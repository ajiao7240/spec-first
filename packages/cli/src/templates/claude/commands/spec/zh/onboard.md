## 核心哲学：为什么这个工作流存在

AI 辅助开发有三个基本挑战：

### 挑战 1：AI 没有记忆

每个 AI 会话都从空白开始。不像人类工程师在数周/数月中积累项目知识，AI 在会话结束时忘记一切。

**问题**：没有记忆，AI 会重复问同样的问题，犯同样的错误，无法在之前的工作基础上构建。

**解决方案**：`.spec-first/workspace/` 系统捕获每个会话中发生的事情——做了什么、学到了什么、解决了什么问题。`/spec:start` 命令在会话开始时读取这个历史，给 AI "人工记忆"。

### 挑战 2：AI 有通用知识，没有项目特定知识

AI 模型在数百万代码库上训练——它们知道 React、TypeScript、数据库等的通用模式。但它们不知道你项目的约定。

**问题**：AI 写的代码"能用"但不符合你项目的风格。它使用与现有代码冲突的模式。它做出违反不成文团队规则的决定。

**解决方案**：`.spec-first/spec/` 目录包含项目特定的规范。`/before-*-dev` 命令在编码开始前将这些专门知识注入到 AI 上下文中。

### 挑战 3：AI 上下文窗口有限

即使注入了规范，AI 的上下文窗口也是有限的。随着对话增长，早期上下文（包括规范）被推出或变得不那么有影响力。

**问题**：AI 开始遵循规范，但随着会话进展和上下文填满，它"忘记"规则并恢复到通用模式。

**解决方案**：`/check-*` 命令在编写代码后重新验证代码是否符合规范，捕获开发过程中发生的偏离。`/spec:finish-work` 命令进行最终的整体审查。

## 系统结构

```
.spec-first/
|-- .developer              # 你的身份（gitignored）
|-- workflow.md             # 完整工作流文档
|-- workspace/              # "AI 记忆" - 会话历史
|   |-- index.md            # 所有开发者的进度
|   +-- {developer}/        # 每个开发者的目录
|       |-- index.md        # 个人进度索引
|       +-- journal-N.md    # 会话记录（最多 2000 行）
|-- tasks/                  # 任务跟踪（统一）
|   +-- {MM}-{DD}-{slug}/   # 任务目录
|       |-- task.json       # 任务元数据
|       +-- prd.md          # 需求文档
|-- spec/                   # "AI 训练数据" - 项目知识
|   |-- frontend/           # 前端约定
|   |-- backend/            # 后端约定
|   +-- guides/             # 思维模式
+-- scripts/                # 自动化工具
```

### 理解 spec/ 子目录

**frontend/** - 单层前端知识：
- 组件模式（如何在这个项目中编写组件）
- 状态管理规则（Redux？Zustand？Context？）
- 样式约定（CSS modules？Tailwind？Styled-components？）
- Hook 模式（自定义 hooks、数据获取）

**backend/** - 单层后端知识：
- API 设计模式（REST？GraphQL？tRPC？）
- 数据库约定（查询模式、迁移）
- 错误处理标准
- 日志和监控规则

**guides/** - 跨层思维指南：
- 代码复用思维指南
- 跨层思维指南
- 实现前检查清单

## 命令深入

### /spec:start - 恢复 AI 记忆

**为什么存在**：
当人类工程师加入项目时，他们花费数天/数周学习：这是什么项目？已经构建了什么？正在进行什么？当前状态是什么？

AI 需要同样的入职——但压缩到会话开始的几秒钟。

**它实际上做什么**：
1. 读取开发者身份（我在这个项目中是谁？）
2. 检查 git 状态（什么分支？未提交的更改？）
3. 从 `workspace/` 读取最近的会话历史（之前发生了什么？）
4. 识别活动功能（正在进行什么？）
5. 在做任何更改之前了解当前项目状态

**为什么这很重要**：
- 没有 /spec:start：AI 是盲目的。它可能在错误的分支上工作，与其他人的工作冲突，或重做已完成的工作。
- 有 /spec:start：AI 知道项目上下文，可以从上一个会话中断的地方继续，避免冲突。

---

### /spec:before-dev - 注入专门知识

**为什么存在**：
AI 模型有"预训练知识"——来自数百万代码库的通用模式。但你的项目有不同于通用模式的特定约定。

**它实际上做什么**：
1. 通过 `get_context.py --mode packages` 发现规范层并读取相关规范
2. 将项目特定模式加载到 AI 的工作上下文中：
   - 组件命名约定
   - 状态管理模式
   - 数据库查询模式
   - 错误处理标准

**为什么这很重要**：
- 没有 before-dev：AI 写的通用代码不符合项目风格。
- 有 before-dev：AI 写的代码看起来像代码库的其余部分。

---

### /spec:check - 对抗上下文偏离

**为什么存在**：
AI 上下文窗口容量有限。随着对话进展，会话开始时注入的规范变得不那么有影响力。这导致"上下文偏离"。

**它实际上做什么**：
1. 重新读取之前注入的规范
2. 将编写的代码与这些规范进行比较
3. 运行类型检查器和 linter
4. 识别违规并建议修复

**为什么这很重要**：
- 没有 check-*：上下文偏离不被注意，代码质量下降。
- 有 check-*：偏离在提交前被捕获和纠正。

---

### /spec:check-cross-layer - 多维度验证

**为什么存在**：
大多数 bug 不是来自缺乏技术技能——它们来自"没想到"：
- 在一个地方更改了常量，漏了其他 5 个地方
- 修改了数据库模式，忘记更新 API 层
- 创建了一个工具函数，但类似的已经存在

**它实际上做什么**：
1. 识别你的变更涉及哪些维度
2. 对于每个维度，运行针对性检查：
   - 跨层数据流
   - 代码复用分析
   - 导入路径验证
   - 一致性检查

---

### /spec:finish-work - 整体提交前审查

**为什么存在**：
`/check-*` 命令专注于单层内的代码质量。但实际变更通常有跨切面关注点。

**它实际上做什么**：
1. 整体审查所有变更
2. 检查跨层一致性
3. 识别更广泛的影响
4. 检查新模式是否应该被记录

---

### /spec:record-session - 为未来持久化记忆

**为什么存在**：
AI 在此会话中构建的所有上下文在会话结束时将丢失。下一个会话的 `/spec:start` 需要这些信息。

**它实际上做什么**：
1. 将会话摘要记录到 `workspace/{developer}/journal-N.md`
2. 捕获做了什么、学到了什么以及剩余什么
3. 更新索引文件以便快速查找

## 真实世界工作流示例

### 示例 1：Bug 修复会话

**[1/8] /spec:start** - AI 在接触代码之前需要项目上下文
**[2/8] python3 ./.spec-first/scripts/task.py create "Fix bug" --slug fix-bug** - 跟踪工作以供将来参考
**[3/8] /spec:before-dev** - 注入项目特定的开发规范
**[4/8] 调查并修复 bug** - 实际的开发工作
**[5/8] /spec:check** - 重新验证代码是否符合规范
**[6/8] /spec:finish-work** - 整体跨层审查
**[7/8] 人类测试并提交** - 人类在代码进入仓库前验证
**[8/8] /spec:record-session** - 为未来会话持久化记忆

### 示例 2：规划会话（无代码）

**[1/4] /spec:start** - 即使是非编码工作也需要上下文
**[2/4] python3 ./.spec-first/scripts/task.py create "Planning task" --slug planning-task** - 规划是有价值的工作
**[3/4] 审查文档、创建子任务列表** - 实际的规划工作
**[4/4] /spec:record-session (with --summary)** - 规划决定必须被记录

### 示例 3：代码审查修复

**[1/6] /spec:start** - 从上一个会话恢复上下文
**[2/6] /spec:before-dev** - 在修复前重新注入规范
**[3/6] 修复每个 CR 问题** - 在上下文中有规范的情况下解决反馈
**[4/6] /spec:check** - 验证修复没有引入新问题
**[5/6] /spec:finish-work** - 记录 CR 的经验教训
**[6/6] 人类提交，然后 /spec:record-session** - 保留 CR 经验教训

### 示例 4：大型重构

**[1/5] /spec:start** - 在重大更改前清晰的基线
**[2/5] 规划阶段** - 分解为可验证的块
**[3/5] 逐阶段执行，每个阶段后使用 /spec:check** - 增量验证
**[4/5] /spec:finish-work** - 检查新模式是否应该被记录
**[5/5] 用多个提交哈希记录** - 将所有提交链接到一个功能

### 示例 5：调试会话

**[1/6] /spec:start** - 查看这个 bug 是否之前被调查过
**[2/6] /spec:before-dev** - 规范可能记录了已知的陷阱
**[3/6] 调查** - 实际的调试工作
**[4/6] /spec:check** - 验证调试更改不会破坏其他东西
**[5/6] /spec:finish-work** - 调试发现可能需要记录
**[6/6] 人类提交，然后 /spec:record-session** - 调试知识是有价值的

## 要强调的关键规则

1. **AI 永远不提交** - 人类测试和批准。AI 准备，人类验证。
2. **代码前规范** - /before-dev 命令注入项目知识。
3. **代码后检查** - /check-* 命令捕获上下文偏离。
4. **记录一切** - /spec:record-session 持久化记忆。

# 第三部分：自定义你的开发规范

解释第一部分和第二部分后，检查项目的开发规范是否需要自定义。

## 步骤 1：检查当前规范状态

检查 `.spec-first/spec/` 是否包含空模板或自定义规范：

```bash
# Check if files are still empty templates (look for placeholder text)
grep -l "To be filled by the team" .spec-first/spec/backend/*.md 2>/dev/null | wc -l
grep -l "To be filled by the team" .spec-first/spec/frontend/*.md 2>/dev/null | wc -l
```

## 步骤 2：确定情况

**情况 A：首次设置（空模板）**

如果规范是空模板（包含 "To be filled by the team"），这是第一次在这个项目中使用 spec-first。

向开发者解释：

"我看到 `.spec-first/spec/` 中的开发规范仍然是空模板。这对于新的 spec-first 设置来说是正常的！

模板包含需要用你项目实际约定替换的占位符文本。没有这个，`/before-*-dev` 命令不会提供有用的指导。

**你的第一个任务应该是填写这些规范：**

1. 查看你现有的代码库
2. 识别已经在使用的模式和约定
3. 在规范文件中记录它们

例如，对于 `.spec-first/spec/backend/database-guidelines.md`：
- 你的项目使用什么 ORM/查询库？
- 迁移是如何管理的？
- 表/列的命名约定是什么？

你想让我帮助你分析代码库并填写这些规范吗？"

**情况 B：规范已自定义**

如果规范有真实内容（没有 "To be filled" 占位符），这是一个现有的设置。

向开发者解释：

"太好了！你的团队已经自定义了开发规范。你可以立即开始使用 `/before-*-dev` 命令。

我建议阅读 `.spec-first/spec/` 以熟悉团队的编码标准。"

## 步骤 3：帮助填写规范（如果为空）

如果开发者想要帮助填写规范，创建一个功能来跟踪这个：

```bash
python3 ./.spec-first/scripts/task.py create "Fill spec guidelines" --slug fill-spec-guidelines
```

然后系统地分析代码库并填写每个规范文件：

1. **分析代码库** - 查看现有代码模式
2. **记录约定** - 写你观察到的，而不是理想的
3. **包含示例** - 引用项目中的实际文件
4. **列出禁止模式** - 记录团队避免的反模式

一次处理一个文件：
- `backend/directory-structure.md`
- `backend/database-guidelines.md`
- `backend/error-handling.md`
- `backend/quality-guidelines.md`
- `backend/logging-guidelines.md`
- `frontend/directory-structure.md`
- `frontend/component-guidelines.md`
- `frontend/hook-guidelines.md`
- `frontend/state-management.md`
- `frontend/quality-guidelines.md`
- `frontend/type-safety.md`

# spec-app-consistency-audit 50 轮演化审查记录

> 记录时间：2026-05-01 15:00:22
> 目标文档：`docs/02-架构设计/spec_app_consistency_audit_技术方案.md`
> 姿态：Spec-First Evolution Architect；小步迭代；以 `Light contract / Explicit boundaries / Scripts prepare, LLM decides` 为判断基线。

## 总体结论

本轮 50 次审查将方案从“功能蓝图”推进为更可落地的 spec-first workflow 设计：修正 source/runtime 真相源边界，收紧输入降级和本地读取策略，明确 Figma MCP 的宿主能力边界，补齐 candidate/confirmed/provenance 证据契约，限制 rule pack 只能作为审查参考，加入行业包激活确认、产物安全、writeback 确认、工程质量候选事实、报告降级展示和 v0.1 pilot 验证门槛。

## 逐轮记录

### Round 01
- 问题描述：Skill 目录仍指向 `.claude/` runtime，违背 source-of-truth 边界。
- 具体修复：将 6.1 改为 `skills/spec-app-consistency-audit/`，并说明 `.claude/`、`.codex/`、`.agents/skills/` 只由 init 生成。
- 修复结论：源码和 runtime 边界重新对齐。
- 修复效果：避免实现者手改生成资产导致 drift。

### Round 02
- 问题描述：公开入口治理缺失，新增 workflow 仅有 skill 目录不够。
- 具体修复：补充 `templates/claude/commands/spec/app-consistency-audit.md` 与 `skills-governance.json` 作为入口治理落点。
- 修复结论：方案覆盖双宿主发现链路。
- 修复效果：降低 Claude/Codex 入口不一致风险。

### Round 03
- 问题描述：PRD、源码、Figma 被写成同等必选，与降级策略冲突。
- 具体修复：将 5.1 改为“最低可运行输入”和“完整审查输入”两层。
- 修复结论：输入语义从硬失败改为可降级。
- 修复效果：缺 PRD 或 Figma 时仍可输出有边界的审查。

### Round 04
- 问题描述：缺少 PRD 和 Figma 同时缺失的处理路径。
- 具体修复：降级策略新增 Code-Architecture-Engineering Quality 审查模式。
- 修复结论：最小输入路径闭合。
- 修复效果：避免空产品/设计证据下伪造一致性结论。

### Round 05
- 问题描述：本地文件读取缺少路径安全策略。
- 具体修复：新增 5.3.1 本地输入读取边界，覆盖 realpath、repo root、allowlist、symlink、大小上限和纯文本解析。
- 修复结论：输入读取职责脚本化。
- 修复效果：减少路径逃逸、解析器风险和敏感文件误读。

### Round 06
- 问题描述：HTML/PDF/DOCX 解析边界不清。
- 具体修复：要求无网络、无脚本执行的纯文本解析。
- 修复结论：复杂文档解析不扩大攻击面。
- 修复效果：降低静态审查引入外部执行副作用的风险。

### Round 07
- 问题描述：Figma MCP 被放入 Node 脚本职责，混淆工具边界。
- 具体修复：将 `extract-figma-contract.js` 改为读取宿主会话生成的 `figma-context.json`。
- 修复结论：MCP live access 归宿主会话，脚本只做归一化。
- 修复效果：符合 `Scripts prepare, LLM/session owns MCP` 的边界。

### Round 08
- 问题描述：Figma token 和节点读取范围未定义。
- 具体修复：新增 Figma MCP 信任边界，限制 file/node/frame scope 并禁止 token 进入日志和报告。
- 修复结论：Figma 集成有明确 trust boundary。
- 修复效果：降低凭证泄露和过度读取风险。

### Round 09
- 问题描述：Figma 文本可能作为 prompt injection 进入 LLM。
- 具体修复：明确 Figma 文本和节点名作为不可信数据，进入 prompt 前需隔离和转义。
- 修复结论：外部设计内容不再被默认信任。
- 修复效果：提升 LLM 审查输入安全性。

### Round 10
- 问题描述：`.spec-first/app-audit/` 产物可能包含敏感信息。
- 具体修复：新增产物安全策略，默认视为内部敏感执行产物。
- 修复结论：artifact 数据分类明确。
- 修复效果：减少报告、证据、路径和产品信息泄露面。

### Round 11
- 问题描述：报告可能保存完整 PRD/Figma/源码片段。
- 具体修复：要求默认保存摘要、定位和 evidence summary，不保存完整原文。
- 修复结论：证据粒度从 raw content 降为 summary-first。
- 修复效果：保留可审计性，同时降低敏感信息外泄。

### Round 12
- 问题描述：缺少 secret/PII 脱敏要求。
- 具体修复：产物安全策略加入 token、手机号、证件号、验证码、交易密码等脱敏扫描。
- 修复结论：报告生成前有敏感数据清洗要求。
- 修复效果：提升 App 审查对隐私与合规的基本保护。

### Round 13
- 问题描述：`.spec-first/app-audit/` 是否被 git 跟踪未定义。
- 具体修复：要求 Preflight 对可能被 git 跟踪的 app-audit 目录给出警告。
- 修复结论：产物持久化风险前置。
- 修复效果：避免执行产物误入源码仓库。

### Round 14
- 问题描述：raw evidence 保存策略缺失。
- 具体修复：规定 raw evidence 需要显式 opt-in 并标记保留策略。
- 修复结论：高敏感证据不默认长期保存。
- 修复效果：让审查证据管理更符合最小必要原则。

### Round 15
- 问题描述：PRD contract 脚本输出“草案”但下游可能当事实。
- 具体修复：增加脚本语义条目默认 `status: candidate`，LLM 回查证据后才能 confirmed。
- 修复结论：候选事实和确认事实分层。
- 修复效果：减少抽取误差被专家放大成 confirmed issue。

### Round 16
- 问题描述：Issue 协议缺少事实来源状态。
- 具体修复：新增 `contract_status` 字段。
- 修复结论：issue 可区分 candidate 与 confirmed。
- 修复效果：Evidence Auditor 可确定哪些问题能进入最终报告。

### Round 17
- 问题描述：Issue 协议缺少 provenance。
- 具体修复：新增 `provenance` 数组，记录项目特定 source、file 和 span。
- 修复结论：confirmed issue 必须可追溯。
- 修复效果：提升报告可复查性与下游 spec-plan 可用性。

### Round 18
- 问题描述：Rule pack 可能被当作唯一 evidence。
- 具体修复：字段约束明确 rule pack 不计入项目特定 source。
- 修复结论：规则包回到参考/rationale 角色。
- 修复效果：避免 generic checklist 直接生成项目问题。

### Round 19
- 问题描述：Issue 协议缺少运行验证详情对象。
- 具体修复：补充 `runtime_verification` 对象，包含 required、level、reason。
- 修复结论：布尔字段和验证详情并存。
- 修复效果：报告能解释为什么需要模拟器或真机。

### Round 20
- 问题描述：Issue 协议缺少数据敏感度。
- 具体修复：新增 `data_sensitivity` 字段及 public/internal/confidential/restricted 枚举。
- 修复结论：问题级敏感度可表达。
- 修复效果：便于报告输出、共享和保留策略分级。

### Round 21
- 问题描述：行业识别要求人工确认，但规则包选择可直接消费候选。
- 具体修复：Phase 9 增加非 common 行业规则包必须用户指定或确认 profile 后才能产生 confirmed issue。
- 修复结论：行业候选不再自动变成强规则。
- 修复效果：降低混合业务 App 被错误行业规则误判的风险。

### Round 22
- 问题描述：未确认行业下专家输出等级不明。
- 具体修复：规定未确认行业时行业专家只能输出 advisory risk。
- 修复结论：行业不确定性进入报告语义。
- 修复效果：避免 advisory 被误读为 confirmed blocker。

### Round 23
- 问题描述：Rule Pack Selection 缺少激活规则。
- 具体修复：新增激活规则，区分自动启用基础包和需确认的行业包。
- 修复结论：规则包激活从隐式变显式。
- 修复效果：降低 rule pack 选择对 LLM 的隐形偏置。

### Round 24
- 问题描述：confirmed issue 的 evidence 要求不够硬。
- 具体修复：要求 confirmed issue 至少包含 PRD、Figma、Code、Architecture、Analytics 或 I18n 中一个项目特定证据。
- 修复结论：No evidence, no issue 得到机械约束。
- 修复效果：专家审查更少幻觉和泛化误报。

### Round 25
- 问题描述：Rule pack 在输出中的位置不清。
- 具体修复：规定 rule pack 写入 `related_rule_packs` 或 rationale，不作为唯一 evidence。
- 修复结论：影响解释与事实证据分离。
- 修复效果：让行业规则能增强判断但不替代项目事实。

### Round 26

## 收口校验补记

- 问题描述：最终一致性检查发现证据策略仍有两处残留表达，把 Rule Pack 与项目特定 evidence 并列，可能导致通用规则包被误读为 confirmed issue 的唯一证据。
- 具体修复：将证据策略和 `SKILL.md` 草案统一改为“Rule Pack 只能作为规则依据、解释上下文或 `related_rule_packs`”，confirmed issue 必须绑定 PRD / Figma / Code / Architecture / Analytics / I18n / Contract 中至少一种项目特定证据。
- 修复结论：Rule Pack 从事实证据层退出，保留为知识增强层。
- 修复效果：降低行业包、通用 checklist 和专家偏置直接制造项目问题的风险。

- 问题描述：Artifact 只声明了 `schema_version`，没有系统性表达 source、freshness、consumer 和敏感度，和项目角色契约对 artifact 的要求不完全对齐。
- 具体修复：新增 `6.3 Artifact 元数据契约`，规定 `source_inputs`、`source_hash` / revision、`freshness`、`consumers`、`contract_status` 和 `data_sensitivity`；同时扩展 `validate-artifacts.js` 职责。

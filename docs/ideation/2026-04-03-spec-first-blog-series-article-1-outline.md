# 第一篇详细大纲：为什么 AI 编程需要一套工程系统，而不只是一个更强的模型

这篇是整组连载的锚点。它不负责讲功能细节，而是负责定义问题、定义边界、定义整个系列的阅读预期。

## 文章目标

- 让读者明白 `spec-first` 不是命令集合
- 让读者明白它解决的不是“写代码更快”，而是“工程过程更稳”
- 让读者明白后面每一篇都不是孤立功能，而是同一套系统的不同切面

## 文章定位

- 读者对象：AI Coding 使用者、开源工具用户、想把工作流工程化的开发者
- 传播目标：先建立认知，再引出系列
- 内容风格：问题驱动，避免术语堆叠，尽量用真实场景开场

## 建议标题

- `为什么 AI 编程需要一套工程系统，而不只是一个更强的模型`
- 备选：`从 Vibe Coding 到 Harness Engineering：Spec-First 想解决什么`
- 备选：`从 Vibe Coding 到工程系统：Spec-First 想解决什么`
- 备选：`Spec-First：把 AI 编程从对话变成工程系统`

## 这一篇要增加的外部参照

这一篇不只讲 `spec-first` 自己，还要借 `Harness engineering` 把问题说得更清楚：

- `Vibe Coding` 更强调“自然语言描述需求，先把代码生成出来”
- `Harness engineering` 更强调“人类设计环境、约束和反馈回路，让 Agent 能可靠执行”
- `spec-first` 可以被写成：把 harness 思想继续推进到跨宿主工作流、治理和知识复利层

这里要避免两个误区：

- 不要把 `spec-first` 直接等同于 `Harness engineering`
- 也不要把 harness 只写成一个流行术语，而不解释它为什么对这个项目重要

## 开头引子

先把读者熟悉的体验说出来：

- AI 写代码很快
- 但上下文会散
- 计划会散
- 评审会散
- 知识不会自动沉淀

接着抛出核心问题：

- 问题真的只是模型不够强吗
- 还是我们缺少一套把 AI 编程“工程化”的系统

## 正文结构

### 1. 先讲现状：为什么很多 AI 编程工具“短期爽，长期乱”

**要点**

- 单次对话能解决局部问题
- 项目一旦变大，问题会从“不会写”变成“难以持续”
- 真正拖慢团队的不是生成速度，而是工程边界不稳定

**建议表达**

- 不要直接批评模型
- 先承认它在局部问题上的价值
- 再指出它缺少的是系统层的约束

### 2. 再讲本质：AI 编程缺的不是灵感，而是工程系统

**要点**

- 没有稳定上下文，下一轮还是从零开始
- 没有治理，规则只存在于记忆里
- 没有复盘，修复只能重复发生
- 没有资产沉淀，经验无法复用
- `Harness engineering` 提醒我们：真正该优化的对象，不只是 prompt，而是 Agent 所在的执行环境

**建议表达**

- 把“AI 编程”提升成“工程交付”
- 让读者意识到问题范围不只是 prompt

### 3. 再讲 `spec-first` 的定位

**要点**

- 它不是 prompt 集合
- 它不是某个宿主的附属插件
- 它是一套可安装、可治理、可复用的工程工作流系统
- 它可以被理解为：以 harness 思想为参照，把环境设计继续推进到工作流编排、治理注入和知识沉淀

**建议表达**

- 用一句话定义它
- 再用三个关键词展开：安装、治理、复用

### 4. 再讲整套系统的骨架

**要点**

- `doctor / init / clean` 管运行时资产
- `/spec:*` 和 `$spec-*` 提供稳定入口
- `Stage-0` 先建上下文
- `Ideate -> Brainstorm -> Plan -> Work -> Review -> Compound` 形成闭环
- `docs/solutions`、`docs/plans`、`docs/brainstorms` 形成知识复利
- 如果说 harness 更强调“让 Agent 能在仓库里可靠工作”，那么 `spec-first` 更进一步强调“让整个 AI 交付过程可以被治理、复用和持续演进”

**建议表达**

- 不要一口气把所有名词都解释完
- 只需要让读者知道：后面会逐个拆开

### 5. 再讲 `Harness engineering` 和 `spec-first` 的关系与边界

**要点**

- 两者共同点：都强调仓库内事实来源、环境设计、约束编码、验证回路
- 两者差异：`Harness engineering` 更偏 agent-first 的仓库工程；`spec-first` 更偏工作流产品化、治理和知识闭环
- 因此这组连载不是“蹭概念”，而是借这个概念把 `spec-first` 放到正确坐标系里

**建议表达**

- 用“上位参照”和“具体实现”来表达关系
- 避免写成“我们也有这个概念，所以我们是一样的”

## 建议小标题

1. AI 编程为什么会“短期爽，长期乱”
2. 单次对话为什么不够
3. `Harness engineering` 到底补上了什么
4. `spec-first` 不是 prompt 集合
5. 它到底是一套什么系统
6. 这套系统后面会怎么拆

## 结尾转场

结尾不要收得太死，要留出下一篇的入口。

**建议转场句**

- 如果总论讲的是“为什么需要这套系统”，下一篇就讲它的第一块底座：为什么必须先建上下文，再谈执行。
- 真正决定这套系统能不能成立的，不是某个命令，而是 Stage-0。

# spec-first 工作流分析

本文档只做一件事：把 spec-first 的主工作流按当前代码真实行为讲清楚。

它覆盖两条入口，但最终都会收敛到同一条执行链：

1. 新项目初始化，先补齐项目规范
2. 已存在项目迭代，直接围绕需求创建任务

结论先行：

- `.spec-first/spec/` 是长期知识层，初始化时写入骨架，后续由项目自己维护。
- `.spec-first/tasks/` 是任务层，承载 PRD、状态、阶段动作和上下文清单。
- `.spec-first/workspace/` 是会话层，既有共享入口，也有按开发者隔离的 journal / index。
- `.spec-first/.current-task` 是当前任务锚点，hook、上下文恢复和 session 追踪都会依赖它。

---

## 0. 这条链路里有哪些文件

先把真实落盘点对齐，后面看流程会更清楚。

- `.spec-first/workflow.md`：工作流总说明
- `.spec-first/config.yaml`：工作流配置
- `.spec-first/.developer`：项目级开发者身份
- `.spec-first/workspace/index.md`：共享 workspace 入口页，由 `init` 创建
- `.spec-first/workspace/<developer>/journal-N.md`：开发者会话日志
- `.spec-first/workspace/<developer>/index.md`：开发者会话索引
- `.spec-first/.current-task`：当前任务路径
- `.spec-first/tasks/<task>/task.json`：任务元数据
- `.spec-first/tasks/<task>/prd.md`：任务说明
- `.spec-first/tasks/<task>/implement.jsonl`：实现阶段上下文清单
- `.spec-first/tasks/<task>/check.jsonl`：检查阶段上下文清单
- `.spec-first/tasks/<task>/debug.jsonl`：调试阶段上下文清单
- `.spec-first/spec/`：项目规范知识库

---

## 1. 两条入口

```text
spec-first 开始工作
├─ 路径 A：初始化 / 第一次接入
│  ├─ spec-first init
│  ├─ createWorkflowStructure
│  ├─ （可选）解析开发者身份并初始化 workspace/<developer>/
│  ├─ （可选）创建 00-bootstrap-guidelines 并写入 .current-task
│  └─ 基于真实代码补齐 spec/
└─ 路径 B：已存在项目迭代
   ├─ 需求澄清 / 复杂度判断
   ├─ task create
   ├─ 写 PRD
   ├─ task init-context
   ├─ 激活当前任务
   ├─ 实现 / 检查 / 收尾
   ├─ add_session
   └─ archive / create-pr
```

这两条路不是两套系统，只是起点不同。最终都回到同一套任务、上下文和会话记录机制。

## 2. 代码对应关系

### 2.1 `init_developer`

相关代码：

- `packages/cli/src/commands/init.ts`
- `packages/cli/src/configurators/workflow.ts`
- `.spec-first/scripts/init_developer.py`
- `.spec-first/scripts/common/developer.py`

真实作用要拆成两层看：

第一层，`spec-first init` 一定会创建工作流骨架：

- `.spec-first/scripts/`
- `.spec-first/workflow.md`
- `.spec-first/config.yaml`
- `.spec-first/workspace/index.md`
- `.spec-first/tasks/`
- `.spec-first/spec/` 模板骨架
- `.spec-first/worktree.yaml`，因为当前 `init.ts` 固定以 `multiAgent: true` 调用 `createWorkflowStructure()`

第二层，开发者身份初始化是“条件成立时再做”：

- `init.ts` 会按 `-u/--user` -> 项目级 `.developer` -> 全局 `~/.spec-first/.developer` -> `git config user.name` 的顺序解析开发者名
- 如果解析到了开发者名，就调用 `init_developer.py`
- `init_developer.py` / `common/developer.py` 会创建 `.spec-first/.developer`
- 同时创建 `.spec-first/workspace/<developer>/`
- 并创建 `journal-1.md` 和该开发者自己的 `index.md`

所以，`init` 负责骨架；`init_developer` 负责把“谁在工作”和“这个人的会话空间”写入文件系统。

对应关系可以压缩成下面这张图：

```text
spec-first init
   │
   ├─ createWorkflowStructure(...)
   │   ├─ 写 workflow.md / config.yaml
   │   ├─ 写 workspace/index.md / tasks/
   │   ├─ 写 spec/ 骨架
   │   └─ 写 worktree.yaml
   │
   └─ 若识别到 developerName
       └─ init_developer.py
           ├─ 写 .developer
           ├─ 建 workspace/<developer>/
           ├─ 建 journal-1.md
           └─ 建 workspace/<developer>/index.md
```

### 2.2 `get_context`

相关代码：

- `.spec-first/scripts/get_context.py`
- `.spec-first/scripts/common/git_context.py`
- `.spec-first/scripts/common/session_context.py`
- `.spec-first/scripts/common/packages_context.py`

`get_context.py` 现在只是入口壳，真正分三种模式：

- `default`：完整会话上下文
- `record`：给 record-session 用的聚焦输出
- `packages`：只看 package / spec layer 结构

`default` 模式会采集：

- 开发者身份
- 当前 Git 分支和工作区脏/净状态
- 最近 5 条提交
- 当前任务和 `prd.md` 提示
- 活跃任务树和“我的任务”
- 当前 journal 文件、行数和接近上限提示
- monorepo 下独立 Git package 的状态
- package 与 spec layer 概览

所以它不是“读一个配置”，而是把当前工作面板拼出来。

```text
get_context.py
   │
   └─ git_context.main()
       ├─ --mode default  -> session_context.get_context_text/json
       ├─ --mode record   -> session_context.get_context_text_record/json
       └─ --mode packages -> packages_context.get_context_packages_text/json
```

### 2.3 `create_bootstrap`

相关代码：

- `packages/cli/src/commands/init.ts`
- `.spec-first/scripts/create_bootstrap.py`

当前 CLI 初始化链路里，bootstrap 任务的主入口是 `init.ts` 内部的 `createBootstrapTask()`，不是 shell 再去调用 Python 脚本。

它会创建 `00-bootstrap-guidelines`，并写入：

- `task.json`
- `prd.md`
- `.current-task`

关键特点：

- `status` 是 `in_progress`
- `dev_type` 是 `docs`
- `priority` 是 `P1`
- `relatedFiles` 指向 `.spec-first/spec/` 下待补齐的规范目录
- `prd.md` 明确要求“基于现有代码和现有规则文件”补齐规范

monorepo 下，bootstrap 会按 package 生成 subtasks 和 PRD 区块。

还有两个容易写错的点：

- bootstrap 任务只在 `init` 成功解析出开发者身份后才会创建；`--yes` 且未识别到用户名时，骨架会建好，但 bootstrap 不一定存在
- `.spec-first/scripts/create_bootstrap.py` 仍然是有效脚本入口，但当前 CLI 主路径已经内联了同等逻辑

```text
init.ts
  └─ createBootstrapTask(...)
      ├─ 建 .spec-first/tasks/00-bootstrap-guidelines/
      ├─ 写 task.json
      ├─ 写 prd.md
      └─ 写 .current-task

create_bootstrap.py
  └─ 保留为等价脚本入口
```

### 2.4 `task create`

相关代码：

- `.spec-first/scripts/common/task_store.py`
- `.spec-first/scripts/common/workflow_templates.py`

`cmd_create` 会创建新的任务目录并写入 `task.json`。当前版本里最重要的字段不是一个固定字符串，而是一组结构化阶段信息：

- `status: "planning"`
- `base_branch`：创建任务时的当前 Git 分支
- `workflow_type`：任务采用的工作流模板类型，默认是 `default`
- `next_action`：由工作流模板生成的 phase/action 数组，不同模板不同
- `decision_hints`：按工作流类型生成的默认提示
- `children` / `parent`：子任务关系
- `package`：monorepo 包信息

所以这一步不是简单“开目录”，而是把任务放进一份可执行的流程契约里。

```text
task.py create
   │
   └─ cmd_create
       ├─ 解析 package / assignee / slug
       ├─ 读取当前 Git 分支 -> base_branch
       ├─ 读取 workflow template
       ├─ 生成 next_action / decision_hints
       ├─ 写 task.json
       └─ （可选）维护 parent / children
```

### 2.5 `task init-context`

相关代码：

- `.spec-first/scripts/common/task_context.py`

这个命令会生成三份 JSONL：

- `implement.jsonl`
- `check.jsonl`
- `debug.jsonl`

默认注入策略是“给最小够用的上下文”，而不是把整个 spec 树一次性塞进去。

当前默认规则如下：

- `implement.jsonl` 一定包含 `.spec-first/workflow.md`
- `backend` / `test`：再注入 `.spec-first/spec[/<package>]/backend/index.md`
- `frontend`：再注入 `.spec-first/spec[/<package>]/frontend/index.md`
- `fullstack`：前后端 index 都注入
- `docs`：除了 `workflow.md` 之外没有额外默认实现规范
- `check.jsonl`：注入当前平台适配器解析出来的 `finish-work` 和 `check`
- `debug.jsonl`：默认只注入 `check`

monorepo 下的 package 解析顺序也已经固化：

1. CLI `--package`
2. `task.json.package`
3. `config.yaml` 里的 `default_package`

如果是 monorepo 且最终仍然解析不到 package，`init-context` 会直接报错，而不是悄悄退回全局 `spec/`。

```text
task.py init-context <dir> <dev_type>
   │
   ├─ resolve package
   │   ├─ --package
   │   ├─ task.json.package
   │   └─ config.yaml default_package
   │
   ├─ 写 implement.jsonl
   │   ├─ workflow.md
   │   └─ backend/frontend index.md
   │
   ├─ 写 check.jsonl
   │   └─ finish-work + check
   │
   └─ 写 debug.jsonl
       └─ check
```

### 2.6 `current_task.py switch` / `task.py start`

相关代码：

- `.spec-first/scripts/current_task.py`
- `.spec-first/scripts/task.py`
- `packages/cli/test/commands/current-task.integration.test.ts`

当前任务通过 `.spec-first/.current-task` 记录，内容是 repo-relative 的任务路径。

这里要区分两个入口：

- `current_task.py list/switch`：面向“人先看列表再选任务”的用户入口
- `task.py start`：面向“已经拿到确切任务路径”的低层原语

`current_task.py switch` 现在支持：

- 列表序号
- 精确任务目录名
- 精确 repo-relative 路径
- 仓库内绝对路径

`task.py start` 现在只接受精确名称或精确路径，不再做 suffix 模糊匹配。

所以更准确的表述是：

- 人工切换、多任务选择时，优先 `current_task.py list` / `switch`
- 自动化流程里如果已经持有 `TASK_DIR`，可以直接 `task.py start "$TASK_DIR"`
- 两条路径最终都会写 `.current-task`，并触发 `after_start` hooks

```text
人工入口
  current_task.py list
        │
        └─ current_task.py switch <selection>
             ├─ 序号 / 精确名 / 精确路径
             ├─ 写 .current-task
             └─ run after_start hooks

脚本入口
  task.py start <exact-path-or-name>
        ├─ 仅精确匹配
        ├─ 写 .current-task
        └─ run after_start hooks
```

### 2.7 `add_session`

相关代码：

- `.spec-first/scripts/add_session.py`
- `.spec-first/scripts/common/developer.py`
- `.spec-first/scripts/common/config.py`

记录会话时，`add_session` 当前会做四件事：

1. 追加 session 内容到当前 `journal-N.md`
2. 超过上限时自动新建下一份 journal
3. 更新开发者自己的 `workspace/<developer>/index.md`
4. 默认自动提交 `.spec-first/workspace` 和 `.spec-first/tasks`

几个关键细节：

- journal 上限不是硬编码在 `add_session.py`，而是走 `config.yaml` 的 `max_journal_lines`，默认值是 2000
- `get_context` 在 journal 超过 1800 行时会给出接近上限提示
- 自动提交默认开启，只有显式传 `--no-commit` 才跳过
- 分支解析顺序是 `--branch` -> 当前任务 `task.json.branch` -> 当前 Git 分支

所以它不是“写一段日志”而已，而是顺手把会话索引和工作流状态一起推进。

```text
add_session.py
   │
   ├─ 找到当前 journal-N.md
   ├─ 计算追加后行数
   ├─ 若超 max_journal_lines
   │   └─ 新建 journal-(N+1).md
   ├─ 追加 session 内容
   ├─ 更新 workspace/<developer>/index.md
   └─ 默认 git add + git commit
       .spec-first/workspace
       .spec-first/tasks
```

### 2.8 `spec/` 的写入时机

`spec/` 的第一次写入发生在 `spec-first init` 过程中，但要分“本地空模板”和“远程模板”两种情况理解。

简化调用链：

```text
spec-first init
  -> init(options)
  -> createWorkflowStructure(...)
  -> createSpecTemplates(...)   # 仅在需要写本地空模板时执行
  -> （可选）init_developer.py
  -> （可选）createBootstrapTask(...)
```

更精确一点：

```text
用户执行 spec-first init
        │
        ▼
CLI 入口 packages/cli/src/cli/index.ts
        │
        ▼
init(options)
        │
        ├─ 解析 developer / projectType / monorepo / template 来源
        ├─ 如选择远程模板，先下载模板内容到 .spec-first/spec/
        ├─ createWorkflowStructure(...)
        │    ├─ 拷贝 scripts/
        │    ├─ 写 workflow.md / config.yaml / workspace/index.md / tasks/
        │    ├─ 写 worktree.yaml
        │    └─ 仅在未跳过时写本地 spec 模板
        ├─ （可选）写 monorepo packages 到 config.yaml
        ├─ （可选）初始化项目级 developer
        └─ （可选）创建 bootstrap 任务
```

关于 `spec/`，当前代码有三条很重要的边界：

- 单仓库下，如果使用远程模板，`skipSpecTemplates` 会阻止本地空模板再次覆盖 `spec/`
- monorepo 下，`remoteSpecPackages` 允许“只跳过部分 package 的空模板”，但共享 `spec/guides/` 仍会创建
- `spec-first update` 把 `.spec-first/spec/` 当作受保护的用户内容，不会去覆盖它

后续新增知识的主路径也不是 `spec-first update`，而是 `update-spec` 这类把实现契约写回 `spec/` 的工作流。

## 3. 主流程怎么串起来

### 3.1 新项目初始化

目标是先建立工作流骨架，再把项目规范补齐到可用状态：

1. 运行 `spec-first init`
2. 创建 `.spec-first/` 骨架和 `spec/` 初始内容
3. 如已识别开发者，则初始化个人 workspace
4. 如已识别开发者，则创建 bootstrap 任务并写入 `.current-task`
5. 基于真实代码、现有规则文件和现有目录结构补齐 `.spec-first/spec/`
6. 再进入正常需求迭代

```text
新项目初始化
   │
   ├─ spec-first init
   ├─ 创建 .spec-first/ 骨架
   ├─ 创建 spec/ 初始内容
   ├─ （有 developer 时）初始化个人 workspace
   ├─ （有 developer 时）创建 bootstrap 任务
   └─ 基于真实代码补齐 spec/
```

### 3.2 已存在项目迭代

目标是把需求变成可执行、可恢复、可记录的任务：

1. 接收需求并判断复杂度
2. 创建任务
3. 写 PRD
4. 初始化上下文
5. 激活当前任务
6. 实现、检查、收尾
7. 记录 session
8. 归档任务或创建 PR

两条路最后都落到同一套文件系统状态上，这样 AI 才能恢复上一次工作的现场。

```text
需求进入
  ├─ 澄清 / 判断复杂度
  ├─ task create
  ├─ 写 prd.md
  ├─ task init-context
  ├─ current_task.py switch 或 task.py start
  ├─ implement / check / finish
  ├─ add_session
  └─ archive / create-pr
```

## 4. 写文档时要守住的边界

- `spec` 是长期知识，不是临时草稿
- `tasks` 是任务状态与执行契约，不是随手记笔记的地方
- `workspace` 是会话历史，不是缓存目录
- `.current-task` 是当前任务锚点，不是任务列表本身
- `spec-first update` 升级框架时不应修改 `spec/`，因为那是用户知识

如果这几个边界写乱了，后续读者会把流程、文件和脚本之间的关系看反。

### 4.1 一句话总结

- `spec-first init` 负责初始化工作流骨架，并按条件创建 bootstrap 入口
- `update-spec` 负责把新知识写回 `spec/`
- `spec-first update` 负责升级框架，不负责改用户 `spec/`

# spec-first 工作流分析

本文档只做一件事：把 spec-first 的主工作流按代码真实行为讲清楚。

它覆盖两条入口，但最终都会收敛到同一条执行链：

1. 新项目初始化，先补齐项目规范
2. 已存在项目迭代，直接围绕需求创建任务

结论先行：

- `spec` 是长期知识层，存放项目规范。
- `tasks` 是任务层，存放单次交付的目标、状态和上下文。
- `workspace` 是会话层，存放 journal、index 和当前开发者的历史。
- `.current-task` 是当前锚点，后续上下文生成和恢复都会依赖它。

---

## 0. 这条链路里有哪些文件

先把真实落盘点对齐，后面看流程会更清楚。

- `.spec-first/.developer`：开发者身份
- `.spec-first/workspace/<developer>/journal-N.md`：会话日志
- `.spec-first/workspace/<developer>/index.md`：会话索引
- `.spec-first/.current-task`：当前任务路径
- `.spec-first/tasks/<task>/task.json`：任务元数据
- `.spec-first/tasks/<task>/prd.md`：任务说明
- `.spec-first/tasks/<task>/implement.jsonl`：实现上下文
- `.spec-first/tasks/<task>/check.jsonl`：检查上下文
- `.spec-first/tasks/<task>/debug.jsonl`：调试上下文
- `.spec-first/spec/`：项目规范知识库

---

## 1. 两条入口

```text
spec-first 开始工作
├─ 新项目 / 第一次接入
│  ├─ init_developer
│  ├─ get_context
│  ├─ create_bootstrap
│  └─ 进入规范补齐
└─ 已存在项目 / 正常迭代
   ├─ 需求输入
   ├─ task create
   ├─ 写 PRD
   ├─ task init-context
   ├─ task start
   └─ 进入实现、检查、记录、归档
```

这两条路不是两套系统，只是起点不同。最终都要回到同一套任务、上下文和会话记录机制。

## 2. 代码对应关系

### 2.1 `init_developer`

相关代码：

- `.spec-first/scripts/common/developer.py`
- `packages/cli/src/commands/init.ts`

真实作用：

- 创建 `.spec-first/.developer`
- 创建 `.spec-first/workspace/<developer>/`
- 创建初始 `journal-1.md`
- 创建 `index.md`

这里不是“初始化一个配置”，而是把开发者身份和会话空间写进文件系统。

### 2.2 `get_context`

相关代码：

- `.spec-first/scripts/common/session_context.py`

它会采集：

- 当前 Git 分支
- 工作区是否干净
- 最近 5 条提交
- 活跃任务列表
- 当前任务
- 当前 journal 文件
- monorepo 下各 package 的独立 Git 状态

所以它更像“当前工作面板”，不是简单读取单个配置文件。

### 2.3 `create_bootstrap`

相关代码：

- `.spec-first/scripts/create_bootstrap.py`
- `packages/cli/src/commands/init.ts`

首次初始化时会生成 `00-bootstrap-guidelines`，并写入：

- `task.json`
- `prd.md`
- `.current-task`

它的关键特点：

- `task.json.status` 是 `in_progress`
- `dev_type` 是 `docs`
- `priority` 是 `P1`
- `relatedFiles` 指向 `.spec-first/spec/` 下的规范目录
- `prd.md` 明确要求基于现有代码补齐项目规范

如果是 monorepo，bootstrap 还会按 package 生成不同的规范区块。

这里要注意两件事：

- `create_bootstrap` 创建的是“补规范任务”，不是直接把项目知识写入 spec
- 真正写入 `.spec-first/spec/` 的动作发生在 `createWorkflowStructure()` → `createSpecTemplates()`

### 2.4 `task create`

相关代码：

- `.spec-first/scripts/common/task_store.py`

`cmd_create` 会创建新的任务目录，并写入 `task.json`。这个文件里最重要的字段是：

- `status: "planning"`
- `base_branch`: 当前 Git 分支
- `next_action`: `implement -> check -> finish -> create-pr`
- `children` / `parent`: 子任务关系
- `package`: monorepo 的包信息

这一步完成后，任务已经被系统接管，但还没有进入上下文注入。

### 2.5 `task init-context`

相关代码：

- `.spec-first/scripts/common/task_context.py`

这个命令会生成三份 JSONL：

- `implement.jsonl`
- `check.jsonl`
- `debug.jsonl`

默认注入策略是“给最小够用的上下文”，而不是把整个 spec 树一次性塞进去。

不同 `dev_type` 的默认注入不同：

- `backend` / `test`：注入后端实现指南
- `frontend`：注入前端实现指南
- `fullstack`：前后端都注入

monorepo 下还会先解析 package，再决定该读哪个 `spec/<package>/...`。

### 2.6 `task start`

相关代码：

- `.spec-first/scripts/common/paths.py`
- `.spec-first/scripts/common/session_context.py`

当前任务通过 `.spec-first/.current-task` 记录。后续的上下文恢复、会话记录和任务定位，都依赖这个锚点。

### 2.7 `add_session`

相关代码：

- `.spec-first/scripts/add_session.py`
- `.spec-first/scripts/common/developer.py`

记录会话时会做三件事：

- 追加到当前 `journal-N.md`
- 更新 `index.md`
- 在需要时自动新建下一份 journal

`index.md` 里的 `Total Sessions` 和 `Active File` 会随之更新。默认单份 journal 上限是 2000 行，接近 1800 行时会被标记为接近上限。

### 2.8 `spec/` 的写入时机

`spec/` 的第一次写入发生在 `spec-first init` 过程中，调用链可以简化成：

```text
spec-first init
  -> init(options)
  -> createWorkflowStructure(...)
  -> createSpecTemplates(...)
  -> 写入 .spec-first/spec/
```

ASCII 流程图如下：

```text
用户执行 spec-first init
        │
        ▼
CLI 入口 packages/cli/src/cli/index.ts
        │
        ▼
init(options)
        │
        ├─ 初始化 .developer
        ├─ 创建 workspace/
        ├─ 创建 tasks/
        ├─ 写 config.yaml / worktree.yaml
        │
        ▼
createWorkflowStructure(...)
        │
        ├─ 单仓库 -> createSpecTemplates(cwd, projectType)
        └─ monorepo -> createSpecTemplates(cwd, projectType, packages, ...)
                     │
                     ├─ 写 spec/guides/
                     ├─ 写 spec/backend/ 或 spec/frontend/
                     └─ monorepo 时写 spec/<package>/
```

如果用户选择远程模板，`init()` 会先下载模板内容，再跳过默认 spec 模板写入。

后续新增知识不是走 `spec-first update`，而是走 `update-spec` 体系，把实现契约、边界和新模式写回 `.spec-first/spec/`。

## 3. 主流程怎么串起来

### 3.1 新项目初始化

目标是先建立规范，再让 AI 开始稳定地产出：

1. 初始化开发者
2. 读取上下文
3. 创建 bootstrap 任务
4. 基于真实代码补齐 `.spec-first/spec/`
5. 再进入正常需求迭代

### 3.2 已存在项目迭代

目标是把需求变成可执行任务：

1. 接收需求
2. 判断复杂度
3. 创建任务
4. 写 PRD
5. 初始化上下文
6. 开始执行
7. 记录 session
8. 归档任务

两条路最后都要落到同一套文件系统状态上，这样 AI 才能恢复上一次工作的现场。

---

## 4. 写文档时要守住的边界

- `spec` 是长期知识，不是临时草稿
- `tasks` 是任务状态，不是业务设计文档仓库
- `workspace` 是会话历史，不是缓存目录
- `.current-task` 是当前任务锚点，不是任务列表
- `update` 不应该碰 `spec/`，因为那里是用户知识，不是模板

如果这几个边界写乱了，后续读者会把流程、文件和脚本之间的关系看反。

### 4.1 一句话总结

- `spec-first init` 负责初始化 `spec/` 骨架
- `update-spec` 负责把新知识写回 `spec/`
- `spec-first update` 负责升级框架，不负责改用户 `spec/`

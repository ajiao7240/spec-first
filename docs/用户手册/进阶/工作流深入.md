## 工作流概述

spec-first 的核心工作流可以概括为四个步骤：

```
┌──────────────┐    ┌──────────────┐    ┌──────────────┐    ┌──────────────┐
│  编写 Spec   │ -> │  创建任务    │ -> │  注入上下文  │ -> │  检查验收    │
└──────────────┘    └──────────────┘    └──────────────┘    └──────────────┘
```

### 1. 编写 Spec

在 `.spec-first/spec/` 目录下定义项目规范：

```
.spec-first/spec/
├── architecture.md    # 架构规范
├── coding-standards.md # 编码标准
├── review-rules.md    # 代码审查规则
└── workflow.md        # 工作流偏好
```

### 2. 创建任务

在 `.spec-first/tasks/` 目录下管理任务：

```
.spec-first/tasks/
├── TASK-001/
│   ├── prd.md         # 产品需求文档
│   ├── context.md     # 实现上下文
│   └── status.md      # 任务状态
└── TASK-002/
    └── ...
```

### 3. 注入上下文

spec-first 会自动为当前任务注入相关上下文：

- 相关的 Spec 文件
- 任务的 PRD 和上下文
- 历史会话记录

### 4. 检查验收

完成后进行检查和验收：

- 代码符合 Spec 规范
- 功能满足 PRD 要求
- 更新任务状态

## Spec 自动注入机制

### 工作原理

spec-first 通过 hook 脚本实现上下文自动注入：

1. **会话启动时** (`session-start` hook)
   - 检测当前工作目录
   - 识别活跃任务
   - 加载相关 Spec

2. **工具调用前** (`pre-tool-use` hook)
   - 分析用户意图
   - 注入任务上下文
   - 加载历史记录

### 注入优先级

上下文注入遵循以下优先级：

1. **任务级上下文** - 当前任务的 PRD、状态、历史
2. **Spec 规范** - 相关的编码标准、架构规范
3. **工作空间记忆** - 个人的 journal 和笔记
4. **全局配置** - workflow.md 等共享规则

### 自定义注入

你可以通过编辑 hook 脚本自定义注入逻辑：

```python
# .claude/hooks/inject-subagent-context.py

def get_relevant_specs(task_context):
    """根据任务上下文选择相关的 Spec 文件"""
    specs = []

    # 始终加载编码标准
    specs.append('coding-standards.md')

    # 根据任务类型加载其他 Spec
    if task_context.get('type') == 'api':
        specs.append('api-design.md')
    elif task_context.get('type') == 'frontend':
        specs.append('frontend-standards.md')

    return specs
```

## 任务驱动开发

### 任务生命周期

```
┌─────────┐    ┌─────────┐    ┌─────────┐    ┌─────────┐    ┌─────────┐
│  创建   │ -> │  规划   │ -> │  实现   │ -> │  检查   │ -> │  完成   │
└─────────┘    └─────────┘    └─────────┘    └─────────┘    └─────────┘
```

### 任务状态管理

在 `status.md` 中跟踪任务状态：

```markdown
# TASK-001 状态

## 当前阶段: 实现

- [x] 需求分析
- [x] 技术设计
- [ ] 编码实现 (进行中)
- [ ] 测试验证
- [ ] 代码审查

## 阻塞问题
- 无

## 下一步
- 完成核心功能实现
```

### 任务切换

当需要切换任务时：

```bash
# 1. 记录当前任务进度
# 编辑 .spec-first/tasks/TASK-001/status.md

# 2. 切换到新任务
cd /path/to/project
# spec-first 会自动检测活跃任务

# 3. 恢复时查看历史
# 查看 .spec-first/workspace/<user>/journal/
```

## 会话连续性

### Journal 系统

Journal 记录你的工作历史，保持会话连续性：

```
.spec-first/workspace/<user>/
├── journal/
│   ├── 2024-01-15.md  # 每日工作日志
│   ├── 2024-01-16.md
│   └── ...
├── notes/             # 个人笔记
└── preferences.md     # 个人偏好
```

### Journal 格式

```markdown
# 2024-01-15 工作日志

## 进行中的任务
- TASK-001: 用户认证功能

## 完成的工作
- 实现了登录 API
- 编写了单元测试

## 遇到的问题
- JWT token 过期处理需要确认

## 明天计划
- 完成注册功能
- 集成测试
```

### 恢复上下文

新会话开始时，spec-first 会：

1. 读取最近的 journal
2. 识别进行中的任务
3. 加载任务上下文
4. 恢复工作状态

## 跨平台工作流

### 平台适配

spec-first 支持多个 AI 编码平台，每个平台有不同的接入方式：

| 平台 | 接入方式 | 配置目录 |
|------|----------|----------|
| Claude Code | Python hooks | `.claude/` |
| Cursor | Cursor rules | `.cursor/` |
| Codex | Skills + Config | `.agents/` + `.codex/` |
| OpenCode | 配置集成 | 项目根目录 |

### 统一工作流

无论使用哪个平台，核心工作流保持一致：

```
Spec → Task → Context → Verify
```

平台只是改变了接入方式，不改变工作流本身。

### 平台迁移

从一个平台迁移到另一个平台：

```bash
# 1. 初始化新平台
spec-first init --<new-platform> -u <user> -f

# 2. Spec 和 Task 保持不变
# .spec-first/ 目录是平台无关的

# 3. 使用新平台
# 新平台会读取相同的 Spec 和 Task
```

## 最佳实践

### Spec 组织

- **分层组织**: 按主题组织 Spec 文件
- **保持精简**: 只包含高价值规则
- **持续迭代**: 随项目发展更新 Spec

### 任务管理

- **单一职责**: 每个任务专注一个功能
- **明确边界**: PRD 清晰定义任务范围
- **及时更新**: 状态变化时更新 status.md

### Journal 记录

- **每日记录**: 工作结束前写日志
- **问题追踪**: 记录遇到的问题和解决方案
- **决策记录**: 重要决策及其原因

## Spec 编写最佳实践

### 1. 保持精简

**原则**: 只记录真正重要的规则，不要面面俱到。

```markdown
# ❌ 不好的做法 - 过于详细

## 变量命名
- 使用 camelCase
- 布尔值以 is/has/should 开头
- 常量使用 UPPER_SNAKE_CASE
- 私有变量以 _ 开头
- 接口以 I 开头
- ...（还有 20 条规则）
```

```markdown
# ✅ 好的做法 - 精简关键

## 核心规范
- 遵循项目现有代码风格
- API 响应使用统一格式 `{ code, data, message }`
- 错误处理使用全局异常过滤器
```

### 2. 提供示例

**原则**: 用代码示例说明规范，比纯文字更有效。

```markdown
# ✅ 好的做法

## API 响应格式

成功响应:
```json
{
  "code": 0,
  "data": { ... },
  "message": "success"
}
```

错误响应:
```json
{
  "code": 1001,
  "data": null,
  "message": "用户不存在"
}
```
```

### 3. 分类组织

**原则**: 按 topic 组织 Spec 文件，便于查找和维护。

```
.spec-first/spec/
├── architecture/       # 架构相关
│   ├── api-design.md
│   └── database.md
├── coding/            # 编码相关
│   ├── typescript.md
│   └── testing.md
└── review/            # 审查相关
    └── checklist.md
```

### 4. 持续迭代

**原则**: Spec 是活的文档，随项目发展而更新。

- 每次代码审查发现问题，考虑是否需要更新 Spec
- 每个迭代结束时，回顾并更新 Spec
- 删除过时的规则，添加新的经验

## 任务管理最佳实践

### 1. 单一职责

**原则**: 每个任务只做一件事。

```markdown
# ❌ 不好的做法

## TASK-001: 用户系统
实现用户注册、登录、个人信息管理、密码重置等功能。
```

```markdown
# ✅ 好的做法

## TASK-001: 用户注册功能
实现用户注册 API，包括：
- 邮箱验证
- 密码加密
- 注册成功通知

## TASK-002: 用户登录功能
（单独的任务）
```

### 2. 明确的验收标准

**原则**: PRD 中明确写明验收标准。

```markdown
# ✅ 好的做法

## 验收标准
- [ ] 注册 API 返回正确格式
- [ ] 邮箱格式验证生效
- [ ] 密码长度 >= 8 位
- [ ] 重复邮箱注册返回错误
- [ ] 单元测试覆盖率 >= 80%
```

### 3. 及时更新状态

**原则**: 任务状态变化时立即更新 status.md。

```markdown
# status.md

## 当前阶段: 实现
更新时间: 2024-01-15 14:30

## 进度
- [x] API 设计
- [x] 数据库模型
- [ ] 控制器实现 (进行中)
- [ ] 测试

## 阻塞
- 无

## 备注
- 预计明天完成
```

## 团队协作最佳实践

### 1. 共享 Spec

**原则**: 团队共享同一套 Spec，保持一致性。

- Spec 文件提交到仓库
- 修改 Spec 需要代码审查
- 新成员入职先阅读 Spec

### 2. 个人 Workspace

**原则**: 个人工作空间不提交，保持独立。

```gitignore
# .gitignore
.spec-first/workspace/
```

### 3. 任务分配

**原则**: 明确任务责任人，避免重复工作。

```markdown
# tasks/README.md

## 活跃任务

| 任务 | 负责人 | 状态 |
|------|--------|------|
| TASK-001 | 张三 | 实现中 |
| TASK-002 | 李四 | 测试中 |
| TASK-003 | 王五 | 待开始 |
```

### 4. 定期同步

**原则**: 定期同步进度和问题。

- 每日站会: 同步各任务进度
- 每周回顾: 更新 Spec 和最佳实践
- 迭代结束: 归档完成的任务

## 平台使用最佳实践

### 1. 选择合适的平台

| 场景 | 推荐平台 |
|------|----------|
| 深度 AI 协作 | Claude Code |
| IDE 集成 | Cursor |
| 快速原型 | OpenCode |
| 企业级开发 | Codex |

### 2. 利用平台特性

**Claude Code**:
- 使用 `/agent:xxx` 手动调用特定 Agent
- 利用 Python hooks 自定义行为

**Cursor**:
- 配置 `.cursorrules` 补充 Spec
- 使用 Cmd+K 快速调用 AI

**Codex**:
- 使用 Skills 扩展功能
- 配置自定义 Agent

### 3. 保持平台无关

**原则**: 核心工作流保持平台无关。

- Spec 和 Task 是平台无关的
- 只在接入层适配不同平台
- 迁移平台时只需重新初始化

## 常见反模式

### 反模式 1: Spec 过于庞大

**问题**: Spec 文件越写越多，AI 难以找到重点。

**解决**:
- 定期清理过时内容
- 合并相似规则
- 只保留高价值规则

### 反模式 2: 任务粒度过大

**问题**: 一个任务包含太多功能，难以管理和验收。

**解决**:
- 拆分为多个小任务
- 每个任务有明确的验收标准
- 使用子任务管理复杂任务

### 反模式 3: 状态不更新

**问题**: status.md 长期不更新，失去跟踪意义。

**解决**:
- 每次工作结束更新状态
- 设置提醒或习惯
- 简化状态格式，降低维护成本

### 反模式 4: 忽视 Journal

**问题**: 不写 Journal，每次会话从零开始。

**解决**:
- 养成每日记录习惯
- 简化 Journal 格式
- 利用 Journal 恢复上下文

### 反模式 5: 平台锁定

**问题**: 过度依赖特定平台特性，难以迁移。

**解决**:
- 核心工作流保持通用
- 平台特性作为增强而非依赖
- 定期评估其他平台

## 检查清单

### Spec 检查

- [ ] Spec 是否精简，只包含关键规则？
- [ ] 是否有代码示例说明规范？
- [ ] Spec 是否按主题分类组织？
- [ ] 是否定期更新和清理？

### 任务检查

- [ ] 每个任务是否单一职责？
- [ ] 是否有明确的验收标准？
- [ ] 状态是否及时更新？
- [ ] 是否有清晰的责任人？

### 团队检查

- [ ] Spec 是否共享并版本控制？
- [ ] 个人 workspace 是否在 .gitignore？
- [ ] 是否有任务分配和跟踪机制？
- [ ] 是否定期同步和回顾？

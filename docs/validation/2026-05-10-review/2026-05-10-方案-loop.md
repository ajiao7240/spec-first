### Finding ID: F-004

- Severity: P1
- Category: Script / CLI / Install
- Title: `engines.node >=20` 只靠 npm warning，CLI/postinstall 缺少早期硬校验
- Evidence:
  - File: `package.json` declares `"node": ">=20.0.0"`.
  - File: `bin/spec-first.js` and `bin/postinstall.js` now call `ensureSupportedNodeVersion()` before loading CLI code or printing install summary.
  - File: `tests/unit/package-install-contracts.test.js` asserts Node 19 rejects, Node 20/24 pass, and bin loads the guard before `src/cli`.
- Official Docs Evidence:
  - DOC-004: npm `engines` is advisory unless `engine-strict` is enabled.
  - DOC-005: parsing `process.version` with optional `v` prefix matches Node docs.
- Industry Practice Evidence: CLI entrypoints should fail early with a clear runtime prerequisite error instead of relying on package-manager warnings.
- Problem: unsupported Node users could install or invoke the CLI and hit a less diagnosable runtime failure.
- Why it matters to spec-first: install/init/doctor is a primary user path; runtime prerequisite failure must be deterministic and clear.
- CE-related cause: not a CE completeness issue; found while reviewing actual post-sync working tree.
- Root cause: package metadata was treated as enough enforcement.
- Suggested fix: add a small shared Node version guard and use it from bin/postinstall.
- Fix scope: `src/cli/node-version.js`, `bin/spec-first.js`, `bin/postinstall.js`, `tests/unit/package-install-contracts.test.js`, `CHANGELOG.md`.
- Risk: low; no generated runtime assets changed.
- Verification: package install contract, typecheck, smoke, release install, full unit.
- Status: fixed.

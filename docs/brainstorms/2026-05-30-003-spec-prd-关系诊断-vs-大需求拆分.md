# 诊断：`003 spec-prd 需求` ↔ `大需求拆分.md 终态方案` 的关系

## 结论先行

这两份文档**在"大需求/多端能力的载体"上路线对立**,当前并存即构成角色契约 §7 反对的**多真相源**。必须先定它们的关系,`spec-prd` 才能安全地扩范围。

核心对立一句话：

> **《大需求拆分.md》主张：不新增主流程 skill，把 PRD Intake / Code-aware / Split / packet / multi-surface 全部做成 `spec-brainstorm` 的输出模式。**
> **`003` 主张：新增独立 `spec-prd` skill 承载增量 PRD，`spec-brainstorm` 退回 0-1。**

同一批能力,两份文档指向**两个不同载体**。这不是细节分歧,是架构分叉。

## 二、一致的部分(没有冲突,可直接共用)

两份文档在这些点**完全同向**,无需取舍:

1. **主链路不变**:Codebase→...→Knowledge,brainstorm/plan/write-tasks 职责边界一致。
2. **plan 是技术真相源**:WHAT/WHY/SCOPE 归需求侧,HOW/FILES/TESTS 归 plan,两文档都坚持。
3. **write-tasks 不能前置**:必须 plan→write-tasks,packet 不能直接派生 task。
4. **ID 化风格**:R/A/F/AE 稳定 ID,003 已用,终态方案沿用并扩 CAP/PS/API/DATA/Contract。
5. **code-aware 不反向扩 scope**:003 的 R13 / 终态方案的 code-alignment 都是"现状只约束"。
6. **knowledge 不写回 requirements**:两文档都守。

→ 这 6 点是**共同地基**,无论怎么定关系都保留。

## 三、对立的部分(必须二选一)

C1/C2 是同一枚硬币两面:**能力要么进 brainstorm(终态方案),要么进 spec-prd(003)**,不能两者都是真相源。

## 四、三种可能的关系,及我的评估

### 关系 A：`003` 取代终态方案(spec-prd 为唯一路线)
- 含义:废弃/归档《大需求拆分.md》的 brainstorm-centric 路线,所有能力(含 packet/multi-surface/拆分)收敛到 `spec-prd`。
- 代价:那份 2042 行方案大量 packet/contract 结构设计**作废或迁移**到 spec-prd;`spec-prd` 范围爆炸成需求编译系统。
- 适合:你确信"独立 PRD 入口"是对的,且愿意让 spec-prd 吃下全部复杂度。

### 关系 B：终态方案取代 `003`(回到 brainstorm-centric)
- 含义:推翻"新增 spec-prd"决策,把增量 PRD 能力做回 `spec-brainstorm` 输出模式。
- 代价:`003` 几轮 owner 决策(单入口/intent model/reviewer/by-reference gate)大部分作废。
- 适合:你认同终态方案"不增主流程 skill"的克制,愿意放弃独立 spec-prd。

### 关系 C：分阶段 —— `003` 是终态的第一阶段(我推荐)
- 含义:**`003`/`spec-prd` v1 = 终态方案里"小需求单文件 + code-aware normalization"那一层的独立载体**;packet/multi-surface/拆分作为终态方案描述的**后续阶段**,载体到那时再定(可能是 spec-prd 长出 packet 能力,也可能回流 brainstorm/spec-requirements)。
- 关键动作:
  1. 把《大需求拆分.md》的 status 改为 **"终态愿景 / 多阶段路线图"**,显式声明"§0 的 brainstorm-centric 载体是早期构想,增量 PRD 这一层的当前落地载体以 `003`/`spec-prd` 为准"。
  2. `003` 加一条**前向兼容声明**:spec-prd v1 只做单文件;packet/multi-surface/拆分是 v2+ 演化,届时复用终态方案的 packet/contract 结构设计。
  3. v1 加**复杂度识别闸**:检测到大需求/跨端信号时输出 split-decision 建议并 handoff,而非硬塞单文件——这正是把终态方案的"复杂度决策树"以最低成本提前埋点。
- 代价:最小。两文档各归其位(`003`=当前落地真相源,终态方案=愿景路线图),消除多真相源;终态方案的 packet 设计不浪费,作为 v2 蓝图保留。
- 为何推荐:符合 spec-first "先证明价值再上规模 / 最小 durable mechanism" 原则;不浪费那份 2042 行设计;给 packet 留干净演化路径;立刻消除多真相源。

## 五、推荐

**选关系 C。** 理由:
- 它把两文档从"对立"变成"分层"——`003` 是终态的第一阶段落地,终态方案降为愿景路线图。多真相源问题立即消除。
- 它不要求你现在就决定 packet/multi-surface 最终长在哪(那是 v2 才需答的问题),只要求承认"v1 先做单文件 + 识别闸"。
- 它和我上一轮建议的"v1 + 复杂度 gate"方案天然契合。

**唯一仍需你拍板的点(C 之下)**:终态方案 §0 说"能力长在 brainstorm",`003` 说"长在 spec-prd"。选 C 后,v2 阶段这批 packet 能力的最终载体仍悬而未决——但**好消息是 C 允许把这个决定推迟到 v2**,v1 不被它阻塞。

如果你接受 C,我会做两件小事(都不碰 source skill):
1. 给《大需求拆分.md》加 status/关系声明 frontmatter,标为多阶段愿景、增量 PRD 层以 003 为准。
2. 给 `003` 加一段"与终态方案的关系 + v1/v2 范围边界 + 复杂度识别闸"。

# Repository Code Baseline Review

## Problem Frame
当前项目已有面向变更集的结构化审查能力，但缺少一份面向整个仓库已提交代码状态的基线体检定义。用户希望对当前分支最新提交中的源码与测试进行一次深度、多视角综合审查，用于建立项目健康基线，而不是立即推动修复或上线决策。

## Requirements

**审查对象与边界**
- R1. 审查必须以当前分支的最新已提交状态为准，不包含未提交的工作区修改。
- R2. 审查范围仅包含 `src/` 与 `tests/` 中的代码和测试资产。
- R3. 审查默认不覆盖 README、技能文档、命令模板、工作流资产、历史分析文档与其他非代码资产，除非它们直接影响 `src/` 或 `tests/` 中代码行为的判断。

**审查视角**
- R4. 审查必须同时覆盖维护者视角、产品负责人视角、开源使用者视角，并在结论中区分不同视角下的问题与价值判断。
- R5. 审查必须是“综合体检”而非单一 bug hunt，既要识别 correctness、回归风险、测试缺口，也要评估可维护性、演进成本、能力设计与可采用性。

**产出形式**
- R6. 最终产出必须采用混合型报告：先给整体判断与主题诊断，再附按优先级排序的问题清单。
- R7. 报告必须为深度版，优先追求结论完整性与分析透彻度，而不是只保留少量高层摘要。
- R8. 主题诊断至少要覆盖代码健康、测试策略、架构与模块边界、对外可理解性/可采用性四个方面；如发现更高杠杆主题，可增加但不应替代这四类。

**发现与排序**
- R9. 问题清单必须给出明确的整改优先级，至少区分 P0、P1、P2，并说明排序依据。
- R10. 每个高优先级问题都应尽量指向具体文件、行为或缺口，避免只给抽象判断。
- R11. 报告必须明确区分“事实观察”“风险判断”“建议动作”，避免把推断写成既定事实。

**用途与限制**
- R12. 这次审查的首要目的，是建立项目健康基线并服务后续决策，而不是直接形成必须立即执行的整改承诺。
- R13. 报告应支持后续把高优先级发现转化为计划或工作项，但本次基线体检本身不要求完成修复方案设计。

## Success Criteria
- 读者可以在一份报告中快速理解该项目当前源码与测试层面的整体健康状况。
- 报告能清楚说明不同视角下最重要的问题是什么、为什么重要、优先级如何排序。
- 后续执行者无需重新界定范围或产出格式，就能基于该文档继续规划审查或整改工作。

## Scope Boundaries
- 不把未提交修改纳入基线。
- 不把文档质量、营销表达或 workflow 资产本身作为主要审查对象。
- 不要求在本阶段输出具体实现方案、重构设计或修复计划。

## Key Decisions
- 审查对象限定为 `src/` 与 `tests/`: 用户希望聚焦代码与测试本身，避免被文档和流程资产稀释注意力。
- 基线以当前分支最新提交为准: 这样能避免未提交噪音污染结论。
- 采用深度版混合报告: 用户要的是可复用的健康基线，而不是轻量问题摘录。
- 采用多视角综合口径: 这样既能评估工程健康，也能覆盖项目能力设计与外部采用门槛。
- 必须包含 P0/P1/P2 排序: 只有诊断没有排序，后续很难转化为行动。

## Dependencies / Assumptions
- 假设当前分支最新提交可以代表用户想建立基线的代码状态。
- 假设 `src/` 与 `tests/` 足以承载本次主要风险和质量判断。
- 假设后续若进入规划或执行阶段，可以再补充文档与 workflow 资产的关联检查。

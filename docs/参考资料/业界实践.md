
## GitHub开源Spec工程Prompt列表汇总

### 1. GitHub Spec Kit (github/spec-kit) ⭐ 68.5k
**最权威的Spec-Driven Development工具包，支持多种AI编码助手**

#### 核心Prompt文件

**A. `/specify` - 需求规格生成Prompt**
```
角色：AI软件工程师，专注于Spec-Driven Development
任务：将自然语言功能描述转换为结构化规格文档

核心流程：
1. 生成分支名称（2-4个词，动作-名词格式）
2. 检查现有分支避免冲突
3. 解析用户描述提取：参与者、动作、数据、约束
4. 生成User Stories和EARS格式的验收标准
5. 识别边缘情况和关键实体
6. 创建规格质量检查清单
7. 验证规格完整性（最多3个[NEEDS CLARIFICATION]标记）

关键规则：
- 聚焦WHAT和WHY，避免HOW（无技术实现细节）
- 为业务利益相关者编写，非开发者
- 所有需求必须可测试、无歧义
- 成功标准必须可衡量且与技术无关
```

**B. `/plan` - 技术规划Prompt**
```
任务：执行实现规划工作流，生成设计文档

阶段划分：
Phase 0 - 大纲与研究：
- 从技术上下文中提取未知项
- 为每个NEEDS CLARIFICATION生成研究任务
- 整合研究结果到research.md

Phase 1 - 设计与契约：
- 从功能规格提取实体 → data-model.md
- 生成API契约（REST/GraphQL模式）
- 更新AI代理上下文文件

关键规则：
- 使用绝对路径
- 门控失败或未解决的澄清项时报错
```

**C. `/tasks` - 任务分解Prompt**
```
任务：生成可执行、按依赖排序的任务列表

任务格式（严格）：
- [ ] [TaskID] [P?] [Story?] 描述及文件路径

阶段结构：
Phase 1: 设置（项目初始化）
Phase 2: 基础（阻塞性前置任务）
Phase 3+: 按优先级排序的用户故事
Final Phase: 完善与跨领域关注点

组织规则：
- 按用户故事组织任务
- 测试是可选的（仅在明确要求时生成）
- 每个任务必须包含具体文件路径
- 标记可并行任务[P]
```

**D. `/implement` - 实现执行Prompt**
```
任务：执行实现计划，处理并执行tasks.md中定义的所有任务

执行前检查：
- 扫描所有检查清单文件
- 计算完成状态（通过/失败）
- 如有未完成项，询问用户是否继续

实现策略：
- 按阶段顺序执行
- 每个用户故事独立可测试
- 优先MVP交付
```

### 2. Kiro Workflow Prompts (wirelessr/kiro-workflow-prompts) ⭐ 38
**Kiro IDE风格的spec-driven development prompt集合**

#### 核心Prompt文件

**A. `createSpec.prompt.md` - 需求生成**
```
角色：高级AI软件工程师，专精Spec-Driven Development
使命：将高层功能想法转换为符合EARS标准的正式需求文档

核心工作流（状态化循环）：
1. 接收用户高层功能请求
2. 立即生成requirements.md初稿
3. 暂停并显式等待用户批准
4. 如有修改请求，更新文档并返回步骤3

行为规则：
- 在.kiro/specs/[feature-name]/目录创建requirements.md
- 读取.kiro/steering/目录作为项目基础上下文
- 每个需求必须以User Story格式开头：
  "As a [role], I want [feature], so that [benefit]"
- 验收标准必须使用EARS语法（WHEN/THEN/IF/WHILE/WHERE）
- 不得事先询问多个澄清问题，主动思考各种场景

批准门控：
"Do the requirements look good? If so, we can move on to the next phase."
```

**B. `design.prompt.md` - 技术设计**
```
角色：高级AI软件工程师，担任技术架构师
使命：将批准的需求转换为全面的技术设计蓝图design.md

输入来源：
- 批准的requirements.md
- .kiro/steering/目录（技术栈和约束）
- 现有代码库的静态分析

design.md必须包含：
1. Architectural Overview - 高层解决方案描述
2. Data Flow Diagram - Mermaid.js数据流图
3. Component & Interface Definitions - TypeScript接口定义
4. API Endpoint Definitions - 完整端点契约
5. Database Schema Changes - SQL DDL或ORM模型
6. Security Considerations - 风险分析和缓解策略
7. Test Strategy - 单元/集成/E2E测试规划

批准门控：
"Does the technical design look good? If so, we can proceed to implementation planning."
```

**C. `executeTask.prompt.md` - 任务执行**
```
核心原则：
- 协作优先：未经用户明确批准不得实施
- KISS原则：避免过度工程
- 代码使用英文：确保可维护性
- 实用主义：明显正确的代码优于巧妙技巧
```

**D. `commit.prompt.md` - 提交助手**
```
任务：分析变更并创建专业的Git提交消息

功能：
- 分类文件类型和评估变更模式
- 过滤内部开发产物
- 确定适当的提交策略
- 生成conventional commit消息
- 识别破坏性变更和依赖
```

**E. `prReview.prompt.md` - PR审查**
```
角色：高级开发者的全面PR审查

能力：
- 使用GitHub CLI获取PR元数据
- 执行全面的变更分析
- 识别风险、安全问题和架构关注点
- 提供具体可操作反馈
- 评估测试覆盖率和文档质量
```

### 3. Claude-Spec-Starter (danielrosehill/Claude-Spec-Starter) ⭐ 4
**Claude Code专用的Spec-Driven Development模板**

#### 核心Prompt文件

**A. `build-repo.md` - 仓库初始化**
```
任务：根据starter规范构建仓库

执行顺序：
1. 读取prompt.md（用户提供的长篇项目描述）
2. 创建spec/starter.md - 初始开发规格
   （测试标准：是否足够清晰，能为AI代理提供足够开发方向？）
3. 创建context/index.md - 重写上下文数据
   格式："用户希望使用Python进行此项目"
4. 更新CLAUDE.md - 用项目详情摘要覆盖占位符
5. 将prompt.md归档到archive文件夹
```

**B. `refine-spec.md` - 规格细化**
```
任务：更新现有规格以纳入新信息
- 处理迭代添加的语音会话内容
- 保持与原始愿景的一致性
```

**C. `add-context.md` - 上下文管理**
```
任务：添加上下文信息
- 处理语音驱动的偏好更新
- 将上下文组织到适当文件
```

### 4. OpenSpec (Fission-AI/OpenSpec) ⭐ 23.2k
**轻量级Spec-Driven Development工具，支持20+ AI助手**

#### 核心特点
- **Agree before you build** - 编码前人与AI对齐规格
- **Stay organized** - 每个变更有自己的文件夹（proposal、specs、design、tasks）
- **Work fluidly** - 随时更新任何产物，无严格阶段门控
- **Use your tools** - 通过slash命令与20+ AI助手协作

#### 推荐模型
- 规划和实现：Opus 4.5、GPT 5.2（高推理能力模型）

#### 工作流命令
```
/spec - 创建/更新规格
/plan - 生成技术计划  
/tasks - 分解为可执行任务
/implement - 执行实现
```

## 推荐用于构建公司Spec研发流程的Prompt组合

### 阶段一：需求定义（Specify）
| 来源 | 推荐元素 |
|------|---------|
| GitHub Spec Kit | 分支命名规范、规格质量检查清单、NEEDS CLARIFICATION处理机制 |
| Kiro | EARS语法验收标准、User Story格式、批准门控机制 |
| Claude-Starter | 上下文分离模式（spec vs context） |

### 阶段二：技术规划（Plan）
| 来源 | 推荐元素 |
|------|---------|
| GitHub Spec Kit | Phase 0研究阶段、data-model.md、contracts/目录 |
| Kiro | Mermaid.js数据流图、安全考虑、测试策略 |
| OpenSpec | 灵活迭代方式（无严格阶段门控） |

### 阶段三：任务分解（Tasks）
| 来源 | 推荐元素 |
|------|---------|
| GitHub Spec Kit | 严格任务格式 `- [ ] [TaskID] [P?] [Story?]`、按用户故事组织 |
| Kiro | 独立可测试标准、MVP优先策略 |

### 阶段四：实现（Implement）
| 来源 | 推荐元素 |
|------|---------|
| GitHub Spec Kit | 检查清单预检查、分阶段执行 |
| Kiro | 协作优先原则、Linus Torvalds开发准则 |

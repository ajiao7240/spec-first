## 场景概述

### 业务背景

某电商项目已运行一年，订单支付模块（微信支付、支付宝支付）已上线稳定运行。项目此前**从未使用过 Spec-First**，没有任何 `stage-state.json`、`spec.md`、`design.md`、`traceability-matrix` 等 Spec-First 产物。

产品经理提交需求：**「优化订单支付流程 — 增加支付超时自动取消 + 支付结果页优化」**。

团队决定借此需求**首次引入 Spec-First**，用它管理对已有功能的修改。

### 功能范围

| 维度 | 值 | 说明 |
|------|-----|------|
| 涉及模块 | 2 个 | 支付服务（Payment）、订单服务（Order） |
| AC 数量 | ~4 个 | 超时自动取消、结果页优化等 |
| API 修改 | 1 个 | 修改 `POST /api/pay/create`（增加超时参数） |
| 数据模型 | 无变更 | 复用已有 `timeout_at` 字段（已预留但未启用） |
| 跨团队 | 0 | 前后端同一团队 |
| 涉及端 | H5 + Backend | 前端支付结果页 + 后端超时逻辑 |

### 与场景 1.1 的核心区别

| 对比维度 | 场景 1.1（新增功能） | 场景 1.2（修改功能） |
|---------|-------------------|-------------------|
| Mode | I（Iteration） | **N（New）** |
| 为什么选此 Mode | 项目已有 Spec-First 历史产物 | **项目从未使用 Spec-First，无历史产物** |
| Size | M | **S** |
| Spec-First 状态 | 已初始化，有历史 Feature | **首次引入，零产物** |
| Specify 阶段特殊处理 | Impact Analysis（基于历史 spec.md） | **补建现状基线**（在 spec.md 中描述当前行为） |
| 历史产物定位 | 需要（Mode I 追加活动） | **不需要**（无 Spec-First 产物可定位） |

### 为什么是 Mode N 而不是 Mode I

**关键认知**：Mode I（Iteration）要求项目中已存在 Spec-First 历史产物（`spec.md`、`traceability-matrix` 等），因为 Mode I 的核心增量活动（历史产物定位、Impact Analysis、回归验证）都依赖这些产物作为基线。

本场景中：
- 已有代码工程和已有功能 -- 但这是**业务层面**的"已有"
- 没有任何 Spec-First 产物 -- 这是**工具链层面**的"全新"
- Mode 的判定依据是**工具链层面**是否有历史产物，而非业务层面是否有已有功能

因此，**首次引入 Spec-First 时，无论是新增功能还是修改功能，一律使用 Mode N**。

### Mode/Size 判定

```
Mode = N（New）— 首次使用 Spec-First，无历史产物

Size 判定（5 维度取最高）：
  modules      = 2  → S（1-2 模块）
  ac_count     = 4  → S（1-5 AC）
  api_changes  = 1  → S（0-1 API）
  data_model   = 0  → S（无变更）
  cross_team   = 0  → S（无跨团队）
  ─────────────────
  最终 Size    = S
```

### FEAT 缩写

- **FEAT 缩写**: `PAY`
- **Feature ID**: `001-payment-timeout`
- **全名**: 支付超时自动取消与结果页优化

### Size S 裁剪效果总览

| 产出物/活动 | Size S（本场景） | Size M | Size L |
|------------|----------------|--------|--------|
| `spec.md` | 简版（核心 FR + AC） | 标准版 | 完整版（含 NFR 量化） |
| `research.md` | **可省略** | 标准版 | 完整版（含 Trade-off） |
| `contracts/*.yaml` | **可内嵌 design.md** | 独立文件 | 独立文件 + Mock Server |
| `data-model.md` | **无变更可省略** | 字段级变更必须 | 完整 ERD + 状态机 |
| AC 级覆盖率校验 | **不强制** | 强制 >=90% | 强制 >=90% |
| Architect 参与 | **可选** | 推荐 | 必须 |
| 任务粒度 | **1-3 个 TASK** | 5-15 个 TASK | 15+ 个 TASK |
| 并行策略 | **串行开发** | 分支并行 | Git Worktree 隔离 |
| 安全扫描 | **OWASP + SCA** | + SAST | + DAST + 渗透测试 |
| 部署策略 | **Rolling Update** | Blue-Green | Canary |
| 观察窗口 | **30 分钟** | 2 小时 | 24 小时 |

## Stage 01: Specify（需求规格）

### 阶段目标

编写需求规格 `spec.md`，分配 FR/NFR ID。**本场景的核心特殊性**：虽然是修改已有功能，但没有历史 `spec.md`，需要在 spec.md 中先"补建现状基线"（描述当前行为），再描述期望变更。

### 场景特殊处理：补建现状基线

由于首次引入 Spec-First，不存在已有功能的 `spec.md`。直接写"变更需求"会缺少上下文。建议在 `spec.md` 中增加 **「现状描述」** 章节：

```
spec.md 结构（修改已有功能 + 首次引入 Spec-First）：

  1. Feature 概述
  2. 现状描述（补建基线）    ← 本场景特有
     - 当前支付流程行为
     - 当前已知问题
  3. 变更需求（FR）          ← 描述期望变更
  4. 非功能需求（NFR）
  5. 验收标准（AC）
  6. Sign-off
```

### 具体操作步骤

**步骤 1：生成 FR/NFR ID**

```bash
# 生成功能需求 ID
spec-id next FR PAY
# → FR-PAY-001

spec-id next FR PAY
# → FR-PAY-002

# 生成非功能需求 ID（Size S 下 NFR 可选，但支付场景建议保留性能要求）
spec-id next NFR PAY --dim PERF
# → NFR-PERF-001
```

**步骤 2：编写 spec.md（简版 -- Size S 裁剪）**

在 `specs/001-payment-timeout/spec.md` 中编写：

```markdown
# 需求规格 — 001-payment-timeout (PAY)

> Mode: N | Size: S | Platforms: H5, Backend
> 状态: Draft

## 1. Feature 概述

优化订单支付流程，增加支付超时自动取消功能，并优化支付结果页的用户体验。

## 2. 现状描述（补建基线）

> 本章节描述当前系统行为，作为变更的参照基线。
> 由于项目首次引入 Spec-First，此前无 spec.md 记录，故在此补建。

### 2.1 当前支付流程

1. 用户提交订单后，调用 `POST /api/pay/create` 发起支付
2. 后端创建支付单，返回支付凭证（微信/支付宝）
3. 用户在第三方支付页面完成支付
4. 第三方回调 `POST /api/pay/callback` 通知支付结果
5. 前端轮询支付状态，展示"支付成功/失败"

### 2.2 当前已知问题

- **无超时机制**：用户发起支付后若未完成，订单永久处于"待支付"状态，库存被锁定
- **结果页体验差**：支付完成后仅显示文字"支付成功"，无订单摘要、无后续引导

## 3. 功能需求（FR）

### FR-PAY-001: 支付超时自动取消

用户发起支付后，若 30 分钟内未完成支付，系统自动取消订单并释放库存。

**验收标准（AC）**：
- AC-1: 支付创建时记录超时时间（当前时间 + 30 分钟）
- AC-2: 定时任务每分钟扫描超时订单，执行自动取消
- AC-3: 取消后释放库存，订单状态变更为"已取消（超时）"

### FR-PAY-002: 支付结果页优化

支付完成后，结果页展示订单摘要信息和后续操作引导。

**验收标准（AC）**：
- AC-4: 支付成功页展示订单编号、支付金额、支付方式、支付时间
- AC-5: 提供"查看订单详情"和"继续购物"两个操作入口

## 4. 非功能需求（NFR）

### NFR-PERF-001: 超时扫描性能

定时任务单次扫描耗时 < 5 秒（订单量 < 10 万条场景下）。

## 5. Sign-off

[SIGNED:Leo:2026-02-08T11:00:00Z]
```

> **Size S 裁剪效果**：简版 spec.md，仅包含核心 FR + AC，NFR 仅保留关键项。对比 Size M/L，省略了详细的用户故事、流程图、边界条件分析等。

# → ✓ FR-PAY-002 — Valid

**步骤 4：提交 spec.md 到 Git**

## 场景特殊考量

### 1. 首次引入 Spec-First 修改已有功能的注意事项

**核心认知：Mode 的选择依据是"工具链层面"而非"业务层面"**

| 判断维度 | 本场景情况 | 结论 |
|---------|-----------|------|
| 业务层面是否有已有功能？ | 是（支付模块已上线） | 不影响 Mode 选择 |
| 工具链层面是否有 Spec-First 历史产物？ | **否**（从未使用过） | **决定使用 Mode N** |
| 是否存在 stage-state.json？ | 否 | Mode I 无法执行历史产物定位 |
| 是否存在 spec.md？ | 否 | Mode I 无法执行 Impact Analysis |
| 是否存在 traceability-matrix？ | 否 | Mode I 无法执行回归验证 |

**补建现状基线的最佳实践**

首次引入 Spec-First 修改已有功能时，`spec.md` 中的「现状描述」章节是关键差异点：

```
推荐做法：
  ✅ 在 spec.md 中增加「现状描述」章节，简要描述当前系统行为
  ✅ 现状描述仅作为参照基线，不分配 FR ID
  ✅ 变更需求（FR）基于现状描述来定义"从 A 变为 B"

不推荐做法：
  ❌ 为已有功能补写完整的历史 spec.md（工作量大，ROI 低）
  ❌ 直接写变更需求而不描述现状（缺少上下文，评审困难）
  ❌ 使用 Mode I（无历史产物，Mode I 的增量活动无法执行）
```

### 2. Size S 的轻量化优势

本场景完整走完 8 个阶段，产出物和工作量统计：

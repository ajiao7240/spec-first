# Spec-First 全流程提效分析

> **版本**: v1.0 | **日期**: 2026-02-08 | **作者**: Leo (况雨平)
> **基准场景**: Mode N × Size M（最常见的标准场景）
> **对比基线**: 传统研发流程（无规范管理工具，人工治理）

---

## 1. 分析说明

本文以 **Size M（标准规模）** Feature 为基准，完整梳理 Spec-First 8 阶段流程中的每一项活动，明确标注：

- **执行主体**：AI 自动（CLI/Hook/CI）、人工执行、人机协同
- **传统耗时**：无 Spec-First 时该活动的人工耗时
- **Spec-First 耗时**：引入 Spec-First 后的实际耗时
- **提效幅度**：单项活动的效率提升

> **约定**：「AI 自动」指 CLI 工具、Git Hook、CI Pipeline 自动完成，人工仅需触发命令或查看结果。

### Size M 基准参数

| 参数 | 值 | 说明 |
|------|-----|------|
| 模块数 | 3-5 | 标准复杂度 |
| AC 数 | 6-15 | 中等需求量 |
| API 数 | 2-5 | 标准接口量 |
| 涉及端 | 2 | 如 H5 + Backend |
| 开发团队 | 3-5 人 | 1 Tech Lead + 2 Dev + 1 QA + 1 PM |
| 开发周期 | ~2 周 | 含设计、编码、测试 |
| 总人时预算 | ~200 人时 | 全角色合计 |

## 3. 全流程汇总

### 3.1 各阶段耗时对比

| 阶段 | 传统耗时 | Spec-First 耗时 | 节省 | 提效比 |
|------|---------|----------------|------|--------|
| 前置准备（一次性） | 7h | 2h | 5h | 71% |
| 00 Init | 1.5h | 0.25h | 1.25h | 83% |
| 01 Specify | 17h | 14h | 3h | 18% |
| 02 Design | 20h | 14.5h | 5.5h | 28% |
| 03 Plan | 8h | 4.5h | 3.5h | 44% |
| 04 Implement | 78h | 69h | 9h | 12% |
| 05 Verify | 25h | 16.5h | 8.5h | 34% |
| 06 Wrap-up | 12.5h | 1.5h | 11h | 88% |
| 07 Release | 6h | 4.5h | 1.5h | 25% |
| **全流程合计** | **175h** | **126.75h** | **48.25h** | **28%** |

> 注：前置准备为一次性投入，若排除前置准备，Feature 级提效为 **(168h → 124.75h) = 26%**。

### 3.2 按执行主体分类统计

#### 全部 78 项活动分类

| 执行主体 | 活动数 | 占比 | 典型活动 |
|---------|--------|------|---------|
| **AI 自动** | 35 项 | 45% | ID 生成/校验、覆盖率计算、Gate 评估、SCA 校验、Hook 校验、矩阵检查、报告生成 |
| **人工执行** | 30 项 | 38% | PRD 分析、spec/design/tasks 编写、编码、测试用例编写、Code Review、UAT、签核、部署监控 |
| **人机协同** | 13 项 | 17% | Git 提交（人工提交 + Hook 自动校验）、FEAT 注册、CI/CD 部署 |
| **合计** | **78 项** | **100%** | |

#### 按耗时分类（Spec-First 模式下）

| 执行主体 | Spec-First 耗时 | 占比 | 说明 |
|---------|----------------|------|------|
| **人工执行** | ~113h | 89% | 创造性工作：需求分析、设计、编码、测试、Review、签核 |
| **人机协同** | ~12h | 10% | 人工触发 + AI 辅助校验：Git 提交、CI/CD、FEAT 注册 |
| **AI 自动** | ~1.75h | 1% | 纯 CLI/Hook 自动执行：ID 生成、覆盖率、Gate、SCA、报告 |
| **合计** | **~126.75h** | **100%** | |

> **关键洞察**：AI 自动化仅占总耗时 1%，但替代了传统流程中 ~48h 的人工治理工作。这说明 Spec-First 的价值不在于「AI 做了多少工作」，而在于「AI 消除了多少低价值重复劳动」。

### 3.3 治理开销专项对比

将全流程活动分为「创造性工作」和「治理开销」两类：

| 类别 | 定义 | 传统耗时 | SF 耗时 | 提效 |
|------|------|---------|--------|------|
| **创造性工作** | 需求分析、设计、编码、测试编写、Code Review、UAT、部署监控 | ~127h | ~122h | 4% |
| **治理开销** | ID 管理、覆盖率统计、一致性校验、Gate 评审会议、报告编写、矩阵维护、签核流程 | ~48h | ~4.75h | **90%** |
| **合计** | | **~175h** | **~126.75h** | **28%** |

> **核心结论**：Spec-First 对创造性工作几乎无影响（仅因结构化模板略有提效），**真正的提效来自治理开销的 90% 压缩**。

## 6. 超越时间节省的隐性价值

Spec-First 的价值不仅体现在「节省了多少小时」，更体现在以下难以量化但影响深远的维度：

### 6.1 质量左移（Shift-Left）

| 传统流程 | Spec-First 流程 | 质量影响 |
|---------|----------------|---------|
| 需求遗漏在测试阶段才发现 | Gate 01 强制校验 FR/NFR 完整性 | 缺陷发现成本降低 10-100x |
| 设计与需求脱节在编码时暴露 | sca_design 自动校验 design↔spec 一致性 | 返工率降低 |
| 任务遗漏在集成时才发现 | C1 覆盖率 100% 强制要求 | 功能遗漏归零 |
| 测试用例覆盖不全 | C3 测试覆盖率 Gate 拦截 | 测试盲区消除 |

> **量化估算**：传统流程中，需求/设计阶段的缺陷如果在测试阶段才发现，修复成本是原阶段的 10-100 倍（IBM Systems Sciences Institute 数据）。Spec-First 通过 4 道关键 Gate 将缺陷拦截在产生阶段，预计可减少 **30-50% 的返工成本**。

### 6.2 随时可审计（Always Audit-Ready）

| 维度 | 传统流程 | Spec-First 流程 |
|------|---------|----------------|
| 追踪矩阵 | 项目结束时手动补写（或不写） | 全程自动维护，`spec-matrix show` 实时查看 |
| 覆盖率数据 | 无法实时获取 | `spec-metrics coverage` 随时查询 9 项指标 |
| 变更历史 | 散落在邮件/聊天记录中 | RFC 变更流程 + Gate 历史 JSONL 完整记录 |
| 签核记录 | 邮件/钉钉截图 | `[SIGNED:name:timestamp]` 结构化记录在文档中 |

> **合规价值**：对于金融、医疗等强监管行业，Spec-First 的全链路追溯能力可直接满足 ISO 27001、SOC 2 等审计要求，避免项目结束后「补文档」的痛苦。

### 6.3 知识沉淀与团队传承

| 传统痛点 | Spec-First 解法 | 长期收益 |
|---------|----------------|---------|
| 需求只在 PM 脑中，离职即丢失 | 结构化 spec.md 永久留存 | 新人可独立理解历史需求 |
| 设计决策无记录，后人不知「为什么这样做」 | design.md + ADR 记录决策理由 | 避免重复踩坑 |
| 任务拆解经验无法复用 | tasks.md 积累拆解模式 | 估算准确度逐步提升 |
| 追踪关系散落各处 | 追踪矩阵统一管理 | 影响分析秒级完成 |

> **团队价值**：新成员通过阅读 `specs/` 目录下的历史 Feature 产物，可在 **1-2 天内**理解项目全貌，而传统流程中这一过程通常需要 **1-2 周**的口头传授。

### 6.4 风险前置与持续可见

| 风险类型 | 传统发现时机 | Spec-First 发现时机 | 提前量 |
|---------|------------|-------------------|--------|
| 需求遗漏 | 测试阶段 | Specify 内联校验（ID 完整性） | 提前 3-4 个阶段 |
| 设计与需求不一致 | 编码阶段 | ★ Gate 1: Design Ready（sca_design） | 提前 2-3 个阶段 |
| 任务覆盖不全 | 集成阶段 | ★ Gate 2: Code Ready（C1 覆盖率） | 提前 1-2 个阶段 |
| 测试覆盖不全 | 上线后 | ★ Gate 3: Release Ready（C3 覆盖率） | 提前 1 个阶段 |
| 孤儿代码（无需求对应） | 从不发现 | C9 孤儿率检测 | 全新能力 |

> **风险价值**：每道 Gate 都是一个「质量检查站」，将风险从「事后救火」转变为「事前预防」。4 道关键 Gate + CLI 内联校验构成的防线，使得问题在产生阶段即被拦截，而非在下游阶段以更高成本暴露。

## 7. 不同 Size 的提效差异

Size 决定了产出物深度和 Gate 严格度，直接影响治理开销占比和提效幅度。

### 7.1 Size S/M/L 对比总览

| 维度 | Size S | Size M（基准） | Size L |
|------|--------|---------------|--------|
| 模块数 | 1-2 | 3-5 | 6+ |
| AC 数 | 1-5 | 6-15 | 16+ |
| API 数 | 0-1 | 2-5 | 6+ |
| 开发周期 | 3-5 天 | ~2 周 | 4+ 周 |
| 总人时预算 | ~40h | ~200h | ~500h+ |
| 产出物深度 | 轻量（内联） | 标准（独立文件） | 完整（独立文件 + 评审会议） |
| Gate 条件数 | 较少 | 标准 | 最多（含额外安全/性能门禁） |

### 7.2 各 Size 提效幅度估算

| 阶段 | Size S 提效 | Size M 提效 | Size L 提效 | 规律 |
|------|-----------|-----------|-----------|------|
| 00 Init | 80% | 83% | 85% | Size 越大，自动判定越有价值 |
| 01 Specify | 15% | 18% | 20% | 需求编写本身不可替代，ID 管理量随 Size 增长 |
| 02 Design | 20% | 28% | 35% | Size L 的 API 契约和 SCA 校验量更大 |
| 03 Plan | 35% | 44% | 50% | Size L 任务数更多，覆盖率校验价值更高 |
| 04 Implement | 8% | 12% | 15% | 编码不可替代，但 Hook 校验次数随 commit 数增长 |
| 05 Verify | 25% | 34% | 40% | Size L 测试用例更多，覆盖率校验更复杂 |
| 06 Wrap-up | 85% | 88% | 92% | Size L 的矩阵终审和报告生成工作量更大 |
| 07 Release | 20% | 25% | 28% | 发布流程差异不大 |
| **全流程** | **~20%** | **~28%** | **~35%** | **Size 越大，治理开销占比越高，提效越显著** |

### 7.3 规律总结

1. **治理开销与 Size 正相关**：Size S 治理开销约占总耗时 15%，Size M 约 27%，Size L 约 35%。Size 越大，ID 数量、覆盖率校验复杂度、矩阵规模、Gate 条件数均成倍增长。

2. **AI 自动化收益与 Size 正相关**：Size 越大，被自动化替代的治理工作量越大，绝对节省时间越多。Size L 项目中，仅追踪矩阵终审一项就可节省 8h+（传统流程需人工逐行核对数百条追踪链）。

3. **创造性工作占比与 Size 负相关**：Size S 项目中编码占比高达 70%+，治理开销占比低，因此整体提效幅度有限（~20%）。Size L 项目中编码占比降至 55%，治理开销占比升高，提效空间更大（~35%）。

4. **边际效益递增**：随着团队使用 Spec-First 管理的 Feature 数量增加，模板复用、规范熟练度、AI 辅助起草效率均会提升，实际提效幅度可能高于首次使用的估算值。

## 8. 总结与结论

### 8.1 核心数据一览

| 指标 | 数值 | 说明 |
|------|------|------|
| 全流程活动总数 | 78 项 | 覆盖 8 阶段 + 前置准备 |
| AI 自动化活动 | 35 项（45%） | CLI/Hook/CI 自动执行 |
| 人工执行活动 | 30 项（38%） | 创造性工作，不可替代 |
| 人机协同活动 | 13 项（17%） | 人工触发 + AI 校验 |
| 传统流程总耗时 | 175h | Size M 基准 |
| Spec-First 总耗时 | 126.75h | Size M 基准 |
| 绝对节省 | 48.25h | 约 6 个工作日 |
| 整体提效比 | **28%** | Size M 基准 |
| 治理开销压缩 | **90%** | 48h → 4.75h |
| 创造性工作影响 | 4% | 127h → 122h，几乎不变 |

### 8.2 三个核心结论

**结论一：Spec-First 的本质是「治理自动化」，而非「开发自动化」**

Spec-First 不试图替代人类的创造性工作（需求分析、架构设计、编码实现、测试设计），而是将围绕这些创造性工作的治理开销（ID 管理、覆盖率统计、一致性校验、Gate 评审、报告生成）从人工操作转变为工具自动执行。治理开销从 48h 压缩到 4.75h（**90% 压缩**），而创造性工作仅因结构化模板略有提效（4%）。

**结论二：提效幅度与项目规模正相关**

- Size S（小型）：整体提效 ~20%，治理开销占比低，自动化收益有限
- Size M（标准）：整体提效 ~28%，治理与创造性工作比例均衡，性价比最优
- Size L（大型）：整体提效 ~35%，治理开销占比高，自动化收益最大

**结论三：真正的价值在时间节省之外**

28% 的时间节省是可量化的直接收益，但更大的价值在于：
- **质量左移**：4 道关键 Gate 将缺陷拦截在产生阶段，预计减少 30-50% 返工成本
- **随时可审计**：全链路追溯能力满足合规审计要求，消除「项目结束补文档」
- **知识沉淀**：结构化产物使新人上手周期从 1-2 周缩短到 1-2 天
- **风险前置**：问题在产生阶段即被发现，而非在下游以更高成本暴露

### 8.3 一句话总结

> **Spec-First 用 1% 的 AI 自动化时间，消除了 90% 的治理开销，释放团队 28% 的产能回归创造性工作，同时通过 4 道关键 Gate 将缺陷拦截成本降低一个数量级。**

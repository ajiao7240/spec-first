# Spec-First 安装与更新机制方案

> **版本**: v1.2 | **日期**: 2026-02-23 | **状态**: 已确认（多agent审查 + SessionStart 全局化修订）

## 1. 现状分析

### 当前安装流程（4步）

```bash
cd /Users/kuang/xiaobu/spec-first
pnpm install && pnpm build
pnpm link --global
spec-first setup --global
```

### 当前更新流程（2步）

```bash
cd /Users/kuang/xiaobu/spec-first
pnpm build
# 若新增/删除技能：spec-first setup --global
```

### 问题

| 痛点 | 说明 |
|------|------|
| 依赖源码目录 | 必须 clone 仓库，无法脱离源码使用 |
| 更新需手动 build | 每次升级需进入源码目录执行构建 |
| 技能注册与安装分离 | `setup --global` 是独立步骤，容易遗漏 |
| 无版本升级感知 | 用户不知道何时有新版本可用 |
| Viewer 仅源码项目自动打开 | `session-start.js` 引用 `scripts/stage-viewer/bootstrap.js`，仅在源码仓库存在；目标项目打开 Claude Code 不会自动启动 Viewer |

## 2. 目标流程（对标 OpenSpec）

### 安装（1行）

```bash
npm install -g spec-first@latest
# 自动完成：CLI 全局可用 + Skill 注册 + MCP 配置补齐
```

### 更新（2行）

```bash
npm install -g spec-first@latest
spec-first update
# 刷新：Skill 命令 + MCP 配置 + Git hooks（当前项目）
```

### 项目初始化（不变）

```bash
cd your-project
spec-first init --feat AUTH --mode N --size M --platforms backend
```

## 4. 实现方案

### 4.1 npm 发布准备

**package.json 调整：**

```jsonc
{
  "name": "spec-first",          // 包名（npmjs.com 未被占用）
  "version": "0.1.0",
  "scripts": {
    "build": "tsup",
    "prepublishOnly": "npm run build",   // 发布前强制构建
    "postinstall": "node dist/postinstall.js"  // 全局安装后自动注册
  }
}
```

**files 字段确认（当前已覆盖）：**

```jsonc
"files": ["dist", "skills", "templates", "README.md", "scripts/stage-viewer", "scripts/codex"]
```

> 无需改动，`npm pack --dry-run` 可验证发布内容。

### 4.2 新增 `update` 子命令

**职责**：一条命令完成升级后的所有刷新动作。

```
spec-first update [--dry-run] [--skip-mcp] [--skip-hooks]
```

**执行逻辑（顺序）：**

1. 检查版本升级（update-notifier，有新版本时输出提示）
2. 输出当前版本号
3. 执行 `ensureSkillCommands(cwd, { global: true })` — 刷新 Skill 命令到 Claude Code + Codex
4. 执行 `ensureHostBootstrap()` — 补齐 MCP 配置（可 `--skip-mcp` 跳过）
5. 若当前目录有 `.git`，执行 `installHooks(cwd)` — 刷新 Git hooks（可 `--skip-hooks` 跳过）
6. 执行 `registerAIHooks(cwd)` — 刷新 AI Runtime Hooks（PreToolUse/PostToolUse/Stop）到 `.claude/settings.json`
7. 执行 `registerSessionHooks()` — 注册 SessionStart/SessionEnd 到全局 `~/.claude/settings.json`，实现任意项目打开 Claude Code 自动启动 Viewer
8. 输出变更摘要（updated / skipped / warn）

**`--dry-run` 模式**：仅输出将发生的变更，不写文件。

> ⚠️ 审查发现：5 个核心函数（ensureSkillCommands、ensureHostBootstrap、installHooks、registerAIHooks、registerSessionHooks）当前均不支持 dry-run。需为每个函数添加 `dryRun?: boolean` 参数，控制文件写入行为。ensureHostBootstrap 已支持 `SPEC_FIRST_SKIP_BOOTSTRAP=1` 环境变量跳过。

**新增文件**：`src/cli/commands/update.ts`（~40行）

**路由注册**：`src/cli/index.ts` 增加一行

```typescript
registerCommand('update', '升级后刷新 Skill/MCP/Hooks', handleUpdate);
```

### 4.3 postinstall 钩子

**新增文件**：`src/postinstall.ts`

**逻辑**：

```
if 全局安装（npm_config_global === 'true'）:
    执行 ensureSkillCommands(cwd, { global: true })
    执行 ensureHostBootstrap()
    执行 registerSessionHooks()  — 注册全局 SessionStart Hook
    输出："✓ Skill 命令已注册，MCP 配置已补齐，Viewer 自动启动已配置"
else:
    静默跳过（作为项目依赖安装时不干扰）
```

**全局安装检测方式**：

| 包管理器 | 检测条件 |
|---------|---------|
| npm | `process.env.npm_config_global === 'true'` |
| pnpm | `process.env.PNPM_HOME` 存在且安装路径在其下 |
| yarn | 不支持全局 postinstall，需手动 `spec-first update` |

> 兜底策略：检测失败时静默跳过，不阻塞安装。用户可手动执行 `spec-first update`。

### 4.4 SessionStart 全局 Hook 注册

**问题**：当前 `.claude/hooks/hooks.json` 是项目级配置，`session-start.js` 引用 `scripts/stage-viewer/bootstrap.js`（仅源码仓库存在）。导致目标项目打开 Claude Code 时 Viewer 不会自动启动。

**方案**：新增 `registerSessionHooks()` 函数，将 SessionStart Hook 注册到全局 `~/.claude/settings.json`，使用 CLI 命令替代本地脚本路径。

**注册内容**：

```jsonc
// ~/.claude/settings.json（全局）
{
  "hooks": {
    "SessionStart": [
      { "matcher": "*", "command": "spec-first viewer open --print-url 2>/dev/null || true" }
    ]
  }
}
```

**设计要点**：

| 项 | 说明 |
|----|------|
| 注册目标 | `~/.claude/settings.json`（全局，非项目级） |
| 命令 | `spec-first viewer open --print-url`（通过全局 CLI，不依赖源码路径） |
| 容错 | `2>/dev/null \|\| true` 确保非 spec-first 项目静默跳过 |
| 幂等 | 过滤已有 `spec-first viewer` 条目后追加，重复执行无副作用 |
| Codex 限制 | Codex 无 SessionStart Hook 机制，仅 Claude Code 生效 |

**新增文件**：`src/core/tool-integration/session-hook.ts`（~30行）

**实现逻辑**：复用 `registerAIHooks` 的 settings.json 读写模式，但写入 `~/.claude/settings.json`（全局）而非项目级。

### 4.5 开发者模式（保留）

源码贡献者继续使用 `pnpm link --global` 流程，文档中标注为"开发模式"：

```bash
# 开发模式（仅源码贡献者）
git clone <repo> && cd spec-first
pnpm install && pnpm build && pnpm link --global
spec-first update   # 替代原来的 setup --global
```

`setup --global` 保留但标记为 deprecated，内部转发到 `update`。

## 7. 风险评估

| 风险 | 影响 | 缓解措施 |
|------|------|---------|
| postinstall 在 CI 环境触发 MCP 写入 | 低 | `SPEC_FIRST_SKIP_BOOTSTRAP=1` 已有跳过机制 |
| pnpm 全局安装不触发 postinstall | 中 | 文档明确提示 `spec-first update`；postinstall 仅作锦上添花 |
| npm 包名被抢注 | 低 | 尽早发布占位版本（0.1.0） |
| Skill 路径从 symlink 变为 npm 包内路径 | 中 | `resolveSkillsRoot()` 已兼容 `dist/` 和 `../` 两级查找 |
| `setup --global` 废弃后用户习惯迁移 | 低 | 保留命令但内部转发 + 输出 deprecated 提示 |
| update-notifier 网络请求延迟 | 低 | 异步非阻塞检查，缓存 24h，离线静默跳过 |
| SessionStart Hook 在非 spec-first 项目触发 | 低 | 命令尾部 `2>/dev/null \|\| true` 确保静默失败；`viewer open` 内部检测 `.spec-first/` 不存在时直接退出 |
| Codex 无 SessionStart Hook 机制 | 中 | 文档标注已知限制；Codex 用户需手动 `spec-first viewer open --print-url` |

## 8. 总结

1. 包名 `spec-first`，发布到 npmjs.com 公开 registry
2. 新增 `update` 命令（含 `--dry-run`），合并 `setup --global` + `host-bootstrap` + `hooks install`
3. P0 引入 `update-notifier`，CLI 执行时异步检查新版本
4. postinstall 采用方案 C（try-catch 执行 + 失败提示）
5. `setup --global` 保留但 deprecated，转发到 `update`
6. 开发者模式（pnpm link）保留，文档独立标注
7. SessionStart Hook 全局注册到 `~/.claude/settings.json`，任意项目打开 Claude Code 自动启动 Viewer

**最终用户体验**：

```bash
# 首次安装
npm install -g spec-first@latest

# 进入项目
cd your-project
spec-first init --feat AUTH --mode N --size M --platforms backend

# 后续升级
npm install -g spec-first@latest
spec-first update
```

与 OpenSpec 的 `npm install -g @fission-ai/openspec@latest` + `openspec update` 体验完全对齐。

## 10. 验收标准

1. `npm install -g spec-first@latest` 后，`spec-first --version` 输出正确版本号
2. `spec-first init` 后项目可直接进入 Feature 流程，且 `.claude/settings.json` 包含 AI Runtime Hooks
3. `spec-first update` 刷新 Skill 命令 / MCP 配置 / Git hooks / AI Runtime Hooks，输出变更摘要
4. `spec-first update --dry-run` 输出将发生的变更，不改任何文件
5. 同版本连续执行两次 `update`，第二次输出全部 skipped（幂等）
6. CLI 执行时若有新版本可用，输出升级提示（不阻塞执行）
7. 全局安装与 `npx -y spec-first@latest` 两种方式均可正常运行
8. 全局安装后，在任意项目目录打开 Claude Code，Viewer 自动启动并打开浏览器（`~/.claude/settings.json` 包含 SessionStart Hook）

## 11. 多 Agent 审查发现（v1.1 修订依据）

### 11.1 AI Runtime Hooks 注册缺失 ❌→已补充

`registerAIHooks()`（`ai-runtime-hook.ts:47`）负责将 PreToolUse/PostToolUse/Stop 三个 Hook 写入 `.claude/settings.json`，但当前 **init 和 setup 流程均未调用**。

影响：用户 init 后，AI 编码时的 Gate 校验和 Matrix 检查不会自动触发。

修正：
- update 命令第 6 步新增 `registerAIHooks(cwd)` 调用
- init 流程 `ensureProjectHooks()` 后补充 `registerAIHooks(cwd)` 调用（任务 G）

### 11.2 dry-run 需函数级改造 ❌→已补充

4 个核心函数当前均直接写入文件，不支持 dry-run：

| 函数 | 幂等 | dry-run | 改造方式 |
|------|------|---------|---------|
| `ensureSkillCommands` | ✅ | ❌ | 添加 `dryRun?: boolean` 参数 |
| `ensureHostBootstrap` | ✅ | ⚠️ 仅环境变量 | 添加 `dryRun?: boolean` 参数 |
| `installHooks` | ✅ | ❌ | 添加 `dryRun?: boolean` 参数 |
| `registerAIHooks` | ✅ | ❌ | 添加 `dryRun?: boolean` 参数 |

修正：新增任务 F，统一为 4 个函数添加 dry-run 支持。

### 11.3 tsup 单入口不支持 postinstall ❌→已补充

当前 tsup 仅配置 `src/cli/index.ts` 单入口。`postinstall.ts` 作为独立脚本需要单独构建入口。

修正：新增任务 E，tsup 扩展为多入口 `['src/cli/index.ts', 'src/postinstall.ts']`。

### 11.4 `getCliVersion()` 未导出 ❌→已补充

`router.ts:73` 的 `getCliVersion()` 为私有函数，update 命令需要复用来输出版本号。

修正：任务 B 中导出该函数。

### 11.5 SessionStart Hook 仅源码项目生效 ❌→已补充

**现状**：`.claude/hooks/hooks.json` 是项目级配置，`session-start.js` 中 `startStageViewer()` 引用 `scripts/stage-viewer/bootstrap.js`——该文件仅存在于 spec-first 源码仓库。

**影响**：目标项目打开 Claude Code 时，Viewer 不会自动启动。用户必须手动执行 `spec-first viewer open --print-url`。

**根因分析**：

| 层面 | 问题 |
|------|------|
| hooks.json 作用域 | 项目级，无全局 hooks.json 机制 |
| 脚本路径 | 硬编码 `scripts/stage-viewer/bootstrap.js`，目标项目不存在 |
| Codex 限制 | 无 SessionStart Hook 机制，无法自动触发 |

**修正方案**：

1. 新增 `registerSessionHooks()` 函数（任务 K）
2. 写入全局 `~/.claude/settings.json`（与 AI Runtime Hooks 同机制，但目标为全局路径）
3. 命令改为 `spec-first viewer open --print-url 2>/dev/null || true`（通过全局 CLI，不依赖源码路径）
4. update 命令第 7 步 + postinstall 均调用此函数
5. Codex 暂不支持，文档标注为已知限制

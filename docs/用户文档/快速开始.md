# Spec-First 快速开始指南

> 让 AI 帮你写规范、管流程、保质量 —— 5 分钟上手，小白友好

---

## 💡 这是什么？（30 秒读懂）

**Spec-First 就像"建房子前先画图纸"**

传统开发：需求口头说 → 直接写代码 → 改来改去 → 文档补不上
Spec-First：AI 帮你写需求 → 自动生成设计 → 代码对照规范 → 全程可追溯

**一句话价值**：让 AI 理解你的需求，自动生成规范化文档，确保需求-设计-代码一致。

---

## 🎯 为什么需要它？

| 痛点 | Spec-First 解决方案 |
|------|-------------------|
| 需求散落在微信/文档/脑子里 | 统一存储在 `specs/` 目录，AI 自动生成 |
| 设计和代码对不上 | 自动追踪矩阵，一眼看出哪里没实现 |
| 改需求不知道影响范围 | 自动分析影响，告诉你要改哪些代码 |
| 质量靠人工审查，容易漏 | 自动 Gate 检查（安全、性能、覆盖率） |
| 新人接手项目一脸懵 | 5 分钟生成项目认知文档 |

---

## 📚 核心概念速查（看不懂也能用）

| 术语 | 通俗解释 | 举例 |
|------|---------|------|
| **Feature** | 一个完整的功能需求 | "用户注册功能" |
| **FR** | 功能需求（用户要什么） | FR-001: 用户可以用手机号注册 |
| **NFR** | 非功能需求（性能/安全要求） | NFR-001: 注册接口响应时间 <500ms |
| **DS** | 技术设计方案（怎么实现） | DS-001: 使用 Redis 缓存验证码 |
| **TASK** | 开发任务（具体要写的代码） | TASK-001: 实现发送验证码接口 |
| **Gate** | 质量检查点（确保没问题才能继续） | 检查测试覆盖率是否 ≥75% |
| **追踪矩阵** | 需求-设计-代码对应表 | FR-001 → DS-001 → TASK-001 |

**提示**：第一次使用不需要记住这些，跟着步骤走就行。

## ⚡ 5 分钟快速体验（最简路径）

**适合**：第一次使用，想快速看到效果

只需 3 个命令，不需要理解所有概念：

```bash
# 1. 让 AI 了解你的项目（2 分钟）
/spec-first:first

# 2. 创建一个新功能（1 分钟）
/spec-first:init
# 提示输入时：
# - Feature 标题：用户注册功能
# - 模式：选 N（新功能）
# - 规模：选 S（小型）
# - 平台：backend

# 3. 生成需求文档（2 分钟）
/spec-first:spec
```

**✅ 成功标志**：
- 看到 `specs/FSREQ-20260309-XXX/` 目录
- 里面有 `prd.md` 和 `spec.md` 两个文件
- 打开文件能看到 AI 生成的需求文档

**❌ 如果出错**：运行 `/spec-first:doctor` 检查环境

## 📋 场景 1：开发新功能（完整流程）

**👤 适合谁**：开发者从零开始实现一个新功能

**🎯 解决什么**：不知道怎么写需求文档、设计文档，担心遗漏细节

**⏱️ 预计时间**：30-60 分钟（不含编码）

**📦 你会得到**：完整的需求、设计、任务文档 + 代码实现指导

---

### Step 1: 快速认知项目 (5 分钟)

```bash
/spec-first:first
```

**🤔 这步做什么**：让 AI 快速了解你的项目（就像新同事入职第一天）

**📝 AI 会分析**：
- 用的什么技术（React？Vue？Node.js？）
- 项目结构（代码放哪、测试放哪）
- 编码规范（怎么命名变量、怎么写注释）

**✅ 成功标志**：
- 终端显示 "✓ 已生成 5 份文档"
- 出现 `docs/first/` 目录
- 打开 `docs/first/tech-stack.md` 能看到技术栈清单

**❌ 如果出错**：
- 提示"不是 git 仓库" → 先运行 `git init`
- 提示"找不到代码文件" → 确认当前目录是项目根目录

**💡 小贴士**：
- 第一次运行会慢一点（AI 在分析代码）
- 生成的文档可以手动修改
- 加 `--deep` 参数可以生成更详细的文档

---

### Step 2: 初始化 Feature 工作区 (2 分钟)

```bash
/spec-first:init
```

**交互式引导**:
1. Feature 标题？（如：用户登录功能）
2. 开发模式？（N=新功能 | E=增强 | B=Bug修复 | R=重构）
3. 规模评估？（S=小型 | M=中型 | L=大型）
4. 目标平台？（backend | h5 | admin | miniapp）

**输出**:
- `specs/FSREQ-YYYYMMDD-XXX/` 工作区
- `.spec-first/current` 当前 Feature 指针
- 初始化追踪矩阵 `trace-matrix.yaml`

**可选参数**: `--bootstrap` 自动检查并安装宿主配置（Claude Code/Codex）

---

### Step 3: 编写需求规格 (10 分钟)

```bash
/spec-first:spec
```

**AI 辅助生成**:
- PRD（产品需求文档）- 业务目标、用户故事
- spec.md（需求规格）- 功能需求（FR）、非功能需求（NFR）
- 自动分配追溯 ID（FR-001, NFR-001）

**输出**:
- `specs/FSREQ-xxx/prd.md`
- `specs/FSREQ-xxx/spec.md`
- 更新 `trace-matrix.yaml`

**阶段**: 01_specify

---

### Step 4: 技术设计 (8 分钟)

```bash
/spec-first:design
```

**AI 辅助设计**:
- 架构设计（模块划分、接口定义）
- 数据模型（表结构、字段定义）
- API 规格（接口路径、参数、响应）
- 分配设计 ID（DS-001, API-001）

**输出**: `specs/FSREQ-xxx/design.md`

**阶段**: 02_design

---

### Step 5: 任务拆解 (5 分钟)

```bash
/spec-first:task
```

**AI 自动拆解**:
- 根据设计生成开发任务清单
- 分配任务 ID（TASK-001, TASK-002）
- 建立任务依赖关系
- 估算工作量

**输出**: `specs/FSREQ-xxx/task_plan.md`

**阶段**: 03_plan

---

### Step 6: 代码实现 (按需)

```bash
/spec-first:code
```

**AI 辅助编码**:
- 自动定位进行中的 TASK
- 加载设计规格和代码规范
- 生成符合规范的代码
- 实时代码审查

**阶段**: 04_implement

---

### Step 7: 实现审查 (3 分钟)

```bash
/spec-first:review
```

**自动审查维度**:
- 规范一致性（代码 vs 设计）
- 代码质量（复杂度、重复度）
- 安全扫描（OWASP Top 10）
- 性能检查（N+1 查询、内存泄漏）

**输出**: `specs/FSREQ-xxx/review-report.md`

---

### Step 8: 验证测试 (5 分钟)

```bash
/spec-first:verify
```

**Gate 校验**:
- 追踪完整性（所有 FR/DS/TASK 已实现）
- 测试覆盖率（≥75%）
- 安全扫描通过
- 文档完整性

**阶段**: 05_verify

## 🔧 场景 2：修复 Bug（快速场景）

**适用**: 开发者需要快速修复一个 Bug

**预计时间**: 15-30 分钟（不含编码）

### Step 1: 恢复上下文 (3 分钟)

```bash
/spec-first:catchup
```

**AI 自动恢复**:
- 当前 Feature 状态概览
- 最近变更历史
- 待办任务清单
- 关键决策记录

---

### Step 2: 一致性分析 (5 分钟)

```bash
/spec-first:analyze
```

**跨产物分析**:
- 需求 vs 设计一致性
- 设计 vs 代码一致性
- 测试 vs 需求覆盖度
- 识别潜在问题和遗漏

**输出**: `specs/FSREQ-xxx/analysis-report.md`

---

### Step 3: 修复代码 (按需)

```bash
/spec-first:code
```

**AI 辅助修复**:
- 根因分析
- 生成修复代码
- 回归测试建议

---

### Step 4: 验证修复 (5 分钟)

```bash
/spec-first:verify
```

**确保修复质量**:
- Gate 检查通过
- 无新增安全风险
- 测试覆盖率未下降

---

## 📝 场景 3：产品经理写需求（PM 场景）

**适用**: 产品经理需要编写规范化需求文档

**预计时间**: 40-60 分钟

### Step 1: 快速认知项目 (5 分钟)

```bash
/spec-first:first
```

**了解技术背景**:
- 当前技术栈
- 系统架构
- 已有功能模块

---

### Step 2: 初始化 Feature (2 分钟)

```bash
/spec-first:init
```

**输入**: Feature 标题、模式、规模、平台

---

### Step 3: 编写需求规格 (20 分钟)

```bash
/spec-first:spec
```

**AI 辅助编写**:
- 生成 PRD 模板
- 补充业务目标、用户故事
- 定义验收标准
- 自动分配 FR/NFR ID

**重点**: 确保需求可测试、可追溯

---

### Step 4: 需求质量审查 (10 分钟)

```bash
/spec-first:spec-review
```

**C10 质量指标**:
- 完整性（所有必填项已填写）
- 一致性（无矛盾需求）
- 可测性（验收标准明确）
- 优先级合理性

**输出**: 质量评分 + 改进建议

---

### Step 5: 跟踪状态 (2 分钟)

```bash
/spec-first:status
```

**实时监控**:
- Feature 进度（当前阶段）
- 健康分（0-100）
- 追踪覆盖率（C1-C9）
- 风险预警

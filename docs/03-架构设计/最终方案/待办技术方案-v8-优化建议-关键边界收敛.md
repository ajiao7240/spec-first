# Spec-First 待办技术方案 v8 优化建议

> **版本**: 8.1 边界收敛建议稿
> **日期**: 2026-03-27
> **目标**: 针对 v8 当前 4 个关键边界问题，给出最佳实践级的收敛建议
> **适用对象**: 产品、研发、测试、平台维护者

---

## 1. 文档目的

这份文档只解决 4 个问题：

1. `decision_hints` 应该如何处理
2. `verify_commands` 应该放在哪一层
3. `Codex / Kiro` 的静态物化时机应该如何定义
4. `preset` 的能力边界应该怎么收死

这 4 个问题不解决，v8 虽然方向正确，但仍然存在：

- 模型未完全收敛
- 层次边界仍有泄漏
- 平台接入链路不够稳定
- 易用性增强有滑向第二套策略系统的风险

本建议稿的原则是：

- 不推翻 v8 主架构
- 只修补影响长期稳定性的关键边界
- 优先选择对当前代码落点最友好的方案
- 避免重新引入 v5/v6/v7.1 的结构性复杂度

---

## 2. 总体结论

v8 应继续作为主方案，但需要补 4 条硬约束。

一句话结论：

**核心状态只保留一套真源，执行参数单独收敛，平台安装与任务运行分离，UX 快捷入口绝不反向定义核心策略。**

对应到本项目，建议定为：

1. 删除 `decision_hints` 作为核心任务字段
2. 新增一个最小 `action_config`，只承载非策略型执行参数
3. `verify_commands` 统一收敛到 `action_config.check.verify_commands`
4. `Codex / Kiro` 采用“安装期全量物化 + 任务期轻量选择”两段式模型
5. `preset` 只允许展开为标准输入字段，不允许直接定义 capability

这套方案兼顾：

- 架构合理
- 功能内聚
- 分层清晰
- 扩展性高
- 维护简单
- 用户上手成本低

---

## 3. 最佳实践判断标准

这次裁决采用 5 条判断标准：

### 3.1 单一真源

同一类语义只能有一个稳定真源。

例如：

- 当前动作来自 `workflow`
- 当前能力选择来自 `selector`
- 执行验证命令来自 `action_config`
- 结果证据来自 `evidence`

不能再出现“同一件事有两个字段都在表达”的情况。

### 3.2 关注点分离

不同层只做一件事：

- `workflow` 负责动作拓扑
- `selector` 负责能力选择
- `action_config` 负责动作执行参数
- `evidence` 负责结果记录
- `platform adapter` 负责平台消费

### 3.3 默认可运行

系统的基础闭环不能依赖交互式 LLM、人工解释或平台特例。

关闭 LLM、关闭 explain、关闭 preset 后，主流程仍应稳定运行。

### 3.4 平台安装与任务运行分离

安装期负责铺设平台基础产物，任务运行期负责选择当前任务真正需要的子集。

不能把平台初始化和任务状态推进混成一个生命周期。

### 3.5 UX 增强不反向污染模型

`preset`、`explain`、默认值、引导文案都应该是增强层，而不是新的策略层。

## 4. 决策一：删除 decision_hints，替换为最小 action_config

## 4.1 问题定义

当前 v8 文档虽然强调单一任务模型，但仍然保留了 `decision_hints`。

这会带来 3 个问题：

- 它和 `selector.current` 语义重叠
- 它在现有代码里仍是运行时主输入
- 它没有明确边界，容易继续长成第二套策略系统

这会直接破坏“模型统一”和“功能内聚”。

## 4.1.1 `decision_hints` 与 `action_config` 的本质区别

两者最核心的区别是：

- `decision_hints` 是旧设计里的宽泛提示容器
- `action_config` 是新设计里的动作执行参数容器

对比表如下：

| 维度 | `decision_hints` | `action_config` |
|------|------------------|-----------------|
| 定位 | 泛化提示容器 | 动作执行参数 |
| 语义边界 | 容易混入策略、提示、命令、模式 | 只放动作执行时真正需要的参数 |
| 是否容易和 selector 重叠 | 高 | 低 |
| 是否适合长期作为主模型字段 | 否 | 是 |
| 典型内容 | implement mode、check hints、verify_commands | `check.verify_commands`、`finish.required_fields` |
| 是否允许表达 capability 选择 | 容易误入 | 不允许 |

简单理解：

- `decision_hints` 容易变成“这一步该怎么想、怎么做、跑什么”的混合包
- `action_config` 只负责“这一步执行时要带哪些结构化参数”

## 4.1.2 最小示例

旧方式：

```json
{
  "decision_hints": {
    "implement": {
      "mode": "debug"
    },
    "check": {
      "verify_commands": ["pnpm lint", "pnpm typecheck"]
    }
  }
}
```

这个结构的问题是：

- `implement.mode` 像策略
- `check.verify_commands` 像执行参数
- 两者被放进了同一个字段

推荐的新方式：

```json
{
  "selector": {
    "current": {
      "action": "implement",
      "selected_capabilities": ["before-dev", "systematic-debugging"]
    }
  },
  "action_config": {
    "check": {
      "verify_commands": ["pnpm lint", "pnpm typecheck"]
    }
  }
}
```

这里的边界就很清楚：

- `selector` 决定 implement 需要哪些 capability
- `action_config.check` 决定 check 时具体跑哪些命令

## 4.2 最佳实践建议

**推荐方案：从 v8 核心模型中移除 `decision_hints`，用最小 `action_config` 替代。**

`action_config` 的定位非常窄：

- 它不是 selector
- 它不是 policy engine
- 它不是 prompt model
- 它只是“某个动作在执行时需要的结构化参数”

建议结构：

```json
{
  "action_config": {
    "check": {
      "verify_commands": ["pnpm lint", "pnpm typecheck"]
    }
  }
}
```

## 4.3 为什么这是更好的方案

### 对产品

- 用户不需要理解 `decision_hints` 这种抽象名词
- 文档和命令行更容易解释

### 对研发

- `selector` 和执行参数彻底分开
- hook 不再依赖宽泛 dict
- 类型约束更容易做严

### 对测试

- 可以明确测两类东西：
  - selector 选择是否正确
  - action_config 是否被正确消费

### 对平台

- 平台拿到的是稳定的消费结构，而不是半策略半参数的混合对象

## 4.4 对当前代码的落点建议

建议改动方向：

- [types.py](/Users/kuang/xiaobu/spec-first/.spec-first/scripts/common/types.py)
  - 删除 `decision_hints`
  - 增加 `action_config`

- [task_store.py](/Users/kuang/xiaobu/spec-first/.spec-first/scripts/common/task_store.py)
  - 不再生成默认 `decision_hints`
  - 改为生成最小 `action_config`

- [inject-subagent-context.py](/Users/kuang/xiaobu/spec-first/packages/cli/src/templates/claude/hooks/inject-subagent-context.py)
  - 不再读取 `decision_hints`
  - 改为读取 `selector.current` 和 `action_config`

## 4.5 结论

**v8 的正式主模型里不应再出现 `decision_hints`。**

这是这次优化里最重要的一条。

## 5. 决策二：verify_commands 统一归属到 action_config.check

## 5.1 问题定义

`check` 节点是 v8 质量提升的核心，但当前文档没有把 `verify_commands` 放到一个稳定位置。

如果这个问题不解决：

- `check` 的门禁无法稳定实现
- evidence 只能记录结果，无法追溯输入
- 动态平台与静态平台都无法共享一致语义

## 5.2 最佳实践建议

**推荐方案：`verify_commands` 只属于 `action_config.check.verify_commands`。**

它不应该放在：

- `selector`
- `evidence`
- `preset`
- 平台模板内部

建议结构：

```json
{
  "action_config": {
    "check": {
      "verify_commands": [
        "pnpm lint",
        "pnpm typecheck"
      ]
    }
  },
  "evidence": {
    "verify_result": {
      "commands": [
        "pnpm lint",
        "pnpm typecheck"
      ],
      "status": "passed"
    }
  }
}
```

这满足了非常清楚的因果关系：

- `action_config.check.verify_commands` 是输入
- `evidence.verify_result` 是输出

## 5.3 生成与消费规则

建议采用下面的责任分配：

### 生成方

- 任务创建时生成默认 `verify_commands`
- 用户可以显式覆写
- preset 只能影响默认 workflow / mode，不能直接注入 raw commands

### 消费方

- `check` hook 或 runner 只读取 `action_config.check.verify_commands`
- 执行结果只回写 `evidence.verify_result`

### fallback

为了避免系统不可用，建议保留轻量 fallback：

1. `task.json.action_config.check.verify_commands`
2. `worktree.yaml.verify`
3. 空列表

但从架构定义上，**唯一正式真源是 `action_config.check.verify_commands`**。

## 5.4 对当前代码的落点建议

- [ralph-loop.py](/Users/kuang/xiaobu/spec-first/packages/cli/src/templates/claude/hooks/ralph-loop.py)
  - 把优先级 1 从 `decision_hints.check.verify_commands` 改成 `action_config.check.verify_commands`

- [task_store.py](/Users/kuang/xiaobu/spec-first/.spec-first/scripts/common/task_store.py)
  - 默认值生成逻辑迁移到 `action_config.check`

## 5.5 结论

**`verify_commands` 是动作执行参数，不是策略，不是证据，不是平台配置。**

这个归位后，`check` 才真正成为稳定节点。

## 5.6 谁来配置 verify_commands

最佳实践不是“每个任务都手工填”，而是三层责任分离：

```text
┌──────────────────────────────────────────────────────────────┐
│ Project Defaults Layer                                       │
│ 由系统维护者/项目维护者定义默认 verify profiles              │
│ 例如: app / admin / api / package 的默认检查命令             │
└──────────────────────────────┬───────────────────────────────┘
                               │ 解析默认值
┌──────────────────────────────▼───────────────────────────────┐
│ Task Creation / Update Layer                                 │
│ 系统根据 package / dev_type / task_mode 自动生成             │
│ action_config.check.verify_commands                          │
└──────────────────────────────┬───────────────────────────────┘
                               │ 可选增强
┌──────────────────────────────▼───────────────────────────────┐
│ LLM Suggestion Layer                                         │
│ 基于上下文建议补充检查项，但不作为唯一真源                   │
└──────────────────────────────┬───────────────────────────────┘
                               │ 必要时覆写
┌──────────────────────────────▼───────────────────────────────┐
│ Human Override Layer                                         │
│ 用户只在特殊任务下显式修改 verify_commands                   │
└──────────────────────────────────────────────────────────────┘
```

### 推荐责任分工

#### 系统默认值

负责方：

- 项目维护者
- 框架维护者

职责：

- 提供项目级、包级默认 verify 规则
- 让大多数任务无需人工配置

#### 任务生成

负责方：

- 系统

职责：

- 在任务创建时自动把默认规则解析成最终 `verify_commands`
- 在任务更新时按新上下文重算

#### LLM 建议

负责方：

- LLM 增强层

职责：

- 提示可能遗漏的检查项
- 增强灵活性

限制：

- 不作为唯一真源
- 不直接替代默认规则

#### 人工覆写

负责方：

- 用户

职责：

- 只在特殊场景下改写任务级 verify 配置

### 结论

`verify_commands` 不应默认依赖人工配置。

最合理的机制是：

- 默认自动生成
- LLM 辅助建议
- 人工必要时覆写

## 6. 决策三：Codex / Kiro 采用“两段式物化”

## 6.1 问题定义

当前 v8 已经正确区分了动态平台和静态平台，但对静态平台的运行方式还不够清楚。

核心模糊点是：

- 是不是每次任务推进都重写平台技能目录
- 还是在安装期一次性生成全部技能
- 任务运行时到底消费什么

如果这点不明确，会直接影响：

- 平台实现复杂度
- 文件写入频率
- 缓存稳定性
- 用户对平台行为的理解

## 6.2 最佳实践建议

**推荐方案：Codex / Kiro 使用“两段式物化”。**

### 第一段：安装期全量物化

在 `init / update` 时完成：

- capability 正文分发
- 平台包装文件生成
- 平台目录初始化

这一步的目标是：

- 保证平台运行所需的基础资产始终存在
- 避免任务运行时频繁重写大目录

这与当前实现天然契合：

- [codex.ts](/Users/kuang/xiaobu/spec-first/packages/cli/src/configurators/codex.ts)
- [kiro.ts](/Users/kuang/xiaobu/spec-first/packages/cli/src/configurators/kiro.ts)

### 第二段：任务期轻量选择

在任务运行时不重写全量技能目录，只生成轻量选择结果，例如：

- `task.json.selector.current`
- 可选的 `selected-capabilities.json`
- 可选的 `selected-capabilities.md`

平台消费的是“当前任务激活了哪些 capability”，而不是“重新安装一批技能”。

## 6.3 为什么这是最佳实践

### 对平台维护

- 初始化路径稳定
- 任务推进路径轻量
- 避免大量无意义文件 churn

### 对用户体验

- 平台目录不会在每个动作切换时大改
- 当前任务选择结果可以单独查看和解释

### 对扩展性

- 后续新增平台时也可沿用：
  - 安装期：铺设基础资产
  - 运行期：消费当前选择结果

## 6.4 明确边界

静态平台的推荐边界是：

- 不在任务推进时重写 `.codex/skills/*`
- 不在任务推进时重写 `.kiro/skills/*`
- 可以在任务推进时更新轻量 manifest
- 平台 wrapper / start skill 可以读取当前 manifest

## 6.5 建议结构

```text
.spec-first/runtime/platform/
  current-capabilities.json
  current-capabilities.md
```

或者直接把这份信息作为 `task.json.selector.current` 的唯一来源，由平台 wrapper 去读。

推荐更简单的版本：

- 不额外加 manifest 文件
- 优先让平台读取 `task.json.selector.current`
- 只有在平台消费不便时，才加只读 manifest

## 6.6 结论

**静态平台不应该按“每个任务都重装技能”来设计。**

正确做法是：

- 安装期铺底
- 任务期只切换选择结果

## 7. 决策四：Preset 只展开标准字段，禁止直接定义 capability

## 7.1 问题定义

v8 吸收 preset 是对的，但如果 preset 能直接定义 capability，它很快就会从 UX 快捷入口变成第二套策略系统。

这会带来两个后果：

- selector 失去中心地位
- 用户开始在 preset 里偷写业务规则

## 7.2 最佳实践建议

**推荐方案：preset 只能展开标准输入字段，不能直接声明 capability。**

允许 preset 设置的字段建议限制为：

- `workflow`
- `task_mode`
- 可选 `dev_type`

不建议允许：

- `capabilities`
- `verify_commands`
- 原始 prompt 片段
- 冲突规则
- category quota

推荐示例：

```json
{
  "presets": {
    "bug-debug": {
      "workflow": "default",
      "task_mode": "debug",
      "dev_type": "backend",
      "description": "复杂 bug 调试"
    }
  }
}
```

而不是：

```json
{
  "presets": {
    "bug-debug": {
      "workflow": "default",
      "task_mode": "debug",
      "capabilities": ["systematic-debugging", "check"]
    }
  }
}
```

## 7.3 为什么这样更好

### 对产品

- 用户获得快捷入口
- 不必理解能力系统细节

### 对研发

- selector 仍然是唯一能力决策点
- preset 不会绕过规则系统

### 对测试

- 只需验证“preset 是否正确展开”
- 不必为 preset 单独构建 capability 决策测试矩阵

## 7.4 透明化建议

为了兼顾上手容易和可理解性，建议：

- `task create --preset <id> --explain`
  - 先显示 preset 展开结果
  - 再显示 selector explain

输出顺序建议是：

1. preset 展开了什么
2. 当前 action 是什么
3. selector 为什么选了这些 capability

这样用户会感觉系统是可理解的，而不是魔法。

## 7.5 结论

**preset 必须被严格定义为 UX 语法糖。**

它可以提升上手效率，但不能成为策略入口。

## 8. 推荐后的 v8 主模型

在以上 4 条建议落地后，推荐的最小主模型如下：

```json
{
  "id": "add-v8-selector",
  "title": "实现 v8 selector",
  "status": "planning",
  "dev_type": "backend",
  "task_mode": "default",
  "package": "cli",
  "workflow": {
    "id": "default",
    "steps": [
      { "index": 1, "action": "implement" },
      { "index": 2, "action": "check" },
      { "index": 3, "action": "finish" },
      { "index": 4, "action": "create-pr" }
    ],
    "current_step": 0
  },
  "selector": {
    "current": null,
    "history": []
  },
  "action_config": {
    "check": {
      "verify_commands": []
    }
  },
  "evidence": {
    "selector_trace": {},
    "verify_result": {
      "commands": [],
      "status": "unknown"
    },
    "finish_note": "",
    "release_note": ""
  },
  "meta": {}
}
```

这个结构有 4 个好处：

- 没有双真源
- `check` 闭环完整
- 平台消费边界清楚
- UX 增强无法污染核心模型

## 9. 对多角色的价值

## 9.1 产品

产品最关心的是：

- 用户能否快速开始
- 节点是否清楚
- 结果是否可解释

这套建议保留了：

- `preset`
- `selector explain`
- 关键动作节点

同时避免了：

- 模型越来越难讲
- 用户被迫理解太多内部抽象

## 9.2 研发

研发最关心的是：

- 分层是否稳定
- 代码落点是否清楚
- 未来是否容易扩展

这套建议会把运行时主链压缩成：

- `workflow`
- `selector`
- `action_config`
- `evidence`

这已经足够支撑绝大多数执行场景。

## 9.3 测试

测试最关心的是：

- 输入输出是否可验证
- 是否容易构造用例
- 是否能清楚定位故障

这套方案把关键测试面收敛成：

- selector 选择正确性
- verify_commands 读取正确性
- evidence 写入正确性
- 平台消费正确性

## 9.4 平台维护

平台维护者最关心的是：

- 动态平台和静态平台能否长期共存
- 安装流程和任务运行流程是否打架

“安装期全量物化 + 任务期轻量选择”能明显降低维护成本。

---

## 10. 建议的实施顺序

建议按下面顺序实施：

1. 先删 `decision_hints`，定义 `action_config`
2. 再把 `verify_commands` 收敛到 `action_config.check`
3. 再统一动态平台的消费方式
4. 再明确静态平台的安装期 / 运行期边界
5. 最后把 preset 改成严格受限的 UX sugar

这样做的原因是：

- 前两步先收模型
- 中间两步再稳平台
- 最后一步再修入口

这是最稳的工程路径。

---

## 11. 最终裁决

针对这 4 个问题，推荐裁决如下：

| 问题 | 推荐裁决 | 原因 |
|------|----------|------|
| `decision_hints` | 从 v8 核心模型删除 | 它是双真源入口 |
| `verify_commands` | 收敛到 `action_config.check.verify_commands` | 它是动作输入，不是策略 |
| `Codex / Kiro` 物化时机 | 安装期全量物化，任务期轻量选择 | 最符合当前 configurator 架构 |
| `preset` 边界 | 只展开标准字段，禁止直接定义 capability | 避免第二套策略系统 |

一句话总结：

**v8 最需要的不是再加功能，而是把边界写死。边界一旦收死，这套 agent 框架才真正能长期提升效率和质量。**

# Skill 方案取舍表

> **日期**: 2026-03-27
> **目的**: 对比最小裁剪版与智能决策增强版，明确何时该用哪个方案

---

## 1. 方案对比

| 维度 | `v8-lite-最小裁剪版` | `skill-智能决策增强方案` |
|------|----------------------|---------------------------|
| 目标 | 只解决“按场景选 skill” | 解决“按场景选 skill + 动态组合 skill” |
| 复杂度 | 低 | 中-高 |
| 改动面 | 小 | 明显更大 |
| 是否需要新规则引擎 | 不需要 | 需要，至少有 `enhancement_rules` |
| 是否需要上下文分析 | 不需要 | 需要 `file_count` / `keywords` / `title` / `description` |
| 是否需要依赖/冲突处理 | 不需要 | 需要 |
| 是否需要用户覆写层 | 可选 | 需要 |
| 是否适合当前最小落地 | 是 | 否，除非 skill 数量和场景复杂度都很高 |

---

## 2. 什么时候用最小裁剪版

适合这些情况：

- skill 数量不多
- 任务类型相对稳定
- 只想让系统自动挑一个最合适的 skill 集合
- 希望尽量不改现有系统

这时最小裁剪版就够了：

- `skill-profiles.json`
- `resolve_skills()`
- `explain`
- hook 注入

---

## 3. 什么时候上智能决策增强版

适合这些情况：

- 已经有 20+ 个 skill
- 同一任务经常需要多个 skill 组合
- 任务上下文会显著影响 skill 选择
- 愿意接受一个轻量规则引擎

这时才考虑：

- `enhancement_rules`
- `skill_dependencies`
- `conflicts_with`
- `user_override_skills`

---

## 4. 建议

如果目标是：

**“最小满足，不过度设计”**

就先用：

- `v8-lite-最小裁剪版`

如果后面发现：

- skill 太多
- 组合太复杂
- 查表不够用了

再升级到：

- `skill-智能决策增强方案`

---

## 5. 一句话结论

**先用“查表选 skill”，只有在 skill 数量和组合复杂度真的上来之后，再引入“智能决策增强”。**

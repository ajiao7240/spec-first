# 待办技术方案综合结论 - 专业裁决

> **日期**: 2026-03-27
> **裁决类型**: 架构主方案选择
> **评审维度**: 产品、研发、测试、平台

---

## 执行摘要

**裁决结果**: 以 **v8 作为主架构基线**，选择性吸收 v7.1 的轻量 UX 增强。

**核心理由**:
- v8 的模型更单一、边界更清晰、维护成本更可控
- v7.1 的产品友好性优势不足以推翻 v8 的架构收敛
- 产品入口优化可以在 v8 基础上轻量叠加

---

## 1. 客观对比分析

### 1.1 架构复杂度对比

| 维度 | v7.1 | v8 | 专业判断 |
|------|------|-----|----------|
| **任务模型** | 单一 | 单一 | 相同 ✓ |
| **用户接口** | 双模式（Simple/Expert） | 单一接口 | v8 更简洁 ✓ |
| **LLM 集成** | P2 引入，交互式助手 | 不引入 | v8 更可控 ✓ |
| **配置层次** | Preset + Registry + Rules | Registry + Rules | v8 更统一 ✓ |
| **平台支持** | P1 动态，P2 静态 | P1 同时支持动态+静态 | v8 更完整 ✓ |

**结论**: v8 在架构复杂度上全面优于 v7.1

---

### 1.2 实施成本对比

| 阶段 | v7.1 | v8 | 差异 |
|------|------|-----|------|
| **P1 范围** | Preset + Selector + 动态平台 + explain | Selector + 动态+静态平台 | v8 更聚焦 |
| **P1 工期** | 4-6 周 | 4 周 | v8 更快 |
| **P2 范围** | LLM 助手 + 超时降级 | 优化与增强 | v7.1 风险更高 |
| **测试成本** | 高（双模式 + LLM） | 低（单一路径） | v8 更低 |
| **维护成本** | 中高 | 低 | v8 更低 |

**结论**: v8 的实施和维护成本显著低于 v7.1

## 2. 关键决策点重新审视

### 2.1 Preset Profiles 的定位

**v7.1 的观点**: 核心特性，降低学习曲线

**客观分析**:
- Preset 是**命令层快捷入口**，不是架构核心
- 可以作为可选功能在 v8 基础上实现
- 不应该成为选择主架构的决定性因素

**裁决**:
- ✅ 吸收 Preset 概念
- ❌ 不作为核心任务模型的一部分
- ✅ 定义为命令层语法糖

---

### 2.2 LLM 智能助手的必要性

**v7.1 的观点**: P2 引入，定位为"助手"

**客观分析**:
- LLM 引入了不确定性和测试复杂度
- 规则引擎已经可以覆盖 80% 的场景
- 交互式 LLM 会打断工作流
- 2026 年的 AI 工具不一定需要 LLM 交互

**裁决**:
- ❌ 不引入交互式 LLM 助手
- ✅ 如需智能化，考虑静默优化（不打断用户）
- ✅ 保持规则引擎的确定性

---

### 2.3 双模式接口的价值

**v7.1 的观点**: Simple Mode + Expert Mode

**客观分析**:
- 双模式增加了代码复杂度
- 用户需要理解"什么时候用哪个模式"
- 单一接口 + 良好文档更简洁

**裁决**:
- ❌ 不引入双模式接口
- ✅ 单一接口 + 渐进式文档
- ✅ 通过 Preset 快捷命令降低门槛

---

### 2.4 `selector explain` 的定位

**v7.1 的观点**: 独立命令，提升可理解性

**客观分析**:
- explain 是 trace 的人类可读投影
- 实现成本低，用户价值高
- 不增加架构复杂度

**裁决**:
- ✅ 吸收 `selector explain` 命令
- ✅ 定义为 trace 的格式化输出
- ✅ 作为调试和透明化工具

---

## 3. 最终裁决

### 3.1 主方案选择

**采用 v8 作为唯一主方案**

理由：
1. 架构更单一、边界更清晰
2. 实施和维护成本更低
3. 可测试性更强
4. 平台支持更完整
5. 符合 YAGNI 原则

---

### 3.2 从 v7.1 吸收的内容

**仅吸收以下两项**:

#### 吸收项 1: `selector explain` 命令

**定位**: trace 的人类可读投影

**实现方式**:
```bash
$ spec-first selector explain
```

**输出格式**: 格式化的 trace 信息，包含决策理由

**集成方式**: 作为独立命令，不影响核心流程

---

#### 吸收项 2: Preset 快捷命令

**定位**: 命令层语法糖，不进入核心任务模型

**实现方式**:
```bash
# Preset 只是参数预设
$ spec-first task create "Fix bug" --preset bug-debug

# 等价于
$ spec-first task create "Fix bug" \
  --dev-type backend \
  --task-mode debug \
  --workflow default
```

**关键约束**:
- Preset 不存储在 task.json
- Preset 不参与 selector 逻辑
- Preset 只是命令行参数的批量设置

---

### 3.3 不吸收的内容

**明确不吸收以下内容**:

1. **Simple / Expert Mode**
   - 理由: 增加复杂度，收益不明显

2. **交互式 LLM Assistant**
   - 理由: 引入不确定性，测试成本高

3. **Preset 作为核心模型**
   - 理由: 应该是命令层，不是数据层

4. **分阶段平台支持**
   - 理由: v8 的 P1 即支持全平台更合理

## 5. 风险评估

### 5.1 主要风险

| 风险 | 概率 | 影响 | 缓解措施 |
|------|------|------|----------|
| 用户觉得不够智能 | 中 | 中 | 文档强调确定性优势 |
| 学习曲线仍然陡 | 中 | 中 | Preset + 文档优化 |
| 竞品有 AI 辅助 | 高 | 低 | 强调可测试性和稳定性 |

### 5.2 风险应对

**如果用户强烈要求 AI 辅助**:
- 考虑静默优化（不打断用户）
- 不引入交互式 LLM
- 保持核心流程的确定性

---

## 6. 总结

### 6.1 核心判断

**v8 > v7.1**，但不是"全盘否定 v7.1"

正确的做法是：
- ✅ v8 作为主架构基线
- ✅ 选择性吸收 v7.1 的轻量 UX 增强
- ❌ 不采纳会扩大模型复杂度的部分

### 6.2 专业建议

**给产品团队**:
- 产品友好性很重要，但不能以牺牲架构清晰度为代价
- Preset 和 explain 已经足够降低门槛
- 不要盲目追求"AI 辅助"，确定性也是竞争力

**给研发团队**:
- v8 的架构更符合长期维护需求
- 轻量化不是"功能少"，而是"职责清晰"
- 保持 YAGNI 原则，不过度设计

**给测试团队**:
- v8 的可测试性显著优于 v7.1
- 单一路径、确定性输出是质量保证的基础
- 避免引入不必要的测试复杂度

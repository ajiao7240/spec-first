# Spec-First 待办技术方案 v8 - 轻量最佳实践方案

> **版本**: 8.1 (基于专业裁决更新)
> **日期**: 2026-03-27
> **状态**: 推荐主方案
> **定位**: 面向多角色使用，但不过度设计的终局主方案
> **更新**: 吸收 v7.1 的 `selector explain` 和 Preset 快捷命令

---

## 1. 方案定位

v8 的目标很明确：

- 支持产品、研发、测试等不同角色理解和使用
- 提升关键阶段节点的执行质量
- 保持架构小而硬，不引入重型流程治理系统

v8 不追求做成企业级流程平台。

v8 追求的是：

- **模型统一**
- **职责清晰**
- **节点可控**
- **平台可落地**

一句话总结：

**用最少的结构，稳定提升关键动作节点的执行质量。**

---

## 2. 设计结论

v8 采用以下核心判断：

1. **任务系统以 action 为中心**
   不再以 phase 编号作为规则核心输入。

2. **selector 只负责选择**
   不负责编排，不负责审批，不负责执行。

3. **平台只负责消费**
   规则只在运行时核心层定义，平台不复写。

4. **质量提升聚焦关键节点**
   只强化 `implement`、`check`、`finish`、`create-pr` 四个高价值动作。

5. **角色支持是轻量支持**
   让不同角色看懂同一任务状态，但不引入复杂权限和审批流。

### 动作推进数据流

```text
当前动作完成
  -> 写入最小 evidence
  -> selector.current 追加到 selector.history
  -> workflow.current_step += 1
  -> 推导新的 current_action
  -> 重算 selector.current
  -> 平台消费新的 capability 结果
```

规则约定：

- 只有动作成功完成后才追加 `selector.history`
- 动作失败回退时不追加 history
- 重试时保留失败 evidence，但不把失败快照写入 history

### 失败与回退路径

```text
Selector 失败
  -> 不推进 workflow
  -> 保留当前 action
  -> 记录错误信息到 trace / log
  -> 平台不消费空结果
  -> 不追加 selector.history

Capability 缺失
  -> 标记 selector.current 为无效
  -> 阻止进入关键动作执行
  -> 提示补齐 registry / capability 文件

LLM enhancer 超时
  -> 直接回退到规则结果
  -> trace 记录 llm_adjusted = false
```

### 失败时的端到端回退

```text
implement 执行失败
  -> evidence.verify_result.status = "failed"
  -> workflow.current_step 不推进
  -> selector.current 保持不变
  -> selector.history 不追加失败快照
  -> 用户修复后重新执行当前 action
```

## 2.3 UX 增强层（从 v7.1 吸收）

基于专业裁决，v8 吸收 v7.1 的两项轻量 UX 增强，不增加核心架构复杂度。

### 2.3.1 Preset 快捷命令

**定位**: 命令层语法糖，不进入核心任务模型

**原则**:
- Preset 不存储在 task.json
- Preset 不参与 selector 逻辑
- Preset 只是命令行参数的批量设置
- Preset 只能展开标准输入字段，不能直接定义 capability

**使用示例**:

```bash
# 推荐：使用 preset，零配置
$ spec-first task create "Fix login bug" --preset bug-debug
✓ Task created with preset: bug-debug
✓ Workflow: default (implement → check → finish → create-pr)
✓ Current Action: implement
✓ Capabilities: before-dev, systematic-debugging

# 高级：完整配置（等价于上面）
$ spec-first task create "Fix login bug" \
  --dev-type backend \
  --task-mode debug \
  --workflow default
```

**默认 Presets**:

| Preset ID | 参数预设 | 适用场景 | 推荐用于 |
|-----------|----------|----------|----------|
| `quick-fix` | `task_mode=default, workflow=quick-fix` | 快速修复、typo | 紧急热修复、文案修改 |
| `feature-dev` | `task_mode=default, workflow=default` | 标准功能开发 | 新功能、功能增强、重构 |
| `bug-debug` | `task_mode=debug, workflow=default` | 复杂 bug 调试 | 性能问题、间歇性错误 |
| `docs-update` | `task_mode=docs, workflow=docs-only` | 文档更新 | README、API 文档、注释 |

**Preset 允许展开的字段**:

- `workflow`
- `task_mode`
- 可选 `dev_type`

**Preset 不允许展开的字段**:

- `capabilities`
- `verify_commands`
- 原始 prompt 片段
- 规则优先级
- 冲突规则

**自定义 Preset**:

文件位置：`.spec-first/config/presets.json`

```json
{
  "presets": {
    "my-workflow": {
      "workflow": "custom",
      "task_mode": "default",
      "dev_type": "backend",
      "description": "我的自定义工作流"
    }
  }
}
```

### 为什么要这样限制

这样做是为了防止 preset 滑成第二套策略系统。

v8 里：

- `preset`
  - 只负责提升任务创建体验
- `selector`
  - 仍然是唯一 capability 决策点

一句话：

**preset 可以决定任务从什么标准输入开始，但不能决定最终 capability 结果。**

### 2.3.2 Selector Explain 命令

**定位**: trace 的人类可读投影

**使用场景**:
- 理解为什么选择了某些 capabilities
- 调试 selector 规则
- 学习系统决策逻辑

**命令**:

```bash
$ spec-first selector explain
```

**输出示例**:

```text
Current Action: implement
Selected Capabilities: before-dev, systematic-debugging

Decision Trace:
  1. ✓ Action bundle loaded: [before-dev]
  2. ✓ Mode override applied: debug → [systematic-debugging]
  3. ✓ Conflict check: No conflicts
  4. ✓ Category quota: development(2/3), verification(0/2)
  5. ✓ Final selection: [before-dev, systematic-debugging]

Why these capabilities?
  • before-dev
    - Required for all implement actions
    - Ensures context is loaded before coding

  • systematic-debugging
    - Added by task_mode=debug
    - Provides structured debugging approach

To customize:
  $ spec-first task update --workflow <id>
  $ spec-first task update --task-mode <mode>
  $ spec-first task update --package <path>
```

**集成到工作流**:

```bash
# 创建任务时自动显示
$ spec-first task create "Fix bug" --preset bug-debug --explain

# 任务推进时显示变化
$ spec-first task advance --explain
```

**实现**: 读取 `task.json.selector.current.trace`，格式化输出

- 不引入复杂角色权限系统
- 不实现企业级审批流
- 不建立完整 work hierarchy 平台
- 不让 LLM 成为基础依赖

## 4. 核心设计原则

## 4.0 基础术语边界

为避免混淆，v8 对下面 4 个概念作明确区分：

### `workflow`

`workflow` 不是 agent。

`workflow` 的职责是定义任务步骤拓扑和当前推进位置，例如：

- `implement`
- `check`
- `finish`
- `create-pr`

它回答的是：

**现在任务处于哪一步。**

### `current_action`

`current_action` 由：

- `workflow.steps`
- `workflow.current_step`

共同推导得出。

它回答的是：

**当前这一刻系统要解决的动作语义是什么。**

### `selector`

`selector` 根据当前 `action` 和最小任务画像，选择当前动作需要的 capability。

它回答的是：

**为了完成当前动作，需要激活哪些能力。**

### `agent`

`agent` 是执行者和消费方，例如：

- Claude agent
- iFlow agent
- Codex / Kiro 运行时

它回答的是：

**由谁来执行、如何消费这些能力。**

一句话总结：

```text
workflow
  -> 推导 current_action
  -> selector 选择 capability
  -> agent 消费 capability 并执行
```

### 4.0.1 workflow / action / agent / skill 分层图

这 4 个概念在流程中的层次如下：

```text
┌──────────────────────────────────────────────────────────────┐
│ Task Orchestration Layer                                     │
│ - workflow                                                   │
│   负责步骤拓扑和推进顺序                                     │
└──────────────────────────────┬───────────────────────────────┘
                               │ 推导
┌──────────────────────────────▼───────────────────────────────┐
│ Action Semantics Layer                                       │
│ - current_action                                             │
│   负责表达“当前在做 implement/check/finish/create-pr 哪一步” │
└──────────────────────────────┬───────────────────────────────┘
                               │ 驱动
┌──────────────────────────────▼───────────────────────────────┐
│ Capability / Skill Selection Layer                           │
│ - selector                                                   │
│ - capability / skill                                         │
│   负责表达“为了完成当前 action，需要哪些能力单元”            │
└──────────────────────────────┬───────────────────────────────┘
                               │ 供给
┌──────────────────────────────▼───────────────────────────────┐
│ Execution Layer                                              │
│ - agent                                                      │
│   负责消费当前 skill/capability，并真正执行任务              │
└──────────────────────────────────────────────────────────────┘
```

### 4.0.2 四者在流程里的作用

#### `workflow`

作用层：

- 任务编排层

核心作用：

- 定义任务步骤顺序
- 决定当前推进到哪一步

它不负责：

- 决定 skill
- 执行任务
- 生成 prompt

#### `action`

作用层：

- 动作语义层

核心作用：

- 把当前步骤转成稳定语义键
- 作为 selector 的核心输入

例如：

- `implement`
- `check`
- `finish`
- `create-pr`

#### `skill`

作用层：

- 能力层

核心作用：

- 表达一个可复用的执行能力单元
- 为 agent 提供方法、规则、提示或操作约束

在 v8 里，`skill` 与 `capability` 可以视为同一层概念。

更稳的理解是：

- `capability` 是架构层叫法
- `skill` 是平台消费时更贴近用户的叫法

#### `agent`

作用层：

- 执行层

核心作用：

- 读取当前任务状态
- 消费当前 skill / capability
- 执行当前 action
- 写回 evidence

### 4.0.3 端到端流程图

```text
workflow 定义步骤
  -> 推导 current_action
  -> selector 按 current_action 选择 skill/capability
  -> agent 读取并消费这些 skill/capability
  -> agent 执行当前 action
  -> 写回 evidence
  -> workflow 推进到下一步
```

### 4.0.4 一句话区分

- `workflow` 决定“任务怎么走”
- `action` 决定“当前在做什么”
- `skill` 决定“完成这一步要用什么能力”
- `agent` 决定“谁来执行、如何执行”

### 4.0.5 动态平台 vs 静态平台映射图

同一套 `workflow -> action -> selector -> skill -> agent` 主链，在不同平台上的消费方式不同。

```text
┌──────────────────────────────────────────────────────────────────────┐
│ Core Runtime Model                                                   │
│                                                                      │
│ workflow -> current_action -> selector -> selected skills            │
└──────────────────────────────┬───────────────────────────────────────┘
                               │
                 ┌─────────────┴─────────────┐
                 │                           │
                 ▼                           ▼
┌────────────────────────────────┐   ┌────────────────────────────────┐
│ Dynamic Platforms              │   │ Static Platforms               │
│ Claude / iFlow                 │   │ Codex / Kiro                   │
├────────────────────────────────┤   ├────────────────────────────────┤
│ workflow                       │   │ workflow                       │
│   -> task runtime读取          │   │   -> task runtime读取          │
│                                │   │                                │
│ action                         │   │ action                         │
│   -> hook按当前action选择注入  │   │   -> runtime按当前action决定   │
│                                │   │      当前激活哪些skills       │
│ selector                       │   │ selector                       │
│   -> 直接读取selector.current  │   │   -> 直接读取selector.current  │

└────────────────────────────────┘   └────────────────────────────────┘
```

### 4.0.6 平台差异的关键结论

#### 对动态平台

适用：

- Claude
- iFlow

特点：

- skill 正文在运行时按当前任务动态注入
- 不需要提前把所有能力都变成平台最终产物
- hook 是主要消费点

#### 对静态平台

适用：

- Codex
- Kiro

特点：

- skill 基础资产在安装期就已经写入平台目录
- 任务运行时不重新安装 skill
- 任务运行时只根据 `selector.current` 决定当前激活或强调哪些 skill

这可以进一步收敛为：

**安装期全量物化 + 任务期轻量选择**

### 4.0.7 为什么这样分层

这样设计的目的，是让：

- 核心模型保持统一
- 平台消费方式允许不同
- 平台差异不反向污染 selector 和 workflow

也就是说：

- `workflow / action / selector` 必须跨平台统一
- `skill 的落地形式` 可以因平台而不同
- `agent 的消费方式` 也可以因平台而不同

这正是 v8 所说的：

- 核心规则统一
- 平台只负责消费
- 不把所有平台强行拉成同一种运行方式

### 4.1 单一任务模型

任务只保留一套统一模型，不再引入新的大画像系统。

### 4.2 action-aware

规则始终围绕“当前动作是什么”来判断，而不是围绕数字 phase。

### 4.3 小步门禁

不是每个状态都做治理，只把高价值动作节点做硬化。

### 4.4 轻证据

证据只保留最关键的执行质量信息，不做过度审计。

### 4.5 平台分层

平台只分两类：

- 动态注入平台
- 静态物化平台

不继续扩张抽象层次。

```json
  "dev_type": "backend",
  "task_mode": "default",
  "package": "cli",
  "workflow": {
    "id": "default",
    "steps": [
      { "index": 1, "action": "implement" },
      { "index": 2, "action": "check" },
      { "index": 3, "action": "finish" },
      { "index": 4, "action": "create-pr" }
    ],
    "current_step": 0
  },
  "selector": {
    "current": null,
    "history": []
  },
  "action_config": {
    "check": {
      "verify_commands": []
    }
  },
  "evidence": {
    "selector_trace": {},
    "verify_result": {
      "commands": [],
      "status": "unknown"
    },
    "finish_note": "",
    "release_note": ""
  },
  "meta": {}
}
```

这里有一个明确约定：

- `workflow.steps[*].index`
  - 只是展示用步骤序号，从 `1` 开始
- `workflow.current_step`
  - 是当前步骤在 `steps` 数组中的下标，从 `0` 开始
  - `0` 表示第一步已激活，不表示未开始

这个结构体现了 v8 的 4 条收敛原则：

- `workflow`
  - 定义步骤拓扑
- `selector`
  - 定义当前动作需要哪些 capability
- `action_config`
  - 定义当前动作执行参数
- `evidence`
  - 定义动作执行结果

这也是推荐后的最终主模型，不再依赖：

- `decision_hints`
- `execution_profile`
- preset 直接定义 capability
- 静态平台任务期重装技能

## 5.2 保留字段

### `dev_type`

描述任务类型：

- `backend`
- `frontend`
- `fullstack`
- `test`
- `docs`

### `task_mode`

描述任务执行模式：

- `default`
- `tdd`
- `debug`
- `docs`

### `package`

用于 monorepo 或包级作用域控制。

### `workflow`

负责步骤拓扑，不负责规则判断。

### `selector`

只保存当前动作的 selector 快照和历史快照。

### `action_config`

只保存动作执行参数，不保存策略。

最典型的内容是：

- `check.verify_commands`

未来如果需要扩展，也只能增加同类“执行参数”，例如：

- `finish.required_fields`
- `create-pr.draft`

它不负责：

- capability 选择
- workflow 推导
- prompt 策略

扩展边界约束：

- 只允许预定义字段
- 新字段必须先进入 schema
- 平台运行时不能自由塞入临时 key
- 如需演进，使用 schema version 进行显式升级

### `evidence`

只存最小必要证据。

## 5.3 删除 decision_hints，引入最小 action_config

v8 不再保留 `decision_hints` 作为核心任务字段。

原因：

- 语义过宽，容易混入策略、提示和执行参数
- 会和 `selector.current` 形成职责重叠
- 不利于长期类型收敛和平台统一消费

v8 用最小 `action_config` 替代它。

推荐结构：

```json
{
  "selector": {
    "current": {
      "action": "check",
      "selected_capabilities": ["check"]
    }
  },
  "action_config": {
    "check": {
      "verify_commands": ["pnpm lint", "pnpm typecheck"]
    }
  },
  "evidence": {
    "verify_result": {
      "commands": ["pnpm lint", "pnpm typecheck"],
      "status": "passed"
    }
  }
}
```

这里的边界是：

- `selector` 回答“当前动作需要哪些 capability”
- `action_config` 回答“执行这个动作时要带哪些参数”
- `evidence` 回答“这个动作执行后产生了什么结果”

一句话：

**`decision_hints` 是泛化提示容器，`action_config` 是动作执行参数容器。**

迁移后的字段职责收敛为：

- `workflow`
  - 定义步骤拓扑
- `selector`
  - 定义当前动作需要的 capability
- `action_config`
  - 定义当前动作执行参数
- `evidence`
  - 定义动作执行结果

---

## 5.4 verify_commands 统一归属到 action_config.check

`check` 是 v8 质量提升的核心动作。

为了让 `check` 可执行、可解释、可测试，`verify_commands` 必须有唯一稳定归属。

v8 的结论是：

**`verify_commands` 统一放在 `action_config.check.verify_commands`。**

它不应该放在：

- `selector`
- `evidence`
- `preset`
- 平台模板内部
- LLM 临时输出结果中

推荐结构：

```json
{
  "action_config": {
    "check": {
      "verify_commands": [
        "pnpm lint",
        "pnpm typecheck"
      ]
    }
  },
  "evidence": {
    "verify_result": {
      "commands": [
        "pnpm lint",
        "pnpm typecheck"
      ],
      "status": "passed"
    }
  }
}
```

这里的边界非常明确：

- `action_config.check.verify_commands`
  - 记录 check 执行前要跑什么
- `evidence.verify_result`
  - 记录这些命令执行后的结果

### 5.4.1 verify_commands 的来源层次

最佳实践不是“每个任务都手工填命令”，而是三层生成：

1. 项目级默认 verify profile
2. 任务创建/更新时自动解析
3. 特殊任务下人工覆写

推荐优先级：

```text
Priority 1: task.action_config.check.verify_commands
Priority 2: package-scoped verify profile
Priority 3: dev_type + task_mode default profile
Priority 4: repo default profile
```

约束说明：

- `backend/frontend/fullstack/test` 任务，`verify_commands` 必须非空
- `docs` 任务允许为空，但需要在执行层明确 fallback 到 `check.jsonl` 或 completion markers
- 若为空且无 fallback，`check` 节点必须阻断并提示补齐配置

### 5.4.2 app / pc / h5 / admin 如何适配

不同项目差异不应该写死在 `check` hook 里，而应该按 `package` 解析。

推荐配置思想：

```json
{
  "verify_profiles": {
    "default": ["pnpm lint"],
    "apps/pc": ["pnpm --filter pc lint", "pnpm --filter pc typecheck"],
    "apps/h5": ["pnpm --filter h5 lint", "pnpm --filter h5 typecheck"],
    "apps/admin": ["pnpm --filter admin lint", "pnpm --filter admin typecheck", "pnpm --filter admin test"],
    "services/api": ["pnpm --filter api lint", "pnpm --filter api test"]
  }
}
```

创建任务时系统根据：

- `package`
- `dev_type`
- `task_mode`

解析出最终的 `action_config.check.verify_commands`。

monorepo 示例：

```json
{
  "package": "apps/admin",
  "action_config": {
    "check": {
      "verify_commands": [
        "pnpm --filter admin lint",
        "pnpm --filter admin typecheck"
      ]
    }
  }
}
```

### 5.4.3 LLM 的位置

LLM 可以参与建议，但不能成为 `verify_commands` 的唯一真源。

推荐职责是：

- 规则给出基础 verify commands
- LLM 提示可能遗漏的检查项
- 用户在特殊场景下决定是否采纳

一句话：

**规则保底，LLM 增强，人工只在必要时覆写。**

### 5.4.4 为什么这样设计

对产品：

- 大多数任务无需手动配置
- 不同项目默认行为仍可解释

对研发：

- `check` 的输入位置固定
- 不再依赖宽泛提示字段

对测试：

- 可以稳定验证 `check` 输入和输出
- 不会因为 LLM 波动导致回归不稳定

对平台：

- 动态平台和静态平台都消费同一份 `action_config`

## 5.5 删除 execution_profile

v8 不保留 `execution_profile`。

原因：

- 与 `dev_type + task_mode + package` 重叠
- 容易形成双真源
- 在当前产品阶段收益不高

这里遵循的是最佳实践里的一个现实原则：

**宁可少一个抽象，也不要多一个没人稳定消费的抽象。**

- `workflow.steps`
- `workflow.current_step`

共同推导。

## 7. Selector 设计

## 7.1 职责

selector 只做一件事：

**根据当前 action 和最小任务画像，选择当前动作需要的 capability 列表。**

selector 不做：

- 工作流编排
- 角色审批
- 命令执行
- prompt 拼接
- 结果放行

## 7.2 输入

```python
def select_capabilities(
    *,
    action: str,
    dev_type: str | None,
    task_mode: str,
    package: str | None,
    workflow_id: str,
    user_context: str | None = None,
) -> dict:
    ...
```

## 7.3 输出

```json
{
  "action": "check",
  "selected_capabilities": ["check", "finish-work"],
  "generated_at": "2026-03-27T12:00:00Z",
  "inputs": {
    "dev_type": "backend",
    "task_mode": "default",
    "package": "cli",
    "workflow_id": "default"
  },
  "trace": {
    "bundles": ["check-default"],
    "matched": ["check", "finish-work"],
    "excluded": [],
    "required": [],
    "final": ["check", "finish-work"]
  }
}
```

## 7.4 规则优先

v8 的基础闭环完全依赖规则。

LLM 只能是增强层，不是基础层。

调用顺序建议：

1. 先做规则匹配，得到基础 selector 结果
2. 再调用 LLM enhancer 做增补建议
3. 最终结果仍需回到规则输出的可解释框架里
4. 超时或失败时直接回退规则结果

## 8. Capability 模型

## 8.1 capability 的定义

`capability` 是平台无关的执行能力单元。

每个 capability 包含三部分：

- 元数据
- 通用正文
- 平台包装

## 8.2 真源分层

### 元数据真源

```text
.spec-first/config/capability-registry.json
```

### 规则真源

```text
.spec-first/config/selector-rules.json
```

### 通用正文真源

```text
.spec-first/capabilities/<id>.md
```

### 平台包装真源

各平台模板目录。

## 8.3 不过度设计的 registry

registry 只保留必要字段：

```json
{
  "capabilities": [
    {
      "id": "before-dev",
      "priority": 100,
      "category": "development",
      "requires": [],
      "conflicts_with": [],
      "applicable_to": {
        "actions": ["implement"],
        "dev_types": ["backend", "frontend", "fullstack"],
        "task_modes": ["default", "tdd", "debug"]
      }
    }
  ]
}
```

不提前引入复杂字段。

## 9. 轻量质量门禁

v8 不做完整治理系统，只做最值钱的质量门禁。

## 9.1 关键动作节点表

| Action | 主要目标 | 最小输入 | 最小输出 | 最小证据 | 门禁 |
|--------|----------|----------|----------|----------|------|
| `implement` | 正确开始执行 | `current_action`、`selector.current`、基础 task context | 代码修改或文档修改 | `selector_trace` | 没有 selector 结果时不允许进入执行 |
| `check` | 验证实现质量 | `action_config.check.verify_commands`、check capability | 验证结果 | `verify_result` | 对代码类任务，关键验证命令不可为空 |
| `finish` | 收口并形成交付说明 | check 结果、finish capability | finish 总结 | `finish_note` | check 未完成时不允许 finish |
| `create-pr` | 形成可审阅交付物 | finish 结果、变更摘要 | PR 内容或 PR 草稿 | `release_note` | finish_note 缺失时不允许 create-pr |

### 9.1.1 节点表的使用方式

这张表不是审批系统。

它的用途是统一三件事：

- 产品知道当前节点应产出什么
- 研发知道当前节点至少要补齐什么
- 测试知道哪些证据可以判断节点是否达标

因此 v8 只把它当作轻量执行契约，而不是重型流程引擎。

## 9.2 implement

目标：

- 给工程执行提供正确能力包
- 减少无上下文编码

要求：

- 必须有 `selector.current`
- 必须有基础 task context

## 9.3 check

目标：

- 把验证前置
- 让检查动作有稳定输出

要求：

- 必须能拿到 `action_config.check.verify_commands`
- 必须记录检查结果
- `docs` 任务允许空 verify_commands，但必须有明确 fallback

## 9.4 finish

目标：

- 保证收尾质量
- 避免“代码写完但没收口”

要求：

- 必须能读取 check 结果
- 必须输出 finish note

## 9.5 create-pr

目标：

- 保证最终交付具备最小可审阅性

要求：

- 必须具备 PR summary
- 必须具备最小 release note

## 10. 轻量 evidence 模型

v8 不做复杂 evidence 平台，只保留四类证据：

- `selector_trace`
- `verify_result`
- `finish_note`
- `release_note`

建议结构：

```json
{
  "evidence": {
    "selector_trace": {},
    "verify_result": {
      "commands": [],
      "status": "unknown"
    },
    "finish_note": "",
    "release_note": ""
  }
}
```

这已经足够支撑当前阶段的质量提升。

其中：

- `action_config.check.verify_commands`
  - 定义 check 的输入命令
- `evidence.verify_result.commands`
  - 记录本次实际执行了哪些命令
- `evidence.verify_result.status`
  - 记录执行结果状态

## 11. 多角色支持

v8 要支持不同角色，但方式是“同一数据模型，不同关注点”，不是“复杂角色系统”。

## 11.1 产品

产品主要关注：

- 当前任务目标
- 当前动作处于哪个节点
- finish 是否完成
- release note 是否齐全

## 11.2 研发

研发主要关注：

- current_action
- selected_capabilities
- action_config
- action_config.check.verify_commands

## 11.3 测试

测试主要关注：

- check 动作
- verify_result
- finish_note

## 11.4 结论

不同角色共享同一个 task.json 和 action 状态。

这已经足够满足“支持不同角色人员使用”的目标，不必再单独建一套角色治理模型。

## 12. 平台接入

## 12.1 动态注入平台

适用：

- `Claude`
- `iFlow`

消费方式：

1. 读取当前任务
2. 读取 `selector.current.selected_capabilities`
3. 读取 `action_config`
4. 读取 capability 正文
5. 拼接到现有 agent prompt

这类平台不需要物化平台技能文件。

## 12.2 静态物化平台

适用：

- `Codex`
- `Kiro`

v8 对静态平台采用“两段式物化”。

### 第一段：安装期全量物化

在 `init / update` 时完成：

- 共享 skill 目录生成
- 平台专属 skill 目录生成
- 平台 agent / hook / config 初始化

这与当前实现一致：

- Codex 在安装期写入 `.agents/skills/`、`.codex/skills/`、`.codex/agents/`、`.codex/hooks/` 和配置文件  
  参考 [codex.ts](/Users/kuang/xiaobu/spec-first/packages/cli/src/configurators/codex.ts#L12)
- Kiro 在安装期写入 `.kiro/skills/`  
  参考 [kiro.ts](/Users/kuang/xiaobu/spec-first/packages/cli/src/configurators/kiro.ts#L5)

安装期的目标是：

- 保证平台基础资产完整存在
- 避免任务运行时频繁重写大目录
- 让平台目录结构保持稳定

### 第二段：任务期轻量选择

任务运行时不重新安装平台技能目录，只消费当前任务的选择结果。

任务期的核心输入是：

- `workflow`
- `current_action`
- `selector.current`
- `action_config`

任务期可以做的事情：

- 读取当前任务选择结果
- 在 wrapper / hook / runtime 中强调当前激活的 skill
- 可选生成轻量 manifest

任务期不做的事情：

- 不重写 `.codex/skills/*`
- 不重写 `.kiro/skills/*`
- 不重新生成整个平台基础资产

### 推荐消费方式

1. 读取当前任务的 `selector.current`
2. 读取当前任务的 `action_config`
3. 平台 runtime 决定当前强调或消费哪些已安装 skill
4. 执行当前 action

推荐优先使用最简单的形式：

- 不额外生成全局 manifest
- 直接让平台 runtime 读取 `task.json`

只有在平台消费受限时，才增加只读 manifest 文件。

### Codex / Kiro 消费示例

```text
1. 任务已在 init / update 阶段完成平台技能物化
2. 任务运行时，Codex / Kiro 读取 task.json
3. runtime 解析 workflow.current_step 得到 current_action
4. runtime 读取 selector.current.selected_capabilities
5. runtime 在已安装 skill 目录中激活对应正文
6. 若当前动作是 check，则读取 action_config.check.verify_commands
7. 执行当前动作并写回 evidence
```

## 12.3 平台边界

平台层统一遵守：

- 不决定规则
- 不修改 selector 结果
- 不承担任务编排
- 不在任务运行时重装平台技能目录

## 13. LLM 的正确位置

v8 保留 LLM 增强，但只作为可选层。

### 可以做

- 根据用户上下文增补 capability
- 给出选择解释

### 不可以做

- 替代规则引擎
- 替代 gate
- 替代 evidence
- 替代平台消费逻辑

结论：

**LLM 是优化器，不是裁决器。**

## 17. 为什么这是最佳实践但不过度设计

因为它满足了最佳实践的三个核心要求：

### 统一

只有一套任务模型和一套规则模型。

### 可控

把质量提升聚焦到少数关键动作节点。

### 可扩展

以后可以加 capability、加 action、加 LLM 增强，但不需要推翻基础架构。

同时，它避免了常见的过度设计问题：

- 不引入多余抽象
- 不把未来可能需要的治理系统提前实现
- 不把角色协作复杂化
- 不把 LLM 放到基础闭环里
- 不让 preset 变成第二套 capability 策略系统

## 17.1 快速开始指南（5 分钟）

这套系统的目标不是让用户先理解所有内部概念，而是先跑通第一个任务。

推荐的新用户路径只有 5 步：

### Step 1: 初始化项目

确保项目已经完成 spec-first 初始化，并具备默认配置与平台基础资产。

```bash
spec-first init
```

如果项目已经初始化过，则执行：

```bash
spec-first update
```

### Step 2: 用 preset 创建第一个任务

新用户不要从全量参数开始，直接使用 preset。

```bash
spec-first task create "Fix admin login bug" --preset bug-debug
```

系统会自动展开为标准输入字段，例如：

- `workflow=default`
- `task_mode=debug`
- 可选 `dev_type`

### Step 3: 查看系统是怎么理解这个任务的

创建任务后，立刻用 explain 看当前 action 和 capability 选择结果。

```bash
spec-first selector explain
```

用户此时只需要理解三件事：

- 当前 action 是什么
- 当前选中了哪些 capability
- 如果需要调整，应该改 `task_mode` / `workflow` / `package`

### Step 4: 开始执行当前动作

如果当前 action 是 `implement`，agent 会读取：

- `workflow`
- `selector.current`
- 当前 task context

然后开始执行。

对用户来说，重点不是理解内部流程，而是知道：

- `workflow` 决定当前在哪一步
- `selector` 决定当前用什么能力

### Step 5: 进入 check 并查看验证结果

当任务推进到 `check` 时，系统会读取：

- `action_config.check.verify_commands`

并把结果写入：

- `evidence.verify_result`

用户最终能看到的是：

- 跑了哪些校验命令
- 是否通过
- 是否可以进入 `finish`

### 一个最小完整示例

```bash
# 1. 初始化
spec-first init

# 2. 创建任务
spec-first task create "Fix admin login bug" --preset bug-debug

# 3. 查看选择结果
spec-first selector explain

# 4. 执行任务
# agent 进入 implement -> check -> finish -> create-pr
```

### 新用户第一天只需要理解的 4 个概念

- `workflow`
  - 决定任务步骤顺序
- `action`
  - 决定当前在做什么
- `selector`
  - 决定当前激活哪些 capability
- `evidence`
  - 决定系统记录了什么结果

如果用户能完成上面这 5 步，就已经完成了第一次有效使用。

## 18. 最终结论

v8 的最终口径如下：

- `workflow` 负责步骤拓扑
- `action` 负责当前执行语义
- `selector` 负责 capability 选择
- `platform adapter` 负责消费 selector 结果
- `evidence` 负责最小质量记录
- `LLM` 负责可选增强

这是一套：

**支持不同角色理解和协作，但不过度设计的轻量最佳实践方案。**

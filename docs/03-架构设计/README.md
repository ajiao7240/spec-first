## 当前建议口径

- `workflow` 只负责步骤拓扑与推进
- `action` 是 selector 的稳定语义键
- `task profile` 收敛为 `dev_type + task_mode + package`
- `selector` 只负责产出当前 action 的 capability 快照
- `evidence` 只保留最小必要质量信息
- `runtime hook` 只负责消费 selector 结果，不负责选择
- `Claude / iFlow` 属于动态注入平台
- `Codex / Kiro` 属于静态物化平台
- `LLM` 只能作为增强层，不能成为基础闭环依赖

## 🎯 方案演进

### 为什么选择 v8.1？

**多角色加权评分**:
- 产品: 0.75
- 研发: 1.75 ⭐
- 测试: 1.25 ⭐
- 平台: 0.60
- **总分: 4.35** (v7.1: 3.25)

**核心理由**:
1. 架构更清晰（单一模型，职责明确）
2. 维护成本更低（无双模式，无 LLM 复杂度）
3. 可测试性更强（确定性输出）
4. 平台支持更完整（P1 即支持 4 平台）

## 🚀 实施建议

### P1: 核心闭环（4 周）

**交付**:
- Task 模型 + Selector 核心
- Preset + Explain 命令
- 4 平台适配（Claude/iFlow/Codex/Kiro）
- 测试覆盖 > 80%

**验收**:
- 用户可用 `--preset` 创建任务
- Selector 稳定输出 capabilities
- 4 平台正确消费
- `selector explain` 输出可读

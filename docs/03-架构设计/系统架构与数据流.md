# Spec-First 系统架构与数据流

> **版本**: 1.1
> **日期**: 2026-03-26
> **目的**: 基于当前仓库代码，说明 spec-first 的真实架构边界、任务数据模型、上下文流转与记忆体系。

## 1. 文档边界

这份文档只描述**当前代码能证明的实现**，不把历史方案稿、未来规划或尚未安装到当前仓库根目录的能力写成“已落地”。

本次核对主要基于以下代码路径：

- `.spec-first/scripts/` 与 `.spec-first/hooks/`：本仓库自用的 spec-first 运行时
- `.spec-first/scripts/common/`：任务、上下文、工作流模板、session context 等核心逻辑
- `.spec-first/scripts/multi_agent/`：多 Agent worktree 启动、状态和 PR 流程
- `packages/cli/src/templates/spec-first/`：CLI 安装到用户项目时生成的 spec-first 模板
- `packages/cli/src/templates/claude/hooks/`：Claude 平台的 Python hook 实现
- `packages/cli/src/templates/opencode/plugins/`：OpenCode 平台的插件实现

以下内容**不作为当前实现依据**：

- `docs/01-需求分析/架构升级/90-历史稿/` 下的历史技术方案
- 文档中曾提到、但当前仓库根 `.spec-first/` 下并不存在的脚本名

## 2. 当前实现的两层结构

当前代码实际上分成两层，不应混写：

### 2.1 项目自用运行时层

这是本仓库根目录下真正存在并正在使用的能力：

- 任务与上下文脚本：`.spec-first/scripts/task.py`
- 当前任务切换：`.spec-first/scripts/current_task.py`
- 会话上下文输出：`.spec-first/scripts/get_context.py`
- 会话记录：`.spec-first/scripts/add_session.py`
- 多 Agent worktree 流程：`.spec-first/scripts/multi_agent/start.py`
- 轻量 runtime hook：`.spec-first/hooks/task-context.sh`
- 轻量 runtime hook：`.spec-first/hooks/progress-sync.sh`
- 轻量 runtime hook：`.spec-first/hooks/stop-guard.sh`

这一层负责：

- 创建和维护任务目录
- 生成 `task.json` 与阶段上下文 JSONL
- 写入 `.spec-first/.current-task`
- 输出开发者、Git、任务、journal 的当前上下文
- 记录 session 到 workspace journal
- 启动 worktree 多 Agent 流程

### 2.2 平台模板层

这一层位于 `packages/cli/src/templates/`，是 CLI 生成给用户项目的模板，不等于“当前仓库根目录已经存在同名脚本”。

已在代码中确认的模板能力包括：

- `packages/cli/src/templates/claude/hooks/session-start.py`
- `packages/cli/src/templates/claude/hooks/inject-subagent-context.py`
- `packages/cli/src/templates/claude/hooks/ralph-loop.py`
- `packages/cli/src/templates/claude/agents/dispatch.md`
- `packages/cli/src/templates/opencode/plugins/inject-subagent-context.js`

这意味着：

- `inject-subagent-context.py`、`ralph-loop.py` 是**Claude 模板层**能力
- `dispatch` 当前是**agent 定义/提示模板**，不是本仓库根目录中的 `dispatch.py`
- 平台差异由模板适配，不是所有平台共用一套同名脚本

## 3. 任务生命周期与数据流

### 3.1 真实主流程

```text
需求进入
  -> task.py create
  -> 写 prd.md
  -> task.py init-context
  -> task.py add-context（按需补充）
  -> current_task.py switch 或 task.py start
  -> 实现 / 检查 / 收尾
  -> add_session.py
  -> archive 或 create_pr.py
```


这张图对应到当前 3.2 的层次关系是：

- `task creation / plan` 对应当前图里的 `task.py create`、`task.py init-context` 及其上游规划动作
- `skill selector` 是需要新增的一层，应该位于“任务环境已确定”与“hook 注入”之间
- `runtime hook / plugin` 对应当前图里的“模板层 hook / plugin”
- `phase agent` 对应当前图里的 “Agent 执行 implement / check / finish”
- `add_session / archive / PR` 仍然属于后置记录和收尾层，不参与选择

因此，按该方案定义，**自动选择的决策点在 `skill selector` 层，自动选择的生效点在 `runtime hook / plugin` 层**。

这张图表达的配置化原则是：

- 新增 skill 主要通过 `skills-registry.json` 注册，而不是改 selector 代码
- bundle 选择策略主要通过 `selector-rules.json` 调整，而不是写死在 hook 中
- `skill selector` 只做“匹配、排序、裁决、产出结果”，不直接负责 prompt 注入
- `runtime hook / plugin` 只消费 selector 输出结果并完成注入

#### 3.3.2 `skills-registry.json` 最小 schema

最小目标是让新 skill 接入时，只需要补 metadata 并注册，就能进入候选集。

```json
{
  "version": 1,
  "skills": {
    "react-patterns": {
      "path": ".agents/skills/react-patterns/SKILL.md",
      "category": "framework",
      "language": ["typescript", "javascript"],
      "framework": ["react"],
      "surface": ["frontend", "web"],
      "phases": ["implement", "check"],
      "modes": ["default", "tdd"],
      "priority": 85,
      "requires": [],
      "conflicts_with": []
    }
  }
}
```

最小字段约束：

| 字段 | 必填 | 说明 |
|------|------|------|
| `version` | 是 | registry schema 版本 |
| `skills` | 是 | skill id 到 metadata 的映射 |
| `path` | 是 | `SKILL.md` 路径 |
| `category` | 是 | `base/language/framework/task_mode` |
| `language` | 否 | 适用语言列表 |
| `framework` | 否 | 适用框架列表 |
| `surface` | 否 | 适用端类型列表 |
| `phases` | 否 | 适用 phase 列表 |
| `modes` | 否 | 适用 task_mode 列表 |
| `priority` | 是 | 同类匹配时的优先级整数 |
| `requires` | 否 | 依赖 skill id 列表 |
| `conflicts_with` | 否 | 冲突 skill id 列表 |

建议最小规则：

1. `skills` 的 key 使用稳定的 skill id
2. `priority` 为整数，默认可约定为 `50`
3. `category` 必须为固定枚举值
4. `requires` / `conflicts_with` 只能引用已注册的 skill id

#### 3.3.3 `selector-rules.json` 最小 schema

最小目标是让 selector 的路由和裁决规则可配置，而不是散落在代码里。

```json
{
  "version": 1,
  "defaults": {
    "max_skills_per_phase": 4,
    "category_order": ["base", "language", "framework", "task_mode"]
  },
  "phase_rules": {
    "implement": {
      "allowed_categories": ["base", "language", "framework", "task_mode"],
      "max_skills": 4
    },
    "check": {
      "allowed_categories": ["base", "language", "framework", "task_mode"],
      "max_skills": 4
    },
    "debug": {
      "allowed_categories": ["base", "language", "framework", "task_mode"],
      "max_skills": 3
    },
    "finish": {
      "allowed_categories": ["base", "task_mode"],
      "max_skills": 2
    }
  },
  "match_weights": {
    "surface": 30,
    "language": 30,
    "framework": 20,
    "mode": 20
  },
  "overrides": {
    "force_include": [],
    "force_exclude": []
  }
}
```

最小字段约束：

| 字段 | 必填 | 说明 |
|------|------|------|
| `version` | 是 | rules schema 版本 |
| `defaults.max_skills_per_phase` | 是 | 默认 bundle 上限 |
| `defaults.category_order` | 是 | 默认类别排序 |
| `phase_rules` | 是 | 每个 phase 的允许类别和数量限制 |
| `match_weights` | 是 | 匹配打分权重 |
| `overrides.force_include` | 否 | 强制加入的 skill id |
| `overrides.force_exclude` | 否 | 强制排除的 skill id |

建议最小规则：

1. selector 先按 `phase_rules` 过滤，再按 `match_weights + priority` 排序
2. `force_include` 最后追加，但仍应做 `conflicts_with` 校验
3. `force_exclude` 优先级最高
4. 若无命中，至少回退到一个 `base` skill

#### 3.3.4 配置化接入收益

这样设计后，后续集成新 skill 的主路径会变成：

1. 新增 skill 文件
2. 在 `skills-registry.json` 注册 metadata
3. 如有需要，在 `selector-rules.json` 调整路由规则
4. 无需修改 `skill selector` 核心算法

### 3.4 各节点的真实职责

#### 节点 1: `task.py create`

代码入口：

- `.spec-first/scripts/task.py`
- `.spec-first/scripts/common/task_store.py::cmd_create`

职责：

- 创建任务目录 `.spec-first/tasks/{MM}-{DD}-{slug}/`
- 生成 `task.json`
- 根据 `workflow_type` 生成 `next_action`
- 生成默认 `decision_hints`
- 建立 `parent` / `children` 关系（如果传 `--parent`）

#### 节点 2: `task.py init-context`

代码入口：

- `.spec-first/scripts/common/task_context.py::cmd_init_context`

职责：

- 生成 `implement.jsonl`
- 生成 `check.jsonl`
- 生成 `debug.jsonl`
- 同步更新 `task.json.dev_type`
- monorepo 场景下解析并写回 `task.json.package`

当前实现**不会**在 `init-context` 时生成 `plan.jsonl`。

#### 节点 3: `task.py add-context`

代码入口：

- `.spec-first/scripts/common/task_context.py::cmd_add_context`

职责：

- 将额外文件或目录追加到指定 JSONL
- 支持 `implement` / `check` / `debug` 简写
- 目录条目会写入 `"type": "directory"`

#### 节点 4: 当前任务激活

代码入口：

- `.spec-first/scripts/current_task.py::cmd_switch`
- `.spec-first/scripts/task.py::cmd_start`

职责：

- 将当前任务写入 `.spec-first/.current-task`
- 供后续 hook、session context、multi-agent 流程读取

当前推荐入口是 `current_task.py switch`，`task.py start` 是底层原语。

#### 节点 5: 多 Agent 启动

代码入口：

- `.spec-first/scripts/multi_agent/start.py`

职责：

- 创建或复用 worktree
- 复制必要环境文件
- 在 worktree 中设置 `.current-task`
- 启动平台 Agent
- 写入 registry

这里真正存在的是 `start.py`、`status.py`、`create_pr.py`、`cleanup.py`，而不是仓库根目录下的 `dispatch.py`。

#### 节点 6: 平台注入与 gate

这一部分是**模板层能力**：

- Claude 会使用 `packages/cli/src/templates/claude/hooks/session-start.py`
- Claude 会使用 `packages/cli/src/templates/claude/hooks/inject-subagent-context.py`
- Claude 会使用 `packages/cli/src/templates/claude/hooks/ralph-loop.py`

已从代码确认的行为：

- `inject-subagent-context.py` 会读取 `.current-task`、`task.json`、阶段 JSONL、`prd.md`
- 它会把 `finish` 映射为 `check` agent，并按 `next_action` 推进 `current_phase`
- `ralph-loop.py` 会优先读 `task.json.decision_hints.check.verify_commands`
- 若没有 verify commands，再退回到 `check.jsonl` 的 completion marker 机制

#### 节点 7: session 记录

代码入口：

- `.spec-first/scripts/add_session.py`

职责：

- 追加 session 到当前 `journal-N.md`
- 超过 `max_journal_lines` 时自动轮转
- 更新开发者工作区 `index.md`
- 默认自动提交 `.spec-first/workspace` 与 `.spec-first/tasks` 的变更

- `workflow_type`、`decision_hints`、`evidence`、`worktree_path` 是当前实现中的真实字段

### 4.2 `status` 的真实语义

从当前代码可确认或直接观察到的状态包括：

- `planning`
- `in_progress`
- `completed`
- `rejected`

说明：

- `task.py create` 初始写入 `planning`
- `multi_agent/start.py` 会把任务推进到 `in_progress`
- `create_pr.py` 与 `archive` 流程会把任务推进到 `completed`
- `rejected` 由 plan/multi-agent 流程使用

因此，把 `status` 固定描述为 `planning|implementing|checking|completed|archived` 不准确。

### 4.3 JSONL 条目真实格式

`task_context.py` 当前写入的是这种结构：

```json
{ "file": ".spec-first/workflow.md", "reason": "Project workflow and conventions" }
```

目录条目则可能是：

```json
{ "file": ".spec-first/spec/guides/", "type": "directory", "reason": "Shared guides" }
```

关键点：

- 字段名是 `file`，不是 `path`
- 当前实现没有自动写 `addedAt`
- `init-context` 默认只生成 `implement/check/debug` 三类 JSONL

## 5. 记忆体系

### 5.1 三类记忆载体

当前代码能证实的记忆体系如下：

#### 会话级记忆

- 载体：当前对话上下文
- 生命周期：单次会话
- 仓库内没有单独文件直接保存这层内容

#### 任务级记忆

- 载体：`.spec-first/tasks/{task}/`
- 关键文件：
  - `task.json`
  - `prd.md`
  - `implement.jsonl`
  - `check.jsonl`
  - `debug.jsonl`

#### 项目级记忆

- 载体：`.spec-first/workspace/` 与 `.spec-first/spec/`
- 关键文件：
  - `.spec-first/workspace/{developer}/journal-N.md`
  - `.spec-first/workspace/{developer}/index.md`
  - `.spec-first/spec/**/*.md`

### 5.2 `add_session.py` 的真实写入过程

已从代码确认：

1. 读取开发者工作区和当前活跃 journal
2. 计算新 session 内容行数
3. 若超出 `config.yaml.max_journal_lines`，创建新的 `journal-(N+1).md`
4. 追加 session 内容
5. 更新开发者 `index.md` 中的 auto section
6. 默认自动提交 `.spec-first/workspace` 与 `.spec-first/tasks`

### 5.3 `get_context.py` 的真实读取输出

`get_context.py` 实际委托给：

- `.spec-first/scripts/common/git_context.py`
- `.spec-first/scripts/common/session_context.py`

默认输出包含：

- developer
- Git branch / dirty state / recent commits
- 当前任务
- 活跃任务列表
- 当前 journal 文件与行数

它不是只读 “当前任务状态”，而是一个聚合的 session context 视图。

## 8. 结论

当前 spec-first 的真实架构可以概括为：

1. **任务契约层**：`task.json + prd.md + implement/check/debug.jsonl`
2. **项目自用运行时层**：`.spec-first/scripts/` 与 `.spec-first/hooks/`
3. **模板发布层**：`packages/cli/src/templates/` 中的跨平台适配
4. **项目记忆层**：`.spec-first/workspace/` 与 `.spec-first/spec/`

最重要的事实边界是：

- 本仓库根目录负责“任务、上下文、session、worktree”
- 平台注入与 gate 机制有相当一部分位于模板层，而不是当前根目录直接运行的脚本
- 架构文档必须区分“仓库现状”“模板能力”“历史设计稿”，否则就会失真

## 问题 1: Phase 初始值冲突 🔴 高危

### 问题描述

**V5 方案** (待办技术方案-v5-混合方案.md:445):
```python
initial_phase = 2  # implement phase
```

**现有代码**:
- `task_store.py:199`: `"current_phase": 0` (任务创建时从 0 开始)
- `workflow_templates.py:44`: `{"phase": 1, "action": "implement"}` (implement 是 phase 1)
- `inject-subagent-context.py:149`: 只查找 `phase_num > current_phase` 的下一个 phase

### 影响分析

如果按 V5 方案设置 `current_phase = 2`:
1. ❌ 跳过 phase 1 (implement)
2. ❌ 直接进入 phase 2 (check)
3. ❌ gate/loop 语义错位
4. ❌ 第一次 implement 永远不会被触发

### 正确做法

```python
initial_phase = 0  # 待处理状态
# Hook 会自动推进到 phase 1 (implement)
```

## 问题 2: 平台路径错误 🔴 高危

### 问题描述

**V5 方案** (待办技术方案-v5-混合方案.md:691):
```text
Codex: .codex/agents/skills/
Kiro: .kiro/skills/
```

### 正确路径

```text

Codex 专属技能: .codex/skills/
Codex agents: .codex/agents/
Kiro: .kiro/skills/
```

## 问题 3: Codex/Kiro 动态注入机制缺失 🔴 高危

### 问题描述

**V5 方案声称**: "平台无关设计 - task.json 存储 selector_result"

**现实问题**:
- Codex 只在 SessionStart 注入 start skill (`session-start.py:172`)
- 没有机制将 `task.json` 的 `selected_skills` 动态映射到 Codex/Kiro prompt
- Codex/Kiro 不支持 Python Hook，无法像 Claude/iFlow 那样动态注入

### 影响分析

1. ❌ "平台无关"前提不成立
2. ❌ Codex/Kiro 无法使用 phase-aware skills
3. ❌ 只有 Claude/iFlow 能工作

### 需要补充的设计

```text
方案 A: 在 SessionStart 时预生成所有 phase 的 skills
方案 B: 使用配置文件让 Codex/Kiro 读取 task.json
方案 C: 放弃 Codex/Kiro 的 phase-aware 支持
```

```python
def generate_all_phase_skills():
    """在 SessionStart 时预生成所有 phase 的 skills"""
    task_data = read_task_json()
    
    for phase in [1, 2, 3, 4]:
        selector_result = select_skills(
            execution_profile=task_data['execution_profile'],
            phase=phase,
            registry_path='.spec-first/config/skills-registry.json'
        )
        
        # 生成到平台目录
        for skill_id in selector_result['selected_skills']:
            copy_skill_to_platform(skill_id, phase)
```

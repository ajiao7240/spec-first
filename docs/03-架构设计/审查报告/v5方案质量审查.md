## 三、架构设计质量

### 3.1 规则引擎设计 ⭐⭐⭐⭐⭐

**优势**:
1. **完整的配置化架构**
   - `skills-registry.json`: 技能清单 + metadata
   - `selector-rules.json`: 规则配置 + 策略
   - 支持 priority、requires、conflicts_with

2. **确定性保证**
   - 规则引擎输出可预测
   - selector_trace 完整记录决策过程
   - 支持回溯和审计

3. **扩展性**
   - 新增 skill 只需修改 JSON
   - 规则可独立调整
   - 无需修改代码

**潜在风险**:
- 规则复杂度增长后的维护成本
- 建议: 提供规则验证工具

---

### 3.2 LLM 增强层设计 ⭐⭐⭐⭐⭐

**优势**:
1. **生产级稳定性**
   - 5 秒超时机制
   - 缓存机制减少 API 调用
   - Fallback 到规则引擎
   - LLM 结果验证

2. **边界清晰**
   - 规则引擎处理 80% 标准场景
   - LLM 仅处理 20% 特殊场景
   - 不允许 LLM 完全替换规则结果

**实施建议**:
- P1 先实现规则引擎，验证价值
- P2 再添加 LLM 增强
- 监控 LLM 调用频率和成功率

---

### 3.3 可追溯性设计 ⭐⭐⭐⭐⭐

**核心机制**:
```python
task_data['selector_result'] = {
    "selected_skills": [...],
    "selector_trace": {
        "bundle": [...],
        "matched": [...],
        "final": [...]
    }
}
```

**优势**:
- 每个 phase 的 selector_result 存储在 phase_history
- 可以回溯任意时刻的 skill 选择
- 支持审计和问题排查

**对比 V4**:
- V4: 动态计算，无法追溯
- V5: 预计算+存储，完全可追溯

## 九、源码模板层 vs 运行时副本层

### 9.1 层次划分 ⚠️ 关键架构

V5 方案需要严格区分两个层次：

```text
┌─────────────────────────────────────────────────────────┐
│ 源码模板层 (packages/cli/src/templates/)               │
│ - 只读，由 npm 包分发                                    │
│ - spec-first init 时复制到用户项目                      │
│ - 包含: hooks/, commands/, skills-registry.json 等      │
└─────────────────────────────────────────────────────────┘
                         │ spec-first init
                         ↓ (复制)
┌─────────────────────────────────────────────────────────┐
│ 运行时副本层 (用户项目/.spec-first/)                    │
│ - 可读写，用户可修改                                     │
│ - 运行时实际执行的代码                                   │
│ - 包含: scripts/, skills/, tasks/, .journal/ 等         │
└─────────────────────────────────────────────────────────┘
```

---

### 9.2 V5 方案中的层次问题

**当前方案描述混淆**:

| 文件/目录 | 方案中的描述 | 实际应该在 |
|-----------|-------------|-----------|
| `skills-registry.json` | `packages/cli/src/templates/spec-first/` | ❌ 源码模板层 |
| `selector-rules.json` | `packages/cli/src/templates/spec-first/` | ❌ 源码模板层 |
| `skill_selector.py` | `packages/cli/src/templates/spec-first/scripts/common/` | ❌ 源码模板层 |
| `.spec-first/skills/` | 运行时读取 | ✅ 运行时副本层 |

**问题**:
- 源码模板层的文件在 `spec-first init` 后就是只读的
- 用户无法修改 `skills-registry.json` 和 `selector-rules.json`
- 这与"配置化"目标矛盾

---

### 9.3 正确的层次设计

**方案 A: 默认配置 + 用户覆盖**

```text
源码模板层:
  packages/cli/src/templates/spec-first/
    ├── skills-registry.default.json    (默认配置)
    └── selector-rules.default.json     (默认规则)

运行时副本层:
  .spec-first/
    ├── skills-registry.json            (用户可修改)
    ├── selector-rules.json             (用户可修改)
    └── scripts/common/skill_selector.py (读取运行时配置)
```

**加载逻辑**:
```python
def _load_registry(registry_path: str):
    # 1. 尝试加载用户配置
    user_config = Path('.spec-first/skills-registry.json')
    if user_config.exists():
        return json.loads(user_config.read_text())
    
    # 2. Fallback 到默认配置
    default_config = Path('.spec-first/skills-registry.default.json')
    return json.loads(default_config.read_text())
```

---

**方案 B: 三层配置（推荐）**

```text
源码模板层:
  packages/cli/src/templates/spec-first/
    └── defaults/
        ├── skills-registry.json        (内置默认)
        └── selector-rules.json         (内置默认)

运行时副本层:
  .spec-first/
    ├── config/
    │   ├── skills-registry.json        (项目级配置)
    │   └── selector-rules.json         (项目级配置)
    └── tasks/<task-id>/
        └── selector-overrides.json     (任务级覆盖)
```

**优先级**: 任务级 > 项目级 > 默认

---

### 9.4 修正建议

**V5 方案需要修正**:

1. **文件位置调整**
   - `skills-registry.json` → `.spec-first/config/skills-registry.json`
   - `selector-rules.json` → `.spec-first/config/selector-rules.json`
   - 源码模板层只保留 `defaults/` 目录

2. **skill_selector.py 读取路径**
   ```python
   # 错误（当前方案）
   registry_path = '.spec-first/skills-registry.json'
   
   # 正确
   registry_path = '.spec-first/config/skills-registry.json'
   ```

3. **spec-first init 行为**
   - 复制 `defaults/` → `.spec-first/config/`
   - 用户可修改 `.spec-first/config/` 下的文件

---

### 9.5 影响评估

**如果不修正**:
- ❌ 用户无法自定义 skills
- ❌ 用户无法调整规则
- ❌ "配置化"目标无法实现
- ❌ 每次更新 npm 包会覆盖用户配置

**修正后**:
- ✅ 用户可自定义 skills-registry.json
- ✅ 用户可调整 selector-rules.json
- ✅ 真正的配置化架构

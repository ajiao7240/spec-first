## 问题3: 平台模式混淆

### 问题描述

文档声称"平台无关"，但实际混淆了两种模式：
- **动态注入模式**（Claude/iFlow）- Hook读取task.json动态注入
- **静态生成模式**（Codex/Kiro）- SessionStart预生成到目录

### 解决方案

**明确区分两种模式，各自独立设计**

#### 模式A: 动态注入（Claude/iFlow）

```python
# inject-subagent-context.py
def inject_phase_skills():
    """从task.json读取selector_result，动态注入"""
    task_data = read_task_json()
    selector_result = task_data.get('selector_result', {})
    skill_ids = selector_result.get('selected_skills', [])

    # 从.spec-first/skills/读取内容
    for skill_id in skill_ids:
        skill_path = f".spec-first/skills/{skill_id}.md"
        inject_content(skill_path)
```

#### 模式B: 静态生成（Codex/Kiro）

```python
# session-start.py
def generate_all_phase_skills():
    """预生成所有phase的skills到平台目录"""
    task_data = read_task_json()
    execution_profile = task_data['execution_profile']

    # 动态获取phases
    phases = [a['phase'] for a in task_data['next_action'] if 'phase' in a]

    for phase in phases:
        result = select_skills(execution_profile, phase)
        for skill_id in result['selected_skills']:
            copy_skill_to_platform(skill_id, phase)
```

#### 文档修改

**删除**: "平台无关设计"的表述

**改为**: "双模式架构 - 动态注入 vs 静态生成"

## 问题4: Skills资产source of truth

### 问题描述

三层混淆：
1. `skills-registry.json` - 元数据
2. `.spec-first/skills/*.md` - 内容
3. 平台目录 - 实际使用

没有定义生成链路。

### 解决方案

**明确单向数据流**

```text
Source of Truth:
  .spec-first/skills/{skill-id}.md (内容)
  .spec-first/config/skills-registry.json (元数据)
    ↓
  skill selector (选择)
    ↓
  task.json.selector_result (存储)
    ↓
  模式A: Hook动态读取 .spec-first/skills/
  模式B: SessionStart复制到平台目录
```

#### 生成链路

**spec-first init时**:
```bash
# 1. 复制默认配置
cp templates/defaults/skills-registry.json → .spec-first/config/
cp templates/defaults/skills/*.md → .spec-first/skills/

# 2. 用户可修改
.spec-first/config/skills-registry.json (可编辑)
.spec-first/skills/*.md (可编辑)
```

**task create时**:
```python
# 调用selector，存储结果
selector_result = select_skills(profile, phase=1)
task_data['selector_result'] = selector_result
```

**平台使用时**:
- 模式A: 直接读 `.spec-first/skills/`
- 模式B: 复制到平台目录

#### 文档修改

**新增章节**: "Skills资产管理"，明确单向数据流

## 问题5: execution_profile推导回退现有能力

### 问题描述

v5引入更弱的推导器：
- 只用根目录 `package.json` / `pyproject.toml`
- 忽略现有的 `dev_type` 和 `package` 字段
- 忽略 `project-detector.ts` 的工程识别

### 解决方案

**复用现有能力，不重新发明**

#### 方案A: 复用task_context.py的dev_type

```python
def _infer_execution_profile(workflow_type, task_dir):
    """复用现有能力"""
    # 1. 读取task.json的dev_type（如果已有）
    task_json = Path(task_dir) / "task.json"
    if task_json.exists():
        task_data = json.loads(task_json.read_text())
        dev_type = task_data.get('dev_type')
        if dev_type:
            return _map_dev_type_to_profile(dev_type)

    # 2. Fallback: 调用现有的detect_dev_type
    from .task_context import detect_dev_type
    dev_type = detect_dev_type(task_dir)
    return _map_dev_type_to_profile(dev_type)

def _map_dev_type_to_profile(dev_type: str) -> dict:
    """映射现有dev_type到execution_profile"""
    mapping = {
        "frontend": {"surface": "frontend", "language": "typescript"},
        "backend": {"surface": "backend", "language": "python"},
        "fullstack": {"surface": "fullstack", "language": "typescript"},
    }
    profile = mapping.get(dev_type, {"surface": "backend", "language": "python"})
    profile.setdefault("task_mode", "default")
    return profile
```

#### 方案B: 直接使用dev_type，不引入execution_profile

**更激进的方案**: 不引入新字段，直接用现有的 `dev_type` + `task_mode`

```python

# selector使用

    """直接用现有字段"""

    surface = _map_dev_type_to_surface(dev_type)
    language = _map_dev_type_to_language(dev_type)

```

#### 文档修改

**推荐方案B**: 删除execution_profile，直接用 `dev_type` + `task_mode`

**理由**:
- 不引入新概念
- 复用现有字段
- 减少维护成本

## 问题7: Hook注入与现有系统集成

### 问题描述

现有是三层拼装：
- jsonl任务上下文
- decision_hints
- 平台命令模板

v5只描述skills注入，没说如何集成。

### 解决方案

**Skills作为增强层，不替换现有层**

```python
# inject-subagent-context.py (伪代码)
def build_context():
    """三层拼装 + skills增强"""
    context = []

    # Layer 1: 现有jsonl上下文（保留）
    context.append(read_jsonl_context(phase))

    # Layer 2: decision_hints（保留）
    context.append(read_decision_hints())

    # Layer 3: 平台命令模板（保留）
    context.append(read_platform_commands())

    # Layer 4: Skills增强（新增）
    selector_result = task_data.get('selector_result', {})
    for skill_id in selector_result.get('selected_skills', []):
        context.append(read_skill(skill_id))

    return "\n\n".join(context)
```

#### 文档修改

**明确**: Skills是增强层，不替换现有上下文系统

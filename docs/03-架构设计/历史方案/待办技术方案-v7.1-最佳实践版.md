# Spec-First 待办技术方案 v7.1 - 最佳实践版

> **版本**: 7.1 (基于 v7.0 优化)
> **日期**: 2026-03-27
> **状态**: 推荐实施方案
> **定位**: 在 v7.0 基础上，融合多角色评审结论的最佳实践版本

---

## 变更摘要

v7.1 在保留 v7.0 核心设计的基础上，针对以下问题进行优化：

| 问题 | v7.0 | v7.1 优化 |
|------|------|-----------|
| **易用性** | 需要理解 registry/rules/selector | 引入 Preset Profiles，零配置启动 |
| **实施风险** | 4 平台并行开发 | P1 只做动态注入平台（Claude/iFlow） |
| **LLM 定位** | "可选增强"，定位模糊 | "智能助手"，建议而非决策 |
| **可理解性** | trace 存在但不可视 | 新增 `selector explain` 命令 |

**核心原则**：在工程严谨性和产品可用性之间找到平衡。

## 1. 设计目标

v7.1 的目标是建立一套**既工程正确又用户友好**的架构：

### 1.1 保留 v7.0 的优势

- ✅ Action-aware 核心设计
- ✅ 单一真源原则
- ✅ 关注点分离
- ✅ 完整的测试策略

### 1.2 新增的优化

- 🎯 **约定优于配置**：提供智能默认值，80% 场景零配置
- 🎯 **渐进式复杂度**：简单场景简单用，复杂场景可扩展
- 🎯 **决策透明化**：关键决策可视化，用户可理解
- 🎯 **分阶段验证**：先做好一类平台，再扩展第二类

## 2. 核心架构

### 2.1 分层设计

```text
┌─────────────────────────────────────────┐
│         User Interface Layer            │
│  ┌─────────────┐    ┌─────────────┐    │
│  │ Simple Mode │    │ Expert Mode │    │
│  │  (Presets)  │    │ (Full Config)│   │
│  └─────────────┘    └─────────────┘    │
└─────────────────────────────────────────┘
              ↓
┌─────────────────────────────────────────┐
│      Action-aware Selector Core         │
│  ┌──────────┐  ┌──────────┐  ┌───────┐ │
│  │ Registry │→ │  Rules   │→ │ Trace │ │
│  └──────────┘  └──────────┘  └───────┘ │
└─────────────────────────────────────────┘
              ↓
┌─────────────────────────────────────────┐
│         LLM Assistant Layer             │
│     (Suggestions, not Decisions)        │
└─────────────────────────────────────────┘
              ↓
┌─────────────────────────────────────────┐
│      Platform Adapters (P1)             │
│  ┌─────────────┐    ┌─────────────┐    │
│  │   Claude    │    │    iFlow    │    │
│  │  (Dynamic)  │    │  (Dynamic)  │    │
│  └─────────────┘    └─────────────┘    │
└─────────────────────────────────────────┘
```

### 2.2 关键改进点

#### 改进 1: 双模式接口

**Simple Mode（简单模式）**：
- 使用 preset profiles
- 零配置启动
- 适合 80% 的常见场景

**Expert Mode（专家模式）**：
- 完整配置能力
- 自定义 registry/rules
- 适合 20% 的高级场景

#### 改进 2: LLM 智能助手

**定位变化**：
- ❌ v7.0: "可选增强层"（定位模糊）
- ✅ v7.1: "智能助手"（建议 + 解释）

**核心原则**：
- 不参与核心决策（保证可测试性）
- 提供建议和解释（提升用户体验）
- 最终决策权在用户（保持透明性）

#### 改进 3: 决策可视化

新增 `selector explain` 命令，让用户理解系统决策。

## 3. Preset Profiles 设计

### 3.1 核心概念

Preset Profile 是预定义的任务配置模板，封装了常见开发场景的最佳实践。

### 3.2 默认 Presets

```json
{
  "presets": {
    "quick-fix": {
      "workflow": "quick-fix",
      "capabilities": ["before-dev", "check"],
      "description": "快速修复 bug，跳过复杂验证",
      "use_cases": ["修复 typo", "小改动", "紧急热修复"]
    },
    "feature-dev": {
      "workflow": "default",
      "capabilities": ["before-dev", "check", "finish-work"],
      "description": "标准功能开发流程",
      "use_cases": ["新功能", "功能增强", "重构"]
    },
    "bug-debug": {
      "workflow": "default",
      "task_mode": "debug",
      "capabilities": ["systematic-debugging", "check"],
      "description": "系统化调试流程",
      "use_cases": ["复杂 bug", "性能问题", "间歇性错误"]
    },
    "docs-update": {
      "workflow": "docs-only",
      "capabilities": ["check"],
      "description": "仅更新文档",
      "use_cases": ["README 更新", "API 文档", "注释补充"]
    }
  }
}
```

### 3.3 用户体验

**简单模式（推荐）**：
```bash
# 使用 preset，零配置
$ spec-first task create "Fix login bug" --preset bug-debug
✓ Task created with preset: bug-debug
✓ Workflow: default (implement → check → finish)
✓ Capabilities: systematic-debugging, check
```

**专家模式（高级用户）**：
```bash
# 完整配置
$ spec-first task create "Fix login bug" \
  --dev-type backend \
  --task-mode debug \
  --workflow default \
  --capabilities systematic-debugging,check
```

### 3.4 配置文件

文件位置：`.spec-first/config/presets.json`

用户可以自定义 presets：
```json
{
  "presets": {
    "my-workflow": {
      "workflow": "custom",
      "capabilities": ["my-capability"],
      "description": "我的自定义工作流"
    }
  }
}
```

## 4. LLM 智能助手设计

### 4.1 定位变化

| 维度 | v7.0 | v7.1 |
|------|------|------|
| **角色** | 可选增强层 | 智能助手 |
| **职责** | 调整 selector 结果 | 提供建议和解释 |
| **决策权** | 可能影响核心决策 | 用户保留最终决策权 |
| **依赖性** | P2 才考虑 | P2 实施，但不影响 P1 |

### 4.2 工作模式

```text
User Input
    ↓
Selector (规则驱动)
    ↓
LLM Assistant (可选)
    ├─ 分析上下文
    ├─ 生成建议
    └─ 提供解释
    ↓
User Decision
    ├─ 接受建议 → 应用
    └─ 拒绝建议 → 使用原结果
```

### 4.3 示例交互

```bash
$ spec-first task create "Fix auth bug" --preset bug-debug

✓ Task created: fix-auth-bug
✓ Workflow: default (implement → check → finish)
✓ Capabilities: systematic-debugging, check

💡 AI Suggestion:
   Based on "auth bug", you might also want:
   • Add capability: security-audit
   • Enable mode: review-heavy

   Reason: Auth bugs often require security review

   Apply suggestions? [y/N] _
```

### 4.4 技术约束

- **超时机制**：3 秒内无响应，使用规则结果
- **降级策略**：LLM 不可用时，系统正常工作
- **隐私保护**：不发送敏感代码内容
- **可关闭**：用户可以完全禁用 LLM 助手

## 5. 决策可视化

### 5.1 新增命令

```bash
$ spec-first selector explain
```

### 5.2 输出示例

```
Current Action: implement
Selected Capabilities: before-dev, systematic-debugging

Decision Trace:
  1. ✓ Action bundle loaded: [before-dev]
  2. ✓ Mode override applied: debug → [systematic-debugging]
  3. ✓ Conflict check: No conflicts
  4. ✓ Category quota: development(2/3), verification(0/2)
  5. ✓ Final selection: [before-dev, systematic-debugging]

Why these capabilities?
  • before-dev
    - Required for all implement actions
    - Ensures context is loaded before coding

  • systematic-debugging
    - Added by task_mode=debug
    - Provides structured debugging approach

To customize:
  $ spec-first task update --capabilities <list>
  $ spec-first task update --task-mode <mode>
```

### 5.3 集成到工作流

```bash
# 创建任务时自动显示
$ spec-first task create "Fix bug" --preset bug-debug --explain

# 任务推进时显示变化
$ spec-first task advance --explain
```

## 6. 平台实施优先级

### 6.1 分阶段策略

| 阶段 | 平台 | 类型 | 优先级原因 |
|------|------|------|-----------|
| **P1** | Claude | 动态注入 | 用户群最大，反馈最快 |
| **P1** | iFlow | 动态注入 | 国内市场重要 |
| **P2** | Codex | 静态物化 | 验证后扩展 |
| **P2** | Kiro | 静态物化 | 验证后扩展 |

### 6.2 收益分析

**P1 只做动态注入平台**：
- ✅ 减少 50% 初期开发工作量
- ✅ 先验证核心价值
- ✅ 避免过早优化
- ✅ 更快获得用户反馈

**P2 再做静态物化平台**：
- ✅ 基于 P1 经验优化设计
- ✅ 复用已验证的 selector 逻辑
- ✅ 降低技术风险

## 10. 与 v7.0 的对比

### 10.1 保留的设计

| 设计 | 说明 |
|------|------|
| Action-aware 核心 | 以 action 为核心，而非 phase |
| 单一真源原则 | 每类信息只有一个权威来源 |
| 关注点分离 | Selector/Workflow/Platform 职责清晰 |
| Trace 机制 | 完整的决策审计链 |

### 10.2 新增的优化

| 优化 | 价值 |
|------|------|
| Preset Profiles | 降低学习曲线，80% 场景零配置 |
| 双模式接口 | 简单场景简单用，复杂场景可扩展 |

### 10.3 删除的复杂度

| 删除项 | 原因 |

## 11. 最佳实践总结

### 11.1 采纳的业界实践

1. **约定优于配置**（Ruby on Rails）
   - 提供智能默认值
   - 减少必填配置项

2. **渐进式复杂度**（Vue.js）
   - 简单场景简单用
   - 复杂场景可扩展

3. **可观测性设计**（OpenTelemetry）
   - 完整的 trace 机制
   - 决策过程可审计

4. **分阶段交付**（Agile/Lean）
   - MVP 验证核心价值
   - 迭代增强功能

5. **关注点分离**（SOLID 原则）
   - 单一职责
   - 依赖倒置

### 11.2 关键创新点

1. **Preset Profiles**
   - 场景化配置
   - 降低 AI 编码工具学习曲线

2. **LLM 智能助手定位**
   - 建议 + 解释，而非自动化决策
   - 平衡 AI 能力和用户控制

3. **Action-aware 设计**
   - 比 phase 编号更稳定
   - 符合开发者心智模型

# Spec-First 待办技术方案 v7 - 动作感知统一架构

> **版本**: 7.0
> **日期**: 2026-03-27
> **状态**: 终局设计
> **定位**: 基于当前代码结构重写的主方案，替代 v5 / v6

---

## 1. 文档目标

本方案重新定义 spec-first 的待办能力架构，不沿用 v5 / v6 的核心前提。

v7 的目标不是在旧设计上继续打补丁，而是建立一套新的、一致的、可扩展的主设计，用于指导后续实现。

本方案明确采用以下前提：

- 不要求向下兼容旧方案文档
- 不要求提供迁移方案
- selector 以 `action` 为核心输入，而不是以 phase 编号为核心输入
- 只将 `Claude / iFlow / Codex / Kiro` 写到可实施级
- `LLM 增强` 作为终局可选层存在，但 P1 架构闭环不依赖它

---

## 2. 设计摘要

v7 采用 `Action-aware Core`。

系统围绕三个核心事实构建：

1. **任务是状态机**
   任务有固定工作流步骤，步骤包含稳定的 `action` 语义，如 `implement`、`check`、`review`、`finish`。

2. **selector 只做一件事**
   selector 根据当前任务状态，特别是 `current_action`，选择当前动作需要的 capability bundle。

3. **平台只负责消费**
   各平台不参与选择，不复写规则，只消费 selector 已经产出的结果。

核心分层如下：

```text
Task State
  - workflow
  - profile
  - current_action
  - selector snapshot
    ↓
Action-aware Selector
  - rules
  - capability registry
  - optional llm enhancer
    ↓
Platform Adapters
  - 动态注入: Claude / iFlow
  - 静态物化: Codex / Kiro
```

## 3. 设计原则

### 3.1 单一真源

每类信息只能有一个真源：

- **任务状态真源**: `task.json`
- **selector 规则真源**: `.spec-first/config/selector-rules.json`
- **capability 元数据真源**: `.spec-first/config/capability-registry.json`
- **capability 通用内容真源**: `.spec-first/capabilities/<id>.md`
- **平台适配包装真源**: 平台模板目录

平台目录中的技能文件不是业务真源，只是产物。

### 3.2 action 优先于 phase

phase 只负责工作流中的排序与推进。

真正稳定的语义键是 `action`：

- `implement`
- `check`
- `review`
- `finish`
- `debug`
- `research`

selector 不允许以 phase 编号直接驱动规则判断。

### 3.3 selector 只选择，不执行

selector 负责：

- 读取任务状态
- 解析规则
- 产出 capability 集合
- 记录 trace

selector 不负责：

- 修改工作流
- 推进任务步骤
- 运行命令
- 注入 prompt

### 3.4 平台分层

平台按能力分为两类：

- **动态注入平台**: Claude / iFlow
- **静态物化平台**: Codex / Kiro

两类平台共享 selector 结果，但消费方式不同。

### 3.5 P1 可闭环

P1 必须在没有 LLM 的情况下完成以下闭环：

- 创建任务
- 推导当前 action
- 选择 capability
- 平台消费 selector 结果
- 通过测试验证

---

## 4. 非目标

v7 明确不做以下事情：

- 不在 selector 内实现完整任务编排器
- 不让平台自行决定 selection 规则
- 不把 `execution_profile` 作为新的复杂画像系统引入
- 不让 LLM 成为基础依赖
- 不把 capability 选择和 prompt 拼接耦合在一起
- 不尝试一次性覆盖所有平台到同等深度

---

## 5. 核心对象模型

### 5.1 任务状态模型

v7 不再引入 `execution_profile`。

任务画像收敛为最小必要字段：

- `dev_type`
- `task_mode`
- `package`
- `workflow`
- `current_action`

建议的 `task.json` 结构如下：

```json
{
  "id": "add-selector-v7",
  "title": "实现 v7 selector",
  "status": "planning",
  "dev_type": "backend",
  "task_mode": "default",
  "package": "cli",
  "workflow": {
    "id": "default",
    "steps": [
      { "index": 1, "action": "implement" },
      { "index": 2, "action": "check", "loop": { "max": 5, "gate": "lint_and_typecheck_pass" } },
      { "index": 3, "action": "finish" },
      { "index": 4, "action": "create-pr" }
    ],
    "current_step": 0
  },
  "selector": {
    "current": null,
    "history": []
  },
  "decision_hints": {},
  "evidence": null,
  "meta": {}
}
```

### 5.1.1 字段说明

#### `dev_type`

描述任务开发类型，沿用当前项目认知：

- `backend`
- `frontend`
- `fullstack`
- `test`
- `docs`

#### `task_mode`

描述任务执行策略：

- `default`
- `tdd`
- `debug`
- `review-heavy`
- `docs`

它是规则选择输入，而不是 workflow 类型的别名。

#### `workflow`

`workflow` 是任务状态机。

它包含：

- `id`: workflow 模板标识
- `steps`: 稳定步骤列表
- `current_step`: 当前步骤游标

#### `current_action`

`current_action` 不单独存储为真源字段。

它由 `workflow.steps[workflow.current_step]` 推导得到。

如果需要缓存，可缓存为只读冗余字段，但逻辑真源仍是 `workflow.current_step + workflow.steps`。

### 5.1.2 为什么删除 execution_profile

原因很简单：

- 它和 `dev_type / task_mode / package` 的职责高度重叠
- 容易演化成“画像很完整但没有稳定消费方”的死数据
- 会让任务模型出现双真源

v7 采用更小、更稳定的输入集合，优先保证可读性和可验证性。

---

### 5.2 selector 结果模型

selector 只产出当前动作的快照，以及历史快照。

```json
{
  "selector": {
    "current": {
      "action": "implement",
      "selected_capabilities": ["before-dev", "systematic-debugging"],
      "generated_at": "2026-03-27T12:00:00Z",
      "inputs": {
        "dev_type": "backend",
        "task_mode": "debug",
        "package": "cli",
        "workflow_id": "default"
      },
      "trace": {
        "matched": ["before-dev", "systematic-debugging", "check"],
        "excluded": ["check"],
        "required": [],
        "bundles": ["debug-implement"],
        "final": ["before-dev", "systematic-debugging"]
      }
    },
    "history": []
  }
}
```

### 5.2.1 规则

- `selector.current` 永远表示当前 action 的最终结果
- `selector.history` 只存已完成 action 的快照
- history 中每一项必须包含当时的 inputs 与 trace
- 平台只读取 `selector.current`

### 5.2.2 不预计算全部 action

v7 不采用“创建任务时预计算所有 phase/action”的设计。

原因：

- action 的输入会随 `task_mode`、`package`、用户上下文而变化
- 预计算全部 action 会制造大量过期数据
- 真正需要消费的只有“当前 action 的当前结果”

因此 v7 只在以下时机重算：

- 任务创建完成后，初始化当前 action 的 selector 快照
- 任务步骤推进后
- `task_mode` 明确变化后
- 用户显式触发 `selector refresh`

## 6. capability 资产模型

### 6.1 capability 与平台技能的关系

v7 统一使用 `capability` 作为平台无关概念。

一个 capability 包含：

- 元数据
- 通用指令内容
- 平台适配包装

关系如下：

```text
capability
  ├─ metadata           -> registry
  ├─ common content     -> .spec-first/capabilities/<id>.md
  └─ platform wrappers  -> template adapters
```

`Codex skill`、`Kiro skill`、`Claude injected section` 都只是 capability 的不同消费形态。

### 6.2 capability registry

文件：

```text
.spec-first/config/capability-registry.json
```

建议结构：

```json
{
  "capabilities": [
    {
      "id": "before-dev",
      "kind": "instruction",
      "priority": 100,
      "category": "development",
      "requires": [],
      "conflicts_with": [],
      "applicable_to": {
        "actions": ["implement"],
        "dev_types": ["backend", "frontend", "fullstack"],
        "task_modes": ["default", "tdd", "debug"]
      }
    },
    {
      "id": "check",
      "kind": "instruction",
      "priority": 100,
      "category": "verification",
      "requires": [],
      "conflicts_with": [],
      "applicable_to": {
        "actions": ["check", "finish"]
      }
    }
  ]
}
```

### 6.2.1 registry 必须满足

- 每个 capability 必须有唯一 `id`
- `actions` 是必填项
- `category` 必须显式存在，避免 quota 无法执行
- `requires` 与 `conflicts_with` 只能引用已存在 capability

### 6.3 selector rules

文件：

```text
.spec-first/config/selector-rules.json
```

建议结构：

```json
{
  "action_bundles": {
    "implement": ["before-dev"],
    "check": ["check"],
    "finish": ["finish-work"]
  },
  "mode_overrides": {
    "tdd": {
      "implement": ["test-driven-development"]
    },
    "debug": {
      "implement": ["systematic-debugging"]
    }
  },
  "category_quotas": {
    "verification": 2,
    "development": 3
  },
  "conflict_resolution": "priority"
}
```

### 6.3.1 规则职责

- `registry` 定义 capability 是什么
- `rules` 定义什么时候优先选什么

二者不混写。

### 6.4 capability 内容真源

文件：

```text
.spec-first/capabilities/<id>.md
```

这是 capability 的平台无关正文。

例如：

```markdown
# before-dev

Read the workflow and task context first.

- Inspect relevant specs
- Confirm package scope
- Avoid coding before context is clear
```

这个文件不携带平台 front matter，也不直接写成某个平台的专用 skill 格式。

## 7. Action-aware Selector 设计

### 7.1 输入

selector 的最小输入如下：

- `action`
- `dev_type`
- `task_mode`
- `package`
- `workflow_id`
- 可选 `user_context`

函数签名建议：

```python
def select_capabilities(
    *,
    action: str,
    dev_type: str | None,
    task_mode: str,
    package: str | None,
    workflow_id: str,
    registry_path: str,
    rules_path: str,
    user_context: str | None = None,
) -> dict:
    ...
```

### 7.2 基础算法

```text
1. 读取 registry
2. 读取 rules
3. 加载 action bundle
4. 过滤 applicable_to
5. 处理 requires
6. 处理 conflicts_with
7. 应用 category quota
8. 产出 current snapshot
9. 可选进入 LLM enhancer
```

### 7.3 trace 设计

trace 必须服务于调试和审计。

建议包含：

- `bundles`
- `matched`
- `excluded`
- `required`
- `final`
- `llm_adjusted`
- `llm_reason`

没有 trace 的 selector 不算完成。

### 7.4 LLM 增强层

LLM 增强是可选层，只能作用于规则结果之上。

允许做的事：

- 增补 capability
- 降低部分 capability 优先级
- 生成解释

不允许做的事：

- 跳过冲突校验
- 删除必须依赖
- 改写 workflow
- 直接生成平台产物

设计要求：

- 没有 `user_context` 时不调用
- 超时必须回退到规则结果
- 结果必须进入 trace
- 可以完全关闭

## 8. 工作流与任务推进

### 8.1 工作流定义

workflow 仍然存在，但它只负责步骤序列，不负责 capability 决策。

建议保留最小 workflow 集合：

- `default`
- `quick-fix`
- `docs-only`
- `review`

每个 workflow 都由 `steps[].action` 组成。

示例：

```json
{
  "id": "default",
  "steps": [
    { "index": 1, "action": "implement" },
    { "index": 2, "action": "check" },
    { "index": 3, "action": "finish" },
    { "index": 4, "action": "create-pr" }
  ]
}
```

### 8.2 推进规则

任务推进由任务状态机负责，不由 selector 负责。

推进动作：

- `advance`
- `set-action`
- `refresh-selector`

行为如下：

### `advance`

- 将 `selector.current` 追加到 `selector.history`
- `workflow.current_step += 1`
- 计算新的 `current_action`
- 重算 selector.current

### `set-action`

用于显式切换，如 debug / review 插入场景。

它只能修改工作流游标或工作流步骤，不直接操作 selector 结果。

### `refresh-selector`

不推进 action，只重算当前 action 快照。

## 9. 平台接入设计

### 9.1 Claude / iFlow：动态注入平台

这两类平台继续使用 hook 注入。

### 9.1.1 消费契约

hook 只做以下事情：

1. 读取当前任务
2. 获取 `selector.current.selected_capabilities`
3. 读取 capability 正文
4. 拼接到对应 action 的 prompt section

hook 不做：

- selector 规则判断
- workflow 推进决策
- capability 元数据解析

### 9.1.2 prompt 组装策略

每个 agent prompt 由三段构成：

```text
A. 平台固定 prompt 壳
B. 任务上下文（jsonl / prd / decision_hints）
C. capability 注入段
```

顺序固定：

```text
平台固定壳
  → 任务上下文
  → capability 指令
  → 原始用户 prompt
```

原因：

- 先建立平台与任务边界
- 再注入当前动作能力
- 最后保留用户具体意图

### 9.2 Codex / Kiro：静态物化平台

这两类平台不做运行时动态注入。

它们消费的是“物化后的平台技能”。

### 9.2.1 物化原则

selector 不直接把 capability 内容注入平台。

相反，系统为静态平台生成稳定技能文件：

- `.codex/skills/generated-<action>.md` 或等价目录
- `.kiro/skills/generated-<action>/SKILL.md`

这些文件由 capability 正文和平台包装模板编译生成。

### 9.2.2 静态平台的消费模型

静态平台只读取当前 action 对应的 generated skill。

因此需要一个“物化层”：

```text
selector.current
  -> materializer
  -> platform skill artifact
```

### 9.2.3 为什么不直接复制 capability 文件

因为平台技能格式并不等价于 capability 正文：

- Codex 可能需要额外说明块
- Kiro 需要 `SKILL.md` 目录结构
- 平台可能需要 action-specific 标题、说明或 usage 片段

因此 v7 明确区分：

- capability content
- platform wrapper
- generated artifact

## 10. 运行时命令与模块边界

### 10.1 运行时模块

建议新增或重构以下运行时模块：

```text
.spec-first/scripts/common/workflow_state.py
.spec-first/scripts/common/task_profile.py
.spec-first/scripts/common/capability_registry.py
.spec-first/scripts/common/action_selector.py
.spec-first/scripts/common/selector_trace.py
.spec-first/scripts/common/platform_materializer.py
```

### 职责划分

#### `workflow_state.py`

- 读取 workflow
- 解析 current_action
- 推进步骤

#### `task_profile.py`

- 解析 `dev_type / task_mode / package`
- 校验 profile 基础字段

#### `capability_registry.py`

- 加载 registry
- 校验 capability schema

#### `action_selector.py`

- 执行 selection
- 输出 snapshot

#### `selector_trace.py`

- 构建 trace
- 提供 debug 输出

#### `platform_materializer.py`

- 将 selector 结果编译为静态平台技能产物

### 10.2 task.py 接口变化

建议在 `task.py` 中新增：

- `task.py action current`
- `task.py action advance`
- `task.py action set <action>`
- `task.py selector refresh`
- `task.py selector show`

这些命令是 v7 的用户侧显式入口。

## 13. 测试设计

### 13.1 单元测试

必须覆盖：

- workflow state 解析
- current_action 推导
- registry schema 校验
- selector conflict / requires / quota
- materializer 输出

### 13.2 集成测试

必须覆盖：

- task create -> selector current 初始化
- action advance -> history 追加 + current 重算
- Claude/iFlow hook 读取 selector.current
- Codex/Kiro generated skill 输出正确

### 13.3 回归测试

必须覆盖：

- 无 capability 时的空结果
- 非法 registry 配置
- 冲突 capability
- quota 截断
- LLM enhancer timeout fallback

### 13.4 验收标准

v7 通过的最低标准：

1. 当前 action 可以稳定得到 capability 结果
2. selector trace 可读可审计
3. Claude/iFlow 不自行做选择
4. Codex/Kiro 能消费物化产物
5. 关闭 LLM 时系统仍完整工作

---

## 14. 与旧方案的关系

v7 不再以 `execution_profile + phase-aware 预计算` 为核心。

v7 明确替换为：

- `最小任务画像`
- `action-aware selector`
- `单当前快照 + 历史追溯`
- `动态注入 / 静态物化` 双平台模型

这意味着 v7 的核心价值不是“画像更完整”，而是：

- 模型更小
- 职责更清晰
- 平台边界更明确
- P1 更容易做成高质量闭环

---

## 15. 最终结论

v7 的最终架构判断如下：

- 任务系统的稳定语义键应该是 `action`
- selector 应该是独立的规则层，而不是编排器
- capability 应该成为平台无关资产
- 平台接入必须拆成动态注入与静态物化两类
- LLM 只能作为增强层，不能成为基础闭环依赖

这套设计比 v5 / v6 更适合作为主方案，因为它解决了之前最核心的五个问题：

1. 去掉双真源任务模型
2. 去掉 phase 编号主导的规则选择
3. 去掉“平台无关”的伪抽象
4. 建立 capability 真源与平台产物的边界
5. 建立可验证、可裁剪、可分阶段落地的终局架构

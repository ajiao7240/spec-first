# 前端设计

指导创建独特的生产级前端界面，避免通用的人工智能美学。这项技能涵盖了整个生命周期：检测存在的内容、规划设计、有意构建并进行视觉验证。

## 权限层次结构

该技能中的每条规则都是默认规则，而不是强制规则。

1. **现有的设计系统/代码库模式**——最高优先级，始终受到尊重
2. **用户的明确指示** -- 覆盖技能默认值
3. **技能默认值**——适用于全新工作或当用户要求设计指导时

当使用具有既定模式的现有代码库时，请遵循这些模式。当用户指定的方向与默认值相矛盾时，请遵循用户。

## 工作流程
```
Detect context -> Plan the design -> Build -> Verify visually
```

## 第 0 层：上下文检测

在进行任何设计工作之前，请检查现有设计信号的代码库。这决定了该技能的固执己见指导的适用程度。

### 寻找什么

- **设计标记/CSS变量**：`--color-*`、`--spacing-*`、`--font-*`自定义属性、主题文件
- **组件库**：shadcn/ui、Material UI、Chakra、Ant Design、Radix 或特定于项目的组件目录
- **CSS 框架**：`tailwind.config.*`、`styled-components` 主题、Bootstrap 导入、具有一致命名的 CSS 模块
- **版式**：HTML/CSS 中的字体导入、`@font-face` 声明、Google Fonts 链接
- **调色板**：定义的色阶、品牌颜色文件、设计令牌导出
- **动画库**：Framer Motion、GSAP、anime.js、Motion One、Vue Transition 导入
- **间距/布局模式**：一致的间距比例使用、网格系统、布局组件

使用平台的本机文件搜索和内容搜索工具（例如 Claude Code 中的 Glob/Grep）来扫描这些信号。不要使用 shell 命令进行例行文件探索。

### 模式分类

根据检测到的信号，选择一种模式：- **现有系统**（跨多个类别的 4 个以上信号）：遵循它。技能的审美观点（版式、颜色、动作）屈服于既定的系统。结构指导（构图、副本、可访问性、验证）仍然适用。
- **部分系统**（1-3个信号）：遵循现有的；仅对未检测到约定的区域应用技能默认值。例如，如果配置了Tailwind但不存在组件库，请按照Tailwind令牌并应用组件结构的技能指导。
- **Greenfield**（未检测到信号）：适用完整技能指导。
- **不明确**（信号矛盾或不清楚）：在继续之前询问用户。

### 询问用户

当上下文不明确时，请使用平台的阻塞问题工具（Claude Code 中的 `AskUserQuestion`、Codex 中的 `request_user_input`、Gemini 中的 `ask_user`。如果没有可用的提问工具，则采用“部分”模式并保守地进行。

示例问题：“我发现[检测到的信号]。我应该遵循您现有的设计模式还是创建一些独特的东西？”

## 第 1 层：预建规划

在编写代码之前，先写三个简短的语句。这些创建了一致性，并为用户提供了在编写代码之前重定向的检查点。

1. **视觉论文**——一句话描述心情、物质和能量
   - Greenfield 示例：“干净的编辑感觉、大量空白、衬线标题、柔和的大地色调”或“密集数据转发仪表板、等宽字体、深色表面层次结构”
   - 现有代码库：描述“现有”美学以及新作品将如何扩展它2. **内容计划** -- 页面上显示的内容以及顺序
   - 登陆页面：英雄、支持、详细信息、CTA
   - 应用程序：主要工作区、导航、辅助上下文
   - 组件：它有什么状态，它传达什么

3. **交互计划** -- 2-3个改变感觉的具体动作想法
   - 不是“添加动画”，而是“英雄加载时交错淡入、各部分之间滚动时的视差、卡片悬停时放大”
   - 在现有代码库中，仅描述使用现有运动库添加的交互

## 第 2 层：设计指导核心

这些原则适用于所有上下文类型。每个都根据权限层次结构屈服于现有的设计系统和用户指令。

### 版式

- 选择独特、有特色的字体。避免常见的嫌疑人（Inter、Roboto、Arial、系统默认值），除非现有代码库使用它们。
- 最多两种字体，没有明确的理由需要更多。将显示/标题字体与正文字体配对。
- *在第 0 层检测到时，屈服于现有字体选择。*

### 颜色和主题

- 使用 CSS 变量致力于打造一个有凝聚力的调色板。带有鲜明强调的主色调胜过胆小、分布均匀的调色板。
- 无紫白偏差，无暗模式偏差。根据上下文在浅色和深色之间变化。
- 默认情况下使用一种强调色，除非产品已经具有多色系统。
- *检测到时屈服于现有的颜色标记。*

### 成分- 从组合开始，而不是组件。将第一个视口视为海报，而不是文档。
- 在添加镶边（边框、阴影、卡片）之前使用空白、对齐、缩放、裁剪和对比度。
- 默认为无卡布局。当卡片用作用户交互的容器（可点击项目、可拖动单元、可选择选项）时，允许使用卡片。如果删除卡片样式不会影响理解，那么它就不应该是卡片。
- *所有组合规则均为默认。用户可以覆盖它们。*

### 运动

- 为视觉主导的工作提供 2-3 个有意的动作：一个进入序列、一个滚动链接或深度效果、一个悬停/显示过渡。
- 使用项目现有的动画库（如果存在）。
- 当没有找到现有库时，使用框架条件默认值：
  - **CSS动画**作为通用基线
  - React 项目的 **Framer Motion**
  - **Vue Transition / Motion One** 适用于 Vue 项目
  - Svelte 项目的 **Svelte 过渡**
- 快速录制时的动作应该是明显的，在移动设备上应该流畅，并且在整个页面上保持一致。如果纯粹是装饰性的，请移除。

### 辅助功能

- 默认情况下语义 HTML：`nav`、`main`、`section`、`article`、`button` - 不是所有内容的 div。
- 颜色对比度符合 WCAG AA 最低要求。
- 将状态集中在所有交互元素上。
- 如果做得好的话，可访问性和美观性不会紧张。

### 图像

- 当需要图像时，与抽象渐变或虚假 3D 对象相比，更喜欢真实或逼真的摄影。
- 选择或生成具有稳定色调区域的图像以进行文本叠加。
- 如果环境中存在图像生成工具，请使用它们来创建适合上下文的视觉效果，而不是占位符库存。

## 上下文模块选择适合正在构建的模块。在现有应用程序中工作时，无论功能是什么，都默认使用模块 C。

### 模块 A：登陆页面和营销（新建）

**默认部分顺序：**
1. 英雄——品牌/产品、承诺、CTA、一个主导视觉
2. 支持——一项具体的功能、提议或证明点
3. 细节——氛围、工作流程、产品深度或故事
4. 最终 CTA——转化、开始、访问或联系

**英雄规则（默认）：**
- 一种组合，而不是仪表板。全出血图像或主视觉平面。
- 品牌第一，标题第二，正文第三，CTA 第四。
- 保持文本栏狭窄并固定在图像的安静区域。
- 无明确理由的总共不超过 6 个部分。
- 一个 H1 标题。首屏上方有一个主要 CTA。

**复制：**
- 让标题有意义。支持文案通常是一小句。
- 用产品语言撰写，而不是设计评论。 UI 中没有提示语言或 AI 注释。
- 每个部分都有一项工作：解释、证明、深化或转换。
- 每一句话都应该占有一席之地。默认为较少副本，而不是更多副本。

### 模块 B：应用程序和仪表板（新建）

**默认模式：**
- 平静的表面层次结构，强大的排版和间距，很少的颜色，密集但可读的信息，最少的镶边。
- 围绕以下方面进行组织：主要工作区、导航、次要上下文/检查器、针对操作或状态的明确强调。
- 仅当卡片是交互时才显示卡片（可点击项目、可拖动单元、可选择选项）。如果面板可以变得简单布局而不失去意义，则删除卡片处理。**复制（实用，而非营销）：**
- 优先考虑方向、地位和行动，而不是承诺、情绪或品牌声音。
- 章节标题应说明该区域是什么或用户可以在那里做什么。好：“计划状态”、“搜索指标”。坏处：“释放你的潜力”。
- 如果一个句子可以出现在主页英雄中，请重写它，直到听起来像产品 UI。
- Litmus：如果操作员只扫描标题、标签和数字，他们能立即理解页面吗？

### 模块 C：组件和功能（现有应用程序中的默认设置）

要添加到现有应用程序：

- 匹配现有的视觉语言。这个模块是关于制作一些属于自己的东西，而不是一些引人注目的东西。
- 从周围的代码继承间距比例、边框半径、颜色标记和版式。
- 注重交互质量：清晰的状态（默认、悬停、活动、禁用、加载、错误）、状态之间的平滑过渡、明显的可供性。
- 一个新组件不应引入新的设计系统。如果现有应用程序使用 4px 边框半径，请勿添加 8px 的组件。

## 硬规则和反模式

### 默认反对（可覆盖）

这些都是自以为是的技能。用户可以覆盖其中任何一个。- 通用 SaaS 卡网格作为第一印象
- 白底紫配色方案，暗模式偏差
- 在新工作中过度使用字体（Inter、Roboto、Arial、Space Grotesk、系统默认值）
- 英雄部分充斥着统计数据、时间表、药丸簇、徽标云
- 用不同的词语重复相同情绪陈述的部分
- 没有叙事目的的轮播
- 多种相互竞争的强调色
- 装饰渐变或抽象背景代表真实的视觉内容
- 听起来像是设计评论的文案（“体验无缝集成”）
- 分屏英雄，其中文本位于图像繁忙的一侧

### 始终避免（优质地板）

这些都是真正的质量缺陷，用户不会愿意。

- 提示语言或人工智能评论泄漏到用户界面中
- 对比度不高——图像或背景上的文字无法辨认
- 没有可见焦点状态的交互元素
- 当存在正确的 HTML 元素时，语义 div 汤

## 石蕊检查

在进行目视验证之前进行快速自我检查。并非所有检查都适用于所有情况 - 对哪些内容相关进行判断。

- 品牌或产品在第一个屏幕上是否清晰可见？
- 有没有一个强大的视觉锚点？
- 仅扫描标题就能看懂页面吗？
- 每个部门都有一份工作吗？
- 在使用卡的地方真的有必要吗？
- 运动是否改善了层次或氛围，或者只是存在？
- 如果所有装饰性阴影都被移除，设计会不会感觉很高级？
- 文案听起来是否像产品，而不是提示？
- 新作品是否符合现有的设计体系？ （模块C）

## 目视验证实施后，目视验证。这是一次健全性检查，而不是像素完美的审查。一过。如果存在明显的问题，请解决它。如果看起来很牢固，请继续。

### 工具偏好级联

使用第一个可用选项：

1. **现有项目浏览器工具** - 如果 Playwright、Puppeteer、Cypress 或类似工具已在项目的依赖项中，请使用它。不要仅仅为了验证而引入新的依赖项。
2. **浏览器 MCP 工具** - 如果代理环境中提供浏览器自动化工具（例如 claude-in-chrome），请使用它们。
3. **代理浏览器 CLI** -- 如果没有其他可用的，则这是默认设置。加载`agent-browser`技能以获取安装和使用说明。
4. **心理审查** - 如果无法通过浏览器访问（无头 CI，无安装权限），则应用石蕊检查作为自我审查，并注意跳过视觉验证。

### 评估什么

- 输出结果是否与预构建计划中的视觉主题相符？
- 是否存在明显的视觉问题（布局损坏、文本不可读、图像丢失）？
- 它看起来是否像预期的上下文模块（登陆页面感觉像登陆页面，仪表板感觉像仪表板，组件适合其周围环境）？

### 范围控制

一次迭代。截取屏幕截图，根据试金石检查进行评估，修复任何明显的问题，然后继续。将屏幕截图包含在可交付成果中（公关描述、对话输出等）。

对于单次之外的迭代细化（多轮屏幕截图-评估-修复），请参阅 `spec-first:design:design-iterator` 代理。

## 创意能量

这项技能提供了结构，但目标是避免人工智能失误的独特工作——而不是公式化的输出。对于未开发的工作，致力于大胆的美学方向。考虑基调：极简主义、极繁主义、复古未来主义、有机/自然、奢华/精致、俏皮、社论、野兽派、装饰艺术、柔和/柔和、工业——或者发明一些适合背景的东西。有无穷无尽的味道。使用这些作为灵感，但设计一个适合项目的设计。

问：是什么让这件事令人难忘？人们会记住的一件事是什么？

将实现的复杂性与审美愿景相匹配。极繁主义设计需要带有大量动画和效果的复杂代码。极简主义设计需要克制、精确，并仔细注意间距、排版和微妙的细节。优雅来自于良好地执行愿景，而不是来自于强度。

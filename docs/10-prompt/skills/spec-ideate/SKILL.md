# 产生改进想法

**注意：当前年份是 2026 年。** 在确定构思文档的日期和检查最近的构思工件时使用此选项。

`spec:ideate` 位于 `spec:brainstorm` 之前。

- `spec:ideate` 回答：“最值得探索的最强想法是什么？”
- `spec:brainstorm` 回答：“一个选定的想法到底意味着什么？”
- `spec:plan`回答：“应该如何建造？”

此工作流程在 `docs/ideation/` 中生成排名的构思工件。它**不**产生需求、计划或代码。

## 交互方式

如果可用，请使用平台的阻塞问题工具（Claude Code 中的 `AskUserQuestion`、Codex 中的 `request_user_input`、Gemini 中的 `ask_user`。否则，在聊天中显示编号选项并等待用户回复后再继续。

一次问一个问题。当存在自然选项时，更喜欢简洁的单选选项。

## 焦点提示

<focus_hint> #$ARGUMENTS </focus_hint>

将任何提供的参数解释为可选上下文。它可能是：

- 诸如 `DX improvements` 之类的概念
- 诸如 `plugins/spec-first/skills/` 的路径
- 约束，例如 `low-complexity quick wins`
- 音量提示，例如 `top 3`、`100 ideas` 或 `raise the bar`

如果没有提供论据，则继续进行开放式构思。

## 核心原则1. **构思之前先进行基础** - 首先扫描实际的代码库。不要生成脱离存​​储库的抽象产品建议。
2. **判断前发散** - 在评估任何个人想法之前生成完整的想法集。
3. **使用对抗性过滤** - 质量机制是明确拒绝并给出理由，而不是乐观排名。
4. **保留原来的提示机制** - 产生许多想法，批评整个列表，然后仅详细解释幸存者。不要让额外的过程掩盖了这种模式。
5. **使用代理多样性来改进候选池** - 并行子代理是更丰富的想法生成和批评的支持机制，而不是核心工作流程本身。
6. **尽早保留工件** - 在呈现结果之前编写构思文档，以便工作不会受到干扰。
7. **将行动转化为头脑风暴** - 构想确定有希望的方向； `spec:brainstorm` 足够精确地定义所选的一个以进行规划。

## 执行流程

### 第 0 阶段：简历和范围

#### 0.1 检查最近的构思工作

在 `docs/ideation/` 中查找过去 30 天内创建的构思文档。

在以下情况下将先前的构思文档视为相关：
- 主题符合要求的焦点
- 路径或子系统与请求的焦点重叠
- 该请求是开放式的，并且有一个明显的近期开放构思文档
- 基于问题的状态匹配：当当前参数表明问题跟踪者意图时，不要提出恢复非问题构想，反之亦然 - 将这些视为不同的主题

如果存在相关文档，询问是否：
1. 继续
2. 重新开始如果继续：
- 阅读文档
- 总结已经探索过的内容
- 保留以前的想法状态和会话日志条目
- 更新现有文件而不是创建重复文件

#### 0.2 解释焦点和音量

从论证中推断出三件事：

- **焦点上下文** - 概念、路径、约束或开放式
- **卷覆盖** - 任何改变候选者或幸存者计数的提示
- **问题跟踪器意图** - 用户是否希望问题/错误数据作为输入源

当参数的主要意图是关于分析问题模式时，问题跟踪器意图会触发：`bugs`、`github issues`、`open issues`、`issue patterns`、`what users are reporting`、`bug reports`、`issue themes`。

不要触发仅提及错误作为焦点的参数：`bug in auth`、`fix the login issue`、`the signup bug` - 这些是焦点提示，而不是分析问题跟踪器的请求。

组合时（例如，`top 3 bugs in authentication`）：首先检测问题跟踪器意图，其次检测音量覆盖，其余部分是焦点提示。焦点缩小了哪些问题是重要的；音量覆盖控制幸存者数量。

默认音量：
- 每个构思子代理生成大约 7-8 个想法（跨代理产生 30-40 个原始想法，重复数据删除后约 20-30 个）
- 保留前5-7名幸存者

遵守明确的覆盖，例如：
- `top 3`
- `100 ideas`
- `go deep`
- `raise the bar`

使用合理的解释而不是形式化的解析。

### 第 1 阶段：代码库扫描

在产生想法之前，收集代码库上下文。

在 **前台** 并行运行代理（不要使用后台调度 - 在继续之前需要结果）：

1. **快速上下文扫描** — 使用此提示分派通用子代理：> 读取项目的 AGENTS.md（或仅作为兼容性回退的 CLAUDE.md，如果两者都不存在，则读取 README.md），然后使用本机文件搜索/glob 工具发现顶级目录布局（例如，Claude 代码中具有模式 `*` 或 `*/*` 的 `Glob`。返回简洁的摘要（30 行以下），内容包括：
   > - 项目形态（语言、框架、顶级目录布局）
   > - 值得注意的模式或惯例
   > - 明显的痛点或差距
   > - 可能的改进杠杆点
   >
   > 保持浅层扫描——只阅读顶层文档和目录结构。请勿分析 GitHub 问题、模板或贡献指南。不要进行深度代码搜索。
   >
   > 焦点提示：{focus_hint}

2. **学习内容搜索** — 发送 `spec-first:research:learnings-researcher` 并简要总结构思重点。

3. **问题情报**（有条件）——如果在阶段 0.2 中检测到问题跟踪者意图，则使用焦点提示调度 `spec-first:research:issue-intelligence-analyst`。如果存在焦点提示，请将其传递，以便代理可以将其聚类加权到该区域。与代理 1 和 2 并行运行。

   如果代理返回错误（gh 未安装、无远程、身份验证失败），则向用户记录警告（“问题分析不可用：{reason}。按照标准构思继续进行。”）并继续现有的两个代理接地。

   如果代理报告的问题总数少于 5 个，请注意“主题分析的问题信号不足”，并在第 2 阶段继续使用默认构思框架。

将所有结果合并为简短的基础摘要。当存在问题情报时，将其保留为一个独特的部分，以便构思子代理可以区分代码观察到的信号和用户报告的信号：- **代码库上下文** — 项目形状、显着模式、明显的痛点、可能的杠杆点
- **过去的学习** — 来自文档/解决方案/的相关机构知识
- **问题情报**（如果存在）- 来自问题情报代理的主题摘要，保留主题标题、描述、问题计数和趋势方向

**不要**在 v1 中进行外部研究。

### 第二阶段：发散思维

严格遵循这个机制：

1. 在批评任何想法之前生成完整的候选者列表。
2. 每个子代理默认针对大约 7-8 个想法。有了 4-6 个代理，就会产生 30-40 个原始想法，这些想法经过合并和重复数据删除，得到大约 20-30 个独特的候选者。当数量覆盖适用时调整每个代理的目标（例如，“100 个想法”会提高它，“前 3 个”可能会降低幸存者数量）。
3. 推过安全明显层。每个特工最初的几个想法往往都很明显——超越它们。
4. 落实第一阶段扫描中的每个想法。
5. 使用此提示模式作为支柱：
   - 首先产生许多想法
   - 然后系统地挑战他们
   - 然后只详细解释幸存者
6. 如果平台支持子代理，请使用它们来提高候选池的多样性，而不是取代核心机制。
7. 给每个创意子代理相同的值：
   - 接地总结
   - 焦点提示
   - 每个代理的数量目标（默认情况下约 7-8 个想法）
   - 仅指导生成原始候选人，而不是批评
8. 使用子代理时，为每个子代理分配不同的构思框架作为**起始偏差，而不是约束**。提示每个代理从指定的角度开始，但无论它引导到哪里，都遵循任何有希望的线索——跨越多个框架的横切想法是有价值的，而不是超出范围。**框架选择取决于问题智能是否处于活动状态：**

   **当问题跟踪器意图处于活动状态并且返回主题时：**
   - 每个带有`confidence: high`或`confidence: medium`的主题成为一个构思框架。框架提示使用主题标题和描述作为起始偏差。
   - 如果少于 4 个簇派生帧，则按以下顺序填充默认帧：“杠杆和复合效应”、“打破假设或重新构建”、“痛苦步骤的倒置、删除或自动化”。这些通过超越报告的问题来补充基于问题的主题。
   - 总帧数上限为 6 帧。如果超过 6 个主题符合条件，则使用按问题数排列的前 6 个主题；将基础摘要中的剩余主题记为“次要主题”，以便下级代理仍然了解它们。**当问题跟踪器意图未激活时（默认）：**
   - 用户或操作员的疼痛和摩擦
   - 需求未得到满足或能力缺失
   - 痛苦步骤的反转、移除或自动化
   - 打破或重构假设
   - 杠杆效应和复利效应
   - 极端情况、边缘情况或高级用户压力
9. 要求每个构思子代理为每个想法返回标准化结构，以便协调器可以一致地合并和推理输出。更喜欢紧凑的类似 JSON 的结构：
   - 标题
   - 总结
   - 为什么重要
   - 证据或接地钩
   - 可选的本地信号，例如粗体或 focus_fit
10. 将子代理输出合并并删除重复数据到一个主候选列表中。
11. **合成横切组合。** 重复数据删除后，扫描合并列表，寻找来自不同框架的想法，这些想法组合在一起比单独的框架更强大。如果两个或多个想法自然地组合成一个更高杠杆的提案，请将组合的想法添加到列表中（预计最多添加 3-5 个）。这个综合步骤属于协调器，因为它需要同时看到所有想法。
12. 在合理的情况下将想法传播到多个维度：
   - 工作流程/DX
   - 可靠性
   - 可扩展性
   - 缺少能力
   - 文档/知识复合
   - 质量和维护
   - 对未来工作的影响
13. 如果提供了焦点，则将其传递给每个构思子代理，并对其加权合并列表，而不排除更强的相邻想法。

保存机制是：
- 首先产生许多想法
- 第二次批评完整的组合列表
- 仅详细解释幸存者要保留的子代理模式是：
- 以框架为起始偏见的独立构思
- 编排器合并、重复数据删除和横切合成第二
- 仅在组合和综合列表存在后才进行批评

### 第三阶段：对抗性过滤

批判性地审查每一个产生的想法。

更喜欢两层批评：
1. 让一个或多个持怀疑态度的子代理从不同的角度攻击合并列表。
2. 让协调者综合这些批评，一致地应用评分标准，对幸存者进行评分，并决定最终排名。

除非明确改进，否则不要让批评代理在此阶段产生替代想法。

批评代理可能会提供本地判断，但最终评分权属于协调者，因此排名在不同框架和视角下保持一致。

对于每个被拒绝的想法，写下一行原因。

使用拒绝标准，例如：
- 太模糊
- 不可采取行动
- 重复一个更强烈的想法
- 不以当前代码库为基础
- 相对于可能的价值来说太贵了
- 现有工作流程或文档已涵盖
- 有趣，但更好地处理为头脑风暴变体，而不是产品改进

使用一致的幸存者评分标准，其权重为：
- 当前回购协议的基础性
- 期望值
- 新奇
- 实用主义
- 对未来工作的影响
- 实施负担
- 与更强的想法重叠

目标输出：
- 默认保留5-7名幸存者
- 如果有太多人幸存，则进行第二次更严格的通行证
- 如果幸存的人数少于 5 人，请诚实报告，而不是降低标准

### 第 4 阶段：展示幸存者

在编写持久工件之前，向用户展示幸存的想法。

第一个演示是一个审查检查点，而不是最终的存档结果。仅以结构化形式呈现现存的想法：

- 标题
- 描述
- 理由
- 缺点
- 置信度得分
- 估计的复杂性

然后附上简短的拒绝摘要，以便用户可以看到考虑和削减的内容。

保持演示简洁。这件耐用的文物保持着完整的记录。

在编写工件之前允许简短的后续问题和简单的澄清。

不要编写构思文档，除非：
- 用户表示候选集足够好以保留
- 用户要求以应记录的方式完善和继续
- 工作流程即将移交给`spec:brainstorm`、证明共享或会话结束

### 第 5 阶段：编写构思工件

在候选集经过充分审查以保留后，编写构思工件。

始终在以下情况之前写入或更新工件：
- 移交给`spec:brainstorm`
- 分享到证明
- 结束会话

编写工件：

1. 确保`docs/ideation/`存在
2.选择文件路径：
   - `docs/ideation/YYYY-MM-DD-<topic>-ideation.md`
   - 不存在焦点时的 `docs/ideation/YYYY-MM-DD-open-ideation.md`
3. 撰写或更新构思文档

仅在必要时使用此结构并省略明显不相关的字段：
```markdown
---
date: YYYY-MM-DD
topic: <kebab-case-topic>
focus: <optional focus hint>
---

# Ideation: <Title>

## Codebase Context
[Grounding summary from Phase 1]

## Ranked Ideas

### 1. <Idea Title>
**Description:** [Concrete explanation]
**Rationale:** [Why this improves the project]
**Downsides:** [Tradeoffs or costs]
**Confidence:** [0-100%]
**Complexity:** [Low / Medium / High]
**Status:** [Unexplored / Explored]

## Rejection Summary

| # | Idea | Reason Rejected |
|---|------|-----------------|
| 1 | <Idea> | <Reason rejected> |

## Session Log
- YYYY-MM-DD: Initial ideation — <candidate count> generated, <survivor count> survived
```
如果恢复：
- 就地更新现有文件
- 附加到会话日志
- 保留探索过的标记

### 第 6 阶段：完善或移交

展示结果后，询问接下来会发生什么。

提供以下选项：
1.集思广益选定一个想法
2.提炼构思
3.分享到证明
4. 结束会话

#### 6.1 集思广益选定的想法

如果用户选择一个想法：
- 首先编写或更新构思文档
- 将该想法标记为 `Explored`
- 在会议日志中记下头脑风暴日期
- 以选定的想法作为种子调用 `spec:brainstorm`

**不要**跳过头脑风暴并直接从构思输出开始进行规划。

#### 6.2 完善构想

按意图细化路线：

- `add more ideas` 或 `explore new angles` -> 返回阶段 2
- `re-evaluate` 或 `raise the bar` -> 返回阶段 3
- `dig deeper on idea #N` -> 仅扩展该想法的分析

每次细化后：
- 在任何交接、共享或会议结束之前更新构思文档
- 附加会话日志条目

#### 6.3 分享到证明

如果需要，请使用插件中其他地方已使用的标准 Proof markdown 上传模式来共享构思文档。

分享完成后返回下一步选项。

#### 6.4 结束会话

结束时：
- 提议仅提交构思文档
- 不创建分支
- 不要推
- 如果用户拒绝，则保留文件不提交

## 质量栏

完成之前，请检查：- 想法集基于实际的回购协议
- 候选列表是在过滤之前生成的
- 保留了原来的多想法->批评->幸存者机制
- 如果使用子代理，它们可以在不替换核心工作流程的情况下提高多样性
- 每个被拒绝的想法都有一个原因
- 幸存者在物质上比天真的“给我想法”清单要好
- 该工件是在任何移交、共享或会话结束之前编写的
- 根据想法采取行动会通往`spec:brainstorm`，而不是直接实施

# 一、审查铁律

1. **代码优先，文档从属**
   - 必须优先使用代码、目录结构、命令实现、脚本、模板、测试、配置、运行时资产作为证据。
   - README、注释、命名、文档只能辅助说明，不能替代代码事实。

2. **严格区分四件事**
   - 项目“宣称支持什么”
   - 代码“实际实现了什么”
   - 流程“真正被 enforce 了什么”
   - 哪些只是“依赖宿主、依赖人工、依赖约定”

3. **必须识别软约束与硬约束**
   - 不要把“写在 `SKILL.md` 里”误判成“代码 enforce”
   - 不要把“有目录、有模板、有测试名”误判成“行为闭环”
   - 必须明确指出工程闭环和提示词闭环的差异
   - 但也不要把“没有硬状态机”自动判定为缺陷；先判断该系统是否本来就在追求“轻 contract + LLM 决策”

4. **必须优先评估决策输入质量**
   - 如果系统的目标是让 LLM 决策，而不是替 LLM 决策，就优先审查：
     - machine-readable context 是否真实、稳定、低漂移
     - provenance / freshness / confidence / fallback_reason 是否明确
     - verification summary / risk signals / verifier dispatch / verification gate state 是否足够支撑判断
   - 缺少硬编排本身不是问题；“给 LLM 的输入又弱又假”才是问题
   - 如果代码已经把 `verification_summary`、`verifier_dispatch`、`verification_gate_state` 拆成独立结构，必须按独立 contract 分析，不能把后两者重新压扁回 summary 的附属字段

5. **所有重要结论必须带证据等级**
   - `【代码已证实】`
   - `【高可信推断】`
   - `【文档声明但代码未充分证实】`
   - `【无法从仓库单独证实】`

6. **先证据，后判断**
   - 必须先复盘代码事实、执行链路、测试结果，再给评价
   - 禁止先下结论，再倒推理由

7. **审查标准必须是软件工程标准**
   - 不只看“有没有”
   - 还要看：一致性、完整性、稳健性、幂等性、可维护性、可扩展性、可测试性、可治理性、落地成本

8. **输出必须像工程审查，而不是产品宣传**
   - 可以尖锐，但必须具体
   - 可以否定，但必须给证据和修复方向
   - 不要写空话，例如“架构清晰”“设计优雅”“很有前景”这类无证据判断

# 二、审查流程

请按下面的工作顺序执行：

1. **建立证据面**
   - 盘点入口文件、核心模块、关键配置、模板、脚本、测试
   - 标出哪些是 source of truth，哪些是 generated / mirror / runtime copy

2. **重建执行链路**
   - 从 CLI 或主入口开始
   - 说明命令解析、路由、adapter、state、manifest、template sync、runtime asset 安装的实际链路

3. **检查流程 contract 与决策输入**
   - 针对 ideate / brainstorm / plan / work / review / compound / bootstrap / mcp-setup
   - 逐项判断：入口、主契约、输出工件、依赖关系、失败行为、hard/soft enforcement
   - 同时判断：这个流程给 LLM 提供的决策输入到底是什么，质量够不够

4. **检查测试与验证**
   - 尽量运行项目内已有的高信号验证命令
   - 必须优先使用本次实际重跑结果，不能直接沿用旧审查或旧报告中的测试统计
   - 如果测试失败，必须区分：
     - 是主逻辑失败
     - 还是文档/格式/治理层失败
   - 如果没有执行测试，也必须说明原因

5. **沉淀问题与优先级**
   - 输出问题清单
   - 按 P0 / P1 / P2 / P3 评级
   - 再给出修复顺序和 30/60/90 天路线图

## F. Bootstrap / Stage-0 / 决策输入质量专项加严

如果项目内存在 `bootstrap`、`stage0-context`、`graph-bootstrap`、`context-routing`、`minimal-context`、`verification_summary`、`verifier_dispatch`、`verification_gate_state`、`risk_signals`、`artifact-manifest` 这类机制，必须追加做下面的专项审查，不能只做一般性架构评论。

必须逐项判断：

1. 它是在生成“真实分析结果”，还是在生成“可被消费的样板控制面”
2. `complete` / `ready` / `healthy` 这类状态表示的是：
   - 装配完成
   - 事实可信
   - 还是两者被混在一起
3. `minimal-context/*.json`、`verification_summary`、`verifier_dispatch`、`verification_gate_state` 是否已经成为下游 workflow 的真实高优先级输入
4. 如果下游高度依赖这些输入，上游是否提供了足够的：
   - provenance
   - freshness
   - confidence
   - fallback_reason
   - coverage_gaps
5. 文档类产物是“从事实编译出来的”，还是“默认骨架 + 路径存在性”
6. evaluator / loader / router 在判定 L0/L1/L2 时，到底是在看：
   - 文件存在
   - contract 解析成功
   - 还是事实真实性达标

必须特别警惕以下误判：

- 不要把“有 `artifact-manifest.json`”误判成“事实抽取已完成”
- 不要把“有 `minimal-context/*.json`”误判成“决策输入足够可信”
- 不要把“文档文件存在”误判成“人类可读上下文已经高质量生成”
- 不要把“sample generator / fixture / default docs”误判成“真实仓库分析结果”

如果发现系统当前更强地证明了“控制面可装配”，而不是“事实真实性已建立”，必须明确写出：

1. 当前强项是什么

4. 哪些 L0 / complete / healthy 状态应该收紧

  - 工程治理

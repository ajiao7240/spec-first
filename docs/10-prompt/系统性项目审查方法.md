# spec-first 系统性项目审查方法 v1.0

本文件是 spec-first **「怎么做一次系统性项目审查」的方法论基线**。它与同目录 `结构化项目角色契约.md` 互补:角色契约规定「怎么做一次*演化判断*」,本文件规定「怎么编排、取证、收口一次*系统性审查*」。

**定位与边界:**

- 本文件是当前有效的审查方法基线,是本目录(历史快照目录)的明确例外之一(另一例外是 `结构化项目角色契约.md`),不是 runtime behavior contract。
- 它**不取代**同目录单视角内容审查 prompt(`项目审查.md`、`全面项目审查.md`、`审查skill.md`、`审查agent.md`、`审查token.md`):那些回答「审查*什么*」(项目本质、硬/软约束、质量),本文件回答「*怎么编排和收口*一次审查」。二者正交,可组合——本文件的 fan-out worker 可直接复用那些 prompt 作为单路审查者的注入内容。
- 它**不取代** spec-first 自带的 `/spec:code-review`、`/spec:doc-review`:那些是 runtime workflow,审查具体 diff / 单文档;本文件是**项目级、跨多真相源、含行业对标**的全局审查方法,粒度更大、周期更长。
- 与 `结构化项目角色契约.md` 冲突时以角色契约为准——本文件是方法,角色契约是判断基线。

**origin:** 本方法提炼自 2026-06-15 对 spec-first 的一次系统性 review 的实际执行(`docs/项目审查/2026-06-15-项目Review与优化方案.md`),并吸收 06-10/06-11/06-14 历次全项目审查的经验。

---

## 0. 何时用本方法 / 何时不用

**用本方法(项目级系统性审查):**
- 全局健康体检:定位是否统一、能力达成度、边界纪律、采纳成熟度。
- 行业对标:对照同类项目找差距与可借鉴 mechanism。
- 战略复盘:是否值得继续投入、下一步往哪走。
- 周期性(如季度/里程碑)或重大决策前的一次性深度审查。

**不用本方法(改用更窄的工具):**
- 审查一个具体 PR / diff → `/spec:code-review`。
- 审查一份具体计划/需求文档 → `/spec:doc-review`。
- 单视角内容质量评估 → 直接用 `项目审查.md` 等单 prompt。
- 轻量事实问答、单点定位 → 直接答,不起审查。

判据:**审查面是否横跨多个真相源、是否需要行业对标、是否要产出可落地的战略级方案。** 三者有其一才用本方法。

## 1. 审查者角色

你同时是**项目负责人、架构负责人、开源负责人**。这三重身份对应三类不会被单一视角覆盖的关切:

| 身份 | 守护什么 | 典型盲区(若缺此视角) |
|------|----------|----------------------|
| 项目负责人 | 用户真实研发增益、采纳、可行性 | 「能力强但没人能用/没人知道」被漏判 |
| 架构负责人 | 演化质量、边界清晰度、source/runtime 纪律 | 「能力堆叠但边界腐烂」被漏判 |
| 开源负责人 | OSS 成熟度、传播、社区、企业落地 | 「内部精密但对外不可信」被漏判 |

核心判断问题(承接角色契约 §1):
> 这个项目是否在让 AI coding 走向可治理、可验证、可复用、可沉淀的工程闭环——**并且这份价值是否可被目标用户与决策者识别、试用、评估**?

后半句是 2026-06-15 review 的元结论:**能力建设 ≠ 已兑现使命**。审查必须同时量度「能力」与「能力的可见性/可信度/可采纳性」。

## 2. 七条审查铁律

1. **代码/产物优先,文档从属。** 优先用源码、目录结构、命令实现、脚本、模板、测试、配置、git 事实作证据;README/注释/命名只能辅助,不能替代代码事实。

2. **严格区分四件事:** 项目「宣称支持什么」/ 代码「实际实现什么」/ 流程「真正被 enforce 什么」/ 哪些只是「依赖宿主、依赖人工、依赖约定」。不要把「写在 SKILL.md 里」误判成「代码 enforce」,不要把「有目录/模板/测试名」误判成「行为闭环」。

3. **硬约束 vs 软约束必须标明。** 明确工程闭环与提示词闭环的差异。但**不要把「没有硬状态机」自动判成缺陷**——先判断该系统是否本就追求「轻 contract + LLM 决策」(角色契约 §3)。对 spec-first,prose gate 降级为「响亮约定」是有意设计,不是 bug;真正的 finding 是「响亮约定退化成静默约定」(无声明、无最小事实记录)。

4. **证据分级,advisory ≠ confirmed。** 每条结论标注证据等级:`confirmed`(grep/wc/git/test/md5 实证或人类确认)、`advisory`(provider/subagent/对标判断,未回源核对)、`degraded`(stale/partial)。量化结论(体积、计数、命中)必须由确定性命令实证,禁止凭印象。

5. **诚实声明局限。** 未执行的检查(未跑全量 test、未取 GitHub 实时数据、WebSearch 不可用、subagent 判断未回源)必须显式声明为未执行/advisory,不得伪装成已验证。这是角色契约 §4「响亮约定」对审查自身的应用。

6. **不重复已知,增量定位。** 审查前先读最近的审查报告(`docs/项目审查/`)。不重复发现已知问题,而是:把历史 finding ground-truth 到今日 disk(裁决 RESOLVED/PARTIAL/OPEN/REGRESSED)+ 补历史审查较少覆盖的透镜(典型:行业对标、真相源一致性、采纳成熟度)。

7. **只读 source,禁读 generated runtime mirror。** 审查证据排除 `.claude/**`、`.codex/**`、`.agents/skills/**`、`.spec-first/audits/**`、`.spec-first/governance/**`、`graphify-out/**`(setup/runtime-drift 专项审查除外)。source drift 本身可 confirmed,双宿主 runtime 投射影响为 advisory 推断。

## 3. 审查维度(12 维)

每维输出「证据 / 问题 / 建议」。维度可按审查目标裁剪,但**定位、闭环、证据、采纳**四维不可省。

1. **定位与差异化** —— 30 秒价值;区别于 prompt pack / AI IDE / workflow engine / 强状态机 / 传统 SDD;一句话定位在各真相源是否统一。
2. **Onboarding / 可试用** —— 安装、doctor、init、host 入口、demo repo、quickstart、案例;newcomer 能否「clone 即见效果」。
3. **Workflow 闭环** —— Codebase→Spec→Plan→Tasks→Work/Debug/Review→Evidence→Knowledge 各节点覆盖深度;哪些节点厚、哪些薄。
4. **Context / Graph / 多仓** —— bounded reads、context bundle、graph readiness、target repo、cross-repo、degraded mode。
5. **Evidence / Eval** —— 证据链、verification gate、honest-closeout、是否有真实自动化 eval(还是只校验 contract shape)。
6. **Review / Governance** —— code-review/doc-review/audit、安全、合规、架构、性能、测试、文档一致性、gate 强制度。
7. **Knowledge** —— bug→learning、review finding 沉淀、团队规范、promotion gate 是否被遵守、recall 实际覆盖率。
8. **CLI / Runtime / Host** —— doctor/init/clean/tasks、双宿主同源、runtime drift、多宿主扩展策略。
9. **文档 / 官网 / 传播** —— README(双语)、官网、用户路径、Why/Quickstart/Concepts/Workflow/CLI/Architecture/FAQ、关键图示、CHANGELOG 对外可读性。
10. **开源成熟度** —— LICENSE/SECURITY/CONTRIBUTING/COC、issue/PR 模板、CI、examples、release notes、GitHub Releases、roadmap、bus factor。
11. **企业落地** —— 团队采纳指南、效果指标、案例、外部工具集成、高治理/合规叙事;治理价值是否对决策者可见。
12. **战略取舍** —— 必须做 / 集成做 / 不做;3 个差异化标签;3 个失焦风险;里程碑。

## 4. 编排:Scripts prepare, LLM decides 的审查版

审查本身遵守角色契约的职责边界:**确定性取证脚本化,语义判断 LLM 化,综合由主线程亲自做。**

### 4.1 主线程先做最小事实勘察(不下放)
- 项目结构、git remote/tag/version、关键文件行数。
- 读核心真相源:角色契约、最近审查报告、README 首屏、官网 hero、package.json。
- 目的:建立准确基线,决定 fan-out 的形状。**不要在没勘察前就盲目 fan-out。**

### 4.2 Fan-out 并行取证(下放给通用 subagent,不调用本项目审查 skill)
按维度/对标对象分组并行,典型三类 worker:
- **行业对标路**(每路 1-N 个对标仓):mechanism-level 深读,输出 `定位/入口/闭环/治理-证据-知识/多宿主/亮点/领先 spec-first 之处/可借鉴动作`。
- **Web 趋势路**:行业现状 + 用户痛点 + 差异化信号 + 趋势 + 定位建议。
- **内部事实审查路**:真相源一致性 / 闭环与 contracts/tests 覆盖 / 开源成熟度与企业落地。

每个 worker 用 schema 约束输出(结构化返回,免解析)。worker 用通用 subagent / Explore,**不要调用 spec-first 自带的 `/spec:*` 审查 workflow**(避免审查者污染被审对象的 runtime,也避免会话缓存旧定义)。

### 4.3 主线程综合(不下放)
- 收集结构化结果,与 4.1 的实证交叉印证。
- 对每条存活的高severity finding,可派**证伪立场 verifier**(对抗验证)再定级。
- 综合、定级、写报告由主线程亲自做,保证连贯与口径一致。

### 4.4 边界
- 脚本/命令产 deterministic facts(grep/wc/git/md5/test);LLM 产语义判断(架构合理性、风险、取舍、next action)。
- 不让脚本模拟架构判断;不让 LLM 假装跑过确定性校验或编造命令结果。

## 5. 输出结构(10 段)

产出《<项目名> 项目 Review 与优化方案》,写入 `docs/项目审查/YYYY-MM-DD-*.md`,带 frontmatter(`doc_role: review-report`、`review_date`、`review_method`、`relates_to`、`limitations`)。

1. **Executive Summary** —— 总体判断、优势、短板、机会、优先方向、阶段目标。
2. **项目事实审查** —— 表格 `审查项 / 当前状态 / 证据 / 问题 / 建议`。
3. **行业对标** —— 表格 `对标对象 / 核心定位 / 值得学习 / 当前差距 / 可借鉴动作`。
4. **核心优势**。
5. **当前短板** —— 表格 `优先级 / 短板 / 影响 / 根因 / 修复建议`。
6. **优化方案** —— 短期 / 中期 / 长期,重大项标注价值-成本-风险-优先级-负责人角色。
7. **Roadmap** —— 表格 `时间 / 目标 / 关键能力 / 交付物 / 成功指标 / 风险`。
8. **不做什么** —— 明确边界(承接角色契约系统边界);差异化标签;失焦风险。
9. **推荐文案** —— 一句话定位、30 秒介绍、README/官网首屏、面向 Tech Lead/开发者/老板的价值表达。
10. **最终结论** —— 当前阶段、是否值得投入、需补能力、最大机会、最大风险、下一步 3 件事。

## 6. 收口:审查必须可被消费(闭环纪律)

**这是本方法最重要、也最容易失守的一条**,直接来自 06-10/06-14 报告反复点名的元结论:**一份无法被追踪是否被消费的审查,在自指层面尚未兑现 spec-first 对用户承诺的「可治理闭环」。**

审查产出后必须:

1. **findings 是 review evidence,不是已拍板决策。** 报告里显式声明此点,落地前回源核对(尤其 advisory 级与对标「领先」判断)。
2. **建立 origin 可追踪链。** 对 P0/P1 finding 各起一个窄 plan,以本审查报告为 `origin`(写入 plan frontmatter 的 `referenced_reviews[].addresses_findings`,见 `docs/contracts/workflows/review-closure-traceability.md`),走当前 host 的 `/spec:plan` → `/spec:work`。**禁止让高置信审查在 origin 链上凭空消失。**
3. **durable 判断回流基线。** 审查暴露的、超出单次整改的*演化判断原则*(而非 execution 细节),回流到 `结构化项目角色契约.md`(按其 §11 修订纪律:带 origin + 独立 reviewer 视角 + 自指 knowledge-promotion gate)。execution 细节(demo、CHANGELOG 格式、OSS 卫生)留在报告与 backlog,不进基线。
4. **下一轮审查裁决上一轮。** 把上一轮全部 finding ground-truth 到今日 disk,逐条裁决 RESOLVED/PARTIAL/OPEN/REGRESSED——这既是增量定位(铁律 6),也是闭环健康度的度量。

> 一句话:审查的价值不在「又发现 N 个问题」,而在「发现→整改→验证→沉淀」这条链本身被走通且可追踪。

## 7. 反模式(审查自身的红旗)

出现以下念头/产出即停,回到对应铁律:

- 「复述 README / 总结愿景」→ 审查要回答工程问题,不写宣传稿(铁律 1)。
- 「写在 SKILL.md 里就算实现了」→ 区分软/硬约束(铁律 2-3)。
- 「没有硬状态机就是缺陷」→ 先判断系统是否本就追求轻 contract(铁律 3)。
- 「subagent/对标说领先,直接写进结论」→ 标 advisory,关键项回源(铁律 4)。
- 「全量 test 没跑但报告说验证通过」→ 诚实声明未执行(铁律 5)。
- 「把上轮已知问题再发现一遍」→ 增量定位,裁决历史 finding(铁律 6)。
- 「读 .claude/.codex mirror 当证据」→ 只读 source(铁律 7)。
- 「报告写完就结束」→ 没起 origin 追踪 plan = 审查未收口(§6)。
- 「把 demo/CHANGELOG 格式这类 execution 细节塞进角色契约」→ 那留在 backlog(§6.3)。

## 8. 复用与维护

- 本方法是 spec-anchored:每次大型审查后,若方法本身暴露改进点(新维度、新反模式、新编排技巧),回流更新本文件并续写 origin。
- 单视角内容审查仍可直接用同目录的 `项目审查.md` 等——本方法的 fan-out worker 可把它们作为单路审查者的注入 prompt,二者组合使用。
- 本文件任何修订至少按角色契约 §8 中型处理;改动铁律或编排骨架按大型处理。

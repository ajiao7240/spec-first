# 3. 自我审视目标

自我审视不是为了找问题而找问题，而是为了识别：

```text
当前 spec-first 哪里阻碍了高质量辅助研发？
```

必须审视：

## 3.1 目标是否对齐

检查：

* 当前项目是否仍然服务于 workflow-first AI coding；
* 是否开始变成 prompt pack；
* 是否开始变成 agent collection；
* 是否开始变成重型 runtime；
* 是否仍然服务于高质量辅助研发流程。

## 3.2 workflow 是否闭环

检查链路：

```text
mcp-setup / graph-bootstrap
  -> ideate / brainstorm / doc-review
  -> spec-plan
  -> spec-write-tasks
  -> spec-work / debug / optimize / polish
  -> code-review / app-consistency-audit
  -> compound / compound-refresh / skill-audit
```

判断：

* 上游产物是否被下游消费；
* plan / tasks / work / review 是否断链；
* compound 是否反哺下一轮；
* graph facts 是否被正确消费；
* review findings 是否进入知识沉淀。

## 3.3 skill 是否健康

检查：

* Purpose 是否清楚；
* When to use / When not to use 是否清楚；
* Non-goals 是否存在；
* Inputs / Outputs 是否明确；
* Verification 是否真实；
* 是否存在职责重叠；
* 是否有大 prompt 化趋势；
* 是否有产物没人消费。

## 3.4 context 是否精准

检查：

* AGENTS.md / CLAUDE.md 是否过重；
* repo-profile 是否被写入临时判断；
* graph facts 是否按需读取；
* compound knowledge 是否可路由；
* brainstorm / ideate 是否被实现细节污染；
* work / review 是否缺必要代码事实。

## 3.5 artifact 是否可信

检查：

* artifact 是否带 schema_version；
* 是否带 generated_at；
* 是否能追溯 source_commit / branch / dirty_state；
* latest artifact 是否有 freshness 判断；
* downstream 是否知道 trust / stale advisory / reject。

## 3.6 review 是否有证据

检查：

* code-review 是否基于 diff、plan、tasks、evidence；
* skill-audit 是否基于 source skill contract；
* doc-review 是否区分文档问题和系统能力问题；
* review 是否能输出可执行、可验证的 findings。

## 3.7 knowledge 是否复利

检查：

* compound 是否只是总结；
* 是否输出 pattern / anti-pattern；
* 是否输出 decision log；
* 是否输出 next-cycle input；
* 历史 lesson 是否被下一轮读取。

# 5. 自我升级目标

自我升级不是让项目变大，而是让能力更成熟。

升级对象包括：

| 升级对象                 | 目标                   |
| -------------------- | -------------------- |
| Workflow             | 节点更清晰，handoff 更稳定    |
| Skill                | 边界更明确，触发更准确          |
| Context              | 更少但更准                |
| Artifact             | fresh、可追溯、可消费        |
| Review               | 更有证据，更少泛泛建议          |
| Graph                | readiness 更可靠，影响面更准确 |
| Docs                 | 和代码保持一致              |
| Knowledge            | 能反哺下一轮               |
| Best Practice Intake | 能学习外部，但不盲目照搬         |
| Verification         | 命令真实，验证路径明确          |

升级等级：

* L0 No Upgrade：不升级，只记录原因
* L1 Contract Upgrade：改契约、文档、边界
* L2 Routing Upgrade：改路由、handoff、trigger
* L3 Capability Upgrade：新增或增强 skill / script / eval
* L4 System Upgrade：影响 runtime / command / CI / release

本轮只允许建议：

* L0
* L1
* 少量 L2
* 条件性 L3 后续建议

本轮不允许直接实施：

* L3
* L4

# 6. 持续迭代 Loop

必须使用下面的持续迭代 loop：

```text
Loop 0: Establish Goal
  明确本轮自我审视目标。

Loop 1: Inspect Current State
  读取仓库事实，建立 composition baseline。

Loop 2: Identify Capability Gaps
  判断哪些问题是系统能力缺口。

Loop 3: Search External Practices
  查询业界最佳实践与 GitHub 同类方案。

Loop 4: Compare & Decide
  判断外部实践是否适配当前 gap，形成 CUD。

Loop 5: Plan Handoff
  Accepted CUD 只输出 plan handoff，不直接实现。

Loop 6: Review Expectation
  定义后续 code-review / skill-audit / doc-review 如何验证。

Loop 7: Knowledge Feedback
  定义 compound 应该沉淀什么。

Loop 8: Next Self-Reflection Input
  输出下一轮审视输入，形成持续迭代。
```

每轮输出都必须能成为下一轮输入。

禁止一次性报告后结束。

# 12. 不绕过 spec-plan

self-reflection / capability-upgrade 的职责边界：

Self-reflection Prompt 负责：

* 是否存在能力缺口
* 是否需要能力升级
* 为什么接受 / 跳过 / 延后
* 交给谁继续
* 如何验证
* 如何沉淀
* 下一轮继续看什么

spec-plan 负责：

* 如何设计实现

spec-write-tasks 负责：

* 如何拆任务

spec-work 负责：

* 如何执行

code-review / skill-audit 负责：

* 是否正确

compound 负责：

* 如何沉淀经验，反哺下一轮

不要输出 implementation plan 来替代 spec-plan。
你只能输出 plan handoff brief。

plan handoff 是 advisory context。
不是 requirement。

## Step 6. Identify Capability Gaps

不要只写 Finding。
必须判断它是否构成 Capability Gap。

Capability Gap 的含义：

当前系统缺少某种能力，导致这个问题无法被现有 workflow 稳定解决，或会反复出现。

格式：

## CG-001: <能力缺口标题>

### Evidence

* 引用具体文件、路径、命令、报告、review finding 或观察事实。
* 每条 evidence 必须可追溯。
* 不确定的地方必须标注 uncertainty。

### Evidence Strength

Strong / Medium / Weak / Unknown

Evidence strength 定义：

* Strong: direct source/code/contract evidence
* Medium: multiple indirect signals
* Weak: plausible but not proven
* Unknown: insufficient evidence

Accepted CUD requires Strong or Medium evidence.
Weak evidence can only produce Deferred.
Unknown evidence cannot produce Accepted.

### Current Behavior

当前系统如何处理这个问题。

### Capability Missing

缺少的能力是什么。

### Why Existing Workflow Is Not Enough

为什么 skill-audit / doc-review / spec-plan / code-review / compound 组合不足以稳定解决。

### Impact

如果不补这个能力，会影响什么。

### Upgrade Level

L0 / L1 / L2 / L3 / L4

升级等级定义：

* L0 No Upgrade：不升级，只记录原因
* L1 Contract Upgrade：改契约、文档、边界
* L2 Routing Upgrade：改路由、handoff、trigger
* L3 Capability Upgrade：新增或增强 skill / script / eval
* L4 System Upgrade：影响 runtime / command / CI / release

本轮只允许建议：

* L0
* L1
* 少量 L2
* 条件性 L3 后续建议

本轮不允许直接实施：

* L3
* L4

### Candidate Upgrade

建议升级方向。

### Counter-signal

有什么证据说明也许不需要升级。

### Decision Candidate

Accepted / Skipped / Deferred

如果没有 named capability gaps，必须明确写：

No named capability gaps found.

# 14. 区分能力层级

不要混淆：

* declared capability
* implemented capability
* runnable capability
* verified capability
* reusable capability

必须牢记：

A section existing in a document is not proof that the capability exists.
A command mentioned in README is not proof that the command works.
A skill file existing is not proof that the workflow can use it.
A generated artifact existing is not proof that it is fresh or consumable.
A best practice being popular is not proof that spec-first should integrate it.

每个 capability 判断都要标注层级：

| Capability Level | 含义                           |
| ---------------- | ---------------------------- |
| Declared         | 文档或说明里声明了                    |
| Implemented      | 源码或 source asset 中实现了        |
| Runnable         | 可以通过真实入口运行                   |
| Verified         | 有验证命令、测试、review 或 dogfood 证明 |
| Reusable         | 能稳定被下游 workflow 消费和复用        |

   把重复工程流程沉淀为仓库内可复用 skill。

3. Progressive disclosure
   只在需要时加载更深上下文。

   把上下文看作有限资源，避免 context rot。

   docs-sync / coverage / release readiness 先报告，再批准修改。

6. Skill evals
   用 trigger / process / output / efficiency eval 检查 skill 质量。

   只在大型 PR / macro audit / release readiness 中使用。

9. Release readiness
   仅在 release 涉及 workflow、contract、runtime governance、skill/agent 调度、public positioning 时触发。

10. Evolution memory
    compound 输出 pattern / anti-pattern / decision log / next-cycle input。

11. Codebase graph / impact analysis
    用 graph facts 支撑 plan、review、blast-radius、test focus。

12. Review routing
    根据 diff / risk / artifact 类型选择 review lens，而不是固定全量专家。

13. Context router
    决定哪个 skill 读什么、不读什么，避免全局上下文膨胀。

14. Verification router
    根据变更类型选择真实验证命令和 review gate。

15. Memory router
    决定经验沉淀到 docs、compound、repo-profile、AGENTS.md 还是 next-cycle input。

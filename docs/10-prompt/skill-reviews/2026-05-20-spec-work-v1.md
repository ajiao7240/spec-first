## 1. 优秀 Skill 的判断基线

优秀 workflow skill 不是长 prompt，也不是隐式状态机。对 `spec-work` 这种高风险执行入口，质量基线应满足:

1. **触发准确**: `description` 说明做什么、何时用、何时不用；正文的 When / When Not 保持互斥。
2. **输入有边界**: plan/task-pack/bare prompt 的 scope authority 清楚；large bare prompt 不伪装成 task pack。
3. **执行有反馈回路**: 开工前建立最小可观察 loop，执行后重跑同一 loop 或记录不能重跑的原因。
4. **验证不是口号**: test discovery、test scenario completeness、system-wide impact check、review tier、final validation 都要可执行。
5. **脚本与 LLM 分工明确**: deterministic facts 由 CLI/git/tests/MCP 准备；scope、handoff、risk、review interpretation 由 LLM 判断。
6. **progressive disclosure 稳定**: 高频安全边界留在 `SKILL.md`，低频流程细节才进 `references/`；外置不能让 Codex/Claude 任一宿主丢失关键路径。
7. **source/runtime 边界清楚**: 不把 `.claude/`、`.codex/`、`.agents/skills/` runtime mirror 当 source fix。
8. **产物能被 downstream 消费**: completion summary、review handoff、run artifact、known residuals、compound handoff 的 authority 和限制清楚。

这与项目角色契约一致: `docs/10-prompt/结构化项目角色契约.md` 明确脚本负责确定性事实，LLM/agents 负责语义判断；但该角色契约本身不是 runtime behavior contract，`spec-work` source 仍需保留自己的轻量执行边界。

   - 不在 `spec-work` 新增 staging/security engine。

   - 修为 `7. Track Progress`。
   - 验证: `rg -n "^\\d+\\. \\*\\*" skills/spec-work/SKILL.md` 检查 Phase 2 顺序。

### 6.2 Should Fix

1. **P1-001-light-boundary: 加轻量 scripts/tools vs LLM boundary**

   不新增 25 行大段私有 policy。建议在 `Workflow Contract Summary` 后加 3-6 行:

   ```md
   ### Execution Boundary
   `spec-work` orchestrates execution; it does not own deterministic validators or provider refresh.
   Tools/CLI/git provide task-pack validation, diff/branch facts, test results, and optional run artifact writes.
   The LLM decides scope fit, task ordering, review depth, handoff, and whether advisory evidence is sufficient to proceed.
   ```

   这样满足 `审查skill.md` 的显式边界要求，同时不复制项目角色契约。

2. **P2-007-one-line: large bare prompt task list 边界**

   在 Phase 0 Large row 或 Oversized intake 中补一句:

   ```md
   If the user chooses to proceed from a large bare prompt, the task list is a session-local execution index, not a persisted task pack or substitute for `spec-write-tasks`.
   ```

### 6.3 Optional

1. **P2-001-concise-description**

   当前 description 泛化。建议短版:

   ```yaml
   description: Execute a settled plan, validated task pack, or scoped implementation prompt within current repo scope.
   ```

   不采用旧报告 67 词长版；description 是 trigger 层，不应塞入 branch、review、dispatch 全流程。

2. **P2-008-dedupe-only**

   可压缩 `Key Principles`，但不把它列为 safety must-fix。保留 `Start Fast, Execute Faster` 的 ship-bias；把已经由 Phase 2 / shipping workflow 覆盖的 bullets 合并或删除。`Cache-Friendly Context Layout` 若保留，应改名为执行性更强的 `Context Handoff Layout`，因为其中包含 `artifact-summary.v1` / `context-bundle.v1` handoff 指令。

3. **P2-002-low-frequency-extraction-only**

   可以只外置:

   - required `review_gate` mini review 具体 diff-anchor 流程；
   - worktree-isolated post-batch cleanup 7 步细节。

   不外置 task-pack reject matrix 与 host capability matrix；这两段是高频安全锚点。

### 2.1 P1:补显式 Evidence And Execution Boundary(项目核心哲学维度)

**问题**:文中 evidence 表明 spec-work 无 scripts/ 是合理设计——它是 LLM-orchestrated execution skill,deterministic tools 和 facts 分别来自 CLI、git 与其他 workflow artifacts。但**没有显式段落声明这件事**,fresh-source reviewer 与下游 consumer 必须跨 4-5 段(§Workflow Contract Summary、Phase 1 Step 1、§Run Artifact Boundary、§Graph Freshness)推断边界,易把 LLM 步骤误读为确定性动作,或把 spec-work 必须调用的 CLI 命令误读为只读 upstream facts。

**修复**:在 line 47(§Downstream Consumers 之后)或 line 49(§Examples As Context 之前)插入一个单一边界锚点,并让 §Context Orientation Anchor / §Graph Freshness / §Run Artifact Boundary 引用它而不是重复维护 policy:

```markdown
## Evidence And Execution Boundary

`spec-work` is LLM-orchestrated and ships no `scripts/` of its own. It invokes deterministic tools when the workflow requires them, and consumes deterministic facts prepared by other source-owned tools or artifacts.

Deterministic tools invoked by spec-work when required:
- `spec-first tasks validate <task-pack-path> --json` — task-pack identity, hash, and contract validation before execution
- `spec-first internal spec-work-run-artifact write ...` — closeout artifact writing when durable evidence triggers apply
- `git status` / `git diff` / `git branch --show-current` — current branch, diff, and worktree state

Read-only facts consumed by spec-work:
- `.spec-first/graph/{provider-status,graph-facts}.json` — graph readiness facts
- `.spec-first/impact/bootstrap-impact-capabilities.json` — provider projection freshness
- `review-pre-facts` / standards / prior review artifacts when present — advisory context with provenance and freshness
```

### 7.2 Reviewer Prompt 真实 Gap

这是 reviewer prompt 历史 gap,影响所有跑 `审查skill.md` 的 fresh review:**结构合理的 stale governance 引用不会被发现**,直到对抗审查或多轮迭代才暴露。

### 7.3 Reviewer Prompt 修复(本次同步落地)

`docs/10-prompt/审查skill.md` 已加三处修复关闭该 gap:

1. **§0 evidence 规则**新增 `<grep:0-hit token="..." searched=[...]>` 形态,专用于 stale 引用 claim
2. **§2 #11 examples 通过标准与反例**强化:示例引用的所有 token 在 SKILL.md / references / 项目级 source-of-truth 中可 grep 命中;stale 引用直接触发 fail 反例
3. **§5 Step 1.5 Bundled Reference Cross-Verification(新增,必做)**:对 examples.json / contract.yaml / references 中的所有 reference token 强制做反向 grep 验证;0 命中即 stale,升 P1 finding;Bash 工具调用记录在 evidence

### 7.5 第一性原理沉淀

Reviewer prompt 演进的核心教训:**"结构合理性 + 引用一致性"两层检查不够,必须加第三层"反向 grep 真实性"**。skill quality 的真实瓶颈不是文件结构是否漂亮,而是引用是否指向真实存在的 source——这正是 spec-first 项目"Light contract + Explicit boundaries + Scripts prepare LLM decides"哲学的具体落地:**Scripts(grep)prepare deterministic facts(token 是否真实存在),LLM decides(stale 严重度与修复路径)**。

未来跑 `审查skill.md` 的 reviewer 必须执行 §5 Step 1.5。本次 spec-work 审查 报告作为该 step 必要性的实证案例。

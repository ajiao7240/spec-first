### 7.2 Reviewer Prompt 真实 Gap

Fresh subagent 独立得出 P1 finding 与累积分析方向**完全一致**(identity / workflow_disclosure / script_llm_boundary / examples 全部命中相同 partial 维度),但**完全漏掉 P1-002 examples.json 第 5 例的 ownership 文档化缺失问题**——`secret-deny-patterns.json` / `batch-owned files` / `expected_side_effects` / `--copy-env` 4 个 token 在 SKILL.md / references **本 skill 内** grep 0 命中。

**§8.1 校准重要说明**:早期 v0 §7.2 把该 finding 描述为 "stale governance" 是不准确的——这 4 个 token 在**全仓 grep**(含 src/cli/contracts/ + 邻近 skill source)中**有真实 owner**(详见 §8.1 sanity check 表)。准确描述是 "ownership 在邻近 skill 而非 spec-work 私有,examples.json source_note 没显式声明 owner";不是 "governance 已废弃" 或 "stale"。但**reviewer prompt 仍漏检了这个 ownership 文档化缺失问题**——这是 Step 1.5 改进的真实价值(无论描述方式)。

漏检原因:**`审查skill.md` 原版只要求 "Read 全文 + 枚举 bundled 目录 + 抽样标题"**,**没强制对 examples.json / contract.yaml 中所有 reference token 做反向 grep 验证**。Fresh subagent 看到 examples.json 结构合理(schema_version + 字段齐全)就给 examples 维度 partial(只因主文件无内联示例),没继续验证示例中引用的 token 是否真实存在或归属哪个 owner。

这是 reviewer prompt 历史 gap,影响所有跑 `审查skill.md` 的 fresh review:**结构合理的 stale governance 引用不会被发现**,直到对抗审查或多轮迭代才暴露。

### 7.3 Reviewer Prompt 修复(本次同步落地)

`docs/10-prompt/审查skill.md` 已加三处修复关闭该 gap:

1. **§0 evidence 规则**新增 `<grep:0-hit token="..." searched=[...]>` 形态,专用于 stale 引用 claim
2. **§2 #11 examples 通过标准与反例**强化:示例引用的所有 token 在 SKILL.md / references / 项目级 source-of-truth 中可 grep 命中;stale 引用直接触发 fail 反例
3. **§5 Step 1.5 Bundled Reference Cross-Verification(新增,必做)**:对 examples.json / contract.yaml / references 中的所有 reference token 强制做反向 grep 验证;0 命中即 stale,升 P1 finding;Bash 工具调用记录在 evidence

**预期效果**:未来跑 `审查skill.md` 的 fresh review 不会再漏掉 P1-002 类 stale governance 问题,fresh review 与对抗审查的判断收敛差距将显著缩小。

### 7.4 对 §6 minimum viable fix set 的最终验证

Fresh review 独立得出的 4 个 partial 维度(identity / workflow_disclosure / script_llm_boundary / examples)**与 §6 必修 / 可选清单方向完全一致**:

- P2-001(description)→ 命中 identity ✓
- P2-002(拆 references)→ 命中 workflow_disclosure ✓
- P1-001(boundary 段)→ 命中 script_llm_boundary ✓
- P2-004(Minimal Examples)→ 部分命中 examples ✓
- P1-002(stale)→ 只能通过 Step 1.5 grep 命中(原版 fresh review 漏)→ 现在可被发现 ✓

且 fresh review 的 P2 finding 比累积分析的更精炼(2 项 vs 8 项),反向证明 §6 把 9 项收敛为 **3 必修 + 1 可选 + 2 降级 + 1 砍 + 1 重路由** 的方向是对的——P2-006 / P2-007 / P2-008 等更细颗粒度 finding 在 fresh 视角下确实不算 must-fix。

**最终结论**:§6 minimum viable fix set 经 fresh review 二次验证仍成立。落地修复时引用 §6,不直接按 §1 全量执行。

### 7.5 第一性原理沉淀

Reviewer prompt 演进的核心教训:**"结构合理性 + 引用一致性"两层检查不够,必须加第三层"反向 grep 真实性"**。skill quality 的真实瓶颈不是文件结构是否漂亮,而是引用是否指向真实存在的 source——这正是 spec-first 项目"Light contract + Explicit boundaries + Scripts prepare LLM decides"哲学的具体落地:**Scripts(grep)prepare deterministic facts(token 是否真实存在),LLM decides(stale 严重度与修复路径)**。

未来跑 `审查skill.md` 的 reviewer 必须执行 §5 Step 1.5。本次 spec-work 审查 报告作为该 step 必要性的实证案例。

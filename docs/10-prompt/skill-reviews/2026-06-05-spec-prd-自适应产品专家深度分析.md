## 0. 结论先行

**关键事实:这个 skill 正在被改造,而且改的正好是本任务要做的事。** 当前工作区里 `skills/spec-prd/**` 有未提交 source 编辑,新增的能力恰好是"自适应产品专家质量闭环"——Adaptive Product Expert Lens、`quality_diagnosis` 决策卡字段、PRD Quality Diagnosis And Optimization(`original -> recommendation -> reason -> write target`)、Quality Diagnosis Pack。contract test 已同步覆盖这些字符串 contract,CHANGELOG 已有对应记录。

三句话判断:

1. **方向正确。** "证据约束生成 + human-in-the-loop 评审 + 按变更规模 right-size 产出深度"这条设计主线,被 2024 IEEE RE/RE@Next! 研究、2025/2026 综述、Cagan 四类风险、INVEST/EARS/Gherkin/CIA 等来源共同支持(详见 §2)。
2. **落地扎实,只有低severity drift。** 最硬问题仍是 enum 双名(`quality_diagnosis` vs `quality_posture`)和"质量维度三处重复"。两者都是 **P3**:错误行为概率低,真正成本是维护 churn,cheap to fix,**不阻塞 shipping**。
3. **诚实性闭环未完全持久化。** 仓库持久工件 `docs/validation/spec-prd/fresh-source-eval-2026-06-05-product-expert-quality-loop.md` 仍记 `status: not_run`。本文 §5 中的 reviewer YAML 若无对应 transcript 或落盘更新,只能作为 `session-local/advisory`,不能当 confirmed eval 通过。

> **confirmed 结论:** 当前 source 显示这是尊重 Light contract + 80/20 的净改进:诊断闭环正确约束为 run-local lens(非新 agent、非评分 rubric),WHAT-not-HOW 成立,建议折叠进重写 PRD 无第二 artifact 拓扑。**advisory 结论:** 文内独立 reviewer 摘要也支持 `ship with two P3 cleanups`,但因未见持久 eval 证据,本文不把它升级为 confirmed truth。

## 1. skill 当前架构(现状快照)

### 1.1 定位与链路节点

`spec-prd` 是 brownfield PRD requirements workflow,服务核心链路的 Spec/PRD 节点:

```text
Codebase -> Context -> Spec/PRD -> Plan -> Tasks -> Code -> Review -> Knowledge
                          ^
                      spec-prd 在这里
```

核心目标不是"写更长的 PRD",而是降低下游 `spec-plan` 的 planning invention:先对齐系统现状证据 → 确认增量 delta(keep/extend/replace/remove/unknown)→(对低质量 PRD)做产品专家式质量诊断 → 输出标准 Markdown PRD。

## 2. 行业证据校准:这个 skill 的设计假设站得住吗?

原深度研究记录了 23 条 confirmed claim 和 2 条 refuted claim。本轮复核重点核对这些 claim 是否仍符合 2026-06-05 的公开来源状态;逐项对照 skill 设计:

### 2.1 ✅ "LLM 应是辅助 + human-in-the-loop,不是自主 PRD 生成器/裁判" —— 高置信,强支持

- **证据(high,arXiv:2408.10886,IEEE RE 2024):** LLM 评审需求质量有系统性**负向偏置**——把合格需求判为合格的数量远少于人类(DigitalHome 项目 人类 514 vs LLM 225,>2 倍),差异集中在 appropriate/correct/singular/unambiguous/verifiable 五维。结论原文:LLM"目前最有益的角色是辅助人类...不足以完全自动化需求质量保证"。
- **证据(high,arXiv:2404.17842,IEEE RE 2024):** 生成 SRS 时 GPT-4/CodeLlama 只达**入门级**(非资深)工程师质量;生成与评审是两种不同能力(GPT-4 能评审改进,CodeLlama 评审差)。
- **证据(medium,2025 LLM4RE SLR + 2026 AI requirements-quality SLR):** 最新综述没有推翻上述边界,反而强调研究多集中在 elicitation/validation 或问题检测,真实工业集成、复杂 workflow 和 actionable improvement 仍不足。
- **对 skill 的含义:** `spec-prd` 坚持 owner confirmation、最小阻塞提问、evidence tag、"不编造目标值"、readiness 由 LLM 判断而非脚本——**完全符合这条业界边界**。新增的 Adaptive Lens 明确写"This lens adapts the questions; it does not relax the evidence rules or replace owner confirmation",方向对。
- ⚠️ **校准提示(研究 caveat):** 上述证据基于 2023-era 模型(Llama 2 70B / GPT-4 / CodeLlama)和小/玩具数据集,部分发表于 RE@Next! 预研轨。**不应外推为"现代前沿模型的固有属性"。** skill 不要把"LLM 评审不可信"写死成永久假设。

### 2.2 ✅ "模糊/不完整需求会放大意图偏离,证据约束可缓解" —— 高置信,但需按 2026 v3 口径表述

- **证据(high,arXiv:2404.00971 v3,2026-01-21):** 该论文已更新为 "Beyond Functional Correctness: Exploring Hallucinations in LLM-Generated Code",最新摘要把代码幻觉组织为 3 个 primary categories / 12 个 detailed categories,包括 user intent、internal consistency、external real-world knowledge 等维度;并指出 syntax/semantic checks 不足以发现这类问题,需要 prompt-enhancing 等 mitigation。
- **对 skill 的含义:** 这正是 `spec-prd` 的 evidence-tag 纪律(`confirmed-source` 必须回源)+ Change Delta + 术语 glossary 的学理依据。**"先用证据约束、再让 LLM 判断"的整条主线被一手研究背书。**
- ⚠️ **caveat:** 旧版材料里的"`Requirement Conflicting` 最普遍"不应继续作为最新精确 claim。该论文研究对象仍是"LLM 生成代码",对"PRD 文档本身"是概念性可迁移而非直接证据;因此 `spec-prd` 应保持一次性、收敛式诊断重写,不要做无限 self-refine 循环。

### 2.3 ✅ 质量标准操作化工具(INVEST / EARS / Gherkin / 3C's) —— 多为一手源高置信

| 工具 | 证据 | skill 现状 | 建议 |
|---|---|---|---|
| **INVEST**(Independent/Negotiable/Valuable/Estimable/Small/Testable;可测试性是定义性特征) | high,缩写发明人 Bill Wake 自有站;2026 AI requirements-quality SLR 也用 INVEST 作为质量维度映射 | readiness 有 testability,但未显式用 INVEST 锚定"requirement quality" | Adaptive Lens 的"requirement quality: atomic/necessary/prioritized/testable"可显式映射 INVEST,提升可解释性 |
| **EARS**(While/When/Where/If-Then + 固定语序,为减少 NL 歧义而生) | high,创建者 Mavin 官方站 + RE'09 论文 | readiness 有 wording/testability,brainstorm 侧已提 EARS-style | 可在 Acceptance/wording 检查里引用 EARS 作为消歧句式(triggered,不强制) |
| **Gherkin Given/When/Then**(可执行规约,强制量化领域语言) | high,Cucumber 官方文档 | template 已用 `Given/When/Then` 验收骨架 + `AE-*` trace | ✅ 已对齐 |
| **3 C's + 验收标准 = 可测试性 + DoR** | medium,Atlassian 二手+Ron Jeffries 一手起源 | readiness 有 acceptance trace | ✅ 概念已覆盖 |

### 2.4 ✅ 棕地 = 变更影响分析(CIA) —— 中置信,框架级支撑

- **证据(medium,Bohner & Arnold 1996 经教科书共识,Wikipedia 二手引用):** CIA = "识别一次变更的潜在后果 / 估计需要改什么";三类技术 traceability-based / dependency-based / experiential;可追溯 IA 捕获 需求↔规约↔设计↔测试 链接来界定变更范围。
- **对 skill 的含义:** `evidence-and-topology.md` 的 Surface Map / Producer-Artifact-Consumer / Source-Of-Truth Resolution / 10 种 topology + Owner Question Ladder **就是 CIA 在 PRD 层的工程化落地**,而且做得比业界泛泛而谈更具体。这是 skill 相对通用 PRD 模板的**核心差异化优势**,应继续强化而非稀释。

### 2.5 ✅ PRD 组成要素 + 显式范围/假设 —— 中置信(厂商二手,柔性建议)

- Atlassian 8 节 / ProductPlan 要素 / Cagan 四类风险(价值/可用性/可行性/商业可行性)高度一致,可作**完整性 checklist**;显式"What we're not doing"和显式 Assumptions 是高价值要素。
- **对 skill 的含义:** template 的 core sections(Summary/Change Delta/Requirements/Acceptance/Scope Boundaries/Evidence And Assumptions)+ Scope Boundaries 的 In/Out + Evidence And Assumptions **已覆盖**。Cagan 四类风险可作为 Adaptive Lens "user/problem/outcome" 维度的可选 framing。
- ⚠️ 这些是 `recommended/just enough` 柔性建议,**不是硬性 must**——符合 skill 的 right-size 哲学,别因为"业界清单"就把所有 section 变成强制。

### 2.6 ❌ 两条被否决的 claim(避免写进 skill)

研究用 3-vote 对抗验证否决了两条,**skill 不应采纳**:

1. **"LLM 评审高召回低精确(catch most flawed, over-flag good)的数值化表述"** —— 0-3 否决。正确表述是**"负向偏置/过度标记"**,不要量化成精确率/召回率。
2. **"PRD 每个能力都必须有配套 use case 才能建立 requirement-to-test 可追溯"** —— 0-3 否决,证据不足。skill 现在用"core requirements have acceptance coverage **or an explicit trace gap**"——**这个'or explicit trace gap'的柔性表述反而比被否决的强制版更正确**,保持。

### 2.7 研究暴露的开放问题(skill 的真实能力边界)

- **没有可直接落地的"量化 PRD 评分卡"**:本轮复核未找到 Lenny/Reforge/SVPG 这类来源的可验证数值阈值 scorecard;2026 SLR 反而提示 actionable improvement 仍不足。→ **支持 skill 现在的选择:用定性 pack + LLM 判断,而非伪精确打分。不要造一个假的 0-100 评分 schema。**
- **缺"变更类型→PRD 深度"的业界决策矩阵**:CIA 给了影响面框架,但没给 right-sizing 规则。→ skill 的 `output_shape`(bypass/compact/normal/topology-heavy)+ surface_lens 实际上**自己造了这个矩阵**,是合理的工程补位,但要注意这是 project-local 发明,不是业界标准。

## 3. 代码深度分析:本次增强的质量复核

对 in-flight 编辑逐项审查(对照项目角色契约:Light contract / Explicit boundaries / Scripts prepare, LLM decides / 80/20)。

### 3.1 🟡 P3 — 单一真相源 drift:`quality_diagnosis` vs `quality_posture` 双名双枚举

同一个"PRD 质量档位"概念,出现两个名字 + 两套不一致的取值:

| 位置 | 字段名 | 枚举值 |
|---|---|---|
| `SKILL.md:97` Decision Card | `quality_diagnosis` | `not-run \| minor-gaps \| material-gaps \| blockers \| ready` |
| `prd-output-template.md:195` 诊断块 | `quality_posture` | `ready \| minor-gaps \| material-gaps \| blockers`(无 `not-run`) |

**严重度校准(我初判 P2,独立 reviewer 下调为 P3,我采纳 P3):** 这确实是真 drift——一个概念两个字段名违反"是否导致多真相源"。但 reviewer 给出了更精确的判断:**四个共享值完全一致,且两处语境不同**(scratch card 是 run-local 草稿 vs 诊断块是 emitted 输出),`not-run` 在输出块缺席**语义上是合理的**(refine/validate 输出里不会 emit 一个 not-run 档)。所以"产生错误行为"的概率低,真正的风险是**维护 churn**:未来 agent 改一处不知另一处分叉,可能反向"修"错。因此 P3,不是 P2——cheap to unify,但不阻塞 shipping。

**最小修复:** 统一字段名(两处都用同一个)+ 加一句"`not-run` 是 decision-card-only 状态,不在诊断块 emit"。无需新 schema。

### 3.2 🟡 P3 — 质量维度清单三处重复(carrying cost,且违反 readiness-lens 自己的纪律)

同一组质量维度(outcome / code-alignment / 需求原子性 / acceptance / metrics-priority / industry / planning-invention)出现在至少三处:

1. `SKILL.md:127` Phase 1 的诊断句(8 项平铺)
2. `prd-output-template.md:113-125` 的 Adaptive Product Expert Lens(7 个 bullet,措辞不同)
3. `prd-readiness-lens.md:22-61` 的 Core/Metrics packs(current-state provenance ≈ code alignment、wording/testability ≈ requirement quality、interaction/exception ≈ acceptance coverage、goal-measurability ≈ goals and metrics、overlay check ≈ industry overlay)

**为什么留意(独立 reviewer 强化了这条):** 这不只是 80/20 carrying cost——它**直接违反 `prd-readiness-lens.md:7` 自己立的纪律**:"Reuse the existing Requirements Readiness Gate by reference. Do not copy its full prose"。三份近重复清单措辞各有侧重(Phase 1 = "诊断什么"、Lens = "问什么"、Pack = "readiness 检查什么"),不是致命问题,但任何维度调整要改三处否则 drift。

**最小修复(reviewer 建议,我采纳):** 指定 `prd-output-template.md` 的 Adaptive Product Expert Lens 为**唯一 canonical 维度清单**;`SKILL.md` Phase 1 和 readiness Quality Diagnosis Pack 用 by-name 引用而非各自重列。保留 readiness pack 真正互补的 meta-check(`adaptive product lens fit` / `optimization suggestion closure` / `rewrite integrity`)——这三条是补充不是重复。

### 3.3 🟢 守得好的边界(确认无问题)

- **No new agent type:** template 明确"It is not an agent type, persona taxonomy, scoring schema, or separate reviewer workflow"——✅ 正确把"产品专家"约束为 run-local lens,没有膨胀成新 agent。符合角色契约"不要把 internal helper 暴露成用户入口"。
- **No second artifact topology:** 明确"Ensure there is no standalone quality report artifact unless the user explicitly asks;put...into Decision Notes / Evidence And Assumptions / Outstanding Questions"——✅ 优化建议折叠进重写后的 PRD,不产独立 critique 报告,守住 `docs/brainstorms/*-requirements.md` 单一链路。
- **Industry overlay 保持 advisory:** "only raises questions...never asserts an industry rule as confirmed truth"——✅ 与研究 §2.5"柔性建议非 must"一致,且符合"不把行业规则写成 confirmed"。
- **Scripts prepare / LLM decides:** 诊断/lens 全是 LLM 判断,`check-glossary-drift.js` 仍只产 `avoid_term_used` 字面事实——✅ 边界清晰。
- **WHAT not HOW:** 诊断维度"requirement atomicity / code alignment"措辞停在 WHAT 层(template 仍写"implementation units, schemas, exact API fields...are not")——✅ 但见 §3.4 的轻微风险。
- **source/runtime 边界:** 未读取/编辑 generated mirrors——✅。

### 3.4 🟡 P3 — "code alignment"维度的 HOW 渗漏风险(措辞预防)

Adaptive Lens 有"current-state and code alignment"维度。研究 §2.2 表明 code-evidence grounding 是对的,但要防止诊断滑向"这段代码该怎么改"。当前 template 靠通用的"implementation units...are not"兜底,**够用但不够显式**。建议在该维度后加半句:"code alignment confirms WHAT the system currently does, not HOW to change it"。低优先,属预防性。

### 3.5 🟡 fresh-source eval 缺口(持久证据仍未补)

原 `docs/validation/spec-prd/fresh-source-eval-2026-06-05-product-expert-quality-loop.md` 记 `status: not_run`(原编辑会话 Codex sub-agent 需用户显式授权,未跑语义 eval,只跑 15 个字符串 contract test)。

**为什么这曾是最该补的缺口:** 本次新增的是**纯 LLM 语义判断能力**(诊断、lens、建议)。字符串 contract test 只能证明"措辞存在",证明不了"agent 真的会在低质量 PRD 上做出对的诊断、不过度提问、不滑 HOW"。

**本轮复核结论:** 持久验证工件仍是 `not_run`。本文 §5 的 reviewer YAML 可作为 session-local advisory 证据,但在没有 transcript 或 validation artifact 更新前,不能写成"fresh-source eval 已通过/已完成"。这不改变 P3 finding 本身,但改变证据等级:direct source findings 是 confirmed,独立 reviewer verdict 是 advisory。

### 3.6 🟡 P3 — Quality Diagnosis Pack 触发面与 lens scope 轻微不一致(reviewer 发现)

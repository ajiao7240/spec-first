## 核心方法论

对于每个迭代周期，您必须：

1. **截取屏幕截图**：使用聚焦屏幕截图仅捕获目标元素/区域（见下文）
2. **分析**：确定 3-5 个可以增强设计的具体改进
3. **实施**：对代码进行有针对性的更改
4. **文档**：记录更改的内容以及原因
5. **Repeat**：继续指定的迭代次数

## 重点截图（重要）

**始终仅对您正在处理的元素或区域进行屏幕截图，而不是整个页面。**这可以保持上下文焦点并减少噪音。

### 设置：设置适当的窗口大小

在开始迭代之前，以 head 模式打开浏览器以查看并根据需要调整大小：
```bash
agent-browser --headed open [url]
```
推荐视口尺寸供参考：
- 小部件（按钮、卡片）：800x600
- 中等部分（英雄、特征）：1200x800
- 整页部分：1440x900

### 获取元素屏幕截图

1.首先通过`agent-browser snapshot -i`获取元素引用
2. 找到目标元素的引用（例如@e1、@e2）
3. 使用`agent-browser scrollintoview @e1`关注特定元素
4. 截图：`agent-browser screenshot output.png`

### 视口截图

对于重点截图：
1. 使用`agent-browser scrollintoview @e1`将元素滚动到视图中
2. 截取视口截图：`agent-browser screenshot output.png`

### 示例工作流程
```bash
1. agent-browser open [url]
2. agent-browser snapshot -i  # Get refs
3. agent-browser screenshot output.png
4. [analyze and implement changes]
5. agent-browser screenshot output-v2.png
6. [repeat...]
```
**保持屏幕截图集中** - 仅捕获您正在处理的元素/区域以减少噪音。

## 应用的设计原则

分析组件时，寻找以下领域的机会：

### 视觉层次结构

- 标题大小和权重级数
- 色彩对比和强调
- 留白和呼吸空间
- 部分分离和分组

### 现代设计模式

- 渐变背景和微妙的图案
- 微交互和悬停状态
- 徽章和标签样式
- 图标处理（大小、颜色、背景）
- 边界半径一致性

### 版式

- 字体配对（衬线标题、无衬线正文）
- 行高和字母间距
- 文本颜色变化（slate-900、slate-600、slate-400）
- 关键短语的斜体强调

### 布局改进

- 英雄卡牌图案（特色物品较大）
- 网格排列（不对称可以更有趣）
- 视觉节奏的交替模式
- 适当的响应断点

### 波兰细节

- 阴影深度和颜色（蓝色按钮的蓝色阴影）
- 动画元素（微妙的脉冲、过渡）
- 社会证明徽章
- 信任指标
- 编号或标签的物品

## 竞争对手研究（如有要求）

如果被要求研究竞争对手：

1. 导航至 2-3 个竞争对手网站
2. 相关部分截图
3. 提取他们使用的具体技术
4. 在后续迭代中应用这些见解

流行设计参考：

条纹：干净的渐变、深度、优质的感觉
- 线性：黑暗主题，简约，集中
- Vercel：排版前卫、自信的空白
- 理念：友好、平易近人、插图前卫
- Mixpanel：数据可视化，价值道具清晰
- Wistia：对话文案，问题式标题

## 迭代输出格式

对于每次迭代，输出：
```
## Iteration N/Total

**What's working:** [Brief - don't over-analyze]

**ONE thing to improve:** [Single most impactful change]

**Change:** [Specific, measurable - e.g., "Increase hero font-size from 48px to 64px"]

**Implementation:** [Make the ONE code change]

**Screenshot:** [Take new screenshot]

---
```
**规则：如果您无法确定一项明显的改进，则设计已完成。停止迭代。**

## 重要准则

- **仅进行小更改** - 每次迭代进行 1-2 次有针对性的更改，切勿更多
- 每个更改都应该具体且可衡量（例如，“将标题大小从 24 像素增加到 32 像素”）
- 在每次更改之前，决定：“现在最能改善这一点的一件事是什么？”
- 不要撤消之前迭代中的良好更改
- 逐步构建 - 早期迭代侧重于结构，后期侧重于完善
- 始终保留现有功能
- 牢记可访问性（对比度、语义 HTML）
- 如果某件事看起来不错，就不要管它 - 抵制“改进”工作元素的冲动

## 开始迭代周期

调用时，您应该：

### 第 0 步：检查上下文中的设计技巧

**当用户调用时，swiss-design、frontend-design 等设计技能会自动加载。** 检查您的上下文以获取活动技能说明。

如果用户提到设计风格（瑞士、简约、条纹等），请查找：
- 在您的系统上下文中加载技能说明
- 在所有迭代中应用这些原则

从任何负载的设计技能中提取的关键原则：
- 网格系统（列、排水沟、基线）
- 版式规则（比例、对齐方式、层次结构）
- 色彩哲学
- 布局原则（不对称、空白）
- 要避免的反模式

### 步骤 1-5：继续迭代周期

1. 确认目标组件/文件路径
2. 确认请求的迭代次数（默认：10）
3. （可选）确认要研究的任何竞争对手网站
4. 使用 `agent-browser` 设置浏览器以获取适当的视口
5. 以加载的技能原则开始迭代周期首先拍摄目标元素的初始屏幕截图以建立基线，然后继续进行系统改进。

避免过度设计。仅进行直接要求或明显必要的更改。保持解决方案简单且重点突出。不要添加功能、重构代码或进行超出要求的“改进”。错误修复不需要清理周围的代码。简单的功能不需要额外的可配置性。不要为不可能发生的场景添加错误处理、后备或验证。信任内部代码和框架保证。仅在系统边界（用户输入、外部 API）进行验证。当您只能更改代码时，不要使用向后兼容的垫片。不要为一次性操作创建助手、实用程序或抽象。不要为假设的未来需求进行设计。适当的复杂度是当前任务所需的最低限度。尽可能重用现有的抽象并遵循 DRY 原则。

在提出代码编辑建议之前，请务必阅读并理解相关文件。不要猜测您没有检查过的代码。如果用户引用特定文件/路径，您必须在解释或提出修复之前打开并检查它。严格并坚持不懈地在代码中搜索关键事实。在实现新功能或抽象之前，彻底检查代码库的风格、约定和抽象。

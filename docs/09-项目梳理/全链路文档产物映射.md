# Spec-First 全链路文档产物映射

本文按当前仓库中的 `README.md`、`README-CN.md`、`specs/*/stage-state.json` 与 `docs/07-用户文档/使用手册.md` 进行整理，目标是回答两个问题：

1. 全链路会生成哪些文档。
2. 每个流程节点对应哪些文档产物。

## 一、全链路主流程

Spec-First 的主链路是一个 8 个活跃阶段 + 2 个终态的状态机：

`00_init -> 01_specify -> 02_design -> 03_plan -> 04_implement -> 05_verify -> 06_wrap_up -> 07_release -> 08_done`

`09_cancelled` 是可从任意活跃阶段进入的终态。

## 二、流程节点与文档产物

下表优先按仓库里真实的 `stage-state.json` / README 口径整理，列出每个节点通常会产出的文档。

| 节点 | 主要文档产物 | 说明 |
|---|---|---|
| `00_init` | `stage-state.json`、目录骨架、配置文件 | 初始化 Feature 工作区，建立后续阶段的状态容器。 |
| `01_specify` | `spec.md`、`impact-analysis.md` | 需求规格与变更影响分析。Mode I 场景通常必须有 `impact-analysis.md`。 |
| `02_design` | `design.md`、`api-contract.yaml`、`contracts/openapi.yaml`、`data-model.md` | 技术设计和接口/数据契约。部分 Feature 还会产生 `adr/`、`ui-information-architecture.md` 等。 |
| `03_plan` | `task_plan.md` | 任务拆解与执行计划，通常带 traceability ID。 |
| `04_implement` | 源代码、`traceability-matrix.md` 更新、`findings.md` 更新 | 这个阶段的主产物是代码；文档侧通常是追踪矩阵和发现记录的增量更新。 |
| `05_verify` | `reports/test-report.md`、`reports/regression-report.md` | 验证阶段的测试与回归证据。部分项目还会产出性能、安全等补充报告。 |
| `06_wrap_up` | `retro.md` | 复盘与归档记录。 |
| `07_release` | `reports/smoke-test-report.md`、`reports/release-note.md` | 发布阶段的上线验证与发布说明。 |
| `08_done` | 无新增标准文档 | 终态，只做收口。 |
| `09_cancelled` | 无新增标准文档 | 终态，只记录取消状态。 |

## 三、当前仓库里可见的标准文档集

结合 `README.md` 中的阶段定义，以及 `specs/*/stage-state.json` 中的 deliverables 字段，当前仓库的标准文档产物可以分成四类：

### 1. 需求类

- `spec.md`
- `impact-analysis.md`
- `user-stories.md` 通过可选产物出现

### 2. 设计类

- `design.md`
- `api-contract.yaml`
- `contracts/openapi.yaml`
- `data-model.md`
- `adr/` 目录
- `ui-information-architecture.md`
- 其他平台/项目特有设计文件，如 `system-architecture.md`、`page-routing-map.md`

### 3. 计划与执行类

- `task_plan.md`
- `findings.md`
- `traceability-matrix.md`
- `stage-state.json`

### 4. 验证与交付类

- `reports/test-report.md`
- `reports/regression-report.md`
- `reports/smoke-test-report.md`
- `reports/release-note.md`
- 其他可选报告，如 `reports/perf-report.md`、`reports/security-scan.md`、`reports/api-change-log.md`

## 四、运行态文件

这几类文件不是阶段“交付物”本身，但它们是全链路运行必需的上下文文件：

- `stage-state.json`：当前 Feature 的阶段状态、历史流转、门禁规则、交付物定义。
- `traceability-matrix.md`：需求、设计、任务、测试之间的追踪映射。
- `gate-history.jsonl`：门禁评估记录。
- `findings.md`：阶段推进中的发现、决策、偏差记录。

从实际仓库结构看，`stage-state.json` 是最核心的流程总账，很多自动化判断都会围绕它展开。

## 五、按项目类型变化的可选产物

不同 Feature 会因为平台、规模、模式不同而增加额外文档。仓库里的真实示例包括：

- Web / Admin 场景：`reports/admin-bundle-analysis.md`、`reports/bundle-analysis.md`
- Python 后端场景：`reports/python-service-slo.md`
- 移动端场景：`reports/apk-size.md`、`reports/ipa-size.md`
- 数据迁移场景：`reports/migration-execution-log.md`
- 风险较高的改造：`risk-matrix.md`、`known-exceptions.md`
- 架构决策补充：`adr/ADR-001-test-strategy.md`

这些都不是每个 Feature 的强制产物，但会在对应的 `stage-state.json` 中作为可选 deliverables 出现。

## 六、文档内部关联字段关系

这一部分回答“这些文档彼此怎么串起来”。当前仓库里最重要的关联主轴是 `featureId`，所有阶段文档、运行态文件和追踪矩阵都围绕它展开。

### 1. `stage-state.json` 的字段关系

`stage-state.json` 是状态机总账，核心字段之间的关系如下：

| 字段 | 作用 | 关联对象 |
|---|---|---|
| `featureId` | Feature 的全局主键 | 目录名、所有阶段产物、`traceability-matrix.md`、`gate-history.jsonl` |
| `mode` | 决定是新建还是增量迭代 | 影响 `impact-analysis.md` 是否必需，以及部分门禁规则 |
| `size` | 决定裁剪力度和产物密度 | 影响 `spec.md`、`design.md`、`reports/*` 的详细程度 |
| `platforms` | 标记平台范围 | 影响设计/验证文档的分支产物，例如 Web、后端、移动端特有报告 |
| `backgroundInputStatus` | 标记背景输入是否完整 | 影响初始化和补齐文档的深度 |
| `stageStatus` | 当前阶段处理状态 | 决定是 drafting、ready 还是 terminal 前的待推进状态 |
| `autoAdvancePolicy` | 自动推进策略 | 决定门禁通过后是建议推进还是自动推进 |
| `currentStage` | 当前所处阶段 | 直接决定当前应产出的文档类型 |
| `history` | 阶段流转日志 | 与 `gate-history.jsonl` 互补，用于回放路径 |
| `terminal` | 是否已终态 | 终态为 `true` 时不再产生新的阶段性产物 |
| `mergedRules.gateConditions` | 各阶段门禁规则 | 以阶段名为 key，决定该阶段是否可推进 |
| `mergedRules.deliverables` | 各阶段交付物清单 | 以阶段名为 key，定义该阶段应生成哪些文档 |
| `mergedRules.thresholds` | 指标阈值 | 影响验证、发布、健康评分等检查 |

### 2. `mergedRules` 和阶段文档的绑定关系

`mergedRules` 里最关键的是两个映射：

| 关系 | 含义 |
|---|---|
| `mergedRules.deliverables[stage]` -> 文档清单 | 规定某个阶段需要输出哪些文件 |
| `mergedRules.gateConditions[stage]` -> 校验条件 | 规定某个阶段要满足哪些门禁才能进入下一阶段 |

这意味着阶段与文档不是“松散建议关系”，而是显式绑定关系。例如：

- `01_specify` 绑定 `spec.md` 和 `impact-analysis.md`
- `02_design` 绑定 `design.md`、`api-contract.yaml`、`contracts/openapi.yaml`
- `03_plan` 绑定 `task_plan.md`
- `05_verify` 绑定 `reports/test-report.md`、`reports/regression-report.md`
- `06_wrap_up` 绑定 `retro.md`
- `07_release` 绑定 `reports/smoke-test-report.md`、`reports/release-note.md`

### 3. 当前模型的问题与收敛方向

当前仓库里的 `traceability-matrix.md` 原本承担的是条目级关系映射，但这个模型已经被证明过于复杂、过于脆弱。

因此更合理的收敛方向不是继续维护：

- `FR`
- `DS`
- `TASK`
- `TC`

这种条目级追踪链，而是退回到文档级关系：

- `spec.md` 引用或作为 `design.md` 的上游文档
- `design.md` 引用或作为 `task_plan.md` 的上游文档
- `task_plan.md` 引用或作为验证报告的上游文档

也就是说，未来更推荐的模型是：

`spec.md -> design.md -> task_plan.md -> reports/* -> retro.md`

### 4. 各文档之间的常见引用链

| 文档 | 主要引用谁 | 主要被谁引用 |
|---|---|---|
| `spec.md` | `featureId`、业务范围、验收描述 | `design.md`、`impact-analysis.md`、验证报告 |
| `impact-analysis.md` | `spec.md`、现状基线、变更范围 | `specify` 阶段门禁、设计输入 |
| `design.md` | `spec.md`、接口契约、方案说明 | `task_plan.md`、验证报告 |
| `api-contract.yaml` / `contracts/openapi.yaml` | 接口定义、schema | `design.md`、代码实现、验证报告 |
| `task_plan.md` | `design.md`、执行范围、文件路径 | `code` 阶段、验证报告 |
| `traceability-matrix.md` | 当前仍承担条目级追踪 | `gate`、`verify`、`archive`、`review` |
| `reports/test-report.md` / `reports/regression-report.md` | `spec.md`、`design.md`、`task_plan.md`、执行结果 | `05_verify`、`07_release` |
| `retro.md` | `history`、`gate-history.jsonl`、`findings.md` | `06_wrap_up`、交接与回顾 |
| `reports/smoke-test-report.md` / `reports/release-note.md` | 验证结论、发布范围、变更摘要 | `07_release`、`08_done` 收口 |

### 5. 运行态文件之间的关系

运行态文件不只是并列存在，它们的关系也固定：

- `stage-state.json` 负责“当前状态是什么”
- `gate-history.jsonl` 负责“哪些门禁在什么时候怎么判定”
- `findings.md` 负责“过程中发现了什么、怎么决策的”
- `traceability-matrix.md` 负责“文档和需求怎么连起来”

可以把它们理解成：

`stage-state.json` 是状态中心，`gate-history.jsonl` 是事件日志，`traceability-matrix.md` 是关系索引，`findings.md` 是过程记录。

## 七、一个最小闭环视图

如果只看“最小可闭环文档链”，通常可以记成：

`spec.md -> design.md -> task_plan.md -> code + traceability-matrix.md -> reports/* -> retro.md -> smoke-test-report.md + release-note.md`

其中 `impact-analysis.md` 是 Mode I 的关键补充，`stage-state.json` 则贯穿整个链路，负责状态与门禁控制。

## 九、与 `get-shit-done` 的对比

你前面问的是“矩阵字段关系有没有必要”。`get-shit-done` 的实现是一个很好的对照样本：它保留了追踪关系，但把关系压缩到了更稳定的最小模型。

| 维度 | Spec-First | get-shit-done | 结论 |
|---|---|---|---|
| 关系粒度 | 当前是条目级多层 ID 体系 | 主要是 `REQ -> Phase -> Status` | Spec-First 过于细；GSD 更轻 |
| 关系载体 | `traceability-matrix.md` + `stage-state.json` + 各阶段文档 | `REQUIREMENTS.md` 中的一张追踪表 + `ROADMAP.md` + `STATE.md` | GSD 只保留一张最小追踪表 |
| 状态职责 | `stage-state.json` 同时承载状态、门禁、deliverables、阈值 | `STATE.md` 承担当前状态，`ROADMAP.md` 承担阶段，`REQUIREMENTS.md` 承担追踪 | GSD 职责更分离 |
| 更新方式 | 多文件联动，关系容易被中途变更打穿 | 完成阶段时同步更新 requirements traceability table | GSD 的关系更新更局部 |
| 变更抗性 | 容易受需求变更、bug 回退、阶段重排影响 | 需求变更主要影响 REQUIREMENTS + ROADMAP，不会牵连多层矩阵 | GSD 更抗变更 |
| 工具实现 | 矩阵与状态机强绑定 | 代码里仅更新 checkbox + traceability status | GSD 没有做复杂矩阵引擎 |

### 对照结论

`get-shit-done` 证明了一点：**追踪关系本身是有必要的，但不需要复杂矩阵字段。**

它更接近下面这种稳定模型：

`Requirement -> Phase -> Status`

而不是：

`条目级关系 -> 门禁 -> 阈值 -> 状态 -> 运行态 -> 交付物`

这说明如果你的目标是提高健壮性，最好的方向不是增加矩阵字段，而是把矩阵降级成“轻量索引”：

- 需求层只保留需求与阶段的映射
- 状态层只保留当前阶段和进度
- 交付物层留在各自文档里
- 验证结果留在验证报告里

### 对当前 spec-first 的启发

如果要向更稳的方向收敛，可以考虑：

- 把 `traceability-matrix.md` 简化为“关系索引”
- 把 `stage-state.json` 收缩为“状态快照”
- 把变更、bug、回退写回原始源文档，而不是继续扩充矩阵字段

这样才能避免“中途一变更，整张矩阵就失真”的问题。

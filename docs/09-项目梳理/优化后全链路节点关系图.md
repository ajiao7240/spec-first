# 优化后全链路节点关系图

本文档描述当前优化后代码中的全链路节点关系，重点说明三件事：

1. 文档之间如何关联
2. Gate 如何校验这些文档
3. Skill 如何在阶段流中读写这些文档

当前模型已经从“条目级追踪矩阵”收敛为“文档级关联索引”：

- 不再维护 `FR / DS / TASK / TC` 的多跳矩阵
- 只保留文档级引用关系
- `document-links.yaml` 是关系索引真源
- `stage-state.json` 是阶段状态真源
- `findings.md` 是过程结论与证据落盘点

## 节点分类

### 1. 文档节点

| 节点 | 作用 | 读写方式 |
|------|------|----------|
| `prd.md` | 业务目标与问题定义 | `spec` / `design` / `status` 读取 |
| `spec.md` | 功能需求与验收标准 | `spec` 写入，`design` / `task` / `verify` 读取 |
| `design.md` | 技术设计与接口契约 | `design` 写入，`task` / `verify` 读取 |
| `task_plan.md` | 任务拆解与执行顺序 | `task` 写入，`code` / `orchestrate` 读取 |
| `document-links.yaml` | 文档关联索引 | `spec links validate` / `sync` / `verify` 读写 |
| `stage-state.json` | 当前阶段与状态 | `init` / `stage advance` / `orchestrate` 读写 |
| `findings.md` | 发现、风险、决策、阻塞 | `spec` / `design` / `task` / `code` / `verify` / `sync` 追加 |
| `reports/*` | 验证与分析证据 | `verify` / `archive` 写入 |
| `retro.md` | 复盘总结 | `archive` 写入 |

| Gate | 触发阶段 | 对应 Skill 节点 | 主要职责 | 关键检查 | Gate 判断规则 |

## 关键关系

### 1. `stage-state.json` 是状态中心

它回答的是：

- 当前在哪个阶段
- 当前状态是否允许推进
- 最近一次阶段流转是什么

它不保存文档内容，也不保存条目级矩阵关系。

### 2. `document-links.yaml` 是关系中心

它回答的是：

- 哪些文档互相引用
- 哪些文档是当前阶段的直接上下游
- 哪些引用已断链

它只表达文档级关联，不再表达 FR/DS/TASK/TC 的多跳关系。

### 3. `findings.md` 是证据中心

它回答的是：

- 为什么做这个决策
- 验证命令输出是什么
- 当前阻塞项是什么
- 下一步应该执行什么

### 4. Gate 是真伪裁判

Gate 不负责“生成内容”，只负责判断：

- 必需文档是否存在
- 文档关联是否可解析
- 证据是否足够推进

---

## Gate 与文档的关系

Gate 和文档不是并列竞争关系，而是分工关系：

- **文档负责产出内容**，例如 `spec.md`、`design.md`、`task_plan.md`、`retro.md`
- **`document-links.yaml` 负责表达文档关联**，例如哪些文档是上下游引用
- **`stage-state.json` 负责表达阶段状态**，例如当前是否处于 `02_design`
- **Gate 负责判断能否推进**，即当前阶段所需文档、关联、证据是否齐备

因此，Gate 的卡点不再是“条目级矩阵是否填满”，而是：

1. 当前阶段该有的文档是否存在
2. 文档之间的引用是否可解析
3. 当前阶段的证据是否足够审计
4. 阶段状态是否与工作流一致

---

## 补充卡点规则

### 1. 硬卡点

满足任一条即阻断，不允许推进：

- 当前阶段要求的核心文档缺失
- `document-links.yaml` 缺失
- `document-links.yaml` 存在断链引用或无法解析的目标
- `stage-state.json` 与当前阶段不一致
- 当前阶段要求的验证证据缺失
- Gate 条件本身失败

### 2. 软卡点

不阻断，但必须提醒用户：

- 说明性文档缺失，但不影响当前阶段主目标
- `findings.md` 过旧，但阶段主文档完整
- 可选报告缺失，例如非强制的 `research.md`、`retro.md`
- 文档引用存在，但尚未覆盖到非关键附属文档

### 3. 阶段卡点逻辑

- `01_specify`：`prd.md`、`spec.md`、`document-links.yaml` 必须齐
- `02_design`：`design.md` 必须齐，且文档关联可解析
- `03_plan`：`task_plan.md` 必须齐，且关联完整
- `04_implement`：代码和 `findings.md` 必须有证据
- `05_verify`：测试结果、`gate check`、`docs links validate`、`metrics report` 必须齐
- `06_wrap_up`：`retro.md` 和归档证据必须齐，关联必须闭合

### 4. 当前模型的核心判定

- 内容缺失卡文档
- 断链卡关联
- 证据不足卡 Gate
- 状态不一致卡推进

## 最小闭环

```text
spec.md
  -> design.md
  -> task_plan.md
  -> code / tests
  -> reports/*
  -> retro.md
  -> document-links.yaml 校验通过
```

如果某个阶段出现变更、bug 或回退，不修改矩阵结构，只做三件事：

1. 更新源文档
2. 追加 `findings.md`
3. 重新运行 `docs links validate` / `gate check`

---

## 结论

优化后的全链路节点关系可以概括为：

- `stage-state.json` 管状态
- `document-links.yaml` 管关系
- `findings.md` 管证据
- `Gate` 管推进
- `Skill` 管产物生成

这套结构比条目级追踪矩阵更稳，原因是它把复杂关系收缩成了可重算的文档级索引。

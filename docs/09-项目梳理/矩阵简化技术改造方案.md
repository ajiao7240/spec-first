# Spec-First 文档关联简化技术改造方案

> 日期：2026-03-23
> 范围：`spec-first`
> 决策前提：**不考虑向下兼容**
> 参考对象：`get-shit-done` 的轻量追踪方式

## 1. 结论

这次改造不再保留矩阵，也不再保留 `FR / DS / TASK / TC` 这类中间关系模型。

目标模型只保留三件事：

- 阶段状态
- 文档本身
- 文档级关联

也就是说，后续系统不再回答：

- 哪个 FR 对应哪个 DS
- 哪个 TASK 对应哪个 TC
- 哪条设计项映射哪条需求项

系统只回答：

- 当前 Feature 在哪个阶段
- 当前阶段需要哪些文档
- 这些文档引用了哪些其他文档
- 这些文档是否已经存在、是否满足基本检查

## 2. 现状问题

当前模型的问题不只是“字段太多”，而是建模粒度错了。

它把真实工作流中的文档，强行拆成了很多中间关系对象：

- `FR`
- `DS`
- `TASK`
- `TC`
- 以及其他扩展 ID

这会带来三个问题：

1. **关系过密**
任何一次需求变更、返工、补测、文档重写，都会牵动多条关系链。

2. **维护成本高**
需要维护的不是文档，而是文档背后的抽象对象和映射表。

3. **收益不足**
这些细粒度关系并没有形成足够稳定的自动化价值，反而放大了脆弱性。

因此，应该直接删掉这一层。

## 3. 新模型

## 3.1 状态层

保留：

- `specs/{featureId}/stage-state.json`

职责：

- 当前阶段
- 当前状态
- 历史流转
- 阶段所需文档
- Gate 所需规则引用

不再承担：

- 需求到设计到任务的关系存储
- 覆盖率原始输入
- 复杂 profile 合并

## 3.2 文档层

保留并强化这些源文档：

- `spec.md`
- `impact-analysis.md`
- `design.md`
- `task_plan.md`
- `findings.md`
- `retro.md`
- `reports/*.md`

原则：

- 每份文档只写自己的内容
- 系统只把文档当作一等对象
- 不再从文档里抽象出 `FR / DS / TASK / TC`

## 3.3 关联层

新增一个最小文档关联文件：

- `specs/{featureId}/document-links.yaml`

它只记录：

- 当前 Feature 有哪些文档
- 某份文档引用了哪些其他文档

它不记录：

- 文档内锚点
- 文档内条目之间的关系
- 中间对象类型
- 覆盖率关系

## 4. 新的文件结构

```text
specs/{featureId}/
├── stage-state.json
├── document-links.yaml
├── spec.md
├── impact-analysis.md
├── design.md
├── task_plan.md
├── findings.md
├── retro.md
├── gate-history.jsonl
└── reports/
   ├── test-report.md
   ├── regression-report.md
   ├── smoke-test-report.md
   └── release-note.md
```

## 5. 数据结构设计

## 5.1 `stage-state.json`

建议收缩为：

```json
{
  "featureId": "FSREQ-20260323-TRACE-001",
  "title": "Trace model simplification",
  "mode": "I",
  "size": "M",
  "platforms": ["web", "backend"],
  "currentStage": "03_plan",
  "stageStatus": "ready",
  "terminal": false,
  "history": [
    {
      "from": "00_init",
      "to": "01_specify",
      "timestamp": "2026-03-23T10:00:00.000Z",
      "gateResult": "PASS"
    }
  ],
  "deliverables": {
    "01_specify": ["spec.md", "impact-analysis.md"],
    "02_design": ["design.md"],
    "03_plan": ["task_plan.md"],
    "05_verify": ["reports/test-report.md", "reports/regression-report.md"],
    "06_wrap_up": ["retro.md"],
    "07_release": ["reports/smoke-test-report.md", "reports/release-note.md"]
  },
  "gatePolicy": {
    "01_specify": ["required_docs_exist"],
    "02_design": ["required_docs_exist"],
    "03_plan": ["required_docs_exist"],
    "05_verify": ["required_docs_exist"],
    "06_wrap_up": ["required_docs_exist"]
  }
}
```

删除：

- `mergedRules.thresholds`
- 复杂 deliverables 元数据
- 与矩阵和 coverage 绑定的结构

## 5.2 `document-links.yaml`

建议结构如下：

```yaml
version: 1
featureId: FSREQ-20260323-TRACE-001
documents:
  - path: spec.md
    kind: requirements
    stage: 01_specify
    references: []
  - path: impact-analysis.md
    kind: impact
    stage: 01_specify
    references:
      - spec.md
  - path: design.md
    kind: design
    stage: 02_design
    references:
      - spec.md
      - impact-analysis.md
  - path: task_plan.md
    kind: execution_plan
    stage: 03_plan
    references:
      - design.md
  - path: reports/test-report.md
    kind: verification
    stage: 05_verify
    references:
      - spec.md
      - design.md
      - task_plan.md
  - path: retro.md
    kind: archive
    stage: 06_wrap_up
    references:
      - findings.md
      - reports/test-report.md
```

这里没有：

- 条目 ID
- 文档内局部锚点
- 关系类型白名单
- 多跳关系计算

只有：

- 文档路径
- 文档类型
- 所属阶段
- 引用的其他文档

## 6. 新的系统行为

## 6.1 文档关联规则

系统以后只做两类检查：

1. 文档是否存在
2. 文档引用的其他文档是否存在

不再检查：

- 设计项是否覆盖需求项
- 任务是否覆盖设计项
- 测试是否映射到某个需求 ID

这些判断统一降级为：

- 设计文档是否引用需求文档
- 任务计划是否引用设计文档
- 验证报告是否引用需求/设计/计划文档

## 6.2 Skill 行为改造

Skill 的职责改为：

- 写源文档
- 更新 `document-links.yaml`
- 更新 `stage-state.json`

不再做：

- `matrix update`
- 生成或维护 `FR / DS / TASK / TC`
- coverage 驱动的关系回填

例如：

- `spec`
  - 写 `spec.md`
  - 在 `document-links.yaml` 中注册 `spec.md`
- `design`
  - 写 `design.md`
  - 在 `document-links.yaml` 中将 `design.md` 指向 `spec.md`
- `task`
  - 写 `task_plan.md`
  - 在 `document-links.yaml` 中将 `task_plan.md` 指向 `design.md`
- `verify`
  - 写验证报告
  - 在 `document-links.yaml` 中将验证报告指向 `spec.md`、`design.md`、`task_plan.md`

## 6.3 Gate 行为改造

Gate 改成只做文档级检查。

每个阶段只检查：

- 需要的文档是否存在
- 需要的文档关联是否存在

例如：

- `01_specify`
  - `spec.md` 存在
  - 若 `mode=I`，`impact-analysis.md` 存在

- `02_design`
  - `design.md` 存在
  - `design.md` 在 `document-links.yaml` 中引用了 `spec.md`

- `03_plan`
  - `task_plan.md` 存在
  - `task_plan.md` 在 `document-links.yaml` 中引用了 `design.md`

- `05_verify`
  - 验证报告存在
  - 验证报告在 `document-links.yaml` 中引用了前序文档

这样 Gate 不再需要理解复杂的业务关系，只需要理解文档流。

## 7. CLI 改造

建议删除：

- `spec-first matrix check`
- `spec-first matrix update`
- `spec-first matrix export`
- 旧的 coverage 计算相关命令

建议新增：

- `spec-first docs links validate <featureId>`
- `spec-first docs links rebuild <featureId>`
- `spec-first docs links export <featureId> --format md`
- `spec-first docs audit <featureId>`

职责如下：

- `docs links validate`
  - 校验 `document-links.yaml` 结构
  - 校验引用的文档是否存在

- `docs links rebuild`
  - 从现有文档重建 `document-links.yaml`

- `docs links export`
  - 输出文档引用视图，供人阅读

- `docs audit`
  - 校验当前阶段的必需文档与必需引用

## 8. Metrics 改造

现有 C1-C9 体系应该直接移除。

替换为更简单的指标：

- `required_doc_count`
- `missing_doc_count`
- `linked_doc_count`
- `broken_reference_count`
- `stage_ready`

这些指标只反映文档链路是否完整，不反映细粒度语义覆盖率。

这是刻意收缩，而不是能力退化。

原因很简单：

- 当前系统并没有稳定地证明自己能维护细粒度语义覆盖率
- 但它完全可以稳定地维护文档链路完整性

## 9. 直接删除的概念

由于不考虑向下兼容，以下概念可以直接退场：

- `traceability-matrix.md`
- `traceability-matrix.yaml`
- `FR`
- `DS`
- `TASK`
- `TC`
- `RFC`
- `REQ / SYS / ARCH / MOD / ATP / STP / ITP / UTP` 这套矩阵型追踪对象
- 基于这些对象的 coverage 计算

保留的唯一“追踪”概念变成：

- 文档是否引用了前序文档

## 11. 迁移顺序

### P0

- 定义 `document-links.yaml` schema
- 收缩 `stage-state.json`
- 新增 `docs links validate`

### P1

- Skill 改为只写文档和文档引用
- 新增 `docs audit`
- Gate 切换到文档检查模型

### P2

- 删除 `traceability-matrix.*`
- 删除 `matrix` 命令
- 删除 coverage 逻辑
- 重写 status / metrics / dashboard

## 12. 风险与收益

### 风险

- 旧的矩阵型工具和文档会整体失效
- 很多现有测试需要重写
- status / metrics 展示要重做

### 收益

- 数据结构显著简单
- 需求变更和返工时不再需要维护多跳关系
- Gate 规则更容易理解和稳定执行
- Skill 写入逻辑更直接

## 13. 最终定义

这次改造的目标不是“保留关系但简化字段”，而是：

**删除条目级追踪，退回到文档级关联。**

系统以后维护的最小事实只有：

- 当前在哪个阶段
- 当前阶段应该有哪些文档
- 这些文档之间是否有基本引用关系

这比旧模型弱，但更真实，也更稳。

# First Skill 质量保障设计

> 日期: 2026-02-28 | 版本: v1.0 | 状态: 已批准

## 背景

`spec-first:first` skill 自动生成最多 10 份项目认知文档。当前质量保障仅停留在声明层面（"以代码为准，禁止捏造"），缺少执行层面的验证机制。需要系统性方案确保产出的事实性、完整性和一致性。

## 方案选型

评估了三种方案：

| 方案 | 思路 | 结论 |
|------|------|------|
| A 证据前置 | 重预防，agent 强制取证后写作 | 无法检测遗漏，无跨文档校验 |
| B 独立校验 | 重校验，P5 新增 Validator Agent | 额外 30-60s + 8K tokens，成本高 |
| **C 分层混合** | **三层防御纵深** | **选定方案，性价比最优** |

## 设计：三层校验模型

### 第一层：Agent 证据协议（预防）

**原则**：先取证，后写作。Agent 生成过程中无法偏离事实。

三类关键事实必须有工具调用证据：

| 类别 | 示例 | 证据来源 |
|------|------|----------|
| 存在性声明 | "项目使用了 Redis" | Grep/Serena 命中结果 |
| 数值性声明 | "TypeScript ^5.4.0" | Read 文件具体行 |
| 关系性声明 | "module A 依赖 module B" | Serena find_referencing_symbols 或 import 扫描 |

证据标注格式（嵌入文档）：

```
- <结论> (`<file_path>:<line>` — `<关键代码片段>`)
```

非关键事实（描述性文字）不强制取证，避免过度开销。无证据的关键事实标注 `[待确认]`。

**改动范围**：SKILL.md 核心约束 + 5 个 agent 规格文件。

### 第二层：P5 抽样验证（检测）

P5 汇总阶段，对高风险内容做 9 次自动化抽检：

| 规则 | 检查内容 | 方法 | 失败处理 |
|------|----------|------|----------|
| S1 | API 端点存在性（×3） | Grep 源码搜索路由定义 | 标注 `[待确认]` |
| S2 | 依赖版本准确性（×3） | Read 依赖声明文件比对 | 直接修正 |
| S3 | 模块关系真实性（×3） | Grep 搜索 import 语句 | 标注 `[待确认]` |

9 次中 ≥3 次失败 → 输出全局质量警告。预计耗时 10-15s。

### 第三层：跨文档一致性（集成验证）

紧接抽检之后，主线程做集合运算比对：

| 规则 | 校验内容 | 比对文档 |
|------|----------|----------|
| C1 | 技术栈覆盖：tech-stack ⊇ 其他文档引用的技术 | tech-stack ↔ 全部 |
| C2 | 模块一致：codebase-overview 模块集 = architecture 模块集 | A1 ↔ A2 |
| C3 | 外部服务闭环：external-deps ⊆ local-setup 所需服务 | C1 ↔ C2 |
| C4 | DB 类型一致：tech-stack 声明 = database-er 实际连接 | tech-stack ↔ D |

不一致时：回溯代码确认正确方 → 修正错误文档 → README 追加修正记录。预计耗时 5-10s。

## 输出与处理

### 不一致处理流程

1. 回溯代码源文件，确认哪个 agent 的结论正确
2. 修正错误文档中的对应内容
3. README.md 底部追加修正记录（无不一致时不生成此节）：
   ```
   ## ⚠️ 交叉验证修正记录
   - [V2] architecture.md 模块列表与 codebase-overview.md 不一致，以代码实际目录结构为准，已修正 architecture.md
   ```

### 证据抽检

随机抽取各文档中 2-3 条结论，验证 `(file:line)` 标注是否真实存在。标注指向不存在的文件/行号须修正或改标 `[待确认]`。

### P5 输出清单

1. 生成文件清单及路径（新建/已更新/未变化/跳过）
2. 交叉验证结果摘要（通过项/修正项/抽检结果）
3. 全局质量警告（9 次抽检中 ≥3 次失败时触发）

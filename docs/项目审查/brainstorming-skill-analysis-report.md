# `brainstorming` Skill 分析报告

## 1. 文档对象

- Skill 目录：`superpowers/skills/brainstorming`
- 主定义文件：`SKILL.md`
- 视觉辅助指南：`visual-companion.md`
- 相关提示模板：`spec-document-reviewer-prompt.md`
- 运行脚本：
  - `scripts/start-server.sh`
  - `scripts/stop-server.sh`
  - `scripts/server.cjs`
  - `scripts/helper.js`
  - `scripts/frame-template.html`

---

## 2. 一句话结论

这个 `brainstorming` skill 不是“帮你随便想想”的轻量提示词，而是一个**强约束的前置设计工作流**。它要求代理在任何创意型实现前，先做上下文探索、逐步澄清、方案比较、设计呈现、文档沉淀和用户审批，然后才允许进入 `writing-plans`。如果问题涉及明显的视觉内容，还可以挂接一个浏览器可视化辅助回路。

换句话说，它的职责不是“生成想法”，而是把模糊想法收敛成**已获批准的设计规格**。

---

## 3. Skill 的核心定位

从 `SKILL.md` 的开头描述可以看出，这个 skill 的触发条件很宽：

- 创建新功能
- 构建组件
- 添加功能
- 修改行为

它默认认为，只要任务涉及“设计选择”或“行为变化”，就不能直接写代码，而应该先经过设计澄清。

这意味着它是一个典型的 **process skill**，不是 implementation skill。

它的根本意图有三点：

1. 防止代理跳过设计阶段，直接开写代码。
2. 防止“简单任务”因为未经检验的假设而返工。
3. 把设计过程沉淀成可审阅、可提交、可继续规划的文档资产。

---

## 4. 最重要的硬门禁

这个 skill 最核心的约束来自 `<HARD-GATE>`：

- 不允许调用 implementation skill
- 不允许写代码
- 不允许搭脚手架
- 不允许采取任何实现动作
- 直到设计已经呈现给用户，并且用户已经批准

这是一条真正的流程闸门，不是建议。

它的含义是：

- brainstorming 阶段的产物是“设计与规格”
- implementation 阶段的起点是“用户批准后的设计”
- 两者之间不能混做

这也是为什么文档明确写死：

- brainstorming 结束后的唯一合法后继 skill 是 `writing-plans`
- 不能直接跳到 `frontend-design`、`mcp-builder` 或其他实现型 skill

---

## 5. 总体工作流

`SKILL.md` 里给出了一条 9 步清单，这其实就是该 skill 的主状态机：

1. 探索项目上下文
2. 如果后续问题明显涉及视觉表达，则单独征求是否启用 visual companion
3. 一次一个问题地做澄清
4. 提出 2–3 个方案并说明取舍
5. 分段展示设计，并逐段获得确认
6. 把设计写成 spec 文档
7. 做一次 inline spec 自检
8. 让用户审阅写好的 spec 文档
9. 切换到 `writing-plans`

这个顺序非常严格，尤其强调两件事：

- **不能边问边实现**
- **不能只在对话里达成共识而不落文档**

---

## 6. Skill 逻辑拆解

### 6.1 上下文探索优先

skill 首先要求代理检查：

- 文件
- 文档
- 最近提交

这是为了避免代理在不了解项目结构的前提下提出脱离现状的设计。

如果在已有代码库里工作，这一步还有额外目标：

- 识别既有模式
- 遵循已有结构
- 发现影响本次工作的局部设计问题
- 将必要的、与任务直接相关的结构改善纳入设计

它明确反对“无关重构”，也就是：

- 可以顺手修正会妨碍当前工作的结构问题
- 不能借题发挥做大规模 unrelated refactor

### 6.2 范围评估先于细节提问

在进入问题澄清前，skill 要先判断需求是不是过大。

如果用户说的是多子系统的大工程，例如：

- chat
- file storage
- billing
- analytics

那么不应该立刻问细节，而应该先做系统拆分：

- 哪些是独立子项目
- 它们之间怎么关联
- 先做哪个

之后只对第一个子项目走完整的 brainstorm → spec → plan → implementation 闭环。

这说明这个 skill 不只是“问问题”，还承担了**防止 scope 失控**的职责。

### 6.3 一次只问一个问题

这是整份 skill 中最关键的交互约束之一。

目标不是减少问题总数，而是提高每个问题的价值：

- 避免一次给用户一串问题造成答复质量下降
- 让代理根据上一个回答及时调整后续问题
- 把需求收敛做成一个迭代过程，而不是问卷调查

文档里还建议：

- 优先用 multiple choice
- 但开放问题也允许

本质上，这个 skill 偏好“高带宽、低负担”的提问方式。

### 6.4 先给方案，再让用户选

skill 要求代理一定要给出 2–3 个 approach，并带上：

- 各自的做法
- 取舍
- 推荐方案
- 推荐理由

这意味着代理不能把“方案设计责任”丢给用户。

它的默认假设是：

- 用户更擅长表达目标和偏好
- 代理更应该负责整理路径和权衡

这是一个很成熟的协作姿态，因为它避免了这类低质量问题：

- “你想怎么实现？”
- “架构你来定？”
- “你选数据库还是文件？”

正确做法应该是代理先形成结构化选项，再让用户决策。

### 6.5 设计必须显式呈现并获得批准

当代理认为自己已经理解要做什么后，不是直接记在心里，而是要显式展示设计。

设计展示要求包括：

- 按复杂度分段
- 简单的部分几句话即可
- 复杂部分可到 200–300 字
- 每一段后都询问“这里是否正确”
- 覆盖 architecture、components、data flow、error handling、testing

这说明它要求的是**增量式验证**，而不是一次性大段倾倒。

### 6.6 设计必须可隔离、可解释、可替换

这是该 skill 比较有工程味的一部分。

文档要求设计时考虑：

- 单元职责是否清晰
- 单元是否通过明确定义的接口通信
- 外部是否能在不看内部实现的前提下理解其用途
- 内部是否可以在不破坏调用方的前提下替换

这其实是一套偏模块化设计的标准。

它不是只关心“需求对不对”，还关心“设计边界是否健康”。

文档甚至把“大文件正在变大”视为边界失控的信号，这说明这个 skill 明确偏向：

- 小而聚焦的文件
- 高内聚、低耦合的边界
- 便于 AI 和人类同时理解与修改的结构

### 6.7 设计结束后，必须写成文档

设计被验证后，必须写入：

`docs/superpowers/specs/YYYY-MM-DD-<topic>-design.md`

这里的关键不是路径，而是流程含义：

- 设计共识不能只存在于会话上下文
- 设计必须成为 repo 中可审阅、可版本化的资产

文档还要求：

- 如果用户偏好别的 spec 位置，可以覆盖默认路径
- 如果有可用的写作优化 skill，可以调用辅助优化表达
- 设计文档要提交到 git

也就是说，这不是“临时草稿”，而是正式产物。

### 6.8 自检不是再开评审，而是直接修正

设计文档写完后，skill 要求进行一次 `Spec Self-Review`，检查四类问题：

1. 占位符，如 `TBD`、`TODO`
2. 内部不一致
3. 范围过大
4. 歧义表达

一个很重要的细节是：

- 发现问题后，直接 inline 修掉
- 不要求再次起一轮正式评审

这让 self-review 更像“质量闸门”，而不是“新一轮流程膨胀”。

### 6.9 用户对文档的审阅是第二道门

自检通过后，skill 还要求代理把 spec 文件路径明确告诉用户，请用户审阅。

只有当用户批准文档后，才允许进入 `writing-plans`。

这形成两级门禁：

- 第一级：用户批准“对话中的设计”
- 第二级：用户批准“落盘后的文档版本”

这样做的意义是：

- 防止对话中说得对，但写入文档时漂移
- 防止设计理解和文档表达之间出现偏差

## 8. Visual Companion 的逻辑角色

`visual-companion.md` 不是第二个 brainstorming skill，而是这个 skill 的一个辅助子系统。

它解决的问题是：

- 有些问题“看”比“读”更容易理解
- 尤其是布局、线框图、架构图、视觉方向对比时

但它非常强调：

- 这是 **tool**，不是 **mode**
- 即使用户同意启用，也不是整场会话都强制走浏览器
- 每个问题都要重新判断是否真的适合视觉化

其判断准则很明确：

- 视觉内容本身适合浏览器
  - mockup
  - wireframe
  - layout comparison
  - architecture diagram
  - spatial relationship
- 文本或概念问题仍然应在终端里处理
  - scope
  - tradeoff
  - conceptual choice
  - technical decision

这个设计非常克制。它避免了两种常见误区：

1. 一提 UI 就全部用浏览器
2. 开了 visual companion 后，所有问题都被迫视觉化

---

## 9. Visual Companion 的使用协议

### 9.1 启用前必须单独征求许可

如果预判后续问题会涉及视觉内容，代理必须先发一条**单独消息**询问用户是否愿意启用浏览器辅助。

这条消息不能和别的内容混在一起，不能同时：

- 总结上下文
- 问澄清问题
- 讨论设计

这说明该 skill 对交互形式有很强的控制，目的是避免用户漏掉“是否打开本地 URL”这件事。

### 9.2 运行机制不是“远程画布”，而是“本地目录 + 浏览器轮询/推送”

从 `visual-companion.md` 和脚本实现可以看出，其机制是：

- 代理启动本地 server
- server 监听一个 `content` 目录
- 代理往目录里写新的 HTML 文件
- server 始终展示最新文件
- 浏览器通过 WebSocket 接收 reload 通知
- 用户在浏览器点击选项
- 这些交互事件被写入 `state/events`
- 代理下一轮读取 `events` 与终端文本一起理解用户反馈

所以它并不是复杂的前端应用，而是一个极简的“文件驱动式互动浏览器层”。

## 10. Visual Companion 的实现架构

### 10.1 `start-server.sh`

这个脚本负责：

- 解析参数
- 选择前台 / 后台运行模式
- 创建唯一 session 目录
- 设置 `content` 和 `state` 目录
- 启动 `server.cjs`
- 返回包含 `url`、`screen_dir`、`state_dir` 的 JSON

它有几个很实用的设计点：

- 默认使用随机高位端口，降低冲突概率
- 可把 session 文件写到项目内 `.superpowers/brainstorm/`，避免 `/tmp` 丢失
- 能识别 Codex、Windows 等会清理后台进程的环境，并自动切到前台模式
- 会记录 `server-info`，方便后续恢复连接信息

这说明 visual companion 不是一个只为单一宿主写的脚本，而是考虑了多代理环境差异。

### 10.2 `server.cjs`

这个文件是核心服务端，职责包括：

- 提供 HTTP 页面访问
- 手写实现 WebSocket 协议的基础部分
- 监听内容目录变化
- 在新 screen 推送时通知浏览器 reload
- 记录用户点击事件
- 维护 session 生命周期

其中几个关键点值得注意：

#### a. 内容模型

server 从 `content` 目录中选取**最新的 HTML 文件**作为当前 screen。

这带来几个后果：

- 不需要维护复杂的页面路由
- 每个 screen 天然是一个版本快照
- 强制鼓励用新文件名表示新轮次，而不是反复覆盖同一文件

#### b. 文档包装模型

如果 HTML 本身是完整文档，原样服务。
如果只是内容片段，则自动注入 `frame-template.html`。

这让代理大多数时候只需要写内容片段，例如：

- 标题
- 选项卡片
- mockup 区块

而无需每次重写整页 CSS 和脚本。

#### c. 浏览器交互模型

浏览器侧的 `helper.js` 负责：

- 建立 WebSocket 连接
- 自动重连
- 捕获用户点击
- 上报 `choice`
- 更新界面上的选中状态提示

这意味着 HTML screen 本身可以非常轻，交互能力由公共脚本托底。

#### d. 事件记录模型

用户点击会被追加写入：

- `$STATE_DIR/events`

而且不是只记最终结果，而是记录整个点击序列。

这样代理就能读到：

- 用户先看了什么
- 点过哪些选项
- 是否存在犹豫或来回切换

这比只有一个“最终值”更有信息量。

#### e. 生命周期管理

`server.cjs` 会在以下情况自动退出：

- owner process 已退出
- 30 分钟无活动

并且会写：

- `server-stopped`
- 删除 `server-info`

这是为了避免留下僵尸服务进程。

### 10.3 `helper.js`

客户端脚本提供三类能力：

- WebSocket 自动连接与重连
- 点击事件上报
- 选项选中态维护

其设计非常轻量，说明整个 visual companion 的目标不是构建通用前端系统，而是为 brainstorming 提供最低限度但够用的交互能力。

## 11. `spec-document-reviewer-prompt.md` 的角色

这个文件不是主流程的一部分，但它揭示了一个重要设计思路：

- brainstorming 产出的 spec 文档，后面可以交给 reviewer 子代理审查

它检查的重点不是文风，而是是否会影响后续 planning：

- Completeness
- Consistency
- Clarity
- Scope
- YAGNI

也就是说，这个 skill 不是只关注“想法”，它非常在意：

- 这份 spec 能不能继续喂给 planning 阶段
- 它是否足够稳定到不会把 implementation 带偏

不过要注意：

- 当前 `SKILL.md` 主流程里强调的是 inline self-review
- reviewer prompt 更像一个补充模板，体现了该体系曾考虑过或支持更正式的 spec 复核

## 12. 这个 Skill 真正在强制什么

如果把所有显式规则压缩成一句话，这个 skill 真正强制的是：

**任何实现前，都必须先形成经过用户批准的设计文档。**

为了实现这点，它具体强制了以下行为：

- 先看上下文，再提问
- 一次只问一个问题
- 先给方案，再让用户选
- 设计必须分段讲清楚
- 设计必须获得用户批准
- 设计必须写成文档
- 文档必须经过自检
- 文档还要经过用户二次确认
- 后续只能进入 `writing-plans`

所以这个 skill 的真实输出不是“一个好主意”，而是：

- 结构化设计共识
- 落盘 spec
- 进入 planning 的合法前置条件

## 13. 这个 Skill 的优点

### 13.1 避免代理直接开写

很多代理在拿到需求后会直接进入实现，导致：

- 误解需求
- 漏掉边界情况
- 在错误架构上快速前进

这个 skill 用硬门禁强行阻止了这一点。

### 13.2 让“简单任务”也接受最基本的设计验证

文档明确反对“这太简单，不需要设计”的想法。

这个原则很有价值，因为很多返工正是来自看似简单的小改动。

### 13.3 强迫代理承担方案整理责任

而不是把技术选择和结构选择甩给用户。

### 13.4 把设计从会话文本升级为仓库资产

spec 文档进入 repo 后，才具备：

- 可版本化
- 可评审
- 可追踪
- 可作为 planning 输入

### 13.5 Visual Companion 提供了低成本视觉协作能力

它不是重型原型系统，而是一个很轻的 HTML screen 机制，但足够覆盖：

- 线框图
- 布局对比
- 流程图
- 视觉偏好选择

## 14. 这个 Skill 的代价与风险

### 14.1 流程成本偏高

对于非常小的任务，这套流程依然要求：

- 探索
- 提问
- 方案比较
- 设计呈现
- 文档写作
- 用户复核

这在高频小改动场景下成本不低。

### 14.2 容易让代理“过度流程化”

如果执行过于教条，代理可能会：

- 明明问题很清楚，还不停问
- 明明设计已经足够稳定，还拉长审批过程
- 为了满足模板而输出过多结构化内容

### 14.3 对用户耐心有要求

这个 skill 假设用户愿意参加一个分阶段设计过程，而不是只想说一句“去做”。

### 14.4 对代理的上下文判断要求很高

例如：

- 什么时候该拆分子项目
- 什么时候问题适合 multiple choice
- 什么时候适合 visual companion
- 什么时候设计已经足够成熟可以落文档

这些都需要代理有较强的产品判断和工程判断。

## 15. 与 Trellis 版 `brainstorm.md` 的关系

如果和 Trellis 的 `brainstorm.md` 对比，这个 skill 更像一个**通用上层流程规范**，而不是深度嵌入任务系统的操作协议。

大致区别如下：

- 当前 skill 更关注：
  - 设计澄清
  - 用户批准
  - spec 文档产出
  - 进入 `writing-plans`

- Trellis 版更关注：
  - task 创建
  - PRD 持续更新
  - research-first
  - start workflow 集成

所以可以把它们理解为：

- `superpowers/skills/brainstorming`：通用设计前置 skill
- Trellis `brainstorm.md`：产品化任务系统下的 requirements discovery protocol

---

## 16. 最终评价

这个 `brainstorming` skill 的成熟度很高，原因不在于它写得“像提示词”，而在于它具备一套清晰的流程边界：

- 输入：模糊想法 / 待实现需求
- 中间过程：上下文探索、逐步澄清、方案比较、设计验证、文档沉淀
- 输出：已获用户批准的 spec
- 下一阶段入口：`writing-plans`

它把“先思考再实现”从一句口号变成了可执行的状态机。

如果把它放在团队协作或代理式开发体系里看，它最有价值的地方是：

- 阻止错误实现过早发生
- 强制把设计知识沉淀进 repo
- 用用户审批把设计阶段和实现阶段清晰切开

它的代价则是流程变重，尤其在小任务上容易显得有点严格。

但如果目标是提高多轮代理协作的可靠性，而不是单次对话的轻快感，那么这种严格其实是有意识的权衡。

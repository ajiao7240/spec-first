## 0. 结论先行(TL;DR)

50 轮审查收敛出一个核心判断:

> **spec-first 的方法论已对齐业界最佳实践,但它在三个结构性维度上"长得不均衡":事实层被 prose 稀释、链路是单向流水线、价值无法被自身度量。下一阶段的演化不是"更多能力",而是这三个维度的"补齐与升维"。**

三大结构性主题(统领全部 50 轮):

1. **事实-语义未分层** — 大量确定性事实(校验、追踪、freshness、完成度、信任规则)被迫用 LLM prose 维持,既不可靠又消耗注意力。涉及 R5/R7/R10/R16/R23/R29/R36/R41/R43。
2. **链路是单向流水线,缺反馈飞轮** — 下游发现的上游缺陷不回流,知识写入不被消费,handoff 靠 LLM 手动接续易断。涉及 R8/R11/R15/R21/R39/R47/R48。
3. **Evaluation Harness 近乎空,价值不可证明** — 六层 harness 唯一空层,导致所有演化判断盲目、80/20 无数据、自进化靠人工偶发触发。涉及 R17/R24/R26/R28/R30/R40/R42/R49。

这三者**互相依赖**:没有 ③ 的数据,①② 的改进无法被证明;没有 ① 的事实层,③ 的 telemetry 无处采集;没有 ② 的反馈,①③ 的成果无法复用沉淀。

## 1. 50 轮 → 9 个可执行主题(去重聚类)

50 轮审查的优化建议,去重后聚为 9 个主题。每个主题标注:涉及轮次、问题本质、归属边界(script/LLM)、成本、价值。

### 主题 A:收敛重复规则为单一真相源(事实层)
- **轮次:** R5(task-pack 三重校验)、R10(graph 消费四份解读)、R23(subagent TEXT-only 各处强调)、R29(invariant 横向不一致)
- **本质:** 同一条规则/invariant 在多个 skill 各写一份 prose,独立演化必然 drift。最危险的是"防 stale 执行"的规则本身存在 drift 风险(自指脆弱)。
- **归属:** 事实判定归 script(`spec-first tasks validate` 强化、graph 消费契约、trace check);决策归 LLM。
- **成本:** 中 · **价值:** 高(消除 drift + 减 prose 重量 + 闭合自指脆弱)

### 主题 B:确定性算法脚本化(事实层)
- **轮次:** R4(spec_id 冲突扫描/U-ID diff)、R5(hash/lint)、R7(code-review merge/dedup/confidence)、R43(spec_id/U-ID 追踪链完整性)
- **本质:** merge/dedup/confidence 锚定、冲突扫描、追踪链校验是确定性算法,却靠 LLM 在 prose 指引下手算——慢、不可复现、易错。违背"scripts prepare"。
- **归属:** 算法归 script,语义判断(finding 是否同源、降级是否合理)归 LLM。
- **成本:** 中-高(code-review merge 脚本化是大改) · **价值:** 高(可复现 + 速度 + 边界纠偏)

### 主题 C:prose 预算与长 skill 拆分(事实层 + 上下文经济学)
- **轮次:** R7/R16(code-review 1101 行超 cap)、R4(plan readiness 噪声)、R6(work Phase1 过载)、R36(单 skill 内多层混存)
- **本质:** 长 skill + 多 reference + 累积 artifact 制造 Trellis "约束随上下文衰减"。code-review 超 token cap 是已触顶信号。
- **归属:** prose 分层(invariant→共享契约,流程→主体,失败枚举→脚本/reference)。
- **成本:** 中 · **价值:** 高(链路可扩展性的根本约束)

### 主题 D:Codebase→Spec 的 current-state layer(Context Harness 补强)
- **轮次:** R1(隐式起点无 artifact)、R19(降级态四重重读)、R34(PRD 竞品真空白)、R45(advisory→confirmed 固化)
- **本质:** 链路真正入口事实残缺;current-state 理解在 brainstorm/plan/work/debug 四次重复且不共享、不可复现、不可审计。这是 spec-first 相对竞品最该建立的差异化。
- **归属:** script 产结构化骨架,LLM 填语义,authority=advisory(可经确认升 confirmed)。
- **成本:** 中 · **价值:** 高(消除四重重复 + 差异化 + 降级态质量)

### 主题 E:Graph impact 能力与诚实性对齐(Evidence Harness)
- **轮次:** R2(definitions-only 半截事实)、R9(投影边界)、R26(证据消费未验证)、R33(CRG 第一闭环)
- **本质:** 链路讲 blast-radius/impact-aware,但最有 ground truth 的 impact 闭环 `unavailable`。叙事 > 能力。
- **归属:** 诚实性加固(禁止 impact 不可用时输出 blast-radius 结论)立刻做;impact 能力是中长期投资。
- **成本:** 诚实性=低,impact 能力=高 · **价值:** 高(单点最高 ROI 演化方向)

### 主题 F:知识反向消费与反哺闭环(Knowledge Harness + 飞轮)
- **轮次:** R8(写入治理/消费黑盒)、R15(review→compound 接力)、R21(检索命中黑盒)、R39(知识层形态)、R47(回流)、R48(自审机制)
- **本质:** Knowledge 节点写入有治理,但"是否被检索命中、是否影响判断"是黑盒;下游缺陷不回流上游;链路是单向 pipeline 而非飞轮。
- **归属:** 检索索引/命中 telemetry 归 script;回流建议/沉淀判断归 LLM。
- **成本:** 中 · **价值:** 高(让"越用越好"从口号到机制)

### 主题 G:Evaluation Harness 三支柱实质化(度量)
- **轮次:** R17(系统指标)、R24(eval 灵敏度)、R28(行为质量)、R30(用户增益)、R40(数据证明)、R42(立项路径)、R49(数据驱动 80/20)
- **本质:** 六层 harness 唯一空层。一个以"可验证"为信条的系统,却无法验证自身价值。
- **归属:** telemetry 纯 script 记录(非 LLM 自评);golden case 行为基线。严格 advisory + redaction,非 KPI。
- **成本:** 中(Phase 1 最小) · **价值:** 最高战略(其余改进的证明前提)

### 主题 H:安全 invariant 横向一致性(Governance Harness)
- **轮次:** R25(供应链/mirror drift)、R29(invariant 矩阵)、R31(mutation boundary)、R32(prompt injection)、R44(preview-first 落实)
- **本质:** mutation boundary / preview-first / untrusted-content / 供应链是安全四角,但在不同 workflow 落实强度不均——木桶短板即漏洞。
- **归属:** invariant 矩阵(治理文档)+ prompt source lint 扩展 + runtime mirror hash 校验。
- **成本:** 中 · **价值:** 高(安全不可妥协,一致性比单点强度重要)

### 主题 I:事实状态固化与可恢复性(事实层 + 反状态机精化)
- **轮次:** R35(task pack 完成状态)、R41(反状态机张力终局判断)、R45(advisory→confirmed)、R46(长任务续跑)
- **本质:** "反状态机"被部分误读为"连事实状态也不能固化",导致事实状态(完成度、freshness、校验结论)被迫 LLM prose 维持。区分语义状态(LLM)vs 事实状态(script)。
- **归属:** 事实状态 script-owned advisory facts;语义状态 LLM。**不是** active pointer/phase routing(Trellis 拒绝项)。
- **成本:** 中 · **价值:** 中-高(可恢复性 + 判断基线精化)

## 3. 三个"无论如何都该立刻做"的零/低成本项

从 50 轮中提取**成本极低、价值明确、无依赖**的诚实性加固,可不经大 plan 直接做:

1. **Graph 越权断言禁令(R2/R26/E):** 所有消费 graph 的下游 skill 明确——`impact_context=false` 时不得输出 blast-radius 级结论,必须在 evidence 段落引用具体 graph-facts 字段(query_ready/impact_context)而非笼统 "based on graph"。这把 advisory≠confirmed 从声明变约束。
2. **run artifact 单一定位(R6):** schema 的 authority_level 显式标 advisory,杜绝下游把 run JSON 的"测试通过"当 confirmed。
3. **"definitions-only 是常态"标记(R9):** 把它从 limitations 文本提升为稳定 evidence_posture 标记,避免下游每次当降级而触发无谓 graph-bootstrap。

这三项都是**让证据诚实、防止 advisory 被当 confirmed**——直接服务角色契约最高优先级"可信证据 > 自动化便利"。

## 4. 必须坚持的差异化内核(审查确认不动摇的部分)

50 轮审查不是只找问题,也确认了 spec-first **做对了、应坚持**的部分:

- **vendor-neutral 本地代码事实层 + 全链路消费 + 可审计 evidence/limitations**(R33):护城河,竞品(Cursor/Copilot/Sourcegraph/CodeRabbit)难复制,继续坚持。
- **反中心化流程引擎 / 反强状态机**(R35/R41):大部分收缩是有意的、正确的。精化为"反语义状态中心化,不反事实状态固化"即可,不推翻。
- **单一需求真相源,不建第二需求库**(R34):current-state layer 是 advisory 输入,代码仍是真相,不新建需求库。
- **六层 Harness 框架**(R38):竞品多数没有这么完整的 harness 自觉,是优势。
- **数据是 advisory 自观测,不是 KPI 考核**(R40):避免 Goodhart's law,区别于团队监控工具。
- **closeout 冗余 > per-turn hook**(R3/R11/R18/R37):更克制、双宿主一致的 breadcrumb 落地,优于照搬 Trellis per-turn hook。
- **Light contract · Explicit boundaries · Let the LLM decide**(R50):所有演化强化内核,不偏离。冲突时:边界 > 能力,证据 > 自动化,最小方案 > 完整设计。

## 5. 终局架构假说(方向收束)

spec-first 三年后不应是"更大的 prompt 集合或更强的状态机",而应完成**三个升维**(R50):

1. **事实层确定,语义层灵活** — 确定性事实(校验/追踪/freshness/完成度/信任规则)沉淀为 script-owned facts + 共享契约;LLM prose 只保留语义判断与流程骨架。可靠性来自事实层确定性,灵活性来自语义层 LLM 判断。这是 "Scripts prepare, LLM decides" 的彻底化。(主题 A/B/C/I)
2. **单向流水线 → 有反馈的飞轮** — 下游→上游回流、Knowledge 反哺、telemetry 驱动自审。链路闭合成飞轮,错误被源头修正,知识被复用,系统真正"越用越好"。(主题 F + R47/R48)
3. **不可证明 → 可度量自观测** — Evaluation Harness 三支柱实质化,让所有演化判断有数据,让"用数据证明价值"对自身成立。(主题 G)

**终局形态:** 事实层确定、语义层灵活、闭环有反馈、价值可度量的 AI Coding Harness。护城河 = vendor-neutral 本地事实层 + 全链路可审计证据 + 不可证伪的自我度量。

## 6. 落地建议与边界声明

**落地路径:** 本文是方向文档,不是 plan。建议:
1. 把 W1 四项(E 诚实性 / A 收敛 / D current-state / G telemetry Phase 1)各转一个 `/spec:plan`,其中 B(code-review merge 脚本化)和 G(Evaluation 完整体)是大型任务需独立 plan + goals/non-goals/migration/test plan。
2. 把本审查 50 轮结论进入**可追踪 backlog**(R48):否则审查本身也是单向写入,违背反馈闭环原则。
3. 每个落地项必须能指向至少一个 Evaluation Harness 指标(角色契约设计判断矩阵要求)——这也反向论证了为何 G 应 W1 启动。

**双宿主:** 所有 prose/契约改动(A/C/D/H)必须同步验证 Claude 与 Codex 投影一致(R18 行为层 eval)。

**source/runtime 边界:** 改动走 source(skills/agents/templates/src/cli/contracts/docs),不手改 `.claude/`/`.codex/`/`.agents/skills/`;runtime drift 用 `spec-first init` 修复。

**CHANGELOG:** 任一主题落地为 source 变更时,按仓库格式 + 全局 developer profile 更新 `CHANGELOG.md`;用户可见行为变化追加 `(user-visible)`。

**验证状态声明(诚实交代):** 本次 50 轮审查与综合为**静态源码精读 + 既有文档归纳 + 业界对照**。基于已读的 8 个核心 skill 源码摘要、角色契约全文、using-spec-first 全文、graph-facts 实况、既有两份审查、6 份业界分析。**未执行:** fresh-source eval、测试运行、双宿主 runtime 对比、CLI 行为实测。所有"实现现状"判断可能与最新磁盘 source 有 commit 级偏差;所有"模型实际行为"判断为推断。落地前需回源核对具体实现,advisory 结论需经验证升为 confirmed——这本身正是主题 G/E 要解决的问题,审查方法论与审查结论在此自洽。

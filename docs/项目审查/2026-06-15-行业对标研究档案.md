# spec-first 行业对标研究档案（2026-06-15）

## 0. 用途与读法

本档案回答一个问题：**同类 AI coding workflow / SDD 项目各自靠什么 mechanism 工作，spec-first 在哪些地方真落后、哪些 mechanism 值得借鉴。**

- 取证方式：11 路并行 subagent 深读本地 clone（非读 README，而是读 CLI/skill/agent/command/template/脚本源码，理解 mechanism）。
- 每个对标对象记 7 维：**定位 / 入口 / 闭环 / 治理·证据·知识 / 多宿主 / 亮点 / 领先 spec-first 之处 / 可借鉴动作**。
- 所有「领先」与「可借鉴」为 advisory；借鉴前过角色契约 §7 矩阵 + §4 边界。

**跨 11 仓的三条核心结论（已写入 review 报告 §3）：**

1. **多宿主可移植性是 spec-first 唯一系统性落后的硬能力。** 几乎每个活跃对标仓支持 7–30+ host（spec-kit IntegrationBase、OpenSpec adapter、Trellis bracket marker、BMAD template、compound converter、planning-with-files SKILL.md）；spec-first 锁定 Claude+Codex。但宿主 primitive 正商品化，结论是转向 standards-native 而非硬追 adapter 数量（见角色契约 §7 新增判断）。
2. **三个 mechanism 被反复证明有效、spec-first 最该补：**（a）per-turn / hook-based 状态注入（Trellis / pro-workflow / planning-with-files，抗 context 压缩）；（b）artifact preflight gate（flow-x / OpenSpec，程序化防跳阶段）；（c）anti-rationalization 表（superpowers，最低成本反 LLM 偷懒）。
3. **spec-first 的独有优势在对标后更清晰：** deterministic-facts/LLM-judgment 边界纯粹度、honest-closeout 诚实降级文化、Review→Knowledge 闭环契约化——**没有任一对标仓同时具备**。scale-engine 进化引擎更自动但牺牲边界清晰；BMAD retrospective 更深但靠 LLM 自查无确定性证据。

## 1. GitHub Spec Kit

- **仓库**：`/Users/kuang/xiaobu/spec-kit`（github/spec-kit）
- **定位**：Agent-agnostic SDD scaffold CLI。Primary unit = "feature"（`spec.md + plan.md + tasks.md` in `specs/###-feature/`）。目标 30+ AI coding agent（Claude/Copilot/Gemini/Cursor/Windsurf/Codex 等）经 integration adapter。
- **入口**：① Python CLI（`specify init/self/extension/preset/workflow`）；② 注入到用户选定 agent 的 slash commands（`/speckit.specify` `.plan` `.tasks` `.implement` `.clarify` `.analyze` `.checklist` `.constitution` `.taskstoissues`）；③ YAML workflow engine（`specify workflow run speckit`）做 gate 驱动的多步自动化；④ extension/preset marketplace。无 MCP server，无 workflow engine 以外的状态机。
- **闭环**：覆盖 spec / plan / task / work(implement) / 只读跨产物 analyze；另有 clarify（plan 前消歧）、checklist（需求质量校验）。**未覆盖**：专门 debug、code-review、实现后 evidence/eval、knowledge/learning 捕获。workflow engine 在 spec/plan 有 approve/reject 人类 gate，但实现后无 evidence gate——**非全闭环**（代码执行结果不回流 spec/knowledge）。
- **治理·证据·知识**：治理=Constitution 模板（项目原则）在 plan 时经「Constitution Check」gate 校验；engine 支持 specify↔plan、plan↔tasks 的 approve/reject gate。证据=实现只在 tasks.md 标 `[X]`，无自动 test-pass 证据、无 eval 评分、无 LLM-judge。知识=无显式 learning 捕获，无 compound/solution 等价物。
- **多宿主**：激进多宿主，30+ adapter。`specify init --integration <key>` scaffold 出正确格式（`.claude/commands/*.md`、`.github/copilot-instructions.md`、`.gemini/commands/*.toml` 等）。IntegrationBase 类层级处理格式差异（Markdown/TOML/YAML/Skills 子类），一套模板装到任意 agent。
- **亮点**：① 极致多 agent 广度（IntegrationBase 类层级，写一次部署到处）；② extension/preset/workflow 三层定制（extension 加命令+hook、preset 改模板/术语、workflow 编排 gate/fan-out/fan-in/loop/可恢复 state）；③ hook 架构（`extensions.yml` 在每个生命周期 phase 有 before_*/after_* hook，optional/mandatory 语义，第三方不 fork 即可注入行为）。
- **领先 spec-first**（advisory）：① 多 agent 可移植（30+ vs Claude+Codex，加新 agent ≈10 行类）；② 可恢复 workflow engine（command/shell/gate/if/switch/while/fan-out/fan-in + 持久 RunState，比 spec-first 线性 skill 链更强）；③ extension marketplace（catalog.json + community catalog + 优先级叠加 preset）；④ checklist-as-requirements-test（「unit tests for English」校验需求质量）；⑤ tasks→GitHub Issues 桥。
- **可借鉴**（候选）：① 生命周期 hook 模型（before/after，spec-first 可加 hooks.yml 声明 skill+phase+optional/mandatory）；② checklist-as-requirements-gate（/spec:work 前跑 checklist 校验 spec 完整/清晰/一致，缺关键项即阻断）；③ 可恢复 run state（每步序列化到 `.spec-first/runs/<id>.json`，中断的 workflow 可单命令恢复）。

## 2. OpenSpec ⚠️ 最近赛道竞品

- **仓库**：`/Users/kuang/xiaobu/OpenSpec`（github.com/Fission-AI/OpenSpec，npm `@fission-ai/openspec` v1.4.1）
- **定位**：轻量、tool-agnostic 的 spec-layer CLI，给任意 AI coding 助手加「change proposal + delta spec + archive」生命周期，25+ tool。Primary unit = 一个 "change" 文件夹（proposal/specs/design/tasks）。**显式 anti-Spec-Kit**：自我定位「比 Spec Kit 轻、可自由迭代、无刚性 phase gate、brownfield-first」。
- **入口**：CLI（`openspec init/update/validate/list/status/new/archive`）+ 每个 tool 生成的 slash-command skill（如 `/opsx:propose` `/opsx:apply` `/opsx:archive`）。schema 驱动的 artifact graph 解析依赖序。两 profile：core（propose/explore/apply/sync/archive）与 expanded（加 new/continue/ff/verify/bulk-archive/onboard）。
- **闭环**：spec（delta spec，ADDED/MODIFIED/REMOVED/RENAMED 操作）/ plan（proposal.md + design.md）/ task（tasks.md checkbox）/ work（apply 读 tasks 标 checkbox）/ review·debug（verify-change skill 做 3 维校验：对 specs/tasks/design）/ evidence（archive gate 的 Zod schema + 结构校验）/ knowledge（archive 进 `openspec/changes/archive/` + delta 合并进 `openspec/specs/` source-of-truth）。**是闭环**：change 从 proposal 经实现、校验，delta spec 合并进持久 spec corpus，change 文件夹归档。
- **治理·证据·知识**：治理=artifact 依赖图（schema.yaml）定义 apply 前哪些 artifact 必须存在；archive gate 用 Zod 校验 delta、ERROR 级拒绝。证据=`openspec validate` 并发批量校验出 JSON 结构化报告；archive 前对重建的全量 spec 预校验。知识=delta 在 archive 时原子合并进 `openspec/specs/`（ADDED/MODIFIED/REMOVED/RENAMED，校验失败回滚），归档目录日期前缀做审计轨。无显式 learnings 抽取（spec corpus 本身即知识）。
- **多宿主**：25+ AI 助手 adapter，带各 tool 命令格式知识（frontmatter/TOML/prompt.md 等），生态里最可移植的 spec workflow。
- **亮点**：① 无可匹敌的 tool 广度；② **delta-spec merge 系统**（change 带 ADDED/MODIFIED/REMOVED/RENAMED delta，archive 时原子合并进持久 corpus，预写校验——真正的「spec as living document」机制，累积制度知识）；③ 可定制 artifact schema（用户自定义 workflow 形状：schema.yaml 定 artifacts/dependencies/apply 配置，无需改代码）。
- **领先 spec-first**（advisory）：① 多 tool 可移植；② **delta-spec 合并进持久 corpus**（spec-first 只 archive 冻结快照，无 ADDED/MODIFIED/REMOVED 原子合并 + 回滚的 living source-of-truth）；③ 用户可定制 workflow schema（spec-first 生命周期硬编码在 skill 模板，OpenSpec 让用户改 schema.yaml 而不 fork）；④ workspace coordination（beta，跨仓规划 + linked context store）。
- **可借鉴**（候选）：① delta-spec merge（借 requirement-block 解析器 + apply 逻辑，维护累积 living spec corpus 而非冻结归档）；② 可定制 workflow schema（YAML artifact graph，让团队加「security review」artifact 而不 fork）；③ archive 前并发 bulk validation + 结构化 JSON 报告（ERROR/WARNING/INFO + duration + per-item，既作 CLI gate 又作 CI 机器证据）。

## 3. Trellis

- **仓库**：`/Users/kuang/xiaobu/Trellis`（v0.5.19，npm `@mindfoldhq/trellis`）
- **定位**：AI coding workflow framework，把 spec/task/memory 持久化进仓库，让任意 coding agent 按工程标准工作。Primary unit = task 目录（`.trellis/tasks/MM-DD-name/`）。14+ AI coding 平台。
- **入口**：CLI（`trellis init/update/uninstall`）做 scaffold。运行时完全 prompt 驱动，经平台 hook（SessionStart、UserPromptSubmit/BeforeAgent、PreToolUse）每轮注入 workflow-state 面包屑。运行时 skill/command：`/trellis:start` `/trellis:finish-work` `trellis-brainstorm` `trellis-check` `trellis-update-spec` `trellis-break-loop`。Python（`task.py`）提供 task 状态机（create/start/finish/archive/add-context/add-subtask）。无 MCP——纯文件 state + hook 注入。
- **闭环**：spec（`.trellis/spec/`）/ plan（brainstorm→prd.md）/ task（task.py 父子层级）/ work（trellis-implement 子 agent）/ review（trellis-check 子 agent）/ debug（trellis-break-loop 复盘）/ knowledge（trellis-update-spec 把 learnings 写回 spec）。**是闭环**：Plan→Execute→Verify→Finish→Spec-update 喂下次会话的 context 注入。finish-work 归档 task + 记 session journal 做跨会话连续。
- **治理·证据·知识**：治理=workflow-state 面包屑每轮充当 gate（UserPromptSubmit hook 每轮注入 phase 指令，强制 no_task→planning→in_progress→completed；必做步骤标 `[required · once]` 并有回归测试断言面包屑覆盖）。证据=trellis-check 跑 lint/typecheck/tests + 对 spec 审 diff，但无超出 task.json status 的持久证据 artifact。知识=trellis-update-spec 用结构化模板把 learnings（设计决策/模式/坑/禁用模式）写回 `.trellis/spec/`；workspace journal 持久化 session 历史。
- **多宿主**：14 平台，同一 `.trellis/` 生成 Claude hook / Cursor command / Codex skill / Gemini agent / Copilot hook；平台差异编码在 workflow.md 的 bracket marker，运行时过滤。
- **亮点**：① **per-turn workflow-state 面包屑注入**（UserPromptSubmit hook 每条用户消息触发，解析 active task status 注入 phase 指令——hook 是 enforcer，不靠 AI 记忆）；② 14 平台单源生成；③ session-scoped active task 多窗口隔离（`.trellis/.runtime/sessions/<key>.json`，多终端不冲突）。
- **领先 spec-first**（advisory）：① **per-turn 面包屑强制**（spec-first 无 UserPromptSubmit 每轮重注，靠会话启动加载 + 记忆，抗 context 压缩弱）；② 多平台；③ JSONL context manifest 给子 agent（implement.jsonl/check.jsonl 声明每个子 agent 必读哪些 spec/research，结构化注入而非靠子 agent 自找）；④ 父子 task 层级（spec-first task 是扁平的）。
- **可借鉴**（候选）：① **per-turn 状态注入**（spec-first 加 UserPromptSubmit hook 读 task status 每轮注入短面包屑，抗 context 压缩漂移）；② 子 agent JSONL context manifest（/spec:work 生成显式 manifest 列子 agent 预读文件）；③ session-scoped task 指针（按 CLAUDE_SESSION_ID 存 active 指针，多窗口不互覆）。

## 4. BMAD-METHOD

- **仓库**：`/Users/kuang/xiaobu/BMAD-METHOD`（github bmad-code-org/BMAD-METHOD，npm bmad-method，V6）
- **定位**：Agentic agile 开发框架，persona-agent（Analyst/PM/Architect/Dev/QA/UX/SM/Tech Writer）经 prompt-document 架构编排全生命周期 SDLC。Primary unit = "workflow"（step-file 驱动的 markdown 指令序列）。20+ AI IDE/CLI（Claude/Cursor/Windsurf/Gemini CLI/Kiro/Codex）。
- **入口**：CLI installer（`npx bmad-method install`）scaffold `_bmad/` 进项目，带平台特定 skill 文件。经原生 IDE skill 机制调用（如 Claude `/bmad-pm` 加载 PM persona，再 `CP` 触发 create-PRD）。master "bmad-help" skill 按 artifact 存在性 + phase 序路由用户到下一 workflow。无 MCP、无运行时状态机——纯 prompt-document 驱动。
- **闭环**：spec（Product Brief/PRD/UX Design）/ plan（Architecture/Epics & Stories/Sprint Planning）/ task（Story 文件含 tasks/subtasks/ACs）/ work（bmad-dev-story red-green-refactor）/ review（对抗 code review + 实现就绪检查）/ debug（correct-course workflow）/ evidence（Definition of Done checklist、sprint-status.yaml 跟踪 story→review→done）/ knowledge（retrospective 存 lesson 文档、project-context.md）。**是闭环**：retrospective 显式载入上轮 retro 的 action item 查 follow-through，「重大发现」触发 epic 重规划。但无自动 evidence gate（全为 LLM 自查，非程序化 test 执行跟踪）。
- **治理·证据·知识**：治理=phase 序由 help 路由 CSV 强制（required=true 阻断）；quick-dev 的复杂度阈值把复杂工作路由回完整 PRD 流程；「实现就绪」gate 在 dev 前查 PRD/UX/Architecture/Epics 对齐。证据=DoD checklist 验所有 AC 满足、测试过、File List 完整；code review 对 `git diff` 交叉验 story File List；sprint-status.yaml 跟踪 story 状态机。无 CI 集成（证据皆 LLM 观察）。知识=retrospective 是所见最丰富的知识捕获（从所有 story 抽 dev note/review 模式/velocity/技术债，交叉对照上轮 retro 的 action item，检测需更新 epic 的架构发现，存结构化 retro markdown）；project-context.md 作项目级持久知识。
- **多宿主**：20+ IDE/CLI，统一 installer + per-platform 模板（platform-codes.yaml + 模板文件）。
- **亮点**：① **MULTI-AGENT PARTY MODE**（单会话载入多 persona 协作讨论，retrospective 模拟完整 Scrum 团队对话）；② **scale-adaptive escalation**（quick-dev 按复杂度信号路由：简单直接执行 / 中等给 tech-spec / 系统级走完整 PRD→Architecture→Epics）；③ retrospective 连续性追踪（载入上轮 retro 查哪些 action item 完成/忽略，关联当前困境与未处理的旧承诺，检测需重规划下个 epic 的「重大发现」）。
- **领先 spec-first**（advisory）：① IDE/host 覆盖广度（20+ vs 2，template 驱动）；② agent persona 专精 + menu（每 agent YAML 定 persona + 可触发 workflow 菜单，角色分离强于 spec-first 单声部）；③ **sprint-status.yaml 跨会话状态机**（committed YAML 跟踪 story 生命周期，workflow 原子读写，spec-first 缺此粒度的跨会话 artifact 状态）；④ retrospective 学习循环深度。
- **可借鉴**（候选）：① **跨会话 artifact 状态文件**（spec-first 可维护 `spec-first-status.yaml` 跟踪 spec→plan→tasks→work→review→done per feature，靠文件而非 LLM 记忆做跨会话连续）；② **scale-adaptive routing**（work 前按复杂度信号分流，自动升级到 plan，防小任务过度规划 + 大任务规划不足）；③ retrospective 连续性检查（新 feature 周期开始时载入上轮 lessons，查 action item 是否落地，未落地标为当前风险）。

## 5. get-shit-done (v1 / v2)

- **仓库**：`/Users/kuang/xiaobu/get-shit-done`（v1）、`/Users/kuang/xiaobu/gsd-2`（v2）
- **定位**：v1=Meta-prompt + context-engineering 的 SDD 系统（slash-command workflow + SDK），targets Claude Code/Gemini/Codex；v2=独立 CLI coding agent（gsd-pi，基于 Pi SDK，fresh-session-per-task 状态机）。Primary unit = Phase(v1) / Slice+Task(v2)。
- **入口**：v1=slash command 装进 `~/.claude/commands/`（`/gsd:do` dispatcher 路由到 new-project/plan-phase/execute-phase/autonomous/debug/verify-work/quick）；SDK（gsd-sdk）提供 init/state/config 查询；PreToolUse hook（prompt 注入守卫、workflow 守卫、context 监控）。v2=独立 CLI 二进制 `gsd` + 交互 TUI（/gsd auto/next/discuss/status），TS 扩展系统 + 程序化 session 控制。
- **闭环**：v1 覆盖 spec（spec-phase.md 含量化 ambiguity 评分）/ plan（researcher+planner+plan-checker 修订环）/ task（execute-phase.md wave 并行）/ work（gsd-executor 子 agent）/ review（code-review.md + verify-work.md UAT）/ debug（debug.md 科学方法 + session manager）/ evidence（VERIFICATION.md/UAT.md/SUMMARY.md）/ knowledge（extract-learnings.md → LEARNINGS.md）。**是闭环**（autonomous.md 自动 discuss→plan→execute→verify→next）。v2 加 crash recovery（lock 文件）、每 slice 自适应重规划、cost/token metrics ledger、stuck 检测（retry once then stop）。
- **治理·证据·知识**：治理=4 型 gate taxonomy（Pre-flight/Revision/Escalation/Abort）formalized 在 references/gates.md 的 Gate Matrix；revision gate 有迭代上限（max 3）+ stall 检测；checkpoint 系统（human-verify/decision/human-action，auto 模式可绕 verification checkpoint）。证据=verification-patterns.md 强制 4 级检查（Exists→Substantive→Wired→Functional）+ stub 检测（grep 模式）；UAT.md 跟踪对话式验收；VERIFICATION.md 的 `status:passed` gate phase 关闭。知识=extract-learnings.md 从 phase artifact 抽 decisions/lessons/patterns/surprises 进 LEARNINGS.md；CONTEXT.md 捕获设计决策；decisions register（append-only）；v2 加 metrics.json（per-unit token/cost）。

- **定位**：myclaude=Claude Code 多 agent workflow 系统 + 多后端执行（Codex/Claude/Gemini/OpenCode），primary unit = /do 5-phase feature dev + codeagent-wrapper 二进制，Claude Code only；baoyu-skills=内容发布工具（公众号/X/小红书/信息图）；vibe-coding-cn=中文知识 hub（prompt+skill+入门）；chen-skillshub=PRD 写+审管线；yunshu_skillshub=31 个独立 skill（PM/写作/运维），无编排层。

## 11. planning-dev-skills（独立 planning/PRD/task skill 5 仓）

- **仓库**：planning-with-files、dev-planner-skill、prd-manager、prd-taskmaster、demand-detective-skill
- **定位**：一组面向 planning-to-dev 不同 phase 的独立 Claude Code skill：文件式 context engineering（planning-with-files）、对话式项目 scaffold（dev-planner）、结构化 PRD 生成/版本化（prd-manager、prd-taskmaster）、需求消歧（demand-detective）。Primary unit = 独立 SKILL.md，Claude/Cursor/Codex/Gemini CLI 消费。无统一框架——各自独立可装。
- **入口**：Claude slash（/plan、/planning-with-files:start、/prd-manager 等）或自然语言触发短语（"I want a PRD for..."、"帮我开发..."）。无 CLI 二进制、无状态机。各 skill 按 frontmatter description 关键词在 host IDE 激活。prd-taskmaster 用 Python helper（script.py）做确定性操作。
- **闭环**：planning-with-files=仅 work（plan/task/work 经 3 markdown + hook），无 spec/review/evidence/knowledge；prd-taskmaster=spec/plan/task/work（PRD→validate→taskmaster parse→执行模式），有初级证据（13 validation check + datetime 跟踪）无 review/knowledge 环；prd-manager=仅 spec/plan（PRD/design/dev/plan/ops/test docs）；demand-detective=仅 spec（需求消歧）；dev-planner=spec/plan/work（对话→docs→agent-team 执行）。**皆非闭环**，无项目从 evidence/review 回流到 knowledge/spec 精炼。
- **治理·证据·知识**：planning-with-files hook（PreToolUse 重读 plan、PostToolUse 提示更新 status、Stop hook 验 phase 完成）——轻量强制无正式 gate；prd-taskmaster 13 项自动 PRD 质量 check（章节存在/含糊语言检测/可测性/SMART 度量）+ 评级（EXCELLENT/GOOD/ACCEPTABLE/NEEDS_WORK）——task 生成前真质量门；prd-manager 在 prd.md/dev.md 阶段需显式用户确认。证据=prd-taskmaster scored PRD 质量 + 执行时间跟踪（实际 vs 估算）；planning-with-files markdown 记错误。知识=prd-manager 生成 CLAUDE.md 作项目 memory（ADR/known issues/技术债），无系统 learning 回流。
- **多宿主**：planning-with-files 支持 16 IDE/CLI（per-platform skill 变体 + hook 格式）。
- **亮点**：① planning-with-files **hook-based 注意力管理**（PreToolUse hook 每次工具调用前重读 task_plan.md，PostToolUse 提醒更新 status，Stop hook 验完成——真 context engineering 机制，保持 agent 在轨无需人工）；② demand-detective **结构化决策树方法**（L1-L4 影响半径排序，把模糊业务请求转为确认/优先的决策，带显式权衡 + 决策审计日志）；③ prd-taskmaster **script.py 关注点分离**（确定性操作 preflight/validation/task counting/crash recovery 由 Python 输出 JSON，AI 处理判断——可靠性架构）。
- **领先 spec-first**（advisory）：① 多 host 可移植（planning-with-files 16 平台，SKILL.md 格式天然可移植）；② **hook-based context 强化**（PreToolUse hook 每次工具调用前重注 plan 进注意力窗口——spec-first 无等价抗 context drift 的自动注意力机制）；③ **需求消歧方法论**（demand-detective L1-L4 决策树 + A/B 结构化选择 + 业务成本标注 + 决策审计日志，比 spec-first brainstorm 更严谨地把模糊需求转可执行 spec）；④ **PRD 校验作自动质量门**（prd-taskmaster 13-check 脚本：含糊语言检测/章节完整/可测性评分 + 评级——task 生成前的证据式门，spec-first 靠 LLM 判断无 spec 质量结构化校验）。
- **可借鉴**（候选）：① **attention hook**（PreToolUse hook 每次工具调用前重读当前 spec/task context 进注意力窗口，防长会话目标漂移，如 planning-with-files `cat task_plan.md | head -30`）；② **结构化 PRD 校验脚本**（spec 生成后 plan 前跑自动 spec 质量门：含糊语言模式/缺章节/不可测需求/SMART 度量，返回 scored JSON，似 prd-taskmaster validate-prd）；③ **决策审计日志**（brainstorm/spec 阶段采 demand-detective L1-L4：模糊点作优先 A/B 选择 + 业务成本，持久决策日志表 date/keyword/uncertainty/decision/person/risk 供可追溯）。

# 2026-05-30 Codex：100 轮审查后当前 spec-first 优化点

## 结论

基于 `2026-05-30-codex-全面审查-自我进化.md` 的 100 轮审查、`2026-05-30-codex-全面审查-自我进化-result.md` 的复审结论，以及 Claude prior report `2026-05-30-全面审查-自我进化.md` 的 advisory 洞察，当前 spec-first 最值得优化的方向不是继续增加公开 workflow，而是把已有链路变成可证明、可复盘、可消费的工程闭环。

下一阶段主线应收敛为：

> 先证明 artifacts 被下游消费，再证明消费改善结果，最后才扩展能力。

### 四、综合方向定论(五份合一的单一结论)

两链、五份文档、150 轮审查,最终收敛为一句话,且双链表述一致:

> **下一阶段不是"把链路搭得更全/更强",而是"先证明现有闭环真的提升交付质量"——用 light contracts 把上下文、证据、评估、知识变成 repo-local 可消费、可度量的事实。**

执行上收敛为:**先做消费证明 + Evaluation MVP + invariant matrix(P0/P1/P6 的 MVP),用 friction audit 的真实数据再排序 current-state、review merge、knowledge 等后续 CUD;能力增强(尤其 graph-primary)必须由 P-friction 实证驱动,不接受纯架构偏好。** 这同时满足 Claude 终局假说的三个升维和 Codex 的"减少重复认知负担 > 新增能力"排序原则——两链在此完全合流。

## 优先级总览

| 优先级 | 优化点 | 核心问题 | 建议状态 |
| --- | --- | --- | --- |
| P0 | Consumption Evidence MVP | artifacts 是否真的被读到并影响决策不可见 | 立即进入窄 plan |
| P1 | Workflow Outcome Ledger | workflow 做了什么可见，但是否减少返工不可见 | 与 P0 同步设计最小字段 |
| P2 | Friction Audit | 缺真实任务摩擦数据，无法排序后续能力 | P0/P1 后用真实样本验证 |
| P3 | Core Workflow Semantic Eval | 关键 workflow 行为依赖 prose，缺 fresh-source eval | P0 后启动核心 fixture |
| P4 | Artifact Discoverability Index | artifacts 多，但 producer/consumer/authority 不够易消费 | 先文档化，不做 registry |

## P0：Consumption Evidence MVP

目标：证明现有 artifacts 是否被下游 workflow 消费，并区分“读过”与“用于决策”。

为什么现在：100 轮审查多次发现 graph fields、run artifacts、solutions、review summaries 已存在，但消费证据弱。继续扩能力前，应先证明已有证据链真的进入了 downstream decision。

最小落地：

- 在 work/review/compound closeout 或相关 artifact 中记录 `artifact_ref`、`read_by_workflow`、`used_for_decision`、`ignored_reason`、`source_confirmation`。
- 首批只覆盖三类高价值对象：graph fields consumed、solutions hit telemetry、run artifact downstream consumption。
- 对 graph-backed claim 要显式写明消费字段，例如 `query_global_graph`、`impact_context`、`freshness_state`；缺字段引用时只能称 bounded source read。

script-owned facts：

- artifact path 是否存在。
- workflow 是否记录了 consumption 字段。
- source confirmation 是否有 repo-relative path 或验证命令引用。

LLM-owned judgment：

- artifact 是否真的影响了架构/实现/review 判断。
- ignored reason 是否合理。
- stale/degraded evidence 是否仍足以支持当前结论。

Artifacts / consumers：

| Artifact | Producer | 初始 consumer |
| --- | --- | --- |
| `consumption_evidence` closeout block | plan/work/review/compound/debug | result 文档、review、compound、后续 plan |
| graph fields consumed summary | graph-aware workflows | plan/work/review/debug |
| solutions hit telemetry | plan/work/debug/compound | knowledge freshness / compound-refresh |
| run artifact downstream consumption | spec-work closeout / review handoff | review、compound、resume |

Anti-pattern：

- 不把 `read_by_workflow=true` 当作“产生价值”的证明。
- 不把 graph provider output 当语义权威。
- 不为首版引入 dashboard、数据库或全局 registry。

验证：

- contract/lint 层面检查必需字段存在。
- fresh-source eval 检查 LLM 在 stale graph、no-hit solution、ignored artifact 三种场景下是否正确披露。

## P1：Workflow Outcome Ledger

目标：用最小事实记录 workflow outcome，让后续能比较“是否减少返工、漏判和重复解释”。

最小字段：

`goal`、`workflow_path`、`artifact_refs`、`feedback_loop`、`validation_result`、`review_result`、`residual_risk`、`rework_signal`、`downstream_consumed`。

script-owned facts：

- 命令是否运行、exit code、artifact path、validation result path。
- ledger 字段是否符合 schema。

LLM-owned judgment：

- residual risk 的解释。
- rework signal 是否代表真实返工。
- outcome 是否足以支撑下一步。

边界：

- ledger 是 evidence ledger，不是 KPI dashboard。
- 脚本不判断“成功”；脚本只记录事实，LLM/人类解释成功与否。
- 可预留 `trace_ref`、`span_kind_hint`、`genai_operation_hint`，但不接入 OTel 平台。

首批使用场景：

- `spec-work`：记录反馈环、验证结果、review residual。
- `spec-code-review`：记录 actionable finding、false-positive signal、deferred risk。
- `spec-debug`：记录 repro、root cause confidence、fix validation。

## P2：Friction Audit

目标：用真实任务数据排序后续能力，而不是按架构偏好投资。

样本：

| 样本 | 要回答的问题 |
| --- | --- |
| first-run | 新用户是否被 setup/runtime/graph 状态卡住 |
| standard feature | plan/tasks/work 是否减少重复上下文输入 |
| bug debug | debug 是否更快定位 root cause |
| review autofix | review 是否发现 actionable 问题，误报成本多高 |
| compound learning | solution 是否在后续任务中被命中和使用 |

记录字段：

- `elapsed_time_estimate`
- `blockers`
- `re_explanation_count`
- `degraded_reason`
- `review_wait_cost`
- `review_found_actionable_count`
- `review_false_positive_count`
- `ship_outcome`

边界：

- Friction Audit 是产品/工程实证，不是新增 workflow。
- 不用小样本直接制定全局默认策略；只作为 CUD 排序证据。

## P3：Core Workflow Semantic Eval

目标：给核心 workflow 建立 fresh-source eval，防止 prose 行为漂移。

首批覆盖：

| Workflow | 关键 eval 场景 |
| --- | --- |
| `using-spec-first` | 应触发/应跳过、explicit route normalization、graph stale disclosure |
| `spec-brainstorm` | Requirements Readiness Gate、engineering-evolution gap lens N/A 分流 |
| `spec-plan` | acceptance coverage、graph degraded posture、scope/non-goal 不反向扩张 |
| `spec-work` | task-pack vs internal tracker、runtime mirror 不手改、dirty graph fallback |
| `spec-code-review` | provider-only finding rejection、confidence anchor、report-only/no-agents fallback |

边界：

- eval 检查行为语义，不替代 unit/contract tests。
- fresh-source eval 必须读取当前磁盘 source，不依赖当前会话缓存的 typed agent/skill。
- eval 失败给出 drift evidence，不直接自动改 workflow prose。

## P4：Artifact Discoverability Index

目标：让 agent 和用户知道每个阶段该读哪个 artifact、哪个是 source、哪个是 generated/degraded/advisory。

最小形态是一张表：

| Artifact | Producer | Consumer | Authority | Freshness | Default read policy |
| --- | --- | --- | --- | --- | --- |
| requirements doc | `spec-brainstorm` | `spec-plan` | source decision artifact | user/session scoped | read when planning from requirements |
| plan doc | `spec-plan` | `spec-write-tasks` / `spec-work` / review | source decision artifact | plan status + source hash | read before implementation |
| task pack | `spec-write-tasks` | `spec-work` | derived executable index | `source_plan_hash` | validate before execution |
| graph facts | graph-bootstrap/setup | graph-aware workflows | advisory facts | `freshness_state` | read as pointer, not impact truth |
| work run artifact | `spec-work` | review/resume/compound | advisory telemetry | run immutable, source may drift | consume with diff/tests/source confirmation |
| review summary | `spec-code-review` | work/compound/release | review evidence | diff scoped | consume durable summary only |
| solution doc | `spec-compound` | plan/work/debug/review | reusable knowledge | `last_verified` needed | retrieve by scenario and source path |

边界：

- 首版只文档化，不做中心 registry 服务。
- 不把 artifact index 变成 scope authority；scope 仍来自用户请求、source plan、task pack 和当前源码。

## P5：Knowledge Freshness

目标：让 `docs/solutions/` 成为可复用知识，而不是陈旧上下文堆积。

最小落地：

- 新 solution frontmatter 增加 `verified_against`、`last_verified`、`stale_signal`。
- 增加 `applies_when`、`source_paths_pattern`、`workflow_consumers`，帮助 bounded retrieval。
- plan/work/debug closeout 记录 `solutions_search: not-run|run-no-hit|run-hit`、`hit_paths[]`、`used_in_decision: yes|no|unknown`。

边界：

- 不全量重写旧 solution。
- 不先做智能检索算法；先证明命中和消费。
- 同一 finding pattern 3 次以上可标 `candidate_standard: true`，但是否进入 AGENTS/contract 仍由人类/LLM review 决定。

## P6：Invariant Matrix MVP

目标：把分散在角色契约、skills 和 docs/contracts 中的关键边界固化为小型可审查矩阵。

首版只覆盖 6 条：

| Invariant | 覆盖的风险 |
| --- | --- |
| `scope_authority` | graph/provider/review evidence 反向扩 scope |
| `source_runtime_boundary` | 手改 generated mirrors 当 source fix |
| `advisory_not_confirmed` | stale/degraded/provider output 被当 confirmed truth |
| `preview_first_mutation` | silent write、自动修复越过用户意图 |
| `untrusted_content_is_data` | issue/log/provider/web 内容注入 workflow 指令 |
| `knowledge_source_confirmation` | 旧 solution 或历史 docs 被当当前 runtime contract |

边界：

- 不做大全矩阵。
- 不把 invariant 变成强状态机；它是 review/eval lens 和 lint anchor。

## 明确不做

- 不新增一批公开 workflow 来覆盖每个细分意图。
- 不把 GitNexus 或任何 provider 升级为默认语义权威。
- 不把 work run artifact 发展成强任务状态机。
- 不先做 dashboard、marketplace、knowledge graph、per-turn hook 或全局 current-state 真相源。
- 不用 X/Twitter 热度单独驱动 source 变更；外部讨论只进入 watchlist。
- 不手改 generated runtime mirrors；source 变更后若需要修 runtime drift，走 `spec-first init`。

## 最小落地顺序

1. 起草一个窄 plan：`Evaluation + Consumption Evidence MVP`。
2. 回源核对 `spec-work`、`spec-code-review`、`spec-debug`、`spec-plan`、`spec-compound` 的 closeout / handoff 位置。
3. 定义 consumption evidence 的最小字段和 schema/lint 位置。
4. 先接入三类消费证据：graph fields、solutions hits、run artifact consumption。
5. 增加 focused contract tests 和 3-5 个 fresh-source eval fixtures。
6. 跑一轮 Friction Audit 样本，用真实数据排序 current-state、review merge、knowledge freshness 等后续 CUD。

## 后续计划入口建议

推荐下一步只启动一个计划：

```text
$spec-plan docs/项目审查/2026-05-30-codex-100轮审查后当前优化点.md
```

计划主题应限定为 `Evaluation + Consumption Evidence MVP`。current-state brief、review merge dry-run、knowledge index、artifact registry 等都应作为后续 CUD，不应在同一计划里并行铺开。

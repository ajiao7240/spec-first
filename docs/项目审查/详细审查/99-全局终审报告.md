## 2. 全局事实、推断、假设与待验证项

### 2.1 全局事实

- 原报告覆盖 36 个 Skill、51 个 Agent；本次已生成同数量单项报告。
- 当前 source inventory 与原报告一致：`skills/` 下 36 个 `SKILL.md`，`agents/` 下 51 个 `.agent.md`。
- 本次使用 Graphify 只作导航，结论依据仍为原报告与 source 文件。
- 本次未修改 generated runtime mirrors：`.claude/`、`.codex/`、`.agents/skills/`。

### 2.2 全局推断

- 推断 1：最大系统性收益来自三类收敛：大 workflow progressive disclosure、旧 helper/agent contract 补齐、reviewer persona catalog/schema 对齐。
  - 推断依据：P1 对象集中在入口过重、缺输出契约、高权限/外部工具边界和重复 reviewer 角色。
  - 可信度：高
- 推断 2：当前不需要批量新增 Skill/Agent；新增会扩大 catalog 噪声和治理成本。
  - 推断依据：原报告已指出 51 个 agent 中存在多组重叠，36 个 skill 中多个缺 eval/contract。
  - 可信度：高

### 2.3 全局假设

- 假设 1：后续优化会按 P1/P2 分批进入 source 修改，不在本次审查中直接改 skill/agent 行为。
  - 如果假设不成立：必须补 CHANGELOG、focused tests 和 fresh-source eval。
- 假设 2：P1 对象仍有真实消费者。
  - 如果假设不成立：应优先退役/合并，而不是扩写 prompt。

### 2.4 全局待验证项

| 待验证项 | 验证方式 | 影响范围 | 优先级 |
|---|---|---|---|
| Agent lifecycle 分类是否与实际消费者一致 | 对 `agents/` 做引用回源，建立 persona catalog | 所有 reviewer/analyst agent | P1 |
| 大 workflow 下沉是否不破坏可执行性 | focused contract tests + fresh-source eval | `spec-code-review`、`spec-plan`、`spec-work` 等 | P1 |
| 外部工具/API 边界是否当前有效 | 浏览官方/primary source，记录日期与限制 | Gemini、Proof、GitHub、Slack、Figma、Context7/Web、XcodeBuildMCP | P1 |
| eval fixture 是否被 readiness checker 正确识别 | 运行 eval normalizer / skill-audit focused tests | production/library/governed skill | P2 |

## 3. 角色 1：spec-first 总架构师终审

### 3.1 核心判断

当前 Skill / Agent 体系的骨架成立，但入口体量、重复 persona 和证据回路不均衡。下一阶段应优先收敛边界，而不是扩展能力数量。

### 3.2 主要架构问题

- 大 workflow 将流程细节、catalog、fallback、示例和治理规则混在入口，增加初载成本。
- Reviewer agent 分层不够稳定，always-on / conditional / deep-dive / helper / deprecated 候选混在同一 catalog。
- 高权限 helper 与外部工具 agent 的 authority、input、output、fallback 边界不足。

### 3.3 架构优化建议

- 以 `spec-code-review`、`spec-plan`、`spec-work` 为第一批 progressive disclosure 样板。
- 建立 Agent lifecycle catalog，并把重复角色合并或分层。
- 将 Evidence/Knowledge 回流字段作为轻合同，不做复杂状态机。

### 3.4 风险与代价

- 入口瘦身过度会让执行者找不到关键规则；必须用 references 路由和 focused tests 保护。
- 合并 agent 会影响已有 workflow 选择逻辑；需先做引用回源和兼容 mapping。

## 4. 角色 2：企业研发落地负责人终审

### 4.1 核心判断

落地层面的首要问题是使用成本和可解释性：团队需要知道何时用哪个对象、输出如何被接收、失败时如何降级。

### 4.2 主要落地问题

- 部分 helper skill 缺输入输出和失败处理，用户容易把高权限动作当普通辅助。
- 外部工具/组织资料 agent 需要更强隐私、权限和当前性边界。
- 缺少足够 eval/fixture 让团队相信 prompt 改动不会漂移。

### 4.3 落地优化建议

- 先修 P1 helper 和 external-tool 对象：`git-worktree`、`git-clean-gone-branches`、`resolve-pr-feedback`、`proof`、`gemini-imagegen`、`test-browser`、`test-xcode`。
- 每个 P1 对象补“何时不用”和“失败时输出什么”，降低培训成本。
- 用 README/索引式 catalog 告诉团队核心链路、扩展链路和实验链路。

### 4.4 风险与代价

- 如果先做深层重构而非可见 contract，团队短期收益不明显。
- 过度强调治理术语会降低普通使用者采纳度。

## 5. 角色 3：AI Agent 工程化专家终审

### 5.1 核心判断

Agent/Skill 工程化缺口集中在可评测、可回放、工具边界和防幻觉机制。当前 source 已有强治理意图，但部分对象仍停留在 prose 约定。

### 5.2 主要工程化问题

- 多数旧 Agent 缺稳定 output schema、confidence calibration 或 What-you-don't-flag。
- 部分 Agent 声称需要 Web/Figma/Slack/GitHub 等事实，但 tools 为空或依赖 orchestrator 提供事实。
- Skill eval fixture 格式与 readiness checker 尚需统一。

### 5.3 工程化优化建议

- 为 code-review/doc-review persona 建立最小 finding schema。
- 对无工具但需外部证据的 Agent 增加 `insufficient_evidence` fallback。
- 对 production/library/governed Skill 补 trigger + output eval，区分 recorded fixture、command-runner evidence 和 model-executed evidence。

### 5.4 风险与代价

- eval 过窄会诱导过拟合；需要 near-neighbor 和 failure cases。
- 工具声明扩张可能越权；应优先让 orchestrator 提供事实，Agent 只做语义判断。

## 6. 三方终审交叉讨论

### 6.1 共识

- 不新增批量对象，先治理已有对象。
- P1 优先：false-positive scanner、helper contract、大 workflow 瘦身、persona catalog/schema。
- Evidence 和 Knowledge 必须回源，不把 advisory facts 写成 confirmed truth。

### 6.2 分歧

| 分歧点 | 总架构师观点 | 研发落地负责人观点 | Agent 工程化专家观点 | 最终取舍 | 取舍理由 |
|---|---|---|---|---|---|
| 是否先做大 workflow 重构 | 应优先处理最高频主链路 | 先修高权限 helper，用户可见收益更直接 | 两者都需 eval 保护 | 并行分两批 P1 | 大 workflow 影响面大，helper 风险高；两者都是 P1 但验证路径不同。 |
| 旧 Agent 是合并还是补强 | 重叠角色应合并 | 若团队已熟悉名称，直接删除会增加迁移成本 | 先 catalog 再合并 | 先分类后合并 | 没有消费者回源前不直接删除，但不继续扩写重复角色。 |
| 外部研究 Agent 是否应声明工具 | 架构上可由 orchestrator 提供事实 | 落地上工具能力要显式 | 工程上避免 Agent 越权 | 写清事实来源边界 | 有工具则声明；无工具则必须消费父 workflow facts 并能 degraded。 |

### 6.3 最终裁决原则

1. 不破坏 spec-first 主链路闭环。
2. 优先保证可执行、可验证、可沉淀。
3. 优先解决高频、高风险、高阻塞问题。
4. 避免为了概念完整而增加团队使用成本。
5. 不为局部 Skill 优化牺牲整体架构清晰度。

## 11. 终审结论

1. 当前体系的核心优势是主链路已完整覆盖需求、计划、执行、审查和知识沉淀。
2. 最关键问题是入口体量、重复 agent、缺稳定输出契约、缺 eval/fallback、外部工具边界弱。
3. 最推荐路径是先修审计误报，再补高权限 helper contract，同时启动主 workflow 瘦身样板。
4. 应立即改造的对象集中在 P1：spec-skill-audit、高权限 helper、spec-code-review、spec-plan、spec-work、Agent catalog。
5. 应暂缓的是批量新增 Skill/Agent，以及没有消费者证据的旧 agent 扩写。
6. 长期演进应围绕 Evidence/Knowledge 回流、eval fixture 标准化和治理元数据最小投影。
7. 这些单项报告可作为后续 $spec-work 分批优化依据；每批修改仍需 source 回源、CHANGELOG、focused tests 和 fresh-source eval 说明。

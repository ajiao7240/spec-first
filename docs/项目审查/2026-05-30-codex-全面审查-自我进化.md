# 2026-05-30 Codex 全面审查：spec-first 自我进化

> 本文档按用户要求，对 `Codebase -> Graph -> Spec -> Plan -> Tasks -> Code -> Review -> Knowledge` 核心链路做 50 轮多视角审查。每轮次均追加到同一文档中，最后由 `2026-05-30-codex-全面审查-自我进化-result.md` 汇总复审结论。

## 执行边界

- 角色基线：`docs/10-prompt/结构化项目角色契约.md`。核心判断是 spec-first 是否进一步成为可治理、可验证、可复用、可沉淀的 AI coding harness。
- Source/runtime 边界：本次只写 `docs/项目审查/` 与 `CHANGELOG.md`，不修改 `.claude/`、`.codex/`、`.agents/skills/` generated runtime mirrors。
- 图谱证据：`.spec-first/graph/graph-facts.json` 显示 `query_global_graph=true` 且 ready provider 包含 `gitnexus`，但 startup reminder 报告 freshness stale/dirty、impact/review none。因此 GitNexus 查询仅作为 session-local orientation，不作为 confirmed impact evidence。
- 内部 confirmed evidence：角色契约、README/README.zh-CN、`docs/contracts/**`、`skills/spec-*`、`src/cli/task-pack.js`、`src/cli/helpers/spec-work-run-artifact.js`、`src/cli/helpers/review-pre-facts/**`、现有审查/路线图/solutions。
- 外部一手资料：Anthropic Claude Code best practices、OpenAI Codex AGENTS/skills/eval/self-improving agents 资料、GitHub Spec Kit、Kiro specs/steering/hooks、GitHub Copilot coding-agent/custom-instructions、Gemini CLI GEMINI.md、METR long-horizon task measurement、OpenTelemetry GenAI semantic conventions。

## 总体判断

spec-first 的方向是对的：它不是 prompt collection，而是围绕 AI coding 生命周期建立 repo-local harness。当前最强资产是 source/runtime 边界、graph readiness、task-pack identity、review closure、run artifact、context governance 与 docs/solutions 知识沉淀。当前最大缺口不是再增加 workflow，而是把这些机制连成可度量的 Evaluation Harness：知道每次流程是否真的降低了用户摩擦、减少了返工、提升了交付质量。

## 既有草稿吸收说明

复审时发现同目录存在一个未跟踪的非目标文件 `2026-05-30-全面审查-自我进化.md`，其内容只推进到前 8 轮，且文件名不符合用户指定的 `2026-05-30-codex-全面审查-自我进化.md`。本次不删除、不改写该文件；本文按指定文件名完成 50 轮，同时吸收其中 8 个高价值洞察：

- Codebase 节点缺少可复用的 current-state context artifact，导致 brainstorm/plan/work/debug 重复做 bounded reads。
- 当前 Graph 更接近 definitions-only orientation，不能被下游误读为 blast-radius impact evidence。
- brainstorm rigor probe 存在“prose 维持的隐式状态机”风险，适合拆成半结构化 checklist。
- plan 的 Graph Readiness prose 对 HOW 决策文档形成认知超载，U-ID/spec_id 冲突扫描更适合脚本化。
- task-pack identity/freshness 校验规则在 plan/write-tasks/work 三处重复，存在自指 drift 风险。
- spec-work Phase1 承载过多分布式执行协议语义，run artifact 应明确是 advisory execution telemetry。
- spec-code-review 的 deterministic merge/dedup/confidence 部分应从 1100 行 prose 中抽给脚本，LLM 只保留语义判断。
- compound 是链路末端一致性较弱的一环，knowledge 写入已有治理，反向消费命中率尚未可观测。

## 50 轮次审查

### 第 01 轮：使命与主链路完整性

- 视角：Spec-First Evolution Architect。
- 证据：角色契约定义 `Codebase -> Graph -> Spec -> Plan -> Tasks -> Code -> Review -> Knowledge`，README 把 spec-first 定位为 Claude/Codex 的 spec-driven AI engineering workflows。
- 判断：链路表达清晰，但日常改动仍容易落入局部 contract 优化，缺少每次优化对整条链路的回溯说明。
- 优化：为重大 plan 增加 `chain_impact` 小节，强制说明影响哪个节点、改善哪个输入/证据/复用环节、不会触碰哪个边界。

### 第 02 轮：Codebase 输入质量

- 视角：代码库事实入口。
- 证据：`context-governance.md` 要求普通 workflow 先消费 source-of-truth 和 bounded source reads，排除 generated runtime/audit artifacts。
- 判断：默认排除策略正确，但 worker/reviewer 是否实际拿到最小充分 context 仍依赖 prompt 执行质量。
- 优化：把 `context-bundle.v1` 从可选 helper 推成中大型 workflow 的默认 handoff envelope，并记录 included/excluded reason_code。

### 第 03 轮：Graph readiness 可信度

- 视角：Graph 证据治理。
- 证据：graph facts 当前 `freshness_state=dirty-advisory`，startup reminder 显示 query/context full、impact/review none。
- 判断：GitNexus 可用于定位，但不能声称全量影响分析。当前系统对 stale/dirty 已诚实，但用户心智仍可能把“有图谱”误解为“图谱可信”。
- 优化：所有 graph-consuming artifact 都显示 `freshness_state + capability_used + source_reads_required` 三元组，避免单独展示 ready/provider 名称。

### 第 04 轮：Graph 到 source 的闭环

- 视角：provider facts 消费。
- 证据：`review-pre-facts-extraction.md` 要求 provider results 经过 normalization、snapshot validation、provenance 后才进入 review context。
- 判断：review 侧已经成熟，plan/debug/work 的 graph evidence 同等成熟度仍不完全一致。
- 优化：复用 review-pre-facts 的 fact normalization 思路，给 plan/debug 输出统一的 compact graph-evidence summary，不复制 provider raw output。

### 第 05 轮：Provider 内部实现泄漏

- 视角：外部工具边界。
- 证据：角色契约禁止 downstream 依赖 provider 内部实现；README 指向 GitNexus readiness/capability vocabulary。
- 判断：当前文档已经反复声明 provider 是 evidence input，不是 semantic authority。但 CHANGELOG 中 provider pin、rc、上游 bug 仍频繁进入用户可见叙事。
- 优化：建立 provider incident appendix，把 pin/crash 细节沉到维护者文档，用户文档只暴露 capability state、degraded reason 与 next action。

### 第 06 轮：Spec 入口质量

- 视角：需求澄清。
- 证据：`spec-brainstorm` 有 Requirements Readiness Gate，要求 goal/non-goal/assumption/risk/handoff。
- 判断：它能显著提高 WHAT 质量，但对于已有系统增量需求，brownfield current-state analysis 仍是刚形成的增强方向。
- 优化：在 brainstorm/PRD 需求文档中标准化 `Current State Evidence`，区分用户观察、source-confirmed fact、graph-advisory pointer。

### 第 07 轮：Spec identity

- 视角：artifact identity。
- 证据：`requirements-capture.md` 要求新需求文档带 `spec_id`，并说明它是跨 requirements/plan/task-pack 的 identity，不是 freshness。
- 判断：`spec_id` 是轻量 contract 的好例子。风险是 legacy 文档无 identity 时链路降级解释不够醒目。
- 优化：在 plan frontmatter 附近显示 `trace_strength: inherited | plan-local | weak-legacy`，让 reviewer 一眼看到 spec chain 可信度。

### 第 08 轮：Spec 到 Plan 的单真相源

- 视角：handoff ownership。
- 证据：`spec-plan` 会搜索 `docs/brainstorms/*-requirements.md`，并把 origin requirements 作为 plan 输入。
- 判断：这避免了“另起 PRD producer”的多真相源风险。近期 PRD 文档也已识别新增 `spec-prd` 可能制造双 producer。
- 优化：除非明确改造 artifact ownership，不新增并行需求 producer；已有系统的 PRD 增强优先作为 brainstorm reference 或 plan intake lens。

### 第 09 轮：Plan 的范围控制

- 视角：计划质量。
- 证据：`spec-plan` 要求 implementation units、scope boundaries、test scenarios、deferred unknowns、Graph/GitNexus Evidence。
- 判断：计划能力强，但容易变长。对小改动而言，计划成本可能高于收益。
- 优化：按 risk/uncertainty 而不是文件数量决定 plan 深度；小任务允许 `mini-plan`，但仍保留 acceptance/test/source-runtime 边界。

### 第 10 轮：Plan 的 graph evidence posture

- 视角：graph-informed planning。
- 证据：`spec-plan` 要求读取 `references/graph-evidence-posture.md`，并区分 evidence_grade、evidence_posture、source_tags、source_reads_required。
- 判断：这是当前 graph 消费最关键的语义 contract。风险是执行者从 memory 重建规则导致 drift。
- 优化：把 graph evidence posture 的关键 enum 做成 source contract test，扫描 plan template 和 downstream workflow 是否使用同一字段名。

### 第 11 轮：Tasks 派生边界

- 视角：任务编译。
- 证据：`spec-write-tasks` 明确 task pack 只能从 source plan 派生，不能二次发明 scope。
- 判断：`source_plan_hash + spec_id + Task Pack Contract` 是防止 AI 偷换任务的核心护栏。
- 优化：在 task pack 顶部增加 `not_scope_authority` 明示，提醒 executor free-form task cards 不能越过 source plan。

### 第 12 轮：Task granularity

- 视角：可执行性。
- 证据：`spec-write-tasks` 支持 execution_waves、dependencies、stop_if、test_focus。
- 判断：粒度已经能支撑中大型工作，但缺少对“任务太小导致上下文碎片化”的反向约束。
- 优化：增加 task-pack quality heuristic：每个 task 应闭合一个可验证 vertical slice；过小 task 只能作为 checklist，不生成独立 wave。

### 第 13 轮：Code 执行反馈环

- 视角：执行质量。
- 证据：`spec-work` 要求编辑前建立或尝试最小反馈 loop，docs-only 也应有 contract/render/diff-shape check。
- 判断：这是把 AI coding 从“写完再说”拉回工程闭环的关键。当前缺口是反馈 loop 未结构化沉淀。
- 优化：在 spec-work run artifact 中加入 `feedback_loop.kind/status/coverage_note`，用于后续评估哪些 loop 真正发现问题。

### 第 14 轮：Run artifact 集成

- 视角：执行证据。
- 证据：`spec-work-run-artifact.schema.json` 已把 `workflow_integrated`、reason_code、validation、changed_files、resume_evidence 结构化。
- 判断：run artifact 是连接 Code、Review、Knowledge 的关键新资产。风险是“写了 artifact”被误当作“产生价值”。
- 优化：把 artifact 价值度量延后到 Evaluation Harness：记录 downstream 是否实际消费 run artifact、是否减少 review/debug 反复。

### 第 15 轮：Review 入口强度

- 视角：质量门。
- 证据：`spec-code-review` 有多 persona、confidence anchor、merge/dedup、validator、autofix_class、coverage disclosure。
- 判断：Review 是 spec-first 当前最强能力，但也是复杂度最高的能力。
- 优化：引入 review quality fixture matrix，按缺测试、需求偏离、provider stale、security、cross-platform 等场景评估 reviewer 发现率和假阳性率。

### 第 16 轮：Review pre-facts

- 视角：审查前事实准备。
- 证据：`review-pre-facts` 会读取 canonical graph readiness、验证 snapshot、生成 MCP query plan 或 fallback direct reads。
- 判断：这是 scripts prepare facts, LLM decides 的正例。风险是 helper 输出被当成 finding。
- 优化：所有 pre-facts 输出必须带 `not_a_finding` 语义标记，review synthesis 只能把它作为 evidence pointer。

### 第 17 轮：Finding schema 与证据锚点

- 视角：review output contract。
- 证据：`spec-code-review` 要求 surviving finding 必须有 source/test/contract evidence，provider output alone 不够。
- 判断：该约束对抗“图谱/LLM 幻觉 finding”非常关键。
- 优化：把 `provider_only_finding_rejected` 计数进入 Coverage，让系统能度量 provider 噪声和 reviewer 过滤质量。

### 第 18 轮：Debug root-cause

- 视角：故障诊断。
- 证据：`spec-debug` 要求先建立复现、假设、probe evidence，再修复；graph/MCP evidence 只作 advisory debugging context。
- 判断：debug workflow 方向正确，但 root-cause ledger 与 work/review artifact 的衔接不如 spec-work run artifact 结构化。
- 优化：定义最小 `debug-ledger.v1` summary，不做全量状态机，只记录 symptom、repro、hypothesis、probe、confirmed_root_cause、fix evidence。

### 第 19 轮：Knowledge capture

- 视角：知识沉淀。
- 证据：`spec-compound` 将 solved problem 写入 `docs/solutions/`，并要求 graph-informed lesson 必须 source-confirmed。
- 判断：Knowledge 节点存在且边界清晰。缺口是 refresh 与发现机制仍偏人工。
- 优化：为 docs/solutions 增加轻量 index：category、tags、last_verified、known_stale_signal、consumer workflows。

### 第 20 轮：Knowledge freshness

- 视角：可复用知识可信度。
- 证据：`spec-compound-refresh` 只在窄范围触发，不默认广扫；当前 docs/solutions 约 15 个。
- 判断：知识库规模小而有价值，适合建立 freshness discipline；过早引入复杂知识图谱会过度设计。
- 优化：每次 compound 写入 `verified_against` 与 `review_due_hint`，refresh 只按明确 stale signal 触发。

### 第 21 轮：Source/runtime 边界

- 视角：双宿主治理。
- 证据：角色契约、README、context-governance 均声明 source assets 与 generated runtime mirrors 分离。
- 判断：这是 spec-first 相比普通 prompt repo 的核心差异化。风险是用户为了快速修 runtime 手改 mirror。
- 优化：doctor/update 输出中继续把“修 source 后 init”作为唯一修复路径，并在 runtime mirror 文件头放 source pointer 与 no-edit reason。

### 第 22 轮：Claude/Codex 双宿主

- 视角：host parity。
- 证据：README.zh-CN 记录 Claude 19 commands + Codex 19 workflow skills 等 runtime 交付面；`dual-host-governance` 是 contract 源。
- 判断：双宿主 parity 是护城河，也是维护成本源。
- 优化：新增 host parity snapshot test，把 public workflow inventory、runtime projection、README 数量和 CHANGELOG 说明绑定在同一检查中。

### 第 23 轮：Host 指令层复用

- 视角：instruction hygiene。
- 证据：`context-governance.md` 的 Host Instruction Reuse Policy 要求普通 workflow 优先使用已加载指令，而不是反复读取 AGENTS/CLAUDE。
- 判断：这有效降低 token 噪声，也避免 generated/runtime 指令被误当 source。
- 优化：在 workflow closeout 中可选记录 `host_instruction_reread_reason`，用于发现哪些 workflow 仍过度读取入口文档。

### 第 24 轮：行业 host 指令标准化

- 视角：外部趋势。
- 证据：OpenAI Codex 支持 `AGENTS.md`，Gemini CLI 使用 `GEMINI.md` 分层记忆，GitHub Copilot 支持 repository custom instructions。
- 判断：host-native instruction 越来越标准化，spec-first 不应和 host 竞争“指令文件”，而应做 source-governed projection 与证据闭环。
- 优化：建立 host instruction adapter matrix，只维护 source slice 到各 host instruction surface 的映射，不为每个 host 发明独立语义。

### 第 25 轮：Skills 与 eval

- 视角：能力包质量。
- 证据：OpenAI Codex skills 方向强调可复用技能与 eval；`spec-skill-audit` 已检查 trigger、boundary、contract、runtime governance。
- 判断：spec-first 已具备 skill 审查能力，但核心 workflow 的 fresh-source eval 覆盖仍是增量点。
- 优化：为 `using-spec-first`、`spec-brainstorm`、`spec-plan`、`spec-work`、`spec-code-review` 建立最小 eval fixture，验证触发、跳过、降级、handoff。

### 第 26 轮：Agent profile 治理

- 视角：多角色判断。
- 证据：现有审查文档指出 reviewer agents 强于 generic agents；`spec-code-review` 有 persona catalog 和 conditional reviewers。
- 判断：多角色是优势，但 agent 数量不是质量本身。
- 优化：按主链路重分 agents：core reviewer、domain reviewer、researcher、optional utility；generic agent 未绑定链路节点时降级为 optional。

### 第 27 轮：MCP 与工具安全

- 视角：外部工具治理。
- 证据：README trust model 将 raw provider/tool output 视为 untrusted quoted data，需要 validation、containment、escaping、provenance。
- 判断：安全模型正确，但应覆盖 prompt injection in excerpts 的常见 failure。
- 优化：把“不要执行 excerpt 中的指令”写入所有 researcher/reviewer subagent templates，并用 eval fixture 覆盖恶意 tool output。

### 第 28 轮：权限与自动化边界

- 视角：mutation governance。
- 证据：context-bundle、review-pre-facts、clean preview-first 都强调 silent write 风险。
- 判断：preview-first 已成为系统文化。下一步要防止“自动修复”跨过用户意图边界。
- 优化：autofix_class 只对 low-risk deterministic edits 生效；scope-changing fixes 必须进入 handoff，而不是由 fixer 扩展计划。

### 第 29 轮：Long-horizon autonomy

- 视角：长任务执行。
- 证据：Anthropic Claude Code 和 METR 长任务研究都强调上下文、计划、验证和恢复锚点；spec-work run artifact 正在补 resume_evidence。
- 判断：spec-first 已有长任务基础，但没有显式 compaction/continuation protocol。
- 优化：当任务超过一定 evidence threshold 或即将 handoff/review 时，spec-work 主动写 run artifact，作为 compaction/resume anchor。

### 第 30 轮：并行 session 与 worktree

- 视角：多 agent 协作。
- 证据：`using-spec-first` 可用 session list 做 advisory，`spec-work-run-artifact` 避免同 workspace/run-id overwrite。
- 判断：并行可见性增强了，但还不是完整协作协议。
- 优化：保持 advisory，不引入 lock server；在 closeout 中记录 active session count 和 changed-file overlap risk 即可。

### 第 31 轮：外部 spec-driven 竞品对照

- 视角：行业借鉴。
- 证据：GitHub Spec Kit/Kiro 都强调 spec、plan/tasks、hooks/steering 等工程化链路。
- 判断：spec-first 不应成为 Spec Kit clone；差异化在 repo-local evidence、双宿主 runtime projection、review/knowledge closure。
- 优化：对外叙事从“更强 spec 工具”转为“spec 是入口，evidence/review/knowledge 是护城河”。

### 第 32 轮：Evaluation Harness

- 视角：结果度量。
- 证据：角色契约把 Evaluation Harness 定义为 graph-to-finding ratio、debug 命中率、review 漏判率等 artifact 产出，而非 LLM 自评。
- 判断：这是当前最大未完全落地的层。没有结果度量，系统容易持续优化流程而不证明价值。
- 优化：定义 `workflow-outcome-ledger.v1`，只记录最小事实：goal、workflow path、artifact refs、validation result、review result、residual、rework signal。

### 第 33 轮：用户摩擦

- 视角：研发效率。
- 证据：`2026-05-28-spec-first-用户体感提升路线图.md` 已提出 P-friction first-run/standard feature/debug 等实证审计。
- 判断：这是比继续加 contract 更高杠杆的下一步。
- 优化：优先跑 5 个真实任务 friction audit，记录时间、卡点、重复输入、降级原因、最终是否 ship。

### 第 34 轮：Artifact discoverability

- 视角：产物地图。
- 证据：README 有 Durable entities 与 artifact roots；README.zh-CN 有 trust model 和 runtime reference。
- 判断：产物越来越多，用户和 agent 都需要“当前任务该读哪个 artifact”的指南。
- 优化：生成 `artifact-summary index`，按 workflow 列出 producer、consumer、authority、freshness、default read policy。

### 第 35 轮：CHANGELOG 治理成本

- 视角：维护成本。
- 证据：仓库要求任何 source 变更更新 CHANGELOG，当前 CHANGELOG 信息密度极高。
- 判断：可追溯性强，但人工成本高，且重要信号被长条目淹没。
- 优化：保留详细条目，同时为 release note 生成短摘要视图；CHANGELOG 作为事实 ledger，不承担完整设计说明。

### 第 36 轮：路线图弹性

- 视角：长期演化。
- 证据：版本规划已有 anti-thesis，要求每个 Milestone 标注 complement/compete，并每 90 天复核 host 能力。
- 判断：这是成熟的反脆弱设计。缺口是复核尚未制度化为脚本或日历。
- 优化：新增 `docs/业界分析/host-capability-watchlist` 的 recurring review template，避免路线图被过期行业事实支撑。

### 第 37 轮：Spec-centric vs evidence-centric

- 视角：战略定位。
- 证据：路线图 Open Questions 已指出 spec-driven 范式可能降温，Evidence/Review 仍成立。
- 判断：产品名叫 spec-first，但护城河不应只绑定 spec。真正稳定的是 evidence-first delivery loop。
- 优化：对外心智调整为“spec anchors intent, evidence proves progress, review closes quality, knowledge compounds lessons”。

### 第 38 轮：Brownfield 需求迭代

- 视角：已有系统上的需求。
- 证据：近期 PRD 文档指出 `spec-prd` 差异化是 GitNexus + bounded source reads 的 current-state analysis。
- 判断：brownfield PRD 是真实增量，但不一定需要新公开 workflow。
- 优化：先在 brainstorm/plan 中添加 brownfield lens，跑 3 个真实需求后再决定是否升格为 `spec-prd`。

### 第 39 轮：Schema 与 contract budget

- 视角：轻量契约。
- 证据：角色契约要求 Light contract；现有 schema 包含 task-pack、review finding、work run artifact、quality gates。
- 判断：schema 已经不少，继续增加必须证明 consumer 存在。
- 优化：每个新 schema 必填 `producer_available`、`first_consumer`、`drop_condition`，无 consumer 不进入 source contract。

### 第 40 轮：Release 与 package delivery

- 视角：发布可信度。
- 证据：package.json scripts 有 release install、website sync、runtime catalog、release continuity checks。
- 判断：发布面较成熟。风险在 docs/source/runtime 三者数量漂移。
- 优化：release gate 增加 public workflow inventory diff：skills、README 表格、runtime catalog、init expected output 必须一致。

### 第 41 轮：测试矩阵

- 视角：验证覆盖。
- 证据：unit/smoke/integration/e2e、mcp-setup、graph-bootstrap、ai-dev quality gate、release tests 已存在。
- 判断：确定性代码测试强，workflow 语义 eval 仍偏少。
- 优化：把 workflow semantic eval 与 unit tests 分层：脚本 contract 用 Jest/shell，prompt behavior 用 fresh-source eval fixture。

### 第 42 轮：Codebase indexing beyond Graph

- 视角：代码理解输入。
- 证据：当前 graph provider 主要是 GitNexus，context bundle 支持 related paths 和 evidence paths。
- 判断：不要把 GitNexus 变成唯一代码理解方式。AST-grep、rg、language server、test discovery 都应作为 bounded evidence。
- 优化：定义 codebase orientation palette：Graph、AST、text search、package scripts、test topology，各自有 authority/freshness 限制。

### 第 43 轮：数据 provenance 与脱敏

- 视角：安全与合规。
- 证据：README trust model 要求 raw provider/tool output 进入 durable artifacts 前经过 validation、containment、escaping、excerpt cap、provenance/readiness classification。
- 判断：原则完备，但需要更接近实现的检查。
- 优化：给 artifact writers 增加 shared sanitizer contract，禁止写入 raw logs、credentials、absolute sensitive paths，除非 artifact 类型显式允许且脱敏。

### 第 44 轮：人类确认点

- 视角：协作体验。
- 证据：workflow 里已有 preview-first、question tool、handoff contract、report-only modes。
- 判断：确认点太多会降低速度，太少会越权。关键是只在 scope/authority/mutation 变化处确认。
- 优化：统一 gate taxonomy：scope gate、mutation gate、degraded-evidence gate、externalization gate；普通信息不弹确认。

### 第 45 轮：Review 到 Knowledge 的学习机会

- 视角：复用闭环。
- 证据：`spec-code-review` Stage 6 有 Learning Capture Recommendation，要求只在有通用 lesson 时建议 compound。
- 判断：方向正确，避免每次 review 都写知识。
- 优化：把 review finding 的 repeated-pattern count 作为 compound 建议输入，3+ 同类 finding 时强提示，单次机械问题跳过。

### 第 46 轮：Skill audit 自检

- 视角：自我审查机制。
- 证据：`spec-skill-audit` 审查 trigger precision、scope boundary、eval readiness、runtime governance，并强调 deterministic facts first。
- 判断：这是 meta-harness 的关键能力，但不应变成每次小改必跑的大流程。
- 优化：按变更类型触发：workflow/agent/skill prose 变更跑 focused audit；runtime generation 变更跑 dual-host/governance audit；docs-only 局部变更只做 direct source review。

### 第 47 轮：Scenario capability

- 视角：场景自适应。
- 证据：`scenario-capability-matrix.md` 用 first-time、foreign-residual、unavailable-provider 等 scenario class 解释 degraded/blocked/fallback。
- 判断：它把环境复杂性表达为 advisory facts，而不是硬状态机，符合角色契约。
- 优化：避免 scenario class 膨胀；新增 scenario 必须证明会改变 workflow 行为或用户 next action。

### 第 48 轮：父级多仓 workspace

- 视角：repo scope。
- 证据：AGENTS/bootstrap 与 spec-work 都要求写入前明确 target_repo，parent workspace artifacts 只是 advisory。
- 判断：多仓边界已被清晰识别。风险是 graph group evidence 诱导跨 repo scope expansion。
- 优化：所有 cross-repo plan/task 必须用 per-unit target_repo，review closeout 也要按 child repo 分列 graph evidence posture。

### 第 49 轮：Failure modes 与 reason_code

- 视角：可诊断性。
- 证据：contracts 和 scripts 大量使用 reason_code，如 graph-stale、provider-unavailable、wrong-chain、stale_hash、artifact-already-exists。
- 判断：reason_code 是可维护性的基础，但过细会制造伪精度。
- 优化：建立 reason_code registry，分类为 input-missing、freshness、scope、provider、mutation、validation、runtime；新增 code 前检查是否复用已有类别。

### 第 50 轮：最终收敛

- 视角：系统演化方向。
- 证据：内部链路已覆盖 source/runtime、graph readiness、spec/plan/task identity、work artifact、review closure、compound；外部趋势显示 host 能力、spec-driven artifact、agent eval、long-horizon execution 都在快速增强。
- 判断：spec-first 下一阶段不应主要扩 workflow 数量，而应证明现有闭环真实提升交付质量。
- 优化：优先级应转向 Evaluation Harness、friction audit、workflow semantic eval、artifact discoverability、knowledge freshness。新功能只有在明确改善主链路节点且有 consumer 时才进入核心。

## 第二轮 50 轮次调研分析优化（第 51-100 轮）

### 第二轮证据边界

- 内部证据：继续以角色契约、AI Coding Harness 合同、self-reflection capability upgrade contract、downstream graph evidence contract、workflow source、现有 result 文档为 confirmed source。
- Claude 参考：`docs/项目审查/2026-05-30-全面审查-自我进化.md` 作为 `prior_report/advisory`，其 40 轮已覆盖 Codebase current-state、definitions-only graph、handoff、Evaluation 三角、invariant matrix、mutation boundary、prompt injection、CRG/Trellis/Qoder/知识基座等洞察。
- GitHub 调研：使用 `gh search repos` 检索 `spec-driven development ai agent`、`agent skills codex`、`MCP coding agent`、`AI coding agent benchmark`、`context engineering agent`。结果显示高星且相关的主要是 benchmark/context engineering/handbook 类项目，agent skills 生态仍碎片化、低星、个人化。
- X/Twitter 调研：使用 `xreach search` 检索 context engineering、spec-driven development、SWE-bench/coding-agent eval、AGENTS.md/Codex。结果作为 weak/advisory trend signal：讨论集中在 context engineering、spec-driven 对抗 vibe coding、leaderboard 与真实代码库落差、AGENTS.md 规则沉淀。
- 外部一手资料：Anthropic context engineering、OpenAI Codex AGENTS.md/skills、GitHub Spec Kit、Kiro specs/steering/hooks、OpenTelemetry GenAI semantic conventions、METR long tasks。

### 第 51 轮：把 Claude prior report 升级为证据源，而不是重复报告

- 证据：Claude 参考文档已写到 Round 40，且多个点比 Codex 第一轮更细：Graph→Spec/Plan 四份 prose drift、Spec→Plan 语义消费缺口、review merge deterministic 化、knowledge 反向消费黑盒。
- 判断：第二轮不应重新覆盖相同主题，而应把这些洞察转换为可执行的 CUD 候选。
- 优化：把 prior report 的 40 轮归入 self-reflection contract 的 `prior_report` evidence，按 `gap_id -> candidate_upgrade_level -> counter_signal -> plan_handoff_target` 重新索引。

### 第 52 轮：Self-reflection contract 已存在，不需要新 `spec-evolve`

- 证据：`self-reflection-capability-upgrade.md` 明确 non-goal：不默认新增 `spec-evolve`、不新增自我反思 agent、不做 auto-rewrite。
- 判断：用户要求“再次 50 轮”本质是 self-reflection cycle，但现有 contract 已能承载，不需要再发明公开 workflow。
- 优化：后续报告统一使用该 contract 的 CUD vocabulary；新增能力必须先成为 CUD，再进入 `spec-plan`，避免“审查报告直接变实施指令”。

### 第 53 轮：CUD 粒度需要从“建议”压缩为“可计划单元”

- 证据：第一轮和 Claude 报告累计产生 30+ 建议，过多建议本身会成为噪声。
- 判断：自我进化报告的价值不在建议数量，而在能否压缩成 5-7 个有明确 consumer 的 plan candidate。
- 优化：把候选收敛为 CUD-A Evaluation Harness、CUD-B current-state context、CUD-C review deterministic merge、CUD-D knowledge retrieval/freshness、CUD-E invariant matrix、CUD-F graph consumption contract tightening。

### 第 54 轮：Evaluation Harness 应先做 ledger，不先做 dashboard

- 证据：AI Coding Harness contract 把 Evaluation Harness 映射到 quality gates、verification evidence、self-reflection contract，而不是 dashboard。
- 判断：最小可维护机制是本地 ledger + summary，不是可视化平台。
- 优化：定义 `workflow-outcome-ledger.v1` 为 append-only JSONL 或 run artifact summary 扩展，先记录事实：workflow、artifact refs、validation、review result、handoff continued、residual。禁止脚本判断“质量提升”。

### 第 55 轮：Evaluation 三角要区分系统指标、行为质量、用户增益

- 证据：Claude prior report 把 R17/R28/R30 收敛为三支柱：系统指标、fresh-source behavior eval、用户采纳/返工数据。
- 判断：单一指标会诱导 Goodhart。三角互相制衡：调用次数高不等于质量好，eval 过了不等于用户愿意用。
- 优化：每个 CUD 的 effectiveness_check 至少归到三角之一，并说明另外两角为什么暂不测。

### 第 56 轮：GitHub benchmark 项目说明“通用榜单”和“本仓库表现”分离

- 证据：GitHub 搜索到 `mame/ai-coding-lang-bench`、SWE-bench utilities、awesome coding-agent eval 等；X 上也有“SWE-bench 不测你自己的代码库习惯”的讨论。
- 判断：spec-first 不该追逐通用榜单，而应建立 repo-local eval fixtures。
- 优化：把 AI Dev Benchmark 与 fresh-source eval 设计成 repo-local fixtures：本仓库历史 bug、历史 review finding、历史 plan drift，而不是只引用 SWE-bench/Terminal-Bench。

### 第 57 轮：X 的弱信号指向“上下文工程”已成为公共心智

- 证据：X context engineering 查询返回多个关于复杂工程中 coding agent 缺上下文、code graph/queryable repo 的讨论。
- 判断：context engineering 已不是差异化口号，差异化在于 source-confirmed、freshness-aware、scope-bounded 的上下文工程。
- 优化：对外文案避免只说 context engineering；强调 spec-first 的 Context Harness 是 `bounded + provenance + freshness + source/runtime boundary`。

### 第 58 轮：Spec-driven 外部热度增加，反而要求 spec-first 降低 spec-only 叙事

- 证据：X 查询出现 Spec-Driven Development 课程、Spec Kit、对抗 vibe coding 的讨论；GitHub Spec Kit/Kiro 官方 docs 也强化 specs/design/tasks。
- 判断：spec-driven 变热会让“有 spec 流程”迅速商品化。
- 优化：spec-first 叙事从 “spec-driven workflow” 升级为 “spec anchors intent; evidence, review and knowledge close the loop”。

### 第 59 轮：AGENTS.md 生态碎片化说明 source-governed projection 仍有价值

- 证据：OpenAI 官方 AGENTS.md guide、X 上大量“复制 AGENTS.md 模板”的传播、GitHub 搜索到个人 agent-skills/AGENTS 片段。
- 判断：规则复制会带来过期、冲突、不可验证。spec-first 的优势是 AGENTS/CLAUDE 只是 source-managed entry slice。
- 优化：新增 host instruction adapter matrix，记录每个 host 的 instruction surface、source slice、runtime projection、drift repair command、unsupported semantics。

### 第 60 轮：GitHub agent-skills 生态低星碎片化，说明不要急着做 marketplace

- 证据：`gh search repos "agent skills codex"` 返回多为 0-5 星个人 skill 集合。
- 判断：skills 分发生态还未稳定，spec-first 不应把维护重心转向 marketplace/catalog。
- 优化：坚持本地 source-controlled capability pack 方向；外部 skills 只作为 reviewable source artifact 引入，不做 install/search/publish 平台。

### 第 61 轮：MCP coding-agent 搜索显示“检索层”正在独立成类

- 证据：GitHub 搜索到 `ham-zax/satori` 这类“agent-safe code retrieval/runtime-first search/exact reads/call graph context”项目。
- 判断：spec-first 的 Graph/Context Harness 方向正确，但必须避免 provider lock-in。
- 优化：定义 codebase orientation palette：GitNexus、AST-grep、rg、language server、package/test topology 都是 lane；plan/work 选择 lane，不能让 GitNexus 成唯一入口。

### 第 62 轮：Anthropic context engineering 的 write/select/compress/isolate 可映射到 spec-first

- 证据：Anthropic context engineering 强调上下文写入、选择、压缩、隔离和 compaction/memory。
- 判断：spec-first 已有部分实现：write=artifacts，select=context-bundle/review-pre-facts，compress=artifact-summary，isolate=subagent boundaries。但缺少统一映射。
- 优化：在 AI Coding Harness contract 增加一张 context operation map，把每种 context 操作对应到现有 contract，避免为每篇文章新增一套术语。

### 第 63 轮：Kiro specs/steering/hooks 给出三层参照

- 证据：Kiro docs 暴露 specs(requirements/design/tasks)、steering(project knowledge/context)、hooks(agent automation trigger)。
- 判断：spec-first 有 specs/plans/tasks，有 docs/solutions 和 AGENTS/CLAUDE，但 hooks 选择更克制。
- 优化：明确“不做 per-turn hook”的原因：双宿主不等价、状态机风险、上下文噪声。采用 closeout breadcrumb 和 workflow invariant lens 替代。

### 第 64 轮：OTel GenAI 是未来对接层，不是当前内部模型

- 证据：OpenTelemetry GenAI semantic conventions 已定义 GenAI spans/operations 等观测术语。
- 判断：spec-first 不应内建 observability platform，但应避免 ledger 字段将来无法映射到 OTel。
- 优化：`workflow-outcome-ledger.v1` 字段设计保留 `trace_ref`、`span_kind_hint`、`genai_operation_hint`，但不要求当前实现发 OTel。

### 第 65 轮：METR 长任务度量提醒“时间跨度”不是“步骤数”

- 证据：METR long tasks 关注 AI 能完成的人类时间跨度任务，而不是简单 action count。
- 判断：spec-first 的 long task 判断不能只按文件数、轮次数、token 数。
- 优化：run artifact 增加 `human_time_horizon_estimate: short|medium|long|unknown` 的 LLM-owned field，帮助后续分析长任务失败模式，但不作为脚本 gate。

### 第 66 轮：current-state context 必须是 advisory cache，而非代码真相源

- 证据：Claude Round 1 建议 current-state context artifact，但也提醒代码仍是真相。
- 判断：这是高价值方向，也是高漂移风险方向。
- 优化：首版只做 `current-state-brief.md` 或 JSON summary，字段含 source paths、generated_at、source_revision、confidence、stale_after_hint；每条语义 claim 必须有 source path anchors。

### 第 67 轮：current-state 不应覆盖全仓，而应按 spec scope 生成

- 证据：context-bundle 合同强调最小充分上下文，不广播 full repo。
- 判断：全仓 current-state 会变成第二知识库，维护成本不可控。
- 优化：按需求/计划 scope 生成 scoped current-state brief，例如 `billing-current-state`、`runtime-delivery-current-state`，而不是全局 `PROJECT_CONTEXT.md`。

### 第 68 轮：Codebase→Graph→Current-state 需要三态关系

- 证据：当前 graph facts 是 definitions-only/stale/dirty-advisory；current-state 若基于 graph，也会继承限制。
- 判断：current-state 的 freshness 不能只看生成时间，还要看 graph/source 双层 freshness。
- 优化：current-state brief 写明 `source_reads_confirmed`、`graph_pointers_used`、`graph_freshness_at_generation`、`claims_requiring_reread`。

### 第 69 轮：Spec→Plan 语义消费应以 acceptance coverage 证明

- 证据：Claude Round 11 指出 spec_id 可形式正确但语义断裂。
- 判断：identity trace 是必要条件，不是充分条件。
- 优化：plan quality pass 增加轻量 “Acceptance Coverage” 段：requirements 的 actor/flow/acceptance example 对应 U-ID 或 non-goal。

### 第 70 轮：Plan→Tasks 双派生路径需要显式优先级

- 证据：work 可直接从 plan 建内部 tracker，也可消费 task pack。
- 判断：双派生路径不必消灭，但优先级必须明确。
- 优化：规则：validated task pack present -> work consumes task pack；no task pack -> internal tracker is ephemeral and not handoff authority。

### 第 71 轮：Task dependency confidence 是并行安全前置

- 证据：task pack wave/parallelizable 由 LLM 语义推断；parallel work 依赖其正确性。
- 判断：parallelizable=true 不应被当作脚本确认事实。
- 优化：task card 增加 `parallel_confidence: high|medium|low` 和 `basis: explicit-independent-files|shared-files|semantic-only`；low 不进入并行 subagent。

### 第 72 轮：Review deterministic merge 是最值得脚本化的 review 子问题

- 证据：spec-code-review 1100+ 行包含 dedup、confidence anchor、demotion、validator budget 等确定性/半确定性流程。
- 判断：不是要把 review 变脚本，而是把机械排序/合并从 LLM working memory 中移出。
- 优化：CUD-C 首版只做 `review merge --dry-run`，输入 reviewer JSON fixture，输出 merged candidates；语义同源判断仍可由 LLM 提供 `semantic_group_id`。

### 第 73 轮：Review artifact 应区分 full-fidelity temp 与 durable compact

- 证据：spec-code-review 已声明 full-detail JSON 走 temp，durable 只写 concise summary。
- 判断：这个边界正确，但第二轮应强调 downstream 不得依赖 temp artifact 长期存在。
- 优化：review closeout 强制给 `durable_summary_available: yes|no`；compound 只能消费 durable summary 或 source-confirmed finding。

### 第 74 轮：Graph evidence 消费需要“显式字段引用”

- 证据：Claude Round 26 指出 limitations 字段存在不等于被消费。
- 判断：这是真实执行缺口，适合用 eval 和 output contract 双重验证。
- 优化：当输出 graph-backed claim 时，必须写 `Graph fields consumed: query_ready=<>, impact_context=<>, freshness_state=<>`；没有字段引用则只能称 bounded source read。

### 第 75 轮：definitions-only 不应反复触发 graph-bootstrap

- 证据：当前 startup reminder 显示 query/context full、impact none、definitions-only limitations。
- 判断：definitions-only 在当前仓库是能力边界，不一定是可修故障。
- 优化：graph readiness vocabulary 增加 “capability-ceiling” 解释：bootstrap 可刷新 freshness，但不能凭刷新获得 provider 不支持的 impact/review 能力。

### 第 76 轮：Graph primary 升级应排在 P-friction 后

- 证据：用户体感路线图明确 P-graph-primary 最易被砍，需 P-friction 证明摩擦集中在 graph 不够强。
- 判断：不要因为架构上喜欢 graph 就优先升级。
- 优化：graph-primary CUD 的 `why_now` 必须引用 P-friction finding 或 review/debug repeated failure，不接受纯 roadmap-gap。

### 第 77 轮：docs lifecycle 是当前知识污染风险

- 证据：Claude Round 27 指出 docs 目录多、历史口径易被当 current；角色契约也禁止把历史 docs 当 runtime contract。
- 判断：随着审查文档增多，这个风险比以前更大。
- 优化：新增 docs lifecycle frontmatter 约定：`status: current-source|review-evidence|historical-input|superseded`。先从 `docs/项目审查`、`docs/业界分析`、`docs/plans` 开始，不全仓强制。

### 第 78 轮：审查报告本身需要“可消费摘要”

- 证据：本轮和 Claude 参考文档加起来超过 1000 行。
- 判断：长报告如果没有索引，会变成不可消费历史材料。
- 优化：在 result 文档增加 `CUD candidate index`，每个候选指向 source rounds，而不是让后续 planner 通读全部 100 轮。

### 第 79 轮：Prompt injection 防护应成为 invariant matrix 行

- 证据：Claude Round 32 指出 debug/issues/logs/PR comments/provider output 都是不受信外部内容。
- 判断：只靠 researcher/web 防护不够。
- 优化：invariant matrix 增加 `untrusted_content_is_data_not_instruction` 行；覆盖 brainstorm research、plan research、debug、review、compound、graph evidence consumers。

### 第 80 轮：Mutation boundary 需要跨 workflow 统一词表

- 证据：work、debug、review autofix、compound、clean、graph-bootstrap 都可能写文件或运行命令。
- 判断：各自描述 preview-first/safe_auto/silent write，词表不统一。
- 优化：定义 mutation gate taxonomy：`preview-only`、`safe-auto`、`gated-auto`、`manual-only`、`forbidden`，各 workflow 显式声明支持哪些。

### 第 81 轮：Invariant matrix 应优先覆盖 6 条，不要做大全

- 证据：角色契约已有核心哲学，Claude Round 29 提出 invariant matrix。
- 判断：过大的矩阵会变维护负担。
- 优化：首版只覆盖 `scope_authority`、`source_runtime_boundary`、`advisory_not_confirmed`、`preview_first_mutation`、`untrusted_content`、`knowledge_source_confirmation`。

### 第 82 轮：Knowledge retrieval 应先做命中记录，不先做智能检索

- 证据：spec-compound 写入已有，反向消费不可观测。
- 判断：不知道是否被用，就不该先优化检索算法。
- 优化：plan/work/debug 在 closeout 记录 `solutions_search: not-run|run-no-hit|run-hit`、`hit_paths[]`、`used_in_decision: yes|no|unknown`。

### 第 83 轮：docs/solutions 需要场景标签，而非只靠 category

- 证据：当前 solutions 目录按 architecture-patterns、workflow-issues 等分类；这对人类有用，但不一定匹配 agent 触发。
- 判断：agent 需要“什么时候读”，不只是“属于哪类”。
- 优化：solution frontmatter 增加 `applies_when`、`source_paths_pattern`、`workflow_consumers`，便于 bounded retrieval。

### 第 84 轮：高频 review finding 到 project standard 的路径需要轻量化

- 证据：Review→Knowledge 已能建议 compound，但 compound 后是否反哺 plan/work 未验证。
- 判断：知识闭环缺最后一步：lesson 成为后续默认检查。
- 优化：当同一 finding pattern 3+ 次出现，compound doc 可标 `candidate_standard: true`；是否进 AGENTS/contract 由人类/LLM review 决定，不自动修改 instruction。

### 第 85 轮：Changelog 已成事实 ledger，但需要摘要消费层

- 证据：CHANGELOG 条目非常长，承载大量设计背景。
- 判断：详细 ledger 有价值，但 planner/reviewer 需要短摘要。
- 优化：release notes 或 `changelog-summary` 生成按主题聚合的近期变更索引；CHANGELOG 保持原始事实，不承担导航。

### 第 86 轮：GitHub 搜索显示“context engineering handbook”类项目在增长

- 证据：`Prompthon-IO/agent-systems-handbook` 等项目把 context engineering、memory、evaluation、observability、multi-agent architecture 放在一起。
- 判断：spec-first 的六层 harness 与行业趋势一致，但需要更强的“边界裁判”叙事。
- 优化：对外文档增加一段：spec-first adopts context/eval/observability principles only as repo-local light contracts, not as a platform runtime。

### 第 87 轮：数据库 context enrichment 项目说明 domain-specific context 可能更有效

- 证据：`GoogleCloudPlatform/db-context-enrichment` 以数据库 schema 场景做结构化上下文编译/评估。
- 判断：全局通用 current-state 不如 domain-specific current-state。
- 优化：current-state brief 支持 domain lens：CLI/runtime、workflow docs、graph provider、review system、package/release，每个 lens 有不同 field。

### 第 88 轮：Host native custom agents 正在逼近 spec-first 的 agent/skill 层

- 证据：GitHub Copilot custom agents docs、OpenAI skills、Gemini CLI extensions/MCP、Kiro hooks 都在增强 host native capability。
- 判断：spec-first 不应和 host 比“谁有更多 agent”，应定位为 source-governed workflow contract layer。
- 优化：每个 agent/skill 变更都标注 complement/compete：是补 host 缺口，还是消费 host 能力，还是可能被 host 原生替代。

### 第 89 轮：Spec Kit Agents 的 context-grounding hook 对 spec-first 是提醒

- 证据：Spec Kit 生态已出现 agent/context grounding hooks，用来把实现回绑到 spec/context。
- 判断：spec-first 的 equivalent 不是 hook，而是 plan acceptance coverage + work source-read confirmation + review finding schema。
- 优化：将 `grounding` 定义为三段：spec grounding、source grounding、review grounding；不要新增 hook，先补 report fields。

### 第 90 轮：X 上 “leaderboard gap” 讨论支持 repo-local eval

- 证据：X eval 查询中出现“leaderboard 不测每天重复解释自家代码库”的观点。
- 判断：这与 spec-first 的痛点高度一致：不是模型会不会修公开 issue，而是能否在你的 repo 中少重读、少跑偏。
- 优化：P-friction 增加指标 `re-explanation_count`：用户/agent 在同一任务中重复输入背景的次数。

### 第 91 轮：AGENTS.md 规则沉淀需要防止规则膨胀

- 证据：X 上大量传播可复制 AGENTS.md 模板，甚至 200+ 文件集合。
- 判断：复制规则容易短期有效、长期膨胀。
- 优化：spec-first instruction update 必须走 `rule_scope: global|project|workflow|none` 判断；未验证 lesson 不进入 AGENTS/CLAUDE。

### 第 92 轮：Long-horizon resume 不等于 task state machine

- 证据：spec-work run artifact 已有 resume_evidence，Claude Round 35 提到 advisory 完成状态可辅助续跑。
- 判断：resume anchor 有必要，但不能变 active task pointer 真相源。
- 优化：run artifact 可以记录 `resume_suggestions[]`，但必须带 `authority=advisory`，恢复时仍以 diff/tests/source plan/task pack 校验。

### 第 93 轮：Review inline 是否值得做，需要先测 review friction

- 证据：用户体感路线图将 P-review-inline 置于 P-friction/P-task-pack-default 后。
- 判断：内联 review 可能降低摩擦，也可能拖慢小任务。
- 优化：P-friction 单独统计 `review_wait_cost`、`review_found_actionable_count`、`review_false_positive_count`，再决定 inline threshold。

### 第 94 轮：Task-pack default 不应从中大型任务一刀切开始

- 证据：路线图提到 task-pack default 依赖真实摩擦数据。
- 判断：task-pack 对复杂任务有价值，对小任务可能是流程税。
- 优化：用 P-friction 数据建立 `task_pack_recommended_when` heuristic，例如跨 3+ implementation units、review_gate required、parallelizable candidates、source_plan_hash needed。

### 第 95 轮：External research intake 应标准化 source quality

- 证据：self-reflection contract 已定义 external practice fields：source_type、freshness、evidence_quality、counter_signal、decision_use。
- 判断：本轮 GitHub/X 调研质量参差，必须纳入统一评级。
- 优化：所有外部趋势进入报告时标 `official_docs(primary)`、`github_repo(secondary)`、`x_discussion(weak)`，X 只可形成 watchlist，不可单独驱动 CUD accepted。

### 第 96 轮：第二轮 CUD 优先级应按“减少重复认知负担”排序

- 证据：current-state、graph consumption contract、task validation、review merge、knowledge retrieval 都在减少 LLM 重复判断。
- 判断：减少重复认知负担比新增能力更直接提升质量。
- 优化：优先顺序：graph consumption contract tightening、task validation single source、review merge dry-run、solutions hit telemetry、current-state scoped brief。

### 第 97 轮：先做“消费证明”，再做“能力增强”

- 证据：graph fields、run artifact、solutions、review summaries 都已有，但消费证据弱。
- 判断：增强能力前，先证明现有 artifacts 被下游读到并影响决策。
- 优化：每个 artifact 增加 consumption evidence：`read_by_workflow`、`used_for_decision`、`ignored_reason`。先从 work/review/compound closeout prose 做起。

### 第 98 轮：报告需要把 anti-pattern 也结构化

- 证据：角色契约已有禁止项，但报告建议中 anti-pattern 分散。
- 判断：后续 planner 很容易只读建议、不读反向边界。
- 优化：result 文档为每个 CUD 写 `anti_patterns`：如 current-state 不得变全局真相源、ledger 不得变 KPI、review merge 不得替代语义审查。

### 第 99 轮：第二轮结论应推动一个“窄计划”，不是五个并行计划

- 证据：建议数量多，且多个方向互相依赖。
- 判断：并行推进会造成治理面膨胀。
- 优化：下一步只建议起草一个 plan：`Evaluation + Consumption Evidence MVP`，包含 graph field consumed、solutions hit telemetry、workflow outcome ledger minimal fields。current-state 和 review merge 作为后续 CUD。

### 第 100 轮：第二轮最终收敛

- 证据：第一轮指出方向；Claude prior report 深化到 40 轮；本轮 GitHub/X/official docs 进一步证明行业正在向 context engineering、spec-driven workflows、repo-local eval、host-native instructions 聚拢。
- 判断：spec-first 的突破点不是追逐所有趋势，而是用 light contracts 把“上下文、证据、评估、知识”变成 repo-local 可消费事实。
- 优化：立即收敛到三件事：消费证明、Evaluation MVP、invariant matrix。其余 current-state、review merge、knowledge index 作为 CUD backlog，等待 P-friction 和 eval 证据排序。

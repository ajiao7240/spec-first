# spec-first 项目 Review 与优化方案（2026-06-15）

## 1. Executive Summary

**一句话判断:spec-first 已是「AI Coding Harness」品类里能力建设最完整、边界纪律最严、证据/知识闭环最诚实的项目之一——它的问题不在「不够强」,而在「太强但没说清,且没被外面看见」。**

把 16 路并行调研 + 内部 06-14 达成度报告交叉印证后,三层判断如下:

### 总体判断

- **架构与哲学:行业领先,且踩在趋势正中央。** 「Scripts prepare facts, LLM decides」+「Light contract / Explicit boundaries」与 2025-2026 Anthropic Agent Skills 指南、OpenAI Codex best-practices、Martin Fowler 的 SDD 评估**1:1 收敛**。市场正从「prose-spec」转向「machine-checkable evidence」,而这恰是 spec-first 的 deterministic-facts-vs-LLM-judgment 边界——**没有任何对标项目把这条边界讲得这么干净**。
- **工程落地:六层 harness 五层结构健康**(Context/Execution/Evidence/Governance/Knowledge 契约面 + Evaluation 机制已就位待数据),37 source skill / 51 agent / 154 测试文件 / 双宿主同源生成 / 契约化 schema 体系——这是 source-code 级成熟度,远超绝大多数对标仓的「prompt 集合 + 几个脚本」。
- **传播与采纳:这是最大短板,且与能力严重不匹配。** 严密的内部治理对决策者完全不可见;无 demo repo、无企业采纳内容、无效果指标、无外部集成;定位在四个真相源里有四种措辞;CHANGELOG 对外部消费者不可读;bus factor = 1。

### 优势 / 短板 / 机会

| 维度 | 判断 |
| --- | --- |
| 最大优势 | deterministic-facts/LLM-judgment 边界 = 正在到来的「Verified SDD」趋势的最佳载体;Review→Knowledge 闭环是对标仓普遍缺失的节点 |
| 最大短板 | 「能力强但不可见 / 不可试 / 不可被决策者评估」——采纳漏斗第一格(试用)和最后一格(企业决策)都缺 |
| 最大机会 | 市场刚进入「反 ceremony + 重 verification + 焦虑 staleness」的批判期,spec-first 的 task 分级(小/中/大)+ 证据闭环正好命中三大痛点,但**当前未作为头部卖点** |
| 最大风险 | 与 OpenSpec 处于同一「lighter-than-Spec-Kit」赛道却没说清差异;宿主 primitive 正在商品化(Codex/Claude 都有了 subagent/review/skills),「双宿主」本身的护城河在变薄 |

### 优先方向（6 个月目标）

把项目从「能力完整的内部工程范本」推进到「外部可试、可评估、可采纳的开源 harness」。**不新增核心能力,优先把已有能力变得可见、可试、可度量、可治理性外显。**

**对标提炼(跨 11 仓的共性结论):**

1. **多宿主可移植性是 spec-first 唯一系统性落后的硬能力。** 几乎每个活跃对标仓都支持 7-30+ host(spec-kit IntegrationBase、OpenSpec adapter、Trellis bracket marker、BMAD template、compound converter、planning-with-files SKILL.md)。spec-first 锁定 Claude+Codex。但——见 §5 战略取舍——这未必要全做。
2. **三个 mechanism 被对标仓反复证明有效、且 spec-first 最该补:** (a) **per-turn / hook-based 状态注入**(Trellis/pro-workflow/planning-with-files 都有,抗 context 压缩,spec-first 完全没有);(b) **artifact preflight gate**(flow-x/OpenSpec,程序化防跳阶段);(c) **anti-rationalization 表**(superpowers,最低成本反 LLM 偷懒)。
3. **spec-first 的独有优势在对标后更清晰:** deterministic-facts/LLM-judgment 边界的纯粹度、honest-closeout 的诚实降级文化、Review→Knowledge 闭环的契约化——这三点**没有任一对标仓同时具备**。scale-engine 的进化引擎更自动,但牺牲了边界清晰;BMAD 的 retrospective 更深,但靠 LLM 自查无确定性证据。

## 4. 核心优势

1. **定位哲学踩中趋势正中央。** 「Scripts prepare facts, LLM decides」与 Anthropic「bundle code for deterministic operations」、OpenAI Codex「deterministic read-only review」、市场「Verified SDD」趋势 1:1 吻合。这是**可防御的、可拥有的差异化**:市场正从信任 generated Markdown 转向信任 machine-checkable facts。
2. **架构边界纪律是 source-code 级,非 prompt 级。** source/runtime 边界 + 双宿主同源生成 + 契约化 schema(verification/context-bundle/project-graph/honest-closeout)+ 154 测试文件。对标仓里只有 scale-engine 和 spec-kit 接近这个工程度,但都没有 spec-first 的边界清晰度。
3. **Workflow 闭环覆盖最完整。** Codebase→Spec→Plan→Tasks→Work/Debug/Review→Evidence→Knowledge 全节点有公开 workflow;**Review 和 Knowledge 这两个节点是绝大多数对标仓的盲区**(spec-kit/Kiro/OpenSpec/rapid 都停在 implement)。
4. **双宿主同源生成是真·工程方案。** 一份 source 投射 Claude `/spec:*` + Codex `$spec-*`,init 重生成消除 drift——比「往每个工具塞 prompt 文件」干净。
5. **Evidence/Knowledge 的诚实文化独一份。** honest-closeout 主动把 unsupported claim 降级而非标 verified;coverage 诚实标 aspirational;06-14 报告连「自指闭环失守」都白纸黑字写出来。这种**诚实降级文化**是信任的根基,对标仓普遍是「LLM 自查即通过」。
6. **中文生态 + 工程严谨度的组合在中文 AI coding 社区稀缺。** 对标的中文仓(myclaude/yunshu/chen)多为 skill 堆叠;spec-first 的契约化工程度在中文生态里是降维优势。

| P2 | bug→knowledge 自动化、artifact preflight gate 缺失 | 对标仓已证明有效的低成本 mechanism 未享受 | 80/20 取舍未排到 | 中期:artifact preflight gate + anti-rationalization 表 + shadow-rule maturity(择优) |
| P3 | bus factor = 1 | 企业采纳的长期性担忧 | 单维护者 | 中长期:公开 RFC 流程 + 降低贡献门槛 |

## 6. 优化方案

> 原则:不新增核心能力,先让已有能力**可见、可试、可度量、可治理性外显**;新增 mechanism 严格走 80/20,优先对标已证明有效项。

### 短期（0-6 周）— 让能力可见可试

1. **Demo repo(P0)**:独立仓预置完整 workflow 产物链,配 1 张产物截图替换 README 当前 SVG demo slot。价值高/成本中/风险低。负责人:开源负责人。
2. **定位收口(P1)**:选定 canonical 一句话(见 §9),同步 README en/zh + package.json + website hero + CONCEPTS,harness 身份上 hero。价值高/成本低/风险低(走中型任务,影响用户可见 + 双宿主同源,需同步验证)。
3. **GitHub Releases + 精简 release notes(P1)**:每个 tag 发面向用户的版本价值摘要。价值中/成本低/风险低。
4. **OSS 卫生补齐(P2)**:CODE_OF_CONDUCT + .github/ISSUE_TEMPLATE + CODEOWNERS + SECURITY 私密渠道 + root ROADMAP.md。价值中/成本低/风险低。
5. **CLI 文档补口(P2)**:README CLI reference 补 `repair-worktree`/`session` 或标 advanced。价值低/成本低。

### 中期（2-4 月）— 补证据/上下文/对标 mechanism

1. **语义 eval 最小回归(P1)**:用已有 eval fixture 搭 LLM-as-judge,先覆盖 spec-plan/code-review/debug/compound 4 个 first-wave workflow,接入 ai-dev-quality-gate CI。价值高/成本高/风险中(新增 eval 执行基础设施,需防过度设计)。
2. **per-turn 状态注入 hook(P1)**:UserPromptSubmit hook 读 task 状态注入 breadcrumb,抗 context 压缩。对标 Trellis/pro-workflow。价值高/成本中/风险中(双宿主 hook 支持差异需验证)。
3. **artifact preflight gate(P2)**:进入 plan/work 前程序化校验上游 artifact 存在,缺失则路由到前置 phase。对标 flow-x/OpenSpec。**这是角色契约 §4「handoff gate」的确定性强化,方向自洽**。价值中/成本中。
4. **anti-rationalization 表(P2)**:在 work/debug/review prompt 加 Red Flags / Common Rationalizations 表(spec-first CLAUDE.md 已有红旗雏形,可系统化)。对标 superpowers。价值中/成本低/风险低。
5. **多仓集成测试 + degraded-mode 仿真(P2)**:把 context/quarantine 契约从 advisory 推向有测试守护。价值中/成本中。
6. **standards-native 显式化(P2)**:显式声明并测试 AGENTS.md/SKILL.md/MCP 兼容,把「双宿主」叙事升级为「standards-native harness」。价值高/成本中(对冲宿主商品化风险)。

### 长期（4-12 月）— 企业护城河 + 知识进化

1. **企业采纳叙事 + 治理价值外显(P0 级价值)**:团队采纳指南、治理/审计价值页、1 个高治理(金融/信贷/证券)落地案例、效能指标框架(debug 命中率/review 漏判率——angle 已在角色契约标 aspirational)。
2. **bug→knowledge / shadow-rule maturity(择优)**:compound 捕获→候选规则 shadow N 次→达阈值晋升。对标 scale-engine,但**保持 spec-first 的边界清晰,不照搬其自动化激进度**。
3. **delta-spec living corpus(评估)**:对标 OpenSpec,回答 Fowler 的 staleness 拷问——但需评估是否与「artifact 是证据非 source」哲学冲突,**可能更适合定位为 spec-anchored 而非引入 spec-as-source**。
4. **AI review evidence 外发 + eval 平台对接(评估)**:让 review/verification 证据 machine-readable,可喂 Braintrust/LangSmith 式 eval 与企业审计。定位为「workflow-level evidence layer」——介于 in-IDE agent 与 PR-level review/eval 之间的无主地带。
5. **效能指标 / VibOps**:把 Evaluation harness 从「机制就位」推到「数据积累」,产出可对决策者展示的效果度量。

## 8. 不做什么

严格遵循角色契约 §10 与系统边界,**以下方向明确不做或仅 opt-in**:

- **不做泛 prompt collection / agent marketplace。** spec-first 是 workflow harness,不是 prompt/agent 堆。
- **不做强状态机 / 中心化流程引擎。** 对标 spec-kit 的 YAML workflow engine 虽强,但与「强制不变量、不规定路径」冲突——只借鉴**可恢复 run state**(handoff gate 的确定性强化),不引入刚性状态图。
- **不替代 Claude / Codex / Gemini。** 不做独立 IDE(Kiro 的 lock-in 是反面);不做独立 agent(gsd v2 路线)。
- **不无边界堆脚本。** 新增 mechanism 必须服务核心链路节点,过对标不等于全抄。
- **不硬编码 LLM 判断。** scale-engine 的自动进化引擎、BMAD 的 persona 自查再强,也不把语义裁决脚本化。
- **不把 generated runtime 当 source。** 多宿主扩展若做,走 source 投射 + init 重生成,不手改 mirror。
- **不盲目追多宿主数量。** 对标仓 7-30+ host 看似领先,但宿主 primitive 正商品化;转向 standards-native(AGENTS.md/SKILL.md/MCP)比硬编码 N 个 adapter 更省更稳。
- **不出文档量去卷 Spec Kit。** 市场已把 verbose markdown 视为毒药;不引入强制 phase gate 的 ceremony。
- **不claim specs-as-source / 人不碰代码。** 那是 HN 社区嘲讽的 waterfall 触发点;spec-first 定位 spec-anchored,artifact 是证据不是 source。

**3 个差异化标签:** ① Scripts-prepare-facts / LLM-decides 的 verifiable evidence;② Review→Knowledge 闭环(对标普遍缺失);③ honest-closeout 诚实降级文化。

**3 个失焦风险方向:** ① 追多宿主数量而非 standards-native;② 引入强状态机/重 ceremony;② 把自动化做到牺牲边界清晰(scale-engine 化)。

**6 个月 5 个里程碑:** 见 §7(M1 可试用 / M2 可信证据 / M3 可治理外显 / M4 企业可评估 / M5 知识进化)。

## 9. 推荐文案

> 基于 §3 趋势分析(反 ceremony + 重 verification + 焦虑 staleness + tool-neutral)与 spec-first 实际能力。harness 身份上 hero,verification 边界作头部卖点。

**一句话定位（canonical,建议全 surface 统一）:**
> spec-first —— 把一次性 AI coding 对话,变成可治理、可验证、可复用的工程闭环。脚本产出可信事实,LLM 做判断,证据留在你的仓库里。

**30 秒介绍:**
> AI 写代码很快,但塑造代码的判断随对话窗口消失:下一次会话没上下文,reviewer 看不到计划为什么变,团队复用不了成功经验。spec-first 是面向 Claude Code 与 Codex 的 AI Coding Harness——它把需求、计划、任务、评审、经验作为持久 artifact 留在仓库里。脚本只产确定性事实(git 状态、schema 校验、readiness),LLM 只做语义判断。从一行修复到一次迁移,workflow 按任务大小伸缩,不给小任务套 ceremony。

**README 首屏(替换/强化当前 hero):**
> **把 AI Coding 从临时对话,升级为可验证的工程闭环。**
> spec-first 是面向 Claude Code 与 Codex 的 AI Coding Harness。不让你信任生成的 Markdown——脚本产出可验证事实,LLM 在事实上做判断,需求/计划/评审/经验作为持久 artifact 留在仓库。一份 source,两个宿主,无 drift。

**官网首屏 eyebrow + title(微调现有,加 harness + verifiable):**
> eyebrow: `AI CODING HARNESS · CLI`
> title: 把 AI Coding 从临时对话,升级为**可验证的工程化工作流**
> subtitle: 脚本产可信事实 · LLM 做判断 · 需求写入仓库 · 评审有证据 · 经验可复用

**面向技术负责人(Tech Lead):**
> 你的团队已经在用 Claude Code / Codex,但 AI 改动的决策、证据、评审结论散落在聊天里,无法审计、无法复用、无法治理。spec-first 给每次 AI coding 留下可追踪的需求、计划、评审证据和经验沉淀——source/runtime 边界清晰,证据可验证,谁决定了什么、验证了什么、留下什么,都在仓库里。

**面向开发者:**
> 不用再把上下文反复粘进新会话。spec-first 把你和 AI 的工程判断写回仓库,下一次会话、reviewer、同事直接继承。小任务不套流程,大任务有结构。`npm i -g spec-first && spec-first init`,在你已有的项目里就能跑。

**面向老板/决策者:**
> AI coding 的风险不是写不出代码,是没人能证明改动对不对、可不可复用、能不能治理。spec-first 把 AI 研发变成可治理、可验证、可审计的工程闭环——降低返工,沉淀团队资产,为规模化 AI 研发提供证据基线。

## 10. 最终结论

**当前阶段:** spec-first 处于「能力成熟、表达滞后」的关键拐点。六层 harness 五层健康、契约化工程度 source-code 级、定位哲学踩中行业趋势正中央——这是**能力已就位**。但它对外不可试、对决策者不可见、定位措辞不统一、最强差异化(verification 边界)不在头部——这是**表达与采纳严重滞后于能力**。

**是否值得投入:** 值得,且时机正好。市场刚进入 SDD 的批判性重估期(反 ceremony、重 verification、焦虑 staleness),spec-first 的 task 分级 + deterministic-facts 边界 + Review→Knowledge 闭环恰好命中三大痛点,而 OpenSpec 等竞品尚未占住「verifiable evidence」心智。窗口期内把表达和采纳补上,能从「内部工程范本」变成「品类标杆」。

**需补充/优化能力(按 ROI 排序):** ① demo repo + 定位收口(让能力可见可试);② 语义 eval 自动执行(让 Evaluation 层真正启动);③ per-turn hook + artifact preflight gate(对标已证明有效的低成本 mechanism);④ 企业采纳叙事 + 治理外显(打开决策者漏斗);⑤ standards-native(对冲宿主商品化)。

**最大机会:** 占住「Verified SDD / AI Coding Harness」心智——把「Scripts prepare facts, LLM decides」从内部哲学变成市场可识别的差异化标签。这是无竞品占据、且与趋势完全吻合的高地。

**最大风险:** 表达滞后导致窗口错过——OpenSpec 在同赛道更激进地做传播,宿主 primitive 商品化侵蚀「双宿主」叙事。能力强但没人知道、没人能试,是最现实的失败模式。

**下一步 3 件事:**
1. 建 demo repo,配一张完整产物链截图替换 README demo slot——打开试用漏斗第一格。
2. 收口一句话定位(harness 身份上 hero,verification 作头部卖点),同步 4 个真相源。
3. 发 GitHub Releases 精简 release notes + 补 OSS 卫生缺项——低成本拉高外部可信度。

# 2026-06-14 spec-first 达成度与闭环审查报告

## 0. 结论先行

**总体判断:spec-first 作为 AI Coding Harness 的「能力建设」已是同类范本级——六层 harness 中五层(Context/Execution/Evidence/Evaluation/Governance/Knowledge 的契约面)结构健康、边界清晰、诚实降级文化贯穿。Evaluation 层已从「唯一空层」转为「机制就位、待数据积累」,这是 6 月最大的正向进展。但本次审查暴露一个比任何单点 finding 更重要的元结论:这个把「可治理、可验证、可复用、可沉淀」作为使命的 harness,对用户项目执行了严密的「审查→整改」闭环纪律,却对自身资产的同一闭环失守——审查发现可以正确产出、却无法保证被消费。**

核心证据(经对抗验证):

- **06-10 综合审查报告的 12 个 P1,无任何 plan 以其为 `origin` 承接。** 对照之下,05-07 的两份审查报告都各有 plan 以其为 origin。06-10 是唯一一份高置信审查在 origin 链上「凭空消失」的。(对抗验证下修:存在 06-10→06-12→plans 的**间接语义承接链**,故非"全部静默搁置",但 12 个 P1 的逐条闭环状态**不可机器追踪**这一缺口属实且为 P1 级。)
- **12 个 P1 实测裁决:3 RESOLVED / 2 PARTIAL / 6 OPEN / 1 REGRESSED。** 即四天后仍有 **6 项 OPEN + 1 项不降反增(P1-1 体积)**——半数以上从未真正落盘。
- 这精确复现了 06-10 报告自己点名的最高级别元风险(「最高置信决策被静默搁置」),并坐实了 Evaluation Harness 缺位的**实际代价**:plan 可以标 `status: completed`,而它根本不负责的 P1 照样留在 source 里,无任何机制报警。

> **一句话:本次审查最该守护的不是"又发现 N 个新问题",而是"审查→整改闭环本身需要一个最小确定性追踪机制"——这正落在角色契约 §4「knowledge promotion gate / verification gate 在缺 runtime 支撑时降级为响亮约定」那条上。**

裁决分布:**P0 = 0**(无链路断点、无安全漏洞、无数据破坏);**元结论级 = 1**(闭环 origin 追踪断点);**P1 残留 = 6**;**P2 = 7**;**P3 = 一批治理卫生**。对抗验证 5 项全部确认成立(0 证伪),仅严重度/框定有校准。

## 2. 最重要结论:审查→整改闭环的 origin 追踪断点(元结论级)

**印证来源:A1-06-10 路 + 对抗验证 + 主线程独立复核。**

### 2.1 事实链(全部 confirmed)

1. `grep -rln "2026-06-10-全项目综合审查" docs/plans/` = **零命中**。没有任何 plan 以 06-10 综合报告为 `origin`。
2. 对照组:`2026-05-07-001` plan origin = 05-07 skill-agent review;`2026-05-08-001` plan origin = 05-07 source-code review。**05-07 的审查进了 plan,06-10 的没有。**
3. 06-14-001 plan(`status: completed`)的 origin 是 **06-12 agent-native 报告**,不是 06-10。所以「plan completed 但 P1 还在」不是回归——是这个 plan 根本不负责 06-10 的 P1。
4. 06-10 报告第 9 章白纸黑字要求「每条 P1 走 `/spec:plan` → `/spec:work`,每批一个窄 plan」。该路径**从未对 06-10 的 P1 集合启动**。

### 2.2 对抗验证的校准

证伪 verifier 确认两条核心事实站得住,但把原 finding 的「12 P1 整体闭环断裂/从未消费」框定**下修为 PARTIAL**:06-12 报告 :307 显式声明 06-10 为「前序」并至少承接了 P1-10(legacy e2e 退役)。即存在 **06-10 → 06-12 → plans** 的间接语义承接链。

**但这个间接链恰恰是问题本身,不是反驳:** 因为承接是「人脑记得 06-12 引用过 06-10」,而非 origin 字段可机器追踪。结果就是 12 个 P1 中——哪些经 06-12 顺带承接、哪些静默漏掉——**没有任何确定性事实可查**。这正是 Evaluation Harness 缺位的定义性症状。

### 2.3 为什么这是元结论而非普通 finding

角色契约把 spec-first 的使命定义为「让 AI coding 走向**可治理、可验证、可复用、可沉淀**的工程闭环」。审查 artifact 是 Knowledge/Evaluation 层的核心产物。**一个连自己的审查 artifact 都无法追踪是否被消费的 harness,在自指层面尚未兑现它对用户承诺的闭环。** 06-10 报告自己把这列为「本次审查最重要的元结论」,四天后它**原样复现**——这本身就是最强的证据。

### 4.1 跨层共性:prose gate 在缺机器约束时反复失守

三层在不同资产域独立指向同一模式——**「契约写了规则,但无确定性 test 校验规则是否被执行」**:

- **Knowledge**:promotion gate 是 prose,validate-frontmatter.py 只做 YAML parser-safety,结果连续两篇新 promote 跳过必填字段无 CI 信号(KH-promote-gate-1 + KH-contract-test-5)。
- **Evidence**:workflow_integrated=true 是给 LLM 的 prose 指令,test 只验"payload 提交时写入正确",不验"closeout 是否真调了 producer"(EVAL-gap-1)。
- **闭环 origin**:plan 必须回链审查 P1 是 prose 约定,无 test 守护 origin 完整性(META-closure-break)。

这与角色契约 §4 完全自洽——这些都属「verification/handoff/knowledge-promotion gate 在缺 runtime 支撑时降级为响亮约定」。**问题不在于"没硬 gate"(那是有意取舍),而在于"响亮约定"在实践中变成了"静默约定":没有 loud convention 的显式声明,也没有最小事实记录证明它被遵守。**

## 7. 建议优先级与最小落地顺序

遵循 80/20 与「先证明再扩展」,**每批一个窄 plan,不并行铺开,全部走 `/spec:plan` → `/spec:work`**。

### 第一批(元结论补账,docs-only,最高治理收益/最低成本)
1. **给 06-10 报告 12 个 P1 补一份显式闭环裁决文档**:逐项标 RESOLVED/继续/裁剪,并交代承接去向(本报告第 3 章已提供裁决底稿)——消除「不可追踪」状态本身。
2. **修最小闭环追踪机制**:在 plan frontmatter 约定 `origin` 可多值或加 `addresses_findings` 字段 + 一条 contract test 断言「引用审查报告的 plan 必须保留可追踪 finding id」。这是把「静默约定」升级为「响亮约定」的最小确定性机制,直接回应元结论。

### 第二批(最高风险残留,小)
3. **figma agent 脱敏(P1-9)**:移除 max-w-screen-xl/w-full/ERB wrapper 硬编码与 "Yes, I did it." 口令,或拆为 read-only diagnosis;补一条 contract test 断言 agent prose 不含来源项目私有字面规范。这是唯一会主动造成错误改动的资产。

### 第三批(source 卫生,中,可批量)
4. 虚挂依赖移除(P1-11,`npm uninstall ignore simple-git` 级);`.context/` 陈旧路径清理 + validate-frontmatter.py 收敛为单一来源(P1-4)。
5. 孤儿/悬空/抢触发 skill 处置(P1-6/7/8):删除真孤儿、host-provided 补标注、internal description 改写为非抢触发。
6. init apply 摘要中文化 + `-y` 输出路径清单(P1-12);workflow-skill-agent-map 补三映射 + contract test(P1-2)。

### 第四批(prose gate 加固,中)
7. **Knowledge promotion gate 机器化(P2)**:把 validate-frontmatter.py 扩展为校验 new_promote_required_fields,或补一条 test 读 docs/solutions/** 断言新 promote 合规——堵住「prose gate 静默失守」。

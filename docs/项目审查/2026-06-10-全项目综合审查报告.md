# 2026-06-10 spec-first 全项目综合审查报告

## 0. 结论先行

**总体判断：spec-first 是一个边界意识极强、治理纪律远超同类的成熟 AI Coding Harness，链路无 P0 断点；当前最大的风险不是能力缺失，而是三类「自身哲学未自指执行」的债务——上下文经济学失控（SKILL.md 体积/双重注入）、source 边界积灰（孤儿资产/拷贝漂移/导航图失真）、历史最高置信决策（P0/P1 消费证明）被静默搁置。**

- 统一裁决：**P0 = 0 项**（无链路断点、无安全漏洞、无数据破坏风险）；**P1 = 12 项**（合并跨路重复后）；**P2 = 15 项左右**（择要列出）。
- 各路评分：链路 4/5、核心 skill 无 P0（27~21 分）、辅助 skill「runtime 边界守住、source 边界积灰」、agent 资产 3.5/5、上下文工程 4/5、工程质量 4/5、init UX 成熟度高、业界对标为 advisory 级。
- 与 2026-05-30 双链 150 轮审查对照：当时收敛的 7 个优先级（P0-P6）中，**2 个未落地（P1 Outcome Ledger、P2 Friction Audit）、1 个承载 plan 被删仅部分承接（P0 Consumption Evidence）**，构成本次审查识别的最大治理风险。

## 2. 跨路交叉印证的高置信发现（总裁决核心）

多路独立评审指向同一处的发现，置信度最高，列为本次审查的核心结论。

### 发现①：SKILL.md 体积失控 + 渐进披露机制失效（全场最高置信）

**印证来源：R4 + R2a + R6 + 官方文档一手锚点。**

- R4：`skills/spec-code-review/SKILL.md` 114KB ≈ 30k token，单次加载即吃掉相当比例的有效上下文窗口；37 个 SKILL.md 合计 11,524 行。
- R2a：spec-code-review 通过 `@include` 把 5 个 references 文件强制内联进主体，等于主动放弃了 references 目录本应提供的渐进披露（按需加载）能力。
- 官方一手锚点：Claude Code 官方文档明确「skill 加载后内容跨轮驻留上下文，每一行都是 recurring token cost」。
- R6（advisory）：业界通行做法是给 SKILL.md 体积加 lint 上限，超限内容下沉 references 并按需读取。

**裁决：P1。** 这与角色契约「Context Harness：给 AI 正确上下文，不给无限上下文」直接冲突——harness 对用户项目执行上下文纪律，却对自身 skill 资产豁免。

### 发现②：`docs/workflow-skill-agent-map.md` 双重漂移（治理导航 source 失效）

**印证来源：R1 + R3。**

- R1：全景图缺 3 个 workflow 映射（spec-compound-refresh、spec-release-notes、spec-polish-beta），导航不完整。
- R3：该文件 :110 备注声称「11 个 agent 未被任何 workflow dispatch」，但其中 8 个实际被 deepening-workflow 调用；且与同文件 :29/:33 的表述自相矛盾。

**裁决：P1。** 这是 source-of-truth 文档自身失真——按角色契约，导航/治理文档失真比缺失更危险，因为下游会基于错误事实做裁剪决策（例如把仍在使用的 agent 当孤儿删除）。

### 发现③：bootstrap 双重注入（每会话 ~1100–1200 token 纯浪费）

**印证来源：R4 + R6。**

CLAUDE.md managed block 与 SessionStart hook 注入了同一份 bootstrap 内容，每个会话重复一次，约 1100–1200 token；源码注释自认了这一重复。R6（advisory）佐证业界注入机制均做去重。

**裁决：P1。** 单点修复收益直接（每会话固定节省），且与发现①同属「上下文经济学自指失守」。

### 发现④：拷贝式复用漂移是系统性模式（非孤立事件）

**印证来源：R2a + R3 + R2b，三路在不同资产域独立发现同一模式。**

- R2a：spec-optimize 引用已不存在的 `.context/` 路径（陈旧拷贝）；`validate-frontmatter.py` 在多个 skill 目录间 md5 一致的整文件拷贝；治理样板 8 段在多个 SKILL.md 间逐字重复。
- R3：spec-best-practices-researcher 硬编码的 skill 名称映射已与现实漂移。
- R2b：swiss-design 存在死引用。

**裁决：P1（作为模式）。** 单点都是 P2，但三路印证说明仓库缺「单一来源 + 引用」的 prose 复用机制，拷贝即埋下漂移种子，这与「多真相源必须重构」的角色契约判断矩阵直接冲突。

### 发现⑤：Evaluation Harness 自指缺位 + 历史最高置信决策静默搁置

**印证来源：R1 + 主线程历史核对（详见第 4 章）。**

2026-05-30 双链 150 轮审查的最终收敛点是「先证明 artifacts 被消费，再证明消费改善结果，最后才扩展能力」（P0/P1）。本次核对发现：P1 Workflow Outcome Ledger 完全未落地；P2 Friction Audit 完全未动；P0 的承载 plan（2026-05-30-002）已于 2026-06-08 删除，删除理由只针对 graph 消费面，未逐项交代 solutions hit telemetry、run artifact consumption 两类证据的承接去向。

**裁决：P1（治理层面是本次审查最严重的发现）。** 六层 Harness 中 Evaluation 仍是唯一空层，而 5 月 30 日之后的多个里程碑（governance lens、knowledge harness、runtime setup）都是能力扩展——正是历史审查明令的反序。

### 两大治理风险（本次审查最重要的元结论）

1. **双链 150 轮最高置信收敛点（P0/P1）被静默搁置。** 当时的结论原文是「先证明现有闭环真的提升交付质量，再扩展能力」，且明确「能力增强必须由 P-friction 实证驱动，不接受纯架构偏好」。此后五个月的实际投资（governance lens、knowledge harness、provider-native runtime setup、capability-aware provider）全部是能力/治理扩展。每一项单独看都有合理 plan 与评审背书，但合起来正是历史审查警告的反序。搁置本身可以是合理决策，**问题在于「静默」**——没有任何文档记录「我们决定推迟 P0/P1，理由是 X」，删除承载 plan 时也未逐项交代三类证据的承接去向。
2. **P2 Friction Audit 未动 = 能力投资排序至今仍靠架构判断而非摩擦数据。** 这使得所有后续优先级争论（包括本报告的建议）都缺真实用户数据校准，回到了历史审查试图打破的循环。

## 6. 专章：上下文工程（R4）

总评 4/5。spec-first 对用户项目的上下文纪律是业界一流（context bundle、excluded_context+reason_code、summary-first handoff、bounded direct reads），但对**自身资产**的上下文经济学失守：

1. bootstrap 双重注入（P1-3）：CLAUDE.md managed block 与 SessionStart hook 内容重复，每会话 ~1100–1200 token。
2. SKILL.md 体积（P1-1）：spec-code-review 114KB≈30k token；@include 内联机制使 references 失去渐进披露价值。skill 一旦加载即跨轮驻留，体积成本是每轮复利。
3. 亮点：context-bundle.v1 的 budget/budget_used 语义、knowledge-harness 的 expand-on-trigger 设计（语义触发而非行数阈值）是同类项目中少见的精细设计。

建议方向：① 注入去重（hook 检测 managed block 已在场则降级为短指针）；② SKILL.md 体积 lint（警戒线建议 500 行/30KB，超限内容下沉 references 且禁止 @include 全量内联回来）；③ 把「上下文经济学」补进设计判断矩阵作为显式问题。

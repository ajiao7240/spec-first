# AI辅助工作框架设计文档

## 文档信息

- **文档名称**：AI辅助工作框架（Spec-First Framework）设计文档
- **创建日期**：2026-03-29
- **版本**：v1.0
- **状态**：待审阅

## 1. 设计概述

### 1.1 设计目标

基于需求文档《AI辅助工作框架需求文档》，本设计旨在构建一套围绕 Claude Code CLI 的轻量级工作框架，通过五步闭环（brainstorm → plan → work → review → compound）推动软件交付从脆弱、手工、割裂走向智能、自动、可控。

一期聚焦 Claude 工具支撑，覆盖产品、研发（含全栈）、测试三类角色，建立最小可控的状态管理能力，形成可复制的落地样板。

### 1.2 核心设计原则

- **方法论先行**：优先统一协作方式，而非堆叠工具
- **一期收敛**：仅聚焦 Claude Code CLI，不做多模型协同
- **轻量可控**：建立最小状态管理，避免过度工程化
- **契约串联**：通过输入输出契约串联各阶段，而非中心化状态机
- **阶段自治**：每个阶段自己守门、自己产出，下一阶段只消费上游确认过的结果
- **人机协同**：AI 增强效率，关键决策保留人工把关

### 1.3 技术方案选型

**采用 A-Plus 方案**：轻量级 Skills 集合 + 最小元数据占位

核心思路是不构建完整的工作流引擎，避免过度工程化；但从首期开始建立最小状态管理能力，通过输入输出契约串联各阶段，而非依赖中心化状态机。

**技术选型**：
- 基于 Claude Code CLI 实现
- 使用 skills 作为核心交互方式
- 使用 steering 作为角色模式和场景约束的辅助承载方式
- 使用文件系统组织产物（`.claude/tasks/`）
- 使用 frontmatter + markdown 混合格式存储结构化信息
- 使用 `task.yaml` 承担最小控制面职责

## 2. 目录结构与产物组织

### 2.1 运行态目录结构

```
.claude/
  └── tasks/
      └── <task-id>/
          ├── task.yaml           # 任务元数据（控制面）
          ├── 01-brainstorm.md    # 头脑风暴产物
          ├── 02-plan.md          # 规划产物
          ├── 03-work.md          # 执行产物
          ├── 04-review.md        # 评审产物
          └── 05-compound.md      # 沉淀产物
```

**设计说明**：
- 按任务聚合组织产物，一个任务一个目录
- 运行态文件存放于 `.claude/tasks/`，与正式文档区分离
- 阶段文件按序号命名，便于识别流程顺序
- 支持多轮 revision 时可在任务目录下增加 `revisions/` 子目录

### 2.2 task.yaml 最小字段集

一期仅包含以下字段：

```yaml
task_id: 2026-03-29-feature-x
title: 功能X开发
role: fullstack
level: L2
current_stage: plan
status: in_progress
created_at: 2026-03-29T10:30:00+08:00
updated_at: 2026-03-29T11:10:00+08:00
```

**字段说明**：
- `task_id`：任务唯一标识，格式为 `YYYY-MM-DD-<slug>[-suffix]`，例如 `2026-03-29-feature-x`
- `title`：任务标题
- `role`：角色模式（product | dev | fullstack | test | generic），未指定角色时使用 `generic`
- `level`：任务级别（L1 | L2 | L3）
- `current_stage`：当前阶段（brainstorm | plan | work | review | compound）
- `status`：任务状态（pending | in_progress | blocked | completed | failed）
- `created_at`：创建时间（ISO 8601 格式）
- `updated_at`：更新时间（ISO 8601 格式）

**设计约束**：
- `task.yaml` 仅承担最小控制面职责，不承担完整工作流引擎功能
- 一期不包含依赖关系、并行节点、路由规则等复杂编排能力
- 状态更新由各 skill 在执行时写入，不依赖外部调度器
- `current_stage` 表示当前阶段，`status` 表示任务整体状态

### 2.3 沉淀态目录结构

通过 compound 阶段确认的资产，可选地迁移到正式知识库：

```
.claude/
  └── assets/
      ├── prompts/      # 提示词模板
      ├── workflows/    # SOP 和流程模板
      ├── domain/       # 领域知识卡片
      └── cases/        # 案例库
```

**资产文件要求**：
- 每个资产文件的 frontmatter 中应包含来源信息：
  - `source_task_id`：来源任务ID
  - `source_role`：来源角色
  - `source_stage`：来源阶段
  - `created_at`：创建时间
- 便于后续回溯和资产管理

## 3. 五个核心 Skills 设计

### 3.1 Skills 命名与职责

一期实现5个核心 skills，统一使用 `spec:` 前缀：

| Skill | 职责 | 主要输入 | 主要输出 |
|-------|------|---------|---------|
| `/spec:brainstorm` | 思路发散、方案探索 | 问题定义、背景、目标 | 候选方案、取舍分析 |
| `/spec:plan` | 任务拆解、路径设计 | brainstorm 结论、约束 | 任务清单、验收标准 |
| `/spec:work` | 执行落地、产出生成 | plan 结果、模板 | 文档草稿、分析结论、执行记录或阶段性交付物 |
| `/spec:review` | 质量校验、问题识别 | work 产物、评审维度 | 评分、问题清单、修订建议 |
| `/spec:compound` | 资产沉淀、知识提炼 | 已通过产物 | 模板、SOP、案例 |

### 3.2 参数设计

所有 skills 支持以下可选参数：

```bash
--role <product|dev|fullstack|test>   # 可选，启用角色视角模式
--level <L1|L2|L3>                     # 可选，未指定则智能建议
--task-id <task-id>                    # 可选，未指定则自动生成
```

**参数说明**：
- `--role`：可选参数，不指定时使用通用模式（`generic`）；指定后启用对应角色视角，影响提示模板、检查维度与输出格式。一期默认允许在不指定角色的情况下使用统一 skills 跑通闭环；`--role` 作为可选视角模式，在用户显式指定、系统建议或高差异场景（如 review、compound）下启用
- `--level`：可选参数，不指定时由 skill 根据任务复杂度智能建议并默认采用；用户可通过参数覆盖
- `--task-id`：可选参数，不指定时根据当前日期和任务标题自动生成，格式为 `YYYY-MM-DD-<slug>[-suffix]`，例如 `2026-03-29-feature-x`

**角色模式定位**：
- 一期不将角色区分作为强制前置步骤
- 角色能力以"视角模式"形式存在，在用户显式指定、系统建议或特定高差异场景（如 review、compound）下启用
- 默认允许用户在不指定角色的情况下跑通完整闭环

### 3.3 阶段契约与守门机制

每个 skill 执行时的标准流程：

1. **上游产物检查**：验证上一阶段产物文件是否存在，并满足基本可消费条件：
   - 文件存在且非空
   - 如果有 frontmatter，则 `status` 不处于失败态
   - 对 review 之后的流转，必须 `pass=true`
2. **元数据读取**：读取 `task.yaml`，确认当前状态和任务信息
3. **level 决策**：如未指定 `--level`，根据任务描述分析并建议任务级别，默认采用并提示可覆盖
4. **输入完整性检查**：根据确定的 level 验证必需输入是否完整
5. **执行与输出**：生成阶段产物（markdown 文件）
6. **状态更新**：更新 `task.yaml` 的 `current_stage`、`status` 和 `updated_at`，遵循以下规则：
   - skill 成功完成并产出可消费结果时，方可推进 `current_stage`
   - skill 执行失败时更新 `status=failed` 或 `blocked`
   - review 不通过时不得推进到 compound

## 4. 任务分级机制

### 4.1 三级任务定义

| 级别 | 特征 | 典型场景 | 流程要求 |
|------|------|---------|---------|
| L1 | 低风险、单轮可完成、输出较短 | 会议纪要整理、简单文案优化、测试点补充 | 可走简化链路 |
| L2 | 有明确交付物、需要标准流程、需要 review | PRD 辅助撰写、技术方案分析、测试方案设计 | 应走完整闭环 |
| L3 | 多角色参与、高风险或高复杂度、需要多轮 review | 跨团队需求方案、核心模块设计、复杂测试策略 | 必须走完整闭环 + 人工审查 |

### 4.2 Level 控制的六个方面

不同 level 影响以下方面：

1. **输入要求强度**
   - L1：最小上下文
   - L2：标准上下文
   - L3：完整上下文 + 风险说明

2. **输出结构复杂度**
   - L1：简版输出
   - L2：标准结构化输出
   - L3：增强结构化输出 + 风险项 + 人工确认项

3. **阶段完整度**
   - L1：可走简化链路
   - L2：应走标准链路
   - L3：必须走完整链路

4. **review 强度**
   - L1：轻 review
   - L2：标准 review
   - L3：增强 review + 强制返工规则

5. **人工介入要求**
   - L1：通常不强制
   - L2：关键场景人工确认
   - L3：必须人工确认

6. **compound 进入策略**
   - L1：默认不沉淀或按需沉淀
   - L2：合格内容可沉淀
   - L3：重点沉淀

## 5. Review 机制详细设计

### 5.1 Review 产物格式

review 阶段采用 frontmatter + markdown 正文的混合格式：

**Frontmatter（控制面）**：
```yaml
---
task_id: 2026-03-29-feature-x
stage: review
review_round: 1
reviewer_role: test
level: L2
status: failed

scores:
  correctness: 3
  completeness: 2
  executability: 4
  reusability: 3

summary_score: 12
blocking_issues: 2
major_issues: 1
minor_issues: 3

pass: false
rework_required: true
human_confirmation_required: false
compound_recommended: false

reviewed_at: 2026-03-29T14:30:00+08:00
---
```

**Markdown 正文（解释面）**：
```markdown
# Review Summary

本轮评审未通过，主要原因是需求覆盖不完整，且测试边界条件缺失。

## 1. 正确性 (3/5)
实现逻辑整体方向正确，但接口返回异常场景未被纳入验证...

## 2. 完整性 (2/5)
当前方案缺少边界条件、异常路径和权限校验场景...

## 3. 可执行性 (4/5)
测试计划可以执行，但需要补充明确的前置条件...

## 4. 可复用性 (3/5)
本轮内容具备部分复用价值，但由于场景覆盖不足...

## Blocking Issues
1. 缺少异常分支测试
2. 缺少权限控制验证

## Rework Suggestions
1. 补充异常路径测试用例
2. 增加权限边界校验
3. 明确测试前置数据准备要求
```

### 5.2 通过判定规则

review 是否通过采用双重判定机制，而非单纯依赖总分：

**规则1：关键维度底线**
- 正确性 < 3 → 不通过
- 完整性 < 3 → 不通过

**规则2：阻断问题优先**
- 存在 `blocking_issues > 0` → 默认不通过

**规则3：总分参考**
- 总分 < 12：通常不通过
- 总分 12-15：作为人工参考区间
- 总分 ≥ 16：可进入通过候选

**说明**：`summary_score` 仅作为参考，不作为唯一通过依据；是否通过以阻断问题、关键维度和返工要求综合判断。

### 5.3 不同 Level 的 Review 强度

- **L1**：可保留结论性评分，允许简化维度展开；正文可简化；不强制人工确认
- **L2**：标准四维评分；标准正文分析；关键场景人工确认
- **L3**：标准四维 + 强化问题分类；必须详细正文 + 风险项；必须人工确认

## 6. 角色适配设计

### 6.1 角色定位

一期覆盖三类核心角色：

| 角色 | 核心职责 | 重点使用场景 |
|------|---------|-------------|
| 产品人员 | 需求主导、业务定义、方案验收 | brainstorm、plan、review（需求视角）、compound |
| 研发人员（含全栈） | 技术落地、方案实现、问题排查 | brainstorm、plan、work、review（技术视角）、compound |
| 测试人员 | 质量把关、风险识别、缺陷分析 | brainstorm、plan、work（测试执行）、review（质量视角）、compound |

**全栈开发人员特殊说明**：
- 在一人公司或小团队场景下，全栈开发人员承担从需求讨论到交付验证的全流程职责
- 不仅是技术实现者，也是需求承接和需求输出角色
- 框架需支持其在需求、方案、开发、联调、质量校验等多个环节的连续性工作

### 6.2 角色区分策略

一期采用"统一 skill 骨架 + 角色模式切换 + 少量专属 skill 补充"的方式：

**实现原则**：
- 五步闭环类能力采用统一 skill，不按角色拆成独立 skill 集合
- 产品、研发、全栈、测试通过 `--role` 参数或 steering 切换不同提示模板、上下文结构、输出格式和检查维度
- 一期默认允许在不指定角色的情况下使用统一 skills 跑通闭环；`--role` 作为可选视角模式，在用户显式指定、系统建议或高差异场景（如 review、compound）下启用，用于调整提示模板、检查维度与输出结构
- 对于 PRD、全栈协同、测试执行、缺陷分析等高差异场景，可补充角色专属 skill（二期考虑）
- 不采用完全依赖文档说明的无差别模式

**角色视角的启用时机**：
- 用户显式指定 `--role` 参数
- 系统根据任务类型建议角色（用户可接受或忽略）
- 在 review、compound 等高差异场景下，角色视角更有价值

## 7. 五步闭环详细设计

### 7.1 Brainstorm 阶段

**目标**：思路发散、需求理解、方案探索、问题拆解、备选路径生成

**输入要求**：
- 问题定义
- 背景信息
- 业务目标
- 约束条件
- 已知风险或限制

**输出要求**：
- 至少 3 个候选方向
- 每个方向的优劣分析
- 推荐方向和原因

**完成标准**：
- 至少形成多个可比较的备选方向
- 输出包含取舍维度，而非仅有观点堆砌
- 可为 plan 阶段提供明确输入

### 7.2 Plan 阶段

**目标**：将发散思路收敛为可执行方案，形成任务计划与实施路径

**输入要求**：
- brainstorm 阶段产出
- 已确认的优选方案
- 资源与依赖信息
- 交付目标

**输出要求**：
- 目标定义
- 任务拆解
- 执行顺序
- 负责人建议
- 依赖与风险项
- 验收标准

**完成标准**：
- 任务可执行
- 风险与依赖可识别
- 后续 work 阶段无需重新理解任务目标

### 7.3 Work 阶段

**目标**：围绕既定计划完成实际产出

**输入要求**：
- 计划单
- 模板或规范
- 相关资料与上下文

**输出要求**：
- 文档草稿
- 分析结论
- 技术说明
- 代码相关辅助内容
- 测试执行记录、缺陷分析说明、验证结论等

**完成标准**：
- 输出符合既定格式和目标要求
- 可进入 review 阶段校验
- 可追溯到对应计划项

### 7.4 Review 阶段

**目标**：对 work 阶段产出进行质量校验与问题识别

**输入要求**：
- 执行产出物
- 原始目标与验收标准
- 相关规范或评审维度

**输出要求**：
- 问题清单
- 修订建议
- 通过/返工结论
- 风险说明

**完成标准**：
- 明确指出关键问题与优先级
- 给出可执行修订建议
- 能支持人工判断是否通过

**详细设计**：参见第 5 节 Review 机制详细设计

### 7.5 Compound 阶段

**目标**：将本轮协作成果沉淀为可复用资产，形成长期价值

**输入要求**：
- 已通过 review 的成果
- 过程中的高质量模板、方法、案例
- 可抽象复用的知识内容

**输出要求**：
- 模板
- SOP
- 知识卡片
- 评审清单
- 场景案例
- 提示词规范

**完成标准**：
- 沉淀内容可复用
- 可归类、可检索、可维护
- 能在后续同类任务中直接调用

**沉淀门槛**：
- 同类任务重复出现 3 次以上
- 可被跨角色复用
- 明显提升效率或质量
- 具有培训与示范价值

## 8. 状态流转与回退规则

### 8.1 基本流转

默认状态流转为：**Brainstorm → Plan → Work → Review → Compound**

### 8.2 进入条件

- Brainstorm 可在目标、背景、问题定义基本清晰时进入
- Plan 仅在 Brainstorm 已形成可比较候选方案时进入
- Work 仅在 Plan 已形成可执行任务和验收标准时进入
- Review 仅在 Work 已形成明确交付物时进入
- Compound 仅接收已通过 Review 的成果

### 8.3 回退规则

- Brainstorm 输出候选方案不足或目标不清时，不得进入 Plan
- Plan 缺失验收标准、依赖关系或关键任务拆解时，不得进入 Work
- Work 输出不完整或不符合格式要求时，不得进入 Review
- Review 存在阻断问题时，必须回退到 Work
- 任一阶段若上下文不足、目标冲突或信息缺失，可回退到上游阶段
- 任一高风险任务在关键判断节点可升级为人工接管

### 8.4 终止与升级

任务在以下场景可被终止或升级：
- 输入长期不完整
- 输出连续不满足要求
- 工具异常无法恢复
- 涉及高风险业务判断
- 需要人工定夺的争议性结论

此时任务需保留状态记录和失败原因，用于后续复盘与模板优化。

## 9. 异常与失败处理机制

### 9.1 输入失败

**典型场景**：
- 需求不完整
- 背景不足
- 目标不明确
- 上下文冲突

**处理方式**：
- 返回补充输入模板
- 不进入下一阶段
- 记录失败类型，用于优化输入规范

### 9.2 生成失败

**典型场景**：
- 输出空泛
- 逻辑跳跃
- 与约束冲突
- 结构不符合要求

**处理方式**：
- 允许一次受控重试
- 重试失败后进入 review 失败态
- 根据问题类型回退上游阶段

### 9.3 评审失败

**典型场景**：
- 阻断问题过多
- 关键缺漏
- 与业务目标不一致
- 无法满足交付标准

**处理方式**：
- 强制返工
- 标注失败原因
- 不得进入 compound

### 9.4 系统失败

**典型场景**：
- Claude 调用异常
- 工具不可用
- 上下文丢失
- 记录失败

**处理方式**：
- 人工接管
- 留痕记录
- 不中断任务可追踪性

### 9.5 失败回灌

所有失败类型应进入复盘体系，用于：
- 优化输入模板
- 优化 review checklist
- 形成反例案例库
- 优化阶段切换规则

## 10. 治理与人工兜底机制

### 10.1 治理目标

保障框架使用过程中的可控性、可追踪性和风险可管。

### 10.2 一期治理最小集

1. **权限分级**：按角色区分任务查看、模板维护、资产发布等权限
2. **日志留痕**：留存任务输入、AI 输出、review 结论、人工处理记录
3. **高风险动作人工确认**：正式发布、关键需求确认、核心技术结论采纳等必须人工确认
4. **问题回溯能力**：重要任务需支持从结果回溯到输入、过程与评审意见

### 10.3 人工介入节点

以下场景必须保留人工把关：
- 需求最终定稿
- 研发关键方案采纳
- 测试最终放行判断
- 对外输出内容
- 正式纳入知识库的核心资产

### 10.4 角色-阶段-权限矩阵原则

- 产品人员可发起 brainstorm、plan，可参与 review，可申请沉淀需求资产
- 研发人员（含全栈开发人员）可参与 brainstorm、plan、work、review、compound，可沉淀技术资产
- 测试人员可参与 brainstorm、plan、work、review、compound，可发起质量返工；其中 work 阶段重点包括执行测试用例、记录测试结果、提交缺陷、回归验证与输出测试结论
- 模板维护者或管理员可配置规则、发布模板、归档资产

## 14. 风险与应对

### 14.1 技术风险

**风险**：Claude Code CLI 的 skills 机制可能存在限制
**应对**：一期保持轻量设计，遇到限制时可通过外部脚本辅助

**风险**：YAML 文件读写可能出现并发冲突
**应对**：一期不支持并行任务，避免并发问题

### 14.2 使用风险

**风险**：用户可能不习惯结构化流程
**应对**：提供简化的 L1 模式，降低使用门槛

**风险**：角色区分可能增加理解成本
**应对**：角色设为可选，默认通用模式即可使用

### 14.3 质量风险

**风险**：AI 输出质量波动
**应对**：通过 review 机制和人工兜底保障质量

**风险**：资产沉淀质量参差不齐
**应对**：设置沉淀门槛，定期清理低质量资产

### 14.4 实现风险防范

**风险1：文件状态不一致**
- 典型场景：阶段文件已生成但 task.yaml 未更新，或 review 失败但 status 仍为 in_progress
- 应对原则：`task.yaml` 为任务控制面的事实来源，阶段文件为内容事实来源；两者不一致时，以 task.yaml 为状态判断基准，并记录修复日志

**风险2：任务分级沦为标签**
- 典型场景：L1/L2/L3 在实现时仅影响文案，不影响实际行为
- 应对原则：`level` 必须真实影响输入要求、review 强度、人工确认和 compound 策略，在实现时需验证这些差异确实生效

## 15. 附录

### 15.1 术语表

- **Spec-First**：规范优先，强调在执行前明确输入输出规范

- **执行面**：负责内容生成、分析输出的 markdown 层
- **运行态**：任务执行过程中的临时产物
- **沉淀态**：经过 compound 确认的正式资产

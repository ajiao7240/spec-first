# spec-first 终版规划

## 从 Spec-Driven Workflow 升级为 AI Coding Harness

这份规划按 **spec-first 开源维护者** 视角制定。核心目标不是继续堆 prompt、skill、agent，而是把 spec-first 打造成一套完整的：

```text
repo-local, evidence-governed, review-closed AI coding Harness
```

也就是帮助用户把 AI 辅助编码从：

```text
一次性聊天 / vibe coding / 临时 prompt / 单点 agent 执行
```

升级为：

```text
需求有锚点
上下文有路由
计划有证据
任务可交接
执行有边界
审查有闭环
知识可复用
团队可治理
```

# 1. 终局定位

## 1.1 一句话定位

```text
spec-first 是一套仓库本地的 AI Coding Harness，
用于把 Claude Code、Codex、Copilot、Gemini CLI、OpenCode 等 AI coding agent
纳入可控、可证、可审、可复用的工程闭环。
```

英文定位：

```text
spec-first is a repo-local Harness for evidence-governed AI-assisted software engineering.
```

## 1.2 spec-first 不是什么

```text
不是 another coding agent
不是 prompt collection
不是 slash command collection
不是 Spec Kit clone
不是 IDE
不是 SaaS 平台
不是重 workflow engine
不是 agent marketplace
```

## 1.3 spec-first 是什么

七层 Harness 能力 + 一层横切治理(详细映射见 §4.2):

```text
1. Workflow Harness        组织研发阶段、handoff、context budget
2. Artifact Protocol       artifact-header、task-pack、统一人机可读产物
3. Evidence Governance     evidence-packet、claim、risk、unknowns
4. Context + Tool Routing  context-request/bundle、tool-budget、provider-risk
5. Provider Adapter        runtime-capabilities、provider-artifacts、graph readiness
6. Review Closure          review-finding、fix-plan、re-review
7. Knowledge Compound      compound-delta、repo-profile、glue-map

横切:Skill Governance     stage-contract、skill-audit、capability-pack
```

最终产品心智:

```text
Spec 是入口。
Graph 是证据。
Plan 是决策。
Task 是交接。
Work 是受控执行。
Review 是质量闭环。
Compound 是知识复利。
```

# 2. 业界趋势判断

当前业界已经从“模型会不会写代码”进入“谁能把 agent 组织进真实工程系统”的阶段。

GitHub Spec Kit 已经把 Spec-Driven Development 定义为以 specification 为中心的 AI-assisted software development 方法：先定义要构建什么，再通过结构化阶段细化，并交给任意 AI coding agent 实现。这个趋势证明 `Spec > Code` 方向成立，但也意味着 spec-first 不能停在“spec 模板”，必须往完整 Harness 走。([github.github.com][1])

OpenAI Codex Skills 官方把 skill 定义成可复用能力包，并通过 progressive disclosure 管理上下文：模型先看到 name、description、path，只有决定使用时才加载完整 `SKILL.md`。这说明未来 skill 不是 prompt，而是 instructions + resources + scripts + context budget 的工程能力单元。([OpenAI 开发者][2])

Google Agent Skills 的 source snapshot(GitHub repo,local checkout hint:`../skill-google`,HEAD `caeec93`,2026-05-13)也验证了同一方向:产品域能力以 `SKILL.md` 作为入口,把深入材料下沉到 `references/`,把可复用配置样例放入 `assets/`。其中 GKE golden path、Google Cloud networking observability results-first 纪律、WAF security/reliability/cost checklist 说明 domain skill pack 的价值不在复制产品命令,而在把领域判断、默认路径、偏离取舍和验证清单包装成可按需加载的能力包。spec-first 应吸收这种能力包化结构,但不把外部产品 skill collection 直接并入核心 workflow。([Google Skills source snapshot][8])

Anthropic 在 MCP code execution 文章里指出，agent 连接数百甚至上千工具时，如果把所有工具定义和中间结果都塞进上下文，会增加成本、延迟并让流程变脆。这直接说明未来必须做 tool/context routing，而不是无脑扩 MCP。([Anthropic][3])

Anthropic 对 long-running agents 和 Managed Agents 的讨论也说明，未来 agent 会做更长周期、更复杂的工作，而 harness、context management、sandbox、工具抽象和可恢复机制会成为核心能力。([Anthropic][4])

GitHub Copilot coding agent 已经在强化 model picker、self-review、安全扫描、custom agents、CLI handoff；OpenAI Codex 也在往端到端工程任务、代码审查、并行 threads、worktrees、automations、cloud background tasks 发展。这说明 agent runtime 会越来越强，spec-first 不应该重复造 agent，而应该做 agent 之上的工程控制层。([The GitHub Blog][5])

所以未来 12–24 个月的核心趋势是：

```text
Agent Runtime 会商品化。
Skills / Plugins 会能力包化。
MCP / Tools 会爆炸。
Context Engineering 会成为基础设施。
Review / Verification 会成为质量瓶颈。
Long-running / multi-agent 会变成常态。
企业会从个人 AI 工具试用转向 AI-native engineering governance。
```

spec-first 必须提前解决未来问题，而不仅是当下问题。

> **中国/亚太生态 advisory(advisory only,不进入决议层)**:DeepSeek / 智谱 / 通义 / Kimi / 文心等本地模型,以及国内一体机、隔离网与受监管行业,会带来与英文公开材料不完全重叠的 host runtime / 网络 / 合规要求。本节仅作判断锚点提示,不在 §3 总体战略层级承担约束;具体落点见 §15 M7 capability-pack 的 "国内适配 advisory"。

# 3. 总体战略

## 3.1 战略目标

```text
把 spec-first 从 spec-driven workflow collection
升级为 evidence-governed AI coding Harness。
```

## 3.2 战略路径

```text
先统一定位
再建立协议
再打通证据
再路由上下文
再闭合 review
再支持长任务
再沉淀知识
再扩展能力生态
最后进入团队治理
```

## 3.3 核心原则

```text
1.  Repo-local first, not platform-first
2.  Evidence-first, not opinion-first
3.  Review-closed, not write-and-forget
4.  Context-budgeted, not context-stuffed
5.  Tool-risk-aware, not tool-permissive
6.  Skill-contract-based, not prompt-based
7.  Provider-replaceable, not provider-locked
8.  Host-agnostic, not host-locked
9.  Light contract, not heavy workflow engine
10. Scripts prepare, LLM decides
```

## 3.4 不做原则

```text
不和 Claude Code / Codex / Copilot 抢 agent runtime。
不做重状态机。
不做中心化 SaaS。
不做无边界 skill/agent 大杂烩。
不把 MCP 工具越接越多当成能力升级。
不把未验证猜测沉淀成知识。
不让 agent 越权成为最终裁判。
```

# 5. Repo-local 目录与边界

`.spec-first/` 是 Harness 的核心运行时工作目录,**不是 source-of-truth**。Source-of-truth 仍是 `skills/`、`agents/`、`templates/`、`src/cli/`、`src/cli/contracts/**`、`docs/`、`CLAUDE.md`、`AGENTS.md`(详见 `docs/10-prompt/结构化项目角色契约.md` §9)。

## 5.1 边界声明

`.spec-first/` 内部按 ownership 划分:

- **setup/script-owned facts**(确定性事实,setup 命令产出):`config/`、`providers/`、`graph/`、`standards/`
- **workflow-owned artifacts**(workflow 运行时产出):`runs/`、`reviews/`、`compound/`、`specs/`、`workflows/`、`sessions/`、`impact/`、`app-audit/`
- **audit snapshots**:`audits/` —— 默认不进入 runtime context(CLAUDE.md bootstrap 已声明)

任何区域都不替代 `docs/contracts/`、`skills/`、`src/cli/contracts/**` 作为 source-of-truth;任何区域内的 markdown 文档不被升级为当前 runtime contract。

`docs/` 给人读;`.spec-first/` 给 workflow / agent / review / resume / compound 读。

## 6.1 Stage Contract

最终每个核心 workflow / standalone skill 必须有 `contract.yaml`。

```yaml
schema_version: spec-first.stage-contract.v1
stage_id: plan
stage_type: planning

inputs:
  required:
    - requirements_packet
    - graph_readiness
  optional:
    - standards_preview
    - repo_profile
    - prior_compound_notes

outputs:
  required:
    - plan
    - evidence_packet
  optional:
    - risk_register
    - open_questions

rules:
  - do_not_write_code
  - cite_evidence_for_architecture_claims
  - mark_unknowns_explicitly

handoff:
  next:
    - write-tasks
```

## 6.3 Evidence Packet

所有关键阶段都要生成或更新 `evidence-packet.json`。

```json
{
  "schema_version": "spec-first.evidence-packet.v1",
  "spec_id": "2026-05-13-001-feature",
  "stage": "plan",
  "repo": {
    "branch": "feature/demo",
    "commit": "abc123"
  },
  "sources": [],
  "claims": [],
  "unknowns": [],
  "risks": []
}
```

证据链标准：

```text
claim → evidence → decision → task → diff → review → compound
```

默认 handoff 路径(advisory,不是 hard gate;偏离时由当前 workflow contract 与 reason_code 显式说明,不由本文件强制)：

```text
没有 evidence 的结论必须标 assumption(确定性 facts,不可豁免)。
默认进入 spec-work 前,task-pack 应先生成;若缺失,workflow 应 fallback 到 plan-mode 并记录 reason_code。
默认进入 code-review 前,diff / context / task-pack 三类输入应齐备;若缺一,reviewer 应在 finding 中标 evidence-degraded。
默认进入 compound 前,review closure(finding → fix-plan → re-review)应已完成;若强制 compound 未结案的 finding,产物必须显式标 unresolved。
```

边界声明:本节的"默认路径"是 light contract,不是中心化 gate 系统;实际 gate 由 `docs/contracts/quality-gates/` 与各 workflow contract 维护,与角色契约 §11–§12 反对中心化 gate / 重状态机的原则一致。

## 6.4 Task Pack

```yaml
schema_version: spec-first.task-pack.v1
spec_id: 2026-05-13-001              # 必选;短名 (-feature-name) 可选,不强制后缀

tasks:
  - id: T-001
    title: 增加 runtime-capabilities.json 生成逻辑
    source_requirements:
      - R-001
    evidence:
      - CLAIM-001
    expected_files:
      - skills/spec-mcp-setup/scripts/check-health
    acceptance:
      - 运行当前 host 的 setup entrypoint(Claude `/spec:mcp-setup` / Codex `$spec-mcp-setup`)后生成 runtime-capabilities.json
      - 缺少必需 helper 时 baseline_ready=false
    risk:
      level: medium
      reason: 会影响 graph-bootstrap readiness 判断
    review_focus:
      - provider readiness
      - cross-platform path handling
```

> spec_id 命名规范:`YYYY-MM-DD-NNN[-short-name]`。短名仅作为人类可读提示,validator 与 anchor lookup 以 `YYYY-MM-DD-NNN` 为准。

## 6.6 Context Bundle

`context-bundle.v1` 协议已落地在 `docs/contracts/context-bundle.md` + `docs/contracts/context-governance.md` + `docs/contracts/artifact-summary.md`,本节不重复 schema 定义,只示意核心字段:

```json
{
  "schema_version": "spec-first.context-bundle.v1",
  "providers_used": [],
  "included_context": [],
  "excluded_context": [],
  "tool_risk": [],
  "confidence": "medium"
}
```

Context Bundle 必须说明：

```text
为什么包含这些上下文。
为什么排除这些上下文。
用了哪些 provider。
哪些 provider degraded。
```

## host deprecation playbook(对 Anthropic / Codex 等单宿主退役兜底)

```text
  - 任何 host-specific 行为以 capability facts 形式声明,不进入核心 prompt。
  - 退役场景演练:在 docs/contracts/host-deprecation.md(待建)中维护一份
    "当前所有支持 host + EOL 风险评分 + 替换备份 host" 的列表,每 90 天复核。
  - 至少保证 1 个 fallback host 可在不修改源端 SKILL.md 的前提下,通过
    spec-first init --<other-host> 重建 runtime mirror。

不做:
  不锁定到 Anthropic / OpenAI / Google / GitHub 任一单一 host;
  不允许 host-specific 调用直接出现在 SKILL.md prose 中。
```

### 国内适配 advisory(advisory only,不进 M7 必通过项)

国内/亚太大量企业用户与外部 host 接入条件不同,以下点位仅作 advisory,不在 M7 出 release 前做强承诺,但需要在 capability-pack manifest 与 host adapter 设计中保留足够 hook,避免后续被迫做侵入式重构:

```text
1. 国产模型适配:DeepSeek / 智谱 / 通义 / Kimi / 文心等模型可能成为某些团队的默认
   coding host,capability-pack 不应在 prompt 中硬编码任何 host-specific 名字。
2. 离线模式:对内网/隔离网用户,spec-first init / doctor 必须能在无外网情况下完成
   一次完整 runtime 重建;capability-pack source 默认走本地 vendored 路径。
3. 国密签名(SM2/SM3/SM4):部分受监管行业可能要求 capability-pack source-lock /
   commit hash 走国密 hash;这是 long-tail capability,M7 默认仍用 SHA-256,
   保留 advisory 接入点,后续按真实需求再决定是否加 plugin 化签名 backend。
```

判断锚定:仅在出现 ≥3 个真实国内用户提出该需求,且对应 host runtime 在用户群形成稳定份额时,才把上述 advisory 升级为 M7+ 必做项。

## OTel GenAI 对齐与合规节点(原 V1.0 主线 E + §2 业界趋势判断)

M8 输出的 metrics / facts 必须对齐 **OpenTelemetry GenAI semantic conventions**(facts emission 思路,而非内化为自有可视化平台);spec → plan → task → code → review → artifact 链路在 trace 层可追溯;基础日志留痕与人工兜底机制可用。

```text
AI 改动 commit / PR 标注 contract:
  在 commit footer 或 PR metadata 注入 ai-attribution / spec-anchor / workflow 等字段,
  作为 "AI 改动分组度量" 与 "AI 改动回归率" 指标的前置条件。
  在 M8 阶段以 advisory facts emission 起步,不构成强 gate。
```

合规锚点:**EU AI Act 分阶段适用(advisory anchor)**。官方 Article 113 / FAQ 将适用日期拆为多个节点:主体规则 2026-08-02 起适用,Article 6(1) 相关高风险系统要求在 2027-08-02 起适用;同时 high-risk timeline 正受 Digital Omnibus 调整影响。M9 GA 不绑定单一法规日期;若 spec-first 进度滞后或外部法规节点变化,必须从对外承诺中剥离 "合规级 audit trail" 叙事,避免对客户做出无法兑现的合规声明。**判断锚定日期需每 90 天复核一次,超过 180 天未更新视为过期。**([EU AI Act Article 113][6])([EU AI Act FAQ][7])

### EU AI Act fallback path(实操要求超出 OTel facts 范围时的兜底)

OTel GenAI semantic conventions 覆盖的是 "AI 改动可被追踪 + 可被回放" 这一层;EU AI Act / 行业监管(金融、医疗、政务等)可能要求更强的:**审计证据保留期、签名链、人工 sign-off、access log、operator-specific identity binding**。这些要求大概率超出 OTel facts emission 自身的承诺范围。

```text
触发条件:
  外部合规要求中出现以下任一项,且 spec-first 现有 OTel facts 无法直接覆盖:
    - artifact 长期保留(≥ 1 年)+ 防篡改要求
    - 人工 sign-off 链路、双人确认或外部审计员介入
    - operator identity / role binding(超出当前 .developer profile 表达力)
    - 强制审计员日志格式(非 OTel)

fallback 动作(按优先级顺序):
  1. 显式声明 spec-first 不承担该层合规叙事,refer 用户到外部 governance 平台
     (例如外部 DLP / SIEM / 审计 SaaS)。
  2. 将 spec-first OTel facts 作为下游审计平台的输入源之一,不作为唯一真相。
  3. 仅当至少 3 个真实合规客户出具明确需求,且能用 advisory capability-pack 形式解决,
     才考虑作为 M9 之后的 capability-pack 候选,不进 M8 必通过项。

不做:
  - 不内化"合规审计平台"叙事;
  - 不承诺 EU AI Act / SOC2 / ISO 27001 / 等保 等任意一项的 "我们已合规";
  - 不让 review / compound 工作流去模拟人工 sign-off。
```

# 20. 关键维护者决策

## 决策 1：不做 another coding agent

原因：

```text
Claude Code / Codex / Copilot / Gemini CLI 会持续变强。
spec-first 应站在它们之上，做工程控制层。
```

## 决策 2：不做重状态机

原因：

```text
spec-first 的优势是 light contract。
重状态机会降低灵活性，并制造流程债。
```

## 决策 3：先 repo-local，再 workspace，再 team

路径：

```text
repo-local artifacts
→ workspace advisory
→ team governance
→ future platform optional
```

## 决策 4：review 是核心壁垒

原因：

```text
AI 写代码会越来越便宜。
证明代码是对的会越来越值钱。
```

## 决策 5：compound 是长期复利

原因：

```text
普通 agent 的产物是代码。
spec-first 的产物是代码 + 证据 + 决策 + 复盘 + 可复用知识。
```

## 决策 6:roadmap-gap 优先级 ≠ user-pain 优先级

本规划目前的 P0/P1/P2 优先级,反映的是 **roadmap 推导出的能力缺口**,不是经过用户访谈或外部 telemetry 验证过的真实 user-pain 排序。

```text
判断准则:
  roadmap gap = 在 Harness 抽象内部"少一个 contract / 少一个 readiness 证据"。
  user pain   = 真实开发团队在使用 spec-first 完成 brainstorm → plan → tasks → code →
                review → compound 时,反复抱怨 / 流程卡死 / 证据缺失的具体场景。

落地动作:
  - 任何 milestone 推进前,允许 roadmap-gap 排序作为工作起点,但禁止把它伪装成"用户痛点"。
  - 继续推进 M3 增量或对外声称 "user-pain driven priority" 前,必须完成 ≥5 个真实用户(同时使用 Claude Code / Codex / 其他 host 各 ≥1)
    的访谈,把结果写入 docs/research/user-pain-validation.md(待建)。
  - 该访谈未完成前,Brainstorm/Plan/Review 工作流可继续基于 roadmap-gap 演化,
    但对外公开材料须使用 advisory 措辞,不得以 "用户痛点驱动的优先级" 自述。

风险:
  混淆 roadmap-gap 与 user-pain 会让 spec-first 在外部传播中被解读为 "PM 拍脑袋的工程清单"。
```

# 22. 最终结论

```text
spec-first 不是更强的 spec 工具,而是面向 AI coding 未来形态的 Harness:
repo-local、evidence-governed、review-closed、knowledge-compounding。

它的价值不是让 AI 多写代码,
而是让 AI coding 同时具备需求锚点、上下文路由、证据链、任务交接、质量审查、
修复闭环、知识复用、团队治理。

它要解决的不只是今天 AI coding 混乱的问题,
还要提前应对 agents、tools、context、review、long-running 在未来变得更复杂的问题。
```

详细路线见 §7 / §21,各 Milestone 边界见 §8–§17。

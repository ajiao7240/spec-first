# AI辅助工作框架版本路线图

## 1. 路线图目标

本路线图用于明确 AI 辅助工作框架的分阶段建设路径，避免一期范围过大、目标发散、建设失焦。路线图以 **Harness Engineering** 为指导思想，以“让软件交付从脆弱、手工、割裂，变成智能、自动、可控”为核心目标，按照“先验证、再可用、再首发、再扩展”的方式推进。

路线图遵循以下原则：

* 先收敛 Claude 单模型工具支撑，再谈模型扩展；
* 先打通 spec-first 最小闭环，再补全治理和平台能力；
* 先覆盖产品、研发、测试三类核心角色，再扩展其他角色；
* 先让用户能稳定使用，再提升自动化、资产化和可观测性。

---

## 2. 版本划分总览

| 版本   | 定位    | 核心目标                                  | 关键词  |
| ---- | ----- | ------------------------------------- | ---- |
| V0.1 | 概念验证版 | 验证 spec-first + Claude CLI + 五步闭环是否可行 | 能跑通  |
| V0.5 | 可用版   | 建立稳定使用规则，补齐流程控制和失败处理                  | 能稳定用 |
| V1.0 | 首发版   | 形成正式的一期 Claude 工具支撑框架                 | 能推广  |
| V2.0 | 扩展版   | 在框架母体不变前提下扩展 Codex 和平台能力              | 能扩展  |

## 3. V0.1 概念验证版

### 3.1 版本定位

V0.1 的目标不是做完整系统，而是验证这套思路是否成立，证明：

* Claude 是否能支撑 spec-first 模式；
* `/spec:xxx` 命令方式用户是否能接受；
* brainstorm → plan → work → review → compound 这条链路是否能跑通；
* 产品、研发、测试三个角色是否存在真实使用价值。

### 3.2 核心目标

V0.1 需要回答的核心问题只有一个：

**这套框架是否值得继续投入。**

### 3.3 范围

V0.1 只做最小闭环，不追求完整治理和复杂系统能力。

包含内容：

* 明确 Harness 核心目标与框架定位；
* 明确五步闭环流程；
* 明确一期角色：产品、研发、测试；
* 明确 Claude 是唯一支撑模型；
* 明确 `npm i spec-first` + Claude CLI + `/spec:xxx` 使用模式；
* 支持少量核心 spec。

### 3.4 首批建议命令

```bash
/spec:brainstorm
/spec:plan
/spec:code-review
/spec:prd
/spec:fullstack
/spec:test-plan
/spec:test-exec
```

### 3.5 暂不建设内容

* 复杂权限体系；
* 完整模板治理；
* 完整可观测性指标；
* Codex 扩展；
* 自动回灌机制；
* 大规模系统集成。

### 3.6 成功标准

* 有真实用户能用起来；
* 至少 2 到 3 类典型场景跑通；
* 用户理解 spec-first 交互方式；
* 框架不是停留在文档层，而是已有实际使用入口。

## 4. V0.5 可用版

### 4.1 版本定位

V0.5 的目标，是从“能跑通”升级到“能稳定使用”。

### 4.2 核心目标

让产品、研发、测试三类角色可以在日常工作中，用相对稳定、低学习成本的方式使用框架，而不依赖临时 prompt 和个人经验。

### 4.3 核心建设内容

V0.5 重点补齐“可控性”：

* 完整定义状态流转规则；
* 完整定义各阶段输入输出；
* 明确控制面与执行面分离；
* 完整定义上下文装配规范；
* 完整定义异常与失败处理机制；
* 完整定义角色—阶段—权限矩阵原则；
* 补充首批模板与 SOP；
* 明确 CLI 命令行为规范。

### 4.4 用户使用表现

用户可以比较稳定地完成以下链路：

1. 安装 spec-first；
2. 进入 Claude CLI；
3. 使用 `/spec:xxx`；
4. 按提示补充上下文；
5. 得到结构化输出；
6. 进入 review 或 compound。

### 4.5 版本重点

V0.5 的关键词不是“更多功能”，而是：

**更稳定、更可控、更少依赖个人发挥。**

### 4.6 成功标准

* 输出质量明显比自由聊天稳定；
* 用户不需要自己组织复杂 prompt；
* 三类角色都能在日常工作中使用；
* 失败场景可控，不会一坏就乱；
* 开始形成初步资产沉淀。

## 5. V1.0 首发版

### 5.1 版本定位

V1.0 是正式的一期发布版本，目标是形成：

**可推广、可评审、可复制的 Claude 工具支撑框架。**

### 5.2 核心目标

V1.0 要实现的不是“试用”，而是“形成团队统一使用方式”。

### 5.3 核心建设内容

* 完整框架文档定稿；
* Claude 单模型支撑规范定稿；
* spec-first CLI 使用模式定稿；
* 首批核心 spec 定稿并稳定支持；
* 资产沉淀机制可用；
* 模板治理机制可用；
* 基础日志留痕和人工兜底机制可用；
* 核心指标可观测。

### 5.4 推荐首发命令集

#### 流程类

```bash
/spec:brainstorm
/spec:plan
/spec:work
/spec:code-review
/spec:compound
```

#### 产品类

```bash
/spec:prd
/spec:requirement
```

#### 研发类

```bash
/spec:dev
/spec:fullstack
/spec:tech-plan
/spec:api
```

#### 测试类

```bash
/spec:test-plan
/spec:test-exec
/spec:defect
/spec:qa-review
```

### 5.5 首发版本重点能力

* 产品、研发、测试三类角色形成稳定使用习惯；
* 全栈开发人员获得重点支持；
* 测试执行链路定义清晰；
* 资产开始反向支撑后续任务；
* review 和 compound 不再是口号，而是实际可执行环节。

### 5.6 成功标准

* 可以作为团队统一使用方式推广；
* 首批高复用模板和案例资产形成；
* 任务通过率、返工率、复用率开始可观测；
* 软件交付链路开始从“手工割裂”向“受控连续”演进。

## 6. V2.0 扩展版

### 6.1 版本定位

V2.0 的目标是在框架母体不变前提下，扩展模型与平台能力。

### 6.2 核心目标

从“Claude 单模型支撑框架”升级到“可扩展的 Harness 母体”。

### 6.3 核心建设内容

* 扩展 Codex 工具支撑规范；
* 抽象模型接入标准；
* 强化模板治理和资产治理；
* 增强可观测性与反馈回路；
* 补充更多角色适配；
* 加强与 IDE、办公系统、测试工具的集成；
* 提升自动推荐、自动回灌、自动检查能力。

### 6.4 重点说明

V2.0 仍然不是以“多模型协同编排”为首要目标，而是：

* 先复制 Claude 成功经验；
* 再做 Codex 的独立支撑规范；
* 在统一 spec-first 入口下扩展能力。

### 6.5 成功标准

* 不同模型可复用同一框架母体；
* 使用方式不因模型扩展而碎裂；
* 资产、规则、治理可跨版本延续；
* 框架开始具备平台化能力。

## 7. 版本边界建议

### 7.1 第一批必须落地

建议纳入 V0.5 / V1.0：

* Claude 单模型支撑；
* spec-first CLI 入口；
* 五步闭环；
* 产品 / 研发 / 测试三角色；
* 全栈开发重点支持；
* 测试执行链路；
* 状态流转；
* 失败处理；
* 基础资产沉淀。

### 7.2 第二批补强

建议纳入 V1.0+：

* 模板治理完善；
* 可观测指标完善；
* 更多 spec；
* 更完整的回灌机制；
* 更细粒度权限治理。

### 7.3 第三批扩展

建议纳入 V2.0：

* Codex 支撑；
* 更强系统集成；
* 更强自动化能力；
* 更丰富角色生态。

---

## 8. 当前阶段判断

结合目前需求成熟度，当前状态更接近：

**方法论与框架设计已接近 V0.5，若补齐命令清单、状态规则、模板与失败机制，可进入 V1.0 设计定稿阶段。**

这意味着当前最合理的推进节奏不是直接谈 V2.0，而是：

**先完成 V0.5，再锁定 V1.0 首发范围。**

---

## 9. 推荐推进顺序

### 第一阶段

完成 V0.1 验证：

* 跑通最小闭环；
* 确认命令模式可接受；
* 验证角色场景价值。

### 第二阶段

推进 V0.5：

* 补流程控制；
* 补失败处理；
* 补上下文规范；
* 补角色矩阵；
* 让框架可稳定使用。

### 第三阶段

锁定 V1.0：

* 固化首发 spec；
* 固化核心模板与 SOP；
* 建立治理最小集；
* 形成可推广版本。

### 第四阶段

进入 V2.0：

* 扩 Codex；
* 扩平台能力；
* 扩更多集成与反馈机制。

---

## 10. 总结

本路线图建议将 AI 辅助工作框架拆分为四个版本推进：

* **V0.1** 证明可行；
* **V0.5** 建立可用；
* **V1.0** 形成首发；
* **V2.0** 进入扩展。

这样拆分的好处是：

* 避免一期做成大而全；
* 保持 Claude 单模型支撑的收敛性；
* 让 spec-first 使用模式尽快落地；
* 让软件交付升级目标逐步兑现，而不是停留在理念层。

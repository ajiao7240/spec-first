# AI 辅助工作框架版本路线图

## 1. 路线图目标

本路线图用于明确 AI 辅助工作框架的分阶段建设路径，避免一期范围过大、目标发散、建设失焦。路线图以 **Harness Engineering** 为指导思想，以"让软件交付从脆弱、手工、割裂，变成智能、自动、可控"为核心目标，按照"先验证、再可用、再首发、再扩展"的方式推进。

路线图遵循以下原则：

* 守护核心哲学：**Light contract、Explicit boundaries、Scripts prepare 与 LLM decides**；
* 守护核心链路：`Codebase → Graph → Spec → Plan → Tasks → Code → Review → Knowledge`；
* 先打通 spec-first 最小闭环，再补全治理和平台能力；
* 先覆盖产品、研发、测试三类核心角色，再扩展其他角色；
* 先让用户能稳定使用，再提升自动化、资产化和可观测性；
* 对齐 2025–2026 业界主线：Context Engineering、Spec-Driven Development、Single-agent + ephemeral subagent、MCP、AgentOps；
* 优先建立"大厂平台不会做"的差异化层：source/runtime 边界、scripts/LLM 边界、artifact provenance、review→knowledge 闭环。

本路线图不规定运行时 workflow 行为，也不替代 `docs/10-prompt/结构化项目角色契约.md`。任何与角色契约冲突的措辞，必须先按角色契约重写。

## 3. V0.1 概念验证版

### 3.1 版本定位

V0.1 的目标不是做完整系统，而是验证这套思路是否成立，证明：

* Claude 是否能支撑 spec-first 模式；
* `/spec:xxx` 命令方式用户是否能接受；
* brainstorm → plan → work → review → compound 这条链路是否能跑通；
* 产品、研发、测试三个角色是否存在真实使用价值。

### 3.2 核心目标

V0.1 需要回答的核心问题只有一个：

**这套框架是否值得继续投入。**

### 3.3 范围

V0.1 只做最小闭环，不追求完整治理和复杂系统能力。

包含内容：

* 明确 Harness 核心目标与框架定位；
* 明确五步闭环流程；
* 明确一期角色：产品、研发、测试；
* 明确 Claude 是首发支撑模型（Codex 双宿主作为 V1.0 首发能力，见 §5）；
* 明确 `npm i spec-first` + Claude CLI + `/spec:xxx` 使用模式；
* 支持少量核心 spec。

### 3.4 首批建议命令

以下命令仅在 Claude Code host 上进入 V0.1 验证范围，Codex host 行为留待 V1.0 双宿主首发时统一交付。

```bash
/spec:brainstorm     # Claude only, V0.1 scope
/spec:plan           # Claude only, V0.1 scope
/spec:code-review    # Claude only, V0.1 scope
/spec:prd            # Claude only, V0.1 scope
/spec:fullstack      # Claude only, V0.1 scope
/spec:test-plan      # Claude only, V0.1 scope
/spec:test-exec      # Claude only, V0.1 scope
```

### 3.5 暂不建设内容

* 复杂权限体系；
* 完整模板治理；
* 完整可观测性指标；
* 自动回灌机制；
* 大规模系统集成。

### 3.6 成功标准

* 有真实用户能用起来；
* 至少 2 到 3 类典型场景跑通；
* 用户理解 spec-first 交互方式；
* 框架不是停留在文档层，而是已有实际使用入口。

## 4. V0.5 可用版

### 4.1 版本定位

V0.5 的目标，是从"能跑通"升级到"能稳定使用"，并补齐 2025–2026 业界主线中两条必须的能力底座：**Context Engineering** 与 **Artifact Provenance**。

### 4.2 核心目标

让产品、研发、测试三类角色可以在日常工作中，用相对稳定、低学习成本的方式使用框架，而不依赖临时 prompt 和个人经验；同时为 V1.0 的双宿主 harness 与 spec drift 治理铺好 schema 与脚本基线。

### 4.3 核心建设内容

V0.5 重点补齐"可控性"与"输入质量"，按四条主线推进：

**主线 A：流程稳定**

* 定义**轻量 confirmation point + 失败回路**：每个 phase 末尾一个 advisory 检查点，不引入多状态流转、不引入中心化 gate、不引入强编排（严格遵守角色契约 §12 反模式约束）；
* 定义各阶段输入输出 schema；
* 明确控制面（workflow 入口治理）与执行面（skill / agent / CLI）分离；
* 定义异常与失败处理的最小契约：facts emission + reason_code + human fallback；
* 定义角色—阶段—权限矩阵原则（原则即可，不构建权限治理引擎）。

**主线 B：Context Engineering 能力层**

对齐 Anthropic *Effective Context Engineering for AI Agents*、Claude Agent SDK 五层 compaction 实践，以及 arxiv 关于 Agentic Context Engineering 的研究主张：

* 定义 context budget contract：每个 workflow 进入时声明 token budget 与 critical context 集合；
* 定义 compaction trigger 与降级策略：长会话与跨 phase handoff 时由脚本输出 facts，由 LLM 决定 compaction 边界；
* 定义 progressive disclosure 规则：skill / spec / context 按需注入，不在会话启动时一次性灌入；
* 定义 subagent context isolation 规则：bounded subagent 默认无主会话历史，只接收 task contract 与必要 artifact 摘要。

落地点：`docs/contracts/workflows/context-governance.md` 作为 contract source，各 `SKILL.md` 在 frontmatter 或入口章节声明 context budget 与 critical context；不新建中心化 context engine。

**主线 C：Artifact Provenance 基线**

消费角色契约 §7.5 已经定义的 schema 字段，把 advisory / confirmed / generated / degraded 区分落到所有首批 artifact：

* 所有 workflow 产出的 artifact 必须携带 `schema_version`、`producer`、`source`、`freshness`、`hash` 或 `timestamp`、`authority_level`、`degraded_status`、`reason_code`、`consumer`；
* 明确 advisory 与 confirmed 的边界，禁止把 advisory facts 当作 confirmed truth 流入下游；
* 明确 source-of-truth 与 generated runtime 的产物分类，并把 drift 检测脚本化（具体 drift 治理在 V1.0 落地）。

**主线 D：模板与 SOP**

* 补充首批模板与 SOP；
* 明确 CLI 命令行为规范；
* 明确双宿主初始化与 runtime 重建命令（`spec-first init --claude|--codex`）的预期产物与失败模式。

### 4.4 用户使用表现

用户可以比较稳定地完成以下链路：

1. 安装 spec-first；
2. 进入 Claude CLI（或 Codex CLI，V0.5 仅作为预备能力，主推 V1.0 首发）；
3. 使用 `/spec:xxx`；
4. 按提示补充上下文；
5. 得到带 provenance 字段的结构化输出；
6. 进入 review 或 compound。

### 4.5 版本重点

V0.5 的关键词不是"更多功能"，而是：

**更稳定、更可控、更少依赖个人发挥、Context 与 Artifact 可被脚本验证。**

### 4.6 成功标准

* 输出质量明显比自由聊天稳定；
* 用户不需要自己组织复杂 prompt；
* 三类角色都能在日常工作中使用；
* 失败场景可控，不会一坏就乱；
* 所有首批 artifact 携带 provenance schema，可被脚本验证；
* 长 horizon 会话不再因 context rot 失控；
* 开始形成初步资产沉淀。

## 5. V1.0 首发版

### 5.1 版本定位

V1.0 是正式的一期发布版本，目标是形成：

**可推广、可评审、可复制、可取证的双宿主 spec-first harness。**

V1.0 首发就必须具备双宿主能力。原因：2026 上半年业界已收敛为 agent-agnostic 主张（GitHub Spec-Kit 等已支持 30+ agent），把"Codex 扩展"留到 V2.0 会丢失差异化窗口。

> **本路线对 "agent-agnostic" 的边界声明**：
> spec-first 的 agent-agnostic 是 **protocol-level**，不是 host-level：
> 通过 artifact contract、facts schema、MCP 与 capability declaration 让任意符合 contract 的 host 可消费；
> spec-first 自身**首发只验证 Claude Code 与 Codex CLI 两个 host**，不承诺把每个新 host 一一适配。
> 接入新 host 是 downstream 实现工作，不是核心母体职责。

### 5.2 核心目标

V1.0 要实现的不是"试用"，而是"形成团队统一使用方式，并具备企业取证基线"。

## 6. V2.0 扩展版

### 6.1 版本定位

V2.0 的目标是在框架母体不变前提下，扩展跨项目知识能力与 AgentOps 生态对接能力。

V2.0 不再以"扩 Codex"为主线——双宿主能力已在 V1.0 首发完成。V2.0 关注的是 spec-first 与外部知识网络、外部 trace/eval 生态、外部规范注册表之间的 artifact contract。

### 6.2 核心目标

从"双宿主 harness"升级到"可对接外部知识与可观测生态的 Harness 母体"。

### 6.3 核心建设内容

**主线 A：跨项目知识 Artifact Contract**

* 把项目级 knowledge 沉淀（CHANGELOG、docs、compound 产物）抽象为可被外部 registry 消费的 artifact contract；
* 通过 artifact contract 让**任意符合 contract 的外部 registry 可消费**——Tessl Spec Registry 仅作为参考实现案例，不作为默认 / 单一 consumer，避免对单一第三方平台形成单点依赖；
* 明确跨项目知识的 freshness、authority_level 与 conflict resolution 边界。

**主线 B：AgentOps 与持续 Eval 对接**

* 与 LangSmith / Datadog / Phoenix 等外部 AgentOps 平台对接 trace；
* 把 fresh-source eval 与 continuous eval 串成可重放的证据链；
* **spec-first 只输出 OTel-compatible facts，不内化 dashboard、retention 治理、alert 与 viz**；这些能力由外部 AgentOps 平台承担。

**主线 C：更强模型接入标准与角色生态**

* 抽象模型接入标准（保持 agent-agnostic）；
* 补充更多角色适配；
* 加强与 IDE、办公系统、测试工具的集成；
* 提升自动推荐、自动回灌、自动检查能力。

**主线 D：模板与资产治理深化**

* 强化模板治理和资产治理；
* 增强可观测性与反馈回路；
* 明确 schema 演进与 downstream consumer 兼容策略。

### 6.4 重点说明

V2.0 不是以"多模型协同编排"为首要目标。spec-first 的核心哲学反对强编排、多状态流转和中心化 gate 系统。V2.0 仍然走"主 agent + 工具 + 结构化 workflow + ephemeral subagent"这条业界已收敛的路线（对齐 Anthropic 研究系统、Cognition 工程主张，以及关于 single-agent + ephemeral subagent 模式的 arxiv 论文；具体编号在引用前由 `docs/09-业界借鉴/` 校验后填入，避免引用未核实的编号）。

### 6.5 成功标准

* 不同模型可复用同一框架母体；
* 使用方式不因模型扩展而碎裂；
* 资产、规则、治理可跨项目延续；
* artifact contract 可被外部 registry / trace / eval 平台消费；
* 框架开始具备平台化能力，但不退化为中心化平台。

### 8.1 已收敛的业界共识与本路线的对齐

### 8.2 与主要竞品的差异化壁垒

* **vs. GitHub Spec-Kit**：spec-kit 是 agent-agnostic 脚手架，覆盖广、社区认知强，但**不做 source/runtime 边界、不做 drift 检测、不做 review→knowledge 闭环**。本路线必须以差异化壁垒为首发叙事，不与 spec-kit 在"spec-driven"标签上正面竞争。
* **vs. AWS Kiro**：Kiro 是封闭 IDE，spec-first 是宿主无关 harness。差异化在双宿主与 CLI-first。
* **vs. Tessl**：Tessl 走 spec-as-source 路线 + 自建 Spec Registry。本路线**不自建中心化 registry**，通过 artifact contract 让外部 registry 可消费。
* **vs. Claude Agent SDK / Claude Code 原生能力**（snapshot 时间窗内观察）：Anthropic 正在持续内化 skills、hooks、subagent、compaction 等 harness 基础设施，平台方功能边界处于快速变化期。本路线必须保持双宿主、review→knowledge 闭环、source/runtime 边界这三层平台方在当前 snapshot 内尚未提供的能力；该判断需随 Claude / Codex 官方能力更新滚动复核（建议每个 minor release 校准一次）。
* **vs. Cursor / Cline / Aider / OpenHands**：这些是执行层工具，与本路线不直接竞争，可作为执行层组合使用。

### 8.3 必须警惕的趋势信号

1. **平台方内化 harness 能力**：Claude Agent SDK、Codex Skills 在持续吃 harness 基础设施。本路线必须把差异化能力放在 V1.0 首发，而不是等到 V2.0。
2. **模型能力跃迁带来的 ROI 举证压力**：200K+ context、subagent 原生、computer use 普及后，"harness 还有必要吗"的质疑会上升。本路线必须用可度量证据（任务通过率、回归率、可审计性）回应。
3. **EU AI Act 2026-08 高风险系统全面执行**：企业采购会把"合规级 audit trail"作为必选项。V1.0 必须具备基础 trace 能力，否则会被合规层绕过。

## 9. 反模式提示

本节的反模式来自三条正面原则：**Light contract + Explicit boundaries + Scripts prepare, LLM decides**，以及"用户真实研发增益优先于架构炫技"。完整正面规则与判断矩阵见角色契约 §11–§12；本节只列出明确不会走的方向，与角色契约 §12 保持一致：

* 不引入多状态流转或中心化 gate 系统；
* 不构建强编排或复杂规则引擎替代 LLM 判断；
* 不构建自有 IDE、自有 trace 可视化平台、自有 Spec Registry（除非有充分必要并已通过角色契约 §11 判断矩阵）；
* 不让脚本决定语义范围、业务优先级或 review 结论；
* 不让 LLM 伪造确定性校验或编造命令结果；
* 不把 advisory facts 当 confirmed truth；
* 不把 internal helper skill / agent 暴露成用户入口；
* 不把 generated runtime mirror 当作 source-of-truth；
* 不把 historical docs 升级为当前 runtime contract；
* 不把 provider 内部实现细节泄漏成 workflow contract。

## 12. 总结

本路线图建议将 AI 辅助工作框架拆分为四个版本推进：

* **V0.1** 证明可行；
* **V0.5** 建立可用，补齐 Context Engineering 与 Artifact Provenance；
* **V1.0** 形成双宿主首发，建立 source/runtime 边界、spec drift 治理与基础 Agent Trace；
* **V2.0** 进入扩展，对接外部知识与 AgentOps 生态。

这样拆分的好处是：

* 避免一期做成大而全；
* 保持 Light contract + Explicit boundaries + Scripts prepare, LLM decides 的核心哲学；
* 让 spec-first 使用模式尽快落地；
* 把差异化壁垒（source/runtime 边界、双宿主、artifact provenance、review→knowledge 闭环）放在 V1.0 首发，不被竞品时间差吞掉；
* 让软件交付升级目标逐步兑现，而不是停留在理念层。

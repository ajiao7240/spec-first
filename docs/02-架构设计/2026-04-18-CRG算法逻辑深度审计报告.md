## 一、关键错误（Critical Bugs）—— 会污染产出事实

### ⚠️ 1. `detectFlows` 的 criticality F2 `depth_score` —— 降级为 P2 设计质疑

**位置**：`src/crg/flows.js:114`

```js
const depthScore = Math.min(flowNodeIds.length / 20, 1.0);
```

**定性更正（原 P0 降为 P2）**：

注释原文是"F2 depth_score (0.20)：BFS 节点数近似深度"——实现**与注释一致**，是**显式的用节点数做深度代理**，不是"用错变量"。`bfsFlow` 返回的 `reachedDepth` 已用于写入 `flows.depth` 列（审计/查询用途），不是被丢弃。

**值得讨论的点**（非 bug，是启发式权衡）：
- 真实 depth 取值域只有 0-5（maxDepth=5），区分度只有 6 档
- 节点数 0-20 有 21 档，在 100 条 flow 间排序时分辨率更高
- 两种口径都不完美：节点数偏向 fan-out 宽的浅流，真实 depth 偏向瘦长深流

**建议**：改为设计讨论。如果要改，可考虑并存两个因子 `depth_factor + size_factor` 各占一半权重；或把 `depth_score` 改名为 `breadth_score` 让语义对齐。

### 🔴 2. `surprising_connections` 的 F3 `cross_community` 权重与文档不符且被门控

**位置**：`src/crg/analyze.js:122-127`

```js
const crossCommunity = src.community_id !== tgt.community_id;
if (crossCommunity && (crossLanguage || peripheralToHub)) {
  score += 30;
  reasons.push('cross_community');
}
```

函数头注释写的是"F3. cross_community (40分)"，代码给 30 分，且**必须先 F2 或 F4 触发才加**。

**后果**：
- "惊喜连接"的核心语义（跨社区调用）单独发生时得 0 分
- 最高分只到 90（不是 100），`Math.min(score, 100)` 永远达不到上限
- score ≥ 30 阈值下：cross_community 孤立出现**直接被过滤**，而 peripheral_to_hub + confidence_weight（20+10=30）却能通过 → 噪声多、真信号少

**修法**：解除门控、F3 独立加 40 分、过滤阈值调到 40 保持"惊喜"稀有性。

### 🔴 3. `analyze.js` 计算 `in_degree` 污染了结构边

**位置**：`src/crg/analyze.js:75-78` 与 `:167-170`

```js
// surprisingConnections
for (const edge of edges) { // ← 用的是全部 edges
  inDegree.set(edge.target_id, (inDegree.get(edge.target_id) || 0) + 1);
}

// godNodes
SELECT ... COUNT(e.source_id) AS in_degree
FROM nodes n LEFT JOIN edges e ON e.target_id = n.id
```

文件头明确定义 `STRUCTURAL_EDGE_KINDS = {imports_from, contains, defined_in}` 要"过滤结构边"，但 `in_degree` 计算两处都没过滤。`contains` 是 module → function 的强关联边，**每个 function 都天然 +1 in_degree 噪声**；`defined_in` 同理；`imports_from` 跨文件时也会 target 到 module 节点。

**后果**：
- `god_nodes` 的 `in_degree` 系统性偏高，所有 function 都至少 +1 噪声
- `hubThreshold` 被结构边撑高 → `peripheral_to_hub` 的 F4 假阴率上升
- class 节点因有多个 method `contains` 边汇入被误判 god node

**修法**：两处都加 `WHERE kind NOT IN ('imports_from','contains','defined_in')`。

### 🔴 4. `writeCommunities` Pass3 在无内部边时把社区切成 N 个单点

**位置**：`src/crg/communities.js:233-298`

当一个目录社区 `n > threshold && n >= 4` 但 `intra_edges == 0`（例如纯配置目录、国际化资源、无相互引用的 DTO 集合）：
- `intraAdjMap[communityId]` 是空 Map
- `bfsComponents` 对每个节点返回独立连通分量 → `components.length === n`
- 走 else 分支 → 生成 `n` 个形如 `communityId/0`, `communityId/1`, ... 的**单节点子社区**

**后果**：
- 社区表被垃圾 ID 淹没，下游 `crg community --id=X/42` 返回单文件社区毫无价值
- `testing_surface` 的"每社区至少 1 subject"规则被这堆单点社区稀释预算

**修法**：
```js
// 只有有意义的拆分才执行
if (components.length > 1 && components.some(c => c.size >= 2)) {
  // 拆分（可选：将 size<2 的零散节点合并回父社区）
} else {
  baseCommunity.health_note = 'oversized, no split boundary found';
  finalCommunities.push(baseCommunity);
}
```

### 🔴 5. `writeCommunities` 密度公式分母错误

**位置**：`src/crg/communities.js:201-202`

```js
const totalPossible = n * (n - 1);
const density = stats.intra_edges / Math.max(totalPossible, 1);
```

`moduleEdges` 已经做了无向去重（`seenModuleEdgePairs`，`communities.js:129-141`），即每对文件只计一次。无向图的最大可能边数是 `n*(n-1)/2`，不是 `n*(n-1)`。

**后果**：所有社区的 density **系统性低估 2 倍**。完全图（intra=n*(n-1)/2）在错误分母下得 density=0.5，仍能通过 `>0.3` 阈值进入 healthy；但真实密度 0.4-0.5 的内聚社区（显示值 0.2-0.25）会被错判为 scattered/fragmented。真实影响是 **healthy 门槛被相对抬高**，而非"不可达"。子社区拆分时（`communities.js:252`）同样有此错。

**修法**：
```js
const totalPossible = n * (n - 1) / 2;
```

### 🔴 6. `detectFlows` 的 entry 判定方法学薄弱

**位置**：`src/crg/flows.js:164-170`

```sql
SELECT id, name FROM nodes
WHERE kind != 'module'
  AND id NOT IN (SELECT target_id FROM edges WHERE kind = 'calls')
```

"无 incoming calls 即 entry"作为**候选判定**，存在两类系统偏差：
- **误报**：fan-in=0 且**仍有出边**的 helper / dead-code 会成为候选（`flows.js:194-198` 已过滤"完全孤立无出边的叶节点"，但对"有出边的死代码链"不过滤）
- **漏报**：框架分发的真实 entrypoint（Express handler、Rails action、Django view）通常不会被静态 calls 边命中，反被漏掉

**与 Full 模式 §1.6 校验 6 的关系（更正）**：

原报告把 detectFlows 直接定为"§1.6 校验 6『entrypoints 数组不得为空』的根因"，归因过强。实际情况是：

1. SKILL 文本 `SKILL.md:223` 首要嫌疑**归因于 client 侧读字段错误**（`flow_id` 误读为 `id`）
2. `phase1-crg-extraction.md:84` 已有独立的"crg flows 零结果 → Glob fallback"分支处理空集情况
3. detectFlows 的 entry heuristic 偏弱只是**压力来源之一**，不是该校验存在的直接触发原因

**方向性修法**（不改变以上定性）：Entry 判定需要融合：
1. 静态无 incoming calls（当前）
2. 名称/路径启发（`/routes/`、`handlers/`、`main.go` 等）
3. 装饰器/注解（`@app.route`、`@RestController`）
4. 框架特征 symbol（`app.listen`、`urlpatterns`）

无法完美静态判定时，应在 envelope 标 `confidence: Inferred` + `inference_reason: "static-fan-in-zero"`。

## 二、算法缺陷（Significant Flaws）—— 降低事实质量

### 🟠 7. `computeFlowCriticality` F3 `security_score` 关键词过泛

**位置**：`src/crg/constants.js:7-12`

```js
const SECURITY_KEYWORDS = new Set([
  'auth', 'login', 'password', 'token', 'session', 'crypt', 'secret',
  'credential', 'permission', 'sql', 'query', 'execute', 'connect',
  'socket', 'request', 'http', 'sanitize', 'validate', 'encrypt',
  'decrypt', 'hash', 'sign', 'verify', 'admin', 'privilege',
]);
```

`request` / `http` / `query` / `connect` / `execute` / `validate` 在任何 Web 项目都遍地都是。子串匹配（`n.name.toLowerCase().includes(kw)`）会把 `requestBody`、`fetchUserRequest`、`queryDb`、`validateForm` 全打进去。

**后果**：`security_score` 在典型 Web 项目里几乎永远 ≥ 0.5，贡献稳定的 12.5 分虚假 criticality。

**修法**：分层关键词集
- **强特征**（直接加分）：password / credential / crypt / decrypt / encrypt / sign / hash / token / secret
- **上下文弱特征**（需要 package/路径限定）：request / http / sql / connect / execute

### 🟠 8. `computeFlowCriticality` F4 `test_gap` 语义完全错了

**位置**：`src/crg/flows.js:122-124`

```js
const testCount = nodeRows.filter(n => n.is_test).length;
const testGap = 1 - (nodeRows.length > 0 ? testCount / nodeRows.length : 0);
```

这在算"flow **自身**包含多少测试节点"。但业务 flow 沿 `calls` 边 BFS 出来，几乎**永远不会**包含测试函数（测试函数不会被业务代码调用）。所以 `testCount = 0`、`testGap = 1`、永远满分 15。

真正想表达的是"**有没有测试覆盖该 flow**"——这需要 query `tests_for` 反查，不是在 flow 节点集合里找 is_test。

**后果**：F4 变成恒定 +15 的噪声常数项，对 criticality 排序毫无贡献。

**修法（更正）**：
- 方案 A：复用 `src/crg/cli/query.js:213` 现有 `tests_for` pattern 的推断逻辑——以 `imports_from` 边 + `nodes.is_test=1` 反查被测符号，而非假设存在 `tests_for` 持久化边（该边类型不存在，`tests_for` 只是 query 层 pattern）
- 方案 B：直接删除此因子，把权重重分配给 F1 + F2

> 原报告建议的 `SELECT ... WHERE kind='tests_for'` 与代码事实不符，已更正。

---

### 🟠 9. `surprising_connections` 的 `peripheral_to_hub` 判定过窄

**位置**：`src/crg/analyze.js:114-116`

```js
const srcDeg = inDegree.get(edge.source_id) || 0;
const tgtDeg = inDegree.get(edge.target_id) || 0;
const peripheralToHub = srcDeg === 0 && tgtDeg >= hubThreshold;
```

只有 `srcDeg === 0`（严格零）才算 peripheral。但实际 entry 节点常被 1-2 个 router/wrapper 调用。

**修法**：放宽到 `srcDeg <= 2` 或 `srcDeg < hubThreshold / 10`。

### 🟠 10. `resolveEdges` 相对路径扩展名列表不全

**位置**：`src/crg/graph.js:305`

```js
for (const suffix of ['', '.js', '.mjs', '.ts', '/index.js']) {
```

缺 `.cjs` / `.tsx` / `.jsx` / `.d.ts` / `/index.ts` / `/index.tsx`。React 项目 `require('./Button')` 指向 `Button.tsx` 会失败 → 计入 unresolved → `unresolved_rate > 0.1` 的 warning 误触。

**修法**：
```js
const SUFFIX_CHAIN = [
  '', '.js', '.mjs', '.cjs', '.ts', '.tsx', '.jsx',
  '/index.js', '/index.ts', '/index.tsx',
];
```

### 🟠 11. `detectFlows` entry 查询无确定性顺序

**位置**：`src/crg/flows.js:164-170`

SELECT 没有 ORDER BY。`flowIndex >= 100` 截断时，**不同 build 选出的 flows 不稳定**——`crg flows` 输出、SKILL B-Round2 `top-5 criticality` 都被非确定性污染。

**修法**：加 `ORDER BY id ASC`，或先对所有 entry 算 criticality 再排序截取 top-100。

---

### 🟠 12. `bfsFlow` maxNodes=20 截断真实业务流程

Express middleware 链或 Spring Controller 调用栈常有 10-30 个节点，5 层以上深度。当前硬编码 `maxDepth=5 / maxNodes=20`，一个真实 flow 会**在截断点上输出局部前缀**——后续节点永远不会被 BFS 发现。

**方向**：
- 按 criticality 预算动态调（例如发现 flow 的"骨架节点"后允许更深展开）
- 显式在 envelope 标 `truncated: true`
- 为 CLI 流量高的业务 flow 提供 `--max-depth` / `--max-nodes` 可配置上限

## 三、设计缺陷（Design Issues）—— 需架构级讨论

### 🟡 13. Pass1 目录合并规则"单文件目录 → (root)"过度合并

**位置**：`src/crg/communities.js:98-103`

1 个文件的目录合并到 `(root)`。但一些独立而重要的顶层工具（例如 `banner.ts`, `version.ts`, `cli.ts` 各自独立）会被全部丢进 `(root)` 形成"杂物社区"。

**更好策略**：只合并"完全孤立（无跨社区边）的单文件目录"。

---

### 🟡 14. 社区检测完全基于目录而非图结构

**根本性问题**：当前 3-Pass 算法本质是"**目录分桶 + 健康评估 + 连通性精化**"，并非真正的社区检测（Louvain / Leiden / Label Propagation）。

**代价**：
- 物理目录结构好 → 社区结果好
- 目录结构混乱（遗留项目、monorepo 把多个项目放 `packages/*`）→ 社区无意义
- 跨语言项目（如 `backend/` + `frontend/` 共用类型）的共享域永远不会被识别为同一社区

**收益**：
- O(V+E) 速度
- 零随机性
- 结果可解释
- 符合 SKILL §14.5 规格要求

**建议演化路径**：保留目录 Pass1 作为 seed → Pass3 从 BFS 连通分量**升级为 Louvain 单遍迭代**（社区规模上界约束下只跑 1-2 轮）→ 在保持可解释的同时吸收真实图结构。

---

### 🟡 15. `hubThreshold` 使用节点数分位而非度分布分位

**位置**：`src/crg/analyze.js:82-84`

```js
const sortedDegrees = Array.from(inDegree.values()).sort((a, b) => b - a);
const hubThresholdIdx = Math.max(0, Math.floor(sortedDegrees.length * 0.1) - 1);
const hubThreshold = sortedDegrees.length > 0 ? (sortedDegrees[hubThresholdIdx] || 1) : 1;
```

度分布通常极度长尾（power law）。用"排名前 10% 的度值"做阈值时，在 10000 节点的大仓库里可能 top-10% 处度值已经只有 3——阈值过低。

**更稳健做法**：均值 + K·标准差，或度值 z-score > 2。

---

### 🟡 16. `godNodes` 顶部硬切 `Math.min(10, ceil(n*0.05))`

**位置**：`src/crg/analyze.js:181`

大仓库里真实 god node 可能有 50 个，只取前 10 丢失信息；小仓库 < 20 节点永远输出 1 个（含义稀薄）。

**修法**：按度值断崖（degree gap）或对数尺度选取，而非固定百分比。

---

### 🟡 17. `chunking.js` 按固定行数切分，非 AST 感知

**位置**：`src/crg/chunking.js:15-22`

```js
const lines = rawText.split(/\r?\n/);
const slice = lines.slice(index * maxLines, (index + 1) * maxLines).join('\n').trim();
```

把 300 行函数切成 4 个 75 行 chunk，**跨越表达式/循环边界**。下游向量检索的 embedding 质量会打折——语义单位被生硬切开。

**更好做法**：
- 以 AST 节点（function/method/class）为第一优先切分单位
- 仅当单节点 > maxLines 才按 if/for/switch 子结构二次切分
- tree-sitter 已经给出 AST，几乎零成本复用

---

### 🟡 18. `incremental.js` 每次 build 重算全量 SHA

`computeFileSHA` 每次都 `readFileSync` 整个文件。大仓库（10 万文件）每次 build 重读 + 重 hash，单独耗时可能 30-60s。

**优化**：
- mtime + size 快速筛选 → 只对可疑文件算 SHA
- `fingerprints` 表加 `mtime_ms` / `size` 列

---

### 🟡 19. Phase 0.3 stale 警告"不阻断流程"

SKILL 合约里 stale 只打 warning，Phase 1 仍会读**过期**的 `graph.db` 产生过期事实。这在语义上已经违反了"Full 模式 = high confidence"的承诺。

**建议**：stale 检测触发时，强制降级到 Enhanced/Basic（或要求先 rebuild）。当前设计把"data freshness"和"analyzer mode"解耦——但事实上 graph 过期等价于 graph 不可靠。

## 五、正确的设计（Validated as Correct）

为平衡视角，以下是**经得起检验的部分**：

- ✅ **Generation 隔离机制**（`generations/*`）：写新 generation → health 检查 → 仅在健康时升 current pointer；reader 通过 `resolveActiveGraphDb` 优先 current → last-known-good。这是标准的 blue-green DB 切换，正确。

- ✅ **`resolveEdges` 多阶段 + 缓存用 `Object.create(null)`**：防原型链污染，用 `!== undefined` 区分"未查询"与"已知失败"——扎实的 JS 陷阱规避。

- ✅ **`writeCommunities` 外键安全写入顺序**（UPDATE NULL → DELETE → INSERT → UPDATE → 传播）：正确处理了 `ON DELETE SET NULL` 与事务。

- ✅ **ObjC `#import "file.h"` 的 basename 模糊匹配 + "目录公共前缀最长" proximity heuristic**：在 C 家族缺乏路径信息的前提下，合理的近邻启发式。

- ✅ **PRAGMA 配置**：WAL + synchronous=NORMAL + 64MB cache + 256MB mmap，是 SQLite 在批量写 + 并发读场景下的标准优化。

- ✅ **FTS5 独立存储而非 `content=nodes`**：避免了 content 表列名强绑定，迁移时 DROP-rebuild 语义清晰。

- ✅ **SENSITIVE_PATTERNS 硬规则不可被白名单绕过**：安全兜底设计正确。

- ✅ **unresolved_edges 审计表 + `top_kinds/top_source_files/samples` 摘要**：治理向设计，便于定位解析器问题。

## 六、总体评价

### 工程度：中上

PRAGMA / generation / 外键事务 / 缓存防陷阱 / 输入收敛的 SENSITIVE_PATTERNS / unresolved 审计表 —— 这些系统工程要素都做到位。

### 算法严谨度：中偏下

两处核心算法存在**需要修的真 bug**：
- depth_score 用错变量
- cross_community 权重错且被门控
- in_degree 没过滤结构边
- Pass3 边集为空时切碎社区
- 密度分母错误

还有一批**语义上下游不自洽**：
- entry 定义过窄且漏掉框架分发
- F4 test_gap 语义错位
- F3 security_score 关键词过泛

### 根本性问题

**"社区检测"和"entry 探测"本质上不是图算法，而是"目录启发式 + SQL 过滤"**。这与 SKILL 声称的"Full 模式 = AST 级 high confidence"存在 gap。

当前补丁方式是 Phase 1.6 校验 + `generation_errors` 降级标注，**但算法本身没被修好**。这种"在合约层校验，不在算法层修复"的做法会让补丁不断累积，长期维护性下降。

## 八、修复的回归风险矩阵

| 修复项 | 风险 | 测试需求 |
|---|---|---|
| P0-1 depth_score | 低，纯字段替换 | 单元测试覆盖 flow 评分排序稳定性 |
| P0-2 cross_community 权重 | 中，下游 surprising 列表会变 | 基线 fixture 对比 + 阈值调优 |
| P0-3 in_degree 过滤 | 中，god_nodes 列表会变 | 基线 fixture 对比 |
| P0-4 Pass3 保护 | 低，只影响边集为空的超大社区 | 新增空边集社区用例 |

**关键原则**：凡是改变输出 schema 或稳定标识的修复，都应该升级 `schema_versions.<analyzer>` 从 `v1` → `v2`，让 `artifact-manifest.json.inputs.analyzer_versions` 能感知差异。

## 九、结论

这些修正如果落地，可以把 CRG 从"**有偏的启发式图数据库**"推进到"**可作为事实真源的 AST 知识库**"。

当前 CRG 在工程度上已经具备生产级基础（Generation / 事务 / 缓存），但算法层的若干逻辑错误和语义错位意味着：

- **对大多数场景仍能给出可用信号**（尤其是小-中型单语言项目）
- **但在跨语言 monorepo、遗留代码、框架重路由场景下置信度会显著下降**
- **当前合约层（Phase 1.6 校验 + fallback 标注）已经在为这些问题做兜底**，但这是症状治疗而非根因治疗

建议按 P0-P1-P2-P3 顺序推进修复，每个 P 级完成后重新跑一轮对比基线（建议在 `spec-first` 自举仓库 + 外部 OSS 仓库上做双重基线），量化置信度提升。

# 移动端交互审查方案（历史设计输入）

> 当前状态：历史设计输入，已被 `spec_app_consistency_audit_升级技术方案.md` 吸收和收敛
> 最新落地依据：`docs/02-架构设计/spec_app_consistency_audit_升级技术方案.md`
> 文档定位：用于保留移动端交互审查、Mobile UX、runtime verification 和分期拆分的设计背景，不作为 runtime execution source-of-truth
> Skill 名称：`spec-app-consistency-audit`
> 设计状态：superseded design input / historical reference
> 默认模式：静态审查优先，不默认启动真机、模拟器、打包或云真机
> 核心范式：Skill 作为 Orchestrator；脚本产出 Impact Facts；LLM 生成 Audit Plan；按需调度专家 Agent；Evidence Gate 控制输出质量

---

## 1. 最终结论

`spec-app-consistency-audit` 值得作为独立 Skill 保留，但它不应该做成每次全量运行的重型平台。

最终形态是：

```text
一个主 Skill
  +
Deterministic Impact Facts
  +
LLM Audit Planner
  +
按需专家 Agent 调度
  +
Deterministic Evidence Gate
  +
LLM Evidence Auditor
  +
Report Writer
```

它与 `spec-code-review` 的关系不是替代，也不是合并，而是协作：

```text
spec-code-review
  = 通用代码变更审查 orchestrator

spec-app-consistency-audit
  = App 产品-设计-架构-代码一致性专项审查 workflow

两者通过 impact-facts / audit-plan / headless / report-summary 协作
```

一句话：

> `spec-code-review` 负责通用 PR / diff 审查；`spec-app-consistency-audit` 负责 App 专项一致性审查；当 code-review 命中 App/KMP/Figma/PRD/埋点/i18n/行业关键路径时，再推荐或委托 app-audit 运行专项审查。

这个定位与最新方案中的核心模式一致：Skill 作为 Orchestrator，脚本产出 Impact Facts，LLM 生成 Audit Plan，按需调度专家 Agent，并通过确定性 Evidence Gate 与 LLM Evidence Auditor 控制输出质量。

---

## 2. 核心目标

`spec-app-consistency-audit` 解决的不是“App 能不能跑”，而是：

> **App 在真机 / 模拟器运行之前，产品需求、Figma 设计、KMP/Clean 架构、代码实现、交互状态、埋点、国际化和行业关键规则之间是否一致。**

核心目标：

```text
1. 在不默认启动真机 / 模拟器的前提下，提前发现 App 交付断点。
2. 将 PRD、Figma、源码、架构、埋点、i18n、行业规则编译成可审查证据链。
3. 根据改动影响面动态选择专家，而不是每次全量专家审查。
4. 所有问题必须有项目证据，避免 AI 幻觉和 rule-pack-only 结论。
5. 将问题分为 confirmed / candidate / advisory / runtime verification suggestion。
6. 与 spec-code-review 协作，但不污染其通用 code review 心智模型。
7. 输出可定位、可修复、可验证、可沉淀的 App 专项审查报告。
```

---

## 3. 非目标

这个 Skill 不做以下事情：

```text
1. 不替代 spec-code-review。
2. 不替代真机测试、模拟器测试、Maestro、Appium、BrowserStack。
3. 不自动打包、不自动启动 App、不默认访问云真机。
4. 不直接修改产品源码。
5. 不自动写 repo-profile.yaml。
6. 不自动写入 durable standards。
7. 不把行业 rule-pack 当作 confirmed issue 的唯一证据。
8. 不把 Figma node id / URL 当作设计证据。
9. 不把 PRD 缺失、Figma 缺失、任务文档缺失直接当作 run failed。
10. 不做中心化规则引擎，不做 hard gate policy engine。
```

## 4. 与 `spec-code-review` 的边界

### 4.1 `spec-code-review` 的职责

`spec-code-review` 面向通用代码变更审查，适合处理：

```text
代码正确性
测试覆盖
安全风险
性能风险
可靠性
API 合约
通用可维护性
diff 风险
PR 合并前检查
```

### 4.2 `spec-app-consistency-audit` 的职责

`spec-app-consistency-audit` 面向 App 专项一致性审查，适合处理：

```text
PRD ↔ Figma ↔ Code 一致性
Figma 状态是否进入 UiState
设计组件是否映射代码组件
KMP + Clean Architecture 是否承载业务规则
Android / iOS 行为是否漂移
移动端交互风险
App 工程质量
组件化 / 模块化 / 复用
埋点完整性
国际化完整性
行业关键流程审查
运行验证建议
```

### 4.3 为什么不能直接并入 code-review

如果直接把 app-audit 并入 code-review，会出现：

```text
1. code-review 入口变重。
2. PRD / Figma / 行业规则等非 diff 输入污染通用 review 流程。
3. App 专项 artifacts 混进 code-review artifacts。
4. 用户想单独做 App 一致性审查时没有清晰入口。
5. App 专项能力无法独立演进。
6. 专家数量过多，容易每次开“专家大会”。
```

因此，最终设计是：

```text
code-review 负责通用审查；
app-audit 负责 App 专项审查；
code-review 可以 recommend 或 headless 调用 app-audit；
两者 artifacts 独立；
两者通过 summary / handoff 协作。
```

## 5. 总体架构

最终流水线：

```text
Host app-consistency-audit entrypoint
        ↓
Argument Parsing / Mode Detection
        ↓
Scope Detection / Preflight
        ↓
Impact Facts Builder / 确定性改动事实与候选 signals
        ↓
Contract Extraction / 契约抽取
        ↓
App Audit Context / 合并候选事实
        ↓
LLM Audit Planner / 审查计划生成
        ↓
Selected Expert Agents / 按需专家审查
        ↓
Normalize / Enrich Issue Candidates
        ↓
Issue Synthesis / Dedup
        ↓
Deterministic Evidence Gate / 结构化证据校验
        ↓
LLM Evidence Auditor / 语义证据审计
        ↓
Validation Pass / 独立复核外部化问题
        ↓
Final Partition + Code Review Handoff
        ↓
Report Writer + Headless Envelope
        ↓
Run-scoped Issues + Report + Summary + Regression Suggestions + Writeback Preview
```

设计边界：

```text
1. Gate 是确认点，不是中心化状态机。
2. 所有脚本阶段只处理确定性事实、结构校验和 artifact 写入。
3. 所有语义取舍由 LLM planner / expert / evidence auditor 完成。
4. report-only 的 no-write contract 高于任何 artifact 章节描述。
```

你上传的方案已经明确了这条权威总流水线，并强调局部章节只能补充字段和失败语义，不能重新定义顺序。

## 6. Skill / Script / Agent / Rule Pack 职责边界

### 6.1 Skill 职责

`spec-app-consistency-audit` 是 workflow orchestrator。

负责：

```text
1. 读取用户输入。
2. 运行 preflight。
3. 组织 diff / changed files / changed symbols 的事实收集。
4. 抽取 PRD / Figma / Code / KMP / Component / Analytics / I18n / Industry facts。
5. 合并 app-audit-context.json。
6. 让 LLM Audit Planner 基于 facts 生成 audit-plan.json。
7. 向专家分发最小必要上下文。
8. 收集专家 findings。
9. normalize / enrich issue candidates。
10. 合成、去重和冲突记录。
11. 调用 deterministic evidence gate。
12. 调用 LLM Evidence Auditor。
13. 按条件运行 Validation Pass。
14. 调用 Report Writer。
15. 输出 run-scoped artifacts。
16. 生成 writeback preview。
```

### 6.2 Script 职责

Script 是确定性执行器，不做最终语义判断。

负责：

```text
1. 文件发现、路径解析、hash / freshness 计算。
2. preflight 与 degraded modes。
3. git diff / changed files / changed symbols 候选事实。
4. PRD / Figma / Code / KMP / Component / Analytics / I18n / Industry 候选 contract 抽取。
5. impact-facts.json 和 app-audit-context.json 写入。
6. JSON schema 校验。
7. deterministic evidence gate：
   - 空 evidence
   - rule-pack-only evidence
   - 缺少 provenance
   - data_sensitivity 缺失
   - 未确认 industry confirmed issue
8. run_id、artifact 目录和 summary 指针写入。
```

不能：

```text
1. 决定业务影响是否成立。
2. 决定专家组合是否足够。
3. 将行业候选升级为 confirmed 行业事实。
4. 判断严重等级和修复优先级。
5. 用规则引擎替代专家 review。
```

### 6.3 Agent 职责

Agent 是专业审查执行者。

负责：

```text
1. 在自己的领域内审查事实。
2. 输出有证据的问题。
3. 给出影响分析和修复建议。
4. 标记置信度。
5. 标记是否需要运行验证。
6. 不越权审查无关领域。
```

### 6.4 Rule Pack 职责

Rule Pack 是知识增强材料，不是硬编码规则引擎。

负责：

```text
1. 提供领域术语。
2. 提供关键路径 checklist。
3. 提供常见风险。
4. 提供严重等级参考。
5. 提供行业专项规则。
```

不能：

```text
1. 单独作为 issue 证据。
2. 自动强制失败。
3. 代替 LLM 判断。
```

## 7. 运行模式与参数协议

### 7.1 Canonical Token 参数

建议使用 token 风格，便于 Claude / Codex 双宿主统一：

```text
mode:headless
mode:report-only
base:<sha-or-ref>
source:<repo-relative-path>
prd:<repo-relative-path>
figma-context:<repo-relative-path>
figma-ref:<id-or-url>
industry:<name>
depth:deep
from:code-review
tech-plan:<repo-relative-path>
task-doc:<repo-relative-path>
```

### 7.2 Mode Contract

#### default / interactive

```text
允许写 run-scoped artifacts。
允许在必要时询问用户。
允许在用户授权下 materialize Figma context。
不允许自动 apply writeback。
不允许改源码。
```

#### mode:headless

```text
程序化调用模式。
禁止用户提问。
必须写 run-scoped artifacts。
必须返回 headless envelope。
不得切换 checkout。
不得主动拉取 Figma。
不得 apply writeback。
```

#### mode:report-only

```text
严格只读模式。
不写任何 artifact。
不 materialize Figma。
不写 preview。
只能基于当前响应输出报告。
如果 extractor 只能写 artifact，则该能力降级。
```

### 7.3 冲突规则

```text
1. 多个 mode 同时出现时停止，不调度专家。
2. mode:headless 缺少可确定 diff scope 时停止并返回 failed envelope。
3. mode:report-only 不允许写 run artifact、input materialization 或 preview。
4. from:code-review 不允许切换 shared checkout。
5. figma-ref 在 headless/report-only 中不能自动 materialize，只能产生 degraded mode。
6. 宿主 CLI 支持 --prd 等选项时，必须先归一化为 canonical token。
```

## 8. Scope Detection 与 Preflight

### 8.1 Scope Detection 职责

```text
1. 解析 source root，并拒绝 repo 外路径或 symlink escape。
2. 解析 base:<ref>。
3. 收集 changed files、changed symbols、untracked files。
4. 标记 untracked files 是否纳入 scope。
5. 检查 PRD、Figma materialized context、App source、KMP、analytics、i18n、design-system 是否可读。
6. default/headless 生成 run-scoped preflight.json 和 degraded_modes。
7. report-only 只保留内存 preflight facts，并在输出中呈现 degraded_modes。
```

### 8.2 输入期望等级

输入分为四类：

```text
required:
  当前 mode 的硬前提。缺失即 failed。

expected:
  用户显式提供、父 workflow 声明 expected_inputs、或当前审查目标明确依赖的输入。缺失通常 degraded。

opportunistic:
  有则增强审查，缺失不应把 run 标记为 degraded，只应体现在 coverage gap。

not_applicable:
  当前项目形态或用户目标不需要该输入。缺失不记录 degraded。
```

### 8.3 Run Status

```text
failed:
  - required input 缺失或不可读
  - mode 冲突
  - headless/from:code-review 无法确定 required diff scope
  - required artifact schema invalid 且无法降级

degraded:
  - expected input 缺失
  - provider / graph / validator / expert 失败但可继续
  - source scan truncated 或 fallback 到 bounded source scan

complete:
  - required inputs 可用
  - expected inputs 满足或不存在
  - opportunistic inputs 缺失只形成 coverage gap，不降级 run_status
```

### 8.4 降级原则

缺少 PRD / Figma / 技术方案 / 开发任务文档，不应默认中断。

```text
1. 缺 PRD：
   不能输出 “违反 PRD” confirmed issue。
   可以继续输出源码静态问题。

2. 缺 Figma：
   不能输出 “违反 Figma” confirmed issue。
   可以继续输出 UI 源码静态问题。

3. 缺技术方案：
   不能输出 “违反技术方案” confirmed issue。
   可以继续输出 architecture_static finding。

4. 缺任务文档：
   不能输出 “任务未完成 / 偏离任务” confirmed issue。
   可以继续输出 implementation_quality finding。

5. 缺 Graph：
   不能输出全调用链 confirmed impact。
   可以输出 bounded source scan finding。

6. 未确认 industry：
   只能 advisory / candidate。
   不能输出行业 confirmed issue。
```

你提供的方案已经明确了类似的输入缺失处理矩阵，例如 PRD/Figma 缺失不应中断，Graph 不可用时回退 bounded source scan，industry 只有术语候选时只能 advisory。

## 9. Impact Facts Builder

### 9.1 目标

脚本产出确定性改动事实和候选 signals，但不做最终语义判断。

```text
Impact Facts Builder
  = 脚本产出确定性改动事实和候选 signals

LLM Audit Planner
  = 基于 facts 和用户意图生成 audit-plan.json
```

### 9.2 输入

```text
1. git diff
2. changed files
3. changed symbols
4. import / dependency changes
5. route / screen / ViewModel / UseCase / Repository 影响面
6. PRD 是否变更
7. Figma context 是否提供或 materialized
8. analytics / i18n 文件是否变更
9. industry profile / rule-pack signals
10. graph artifacts，如果存在
11. user tokens，例如 depth:deep / industry:<name>
```

### 9.3 上下文扩展

不能只看 diff 附近几行。

```text
changed lines
  → enclosing function / class
  → related UiState / UiEvent / Effect
  → related UseCase / Repository
  → related Screen / Route
  → related analytics / i18n / Figma state
  → related product journey / industry critical flow
```

### 9.4 输出

```text
.spec-first/app-audit/runs/<run-id>/impact-facts.json
```

最小字段：

```json

  "artifact_id": "impact-facts",
  "generated_at": "2026-05-01T00:00:00.000Z",
  "source_inputs": [],

  "contract_status": "candidate",
  "data_sensitivity": "internal",
  "changed_files": [],
  "candidate_signals": [],
  "available_context": {
    "prd": false,
    "figma_materialized": false,
    "graph_artifacts": false,
    "analytics": false,

}
```

## 11. LLM Audit Planner

### 11.1 职责

LLM Audit Planner 基于 `impact-facts.json`、`app-audit-context.json` 和用户模式生成 `audit-plan.json`。

它负责：

```text
1. 解释改动影响面。
2. 选择 selected_experts。
3. 记录 skipped_experts。
4. 标记专家模式：focused / deep / advisory / required。
5. 给出选择理由。
6. 遵守 degraded_modes 与 conclusion_caps。
```

### 11.2 Planner Guardrail

LLM 可以决策，但必须在轻量 guardrail 内。

建议 schema 约束：

```json
{
  "allowed_experts": [
    "Product Expert",
    "Design Expert",
    "Mobile UX Expert",
    "KMP Architect",
    "Component Module Expert",
    "Engineering Quality Expert",
    "Analytics Expert",
    "I18n Expert",
    "Industry Expert",
    "Regression Expert",
    "LLM Evidence Auditor",
    "Report Writer"
  ],
  "must_select": [
    "Engineering Quality Expert",
    "LLM Evidence Auditor",
    "Report Writer"
  ],
  "max_selected_experts_default": 5
}
```

### 11.3 输出

```text
.spec-first/app-audit/runs/<run-id>/audit-plan.json
```

必须包含：

```text
schema_version
artifact_id
source_inputs
mode
planner_decisions
selected_experts
skipped_experts
data_sensitivity
contract_status
```

## 12. 专家 Agent 调度

### 12.1 永远启用

#### Deterministic Evidence Gate

```text
1. 校验 issue JSON schema。
2. 拒绝缺少 evidence / provenance / data_sensitivity / runtime_verification 的 issue。
3. 拒绝 rule-pack-only evidence 的 confirmed issue。
4. 拒绝未确认 industry profile 产生的行业 confirmed issue。
5. 规范化 confidence、impact、recommendation、related_rule_packs。
6. 输出 evidence-gate-result.json。
```

#### LLM Evidence Auditor

```text
1. 复核候选问题。
2. 检查证据是否支持标题、严重等级、影响和建议。
3. 删除语义证据不足的问题。
4. 将弱证据问题降级为 candidate / advisory / follow-up。
```

## 15. Claim Type Evidence Matrix

不同 claim_type 需要不同 evidence。

| claim_type / claim family         | confirmed 所需 evidence                                  | 缺证据时最高状态                                |
| --------------------------------- | ------------------------------------------------------ | --------------------------------------- |
| `product_alignment`               | PRD/product contract + code evidence                   | candidate / advisory / out_of_scope     |
| `design_alignment`                | materialized Figma contract + code evidence            | candidate / advisory / out_of_scope     |
| `product_design_code_alignment`   | PRD + materialized Figma + code evidence               | candidate / advisory / out_of_scope     |
| `architecture_static`             | code / module / KMP contract evidence                  | confirmed                               |
| `architecture_intent_conformance` | tech-plan + code evidence                              | advisory / out_of_scope                 |
| `task_fidelity`                   | task-doc + diff/source evidence                        | advisory / out_of_scope                 |
| `analytics_static`                | analytics code/resource evidence                       | confirmed                               |
| `analytics_requirement_alignment` | PRD analytics requirement + code evidence              | candidate / confirmed when both present |
| `i18n_static`                     | user-facing code/resource evidence                     | confirmed                               |
| `industry_compliance`             | confirmed industry profile + project-specific evidence | advisory / candidate                    |
| `runtime_behavior`                | runtime validation result                              | runtime_verification_suggestion         |

规则：

```text
1. Evidence Gate 根据 required evidence sources 做确定性 reject / downgrade。
2. Evidence Gate 不判断证据语义是否足够强。
3. 证据语义充分性由 LLM Evidence Auditor 判断。
4. 专家输出 finding 时必须填写 claim_type。
5. 缺 claim_type 的 finding 不能进入 confirmed 流程。
```

## 24. 安全、隐私与执行边界

## 27. MVP 验收标准

v0.1 完成后，必须满足：

```text
1. 普通 App/KMP diff 能生成 impact-facts.json。
2. LLM Audit Planner 能生成稳定 audit-plan.json。

4. 每个 issue 都有 claim_type、affected_surface、evidence、provenance。
5. rule-pack-only issue 会被 Evidence Gate 拒绝。
6. 未确认 industry 只能 advisory。
7. 缺 PRD 不会阻断源码静态审查。
8. 缺 Figma 不会产生 design confirmed issue。
9. report-only 不写任何文件。
10. headless 不询问用户。
11. from:code-review 不切换 checkout。
12. 所有 artifact 写入 runs/<run-id>/。
13. latest-summary 只作为指针，不作为事实源。
14. CLI subprocess e2e 测试通过。
```

## 28. 最终技术结论

`spec-app-consistency-audit` 的最终正确形态是：

```text
一个 App 专项 Skill
  作为 Orchestrator
  先产出确定性 Impact Facts
  再由 LLM Audit Planner 生成审查计划
  按需调度专家 Agent
  通过 Deterministic Evidence Gate 和 LLM Evidence Auditor 控制输出质量
  最后输出 run-scoped issues / report / summary / handoff
```

它与 `spec-code-review` 的关系是：

```text
code-review 是通用代码审查入口
app-consistency-audit 是 App 专项一致性审查入口
code-review 可以根据 diff 命中情况推荐或 headless 调用 app-audit
两者不合并，但协作
```

最终一句话：

> 不要把 `spec-app-consistency-audit` 做成每次全量跑的重型平台；要把它做成 App 专项审查 orchestrator，根据改动影响面组织最小专家小组，做到轻入口、强专家、证据驱动、按需深审。

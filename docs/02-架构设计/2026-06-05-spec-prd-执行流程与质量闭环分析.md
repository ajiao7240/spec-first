## 1. 定位

`spec-prd` 是 brownfield PRD requirements workflow。它把已有系统上的增量想法、粗糙产品稿、低质量 PRD 或规划前验证请求，转成下游 `spec-plan` 能直接消费的 PRD-grade requirements。

它的核心目标不是“写更长的 PRD”，而是降低下游 planning invention：

- 先对齐当前系统事实和证据；
- 再确认产品增量的 `keep / extend / replace / remove / unknown`；
- 对低质量 PRD 做产品专家式质量诊断和优化建议；
- 最终输出标准 Markdown PRD artifact；
- 让 `spec-plan` 规划 HOW 时不再发明 WHAT。

核心边界仍是：

```text
Codebase -> Context -> Spec/PRD -> Plan -> Tasks -> Code -> Review -> Knowledge
```

`spec-prd` 只负责 Spec/PRD 节点的 WHAT / WHY / acceptance / scope / evidence，不负责 implementation plan、task breakdown、schema、API 字段或具体文件改动。

## 3. Reference 加载模型

`spec-prd` 的 entrypoint 保持轻量，具体规则按触发条件加载 4 个 reference：

```text
skills/spec-prd/SKILL.md
  |
  +-- current-state / evidence / topology / source-of-truth
  |     -> references/evidence-and-topology.md
  |
  +-- terminology / contradiction / owner decision
  |     -> references/domain-language-and-decision-ledger.md
  |
  +-- output shape / template / product lens / rewrite rules
  |     -> references/prd-output-template.md
  |
  +-- final readiness / handoff decision
        -> references/prd-readiness-lens.md
```

这保持了 “steel frame” 结构：

```text
entrypoint spine
  + evidence/topology steel
  + domain/decision steel
  + output contract steel
  + readiness steel
```

没有新增第五个 reference、schema、agent type 或独立 PRD 目录，避免 skill 再次膨胀成模板平台。

## 4. 自适应产品专家质量闭环

增强后的 refine / validate 路径增加了一个明确闭环：

```text
Existing low-quality PRD
  |
  v
Adaptive Product Expert Lens
  |
  +-- infer surface lens
  |     App / H5-PC / Admin / Backend / CLI / Mixed / Generic
  |
  +-- infer industry overlay when triggered
  |     securities / trading / credit / admin / backend platform
  |     note: overlay raises questions, not confirmed truth
  |
  +-- inspect quality dimensions
        product outcome
        current-state and code alignment
        requirement atomicity
        acceptance coverage
        metrics and priority
        domain / industry risk
        scope and handoff entropy
  |
  v
PRD Quality Diagnosis
  |
  v
Optimization Suggestions
  |
  |  original -> recommendation -> reason -> write target
  |
  v
Final PRD Rewrite
  |
  v
Readiness Lens
  |
  v
$spec-plan or revise / ask-owner / doc-review
```

这个闭环的关键不是“给 PRD 打分”，而是把诊断结果转成可写入最终 PRD 的改进：

| 阶段 | 输出 | 持久化位置 |
| --- | --- | --- |
| quality diagnosis | `quality_posture`、`top_gaps`、`rewrite_strategy` | run-local，不默认写入 artifact |
| optimization suggestions | `original -> recommendation -> reason -> write target` | closeout 或当前响应，可按需进入 PRD 对应 section |
| final rewrite | 标准 PRD sections | `docs/brainstorms/*-requirements.md` |
| unresolved blocker | 最小 owner question | `Outstanding Questions` 或 handoff |

这样既能满足“深度分析 PRD 质量”，又不会产生第二套 `docs/prds/`、质量报告 artifact 或 reviewer workflow。

## 5. Evidence 与代码分析边界

`spec-prd` 做代码深度分析时，遵守 evidence tag：

```text
confirmed-source
  repo source / tests / docs / contracts / deterministic command output

user-stated
  owner 明确说法，且未被 confirmed source 反驳

source-candidate
  rg / ast-grep / code index / knowledge-base 找到候选，但未直接读源确认

external-research
  外部来源，必须带日期和出处；只作 advisory

assumption
  安全、显式、可审查的推断
```

代码分析不等于实现规划：

```text
spec-prd may say:
  当前系统已有批量操作入口
  当前权限边界未确认
  当前 artifact consumer 是 spec-plan
  当前 source-of-truth 是 skills/spec-prd/**

spec-prd must not say:
  新增哪个函数
  改哪个 API 字段
  数据库 schema 怎么设计
  task 拆成哪些 implementation units
```

材料可信度排序：

```text
direct source/docs/tests
  > deterministic command output
  > user-stated current product decision
  > source-candidate pointer
  > external research
  > assumption
```

外部行业材料只能提出问题或触发 conditional sections，不能把行业规则写成项目确认事实。

## 6. Output Shape 决策

`spec-prd` 先决定输出形态，再写内容：

```text
                 +----------------+
                 | output_shape   |
                 +----------------+
                         |
      +------------------+------------------+
      |                  |                  |
      v                  v                  v
   bypass           compact-prd        normal-prd
 no artifact       core sections       core + triggered
 plan/work         minimal evidence    surface/domain
 handoff           and acceptance      sections
      |
      v
 topology-heavy-prd
 core + topology + surface map
 producer/consumer + source-of-truth
 negative acceptance + decision notes
```

选择原则：

- 小修小补或 implementation-ready work：`bypass`
- 小型 brownfield 增量：`compact-prd`
- 普通产品/系统增量：`normal-prd`
- workflow、contract、migration、replace/remove、source-of-truth、mixed-surface：`topology-heavy-prd`

`output_shape` 是 run-local authoring posture，不是 schema、frontmatter 或第二套 artifact taxonomy。

## 7. Readiness 决策

Readiness lens 的核心问题：

> 这份 PRD 是否已经让 `spec-plan` 不需要发明 WHAT？

执行方式：

```text
Core Pack
  current-state provenance
  change delta and boundary clarity
  planning-invention and trace risk
  wording and testability
  interaction and exception readiness

Quality Diagnosis Pack
  adaptive product lens fit
  optimization suggestion closure
  rewrite integrity

Topology Pack
  topology and surface fit
  producer-consumer and source-of-truth closure
  negative-space coverage
  framing-evidence alignment

Domain And Decision Pack
  terminology and contradiction handling
  owner-question minimality
  domain-grill and decision-note adequacy
  no context-artifact inflation

Metrics And Overlay Pack
  goal-measurability
  project-local overlay check
```

最终 outcome：

```text
ready-for-planning
  -> planning can consume without inventing WHAT

revise-prd
  -> concrete PRD gaps need fixing

ask-owner
  -> one smallest blocking product question

doc-review
  -> broad/subtle quality risk needs independent critique

route-out
  -> brainstorm / app-audit / plan / debug / work is the correct route
```

在 handoff 前必须做 entropy check：

```text
remaining WHAT decisions?
  behavior
  scope
  affected surfaces
  artifact consumers
  source-of-truth
  negative boundaries
  unresolved framing risks

if any load-bearing item remains unresolved:
  outcome = revise-prd or ask-owner
```

## 10. 关键结论

`spec-prd` 当前正确的执行模型是：

```text
低质量 PRD / 增量需求
  -> brownfield current-state evidence
  -> adaptive product expert lens
  -> quality diagnosis
  -> optimization suggestions
  -> final PRD rewrite
  -> readiness lens
  -> spec-plan handoff
```

它的设计取舍是：

- 用产品专家 lens 提升 PRD 质量；
- 用 evidence tag 防止把猜测写成事实；
- 用 output template 保持标准 PRD 形态；
- 用 readiness lens 防止把 WHAT 缺口甩给 planning；
- 不新增复杂 artifact topology，不让 skill 变成 PRD 平台。

一句话总结：

> `spec-prd` 的价值不是生成漂亮文档，而是把产品输入、行业关注点和代码现状压缩成 planning-ready 的 WHAT 事实与边界。

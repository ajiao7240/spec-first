## 1. 背景

当前 `spec-first` 已完成 `CRG` 与 `spec-graph-bootstrap` 的主要代码开发，系统已经具备以下基础：

- `CRG` 已经是可运行的本地 AST-first 代码事实索引底座
- `spec-graph-bootstrap` 已经形成 Phase 0-4 的 Stage-0 supporting workflow
- `docs/contexts/<slug>/` 已经成为 durable context assets 的承载目录
- `spec-plan`、`spec-work`、`spec-review` 已经开始接入 Stage-0 产物

但从终局目标看，当前系统还没有形成真正的闭环。当前更接近：

- 一个方向正确的索引器
- 一个方向正确的上下文生成器
- 一组尚未完全硬化的消费约定

而不是：

- 一个高质量的代码库理解系统
- 一个稳定的最小上下文分发系统
- 一个可评测、可维护、可持续演进的上下文工程底座

因此，接下来的技术方案不应继续停留在“补若干点”，而必须完成一次架构收口，明确单一正确方向。

## 2. 最终结论

本方案确定以下最终定位，不再保留其他竞争性定义。

### 2.1 `CRG` 的最终定位

`CRG` 的最终定位是：

**代码事实编译器 + 任务检索底座**

它不是：

- 单纯的代码图工具
- 更高级的 grep
- 单纯服务 review 的分析器

它负责：

1. 从仓库中编译结构化、可审计、可增量、可回滚的代码事实
2. 基于这些事实生成面向任务的最小上下文检索结果
3. 为上层 Stage-0 和 workflow 提供稳定、版本化、可解释的事实与检索接口

### 2.2 `spec-graph-bootstrap` 的最终定位

`spec-graph-bootstrap` 的最终定位是：

**Compiled Context Compiler**

它不是：

- 文档生成器
- narrative 报告流水线
- 第二套代码索引器

它负责：

1. 把 `CRG` 和仓库其他来源的事实编译成 machine-first 上下文资产
2. 生成最小上下文卡片、路由 contract、freshness/lint/provenance 资产
3. 从 machine artifacts 派生人类可读的导航文档
4. 为 `plan / work / review / verify` 提供稳定消费入口

### 2.3 系统终局目标

最终目标不是“生成更多上下文文件”，而是：

**让 Agent 在真实团队协作场景中，用更少 token、更少无关文件、更高证据召回率，稳定完成 plan / work / review / verify。**

## 3. 设计目标

本方案必须同时满足以下 6 个目标：

### 3.1 正确性

- 事实层可审计
- 产物有 schema contract
- drift 可检测
- fallback 可解释

### 3.2 可维护性

- 支持增量更新
- 支持 generation 切换
- 支持 sample 自动生成
- 支持兼容性治理

### 3.3 上下文最小化

- 不再默认整文件注入
- 按任务类型选择最小证据包
- narrative 永远服从 machine-first 资产

### 3.4 代码库理解能力

- 支持入口定位
- 支持调用链与 blast radius
- 支持测试面与风险面识别
- 支持跨模块、跨文件的结构理解

### 3.5 团队协作可扩展性

- 支持多 workflow 共享
- 支持 checked-in sample 与 runtime 产物一致
- 支持长期演进而不失控

### 3.6 可证明性

- 有 retrieval benchmark
- 有 repo QA benchmark
- 有 review benchmark
- 有 context efficiency benchmark

## 4. 非目标

本方案明确不把以下事项放入当前最终方案的主线：

- 不优先做 vector-first 大重构
- 不优先做复杂 GNN / end-to-end learned ranker
- 不继续横向膨胀 narrative 文档
- 不让 workflow 直接依赖 `CRG` SQLite 内部表结构
- 不把 Stage-0 做成第二套分析器

## 6. 关键架构决策

本节给出最终决策，不再保留备选方案。

### 6.1 决策一：先 evaluator，后大 schema

最终决策：

- `P0` 先做 evaluator
- schema 只做满足 evaluator 所需的最小扩展

原因：

- 当前系统最大缺口是消费闭环，而不是字段丰富度
- evaluator 能反向约束 machine artifacts 的最小必要集合
- 先做大 schema，风险是过度设计

### 6.2 决策二：引入 generation model

最终决策：

- `CRG` 引入 generation snapshot
- 只允许通过 promote 把新 generation 切为 current
- 保留 `last-known-good`

原因：

- 这是“团队级可用”的最低前提
- 否则构建失败会污染消费链

### 6.3 决策三：machine-first 优先于 narrative

最终决策：

- workflow 优先消费 `minimal-context/*.json` 与 machine artifacts
- narrative 文档只作为导航和解释层

原因：

- narrative 可读，但不稳定、不利于精确选择
- machine-first 才能做 evaluator、freshness、budget、regression

### 6.4 决策四：`injection-index.yaml` 降为人类视图

最终决策：

- 保留 `injection-index.yaml`
- 但它不再承担最终执行器职责
- 真正的执行 contract 由 `context-routing.json + evaluator` 承担

原因：

- YAML 路由表适合作为人类可读视图
- 不适合作为唯一的运行时判定逻辑

### 6.5 决策五：检索层采用 hybrid route，而不是单一方案

最终决策：

检索层采用：

1. lexical seed
2. graph expansion
3. heuristic rerank
4. token-budget pack
5. optional semantic enhancement

不采用：

- 一上来全量向量化
- 一上来复杂 learned ranking

原因：

- 当前工程基础更适合 lexical + graph
- 先把 deterministic、可解释、可测做好

## 7. `CRG` 最终优化方案

## 7.1 目标

把 `CRG` 从“AST 图构建器”升级成“稳定的代码事实与任务检索底座”。

## 7.2 存储模型

### 7.2.1 generation 目录

最终结构：

```text
.spec-first/graph/
  generations/
    <generation-id>/
      graph.db
      build-report.json
      retrieval-report.json
  current.json
  last-known-good.json
```

规则：

- 每次 build 写入新的 generation 目录
- 构建通过 health checks 后写 `current.json`
- 上一个稳定 generation 写入 `last-known-good.json`
- evaluator 和 workflow 永远只读 current 或 fallback generation

### 7.2.2 schema 设计

P0 最小 schema：

- `nodes`
  - `generation_id`
  - `parser_quality`
  - `summary`
  - `retrieval_text`
- `graph_meta`
  - `generation_id`
  - `build_health`
  - `parser_degradation_count`

P1 扩展 schema：

- `chunks`
- `retrieval_features`
- `signature_text`
- `tags_json`
- `freshness_score`

原则：

- 先最小可用
- 再按 benchmark 结果扩展

## 7.3 检索模型

最终检索链路固定为：

```text
task request
  -> seed
  -> expand
  -> rerank
  -> pack
  -> ranked_context
```

### 7.3.1 seed

来源：

- lexical search
- direct symbol hit
- changed file hints
- entrypoint hint

### 7.3.2 expand

来源：

- call graph
- import graph
- flow prior
- risk prior
- test prior

### 7.3.3 rerank

P0 / P1 用 heuristic：

- stage relevance
- structural proximity
- risk score
- test relevance
- freshness

### 7.3.4 pack

规则：

- 先保核心 evidence
- 预算不足时先裁 narrative
- 再裁辅助 facts
- 最后才裁核心 minimal context

## 7.4 profile 模型

最终固定 4 类 profile：

- `plan-default`
- `work-default`
- `review-default`
- `verify-default`

各自优先级：

### `plan-default`

- entrypoints
- module boundaries
- integrations
- data shapes

### `work-default`

- changed files
- adjacent tests
- impact
- data flow

### `review-default`

- blast radius
- candidate tests
- high-risk modules
- cross-file dependencies

### `verify-default`

- target tests
- coverage gaps
- high-risk signals
- impacted modules

### sample contract

- checked-in sample 由 generator 自动生成
- 不允许长期手工维护

### consumer contract

规则：

- workflow 只依赖 stable contract
- 不直接依赖运行时偶然字段

## 9. Evaluator 最终方案

这是本方案最关键的模块，规则必须具体、固定、可测试。

## 9.1 输入

P0 最小输入固定为：

- `stage`
- `contextDir`
- `controlPlaneDir`
- `context-routing.json`
- `artifact-manifest.json`
- `freshness.json`
- `minimal-context/*.json`

## 9.2 规则顺序

固定求值顺序：

1. manifest 检查
2. compatibility 检查
3. freshness 检查
4. profile 解析
5. always 资产加载
6. stage 资产加载
7. selection_rules 求值
8. 存在性过滤
9. 去重
10. 排序
11. budget 裁剪
12. fallback reason 输出

## 9.3 资产优先级

固定优先级：

1. `minimal-context/*.json`
2. machine artifacts
3. `code-facts/*.md`
4. `architecture/*.md`
5. `README.md` / `00-summary.md`

## 9.4 fallback 等级

固定四级：

- `L0`
  - machine + minimal context 全量可用
- `L1`
  - minimal context 缺失
- `L2`
  - machine artifacts 不完整
- `L3`
  - 仅剩 summary/readme

## 9.5 fallback reason 枚举

固定枚举：

- `manifest_incomplete`
- `routing_missing`
- `minimal_context_missing`
- `machine_artifact_missing`
- `freshness_stale`
- `compatibility_mismatch`
- `context_dir_missing`

## 9.6 输出

Evaluator 最终返回：

```json
{
  "stage": "review",
  "profile": "review-default",
  "level": "L0",
  "selected_assets": [],
  "estimated_tokens": 0,
  "fallback_reason": null,
  "skipped_rules": [],
  "freshness_status": "healthy",
  "advice": ""
}
```

## 9.7 伪代码

```text
resolve(stage, slug):
  load manifest
  if missing or incomplete:
    return L3(summary/readme, manifest_incomplete)

  load routing
  if missing:
    return L2(core docs, routing_missing)

  profile = resolveProfile(stage)
  assets = always + stageAssets(profile)

  for rule in selection_rules:
    if eval(rule.condition):
      assets += rule.inject

  assets = filterExists(assets)
  assets = applyFreshness(assets)
  assets = dedupe(assets)
  assets = sortByPriority(assets)
  assets = applyBudget(assets)

  if minimal-context missing:
    level = L1
  else:
    level = L0

  return result
```

## 10. Workflow 接入方案

最终接入方式固定为：

- `spec-plan`、`spec-work`、`spec-review` 在运行前调用 evaluator
- evaluator 输出 selected assets
- selected assets 以结构化方式进入 prompt/context
- 不再只依赖 skill 文本中的“请优先阅读”

P0 桥接说明：

- `P0` 先统一 evaluator output contract 与共享测试夹具
- skill runtime 是否完全代码化可以后续迭代
- 但从 `P0` 开始，`SKILL.md`、tests、sample contract 的真源都必须切到 evaluator 输出格式

### 10.1 `spec-plan`

消费：

- `minimal-context/plan.json`

## 13. 风险与治理

## 13.1 主要风险

### 风险一：schema 继续膨胀

治理：

- schema 只按 evaluator 与 benchmark 反推扩展

### 风险二：evaluator 退化为另一个 prompt

治理：

- 所有规则必须结构化
- 所有 fallback 必须可枚举

### 风险三：narrative 文档继续成为事实真源

治理：

- 机器资产先生成
- narrative 只能派生，不可反向定义

### 风险四：generation 引入复杂度后没人维护

治理：

- promote / rollback / current 规则必须最小且硬编码明确

### 风险五：benchmark 长期烂尾

治理：

- P0 就建立最小 review benchmark
- 不等全套系统完成后再补

## 14. 验收标准

只有满足以下标准，方案才视为达成：

### 14.1 正确性

- evaluator 可以稳定决定 selected assets
- fallback reason 可解释
- schema drift 可检测并阻断

### 14.2 可维护性

- generation build 失败不影响当前消费
- sample 不再手工维护
- compatibility 有明确治理

### 14.3 效能

- review evidence recall 提升
- plan 命中入口/模块/测试面的准确性提升
- work 无关扫描明显下降
- token 使用量下降

### 14.4 长期性

- freshness / stale claim / consumer mismatch 可治理
- benchmark 能持续回归

---

## 15. 最终方案摘要

本最终方案只保留一个明确方向：

1. `CRG` 升级为 **generation-aware、retrieval-aware** 的代码事实编译器与检索底座
2. `spec-graph-bootstrap` 升级为 **machine-first、schema-driven** 的 compiled context compiler
3. workflow 通过 **deterministic evaluator** 消费上下文
4. 系统通过 **benchmark + telemetry + drift/freshness 治理** 形成长期闭环

一句话概括：

**当前系统要从“抽事实、生成文档、声明路由”进化为“解析控制面、选择健康 generation、检索最小证据、编译上下文、确定性分发、持续评测反哺”的完整上下文工程系统。**

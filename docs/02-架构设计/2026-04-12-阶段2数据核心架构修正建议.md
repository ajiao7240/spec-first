# 阶段2数据核心架构修正建议

## 1. 结论

阶段2“围绕数据做核心”这个方向本身是正确的，但当前前提不成立。

如果阶段0数据基座不先修正，阶段2继续往前推进，最终会出现：

1. 数据看起来更多
2. 命令看起来更强
3. 结论看起来更完整
4. 但业务可信度反而下降

这不是实现问题，而是架构时序问题。

核心原则只有一句：

> 阶段2不要继续做“更多数据”，而要做“更可信的数据核心”。

## 2. 当前问题背景

基于对真实项目 `/Users/kuang/java_work/huashengsecurities/.spec-first-graph/graph.db` 的结果级审查，当前阶段0已经暴露出以下关键问题：

1. 图主体被 `Pods/` 三方依赖吞没
2. 主工程业务源码几乎未入图
3. `communities` 严重失真
4. `flows = 0`
5. `unresolved_edge_count` 只有总数，没有明细
6. 审计已能证明数据库物理健康，但还不能证明业务结果可信

这意味着当前数据层还停留在：

1. 能生成
2. 能对账
3. 能存储

但还没有达到：

1. 数据范围正确
2. 数据语义正确
3. 数据结果可解释
4. 数据可以稳定支撑阶段2高阶能力

## 3. 目标

阶段2真正应该达成的目标，不是“命令更多”或“表更多”，而是以下六项：

1. 数据范围正确
2. 数据语义正确
3. 数据结果可解释
4. 数据输出可审计
5. 数据模型可演进
6. 兼容多类仓库

其中“兼容多类仓库”至少应覆盖：

1. 通用代码仓
2. iOS/Xcode/CocoaPods
3. Node/Monorepo
4. Java/Kotlin/Gradle

## 4. 非目标

阶段2不应同时追求以下目标，否则容易失控：

1. 一次性把所有语言都做深
2. 一次性把 flow 做得很智能
3. 把所有分析结果都持久化
4. 让 CLI contract 直接等同于 DB schema
5. 所有仓库统一用一套输入规则

## 5. 数据分层模型

阶段2必须将数据核心拆成 4 层。

### 5.1 Facts 层

这是唯一的事实来源，要求最稳定。

建议包含：

1. `nodes`
2. `edges`
3. `fingerprints`
4. `graph_meta`
5. `unresolved_edges`
6. `excluded_inputs`
7. `parse_skips`

这一层回答的问题是：

1. 什么进图了
2. 什么边进图了
3. 什么没进图
4. 什么没解析出来
5. 本次构建到底处理了什么

### 5.2 Derived 层

这是从事实推导出来的结果，允许重算。

建议包含：

1. `communities`
2. `flow_candidates`
3. `flows`
4. `flow_nodes`
5. `derived_meta`

`derived_meta` 建议记录：

1. `algorithm_version`
2. `computed_at`
3. `input_snapshot`
4. `derivation_scope`

### 5.3 Audit / Explain 层

这是阶段2必须补齐的层，用于解释“为什么”。

建议包含：

1. `unresolved_edges`
2. `excluded_inputs`
3. `parse_skips`
4. `ownership_summary`
5. `build_warnings`

这层要能解释：

1. 为什么 unresolved 高
2. 为什么某些目录没进图
3. 为什么 flow 为 0
4. 为什么 community 失真
5. 为什么某次构建结果不可靠

### 5.4 Contract 层

这是对外提供稳定能力的层，建议继续通过命令投影，而不是直接暴露表结构。

建议继续维护以下 contract：

1. `stats`
2. `context`
3. `impact`
4. `communities`
5. `community`
6. `flow`
7. `affected-flows`
8. `review-context`

要求：

1. contract 稳定
2. DB 可演进
3. CLI 输出不被底层 schema 绑死

## 6. 必要表设计建议

### 6.1 `unresolved_edges`

这是第一优先级。

建议字段：

1. `id`
2. `source_id`
3. `source_file_path`
4. `target_name`
5. `target_path_raw`
6. `kind`
7. `reason`
8. `created_at`

`reason` 建议枚举化：

1. `missing_target_id`
2. `target_path_not_found`
3. `target_name_not_found`
4. `target_name_ambiguous`
5. `same_file_disambiguation_failed`
6. `unsupported_target_kind`

### 6.2 `excluded_inputs`

建议字段：

1. `file_path`
2. `stage`
3. `reason`
4. `matched_rule`
5. `created_at`

`stage` 建议取值：

1. `default_exclude`
2. `graphignore`
3. `extra_exclude`
4. `ios_pods_filter`
5. `sensitive_pattern`
6. `binary_ext`

`reason` 可以聚焦为：

1. `runtime_dir`
2. `third_party_dependency`
3. `generated_output`
4. `sensitive_file`
5. `binary_file`
6. `custom_graphignore_rule`

### 6.3 `parse_skips`

建议字段：

1. `file_path`
2. `lang_inferred`
3. `reason`
4. `details`
5. `created_at`

`reason` 至少支持：

1. `unsupported_lang`
2. `read_error`
3. `sensitive`
4. `objc_header_detected`
5. `parser_unavailable`

### 6.4 `ownership_summary`

这一层可用表或视图实现。

建议维度：

1. `bucket`
2. `node_count`
3. `edge_count`
4. `module_count`
5. `symbol_count`

`bucket` 在阶段2必须显式化：

1. `app`
2. `local_module`
3. `third_party`
4. `generated`
5. `runtime`
6. `infra`
7. `other`

## 7. 仓库类型适配建议

当前最大的结构性问题不是解析器，而是输入适配没有成为一等公民。

建议显式引入 `repo profile` 概念。

### 7.1 通用仓库 profile

默认规则：

1. 排除 runtime / build / cache / vendor-like 目录
2. 保留源码主目录
3. 不做平台特化

### 7.2 iOS/Xcode profile

这是当前最迫切需要的 profile。

建议规则：

1. 默认排除 `Pods/**`
2. 若 `Podfile.lock` 存在 `EXTERNAL SOURCES :path:`，仅白名单本地 Pod
3. 优先保留：
   - app 主目录
   - extension 目录
   - modules 目录
   - business framework source
4. 对 `.framework/Headers`、`.xcframework/Headers` 设定更严格策略
   - 默认作为 `third_party/header` bucket
   - 不允许其主导主图

### 7.3 Node/Monorepo profile

建议规则：

1. 排除 `node_modules/**`
2. 优先保留 `packages/*`、`apps/*`
3. 识别 workspace package

### 7.4 Java/Kotlin/Gradle profile

建议规则：

1. 优先保留 `src/main/*`、`src/test/*`
2. 排除 `.gradle/`、`build/`
3. 区分 generated source 与 hand-written source

## 8. 审计门禁建议

阶段2必须把“质量门禁”从结构正确，升级到“结果可信”。

### 8.1 结构门禁

继续保留当前已有门禁：

1. `integrity_check`
2. `foreign_key_check`
3. orphan / duplicate / count mismatch
4. FTS row consistency

### 8.2 结果门禁

建议新增：

1. `orphan_fingerprints = 0`
2. `ownership.third_party_ratio < threshold`
3. `app_node_ratio >= threshold`
4. `isolated_module_ratio < threshold`
5. `files_without_symbols_ratio < threshold`

### 8.3 解释门禁

建议新增：

1. `graph_meta.unresolved_edge_count == unresolved_edges.count`
2. `parse_skips.count` 与构建期 skip 统计一致
3. `excluded_inputs` 计数与 ignore 统计一致
4. `flow_count = 0` 时必须给出 explain code

## 9. 演进顺序建议

### 9.1 里程碑 1：输入基座可信化

优先做：

1. iOS profile 接入
2. Pods 默认排除
3. ownership 分类
4. `fingerprints` 只跟踪真实图输入
5. `excluded_inputs` 落库

验收标准：

1. 真实业务仓中，三方依赖不再主导节点占比
2. 主工程代码明显回到图主体
3. `communities` 不再全是碎片化

### 9.2 里程碑 2：解释层补齐

优先做：

1. `unresolved_edges`
2. `parse_skips`
3. 审计脚本 explain 升级
4. `stats` 输出 explain summary

验收标准：

1. 所有高风险统计值都能 drill-down
2. 不再只有“数字”，而有“原因结构”

### 9.3 里程碑 3：Derived 层稳定化

优先做：

1. communities 稳定化
2. flow 生成规则收紧
3. `flow = 0` 降级契约
4. architecture 只看业务 ownership 的主视图

验收标准：

1. `context / impact / architecture / review-context` 对真实项目有解释力
2. flow 空结果不再是黑盒空

### 9.4 里程碑 4：扩展能力

优先做：

1. 多 profile 抽象
2. derived algorithm versioning
3. migration strategy
4. cache / rebuild 策略
5. 更多语言细化

验收标准：

1. 新增仓库类型不破坏旧 contract
2. schema 演进有边界，不会把底层和上层绑死

## 10. 当前阻塞项

如果要推进阶段2，当前至少存在以下五个硬阻塞：

1. 输入范围治理不足
2. unresolved 不可解释
3. flow 语义不稳定
4. 解释性数据缺失
5. 多仓库类型适配未抽象

## 11. 对“合理、准确、完善、扩展”的最终判断

如果阶段2按本文方案修正：

1. 合理性：成立
2. 准确性：可以逐步建立
3. 完整性：可以达成
4. 扩展性：可以兼顾

如果不修正，而是在当前结构上继续叠加更多 `context / flow / impact` 能力，则结果会是：

1. 方向看似合理
2. 实际不准确
3. 不完善
4. 扩展后只会更脆

## 12. 直接建议

建议立刻做三件事：

1. 先做 iOS 输入收敛修复，解决 `Pods` 吞图
2. 再做 `unresolved / excluded / parse-skips` 三张解释表
3. 最后再评估阶段2后续能力设计

## 13. 总结

阶段2的数据核心可以成立，但必须先从“数据中心”升级为：

1. 事实中心
2. 解释中心
3. 分层中心

少了这三个中心，阶段2不是架构升级，而只是把阶段0的问题放大。

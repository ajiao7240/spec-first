# Spec-First Framework 整体架构

## 1. 架构概述

Spec-First Framework 是一套基于 `spec-first` npm CLI 的轻量级 AI 辅助工作框架，通过五步闭环（brainstorm → plan → work → review → compound）推动软件交付从脆弱、手工、割裂走向智能、自动、可控。

### 1.1 核心设计原则

- **方法论先行**：优先统一协作方式，而非堆叠工具
- **轻量可控**：建立最小状态管理，避免过度工程化
- **契约串联**：通过输入输出契约串联各阶段，而非中心化状态机
- **阶段自治**：每个阶段自己守门、自己产出，下一阶段只消费上游确认过的结果

### 1.2 技术选型

- **发布入口**：npm 包 `spec-first`
- **CLI 入口**：`spec-first doctor` / `spec-first init --claude`
- **实现方式**：Node.js CLI + Claude Code 命令模板
- **状态管理**：文件系统 + task.yaml 最小元数据
- **产物格式**：frontmatter + markdown 混合格式
- **目录组织**：`.claude/tasks/<task-id>/` 按任务聚合

## 3. 核心组件

### 3.1 核心 Skills

**五阶段主链路：**

| Skill | 职责 | 输入 | 输出 |
|-------|------|------|------|
| `/spec:brainstorm` | 思路发散、方案探索 | 问题定义、背景、目标 | 候选方案、取舍分析 |
| `/spec:plan` | 任务拆解、路径设计 | brainstorm 结论、约束 | 任务清单、验收标准 |
| `/spec:work` | 执行落地、产出生成 | plan 结果、模板 | 文档草稿、执行记录 |
| `/spec:review` | 质量校验、问题识别 | work 产物、评审维度 | 评分、问题清单、修订建议 |
| `/spec:compound` | 资产沉淀、知识提炼 | 已通过产物 | 模板、SOP、案例 |

**Supporting Workflows（不属于五阶段主链路）：**

| Skill | 定位 | 输入 | 输出 |
|-------|------|------|------|
| `/spec:graph-bootstrap` | Stage-0 上下文 bootstrap | 目标项目路径或 slug | `docs/contexts/<slug>/` 长期上下文资产 |
| `/spec:audit` | 代码库审计 | 目标项目 | 技术债和高风险模式报告 |

### 3.2 控制面组件

**task.yaml**：最小元数据文件
```yaml
task_id: 2026-03-29-feature-x
title: 功能X开发
role: fullstack
level: L2
current_stage: plan
status: in_progress
created_at: 2026-03-29T10:30:00+08:00
updated_at: 2026-03-29T11:10:00+08:00
```

**阶段契约机制**：六步标准流程
1. 上游产物检查
2. 元数据读取
3. level 决策
4. 输入完整性检查
5. 执行与输出
6. 状态更新

### 3.3 执行面组件

**产物格式**：frontmatter + markdown
- frontmatter：结构化控制信息
- markdown 正文：详细内容说明

**Review 判定**：双重判定机制
- 规则1：关键维度底线
- 规则2：阻断问题优先
- 规则3：总分参考

**Rationale capture**：决策理由记录
- 为什么选这个方案
- 为什么不选其他方案
- 当前选择的适用边界

## 4. 目录结构

### 4.1 运行态目录

```
.claude/
  └── tasks/
      └── <task-id>/
          ├── task.yaml           # 任务元数据
          ├── 01-brainstorm.md    # 头脑风暴产物
          ├── 02-plan.md          # 规划产物
          ├── 03-work.md          # 执行产物
          ├── 04-review.md        # 评审产物
          └── 05-compound.md      # 沉淀产物
```

### 4.2 沉淀态目录

```
.claude/
  └── assets/
      ├── prompts/      # 提示词模板
      ├── workflows/    # SOP 和流程模板
      ├── domain/       # 领域知识卡片
      └── cases/        # 案例库
```

## 5. 数据流

```
用户输入
  ↓
/spec:brainstorm
  ↓ 01-brainstorm.md
/spec:plan
  ↓ 02-plan.md
/spec:work
  ↓ 03-work.md
/spec:review
  ↓ 04-review.md (pass=true)
/spec:compound
  ↓ 05-compound.md
  ↓
.claude/assets/ (可选沉淀)
```

每个阶段：
1. 读取上游产物
2. 读取 task.yaml
3. 执行阶段逻辑
4. 生成阶段产物
5. 更新 task.yaml
6. 下一阶段消费

## 6. 状态机

```
pending → in_progress → completed
   ↓           ↓            ↓
blocked ← failed      (终态)
```

**状态流转规则**：
- skill 成功完成 → 推进 current_stage
- skill 执行失败 → status=failed 或 blocked
- review 不通过 → 不得推进到 compound
- 任一阶段上下文不足 → 可回退上游阶段

## 7. 角色与任务分级

### 7.1 角色模式（可选）

- `product`：产品人员视角
- `dev`：研发人员视角
- `fullstack`：全栈开发人员视角
- `test`：测试人员视角
- `generic`：通用模式（默认）

### 7.2 任务分级

| 级别 | 特征 | 流程要求 |
|------|------|---------|
| L1 | 低风险、单轮可完成 | 可走简化链路 |
| L2 | 有明确交付物、需要标准流程 | 应走完整闭环 |
| L3 | 多角色参与、高风险或高复杂度 | 必须走完整闭环 + 人工审查 |

## 8. 扩展点

### 8.1 一期不包含

- 多模型协同
- Session recovery
- 复杂工作流引擎
- 重运行时能力

### 8.2 二期扩展方向

- 扩展 Codex 等其他模型支撑
- 补强资产检索能力
- 增强模板治理机制
- 补充角色专属 skills

## 9. 关键约束

1. **不构建完整工作流引擎**：避免过度工程化
2. **最小状态管理**：task.yaml 仅承担最小控制面职责
3. **阶段自治**：每个阶段自己守门、自己产出
4. **人机协同**：AI 增强效率，关键决策保留人工把关
5. **一期收敛**：仅聚焦 Claude Code CLI，不做多模型协同

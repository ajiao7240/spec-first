# CRG 阶段0：代码图构建流水线技术文档

> 对应命令：`spec-first crg build --repo=<path>`  
> 最后更新：2026-04-12  
> 版本：v1.6.0

---

## 一、概览

CRG（Code Review Graph）阶段0 是整个代码图系统的地基，负责将一个本地代码仓库转换为结构化的 SQLite 图数据库，供后续查询、分析、代码评审辅助使用。

**输入**：代码仓库根目录  
**输出**：`.spec-first-graph/graph.db` + `.spec-first-graph/fingerprints.json`  
**特性**：增量构建（SHA-256 文件指纹）、iOS 自动检测、多语言 AST 解析

**三个核心目标：**

| 目标 | 说明 |
|------|------|
| **完整性** | 纳入所有业务代码，排除三方库、构建产物、敏感文件 |
| **准确性** | AST 级别的节点与边提取，节点 id 全局唯一、语义正确 |
| **增量性** | SHA-256 指纹跟踪，只重建变更文件，重建代价 O(Δ) 而非 O(N) |

│
│  ┌─────────────────────────────────────────────────────────────────┐
│  │ PHASE D：AST 解析   parser.js                                   │
│  └─────────────────────────────────────────────────────────────────┘
│
├─[D1] 对 changed[] 中每个文件调用 parseFile(filePath, repoRoot)
│
│       parseFile 内部流程：
│       │
│       ├─ inferLanguage(filePath)
│       │   ext → lang 映射（.m → 'objc', .tsx → 'tsx' 等）
│       │   → null 表示不支持 → {skipped:true, reason:'unsupported_lang'}
│       │
│       ├─ isSensitiveFile(basename) 二次过滤（防路径拼接绕过）
│       │   → true → {skipped:true, reason:'sensitive'}
│       │
│       ├─ 读取文件内容（fs.readFileSync）
│       │   → 失败 → {skipped:true, reason:'read_error'}
│       │
│       ├─ .h 文件 ObjC 启发式路由
│       │   lang='c' + 内容含 @interface|@protocol
│       │   → 递归 parseFile(..., {lang:'objc'})
│       │
│       ├─ ObjC 预处理（lang='objc'）
│       │   剥离独立行宏（正则替换为空行）：
│       │     NS_ASSUME_NONNULL_BEGIN / NS_ASSUME_NONNULL_END
│       │     NS_HEADER_AUDIT_BEGIN(...) / NS_HEADER_AUDIT_END
│       │   原因：tree-sitter-objc 3.0.2 无法解析这些 Apple 宏，

│       │   getParser(lang) → Parser 实例
│       │   → null（parser 不可用）→ 仅产出 module 节点（优雅降级）
│       │   parser.parse(content) → tree
│       │
│       ├─ module 节点（每个文件必有，与 tree-sitter 是否可用无关）
│       │   id = buildSymbolKey(filePath, 'module', basename, 0)
│       │      = "<filePath>#module#<basename>#L0"
│       │   is_test = TEST_FILE_RE.test(filePath)
│       │     (.test.js / .spec.ts / __tests__/ 等)
│       │
│       └─ getExtractFn(lang) → 语言专用 AST 遍历器
│
│           ┌──────────────────────────────────────────────────────┐
│           │ extractJSNodes（js / ts / tsx）                      │
│           │                                                      │
│           │ 节点提取：                                           │
│           │   function_declaration       → kind='function'      │
│           │   arrow_function（var 赋值） → kind='function'      │
│           │   class_declaration          → kind='class'         │
│           │   method_definition          → kind='method'        │
│           │                                                      │
│           │ 边提取：                                             │
│           │   import_statement           → imports_from 边      │
│           │   require('path')            → imports_from 边      │
│           │   call_expression            → calls 边             │
│           │   子节点属于父节点           → contains 边          │
│           │   节点定义于文件             → defined_in 边        │
│           └──────────────────────────────────────────────────────┘
│
│           ┌──────────────────────────────────────────────────────┐
│           │ extractObjCNodes（objc）                             │
│           │                                                      │
│           │ 节点提取：                                           │
│           │   class_interface      → kind='class'               │
│           │   class_implementation → kind='class'               │
│           │   protocol_declaration → kind='interface'           │
│           │   method_declaration   → kind='function'            │
│           │   method_definition    → kind='function'            │
│           │                                                      │
│           │ ObjC 方法 selector 重建：                            │
│           │   identifier + ':' (若后跟 method_parameter)        │
│           │   示例：application:didFinishLaunching:options:     │
│           │                                                      │
│           │ 边提取：                                             │
│           │   preproc_include "#import \"file.h\""              │
│           │     → imports_from（target_path_raw = 'file.h'）    │
│           └──────────────────────────────────────────────────────┘
│
│       → { nodes[], rawEdges[], skipped:false, sha256 }
│
│  ┌─────────────────────────────────────────────────────────────────┐

          }
        }

## 五、语言支持矩阵

结合当前代码实现，阶段0的语言口径需要分成四层看：

- `EXT_TO_LANG`：面向 `presentLanguages` 的宽口径识别
- `INDEXABLE_EXTS`：会进入 `finalInputs`、参与 build 的真实入图口径
- `getParser(lang)`：对应 tree-sitter grammar 是否已接入
- `getExtractFn(lang)`：是否有语言专用 AST 提取器，而非只保留 module 节点

| 扩展名 | `presentLanguages` 识别语言 | 可入图 | parser / extractor | 当前语义提取能力 |
|--------|-----------------------------|--------|--------------------|------------------|
| `.js` `.jsx` `.mjs` `.cjs` | `javascript` | ✅ | `tree-sitter-javascript` + `extractJsNodes` | function / class / method / imports / require / calls |
| `.ts` `.mts` `.cts` | `typescript` | ✅ | `tree-sitter-typescript.typescript` + `extractJsNodes` | function / class / method / imports / require / calls |
| `.tsx` | `typescript` | ✅ | `tree-sitter-typescript.tsx` + `extractJsNodes` | JSX/TSX 文件按 JS/TS 语义提取 |
| `.py` `.pyw` | `python` | ✅ | `tree-sitter-python` + `extractPyNodes` | function / class / imports / contains |
| `.go` | `go` | ✅ | `tree-sitter-go` + `extractGoNodes` | function / method / type / imports / contains |
| `.java` | `java` | ✅ | `tree-sitter-java` + `extractJavaNodes` | class / interface / method / imports / contains |
| `.rs` | `rust` | ✅ | `tree-sitter-rust` + `extractRustNodes` | function / struct / enum / trait / use / contains |
| `.c` `.h` | `c` | ✅ | `tree-sitter-c` + `extractCNodes` | C 函数/类型声明；`.h` 若命中 `@interface/@protocol` 会升级走 ObjC |
| `.cc` `.cpp` `.cxx` `.hpp` `.hxx` | `cpp` | ✅ | `tree-sitter-cpp` + `extractCNodes` | C/C++ 共用基础提取器，偏函数/类型声明 |
| `.m` `.mm` | `objc` | ✅ | `tree-sitter-objc` + `extractObjCNodes` | class / interface / selector 方法 / `#import` |
| `.swift` | `swift` | ✅ | `tree-sitter-swift` + `extractSwiftNodes` | class / struct / enum / protocol / function / import |
| `.kt` `.kts` | `kotlin` | ✅ | `tree-sitter-kotlin` + `extractKotlinNodes` | class / object / interface / function / import |
| `.rb` | `ruby` | ✅ | `tree-sitter-ruby` + `extractRubyNodes` | class / module / method / require / include |
| `.php` | `php` | ✅ | `tree-sitter-php` + `extractPhpNodes` | class / interface / trait / function / namespace / use |
| `.cs` | `csharp` | ✅ | `tree-sitter-c-sharp` + `extractCSharpNodes` | class / interface / struct / method / using |
| `.scala` `.sc` | `scala` | ✅ | `tree-sitter-scala` + `extractScalaNodes` | class / object / trait / def / import |
| `.lua` | `lua` | ❌ | 无 | 仅参与 `presentLanguages` 检测，不进入 `finalInputs` |
| `.dart` | `dart` | ❌ | 无 | 仅参与 `presentLanguages` 检测，不进入 `finalInputs` |
| 其余（`.md` `.json` `.sh` 等） | — | ❌ | — | 不参与阶段0代码图构建 |

**复核结论**：

- 当前 `INDEXABLE_EXTS` 已由 `lang-config.js` 自动派生，真实入图语言是 16 种：`javascript`、`typescript`、`tsx`、`python`、`go`、`java`、`rust`、`c`、`cpp`、`objc`、`swift`、`kotlin`、`ruby`、`php`、`csharp`、`scala`。
- `EXT_TO_LANG` 只比真实入图口径多出 `lua`、`dart` 两类“可识别但不可入图”的语言。
- 当前 16 种可入图语言在 `getExtractFn(lang)` 中都已有专用提取器；默认 `extractGenericNodes` 只作为未来新增语言的兜底路径，当前主路径不会走到。
- 即使 grammar 缺失或 parser 初始化失败，`parseFile()` 仍会保留每个文件的 `module` 节点，这属于优雅降级，不等于该语言“不支持”。

## 七、增量构建机制

```
全量构建（首次 or --force）          增量构建（后续运行）
─────────────────────────────        ────────────────────────────────
fingerprints 表为空                  fingerprints 表有历史记录
↓                                    ↓
所有文件均为 changed                 SHA-256 对比 → 仅处理变更文件
↓                                    ↓
解析所有文件 → 全量写入 nodes/edges  解析 changed 文件 → UPSERT/删除
↓                                    ↓
典型耗时 ~500ms（147 文件）          典型耗时 ~50ms（2-3 文件变更）

注意事项：
  parsedChanged=0 时不更新 unresolved_edge_count（保留上次全量值）
  prunedPaths 清理：排除规则变更导致的历史残留节点
  skippedChanged 文件：解析跳过 → DELETE fingerprints（防脏指纹）
  changedShas 复用：detectChangedFiles 计算的 SHA 传给 updateFingerprints，
                    避免同一文件被读取两次
```

## 八、关键设计决策与权衡

| 决策 | 方案 | 原因 |
|------|------|------|
| `Pods/**` 毯式排除 | 兜底排除 + `:path:` 白名单 | PODS 段 Pod 数可达 100+，逐一枚举不可靠；Pods/ 在 .gitignore 中，tracked 模式天然排除 |
| `maxBuffer=256MB` | execSync maxBuffer 扩大 | 万级文件仓库超出默认 1MB → 静默 null → 误降级为 all-files 模式 |
| `.h` 文件 ObjC 路由 | 正则启发式检测 `@interface`/`@protocol` | 同扩展名承载不同语义（C header vs ObjC header），无更好区分依据 |
| `NS_ASSUME_NONNULL` 预处理 | 正则剥离独立行宏 | tree-sitter-objc 3.0.2 无法解析，导致后续 `@interface` 全部产出 ERROR 节点 |
| basename 模糊匹配（阶段1.5） | 公共前缀最长者（最近邻） | ObjC `#import "file.h"` 无路径信息，git 无 include path，最近邻是最可靠的启发式策略 |
| `EXT_TO_LANG` / `INDEXABLE_EXTS` 分离 | 双口径 | 语言识别（宽）与 parser 能力（窄）解耦，便于未来扩展不影响现有语言统计 |
| `.gitignore` 仅在 `all-files` 模式读取 | 按模式条件加载 | tracked 模式 git 天然处理；all-files fallback 手动补偿，行为对齐 |
| FTS5 全量重建 | drop + recreate | content= 外部内容表要求列名严格对应；独立 FTS 无此约束，逻辑更简单稳定 |
| `resolveEdges` 缓存用 `Object.create(null)` | 无原型链缓存对象 | 普通 `{}` 继承 `Object.prototype`，`symbolCache['toString']` 会命中原生函数而非 undefined |

## 十、已知限制（阶段0 边界）

| 限制 | 说明 | 计划 |
|------|------|------|
| Lua / Dart 不入图 | 仅存在于 `EXT_TO_LANG`，用于 `presentLanguages` 报告；不在 `LANG_CONFIG` 中 | 待存在稳定 grammar 与明确收益后再评估 |
| 语言 grammar 缺失时降级为 module-only | `getParser(lang)` 返回 null 时，文件仍保留 `module` 节点，但无 AST 语义节点与边 | 通过依赖安装与 CI 校验降低发生概率 |
| 系统框架 import 未解析 | ObjC `#import <Foundation/Foundation.h>` → unresolved（正常，系统头不入图） | 设计如此 |
| 嵌套 `.gitignore` 不处理 | all-files 模式仅读根目录 `.gitignore` | 复杂嵌套语义依赖完整 git 实现，超出 fallback 范围 |
| 跨文件 `calls` 边解析率 | 依赖函数名全局唯一性，重名时降级为 unresolved | 阶段2 引入类型推断提升精度 |

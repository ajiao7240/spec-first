# CRG-first 规范生成与按需加载方案

> 文档状态：修订方案  
> 修订日期：2026-04-26  
> 核心裁决：先用 `$spec-graph-bootstrap` 准备工程事实，再由规范生成 skill 基于事实提出规范建议；CRG 是推荐上游，不是硬前置；规范写回必须 preview-first，避免自动抽取规则、自动刷新规范或引入质量 gate。

---

# 1. 一句话结论

`spec-first` 需要的是 **CRG-first 的规范输入增强链路**，不是一套完整的规范治理平台。

正确闭环是：

```text
$spec-graph-bootstrap
  准备代码事实、导航事实、结构证据
        ↓
规范生成 skill
  基于 CRG evidence + README/manifest + 现有规范输入提出建议
        ↓
preview-first
  人确认哪些建议能成为长期规范
        ↓
写入轻量规范输入
  repo-profile / manual standards index
        ↓
plan / work / review
  按需加载规范输入，LLM 做最终判断
```

不应该做成：

```text
扫描代码
  → 自动抽取规则
  → 自动写入规范
  → 自动检查 fail/warn
  → 自动刷新规范
```

后者会把事实层、规范层、review 层和质量 gate 混在一起，违背当前项目的核心哲学：

- Light contract
- Explicit boundaries
- Let the LLM decide
- Scripts prepare, LLM decides

---

# 2. 核心目标

本方案只解决一个问题：

> **在不引入重型规则系统的前提下，用 CRG 事实提升项目规范生成与规范加载质量。**

具体目标：

1. 让规范建议建立在真实工程事实上，而不是只靠 README 和目录名猜测。
2. 明确区分“代码事实”和“项目规范”。
3. 让规范生成 skill 输出可审查的建议，而不是静默写回。
4. 让后续 plan / work / review 按需读取轻量规范输入，而不是全量加载文档。
5. 避免 `specs extract / check / refresh / sync` 演化为新的中心化平台。

非目标：

1. 不做完整 `docs/specs/**` 多目录规范操作系统。
2. 不做自动从代码抽取规范并直接生效。
3. 不做规范违反的 CLI hard gate。
4. 不做规则引擎、policy engine、routing engine。
5. 不把规范正文同步进 `AGENTS.md` / `CLAUDE.md` / Cursor / Copilot。
6. 不让 CRG 事实自动升格为规范。

# 3. 职责边界

## 3.1 CRG / spec-graph-bootstrap

职责：

- 构建和验证本地工程事实图。
- 提供 query-first 的结构事实、入口事实、影响面事实、导航事实。
- 输出 advisory evidence，供 LLM 判断。

典型产物：

```text
.spec-first/graph/graph.db
.spec-first/graph/graph-index-status.json
.spec-first/graph/code-navigation.json
.spec-first/graph/graph-operations.jsonl
```

典型查询：

```bash
spec-first crg workflow-context --stage=plan --repo=.
spec-first crg locate --repo=. --query="public entrypoints" --limit=10
spec-first crg locate --repo=. --query="test entrypoints" --limit=10
spec-first crg architecture --repo=.
spec-first crg god-nodes --repo=.
```

CRG 只回答：

```text
代码库现在是什么
哪些模块/入口/测试/关系真实存在
哪些证据可供 LLM 参考
```

CRG 不回答：

```text
团队长期应该怎么做
哪些现状应该成为规范
违反某规范是否应该 fail
```

## 3.2 规范生成 skill

职责：

- 读取现有规范输入和 CRG evidence。
- 基于证据提出 repo-level 规范建议。
- 标注证据、置信度、适用边界和不确定项。
- 默认 preview，不静默写回。

规范生成 skill 只回答：

```text
基于当前事实，哪些内容可能值得成为长期项目规范
哪些建议证据充分
哪些建议只是低置信度推断
哪些内容需要人确认
```

它不应该：

```text
自动把所有重复代码模式写成规范
自动刷新人工规范
自动改变 workflow state
自动触发 plan/work/review
自动创建质量 gate
```

## 3.3 人

职责：

- 判断建议是否代表团队真正想坚持的长期规范。
- 确认、修改或拒绝规范写回。
- 对冲突规范做最终裁决。

这是必须保留的人类决策点，因为“现状”不等于“规范”。

# 4. 推荐使用顺序

## 4.1 首次为项目生成规范

推荐顺序：

```bash
$spec-graph-bootstrap .
```

然后运行规范生成 skill：

```text
规范生成 skill
  读取 CRG readiness
  查询核心入口/模块/测试/风险节点
  读取 README / package manifest / 现有规范输入
  输出 preview 建议
```

最后人工确认写回。

## 4.2 已有规范，只做小修改

可以跳过 `$spec-graph-bootstrap`，直接运行规范生成 skill 或手工修改。

适用场景：

- 用户明确要新增一条人工规范。
- 只调整文案表达。
- 不涉及从代码事实推导长期原则。

## 4.3 graph 未准备或不可用

规范生成 skill 必须支持降级，不能把 `$spec-graph-bootstrap` 做成硬前置。

默认策略：

```text
先提示，再降级，继续产出低置信度 preview。
```

具体行为：

1. 启动时先探测 CRG readiness。
2. 如果 graph ready，使用 CRG evidence。
3. 如果 graph missing / unavailable / degraded：
   - 明确提示建议先执行 `$spec-graph-bootstrap .`
   - 同时继续降级读取 repo 直接文件
   - 在 preview 里标注 `graph_state` 和 `limitations`
   - 将建议置信度整体下调，不生成高置信度架构规范
4. 只有当用户显式要求“必须基于 CRG evidence 生成”时，才停止并要求先运行 `$spec-graph-bootstrap`。

降级模式：

```text
fallback mode: direct_repo_reads
```

允许读取：

- `README.md`
- `package.json` / `pyproject.toml` / `go.mod` / `Cargo.toml`
- `src/`
- `tests/`
- 关键配置文件

必须说明：

- 当前没有 CRG evidence。
- 本轮建议置信度较低。
- 建议后续运行 `$spec-graph-bootstrap` 后再复核。

推荐提示文案：

```text
当前未检测到可用 CRG graph。建议先运行 `$spec-graph-bootstrap .` 以获得更可靠的代码结构证据。
本次将降级为 direct repo reads，仅基于 README、manifest、源码入口和测试目录生成低置信度 preview；不会自动写回规范。
```

## 4.4 失败处理矩阵

| 状态 | 行为 | 是否继续 | 输出要求 |
| --- | --- | --- | --- |
| graph ready | 使用 CRG 查询增强建议 | 继续 | `graph_state: ready`，引用 CRG evidence |
| graph missing | 提示 `$spec-graph-bootstrap .`，降级 direct reads | 继续 | `graph_state: missing`，标注低置信度 |
| graph degraded | 使用可用 CRG evidence + direct reads 补足 | 继续 | `graph_state: degraded`，列出 limitations |
| graph build failed | 不重试构建，提示错误和 bootstrap 命令，降级 direct reads | 继续 | `graph_state: unavailable`，记录失败原因 |
| 用户要求 CRG-only | 不降级 | 停止 | 要求先运行 `$spec-graph-bootstrap .` |

这保证两点：

- 首次使用时不会因为没跑 graph-bootstrap 而卡死。
- 高质量路径仍然清楚引导用户先准备 CRG 事实。

# 5. 决策输入模型

规范生成 skill 的输入按优先级分层：

## 5.1 已确认规范输入

最高优先级。

建议目标形态：

```text
.spec-first/specs/repo-profile.yaml
```

如果当前仓库版本没有自动初始化该文件，也可以由用户显式创建。它只承载轻量 repo-level 规范输入：

```yaml
schema_version: 1
repo_id: example
project_type: cli-tooling
languages:
  - javascript

project_intent:
  summary: Short, stable project intent.

principles:
  - Prefer explicit stage boundaries

non_negotiables:
  - Generated runtime assets must not be edited directly

review_defaults:
  - boundary integrity
  - fallback clarity
```

字段边界：

- `project_intent.summary`：项目长期意图。
- `principles`：长期稳定原则。
- `non_negotiables`：不可违反项。
- `review_defaults`：默认 review 关注方向。

禁止字段：

- workflow state
- quality gate state
- routing rules
- transition rules
- task-specific requirements
- verifier dispatch

## 5.2 CRG observed evidence

第二优先级。

来自：

```text
.spec-first/graph/graph.db
.spec-first/graph/code-navigation.json
spec-first crg workflow-context / locate / architecture / god-nodes
```

用途：

- 识别真实入口和模块边界。
- 识别测试入口和验证面。
- 识别高风险枢纽或高耦合模块。
- 发现 README 与代码事实的明显漂移。

限制：

- CRG evidence 只能证明“现状”。
- 不能直接证明“应该作为规范”。
- 必须经过 LLM 解释和人确认。

## 5.3 README / manifest / config

第三优先级。

用途：

- 建立项目身份。
- 获取项目描述和主语言。
- 判断用户可见目标。
- 补足 CRG 不覆盖的配置事实。

## 5.4 手工补充说明

用户可以在运行规范生成 skill 时提供：

```text
本项目最重要的规范是...
不要把历史包袱写成规范...
这次只生成 review_defaults...
```

这类输入优先级高于自动推断。

# 6. 输出模型

规范生成 skill 必须先输出 preview。

## 6.1 Preview 结构

```markdown
# Repo Profile Suggestions

## Summary

- graph_state: ready | degraded | unavailable
- fallback_mode: none | direct_repo_reads
- limitations:
  - graph unavailable, suggestions are lower confidence
- evidence_sources:
  - crg_workflow_context
  - crg_locate
  - README.md
  - package.json
- write_target: .spec-first/specs/repo-profile.yaml

## Suggested principles

### Prefer query-first code evidence over stale context packs

- confidence: high
- evidence:
  - `skills/spec-graph-bootstrap/SKILL.md`
  - `src/crg/workflow-context/stage.js`
- rationale:
  CRG 当前主路径已切到 query-first evidence，规范应提醒后续 workflow 优先使用现行 graph evidence。

## Suggested non_negotiables

### Do not edit generated runtime assets directly

- confidence: high
- evidence:
  - `AGENTS.md`
  - `src/cli/adapters/*`
- rationale:
  当前项目有 source/runtime mirror 边界，直接编辑生成资产会导致 drift。

## Rejected / uncertain candidates

### Use current repeated style X as mandatory rule

- confidence: low
- reason:
  只能证明代码里存在该模式，不能证明团队希望长期坚持。

## Proposed patch

```yaml
principles:
  - Prefer query-first code evidence over stale context packs

non_negotiables:
  - Do not edit generated runtime assets directly

review_defaults:
  - source/runtime boundary integrity
```
```

## 6.2 写回规则

默认不写回。

只有用户确认后，才允许写：

```text
.spec-first/specs/repo-profile.yaml
```

写回必须遵守：

1. 不覆盖用户未授权字段。
2. 不写 runtime / workflow / gate state。
3. 不写任务临时要求。
4. 不把 inferred 低置信度建议写成 hard rule。
5. 保持文件短小，优先 30 到 60 行。

# 7. 规范索引的合理边界

本方案保留“按需加载规范”的价值，但不在第一版实现完整 `docs/specs` 平台。

## 7.1 第一版只需要 repo-profile

第一版只验证：

```text
.spec-first/specs/repo-profile.yaml
```

是否能改善：

- `spec-plan` 的方案质量。
- `spec-code-review` 的默认关注点。
- agent 对项目边界的理解。

如果这个文件长期没人维护，说明扩展成多文件规范系统没有意义。

## 7.2 第二版才考虑 manual standards index

只有当 `repo-profile.yaml` 被真实维护，并且团队确实有多份人工规范时，才引入轻量索引：

```text
.spec-first/specs/manual-index.json
```

建议字段：

```json
{
  "version": 1,
  "standards": [
    {
      "id": "generated-assets-boundary",
      "path": "docs/standards/generated-assets-boundary.md",
      "title": "Generated assets boundary",
      "source": "manual",
      "summary": "Do not edit generated runtime assets directly.",
      "applies_to_paths": [
        ".claude/**",
        ".codex/**"
      ],
      "keywords": [
        "runtime",
        "generated",
        "drift"
      ],
      "token_estimate": 900
    }
  ]
}
```

注意：

- 索引只服务加载决策。
- 不包含 fail/warn gate。
- 不包含自动抽取规则。
- 不包含自动刷新逻辑。

## 7.3 不推荐的第一版能力

暂缓：

- `spec-first specs extract`
- `spec-first specs check --changed`
- `spec-first specs refresh --changed`
- `spec-first specs sync`
- `rules-map.json`
- inferred rule promotion
- conflict auto-resolution
- embedding 检索

这些能力都可能把轻量输入层推向规范平台和质量 gate。

# 8. 与 plan / work / review 的集成

## 8.1 spec-plan

推荐消费：

- `project_intent.summary`
- `principles`
- `non_negotiables`

行为：

- 作为 optional planning input。
- 空值按 absent 处理。
- 不把 repo-profile 展开成 checklist。
- 不因为缺失 repo-profile 阻断 plan。

## 8.2 spec-work

推荐消费：

- `non_negotiables`
- 与当前改动命中的 manual standards summary

行为：

- 只提醒实现边界。
- 不自动阻断执行。
- 如果发现规范与实现目标冲突，停止并说明 tradeoff。

## 8.3 spec-code-review

推荐消费：

- `review_defaults`
- `non_negotiables`
- 与 diff 命中的 manual standards
- CRG `before-review` / `review-context` evidence

行为：

- 把规范作为 reviewer context。
- LLM 结合 diff、CRG evidence 和测试结果判断问题。
- 输出 findings 时必须引用证据。
- 不把规范索引当成 hard gate。

# 9. 质量保障

## 9.1 好规范的判定标准

一条规范只有满足以下条件，才值得写入：

1. 长期有效，不是当前任务临时要求。
2. 可执行，可判断，不是空泛价值观。
3. 有证据支撑，或由用户明确声明。
4. 能提升 plan / work / review 的决策质量。
5. 不与已有规范或项目事实明显冲突。

## 9.2 坏规范信号

以下内容不应写入：

- “代码要高质量”
- “注意安全”
- “保持简洁”
- “遵循最佳实践”
- “当前代码里大量这么写，所以必须继续这么写”
- “修改某类文件必须自动触发某 workflow”
- “命中某规则必须 fail”

## 9.3 Evidence 等级

建议使用轻量 evidence 分类：

```text
confirmed
  用户明确说明，或已有人工规范明确写入。

observed
  CRG / 源码 / 测试 / 配置中可重复观察到。

inferred
  LLM 基于多项证据推断，但未被人确认。

uncertain
  证据不足，不应写入，只能列为候选问题。
```

写回策略：

```text
confirmed → 可写入
observed + rationale strong → preview 中建议，人确认后写入
inferred → 默认不写入，除非用户确认
uncertain → 不写入
```

# 10. MVP

## 10.1 MVP 必做

1. 规范生成 skill 启动时检查 CRG readiness。
2. graph ready 时执行至少三类查询：
   - workflow context
   - public entrypoints / main modules locate
   - tests / validation surface locate
3. graph missing / unavailable / degraded 时提示 `$spec-graph-bootstrap .`。
4. 默认降级 direct repo reads 并继续输出低置信度 preview。
5. 用户显式要求 CRG-only 时停止，不降级。
6. 读取现有 repo-profile。
7. 输出 preview suggestions，带 evidence / confidence / proposed patch / limitations。
8. 用户确认后只写允许字段。
9. `spec-plan` 能把非空 repo-profile 字段作为 optional input。

## 10.2 MVP 不做

1. 不新增完整 `specs` CLI。
2. 不做 `docs/specs` 多目录初始化。
3. 不做自动抽取规则。
4. 不做自动刷新规范。
5. 不做规范检查 hard gate。
6. 不同步规范正文到 host instruction files。
7. 不把 CRG query 输出直接写成规范。

## 10.3 Done signals

MVP 完成后应能证明：

1. 有 graph 时，规范建议比 direct reads 更贴近真实架构。
2. 无 graph 时，skill 会提示 `$spec-graph-bootstrap .`，并安全降级 direct reads。
3. CRG-only 模式下没有 graph 会停止，而不是生成伪 CRG evidence。
4. preview 中每条建议都有 evidence、confidence 和 limitations。
5. 写回文件没有 runtime / workflow / gate 字段。
6. `spec-plan` 只消费规范输入，不把它变成 rules engine。

# 11. 推荐命令与交互

## 11.1 首次生成

```bash
$spec-graph-bootstrap .
```

然后：

```text
运行规范生成 skill
```

skill 内部建议执行：

```bash
spec-first crg workflow-context --stage=plan --repo=.
spec-first crg locate --repo=. --query="public entrypoints" --limit=10
spec-first crg locate --repo=. --query="test entrypoints" --limit=10
spec-first crg architecture --repo=.
```

## 11.2 复核现有规范

```text
运行规范生成 skill --mode=preview
```

只输出建议，不写文件。

## 11.3 应用建议

```text
运行规范生成 skill --mode=apply
```

应用前必须展示 patch preview。

# 12. 最终架构判断

这条路线值得做，但只能按轻量输入层做。

应该保留：

- `$spec-graph-bootstrap` 作为推荐上游。
- CRG observed evidence。
- 规范建议 preview。
- 人工确认写回。
- 按需加载规范输入。

必须避免：

- 多目录规范平台。
- 自动抽取规则直接生效。
- 自动刷新人工规范。
- CLI hard gate。
- `AGENTS.md` / `CLAUDE.md` 规范正文膨胀。
- 把 CRG 事实当成规范真相。

最终定版：

> **先用 CRG 提升事实输入质量，再让 LLM 基于事实提出规范建议，最后由人确认哪些建议能成为长期规范。**

这符合 spec-first 的长期方向：脚本准备确定性事实，LLM 做语义判断，人确认长期约束。

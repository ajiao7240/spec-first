# CRG-first 规范输入增强方案

> 文档状态：修订方案  
> 修订日期：2026-04-26  
> 核心裁决：新增独立 `$spec-standards` workflow skill；本文是 `$spec-standards` 的 MVP-A / MVP-B 技术路线，不是完整规范资产系统的最终形态；先用 `$spec-graph-bootstrap` 准备工程事实，并在完成时提示 `$spec-standards` 作为推荐下一步，但不自动调用；`$spec-standards` 是规范输入入口，不是规范执行引擎；MVP-A 只验证 `repo-profile + CRG-first preview`，MVP-B 支持确认写回与 workflow 可选消费，并为后续 manual standards index 留出兼容接口；manual standards index 是 Phase 2A 的第一扩展，共享规范、`standards attach` 和外挂 Git 生命周期属于 Phase 2B+；规范写回必须 preview-first，避免自动抽取规则、自动刷新规范或引入质量 gate。

---

# 1. 一句话结论

`spec-first` 当前阶段需要的是 **CRG-first 的轻量规范输入增强链路**，而不是一开始就建设完整规范治理平台。

`$spec-standards` 在 MVP 阶段只管理轻量 repo-level standards input。它不是完整规范平台，也不是规范检查器；它的价值在于用 CRG evidence 提升规范建议质量，用 preview-first 保留人工裁决，再把确认后的轻量规范输入交给 plan / work / review 按需消费。

正确闭环是：

```text
$spec-graph-bootstrap
  准备代码事实、导航事实、结构证据
  完成后提示 next step: $spec-standards，不自动执行
        ↓
$spec-standards
  基于 CRG evidence + README/manifest + 现有规范输入提出建议
        ↓
preview-first
  人确认哪些建议能成为长期规范
        ↓
写入轻量规范输入
  repo-profile（MVP）/ manual standards index（Phase 2A+）
        ↓
plan / work / review
  按需加载规范输入，LLM 做最终判断
```

不应该做成：

```text
扫描代码
  → 自动抽取规则
  → 自动写入规范
  → 自动检查 fail/warn
  → 自动刷新规范
```

后者会把事实层、规范层、review 层和质量 gate 混在一起，违背当前项目的核心哲学：

- Light contract
- Explicit boundaries
- Let the LLM decide
- Scripts prepare, LLM decides

---

# 2. 核心目标

本方案只解决一个问题：

> **在不引入重型规则系统的前提下，用 CRG 事实提升项目规范生成与规范加载质量。**

具体目标：

1. 让规范建议建立在真实工程事实上，而不是只靠 README 和目录名猜测。
2. 明确区分“代码事实”和“项目规范”。
3. 让 `$spec-standards` 输出可审查的建议，而不是静默写回。
4. 让后续 plan / work / review 按需读取轻量规范输入，而不是全量加载文档。
5. 避免 `specs extract / check / refresh / sync` 演化为新的中心化平台。

非目标：

1. MVP 不引入完整 `docs/specs/**` 多目录规范平台；如果 `repo-profile` 和 manual standards 被真实维护，再在 Phase 2A+ 引入多文件规范目录与轻量索引。
2. 不做自动从代码抽取规范并直接生效。
3. 不做规范违反的 CLI hard gate。
4. 不做规则引擎、policy engine、routing engine。
5. 不把规范正文同步进 `AGENTS.md` / `CLAUDE.md` / Cursor / Copilot。
6. 不让 CRG 事实自动升格为规范。

# 3. 职责边界

## 3.1 CRG / spec-graph-bootstrap

职责：

- 构建和验证本地工程事实图。
- 提供 query-first 的结构事实、入口事实、影响面事实、导航事实。
- 输出 advisory evidence，供 LLM 判断。
- 完成后输出 next-step guidance，引导用户运行 `$spec-standards`。

典型产物：

```text
.spec-first/graph/graph.db
.spec-first/graph/graph-index-status.json
.spec-first/graph/code-navigation.json
.spec-first/graph/graph-operations.jsonl
```

CRG 通过轻量 Evidence Adapter 暴露给 workflow。协议层关心 evidence 类型，不绑定具体命令名：

```text
workflow_context
architecture_summary
navigation_index
locate_result
explain_result
path_result
graph_health
limitations
```

当前实现示例：

```bash
spec-first crg workflow-context --stage=plan --repo=.
spec-first crg architecture --repo=.
spec-first crg locate --repo=. --query="<具体文件/模块/symbol>" --limit=10
spec-first crg explain --repo=. --node="<具体 id/name/path>"
spec-first crg path --repo=. --from="<具体 source node>" --to="<具体 target node>"
spec-first crg god-nodes --repo=.
```

CRG 只回答：

```text
代码库现在是什么
哪些模块/入口/测试/关系真实存在
哪些证据可供 LLM 参考
```

CRG 不回答：

```text
团队长期应该怎么做
哪些现状应该成为规范
违反某规范是否应该 fail
```

`$spec-graph-bootstrap` 可以提示：

```text
Graph ready. Next recommended step:
$spec-standards
```

但不能自动调用 `$spec-standards`。自动串联会把事实准备和语义规范判断变成强编排，削弱边界。

## 3.2 `$spec-standards`

职责：

- 读取现有规范输入和 CRG evidence。
- 基于证据提出 repo-level 规范建议。
- 标注证据、置信度、适用边界和不确定项。
- 默认 preview，不静默写回。
- 支持直接运行：自行检测 graph readiness，ready 时使用 CRG，degraded 时部分使用 CRG，missing / unavailable 时提示 `$spec-graph-bootstrap` 并可降级。

`$spec-standards` 只回答：

```text
基于当前事实，哪些内容可能值得成为长期项目规范
哪些建议证据充分
哪些建议只是低置信度推断
哪些内容需要人确认
```

MVP 阶段必须明确展示：

```text
$spec-standards is the entry point for standards input, not a standards enforcement engine.
```

中文表达：

> `$spec-standards` 是规范输入入口，不是规范执行引擎。

它不应该：

```text
自动把所有重复代码模式写成规范
自动刷新人工规范
自动改变 workflow state
自动触发 plan/work/review
自动创建质量 gate
自动要求 $spec-graph-bootstrap 立即串联执行自己
```

## 3.3 人

职责：

- 判断建议是否代表团队真正想坚持的长期规范。
- 确认、修改或拒绝规范写回。
- 对冲突规范做最终裁决。

这是必须保留的人类决策点，因为“现状”不等于“规范”。

## 3.4 术语：target repo

本文中的 `target repo` 在 MVP 语境下默认指当前执行目录 `.` 中的项目，或用户显式传入的本地仓库目录。

MVP 不包含：

- 远程 Git URL resolver。
- 自动 clone / pull / push。
- pinned ref cache。
- 跨 repo 隐式写回。

当文档讨论“外挂 Git 工程”时，指 Phase 2+ 的完整能力；它可以建立在本地 target repo path 之上，但远程仓库生命周期管理和跨 repo 写回都不属于 MVP。

# 4. 推荐使用顺序

## 4.1 首次为项目生成规范

推荐顺序：

```bash
$spec-graph-bootstrap .
```

然后运行 `$spec-standards`：

```text
$spec-standards
  读取 CRG readiness
  查询核心入口/模块/测试/风险节点
  读取 README / package manifest / 现有规范输入
  输出 preview 建议
```

最后人工确认写回。

## 4.2 已有规范，只做小修改

可以跳过 `$spec-graph-bootstrap`，直接运行 `$spec-standards` 或手工修改。

适用场景：

- 用户明确要新增一条人工规范。
- 只调整文案表达。
- 不涉及从代码事实推导长期原则。

## 4.3 graph 未准备或不可用

`$spec-standards` 必须支持降级，不能把 `$spec-graph-bootstrap` 做成硬前置。

默认策略：

```text
interactive 模式先让用户明确选择；headless / 无交互模式才自动降级。
```

具体行为：

1. 启动时先探测 CRG readiness。
2. 如果 graph ready，使用 CRG evidence。
3. 如果 graph missing / unavailable：
   - 明确提示建议先执行 `$spec-graph-bootstrap .`
   - interactive 模式给出三种选择：
     - 运行 `$spec-graph-bootstrap .`
     - 继续 direct-only 低置信度 preview
     - CRG-only stop
   - headless / 无交互模式才默认读取 repo 直接文件
   - 在 preview 里标注 `graph_state` 和 `limitations`
   - 将建议置信度整体下调，不生成高置信度架构规范
   - 禁止写入基于代码推断或 LLM inferred 的规范
   - 允许用户显式声明的规范，或已有 `repo-profile` 的纯文本整理类修改，在 patch preview + 人工确认 + 确定性写回器存在时进入 apply
4. 如果 graph degraded：
   - 使用仍然可用的 CRG evidence
   - 同时用 direct repo reads 补足缺口
   - 在 preview 里列出不可用的 graph 能力或缺失产物
   - 只下调缺少可用 CRG evidence 支撑的 claim，不把全部建议一刀切降为低置信度
5. 只有当用户显式要求“必须基于 CRG evidence 生成”时，才停止并要求先运行 `$spec-graph-bootstrap`。

降级模式：

```text
fallback mode:
  direct_repo_reads                  # graph missing / unavailable
  crg_partial_plus_direct_repo_reads # graph degraded
```

允许读取：

- `README.md`
- `package.json` / `pyproject.toml` / `go.mod` / `Cargo.toml`
- `src/`
- `tests/`
- 关键配置文件

graph missing / unavailable 的 direct-only fallback 必须说明：

- 当前没有 CRG evidence。
- 本轮建议置信度较低。
- 建议后续运行 `$spec-graph-bootstrap` 后再复核。

graph degraded 必须说明：

- 哪些 CRG evidence 可用。
- 哪些 graph 能力或产物不可用。
- 哪些 claim 因缺少可用 CRG 支撑而被下调置信度。

推荐提示文案：

```text
当前未检测到可用 CRG graph。建议先运行 `$spec-graph-bootstrap .` 以获得更可靠的代码结构证据；你也可以继续让 `$spec-standards` 降级生成低置信度 preview。
请选择：
1. 运行 `$spec-graph-bootstrap .`
2. 继续 direct-only 低置信度 preview（代码推断禁用 apply）
3. CRG-only stop
```

## 4.4 失败处理矩阵

| 状态 | 行为 | 是否继续 | 输出要求 |
| --- | --- | --- | --- |
| graph ready | 使用 CRG 查询增强建议 | 继续 | `graph_state: ready`，引用 CRG evidence |
| graph missing | interactive 模式让用户选择 bootstrap / direct-only / CRG-only stop；headless 才默认 direct reads | 视选择继续 | `graph_state: missing`，标注低置信度，不生成高置信度架构规范；direct-only 时禁止写入代码推断规范，但允许用户显式声明规范经确认写回 |
| graph degraded | 使用可用 CRG evidence + direct reads 补足 | 继续 | `graph_state: degraded`，列出 limitations，只下调未被可用 CRG 支撑的 claim |
| graph build failed | 不重试构建；interactive 模式让用户选择 bootstrap / direct-only / CRG-only stop；headless 才默认 direct reads | 视选择继续 | `graph_state: unavailable`，记录失败原因，标注低置信度；direct-only 时禁止写入代码推断规范，但允许用户显式声明规范经确认写回 |
| 用户要求 CRG-only | 不降级 | 停止 | 要求先运行 `$spec-graph-bootstrap .` |

这保证两点：

- 首次使用时不会因为没跑 graph-bootstrap 而卡死。
- 高质量路径仍然清楚引导用户先准备 CRG 事实。

# 5. 决策输入模型

`$spec-standards` 只维护一套输入优先级，避免不同章节形成第二套排序：

```text
1. 用户显式说明
2. repo-profile confirmed fields
3. Phase 2A optional manual standards index
4. Phase 2B+ optional pinned shared standards
5. CRG observed evidence
6. README / manifest / config
```

MVP-A 激活的默认输入只有 `1 / 2 / 5 / 6`。MVP-B 写回器必须为后续 manual standards index 留出兼容边界，但不实现自动命中。`manual standards index` 是验证 `repo-profile` 有真实使用价值后的第一扩展（Phase 2A），`pinned shared standards` 是更靠后的团队共享扩展（Phase 2B+）。如果用户在 MVP 阶段显式提供某份规范文件或摘录，可以把它当作“用户显式说明”的补充材料读取，但不实现自动索引解析、命中或加载。

## 5.1 用户显式说明

最高优先级。用户可以在运行 `$spec-standards` 时提供：

```text
本项目最重要的规范是...
不要把历史包袱写成规范...
这次只生成 review_defaults...
```

这类输入代表当前任务的显式裁决，但仍不能让 skill 绕过 preview-first 写回。

## 5.2 repo-profile confirmed fields

第二优先级。

MVP 目标形态：

```text
.spec-first/specs/repo-profile.yaml
```

如果当前仓库版本没有自动初始化该文件，也可以由用户显式创建。它只承载轻量 repo-level 规范输入，但要能描述多端项目的基本形态，避免把规范输入压缩成几条抽象原则：

```yaml
schema_version: 1
repo_id: example

project_intent:
  summary: Short, stable project intent.

repo_profile:
  project_type: cli-tooling
  languages:
    - javascript
  frameworks: []
  apps:
    - cli
  packages: []

standards_summary:
  common:
    - Prefer explicit source/runtime boundaries.
  frontend: []
  backend: []
  mobile: []
  desktop: []

principles:
  - Prefer explicit stage boundaries

non_negotiables:
  - Generated runtime assets must not be edited directly

review_defaults:
  - boundary integrity
  - fallback clarity

loading_hints:
  prefer:
    - .spec-first/specs/repo-profile.yaml
  avoid_full_load:
    - docs/**

metadata:
  generated_by: spec-standards
  updated_at: "2026-04-26T00:00:00Z"
  evidence_mode: crg-first
```

字段边界：

- `project_intent.summary`：项目长期意图。
- `repo_profile`：项目类型、语言、框架、应用和包的轻量摘要；只描述稳定事实，不承载任务状态。
- `standards_summary`：多端项目的轻量规范摘要，用于让 plan / work / review 快速理解 common / frontend / backend / mobile / desktop 等维度。
- `principles`：长期稳定原则。
- `non_negotiables`：不可违反项。
- `review_defaults`：默认 review 关注方向。
- `loading_hints`：按需加载提示，只帮助 LLM 避免全量加载，不做路由规则。
- `metadata`：生成来源、更新时间和 evidence mode。

允许字段白名单：

```text
schema_version
repo_id
project_intent
repo_profile
standards_summary
principles
non_negotiables
review_defaults
loading_hints
metadata
```

禁止字段：

- `workflow_state`
- `quality_gate`
- `quality_gate_state`
- `routing`
- `routing_rules`
- `transition_rules`
- `verifier`
- `verifier_dispatch`
- `task_state`
- `task-specific requirements`

## 5.3 Phase 2A optional manual standards index

第三优先级，但不属于 MVP-A 默认输入。它不应该被推到共享规范之后，因为它直接承接“用户手动放规范文档、workflow 按需加载”的核心价值。

Phase 2A 推荐的本地结构：

```text
.spec-first/specs/manual/
  README.md
  api-style.md
  generated-assets-boundary.md
.spec-first/specs/manual-index.json
```

只有当 `repo-profile.yaml` 被真实维护，并且团队确实有多份人工规范时，才启用轻量索引：

```text
.spec-first/specs/manual-index.json
```

它只服务“按需找到人工规范摘要或正文”，不承载规则引擎、质量 gate 或自动刷新逻辑。

示例：

```json
{
  "version": 1,
  "standards": [
    {
      "id": "api-style",
      "path": ".spec-first/specs/manual/api-style.md",
      "title": "API style",
      "summary": "接口响应、错误处理和分页约定",
      "keywords": ["api", "response", "error", "pagination"],
      "applies_to_paths": ["src/**/controller/**"],
      "token_estimate": 900
    }
  ]
}
```

边界：

- 索引只服务加载决策。
- `summary` 是 LLM 的快速判断输入，不是 hard rule。
- `applies_to_paths` 只帮助候选命中，不做确定性 gate。
- 不包含 fail/warn gate。
- 不包含自动抽取规则。
- 不包含自动刷新逻辑。

## 5.4 Phase 2B+ optional pinned shared standards

第四优先级，也不属于 MVP 默认输入。

共享规范必须是 pinned ref，且必须通过 target repo 本地采用记录生效。共享规范只能提高输入质量，不能覆盖 repo-local confirmed fields。

## 5.5 CRG observed evidence

第五优先级。

来自：

```text
.spec-first/graph/graph.db
.spec-first/graph/code-navigation.json
spec-first crg workflow-context / locate / architecture / god-nodes
```

用途：

- 识别真实入口和模块边界。
- 识别测试入口和验证面。
- 识别高风险枢纽或高耦合模块。
- 发现 README 与代码事实的明显漂移。

限制：

- CRG evidence 只能证明“现状”。
- 不能直接证明“应该作为规范”。
- 必须经过 LLM 解释和人确认。

## 5.6 README / manifest / config

第六优先级。

用途：

- 建立项目身份。
- 获取项目描述和主语言。
- 判断用户可见目标。
- 补足 CRG 不覆盖的配置事实。

## 5.7 repo-profile 与 manual standards 的关系

`repo-profile` 是项目级规范摘要和默认约束；manual standards 是较长、较具体的人工规范文档。

默认加载顺序：

```text
先读 repo-profile
  ↓
如果任务需要更具体规范，Phase 2A 再按 manual-index 命中 summary
  ↓
只有 summary 不足以判断时，才读取命中的 manual standard 正文
```

MVP-A / MVP-B 只实现 `repo-profile` 主线，但 schema 和输出协议必须为 Phase 2A 的 manual standards 留口，避免后续为按需加载能力重做入口。

# 6. 输出模型

`$spec-standards` 必须先输出 preview。

Preview 的建议必须按类型分组，避免用户把“观察到的现状”“可写入规范”和“应拒绝候选”混在一起：

```text
project_intent suggestion
repo_profile suggestion
standards_summary suggestion
principle suggestion
non_negotiable suggestion
review_default suggestion
manual_standard candidate（Phase 2A+）
rejected candidate
uncertain candidate
conflict
```

## 6.1 Preview 结构

```markdown
# Repo Standards Suggestions

## Summary

- graph_state: ready | degraded | missing | unavailable
- fallback_mode: none | direct_repo_reads | crg_partial_plus_direct_repo_reads
- limitations:
  - graph unavailable, suggestions are lower confidence
- evidence_sources:
  - crg_workflow_context
  - crg_locate
  - README.md
  - package.json
- write_targets:
  - .spec-first/specs/repo-profile.yaml
  - .spec-first/specs/manual-index.json # Phase 2A+, only when enabled

## Suggested project_intent

### Summary

- confidence: medium
- evidence:
  - `README.md`
  - CRG `workflow_context`
- proposed_value:
  Short, stable project intent.

## Suggested repo_profile

- project_type: cli-tooling
- languages:
  - javascript
- apps:
  - cli

## Suggested standards_summary

### common

- Prefer explicit source/runtime boundaries.

## Suggested principles

### Prefer query-first code evidence over stale context packs

- confidence: high
- evidence:
  - `skills/spec-graph-bootstrap/SKILL.md`
  - `src/crg/workflow-context/stage.js`
- rationale:
  CRG 当前主路径已切到 query-first evidence，规范应提醒后续 workflow 优先使用现行 graph evidence。

## Suggested non_negotiables

### Do not edit generated runtime assets directly

- confidence: high
- evidence:
  - `AGENTS.md`
  - `src/cli/adapters/*`
- rationale:
  当前项目有 source/runtime mirror 边界，直接编辑生成资产会导致 drift。

## Rejected / uncertain candidates

### Use current repeated style X as mandatory rule

- confidence: low
- reason:
  只能证明代码里存在该模式，不能证明团队希望长期坚持。

## Conflicts

### Existing repo-profile says X, CRG observed evidence suggests Y

- source: repo-profile + CRG evidence
- handling:
  Report only. Human decides whether to update the confirmed field.

## Proposed patch

```yaml
repo_profile:
  project_type: cli-tooling
  languages:
    - javascript

standards_summary:
  common:
    - Prefer explicit source/runtime boundaries.

principles:
  - Prefer query-first code evidence over stale context packs

non_negotiables:
  - Do not edit generated runtime assets directly

  - source/runtime boundary integrity
```
```

## 6.2 写回规则

默认不写回。

只有用户确认后，才允许写：

```text
.spec-first/specs/repo-profile.yaml
```

MVP-B 只写 `repo-profile`。Phase 2A 才允许写 `manual-index.json` 或 manual standards adoption metadata，而且同样必须走确定性 helper。

写回必须遵守：

1. 不覆盖用户未授权字段。
2. 不写 runtime / workflow / gate state。
3. 不写任务临时要求。
4. 不把 inferred 低置信度建议写成 hard rule。
5. 保持文件短小，优先 30 到 60 行。

fallback 模式下也必须先展示 patch preview。`apply` 只表示“用户确认将某些建议写回”，不表示自动信任本轮全部推断。

apply 分成两类：

```text
user-confirmed apply
  用户显式声明的规范，或用户要求整理已有 repo-profile 文案。

evidence-derived apply
  基于 CRG / 源码 / 配置证据提出的规范建议。
```

具体规则：

- graph missing / unavailable：interactive 模式必须先让用户选择 bootstrap / direct-only / CRG-only stop；direct-only 时禁止 `evidence-derived apply`，但允许 `user-confirmed apply`。
- graph degraded：可用 CRG evidence 明确支撑的建议可以按正常流程进入 proposed patch；未被可用 CRG 支撑的建议必须标低置信度，并按 preview-only 处理。
- CRG-only 模式：没有可用 graph 时不进入 apply。

确定性写回边界：

- `$spec-standards` 负责语义建议、evidence、confidence、limitations、proposed patch 和用户确认。
- repo-profile 的 parse / schema validation / 字段 allowlist / merge / diff / write 必须由确定性 CLI 或内部 helper 执行。
- 该写回器必须保留用户未授权字段，拒绝 runtime / workflow / gate state，并只写 `.spec-first/specs/repo-profile.yaml` 的允许字段白名单。
- 如果 MVP-B 尚未实现该 helper，`$spec-standards --mode=apply` 只能展示 patch preview，不能直接改文件。

# 7. 规范索引的合理边界

本方案保留“按需加载规范”的价值，但不在第一版实现完整 `docs/specs` 平台。

## 7.1 MVP 先验证 repo-profile，但不把它当最终形态

MVP-A / MVP-B 先验证：

```text
.spec-first/specs/repo-profile.yaml
```

是否能改善：

- `spec-plan` 的方案质量。
- `spec-work` 的实现边界感。
- `spec-code-review` 的默认关注点。
- agent 对项目边界的理解。

`repo-profile` 不是最终完整规范资产系统。它是项目级摘要和默认约束，负责承载多端项目的基本 profile、稳定原则、不可违反项和默认 review 关注点。如果这个文件长期没人维护，说明扩展成多文件规范系统没有意义；如果它被真实使用，再进入 Phase 2A 的 manual standards index。

## 7.2 Phase 2A 优先考虑 manual standards index

manual standards index 不应排在共享规范之后。它是本地项目已有人工规范的轻量索引，不是规则平台。

推荐目录：

```text
.spec-first/specs/manual/
  README.md
  generated-assets-boundary.md
.spec-first/specs/manual-index.json
```

建议字段：

```json
{
  "version": 1,
  "standards": [
    {
      "id": "generated-assets-boundary",
      "path": ".spec-first/specs/manual/generated-assets-boundary.md",
      "title": "Generated assets boundary",
      "source": "manual",
      "summary": "Do not edit generated runtime assets directly.",
      "applies_to_paths": [
        ".claude/**",
        ".codex/**"
      ],
      "keywords": [
        "runtime",
        "generated",
        "drift"
      ],
      "token_estimate": 900
    }
  ]
}
```

注意：

- 索引只服务加载决策。
- 先用 `summary` 判断是否需要，再按需读取正文。
- 不包含 fail/warn gate。
- 不包含自动抽取规则。
- 不包含自动刷新逻辑。

## 7.3 不推荐的第一版能力

暂缓：

- `spec-first specs extract`
- `spec-first specs check --changed`
- `spec-first specs refresh --changed`
- `spec-first specs sync`
- `rules-map.json`
- inferred rule promotion
- conflict auto-resolution
- embedding 检索

这些能力都可能把轻量输入层推向规范平台和质量 gate。

## 7.4 Phase 2B+：团队共享与外挂 Git 工程只保留原则

规范需要方便团队之间共享，但这不是 MVP 默认范围，也不应挤占主线。正文只保留原则，详细 attach / shared standards / external target repo 设计见附录 A。

原则：

- 共享规范是只读输入来源，不是 hard gate。
- 共享规范必须通过 target repo 本地采用记录生效。
- repo-local `repo-profile` 和 manual standards 优先于 shared standards。
- 共享规范冲突时只报告冲突，不自动裁决。
- 外挂 target repo 的 CRG 与规范产物落在 target repo 本地，而不是 `spec-first` 工具仓库。
- 远程 Git resolver、clone / fetch、pinned ref cache 和 auth failure reporting 属于 Phase 2B+，不得混入 `init` 或 MVP 主线。

# 8. init 后执行流程

流程分为默认核心路径、直接入口路径与可选团队共享路径。默认核心路径不包含 `standards attach`；共享规范挂载属于 Phase 2+，只有团队已经有独立 standards repo 且用户显式要求采用时才进入。

## 8.1 默认核心路径（MVP）

MVP 按“确定性初始化 → 准备代码事实 → 语义规范建议 → 人工确认 → repo-profile 记录 → workflow 消费”推进：

```text
spec-first init
  ↓
$spec-graph-bootstrap
  ↓
完成提示 next step: $spec-standards（不自动执行）
  ↓
$spec-standards preview
  ↓
人工确认 apply
  ↓
repo-profile 记录
  ↓
$spec-plan / $spec-work / $spec-code-review
```

`spec-first init` 的职责：

- 安装 Claude / Codex runtime assets。
- 写 instruction managed block。
- 写 developer profile。
- 准备本项目最小 spec-first 工作区。

不做：

- 不拉取远程共享规范仓库。
- 不执行 `standards attach`。
- 不判断当前项目应采用哪些团队规范。
- 不基于代码事实生成长期规范。
- 不写 workflow / gate / routing 状态。

`init` 必须保持确定性、轻量和可重复执行。

## 8.2 `$spec-graph-bootstrap`

职责：

- 为当前 repo / target repo（MVP 为当前 repo 或显式本地目录）构建 CRG 代码事实层。
- 生成 `.spec-first/graph/graph.db`。
- 验证 query-first 能力。
- 为 `$spec-standards`、plan、work、review 提供 observed evidence。
- 结束时输出 `$spec-standards` 作为推荐下一步。

不做：

- 不维护规范来源。
- 不写 repo-profile。
- 不判断团队长期规范。
- 不自动执行 `$spec-standards`。

完成提示建议：

```text
Graph ready. Next recommended step:
$spec-standards
```

## 8.3 `$spec-standards`

职责：

- 读取用户显式说明。
- 读取当前 repo / target repo（MVP 为当前 repo 或显式本地目录）本地 repo-profile。
- 读取 CRG observed evidence。
- 读取 README / manifest / config。
- 输出按类型分组的 preview suggestions。

MVP 不默认读取 target repo 本地 manual standards index，也不读取 pinned shared standards。若用户显式给出某份规范文件或摘录，可作为“用户显式说明”的补充材料读取；不实现自动索引解析、命中或加载。

`$spec-standards` 是独立入口，可以在 `$spec-graph-bootstrap` 之后运行，也可以由用户直接运行。直接运行时必须先检测 CRG readiness，再决定使用 CRG、partial CRG + direct reads，或 direct-only fallback。

默认行为：

```text
graph ready
  → 使用 CRG evidence 增强建议

graph degraded
  → 使用可用 CRG evidence
  → direct repo reads 补足缺口
  → 只下调未被可用 CRG evidence 支撑的 claim
  → 输出带 limitations 的 preview

graph missing / unavailable
  → interactive 模式给出 bootstrap / direct-only preview / CRG-only stop 三选一
  → headless / 无交互模式才默认 direct-only preview
  → direct-only 输出低置信度 preview
  → 禁止 evidence-derived apply
  → 允许用户显式声明规范经确认写回

用户指定 CRG-only
  → 没有 graph 时停止
  → 要求先运行 $spec-graph-bootstrap .
```

preview 必须包含：

- `graph_state`
- `fallback_mode`
- `limitations`
- `evidence_sources`
- suggested `project_intent`
- suggested `repo_profile`
- suggested `standards_summary`
- suggested `principles`
- suggested `non_negotiables`
- suggested `review_defaults`
- rejected / uncertain candidates
- conflicts
- proposed patch

## 8.4 人工确认 apply

确认后只允许写 repo-local 规范输入：

```text
.spec-first/specs/repo-profile.yaml
```

写回禁止：

- workflow state
- quality gate state
- routing rules
- 当前任务临时需求
- 低置信度 inferred rule

fallback apply 规则：

- graph missing / unavailable 时，interactive 模式必须先让用户选择 bootstrap / direct-only / CRG-only stop；direct-only preview 禁止写入代码推断规范，但允许用户显式声明规范或已有 repo-profile 的纯文本整理经确认写回。
- graph degraded 时，只有可用 CRG evidence 或用户显式说明支撑的建议才能进入 proposed patch。
- 每次 apply 前都必须展示 patch preview。

## 8.5 `$spec-standards` 直接入口路径

为了减少用户要记忆的命令，`$spec-standards` 必须能作为直接入口运行：

```text
$spec-standards
  ↓
检测 graph readiness
  ↓
ready: 使用 CRG evidence 生成 preview
degraded: 使用可用 CRG + direct reads 生成带 limitations 的 preview
missing / unavailable: interactive 模式选择 bootstrap / direct-only preview / CRG-only stop；headless 默认 direct-only preview，并禁止 evidence-derived apply
```

这条路径解决“用户没有先跑 `$spec-graph-bootstrap`”的情况，但不会把 `$spec-graph-bootstrap` 变成隐藏副作用。interactive 模式必须先让用户选择运行 bootstrap、继续低置信度 preview，或 CRG-only stop；headless / 无交互模式才默认 direct-only preview，并禁止 evidence-derived apply。只有用户显式要求 CRG-only 时才停止。

## 8.6 可选团队共享路径（Phase 2B+）

当团队已经有共享 standards repo，并且用户希望当前项目采用其中一部分规范时，才运行 `standards attach`。它不属于 `init` 的默认后续步骤，也不属于 MVP-A / MVP-B 主线。

```text
spec-first init
  ↓
spec-first standards attach <standards-repo> --ref <sha-or-tag>
  ↓
$spec-graph-bootstrap
  ↓
$spec-standards preview
  ↓
人工确认 apply
  ↓
repo-profile 记录
  （shared-standards adoption record 已由 standards attach 写入）
  ↓
$spec-plan / $spec-work / $spec-code-review
```

详细边界、命令示例与 adoption record 见附录 A。正文只保留两条约束：

- `spec-first standards attach` 写 `.spec-first/specs/shared-standards.yaml`，`$spec-standards apply` 默认只写 `repo-profile`。
- attach resolver 先支持本地 standards repo 路径，再考虑 Git URL fetch / pinned ref cache；skill 不直接执行隐式 clone / pull。

## 8.7 后续 workflow 消费

`$spec-plan`：

- 读取 `project_intent.summary`、`repo_profile`、`standards_summary`、`principles`、`non_negotiables`。
- 读取 CRG before-plan / workflow-context evidence。
- 把这些作为 planning input；关键方案选择需要说明是否受这些约束影响。
- 如果方案与 `non_negotiables` 冲突，必须显式标出冲突并请求用户裁决。
- 不自动修改 repo-profile。

`$spec-work`：

- 读取 `non_negotiables`。
- 读取与当前任务相关的 `principles` / `standards_summary`。
- 结合 CRG before-work evidence 判断影响面。
- 编码前只列出本任务相关的 3 到 5 条约束，不展开成全量 checklist。
- 不自动写回规范。

`$spec-code-review`：

- 读取 `review_defaults`。
- 读取 `non_negotiables`。
- 读取 CRG before-review / review-context evidence。
- LLM 结合 diff 和测试结果产出 review findings。
- findings 必须标注主要来源：`repo-profile`、`CRG evidence`、`diff evidence` 或 `test evidence`。
- 不把 repo-profile 直接当 fail gate。

Phase 2A 才能增加“命中的 manual standards summary”；Phase 2B+ 才能增加 shared standards summary。即使启用，也仍由 LLM 结合任务和 diff 判断，不做 hard gate。

## 8.8 降级链路

如果用户没有跑 `$spec-graph-bootstrap`：

```text
$spec-standards
  ↓
检测 graph missing
  ↓
interactive 选择：运行 $spec-graph-bootstrap / 继续 direct-only preview / CRG-only stop
  ↓
选择 direct-only 时只用 direct repo reads
  ↓
输出低置信度 preview
  ↓
禁止写入代码推断规范；用户显式声明规范仍可经确认写回
```

如果用户要求高质量或 CRG-only：

```text
$spec-standards
  ↓
检测 graph missing
  ↓
停止
  ↓
要求先运行 $spec-graph-bootstrap .
```

如果 graph degraded：

```text
$spec-standards
  ↓
检测 graph degraded
  ↓
读取可用 CRG evidence
  ↓
direct repo reads 补足缺口
  ↓
输出带 limitations 的 preview
  ↓
仅下调未被可用 CRG 支撑的 claim
```

# 10. 质量保障

## 10.1 好规范的判定标准

一条规范只有满足以下条件，才值得写入：

1. 长期有效，不是当前任务临时要求。
2. 可执行，可判断，不是空泛价值观。
3. 有证据支撑，或由用户明确声明。
4. 能提升 plan / work / review 的决策质量。
5. 不与已有规范或项目事实明显冲突。

## 10.2 坏规范信号

以下内容不应写入：

- “代码要高质量”
- “注意安全”
- “保持简洁”
- “遵循最佳实践”
- “当前代码里大量这么写，所以必须继续这么写”
- “修改某类文件必须自动触发某 workflow”
- “命中某规则必须 fail”

## 10.3 Evidence 等级

建议使用轻量 evidence 分类：

```text
confirmed
  用户明确说明，或已有人工规范明确写入。

observed
  CRG / 源码 / 测试 / 配置中可重复观察到。

inferred
  LLM 基于多项证据推断，但未被人确认。

uncertain
  证据不足，不应写入，只能列为候选问题。
```

写回策略：

```text
confirmed → 可写入
observed + rationale strong → preview 中建议，人确认后写入
inferred → 默认不写入，除非用户确认
uncertain → 不写入
```

graph missing / unavailable 时，`confirmed` 仍然可以通过 `user-confirmed apply` 写入；`observed` / `inferred` 的代码推断不允许写入。这样保证没有 CRG 时不会把代码现状误升格为规范，同时不阻断用户显式声明的长期约束。

# 11. MVP

MVP 拆成两段，避免把 preview 质量验证、写回和 workflow 消费一次性绑死。

## 11.1 MVP-A：preview-only

目标：新增独立 `$spec-standards`，先证明 CRG-first preview 的质量值得做。

必做：

1. 新增 `$spec-standards` workflow skill，作为规范生成的唯一公开入口。
2. `$spec-standards` 启动时检查 CRG readiness。
3. graph ready 时优先读取 `workflow-context`、`architecture`、`code-navigation.json` 等结构性输出，再基于其中出现的具体文件、模块或 symbol 执行 `locate` / `explain` / `path`。
4. `locate` 返回空时，明确记录 limitation，并回退 direct repo reads，不伪造 CRG evidence。
5. graph missing / unavailable 时提示 `$spec-graph-bootstrap .`；interactive 模式让用户选择 bootstrap / direct-only / CRG-only stop，headless 才默认 direct-only preview 并禁用 apply。
6. graph degraded 时使用可用 CRG evidence + direct repo reads，只下调未被可用 CRG 支撑的 claim。
7. 用户显式要求 CRG-only 时停止，不降级。
8. 读取现有 repo-profile confirmed fields。
9. 输出 preview suggestions，带 evidence / confidence / proposed patch / limitations。
10. `$spec-graph-bootstrap` 完成时增加 next-step guidance，提示 `$spec-standards`，但不自动执行。
11. 不写回任何文件。
12. 不接入 `spec-plan` 消费。

CRG value validation：

- 至少选择 2 到 3 个真实仓库作为样本。
- 对同一仓库、同一用户输入分别生成 direct-only preview 与 CRG-first preview。
- 记录并比较：
  - factuality：建议是否准确引用真实入口、模块、测试和配置。
  - evidence coverage：每条建议是否有可追溯证据。
  - actionable suggestions：可被人确认采用的建议数量。
  - rejected / uncertain candidates：被拒绝或不确定候选的比例。
  - human adoption rate：人工最终采纳率。
- 只有 CRG-first 明显优于 direct-only，才进入 MVP-B。

## 11.2 MVP-B：确认写回与可选消费

前置条件：MVP-A 的 preview 质量通过真实项目验证。

必做：

1. 用户确认后只写 `.spec-first/specs/repo-profile.yaml` 的允许字段。
2. fallback apply 必须展示 patch preview。
3. graph missing / unavailable 的 direct-only preview 禁用 apply。
4. 引入确定性写回器：
   - `$spec-standards` 只负责语义建议、evidence、confidence、limitations 和用户确认。
   - CLI 或内部 helper 负责读取 YAML、校验 `schema_version`、执行字段 allowlist、合并未授权字段、输出 diff 并写入 `.spec-first/specs/repo-profile.yaml`。
   - helper 未实现前，`--mode=apply` 只能展示 patch，不能直接写文件。
5. `spec-plan` 可把非空 repo-profile 字段作为 optional planning input。
6. `spec-work` / `spec-code-review` 可读取 `non_negotiables`、`review_defaults` 等 repo-profile 字段作为 context。
7. workflow 消费不把 repo-profile 展开成 checklist，也不把它变成 hard gate。

## 11.3 MVP 不做 / Phase 2+

MVP 不做：

1. 不新增完整 `specs` CLI。
2. 不做 `docs/specs` 多目录初始化。
3. 不做自动抽取规则。
4. 不做自动刷新规范。
5. 不做规范检查 hard gate。
6. 不同步规范正文到 host instruction files。
7. 不把 CRG query 输出直接写成规范。
8. 不实现 manual standards index 的自动解析、命中或加载。
9. 不实现 shared standards repo 的 attach / fetch / adoption。
10. 不实现外挂 target repo 的完整 resolver、clone、pull、push 或跨 repo 写回。

Phase 2+ 才考虑：

- `spec-first standards attach`
- `.spec-first/specs/manual-index.json`
- `.spec-first/specs/shared-standards.yaml`
- pinned shared standards adoption
- 外挂本地 target repo 支持
- full Git resolver / clone / fetch / pinned ref cache
- manual/shared standards 对 plan / work / review 的自动命中与摘要加载

## 11.4 Done signals

MVP-A 完成后应能证明：

1. CRG-first preview 在真实仓库样本上明显优于 direct-only preview。
2. 无 graph 时，`$spec-standards` 会提示 `$spec-graph-bootstrap .`；interactive 模式让用户选择 bootstrap / direct-only / CRG-only stop，headless 才默认 direct reads。
3. CRG-only 模式下没有 graph 会停止，而不是生成伪 CRG evidence。
4. preview 中每条建议都有 evidence、confidence 和 limitations。
5. `locate` / `explain` / `path` 的失败会被表达为 limitation，而不是被隐藏。

MVP-B 完成后应能证明：

1. 写回文件没有 runtime / workflow / gate 字段。
2. repo-profile 写回由确定性 helper 完成 parse / validate / allowlist / merge / diff / write。
3. fallback apply 不会把低置信度 inferred 建议静默写入。
4. `spec-plan` 只消费规范输入，不把它变成 rules engine。

# 12. 推荐命令与交互

## 12.1 首次生成

```bash
$spec-graph-bootstrap .
```

`$spec-graph-bootstrap` 完成后只提示下一步，不自动执行：

```text
Graph ready. Next recommended step:
$spec-standards
```

然后用户运行：

```text
$spec-standards
```

也可以跳过 bootstrap 直接运行：

```text
$spec-standards
```

直接运行时，`$spec-standards` 自行检测 graph readiness；没有可用 graph 时提示 `$spec-graph-bootstrap`。interactive 模式让用户选择运行 bootstrap、继续低置信度 preview，或 CRG-only stop；headless / 无交互模式才默认 direct-only preview，并禁用 apply。

`$spec-standards` 内部建议执行：

```bash
spec-first crg workflow-context --stage=plan --repo=.
spec-first crg architecture --repo=.
# 读取 .spec-first/graph/code-navigation.json 中的 suggested queries / entrypoints
spec-first crg locate --repo=. --query="<workflow-context 或 code-navigation 给出的具体文件/模块/symbol>" --limit=10 --detail=minimal
spec-first crg explain --repo=. --node="<locate 返回的具体 id/name/path>" --detail=standard
spec-first crg path --repo=. --from="<具体 source node>" --to="<具体 target node>" --detail=standard
```

查询策略：

1. 先用 `workflow-context`、`architecture`、`code-navigation.json` 取得结构性事实和具体候选。
2. 再用 `locate` / `explain` / `path` 验证具体文件、模块、symbol 或关系。
3. 不把 `public entrypoints` 这类泛查询作为主要证据来源；如果泛查询返回空，必须标注 limitation，并回退 direct repo reads。

## 12.2 复核现有规范

```text
$spec-standards --mode=preview
```

只输出建议，不写文件。

## 12.3 应用建议（MVP-B 起）

MVP-A 不提供写回；该命令从 MVP-B 开始实现。MVP-B 必须先有确定性写回器，否则只能展示 patch preview。

```text
$spec-standards --mode=apply
```

应用前必须展示 patch preview。

# 13. 最终架构判断

这条路线值得做，但只能按轻量输入层做。

应该保留：

- `$spec-graph-bootstrap` 作为推荐上游。
- `$spec-standards` 作为新增独立规范生成入口。
- CRG observed evidence。
- 规范建议 preview。
- 人工确认写回。
- 按需加载规范输入。

必须避免：

- 多目录规范平台。
- 自动抽取规则直接生效。
- 自动刷新人工规范。
- CLI hard gate。
- `AGENTS.md` / `CLAUDE.md` 规范正文膨胀。
- 把 CRG 事实当成规范真相。
- `$spec-graph-bootstrap` 自动调用 `$spec-standards` 的强编排。

最终定版：

> **先用 `$spec-graph-bootstrap` 提升事实输入质量，再用独立 `$spec-standards` 让 LLM 基于事实提出规范建议，最后由人确认哪些建议能成为长期规范。**

这符合 spec-first 的长期方向：脚本准备确定性事实，LLM 做语义判断，人确认长期约束。

# CRG-first 规范资产系统方案

> 文档状态：重写方案
> 修订日期：2026-04-27
> 核心裁决：新增独立 `$spec-standards` workflow skill，并配套 `spec-first specs` CLI 能力；`$spec-graph-bootstrap` 负责准备工程事实，`$spec-standards` 基于 CRG evidence 生成规范文档草案、索引草案和加载计划；正式规范资产存放在 `docs/specs/**`，生成草案和运行证据存放在 `.spec-first/standards/runs/**`；MVP-A 必须生成规范草案而不是只输出 repo-profile preview；MVP-B 才允许人工确认后 promote 到正式 `docs/specs/**`。

---

# 1. 一句话结论

`spec-first` 需要的是 **CRG-first 的规范资产生成、索引与按需加载系统**，不是只生成一个轻量 `repo-profile` 摘要。

正确闭环是：

```text
$spec-graph-bootstrap
  构建 CRG 工程事实：语言、框架、入口、模块、依赖、测试、风险节点
        ↓
$spec-standards
  基于 CRG evidence + README/manifest/config + 人工输入生成规范文档草案
        ↓
preview-first proposal
  草案落到 .spec-first/standards/runs/<run-id>/，不自动生效
        ↓
human confirm / promote
  人确认后才写入 docs/specs/** 正式规范资产
        ↓
index / resolve
  生成机器索引，后续 plan / work / review 按需加载规范
        ↓
check / refresh
  作为审查辅助和刷新建议，不做默认 hard gate
```

关键判断：

```text
CRG = 工程事实层
docs/specs/** = 正式规范资产源
.spec-first/standards/runs/** = 本次生成草案和证据
_index/** = 机器索引，可重建
custom/** = 人工规范，最高优先级
LLM = 语义抽取、归纳、resolve 决策
脚本 / CLI = 目录、索引、schema、diff、写回、缓存等确定性执行
人 = 长期规范裁决
```

这保留原始目标：

```text
识别项目 → 抽取规范 → 手动接入 → 建立索引 → LLM 按需加载 → 检查 → 刷新
```

同时避免跑偏为：

```text
扫描代码 → 自动把现状升格为规范 → 自动写入 → 自动 fail gate
```

# 2. 设计基线

本方案必须遵循 `docs/10-prompt/项目角色.md` 的判断基线：

- Light contract
- Explicit boundaries
- Let the LLM decide
- Scripts prepare, LLM decides
- Deterministic execution belongs to scripts
- Semantic analysis belongs to LLM

因此：

```text
CRG 只能证明“代码库现在是什么”
CRG 不能证明“团队长期应该怎么做”
LLM 可以提出规范草案
LLM 不能静默写入正式规范
人确认后规范才成为长期输入
CLI/helper 负责确定性写回和索引
```

参考实践：

- Trellis 的 `.trellis/spec/**` 表明规范资产应是可读、可版本化、按领域组织的 Markdown 文档，而不是单个 YAML 摘要。
- spec-kit 的 constitution / specs / plan / tasks 结构表明长期原则应作为稳定输入源，被后续 workflow 读取，而不是混进每次任务产物。

# 3. 核心目标

本方案解决的问题：

```text
项目中有很多隐性规范：
- 代码里有事实
- 配置里有约束
- CI 里有质量信号
- 团队里有习惯
- 人工文档散落在不同位置

AI 开发时如果没有规范基线，就会：
- 凭经验生成代码
- 误判项目架构
- 忽略团队约定
- 全量读文档浪费 token
- 让 plan / work / review 首稿质量不稳定
```

`spec-standards` 的目标是：

1. 基于 CRG facts 识别项目端类型、语言、框架和模块边界。
2. 从代码、配置、测试、CI、README 中生成规范文档草案。
3. 支持用户手动放入规范文档，并保证 manual/custom 优先。
4. 生成机器可读索引，避免后续 workflow 全量读取 `docs/specs/**`。
5. 让 LLM 根据任务、文件、profile、关键词和 token budget 决策加载哪些规范。
6. 为 check / refresh 提供规范资产基础，但默认不做 hard gate。
7. 让规范跟随仓库版本化，方便团队共享。

# 4. 职责边界

## 4.1 `$spec-graph-bootstrap`

职责：

- 构建本地 CRG graph。
- 识别真实入口、模块、依赖、测试、配置、风险节点。
- 提供 query-first evidence。
- 完成后提示 `$spec-standards` 作为推荐下一步。

产物：

```text
.spec-first/
  graph/
    graph.db
    graph-index-status.json
    code-navigation.json
    graph-operations.jsonl
    repo-topology.json
```

不做：

- 不生成规范文档。
- 不判断哪些现状应成为规范。
- 不写 `docs/specs/**`。
- 不自动调用 `$spec-standards`。

完成提示：

```text
Graph ready. Next recommended step:
$spec-standards
```

## 4.2 `$spec-standards`

职责：

- detect：识别项目 profiles。
- extract：生成规范文档草案。
- intake：接入用户手动规范。
- index：生成规范索引。
- resolve：为任务决策加载哪些规范。
- check：基于规范进行审查辅助。
- refresh：根据变更生成规范刷新建议。
- sync：同步入口说明到 AGENTS.md / CLAUDE.md 等 host 文件，但不复制规范正文。

`$spec-standards` 是公开 workflow skill；确定性文件操作由 `spec-first specs` CLI/helper 执行。

不做：

- 不把所有重复代码模式直接写成正式规范。
- 不覆盖 manual/custom 规范。
- 不自动解决规范冲突。
- 不把 inferred 规范变成默认 fail gate。
- 不把规范正文膨胀进 AGENTS.md / CLAUDE.md。

## 4.3 `spec-first specs` CLI/helper

职责：

- 创建目录。
- 解析 Markdown frontmatter。
- 生成索引。
- 校验 schema。
- 输出 diff。
- 执行 promote。
- 生成 resolve result / JSONL context plan。
- 写 reports。

这些是确定性流程，应由 CLI/helper 做，不让 LLM 模拟。

## 4.4 人

职责：

- 判断 extracted / inferred 草案是否代表团队长期规范。
- 修改、确认或拒绝规范草案。
- 裁决冲突。
- 决定哪些规范进入 `docs/specs/**` 正式资产。

# 7. 规范来源与优先级

规范来源：

```text
manual      用户手动编写或放入
imported    外部规范导入
confirmed   用户确认后的 extracted 规范
extracted   LLM 基于 CRG evidence 生成的草案
template    系统模板初始化
generated   索引、报告、证据等系统产物
```

优先级：

```text
custom/manual
  > manual
  > imported
  > confirmed
  > extracted inferred
  > template
```

规则：

- `docs/specs/custom/**` 最高优先级。
- `source=manual` 不允许被 refresh 自动覆盖。
- `source=extracted` 默认先进入 `.spec-first/standards/runs/**` 草案区。
- `source=confirmed` 才允许进入正式规范并参与 check 的高置信度判断。
- `uncertain` / `conflict` 只能进入 evidence 或 reports，不进入正式规则。

# 8. 规范文件格式

正式规范文件使用 Markdown + frontmatter。无 frontmatter 的 manual 文档也必须支持，但会在 validate 报告中提示补齐。

示例：

```markdown
---
spec_id: backend-api
title: 后端接口规范
source: confirmed
level: L3
scope:
  - backend
profiles:
  - backend-java-spring
categories:
  - api
  - error-handling
keywords:
  - controller
  - dto
  - response
applies_to_paths:
  - "src/main/java/**/controller/**"
priority: 80
status: active
owners:
  - backend-team
---

# 后端接口规范

## Summary for Agent

- 新增 HTTP API 应遵循现有 Controller 风格。
- Controller 只负责参数接收、鉴权入口和响应封装。
- 复杂业务逻辑应下沉到 Service。
- 错误处理应参考 `backend/error-handling.md`。

## Rules

### RULE-BACKEND-API-001 统一响应格式

- 状态：confirmed
- 层级：L3 Project Spec
- 适用范围：backend/java-spring
- 类别：api
- 优先级：high
- 规则：新增 HTTP API 应使用项目统一响应对象。
- 证据：
  - `src/main/java/com/example/common/Result.java`
  - `src/main/java/com/example/user/UserController.java`
- 置信度：high
- 后续 Agent 约束：生成 Controller 时不得自行创建新的响应结构，除非 custom 规范明确覆盖。
- 检查方式：
  - llm_review_api_style
```

无 frontmatter 的用户文档：

```text
docs/specs/backend/payment.md
```

index helper 应根据路径推断：

```json
{
  "source": "manual",
  "scope": ["backend"],
  "categories": ["payment"],
  "priority": 85,
  "metadata_inferred": true
}
```

# 10. Profile Registry

Profile 定义：

```text
1. 如何识别项目
2. 需要扫描哪些文件
3. 应生成哪些规范草案
4. 路径对应哪些规范
5. resolve/check 时如何匹配规则
```

示例：

```yaml
id: backend-java-spring
name: Java Spring Backend
type: backend

detect:
  files:
    - pom.xml
    - build.gradle
  contains:
    - "@SpringBootApplication"
    - "@RestController"
    - "spring-boot-starter-web"

scan:
  required:
    - pom.xml
    - src/main/java/**
    - src/main/resources/application*.yml
  optional:
    - Dockerfile
    - .github/workflows/**
    - src/test/java/**

specs:
  required:
    - backend/java.md
    - backend/java-spring.md
    - backend/api.md
    - backend/error-handling.md
    - backend/database.md
    - backend/testing.md
  optional:
    - backend/auth.md
    - common/security.md
    - common/ci-cd.md

resolve:
  path_rules:
    - pattern: "**/controller/**"
      load:
        - backend/api.md
        - backend/error-handling.md
        - backend/java-spring.md
    - pattern: "**/service/**"
      load:
        - backend/java-spring.md
        - backend/error-handling.md
```

Profile detection 的事实来源优先使用 CRG Evidence Adapter，再补 README / manifest / config direct reads。

# 11. LLM 决策加载

目标：

```text
任务理解 → 索引匹配 → 加载计划 → 精准读取
```

输入：

- 任务描述。
- 任务类型，可选。
- 变更文件列表，可选。
- detected profiles。
- `specs-index.json`。
- `rules-map.json`。
- custom/manual 规范摘要。

输出：

```json
{
  "task_id": "login-api",
  "task_type": "backend_api_development",
  "profiles": ["backend-java-spring"],
  "token_budget": 12000,
  "estimated_tokens": 6200,
  "load_full": [
    {
      "path": "docs/specs/custom/team-overrides.md",
      "reason": "人工规范优先级最高",
      "priority": 100
    },
    {
      "path": "docs/specs/backend/api.md",
      "reason": "任务涉及新增 HTTP API",
      "priority": 90
    }
  ],
  "load_summary": [
    {
      "path": "docs/specs/backend/error-handling.md",
      "reason": "登录接口需要处理失败场景",
      "priority": 75
    }
  ],
  "load_reference": [
    {
      "path": "docs/specs/common/ci-cd.md",
      "reason": "非当前任务核心，只在涉及 CI 时引用"
    }
  ],
  "excluded": [
    {
      "path": "docs/specs/frontend/component.md",
      "reason": "任务不涉及前端组件"
    }
  ]
}
```

加载策略：

```text
load_full
  必须全文加载。

load_summary
  只加载 Summary for Agent。

load_reference
  仅在需要时引用，不主动加载。
```

优先级：

```text
1. custom/manual 且命中任务 → load_full
2. changed files 命中的规范 → load_full
3. 直接任务类别规范 → load_full
4. 依赖规范 → load_summary
5. 通用规范 → load_summary
6. 未命中规范 → skip
```

# 13. Graph 缺失与降级

`$spec-standards` 可以直接运行，但必须先检测 CRG readiness。

```text
graph ready
  使用 CRG evidence 生成 detect/extract 草案。

graph degraded
  使用可用 CRG evidence + direct repo reads，并在 preview 中标注 limitations。

graph missing / unavailable
  interactive 模式让用户选择：
    1. 先运行 $spec-graph-bootstrap
    2. 继续 direct-only 低置信度草案
    3. CRG-only stop
  headless 模式默认 direct-only proposal，但必须标记 low confidence。
```

direct-only 规则：

- 可以生成 proposal 草案。
- 不允许把基于代码推断的规范直接 promote。
- 用户显式声明的 manual 规范可以写入 `docs/specs/custom/**`。
- preview / evidence 必须说明缺少 CRG evidence。

# 14. Check 与 Refresh

## 14.1 Check

`spec-first specs check` 是审查辅助，不是默认 hard gate。

流程：

```text
1. 获取 changed files
2. 调用 specs resolve
3. 加载 check.jsonl 中的规范
4. 执行确定性检查，可选
5. 执行 LLM review
6. 运行项目命令：lint / test / typecheck，可选
7. 输出 docs/specs/reports/spec-check-report.md
```

默认语义：

```text
manual critical/high → 可以作为阻塞建议，但仍由 workflow/user 裁决
confirmed → warning 或 blocking suggestion
inferred → warning
uncertain/conflict → 只提示人工确认
```

## 14.2 Refresh

`spec-first specs refresh` 不覆盖人工规范。

模式：

```bash
spec-first specs refresh --changed
spec-first specs refresh --full
spec-first specs refresh --profile backend-java-spring
spec-first specs refresh --index-only
```

原则：

- 不覆盖 `docs/specs/custom/**`。
- 不覆盖 `source=manual`。
- confirmed 规则谨慎修改，需要 preview。
- inferred 可根据新证据生成更新草案。
- conflict 需要人工确认。
- 每次刷新必须生成报告。
- 刷新后必须重新生成 index。

# 15. Agent Context Sync

`docs/specs/**` 是唯一规范源。Host instruction 文件只是入口说明和加载策略，不承载规范正文。

可以同步到：

```text
AGENTS.md
CLAUDE.md
.cursor/rules/spec-first.mdc
.github/copilot-instructions.md
```

同步内容只包括：

```text
1. 规范入口路径
2. 索引路径
3. 加载策略
4. custom/manual 优先原则
5. 不全量加载原则
6. 发现规范冲突时如何报告
```

禁止：

- 把 `docs/specs/**` 正文复制进 host instruction。
- 让多个 host 文件成为规范真源。
- 自动覆盖用户手写的 host instruction 内容。

# 17. MVP 分阶段

## 17.1 MVP-A：CRG-first detect + extract proposals

目标：证明 CRG-first 能生成高质量规范文档草案。

必做：

1. 新增 `$spec-standards` workflow skill。
2. `$spec-standards` 启动时检测 CRG readiness。
3. graph ready 时使用 CRG Evidence Adapter 读取项目事实。
4. graph missing 时提示 `$spec-graph-bootstrap`，但允许 direct-only low-confidence proposal。
5. 生成 `.spec-first/standards/runs/<run-id>/preview.md`。
6. 生成 `.spec-first/standards/runs/<run-id>/detected-profiles.json`。
7. 生成 `.spec-first/standards/runs/<run-id>/extract-plan.json`。
8. 生成 `.spec-first/standards/runs/<run-id>/evidence-map.json`。
9. 生成 `.spec-first/standards/runs/<run-id>/drafts/**/*.md`。
10. 生成 `.spec-first/standards/runs/<run-id>/index-draft/specs-index.json`。
11. 不写 `docs/specs/**` 正式规范。
12. `$spec-graph-bootstrap` 完成后只提示 `$spec-standards`，不自动调用。

MVP-A 不做：

- 不 promote。
- 不 check。
- 不 refresh。
- 不 sync host instruction。
- 不实现 full profile registry。
- 不支持远程共享规范仓库。

## 17.2 MVP-B：promote 到正式规范资产

前置：MVP-A 草案质量通过真实仓库验证。

必做：

1. `spec-first specs init` 创建 `docs/specs/**` 基础结构。
2. `spec-first specs promote <run-id>` 展示 diff，经用户确认后写入 `docs/specs/**/*.md`。
3. `spec-first specs index` 生成 `_index/specs-index.json`、`rules-map.json`、`profiles.json`。
4. manual/custom 文档优先，promote 不覆盖 manual。
5. 无 frontmatter manual 文档可被索引，并提示 normalize。
6. 所有写回由确定性 helper 完成 parse / validate / allowlist / merge / diff / write。

## 17.3 MVP-C：resolve 接入 workflow

必做：

1. `spec-first specs resolve --task --files` 输出 resolve plan。
2. 为 `$spec-plan`、`$spec-work`、`$spec-code-review` 提供 optional context。
3. 支持 load_full / load_summary / load_reference。
4. 输出 `.spec-first/tasks/<task-id>/resolve-result.json` 和 JSONL context plan。
5. 不把 resolve 结果变成 hard gate。

## 17.4 后续阶段

Phase 2：

- `spec-first specs check --changed`
- `spec-first specs refresh --index-only`
- `spec-first specs refresh --changed`
- `spec-first specs validate`
- `spec-first specs normalize`

Phase 3：

- shared standards repo attach
- pinned ref adoption
- remote Git resolver / clone / fetch
- PR history learning
- embedding 检索

# 18. 质量验证

MVP-A 必须用真实仓库验证：

```text
CRG-first proposal vs direct-only proposal
```

指标：

- factuality：是否准确引用真实入口、模块、测试、配置。
- evidence coverage：每条规范是否有证据。
- draft usefulness：草案是否接近可人工确认。
- uncertainty handling：是否正确区分 inferred / uncertain / conflict。
- token efficiency：是否比全量读代码更省。
- human adoption rate：最终被 promote 的比例。

只有 CRG-first 明显优于 direct-only，才进入 MVP-B。

# 19. 最终架构判断

这条路线值得做，但必须坚持两条边界：

```text
1. spec-standards 是规范资产系统，不是 repo-profile 摘要生成器。
2. spec-standards 不是强规则引擎，不替代 LLM 和人的判断。
```

应该保留：

- CRG-first。
- preview-first proposal。
- human-confirmed promote。
- `docs/specs/**` 作为正式规范资产源。
- `_index/**` 作为机器索引。
- manual/custom 最高优先级。
- LLM 决策加载，避免全量读取。
- check / refresh 作为后续辅助能力。

必须避免：

- 把代码现状直接升格为规范。
- 自动覆盖人工规范。
- 自动解决冲突。
- 一开始就建设复杂规则引擎。
- 把规范正文同步进 host instruction。
- 让 `repo-profile` 替代规范文档。

最终定版：

> **`spec-standards` 是 spec-first 的项目工程规范资产层：它使用 `spec-graph-bootstrap` 的工程事实生成规范文档草案，经过人工确认后沉淀到 `docs/specs/**`，再通过机器索引和 LLM resolve 实现少读、准读、按需读。**

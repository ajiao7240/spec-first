# CRG-first 规范资产系统方案

> 文档状态：重写方案
> 修订日期：2026-04-27
> 核心裁决：新增独立 `$spec-standards` workflow skill，并配套 `spec-first specs` CLI 能力；`$spec-graph-bootstrap` 负责准备 CRG query-first 工程事实，`$spec-standards` 以 `spec-first crg workflow-context` 为稳定 evidence 入口，结合 `graph-quality.json`、`code-navigation.json`、`repo-topology.json` 和必要 query 生成规范文档草案；正式规范资产存放在 `docs/specs/**`，生成草案和运行证据存放在 `.spec-first/standards/runs/**`；MVP-A 必须生成规范草案与 evidence map，而不是只输出 repo-profile preview；MVP-B 才允许人工确认后 promote 到正式 `docs/specs/**` 并生成索引。

---

# 1. 一句话结论

`spec-first` 需要的是 **CRG-first 的规范资产生成、索引与按需加载系统**，不是只生成一个轻量 `repo-profile` 摘要。

正确闭环是：

```text
$spec-graph-bootstrap
  构建 CRG query-first 工程事实：graph、quality、navigation、topology、workspace candidates
        ↓
$spec-standards
  基于 workflow-context + targeted CRG queries + README/manifest/config + 人工输入生成规范文档草案
        ↓
preview-first proposal
  草案落到 .spec-first/standards/runs/<run-id>/，不自动生效
        ↓
human confirm / promote
  人确认后才写入 docs/specs/** 正式规范资产
        ↓
index / resolve
  生成机器索引，后续 plan / work / review 按需加载规范
        ↓
check / refresh
  作为审查辅助和刷新建议，不做默认 hard gate
```

关键判断：

```text
CRG = 工程事实层
docs/specs/** = 正式规范资产源
.spec-first/standards/runs/** = 本次生成草案和证据
_index/** = 机器索引，可重建
custom/** = 人工规范，最高优先级
LLM = 语义抽取、归纳、resolve 决策
脚本 / CLI = 目录、索引、schema、diff、写回、缓存等确定性执行
人 = 长期规范裁决
```

这保留原始目标：

```text
识别项目 → 抽取规范 → 手动接入 → 建立索引 → LLM 按需加载 → 检查 → 刷新
```

同时避免跑偏为：

```text
扫描代码 → 自动把现状升格为规范 → 自动写入 → 自动 fail gate
```

# 2. 设计基线

本方案必须遵循 `docs/10-prompt/项目角色.md` 的判断基线：

- Light contract
- Explicit boundaries
- Let the LLM decide
- Scripts prepare, LLM decides
- Deterministic execution belongs to scripts
- Semantic analysis belongs to LLM

因此：

```text
CRG 只能证明“代码库现在是什么”
CRG 不能证明“团队长期应该怎么做”
LLM 可以提出规范草案
LLM 不能静默写入正式规范
人确认后规范才成为长期输入
CLI/helper 负责确定性写回和索引
```

参考实践：

- Trellis 的 `.trellis/spec/**` 表明规范资产应是可读、可版本化、按领域组织的 Markdown 文档，而不是单个 YAML 摘要。
- spec-kit 的 constitution / specs / plan / tasks 结构表明长期原则应作为稳定输入源，被后续 workflow 读取，而不是混进每次任务产物。

# 3. 核心目标

本方案解决的问题：

```text
项目中有很多隐性规范：
- 代码里有事实
- 配置里有约束
- CI 里有质量信号
- 团队里有习惯
- 人工文档散落在不同位置

AI 开发时如果没有规范基线，就会：
- 凭经验生成代码
- 误判项目架构
- 忽略团队约定
- 全量读文档浪费 token
- 让 plan / work / review 首稿质量不稳定
```

`spec-standards` 的目标是：

1. 基于 CRG facts 识别项目端类型、语言、框架和模块边界。
2. 从代码、配置、测试、CI、README 中生成规范文档草案。
3. 支持用户手动放入规范文档，并保证 manual/custom 优先。
4. 生成机器可读索引，避免后续 workflow 全量读取 `docs/specs/**`。
5. 让 LLM 根据任务、文件、profile、关键词和 token budget 决策加载哪些规范。
6. 为 check / refresh 提供规范资产基础，但默认不做 hard gate。
7. 让规范跟随仓库版本化，方便团队共享。

# 4. 职责边界

## 4.1 `$spec-graph-bootstrap`

职责：

- 构建本地 CRG graph。
- 识别真实入口、模块、依赖、测试、配置、风险节点。
- 写入 graph quality、code navigation、repo topology 等 control-plane 产物。
- 在 parent workspace 场景提供 workspace candidates，但不自动选择 child repo。
- 提供 query-first evidence 与 lifecycle hooks。
- 完成后提示 `$spec-standards` 作为推荐下一步。

产物：

```text
.spec-first/
  graph/
    graph.db
    current.json
    graph-index-status.json
    graph-quality.json
    code-navigation.json
    repo-topology.json
    graph-operations.jsonl
    generations/
      <generation-id>/
        graph.db
        graph-quality.json

  workspace/
    workspace-index.json
    workspace-status.json
```

`graph.db` 是代码事实数据库；`graph-quality.json` 是 active generation 的质量投影；`workflow-context` 是 `$spec-standards` 推荐读取入口。`$spec-standards` 不应该优先直接解析内部 graph 产物，只有在 debug 或 CLI helper 需要确定性 schema 校验时才直接读取。

不做：

- 不生成规范文档。
- 不判断哪些现状应成为规范。
- 不写 `docs/specs/**`。
- 不自动调用 `$spec-standards`。
- 不在 parent workspace 上创建混合 graph 或自动选择 child repo。

完成提示：

```text
Graph ready. Next recommended step:
$spec-standards
```

## 4.2 `$spec-standards`

职责：

- detect：识别项目 profiles。
- extract：生成规范文档草案。
- intake：接入用户手动规范。
- index：生成规范索引。
- resolve：为任务决策加载哪些规范。
- check：基于规范进行审查辅助。
- refresh：根据变更生成规范刷新建议。
- sync：同步入口说明到 AGENTS.md / CLAUDE.md 等 host 文件，但不复制规范正文。

`$spec-standards` 是公开 workflow skill；确定性文件操作由 `spec-first specs` CLI/helper 执行。

不做：

- 不把所有重复代码模式直接写成正式规范。
- 不覆盖 manual/custom 规范。
- 不自动解决规范冲突。
- 不把 inferred 规范变成默认 fail gate。
- 不把规范正文膨胀进 AGENTS.md / CLAUDE.md。

## 4.3 `spec-first specs` CLI/helper

职责：

- 创建目录。
- 解析 Markdown frontmatter。
- 生成索引。
- 校验 schema。
- 输出 diff。
- 执行 promote。
- 生成 resolve result / JSONL context plan。
- 写 reports。

这些是确定性流程，应由 CLI/helper 做，不让 LLM 模拟。

## 4.4 人

职责：

- 判断 extracted / inferred 草案是否代表团队长期规范。
- 修改、确认或拒绝规范草案。
- 裁决冲突。
- 决定哪些规范进入 `docs/specs/**` 正式资产。

## 4.5 CRG Evidence Adapter v1

`$spec-standards` 不直接把 `.spec-first/graph/**` 当成稳定 API。MVP-A 的稳定 evidence 入口是：

```bash
spec-first crg workflow-context --stage=plan --repo=<target> --task="generate project standards"
```

Adapter v1 输入：

- target repo root。
- 可选 task hint，默认使用 `generate project standards`。
- 可选 changed files，用于 workspace context 或后续 resolve。
- 可选 parent workspace root。

Adapter v1 输出给 LLM 的 evidence envelope：

```text
graph_status
  state / capabilities / stats / active_db_path / limitations

graph_quality
  generation_id / parser_success_rate / parse_error_count
  skipped_sensitive_count / unresolved_edges.rate
  confidence_distribution / communities / flows / limitations

code_navigation
  entrypoints / test_roots / high_risk_nodes / top_communities / top_flows

repo_topology
  kind / units / detectors / limitations

decision_inputs
  observed / ambiguous evidence kinds

recommended_queries
  locate / explain / path / impact / review-context suggestions

fallback
  direct_repo_reads suggested reads and limitations
```

Adapter v1 可按需追加 targeted CRG queries：

```bash
spec-first crg locate --repo=<target> --query="<area>" --limit=5
spec-first crg explain --repo=<target> --node="<node-or-file-id>"
spec-first crg path --repo=<target> --from=<a> --to=<b>
spec-first crg architecture --repo=<target>
spec-first crg review-context --repo=<target> --since=<base>
```

错误语义：

- `graph_status.state=ready`：可以生成 CRG-first proposal。
- `graph_status.state=degraded`：可以生成 proposal，但必须把 graph limitations 写入 preview 和 evidence-map。
- `graph_status.state=missing|unavailable`：进入 direct-only 降级，不能把代码推断直接 promote。
- `graph_quality.state=missing|degraded`：不阻断 proposal，但降低 extracted/inferred 草案置信度。
- query error：记录到 run limitations，继续使用 workflow-context 与 targeted direct reads。

## 4.6 Graph Quality Advisory Gating

`graph-quality.json` 不决定“什么是规范”，但决定“当前 CRG evidence 能支撑多高置信度的规范草案”。

建议规则：

```text
parser_success_rate 高、parse_error_count 低、skipped_sensitive_count 低
  → 可以生成较高置信 inferred draft。

unresolved_edges.rate 高、flows.status unknown、communities.status unknown
  → 架构/依赖类规范必须标注 limitation，不能建议直接 confirmed。

skipped_sensitive_count > 0
  → evidence-map 只能引用路径、symbol、line range 和结构事实，不写原始敏感值。

graph_quality missing
  → 继续生成草案，但 preview 必须说明缺少质量投影。
```

Graph quality 是 advisory input，不是 hard gate。脚本只报告 coverage、limitations 和 generation；LLM 负责判断哪些候选规则应进入 draft / uncertain / rejected。

为减少实现分歧，helper 可以生成 `quality_band`，但 band 仍然只是 advisory：

```text
strong
  parser_success_rate >= 0.9，且 parse_error_count / skipped_sensitive_count / unresolved_edges limitation 不影响目标范围。

usable
  parser_success_rate >= 0.7，存在 limitations，但仍足以支持路径、入口、模块类事实。

weak
  parser_success_rate < 0.7，或 unresolved_edges / parse errors 明显影响目标范围。

unavailable
  graph_quality missing 或 graph unavailable。
```

`quality_band` 由 helper 根据数字和 limitation 粗分；LLM 结合任务语义决定具体草案进入 `draft`、`uncertain` 还是 `rejected`。不要把 band 做成 fail gate。

## 4.7 Workspace Target Handling

如果 target 是 parent workspace，`$spec-standards` 必须先复用 CRG workspace preflight：

```bash
spec-first crg workspace scan --root=<workspace>
spec-first crg workspace status --root=<workspace>
spec-first crg workspace context --root=<workspace> --task="generate project standards"
```

规则：

- workspace context 的 candidates 只是 advisory input。
- LLM/user 必须选择一个 child repo 后，才运行 repo-local `$spec-standards`。
- 默认不在 parent workspace 写 `docs/specs/**`。
- 默认不在 parent workspace 创建 `.spec-first/graph/graph.db`。
- 跨多个 child repos 的规范生成必须拆成显式顺序 repo-local runs。
- 只有用户明确声明 parent workspace 本身是规范仓库，或后续 Phase 3 attach shared standards repo，才允许 parent 级 standards 资产。

parent workspace 被视为 standards repo 必须满足至少一个确定性条件：

```text
1. 命令显式传入 --target-kind=standards-repo。
2. parent 下已存在 docs/specs/SPEC.md，且用户确认继续。
3. parent 下已存在 .spec-first/standards-root.json。
```

否则 `$spec-standards` 只能展示 workspace candidates，并要求选择 child repo；不能依靠 LLM 从自然语言中猜测 parent 是否应该承载正式规范。

## 5.2 目录职责

| 路径 | 职责 | 写入者 | 是否正式生效 |
| --- | --- | --- | --- |
| `.spec-first/graph/**` | CRG 工程事实 | `$spec-graph-bootstrap` / CRG CLI | 否，只是 evidence |
| `.spec-first/workspace/**` | parent workspace 候选 repo、readiness、recommended commands | CRG workspace CLI | 否，只是 advisory evidence |
| `.spec-first/standards/runs/**` | 本次规范生成草案、preview、证据 | `$spec-standards` + helper | 否，proposal-only |
| `.spec-first/standards/resolve-cache/**` | resolve 缓存 | `spec-first specs resolve` | 否，可重建 |
| `docs/specs/**/*.md` | 正式规范资产 | 人工 / promote helper | 是 |
| `docs/specs/custom/**` | 人工最高优先级规范 | 人工 | 是 |
| `docs/specs/_index/**` | 机器索引 | index helper | 是，但可重建 |
| `docs/specs/evidence/**` | 证据、冲突、不确定项 | helper / 人工整理 | 否，辅助判断 |
| `docs/specs/reports/**` | check / refresh 报告 | helper | 否，辅助判断 |

## 5.3 MVP-A Evidence Map

MVP-A 的 `evidence-map.json` 必须记录 CRG generation 与 graph quality 摘要，避免草案和事实层脱钩。

最小结构：

```json
{
  "version": 1,
  "run_id": "2026-04-27-001",
  "target_repo": "/path/to/repo",
  "evidence_mode": "crg-first",
  "crg": {
    "generation_id": "20260426175341061",
    "graph_state": "ready",
    "active_db_path": ".spec-first/graph/generations/20260426175341061/graph.db",
    "graph_quality_state": "available",
    "parser_success_rate": 0.9791,
    "parse_error_count": 4,
    "skipped_sensitive_count": 3,
    "unresolved_edges_rate": 0.5837,
    "limitations": []
  },
  "drafts": [
    {
      "path": ".spec-first/standards/runs/2026-04-27-001/drafts/common/architecture.md",
      "evidence_kind": "observed",
      "source_queries": [
        "spec-first crg workflow-context --stage=plan --repo=<target>"
      ],
      "limitations": []
    }
  ]
}
```

证据写入规则：

- 优先记录 file path、symbol、line range、query command、generation id 和结构事实。
- 不写 raw secret、token、credential、环境变量值或敏感配置值。
- 如果 CRG 报告 `skipped_sensitive_count > 0`，preview 必须明确说明 evidence 已做 redaction。
- 如果 direct-only 降级，`evidence_mode=direct-only` 且每条草案必须标记 low confidence。

## 5.4 Proposal Run 共享策略

`.spec-first/standards/runs/**` 默认是本地 proposal artifact，不是团队共享规范源。

规则：

- 默认应加入 `.gitignore` 或由 helper 提示忽略。
- `docs/specs/**` 才是正式可版本化共享规范资产。
- 如果团队希望共享 proposal run，必须显式执行 export。
- export 前必须重新执行 redaction / validate。
- export 产物仍然不能直接作为正式规范参与 resolve / check，除非被 promote。

## 5.5 确定性写入安全边界

`spec-first specs` helper 的核心价值不是替 LLM 做语义判断，而是把文件写入、schema 校验、diff 和路径约束做成确定性边界。

最低要求：

- 所有 run 写入必须先写临时目录，再 atomic rename 到 `.spec-first/standards/runs/<run-id>/`。
- run 完成前写入 `run-state.json`，区分 `running | completed | failed`，避免半成品被误 promote。
- `validate-run` 必须校验 `preview.md`、`detected-profiles.json`、`evidence-map.json`、`drafts/**`、`rejected/**` 的存在性和 schema。
- helper 只能写 repo-relative allowlist 路径，MVP-A 只允许 `.spec-first/standards/runs/**`。
- promote helper 只能写 `docs/specs/**`，不能写 host instruction、源码目录或任意绝对路径。
- 禁止跟随会把写入目标带出 repo 的 symlink。
- 所有 overwrite 必须先生成 diff；manual/custom 被覆盖时直接拒绝，除非用户显式选择创建新文件或人工合并。
- `evidence-map.json` 和 `preview.md` 必须经过 redaction 检查后才允许 export。

## 5.6 Proposal Payload Contract

MVP-A 中，LLM 不直接写 `.spec-first/standards/runs/**`。`$spec-standards` 先生成一个机器可校验的 proposal payload，再交给 deterministic helper 写盘。

命令形态：

```bash
spec-first specs write-proposal \
  --run-id <run-id> \
  --target <repo> \
  --payload <proposal-payload.json>
```

最小 payload：

```json
{
  "schema_version": "standards-proposal-payload/v1",
  "run_id": "2026-04-27-001",
  "target_repo": "/path/to/repo",
  "evidence_mode": "crg-first",
  "graph": {
    "generation_id": "20260426175341061",
    "quality_band": "usable",
    "limitations": []
  },
  "preview_markdown": "# Standards Proposal Preview\n",
  "detected_profiles": [
    {
      "id": "node-commonjs-cli",
      "scope": ["backend", "cli"],
      "confidence": "high",
      "evidence": ["package.json", "bin/spec-first.js"],
      "limitations": []
    }
  ],
  "evidence_map": {
    "version": 1,
    "run_id": "2026-04-27-001",
    "evidence_mode": "crg-first",
    "drafts": []
  },
  "drafts": [
    {
      "path": "drafts/common/architecture.md",
      "content": "---\nspec_id: common-architecture\nsource: extracted\nconfirmation_status: inferred\nlifecycle_status: active\n---\n# Architecture\n",
      "source": "extracted",
      "confirmation_status": "inferred",
      "lifecycle_status": "active",
      "evidence_refs": []
    }
  ],
  "rejected": {
    "inferred_rules_markdown": "No entries in this run.\n",
    "uncertain_rules_markdown": "No entries in this run.\n",
    "conflicts_markdown": "No entries in this run.\n"
  },
  "limitations": []
}
```

helper 写入映射：

| payload 字段 | 写入位置 |
| --- | --- |
| `preview_markdown` | `preview.md` |
| `detected_profiles` | `detected-profiles.json` |
| `evidence_map` | `evidence-map.json` |
| `drafts[].path/content` | 对应 `drafts/**/*.md` |
| `rejected.inferred_rules_markdown` | `rejected/inferred-rules.md` |
| `rejected.uncertain_rules_markdown` | `rejected/uncertain-rules.md` |
| `rejected.conflicts_markdown` | `rejected/conflicts.md` |

helper 必须执行的确定性校验：

- `run_id` 必须匹配命令参数，且只能包含安全字符。
- `target_repo` 必须等于 `--target` 解析后的 repo root。
- `evidence_mode` 只能是 `crg-first | direct-only | mixed`。
- `drafts[].path` 必须是 repo-relative run-local path，且只能位于 `drafts/` 下。
- `drafts[].content` 必须是 Markdown，frontmatter 中的 `source`、`confirmation_status`、`lifecycle_status` 必须与 payload 元数据一致。
- `rejected.*` 缺失时 helper 必须补写 `No entries in this run.`。
- payload 不允许携带 raw secret、token、credential 或环境变量值。
- payload schema validate 通过后，helper 才能写临时目录并 atomic rename。

职责边界：

```text
LLM / $spec-standards
```

# 7. 规范来源与优先级

规范来源：

```text
manual      用户手动编写或放入
imported    外部规范导入
extracted   LLM 基于 CRG evidence 生成的草案
template    系统模板初始化
generated   索引、报告、证据等系统产物
```

确认状态使用独立 `confirmation_status` 字段，不混进 `source`：

```text
manual       用户明确声明或手写
confirmed    用户确认 extracted 草案代表长期规范
inferred     LLM 从 CRG / repo facts 推断，未被确认
uncertain    evidence 不足或 graph quality 限制较强
conflict     与现有规范或证据冲突
```

生命周期状态使用独立 `lifecycle_status` 字段，不混进确认状态：

```text
active       已生效规范文件的普通状态
deprecated   已废弃但保留历史记录
overridden   被更高优先级规范覆盖
```

优先级：

```text
custom/manual
  > manual
  > imported
  > source=extracted,confirmation_status=confirmed
  > source=extracted,confirmation_status=inferred
  > template
```

规则：

- `docs/specs/custom/**` 最高优先级。
- `source=manual` 不允许被 refresh 自动覆盖。
- `source=extracted` 默认先进入 `.spec-first/standards/runs/**` 草案区。
- `source=extracted,confirmation_status=confirmed,lifecycle_status=active` 才允许进入正式规范并参与 check 的高置信度判断。
- `confirmation_status=uncertain|conflict` 只能进入 evidence 或 reports，不进入正式规则。

# 10. Profile Registry

Profile Registry 是 Phase 2+ 的扩展能力，不是 MVP-A 的前置条件。MVP-A 只需要一个轻量 detected profile contract：

```json
{
  "detected_profiles": [
    {
      "id": "node-commonjs-cli",
      "scope": ["backend", "cli"],
      "confidence": "high",
      "evidence": [
        "package.json",
        "bin/spec-first.js",
        "src/cli/**"
      ],
      "crg_generation_id": "20260426175341061",
      "limitations": []
    }
  ]
}
```

MVP-A 不引入完整 registry，不用 path rules 代替 LLM 判断，也不把 profile 当成规则引擎。

Phase 2+ 的 Profile 才定义：

```text
1. 如何识别项目
2. 需要扫描哪些文件
3. 应生成哪些规范草案
4. 路径对应哪些规范
5. resolve/check 时如何匹配规则
```

示例：

```yaml
id: backend-java-spring
name: Java Spring Backend
type: backend

detect:
  files:
    - pom.xml
    - build.gradle
  contains:
    - "@SpringBootApplication"
    - "@RestController"
    - "spring-boot-starter-web"

scan:
  required:
    - pom.xml
    - src/main/java/**
    - src/main/resources/application*.yml
  optional:
    - Dockerfile
    - .github/workflows/**
    - src/test/java/**

specs:
  required:
    - backend/java.md
    - backend/java-spring.md
    - backend/api.md
    - backend/error-handling.md
    - backend/database.md
    - backend/testing.md
  optional:
    - backend/auth.md
    - common/security.md
    - common/ci-cd.md

resolve:
  path_rules:
    - pattern: "**/controller/**"
      load:
        - backend/api.md
        - backend/error-handling.md
        - backend/java-spring.md
    - pattern: "**/service/**"
      load:
        - backend/java-spring.md
        - backend/error-handling.md
```

Profile detection 的事实来源优先使用 CRG Evidence Adapter v1，再补 README / manifest / config direct reads。CRG 只能证明项目事实；profile 只帮助确定应生成哪些草案类型，不能证明这些草案是团队长期规范。

# 13. Graph 缺失与降级

`$spec-standards` 可以直接运行，但必须先通过 CRG Evidence Adapter v1 检测 target 类型、graph readiness 和 graph quality。

```text
target is parent workspace
  先运行 workspace scan/status/context。
  LLM/user 选择 child repo 后进入 repo-local standards run。
  默认不在 parent workspace 写 docs/specs/**。

graph ready + graph_quality available
  使用 workflow-context + targeted CRG queries 生成 detect/extract 草案。

graph degraded
  使用可用 workflow-context + direct repo reads，并在 preview / evidence-map 中标注 limitations。

graph ready + graph_quality missing/degraded
  可以生成草案，但降低置信度；架构/依赖类规范必须标注 quality limitation。

graph missing / unavailable
  interactive 模式让用户选择：
    1. 先运行 $spec-graph-bootstrap
    2. 继续 direct-only 低置信度草案
    3. CRG-only stop
  headless 模式默认 direct-only proposal，但必须标记 low confidence。
```

direct-only 规则：

- 可以生成 proposal 草案。
- 不允许把基于代码推断的规范直接 promote。
- preview / evidence 必须说明缺少 CRG evidence。
- direct-only 草案必须标记 `evidence_mode=direct-only` 和 low confidence。
- MVP-A 下，用户显式声明的 manual 规范只能进入 proposal 或提示用户手动写入；`$spec-standards` 不自动写 `docs/specs/**`。
- MVP-B+ 下，用户显式声明的 manual 规范可以通过 deterministic helper 写入 `docs/specs/custom/**`。

CRG 降级不是失败态，而是 evidence 质量提示。脚本只报告 ready/degraded/missing/unavailable、quality limitations 和 suggested direct reads；LLM 决定是否继续生成草案，人决定是否 promote。

# 14. Check 与 Refresh

## 14.1 Check

`spec-first specs check` 是审查辅助，不是默认 hard gate。

流程：

```text
1. 获取 changed files
2. 调用 specs resolve
3. 加载 check.jsonl 中的规范
4. 执行确定性检查，可选
5. 执行 LLM review
6. 运行项目命令：lint / test / typecheck，可选
7. 输出 docs/specs/reports/spec-check-report.md
```

默认语义：

```text
manual critical/high → 可以作为阻塞建议，但仍由 workflow/user 裁决
source=extracted,confirmation_status=confirmed → warning 或 blocking suggestion
source=extracted,confirmation_status=inferred → warning
confirmation_status=uncertain|conflict → 只提示人工确认
```

trust 规则：

- 使用 `_index/**` 前必须校验它与当前 `docs/specs/**` 的 freshness。
- 默认忽略 `.spec-first/standards/runs/**` 草案，除非用户显式指定 draft run。
- remote/imported 规范在本地人工确认前只能作为 advisory input。
- 报告必须标注使用的 specs index generation、CRG generation 和可能的 stale / tampered / missing 状态。

## 14.2 Refresh

`spec-first specs refresh` 不覆盖人工规范。

模式：

```bash
spec-first specs refresh --changed
spec-first specs refresh --full
spec-first specs refresh --profile backend-java-spring
spec-first specs refresh --index-only
```

原则：

- 不覆盖 `docs/specs/custom/**`。
- 不覆盖 `source=manual`。
- `confirmation_status=confirmed` 规则谨慎修改，需要 preview。
- `confirmation_status=inferred` 可根据新证据生成更新草案。
- `confirmation_status=conflict` 需要人工确认。
- 每次刷新必须生成报告。
- 刷新后必须重新生成 index。

`docs/specs/**` 是唯一规范源。Host instruction 文件只是入口说明和加载策略，不承载规范正文。

- `spec-first specs sync` 只能更新带明确 start/end marker 的 spec-first managed block。

# 17. MVP 分阶段

## 17.1 MVP-A：CRG-first detect + extract proposals

目标：证明 CRG-first 能生成高质量规范文档草案，并证明 `graph-quality.json` 能帮助 LLM 正确降级、标注不确定性和避免把低质量 evidence 升格为规范。

必做：

1. 新增 `$spec-standards` workflow skill。
2. `$spec-standards` 启动时通过 CRG Evidence Adapter v1 读取 `workflow-context`。
3. graph ready 时使用 `graph_status`、`graph_quality`、`code_navigation`、`repo_topology`、`decision_inputs` 和 targeted CRG queries 生成草案。
4. parent workspace target 先运行 workspace preflight，并要求明确 child repo 后再执行 repo-local run。
5. graph missing 时提示 `$spec-graph-bootstrap`，但允许 direct-only low-confidence proposal。
6. 生成 `.spec-first/standards/runs/<run-id>/preview.md`。
7. 生成 `.spec-first/standards/runs/<run-id>/detected-profiles.json`。
8. 生成 `.spec-first/standards/runs/<run-id>/evidence-map.json`，记录 CRG generation、graph quality 摘要、source queries、limitations 和 redaction 状态。
9. 生成 `.spec-first/standards/runs/<run-id>/drafts/**/*.md`。
10. 固定生成 `.spec-first/standards/runs/<run-id>/rejected/inferred-rules.md`、`uncertain-rules.md`、`conflicts.md`；没有内容时写入 `No entries in this run.`。
11. 不写 `docs/specs/**` 正式规范。
12. `$spec-graph-bootstrap` 完成后只提示 `$spec-standards`，不自动调用。

MVP-A 可选但不作为验收必需：

- `.spec-first/standards/runs/<run-id>/extract-plan.json`
- `.spec-first/standards/runs/<run-id>/index-draft/specs-index.json`

这些产物只有在真实验证证明它们能提升草案评审效率时再纳入 MVP-A，否则后移到 MVP-B。

MVP-A 不做：

- 不 promote。
- 不 check。
- 不 refresh。
- 不 sync host instruction。
- 不实现 full profile registry。
- 不生成 `rules-map.json`。
- 不支持远程共享规范仓库。
- 不在 parent workspace 自动写正式规范。

MVP-A runtime 注册清单：

1. 新增 `skills/spec-standards/SKILL.md`。
2. 新增 Claude command 与 Codex skill runtime 安装来源。
3. 更新 plugin manifest / dual-host governance 中公开 workflow 清单。
4. 补 source-of-truth contract 测试，确认 Claude `/spec:standards` 与 Codex `$spec-standards` 能被安装。
5. 不手改 `.claude/`、`.codex/` 或 `.agents/skills/` 生成资产。

## 17.2 MVP-B：promote 到正式规范资产

前置：MVP-A 草案质量和端到端 workflow 价值通过真实仓库验证。

必做：

1. `spec-first specs init` 创建 `docs/specs/**` 基础结构。
2. `spec-first specs promote <run-id>` 支持按 spec / rule 逐项确认、拒绝、编辑、延后；run-level promote 只能作为 all-confirmed 快捷路径。
3. `spec-first specs index` 生成 `_index/specs-index.json`、`rules-map.json`、`profiles.json`。
4. manual/custom 文档优先，promote 不覆盖 manual。
5. 无 frontmatter manual 文档可被索引，并提示 normalize。
6. 所有写回由确定性 helper 完成 parse / validate / allowlist / merge / diff / write。

## 17.3 MVP-C：resolve 接入 workflow

前置：MVP-B 已生成 `docs/specs/**` 和 `docs/specs/_index/**`。如果要支持 draft-run resolve，必须作为单独路径设计，不能把 proposal 草案伪装成正式规范。

必做：

1. `spec-first specs resolve --task --files` 输出 resolve plan。
2. 为 `$spec-plan`、`$spec-work`、`$spec-code-review` 提供 optional context。
3. 支持 load_full / load_summary / load_reference。
4. 输出 `.spec-first/tasks/<task-id>/resolve-result.json` 和 JSONL context plan。
5. 不把 resolve 结果变成 hard gate。

## 17.4 后续阶段

Phase 2：

- `spec-first specs check --changed`
- `spec-first specs refresh --index-only`
- `spec-first specs refresh --changed`
- `spec-first specs validate`
- `spec-first specs normalize`

Phase 3：

- shared standards repo attach
- pinned ref adoption
- remote Git resolver / clone / fetch
- PR history learning
- embedding 检索

Shared standards repo trust model：

- 默认只接受 pinned commit ref，不接受 floating branch 作为生效规范。
- remote 必须 allowlist，不能从任意 URL 自动拉取规范。
- 更新 remote standards 必须展示 diff 并由用户确认。
- remote/imported 规范不得自动高于 local `docs/specs/custom/**` 或 `source=manual`。
- private repo credential 不写入 `docs/specs/**`、reports、evidence-map 或 host instruction。
- remote standards 在本地 promote/confirm 前只能作为 advisory input。
- attach 记录写入 target repo 的本地采用记录，不把 shared repo 当第二真相源。

# 18. 质量验证

MVP-A 必须用真实仓库验证：

```text
CRG-first proposal vs direct-only proposal
```

指标：

- factuality：是否准确引用真实入口、模块、测试、配置。
- graph quality handling：是否正确消费 parser coverage、unresolved edge rate、skipped sensitive、limitations。
- evidence coverage：每条规范是否有事实证据，并能区分事实证据与规范意图证据。
- draft usefulness：草案是否接近可人工确认。
- uncertainty handling：是否正确区分 inferred / uncertain / conflict。
- token efficiency：是否比全量读代码更省。
- human adoption rate：最终被 promote 的比例。
- downstream usefulness：在 2-3 个真实任务中，是否提升 `$spec-plan`、`$spec-work`、`$spec-code-review` 首稿质量。
- maintenance cost：人工确认每条规范的时间是否低于收益。

进入 MVP-B 的最低门槛：

```text
1. 至少在 2 个真实仓库对比 CRG-first proposal 与 direct-only proposal。
2. CRG-first factuality 明显更高，且不能增加严重 hallucination。
3. 每条高置信草案都能追溯到 CRG generation id 和 evidence source。
4. graph quality limitation 能正确降低草案置信度，而不是被忽略。
5. 至少 2-3 个真实任务证明 standards context 能改善 plan/work/review 输出质量或减少人工修改。
6. 人工确认成本可接受；如果草案大量被拒绝，停止进入 MVP-B，回到 adapter / prompt / evidence 设计。
```

downstream usefulness 的最小评估方式：

```text
对同一真实任务执行两组对比：
1. without standards context
2. with standards context

比较：
- factual errors
- missed constraints
- unnecessary files / wrong change surface
- review finding quality
- token cost
- 人工修改量
- 完成时间
```

评估可以先由人工或 LLM-as-judge 完成，但必须保留对比记录。不要把“草案看起来更完整”当作进入 MVP-B 的充分理由。

只有这些条件成立，才进入 MVP-B。

# 19. 最终架构判断

这条路线值得做，但必须坚持两条边界：

```text
1. spec-standards 是规范资产系统，不是 repo-profile 摘要生成器。
2. spec-standards 不是强规则引擎，不替代 LLM 和人的判断。
```

应该保留：

- CRG-first。
- preview-first proposal。
- human-confirmed promote。
- `docs/specs/**` 作为正式规范资产源。
- `_index/**` 作为机器索引。
- manual/custom 最高优先级。
- LLM 决策加载，避免全量读取。
- check / refresh 作为后续辅助能力。

必须避免：

- 把代码现状直接升格为规范。
- 自动覆盖人工规范。
- 自动解决冲突。
- 一开始就建设复杂规则引擎。
- 把规范正文同步进 host instruction。
- 让 `repo-profile` 替代规范文档。

最终定版：

> **`spec-standards` 是 spec-first 的项目工程规范资产层：它使用 `spec-graph-bootstrap` 的工程事实生成规范文档草案，经过人工确认后沉淀到 `docs/specs/**`，再通过机器索引和 LLM resolve 实现少读、准读、按需读。**

# 20. 健壮性问题修复清单

本节把当前方案审查中暴露的问题逐个落成开发约束，避免实现时重新跑偏。

## 20.1 状态字段混用

问题：

```text
source、确认状态、生命周期状态容易混在一个 status 字段里，导致 manual、confirmed、active 语义互相污染。
```

修复：

- `source` 只描述来源：`manual | imported | extracted | template | generated`。
- `confirmation_status` 只描述确认程度：`manual | confirmed | inferred | uncertain | conflict`。
- `lifecycle_status` 只描述生命周期：`active | deprecated | overridden`。
- check / refresh / index 全部按这三个字段组合判断，不能再使用 `source=confirmed` 或 `status=confirmed`。

## 20.2 Graph quality 被误做 hard gate

问题：

```text
graph-quality 如果直接变成 fail gate，会把 CRG 从 evidence provider 变成强规则控制器。
```

修复：

- `quality_band=strong|usable|weak|unavailable` 只作为 advisory。
- helper 可以根据数字生成 band，但不裁决规范是否成立。
- LLM 结合任务语义决定 draft / uncertain / rejected。
- 人决定是否 promote。

## 20.3 direct-only 降级写回过宽

问题：

```text
没有 CRG 时如果仍能自动写 docs/specs/**，容易把低证据推断沉淀为团队规范。
```

修复：

- direct-only 可以生成 proposal，但必须 low confidence。
- MVP-A 禁止任何自动写 `docs/specs/**`。
- direct-only 的代码推断不能直接 promote。
- 用户显式声明的 manual 规范在 MVP-A 只进入 proposal 或提示用户手动写入；MVP-B+ 才能通过 deterministic helper 写入 `docs/specs/custom/**`。

## 20.4 parent workspace 误写正式规范

问题：

```text
在 monorepo parent 或多 repo workspace 下运行时，LLM 可能把 parent 当成 target repo，生成混合规范。
```

修复：

- parent workspace 必须先跑 workspace scan/status/context。
- 必须选择 child repo 后才进入 repo-local run。
- parent 只有满足 `--target-kind=standards-repo`、已有 `docs/specs/SPEC.md` 并确认、或已有 `.spec-first/standards-root.json` 时，才可作为 standards repo。
- 多 child repo 规范生成必须拆成多个显式 run。

## 20.5 proposal run 被误当共享规范

问题：

```text
.spec-first/standards/runs/** 是草案和证据，如果被团队直接共享使用，会绕过人工确认和 redaction。
```

修复：

- proposal run 默认本地、默认忽略。
- 只有 `docs/specs/**` 是正式共享规范资产。
- proposal export 必须显式执行，并重新 redaction / validate。
- export 后仍不能参与 resolve/check，除非 promote。

## 20.6 MVP-A CLI 暴露过多

问题：

```text
文档同时列出 init/detect/extract/index/resolve/check/refresh/sync，容易让 MVP-A 被实现成完整平台。
```

修复：

- MVP-A 用户入口只承诺 `$spec-standards`。
- MVP-A CLI 只实现内部 helper：`write-proposal` 和 `validate-run`。
- 其他命令保留为长期目标，按 MVP-B/C/Phase 2 分阶段进入。

## 20.7 rejected / uncertain / conflict 产物不稳定

问题：

```text
如果 rejected 目录按需生成，下游 review 和测试很难判断是“没有问题”还是“漏生成”。
```

修复：

- MVP-A 固定生成 `rejected/inferred-rules.md`、`uncertain-rules.md`、`conflicts.md`。
- 没有内容时写入 `No entries in this run.`。

## 20.8 shared standards repo 信任边界不足

问题：

```text
外挂 Git 规范仓库如果没有 pinned ref、allowlist 和本地采用记录，会形成第二真相源或供应链风险。
```

修复：

- Phase 3 才引入 shared standards repo attach。
- 默认只接受 pinned commit ref。
- remote 必须 allowlist。
- 更新必须展示 diff 并人工确认。
- remote/imported 在本地 promote/confirm 前只能 advisory。
- local `docs/specs/custom/**` 和 `source=manual` 永远优先。

## 20.9 helper 写入路径不受控

问题：

```text
如果 LLM 或 helper 可写任意路径，promote/export/sync 都可能越界改源码、host instruction 或外部目录。
```

修复：

- MVP-A helper 只能写 `.spec-first/standards/runs/**`。
- promote helper 只能写 `docs/specs/**`。
- sync helper 只能更新 host instruction 中带 marker 的 managed block。
- 禁止 absolute path 写入和逃逸 repo 的 symlink。
- 写入前必须 validate，覆盖前必须 diff。

## 20.10 下游价值验证不具体

问题：

```text
只看草案是否完整，无法证明 spec-standards 真正提升 plan/work/review 质量。
```

修复：

- MVP-B 前必须做 CRG-first proposal vs direct-only proposal 对比。
- 至少用 2-3 个真实任务比较 with/without standards context。
- 指标覆盖 factual errors、missed constraints、wrong change surface、review finding quality、token cost、人工修改量和完成时间。
- 草案“看起来完整”不能作为进入 MVP-B 的充分理由。

## 20.11 LLM 与 helper 交接不明确

问题：

```text
如果没有 proposal payload contract，LLM 可能直接写 run 文件，或 helper 被迫接受隐式自由文本输入。
```

修复：

- `$spec-standards` 只生成 `standards-proposal-payload/v1`。
- `write-proposal` 只接受 `--payload <proposal-payload.json>`。
- helper 负责 schema、path allowlist、frontmatter consistency、redaction、atomic write。
- `validate-run` 负责校验落盘后的 proposal run 是否完整可审查。

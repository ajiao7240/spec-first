做这个事情的核心目的，其实一句话就够：

> **让 AI 在存量项目里，不再靠临时猜测写代码，而是基于团队规范、项目事实、图谱证据和已有能力复用来工作。**

它不是为了多生成几个规范文件。
它是为了给 spec-first 补上一个关键节点：

```text
存量项目接入 spec-first 时，
先把“隐性规范 + 团队标准 + 代码事实 + 复用能力”
编译成可复用的项目基线。
```

---

# 1. 为什么要做？

因为真实企业项目里，最大的问题不是“没有代码”，而是：

```text
代码很多，但规范不显性
经验很多，但沉淀在人脑里
文档有一些，但可能过期
AI 能写代码，但不知道项目边界
新人能改代码，但不知道哪些模式应该复用
review 能发现问题，但经验很难反哺
```

最终结果就是：

```text
AI 反复读项目
反复问上下文
反复重复造轮子
反复违反项目约定
反复在 review 阶段被纠正
```

所以需要 `/spec:standards`。

它要解决的是：

> **把项目里的隐性工程知识，变成 AI 和团队都能稳定消费的显性标准。**

---

# 2. 这个 skill 的目标是什么？

`/spec:standards` 的目标不是“写规范文档”。

真正目标有 6 个。

---

## 2.1 识别项目是什么

先搞清楚当前项目模式：

```text
单仓单项目？
单仓多模块？
父目录下多个独立 git 仓库？
```

再识别：

```text
有没有前端？
有没有后端？
有没有 API？
有没有数据库？
有没有移动端？
有没有 admin？
是不是 CLI？
是不是 AI workflow 项目？
有哪些 graph 能力？
```

目的：

> **避免一上来盲目生成 frontend/backend/database/mobile 规范。**

---

## 2.2 导入团队已有规范

如果团队已经有编码规范，不让 AI 重新发明一套。

而是：

```text
从独立 standards Git 仓库导入
按项目技术栈和场景自动匹配
锁定版本
解析成 imported-standards
和当前项目代码做对齐
再给团队确认
```

目的：

> **让团队已有规范成为 AI coding 的高优先级输入源。**

---

## 2.3 从代码和 graph 中发现项目事实

不是靠 AI 猜。

而是基于：

```text
graph-facts.json
architecture-facts.json
reuse-candidates.json
semantic query
代码证据
模块依赖
调用关系
已有实现
```

目的：

> **让规范候选有证据，而不是拍脑袋。**

---

## 2.4 生成胶水能力地图

很多需求不是从零写系统，而是把已有能力串起来。

所以要识别：

```text
已有能力在哪里？
哪些模块应该复用？
哪些地方只需要写 adapter？
哪些能力不能重复实现？
跨 skill / module / repo 怎么交接？
```

目的：

> **让 AI 少写新代码，多复用已有能力，用最小胶水完成需求。**

---

## 2.5 形成项目级标准基线

通过：

```text
standards-candidates.json
standards-preview.md
repo-profile.patch.yaml
repo-profile.yaml
```

把候选规范变成团队确认后的项目基线。

目的：

> **让后续 brainstorm / plan / work / review 都有统一标准可读。**

---

## 2.6 形成持续更新闭环

规范不是一次性文档。

它应该能被这些信号更新：

```text
项目结构变化
graph 产物变化
团队规范升级
review 多次发现同类问题
compound-refresh 发现长期经验
新 domain 出现
```

目的：

> **让规范随着项目演进，而不是写完就过期。**

---

# 3. 它最终服务谁？

## 3.1 服务 AI

让 AI 不再每次临时猜：

```text
这个项目怎么分层？
接口怎么返回？
组件放哪里？
数据库怎么改？
已有导出能力在哪里？
这个需求该改几个仓库？
```

AI 可以直接读取：

```text
repo-profile.yaml
project-shape.json
imported-standards.json
standards-candidates.json
glue-map.json
graph-query-index.json
```

---

## 3.2 服务开发者

让开发者不用反复口头告诉 AI：

```text
我们项目不是这样写的
这个接口要走统一封装
这个能力已经有了
不要直接访问数据库
不要绕过 SDK
记得更新 changelog
```

这些会沉淀成标准和 review 依据。

---

## 3.3 服务团队

让团队规范可以：

```text
集中维护
版本化
按项目导入
按项目确认
多人共享
持续演进
```

而不是每个项目各写一份、各自漂移。

---

## 3.4 服务 spec-first 自身

它让 spec-first 从：

```text
AI coding workflow 工具
```

升级成：

```text
项目规范 + 代码图谱 + 胶水能力 + 工程闭环的工作流系统
```

这就是核心竞争力。

---

# 4. 最终业务价值

做这个事情的最终价值是：

```text
降低 AI coding 的上下文成本
降低重复造轮子的概率
降低 review 返工
提高需求方案贴合项目的程度
提高团队规范复用
提高新人和 AI 的项目理解速度
提高多端 / 多模块 / 多仓协作质量
把一次次 review 经验沉淀成长期资产
```

更直白一点：

> **它让 AI 从“会写代码”变成“会按这个团队、这个项目、这个仓库的方式写代码”。**

---

# 5. 最终目标状态

理想状态下，一个存量项目接入 spec-first 后：

```text
/spec:mcp-setup
  ↓
/spec:graph-bootstrap
  ↓
/spec:standards
```

生成项目基线：

```text
项目是什么
有哪些模块
有哪些团队规范
哪些规范适用
当前代码是否符合
有哪些已有能力
哪些能力应该复用
哪些规范需要确认
哪些冲突需要处理
```

然后后续每个需求：

```text
brainstorm 不再空想需求
plan 不再泛泛设计
write-tasks 不再脱离项目结构
work 不再重复造轮子
code-review 不再泛泛建议
compound-refresh 不再只写总结，而是能反哺规范
```

---

# 6. 一句话总结

做 `/spec:standards` 的目的不是“生成规范文档”，而是：

> **把团队规范、代码事实、图谱证据、已有能力和胶水边界编译成项目级共享基线，让 AI 和团队在同一套工程标准下完成需求、计划、开发、审查和知识沉淀。**

它的目标是：

> **让 spec-first 支持真实存量项目、真实团队规范、真实多端/多仓协作，并让 AI coding 从临时上下文驱动，升级为项目标准和代码事实驱动。**

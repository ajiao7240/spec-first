
```text
Shared Standards Repository
团队共享规范仓库
```

它不是业务项目的一部分，而是一个独立 Git 仓库，用来沉淀公司/部门/团队级规范。业务项目在开发过程中按需拉取、锁定版本、导入到本项目的 `.spec-first/standards/`，再和当前项目代码事实做对齐。

---

# 1. 最终定位

你的设想可以变成这条链路：

```text
shared-standards-repo
  ↓
按端 / 技术栈 / 研发场景组织团队规范
  ↓
业务项目通过 /spec:standards --import-source 拉取
  ↓
生成 imported-standards.json
  ↓
结合 graph facts / glue-map / 当前代码事实做 alignment
  ↓
生成 standards-preview.md
  ↓
项目团队确认
  ↓
写入 repo-profile.yaml confirmed_standards
  ↓
plan / work / review 全链路消费
```

也就是说：

> **团队规范仓库提供“标准输入”，业务项目负责“项目化确认”。**

这点很关键。
共享规范不能直接变成某个项目的强规则，必须经过当前项目确认。

---

# 2. 为什么独立 Git 仓库是好设计？

因为它解决了几个问题：

```text
1. 多项目复用
   不同业务项目可以共用一套前端、后端、API、数据库、移动端规范。

2. 规范版本化
   每个项目可以锁定某个 commit / tag，避免规范突然变化影响业务开发。

3. 团队协作
   规范仓库可以独立 PR、review、发布版本。

4. 权责清晰
   平台组/架构组维护共享规范，业务项目只导入和确认适用部分。

5. 避免业务仓污染
   业务仓只保存适配后的项目基线，不需要塞完整公司规范库。

6. 可演进
   未来可以支持公司级、部门级、团队级、项目级多层 standards。
```

# 5. 必须有一个总索引：`standards.yaml`

这是整个共享规范仓库的核心。

它让业务项目可以按需导入，而不是全仓复制。

示例：

```yaml

name: engineering-standards
version: 1.0.0

standards:
  - id: global.review.changelog-required
    title: Changelog 治理规则
    domain: global
    path: global/changelog-governance.md
    scope: company
    priority: 90
    applies_to:
      project_modes:
        - single_project_repo
        - monorepo_multi_module
        - multi_repo_workspace
      stacks: []
    tags:
      - review
      - governance
      - changelog

  - id: frontend.react.component-standards
    title: React 组件规范
    domain: frontend
    stack: react
    path: frontend/react/react-component-standards.md
    scope: team
    priority: 70
    applies_to:
      domains:
        - frontend
      frameworks:
        - react
      project_modes:
        - single_project_repo
        - monorepo_multi_module
    tags:
      - frontend
      - react
      - component

  - id: backend.spring.layering
    title: Spring Boot 分层规范
    domain: backend
    stack: spring-boot
    path: backend/java/spring-boot-standards.md
    scope: team
    priority: 70
    applies_to:
      domains:
        - backend
      frameworks:
        - spring-boot

  - id: api.response-shape
    title: 统一接口响应规范
    domain: api
    path: api/response-shape-standards.md
    scope: company
    priority: 85
    applies_to:
      domains:
        - api
    tags:
      - api
      - response
      - contract

  - id: glue.artifact-handoff
    title: Artifact Handoff 胶水规范
    domain: glue
    path: glue/artifact-handoff-standards.md
    scope: team
    priority: 80
    applies_to:
      domains:
        - glue
        - ai_workflows
    tags:
      - glue
      - artifact
      - workflow
```

这个文件回答：

```text
有哪些规范？

适用于哪些技术栈？
优先级是什么？
scope 是公司级、团队级还是项目级？
```

# 6. 每个规范文件的标准结构

共享规范文档不能只是散文，要有固定结构，方便 `/spec:standards --import` 解析。

推荐模板：

````markdown
# React 组件规范

## Metadata

- id: frontend.react.component-standards
- domain: frontend
- stack: react
- scope: team
- status: active
- version: 1.0.0
- owner: frontend-architecture-team
- applies_to:
  - React
  - Vite
  - Next.js

## Purpose

说明这条规范解决什么问题。

## Rules

### Rule 1: 组件命名使用 PascalCase

组件名和组件文件名使用 PascalCase。

Good:

```tsx
UserCard.tsx
OrderTable.tsx
````

Bad:

```tsx
user-card.tsx
order_table.tsx
```

### Rule 2: 跨业务复用组件放入 shared components

...

## Recommended Project Mapping

* `src/components`
* `src/features/*/components`
* `packages/ui`

## Review Checklist

* [ ] 组件命名是否符合 PascalCase
* [ ] 是否复用已有组件
* [ ] 是否避免把业务逻辑塞进通用组件
* [ ] 是否有必要的测试或 Storybook 示例

## Common Violations

* 重复创建 Button / Modal / Table
* 在页面组件中写复杂业务状态
* 通用组件依赖具体业务 domain

## Exceptions

说明哪些情况下可以例外。

## Related Standards

* frontend.api-client-standards
* frontend.styling-standards

````

重点是必须有：

```text
Metadata
Rules
Good / Bad
Review Checklist
Common Violations
Exceptions
Related Standards
````

这样导入后才能变成结构化 candidate。

# 7. 业务项目如何导入？

建议 `/spec:standards` 支持这几种方式。

## 7.1 从 Git URL 导入

```bash
/spec:standards --import-source git:https://github.com/company/spec-first-standards.git
```

可以指定版本：

```bash
/spec:standards --import-source git:https://github.com/company/spec-first-standards.git --ref v1.2.0
```

或者 commit：

```bash
/spec:standards --import-source git:https://github.com/company/spec-first-standards.git --commit abc123
```

---

## 7.2 从本地目录导入

```bash
/spec:standards --import-source ../spec-first-standards
```

适合公司内网、私有仓库、本地开发。

---

## 7.3 只导入某个 domain

```bash
/spec:standards --import-source ../spec-first-standards --domain frontend
/spec:standards --import-source ../spec-first-standards --domain backend
/spec:standards --import-source ../spec-first-standards --domain api
/spec:standards --import-source ../spec-first-standards --domain glue
```

---

## 7.4 自动匹配当前项目

最理想的是：

```bash
/spec:standards --import-source ../spec-first-standards --auto
```

流程：

```text
读取当前项目 project-shape.json
读取 shared standards standards.yaml
自动匹配适用规范
生成 import plan
```

例如当前项目识别为：

```text
React + Vite + Node backend + MySQL
```

则自动导入：

```text
global/review-governance.md
frontend/react/react-component-standards.md
frontend/api-client-standards.md
backend/node/nestjs-standards.md
api/response-shape-standards.md
database/mysql/mysql-standards.md
glue/reuse-first-standards.md
```

# 8. 业务项目内新增导入记录

建议业务项目生成：

```text
.spec-first/standards/standards-sources.json
.spec-first/standards/imported-standards.json
.spec-first/standards/import-lock.json
```

## 8.1 `standards-sources.json`

记录规范源。

```json
{
  "schema_version": "spec-first.standards-sources.v1",
  "sources": [
    {
      "id": "company-shared-standards",
      "type": "git",
      "url": "git@github.com:company/spec-first-standards.git",
      "ref": "v1.2.0",
      "commit": "abc123",
      "path": ".",
      "priority": 60,
      "scope": "company"
    }
  ]
}
```

## 8.2 `import-lock.json`

类似 package-lock，锁定导入版本，避免规范漂移。

```json
{
  "schema_version": "spec-first.import-lock.v1",
  "imports": [
    {
      "source_id": "company-shared-standards",
      "ref": "v1.2.0",
      "commit": "abc123",
      "imported_at": "2026-05-04T00:00:00Z",
      "standards": [
        {
          "id": "frontend.react.component-standards",
          "source_path": "frontend/react/react-component-standards.md",
          "content_hash": "sha256:xxx"
        },
        {
          "id": "api.response-shape",
          "source_path": "api/response-shape-standards.md",
          "content_hash": "sha256:yyy"
        }
      ]
    }
  ]
}
```

## 8.3 `imported-standards.json`

把导入文档解析成结构化 items。

```json
{
  "schema_version": "spec-first.imported-standards.v1",
  "source_id": "company-shared-standards",
  "items": [
    {
      "id": "api.response-shape",
      "domain": "api",
      "type": "interface_contract",
      "status": "imported",
      "rule": "接口响应结构统一为 { code, message, data }。",
      "source": {
        "path": "api/response-shape-standards.md",
        "section": "Rules"
      },
      "requires_confirmation": true,
      "suggested_action": "confirm"
    }
  ]
}
```

# 9. 导入后不能直接生效，必须做项目适配

共享规范是通用标准。
业务项目有自己的历史和现实。

所以导入后必须经过：

```text
Imported Standards
  ↓
Project Shape
  ↓
Graph / Code Facts
  ↓
Alignment Check
  ↓
Standards Preview
  ↓
Human Confirm
  ↓
Repo Profile
```

对齐结果：

```text
aligned
partially_aligned
conflict
not_observable
not_applicable
```

示例：

```json
{
  "id": "backend.layering.controller-service-repository",
  "source_type": "shared_standard",
  "status": "imported",
  "alignment": {
    "result": "partially_aligned",
    "matched_paths": [
      "src/modules/user/UserController.ts",
      "src/modules/user/UserService.ts"
    ],
    "conflict_paths": [
      "src/modules/order/OrderController.ts"
    ],
    "summary": "用户模块符合分层规范，订单模块存在 Controller 直接访问 Repository 的旧实现。"
  },
  "suggested_action": "confirm_as_target_standard"
}
```

这里的建议是：

```text
新代码遵循团队标准；
旧代码不强行重构；
触达时渐进改善；
冲突模块在 review 中提示风险。
```

# 10. 导入优先级

建议优先级：

```text
用户显式输入
  >
项目 repo-profile confirmed standards
  >
项目本地 domains/*.md
  >
团队共享规范仓库导入的 team standards
  >
公司共享规范仓库导入的 company standards
  >
graph observed conventions
  >
README / manifest / config
  >
AI suggested best practice
```

注意：

```text
共享规范优先于 graph observed conventions，
因为旧代码可能不是好规范。
```

但：

```text
项目 confirmed standards 优先于共享规范，
因为项目本地约束更具体。
```

# 11. 多端团队规范的组织方式

你说“各个端的开发，按标准目录结构、文档结构录入”，这个非常适合这样划分：

```text
frontend/
backend/
api/
database/
mobile/
admin/
glue/
ai-workflow/
global/
scenarios/
```

其中 `scenarios/` 很重要。

因为真实研发规范不只是按技术端，还要按场景：

```text
大需求多团队协作
前后端联调
多端一致性
接口变更
数据库变更
灰度发布
历史代码改造
新增 provider
新增 skill
代码审查
```

所以 `scenarios/` 建议保留。

例如：

```text
scenarios/
  multi-team-feature.md
  api-change.md
  database-change.md
  frontend-backend-collaboration.md
  mobile-h5-consistency.md
  legacy-refactor.md
  new-provider-integration.md
```

# 12. 和 `/spec:standards` 的关系

共享规范仓库不是替代 `/spec:standards`。

它是 `/spec:standards` 的输入源。

```text
shared standards repo
  ↓
/spec:standards --import-source
  ↓
imported-standards.json
  ↓
graph/code alignment
  ↓
standards-candidates.json
  ↓
standards-preview.md
  ↓
repo-profile.yaml
```

边界：

| 层级                        | 作用                       |
| ------------------------- | ------------------------ |
| shared standards repo     | 公司/团队通用规范库               |
| imported-standards.json   | 当前项目导入了哪些规范              |
| standards-candidates.json | 导入规范 + 代码观察 + AI 建议的候选合集 |
| standards-preview.md      | 给项目团队确认                  |
| repo-profile.yaml         | 当前项目确认后的核心标准             |
| domains/*.md              | 当前项目适配后的领域规范             |

# 13. 项目开发过程中如何拉取？

建议支持三种策略。

## 13.1 手动拉取

适合早期。

```bash
/spec:standards --import-source ../spec-first-standards --ref v1.2.0 --auto
```

---

## 13.2 需求开始前检查

在 `/spec:plan` 前检查：

```text
当前需求涉及 domain 是否已有导入规范？
import-lock 是否过期？
shared standards 是否有新版本？
```

如果有变化，提示：

```text
检测到团队 backend 规范有新版本 v1.3.0。
当前项目锁定 v1.2.0。
是否本次需求升级导入？
```

不要自动升级。

---

## 13.3 定期升级

例如每月或每个版本周期：

```bash
/spec:standards --import-source ../spec-first-standards --upgrade --ref v1.3.0
```

生成：

```text
standards-import-diff.md
standards-preview.md
repo-profile.patch.yaml
```

团队 review 后再合并。

# 14. 是否提交共享规范导入结果？

业务项目建议提交：

```text
.spec-first/standards/standards-sources.json
.spec-first/standards/import-lock.json
.spec-first/standards/imported-standards.json
.spec-first/standards/standards-candidates.json
.spec-first/standards/standards-preview.md
.spec-first/specs/repo-profile.yaml
```

不建议把共享规范仓库完整复制到业务项目。

原因：

```text
避免重复维护
避免业务仓膨胀
通过 import-lock 保证可追溯
通过 imported-standards 保存项目实际导入结果
```

# 17. 最终建议

这个方向非常值得做，而且应该进入 `/spec:standards` 的核心设计。

最终应该定义成三层 standards 来源：

```text
1. Shared Standards Repository
   公司/团队级通用规范库。

2. Project Standards Baseline
   当前项目导入、对齐、确认后的规范基线。

3. Runtime Standards Consumption
   plan/work/review 按需读取项目确认后的标准。
```

一句话：

> **共享规范仓库负责“标准化输入”，业务项目负责“项目化确认”，spec-first 负责“导入、对齐、证据化、预览、确认和消费”。**

这会让 spec-first 从单项目工具升级成：

```text
企业级 AI 研发规范治理与复用体系
```

但仍然保持轻量，因为：

```text
共享规范不直接强制生效；
项目只导入适用部分；
repo-profile 只写确认后的核心标准；
后续 skill 按需消费，不全量读取。
```

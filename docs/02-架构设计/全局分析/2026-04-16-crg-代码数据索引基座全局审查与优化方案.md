## 1. 结论摘要

### 1.1 总判断

`src/crg` 已经不是一个零散脚本集合，而是一套**结构完整的本地代码索引基座**。  
它已经具备以下闭环能力：

1. 输入收敛  
2. 多语言 AST 解析  
3. SQLite 图存储  
4. 增量构建  
5. generation/promote  
6. 社区/flow/风险/FTS 后处理  
7. retrieval 排序打包  
8. 面向 CLI 和上层工作流的查询/评审/上下文输出

如果把目标定义为：

- “成为 `spec-first` 当前单仓、本地优先、Stage-0/Review/Context 的代码数据索引基座”

那么当前体系已经**基本完整，且执行链路清晰**。

但如果把目标定义为：

- “成为业界领先级、可支撑高精度 AI 代码理解与上下文装配的通用代码图引擎”

那么当前体系还**没有达到终局形态**。核心差距不在“有没有图”，而在：

1. 语义边生产还偏薄  
2. 跨文件/跨语言解析精度还不够  
3. 结果去噪与代码域分层不够  
4. generation 健康门禁过浅  
5. retrieval 仍以 lexical + heuristic 为主，尚非强语义检索

### 1.2 一句话判断

**当前 CRG 是“可用、可维护、可演进”的 v1 基座，但还不是“业界领先级 AI 代码理解引擎”的终局版本。**

## 3. `src/crg` 当前在全局体系中的角色

从代码事实看，`CRG` 当前承担的是：

1. `spec-first` 的本地代码图构建器
2. 上层 Stage-0 / Context / Review 工作流的代码事实供应层
3. 本地 SQLite 形态的低成本可复用索引底座
4. 命令行可消费的统一代码 intelligence surface

它不是：

1. 纯粹的 parser 库
2. 只做静态检索的 FTS 工具
3. 只服务 bootstrap 的专用模块

从代码入口看，`bin/spec-first.js` 在进入 `crg` 子命令时会直接切到 `src/crg/cli/router.js`，说明 `CRG` 已经是 CLI 一级能力，而不是隐藏在内部的实验模块。

## 5.3 AST 解析层

相关文件：

- `src/crg/parser.js`
- `src/crg/chunking.js`

### 代码事实

`parseFile()` 当前真实做了这些事：

1. 敏感文件前置过滤
2. 按扩展名路由语言
3. 单次读文件同时做 SHA 与解析，避免 TOCTOU
4. 每个文件必建 `module` 节点
5. tree-sitter 可用时做 AST 提取
6. tree-sitter 不可用时 graceful degradation 为 `module_only`
7. JS/TS 额外生成 `calls`
8. 所有提取器普遍生成 `contains`
9. 模块导入普遍生成 `imports_from`
10. 所有非 module 节点补 `defined_in`
11. 基于跨度生成 chunk

### 当前真实产出的语义边

从 `parser.js` 的 raw edge 生产逻辑和当前数据库实际统计看，当前主图真实边种类只有 4 类：

1. `imports_from`
2. `contains`
3. `calls`
4. `defined_in`

`current graph.db` 中也只出现这 4 类边。

### 关键判断

这意味着当前 CRG 的 AST 层是：

- **结构图完整**
- **语义图不完整**

更准确地说：

1. “文件-符号-局部调用”已经有了
2. “跨文件高精度语义关系图”还没有

### 最大问题

虽然 `query.js` 暴露了 `inheritors_of`，并且按 `inherits/implements` 去查，但 `parser.js` 当前没有任何地方生产这两类边。  
这是明确的体系断层：

- 上层查询面已经承诺
- 下层生产面并未实现

这类能力在体系上应归类为“表面完整、实质缺失”。

### 另一个关键问题

`calls` 边目前只在 JS/TS 系提取器中生成。  
因此：

1. `flows.js` 基于 `calls` 构建 flow
2. `impact.js` 也高度依赖 `calls`
3. `review-context` 的 graph expansion 同样依赖 `calls`

这会直接导致：

- 非 JS/TS 语言虽然能入图，但在“影响分析 / flow / review blast radius”层面能力明显偏弱

### 结论

**AST 层已经能支撑 v1，但还没有形成业界领先所需的“强语义图生产能力”。**

## 5.4 图存储与增量层

相关文件：

- `src/crg/migrations.js`
- `src/crg/graph.js`
- `src/crg/incremental.js`
- `src/crg/cli/build.js`

### 代码事实

这一层已经具备：

1. `nodes / edges / flows / flow_nodes / communities / fingerprints / unresolved_edges / chunks / graph_meta / fts_nodes`
2. 外键约束
3. 索引
4. 增量 fingerprint 比对
5. 删除 stale 节点
6. 局部替换 changed 文件
7. unresolved_edges 审计表
8. fingerprint JSON 产物输出

`resolveEdges()` 当前使用 4 段解析策略：

1. 直接 `target_id`
2. `target_path_raw -> module`
3. `target_name -> symbol`
4. 同文件消歧

### 判断

这一层的工程结构是**扎实的**。  
尤其是下面三点很关键：

1. `unresolved_edges` 没有被丢弃，而是被保存为治理对象
2. changed 文件采用“先删旧事实，再写新事实”，避免幽灵节点
3. fingerprints 与输入收敛联动，而不是单纯按 git diff 更新

### 当前最大的事实缺口

当前仓库运行结果显示：

- `resolved edges = 1954`
- `unresolved edges = 2758`
- 其中 `calls unresolved = 2464`

这说明当前系统最大的瓶颈不是存储，而是**解析与解析后的语义解析率**。

也就是说：

- SQLite 层没有成为瓶颈
- `edge resolution accuracy` 才是当前核心瓶颈

### 结论

**图存储与增量层完整，语义解析率不足。**

- `src/crg/generations/rollback.js`

### 代码事实

当前已实现：

1. generation 目录隔离构建
2. 构建成功后 promote 到 `current.json`
3. 同步维护 `last-known-good.json`
4. 回退模块 `rollback.js` 已存在

### 当前不足

`assessGenerationHealth()` 当前只做了非常浅的健康判定：

1. `dbPath` 是否存在
2. `nodeCount > 0`

这离真正的 generation health gate 还差很远。  
因为当前仓库其实已经有更强的审计标准，但它们存在于：

此外，`rollback.js` 中的 `discardFailedGeneration()` 当前并没有接入 `build.js` 主链。  
这意味着：

1. 失败 generation 不会被 promote
2. 但失败 generation 目录也不会被主动清理

### 结论

**generation 模型方向正确，但健康门禁过浅，回滚闭环没有真正接入运行时主链。**

## 5.6 后处理分析层

相关文件：

- `src/crg/communities.js`
- `src/crg/flows.js`
- `src/crg/analyze.js`
- `src/crg/search.js`
- `src/crg/cli/postprocess.js`

### 代码事实

`runPostprocess()` 的顺序明确且正确：

1. `writeCommunities`
2. `detectFlows`
3. `analyzeGraph`
4. `rebuildFTS`

#### 社区层

`communities.js` 采用：

1. 顶层目录聚类
2. 密度/独立性健康度评估
3. 超大社区再做 BFS 拆分

#### flow 层

`flows.js` 采用：

1. `calls` 图
2. fan-in 为 0 的节点作为入口
3. BFS 深度上限 `5`
4. 单 flow 节点数上限 `20`
5. flow 数量上限 `100`

#### 图分析层

`analyze.js` 当前输出：

1. `surprisingConnections`
2. `godNodes`

#### 搜索层

`search.js` 当前输出：

1. FTS5 lexical 检索
2. 全量 rebuild
3. snippet 摘要

### 当前问题

#### 问题 1：社区模型偏“目录结构”，不偏“业务结构”

当前运行结果里已经可以看到两类现象：

1. `context` 的 `top_communities` 出现 `crg/0` 这类 machine-friendly 社区标识
2. `architecture` 的 `cross_community_edges` 中出现 `tests/8`、`tests/41` 这类测试社区 ID

这说明社区模型能工作，但仍然明显带有：

1. 目录优先
2. 测试/fixture 干扰
3. machine-friendly 多于 human-friendly

#### 问题 2：flow 更像“浅层调用扩展”，还不是“业务执行流”

原因很直接：

1. 入口定义仅依赖 `calls fan-in = 0`
2. 只看 `calls`
3. 深度和宽度都做了硬截断

这非常适合 v1 控制复杂度，但不适合被误认为“业务流程理解已完成”。

#### 问题 3：FTS 已有，但 semantic retrieval 仍较弱

`search.js` 是 FTS；`semantic-rerank.js` 只是 token overlap 加分。  
这意味着当前 retrieval 是：

- lexical + heuristic + graph neighborhood

还不是：

- 强语义 hybrid retrieval

### 结论

**后处理分析层已经形成可工作的 intelligence layer，但离“高精度架构理解层”还有明显距离。**

## 5.7 Retrieval 与 AI 上下文赋能层

相关文件：

- `src/crg/retrieval/api.js`
- `src/crg/retrieval/seed.js`
- `src/crg/retrieval/expand.js`
- `src/crg/retrieval/rerank.js`
- `src/crg/retrieval/semantic-rerank.js`
- `src/crg/retrieval/pack.js`
- `src/crg/retrieval/query-plan.js`
- `src/crg/commands/review-context.js`
- `src/crg/cli/context.js`

### 当前 retrieval 链路

```text
[用户问题 / review 场景]
          |
          v
   buildQueryPlan()
          |
          v
    buildSeedSet()
          |
          v
    expandSeedSet()
          |
          v
    rerankContext()
          |
          v
   semanticRerank()
          |
          v
      packContext()
          |
          v
  ranked_context + estimated_tokens
```

### 代码事实

当前 retrieval 已经具备：

1. `review / plan / work / search` profile
2. changed file / candidate tests / risk paths 的加权
3. graph neighbor 扩展
4. chunk 扩展
5. token budget 裁剪
6. review-context 中与 diff / graph expansion / candidate tests 联动

### 当前不足

#### 不足 1：semantic 并不是真正 semantic

`semantic-rerank.js` 当前只是：

- query term 和 `name + retrieval_text` 的 overlap 计数

它不是 embedding，不是 learned ranker，也不是 code semantic model。

#### 不足 2：检索对象还不够“角色感知”

当前 `packContext()` 主要按：

1. score
2. 单文件最多 2 个条目
3. token budget

来裁剪，而不是按：

1. product code / test / fixture / tool
2. call path completeness
3. ownership / risk / contract relevance

所以当前 `context` 输出里会把 `skills/onboarding/scripts/inventory.mjs` 之类的脚本推到非常靠前的位置。  
这不是错误，而是说明当前 retrieval 还没有把“仓库角色分层”纳入核心排序。

### 结论

**Retrieval 已经形成 AI 可消费主链，但还没有形成“高信噪比、强角色感知”的领先级上下文装配器。**

这类问题不影响代码运行，但会直接伤害体系可维护性与团队认知一致性。

## 7. 当前最关键的系统性问题

## 7.1 语义图过薄，是当前第一瓶颈

当前最核心的问题不是“没有 graph”，而是：

- graph 的结构完整度高于语义完整度

具体表现：

1. 只有 4 类主边
2. `inherits / implements / tested_by / route_to / exports / reads / writes` 等关键边完全缺席
3. 非 JS/TS 语言缺少实用级调用边
4. unresolved edge 数量高于 resolved edge 数量

这是当前离“领先级 AI 理解”最远的一层。

## 7.2 代码域未分层，导致架构理解噪声偏大

当前主图把：

1. `src`
2. `tests`
3. `skills`
4. `agents`
5. fixture

都放在同一主图里，并且 retrieval 没有强角色权重。  
所以 `context` 的 top hub / ranked context 已经出现非核心产品代码抢占高位的情况。

这不是 parser 问题，而是**索引域设计问题**。

## 7.3 generation 健康门禁太浅

当前健康判定只看：

1. DB 是否存在
2. nodeCount 是否大于 0

但当前系统明明已经具备更丰富的质量指标：

1. unresolved rate
2. parser degradation
3. orphan / duplicate / mismatch
4. FTS row consistency
5. community / flow availability

这些没有进入 runtime 主链，是明显短板。

## 7.4 retrieval 还不是强语义 retrieval

当前 retrieval 已经可用，但距离领先级还有 3 个缺口：

1. 没有真正 semantic ranker
2. 没有 call-path / ownership / contract pack
3. 没有 code role aware packing

## 7.5 维护热点过大

当前最明显的维护热点：

1. `parser.js`
2. `input-convergence.js`
3. `cli/build.js`
4. `cli/query.js`

这些文件都已经过大。  
在当前阶段仍可维护，但如果继续往里叠能力，不先拆模块，复杂度会快速失控。

---

## 8. 正确的优化方向

为避免与配套总纲文档混淆，这一节只定义 **CRG 子系统内部施工顺序**。  
平台级应理解为：

1. 先完成本文 `8.1-8.3`
2. 再推进 bootstrap/context compiler 的真实编译闭环
3. 最后再做统一 workflow runtime / observability

也就是说，本文并没有主张“先把全平台做成重型统一引擎”；本文主张的是先把 `CRG` 底座质量补齐。

## 8.1 P0：先把“强语义生产层”补齐

这是最关键的一层，优先级高于再加新命令。

### 必做项

1. 把 `parser.js` 的“结构提取”升级为“结构提取 + 语义关系提取”分层
2. 增加统一 `relation extractors` 概念，而不是继续把所有关系塞进各语言提取器
3. 第一批新增边建议：
   - `inherits`
   - `implements`
   - `tested_by`
   - `exports`
   - `route_to`
4. 为 Python / Go / Java / Ruby / Swift / Kotlin / C# 增加最小可用调用边生产
5. 引入 per-language import resolver / symbol resolver，而不是只靠 name/path heuristic

### 为什么这是 P0

因为 `impact / flow / architecture / review-context / query` 的上层价值都直接依赖这层。  
不先补这一层，继续扩命令面只会放大“表面丰富、底层偏薄”的问题。

---

## 8.2 P0：引入代码域分层

正确做法不是简单排除 `tests/skills`，而是给每个文件增加 `code_role`：

1. `product`
2. `test`
3. `fixture`
4. `tooling`
5. `workflow_runtime`
6. `knowledge_script`

然后：

1. 主架构分析默认优先 `product`
2. review 场景按 `product > test > tooling`
3. debug / workflow 相关问题可再拉高 `tooling / skills`

### 这样做的收益

1. 不牺牲索引完整性
2. 明显提升 `context` 的信噪比
3. 可以让同一套图服务多种工作流

---

## 8.3 P0：把 generation 健康门禁升级为真正 gate

当前正确做法应是：

### `assessGenerationHealth()` 最少纳入

1. `node_count > 0`
2. `edge_count > 0`
3. `unresolved_edge_rate` 阈值
4. `parse_error_count / no_parser_count / module_only_count`
5. `fts_row_mismatch == 0`
6. `orphan_edges == 0`
7. `community_count > 0`
8. `flow_count > 0` 或明确允许为空的条件

### 运行时动作

1. 健康通过才 promote
2. 健康失败就调用 `discardFailedGeneration()`
3. 把失败原因写入结构化 warning

### 原因

当前仓库已有 SQLite audit 标准，但没有进入 build 主链。  
这相当于“知道什么是健康，但没有在生产路径里真正执行”。

## 8.5 P1：把 retrieval 升级为 hybrid retrieval

当前正确方向不是一上来引入重型向量库，而是按顺序升级：

### 第一步

1. 保留 FTS
2. 保留 graph expansion
3. 引入 `code_role` 权重
4. 引入 `call_path pack`
5. 引入 `ownership / contract / risk` pack

### 第二步

1. chunk 级 FTS 检索成为一等能力，不只在 changedFiles 下触发
2. `semantic-rerank` 升级为真正的 embedding / model ranker
3. 支持 query intent 到 pack strategy 的更细粒度映射

### 目标

从“找到若干相关节点”升级为“组装出最小但足够的行动上下文包”。

---

## 8.6 P1：把 flow 从“浅调用扩展”升级为“入口驱动流程图”

当前 flow 入口是：

- `kind != module && no incoming calls`

终局应增加真正入口识别器：

1. CLI entry
2. HTTP route handler
3. framework lifecycle
4. job / worker consumer
5. test entry

然后把 flow 分成：

1. `runtime flow`
2. `integration flow`
3. `test flow`
4. `tooling flow`

这样 `context / architecture / review-context` 的解释能力会明显提升。

---

## 8.7 P2：补齐 contract 真源

当前正确做法：

1. 补回 `docs/contracts/crg-cli-v1.schema.json`
2. 让 Jest contract 与 schema 同时成为真源
3. build / command 输出在 CI 中做 schema 校验

这是维护性问题，不是算法问题，但必须做。

## 10. 最终判断

### 10.1 当前已经具备的能力

当前 `src/crg` 已经真正具备：

1. 本地优先代码索引构建
2. 多语言 AST 结构提取
3. SQLite 图存储与增量刷新
4. generation/promote 基础能力
5. 社区、flow、风险、FTS、review-context 等上层消费面
6. 强测试和审计支撑

### 10.2 当前还没有完成的能力

当前还没有完成：

1. 强语义边生产
2. 高精度跨文件符号解析
3. 角色感知的主图去噪
4. 真正的 generation 健康门禁
5. 领先级 hybrid retrieval
6. schema contract 真源闭环

### 10.3 最客观的结论

以代码事实为准，当前 `src/crg` 的正确评价应是：

- 它已经是一个**结构清晰、可运行、可审计、可演进的代码索引基座**
- 但它还不是一个**业界领先级、强语义、强去噪、强上下文装配的终局代码理解引擎**

### 10.4 正确的优化顺序

不要先继续扩命令面，也不要先做重型平台化。  
正确顺序应是：

1. 先补语义生产
2. 再补代码域分层
3. 再把 generation 健康门禁拉齐
4. 再升级 retrieval
5. 最后再扩更高层工作流赋能

这是当前既不过度设计、又能真正把 `CRG` 推到高质量终局的正确路径。

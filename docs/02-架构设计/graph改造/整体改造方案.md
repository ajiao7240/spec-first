> 状态说明：本文是 CRG 删除前后的探索性设计记录，不代表当前 runtime contract。
> 当前实现已移除内置 CRG 与 graph-bootstrap workflow。
> 如需恢复图谱能力，请另见后续 external graph provider 集成方案。

不考虑向下兼容，方案可以更激进、更干净：**不要再保留 `src/crg` 作为旧入口，不要继续把内置 CRG 当主线，而是把 spec-first 升级成“Graph Context Layer + Harness Workflow Orchestrator”。**

最终目标：

> **GitNexus 负责全局代码库认知。**
> **upstream code-review-graph 负责增量影响分析与最小上下文。**
> **spec-first 负责 workflow、产物协议、上下文路由、review gate 和证据闭环。**

当前 master 里，spec-first 还是以 “CRG query-first runtime” 为核心，`graph-bootstrap` 会构建 CRG graph index，并产出 `graph-quality.json`、`graph-index-status.json`、`code-navigation.json`、`repo-topology.json`、`graph-operations.jsonl` 等证据；README 也明确写了 `spec-first crg *` 是内置 Node.js runtime over SQLite + FTS5。既然你说“不需要考虑向下兼容”，就可以把这套内置 CRG 主线整体替换为 provider-based graph runtime。([GitHub][1])

# 2. 两个工具的职责边界

## 2.1 GitNexus：Global Knowledge Provider

GitNexus 适合做全局项目认知，因为它的定位是将代码库索引成 knowledge graph，并支持依赖、调用链、cluster、execution flow、MCP、多 repo registry 等能力；它也说明一个 MCP server 可以服务多个 indexed repos，不需要每个项目单独配置 MCP。([GitHub][2])

在 spec-first 中，GitNexus 负责：

```text
1. 全局代码库索引
2. architecture map
3. module clusters
4. dependency map
5. call chain / execution flow
6. entrypoints
7. repo wiki / architecture facts
8. plan / tasks 阶段的项目认知输入
```

它不负责：

```text
1. spec-first 的需求分析
2. task 拆分
3. work 执行策略
4. review gate
5. changelog/docs/tests 治理
```

---

## 2.2 upstream code-review-graph：Impact Context Provider

upstream code-review-graph 适合做增量影响和最小上下文，因为它 README 里明确说它用 Tree-sitter 构建结构图，跟踪变化，并通过 MCP 给 AI assistant 精准上下文，避免重复读取整个代码库；它要求 Python 3.10+，推荐 uv，MCP 配置会优先使用 `uvx`。([GitHub][3])

在 spec-first 中，code-review-graph 负责：

```text
1. changed files
2. blast radius
3. impacted callers / dependents
4. related tests
5. minimal context
6. review context
7. work 前 context pack
8. work 后 impact diff
```

它不负责：

```text
1. 全局架构解释的主事实源
2. spec-first 阶段协议
3. task acceptance criteria
4. review persona routing
5. workflow governance
```

# 12. Context Select 详细流程

```text
输入：
  task.md

步骤：
  1. 解析 task objective / scope / acceptance criteria
  2. GitNexus 查询相关模块、入口、架构事实
  3. code-review-graph 查询 minimal context
  4. 合并两边结果
  5. 去重、排序、打分
  6. 生成 reading order
  7. 生成 avoid_reading
  8. 输出 context-pack
```

打分规则：

```text
score = task_relevance * 0.40
      + impact_relevance * 0.30
      + architecture_relevance * 0.20
      + test_relevance * 0.10
```

注意：这个打分只用于排序，不做 hard gate，继续符合 spec-first “Let the LLM decide” 的原则。README 也强调 spec-first 是提升 LLM decision input，不用状态机替代 LLM 判断。([GitHub][1])

# 17. 错误与 fallback 设计

不考虑旧 CRG 兼容后，fallback 不再是 builtin CRG，而是：

```text
1. GitNexus 不可用：
   - global knowledge state = unavailable
   - fallback = targeted direct repo reads
   - graph-bootstrap 仍可继续 code-review-graph build
   - 但 project-knowledge docs 标记 global facts low confidence

2. code-review-graph 不可用：
   - impact context state = unavailable
   - fallback = git diff + targeted direct repo reads
   - context-select 不生成 minimal context，只生成 architecture-first context

3. 两者都不可用：
   - graph-bootstrap failed
   - workflow 可继续，但必须显式报告 no graph provider
```

保留 spec-first 的核心理念：

```text
不要伪造 graph evidence。
不要读取旧静态包当事实。
fallback 必须显式写入 provenance / limitations。
```

# 21. 最终形态总结

不考虑向下兼容后，最干净的最终形态是：

```text
spec-first
  不再内置 CRG 图引擎
  不再维护 spec-first crg 子命令
  不再用 graph.db 作为主事实路径

spec-first
  维护 Graph Context Layer
  默认使用 GitNexus + code-review-graph
  统一 normalize 成 spec-first evidence
  让 workflow skills 消费 evidence
```

最终数据流：

```text
Source Repo
  ├─ GitNexus
  │    └─ Global Graph Facts
  │         └─ project-map / module-boundaries / architecture-facts
  │
  └─ code-review-graph
       └─ Impact Context Facts
            └─ minimal-context / blast-radius / related-tests

spec-first
  └─ Normalize + Workflow Contracts
       └─ Spec → Plan → Tasks → Context → Work → Review
```

最终一句话：

> **直接删除内置 `src/crg` 主线，把 GitNexus 和 upstream code-review-graph 作为两个强制 graph providers 接入；spec-first 只保留 provider 编排、事实归一化、workflow 产物、context routing 和 review evidence。这样 spec-first 才不会继续重复造图谱轮子，而是成为真正的 Harness Engineering 编排层。**

# Spec-First Framework 目录结构

## 1. 目录视角

这个项目同时存在两种目录视角：

1. 仓库源码视角：`spec-first` npm 包本身包含的目录结构
2. 目标项目运行态视角：用户执行 `spec-first init --claude` 后，在目标项目里生成的 `.claude/` 目录

```text
├── bin/

├── src/

│       ├── index.js
│       ├── commands/
│       │   ├── doctor.js
│       │   └── init.js
│       ├── plugin.js
│       ├── spec-commands.js
│       ├── skills.js
│       └── templates.js
├── skills/
│   ├── spec-brainstorm/
│   ├── spec-plan/
│   ├── spec-work/
│   ├── spec-code-review/
│   ├── spec-graph-bootstrap/     ← Stage-0 supporting workflow
│   └── ...  # 全量随包发布的 skill 目录
├── agents/
│   ├── review/
│   ├── research/
│   ├── workflow/
│   └── ...  # 全量随包发布的 agent 目录
├── templates/

```

## 3. 目标项目运行态视角

`spec-first init --claude` 会在目标项目里生成：

```text
your-project/
└── .claude/
    ├── commands/
    │   └── spec/
    │       ├── brainstorm.md
    │       ├── plan.md
    │       ├── work.md
    │       ├── review.md
    │       └── compound.md
    ├── skills/
    │   ├── spec-brainstorm/
    │   ├── spec-plan/
    │   ├── spec-work/
    │   ├── spec-code-review/
    │   └── ...  # 与 npm 包内 skills/ 同步
    ├── agents/
    │   ├── review/
    │   ├── research/
    │   ├── workflow/
    │   └── ...  # 与 npm 包内 agents/ 同步
    └── ...
```

## 4. 运行态产物视角

各 workflow 运行后，目标项目新增：

```text
your-project/
├── .spec-first/                           ← 整体 gitignored，机器产物
│   ├── graph/
│   │   ├── graph.db                       ← SQLite 图数据库（crg build）
│   │   └── input-fingerprints.json        ← 增量构建指纹快照（crg build）
│   └── workflows/
│       ├── bootstrap/<slug>/              ← bootstrap 控制面（spec:graph-bootstrap）
│       │   ├── fact-inventory.json
│       │   ├── risk-signals.json
│       │   ├── test-surface.json
│       │   ├── artifact-manifest.json
│       │   ├── tasks/
│       │   └── backups/
│       ├── spec-code-review/<run-id>/          ← review per-run 产物（spec:code-review interactive/autofix）
│       ├── spec-plan/deepen/              ← plan deepen 暂存（spec:plan artifact-backed mode）
│       ├── feature-video/<run-id>/        ← 截图/视频暂存（feature-video，上传后删除）
│       │   ├── screenshots/
│       │   └── videos/
│       └── todo-resolve/<run-id>/         ← resolve 并发暂存（todo-resolve 大集合模式）
└── docs/                                  ← VCS 资产，可入库
    ├── brainstorms/                       ← 需求文档（spec:brainstorm）
    ├── plans/                             ← 实施计划（spec:plan）
    ├── solutions/                         ← 知识沉淀（spec:compound）
    ├── contexts/<slug>/                   ← bootstrap 上下文产物（spec:graph-bootstrap）
    └── todos/                             ← 持久工作项（todo-create/triage/resolve）
```

## 5. 说明

- `templates/claude/commands/spec/` 是 npm 包内部模板来源，不是目标项目运行态目录
- `.claude/commands/spec/` 是用户真正执行 `/spec:*` 时的命令来源，这层只负责稳定入口
- `.claude/skills/` 是 `spec-first init --claude` 同步下发的 skill 运行态目录，用于保留发布后可引用的完整 skill 资产
- `.claude/agents/` 是 `spec-first init --claude` 同步下发的 agent 运行态目录，用于给 skills 调度子代理
- `.claude-plugin/plugin.json` 是发布包内的统一资产清单，`init` / `doctor` 会基于它发现 commands、skills、agents
- 当前流程的稳定产物以 `docs/brainstorms/`、`docs/plans/`、`docs/solutions/` 为主，不要求目标项目存在 `.claude/tasks/`
- `spec-graph-bootstrap` 运行后，目标项目会新增 `docs/contexts/<slug>/`（长期上下文资产）和 `.spec-first/workflows/bootstrap/<slug>/`（控制面，可不提交）
- `skills/` 和 `agents/` 现在属于 npm 包发布产物；`scripts/` 仍属于仓库级开发与历史资产，不进入发布包
- `docs/todos/` 是持久工作项目录（VCS 资产），由 todo-create/triage/resolve 工作流共享；默认可入库，用户如不想提交可在项目 `.gitignore` 中添加 `docs/todos/`
- `.spec-first/workflows/` 下除 `bootstrap/` 外的子目录均为运行时暂存产物，随 `.spec-first/` 整体 gitignored

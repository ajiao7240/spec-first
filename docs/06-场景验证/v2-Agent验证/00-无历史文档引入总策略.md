# 无历史文档引入总策略（Spec-First）

> 适用范围：已有或新建代码工程，当前开始引入 Spec-First，历史规范文档为空  
> 目标：用最小成本建立可执行基线，并平滑进入标准研发流程

## 1. 核心原则

1. 先建“可分析基线”，再做新增/修改需求。
2. 基线只做最小闭环，不追求一次性补齐全部历史。
3. 每个需求都走 00-07 阶段，并保留 Gate 与度量证据。

## 2. 一次性引入步骤（全团队统一）

## Step 1：工具链初始化

```bash
spec-first init
spec-first doctor
```

产出：

1. `.spec-first/` 基础目录与配置。
2. `specs/.feat-registry.md`。
3. Hook 安装与环境可用性确认。

## Step 2：建立项目级规则

补齐：

1. `.spec-first/constitution.md`（技术约束、质量门槛、角色映射、RACI）。
2. `.spec-first/config.yaml`（默认模式、平台、Gate 配置、可选 Jira 配置）。

## Step 3：创建“遗留基线 Feature”（一次）

```bash
spec-first init feature FSREQ-000000-legacy-baseline --mode I --platforms Backend
```

最小补齐内容：

1. `spec.md`：当前已上线能力摘要（仅主流程/主接口）。
2. `design.md`：现状架构与关键模块边界。
3. `traceability-matrix.*`：先登记核心链路（高频/高风险功能）。

## Step 4：开始真实需求研发

按需求类型进入对应场景文档执行：

1. 新增功能：`01-AgentA-现有工程新增功能场景.md`
2. 修改功能：`02-AgentB-现有工程修改功能场景.md`
3. 全新工程：`03-AgentC-全新项目全新工程场景.md`

## 3. 三场景统一命令骨架

```bash
# 启动需求
spec-first init feature <feature-id> --mode <N|I> --platforms <list>

# 阶段推进（每阶段执行）
spec-gate check <feature-id> --stage 01_specify --fix-hints
spec-gate check <feature-id> --stage 02_design --fix-hints
spec-gate check <feature-id> --stage 03_plan --fix-hints
spec-gate check <feature-id> --stage 04_implement --fix-hints
spec-gate check <feature-id> --stage 05_verify --fix-hints
spec-gate check <feature-id> --stage 06_wrap_up --fix-hints
spec-gate check <feature-id> --stage 07_release --fix-hints

# 收尾度量
spec-metrics report <feature-id> --output specs/<feature-id>/reports/metrics-report.md
```

## 4. 角色分工建议（首次引入期）

1. PM：需求清晰度、验收标准、UAT 签核。
2. Tech Lead：阶段推进、Gate 质量、架构一致性。
3. Dev：任务拆解、实现与追踪链维护。
4. QA：验证策略、覆盖率与缺陷闭环。
5. Ops：发布策略、回滚演练、观察窗口。
6. DevEff：模板固化、命令规范、执行效率度量。

## 5. 验收标准（引入成功定义）

1. `doctor` 全绿或仅非阻断告警。
2. `FSREQ-000000-legacy-baseline` 完成最小三件套（spec/design/matrix）。
3. 至少 1 个真实需求从 `00_init` 走到 `07_release`。
4. Gate 历史、追踪矩阵、度量报告三类证据完整。

## 6. 常见误区

1. 跳过基线直接做新需求：后续 Impact Analysis 无输入，容易漏改。
2. 只建目录不建规则：`constitution` 缺失会导致执行标准漂移。
3. 只做文档不跑 Gate：流程会退化成“写文档动作”，无法形成质量防线。

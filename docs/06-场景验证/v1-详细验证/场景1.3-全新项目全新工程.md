## 场景概述

### 业务背景

公司决定开发一个全新的「内部知识库管理系统」（Knowledge Base，简称 KB），从零开始搭建。无历史代码、无已有规范、无存量产物。这是最纯粹的 Spec-First 场景——没有任何历史包袱，完整展示 Spec-First 在大型项目中从 0 到 1 的全链路管控能力。

### 与场景 1.1 / 1.2 的核心区别

| 对比维度 | 场景 1.1（现有工程+新功能） | 场景 1.2（现有工程+改功能） | **场景 1.3（全新项目+全新工程）** |
|---------|------------------------|------------------------|-------------------------------|
| 项目状态 | 已有工程，已 `spec-first init` | 已有工程，已有历史 Feature | **全新 Git 仓库，从零初始化** |
| Mode | I（Iteration） | I（Iteration） | **N（New）** |
| Size | M | S | **L（完整深度）** |
| 历史产物 | 有，需 Impact Analysis | 有，需 Baseline 定位 | **无，纯净起步** |
| 前置工作 | 环境检查即可 | 环境检查 + 历史定位 | **Git 仓库创建 + 项目级初始化 + constitution.md** |
| 流程深度 | 标准产出物 | 轻量产出物 | **完整产出物 + 深度评审** |
| 核心价值 | 验证迭代兼容性 | 验证轻量裁剪 | **验证 Spec-First 完整能力上限** |

### 第一个 Feature 定义

| 属性 | 值 |
|------|-----|
| Feature 名称 | `001-kb-article-crud-search` |
| Feature 描述 | 知识文章的 CRUD + 全文搜索 + 权限管控 |
| FEAT 缩写 | `KB` |
| Mode | N（全新功能，无历史产物） |
| Size | L（6+ 模块、22 AC、8 个新 API、表级数据模型、2 个跨团队协作） |
| 涉及端 | H5 + PC + Backend |

### Size L 判定依据（5 维度取最高）

| 维度 | 实际值 | 判定 | 说明 |
|------|--------|------|------|
| 模块数 | 6 | **L**（>=6） | 文章服务、搜索服务、用户服务、权限服务、存储服务、前端 |
| AC 数 | 22 | **L**（>=16） | 涵盖 CRUD、搜索、权限、版本管理等 |
| API 变更数 | 8 个新 API | **L**（>=6） | 全部为新建 API |
| 数据模型 | 表级变更（新建 5 张表） | **L**（level=2） | kb_article, kb_category, kb_tag, kb_permission, kb_search_index |
| 跨团队 | 2 | **L**（>=2） | 前端团队 + 后端团队 |

> **5 维度全部命中 L**，无需 Tech Lead 手动覆盖。

### Step 3：编写 `constitution.md`（项目宪法）

constitution.md 是项目级的规范宪法，定义团队约定、命名规范、技术栈约束等。**全新项目必须在首个 Feature 启动前完成**——Init 内联校验会检查 constitution.md 存在性。

```markdown
<!-- .spec-first/constitution.md -->
# Knowledge Base — 项目宪法

## 技术栈
- 后端：Node.js 20 + NestJS + PostgreSQL + Elasticsearch
- 前端（PC）：React 18 + TypeScript + Ant Design Pro
- 前端（H5）：React 18 + TypeScript + Ant Design Mobile
- 部署：Docker + Kubernetes
- 搜索引擎：Elasticsearch 8.x

## 命名规范
- API 路径：`/api/v1/<resource>` RESTful 风格
- 数据库表名：`kb_<entity>` 前缀，snake_case
- 前端组件：PascalCase，按功能域分目录
- ID 格式：严格遵循 Spec-First 8 种 ID 规范

## 分支策略
- 主干：main
- Feature 分支：feat/<feature-id>
- 合并方式：Squash Merge
- Size L Feature：使用 Git Worktree 隔离并行开发

## 代码审查
- 所有 PR 至少 1 人 Review
- Size L Feature 需 Architect 参与 Review
- 跨团队 PR 需双方 Lead 签核

## 非功能约束
- API 响应时间 P99 < 500ms
- 搜索响应时间 P99 < 200ms
- 可用性 SLA >= 99.9%
- 单文章最大 10MB
- 并发用户 >= 500
```

# 技术调研 — 知识文章 CRUD + 全文搜索

## DS-KB-001 搜索引擎选型

### 候选方案

| 方案 | 优势 | 劣势 | 成本 |
|------|------|------|------|
| Elasticsearch 8.x | 成熟生态、中文分词插件丰富 | 运维复杂度高、内存消耗大 | 中 |
| Meilisearch | 轻量、开箱即用 | 中文分词弱、社区较小 | 低 |
| PostgreSQL FTS | 无额外依赖 | 中文分词差、大数据量性能不足 | 低 |

### Trade-off 分析（Size L 必须）

| 维度 | Elasticsearch | Meilisearch | PostgreSQL FTS |
|------|--------------|-------------|----------------|
| NFR-PERF-001 达标 | ✅ P99 < 100ms | ✅ P99 < 150ms | ❌ P99 > 500ms |
| 中文分词质量 | ✅ IK 分词器 | ⚠️ 基础分词 | ❌ 需自建 |
| 运维复杂度 | ⚠️ 中等 | ✅ 低 | ✅ 低 |
| 团队经验 | ✅ 有经验 | ❌ 无经验 | ✅ 有经验 |

### 决策

选择 **Elasticsearch 8.x**。理由：NFR-PERF-001 要求搜索 P99 < 200ms，仅 ES 可稳定达标；团队有 ES 运维经验。

### ADR 记录

ADR-001: 搜索引擎选型 Elasticsearch 8.x
- 决策日期: 2026-02-08
- 决策人: 张工(Architect) + Leo(Tech Lead)

### 阶段目标

执行 Canary 部署（Size L 要求），通过 24 小时观察窗口，Smoke Test 通过，核心指标无异常，Feature 进入 Done 终态。

> **Size L 完整性要求**：
> - 部署策略为 **Canary 部署**（Size S 为 Rolling Update，Size M 为 Blue-Green）
> - 观察窗口为 **24 小时**（Size S 为 30 分钟，Size M 为 2 小时）
> - Gate Owner 为 Tech Lead + Ops 双人签核

### 具体操作步骤

**步骤 1：Canary 部署（Size L 要求）**

Size L 部署策略对比：

| 部署策略 | Size S | Size M | Size L |
|---------|--------|--------|--------|
| Rolling Update | ✅ | ❌ | ❌ |
| Blue-Green | ❌ | ✅ | ❌ |
| **Canary** | ❌ | ❌ | **✅** |

Canary 部署流程：

```
阶段 1: 5% 流量 → Canary 实例（1 小时观察）
阶段 2: 25% 流量 → 扩展 Canary（4 小时观察）
阶段 3: 50% 流量 → 半量部署（8 小时观察）
阶段 4: 100% 流量 → 全量发布（持续观察至 24 小时）
```

**步骤 2：执行 Smoke Test**

```bash
# Smoke Test 覆盖核心路径
# 1. 文章创建（PC 端）
# 2. 文章搜索（H5 端）
# 3. 权限校验（未授权访问拒绝）
# 4. API 响应时间（P99 < 500ms）
```

**步骤 3：24 小时观察窗口（Size L 要求）**

观察窗口对比：

| Size | 观察窗口 | 监控指标 |
|------|---------|---------|
| S | 30 分钟 | 错误率、响应时间 |
| M | 2 小时 | 错误率、响应时间、资源使用 |
| **L** | **24 小时** | **错误率、响应时间、资源使用、业务指标、用户反馈** |

**步骤 4：通过 Gate 4: Go Live**

```bash
spec-gate check 001-kb-article-crud-search --fix-hints
```

## 场景特殊考量

### 1. Size L 的完整流程优势

Size L 是 Spec-First 能力的完整展示，体现了"不跳过阶段，只调节深度"的核心理念：

| 阶段 | Size S（轻量） | Size L（完整） | 深度差异 |
|------|--------------|--------------|---------|
| 01 Specify | 简版 spec.md | 完整版 + NFR 量化 | AC 覆盖率强制 >= 90% |
| 02 Design | research.md 可省略 | 完整版 + Trade-off + Mock Server | Architect 必须参与 |
| 03 Plan | 1-3 TASK | 15+ TASK | 细粒度到可独立开发 |
| 04 Implement | 串行开发 | Git Worktree 隔离并行 | 跨团队物理隔离 |
| 05 Verify | OWASP + SCA | + SAST + DAST + 渗透测试 | 安全扫描全覆盖 |
| 07 Release | Rolling Update / 30min | Canary / 24h 观察 | 渐进式发布 |

### 2. 多端（H5+PC+Backend）的 Layer 2 规则合并

本场景涉及 3 个端，Layer 2 规则合并是关键复杂点：

**合并策略**：

| 规则类型 | 合并方式 | 本场景示例 |
|---------|---------|----------|
| entry_criteria | 追加（取并集） | H5 兼容性 + PC 浏览器兼容 + Backend 性能基线 |
| deliverables | 追加（取并集） | H5 适配方案 + PC 响应式方案 + Backend API 文档 |
| exit_gate_items | 追加（取并集） | H5 性能预算 + PC 可访问性 + Backend 压测报告 |
| 阈值类 | 各端隔离 | H5 首屏 < 2s / PC 首屏 < 1.5s / Backend P99 < 500ms |

**各阶段的 Layer 2 影响**：

| 阶段 | H5 端追加 | PC 端追加 | Backend 端追加 |
|------|----------|----------|---------------|
| 01 Specify | 移动端交互规格 | 桌面端交互规格 | API 性能基线 |
| 02 Design | 响应式适配方案 | 大屏布局方案 | 数据库索引设计 |
| 03 Plan | H5 端 TASK 拆分 | PC 端 TASK 拆分 | Backend TASK 拆分 |
| 04 Implement | Ant Design Mobile | Ant Design Pro | NestJS + ES |
| 05 Verify | 移动端兼容性测试 | 浏览器兼容性测试 | 压力测试 |
| 07 Release | H5 CDN 部署 | PC CDN 部署 | K8s Canary 部署 |

### 3. 跨团队协作的 Gate 签核

Size L 涉及 2 个跨团队协作（前端团队 + 后端团队），Gate 签核需要多方参与：

| 阶段 Gate | 签核人 | 说明 |
|----------|--------|------|
| 00 Init 内联校验 | Tech Lead | 单人签核即可 |
| 01 Specify 内联校验 | PM + Tech Lead | DoR 需 PM 确认业务需求 |
| ★ Gate 1: Design Ready | **Architect** + Tech Lead | Size L 强制 Architect 参与 |
| 03 Plan 内联校验（合并到 Gate 2） | Tech Lead + **Frontend Lead** + **Backend Lead** | 跨团队任务分配需双方确认 |
| ★ Gate 2: Code Ready | Tech Lead + Peer Reviewer | Code Review 签核 |
| ★ Gate 3: Release Ready | **QA Lead** + **PM** | UAT 需业务和质量双方确认 |
| 06 Wrap-up 内联校验（合并到 Gate 3） | Tech Lead | 归档终审 |
| ★ Gate 4: Go Live | Tech Lead + **Ops** | 发布需运维确认 |

### 4. 为后续 Feature 建立 Spec-First 基线

全新项目的首个 Feature 完成后，将为后续 Feature 建立以下基线：

| 基线类型 | 内容 | 后续 Feature 如何使用 |
|---------|------|---------------------|
| **ID 命名空间** | KB 缩写已注册，FR-KB-001~008 已分配 | 后续 Feature 的 FR ID 从 FR-KB-009 起续编 |
| **API 契约基线** | 8 个 API 契约已锁定 | 后续迭代（Mode I）修改 API 需走 RFC 变更流程 |
| **数据模型基线** | 5 张表结构已定义 | 后续迭代修改表结构需 Impact Analysis |
| **度量基线** | 9 项覆盖率/合规率首次采集 | 后续 Feature 可对比度量趋势 |
| **流程模板** | constitution.md + config.yaml 已定义 | 后续 Feature 直接复用，无需重新初始化 |
| **团队协作模式** | 跨团队签核流程已验证 | 后续 Feature 沿用相同签核机制 |

### 5. Size L 全流程耗时参考

基于本场景的完整 8 阶段流程，Size L 的典型耗时参考：

| 阶段 | 预估耗时 | 关键瓶颈 |
|------|---------|---------|
| 00 Init | 0.5d | constitution.md 编写 |

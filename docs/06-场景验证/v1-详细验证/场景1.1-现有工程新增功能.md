## 场景概述

### 业务背景

某电商项目已运行一年，拥有用户服务（User）、商品服务（Product）、订单服务（Order）等核心模块，代码仓库成熟、CI/CD 流水线完备。但该项目**从未使用过 Spec-First 工具链**——没有 `.spec-first/` 目录、没有 `specs/` 目录、没有任何 stage-state.json、spec.md、design.md、traceability-matrix 等 Spec-First 产物。

现在，团队决定引入 Spec-First 来管理一个新增功能：**用户收货地址管理**。这是该项目首次使用 Spec-First，同时也是一个全新功能（项目中不存在地址管理的任何历史代码）。

| 维度 | 值 | 说明 |
|------|-----|------|
| 涉及模块 | 3 个 | 用户服务（User，扩展）、地址服务（Address，新增）、订单服务（Order，改造） |
| AC 数量 | ~8 个 | 地址 CRUD、默认地址设置、订单关联地址等 |
| 新增 API | 3 个 | 地址列表、地址创建/编辑、地址删除 |
| 数据模型 | 字段级变更 | 新增 `address` 表 + `order` 表增加 `address_id` 字段 |
| 涉及端 | H5 + Backend | 前端地址管理页面 + 后端 API |
| 跨团队 | 无 | 同一团队内完成 |

### 为什么是 Mode N 而非 Mode I

**关键判定**：Mode 的选择取决于 Spec-First 体系内是否存在历史产物，而非项目本身是否有历史代码。

- Mode I（Iteration）要求：项目中已有 Spec-First 管理的历史 Feature（如 `specs/001-xxx/spec.md`），本次是在已有规范基础上迭代
- Mode N（New）适用：项目首次引入 Spec-First，不存在任何历史 Spec-First 产物，即使项目代码已运行多年

本场景中，电商项目虽然代码成熟，但 Spec-First 是全新引入的，`specs/` 目录下空无一物。因此**必须使用 Mode N**。

### Mode/Size 判定

```
Mode = N（New）— 首次引入 Spec-First，无历史 Spec-First 产物

Size 判定（5 维度取最高）：
  modules      = 3  -> M（3-5 模块）
  ac_count     = 8  -> M（6-15 AC）
  api_changes  = 3  -> M（2-5 API）
  data_model   = 1  -> M（字段级变更）
  cross_team   = 0  -> S（无跨团队）
  -----------------
  最终 Size    = M（取最高维度）
```

### FEAT 缩写

- **FEAT 缩写**: `ADDR`
- **Feature ID**: `001-user-address`（注意：这是 Spec-First 管理的第一个 Feature，编号从 001 开始）
- **全名**: 用户收货地址管理

### 首次引入的特殊注意事项

1. **不影响已有代码** — `spec-first init` 仅创建 `.spec-first/` 和 `specs/` 目录，不修改任何已有项目文件
2. **Git Hook 共存** — 如果项目已有 Git Hook（如 husky），需手动合并 Hook 脚本，或使用 `--skip-hooks` 跳过自动安装后手动集成
3. **`.gitignore` 更新** — 建议将 `.spec-first/runtime/` 加入 `.gitignore`（运行时数据不入库），但 `.spec-first/config.yaml` 和 `specs/` 必须入库
4. **团队培训** — 首次引入建议先用一个中等规模的 Feature 走完全流程，积累经验后再推广

## Stage 01: Specify（需求规格）

### 阶段目标

将产品 PRD 转化为结构化的 `spec.md`，分配 FR/NFR ID，确保需求无歧义、可追踪。

### 操作步骤

**步骤 1：确认当前阶段**

```bash
spec-gate status 001-user-address
```

预期输出：
```
Feature:  001-user-address (ADDR)
Stage:    01_specify (active)
Mode:     N | Size: M | Platforms: H5, Backend
Gate attempts: 0
```

**步骤 2：生成 FR/NFR ID**

```bash
# 逐个生成功能需求 ID
spec-id next FR ADDR
# -> FR-ADDR-001

spec-id next FR ADDR
# -> FR-ADDR-002

# ... 依次生成至 FR-ADDR-008

# 生成非功能需求 ID
spec-id next NFR ADDR --dim PERF
# -> NFR-PERF-001

spec-id next NFR ADDR --dim SEC
# -> NFR-SEC-001
```

**步骤 3：编写 spec.md**

在 `specs/001-user-address/` 下创建 `spec.md`，将产品 PRD 转化为结构化规格：

```markdown
<!-- specs/001-user-address/spec.md -->
# 需求规格 — 用户收货地址管理 (ADDR)

> Feature: 001-user-address | Mode: N | Size: M

## 1. 功能需求（FR）

### FR-ADDR-001 地址列表查询

- **描述**: 用户可查看自己的所有收货地址列表
- **AC**:
  1. 已登录用户访问地址管理页面，展示该用户所有收货地址
  2. 列表按「默认地址优先 + 更新时间倒序」排列
  3. 每条地址展示：收件人、手机号、省市区、详细地址、是否默认
- **优先级**: P0
- **涉及端**: H5, Backend

### FR-ADDR-002 新增收货地址

- **描述**: 用户可新增收货地址，最多 20 条
- **AC**:
  1. 用户填写收件人、手机号、省市区（级联选择）、详细地址
  2. 手机号格式校验（11 位中国大陆手机号）
  3. 地址数量达到 20 条上限时，提示「地址数量已达上限，请删除后再添加」
  4. 新增第一条地址时自动设为默认地址
- **优先级**: P0
- **涉及端**: H5, Backend

### FR-ADDR-003 编辑收货地址

- **描述**: 用户可编辑已有收货地址
- **AC**:
  1. 编辑页面回填当前地址信息
  2. 校验规则与新增一致
- **优先级**: P0
- **涉及端**: H5, Backend

### FR-ADDR-004 删除收货地址

- **描述**: 用户可删除非默认收货地址
- **AC**:
  1. 删除前二次确认
  2. 默认地址不可直接删除，需先取消默认
- **优先级**: P1
- **涉及端**: H5, Backend

### FR-ADDR-005 设置默认地址

- **描述**: 用户可将任一地址设为默认收货地址
- **AC**:
  1. 设置新默认地址时，原默认地址自动取消
  2. 同一时刻仅有一个默认地址
- **优先级**: P0
- **涉及端**: H5, Backend

### FR-ADDR-006 订单关联地址

- **描述**: 创建订单时自动关联用户默认收货地址
- **AC**:
  1. 下单时默认选中默认地址
```

# Spec-First 使用手册

> **版本**: v3.1 | **日期**: 2026-02-24
> **适用对象**: 工具维护者、Tech Lead、开发/测试人员
> **状态**: ✅ 全链路可用（17 命令组 44 子命令 + 19 Skill）

---

## 1. 目的与范围

本文档回答两个问题：

1. 当前这套 Spec-First 工具如何提供给其他人员使用。
2. 其他人员拿到后，如何按阶段流程实际执行。

当前交付形态是 **CLI 工具（流程状态机 + 17 命令组） + 19 个 Skill（6 阶段执行模型） + 3 个协同命令（/plan /verify /orchestrate）**。

### 1.1 产出物命名规则

| 领域 | 统一命名 |
|---|---|
| 需求文档 | `spec.md` |
| 设计文档 | `design.md` |
| 任务规划 | `task_plan.md` |
| 技术调研 | `research.md` |
| 接口契约 | `contracts/*.yaml`（OpenAPI 3.0） |
| 追踪矩阵 | `traceability-matrix.md` |
| 回顾总结 | `retro.md` |
| 进度记录 | `stage-state.json` |
| 发现记录 | `findings.md` |

## 2. 对外提供方式

### 2.1 方式 A：源码模式（推荐当前阶段）

适用：内部试点、小团队、迭代频繁。

```bash
git clone <repo-url>
cd spec-first
pnpm install
pnpm build
```

使用方式（二选一）：

```bash
# A1: 直接运行构建产物
node dist/cli/index.js --help

# A2: 全局链接
pnpm link --global
spec-first --help
```

### 2.2 方式 B：离线包分发（tarball）

适用：多人安装但暂不搭内网 npm。

```bash
pnpm build && npm pack
# 生成: spec-first-0.1.0.tgz
```

接收方：

```bash
npm i -g ./spec-first-0.1.0.tgz
spec-first --help
```

### 2.3 方式 C：内网 npm 仓库发布（长期推荐）

适用：组织级推广、统一升级。

1. CI 构建并产出 npm 包。
2. 发布到内网 registry（Verdaccio/Nexus/Artifactory）。
3. 团队统一安装 `npm i -g spec-first`。
4. 通过版本策略控制升级窗口。

### 2.4 方式 D：AI Agent 配套资源

适用：Claude Code / Codex CLI 场景。

仓库内置位置：

| 资源 | 路径 | 状态 |
|------|------|------|
| 协同命令 | `.claude/commands/plan.md`、`verify.md`、`orchestrate.md` | ✅ |
| 19 个 Skill | `skills/spec-first/01-init/` ~ `19-feature-current/SKILL.md` | ✅ |
| 共享上下文 | `skills/spec-first/AGENTS.md` | ✅ |
| 会话 Hook | `.claude/hooks/hooks.json`、`session-start.js`、`session-end.js` | ✅ |
| Legacy Skill | 历史遗留 Skill 已移除（不参与当前路由） | ✅ |

职责分工：

- **Skill（驱动层）**：编排流程、交互式引导、阶段产物生成（6 阶段执行模型）。
- **CLI（能力层）**：状态机推进、Gate/Matrix/Metrics 的确定性校验。
- **人工（决策层）**：阶段产物内容质量与最终签核。

  pilot_mode: true  # 软门禁放行（试点期建议启用）

  token_budget: 16000  # 8000-64000

catchup:
  trigger: prompt  # auto | prompt | off

    w1: 0.10  # C1 Design Coverage
    w2: 0.10  # C2 API Coverage

    w4: 0.15  # C4 Test Coverage (FR)
    w5: 0.10  # C5 Test Coverage (AC)
    w6: 0.15  # C6 Impl Coverage
    w7: 0.10  # C7 PR Compliance
    w8: 0.10  # C8 Task Compliance
    w9: 0.10  # C9 TC Compliance

### 3.4 平台配置

`.spec-first/layer2/` 下可放置项目自定义平台 YAML；当前默认模板包含 11 个：

`admin-frontend.yaml`、`employee-frontend.yaml`、`h5.yaml`、`pc.yaml`、`app-ios.yaml`、`app-android.yaml`、`backend.yaml`、`java-backend.yaml`、`python-backend.yaml`、`go-backend.yaml`、`database.yaml`

### 3.5 stage-state 可视化面板

内置 Web 面板（参考 Serena 网页）用于实时查看 `specs/*/stage-state.json`：

- 端口策略：项目级动态端口（默认自动分配，不固定 4319）
- Claude Code：会话启动时由 `.claude/hooks/session-start.js` 自动拉起并自动打开浏览器
- Codex：可使用 `scripts/codex/codex-with-stage-viewer.sh` 启动（先拉起面板再进入 Codex）
- 当前 URL：记录在 `.spec-first/runtime/viewer-state.json`

Codex 全局自动拉起（一次性安装）：

```bash
bash scripts/codex/install-codex-autostart.sh
source ~/.zshrc
```

手动启动/调试：

```bash
# 启动并自动打开浏览器，同时输出当前 URL
spec-first viewer open --print-url

# 非阻塞模式（用于 SessionStart Hook）
spec-first viewer open --print-url --background

# 查看当前项目面板地址
cat .spec-first/runtime/viewer-state.json

# 如需固定端口（可选）
SPEC_FIRST_VIEWER_PORT=4319 spec-first viewer open --print-url
```



### 3.6 npx 零安装用法

当你不想全局安装时，可在目标项目直接用 `npx`：

```bash
cd /path/to/your-project

# 环境诊断
npx -y spec-first@latest doctor

# 初始化 Feature
npx -y spec-first@latest init --feat REPORT --mode N --size M --platforms backend,admin-frontend --title "个人报告"

# 启动面板（动态端口）
npx -y spec-first@latest viewer open --print-url
```

## 4. 流程模型（8+2 阶段）

主流程阶段（8 个）：

`00_init → 01_specify → 02_design → 03_plan → 04_implement → 05_verify → 06_wrap_up → 07_release`

终态（2 个，不可逆）：

- `08_done` — 正常完结，由 `07_release` 推进到达。
- `09_cancelled` — 任意阶段可通过 `stage cancel` 进入。

阶段顺序由状态机强约束，不允许跳过和回退。到达终态后不可再变更。

## 5. 日常使用（实操）

下面假设 Feature 为 `FSREQ-20260209-AUTH-001`。

### 5.0 CLI + Skill 协同流程（推荐）

> **核心原则**：Skill 驱动整个流程，CLI 提供底层原子能力。

```bash
# 1) 初始化 Feature（CLI）
spec-first init --feat AUTH --mode N --size M --platforms h5,java-backend

# 2) 查看当前阶段（CLI）
spec-first stage current FSREQ-20260209-AUTH-001

# 3) 生成阶段计划（Skill 驱动）
/plan FSREQ-20260209-AUTH-001 "实现用户认证需求"

# 4) 按计划执行阶段产物与代码工作（人工 / 阶段 Skill 辅助）

# 5) 质量校验（Skill 编排 + CLI 执行）
/verify FSREQ-20260209-AUTH-001 full

# 6) 通过后推进阶段（CLI）
spec-first stage advance FSREQ-20260209-AUTH-001

# 7) 循环执行 3-6，直到 08_done
```

一键编排（Skill）：

```bash
/orchestrate FSREQ-20260209-AUTH-001 "实现用户认证需求"
```

### 5.0.1 多需求处理（需求2）

- 原则：一个独立需求 = 一个 Feature（独立 featureId、独立 stage-state）。
- 需求2是独立需求：再次执行 `/spec-first:init`（或 `spec-first init ...`），使用新的 `feat`。
- 需求2是需求1的补充/变更：不要重新 init；先执行 `/spec-first:catchup` 回到原 Feature，必要时使用 `spec-first rfc create <featureId> --title "..."` 记录变更。
- 多需求并行：使用 `spec-first feature list` 查看全部 Feature，使用 `spec-first feature switch <featureId>` 切换当前上下文。

### 5.3 Skill 6 阶段执行模型

每个 Skill 遵循统一的 6 阶段流程：

```
P0_LOCATE      → 定位 Feature 工作区，校验阶段约束
P1_CONTEXT     → 加载 Context Pack，读取阶段相关产物
P2_GENERATE    → AI 推理生成内容（纯推理，不调 CLI）
P3_CONFIRM     → 展示生成内容，用户确认/修改/拒绝
P4_WRITE       → 写入目标文件，注册新 ID
P5_SIDE_EFFECT → 矩阵校验、Gate 检查、更新运行态文件
```

### 5.5 典型操作示例

#### 初始化 Feature

```bash
spec-first init --feat AUTH --mode N --size M --platforms h5,java-backend
# 输出: Initialized: FSREQ-20260214-AUTH-001

# 指定 Feature ID
spec-first init --feat AUTH --mode N --size M --platforms h5,java-backend \
  --feature-id FSREQ-20260209-AUTH-001 --title "用户认证功能"
```

#### 阶段管理

```bash
spec-first stage current FSREQ-20260209-AUTH-001
spec-first stage advance FSREQ-20260209-AUTH-001          # 需 Gate 通过或 pilot_mode
spec-first stage advance FSREQ-20260209-AUTH-001 --force  # 强制跳过（写审计记录）
spec-first stage cancel FSREQ-20260209-AUTH-001 --reason "需求变更"
```

#### ID 管理

```bash
spec-first id next FR AUTH --feature FSREQ-20260209-AUTH-001
# 输出: Generated: FR-AUTH-001

spec-first id next TC AUTH --feature FSREQ-20260209-AUTH-001 --level UT
# 输出: Generated: TC-UT-AUTH-001

spec-first id validate FR-AUTH-001
spec-first id search AUTH --feature FSREQ-20260209-AUTH-001
spec-first id list --feature FSREQ-20260209-AUTH-001 --type FR
```

ID 类型：`FR`（需求）| `DS`（设计）| `TASK`（任务）| `TC`（测试）| `RFC`（变更请求）

#### 追踪矩阵

```bash
spec-first matrix check FSREQ-20260209-AUTH-001
spec-first matrix export FSREQ-20260209-AUTH-001 --format yaml
spec-first matrix update FSREQ-20260209-AUTH-001 FR-AUTH-001 \
  --status Implemented --downstream DS-AUTH-001,TASK-AUTH-001
```

#### 质量门禁

```bash
spec-first gate check FSREQ-20260209-AUTH-001
spec-first gate conditions FSREQ-20260209-AUTH-001
spec-first gate history FSREQ-20260209-AUTH-001
```

Gate 结果：`PASS`（通过）| `PASS_WITH_WAIVER`（有豁免的通过）| `FAIL`（阻断）

#### 覆盖率指标

```bash
spec-first metrics coverage FSREQ-20260209-AUTH-001   # C1-C9 九项指标
spec-first metrics report FSREQ-20260209-AUTH-001      # 完整报告
spec-first metrics health FSREQ-20260209-AUTH-001      # 健康分
```

#### RFC 变更管理

```bash
spec-first rfc create FSREQ-20260209-AUTH-001 \
  --title "新增邮箱登录" --level Major --by "Leo"
spec-first rfc submit RFC-001 --feature FSREQ-20260209-AUTH-001
spec-first rfc transition RFC-001 approved --feature FSREQ-20260209-AUTH-001
spec-first rfc list FSREQ-20260209-AUTH-001
spec-first rfc get RFC-001 --feature FSREQ-20260209-AUTH-001
```

语义快捷方式（Dispatcher SEMANTIC_MAP）：

```bash
/spec-first:rfc approve RFC-001    # → rfc transition RFC-001 approved
/spec-first:rfc reject RFC-001     # → rfc transition RFC-001 rejected
/spec-first:defect fix AUTH 1      # → defect update AUTH 1 --status fixing
/spec-first:defect verify AUTH 1   # → defect update AUTH 1 --status verified
```

#### 缺陷管理

```bash
spec-first defect register FSREQ-20260209-AUTH-001 \
  --severity S2 --title "登录超时" --reporter "Leo" \
  --discovered-in 05_verify --linked-fr FR-AUTH-001
spec-first defect update FSREQ-20260209-AUTH-001 1 --status fixing
spec-first defect list FSREQ-20260209-AUTH-001
spec-first defect get FSREQ-20260209-AUTH-001 1
spec-first defect escape-rate FSREQ-20260209-AUTH-001
```

#### AI 协作

```bash
spec-first ai context FSREQ-20260209-AUTH-001   # Context Pack 生成
spec-first ai catchup FSREQ-20260209-AUTH-001   # 会话恢复摘要
spec-first ai stats FSREQ-20260209-AUTH-001     # AI 编码统计
```

#### Git 集成

```bash
spec-first commit -m "实现短信登录 API" --task TASK-AUTH-002
# 自动注入 traces trailer

spec-first feature list
spec-first feature current
spec-first feature switch FSREQ-20260209-AUTH-001
```

#### 上线检查

```bash
spec-first golive check FSREQ-20260209-AUTH-001
```

#### 健康检查

```bash

```

### 6.2 Dispatcher 路由规则

Skill 调用时，Dispatcher 按以下优先级路由：

1. **SEMANTIC_MAP** — 语义快捷方式（如 `rfc approve` → `rfc transition ... approved`）
2. **RUNTIME_COMMANDS** — 直接走 CLI 原子命令（id/matrix/stage/rfc/defect/metrics/gate/golive/ai/commit/feature）
3. **Skill 路由** — 加载 `NN-skillName/SKILL.md` 进入 6 阶段执行

### 8.1 统一入口

## 10. 常见问题（FAQ）

**Q1: advance 命令报 Gate 错误怎么办？**
A: 启用 `pilot_mode: true`（软门禁放行），或使用 `--force`（写审计记录到 findings.md）。

**Q2: 多人同时初始化 Feature 会冲突吗？**
A: 可能。建议单进程使用，或用 `--feature-id` 手动指定 ID。

**Q3: 矩阵文件可以手动编辑吗？**
A: 不建议。使用 `spec-first matrix update` 命令操作。

**Q4: 如何备份 Feature 数据？**
A: 直接备份 `specs/<featureId>/` 目录，所有状态都在文件中。

**Q5: 覆盖率如何计算？**
A: 基于追踪矩阵计算 C1-C9 九项指标，排除 Deferred/Cancelled/Exception 状态。

**Q6: 如何查看历史推进记录？**
A: `specs/<featureId>/stage-state.json` 的 `history` 字段，或 `gate-history.jsonl`。

**Q7: 如何升级到新版本？**
A: `npm install -g spec-first@latest`（postinstall 自动执行 `spec-first update`）。如自动注册未生效，手动执行 `spec-first update`。开发模式：`git pull && pnpm build`。

**Q8: Skill 和 CLI 命令的区别？**
A: CLI 是原子命令（确定性操作），Skill 是编排流程（AI 驱动 + 人工确认）。Skill 内部调用 CLI 完成实际操作。

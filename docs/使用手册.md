## 1. 目的与范围

本文档回答两个问题：

1. 当前这套 Spec-First 工具如何提供给其他人员使用。

当前交付形态是 **CLI 工具（流程状态机 + 命令组） + 外部 Skill 最小集合（/plan /verify /orchestrate）**。

## 2. 对外提供方式（给其他人员）

### 2.1 方式 A：源码模式（推荐当前阶段）

适用：内部试点、小团队、迭代频繁。

管理员操作：

```bash
git clone <repo-url>
cd spec-first
pnpm install
pnpm build
```

使用方式（二选一）：

```bash
# 方式 A1: 直接运行构建产物（最稳妥）
node dist/index.js --help

# 方式 A2: 全局链接命令（开发机方便）
pnpm link --global
spec-first --help
```

优点：版本可控、问题定位快。  
缺点：每台机器都要安装依赖并构建。

### 2.2 方式 B：离线包分发（tarball）

适用：多人安装但暂不搭内网 npm。

打包与分发：

```bash
pnpm build
npm pack
# 生成类似: spec-first-cli-0.1.0.tgz
```

接收方安装：

```bash
npm i -g ./spec-first-cli-0.1.0.tgz
spec-first --help
```

### 2.3 方式 C：内网 npm 仓库发布（长期推荐）

适用：组织级推广、统一升级。

流程建议：

1. CI 构建并产出 npm 包。  
2. 发布到内网 registry（如 Verdaccio/Nexus/Artifactory）。  
3. 团队统一安装 `npm i -g @spec-first/cli` 或项目内 devDependency。  
4. 通过版本策略控制升级窗口。

### 2.4 方式 D：AI Agent 配套资源（最小集合）

适用：Claude Code / Codex CLI 场景下，需要"可调用工作流能力"而非仅 CLI 命令。

仓库内置位置：

- 协同 Skill 命令模板（✅ 已可用）：`.claude/commands/plan.md`、`.claude/commands/verify.md`、`.claude/commands/orchestrate.md`
- 8 个阶段 Skill 指令（🔧 初版已编写，待联调）：`skills/spec-first/00-session-catchup/SKILL.md` ~ `skills/spec-first/07-archive/SKILL.md`
- 会话 Hook：`.claude/hooks/hooks.json`、`.claude/hooks/session-start.js`、`.claude/hooks/session-end.js`

职责分工：

- **Skill（驱动层）**：编排流程、交互式引导、阶段产物生成。
- **CLI（能力层）**：状态机推进、Gate/Matrix/Metrics 的确定性校验。
- **人工（决策层）**：阶段产物内容质量与最终签核。

校验命令：

```bash
pnpm validate:ai-assets
```

## 4. 流程模型（8+2 阶段）

主流程阶段（8 个）：

`00_init → 01_specify → 02_design → 03_plan → 04_implement → 05_verify → 06_wrap_up → 07_release`

终态（2 个，不可逆）：

- `08_done` — 正常完结，由 `07_release` 推进到达。
- `09_cancelled` — 任意阶段可通过 `stage cancel` 进入。

说明：阶段顺序由状态机强约束，不允许跳过和回退。到达终态后不可再变更。

## 5. 其他人员日常如何用（实操）

下面假设 Feature 为 `FEAT-AUTH-001`。

### 5.0 CLI + Skill 协同简要流程（推荐）

> **核心原则**：Skill 驱动整个流程，CLI 提供底层原子能力。

```bash
# 1) 初始化 Feature（CLI）
spec-first init --feat AUTH --mode N --size M --platform github --feature-id FEAT-AUTH-001

# 2) 查看当前阶段（CLI）
spec-first stage current FEAT-AUTH-001

# 3) 生成阶段计划（Skill 驱动）
/plan FEAT-AUTH-001 "实现用户认证需求"

# 4) 按计划执行阶段产物与代码工作（人工 / 阶段 Skill 辅助）

# 5) 质量校验（Skill 编排 + CLI 执行）
/verify FEAT-AUTH-001 full
# Skill 内部调用 CLI：stage current / doctor / matrix check / metrics coverage / gate check

# 6) 通过后推进阶段（CLI）
spec-first stage advance FEAT-AUTH-001

# 7) 循环执行 3-6，直到 08_done
```

一键编排（Skill）：

```bash
/orchestrate FEAT-AUTH-001 "实现用户认证需求"
```

协同 Skill 命令清单（✅ 已可用）：

| 命令 | 用途 |
|------|------|
| `/plan <featureId> "<task>"` | 生成阶段执行计划 |
| `/verify <featureId> [quick\|full]` | 质量校验（编排 CLI 命令） |
| `/orchestrate <featureId> "<task>"` | 全流程一键编排 |

8 个阶段 Skill（🔧 初版已编写，待联调）：

| `02-design-write` | 02 Design | `skills/spec-first/02-design-write/SKILL.md` |
| `03-research` | 02 Design | `skills/spec-first/03-research/SKILL.md` |
| `04-task-decompose` | 03 Plan | `skills/spec-first/04-task-decompose/SKILL.md` |
| `05-code-trace` | 04 Implement | `skills/spec-first/05-code-trace/SKILL.md` |

### 5.1 初始化（进入 00）

```bash
spec-first init --feat AUTH --mode N --size M --platform github --feature-id FEAT-AUTH-001
```

### 5.2 查看当前阶段

```bash
spec-first stage current FEAT-AUTH-001
```

### 5.3 推进阶段

⚠️ **Phase A 限制**: GateEngine 未集成，需启用 pilot_mode 或使用 --force

**方式 1: 启用 pilot_mode（推荐）**
```bash
# 编辑 .spec-first/config.yaml
gate:
  pilot_mode: true

# 推进阶段（软门禁放行）
spec-first stage advance FEAT-AUTH-001
# 输出: Advanced: 00_init → 01_specify
#       Gate: PILOT_PASS
```

**方式 2: 使用 --force（临时）**
```bash
spec-first stage advance FEAT-AUTH-001 --force
# 输出: Advanced: 00_init → 01_specify
#       Gate: FORCE_SKIPPED
# ⚠️ 会写入 findings.md 审计记录
```

**方式 3: 补充校验（严格）**
```bash
# 先手动校验
spec-first matrix check FEAT-AUTH-001
spec-first metrics coverage FEAT-AUTH-001

# 确认无问题后推进
spec-first stage advance FEAT-AUTH-001 --force
```

### 5.4 Specify / Design / Plan 典型动作

```bash
# ID 生成（--feature 运行时必填）

spec-first id validate FR-AUTH-001

# 矩阵检查

```

```bash

```

### 5.6 终止流程


### 6.2 命令使用示例

#### 初始化 Feature
```bash
# 基础用法
spec-first init --feat AUTH --mode N --size M --platforms github

# 指定 Feature ID（避免并发冲突）
spec-first init --feat AUTH --mode N --size M --platforms github \
  --feature-id FSREQ-20260211-AUTH-001 --title "用户认证功能"
```

#### 阶段管理
```bash
# 查看当前阶段
spec-first stage current FSREQ-20260211-AUTH-001

spec-first stage cancel FSREQ-20260211-AUTH-001 --reason "需求变更"
```

#### ID 管理
```bash
# 生成 FR ID
spec-first id next FR AUTH --feature FSREQ-20260211-AUTH-001
# 输出: Generated: FR-AUTH-001

# 生成 TC ID（需指定级别）
spec-first id next TC AUTH --feature FSREQ-20260211-AUTH-001 --level UT
# 输出: Generated: TC-UT-AUTH-001

# 校验 ID 格式
spec-first id validate FR-AUTH-001
# 输出: Valid FR ID: FR-AUTH-001

# 搜索 ID
spec-first id search AUTH --feature FSREQ-20260211-AUTH-001

# 列出所有 ID
spec-first id list --feature FSREQ-20260211-AUTH-001 --type FR
```

## 11. 常见问题（FAQ）

### Q1: 为什么 advance 命令需要 --force？
**A**: Phase A 的 GateEngine 尚未集成，临时使用硬编码抛出错误。建议启用 `pilot_mode: true` 或使用 `--force`。Phase B 会集成真实 Gate 校验。

### Q2: 多人同时初始化 Feature 会冲突吗？
**A**: 会。当前版本无文件锁机制，建议：
1. 单进程使用
2. 或使用 `--feature-id` 手动指定 ID
3. 本周会修复并发安全问题

### Q3: 矩阵文件可以手动编辑吗？
**A**: 不建议。当前解析逻辑依赖固定列顺序，手动编辑可能导致解析失败。建议使用 CLI 命令操作矩阵。

### Q4: 如何备份 Feature 数据？
**A**: 直接备份 `specs/<featureId>/` 目录即可，所有状态都在文件中。

### Q5: 测试覆盖率如何计算？
**A**: 基于追踪矩阵计算 9 项指标（C1-C9），排除 Deferred/Cancelled/Exception 状态。详见 `v2-05-追踪矩阵与ID引擎.md`。

### Q6: 如何查看历史推进记录？
**A**: 查看 `specs/<featureId>/stage-state.json` 的 `history` 字段，或 `gate-history.jsonl`。

### Q7: 性能如何？大文件场景支持吗？
**A**: 
- getCoverage 目标 < 50ms（未验证）
- 矩阵 >1000 行性能未测试
- 建议先在小型 Feature 试点

### Q8: 如何升级到新版本？
**A**: 
```bash
git pull
pnpm install
pnpm build
spec-first doctor  # 验证升级
```

## 2. 模版来源详解

### 2.1 模版文件位置

| 层级 | 路径 | 文件数 |
|------|------|--------|
| Backend | `packages/cli/src/templates/markdown/spec/backend/` | 6 个 |
| Frontend | `packages/cli/src/templates/markdown/spec/frontend/` | 7 个 |
| Guides | `packages/cli/src/templates/markdown/spec/guides/` | 3 个 |

### 2.2 模版加载机制

```typescript
// packages/cli/src/templates/markdown/index.ts

function readLocalTemplate(filename: string): string {
  const filePath = join(__dirname, filename);
  return readFileSync(filePath, "utf-8");
}

// 导出为常量
export const backendIndexContent: string = readLocalTemplate("spec/backend/index.md.txt");
export const frontendIndexContent: string = readLocalTemplate("spec/frontend/index.md.txt");
// ...
```

### 2.3 复制触发时机

| 时机 | 命令 | 触发条件 |
|------|------|----------|
| 项目初始化 | `npx spec-first init` | 首次设置项目 |
| 项目更新 | `npx spec-first update` | 更新到新版本 |

### 2.4 复制流程

```
init.ts
    │
    ├── createWorkflowStructure()
    │       │
    │       ├── copySpecFirstDir("scripts", ...)  # 复制 Python 脚本
    │       │
    │       └── createSpecTemplates()
    │               │
    │               ├── ensureDir(".spec-first/spec")
    │               │
    │               ├── writeGuidesDocs()         # guides 总是创建
    │               │
    │               ├── if (monorepo):
    │               │       for pkg in packages:
    │               │           writeSpecForType(.spec-first/spec/{pkg}/)
    │               │
    │               └── else (single-repo):
    │                       writeSpecForType(.spec-first/spec/)
    │
    └── writeSpecForType(specBase, projectType)
            │
            ├── if (projectType !== "frontend"):
            │       writeBackendDocs()
            │
            └── if (projectType !== "backend"):
                    writeFrontendDocs()
```

## 3. 规范使用场景

### 3.1 使用场景汇总

| 场景 | 触发点 | 读取方式 | 注入位置 |
|------|--------|----------|----------|
| Session 启动 | `session-start.py` | 直接读取 index.md | `<guidelines>` 块 |
| 子代理调用 | `inject-subagent-context.py` | 读取 JSONL 列表 | 子代理 prompt |
| 任务初始化 | `task.py init-context` | 硬编码默认路径 | JSONL 文件 |
| Slash 命令 | `/spec:start` 等 | 引导用户读取 | 命令描述 |

### 3.2 JSONL 上下文机制

```
┌─────────────────────────────────────────────────────────────────┐
│  JSONL 上下文注入机制                                            │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  任务目录: .spec-first/tasks/03-26-my-task/                     │
│  │                                                              │
│  ├── implement.jsonl    # 实现 Agent 的上下文                    │
│  │   {"file": ".spec-first/spec/backend/index.md",              │
│  │    "reason": "Backend development guide"}                    │
│  │   {"file": ".spec-first/spec/backend/error-handling.md",     │
│  │    "reason": "Error handling patterns"}                      │
│  │                                                              │
│  ├── check.jsonl        # 检查 Agent 的上下文                    │
│  │   {"file": ".spec-first/spec/backend/quality-guidelines.md", │
│  │    "reason": "Code quality standards"}                       │
│  │                                                              │
│  └── debug.jsonl        # 调试 Agent 的上下文                    │
│      {"file": ".spec-first/spec/backend/error-handling.md",     │
│       "reason": "Error patterns for debugging"}                 │
│                                                                 │
│  Hook 读取流程:                                                  │
│  1. 拦截 Task 工具调用                                          │
│  2. 解析 agent 类型 (implement/check/debug/research)            │
│  3. 读取对应的 JSONL 文件                                        │
│  4. 解析 file 字段，读取实际文件内容                              │
│  5. 注入到子代理的 prompt 中                                     │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

## 4. 规范更新机制

### 4.1 核心原则

```
┌─────────────────────────────────────────────────────────────────┐
│  规范更新原则                                                    │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  ✗ 不自动更新 - spec 文档需要手动维护                            │
│                                                                 │
│  ✓ 何时更新:                                                     │
│    - 实现了新功能，需要记录设计决策                               │
│    - 修复了 bug，需要记录常见错误                                 │
│    - 发现了新模式，需要记录最佳实践                               │
│    - 做出了架构决策，需要记录约束                                 │
│                                                                 │
│  ✓ 强制更新 (跨层/基础设施变更):                                  │
│    - API 签名变更                                                │
│    - 数据库 schema 变更                                          │
│    - 基础设施集成                                                │
│    - 跨层契约变更                                                │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 4.2 更新命令流程

```
/spec:update-spec
    │
    ├── Step 1: 识别学习内容
    │   - 学到了什么？
    │   - 为什么重要？
    │   - 应该放在哪个文件？
    │
    ├── Step 2: 分类更新类型
    │   - Design Decision
    │   - Project Convention
    │   - New Pattern
    │   - Forbidden Pattern
    │   - Common Mistake
    │
    ├── Step 3: 读取目标 spec
    │   cat .spec-first/spec/<category>/<file>.md
    │
    └── Step 4: 执行更新
        - 按模板格式添加内容
        - 保持文档结构一致
```

### 4.3 finish-work 检查清单

```markdown
### 2. Code-Spec Sync
- [ ] Does `.spec-first/spec/backend/` need updates?
- [ ] Does `.spec-first/spec/frontend/` need updates?
- [ ] Does `.spec-first/spec/guides/` need updates?

### 2.5. Code-Spec Hard Block (Infra/Cross-Layer)
If this change touches infra or cross-layer contracts, this is blocking
```

## 7. 总结

### 7.1 四阶段生命周期

| 阶段 | 时机 | 动作 | 触发者 |
|------|------|------|--------|
| 定义 | npm 包开发 | 编写模版文件 | 包维护者 |
| 分发 | 项目初始化 | 复制模版到用户项目 | `spec-first init` |
| 使用 | 开发会话 | 注入 spec 到 AI 上下文 | Hooks/Scripts |
| 更新 | 任务完成 | 手动更新 spec 文档 | 用户/命令 |

### 7.2 核心设计原则

1. **模版是静态的** - 从 npm 包复制，不在运行时修改
2. **用户 spec 是可编辑的** - 用户可以根据项目特点定制
3. **注入是自动的** - Hooks 自动读取并注入相关 spec
4. **更新是手动的** - 需要用户主动触发更新命令
5. **上下文是 JSONL 配置的** - 通过 JSONL 文件声明需要注入的内容

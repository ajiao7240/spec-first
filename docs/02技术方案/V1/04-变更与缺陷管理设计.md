## 1. 模块职责与边界

### 1.1 核心职责

| 职责 | 说明 |
|------|------|
| **RFC 生命周期管理** | 管理变更请求（RFC）从提交到关闭的完整生命周期，包括分级、审批、执行、验证 |
| **Impact Analysis** | 基于 M2 追踪链自动定位变更影响范围，输出受影响产物清单和修改建议 |
| **变更分级与审批** | 根据影响范围自动判定变更级别（Minor/Major/Critical），驱动对应审批流 |
| **缺陷追踪集成** | 与外部 Bug Tracker（Jira）集成，实现缺陷登记、溯源、修复验证的闭环 |
| **变更历史记录** | 记录所有变更事件（RFC + 缺陷），支撑变更频率、返工率等度量指标 |

### 1.2 模块边界

### 1.4 专家会审记录

> **[Arch]**: M4 是「变更中枢」——所有对已完成阶段产出物的修改都必须经过 M4。M4 不直接修改产出物（那是开发者的事），而是管理变更的「审批→执行→验证」闭环。Impact Analysis 依赖 M2 的追踪链数据，M4 本身不维护追踪关系。
> **[DevEff]**: 变更管理的核心矛盾是「流程严谨性 vs 开发敏捷性」。Minor 变更（如修正拼写、补充 AC 细节）不应走完整 RFC 流程，否则开发者会绕过变更管理直接改文件。建议 Minor 变更走快速通道——自动审批 + 事后记录。
> **[QA]**: 缺陷管理与变更管理的边界需要明确——缺陷修复是否需要走 RFC？建议：已发布功能的缺陷修复走 RFC（因为涉及已完成阶段的产出物变更），开发阶段的缺陷直接修复（当前阶段内的修正不算变更）。
> **[决议]**: M4 定位为「变更中枢」，不直接修改产出物。Minor 变更走快速通道（自动审批）。缺陷修复的 RFC 要求取决于缺陷发现阶段——当前阶段内直接修复，跨阶段回溯修复走 RFC。

## 2. RFC 变更管理

### 2.1 RFC 生命周期状态机

```
RFC 生命周期：

  Draft ──▶ Submitted ──▶ Analyzing ──▶ Approved ──▶ Executing ──▶ Verifying ──▶ Closed
              │              │              │                          │
              │              │              └──▶ Rejected ──▶ Closed   │
              │              │                                         │
              └──▶ Withdrawn │              ┌──────────────────────────┘
                   (撤回)    │              ▼
                             │         Verify Failed ──(修正后)──▶ Verifying
                             │
                             └──▶ Needs Info ──(补充后)──▶ Analyzing
```

**状态说明**：

| 状态 | 说明 | 允许操作 |
|------|------|---------|
| `Draft` | RFC 草稿，尚未正式提交 | 编辑、提交、撤回 |
| `Submitted` | 已提交，等待 Impact Analysis | 撤回 |
| `Analyzing` | Impact Analysis 进行中 | 补充信息 |
| `Needs Info` | 信息不足，需补充后重新分析 | 补充信息、撤回 |
| `Approved` | 审批通过，等待执行 | 开始执行 |
| `Rejected` | 审批拒绝 | 关闭、重新提交（新 RFC） |
| `Executing` | 变更执行中（修改产出物） | 完成执行 |
| `Verifying` | 执行完成，SCA 重新校验中 | — |
| `Verify Failed` | SCA 校验未通过，需修正 | 修正后重新验证 |
| `Closed` | RFC 已关闭（正常完成或拒绝） | — |
| `Withdrawn` | 提交者主动撤回 | — |

### 2.3 变更分级标准

#### 自动分级算法

```
function classifyRFCLevel(impactResult):
  affectedArtifacts = impactResult.affectedArtifacts
  affectedGates = impactResult.affectedGates
  involvesConstitution = impactResult.involvesConstitution
  involvesArchitecture = impactResult.involvesArchitecture

  // Critical: 涉及 Constitution 或架构变更
  if involvesConstitution or involvesArchitecture:
    return "Critical"

  // Major: 影响 3+ 产物，或需重新触发已通过 Gate
  if affectedArtifacts.length >= 3 or affectedGates.length > 0:
    return "Major"

  // Minor: 影响 ≤2 个产物，不触发 Gate 重评估
  return "Minor"
```

#### 分级详细标准

| 级别 | 影响范围 | Gate 影响 | 审批要求 | 典型场景 |
|------|---------|----------|---------|---------|
| **Minor** | ≤2 个产物 | 不需重新触发已通过 Gate | Tech Lead 审批 | AC 细节修正、文案调整、补充说明 |
| **Major** | 3-5 个产物 | 需重新触发已通过 Gate | Tech Lead + PM 审批 | FR 修改、新增 NFR、API 变更 |
| **Critical** | 不限 | 全量 Gate 重评估 | Tech Lead + PM + Architect 审批 | Constitution 变更、架构重构、安全策略调整 |

#### 分级辅助判定维度

| 维度 | Minor | Major | Critical |
|------|-------|-------|----------|
| 受影响 FR/NFR 数量 | 0-1 | 2-5 | 6+ |
| 追踪链深度 | 1 层 | 2-3 层 | 4+ 层 |
| 跨 Feature 影响 | 无 | 可能 | 大概率 |
| 已通过 Gate 需重评估 | 否 | 是 | 全量重评估 |
| 预估修改工作量 | < 0.5 人天 | 0.5-3 人天 | > 3 人天 |

### 2.4 Minor 快速通道

Minor 变更不需要提交正式 RFC 文档，走简化流程：

```
Minor 快速通道流程：

  开发者发起变更请求
      │
      ▼
  系统自动执行 Impact Analysis
      │
      ├── 影响范围 ≤2 个产物 且 不触发 Gate 重评估
      │       │
      │       ▼
      │   Tech Lead 在线审批（Approve/Reject）
      │       │
      │       ├── Approved ──▶ 开发者执行变更
      │       │                    │
      │       │                    ▼
      │       │               增量 SCA 校验
      │       │                    │
      │       │                    ▼
      │       │               自动记录到 change-log.jsonl
      │       │
      │       └── Rejected ──▶ 升级为 Major，走正式 RFC 流程
      │
      └── 影响范围超出 Minor 标准
              │
              ▼
          自动升级为 Major/Critical，走正式 RFC 流程
```

**快速通道约束**：

| 约束 | 说明 |
|------|------|
| 频率限制 | 同一 Feature 同一阶段内 Minor 变更 ≤ 5 次，超限自动升级为 Major |
| 累积效应检测 | 连续 3 次 Minor 变更影响同一 FR，触发累积影响评估 |
| 审计轨迹 | 所有 Minor 变更记录到 `change-log.jsonl`，可追溯 |
| 不可降级 | 系统判定为 Major/Critical 的变更不可手动降级为 Minor |

### 2.5 专家会审记录

> **[Arch]**: RFC 生命周期状态机的关键是「Analyzing → Approved」这一步——Impact Analysis 的结果直接决定变更级别和审批路径。状态机中 `Needs Info` 状态很重要，避免信息不足时强行推进分析导致遗漏。`Verify Failed` 状态确保变更执行后必须通过 SCA 校验，不允许「改了但没验」的情况。
> **[DevEff]**: Minor 快速通道是开发者体验的关键设计。实际项目中 70%+ 的变更是 Minor 级别（AC 细节修正、文案调整），如果每次都走完整 RFC 流程，开发者会直接绕过变更管理。频率限制（≤5 次/阶段）和累积效应检测是防止滥用的安全阀。
> **[QA]**: 变更分级的自动化判定需要依赖 Impact Analysis 的准确性——如果追踪链不完整，影响范围可能被低估，导致 Major 变更被误判为 Minor。建议在分级结果中标注「置信度」，低置信度时自动升级一个级别。
> **[决议]**: RFC 状态机确认。Minor 快速通道启用，含频率限制和累积效应检测。自动分级增加置信度标注——追踪链覆盖率 < 90% 时，分级结果自动上浮一级。

## 3. Impact Analysis 引擎

### 3.1 分析原理

Impact Analysis 基于 M2 TraceEngine 提供的追踪链数据，从变更起点出发，沿正向和反向追踪链遍历所有受影响节点，输出完整的影响范围清单。

```
Impact Analysis 核心流程：

  变更起点（FR/NFR/DS/API/TASK 等）
      │
      ▼
  调用 M2.getTraceChain(id) 获取追踪链
      │
      ├── 正向遍历：FR → DS → API → TASK → TC → PR
      │   （变更向下游传播：需求变了，设计/代码/测试都要改）
      │
      └── 反向遍历：PR → TASK → FR
          （变更向上游溯源：实现偏差需要确认需求是否需要调整）
      │
      ▼
  汇总所有受影响节点 → 去重 → 按产物文件分组
      │
      ▼
  计算影响深度和广度 → 输出分级建议
```

### 3.2 影响分析算法

```javascript
/**
 * Impact Analysis 核心算法
 * 基于 BFS 遍历追踪链，收集所有受影响节点
 */
function analyzeImpact(changeIds, traceMatrix) {
  const visited = new Set();
  const queue = [...changeIds];
  const impactNodes = [];

  while (queue.length > 0) {
    const currentId = queue.shift();
    if (visited.has(currentId)) continue;
    visited.add(currentId);

    // 获取追踪链（正向 + 反向）
    const chain = traceMatrix.getTraceChain(currentId);

    // 正向遍历：变更向下游传播
    for (const link of chain.forward) {
      if (!visited.has(link.toId)) {
        queue.push(link.toId);
        impactNodes.push({
          id: link.toId,
          impactedBy: currentId,
          direction: "forward",
          relation: link.relationType,
          depth: getDepth(changeIds, link.toId, chain),
        });
      }
    }

    // 反向遍历：变更向上游溯源
    for (const link of chain.backward) {
      if (!visited.has(link.fromId)) {
        queue.push(link.fromId);
        impactNodes.push({
          id: link.fromId,
          impactedBy: currentId,
          direction: "backward",
          relation: link.relationType,
          depth: getDepth(changeIds, link.fromId, chain),
        });
      }
    }
  }

  return groupByArtifact(impactNodes);
}
```

**算法复杂度**：
- 时间：O(V + E)，V = 追踪链节点数，E = 追踪链边数
- 空间：O(V)，visited 集合 + 结果集
- 典型规模：S 规模 V < 30，M 规模 V < 150，L 规模 V < 500

### 3.3 影响范围分组与输出

Impact Analysis 结果按产物文件分组，输出结构化的影响清单：

**分组规则**：

| 分组维度 | 分组方式 | 用途 |
|---------|---------|------|
| 按产物文件 | 同一文件的受影响 ID 归为一组 | 指导开发者按文件修改 |
| 按追踪层级 | FR/NFR → DS → API → TASK → TC → PR | 展示影响传播路径 |
| 按影响方向 | 正向（下游）/ 反向（上游） | 区分「需要改」和「需要确认」 |
| 按影响深度 | 直接影响（depth=1）/ 间接影响（depth≥2） | 优先处理直接影响 |

**影响清单输出格式**（`impact-analysis.md`）：

```markdown
# Impact Analysis Report

> **RFC**: RFC-003 | **变更起点**: FR-AUTH-001, FR-AUTH-002
> **分析时间**: 2026-02-08T14:30:00Z | **置信度**: 高（追踪链覆盖率 98%）

## 影响摘要

| 指标 | 值 |
|------|-----|
| 受影响产物数 | 4 |
| 受影响 ID 数 | 12 |
| 最大影响深度 | 3 层 |
| 建议变更级别 | Major |

## 直接影响（depth = 1）

| ID | 类型 | 产物文件 | 影响方向 | 修改建议 |
|----|------|---------|---------|---------|
| DS-AUTH-001 | Design | design.md | 正向 | 更新设计方案 §2.1 |
| DS-AUTH-002 | Design | design.md | 正向 | 更新设计方案 §2.3 |
| API-AUTH-001 | API | contracts/API-AUTH-001.yaml | 正向 | 更新请求参数 |

## 间接影响（depth ≥ 2）

| ID | 类型 | 产物文件 | 影响路径 | 修改建议 |
|----|------|---------|---------|---------|
| TASK-AUTH-001 | Task | tasks.md | FR-AUTH-001→DS-AUTH-001→TASK-AUTH-001 | 更新任务描述 |
| TC-E2E-AUTH-001 | TC | tests/TC-E2E-AUTH-001.test.md | FR-AUTH-001→...→TC | 更新测试用例 |
```

### 3.4 回归范围推导

Impact Analysis 的一个重要输出是回归测试范围——哪些已有测试用例需要重新执行。

```
回归范围推导算法：

  1. 从影响清单中筛选所有受影响的 TC（Test Case）ID
  2. 区分两类回归需求：
     a. 直接回归：TC 关联的 FR/NFR 被变更 → 必须重新执行
     b. 间接回归：TC 关联的 TASK/API 被变更 → 建议重新执行
  3. 按测试级别排序：E2E > IT > UT > ST
  4. 输出回归测试清单，标注优先级

回归优先级：
  P0: TC 直接 verifies 被变更的 FR/NFR
  P1: TC verifies 的 FR/NFR 未变更，但关联的 API/TASK 被变更
  P2: TC 与变更无直接关联，但在同一模块内（防御性回归）
```

**回归范围输出**：

| 优先级 | TC ID | 测试级别 | 关联变更 | 回归原因 |
|--------|-------|---------|---------|---------|
| P0 | TC-E2E-AUTH-001 | E2E | FR-AUTH-001 变更 | 直接验证变更需求 |
| P1 | TC-IT-AUTH-003 | IT | API-AUTH-001 变更 | API 接口变更 |
| P2 | TC-UT-AUTH-005 | UT | 同模块 | 防御性回归 |

### 3.5 跨 Feature 影响检测

当变更涉及被其他 Feature 引用的 ID 时（跨 Feature 引用格式：`ID@featureId`），Impact Analysis 需要扩展分析范围：

```
跨 Feature 影响检测流程：

  1. 在当前 Feature 的影响清单中，标记所有被变更的 ID
  2. 扫描其他 Feature 的追踪矩阵，查找引用了这些 ID 的条目
  3. 对每个跨 Feature 引用，生成「外部影响通知」
  4. 通知内容：被引用 ID + 变更类型 + 建议动作
  5. 外部影响不计入当前 RFC 的影响范围，但记录在 RFC 文档中
```

**外部影响通知格式**：

| 被引用 ID | 引用方 Feature | 引用方 ID | 建议动作 |
|-----------|---------------|-----------|---------|
| FR-AUTH-001 | 002-payment | TASK-PAY-003 | 确认支付模块的认证依赖是否受影响 |
| API-AUTH-001 | 003-order | TASK-ORD-007 | 确认订单模块的 API 调用是否需要适配 |

### 3.6 专家会审记录

> **[Arch]**: Impact Analysis 的 BFS 遍历算法简洁有效。关键是追踪链的完整性——如果 M2 的追踪矩阵有遗漏（某个 FR 没有关联 TASK），Impact Analysis 就会漏报。因此 Impact Analysis 结果中的「置信度」字段至关重要，它直接反映追踪链的覆盖率。跨 Feature 影响检测是架构层面的重要扩展——变更的影响不应止步于 Feature 边界。
> **[DevEff]**: 回归范围推导的三级优先级（P0/P1/P2）对开发者很实用——P0 必须执行，P1 建议执行，P2 可选。这避免了「全量回归」的高成本。建议 CLI 输出回归清单时，附带测试执行命令（如 `npm test -- --grep "TC-E2E-AUTH-001"`），降低执行门槛。
> **[QA]**: 间接影响（depth ≥ 2）容易被忽视——开发者倾向于只修改直接影响的产物。建议 RFC 执行完成后的 SCA 校验，不仅检查直接影响产物，还要检查间接影响产物的一致性。P2 防御性回归在 S 规模可省略，M/L 规模建议执行。
> **[决议]**: BFS 遍历算法确认。置信度基于追踪链覆盖率计算。回归清单附带执行命令。SCA 校验覆盖直接 + 间接影响产物。P2 回归按 Size 裁剪（S 省略，M/L 执行）。

## 4. 缺陷管理

### 4.1 缺陷生命周期

```
缺陷生命周期状态机：

  New ──▶ Confirmed ──▶ Assigned ──▶ Fixing ──▶ Fixed ──▶ Verified ──▶ Closed
   │         │              │           │                      │
   │         │              │           └──▶ Cannot Reproduce   │
   │         │              │                    │              │
   │         │              │                    ▼              │
   │         │              │               Needs Info ────────┘
   │         │              │                                   (补充后重新验证)
   │         │              │
   │         └──▶ Duplicate (关闭，关联原始缺陷)
   │         └──▶ Not a Bug (关闭，记录原因)
   │
   └──▶ Won't Fix ──▶ Closed (记录原因 + 审批人)
   └──▶ Deferred  ──▶ (下一迭代重新进入 New)
```

**状态转换规则**：

| 当前状态 | 目标状态 | 触发条件 | 操作人 |
|---------|---------|---------|--------|
| New | Confirmed | QA Lead 确认缺陷有效 | QA Lead |
| New | Duplicate | 与已有缺陷重复 | QA Lead |
| New | Not a Bug | 确认为非缺陷（预期行为） | QA Lead + Tech Lead |
| Confirmed | Assigned | 分配修复责任人 | Tech Lead |
| Assigned | Fixing | 开发者开始修复 | Developer |
| Fixing | Fixed | 修复完成，提交 PR | Developer |
| Fixing | Cannot Reproduce | 无法复现 | Developer |
| Fixed | Verified | QA 验证修复有效 | QA Lead |
| Fixed | Needs Info | 修复不完整，需补充 | QA Lead |
| Verified | Closed | 缺陷关闭 | QA Lead |
| 任意 | Won't Fix | 决定不修复（需审批） | Tech Lead |
| 任意 | Deferred | 延迟到下一迭代 | Tech Lead + PM |

### 4.2 缺陷严重等级与 SLA

| 等级 | 定义 | 响应 SLA | 修复 SLA | 升级规则 |
|------|------|---------|---------|---------|
| **S1-Critical** | 系统不可用 / 数据丢失 / 安全漏洞 | 4 小时内 | 24 小时内 | 超 SLA 自动升级至 CTO |
| **S2-Major** | 核心功能不可用，无 workaround | 8 小时内 | 3 工作日内 | 超 SLA 自动通知 Tech Lead |
| **S3-Minor** | 功能异常但有 workaround | 1 工作日内 | 下一迭代修复 | 累积 5+ 个 S3 触发专项修复 |
| **S4-Trivial** | UI 瑕疵 / 文案错误 | 无强制 | 下一迭代修复 | 无 |

**等级判定规则**：

```
function classifyDefectSeverity(defect):
  // S1: 系统级影响
  if defect.systemDown or defect.dataLoss or defect.securityVulnerability:
    return "S1"

  // S2: 核心功能影响，无替代方案
  if defect.coreFunction and not defect.hasWorkaround:
    return "S2"

  // S3: 功能异常，有替代方案
  if defect.functionalIssue and defect.hasWorkaround:
    return "S3"

  // S4: 外观/文案问题
  return "S4"
```

**S1/S2 特殊处理**：
- S1/S2 缺陷需 Tech Lead 确认等级（防止误判）
- S1 缺陷自动触发 RFC 变更流程（UC-012），补充遗漏的 AC 或 TC
- S1/S2 修复后必须触发回归测试（P0 级别）
- S1/S2 缺陷逃逸率目标 = 0%

### 4.3 缺陷溯源机制

缺陷溯源是 Spec-First 体系的核心能力——每个缺陷都能反向追踪到需求层，定位追踪链的断裂点。

```
缺陷溯源流程：

  缺陷报告（Bug Ticket）
      │
      ▼
  通过 verifies 字段关联到 FR/AC/NFR
      │
      ├── 有关联 → 正常溯源
      │       │
      │       ▼
      │   查询追踪矩阵，定位该 FR/NFR 的完整追踪链
      │       │
      │       ▼
      │   分析断裂点：哪个环节遗漏导致缺陷逃逸？
      │       │
      │       ├── AC 遗漏 → spec.md 需补充 AC
      │       ├── TC 遗漏 → 测试用例未覆盖该 AC
      │       ├── TC 不充分 → 测试用例存在但验证不完整
      │       └── 实现偏差 → 代码未按 spec 实现
      │
      └── 无关联 → 孤儿缺陷
              │
              ▼
          标记为「追踪链外缺陷」，需人工判定：
            a. 遗漏的 FR/NFR → 补充需求 + 走 RFC 流程
            b. 非功能性问题 → 关联到 NFR 或新建 NFR
            c. 环境/配置问题 → 不计入追踪体系
```

**溯源结果分类**：

| 断裂类型 | 定义 | 修复动作 | 度量影响 |
|---------|------|---------|---------|
| AC 遗漏 | FR 的验收条件不完整 | 补充 AC → 走 RFC | 需求完整性指标下降 |
| TC 遗漏 | AC 存在但无对应 TC | 补充 TC → 更新矩阵 | Test 覆盖率(AC级)下降 |
| TC 不充分 | TC 存在但验证逻辑不完整 | 增强 TC → 重新执行 | TC 有效性指标下降 |
| 实现偏差 | 代码未按 spec 实现 | 修复代码 → CR | PR 合规率可能下降 |
| 需求遗漏 | 缺陷对应的需求未被识别 | 新增 FR/NFR → 走 RFC | 需求覆盖率下降 |

### 4.4 缺陷逃逸率

**定义**：

```
缺陷逃逸率 = 生产环境发现的缺陷数 / (测试阶段发现 + 生产环境发现) × 100%
```

**统计规则**：

| 规则 | 说明 |
|------|------|
| 统计窗口 | 发布后 30 天 |
| 目标阈值 | 总体 ≤ 2%，S1/S2 = 0% |
| 小样本替代 | 总缺陷数 < 10 时，改用绝对值：生产缺陷 ≤ 1 个 |
| 统计口径 | 仅统计功能缺陷（S1-S3），S4 不计入 |
| 排除项 | 环境/配置问题、第三方依赖缺陷不计入 |

**逃逸率分析维度**：

| 维度 | 分析内容 | 改进方向 |
|------|---------|---------|
| 按 FR 分布 | 哪些 FR 的缺陷逃逸最多 | 加强该 FR 的 AC 和 TC |
| 按断裂类型 | AC 遗漏 vs TC 遗漏 vs 实现偏差 | 针对性改进流程 |
| 按测试级别 | UT/IT/E2E/ST 各级别的拦截率 | 优化测试金字塔 |
| 按模块分布 | 哪些模块逃逸率最高 | 加强该模块的质量投入 |

### 4.5 Jira 集成

M4 通过 M7 ToolIntegration 与 Jira Bug Tracker 集成，实现缺陷数据的双向同步。

**同步方向**：

| 方向 | 触发时机 | 同步内容 |
|------|---------|---------|
| Spec-First → Jira | 缺陷登记时 | 创建 Jira Issue，填充 summary/description/severity/labels |
| Spec-First → Jira | 状态变更时 | 同步 Jira Issue 状态（Confirmed→In Progress→Done 等） |
| Jira → Spec-First | Jira Webhook | 外部状态变更回写（如 PM 在 Jira 中标记 Won't Fix） |
| Jira → Spec-First | 定时轮询（备用） | 每 5 分钟轮询 Jira API，同步最新状态（Webhook 不可用时） |

**Jira 字段映射**：

| Spec-First 字段 | Jira 字段 | 说明 |
|-----------------|-----------|------|
| defectId | Issue Key | 如 `PROJ-123` |
| severity | Priority | S1→Highest, S2→High, S3→Medium, S4→Low |
| status | Status | 按 Jira Workflow 映射 |
| assignee | Assignee | 修复责任人 |
| relatedFR | Labels + Custom Field | `FR-AUTH-001` 写入 Labels 和自定义字段 |
| featureId | Epic Link | 关联到 Feature 对应的 Epic |
| description | Description | 缺陷描述 + 复现步骤 |
| traceSource | Custom Field `Trace Source` | 溯源结果（AC 遗漏/TC 遗漏等） |

**集成约束**：

| 约束 | 说明 |
|------|------|
| Jira 可选 | Jira 集成为可选功能，未配置时缺陷仅在本地 JSONL 中记录 |
| 幂等同步 | 同一缺陷的重复同步请求不会创建重复 Issue |
| 离线容错 | Jira 不可达时，同步请求入队，恢复后自动重试（最多 3 次） |
| 数据主权 | Spec-First 为数据主源，Jira 为展示/协作渠道 |

### 4.6 专家会审记录

> **[Arch]**: 缺陷生命周期状态机覆盖了所有合理的状态转换路径，包括 `Cannot Reproduce`、`Duplicate`、`Not a Bug` 等非正常路径。关键设计是「Deferred 重入」——延迟的缺陷在下一迭代重新进入 New 状态，而不是直接进入 Confirmed，确保每次迭代都重新评估缺陷的有效性和优先级。Jira 集成采用「数据主权在 Spec-First」的策略是正确的——避免 Jira 成为数据孤岛。
> **[DevEff]**: 缺陷溯源机制是 Spec-First 体系的差异化能力——传统 Bug Tracker 只记录「是什么」和「怎么修」，Spec-First 还能回答「为什么逃逸」。溯源结果的 5 种断裂类型（AC 遗漏/TC 遗漏/TC 不充分/实现偏差/需求遗漏）直接指向改进方向，避免「修了 Bug 但不改流程」的恶性循环。Jira 离线容错（入队 + 重试）对网络不稳定的开发环境很重要。
> **[QA]**: S1/S2 缺陷自动触发 RFC 是质量保障的关键闭环——高危缺陷不仅要修代码，还要补充遗漏的 AC/TC，防止同类缺陷再次逃逸。缺陷逃逸率的「小样本替代」规则（总缺陷 < 10 时用绝对值）很务实——小样本下百分比指标波动太大，没有统计意义。
> **[决议]**: 缺陷生命周期状态机确认。溯源机制的 5 种断裂类型纳入度量体系。S1/S2 自动触发 RFC。Jira 集成为可选功能，离线容错入队重试。逃逸率小样本替代规则确认。

## 5. 变更审批流

### 5.1 审批矩阵

| 变更级别 | 审批人 | 审批方式 | 超时策略 | 审批记录位置 |
|---------|--------|---------|---------|------------|
| **Minor** | Tech Lead | 在线审批（CLI 确认） | 24 小时无响应自动提醒 | `change-log.jsonl` |
| **Major** | Tech Lead + PM | 异步审批（RFC 文档签核） | 48 小时无响应升级至上级 | RFC 文档 §4 审批记录 |
| **Critical** | Tech Lead + PM + Architect | 会议审批（同步讨论） | 72 小时内必须完成审批 | RFC 文档 §4 审批记录 + 会议纪要 |

### 5.2 审批流程详述

#### Major 审批流程

```
RFC 提交（状态: Submitted）
    │
    ▼
Impact Analysis 完成（状态: Analyzing → Approved/Rejected）
    │
    ├── 自动分级确认为 Major
    │
    ▼
Tech Lead 审批
    │
    ├── Approved ──▶ PM 审批
    │                   │
    │                   ├── Approved ──▶ 状态: Approved → 可执行
    │                   │
    │                   └── Rejected ──▶ 状态: Rejected
    │                                    附带拒绝理由
    │
    └── Rejected ──▶ 状态: Rejected
                      附带拒绝理由 + 替代建议
```

#### Critical 审批流程

```
RFC 提交（状态: Submitted）
    │
    ▼
Impact Analysis 完成 + 自动分级确认为 Critical
    │
    ▼
发起审批会议邀请（Tech Lead + PM + Architect）
    │
    ▼
同步审批会议
    │
    ├── 全票 Approved ──▶ 状态: Approved → 可执行
    │
    ├── 多数 Approved ──▶ 条件性批准（附带约束条件）
    │                      状态: Approved（with conditions）
    │
    └── 任一 Rejected ──▶ 状态: Rejected
                           附带拒绝理由 + 改进建议
```

### 5.3 审批记录格式

**RFC 文档内嵌审批记录**（§4 审批记录）：

```markdown
## 4. 审批记录

| 角色 | 审批人 | 结果 | 日期 | 签名 | 备注 |
|------|--------|------|------|------|------|
| Tech Lead | Leo | ✅ Approved | 2026-02-08 | `[SIGNED:Leo:2026-02-08T14:30:00Z]` | — |
| PM | Alice | ✅ Approved | 2026-02-09 | `[SIGNED:Alice:2026-02-09T10:00:00Z]` | 建议优先处理 FR-AUTH-001 |
| Architect | — | ⏳ Pending | — | — | — |
```

**签名格式**：`[SIGNED:<name>:<ISO8601>]`

签名校验规则：
- M3 GateEngine 在 Gate 评估时解析签名标记，确认审批完整性
- 签名标记不可伪造——通过 Git commit 历史验证签名时间与提交时间一致
- 缺少必要签名的 RFC 不可进入 `Executing` 状态

### 5.4 变更执行与验证

RFC 审批通过后进入执行阶段，执行完成后必须经过验证才能关闭。

```
变更执行与验证流程：

  RFC 状态: Approved
      │
      ▼
  开发者按影响清单逐项修改产出物
      │
      ├── 更新 spec.md（FR/NFR/AC 变更）
      ├── 更新 design.md（DS 变更）
      ├── 更新 contracts/*.yaml（API 变更）
      ├── 更新 tasks.md（TASK 变更）
      ├── 更新 tests/*.test.md（TC 变更）
      └── 同步更新追踪矩阵
      │
      ▼
  RFC 状态: Executing → Verifying
      │
      ▼
  自动触发验证：
      │
      ├── 1. SCA 重新校验（增量或全量，取决于变更级别）
      ├── 2. 覆盖率重算（确认变更未破坏追踪完整性）
      ├── 3. 影响清单核对（所有受影响产物是否已更新）
      └── 4. Gate 重评估（Major/Critical 需重新触发已通过 Gate）
      │
      ├── 全部通过 ──▶ RFC 状态: Closed（成功）
      │
      └── 未通过 ──▶ RFC 状态: Verify Failed
                      输出未通过项清单 → 修正后重新验证
```

### 5.5 专家会审记录

> **[Arch]**: 审批矩阵的三级设计（Minor/Major/Critical）与变更分级一一对应，职责清晰。Critical 的「任一 Rejected 即拒绝」规则确保了架构变更的审慎性。签名格式 `[SIGNED:name:timestamp]` 简洁且可机器解析，通过 Git 历史交叉验证防伪。变更执行后的四步验证（SCA + 覆盖率 + 影响清单核对 + Gate 重评估）形成完整闭环。
> **[DevEff]**: Minor 的在线审批（CLI 确认）是关键体验——开发者不需要离开终端就能完成审批。Major 的异步审批适合跨时区团队。超时策略（24h/48h/72h）避免审批流程成为瓶颈。建议 CLI 提供 `spec-rfc approve <rfc-id>` 命令，一键完成审批签名。
> **[QA]**: 变更执行后的验证步骤中，「影响清单核对」是最容易被忽视的——开发者可能只修改了直接影响的产物，遗漏间接影响。建议验证步骤自动对比 Impact Analysis 输出的影响清单与实际修改的文件列表，差异项自动标记为「未处理」。
> **[决议]**: 三级审批矩阵确认。签名格式采用 `[SIGNED:name:timestamp]`。CLI 提供 `spec-rfc approve` 命令。验证步骤增加影响清单与实际修改的自动对比。

## 6. 核心数据结构

### 6.1 RFC 数据模型

```typescript
/** RFC 变更请求 */
interface RFC {
  rfcId: string;                     // 如 "RFC-003"
  title: string;                     // 变更标题
  featureId: string;                 // 关联 Feature
  status: RFCStatus;
  level: RFCLevel;                   // 自动分级结果
  levelConfidence: "high" | "medium" | "low";  // 分级置信度
  submittedBy: string;               // 提交人
  submittedAt: string;               // ISO 8601
  closedAt?: string;

  // 变更内容
  motivation: string;                // 变更动机
  changeDescription: string;         // 变更描述
  affectedIds: AffectedId[];         // 受影响的 FR/NFR 列表

  // Impact Analysis 结果
  impactResult?: ImpactResult;

  // 审批记录
  approvals: ApprovalRecord[];

  // 执行记录
  executionLog: ExecutionEntry[];

  // 验证结果
  verificationResult?: VerificationResult;
}

type RFCStatus =
  | "draft" | "submitted" | "analyzing" | "needs_info"
  | "approved" | "rejected" | "executing" | "verifying"
  | "verify_failed" | "closed" | "withdrawn";

type RFCLevel = "Minor" | "Major" | "Critical";
```

### 6.2 Impact Analysis 结果

```typescript
/** Impact Analysis 结果 */
interface ImpactResult {
  rfcId: string;
  changeIds: string[];               // 变更起点 ID 列表
  analyzedAt: string;                // ISO 8601
  confidence: "high" | "medium" | "low";  // 置信度（基于追踪链覆盖率）
  traceCoverageRate: number;         // 追踪链覆盖率（0.0 ~ 1.0）

  // 影响节点
  directImpacts: ImpactNode[];       // 直接影响（depth = 1）
  indirectImpacts: ImpactNode[];     // 间接影响（depth ≥ 2）

  // 影响统计
  summary: ImpactSummary;

  // 回归范围
  regressionScope: RegressionItem[];

  // 跨 Feature 影响
  crossFeatureImpacts: CrossFeatureImpact[];

  // 分级建议
  suggestedLevel: RFCLevel;
}

/** 影响节点 */
interface ImpactNode {
  id: string;                        // 受影响 ID
  type: IdType;                      // ID 类型
  impactedBy: string;                // 被谁影响
  direction: "forward" | "backward"; // 影响方向
  relation: "traces" | "verifies" | "implements";
  depth: number;                     // 影响深度
  sourceFile: string;                // 所在文件
```

> **[决议]**: 三个核心数据模型确认。`ChangeLogEntry` 统一三类变更事件。CLI 引导填写 `traceSource`。`VerificationResult` 增加 `attemptNumber` 字段。

> **[Arch]**: 三个接口职责清晰——RFCManager 管变更生命周期，ImpactAnalyzer 管影响分析，DefectTracker 管缺陷追踪。RFCManager 的 `hasOpenRFCs` 方法是 M1 的关键依赖——阶段转换前必须确认无未关闭的 RFC，否则变更可能在阶段间「悬空」。ImpactAnalyzer 的 `classifyLevel` 返回置信度，是分级准确性的透明化机制——调用方可根据置信度决定是否需要人工复核。

> **[QA]**: DefectTracker 的 `getOpenDefects` 方法支持 `minSeverity` 过滤——Gate 评估时只关心 S1/S2 未修复缺陷（S3/S4 不阻断 Gate）。`EscapeRateResult` 的 `smallSampleMode` 标记很重要——小样本下百分比指标不可靠，必须明确告知调用方当前使用的是替代规则。建议 `registerDefect` 在 S1/S2 缺陷登记时自动调用 RFCManager 创建 RFC 草稿。
> **[决议]**: 三接口划分确认。CLI 子命令与 RFCManager 方法一一映射。S1/S2 缺陷登记时自动创建 RFC 草稿。Gate 评估仅检查 S1/S2 未修复缺陷。

## 8. 错误处理与边界场景

### 8.1 错误分类与处理策略

| 错误类别 | 触发场景 | 处理策略 | 用户提示 |
|---------|---------|---------|---------|
| **RFC 文件格式错误** | RFC Markdown 模板字段缺失或格式不合规 | Fail-Fast，阻断提交 | 输出缺失字段列表 + 模板示例 |
| **RFC 状态转换非法** | 请求不合法的状态转换（如 Draft → Executing） | 拒绝操作 | 输出当前状态 + 允许的目标状态列表 |
| **Impact Analysis 超时** | 追踪链过深或矩阵数据量过大导致分析超时 | 返回部分结果 + 超时标记 | 提示手动缩小分析范围或检查追踪链循环 |
| **追踪链循环检测** | BFS 遍历时检测到循环引用 | 中断遍历，标记循环节点 | 输出循环路径 + 建议修复追踪矩阵 |
| **审批人角色不匹配** | Minor 变更由非 Tech Lead 审批 | 拒绝审批 | 输出变更级别 + 所需审批角色 |
| **审批超时** | 审批未在规定时间内完成 | 自动提醒 + 升级通知 | 发送提醒给审批人 + 通知上级 |
| **Jira 同步失败** | Jira API 不可达或认证失败 | 入队重试（最多 3 次） | 提示 Jira 连接状态 + 本地记录已保存 |
| **缺陷状态转换非法** | 请求不合法的缺陷状态转换 | 拒绝操作 | 输出当前状态 + 允许的目标状态列表 |
| **溯源失败** | 缺陷未关联 FR/AC/NFR，无法溯源 | 标记为「追踪链外缺陷」 | 提示手动关联 FR/AC/NFR |
| **RFC 文件冲突** | 多人同时编辑同一 RFC 文档 | Git 冲突解决 | 提示合并冲突 + 以最新审批记录为准 |

### 8.2 边界场景处理

#### 场景 1：RFC 执行期间发现新的影响范围

```
触发：开发者在执行 RFC 变更时，发现 Impact Analysis 遗漏了部分受影响产物
影响：原始影响清单不完整，可能导致验证阶段失败
处理：
  1. 开发者通过 CLI 补充受影响 ID：spec-rfc add-impact <rfc-id> <id>
  2. 系统重新执行增量 Impact Analysis，扩展影响清单
  3. 若新增影响导致变更级别上升（如 Minor → Major），自动升级审批流
  4. 升级后需补充审批（已有审批保留，追加新审批人）
  5. 记录影响清单变更历史，作为 Impact Analysis 准确性的度量数据
```

#### 场景 2：同一 Feature 多个 RFC 并行

```
触发：Feature 在同一阶段内有多个 RFC 同时处于 Executing 状态
冲突风险：多个 RFC 可能修改同一产出物的同一区域
处理：
  1. 系统检测 RFC 影响范围是否重叠（受影响 ID 交集）
  2. 无重叠 → 允许并行执行
  3. 有重叠 → 警告，建议串行执行或合并为一个 RFC
  4. 强制并行时，后提交验证的 RFC 必须基于前一个 RFC 的结果重新校验
  5. 建议同一 Feature 同时 Executing 的 RFC ≤ 2 个
```

#### 场景 3：缺陷修复引入新缺陷（回归缺陷）

```
触发：缺陷修复后的回归测试发现新缺陷
影响：修复 PR 引入了副作用
处理：
  1. 新缺陷登记时标记 relatedDefect 字段，关联原始缺陷
  2. 原始缺陷状态回退到 Fixing（而非 Closed）
  3. 回归缺陷的严重等级独立判定，不继承原始缺陷等级
  4. 若回归缺陷等级 ≥ 原始缺陷，触发 Tech Lead 复审修复方案
  5. 回归缺陷计入「修复引入率」度量指标
```

#### 场景 4：Constitution 变更的级联影响

```
触发：Constitution（项目宪法）被修改，影响所有 Feature 的流程规则
影响：所有活跃 Feature 的裁剪结果可能需要重新计算
处理：
  1. Constitution 变更自动分级为 Critical
  2. Impact Analysis 扩展到所有活跃 Feature（非仅当前 Feature）
  3. 每个受影响 Feature 生成独立的影响通知
  4. 已完成阶段不受影响（历史数据不可变）
  5. 当前活跃阶段和未来阶段使用新 Constitution 规则
  6. M1 ProcessEngine 重新执行三层规范合并，更新 mergedSpec
```

#### 场景 5：缺陷与 RFC 的交叉关联

```
触发：S1/S2 缺陷自动触发 RFC，但该 Feature 已有一个相关的 RFC 在执行中
影响：两个 RFC 可能修改同一产出物
处理：
  1. 系统检测新 RFC 与已有 RFC 的影响范围重叠度
  2. 重叠度 > 50% → 建议合并到已有 RFC（追加变更内容）
  3. 重叠度 ≤ 50% → 允许独立 RFC，但标记关联关系
  4. 合并时，已有 RFC 的审批记录保留，新增内容需补充审批
  5. 缺陷触发的 RFC 优先级高于普通 RFC（S1 缺陷 → 紧急处理）
```

### 8.3 专家会审记录

> **[Arch]**: 场景 4（Constitution 变更）是最复杂的边界场景——影响范围从单 Feature 扩展到全局。关键约束是「已完成阶段不可变」，这保证了 Constitution 变更不会破坏历史数据的一致性。M1 的三层规范合并需要在 Constitution 变更后重新执行，但仅影响当前和未来阶段。场景 1（执行期间发现新影响）的「影响清单变更历史」是 Impact Analysis 准确性的重要度量数据——如果频繁出现遗漏，说明追踪链覆盖率不足。
> **[DevEff]**: 场景 2（多 RFC 并行）在实际项目中很常见——大型 Feature 可能同时有需求变更和设计优化两个 RFC。建议 CLI 在创建新 RFC 时自动检测已有 RFC 的影响范围重叠，给出合并建议。场景 3（回归缺陷）的「修复引入率」是开发质量的重要指标——高修复引入率说明修复方案的评审不充分。
> **[QA]**: 场景 5（缺陷与 RFC 交叉）的合并策略需要谨慎——合并后的 RFC 审批记录必须完整，不能因为合并而跳过审批。建议合并操作生成审计日志，记录合并前后的 RFC 内容差异。场景 3 的回归缺陷处理中，原始缺陷状态回退到 Fixing 而非重新打开（New），这保持了缺陷生命周期的连续性。
> **[决议]**: 5 个边界场景处理方案确认。Constitution 变更影响全局但不破坏历史数据。多 RFC 并行限制 ≤ 2 个。回归缺陷关联原始缺陷。RFC 合并生成审计日志。

| D2 | Minor 快速通道 | 启用，含频率限制（≤5 次/阶段）和累积效应检测 | 70%+ 变更为 Minor，完整 RFC 流程会被绕过 | DevEff |
| D3 | 自动分级置信度 | 追踪链覆盖率 < 90% 时，分级结果自动上浮一级 | 追踪链不完整时影响范围可能被低估 | QA |
| D4 | 缺陷修复 RFC 规则 | 当前阶段内直接修复，跨阶段回溯修复走 RFC | 平衡流程严谨性与开发敏捷性 | QA |
| D5 | Impact Analysis 算法 | BFS 遍历追踪链，正向+反向，O(V+E) 复杂度 | 简洁有效，追踪链天然是 DAG 结构 | Arch |
| D6 | 回归测试优先级 | P0（直接验证）/ P1（间接关联）/ P2（防御性），P2 按 Size 裁剪 | 避免全量回归的高成本，S 规模省略 P2 | DevEff+QA |
| D7 | 缺陷溯源断裂类型 | 5 种：AC 遗漏/TC 遗漏/TC 不充分/实现偏差/需求遗漏 | 直接指向改进方向，避免「修 Bug 不改流程」 | DevEff |
| D8 | S1/S2 缺陷自动触发 RFC | 高危缺陷登记时自动创建 RFC 草稿 | 补充遗漏的 AC/TC，防止同类缺陷再次逃逸 | QA |
| D9 | 逃逸率小样本替代 | 总缺陷 < 10 时用绝对值（生产缺陷 ≤ 1 个） | 小样本下百分比指标波动太大，无统计意义 | QA |
| D10 | Jira 集成策略 | 可选功能，数据主权在 Spec-First，离线入队重试 | 避免 Jira 成为数据孤岛，网络不稳定时容错 | Arch |
| D11 | 审批签名格式 | `[SIGNED:<name>:<ISO8601>]`，Git 历史交叉验证 | 简洁可机器解析，防伪通过 Git commit 时间校验 | Arch |
| D12 | 变更执行验证 | 四步闭环：SCA + 覆盖率 + 影响清单核对 + Gate 重评估 | 确保变更执行后不破坏追踪完整性 | QA |
| D13 | 多 RFC 并行限制 | 同一 Feature 同时 Executing 的 RFC ≤ 2 个 | 降低影响范围重叠导致的冲突风险 | Arch |
| D14 | VerificationResult 增强 | 增加 `attemptNumber` 字段记录验证尝试次数 | 作为变更执行效率的度量数据 | QA |
| D15 | Gate 评估仅检查 S1/S2 | `getOpenDefects` 支持 `minSeverity` 过滤 | S3/S4 不阻断 Gate，避免低优先级缺陷阻塞流程 | QA |

### 9.3 与其他模块的设计约定

| 约定 | M4 承诺 | 对方模块承诺 |

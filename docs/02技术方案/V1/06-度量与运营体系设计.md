## 1. 模块职责与边界

### 1.1 核心职责

| 职责 | 说明 |
|------|------|
| **指标定义与注册** | 维护 12 项度量指标的定义、公式、采集方式和目标阈值 |
| **数据采集调度** | 在 Feature 完成、Gate 评估、AI 会话结束等时机触发数据采集 |
| **统计聚合计算** | 按 Feature / 开发者 / 阶段 / 时间维度聚合原始数据，生成趋势报告 |
| **运营闭环驱动** | 基于度量数据识别瓶颈阶段，输出改进建议，驱动流程优化 |
| **报告生成与分发** | 生成 Markdown 格式度量报告，支持 PR Comment、CLI 输出、归档存储 |
| **历史数据管理** | JSONL 文件的轮转、归档、清理，确保数据可查询且不膨胀 |

### 1.3 与其他模块的接口

| 调用方 | 接口 | 说明 |
|--------|------|------|
| M6 → M2 | `getCoverage(featureId, type)` | 采集覆盖率指标（C1-C9）|
| M6 → M1 | `getFeatureState(featureId)` | 获取 Feature 状态，用于流程健康指标计算 |
| M6 → M3 | `getGateHistory(featureId)` | 获取 Gate 评估历史，计算返工率和首次通过率 |
| M5 → M6 | `recordAIStats(stats)` | AI 会话结束时写入统计数据 |

| M6 → M7 | `postReport(report, target)` | 将度量报告发布到 PR Comment 或 CLI |

### 1.4 专家会审记录

> **[Arch]**: M6 是「度量消费者」——它不产生原始数据（覆盖率由 M2 计算，Gate 历史由 M3 记录，AI 统计由 M5 采集），而是聚合、分析和呈现这些数据。M6 与其他模块的关系是「只读消费」——M6 查询数据但不修改其他模块的状态。这种设计确保度量采集不会干扰核心流程。
> **[DevEff]**: 度量体系的核心矛盾是「数据完整性 vs 采集开销」。12 项指标中，覆盖率指标（C1-C5）和合规率指标（C6-C8）可以从追踪矩阵实时计算，无需额外采集；流程健康指标（返工率、首次通过率）需要从 Gate 历史聚合；AI 效能指标需要从 JSONL 文件解析。建议按数据源分类采集，避免重复查询。
> **[QA]**: 度量数据的准确性是运营闭环的基础——如果返工率计算错误，改进方向就会偏离。建议每项指标都有「校验公式」，在报告生成时自动校验数据一致性（如覆盖率分子不应大于分母）。
> **[决议]**: M6 定位为「只读消费者」，不修改其他模块状态。按数据源分类采集。每项指标增加校验公式。

## 2. 度量指标体系

### 2.1 指标分类总览

12 项度量指标按业务语义分为 4 大类别，每类 3 项：

| 类别 | 关注点 | 指标数 | 数据源 |
|------|--------|--------|--------|
| **需求覆盖** | 需求是否被充分实现和验证 | 3 | M2 追踪矩阵 |
| **实现合规** | 实现是否有需求依据，无孤儿项 | 3 | M2 追踪矩阵 |
| **流程健康** | 流程运转效率和质量 | 3 | M3 Gate 历史 + Bug 系统 |
| **AI 效能** | AI 辅助开发的产出和可靠性 | 3 | M5 AI 统计 JSONL |

### 2.2 需求覆盖指标（3 项）

| # | 指标 | 代号 | 公式 | 采集方式 | 校验阶段 | 目标 |
|---|------|------|------|---------|---------|------|
| M1 | Task 覆盖率 | `task_coverage` | Active FR∪NFR with ≥1 TASK / Active FR+NFR | M2 `getCoverage(fid, "task_coverage")` | ★ Gate 2: Code Ready | = 100% |
| M2 | Test 覆盖率 | `test_coverage_fr` | Active FR∪NFR with ≥1 TC / Active FR+NFR | M2 `getCoverage(fid, "test_coverage_fr")` | ★ Gate 3: Release Ready | = 100% |
| M3 | 实现覆盖率 | `impl_coverage` | Active FR∪NFR with ≥1 PR / Active FR+NFR | M2 `getCoverage(fid, "impl_coverage")` | ★ Gate 3: Release Ready | = 100% |

**校验公式**：

```
对于 M1/M2/M3：
  断言: covered ≤ total          // 分子不超过分母
  断言: total ≥ 0                // 分母非负
  断言: total == 0 → rate == 1.0 // 空集视为全覆盖
  断言: uncoveredIds.length == total - covered  // 遗漏清单完整
```

**语义说明**：需求覆盖指标回答「每条需求是否都有对应的实现产物」。三项指标分别在 Plan、Verify、Wrap-up 阶段校验，形成渐进式覆盖保障。

### 2.3 实现合规指标（3 项）

| # | 指标 | 代号 | 公式 | 采集方式 | 校验阶段 | 目标 |
|---|------|------|------|---------|---------|------|
| M4 | Task 合规率 | `task_compliance` | TASK with ≥1 FR/NFR ref / Total TASK | M2 `getCoverage(fid, "task_compliance")` | ★ Gate 2: Code Ready | = 100% |
| M5 | PR 合规率 | `pr_compliance` | PR with ≥1 TASK ref / Total PR | M2 `getCoverage(fid, "pr_compliance")` | ★ Gate 2: Code Ready | = 100% |
| M6 | 孤儿项率 | `orphan_rate` | 未关联 FR/NFR 的(TASK+TC+PR) / 全部(TASK+TC+PR) | M2 `getCoverage(fid, "orphan_rate")` | ★ Gate 3: Release Ready | = 0% |

**校验公式**：

```
对于 M4/M5：
  断言: compliant ≤ total         // 合规数不超过总数
  断言: rate == 1.0 → orphanIds.length == 0  // 全合规则无孤儿

对于 M6：
  断言: orphanCount ≤ total       // 孤儿数不超过总数
  断言: rate == 0.0 → orphanIds.length == 0  // 零孤儿则清单为空
  断言: orphanIds 中每个 ID 确实无 FR/NFR 关联  // 交叉验证
```

**语义说明**：实现合规指标回答「每个实现产物是否都有需求依据」。与需求覆盖互为正反向——覆盖率防遗漏，合规率防过度实现。

### 2.4 流程健康指标（3 项）

| # | 指标 | 代号 | 公式 | 采集方式 | 校验阶段 | 目标 |
|---|------|------|------|---------|---------|------|
| M7 | 返工率 | `rework_rate` | (Gate 驳回次数 + PR Changes Requested) / (Gate 总触发数 + PR 总数) | M3 `getGateHistory` + CI 统计 | Feature 完成后 | < 10% |
| M8 | Gate 首次通过率 | `gate_first_pass` | 首次通过 Gate 的阶段数 / Gate 总评估数 | M3 `getGateHistory` | Feature 完成后 | > 85% |
| M9 | 缺陷逃逸率 | `defect_escape` | 生产 Bug 数 / (生产 Bug + 测试阶段 Bug) | Bug 系统 API（Jira） | 发布后 30 天 | < 2% |

**校验公式**：

```
对于 M7：
  断言: gateRejects + prChanges ≤ gateTriggers + prTotal  // 分子不超过分母
  断言: rate ∈ [0.0, 1.0]                                  // 比率合法范围

对于 M8：
  断言: firstPassCount ≤ totalGateCount                    // 首次通过数不超过总数
  断言: firstPassCount + retryPassCount + failCount == totalGateCount  // 三态完整

对于 M9：
  断言: prodBugs ≤ prodBugs + testBugs                     // 分子不超过分母
  断言: prodBugs == 0 → rate == 0.0                        // 无生产 Bug 则逃逸率为 0
```

**语义说明**：流程健康指标回答「流程运转是否高效」。返工率和首次通过率反映前期质量（越低/越高说明前期工作越扎实），缺陷逃逸率反映整体质量防线的有效性。

### 2.5 AI 效能指标（3 项）

| # | 指标 | 代号 | 公式 | 采集方式 | 校验阶段 | 目标 |
|---|------|------|------|---------|---------|------|
| M10 | AI 代码生成量 | `ai_code_volume` | Σ(code.additions + code.deletions) per session | M5 Stop Hook → `ai-stats.jsonl` | 持续采集 | 趋势观察 |
| M11 | AI 文档生成量 | `ai_doc_volume` | Σ(docs.additions + docs.deletions) per session | M5 Stop Hook → `ai-stats.jsonl` | 持续采集 | 趋势观察 |
| M12 | 会话恢复成功率 | `session_recovery` | 成功恢复次数 / 恢复触发次数 | M5 Session Catchup 日志 | 持续采集 | > 95% |

**校验公式**：

```
对于 M10/M11：
  断言: additions ≥ 0 && deletions ≥ 0   // 行数非负
  断言: filesChanged ≥ 0                  // 文件数非负
  断言: additions + deletions > 0 → filesChanged > 0  // 有变更则有文件

对于 M12：
  断言: successCount ≤ triggerCount       // 成功数不超过触发数
  断言: triggerCount == 0 → rate == 1.0   // 无触发视为全成功
  断言: failureReasons 中每条都有对应的 sessionId  // 失败原因可追溯
```

**语义说明**：AI 效能指标回答「AI 辅助开发的产出量和可靠性」。M10/M11 无硬性阈值，用于趋势观察和团队对比；M12 是可靠性指标，低于 95% 说明 Session Catchup 机制需要优化。

### 2.6 指标间关联与交叉校验

```
指标关联图：

  需求覆盖（M1-M3）──── 正反向互补 ────实现合规（M4-M6）
       │                                    │
       │  覆盖率高 + 合规率高 → 追踪链完整    │
       │  覆盖率高 + 合规率低 → 存在孤儿实现   │
       │  覆盖率低 + 合规率高 → 存在遗漏需求   │
       │                                    │
       └──────────┬─────────────────────────┘
                  │
                  ▼
          流程健康（M7-M9）
          返工率高 → 前期覆盖/合规不足的信号
          首次通过率低 → Gate 条件与团队能力不匹配
          缺陷逃逸率高 → 测试覆盖率虽 100% 但质量不足
                  │
                  ▼
          AI 效能（M10-M12）
          代码生成量高 + 返工率高 → AI 生成质量需关注
          文档生成量高 + 覆盖率低 → AI 生成但未正确关联
          恢复成功率低 → 影响开发连续性
```

**交叉校验规则**：

| 规则 | 条件 | 预期 | 异常信号 |
|------|------|------|---------|
| R1 | M1=100% 且 M4=100% | M6=0% | 孤儿项率不为 0 说明数据不一致 |
| R2 | M8 < 50% | M7 > 15% | 首次通过率极低时返工率应偏高 |
| R3 | M3=100% | M9 观察期内 | 实现覆盖完整但逃逸率仍需观察 |
| R4 | M10 持续增长 | M7 不应同步增长 | AI 产出增加但返工也增加需预警 |

### 2.7 专家会审记录

> **[Arch]**: 12 项指标按数据源分为三组：(1) M2 实时计算组（M1-M6，覆盖率+合规率），无需额外存储，调用 M2 接口即可；(2) M3 历史聚合组（M7-M8，返工率+首次通过率），从 `gate-history.jsonl` 聚合；(3) M5 JSONL 解析组（M10-M12，AI 效能），从 `ai-stats.jsonl` 解析。M9（缺陷逃逸率）是唯一需要外部数据源（Jira）的指标，建议作为可选项，无 Jira 集成时跳过。
> **[DevEff]**: 指标间交叉校验是度量体系的「自检机制」——如果 M1=100% 且 M4=100% 但 M6≠0%，说明数据采集或计算有 Bug。建议在报告生成时自动执行交叉校验，异常时在报告中标注 ⚠️ 警告。这比事后人工发现数据不一致高效得多。
> **[QA]**: M10/M11（AI 代码/文档生成量）没有硬性阈值是正确的——AI 产出量因项目而异，强制阈值无意义。但建议增加「AI 生成占比」派生指标：AI 代码行数 / 总代码行数，用于评估 AI 辅助程度。这个派生指标同样只做趋势观察，不设阈值。
> **[决议]**: 12 项指标按数据源三组采集。M9 作为可选指标（依赖 Jira）。报告生成时自动执行交叉校验。增加「AI 生成占比」作为派生观察指标（不纳入 12 项核心指标）。

## 3. 采集引擎设计

### 3.1 采集时机与触发方式

| 触发时机 | 触发方 | 采集指标 | 触发方式 |
|---------|--------|---------|---------|
| **Gate 评估完成** | M3 GateEngine | M1-M6（覆盖率+合规率）、M7-M8（返工率+首次通过率） | M3 评估后回调 M6 |
| **AI 会话结束** | M5 Stop Hook | M10-M11（AI 代码/文档生成量） | M5 写入 JSONL 后通知 M6 |
| **Session Catchup 触发** | M5 SessionManager | M12（会话恢复成功率） | M5 恢复完成后回调 M6 |
| **Feature 完成（Wrap-up）** | M1 ProcessEngine | M1-M12 全量采集 | M1 阶段转换时触发 |
| **CI Pipeline 触发** | M7 ToolIntegration | M1-M8（覆盖率+合规率+流程健康） | CI Job 调用 M6 接口 |
| **定时采集（可选）** | Cron / CI Schedule | M9（缺陷逃逸率） | 发布后 30 天定时查询 Jira |

### 3.2 采集调度架构

```
触发源                    采集调度器                      数据源
┌──────────┐            ┌──────────────────┐           ┌──────────┐
│ M3 Gate  │──回调──▶  │                  │──查询──▶  │ M2 矩阵  │
│ 评估完成  │            │  Collection      │           └──────────┘
└──────────┘            │  Scheduler       │           ┌──────────┐
┌──────────┐            │                  │──查询──▶  │ M3 Gate  │
│ M5 Stop  │──通知──▶  │  ┌────────────┐  │           │ History  │
│ Hook     │            │  │ Task Queue │  │           └──────────┘
└──────────┘            │  │ (内存队列)  │  │           ┌──────────┐
┌──────────┐            │  └────────────┘  │──解析──▶  │ M5 JSONL │
│ M1 阶段  │──触发──▶  │                  │           │ ai-stats │
│ 转换     │            │  ┌────────────┐  │           └──────────┘
└──────────┘            │  │ Dedup Cache│  │           ┌──────────┐
┌──────────┐            │  │ (去重缓存)  │  │──API───▶  │ Jira     │
│ M7 CI    │──调用──▶  │  └────────────┘  │           │ (可选)   │
│ Pipeline │            └──────────────────┘           └──────────┘
└──────────┘                     │
                                 ▼
                        ┌──────────────────┐
                        │ metrics-data.jsonl│
                        │ (度量数据存储)     │
                        └──────────────────┘
```

### 3.3 采集去重与幂等

同一 Feature 的同一指标可能被多个触发源重复采集（如 Gate 评估和 CI Pipeline 都触发覆盖率采集）。采集调度器通过去重缓存确保幂等性：

```
去重键 = featureId + indicatorCode + collectionTimestamp(精确到分钟)

采集流程：
  1. 收到采集请求(featureId, indicators[])
  2. 对每个 indicator 生成去重键
  3. 查询 Dedup Cache：
     ├── 命中 → 跳过采集，返回缓存结果
     └── 未命中 → 执行采集，写入缓存（TTL = 5 分钟）
  4. 将采集结果追加写入 metrics-data.jsonl
```

**幂等保证**：

| 场景 | 处理方式 |
|------|---------|
| 5 分钟内重复触发 | 返回缓存结果，不重复写入 JSONL |
| 5 分钟后重复触发 | 重新采集，生成新记录（时间戳不同） |
| Gate 和 CI 同时触发 | 先到者写入，后到者命中缓存 |
| 采集中途失败 | 不写入 JSONL，不更新缓存，下次触发重新采集 |

### 3.4 专家会审记录

> **[Arch]**: 采集调度器是「被动触发」而非「主动轮询」——M6 不定时扫描数据源，而是等待其他模块的回调或通知。这降低了系统耦合度和资源消耗。唯一的例外是 M9（缺陷逃逸率），需要定时查询 Jira API，建议通过 CI Schedule Job 实现，而非 M6 内置定时器。
> **[DevEff]**: 去重缓存的 5 分钟 TTL 是合理的——Gate 评估和 CI Pipeline 的时间差通常在秒级到分钟级。但需要注意：如果开发者在 5 分钟内修复了 Gate 失败项并重新提交，第二次 Gate 评估应该重新采集（因为数据已变化）。建议去重键增加 `gateAttempt` 序号，区分同一阶段的不同 Gate 尝试。
> **[QA]**: 采集失败的「不写入」策略是正确的——宁可缺数据也不写脏数据。但需要有失败告警机制——如果某个 Feature 的度量数据连续 3 次采集失败，应该在 CLI 输出警告，提示开发者检查数据源。
> **[决议]**: 采集调度器采用被动触发模式。去重键增加 `gateAttempt` 序号。采集失败不写入 + 连续 3 次失败告警。M9 通过 CI Schedule 实现。

## 4. 聚合与计算引擎

### 4.1 聚合维度

度量数据按 4 个维度聚合，支撑不同粒度的分析需求：

| 维度 | 聚合粒度 | 用途 | 示例 |
|------|---------|------|------|
| **Feature** | 单个 Feature | Feature 级度量报告，Wrap-up 复盘 | `001-user-auth` 的覆盖率趋势 |
| **开发者** | 单个开发者 | 个人效能分析，AI 使用习惯 | Leo 的 AI 代码生成量趋势 |
| **阶段** | 单个阶段 | 阶段瓶颈识别，Gate 通过率分析 | 02 Design 阶段的平均返工率 |
| **时间** | 周 / 月 / 季度 | 团队趋势分析，运营节奏报告 | 本月所有 Feature 的平均首次通过率 |

### 4.2 聚合算法

#### 4.2.1 Feature 维度聚合

```
function aggregateByFeature(featureId):
  // 从 metrics-data.jsonl 中筛选该 Feature 的所有记录
  records = filterByFeature(metricsData, featureId)

  // 按指标分组，取每个指标的最新值
  latestByIndicator = {}
  for record in records.sortBy(timestamp, DESC):
    if record.indicator not in latestByIndicator:
      latestByIndicator[record.indicator] = record

  // 生成 Feature 级汇总
  return {
    featureId,
    snapshot: latestByIndicator,           // 最新快照
    timeline: groupByTimestamp(records),    // 时间线（用于趋势图）
    health: calcHealthScore(latestByIndicator)  // 综合健康分
  }
```

#### 4.2.2 综合健康分算法

```
function calcHealthScore(indicators):
  // 加权评分：覆盖率类权重高，AI 效能类权重低
  weights = {
    task_coverage: 15,    test_coverage_fr: 15,  impl_coverage: 15,
    task_compliance: 10,  pr_compliance: 10,     orphan_rate: 10,
    rework_rate: 8,       gate_first_pass: 8,    defect_escape: 5,
    ai_code_volume: 0,    ai_doc_volume: 0,      session_recovery: 4
  }
  // 注：ai_code_volume 和 ai_doc_volume 权重为 0（趋势观察，不参与评分）

  totalWeight = sum(weights.values())  // = 100
  score = 0
  for indicator, weight in weights:
    if weight == 0: continue
    score += weight * normalizeScore(indicators[indicator])

  return score  // 0-100 分
```

**normalizeScore 规则**：

| 指标类型 | 满分条件 | 零分条件 | 线性插值 |
|---------|---------|---------|---------|
| 覆盖率类（= 100%） | rate = 1.0 | rate < 0.5 | [0.5, 1.0] → [0, 100] |
| 合规率类（= 100%） | rate = 1.0 | rate < 0.5 | [0.5, 1.0] → [0, 100] |
| 孤儿项率（= 0%） | rate = 0.0 | rate > 0.5 | [0.0, 0.5] → [100, 0] |
| 返工率（< 10%） | rate ≤ 0.10 | rate > 0.30 | [0.10, 0.30] → [100, 0] |
| 首次通过率（> 85%） | rate ≥ 0.85 | rate < 0.50 | [0.50, 0.85] → [0, 100] |
| 缺陷逃逸率（< 2%） | rate ≤ 0.02 | rate > 0.10 | [0.02, 0.10] → [100, 0] |
| 恢复成功率（> 95%） | rate ≥ 0.95 | rate < 0.80 | [0.80, 0.95] → [0, 100] |

### 4.3 趋势计算

趋势分析基于时间维度聚合，支撑月度/季度运营报告：

```
function calcTrend(indicatorCode, timeRange, granularity):
  // granularity: "week" | "month" | "quarter"
  records = filterByTimeRange(metricsData, timeRange)
  buckets = groupByGranularity(records, granularity)

  // 每个时间桶取该指标的平均值
  series = buckets.map(bucket => ({
    period: bucket.label,           // 如 "2026-W06", "2026-02"
    avgRate: avg(bucket.records.filter(r => r.indicator == indicatorCode).map(r => r.rate)),
    sampleCount: bucket.records.filter(r => r.indicator == indicatorCode).length,
  }))

  // 计算趋势方向
  if series.length >= 2:
    delta = series.last.avgRate - series.first.avgRate
    direction = delta > 0.05 ? "improving" : delta < -0.05 ? "declining" : "stable"
  else:
    direction = "insufficient_data"

  return { indicatorCode, timeRange, granularity, series, direction }
```

**趋势方向判定**：

| 方向 | 条件 | 报告标记 | 运营动作 |
|------|------|---------|---------|
| `improving` | 末值 - 首值 > 5% | ↑ 改善 | 继续当前策略 |
| `stable` | 变化幅度 ≤ 5% | → 稳定 | 维持现状 |
| `declining` | 末值 - 首值 < -5% | ↓ 下降 | 触发改进分析 |
| `insufficient_data` | 数据点 < 2 | — 数据不足 | 等待更多数据 |

### 4.4 专家会审记录

> **[Arch]**: 综合健康分的权重分配体现了度量体系的价值观——覆盖率和合规率权重最高（共 75%），因为它们是追踪链完整性的直接度量；流程健康次之（21%）；AI 效能中只有恢复成功率参与评分（4%），代码/文档生成量不参与。这避免了「AI 产出量越大分数越高」的错误激励。
> **[DevEff]**: 趋势方向的 5% 阈值需要在实际运营中校准——如果团队覆盖率长期在 95%-100% 之间波动，5% 的变化可能是噪声而非趋势。建议增加「连续 N 期同方向变化」的二次确认机制，避免单期波动触发误报。
> **[QA]**: `normalizeScore` 的线性插值是简化模型——实际中覆盖率从 90% 提升到 95% 的难度远大于从 50% 到 55%。但作为 v1.0 的初始实现，线性插值足够。后续可根据运营数据调整为对数曲线或分段函数。
> **[决议]**: 健康分权重确认。趋势判定增加「连续 2 期同方向」二次确认。normalizeScore v1.0 用线性插值，后续根据数据调整。

## 5. 运营闭环机制

### 5.1 闭环四步模型

```
┌──────────┐     ┌──────────┐     ┌──────────┐     ┌──────────┐
│  采集    │────▶│  分析    │────▶│  改进    │────▶│  验证    │
│ Collect  │     │ Analyze  │     │ Improve  │     │ Verify   │
└──────────┘     └──────────┘     └──────────┘     └──────────┘
     ▲                                                   │
     │                                                   │
     └───────────── 下一个 Feature ──────────────────────┘

采集：每个 Feature 完成后自动汇总 12 项度量数据
分析：识别瓶颈阶段（返工率最高的阶段）+ 趋势异常
改进：调整流程裁剪深度 / Gate 检查项 / 模板内容
验证：下一个 Feature 验证改进效果，对比改进前后数据
```

### 5.2 运营节奏

| 频率 | 活动 | 参与者 | 输入 | 输出 |
|------|------|--------|------|------|
| **每个 Feature** | 度量数据自动采集 + Wrap-up 复盘 | Tech Lead + Dev | 12 项指标快照 | Feature 度量报告 |
| **每月** | 度量趋势分析 + 流程优化建议 | Tech Lead + PM | 月度聚合数据 | 月度运营报告 + 改进 Action |
| **每季度** | 流程裁剪深度校准 + Gate 标准调整 | 全团队 | 季度趋势 + 改进效果 | config.yaml 调整提案 |

**每个 Feature 复盘（自动触发）**：

```
触发时机：Feature 进入 06 Wrap-up 阶段
自动动作：
  1. 全量采集 12 项指标
  2. 生成 Feature 度量报告（Markdown 格式）
  3. 与该 Feature 的 Size 基线对比（S/M/L 各有基线）
  4. 标注异常指标（偏离基线 > 10%）
  5. 输出到 specs/NNN-feat/reports/metrics-report.md
```

**月度运营报告（手动触发 / CI Schedule）**：

```
触发方式：CLI 命令 `spec-metrics report --period monthly`
自动动作：
  1. 聚合本月所有已完成 Feature 的度量数据
  2. 按指标计算月度平均值 + 趋势方向
  3. 识别 Top 3 瓶颈（返工率最高的阶段 / 覆盖率最低的指标）
  4. 生成改进建议（基于瓶颈分析规则）
  5. 输出到 .spec-first/runtime/reports/monthly-YYYY-MM.md
```

### 5.4 专家会审记录

> **[Arch]**: 运营闭环的关键是「改进建议可执行」——不能只说「覆盖率不足」，必须指出具体调整哪个模板、哪个 Gate 条件、哪个 Hook 配置。瓶颈分析规则表本质上是一个「异常信号 → 改进动作」的映射表，可以随运营经验持续扩充。建议将规则表外置为 YAML 配置，而非硬编码。

> **[决议]**: 瓶颈分析规则表外置为 YAML 配置。月度报告自动生成 Action Item。改进建议增加根因分析步骤（条件问题 vs 质量问题）。

### 6.2 JSONL 轮转策略

`metrics-data.jsonl` 采用按月轮转 + 6 个月保留策略，与 `ai-stats.jsonl` 保持一致：

```
轮转规则：
  文件命名：metrics-data.jsonl（当月活跃文件）
  归档命名：metrics-data-YYYY-MM.jsonl.gz（历史月份压缩归档）

轮转流程（每月 1 日 00:00 触发，或首次写入时检测）：
  1. 检查当前 metrics-data.jsonl 的首条记录时间戳
  2. 若首条记录属于上月：
     a. 将文件重命名为 metrics-data-YYYY-MM.jsonl
     b. gzip 压缩为 metrics-data-YYYY-MM.jsonl.gz
     c. 创建新的空 metrics-data.jsonl
  3. 清理超过 6 个月的归档文件

保留策略：
  当月文件：metrics-data.jsonl（活跃，可读写）
  近 6 月归档：metrics-data-YYYY-MM.jsonl.gz（只读，可查询）

```

**存储容量估算**：

| 场景 | 每月 Feature 数 | 每 Feature 采集次数 | 每条记录大小 | 月度数据量 |
|------|----------------|-------------------|-------------|-----------|
| 小团队（10人） | 5-10 | ~20 | ~500B | ~100KB |
| 中团队（30人） | 15-30 | ~20 | ~500B | ~300KB |
| 大团队（50人） | 30-50 | ~20 | ~500B | ~500KB |

### 6.3 查询模式

M6 提供三种查询模式，覆盖不同使用场景：

| 模式 | 适用场景 | 数据源 | 性能 |
|------|---------|--------|------|
| **快照查询** | Gate 评估、CLI 实时查看 | `metrics-summary.json` | O(1)，直接读取 JSON |
| **历史查询** | 趋势分析、月度报告 | `metrics-data.jsonl` | O(N)，逐行扫描过滤 |
| **跨月查询** | 季度报告、年度回顾 | `metrics-data-*.jsonl.gz` | O(N×M)，解压 + 扫描 |

**快照查询流程**：

```
function querySnapshot(featureId, indicatorCode):
  summary = readJSON(".spec-first/runtime/metrics-summary.json")
  return summary.features[featureId]?.indicators[indicatorCode]
  // 快照不存在时触发一次全量采集并更新快照
```

**历史查询流程**：

```
function queryHistory(featureId, indicatorCode, timeRange):
  lines = readLines("metrics-data.jsonl")
  return lines
    .map(JSON.parse)
    .filter(r => r.featureId == featureId
              && r.indicator == indicatorCode
              && r.timestamp >= timeRange.start
              && r.timestamp <= timeRange.end)
    .sortBy("timestamp")
```

### 6.4 专家会审记录

> **[Arch]**: 三种查询模式的分层设计是合理的——快照查询覆盖 90% 的实时场景（Gate 评估、CLI 查看），历史查询覆盖趋势分析，跨月查询仅在季度/年度报告时使用。`metrics-summary.json` 作为快照缓存，在每次采集后同步更新，避免高频场景读取 JSONL。
> **[DevEff]**: JSONL 的逐行扫描在数据量小（< 6MB/年）时性能完全可接受。但如果未来团队规模扩大或指标增多，可以考虑在 JSONL 基础上建立简单的内存索引（按 featureId + indicatorCode 建 Map），避免全量扫描。这是 v2.0 的优化方向，v1.0 不需要。
> **[QA]**: 跨月查询需要解压 `.gz` 文件，建议在查询前检查归档文件完整性（gzip CRC 校验）。如果归档文件损坏，应该跳过该月数据并在报告中标注「数据缺失」，而非报错中断。
> **[决议]**: 三层查询模式确认。v1.0 用逐行扫描，v2.0 按需引入内存索引。归档文件查询前做 CRC 校验，损坏时优雅降级。

```typescript
type TrendDirection = "improving" | "stable" | "declining" | "insufficient_data";

/** 交叉校验结果 */
interface CrossValidationResult {
  ruleId: string;                  // 如 "R1", "R2"
  description: string;             // 规则描述
  passed: boolean;
  message?: string;                // 异常时的说明
}

/** 基线对比 */
interface BaselineComparison {
  baselineSource: string;          // 基线来源，如 "size_M_average"
  deviations: DeviationItem[];     // 偏离项
}

/** 偏离项 */
interface DeviationItem {
  indicator: IndicatorCode;
  baseline: number;                // 基线值
  actual: number;                  // 实际值
  deviation: number;               // 偏离百分比

}
```

### 7.5 专家会审记录

> **[Arch]**: 数据结构分三层：(1) `IndicatorDefinition` 是静态配置，定义指标的元数据；(2) `MetricsRecord` 是原始采集数据，追加写入 JSONL；(3) `MetricsSummary` 是聚合快照，覆盖写入 JSON。三层数据的生命周期不同——定义随代码版本管理，原始数据按月轮转，快照实时更新。`AnomalyItem` 嵌入快照而非独立存储，是因为异常项与快照强关联，独立存储会增加一致性维护成本。
> **[DevEff]**: `MetricsRecord` 的 `details` 字段使用 `Record<string, any>` 是有意为之——不同指标的详情结构不同（覆盖率有 uncoveredIds，AI 效能有 filesChanged），统一类型会过度约束。但建议在文档中列出每种指标的 details 结构示例，方便开发者理解。
> **[QA]**: `CrossValidationResult` 纳入报告数据结构是正确的——交叉校验结果应该在报告中可见，而非仅在日志中。`BaselineComparison` 的基线数据来源需要明确——建议在 `config.yaml` 中维护 S/M/L 三种 Size 的基线值，初始值由前 5 个 Feature 的平均值确定，后续滚动更新。
> **[决议]**: 三层数据结构确认。`details` 保持灵活类型，文档补充示例。基线值在 `config.yaml` 中维护，初始值由前 5 个 Feature 平均值确定。

### 8.6 专家会审记录

> **[Arch]**: 四个接口职责清晰——IndicatorRegistry 管「定义」，CollectionScheduler 管「采集」，AggregationEngine 管「算」，ReportGenerator 管「呈现」。`CollectionScheduler.recordAIStats` 是 M5 → M6 的唯一入口，M5 不直接写入 `metrics-data.jsonl`，而是通过 M6 统一写入，确保数据格式一致和校验公式执行。`ReportTarget` 的联合类型设计支持多种分发渠道，新增渠道只需扩展类型定义。

## 9. 错误处理与边界场景

### 9.1 错误分类与处理策略

| 错误类别 | 触发场景 | 处理策略 | 用户提示 |
|---------|---------|---------|---------|
| **数据源不可达** | M2/M3/M5 接口调用失败或超时 | 跳过该指标，标记 `source_unavailable` | 输出不可达的数据源 + 重试命令 |
| **JSONL 写入失败** | `metrics-data.jsonl` 文件锁定或磁盘满 | 重试 3 次（间隔 100ms），失败则丢弃本次记录 | 输出写入失败原因 + 手动触发命令 |
| **JSONL 解析错误** | `metrics-data.jsonl` 中存在格式错误的行 | 跳过错误行，继续解析后续行 | 输出错误行号 + 建议手动修复 |
| **快照文件损坏** | `metrics-summary.json` 格式错误 | 删除快照，下次采集时重建 | 输出损坏原因 + 自动重建提示 |
| **校验公式失败** | 指标值不满足校验规则（如分子 > 分母） | 标记该指标为 `anomaly`，写入报告但不参与健康分 | 输出违反的校验规则 + 数据源检查建议 |
| **归档文件损坏** | `.gz` 文件 CRC 校验失败 | 跳过该月数据，报告标注「数据缺失」 | 输出损坏文件名 + 从 Git 恢复建议 |
| **去重缓存溢出** | 缓存条目超过上限（默认 1000） | LRU 淘汰最旧条目 | 无需用户干预，日志记录淘汰事件 |
| **外部 API 失败** | Jira API 调用失败（M9 缺陷逃逸率） | 标记 M9 为 `na`，不影响其他指标 | 输出 API 错误 + Jira 连接检查建议 |
| **瓶颈规则配置错误** | `bottleneck-rules.yaml` 格式错误 | 跳过瓶颈分析，报告中标注「规则配置错误」 | 输出 YAML 解析错误位置 + 期望格式 |
| **基线数据不足** | Feature 数 < 5，无法建立基线 | 跳过基线对比，报告中标注「基线待建立」 | 提示需要更多 Feature 数据 |

### 9.2 边界场景处理

#### 场景 1：Feature 中途取消时的度量数据处理

```
触发：Feature 在任意阶段被 cancelFeature() 取消
影响：已采集的度量数据是否保留？是否参与团队聚合？
处理：
  1. 已采集的度量数据保留在 metrics-data.jsonl 中（不删除）
  2. metrics-summary.json 中该 Feature 标记为 cancelled
  3. 月度聚合时，cancelled Feature 不参与平均值计算
  4. 但 cancelled Feature 的返工率数据仍参与流程健康分析
     （取消本身可能是流程问题的信号）
  5. 报告中单独列出 cancelled Feature 及其取消阶段
```

#### 场景 2：多个 Feature 并发采集时的 JSONL 写入冲突

```
触发：两个 Feature 同时完成 Gate 评估，同时触发度量采集
影响：两个采集任务同时追加写入 metrics-data.jsonl
处理：
  1. JSONL 追加写入是原子操作（单行 append，OS 级文件锁）
  2. 每条记录独立一行，不存在跨行事务
  3. 两个采集任务的记录可能交错写入，但每行完整
  4. 读取时按 featureId 过滤，交错不影响查询正确性
  5. metrics-summary.json 的更新需要读-改-写，
     使用文件锁（flock）确保原子性
```

#### 场景 3：度量数据与实际状态不一致（数据漂移）

```
触发：开发者手动修改了追踪矩阵但未触发 Gate 评估，
      导致 metrics-summary.json 中的覆盖率快照与实际矩阵不一致
检测：
  1. 每次 Gate 评估时全量重算覆盖率，与快照对比
  2. 偏差 > 1% 时标记为 data_drift 异常
处理：
  1. 以全量重算结果为准，覆盖快照中的旧值
  2. 在报告中标注「数据已校正」+ 校正前后差值
  3. 记录漂移事件到日志，用于分析漂移频率
预防：
  1. 建议开发者修改矩阵后执行 `spec-metrics refresh`
  2. CI Pipeline 中强制全量重算，作为兜底
```

#### 场景 4：轮转期间的采集请求

```
触发：月初轮转正在执行（重命名 + 压缩），此时收到新的采集请求
影响：写入目标文件可能不存在（旧文件已重命名，新文件未创建）
处理：
  1. 轮转操作加文件锁，采集写入等待锁释放
  2. 轮转流程保证原子性：
     a. 创建新的空 metrics-data.jsonl
     b. 将旧文件重命名为 metrics-data-YYYY-MM.jsonl
     c. 异步压缩（不阻塞写入）
  3. 采集请求最多等待 5s，超时则写入临时文件
     下次采集时合并临时文件到主文件
```

#### 场景 5：纯人工场景下的度量体系行为

```
触发：团队未使用 AI 辅助开发（无 M5 数据源）
影响：M10/M11/M12 三项 AI 效能指标无数据
处理：
  1. AI 效能指标标记为 na（not applicable）
  2. 健康分计算时，AI 效能权重（4%）重新分配到其他类别
     调整后权重：覆盖率 78.1%，合规率 10.4%，流程健康 11.5%
  3. 报告中 AI 效能区域显示「未启用 AI 辅助」
  4. 月度报告中不包含 AI 效能趋势
  5. 不影响其他 9 项指标的正常采集和分析
```

### 9.3 专家会审记录

> **[Arch]**: M6 的错误处理核心原则是「优雅降级」——任何单点故障都不应导致整个度量体系不可用。数据源不可达时跳过该指标，JSONL 损坏时跳过错误行，归档损坏时跳过该月。这与 M5 的错误处理原则一致（见 05-AI协作编排设计 §9）。场景 2 的并发写入通过 JSONL 追加模式天然解决，`metrics-summary.json` 的读-改-写需要文件锁，建议使用 `proper-lockfile` npm 包。
> **[DevEff]**: 场景 3（数据漂移）是实际运营中最常见的问题——开发者修改了矩阵但忘记触发重算。`spec-metrics refresh` 命令是必要的，但更好的方案是在 `traceability-matrix.yaml` 的 `fs.watch` 变更时自动触发增量重算。这与 M1 的 `getCurrentStage` 缓存失效策略类似（见 01-流程引擎设计 §7.5）。
> **[QA]**: 场景 5（纯人工场景）的权重重分配算法需要在单元测试中覆盖——确保重分配后权重总和仍为 100%。场景 1（Feature 取消）的「cancelled Feature 不参与平均值但参与流程健康分析」规则需要在月度报告生成逻辑中明确实现，避免遗漏。
> **[决议]**: 错误处理遵循「优雅降级」原则。并发写入用 `proper-lockfile`。数据漂移通过 `fs.watch` + CLI 命令双重机制检测。纯人工场景权重重分配需单元测试覆盖。

| D6 | AI 生成占比 | 作为派生观察指标，不纳入 12 项核心指标 | 避免「AI 产出量越大分数越高」的错误激励 | QA |
| D7 | 采集调度模式 | 被动触发，非主动轮询 | 降低系统耦合度和资源消耗 | Arch |
| D8 | 去重键设计 | featureId + indicatorCode + timestamp(分钟) + gateAttempt | 区分同一阶段的不同 Gate 尝试 | DevEff |
| D9 | 采集失败策略 | 不写入 + 连续 3 次失败告警 | 宁可缺数据也不写脏数据 | QA |
| D10 | 健康分权重 | 覆盖率 75% + 流程健康 21% + AI 效能 4% | 追踪链完整性是核心价值 | Arch |
| D11 | 趋势确认机制 | 连续 2 期同方向变化才确认趋势 | 避免单期波动触发误报 | DevEff |
| D12 | 瓶颈规则外置 | bottleneck-rules.yaml 配置文件 | 规则可随运营经验持续扩充 | Arch |
| D13 | 改进建议增加根因分析 | 区分「条件过严」vs「质量不足」 | 避免盲目放宽 Gate 条件 | QA |
| D14 | 存储三层分离 | 定义(代码) / 原始数据(JSONL) / 快照(JSON) | 生命周期不同，管理策略不同 | Arch |
| D15 | 基线值来源 | config.yaml 维护，前 5 个 Feature 平均值初始化 | 基线需要数据积累，初期无法预设 | QA |

## 3. 合并规则

### 3.1 三类合并策略

| 字段 | 合并策略 | 说明 |
|------|---------|------|
| `gate_conditions` | AND 叠加 | 各端条件追加到同一阶段，全部满足才 PASS |
| `extra_deliverables` | 追加去重 | 合并到阶段产出物列表，去除重复项 |
| `quality_thresholds` | 取更严格值 | 多端冲突时选择约束更强的值 |

### 3.2 "更严格"的比较语义

每个阈值通过显式 `direction` 字段声明比较方向：

| direction 值 | 比较规则 | 适用场景 |
|-------------|---------|---------|
| `lower_is_better` | 取较小值 | 体积、延迟、错误率 |
| `higher_is_better` | 取较大值 | 分数、覆盖率、通过率 |

**向后兼容**：`direction` 缺失时，退化为命名推断（含 `max/size/latency/time` → `lower_is_better`；含 `min/score/rate/coverage` → `higher_is_better`）。无法推断时阻断并提示用户补充 `direction`。

## 4. 合并时机与输出

### 4.1 触发时机

`spec-first init` 执行时，根据 `--platforms` 参数加载对应 YAML 文件并合并。

### 4.2 合并流程

```text
init --platforms h5,java-backend
  │
  ├── 1. 加载 Layer 0 基线
  ├── 2. 应用 Layer 1 (Mode × Size) 裁剪
  ├── 3. 加载 .spec-first/layer2/h5.yaml
  ├── 4. 加载 .spec-first/layer2/java-backend.yaml
  ├── 5. 合并 gate_conditions（AND 叠加）
  ├── 6. 合并 extra_deliverables（追加去重）
  ├── 7. 合并 quality_thresholds（取更严格值）
  └── 8. 写入 stage-state.json
```

### 4.3 platforms 必填与交互引导

`--platforms` 为必填参数（对齐 aux-02-cli-system §1 `spec-first init` 命令定义）。

**Skill 交互引导**：`/spec-first:init` 扫描 `.spec-first/layer2/*.yaml`，以多选列表呈现可用平台供用户勾选，选中结果作为 `--platforms` 传入 CLI。无可用 YAML 时提示用户先创建端规范文件。

**CLI 校验**：未传 `--platforms` 时 init 阻断（ExitCode = VALIDATION_ERROR），提示"--platforms 为必填，请至少选择一个平台"。

## 5. 冲突处理

| 场景 | 处理 |
|------|------|
| 平台 YAML 文件不存在 | init 阻断，提示创建或移除该平台 |
| 同阶段同 gate id 冲突 | 阻断，提示重命名 |
| 阈值类型无法自动判定 | 阻断，提示手动指定比较规则 |
| YAML 格式错误 | 阻断，输出解析错误详情 |

---

## 6. 平台名约束

- 仅技术端平台：`h5` / `java-backend` / `app-ios` / `app-android` / `pc` 等
- 命名格式：kebab-case
- `platforms` 数组按字典序排列
- 与 Context Pack 中 `feature_meta.platforms` 共享同一枚举

---

## 7. 最小实现清单

1. `loadPlatformRules(platforms[])` — 加载并校验 YAML
2. `mergeGateConditions(base, ...platformRules)` — AND 叠加
3. `mergeDeliverables(base, ...platformRules)` — 追加去重
4. `mergeThresholds(base, ...platformRules)` — 取更严格值 + 类型推断

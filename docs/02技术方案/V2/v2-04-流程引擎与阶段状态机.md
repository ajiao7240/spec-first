## 1. 目标

用最小状态机支撑 8+2 阶段流程，保证阶段推进可审计、可阻断、可恢复。

---

## 2. 状态定义

### 2.1 阶段主链

```text
00_init → 01_specify → 02_design → 03_plan → 04_implement
  → 05_verify → 06_wrap_up → 07_release → 08_done
```

### 2.2 终态

- `08_done` — 不可逆，仅从 `07_release` 进入
- `09_cancelled` — 不可逆，任意非终态阶段均可进入

### 2.3 合法转换表

| 当前阶段 | 允许目标 |
|---------|---------|
| 00_init | 01_specify |
| 01_specify | 02_design |
| 02_design | 03_plan |
| 03_plan | 04_implement |
| 04_implement | 05_verify |
| 05_verify | 06_wrap_up |
| 06_wrap_up | 07_release |
| 07_release | 08_done |
| 任意非终态 | 09_cancelled |

## 4. 推进规则

### 4.1 advance 前置条件

`spec-first stage advance <featureId>` 执行前必须满足：

1. 当前阶段非终态（`08_done` / `09_cancelled`）
2. 目标阶段为当前阶段的唯一合法后继（不允许跳阶段）
3. Gate 结果为 `PASS` 或 `PASS_WITH_WAIVER`
4. `PASS_WITH_WAIVER` 时必须存在有效豁免记录（关联 RFC + 未过期）

### 4.2 advance 执行流程

```text
advance(featureId, options?)
  │
  ├── 1. 读取 stage-state.json
  ├── 2. 断言非终态
  ├── 3. 计算目标阶段（nextStage）
  ├── 4. 判断是否 --force 模式
  │     ├── --force → 跳过 Gate 校验，进入步骤 5（降级路径）
  │     └── 正常 → 调用 GateEngine.check(featureId, currentStage)
  │           ├── PASS → 继续
  │           ├── PASS_WITH_WAIVER → 校验豁免有效性 → 继续
  │           └── FAIL → 返回错误 + 修复建议，中止
  ├── 5. 更新 stage-state.json
  │     ├── currentStage = nextStage
  │     ├── history 追加当前阶段记录（exitedAt + gateResult）
  │     │     └── --force 时 gateResult = "FORCE_SKIPPED"
  │     └── terminal = (nextStage ∈ {08_done, 09_cancelled})
  └── 6. 写入审计记录
        ├── gate-history.jsonl（含 force 标记）
        └── --force 时追加写入 findings.md（记录跳过原因）
```

### 4.2.1 `--force` 降级路径

`--force` 用于 GateEngine 尚未就绪（阶段 A）或紧急场景下跳过 Gate 校验：

1. Gate 校验完全跳过，不调用 `GateEngine.check()`
2. `gateResult` 记录为 `"FORCE_SKIPPED"`（区别于 PASS/FAIL）
3. 强制写入 `findings.md` 审计记录，包含：操作人、时间、跳过原因
4. 阶段 B 交付 M3 GateEngine 后，应逐步收回 `--force` 使用权限

### 4.3 cancel 规则

- 任意非终态阶段均可 cancel
- 必须提供 `--reason`，写入 history
- cancel 后 `terminal = true`，不可恢复

## 5. 三层合并（init 时执行）

### 5.1 合并流程

```text
init(opts)
  │
  ├── 1. Layer 0：加载基线 Gate 条件 + 基线产出物清单
  ├── 2. Layer 1：根据 Mode × Size 裁剪
  │     ├── Mode I 追加：历史产物定位(00)、影响分析(01)、回归验证(05)
  │     └── Size 调节产出物深度（不跳过阶段）
  ├── 3. Layer 2：读取 .spec-first/layer2/<platform>.yaml
  │     ├── gate_conditions → AND 叠加到对应阶段
  │     ├── extra_deliverables → 追加到阶段产出物列表
  │     └── quality_thresholds → 多端冲突取更严格值
  └── 4. 写入 stage-state.json（含合并后的完整配置）
```

### 5.2 Size 裁剪规则

| 阶段 | S | M | L |
|------|---|---|---|
| 01 Specify | FR/NFR 列表 | + 用户故事 | + 竞品分析 |
| 02 Design | 简要设计 + API 契约（必须） | + 数据模型 | + ADR + 架构图 |
| 03 Plan | 任务列表 | + 依赖关系 | + 风险矩阵 |
| 05 Verify | 功能测试 | + 集成测试 | + 性能测试 + 安全扫描 |

## 6. Mode I 强制逻辑

Mode I（迭代模式）相比 Mode N 多出 3 个必须处理的环节：

| 阶段 | 增量要求 | 产出物 |
|------|---------|--------|
| 00_init | 历史产物定位（找到已有 spec/design/code） | 历史产物清单写入 progress.md |
| 01_specify | Impact Analysis（评估变更影响） | `impact-analysis.md` |
| 05_verify | 回归验证（确保不破坏现有功能） | `reports/regression-report.md` |

Gate 条件自动叠加：Mode I 的 01 阶段 Gate 增加 "impact-analysis.md 存在且非空"。

## 7. 异常处理

| 场景 | ExitCode | 处理 |
|------|----------|------|
| stage-state.json 缺失 | 4 (IO_ERROR) | 提示先执行 `spec-first init` |
| 非法跳阶段 | 3 (CONFIG_ERROR) | 返回 `CONFIG_ERROR: 不允许从 X 跳到 Y` |
| 终态重复推进 | 3 (CONFIG_ERROR) | 返回 `CONFIG_ERROR: Feature 已处于终态` |
| Gate FAIL | 1 (GATE_FAILED) | 返回失败条件列表 + 修复建议 |
| 豁免过期 | 1 (GATE_FAILED) | 返回 `WAIVER_EXPIRED`，提示更新或移除豁免 |

## 8. 最小实现清单

1. `current(featureId)` — 读取并返回 stage-state.json
2. `advance(featureId)` — 校验 + 推进 + 写入 history
3. `cancel(featureId, reason)` — 进入 09_cancelled
4. `assertTransitionAllowed(from, to)` — 合法转换校验
5. `appendHistory(featureId, event)` — 追加 history 记录
6. `mergeLayerRules(mode, size, platforms)` — 三层合并逻辑

## 2. 目录规范

### 2.1 项目级目录

```text
.spec-first/                        # 项目级配置（纳入 Git）
├── current                         # 当前 Feature 指针（纯文本，纳入 .gitignore）
├── config.yaml                     # 全局配置（可选）
├── constitution.md                 # 项目宪法（单例，所有 Feature 共享）
└── layer2/                         # Layer 2 端规范
    ├── h5.yaml
    ├── java-backend.yaml
    ├── go-backend.yaml
    └── ...
```

**约束**：
- `current` 文件内容仅为 featureId 字符串，不纳入版本控制
- `constitution.md` 项目级唯一，Feature 目录下为符号引用（复制副本）
- `config.yaml` 缺失时使用内置默认值，不报错

## 4. 文件格式规范

| 文件类型 | 格式 | 解析方式 | 说明 |
|----------|------|----------|------|
| 规范文档（spec/design/task_plan） | Markdown + YAML frontmatter | remark AST | 支持结构化提取 |
| API 契约 | OpenAPI 3.x YAML | js-yaml | 标准化接口定义 |
| 状态数据（stage-state） | JSON | JSON.parse | 原生解析，无依赖 |
| 时序数据（gate/metrics/ai-stats） | JSONL | 逐行 JSON.parse | 追加写入，不修改历史 |
| 追踪矩阵 | Markdown 表格 | remark AST | 与 spec 同格式 |
| 端规范 | YAML | js-yaml | Layer 2 配置 |

**JSONL 写入规则**：
- 追加模式（`fs.appendFile`），不读取-修改-写回
- 每行一条完整 JSON，以 `\n` 结尾
- 解析时逐行 `JSON.parse`，跳过空行

## 5. 模板系统

### 5.1 模板目录

```text
templates/
├── init/
│   ├── stage-state.json.hbs        # 阶段状态机初始模板
│   └── constitution.md.hbs         # 宪法模板
├── matrix/
│   ├── traceability-matrix.md.hbs  # Markdown 格式矩阵
│   └── traceability-matrix.yaml.hbs # YAML 格式矩阵
├── gate/
│   └── gate-report.md.hbs         # Gate 评估报告
├── review/
│   └── code-review-report.md.hbs  # 代码评审报告
└── metrics/
    └── health-report.md.hbs       # 健康度报告
```

### 5.2 模板变量

所有模板共享的基础变量：

```typescript
interface TemplateContext {
  featureId: string;       // FSREQ-YYYYMMDD-FEAT-NNN
  title: string;           // Feature 标题
  mode: 'N' | 'I';        // 开发模式
  size: 'S' | 'M' | 'L';  // 规模
  platforms: string[];     // 技术端列表
  timestamp: string;       // ISO8601
  author: string;          // 操作人
}
```

阶段特定变量按需注入，不做全局膨胀。

### 5.3 渲染规则

1. **引擎**：Handlebars（零逻辑模板，避免模板内嵌复杂判断）
2. **渲染时机**：仅在 `init` 和 `ensureArtifacts` 时触发，不重复渲染已存在文件
3. **覆盖策略**：文件已存在则跳过，不覆盖用户修改
4. **缺失模板**：模板文件不存在时报错并阻断，不使用空文件兜底

### 5.4 模板版本管理

`stage-state.json` 记录创建时使用的模板版本：

```json
{
  "template_version": "1.0.0",
  "...": "..."
}
```

**版本策略**：

| 场景 | 行为 |
|------|------|
| 新建 Feature | `init` 写入当前 `template_version` |
| 模板升级（minor） | 已有 Feature 不受影响，新 Feature 使用新版 |
| 模板升级（major） | `doctor` 检测版本差异，输出迁移建议（不自动迁移） |

**迁移原则**：模板升级不自动修改已有产出物。`doctor` 输出差异报告，由用户决定是否手动迁移。

## 6. 运行态三文件

Skill 执行过程中持续更新的 3 个文件，用于 Session Catchup 和进度追踪。

### 6.1 文件职责

| 文件 | 职责 | 写入者 |
|------|------|--------|
| `stage-state.json` | 记录每个阶段的完成状态和关键里程碑 | Skill Phase 4（写入阶段） |
| `findings.md` | 记录过程中的发现、决策、风险 | Skill Phase 2（推理阶段） |
| `task_plan.md` | 当前任务计划与执行进度 | plan Skill + 各阶段 Skill |

### 6.2 更新时机

| 触发事件 | stage-state.json | findings.md | task_plan.md |
|----------|-------------|-------------|--------------|
| 阶段推进成功 | ✅ 追加里程碑 | — | — |
| 发现风险/决策 | — | ✅ 追加条目 | — |
| 任务开始/完成 | — | — | ✅ 更新状态 |
| Gate 评估完成 | ✅ 记录结果 | ✅ 记录 FAIL 原因 | — |
| Sync 回填完成 | ✅ 记录同步 | ✅ 记录变更 | — |

### 6.3 格式约定

三个文件均为纯 Markdown，采用追加写入模式：

```markdown
## [2026-02-10T10:30:00Z] 01_specify → 02_design
- Gate 结果: PASS
- 关键产出: spec.md (FR×5, NFR×2)
```

不做结构化解析，仅供人类阅读和 Catchup 恢复使用。

### 6.4 容量治理

运行态三文件采用追加写入模式，长期运行的 Feature 可能导致文件膨胀。滚动归档机制：

| 文件 | 归档阈值 | 归档策略 |
|------|----------|----------|
| `stage-state.json` | 500 行 | 保留最近 200 行，历史归档为 `progress-<ISO8601>.md` |
| `findings.md` | 500 行 | 保留最近 200 行，历史归档为 `findings-<ISO8601>.md` |
| `task_plan.md` | 不归档 | 已完成 TASK 标记 `[DONE]` 但保留（供追踪查询） |

**触发时机**：每次 Skill Phase 5 写入前检查行数，超限时自动归档。

**Catchup 兼容**：`ai catchup` 优先读取当前文件；需要历史上下文时按时间倒序扫描归档文件。

## 7. 最小实现清单

1. `initWorkspace(featureId, opts)` — 创建 Feature 目录 + 渲染初始模板
2. `renderTemplate(templateName, context)` — Handlebars 渲染 + 写入文件
3. `ensureArtifacts(featureId, stage)` — 按 Mode×Size 检查并补全阶段产出物
4. `updateProgress(featureId, entry)` — 追加 stage-state.json 条目
5. `updateFindings(featureId, entry)` — 追加 findings.md 条目
6. `listArtifacts(featureId)` — 列出 Feature 下所有产出物及状态

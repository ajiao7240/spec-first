# V2-01 总体技术架构

> **对齐需求**: core-01-overview · core-02-architecture · README
> **版本**: v2.1 | **日期**: 2026-02-10 | **原则**: KISS

---

## 1. 目标

构建 **Skill 驱动编排 + CLI 确定性执行** 的研发效能引擎：

1. 用户入口统一为 `/spec-first:*`，禁止裸调 CLI
2. 状态变更全量可追踪、可回放（JSONL 审计）
3. 覆盖率 / Gate 可量化、可审计
4. 支持 Mode × Size × Platform 三维裁剪

---

## 2. 架构分层

```text
┌─────────────────────────────────────────────────────────┐
│  人类（PM / Tech Lead / Dev / QA）                       │
│  决策 · 确认 · 签核                                      │
└────────────────────┬────────────────────────────────────┘
                     │ /spec-first:*
┌────────────────────▼────────────────────────────────────┐
│  入口路由层（Runtime Router）                             │
│  ┌────────────────────────────────────────────────────┐ │
│  │ Skill 路由 → .md Skill 文件（生成/编排/交互）        │ │
│  │ Runtime 路由 → CLI 原子命令（stage/id/gate/rfc/...） │ │
│  └────────────────────────────────────────────────────┘ │
├─────────────────────────────────────────────────────────┤
│  Skill 层（流程编排 · 16 个 Skill）                      │
│  职责: 流程推进 · 阶段流转触发 · AI 推理生成 · 交互引导   │
│  实现: .md 文件 + 6 阶段执行模型（Phase 0-5）        │
│  宿主: Claude Code / Codex CLI / 其他 Agent              │
├─────────────────────────────────────────────────────────┤
│  CLI 层（确定性原子能力 · 13 命令组 × 7 模块）            │
│  职责: ID 注册 · Gate 判定 · 状态推进 · 覆盖率计算        │
│  实现: TypeScript ESM · Node.js 20+                      │
│  约束: 相同输入 = 相同输出，不主动编排                     │
├─────────────────────────────────────────────────────────┤
│  持久层                                                   │
│  ┌──────────────────┬───────────────────────────────┐   │
│  │ specs/<featureId> │ .spec-first/                  │   │
│  │ 产出物 + 状态文件  │ config + constitution + layer2│   │
│  └──────────────────┴───────────────────────────────┘   │
│  ┌──────────────────────────────────────────────────┐   │
│  │ Git Hook (Layer B) + CI Pipeline                  │   │
│  └──────────────────────────────────────────────────┘   │
└─────────────────────────────────────────────────────────┘
```

### 2.1 层间调用规则

| 调用方向 | 允许 | 说明 |
|---------|------|------|
| 人 → 入口路由 | ✅ | 唯一用户入口 |
| 入口路由 → Skill | ✅ | 生成/编排类命令 |
| 入口路由 → CLI | ✅ | 状态变更类命令（Runtime 路由） |
| Skill → CLI | ✅ | Skill 调度 CLI 执行确定性操作 |
| CLI → Skill | ❌ | CLI 不感知 Skill 存在 |
| CLI → 持久层 | ✅ | 读写 specs/ 和 .spec-first/ |
| Hook/CI → CLI | ✅ | 自动化场景直调 CLI |

### 2.2 路由判定逻辑

```text
用户输入 /spec-first:<cmd> [args]
  │
  ├── cmd ∈ {stage, id, gate, matrix, rfc, defect, metrics, feature, commit, golive}
  │     → Runtime 路由：直接映射 CLI `spec-first <cmd> <args>`
  │
  └── cmd ∈ {init, catchup, spec, design, research, task, code,
  │          code-review, test, archive, plan, verify, orchestrate,
  │          status, doctor, sync}
  │     → Skill 路由：加载 .md Skill 文件，进入 6 阶段执行模型
```

## 3. 核心运行对象

### 3.1 Feature

- **ID 格式**: `FSREQ-YYYYMMDD-<FEAT>-NNN`
- **正则**: `^FSREQ-\d{8}-[A-Z][A-Z0-9]{1,15}-\d{3}$`
- **生命周期**: `00_init` → `07_release` → `08_done` | 任意阶段 → `09_cancelled`
- **工作区**: `specs/<featureId>/`

### 3.2 Stage（阶段）

```typescript
enum Stage {
  INIT         = '00_init',
  SPECIFY      = '01_specify',
  DESIGN       = '02_design',
  PLAN         = '03_plan',
  IMPLEMENT    = '04_implement',
  VERIFY       = '05_verify',
  WRAP_UP      = '06_wrap_up',
  RELEASE      = '07_release',
  DONE         = '08_done',      // 终态
  CANCELLED    = '09_cancelled', // 终态
}
```

### 3.3 三层规范

| 层 | 内容 | 数据来源 |
|----|------|---------|
| Layer 0 | 8+2 阶段基线 Gate + 基线产出物 | 硬编码在 ProcessEngine |
| Layer 1 | Mode × Size 裁剪规则 | 硬编码在 ProcessEngine |
| Layer 2 | 端特有规范（gate / deliverables / thresholds） | `.spec-first/layer2/*.yaml` |

**合并时机**：`spec-first init` 执行时读取 Layer 0 + 应用 Layer 1 + 合并 Layer 2，写入 `stage-state.json`。

### 3.4 运行态指针

| 文件 | 用途 | 纳入 Git |
|------|------|---------|
| `.spec-first/current` | 当前 Feature ID（纯文本） | ❌ (.gitignore) |
| `specs/<fId>/stage-state.json` | 阶段状态 + 元数据 | ✅ |
| `specs/<fId>/progress.md` | 已完成进度 | ✅ |
| `specs/<fId>/findings.md` | 发现与审计记录 | ✅ |
| `specs/<fId>/task_plan.md` | 任务规划状态 | ✅ |

## 4. 7 个核心模块概览

| 模块 | 名称 | 入口命令 | 核心职责 |
|------|------|---------|---------|
| **M1** | ProcessEngine | `stage` | 阶段状态机、三层合并、advance/cancel |
| **M2** | TraceEngine | `id` / `matrix` / `metrics coverage` | ID 注册/校验/搜索、矩阵管理、覆盖率 |
| **M3** | GateEngine | `gate` | Gate 条件评估、SCA 校验、PASS/FAIL/WAIVER |
| **M4** | ChangeMgr | `rfc` / `defect` | RFC 状态机、缺陷管理 |
| **M5** | AIOrchestrator | `ai` | Context Pack 生成、Catchup、AI 统计 |
| **M6** | MetricsEngine | `metrics report` | 12 指标计算、健康分、瓶颈分析 |
| **M7** | ToolIntegration | `doctor` / `commit` | Git Hook 安装、CI 模板、环境诊断 |

### 4.1 模块间依赖

```text
M1 ProcessEngine ← M3 GateEngine     （advance 前调用 gate check）
M2 TraceEngine   ← M3 GateEngine     （gate check 调用覆盖率计算）
M2 TraceEngine   ← M6 MetricsEngine  （metrics 调用覆盖率计算）
M4 ChangeMgr     ← M2 TraceEngine    （影响分析基于追踪链）
M5 AIOrchestrator← M2 TraceEngine    （Context Pack 读取矩阵）
M7 ToolIntegration← M3 GateEngine    （Hook 调用 gate check）
```

**依赖原则**：单向依赖，禁止循环。M2 TraceEngine 是被依赖最多的模块，优先实现。

## 5. 关键设计决策

| # | 决策 | 理由 |
|---|------|------|
| D1 | **文件即状态，不引入数据库** | `stage-state.json` + `traceability-matrix.md` 作为单一真相；Git 天然提供版本管理 |
| D2 | **JSONL 审计日志** | `gate-history.jsonl`、`metrics.jsonl`、`ai-stats.jsonl`；追加写入，天然审计友好 |
| D3 | **路由单点** | 所有用户入口走 `/spec-first:*`，杜绝多入口分裂 |
| D4 | **Layer 2 可插拔** | 端规范文件独立 YAML，无端规范时退化为纯 Layer 0+1 |
| D5 | **Skill 无状态** | Skill 只做编排和生成，不持久化任何状态；所有状态变更通过 CLI 完成 |
| D6 | **Handlebars 模板** | 初始化文件通过模板生成，避免硬编码字符串拼接 |

## 6. 非功能要求

| 维度 | 要求 | 验证方式 |
|------|------|---------|
| **确定性** | CLI 同输入同输出 | 单元测试 + 快照测试 |
| **性能** | `validateId` < 10ms, `getCoverage` < 50ms, `evaluateGate` < 200ms | 基准测试 |
| **可观测** | 每次 stage advance / gate check 写入 JSONL | 审计日志检查 |
| **可恢复** | Catchup 从三文件 + stage-state.json 恢复到可执行状态 | E2E 测试 |
| **可裁剪** | Mode × Size × Platform 组合不改变核心逻辑 | 参数化测试 |
| **跨平台** | CLI 在 Node.js 20+ 运行；Skill 在 Claude Code / Codex CLI 运行 | CI 多环境测试 |

## 10. 实施顺序（推荐）

```text
Phase 1 — 基础能力
  M2 TraceEngine → M1 ProcessEngine → M3 GateEngine

Phase 2 — 质量闭环
  M4 ChangeMgr → M7 ToolIntegration（Hook）

Phase 3 — AI 协作
  M5 AIOrchestrator → Skill 文件编写

Phase 4 — 扩展与运营
  M6 MetricsEngine → Layer 2 端规范 → 模板系统

Phase 5 — 上线
  E2E 测试 → GoLive 门槛校验 → 灰度发布
```

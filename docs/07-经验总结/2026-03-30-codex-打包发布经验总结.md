# Codex 打包发布经验总结

## 背景

这次工作目标是让 `spec-first init --codex` 在当前 `codex-cli 0.117.0` 环境下真正可用，并且能通过本地打包、全局安装、项目初始化、重启 Codex 之后的真实交互验证。

在修复过程中，最大的教训不是“文件有没有生成”，而是“Codex 到底按什么约定发现这些资产”。

## 结论先行

当前版本下，`spec-first` 对 Codex 的可靠接入方式是：

- 项目级 skills：`.agents/skills/`
- 项目级 agents：`.codex/agents/`
- 兼容命令层：`.codex/commands/spec/`
- 项目级状态：`.codex/spec-first/`
- 正式 discovery / skill 入口：`$spec-brainstorm`、`$spec-plan`、`$spec-work`、`$spec-review`、`$spec-compound`
- `/spec:*` 命令文件可以生成到 `.codex/commands/spec/`，但它们是兼容命令层，不是 Codex 正式可用性的唯一判定依据

不再采用以下方案：

- `.codex/skills/*`
- repo-local plugin marketplace
- 把 `/spec:*` 当成 Codex 的主入口或正式 discovery 契约

## 这次踩过的坑

### 1. 不能把 Claude 的运行时模型直接平移到 Codex

最早的实现把 Claude Code 的项目内命令模型直接类推到了 Codex：

- `.codex/commands/spec/*.md`
- `.codex/skills/*`
- `.codex/agents/*`

这个模型对 Claude 成立，但对当前 Codex CLI 不成立。结果是：

- `spec-first init --codex` 看起来生成成功
- `spec-first doctor --codex` 也可能全绿
- 但真实 `codex` 会话里并没有可用入口

经验：跨平台支持不能靠“目录长得像”，必须先确认目标平台的真实加载契约。

### 2. repo-local plugin slash commands 在当前环境里并不可靠

中间一度改成了 repo-local plugin 方案，尝试通过：

- `.agents/plugins/marketplace.json`
- `plugins/spec/.codex-plugin/plugin.json`
- `plugins/spec/commands/*`

把 Codex 做成 `/spec:*` slash commands。

这个方案结构上自洽，但在当前 `codex-cli 0.117.0` 环境里，真实交互验证没有通过：

- 输入 `$spec`，出来的是 skill 搜索
- 输入 `/spec`，显示 `no matches`

经验：只要真实 CLI 不认，这条链路就不能继续投入。能不能“理论上接通”不重要，重要的是用户当前版本能不能用。

### 3. `doctor` 不能只检查“有没有文件”

这次一个明显问题是：早期 `doctor` 只是检查某些目标路径是否存在，因此错误目录结构也会 `PASS`。

这会带来非常强的误导：

- 用户以为平台已经安装成功
- 实际上 Codex 根本不会加载这些目录

经验：平台诊断必须围绕“真实运行时契约”设计，而不是围绕“我们自己猜的目标路径”设计。

### 4. Codex skill 的目录名和 `name:` 必须一致

最终快修通时，还遇到一个更隐蔽的问题：

- 目录名已经是 `spec-brainstorm`
- 但 `SKILL.md` 里的 frontmatter 仍然是 `name: brainstorm-workflow`

这会导致 Codex 不能按 `$spec-brainstorm` 正确命中。

最后的修复方式是：生成 Codex runtime skill 时，把 frontmatter 的 `name:` 重写为目录名，例如：

- `spec-brainstorm`
- `spec-plan`
- `spec-work`
- `spec-review`
- `spec-compound`

经验：对 Codex 来说，skill 的“磁盘目录名”和“被发现时的名字”不是两回事，不能脱节。

### 5. 只在源码仓库里验证不够，必须验证 tarball 安装

如果只在源码目录里直接跑，很容易被这些因素误导：

- shell 缓存
- 旧的全局 shim
- `npm install -g .` 带来的软链行为
- 当前工作区里已有的手工生成产物

因此这次后面把发布验证标准收紧成：

1. `npm pack`
2. `npm install -g ./spec-first-<version>.tgz`
3. `hash -r`
4. 在目标项目运行 `spec-first clean --codex`
5. 在目标项目运行 `spec-first init --codex`
6. 完全退出并重启 `codex`
7. 在新会话里输入 `$spec-brainstorm`

只有走完整链路，才算发布物真的可用。

### 6. 文档如果不跟着改，用户还会按旧入口继续失败

代码修好之后，如果 README、快速开始、FAQ、本地安装指南还在把这些过时口径写成事实：

- Codex 只有 `/spec:*`，没有 `$spec-*`
- Codex 正式 discovery 看 `.codex/commands/spec/`
- Codex skills 生成在 `.codex/skills/`

那用户仍然会按旧方式操作，然后判断“安装失败”。

经验：多平台入口一旦分化，用户文档必须明确区分：

- Claude Code：`/spec:*`
- Codex：正式 discovery 看 `$spec-*` / `.agents/skills/`，`.codex/commands/spec/` 只是兼容命令层

## 最终稳定模型

### Claude Code

- 入口：`/spec:*`
- 命令目录：`.claude/commands/spec/`
- skills：`.claude/skills/`
- agents：`.claude/agents/`
- 状态：`.claude/spec-first/`

### Codex

- 正式入口：`$spec-*`
- 兼容命令层：`.codex/commands/spec/`
- skills：`.agents/skills/`
- agents：`.codex/agents/`
- 状态：`.codex/spec-first/`

## 本次沉淀出的发布验收清单

以后每次涉及 Codex 发布，至少执行以下检查。

### A. 打包产物检查

```bash
npm pack --dry-run
```

确认：

- CLI 入口存在
- `skills/` 和 `agents/` 被打进包里
- 没有把仓库内部调试脚本误打进去

### B. 全局安装检查

```bash
npm pack
npm install -g ./spec-first-<version>.tgz
hash -r
which spec-first
spec-first --version
```

确认：

- 当前 shell 指向的是新安装版本
- 不是旧的 pnpm shim
- 不是旧 tarball

### C. 项目初始化检查

```bash
spec-first clean --codex
spec-first init --codex
spec-first doctor --codex
```

确认：

- `.agents/skills/spec-brainstorm/SKILL.md` 存在
- `.codex/commands/spec/brainstorm.md` 存在（兼容命令层）
- `.codex/agents/` 存在
- `.codex/spec-first/.developer` 存在
- `.codex/spec-first/state.json` 中记录的 commands / skills / agents 数量与当前生成资产一致，不要硬编码历史数量

### D. 真实 Codex 交互检查

```bash
codex
```

在新会话里输入：

```text
$spec-brainstorm
```

确认：

 - 可以出现 `spec-ideate [Skill]` 或 `spec-brainstorm [Skill]` 候选
- 描述来自项目内生成的 skill

### E. 回归测试检查

```bash
bash tests/smoke/cli.sh
npm test
```

确认：

- smoke 覆盖 Codex 运行时目录
- 集成测试不被这次修改破坏

## 后续建议

### 1. 不再把 Codex 设计成 Claude 的镜像平台

未来新增平台支持时，应该先回答：

- 平台入口是什么
- 平台发现的是 command、skill 还是 prompt
- 项目级资产目录是什么
- 用户级资产目录是什么

不要先写代码，再靠目录命名去猜。

### 2. 把“真实 CLI 可发现性”纳入自动化测试思路

目前 smoke 已经覆盖了文件结构和迁移清理，但“Codex 真能否发现 skill”仍主要依赖人工会话验证。

后续如果有稳定办法，建议把以下断言自动化：

- 生成后的 skill frontmatter `name:` 与目录名一致
- Codex 入口 skill 的命名固定为 `spec-*`

### 3. 保持文档与运行时模型同步

以后只要 Codex 入口、目录、命名规则发生变化，必须同步更新：

- `README.md`
- `docs/05-用户手册/01-快速开始.md`
- `docs/05-用户手册/04-常见问题.md`
- `docs/05-用户手册/06-本地源码安装.md`

否则代码修好了，用户体验仍然是坏的。

## 一句话总结

这次 Codex 打包发布最大的经验是：

**对 Codex，真正重要的不是“我们生成了什么文件”，而是“Codex 当前版本到底会发现什么、按什么名字发现、正式 discovery 与兼容命令层各自扮演什么角色、用户最终怎么触发”。**

## 1. Skill 概述

### 1.1 核心定位

**check-cross-layer** 是一个多维度验证命令，检查变更是否考虑了所有相关维度。

| 维度 | 描述 |
|------|------|
| **目标** | 验证变更的完整性 |
| **触发时机** | 实现后（安全网） |
| **核心理念** | 大多数 Bug 来自"没想到"，而非技术能力不足 |

## 3. 检查维度

### 3.1 维度概览

```
┌─────────────────────────────────────────────────────────────┐
│                  check-cross-layer 维度                      │
└─────────────────────────────────────────────────────────────┘

  A. Cross-Layer Data Flow    ← 3+ 层时必需
  B. Code Reuse               ← 修改常量/配置时必需
     B2. New Utility Functions ← 创建新工具函数
     B3. After Batch Mods      ← 批量修改后
  C. Import/Dependency Paths  ← 创建新文件时必需
  D. Same-Layer Consistency   ← 同层一致性
```

### 3.2 维度 A: 跨层数据流

**触发条件**: 变更涉及 3 层或更多

| 层级 | 常见位置 |
|------|---------|
| API/Routes | `routes/`, `api/`, `handlers/`, `controllers/` |
| Service/Business | `services/`, `lib/`, `core/`, `domain/` |
| Database/Storage | `db/`, `models/`, `repositories/`, `schema/` |
| UI/Presentation | `components/`, `views/`, `templates/`, `pages/` |
| Utility | `utils/`, `helpers/`, `common/` |

**检查清单**:
- [ ] 读流程: Database → Service → API → UI
- [ ] 写流程: UI → API → Service → Database
- [ ] 类型/schema 正确传递？
- [ ] 错误正确传播到调用者？
- [ ] 加载/pending 状态在每层都处理？

**详细指南**: `.spec-first/spec/guides/cross-layer-thinking-guide.md`

### 3.3 维度 B: 代码复用

**触发条件**:
- 修改 UI 常量（标签、图标、颜色）
- 修改任何硬编码值
- 多处看到相似代码
- 创建新的工具/辅助函数
- 刚完成跨文件批量修改

**检查清单**:
- [ ] 先搜索: 多少地方定义了这个值？
  ```bash
  grep -r "value-to-change" src/
  ```
- [ ] 如果 2+ 处定义相同值 → 应提取到共享常量
- [ ] 修改后，所有使用点都更新了？
- [ ] 如果创建工具: 是否已存在类似工具？

**详细指南**: `.spec-first/spec/guides/code-reuse-thinking-guide.md`

### 3.4 维度 B2: 新工具函数

**触发条件**: 即将创建新的工具/辅助函数

**检查清单**:
- [ ] 先搜索是否已存在类似工具
  ```bash
  grep -r "functionNamePattern" src/
  ```
- [ ] 如果存在类似，能否扩展它？
- [ ] 如果创建新的，位置是否正确（共享 vs 领域特定）？

### 3.5 维度 B3: 批量修改后

**触发条件**: 刚在多个文件中修改了相似模式

**检查清单**:
- [ ] 是否检查了所有有相似模式的文件？
  ```bash
  grep -r "patternYouChanged" src/
  ```
- [ ] 是否有遗漏的文件？
- [ ] 是否应该抽象此模式以防止未来重复？

### 3.6 维度 C: 导入/依赖路径

**触发条件**: 创建新的源文件

**检查清单**:
- [ ] 使用正确的导入路径（相对 vs 绝对）？
- [ ] 无循环依赖？
- [ ] 与项目模块组织一致？

### 3.7 维度 D: 同层一致性

**触发条件**:
- 修改显示逻辑或格式化
- 同一领域概念在多处使用

**检查清单**:
- [ ] 搜索其他使用相同概念的地方
  ```bash
  grep -r "ConceptName" src/
  ```
- [ ] 这些使用是否一致？
- [ ] 是否应该共享配置/常量？

## 4. 常见问题速查

| 问题 | 根因 | 预防 |
|------|------|------|
| 改了一处，漏了其他 | 没有搜索影响范围 | 修改前 `grep` |
| 数据在某层丢失 | 没有检查数据流 | 追踪数据源到目标 |
| 类型/schema 不匹配 | 跨层类型不一致 | 使用共享类型定义 |
| UI/输出不一致 | 同一概念在多处 | 提取共享常量 |
| 类似工具已存在 | 没有先搜索 | 创建前搜索 |
| 批量修复不完整 | 没有验证所有出现 | 修复后 grep |

## 5. 执行流程

```
┌─────────────────────────────────────────────────────────────┐
│                check-cross-layer 执行流程                    │
└─────────────────────────────────────────────────────────────┘

  ┌─────────────────┐
  │ 1. 识别变更范围 │
  │ git status/diff │
  └────────┬────────┘
           │
           ▼
  ┌─────────────────┐
  │ 2. 选择适用维度 │
  │ 根据变更类型   │
  └────────┬────────┘
           │
           ├─▶ 3+ 层 → 维度 A
           ├─▶ 修改常量 → 维度 B/B2/B3
           ├─▶ 创建文件 → 维度 C
           └─▶ 同层变更 → 维度 D
           │
           ▼
  ┌─────────────────┐
  │ 3. 执行检查清单 │
  └────────┬────────┘
           │
           ▼
  ┌─────────────────┐
  │ 4. 输出报告     │
  │ 维度 + 结果     │
  └─────────────────┘
```

---

## 6. 输出格式

报告包含：
1. 变更涉及的维度
2. 每个维度的检查结果
3. 发现的问题和修复建议

## 7. 设计分析

### 7.1 预防 vs 检测

```
理想流程:
  Pre-Implementation Checklist → 实现代码 → check-cross-layer
         ↑                              ↑
      预防思考                        检测验证

现实流程:
  实现代码 → check-cross-layer → 修复遗漏
                 ↑
             安全网兜底
```

### 7.2 维度触发逻辑

| 变更类型 | 触发维度 |
|---------|---------|
| 跨多层变更 | A |
| 修改常量/配置 | B, B2, B3 |
| 创建新文件 | C |
| 同层多处修改 | D |

---

## 8. 最佳实践

### 8.1 使用时机

- 完成功能实现后
- 代码审查前
- 提交前作为 finish-work 的补充

### 8.2 检查顺序

1. 先运行 `git diff` 了解变更范围
2. 根据变更类型选择维度
3. 逐一执行检查清单
4. 输出报告并修复问题

---

## 9. 总结

**check-cross-layer** 是多维度安全网：

```
变更 → check-cross-layer
            │
            ├── 维度 A: 跨层数据流
            ├── 维度 B: 代码复用
            ├── 维度 C: 导入路径
            └── 维度 D: 同层一致性
                 │
                 ▼
            完整性报告
```

**核心价值**:
- 捕获"没想到"的问题
- 多维度系统性验证
- 防止常见遗漏
- 提供结构化检查清单

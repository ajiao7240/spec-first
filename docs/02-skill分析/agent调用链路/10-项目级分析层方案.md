# 方案 A：项目级分析层设计

## 目标

在现有 `spec-first` 的单任务执行链路之前，补一层**项目级分析层**，用于处理“大需求 -> 完整需求分析 -> 技术方案 -> 任务拆分”这类场景。

这层只负责把项目级大需求拆成可执行的结构化产物，不改变现有 Claude 执行层的实现方式。

---

## 背景

当前 `spec-first` 更偏向单任务驱动：

```text
/spec:start -> /spec:brainstorm -> task directory -> dispatch -> implement/check -> finish-work -> create-pr
```

它适合：
- 单个任务
- 小中型改动
- task directory 驱动的开发

但它不擅长：
- 一个大需求同时包含多个独立子系统
- 先产出项目级 PRD
- 再产出完整技术方案
- 再拆分任务列表
- 最后把任务分发到现有执行链路

方案 A 的目标就是补这一层，而不是重写执行层。

---

## 总体原则

1. **分析层和执行层分离**
   - 分析层负责“想清楚、拆清楚”
   - 执行层负责“做出来、验证掉”

2. **分析层产物先于任务目录**
   - 大需求先形成项目级文档
   - 再由文档拆出多个任务目录

3. **不改现有执行链路的语义**
   - 仍然保留 `dispatch / implement / check / finish-work / create-pr`
   - 仍然保留现有 Claude hook、task directory、subagent 机制

4. **只在“大需求”场景启用**
   - 小任务继续走现有 `brainstorm -> task directory -> implement`
   - 只有多子系统、多阶段、多依赖需求才进入项目级分析层

---

## 适用场景

适合进入项目级分析层的需求通常具备以下特征：

- 同时涉及多个独立子系统
- 需要统一的产品目标和范围边界
- 需要完整技术方案再拆任务
- 任务之间存在明确依赖关系
- 需要多个里程碑或多个 PR

不建议进入该层的场景：

- 单文件修复
- 小范围 UI / 文案 / 样式调整
- 单个 API 或单个 hook 改造
- 已经很明确的低风险任务

---

## 推荐流程

### 1. 项目级需求分析

先把用户的大需求整理成项目级 PRD。

建议产物：
- `project-prd.md`

内容建议包含：
- 背景和目标
- 用户场景
- 范围边界
- 非目标
- 关键约束
- 验收标准
- 风险与未决问题
- 子系统拆分建议

### 2. 项目级技术方案

在 PRD 基础上生成技术方案。

建议产物：
- `tech-design.md`

内容建议包含：
- 总体架构
- 模块边界
- 数据流
- 关键接口
- 配置与约定
- 错误处理
- 测试策略
- 迁移/兼容策略（如适用）

### 3. 任务拆分文档

把技术方案拆成可以独立执行的任务。

建议产物：
- `tasks.md`

内容建议包含：
- 任务顺序
- 依赖关系
- 可并行任务
- 每个任务的输入/输出
- 每个任务的文件落点
- 每个任务的验收条件

### 4. 映射到现有执行层

将 `tasks.md` 中的任务映射为现有 `spec-first` 的任务目录。

映射后，每个子任务仍然使用现有执行链路：

```text
/spec:brainstorm -> task directory -> dispatch -> implement/check -> finish-work -> create-pr
```

---

## 推荐目录结构

项目级分析层建议落在独立目录下，避免和单任务目录混淆。

示例：

```text
.spec-first/
  projects/
    <project-name>/
      project-prd.md
      tech-design.md
      tasks.md
      analysis.json
      subprojects/
        <subproject-1>/
          project-prd.md
          tech-design.md
          tasks.md
        <subproject-2>/
          project-prd.md
          tech-design.md
          tasks.md
```

### 目录职责

- `project-prd.md`
  - 项目级需求分析结果
- `tech-design.md`
  - 项目级技术方案
- `tasks.md`
  - 项目级任务拆分
- `analysis.json`
  - 结构化元数据，例如项目 id、状态、阶段、关联 task directory
- `subprojects/`
  - 当一个大需求需要拆成多个独立子项目时使用

---

## 任务流转方式

项目级分析层的产物不是直接执行，而是作为**任务生成输入**：

```text
用户大需求
  -> 项目级 PRD
  -> 项目级技术方案
  -> 项目级 tasks.md
  -> 生成多个 task directory
  -> 进入现有 Claude 执行链路
```

### 任务映射规则

建议按以下方式映射：

- `tasks.md` 作为单一任务源文件
- `tasks.md` 的每个**独立交付单元** -> 执行时分发给一个 subagent
- 独立子系统 -> 作为一个 subproject 记录在任务源文件中
- 强耦合步骤 -> 保留在同一个任务条目内作为子步骤
- 可并行任务 -> 在 `tasks.md` 中标记并由 controller 并发分发多个 subagent

### 父子关系

仍然可以使用当前已有的：
- `task.py create --parent`
- `task.py add-subtask`

但它们应只用于**确实需要独立 task directory** 的情况，例如：
- 子项目需要独立上下文
- 子项目需要独立 PR
- 子项目需要独立生命周期管理

默认情况下，项目级任务拆分只保留在 `tasks.md` 中，不强制生成大量目录。

---

## 与现有 spec-first 的关系

### 保留的部分

现有执行层保留不变：
- `/spec:start`
- `/spec:brainstorm`
- `dispatch`
- `implement`
- `check`
- `finish-work`
- `create-pr`

### 新增的部分

新增一个项目级分析入口，例如：
- `/project:analyze`
- 或 `/spec:project`
- 或独立 CLI 命令 `project-plan`

这个入口专门处理：
- 大需求归纳
- 项目级 PRD
- 技术方案
- 任务拆分

### 不修改的部分

不修改现有：
- task directory 的执行逻辑
- Claude hook 的消费逻辑
- `task.json` 的基本执行语义
- 现有单任务流程

---

## 建议命令设计

### 项目级分析命令

建议新增一个入口命令：

```text
/project:analyze
```

职责：
- 收敛项目级需求
- 生成项目级 PRD
- 生成技术方案
- 拆分任务列表
- 产出可执行的项目级任务包

### 单任务命令

继续保留现有：

```text
/spec:start
/spec:brainstorm
```

职责：
- 单任务范围内的需求澄清
- 单个 task directory 的生成
- 单任务执行上下文准备

---

## 何时进入项目级分析层

建议加入一个明确的判断门：

### 进入条件

满足以下任意一条即可进入：
- 需求明显包含多个独立子系统
- 需要统一技术路线再拆任务
- 需要多个并行 PR
- 需要一份可供团队评审的项目级技术方案

### 不进入条件

以下情况继续走现有单任务流程：
- 需求范围单一
- 改动集中
- 可在一个 task directory 内完成
- 不需要项目级任务拆分

## 风险与约束

### 风险 1：分析层变成第二套执行系统

避免方式：
- 分析层只产文档，不直接执行代码
- 执行仍然走现有 task directory 流程

### 风险 2：项目级文档和 task directory 重复

避免方式：
- 项目级文档只做全局决策
- task directory 只做局部执行
- 两者职责不重叠

### 风险 3：流程过重

避免方式：
- 小任务不走项目级分析层
- 只有复杂大需求才启用

## 端到端流程图

```text
用户提出大需求
  -> 进入项目级分析层
  -> 需求澄清 / 范围收敛
  -> 生成 project-prd.md
  -> 生成 tech-design.md
  -> 生成 tasks.md
  -> 记录 analysis.json
  -> 按任务边界拆分 subprojects/
  -> 为每个子任务创建 task directory
  -> 进入现有 spec-first 执行层
      -> /spec:brainstorm（如仍需任务级澄清）
      -> dispatch
      -> implement
      -> check
      -> finish-work
      -> create-pr
```

### 流程分层

```text
项目级分析层
  1. 需求分析
  2. 技术方案
  3. 任务拆分

执行层
  4. task directory
  5. dispatch / implement / check
  6. finish-work / create-pr
```

### 分层职责

- 项目级分析层回答：
  - 需求是什么
  - 技术方案怎么定
  - 任务怎么拆
  - 哪些子任务可以并行

- 执行层回答：
  - 当前 task 如何实现
  - 当前 task 如何验证
  - 当前 task 如何收尾

---

## 最小实施顺序

建议先做最小版本：

1. 先支持 `project-prd.md`
2. 再支持 `tech-design.md`
3. 再支持 `tasks.md`
4. 最后再接入任务目录生成

不要一开始就做完整项目管理系统。

---

## 推荐结论

方案 A 的本质是：

**在现有单任务执行链前面，加一层项目级分析层，用来处理大需求的 PRD、技术方案和任务拆分。**

它的价值在于：
- 让大需求有统一入口
- 让技术方案先行
- 让任务拆分有依据
- 让现有执行层保持不变

这是一种“补前置，不重写后置”的方案。

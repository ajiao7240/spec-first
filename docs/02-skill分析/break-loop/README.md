# Break-Loop 深度分析

> 源文件: `/packages/cli/src/templates/claude/commands/spec/break-loop.md`

---

## 1. Skill 概述

### 1.1 核心定位

**break-loop** 是深度 Bug 分析命令，用于打破"修复 Bug → 遗忘 → 重复"的循环。

| 维度 | 描述 |
|------|------|
| **目标** | 从 Bug 中提取系统性知识 |
| **触发时机** | 调试完成后 |
| **输出** | 5 维度分析报告 + spec 更新 |

### 1.2 核心哲学

```
┌─────────────────────────────────────────────────────────────┐
│                    核心哲学                                  │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   调试的价值不在于修复 Bug，                                │
│   而在于让这类 Bug 永远不再发生。                           │
│                                                             │
│   30 分钟的分析节省 30 小时的未来调试。                     │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## 2. 分析框架

### 2.1 五维度分析

```
┌─────────────────────────────────────────────────────────────┐
│                   break-loop 分析框架                        │
└─────────────────────────────────────────────────────────────┘

  ┌─────────────────┐
  │ 1. 根因分类     │ → A/B/C/D/E 五类
  └────────┬────────┘
           │
           ▼
  ┌─────────────────┐
  │ 2. 修复失败分析 │ → 为什么之前的尝试失败了？
  └────────┬────────┘
           │
           ▼
  ┌─────────────────┐
  │ 3. 预防机制     │ → 如何防止再次发生？
  └────────┬────────┘
           │
           ▼
  ┌─────────────────┐
  │ 4. 系统性扩展   │ → 揭示什么更广泛的问题？
  └────────┬────────┘
           │
           ▼
  ┌─────────────────┐
  │ 5. 知识捕获     │ → 如何固化洞察？
  └─────────────────┘
```

### 2.2 维度详解

#### 1. 根因分类

| 类别 | 特征 | 示例 |
|------|------|------|
| **A. Missing Spec** | 没有文档说明如何做 | 新功能没有检查清单 |
| **B. Cross-Layer Contract** | 层间接口不清晰 | API 返回格式与预期不同 |
| **C. Change Propagation Failure** | 改了一处，漏了其他 | 修改函数签名，漏了调用点 |
| **D. Test Coverage Gap** | 单元测试通过，集成失败 | 单独工作，组合后失败 |
| **E. Implicit Assumption** | 代码依赖未记录的假设 | 时间戳秒 vs 毫秒 |

#### 2. 修复失败分析

如果多次尝试才成功，分析每次失败：

| 失败类型 | 描述 |
|---------|------|
| **表面修复** | 修复症状，非根因 |
| **范围不完整** | 找到根因，但未覆盖所有情况 |
| **工具限制** | grep 漏了，类型检查不够严格 |
| **心智模型** | 一直在同一层找，没想到跨层 |

#### 3. 预防机制

| 类型 | 描述 | 示例 |
|------|------|------|
| **Documentation** | 写下来让人知道 | 更新思考指南 |
| **Architecture** | 结构上让错误不可能 | 类型安全包装 |
| **Compile-time** | TypeScript 严格，禁止 any | 签名变更导致编译错误 |
| **Runtime** | 监控、告警、扫描 | 检测孤儿实体 |
| **Test Coverage** | E2E 测试、集成测试 | 验证完整流程 |
| **Code Review** | 检查清单、PR 模板 | "你检查 X 了吗？" |

#### 4. 系统性扩展

- **类似问题**: 其他地方是否可能有同样问题？
- **设计缺陷**: 是否存在根本性的架构问题？
- **流程缺陷**: 开发流程是否需要改进？
- **知识缺口**: 团队是否缺少某些理解？

#### 5. 知识捕获

- [ ] 更新 `.spec-first/spec/guides/` 思考指南
- [ ] 更新 `.spec-first/spec/backend/` 或 `frontend/` 文档
- [ ] 创建问题记录（如适用）
- [ ] 创建根因修复的功能工单
- [ ] 更新检查命令（如需要）

## 5. 分析后：立即行动

**重要**: 完成分析后，必须立即：

1. **更新 spec/guides** - 不要只列出 TODO，实际更新相关文件：
   - 跨平台问题 → `cross-platform-thinking-guide.md`
   - 跨层问题 → `cross-layer-thinking-guide.md`
   - 代码复用问题 → `code-reuse-thinking-guide.md`
   - 领域特定 → `backend/*.md` 或 `frontend/*.md`

2. **同步模板** - 更新 `.spec-first/spec/` 后，同步到 `src/templates/markdown/spec/`

3. **提交 spec-first 更新** - 这是主要输出，不仅是分析文本

> **分析如果留在聊天中就毫无价值。价值在于更新的 spec。**

## 6. 使用场景

### 6.1 典型工作流

```
Development Flow:
  Write code -> Test -> /spec:finish-work -> git commit -> /spec:record-session

Debug Flow:
  Hit bug -> Fix -> /spec:break-loop -> Knowledge capture
                       |
                  Deep analysis
                       |
                  Update specs
```

### 6.2 触发时机

- 修复了一个复杂的 Bug
- 多次尝试才成功修复
- Bug 涉及跨层问题
- 发现了系统性问题

## 7. 设计分析

### 7.1 知识固化流程

```
Bug 发生
    ↓
修复 Bug
    ↓
break-loop 分析
    ├── 根因分类
    ├── 失败分析
    ├── 预防机制
    ├── 系统扩展
    └── 知识捕获
    ↓
更新 spec 文档
    ↓
知识固化到系统
```

### 7.2 与其他命令的关系

| 命令 | 时机 | 目的 |
|------|------|------|
| `finish-work` | 提交前 | 检查工作完整性 |
| `break-loop` | 调试后 | 深度分析和知识捕获 |
| `record-session` | 提交后 | 记录会话 |

---

## 8. 最佳实践

### 8.1 分析深度

- 不要停留在"怎么修复"
- 必须回答"如何防止"
- 必须更新规范文档

### 8.2 常见错误

| 错误 | 正确做法 |
|------|---------|
| 只记录分析，不更新 spec | 必须实际更新文件 |
| 分析停留在表面 | 深入到系统性问题 |
| 忘记同步模板 | 更新后同步到 templates |

---

## 9. 总结

**break-loop** 是知识提取工具：

```
Bug → break-loop → 知识固化
         │
         ├── 5 维度分析
         ├── 3 层洞察
         └── spec 更新
```

**核心价值**:
- 打破重复 Bug 循环
- 提取系统性知识
- 固化到规范文档
- 提升团队认知

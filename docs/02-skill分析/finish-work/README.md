# Finish-Work 深度分析

> 源文件: `/packages/cli/src/templates/claude/commands/spec/finish-work.md`

---

## 1. Skill 概述

### 1.1 核心定位

**finish-work** 是提交前检查清单，确保工作完整性。

| 维度 | 描述 |
|------|------|
| **目标** | 验证工作完整性 |
| **触发时机** | 代码写完并测试后，提交前 |
| **输出** | 检查清单通过/不通过 |

### 1.2 核心原则

```
┌─────────────────────────────────────────────────────────────┐
│                    核心原则                                  │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   交付不仅是代码，还包括文档、验证和知识捕获                │
│                                                             │
│   Complete work = Code + Docs + Tests + Verification        │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## 2. 检查清单详解

### 2.1 清单结构

```
┌─────────────────────────────────────────────────────────────┐
│                   finish-work 检查清单                       │
└─────────────────────────────────────────────────────────────┘

  1. Code Quality          ← 代码质量
     ├── lint
     ├── type-check
     ├── test
     └── 禁止模式检查

  1.5. Test Coverage       ← 测试覆盖
     ├── 新函数 → 单元测试
     ├── Bug 修复 → 回归测试
     └── 行为变更 → 集成测试

  2. Code-Spec Sync        ← 规范同步
     ├── backend spec 更新?
     ├── frontend spec 更新?
     └── guides 更新?

  2.5. Code-Spec Hard Block ← 基础设施/跨层强制检查
     ├── 可执行规范（签名/契约）
     ├── 验证和错误矩阵
     └── Good/Base/Bad 用例

  3. API Changes           ← API 变更
  4. Database Changes      ← 数据库变更
  5. Cross-Layer Verify    ← 跨层验证
  6. Manual Testing        ← 人工测试
```

### 2.2 各部分详解

#### 1. Code Quality

```bash
# 必须通过
pnpm lint
pnpm type-check
pnpm test
```

检查项：
- [ ] `pnpm lint` 通过（0 错误）
- [ ] `pnpm type-check` 通过（无类型错误）
- [ ] 测试通过
- [ ] 无 `console.log`（使用 logger）
- [ ] 无非空断言（`x!` 操作符）
- [ ] 无 `any` 类型

#### 1.5. Test Coverage

| 变更类型 | 测试要求 |
|---------|---------|
| 新纯函数 | 添加单元测试 |
| Bug 修复 | 在 `test/regression.test.ts` 添加回归测试 |
| 修改 init/update 行为 | 添加/更新集成测试 |
| 无逻辑变更（仅文本/数据） | 无需测试 |

#### 2. Code-Spec Sync

关键问题：
> "如果我修复了一个 bug 或发现了非显而易见的事情，是否应该记录下来，以便未来的我（或其他人）不会遇到同样的问题？"

如果 YES → 更新相关 code-spec 文档。

#### 2.5. Code-Spec Hard Block

**适用于**: 基础设施或跨层契约变更

强制检查：
- [ ] 规范内容是可执行的（真实签名/契约），而非仅原则性文本
- [ ] 包含文件路径 + 命令/API 名称 + 负载字段名
- [ ] 包含验证和错误矩阵
- [ ] 包含 Good/Base/Bad 用例
- [ ] 包含所需测试和断言点

**阻塞规则**: 在 pipeline 模式下，finish agent 会自动检测并执行 spec-first 更新。

#### 3. API Changes

如果修改了 API 端点：
- [ ] 输入 schema 更新？
- [ ] 输出 schema 更新？
- [ ] API 文档更新？
- [ ] 客户端代码同步更新？

#### 4. Database Changes

如果修改了数据库 schema：
- [ ] 创建迁移文件？
- [ ] 更新 schema 文件？
- [ ] 更新相关查询？
- [ ] 更新种子数据（如适用）？

#### 5. Cross-Layer Verification

如果变更跨越多层：
- [ ] 数据正确流经所有层？
- [ ] 错误处理在每个边界都有效？
- [ ] 类型在各层间一致？
- [ ] 加载状态已处理？

#### 6. Manual Testing

- [ ] 功能在浏览器/应用中工作？
- [ ] 边缘情况已测试？
- [ ] 错误状态已测试？
- [ ] 页面刷新后仍工作？

## 3. 快速检查流程

```bash
# 1. 代码检查
pnpm lint && pnpm type-check

# 2. 查看变更
git status
git diff --name-only

# 3. 根据变更文件检查相关项目
```

---

## 4. 常见疏忽

| 疏忽 | 后果 | 检查方式 |
|------|------|---------|
| code-spec 文档未更新 | 其他人不知道变更 | 检查 `.spec-first/spec/` |
| 规范仅是抽象文本 | 基础设施/跨层变更容易回归 | 要求签名/契约/矩阵/用例/测试 |
| 未创建迁移 | Schema 不同步 | 检查 `db/migrations/` |
| 类型未同步 | 运行时错误 | 检查共享类型 |
| 测试未更新 | 虚假信心 | 运行完整测试套件 |
| console.log 残留 | 生产日志嘈杂 | 搜索 console.log |

## 5. 与其他命令的关系

```
Development Flow:
  Write code -> Test -> /spec:finish-work -> git commit -> /spec:record-session
                          |                              |
                   Ensure completeness              Record progress

Debug Flow:
  Hit bug -> Fix -> /spec:break-loop -> Knowledge capture
                       |
                  Deep analysis
```

| 命令 | 时机 | 目的 |
|------|------|------|
| `finish-work` | 提交前 | 检查工作完整性 |
| `record-session` | 提交后 | 记录会话和提交 |
| `break-loop` | 调试后 | 深度分析 |

## 6. 设计分析

### 6.1 分层检查策略

```
┌─────────────────────────────────────────────────────────────┐
│                    检查层次                                  │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   Layer 1: 代码质量 (lint, type, test)                      │
│      ↓                                                      │
│   Layer 2: 测试覆盖 (单元, 回归, 集成)                      │
│      ↓                                                      │
│   Layer 3: 规范同步 (spec 更新)                             │
│      ↓                                                      │
│   Layer 4: 强制阻塞 (基础设施/跨层)                         │
│      ↓                                                      │
│   Layer 5: 特定检查 (API, DB, 跨层)                         │
│      ↓                                                      │
│   Layer 6: 人工验证 (手动测试)                              │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 6.2 硬阻塞机制

**触发条件**: 基础设施或跨层契约变更

**强制要求**:
1. 规范必须是可执行的
2. 必须包含具体契约
3. 必须有验证矩阵
4. 必须有用例
5. 必须有测试

**目的**: 防止抽象规范导致的回归

---

## 7. 最佳实践

### 7.1 使用时机

- 代码写完后
- 自测通过后
- git commit 前

### 7.2 检查顺序

1. 先运行自动化检查（lint, type, test）
2. 查看变更文件列表
3. 根据变更类型检查对应项目
4. 最后进行人工验证

---

## 8. 总结

**finish-work** 是工作交付的最后一道关卡：

```
代码 + 测试
    ↓
finish-work
    ├── 代码质量 ✓
    ├── 测试覆盖 ✓
    ├── 规范同步 ✓
    ├── 特定检查 ✓
    └── 人工验证 ✓
    ↓
可以提交
```

**核心价值**:
- 确保工作完整性
- 防止遗漏关键步骤
- 同步规范文档
- 知识捕获

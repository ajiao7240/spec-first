# Check 深度分析

> 源文件: `/packages/cli/src/templates/claude/commands/spec/check.md`

---

## 1. Skill 概述

### 1.1 核心定位

**check** 是一个代码质量验证命令，用于检查刚写的代码是否符合开发指南。

| 维度 | 描述 |
|------|------|
| **目标** | 验证代码符合 spec 规范 |
| **触发时机** | 编写代码后、提交前 |
| **输出** | 违规报告 + 修复建议 |

### 1.2 与其他命令的关系

```
开发流程中的位置:

before-dev → [写代码] → check → finish-work → commit → record-session
   ↑                          ↑
   │                          └── 验证阶段
   └── 准备阶段
```

## 2. 执行流程

### 2.1 流程图

```
┌─────────────────────────────────────────────────────────────┐
│                    check 执行流程                            │
└─────────────────────────────────────────────────────────────┘

  ┌─────────────────┐
  │ 1. 识别变更文件  │
  │ git diff        │
  └────────┬────────┘
           │
           ▼
  ┌─────────────────┐
  │ 2. 确定适用的    │
  │    spec 模块    │
  └────────┬────────┘
           │
           ▼
  ┌─────────────────┐
  │ 3. 读取 spec    │
  │    索引文件     │
  └────────┬────────┘
           │
           ▼
  ┌─────────────────┐
  │ 4. 读取具体      │
  │    指南文件     │
  └────────┬────────┘
           │
           ▼
  ┌─────────────────┐
  │ 5. 运行 lint    │
  │    和 typecheck │
  └────────┬────────┘
           │
           ▼
  ┌─────────────────┐
  │ 6. 报告违规     │
  │    并修复       │
  └─────────────────┘
```

### 2.2 步骤详解

#### Step 1: 识别变更文件

```bash
git diff --name-only HEAD
```

获取本次会话中修改的文件列表。

#### Step 2: 确定适用的 spec 模块

```bash
python3 ./.spec-first/scripts/get_context.py --mode packages
```

根据文件路径判断需要检查哪些 spec 模块。

#### Step 3: 读取 spec 索引

```bash
cat .spec-first/spec/<package>/<layer>/index.md
```

**重点**: 关注索引中的 **"Quality Check"** 部分。

#### Step 4: 读取具体指南文件

**关键原则**: 索引不是目标，它指向实际的指南文件。

```
index.md → quality-guidelines.md
        → conventions.md
        → 其他相关文件
```

#### Step 5: 运行代码检查

```bash
pnpm lint        # ESLint
pnpm typecheck   # TypeScript
```

#### Step 6: 报告并修复

发现违规时：
1. 报告违规内容
2. 提供修复建议
3. 直接修复（如果可行）

## 3. 设计分析

### 3.1 核心设计理念

```
┌─────────────────────────────────────────────────────────────┐
│                      设计理念                                │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   索引是导航 → 指向实际的指南文件                           │
│   索引 ≠ 规范内容                                           │
│                                                             │
│   正确: index.md → quality-guidelines.md → 检查代码         │
│   错误: 只读 index.md 就开始检查                            │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 3.2 检查维度

| 维度 | 检查方式 | 工具 |
|------|---------|------|
| **代码风格** | 读取 conventions.md | ESLint |
| **类型安全** | 读取 type-safety.md | TypeScript |
| **质量标准** | 读取 quality-guidelines.md | 手动审查 |
| **禁止模式** | 读取 forbidden-patterns | 手动审查 |

## 4. 使用场景

### 4.1 触发时机

| 时机 | 说明 |
|------|------|
| **写完代码后** | 验证代码符合规范 |
| **提交前** | 作为 finish-work 的一部分 |
| **代码审查时** | 辅助审查工具 |

### 4.2 典型工作流

```
1. /spec:before-dev     ← 读取规范
2. [编写代码]
3. /spec:check          ← 验证代码
4. 修复违规
5. /spec:finish-work    ← 最终检查
6. git commit
7. /spec:record-session
```

---

## 5. 与相关命令对比

| 命令 | 时机 | 目的 |
|------|------|------|
| `before-dev` | 写代码前 | 理解规范 |
| `check` | 写代码后 | 验证合规 |
| `finish-work` | 提交前 | 完整检查清单 |

---

## 6. 最佳实践

### 6.1 检查清单

- [ ] 运行 `git diff` 确认变更范围
- [ ] 读取正确的 spec 索引
- [ ] **不要只读索引** - 继续读取具体指南文件
- [ ] 运行 lint 和 typecheck
- [ ] 修复所有违规

### 6.2 常见错误

| 错误 | 正确做法 |
|------|---------|
| 只读 index.md 就开始检查 | 继续读取指南文件 |
| 跳过 lint/typecheck | 必须运行 |
| 忽略违规报告 | 修复后再继续 |

---

## 7. 总结

**check** 是代码质量守门员：

```
代码 ──▶ check ──▶ 符合规范的代码
         │
         └── 检查: spec规范 + lint + typecheck
```

**核心价值**:
- 确保代码符合项目规范
- 自动化静态检查
- 及早发现质量问题

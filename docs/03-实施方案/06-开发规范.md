# 开发规范

## 1. 概述

本文档定义 spec-first 框架的开发规范和最佳实践。

## 2. 代码组织

### 2.1 目录规范

```
spec-first/
├── skills/           # 每个 skill 独立目录
├── agents/           # 每个 agent 独立目录
├── scripts/          # 工具脚本
├── templates/        # 模板文件
├── resources/        # 资源文件
└── tests/            # 测试文件
```

### 2.2 命名规范

**Skills**：
- 目录名：`spec-<name>`
- 文件名：`skill.md`
- Skill 名称：`spec-<name>`

**Agents**：
- 目录名：`<category>/<agent-name>`
- 文件名：`agent.md`

**Scripts**：
- 文件名：`<name>.sh`
- 函数名：`snake_case`

## 3. Markdown 规范

### 3.1 Skill 文件结构

```markdown
---
name: spec-<name>
description: <简短描述>
parameters:
  - name: <param>
    type: string
    required: false

[契约检查逻辑]

[核心逻辑]

## 输出要求
[输出规范]

## 状态更新
[状态更新逻辑]
```

### 3.2 Frontmatter 规范

**task.yaml**：
```yaml
task_id: YYYY-MM-DD-<slug>
title: <标题>
role: product|dev|fullstack|test|generic
level: L1|L2|L3
current_stage: brainstorm|plan|work|review|compound
status: pending|in_progress|blocked|completed|failed
created_at: <ISO8601>
updated_at: <ISO8601>
```

**Review 产物**：
```yaml
---
task_id: <task-id>
stage: review
scores:
  correctness: 1-5
  completeness: 1-5
  executability: 1-5
  reusability: 1-5
summary_score: <total>
blocking_issues: <count>
pass: true|false
---
```

## 4. Shell Script 规范

### 4.1 脚本模板

```bash
#!/bin/bash
set -e
set -u

# 函数定义
function_name() {
    local param1=$1
    local param2=${2:-default}

    # 实现逻辑
}

# 主逻辑
main() {
    # 参数解析
    while [[ $# -gt 0 ]]; do
        case $1 in
            --param)
                param="$2"
                shift 2
                ;;
            *)
                echo "Unknown: $1" >&2
                exit 1
                ;;
        esac
    done

    # 执行
    function_name "$param"
}

main "$@"
```

### 4.2 错误处理

```bash
# 检查文件存在
if [ ! -f "$file" ]; then
    echo "Error: $file not found" >&2
    exit 1
fi

# 检查命令执行
if ! command -v tool &> /dev/null; then
    echo "Error: tool not installed" >&2
    exit 1
fi
```

### 4.3 日志输出

```bash
# 标准输出
echo "Info: Processing..."

# 错误输出
echo "Error: Failed" >&2

# 调试输出（可选）
[ "$DEBUG" = "1" ] && echo "Debug: $var"
```

## 5. 开发流程

### 5.1 新增 Skill

1. 创建目录：`skills/spec-<name>/`
2. 创建 `skill.md`
3. 增加阶段契约检查
4. 实现核心逻辑
5. 增加状态更新
6. 编写测试
7. 更新文档

### 5.2 修改 Skill

1. 阅读现有实现
2. 确认改动范围
3. 修改代码
4. 更新测试
5. 验证功能

### 5.3 新增 Script

1. 创建 `scripts/<name>.sh`
2. 实现核心函数
3. 增加错误处理
4. 编写测试
5. 更新文档

## 6. 测试规范

### 6.1 测试覆盖

- 每个脚本函数必须有单元测试
- 每个 skill 必须有集成测试
- 关键路径必须有 E2E 测试

### 6.2 测试命名

```bash
# 测试文件
tests/test_<component>.sh

# 测试函数
test_<function>_<scenario>()
```

## 7. 文档规范

### 7.1 代码注释

```bash
# 函数说明
# 参数：$1 - task_id
# 返回：0 成功，1 失败
function_name() {
    # 实现
}
```

### 7.2 README 更新

每次重要改动需更新：
- 功能说明
- 使用示例
- 变更日志

## 8. Git 规范

### 8.1 分支管理

- `main`：稳定版本
- `develop`：开发版本
- `feature/<name>`：功能分支
- `fix/<name>`：修复分支

### 8.2 提交信息

```
<type>: <subject>

<body>

<footer>
```

**Type**：
- `feat`：新功能
- `fix`：修复
- `docs`：文档
- `refactor`：重构
- `test`：测试

**示例**：
```
feat: 增加 spec-brainstorm skill

- 增加阶段契约检查
- 修改输出路径
- 增加 rationale capture
```

## 9. 版本管理

### 9.1 语义化版本

- `MAJOR.MINOR.PATCH`
- `1.0.0`：首个稳定版本
- `1.1.0`：新增功能
- `1.1.1`：修复 bug

### 9.2 发布流程

1. 更新版本号
2. 更新 CHANGELOG
3. 创建 git tag
4. 发布到 npm

## 10. 总结

本文档定义了开发规范。

**核心规范**：
- 代码组织：清晰的目录结构
- 命名规范：统一的命名约定
- 文档规范：完善的注释和文档
- 测试规范：充分的测试覆盖
- Git 规范：规范的提交和分支管理

# 测试方案

## 1. 概述

本文档定义 spec-first 框架的测试策略和测试用例。

## 2. 测试层次

### 2.1 测试金字塔

```
        /\
       /  \      E2E 测试（少量）
      /----\
     /      \    集成测试（适量）
    /--------\
   /          \  单元测试（大量）
  /------------\
```

### 2.2 测试范围

| 测试层次 | 测试对象 | 测试数量 | 优先级 |
|---------|---------|---------|--------|
| 单元测试 | 脚本函数 | 多个 | P0 |
| 集成测试 | Skills 流程 | 5个 | P0 |
| E2E 测试 | 完整闭环 | 1-2个 | P1 |

## 3. 单元测试

### 3.1 task-manager.sh 测试

**测试用例**：

```bash
# 测试创建任务
test_create_task() {
    ./scripts/task-manager.sh create \
        --task-id "2026-03-29-test" \
        --title "测试任务" \
        --role "dev" \
        --level "L2"

    # 验证
    assert_file_exists ".claude/tasks/2026-03-29-test/task.yaml"
    assert_contains ".claude/tasks/2026-03-29-test/task.yaml" "task_id: 2026-03-29-test"
}

# 测试读取任务
test_read_task() {
    local output=$(./scripts/task-manager.sh read --task-id "2026-03-29-test")
    assert_contains "$output" "task_id: 2026-03-29-test"
}

# 测试更新任务
test_update_task() {
    ./scripts/task-manager.sh update \
        --task-id "2026-03-29-test" \
        --stage "plan" \
        --status "in_progress"

    assert_contains ".claude/tasks/2026-03-29-test/task.yaml" "current_stage: plan"
}
```

### 3.2 stage-gate.sh 测试

**测试用例**：

```bash
# 测试 brainstorm 阶段（无依赖）
test_stage_gate_brainstorm() {
    ./scripts/stage-gate.sh check --task-id "2026-03-29-test" --stage "brainstorm"
    assert_exit_code 0
}

# 测试 plan 阶段（需要 brainstorm 产物）
test_stage_gate_plan_without_upstream() {
    ./scripts/stage-gate.sh check --task-id "2026-03-29-test" --stage "plan"
    assert_exit_code 1
}

test_stage_gate_plan_with_upstream() {
    touch .claude/tasks/2026-03-29-test/01-brainstorm.md
    ./scripts/stage-gate.sh check --task-id "2026-03-29-test" --stage "plan"
    assert_exit_code 0
}
```

### 3.3 review-judge.sh 测试

**测试用例**：

```bash
# 测试关键维度底线
test_review_judge_low_correctness() {
    local output=$(./scripts/review-judge.sh judge --correctness 2 --completeness 4 --executability 4 --reusability 3 --blocking 0)
    assert_contains "$output" "pass: false"
}

# 测试阻断问题
test_review_judge_blocking_issues() {
    local output=$(./scripts/review-judge.sh judge --correctness 4 --completeness 4 --executability 4 --reusability 4 --blocking 1)
    assert_contains "$output" "pass: false"
}

# 测试通过场景
test_review_judge_pass() {
    local output=$(./scripts/review-judge.sh judge --correctness 4 --completeness 4 --executability 4 --reusability 4 --blocking 0)
    assert_contains "$output" "pass: true"
}
```

## 4. 集成测试

### 4.1 spec-brainstorm 集成测试

**测试场景**：完整执行 brainstorm skill

**测试步骤**：
1. 初始化任务
2. 执行 /spec:brainstorm
3. 验证产物生成
4. 验证 task.yaml 更新

**验证点**：
- 01-brainstorm.md 已生成
- 包含候选方案
- 包含推荐方案与理由
- task.yaml 的 current_stage 为 plan

### 4.2 spec-plan 集成测试

**测试场景**：完整执行 plan skill

**前置条件**：01-brainstorm.md 已存在

**验证点**：
- 02-plan.md 已生成
- 包含实施步骤
- 包含方案选择理由
- task.yaml 的 current_stage 为 work

### 4.3 spec-work 集成测试

**测试场景**：完整执行 work skill

**前置条件**：02-plan.md 已存在

**验证点**：
- 03-work.md 已生成
- 包含实施记录
- task.yaml 的 current_stage 为 review

### 4.4 spec-review 集成测试

**测试场景**：完整执行 review skill

**前置条件**：03-work.md 已存在

**验证点**：
- 04-review.md 已生成
- frontmatter 包含评分
- frontmatter 包含 pass 字段
- pass=true 时 current_stage 为 compound
- pass=false 时 current_stage 保持 work

### 4.5 spec-compound 集成测试

**测试场景**：完整执行 compound skill

**前置条件**：04-review.md 已存在且 pass=true

**验证点**：
- 05-compound.md 已生成
- 资产已迁移到 .claude/assets/
- 资产文件包含正确的 frontmatter
- task.yaml 的 status 为 completed

## 5. E2E 测试

### 5.1 完整五步闭环测试

**测试场景**：从 brainstorm 到 compound 的完整流程

**测试步骤**：
1. 执行 /spec:brainstorm --role fullstack --level L2
2. 执行 /spec:plan
3. 执行 /spec:work
4. 执行 /spec:review
5. 执行 /spec:compound

**验证点**：
- 所有阶段产物已生成
- task.yaml 状态正确流转
- 资产已正确沉淀

### 5.2 Review 不通过场景测试

**测试场景**：Review 不通过后的重做流程

**测试步骤**：
1. 执行前三步（brainstorm → plan → work）
2. 执行 review，模拟不通过场景
3. 修改 work 产物
4. 重新执行 review
5. 执行 compound

**验证点**：
- Review 不通过时 current_stage 保持 work
- 可以重新执行 work
- 重新 review 后可以继续

## 6. 测试工具

### 6.1 断言函数

```bash
assert_file_exists() {
    if [ ! -f "$1" ]; then
        echo "FAIL: File $1 does not exist"
        exit 1
    fi
}

assert_contains() {
    if ! grep -q "$2" "$1"; then
        echo "FAIL: $1 does not contain $2"
        exit 1
    fi
}

assert_exit_code() {
    if [ $? -ne $1 ]; then
        echo "FAIL: Expected exit code $1"
        exit 1
    fi
}
```

### 6.2 测试框架

使用 bats（Bash Automated Testing System）或简单的 bash 脚本。

**示例测试脚本**：
```bash
#!/bin/bash
# tests/test_task_manager.sh

source tests/assert.sh

# 设置
setup() {
    rm -rf .claude/tasks/test-*
}

# 清理
teardown() {
    rm -rf .claude/tasks/test-*
}

# 运行测试
setup
test_create_task
test_read_task
test_update_task
teardown

echo "All tests passed"
```

```bash

./tests/run_all_tests.sh

./tests/run_unit_tests.sh

# 运行集成测试
./tests/run_integration_tests.sh
```

### 8.2 CI/CD 集成

在 GitHub Actions 中运行测试：
```yaml
name: Test
on: [push, pull_request]
jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v2
      - name: Run tests
        run: ./tests/run_all_tests.sh
```

## 1. 概述

### 1.1 项目背景

本项目基于对四篇外部开源项目的深度分析，识别出可借鉴到 Spec-First 的机制要素，并转化为具体的开发任务。分析涵盖：

| 来源项目 | 分析报告版本 | 核心价值 |
|---------|-------------|----------|
| Superpowers | v4.3.1 | AI 行为约束工程 |
| Planning-with-Files | v2.16.0 | 上下文/注意力工程 |
| Spec-Kit | v0.1.6 | 规范质量工程 |
| Oh-My-OpenCode | v3.8.5 | 执行可靠性工程 |

### 1.2 目标

- 短期（P0）：通过 7 项配置层改动，零 TS 运行时代码变更，显著提升 AI 执行纪律
- 中期（P1）：通过 21 项行动，补齐多 TASK 执行、规范质量、执行可靠性等能力

### 1.3 范围

| 优先级 | 任务数 | 工作量 | 覆盖率 |
|--------|-------|--------|--------|
| P0 | 7 项 | 4.0-4.5 人天 | 100% |
| P1 | 21 项 | ~25 人天（含待细化项与联调缓冲） | 100% |

### 1.4 设计原则

1. **最小侵入**：优先使用 Prompt/Hook 配置层改动，避免大规模代码重构
2. **Hook 优先于指令**：能用 Hook 自动化的不依赖 AI 自觉遵守
3. **渐进验证**：先在 Daily Path 验证，再推广到全阶段
4. **可量化反馈**：通过 C7/C8/C9 合规率指标量化改进效果

### 1.5 任务执行管理字段（最佳实践）

每个任务卡除“优先级/类型/来源/工作量/依赖”外，实施时必须补齐以下字段：

- **Owner**: 负责人（单人）
- **看板状态**: Not Started / In Progress / Blocked / Done（项目执行看板）
- **任务状态**: planned / in_progress / blocked / complete / verified（`task_plan.md` 任务明细）
- **开始/截止**: 计划开始与目标完成日期
- **证据链接**: PR、提交记录、验证日志或截图路径
- **回滚方案**: 配置回退步骤与影响范围

### 1.6 核实结论驱动原则（来自 19 Skill 审查）

仅将“已核实真问题”纳入执行面，避免误报驱动开发偏航：

- 统计口径：真 19 / 假 18 / 已修复 1
- 执行约束：真问题必须映射到具体任务卡与验收证据
- 去噪约束：假问题不入排期；已修复项只做回归检查不重复开发

**目标文件**: `.claude/settings.json` + `.spec-first/hooks/task-context.sh`

**改动内容**:
```json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "write|edit|create",
        "hooks": [
          {
            "type": "command",
            "command": "sh .spec-first/hooks/task-context.sh"
          }
        ]
      }
    ]
  }
}
```

```bash
#!/usr/bin/env sh
# .spec-first/hooks/task-context.sh
set -eu

FEAT="$(head -1 .spec-first/current 2>/dev/null || true)"
FILE="specs/$FEAT/task_plan.md"
[ -n "$FEAT" ] && [ -f "$FILE" ] || exit 0

echo "=== TASK 上下文刷新 ==="
awk -F'|' '
  /^\|/ {
    first=""; last=""; title=""
    for (i=1; i<=NF; i++) {
      c=$i
      gsub(/^[ \t]+|[ \t]+$/, "", c)
      if (c != "") {
        if (first == "") {
          first=c
          if (i+1<=NF) {
            title=$(i+1)
            gsub(/^[ \t]+|[ \t]+$/, "", title)
          }
        }
        last=c
      }
    }
    if (first ~ /^TASK-/) {
      s=tolower(last)
      if (s=="in_progress" || s=="in progress") {
        printf("Current TASK: %s | %s | %s\n", first, title, last)
        found=1
      }
    }
  }
  END {
    if (!found) print "Current TASK: (no in_progress task found)"
  }
' "$FILE"
```

> 合并要求：仅追加 `hooks.PreToolUse` 条目，不覆盖既有 `PostToolUse`/`Stop` 条目。
> 双宿主策略：Claude 走 Hook 注入；Codex 不注入宿主 Hook，依赖 T002/T003/T006 的 Skill 约束补偿。
> 安装场景要求：若 `.claude/settings.json` 不存在需先创建；若 `.spec-first/config.yaml` 缺失需通过 `init/setup` 补齐并记录证据。

```markdown
## 反合理化守卫

当你产生以下念头时，立即停止并回到流程：

| AI 的借口 | 封堵 |
|-----------|------|
| "这个改动太小，不需要走 code-review" | 小改动也有回归风险，review 耗时 < 2 分钟 |
| "我已经手动检查过了" | 手动检查 ≠ 自动校验证据 |
| "先写完再补测试" | 事后测试证明不了什么，TDD 先行 |
| "这只是重构，不影响功能" | 重构不改行为 ≠ 重构不引入 bug |
| "快速修一下，之后再调查" | 快速修复掩盖根因，系统化调试更快 |
| "我看到问题了，让我直接修" | 看到症状 ≠ 理解根因 |
| "同时改多处，一起测试" | 无法隔离哪个改动有效，会引入新 bug |
| "再试一次修复"（已失败 2+ 次） | 3 次失败 = 架构问题，停止修复，质疑设计 |
| "我记得刚才看到的内容" | 上下文会被压缩，记忆不可靠，写入文件 |

## 上下文持久化规则

Context Window = RAM（易失、有限），Filesystem = Disk（持久、无限）。
任何重要的东西都写入磁盘。

| 场景 | 动作 | 原因 |
|------|------|------|
| 刚写完一个文件 | 不要读 | 内容还在上下文中 |
| 查看了图片/PDF/网页 | 立即写 findings.md | 多模态内容压缩时丢失 |
| 浏览器/MCP 返回数据 | 写入文件 | 外部数据不持久 |
| 开始新 TASK | 读 task_plan.md + findings.md | 重新定向上下文 |
| 发生错误 | 读相关文件 | 需要当前状态来修复 |
| 间隔后恢复 | 读所有运行态文件 | 恢复状态 |

压缩必须可恢复：即使丢弃了内容，也要保留 URL / 文件路径 / ID 指针。

## 调试流程（测试失败时）

铁律：NO FIXES WITHOUT ROOT CAUSE INVESTIGATION FIRST

| 阶段 | 关键活动 | 成功标准 |
|------|---------|---------|
| 1. 根因调查 | 读错误、复现、检查近期变更、追踪数据流 | 理解 WHAT 与 WHY |
| 2. 模式分析 | 找到可工作的类似实现并对比差异 | 差异点可定位 |
| 3. 假设验证 | 单一假设、最小实验、一次只改一个变量 | 假设被确认或被否定 |
| 4. 实现修复 | 先失败测试，再单一修复，再全量验证 | 问题关闭且无回归 |

硬规则：同类修复失败 3 次后必须停止并升级到架构审查。
```

| 属性 | 值 |

**目标文件**: `skills/spec-first/12-verify/SKILL.md` + `skills/spec-first/13-orchestrate/SKILL.md`

**改动内容**:

```markdown
## 证据铁律

铁律：声称任何阶段通过前，必须提供新鲜的验证证据。

五步 Gate Function：
1. **IDENTIFY** — 什么命令能证明这个声明？
2. **RUN** — 执行完整命令（新鲜的、完整的）
3. **READ** — 完整输出，检查退出码，计数失败项
4. **VERIFY** — 输出是否确认了声明？
5. **ONLY THEN** — 发出声明

禁止使用的表述：
- "should pass" / "looks good" / "已完成" / "我检查过了"
- "应该没问题" / "上一轮通过了" / "差不多了"

## Common Failures 表

| 声明 | 需要的证据 | 不充分的证据 |
|------|-----------|-------------|
| Gate 通过 | `spec-first gate check <featureId>` 输出: `PASS` 或 `PASS_WITH_WAIVER` | "我检查过了"、"应该没问题" |
| 覆盖率达标 | `spec-first metrics coverage <featureId>` 输出: C1-C9 ≥ 阈值 | "所有 FR 都有对应 TASK" |
| 阶段可推进 | `gate check` 退出码 0（`PASS/PASS_WITH_WAIVER`）+ `matrix check` 满足当前阶段策略 | "上一轮通过了" |
| TASK 完成 | 测试命令输出 + code-review 通过 | "代码写完了" |
| Feature 可归档 | `spec-first gate check <featureId>` + `spec-first matrix check <featureId>` + 归档产物证据 | "所有 TASK 都标记完成了" |

## 量化通过条件（补强）

- Gate 通过：`spec-first gate check <featureId>` 退出码为 0，且状态为 `PASS` 或 `PASS_WITH_WAIVER`
- 阶段可推进：`gate check` 通过；`matrix check` 按阶段策略执行（`01_specify` 可仅诊断，`03_plan` 起建议收敛到退出码 0）
- 覆盖率可接受：`metrics coverage` 结果满足当前阶段阈值（由 stage-state/mergedRules 定义）
- 结论可宣告：以上证据均为本次会话“新鲜执行”结果（非历史缓存）

## 判定证据链要求

- 失败条目必须映射到具体 ID（FR/DS/TASK/TC）
- 每个失败条目必须附带至少 1 条可执行修复建议
```

#### T007 — Stop Hook 完成度守门（阻断型，command-based 默认）

**目标文件**: `.claude/settings.json` + `.spec-first/hooks/stop-guard.sh` + `skills/spec-first/AGENTS.md`

**改动内容（推荐：command-based，完整 JSON 示例）**:

```json
{
  "hooks": {
    "Stop": [
      {
        "hooks": [
          {
            "type": "command",
            "command": "sh .spec-first/hooks/stop-guard.sh"
          }

      }
    ]
  }
}
```

```bash
#!/usr/bin/env sh
# .spec-first/hooks/stop-guard.sh
set -eu

FEAT="$(head -1 .spec-first/current 2>/dev/null || true)"
FILE="specs/$FEAT/task_plan.md"
[ -z "$FEAT" ] || [ ! -f "$FILE" ] && exit 0

PENDING_IDS="$(
  awk -F'|' '
    /^\|/ {
      first=""; last=""
      for (i=1; i<=NF; i++) {
        c=$i
        gsub(/^[ \t]+|[ \t]+$/, "", c)
        if (c != "") {
          if (first == "") first=c
          last=c
        }
      }
      if (first !~ /^TASK-/) next
      s=tolower(last)
      if (s != "complete" && s != "done" && s != "verified") print first
    }
  ' "$FILE"
)"

if [ -n "$PENDING_IDS" ]; then
  echo "⚠️ 仍有未完成 TASK：" >&2
  echo "$PENDING_IDS" >&2
  exit 2
fi
```

**任务明细表解析契约（与 T009 联动）**:

- 首个非空单元格必须为 `TASK-*`（任务 ID）
- 最后非空单元格必须为任务状态（`planned/in_progress/blocked/complete/verified`）
- 若 T009 调整上述契约，必须与 T007 脚本适配同 PR 提交

> 合并要求：仅追加 `hooks.Stop` 条目，不替换现有 Stop Hook（如 `ai stats` 等运行态统计逻辑）。
> 兼容性要求：`Stop` 事件不使用 `matcher` 字段（事件不支持 matcher）。
> 守门策略：阻断型。检测到未完成 TASK 时必须非 0 退出，阻止会话“正常结束”。
> 输出要求：阻断原因必须写入 stderr（`>&2`），保证 Hook 失败信息可见。
> 双宿主策略：Claude 走 Stop Hook 阻断；Codex 继续通过 T003/T006 证据链与规则约束防“提前收工”。
> 文档同步要求：在 `skills/spec-first/AGENTS.md` 增加“双宿主核心约束摘要”（Claude Hook 阻断路径 / Codex Skill 证据链路径）。

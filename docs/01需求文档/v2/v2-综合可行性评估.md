## 核心结论

### v1.3 的判断依然成立

v1.3 的核心判断——"Spec-First v7.1 规范完备、交付部分就绪"——在 v2.1 中依然成立。M1/M2/M4/M6 已交付；M3/M5 需补齐，M7 为部分交付待稳健性增强。

### v2.1 的新增洞察

**四大业界项目的分析揭示了 v1.3 未覆盖的四个能力盲区**：

| 盲区 | 表现 | 补齐来源 | 影响 |
|------|------|---------|------|
| **AI 行为约束** | Skill 是指令式的，缺少逃逸封堵 | Superpowers | 流程遵从率不可控 |
| **上下文工程** | 无 Hook 级注意力刷新，catchup 被动触发 | Planning-with-Files | 长任务注意力衰减 |
| **执行可靠性** | 无续航、无完成检测、无编辑校验 | Oh-My-OpenCode | 多 TASK 执行不稳定 |
| **规范质量验证** | Gate 检查产物存在性，不检查质量 | Spec-Kit | 规范文档质量无保障 |

**核心公式**：

```
Spec-First v7.1 完整能力 =
  流程治理（已有）
  + AI 行为约束（Superpowers）
  + 上下文工程（Planning-with-Files）
  + 执行可靠性（Oh-My-OpenCode）
  + 规范质量验证（Spec-Kit）
```

### 量化预期（v2.1 更新）

| 指标 | v1.3 预期 | v2.1 预期（含借鉴） | 提升来源 | 度量方式 |
|------|----------|-------------------|---------|---------|
| Spec→PR 周期（Size S） | 3-5 天 | 2-4 天 | 续航执行 + 自迭代循环 | Feature 创建到 PR 合并的日历天数 |

| 规范文档质量分 | 未量化 | ≥80%（Spec Quality Score） | 英语单元测试 + 歧义消解 | spec-review skill 的 Checklist 通过率（Gate C10） |
| 长任务注意力保持率 | 未量化 | ≥85%（TASK 完成率） | PreToolUse Hook + Stop Hook | 单次 session 内 TASK 完成数 / TASK 总数 |
| 会话中断恢复 | 70-85% | 85-95% | 5-Question Test + Session Recovery | catchup 后 5-Question Test 通过率 |

## 六大能力主题与 42 项机制映射

> **分类原则**：42 项机制按**能力域**（而非来源项目）归类。同一来源的机制可能分布在不同主题下——例如 Superpowers 的"新鲜上下文隔离"归入上下文工程主题，PwF 的"3-Strike Error Protocol"归入执行可靠性主题。这确保每个主题内的机制在能力上内聚，便于按主题分批落地。

### 主题 1: AI 行为约束工程

**核心问题**：Spec-First 的 Skill 是指令式的——告诉 AI 该做什么，但没有预防 AI 不做什么。AI 会"合理化"地跳过纪律。

**来源**：Superpowers（主）+ Spec-Kit（辅）+ OmO（辅）

| # | 机制 | 来源 | 优先级 | 复杂度 | 解决的问题 |
|---|------|------|--------|--------|-----------|
| 1 | 反合理化设计（Red Flags + Rationalization Tables） | Superpowers | **P0** | 极低 | AI 绕过 review/测试/澄清的逃逸路径 |
| 2 | 证据优先于声明（Iron Law + Common Failures） | Superpowers | **P0** | 极低 | Gate 声明无证据支撑 |
| 3 | Description Trap 防护（description 仅触发条件） | Superpowers | **P0** | 极低 | AI 按 description 执行而非读完整 SKILL |
| 4 | 模板驱动 LLM 行为约束（WHAT/HOW 分离 + 不确定性标记） | Spec-Kit | **P1** | 极低 | AI 在 spec 阶段过早引入实现细节 |
| 5 | 意图门控 IntentGate（歧义检测 + 确认） | OmO | P3 | 低 | AI 字面误解用户指令 |

**落地策略**：在 `spec`、`code`、`verify` 三个核心 Skill 的 SKILL.md 中追加反合理化表和证据铁律段落。纯 Prompt 修改，零代码改动。

**P0 具体行动**：
1. `/spec-first:code` 加入反合理化表（封堵"跳过 review/测试"）
2. `/spec-first:verify` 加入证据铁律（Gate 声明必须附带执行证据）
3. `/spec-first:spec` 加入反合理化表（封堵"跳过澄清/AC 结构化"）
4. 制定 Common Failures 表（统一"什么算证据、什么不算"）
5. 巡检 19 个 Skill 的 description，收敛为触发条件 only

### 主题 2: 上下文工程

**核心问题**：Spec-First 在"AI 执行流程时如何保持注意力聚焦"上缺少机制。长任务中 AI 会遗忘目标、丢失发现、重复错误。

**来源**：Planning-with-Files（主）+ OmO（辅）+ Spec-Kit（辅）

| # | 机制 | 来源 | 优先级 | 复杂度 | 解决的问题 |
|---|------|------|--------|--------|-----------|
| 6 | PreToolUse Hook 注意力操控 | PwF | **P0** | 极低 | 长任务注意力衰减 |
| 7 | 文件系统即外部记忆（Read/Write 决策矩阵） | PwF | **P0** | 极低 | 重要信息未持久化 |
| 8 | 2-Action Rule 强制持久化 | PwF | **P1** | 极低 | 研究阶段信息丢失 |
| 9 | 5-Question Reboot Test | PwF | **P1** | 极低 | catchup 恢复质量无校验 |
| 10 | 动态 Prompt 组装（元数据驱动） | OmO | **P1** | 低 | Skill 维护成本高 |
| 11 | 新鲜上下文隔离（Fresh Context Per Task） | Superpowers | P2 | 中 | 多 TASK 上下文污染 |
| 12 | Session Recovery 自动恢复 | PwF | P2 | 中 | catchup 需手动触发 |
| 13 | KV-Cache 优化原则 | PwF | P2 | 低 | 运行效率（成本/延迟） |
| 14 | 层级化 AGENTS.md 自动生成 | OmO | P2 | 中 | AI 对项目结构理解不足 |
| 15 | Agent 上下文自动同步 | Spec-Kit | P2 | 低 | Agent 上下文与项目状态脱节 |

**落地策略**：P0 项仅需 Hook 配置 + Skill 文本修改，零代码改动，立即生效。P1 项需 SKILL.md 元数据扩展。P2 项需新增模块。

**P0 具体行动**：
1. 在 `/spec-first:code` 的 Hook 配置中加入 PreToolUse 注意力刷新（每次 Write/Edit/Bash 前读取 task_plan.md 前 30 行）
2. 在 `/spec-first:code` Skill 中嵌入 Read/Write 决策矩阵

### 主题 3: 执行可靠性工程

**核心问题**：Spec-First 有 Skill 框架但缺少生产级的 Agent 调度、续航、完成检测机制。AI 可能中途放弃、假完成、编辑腐蚀。

**来源**：Oh-My-OpenCode（主）+ PwF（辅）+ Superpowers（辅）

| # | 机制 | 来源 | 优先级 | 复杂度 | 解决的问题 |
|---|------|------|--------|--------|-----------|
| 16 | Todo 续航执行器 | OmO | **P1** | 中 | AI 中途放弃任务 |
| 17 | 自迭代开发循环（Ralph Loop） | OmO | **P1** | 中 | 多 TASK 需反复手动触发 |
| 18 | Stop Hook 完成度守门 | PwF | **P1** | 低 | AI 提前收工 |
| 19 | 3-Strike Error Protocol | PwF | **P1** | 极低 | 错误处理无章法 |
| 20 | 批量执行 + 人工检查点 | Superpowers | **P1** | 低 | 04 阶段节奏失控 |
| 21 | 系统化调试流程（四阶段 + 3 次失败硬规则） | Superpowers | P2 | 低 | 调试质量不可控 |
| 22 | Hash 锚定编辑（Hashline Edit） | OmO | P2 | 中 | 编辑腐蚀（stale-line corruption） |
| 23 | 双重完成检测 | OmO | P2 | 中 | 假完成误判 |

**落地策略**：P1 项中，Stop Hook 和 3-Strike 是纯 Skill 文本修改；续航执行器和自迭代循环需修改 phase-machine.ts 和 orchestrate skill。

**P1 具体行动**：
1. `/spec-first:code` 和 `/spec-first:orchestrate` 的 Stop Hook 加入完成度检查
2. `/spec-first:code` 嵌入 3-Strike Error Protocol
3. Phase Machine P4_WRITE 后检查剩余 TASK，自动续航（max 10）
4. `orchestrate` 增加 `--auto` 模式（自迭代直到所有 TASK 完成，max 20）
5. `orchestrate` 引入 batch checkpoint（Size S: 每 2-3 TASK 暂停，Size M: 每 2 TASK，Size L: 每 TASK）

### 主题 4: 规范质量工程

**核心问题**：Spec-First 的 Gate 检查产物是否存在，不检查产物质量是否达标。好的流程 + 差的规范 = 差的交付。

**来源**：Spec-Kit（主）+ Superpowers（辅）

| # | 机制 | 来源 | 优先级 | 复杂度 | 解决的问题 |
|---|------|------|--------|--------|-----------|
| 24 | "英语单元测试"式 Checklist | Spec-Kit | **P1** | 中 | 规范文档质量无保障 |
| 25 | 结构化歧义消解（10 类分类法 + 5 轮提问） | Spec-Kit | **P1** | 低 | spec 中歧义无系统性消解 |
| 26 | 两阶段审查（Spec Compliance → Code Quality） | Superpowers | **P1** | 低 | code-review 合规与质量混为一谈 |
| 27 | 跨产物一致性分析（6 类检测 + 4 级严重度） | Spec-Kit | P2 | 中 | spec/design/task 语义脱节 |
| 28 | Constitution 语义化版本管理 | Spec-Kit | P2 | 低 | 项目原则变更无追踪 |

**落地策略**：P1 项中，歧义消解和两阶段审查是 SKILL.md Prompt 增强；Checklist 需新增 `spec-review` skill。

**P1 具体行动**：
1. 在 `spec` skill 的 P2→P3 之间插入歧义扫描步骤（10 类分类法，最多标记 3 个 `[NEEDS CLARIFICATION]`）
2. 新增 `spec-review` skill，按 10 类检查维度生成 Checklist；Gate 新增 C10（Spec Quality Score ≥ 80%）
3. 将 `code-review` 拆为两轮：第一轮合规审查（对照 AC 逐条校验），第二轮质量审查（九维度）

### 主题 5: Hook 与工具链工程

**核心问题**：Spec-First 已有 AI Runtime / Session / Git Hook，但缺少统一分层治理（Tool-Guard、Continuation、Transform 等关注点未显式建模）；单点故障隔离仍不足。

**来源**：OmO（主）+ Superpowers（辅）

| # | 机制 | 来源 | 优先级 | 复杂度 | 解决的问题 |
|---|------|------|--------|--------|-----------|
| 29 | 安全 Hook 创建（safeCreateHook 包装） | OmO | **P1** | 极低 | 单个 Hook 异常崩溃整个插件 |
| 30 | Hook 跨平台稳健性（polyglot wrapper + 时序兜底） | Superpowers | **P1** | 低 | 首轮注入失效 |
| 31 | Session Hook 决策树（元技能 + 1% 规则） | Superpowers | **P1** | 低 | Skill 路由依赖 AI 自行判断 |
| 32 | Worktree First（实现前隔离工作区） | Superpowers | **P1** | 低 | 主分支误操作风险 |
| 33 | AI 注释质量检查（Comment Checker / Slop Detection） | OmO | P2 | 低 | AI 生成低质量注释 |
| 34 | 分层 Hook 架构（五层职责） | OmO | P2 | 高 | Hook 体系缺少关注点分离 |

**落地策略**：P1 项中，安全 Hook 包装约 30-50 LOC；Hook 跨平台需补充回归用例；Session Hook 决策树是 Prompt 修改。

**P1 具体行动**：
1. 为 `session-start.js`、`session-end.js` 增加 try-catch 包装；Git hooks 增加 `set +e` 降级逻辑
2. 为 session 级注入链补充跨平台回归用例（Linux/macOS/Windows）
3. session hook 注入的 bootstrap 内容包含 skill 优先级路由表
4. `/spec-first:code`、`/spec-first:orchestrate` 增加"工作区安全前置检查"

### 主题 6: 平台与扩展工程

**核心问题**：Spec-First 已支持 Claude Code + Codex 双宿主，但缺少 Agent 无关抽象、扩展系统、任务并行标记等面向规模化的能力。

**来源**：Spec-Kit（主）+ OmO（辅）

| # | 机制 | 来源 | 优先级 | 复杂度 | 解决的问题 |
|---|------|------|--------|--------|-----------|
| 35 | Agent 无关架构（18+ Agent 支持） | Spec-Kit | P2 | 中 | 单宿主依赖 |
| 36 | 用户故事组织的任务分解（`[P]` 并行标记） | Spec-Kit | P2 | 低 | TASK 缺少并行性标注 |
| 37 | Handoff 接力机制 | Spec-Kit | P2 | 极低 | Skill 间缺少自动衔接提示 |
| 38 | Skill 内嵌 MCP 生命周期管理 | OmO | P3 | 低 | MCP 依赖管理碎片化 |
| 39 | 类别化模型委派（deep/standard/quick） | OmO | P3 | 低 | 单模型无差异化 |
| 40 | V-Model 四层配对追踪 | Spec-Kit | P3 | 高 | 追踪矩阵只有单层映射 |
| 41 | 扩展系统（Extension System，RFC 起步且已有实现） | Spec-Kit | P3 | 高 | 无法让社区贡献方法论扩展 |
| 42 | 前缀匹配 Feature 目录定位 | Spec-Kit | P3 | 极低 | Feature 目录定位脆弱 |

**落地策略**：全部为 P2/P3，在核心能力（P0/P1）稳定后按需推进。P2 项以声明式/配置式预埋为主。

**P2 具体行动**：
1. 新增 `agent-config.ts`，定义各 Agent 的目录结构和命令格式；在 `spec-first setup` 增加 `--host`（或等价参数）
2. TASK 条目增加 `[P]` 并行标记；task_plan.md 按 Phase 分组
3. 每个 SKILL.md 末尾增加 `## Next Steps` 段，声明完成后的建议操作

### 选择边界：不适用 / 无需借鉴的部分

> 四份源报告共分析了约 72 项机制（Superpowers 10 + PwF 8 + OmO 12 + Spec-Kit 12 = 42 项借鉴 + ~30 项排除）。以下汇总各报告中明确排除的机制及排除理由，说明 42 项的选择边界。

| 来源 | 排除机制 | 排除理由 |
|------|---------|---------|
| Superpowers | Brainstorming / Writing-plans / Finishing-branch / Writing-skills / 跨平台插件系统 | Spec-First 的 8+2 阶段 + Gate 已覆盖且更完备 |
| PwF | 3-File Pattern / Phase 自由定义 / PostToolUse 进度提醒 / 多 IDE 适配 / TodoWrite 替代 / Single-Action Execution | Spec-First 已有同名三文件（结构化程度更高）、固定阶段 + Gate 更严格，且 Claude Code/Codex 均支持并行工具调用 |
| OmO | 多模型运行时切换 / Tmux 子代理 / AST-Grep / LSP 工具集 / Background Agent 并发队列 / 11 Agent 体系 / Claude Code 兼容层 / 平台二进制编译 / Zod v4 配置校验 | 模型调度/代码操作/LSP 由宿主负责；Spec-First 的 19 Skill 比 11 Agent 更轻量可控；已有成熟 TypeScript CLI |

## 实施原则

### 原则 1：注意力层补齐，不改流程层

四大来源的借鉴核心是补齐 AI 行为约束、上下文工程、执行可靠性、规范质量四个能力层。所有改动都是在现有 8+2 阶段流程之上追加能力，不改变流程本身的阶段定义、Gate 条件结构、追踪矩阵体系。

### 原则 2：Hook 优先于指令

能用 Hook 自动化的（PreToolUse 注意力刷新、Stop 完成度检查、安全 Hook 包装），不依赖 AI 自觉遵守指令。Hook 是强制性的，指令是建议性的。

### 原则 3：渐进验证，Daily Path 先行

先在 Daily Path（plan → code → code-review → verify）上验证 P0 效果，再推广到全阶段。通过 C7/C8/C9 合规率指标量化"加入约束前后"的 AI 流程遵从率变化。

### 原则 4：最小侵入，纯增量

P0 项全部是纯 Prompt/Hook 修改，零代码改动。P1 项以 SKILL.md 增强为主，轻量代码修改为辅。确保每一步都可独立回滚，不影响现有功能。

### 原则 5：四源协同，不重复造轮

四大来源各有侧重，借鉴时优先使用原始设计最成熟的机制：
- AI 行为约束 → 以 Superpowers 为主
- 上下文工程 → 以 Planning-with-Files 为主
- 执行可靠性 → 以 Oh-My-OpenCode 为主
- 规范质量 → 以 Spec-Kit 为主

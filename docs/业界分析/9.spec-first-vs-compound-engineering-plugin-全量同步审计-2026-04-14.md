# spec-first 与 compound-engineering-plugin 全量同步审计（2026-04-14）

## 审计结论

**结论：当前两个项目不属于“完全同步一致”。**

从代码与脚本实物核对结果看，核心 workflow 区域已经大幅同步，但仓库级功能面仍明显分叉：

- `skill` 层：存在 9 组“重命名映射”、15 组“直接对应但有差异”、1 组“拆分映射”、6 个“仅上游存在”、10 个“仅当前仓库存在”，因此不能判定为全量功能一致。
- `agent` 层：虽然 49 个上游 agent 中已有 34 个与当前仓库字节级一致，但仍有 11 个“直接对应但有差异”、4 个“仅上游存在”，另有 2 个“仅当前仓库存在”。
- `脚本/控制面`：直接可比的 skill 内脚本共有 15 个，其中 13 个字节级一致、2 个存在差异；同时两边各自还有大量独有脚本与独立控制面。
- `仓库级能力`：上游保留多目标插件同步/转换平台与 `ce-update`/`ce-debug`/`ce-optimize` 等控制面；当前仓库则新增 `spec-first` CLI、CRG、`spec-bootstrap`、`spec-graph-bootstrap`、`spec-mcp-setup`、`agent-browser` 等能力。

这意味着：**“核心链路同步度高”成立，但“两个项目已经完全同步一致”不成立。**

## 审计方法

- 核对范围：`skills/`、`agents/`、skill 内 `scripts/`、仓库级 `bin/` / `src/` / `scripts/` 控制面。
- 核对方式：
  - 目录级：枚举两个仓库全部 skill 目录与 agent 文件，不抽样。
  - 文件级：对同目录/同路径文件做 SHA-256 比对。
  - 映射级：对 `ce-* -> spec-*` 做“重命名映射”；对 `ce-setup` 做“拆分映射”；对无法一一落点的能力做“仅上游存在 / 仅当前仓库存在”判定。
- 判定口径：
  - `完全一致`：字节级一致
  - `直接对应但有差异`：存在直接对应，但内容不一致
  - `重命名映射`：能力对应，但命名/控制面已切换
  - `拆分映射`：上游单能力已被当前仓库拆到多个落点
  - `仅上游存在` / `仅当前仓库存在`：只存在于其中一个仓库

## 基于当前产品边界的再判定

上面的差异枚举是代码事实，但是否要把这些差异都视为“待补齐缺口”，取决于当前产品边界。

结合已确认的产品方向，本文补充以下前提：

- `spec-first` **不扩展多平台能力**
- `spec-first` 只维持 **Claude** 与 **Codex** 两类宿主
- 因此审计目标不应再定义成“把上游多宿主插件平台完整搬回 spec-first”

在这个前提下，主文档里的差异需要分成两类看：

1. **应保留的结构性分叉**
2. **应补齐的当前产品闭环缺口**

### 应保留的结构性分叉

| 差异域 | 是否需要同步 | 结论 | 原因 |
|---|---|---|---|
| `src/commands` + `src/sync` + `src/targets` + `src/converters` 多宿主平台 | 否 | 明确保留分叉 | 这套控制面服务于上游的多目标 plugin 转换/安装/同步平台，不符合 `spec-first` 只服务 Claude/Codex 的产品边界 |
| `scripts/release/*.ts` 发布子系统 | 否 | 明确保留分叉 | 这是上游 plugin marketplace / release 模型的一部分，当前仓库没有必要 1:1 复制 |
| `src/crg/**/*.js`、`bootstrap`、`graph-bootstrap` | 不适用 | 作为 `spec-first` 独有能力长期保留 | 这是当前仓库已经形成的产品基础设施，不应因为上游没有而被视为负向差异 |

### 当前真正需要补齐的 P0 缺口

如果按“让 `spec-first` 在 **Claude + Codex** 范围内形成更完整闭环”来定义同步目标，那么真正重要的，不是多平台控制面，而是下面三项入口级能力：

| 候选能力 | 上游代码证据 | 当前状态 | 当前近邻 | 建议落点 | 为什么是 P0 |
|---|---|---|---|---|---|
| `spec-update` | `plugins/compound-engineering/skills/ce-update/SKILL.md` | 无显式等价 skill / 命令 | `src/cli/version-reminder.js` | 新增 `skills/spec-update/`，并视需要在 `src/cli/index.js` 增加入口 | 解决升级检查、陈旧缓存、runtime 漂移等修复闭环，用户感知最直接 |
| `spec-debug` | `plugins/compound-engineering/skills/ce-debug/` | 当前缺少完整 debug-and-fix workflow | `skills/reproduce-bug/SKILL.md` | 新增 `skills/spec-debug/`，或将 `reproduce-bug` 升级为完整 debug workflow | 当前只有 issue-grounded reproduction，缺少从复现到根因、修复建议、验证收口的完整链路 |
| `spec-sessions` | `plugins/compound-engineering/skills/ce-sessions/SKILL.md` + `agents/research/session-historian.md` | 无会话历史研究控制面 | 无直接落点 | 新增 `skills/spec-sessions/`，必要时引入独立 agent | 长任务、多轮排障、经验追溯都依赖会话历史检索，这属于控制面基础能力 |

### 对“完全同步一致”的口径修正

因此，基于当前产品边界，主文档的结论应进一步收敛为：

- `spec-first` **不需要**与上游在“多宿主平台控制面”上完全一致
- `spec-first` **需要**继续维持 `skill/agent` 核心链路高同步
- `spec-first` **需要**优先补齐 `update / debug / sessions` 这三类当前用户可感知的闭环缺口

换言之，当前最合理的目标不是“与上游全量同构”，而是：

**在 Claude + Codex 范围内，实现核心工作流高同步，并补齐关键控制面缺口。**

## 关键发现

1. `skill` 不是“全量一致”，而是“核心链路高同步 + 外围控制面分叉”。
2. `agent` 同步度明显高于 `skill`：当前 live 状态是 34 个“完全一致”，比旧版映射基线更接近上游。
3. 直接可比脚本多数已经同步，当前差异主要集中在：
   - `skills/spec-review/references/resolve-base.sh`
   - `skills/resolve-pr-feedback/scripts/get-pr-comments`
4. “是否完全同步一致”的真正阻断项不在同路径 agent，而在仓库级控制面：
   - 上游独有：`ce-update`、`ce-debug`、`ce-optimize`、`ce-sessions`、`ce-slack-research`、多目标插件同步/转换平台
   - 当前独有：`spec-first` CLI、CRG、`spec-bootstrap`、`spec-graph-bootstrap`、`spec-mcp-setup`、`agent-browser`、`rclone`、`orchestrating-swarms`
5. 因此当前最准确的架构判断是：
   - **核心 workflow prompt/agent 层：高同步**
   - **仓库产品形态与平台能力层：非同构，不完全同步**

## 最终判定

如果判定标准是“核心工作流是否大体跟上游对齐”，答案是：**是，且同步度较高。**

如果判定标准是“spec-first 与 compound-engineering-plugin 是否已经功能完全同步一致”，答案是：**否。**

阻断“完全一致”结论的不是零星文案差异，而是以下三类结构性事实：

- 仍存在 15 个“直接对应但有差异” skill / 11 个“直接对应但有差异” agent
- 仍存在 6 个上游独有 skill、4 个上游独有 agent，以及对应 source-only 脚本/控制面
- 当前仓库已经长出多项上游不存在的产品能力与 CLI/CRG 子系统

所以更准确的说法应是：**spec-first 已完成大规模核心链路同步，但没有也不应该被描述为与 compound-engineering-plugin 全量功能完全一致。**

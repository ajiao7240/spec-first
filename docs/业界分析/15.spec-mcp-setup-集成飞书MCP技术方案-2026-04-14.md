# spec-mcp-setup 集成飞书 MCP 技术方案（2026-04-14）

## 目标

在 `skills/spec-mcp-setup` 中集成 `@larksuiteoapi/lark-mcp`（飞书官方 MCP），使 `spec-mcp-setup` 能够：

1. 检测用户环境中飞书 MCP 是否已配置
2. 在缺失时给出安装与配置引导
3. 为后续 `feishu-chat-researcher` / `feishu-doc-reader` agent 提供可靠的运行依赖检测基础

## 基线事实

```
包名:    @larksuiteoapi/lark-mcp
版本:    0.5.1（截至 2026-04-14）
二进制:  lark-mcp
启动命令: npx @larksuiteoapi/lark-mcp mcp \
           --app-id <APP_ID> \
           --app-secret <APP_SECRET> \
           --language zh
必填参数: --app-id / --app-secret（飞书开放平台应用凭据）
可选参数: --token-mode auto|user_access_token|tenant_access_token
         --tools preset.default（或具体 API 列表）
         --oauth（开启 OAuth 自动登录，Beta）
传输模式: stdio（默认）/ sse / streamable
```

飞书开放平台应用申请入口：<https://open.feishu.cn/app>

## 关键挑战：凭据变量与 detect-tools.sh 的兼容性

现有 `mcp-tools.json` 的两种检测方式：

| 方式 | 逻辑 | 适用场景 |
|---|---|---|
| `mcp_config` | 严格匹配 `command` + `args` | args 固定（如 sequential-thinking） |
| `command` | 执行探测命令，检查退出码 | 二进制是否可运行 |

飞书 MCP 的 `--app-id` 和 `--app-secret` 是每个用户不同的运行时凭据，**`mcp_config` 严格 args 匹配会对所有用户失败**。

需要新增第三种检测方式：`mcp_key_only`——只检查 `mcpServers` 中该 key 是否存在，不校验 args。

## 实施方案

### 一、mcp-tools.json 新增条目

```json
{
  "id": "feishu",
  "name": "飞书 MCP",
  "category": "optional",
  "description": "飞书聊天与文档 API 接入，spec-brainstorm 组织知识增强依赖（feishu-chat-researcher / feishu-doc-reader）",
  "dependencies": ["node"],
  "mcp_config": {
    "command": "npx",
    "args": [
      "-y",
      "@larksuiteoapi/lark-mcp",
      "mcp",
      "--app-id",
      "__FEISHU_APP_ID__",
      "--app-secret",
      "__FEISHU_APP_SECRET__",
      "--language",
      "zh"
    ]
  },
  "detect": {
    "method": "mcp_key_only",
    "key": "feishu"
  }
}
```

`__FEISHU_APP_ID__` / `__FEISHU_APP_SECRET__` 是模板占位符，`install-coordinator.sh` 在引导用户完成配置时负责替换。

### 二、detect-tools.sh 新增 mcp_key_only 检测分支

在 `case "$detect_method" in` 中增加：

```bash
"mcp_key_only")
  detect_key=$(jq -r --arg id "$tool_id" '.tools[] | select(.id == $id) | .detect.key' "$TOOLS_JSON")
  if [ "$HOST" = "claude" ]; then
    if [ -f "$CONFIG_PATH" ] && jq -e \
      --arg key "$detect_key" \
      '.mcpServers[$key] != null' \
      "$CONFIG_PATH" >/dev/null 2>&1; then
      found=true
    fi
  elif [ "$HOST" = "codex" ]; then
    if [ -f "$CONFIG_PATH" ] && grep -q "^\[mcp_servers\.${detect_key}\]" "$CONFIG_PATH" 2>/dev/null; then
      found=true
    fi
  fi
  ;;
```

逻辑：只检查 key 存在，不校验 command/args，兼容凭据差异。

### 三、install-coordinator.sh 新增飞书 MCP 安装引导

当检测到 `feishu` 在 `missing` 列表中时，新增安装引导段：

```bash
install_feishu() {
  echo ""
  echo "=== 飞书 MCP 配置引导 ==="
  echo "需要飞书开放平台应用凭据。若尚未创建，请前往："
  echo "  https://open.feishu.cn/app"
  echo ""
  read -r -p "请输入 Feishu App ID: " FEISHU_APP_ID
  read -r -p "请输入 Feishu App Secret: " FEISHU_APP_SECRET

  if [ -z "$FEISHU_APP_ID" ] || [ -z "$FEISHU_APP_SECRET" ]; then
    echo "⚠️  凭据为空，跳过飞书 MCP 配置。可后续手动配置。"
    return
  fi

  # 生成 Claude mcpServers 配置块
  FEISHU_CONFIG=$(jq -n \
    --arg app_id "$FEISHU_APP_ID" \
    --arg app_secret "$FEISHU_APP_SECRET" \
    '{
      "command": "npx",
      "args": ["-y", "@larksuiteoapi/lark-mcp", "mcp",
               "--app-id", $app_id,
               "--app-secret", $app_secret,
               "--language", "zh"]
    }')

  echo ""
  echo "将以下配置写入 claude_desktop_config.json（或 settings.json）的 mcpServers 中："
  echo ""
  echo "  \"feishu\": $FEISHU_CONFIG"
  echo ""
  echo "或执行以下命令先验证凭据："
  echo "  npx @larksuiteoapi/lark-mcp whoami"
}
```

### 四、verify-tools.sh 新增飞书 MCP 可达性验证

```bash
# 飞书 MCP 可达性验证（仅在已配置时执行）
if echo "$installed_tools" | grep -q '"feishu"'; then
  echo "验证飞书 MCP..."
  if npx @larksuiteoapi/lark-mcp whoami >/dev/null 2>&1; then
    echo "✓ feishu MCP 可达"
  else
    echo "⚠️  feishu MCP 已配置但 whoami 失败，请检查 App ID / App Secret 是否正确"
  fi
fi
```

`lark-mcp whoami` 会尝试打印已登录的用户 session，可作为凭据有效性的最小验证。

## 飞书开放平台应用权限说明

飞书 MCP 的可用能力取决于应用申请的权限。最小可用集建议：

| 权限 | 用途 | 是否必须 |
|---|---|---|
| `docx:document:readonly` | 读取飞书文档 | 是（feishu-doc-reader） |
| `wiki:wiki:readonly` | 读取知识库 | 推荐 |
| `im:message:readonly` | 搜索聊天消息 | 是（feishu-chat-researcher） |
| `im:resource:readonly` | 读取消息附件 | 可选 |

`--token-mode` 建议设为 `user_access_token`（访问用户自己可见的内容），而非 `tenant_access_token`（访问整个企业内容，权限过宽）。

## 与 doc 14 飞书 MCP 接入方案的关系

本文是 `spec-mcp-setup` 的基础设施层集成。doc 14 定义的两个 agent（`feishu-chat-researcher` / `feishu-doc-reader`）在运行时依赖飞书 MCP 已正确配置。

依赖关系：

```
spec-mcp-setup 集成飞书 MCP（本文）
  └─ feishu MCP 检测/安装/验证流程完善
       └─ feishu-chat-researcher / feishu-doc-reader 可在 SKILL.md 中
          明确标注"需要 feishu MCP，可通过 spec-mcp-setup 安装"
```

## 完成标准

1. `detect-tools.sh` 对已配置飞书 MCP 的环境输出 `installed: ["feishu"]`
2. `detect-tools.sh` 对未配置环境输出 `missing: ["feishu"]`
3. `install-coordinator.sh` 能引导用户完成 App ID / Secret 输入并生成配置块
4. `verify-tools.sh` 能通过 `whoami` 确认凭据有效性
5. Windows 版脚本与 shell 版行为一致
6. `spec-mcp-setup` 原有 required tools（serena / sequential-thinking / context7）的检测不受影响

# spec-review 完整方案文档：`ce-review` 基线对齐与可演进增强（2026-04-15）

## 文档定位

本文基于 `ce-review` / `spec-review` 全目录 diff、reference 文件、脚本、review agent 及运行时适配逻辑写成。

---

## Part A：上游基线与代码事实

### A.1 核对范围

- source:
  - `plugins/compound-engineering/skills/ce-review/`
- target:
  - `skills/spec-review/`
- 邻接依赖：
  - `agents/review/*`
  - `agents/research/learnings-researcher.md`
  - `agents/workflow/pr-comment-resolver.md`
  - `src/cli/adapters/claude.js`
  - `src/cli/adapters/codex.js`
  - `tests/smoke/cli.sh`

### A.2 关键差异总览

`ce-review -> spec-review` 的差异不是简单 rename，而是四类叠加：

1. 命名空间与 artifact 路径迁移
2. `spec-first` 新增 Stage-0 预载
3. 明确删除上游 `mode:headless` 契约
4. reviewer 目录与模板内容收口

### A.3 核心代码差异

#### A.3.1 `spec-review` 新增 Stage-0 上下文预载

`spec-review/SKILL.md` 开头新增了 review 阶段的 Stage-0 预载流程，依赖 `docs/contexts/<slug>/...`。

这在 `ce-review` 中不存在。

#### A.3.2 `spec-review` 明确删除 `mode:headless`

这是最大分叉之一。

具体体现：

- mode 表中删掉 `mode:headless`
- headless rules 全段删除
- findings schema 删除对 headless output enrichment 的依赖
- review-output-template 删除 headless format 一整段

这说明 `spec-review` 已不再支持上游的 programmatic headless review contract。

#### A.3.3 reviewer persona 收口

对比 `references/persona-catalog.md`：

- 上游为 17 personas
- 当前为 15 personas

缺失的是：

- `cli-readiness-reviewer`
- `previous-comments-reviewer`

这说明 `spec-review` 当前对 CLI readiness 和 PR 历史 comment 复核的内建支持弱于上游。

#### A.3.4 artifact 与 scratch 路径迁移

上游大量 reference 使用：

- `.context/compound-engineering/ce-review/...`

当前迁移为：

- `.spec-first/workflows/spec-review/...`

这属于本地 runtime contract，不能机械回退。

#### A.3.5 `spec-review` 保留并强化 report-only / interactive 口径

`spec-review` 当前的收口方向是：

- 保留 interactive / autofix / report-only
- 删除 headless
- 保持 mutating review 与 shared checkout 的边界更清楚

这是有意分叉，不是缺失。

#### A.3.6 reference 级细节变化

代码证据表明以下差异均真实存在：

- `resolve-base.sh` 只有命名与注释细差异，逻辑等价
- `subagent-template.md` 强调 `spec-first` review workflow 与 confidence floor
- `review-output-template.md` 彻底删去 headless 输出格式
- `findings-schema.json` 不再说明 headless output 依赖

### A.4 现状判断

#### 结论 1：按上游严格等价口径，`spec-review` 未完全追平

主要原因：

1. 移除了 `mode:headless`
2. 少了 2 个 conditional reviewer
3. 相关 reference/schema/template 也围绕 headless 与 persona 缩减做了收口

#### 结论 2：按当前 `spec-first` 产品边界，`spec-review` 是带有强治理口径的本地化 review workflow

增强与冻结点：

1. Stage-0 上下文预载
2. `.spec-first/workflows/` artifact 契约
3. interactive / report-only / autofix 三态更清晰
4. 与 `document-review` 的 interactive contract 保持一致

## Part B：当前产品边界下的正式实施方案

### B.1 目标

1. 保留 `spec-review` 当前 interactive-centered 审查模型
2. 补足上游 review quality contract 中仍有价值的非 headless 差异
3. 把 reviewer catalog 与 output contract 固化成自动化测试

### B.2 非目标

1. 不恢复 `mode:headless`
2. 不回退 `.spec-first/workflows/` artifact 路径
3. 不为了“人数追平”盲目恢复所有 reviewer，而不验证真实价值

### B.3 冻结口径

1. `headless` 不纳入当前实现范围。
2. review artifact 继续使用 `.spec-first/workflows/spec-review/`。
3. Stage-0 预载属于本地正式能力。

### B.4 分阶段方案

#### Phase 1：明确 reviewer 差异是否要补

建议重点评估：

- `cli-readiness-reviewer`
- `previous-comments-reviewer`

处理方式应二选一：

1. 正式补回，并同步到 persona catalog / tests
2. 在方案与注释中明确“当前 deliberately excluded”

不建议保持模糊状态。

#### Phase 2：补 contract test

建议新增：

- `tests/unit/spec-review-contracts.test.js`

至少覆盖：

- source skill 使用内部名称
- source skill 不包含 `mode:headless`
- source skill 保留 Stage-0 预载
- persona catalog 当前数量与关键 reviewer 列表
- findings schema 不再包含 headless dependency
- Claude / Codex runtime 变换正确

#### Phase 3：收口 review output 与 reviewer template

建议校验并补强：

- `references/subagent-template.md`
- `references/review-output-template.md`
- `references/findings-schema.json`

目标是防止后续同步把 headless 相关内容错误带回。

### B.5 影响文件

建议修改：

- `tests/unit/spec-review-contracts.test.js`
- 视 reviewer 取舍结果修改 `skills/spec-review/references/persona-catalog.md`

只读校验：

- `skills/spec-review/SKILL.md`
- `skills/spec-review/references/*.md`
- `skills/spec-review/references/findings-schema.json`
- `tests/smoke/cli.sh`

### B.6 建议验证

```bash
npx jest tests/unit/spec-review-contracts.test.js --runInBand
npm run test:unit
bash tests/smoke/cli.sh
```

### B.7 最终建议

`spec-review` 当前最合理的推进方式是：

1. **不回退 headless**
2. **把 reviewer catalog 的取舍显式化**
3. **把 Stage-0 / artifact / output schema 收敛成测试保护**

换句话说，`spec-review` 要追求的是“审查质量 contract 稳定”，不是“上游文本字节级复刻”。

## 结论

如果团队主要使用：

- 飞书聊天
- 飞书文档

那么增加飞书 MCP **非常值得，而且应直接成为当前方案里的主路径能力**。

但架构上不建议：

- 把飞书做成一个孤立外挂

更合理的做法是：

1. 在增强版中新增 **Feishu adapters**
2. 将飞书拆成两个独立能力面：
   - `feishu-chat-researcher`
   - `feishu-doc-reader`
3. 将它们纳入统一的 `External Context / Organizational Context` 路由体系
4. 明确 `Slack context` 不纳入当前产品方案，只保留在基线事实文档中

## 为什么飞书 MCP 值得优先做

### 原因 1：它更接近团队真实知识源

对于大量中文团队，真正沉淀决定与约束的地方往往不是 GitHub，也不是公开网页，而是：

- 飞书群聊
- 飞书文档
- 飞书知识库
- 会议纪要
- 需求/设计讨论记录

而 `spec-brainstorm` 最需要的，恰恰就是这些内容：

- 团队之前讨论过什么
- 哪些决定已经做过
- 哪些约束已经存在
- 哪些方案已经被否掉
- 哪些文档已经写好，不该重复发明

### 原因 2：飞书同时覆盖“聊天上下文”和“文档上下文”

对 brainstorm 来说，这两类上下文都很重要：

| 上下文类型 | 飞书载体 | 对 brainstorm 的价值 |
|---|---|---|
| 组织讨论上下文 | 飞书聊天 | 决策背景、争议点、团队共识、隐含约束 |
| 正式文档上下文 | 飞书文档 | PRD、方案文档、会议纪要、操作规范、项目说明 |

这意味着飞书 MCP 的价值不只是“再多一个搜索源”，而是：

**把团队真正的组织知识系统接入了 brainstorm。**

### 原因 3：它比“任意网页读取”更贴近高价值场景

网页 / GitHub URL 当然有价值，但飞书更接近内部决策语境。  
对于 brainstorm 阶段，很多关键问题是：

- “团队之前怎么定的？”
- “这是不是已经在飞书文档里写过？”
- “群里是不是已经讨论过这件事？”

这些问题的价值，通常高于“公网搜索一篇网页”。

## 在整体架构中的位置

飞书 MCP 不应被单独理解成一个工具，而应放进增强版的统一上下文架构中。

### 总体分层

| 层级 | 作用 |
|---|---|
| `Brainstorm Core` | 需求澄清、方案探索、requirements capture、handoff |
| `Context Router` | 判断是否需要 repo 外上下文 |
| `Organizational Context Adapters` | Feishu chat |
| `Document Context Adapters` | Feishu doc、GitHub docs、web docs |
| `Synthesis Layer` | 统一把上下文整理成 digest 给 brainstorm 使用 |
| `Fallback Discovery` | 必要时使用 `find-skills` 做能力发现 |

### 飞书在其中的落位

| 飞书能力 | 所属层 |
|---|---|
| 飞书聊天 | `Organizational Context` |
| 飞书文档 | `Document Context` |

这意味着飞书应当被建模为两个 adapter，而不是一个模糊的“飞书读取器”。

## 能力拆分建议

## 一、`feishu-chat-researcher`

### 目标

面向飞书聊天内容，提供组织研究能力。

### 适用场景

- “团队之前在群里讨论过这个吗？”
- “飞书里有没有关于这个方向的历史争议或结论？”
- “大家对这个方案的主要顾虑是什么？”
- “最近关于这个需求有没有新的讨论？”

### 主要输入

- 主题词
- 自然语言问题
- 群名 / 会话线索
- 时间范围
- 关键词组合

### 主要输出

统一输出为 research digest，而不是 raw chat dump：

- Workspace / tenant 标识
- 搜索范围说明
- 研究价值评估
- 关键讨论主题
- 关键结论
- 冲突点 / 争议点
- 约束 / 风险
- 相关时间与来源

### 建议落点

- `agents/research/feishu-chat-researcher.md`

## 二、`feishu-doc-reader`

### 目标

面向飞书文档与知识库内容，提供文档上下文读取能力。

### 适用场景

- “这个飞书文档里已经定义了什么？”
- “这份 PRD/会议纪要里有哪些约束？”
- “给定一篇飞书文档，提炼出与当前 brainstorm 相关的 requirements / scope / assumptions”
- “多个飞书文档一起看，做综合摘要”

### 主要输入

- 飞书文档 URL
- 文档标题 / 线索
- 指定文档集合
- 与当前 brainstorm 主题相关的问题

### 主要输出

- 文档摘要
- 关键约束
- 已确定义的需求
- 范围边界
- 决策与 rationale
- 未决问题
- 与当前 brainstorm 的关系

### 建议落点

- `agents/research/feishu-doc-reader.md`

## 三、未来可扩展能力

在完成上面两个最核心的 adapter 之后，可再扩：

| 能力 | 是否优先 |
|---|---|
| 飞书知识库多文档聚合检索 | 中 |
| 飞书会议纪要专门解析器 | 中 |
| 飞书表格 / 多维表格读取 | 低 |
| 飞书写回能力（评论/写文档） | 低，不建议前置 |

## 为什么要拆成 chat 和 doc 两个能力

### 原因 1：语义不同

聊天与文档不是一类内容：

- 聊天更噪声、更动态、更像讨论流
- 文档更结构化、更稳定、更像正式事实

### 原因 2：输出结构不同

聊天更适合输出：

- 讨论轨迹
- 争议点
- 共识
- 新旧观点变化

文档更适合输出：

- 已定义的 requirement
- scope
- 决策
- assumptions

### 原因 3：权限与查询策略不同

飞书聊天和飞书文档在实际接入时往往：

- API 不同
- 权限模型不同
- 搜索方式不同
- 摘要策略不同

所以不建议合并成一个“飞书万能读取器”。

## 与 GitHub、网页的关系

飞书不是要替代其他 source，而是补齐团队真实场景中的高价值 source。

### 推荐 source 关系模型

| source | 定位 | 适用问题 |
|---|---|---|
| `Feishu Chat` | 当前团队真实组织上下文 source | 群讨论、共识、争议、历史决策 |
| `Feishu Doc` | 当前团队真实文档上下文 source | PRD、方案、纪要、规范 |
| `GitHub` | 工程与公共协作上下文 source | issue、PR、discussion、README |
| `Web/Docs URL` | 外部公开内容 source | 博客、帮助文档、产品文档、公告 |

### 结论

如果团队主要用飞书，那么：

- `Feishu` 应成为实际产品主路径
- `GitHub` 与 `Web/Docs URL` 作为辅助外部上下文 source
- `Slack` 不纳入当前产品方案

## 对 `spec-brainstorm` 的具体改造建议

增强版建议直接把 Phase 1.1 的外部上下文路由建立在飞书与在线内容之上，不再把 `Slack context` 作为当前方案的一部分。

### 推荐结构

1. `Repo context`
2. `Organizational context`
   - Feishu chat
3. `Document context`
   - Feishu doc
   - GitHub / docs URL
4. `Fallback discovery`

### 路由规则建议

| 用户输入特征 | 推荐路由 |
|---|---|
| 明确提到“飞书群/飞书聊天/群里讨论” | Feishu chat adapter |
| 给出飞书文档链接 | Feishu doc adapter |
| 给出 GitHub URL | GitHub adapter |
| 给出普通网页/文档链接 | Web/docs adapter |
| 只说“查一下外部资料” | 先列出支持类型，必要时 fallback 到 `find-skills` |

## `find-skills` 在飞书方案中的位置

`find-skills` 可以参与，但不能作为主路径。

### 正确定位

- 用来发现当前环境有没有飞书相关能力
- 用来辅助安装或发现缺失 skill
- 用来做 fallback，而不是主执行器

### 不推荐的做法

- 每次需要飞书上下文时，先让 `spec-brainstorm` 动态去找 skill 再决定怎么读

原因：

- 不稳定
- 不可测试
- 不利于治理

### 推荐做法

优先走显式 adapter：

- `feishu-chat-researcher`
- `feishu-doc-reader`

只有在 adapter 不存在时，才允许：

- `find-skills`
- 提示能力缺失
- 指导安装

## 权限与治理建议

飞书 MCP 接入必须单独考虑权限边界，不建议默认做“全局可读”。

### 推荐治理原则

1. 默认只读
2. 默认不写回
3. 对聊天搜索与文档读取分开授权
4. 对跨空间搜索保持保守策略
5. 输出只给 digest，不大段原文回灌

### 最小可接受权限模型

| 能力 | 建议权限 |
|---|---|
| 飞书文档读取 | 用户显式提供链接，或授权限定空间 |
| 飞书聊天检索 | 仅在用户明确要求时触发 |
| 飞书写回 | 默认关闭 |

## 推荐实施优先级

如果资源有限，我建议按以下顺序落地。

### P0：飞书文档读取

优先实现：

- `feishu-doc-reader`

原因：

- 结构化程度高
- 噪声低
- 对 brainstorm 价值直接
- 风险和权限复杂度低于聊天检索

### P1：飞书聊天研究

其次实现：

- `feishu-chat-researcher`

原因：

- 对组织决策上下文价值极高
- 但摘要难度、噪声控制、权限治理都更复杂

### P2：路由集成与统一 digest

在两种 adapter 有雏形后，再把 `spec-brainstorm` 的 Phase 1.1 正式升级成：

- `Organizational Context`
- `Document Context`
- `Fallback Discovery`

## 最终建议

如果团队主要使用飞书，那么飞书 MCP 不只是“可以考虑”，而是：

**应当成为 `spec-brainstorm` 增强版里最优先建设的组织上下文能力。**

最正确的架构动作是：

1. 从当前方案中删除 `Slack context`
2. 新增 `Feishu Chat` 与 `Feishu Doc` 两个 adapter
3. 把飞书放进统一的 `External Context` 架构，而不是单独打补丁
4. 优先做只读研究与摘要，不急于做写回

这样 `spec-brainstorm` 才能真正从“上游兼容版 brainstorm”，升级为“适配团队真实协作方式的增强版 brainstorm”。

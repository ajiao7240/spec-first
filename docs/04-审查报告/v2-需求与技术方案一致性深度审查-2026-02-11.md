## 1. 审查摘要

### 1.1 总体结论

Spec-First v2 需求文档与技术方案整体一致性**良好**，双层架构（Skill 编排 + CLI 原子执行）在两侧文档中定义清晰、边界明确。7 个核心模块（M1-M7）的职责划分、6 种 ID 体系、8+2 阶段流程、Gate 三态语义、覆盖率算法（C1-C9）等核心设计在需求与技术方案之间高度对齐。

主要风险集中在**接口契约完整性**维度：3 个规划中的命令组（feature/commit/golive）在技术方案中有 CLI 签名但缺少对应的模块接口定义，以及 1 个已实现命令（matrix export）缺少模块方法映射。此外存在少量文档间数值不一致（命令组计数）和产出物定义缺口。

## 3. 关键发现

### 3.1 P1 高优先级（5 项）

#### P1-01: 3 个规划命令组缺少模块接口定义

| 维度 | 详情 |
|------|------|
| **涉及文件** | v2-03 (CLI签名) vs v2-03 (模块接口) |
| **问题描述** | `feature`（list/switch/current）、`commit`（--task/--message）、`golive`（check）三个命令组在 v2-03 中有完整的 CLI 签名定义，但对应的模块接口中没有相应方法 |
| **需求侧** | aux-02 明确列出这三个命令组，标记为 Planned |
| **技术侧** | v2-03 M1 ProcessEngine 接口缺 `featureList/featureSwitch/featureCurrent`；M7 ToolIntegration 接口缺 `commit` 相关方法；无模块接口覆盖 `golive check` |
| **影响** | 实现阶段需要补充接口定义，可能导致模块职责边界模糊 |
| **建议** | 在 M1 接口补充 feature 管理方法；在 M7 接口补充 commit 封装方法；golive check 归入 M3 GateEngine 或新增独立方法 |

#### P1-02: matrix export 命令缺少模块方法映射

| 维度 | 详情 |
|------|------|
| **涉及文件** | aux-02 (需求) + v2-03 (技术方案) |
| **问题描述** | `spec-first matrix export <featureId> [--format markdown\|yaml]` 在 aux-02 和 v2-03 均有 CLI 签名，但 v2-03 的 M2 TraceEngine 接口仅定义了 `checkMatrix()` 和 `getCoverage()`，缺少 `exportMatrix()` 方法 |
| **影响** | 实现时需在 M2 接口补充导出方法，或在命令处理器中直接实现（绕过模块层），后者会破坏分层架构 |
| **建议** | 在 M2 TraceEngine 接口补充 `exportMatrix(featureId: string, format: 'markdown' \| 'yaml'): string` |

#### P1-03: 需求文档命令组计数不一致

| 维度 | 详情 |
|------|------|
| **涉及文件** | core-01-overview.md vs aux-02-cli-system.md |
| **问题描述** | core-01 在用例描述中提及"12 命令组"，但 aux-02 实际定义了 13 个命令组（含 golive）。技术方案 v2-03 也计为 13 个命令组、35 个子命令 |
| **影响** | 需求文档内部不一致，可能导致实现团队对交付范围产生歧义 |
| **建议** | 将 core-01 中的"12 命令组"更新为"13 命令组"，与 aux-02 和 v2-03 对齐 |

#### P1-04: 技术方案新增产出物未在需求文档中定义

| 维度 | 详情 |
|------|------|
| **涉及文件** | aux-04-deliverables.md vs v2-10-产出物与模板系统.md |
| **问题描述** | v2-10 在 07_release 阶段新增了 `reports/release-note.md` 和 `reports/smoke-test-report.md` 两个必须产出物，但 aux-04 的目录结构和产出物清单中均未列出这两个文件 |
| **影响** | 技术方案定义了需求未要求的产出物，可能导致实现团队产生"过度实现"或"需求遗漏"的判断分歧 |
| **建议** | 在 aux-04 的目录结构和产出物清单中补充这两个文件，或在 v2-10 中标注为"技术方案建议新增"并提交 RFC |

#### P1-05: GATE_UNAVAILABLE 状态未纳入类型定义

| 维度 | 详情 |
|------|------|
| **涉及文件** | v2-12-上线路线图与GoLive.md vs v2-03/v2-06 |
| **问题描述** | v2-12 阶段 A Gate 降级策略明确提出"GateEngine 不可用时返回 `GATE_UNAVAILABLE` 状态并阻断"，但 v2-03 的 `GateResult.status` 类型定义为 `'PASS' \| 'PASS_WITH_WAIVER' \| 'FAIL'`，v2-06 的 Gate 结果语义表也仅含三态 |
| **影响** | 阶段 A 降级路径引入了类型系统未覆盖的第四态，TypeScript 编译期无法捕获此状态的处理遗漏 |
| **建议** | 方案一：在 GateResult.status 中增加 `'GATE_UNAVAILABLE'` 第四态；方案二：将 GATE_UNAVAILABLE 作为独立错误类型（非 GateResult），通过异常路径处理。推荐方案二，保持 Gate 语义纯净 |

### 3.2 P2 中优先级（4 项）

#### P2-01: defect discovered_in_stage 必填性不一致

| 维度 | 详情 |
|------|------|
| **涉及文件** | aux-02 (CLI签名) vs v2-07 (DefectRecord) |
| **问题描述** | aux-02 的 `defect register` 签名中 `--discovered-in <stage>` 标记为可选参数（方括号），但 v2-07 的 DefectRecord 数据结构中 `discovered_in_stage` 为必填字段（无 `?` 标记） |
| **影响** | CLI 层允许不传此参数，但数据层要求必填，需要默认值策略 |
| **建议** | 统一为可选（CLI 不传时默认取 stage-state.json 的 currentStage），并在 v2-07 DefectRecord 中标记为 `discovered_in_stage?: Stage` |

#### P2-02: Context Pack 总预算仅在技术方案中定义

| 维度 | 详情 |
|------|------|
| **涉及文件** | aux-01-skill-system.md vs v2-08-context-pack与catchup.md |
| **问题描述** | v2-08 定义了 Context Pack 总预算为 8K tokens（control < 2KB + references 动态裁剪），但 aux-01 仅定义了 control < 2KB 的硬限制，未明确总预算上限 |
| **影响** | 需求侧缺少总预算约束，技术方案的 8K tokens 限制缺乏需求依据，后续调整时可能产生需求-设计不同步 |
| **建议** | 在 aux-01 Context Pack v2.0 标准中补充总预算上限定义（如"总输出不超过 8K tokens"） |

#### P2-03: stage-state.json 合并字段为技术方案独有

| 维度 | 详情 |
|------|------|
| **涉及文件** | core-04-process.md vs v2-04-流程引擎与阶段状态机.md |
| **问题描述** | v2-04 的 stage-state.json 结构中包含三层合并结果字段（如合并后的 Gate 条件、裁剪后的产出物清单等），这些字段在 core-04 中未提及。core-04 仅定义了阶段语义和流转规则，未约束状态文件的具体字段 |
| **影响** | 低风险。属于正常的需求→设计细化，但合并字段的存在意味着 init 时的三层合并逻辑成为隐式依赖 |
| **建议** | 在 core-02 或 core-04 中补充一句说明："stage-state.json 在 init 时完成三层规范合并，合并结果持久化供后续阶段使用" |

#### P2-04: defect register 缺少 --linked-fr 参数在技术方案中的映射

| 维度 | 详情 |
|------|------|
| **涉及文件** | v2-03-cli命令与核心模块设计.md vs v2-07-变更管理与同步机制.md |
| **问题描述** | v2-03 的 `defect register` 签名包含 `[--linked-fr <frId>]` 可选参数，但 v2-07 的 DefectRecord 接口中没有对应的 `linked_fr` 字段。缺陷与 FR 的关联关系在数据结构中缺失 |
| **影响** | CLI 接受的参数无法持久化到数据记录中，缺陷→FR 的追踪链断裂 |
| **建议** | 在 v2-07 DefectRecord 中补充 `linked_fr?: string` 字段，或在 v2-03 CLI 签名中移除此参数 |

## 2. P0 Critical — 阻断级问题（1 项）

### P0-01: Layer 2 YAML Schema 需求与技术方案完全不一致

| 属性 | 详情 |
|------|------|
| **发现者** | 效能总监 ✕ AI 架构师 ✕ 研发架构师（3/3 独立发现） |
| **置信度** | 极高 |
| **需求侧** | `aux-03-multi-platform.md` L54-101 |
| **技术方案侧** | `v2-09-多端规范合并.md` L29-52 |

**不一致详情**：

| 维度 | aux-03（需求） | v2-09（技术方案） |
|------|---------------|--------------------|
| 顶层字段 | `name: H5 前端` | `platform: h5` |
| gate 条件结构 | `{id, description, type, command, threshold}` 5 字段 | `{id, description, check}` 3 字段 |
| gate 条件 ID 格式 | `L2-H5-VERIFY-001`（层级+端+阶段+序号） | `h5-lighthouse-score`（自然语言 kebab-case） |
| Lighthouse 阈值 | `>= 80`（L76） | `>= 90`（L36） |
| extra_deliverables | 对象数组 `{name, required, description}` | 纯字符串数组 `["responsive-breakpoints.md"]` |
| quality_thresholds | 扁平 key-value `{bundle_size_kb: 500}` | 含语义字段 `{value: 200, direction: lower_is_better}` |

**影响**：实现团队无法确定以哪个 schema 为准。两端 YAML 结构完全不同，无法互相解析。

**修复建议**：统一为一个 schema。建议以 v2-09 的 `direction` 语义字段为基础（更严谨），同时采纳 aux-03 的 `type: auto|manual` gate 条件区分能力（GateEngine 需要区分自动/手动条件）。统一后同步更新两端文档。

### 6.2 架构亮点（三方共识）

1. **双层架构（Skill + CLI）职责分离彻底** — Skill 负责 AI 编排（非确定性），CLI 负责原子操作（确定性），单向依赖（Skill → CLI），CLI 不感知 Skill 存在
2. **6-ID 追踪链设计精简** — 将 NFR 合并到 FR（标签区分）、API 合并到 DS（属性引用）、ADR 合并到 RFC（子类型），从 9+ 种 ID 精简到 6 种
3. **3 层规范合并（Layer 0/1/2）渐进式裁剪** — Layer 0 基线 → Layer 1 按 Mode×Size 裁剪 → Layer 2 按技术端叠加，三层正交组合
4. **confirm_policy 三档信任度模型** — strict → assisted → auto，配合四维判定矩阵，实现 AI 自主性与人类控制权的平衡
5. **12 项度量指标体系完整** — C1-C9 + E1 + Q1 + H1，公式、权重、惩罚项在需求与技术方案间高度一致

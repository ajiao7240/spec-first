# 代码功能完整性修复任务清单

> 来源报告：`docs/03审查报告/review-代码功能完整性-多agent审查-2026-02-09.md`  
> 目标：将当前版本从“可测试但不可用”修复到“主流程可跑通、关键命令可用、可复验”。

## 1. 执行策略

1. 先清零 `P0`（阻断主流程的功能断点）。  
2. 再处理 `P1`（质量与运维闭环）。  
3. 最后处理 `P2`（运营与可持续性）。  
4. 每完成一个任务，必须执行对应复验命令并记录结果。

---

## 2. P0 修复任务（必须优先完成）

### P0-1 修复 `stage` 命令 Gate 参数传递错误

- 问题：`GateDefinition | undefined` 误传为 `GateCondition[]`，导致 `conditions is not iterable`。
- 修改点：
  - `src/commands/stage.ts`
  - `src/commands/gate.ts`
- 修复要求：
  - 对 `getGateDefinitions(...)` 做空值保护。
  - 统一传 `gateDef.conditions` 给 `evaluateGate(...)`。
  - 无 Gate 阶段应直接跳过校验，不抛异常。
- 验收标准：
  - `spec-first stage advance FEAT-XXX` 在 `00_init` 不崩溃。
  - `spec-first gate check FEAT-XXX --stage 02_design` 可正常输出结果。

### P0-2 修复阶段状态机 Gate 写回逻辑

- 问题：Gate 通过后未写回 `gate_passed`，导致 `02->03` 永久卡死。
- 修改点：
  - `src/commands/stage.ts`
  - `src/core/process-engine/stage-machine.ts`（按需）
- 修复要求：
  - Gate 评估前调用 `markGateChecking(...)`。
  - PASS 时调用 `markGatePassed(...)`，FAIL 时调用 `markGateFailed(...)`。
  - `--force` 语义需明确：是否绕过并直接允许推进（需与产品规则一致）。
- 验收标准：
  - 可连续推进 `00_init -> 01_specify -> 02_design -> 03_plan`。

### P0-3 修复 AI 命令层与核心层接口漂移

- 问题：`ai context/catchup/stats` 与核心模块方法签名不一致。
- 修改点：
  - `src/commands/ai.ts`
  - `src/core/ai-orchestrator/context-builder.ts`
  - `src/core/ai-orchestrator/session-manager.ts`
  - `src/core/ai-orchestrator/stats-collector.ts`
- 修复要求：
  - 统一 `buildContextPack`、`catchup` 调用签名。
  - 替换不存在的方法（`getSummary/getStats`），或在核心层补齐并保持类型一致。
  - `--stage` 参数语义需与 `StageId` 对齐。
- 验收标准：
  - `spec-first ai context FEAT-XXX --validate` 成功。
  - `spec-first ai catchup FEAT-XXX` 成功输出摘要。
  - `spec-first ai stats FEAT-XXX` 无运行时函数不存在错误。

### P0-4 修复 defect 命令数据写坏与参数错位

- 问题：命令层对象入参传给位置参数接口，导致数据结构污染。
- 修改点：
  - `src/commands/defect.ts`
  - `src/core/change-mgr/defect-tracker.ts`（如需兼容）
  - `src/core/change-mgr/types.ts`（字段统一）
- 修复要求：
  - 命令层按核心接口签名传参，或重构核心接口接收对象并全链路统一。
  - 字段统一使用 `defectId`，消除 `id` 混用。
  - `update/get/list/escape-rate` 输出字段与类型一致。
- 验收标准：
  - `defect register` 后 JSON 结构正确。
  - `defect get`、`defect update`、`defect list` 全部可用。

### P0-5 修复类型检查失败

- 问题：`pnpm check` 在 `typecheck` 失败，接口未闭环。
- 修改点：
  - `src/commands/*.ts`（AI/defect/gate/stage）
  - `src/parsers/markdown-parser.ts`
  - `package.json`（补依赖类型，如 `@types/mdast`，按需）
- 修复要求：
  - 消除所有 TS 编译错误。
  - 保证 `strict` 下通过。
- 验收标准：
  - `pnpm check` 全量通过。

**需求原文** (`spec-first-v5.md:1945`):
> 安全风险叠加规则：当 Feature 存在 `NFR-SEC-*` 类型的非功能需求时，无论 Size 大小，SAST 静态扫描为**必须项**。即 Size S + NFR-SEC 场景下，安全测试要求上浮为"OWASP Top 10 + SCA + SAST"。

**差距分析**：技术方案的 `security_clear` Gate 条件固定为 "OWASP + SCA"，**完全缺失 SAST 动态上浮逻辑**。当 Size S 的 Feature 包含 `NFR-SEC-*` 需求时，Gate 不会要求 SAST 扫描，导致安全漏洞可能逃逸。

#### ❌ V-02 | 追踪矩阵状态枚举与需求不一致 [P0] — 已确认

| 属性 | 内容 |
|------|------|
| **来源** | 报告 B (P0-2) |
| **报告 A 是否发现** | ❌ 未发现 |
| **报告 C 是否发现** | ❌ 未发现 |
| **验证结论** | **确认存在，P0 级** |

**需求定义** (`spec-first-v5.md:1604-1611`) — **8 种状态**：
> 📋 Planned → 🔨 Implemented → ✅ Verified → 🎯 Accepted → ⏸️ Deferred → 🚫 Cancelled → ⚠️ Exception → ❌ Not Implemented

**技术方案定义** (`02-ID与追踪体系设计.md:254-259`) — **6 种状态**：
> 📝 Draft → 🔄 In Progress → ✅ Verified → 🎯 Accepted → 🚫 Cancelled → ❌ Not Implemented

**差距分析**：

| 差异点 | 需求 | 技术方案 | 影响 |
|--------|------|---------|------|
| 初始状态命名 | 📋 Planned | 📝 Draft | 语义不同：Planned=已规划，Draft=草稿 |
| 中间状态命名 | 🔨 Implemented | 🔄 In Progress | 语义不同：已实现 vs 进行中 |
| **缺失状态** | ⏸️ Deferred | — | 覆盖率分母计算错误 |
| **缺失状态** | ⚠️ Exception | — | Known Exception 机制无法落地 |

**核心风险**：覆盖率公式 `Active = Total - (Deferred + Cancelled + Exception)`，技术方案缺少 Deferred 和 Exception 两种状态，导致 Active 集合的分母定义与需求不一致，**所有覆盖率指标（C1-C9）的计算结果将失真**。

#### ⚠️ V-04 | Hook 触发点缺口：缺少 Pre-commit ID lint [P1] — 已确认

| 属性 | 内容 |
|------|------|
| **来源** | 报告 B (P1-1) |
| **报告 A 是否发现** | ❌ 未发现 |
| **报告 C 是否发现** | ❌ 未发现 |
| **验证结论** | **确认存在，P1 级** |

**需求原文** (`spec-first-v5.md:2348`):
> | ID 格式校验 | Regex lint | **Pre-commit** |

**技术方案原文** (`03-质量门禁与Hook体系设计.md:185-189`):
> 仅定义 3 种 Hook：commit-msg、pre-push、CI Pipeline

**差距分析**：需求明确要求在 `pre-commit` 阶段执行 ID 格式校验（Regex lint），但技术方案仅设计了 `commit-msg`（校验提交消息）和 `pre-push`（增量 SCA）。ID 格式校验的执行时机从 pre-commit 后移到了 commit-msg，**反馈时机延后**，开发者在编写代码时无法即时发现 ID 格式错误。

**降级评估**：虽然 commit-msg Hook 也能拦截不合规提交，但 pre-commit 的优势在于可以校验**文件内容中的 ID 格式**（如 spec.md 中的 FR-XXX），而 commit-msg 仅校验提交消息中的 ID 标签。两者校验对象不同，不可互相替代。

### 2.3 报告 A 独有问题的重新评估

报告 A 提出了 8 个 P0，其中 V-03（evaluateGate 签名）已在 §2.2 确认。以下对报告 A 独有的其余 7 个 P0 逐项重新评估，区分**设计缺陷**与**增强建议**。

#### RA-01 | 测试策略完全缺失 [报告A: P0-01] — 重新评估：**降级为增强建议**

| 属性 | 内容 |

| **报告 B 是否提及** | ❌ 未提及 |
| **报告 C 是否提及** | ❌ 未提及 |
| **重新评估** | **增强建议，非设计缺陷** |

**分析**：技术方案的定位是"详细设计文档"，其职责是定义系统如何实现，而非定义如何测试系统。测试策略通常作为独立文档（如 `08-测试策略.md`）补充，不属于 M1-M7 详细设计的必要组成部分。报告 B 和 C 均未将此视为问题，说明这是**文档体系完整性**的增强需求，而非技术方案本身的设计缺陷。

**建议**：作为独立行动项补充，不阻塞技术方案审批。

**分析**：Hook 脚本随仓库提交确实存在被恶意修改的风险，但这是所有基于 Git Hook 的工具链（Husky、lefthook 等）的通用风险，并非 Spec-First 特有的设计缺陷。实际场景中，PR Review 流程本身就是对 Hook 脚本变更的审查防线。SHA256 校验和是合理的加固措施，但不应定为 P0 阻断项。

#### FIX-01 | V-01 安全门禁 NFR-SEC 上浮规则

**问题本质**：`security_clear` Gate 条件为静态规则，缺少基于 NFR-SEC 的动态上浮逻辑。

**解决方案**：在 `03-质量门禁与Hook体系设计.md` 的 Gate 条件表中，将 `security_clear` 从固定条件改为**条件矩阵**。

**修改位置**：`03-质量门禁与Hook体系设计.md:326` 附近

**修改前**：
```
security_clear | 安全无高危漏洞 | auto | OWASP + SCA 扫描无 High/Critical
```

**修改后**：
```
security_clear | 安全无高危漏洞 | auto | 基础：OWASP Top 10 + SCA 扫描无 High/Critical
                                        上浮：当 Feature 含 NFR-SEC-* 需求时，
                                        追加 SAST 静态扫描（无论 Size）
```

**补充设计**：在 Gate Evaluator 中增加 NFR-SEC 检测逻辑：

```typescript
// gate-evaluator.mjs 伪代码
function resolveSecurityGate(featureId: string, size: Size): SecurityGateCondition {
  const hasNfrSec = traceMatrix.hasNfrType(featureId, 'NFR-SEC-*');
  const base = ['OWASP_TOP_10', 'SCA'];
  if (hasNfrSec) {
    base.push('SAST');  // 风险上浮：追加 SAST
  }
  return { checks: base, threshold: 'NO_HIGH_CRITICAL' };
}
```

**验收标准**：Size S + NFR-SEC-001 场景下，Gate 要求包含 SAST 扫描。

#### FIX-02 | V-02 追踪矩阵状态枚举对齐

**问题本质**：技术方案定义了 6 种状态，需求定义了 8 种状态，缺失 Deferred 和 Exception，且已有状态命名不一致。

**修改后状态枚举（8 种）**：

| 状态 | 图标 | 语义 | 覆盖率计算 |
|------|------|------|-----------|
| Planned | 📋 | 已规划，尚未开始实现 | 计入 Active |
| Implemented | 🔨 | 已完成编码实现 | 计入 Active |
| Verified | ✅ | 已通过测试验证 | 计入 Active |

#### FIX-04 | V-04 补充 Pre-commit ID lint Hook

**修改位置**：`03-质量门禁与Hook体系设计.md:185-189`

**方案**：在现有 3 种 Hook（commit-msg、pre-push、CI）基础上，增加 `pre-commit` Hook。

| Hook | 触发时机 | 校验内容 | 超时 |
|------|---------|---------|------|

#### FIX-08 | RA-05 JSONL 并发写入安全

**修改位置**：`03-质量门禁与Hook体系设计.md` 和 `06-度量与运营体系设计.md`

**方案**：采用**写临时文件 + 原子 rename** 模式，辅以 `flock` 文件锁。

```javascript
// 伪代码：原子追加写入
async function appendJsonl(filePath, record) {
  const tmpFile = `${filePath}.${process.pid}.tmp`;
  const fd = await fs.open(filePath, 'a');
  try {
    await flock(fd, LOCK_EX);           // 获取排他锁
    await fs.appendFile(fd, JSON.stringify(record) + '\n');
  } finally {
    await flock(fd, LOCK_UN);           // 释放锁
    await fd.close();
  }
}
```

**跨平台兼容**：macOS/Linux 使用 `flock`，Windows 使用 `fs.open` 的 `O_EXCL` 标志或 `proper-lockfile` npm 包。

**落地时机**：编码阶段实现。

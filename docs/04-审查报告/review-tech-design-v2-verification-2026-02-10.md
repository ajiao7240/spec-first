## 2. 详细核实记录

### 2.1 [P0] ID 主模型冲突
- **需求 (`core-03`)**: 明确定义 `FR`, `NFR`, `DS`, `API`, `TASK`, `TC`, `ADR`, `RFC` 为独立 ID 类型，有独立前缀和正则。
- **方案 (`v2-05`)**: 试图将 ID 精简为 6 种，声明 "API → DS 属性", "ADR → RFC 属性", "NFR → FR 标签"。
- **方案内矛盾**: `v2-05` 废弃了独立 API ID，但 `v2-07` 的 RFC 示例中仍使用 `API-AUTH-001`。
- **结论**: 技术方案擅自修改核心领域模型，且导致内部逻辑不自洽，必须修正 `v2-05` 以完全对齐需求。

### 2.2 [P1] Gate 条件弱化
- **需求 (`core-05`)**: Gate 3 (Verify) 必须校验 `Test Coverage(AC) ≥ 90%` 和 `Implementation Coverage = 100%`。
- **方案 (`v2-06`)**: `05_verify` 阶段仅列出 `Test Coverage(FR) = 100%`，遗漏了 AC 级覆盖率和实现覆盖率校验。
- **结论**: 方案门禁标准低于需求，存在质量风险，需补齐 `v2-06`。

### 2.3 [P1] Hook 协议与命令缺失
- **Hook 部分**: 需求要求 `prepare-commit-msg` 自动预填 ID。方案 `v2-06` §7 摘要表中未列出此 Hook，细节缺失。
- **命令部分 (驳回)**: 用户反馈 `v2-03` 缺失以下命令，经核实**均已存在**：
    - `gate conditions`: 见 `v2-03` §2.4
    - `rfc submit/get`: 见 `v2-03` §2.8
    - `defect update/get/escape-rate`: 见 `v2-03` §2.9
    - `feature current`: 见 `v2-03` §2.11
    - `commit --task`: 见 `v2-03` §2.12
- **结论**: 需完善 `v2-06` 文档对 Hook 的描述；命令缺失问题不成立。

### 2.4 [P1] ExitCode 冲突
- **需求 (`aux-02`)**: `1 = GATE_FAILED`, `2 = VALIDATION_ERROR`
- **方案 (`v2-03`)**: `1 = VALIDATION_ERROR`, `2 = GATE_FAILED`
- **结论**: 编码定义互换，需修正方案以对齐需求（通常需求定义了外部契约，应以需求为准）。

### 2.5 [P1] 包管理器冲突
- **需求 (`aux-02`)**: 明确 "仅 npm 安装模式"。
- **方案 (`v2-01`)**: 技术栈选型为 `pnpm`，`doctor` 命令检查 `pnpm`。
- **结论**: 方案违反了明确的非功能性约束，需改为 `npm`。

### 2.6 [P2] 评审报告陈旧
- **事实**: `v2-09` 当前文本已明确区分了两种场景：
    1. 用户**未传** `--platforms` -> 退化模式 (No Error)
    2. 用户**传了** `--platforms` 但文件缺失 -> 阻断 (Error)
- **报告 (`v2-99`)**: 仍声称这是 "内部矛盾 (P0-3)"，显然是基于旧版文档的评审结论。
- **结论**: `v2-99` 报告不再准确反映当前方案状态，具有误导性。

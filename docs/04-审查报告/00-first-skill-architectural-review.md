# 00-first Skill 架构审查报告

> **审查日期**: 2026-02-28
> **审查范围**: `skills/spec-first/00-first/` 目录
> **审查角色**: Software Architect
> **架构模式**: Subagent-Driven Architecture

---

## 执行摘要

00-first skill 采用 Subagent-Driven 架构模式，通过主线程编排和子 agent 并行执行实现项目认知文档的自动化生成。整体架构设计合理，但在文档一致性、阶段命名、依赖管理和扩展性方面存在改进空间。

**关键发现**:
- 1 个严重问题：文档间存在结构性不一致
- 2 个高优先级问题：阶段命名混淆、A4 扇入瓶颈
- 4 个中等问题：中间数据格式规范缺失
- 3 个低优先级改进建议

### 1.3 发现问题

#### [CRITICAL] 文档结构性不一致

**严重程度**: Critical
**影响范围**: 架构可理解性、实现一致性

**问题描述**:
`SKILL.md` 与 `references/subagent-architecture.md` 对 agent 拆分策略存在根本性不一致：

| 方面 | SKILL.md | subagent-architecture.md |
|------|----------|--------------------------|
| Agent A1 | 单一 agent | 拆分为 A1-Backend + A1-Frontend |
| Agent B | 单一 agent | 拆分为 B-Backend + B-Frontend |
| Agent A3 | 单一 agent | 拆分为 A3-Backend + A3-Frontend |
| Agent C2 | 单一 agent | 拆分为 C2-1~C2-7（7个子任务） |

**影响**:
- 实现者可能按任一文档实现，导致不一致
- subagent-architecture.md 中的评估矩阵（推荐拆分列）与 SKILL.md 实际定义冲突

**建议**:
1. 明确 subagent-architecture.md 是"设计提案"还是"实现规范"
2. 如为设计提案，需标注"待实现"状态
3. 如为规范，需同步更新 SKILL.md 的 agent 定义

### 2.2 依赖链分析

| 依赖关系 | 类型 | 风险等级 |
|----------|------|----------|
| A1 → A2 | 数据依赖（模块清单 JSON） | 低 |
| C1 → C2 | 数据依赖（外部服务列表） | 低 |
| P1b → C2 | 数据依赖（Context7 映射） | 低 |
| A2 → A4 | 数据依赖（架构关系） | 中 |
| B → A4 | 数据依赖（API 文档） | 中 |
| D → A4 | 数据依赖（数据库 Schema） | 中 |

### 2.3 发现问题

#### [HIGH] A4 扇入瓶颈

**严重程度**: High
**影响范围**: 执行效率、失败传播

**问题描述**:
Agent A4（领域模型分析）依赖 3 个不同的 agent 产出：
- A2（架构关系）
- B（API 文档）
- D（数据库 Schema）

这意味着：
1. A4 必须等待 3 个 agent 全部完成才能启动
2. 任一上游 agent 失败，A4 输出将不完整
3. 依赖跨越 Wave1 和 Wave3，增加了关键路径长度

**建议**:
1. 考虑将 A4 拆分为多个独立分析任务：
   - A4a: 实体识别（仅依赖 A1 模块清单）
   - A4b: 关系分析（依赖 A2）
   - A4c: API 语义分析（依赖 B）
   - A4d: 持久化分析（依赖 D）
2. 最终由 P5 汇总阶段合并各部分结果

#### [MEDIUM] 依赖状态管理缺失

**严重程度**: Medium
**影响范围**: 错误处理、调试

**问题描述**:
当前设计未明确定义：
1. 上游 agent 部分成功时，下游 agent 如何处理
2. 中间数据格式校验规则
3. 依赖超时后的降级策略

**建议**:
1. 定义 agent 产出状态枚举：`SUCCESS | PARTIAL | FAILED | TIMEOUT`
2. 为每个依赖关系定义降级策略表
3. 增加中间数据 JSON Schema 校验

## 3. 数据流设计分析

### 3.1 中间数据格式

当前定义的中间数据格式：

```json
// A1 模块清单 → A2, A4
{
  "modules": [
    {"name": "auth", "path": "src/auth/", "responsibility": "认证与授权"}
  ]
}

// B API 文档 → A4
{
  "apis": [
    {"method": "POST", "path": "/api/users", "file": "...", "line": 76}
  ]
}

// D 数据库 Schema → A2, A4
{
  "tables": [
    {"name": "users", "columns": [...], "foreign_keys": [...]}
  ]
}
```

### 3.2 发现问题

#### [MEDIUM] 中间数据格式规范缺失

**严重程度**: Medium
**影响范围**: agent 间兼容性、版本升级

**问题描述**:
1. 无 JSON Schema 定义，字段变更时无校验
2. 无版本号字段，格式演进时无兼容性保证
3. B → A4 的 API 文档格式在 SKILL.md 中未明确定义

**建议**:
```json
{
  "$version": "1.0.0",
  "$producer": "A1",
  "$timestamp": "2026-02-28T10:00:00Z",
  "modules": [...]
}
```

#### [MEDIUM] 数据传递机制未定义

**严重程度**: Medium
**影响范围**: 实现一致性

**问题描述**:
SKILL.md 未明确 agent 间数据传递的实现方式：
- 内存共享？
- 文件系统？
- 上下文传递？

**建议**:
明确传递机制并在架构文档中定义。推荐使用上下文传递（context passing）模式，由主线程管理所有中间数据。

## 4. 阶段编排分析

### 4.1 阶段定义

| 阶段 | 执行者 | 职责 |
|------|--------|------|
| P0 | 主线程 | 定位与校验 |
| P1a | 主线程 | 技术栈识别 |
| P1b | 主线程 | Context7 映射收集 |
| P2 | Subagents | Agent 并行执行（3 波） |
| P5 | 主线程 | 汇总与 README 生成 |

### 4.2 发现问题

#### [HIGH] 阶段命名混淆

**严重程度**: High
**影响范围**: 可理解性、文档维护

**问题描述**:
1. 阶段编号跳跃：P0 → P1a → P1b → P2 → P5（缺少 P3、P4）
2. Agent D 的内部阶段命名为 P3/P4（见 `agent-database.md`），与编排阶段命名冲突
3. SKILL.md 变更日志提到 "P3/P4 阶段引用消歧"，但正文仍有混淆

**建议**:
1. 编排层面采用统一命名：`O1, O2, O3...`（Orchestration）
2. Agent 内部阶段保持 `P1, P2, P3...` 命名
3. 或采用完全不同的命名空间

#### [MEDIUM] Wave 边界条件不明确

**严重程度**: Medium
**影响范围**: 并发正确性

**问题描述**:
1. Wave2 的触发条件："P1b 完成后派发" 同时 "C2 等待 C1 完成"
2. 如果 C1 先于 P1b 完成，C2 是否立即启动？
3. 如果 P1b 先于 C1 完成，C2 是否等待 C1？

**建议**:
明确 Wave2 启动的充分必要条件：
```
Wave2 启动条件 = P1b 完成 AND C1 完成
```

## 5. 可扩展性分析

### 5.1 当前扩展点

| 扩展点 | 当前机制 | 评估 |
|--------|----------|------|
| 新增检测规则 | 修改 detection-rules.md | 手动，集中式 |
| 新增 agent | 修改 SKILL.md + 新建 reference 文件 | 手动，侵入式 |
| 新增语言支持 | 修改 detection-rules.md | 手动，集中式 |
| 自定义产物模板 | 无 | 不支持 |

### 5.2 发现问题

#### [LOW] 无插件/扩展机制

**严重程度**: Low
**影响范围**: 定制化能力

**问题描述**:
当前架构不支持：
1. 用户自定义 agent
2. 第三方检测规则扩展
3. 产物模板定制

**建议**:
1. 定义 agent 接口规范（输入/输出契约）
2. 支持通过配置文件注册自定义 agent
3. 支持模板覆盖机制

#### [LOW] 检测规则集中管理

**严重程度**: Low
**影响范围**: 规则维护效率

**问题描述**:
所有检测规则集中在 `detection-rules.md`，无分层：
- 核心规则（必须）
- 可选规则（按需）
- 项目特定规则（覆盖）

**建议**:
```
detection-rules/
├── core/           # 核心规则（内置）
├── frameworks/     # 框架规则（按需加载）
└── custom/         # 项目特定规则（覆盖）
```

## 6. 可伸缩性分析

### 6.1 当前限制

| 维度 | 当前限制 | 影响 |
|------|----------|------|
| 超时 | 单 agent 120s，总阶段 300s | 大型项目可能超时 |
| 文件数 | >10000 时截断目录树 | 信息丢失 |
| Context7 | 最多 5 个库，30s 超时 | 依赖覆盖不全 |
| Serena | 单实例 LSP | 多语言项目受限 |

### 6.2 发现问题

#### [MEDIUM] 大型项目支持不足

**严重程度**: Medium
**影响范围**: 适用范围

**问题描述**:
1. 无分批/分页处理机制
2. 无渐进式加载策略
3. Monorepo 50+ 包时，当前拆分策略无法有效并行

**建议**:
1. 引入任务分片机制：按包/模块分片，每片独立 agent
2. 支持增量分析：基于 git diff 仅分析变更部分
3. 引入任务队列：动态调度，而非静态 3 波

#### [MEDIUM] 多语言项目支持受限

**严重程度**: Medium
**影响范围**: 微服务架构项目

**问题描述**:
1. Serena 单实例 LSP，无法同时支持多语言
2. 检测规则按项目整体识别，无法区分服务边界

**建议**:
1. 支持 Serena 多实例（每语言一个）
2. 支持按服务/包独立分析，最后汇总

## 7. 韧性分析

### 7.1 当前韧性机制

| 机制 | 实现状态 | 评估 |
|------|----------|------|
| 超时控制 | 已实现（60-120s） | 良好 |
| 降级策略 | 已实现（Serena→静态） | 良好 |
| 失败隔离 | 已实现（agent 间不阻塞） | 良好 |
| 串行断链 | 已实现（跳过后续文档） | 良好 |
| 重试机制 | 未实现 | 缺失 |
| 检查点/恢复 | 未实现 | 缺失 |

### 7.2 发现问题

#### [MEDIUM] 无重试机制

**严重程度**: Medium
**影响范围**: 瞬态失败处理

**问题描述**:
对于网络请求（Context7 API）或外部服务（DB 连接）的瞬态失败，无自动重试。

**建议**:
1. 对网络请求实现指数退避重试（最多 3 次）
2. 区分可重试错误和不可重试错误

#### [LOW] 无检查点/恢复机制

**严重程度**: Low
**影响范围**: 长时间运行任务

**问题描述**:
如果 P5 阶段失败，需要从头重新执行所有 agent。

**建议**:
1. 在每个 agent 完成后保存检查点
2. 支持从检查点恢复执行

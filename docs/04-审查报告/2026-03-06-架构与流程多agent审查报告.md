# Spec-First 架构与流程多 Agent 审查报告

## 1. 审查目标

从架构师全局视角审查 `spec-first` 的系统设计、端到端流程、Skill/CLI 契约、近期代码变更与运行态一致性，判断当前系统是否满足“流程流转顺畅、健壮性高、质量高”的目标。

## 2. 审查范围

- 系统主链路：阶段状态机、Gate、阶段推进、运行态文件
- Skill 体系：`code-review`、`orchestrate`、共享 AGENTS 约束、hard-gate 运行时
- 近期变更：`trace-context` / `ratio` / `coverage` / `gate-evaluator` / `sca`
- 运行实例：`FSREQ-20260306-DASHBOARD-001`

## 3. 审查方法

三条并行审查线：

1. 流程/状态线：检查 `stage advance`、Gate、阶段历史与终态一致性
2. 追踪/覆盖率线：检查近期 trace 重构与覆盖率/Gate/SCA 语义一致性
3. Skill/编排线：检查 Skill 文本约束是否被 runtime 真正执行

验证命令：

```bash
npx vitest run tests/unit/coverage.test.ts tests/unit/trace-context.test.ts tests/unit/trace-ratio.test.ts tests/unit/gate-evaluator.test.ts tests/unit/advance.test.ts
```

结果：5 个测试文件、53 个测试全部通过。结论是近期 `trace` 重构在单元层面未出现直接回归；主要风险集中在流程治理与契约执行层，而不是这轮重构本身。

## 4. Findings

### CRITICAL-01: `--force` 可以绕过失败 Gate 直达 `08_done`，系统允许“未验收完成”

**结论**

`stage advance --force` 会直接跳过依赖检查与 Gate 检查，并仍然写入下一阶段与终态。这不是“应急通道”，而是完整的流程绕过能力。

**证据**

- `src/core/process-engine/advance.ts` 在 `options.force` 分支直接写入 `FORCE_SKIPPED`，跳过依赖与 Gate，再持久化新阶段。
- 当前运行实例 `FSREQ-20260306-DASHBOARD-001` 已从 `04_implement` 连续四次 `FORCE_SKIPPED` 推进到 `08_done`。
- 同一实例的 Gate 历史显示：
  - `04_implement` 时 C4=0%，`pytest` 失败
  - `05_verify` 时 C4/C5 未达标，diff-cover 命令失败
  - `06_wrap_up` 时 C6=0%，矩阵非终态
  - `07_release` 时 smoke test/release note 缺失
  - 尽管如此，仍推进到 `08_done`

**影响**

- “Done” 不再代表质量闭环完成，而只代表有人执行过 `--force`
- 所有依赖 `currentStage=08_done` 的下游流程都会被污染
- 审查、统计、viewer、恢复提示都会基于错误终态继续工作

**建议**

- P0：禁止 `04_implement` 之后使用 `--force` 推进，至少禁止 `05_verify`、`06_wrap_up`、`07_release`、`08_done` 路径被强制推进
- P0：将 `--force` 改为写入“override requested / blocked”状态，而不是直接推进阶段
- P0：若确需保留，应强制记录审批人、原因、过期时间，并使 Feature 标记为 `PASS_WITH_OVERRIDE` 而非 `08_done`

### HIGH-01: hard-gate 只是提示文本，不是强制执行；Skill 约束无法真正阻断错误流程

**结论**

当前 hard-gate 机制只是把“警告/阻断说明”拼接到 Skill prompt 前面，没有在 runtime 层真正拒绝执行。Skill 文本里的“阻断”“必须”“禁止”多数仍是约定而非约束。

**证据**

- `src/core/skill-runtime/dispatcher.ts` 只把 `buildHardGateRuntimeNotice()` 的返回值拼接到内容前部
- `src/core/skill-runtime/hard-gate.ts` 虽然能给出 `allowed=false` / `severity=BLOCKED`，但 `buildHardGateRuntimeNotice()` 只是返回字符串，没有抛错或中止
- `skills/spec-first/08-code-review/SKILL.md` 要求仅在 `04_implement` 阶段执行，且必须“先合规后质量”
- 但 `src/core/skill-runtime/hard-gate.ts` 的 `HARD_GATE_STAGE_REQUIREMENTS` 只覆盖 `design` 与 `code`，并不覆盖 `code-review`、`verify`、`archive`

**影响**

- 文档声明与运行时行为脱节，用户会误以为系统已经具备流程硬约束
- `code-review` / `verify` / `archive` 等关键 Skill 可在错误阶段被调用
- “Codex 侧以流程规则拒绝已完成/可推进声明”的总原则无法由代码保证

**建议**

- P0：把 hard-gate 从“提示注入”改为“真实阻断”，在 `BLOCKED` 时直接返回错误
- P0：把 `code-review`、`verify`、`archive`、`spec-review` 纳入 stage hard-gate
- P1：将 Skill 声明中的关键协议映射到可执行 runtime 检查，不再依赖提示语

### HIGH-02: 平台规则合并只校验 YAML 名称，不校验实际项目存在，导致假阻断并诱发流程绕过

**结论**

Layer 2 平台规则是“声明驱动”而非“项目存在驱动”。系统会无差别把所选平台 YAML 中的命令 Gate 合并进运行态，但不会验证当前仓库是否真的存在对应平台实现。

**证据**

- `src/cli/commands/init.ts` 的平台校验只检查 `.spec-first/layer2/*.yaml` 是否存在
- `src/core/process-engine/layer-merger.ts` 会对所选 `platforms` 逐个加载 YAML 并合并命令 Gate，没有任何仓库结构探测
- 当前实例的 `stage-state.json` 声明了 `admin-frontend` + `python-backend`
- 因而注入了 `ruff check .`、`pytest`、`pytest --cov ... && diff-cover ...`
- Gate 历史中，这些命令在当前 TypeScript 仓库内失败，直接造成 `04_implement` 与 `05_verify` Gate 失败

**影响**

- Gate 失败并不一定代表质量问题，可能只是平台声明与仓库结构不匹配
- 用户为了越过假失败，更容易转向 `--force`
- 多平台能力看起来完整，实际缺少“平台声明真实性”校验

**建议**

- P0：在 init / merge 阶段增加平台探测与 capability check，未探测到对应代码根时拒绝合并命令 Gate
- P1：将平台 Gate 与具体工作目录、依赖存在性、命令可用性绑定，而不是固定在仓库根目录执行
- P1：为“声明存在但当前工作区未挂载”的平台引入 `inactive` 状态，而不是直接 FAIL

### HIGH-03: 矩阵状态的 canonical 定义不统一，`done` 会被 Gate 接受但不在类型系统中

**结论**

矩阵状态在类型、解析器、Gate、文档和运行实践之间并不统一。`done` 不是 `MatrixStatus` 的合法值，但解析器会无校验接受，`06_wrap_up` Gate 还把它当成终态。

**证据**

- `src/shared/types.ts` 的 `MatrixStatus` 定义仅包含 `Planned / Implemented / Verified / Accepted / Deferred / Cancelled / Exception`
- `src/core/trace-engine/matrix.ts` 解析状态时直接 `as MatrixStatus`，没有枚举校验
- `src/core/gate-engine/gate-evaluator.ts` 的 `G-WRAP-02` 终态集合包含 `done`
- 仓库内已有分析文档记录“手动更新 traceability-matrix.md 状态为 done”

**影响**

- 非法状态值可悄悄穿透类型系统，进入 Gate 决策
- Wrap-up 可能在 undocumented 状态值上通过，降低追踪矩阵可信度
- “Task 状态 done” 与 “Matrix 状态 Accepted” 的两个语义层被混用

**建议**

- P0：解析矩阵状态时做严格枚举校验，非法值直接报错
- P0：删除 `done` 作为矩阵终态的兼容逻辑，统一采用 `Accepted / Cancelled / Exception`
- P1：把 todo-runner 状态与 traceability matrix 状态做显式映射，不再共享词汇表

### MEDIUM-01: 运行态文件已经出现双写状态，`stage-state.json` 不再是单一真理源

**结论**

当前实例的 `stage-state.json` 同时存在 `currentStage=08_done` 与残留字段 `current_stage=04_implement`。虽然现有代码读取的是 `currentStage`，但运行态文件本身已经失去单义性。

**证据**

- `specs/FSREQ-20260306-DASHBOARD-001/stage-state.json` 尾部残留 `current_stage`
- 全仓搜索只有该运行态文件包含 `current_stage`，说明它不是当前 schema 的合法字段
- `skills/spec-first/AGENTS.md` 明确要求运行态文件是动态 Prompt 的统一来源

**影响**

- 人工审查会得到矛盾结论
- 外部脚本、未来迁移工具或其它宿主若读取旧字段，可能恢复到错误阶段
- 审计对象被污染时，系统很难证明“谁是权威字段”

**建议**

- P1：对 `stage-state.json` 使用严格 schema 校验并在写回时清理未知字段
- P1：增加“运行态自愈/doctor”命令，扫描并清洗残留字段

## 5. 正向结论

- 近期 `trace-context` / `ratio` / `coverage` 重构在当前聚焦单测中全部通过，没有发现直接回归
- `FR -> TASK` 传递链计算抽象为 `createTraceContext()` 后，`coverage`、`sca`、`gate-evaluator` 的内部复用度更高，局部代码质量比改动前更整齐
- 当前主要风险不在“近期重构把算法写坏了”，而在“系统治理层允许错误流程继续向前”

## 6. 结论判定

对“流程流转顺畅、健壮性高、质量高”的目标，本次判定如下：

- 流程顺畅性：表面顺畅，实质存在绕过通道，不合格
- 健壮性：核心质量门禁可被绕开，不合格
- 质量：单元层局部稳定，但系统级质量闭环不成立，不合格

总体结论：**当前系统更像“有完整流程外观的可绕过引擎”，尚不能作为严格可信的全链路质量闭环。**

## 7. 优先级行动建议

### P0（必须先做）

1. 收紧 `stage advance --force`，禁止推进到 `05_verify` 之后的阶段
2. 把 hard-gate 从提示改为真实阻断，并把 `code-review` / `verify` 纳入约束
3. 对矩阵状态做严格校验，移除 `done` 作为矩阵终态的兼容
4. 复盘并回滚当前 `FSREQ-20260306-DASHBOARD-001` 的无效完成态

### P1（紧随其后）

1. 给平台 YAML 增加 capability check，避免空平台/假平台命令 Gate
2. 清洗 `stage-state.json` 非 schema 字段，确保运行态单一真理源
3. 为“override / waiver / forced advance”建立独立状态与审计模型，不再复用正常 Done

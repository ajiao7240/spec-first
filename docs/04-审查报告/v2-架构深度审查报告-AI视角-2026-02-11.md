## 1. 审查摘要

### 1.1 总体评价

Spec-First V2 是一套**高度完整且内部自洽的规范驱动研发流程引擎**。需求文档（12 篇）与技术方案（14 篇）之间的对齐度达到了**工业级水准**，核心概念（双层架构、6-ID 体系、8+2 阶段、三层规范、Gate 三态语义）在两侧文档中保持了高度一致。

**核心优势**：
1. **架构分层清晰** — Skill（编排）+ CLI（执行）的双层架构是 AI-Native 工具链的正确范式
2. **追踪体系完备** — 6-ID + 9 覆盖率 + 3 合规率形成了闭环追踪网络
3. **裁剪机制灵活** — Mode x Size x Platform 三维裁剪避免了"一刀切"的流程僵化
4. **降级策略务实** — 从 confirm_policy 到 Gate 降级到 Go-Live 分阶段，处处体现渐进式推行思维

**核心风险**：
1. **复杂度天花板** — 16 Skill + 13 命令组 + 7 模块 + 12 指标，对 10-50 人团队而言认知负荷偏高
2. **AI 编排层缺乏形式化** — Skill 层依赖 .md 文件 + AI 推理，缺少可测试的编排状态机
3. **Context Pack 实际效果存疑** — 8K token 预算在复杂 Feature 中可能严重不足
4. **文档驱动的追踪矩阵** — Markdown 表格作为追踪矩阵的持久化格式，在并发编辑和大规模场景下脆弱

## 3. 关键发现（P0-P3）

### 3.1 P0 Critical（必须修复）

#### P0-01: Markdown 追踪矩阵在并发场景下的数据完整性风险

**来源**: core-03-traceability + v2-05-追踪矩阵与ID引擎

**问题描述**:
追踪矩阵（`traceability-matrix.md`）是整个系统的核心数据结构，所有覆盖率计算（C1-C9）、Gate 校验、SCA 校验都依赖它。然而，该矩阵以 Markdown 表格格式存储，存在以下风险：

1. **并发写入冲突**: 多个开发者同时执行 `spec-first id next` 注册新 ID 时，可能产生 Git 合并冲突
2. **解析脆弱性**: Markdown 表格对格式敏感，手动编辑（文档允许）可能破坏表格结构导致解析失败
3. **性能瓶颈**: Size L 的 Feature 可能有 50+ FR、100+ TASK，Markdown 表格的全量读写在 `getCoverage < 50ms` SLA 下可能不达标

**影响范围**: M2 TraceEngine 的所有读写操作、M3 GateEngine 的所有评估操作、M6 MetricsEngine 的所有计算操作

**建议**:
- 短期：增加文件锁机制（`lockfile`），防止并发写入
- 中期：内部使用 JSON/YAML 作为规范化存储格式，Markdown 仅作为导出视图
- 长期：考虑 SQLite 作为本地持久化方案（单文件、零配置、支持事务）

#### P0-02: Skill 编排层缺乏可测试的形式化定义

**来源**: aux-01-skill-system + v2-02-skill运行时与路由

**问题描述**:
Skill 层是整个系统的"指挥官"，负责流程编排和阶段流转触发。然而：

1. **Skill 行为不可测试**: Skill 以 `.md` 文件定义，执行依赖 AI 推理，无法编写确定性单元测试。v2-03 的测试策略中明确写了 CLI 层有"单元测试 + 集成测试 + 快照测试 + 基准测试"，但 Skill 层仅有"通过场景验证"
2. **编排状态机隐式化**: `/spec-first:orchestrate` 内部按阶段状态机调度阶段 Skill，但这个编排逻辑完全隐含在 .md 文件的自然语言指令中，没有形式化的状态转换定义
3. **Phase 3 交互协议的执行保证**: 确认口令标准化、最大修订轮次等约束写在文档中，但 AI Agent 是否严格遵守取决于 LLM 的指令遵循能力，无硬性保证

**影响范围**: 所有 16 个 Skill 的执行正确性、编排 Skill 的调度逻辑、Phase 3 的用户交互安全性

**建议**:
- 为编排 Skill 的调度逻辑提取一个可测试的 TypeScript 状态机（`src/core/skill-orchestrator/`），Skill .md 文件调用该状态机而非自行推理
- 为 Phase 3 交互协议实现一个 CLI 侧的 `confirmationGuard` 中间件，在 Phase 4 写入前强制校验确认状态
- 增加 Skill E2E 测试框架：模拟 AI Agent 输入，验证 CLI 调用序列和产出物正确性

### 3.2 P1 High（强烈建议修复）

#### P1-01: Context Pack 8K token 预算在 Size L 场景下严重不足

**来源**: aux-01-skill-system Context Slicing + v2-08-context-pack

**问题描述**:
需求文档和技术方案均定义了 8K token 的推荐预算（L1 <= 20%, L2 <= 30%, L3 >= 50%）。但对于 Size L 的 Feature（6+ 模块、16+ AC、6+ API），仅 spec.md 一个文件就可能超过 8K token。降级策略虽然定义了三级裁剪，但最终降级到"control 仅保留 ID 列表 + 正文通过 references 按需拉取"时，AI Agent 需要多次文件读取才能恢复完整上下文，这与"减少 AI 重复劳动"的设计目标（core-01 问题 #8）相矛盾。

**建议**:
- 将推荐预算从 8K 提升到 16K-32K（当前主流 LLM 上下文窗口已达 128K-200K）
- 为 Size L 场景设计"摘要 + 全文"双模式：AI 先读摘要决定需要哪些全文，再按需加载
- 在 `config.yaml` 中允许用户自定义 token 预算上限

#### P1-02: Layer A (AI Runtime Hook) 缺乏技术实现路径

**来源**: core-05-cross-cutting Hook 双层体系 + v2-06-质量门禁与豁免 S7

**问题描述**:
需求文档定义了 Layer A（AI Runtime Hook）包含 PreToolUse / PostToolUse / Stop 三种 Hook 类型，技术方案 v2-06 也提到"先实现 Layer B，再补 Layer A"。但整个技术方案体系中，**没有任何一篇文档定义 Layer A 的技术实现路径**：
1. PreToolUse / PostToolUse 是 Claude Code Hooks 的特有机制，如何在 Codex CLI 等其他宿主上实现？
2. Layer A 的 Hook 注册、触发、阻断机制没有对应的 TypeScript 接口定义
3. v2-12 的三阶段实施中，Layer A 未出现在任何阶段的交付项中

**建议**:
- 在 v2-06 或 v2-07 中补充 Layer A 的技术实现方案（至少包含 Claude Code Hooks 的注册方式）
- 在 v2-12 阶段 C 中明确 Layer A 的交付计划
- 如果 Layer A 短期不实现，应在需求文档中将其标记为"规划中"而非"强制"

#### P1-03: 认知负荷过高 — 16 Skill + 13 命令组的用户心智模型

**来源**: aux-01-skill-system + aux-02-cli-system + v2-02 + v2-03

**问题描述**:
系统定义了 16 个 Skill + 13 个 CLI 命令组（35 个子命令），加上 Skill 路由 vs Runtime 路由的双入口模型，用户需要理解：
1. 哪些命令走 Skill 路由（16 个），哪些走 Runtime 路由（9 个）
2. 编排 Skill 与阶段 Skill 的层级关系
3. `/spec-first:verify quick` vs `/spec-first:verify full` 的差异
4. `spec-first gate check` vs `/spec-first:verify` 的区别（前者是 CLI 原子操作，后者是 AI 辅助评估）

虽然文档提供了"3 分钟上手"的 Daily Path（init -> plan -> code -> code-review -> verify），但实际使用中用户仍需频繁查阅命令速查表。

**建议**:
- 考虑将 Runtime 路由命令也封装为 Skill 的子能力，用户只需记住 Skill 入口
- 提供 `/spec-first:help` 交互式命令导航（类似 `gh` CLI 的交互模式）
- 在 `status` Skill 输出中自动推荐下一步可执行的命令

#### P1-04: M5 AIOrchestrator 类型签名漂移已被标记但无修复计划

**来源**: aux-02-cli-system M5 交付状态 + v2-08-context-pack

**问题描述**:
需求文档 aux-02 明确标记 M5 AIOrchestrator 为"Partial（类型签名漂移）"，且 `npm run typecheck` 当前不通过。技术方案 v2-08 定义了完整的 Context Pack 和 Catchup 接口，但未提及类型漂移的修复方案。v2-12 阶段 B 将 M5 列为交付项，验收标准为"签名与核心模块一致，typecheck 归零"，但缺少具体的修复路径。

**建议**:
- 在 v2-08 中补充一节"已知类型漂移清单"，列出具体的签名不一致项
- 在 v2-12 阶段 B 的 M5 交付项中增加"类型对齐"子任务

#### P1-05: 缺陷管理（Defect）无独立 ID 体系，与 6-ID 模型不一致

**来源**: core-03-traceability + v2-05-追踪矩阵与ID引擎 + v2-07-变更管理

**问题描述**:
v2-05 明确写道"DEF/EXC 无独立 ID：缺陷用 Feature 内自增序号（#1, #2...）"。然而：
1. 缺陷是质量管理的核心实体，使用简单序号（#1, #2）无法跨 Feature 唯一标识
2. 缺陷逃逸率（Q1）是 12 项核心指标之一，但缺陷记录缺乏结构化 ID 使得跨 Feature 的缺陷趋势分析困难
3. v2-07 的缺陷状态转换表定义了完整的 6 态 FSM（Open -> Confirmed -> Fixing -> Fixed -> Verified -> Closed），这种复杂度暗示缺陷应有更正式的标识

**建议**:
- 短期保持现状（KISS 原则），但在 v2-07 中明确"缺陷序号的作用域为 Feature 内"
- 中期考虑引入 `DEF-<FEAT>-NNN` 格式，纳入 ID 体系（升级为 7-ID 模型）

### 3.3 P2 Medium（建议改进）

#### P2-01: Skill 文件的开发态与部署态目录映射未文档化

**来源**: aux-01-skill-system + v2-01-总体技术架构

**问题描述**:
需求文档定义 Skill 部署目录为 `.claude/commands/spec-first/<skill-name>.md`，技术方案定义开发态目录为 `skills/spec-first/<NN>-<skill-name>/SKILL.md`。两者是同一文件的不同视角，但没有任何文档说明构建/部署时如何从开发态映射到部署态（是符号链接、复制、还是构建脚本转换）。

**建议**: 在 v2-02 或 v2-10 中补充 Skill 文件的构建与部署流程说明。

#### P2-02: 07_release 阶段的 Gate 条件依赖外部系统但无降级方案

**来源**: core-05-cross-cutting + v2-06-质量门禁与豁免

**问题描述**:
07_release 的 Gate 条件包含"错误率 <= 基线 110%"、"P95 延迟 <= 基线 120%"、"回滚率 = 0"，这些指标依赖 APM/日志平台等外部系统。但技术方案中 M3 GateEngine 的接口定义（`evaluateGate`）未包含外部系统集成的接口，也未定义当 APM 不可用时的降级策略。

**建议**: 在 v2-06 中为 07_release Gate 增加"外部指标不可用时的降级策略"（如：降级为人工确认 + 审计记录）。

#### P2-03: Session Catchup 的"只读"约束与"自动触发"存在潜在冲突

**来源**: aux-01-skill-system + v2-08-context-pack

**问题描述**:
文档要求自动触发的 Catchup 为"只读操作，不产生写入副作用"。但 Catchup 第 7 步"输出恢复摘要到终端"后，如果检测到 `current_phase` 与 `stage-state.json` 不一致，仅输出 warning。在 `auto` 策略下，用户可能未注意到 warning 就继续操作，导致后续 Skill 在错误的阶段上下文中执行。

**建议**: 当 Catchup 检测到不一致时，除了 warning 外，应在 `.spec-first/` 下写入一个 `catchup-warning.flag` 文件，后续 Skill 的 Phase 0 检测到该 flag 时强制提示用户确认。

#### P2-04: Code Review 的 A+B 9 维度审查标准缺乏权重定义

**来源**: core-04-process + code-review-integration.md

**问题描述**:
Code Review 定义了 A 类（4 维度追踪合规）+ B 类（5 维度代码质量）共 9 个审查维度，Gate 通过判定为 `APPROVE = A_all_pass AND P0==0 AND P1==0`。但 B 类 5 个维度之间没有权重区分 -- B2（安全与可靠性）和 B5（边界条件）在 Gate 判定中被同等对待，而实际上安全问题的严重性远高于边界条件问题。

**建议**: 为 B 类维度增加权重或优先级排序，至少将 B2（安全）标记为"优先审查"维度。

#### P2-05: 健康分公式的权重分配缺乏数据支撑

**来源**: aux-05-metrics + v2-11-度量与健康分

**问题描述**:
健康分 H1 的 9 项权重（w1-w9）均为经验值（0.10-0.15），文档注明"可通过 config.yaml 覆盖"。但缺乏权重选择的理论依据或历史数据支撑。例如 C4（Test Coverage FR）权重 0.15 而 C5（Test Coverage AC）权重 0.10，为何 FR 级覆盖比 AC 级覆盖更重要？

**建议**: 在 v2-11 中补充权重选择的决策理由（即使是经验判断也应记录），并在阶段 C 中基于实际项目数据校准权重。

#### P2-06: 运行态三文件的"追加写入"模式缺乏容量治理

**来源**: aux-04-deliverables + v2-10-产出物与模板系统

**问题描述**:
运行态三文件（stage-state.json / findings.md / task_plan.md）采用纯追加写入模式，v2-10 明确写道"不做结构化解析，仅供人类阅读和 Catchup 恢复使用"。但对于 Size L 的 Feature（可能持续数周），这三个文件会持续膨胀。当 Catchup 需要读取这些文件恢复上下文时，文件过大会：
1. 增加 Context Pack 的 token 消耗（与 8K 预算冲突）
2. 降低 AI Agent 的恢复准确性（信噪比下降）
3. 在 Git 中产生频繁的大文件变更

**建议**: 为运行态文件增加"滚动归档"机制 — 当文件超过阈值（如 500 行）时，将历史内容归档到 `<filename>-YYYY-MM-DD-<ts>.md`，仅保留最近 N 个阶段的记录供 Catchup 使用。

#### P2-07: Layer 2 端规范的阈值类型推断规则过于脆弱

**来源**: core-02-architecture + v2-09-多端规范合并

**问题描述**:
v2-09 定义了多端规范合并时 `quality_thresholds` 的"更严格"比较语义，判定规则为"阈值名称中含 `max/size/latency/time` 取较小值；含 `min/score/rate/coverage` 取较大值"。这种基于命名约定的类型推断存在风险：
1. 用户自定义阈值名称可能不符合命名约定（如 `error_threshold` 应取较小值，但名称中无 `max/size/latency/time` 关键词）
2. 文档虽然写了"无法自动判定时阻断并提示用户手动指定"，但未定义手动指定的格式（在 YAML 中如何标注比较方向？）

**建议**: 在 Layer 2 YAML 格式中为每个阈值增加显式的 `direction` 字段（`lower_is_better` / `higher_is_better`），命名推断仅作为缺省值。

#### P2-08: 模板系统缺少版本管理与升级迁移策略

**来源**: aux-04-deliverables + v2-10-产出物与模板系统

**问题描述**:
v2-10 定义了 7 个 Handlebars 模板，渲染规则为"文件已存在则跳过，不覆盖用户修改"。但当模板本身升级时（如 `stage-state.json.hbs` 新增字段），已初始化的 Feature 不会自动获得新字段。文档未定义：
1. 模板版本号管理机制（如何知道当前 Feature 使用的是哪个版本的模板？）
2. 模板升级时的迁移策略（是否需要 `spec-first migrate` 命令？）
3. 新旧模板共存时的兼容性保证

**建议**: 在 `stage-state.json` 中增加 `template_version` 字段，当 CLI 检测到版本不匹配时输出 warning 并提供 `doctor` 级别的修复建议。

### 3.4 P3 Low（可选优化）

#### P3-01: aux-02 开头的"12 个命令组"数字未更新为 13

**来源**: aux-02-cli-system vs v2-03-cli命令与核心模块设计

**问题描述**:
需求文档 aux-02 标题区域仍写"12 个命令组"，但正文已包含 golive 命令组的定义（共 13 个）。技术方案 v2-03 标题明确写"13 命令组"。这是一个纯文档表述不一致，不影响功能。

**建议**: 更新 aux-02 标题中的数字为 13。

#### P3-02: UC-012 用例表简写未体现 RFC create 完整参数

**来源**: core-01-overview 用例表 vs v2-07 变更管理

**问题描述**:
core-01 用例表中 UC-012 写 `spec-first rfc create`，而 v2-03 的完整签名含 `--level <Minor|Major|Critical>` 等多个参数。用例表作为概览性质的简写是合理的，但可能误导读者认为 `rfc create` 无需额外参数。

**建议**: 在用例表的"对应 Skill/CLI"列中补充 `(详见 v2-03 §2.8)` 的交叉引用。

#### P3-03: constitution.md 的"复制副本"策略可能导致版本漂移

**来源**: aux-04-deliverables + v2-10-产出物与模板系统

**问题描述**:
v2-10 定义 `.spec-first/constitution.md` 为项目级单例，Feature 目录下为"复制副本"（`00_init` 生成时复制）。如果项目宪法在某个 Feature 生命周期中被更新，已初始化的 Feature 目录下的副本不会自动同步，可能导致不同 Feature 引用不同版本的宪法。

**建议**: 考虑使用符号链接（symlink）替代复制，或在 `doctor` 命令中增加宪法版本一致性检查。

#### P3-04: FEAT 注册表的并发注册缺乏原子性保证

**来源**: core-03-traceability + v2-05-追踪矩阵与ID引擎

**问题描述**:
FEAT 注册表（`specs/.feat-registry.md`）用于确保 FEAT 缩写全局唯一。但当两个开发者同时执行 `spec-first init` 注册不同 FEAT 缩写时，可能出现竞态条件：两者都读取到当前注册表无冲突，然后各自写入，最终 Git 合并时才发现冲突。虽然 Git 合并冲突会暴露问题，但用户体验不佳。

**建议**: 在 `init` 命令中增加 `lockfile` 机制，或在 `doctor` 命令中增加 FEAT 注册表唯一性校验。

#### P3-05: `stage advance --force` 的滥用风险缺乏审计闭环

**来源**: core-04-process + v2-04-流程引擎与阶段状态机

**问题描述**:
`stage advance --force` 允许跳过 Gate 校验强制推进，文档要求"需人工确认，写入 findings.md 记录"。但 `--force` 的使用没有频率限制或升级机制 -- 理论上一个 Feature 可以全程 `--force` 推进到 `08_done`，完全绕过质量门禁。aux-06-roadmap 的风险矩阵中也提到了 `--force` 有滥用风险。

**建议**: 增加 `--force` 使用次数统计，当同一 Feature 累计使用超过 2 次时，自动升级为 RFC 审批流程。

#### P3-06: ExitCode 5（UNKNOWN_ERROR）缺乏结构化错误上报机制

**来源**: aux-02-cli-system + v2-03-cli命令与核心模块设计

**问题描述**:
v2-03 定义了 6 档 ExitCode（0-5），其中 ExitCode 5 为 UNKNOWN_ERROR（未预期异常）。但文档未定义 UNKNOWN_ERROR 的错误上报机制 -- 当用户遇到 ExitCode 5 时，除了终端输出的错误信息外，没有结构化的错误日志可供排查。对于一个面向 10-50 人团队的工具，缺乏错误上报会增加维护成本。

**建议**: 在 `doctor` 命令中增加"最近错误日志"查看功能，UNKNOWN_ERROR 发生时自动写入 `.spec-first/error.log`。

## 5. 技术债务预警

> 以下 4 项技术债务虽然不影响当前文档的正确性，但如果在实现阶段不加以关注，可能在中长期演变为架构级问题。

### TD-01: Markdown 作为核心数据格式的可扩展性天花板

**关联发现**: P0-01

**债务描述**:
追踪矩阵（traceability-matrix.md）、运行态文件（findings/task_plan）与阶段状态文件（stage-state.json）、豁免清单（known-exceptions.md）均以文件形式存储。当前设计选择 Markdown + JSON 的理由是"人类可读 + Git 友好"，这在 Size S 场景下完全合理。但随着使用规模扩大：

1. **查询性能**：覆盖率计算需要全量解析 Markdown 表格，Size L 场景下可能无法满足 `getCoverage < 50ms` 的 SLA
2. **并发安全**：多人同时操作同一 Feature 时，Markdown 文件的读写缺乏事务保证
3. **结构化查询**：跨 Feature 的聚合查询（如"所有 Feature 的平均健康分"）需要遍历所有目录

**演进建议**:
- 阶段 A/B：保持 Markdown 格式（KISS），但内部实现使用 AST 缓存减少重复解析
- 阶段 C：引入 JSON 作为规范化存储格式，Markdown 仅作为导出视图
- vNext：评估 SQLite 作为本地持久化方案（单文件、零配置、支持事务和索引）

### TD-02: Skill 层与 CLI 层的契约缺乏形式化验证

**关联发现**: P0-02, P1-04

**债务描述**:
Skill 层通过 .md 文件中的自然语言指令调用 CLI 命令，两层之间的契约是隐式的。当 CLI 命令签名变更时（如新增参数、修改返回格式），Skill .md 文件中的调用指令不会自动感知变更。M5 AIOrchestrator 当前已出现的"类型签名漂移"就是这一债务的早期症状。

**演进建议**:
- 短期：在 CI 中增加 Skill .md 文件的 CLI 命令引用校验（正则扫描 .md 中的 `spec-first` 命令调用，与 v2-03 签名表交叉验证）
- 中期：为 Skill .md 文件定义结构化的"CLI 依赖声明"区块，工具链可自动校验
- 长期：考虑 Skill DSL（领域特定语言）替代纯自然语言指令

### TD-03: 单宿主绑定风险（Claude Code 依赖）

**关联发现**: P1-02

**债务描述**:
当前架构深度绑定 Claude Code 生态：Skill 文件部署到 `.claude/commands/`，Layer A Hook 依赖 Claude Code Hooks（PreToolUse/PostToolUse/Stop），confirm_policy 的 `auto` 策略依赖 Claude Code 的 AI Agent 能力。虽然需求文档 aux-01 提到了"跨平台兼容"（Claude Code / Codex CLI / Cursor），但技术方案中没有任何抽象层来隔离宿主差异。

**演进建议**:
- 短期：在 v2-02 中增加"宿主适配层"的接口定义（即使只实现 Claude Code 适配器）
- 中期：为 Codex CLI 实现第二个适配器，验证抽象层的可行性
- 长期：考虑 LSP（Language Server Protocol）风格的标准化协议

### TD-04: 度量体系的"自我评估"悖论

**关联发现**: P2-05

**债务描述**:
Spec-First 的度量体系（12 项指标 + 健康分）用于评估 Feature 的研发质量，但这些指标本身是由 Spec-First 工具链计算的。这意味着：

1. **指标可被游戏化**：开发者可以通过"形式合规但实质空洞"的方式达标（如创建空壳 TC 提升覆盖率）
2. **健康分缺乏外部校准**：H1 健康分的权重是内部经验值，没有与外部质量指标（如生产缺陷率、客户满意度）的关联验证
3. **瓶颈规则的触发阈值缺乏自适应**：R1-R5 的触发条件是固定值，不会随团队成熟度调整

**演进建议**:
- 阶段 C：增加"指标质量审计"机制（如 TC 内容非空校验、PR diff 行数与 TASK 粒度的合理性检查）
- vNext：将健康分与外部质量指标（生产缺陷率、MTTR）关联，建立校准反馈环

## 7. 审查结论

### 7.3 架构亮点（值得推广的设计）

1. **Gate 三态语义 + 豁免快通道**：`PASS / PASS_WITH_WAIVER / FAIL` 三态模型比传统的二元 Pass/Fail 更贴近工程实际。豁免必须关联 RFC + 到期时间 + 回滚点的设计，在"灵活性"与"可审计性"之间取得了优秀的平衡。这一设计可推广到任何需要质量门禁的研发流程中。

2. **Skill 驱动 + CLI 执行的双层架构**：将"编排智能"（Skill 层，依赖 AI 推理）与"执行确定性"（CLI 层，纯 TypeScript）分离，是 AI-Native 工具链的正确范式。Skill 层可以随 LLM 能力提升而演进，CLI 层保持稳定可测试。这一分层思想值得在其他 AI 辅助工具中借鉴。

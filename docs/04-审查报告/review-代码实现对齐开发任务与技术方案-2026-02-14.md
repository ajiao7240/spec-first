## ⚠️ 关键差距清单（按优先级）

### P0：会直接破坏“质量闭环可信度”的差距

1) Gate 条件来源未按文档落地  
- 文档要求：`gate check` 需读取 `stage-state.json` 中 **合并后的 Gate 条件（Layer 0/1/2）** 并逐项评估（`v2-06 §2`，`v2-03 §2.4`）。  
- 当前实现：`src/core/gate-engine/gate-evaluator.ts` 使用硬编码 `GATE_CONDITIONS`；与 `stage-state.json.mergedRules.gateConditions` 脱钩。  
- 影响：Layer2 平台规范、Mode/Size 裁剪对 Gate 的影响无法生效；“配置存在但 Gate 不认”，会导致误放行/误阻断。

2) SCA 未纳入 Gate 判定（只存在“孤立实现”）  
- 文档要求：SCA FAIL 等同 Gate FAIL，且写入 Gate 条件评估结果（`v2-06 §4`，`phase-B` T-BM3-002）。  
- 当前实现：`src/core/gate-engine/sca.ts` 已实现，但 `evaluateGate()` 未调用；`golive.ts` 才调用 `runSca()`。  
- 影响：阶段推进可能绕开一致性审计，Gate 的核心价值被削弱。

3) ExitCode 语义不一致（Gate 失败未返回 `GATE_FAILED=1`）  
- 文档要求：Gate 失败（含 GoLive）返回 `ExitCode.GATE_FAILED`（`v2-03`、`src/shared/types.ts`）。  
- 当前实现：`src/cli/commands/gate.ts` 对 FAIL 多数返回 `VALIDATION_ERROR=2`。  
- 影响：Hook/CI 生态无法通过 ExitCode 精确区分“质量门禁失败”与“参数/输入校验失败”，后续自动化策略会变复杂且易误判。

4) CLI 签名/参数与文档不一致  
- 文档要求：`spec-first gate check <featureId> [--stage <stageId>]`，`gate conditions` 同理（`v2-03 §2.4`）。  
- 当前实现：`src/cli/commands/gate.ts` 不支持 `--stage`，仅检查当前阶段。  
- 影响：无法在 CI/回溯/对比时指定阶段评估；也不利于 Debug 与回放。

5) GoLive 入口/审计落点与文档不一致  
- 文档要求：`spec-first golive check`（无 featureId），写入 `specs/_global/golive-history.jsonl`，并包含外部指标源 fallback（`v2-03 §2.13`、`v2-06 §3.1`）。  
- 当前实现：`src/core/gate-engine/golive.ts` 需要传 `featureId`；未写全局 JSONL；外部指标源（错误率/P95/回滚率）未实现，只读取 `reports/security-scan.md`。  
- 影响：上线门槛无法形成全局审计链；GoLive 语义会被误用成“Feature 局部检查”。

6) Hook “可安装但不可用/不可操作”的问题集中爆发  
- `doctor` 提示 “Run spec-first hooks install”，但 CLI 没有 `hooks` 命令入口（仅存在 `src/core/tool-integration/hook-installer.ts`）。  
- `prepare-commit-msg` 解析 `task_plan.md` 的列位置疑似错误：当前脚本 `awk -F'|' '{print $2}'` 会更可能取到序号列而非 TASK ID 列。  
- 影响：提交前缀/追踪 trailer 失真，直接破坏 Traceability 的最小闭环。

### P1：会导致“机制名义存在但执行不到位”的差距

7) known-exceptions / waiver schema 与 `v2-06 §6` 偏差较大  
- 文档要求：`known-exceptions.md` 是豁免主文件，`waiver_id=WVR-NNN`，字段极全（owner/risk/mitigation/approved_by/expires_at 等），并有上限约束（豁免数量≤10%，`expires_at` 必填等）。  
- 当前实现：`src/core/trace-engine/exception-validator.ts` 仅校验少数字段；`src/core/change-mgr/rfc.ts` 回填使用 `EX-xxx`，且字段集合不同。  
- 影响：豁免机制很容易被“滥用且无法治理”；到期/回滚/审计链条不完整。

8) Context Pack 与 `v2-08` 的关键约束未严格执行  
- 文档要求：control 区 **<2KB 超限失败**；references 需包含 checksum+mtime；L3 邻居需基于矩阵 depth=1；token budget 来自 `config.yaml`（8K-64K）。  
- 当前实现：`src/core/ai-orchestrator/context-pack.ts` 计算了 controlSize，但未在构建时强制失败；references 主要是“阶段文件列表”，未实现“矩阵邻居”；checksum 截断为 16 hex；`validateConsistency()` 未落地。  
- 影响：一旦真实项目变大，Context Pack 会出现“看似生成、实际不可控/不可校验/不可恢复”的问题。

9) 配置默认值与使用方式存在漂移  
- `catchup.trigger` 默认值为 `prompt`，与 `v2-08` 的 `auto` 默认不一致。  
- `health.weights` 在 `config-schema.ts` 定义但在 `health-score.ts` 未使用，属于“配置存在但不生效”。  
- 影响：用户对配置的心智模型会被破坏（改了不生效最致命）。

10) Skill Runtime 分发态隐患  
- `src/core/skill-runtime/dispatcher.ts` 使用 `import.meta.dirname`，该字段不是标准 Node ESM 能力；在“安装包运行（无本地 skills/）”场景下可能直接失效。  
- 语义映射存在可疑项（如 `defect fix` → `transition`，但 CLI 更像是 `update --status`）。  
- 影响：Skill 路由/运行时在真实宿主（Codex CLI/Claude Code）中稳定性存疑。

### P2：偏技术债/一致性问题

11) 技术栈与文档不一致：Markdown 解析未使用 `remark + unified`  
- 文档约束：`remark + unified`（`docs/03开发任务/README.md` 技术栈约束）。  
- 当前实现：多数解析依赖字符串/表格扫描。  
- 影响：短期 KISS 成本低；长期遇到复杂 Markdown/表格时会出现边界 bug（需要决定“以文档为准”还是“修订文档以实现为准”）。

---

## 🔀 可选路径（治理策略）

1) 以“文档为准”回收实现语义（推荐）  
优点：能把 Gate/豁免/GoLive/Context Pack 做成可信的“硬机制”；适合后续 CI/Hook/多端扩展。  
代价：需要一轮集中修正 P0/P1 偏差，涉及多个模块联动。

2) 以“实现为准”回收文档与开发任务  
优点：保持当前 KISS 实现；避免引入更多解析/治理复杂度。  
代价：必须重写 `v2-06/v2-08` 的约束与验收标准，否则团队会长期处于“对照文档永远不通过”的状态。

---

## 🎯 我的建议（结论）

建议选择路径 1（以文档为准）先收敛 P0：  
原因（≤3条）：
1) Gate/豁免/GoLive 是 V2 的“价值核心”，语义偏差会导致体系不可用；  
2) 现有测试与覆盖率很强，具备“在不降质量的情况下重构语义”的工程基础；  
3) P0 修正后，Hook/CI/多端与 Skill 宿主才能获得稳定可预期的自动化行为。

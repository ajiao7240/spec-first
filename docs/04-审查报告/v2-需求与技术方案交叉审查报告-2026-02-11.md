### P1-04: IdType 联合体成员数与注释不一致（5 vs 6）

**涉及文件**:
- 文件: `v2-03-cli命令与核心模块设计.md`
- 行号: L244
- 内容: `type IdType = 'FR' | 'DS' | 'TASK' | 'TC' | 'RFC';  // 6 种（Feature 由 init 生成，不走 nextId）`

**交叉参考**:

## AI-Native 架构专项分析

### 1. Skill 系统一致性评估

**评分: 7.5/10**

**一致性亮点**:
- 16 个 Skill 的命名、职责、阶段归属在 aux-01 和 v2-02 间完全一致
- 6 阶段执行模型（Phase 0-5）在两端定义一致，Phase 职责无歧义
- Skill 与 CLI 的调用关系（Skill → CLI 单向依赖）在 v2-01 L60-65 和 aux-01 均明确
- confirm_policy 三档（strict/assisted/auto）+ 四维判定矩阵在两端完全对齐

**关键不一致**:
- **Skill 文件目录双轨定义**（P1-03）：aux-01 定义 `.claude/commands/spec-first/<cmd>.md`，v2-02 定义 `skills/spec-first/<NN>-<cmd>/SKILL.md`。这是 Skill 系统最严重的不一致，直接影响路由器的文件定位逻辑。
- **golive 路由归属缺失**（P1-02）：golive 在 v2-03 中有完整 CLI 定义，但 v2-01/v2-02 的路由表均遗漏。

**架构合理性判定**: Skill 系统的分层设计（Skill 编排 → CLI 执行）是 AI-Native 工具链的正确范式。Skill 作为"AI 可读的 Markdown 指令"，CLI 作为"确定性原子操作"，职责边界清晰。主要问题集中在文件路径等硬数据层面，架构本身无设计缺陷。

### 2. Context Pack 设计评估

**评分: 7.0/10**

**一致性亮点**:
- 双区结构（control + references）在 aux-01 和 v2-08 间定义一致
- 3 层上下文（L1 核心、L2 阶段、L3 邻居）的分层逻辑两端对齐
- control 区 <2KB 硬限制在两端均有明确声明
- 降级策略（超预算时按 L3 → L2 → L1 顺序裁剪）两端一致

**关键不一致**:
- **Token 预算默认值 2 倍偏差**（P0-02）：aux-01 定义 8K，v2-08 定义 16K。这是 Context Pack 最严重的不一致，直接影响 AI 推理质量。

**架构合理性判定**: Context Pack 的双区 + 三层设计是 LLM 上下文管理的成熟模式。control 区硬限制 <2KB 确保指令不被截断，references 区动态切片确保上下文利用率最大化。L1/L2/L3 的分配比例（20%/30%/50%）合理——核心上下文占比小但不可裁剪，邻居上下文占比大但可降级。修复 P0-02 后，Context Pack 设计无架构层面问题。

### 3. Hook 双层体系评估

**评分: 8.0/10**

**一致性亮点**:

### 架构优势（值得肯定的设计决策）

1. **双层架构（Skill + CLI）职责分离彻底**: Skill 负责 AI 编排（非确定性），CLI 负责原子操作（确定性），单向依赖（Skill → CLI），CLI 不感知 Skill 存在。这是 AI-Native 工具链的正确范式。

2. **6-ID 追踪链设计精简**: 将 NFR 合并到 FR（标签区分）、API 合并到 DS（属性引用）、ADR 合并到 RFC（子类型），从潜在的 9+ 种 ID 精简到 6 种，大幅降低追踪复杂度。

3. **3 层规范合并（Layer 0/1/2）渐进式裁剪**: Layer 0 提供基线，Layer 1 按 Mode x Size 裁剪，Layer 2 按技术端叠加。三层正交组合，避免了"一刀切"或"无限配置"两个极端。

4. **confirm_policy 三档信任度模型**: 从 strict（逐一确认）到 assisted（推荐确认）到 auto（自主执行），配合四维判定矩阵，实现了 AI 自主性与人类控制权的优雅平衡。

5. **12 项度量指标体系完整**: C1-C9 覆盖率/合规率 + E1 效率 + Q1 质量 + H1 综合健康分，公式、权重、惩罚项在需求与技术方案间高度一致，可直接落地实现。

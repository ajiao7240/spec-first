# V2 需求文档 × V2 技术方案一致性与合理性审查报告（三角色并行）

> 审查日期：2026-02-10  
> 审查范围：`docs/01需求文档/v2` 与 `docs/02技术方案/V2` 全量交叉核对  
> 审查角色：研发效能总监 / AI 顶尖架构师 / 研发架构师（并行视角）

---

## 一、审查方法（分步执行）

1. 建立映射基线：按 `core/auxiliary` 与 `v2-01~v2-12` 建立一对一能力映射（流程、命令、数据契约、门禁、度量、上线）。
2. 逐条契约对账：针对高风险契约（ID 模型、ExitCode、Gate、Hook、豁免、Catchup、CLI 命令面）做逐条比对。
3. 反向合理性检查：识别“方案超需求”“需求反向依赖方案”“可执行性弱约束”等治理风险。
4. 三角色并行评审：
   - 研发效能总监：流程治理与上线准入
   - AI 顶尖架构师：Skill/Context/Catchup 与 AI 人在回路
   - 研发架构师：模块边界、命令契约与数据模型闭环
5. 合并裁决：按 P0/P1/P2 输出问题清单与修复优先级。

---

## 二、并行评审结论

- 研发效能总监结论：**有阻断项，暂不建议签核**（需求主权与阶段门禁治理存在结构性问题）。
- AI 顶尖架构师结论：**核心架构可用，但 AI 协作契约出现行为漂移**（Catchup 语义偏离）。
- 研发架构师结论：**主体模块一致，但存在接口面和数据语义未闭环项**（命令面扩张、豁免字段语义不严谨）。

综合结论：**当前不满足“最终签核”标准**。需先完成下述 P0/P1 闭环。

- 证据：`docs/01需求文档/v2/core/core-03-traceability.md:24`

## P1（高优先级）

### P1-1 Catchup 契约漂移：需求“轻量手动 6 步”，方案变为“自动触发 7 步 + 默认 auto”
- 证据（需求）：`docs/01需求文档/v2/auxiliary/aux-01-skill-system.md:455`
- 证据（需求）：`docs/01需求文档/v2/auxiliary/aux-01-skill-system.md:457`
- 证据（需求）：`docs/01需求文档/v2/auxiliary/aux-02-cli-system.md:175`
- 证据（方案）：`docs/02技术方案/V2/v2-08-context-pack与catchup.md:106`
- 证据（方案）：`docs/02技术方案/V2/v2-08-context-pack与catchup.md:108`
- 证据（方案）：`docs/02技术方案/V2/v2-08-context-pack与catchup.md:134`
- 问题：行为语义从“手动轻量恢复”扩展为“自动静默恢复”，且流程复杂度增加。
- 影响：用户可预期性下降；额外 I/O 与自动副作用增加；与 CLI 文案“轻量流程”不一致。
- 修复建议：
  - 二选一统一契约：
    1. 保持需求口径：默认手动（或 prompt），6 步轻量；
    2. 升级需求口径：明确自动触发策略、审计与开关默认值。

### P1-2 CLI 契约面扩张未走需求变更：新增 `spec-first golive check`
- 证据（需求）：`docs/01需求文档/v2/auxiliary/aux-02-cli-system.md:69`
- 证据（方案）：`docs/02技术方案/V2/v2-12-上线路线图与GoLive.md:148`
- 问题：需求明确 12 命令组，方案在路线图新增 `golive` 命令，但未在需求与 `v2-03` 命令规范统一登记。
- 影响：命令面失控；帮助文档、权限、测试基线可能漂移。
- 修复建议：
  - 若保留 `golive`：先补需求（aux-02）与 `v2-03` 命令契约。
  - 若不保留：改为内部脚本/非公开命令，不进入 CLI 对外面。

### P1-3 豁免 `scope_ids` 语义与 6-ID 模型不闭环
- 证据（需求 ID 模型）：`docs/01需求文档/v2/core/core-03-traceability.md:24`
- 证据（需求字段）：`docs/01需求文档/v2/core/core-03-traceability.md:182`
- 证据（方案字段）：`docs/02技术方案/V2/v2-06-质量门禁与豁免机制.md:109`
- 证据（方案 ID 模型）：`docs/02技术方案/V2/v2-05-追踪矩阵与ID引擎.md:16`
- 问题：已声明 NFR/API 无独立 ID，但 `scope_ids` 仍允许 `NFR/API`，缺少可验证的 canonical 引用格式。
- 影响：豁免记录难以自动校验，审计口径不一致。
- 修复建议：
  - 将 `scope_ids` 改为明确可解析的引用协议（例如 `FR-*` + NFR tag 引用、`DS.api_ref` 或 `api:<name>`），并给出正则/示例。

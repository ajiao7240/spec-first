## 3. 跨角色共识问题

以下问题被 **2 个及以上角色** 独立提出，具有高置信度：

### 3.1 测试策略完全缺失

| 属性 | 内容 |
|------|------|
| **提出角色** | Dev、QA |
| **严重程度** | P0 |
| **问题描述** | 8 篇技术方案文档中无任何一篇包含测试策略章节。缺少单元测试、集成测试、E2E 测试的分层策略，缺少测试框架选型，缺少跨平台兼容性测试矩阵 |
| **影响范围** | 全部 7 个子系统（M1-M7） |
| **建议** | 补充独立的测试策略文档或在每个模块详细设计中增加测试章节 |

### 3.2 跨模块接口签名不一致

| 属性 | 内容 |
|------|------|
| **提出角色** | Dev、Arch |
| **严重程度** | P0 |
| **问题描述** | `evaluateGate` 在 M1（流程引擎）中定义为 2 参数签名 `(stageId, context)`，在 M3（质量门禁）中定义为 3 参数签名 `(stageId, gateConditions, context)`。跨文档接口定义缺乏统一约束 |
| **影响范围** | M1 ↔ M3 模块交互 |
| **建议** | 建立跨模块接口契约文档（Interface Contract），所有模块引用同一份接口定义 |

### 3.3 JSONL 并发写入竞态风险

| 属性 | 内容 |
|------|------|
| **提出角色** | Dev、Ops |
| **严重程度** | P0 |
| **问题描述** | `gate-history.jsonl` 和 `ai-stats.jsonl` 采用追加写入模式，但未设计文件锁或原子写入机制。多人并行开发或 CI 并发执行时存在数据交错/损坏风险 |
| **影响范围** | M6 度量引擎、M3 Gate 历史记录 |
| **建议** | 采用 `flock` 文件锁（Linux/macOS）或写入临时文件后原子 rename；写入中断时需有恢复机制 |

### 3.4 Hook 脚本安全性

| 属性 | 内容 |
|------|------|
| **提出角色** | Ops、QA |
| **严重程度** | P0 |
| **问题描述** | `.spec-first/hooks/` 下的脚本随仓库提交，恶意 PR 可修改 Hook 脚本注入任意代码。缺少脚本完整性校验（如 SHA256 校验和） |
| **影响范围** | 全部本地 Hook 执行路径 |
| **建议** | Hook 脚本增加 SHA256 完整性校验；CI 中对 Hook 脚本变更做专项审查 |

### 3.5 config.yaml Schema 版本管理缺失

| 属性 | 内容 |
|------|------|
| **提出角色** | Ops、Dev |
| **严重程度** | P0 |
| **问题描述** | `config.yaml` 无 `schemaVersion` 字段，工具升级后无法检测配置兼容性，也无自动迁移路径 |
| **影响范围** | 全局配置管理、工具升级流程 |
| **建议** | 增加 `schemaVersion` 字段 + 版本迁移脚本框架 |

### 3.6 开发者体验（DX）与学习曲线

| 属性 | 内容 |
|------|------|
| **提出角色** | Dev、DevEff、PM |
| **严重程度** | P1 |
| **问题描述** | 7+3 阶段 × 32 Gate × 8 种 ID 类型，新成员上手成本高。错误信息设计不足，commit 被拒时缺少明确的修复指引 |
| **影响范围** | 全部开发者日常工作流 |
| **建议** | Hook 校验失败时输出结构化错误信息（含修复建议和文档链接）；提供交互式引导模式 |

#### 5.1.2 五大核心问题对齐

| 核心问题 | 技术解法 | 对齐度 |
|---------|---------|--------|
| 需求失真 | 全链路追踪矩阵 + SCA 自动校验 | ✅ 完全对齐 |

| AI 不可控 | 双层 Hook + Context Pack + Agent 路由 | ✅ 完全对齐 |
| 流程形式化 | 自动化校验 + 度量运营闭环 | ✅ 完全对齐 |

#### 5.1.3 PM 关注事项

1. **M6 健康评分边界**：评分模型可能被误用于个人绩效评估，需在文档中明确声明使用边界
2. **commit-msg 错误提示**：阶段感知规则拒绝提交时，错误信息应包含"当前阶段允许的 ID 类型"和"修复建议"
3. **7 种角色体验**：PM/Arch/Dev/QA/TL/Ops/AI Agent 七种角色的交互路径均有覆盖，但 Ops 角色的日常运维体验设计较薄

### 5.2 架构师 (Arch) — 详细摘要（部分）

> 因上下文限制，架构师仅完成 M7 工具链集成设计的审查。以下为已审查内容摘要，其余观点从其他角色交叉引用中综合提取。

#### 5.2.1 M7 接口设计评审

| 接口 | 职责 | 评价 |
|------|------|------|
| `HookManager` | Hook 生命周期管理（安装/卸载/状态查询） | 职责明确，方法签名合理 |
| `ScriptRunner` | 校验脚本执行调度（超时控制/结果收集） | 需补充并发执行时的资源隔离 |
| `ExternalAPIAdapter` | 外部系统集成（Jira/GitHub API） | 适配器模式合理，错误重试策略需细化 |
| `AgentRegistryLoader` | AI Agent 注册表加载与路由 | 与 M5 的 Agent 路由矩阵需保持一致 |

#### 5.2.2 架构层面综合观点（交叉提取）

1. **4 层架构合理**：交互→编排→核心服务→数据的分层符合 CLI 工具链定位，无需微服务
2. **模块依赖方向正确**：M1 作为编排中枢向下调用 M2/M3/M5，无循环依赖
3. **接口一致性不足**：跨文档的同名接口（如 `evaluateGate`）签名不一致，需统一契约
4. **CLI 退出码设计合理**：0/1/2/3 四级退出码（成功/校验失败/配置错误/系统错误）覆盖主要场景

| M4 变更管理 | 800-1,500 行 | 中高 | RFC 状态机过度设计（11→5 状态） |

| M7 工具链 | 900-1,500 行 | 中 | 跨平台 Hook 安装兼容性 |

#### 5.3.2 Dev 关键发现

1. **接口签名不一致**：`evaluateGate` 在 M1 为 `(stageId, context)`，M3 为 `(stageId, gateConditions, context)`
2. **RFC 状态机过度设计**：11 个状态（Draft/Submitted/Triaged/Analyzing/Reviewed/Approved/Rejected/Implementing/Implemented/Verified/Closed）建议简化为 5 个
3. **Worktree 编排质疑**：Git Worktree 并行执行 AI 任务的场景在 10-50 人团队中需求不明确，建议 MVP 阶段采用顺序执行
4. **JSONL 竞态**：`fs.appendFile` 在 Node.js 中非原子操作，并发写入可能导致 JSON 行交错损坏
5. **测试策略空白**：8 篇文档无任何测试相关章节，这是最大的缺失项

### 5.4 运维 (Ops) — 详细摘要

#### 5.4.1 七维度审查结果

| 维度 | 评级 | 关键发现 |
|------|------|---------|
| **部署架构** | ✅ 良好 | "零服务进程"架构合理，本地 Hook + CI 兜底双保险策略设计良好 |
| **环境兼容性** | ⚠️ 有风险 | macOS/Linux 原生支持；Windows 需 copy 模式替代 symlink，或推荐 WSL2 |
| **可观测性** | ❌ 缺失 | 无集中式日志框架（P0）；无告警机制（P0）；无运行状态仪表盘 |
| **故障恢复** | ❌ 缺失 | JSONL 写入中断无恢复机制；Gate 评估超时无状态回滚 |
| **安全性** | ❌ 高风险 | Hook 脚本无完整性校验，恶意 PR 可注入任意代码执行 |

#### 5.4.2 Ops 关键建议

1. **安全加固**：Hook 脚本增加 SHA256 校验和，安装时验证完整性；CI 中对 `.spec-first/hooks/` 变更做专项 diff 审查
2. **日志框架**：统一日志输出到 `.spec-first/runtime/logs/`，支持 DEBUG/INFO/WARN/ERROR 四级，按日轮转
3. **告警规则**：Gate 连续失败 ≥ 3 次、覆盖率下降 > 10%、SCA 校验异常时触发通知（Slack/邮件）
4. **配置版本化**：`config.yaml` 增加 `schemaVersion: "1.0"` 字段，工具升级时自动检测并执行迁移脚本

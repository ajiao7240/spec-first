# Spec-First 快速开始指南

> 让 AI 帮你写规范、管流程、保质量 —— 5 分钟上手，小白友好

---

## 💡 这是什么？（30 秒读懂）

**Spec-First 就像"建房子前先画图纸"**

传统开发：需求口头说 → 直接写代码 → 改来改去 → 文档补不上
Spec-First：AI 帮你写需求 → 自动生成设计 → 代码对照规范 → 全程可追溯

**一句话价值**：让 AI 理解你的需求，自动生成规范化文档，确保需求-设计-代码一致。

---

## 🎯 为什么需要它？

| 痛点 | Spec-First 解决方案 |
|------|-------------------|
| 需求散落在微信/文档/脑子里 | 统一存储在 `specs/` 目录，AI 自动生成 |
| 设计和代码对不上 | 自动追踪矩阵，一眼看出哪里没实现 |
| 改需求不知道影响范围 | 自动分析影响，告诉你要改哪些代码 |
| 质量靠人工审查，容易漏 | 自动 Gate 检查（安全、性能、覆盖率） |
| 新人接手项目一脸懵 | 5 分钟生成项目认知文档 |

---

## 📚 核心概念速查（看不懂也能用）

| 术语 | 通俗解释 | 举例 |
|------|---------|------|
| **Feature** | 一个完整的功能需求 | "用户注册功能" |
| **FR** | 功能需求（用户要什么） | FR-001: 用户可以用手机号注册 |
| **NFR** | 非功能需求（性能/安全要求） | NFR-001: 注册接口响应时间 <500ms |
| **DS** | 技术设计方案（怎么实现） | DS-001: 使用 Redis 缓存验证码 |
| **TASK** | 开发任务（具体要写的代码） | TASK-001: 实现发送验证码接口 |
| **Gate** | 质量检查点（确保没问题才能继续） | 检查测试覆盖率是否 ≥75% |
| **追踪矩阵** | 需求-设计-代码对应表 | FR-001 → DS-001 → TASK-001 |

**提示**：第一次使用不需要记住这些，跟着步骤走就行。

## 📋 场景 1：开发新功能（完整流程）

**👤 适合谁**：开发者从零开始实现一个新功能

**🎯 解决什么**：不知道怎么写需求文档、设计文档，担心遗漏细节

**⏱️ 预计时间**：30-60 分钟（不含编码）

**📦 你会得到**：完整的需求、设计、任务文档 + 代码实现指导

---

### Step 1: 快速认知项目 (5 分钟)

```bash
/spec-first:first
```

**🤔 这步做什么**：让 AI 快速了解你的项目（就像新同事入职第一天）

**📝 AI 会分析**：
- 用的什么技术（React？Vue？Node.js？）
- 项目结构（代码放哪、测试放哪）
- 编码规范（怎么命名变量、怎么写注释）

**✅ 成功标志**：
- 终端显示 "✓ 已生成 5 份文档"
- 出现 `docs/first/` 目录
- 打开 `docs/first/tech-stack.md` 能看到技术栈清单

**❌ 如果出错**：
- 提示"不是 git 仓库" → 先运行 `git init`
- 提示"找不到代码文件" → 确认当前目录是项目根目录

**💡 小贴士**：
- 第一次运行会慢一点（AI 在分析代码）
- 生成的文档可以手动修改
- 加 `--deep` 参数可以生成更详细的文档

---

### Step 2: 初始化 Feature 工作区 (2 分钟)

```bash
/spec-first:init
```

**🤔 这步做什么**：创建一个新功能的工作目录（就像新建一个项目文件夹）

**💬 AI 会问你**：
1. **Feature 标题**？
   - 例子：用户注册功能
   - 提示：用一句话说清楚要做什么

2. **开发模式**？
   - N = 新功能（从零开始）
   - E = 增强现有功能
   - B = 修复 Bug
   - R = 重构代码
   - 👉 第一次选 N 就对了

3. **规模评估**？
   - S = 小型（1-3 天）
   - M = 中型（1-2 周）
   - L = 大型（2 周以上）
   - 👉 不确定就选 M

4. **目标平台**？
   - backend = 后端接口
   - h5 = 移动端网页
   - admin = 管理后台
   - miniapp = 小程序
   - 👉 根据实际情况选

**✅ 成功标志**：
- 看到 "✓ Feature 工作区已创建"
- 出现 `specs/FSREQ-20260309-XXX/` 目录
- 目录里有 `trace-matrix.yaml` 文件

**❌ 如果出错**：
- 提示"已存在进行中的 Feature" → 先运行 `/spec-first:feature-list` 查看

**💡 小贴士**：
- Feature ID 会自动生成（格式：FSREQ-日期-序号）
- 可以同时管理多个 Feature，用 `/spec-first:feature-switch` 切换

---

### Step 3: 编写需求规格 (10 分钟)

```bash
/spec-first:spec
```

**🤔 这步做什么**：让 AI 帮你写需求文档（就像产品经理写 PRD）

**📝 AI 会生成**：
- **PRD**（产品需求文档）：业务目标、用户故事、使用场景
- **spec.md**（需求规格）：详细的功能需求列表
- **追溯 ID**：每个需求自动分配编号（FR-001, FR-002...）

**✅ 成功标志**：
- 看到 "✓ 需求规格已生成"
- `specs/FSREQ-xxx/prd.md` 文件存在
- `specs/FSREQ-xxx/spec.md` 文件存在
- 打开文件能看到需求列表（带 FR-001 这样的编号）

**❌ 如果出错**：
- 提示"阶段不匹配" → 运行 `/spec-first:status` 查看当前阶段

**💡 小贴士**：
- AI 生成的文档是草稿，你可以手动修改补充
- 需求要写清楚"用户能做什么"，不要写"怎么实现"
- 非功能需求（性能、安全）也很重要，别忘了

**📖 真实例子**：
```markdown
## 功能需求

### FR-001: 手机号注册
用户可以使用手机号 + 验证码完成注册

验收标准：
- 支持中国大陆 11 位手机号
- 验证码 6 位数字，5 分钟有效
- 同一手机号 1 分钟内只能发送 1 次验证码
```

---

### Step 4: 技术设计 (8 分钟)

```bash
/spec-first:design
```

**🤔 这步做什么**：让 AI 帮你设计技术方案（就像架构师画设计图）

**📝 AI 会生成**：
- 架构设计（用什么技术、怎么分模块）
- 数据模型（数据库表结构）
- API 接口（接口路径、参数、返回值）
- 设计 ID（DS-001, API-001...）

**✅ 成功标志**：
- 看到 "✓ 技术设计已生成"
- `specs/FSREQ-xxx/design.md` 文件存在
- 文件里有接口定义、数据库表结构

**❌ 如果出错**：
- 提示"需求规格不存在" → 先完成 Step 3

**💡 小贴士**：
- 设计要考虑现有系统，不要重复造轮子
- API 接口要符合 RESTful 规范
- 数据库字段要考虑扩展性

**📖 真实例子**：
```markdown
## API 设计

### API-001: 发送注册验证码
POST /api/v1/auth/register/send-code

请求参数：
- phone: string, 必填, 手机号

响应：
- code: number, 状态码
- message: string, 提示信息
```

---

### Step 5: 任务拆解 (5 分钟)

```bash
/spec-first:task
```

**🤔 这步做什么**：把设计拆成具体的开发任务（就像把大目标分解成小步骤）

**📝 AI 会生成**：
- 开发任务清单（TASK-001, TASK-002...）
- 任务依赖关系（哪个任务要先做）
- 工作量估算（每个任务预计多久）

**✅ 成功标志**：
- 看到 "✓ 任务计划已生成"
- `specs/FSREQ-xxx/task_plan.md` 文件存在
- 文件里有任务列表和依赖关系图

**💡 小贴士**：
- 任务要足够小（1 个任务 ≤ 4 小时）
- 优先做基础任务（其他任务依赖的）
- 可以手动调整任务顺序

---

### Step 6: 代码实现 (按需)

```bash
/spec-first:code
```

**🤔 这步做什么**：AI 帮你写代码（根据设计和任务）

**📝 AI 会做**：
- 自动找到当前要做的任务
- 加载设计文档和代码规范
- 生成符合规范的代码
- 实时检查代码质量

**✅ 成功标志**：
- 代码文件已创建/修改
- 代码能通过编译/运行
- AI 提示 "✓ 代码实现完成"

**💡 小贴士**：
- 一次只实现一个 TASK
- 实现完记得自己测试一下
- 不满意可以让 AI 重新生成

---

### Step 7: 实现审查 (3 分钟)

```bash
/spec-first:review
```

**🤔 这步做什么**：AI 自动审查代码质量（就像 Code Review）

**📝 AI 会检查**：
- 代码是否符合设计
- 是否有安全漏洞（SQL 注入、XSS 等）
- 是否有性能问题（N+1 查询、内存泄漏）
- 代码复杂度是否过高

**✅ 成功标志**：
- 看到审查报告
- `specs/FSREQ-xxx/review-report.md` 文件存在
- 报告里没有严重问题（或已修复）

**💡 小贴士**：
- 有问题不要慌，AI 会给出修复建议
- 可以多次审查，直到通过

---

### Step 8: 验证测试 (5 分钟)

```bash
/spec-first:verify
```

**🤔 这步做什么**：最后的质量检查（确保可以上线）

**📝 AI 会检查**：
- 所有需求都实现了吗？
- 测试覆盖率够吗？（≥75%）
- 安全扫描通过了吗？
- 文档都写全了吗？

**✅ 成功标志**：
- 看到 "✓ 验收通过"
- 所有 Gate 检查都是绿色 ✓

**❌ 如果不通过**：
- 看提示信息，缺什么补什么
- 常见问题：测试覆盖率不够 → 补充测试用例

**💡 小贴士**：
- 这是上线前的最后一关，要认真对待
- 通过后就可以提交代码、发布了

## ❓ 常见问题（FAQ）

### Q1: 我不知道该填什么怎么办？

**A**: 交互式问题时，AI 会给出例子和提示。实在不确定就：
- Feature 标题：用一句话说清楚要做什么
- 模式：第一次选 N（新功能）
- 规模：不确定选 M（中型）
- 平台：根据实际情况选，不确定问技术负责人

### Q2: 命令运行没反应，是卡住了吗？

**A**: 不是卡住，AI 在分析处理（可能需要 10-30 秒）
- 看到 "思考中..." 就是在处理
- 复杂任务（如 first、design）会慢一点
- 如果超过 2 分钟没反应，按 Ctrl+C 取消重试

### Q3: 生成的文档在哪里？

**A**: 所有文档都在 `specs/FSREQ-xxx/` 目录下：
- `prd.md` - 产品需求文档
- `spec.md` - 需求规格
- `design.md` - 技术设计
- `task_plan.md` - 任务计划
- `trace-matrix.yaml` - 追踪矩阵

### Q4: 出错了怎么办？

**A**: 按这个顺序排查：
1. 看错误提示（通常会告诉你怎么做）
2. 运行 `/spec-first:doctor` 检查环境
3. 运行 `/spec-first:status` 查看当前状态
4. 查看 `docs/用户文档/使用手册.md` 详细文档

### Q5: 我能改 AI 生成的文档吗？

**A**: 当然可以！
- 所有文档都是 Markdown 格式，可以直接编辑
- 改完后运行 `/spec-first:sync` 同步追踪矩阵
- AI 下次会基于你修改后的内容继续工作

### Q6: 多人协作怎么办？

**A**: Spec-First 支持多人协作：
- 所有文档都在 `specs/` 目录，提交到 Git 即可
- 每个人可以用 `/spec-first:feature-switch` 切换 Feature
- 建议：一个 Feature 一个人负责，避免冲突

### Q7: 我做错了能重来吗？

**A**: 可以重来：
- 删除 `specs/FSREQ-xxx/` 目录
- 重新运行 `/spec-first:init`
- 或者直接编辑文档修改错误内容

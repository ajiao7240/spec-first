# spec-first 记忆体系分析

spec-first 的“记忆”不是模型参数，而是落在文件系统里的可恢复状态。

它的目标很明确：

- 让 AI 能恢复上一次会话的工作现场
- 让项目规范长期可复用
- 让任务执行可以暂停、继续、追溯
- 让每次 session 都能被记录和回放

---

## 0. 记忆分层总图

```text
.spec-first/spec/
  └─ 项目知识记忆

.spec-first/tasks/<task>/
  ├─ task.json
  ├─ prd.md
  ├─ implement.jsonl
  ├─ check.jsonl
  └─ debug.jsonl
  └─ 任务记忆

.spec-first/.current-task
.spec-first/workspace/<developer>/
  ├─ journal-N.md
  └─ index.md
  └─ 会话记忆

journal / index / git / tasks / spec
  └─ 恢复与回放记忆
```

从代码看，这套体系是分层落盘，而不是依赖模型内部状态。

---

## 1. 四层记忆分别是什么

### 1.1 项目知识记忆

位置：

- `.spec-first/spec/`
- `.spec-first/spec/guides/`

内容：

- 项目目录结构与层级约定
- 编码规范、命名规范、模块边界
- 错误处理、日志、异常约束
- 质量标准、检查清单、review 要点
- 跨层思考指南和复用原则
- 不同 package / 层的专属规范索引
- 反模式、常见坑、禁止事项

职责：

- 为 AI 提供长期、稳定、可复用的项目知识
- 作为 `before-dev`、`check`、`finish-work` 的基础上下文
- 作为后续任务创建和实现时的优先读取源

### 1.2 任务记忆

位置：

- `.spec-first/tasks/<task>/task.json`
- `.spec-first/tasks/<task>/prd.md`
- `.spec-first/tasks/<task>/implement.jsonl`
- `.spec-first/tasks/<task>/check.jsonl`
- `.spec-first/tasks/<task>/debug.jsonl`

内容：

- 任务状态
- 责任人
- 关联 package
- 子任务关系
- 当前分支和 base branch
- 下一步动作序列
- 任务需求和验收说明
- 任务专属上下文注入文件

职责：

- 把“要做什么”保存成结构化状态
- 把每个阶段该看什么文件固定下来
- 支持任务状态流转和归档

### 1.3 会话记忆

位置：

- `.spec-first/.developer`
- `.spec-first/.current-task`
- `.spec-first/workspace/<developer>/`
- `.spec-first/workspace/<developer>/journal-N.md`
- `.spec-first/workspace/<developer>/index.md`

内容：

- 当前开发者身份
- 当前活跃任务路径
- 当前 journal 文件
- session 索引
- 过程摘要
- commit / branch / testing 信息

职责：

- 恢复上一次会话时的执行锚点
- 告诉 AI 当前工作在哪个任务上
- 让 `get_context` 和 `add_session` 能找到最新现场

### 1.4 历史沉淀记忆

位置：

- journal 文件中的历史 session 块
- `index.md` 中的 session history

内容：

- 每次会话的标题、摘要、提交、分支
- 检查结果
- 后续步骤
- 工作区累计历史

职责：

- 提供可追溯历史
- 让“今天干了什么”可以回放
- 为后续记录 session 和审计提供基础

## 2. 这些记忆是怎么流动的

### 2.1 初始化阶段

相关代码：

- `.spec-first/scripts/common/developer.py`
- `.spec-first/scripts/init_developer.py`

会创建：

- `.spec-first/.developer`
- `.spec-first/workspace/<developer>/`
- `.spec-first/workspace/<developer>/journal-1.md`
- `.spec-first/workspace/<developer>/index.md`

这一步建立的是会话记忆的容器，而不是任务本身。

### 2.2 任务创建阶段

相关代码：

- `.spec-first/scripts/common/task_store.py`

会创建：

- `.spec-first/tasks/<task>/task.json`
- `.spec-first/tasks/<task>/prd.md`（由人工或 bootstrap 写入）

`task.json` 是任务真相源，里面记录状态、分支、父子任务关系、package 等元数据。

### 2.3 上下文初始化阶段

相关代码：

- `.spec-first/scripts/common/task_context.py`
- `.spec-first/scripts/task.py`

`task init-context` 会生成三份 JSONL：

- `implement.jsonl`
- `check.jsonl`
- `debug.jsonl`

默认策略是“最小够用”的上下文，而不是把整个 spec 树一次性塞进去。

不同 `dev_type` 的默认注入不同：

- `backend` / `test`：注入后端实现指南
- `frontend`：注入前端实现指南
- `fullstack`：前后端都注入

monorepo 下还会解析 package，再决定该读哪个 `spec/<package>/...`。

### 2.4 当前任务锚定阶段

相关代码：

- `.spec-first/scripts/common/paths.py`
- `.spec-first/scripts/current_task.py`
- `.spec-first/scripts/task.py`（低层兼容）

`current_task.py switch` 会把任务路径写入 `.spec-first/.current-task`。

后续的上下文恢复、会话记录和任务定位，都依赖这个锚点。

### 2.5 只切指针 vs 切指针 + 初始化上下文

`current_task.py switch` 只负责切换当前任务指针，不会自动创建或补全任务上下文文件。`task.py start` 只保留为底层兼容原语。

如果目标任务已经有完整的 `prd.md` / `implement.jsonl` / `check.jsonl` / `debug.jsonl`，只切指针通常就够了。

如果目标任务还是新任务，建议先跑 `task init-context`，再 `task start`。

这也是一次有意的接口收口：

- 用户级切换应优先使用 `current-task`
- `task.py start my-task` 这类短名兼容行为已经取消
- 若有自动化脚本，迁移到 `current_task.py switch <selection>` 或精确路径调用

ASCII 流程图如下：

```text
方案 A: 只切指针

┌──────────────────────────┐
│  task start <task-b>     │
└─────────────┬────────────┘
              │
              ▼
┌──────────────────────────┐
│ 写入 .spec-first/.current-task
└─────────────┬────────────┘
              │
              ▼
┌──────────────────────────┐
│ get_context / session hook│
│ 读取当前任务指针          │
└─────────────┬────────────┘
              │
              ▼
┌──────────────────────────┐
│ 读取 task-b 的现有上下文  │
│（有多少就注入多少）      │
└──────────────────────────┘

方案 B: 切指针 + 初始化上下文

┌──────────────────────────┐
│ task init-context <task-b>│
└─────────────┬────────────┘
              │
              ▼
┌──────────────────────────┐
│ 生成 implement/check/debug│
│ 并同步 task.json          │
└─────────────┬────────────┘
              │
              ▼
┌──────────────────────────┐
│  task start <task-b>     │
└─────────────┬────────────┘
              │
              ▼
┌──────────────────────────┐
│ 写入 .spec-first/.current-task
└─────────────┬────────────┘
              │
              ▼
┌──────────────────────────┐
│ get_context / session hook│
│ 读取当前任务指针          │
└─────────────┬────────────┘
              │
              ▼
┌──────────────────────────┐
│ 读取 task-b 的完整上下文  │
│（注入更完整）            │
└──────────────────────────┘
```

简单说：

- `start` = 切当前任务
- `init-context` = 补任务上下文
- 两者一起用，切换体验最完整

### 2.6 会话记录阶段

相关代码：

- `.spec-first/scripts/add_session.py`

`add_session` 会把本次工作结果写进 journal，并同步 `index.md`。

它会：

- 读取当前 journal
- 计算当前 session 序号
- 追加 session 内容
- 更新 `index.md`
- 在需要时新建下一份 journal
- 按配置自动提交 workspace 变更

### 2.7 上下文读取阶段

相关代码：

- `.spec-first/scripts/common/session_context.py`
- `.spec-first/scripts/common/git_context.py`

`get_context.py` 实际上是 `git_context.py` 的入口，支持三种模式：

- `default`
- `record`
- `packages`

这意味着“记忆读取”不是单点，而是按不同场景读取不同视图。

## 3. 记忆读取顺序

当 AI 重新开始工作时，比较合理的读取顺序是：

1. 当前开发者身份
2. 当前任务
3. 当前 journal / workspace
4. 任务上下文 JSONL
5. 项目规范 `spec`
6. 最近 Git 状态和提交

这样做的原因很直接：

- 先知道“我是谁”
- 再知道“我在做什么”
- 再知道“我上次做到哪”
- 最后再扩展到“这个项目怎么写”

## 4. 代码里的实际行为

### 4.1 `init_developer`

`init_developer` 会创建：

- `.spec-first/.developer`
- `.spec-first/workspace/<name>/`
- `.spec-first/workspace/<name>/journal-1.md`
- `.spec-first/workspace/<name>/index.md`

这意味着 workspace 不是临时目录，而是开发者级别的连续性容器。

### 4.2 `get_context`

默认上下文会采集：

- 当前 Git 分支
- 工作区是否干净
- 最近 5 条提交
- 活跃任务列表
- 当前任务
- 当前 journal 文件
- monorepo 下各 package 的独立 Git 状态

所以它更像“当前工作面板”，不是简单读取单个配置文件。

### 4.3 `task init-context`

默认注入路径是：

- `workflow.md`
- `spec/<package>/backend/index.md`
- `spec/<package>/frontend/index.md`
- `finish-work`
- `check`

这说明实现阶段、检查阶段、调试阶段看的上下文并不相同。

### 4.4 `add_session`

会写入：

- 当前 journal 文件
- `index.md`
- 必要时新的 journal 文件

默认 journal 上限来自 `config.yaml` 的 `max_journal_lines`。

## 5. `spec/` 是怎么被写入的

这一层要和 `tasks/`、`workspace/` 明确区分开：

- `spec/` 不是任务产物
- `spec/` 不是 session 日志
- `spec/` 是项目知识层，存放会长期复用的实现约束

### 5.1 第一次写入：`spec-first init`

第一次生成 `spec/` 的触发点是 CLI 命令：

```bash
spec-first init
```

触发链路如下：

```text
用户执行 spec-first init
        │
        ▼
packages/cli/src/cli/index.ts
        │  注册 init 命令并调用 init(options)
        ▼
packages/cli/src/commands/init.ts
        │  处理参数、模板来源、单仓库/monorepo 模式
        ▼
createWorkflowStructure(...)
        │
        ├─ 创建 workspace/
        ├─ 创建 tasks/
        ├─ 写 config.yaml / worktree.yaml
        ▼
createSpecTemplates(...)
        │
        ├─ 写 .spec-first/spec/guides/
        ├─ 写 .spec-first/spec/backend/ 或 frontend/
        └─ monorepo 时写 .spec-first/spec/<package>/
```

更具体地说，`createSpecTemplates()` 会按项目形态写入模板：

- 单仓库：
  - `spec/backend/`
  - `spec/frontend/`
  - `spec/guides/`
- monorepo：
  - `spec/<package>/backend/`
  - `spec/<package>/frontend/`
  - `spec/guides/`

相关实现见 [workflow.ts](/Users/kuang/xiaobu/spec-first/packages/cli/src/configurators/workflow.ts#L214)。

### 5.2 `init` 过程中是否写 spec，取决于模板来源

`spec-first init` 并不总是写默认模板。

如果用户选择了远程模板，`init()` 会先下载模板内容，再把 `skipSpecTemplates` 打开，避免重复写默认的空模板。见 [init.ts](/Users/kuang/xiaobu/spec-first/packages/cli/src/commands/init.ts#L1083) 和 [workflow.ts](/Users/kuang/xiaobu/spec-first/packages/cli/src/configurators/workflow.ts#L136)。

所以可以把初始化分成两种结果：

```text
spec-first init
├─ 本地默认模板
│  └─ createSpecTemplates() 写入默认 spec/
└─ 远程模板
   ├─ 先下载模板
   └─ 跳过默认 spec 模板写入
```

### 5.3 什么时候是“首次把知识写进 spec/”

严格来说，第一次把项目知识写进 `spec/` 的时机有两类：

1. 初始化时写模板骨架
2. 后续根据真实代码把骨架补成项目专属规范

第一类是 `spec-first init` 自动发生的。  
第二类通常发生在你完成第一次分析、补规范、或把新经验沉淀回文档的时候。

这也是为什么 `spec/` 既包含“生成时的初始模板”，也包含“后续演化后的项目知识”。

### 5.4 后续知识回写：`update-spec`

当你在实现、调试、设计决策后学到了新东西，应该把知识写回 `spec/`。这条路不是 `spec-first update`，而是平台里的 `update-spec` 体系：

- Claude Code: `/spec:update-spec`
- Codex / 其他技能模式: `$update-spec`

其定位是：

- 记录可执行的实现契约
- 补充新的边界和错误行为
- 更新共享 guides 或具体 backend/frontend 文档

相关技能模板明确写了：在实现、修 bug、发现新模式之后，应把这些知识捕捉回 `.spec-first/spec/`。见 [update-spec SKILL](/Users/kuang/xiaobu/spec-first/packages/cli/src/templates/codex/skills/update-spec/SKILL.md#L3)。

对应流程可以画成这样：

```text
代码实现 / 修复 / 设计决策
          │
          ▼
发现新约束或新模式
          │
          ▼
调用 /spec:update-spec 或 $update-spec
          │
          ▼
更新 .spec-first/spec/backend/ | frontend/ | guides/
          │
          ▼
后续 session 读取到新的项目知识
```

### 5.5 哪个命令不会写 `spec/`

`spec-first update` 不负责写 `spec/`。

它的职责是升级和迁移项目自己的 workflow、commands、skills、templates 等管理文件，代码里明确把 `spec/` 标成受保护的用户自定义内容，不会在 update 流程里覆盖。见 [update.ts](/Users/kuang/xiaobu/spec-first/packages/cli/src/commands/update.ts#L78)。

所以这三个动作不要混淆：

- `spec-first init`：首次生成 spec 骨架
- `/spec:update-spec` 或 `$update-spec`：把新知识写回 spec
- `spec-first update`：更新框架本身，不改用户 spec

## 6. 当前实现中的约束

### 6.1 journal 的轮转阈值来自配置

`add_session.py` 读取的是 `max_journal_lines`。

这意味着 journal 上限不是写死的，而是项目级可配置的。

### 6.2 任务状态最终以 `task.json` 为准

任务状态、分支、阶段、子任务关系，最终都应该以 `task.json` 为准，而不是以 journal 文本为准。

### 6.3 workspace 是历史，不是缓存

workspace 承载的是可恢复的会话状态，不是杂物区。

### 6.4 `.current-task` 是锚点，不是列表

它只记录当前任务路径，负责把“现在正在做什么”固定住。

## 7. 当前代码和文档的已知偏差

这一节很重要，因为它直接影响你对“记忆体系”的理解。

### 7.1 `default_package` 配置有问题

当前配置里：

- `packages` 的 key 是 `spec-first`
- `default_package` 却写成了 `@leokuang/spec-first`

但 `resolve_package()` 只接受 package key，不接受 npm 包名。

结果是：

- `task init-context` 在 monorepo 场景下可能无法正确推断默认包
- 没有显式 `--package` 时，包解析会警告并跳过

### 7.2 `nearLimit` 仍然是硬编码

`add_session.py` 用 `max_journal_lines` 做轮转。

但 `session_context.py` 里的 `nearLimit` 和告警仍然使用硬编码的 `1800`，这和配置不是同一来源。

### 7.3 根 `workspace/index.md` 还不是活索引

根目录的 `workspace/index.md` 描述了“全开发者总索引”，但当前代码只会维护每个开发者自己的 `workspace/<name>/index.md`。

也就是说：

- `workspace/index.md` 目前更像模板
- 真实会话历史还没有汇总到全局索引

### 7.4 一些入门文档还保留旧结构

部分用户手册仍然沿用旧的单文件会话日志和个人记忆子目录写法。

但当前代码实际使用的是：

- 分段的 `journal-N.md`
- 没有独立的个人记忆子目录

这类文档和代码不一致，容易把读者带偏。

---

## 8. 结论

spec-first 的记忆系统不是“模型自己记住了什么”，而是：

- 用 `spec` 记长期知识
- 用 `task` 记短期目标
- 用 `workspace` 记会话过程
- 用 `journal` 记历史沉淀

四层一起工作，AI 才能在下次会话里真正恢复现场。

当前代码已经把这套机制做成了可运行链路，但还需要继续补齐两类一致性工作：

- 配置项和包名解析要对齐
- 文档示意图要和真实目录结构对齐

# spec-first 记忆体系分析

spec-first 的“记忆”不是模型内部参数，而是外置在文件系统里的可恢复状态。

它解决的是四件事：

- 让 AI 能恢复上一次会话的工作状态
- 让项目规范不会丢失
- 让任务执行可以暂停、继续、追溯
- 让每次 session 都有历史记录

---

## 0. 记忆分层总图

```text
.spec-first/spec/
  └─ 项目知识记忆

.spec-first/tasks/<task>/
  ├─ task.json
  ├─ prd.md
  ├─ implement.jsonl
  ├─ check.jsonl
  └─ debug.jsonl
  └─ 任务记忆

.spec-first/.current-task
.spec-first/workspace/<developer>/
  ├─ journal-N.md
  └─ index.md
  └─ 会话记忆

journal / index / git / tasks / spec
  └─ 恢复与回放记忆
```

---

## 1. 四层记忆分别是什么

### 1.1 项目知识记忆

位置：

- `.spec-first/spec/`
- `.spec-first/spec/guides/`

内容：

- 项目目录结构与层级约定
- 编码规范、命名规范、模块边界
- 错误处理、日志、异常约束
- 质量标准、检查清单、review 要点
- 跨层思考指南和复用原则
- 不同 package / 层的专属规范索引
- 反模式、常见坑、禁止事项

职责：

- 为 AI 提供长期、稳定、可复用的项目知识
- 作为 `before-dev`、`check`、`finish-work` 的基础上下文
- 作为后续任务创建和实现时的优先读取源

### 1.2 任务记忆

位置：

- `.spec-first/tasks/<task>/task.json`
- `.spec-first/tasks/<task>/prd.md`
- `.spec-first/tasks/<task>/implement.jsonl`
- `.spec-first/tasks/<task>/check.jsonl`
- `.spec-first/tasks/<task>/debug.jsonl`

内容：

- 任务状态
- 责任人
- 关联 package
- 子任务关系
- 当前分支和 base branch
- 下一步动作序列
- 任务需求和验收说明
- 任务专属上下文注入文件

职责：

- 把“要做什么”保存成结构化状态
- 把每个阶段该看什么文件固定下来
- 支持任务状态流转和归档

### 1.3 会话记忆

位置：

- `.spec-first/.current-task`
- `.spec-first/workspace/<developer>/`
- `.spec-first/workspace/<developer>/journal-N.md`
- `.spec-first/workspace/<developer>/index.md`

内容：

- 当前活跃任务路径
- 当前 journal 文件
- session 索引
- 过程摘要
- commit / branch / testing 信息

职责：

- 恢复上一次会话时的执行锚点
- 告诉 AI 当前工作在哪个任务上
- 让 `get_context` 和 `record` 能找到最新现场

### 1.4 历史沉淀记忆

位置：

- journal 文件中的历史 session 块
- `index.md` 中的 session history

内容：

- 每次会话的标题、摘要、提交、分支
- 检查结果
- 后续步骤
- 工作区累计历史

职责：

- 提供可追溯历史
- 让“今天干了什么”可以回放
- 为后续记录 session 和审计提供基础

## 2. 这些记忆是怎么流动的

### 2.1 初始化阶段

相关代码：

- `.spec-first/scripts/common/developer.py`

会创建：

- `.developer`
- `workspace`
- `journal-1.md`
- `index.md`

这一步建立的是会话记忆的容器。

### 2.2 Bootstrap 阶段

相关代码：

- `.spec-first/scripts/create_bootstrap.py`

会创建：

- `task.json`
- `prd.md`
- `.current-task`

这一步把“项目规范”变成第一条任务记忆。

### 2.3 任务执行阶段

相关代码：

- `task create`
- `task init-context`
- `task start`
- `.spec-first/scripts/common/session_context.py`

会生成：

- `implement.jsonl`
- `check.jsonl`
- `debug.jsonl`
- 当前任务上下文

这一步把长期规范注入到短期任务中。

### 2.4 会话记录阶段

相关代码：

- `.spec-first/scripts/add_session.py`

会写入：

- journal
- index
- 必要时新的 journal 文件

这一步把短期任务执行结果沉淀成历史记录。

## 3. 记忆读取顺序

当 AI 重新开始工作时，比较合理的读取顺序是：

1. 当前开发者身份
2. 当前任务
3. 当前 journal / workspace
4. 任务上下文 JSONL
5. 项目规范 `spec`
6. 最近 Git 状态和提交

这样做的原因很简单：

- 先知道“我是谁”
- 再知道“我在做什么”
- 再知道“我上次做到哪”
- 最后再扩展到“这个项目怎么写”

---

## 4. 实际约束

### 4.1 `spec` 不能当缓存

`spec` 属于长期知识层，`update` 不应该把它当成临时生成物覆盖。

### 4.2 journal 有长度上限

单份 journal 默认 2000 行上限，接近 1800 行时就要考虑拆分。

### 4.3 task.json 是任务真相源

任务状态、分支、阶段、子任务关系，最终都应该以 `task.json` 为准。

### 4.4 workspace 不是临时目录

workspace 承载的是可恢复的会话状态，不是杂物区。

---

## 5. 结论

spec-first 的记忆系统不是“模型自己记住了什么”，而是：

- 用 `spec` 记长期知识
- 用 `task` 记短期目标
- 用 `workspace` 记会话过程
- 用 `journal` 记历史沉淀

四层一起工作，AI 才能在下次会话里真正恢复现场。

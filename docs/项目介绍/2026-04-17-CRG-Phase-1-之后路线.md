# CRG Phase 1 之后的路线

> 用途：Phase 1a + 1b（`2026-04-17-CRG-优化工作清单.md` 11 项）完成后的后续安排。
> 核心原则：**让数据和消费者反馈驱动，不提前排期，不脑补需求**。
> 关联：短清单见 `2026-04-17-CRG-优化工作清单.md §4 Future Work`；完整调研素材见 `2026-04-17-CRG-深度优化审查报告-full-trace.md`。

---

## 0. 为什么需要这份文档

当工作清单 ship 到 CHANGELOG 后，最容易犯的错是**立刻展开 Future Work 20+ 条**——那就会回到之前的过度设计循环。这份文档的作用是把"Phase 1 之后"这个时间点的行动规则写死，保证后续改造是**证据驱动**的，而不是**计划驱动**的。

三条铁律：

1. **不脑补需求**：当前场景没出现的问题，不提前修；
2. **不提前排期**：下一步做什么，等 Phase 1 ship 后的实测数据和消费反馈告诉你；
3. **每轮最多 1-2 条**：拒绝"一次批准 5 条做完"的诱惑。

---

## 1. 阶段 A：观察期（1-2 周，不做新开发）

Phase 1a + 1b ship 完成后，**不立刻开下一个 Finding**。这段时间做三件事：

### 1.1 跑 benchmarks 填实测数字（历史背景）

> 说明：下面这组 benchmark 命令属于当时 Phase 1 之后的历史运行方式；`test:crg:benchmarks` 与 `test:crg:gate` 已在当前实现中退役，保留这里只是为了说明当时的观察方法。

```bash
npm run test:crg:benchmarks     # review + repo-qa + context-efficiency 三套
npm run test:crg:gate           # regression 基线
```

当时的做法是把实测差值补进 `CHANGELOG.md` 对应的 Phase 1 条目后面，形成 **"改造前 / 改造后" 对照基线**。

### 1.2 观察真实消费

`/spec:code-review`、`/spec:plan`、`/spec:work` 在日常 PR 上跑，**记录具体痛点**：

- 哪些 query 在 `crg search` / retrieval 里召不回？
- `/spec:code-review` 的 `review_guidance` 还有哪些假阳性？
- `crg impact --symbol=X` 的 `blast_radius` 哪里明显错？
- 哪些 `callers_of` 输出缺行号信息导致 AI 定位慢？

**不要凭直觉列问题**；用真实 PR 场景记录。记不到就是没问题。

### 1.3 收集"还差什么"的证据

把 1.1 和 1.2 的产出整理为下一步候选证据。格式：

```
场景：<具体 PR / 查询 / 消费点>
痛点：<行为不符合预期的地方>
证据：<日志 / benchmark 差值 / 用户反馈>
候选修复：<对应 Future Work 编号，可空>
```

阶段 A 的产出是**"我们实际需要什么" 的证据清单**，不是代码。

---

## 2. 阶段 B：基于证据的小迭代（每轮 1-2 项，每轮 2-3 天）

阶段 A 收集到证据后，从 Future Work 里挑。**挑选规则**（三条全满足才做）：

- ✅ 阶段 A 或之前有具体消费者场景证明需要；
- ✅ 有可测量的验证指标（benchmark 数字 / 断言 / 用例）；
- ✅ 工作量 ≤ 3 人日。

### 2.1 候选池（按"证据门槛低 → 高"排序）

> 这不是路线图，是**可供挑选的菜单**。证据到了就做哪条，没到就不做。

| 候选 | 触发条件 | 工作量 | 预期收益 |
|---|---|---|---|
| **Node builtins 集合建模** | Phase 1 完成后 unresolved 仍是 spec-code-review 投诉热点 | 1 人日 | 本仓库直接解 128 条 unresolved（~5%） |
| **S6.1 path_shape 形态分类** | 想把 unresolved 分布作为 benchmark 回归指标 | 1 人日 | 为量化基线提供形态级审计（仍不可做语义断言） |
| **Q5 generations/ 回收** | 磁盘占用实际报警 | 0.5 人日 | 防磁盘失控 |
| **P5 FTS5 增量重建** | Q4 pragma 后 build 时间仍是 bottleneck（有 benchmark 证据） | 2 人日 | 增量 build 显著加速 |
| **D5 call-site 行号** | AI patch 场景明确需要精准定位（有具体用例） | 3 人日 | `callers_of` 输出含 `call_sites` 数组 |
| **S2 unresolved_edges 唯一约束** | 观察到实际重复行 | 0.5 人日 | 数据一致性 |
| **S6.2 resolveEdges failure_reason** | S6.1 落地后，想升级为 CI 真审计门禁 | 3 人日 | unresolved 分布可被 CI 强制断言 |
| **C3a JS/TS suffix 扩展** | 顺手写，基本无证据要求 | 0.5 人日 | 本仓库 `relative_path` bucket 从 10 降到 ≤5 |

### 2.2 每做一条都要过的检查

开工前填一张纸（可直接写 commit body）：

```
候选：<编号 + 标题>
触发场景：<阶段 A 记录的具体 PR / 日志 / 用户反馈>
验收指标：<benchmark 数字 / test 断言 / 用例>
预计工作量：<人日>
不做什么：<明确划线，避免 scope creep>
```

没证据就 **推迟** 而不是 "先做了说不定用得上"。这是防止过度设计的主要门禁。

### 2.3 每轮节奏

- 1-2 条小改动 → benchmark 验证 → CHANGELOG 记录 → ship → 回到阶段 A 观察
- 不要同时开 3 条以上；
- 如果某轮找不到满足判断标准的候选，**直接跳过**这一轮，继续观察。

## 3. 阶段 C：决策节点（阶段 B 跑完 3-5 轮后）

阶段 B 做了 3-5 轮（累积 4-6 周）之后做一次正式评估，回答一个问题：

> **CRG 作为 AI 代码理解底座，现在主要瓶颈在哪？**

根据答案决定是否启动 v2 工程：

| 主要瓶颈 | 启动项 | 工期 |
|---|---|---|
| 检索召回上限（lexical 召不回的场景多） | **D2 dense retrieval**（embedding + ANN + CI cache） | 2-3 周 |
| 社区/依赖信号精度（similar_to 不准 + surprising 噪音） | **Louvain 真社区检测 + 外部模块建模** | 1-2 周 |
| AI patch 精度（agent 需要精确 call-site） | **D5 call-site + S5 稳定 id + FactItem 增强** | 1-2 周 |
| 多语言支持（CRG 被 Python/Ruby 项目接入） | **C3b resolvers 分层** | 1-2 周 |
| **都不是瓶颈** | **停止 CRG 大改造**，资源投到别处 | — |

**"都不是瓶颈" 是一个合法的答案**。如果阶段 A + B 已经让 CRG 满足消费者需求，v2 完全不必做。

---

## 4. 阶段 D：v2 架构（真要做才启动）

阶段 C 识别出瓶颈后，**不直接塞进 Future Work 清单**。启动方式：

1. 走完整 `/spec:ideate → /spec:brainstorm → /spec:plan` 流程；
2. 产出独立 `docs/plans/plan-XXX-crg-v2-<focus>.md`；
3. `full-trace` 报告里对应章节（Phase 4 / D1-D6 / S6.2 / C3b）可作为立项素材输入；
4. v2 立项评审通过后再开工；
5. v2 执行期间，日常工作清单保持冻结，不加新 Finding。

---

## 5. 明确不做的事（现在就划线）

避免再陷入 speculative 工程。以下条目 **不排期、不写 Finding、不进清单**，除非走阶段 D 的完整 spec 流程：

- **parser.js 按语言拆分（D6）**——只在单语言频繁修改时才值得；
- **权重配置化 weights/defaults.json（D3）**——没有 A/B 数据驱动前纯属 YAGNI；
- **AST-aware chunking（D4）**——当前硬切边界没成为实际痛点；
- **P1-P8 的 P2/P3/P4/P6/P8** 剩余性能优化——本仓库规模触发不了收益；
- **C4 basename fallback 多语言泛化**——本仓库是 JS/TS，无收益；
- **S3 expand 分块**——`seed_limit` 还没扩到触发上限；
- **S4 FTS5 转义加固**——没有真实注入场景报告；
- **S5 nodes.id 稳定化**——AI agent 跨 build 引用没成痛点。

这些条目**保留在 full-trace 档案里就够了**，日常不引用、不规划、不讨论。

## 7. 为什么这样安排

三个反面教训来自 `full-trace` 档案：

1. **三轮元审查套娃**：不是从"用户反馈"驱动的修订，是从"报告自己"驱动，越修越厚；
2. **35 条 Finding 里过半是 speculative**：没有消费者场景就把 "将来可能需要" 写成 Finding；
3. **七段式 / 量化基线 / 证据层级** 等 meta-governance 占走大量篇幅，实际能执行的 Finding 反而被稀释。

本路线的核心反制：**先让代码 ship 和用户反馈跑，再决定下一步**。这是从"计划先行"切回"观察先行"的工作方式。

---

## 8. 一句话总结

> **Phase 1 ship → 观察 1-2 周 → 有证据就做 1-2 条小迭代 → 3-5 轮后评估是否需要 v2 → 绝大多数情况 v2 根本不必做。**

"接下来做哪些内容"不是一份可以提前列出来的清单；它是一串由实测数据和消费者反馈触发的判断。

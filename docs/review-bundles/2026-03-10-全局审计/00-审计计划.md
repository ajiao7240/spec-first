# Spec-First ID 链路深度审计计划

## 审计目标

验证从 ID 生成到持久化、恢复的全链路中，ID 在以下维度的可靠性：

1. **一致性** - 同一 ID 在不同模块中的格式、类型、语义是否一致
2. **完整性** - ID 引用链是否完整，无断链、漏链
3. **健壮性** - 异常情况下 ID 是否能正确处理（重试、并发、失败恢复）
4. **幂等性** - 重复执行是否产生重复 ID 或状态错乱
5. **可恢复性** - 中断后能否从正确的 ID 状态恢复

---

## 审计范围

### 核心链路
```
ID 生成 → 传播 → 校验 → 持久化 → 恢复
   ↓        ↓       ↓        ↓        ↓
 init    skill   gate    matrix   catchup
```

### 关键模块
- `trace-engine/` - ID 生成、校验、矩阵管理
- `process-engine/` - 阶段流转、Feature 初始化
- `skill-runtime/` - Skill 分发、上下文传递
- `gate-engine/` - 门禁校验
- `ai-orchestrator/` - 自动循环、上下文恢复

## 审计方法

### 阶段 1: 建立 ID 生命周期地图 (1h)
- 追踪所有 ID 类型的生成点
- 绘制 ID 在模块间的传播路径
- 标注持久化点和恢复点

### 阶段 2: 审计主流程 (2h)
- Feature 初始化 → FR 生成 → DS 生成 → TASK 生成 → TC 生成
- 验证每个环节的 ID 一致性
- 检查矩阵更新时机

### 阶段 3: 审计异常路径 (2h)
- 重试场景：nextId 重复调用
- 并发场景：多进程同时生成 ID
- 失败恢复：中断后重新执行
- 循环检测：orchestrate / dispatcher 死循环风险

### 阶段 4: 审计边界场景 (1h)
- 空矩阵、单 ID、大量 ID
- 特殊字符、边界长度
- 跨阶段 ID 引用

### 阶段 5: 输出审计报告 (1h)
- 问题清单 + 证据
- 根因分析
- 影响范围评估
- 风险等级评定
- 修复建议

## 审计检查点

### 一致性检查点
- [ ] ID 格式定义在所有模块中一致
- [ ] ID 类型枚举完整且无冲突
- [ ] 同一 ID 在不同文件中的解析结果一致

### 完整性检查点
- [ ] 矩阵中所有 upstream 引用都存在
- [ ] 矩阵中所有 downstream 引用都存在
- [ ] FR → DS → TASK → TC 链路完整

### 健壮性检查点
- [ ] nextId 在并发时使用锁机制
- [ ] ID 生成失败时有明确错误信息
- [ ] 循环依赖能被检测并阻断

### 幂等性检查点
- [ ] 重复调用 init 不产生新 Feature ID
- [ ] 重复调用 nextId 不产生重复 ID
- [ ] 矩阵更新操作幂等

### 可恢复性检查点
- [ ] catchup 能正确恢复 Feature 上下文
- [ ] 中断后重新执行不破坏 ID 链路
- [ ] 状态文件损坏时有降级方案

---

## 重点排查项

### 死循环风险点
1. `orchestrate` 自动推进逻辑
2. `dispatcher` 重试机制
3. `advance` 阶段流转条件
4. `getAncestors` 循环依赖处理

### ID 漂移风险点
1. 并发写入 `.feat-registry.md`
2. 矩阵文件并发修改
3. `current` 文件被外部修改
4. 序号计算错误

### 状态错乱风险点
1. 阶段推进但矩阵未更新
2. ID 生成但未写入矩阵
3. 矩阵更新但状态文件未同步

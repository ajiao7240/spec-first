# ID 链路优化方案（方案A / 最小治理版）

**目标**：在不把流程收得过紧的前提下，补齐当前代码库里最容易导致 ID 漂移、断链、重复写入、恢复失真的关键薄弱点。

**设计原则**：
- 以代码实现为准，不以 README 或历史承诺为准。
- 只固化少数硬约束，其余保持告警或后续治理，避免流程脆弱化。
- 优先修“规则源头”，不在各调用点继续打补丁。
- 优先保证恢复与幂等，其次才是更强的治理闭环。

---

## 1. 方案A定位

本方案不是“全量治理版”，而是**最小强治理版**：只收紧 4 条必须稳定的基础契约。

### 4 条硬约束

1. **一个状态文件只对应一种 schema**
   - 避免 `todo-state.json` / batch checkpoint 混写，导致恢复读错状态。

2. **`task_plan.md` 只允许一个核心 parser 解释**
   - 避免 `commit/catchup/prompt/hard-gate/batch-test` 对同一 TASK 链路各读各的。

3. **Feature 身份只允许一个核心 resolver 解释**
   - 避免目录名、`stage-state.featureId`、CLI 参数各自推断，造成错链与误命中。

4. **Artifact ID 分配必须走一条更安全的 allocator 路径**
   - 至少保证串行分配、重复 append 防护、重复 ID 可见。

### 保持软约束的部分

以下内容在方案A中**不强制上硬门**，只做轻量集成或保留为后续项：
- request key / ledger 全量分配账本
- 全局锁化所有 stage advance / current pointer / audit append
- 所有 shell/hook 脚本立即统一 parser
- 把所有 unknown failure 都自动重试
- 用 gate/doctor 对历史脏数据一刀切阻断

原因很简单：这些动作虽然“更完整”，但会明显提高流程刚性，增加恢复失败、误阻断和维护成本。

---

## 2. 方案A已落地项

### A1. 状态文件 schema 拆分

**目标**：修复 `todo-state.json` 被不同运行时复用导致的断链与恢复误读。

**已落地**：
- `src/core/ai-orchestrator/todo-runner.ts`
  - 增加 `kind: 'todo-runner-state'`
  - 增加 `version: 1`
  - `loadTodoState()` 增加 schema guard
  - 保留对旧 `todo-state.json` 的有限兼容，避免一次性收得太死
- `src/core/batch-executor/checkpoint.ts`
  - batch checkpoint 独立为 `batch-checkpoint.json`
  - 增加 `kind: 'batch-checkpoint-state'`
  - 增加 `version: 1`

**解决的问题**：
- batch executor 与 auto-loop 不再争用同一个状态文件
- 恢复时不会把 checkpoint 当成 todo-runner state 读取
- 状态损坏时能尽早暴露，而不是继续静默漂移

### A2. `task_plan.md` 统一解析

**目标**：修复 TASK 当前项、状态、依赖在不同模块中解析不一致。

**已落地**：
- `src/core/task-plan/parser.ts`
  - 提供统一的 `parseTaskPlanContent()` / `readTaskPlan()` / `getCurrentTaskId()` / `toTaskNodes()`
- 以下调用方已经切到统一 parser：
  - `src/cli/commands/batch-test.ts`
  - `src/cli/commands/commit.ts`
  - `src/core/ai-orchestrator/catchup.ts`
  - `src/core/skill-runtime/prompt-assembler.ts`
  - `src/core/skill-runtime/hard-gate.ts`

**解决的问题**：
- 同一份 `task_plan.md` 不再出现“一个模块认为有 current task，另一个模块认为没有”的分叉
- `TASK` 状态归一化有了统一入口，不再由调用方各写一套文本扫描

### A3. Feature 解析统一

**目标**：修复 Feature 身份识别源头不一致导致的错链、漏链、误命中。

**已落地**：
- `src/core/process-engine/feature.ts`
  - 新增 `readValidatedFeatureState()`
  - `getFeatureState()` 校验“目录名 === stage-state.featureId”
  - `listFeatures()` 跳过损坏目录，避免整批命令被脏数据拖死
  - `resolveFeatureId()` 收敛为统一入口
- `src/cli/commands/id.ts`
  - `next/search/list` 统一通过 `resolveFeatureId()` 处理 feature 参数

**解决的问题**：
- CLI 与底层 feature 识别规则收敛
- 目录名错了但 state 没改、或 state 改了目录没改，这类半损坏状态更容易暴露
- 同时保留了适度 fallback，不会把简单查询路径收得太脆

### A4. ID allocator 最小加固

**目标**：修复 `nextId()` 在并发/重试/重复执行下的重复分配与重复写入风险。

**已落地**：
- `src/shared/file-lock.ts`
  - 新增 feature-local 文件锁能力
- `src/core/trace-engine/id-generator.ts`
  - `nextId()` 在矩阵分配路径外包 `withFileLock()`
  - `appendToMatrix()` 在写入前检查同 ID 是否已存在，避免重复 append
- `src/core/validators/format-validator.ts`
  - 增加 matrix 中重复 ID 检测

**解决的问题**：
- 同一 feature 下的 ID 分配不再是裸读-算-写
- 重试或重复执行时，不会把同一个已分配 ID 连续 append 多次
- 重复 ID 不再只靠链路人工排查，格式校验可直接暴露

**本轮刻意不做**：
- 不引入全量 `.id-ledger.json`
- 不引入 `--request-key` 幂等账本

原因：这两项收益确实存在，但会显著提高契约复杂度，和方案A“先稳住、不过度刚性”的目标相冲突。

### A5. retry 接入 auto-loop 主路径

**目标**：修复 `retry-controller` 已存在但未进入 auto-loop 主流程的问题。

**已落地**：
- `src/core/ai-orchestrator/auto-loop.ts`
  - executor 失败后引入 `makeRetryDecision()` / `applyRetryToState()`
  - 仅对 **temporary** 错误执行自动重试调度
  - unknown / permanent 仍维持原有 blocked 语义，避免误重试扩大状态扰动
  - 追加 `task_retry_scheduled` 审计事件

**解决的问题**：
- `timeout` / 临时失败不再直接坠入 blocked
- retry 计数、失败原因、累计 backoff 进入持久状态，可被 catchup / 恢复路径读取
- 保留默认 `stop_on_blocked` 语义，不会把所有失败都改成自动重试，降低流程脆弱性

## 3. 方案A为何不更重

用户担心的问题是成立的：**规则一旦过紧、门过多、恢复链路过硬，系统就会从“不一致”变成“容易崩”**。

所以本方案有意避免以下做法：
- 不把所有失败一律自动重试
- 不要求所有历史状态立刻补齐 schema tag 才能运行
- 不把所有 feature 脏数据直接升级为全局阻断
- 不把 matrix 立刻替换成 ledger-only 单真源架构
- 不把所有 shell / hook / 文档解析一步重构到底

换句话说，方案A追求的是：
- **关键链路更稳**
- **旧数据仍可恢复**
- **新增规则足够少**
- **遇错更可见，但不轻易把流程锁死**

---

## 4. 剩余问题与后续项

以下事项仍建议保留在后续治理中，但**不属于方案A的强制开发范围**：

### 4.1 可选增强项
- 为 ID allocator 增加 request key / ledger，进一步提升真正幂等分配能力
- 为 `advance/current pointer/audit append` 增加统一锁，补齐更多并发写保护
- 将 shell/hook 路径也切到统一 `task_plan` parser
- 为 matrix / state / current pointer 增加更强的 doctor 检查

### 4.2 触发条件
只有当以下问题在真实使用中持续出现时，再建议进入更重治理：
- 多 agent / 并发调用 `id next` 仍频繁冲突
- shell hook 与 TS runtime 的 TASK 解析仍出现分叉
- current pointer / stage advance 在恢复路径中频繁错位
- audit / retry / watchdog 出现明显竞态写损坏

---

## 5. 验收口径

方案A验收不看“规则是否最完整”，而看下面 5 件事是否成立：

1. batch checkpoint 不再污染 todo-runner state
2. `task_plan.md` 在核心 TS 消费者里只存在一种解释方式
3. Feature 解析入口统一，目录名与 state 不一致时不再静默错链
4. `nextId()` 具备最小串行分配保障，重复 append 可避免，重复 ID 可见
5. auto-loop 对临时错误具备可恢复重试路径，但默认失败语义不过度改变

如果这 5 件事成立，就说明当前代码库的 ID 生命周期已经从“脆弱分叉”收敛到了“最小可控”。

# first 运行时后续优化问题与最佳方案（2026-03-09）

## 文档目的

本文聚焦当前 `spec-first first` 最小实现落地后的后续优化事项，只回答两个问题：

1. **现在还存在哪些问题**
2. **每个问题的最佳解决方案是什么**

约束原则：

- 坚持 **KISS**
- 坚持 **职责分离**
- 避免继续扩 scope
- 优先做**收口型优化**，不做“大而全重构”

---

## 当前判断

当前 `first` 的主方向已经明确：

- `.spec-first/runtime/first/` 是唯一真源
- `docs/first/` 是从真源投影出来的人类可读视图
- `spec-first first` 已具备最小可执行入口

所以，下一阶段的重点**不是继续扩分析能力**，而是把“真源 -> 执行 -> 投影恢复 -> 校验”这条链条做得更稳、更薄、更一致。

---

## 待优化问题与最佳方案

| ID | 待优化问题 | 问题说明 | 最佳解决方案 |
|---|---|---|---|
| F1 | `first` 命令层职责偏厚 | 当前 `src/cli/commands/first.ts` 同时负责参数解析、runtime 健康判断、首次 bootstrap、refresh、docs 恢复，命令层已有业务编排倾向 | 将“runtime 健康但 docs 缺失时自动补投影”的逻辑下沉到 `src/core/skill-runtime/first-context.ts` 或同层 runtime service；CLI 命令层只保留“解析参数 -> 调用单一入口 -> 输出结果” |
| F2 | `first` 的确认策略收口不够 | `src/cli/index.ts` 对 `first` 的确认判断仍是专门内联，注册层知道了过多命令细节 | 抽一个极小 helper，例如 `shouldConfirmFirst(args)`，统一放到 `src/core/skill-runtime/first-args.ts` 或 `src/cli/commands/first.ts`，`src/cli/index.ts` 只做引用 |
| F3 | `refresh-all` 对 docs 缺失场景感知不足 | 当前 runtime 健康但 `docs/first/` 缺失时，需要命令层额外再调一次 `refresh-docs-from-runtime`，说明核心 refresh 闭环还没完全收在 runtime 层 | 在 `refreshFirstArtifacts(...)` 内部补充规则：当 runtime 真源健康但 docs 投影缺失或索引缺失时，自动补投影，避免命令层兜底 |
| F4 | `--skip` 语义与“真源/投影模型”还不完全一致 | 当前 `--skip` 仅复用已有 runtime 并输出摘要，不主动恢复 `docs/first/`，对使用者来说不够符合“有真源就应可恢复投影”的直觉 | 将 `--skip` 明确定义为：“跳过 runtime 重建，但允许从健康 runtime 恢复 docs 投影”；这样不做重建，也不破坏真源模型 |
| F5 | `--deep` 目前仍只是 mode 标记 | 当前首次 bootstrap 时写入 `mode=deep`，但内容仍是最小启发式摘要；如果用户把它理解成完整 deep 分析，会产生预期落差 | 先定义“最小 deep 语义”：只增强 `summary.techStack`、`modules`、`evidence`、`entryPoints` 的采样密度，不立刻恢复历史上 10-11 份 deep 文档 |
| F6 | bootstrap 启发式边界需要尽快冻结 | 现在 `first-bootstrap` 已具备轻量分析能力，如果后续持续堆规则，很容易演变成第二套分析引擎，破坏 KISS | 明确边界：bootstrap 仅允许“文件存在性 + `package.json`/少量配置读取 + 平台探测器”，禁止引入 AST 深扫、多 agent 编排、大量推断式规则 |
| F7 | 缺少 router 级确认策略测试 | 当前已有命令级测试，但还缺“默认需确认 / `--quick` 跳确认 / `--force` 跳确认”的路由层保护，后续 CLI 行为可能回归 | 补 2-3 条 router 级单测，只验证确认策略，不扩展到业务逻辑 |
| F8 | 闭环证据输出仍不够稳定 | 现在 index 已记录 docsProjection 哈希和健康度，但一次 `first` 执行究竟恢复了哪些 runtime / docs，用户感知仍偏弱 | 保持最小方案，不新增存储层；仅把命令输出统一成稳定摘要，例如“生成/刷新了哪些 runtime 资产、哪些 docs 投影” |
| F9 | 用户侧最小语义说明仍不足 | 代码已经有最小 `first` 命令，但文档层还没清晰同步“CLI 可执行、runtime 为真源、docs 为投影、`--check-health/--skip` 语义” | 在 `skills/spec-first/00-first/SKILL.md` 或 README 中补一段最小语义说明，避免使用者继续按旧 mental model 理解 |
| F10 | 优先级仍需持续收口 | 当前最有价值的是把已有闭环做稳，而不是继续做 full analyzer；如果不控 scope，复杂度会再次上升 | 后续只优先推进收口项：`F1/F2/F3/F4/F7`；其余项放在第二优先级，且必须坚持最小化实现 |

---

## 推荐执行优先级

### P0：建议优先处理

1. **F1 命令层瘦身**
2. **F2 确认策略 helper 收口**
3. **F3 refresh 核心闭环内建化**
4. **F4 `--skip` 语义对齐真源模型**
5. **F7 router 级确认测试补齐**

这 5 项都是**收口型优化**，特点是：

- 不改变系统边界
- 不引入额外抽象层
- 不增加新的运行时复杂度
- 直接增强“规范定义 -> 运行时执行 -> 证据记录 -> 放行验收”的一致性

### P1：确认后再处理

6. **F5 最小 deep 语义澄清**
7. **F6 bootstrap 边界冻结**
8. **F8 闭环证据输出稳定化**
9. **F9 文档语义同步**

这些项是增强型优化，应该在 P0 稳定后再做。

---

## 不建议现在做的事情

为了避免过度设计，当前阶段**不建议**：

- 直接恢复完整 deep analyzer
- 为 bootstrap 再抽一层复杂框架
- 引入 AST 深度分析链路
- 做多 agent 认知编排
- 新增数据库、事件总线、持久化审计中心

这些动作都会让当前“最小可执行闭环”重新变重。

---

## 总体建议

下一步最值得做的，不是“让 `first` 更强”，而是“让 `first` 更稳”：

- **真源唯一**：`.spec-first/runtime/first/` 只做唯一机器真源
- **投影可恢复**：`docs/first/` 永远可以从真源重建
- **命令层更薄**：CLI 不承担业务编排
- **策略可验证**：确认策略与测试一起收口
- **增强要克制**：所有优化都围绕闭环收紧，不围绕能力扩张

结论上，当前最佳路线是：

> 先做收口，再做增强；先做闭环一致性，再做分析能力扩展。

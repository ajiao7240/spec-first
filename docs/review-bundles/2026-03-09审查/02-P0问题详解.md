# P0 阻断性问题详解

## P0-1: Skill 过长违反 500 行限制

### 问题描述

**当前状态**: 725 行
**建议上限**: 500 行
**超出比例**: 45%

**skill-creator 原则**:
> "Keep SKILL.md body to the essentials and under 500 lines to minimize context bloat."

### 影响分析

1. **Context Window 占用过多**
   - 725 行约占用 ~15k tokens
   - 挤压其他 skill 和对话历史的空间
   - 降低 AI 的整体响应质量

2. **认知负担过重**
   - AI 需要理解整个 725 行才能执行
   - 增加理解错误和遗漏的风险
   - 降低执行效率

3. **维护成本高**
   - 单文件过长难以定位问题
   - 修改一处可能影响多处
   - 版本控制 diff 难以阅读

### 解决方案

**方案 A: Progressive Disclosure 重构（推荐）**

将 SKILL.md 精简到 ~200 行，拆分为：

```
skills/spec-first/03-spec/
├── SKILL.md (~200 行)
└── references/
    ├── phase0-prd-workflow.md
    ├── steps-fr-ac-workflow.md
    ├── complexity-matrix.md
    ├── anti-rationalization-guards.md
    ├── cli-commands-reference.md
    └── quality-gates.md
```

**方案 B: 拆分为多个 Skill（备选）**

```
skills/spec-first/
├── 03-spec-phase0/  (PRD 生成)
├── 03-spec-main/    (FR/AC 定义)
└── 03-spec-gate/    (质量门禁)
```

**推荐方案 A**，因为：
- 保持单一入口，用户体验更好
- 按需加载，灵活性更高
- 符合 skill-creator 最佳实践

### 实施步骤

1. 创建 `references/` 目录
2. 提取 Phase 0 详细流程到 `phase0-prd-workflow.md`
3. 提取 Step 0-8 详细流程到 `steps-fr-ac-workflow.md`
4. 提取复杂度判定表到 `complexity-matrix.md`
5. 精简 SKILL.md 到核心流程 + 引用链接
6. 验证所有引用路径正确
7. 测试 skill 功能完整性

### 验收标准

- [ ] SKILL.md ≤ 300 行
- [ ] 所有 references 文件创建完成
- [ ] SKILL.md 中所有引用链接有效
- [ ] skill 功能测试通过
- [ ] `scripts/package_skill.py` 验证通过

## P0-2: ID 类型自相矛盾

### 问题描述

**位置**: 第 62 行 vs 第 686 行

**矛盾内容**:
```markdown
# 第 62 行（CLI 硬约束）
❌ 禁止使用 `REQ-PRD`（不存在此类型，应使用 `REQ`）

# 第 686 行（CLI 依赖）
- `spec-first id next REQ-PRD <abbr> --feature <featureId>`
```

### 影响分析

1. **AI 执行困惑**
   - AI 读到第 62 行：不能用 REQ-PRD
   - AI 读到第 686 行：要用 REQ-PRD
   - 导致 AI 不知道该遵循哪个规则

2. **用户体验差**
   - 用户看到矛盾信息会质疑 skill 质量
   - 可能导致错误的 ID 类型使用

3. **追溯矩阵错误**
   - 如果使用错误的 ID 类型，追溯矩阵会失效
   - 影响后续阶段的依赖关系

### 根因分析

需要检查 `references/id-types-and-status.md` 确认权威定义：
- 如果 `REQ-PRD` 是有效类型，删除第 62 行的禁止说明
- 如果 `REQ-PRD` 无效，修正第 686 行为 `REQ`

### 解决方案

**步骤 1: 确认权威定义**
```bash
# 读取 ID 类型定义
cat references/id-types-and-status.md | grep -A 5 "ID 类型"
```

**步骤 2: 统一修正**

**情况 A: REQ-PRD 有效**
- 删除第 62 行的禁止说明
- 更新第 62 行为：`✅ 使用 REQ-PRD 表示 PRD 级别需求`

**情况 B: REQ-PRD 无效**
- 修正第 686 行为：`spec-first id next REQ <abbr> --feature <featureId>`
- 保留第 62 行的禁止说明

**步骤 3: 全文搜索验证**
```bash
# 搜索所有 REQ-PRD 引用
grep -n "REQ-PRD" skills/spec-first/03-spec/SKILL.md
```

### 验收标准

- [ ] ID 类型定义统一
- [ ] 全文无矛盾说明
- [ ] CLI 命令示例正确
- [ ] 追溯矩阵测试通过

## P0-3: Gate Check 时机不明确

### 问题描述

**位置**: 第 609-613 行 vs 第 709 行

**矛盾内容**:
```markdown
# 第 609 行
**Gate Check（Step 8 完成后立即执行）**:
```bash
spec-first gate check <featureId>
```

# 第 709 行（成功标准）
- `gate check` 在 `01_specify` 阶段通过（含 G-SPEC-00 和 C-PRD 校验）
```

**不明确点**:
1. gate check 是在 Step 8 内执行还是 Step 8 后执行？
2. gate check 失败是否阻断后续流程？
3. gate check 失败后如何恢复？

### 影响分析

1. **质量门禁失效**
   - AI 可能在 gate check 失败后继续推进
   - 导致低质量 spec 进入下一阶段

2. **流程不可控**
   - 失败后不知道如何恢复
   - 可能陷入死循环或跳过检查

3. **用户体验差**
   - 用户不知道 gate check 何时执行
   - 失败时不知道如何处理

### 解决方案

**明确 Gate Check 为 Step 8 的阻断门禁**

```markdown
### Step 8: Final Confirmation + Implementation Plan

**执行**:
1. 生成确认包（Goal/Requirements/AC/DoD/OOS）
2. 向用户展示完整确认包
3. 用户确认后生成实施计划
4. 注册所有 FR 到追溯矩阵
5. **执行 Gate Check（阻断门禁）**:
   ```bash
   spec-first gate check <featureId>
   ```

   **检查项**:
   - PRD 存在且 C-PRD ≥ 85%
   - spec.md 存在且包含所有 FR/AC
   - FR 已注册到追溯矩阵
   - spec-review.md 存在（如需要）

   **失败处理**:
   - C-PRD < 85%: 返回 Phase 0.3 修正 PRD，重新执行 Phase 0.4-0.6
   - 缺少文件: 补齐后重新执行 gate check
   - FR 未注册: 执行注册后重新执行 gate check
   - **不得跳过 gate check 进入下一阶段**

   **通过后**:
   - 更新 `stage-state.json` 为 `02_design`
   - 记录 gate check 通过时间到 `findings.md`
   - 提示用户："Gate check 通过，可执行 /spec-first:design"

**输出**:
- `spec.md` 最终版本
- `traceability-matrix.md` 注册所有 FR
- `findings.md` 记录 Step 8 完成 + gate check 通过
```

### 验收标准

- [ ] Gate Check 时机明确（Step 8 第 5 步）
- [ ] 失败处理流程清晰
- [ ] 阻断规则明确（不得跳过）
- [ ] 通过后行为明确（更新阶段）
- [ ] 测试 gate check 失败场景

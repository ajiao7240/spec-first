# Spec-First — 项目定位与背景

## 一、一句话定位

**Spec-First 是面向 AI 协同开发场景的规范驱动研发流程引擎**——以结构化规范为单一真理源，通过全链路追踪、AI 协同与自动化门禁，把"需求→设计→任务→实现→校验→交付"串成可追踪、可验证、可恢复的研发闭环。

> 代码证据：`package.json` — `"description": "Specification-driven development process engine for AI-era software development"`

---

## 二、项目背景：AI 时代的新矛盾

### 2.1 AI Coding 工具的普及带来新问题

2023-2025 年，GitHub Copilot、Cursor、Claude Code 等 AI 编码工具迅速普及，带来了生产力跃升，但同时暴露出四大系统性矛盾：

| 矛盾 | 表现 |
|------|------|
| **上下文丢失** | 多轮对话后 AI 忘记之前的需求和设计决策 |
| **质量失控** | AI 生成代码直接提交，缺少质量门禁和验证 |
| **追溯困难** | 不知道某段代码为什么这么写，需求变更影响范围不清 |
| **协作混乱** | 团队成员使用 AI 的方式不统一，缺少规范约束 |

### 2.2 传统工具的局限

| 传统工具 | 局限 |
|---------|------|
| Jira/Linear | 纯任务追踪，无规范强制，AI 无法理解其语义 |
| Notion/Confluence | 文档存储，无自动校验，与代码脱钩 |
| GitHub Actions | 事后 CI/CD，无前置规范约束 |
| OpenAPI/Swagger | 只管接口层，不覆盖业务规范和实现规范 |
| Cursor/Copilot | 代码补全强，流程管控弱 |

## 三、核心设计原则

> 来源：`PROJECT-INTRODUCTION.md` 设计原则章节

| 原则 | 说明 |
|------|------|
| **规范即契约** | 所有开发活动以规范为准，规范是团队协作的唯一真理源 |
| **全链路追溯** | 从需求到上线，每个环节都可追溯到对应规范 |
| **自动化校验** | 规范可被工具自动解析和校验，减少人工审查成本 |
| **结构化定义** | 采用标准化的规范格式，确保跨项目一致性 |
| **规范先行** | 先定义规范，再进行实现。规范是开发的输入，而非输出 |
| **双向追溯** | 正向：需求是否被实现；反向：实现是否有需求依据 |
| **门禁驱动** | 每个阶段结束必须通过 Gate 校验，不合格不得推进 |
| **AI 原生** | 结构化规范天然适配 AI 理解，实现高效人机协作 |
| **规范注入** | 规范通过工具自动注入上下文（Specs Injected, Not Remembered） |

## 五、目标用户

| 用户类型 | 使用场景 |
|---------|---------|
| **AI 协同团队** | 使用 Claude Code/Cursor 开发，需要规范约束 |
| **中小型研发团队** | 需要轻量化规范体系，但不想引入重型 ERP/PLM |
| **平台/工具开发者** | 开发面向 AI 时代的 SaaS 产品 |
| **质量意识强的独立开发者** | 个人项目也需要完整的需求追踪 |

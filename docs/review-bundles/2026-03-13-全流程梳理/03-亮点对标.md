## 二、差异化亮点详解

### 亮点 1：全链路追溯体系（Spec-First 独有）

**核心差异**：从需求到上线的完整追溯链，而非孤立的任务管理

```
需求（FR）→ 设计（DS）→ 任务（TASK）→ 实现（Code）→ 测试（TC）→ 上线
   │                          │                              │
   └──── 正向：覆盖率检查 ───→┘        ←──── 反向：合规性检查 ──┘
```

**代码证据**：
- `src/core/trace-engine/id-generator.ts` — 14 类 ID 生成
- `src/core/trace-engine/coverage.ts` — 5 项覆盖率计算（C3/C4/C6/C8/C9）
- `src/core/trace-engine/upstream-lineage.ts` — 上游链路追踪

**vs Jira**：Jira 只能追踪 Issue 关联，无统一 ID 体系，跨项目无法比对
**vs GitHub**：GitHub 的 Linked Issues 仅是手动链接，无自动覆盖率计算

---

### 亮点 2：Hard Gate 质量门禁（与 CI/CD 的本质区别）

**核心差异**：阶段推进前的强制质量卡点，而非事后 CI/CD 检查

```
传统 CI：
  代码提交 → PR Review → CI 失败 → 修改 → 重新触发 → 上线

Spec-First：
  阶段推进时 → Gate 检查（blocking 条件） → 不满足则直接阻断 → 修改后才能推进
```

**代码证据**：
```typescript
// src/core/gate-engine/gate-evaluator.ts
// GateResult = { status: 'PASS' | 'FAIL' | 'PASS_WITH_WAIVER', ... }

// src/core/gate-engine/condition-registry.ts (~386 行)
// 每个阶段 8-25 条检查条件
// 07_release 阶段 7 条强制条件（含安全扫描、SCA、覆盖率达标）
```

**vs GitHub Actions**：Actions 的 CI 在代码推上去后才 fail，Spec-First 在阶段推进前就拦截

---

### 亮点 3：规范即代码（Executable Specification）

**核心差异**：规范不只是文档，而是可被机器自动解析、校验、执行的结构化数据

```
传统文档：
  PRD.docx → 人读 → 理解需求 → 开发

Spec-First：
  spec.md（结构化） → 机器解析 → 自动校验 → 生成追踪 ID → Gate 检查
```

**代码证据**：
- `src/core/gate-engine/prd-validator.ts` — PRD 结构自动评分（C-PRD 三维度）
- `src/core/validators/format-validator.ts` — 产物格式自动校验
- `src/core/trace-engine/matrix.ts` — 追踪矩阵自动解析

---

### 亮点 4：AI 协同的上下文恢复机制

**核心差异**：多轮对话后 AI 能自动恢复完整上下文，无需重新梳理

**代码证据**：
```typescript
// src/core/ai-orchestrator/catchup.ts
// 6步恢复：stage-state → task_plan → findings → 定位 → 缺失文件 → 输出摘要
// 包含 FiveQuestions 结构（5 个关键问题快速定位状态）

// src/core/ai-orchestrator/context-pack.ts
// Context Pack < 2KB：spec + design + task + 相关 FR/DS/API
// token_budget: 16000（可配置）
```

**vs Cursor/Copilot**：每次开新文件就失去前文上下文，无法跨轮次保持语义一致性

---

### 亮点 5：指标驱动的阶段化目标

**核心差异**：覆盖率目标随阶段动态调整，贴近真实研发节奏

```
04_implement：C4 目标 = 67%（允许还在开发中）
05_verify：   C4 目标 = 100%（全部功能必须有测试）
07_release：  C4 目标 = 100%（上线前最终检查）
```

**代码证据**：
- `src/core/metrics-engine/core-metric-thresholds.ts` — 分阶段阈值配置
- `src/core/metrics-engine/bottleneck.ts` — R1-R5 瓶颈检测规则

**vs GitHub Actions**：CI 的覆盖率检查是固定阈值（如 80%），不区分阶段

---

### 亮点 6：Skill 体系 — AI 协同的标准协议

**核心差异**：20 个内置 Skill，每个都是可复用、可扩展的 AI 协同协议

```
Skill 执行模型（P0-P5）：
P0_LOCATE      → 定位 Feature + 阶段合法性校验
P1_CONTEXT     → 加载 Context Pack（结构化上下文）
P2_GENERATE    → AI 推理生成交付物草稿
P3_CONFIRM     → 用户确认/修改/拒绝（多轮迭代）
P4_WRITE       → 写入 + 注册追溯 ID
P5_SIDE_EFFECT → 同步矩阵 + 触发 Gate + 更新运行态
```

**vs Claude Code 默认能力**：裸 Claude Code 无规范约束，Spec-First Skills 在 P0 强制检查阶段合法性

---

### 亮点 7：Extension 系统（企业定制）

**核心差异**：团队可以扩展 Skill、添加自定义 Hook，不需要 fork 项目

```yaml
# .spec-first/extensions/my-team/extension.yaml
namespace: my-team
version: 1.0.0
hooks:
  - type: PostToolUse
    matcher: "Write"
    command: "scripts/validate-output.sh"
```

**代码证据**：
- `src/core/process-engine/extensions.ts` — Extension 系统，扫描 `.spec-first/extensions/`

---

### 亮点 8：防篡改审计日志

**核心差异**：哈希链保证审计日志不可篡改，适合合规场景

```typescript
// src/shared/config-schema.ts
audit_log: {
  enabled: true,
  tamper_proof: 'hash_chain',  // 哈希链防篡改
  rotation_size_mb: 10,
}
```

## 三、核心竞争力总结

| 维度 | Spec-First 的主张 |
|---|---|
| **定位** | AI 时代的规范驱动研发流程引擎，不只是任务追踪或代码生成 |
| **核心痛点** | 解决 AI 协同的上下文丢失、质量失控、追溯困难、协作混乱 |
| **差异化** | 规范→代码→测试全链路追溯 + 阶段强制质量门禁 + AI 上下文恢复 |
| **商业价值** | 降低 AI 协同的合规风险、加速研发交付、提升质量可视性 |
| **最佳场景** | AI 驱动开发 + 团队规范统一 + 质量管控严格的项目 |

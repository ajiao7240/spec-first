## 四、数据流图（一次 Skill 执行）

```
用户：/spec-first:spec（需求规格 Skill）
           │
           ▼
  skill-runtime/dispatcher.ts
  ├── Layer 1: Semantic Map 查找 → 未命中
  ├── Layer 2: Runtime Route 查找 → 未命中
  └── Layer 3: resolveSkillPath("spec") → skills/spec-first/03-spec/SKILL.md
           │
           ▼
  Skill 加载管线：
  1. readFile(SKILL.md)
  2. ensureNextStepsPolicy()（后置步骤注入）
  3. assemblePrompt()（KV-cache 稳定性检查）
  4. buildHardGateRuntimeNotice()（阶段合法性注入）
           │
           ▼
  用户执行 Prompt，AI 遵循 P0-P5：
  ├── P0: 定位 Feature（stage-state.json）
  ├── P1: 加载 Context Pack
  │       ├── spec.md（如已有）
  │       ├── design.md
  │       └── 相关 FR/DS/TC
  ├── P2: 生成 spec.md 草稿
  ├── P3: 用户确认/修改
  ├── P4: 写入 spec.md，注册 FR-* ID
  └── P5: sync matrix, evaluate gate
           │
           ▼
  持久化到文件系统：
  specs/{featureId}/spec.md
  specs/{featureId}/traceability-matrix.md（更新）
  .spec-first/runtime/{featureId}/stage-state.json（更新）
```

### SpecFirstConfig 结构

```typescript

interface SpecFirstConfig {
  catchup: {
    trigger: 'auto' | 'prompt' | 'off'  // 默认：prompt

      enabled: boolean           // 默认：false
      max_parallel: number       // 默认：1
      max_retry_per_task: number // 默认：3

      tamper_proof: 'hash_chain' // 防篡改
```

## 七、技术选型理由

| 选择 | 理由 |
|------|------|
| **ESM-only** | Node.js 20+ 原生支持，与现代工具链兼容 |
| **TypeScript strict** | AI 时代代码需要强类型保障，减少运行时错误 |
| **Handlebars 模板** | 逻辑最简的模板引擎，规范文档不需要复杂模板逻辑 |
| **js-yaml** | 配置文件可读性强，YAML 是业界标准 |
| **Vitest** | 与 ESM 原生兼容，v8 覆盖率准确 |
| **tsup** | 零配置打包，专为 TypeScript 库设计 |
| **文件系统作为数据库** | 规范文档天然是文本文件，Git 版本控制，无需额外数据库 |

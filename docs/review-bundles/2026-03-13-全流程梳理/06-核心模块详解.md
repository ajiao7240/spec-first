## 模块 1：process-engine（流程引擎）

**目录**：`src/core/process-engine/`
**职责**：Feature 生命周期驱动，阶段状态机管理

### 关键文件

| 文件 | 职责 |
|------|------|
| `init.ts` | Feature 初始化：ID 生成、目录创建、骨架渲染 |
| `stage-machine.ts` | 阶段状态机：合法转换校验、advance/cancel 逻辑 |
| `layer-merger.ts` | 配置层合并：项目级 + Feature 级 + 默认配置合并 |

| `extensions.ts` | Extension 系统：加载 `.spec-first/extensions/` |

### Feature ID 格式

```
FSREQ-{YYYYMMDD}-{FEAT}-{NNN}
│       │          │      │
│       │          │      └── 3 位序号（同 FEAT 自动递增）
│       │          └── FEAT 缩写（1-16 位，A-Z0-9）
│       └── 日期（当天）
└── 固定前缀
```

示例：`FSREQ-20260313-UIOPT-001`

### 阶段转换规则


## 模块 2：skill-runtime（Skill 运行时）

**目录**：`src/core/skill-runtime/`
**职责**：Skill 分发、Prompt 组装、Hard-Gate 校验

### 三层路由详解

```typescript
// dispatcher.ts
function resolveSkill(input: string, projectRoot: string) {
  // Layer 1: Semantic Map（复合命令映射）
  const semanticMatch = SEMANTIC_MAP.get(input);
  if (semanticMatch) return semanticMatch;

  // Layer 2: Runtime Route（直接 CLI 命令）
  if (RUNTIME_COMMANDS.has(input)) return runtimeRoute(input);

  // Layer 3: Skill File Route
  const skillPath = resolveSkillPath(input, projectRoot);
  if (skillPath) return skillFileRoute(skillPath);

  throw new Error(`未知 Skill: ${input}`);
}
```

### Prompt 组装流程

```
读取 SKILL.md
    │
    ├── 解析 frontmatter（name/description/version）
    ├── 注入 runtime 通知（当前 Feature ID、阶段、上次执行状态）
    ├── 注入 Hard-Gate 说明（当前阶段允许的操作）
    └── 组装最终 Prompt（KV-cache 稳定头部 + 动态尾部）
```

### Hard-Gate 校验

防止在错误阶段执行 Skill，例如：
- 在 `00_init` 阶段执行 `code` Skill → 阻断，提示"请先完成 spec 和 design"
- 在 `07_release` 阶段执行 `spec` Skill → 警告，规范已锁定

### auto-loop.ts — 任务自动循环

**算法**：
```
while (iteration < maxIterations && !halted):
  1. pickReadyTodos()  // 依赖解析，拓扑排序
     └── ready = items where:
           status == 'pending'
           && all deps done
           && resumeAt < now
           && len(running) < max_parallel

  2. ready.length == 0:
     ├── 全部完成 → halt('completed')
     ├── 有退避任务 → wait(earliestResume - now)
     └── 有 blocked → halt('has_blocked') if stop_on_blocked

  3. 执行任务：
     └── executor(task, state) → TaskResult
         ├── success → status = 'done', checkpoint()
         └── failure → retry with backoff
                        ├── retries < max_retry → resumeAt += backoff
                        └── retries >= max_retry → status = 'blocked'

  4. checkpoint()：原子写入 todo-state.json
  5. iteration++
```

## 模块 4：gate-engine（质量门禁）

**目录**：`src/core/gate-engine/`
**职责**：阶段质量门禁评估、PRD 评分、豁免管理

### 条件类型

| 类型 | 说明 | 处理方式 |
|------|------|---------|
| `blocking` | 阻断性条件 | 不满足则 FAIL，阶段无法推进 |
| `warning` | 警告性条件 | 不满足则记录警告，允许推进 |
| `waiver` | 可豁免条件 | 豁免后记为 PASS_WITH_WAIVER |

### 条件分布（各阶段）

| 阶段 | 条件数 | 实际 Gate ID（类型） |
|------|--------|----------------------|
| `01_specify` | 4 | G-SPEC-01/02（**blocking**）· G-SPEC-00/03（warning） |
| `02_design` | 2 | G-DESIGN-01（**blocking**）· G-DESIGN-03（warning） |
| `03_plan` | 3 | G-PLAN-01/02/03（均 **blocking**） |
| `04_implement` | 1 | G-IMPL-01（**blocking**，构建/测试基础检查） |
| `05_verify` | 2 | G-VERIFY-01（**blocking**，C4=100%）· G-VERIFY-03（**blocking**，C6=100%） |
| `06_wrap_up` | 2 | G-WRAP-01/02（均 **blocking**） |
| `07_release` | 2 | G-REL-01/02（均 **blocking**，由 RELEASE_REQUIRED_ARTIFACTS 动态生成） |

### PRD 评分算法（prd-validator.ts）

```
C-PRD = 元信息完整性(30%) + 章节完整性(40%) + 内容质量(30%)

元信息(30%)：
  - 版本号：10%
  - 作者：10%
  - 日期：5%
  - 状态：5%

章节(40%)：
  - 背景/目标：10%
  - 功能需求：15%
  - 非功能需求：5%
  - 验收标准：10%

内容(30%)：
  - FR 列表非空：15%
  - 用户故事存在：10%
  - 优先级标注：5%
```

阈值：< 85% 触发 warning（G-SPEC-00），不阻断阶段推进，但会记录到 Gate 报告

## 模块 5：trace-engine（追溯引擎）

**目录**：`src/core/trace-engine/`
**职责**：追溯 ID 生成/校验/搜索、追踪矩阵、覆盖率计算（共 14 类 ID）

### 5 项核心覆盖率指标

```typescript
// src/core/trace-engine/coverage.ts

/**
 * C3: Task Coverage（正向）
 * = FR 中有 TASK 映射的比例
 * 分子：被任何 TASK 关联的 FR 数量
 * 分母：有效 FR 总数（排除 Deferred/Cancelled）
 */
C3 = coveredFrIds.size / frRows.length

/**
 * C4: Test Coverage - FR（正向）
 * = FR 中有 TC 映射的比例
 */
C4 = coveredByTC.size / frRows.length

/**
 * C6: Implementation Coverage（正向）
 * = TASK 中状态为 Implemented/Verified/Accepted 的比例
 */
C6 = implementedTasks.length / taskRows.length

/**
 * C8: Task Compliance（反向）
 * = TASK 有上游 FR/NFR/DS 的比例（无孤儿 TASK）
 * NFR 通过 fr.nfrTag 字段支持 NFR-PERF-001 格式
 */
C8 = compliantTasks.length / taskRows.length

/**
 * C9: TC Compliance（反向）
 * = TC 有对应 FR 的比例（无孤儿 TC）
 */
C9 = compliantTCs.length / tcRows.length
```

## 模块 6：change-mgr（变更管理）

**目录**：`src/core/change-mgr/`
**职责**：RFC 变更请求、Defect 缺陷追踪

### RFC 状态机

```
Draft → Submitted → UnderReview → Approved → Implementing → Closed
                               └──→ Rejected → Closed
                               └──→ Waived   → Closed（豁免通过）
```

**RFC 分级**：
- `Critical`：架构变更、数据模型变更
- `Major`：接口变更、业务流程变更
- `Minor`：文案调整、样式优化

### Defect 状态机

```
Open → Analyzing → Reproduced → Fixed → Verified → Closed
           ↓                              ↓
         Won't Fix                     Reopened（回归）
```

**Defect 分级（S1-S4）**：
- `S1 Critical`：系统崩溃、数据丢失 — 立即响应
- `S2 High`：核心功能不可用 — < 4h
- `S3 Medium`：功能受损，有 workaround — < 24h
- `S4 Low`：UI 问题、非核心功能 — 下迭代

## 模块 7：metrics-engine（指标引擎）

**目录**：`src/core/metrics-engine/`
**职责**：健康分计算、瓶颈检测、指标报告

### 健康分公式（H1）

```typescript
// src/core/metrics-engine/health-score.ts
H1 = (w1×C3 + w2×C4 + w3×C6 + w4×C8 + w5×C9) × 100 - Math.min(escapedDefectRate × 200, 50)

// 默认权重（可在 config.yaml 中配置）：
w1 = 0.25  // C3 Task Coverage
w2 = 0.20  // C4 Test Coverage
w3 = 0.25  // C6 Impl Coverage
w4 = 0.15  // C8 Task Compliance
w5 = 0.15  // C9 TC Compliance

// Q1 惩罚：缺陷逃逸率，每 1% 扣 2 分，最多扣 50 分
Q1_penalty = Math.min(escapedDefectRate × 200, 50)

// 评分段：
// 90+ → A（优秀）
// 80+ → B（良好）
// 70+ → C（合格）
// 60+ → D（警告）
// <60 → F（不及格）
```

### 瓶颈检测规则（R1-R5）

```typescript
// src/core/metrics-engine/bottleneck.ts
R1: C3 < 0.6 → 需求瓶颈（FR 未拆解为 TASK）
R2: C4 < 0.6 → 测试瓶颈（FR 未覆盖测试用例）
R3: C6 < 0.7 → 实现滞后（TASK 未实现）
R4: C8 < 0.7 → 合规缺口（TASK 无上游需求）
R5: C9 < 0.7 → 追溯缺口（TC 无关联 FR）
```

## 二、各阶段详解

### 00_init — 初始化

**职责**：创建 Feature 工作区，生成唯一 ID，初始化骨架文件

**输入**：用户提供 `feat`（缩写）、`title`、`mode`、`size`、`platforms`

**核心逻辑**（`src/core/process-engine/init.ts`）：
```
1. 校验 FEAT 缩写（1-16 位，A-Z0-9 开头）
2. 生成 Feature ID：FSREQ-{YYYYMMDD}-{FEAT}-{NNN}（自动递增序号）
3. 创建 specs/{featureId}/ 目录
4. 渲染初始骨架文件（Handlebars 模板）
5. 检测背景输入状态（full/degraded/blind）
6. 写入 stage-state.json（初始阶段：00_init）
```

**产出文件**：
```
specs/FSREQ-YYYYMMDD-FEAT-NNN/
├── stage-state.json    # 阶段状态机（核心运行态）
├── config.yaml         # Feature 级配置
├── prd.md              # 空白 PRD 模板
└── spec.md             # 空白规格模板
```

**触发命令**：`spec-first init --feat UIOPT --title "UI优化" --mode design`
**对应 Skill**：`/spec-first:init`

---

### 01_specify — 需求规格

**职责**：定义功能需求（FR）、用户故事、验收标准

**前置条件**：`00_init` 完成，`stage-state.json` 存在

**Gate 条件**（`condition-registry.ts`，3 条）：
- `G1-01`：PRD 文件存在且非空
- `G1-02`：PRD C-PRD 评分 ≥ 85%（`prd-validator.ts`）
- `G1-03`：至少定义 1 条 FR（追踪矩阵中存在 FR 行）

**PRD 评分算法**：
```
C-PRD = 元信息完整性(30%) + 章节完整性(40%) + 内容质量(30%)
元信息：版本号、作者、日期、状态
必需章节：背景/目标、功能需求、非功能需求、验收标准
```

**产出文件**：
- `prd.md` — 产品需求文档
- `spec.md` — 需求规格（FR 列表 + 优先级 + 验收标准）

**对应 Skill**：`/spec-first:spec`（含 spec-review 子 Skill）

---

### 02_design — 技术设计

**职责**：技术方案设计、架构决策、接口定义

**前置条件**：`spec.md` 存在（`config.yaml` 中配置）

**Gate 条件**（2 条）：
- `G2-01`：`design.md` 存在且包含必要章节
- `G2-02`：设计文档关联至少 1 个 FR（DS-FR 追踪）

**产出文件**：
- `design.md` — 技术设计文档（含 DS-* 追溯 ID）

**对应 Skill**：`/spec-first:design`

---

### 03_plan — 任务拆解

**职责**：将设计拆解为可执行的 TASK，排期与里程碑

**前置条件**：`design.md` 存在

**Gate 条件**（4 条）：
- `G3-01`：`task_plan.md` 存在
- `G3-02`：每个 TASK 有唯一 TASK-* ID
- `G3-03`：TASK 总数 ≥ 1
- `G3-04`：TASK 有上游 FR/DS 关联（C8 ≥ 60%）

**task_plan.md 格式**：
```markdown
## TASK-FEAT-001: 实现登录接口
- upstream: FR-001, DS-001
- priority: P0
- estimate: 2h
- status: Pending
```

**对应 Skill**：`/spec-first:task`

---

### 04_implement — 代码实现

**职责**：按任务拆解执行代码实现，持续提交

**前置条件**：`npm run test` 和 `npm run build` 可通过

**自动循环机制**（`auto-loop.ts`）：
```typescript
while (iteration < maxIterations && !halted) {
  1. pickReadyTodos()  // 依赖解析，找可执行任务
  2. 无任务 → done 或 退避等待
  3. 执行 TASK（更新 heartbeat/taskStartedAt）
  4. 原子写入 checkpoint
  5. advance iteration
}
// 配置：max_parallel=1, max_retry=3, retry_backoff=2s
```

**并发执行**：
- `stop_on_blocked: true` — 遇到阻塞停止
- `max_task_duration_ms: 600_000` — 单任务最长 10 分钟
- `heartbeat_timeout_ms: 300_000` — 5 分钟无心跳则告警

**对应 Skill**：`/spec-first:code`

---

### 05_verify — 测试验收

**职责**：系统测试、集成测试、覆盖率达标验收

**前置条件**：`npm run test` 通过

**Gate 条件**：
- C4（测试覆盖率）必须达到 100%（所有 FR 有 TC）
- C6（实现覆盖率）必须达到 100%（所有 TASK 已实现）
- 无 P0/P1 级别的未关闭缺陷（Defect 检查）

**对应 Skill**：`/spec-first:verify`

---

### 06_wrap_up — 复盘归档

**职责**：撰写复盘报告，归档规范文档

**前置条件**：`retro.md` 存在

**产出文件**：
- `retro.md` — 复盘报告（包含 Gate 首次通过率、缺陷分布等）

**对应 Skill**：`/spec-first:archive`

---

### 07_release — 上线发布

**职责**：上线就绪检查，获得 Go-Live 批准

**前置条件**：全套产物存在 + `npm run contract:check` 通过

**Gate 条件**（最严格，7 条）：
- 安全扫描通过（SCA/SAST）
- 覆盖率全部达标（C3/C4/C6/C8/C9）
- 无未关闭的 blocking Defect
- constitution.md 哈希校验通过
- Go-Live 批准记录存在

**触发命令**：
```bash
spec-first golive check   # 就绪检查
spec-first golive approve # 批准上线
spec-first done           # 收口到 08_done
```

## 三、Skill 分发机制（三层路由）

```
用户输入 "/spec-first:spec"
        │
        ▼
┌──────────────────────┐
│  Layer 1: Semantic   │  复合命令映射
│  Map                 │  "rfc approve" → runtime cmd + 参数
└──────────┬───────────┘
           │ 未命中
           ▼
┌──────────────────────┐
│  Layer 2: Runtime    │  RUNTIME_COMMANDS 集合
│  Route               │  直接分发为 CLI 命令
│                      │  (id/matrix/stage/gate/metrics...)
└──────────┬───────────┘
           │ 未命中
           ▼
┌──────────────────────┐
│  Layer 3: Skill      │  resolveSkillPath()
│  Route               │  搜索 skills/spec-first/NN-name/SKILL.md
│                      │  → 包级 fallback
└──────────┬───────────┘
           │
           ▼
  Skill 加载管线：
  1. 读文件（SKILL.md）
  2. ensureNextStepsPolicy()
  3. assemblePrompt()（KV-cache 稳定性检查）
  4. buildHardGateRuntimeNotice()（前置校验注入）
  5. 输出最终 Prompt → 用户执行
```

## 四、Gate 评估流程

```
spec-first gate check --feature FSREQ-20260313-UIOPT-001
          │
          ▼
   loadGateConditions(stage)
          │
          ▼
   for each condition:
     ├── evaluate(condition, featureContext)
     │         │
     │         ├── PASS     → 通过
     │         ├── FAIL     → 记录为 blocking failure
     │         └── WAIVER   → 检查豁免记录是否有效
          │
          ▼
   GateResult {
     status: 'PASS' | 'FAIL' | 'PASS_WITH_WAIVER',
     passed: Condition[],
     failed: Condition[],
     waived: Condition[],
     score: number,
   }
          │
          ├── PASS → 允许 stage advance
          ├── PASS_WITH_WAIVER → 记录豁免，允许推进
          └── FAIL → 阻断推进，输出修复建议
```

## 五、追踪 ID 生命周期

```
1. 生成：spec-first id generate FR --feature FSREQ-...
   ├── 格式：FR-{FEAT}-{NNN}（如 FR-UIOPT-001）
   └── 写入：追踪矩阵 traceability-matrix.md

2. 引用：在 spec.md / design.md / task_plan.md 中使用
   └── 格式：TASK-001 upstream: FR-001, DS-001

3. 同步：spec-first matrix sync
   └── 解析所有产物文件 → 更新矩阵 → 计算覆盖率

4. 校验：gate check 或 validate
   ├── ID 格式校验（format-validator.ts）
   └── 追溯链完整性（trace-context.ts）

5. 追溯：spec-first id search TASK-001
   └── 输出：上游 FR/DS、下游 TC、实现状态
```

## 六、Auto-Loop 任务编排

> **代码证据**：`src/core/ai-orchestrator/auto-loop.ts:101-307`

```
task_plan.md（解析）→ TodoState（依赖解析）→ runAutoLoop()
                                                │
                              ┌─────── while iteration < maxIterations && !halted ─────┐
                              │                                                          │
                              │        pickReadyTodos(state, { maxParallel })           │
                              │                    │                                    │
                              │        ┌────── ready.length == 0? ──────────┐           │
                              │        │                                    │           │
                              │   unfinished == 0          有退避中任务    依赖未就绪    │
                              │   halt('completed')         │              advanceTodoIteration() │
                              │   checkpoint() · return     sleep(waitMs)  checkpoint() · continue│
                              │                             continue                    │
                              │                                                          │
                              │        for each ready task:                             │
                              │                    │                                    │
                              │        markTaskStarted() · checkpoint()                 │
                              │        writeAuditLog('task_started')                    │
                              │                    │                                    │
                              │        executor(task, state) → TaskResult               │
                              │                    │                                    │
                              │        updateHeartbeat() · updateWatchdogCheckedAt()    │
                              │                    │                                    │
                              │        ┌──── runWatchdogCheck() ────┐                  │
                              │        │                            │                  │
                              │   'task_timeout'           'heartbeat_stalled'         │
                              │   status=blocked            status=blocked             │
                              │   halt('task_timeout')      halt('stalled_timeout')    │
                              │   checkpoint() · return     checkpoint() · return      │
                              │                            （均不进入 retry）           │
                              │                                                          │
                              │        result.success ?                                 │
                              │     ┌──────┴──────┐                                    │
                              │    是              否                                   │
                              │     │              │                                    │
                              │  runPostWriteGuards()   makeRetryDecision(errorCategory) │
                              │  ┌───┴───┐         ┌──────────────┴──────────────┐     │
                              │ 通过   失败    temporary（可重试）    非临时/重试耗尽   │
                              │  │      │         │                       │           │
                              │  │  status=blocked status=pending     status=blocked  │
                              │  │      │         setTodoResumeAt(backoffMs)   │      │
                              │  │      │         writeAuditLog('task_retry')  │      │
                              │  │  stop_on_blocked?  checkpoint() · continue  │      │
                              │  │  → halt('blocked') ·return            stop_on_blocked? │
                              │  │                                       → halt('blocked')·return │
                              │ status=done                                             │
                              │ completedTasks.push(taskId)                            │
                              │ writeAuditLog('task_done')                             │
                              │                                                          │
                              │        advanceTodoIteration()                           │
                              │        checkpoint() · onIteration()                     │
                              └──────────────────────────────────────────────────────── ┘
                                                    │
                                           buildResult() · return
```

**关键配置项**（`config.yaml → runtime.auto_orchestrate`）：

| 配置项 | 默认值 | 作用 |
|--------|--------|------|
| `max_parallel` | 1 | 每轮最多并发执行的 TASK 数 |
| `max_retry_per_task` | 3 | 临时错误最大重试次数 |
| `retry_backoff_ms` | 2000 | 初始退避时长（指数增长） |
| `stop_on_blocked` | true | 任意 TASK blocked 时立即 halt |
| `max_task_duration_ms` | 600_000 | 单任务超时（10 min）→ task_timeout |
| `heartbeat_timeout_ms` | 300_000 | 无心跳超时（5 min）→ stalled_timeout |

**halt 原因速查**：

| haltReason | 触发时机 |
|------------|---------|
| `completed` | 所有 TASK status=done |
| `task_timeout` | 单任务执行超过 max_task_duration_ms |
| `stalled_timeout` | 心跳停止超过 heartbeat_timeout_ms |
| `blocked` | TASK 失败且重试耗尽 / post-write guard 失败 |
| `no_state_file` | todo-state.json 不存在 |

## 七、变更与缺陷管理

### RFC 变更流程
```
需求变更申请 → RFC-FEAT-001 (Draft)
           → 影响分析（哪些 FR/TASK/TC 受影响）
           → 评审（Approved/Rejected）
           → 实施（更新受影响产物）
           → RFC closed
```

### Defect 状态机
```
Open → Analyzing → Reproduced → Fixed → Verified → Closed
                                    ↓
                                 Reopened（回归）
```

**分级响应**：
- P0 Critical — 立即响应（< 4h）
- P1 High — 4 小时内
- P2 Medium — 24 小时内
- P3 Low — 下迭代

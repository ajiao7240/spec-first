# 痛点分析：Spec-First 解决了什么

## 一、总览

Spec-First 针对 **AI 协同研发**场景的四大核心痛点，提供了系统性解决方案：

| # | 痛点 | 症状 | 解决方案 |
|---|------|------|---------|
| P1 | **上下文丢失** | AI 忘记之前的需求和设计决策 | 规范注入 + Context Pack + Catchup |
| P2 | **质量失控** | AI 生成代码直接提交，无质量门禁 | 8+2 阶段状态机 + Gate 门禁 |
| P3 | **追溯困难** | 需求变更影响范围不清，代码原因不明 | 追踪 ID 体系 + 追踪矩阵 |
| P4 | **协作混乱** | 团队 AI 使用方式不统一 | Skill 执行模型 + 硬 Gate 约束 |

---

## 二、P1：上下文丢失

### 症状描述

AI 编程助手（如 Claude Code、Cursor）每次会话都是独立的，无法记住：
- 需求背景（"为什么要做这个功能"）
- 设计决策（"为什么选这个技术方案"）
- 当前进度（"上次做到哪里了"）

导致每次开始都要重复解释，AI 给出的方案可能与之前的决策矛盾。

### Spec-First 解决方案

**方案 1：规范即持久化上下文**

规范文件（`specs/<featureId>/`）是跨会话的持久化存储，AI 通过读取规范文件恢复完整背景：

```
specs/FSREQ-20260313-XXX-001/
├── prd.md          ← 产品需求文档（Why）
├── spec.md         ← 需求规格（What）
├── design.md       ← 技术设计（How）
├── task_plan.md    ← 任务拆解（具体步骤）
├── stage-state.json ← 当前阶段状态
└── findings.md     ← 风险与发现日志
```

**方案 2：Context Pack 自动打包**

```typescript
// src/core/ai-orchestrator/context-pack.ts
// 自动打包 < 2KB 的关键上下文
// 内容：当前阶段产物、风险、进度、相关 FR/DS/API
export function buildTaskContextPack(...)
```

**方案 3：Catchup 6 步恢复机制**

```typescript
// src/core/ai-orchestrator/catchup.ts
// 6步恢复：stage-state → task_plan → findings → 定位 → 扫描缺失 → 输出摘要
export function catchup(featureId, projectRoot): CatchupResult
```

恢复结果包含：
- `fiveQuestions`（5-Question Reboot Test）：当前 Feature/阶段、当前任务、上次结论、当前阻塞、下一步行动
- `autoLoopSummary`：Auto-loop 运行时摘要
- `stepRecovery`：Step 级恢复状态

**方案 4：Skill P1_CONTEXT 阶段**

每个 Skill 执行的 P1 阶段会自动加载：
- Context Pack（规范 + 设计 + 任务 + 合同）
- 历史阶段产物
- 当前风险项

**对应命令**：
- `spec-first ai summary` — 会话恢复摘要
- `/spec-first:catchup` — 恢复当前 Feature 上下文

## 三、P2：质量失控

### 症状描述

AI 生成代码的常见质量问题：
- 未测试直接提交（无测试覆盖）
- 代码实现与需求规格不一致
- 关键功能缺少错误处理
- 上线前无系统性验收标准

### Spec-First 解决方案

**方案 1：8+2 阶段状态机**

```
00_init → 01_specify → 02_design → 03_plan → 04_implement
        → 05_verify → 06_wrap_up → 07_release → 08_done
                                                → 09_cancelled
```

每个阶段有明确的：
- **输入前提**（前一阶段完成）
- **交付物要求**（必须存在的文件）
- **验收标准**（Gate 条件）

**方案 2：Gate 质量门禁（三态评估）**

```typescript
// src/core/gate-engine/gate-evaluator.ts
// 三态结果：PASS / FAIL / PASS_WITH_WAIVER
export function evaluateGate(featureId, projectRoot, options): GateResult
```

条件注册表（`condition-registry.ts`，~386 行）按阶段组织：
- `01_specify`：PRD 完整性（3 条）
- `02_design`：设计文档（2 条）
- `03_plan`：任务拆解（4 条）
- `07_release`：上线就绪（7 条）

条件分为：
- **blocking**（强制）— 不满足不得推进
- **warning**（警告）— 提示但不阻断
- **waiver**（豁免）— 记录原因后可通过

**方案 3：PRD 自动评分**

```typescript
// src/core/gate-engine/prd-validator.ts
// C-PRD 评分 = 元信息 30% + 章节 40% + 内容 30%
// 阶段 01_specify 时自动评估，< 85% 则阻断推进
```

**方案 4：健康分综合评估**

```typescript
// src/core/metrics-engine/health-score.ts
H1 = (25%×C3 + 20%×C4 + 25%×C6 + 15%×C8 + 15%×C9) × 100 - Q1_penalty
// 评分段：90+ A, 80+ B, 70+ C, 60+ D, <60 F
```

**方案 5：阶段依赖检查**

```yaml
# config.yaml 默认配置
dependencies:
  stages:
    '02_design': files: [prd.md, spec.md]  # 必须存在
    '03_plan':   files: [design.md]
    '04_implement': npmScripts: [test, build]  # 必须通过
    '07_release': files: [全套产物], npmScripts: [contract:check]
```

**对应命令**：
- `spec-first gate check` — 质量门禁检查
- `spec-first metrics report` — 覆盖率报告
- `spec-first golive` — 上线就绪检查

## 四、P3：追溯困难

### 症状描述

- 代码里看到一个函数，不知道它对应哪个需求
- 产品说要改一个功能，不知道会影响多少测试用例
- 复盘时无法还原当时的决策过程
- 缺陷报告无法快速定位到对应的需求规格

### Spec-First 解决方案

**方案 1：12 类追溯 ID 体系**

```typescript
// src/shared/types.ts
type IdType =
  | 'FR'      // Feature Requirement（功能需求）
  | 'DS'      // Design Spec（设计规格）
  | 'TASK'    // 任务
  | 'TC'      // Test Case（测试用例）
  | 'RFC'     // Request for Change
  | 'REQ'     // 需求
  | 'SYS'     // 系统级需求
  | 'ARCH'    // 架构决策
  | 'MOD'     // 模块
  | 'ATP'     // 验收测试计划
  | 'STP'     // 系统测试计划
  | 'ITP'     // 集成测试计划
  | 'UTP'     // 单元测试计划
  | 'Feature' // Feature 级别
```

**方案 2：追踪矩阵（双向追溯）**

```
FR-001 ──→ TASK-001 ──→ TC-001
    │          │
    └──→ DS-001  └──→ TC-002
              │
              └──→ TASK-002 ──→ TC-003
```

- **正向**：FR 是否被拆解为 TASK？（C3）
- **正向**：FR 是否有对应 TC？（C4）
- **反向**：TASK 是否有上游 FR/DS？（C8，检测孤儿 TASK）
- **反向**：TC 是否关联 FR？（C9，检测孤儿 TC）

**方案 3：覆盖率矩阵（C1-C9）**

| 指标 | 含义 | 计算方式 |
|------|------|---------|
| C3 | Task Coverage | FR 中有 TASK 的比例 |
| C4 | Test Coverage | FR 中有 TC 的比例 |
| C6 | Impl Coverage | TASK 中已实现的比例 |
| C8 | Task Compliance | TASK 有上游需求的比例 |
| C9 | TC Compliance | TC 有对应 FR 的比例 |

**方案 4：RFC 变更追踪**

```typescript
// src/core/change-mgr/rfc.ts
// RFC 状态机：Draft → Submitted → Approved/Rejected → Waived
// 每次变更都记录：影响范围 + 变更原因 + 审批人
```

**对应命令**：
- `spec-first id generate FR` — 生成追溯 ID
- `spec-first matrix sync` — 同步追踪矩阵
- `spec-first trace repair` — 修复追踪链断链
- `spec-first analyze` — 跨产物一致性分析
- `spec-first rfc create` — 创建变更请求

## 五、P4：协作混乱

### 症状描述

团队 5 个人用 AI 开发，5 种方式：
- 有人直接让 AI 写代码，有人让 AI 写文档
- PRD 格式各不相同，无法互相理解
- 交付物命名混乱，无法自动解析
- 新成员不知道从哪里开始

### Spec-First 解决方案

**方案 1：Skill 统一执行模型（P0-P5）**

```
P0_LOCATE      → 定位 Feature 并校验阶段合法性
P1_CONTEXT     → 加载 Context Pack、阶段产物、历史状态
P2_GENERATE    → AI 推理生成交付物草稿
P3_CONFIRM     → 用户确认、修改或拒绝（支持多轮迭代）
P4_WRITE       → 写入最终交付物并注册追溯 ID
P5_SIDE_EFFECT → 同步追踪矩阵、触发 Gate 评估、更新运行态
```

所有 Skill 都遵循这个模型，确保输出可追溯、格式统一。

**方案 2：22 个内置 Skill 覆盖全流程**

| 类型 | Skill 数量 | 覆盖范围 |
|------|-----------|---------|
| 认知层 | 2 | onboarding, first |
| 核心阶段 | 8 | init, spec, design, research, task, code, review, archive |
| 支撑编排 | 7 | plan, verify, orchestrate, status, doctor, sync, feature |
| 质量扩展 | 2 | spec-review, analyze |
| 会话恢复 | 1 | catchup |
| 其他 | 2 | onboarding, first |

**方案 3：硬 Gate 前置校验**

```typescript
// src/core/skill-runtime/hard-gate.ts
// Skill 执行前必须通过的前置检查
// 防止在错误阶段执行错误 Skill
```

**方案 4：产物格式校验**

```typescript
// src/core/validators/format-validator.ts
// 自动检查 PRD/规格文档的格式合规性
// ID 格式、章节完整性、必填字段
```

**方案 5：Extension 系统（团队定制）**


## 六、额外痛点：规范漂移

### 症状描述

项目进行到一半，需求变了，但：
- 设计文档没更新
- 测试用例还是旧的
- 代码已经按新需求实现了
- 三者不一致，没有人知道

### Spec-First 解决方案

**跨产物一致性分析**（`analyze` 命令）：
- 自动检测规范→设计→任务→测试的链路断裂
- 生成"孤儿"清单（无上游的 TASK、无对应 FR 的 TC）
- 计算覆盖率缺口

**RFC 变更控制**：
- 任何需求变更必须走 RFC 流程
- RFC 记录影响范围（哪些 TASK/TC 需要更新）
- 审批后才能推进

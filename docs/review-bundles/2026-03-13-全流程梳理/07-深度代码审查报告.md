### 3.2 C3/C4/C8/C9 计算逻辑对比

| 指标 | 分子 | 分母 | 遍历方式 | 关键差异 |
|------|------|------|---------|---------|
| **C3** | 被任意 TASK 覆盖的 FR 数 | 有效 FR 总数 | **传递祖先**（TASK→DS→FR 算覆盖）| 允许间接链路 |
| **C4** | 被任意 TC 直接引用的 FR 数 | 有效 FR 总数 | **直接 upstream**（TC.upstream 含 FR-* 才算）| 不允许间接链路 |
| **C6** | TASK.status 为 Implemented/Verified/Accepted | 有效 TASK 总数 | 仅看状态字段 | 无追溯逻辑 |
| **C8** | TASK 有祖先/直接 upstream 为 FR 或 DS | 有效 TASK 总数 | **传递祖先 + 直接 upstream** | 支持 NFR-TAG 格式 |
| **C9** | TC 直接 upstream 含 FR-* | 有效 TC 总数 | **直接 upstream** | 无传递逻辑 |

### 3.3 🔴 核心设计不对称：C3 传递 vs C4 直接

**代码证据**：

```typescript
// coverage.ts — C3：传递祖先遍历
function calcTaskCoverage(frRows, taskRows, lineage) {
  const coveredFrIds = lineage.collectCoveredTargetIds(
    taskRows.map((task) => task.id),
    frIds  // 向上找到 FR 祖先
  );
  return pct(coveredFrIds.size, frRows.length);
}

// coverage.ts — C4：只看直接 upstream
function calcTestCoverageFR(frRows, tcRows) {
  return calcUpstreamCoverage(frRows, tcRows);  // 只检查 row.upstream[] 是否含 FR-*
}
```

**实际影响演示**：

```
矩阵内容：
  FR-FEAT-001    upstream=[]
  DS-FEAT-001    upstream=[FR-FEAT-001]
  TASK-FEAT-001  upstream=[DS-FEAT-001]   ← 不直接引用 FR
  TC-FEAT-001    upstream=[DS-FEAT-001]   ← 不直接引用 FR

C3 计算：TASK-FEAT-001 → DS-FEAT-001 → FR-FEAT-001 ✓（传递找到 FR）
         → C3 = 1/1 = 100%  ✅

C4 计算：TC-FEAT-001.upstream = [DS-FEAT-001]
         frIds = {FR-FEAT-001}
         DS-FEAT-001 ∉ frIds → FR-FEAT-001 未被覆盖
         → C4 = 0/1 = 0%    ❌

C8 计算：TASK-FEAT-001 has DS-FEAT-001 in allowedUpstreamIds ✓
         → C8 = 100%  ✅

C9 计算：TC-FEAT-001.upstream = [DS-FEAT-001]
         DS-FEAT-001 ∉ frIds → TC-FEAT-001 不合规
         → C9 = 0/1 = 0%    ❌
```

**结论**：TC 必须在 `upstream` 字段**直接列出 FR ID**，通过 DS 的间接链路不被 C4 和 C9 认可。但 TASK 可以通过 DS 间接关联 FR（C3 和 C8 都支持传递链）。这种不对称性如果用户不了解，会导致 C4/C9 在"合理配置"下仍然为 0。

### 3.4 上下游 upstream/downstream 字段的真实用途

**代码证据**：`upstream-lineage.ts:29`

```typescript
// 覆盖率计算只读取 upstream 字段
for (const upstreamId of row?.upstream ?? []) {
  ancestors.add(upstreamId);
}
```

| 字段 | 用于覆盖率（C3-C9）| 用于矩阵完整性校验 | 用于 V-Model 对验证 |
|------|-------------------|-------------------|---------------------|
| `upstream` | ✅ 核心数据来源 | ✅ 孤儿检测 | ✅ |
| `downstream` | ❌ **不使用** | ✅ 仅用于 checkVModelPairs | ✅ |

**关键结论**：`downstream` 字段对覆盖率指标**完全无效**。所有覆盖率计算均通过从 child item 向上查找 `upstream` 实现。如果只维护 `downstream` 而不在子项维护 `upstream`，覆盖率将错误为 0。

## 四、Exception 机制深度审查

### 4.1 Exception 有效性条件

**代码证据**：`exception-validator.ts:45-72`

```typescript
function checkException(ex: KnownException, rfcStatuses: Map<string, string>): string[] {
  const reasons: string[] = [];

  // 条件 1：RFC 必须存在且状态为 approved
  const rfcStatus = rfcStatuses.get(ex.rfcId);
  if (!rfcStatus) reasons.push(`未找到 RFC ${ex.rfcId}`);
  else if (rfcStatus !== 'approved') reasons.push(`RFC 状态为 "${rfcStatus}"，预期 "approved"`);

  // 条件 2：expiresAt 必须存在且未过期
  if (!ex.expiresAt) reasons.push('缺少 expires_at');
  else if (new Date(ex.expiresAt) < new Date()) reasons.push(`已过期：${ex.expiresAt}`);

  // 条件 3：rollbackPoint 必须存在
  if (!ex.rollbackPoint) reasons.push('缺少 rollback_point');

  return reasons; // 非空 → Exception 无效
}
```

**Exception 失效的影响**：

```
valid Exception → FR 排除出覆盖率分母（不影响 C3/C4 计算）
invalid Exception → FR 计入覆盖率分母（被视为普通 FR）
```

### 4.2 Exception 的 known-exceptions.md 格式

| EX-ID | RFC-ID | FR-ID | Reason | ExpiresAt | RollbackPoint | ApprovedBy | ApprovedAt |
|-------|--------|-------|--------|-----------|---------------|------------|------------|
| EX-001 | RFC-001 | FR-FEAT-001 | ... | 2026-12-31 | v1.2.0 | leo | 2026-03-13 |

## 六、全流程上下游关联正确性总结

### 6.1 合法的上下游关联模式

```
合规关联图（✅ = 覆盖率计算认可）

REQ-* ──→ FR-*     （FR.upstream 含 REQ-*）
          FR-*
           ├──→ DS-*     （DS.upstream 含 FR-*）
           │     └──→ TASK-*  （TASK.upstream 含 DS-* 或 FR-*，C3/C8 传递认可）
           └──→ TC-*     （TC.upstream 必须直接含 FR-*，C4/C9 不传递）

NFR 特殊格式：
  FR-PERF-001 [title 含 [NFR:PERF]]
  → nfrTag = 'PERF'
  → TASK.upstream 可写 NFR-PERF（C8 认可）
```

### 6.2 常见错误配置及后果

| 错误配置 | 现象 | 影响指标 |
|---------|------|---------|
| TC.upstream 只引用 DS（无直接 FR）| C4 = 0%，G-VERIFY-01 FAIL | C4↓, C9↓ |
| TASK.upstream 为空 | C8 < 100%，G-PLAN-02 FAIL | C8↓ |
| 只维护 downstream，不维护 upstream | 所有覆盖率指标为 0 | 全部↓ |
| Exception RFC 未 approved | Exception 无效，FR 计入分母 | C3/C4 可能↓ |
| status 写 "blocked" | 映射为 Deferred → 排出分母 | C3/C4 虚高 |
| FR 无任何 TC 直接引用 | checkMatrix 警告（不阻塞 Gate） | — |

### 6.3 追溯链完整性验证路径

1. **`spec-first id search {ID}`** → 查看某 ID 的上下游关联
2. **`spec-first matrix sync`** → 重新解析并更新追踪矩阵
3. **`spec-first validate`** → format-validator 校验 ID 格式
4. **`spec-first trace repair`** → 修复断链（需用户确认）
5. **`spec-first metrics`** → 实时查看 C3/C4/C6/C8/C9

## 七、审查发现与优化建议

### 7.1 🔴 高优先级问题

#### 问题 1：C4/C9 不支持传递链，但用户容易误配置

- **根因**：C4 用 `calcUpstreamCoverage(frRows, tcRows)` 只检查 `TC.upstream` 是否含 FR-*；C9 同理
- **用户误区**：认为 TC→DS→FR 等同于 TC→FR，实际上不计入
- **建议**：在 Skill 文档和 Gate 报告中明确"TC 必须直接引用 FR"

#### 问题 2：DS→FR 追踪关联在 Gate 层无 blocking 条件

- **根因**：G-DESIGN-02 已被移除，当前 02_design 阶段 Gate 只检查 design.md 存在
- **风险**：设计文档中的 DS 项可以不关联任何 FR，直至 03_plan 阶段 G-PLAN-02（C8 = 100%）才间接捕获
- **建议**：考虑在 02_design 阶段补充 blocking 条件：`至少 1 条 DS 关联了 FR`

### 7.2 🟡 中优先级问题

#### 问题 3：Gate 条件 ID 编号存在跳号（改造方案有意设计）

跳号 ID：G-DESIGN-02（C1/C2）、G-IMPL-02（C7）、G-VERIFY-02（C5）均已按改造方案删除，跳号是已删除条件的编号占位。

建议在代码注释中标注这些 ID 为"已删除（改造方案 v2.0）"，防止未来维护者误以为是漏实现。

#### 问题 4：shouldSkipCondition 逻辑未文档化

frontend/css-only 项目静默跳过 G-IMPL-01 和 G-VERIFY-01，建议在 Gate check 输出中显示 `SKIPPED` 状态。

#### 问题 5：RELEASE_REQUIRED_ARTIFACTS 数组顺序与 Gate ID 顺序相反

index 0（smoke-test-report.md）→ G-REL-02；index 1（release-note.md）→ G-REL-01。反直觉，建议添加注释说明。

### 7.3 ⚠️ 需知晓的行为

#### 行为 1：状态别名映射的副作用

将 TASK 状态写为 `blocked` 会映射为 `Deferred`（从分母排除），C6 看似通过但实际有未完成任务。

#### 行为 2：Exception 失效的静默影响

当 RFC 审批状态变化后（如从 approved 变回 closed），对应 Exception 自动失效，原本豁免的 FR 重新计入分母，覆盖率可能突然下降。

#### 行为 3：G-PLAN-03 自动刷新 analysis-report.md

报告超过 5 分钟自动重新分析，这会在 Gate check 时产生额外的 I/O 副作用。在 CI 环境中可能导致不确定性。

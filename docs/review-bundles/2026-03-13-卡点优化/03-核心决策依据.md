# 核心决策依据 - 为什么是这 4+1 个卡点?

> **版本**: v1.0
> **日期**: 2026-03-13

---

## 🎯 决策原则

### 原则 1: 防御灾难性失败,而非小问题

**保留**: C6/C7/C8/C9 - 防止"孤儿数据"和"未完成交付"
**移除**: C1/C2/C3/C10/C11 - 小问题可以通过 Code Review 发现

### 原则 2: 自动化检查 vs 人工判断

**自动化适用**: 二元判断(有/无关联)
**人工判断适用**: 主观质量(文档好坏、架构合理性)

### 原则 3: 成本-收益比

**高 ROI**: 实现简单、维护成本低、收益高
**低 ROI**: 实现复杂、容易误报、收益有限

---

## 🔍 逐项分析

### ✅ C8: Task Compliance = 100%

**为什么必须保留?**
1. **防止孤儿任务**: 没有上游关联的任务 = 技术债
2. **实现简单**: 只需检查 `upstream` 字段是否为空
3. **误报率低**: 二元判断,不存在灰色地带
4. **行业标准**: Jira/Azure DevOps 都强制 Issue 关联

**真实案例**:
```
场景: 开发者创建 TASK-123 "优化数据库查询"
问题: 没有关联任何 FR/DS
后果: 6 个月后无人知道为什么要优化,代码被误删
```

**成本**: 低(简单的字段检查)
**收益**: 高(防止技术债累积)
**ROI**: ⭐⭐⭐⭐⭐

---

### ✅ C7: PR Compliance = 100%

**为什么必须保留?**
1. **防止无法追溯的代码**: 每个 PR 必须能追溯到需求
2. **实现简单**: 检查 PR 描述中是否包含 TASK ID
3. **误报率低**: 格式化检查,容易自动化
4. **行业标准**: GitLab/GitHub 都推荐 PR 关联 Issue

**真实案例**:
```
场景: 开发者提交 PR "fix bug"
问题: 没有关联任何 TASK
后果: 代码审查时无法理解上下文,可能引入新 bug
```

**成本**: 低(PR 模板 + 自动化检查)
**收益**: 高(确保变更可追溯)
**ROI**: ⭐⭐⭐⭐⭐

---

### ✅ C9: TC Compliance = 100%

**为什么必须保留?**
1. **防止孤儿测试**: 测试必须验证具体需求
2. **实现简单**: 检查 TC 的 `upstream` 字段
3. **误报率低**: 二元判断
4. **质量保障**: 确保测试有明确目标

**真实案例**:
```
场景: 测试工程师创建 TC-456 "测试登录功能"
问题: 没有关联任何 FR
后果: 需求变更时,不知道哪些测试需要更新
```

**成本**: 低(简单的字段检查)
**收益**: 高(确保测试覆盖有效)
**ROI**: ⭐⭐⭐⭐⭐

---

### ✅ C6: Implementation Coverage = 100%

**为什么必须保留?**
1. **确保交付完整性**: 归档前必须完成所有任务
2. **实现简单**: 统计 TASK 状态
3. **误报率低**: 状态枚举,明确定义
4. **合同保障**: 避免客户投诉

**真实案例**:
```
场景: Feature 进入归档阶段
问题: 还有 3 个 TASK 状态为 "In Progress"
后果: 客户发现功能不完整,要求退款
```

**成本**: 低(状态统计)
**收益**: 高(避免交付风险)
**ROI**: ⭐⭐⭐⭐⭐

---

### ⚠️ C4: Test Coverage ≥ 60%

**为什么保留但降低阈值?**
1. **平衡质量与效率**: 60% 是工程实践的甜蜜点
2. **支持豁免**: 允许部分 FR 延后测试
3. **行业实践**: Google 不强制阈值,团队自主决定
4. **边际收益递减**: 80% → 100% 的成本远高于收益

**数据支持**:
```
覆盖率 | 发现 Bug 比例 | 成本
-------|--------------|------
60%    | 85%          | 1x
80%    | 92%          | 2x
100%   | 95%          | 5x
```

**成本**: 中(覆盖率统计)
**收益**: 中(减少 bug)
**ROI**: ⭐⭐⭐

---

### ❌ C1/C2: Design Coverage

**为什么移除?**
1. **小需求不需要独立设计**: 可以在 spec.md 中内联
2. **100% 过严**: 配置类/文档类需求不需要设计
3. **可替代**: Code Review 可以检查设计质量
4. **误报率高**: 自动化难以判断"设计是否充分"

**真实案例**:
```
场景: FR-001 "更新 README 文档"
问题: 被要求创建 DS-001 设计文档
后果: 团队花 2 小时写形式化文档,实际价值为 0
```

**成本**: 中(覆盖率计算)
**收益**: 低(可替代)
**ROI**: ⭐⭐

---

### ❌ C3: Task Coverage

**为什么降级为警告?**
1. **部分 FR 不需要拆任务**: 文档类/配置类
2. **100% 过严**: 可能是合理的范围缩减
3. **可替代**: 项目管理会议可以检查
4. **灵活性**: 允许团队根据实际情况调整

**真实案例**:
```
场景: FR-002 "配置 CI/CD 环境变量"
问题: 被要求创建 TASK
后果: 5 分钟的工作被拆成 1 小时的任务管理
```

**成本**: 中(覆盖率计算)
**收益**: 低(过度管理)
**ROI**: ⭐⭐

---

### ❌ C5: AC Coverage

**为什么删除?**
1. **与 C4 重复**: AC 是 TC 的子集
2. **粒度过细**: 实际价值有限
3. **维护成本高**: 需要区分 AC 级别和功能级别测试
4. **行业实践**: 主流工具不区分 AC 覆盖率

**真实案例**:
```
场景: FR-003 有 5 个 AC,创建了 10 个 TC
问题: 系统要求每个 AC 至少有 1 个 TC
后果: 团队花时间调整 TC 的 upstream 映射,而非写测试
```

**成本**: 高(复杂计算)
**收益**: 低(与 C4 重复)
**ROI**: ⭐

---

### ❌ C10: Spec Quality Score

**为什么移除?**
1. **主观质量难量化**: 自动化打分容易被"刷分"
2. **应该人工评审**: Code Review 更有效
3. **误导性**: 分数高 ≠ 质量好
4. **行业实践**: 没有工具用自动化打分评估文档质量

**真实案例**:
```
场景: Spec 质量评审得分 85%
问题: 所有检查项都勾选了,但内容空洞
后果: 通过了自动化检查,但实际质量很差
```

**成本**: 高(复杂规则)
**收益**: 低(易被绕过)
**ROI**: ⭐

---

### ❌ C11: Constitution Compliance

**为什么移除?**
1. **架构合规应该人工检查**: 架构师评审会议
2. **自动化检查容易误报**: 规则难以穷举
3. **灵活性不足**: 创新方案可能不符合现有规则
4. **行业实践**: 架构决策通过 ADR(Architecture Decision Record)

**真实案例**:
```
场景: 设计使用新的缓存方案
问题: Constitution 中没有定义,自动化检查失败
后果: 团队被迫使用旧方案,错过性能优化机会
```

**成本**: 高(规则维护)
**收益**: 低(误报率高)
**ROI**: ⭐

---

### ❌ C-PRD: PRD Completeness

**为什么移除?**
1. **章节完整性 ≠ 内容质量**: 形式主义
2. **小需求不需要完整 PRD**: 过度文档化
3. **应该人工评审**: 产品经理 Review
4. **行业实践**: 敏捷开发推崇"Just Enough Documentation"

**真实案例**:
```
场景: 小功能优化,PRD 只有 2 页
问题: 缺少"术语表"章节,得分 80%
后果: 团队被迫添加空章节,浪费时间
```

**成本**: 中(章节检查)
**收益**: 低(形式主义)
**ROI**: ⭐

## 🎯 最终结论

**保留的 4+1 个卡点**:
1. C8/C7/C9: 防止"孤儿数据" - 这是追溯性的核心
2. C6: 防止"未完成交付" - 这是合同保障
3. C4: 防止"Bug 风险" - 但降低阈值到 60%

**核心逻辑**:
- **追溯性 > 覆盖率**: 能追溯比覆盖率高更重要
- **人工判断 > 自动化打分**: 质量靠 Review,不靠指标
- **防御灾难 > 防御小问题**: 卡点应该防止严重问题

**行业对标**:
- Google/Microsoft/Netflix 都强制追溯性
- 没有主流工具强制 100% 覆盖率
- 质量保障靠 Code Review + CI/CD

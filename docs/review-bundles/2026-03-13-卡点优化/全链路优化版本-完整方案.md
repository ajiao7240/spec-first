# Spec-First 全链路优化版本：完整方案

## 1. 文档目的

本文档给出一份完整的全链路优化方案，用来替代只讨论单点 Gate 精简的局部方案。

目标不是简单减少几个校验项，而是重新定义 Spec-First 的默认推进链，使其同时满足三点：

- 保留研发质量底线
- 显著降低默认流程阻力
- 保证系统实现、CLI 输出、指标体系、审计记录彼此一致

本文档覆盖的范围包括：

- 阶段推进链
- 依赖检查链
- Gate 条件链
- Layer2 / 扩展 Gate 链
- coverage / metrics / health-score / bottleneck 指标链
- CLI / findings / gate-history 输出链
- GoLive 消费链
- profile / 配置化扩展链

---

## 2. 当前系统的真实阻断链路

要做全链路优化，首先必须承认当前系统的阻断并不只来自 `GateEvaluator`。

当前真实推进路径如下：

1. `advance()` 读取当前阶段状态
2. 计算目标阶段
3. 先执行 `checkDependencies()`
4. 再执行 `evaluateGate()`
5. Gate 内部先评估内建条件
6. 再执行 Layer2 / 扩展 command gate
7. 再做 waiver 匹配
8. 聚合出 `PASS / PASS_WITH_WAIVER / FAIL`
9. 写入 `gate-history.jsonl`
10. `CLI`、`status`、`GoLive`、`metrics` 再继续消费这些结果

这意味着：

- 只删 Gate 条件，不等于默认流程真的变轻
- 只改 evaluator，不等于 CLI 和历史会正确表达 warning
- 只改 Gate，不等于健康分和瓶颈分析不会继续放大旧复杂度

因此，这次优化必须按整条链路来做。

---

## 3. 当前链路的核心问题

## 3.1 Gate 存在重复卡点

当前实现中的重复关系比较明确：

- `C1` 与 `C2` 当前都基于 `FR -> DS` 计算
- `C4` 与 `C5` 当前都基于 `FR -> TC` 计算
- `C7` 与 `C8` 都在约束 TASK 上游关系，`C8` 语义更强
- `C-PRD` 与 `C10` 都在 `01_specify` 阶段评估规格质量

结果是用户会感知到：

- 同一问题被重复阻断
- 同一问题在不同阶段再次出现
- 对应的 waiver 也必须重复兜底

## 3.2 默认推进链阻断面过宽

当前默认链路中，很多问题会同时出现在：

- 依赖检查
- Gate 条件
- 指标报告
- CLI 失败输出

因此用户不是只“被 Gate 卡住”，而是被整套系统连续施压。

## 3.3 warning 语义缺失

当前系统只有：

- `PASS`
- `WAIVER`
- `FAIL`

没有“失败但不阻断”的一等语义。

结果是：

- 文档可以说“这个条件只是提醒”
- 但 CLI 仍会把它显示为 `[FAIL]`
- 历史记录里也无法区分“硬失败”和“告警”

## 3.4 指标消费层和默认 Gate 已经失配

即使后续删掉某些默认 Gate，如果以下模块不一起改：

- `health-score`
- `bottleneck`
- `metrics`
- `status/analyze`

系统仍会继续强调那些已经不再属于默认治理面的问题。

## 3.5 配置化能力存在，但没有贯穿整条链路

当前系统已经有：

- Layer 0 / 1 / 2 / 3 规则合并
- `mode / size / platform`
- `thresholds`
- `mergedRules.gateConditions`

但这些能力主要停留在“有合并结构”，没有真正变成一条可切换的默认治理 profile。

---

## 4. 全链路优化目标模型

## 4.1 默认链路的治理目标

默认链路只负责守住 5 个最关键问题：

- 需求是否已落成任务
- TASK 是否有合法来源
- 需求是否获得测试覆盖
- 测试是否可追溯到需求
- 任务是否真实完成

这 5 个问题分别对应：

- `C3`
- `C8`
- `C4`
- `C9`
- `C6`

## 4.2 默认链路的指标模型

默认 blocking：

- `C3`
- `C4`
- `C6`
- `C8`
- `C9`

默认删除：

- `C1`
- `C2`
- `C5`
- `C7`

默认 warning：

- `C-PRD`
- `C10`
- `C11`

这是整个方案的核心决策，不是只针对某一个模块的局部建议。

## 4.3 为什么保留这 5 项

`C3` 决定需求是否真正进入执行层。

`C4` 决定需求是否有基本测试覆盖。

`C6` 决定归档前任务是否真实闭合。

`C8` 决定 TASK 是否有合法来源，是防止孤儿任务的主指标。

`C9` 决定测试是否可追溯，是验证阶段的核心指标。

## 4.4 为什么删除这 4 项

`C1/C2` 当前实现重复，不适合作为两个独立的默认阻断。

`C5` 当前与 `C4` 逻辑重复，在 AC 级追踪没有落地前不应单独成门。

`C7` 已被 `C8` 覆盖，默认保留只会增加理解成本。

## 4.5 为什么把质量项降为 warning

`C-PRD`、`C10`、`C11` 仍然重要，但更适合：

- 作为质量提示
- 作为评审反馈
- 作为 strict 模式的增强治理项

它们不应该继续作为大多数团队的默认硬阻断。

---

## 5. 默认推进链的阶段定义

以下定义描述的是默认模式下的正式阻断面，不包含 strict profile 和 Layer2 额外增强。

## 5.1 `01_specify`

默认 blocking：

- `spec.md exists`
- `FR/NFR IDs assigned`

默认 warning：

- `C-PRD`
- `C10`

阶段目标：

- 保证需求已被结构化表达
- 不再在默认模式下叠加双重质量评分阻断

## 5.2 `02_design`

默认 blocking：

- `design.md exists`

默认 warning：

- `C11`

默认删除：

- `C1`
- `C2`

阶段目标：

- 保证存在设计产物
- 不再把设计完整映射与架构合规作为默认必过项

## 5.3 `03_plan`

默认 blocking：

- `C3 = 100%`
- `C8 = 100%`
- `critical findings = 0`

阶段目标：

- 保证所有需求已经落成任务
- 保证任务本身合法

## 5.4 `04_implement`

默认 blocking：

- `C4 >= 80%`

默认删除：

- `C7`

阶段目标：

- 保证进入实现阶段后，至少已有基本测试覆盖

## 5.5 `05_verify`

默认 blocking：

- `C4 = 100%`
- `C9 = 100%`

默认删除：

- `C5`

阶段目标：

- 保证需求覆盖完整
- 保证测试用例可追溯

## 5.6 `06_wrap_up`

默认 blocking：

- `C6 = 100%`
- `all entries terminal`

阶段目标：

- 保证任务已真实完成
- 保证矩阵进入可归档终态

## 6. 全链路改造分层

全链路优化不能混成一锅，需要分层改。

## 6.1 第一层：推进链

核心文件：

- `src/core/process-engine/advance.ts`

职责：

- 调用依赖检查
- 调用 GateEvaluator
- 写 findings
- 更新阶段状态

改造重点：

- 让 warning 不再阻断推进
- 为 warning 增加审计留痕
- 保持硬失败、waiver、warning 语义清晰分离

## 6.2 第二层：依赖检查链

核心文件：

- `src/shared/config-schema.ts`
- `src/core/process-engine/dependency-checker.ts`

职责：

- 决定阶段推进前的脚本和文件前置要求

改造重点：

- 把 dependency 从“统一默认硬阻断”调整为“默认阻断 + strict 增强”
- 明确哪些属于默认最小要求

## 6.3 第三层：Gate 条件链

核心文件：

- `src/core/gate-engine/gate-evaluator.ts`

职责：

- 执行内建 Gate
- 注入 Layer2 / 扩展 Gate
- 匹配 waiver
- 聚合 gate 状态

改造重点：

- 删除重复默认 Gate
- 引入 warning-only 语义
- 只让 blocking 失败参与 waiver 和 FAIL 聚合

## 6.4 第四层：指标链

核心文件：

- `src/core/trace-engine/coverage.ts`
- `src/core/metrics-engine/health-score.ts`
- `src/core/metrics-engine/bottleneck.ts`

职责：

- 计算 coverage
- 计算健康分
- 输出瓶颈分析

改造重点：

- 允许兼容保留旧指标字段
- 但默认消费链不再把它们作为主指标

## 6.5 第五层：输出链

核心文件：

- `src/cli/commands/gate.ts`
- `gate-history.jsonl`
- `findings.md`

职责：

- 对用户展示 gate 结果
- 为后续追溯提供历史记录

改造重点：

- 引入 `[WARN]`
- 历史记录中持久化 warning 语义
- findings 中区分 warning、waiver、hard fail

## 6.6 第六层：GoLive 消费链

核心文件：

- `src/core/gate-engine/golive.ts`

职责：

- 消费最近一次 Gate 结果
- 参与上线前检查

改造重点：

- 顶层状态判定继续保持简洁
- 但 detail 里应能体现 warning 数量

## 6.7 第七层：Profile / 配置化链

核心文件：

- `src/core/process-engine/layer-merger.ts`
- `src/shared/config-schema.ts`

职责：

- 提供默认模式与 strict 模式的切换能力

改造重点：

- 不是只让 Gate profile 化
- 而是让 Gate、dependencies、metrics、Layer2 rules 一起 profile 化

---

## 7. 详细改造清单

## 7.1 GateEvaluator 改造

### 下线默认内建 Gate

删除：

- `G-DESIGN-02`
- `G-IMPL-02`
- `G-VERIFY-02`

### 引入 warning-only 条件

改造对象：

- `G-SPEC-00`
- `G-SPEC-03`
- `G-DESIGN-03`

推荐结构：

```ts
interface GateConditionDef {
  id: string;
  description: string;
  evaluate: (ctx: EvalContext) => {
    pass: boolean;
    detail?: string;
    scopeFrIds?: string[];
    blocking?: boolean;
  };
}
```

以及：

```ts
interface ConditionResult {
  id: string;
  description: string;
  status: 'PASS' | 'WAIVER' | 'FAIL';
  detail?: string;
  scopeFrIds?: string[];
  blocking?: boolean;
}
```

规则：

- `blocking !== false && status === 'FAIL'` 才算硬失败
- `blocking === false && status === 'FAIL'` 只算 warning
- warning 不参与 waiver

### Layer2 gate 处理

短期：

- 保留现有注入逻辑

中期：

- 为 Layer2 gate 增加 profile 过滤能力

---

## 7.2 推进链改造

### `advance()` 增加 warning 审计

建议在 `PASS` 或 `PASS_WITH_WAIVER` 情况下，如果存在 warning 条件，写入：

```text
GATE_WARNING: <conditionId> <detail>
```

这样可以保留提醒，又不会阻断推进。

### 明确 dependency 与 gate 的职责边界

建议在推进链文档与实现中明确：

- dependency 是前置能力要求
- gate 是规范与质量要求

否则用户会继续把所有阻断都误认为“Gate 卡点”。

---

## 7.3 依赖检查链改造

### 复审默认依赖

当前默认依赖里最值得优先复审的是：

- `04_implement` 的 `build`
- `05_verify` 的 `test`

原因：

- 这是最容易在“Gate 变轻”后依然形成高阻力的地方

### 引入 profile

建议新增一个统一 profile，例如：

```yaml
workflow:
  profile: default-simplified
```

或：

```yaml
gate:
  profile: default-simplified
```

默认：

- `default-simplified`

增强：

- `strict`

---

## 7.4 coverage 与指标链改造

### coverage 字段策略

短期不建议立刻删除：

- `C1`
- `C2`
- `C5`
- `C7`

因为这会带来更大兼容风险。

短期建议：

- 保留字段
- 补充注释
- 在默认消费链中降级或隐藏

### health-score 改造

默认 profile 下只使用：

- `C3`
- `C4`
- `C6`
- `C8`
- `C9`

strict profile 不再恢复已删除指标，只对保留指标、warning 策略和依赖策略做增强。

### bottleneck 改造

默认 profile 下只基于默认 blocking 指标输出瓶颈。

不再在默认模式下因为：

- `C1`
- `C5`
- `C7`

生成主要瓶颈信号。

---

## 7.5 CLI / 历史 / 审计链改造

### `gate check`

需要从当前的三态图标：

- `[OK]`
- `[WVR]`
- `[FAIL]`

扩展为：

- `[OK]`
- `[WVR]`
- `[WARN]`
- `[FAIL]`

其中：

- `status='FAIL' && blocking=false` 显示为 `[WARN]`

### `collectFixSteps()`

只应对 blocking 失败项提取修复步骤。

warning-only 条件不应进入 remediation 流程。

### `gate-history.jsonl`

必须持久化：

- `conditions[].blocking`

否则后续历史读取无法复原 warning 语义。

### `findings.md`

建议统一格式：

- `DEPENDENCY_CHECK_FAIL: ...`
- `WAIVER: ...`
- `GATE_WARNING: ...`

---

## 7.6 GoLive 改造

当前 `GL-01` 只认：

- `PASS`
- `PASS_WITH_WAIVER`

这可以保留。

但应增强 detail，示例：

```text
最近 Gate：PASS（阶段 05_verify，2 warnings）
```

这样 GoLive 不需要改变顶层判定，也能保留风险上下文。

---

## 7.7 Profile / 配置化改造

建议最终引入两个 profile：

- `default-simplified`
- `strict`

profile 应同时作用于：

- 内建 Gate
- dependencies
- health-score
- bottleneck
- Layer2 / 扩展 Gate

而不是只让某一层单独切换。

---

## 8. 实施顺序建议

## Phase 1：统一默认阻断模型

完成内容：

- 删除默认 Gate：`C1/C2/C5/C7`
- 引入 warning-only：`C-PRD/C10/C11`
- 扩展 `ConditionResult`
- 改造 Gate 聚合与 waiver 匹配

目标：

- 先让默认链路的正式阻断面收缩下来

## Phase 2：打通输出层与消费层

完成内容：

- `gate check` / `gate history` / `findings`
- `health-score`
- `bottleneck`
- 默认 metrics 说明

目标：

- 让用户真实感知到链路已变轻

## Phase 3：收缩非 Gate 阻断

完成内容：

- 复审 dependency defaults
- 复审 Layer2 command gate
- 明确默认与 strict 的边界

目标：

- 让“默认链路优化”不再只停留在 evaluator 层

## Phase 4：profile 化

完成内容：

- 正式引入 profile
- 用配置控制不同治理层级

目标：

- 建立可持续演进的治理模型

---

## 9. 风险与边界

## 9.1 主要风险

### 风险一：只改 Gate，不改 metrics

后果：

- 流程虽然能过
- 但 health score / bottleneck 仍然报告“旧问题”

### 风险二：只加 `blocking=false`，不改 CLI

后果：

- 系统内部已放行
- 用户界面仍是一堆 `[FAIL]`

### 风险三：只改内建 Gate，不改 dependencies

后果：

- 文档说流程变轻
- 实际推进仍被脚本或文件依赖拦住

### 风险四：只改默认链路，不处理 Layer2

后果：

- 某些平台下仍会被 command gate 阻断
- 用户误以为“默认简化”没有生效

## 9.2 边界说明

本方案的核心是：

- 先重定义默认链路
- 再保留 strict 模式承接高治理团队

它不是在否定治理，而是在重新安排治理强度的默认位置。

---

## 10. 验收标准

如果要判断“全链路优化是否真正完成”，建议至少满足以下标准：

1. 默认推进链中，`C1/C2/C5/C7` 不再构成硬阻断
2. `C-PRD/C10/C11` 失败不会阻断推进
3. `gate check` 可以明确显示 warning
4. `gate-history` 能还原 warning 语义
5. `advance()` 对 warning 保留审计记录
6. 默认 health-score 不再受 `C1/C2/C5/C7` 主导
7. 默认 bottleneck 不再因 `C1/C5/C7` 输出主要告警
8. 默认 dependency policy 已明确
9. Layer2 / 扩展 Gate 的默认行为已定义清楚
10. GoLive 能读取带 warning 的最近 Gate 结果并给出完整上下文

---

## 11. 最终结论

Spec-First 要做的不是“减少几个 Gate 条件”，而是把默认推进链从局部复杂，调整为整体一致。

最终推荐结论如下：

```text
默认 blocking: C3, C4, C6, C8, C9
默认删除: C1, C2, C5, C7
默认 warning: C-PRD, C10, C11
同步收敛: dependencies, layer2 gates, metrics, cli/history, golive
```

只有把这些链路一起改掉，系统才会真正从“卡点很多的治理框架”，转向“默认可落地、严格可升级”的规范驱动研发引擎。

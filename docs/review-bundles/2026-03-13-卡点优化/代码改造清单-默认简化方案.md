# Spec-First 全链路优化：代码改造清单

## 1. 目标

本清单不再只针对 `GateEvaluator` 做减法，而是面向整条默认推进链做改造。

覆盖范围：

- `advance()` 依赖检查链
- 内建 Gate 条件
- Layer2 / 扩展命令 Gate
- coverage / metrics / health-score / bottleneck
- CLI / findings / gate-history
- GoLive 对 gate 结果的消费

目标模型：

```text
默认 blocking: C3, C4, C6, C8, C9
默认下线: C1, C2, C5, C7
默认 warning: C-PRD, C10, C11
```

---

## 2. 实施顺序

### Phase 1: 统一 warning 语义与默认 Gate

先做这一步，因为它决定系统是否真的“放松默认阻断”。

### Phase 2: 同步输出层和指标消费层

确保 CLI、history、health-score、bottleneck 不再放大旧复杂度。

### Phase 3: 收缩非 Gate 阻断链

处理 dependencies、Layer2 command gate 和 profile。

---

## 3. Gate 链改造

## 3.1 `src/core/gate-engine/gate-evaluator.ts`

### A1. 删除默认内建 Gate

删除：

- `G-DESIGN-02`
- `G-IMPL-02`
- `G-VERIFY-02`

原因：

- `C2` 与 `C1` 重复
- `C7` 被 `C8` 覆盖
- `C5` 与 `C4` 重复

### A2. 将质量项改为 warning-only

改造对象：

- `G-SPEC-00`
- `G-SPEC-03`
- `G-DESIGN-03`

建议改造方式：

- 在 `GateConditionDef.evaluate()` 返回结构中增加：

```ts
blocking?: boolean
```

- 在 `ConditionResult` 中同步增加：

```ts
blocking?: boolean
```

行为规则：

- `pass=false && blocking=false` 时，该条件只作为 warning 展示
- 不参与顶层 `FAIL`
- 不参与 waiver 匹配

### A3. 修改 Gate 聚合逻辑

当前聚合逻辑的问题：

- 只认识 `PASS/FAIL/WAIVER`
- 无法区分“失败但不阻断”

需要改造：

- `hasFailure` 只统计 `blocking !== false && status === 'FAIL'`
- `waiver` 只匹配 blocking 条件
- `suggestions` 只为 blocking 失败项生成

### A4. 保留 Layer2 command gate，但纳入 profile 规划

当前代码会额外执行：

- `state.mergedRules.gateConditions[stage]`

短期策略：

- 第一阶段不删除这条链路
- 但文档与实现都要明确：默认简化只先收缩内建 Gate

第二阶段策略：

- 为 Layer2 command gate 引入 profile 过滤
- 至少支持 `default-simplified` 与 `strict`

## 4. 推进链改造

## 4.1 `src/core/process-engine/advance.ts`

当前推进顺序：

1. `checkDependencies(featureId, to, projectRoot)`
2. `evaluateGate(featureId, projectRoot)`
3. 更新状态

### B1. 明确默认模式下的 dependency policy

当前默认依赖是硬阻断，不区分 profile。

需要改造：

- 定义哪些依赖属于默认模式必需
- 定义哪些依赖迁移到 strict

### B2. 补充 findings 里的 warning 记录

当前 `advance()` 只对 `PASS_WITH_WAIVER` 追加 findings。

建议补充：

- 当 gate 存在 warning-only 条件时，记录：
  - `GATE_WARNING: <condition id> <detail>`

目的：

- 让推进链对 warning 有审计留痕

## 5. 依赖检查链改造

## 5.1 `src/shared/config-schema.ts`

当前默认依赖：

- `02_design`: `prd.md/spec.md`
- `03_plan`: `design.md`
- `04_implement`: `test/build`
- `05_verify`: `test`
- `06_wrap_up`: `retro.md`
- `07_release`: release artifacts + `contract:check`

### C1. 需要 profile 化

建议新增：

```ts
gate: {
  profile: 'default-simplified' | 'strict'
}
```

或者：

```ts
workflow: {
  profile: 'default-simplified' | 'strict'
}
```

### C2. 默认模式下优先复审的依赖

建议优先处理：

- `04_implement` 的 `build`
- `05_verify` 的 `test`

原因：

- 这是最直接的默认阻断项
- 也是最容易让“Gate 变轻但推进仍卡住”的来源

## 6. 指标链改造

## 6.1 `src/core/trace-engine/coverage.ts`

短期目标不是立即删除字段，而是让默认消费链不再把旧指标当主指标。

### D1. 保留兼容字段，但标记默认不消费

保留：

- `C1`
- `C2`
- `C5`
- `C7`

但要明确：

- 不再参与默认 Gate
- 不再参与默认健康分
- 不再参与默认瓶颈判定

### D2. 补充注释与输出说明

例如：

- `C2`: duplicate of `C1` in current implementation
- `C5`: duplicate of `C4` until AC-level trace is implemented
- `C7`: weaker than `C8`

---

## 6.2 `src/core/metrics-engine/health-score.ts`

### E1. 调整默认权重

当前权重仍覆盖：

- `C1`
- `C2`
- `C5`
- `C7`

需要改造：

- 默认 profile 下，仅保留 `C3/C4/C6/C8/C9`
- strict profile 下再恢复全量权重

推荐方式：

- 将权重表按 profile 切换
- 不建议只把旧指标权重硬改为 `0` 且不说明 profile

---

## 6.3 `src/core/metrics-engine/bottleneck.ts`

### F1. 调整默认瓶颈规则

当前规则仍依赖：

- `C1`
- `C5`
- `C7`

需要改造：

- 默认 profile 下的 bottleneck 只基于：
  - `C3`
  - `C4`
  - `C6`
  - `C8`
  - `C9`

strict profile 再恢复完整分析。

---

## 7. 类型与结果结构改造

## 7.1 `src/shared/types.ts`

### G1. 扩展 `ConditionResult`

新增：

```ts
blocking?: boolean
```

必要时可进一步新增：

```ts
severity?: 'error' | 'warning'
```

但第一阶段推荐只加 `blocking`。

### G2. 评估是否扩展 `GateResult`

第一阶段建议：

- 不单独新增 `warnings[]`
- 继续用 `conditions[] + blocking`

原因：

- 对历史兼容更好
- CLI 改造更直接

## 8. CLI / 历史 / 审计链改造

## 8.1 `src/cli/commands/gate.ts`

### H1. `gate check` 必须识别 warning-only 条件

当前显示：

- `PASS -> [OK]`
- `WAIVER -> [WVR]`
- 其他 -> `[FAIL]`

需要改造：

- `blocking=false && status='FAIL'` 显示为 `[WARN]`

### H2. 修复 fix steps 的提取条件

当前 `collectFixSteps()` 只对 `status === 'FAIL'` 生效。

建议：

- 只对 blocking 失败项提取 fix steps
- warning-only 条件不进入 remediation 流程

## 8.2 `gate-history.jsonl`

### H3. 持久化 warning 语义

要求：

- history 中必须能恢复 `blocking=false`
- 否则历史回放会把 warning 当失败

## 8.3 `findings.md`

### H4. 为 warning 引入单独记法

建议：

- `GATE_WARNING: ...`
- 与 `WAIVER:`、`DEPENDENCY_CHECK_FAIL:` 区分开

---

## 9. GoLive 链改造

## 9.1 `src/core/gate-engine/golive.ts`

当前 `GL-01` 只接受：

- `PASS`
- `PASS_WITH_WAIVER`

这本身可以保留，但前提是 warning-only 条件不会把顶层 gate 状态误导成失败。

### I1. 增强最近一次 gate 结果说明

建议：

- `detail` 中附带 warning 数量

例如：

```text
最近 Gate：PASS（阶段 05_verify，2 warnings）
```

这样上线检查才保留完整上下文。

---

## 10. Layer2 / Profile 链改造

## 10.1 `src/core/process-engine/layer-merger.ts`

### J1. Gate profile 化

需要补一个能贯穿整条链路的 profile：

- `default-simplified`
- `strict`

应用范围不能只限于 GateEvaluator，还应覆盖：

- dependencies
- metrics weights
- bottleneck rules
- layer2 command gates

推荐策略：

- 第一阶段：只在文档中明确默认行为
- 第二阶段：再将 profile 正式接入合并规则

---

## 11. 测试改造清单

必须覆盖：

1. 默认内建 Gate 下线 `G-DESIGN-02`、`G-IMPL-02`、`G-VERIFY-02`
2. `G-SPEC-00/G-SPEC-03/G-DESIGN-03` 失败时，顶层 gate 仍可通过
3. warning-only 条件不参与 waiver
4. `gate check` 能正确显示 `[WARN]`
5. `gate-history` 能恢复 warning 语义
6. `advance()` 在 warning-only 情况下仍可推进
7. 默认 health-score 不再受 `C1/C2/C5/C7` 影响
8. 默认 bottleneck 不再因 `C1/C5/C7` 触发
9. GoLive 能读取带 warning 的最近 Gate 结果

---

## 12. 最小可交付改造包

如果只做一轮最小变更，建议至少完成以下 8 项：

1. 修改内建 Gate，删除 `G-DESIGN-02`、`G-IMPL-02`、`G-VERIFY-02`
2. 引入 `blocking?: boolean`
3. 将 `G-SPEC-00`、`G-SPEC-03`、`G-DESIGN-03` 改为 warning-only
4. 修改 Gate 聚合与 waiver 匹配
5. 修改 `gate check` / `gate-history` / `findings`
6. 调整默认 health-score
7. 调整默认 bottleneck
8. 明确默认依赖检查策略

如果这 8 项没一起完成，就还不能称为“全链路优化”，只能算“Gate 层减负”。

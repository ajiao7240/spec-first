# 可实施改造清单 v2

更新时间：2026-03-13
适用范围：开发阶段，全链路优化，不考虑向下兼容
口径来源：以当前全链路优化方案文档为准

状态图例：

- `已完成`：当前代码已满足，文档仅做确认
- `部分完成`：当前代码已具备部分能力，但仍存在偏差或重复真源
- `待改造`：当前代码尚未满足，需要本轮改造落地

## 1. 目标

将当前流程收敛为一套单一真理源模型，消除以下三类不一致：

- Gate 通过，但 metrics / viewer 仍显示失败
- CLI、viewer、历史记录分别使用不同的指标集合和阈值
- default-simplified 放松范围过宽，误伤 release 边界

最终目标：

- 默认主治理面只保留 `C3 / C4 / C6 / C8 / C9`
- 明确物理删除 `C1 / C2 / C5 / C7`
- `C-PRD / C10 / C11` 仅作为 warning
- `strict` 在 Gate 侧不恢复已删除项，只把 warning 升级为 blocking
- Gate / metrics / viewer 全部读取同一份阶段化阈值定义

备注：

- 上述为目标态，不等于当前代码已完全满足
- 当前工作树中，Gate 侧已部分接近该语义，但 metrics / health / viewer 仍存在多处独立解释器

## 1.1 审查校准

基于当前代码实现，对本清单做如下校准：

- `已完成` 方案合理性整体成立，方向正确，目标也可达
- `待改造` 当前最需要优先处理的高风险问题仍是 `default-simplified` 误放松 `07_release`
- `已完成` 文档中关于 `metrics` 固定 target、`warning` 语义被硬编码覆盖、测试不足的判断成立
- `已完成` “阶段化 target 需要从零新建”这个表述已修正

更准确的口径应为：

- 当前代码里已经存在“阶段化阈值语义”，主要体现在 `gate-engine/condition-registry.ts`
- 例如：
  - `03_plan` 使用 `C3=100%`、`C8=100%`
  - `04_implement` 使用 `C4>=60%`
  - `05_verify` 使用 `C4>=80%`、`C9=100%`
  - `06_wrap_up` 使用 `C6=100%`
- 现有问题不是“完全没有阶段化 target”，而是这些 target 仍散落在 Gate 解释器中，尚未抽取为可被 metrics / viewer / health 复用的统一真源
- 因此 Phase 1 的正确动作应是“抽取并收敛已有阶段化 target 语义”，而不是凭空重新设计一套全新 target
- 同时，本轮目标已明确为物理删除 `C1 / C2 / C5 / C7`，因此不能再把它们仅视为“隐藏展示项”或“兼容保留字段”

## 3. 单一真理源方案

### 3.1 抽取统一阈值注册表

当前状态：`待改造`

从现有 Gate 条件定义中抽取出“阶段化核心指标阈值”真源模块，建议位置：

- `src/core/metrics-engine/core-metric-thresholds.ts`

建议导出：

- `CORE_METRICS`
- `getStageMetricTargets(stage, profile)`
- `getMetricDisplayDefs(stage, profile)`
- `isCoreMetric(metricKey)`

建议结构：

```ts
export interface MetricTargetDef {
  key: 'C3' | 'C4' | 'C6' | 'C8' | 'C9';
  name: string;
  target: number;
  blocking: boolean;
}

export function getStageMetricTargets(stage: Stage): MetricTargetDef[] {
  switch (stage) {
    case Stage.PLAN:
      return [
        { key: 'C3', name: '任务覆盖率', target: 1.0, blocking: true },
        { key: 'C8', name: '任务合规率', target: 1.0, blocking: true },
      ];
    case Stage.IMPLEMENT:
      return [
        { key: 'C4', name: '测试覆盖率 (FR)', target: 0.6, blocking: true },
      ];
    case Stage.VERIFY:
      return [
        { key: 'C4', name: '测试覆盖率 (FR)', target: 0.8, blocking: true },
        { key: 'C9', name: 'TC 合规率', target: 1.0, blocking: true },
      ];
    case Stage.WRAP_UP:
      return [
        { key: 'C6', name: '实现覆盖率', target: 1.0, blocking: true },
      ];
    default:
      return [];
  }
}
```

说明：

- 这里定义“当前阶段应解释哪些核心指标”
- 优先复用 `condition-registry.ts` 中已落地的阶段阈值语义，而不是重新发明一套数值
- 不再让 CLI 和 viewer 自己硬编码 target
- 不再让 health / bottleneck 模块各自维护第二套指标集合语义
- `strict` 如无额外指标恢复需求，可直接共用同一套核心 target；如后续保留 strict 专属展示，也必须从同一真源派生
- 该真源仅覆盖保留指标 `C3 / C4 / C6 / C8 / C9`，不再为 `C1 / C2 / C5 / C7` 提供运行时 target 定义

## 6. 推荐实施顺序

推荐顺序：

1. 先修 dependency 的 release 边界
2. 再抽取阶段化指标真源
3. 再统一 metrics CLI
4. 再去掉 gate-evaluator 的 warning 硬编码覆盖
5. 最后改 stage-viewer

原因：

- `07_release` 边界问题是当前行为级 Bug，风险最高，应先止血
- 阶段化 target 语义已在 Gate 中存在，先抽真源再接 metrics/viewer 更稳妥
- warning 语义应在 Gate 真源稳定后再收敛，避免改一层又被另一层覆盖
- viewer 是消费端且改动面最大，应该最后接入

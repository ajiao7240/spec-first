# Stage Viewer 展示逻辑优化方案

> 日期：2026-03-13
> 范围：`scripts/stage-viewer/index.html` / `scripts/stage-viewer/app.js` / `scripts/stage-viewer/health-utils.js` / `scripts/stage-viewer/server.js`
> 依据：`全链路优化版本-完整方案.md`、`开发任务清单.md`

---

## 1. 目标

本方案不是单纯调整页面样式，而是让 Stage Viewer 的“各种率”展示逻辑与最新全链路治理模型一致。

最新方案已经明确：

- 默认 blocking：`C3 / C4 / C6 / C8 / C9`
- 默认删除：`C1 / C2 / C5 / C7`
- 默认 warning：`C-PRD / C10 / C11`

因此页面不能再继续把 `C1-C9` 按同一层级平铺展示，也不能继续让健康分和 coverage bars 默认放大已删除指标。

页面要完成的事情应变成：

1. 让用户一眼看到“当前默认链路真正关心什么”
2. 让 warning 与 blocking 在视觉上彻底分层
3. 让已删除指标退出默认主视图，只保留为折叠参考
4. 让阶段流、健康分、阈值、历史记录表达同一套治理语义

---

## 2. 当前页面的主要问题

基于当前实现，问题集中在 4 类。

### 2.1 健康看板仍按旧指标体系平铺

`app.js` 当前通过 `/api/feature/:id/metrics` 读取 `metricDefs`，再把所有指标直接渲染为 coverage bars。

结果：

- `C1/C2/C5/C7` 仍和 `C3/C4/C6/C8/C9` 同权出现
- 用户无法区分“默认硬阻断指标”和“历史遗留指标”
- 页面语义仍停留在旧的 `C1-C9` 平铺时代

### 2.2 页面没有表达 warning 语义

当前页面只会展示数值是否达标，但不会告诉用户：

- 这个指标是否会阻断推进
- 这个指标是否只是 warning
- 这个阶段失败后是“卡住”还是“仅提醒”

这与最新开发任务中 `blocking?: false` 的 warning 语义不一致。

### 2.3 页面没有阶段感知

当前 coverage bars 一次性展示全部率，缺少阶段语义。

例如：

- 在 `02_design` 阶段展示 `C6/C9` 没有实际意义
- 在 `03_plan` 阶段应该突出 `C3/C8`
- 在 `05_verify` 阶段应该突出 `C4/C9`

如果不按阶段重排，页面会继续显得“指标很多、重点不清”。

### 2.4 Stage Viewer 自己仍维护了一套旧的度量定义

`health-utils.js` 和 `server.js` 里仍内置旧版 `METRIC_DEFS / WEIGHTS`：

- 仍包含 `C1/C2/C5/C7`
- 健康分仍按旧权重计算
- API 返回仍是“旧度量模型”

这意味着即使 CLI 与核心链路后续收敛，Stage Viewer 也会继续输出旧认知。

---

## 3. 方案对比

### 方案 A：最小改动过滤法

做法：

- 仅在前端隐藏 `C1/C2/C5/C7`
- 保留现有健康分圆环和 coverage bars 结构
- warning 只加一个小标签

优点：

- 改动最小
- 前端工作量低

缺点：

- 健康分来源仍是旧模型
- 阶段感知仍缺失
- 页面本质仍是“旧模型减法版”

### 方案 B：语义分层展示法

做法：

- 把指标拆成 `默认阻断 / 质量提醒 / 历史参考` 三层
- 健康分只基于默认 blocking 指标
- 结合阶段流，动态突出当前阶段相关指标
- 已删除指标折叠到“历史参考”区域

优点：

- 与最新全链路方案完全一致
- 用户认知成本最低
- 能承接后续 profile / warning / CLI 语义升级

缺点：

- 需要调整 API 返回结构
- 页面结构和渲染逻辑需要一起改

### 方案 C：追溯矩阵中心化

做法：

- 页面主视图不再突出率，而是突出 FR / TASK / TC 的追溯链条
- 各种率退到辅助面板

优点：

- 更贴近 spec-first 本质

缺点：

- 改动太大
- 会影响现有使用习惯
- 不适合作为这一轮 viewer 优化的主方案

### 推荐结论

推荐采用 **方案 B：语义分层展示法**。

原因：

- 它和最新开发方案最一致
- 它能解决“率太多、重点不清、warning 不可见、删除指标还在主视图”的核心问题
- 它不需要推翻页面，只需要重构健康看板和阶段联动逻辑

## 4. 推荐展示模型

页面中的“各种率”统一按 3 层表达。

### 4.1 第一层：默认阻断率

这是页面主视图，默认展开，只展示：

- `C3` 任务覆盖率
- `C4` 测试覆盖率
- `C6` 实现覆盖率
- `C8` 任务合规率
- `C9` TC 合规率

每个指标卡片展示：

- 当前值
- 目标值
- 当前状态：`PASS / RISK / FAIL`
- 所属阶段
- 一句话解释“为什么重要”
- 是否纳入健康分
- 是否纳入默认 Gate

这 5 项应替代当前的统一 coverage bars，成为主看板核心。

### 4.2 第二层：质量提醒

单独展示：

- `C-PRD`
- `C10`
- `C11`

它们不应继续展示成“与 C3/C4/C6/C8/C9 同款的覆盖率条”，而应展示成：

- warning 卡片
- 审核提醒条
- 质量说明项

每项重点显示：

- 当前状态：`OK / WARN`
- 失败是否阻断：`否`
- 对应阶段
- 来源文档或检查项

这里的视觉语言应与 blocking 明显不同，例如使用浅黄或蓝灰信息卡，而不是红色失败条。

### 4.3 第三层：历史参考指标

折叠展示：

- `C1`
- `C2`
- `C5`
- `C7`

默认折叠，标题建议为：

- `历史参考指标`
- 或 `已退出默认治理面的指标`

每项要明确标注：

- `已删除`
- `不纳入默认 Gate`
- `不纳入默认健康分`
- `仅用于历史对照`

这样既保留兼容性，又不让用户误以为这些指标仍然是默认治理重点。

## 5. 页面结构优化建议

在不推翻当前布局的前提下，建议把 `detail` 区域改成以下结构。

### 5.1 顶部：总览卡

保留当前 summary，但增加两项：

- `当前治理 profile`
- `默认治理面说明`

建议展示为：

- Profile：`default-simplified`
- 默认重点：`C3 / C4 / C6 / C8 / C9`

这能先把“页面按什么逻辑看”讲清楚。

### 5.2 健康看板：从“一个圆环 + 一排条”改为“两层看板”

建议改成：

1. 左侧：健康分总览
2. 右侧：默认阻断率卡片组
3. 下方：warning 区

健康分卡上补充两行说明：

- `仅基于默认 blocking 指标计算`
- `不含 C1/C2/C5/C7`

否则用户看到分数，仍不知道分数到底受哪些指标影响。

### 5.3 阶段流转图：增加 warning / blocking 语义

当前阶段流的 Gate 徽章只展示：

- `PASS`
- `PASS_WITH_WAIVER`
- `FAIL`

建议改为：

- 通过：绿色
- 通过但有 warning：蓝色或黄色描边
- 失败：红色

徽章文案建议从：

- `passCount / totalCount`

改为：

- `3/3`
- `2/2 + 1 warn`
- `1/3 fail`

这样阶段流也能体现 warning 语义，而不只是顶层状态。

### 5.4 阈值表：从“原始阈值表”改为“治理规则表”

当前阈值表只是把 `mergedRules.thresholds` 平铺出来，用户不容易看懂。

建议改成 4 列：

- 指标/规则
- 当前值
- 目标/要求
- 治理层级

治理层级使用：

- `blocking`
- `warning`
- `archived`
- `dependency`

这样阈值表才能和整页口径一致。

## 6. 各种率的具体展示逻辑

### 6.1 默认阻断率展示规则

#### `C3`

- 阶段：主属于 `03_plan`
- 默认展示：从 `03_plan` 开始显示
- 展示语义：需求是否已经落成任务
- 失败文案：突出“未落成 TASK 的 FR 数量”

#### `C4`

- 阶段：主属于 `04_implement` 与 `05_verify`
- 默认展示：从 `04_implement` 开始显示
- 展示语义：需求是否获得测试覆盖
- 失败文案：突出“未被测试覆盖的 FR”

#### `C6`

- 阶段：主属于 `06_wrap_up`
- 默认展示：从 `04_implement` 开始可见，在 `06_wrap_up` 提升权重
- 展示语义：任务是否真实闭合

#### `C8`

- 阶段：主属于 `03_plan`
- 默认展示：从 `03_plan` 开始显示
- 展示语义：是否存在孤儿 TASK

#### `C9`

- 阶段：主属于 `05_verify`
- 默认展示：从 `05_verify` 开始高亮
- 展示语义：是否存在孤儿测试

### 6.2 warning 项展示规则

#### `C-PRD`

- 展示类型：文档质量提醒
- 不进入 coverage bars
- 推荐展示：`PRD 完整度`

#### `C10`

- 展示类型：规格质量提醒
- 推荐展示：`Spec 质量检查`
- 需要显示来源：`checklists/spec-review.md` 或显式分数来源

#### `C11`

- 展示类型：架构/Constitution 合规提醒
- 推荐展示：`架构约束提醒`
- 不与 C3/C4 这类 coverage 指标混排

### 6.3 已删除指标展示规则

`C1/C2/C5/C7` 统一进入折叠区。

展示时要附加固定说明：

- `该指标已退出默认治理面`
- `当前仅作历史参考`
- `不参与默认 Gate`
- `不参与默认健康分`

折叠区默认不占主视线，不应在页面首屏出现红色或黄色失败态。

## 7. 数据接口优化建议

前端如果要真正做到上述展示，不能继续只依赖当前的：

- `coverage`
- `health`
- `metricDefs`

建议 `/api/feature/:id/metrics` 增加一层“展示模型”。

### 7.1 推荐返回结构

```json
{
  "featureId": "feature-x",
  "profile": "default-simplified",
  "health": {
    "H1": 82,
    "grade": "B",
    "scope": "default-blocking"
  },
  "metrics": [
    {
      "key": "C3",
      "name": "任务覆盖率",
      "value": 1,
      "target": 1,
      "displayRole": "blocking",
      "usedInHealth": true,
      "usedInGate": true,
      "stageOwner": "03_plan",
      "visibleFromStage": "03_plan",
      "summary": "需求是否已落成任务"
    }
  ],
  "groups": {
    "blocking": ["C3", "C4", "C6", "C8", "C9"],
    "warning": ["C-PRD", "C10", "C11"],
    "archived": ["C1", "C2", "C5", "C7"]
  }
}
```

### 7.2 为什么需要这一层

原因不是为了“多包一层 JSON”，而是为了让前端不再硬编码旧指标知识。

目前前端硬编码问题包括：

- `health-utils.js` 自己定义旧版 `METRIC_DEFS`
- `health-utils.js` 自己定义旧版 `WEIGHTS`
- `app.js` 直接按 `metricDefs` 全量平铺

优化后应让 viewer 消费“后端整理后的语义模型”，而不是自己推断。

---

## 8. 前端交互逻辑建议

### 8.1 引入治理视图切换

建议在健康看板顶部增加一个轻量切换：

- `默认治理`
- `全部指标`

默认进入 `默认治理`。

含义：

- `默认治理`：只看 blocking + warning
- `全部指标`：额外展开历史参考指标

这样既兼容老用户，也不会让主视图继续被删除指标污染。

### 8.2 阶段联动高亮

点击阶段流时，健康看板中的指标卡也要联动：

- 若当前点击 `03_plan`，高亮 `C3 / C8`
- 若当前点击 `04_implement`，高亮 `C4`
- 若当前点击 `05_verify`，高亮 `C4 / C9`
- 若当前点击 `06_wrap_up`，高亮 `C6`

其它指标保持低强调或折叠。

### 8.3 文案从“率”改为“意义”

不要只显示：

- `C4 80%`

建议同时显示解释：

- `C4 80% · 需求测试覆盖`
- `仍有 2 个 FR 未被 TC 覆盖`

页面不是面向实现者读原始指标，而是帮助用户快速理解“为什么现在卡在这里”。

## 9. 推荐实施顺序

### Phase A：先改展示语义，不动大布局

完成内容：

- 主视图只展示 `C3/C4/C6/C8/C9`
- 新增 warning 区
- 新增 archived 折叠区
- 健康分补充“计算范围说明”

目标：

- 先把错误认知纠正过来

### Phase B：改接口模型

完成内容：

- `/metrics` API 返回 `profile + metrics + groups`
- `health-utils.js` 不再维护旧版固定权重定义
- viewer 前端不再依赖静态 `METRIC_DEFS`

目标：

- 让展示模型与后端治理模型彻底对齐

### Phase C：做阶段联动增强

完成内容：

- 阶段点击联动高亮指标
- 阶段流显示 warning 数
- 阈值表改为治理规则表

目标：

- 让 viewer 从“静态概览页”升级为“治理驾驶舱”

---

## 10. 最终建议

`scripts/stage-viewer/index.html` 的优化重点，不应放在“把条形图画得更漂亮”，而应放在“让页面表达正确的治理语义”。

本轮最重要的 5 个改动点是：

1. 主视图只保留 `C3/C4/C6/C8/C9`
2. `C-PRD/C10/C11` 从 coverage bars 中拆出，单列 warning 区
3. `C1/C2/C5/C7` 进入折叠的历史参考区
4. 健康分明确声明“只基于默认 blocking 指标”
5. 阶段流、阈值表、历史记录统一显示 blocking / warning / archived 语义

如果只做前端过滤，不改展示模型，页面仍会停留在旧逻辑。

如果按本方案推进，Stage Viewer 才会与最新全链路优化方案真正一致。

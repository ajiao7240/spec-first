## 📋 执行摘要

当前 C1-C11 共 11 个指标过于复杂,卡点严格,影响研发效率。基于软件工程最佳实践和行业标准分析,**建议保留 4 个核心卡点 + 1 个可选指标,移除/降级 6 个指标**。

### 核心结论

**保留 4 个硬卡点**:
- C8: Task Compliance = 100% (防止孤儿任务)
- C7: PR Compliance = 100% (防止无法追溯的代码)
- C9: TC Compliance = 100% (防止孤儿测试)
- C6: Implementation Coverage = 100% (确保交付完整性)

**保留 1 个软卡点**:
- C4: Test Coverage ≥ 60% (降低阈值,支持豁免)

**移除/降级 6 个指标**:
- C1/C2: Design Coverage → 删除或改为警告
- C3: Task Coverage → 改为警告(≥60%)
- C5: AC Coverage → 删除(与C4重复)
- C10/C11/C-PRD → 改为 Code Review Checklist

## 🎯 第一部分: 问题诊断

### 1.1 当前体系的核心问题

#### 问题 1: 指标冗余
- **C1/C2/C3**: 都是"覆盖率",在不同阶段重复检查
- **C4/C5**: 都是"测试覆盖率",只是粒度不同
- **C10/C11/C-PRD**: 都是"质量评审",但用自动化指标衡量主观质量

#### 问题 2: 阈值过严
- **C1/C2/C3 = 100%**: 小需求可能不需要独立设计文档/任务拆解
- **C5 ≥ 90%**: AC 级别测试覆盖率过高,实际价值有限

#### 问题 3: 自动化误用
- **C10**: 规格质量评审应该是人工 Code Review,不是自动化打分
- **C11**: 架构合规应该是架构评审会议,不是自动化检查
- **C-PRD**: 章节完整性 ≠ 内容质量

#### 问题 4: 认知负担高
- 11 个指标 → 团队需要记住太多规则
- 严格阈值 → 团队花时间"刷指标"而非解决真问题
- 豁免流程 → 创建 RFC + Exception → 流程开销大

### 1.2 影响分析

**对团队的影响**:
- 开发效率下降: 等待指标达标,延迟交付
- 形式主义: 为了通过检查而"刷指标"
- 创新受限: 不敢尝试新方法(可能不符合指标)

**对质量的影响**:
- 表面质量: 指标达标但实际问题未解决
- 隐藏问题: 团队学会"绕过"而非"解决"
- 技术债累积: 豁免机制被滥用

## 🔬 第二部分: 理论基础

### 2.1 软件工程最佳实践

#### Google 工程实践
- **强制**: Code Review + 自动化测试
- **不强制**: 覆盖率阈值(团队自主决定)
- **原则**: "Trust but Verify"

#### Microsoft SDL (Security Development Lifecycle)
- **强制**: 威胁建模 + 安全扫描
- **不强制**: 文档完整性(使用 Definition of Done)
- **原则**: "Risk-Based Approach"

#### Netflix 文化
- **强制**: 追溯性(每个变更必须有 Ticket)
- **不强制**: 流程规范(Freedom and Responsibility)
- **原则**: "Context, not Control"

### 2.2 行业标准对标

#### CMMI Level 3-5
- **核心**: Traceability(追溯性) > Coverage(覆盖率)
- **实践**: Peer Review > Automated Metrics
- **原则**: Risk-Based Testing(根据风险分配资源)

#### ISO 9001:2015
- **核心**: Process Approach(关注流程而非指标)
- **实践**: Risk-Based Thinking(基于风险而非合规性)
- **原则**: Evidence-Based Decision Making(证据 ≠ 指标)

#### 主流研发工具对比

| 工具 | 强制追溯性 | 强制覆盖率 | 质量保障方式 |
|------|-----------|-----------|-------------|
| Jira | ✅ Issue→Epic | ❌ | Code Review + CI/CD |
| Azure DevOps | ✅ Work Item 关联 | ⚠️ 可配置 | Pipeline + Review |
| GitLab | ✅ MR→Issue | ❌ | Pipeline + Review |
| **Spec-First(当前)** | ✅ C7/C8/C9 | ✅ C1-C6 | 自动化指标 |
| **Spec-First(建议)** | ✅ C7/C8/C9 | ⚠️ C4(60%) | Review + 指标 |

**结论**: 行业标准都强制"追溯性",但不强制"覆盖率阈值"。

### 2.3 质量门禁的本质

#### 风险控制视角

**高风险(必须防御)**:
- 孤儿数据(orphan task/test) → 导致无法追溯的技术债
- 未完成交付 → 合同风险、客户投诉

**中风险(可接受偏差)**:
- 测试覆盖率不足 → 可以后补,不阻塞交付
- 任务覆盖率不足 → 可能是合理的范围缩减

**低风险(主观判断)**:
- 设计覆盖率 → 小需求可以不写设计文档
- 文档质量 → 应该由人工评审,不是自动化指标

#### 成本-收益分析

| 指标 | 实现成本 | 维护成本 | 流程成本 | 收益 | ROI |
|------|---------|---------|---------|------|-----|
| C8/C7/C9 | 低 | 低 | 低 | 高(防止严重问题) | ⭐⭐⭐⭐⭐ |
| C6 | 低 | 低 | 低 | 高(确保交付) | ⭐⭐⭐⭐⭐ |
| C4 | 中 | 中 | 中 | 中(减少bug) | ⭐⭐⭐ |
| C1/C2/C3 | 中 | 中 | 高 | 低(可替代) | ⭐⭐ |
| C10/C11/C-PRD | 高 | 高 | 高 | 低(易被刷分) | ⭐ |

**结论**: 只有 C6/C7/C8/C9 是高 ROI 的卡点。

## 💡 第三部分: 简化方案

### 3.1 核心卡点(4个)

#### C8: Task Compliance = 100%
- **阶段**: 03_plan
- **定义**: TASK 必须有上游 FR/NFR/DS
- **理由**: 防止孤儿任务,确保每个任务都有业务价值来源
- **豁免**: ❌ 不支持
- **行业对标**: Jira "Issue 必须有 Epic"

#### C7: PR Compliance = 100%
- **阶段**: 04_implement
- **定义**: TASK 必须有上游关联
- **理由**: 防止代码变更无法追溯到需求
- **豁免**: ❌ 不支持
- **行业对标**: GitLab "MR 必须关联 Issue"

#### C9: TC Compliance = 100%
- **阶段**: 05_verify
- **定义**: TC 必须有上游 FR
- **理由**: 防止孤儿测试用例,确保测试有明确目标
- **豁免**: ❌ 不支持
- **行业对标**: Azure DevOps "Test Case 关联 Requirement"

#### C6: Implementation Coverage = 100%
- **阶段**: 06_wrap_up
- **定义**: 所有 TASK 必须完成
- **理由**: 归档前必须完成所有承诺的任务
- **豁免**: ❌ 不支持
- **行业对标**: Definition of Done

### 3.2 可选指标(1个)

#### C4: Test Coverage ≥ 60%
- **阶段**: 04_implement
- **定义**: FR 中有 TC 映射的比例
- **理由**: 确保核心功能有测试保护,但允许部分 FR 延后测试
- **豁免**: ✅ 支持(通过 known-exceptions.md)
- **调整**: 80% → 60%(更符合行业实践)
- **行业对标**: Google 不强制阈值,团队自主决定

### 3.3 移除/降级指标(6个)

#### C1/C2: Design Coverage
- **当前**: 100% 硬卡点
- **建议**: 删除或改为警告(≥60%)
- **理由**: 小需求可能不需要独立设计文档,可以在 spec.md 中内联设计

#### C3: Task Coverage
- **当前**: 100% 硬卡点
- **建议**: 改为警告(≥60%)
- **理由**: 部分 FR 可能是文档类/配置类,不需要拆任务

#### C5: AC Coverage
- **当前**: 90%(M/L) / 60%(S) 硬卡点
- **建议**: 删除
- **理由**: 与 C4 重复,AC 级别细化价值有限

#### C10: Spec Quality Score
- **当前**: ≥80% 硬卡点
- **建议**: 改为 Code Review Checklist
- **理由**: 质量评审是主观的,不应该用自动化打分

#### C11: Constitution Compliance
- **当前**: PASS 硬卡点
- **建议**: 改为架构评审会议
- **理由**: 架构合规应该由架构师人工检查,自动化检查容易误报

#### C-PRD: PRD Completeness
- **当前**: ≥85% 硬卡点
- **建议**: 改为 Code Review Checklist
- **理由**: 章节完整性 ≠ 内容质量,小需求可能不需要完整 PRD

### 4.2 Phase 2: 观察期(Month 1-3)

#### 目标: 收集数据,验证假设

**监控指标**:
1. 流程效率
   - Stage advance 平均耗时
   - 豁免 RFC 创建数量
   - 团队满意度调查

2. 质量指标
   - 生产环境 bug 率
   - 技术债累积速度
   - Code Review 发现问题数

3. 追溯性指标
   - 孤儿 TASK 数量(应该 = 0)
   - 孤儿 TC 数量(应该 = 0)
   - 无法追溯的代码变更数量(应该 = 0)

**决策标准**:
- 如果 bug 率上升 > 20% → 恢复部分指标
- 如果流程效率提升 > 30% → 继续简化
- 如果追溯性指标 > 0 → 加强 C7/C8/C9 检查

### 4.3 Phase 3: 优化调整(Month 4-6)

#### 目标: 根据数据优化方案

**场景 1: 质量下降**
- 恢复 C4 阈值到 80%
- 增加 Code Review 频率
- 加强测试培训

**场景 2: 质量稳定**
- 继续简化,考虑删除 C1/C2/C3
- 优化 Code Review Checklist
- 推广最佳实践

**场景 3: 追溯性问题**
- 加强 C7/C8/C9 的错误提示
- 提供自动修复工具
- 增加培训材料

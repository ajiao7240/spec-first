## 🔄 具体变化

### 删除的卡点 (4个)

#### C1: Design Coverage
- **状态**: ❌ 删除
- **原因**: 与 C2 重复
- **影响**: 02_design 阶段更轻量

#### C2: API Coverage
- **状态**: ❌ 删除
- **原因**: 实现重复，未真正计算 API
- **影响**: 02_design 阶段更轻量

#### C5: AC Coverage
- **状态**: ❌ 删除
- **原因**: 与 C4 重复
- **影响**: 05_verify 阶段减少一个检查

#### C7: PR Compliance
- **状态**: ❌ 删除
- **原因**: C8 已覆盖
- **影响**: 04_implement 阶段减少一个检查

### 降级的卡点 (3个)

#### C-PRD: PRD Completeness
- **状态**: 🔴 硬卡点 → ⚠️ 警告
- **原因**: 章节完整性 ≠ 内容质量
- **影响**: 01_specify 不再因格式问题阻塞

#### C10: Spec Quality Score
- **状态**: 🔴 硬卡点 → ⚠️ 警告
- **原因**: 主观评分不适合硬卡
- **影响**: 01_specify 不再因评分阻塞

#### C11: Constitution Compliance
- **状态**: 🔴 硬卡点 → ⚠️ 警告
- **原因**: 架构合规应人工评审
- **影响**: 02_design 不再因架构检查阻塞

### 调整的卡点 (1个)

#### C4: Test Coverage
- **改造前**: 04阶段 80%, 05阶段 100%
- **改造后**: 04阶段 60%, 05阶段 80%
- **原因**: 渐进式提升，平衡灵活性与质量
- **影响**: 开发阶段更灵活，验收阶段仍有保障

## ⚠️ 风险与缓解

### 风险 1: 测试覆盖率下降

**风险**: C4 从 80% 降到 60%
**概率**: 中
**影响**: 可能增加 bug
**缓解**:
- 05_verify 仍要求 80%
- 加强 Code Review
- 监控 bug 率

### 风险 2: 文档质量下降

**风险**: C-PRD/C10 降级为警告
**概率**: 中
**影响**: 文档可能不完整
**缓解**:
- Code Review Checklist
- 定期质量回顾
- 团队培训

### 风险 3: 架构偏离

**风险**: C11 降级为警告
**概率**: 低
**影响**: 可能违反架构原则
**缓解**:
- 架构评审会议
- 重要项目仍需 C11
- 架构师参与 Review

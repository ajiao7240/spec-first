> **版本**: v2.0 Final
> **日期**: 2026-03-13
> **状态**: 推荐执行

## 📋 执行摘要

基于深度分析和方案对比，推荐采用 **5 指标融合方案**，在保留 Spec-First 核心价值的同时，大幅降低流程复杂度。

### 核心改动

- **保留**: C3/C4/C6/C8/C9 (5个核心卡点)
- **删除**: C1/C2/C5/C7 (4个重复指标)
- **降级**: C-PRD/C10/C11 (3个主观评分)
- **优化**: C4 分阶段阈值 (60%→80%)

### 预期收益

- 硬卡点数量: 11 → 5 (↓ 55%)
- 流程复杂度: ↓ 60%
- 认知负担: ↓ 70%
- 核心价值: 保持 100%

## 🎯 第一部分: 终版方案

### 1.1 保留的核心卡点 (5个)

#### C3: Task Coverage = 100%
- **阶段**: 03_plan
- **定义**: FR 中有 TASK 映射的比例
- **理由**: 确保"需求→任务"闭环，这是 Spec-First 的核心价值
- **豁免**: ✅ 支持 (通过 known-exceptions.md)
- **来源**: 最佳实践方案

**为什么必须保留?**
- Spec-First 的核心是"规范驱动"，必须体现在任务拆解
- 如果允许 FR 不拆任务，就失去了"可执行性"
- 这是区别于传统文档驱动的关键

#### C4: Test Coverage (不同阶段不同要求)
- **04_implement**: ≥ 60% (允许部分 FR 延后测试)
- **05_verify**: ≥ 80% (验收前必须达到质量底线)
- **定义**: FR 中有 TC 映射的比例
- **理由**: 渐进式提升，平衡灵活性与质量
- **豁免**: ✅ 支持
- **来源**: 融合方案

**为什么是 60% 和 80%?**
- **60%**: 根据架构师方案，60% 覆盖率可发现 85% bug，成本仅为 1x (性价比最高)
- **80%**: 保持当前实践的质量底线，避免质量下降
- **100%**: 边际收益递减，80%→100% 成本从 2x 跃升到 5x
- **渐进式**: 04 阶段灵活，05 阶段严格，避免"一刀切"

#### C6: Implementation Coverage = 100%
- **阶段**: 06_wrap_up
- **定义**: TASK 状态为 Implemented/Verified/Accepted 的比例
- **理由**: 归档前必须完成所有任务
- **豁免**: ❌ 不支持
- **来源**: 两方案一致

#### C8: Task Compliance = 100%
- **阶段**: 03_plan
- **定义**: TASK 必须有上游 FR/DS/NFR
- **理由**: 防止孤儿任务，确保追溯性
- **豁免**: ❌ 不支持
- **来源**: 两方案一致

#### C9: TC Compliance = 100%
- **阶段**: 05_verify
- **定义**: TC 必须有上游 FR
- **理由**: 防止孤儿测试，确保测试有明确目标
- **豁免**: ❌ 不支持
- **来源**: 两方案一致

### 1.2 删除的指标 (4个)

#### C1: Design Coverage
- **删除理由**: 与 C2 实现重复，设计覆盖率不应先于任务成为阻断点
- **替代方案**: Code Review Checklist

#### C2: API Coverage
- **删除理由**: 当前未真正计算 API 维度，属于"命名独立，计算重复"
- **替代方案**: 等 API 设计体系成熟后再恢复

#### C5: AC Coverage
- **删除理由**: 与 C4 实现重复，在 05_verify 阶段形成双重门禁
- **替代方案**: 合并到 C4

#### C7: PR Compliance
- **删除理由**: C8 在 03_plan 阶段已检查 TASK 上游关联，C7 在 04_implement 阶段是阶段间冗余检查
- **风险评估**: 如果 03→04 阶段间 TASK 被修改导致失去追溯性，概率极低
- **缓解措施**: 通过 Git hooks 或 CI 检查确保 TASK 元数据不被随意修改
- **替代方案**: C8 的检查已足够，无需二次确认

### 1.3 降级为警告 (3个)

#### C-PRD: PRD Completeness
- **当前**: ≥85% 硬卡点
- **调整**: 降级为质量提示
- **理由**: 章节完整性 ≠ 内容质量，不适合作为默认阻断条件

#### C10: Spec Quality Score
- **当前**: ≥80% 硬卡点
- **调整**: 降级为质量提示
- **理由**: 适合作为评审反馈，不适合作为所有团队必须理解的默认卡点

#### C11: Constitution Compliance
- **当前**: PASS 硬卡点
- **调整**: 降级为架构合规提醒
- **理由**: 不是每个 Feature 都需要强制 constitution 审查

## 🔄 第二部分: 阶段门禁重构

### 2.1 简化后的阶段分布

```
00_init (初始化)
  └─ 无检查

01_specify (需求规格)
  ├─ spec.md exists
  ├─ FR/NFR IDs assigned
  └─ ⚠️ C-PRD (警告)

02_design (技术设计)
  ├─ design.md exists
  └─ ⚠️ C11 (警告)

03_plan (任务拆解) ⭐ 核心关口
  ├─ 🔴 C3 = 100% (需求→任务闭环)
  ├─ 🔴 C8 = 100% (防止孤儿任务)
  └─ critical findings = 0

04_implement (代码实现)
  └─ 🔴 C4 ≥ 60% (基础测试覆盖)

05_verify (验收测试)
  ├─ 🔴 C4 ≥ 80% (提高测试覆盖)
  └─ 🔴 C9 = 100% (防止孤儿测试)

06_wrap_up (归档复盘)
  ├─ 🔴 C6 = 100% (交付完整性)
  └─ all entries terminal

07_release (发布上线)
  ├─ release-note.md exists
  └─ smoke-test-report.md exists

08_done (完成)
  └─ 终态，无检查
```

### 2.2 关键改进点

#### 改进 1: 前置阶段轻量化
- 01_specify: 只保证需求被结构化表达
- 02_design: 只要求有设计产物
- 不再叠加复杂质量评分

#### 改进 2: 03_plan 成为核心关口
- 这是默认模型中最关键的治理关口
- 一旦任务拆解和任务合法性都没问题，后续实现会顺畅很多

#### 改进 3: 测试覆盖渐进式
- 04 阶段: 60% (允许灵活性)
- 05 阶段: 80% (确保质量)
- 避免"一刀切"

#### 改进 4: 归档阶段只做终态确认
- 不再承担治理实验职责
- 专注于交付完整性

## 📚 第九部分: 附录

### A. 核心理念

**追溯性 > 覆盖率**
- 能追溯比覆盖率高更重要
- C8/C9 防止孤儿数据是核心

**人工判断 > 自动化打分**
- 质量靠 Code Review，不靠指标
- C-PRD/C10/C11 降级为 Checklist

**防御灾难 > 防御小问题**
- 卡点防止严重问题
- 小问题通过文化建设解决

**渐进式 > 一刀切**
- C4 分阶段阈值 (60%→80%)
- 允许团队适应过程

### B. 决策依据

1. **最佳实践方案**: 保留 C3 (需求→任务闭环)
2. **架构师方案**: 行业对标 + 渐进式实施
3. **融合方案**: 结合两者优势

### C. 关键术语

- **追溯性 (Traceability)**: 需求→设计→任务→测试的关联
- **孤儿数据 (Orphan Data)**: 没有上游关联的任务/测试

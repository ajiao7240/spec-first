# Spec-First 卡点优化：全链路优化决策提案

## 1. 目标

本提案不再把优化范围局限在 `GateEvaluator` 的内建条件，而是直接面向真实推进链路做收敛。

目标是把 Spec-First 的默认工作流从“多处叠加阻断、理解成本高、治理信号彼此重复”，调整为“一条可解释、可执行、可维护的默认链路”。

本提案覆盖的链路范围包括：

- `stage advance` 前置依赖检查
- GateEngine 内建条件
- Layer2 / 扩展命令 Gate
- coverage / health-score / bottleneck 等指标消费层
- CLI / findings / gate-history 等输出层
- GoLive 对 gate 结果的下游消费

---

## 2. 当前链路为什么会显得“卡得太严”

当前问题不只是指标多，而是多层阻断叠加后形成了真实的链路复杂度。

### 2.1 Gate 层有重复卡点

当前实现中存在多组“语义不同、实现接近”的指标：

- `C1` 与 `C2` 当前都基于 `FR -> DS` 关系计算
- `C4` 与 `C5` 当前都基于 `FR -> TC` 关系计算
- `C7` 与 `C8` 都在约束 TASK 上游合法性，且 `C8` 语义更强
- `C-PRD` 与 `C10` 都在 `01_specify` 阶段约束规格质量

结果是同类问题会在多个阶段重复阻断。

### 2.2 推进链不只有 Gate

真实推进链路里，`advance()` 会先执行目标阶段依赖检查，再执行 gate 评估。

这意味着即使删掉一部分 Gate 条件，以下内容仍可能继续阻断：

- 目标阶段要求的文件
- 默认脚本依赖，如 `test`、`build`
- release 前的证据文件

因此，优化如果只看 Gate 条件表，会低估真实阻力。

### 2.3 Layer2 / 扩展规则仍可继续阻断

当前系统支持从 `mergedRules.gateConditions` 注入命令型 Gate。

这意味着：

- 默认内建 Gate 变轻，不代表整条链路一定变轻
- 平台 YAML 或扩展规则仍可能追加阻断项

如果文档只写“当前阶段只剩某几个卡点”，会和运行时行为不一致。

### 2.4 指标消费层仍在放大旧复杂度

即使下线部分默认 Gate，只要以下模块还继续消费这些指标：

- health score
- bottleneck analysis
- status / metrics 展示

用户依然会看到大量“旧指标异常”，从感知上并不会觉得链路变轻。

### 2.5 输出层没有 warning 语义

当前条件结果只有：

- `PASS`
- `WAIVER`
- `FAIL`

如果只在 evaluator 层面引入 `blocking=false`，但 CLI、history、findings 仍然把它展示成 `FAIL`，用户仍然会被“伪失败”包围。

## 3. 全链路优化原则

### 原则一：默认链路只保留直接影响交付质量的硬阻断

默认流程只应拦截这些问题：

- 需求没有落成任务
- TASK 没有合法来源
- 需求没有获得测试覆盖
- 测试用例没有合法来源
- 任务没有真实完成

### 原则二：一个阶段只回答一个核心决策问题

推荐把默认阶段责任收敛为：

- `01_specify`: 需求是否已结构化
- `02_design`: 是否存在可用设计产物
- `03_plan`: 需求是否已落成合法任务
- `04_implement`: 是否具备基本测试覆盖
- `05_verify`: 覆盖是否完整且测试可追溯
- `06_wrap_up`: 是否真实完成并可归档

### 原则三：默认模式服务多数团队，严格治理通过 profile 追加

默认模式不是弱化治理，而是把治理聚焦到最高价值点。

- 默认模式服务绝大多数团队
- 严格治理通过 `strict` profile、Layer2 或扩展规则追加
- 豁免体系只服务少量必须保留硬卡点的团队

### 原则四：指标简化必须同步到消费链路

一个指标只要不再是默认 Gate 的正式组成部分，就不应该继续在默认 health score、bottleneck、CLI 核心输出里占据同等地位。

---

## 4. 推荐的全链路默认模型

### 4.1 默认硬阻断指标

推荐默认保留：

- `C3`
- `C4`
- `C6`
- `C8`
- `C9`

这 5 项足以覆盖最短闭环：

`需求 -> 任务 -> 测试 -> 完成`

### 4.2 默认下线的 Gate 指标

推荐从默认硬阻断中移除：

- `C1`
- `C2`
- `C5`
- `C7`

原因：

- `C1/C2` 当前实现重复
- `C4/C5` 当前实现重复
- `C7` 已被 `C8` 覆盖

### 4.3 默认降级为 warning-only 的质量项

推荐默认降级为 warning：

- `C-PRD`
- `C10`
- `C11`

原因：

- 这些项重要，但更适合作为质量反馈而非默认硬阻断
- 它们适合在 strict 模式恢复为 blocking

---

## 5. 全链路默认阶段分布

这里描述的是“默认推进链的正式阻断面”，不等同于系统中所有可能存在的附加规则。

### `01_specify`

默认 blocking：

- `spec.md exists`
- `FR/NFR IDs assigned`

默认 warning：

- `C-PRD`
- `C10`

### `02_design`

默认 blocking：

- `design.md exists`

默认 warning：

- `C11`

默认下线：

- `C1`
- `C2`

### `03_plan`

默认 blocking：

- `C3 = 100%`
- `C8 = 100%`
- `critical findings = 0`

### `04_implement`

默认 blocking：

- `C4 >= 80%`

默认下线：

- `C7`

### `05_verify`

默认 blocking：

- `C4 = 100%`
- `C9 = 100%`

默认下线：

- `C5`

### `06_wrap_up`

默认 blocking：

- `C6 = 100%`
- `all entries terminal`

---

## 6. 需要同步优化的非 Gate 链路

如果只改 Gate，不改下面这些链路，系统仍然会表现得像“没有被简化”。

### 6.1 依赖检查链路

应同步复审默认依赖配置：

- `02_design` 依赖 `prd.md/spec.md`
- `03_plan` 依赖 `design.md`
- `04_implement` 依赖 `test/build`
- `05_verify` 依赖 `test`
- `06_wrap_up` 依赖 `retro.md`

目标不是全部删除，而是明确哪些属于默认阻断，哪些应转移到 strict profile。

### 6.2 Layer2 / 扩展 Gate

默认方案必须明确：

- “默认简化”只约束内建 Gate
- Layer2 / 扩展规则可继续追加阻断
- 如需全局默认简化，Layer2 规则也必须支持 profile 化

### 6.3 Metrics / Health / Bottleneck

默认模式下应同步调整：

- health score 权重
- bottleneck 规则
- metrics 输出说明

否则系统会继续因为 `C1/C2/C5/C7` 产生“默认模式下不该出现的负面信号”。

### 6.4 CLI / 历史 / 审计

需要补齐 warning 语义：

- `gate check`
- `gate history`
- `findings.md`
- `gate-history.jsonl`

否则 warning-only 条件仍会被用户视为失败。

### 6.5 GoLive 下游消费

GoLive 依赖最近一次 Gate 结果。

因此必须保证：

- warning-only 条件不会把顶层 gate 状态误判为失败
- history 中能恢复“通过但有 warning”的上下文

---

## 7. 推荐的优化顺序

### Phase 1: 收缩默认阻断面

动作：

- 下线默认 Gate：`C1/C2/C5/C7`
- 将 `C-PRD/C10/C11` 改为 warning-only
- 为 `ConditionResult` / `GateResult` 建立 warning 语义

### Phase 2: 打通输出与消费层

动作：

- 更新 `gate check`、`gate history`、`findings`
- 更新 `health-score`、`bottleneck`
- 更新默认 metrics 说明

### Phase 3: 收缩推进链上的非 Gate 阻断

动作：

- 复审 dependency defaults
- 明确哪些依赖是默认模式保留，哪些迁入 strict
- 复审 Layer2 command gate 的默认注入策略

### Phase 4: profile 化

动作：

- 引入 `default-simplified`
- 引入 `strict`
- 让内建 Gate、依赖检查、metrics、Layer2 规则都能随 profile 切换

---

## 8. 最终决策建议

建议将 Spec-First 的默认链路定义为：

```text
默认 blocking: C3, C4, C6, C8, C9
默认下线: C1, C2, C5, C7
默认 warning: C-PRD, C10, C11
同步收敛: dependencies, layer2 gates, metrics, cli/history, golive
```

核心原则只有一句话：

**优化目标不是“删几个 Gate”，而是重定义默认推进链的整体阻断面。**

只有这样，Spec-First 才会从“治理点很多但难落地”，转向“有治理底线且真正能跑通”的默认研发引擎。

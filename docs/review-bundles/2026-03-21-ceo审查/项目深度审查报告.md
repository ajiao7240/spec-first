# Spec-First 项目深度审查报告

> 审查日期: 2026-03-21
> 审查范围: 项目目标、产品价值、交互体验、架构与维护成本
> 基线分支: `master`

## 一、结论先行

Spec-First 不是一个“多做一点事”的 AI 助手，而是一个把研发过程变成可审计、可推进、可回放的规范引擎。这个方向是对的，而且在 AI 工具泛滥的环境里有明确差异化。

当前的核心问题也很明确: 它已经把内部治理逻辑做得很完整，但对外暴露的概念太多，用户需要记住的文件、阶段、门禁、技能和术语超过了大多数团队的容忍阈值。换句话说，项目的治理能力已经接近成熟，产品可用性还没有跟上。

### 结论评分

| 维度 | 评分 | 说明 |
|------|------|------|
| 项目目标清晰度 | 9/10 | 方向明确，问题定义准确 |
| 治理与审计能力 | 9/10 | 状态机、追溯、门禁链路完整 |
| 架构完整性 | 8/10 | 模块边界基本清晰，但有重叠 |
| 新手可进入性 | 4/10 | 首次接入认知负担偏高 |
| 复杂度控制 | 4/10 | 概念层级过多，术语密度过高 |
| 可维护性 | 6/10 | 规范强，但文档和技能面过大 |
| 产品可扩展性 | 7/10 | 方向正确，但需要先收敛表层 |

### 一个 10/10 会是什么样

10/10 不是功能更多，而是用户只需要知道三件事: 现在处于什么状态、下一步做什么、如果被拦住该补什么。底层仍然保留追溯和门禁，但外层入口必须像一条清晰的路，而不是一套说明书。

## 二、项目目标重述

Spec-First 的目标可以压缩成一句话:

**在保持研发过程可审计的前提下，让 AI 辅助开发更稳、更快、更不容易失控。**

这意味着它要同时完成三件事:

1. 把需求到发布的链路固定下来。
2. 让每一次推进都能追溯到对应规范和证据。
3. 在用户被拦住时，给出下一步动作，而不是只给规则说明。

从产品定位看，它更像研发流程操作系统、质量门禁系统、审计系统的组合，而不是通用聊天助手，也不是一个轻量技能仓库。

## 三、当前最有价值的部分

### 3.1 方向是对的

这个项目抓住了 AI 辅助开发的真实痛点: AI 可以生成内容，但很容易失去结构、失去约束、失去上下文。Spec-First 用阶段状态机、追溯 ID、Gate 条件和技能路由，把“能做”变成“按规矩做”。

### 3.2 结构是有底座的

当前仓库已经有几个很强的底层能力:

- 单向阶段状态机，避免流程倒退和状态污染。
- 追溯矩阵和覆盖率指标，让需求、设计、任务、测试之间有链路。
- Gate 历史和阶段推进记录，让每次决策可回放。
- Docs 和 runtime 分层，机器真源与人类阅读产物分开。
- Skill 路由和 hard gate，把能力边界放在系统里，而不是靠人记忆。

### 3.3 工程纪律比多数同类项目强

从约束来看，这个项目已经不是“随便写写”的状态:

- TypeScript strict、ESM only、named exports only。
- 命名和文件规范统一。
- 测试、lint、模板和配置管理都有明确约束。
- 核心概念集中在 `src/shared/types.ts`，便于追踪和演进。

这些东西的价值不只是工程整洁，而是为后续自动化和审查提供了基础。

## 四、当前最明显的缺点

### 4.1 外部心智模型太重

用户需要理解的东西太多了:

- 阶段名。
- 追溯 ID 类型。
- Gate 分类。
- 真源、记录、索引的区别。
- Skill 层级和路由。
- 什么时候能推进，什么时候只能只读。

这会导致一个典型问题: 系统本来是来帮人省时间的，结果用户先要花时间理解系统。

### 4.2 对外的“下一步动作”不够统一

当前系统很像一个强约束的治理引擎，但用户最常见的体验不是“我理解了规则”，而是“我现在到底该干什么”。如果阻断信息只讲规则，不讲动作，就会把产品从引导工具变成说明书。

### 4.3 概念层级没有压缩

现在对内已经有很多层:

- 状态机层。
- 门禁层。
- 追溯层。
- 技能层。
- 文档层。
- 运行时层。

这些层对于实现是合理的，但对用户来说应该被压缩成更少的可见概念。当前最大问题不是层不够，而是层太多且暴露太多。

### 4.4 缺少稳定的设计和交互约定

仓库里没有独立的 `DESIGN.md`，也没有清晰的交互语义约束文档。对于一个依赖大量文本交互和命令交互的项目，这意味着“怎么说、怎么拦、怎么引导”还没有被正式固定下来。

这不是视觉设计问题，而是交互语言问题。

### 4.5 文档和技能规模偏大

从当前的产品审查记录和仓库结构看，文档、技能和参考文件数量已经开始反向增加认知成本。这个趋势如果继续走下去，会出现两个后果:

- 新用户更难进入。
- 维护者更难判断哪个文档才是真正的权威。

## 五、产品层面的根本判断

Spec-First 最适合的用户，不是想要“更聪明的 AI 助手”的人，而是愿意接受流程约束、愿意为审计和稳定性付出一点学习成本的研发团队。

它的产品价值不在于“多一个命令”，而在于:

- 让团队知道当前处于哪一步。
- 让每一步都有证据。
- 让每一次失败都能回到可执行的下一步。
- 让 AI 不只是生成，而是被流程约束。

因此，这个项目的成功标准也应该改写:

- 不是命令数量更多。
- 不是文档更多。
- 不是技能更多。
- 而是团队更快找到下一步，且仍然保留完整追溯。

## 六、我建议的改进方向

### 6.1 收窄外部入口

把对外可见能力压缩到少数几个稳定入口，例如:

- `status`，看现在在哪里。
- `catchup`，恢复上下文。
- `plan`，决定下一步。
- `doctor`，检查环境和异常。
- `spec`、`design`、`task`、`code`、`verify`，只在需要推进时出现。

目标不是删功能，而是减少用户第一次面对系统时要理解的数量。

### 6.2 把“被拦住”改成“下一步”

所有阻断信息都应该优先回答三件事:

1. 缺什么。
2. 先补什么。
3. 下一条命令是什么。

如果一条消息只告诉用户规则，不告诉他下一步动作，那就是失败的交互。

### 6.3 合并对用户来说等价的概念

内部可以继续保留模块边界，但对外不应该暴露过多平行名词。比如“真源 / 记录 / 索引 / 辅助文档”如果不能稳定区分，就应该收敛成更少的用户层概念。

### 6.4 补一份交互设计约定

如果这个项目继续扩展面向用户的命令和文档，建议补一个 `DESIGN.md` 或等价的交互约定文档，明确:

- 常用命令的语气。
- 被拦截时的提示格式。
- 新用户引导的默认顺序。
- 错误消息中哪些内容必须出现。

这比继续堆说明文字更重要。

### 6.5 先做一致性自动化，再谈生态扩张

当前最值得优先做的不是插件化、遥测和更复杂的扩展架构，而是:

- 修复技能引用完整性。
- 统一文档和运行时真源。
- 清理重复或重叠的概念。
- 把回归检查自动化。

先把底座打稳，再考虑外扩。

## 七、哪些问题现在不该做

下面这些属于明显的“海洋级”工作，不适合作为当前阶段的主目标:

- 彻底重写整个架构。
- 一次性引入插件生态。
- 为多用户协作做完整平台化改造。
- 盲目追求更激进的性能重构。
- 把项目改造成复杂的可视化前端产品。

这些方向不是永远不能做，而是现在做会把注意力从“让核心流程更顺”拉走。

## 八、当前已经存在、应该继续复用的东西

- `process-engine` 的状态机和单向阶段约束。
- `gate-engine` 的门禁评估与历史记录。
- `trace-engine` 的追溯矩阵和覆盖率计算。
- `skill-runtime` 的三层路由思想。
- `ai-orchestrator` 的自动循环和上下文恢复。
- `docs/first/*` 的生成型文档分层。
- `CLAUDE.md` 里的工程约束和操作禁区。

另外也要明确一个缺口:

- 当前没有独立 `DESIGN.md`。
- 当前没有独立 `TODOS.md`。
- 当前没有 `reviews.jsonl` 之类的稳定审查沉淀。

这不是产品错误，但确实是后续演进时的治理缺口。

## 九、优先级判断

### P0

- 压缩对外入口。
- 统一被拦截后的行动导向反馈。
- 修复技能引用和文档一致性问题。

### P1

- 合并用户视角下重复的概念。
- 补交互设计约定。
- 把常用入口做成清晰的默认路径。

### P2

- 只有在核心路径稳定后，再考虑插件化、遥测和更广泛的生态能力。

## 十、最终判断

Spec-First 是一个有明确价值的项目，而且这个价值是稀缺的。它真正的护城河不是 AI，而是把 AI 放进可约束、可审计、可推进的流程里。

但要让这个项目长期成立，必须接受一个现实: 复杂度已经高到不能再靠加文档和加技能解决，必须开始删概念、收入口、统一语言。

**一句话结论: 保住治理内核，收紧产品外壳。**

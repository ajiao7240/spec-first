## Problem Statement

**核心问题**: Spec-First 1.0 是一个"全有或全无"的工具——用户要么接受完整的 8 阶段流程，要么什么都用不了。

**具体表现**:
1. **上手太难** — 学习成本高，步骤多，用户学到一半就放弃
2. **太死板** — 无法适应用户现有的工作流，必须大改才能用
3. **效果不明显** — 功能完整但产出质量没有显著提升
4. **稳定性问题** — Bug 多，测试覆盖率不够，状态管理复杂

**关键洞察**: 用户说"只想用部分功能"（如 Gate 校验），但因为功能耦合太深，无法独立使用，最终选择**放弃不用**。

## Current Architecture Analysis

### 依赖链分析

```
Gate Engine 依赖链:
├── trace-engine/coverage → 需要 Feature 的 matrix
├── trace-engine/matrix → 需要 Feature 的 traceability.md
├── trace-engine/exception-validator → 需要 RFC 状态
├── change-mgr/rfc → 需要 Feature 的 rfc/ 目录
└── stage-state.json → 需要 Feature 初始化

结论: Gate 必须绑定 Feature 才能用
```

### 问题根源

1. **功能耦合**: Gate/Trace/Stage 相互依赖，无法独立使用
2. **强制初始化**: 任何功能都需要先 `spec-first init`
3. **流程刚性**: 8 阶段是硬编码的，无法自定义或跳过

## Target Architecture

### 设计原则

1. **核心极简** — 最小核心引擎，只负责配置/存储/插件加载
2. **能力独立** — 每个能力模块可独立安装和使用
3. **渐进采用** — 从 5 分钟上手的 Level 1 开始
4. **流程可选** — Stage 状态机只是一个插件，不是必需

### Level 统一定义

所有能力模块遵循统一的 Level 定义：

| Level | 名称 | 特点 | 示例 |
|-------|------|------|------|
| **Level 1** | 无状态 | 无配置，无持久化，即用即走 | `gate run "cmd"`, `id generate` |
| **Level 2** | 项目级 | 有配置文件，项目范围生效 | `gate check`, `id search` |
| **Level 3** | Feature 级 | 完整功能，需要 Feature 初始化 | `gate check --feature X` |

### 分层说明

#### Layer 1: Core Engine (核心引擎)

```typescript
// @spec-first/core

/**
 * 配置对象
 */
export interface Config {
  version: string;
  project: {
    name: string;
    root: string;
  };
  capabilities?: Record<string, unknown>;
  presets?: string[];
}

/**
 * 插件接口
 */
export interface Plugin {
  name: string;
  version: string;
  init(core: CoreEngine): Promise<void>;
  destroy?(): Promise<void>;
}

/**
 * 存储错误类型
 */
export class StorageError extends Error {
  constructor(
    message: string,
    public readonly code: 'NOT_FOUND' | 'PERMISSION_DENIED' | 'IO_ERROR' | 'TIMEOUT',
    public readonly path: string
  ) { super(message); }
}

/**
 * 存储适配器接口
 *
 * 错误处理约定:
 * - read() 文件不存在时抛 StorageError(NOT_FOUND)
 * - write() 失败时抛 StorageError(PERMISSION_DENIED | IO_ERROR)
 * - exists() 只检查存在性，不检查权限，返回 boolean
 * - delete() 文件不存在时静默成功
 */
export interface StorageAdapter {
  read(path: string): Promise<string>;
  write(path: string, content: string): Promise<void>;
  exists(path: string): boolean;
  delete?(path: string): Promise<void>;
}

/**
 * 核心引擎接口
 */
export interface CoreEngine {
  // 配置管理
  config: {
    load(path: string): Config;
    get(key: string): unknown;
    set(key: string, value: unknown): void;
    save(): Promise<void>;
  };

  // 存储抽象 (默认使用文件系统)
  storage: StorageAdapter;

  // 插件管理
  plugin: {
    register(plugin: Plugin): Promise<void>;
    get(name: string): Plugin | undefined;
    list(): Plugin[];
    has(name: string): boolean;
  };

  // 日志系统
  logger: {
    debug(message: string, ...args: unknown[]): void;
    info(message: string, ...args: unknown[]): void;
    warn(message: string, ...args: unknown[]): void;
    error(message: string, ...args: unknown[]): void;
  };
}

/**
 * 创建核心引擎实例
 */
export function createCoreEngine(options?: {
  storageAdapter?: StorageAdapter;
  configPath?: string;
}): CoreEngine;
```

**不包含**: 任何业务逻辑（Gate、Trace、Stage 等）

**代码量目标**: < 500 LOC

#### Layer 2: Capability Layer (能力层)

每个能力是独立的 npm 包，可以单独安装使用：

```typescript
// @spec-first/gate
export interface GateCapability {
  // Level 1: 纯命令 (无需配置)
  run(command: string): GateResult;

  // Level 2: 项目级 (需要 gate.yaml)
  check(): Promise<GateReport>;

  // Level 3: Feature 级 (完整功能)
  checkFeature(featureId: string): Promise<GateReport>;
}

// @spec-first/trace
export interface TraceCapability {
  // Level 1: ID 生成 (无状态)
  generateId(type: IdType): string;

  // Level 2: 项目级追溯
  search(query: string): TraceResult[];

  // Level 3: Feature 级矩阵
  getMatrix(featureId: string): Matrix;
}

// @spec-first/stage
export interface StageCapability {
  // Level 1: 状态查询 (无状态)
  parse(statePath: string): StageState;

  // Level 2: 项目级阶段管理
  getCurrent(): Stage;
  advance(): void;

  // Level 3: Feature 级完整流程
  advanceFeature(featureId: string): void;
}
```

#### Layer 3: Presets Layer (预设层)

预定义的使用模式：

**统一配置文件** (`.spec-first/config.yaml`):

```yaml
# .spec-first/config.yaml (统一入口)
version: 2
preset: quick-review  # 或 null 表示自定义

# 能力配置 (统一管理，不分散到多个文件)
capabilities:
  gate:
    enabled: true
    level: 2
    checks:
      - typecheck
      - lint
      - test

  trace:
    enabled: false  # quick-review 不需要

  stage:
    enabled: false  # quick-review 不需要
```

**内置预设**:

```yaml
# preset: quick-review
# 用途: 只用 Gate 校验，跳过完整流程
preset: quick-review
capabilities:
  gate:
    enabled: true
    level: 2
    checks:
      - typecheck
      - lint
      - test
  trace: { enabled: false }
  stage: { enabled: false }

# preset: bugfix
# 用途: 缺陷修复模式，简化到 3 阶段
preset: bugfix
capabilities:
  gate: { enabled: true, level: 2 }
  trace: { enabled: true, level: 1 }
  stage:
    enabled: true
    stages: [init, implement, verify]

# preset: full-flow
# 用途: 完整 8 阶段（当前默认）
preset: full-flow
capabilities:
  gate: { enabled: true, level: 3 }
  trace: { enabled: true, level: 3 }
  stage: { enabled: true, level: 3 }
  skill: { enabled: true, level: 3 }
```

**配置优先级** (从高到低):
1. CLI 参数 (如 `--level 1`)
2. 用户配置文件 (`.spec-first/config.yaml`)
3. Preset 默认值
4. 全局默认值

## Migration Strategy

### Phase 0: Monorepo Setup (0.5 周)

**目标**: 建立多包项目结构

**工作项**:
1. 创建 `pnpm-workspace.yaml`
2. 创建 `packages/` 目录结构
3. 配置 TypeScript 项目引用
4. 配置共享的 `tsconfig.base.json`
5. 更新 CI/CD 支持 monorepo

**产出**:
```yaml
# pnpm-workspace.yaml
packages:
  - 'packages/*'
```

### Phase 1: Core Engine 抽取 (1.5 周)

**目标**: 抽取核心引擎，建立新的包结构

**工作项**:
1. 创建 `@spec-first/core` 包
2. 抽取配置管理 (`config-schema.ts` → `@spec-first/core/config`)
3. 抽取存储抽象 (`fs-utils.ts` → `@spec-first/core/storage`)
4. 抽取日志系统 (`logger.ts` → `@spec-first/core/logger`)
5. 设计插件接口

**产出**:
```
packages/
├── core/
│   ├── src/
│   │   ├── config/
│   │   ├── storage/
│   │   ├── plugin/
│   │   └── logger/
│   └── package.json
```

### Phase 2: Gate 解耦 (1.5 周)

**目标**: Gate 可以独立使用，无需 Feature

**工作项**:
1. 创建 `@spec-first/gate` 包
2. 实现 Level 1 (命令 Gate)
3. 实现 Level 2 (项目 Gate)
4. 保留 Level 3 (Feature Gate) 作为高级功能
5. 重构 CLI `spec-first gate` 命令

**产出**:
```bash
# Level 1: 无需任何配置
spec-first gate run "npm test"

# Level 2: 需要 gate.yaml
spec-first gate check

# Level 3: 需要 Feature
spec-first gate check --feature FSREQ-XXX
```

### Phase 3: Trace 解耦 (1 周)

**目标**: Trace ID 系统可独立使用

**工作项**:
1. 创建 `@spec-first/trace` 包
2. 实现 Level 1 (ID 生成器，无状态)
3. 实现 Level 2 (项目级追溯)
4. 保留 Level 3 (Feature 级矩阵)
5. 重构 CLI `spec-first id` / `spec-first matrix` 命令

**产出**:
```bash
# Level 1: 无状态 ID 生成
spec-first id generate FR

# Level 2: 项目级追溯
spec-first id search "FR-001"

# Level 3: Feature 级矩阵
spec-first matrix sync --feature FSREQ-XXX
```

### Phase 4: Stage 可选化 (1 周)

**目标**: Stage 状态机变成可选插件

**工作项**:
1. 创建 `@spec-first/stage` 包
2. 定义 Stage 插件接口
3. 将 8 阶段流程实现为默认插件
4. 支持自定义阶段配置

**产出**:
```yaml
# .spec-first/config.yaml
stage:
  plugin: "@spec-first/stage-default"  # 或自定义
  config:
    stages:
      - init
      - implement
      - verify
      # 跳过 specify/design/plan...
```

### Phase 5: Presets 系统 (0.5 周)

**目标**: 提供开箱即用的预设模板

**工作项**:
1. 实现 `spec-first init --preset` 命令
2. 内置 quick-review / bugfix / full-flow 预设
3. 支持自定义预设

**产出**:
```bash
spec-first init --preset quick-review
spec-first init --preset bugfix
spec-first init --preset full-flow
spec-first init --preset ./my-preset.yaml
```

## API Design

### 用户采用路径

```
Level 1 (5 分钟):
┌─────────────────────────────────────────────────────┐
│ npm install -g @spec-first/cli                      │
│ spec-first gate run "npm test"                      │
│ → 直接使用，无需初始化                              │

Level 2 (15 分钟):
┌─────────────────────────────────────────────────────┐
│ spec-first init --preset quick-review               │
│ # 生成 .spec-first/gate.yaml                        │
│ spec-first gate check                               │
│ → 项目级 Gate 校验                                  │
└─────────────────────────────────────────────────────┘

Level 3 (1 小时):
┌─────────────────────────────────────────────────────┐
│ spec-first init --preset full-flow                  │
│ # 完整 Feature 初始化                               │
│ spec-first feature create MYFEATURE-001             │
│ spec-first stage advance                            │
│ → 完整 8 阶段流程                                   │
└─────────────────────────────────────────────────────┘
```

### CLI 命令重构

```bash
# 全局命令 (无需 Feature)
spec-first gate run <command>           # 执行单个命令
spec-first gate check                   # 项目级检查
spec-first id generate <type>           # ID 生成
spec-first id search <query>            # 项目级搜索

# Feature 命令 (需要初始化)
spec-first feature create <id>          # 创建 Feature
spec-first feature current              # 查看当前 Feature
spec-first stage current                # 查看当前阶段
spec-first stage advance                # 推进阶段
spec-first gate check --feature <id>    # Feature 级 Gate
spec-first matrix sync --feature <id>   # Feature 级矩阵

# 初始化命令
spec-first init                         # 交互式初始化
spec-first init --preset <name>         # 使用预设
spec-first init --minimal               # 最小化初始化
```

## Risks & Mitigations

| 风险 | 级别 | 缓解措施 |
|------|------|---------|
| 重构工作量大 | 高 | 分 5 个 Phase，每 Phase 可独立交付 |
| 破坏现有用户 | 中 | 保留 Level 3 API 完全兼容 |
| 架构过于复杂 | 中 | Core Engine 保持极简，复杂度在 Capability 层 |
| 插件生态难建立 | 低 | 先做好内置能力，生态是加分项 |

## Success Criteria

1. **Level 1 可用**: 用户无需初始化即可使用 `spec-first gate run`
2. **Level 2 可用**: 用户可以用 `--preset quick-review` 在 5 分钟内上手
3. **Level 3 兼容**: 现有 Feature 流程完全兼容
4. **测试覆盖**: 每个包独立测试，覆盖率 ≥ 80%
5. **文档完整**: 每个包有独立 README 和 API 文档

## Skill 系统解耦方案 (关键设计决策)

### 问题分析

当前 Skill 系统与 Stage 存在**循环依赖风险**：
- Skill 需要知道当前 Stage 才能提供正确的上下文
- Stage 推进需要 Skill 执行
- 如果 Stage 是可选的，Skill 如何工作？

### 解决方案: Skill Context Protocol

```typescript
// @spec-first/skill

/**
 * Skill 执行上下文协议
 * Skill 不直接依赖 Stage，而是依赖这个协议
 */
export interface SkillContext {
  // 必需: 项目根目录
  projectRoot: string;

  // 可选: Feature 信息 (如果用户在使用 Feature 模式)
  featureId?: string;
  featurePath?: string;

  // 可选: 阶段信息 (如果用户在使用 Stage 模式)
  currentStage?: string;
  stageHistory?: string[];

  // 必需: 能力注册表 (Skill 通过这个获取其他能力)
  capabilities: {
    gate?: GateCapability;
    trace?: TraceCapability;
    stage?: StageCapability;
  };
}

/**
 * Skill 定义
 */
export interface Skill {
  name: string;
  description: string;

  // Skill 声明它需要哪些能力 (可选)
  requiredCapabilities?: string[];

  // Skill 执行函数
  execute(context: SkillContext): Promise<SkillResult>;
}
```

### 三种 Skill 模式

```yaml
# Mode 1: 无状态 Skill (不依赖任何能力)
skill: help
requiredCapabilities: []
# → 可以在任何环境下执行

# Mode 2: 项目级 Skill (需要 Gate 能力)
skill: lint
requiredCapabilities: [gate]
# → 需要 @spec-first/gate 包

# Mode 3: Feature 级 Skill (需要完整能力)
skill: code
requiredCapabilities: [gate, trace, stage]
# → 需要完整 Feature 环境
```

### CLI 行为

```bash
# Skill 检查能力是否满足
spec-first code
# → 检测到需要 [gate, trace, stage]
# → 如果当前环境缺少，提示用户:
#    "This skill requires: gate, trace, stage"
#    "Run: spec-first init --preset full-flow"
```

### Issue #4: CLI 动态分发机制

**问题**: CLI 命令应该按需加载，避免启动时加载所有包。

**解决方案**: 使用动态 import 实现命令分发。

```typescript
// packages/cli/src/dispatcher.ts
import type { Command } from 'commander';

export async function dispatchCommand(
  capability: string,
  args: string[]
): Promise<void> {
  const module = await import(`@spec-first/${capability}`);
  const command = module.createCommand() as Command;
  await command.parseAsync(args, { from: 'user' });
}

// 命令注册表 (只加载元数据，不加载实现)
const COMMAND_REGISTRY = {
  'gate check': { capability: 'gate', level: 2 },
  'gate run': { capability: 'gate', level: 1 },
  'id generate': { capability: 'trace', level: 1 },
  'matrix sync': { capability: 'trace', level: 3 },
  // ...
};
```

**收益**:
- 启动时间从 ~500ms 降到 ~50ms
- 按需加载，减少内存占用
- 支持未来插件扩展

### Issue #5: CoreEngine 接口拆分

**问题**: `CoreEngine` 职责过重，应拆分为独立接口。

**解决方案**: 按功能域拆分接口。

```typescript
// packages/core/src/types.ts

/**
 * 配置管理
 */
export interface ConfigManager {
  load(path: string): Promise<ProjectConfig>;
  get<K extends keyof ProjectConfig>(key: K): ProjectConfig[K];
  set<K extends keyof ProjectConfig>(key: K, value: ProjectConfig[K]): void;
  save(): Promise<void>;
}

/**
 * 存储抽象
 */
export interface StorageAdapter {
  read(path: string): Promise<string>;
  write(path: string, content: string): Promise<void>;
  exists(path: string): Promise<boolean>;
  delete(path: string): Promise<void>;
  list(dir: string): Promise<string[]>;
}

/**
 * 插件管理
 */
export interface PluginManager {
  load(name: string): Promise<Plugin>;
  unload(name: string): Promise<void>;
  get(name: string): Plugin | undefined;
  list(): Plugin[];
}

/**
 * 组合接口 (可选，方便依赖注入)
 */
export interface CoreEngine extends ConfigManager, StorageAdapter, PluginManager {
  // 空接口，只做组合
}
```

**收益**:
- 单一职责，每个接口职责清晰
- 方便单元测试 (可以 mock 单个接口)
- 支持部分依赖 (某模块只需 ConfigManager)

### Issue #6: Decorator 模式实现 Level

**问题**: Level 1/2/3 有大量重复代码，需要复用机制。

**解决方案**: 使用 Decorator 模式逐层增强能力。

```typescript
// packages/gate/src/levels.ts

/**
 * Level 1: 无状态 Gate
 */
export function createLevel1Gate(): Gate {
  return {
    async run(command: string): Promise<GateResult> {
      const start = Date.now();
      try {
        await exec(command);
        return { passed: true, duration: Date.now() - start };
      } catch (error) {
        return { passed: false, error: error.message };
      }
    }
  };
}

/**
 * Level 2 Decorator: 增加项目配置
 */
export function withProjectConfig(base: Gate, config: GateConfig): Gate {
  return {
    ...base,
    async run(command?: string): Promise<GateResult> {
      const cmd = command || config.defaultCommand;
      const result = await base.run(cmd);

      // 应用项目级阈值
      if (result.duration > config.timeout) {
        result.passed = false;
        result.error = `Timeout after ${config.timeout}ms`;
      }

      return result;
    }
  };
}

/**
 * Level 3 Decorator: 增加 Feature 追溯
 */
export function withFeatureContext(
  base: Gate,
  featureId: string,
  trace: TraceEngine
): Gate {
  return {
    ...base,
    async run(command?: string): Promise<GateResult> {
      const result = await base.run(command);

      // 记录到 Gate History
      await trace.recordGateHistory(featureId, result);

      return result;
    }
  };
}

// 使用
const gate = withFeatureContext(
  withProjectConfig(createLevel1Gate(), config),
  featureId,
  trace
);
```

**收益**:
- 消除 Level 1/2/3 之间的代码重复
- 每个装饰器职责单一，易于测试
- 支持灵活组合 (如 Level 2 + 自定义装饰器)

### Issue #7: Branded Types for SkillContext

**问题**: `SkillContext` 容易与普通对象混淆，需要类型安全。

**解决方案**: 使用 Branded Types 增加编译时检查。

```typescript
// packages/core/src/types.ts

/**
 * Branded Type 工具
 */
declare const brand: unique symbol;

export type Brand<T, TBrand extends string> = T & {
  [brand]: TBrand;
};

/**
 * Skill Context Brands
 */
export type Level1Context = Brand<SkillContext, 'Level1'>;
export type Level2Context = Brand<SkillContext, 'Level2'>;
export type Level3Context = Brand<SkillContext, 'Level3'>;

/**
 * Context Factory (类型安全)
 */
export function createLevel1Context(): Level1Context {
  return { level: 1 } as Level1Context;
}

export function createLevel2Context(config: ProjectConfig): Level2Context {
  return { level: 2, config } as Level2Context;
}

export function createLevel3Context(
  config: ProjectConfig,
  featureId: string
): Level3Context {
  return { level: 3, config, featureId } as Level3Context;
}

/**
 * Skill 函数签名 (强制类型检查)
 */
export type SkillHandler<TContext extends SkillContext> = (
  ctx: TContext,
  args: string[]
) => Promise<void>;

// 示例: 只能接受 Level3Context
export const matrixSyncHandler: SkillHandler<Level3Context> = async (
  ctx,
  args
) => {
  // ctx.featureId 编译时保证存在
  await syncMatrix(ctx.featureId);
};

// 编译错误: Level1Context 不能赋值给 Level3Context
// matrixSyncHandler(createLevel1Context(), []); // ❌ TypeError
```

**收益**:
- 编译时防止 Context 级别混用
- 无运行时开销 (纯类型检查)
- 代码意图更清晰

### Issue #8: 测试策略

**问题**: 缺少明确的测试分层策略。

**解决方案**: 三层测试金字塔。

```
        ┌─────────────────┐
        │   E2E Tests     │  10%
        │  (CLI 集成)      │  - 用户真实场景
        └─────────────────┘  - 预设端到端流程
               ▲
        ┌─────────────────┐
        │ Integration     │  30%
        │   Tests         │  - 跨包交互
        └─────────────────┘  - Decorator 组合
               ▲              - 配置加载
        ┌─────────────────┐
        │  Unit Tests     │  60%
        │                 │  - 每个 Level 独立测试
        └─────────────────┘  - Decorator 单元测试
                             - Storage Mock
```

**测试覆盖目标**:
- Unit Tests: 90% (核心逻辑全覆盖)
- Integration Tests: 70% (关键路径覆盖)
- E2E Tests: 3-5 个核心场景

**关键测试用例**:

### Issue #10: Gate 命令超时

**问题**: Gate 执行命令时可能永久挂起。

**解决方案**: 增加超时机制与进程管理。

```typescript
// packages/gate/src/executor.ts

export interface ExecOptions {
  timeout?: number; // 毫秒，默认 60000 (1 分钟)
  signal?: AbortSignal;
}

export async function execWithTimeout(
  command: string,
  options: ExecOptions = {}
): Promise<{ stdout: string; stderr: string; exitCode: number }> {
  const timeout = options.timeout || 60000;
  const controller = new AbortController();

  // 超时定时器
  const timeoutId = setTimeout(() => {
    controller.abort();
  }, timeout);

  // 外部信号
  options.signal?.addEventListener('abort', () => {
    controller.abort();
  });

  try {
    const result = await exec(command, {
      signal: controller.signal,
      timeout
    });

    clearTimeout(timeoutId);
    return result;
  } catch (error) {
    clearTimeout(timeoutId);

    if (error.name === 'AbortError') {
      throw new Error(`Command timed out after ${timeout}ms: ${command}`);
    }

    throw error;
  }
}

// packages/gate/src/project-gate.ts
export function withProjectConfig(base: Gate, config: GateConfig): Gate {
  return {
    ...base,
    async run(command?: string): Promise<GateResult> {
      const cmd = command || config.defaultCommand;
      const timeout = config.timeout || 60000; // 默认 1 分钟

      try {
        const result = await execWithTimeout(cmd, { timeout });
        return { passed: result.exitCode === 0, ...result };
      } catch (error) {
        return {
          passed: false,
          error: error.message,
          timeout: error.message.includes('timed out')
        };
      }
    }
  };
}
```

**配置示例**:

```yaml
# .spec-first/config.yaml
gate:
  enabled: true
  level: 2
  timeout: 120000  # 2 分钟
  checks:
    - command: "npm run typecheck"
      timeout: 30000  # 30 秒
    - command: "npm test"
      timeout: 180000  # 3 分钟
```

**收益**:
- 防止命令永久挂起
- 支持命令级别超时配置
- 与 AbortController 集成，支持取消

## ⚠️ 关键问题

### Issue #1: 时间线不一致 🔴 CRITICAL

**问题描述**: 不同文档中的时间估算存在严重冲突

| 文档 | 时间估算 | 说明 |
|------|---------|------|
| **2.0 设计方案** | 6-7 周 | Phase 0-5，关键路径 5 周 |
| **90天实施计划** | 90 天 | Phase 1-3，每个 30 天 |
| **Strategic Review** | 3-4 周 | 立即行动项 |

**影响**:
- 用户无法判断真实工期
- 资源规划困难
- 期望管理混乱

**建议**:
1. **统一时间线**: 90 天计划应该与 6-7 周的 Phase 0-5 对齐
2. **明确里程碑**: 说明 6-7 周是核心重构，90 天是完整落地（含用户验证、迭代）
3. **优先级标注**: 3-4 周应该是 MVP 或第一阶段

**修复方案**:
```markdown
# 90天实施计划 应该说明:
- Phase 1 (1-30 天): 对应设计方案 Phase 0-1 (2 周) + 用户验证
- Phase 2 (31-60 天): 对应设计方案 Phase 2-3 (2.5 周) + 迭代
- Phase 3 (61-90 天): 对应设计方案 Phase 4-5 (1.5 周) + 稳定化

6-7 周是核心开发时间，90 天包含验证、迭代和稳定期。
```

---

### Issue #2: 文档间引用不完整 🟡 MEDIUM

**问题描述**: 部分文档缺少交叉引用

- ❌ `90天实施计划.md` 没有引用 `2.0-设计方案.md`
- ❌ `Brownfield-PRD.md` 没有引用 `Brownfield-总方案.md`
- ❌ 没有文档引用 `项目深度审查报告.md` 的具体章节

**影响**:
- 阅读者需要手动在文档间跳转
- 容易遗漏关键上下文

**建议**:
在每个文档开头添加"相关文档"章节：
```markdown
## 相关文档
- 设计方案: [Spec-First-2.0-设计方案.md](./Spec-First-2.0-设计方案.md)
- 实施计划: [Spec-First-90天实施计划.md](./Spec-First-90天实施计划.md)
- 架构图解: [架构图解.md](./架构图解.md)
```

---

### Issue #3: Phase 命名冲突 🟡 MEDIUM

**问题描述**: Phase 在不同文档中含义不同

- **2.0 设计方案**: Phase 0-5（技术重构阶段）
- **90天实施计划**: Phase 1-3（30 天里程碑）

**影响**:
- 容易混淆
- 讨论时需要额外说明"哪个 Phase"

**建议**:
统一术语：
- 技术重构阶段: Phase 0-5
- 时间里程碑: Milestone 1-3（或 Sprint 1-3）

---

### Issue #4: 部分内容重复 🟢 MINOR

**重复内容**:
1. Level 1/2/3 定义在 4 个文档中重复
2. "全有或全无"问题描述在 3 个文档中重复
3. Phase 依赖图在 2 个文档中重复

**影响**:
- 维护负担：修改时需要同步多处
- 阅读负担：相同内容重复阅读

**建议**:
- 保留核心文档（如 2.0 设计方案）的完整描述
- 其他文档引用核心文档，不重复定义
- 示例：
  ```markdown
  ## Level 定义
  详见 [Spec-First-2.0-设计方案.md#Level-统一定义](./Spec-First-2.0-设计方案.md#level-统一定义)
  ```

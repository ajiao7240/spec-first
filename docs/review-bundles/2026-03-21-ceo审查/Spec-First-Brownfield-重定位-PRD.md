# Spec-First Brownfield 重定位 PRD

> 日期: 2026-03-22
> 范围: 存量项目需求迭代场景
> 目标版本: 1.5 -> 2.0 过渡期

## 1. 产品定义

### 1.1 新定位

Spec-First 不再优先定义为“完整 spec-driven 流程引擎”，而是优先定义为:

**面向存量项目需求迭代的 AI 治理层。**

一句话价值主张:

**让团队在不重构现有工作流的前提下，安全地用 AI 修改已有系统。**

### 1.2 为什么现在要重定位

当前业界对 `0-1` 的 AI 开发流程已经较多，但对 `1-10`、`10-100` 的存量项目迭代还没有形成稳定产品形态。

真实痛点不是“写不出代码”，而是:

- 不知道改动会影响哪里。
- 不知道最小验证集是什么。
- 不知道 reviewer 应该先看什么。
- 不知道如何在不强推全流程的前提下保留审计和约束。

## 2. 用户与场景

### 2.1 核心用户

1. 存量系统研发
   目标: 快速、安全完成已有模块的修改。

2. Reviewer / Tech Lead
   目标: 快速识别高风险点、缺测点和行为回归。

3. 团队流程 owner
   目标: 让 AI 使用可控、可审计，但不要强推完整流程。

### 2.2 核心场景

- Bug 修复
- 小中型功能增量
- 复杂模块重构前影响分析
- PR 前 review 与验证建议
- 需要时再补 traceability 和 evidence

### 2.3 非目标场景

- 全新项目从零搭建
- 纯视觉设计工具
- 多用户协作平台
- 通用 Agent marketplace

## 3. 现状问题

### 3.1 当前 1.x 的主要问题

1. 全有或全无
   用户往往只想要 `review/gate/analyze`，但系统默认要求理解更完整的 flow。

2. 认知负担过高
   Stage、Gate、Trace、Docs、Runtime、Feature 等概念同时暴露。

3. 存量项目接入不够渐进
   需要先初始化、先理解流程、先接受结构，才能享受部分能力。

4. 文档与能力边界太大
   会让用户把它理解成“很完整，但很重”。

## 4. 产品目标

### 4.1 北极星目标

**让存量项目的 AI 改动更安全、更易 review、更容易验证。**

### 4.2 成功标准

- 用户不需要完整 feature flow 也能获得价值。
- 小改动可以在 5-15 分钟内完成接入。
- Reviewer 能直接拿到影响分析和验证建议。
- 高风险改动才需要升级到完整 spec-first flow。

## 5. 产品原则

1. 先解决存量改动，再扩展完整治理。
2. 默认轻量接入，按风险升级。
3. 先输出下一步动作，再解释规则。
4. 最小可用入口优先于完整术语体系。
5. 审计能力保留，但不强制前置。

## 6. 目标产品结构

### 6.1 三层能力

#### Level 1: 无状态即用

解决“我现在就要分析/审查/验证这次改动”。

示例:

- `analyze-impact`
- `review-change`
- `next-action`
- `id generate`

#### Level 2: 项目级治理

解决“这个项目有自己的规则，希望复用”。

示例:

- 项目级 gate 配置
- 默认测试策略
- 默认 review 模板
- 默认命令推荐

#### Level 3: Feature 级完整流程

解决“这是高风险或跨阶段改动，需要完整追溯和阶段推进”。

示例:

- `init/spec/design/task/code/verify/archive`
- 完整 traceability
- Stage gate

### 6.2 核心产品模块

1. Analyze
   输出影响面、风险点、建议阅读文件、建议验证集。

2. Review
   输出行为回归风险、边界条件、缺测风险、建议补测。

3. Verify
   输出最小验证闭环和推荐测试顺序。

4. Track
   在需要时补证据、补 trace、补 findings。

5. Flow
   高风险场景进入完整 spec-first flow。

## 7. 核心需求

### FR-001 渐进接入

系统必须支持用户不初始化 feature 也能获得 `analyze/review/verify` 价值。

### FR-002 影响分析

系统必须能从 `diff/issue/file/module` 生成高信号影响分析。

### FR-003 Review 优先

系统必须在生成更多流程文档之前，优先指出风险、缺测和行为变化。

### FR-004 风险分级升级

系统必须能基于风险判断是否建议用户进入完整流程。

### FR-005 统一交互语言

系统必须统一输出:

- 当前判断
- 风险等级
- 缺失项
- 建议下一步
- 建议命令

## 8. 关键指标

### 8.1 产品指标

- 首次有效使用时间
  目标: <= 15 分钟

- 小改动轻流程覆盖率
  目标: >= 60%

- review 前影响分析使用率
  目标: >= 50%

- 从轻流程升级到完整流程的准确率
  目标: 高风险场景召回率足够高

### 8.2 质量指标

- 影响分析误报率
- 建议测试集可执行率
- 阻断提示命中率
- 文档/skill/引用漂移率

## 9. 版本策略

### 1.5

做出 brownfield 最小楔子，不动大底座。

### 2.0

在 1.5 验证成立后，再推进能力分层、preset 和更深的模块解耦。

## 10. 结论

Spec-First 最值得做的不是“更完整”，而是“更容易从存量项目切入”。如果 1.5 能把 `Analyze + Review + Verify + Next Action` 做成可靠默认路径，2.0 才有必要继续往模块化治理层演进。

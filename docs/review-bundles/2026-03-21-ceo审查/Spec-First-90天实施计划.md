# Spec-First 90 天实施计划

> 日期: 2026-03-22
> 目标: 用 90 天把 Spec-First 从“全流程工具”推进到“存量项目渐进治理层”

## Phase 1: 第 1-30 天

### 目标

验证最小 brownfield 楔子是否成立。

### 交付

1. `analyze-impact`
   输入 `diff/issue/file/module`，输出:
   - 影响模块
   - 风险点
   - 建议阅读文件
   - 建议验证集

2. `review-change`
   输入当前 diff，输出:
   - 行为回归风险
   - 缺测点
   - 边界条件
   - 建议补测

3. `next-action`
   输入当前状态，输出:
   - 缺什么
   - 先补什么
   - 下一步命令

### 工程要求

- 不依赖 feature init。
- 不强制完整 traceability。
- 输出模板统一。
- 支持在已有仓库直接运行。

### 验收标准

- 一个存量项目可以在 15 分钟内用起来。
- 小改动场景不需要完整 8 阶段也能得到价值。
- 输出能直接被作者和 reviewer 消费。

## Phase 2: 第 31-60 天

### 目标

把单次能力升级成项目级轻治理。

### 交付

1. 项目级默认配置
   - 默认 gate
   - 默认测试策略
   - 默认 review 模板

2. 预设模式
   - `quick-review`
   - `bugfix`
   - `full-flow`

3. 一致性自动化
   - skill 引用校验
   - 文档/真源一致性检查
   - 命令输出格式回归检查

4. 交互设计规范初稿
   - 阻断提示格式
   - 下一步推荐格式
   - 常用命令输出层级

### 验收标准

- 项目级接入可复用。
- 新用户不需要先理解完整术语体系。
- 文档和 skill 漂移开始可控。

## Phase 3: 第 61-90 天

### 目标

把轻流程和重流程真正打通。

### 交付

1. `Level 1/2/3` 落地
   - Level 1: 即用即走
   - Level 2: 项目级
   - Level 3: Feature 级

2. 风险驱动升级
   - 低风险场景继续走轻流程
   - 高风险场景建议升级到完整 flow

3. 轻流程到完整 flow 的桥接
   - 从 diff/issue 自动生成 feature 追踪入口
   - 从 review 结果补 trace/evidence

4. 文档重组
   - 用户入口文档
   - reviewer 入口文档
   - advanced flow 文档

### 验收标准

- 同一个团队能混用轻流程和完整流程。
- 高风险场景不会继续被轻流程放过。
- 用户不需要理解底层架构也能做正确选择。

## 不做的事

90 天内明确不做:

- 全面 monorepo 重构
- 全模块 npm 包化
- 插件 marketplace
- 多用户平台化
- 视觉前端产品化

## 风险

1. 做成“又一套新术语”
   缓解: 先压缩输出语言。

2. 做成“半套 spec-first”
   缓解: 明确轻流程就是产品，而不是残缺模式。

3. 做成“分析很多但无法执行”
   缓解: 所有输出都必须附带下一步动作和命令建议。

## 成功标准

90 天后，如果用户可以在存量项目里优先使用 `analyze + review + verify + next-action`，并且只在必要时才进入完整 flow，这次改造就是成功的。

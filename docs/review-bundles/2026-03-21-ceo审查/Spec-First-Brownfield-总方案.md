# Spec-First Brownfield 总方案

> 日期: 2026-03-22
> 当前分支: `leo-2026-03-19`
> 基线分支: `master`
> 目的: 把现有审查结论收敛成一份可讨论、可决策、可执行的统一方案

## 1. 结论

Spec-First 值得改造，而且应该明确转向:

**从“完整流程引擎”优先，转为“存量项目 AI 迭代治理层”优先。**

原因很简单:

- `0-1` 的 AI 开发流程行业里已经很多。
- `1-10`、`10-100` 的存量项目需求迭代流程还不成熟。
- 真正难的不是生成代码，而是在已有系统里安全地改对东西。

所以最值得做的不是继续扩大全流程，而是先吃下 brownfield 这个高价值缺口。

## 2. 当前判断

### 2.1 项目已经做对的部分

- 有清晰的状态机和阶段边界。
- 有追溯和 Gate，治理内核扎实。
- 有 docs/runtime 分层，适合做审计和恢复。
- 有 Skill 路由和工程规范，基础设施不差。

### 2.2 当前最阻碍增长的部分

- 对外概念太多，认知负担过重。
- 用户被拦住时，系统解释规则多于给出动作。
- 默认接入路径偏重，轻量场景没有足够好的第一入口。
- 用户想只用一部分能力时，仍然容易被完整流程绑住。

## 3. 新定位

### 3.1 新一句话定位

**Spec-First 是面向存量项目需求迭代的 AI 治理层。**

### 3.2 新价值主张

**让团队在不重构现有工作流的前提下，安全地用 AI 修改已有系统。**

### 3.3 核心用户

1. 存量系统研发
2. Reviewer / Tech Lead
3. 团队流程 owner

### 3.4 核心场景

- Bug 修复
- 小中型增量功能
- 重构前影响分析
- PR 前 review
- 最小验证闭环
- 必要时升级到完整 flow

## 4. 产品策略

### 4.1 产品原则

1. 先解决存量改动，再扩展完整治理。
2. 默认轻量接入，按风险升级。
3. 先输出下一步动作，再解释规则。
4. 审计能力保留，但不强制前置。
5. 用户按场景进入，不按内部架构进入。

### 4.2 三层能力

#### Level 1: 无状态即用

适合:
- 我现在就想分析这次改动
- 我现在就想 review 当前 diff
- 我现在就想知道最小验证集

代表能力:
- `analyze-impact`
- `review-change`
- `next-action`

#### Level 2: 项目级治理

适合:
- 这个仓库有自己的默认规则
- 团队希望复用 review/gate/verify 约定

代表能力:
- 项目级默认 gate
- 默认测试策略
- 默认 review 模板
- 默认命令推荐

#### Level 3: Feature 级完整流程

适合:
- 高风险改动
- 跨阶段交付
- 需要完整 trace 和 stage gate

代表能力:
- `init/spec/design/task/code/verify/archive`

## 5. 推荐的产品结构

### 5.1 用户第一层只暴露 4 个动作

1. `analyze`
   这次改动会影响什么。

2. `review`
   这次改动哪里最危险。

3. `verify`
   最小验证闭环是什么。

4. `next`
   下一步应该怎么做。

### 5.2 项目级辅助入口

- `status`
- `doctor`
- `sync`
- `catchup`

### 5.3 高级流程入口

- `flow init`
- `flow spec`
- `flow design`
- `flow task`
- `flow code`
- `flow verify`
- `flow archive`

或者继续保留现有 namespace，但在对外表达里明确它们是 advanced flow。

## 6. 90 天路线图

### 第 1-30 天

目标:
- 验证最小 brownfield 楔子。

必须交付:
- `analyze-impact`
- `review-change`
- `next-action`

验收:
- 不依赖 feature init。
- 可以直接在已有仓库用。
- 输出模板稳定，可直接给作者和 reviewer 使用。

### 第 31-60 天

目标:
- 从单次能力升级到项目级轻治理。

必须交付:
- 项目级默认配置
- `quick-review` / `bugfix` / `full-flow` presets
- 一致性自动化
- 交互设计规范初稿

验收:
- 新用户不必先学完整术语体系。
- skill / 文档 / 真源漂移开始可控。

### 第 61-90 天

目标:
- 打通轻流程和完整流程。

必须交付:
- `Level 1/2/3` 落地
- 风险驱动升级
- 从 diff/issue 到完整 feature flow 的桥接
- 文档重组

验收:
- 小改动走轻流程。
- 高风险改动升级完整 flow。
- 同一团队可以混用两套模式。

## 7. 现在不该做的事

90 天内不建议优先做:

- 全仓 monorepo 大迁移
- 全模块 npm 包化
- 插件 marketplace
- 多用户平台化
- 视觉前端产品化

这些都可能是未来方向，但都不是当前最高价值矛盾。

## 8. 最小执行建议

如果只能先做一件事，先做:

**把 `analyze + review + verify + next-action` 做成存量项目默认入口。**

如果还能再做一件事，做:

**统一阻断反馈格式: 缺什么、先补什么、下一步命令是什么。**

如果还能再做第三件事，做:

**把完整 flow 明确降级为高级模式，而不是默认第一入口。**

## 9. 决策建议

### 应该立刻拍板的

- 产品重定位方向
- 轻流程优先
- 90 天不做平台级大重构

### 应该在 Phase 1 内验证的

- 用户是否真的愿意先用 `analyze/review/verify`
- 风险升级到完整 flow 的判断是否足够准确
- 统一交互语言是否显著降低认知负担

## 10. 一句话版本

**保留 spec-first 的治理内核，但把默认入口从“完整流程”改成“存量项目安全迭代”。**

# 2026-03-07 04-design skill 审查与优化建议

## 1. skill 要解决什么问题

按主链边界，`04-design` 不是写代码，也不是重复写需求。

它真正要解决的是：

1. 把 `03-spec` 已确认的 FR 转成系统级 `HOW`
2. 明确模块边界、接口契约、数据模型、一致性与回滚策略
3. 在进入 `task / code` 之前，把关键技术决策和系统约束先落盘
4. 让后续实现、测试、验证都能沿着同一套设计事实推进

一句话：

> `04-design` 解决“需求准备怎么在系统层面落地”。

## 2. 当前实现是否合理

整体判断：合理，而且方向基本正确。

### 2.1 合理的地方

1. `WHAT / HOW` 边界清楚
   - `spec` 负责需求定义
   - `design` 负责系统级 HOW
   - 同时明确禁止类/函数级实现细节

2. 入口守卫是对的
   - 要求当前阶段为 `02_design`
   - 要求 `spec.md` 已存在
   - 这符合主链顺序，也能避免没规格就先画方案

3. 宪法一致性检查应当保留
   - 设计阶段必须对照宪法检查 DS
   - 冲突时先修设计，不允许带冲突推进
   - 这符合系统治理最佳实践

4. FR -> DS -> Matrix 的追踪意识是对的
   - 设计不是孤立文档
   - 它和矩阵、覆盖率、后续 gate 是联动的

5. 对高风险设计采用 `strict` 确认策略，方向合理
   - 技术设计确实比一般状态汇报更需要明确确认

## 3. 当前主要问题

### 3.1 成功标准过低，无法代表“设计完成”

当前成功标准里最显眼的问题是：

- `metrics coverage` C1 (Design Coverage) `> 0%`

这几乎不能作为设计完成标准。

如果只要求 `> 0%`，就意味着：

- 只覆盖了极少数 FR
- 仍然可能有大量 FR 没有设计映射
- 也能被判为“通过”

这会直接削弱 `design` 在全链路里的治理价值。

更合理的门槛应该至少是：

- 所有进入实现范围的 FR 都有 DS 映射
- 不允许 orphan FR
- 关键接口、数据模型、异常/一致性策略完整

### 3.2 系统设计维度还不够全

当前 `design` 明确要求：

- 模块边界
- 接口契约
- 数据模型
- 一致性/回滚策略

这已经不错，但对真正的系统设计来说还不够完整。

目前缺得比较明显的维度有：

- 状态流转 / 时序关系
- 异常路径与失败域
- 迁移策略
- 发布/灰度/回滚路径
- 观测性与告警点
- 性能容量与瓶颈假设
- 安全边界与权限控制面

如果 `design` 只停留在“模块 + API + 表结构”，它更像结构图，而不是可指导实施和验证的系统设计。

### 3.3 没有按复杂度分层，默认路径偏硬

不是所有设计都需要同等深度。

例如：

- 小型 API 字段增强
- 单模块内部重构
- 文案类行为调整

这些场景不一定需要完整的系统级设计包。

而复杂需求则需要更完整的：

- 契约
- 时序
- 一致性
- 回滚
- 风险与发布策略

当前 `design` 没有像 `spec` 一样把轻量路径和完整路径分层，这会让简单需求承受不必要的设计负担。

### 3.4 skill 内混入宿主上下文同步副作用

当前文档要求：

- design 结束并推进阶段后，必须触发 `CLAUDE.md / AGENTS.md` 托管区块同步

这个动作不是完全没价值，但它更像宿主集成或运行时治理能力，不是 `design` 的核心职责。

把它塞进 `design` 会带来两个问题：

1. 技术设计 skill 被环境副作用污染
2. 用户容易把“设计完成”和“宿主上下文同步成功”混成一件事

最佳实践应是：

- `design` 负责设计产物和确认
- 宿主同步由平台层或 `sync / orchestrate` 承接

### 3.5 文档自身存在重复与收口不够干净

当前 `Feature 定位规则` 在文档里重复出现了两次。

这不是大问题，但它说明 `SKILL.md` 还没完全收口，后续继续扩展时容易产生更多重复和漂移。

## 4. 推荐优化方向

### 4.1 把 `design` 收敛成“系统设计主入口”

默认应聚焦这 6 类内容：

1. 模块边界
2. 接口契约
3. 数据模型
4. 状态流转 / 时序
5. 一致性 / 异常 / 回滚
6. 风险与验证要点

这才是后续 `task / code / test / verify` 真正会消费的设计核心。

### 4.2 为不同复杂度提供分层模板

建议至少分两层：

1. 轻量设计
   - 面向 Simple 需求
   - 只要求模块边界、接口变化、数据影响、风险提示

2. 完整设计
   - 面向 Moderate / Complex
   - 额外要求时序图、失败路径、迁移策略、观测性、发布回滚

这样能避免所有需求都被迫走最重设计模板。

### 4.3 提高覆盖和完成门槛

建议把成功标准改成更接近真实完成态：

1. 进入实施范围的 FR 必须 100% 映射到 DS
2. 不允许 orphan FR
3. 关键 DS 必须具备接口/数据/异常/一致性字段
4. 复杂需求必须包含发布与回滚策略

也就是说，`design` 的成功标准必须能约束“设计足够用”，而不是“写了一点就算过”。

### 4.4 把宿主同步从核心流程中剥离

建议改成：

- `design` 完成后可触发上下文同步
- 但它属于附加编排动作，不属于设计主流程成功标准

这样 `design` 的职责会更纯。

### 4.5 增加“设计服务谁”的输出约束

最佳实践下，设计文档不只是给实现者看，也要服务：

- `task` 的任务拆解
- `code` 的实现边界
- `test` 的测试面识别
- `verify` 的验收证据

因此建议在 `design.md` 中固定增加一节：

- `Implementation Notes`
- `Test Focus`
- `Rollback / Release Notes`

这三块能明显增强全链路可消费性。

## 5. 当前审查结论

- 职责定位：正确
- 当前合理性：中高
- 主要问题：成功标准太弱，系统设计维度不够全，且混入非核心副作用
- 最佳优化方向：升级成“分层系统设计入口”，提高覆盖门槛，补齐系统级设计模板

一句话结论：

> `04-design` 方向没有错，但现在更像“设计草图生成器”；下一步应把它收敛成真正可支撑实现、测试、验证的系统设计入口。

# 2026-03-07 全项目详细优化方案 v2

> **定位**：本文件是 `spec-first` 的重构基线文档，不是汇报材料，不追求兼容性说明，不做保守修补，而是直接给出一套可执行的**重构目标架构 + 收敛路线图 + 分任务实施计划**。
>
> **重要前提**：**本方案不考虑向下兼容。**
>
> 这意味着：
>
> - 可以直接调整命令边界
> - 可以直接合并命令族
> - 可以直接修改阶段语义
> - 可以直接收紧或删除重复能力
> - 不为旧文档、旧心智、旧入口额外保留过渡层

---

## 1. 方案目标

本方案要解决的问题不是“现有系统能不能用”，而是：

> 如何把当前这套**方向正确但控制层偏胖**的治理系统，收敛成一套**主干清楚、命令分层清楚、规则真理源唯一、流程阶段语义稳定**的研发流程引擎。

本文采用方案 B 的结构：

1. **架构重构方案**：定义系统应该长成什么样
2. **路线图方案**：定义重构应该按什么顺序推进
3. **分任务实施计划**：定义每一步具体改什么、改到什么程度算完成

---

## 2. 现状判断

### 2.1 现状不是“设计错了”

`spec-first` 当前的核心方向没有问题：

- 规范驱动研发
- 全链路追踪
- Gate 自动校验
- AI 在结构化上下文中参与研发

这些都是项目成立的必要条件，不是附加复杂度。

### 2.2 现状是“主干对了，但控制层长胖了”

当前主要问题集中在 5 个方面：

1. **控制层命令重叠**
   - `catchup / status / plan / orchestrate`
   - `code-review / verify`
   - `spec-review / analyze`

2. **规则真理源分散**
   - Gate、Coverage、Layer、阈值规则分布在 skill、references、runtime、模板里

3. **阶段语义不稳定**
   - `test` 的叙事与阶段定义冲突最明显

4. **命令面膨胀**
   - 用户需要理解太多一级入口和中间层入口

5. **副作用节点过重**
   - `archive / doctor / sync` 承担了过多不同性质动作

### 2.3 重构目标不是增强，而是收敛

本次重构不以“继续加能力”为目标，而以“减少重复、减少壳层、减少规则分叉、减少心智负担”为目标。

这意味着：

- 保留治理主干
- 重做命令分层
- 删掉重复叙事
- 收紧节点边界
- 允许破坏式调整命令和语义

# 第一部分：架构重构方案

## 3. 重构原则

### 原则 1：主干能力不减

以下能力必须保留：

- Stage 状态机
- Traceability Matrix
- Gate 校验
- Coverage 体系
- Feature 工作区
- Skill/Context 注入

这些能力构成产品主干。

### 原则 2：规则只允许一个真理源

后续所有正式规则只允许存在于 runtime/config 中。

skill 文档只负责：

- 说明目的
- 说明输入输出
- 说明触发条件
- 说明调用哪些 runtime 能力

skill 文档不再完整复制：

- 阈值
- Gate 条件
- Layer 定义
- 各阶段完整例外规则

### 原则 3：一个节点只回答一个核心问题

目标模型如下：

- `spec`：要做什么
- `design`：怎么做
- `task`：拆成什么任务
- `code`：如何实现
- `review`：质量是否过关
- `verify`：是否能正式推进
- `archive`：如何沉淀与收口

任何一个节点如果回答超过 1 个主问题，就必须继续拆解或收缩。

### 原则 4：控制层必须轻

控制层不是交付主链，只能承担：

- 恢复
- 观察
- 决策
- 调度

不再承担：

- 大量规则解释
- 反复状态复述
- 过多运行态写入
- 替代主链节点做事

### 原则 5：不保留兼容层

本方案明确不考虑向下兼容，因此不引入：

- 双语义并存
- 旧命令长期别名
- 旧流程文档并列维护
- 过渡层 wrapper 命令

如果一个旧入口已经与新心智冲突，就直接收敛或删除。

## 4. 目标架构

## 4.1 五层架构

### L1. Truth Source Layer

职责：定义唯一正式规则。

内容：

- Stage Rules
- Gate Rules
- Coverage Rules
- Review Layer Rules
- Deliverable Rules

要求：

- 所有正式判断都读取这里
- 其他层不得复制定义

### L2. Runtime Atomic Layer

职责：执行确定性动作。

保留：

- `id`
- `matrix`
- `stage`
- `gate`
- `metrics`
- `feature`
- `rfc`
- `defect`
- `trace`
- `validate`

要求：

- 输入输出稳定
- 结果可测试
- 不承载高层流程叙事

### L3. Delivery Flow Layer

职责：承接交付主链。

重构后建议主链：

- `first`（项目认知入口，不属于 Feature 阶段）
- `init`
- `spec`
- `design`
- `research`（按需）
- `task`
- `code`
- `review`
- `verify`
- `archive`

这里的关键调整是：

- `code-review` 归并到 `review` 心智
- `test` 不再作为独立主链节点存在于现有语义中

### L4. Control Plane Layer

职责：流程辅助。

保留：

- `catchup`
- `status`
- `plan`
- `orchestrate`

要求：

- 轻量
- 不与主链争入口
- 不复制规则

### L5. Maintenance Layer

职责：维护、治理、环境侧动作。

保留：

- `doctor`
- `sync`
- `feature`
- setup/update/hooks/viewer 等宿主与治理工具

要求：

- 与主链分层展示
- 明确副作用
- 明确写入范围

## 4.1.1 `07_release` / `golive` 处置方案

当前系统真实存在 `07_release` 阶段，以及 `golive` 命令。因此 v2 不能只重构到 `archive` 就停止，而必须明确发布层归属。

### v2 决定

- `06_wrap_up` 之后仍保留 `07_release`
- `archive` 负责复盘沉淀，不负责上线批准
- `golive` 负责承接 `07_release` 的发布就绪检查与批准

### Canonical 收尾链路

为避免 `阶段`、`命令`、`阶段编号` 三套说法继续打架，v2 在这里给出唯一标准表达：

#### 阶段流

```text
... -> 05_verify -> 06_wrap_up -> 07_release -> 08_done
```

#### 命令流

```text
... -> verify -> archive -> golive -> done
```

解释：

- `archive` 是 `06_wrap_up` 阶段的主命令
- `golive` 是 `07_release` 阶段的主命令
- `done` 不是独立 skill，而是 `golive` 完成发布批准后的终态收口

后续所有 README、skill、references、流程图、状态面板都必须同时满足这组唯一映射，不再允许出现 `archive -> golive -> done` 与 `archive -> golive -> 07_release -> done` 两套并列表达。

### 职责边界

- `verify`：确认当前 Feature 是否具备进入收尾/发布准备阶段的条件
- `archive`：沉淀 retro、经验、风险、行动项
- `golive`：执行发布就绪检查、上线批准或阻断

### 分层归属

- `07_release` 是阶段状态机中的正式阶段
- `golive` 属于 **Delivery Flow Layer** 的发布收尾节点，而不是维护命令

## 4.2 目标命令心智模型

重构后，用户心智应简化为 4 个问题：

1. **我要推进交付** → 用主链命令
2. **我想看当前情况** → 用控制层命令
3. **我想做底层校验/修复/查询** → 用 runtime 命令
4. **我在处理环境或治理问题** → 用维护层命令

这比当前“命令很多但层级感不足”的状态更稳定。

## 4.3 主链重定义

### 新主链建议

```text
first(认知)
  -> init
  -> spec
  -> design
  -> research(按需)
  -> task(含测试设计)
  -> code(TDD执行)
  -> review
  -> verify
  -> archive
```

### TDD 集成原则

参考 superpowers 中的 TDD 最佳实践，测试不应被视为实现后的补充动作，而应被视为实现协议本身。

要点如下：

- **没有 failing test，不允许写生产代码**
- `task` 阶段必须前置定义测试意图，而不是只拆实现任务
- `code` 阶段必须采用 `RED -> GREEN -> REFACTOR` 循环
- `verify` 只负责验证测试覆盖和放行，不再承担“现在开始补测试”的入口角色

### TDD 集成后的主链职责

#### `task`

除了拆任务，还必须补齐每个任务的测试设计意图：

- 对应哪些 `FR / AC`
- 建议测试层级：`UT / IT / E2E / ST`
- 关键失败场景与边界场景
- 最小完成定义

#### `code`

`code` 不再只是“实现代码”，而是“按 TDD 协议完成实现”。

标准流程：

1. 写一个最小 failing test
2. 明确验证 RED，确认失败原因正确
3. 写最小实现使测试通过
4. 重新运行测试，验证 GREEN
5. 在保持 GREEN 的前提下做最小重构

#### `verify`

`verify` 不做测试设计，不做补测试入口，只负责：

- 检查测试资产是否存在
- 检查覆盖率是否达标
- 检查矩阵中的 `FR / AC / TC` 是否闭环
- 给出正式放行结论

### 为什么去掉独立 `test` 主链节点

当前 `test` 处于一个不稳定状态：

- 如果它是测试设计，应该前置
- 如果它是验证测试，应该归并进 `verify`

因此，继续保留“独立且阶段含义摇摆”的 `test` 节点，只会继续制造复杂度。

**v2 方案建议：直接取消现有 `test` 作为独立主链节点。**

替代方案：

- **测试设计能力**并入 `task`
- **TDD 执行能力**并入 `code`
- **验证测试能力**并入 `verify`

这意味着测试不再是“事后阶段”，而是：

- 在 `task` 阶段先定义
- 在 `code` 阶段按 TDD 执行
- 在 `verify` 阶段统一验收

这是一个明确的破坏式收敛。

## 4.4 Review 体系重定义

当前 review 体系问题在于“名字不同，但边界不稳定”。

v2 建议把 review 体系统一成两类：

### A. 专项审查（Specialized Review）

- `spec-review`
- `analyze`

### B. 交付审查（Delivery Review）

- `review`（承接原 `code-review`）
- `verify`

边界如下：

- `review`：质量是否过关
- `verify`：是否允许正式推进

这样用户不会继续混淆“审查”和“放行”。

---

## 4.5 Feature 管理重定义

`feature-list/current/switch` 不再作为分散一级入口存在。

统一收敛为：

- `feature list`
- `feature current`
- `feature switch`

Skill 层也不再单列三个独立心智节点，而是明确为同一命令族的不同动作。

## 4.6 维护类节点重定义

### `doctor`

目标：环境与宿主诊断入口

收缩方式：

- 项目诊断
- 宿主修复

二者可同命令承接，但在结果和确认阶段必须清楚分层。

### `sync`

目标：矩阵与运行态同步器

收缩方式：

- 明确输入是什么
- 明确写哪些文件
- 明确写入依据
- 明确审计记录

### `archive`

目标：复盘沉淀入口

删掉的职责：

- 不再主导阶段推进
- 不再承担所有运行态文件治理

保留的职责：

- `retro.md`
- 教训总结
- 指标收口
- 后续行动项

## 4.8 产物字段级变更方案

如果只改命令和叙事，不改产物字段，v2 很容易停留在概念层。以下字段调整是让新流程真正成立的关键。

### `task_plan.md` 新增字段

建议在任务明细中为每个 TASK 增加以下字段：

| 字段 | 类型 | 说明 | 用途 |
|---|---|---|---|
| `target_fr_ids` | string[] | 该任务直接覆盖的 FR | 明确测试与实现边界 |
| `target_ac_ids` | string[] | 该任务直接覆盖的 AC | 让 TDD 与 AC 对齐 |
| `recommended_test_levels` | string[] | 建议层级：UT/IT/E2E/ST | 指导测试设计 |
| `failure_cases` | string[] | 关键失败/边界场景 | 作为 RED 设计输入 |
| `definition_of_done` | string[] | 完成定义 | 作为 REVIEW/VERIFY 输入 |
| `test_intent` | string | 该任务的测试意图摘要 | 连接 task 与 code |

### `findings.md` 新增结构化字段

`findings.md` 需要区分“恢复/观察类内容”和“实现/验证证据”。建议至少增加：

| 字段 | 类型 | 说明 |
|---|---|---|
| `red_evidence` | object[] | failing test 命令、退出码、失败原因 |
| `green_evidence` | object[] | passing test 命令、退出码、关联 TASK |
| `gate_evidence` | object[] | gate/matrix/coverage 执行结果 |
| `decision_summary` | object | 当前计划/裁决摘要 |

### 字段落盘格式约定

为了避免字段设计停留在抽象层，v2 必须明确 Markdown 产物中的标准表示方式。

### `task_plan.md` 表格字段表示规则

由于当前 `task_plan.md` 以 Markdown 表格为主，新增字段不建议使用复杂嵌套 JSON。推荐规则如下：

- `string[]`：使用分号分隔
- `definition_of_done`：使用编号短句并以分号分隔
- `recommended_test_levels`：使用 `UT;IT;E2E;ST` 这种枚举列表
- `failure_cases`：只保留最关键的 2-5 项，不展开成长段描述

#### 推荐示例

| Task ID | 标题 | target_fr_ids | target_ac_ids | recommended_test_levels | failure_cases | definition_of_done | test_intent |
|---|---|---|---|---|---|---|---|
| TASK-AUTH-002 | 发送验证码 API | FR-AUTH-001 | AC-AUTH-001-01;AC-AUTH-001-02 | IT;UT | 频控触发;无效手机号;网关失败 | 返回成功;返回频控错误;记录审计日志 | 验证短信发送 happy path 与关键失败路径 |

### `findings.md` 结构化块表示规则

`findings.md` 不建议写成自由散文，也不建议写成完整 JSON 文件。推荐使用 Markdown 二级标题 + YAML 风格代码块。

#### 推荐示例

```markdown
## red_evidence

```yaml
- task: TASK-AUTH-002
  command: pnpm test tests/auth/send-otp.test.ts
  exit_code: 1
  reason: endpoint not implemented
```

## green_evidence

```yaml
- task: TASK-AUTH-002
  command: pnpm test tests/auth/send-otp.test.ts

- `verify` 从“补测试入口”收缩为“统一校验与放行入口”，方向正确

**仍需收紧**：

- 所有 `TASK` 都应默认具备最小测试意图；若是 spike / research 型任务，需显式标注免测原因
- `red_evidence` 与 `green_evidence` 需要在实施时变成 verify 可消费的结构，而不是只作为叙事证据

### 17.4 QA 视角

**结论**：v2 的验收框架比 v1 强很多，但 QA 仍要求所有“通过”都对应可机器检查的对象。

**认可点**：

- 已提出 `FR -> AC -> TASK -> TC` 闭环，这是 QA 真正关心的核心
- 已把 `findings.md` 从散文化日志拉向结构化证据
- 已把验收拆成结构、规则、产物、TDD、命令、用户心智、文档一致性七类，覆盖面足够

**仍需收紧**：

- 后续实施中，所有“已完成”定义都必须落到字段、命令输出或 gate 结果上
- `definition_of_done` 不能只作为 task 描述字段，必须最终映射到 review/verify 可检核项

### 17.5 运维 / 发布治理视角

**结论**：v2 已经把发布链补回来，但上线治理还需要更强的审计与回滚表达。

**认可点**：

- 已恢复 `07_release` 的正式地位
- 已确认 `archive` 不等于发布批准，边界正确
- 已将 `golive` 放回交付主链，而不是丢在维护层

**仍需收紧**：

- `golive` 的输入、输出、阻断条件、审计记录、回滚点应该在后续实施稿中单列
- `07_release` 阶段的 deliverables 不能只写“上线检查通过”，必须定义正式发布证据

### 17.6 产品 / 流程治理视角

**结论**：v2 已从“堆更多命令”转向“收敛主链”，这符合产品演进方向。

**认可点**：

- 文档的目标已从“功能增强”切换为“降低心智成本”
- 控制层、维护层、runtime、主链的用户问题划分更自然
- `research` 被收缩为按需分支，减少了流程僵化

**仍需收紧**：

- 后续对外总览必须把“用户应该何时进入哪一层”讲得更短更硬
- 不能再让 README 同时保留旧测试心智与新 TDD 心智

### 17.7 并行复审总判定

本轮多角色并行复审后的结论是：

- **架构方向：通过**
- **主链收敛方向：通过**
- **TDD 与测试节点重构方向：通过**
- **发布链完整性：通过，但需在实施稿补强发布证据模型**
- **命名收敛彻底性：本轮已修正为必须直接改名，不再允许双轨表述**

因此，当前 v2 可以视为：

> **一份已经达到“可以指导下一阶段实施设计”的重构基线文档。**

但它还不是最终实施稿。真正进入执行前，还应再产出一份更工程化的实施计划，专门回答：

- 每个 skill / command / runtime 文件具体改哪里
- 哪些旧文档要删除
- 哪些 gate / schema / 模板要同步改
- 哪些验证脚本和回归检查要新增

## 17. v2 结论

v2 方案的核心不是“再给出更多建议”，而是把方向彻底收敛成一句话：

> **保留治理主干，直接砍掉重叠壳层；保留追踪与放行，直接统一规则真理源；保留主链交付，直接重做控制层和 review 体系。**

如果按这个 v2 方案执行，`spec-first` 最终会从：

- 一个治理骨架很强、但控制层偏胖的系统

进化为：

- 一个主干稳定、规则统一、入口清楚、AI 可稳定消费的规范驱动研发流程引擎。

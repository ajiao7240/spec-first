# Spec-First v2 收敛重构 Implementation Plan

> **For Claude:** REQUIRED SUB-SKILL: Use superpowers:executing-plans to implement this plan task-by-task.
>
> **保存位置说明：** 按用户要求，本计划保存于 `docs/review-bundles/skill优化思考/`，而不是 `docs/plans/`。

**Goal:** 将 `spec-first` 从当前“`code-review / test / feature-*` 多入口并存、规则真理源分散、发布链表达不一致”的状态，重构为符合 v2 文档的单主链、TDD 内建、发布链完整、规则真理源统一的流程引擎。

**Architecture:** 本次实施采用“三段式收口”方案。第一段先做命名与主链收敛，直接解决 `review / test / feature-*` 的双轨问题；第二段补齐发布链、控制层和全域文档治理，避免 README 改了但细节文档没改；第三段抽出真正的 truth source，把技能映射、阶段主命令、gate/deliverables/thresholds 的正式定义统一到 runtime 可读的位置。

**Tech Stack:** TypeScript、Node.js、Vitest、CLI Router、Skill Runtime、Gate Engine、Layer Merger、Markdown Skills、Handlebars 模板。

---

## 0. 实施总规则

### 0.1 硬约束

- **不考虑向下兼容**：不保留 `code-review`、`test`、`feature-list/current/switch` 的长期别名。
- **不做双轨文档**：主 README、`AGENTS.md`、`SHARED.md`、skills、references、tests 必须同轮收口。
- **所有行为变更必须先 red test**：命名、主链、宿主同步、发布链、规则层都先写失败测试。
- **阶段流与命令流严格分离**：阶段流固定为 `05_verify -> 06_wrap_up -> 07_release -> 08_done`；命令流固定为 `verify -> archive -> golive -> done`。
- **真理源不止是 skill 名称映射**：必须同时覆盖 stage primary action、removed skills、gate rules、deliverables、thresholds、release evidence 要求。

### 0.2 建议执行顺序

1. 建立红线治理测试
2. 重命名 `review`
3. 删除独立 `test`
4. 合并 `feature` 家族
5. 清理全域旧命名文档
6. 补齐发布链与 release evidence
7. 抽规则真理源并接入 runtime
8. 收敛 control plane
9. 结构化 `task_plan.md` / `findings.md` / matrix 模板
10. 更新宿主同步链路
11. 做全量回归与 grep 审计

### 0.3 每个任务统一完成定义

- 失败测试已写出且失败原因正确
- 最小实现已完成
- 聚焦测试已转绿
- 受影响文档与模板已同步
- 旧命名在受影响范围内已无残留
- 若修改了 gate/deliverables/thresholds，则对应 runtime 测试已覆盖

# Spec-First 阶段自动流转可执行开发清单（修订版）

> **生成日期**: 2026-03-09
> **修订日期**: 2026-03-09
> **基线**: 当前仓库真实代码 + 深度架构审查
> **目标**: 在不重做现有状态机与 runtime notice 的前提下，补齐阶段自动流转主链能力
> **范围**: 仅覆盖”仍未完成开发”的事项，已完成能力不重复列入本轮实施
> **修订原则**: 保持职责分离、避免过度设计、渐进式实施

---

## 0. 当前基线结论

### 0.1 已完成，不纳入本轮

- `spec/design/task/code/review/plan/verify/spec-review` 的 runtime notice 已落地。
- `status` 是查询型 skill，不作为 consumer runtime 注入改造对象。
- `orchestrate` 的背景治理口径（`background_status / dependency_strength / risk_category / risk_signals`）已可用。

### 0.2 当前真正未完成的主线

- `StageState` 缺少自动流转元数据。
- `advance()` 写回 `stage-state.json` 时会丢掉扩展字段，连 `backgroundInputStatus` 也未保留。
- 缺少 `next-step-decider`。
- 缺少 `stage suggest` / `flow next`，以及面向 `orchestrate` 的 `auto-advance` 协调逻辑。
- `03_plan → 04_implement → 05_verify` 还没有打通真正的受控自动流转闭环。
- `catchup` 只恢复“任务上下文”，还没有恢复“背景质量状态”。

### 0.3 实施原则

- 不重做主阶段状态机。
- 不回退已经完成的 runtime notice 注入能力。
- **保持职责分离**：next-step-decider 做决策、advance 做执行、auto-loop 做循环、orchestrate 做协调。
- **避免过度设计**：不新增 hooks 抽象层、不在 auto-loop 中做阶段决策。
- P0 先交付”只读建议 + 状态可持久化”。
- P1 再做”决策层 + orchestrate 集成”。
- P2 做架构收口、低优先级补强与文档统一。

### P0-2 新增 `next-step-decider`

**目标**

- 把“下一步该做什么”的判断从 skill 文档和人工脑补中收口到统一决策层。

**文件**

- Create: `src/core/process-engine/next-step-decider.ts`
- Modify: `src/core/process-engine/advance.ts`
- Modify: `src/core/process-engine/dependency-checker.ts`
- Create Test: `tests/unit/next-step-decider.test.ts`

**实施内容**

1. 定义决策结果类型：
   - `BLOCKED` - 存在阻塞条件，无法推进
   - `SUGGEST_NEXT` - 建议执行下一步操作（只读建议）
   - `READY_TO_ADVANCE` - 满足推进条件，等待用户确认
   - `AUTO_ADVANCE` - 可自动推进（需用户配置启用）
   - `AUTO_RUN_NEXT_SKILL` - 可自动执行下一个 skill（需用户配置启用）
2. 决策输入至少覆盖：
   - `currentStage`
   - `stageStatus`
   - `autoAdvancePolicy`
   - `gateStatus`（来自 gate-evaluator）
   - `dependencyCheck`（来自 dependency-checker）
   - `todoState`（来自 todo-runner）
3. **职责边界**：
   - next-step-decider 只做决策，不执行任何写操作
   - 不与 dependency-checker 重复检查逻辑
   - 不与 gate-evaluator 重复评估逻辑
   - 只整合各检查结果，输出统一决策
4. 第一版只要求支持：
   - `01_specify → 02_design` 返回 `SUGGEST_NEXT`
   - `02_design → 03_plan` 返回 `SUGGEST_NEXT` 或 `READY_TO_ADVANCE`
   - `03_plan → 04_implement` 具备 `AUTO_ADVANCE` 候选能力
   - `04_implement → 05_verify` 具备 `AUTO_ADVANCE` / `AUTO_RUN_NEXT_SKILL` 候选能力
5. 第一版不自动推进 `05_verify → 06_wrap_up` 及之后阶段。

**验收标准**

- 存在统一决策入口。
- 相同输入得到稳定、可测试的输出。
- 决策逻辑不再散落在 skill 文档、CLI 文案和人工经验中。

**验证命令**

```bash
pnpm vitest run tests/unit/next-step-decider.test.ts tests/unit/advance.test.ts
```

### P1-1 orchestrate 协调入口集成决策层

**目标**

- 为当前 `orchestrate` 路径补上受控的阶段流转建议与可选自动推进。
- 保持 auto-loop 的单一职责（todo 执行循环），不在其中做阶段决策。

**文件**

- Create: `src/cli/commands/orchestrate.ts`
- Modify: `src/cli/index.ts`
- Modify: `src/core/ai-orchestrator/auto-loop.ts`（仅返回完成状态，不做阶段推进）
- Modify: `src/core/process-engine/next-step-decider.ts`
- Modify: `src/core/skill-runtime/orchestrate-args.ts`
- Modify: `skills/spec-first/13-orchestrate/SKILL.md`
- Create Test: `tests/unit/orchestrate-stage-integration.test.ts`
- Test: `tests/unit/orchestrate-args-parser.test.ts`
- Test: `tests/unit/auto-loop.test.ts`

**实施内容**

1. **orchestrate 协调入口增强**：
   - 新建或接入 `spec-first orchestrate` 协调入口
   - 在 auto-loop 完成后，调用 next-step-decider
   - 根据决策结果输出建议或执行操作
2. **参数协议同步**：
   - 在 `orchestrate-args.ts` 中显式支持 `--auto-advance`
   - 保持 `--auto` / `--resume` 兼容
   - 在 `13-orchestrate` skill 文档中同步参数与默认安全策略
3. **auto-loop 职责明确**：
   - 只负责执行 todo 列表中的任务
   - 返回完成状态（all_done / has_blocked / timeout 等）
   - 不调用 `advance()`，不做阶段决策
4. **决策到执行的流程**：
   - orchestrate 检测 auto-loop 返回 `all_done`
   - 调用 next-step-decider 获取决策
   - 如果是 `READY_TO_ADVANCE` 且用户启用了 `auto-advance`，调用 `advance()`
   - 否则输出建议，等待用户手动执行
5. **安全保障**：
   - 自动推进前必须通过 Gate 检查
   - 自动推进前必须通过依赖检查
   - 有任何 blocked todo 时不自动推进

**验收标准**

- orchestrate 能在 todo 完成后给出阶段推进建议。
- auto-loop 不包含任何阶段推进逻辑。
- 默认模式下只建议，不自动执行。
- 启用 `auto-advance` 后能安全地自动推进阶段。
- 参数解析、skill 文档、测试三者保持一致。

**验证命令**

```bash
pnpm vitest run tests/unit/orchestrate-stage-integration.test.ts tests/unit/orchestrate-args-parser.test.ts tests/unit/auto-loop.test.ts
```

**实施内容**

1. **next-step-decider 增强**：
   - 当 `03_plan` 的前置条件满足时，返回 `READY_TO_ADVANCE`
   - 当 `04_implement` 所有 TASK `done`、无 `blocked`、Gate 通过时，返回 `READY_TO_ADVANCE` 或 `AUTO_ADVANCE`
   - 当 `05_verify` 验收通过时，返回 `READY_TO_ADVANCE`
2. **orchestrate 协调**：
   - 在每个阶段的 todo 完成后，调用 next-step-decider
   - 根据 autoAdvancePolicy 决定是建议还是自动执行

   - 对 `blocked` / `timeout` / `retry exhausted` 必须显式降级回 `BLOCKED`
   - 输出明确的阻塞原因和修复建议
   - 不允许在有阻塞的情况下推进阶段
4. **现有 hooks 协调**：
   - 在 orchestrate 中调用现有的 task-context.sh / progress-sync.sh
   - 不新建 hooks 抽象层，直接使用现有机制

**验收标准**

- `task/code/verify` 阶段形成稳定的受控流转闭环。
- 有阻塞时绝不误推进阶段。

## 5. 架构审查结论

### 5.1 关键修正点

1. **sanitizeStageState 修复方式**：
   - ❌ 原方案：删除 sanitize 机制
   - ✅ 修正方案：在白名单中添加字段，保持类型安全

2. **阶段流转实现方式**：
   - ❌ 原方案：在 auto-loop.ts 中调用 advance()
   - ✅ 修正方案：orchestrate 协调 auto-loop + next-step-decider + advance

3. **hooks 抽象**：
   - ❌ 原方案：新建 stage-hooks 抽象层
   - ✅ 修正方案：在 orchestrate 中协调现有 hooks，不新增抽象

### 5.2 职责分离原则

- **next-step-decider**: 决策”是否可以/应该推进”（只读）
- **advance**: 执行阶段推进（含 Gate/依赖检查/审计）
- **auto-loop**: 执行 todo 循环（不做阶段决策）
- **orchestrate**: 协调上述组件（决策 → 建议/执行）

### 5.3 风险控制

- P0 只做只读能力，零风险
- P1 自动执行默认关闭，需用户显式启用
- 所有自动推进前必须通过 Gate + 依赖检查
- 有任何阻塞时不允许推进

### 5.4 实施建议

- 严格按 P0 → P1 → P2 顺序实施
- P0 完成后充分测试再进入 P1
- P1-1 和 P1-2 可并行开发
- P2 可根据实际需求调整优先级

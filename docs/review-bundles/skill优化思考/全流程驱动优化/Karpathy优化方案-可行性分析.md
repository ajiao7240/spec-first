# Karpathy 优化方案可行性分析

> 基于 spec-first 当前实现的深度评估

**分析日期**: 2026-03-09
**当前 spec-first 版本**: 1.0.0
**总 Skill 文件行数**: 5426 行

---

## 执行摘要

### 核心发现

**spec-first 已有强大的守卫机制**，与 Karpathy 准则有 **60-70% 的重叠**：

| Karpathy 原则 | spec-first 现状 | 重叠度 | 优化价值 |
|--------------|----------------|--------|---------|
| Think Before Coding | ✅ 有歧义消解、澄清机制 | 70% | 中等 |
| Simplicity First | ❌ 缺少明确约束 | 20% | **高** |
| Surgical Changes | ❌ 缺少边界守卫 | 10% | **高** |
| Goal-Driven Execution | ✅ 有 TDD、证据铁律 | 80% | 低 |

### 关键结论

1. **不需要大规模重写** - spec-first 架构已经很好
2. **需要精准补充** - 在 2-3 个关键点增加守卫
3. **避免冗余** - 不要重复现有机制

## 当前实现优势分析

### 1. 已有的强大机制

#### 1.1 字面即精神原则（Literal is Spirit）

**位置**: `SHARED.md` + 各 Skill

**功能**: 防止 AI 合理化跳过流程

**评估**: ✅ **非常强大**，已覆盖 Karpathy 的"不隐藏困惑"

```markdown
| AI 的借口 | 封堵 |
|-----------|------|
| "我理解核心思想，可以灵活执行" | 字面规则的违反就是精神的违反 |
| "这是精神而非仪式" | 仪式（字面规则）是精神的体现 |
```

**结论**: 无需优化，已经足够。

#### 1.2 反合理化守卫表

**位置**: `SHARED.md` + `07-code/SKILL.md`

**功能**: 防止 AI 跳过测试、review、调试

**评估**: ✅ **覆盖全面**，已有 9 条 code 专属守卫

**与 Karpathy 对比**:
- ✅ 已有"先写完再补测试" → 对应 Goal-Driven
- ✅ 已有"快速修一下" → 对应 Systematic Debugging
- ❌ **缺少**"顺手优化不相关代码" → Surgical Changes

**结论**: 需要补充 Surgical Changes 守卫。

#### 1.3 结构化歧义消解

**位置**: `03-spec/SKILL.md`

**功能**: 强制标记 `[NEEDS CLARIFICATION]`，动态生成澄清问题

**评估**: ✅ **已实现 Karpathy 的 Think Before Coding**

**现有机制**:
- 10 种歧义分类标签（BOUNDARY, ERROR, PRIORITY...）
- 5 步动态问题生成流程
- 最多 5 轮澄清，每轮最多 3 问

**与 Karpathy 对比**:
- ✅ 已有"不确定时停下来问" → 完全对应
- ✅ 已有"呈现多种候选" → 每个问题提供 2-3 个候选
- ❌ **缺少**"假设显性化" → 未强制列出隐含假设

**结论**: 需要补充"假设显性化"机制。

#### 1.4 TDD 铁律 + 证据铁律

**位置**: `07-code/SKILL.md`

**功能**: 强制 TDD RED 验证，强制新鲜证据

**评估**: ✅ **已实现 Karpathy 的 Goal-Driven Execution**

**现有机制**:
- "NO PRODUCTION CODE WITHOUT A FAILING TEST FIRST"
- "NO FIXES WITHOUT ROOT CAUSE INVESTIGATION FIRST"
- 3-Strike Error Protocol（3 次失败必须升级）

**结论**: 无需优化，已经足够强大。

## 关键调整建议

### 1. 避免重复现有机制

**问题**: 原优化方案中有 4 项与现有机制重叠度 >70%

**调整**:
- ❌ 删除"目标驱动实现"（TDD 铁律已覆盖）
- ❌ 删除"多解释呈现守卫"（动态澄清已覆盖）
- ❌ 删除"编排简洁性守卫"（收益太小）
- ⚠️ 暂缓"方案呈现守卫"（与 ADR 关系需明确）

### 2. 精简守卫表内容

**问题**: 原方案每个守卫表有 5-6 条规则，过于冗长

**调整**: 每个守卫表最多 3-4 条核心规则

**示例**:
```markdown
### 反过度工程守卫（精简版）
| AI 的借口 | 封堵 |
|-----------|------|
| "这个抽象以后可能用得上" | YAGNI - 未来需求不是当前 TASK |
| "加个配置项更灵活" | TASK 未要求灵活性 = 不需要 |
| "封装一下更优雅" | 单次使用不需要封装 |
```

### 3. 利用 SHARED.md 机制

**问题**: 原方案在每个 Skill 中重复定义守卫

**调整**: 将通用守卫放入 `SHARED.md`，各 Skill 引用

**示例**:
```markdown
# SHARED.md 新增章节

## 简洁性守卫（Simplicity First）

### 通用原则
- 最小代码解决问题
- 不添加未要求的功能
- 不为单次使用创建抽象

**应用 Skills**: 07-code, 04-design
```

### 4. 与现有 references/ 目录整合

**问题**: 原方案新增多个 reference 文档

**调整**: 整合到现有文档中

**示例**:
- `simplicity-checklist.md` → 整合到 `code-standards.md`
- `assumption-template.md` → 整合到 `prd-template-*.md`

## 核心结论

### 关键发现

1. **spec-first 已经很强** - 60-70% 的 Karpathy 准则已被覆盖
2. **需要精准补充** - 只需在 2-3 个关键点增加守卫

| 优化 | 为什么重要 | 预期收益 |

| **代码修改边界守卫** | spec-first 完全缺失 | 减少 40% 超范围修改 |

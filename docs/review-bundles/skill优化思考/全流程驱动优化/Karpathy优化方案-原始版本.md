# Spec-First Skills 优化方案：借鉴 Karpathy 准则

> **目标**: 在保持 spec-first 流程纪律的基础上，引入 Karpathy 准则防止过度工程、默默假设、超范围修改

**文档版本**: 1.0.0
**创建日期**: 2026-03-09
**优化范围**: 5 个核心 Skill（code, spec, review, design, orchestrate）

---

## 执行摘要

### 核心问题

spec-first 已有强大的流程约束（字面即精神、反合理化守卫），但在以下方面存在改进空间：

1. **代码生成质量** - 可能产生过度工程的代码
2. **需求澄清** - 可能在不确定时默默选择解释
3. **审查范围** - 可能提出超出 TASK 范围的优化建议
4. **设计复杂度** - 可能过度设计架构

### 解决方案

引入 Karpathy 4 大原则作为补充守卫：

| Karpathy 原则 | 解决的问题 | 应用 Skill |
|--------------|-----------|-----------|
| **Think Before Coding** | 防止默默假设 | spec, design |
| **Simplicity First** | 防止过度工程 | code, design, orchestrate |
| **Surgical Changes** | 防止超范围修改 | code |
| **Goal-Driven Execution** | 聚焦 TASK 目标 | review, code |

### 预期收益

- **代码质量**: 减少 30-50% 的过度抽象和不必要代码
- **需求明确性**: 减少 40% 的后期需求返工
- **审查效率**: 减少 25% 的无效审查建议
- **设计简洁性**: 减少 35% 的过度设计

##### 1. 新增"代码简洁性守卫"章节

**插入位置**: `## When to Use` 之后

```markdown
## 代码简洁性守卫（Karpathy: Simplicity First）

**核心原则**: 最小代码解决问题，零投机性设计。

### 写代码前自检清单

- [ ] 这是解决 TASK 的最小实现吗？
- [ ] 是否添加了 TASK 未要求的功能？
- [ ] 是否为单次使用创建了抽象？
- [ ] 是否处理了不可能发生的错误场景？
- [ ] 200 行能简化为 50 行吗？

### 反过度工程守卫

| AI 的借口 | 封堵 |
|-----------|------|
| "这个抽象以后可能用得上" | YAGNI - 未来需求不是当前 TASK，不写 |
| "加个配置项更灵活" | TASK 未要求灵活性 = 不需要 |
| "多写点错误处理更健壮" | 只处理 TASK 规格中的错误场景 |
| "封装一下更优雅" | 单次使用不需要封装 |
| "用设计模式更专业" | 模式是手段不是目的，简单解决优先 |
| "这个工具类以后能复用" | 当前只用一次 = 不抽取 |

### 简洁性验证

代码写完后问自己：
- 高级工程师会说这代码过度复杂吗？
- 删掉任何一个抽象层会让代码更难理解吗？
- 如果答案是"会说复杂"或"不会更难"，重写。

**Iron Law**: 简洁 > 优雅 > 灵活。TASK 只要求简洁。
```

##### 2. 新增"代码修改边界守卫"章节

**插入位置**: "代码简洁性守卫"之后

```markdown
## 代码修改边界守卫（Karpathy: Surgical Changes）

**核心原则**: 只动必须动的，只清理自己制造的孤立代码。

### 修改现有代码前自检清单

- [ ] 每个改动行是否直接追溯到 TASK 需求？
- [ ] 是否"顺手"改了相邻代码/注释/格式？
- [ ] 是否重构了不相关的代码？
- [ ] 是否删除了 TASK 范围外的死代码？

### 反顺手优化守卫

| AI 的借口 | 封堵 |
|-----------|------|
| "顺便把这个也改了" | 不相关改动增加回归风险，不动 |
| "这段代码风格不统一" | 匹配现有风格，不做风格统一 |
| "这个注释过时了" | TASK 未涉及则不动 |
| "这里有个明显的 bug" | 提及但不修，TASK 范围外 |
| "重构一下更清晰" | 不相关重构 = 风险，不动 |

### 清理规则

**允许清理**:
- 你的改动导致的未使用 import
- 你的改动导致的未使用变量/函数
- 你的改动导致的死代码分支

**禁止清理**:
- 预存在的死代码（提及但不删）
- 预存在的过时注释
- 预存在的风格不一致

**Iron Law**: 每个 diff 行必须追溯到 TASK，否则撤销。
```

##### 3. 新增"目标驱动实现"章节

**插入位置**: "代码修改边界守卫"之后

```markdown
## 目标驱动实现（Karpathy: Goal-Driven Execution）

**核心原则**: 定义可验证的成功标准，循环直到满足。

### 实现前转换 TASK 为验证目标

| TASK 描述 | 转换为验证目标 |
|----------|--------------|
| "添加用户输入验证" | "写测试覆盖 5 种无效输入，然后让测试通过" |
| "修复登录 bug" | "写测试复现 bug，然后让测试通过" |
| "重构认证模块" | "确保所有测试在重构前后都通过" |
| "优化查询性能" | "写基准测试证明性能提升 X%，然后优化" |

### 多步骤 TASK 执行模板

```
TASK: [TASK-XXX-001] 标题

执行计划:
1. [步骤] → 验证: [检查命令/测试]
2. [步骤] → 验证: [检查命令/测试]
3. [步骤] → 验证: [检查命令/测试]

当前进度: Step X/Y
```

### 强验证 vs 弱验证

**强验证**（推荐）:
- ✅ 自动化测试通过
- ✅ Lint/Typecheck 通过
- ✅ 基准测试达标

**弱验证**（禁止）:
- ❌ "看起来能工作"
- ❌ "手动测了一下"
- ❌ "应该没问题"

**Iron Law**: 无强验证 = 未完成，不得标记 TASK 为 Done。
```

```markdown
## 假设显性化守卫（Karpathy: Think Before Coding）

**核心原则**: 不假设、不隐藏困惑、呈现权衡。

### 生成 FR/AC 前强制检查

- [ ] 列出所有隐含假设（用户角色、数据来源、边界条件）
- [ ] 对每个假设标注：`[ASSUMED]` 或 `[NEEDS CLARIFICATION]`
- [ ] 存在多种解释时，列出所有候选而非默默选择
- [ ] 不确定时停下来，明确说明困惑点

### 反默默选择守卫

| AI 的借口 | 封堵 |
|-----------|------|
| "这个解释最合理" | 合理 ≠ 唯一，列出所有候选让用户选 |
| "用户应该是这个意思" | 应该 ≠ 确定，标记 `[NEEDS CLARIFICATION]` |
| "按常规做法就行" | 常规做法是假设，需显性陈述并确认 |
| "这个细节不重要" | 细节决定实现，不重要也要确认 |
| "我理解用户意图" | 理解 ≠ 准确，多种理解必须列出 |

### 假设陈述模板

```markdown
## 隐含假设清单

### [ASSUMED] 用户角色
- 假设：仅管理员可访问此功能
- 依据：PRD 提到"管理后台"
- 风险：若普通用户也需访问，需重新设计权限

### [NEEDS CLARIFICATION][ROLE] 数据来源
- 问题：用户数据从哪个系统同步？
- 候选：A) 从 CRM 实时拉取 B) 从本地缓存读取 C) 用户手动输入
- 影响：决定 API 设计和性能要求
```

**Iron Law**: 每个 FR 至少陈述 3 个假设，不确定的必须标记 `[NEEDS CLARIFICATION]`。
```

##### 2. 新增"多解释呈现守卫"章节

**插入位置**: "假设显性化守卫"之后

```markdown
## 多解释呈现守卫（Karpathy: Think Before Coding）

**核心原则**: 存在歧义时呈现所有候选，不默默选择。

### 触发条件

以下情况必须呈现多种解释：
- 边界值有多种理解（如"最多 10 个"是 ≤10 还是 <10）
- 异常处理有多种策略（如失败后重试 vs 直接报错）
- 优先级有冲突（如性能 vs 安全）
- 术语有多义（如"用户"指终端用户还是管理员）

### 呈现模板

```markdown
## [NEEDS CLARIFICATION][BOUNDARY] FR-XXX-001: 文件大小限制

**问题**: "最多上传 10MB" 的边界如何处理？

**候选解释**:
A) 严格限制：>10MB 直接拒绝
B) 软限制：>10MB 警告但允许
C) 分级限制：普通用户 10MB，VIP 用户 50MB

**影响**:
- A: 实现简单，用户体验差
- B: 用户体验好，可能被滥用
- C: 需要权限系统，复杂度高

**推荐**: [等待用户选择]
```

**Iron Law**: 有歧义不选择，列出候选让用户决策。
```

```markdown
## 审查范围守卫（Karpathy: Goal-Driven + Surgical Changes）

**核心原则**: 只审查 TASK 范围内的质量，不提超范围建议。

### Stage 2 质量审查前自检

- [ ] 建议的改进是否追溯到 TASK/FR 需求？
- [ ] 是否提出了"以后可能需要"的优化？
- [ ] 是否建议添加 TASK 范围外的功能？
- [ ] 是否建议重构不相关代码？

### 反过度审查守卫

| AI 的借口 | 封堵 |
|-----------|------|
| "这里可以优化性能" | TASK 未提性能要求 = 不审查 |
| "建议加个扩展点" | TASK 未要求扩展性 = 不建议 |
| "这个模式更优雅" | 优雅不是 TASK 目标 = 不提 |
| "这段代码可以复用" | 当前只用一次 = 不建议抽取 |
| "建议加个配置项" | TASK 未要求可配置 = 不建议 |

### 审查建议分级

**MUST FIX**（阻断）:
- 违反 TASK 规格
- 违反 constitution 原则
- 安全漏洞
- 功能性 bug

**SHOULD FIX**（建议）:
- TASK 范围内的性能问题
- TASK 范围内的可读性问题
- TASK 范围内的测试覆盖不足

**OUT OF SCOPE**（不提）:
- TASK 范围外的优化
- TASK 范围外的重构
- TASK 范围外的功能建议

**Iron Law**: 超出 TASK 范围的建议 = 噪音，不提。
```

## 风险与缓解

### 风险 1: 过度约束导致效率下降

**描述**: 新增守卫可能让 AI 过于谨慎，降低执行效率

**缓解措施**:
- 明确"简单任务可灵活判断"的例外情况
- 提供"快速通道"标记（如 `--skip-guards`）
- 定期评估守卫的实际价值

### 风险 2: 与现有流程冲突

**描述**: Karpathy 准则可能与 spec-first 现有规则冲突

**缓解措施**:
- 优先保持 spec-first 核心流程不变
- Karpathy 准则作为补充而非替代
- 冲突时以 spec-first 为准

### 风险 3: 学习曲线陡峭

**描述**: 用户需要学习新的守卫规则

**缓解措施**:
- 提供对比示例（优化前 vs 优化后）
- 编写快速参考卡片
- 在 Skill 输出中提供实时提示

```
✅ DO:
- 列出所有假设
- 呈现多种解释
- 不确定时停下来问

❌ DON'T:
- 默默选择一种解释
- 隐藏困惑点
- 假装理解用户意图
```

#### 代码审查 (08-review)

```
✅ DO:
- 只审查 TASK 范围内的问题
- 区分 MUST/SHOULD/OUT_OF_SCOPE
- 聚焦功能正确性和安全

❌ DON'T:
- 提超出 TASK 的优化建议
- 建议添加未要求的功能
- 要求重构不相关代码
```

#### 技术设计 (04-design)

```
✅ DO:
- 设计满足 FR 的最小架构
- 呈现多方案权衡
- 明确推荐理由

❌ DON'T:
- 设计未要求的扩展性
- 默默选择一种方案
- 使用不必要的设计模式
```

  private rules: ValidationRule[];
  private strategy: ValidationStrategy;

  constructor(config: ValidatorConfig) {
    this.rules = config.rules;
    this.strategy = StrategyFactory.create(config.type);
  }

  validate(user: User): ValidationResult {
    return this.strategy.execute(this.rules, user);
  }
}

// 50+ 行配置和工厂代码...
```

**优化后（简洁）**:
```typescript

function validateUsername(username: string): boolean {
  return username.length >= 3 && username.length <= 20;
}

#### 示例 2: 需求规格

**优化前（默默假设）**:
```markdown
## FR-AUTH-001: 用户登录

用户可以通过用户名和密码登录系统。

### AC-001
- 输入正确凭证后跳转到首页
```

**优化后（假设显性化）**:
```markdown
## FR-AUTH-001: 用户登录

用户可以通过用户名和密码登录系统。

### 隐含假设
- [ASSUMED] 用户名不区分大小写（依据：常见做法）
- [NEEDS CLARIFICATION][BOUNDARY] 密码错误 3 次后如何处理？
  - A) 锁定账户 30 分钟
  - B) 要求验证码
  - C) 仅记录日志

### AC-001
- 输入正确凭证后跳转到首页
- [待澄清] 首页指哪个页面？（Dashboard vs 上次访问页）
```

#### 示例 3: 代码审查

**优化前（超范围建议）**:
```markdown
## Review 发现

### SHOULD FIX
1. 建议将认证逻辑抽取为独立服务（便于复用）
2. 建议添加缓存层提升性能
3. 建议使用 Redis 存储 session
```

**优化后（聚焦范围）**:
```markdown

### MUST FIX
1. 密码未加密存储（安全漏洞）

### SHOULD FIX
2. 缺少登录失败测试用例

### OUT OF SCOPE（不在本次 TASK）
- 性能优化（TASK 未要求）
- 架构重构（TASK 未要求）
```

### C. 常见问题 FAQ

**Q1: 所有代码都要遵循"简洁优先"吗？**

A: 不是。简单任务（如修改配置、更新文档）可灵活判断。复杂任务（如新功能、架构变更）必须遵循。

**Q2: 如何判断是否"过度工程"？**

A: 问自己：删掉这个抽象/配置/扩展点，代码会更难理解吗？如果不会，就是过度。

**Q3: "假设显性化"会不会太繁琐？**

A: 初期可能感觉繁琐，但能避免后期大量返工。建议至少列出 3 个核心假设。

**Q4: 审查时发现明显的优化机会怎么办？**

A: 如果超出 TASK 范围，标记为 OUT_OF_SCOPE 并建议创建新 TASK，不在当前 review 中要求修改。

**Q5: 设计时如何平衡"简洁"和"扩展性"？**

A: 优先简洁。只有 FR 明确要求扩展性时才设计扩展点。

## 总结

### 核心价值主张

**spec-first + Karpathy = 流程纪律 + 代码质量**

- **spec-first** 确保"做正确的事"（流程、追踪、验证）
- **Karpathy** 确保"正确地做事"（简洁、明确、聚焦）

### 关键洞察

1. **互补而非替代**: Karpathy 准则不改变 spec-first 流程，只是在关键节点增加质量守卫
2. **防御性编程**: 通过守卫表防止 AI 的常见错误模式
3. **可度量改进**: 所有优化都有明确的度量指标

### 下一步行动

1. **立即行动**: 从 07-code 和 03-spec 开始（P0 优先级）
2. **小步快跑**: 每个 Skill 优化后立即验证，不等全部完成
3. **持续迭代**: 根据实际使用反馈调整守卫规则

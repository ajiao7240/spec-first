# 2026-03-07 20-spec-review skill 审查与优化建议

## 1. skill 要解决什么问题

`20-spec-review` 解决的是：

1. 在 `spec` 完成后，对规格质量做专项审查
2. 用统一清单给出 C10 质量分
3. 在进入 `design` 前尽早发现规格缺陷

一句话：

> `20-spec-review` 解决“这份规格文档是否足够清晰、可测、可进入设计阶段”。

## 2. 当前实现是否合理

整体判断：方向正确，而且聚焦明确。

### 2.1 合理的地方

1. 它只审 `spec.md`，边界清楚
2. 在 `01_specify` 阶段执行很合理
3. C10 作为规格质量分，对早期治理有价值

## 3. 当前主要问题

### 3.1 定义过轻

它基本只定义了：

- 清单来源
- 写入路径
- C10 输出

但没有更细地说明：

- 常见问题严重度
- 什么问题必须阻断
- 修复后如何复查

### 3.2 C10 口径过于依赖 checklist 计数

如果只按 `checked/total` 算百分比，  
很容易把不同严重度的问题压平成同一个分值。

### 3.3 它天然属于 review 命令族

单独存在没问题，但从用户心智看，它应被明确归入：

- `review spec`

而不是孤立 skill。

## 4. 推荐优化方向

### 4.1 保留 spec 专项审查，但补严重度机制

建议在 C10 之外增加：

- `CRITICAL / HIGH / MEDIUM / LOW`

这样 C10 不再是唯一结论。

### 4.2 把它收进 review 家族心智

短期不改命令也可以，  
但文档上应明确它和 `code-review / analyze` 是同一类审查能力。

## 5. 当前审查结论

- 职责定位：正确
- 当前合理性：中高
- 主要问题：过轻，C10 口径过平
- 最佳优化方向：保留 spec 专项审查，但补严重度和复查规则

一句话结论：

> `20-spec-review` 是必要命令，但还像“轻量清单器”；下一步要把它提升为真正的规格质量关口。

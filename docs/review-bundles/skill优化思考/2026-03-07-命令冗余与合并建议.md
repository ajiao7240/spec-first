# 2026-03-07 命令冗余与合并建议

## 1. 为什么要看命令冗余

当前 `spec-first` 不只是 skill 多，用户可感知的“命令面”也已经变宽。

这会带来 3 个问题：

1. 新用户不知道从哪个入口开始
2. 多个命令都在回答“现在该做什么”
3. 一些命令职责相邻，但边界对用户并不直观

所以这一步要回答的不是“能不能合并”，而是：

> 合并之后，是否能让用户更少思考命令名，更专注当前目标。

## 2. 当前命令面按用户目标分组

如果从用户目标看，当前命令大致分成 6 组：

### 2.1 入口与认知

- `first`
- `onboarding`
- `init`

### 2.2 主链交付

- `spec`
- `design`
- `research`
- `task`
- `code`
- `code-review`
- `test`
- `archive`

### 2.3 规划与推进

- `plan`
- `verify`
- `orchestrate`

### 2.4 质量审查与分析

- `spec-review`
- `analyze`

### 2.5 会话恢复与状态查看

- `catchup`
- `status`

### 2.6 Feature 管理与维护

- `feature-list`
- `feature-switch`
- `feature-current`
- `sync`
- `doctor`

从系统内部看，这样拆没问题。  
但从用户心智看，已经出现了明显的冗余区。

## 3. 最明显的冗余区

### 3.1 `catchup` / `status` / `plan`

这 3 个命令都在不同角度回答：

- 现在在哪
- 有什么阻塞
- 下一步做什么

区别在于：

- `catchup` 偏会话恢复
- `status` 偏状态快照
- `plan` 偏下一步建议

但对普通用户来说，这三者非常容易混成一类“项目现状命令”。

### 3.2 `feature-list` / `feature-current` / `feature-switch`

这 3 个命令本质上属于同一个心智模型：

- 查看有哪些 Feature
- 当前在哪个 Feature
- 切到哪个 Feature

从功能上分开是合理的，  
但从命令面看，更适合作为一个 `feature` 命令的子动作，而不是 3 个一级入口。

### 3.3 `plan` / `orchestrate`

这两个命令都在回答：

- 下一步怎么走
- 要不要推进
- 要不要执行某个 skill

`plan` 偏“想”
`orchestrate` 偏“做”

这个边界在架构上成立，但在用户心智上并不天然清晰。  
如果不做更强约束，它们就会变成两个相邻但容易混淆的命令。

### 3.4 `spec-review` / `analyze`

这两个命令都在做只读质量审查：

- 一个偏 `spec`
- 一个偏跨产物

从系统内部看分开合理；  
但从用户视角看，它们都像“审查报告命令”。

## 4. 哪些不应合并

有些命令虽然相邻，但不建议合并。

### 4.1 `spec` / `design`

这两个必须分开。  
一个定义 `WHAT`，一个定义系统级 `HOW`，这是整条链路最重要的边界之一。

### 4.2 `code` / `code-review` / `verify`

这三个也不建议粗暴合并。

因为它们分别对应：

- 实现
- 实现质量审查
- 阶段放行

如果合并成一个“大而全”的实现命令，治理边界会直接塌掉。

### 4.3 `init` / `spec`

也不建议合并。  
`init` 解决“我要从哪里开始”，`spec` 解决“需求怎么定义”，这两个边界已经比较清晰。

## 5. 推荐的合并方案

我建议采用“中等收敛方案”，而不是最激进的整并。

### 5.1 推荐保留的一级主链命令

这些一级命令建议继续保留：

- `first`
- `init`
- `spec`
- `design`
- `research`
- `task`
- `code`
- `code-review`
- `test`
- `verify`
- `archive`

原因是它们大多对应真实阶段或真实质量关口。

### 5.2 推荐合并为单一入口的命令组

#### A. Feature 管理合并

建议合并为：

- `feature`

子动作：

- `feature list`
- `feature current`
- `feature switch`

这实际上更符合用户直觉，也和 CLI 现有心智一致。

#### B. 状态与恢复合并

建议合并为：

- `context`

子动作：

- `context status`
- `context catchup`
- `context next`

如果不想新增命令，也至少应在产品语义上把：

- `catchup`
- `status`
- `plan`

收敛成一个命令族。

#### C. 审查与分析合并

建议合并为：

- `review`

子动作：

- `review spec`
- `review code`
- `review analyze`

如果短期不想改命令名，也建议至少在文档上明确它们是同一族能力。

## 6. 不建议现在合并，但建议做“主次入口重构”的命令

### 6.1 `onboarding`

不建议直接删除。  
但它不应长期和 `first` 平级竞争入口。

更合理的做法是：

- `onboarding` 作为统一角色入口
- `first` 成为研发/架构认知分支

### 6.2 `plan`

不建议立即并入 `orchestrate`。  
但应从“用户常用一级命令”降级成“控制层辅助命令”。

换句话说：

- 普通用户更多看到 `status/catchup/orchestrate`
- `plan` 更偏高级规划工具

### 6.3 `sync` / `doctor`

这两个都偏维护类命令，不建议并入主链。  
但可以在文档上归到同一类“维护命令”下面，减少主链干扰。

## 7. 一个更合理的最终命令面

如果从最佳实践看，我更推荐最后收敛成 5 组用户心智：

### 7.1 入口组

- `onboarding`
- `first`
- `init`

### 7.2 交付组

- `spec`
- `design`
- `research`
- `task`
- `code`
- `code-review`
- `test`
- `verify`
- `archive`

### 7.3 上下文组

- `context status`
- `context catchup`
- `context next`

### 7.4 Feature 组

- `feature list`
- `feature current`
- `feature switch`

### 7.5 维护组

- `sync`
- `doctor`

### 7.6 高级控制组

- `plan`
- `orchestrate`
- `review spec`
- `review analyze`

## 8. 最推荐的具体收敛动作

优先级建议如下：

1. 先合并 `feature-list/current/switch` 为 `feature` 命令族
2. 再把 `catchup/status/plan` 收敛成一组上下文命令
3. 再把 `spec-review/analyze` 收敛成审查命令族
4. 暂不合并 `code/code-review/verify`
5. 暂不合并 `spec/design/task`

## 9. 当前结论

可以合并，而且有必要。  
但最应该合并的不是主链阶段命令，而是那些对用户来说都在回答“我现在在哪、要去哪、当前 Feature 是什么”的命令。

一句话结论：

> 当前命令面真正的冗余不在主链，而在“上下文/状态/Feature 管理/审查入口”这几组命令；最佳实践是保留阶段命令，收敛辅助命令。

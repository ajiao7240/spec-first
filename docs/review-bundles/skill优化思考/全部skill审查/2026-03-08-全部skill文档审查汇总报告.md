## 最佳实践方案

### 方案目标

把现在“文档已声明、代码已局部实现、注入未统一”的状态，收敛成：

> **一套共享背景解析层 + 一套共享 notice 生成层 + 一张 skill→context 映射表。**

### 推荐方案

#### 1. 抽公共背景解析器

建议新增类似：

- `src/core/skill-runtime/background-context.ts`

统一提供：

- `backgroundStatus`
- `dependencyStrength`
- `riskCategory`
- `riskSignals`
- `missingSources`
- `primaryView`
- `recommendedAction`

其中：

- `backgroundStatus` 复用 `01-init` 的三态判定逻辑
- `dependencyStrength / riskCategory / riskSignals` 复用 `13-orchestrate` 的治理逻辑
- `primaryView` 根据 skill 名自动映射到 `spec/design/code/verify` 或 `role-view`

#### 2. 抽共享 notice builder

不要继续在 `dispatcher.ts` 里增加多个 `if (skillName === ...)`。

建议改为：

- `buildSkillBackgroundNotice(skillName, projectRoot)`

由一个映射表驱动：

- `spec` → `spec-view`
- `design` → `design-view`
- `code` → `code-view`
- `verify` → `verify-view`
- `onboarding` → `role-view`
- `status/doctor/analyze` → 只读背景摘要，不强依赖具体 stage-view

#### 3. 保持 runtime 契约只用 camelCase

`07-code` 的双口径问题，最佳实践不是在类型层增加更多别名字段，而是：

- runtime schema 继续保持 camelCase
- prompt / markdown 渲染层按需做 snake_case 或中文标签映射

这样可以避免：

- 类型重复
- 序列化歧义
- 下游再出现“到底哪组字段才是真的”问题

#### 4. 不建议回退删除 verify-view enrich 字段

原报告建议“只保留一组字段、删除重复定义”。

当前更好的方案是：

- 保留 `testFocus / riskAreas / validationHooks / releaseBlockers` 作为通用验证语义
- 保留 `criticalFlows / validationFocus / recommendedChecks` 作为验证阶段增强语义
- 通过 producer builder 和 projection test 保证二者同时可用

因为当前代码和测试已经围绕这套 richer contract 建好了，不适合再退回到瘦契约。

#### 5. 文档收口也要纳入同一波修复

修 P0/P1 时，建议同步更新：

- `README.md`
- `skills/spec-first/README.md`
- `skills/spec-first/04-design/SKILL.md` 示例输出

否则代码口径收口了，入口文档仍会继续误导用户。

# 2026-03-07 06-task skill 审查与优化建议

## 1. skill 要解决什么问题

`06-task` 不应该只是“把需求列成几条待办”。

它真正要解决的是：

1. 把 `spec + design` 已确认的范围，转换成可执行、可追踪、可交接的任务计划
2. 为后续 `code / test / verify` 提供明确实施顺序、依赖关系和验收口径
3. 让多人协作、阶段交付和中断恢复都有统一任务视图
4. 把“知道要做什么”进一步变成“知道先做哪一项、怎么做完”

一句话：

> `06-task` 解决“如何把设计落实成可执行任务计划”。

## 2. 当前实现是否合理

整体判断：合理，而且任务拆解质量意识很强。

### 2.1 合理的地方

1. 强调不应跳过任务拆解
   - `NO CODE WITHOUT TASK BREAKDOWN`
   - 对全链路治理是有价值的

2. 粒度控制意识明确
   - 任务 2-4 小时
   - 步骤 5-30 分钟
   - 这有助于执行和恢复

3. 标准任务格式比较完整
   - Owner
   - 工期
   - traces
   - depends_on
   - 验收标准
   - 文件清单
   - 执行步骤

   这比普通 Todo 列表强得多。

4. 对依赖、循环和中断恢复有明确考虑
   - 这说明它不是静态文档，而是朝着可执行计划在设计

5. `task_plan.md` 作为运行态核心文件，定位是对的
   - 后续 `code`、`catchup`、`status` 都会消费它

## 3. 当前主要问题

### 3.1 已经不只是“任务拆解”，而是在做执行编排

当前 skill 除了拆任务，还包含：

- 当前会话执行
- 新会话执行
- 并行子代理执行

这说明它已经从“计划层”滑向“执行调度层”。

最佳实践下，`task` 应该负责：

- 生成任务计划
- 明确依赖和并行性
- 交接给后续执行入口

而不应自己继续承担“现在就开始执行哪种模式”的编排职责。

这部分更适合：

- `orchestrate`
- 或显式进入 `code`

### 3.2 成功标准过低，无法代表任务拆解真的完成

当前成功标准仍然只是：

- `metrics coverage` C3 (Task Coverage) `> 0%`

这和 `design` 阶段的问题类似，门槛太低。

只要有少量 TASK 映射，就可能看起来“通过”，但实际上：

- 大量 FR/DS 仍无任务
- 关键路径没有覆盖
- 测试与验证任务可能完全缺失

真正合理的标准应更接近：

- 所有进入实现范围的 FR / DS 都有 TASK
- 关键验收活动也有对应任务
- 无 orphan TASK、无 orphan FR/DS

### 3.3 `auto` 策略与关键交付物写入规则冲突

当前 skill 允许：

- `auto`（低风险）：简单 Feature（< 5 个任务）

但 `task` 会直接写入：

- `task_plan.md`
- `traceability-matrix.md`

而系统治理规则本身又明确：

- `auto` 的 Skill 不应写入关键交付物

这意味着 `06-task` 当前存在治理冲突。

最佳实践应是：

- `task` 生成关键交付物时，至少需要 `assisted`
- `auto` 只能用于建议草案或运行态预分析，不能直接落正式 `task_plan.md`

### 3.4 模板把规划和开发习惯绑定得过死

当前标准任务格式里，默认把：

- `git commit`

写成任务步骤的一部分。

这会带来两个问题：

1. 把任务规划层和代码提交习惯强绑定
2. 容易诱导“每个小任务都必须独立 commit”

但现实里更合理的提交粒度通常是：

- 按逻辑变更
- 按用户故事
- 按可验证批次

而不一定是“每个 TASK 一个 commit”。

### 3.5 任务追踪还偏向 FR，DS 和验证交接不够强

当前文档强调：

- 每个 FR 有对应 TASK

但 `task` 实际承接的是 `spec + design` 之后的实施计划，  
所以更合理的完整追踪应该是：

- FR -> DS -> TASK -> TC / Verify

如果 `task` 只盯 FR，而不显式要求 DS 和验证活动也被任务化，就会出现：

- 有需求任务
- 但没有设计实现任务
- 或没有测试/验证收尾任务

### 3.6 文档细节存在阶段语义漂移

示例中的：

- `## 当前阶段`
- `Phase 2: Implementation`

和本 skill 的实际阶段 `03_plan` 并不一致。

这类细节问题不致命，但会降低文档契约的稳定性。

## 4. 推荐优化方向

### 4.1 把 `task` 收敛成“任务规划主入口”

`task` 的主职责应只保留：

1. 任务拆解
2. 依赖编排
3. 并行性标记
4. 验收与验证任务补齐
5. 交接到后续执行 skill

执行方式选择本身应从核心流程里剥离，改为：

- 规划完成后的附加建议

### 4.2 抬高任务完成门槛

建议成功标准改成：

1. 进入范围的 FR / DS 100% 有 TASK 映射
2. 关键测试与验证动作必须被任务化
3. 不允许 orphan TASK / orphan DS / orphan FR
4. 关键路径和并行路径已明确

这比 `C3 > 0%` 更能代表任务计划可执行。

### 4.3 废除关键交付物上的 `auto`

建议：

- `task_plan.md` 的正式写入只允许 `strict` 或 `assisted`
- `auto` 最多输出“建议草案”到 `findings.md`

这样才能和系统治理规则一致。

### 4.4 放松对 `commit` 的模板硬绑定

推荐把模板中的 `Commit` 步骤改成：

- 可选交付动作
- 或在任务批次层统一描述

保留原则：

- 任务必须可验证
- 但不强制每个任务都绑定 git 提交

### 4.5 显式补齐“实施任务 + 测试任务 + 收尾任务”

任务计划建议固定至少覆盖 3 类任务：

1. 实现任务
2. 测试 / 验证任务
3. 收尾 / 文档 / 发布准备任务

这样 `task` 才真正服务全链路，而不是只服务开发者编码阶段。

## 5. 当前审查结论

- 职责定位：正确
- 当前合理性：中高
- 主要问题：规划和执行边界混杂，覆盖门槛偏低，且 `auto` 策略与关键交付物规则冲突
- 最佳优化方向：收敛为任务规划主入口，抬高覆盖标准，并把执行模式选择剥离出去

一句话结论：

> `06-task` 现在很像“半个 orchestrate”；下一步最重要的不是继续加执行选项，而是让它先成为一份真正可靠、完整、可交接的任务计划。

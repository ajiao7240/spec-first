## 问题 1：Stage Views 构建逻辑过于简单（确认存在，优先级上调）

### 当前代码事实

当前 `buildStageViews()` 仍然是直接从 `summary` 做平铺复制：

- `src/core/skill-runtime/first-stage-views.ts`
- `spec.dependencies <- summary.apiSurface`
- `design.technicalConstraints <- [summary.project.platformType]`
- `verify.testFocus <- summary.capabilities`
- `verify.releaseBlockers <- summary.risks`

对应代码现状可直接概括为：
- 阶段摘要是硬编码模板字符串
- 字段间存在明显语义借位
- 没有按阶段做信息裁剪/增强
- 没有形成“针对 consumer 的最小充分背景输入”

### 为什么这是一个真实问题

原审查意见是成立的，而且当前看 **问题比原文写得更深**：

#### 1. 不只是 builder 简单，schema 也偏弱
当前 `FirstStageViews` 类型只有：
- `spec.dependencies`
- `code.verificationHooks`
- `verify.testFocus / validationHooks / releaseBlockers`

但最近已经落地到 consumer 文档侧的口径，实际上更接近：
- `code-view`: `entry_points` / `likely_change_areas` / `change_hazards`
- `verify-view`: `critical_flows` / `validation_focus` / `recommended_checks`

也就是说，**builder 的简单只是表象，根因是 stage-view schema 还没有真正升级到 consumer 语义**。

#### 2. 当前测试实际上也在固化这种“弱语义映射”
`tests/unit/first-stage-views.test.ts` 目前校验的是“字段被复制过去”，而不是“字段语义是否正确”。
这说明当前实现虽然可运行，但还没有被测试体系要求达到设计目标。

#### 3. 现在已经不是 producer 内部小问题了
随着 `03-spec / 04-design / 07-code / 12-verify` 的文档接入完成，stage-view 已经开始承担“背景输入契约”的角色。
在这种前提下，stage-view 若长期停留在“summary 字段搬运层”，后续 consumer 只会继续在文档里补词，但 runtime 真源本身并没有同步变强。

### 最佳实践方案（推荐）

**不要只在 `buildStageViews()` 里加几个 filter helper 了事。**
最佳实践是走 **schema-first + builder 拆分** 路线。

#### 推荐做法

1. **先升级 `FirstStageViews` schema**
   - 明确 stage-view 真正对外承诺的字段
   - 优先对齐当前 consumer 已经使用的词汇，而不是继续沿用过于抽象的旧字段名
   - 例如：
     - `code` 侧增加“调用路径提示 / 耦合点”类字段
     - `verify` 侧增加 `criticalFlows`、`validationFocus`、`recommendedChecks`

2. **按阶段拆 builder，而不是继续堆在一个函数里**
   - `buildSpecView(summary)`
   - `buildDesignView(summary)`
   - `buildCodeView(summary)`
   - `buildVerifyView(summary)`

3. **每个 builder 只做一类事情**
   - 摘要生成
   - 字段裁剪
   - 风险过滤
   - 少量可解释推导

4. **测试先从“语义正确”出发，而不是“复制成功”出发**
   - `spec.dependencies` 应对应“上下游关系/依赖关系”，而不是简单塞 API surface
   - `verify.recommendedChecks` 应明确来自哪些 runtime 输入推导
   - `summary` 应该能解释“这一阶段最该关注什么”

### 不推荐的做法

- 不建议继续在当前 `buildStageViews()` 里补 6~8 个 `extractXxx()` helper 后就收工
- 不建议先改 consumer 文档再回头补 runtime schema
- 不建议用“字段名字先不动，靠文档解释”来掩盖 schema 不匹配

### 验收标准（复核后修订版）

- [ ] `FirstStageViews` 字段能与当前 consumer 背景输入词汇对齐
- [ ] 每个 stage-view 的 `summary` 是阶段化摘要，而不是模板字符串
- [ ] `spec/design/code/verify` 四个 builder 具备独立测试
- [ ] `tests/unit/first-stage-views.test.ts` 从“复制校验”升级为“语义校验”
- [ ] consumer 文档与 runtime schema 不再出现一套说法、两套字段

### 修复优先级

**P0（应优先于问题 2 和问题 3）**

## 问题 2：缺少增量维护机制（确认存在，但不应按原文直接判为主链阻断）

### 当前代码事实

`refreshFirstArtifacts()` 目前确实还是占位实现：

- `src/core/skill-runtime/first-context.ts`
- `refresh-runtime-only` 直接返回固定 runtime 资产列表
- 其他模式直接调用 `refreshFirstDocsFromRuntime(projectRoot)`
- 没有调用 `checkFirstUpdateContext()`
- 没有局部刷新、没有 no-op、没有索引回写

这一点与原审查意见一致。

### 为什么这是一个真实问题

#### 1. refresh mode contract 已经存在，但执行语义还没落地
当前代码已经公开了 3 个 refresh mode：
- `refresh-runtime-only`
- `refresh-docs-from-runtime`
- `refresh-all`

这说明“增量维护”不是未来幻想，而是已经进入代码契约层。
但契约已存在，执行语义却还是占位，这就是实际问题。

#### 2. 现有基础设施其实已经具备一半
当前仓库已经有：
- `first-change-detector.ts`
- `first-artifact-mapping.ts`
- `writeFirstRuntimeIndex()`
- docs projection mapping

也就是说，这不是“从零设计一整套复杂架构”，而是 **检测层、映射层、写盘层都在，唯独 orchestrator 没接起来**。

### 但为什么不建议继续把它定成 P0

这里需要和原文拉开：

#### 1. 它目前还不是主链热路径故障
从当前代码搜索结果看，`refreshFirstArtifacts()` 主要仍停留在模块内部定义层，尚未成为一条被主链广泛依赖的强约束刷新入口。
所以它更接近：
- 已设计但未做实的能力
- 架构债
- 性能/维护性问题

而不是“今天就会直接让主链行为错误”的 blocker。

#### 2. 如果在 schema 还不稳时先做增量刷新，容易把错误结构固化
问题 1 说明 stage-view schema 还偏弱。
若此时先把“弱 schema 的增量刷新”做完整，后面 schema 升级时又要重做一遍 change mapping、index 回写和 projection 刷新逻辑，返工概率很高。

### 最佳实践方案（推荐）

最佳实践不是“一上来补一个大而全的 refreshFirstArtifacts 重构”，而是走 **contract-first + refresh-plan** 路线。

#### 推荐做法

1. **先定义 refresh plan 数据结构**
   - 输入：`projectRoot + mode + currentIndex + changeContext`
   - 输出：
     - `affectedRuntimeArtifacts`
     - `affectedDocsProjection`
     - `shouldRebuildSummary`
     - `shouldUpdateIndex`
     - `shouldNoop`

2. **把“判断”与“执行”拆开**
   - `planFirstRefresh(...)`
   - `applyFirstRefreshPlan(...)`

3. **先支持 3 个最小能力**
   - 无变化时 no-op
   - runtime 资产按需刷新
   - docs 投影视图按 runtime 变化范围刷新

4. **索引回写独立为 helper**
   - 不把 hash / lastUpdated / healthy / status 更新散落到主函数里
   - 单独封装 `updateFirstRuntimeIndex(...)`

5. **等问题 1 的 schema 收敛后，再把 refresh plan 接到 producer 主链**

### 不推荐的做法

- 不建议现在直接照原文示例，在 `refreshFirstArtifacts()` 内堆一长串 if/else + writeXxx 调用
- 不建议在无实际调用方约束的情况下，一次性做完整增量刷新“大全套”
- 不建议把问题 2 放到问题 1 前面处理

### 验收标准（复核后修订版）

- [ ] `refreshFirstArtifacts()` 至少具备 no-op / targeted refresh / docs-only refresh 三类真实语义
- [ ] 变更检测通过 `checkFirstUpdateContext()` 驱动，而不是固定返回
- [ ] `index.json` 的 hash / 时间戳 / 健康状态可以正确回写
- [ ] 先有 refresh plan 单测，再接调用方
- [ ] schema 升级完成后再把增量刷新纳入 producer 主链

### 修复优先级

**P1（高价值架构债，但不再建议视为当前第一优先级）**

## 问题 3：Docs 投影视图内容过于简单（确认存在，但应按“收敛投影面”处理）

### 当前代码事实

`renderProjectedDoc()` 当前仍然非常简单：

- `README.md` 只输出项目名、模式、阶段列表
- 其他投影文档基本只有：
  - 标题
  - project
  - generatedAt

而 `FIRST_RUNTIME_TO_DOCS_PROJECTION_MAP` 又会把 `summary.json` 投影到很多 `docs/first/*.md` 文档。
这意味着：
- 投影面很大
- 渲染内容却非常薄
- 结果就是生成一批“存在但几乎没有信息量”的文档

### 为什么这是一个真实问题

原审查意见成立，但要补充一条更关键的判断：

> **问题不只是“文档写得不够丰富”，而是“当前 projection surface 与 renderer 能力不匹配”。**

#### 1. 现在的 docs/first 已经不是 truth source
随着 runtime 真源切换推进，`docs/first` 更像：
- 人类可读投影
- 沟通/审查辅助层
- runtime 真源的派生物

因此问题 3 不是 correctness blocker，但它会明显影响：
- 团队审查体验
- 对 runtime 真源的可读性
- 投影视图长期可信度

#### 2. 不能简单理解为“把每个 doc 渲染得更长”
如果继续保持当前映射表，却试图给所有 `docs/first/*.md` 都补全内容，工作量会快速膨胀，而且会重复维护很多模板。

### 最佳实践方案（推荐）

最佳实践不是“所有 docs 全面富化”，而是 **先收敛 projection surface，再为关键投影页做专用 renderer**。

#### 推荐做法

1. **先定义真正需要长期保留的投影页**
   建议优先保留：
   - `docs/first/README.md`
   - `docs/first/summary.md`
   - `docs/first/role-views.md`
   - `docs/first/stage-views.md`

2. **对关键页做专用 renderer**
   - `renderReadmeProjection(context)`
   - `renderSummaryProjection(summary)`
   - `renderRoleViewsProjection(roleViews)`
   - `renderStageViewsProjection(stageViews)`

3. **历史型页面不要强行继续全量投影**
   像 `api-docs.md`、`database-er.md`、`architecture.md` 这类历史文档，若当前没有足够 renderer 语义支持：
   - 要么暂时保留旧文档，不自动刷新
   - 要么明确标记为 deprecated projection
   - 要么等待对应 summary/schema 具备足够结构化信息后再恢复投影

4. **投影页要有 runtime 溯源信息**
   建议每页都带：
   - runtime artifact 来源
   - generatedAt
   - mode
   - index status / projection health（如适用）

### 不推荐的做法

- 不建议继续用 `if (docPath.includes(...))` 这种弱路由去支持越来越多投影文档
- 不建议在当前 projection map 不收敛的情况下，把所有 docs 都补成富文本
- 不建议把 docs projection 重新抬回 truth source 地位

### 验收标准（复核后修订版）

- [ ] 明确 canonical projection docs 列表
- [ ] 关键投影页具备专用 renderer，而不是统一元数据模板
- [ ] 非 canonical docs 要么停刷、要么降级、要么专门设计 renderer
- [ ] 投影文档可读性明显提升，同时不会重新成为真相来源

### 修复优先级

**P2（真实问题，但应在问题 1 和问题 2 之后处理）**

## 复核后的推荐执行顺序

1. **P0：先做问题 1（schema-first 的 stage-view 升级）**
   - 先把 runtime 真源结构做对
   - 再让 consumer 真正消费到有意义的背景输入

2. **P1：再做问题 2（refresh plan / index update / 增量刷新）**
   - 前提是 stage-view schema 已基本稳定

3. **P2：最后做问题 3（projection surface 收敛 + 专用 renderer）**
   - 让 docs/first 成为“轻量但可读”的投影视图层

## 最终判断

原审查文档列出的 3 个问题 **并非误报**，但需要按当前仓库状态做以下修正：

- **问题 1：确认存在，并上调优先级**
- **问题 2：确认存在，但下调为“高价值架构债”而非当前主链 blocker**
- **问题 3：确认存在，但最佳实践不是“全面富化”，而是“收敛投影面 + 专用渲染”**

如果后续要进入实现，最佳实践顺序应该是：

> **先修正 stage-view schema 与 builder，再补增量刷新，最后优化 docs projection。**

这是当前代码状态下返工最少、收益最高的路径。

## 实施进展（2026-03-08 当日已落地）

结合本轮实现，以上 3 个问题已完成如下修复：

- [x] **问题 1 / P0**：`FirstStageViews` 已增量扩展为更贴近 consumer 语义的 schema，`buildStageViews()` 已拆为 `spec/design/code/verify` 分阶段 builder，并补充阶段化摘要、调用路径提示、耦合点、关键链路、验证焦点、建议检查项等信息。
- [x] **问题 2 / P1**：`refreshFirstArtifacts()` 已具备真实的 no-op、targeted refresh、docs-only refresh 语义；已接入 working tree 变更检测、`sourceCommit` 基线比对、已提交源码漂移识别、定向 runtime 重建、docs projection 刷新和 `index.json` 回写。
- [x] **问题 3 / P2**：已新增 canonical projection docs（`docs/first/summary.md`、`docs/first/role-views.md`、`docs/first/stage-views.md`），并为 `README/summary/role-views/stage-views` 提供专用 renderer；其余历史 projection doc 降级为轻量说明页。

### 本轮实现的边界说明

- `summary.json` 的刷新目前是基于**现有 runtime summary 的规范化重建**，而不是重新扫描源码树生成全新摘要。
- 这一处理已经足以解决本次审查指出的“refresh orchestration 缺失 / index 不回写 / projection 过弱”问题，并补齐了“clean tree 下显式 docs refresh 不生效”“仅检测 working tree 导致已提交源码漂移被忽略”这两个边界问题；如果未来要做“源码级全量重建”，仍建议作为独立能力继续演进。

# 2026-03-07 16-sync skill 审查与优化建议

## 1. skill 要解决什么问题

`16-sync` 解决的是：

1. 回填或修正追踪矩阵中的缺失关联
2. 检查 orphan 项
3. 让矩阵和运行态保持一致

一句话：

> `16-sync` 解决“矩阵和状态如何被同步校正”。 

## 2. 当前实现是否合理

整体判断：有必要，但定义偏黑箱。

### 2.1 合理的地方

1. 矩阵确实需要一个维护入口
2. 用户确认后再回填是对的
3. 审计日志写回 `findings.md` 方向成立

## 3. 当前主要问题

### 3.1 同步来源不够透明

当前提到：

- 矩阵
- RFC 状态
- 缺陷状态

但这些外部概念在其他主链 skill 里并不充分展开。

### 3.2 它容易变成“自动修正黑箱”

如果不同步前明确列出：

- 根据什么来源修
- 改哪些行
- 为什么这样改

那同步就可能削弱矩阵可信度。

## 4. 推荐优化方向

### 4.1 收敛为矩阵维护命令

只明确处理：

- 缺失链接
- orphan
- 状态回填

### 4.2 强化同步计划透明度

同步前必须列出：

- 输入来源
- 变更范围
- 预期结果

## 5. 当前审查结论

- 职责定位：合理
- 当前合理性：中
- 主要问题：定义黑箱，输入来源和可改范围不够透明
- 最佳优化方向：收敛为可审计的矩阵维护命令

一句话结论：

> `16-sync` 很有必要，但它必须先做到“同步什么、为什么同步、改了什么”完全透明。

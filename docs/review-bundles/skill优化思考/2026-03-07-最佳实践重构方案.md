# 2026-03-07 最佳实践重构方案

## 1. 重构目标

这份方案不追求大改命令名，而追求 4 个结果：

1. 用户更少思考“该用哪个命令”
2. 主链阶段边界更清楚
3. 控制层不重复定义真理
4. 辅助命令收敛成更清晰的命令族

## 2. 推荐的整体结构

### 2.1 入口层

- `onboarding`
- `first`
- `init`

职责：

- `onboarding`：统一角色入口
- `first`：研发 / 架构项目认知
- `init`：研发需求开发起点

### 2.2 主链交付层

- `spec`
- `design`
- `research`（按需）
- `task`
- `code`
- `code-review`
- `test`
- `verify`
- `archive`

职责：

- 严格保持阶段边界
- 不让辅助命令侵入主链

### 2.3 控制层

- `plan`
- `orchestrate`
- `review` 家族
- `analyze`

职责：

- 做规划、调度、审查、只读分析
- 不重复定义业务产物真理

### 2.4 上下文与维护层

- `feature` 家族
- `context` 家族
- `sync`
- `doctor`

职责：

- 做查询、切换、恢复、维护
- 不和主链阶段命令竞争入口

## 3. 推荐的命令面收敛

### 3.1 `feature` 命令族

将以下命令收敛成一组：

- `feature-list`
- `feature-current`
- `feature-switch`

目标形态：

- `feature list`
- `feature current`
- `feature switch`

### 3.2 `context` 命令族

将以下命令在产品语义上收敛成一组：

- `catchup`
- `status`
- `plan`

推荐目标形态：

- `context catchup`
- `context status`
- `context next`

即便短期不改命令名，也应在文档和入口层按这个心智组织。

### 3.3 `review` 命令族

建议统一 review 心智：

- `spec-review`
- `code-review`
- `analyze`

目标形态可以是：

- `review spec`
- `review code`
- `review system`

## 4. 最重要的边界收敛

### 4.1 统一 `test` 的时间语义

这是第一优先级。

推荐方案：

- 把 `09-test` 明确成“测试规格 / TC 建模”
- 前移到 `code` 前后
- `verify` 只做放行，不再承载测试规划叙事

### 4.2 明确 `code-review` 和 `verify` 分工

- `code-review`：实现阶段质量审查
- `verify`：阶段放行

并把 `completion` 层完全收归 `verify`。

### 4.3 明确 `catchup / status / plan` 分工

- `catchup`：恢复上下文
- `status`：只读观察
- `plan`：决定下一步

### 4.4 明确 `archive` 只负责收口沉淀

不要再让它默认承担：

- 阶段推进
- 后续动作即时执行

## 5. 真理源收敛方案

### 5.1 skill 文档只解释原则，不复制全部规则

例如：

- `verify` 不应长期内嵌大量阶段阈值
- `status` 不应自己定义完整阈值口径
- `plan` 不应硬编码所有阶段推荐

### 5.2 规则下沉到统一来源

建议统一真理源到：

- runtime engine
- gate 配置
- metrics 配置
- 状态枚举

skill 文档只引用这些来源，而不是重新描述完整规则集。

## 6. 状态语义收敛方案

### 6.1 严格统一任务状态

不再保留：

- legacy alias
- 多套“完成态”名称

统一后：

- `planned`
- `in_progress`
- `blocked`
- `complete`
- `verified`

或统一成另一套，但必须全系统只有一套。

### 6.2 统一“谁有权改什么状态”

建议：

- `code`：最多改到 `complete`
- `code-review`：写审查结论，不直接改业务状态
- `verify`：决定是否可推进、是否进入 `verified`

## 7. 优先级建议

### P0

1. 收敛 `09-test` 的阶段定位
2. 收敛 `code-review` / `verify` 边界
3. 收敛 `catchup/status/plan` 边界
4. 清除 legacy 状态语义
5. 清理 `{{DATE}}` 这类未收口文档占位符

### P1

1. 合并 `feature-list/current/switch`
2. 把审查类命令组织成统一 review 心智
3. 把 `archive` 和 `doctor` 的副作用边界收紧

### P2

1. 进一步压缩命令面
2. 补更强的角色化入口
3. 让 `orchestrate` 从阶段驱动变成状态驱动

## 8. 最终推荐形态

如果从最佳实践看，最终系统应收敛成：

### 用户常用入口

- `onboarding`
- `first`
- `init`
- `spec`
- `design`
- `task`
- `code`
- `verify`
- `feature`
- `context`

### 用户按需入口

- `research`
- `code-review`
- `test`
- `archive`
- `review`
- `doctor`
- `sync`
- `orchestrate`

## 9. 一句话结论

> 最佳实践不是继续增加命令，而是把主链保持清晰，把控制层边界收紧，把辅助命令收敛成命令族，并把规则真理源统一下沉到 runtime/config。

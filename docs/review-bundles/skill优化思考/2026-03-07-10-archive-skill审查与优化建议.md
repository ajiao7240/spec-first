# 2026-03-07 10-archive skill 审查与优化建议

## 1. skill 要解决什么问题

`10-archive` 的本质不是“把文件收起来”。

它真正要解决的是：

1. 在 Feature 完成后形成复盘结论
2. 沉淀经验、风险、失败模式和改进建议
3. 对运行态文件做有依据的归档，避免工作区持续膨胀
4. 把本次交付转成后续可复用知识，而不是一次性过程

一句话：

> `10-archive` 解决“如何把一次交付收口成可复用知识并安全归档”。

## 2. 当前实现是否合理

整体判断：方向对，而且 `break-loop` 理念是亮点。

### 2.1 合理的地方

1. 它不只做归档，还强调复盘价值
2. `Immediate Actions` 要求复盘后必须给出行动项，这能防止复盘流于空话
3. 组合门槛归档比单纯按文件大小归档更合理
4. 宪法主副本回写意识是对的，说明它关注长期规则沉淀

## 3. 当前主要问题

### 3.1 把三件事绑得太紧

当前 skill 同时承担：

- 复盘报告
- 运行态归档
- Gate 通过后推进阶段

这三件事相关，但不是同一件事。

最佳实践下：

- 复盘是知识动作
- 归档是文件治理动作
- 阶段推进是流程控制动作

绑在一起会让 `archive` 变得过重。

### 3.2 “关闭阶段”仍要求“立即执行动作”，存在产品张力

文档要求：

- 必须立即输出并执行 Immediate Actions

但 `archive` 本身已经是收尾阶段。  
这意味着它一边在关闭 Feature，一边又要求继续执行动作，容易造成边界混乱。

更合理的做法是：

- 输出后续行动项
- 区分“当前 Feature 收尾动作”和“后续改进 backlog”

### 3.3 阶段推进耦合到 `07_release`，链路一致性存疑

当前成功标准直接写：

- `gate check` 通过后 `stage advance` 已执行，阶段推进至 `07_release`

但当前正式 skill 集里并没有与 `07_release` 对应的主链 skill。  
这会让 `archive` 的收尾变成一个半开放状态。

### 3.4 它更像里程碑型强节点，却又承担运行态文件裁剪

运行态文件归档本身是一个偏机械、偏文件治理的动作。  
而 `archive` 的核心价值其实在于：

- 经验提炼
- 知识沉淀
- 指标总结

这两类动作放在一个 skill 里，会让文档显得既像总结器，又像清理器。

## 4. 推荐优化方向

### 4.1 把 `archive` 收敛为“复盘与沉淀入口”

它的默认主职责应是：

1. 生成 `retro.md`
2. 总结指标与经验教训
3. 输出后续行动项

### 4.2 把“后续行动”与“当前收尾”分层

建议将 Immediate Actions 分成两类：

1. 当前 Feature 收尾动作
2. 后续系统改进动作

后者不应要求在 archive 会话内立刻执行完。

### 4.3 把阶段推进解耦到编排层

建议：

- `archive` 产出复盘结果
- `orchestrate / advance` 再根据结果推进阶段

这样控制层边界更稳定。

## 5. 当前审查结论

- 职责定位：正确
- 当前合理性：中高
- 主要问题：复盘、归档、推进三件事绑得过紧
- 最佳优化方向：收敛为复盘与沉淀入口，把后续行动和阶段推进解耦

一句话结论：

> `10-archive` 的思想是好的，但它现在像“收尾大总管”；最佳实践应让它专注复盘沉淀，把流程控制动作外移。

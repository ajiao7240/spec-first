# 2026-03-07 11-plan skill 审查与优化建议

## 1. skill 要解决什么问题

`11-plan` 解决的不是“写任务计划”，而是：

1. 在当前阶段给出下一步执行建议
2. 显式识别阻塞项和风险等级
3. 为 `orchestrate` 提供可消费的计划摘要

一句话：

> `11-plan` 解决“当前阶段接下来应该怎么走，以及主要风险是什么”。

## 2. 当前实现是否合理

整体判断：有价值，但和控制层其他 skill 有明显重叠。

### 2.1 合理的地方

1. 它明确区分了 `plan` 和 `orchestrate`
2. 风险分类框架是有用的
3. 只写 `findings.md`，没有直接推进阶段，边界方向基本正确

## 3. 当前主要问题

### 3.1 和 `status / orchestrate` 职责重叠

它会做：

- 读取阶段和状态
- 识别阻塞
- 给出下一步命令

这些能力和 `status`、`orchestrate` 都有交叉。  
如果边界不再收紧，`plan` 很容易变成“另一个状态摘要器”。

### 3.2 阶段映射过于线性和理想化

当前硬编码：

- `02_design -> task`
- `04_implement -> test`
- `05_verify -> stage advance`

这忽略了：

- `research`
- `code-review`
- `analyze`
- 返工路径

因此它更像理想主链，而不是实际计划器。

### 3.3 文档自身还存在占位符漂移

`last_updated: {{DATE}}`

这说明它自己作为“计划层 skill”都还没完全收口。

## 4. 推荐优化方向

### 4.1 把 `plan` 收敛为“决策前规划层”

只保留：

1. 当前阶段目标
2. 阻塞项
3. 风险等级
4. 下一步建议

### 4.2 减少硬编码阶段推荐

建议从：

- 固定阶段 -> 固定命令

改成：

- 基于 gate、缺口、阻塞的条件化建议

### 4.3 让它少重复状态，多输出决策依据

`plan` 最有价值的是“为什么下一步是这个”，不是再复述一遍当前状态。

## 5. 当前审查结论

- 职责定位：基本正确
- 当前合理性：中
- 主要问题：与 `status / orchestrate` 重叠，阶段建议过于线性
- 最佳优化方向：收敛为决策前规划层，减少状态复述与硬编码推荐

一句话结论：

> `11-plan` 应该是“下一步为什么这么走”的决策器，而不是另一个状态看板。

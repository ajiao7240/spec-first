# 2026-03-07 14-status skill 审查与优化建议

## 1. skill 要解决什么问题

`14-status` 解决的是：

1. 快速展示 Feature 当前状态
2. 把阶段、覆盖率、任务进度和风险汇总成可读面板
3. 帮用户做观察，而不是直接做控制

一句话：

> `14-status` 解决“当前 Feature 现在是什么状态”。 

## 2. 当前实现是否合理

整体判断：方向正确，而且对汇报和快速判断都很有用。

### 2.1 合理的地方

1. 仪表盘式输出对用户友好
2. 把阶段、覆盖率、任务和风险放在一页里是合理的
3. `auto` 只读策略也符合它的定位

## 3. 当前主要问题

### 3.1 健康分口径偏主观

当前健康分：

- 覆盖率
- 门禁
- 任务进度
- 风险控制

这些维度本身没问题，  
但权重和算法都比较主观，容易给出“看起来精确、实际未必可靠”的分数。

### 3.2 与 `verify / plan` 都在回答“能不能推进”

`status` 本该是观察层。  
但当前文档也在输出：

- 可推进阶段？是 / 否

这会和 `verify` 的正式放行判断重叠。

### 3.3 部分阈值和口径存在漂移风险

状态面板里直接写了很多覆盖率阈值和示例，  
长期会面临和 `verify / metrics` 漂移的问题。

### 3.4 文档同样存在未收口占位符

`last_updated: {{DATE}}`

这在状态类命令里同样不应存在。

## 4. 推荐优化方向

### 4.1 保持它是观察层，不是裁决层

建议：

- `status` 负责展示状态和风险
- `verify` 负责放行
- `plan` 负责建议下一步

### 4.2 弱化总分，强化结构化信号

比起给一个总分，更有价值的是直接展示：

- 红色阻塞项
- 黄色风险项
- 缺失覆盖项
- 当前阶段缺口

### 4.3 尽量读取现有指标，不复制阈值

这样状态面板才不会成为另一套规则定义。

## 5. 当前审查结论

- 职责定位：正确
- 当前合理性：中高
- 主要问题：健康分偏主观，并重复承担推进判断
- 最佳优化方向：保持只读观察层，弱化总分、强化结构化信号

一句话结论：

> `14-status` 适合做“状态观察面板”，不适合做“阶段裁决器”；它越少主观评分，越可靠。

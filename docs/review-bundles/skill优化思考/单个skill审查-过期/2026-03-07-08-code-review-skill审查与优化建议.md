# 2026-03-07 08-code-review skill 审查与优化建议

## 1. skill 要解决什么问题

`08-code-review` 不应该只是“看一遍代码”。

它真正要解决的是：

1. 对 `07-code` 产出的实现做独立质量审查
2. 先确认流程与追踪合规，再讨论代码质量好坏
3. 在进入 `test / verify` 之前，尽早发现高风险实现问题
4. 形成可追溯、可复核的审查结论，而不是口头判断

一句话：

> `08-code-review` 解决“这份实现是否足够合规、足够安全、足够值得继续进入验证阶段”。

## 2. 当前实现是否合理

整体判断：方向对，但明显偏薄。

### 2.1 合理的地方

1. 两阶段审查协议是对的
   - Stage 1：先合规
   - Stage 2：再质量

   这比一上来就讨论代码风格更合理。

2. 分层检查思路是好的
   - `single`
   - `cross`
   - `completion`

   至少说明它意识到不同改动范围需要不同检查深度。

3. 宪法合规纳入 Stage 1 是合理的
   - 违反宪法时，质量讨论本身就失去意义

4. 审查发现写回 `findings.md` 的方向是对的
   - 审查不能只停留在聊天内容里

## 3. 当前主要问题

### 3.1 这个 skill 对其重要性来说，定义过薄

相比 `spec / design / code / task`，`08-code-review` 的规则明显薄很多。

它虽然列了：

- 两阶段审查
- 三个层级
- 几类检查清单

但没有真正展开：

- 如何定义审查范围
- 如何分级问题严重度
- 什么条件算通过
- 什么条件必须阻断
- 怎样和后续 `verify` 衔接

这会让 code review 变成“有概念，但缺审计性”的流程。

### 3.2 审查产物过轻，审计能力不足

当前输出路径只有：

- `findings.md`

这对轻量记录是够的，但对正式 code review 来说不够。

因为 code review 需要回答：

- 审了哪些文件
- 按什么层级审
- 发现了哪些问题
- 每个问题严重度是什么
- 哪些已修，哪些待修
- 为什么可以放行

仅靠 `findings.md` 很难承担正式审查报告角色。

### 3.3 `completion` 层与 `verify` 边界重叠

当前 `code-review` 支持：

- `single`
- `cross`
- `completion`

但系统另一边又明确把：

- `completion`
- 作为 `verify` 的主要层级

这会带来边界重复：

- 到底是 `code-review` 做 completion
- 还是 `verify` 做 completion

最佳实践下，这两者不应重复定义同一层。

更合理的边界是：

- `code-review`：`single / cross`
- `verify`：`completion`

### 3.4 没有明确的放行门槛

当前成功标准只要求：

- Stage 1 和 Stage 2 都生成了结果
- 顺序可复核
- findings 已写入
- 通过审查的 TASK 状态已更新

但没有明确说：

- 出现 `CRITICAL` 是否必须阻断
- 出现 `HIGH` 是否允许放行
- 多少 `MEDIUM` 需要修
- 哪些问题可以带风险进入 verify

如果没有放行门槛，审查就容易退化成“有意见但不影响推进”。

### 3.5 审查后直接更新 TASK 状态，边界也不够清晰

当前 P5 写的是：

- 审查通过则更新 TASK 状态

但 review 本身更适合产出：

- `approved / changes_requested / blocked`

而不是直接接管 TASK 业务状态。

TASK 状态属于实施主线，审查结果属于质量主线。  
两者应该有关联，但不应完全混成同一个状态动作。

### 3.6 “定位变更范围”写在描述里，但没有明确证据来源

skill 说要“定位变更范围并执行代码审查”，但没有规定变更范围应来自：

- git diff
- 当前 TASK 文件清单
- matrix traces
- code skill 的 diff 预览

如果没有明确来源，审查范围就可能靠主观判断，影响一致性。

## 4. 推荐优化方向

### 4.1 把 `code-review` 收敛为实现阶段的正式审查入口

建议边界定义成：

1. 检查实现阶段变更是否合规
2. 检查跨文件与跨模块质量风险
3. 输出正式审查结论
4. 把是否可进入 `verify` 的判断交给后续放行逻辑

这样它不会和 `verify` 抢 completion 职责。

### 4.2 把层级边界重新收紧

建议：

- `single`: 单文件或局部改动
- `cross`: 多文件、跨模块、跨调用链改动
- `completion`: 从 `code-review` 中移出，收归 `verify`

这样职责会更干净。

### 4.3 引入正式审查产物

建议新增：

- `specs/{featureId}/reports/code-review.md`

或等价审查产物，至少包含：

1. 审查范围
2. 审查层级
3. 发现列表
4. 严重度
5. 放行结论
6. 待跟踪问题

`findings.md` 仍可保留，但不应独自承担正式审查报告职责。

### 4.4 明确严重度和放行规则

建议最少定义：

1. `CRITICAL`：必须阻断
2. `HIGH`：默认阻断，需显式接受风险才可继续
3. `MEDIUM`：允许带整改项进入后续阶段
4. `LOW`：记录即可

这样 code review 才真正有门禁价值。

### 4.5 把审查结果和 TASK 状态解耦

建议 review 只更新：

- 审查结论
- 审查报告
- 待修问题

而 TASK 状态仍由：

- `code`
- `test`
- `verify`

按主线控制。

## 5. 当前审查结论

- 职责定位：正确
- 当前合理性：中
- 主要问题：过薄、缺正式审查产物、和 `verify` 存在 completion 边界重叠
- 最佳优化方向：收敛到实现阶段正式审查入口，并补齐严重度和放行机制

一句话结论：

> `08-code-review` 的方向没有错，但目前更像“审查提示器”而不是正式审查关卡；下一步要补的是可审计性、放行规则和与 `verify` 的清晰边界。

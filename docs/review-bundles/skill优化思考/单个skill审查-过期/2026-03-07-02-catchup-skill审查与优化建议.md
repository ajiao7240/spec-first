# 2026-03-07 02-catchup skill 审查与优化建议

## 1. skill 要解决什么问题

`02-catchup` 解决的是：

1. 会话中断后快速恢复上下文
2. 帮用户重新定位当前 Feature、阶段、任务和阻塞
3. 给出最小下一步命令

一句话：

> `02-catchup` 解决“中断后如何快速知道做到哪了、接下来做什么”。 

## 2. 当前实现是否合理

整体判断：恢复框架很完整，但已经明显偏重。

### 2.1 合理的地方

1. 6 步恢复报告和 5-Question 都很系统
2. 信息源优先级设计是对的
3. 对信息缺口给出补齐方案，这一点很实用

## 3. 当前主要问题

### 3.1 过于厚重

对很多恢复场景来说，  
用户真正需要的只是：

- 当前 Feature
- 当前阶段
- 当前任务
- 最大阻塞
- 下一步命令

现在整套 6 步 + 5 问 + 写回流程，对轻恢复过重。

### 3.2 默认写回 `findings.md` 容易污染运行态

恢复报告本质上是读取和总结。  
如果每次 catchup 都默认追加到 `findings.md`，会让运行态混入大量“恢复性摘要”。

### 3.3 与 `status / plan` 重叠明显

它也会输出：

- 当前状态
- 风险
- 下一步

这和 `status / plan` 非常接近。

### 3.4 文档自身还有占位符未收口

`last_updated: {{DATE}}`

这类问题对恢复类命令尤其不应该出现。

## 4. 推荐优化方向

### 4.1 收敛为只读恢复入口

默认只读输出，不自动写回 `findings.md`。

### 4.2 提供轻量 / 完整两档恢复

1. 轻量恢复  
   - 当前 Feature
   - 阶段
   - 当前任务
   - 下一步

2. 完整恢复  
   - 才走 6 步和 5-Question

### 4.3 与 `status / plan` 做清晰分工

- `catchup`：恢复上下文
- `status`：展示当前状态
- `plan`：决定下一步

## 5. 当前审查结论

- 职责定位：正确
- 当前合理性：中
- 主要问题：过重、默认写回运行态、与 `status / plan` 重叠
- 最佳优化方向：收敛为只读恢复入口，并分轻量/完整两档

一句话结论：

> `02-catchup` 不是没价值，而是太想一次做完所有恢复工作；最佳实践是把它做成更轻、更干净的恢复入口。

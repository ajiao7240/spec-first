# 2026-03-07 21-analyze skill 审查与优化建议

## 1. skill 要解决什么问题

`21-analyze` 解决的是：

1. 跨 `spec / design / task / matrix` 找一致性缺口
2. 识别覆盖缺失、语义冲突和产物漂移
3. 输出按严重度排序的只读分析报告

一句话：

> `21-analyze` 解决“跨产物是否彼此一致”。

## 2. 当前实现是否合理

整体判断：方向正确，而且补上了单阶段审查看不到的问题。

### 2.1 合理的地方

1. 它只读分析，不直接改产物，边界方向对
2. 严重度分级是必要的
3. 跨产物一致性确实需要独立命令承接

## 3. 当前主要问题

### 3.1 定义偏薄

和 `spec-review` 类似，它说明了做什么，  
但没有充分展开：

- 分析规则
- 报告结构
- 如何判定阻断

### 3.2 “报告可被 gate 读取”会模糊边界

如果临时分析报告被 gate 直接读取，  
那 `analyze` 就不再是只读辅助，而会变成隐性门禁。

最佳实践下：

- `analyze` 可以提供风险输入
- 但不应成为 gate 真理源

## 4. 推荐优化方向

### 4.1 保持只读一致性分析定位

不要让它承担正式放行职责。

### 4.2 增强报告结构和规则透明度

至少应明确：

- 规则来源
- 发现类型
- 关联位置
- 建议修复路径

## 5. 当前审查结论

- 职责定位：正确
- 当前合理性：中高
- 主要问题：定义偏薄，且和正式门禁的边界需要更清楚
- 最佳优化方向：保持只读分析，不让分析报告成为门禁真理源

一句话结论：

> `21-analyze` 很有价值，但它最稳的角色是“高级分析器”，不是“隐性 gate 引擎”。

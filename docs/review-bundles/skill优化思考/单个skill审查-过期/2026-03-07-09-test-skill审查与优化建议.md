# 2026-03-07 09-test skill 审查与优化建议

## 1. skill 要解决什么问题

`09-test` 的核心价值，不是“把测试代码补一补”。

它真正要解决的是：

1. 把 FR / AC 转成结构化测试用例（TC）
2. 明确每个需求和验收标准由哪类测试覆盖
3. 为后续测试实现、验证和放行提供覆盖依据
4. 避免需求完成后才发现“验收标准根本无法测试”

一句话：

> `09-test` 解决“需求和验收标准如何被测试化并形成覆盖映射”。

## 2. 当前实现是否合理

整体判断：方向对，但阶段定位和能力命名明显有冲突。

### 2.1 合理的地方

1. 它重视 FR / AC -> TC 的映射
   - 这对全链路追踪非常关键

2. 测试层级设计是有体系的
   - UT / IT / E2E / ST
   - 还有 AC 到测试层级的推荐映射

3. 测试用例结构规范写得比较完整
   - 前置条件
   - 步骤
   - 预期结果
   - 优先级

4. “数量不等于覆盖率”的提醒是对的
   - 它不是只追求多写几个 TC

## 3. 当前主要问题

### 3.1 阶段定位和 TDD 叙述互相冲突

这是当前最核心的问题。

文档一方面说：

- 完成任务拆解后
- 开始实现前
- 遵循 TDD
- 先写测试用例文档，再写测试代码，再写实现代码

但另一方面又把 skill 固定在：

- `05_verify`

这会直接带来冲突：

- 如果它真是 TDD 的前置测试建模，就不应放在 `05_verify`
- 如果它真属于 `05_verify`，那它就不应再承担“开始实现前”的主叙事

也就是说，当前 `09-test` 的时间位置和能力定位没有对齐。

### 3.2 skill 名称容易误导成“测试实现”，实际更像“测试规格建模”

当前 skill 实际做的是：

- 生成 TC 文档
- 映射 FR / AC
- 更新矩阵

而不是：

- 真正写自动化测试代码
- 执行测试
- 产出测试结果

因此它更像：

- `test-plan`
- `test-spec`
- `tc-design`

而不是传统意义上的 `test`。

### 3.3 成功门槛仍然过低

当前成功标准同时写了：

- 所有 TC 已注册
- 每个 AC 至少映射一个 TC

但覆盖指标仍然只是：

- C4 > 0%
- C5 > 0%

这和前几个 skill 的问题一样，门槛太低。

如果“每个 AC 至少映射一个 TC”已经成立，那么成功标准就应直接要求：

- C4 = 100%
- C5 = 100%

或者至少要求“进入验证范围的项必须 100% 覆盖”。

### 3.4 `auto` 与关键交付物写入再次冲突

当前 skill 支持：

- `auto`（低风险）：简单功能（< 5 个 AC）

但它会正式写：

- `traceability-matrix.md`
- `specs/{featureId}/tests/*.test.md`

这仍然和系统里的统一规则冲突：

- `auto` 不应写关键交付物

因此这里也需要收敛。

### 3.5 输出路径命名容易和真实测试代码混淆

当前输出：

- `specs/{featureId}/tests/*.test.md`

这会有一个产品语义问题：

- `.test.md` 看起来很像真正的测试代码文件
- 但实际只是测试用例文档

长期看，这种命名容易让“测试规格”和“自动化测试实现”混淆。

## 4. 推荐优化方向

### 4.1 先决定它到底属于哪一层

最佳实践只有两种稳定路径。

1. 路径 A，推荐  
   - 把 `09-test` 明确定义为“测试规格建模”
   - 前移到 `03_plan` 或 `04_implement` 前后
   - 服务 `code` 的 TDD 和 `verify` 的覆盖检查

2. 路径 B  
   - 保留在 `05_verify`
   - 但删掉“TDD 前置入口”的叙事
   - 只做“验证阶段测试用例补齐与覆盖检查”

我更推荐路径 A，因为它和当前文档里“TDD / 实现前”这条主线更一致。

### 4.2 把 skill 名称或文案收敛到“测试规格”

即使不改命令名，也建议把文档明确写成：

- 生成测试规格
- 生成 TC 设计
- 不负责真正的测试代码实现

这样边界会更清楚。

### 4.3 抬高覆盖和合规门槛

建议成功标准改成：

1. 进入验证范围的 FR 100% 有 TC
2. 进入验证范围的 AC 100% 有 TC
3. C9（TC Compliance）必须纳入
4. 关键路径必须有明确层级覆盖

否则它的门禁价值还是太弱。

### 4.4 取消关键交付物上的 `auto`

建议：

- 正式 TC 文档和矩阵写入只允许 `assisted / strict`
- `auto` 最多做预分析和建议，不直接写正式产物

### 4.5 区分“测试规格”和“测试实现”

建议链路分工明确成：

1. `09-test`
   - 负责 TC 设计和覆盖映射

2. `07-code`
   - 负责把部分 TC 落成自动化测试代码

3. `12-verify`
   - 负责检查 TC 覆盖、执行结果和放行门槛

这样全链路就不会把测试规划、测试编码和测试放行混在一起。

## 5. 当前审查结论

- 职责定位：方向正确
- 当前合理性：中
- 主要问题：阶段位置与 TDD 叙事冲突，能力名称与真实职责不匹配，覆盖门槛偏低
- 最佳优化方向：重定义为测试规格入口，或前移阶段并显式服务 TDD

一句话结论：

> `09-test` 现在最大的问题不是内容不够，而是“它到底属于测试规划还是验证阶段”没有讲清楚；这一步必须先收敛，后面的测试链路才会稳。

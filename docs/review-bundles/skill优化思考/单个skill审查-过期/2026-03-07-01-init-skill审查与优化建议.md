# 2026-03-07 01-init skill 审查与优化建议

## 1. skill 要解决什么问题

`01-init` 不应该被理解成“创建目录”。

按我们已经确认的边界，它真正要解决的是：

1. 把 `00-first` 的项目认知结果转成当前需求开发上下文
2. 帮研发明确“这次需求从哪里开始”
3. 初始化后续 `spec/design/task/code/verify` 真会消费的工作区
4. 把用户顺畅交接到 `spec`

一句话：

> `01-init` 解决“我要做什么、从哪里开始”。

## 2. 当前实现是否合理

整体判断：合理，但仍然偏工具化，离最佳实践还有一层。

### 2.1 合理的地方

1. 它有明确前置约束
   - 强制要求先完成 `00-first`
   - 这让研发不是在毫无上下文时直接进入需求开发

2. 初始化结果足够完整
   - `stage-state.json`
   - `findings.md`
   - `task_plan.md`
   - `traceability-matrix.md`
   - `constitution.md`

3. 核心实现较稳
   - 幂等
   - 临时目录 + rename
   - registry lock
   - `.spec-first/current` 自愈

4. 交互体验对新人友好
   - 不是强迫用户一次性提交所有参数
   - 分步引导比纯 CLI flags 更容易上手

## 3. 当前主要问题

### 3.1 仍然偏“参数收集器 + 目录生成器”

虽然它会检查 `first` 是否完成，但真正执行时仍然主要在做：

- 检查前置条件
- 收集 `feat/mode/size/platforms/title`
- 创建 Feature 工作区

它没有真正把“项目背景”转换成“本次需求开发起点”。

换句话说：

- 它知道 `first` 跑过了
- 但它并没有真正理解并消费 `first` 的认知结果

### 3.2 对 `first` 的消费方式不稳定

当前通过读取 markdown 文本并用正则提取：

- 技术栈
- 代码量
- API 端点

这会带来两个问题：

1. `first` 文案一变，`init` 摘要就退化
2. `init` 没法稳定读取“推荐切入点”这类更重要的信息

最佳实践应是：

- `first` 提供结构化认知摘要
- `init` 消费结构化摘要

### 3.3 核心职责和附加职责耦合在一起

当前 `init` 在做完核心初始化后，还会继续做：

- hooks 安装
- `.claude/settings.json` 补齐
- AI hooks 注册
- Skill 命令刷新
- 可选 bootstrap

这些能力本身有价值，但它们会让 `init` 的产品定位变得模糊。

更合理的边界是：

- `init` 只负责需求开发起点
- 环境修复和宿主接入交给 `doctor/setup/update`

### 3.4 文档承诺与实际输出不完全一致

`SKILL.md` 里定义的 P5 是：

- 执行 `stage current`
- 输出更完整的摘要

但当前 CLI 最终只输出：

- 初始化完成
- Feature ID
- 目录

这让 `init` 结束时的“交接感”还不够强。

### 3.5 对 greenfield / brownfield 区分不够清晰

当前实现对 `00-first` 的要求是统一的。

这对 brownfield 很合理，但对真正 `0-1` 的项目会偏硬：

- 空项目也要先完整跑一轮 `first`
- 然后才能 `init`

更合理的做法应该是：

- brownfield：强依赖 `first`
- greenfield：允许轻量初始化，再进入 `spec`

## 4. 推荐优化方向

### 4.1 把 `init` 定位成“需求开发起点向导”

`init` 不应只是创建工作区，而应当完成三件事：

1. 展示项目背景摘要
2. 确认当前需求开发目标
3. 初始化空间并交接下一步

### 4.2 引入结构化认知桥接

建议新增：

- `docs/first/.summary.json`

由 `first` 输出，`init` 直接消费。

建议字段：

- `project_type`
- `tech_stack`
- `core_modules`
- `domain_summary`
- `recommended_entry_points`
- `first_mode`

### 4.3 压缩非核心副作用

建议将以下能力降级为可选或延后：

- hooks 安装
- Skill 命令刷新
- AI hooks 注册
- bootstrap

保留原则：

- `init` 主链必须稳定
- 环境副作用不要抢主职责

### 4.4 强化结束交接

`init` 最终输出应明确包含：

- 当前 Feature 已创建
- 当前阶段为 `00_init`
- 已初始化的关键产物
- 下一步推荐命令：`/spec-first:spec`
- 推荐理由

这样它才真正像“起点”，而不是“创建完成”。

### 4.5 区分 greenfield / brownfield 路径

建议：

- brownfield：继续要求 `first`
- greenfield：允许轻量路径进入 `init`

否则 `0-1` 项目会被当前链路卡得过重。

## 5. 当前审查结论

- 职责定位：正确
- 当前合理性：中高
- 主要问题：对 `first` 的消费仍然太弱，核心职责与附加职责耦合
- 最佳优化方向：从“初始化工具”升级成“研发需求开发起点向导”

一句话结论：

> `01-init` 现在已经够稳，但还不够像“起点”；下一步要做的不是继续加参数，而是让它真正承担认知到开发的交接。

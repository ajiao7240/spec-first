# 2026-03-06 00-onboarding skill 审查与优化建议

## 1. skill 要解决什么问题

`00-onboarding` 的本质不是“欢迎页”，而是：

- 帮新用户识别自己的角色
- 帮用户识别自己当前想做的事
- 给出最合理的下一条 skill 链路

如果按我们前面收敛的最佳实践，这个 skill 应该解决的是：

> 不同角色进入系统时，应该从哪里开始。

也就是说，它应该成为“角色化入口层”的第一站。

## 2. 当前实现是否合理

整体判断：方向是对的，但当前实现还只是一个“静态路径推荐器”，还不是成熟的角色入口。

合理点：

1. 它已经抓住了角色差异
   - 开发者
   - 产品经理
   - 测试工程师
   - 架构师

2. 它已经在尝试做场景化入口
   - 角色
   - 任务类型
   - 项目规模

3. 它和我们现在的结论一致
   - 不同角色不应走完全相同的第一条链路

所以，`00-onboarding` 不是方向错了，而是还不够深入。

## 3. 当前主要问题

### 3.1 仍然是静态推荐表，不是真正的入口编排

当前 `references/scenario-mapping.md` 本质上是静态映射：

- 某个角色
- 某个任务
- 某个规模
- 对应一条固定 skill 序列

问题在于：

- 它没有消费当前项目真实状态
- 没有消费 `00-first` 认知底座
- 没有消费当前 Feature / 当前阶段
- 没有根据运行态动态调整推荐

因此它更像“帮助页”，不像“入口系统”。

### 3.2 对产品和测试的入口仍然偏研发主链

例如当前映射里：

- 产品经理场景仍然先走 `first -> init -> spec`
- 测试工程师场景仍然先走 `first -> catchup -> test`

这和我们已经确认的最佳实践有偏差：

- 产品不应该默认先走 `first`
- 测试也不应该默认先走研发式认知入口

更合理的是：

- 产品入口优先进入 `spec`
- 测试入口优先进入 `test / verify`
- `first` 作为底座，不一定由产品或测试显式执行

### 3.3 skill 文档和当前宿主能力不完全匹配

文档里写的是使用 `AskUserQuestion` 工具收集信息，但当前运行环境并没有这个统一工具契约。

这会导致两个问题：

- 文档写得像“可直接执行”
- 实际只是一个概念性 skill，没有真实运行保障

### 3.4 命名规范未完全接入正式 `spec-first` skill 体系

当前 front matter 是：

- `name: onboarding`

而不是：

- `name: "spec-first:onboarding"`

这说明它虽然放在正式目录里，但还没有完全按 `spec-first` 正式 skill 规范治理。

### 3.5 保存学习路径到 `docs/onboarding/` 的价值偏弱

当前设计会把学习路径保存成角色文档。

问题：

- 这类文档是一次性引导信息，长期价值有限
- 更重要的是动态推荐当前下一步，而不是积累静态学习路径文档

所以这部分价值低于“实时角色化入口”本身。

## 4. 推荐定位

`00-onboarding` 最合理的定位不是：

- 新手欢迎页
- 学习路径推荐器

而应该是：

> 统一角色入口层

它负责：

1. 判断用户角色
2. 判断用户当前目标
3. 判断当前项目状态
4. 给出角色化的下一步 skill 入口

## 5. 推荐优化方向

### 5.1 从静态映射升级为动态入口

建议它的输入从：

- role
- task
- size

升级为：

- role
- current_goal
- project_state
- current_feature_state
- first_summary_available

这样它的推荐才真正与当前系统状态相关。

### 5.2 明确不同角色的推荐主链

推荐改成：

#### 开发者

- 默认链路：`first -> init -> spec`

#### 产品经理

- 默认链路：`spec`
- 如缺少项目认知底座，则展示“当前项目认知不可用，建议研发先运行 `first`”

#### 测试工程师

- 默认链路：`test / verify`
- 如当前 Feature 不存在，则先引导到 `feature-current/status`

#### 架构师

- 默认链路：`first -> design / research`

### 5.3 不再把“保存学习路径文档”作为核心能力

建议降级这部分：

- 可以保留为可选能力
- 但不应作为核心目标

核心目标应转成：

- 当下给出正确入口
- 减少误入错误链路

### 5.4 正式接入 `spec-first` skill 治理规范

建议修正：

- front matter 命名规范
- 角色入口说明
- 运行时依赖说明

至少要做到：

- 在文档层面与正式 skill 体系一致
- 不再依赖当前环境中不存在的抽象工具名

## 6. 推荐演进路径

### P0

- 修正正式命名与文档规范
- 明确它是“角色入口层”
- 修正文档中对产品/测试默认链路的偏差

### P1

- 引入当前项目状态读取
- 动态根据是否存在 `first` 认知底座、当前 Feature、当前阶段调整推荐

### P2

- 让它成为真正的统一入口 skill
- 研发 / 产品 / 测试 / 架构从同一个入口进入不同链路

## 7. 当前审查结论

- 职责方向：正确
- 当前成熟度：中
- 主要问题：仍然偏静态、偏学习路径，不够动态、也未完全按正式 skill 治理
- 最佳优化方向：从“新手引导”升级成“角色化入口层”

一句话结论：

> `00-onboarding` 已经找对了方向，但下一步不应继续堆“推荐文案”，而应把它做成真正的角色化动态入口。

# 2026-03-07 13-orchestrate skill 审查与优化建议

## 1. skill 要解决什么问题

`13-orchestrate` 是全链路主编排器。

它真正要解决的是：

1. 在当前阶段决定应该调度哪个主 skill
2. 控制批次执行、检查点、阻塞暂停和阶段推进
3. 把 `plan -> 执行 -> verify -> advance` 串成可恢复流程

一句话：

> `13-orchestrate` 解决“整条链路如何按证据和批次被安全驱动”。

## 2. 当前实现是否合理

整体判断：方向正确，而且批次/检查点思想是对的。

### 2.1 合理的地方

1. 它明确了批次执行，而不是无限串行推进
2. 它要求阻塞就暂停，不允许带病推进
3. 它把 verify 放在推进前，符合治理主线

## 3. 当前主要问题

### 3.1 序列硬编码过强

当前直接写死：

- `plan -> skill -> verify -> advance`

以及阶段到 skill 的固定映射。

这对基础主链成立，但对真实流程来说太硬：

- `research` 是按需
- `code-review` 是按需
- `analyze` 也是按需
- 返工路径不一定回到同一个节点

### 3.2 仍残留 legacy 状态语义

当前文档还写：

- `complete/verified` 作为 legacy alias 归一到 `done`

这和我们前面已经收敛的“开发期不做向下兼容、状态语义应统一”方向冲突。

### 3.3 它对子 skill 的决策粒度还不够智能

现在更像：

- 阶段驱动调度器

而不是：

- 基于当前缺口、阻塞、产物健康度的调度器

这会让编排器过于依赖固定流程，而弱化“按实际状态决策”的能力。

## 4. 推荐优化方向

### 4.1 把 `orchestrate` 收敛为批次编排器

保留三件核心能力：

1. 选择下一批动作
2. 控制检查点
3. 统一推进或暂停

### 4.2 减少硬编码，多使用状态驱动

建议更多依据：

- 当前阶段
- gate 结果
- coverage 缺口
- findings 中最新计划
- 是否存在阻塞

而不是只按静态阶段映射。

### 4.3 统一状态语义，清除 legacy alias

这一步很关键。  
编排器是链路中枢，不应继续容忍多套任务终态语义。

## 5. 当前审查结论

- 职责定位：正确
- 当前合理性：中高
- 主要问题：线性序列和阶段映射过硬，且仍保留 legacy 状态兼容思维
- 最佳优化方向：升级为状态驱动的批次编排器，并统一状态语义

一句话结论：

> `13-orchestrate` 已经有了主编排骨架，但还不够“按状态决策”；下一步应减少硬编码，真正让它成为链路控制中枢。

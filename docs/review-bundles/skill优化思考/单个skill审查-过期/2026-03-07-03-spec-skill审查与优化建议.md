# 2026-03-07 03-spec skill 审查与优化建议

## 1. skill 要解决什么问题

按我们已经确认的边界，`03-spec` 不是“开始做方案”，也不是“顺手生成计划”。

它真正要解决的是：

1. 帮产品、研发把模糊需求转成清晰、可验证的正式规格
2. 明确业务目标、范围边界、功能需求、验收标准与非功能约束
3. 在进入 `design / research / task` 之前，把需求理解先收敛到可落地状态
4. 把“我想做什么”变成“系统确认要做什么”

一句话：

> `03-spec` 解决“具体需求怎么定义”。

## 2. 当前实现是否合理

整体判断：很强，而且是当前主链里定义最完整的 skill 之一。

### 2.1 合理的地方

1. 职责边界在原则上是清楚的
   - 明确强调 `spec` 只定义 `WHAT`，不定义 `HOW`
   - 这一点非常关键，也符合主链分工

2. 反合理化约束做得足够强
   - 对“需求很清楚，不需要澄清”“PRD 可以跳过”这类常见偷跑路径有明确封堵
   - 能有效抑制 spec 阶段过早滑向设计或实现

3. 澄清机制是结构化的
   - `[NEEDS CLARIFICATION]`
   - 一问一答
   - 每轮最多 3 问
   - 优先级排序

   这说明它不是随意聊天，而是在做可控的需求收敛。

4. PRD-first 是合理的
   - 先产出 `prd.md`
   - 再生成 FR / AC
   - 这个顺序是对的，能减少“先写编号，后补理解”的倒挂

5. 宪法一致性检查必须保留
   - 当前把宪法检查放在 FR/AC 收敛后、推进下一步前
   - 这是好设计，不应删掉，只应保留并收敛表达

## 3. 当前主要问题

### 3.1 流程过厚，已经不像单一的“需求定义 skill”

当前 `03-spec` 不只在做需求定义，还同时承担了：

- Feature 快速初始化
- 质量扫描
- 自动上下文收集
- 研究触发
- 发散扫描
- ADR-lite 决策记录
- 最终确认包
- 实施计划交接
- 追踪矩阵注册

这些动作单看都有价值，但都堆在一个 skill 里后，`spec` 就开始像一个“迷你编排器”，而不是清晰的需求定义入口。

### 3.2 默认路径对产品和新人偏重

从产品和新人视角看，`03-spec` 最重要的是：

- 把需求讲清楚
- 把范围边界讲清楚
- 把验收标准定下来

但当前默认流程包含：

- Phase 0.0 到 0.6
- Step 0 到 Step 8
- 复杂度分档
- Question Gate
- Research-first
- Expansion Sweep
- ADR-lite
- Final Confirmation + Implementation Plan

这更像高级需求工作流，而不是“需求定义入口”。

### 3.3 对 `research / design / task` 的边界有侵入

当前有三块最明显：

1. `Step 4: Research-first Mode`
   - 方向对，但它更像 `research` 的职责触发，而不该成为 `spec` 的默认主流程主体

2. `Step 7: ADR-lite`
   - 这是决策记录，已经开始靠近 `design`

3. `Step 8: Final Confirmation + Implementation Plan`
   - 最终确认合理
   - 但“生成实施计划”更像交接到 `design / plan` 的动作

也就是说，`spec` 现在不是边界不清，而是“边界知道，但做得太多”。

### 3.4 Phase 0.0 与 `init` 的职责存在重叠

`03-spec` 里的 `Feature 快速初始化` 设计，本意是防止信息丢失，这个目标是正确的。

但从整体链路看，这会和我们已经确认的 `init` 边界产生重叠：

- `init` 负责初始化需求目录空间
- `spec` 负责需求理解和正式定义

如果两边都能创建 Feature，就容易让入口职责变得模糊。

最佳实践更适合这样理解：

- `init` 是正式起点
- `spec` 可以提供“兜底式快速锚定”，但不应让它成为常规主路径

### 3.5 默认路径缺少“轻量模式”

不是所有需求都需要完整的：

- 质量扫描
- 调研触发
- ADR-lite
- 实施计划确认

对很多小中型需求，真正需要的只是：

1. 明确业务目标
2. 明确范围和边界
3. 产出 FR / AC / NFR
4. 做一次用户确认

当前 skill 虽然在复杂度上支持跳过部分步骤，但从文档呈现和认知负担上，默认仍然显得很厚。

## 4. 推荐优化方向

### 4.1 把 `spec` 收敛为“需求定义主入口”

默认路径建议只保留 5 件事：

1. 需求理解
2. 歧义澄清
3. FR / AC / NFR 定义
4. 宪法一致性检查
5. 最终确认

这是 `spec` 最核心、也最不能丢的能力。

### 4.2 设计成“双层 spec”

建议把 `03-spec` 分成两个执行层：

1. 轻量路径
   - 面向产品、新人、简单需求
   - 只走：PRD -> 澄清 -> FR/AC/NFR -> 宪法检查 -> 确认

2. 完整路径
   - 面向复杂、高风险、多方案需求
   - 才扩展到质量扫描、研究触发、发散扫描、ADR-lite、实施交接

这样既保留高质量能力，也避免默认把所有用户带进重流程。

### 4.3 把重动作更多地改成“触发”而不是“内嵌”

建议边界如下：

- `research`
  - 由 `spec` 判断是否需要
  - 但实际调研工作由 `research` 承接

- `design`
  - 负责技术方案和 ADR 主体
  - `spec` 只记录“存在决策待解”

- `plan/task`
  - 负责实施计划与任务拆解
  - `spec` 只负责把确认后的需求交接出去

也就是说，`spec` 应该负责“识别需要什么”，不需要把后续动作都自己做完。

### 4.4 重新定义 `Step 8`

`Step 8` 建议拆成两部分：

1. `Final Confirmation`
   - 保留在 `spec`

2. `Implementation Handoff`
   - 不再写成“生成实施计划”
   - 改成“明确下一步建议：进入 `/spec-first:design` 或 `/spec-first:task`”

这样能避免 `spec` 过早吞掉后续阶段的职责。

### 4.5 把 Phase 0.0 改成兜底能力，而不是常规心智模型

建议对外仍然强调：

- 正常路径：`init -> spec`

而不是：

- 用户一说需求就默认由 `spec` 创建 Feature

Phase 0.0 可以保留，但定位应改成：

- 会话中断保护
- 缺少初始化时的兜底锚定

而不是主链正式入口。

## 5. 当前审查结论

- 职责定位：非常正确
- 当前合理性：高
- 主要问题：不是弱，而是过厚
- 最佳优化方向：从“高级需求工作流全集”收敛成“需求定义主入口 + 复杂度分层扩展”

一句话结论：

> `03-spec` 现在已经足够强，但不够轻；下一步最重要的不是继续加步骤，而是把默认路径压缩成真正适合产品与研发协作的需求定义入口。

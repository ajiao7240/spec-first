# 2026-03-07 07-code skill 审查与优化建议

## 1. skill 要解决什么问题

`07-code` 不应该被理解成“让 AI 直接开始写代码”。

它真正要解决的是：

1. 按 `task_plan.md` 中已经明确的 TASK 执行实现
2. 把 `spec / design / task` 的上游约束落实到代码与测试证据上
3. 在实现过程中保持追踪链路、TDD 证据和任务状态同步
4. 为 `code-review / test / verify` 交付一个可审、可测、可追踪的实现结果

一句话：

> `07-code` 解决“如何按既定任务和约束安全实现代码”。

## 2. 当前实现是否合理

整体判断：合理，而且治理意识很强。

### 2.1 合理的地方

1. TDD 入口守卫是强项
   - 要求 RED 证据或 `TDD-WAIVER`
   - 这能明显抬高实现质量下限

2. 与 `task_plan.md` 的绑定是对的
   - 只允许在有 `in_progress` TASK 时进入实现
   - 这样代码不是“想写什么就写什么”

3. 调试纪律比普通 code skill 更成熟
   - 根因调查
   - 模式分析
   - 单变量假设验证
   - 3 次失败后升级

4. `findings.md` 的证据落盘要求合理
   - TDD RED / GREEN
   - 中断恢复
   - 当前阻塞

5. 与上游规格和设计的对齐意识是成立的
   - 不是脱离 FR / DS 的自由编码

## 3. 当前主要问题

### 3.1 混入了过强的技术栈假设

当前文档里大量质量检查和验证命令直接写成：

- `npm run lint`
- `npm run typecheck`
- `npm run build`
- `Vitest`
- 覆盖率固定阈值

这和它前面为测试命令做的跨栈探测形成了不一致。

也就是说：

- 测试命令考虑了多栈
- lint / typecheck / build 仍然强绑定 Node / TypeScript

如果这套 skill 要服务多平台项目，这种假设太重。

### 3.2 `traces trailer` 机制过于侵入式

当前要求：

- 每个新文件末尾都注入 `Related / Task / Author / Date`

这会带来几个问题：

1. 代码被流程注释污染
2. 日期和作者字段会制造无意义 diff
3. 并不适合所有语言、配置文件、模板文件或生成文件
4. 追踪信息写进源码末尾，不一定是最佳长期载体

最佳实践更适合：

- 追踪关系主要留在矩阵、task、review 和提交元信息里
- 源码注入只在确有必要的语言或文件类型中按约定使用

### 3.3 把“实现完成”和“验证完成”混在了一起

当前文档允许把 TASK 状态更新为：

- `complete`
- `verified`

但从主链分工看：

- `code` 应负责实现完成
- `test / verify` 才应负责验证完成

如果 `code` 阶段就把任务标到 `verified`，会稀释后续验证阶段的职责边界。

### 3.4 “不要跳过 code-review”写进规则，但成功标准里没有显式交接

文档明确说：

- 不应跳过 code-review

但成功标准只要求：

- 代码写完
- TASK 更新
- findings 落盘
- lint / typecheck / test 通过

没有把“交接到 `08-code-review`”作为显式收尾要求。

这会导致：

- 文档倡导 review 必须做
- 但 skill 自身仍可能在未 review 时显得“已完成”

### 3.5 P3 要求“先完整 diff 预览再确认”，对实现阶段过重

在高风险变更下，先做 diff 预览是合理的。  
但如果所有代码实现都要求完整预览后再写入，实际会带来两个问题：

1. 对迭代式实现过重
2. 容易让 `code` 退化成“先伪写一遍，再正式写一遍”

更合理的分层是：

- 高风险/大改动：保留完整预览
- 低风险/小改动：允许摘要式确认

### 3.6 回滚策略和提交策略仍偏 git 中心

当前回滚策略高度依赖：

- `git revert`

这对大多数场景是有用的，但如果从 skill 设计看，它更应该先描述：

- 逻辑回滚点
- 配置回滚点
- 数据回滚点

而不是过早把策略收窄成 git 动作。

## 4. 推荐优化方向

### 4.1 把 `code` 收敛为“实现执行入口”，不要吞掉后续验证职责

建议边界明确成：

1. `code`：实现 + 本地验证 + 任务状态更新为 `complete`
2. `code-review`：质量审查
3. `test / verify`：验证闭环与放行

这样任务状态语义会更稳定。

### 4.2 把验证命令从“Node 默认值”升级成“项目能力探测”

建议：

- 测试、lint、typecheck、build 都按项目能力探测
- 若某项能力不存在，明确标记 `N/A`，而不是默认 Node 命令失败

这样才能和系统的多平台目标一致。

### 4.3 把 `traces trailer` 从强制源码注入改成分层追踪策略

建议改成：

1. 主追踪载体：`traceability-matrix.md`
2. 运行态追踪：`task_plan.md` / `findings.md`
3. 源码内追踪：仅在适合的文件类型按约定启用

避免把所有实现文件都变成流程注释容器。

### 4.4 把 code-review 交接写入成功标准

`07-code` 完成时应明确输出：

- 当前实现已完成
- 下一步建议执行 `/spec-first:code-review`
- 触发条件：本地验证通过、diff 已稳定

这样“不得跳过 review”才是真正的闭环规则。

### 4.5 为 P3 引入风险分层

建议：

- 高风险改动：完整 diff 预览
- 常规改动：摘要式变更确认
- 极小改动：在保持证据的前提下轻量确认

这样既保留人在回路，也不让实现流程过于僵硬。

## 5. 当前审查结论

- 职责定位：正确
- 当前合理性：高
- 主要问题：技术栈假设过重，追踪注入过于侵入，并混淆实现与验证边界
- 最佳优化方向：收敛为实现执行入口，统一多栈能力探测，降低源码追踪侵入性

一句话结论：

> `07-code` 已经具备很强的工程治理意识，但还不够“平台化”和“边界清晰”；下一步要让它更像实现入口，而不是把实现、验证和追踪都压进一个 skill 里。

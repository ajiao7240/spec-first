# 2026-03-07 12-verify skill 审查与优化建议

## 1. skill 要解决什么问题

`12-verify` 解决的是：

1. 阶段是否具备放行条件
2. 当前证据是否足以支撑“可推进”声明
3. 失败项应如何修复

一句话：

> `12-verify` 解决“现在是否真的可以推进阶段，以及证据是什么”。

## 2. 当前实现是否合理

整体判断：很强，而且是全链路治理里最关键的 skill 之一。

### 2.1 合理的地方

1. 五步证据铁律很成熟
2. Gate 状态、失败处理、WAIVER 机制都比较完整
3. 它把“主观完成”改成了“命令和退出码驱动的放行”

## 3. 当前主要问题

### 3.1 规则真理源重复过多，存在漂移风险

`verify` 里写了大量：

- Gate 条件
- 指标阈值
- 阶段标准
- 修复命令

这本身可读，但问题是这些规则同时也存在于：

- gate engine
- metrics
- references
- 其他 skill

一旦任意一处变更，`verify` 文档就会过时。

### 3.2 放行层与实现层、测试层的指标语义仍有混淆

例如：

- `04_implement` 直接要求 C4
- 但 `09-test` 又被定义在 `05_verify`

这说明“测试覆盖何时形成、何时被验收”在链路上还没完全统一。

### 3.3 WAIVER 机制很强，但缺少更硬的治理边界

WAIVER 设计得不错，但如果审批、有效期、关闭条件不真正落到统一运行态，  
就容易从“例外机制”退化成“推进通道”。

## 4. 推荐优化方向

### 4.1 把 `verify` 收敛为放行入口，而不是规则仓库

建议：

- `verify` 负责执行、解释和报告
- 真正的阈值、阶段条件、指标口径由引擎或配置统一提供

### 4.2 强化“读取规则，不复制规则”

最佳实践是：

- 文档解释验证原则
- runtime 和配置提供真实条件

这样可以减少文档漂移。

### 4.3 把 WAIVER 做成更强的审计对象

至少需要：

1. 明确审批责任
2. 明确有效期
3. 明确何时关闭
4. 明确在哪个运行态文件里可追踪

## 5. 当前审查结论

- 职责定位：非常正确
- 当前合理性：高
- 主要问题：规则真理源重复，指标语义与其他 skill 仍有少量错位
- 最佳优化方向：收敛为证据化放行入口，把规则和阈值统一下沉

一句话结论：

> `12-verify` 已经接近整条链路的“放行中枢”，下一步最重要的是减少规则复制，确保它读取真理源而不是重新定义真理源。

# 2026-03-07 全项目详细优化方案

## 1. 文档定位

本文不是汇报稿，也不是泛化建议清单，而是一份面向后续重构与实施的**详细优化方案文档**。

目标是回答 4 个问题：

1. `spec-first` 当前到底哪里需要优化
2. 优化的原则应该是什么
3. 目标架构应长成什么样
4. 后续应该按什么顺序落地

本文基于前述审查结论，统一整理为一份可执行的方案基线。

---

## 2. 背景与问题定义

### 2.1 当前项目的正确部分

`spec-first` 当前的产品方向是成立的，核心目标也没有问题：

- 用结构化规范作为单一真理源
- 用追踪矩阵建立需求到实现、测试、变更的全链路关联
- 用 Gate / Coverage / Stage State 做自动化校验
- 用 Skill / Context Pack / Hook 让 AI 在规范约束下参与研发

这部分不是过度设计，而是项目成立的根基。

### 2.2 当前项目的主要问题

当前问题不在主干目标，而在“围绕主干长出的控制层和解释层”。

核心问题可收敛为 5 类：

1. **控制层边界重叠**
   - `catchup / status / plan / orchestrate`
   - `code-review / verify`
   - `spec-review / analyze`

2. **规则真理源分散**
   - Gate、Coverage、Layer、阈值规则散落在 skill、references、runtime、文档里

3. **阶段语义不稳**
   - `test` 的叙事与阶段定义冲突最明显

4. **命令面膨胀**
   - 一级命令和 skill 命令都在增长，层次感不够稳定

5. **副作用命令过重**
   - `archive / doctor / sync` 都承担了过多不同性质动作

### 2.3 本次优化的核心目标

本次优化不是推翻现有体系，而是做一次**收敛型重构**。

目标不是“增加更多能力”，而是：

- 保留治理主干
- 收缩控制层壳子
- 统一规则真理源
- 稳定阶段语义
- 压缩命令面
- 降低用户理解成本和维护成本

---

## 3. 优化总原则

### 原则 1：保留主干，不做方向性推翻

必须保留的核心能力：

- Stage 状态机
- Traceability Matrix
- Gate 校验
- Coverage 指标体系
- Feature 工作区
- Skill / Context 注入机制

这些能力是项目差异化所在，不应因“减复杂度”而削弱。

### 原则 2：规则只保留一个真理源

后续必须明确：

- **runtime/config** 定义规则
- **skill** 解释规则
- **references** 提供辅助说明
- **文档** 只做概念和用法说明

禁止 skill 文档再次定义一套完整规则体系。

### 原则 3：一个节点只解决一类核心问题

每个流程节点必须只回答一个核心问题：

- `spec`：做什么
- `design`：怎么做
- `task`：拆成什么任务
- `code`：如何实现
- `review`：质量是否可接受
- `verify`：是否可以正式推进

如果一个节点同时回答 2 到 3 个问题，后续一定继续长胖。

### 原则 4：控制层必须轻量化

控制层命令不是交付主链，不应承载过多文档生成、状态复述、风险讲解和下一步建议。

控制层只保留：

- 恢复
- 观察
- 规划
- 调度

每类只做最小必要动作。

### 原则 5：按用户心智重构，而不是按内部实现重构

从内部实现看，不同命令可能确实不同；但从用户视角，只关心：

- 我是谁
- 我现在在哪
- 下一步做什么
- 谁负责正式判断是否能推进

后续命令和 skill 设计必须优先服务这套心智，而不是服务现有文件结构。

## 4. 目标架构蓝图

## 4.1 目标分层

建议把系统明确分为 5 层：

### L1. 核心规则层（Truth Source）

职责：定义不可分叉的系统规则。

包含：

- Stage 枚举与状态流转
- Gate 条件
- Coverage 阈值
- Layer 定义
- Deliverable 最低要求
- Feature 状态约束

要求：

- 只有一份真理源
- 由 runtime / config 驱动
- skill 不得复制定义

### L2. 原子能力层（Runtime Commands）

职责：执行确定性动作。

包含：

- `id`
- `matrix`
- `stage`
- `gate`
- `metrics`
- `feature`
- `rfc`
- `defect`
- `trace`
- `validate`

要求：

- 输入输出稳定
- 行为可预测
- 不承载高层叙事

### L3. 阶段主链层（Delivery Flow）

职责：驱动交付主链推进。

建议保留的主链：

- `first`（认知底座，非 Feature 阶段）
- `init`
- `spec`
- `design`
- `research`（按需）
- `task`
- `code`
- `review`
- `verify`
- `archive`

这里建议把 `code-review` 重命名心智上归为 `review` 家族中的实现审查层，但是否真的改名可后置。

### L4. 控制辅助层（Control Plane）

职责：提供流程辅助，而不是交付内容。

只保留：

- `catchup`
- `status`
- `plan`
- `orchestrate`

要求：

- 轻量
- 不复制规则
- 不和主链竞争入口

### L5. 维护治理层（Maintenance Plane）

职责：非主链、偏运维和治理动作。

包含：

- `doctor`
- `sync`
- `feature`
- 其他宿主/安装/修复相关命令

要求：

- 不与主链平级强调
- 明确副作用范围
- 可审计

## 5. 节点级优化方案

## 5.1 入口层优化方案

### `first`

**保留理由**
- 它承担项目认知底座价值，尤其适合研发 / 架构进入陌生项目时使用。

**主要问题**
- 容易被误解为所有角色都应先执行。

**优化方案**
- 明确 `first` 不是 Feature 主链节点，而是“项目认知入口”。
- 文档中减少“默认所有人先跑 first”的暗示。
- 在 `onboarding` 中把 `first` 作为“推荐入口之一”，而非唯一入口。

### `onboarding`

**目标重定义**
- 统一角色入口层。

**优化方案**
- 输入：用户角色、当前是否已有 Feature、当前阶段、当前目标
- 输出：推荐后续入口
- 不自己承担过多主链动作

**推荐角色路由**
- 研发：`init / spec / task / code`
- 产品：`spec`
- 测试：`test / verify`
- 架构：`first / design / research`

### `init`

**目标重定义**
- Feature 工作区初始化器

**优化方案**
- 聚焦创建 Feature 工作区和基础文件
- 宿主自修复、环境诊断、全局 bootstrap 不再混入默认 `init` 叙事
- 保持对 `first` 的依赖检查，但不要继续扩展为维护命令

---

## 5.2 主链交付层优化方案

### `spec`

**保留定位**
- 需求定义主入口

**当前问题**
- 流程过厚，容易混入调研、决策、实施交接

**优化方案**
- `spec` 只负责输出：
  - 目标
  - FR
  - AC
  - 约束
  - 未决问题
- 不默认承担：
  - 技术比选
  - 实施任务建议
  - 过多后续调度建议

### `design`

**保留定位**
- 技术设计入口

**当前问题**
- 系统级设计维度仍偏薄

**优化方案**
- 补齐系统级设计模板：
  - 数据流
  - 模块边界
  - 接口契约
  - 风险点
  - 非功能约束
- 提高完成门槛，避免“有文档就算设计完成”

### `research`

**目标重定义**
- 不确定性处理分支

**优化方案**
- 仅在以下场景触发：
  - 技术方案对比
  - 性能/兼容性不确定
  - 外部依赖评估
- 不将其作为默认主链固定步骤

### `task`

**目标重定义**
- 任务规划器

**优化方案**
- 只产出 TASK 拆解、依赖、验收口径、优先级
- 不在 `task` 阶段隐性承担“执行调度器”职责
- 所有执行顺序建议尽量交给 `plan/orchestrate`

### `code`

**目标重定义**
- 实现执行入口

**优化方案**
- 只负责基于 `task_plan.md` 完成实现
- 只输出最小必要证据
- 不再承担“最终验证是否通过”的叙事

---

## 5.3 质量与放行层优化方案

### `code-review`

**目标重定义**
- 实现阶段审查器

**建议边界**
- 负责：
  - 单层/跨层质量审查
  - 合规与质量检查
- 不负责：
  - 正式阶段放行
  - Gate 最终裁决

**收敛建议**
- 长期把 `completion` 层完全移出 `code-review`
- `code-review` 仅保留 `single/cross`

### `verify`

**目标重定义**
- 唯一正式放行入口

**建议边界**
- 负责：
  - gate check
  - matrix check
  - 覆盖率判定
  - 阶段放行建议
- 不负责：
  - 重复定义一整套规则
  - 过多质量审查细节

**收敛建议**
- `verify` 只读取真理源，不在 skill 文本中重复定义大量阈值

### `test`

**这是当前优先级最高的问题点。**

#### 方案 A：前移为测试规格 / 建模入口（推荐）

适用理解：
- `test` 本质是“从 FR/AC 生成 TC 和测试规格”

则应：
- 前移到 `03_plan` 或 `04_implement` 之前/期间
- 成为实现前或实现并行的测试设计命令
- `verify` 负责最终执行放行

#### 方案 B：保留在 `05_verify`，但彻底改成验证测试命令

适用理解：
- `test` 本质是“验证阶段补齐/执行测试”

则应：
- 完全放弃“TDD 前置测试设计”的主叙事
- 改为验证阶段测试收口工具

#### 推荐

推荐采用**方案 A**。原因：

- 更符合 `FR -> AC -> TC -> 实现 -> 放行` 的认知顺序
- 能减少 `code` 和 `verify` 的时间语义冲突
- 能帮助测试角色更自然地前置参与

### `spec-review` 与 `analyze`

**优化方向**
- 统一到 review 家族心智模型中

建议区分：
- `spec-review`：单产物规格质量审查
- `analyze`：跨产物一致性分析

但对用户展示时，应强调二者是同一家族，而不是完全割裂的两个世界。

## 5.4 控制层优化方案

### 目标边界

- `catchup`：恢复上下文
- `status`：展示状态
- `plan`：决定下一步
- `orchestrate`：调度执行

### `catchup`

**当前问题**
- 偏重
- 写回倾向过强
- 与状态/规划重叠

**优化方案**
- 默认轻量恢复：
  - 当前 Feature
  - 当前阶段
  - 当前任务
  - 关键阻塞
  - 下一步命令
- 完整恢复作为扩展模式
- 默认不写运行态，按需写入交接摘要

### `status`

**当前问题**
- 带有一定推进裁决色彩
- 总分/健康分存在主观性

**优化方案**
- 保持只读观察层
- 优先展示结构化信号，而不是总分
- 只展示：
  - 当前阶段
  - 覆盖率缺口
  - 阻塞项
  - 风险项
  - 当前任务分布

### `plan`

**当前问题**
- 硬编码线性阶段建议过强
- 与 orchestrate 有交叉

**优化方案**
- 输出“下一步为什么是这个”
- 减少固定 stage->command 规则
- 改为基于：
  - 当前阶段
  - gate 状态
  - 覆盖率缺口
  - 阻塞项
  - 风险等级
  给出条件化建议

### `orchestrate`

**当前问题**
- 映射关系过线性
- 像硬编码 router

**优化方案**
- 从“固定阶段 -> 固定 skill”升级为“批次编排器”
- 根据当前状态决定：
  - 是执行主链节点
  - 是先恢复/规划
  - 是先补缺口
  - 还是直接放行

## 5.5 管理与维护层优化方案

### `feature-list / feature-current / feature-switch`

**优化方案**
- 合并为统一 `feature` 命令族
- skill 层也尽量统一对外心智：
  - `feature list`
  - `feature current`
  - `feature switch`

### `doctor`

**当前问题**
- 项目诊断 + 宿主修复绑定过紧

**优化方案**
- 拆成两层语义：
  - 项目诊断
  - 宿主修复
- 保留一个入口也可以，但输出必须明确区分：
  - 这是项目问题
  - 这是宿主问题
  - 这是将被修改的全局配置

### `sync`

**当前问题**
- 黑箱感强
- 输入和输出边界不够透明

**优化方案**
- 明确：
  - 输入来源
  - 将修改哪些文件
  - 修改依据是什么
  - 审计记录写到哪里

### `archive`

**当前问题**
- 复盘、归档、推进三件事绑太紧

**优化方案**
- `archive` 主职责只保留：
  - `retro.md`
  - 指标总结
  - 经验教训
  - 后续行动项
- 文件归档与阶段推进拆到后续编排动作或独立治理逻辑中

## 6. 规则真理源重构方案

这是整个优化方案的核心基础。

## 6.1 当前问题

当前规则存在于：

- skill 文档
- references
- runtime
- 模板
- 其他 skill 的引用文字

这导致：

- 难以维护
- 难以确认谁是最终标准
- AI 容易读到过期规则

## 6.2 目标方案

建立统一规则层，建议至少区分以下几类：

1. **Stage Rules**
   - 阶段枚举
   - 允许流转关系
   - 阶段最小产物

2. **Gate Rules**
   - 各阶段 gate 条件
   - FAIL / WAIVER / PASS 判定逻辑

3. **Coverage Rules**
   - C1-C9 定义
   - 阈值
   - 方向性规则

4. **Review Layer Rules**
   - single / cross / completion 语义
   - 各层入口映射

5. **Deliverable Rules**
   - 各阶段最小产物集合
   - 必填字段
   - 文件路径约束

## 6.3 Skill 层的改造要求

每个 skill 文档只保留：

- 它要解决什么问题
- 它在什么阶段触发
- 它的输入输出是什么
- 它会调用哪些 runtime 能力
- 它成功时会得到什么结果

不再完整复制：

- 所有阈值
- 所有 Gate 条件
- 所有层级定义
- 所有例外规则

## 7.1 收敛目标

从“命令很多但层次感弱”，收敛为“命令仍丰富，但分类稳定”。

- `init`
- `spec`
- `design`
- `research`（按需）
- `task`
- `code`
- `review`
- `verify`
- `archive`

### B. 控制命令
- `catchup`
- `status`
- `plan`
- `orchestrate`

### C. 审查命令
- `spec-review`
- `analyze`
- `code-review`（若未并入 `review`）

### D. 管理命令
- `feature`
- `doctor`
- `sync`

### E. 原子 runtime 命令
- `id`
- `matrix`
- `stage`
- `gate`
- `metrics`
- `trace`
- `validate`
- `rfc`
- `defect`

## 7.3 用户心智目标

- 要做专项审查，用审查命令
- 要处理环境/上下文/治理，用管理命令
- 要做底层确定性动作，用 runtime 命令

## 8. 分阶段落地计划

## P0：结构性冲突修复（最高优先级）

### 目标
解决最影响一致性的结构冲突。

### 范围
1. 重构 `test` 的阶段定位
2. 切清 `code-review` 与 `verify` 边界
3. 收敛 `catchup/status/plan/orchestrate`
4. 建立规则真理源唯一化方案

### 验收标准
- `test` 的时间语义与文档叙事一致
- `review` 和 `verify` 不再双重放行
- 控制层每个命令只有一个核心职责
- Gate/Coverage/Layer 规则只在一处正式定义

## P1：命令面与维护层收敛

### 目标
压缩用户心智成本。

### 范围
1. 合并 `feature-*`
2. 纯化 `doctor / sync / archive`
3. 收敛 `research` 的触发边界
4. 精简 skill 文档厚度

### 验收标准
- Feature 管理只保留一个命令族心智
- 维护类命令的副作用边界可解释
- 用户不再混淆主链命令和维护命令

## P2：角色入口与长期体验优化

### 目标
让系统从“可用”走向“自然”。

### 范围
1. 完成 `onboarding` 的角色入口体系
2. 统一 review 家族心智模型
3. 优化状态、规划、调度的呈现体验
4. 文档体系分层整理

### 验收标准
- 不同角色进入系统的第一步更自然
- review 家族结构稳定
- 控制层不再像重复命令集合

## 9. 风险与取舍

## 9.1 主要风险

1. **收敛时误伤主干价值**
   - 如果“减复杂度”被理解成“删追踪、删 Gate”，会伤到项目根基。

2. **文档改了，runtime 没改**
   - 会继续制造第二轮规则漂移。

3. **命令改名/合并导致迁移成本**
   - 需要设计兼容期或别名策略。

4. **局部优化破坏全局一致性**
   - 尤其是 `test`、`verify`、`orchestrate` 三者必须联动调整。

## 9.2 取舍原则

优化时优先级应遵循：

1. 一致性高于局部便利
2. 真理源统一高于文档丰富
3. 用户心智清晰高于内部实现舒服
4. 主干稳定高于辅助能力扩张

---

## 10. 建议的最终目标状态

优化完成后，理想中的 `spec-first` 应呈现为：

### 对用户
- 命令仍然强大，但层次很清楚
- 不会反复困惑“我现在应该用哪个命令”
- 不同角色能自然进入系统

### 对维护者
- 规则只在一处维护
- skill 文档不再成为另一套规则仓库
- 新增命令时更容易判断应归属哪一层

### 对 AI 协作
- skill 更像入口和工作流提示器
- runtime 更像唯一可靠执行器
- 上下文注入更稳定，误读规则概率更低

### 对产品本身
- 保留治理强度
- 降低控制层重量
- 从“治理骨架很强”进化为“默认顺畅的治理平台”

---

## 11. 最终建议

如果只保留一句总建议，那就是：

> **不要削弱 `spec-first` 的治理主干；真正该做的，是把治理主干外面那些重叠、冗余、漂移、过厚的控制层壳子削薄。**

更具体地说：

- 保留 `traceability / gate / stage / context`
- 收敛 `test / review / verify` 的语义
- 压缩 `catchup / status / plan / orchestrate` 的交叉
- 合并 `feature` 家族
- 纯化 `doctor / sync / archive`
- 统一规则真理源

这样做之后，项目不会变弱，反而会变得更像它原本想成为的样子：

> **一个规范驱动、可追踪、可校验、可放行、可被 AI 稳定消费的研发流程引擎。**

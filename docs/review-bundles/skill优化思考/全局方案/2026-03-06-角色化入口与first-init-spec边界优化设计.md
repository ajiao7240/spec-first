# 2026-03-06 角色化入口与 first-init-spec 边界优化设计

## 1. 问题背景

当前如果把 `first` 当成统一入口，会天然偏研发视角。

这是因为 `first` 的核心输出更偏向：

- 技术栈
- 代码结构
- 模块边界
- API / 数据模型
- 系统代码背景

这对研发、架构师很有价值，但对产品、测试并不是最佳第一入口。

因此，系统需要回答一个更基础的问题：

> 不同角色进入 `spec-first` 时，第一步应该是什么？

## 2. 角色差异

### 2.1 研发

研发最先需要知道的是：

- 项目是什么
- 当前系统代码是怎么组织的
- 从哪里开始看代码
- 当前需求开发应该落在哪个模块

所以研发入口适合从 `first` 开始。

### 2.2 产品

产品最先需要知道的是：

- 项目当前做了什么
- 已有能力边界是什么
- 现在要定义的新需求是什么
- 当前应该进入哪个 Feature 或创建哪个 Feature

产品不需要先看：

- 调用链
- Git hooks
- 宿主 bootstrap
- 代码目录细节

所以产品不适合把 `first` 当第一入口。

### 2.3 测试

测试最先需要知道的是：

- 当前需求是什么
- 哪些模块会受影响
- 风险点在哪
- 现有测试覆盖缺口在哪
- 现在应该进入 `test` 还是 `verify`

测试也不需要先进入完整代码认知链路。

### 2.4 架构 / 技术负责人

架构师最关心的是：

- 系统当前结构
- 技术约束与风险
- 模块边界与耦合点
- 本次需求会不会影响整体设计

所以架构入口适合从 `first` 开始，但应该是“架构视图优先”的 `first`。

## 3. 最佳实践：统一认知底座 + 角色化入口

最佳实践不是：

- 所有人都先执行 `first`

而应该是：

- 系统内部先建立统一认知底座
- 不同角色从不同入口消费这套认知底座

也就是说，`first` 是“底座生产者”，但不一定是每个角色的第一入口。

## 4. 推荐职责边界

### 4.1 `first`

`first` 的最佳职责：

- 面向研发 / 架构
- 建立项目全局认知
- 生成可复用的认知底座

一句话：

> `first` 解决“我是谁、项目是什么”

### 4.2 `init`

`init` 的最佳职责：

- 面向研发主链
- 把认知底座转成当前需求开发上下文
- 初始化需求目录空间
- 给出进入 `spec` 的明确起点

一句话：

> `init` 解决“我要做什么、从哪里开始”

### 4.3 `spec`

`spec` 的最佳职责：

- 面向产品 / 研发共同协作
- 做需求理解与正式定义
- 产出 FR / AC / NFR / 边界 / 开放问题

一句话：

> `spec` 解决“具体需求怎么定义”

### 4.4 `test / verify`

`test` / `verify` 的最佳职责：

- 面向测试 / 研发共同协作
- 明确如何验证当前需求
- 输出测试用例、覆盖关系、放行依据

一句话：

> `test / verify` 解决“怎么验证、是否可放行”

## 5. 推荐角色入口

| 角色 | 入口目标 | 推荐链路 |
|---|---|---|
| 研发 | 理解代码背景并开始需求开发 | `first -> init -> spec` |
| 产品 | 快速理解业务背景并进入需求定义 | `角色入口 -> spec` |
| 测试 | 快速理解需求与风险并进入验证准备 | `角色入口 -> test / verify` |
| 架构 | 快速理解系统结构并进入设计决策 | `first -> design / research` |

## 6. 为什么产品和测试不应先走 first

### 6.1 对产品来说太重

产品并不需要先消费：

- 完整代码结构
- 技术依赖细节
- 模块调用链

产品真正需要的是：

- 当前业务背景
- 已有能力与缺口
- 当前 Feature 的目标
- 需求定义入口

### 6.2 对测试来说路径不自然

测试并不需要先消费：

- 全量研发上下文
- 复杂技术栈识别
- 代码导向的开发入口

测试更需要：

- 需求摘要
- 影响范围
- 风险点
- 测试面
- 当前覆盖状态

## 7. 推荐系统设计

## 7.1 保留 `first`，但明确其目标用户

建议把 `first` 定义为：

- 默认面向研发 / 架构
- 负责生产统一认知底座
- 不强制要求所有角色都显式执行

## 7.2 新增角色化入口层

推荐新增一个统一角色入口层，可基于现有 `onboarding` 演进，也可以新增单独入口 skill。

职责：

- 识别当前角色
- 读取统一认知底座
- 按角色输出裁剪后的上下文
- 引导进入正确的下一步 skill

## 7.3 角色化入口的建议输出

### 产品入口

输出应包含：

- 项目当前业务背景
- 当前已支持能力
- 当前 Feature / 当前阶段
- 新需求建议从哪里进入
- 下一步命令：`/spec-first:spec`

### 测试入口

输出应包含：

- 当前需求摘要
- 风险区域
- 受影响模块
- 当前测试覆盖缺口
- 下一步命令：`/spec-first:test` 或 `/spec-first:verify`

### 架构入口

输出应包含：

- 当前系统结构
- 核心模块边界
- 风险和耦合点
- 下一步命令：`/spec-first:design` 或 `/spec-first:research`

## 7.4 统一认知底座建议

建议由 `first` 额外生成一个结构化认知文件，例如：

- `docs/first/.summary.json`

建议字段：

- `project_type`
- `tech_stack`
- `core_modules`
- `domain_summary`
- `api_summary`
- `risk_summary`
- `recommended_entry_points`
- `generated_by_role`
- `generated_at`

这样：

- 研发入口可以消费完整视图
- 产品入口可以消费业务裁剪视图
- 测试入口可以消费风险与覆盖裁剪视图

## 8. 推荐演进路径

### P0

- 明确文档边界：
  - `first` 面向研发 / 架构
  - `init` 面向研发主链
  - `spec` 面向产品 / 研发协作
  - `test/verify` 面向测试 / 研发协作

### P1

- 给 `first` 增加结构化认知摘要产物
- 让 `init` 消费结构化摘要，而不是猜 markdown

### P2

- 基于 `onboarding` 或新增入口 skill，建设角色化入口层
- 产品和测试可以不经过 `first` 命令，也能拿到适合自己的项目上下文

### P3

- 让不同角色入口共享同一认知底座，但输出按角色裁剪
- 建立“角色 -> 推荐 skill 链路”的稳定导航

## 9. 最终结论

最合理的设计不是：

- 所有人都从 `first` 开始

而是：

- `first` 负责生成统一认知底座
- 不同角色通过不同入口消费它

因此，最佳边界应当是：

- `first`：项目认知入口（偏研发 / 架构）
- `init`：研发需求开发起点
- `spec`：需求理解与定义入口（偏产品 / 研发协作）
- `test / verify`：验证入口（偏测试 / 研发协作）

一句话总结：

> 对研发，`first` 是起点；对产品、测试，`first` 是底座，但不一定是入口。

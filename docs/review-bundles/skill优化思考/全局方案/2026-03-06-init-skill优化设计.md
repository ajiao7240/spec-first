# 2026-03-06 init skill 优化设计

## 1. 背景

当前 `spec-first` 的新人进入链路大致是：

`first -> init -> spec -> design -> task -> code -> verify`

但从新人使用视角看，`init` 的定位仍然偏“参数收集器 + 目录生成器”，没有充分承担“把项目认知转成需求开发起点”的职责。

这会导致两个问题：

1. 新人虽然跑完了 `first`，但并不知道当前应该做什么、从哪里开始。
2. `init` 虽然创建了 Feature 空间，但没有把“项目背景”转换成“当前需求开发上下文”。

## 2. 三个 skill 的最佳职责边界

### 2.1 `first`

`first` 解决：

- 我是谁
- 项目是什么
- 当前系统代码内容和背景是什么

它的核心价值是建立全局认知，而不是直接进入需求定义。

### 2.2 `init`

`init` 解决：

- 我要做什么
- 我应该从哪里开始
- 当前 Feature 的需求目录空间如何初始化

它的核心价值是把 `first` 的认知结果转成当前需求开发上下文，并初始化后续链路可消费的工作区。

### 2.3 `spec`

`spec` 解决：

- 当前这个需求到底是什么
- 功能边界、验收标准、非功能要求、开放问题是什么

它的核心价值是完成需求理解和正式定义。

## 3. 最佳实践下的目标链路

最佳实践不是：

- `first = 项目分析`
- `init = 问参数 + 建目录`
- `spec = 写需求`

而应该是：

- `first = 项目认知入口`
- `init = 需求开发起点`
- `spec = 需求定义入口`

压缩成一句话：

- `first` 解决“我是谁、项目是什么”
- `init` 解决“我要做什么、从哪里开始”
- `spec` 解决“具体需求怎么定义”

## 4. init skill 应该解决的问题

`init` 应该聚焦解决 4 个问题：

1. 把项目认知结果转换成当前 Feature 开发上下文。
2. 帮新人明确“当前这次要做的需求开发范围”。
3. 初始化后续 `spec/design/task/code/verify` 可消费的需求目录空间。
4. 输出清晰的下一步动作，让用户自然进入 `spec`。

因此，`init` 的本质不是“创建目录”，而是“项目认知到需求开发的交接点”。

## 5. 当前实现的合理点

当前实现已经有一些方向是对的：

1. 强制要求先完成 `first`
   - 保证用户不是在毫无上下文时直接开始需求开发。

2. 初始化骨架文件较完整
   - `stage-state.json`
   - `findings.md`
   - `task_plan.md`
   - `traceability-matrix.md`
   - `constitution.md`

3. 幂等与恢复做得较稳
   - 已有 Feature 不覆盖
   - `.spec-first/current` 可自愈
   - 临时目录 + rename + registry lock 的处理合理

4. 交互流程比纯命令参数更适合新人
   - `feat/mode/size/platforms/title/feature-id/bootstrap` 的逐步收集对新手友好

## 6. 当前实现的主要问题

### 6.1 定位偏工具化，不够“认知到开发起点”

现在 `init` 更多是在做：

- 检查 `first` 是否跑过
- 问几个参数
- 创建 Feature 目录

但它没有真正解决：

- 基于当前项目背景，我这次需求应该从哪里切入
- 当前初始化出的 Feature 与项目现状是什么关系
- 下一步为什么是去做 `spec`

### 6.2 对 `first` 的消费方式太脆弱

当前通过读取 markdown 文本再正则提取摘要，这种方式不稳定。

问题：

- 文档措辞一变，摘要就退化
- `init` 无法可靠消费 `first` 的认知结果

最佳实践应改为：

- `first` 输出结构化摘要
- `init` 消费结构化摘要

### 6.3 职责边界过宽

当前 `init` 还承担了较多附加责任：

- 补 `.claude/settings.json`
- 装 hooks
- 注册 AI hooks
- 刷新 skill 命令
- 可选 bootstrap

这些能力有价值，但不属于 `init` 的核心职责。

最佳实践下应区分：

- `init` 核心职责：初始化需求开发空间
- `doctor/setup/update` 扩展职责：环境修复、宿主接入、命令注册

### 6.4 文档承诺和实际输出不完全一致

`SKILL.md` 里对 P5 的定义比实际 CLI 输出更完整，但当前 CLI 只输出初始化成功和目录位置。

这会削弱新人对“当前我已经进入哪一步”的感知。

## 7. 推荐优化方向

## 7.1 重新定义 init 的输入

`init` 的输入不应只是：

- feat
- mode
- size
- platforms

还应包含“从 `first` 得到的项目背景摘要”。

推荐输入模型：

- 项目类型
- 核心技术栈
- 主要业务域 / 模块入口
- 推荐关注区域
- 当前初始化目标（要做的新需求方向）

## 7.2 重新定义 init 的输出

`init` 的输出不应只是：

- Feature ID
- 目录路径

还应输出：

- 当前 Feature 的目标摘要
- 当前阶段：`00_init`
- 已初始化的关键产物
- 推荐下一步命令：`/spec-first:spec`
- 为什么下一步是 `spec`

## 7.3 建立结构化认知桥接

推荐让 `first` 额外输出一个结构化文件，例如：

- `docs/first/.summary.json`

建议内容：

- project_type
- tech_stack
- core_modules
- domain_summary
- api_surface_summary
- recommended_entry_points
- first_mode
- generated_at

然后 `init` 直接消费该文件，而不是猜测 markdown 内容。

## 7.4 把 init 变成“需求开发起点向导”

推荐把 `init` 的交互分成 3 段：

### 段 1：展示项目背景摘要

告诉新人：

- 这是一个什么项目
- 核心技术栈是什么
- 主要模块/业务边界是什么

### 段 2：确认当前需求开发目标

告诉新人：

- 你这次准备创建什么 Feature
- 预计是新功能还是迭代
- 涉及哪些平台

### 段 3：初始化空间并交接下一步

最终输出：

- 当前 Feature 已创建
- 当前阶段已就绪
- 下一步应执行 `spec`

## 7.5 压缩非核心职责

建议把以下能力下沉为“附加能力”：

- hooks 安装
- AI hooks 注册
- Skill 命令刷新
- 宿主 bootstrap

保留策略：

- 默认不阻塞 `init`
- 可以通过 `--with-runtime-setup` 或后续 `doctor/update` 再做

这样 `init` 更聚焦，也更稳定。

## 8. 推荐目标流程

理想流程如下：

1. 用户执行 `first`
2. 系统生成认知产物 + 结构化摘要
3. 用户执行 `init`
4. `init` 读取结构化摘要并展示项目背景
5. `init` 收集本次需求开发信息
6. `init` 初始化 `specs/{featureId}/`
7. `init` 输出当前阶段状态和下一步命令
8. 用户进入 `spec`

也就是：

`项目认知 -> 开发起点 -> 需求定义`

## 9. 推荐改造优先级

### P0

1. 补齐 `init` 结束时的阶段摘要输出
2. 明确输出“下一步执行 `/spec-first:spec`”
3. 保证 `SKILL.md` 与 CLI 行为一致

### P1

1. 让 `first` 生成结构化摘要文件
2. 让 `init` 改为消费结构化摘要
3. 减少对 markdown 文本正则推断的依赖

### P2

1. 将宿主初始化副作用与 `init` 核心职责解耦
2. 把 `init` 打造成完整的“需求开发起点向导”

## 10. 结论

`init` skill 的最佳实践定位应当是：

> 它不是一个目录初始化工具，而是新人从“理解项目”进入“开始需求开发”的正式入口。

所以它最合理的职责定义是：

- `first`：建立项目认知
- `init`：建立当前需求开发上下文并初始化空间
- `spec`：完成需求理解与正式定义

这也是当前 `spec-first` 链路里最应该强化的边界。

# 2026-03-07 全局视角达标性评估

## 1. 评估问题

本次全量审查最终要回答的是：

> 当前这套 `spec-first` skill 链路，是否已经能稳定达成“流程顺畅、健壮性高、质量高、支持多角色协作”的目标？

结论先说：

> **能达成 70% 到 80% 的目标，但还没有达到“默认顺畅、默认高质量、默认低歧义”的成熟状态。**

它已经具备非常好的治理骨架，但控制层边界、命令面和规则真理源还没有完全收敛。

## 2. 已经成立的能力

### 2.1 主链骨架已经成立

当前从：

- `first`
- `init`
- `spec`
- `design`
- `task`
- `code`
- `code-review`
- `test`
- `verify`
- `archive`

这条主链，已经能看出清晰的阶段式研发治理结构。

### 2.2 证据化和追踪意识很强

这套系统最强的部分在于：

- `traceability-matrix`
- Gate / Coverage
- `findings.md`
- 运行态恢复
- verify 证据铁律

这意味着它不是“流程口号”，而是在努力把研发行为变成有证据的系统动作。

### 2.3 角色分层已经开始形成

通过前面几个入口类 skill 的审查，可以确认：

- `first` 更适合研发 / 架构
- `init` 是研发需求开发起点
- 产品更适合从 `spec` 或角色化入口进入
- 测试更适合从测试规格 / verify 链路进入

这说明它已经不是单一开发者工具，而是在往多角色协作系统演进。

## 3. 当前还没完全达标的关键原因

### 3.1 控制层边界仍然重叠

当前至少有 4 组控制层命令存在明显重叠：

- `catchup / status / plan`
- `plan / orchestrate`
- `spec-review / analyze`
- `code-review / verify`

从系统内部看，这些能力都有差异。  
但从用户心智看，边界还不够清晰，容易出现：

- 不知道该先跑哪个
- 多个命令都在回答“下一步做什么”
- 同一件事被不同命令各讲一遍

### 3.2 规则真理源存在漂移风险

这是当前最值得警惕的问题之一。

大量规则同时存在于：

- skill 文档
- references
- gate/metrics runtime
- 其他 skill 的引用文本

这样一旦某处更新，其他地方就可能过时。  
`verify`、`status`、`plan`、`code` 都不同程度存在这个问题。

### 3.3 测试链路的时间位置还没收敛

`09-test` 是目前主链里最明显的时间语义冲突点：

- 文档叙事偏向“实现前测试规格 / TDD 前置”
- 阶段位置却被放在 `05_verify`

这会影响：

- `code`
- `verify`
- 阶段推进

也就是它不仅是单命令问题，而是全链路问题。

### 3.4 命令面已经开始膨胀

真正的冗余不在主链，而在辅助命令：

- `feature-list/current/switch`
- `catchup/status/plan`
- `spec-review/analyze`

这会让新用户的命令选择成本偏高。

### 3.5 维护类和关闭类命令耦合了过多副作用

例如：

- `doctor` 同时操作项目和双宿主全局配置
- `archive` 同时做复盘、归档和阶段推进
- `sync` 定义不够透明

这些命令本来应该是辅助治理能力，  
但现在有的已经像“超级命令”，权责不够干净。

## 4. 从不同角色看，是否达标

### 4.1 研发视角

对研发来说，当前系统已经相当可用，尤其适合：

- 有一定流程纪律的个人或小团队
- 需要追踪和证据化的功能开发

但还没达到“默认顺畅”的程度，因为：

- 控制层命令重叠
- 测试位置冲突
- review/verify 边界还不够干净

### 4.2 产品视角

对产品来说，系统已经能进入，但还不够自然。

原因是：

- `onboarding` 还不够像统一入口
- `spec` 很强，但仍偏厚
- 辅助命令的产品语义还不够直观

### 4.3 测试视角

对测试来说，系统价值明显，但链路位置还没完全打通。

核心问题就是：

- `09-test` 到底是测试规格还是验证阶段命令

这一步不收敛，测试角色的主入口就始终有歧义。

### 4.4 架构 / 技术负责人视角

对架构负责人来说，这套系统已经有明显价值：

- 设计和追踪意识强
- verify 和 analyze 能形成治理闭环

但最需要继续收敛的是：

- 规则真理源
- 控制层边界
- 状态语义统一

## 5. 当前最关键的 5 个系统级短板

1. `test` 的阶段定位与 TDD 主线冲突
2. 控制层命令边界重叠，尤其是 `catchup/status/plan/orchestrate`
3. `verify/status/plan` 等命令里存在大量规则复制
4. `feature` 相关命令和审查类命令的命令面冗余
5. `archive/doctor/sync` 这类强副作用命令的边界还不够纯

## 6. 最终判断

如果问题是：

> 当前系统“能不能辅助完成高质量研发流程”？

答案是：

> **能，而且已经具备明显优势。**

如果问题是：

> 当前系统“是否已经达到最佳实践下的低歧义、低学习成本、低漂移、高一致性的成熟状态”？

答案是：

> **还没有。**

它现在更像：

- **一个治理骨架已经很强、但控制层和命令面还需要一次系统收敛的研发平台雏形。**

## 7. 一句话结论

> `spec-first` 当前已经有很强的研发治理潜力，也已经能支撑中小团队按阶段推进；但要真正达成“默认顺畅、默认健壮、默认高质量”，下一步必须收敛控制层边界、统一规则真理源、重构测试阶段定位，并压缩辅助命令面。

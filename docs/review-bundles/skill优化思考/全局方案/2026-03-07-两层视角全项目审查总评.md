# 2026-03-07 两层视角全项目审查总评

## 1. 审查目标

本文从两个层次对当前 `spec-first` 项目进行整体审查：

1. **全局整体流程视角**：判断这套系统在产品目标、阶段主链、命令面、控制层、规则体系上是否自洽。
2. **流程节点细节视角**：逐类检查各节点是否职责清楚、边界稳定、是否存在重复、漂移或过度复杂化。

本文不是复述单个 skill 的局部问题，而是给出：

- 当前项目到底是不是在解决正确的问题
- 当前流程骨架是否成立
- 复杂度主要长在什么位置
- 哪些能力必须保留
- 哪些设计已经开始过度复杂化
- 后续应该按什么优先级重构

---

## 2. 项目在解决什么问题

从 `README.md` 的产品定义看，`spec-first` 要解决的不是“如何多写几份文档”，而是研发治理中的 4 类核心问题：

1. **真理源分散**  
   需求、设计、实现、测试、变更经常分散在不同文档、聊天、口头沟通和代码里，导致理解漂移。

2. **追踪链断裂**  
   团队无法稳定回答：
   - 这个实现对应哪个需求？
   - 这个测试覆盖了哪个 AC？
   - 这个变更为什么发生？

3. **质量校验过度依赖人工**  
   阶段是否能推进、产物是否完整、覆盖是否达标，往往靠人判断，稳定性不足。

4. **AI 缺少稳定上下文**  
   AI 容易基于碎片信息输出结果，出现“理解过期”“忘记规范”“代码与文档失配”的问题。

因此，项目提出的核心解法是：

- 以**结构化规范**作为单一真理源
- 以**追踪矩阵**建立全链路可追溯关系
- 以**Gate / Coverage / 阶段状态机**形成自动化校验
- 以**Skill + Context 注入**把 AI 变成受约束的执行者，而不是自由发挥的写作者

这个目标本身是成立的，不属于过度设计。

---

## 3. 总评结论

一句话总评：

> `spec-first` 的**产品目标正确、流程主干成立、治理价值明显**；当前真正的问题不是“可追踪、可校验做得太多”，而是**围绕这些目标长出了过厚的控制层、命令层和规则解释层**。

更展开一点：

- **不是目标过度设计**：`traceability + gate + coverage + stage + AI context` 正是本项目的核心价值，不是附加装饰。
- **是实现开始复杂化**：复杂度主要集中在控制层边界、命令面膨胀、规则真理源分散、阶段语义不稳。
- **当前状态可用，但不够成熟**：已能支撑中小团队按阶段推进，但距离“默认顺畅、默认低歧义、默认低维护成本”的成熟状态还有明显差距。

如果用一句更直白的话概括：

> 这不是一个“方向错了”的项目，而是一个“主干长对了，但周边壳子长胖了”的项目。

---

## 4. 第一层视角：全局整体流程审查

## 4.1 全局流程骨架是否成立

结论：**成立。**

当前项目已经具备完整的阶段主链与状态机语义：

- `00_init`
- `01_specify`
- `02_design`
- `03_plan`
- `04_implement`
- `05_verify`
- `06_wrap_up`
- `07_release`
- `08_done / 09_cancelled`

这说明项目不是“文档集合”，而是明确在尝试构建一个**规范驱动的研发流程引擎**。

从这个角度看，全局流程主干是清楚的，产品定位和架构方向也是一致的。

## 4.2 全局价值是否自洽

结论：**自洽。**

项目主叙事是：

- 规范即契约
- 规范即真理源
- 全链路追踪
- 自动化门禁
- AI 原生协作

这几件事之间不是拼盘关系，而是互相支撑：

- 没有规范真理源，追踪会失焦
- 没有追踪，门禁无法判断覆盖完整性
- 没有门禁，阶段推进仍旧靠人拍脑袋
- 没有结构化上下文，AI 无法稳定消费规范

所以从全局产品逻辑看，这套系统不是“为了复杂而复杂”，而是在建立一套完整治理闭环。

## 4.3 全局主要问题不在主干，而在控制层

结论：**复杂度主要集中在控制层，而不是主链交付层。**

当前最显著的问题不是 `spec / design / task / code` 这些主链节点本身，而是以下几组辅助/控制能力：

- `catchup / status / plan / orchestrate`
- `spec-review / analyze`
- `code-review / verify`
- `feature-list / feature-current / feature-switch`

这些命令在系统内部各自有理由存在，但从用户心智看，边界还不够收敛，容易出现：

- 不知道先用哪个
- 多个命令都在回答“下一步做什么”
- 多个命令都在输出“状态 / 风险 / 建议”
- 同一件事从不同入口各讲一遍

这会让系统从“治理平台”滑向“控制层命令拼盘”。

## 4.4 全局规则体系存在真理源漂移风险

结论：**这是当前最需要警惕的问题之一。**

目前 Gate、Coverage、Layer、阈值、阶段推进条件等规则，不只存在于 runtime，也存在于：

- Skill 文档
- references 文档
- 示例模板
- 其他 skill 的引用说明

这会造成一个典型风险：

> runtime 已更新，但 skill 文档仍保留旧规则；或者某个 skill 的引用口径与当前 gate engine 不一致。

一旦发生漂移：

- 用户会被误导
- AI 会读到过期规则
- 审查/放行口径会失真

因此，全局上最需要收敛的是：

> **规则只有一个真理源；skill 只负责解释和引导，不负责重新定义规则。**

## 4.5 全局命令面已经开始膨胀

结论：**命令面正在逼近用户学习成本上限。**

当前项目已有较多 CLI 命令，加上 Skill 层又派生出一套更接近“任务入口”的命令体系。问题不在于命令多，而在于：

- 有些命令是原子能力
- 有些命令是阶段入口
- 有些命令是控制/观察/恢复工具
- 有些命令是维护类命令

但这些层级在用户视角里没有完全分层呈现，导致“命令是很多，但层次感不够强”。

这会影响：

- 新用户首次上手
- 多角色用户选择正确入口
- 维护者持续扩展命令时的边界感

## 4.6 全局成熟度判断

结论：**整体可用，尚未收敛。**

当前系统适合：

- 有流程纪律的个人开发者
- 小型到中型研发团队
- 对追踪、证据、阶段治理有明确需求的场景

但还不够适合“默认低学习成本的大规模推广”，原因在于：

- 控制层边界仍有重叠
- 规则真理源仍然分散
- 测试节点时间语义冲突
- 命令面还没有压缩到最佳心智规模

因此，全局成熟度可判断为：

> **治理骨架已经成立，但还没有完成控制层收敛。**

## 5.2 节点分组审查

### A. 入口层：`first / onboarding / init`

#### 判断

这一组节点承担的是“如何进入系统”的职责。当前方向整体正确，但入口体系还没有完全角色化。

#### 结论

- `first` 应保留，但定位为**项目认知底座**，而非所有角色共同入口。
- `onboarding` 应强化为**统一入口层**，识别角色后再推荐后续入口。
- `init` 应保留为**研发需求开发起点**，但不应继续承载太多宿主/环境/修复逻辑。

#### 当前问题

- 所有角色并不适合走同一套入口链路。
- 入口层对“角色差异”虽有意识，但还未完全产品化。
- `init` 与维护类职责仍有耦合倾向。

#### 推荐方向

- 形成“统一认知底座 + 角色化入口分发”的结构。
- 研发、产品、测试、架构看到的推荐路径不必相同。

### B. 主链交付层：`spec / design / research / task / code`

#### 判断

这是当前系统最稳的一层，问题主要不是“没有价值”，而是“个别节点太厚”。

#### 结论

- `spec` 是当前最强节点之一，应保留其中心地位。
- `design`、`task`、`code` 都有明确存在价值。
- `research` 更适合做**按需分支**，而不是潜在默认主链。

#### 当前问题

- `spec` 太容易继续吸收调研、决策和实施交接职责。
- `task` 有时不仅在拆任务，还在隐性决定执行顺序。
- `code` 文档中混入了部分验证语义。

#### 推荐方向

- `spec`：只解决“要做什么”
- `design`：只解决“技术上怎么做”
- `research`：只解决“存在不确定性时如何证据化比较方案”
- `task`：只解决“如何拆成可执行任务”
- `code`：只解决“如何按既定任务落实现”

### C. 质量与放行层：`code-review / test / verify / spec-review / analyze`

#### 判断

这是当前最需要收敛的一层。

#### 结论

- `code-review` 和 `verify` 都必要，但边界必须切清。
- `test` 是当前全项目最突出的单点冲突。
- `spec-review` 与 `analyze` 应进入统一 review 心智模型，而不是继续散开。

#### 当前问题

1. `code-review` 与 `verify` 都涉及“检查 / 放行 / 层级”语义。  
2. `test` 的叙事偏向前置测试设计，但阶段触发又被放在 `05_verify`。  
3. `analyze` 的价值成立，但不应再继续向 gate 真理源靠近。  
4. review 家族命令目前对用户来说仍不够成体系。

#### 推荐方向

- `code-review`：实现阶段质量审查
- `verify`：唯一正式放行入口
- `test`：二选一重定义  
  - 要么前移为测试规格/建模入口  
  - 要么保留在 `05_verify`，但彻底改成验证测试命令
- `spec-review` / `analyze`：并入统一 review 家族认知

### D. 控制与恢复层：`catchup / status / plan / orchestrate`

#### 判断

这是当前最“长胖”的一层。

#### 结论

- 四者都有存在价值，但重叠过多。
- 当前最需要压缩的是这一组，而不是主链交付层。

#### 推荐边界

- `catchup`：恢复上下文
- `status`：观察当前状态
- `plan`：决定下一步为什么这么走
- `orchestrate`：执行调度与推进

#### 当前问题

- 多个命令都在输出“当前状态 / 风险 / 下一步建议”。
- `orchestrate` 当前映射过线性，像一个硬编码 stage router。
- `plan` 当前更像半个 orchestrator。
- `catchup` 当前比“恢复工具”更接近“重型会话总结器”。

#### 推荐方向

- 全部收缩到最小职责边界。
- 减少对同一信息的多次生成。
- 让控制层从“重型解释器”变成“轻量决策/观察/调度器”。

### E. 管理与维护层：`doctor / sync / feature-* / archive`

#### 判断

这些能力都有必要，但都不适合继续以“超级命令”方式生长。

#### 结论

- `feature-list/current/switch` 应合并成 `feature` 命令族。
- `doctor` 应拆成“项目诊断”和“宿主修复”。
- `sync` 需要提升透明度和可审计性。
- `archive` 应以复盘沉淀为主，不应继续绑住所有收尾动作。

#### 当前问题

- 功能虽然必要，但命令面心智成本偏高。
- 副作用命令的权限边界不够清晰。
- 归档类动作和流程推进类动作耦合过深。

#### 推荐方向

- 统一命令族
- 拆分副作用层次
- 强化可审计性

## 6. 哪些必须保留，哪些属于过度复杂化

## 6.1 必须保留的能力

以下能力不只是“可以有”，而是 `spec-first` 作为产品成立所必需的：

1. 阶段状态机
2. `traceability-matrix`
3. Gate 校验
4. 最小必要覆盖率体系
5. Feature 工作区与运行态文件
6. AI 上下文注入 / Skill 路由

如果这些被削弱，项目会失去核心差异化。

## 6.2 已经开始过度复杂化的部分

以下部分不是没有价值，而是当前实现形态已经开始超过最佳复杂度：

1. 控制层命令过多且重叠
2. 规则在多处重复定义
3. `test` 的阶段与叙事冲突
4. 维护/收尾类命令承担过多不同性质动作
5. 文档和 skill 文本逐渐膨胀成“第二套系统”

也就是说：

> 当前真正的复杂化，不在主干目标，而在解释层、控制层和命令层。

---

## 7. 重构优先级建议

## P0：必须优先处理

1. **重构 `test` 的阶段定位**  
   这是当前最明显的单点冲突，会影响 `plan / orchestrate / verify / code`。

2. **切清 `code-review` 与 `verify` 的边界**  
   明确“审查”和“正式放行”分别由谁负责。

3. **压缩 `catchup / status / plan / orchestrate` 重叠**  
   这是当前控制层最大的问题源。

4. **统一规则真理源**  
   Gate、阈值、Layer、Coverage 规则必须由 runtime/config 统一定义。

## P1：尽快处理

1. 合并 `feature-list/current/switch` 为 `feature` 命令族
2. 纯化 `doctor / archive / sync` 职责
3. 让 `research` 明确成为按需触发分支
4. 精简 `spec / task / code` 中的越界叙事

## P2：持续优化

1. 完成 `onboarding` 的统一角色入口设计
2. 统一 `review` 命令家族的产品心智模型
3. 精简 skill 文本，把规则解释与规则定义彻底分层

---

## 8. 最终结论

从两层视角综合判断：

### 结论 1：项目方向正确

`spec-first` 要解决的问题是真实且高价值的，尤其适用于：

- 需要规范驱动的研发流程
- 需要追踪与证据化的团队
- 希望让 AI 在受约束前提下参与研发的场景

### 结论 2：主干成立，但控制层未收敛

阶段状态机、规范真理源、追踪矩阵、Gate、Skill 路由，这些主干都已经成立。  
当前的主要风险不在主链，而在围绕主链长出的控制层。

### 结论 3：不是“可追踪/可校验过度设计”，而是“控制层过度复杂化”

项目最值得保留的，恰恰就是：

- 可追踪
- 可校验
- 可放行
- 可被 AI 稳定消费

真正需要削减和重构的，是：

- 重叠命令
- 规则复制
- 模糊边界
- 超级命令

### 结论 4：当前最合理的重构原则

> **保留治理主干，压缩控制层壳子；保留规则能力，收缩规则叙事；保留追踪与放行，减少命令重叠与职责漂移。**

---

## 9. 一句话总评

> `spec-first` 不是做错了方向，而是已经走到了一个“必须做控制层收敛”的阶段：主干值得保留，复杂度需要减脂，下一步的关键不是再加能力，而是把已有能力重新分层、收口、定边界。

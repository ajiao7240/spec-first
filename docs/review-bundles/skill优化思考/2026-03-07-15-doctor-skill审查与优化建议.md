# 2026-03-07 15-doctor skill 审查与优化建议

## 1. skill 要解决什么问题

`15-doctor` 解决的是：

1. 检查项目环境和宿主配置是否可用
2. 发现 MCP、skills、hooks、配置等缺失项
3. 在用户确认后执行自动修复

一句话：

> `15-doctor` 解决“本地环境和宿主配置是否足以正常运行 spec-first”。 

## 2. 当前实现是否合理

整体判断：有价值，但权限面和职责面都偏大。

### 2.1 合理的地方

1. 环境诊断是必要能力
2. manifest 驱动检查比硬编码好
3. 修复前后差异展示是对的

## 3. 当前主要问题

### 3.1 把项目诊断和双宿主全局修复绑得太紧

当前同时处理：

- 项目环境
- Codex 全局配置
- Claude Code 全局配置

这会让命令权限面过大，也让用户难以判断它到底在修什么。

### 3.2 更像维护命令，不适合主链心智

它不是交付链一部分，  
而是维护和安装侧能力。

### 3.3 文档仍有占位符未收口

`last_updated: {{DATE}}`

## 4. 推荐优化方向

### 4.1 分离项目诊断与宿主修复

建议拆成两层：

1. 项目诊断
2. 宿主修复

### 4.2 降级为维护命令

不要与主链阶段命令平级强调。

## 5. 当前审查结论

- 职责定位：合理
- 当前合理性：中
- 主要问题：权限面过大，项目诊断与宿主修复绑定过紧
- 最佳优化方向：收敛为维护命令，并分离项目与宿主两个层面

一句话结论：

> `15-doctor` 该保留，但应更像维护入口，而不是主链常用命令。

# 2026-03-07 05-research skill 审查与优化建议

## 1. skill 要解决什么问题

`05-research` 不应该被理解成“去网上搜一圈资料”。

它真正要解决的是：

1. 当主链出现技术选型、不确定方案或高风险决策时，提供证据化调研
2. 对候选方案做结构化比较，而不是凭经验拍板
3. 把调研结论沉淀成 `design / task / verify` 可继续消费的事实基础
4. 显式标记未验证假设，避免把猜测当结论

一句话：

> `05-research` 解决“技术决策为什么这样选，以及证据是什么”。

## 2. 当前实现是否合理

整体判断：合理，而且方向很对。

### 2.1 合理的地方

1. 证据意识非常强
   - 明确提出 `NO TECHNICAL DECISION WITHOUT EVIDENCE`
   - 还细分了证据强度和来源可信度

2. 反证据原则是好的设计
   - 不是只找支持结论的材料
   - 发现反证据时要求记录、解释、必要时更新结论

3. 对未验证假设有显式标记
   - `[NEEDS VERIFICATION]`
   - 这能避免调研结论被误当成已证实事实

4. `findings.md` 的 2-Action Rule 很适合调研场景
   - 调研天然容易上下文漂移
   - 及时落盘是对的

5. 模板化输出方向正确
   - 调研目标
   - 候选方案
   - 对比矩阵
   - 推荐结论

   这能避免 research 退化成散乱笔记。

## 3. 当前主要问题

### 3.1 阶段边界偏松，容易变成“哪里都能插一脚”

当前文档定义：

- 阶段：任意阶段（不限阶段）

这让 `research` 非常灵活，但也会带来边界漂移。

从全链路最佳实践看，`research` 更适合是：

- `spec` 或 `design` 识别到不确定技术决策时的按需分支
- 而不是一个“任何时候都可以独立跑”的泛化 skill

否则会出现两个问题：

1. 主链边界变模糊
2. 用户容易跳过 `spec/design`，直接把 research 当成方案入口

### 3.2 确认策略与成功标准存在自相矛盾

当前同时定义了：

- 低风险场景支持 `auto`
- 成功标准里要求“用户已确认研究结论”

这两个条件不能同时成立。

如果允许 `auto`，就不应把“用户确认”写成所有场景的统一成功标准。  
更合理的做法是按风险等级区分：

- `strict`：必须用户确认
- `assisted`：推荐用户确认
- `auto`：允许自动归档，但要显式标记为“待后续消费确认”

### 3.3 “至少 2 个候选方案”在部分场景会制造伪对比

默认要求 `≥2` 个方案，这在很多常规选型中是对的。  
但有些场景并不适合硬凑两个“真实方案”，例如：

- 公司已有平台标准，只有一种允许方案
- 法规/合规要求已把选择空间压缩到单一方案
- 现有系统强耦合，只存在一种可落地兼容路径

这时如果还硬要求“至少 2 个”，就很容易做出形式化伪对比。

最佳实践应该是：

- 优先要求“有真实备选时必须对比”
- 若只有一个可行方案，也必须说明约束来源，并至少对比：
  - `唯一可行方案`
  - `不做 / 延后 / 维持现状`

### 3.4 本地系统约束的优先级表达还不够强

当前 research 很强调外部资料、官方文档、社区证据，这是对的。  
但对实际项目来说，很多调研结论首先受制于：

- 现有技术栈
- 组织规范
- 已上线依赖
- 当前运维能力
- 已有接口与数据模型

也就是说，最佳方案往往不是“外部世界里最强的方案”，而是“在当前系统里最合适的方案”。

当前 skill 虽然有“加载上下文”的步骤，但这层优先级表达还不够强，容易让调研偏向通用技术评测，而弱化项目约束。

### 3.5 缺少“如何验证调研结论”的后续交接约束

`[NEEDS VERIFICATION]` 是对的，但现在更像标签，还不够像下一步动作。

最佳实践下，research 不应只输出：

- 推荐什么
- 为什么推荐

还应输出：

- 哪些关键假设还没被验证
- 应该通过什么实验 / PoC / benchmark / 小流量验证
- 由哪个后续 skill 承接

否则 research 很容易停在“看起来有道理”，而没有真正进入可执行闭环。

## 4. 推荐优化方向

### 4.1 把 `research` 收敛成“主链按需触发的证据化分支”

推荐边界：

- `spec` 负责识别需求层不确定性
- `design` 负责识别技术层不确定性
- `research` 负责针对不确定技术决策给出证据化比较

也就是说，`research` 更像专家支援分支，而不是默认主入口。

### 4.2 统一风险等级与确认策略

建议把成功标准改成按风险等级分层：

1. 高风险：必须用户确认
2. 中风险：建议用户确认
3. 低风险：允许自动完成，但必须明确“待后续阶段消费确认”

这样才能和 `strict / assisted / auto` 三档策略真正一致。

### 4.3 改造“至少 2 个方案”的规则

建议改成：

1. 有真实备选时，必须至少比较 2 个方案
2. 只有单一可行方案时，必须说明限制条件
3. 单方案场景至少补一个基线对照：
   - `Do Nothing`
   - `Keep Current`
   - `Delay Decision`

这样既保留比较意识，也避免形式主义。

### 4.4 强化“本地约束优先”原则

建议把证据来源排序明确成两层：

1. 本地事实
   - 现有代码
   - 当前平台约束
   - 已接入依赖
   - 组织规则

2. 外部证据
   - 官方文档
   - Benchmark
   - 案例与社区资料

结论应该始终写成：

- “在当前项目约束下推荐什么”

而不是抽象地写成：

- “哪个技术普遍更好”

### 4.5 把未验证假设升级成“验证计划”

每个 `[NEEDS VERIFICATION]` 建议至少补 3 个字段：

- `验证方式`
- `验证阈值`
- `后续承接 skill`

例如：

- `PERF` 假设
- 验证方式：benchmark / 压测
- 验证阈值：P95 < 200ms
- 承接：`design` 或 `verify`

这样 research 才能真正接入闭环，而不是停在文档层。

## 5. 当前审查结论

- 职责定位：正确
- 当前合理性：中高
- 主要问题：阶段边界偏松，确认策略不一致，比较规则在受约束场景下易流于形式
- 最佳优化方向：收敛成按需触发的证据化调研分支，并显式输出验证交接

一句话结论：

> `05-research` 的价值很高，但不该泛化成“任何阶段都能随时跑的资料搜集器”；它最好的形态，是主链在关键技术不确定点触发的证据化调研分支。

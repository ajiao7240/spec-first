# 2026-03-07 全量 skill 深度审查执行计划

## 1. 审查目标

本轮工作不再只做零散 skill 评论，而是完成一套可收敛到全局结论的系统性审查。

最终目标有 3 个：

1. 逐个审查全部 `spec-first` 正式 skill 的职责定位、边界、流程合理性与质量约束
2. 为每个 skill 形成独立的分析优化报告，并持续更新总索引
3. 在全部 skill 审完后，从全局视角回答：
   - 当前整条链路能否达成目标
   - 哪些能力已经成立
   - 哪些短板会阻断“高质量、顺畅、健壮”的全链路执行
   - 最佳实践方案应如何落地

## 2. 审查原则

### 2.1 先单点，后全局

不先写泛化结论，而是先逐个看清每个 skill 的真实职责和问题，再做全局收敛。

### 2.2 先边界，后细节

每个 skill 优先回答：

1. 它到底要解决什么问题
2. 它在整条链路里的边界是什么
3. 当前实现是否真正服务这个目标

### 2.3 关注“可达成目标”，不是只挑文案问题

审查重点不是措辞，而是：

- 流程是否顺畅
- 阶段是否清晰
- 产物是否可被下游消费
- 质量约束是否真实生效
- 是否有职责重叠、遗漏、伪门禁、伪追踪

### 2.4 每轮必须沉淀

每完成一个 skill：

- 产出单独报告
- 更新 `README.md` 总表
- 保证后续结论能复用，不重复推倒重来

## 4. 每个 skill 的固定分析框架

每份报告统一采用 5 段结构：

1. `skill 要解决什么问题`
2. `当前实现是否合理`
3. `当前主要问题`
4. `推荐优化方向`
5. `当前审查结论`

这保证后续能横向比较。

## 5. 每轮输出物

每完成一个 skill，必须产出：

1. 一份独立报告  
   路径：`docs/review-bundles/skill优化思考/YYYY-MM-DD-<skill>-skill审查与优化建议.md`

2. 一次 `README.md` 更新  
   更新项包括：
   - 总表状态
   - 主要问题
   - 优化方向
   - 文档索引

## 6. 最终全局输出物

全部 skill 审完后，再补 2 类总文档：

### 6.1 全局达标性评估

回答：

- 当前整条链路能解决什么问题
- 从 `first -> init -> spec -> design -> task -> code -> review -> test -> verify -> archive` 是否闭环
- 对研发 / 产品 / 测试 / 架构四类角色是否都有合理入口
- 当前是否已经足以支撑高质量研发流程

### 6.2 最佳实践总方案

输出：

- 推荐链路分层
- 关键 skill 的职责重构建议
- 关键治理规则收敛建议
- 哪些能力必须保留
- 哪些能力需要降级、外移、合并或删除

## 7. 执行方式

采用“顺序深审 + 持续同步”的方式：

1. 按既定顺序逐个审查
2. 每完成一个 skill 就落文档
3. 不等待全部结束才回写结论
4. 全部审完后再统一做全局收敛

# 2026-03-06 00-onboarding 最佳实践设计

## 1. 目标重定义

`00-onboarding` 的最佳实践定位不是：

- 欢迎页
- 静态学习路径推荐器
- 新手帮助文档入口

而应该是：

> 作为新人进入 `spec-first` 的统一入口，快速识别角色、目标、当前项目状态，然后把用户送到最正确的下一步。

它的核心价值是：

1. 让不同角色快速进入正确链路
2. 减少误用错误 skill 的概率
3. 把复杂流程压缩成“当前该做什么”

## 2. 它要解决什么问题

新人进入 `spec-first` 时通常会遇到三个问题：

1. 我是谁
   - 我是研发、产品、测试还是架构？

2. 我现在要干什么
   - 我要认知项目？
   - 我要开始一个需求？
   - 我要继续已有需求？
   - 我要做验证？

3. 我现在应该从哪里开始
   - `first`
   - `init`
   - `spec`
   - `test`
   - `verify`
   - `status`
   - `catchup`

因此，`00-onboarding` 的职责不是给用户看一张“学习路径图”，而是回答：

> 你现在最应该执行的下一条命令是什么。

## 3. 最佳实践原则

## 3.1 它是入口层，不是帮助页

最佳实践下，`00-onboarding` 应该是：

- 统一入口
- 角色识别器
- 下一步调度器

而不是一段静态说明文字。

## 3.2 先识别角色，再识别目标

建议最小问题集只保留 3 个：

1. 你的角色是什么？
   - 研发
   - 产品
   - 测试
   - 架构

2. 你现在想做什么？
   - 认知项目
   - 启动一个新需求
   - 继续已有需求
   - 做测试/验证
   - 查看当前状态

3. 当前是否已有 Feature 上下文？
   - 有
   - 没有
   - 不确定

这比“项目规模”更重要，因为入口推荐首先由角色和当前目标决定。

## 3.3 必须读取真实项目状态

最佳实践下，`00-onboarding` 不能只依赖静态推荐表。

它必须读取：

- 是否已有 `docs/first/` 认知底座
- 是否已有 `.spec-first/current`
- 当前 Feature 是否存在
- 当前阶段是什么
- 是否存在可继续的上下文

否则推荐路径和实际项目状态很容易脱节。

## 3.4 推荐“下一步”，不是一次性推荐整套流程

新人最需要的是：

- 现在该做什么

而不是：

- 给我一个 6 步教程

因此最佳输出应为两层：

### 第一层：唯一推荐下一步

例如：

- 推荐命令：`/spec-first:spec`

### 第二层：后续可选链路

例如：

- 下一步完成后通常进入 `design`
- 如需查看状态，可执行 `status`

## 3.5 不同角色必须有不同默认入口

推荐默认链路如下：

| 角色 | 最佳默认入口 |
|---|---|
| 研发 | `first -> init -> spec` |
| 产品 | `spec` |
| 测试 | `test / verify` |
| 架构 | `first -> design / research` |

这意味着：

- `first` 是认知底座入口
- 但不是所有角色都必须显式先运行它

## 3.6 每次推荐都必须解释原因

最佳实践下，每次输出都应包含：

1. 当前判断依据
2. 推荐命令
3. 为什么是这一步
4. 为什么不是其他常见入口

例如：

> 你当前是测试角色，已有激活 Feature，且当前阶段为 `05_verify`，因此推荐先执行 `/spec-first:verify`，而不是重新执行 `init`。

## 3.7 它必须和正式 skill 体系一致

若作为正式 skill，则必须满足：

- 正式命名规范
- 明确输入输出
- 不依赖不存在的抽象工具能力
- 推荐逻辑可被测试

也就是说，它不能只是“想法正确”，而必须“运行时可落地”。

## 4. 推荐输出结构

建议输出统一采用以下结构：

### 4.1 当前判断

- 角色
- 当前目标
- 当前项目状态

### 4.2 推荐下一步

- 推荐命令
- 简要理由

### 4.3 后续链路

- 执行完这一步后通常进入哪里

### 4.4 备用命令

- 如果当前条件不满足，应先执行什么补齐

## 5. 推荐入口逻辑

## 5.1 研发

### 场景 A：完全新进入项目

- 若没有 `docs/first/`
  - 推荐：`/spec-first:first`

### 场景 B：已认知项目，但没有 Feature

- 推荐：`/spec-first:init`

### 场景 C：已有 Feature，准备定义需求

- 推荐：`/spec-first:spec`

## 5.2 产品

### 场景 A：准备开始新需求

- 推荐：`/spec-first:spec`

### 场景 B：不确定当前项目背景

- 若已有认知底座，展示项目摘要后继续 `spec`
- 若没有认知底座，提示需要研发先补认知底座

## 5.3 测试

### 场景 A：需求已进入验证阶段

- 推荐：`/spec-first:verify`

### 场景 B：需要定义测试用例

- 推荐：`/spec-first:test`

### 场景 C：当前上下文不明确

- 推荐：`/spec-first:status` 或 `/spec-first:catchup`

## 5.4 架构

### 场景 A：理解系统再做设计

- 推荐：`/spec-first:first`

### 场景 B：已经进入设计阶段

- 推荐：`/spec-first:design`

### 场景 C：需要方案比较

- 推荐：`/spec-first:research`

## 6. 与 `first / init / spec` 的边界

最佳边界应当是：

- `first`：生产统一认知底座
- `onboarding`：把人送到正确入口
- `init`：研发需求开发起点
- `spec`：需求理解与正式定义

也就是说：

- `onboarding` 不替代 `first`
- `onboarding` 不替代 `init`
- `onboarding` 不替代 `spec`

它只负责：

> 判断现在谁在使用系统，并把他送到最合理的下一步。

## 7. 推荐演进路径

### P0

- 明确它的产品定位：统一角色入口
- 修正文档中“所有角色都走研发主链”的偏差

### P1

- 引入当前项目状态读取
- 从静态映射升级为动态推荐

### P2

- 接入 `first` 的结构化认知底座
- 让产品、测试、架构可以不显式跑 `first` 也拿到适合自己的上下文

### P3

- 为不同角色形成稳定入口模板
- 让入口逻辑可测试、可维护、可演进

## 8. 结论

`00-onboarding` 的最佳实践，不是继续做“学习路径文案优化”，而是：

> 把它做成 `spec-first` 的统一角色入口调度器。

一句话总结：

- `first` 负责产出认知底座
- `onboarding` 负责把人送到正确入口
- `init/spec/test/verify` 负责承接具体工作

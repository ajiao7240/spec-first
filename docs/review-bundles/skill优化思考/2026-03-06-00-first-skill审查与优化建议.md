# 2026-03-06 00-first skill 审查与优化建议

## 1. skill 要解决什么问题

`00-first` 的核心价值不是“生成几份文档”，而是让一个刚进入项目的人快速建立项目全局认知。

它要解决的是：

1. 这个项目是什么
2. 当前系统代码内容和背景是什么
3. 主要技术栈、模块边界、业务域、API、数据模型分别是什么
4. 后续应该从哪里进入需求开发

从整条链路看，`00-first` 是认知入口，不是需求定义入口，也不是目录初始化入口。

一句话：

> `00-first` 解决“我是谁、项目是什么”

## 2. 当前实现是否合理

整体判断：合理，而且是当前链路里定位最清晰的 skill 之一。

理由如下：

1. 目标聚焦
   - 当前 `00-first` 明确围绕技术栈、代码结构、业务流程、API、数据模型产出认知结果
   - 这和新人理解系统背景的目标一致

2. 输出成体系
   - quick / deep 双模式是合理的
   - quick 适合快速进入
   - deep 适合研发、架构、复杂项目的深入认知

3. 参考资产较完整
   - 已经形成较完整的 agent 规格、检测规则、QA 规则、模板与测试策略
   - 这说明它已经不是“散乱 prompt”，而是有治理基础的认知 skill

4. 和主链关系正确
   - 它位于 `init` 之前
   - 为 `init/spec/design/task/...` 提供认知前提

## 3. 当前主要问题

### 3.1 仍然偏研发视角

当前 `00-first` 非常适合研发和架构师，但并不是产品、测试的最佳第一入口。

这不是实现错误，而是定位边界需要明确：

- `00-first` 更适合做统一认知底座的生产者
- 不应被要求成为所有角色的统一入口

### 3.2 认知产物以 markdown 为主，机器可消费性不足

虽然 `docs/first/*` 对人类阅读友好，但对后续 skill 消费不够稳定。

问题：

- `init` 只能用正则猜测摘要
- 角色化入口没法稳定读取认知结果
- 后续自动推荐链路不够稳

### 3.3 角色化视图尚未形成

当前产物主要是“研发全景视图”，还缺：

- 产品视角摘要
- 测试视角摘要
- 架构风险视角摘要

这会导致统一认知底座存在，但不同角色无法自然消费。

## 4. 推荐优化方向

### 4.1 明确定位：认知底座生产者

建议明确把 `00-first` 定义成：

- 面向研发 / 架构的默认入口
- 面向全角色的统一认知底座生成器

这样可以避免后续把产品、测试也硬塞进 `first`。

### 4.2 增加结构化认知摘要

建议新增机器可消费的结构化摘要文件，例如：

- `docs/first/.summary.json`

建议字段：

- `project_type`
- `tech_stack`
- `core_modules`
- `domain_summary`
- `api_summary`
- `risk_summary`
- `recommended_entry_points`
- `generated_at`

### 4.3 补充角色化视图

在不改变主产物结构的前提下，建议增加三类角色化摘要：

- 产品视图：项目业务背景、已有能力、需求切入点
- 测试视图：风险区域、影响范围、验证入口
- 架构视图：系统边界、耦合点、关键技术约束

### 4.4 让后续 skill 直接消费 `00-first`

建议：

- `init` 消费项目摘要与推荐切入点
- `spec` 消费业务摘要与已有能力边界
- `test/verify` 消费风险摘要与覆盖入口

这样 `00-first` 就不仅是阅读文档，更是后续主链的真实输入。

## 5. 推荐结论

`00-first` 当前不需要推翻重做。

最合理的方向不是“继续加更多认知文档”，而是：

1. 保持它作为认知入口的稳定定位
2. 把 markdown 认知结果升级成“人可读 + 机器可消费”
3. 为不同角色输出不同视图

最终目标应当是：

> `00-first` 不只是研发的第一步，而是整个系统的统一认知底座。

## 6. 当前审查结论

- 职责定位：清晰
- 当前合理性：高
- 主要问题：角色视角不均衡、结构化消费不足
- 优化优先级：高，但属于增强，不属于推翻

一句话结论：

> `00-first` 是一块好底座，下一步不是改掉它，而是让更多角色和后续 skill 真正用起来。

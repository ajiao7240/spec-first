# CLI-Anything 借鉴方案汇总

## 执行摘要

基于 `CLI-Anything` 源码、[GIMP 实现案例](../实现案例/01-GIMP-harness实现案例.md) 和当前 `spec-first` 架构，结论已经很明确：

- 值得借鉴的是“真实证据链、用户视角测试、统一输出接口、测试结果可复核、差距分析闭环”
- 不值得照搬的是 Python 文件结构、PEP 420、`setup.py`、`subprocess.run` 风格示例
- 后续工作应以“增强现有 `gate / validate / analyze / tests / reports`”为主，而不是另起一套平行系统

## 分析基线

### CLI-Anything 已核实的证据

- `HARNESS.md`：622 行方法论文档
- `repl_skin.py`：498 行统一终端界面层
- `validate.md`：52 项检查，8 个维度
- `refine.md`：差距分析 -> 用户确认 -> 增量实现
- `GIMP harness`：完整展示了入口层、状态层、真实后端、CLI 子进程测试、`TEST.md` 结果落盘

### spec-first 当前可复用基线

- CLI 入口：`src/cli/index.ts`
- 现有命令：`validate`、`analyze`、`doctor`、`gate`、`metrics`、`trace`
- 门禁引擎：`src/core/gate-engine/gate-evaluator.ts`
- 命令执行能力：`src/core/gate-engine/command-gate.ts`
- 测试体系：`tests/unit`、`tests/integration`、`tests/e2e`
- Feature 产物目录：`specs/<featureId>/`

## 六大借鉴点

### 1. 真实证据链优先于“形式完成”

**CLI-Anything 证据**

- 真实软件是硬依赖
- 不信任退出码，必须检查最终产物
- 测试不仅验证流程，还验证真实导出文件

**spec-first 现状**

- `gate-evaluator.ts` 已具备阶段门禁和条件评估能力
- `command-gate.ts` 已具备白名单命令执行和 PASS/FAIL 细节返回
- 但当前门禁重心仍偏向文档完整性、追溯覆盖率、规则合规

**建议落点**

- `src/core/gate-engine/command-gate.ts`
- `src/core/gate-engine/gate-evaluator.ts`
- `src/cli/commands/validate.ts`

**最小改造**

- 为 `04_implement` / `05_verify` 增加“真实构建、真实测试、最终产物存在性”证据条件
- 保留现有 coverage / trace / gate 体系，不另造 `stage_gates.py`

**优先级**

- `P0`

### 2. 用户视角 CLI 子进程测试

**CLI-Anything 证据**

- `test_full_e2e.py` 中单独存在 `TestCLISubprocess`
- `_resolve_cli` 先找安装态命令，再 fallback 到开发态入口
- 测试 `--help`、JSON 输出、完整工作流

**spec-first 现状**

- 已有 `tests/e2e/`，覆盖流程和异常路径
- 但安装后命令是否可用、二进制是否按用户路径工作，这层证据还不够强

**建议落点**

- `tests/e2e/`
- 可新增 `cli-subprocess.test.ts` 一类文件

**最小改造**

- 用 Node 子进程验证 `spec-first --help`
- 验证 `spec-first init ...` 在临时目录下的真实行为
- 验证已构建命令而不仅是直接调用内部函数

**优先级**

- `P1`

### 3. 统一输出接口，而不是照搬 REPL 皮肤

**CLI-Anything 证据**

- `repl_skin.py` 统一处理 banner、prompt、table、progress、消息分级、NO_COLOR 降级

**spec-first 现状**

- 输出能力分散在多个命令实现中
- 已有 `src/shared/logger.ts`，但偏向日志而不是 CLI 交互输出抽象

**建议落点**

- `src/shared/`
- 或新的 `src/cli/support/`

**最小改造**

- 先统一 `success/error/warning/info/table/status_block` 这类输出接口
- 先解决一致性和非 TTY 降级，不急着引入复杂 REPL 交互

**优先级**

- `P1`

### 4. 差距分析闭环，优先增强现有 analyze 能力

**CLI-Anything 证据**

- `refine` 的核心不是“新命令名字”，而是“盘点当前覆盖 -> 找缺口 -> 用户确认 -> 增量补齐”

**spec-first 现状**

- 已有 `analyze` 命令，负责跨产物一致性分析
- 也有 `review`、`verify`、`metrics` 等辅助视角

**建议落点**

- `src/cli/commands/analyze.ts`
- `src/core/gate-engine/sca.ts`
- 对应 analyze/review skill 文档

**最小改造**

- 不新增平行的 `validate` / `analyze` 同名命令
- 若未来确有必要，再新增“refine”也应建立在 `analyze` 结果和现有 feature 产物之上

**优先级**

- `P1`

### 5. 测试与验证结果需要落盘、可复核

**CLI-Anything 证据**

- `TEST.md` 同时记录测试清单、覆盖面、最终结果

**spec-first 现状**

- 已有 `reports/analysis-report.md`、`findings.md` 等产物
- 但“验证阶段到底跑了什么、为什么算通过”的证据形式还不够固定

**建议落点**

- `specs/<featureId>/reports/`
- `verify` / `review` / `analyze` 输出

**最小改造**

- 约定一份验证证据摘要文档
- 最少包含：执行命令、退出码、最终判定、关键产物路径

**优先级**

- `P1`

### 6. 方法论文档体系应做“统一抽象”，不是从零开始补课

**CLI-Anything 证据**

- `HARNESS.md` 提供共享方法论
- `<SOFTWARE>.md` 提供项目特定 SOP

**spec-first 现状**

- 并非“只有 CLAUDE.md”
- 已有 skills、AGENTS、shared contracts、阶段文档、命令参考
- 问题在于信息分散，不在于完全缺失

**建议落点**

- `skills/spec-first/README.md`
- `skills/spec-first/AGENTS.md`
- 未来若统一抽象，可考虑 `docs/methodologies/`

**最小改造**

- 先做“聚合与梳理”
- 暂不建议把它排在真实证据链和测试增强之前

**优先级**

- `P2`

## 实施路线图

### 阶段 1：证据链增强

目标：

- 在现有 gate / validate 上补真实命令证据和最终产物校验

对应模块：

- `src/core/gate-engine/command-gate.ts`
- `src/core/gate-engine/gate-evaluator.ts`
- `src/cli/commands/validate.ts`

### 阶段 2：测试增强

目标：

- 增加安装态 CLI 子进程测试
- 补用户视角工作流证据

对应模块：

- `tests/e2e/`

### 阶段 3：分析闭环增强

目标：

- 让 analyze 输出更接近“差距分析 + 建议修复”

对应模块：

- `src/cli/commands/analyze.ts`
- `src/core/gate-engine/sca.ts`

### 阶段 4：输出和文档抽象

目标：

- 统一 CLI 输出接口
- 聚合分散的方法论说明

对应模块：

- `src/shared/` 或 `src/cli/support/`
- `skills/spec-first/*`

## 优先级排序

### P0（必须做）

1. Gate 真实证据链增强

### P1（应该做）

2. CLI 子进程 E2E
3. analyze 差距分析增强
4. 验证证据落盘规范
5. 统一输出接口

### P2（可以做）

6. 方法论文档聚合

## 不再采用的旧建议

以下建议已经在本轮审查后废弃：

- 新增 `output_skin.py`
- 新增 `real_verification.py`
- 新增 `stage_gates.py`
- 新增 Python 版 `test_real_env.py`
- 新增 Python 版 `test_cli_subprocess.py`
- 把 `spec-first` 描述成“主要只有阶段命令”

这些写法的问题不在方向，而在于没有落到 `spec-first` 当前真实架构。

## 总结

这份汇总的核心结论是：

- 要借鉴的是模式，不是 Python 文件名
- 要增强的是现有 `spec-first` 模块，不是复制一份 CLI-Anything 的目录结构
- 真正的优先级应从“真实证据链”开始，而不是从 UI 或方法论文档开始

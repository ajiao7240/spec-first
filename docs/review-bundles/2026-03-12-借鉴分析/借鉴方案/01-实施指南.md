# 借鉴方案实施指南

## 文档说明

本指南只讨论 `spec-first` 当前真实架构上的实施路径：

- 技术栈：Node.js + TypeScript + ESM + Vitest
- 当前能力：已有 `gate / validate / analyze / tests / reports`
- 实施原则：增强现有模块，不复制 `CLI-Anything` 的 Python 目录结构

## 方案 1：增强 gate 的真实证据链

### 目标

把“真实构建、真实测试、最终产物存在性”纳入当前阶段门禁，而不是只看规则和追溯指标。

### 目标文件

```text
src/core/gate-engine/
├── command-gate.ts
└── gate-evaluator.ts

src/cli/commands/
└── validate.ts
```

### 实施要点

#### A. 在 `command-gate.ts` 扩展结果模型

目标：

- 不只返回 PASS/FAIL
- 还保留 stdout/stderr 摘要、退出码、执行命令、可选产物路径

示意：

```ts
export interface CommandExecutionResult {
  pass: boolean;
  detail: string;
  stdout: string;
  exitCode?: number;
  command?: string;
  artifactPaths?: string[];
}
```

#### B. 在 `gate-evaluator.ts` 新增真实证据条件

建议位置：

- `04_implement`
- `05_verify`

建议条件示例：

```ts
{
  id: 'G-IMPL-REAL-01',
  description: 'Configured build/test commands succeed with fresh evidence',
  evaluate: (ctx) => {
    const result = runCommandGate(/* read layer2 or feature config */);
    return {
      pass: result.pass,
      detail: result.detail,
    };
  },
}
```

#### C. 在 `validate all` 中展示真实证据摘要

目标：

- `validate all` 不只显示 gate PASS/FAIL
- 还输出最近一次真实证据检查的关键信息

### 验收标准

- 构建命令失败时，`gate` 和 `validate all` 都会阻断
- 错误信息含 stdout/stderr 摘要
- 可选校验最终产物路径是否存在
- 不新增新的 Python 模块或平行 gate 实现

## 方案 2：增加安装态 CLI 子进程测试

### 目标

补齐“用户真的通过命令行调用 `spec-first` 时是否可用”的证据层。

### 目标文件

```text
tests/e2e/
└── cli-subprocess.test.ts
```

### 实施要点

#### A. 提供 Node 版 CLI 解析器

目标：

- 优先找已构建命令
- 必要时 fallback 到 `dist/cli/index.js`
- 不使用 Python 风格 fallback

示意：

```ts
function resolveCli(): string[] {
  const installed = process.env.SPEC_FIRST_FORCE_INSTALLED === '1'
    ? which('spec-first')
    : which('spec-first');

  if (installed) return [installed];
  return [process.execPath, 'dist/cli/index.js'];
}
```

#### B. 最低覆盖范围

建议先测 4 类场景：

1. `spec-first --help`
2. `spec-first init ...`
3. `spec-first validate format <featureId>`
4. 一个错误路径场景，确保 stderr 和退出码正确

#### C. 保持测试真正站在用户视角

要求：

- 用子进程启动命令
- 在临时目录执行
- 不直接 import 内部函数作为替代

### 验收标准

- 子进程测试可在 CI 跑通
- 支持安装态优先
- 覆盖 help、正常流程、错误流程
- 与现有 `tests/e2e/` 风格保持一致

## 方案 3：统一输出接口

### 目标

统一 CLI 输出格式，但不引入一整套 REPL 皮肤。

### 目标文件

```text
src/shared/
├── logger.ts
└── cli-output.ts        # 可新建
```

### 实施要点

#### A. 抽象最小输出接口

建议接口：

```ts
export interface CliOutput {
  success(message: string): void;
  error(message: string): void;
  warning(message: string): void;
  info(message: string): void;
  table(headers: string[], rows: string[][]): void;
  statusBlock(items: Array<{ label: string; value: string }>): void;
}
```

#### B. 支持非 TTY 降级

检查：

- `NO_COLOR`
- `process.stdout.isTTY`

#### C. 先试点，不全量替换

推荐试点命令：

- `validate`
- `gate`
- `doctor`

### 验收标准

- 成功/错误/警告/信息格式统一
- 非 TTY 环境自动降级
- 至少 1 到 2 个命令完成迁移
- 不引入复杂 prompt/REPL 依赖

## 方案 4：增强 analyze 的差距分析能力

### 目标

把 `CLI-Anything refine` 的核心价值吸收到当前 `analyze` 里，而不是直接新增同名命令。

### 目标文件

```text
src/cli/commands/
└── analyze.ts

src/core/gate-engine/
└── sca.ts
```

### 实施要点

#### A. 报告结构增加“缺口与建议”

现有分析结果之外，补以下区块：

- 当前已满足项
- 缺失项
- 风险项
- 最小修复建议

#### B. 优先使用现有 feature 产物

输入优先级：

1. `stage-state.json`
2. `traceability-matrix.md`
3. `design.md / tasks.md / reports/*`

#### C. 不新增重复命令

原则：

- 先把 `analyze` 做强
- 只有在确实需要单独交互模型时，再考虑 `refine`

### 验收标准

- `analyze` 报告能清晰区分“已满足 / 缺口 / 风险 / 建议”
- 可直接服务 review/verify 阶段
- 不新增与现有命令重叠的平行入口

## 方案 5：补验证证据摘要文档

### 目标

让验证阶段的关键证据可复核，而不是只靠终端输出。

### 目标文件

```text
specs/<featureId>/reports/
└── verification-evidence.md
```

### 建议结构

```md
# Verification Evidence

## Commands
- npm run build
- npm test
- spec-first validate all <featureId>

## Exit Codes
- build: 0
- test: 0
- validate: 0

## Key Artifacts
- dist/cli/index.js
- specs/<featureId>/reports/analysis-report.md

## Conclusion
- PASS / FAIL
```

### 生成方式

可选两种：

1. `verify` 阶段自动生成
2. 先由 `validate all` 或 review/verify skill 输出

### 验收标准

- 证据文档路径固定
- 包含命令、退出码、关键产物、结论
- 能和 `findings.md`、`analysis-report.md` 并存

## 建议实施顺序

### 第 1 步：真实证据链

先做：

- `command-gate.ts`
- `gate-evaluator.ts`
- `validate.ts`

原因：

- 这是最直接提升质量可信度的部分

### 第 2 步：CLI 子进程 E2E

再做：

- `tests/e2e/cli-subprocess.test.ts`

原因：

- 可以从用户视角证明 CLI 真能工作

### 第 3 步：analyze 增强

再做：

- `analyze.ts`
- `sca.ts`

原因：

- 帮助后续差距收敛和复盘

### 第 4 步：输出与证据文档

最后做：

- `cli-output.ts`
- `verification-evidence.md`

原因：

- 这些重要，但不应压过真实证据链

## 不再采用的旧写法

以下写法不再保留：

- `src/core/ui/output_skin.py`
- `src/core/gate-engine/real_verification.py`
- `src/core/gate-engine/stage_gates.py`
- `tests/integration/test_real_env.py`
- `tests/e2e/test_cli_subprocess.py`
- `sys.executable -m spec_first.cli`

这些路径和示例都不符合 `spec-first` 当前架构。

## 总结

本指南的实施原则只有一句话：

- 借鉴 `CLI-Anything` 的闭环模式，但所有落点都必须回到 `spec-first` 已有 TypeScript 模块上

如果后续进入真正实施，优先顺序应是：

1. 真实证据链
2. CLI 子进程 E2E
3. analyze 差距分析增强
4. 输出接口与验证证据文档

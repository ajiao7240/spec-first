# 命令系统深度分析

## 文件概览

- **cli-anything.md**: 主构建命令（7 阶段流水线）
- **refine.md**: 增量优化命令（覆盖率扩展）
- **validate.md**: 质量验证命令（52 项检查）
- **test.md**: 测试运行命令
- **list.md**: 列表查询命令

## 核心价值

CLI-Anything 的命令系统体现了"渐进式构建"理念：
1. **cli-anything**: 从零构建完整 CLI
2. **refine**: 增量扩展覆盖率
3. **validate**: 质量门禁验证
4. **test**: 持续测试验证

## 架构设计亮点

### 1. refine 命令 - 增量优化模式

**核心流程**:
```
Step 1: 盘点当前覆盖率
  - 读取现有 CLI 代码
  - 列出所有命令/子命令
  - 构建覆盖率地图

Step 2: 分析软件能力
  - 重新扫描源码
  - 识别所有公开 API
  - 按领域分类

Step 3: 差距分析
  - 对比覆盖率 vs 完整能力
  - 按优先级排序（高影响/易实现/可组合）
  - 向用户展示并确认

Step 4: 实现新命令
  - 遵循现有模式
  - 集成到命令组

Step 5: 扩展测试
  - 单元测试 + E2E 测试
  - 回归测试

Step 6: 更新文档
  - README.md
  - TEST.md
  - SOP 文档
```

**设计亮点**:
- **聚焦模式**: 支持 `[focus]` 参数，针对特定功能领域
- **增量迭代**: 可多次运行，逐步扩展
- **用户确认**: 差距分析后等待用户确认
- **零破坏**: 只增加/增强，不删除

**spec-first 借鉴**:
```python
# 新命令: /spec-first:refine <feature-id>
# 分析当前 Feature 的覆盖率差距

Step 1: 盘点当前实现
  - 读取已实现的 TASK
  - 检查测试覆盖率
  - 构建追溯矩阵

Step 2: 分析需求完整性
  - 重新审视 FR（功能需求）
  - 识别未覆盖的场景
  - 检查边界情况

Step 3: 差距分析
  - 对比实现 vs 需求
  - 识别缺失的测试
  - 发现潜在风险

Step 4: 增量实现
  - 补充缺失的 TASK
  - 增加测试用例
  - 更新文档

Step 5: 验证
  - 运行完整测试套件
  - 更新追溯矩阵
  - 生成覆盖率报告
```

### 2. validate 命令 - 质量门禁系统

**52 项检查清单**:
```yaml
目录结构 (5 checks):
  - 命名空间包结构正确
  - 子包结构完整
  - 核心目录存在

必需文件 (9 checks):
  - README.md
  - CLI 入口
  - 核心模块
  - 测试文件
  - SOP 文档

CLI 实现标准 (7 checks):
  - 使用 Click 框架
  - 命令分组
  - --json 支持
  - REPL 模式
  - 错误处理

核心模块标准 (5 checks):
  - project.py 完整性
  - session.py 撤销/重做
  - export.py 渲染流程
  - 文档字符串
  - 类型提示

测试标准 (10 checks):
  - TEST.md 完整性
  - 单元测试隔离
  - E2E 真实文件
  - 工作流测试
  - CLI 子进程测试
  - _resolve_cli 实现
  - 100% 通过率

文档标准 (4 checks):
  - README 完整性
  - SOP 文档
  - 命令文档
  - 示例代码

PyPI 打包标准 (7 checks):
  - 命名空间包配置
  - 包命名规范
  - 入口点配置
  - 依赖声明
  - Python 版本要求

代码质量 (5 checks):
  - 无语法错误
  - 无导入错误
  - PEP 8 风格
  - 无硬编码路径
  - 错误处理规范
```

**设计亮点**:
- **全面性**: 52 项检查覆盖所有关键维度
- **可追溯**: 每项检查对应 HARNESS.md 的具体要求
- **自动化**: 可编程验证，不依赖人工
- **报告清晰**: 分类汇总，一目了然

**spec-first 借鉴**:
```python
# 新命令: /spec-first:validate <feature-id>
# 验证 Feature 是否符合规范

检查维度:
1. 规范完整性 (10 checks)
   - FR/DS/TASK 完整性

2. 实现质量 (8 checks)

3. 阶段门禁 (6 checks)
   - 阶段产物完整

4. 追溯一致性 (5 checks)
   - ID 引用正确

输出:

  ========================

  Stage: 04_implement

  规范完整性: 10/10 ✓
  实现质量: 7/8 ⚠ (测试覆盖率 68%)

```

### 3. 命令设计模式对比

**CLI-Anything 命令特点**:
```
cli-anything        # 全流程构建（7 阶段）
cli-anything:refine # 增量优化（差距分析）
cli-anything:validate # 质量验证（52 检查）
cli-anything:test   # 测试运行
```

**spec-first 当前命令**:
```
spec-first:init     # 初始化
spec-first:spec     # 需求规格
spec-first:design   # 技术设计
spec-first:task     # 任务拆解
spec-first:code     # 代码实现
spec-first:review   # 实现审查
spec-first:verify   # 阶段验收
```

**差异分析**:
- CLI-Anything: 命令对应"操作类型"（构建/优化/验证）
- spec-first: 命令对应"阶段"（01_specify/02_design/...）

**借鉴建议**:
```
保留阶段命令（主流程）:
  spec-first:spec
  spec-first:design
  spec-first:code
  ...

增加操作命令（辅助）:
  spec-first:refine   # 增量优化
  spec-first:validate # 质量验证
  spec-first:analyze  # 差距分析（已有）
  spec-first:doctor   # 环境诊断（已有）
```

## spec-first 可借鉴的核心模式

### 模式 1: 增量优化命令

**当前问题**: 缺少"回头看"机制，发现遗漏后难以补充

**借鉴方案**:
```bash
# 分析当前 Feature 的覆盖率差距
/spec-first:refine FSREQ-001

# 聚焦特定领域
/spec-first:refine FSREQ-001 --focus="错误处理"
```

**实现要点**:
- 盘点已实现的 TASK
- 分析需求完整性
- 识别测试覆盖率差距
- 生成补充建议

### 模式 2: 质量门禁验证

**当前问题**: gate-engine 存在，但检查项可能不够系统化

**借鉴方案**:
```python
# src/core/gate-engine/checklist.py
VALIDATION_CHECKLIST = {
    "规范完整性": [
        "FR 定义完整",
        "DS 设计完整",
        "TASK 拆解完整",
        "追溯矩阵无断链",
        # ...
    ],
    "实现质量": [
        "代码通过 lint",
        "测试覆盖率 >= 75%",
        "文档与代码同步",
        # ...
    ],
    # ...
}
```

### 模式 3: 聚焦模式

**当前问题**: 大型 Feature 难以一次性完成

**借鉴方案**:
```bash
# 聚焦特定子功能
/spec-first:code FSREQ-001 --focus="用户认证"

# 只实现核心路径
/spec-first:code FSREQ-001 --focus="happy-path"

# 补充边界情况
/spec-first:refine FSREQ-001 --focus="边界情况"
```

## 总结

CLI-Anything 命令系统的核心价值:
1. **渐进式构建**: 构建→优化→验证的闭环
2. **质量门禁**: 52 项系统化检查
3. **增量迭代**: refine 支持多次运行
4. **聚焦模式**: 支持针对性优化

这些模式可以增强 spec-first 的灵活性和质量保障能力。

# stop-guard 待优化问题列表

日期：2026-03-09  
范围：`stop-guard` 当前本地实现、生成真源、测试与治理文档  
结论摘要：本地已落盘的 `stop-guard.sh` 当前**没有死循环问题**，但仍存在“语义真源分裂 → 后续容易被旧实现覆盖回去”的回归风险。当前最优策略不是继续扩展功能，而是先做一次**最小收口**，统一脚本、生成源码、治理说明和测试断言。

---

## 一、总判断

当前 `stop-guard` 的真实定位，已经从“硬阻断 Stop Hook”演化为“Stop 阶段的非阻断提醒 Hook”：

- 仅在 `04_implement` 阶段生效
- 仅提醒 `in_progress` 任务
- 输出到 `stderr`
- 始终 `exit 0`
- 目标是避免 AI 因 Stop Hook 非零退出而陷入重复收尾/重复触发的死循环

因此，当前最需要优化的不是功能强度，而是**口径统一**与**回归防护**。

---

## 二、待优化问题列表

### 问题 1：运行中的脚本与生成真源不一致

**现象**

当前仓库里的落盘脚本 `.spec-first/hooks/stop-guard.sh` 已经是“提醒型”实现：

- 仅在 `04_implement` 阶段检查
- 仅识别 `in_progress`
- 有提醒也 `exit 0`

但生成该脚本的真源 `src/core/tool-integration/ai-runtime-hook-scripts.ts` 仍保留旧版“阻断型”逻辑：

- 检测的是 `PENDING_IDS`
- 发现未完成任务后 `exit 2`

**风险**

- 任何再次执行脚本注册/安装/更新逻辑的场景，都可能把当前正确脚本重新覆盖成旧版阻断实现
- 这会把已经修掉的死循环问题重新引入
- 这是当前最核心、最现实的回归源

**最佳解决方案**

把 `src/core/tool-integration/ai-runtime-hook-scripts.ts` 中的 `STOP_GUARD_SCRIPT_CONTENT` 直接改为与当前落盘脚本完全一致的提醒型实现，并把它确认为唯一维护真源。

**为什么这是最佳方案**

- 改动最小
- 风险最低
- 立刻消除“当前文件正确，但下次安装又写错”的问题
- 不需要引入模板系统、配置模式或额外抽象

**优先级**

`P0 / 必做`

---

### 问题 2：治理文档仍把 stop-guard 定义成阻断型 Hook

**现象**

`skills/spec-first/AGENTS.md` 仍写着：

- 阻断型 Hook（如 `stop-guard.sh`）必须输出 `stderr` + 非 0 退出

这与当前实际实现完全相反。

**风险**

- 后续维护者或 AI 会把当前提醒型实现误判为“不符合规范”
- 容易再次把脚本改回非 0 退出
- 文档会持续制造错误修复方向

**最佳解决方案**

将治理口径改为：

- `stop-guard.sh` 当前属于**非阻断提醒型 Hook**
- 必须把提醒写到 `stderr`
- 必须允许 `exit 0`
- 真正阻断型 Hook 与提醒型 Hook 需要分开定义，不可混用

建议把 Hook 分类明确写成两类：

1. 阻断型 Hook：失败必须非 0 退出
2. 提醒型 Hook：允许 `exit 0`，但不得静默误导

并明确 `stop-guard.sh` 属于第 2 类。

**为什么这是最佳方案**

- 直接消除规则层误导
- 不需要大面积改所有历史文档
- 只修“当前仍作为操作约束使用的活文档”即可

**优先级**

`P0 / 必做`

---

### 问题 3：测试只校验“文件存在”，没有校验“生成后语义正确”

**现象**

当前 `tests/unit/ai-runtime-hook.test.ts` 主要验证：

- 注册后 `stop-guard.sh` 文件存在
- hooks 配置条目存在

但没有验证：

- 生成出来的脚本到底是不是提醒型语义
- 是否只在 `04_implement` 生效
- 是否仍然 `exit 0`
- 是否只提醒 `in_progress`

**风险**

- 一旦真源回滚成旧版，测试仍可能继续通过
- 会出现“看起来所有测试都绿了，但 stop-guard 实际语义已经变坏”的假阳性

**最佳解决方案**

补一条最小语义回归测试：

- 调用 `registerAIHooks`
- 在临时目录生成 `stop-guard.sh`
- 准备最小 `task_plan.md` 和 `stage-state.json`
- 实际执行生成脚本
- 验证以下行为：
  - `04_implement + in_progress` → 提示在 `stderr`，且退出码为 `0`
  - `04_implement + todo` → 不提示，退出码为 `0`
  - 非 `04_implement` → 跳过，退出码为 `0`

**为什么这是最佳方案**

- 用最少的测试补上真正的回归防线
- 直接验证用户真正关心的行为，而不是文件是否存在
- 不需要额外引入测试框架或快照机制

**优先级**

`P0 / 必做`

---

### 问题 4：stop-guard 命名与实际行为存在语义误导

**现象**

文件名 `stop-guard.sh` 很容易让人理解为：

- 阻止结束
- 强制拦截 Stop
- 发现未完成任务必须失败

但当前实现其实只是提醒，不做硬拦截。

**风险**

- 新维护者容易基于名字做错推断
- 文档和讨论会反复出现“它到底该不该 exit 非 0”的歧义

**最佳解决方案**

短期内：**不改文件名**，只改描述语义。

建议在活文档和测试命名中统一描述为：

- `stop-guard`：Stop 阶段完成度提醒 Hook
- 或 `stop-guard`：非阻断收口提醒 Hook

长期如果确实要追求命名精确，再做兼容式重命名，例如：

- 保留 `stop-guard.sh` 作为兼容入口
- 内部新语义名称可演进为 `stop-reminder.sh`

**为什么这是最佳方案**

- 现在直接改文件名收益不大，但会增加宿主配置、安装逻辑、历史兼容成本
- 先修语义描述，能用最小代价消除 80% 的误解

**优先级**

`P1 / 建议尽快`

---

### 问题 5：历史文档仍大量残留“硬阻断 stop-guard”口径

**现象**

仓库内多份历史文档仍描述 `stop-guard` 为：

- 阻断未完成 TASK 结束
- `stderr + 非 0 退出`
- Stop 核心阻断门禁

这些历史描述与当前实现不一致。

**风险**

- 审查时容易被旧文档干扰
- AI 检索上下文时会混入过期结论
- 新一轮“看文档修代码”时容易修反

**最佳解决方案**

不要试图一次性清洗所有历史文档；应采用“活文档优先、历史文档标注过期”的最小策略：

1. 先修活文档：`AGENTS.md`、当前设计文档、当前实现说明、必要的 README/当前清单
2. 对高频引用的历史文档增加一行“该描述基于旧版阻断语义，已过期”
3. 不做全仓库批量重写，避免制造新的噪音改动

**为什么这是最佳方案**

- 能控制改动面
- 不会把本次最小修复升级成一次庞大的文档清洗工程
- 兼顾现实维护成本与信息正确性

**优先级**

`P1 / 建议尽快`

---

### 问题 6：当前缺少一个可复用的 Hook 行为 contract

**现象**

现在关于 `stop-guard` 的关键语义散落在：

- shell 脚本
- TS 生成字符串
- AGENTS 说明
- 单元测试
- changelog 文字

没有一个简洁的、可复核的行为契约来定义：

- 它何时生效
- 它检查什么
- 它输出什么
- 它是否阻断

**风险**

- 语义容易再次漂移
- 每次审查都要靠人工拼上下文
- 未来类似 hook 也会重复踩坑

**最佳解决方案**

增加一个**最小 contract 文档或最小 contract 测试**，不做大框架，只定义 4 条语义：

1. 仅 `04_implement` 生效
2. 仅提醒 `in_progress`
3. 输出到 `stderr`
4. 始终 `exit 0`

这份 contract 可以落在：

- 一个简短 markdown 说明文件，或
- 一个更推荐的“共享语义测试”中

若只能选一个，优先选**测试 contract**，因为它能真正防回归。

**为什么这是最佳方案**

- 成本小
- 收益高
- 不会把当前问题过度工程化

**优先级**

`P1 / 建议尽快`

## 三、最优落地顺序

### 第一步：先修真源一致性

处理内容：

- 对齐 `src/core/tool-integration/ai-runtime-hook-scripts.ts`
- 以当前提醒型脚本为准

**原因**

这是唯一会直接把正确实现重新写坏的入口，优先级最高。

### 第二步：再修治理口径

处理内容：

- 更新 `skills/spec-first/AGENTS.md`
- 明确 `stop-guard` 是提醒型 Hook，不是阻断型 Hook

**原因**

不修这一层，后续很容易再次被“按规范修错”。

### 第三步：补行为回归测试

处理内容：

- 在 AI hook 注册相关测试中，验证“生成后的 stop-guard 实际行为”

**原因**

这样才能保证后续修改不会静默回退。

### 第四步：补一个最小 contract

处理内容：

- 文档或测试中沉淀 4 条 stop-guard 核心语义

**原因**

用最小成本把这次结论固定下来，减少未来重复争论。

### 第五步：按需处理文档漂移

处理内容：

- 只清理高频活文档
- 对少量高频历史文档加“旧语义已过期”提示

**原因**

避免把本次问题扩大成全量文档治理工程。

---

## 四、不建议现在做的事

### 不建议 1：立刻改回阻断型 stop-guard

原因：

- 会重新引入 Stop Hook 死循环风险
- 除非同时完成严格的防重入/防重复触发设计，否则不值得

### 不建议 2：立刻重命名文件为 `stop-reminder.sh`

原因：

- 收益不如修真源和测试大
- 会扩大兼容范围，牵连配置、安装器、文档、历史引用

### 不建议 3：为了 stop-guard 单独引入复杂模板/配置系统

原因：

- 当前问题本质是“漂移”，不是“扩展能力不足”
- 复杂化只会增加新故障面

---

## 五、推荐结论

当前 `stop-guard` 的最优优化方向应明确为：

> **保持提醒型语义不变，做一次最小但彻底的一致性收口。**

也就是：

- 不改产品策略
- 不加复杂模式
- 不做大重构
- 只把“脚本真源、治理口径、行为测试、最小契约”四件事一次对齐

这是当前成本最低、收益最高、最能防止回归的方案。

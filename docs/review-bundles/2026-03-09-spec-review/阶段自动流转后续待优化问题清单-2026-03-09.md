# orchestrate / stage flow 后续待优化问题清单（2026-03-09）

## 目的

本文聚焦当前 `P1-2 / 方案A` 相关实现的**后续待优化问题**，仅记录：

- 问题是什么
- 为什么是问题
- 当前推荐的最佳解决方案是什么

原则：坚持 **KISS**、**职责分离**、**避免过度设计**，优先完成“状态闭环”与“受控推进”的收口，不提前引入真实 `task/code/verify` 执行链。

---

## 待优化问题列表

| ID | 问题 | 现状 / 风险 | 最佳解决方案 |
|---|---|---|---|
| S1 | `next-step-decider` 还不能直接消费 `auto-loop` 结果 | 当前阶段决策主要依赖 `stageStatus / gate / dependency / todoState`，但 `auto-loop` 的 `all_done / has_blocked / timeout / incomplete` 结果没有成为显式输入，导致“任务循环结果”和“阶段推进结果”之间仍存在一层隐式推断 | 在 `src/core/process-engine/next-step-decider.ts` 中新增 `autoLoopStatus` 输入，并将 `has_blocked / timeout / max_iterations / incomplete / no_state_file` 统一折叠为 `BLOCKED`。让阶段判定直接消费循环结果，而不是通过文案和外围判断间接推断 |
| S2 | 决策结果缺少稳定的结构化原因码 | 当前 `reasons` 更偏向面向用户的自然语言输出，适合 CLI 展示，但不适合做稳定测试断言，也不利于后续审计与统一错误处理 | 为 `NextStepDecision` 增加 `reasonCodes`，例如：`AUTO_LOOP_NOT_DONE`、`TODO_BLOCKED`、`TODO_PENDING`、`GATE_FAILED`、`DEPENDENCY_FAILED`、`STAGE_NOT_READY`。CLI 继续打印 `reasons`，测试与集成逻辑改依赖 `reasonCodes` |
| S3 | `03_plan / 04_implement / 05_verify` 的规则还不够收口 | 当前 `next-step-decider` 已有基础逻辑，但对 `03_plan -> 04_implement -> 05_verify` 这条主线仍偏粗，容易出现“建议推进”和“实际不能推进”之间的判断漂移 | 以阶段为单位补齐明确判定表：`BLOCKED / SUGGEST_NEXT / READY_TO_ADVANCE / AUTO_ADVANCE`。每个阶段都统一依赖 `stageStatus + dependency + gate + todo + autoLoopStatus` 五类信号，避免阶段间规则不一致 |
| S4 | `AUTO_RUN_NEXT_SKILL` 仍暴露在 `04_implement` 路径中，超出方案A边界 | 方案A的目标是“状态闭环优先”，不是“真实执行闭环优先”。如果保留 `AUTO_RUN_NEXT_SKILL` 的活跃语义，后续很容易误把 A 做成 B，扩大范围并引入执行链复杂度 | 保留枚举类型以兼容未来扩展，但在 `P1-2 / 方案A` 路径中明确禁用该分支；`03_plan / 04_implement / 05_verify` 三段只允许返回 `BLOCKED / SUGGEST_NEXT / READY_TO_ADVANCE / AUTO_ADVANCE` |
| S5 | `orchestrate` 还没有把 loop 结果完整传入统一决策层 | 当前 `orchestrate` 已能在 loop 后调用 `next-step-decider`，但如果不把 `autoLoopResult.status` 显式转成决策输入，仍会形成“协调层知道、决策层不知道”的信息分裂 | 在 `src/cli/commands/orchestrate.ts` 中把 `autoLoopResult.status` 显式映射为 `autoLoopStatus` 传给 `decideNextStep()`；协调层只负责汇总信号，不重复写阶段判定 |
| S6 | `orchestrate` 缺少最小 hooks 同步点 | 当前已有 `task-context.sh`、`progress-sync.sh` 等机制，但 `orchestrate` 本身还没有固定的“同步前 / 同步后”接入点，容易导致 todo 层状态和阶段层状态在推进瞬间出现轻微漂移 | 在 `src/core/tool-integration/ai-runtime-hook.ts` 增加一个极薄的 orchestrate helper，例如 `runOrchestrateSyncHooks(...)`，只在 `before_decide` 与 `after_advance` 两个点触发现有脚本。hooks 失败只记 warning，不成为推进裁判 |
| S7 | 自动推进的安全条件尚未完全在一个点收口 | 如果 `auto-advance` 的放行条件分别散在 CLI、decider、文档说明里，长期会出现“某处放行、某处阻断”的口径分裂 | 统一规定：只有当 `decision in {READY_TO_ADVANCE, AUTO_ADVANCE}` 且 `autoLoopStatus=all_done` 且 Gate 通过且 dependency 通过且无 blocked todo 时，`orchestrate` 才允许调用 `advance()`；否则一律只输出建议和阻塞原因 |
| S8 | `stageStatus` 的语义还需要进一步固化 | 目前 `drafting / awaiting_review / review_failed / ready_to_advance / advanced` 已具备最小语义，但如果不进一步明确每个状态的含义，后续实现中很容易把 `stageStatus` 当成“随手写的文案”而不是推进契约 | 明确约束：`ready_to_advance` 是唯一可进入推进判定的准备态；`advanced` 只允许作为短暂事件态，不应长期驻留；`advance()` 完成后，新阶段立即回到 `drafting`。这样可以降低跨阶段判断歧义 |
| S9 | 缺少专门覆盖 `03_plan -> 05_verify` 的流转测试 | 现有测试已覆盖部分 orchestrate 和参数语义，但还缺一组专门面向 `03_plan / 04_implement / 05_verify` 的阶段流转测试，后续最容易在这条主线上回归 | 新增 `tests/unit/stage-flow-03-05.test.ts`，专门覆盖：`plan ready -> READY_TO_ADVANCE`、`implement blocked -> BLOCKED`、`all_done + gate fail -> BLOCKED`、`pending todo -> SUGGEST_NEXT`、`safe auto-advance -> advance called` |
| S10 | `--auto-advance` 的产品语义仍有漂移风险 | 如果文档、参数注释、测试口径不完全一致，使用者容易把 `--auto-advance` 理解成“自动执行下一 skill”，从而对当前能力形成误判 | 在 `src/core/skill-runtime/orchestrate-args.ts`、`skills/spec-first/13-orchestrate/SKILL.md` 与相关测试中明确：`--auto-advance` 只代表“自动推进阶段”，不代表自动执行 `task/code/verify`。保持参数、文档、测试三者完全一致 |
| S11 | hooks 与阶段判定之间的职责边界需要显式写清 | 如果后续实现者把 hooks 结果也直接拿来当推进依据，会让“同步器”和“判定器”混淆，导致难以调试和追责 | 明确边界：hooks 只负责刷新任务上下文与进度同步，不直接产出 `READY_TO_ADVANCE` 结论；推进依据仍只来自 decider 聚合的 `stageStatus / gate / dependency / todo / autoLoopStatus` |
| S12 | 需要显式冻结非目标，防止 P1-2 再次扩 scope | 当前最危险的不是做不出来，而是顺手把 A 扩成 B：加入真实 skill 执行桥、重构 runtime notice、扩到 `06_wrap_up` 以后，都会显著抬高复杂度与回归风险 | 在设计文档、实施计划、skill 文档里同时写清本轮非目标：不实现真实 `task -> code -> verify` 执行桥、不重写 `dispatcher` notice 架构、不自动打通 `06_wrap_up` 以后阶段。先把 A 做成稳定闭环，再决定是否进入 B |

## 推荐优先级

### P0：优先立即优化

1. **S1 `autoLoopStatus` 显式接入 decider**
2. **S2 `reasonCodes` 结构化输出**
3. **S3 三阶段规则收口**
4. **S5 orchestrate → decider 信号打通**
5. **S7 自动推进放行条件收口**
6. **S9 `stage-flow-03-05` 专项测试补齐**

这 6 项都属于**闭环收口型优化**：

- 不改总体架构边界
- 不新增真实执行链
- 直接增强“loop 结果 -> 阶段决策 -> 安全推进”的一致性

### P1：收口稳定后再做

7. **S6 hooks 同步点最小接入**
8. **S8 `stageStatus` 语义固化**
9. **S10 `--auto-advance` 文档与参数口径统一**
10. **S11 hooks / decider 职责边界文档化**
11. **S12 非目标冻结与防扩 scope 约束**

这些属于**稳定性增强 / 认知收口型优化**，应在主线规则稳定后补齐。

---

## 总体建议

当前最优策略不是把 `orchestrate` 继续做大、做成新的真实执行中心，而是先把现有最小架构做成真正稳定的受控闭环：

- **`auto-loop` 只负责 todo 执行**
- **`next-step-decider` 负责统一阶段判定**
- **`orchestrate` 只负责汇总信号、打印建议、在安全条件满足时调用 `advance()`**
- **hooks 只负责同步，不负责判定**
- **`03_plan -> 04_implement -> 05_verify` 先做稳定，再谈更强自动化**

换句话说，下一步最值得做的不是“做更多自动执行”，而是“让当前这套判断和推进规则更稳、更薄、更一致”。

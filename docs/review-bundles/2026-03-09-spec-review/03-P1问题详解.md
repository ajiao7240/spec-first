# P1 质量问题详解

## P1-1: 缺少 Progressive Disclosure 设计

### 问题描述

**skill-creator 原则**:
> "When a skill supports multiple variations, keep only the core workflow in SKILL.md. Move variant-specific details into separate reference files."

**当前问题**:
- Phase 0.0-0.6 详细流程（154-344 行）全在 SKILL.md
- Step 0-8 详细流程（346-635 行）全在 SKILL.md
- 复杂度判定表格（384-391 行）全在 SKILL.md
- 反合理化守卫表（26-47 行）全在 SKILL.md

### 影响分析

1. **Context Window 浪费**
   - AI 每次加载 skill 都要读取所有细节
   - 即使只需要执行 Trivial 路径，也要加载 Complex 路径的内容

2. **按需加载失效**
   - 无法根据复杂度动态加载对应流程
   - 无法根据当前步骤只加载相关内容

3. **维护困难**
   - 修改某个步骤需要在 725 行中定位
   - 难以独立测试某个流程分支

### 解决方案

**重构为 Progressive Disclosure 结构**:

```
skills/spec-first/03-spec/
├── SKILL.md (~200 行)
│   ├── 核心流程概览
│   ├── 复杂度自适应规则
│   ├── 关键约束
│   └── 引用链接
└── references/
    ├── phase0-prd-workflow.md (Phase 0.0-0.6 详细流程)
    ├── steps-fr-ac-workflow.md (Step 0-8 详细流程)
    ├── complexity-matrix.md (复杂度判定表 + 示例)
    ├── anti-rationalization-guards.md (反合理化守卫表)
    ├── cli-commands-reference.md (CLI 命令详解 + 错误处理)
    └── quality-gates.md (C-PRD 评分 + Gate Check 规则)
```

**加载策略**:
- SKILL.md 始终加载（~200 行）
- Phase 0 执行时加载 `phase0-prd-workflow.md`
- Step 0-8 执行时加载 `steps-fr-ac-workflow.md`
- 复杂度判定时加载 `complexity-matrix.md`
- 需要时加载其他 references

### 实施步骤

详见 [09-重构方案.md](./09-重构方案.md)

## P1-2: 流程重复（Phase 0.2 vs Step 1）

### 问题描述

**Phase 0.2 Step 2（第 192-209 行）**:
```markdown
**Step 2: 自动上下文收集**
- 读取 constitution.md 获取项目约束
- 扫描仓库识别相关文件（基于关键词）
- 识别外部依赖（API/数据库/第三方服务）
- 检测现有代码/文档（判定 greenfield vs iteration）
```

**Step 1（第 362-377 行）**:
```markdown
### Step 1: Auto-Context
- 读取 prd.md 获取需求理解
- 读取 constitution.md 获取项目约束
- 扫描仓库识别相关文件（基于关键词）
- 识别外部依赖（API/数据库/第三方服务）
- 统计受影响文件数
```

**重复内容**:
- 读取 constitution.md
- 扫描仓库识别相关文件
- 识别外部依赖

### 影响分析

1. **重复工作浪费 Token**
   - 两次扫描仓库
   - 两次识别依赖
   - 约浪费 2-3k tokens

2. **可能产生不一致结果**
   - Phase 0.2 识别的依赖 vs Step 1 识别的依赖可能不同
   - 导致后续决策基于不一致的数据

3. **执行效率低**
   - 重复操作增加执行时间
   - 用户体验差

### 解决方案

**方案 A: 合并到 Phase 0.2（推荐）**

删除 Step 1，将其功能合并到 Phase 0.2：

```markdown
#### Phase 0.2: 质量扫描 + 上下文收集

**执行**:
1. 扫描用户输入完整度
2. 计算初始质量评分
3. **自动上下文收集**:
   - 读取 constitution.md 获取项目约束
   - 扫描仓库识别相关文件（基于关键词）
   - 识别外部依赖（API/数据库/第三方服务）
   - 统计受影响文件数（为 Step 2 复杂度判定准备）
   - 检测现有代码/文档（判定 greenfield vs iteration）
4. 判定场景类型（greenfield/iteration）
5. 生成质量报告

**输出**:
- findings.md 追加质量报告 + 上下文摘要
- 场景类型标记
- 受影响文件列表
- 外部依赖列表
```

**方案 B: 明确分工**

- Phase 0.2: 只做质量扫描 + 场景判定
- Step 1: 基于 PRD 做深度上下文收集

```markdown
#### Phase 0.2: 质量扫描 + 场景判定
- 扫描用户输入完整度
- 读取 constitution.md（仅获取约束）
- 判定场景类型（greenfield/iteration）
- **不做**: 深度依赖分析、受影响文件统计

### Step 1: 深度上下文收集
- 读取 prd.md 提取关键词
- 扫描仓库识别相关文件（基于 PRD 关键词，更精准）
- 识别外部依赖（基于 PRD 描述）
- 统计受影响文件数
- 检测现有实现（判定是否需要重构）
```

**推荐方案 A**，因为：
- 避免重复工作
- 数据一致性更好
- 流程更简洁

## P1-3: 复杂度判定时机不合理

### 问题描述

**当前时机**: Step 2（第 380-409 行）

**问题**:
- 复杂度判定在 Step 2，但 Phase 0 已执行大量工作
- 如果是 Trivial 任务，Phase 0.3-0.6 的很多步骤可能是浪费
- 例如：Phase 0.5 的"三道门禁 + 5 轮 3 问"对 Trivial 任务过度

### 影响分析

1. **简单任务效率低**
   - Trivial 任务也走完整 Phase 0.5（最多 5 轮 3 问）
   - 浪费用户时间和 token

2. **用户体验差**
   - 简单任务被问太多问题
   - 用户感觉流程繁琐

3. **资源浪费**
   - 不必要的 PRD 补全对话
   - 不必要的质量门禁检查

### 解决方案

**在 Phase 0.2 添加初步复杂度判定**:

```markdown
#### Phase 0.2: 质量扫描 + 初步复杂度判定

**执行**:
1. 质量扫描（已有）
2. 自动上下文收集（已有）
3. **初步复杂度判定**:
   - 基于用户输入长度（<100 字 → Trivial 候选）
   - 基于关键词数量（<3 个 → Simple 候选）
   - 基于关键词类型（包含"架构/重构/迁移" → Complex 候选）
   - 基于受影响文件数（已收集）

**复杂度自适应规则**:
- **Trivial/Simple**: Phase 0.5 最多 1 轮 1 问
- **Moderate/Complex**: Phase 0.5 最多 5 轮 3 问

**输出**: 初步复杂度标记（在 Step 2 基于实际文件数重新校准）
```

**Step 2 调整为复杂度校准**:

```markdown
### Step 2: 复杂度校准

**目标**: 基于实际文件数和依赖数校准复杂度。

**执行**:
1. 读取 Phase 0.2 的初步复杂度判定
2. 统计实际受影响文件数（Phase 0.2 已收集）
3. 统计实际外部依赖数（Phase 0.2 已收集）
4. 按四档判定标准校准复杂度
5. 如果校准后复杂度变化，更新 findings.md

**输出**: 最终复杂度档位
```

### 验收标准

- [ ] Phase 0.2 包含初步复杂度判定
- [ ] Phase 0.5 根据复杂度自适应提问轮次
- [ ] Step 2 调整为复杂度校准
- [ ] Trivial 任务测试通过（最多 1 轮 1 问）

## P1-4: CLI 命令错误处理缺失

### 问题描述

**位置**: 第 574-577 行

**当前内容**:
```markdown
4. 注册所有 FR 到追溯矩阵
   - 使用 `spec-first id next FR <abbr> --feature <featureId>` 生成 FR ID
   - 使用 `spec-first matrix update <featureId> <id> --title "标题" --yes` 注册到矩阵
```

**缺失内容**:
- CLI 命令失败时如何处理
- 网络问题、权限问题、ID 冲突如何恢复
- 是否允许跳过注册继续推进

### 影响分析

1. **流程可能卡住**
   - CLI 命令失败后 AI 不知道如何恢复
   - 用户需要手动介入

2. **数据不一致**
   - 如果跳过注册继续推进，追溯矩阵会缺失数据
   - 影响后续阶段的依赖关系

3. **用户体验差**
   - 失败时没有明确的错误提示
   - 不知道如何修复

### 解决方案

**添加完整的错误处理流程**:

```markdown
4. 注册所有 FR 到追溯矩阵

   **执行**:
   ```bash
   # 生成 FR ID
   spec-first id next FR <abbr> --feature <featureId>

   # 注册到矩阵
   spec-first matrix update <featureId> <id> --title "标题" --yes
   ```

   **错误处理**:

   **错误类型 1: ID 生成失败**
   - 原因: Feature 不存在、ABBR 无效、权限不足
   - 处理:
     1. 检查 Feature 是否存在（`ls specs/`）
     2. 检查 ABBR 格式（2-8 个大写字母）
     3. 重试 1 次
     4. 仍失败 → 标记 `[CLI_ERROR]` 到 findings.md，请求用户介入

   **错误类型 2: Matrix 更新失败**
   - 原因: ID 冲突、矩阵文件损坏、权限不足
   - 处理:
     1. 检查 ID 是否已存在（`grep <id> specs/<featureId>/traceability-matrix.md`）
     2. 如已存在，使用 `--force` 覆盖
     3. 重试 1 次
     4. 仍失败 → 标记 `[CLI_ERROR]` 到 findings.md，请求用户介入

   **错误类型 3: 网络超时**
   - 原因: CLI 依赖外部服务（如果有）
   - 处理:
     1. 等待 5 秒后重试
     2. 最多重试 3 次
     3. 仍失败 → 标记 `[CLI_ERROR]` 到 findings.md，请求用户介入

   **阻断规则**:
   - **不得跳过注册步骤继续推进**
   - 所有 FR 必须成功注册后才能执行 gate check
   - 如果注册失败，必须修复后重新执行 Step 8
```

### 验收标准

- [ ] 所有 CLI 命令都有错误处理
- [ ] 错误类型分类清晰
- [ ] 重试机制明确
- [ ] 阻断规则明确
- [ ] 测试各种失败场景

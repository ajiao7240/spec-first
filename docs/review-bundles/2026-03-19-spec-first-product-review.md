# Spec-First 产品视角评审

> 评审时间：2026-03-19  
> 评审视角：产品经理 / 研发流程产品化  
> 评审目标：从用户可用性、流程效率、可理解性、可持续性四个维度，审查 Spec-First 是否真正适合作为团队级研发协作产品。

## 1. 产品定位

Spec-First 不是一个通用型“随点随用”的 AI 工具，而是一套面向研发团队的 **研发流程 skill 集合**。  
它的核心价值不是“帮你多做一点事”，而是：

- 把需求、设计、任务、实现、验证、归档串成可追踪链路
- 把 AI 辅助工作限制在可审计、可回放、可推进的阶段内
- 把机器可消费的 runtime 真源与人类可阅读的 docs 输出分离

从产品角度看，它更接近：

- SDD 工作流编排器
- 质量门禁与追踪系统

而不是：

- 万能 AI 助手
- 随时可调用的通用 skill 库
- 低学习成本的轻量化 CLI

## 2. 目标用户与使用场景

### 2.1 用户分层

#### 新接入用户
- 需要理解阶段、门禁、记录节点、辅助节点、路由节点。
- 最怕第一次使用就被门禁挡住，却不知道下一步该做什么。
- 关心的是“我如何开始”和“为什么被阻止”。

#### 执行型研发
- 主要关注日常交付效率。
- 更希望随时调用诊断、恢复、状态、分析类 skill。
- 不希望每个动作都要先理解完整流程状态机。

#### 治理/审查者
- 关心可追踪、可回放、可审计。
- 看重 gate history、traceability、findings、stage-state 等记录链路。
- 期望系统能解释“为什么通过 / 为什么阻断”。

### 2.2 典型场景

- 新建 Feature，走规范化需求到验收的完整链路。
- 从中断状态恢复上下文，继续推进。
- 对设计、任务、实现、验证阶段进行门禁与质量控制。
- 诊断宿主环境、同步追踪矩阵、分析一致性风险。
- 归档阶段收口交付证据。

## 3. 当前产品状态

### 3.1 已具备的产品能力

- 阶段边界清晰，适合规范驱动开发。
- `runtime / docs / records` 分层明确。
- `gate-history.jsonl`、`traceability-matrix.md`、`findings.md` 等记录机制具备审计价值。
- 关键阶段节点有质量门禁，能降低“边做边漂”的风险。
- `first` 已明确回到“校验 final outputs”定位，不再把 CLI 误做生成器。

### 3.2 当前的产品短板

#### 入口摩擦偏高
- 用户需要先理解大量术语和阶段名。
- 很多 skill 并不能“随时可用”，而是依赖前置状态。
- 首次使用的成功率和理解成本偏高。

#### 认知负担偏重
- 同时存在 `stage-state.json`、`task_plan.md`、`findings.md`、`gate-history.jsonl`、`traceability-matrix.md`、`docs-index.json`、runtime index 等多个状态载体。
- 对产品用户来说，这是一种“信息过剩”，而不是“信息丰富”。

#### 门禁语义过多
- 硬门、软门、伴生门、调度门、路由门、记录门同时存在。
- 这些分类对工程实现有价值，但对产品用户来说过于细碎。

#### 记录压力偏大
- `findings.md` 虽然已经退回记录文件，但在系统里仍然是最容易被滥用的文件。
- 如果用户频繁把它当门禁输入，产品体验会继续变重。

#### 可用性不一致
- 只读 / 诊断 / 分析型 skill 与阶段推进型 skill 的使用门槛差异大。
- 如果没有明确的可用性分层，用户很容易形成“为什么这个能用，那个不能用”的挫败感。

## 4. 关键产品问题

### 4.1 不是“能力不足”，而是“复杂度高于用户容忍度”

当前体系最强的问题，不是功能少，而是：

- 用户要记住的概念太多
- 用户要理解的门禁类型太多
- 用户要判断的文件角色太多

对于研发团队来说，这意味着：

- 工程师需要花额外时间理解流程，而不是直接进入问题解决
- 团队容易在“文件职责”上发生心智混乱
- 流程控制本身会成为新的负担

### 4.2 “随时可用”与“质量控制”之间的张力

Spec-First 天然是流程型产品，不可能让所有 skill 都毫无限制地随时调用。  
但产品层面仍然需要：

- 让诊断、恢复、状态、分析类 skill 低门槛可用
- 让产物型 skill 保留阶段门
- 让被阻断时的反馈更短、更直接、更可执行

如果这条边界不清晰，用户会感觉系统“过度严格”或“不可预测”。

### 4.3 语义对齐问题

当前产品中最容易造成困惑的是：

- skill 文档怎么写
- runtime 实际怎么判
- docs 如何参与上下文
- 哪些是记录，哪些是真源，哪些只是辅助索引

对研发团队来说，产品要么清楚，要么就会被当成“规则太多的黑盒”。

## 5. 产品目标

如果把 Spec-First 作为一个真正的研发流程产品，建议的目标可以定义为：

### 5.1 北极星目标

**让研发团队在保持流程可审计的前提下，更快、更稳地完成从需求到验收的 AI 辅助研发。**

### 5.2 阶段目标

- 降低首次接入门槛
- 降低流程理解成本
- 提高诊断和恢复类 skill 的即时可用性
- 保持关键质量门禁
- 让用户更快知道“当前该做什么”

## 6. 产品策略

### 6.1 保留强流程，但弱化复杂表达

应该保留的是：

- 阶段推进
- 质量门禁
- traceability
- gate history
- 证据落盘

应该弱化的是：

- 过度细碎的门禁术语
- 对用户暴露过多内部文件名
- 对“记录 / 门禁 / 索引 / 真源”的混乱解释

### 6.2 以“可用性分层”代替“统一门禁思维”

建议把 skill 分成三类产品能力：

1. **随时可用**
   - `catchup`
   - `status`
   - `doctor`
   - `sync`
   - `analyze`
   - `plan`

2. **条件可用**
   - `spec`
   - `design`
   - `task`
   - `review`
   - `verify`

3. **阶段推进**
   - `init`
   - `code`
   - `archive`
   - `orchestrate`

这比单纯强调“门禁强弱”更符合产品认知。

### 6.3 用“下一步动作”替代“规则说明书”

用户被拦住后，最需要的是：

- 现在缺什么
- 先补哪一个
- 预计下一步是什么

而不是一大段门禁解释。  
产品层要将门禁结果压缩成“最小修复路径”。

## 7. 优先级建议

### 7.1 必须修

1. **统一可用性表达**
   - 把 skill 分成“随时可用 / 条件可用 / 阶段推进”三类。
   - 让用户在入口就知道预期。

2. **压缩对内部文件的认知负担**
   - 对外只保留“机器真源 / 人类阅读 / 工作记录”的三层抽象。
   - 不要要求用户理解太多状态文件差异。

3. **把被阻断后的反馈改成行动导向**
   - 输出“缺什么、先补什么、下一步是什么”。
   - 不要只输出规则名称。

### 7.2 建议修

1. **减少 `findings.md` 的心理负担**
   - 它应该是记录，不是“万用入口”。
   - 尽量减少它在流程中的存在感。

2. **收紧 `docs-index.json` 的产品语义**
   - 它应只做阅读路由，不做二次门禁。
   - 避免用户误把索引当真源。

3. **给每个 skill 做一张最小卡片**
   - 作用
   - 适用时机
   - 输入
   - 输出
   - 常见阻塞原因

### 7.3 可保留

1. **硬门机制**
   - `code / verify / archive` 等关键节点保留硬门是合理的。

2. **记录体系**
   - `gate-history.jsonl`
   - `traceability-matrix.md`
   - `findings.md`
   - `task_plan.md`

3. **runtime/docs 分层**
   - 这条原则应继续保留。

## 8. 用户旅程建议

### 8.1 新用户旅程

1. 进入项目
2. 看到当前阶段和推荐动作
3. 先用 `status / doctor / catchup`
4. 再进入 `spec / design / task`
5. 遇到阻塞时得到最小修复路径

### 8.2 执行用户旅程

1. 打开当前 Feature
2. 看下一步建议
3. 调用对应 skill
4. 通过门禁后继续推进
5. 在 `verify / archive` 收口

### 8.3 审查用户旅程

1. 看 gate history
2. 看 traceability
3. 看 findings
4. 看最终 runtime/docs
5. 判断是否可推进到 release / done

## 9. 风险与缓解

| 风险 | 影响 | 缓解 |
|---|---|---|
| 入口门槛过高 | 新用户流失 | 做可用性分层与最短修复路径 |
| 记录文件过多 | 认知负担加重 | 统一三层抽象：真源 / 阅读 / 记录 |
| 门禁术语过细 | 用户理解困难 | 简化对外表述，保留内部实现细节 |
| 阶段门过强 | 随用性下降 | 只读/诊断 skill 低门槛化 |
| 规则与文档不一致 | 维护成本上升 | 文档与代码定期对表 |

## 10. 关键指标

如果从产品角度衡量这套体系是否成功，建议关注：

- **首次可用时间**
  - 新用户从安装到第一次成功执行核心 skill 的时间

- **阻断后恢复时间**
  - 用户被门禁挡住后，到能继续推进的时间

- **诊断命中率**
  - `doctor / status / catchup` 是否能快速给出有效建议

- **阶段推进成功率**
  - `spec → design → task → code → verify → archive` 的推进通过率

- **记录完整率**
  - `gate-history / findings / traceability` 是否能支撑回放和审查

- **用户理解成本**
  - 团队是否能在不看实现细节的情况下理解“下一步怎么走”

## 11. 结论

Spec-First 的产品方向是成立的：  
它适合做成一套 **研发流程 skill 集合**，帮助团队用 AI 提高质量、可追踪性和阶段纪律。

但如果要把它真正做成产品，而不是工程系统，下一步重点不是继续加规则，而是：

- 收敛可用性分层
- 降低用户对内部文件和门禁术语的理解负担
- 强化被阻断后的下一步动作
- 保留质量控制，同时提升“随时可用”的体验

一句话总结：

**这套系统的产品价值很强，但产品体验还偏工程化。要把它从“强流程工具”打磨成“真正可用的研发产品”，关键在于降认知负担，而不是继续增加复杂度。**

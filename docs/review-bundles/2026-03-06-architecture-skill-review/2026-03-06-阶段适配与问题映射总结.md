# 2026-03-06 阶段适配与问题映射总结

## 1. 不同规模阶段的适配度

| 项目阶段 | 适配度 | 擅长解决 | 当前短板 |
|---|---|---|---|
| `0-1` | 高 | 需求收敛、方案落盘、任务拆解、建立最小规范 | 对超早期高频变更会偏重 |
| `1-10` | 很高 | 多人协作、阶段治理、追踪矩阵、验证门禁、会话恢复 | 还需补更多自动化集成 |
| `10-100` | 中 | 统一研发流程骨架、沉淀标准产物、审计推进依据 | 多团队、多仓、多平台、发布治理还不够完整 |

## 2. 问题到链路能力映射

| 问题 | 对应链路能力 | 产物 / skill |
|---|---|---|
| 想法很多，但不知道先做什么 | 从需求到计划的主链收敛 | `spec` / `design` / `task`，`spec.md` `design.md` `task_plan.md` |
| 需求和代码经常对不上 | FR-DS-TASK-TC 追踪闭环 | `traceability-matrix.md`，`spec` `design` `task` `test` |
| 还没准备好就开始写代码 | 阶段 hard-gate 阻断 | `hard-gate`，`code` `code-review` `verify` `archive` |
| 多人协作时经常断上下文 | 运行态外部记忆 | `findings.md` `task_plan.md` `stage-state.json`，`catchup` `status` |
| 做到一半不知道现在在哪个阶段 | 阶段状态可视化与切换 | `feature-current` `feature-switch` `status` |
| 设计跳过了，后面返工 | 先规格、后设计、再任务 | `spec` `design` `task` |
| 代码做完但测试没跟上 | 测试用例先建模，验证后推进 | `test` `verify`，`checklists/` `TC` 映射 |
| 团队说“应该没问题”，但没有证据 | Gate + Verify 证据化 | `verify` `gate check` `matrix check` |
| 会话一断，AI/人都不知道做到哪了 | 上下文恢复与摘要 | `catchup` `findings.md` |
| 做完一个功能，没有沉淀经验 | 归档复盘和知识回写 | `archive`，`retro.md` |
| 不同 skill 写法不一致，质量漂移 | skill catalog 治理和自动化校验 | `skill-catalog.test.ts`，各 `SKILL.md` |
| 新项目从零开始没有认知入口 | 项目快速认知与初始化 | `first` `init`，`docs/first/*` |

## 3. 总结

一句话判断：

- `0-1`：能很好辅助，尤其适合把混乱想法收敛成可执行闭环。
- `1-10`：当前最匹配，收益最高。
- `10-100`：能作为研发治理内核，但还需要继续补组织级能力。

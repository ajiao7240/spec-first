# 第一阶段/第二阶段完成后的第三阶段后续收口方案

## 1. 文档目的

本文档用于总结当前 `first` 第三阶段的实现现状，并明确：

- 哪些问题必须继续解决
- 哪些能力属于增强项，可后续再做
- 后续建议的实施顺序
- 每一阶段的验收标准

目标不是继续扩大范围，而是先把第三阶段收口成一个可长期维护、可解释、可审计的稳定能力。

## 2. 当前状态总结

当前第三阶段已经完成的能力：

- `wrap_up / done` 阶段已接入结构性变更检测
- 已支持基于规则的增量回写
- 已支持 `module / api / flow / convention / risk / tech-stack` 的 `add / update / remove`
- 对未支持或不安全的变更，不再伪装成成功，而是阻断自动 writeback
- 阻断时会生成 `project-cognition-suggestions/*.md` 建议文件
- 治理日志已记录 suggestion 路径与阻断结果
- 全量 skill 的 `confirm_policy` 已补齐到 frontmatter

当前已通过验证：

- `pnpm -s typecheck`
- `pnpm -s vitest run`

当前基线结果：

- `174 passed`
- `1536 passed | 2 skipped (1538)`

## 3. 必须继续解决的问题

以下问题属于后续必须解决的问题。这里的“必须”指的是：不解决会持续影响第三阶段的长期稳定性、治理一致性或安全边界。

### 3.1 未覆盖资产的最小安全边界

当前尚未完成 deterministic `update/remove` 的资产包括：

- `domain-model`
- `database-schema`
- `steering`

这三类资产不一定要立即支持自动合并，但必须保证：

- 能被识别为未支持或高风险变更
- 不会被误判成可自动回写
- 遇到此类变更时，统一进入阻断 + suggestion 模式

如果安全边界不清晰，后续会直接污染 `first` runtime 真源。

### 3.2 冲突检测的可解释性

当前已经能阻断不安全变更，但后续必须把冲突分类进一步稳定下来。至少需要明确区分：

- 自动合并成功
- 因缺少旧值而阻断
- 因目标冲突而阻断
- 因当前资产暂不支持而阻断

治理日志、suggestion 文档、控制台输出都应能表达这一层语义。否则后续人工处理成本会很高。

### 3.3 控制面真源一致性

`confirm_policy` 相关治理已经明显改善，但这类控制面字段后续仍必须保持单一真源和一致解释：

- `README`
- `SHARED`
- 各 `SKILL.md`
- 路由/命令实现

后续如果再出现这几层口径不一致，会重新引入“文档说一套、实际跑一套”的治理风险。

### 3.4 control-plane 文档与实现对齐

以下控制面节点必须继续保持文档与实现一致：

- `feature`
- `status`
- `doctor`
- `onboarding`

要求不是把输出做得更复杂，而是：

- 文档承诺什么，实现就提供什么
- 实现不支持的行为，文档不能提前承诺

这属于必须持续守住的边界。

## 4. 非必须但建议后续做的能力

以下能力不属于当前版本的硬门槛。它们是增强项，可以后续按优先级逐步投入。

### 4.1 `domain-model / database-schema / steering` 的 deterministic 自动合并

这是增强项，不是当前必须项。只要当前策略是：

- 检测
- 阻断
- suggestion

就已经满足当前安全要求。

### 4.2 `assisted apply`

当前 suggestion mode 已经能让系统：

- 检测风险
- 生成合并建议
- 不直接污染真源

在此基础上，再做“用户确认后应用建议 patch”的 `assisted apply` 属于下一阶段增强能力，不是本阶段硬门槛。

### 4.3 更智能的 LLM merge 建议

例如：

- 更好的冲突解释
- 更强的 patch 组织方式
- 更细的候选 diff 展示

这些都不是当前必须项。当前重点仍应是“安全和可解释”优先于“更智能”。

### 4.4 更广的文档治理优化

例如：

- 版本字段收缩
- 更多 README 整理
- 历史方案文档去重

这些属于持续治理工作，不应抢在第三阶段核心闭环之前。

## 5. 推荐实施顺序

后续建议按以下顺序推进。

### 第一步：补齐未覆盖资产的安全边界

目标：

- `domain-model / database-schema / steering` 一旦出现 `update/remove`
- 必须被清晰识别
- 必须进入阻断 + suggestion

这一阶段不追求自动合并，只追求“不会错写”。

### 第二步：细化冲突分类与治理日志

目标：

- 让每一种阻断都能说清楚原因
- 让 suggestion 文件可以直接作为人工决策输入
- 让后续排查不依赖读代码

### 第三步：按优先级扩 deterministic 合并

建议顺序：

1. `domain-model`
2. `database-schema`
3. `steering`

原因：

- `domain-model` 最容易规则化
- `database-schema` 次之，但需要更严格冲突判断
- `steering` 最抽象，最不适合先做自动改写

### 第四步：再考虑 `assisted apply`

只有在前三步稳定后，才建议推进：

- suggestion 展示
- 人工确认
- 应用候选 patch

不建议在当前阶段直接让 LLM 自动落盘。

## 6. 推荐验收标准

### 必选验收

1. 未覆盖资产不会被误合并
2. 冲突类型在日志中可明确识别
3. suggestion 文件可定位、可阅读、可复核
4. `README / SHARED / SKILL.md / 实现` 的控制面口径保持一致
5. `pnpm -s typecheck` 通过
6. `pnpm -s vitest run` 通过

### 增强验收

1. `domain-model` 支持 deterministic `update/remove`
2. `database-schema` 支持 deterministic `update/remove`
3. `assisted apply` 可在人工确认后应用建议 patch

## 7. 结论

当前第三阶段已经完成了“最小可用且安全”的闭环：

- 支持一批核心资产的 deterministic 增量合并
- 对高风险场景不再盲写
- suggestion mode 已落地

后续最佳策略不是立刻追求更智能，而是继续把：

- 安全边界
- 冲突可解释性
- 控制面真源一致性

这三件事做扎实。

一句话总结：

第三阶段后续应坚持 `detect -> block -> suggest -> assisted apply` 的路线，而不是直接走 `detect -> llm auto merge -> writeback`。

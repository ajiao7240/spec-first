# 2026-03-18 first 第三阶段后续实施建议

## 1. 目标

在当前第三阶段最小可用闭环已经落地的基础上，继续推进 `first` 项目认知回写能力，但避免过早引入高风险的“智能自动改写”。

核心原则：

1. 先把 deterministic 增量回写做完整
2. 再补冲突检测与阻断
3. 最后才引入 LLM 辅助建议

---

## 2. 当前已完成能力

当前已经具备：

- wrap_up / done 阶段触发 project cognition writeback
- 基于 Feature 产物的结构性变更识别
- 变更类型到 first runtime 资产的映射
- 针对部分资产的增量回写
- 受影响 docs/first 投影刷新
- 治理日志记录

当前已覆盖的结构性变更类型：

- `module`
- `api`
- `risk`
- `flow`
- `convention`
- `tech-stack`

当前已覆盖的 runtime 资产：

- `summary.json`
- `structure-overview.json`
- `api-contracts.json`
- `critical-flows.json`
- `entry-guide.json`
- `conventions.json`

---

## 3. 剩余问题本质

当前未完成的能力，不适合一次性直接做成“智能自动回写”，主要原因有三点：

1. `update/remove` 语义尚未稳定
2. `domain-model / database-schema / steering` 这类资产的增量规则还不完整
3. 冲突场景还没有可靠的阻断与审计机制

因此，下一步最优策略不是“更聪明”，而是“更稳、更可解释”。

---

## 4. 推荐实施路线

### 4.1 Phase 3.1：补完整 deterministic 增量语义

优先目标：

- 为当前已支持的变更类型补齐 `add / update / remove`
- 保持规则化、可预测、可测试

建议顺序：

1. `module`
2. `api`
3. `flow`
4. `convention`
5. `risk`
6. `tech-stack`

原因：

- 这些类型目前已经有基础实现
- 在现有数据结构上最容易定义稳定 merge 规则
- 能最快扩大第三阶段的真实覆盖面

验收标准：

- `add / update / remove` 都有单测
- 治理日志能清晰记录变更动作
- 不出现 silent overwrite

---

### 4.2 Phase 3.2：扩展资产覆盖面

在基础语义稳定后，再扩到更复杂资产：

建议顺序：

1. `domain-model.json`
2. `database-schema.json`
3. `steering.json`

原因：

- `domain-model` 与 Feature 语义最接近，较适合规则识别
- `database-schema` 风险较高，但仍有机会通过规则实现稳定增量
- `steering` 最抽象，应最后处理

验收标准：

- 每类资产都有独立 merge 策略
- 每类资产都有最小 e2e 回写测试
- docs 投影可随受影响资产增量刷新

---

### 4.3 Phase 3.3：先做冲突检测，不急着做智能冲突解决

推荐先实现：

- 冲突识别
- 冲突分类
- 自动阻断
- 治理日志记录候选差异

不要直接做：

- LLM 自动改写 runtime 真源
- 遇冲突仍继续自动写入

建议冲突类型：

- `same-target-different-evidence`
- `remove-with-live-reference`
- `schema-shape-conflict`
- `steering-summary-conflict`

推荐行为：

- 检测到冲突时，返回 `blocked`
- 写入治理日志
- 给出人工处理建议

验收标准：

- 冲突不会 silent pass
- 冲突日志可追溯
- wrap_up / done 不会因为冲突而产生伪稳定写入

---

### 4.4 Phase 3.4：引入 LLM 辅助，但先做 suggestion mode

LLM 第一阶段只做：

- 解释冲突
- 生成 merge 建议
- 生成候选 patch
- 辅助人工确认

不要第一版就做：

- 自动落盘
- 无确认 merge

推荐模式：

- `detect -> block -> suggest`

而不是：

- `detect -> auto-merge`

原因：

- 当前项目对治理和审计要求高
- suggestion mode 风险最低
- 更适合作为生产环境中的渐进式增强

验收标准：

- LLM 输出只作为候选建议
- 最终写入仍需 deterministic apply 或人工确认
- 治理日志保留 suggestion 与最终决策

---

## 5. 不推荐方案

以下方案不建议采用：

1. 在 `update/remove` 规则未稳定前直接接入 LLM 自动合并
2. 没有冲突阻断机制时允许自动写入
3. 在未扩完资产覆盖前就把“第三阶段已完成”定义为智能知识同步
4. 继续依赖 `refresh-all` 伪装成“增量更新”

这些做法会制造新的伪稳定，后续维护成本会更高。

---

## 6. 测试策略建议

后续必须新增的测试类型：

### 6.1 单元测试

- `update/remove` 语义测试
- 各资产 merge 规则测试
- 冲突检测分类测试

### 6.2 集成测试

- wrap_up -> incremental writeback
- done -> incremental writeback
- 冲突出现时 blocked 行为

### 6.3 治理测试

- 治理日志字段完整性
- docs 投影只刷新受影响文档
- 冲突时不得产生 silent write

---

## 7. 推荐优先级

按实施顺序建议：

1. `add/update/remove` 规则补全
2. `domain-model / database-schema / steering` 扩展
3. 冲突检测与阻断
4. LLM suggestion mode
5. 如前述都稳定，再评估 assisted apply

---

## 8. 最终建议

第三阶段后续的最佳实践不是“尽快做智能化”，而是：

`先把规则型增量回写做全做稳，再让 LLM 进入建议层，而不是直接进入自动写入层。`

这样做的好处是：

- 风险最低
- 可解释性最好
- 审计最清晰
- 更符合当前 spec-first 的治理风格

## 详细问题清单

### F-01: 规则源分裂 - REQ-PRD vs REQ-*

**状态**：❌ **未解决**

**问题描述**：
- **文档**：SKILL.md 禁止使用 `REQ-PRD`，要求使用 `REQ-*`
- **代码**：运行时强制要求 `REQ-PRD-*`

**证据**：
```typescript
// src/core/trace-engine/matrix.ts:63
const hasPrd = (fr.upstream ?? []).some(u => u.startsWith('REQ-PRD-'));

// src/core/trace-engine/matrix.ts:70
if (!hasPrd) missing.push('REQ-PRD-*');

// src/core/gate-engine/sca.ts:151, 304, 312
!(r.upstream ?? []).some(u => u.startsWith('REQ-PRD-'))
```

**修复历史**：
- ✅ 提交 `8a5daa5` 修改了 SKILL.md 文档
- ❌ **未修改任何运行时代码**

**影响范围**：
- matrix.ts（断链检测）
- sca.ts（SCA 检查）
- 所有使用 `REQ-*` ID 的 Feature 将在运行时失败

**修复方案**：
1. 统一代码使用 `REQ-*` 而非 `REQ-PRD-*`
2. 新增 `src/shared/id-rules.ts` 作为单一规则源
3. 在 CI 中加入规则一致性检查

### F-02: 非法 ID 被静默降级为 Feature

**状态**：❌ **未解决**

**问题描述**：
`parseMatrixContent()` 对 `validateId()` 失败的 ID 直接降级为 `Feature`，造成非法 ID 继续流转。

**证据**：
```typescript
// src/core/trace-engine/matrix.ts:151
const validation = validateId(id);
const type: IdType = validation.type ?? 'Feature';  // ❌ 非法 ID 降级
```

**影响**：
- 非法 ID 不会触发告警
- 断链/漏链被吞掉
- 数据质量问题无法暴露

**修复方案**：
```typescript
// 推荐：强失败策略
if (!validation.valid) {
  throw new Error(`Invalid ID format: "${id}" - ${validation.error}`);
}
```

### F-03: trace fix 写入文件路径污染链路字段

**状态**：❌ **未解决**

**问题描述**：
`trace fix` 将文件路径作为 downstream 填充，导致链路字段混入非 ID 值。

**证据**：
```typescript
// src/cli/commands/trace.ts:53
const placeholder = `src/core/**/${task.id.toLowerCase()}.ts`;
content = content.replace(
  new RegExp(`(\\| ${task.id} \\| TASK \\| [^|]+ \\| [^|]+ \\| [^|]+ \\|)  \\|`, 'g'),
  `$1 ${placeholder} |`
);
```

**影响**：
- downstream 字段包含路径而非 ID
- 链路追溯失败
- `validate matrix` 无法检测

**修复方案**：
1. 移除自动填充路径的逻辑
2. 或新增专用字段（如 `Artifacts`）存储路径
3. 在 `validate matrix` 中校验 downstream 仅允许合法 ID

---

### F-04: validate format 只检查 ID 列

**状态**：❌ **未解决**

**问题描述**：
格式校验仅扫描矩阵 ID 列，跳过非法 ID，且不检查 upstream/downstream。

**证据**：
```typescript
// src/core/validators/format-validator.ts:88
if (!validateId(id).valid) continue;  // ❌ 非法 ID 被跳过

// src/core/validators/format-validator.ts:85
const match = line.match(/^\|\s*([A-Z][A-Z0-9-]{2,40})\s*\|/);  // ❌ 只匹配第1列
```

**影响**：
- 非法 ID 无法暴露
- upstream/downstream 完全未校验
- 格式校验形同虚设

**修复方案**：
1. 扫描 matrix 全表：ID 列 + upstream/downstream
2. 将非法 ID 列为 `errors` 而非 skip
3. 对 `AC` / `TC` 做显式规则校验

### F-05: C5 号称 AC Coverage，实际仍是 FR Coverage

**状态**：❌ **未解决**

**问题描述**：
`calcTestCoverageAC()` 与 `calcTestCoverageFR()` 使用同一逻辑，且 matrix 没有 AC 类型。

**证据**：
```typescript
// src/core/trace-engine/coverage.ts:80-83
/** C5: Test Coverage (AC) — 同 C4（AC 级别细化留给 Phase B） */
function calcTestCoverageAC(frRows: MatrixRow[], tcRows: MatrixRow[]): number {
  return calcUpstreamCoverage(frRows, tcRows);  // ❌ 复用 C4 逻辑
}
```

**影响**：
- C5 实际无法衡量 AC 覆盖率
- Gate 条件中的 C5 描述误导
- AC 作为概念存在但无法实施

**修复方案**：
1. 将 AC 作为独立 ID 类型（`IdType` 支持 `AC`）
2. 新增 AC→TC/TASK 关联规则
3. 计算真实 C5（而非复用 C4）

### F-06: stage-state 校验过弱 + 写入无锁

**状态**：❌ **未解决**

**问题描述**：
`isStageState()` 仅检查 `currentStage` 字段，损坏的 state 仍被接受；`writeJson` 无锁且非原子。

**证据**：
```typescript
// src/shared/validators.ts:11-14
export function isStageState(data: unknown): data is StageState {
  if (!isObj(data)) return false;
  return typeof data.currentStage === 'string';  // ❌ 校验过弱
}
```

**影响**：
- 损坏的 state 文件仍被接受
- 并发写入覆盖风险
- 状态恢复可靠性不足

**修复方案**：
1. 强化校验：必需字段（featureId/mode/size/history/updatedAt）
2. 引入文件锁或 CAS（基于 `updatedAt`）
3. 使用原子写（tmp + rename）

---

### F-07: auto-loop 退避仅记录不等待

**状态**：❌ **未解决**

**问题描述**：
重试逻辑只记录 backoff 时间，但没有实际等待；`todo-state` 写入无锁。

**证据**：
```typescript
// src/core/ai-orchestrator/auto-loop.ts:195-206
state = updateAutoLoopLastResult(
  state,
  task.id,
  'pending',
  `retry scheduled (${retryDecision.backoffMs}ms): ${result.message}`,  // ❌ 仅记录
);
// ❌ 缺少 sleep(backoffMs) 或定时调度
```

**影响**：
- 重试立即发生而非等待
- 退避策略无效
- 并发执行可能导致状态回退

**修复方案**：
1. 在 retry scheduled 后加入 `sleep(backoffMs)` 或定时调度
2. 引入 `todo-state` 文件锁或版本号 CAS

**证据**：

**影响**：
- 使用者按文档产出但运行时拒绝
- 人工修复被误导
- 规则反复漂移

**修复方案**：
1. 建立单一规则源（`id-rules.ts`）
2. CI 加入"规则源一致性检查"
3. 文档从规则源生成

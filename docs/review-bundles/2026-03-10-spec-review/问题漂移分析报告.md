## 漂移根因分析

### 1. 概念混淆：功能实现 vs 设计合理

**混淆点**：
- "断链检测已实现" ≠ "断链检测规则正确"
- "功能已工作" ≠ "功能设计合理"

**证据**：
```typescript
// v5 验证：断链检测功能已实现
const hasPrd = (fr.upstream ?? []).some(u => u.startsWith('REQ-PRD-'));
// ✅ 功能确实存在

// 但 v5 忽略了原始问题：规则本身有矛盾
// 文档禁止 REQ-PRD，代码强制要求 REQ-PRD
// ❌ 设计不合理
```

### 2. 评估范围收窄

**原始审计范围**（`id-linkage-deep-audit.md`）：
- ID 生成 → 传播 → 校验 → 持久化 → 恢复
- 全链路一致性
- 健壮性、并发安全、异常处理

**v5 评估范围**：
- ✅ 断链检测是否完整
- ✅ 孤儿检测是否合理
- ✅ C6/C7 是否对齐
- ❌ **不包含**：规则分裂、静默降级、健壮性问题

### 3. "事实错误"指控的误导

v5 通过指出 v4 的"事实错误"来否定整个修复方案，但：

- **v4 的"错误"可能只是表述不清**
- **v4 的核心观点（需要修复）是正确的**
- **v5 用"功能已实现"否定了"设计需优化"**

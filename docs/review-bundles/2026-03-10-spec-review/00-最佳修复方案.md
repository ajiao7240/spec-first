# ID 链路最佳修复方案

> 生成日期：2026-03-10
> 原则：重构而非修补，降低复杂度而非增加

---

## 一、核心问题重新定位

### 当前文档的误区

三份文档把**症状当成了疾病**：

| 文档描述 | 实际本质 |
|---------|---------|
| F-02 降级为 Feature | 症状：容错机制 |
| F-01 REQ-PRD 规则分裂 | 症状：历史遗留 |
| F-04 校验跳过非法ID | 症状：防御性编程 |
| F-05 C5 = C4 | 症状：功能未实现 |

### 真正的疾病

1. **规则源头分裂** - 代码、文档、运行时各自定义规则
2. **职责边界混乱** - 解析、校验、门禁逻辑混在一起
3. **类型定义不完整** - 缺少 AC，REQ-PRD 历史遗留

---

## 二、最佳方案：规则驱动 + 职责分离

### 架构对比

**❌ 修补式（文档方案）**：
```
parseMatrix (降级) → checkMatrix (阶段感知) → gate (分层校验)
```
- 问题：3层架构，职责混乱，+300 行代码

**✅ 重构式（最佳方案）**：
```
id-rules.ts (统一规则源)
    ↓
parseMatrix (严格解析，只做格式校验)
    ↓
checkMatrix (简单校验，只做链路校验)
    ↓
gate (清晰门禁，只做完整性校验)
```
- 优势：单一职责，逻辑清晰，-100 行代码

### Step 1: 统一规则源（新增文件）

**文件**：`src/shared/id-rules.ts`

```typescript
export const ID_RULES = {
  Feature: {
    regex: /^FSREQ-\d{8}-[A-Z][A-Z0-9]{1,15}-\d{3}$/,
    upstream: [],
    downstream: ['FR'],
  },
  REQ: {
    regex: /^REQ-[A-Z][A-Z0-9]{1,15}-\d{3}$/,
    upstream: [],
    downstream: ['FR'],
  },
  FR: {
    regex: /^FR-[A-Z][A-Z0-9]{1,15}-\d{3}$/,
    upstream: ['REQ'],
    downstream: ['DS', 'AC', 'TC'],
  },
  AC: {
    regex: /^AC-[A-Z][A-Z0-9]{1,15}-\d{3}-\d{2}$/,
    upstream: ['FR'],
    downstream: ['TC', 'TASK'],
  },
  DS: {
    regex: /^DS-[A-Z][A-Z0-9]{1,15}-\d{3}$/,
    upstream: ['FR'],
    downstream: ['TASK'],
  },
  TASK: {
    regex: /^TASK-[A-Z][A-Z0-9]{1,15}-\d{3}$/,
    upstream: ['DS', 'AC'],
    downstream: [],
  },
  TC: {
    regex: /^TC-(UT|IT|E2E|ST)-[A-Z][A-Z0-9]{1,15}-\d{3}$/,
    upstream: ['FR', 'AC'],
    downstream: [],
  },
} as const;

export type IdType = keyof typeof ID_RULES;

// 自动生成 validator
export function validateId(id: string): IdValidationResult {
  for (const [type, rule] of Object.entries(ID_RULES)) {
    if (rule.regex.test(id)) {
      return { valid: true, type: type as IdType };
    }
  }
  return { valid: false, error: `Unknown ID: "${id}"` };
}

// 自动生成链路校验
export function validateChain(id: string, upstream: string[]): string[] {
  const validation = validateId(id);
  if (!validation.valid) return ['Invalid ID format'];

  const rule = ID_RULES[validation.type!];
  const errors: string[] = [];

  for (const upId of upstream) {
    const upValidation = validateId(upId);
    if (!upValidation.valid) {
      errors.push(`Invalid upstream: ${upId}`);
      continue;
    }
    if (!rule.upstream.includes(upValidation.type!)) {
      errors.push(`${validation.type} cannot have ${upValidation.type} as upstream`);
    }
  }

  return errors;
}
```

**优势**：
- 规则即代码，单一真理源
- 类型安全，自动补全
- 易于扩展新类型

### Step 3: 简化校验（修改文件）

**文件**：`src/core/trace-engine/matrix.ts`

```typescript
export function checkMatrix(
  featureId: string,
  projectRoot: string
): MatrixCheckResult {
  const rows = parseMatrix(featureId, projectRoot);
  const errors: string[] = [];

  // 职责：只做链路合法性校验
  for (const row of rows) {
    const chainErrors = validateChain(row.id, row.upstream ?? []);
    errors.push(...chainErrors.map(e => `${row.id}: ${e}`));

    // 校验 downstream 也是合法 ID
    for (const downId of row.downstream ?? []) {
      if (!validateId(downId).valid) {
        errors.push(`${row.id}: invalid downstream "${downId}"`);
      }
    }
  }

  return {
    total: rows.length,
    errors,
    valid: errors.length === 0
  };
}
```

### Step 4: 清晰门禁（新增文件）

**文件**：`src/core/gate-engine/completeness-gate.ts`

```typescript
export function checkCompleteness(rows: MatrixRow[]): GateResult {
  const frRows = rows.filter(r => r.type === 'FR');
  const errors: string[] = [];

  for (const fr of frRows) {
    // 简单规则：FR 必须有 REQ + DS + (AC 或 TC)
    const hasReq = fr.upstream?.some(u => u.startsWith('REQ-'));
    const hasDs = rows.some(r => r.type === 'DS' && r.upstream?.includes(fr.id));
    const hasAcOrTc = rows.some(r =>
      (r.type === 'AC' || r.type === 'TC') && r.upstream?.includes(fr.id)
    );

    if (!hasReq) errors.push(`${fr.id}: missing REQ upstream`);
    if (!hasDs) errors.push(`${fr.id}: missing DS downstream`);
    if (!hasAcOrTc) errors.push(`${fr.id}: missing AC or TC`);
  }

  return {
    pass: errors.length === 0,
    errors,
  };
}
```

**集成到 gate**：
```typescript
// src/core/gate-engine/gate-evaluator.ts
const completenessResult = checkCompleteness(ctx.rows);
if (!completenessResult.pass) {
  return { pass: false, errors: completenessResult.errors };
}
```

## 四、数据迁移

### 迁移脚本

**文件**：`scripts/migrate-req-prd.sh`

```bash
#!/bin/bash
set -e

echo "🔍 Scanning REQ-PRD usage..."
find specs/ -name "*.md" -exec grep -l "REQ-PRD" {} \;

echo ""
read -p "⚠️  Continue migration? (y/n) " -n 1 -r
echo
if [[ ! $REPLY =~ ^[Yy]$ ]]; then
    exit 1
fi

echo "📦 Backing up..."
cp -r specs/ specs.backup.$(date +%Y%m%d-%H%M%S)

echo "🔄 Migrating REQ-PRD → REQ..."
find specs/ -name "*.md" -type f -exec sed -i '' 's/REQ-PRD-/REQ-/g' {} \;

echo "✅ Migration complete"
echo "🔍 Verifying..."
if grep -r "REQ-PRD" specs/ 2>/dev/null; then
    echo "❌ Migration incomplete"
    exit 1
fi

echo "✅ All REQ-PRD migrated"
```

### 执行迁移

```bash
# 1. 运行迁移脚本
chmod +x scripts/migrate-req-prd.sh
./scripts/migrate-req-prd.sh

# 2. 验证所有矩阵
npm run validate:all

# 3. 提交变更
git add specs/
git commit -m "chore: migrate REQ-PRD to REQ"
```

#### F-04: validate format 校验闭环

**修改文件**：
- `src/core/validators/format-validator.ts` (L88-135)

**修改内容**：
1. 非法 ID 报错而非跳过
2. 添加 upstream/downstream 校验
3. 检测路径占位符（`/` 或 `*`）

```typescript
// 修改前
if (!validateId(id).valid) continue;  // ❌ 跳过

// 修改后
if (!validateId(id).valid) {
  errors.push(`非法 ID 格式：${id}`);
  continue;  // ✅ 报错
}

// 新增：校验 upstream/downstream
function validateRefList(refList: string, field: string, rowId: string, errors: string[]): void {
  // 检测路径占位符
  if (ref.includes('/') || ref.includes('*')) {
    errors.push(`${rowId} ${field} 包含非法引用：${ref}`);
  }
  // 校验 ID 格式
  const validation = validateId(ref);
  if (!validation.valid) {
    errors.push(`${rowId} ${field} 包含非法 ID：${ref}`);
  }
}
```

**影响**：
- ✅ 向后兼容（只增加校验）
- ✅ 数据完整性保障提升

## 💡 经验教训

### 1. 明确问题域

**教训**：架构问题和功能问题是**不同的领域**，应分别评估。

**建议**：
- 架构审查：关注设计合理性、健壮性
- 功能验证：关注功能完整性、正确性
- 明确区分，避免混淆

### 2. 避免概念混淆

**教训**："功能已实现"不等于"设计合理"。

**建议**：
- 使用精确表述（"规则有缺陷"而非"不完整"）
- 明确评估标准（"设计质量"而非"是否工作"）
- 区分"是什么"和"应该是"

### 3. 文档范围澄清

**教训**：文档应明确说明评估范围和局限性。

**建议**：
- 明确说明"评估内容"和"不涵盖内容"
- 添加引用（相关问题的参考文档）
- 避免误导性结论

## 一、核心洞察：数据完整性 vs 业务完整性

### 关键区分

**数据完整性**（Infrastructure Layer）：
- ID 格式合法性
- 引用存在性（引用的 ID 必须存在）
- 链路类型匹配（FR 的 upstream 只能是 REQ）
- **类比**：数据库的外键约束

**业务完整性**（Application Layer）：
- 孤儿检测（非根节点必须有 upstream）
- 覆盖率要求（FR 必须有 DS）
- **类比**：业务规则校验

### 为什么这样划分？

**场景对比**：

```typescript
// 数据完整性问题（基础设施层）
FR-AUTH-001 → upstream: ["REQ-PRD-001"]  // ❌ REQ-PRD 格式非法
FR-AUTH-002 → upstream: ["DS-AUTH-001"]  // ❌ FR 不能有 DS 作为 upstream

// 业务完整性问题（应用层）
FR-AUTH-003 → upstream: []               // ❌ 孤儿（业务规则）
FR-AUTH-004 → 没有 DS downstream         // ❌ 覆盖率不足（阶段要求）
```

**关键**：数据完整性问题会导致系统崩溃，必须在入口处拦截。

## 四、Infrastructure Layer（健壮版）

### parseMatrix（数据完整性保证）

```typescript
// src/core/trace-engine/matrix.ts
function parseMatrixContent(content: string): MatrixRow[] {
  const rows: MatrixRow[] = [];

  for (const cells of parseMarkdownTable(content)) {
    const id = cells[0];
    const upstream = parseRefList(cells[4]);
    const downstream = parseRefList(cells[5]);

    // 1. ID 格式校验
    const validation = validateId(id);
    if (!validation.valid) {
      throw new Error(`Invalid ID format: ${id} - ${validation.error}`);
    }

    // 2. 引用合法性校验
    for (const ref of [...upstream, ...downstream]) {
      const refValidation = validateId(ref);
      if (!refValidation.valid) {
        throw new Error(`Invalid reference: ${ref} in ${id}`);
      }
    }

    // 3. 链路合法性校验（关键：在这里做）
    const chainErrors = validateChain(id, upstream, downstream);
    if (chainErrors.length > 0) {
      throw new Error(`Chain validation failed for ${id}: ${chainErrors.join('; ')}`);
    }

    rows.push({
      id,
      type: validation.type!,
      title: cells[2],
      status: normalizeStatus(cells[3]),
      upstream,
      downstream,
    });
  }

  return rows;
}
```

**关键**：
- ✅ 在 parseMatrix 中做链路校验
- ✅ 任何坏数据都无法进入系统
- ✅ 不依赖人工纪律

### validateChain（双向校验）

```typescript
// src/core/trace-engine/id-validator.ts
export function validateChain(
  id: string,
  upstream: string[],
  downstream: string[]
): string[] {
  const validation = validateId(id);
  if (!validation.valid) return ['Invalid ID format'];

  const rule = ID_RULES[validation.type!];
  const errors: string[] = [];

  // 校验 upstream
  for (const upId of upstream) {
    const upValidation = validateId(upId);
    if (!upValidation.valid) {
      errors.push(`Invalid upstream: ${upId}`);
      continue;
    }
    if (!rule.upstream.includes(upValidation.type!)) {
      errors.push(`${validation.type} cannot have ${upValidation.type} as upstream`);
    }
  }

  // 校验 downstream
  for (const downId of downstream) {
    const downValidation = validateId(downId);
    if (!downValidation.valid) {
      errors.push(`Invalid downstream: ${downId}`);
      continue;
    }
    if (!rule.downstream.includes(downValidation.type!)) {
      errors.push(`${validation.type} cannot have ${downValidation.type} as downstream`);
    }
  }

  return errors;
}
```

## 五、Application Layer（业务完整性）

### checkMatrix（只做孤儿检测）

```typescript
// src/core/trace-engine/matrix.ts
export function checkMatrix(
  featureId: string,
  projectRoot: string
): MatrixCheckResult {
  // parseMatrix 已经保证了数据完整性
  const rows = parseMatrix(featureId, projectRoot);

  // 只做业务完整性检查：孤儿检测
  const orphans = rows.filter(r =>
    r.type !== 'Feature' && r.type !== 'REQ' && r.type !== 'RFC' &&
    (!r.upstream || r.upstream.length === 0)
  );

  return {
    total: rows.length,
    orphans,
    valid: orphans.length === 0,
  };
}
```

**关键**：
- ✅ 不再做链路校验（已在 parseMatrix 完成）
- ✅ 只做业务规则检查（孤儿检测）

```typescript
// src/core/gate-engine/gate-evaluator.ts
export function evaluateGate(
  featureId: string,
  stage: Stage,
  projectRoot: string
): GateResult {
  const rows = parseMatrix(featureId, projectRoot);  // 自动获得数据完整性保护
  const coverage = getCoverage(featureId, projectRoot);
  const matrixCheck = checkMatrix(featureId, projectRoot);

  const ctx: GateContext = {
    featureId,
    stage,
    projectRoot,
    rows,
    coverage,
  };

  const results: GateConditionResult[] = [];

  // 前置：业务完整性
  if (!matrixCheck.valid) {
    results.push({
      id: 'G-MATRIX',
      description: 'Matrix integrity',
      pass: false,
      detail: `Found ${matrixCheck.orphans.length} orphans`,
    });
  }

  // 阶段门禁
  const conditions = GATE_CONDITIONS[stage] || [];
  for (const cond of conditions) {
    results.push({
      id: cond.id,
      description: cond.description,
      ...cond.evaluate(ctx),
    });
  }

  return {
    pass: results.every(r => r.pass),
    results,
  };
}
```

## 九、为什么这样划分？

### 数据完整性 vs 业务完整性

**数据完整性**（Infrastructure Layer）：
- 保证数据结构正确
- 类比：数据库的外键约束
- 失败后果：系统崩溃
- 检查时机：入口处（parseMatrix）

**业务完整性**（Application Layer）：
- 保证业务规则满足
- 类比：业务逻辑校验
- 失败后果：业务不完整
- 检查时机：业务逻辑层（checkMatrix）

### 为什么 parseMatrix 做链路校验是合理的？

1. **链路校验是数据完整性约束**
   - FR 的 upstream 只能是 REQ（类型约束）
   - 引用的 ID 必须存在（外键约束）
   - 这不是"业务规则"，而是"数据结构约束"

2. **类比：数据库外键**
   ```sql
   CREATE TABLE orders (
     id INT PRIMARY KEY,
     user_id INT,
     FOREIGN KEY (user_id) REFERENCES users(id)  -- 在表定义时约束
   );
   ```
   - 外键约束在数据库层（基础设施层）
   - 不在应用层检查

3. **健壮性保证**
   - 坏数据无法进入系统
   - 不依赖人工纪律
   - 系统级保证

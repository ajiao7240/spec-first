## 第一部分：理想架构设计

### 1. 核心洞察

**数据完整性 vs 业务完整性**：

| 层次 | 职责 | 示例 |
|------|------|------|
| 数据完整性 | ID 格式、引用存在性、链路类型匹配 | FR 的 upstream 只能是 REQ |
| 业务完整性 | 孤儿检测、断链检测、覆盖率要求 | FR 必须有 DS 下游 |

### 2. DDD 分层架构

```
Domain Layer (规则定义)
  ↓
Infrastructure Layer (数据完整性)
  ↓
Application Layer (业务完整性)
  ↓
Presentation Layer (质量门禁)
```

### 3. 理想 API 设计

```typescript
// Infrastructure Layer
export function parseMatrix(content: string): ParsedMatrix {
  // 格式 + 引用存在性 + 链路类型校验
}

// Application Layer
export function checkMatrix(parsed: ParsedMatrix, stage?: string): CheckResult {
  // 孤儿检测 + 断链检测
}
```

**优势**：
- 职责单一
- 易于测试
- 可复用性高

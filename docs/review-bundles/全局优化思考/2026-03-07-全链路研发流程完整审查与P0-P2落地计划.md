# 2026-03-07 全链路研发流程完整审查与 P0/P1/P2 落地计划

- `skills/spec-first/*`

- `src/core/skill-runtime/*`

## 3. 总体判断

### 3.1 已经成立的能力

当前系统已经不是“只有文档没有执行”的流程壳子，以下能力已经真实存在：

1. **阶段推进有运行时真约束**
   - `stage-machine.ts` 已定义 8+2 阶段转换
   - `advance` / `gate` / `hard-gate` 已形成基本阻断链

2. **TDD 前置守卫已部分落地**
   - `hard-gate.ts` 在 `code` 前检查 in-progress TASK、TDD RED、plan approval
   - `tests/unit/hard-gate.test.ts` 已覆盖关键阻断路径

3. **Gate 不是空壳**
   - `gate-evaluator.ts` 已评估 `00_init` 到 `07_release` 的条件
   - C10/C11/coverage/rfc/analysis 已进入实际逻辑

4. **CLI 与 Runtime 已具备流程引擎雏形**
   - `dispatcher.ts` 已区分 runtime route 与 skill route
   - `feature.ts` 已把 `list/current/switch` 合并为命令族

### 3.2 尚未收口的核心问题

系统的主要问题不是“没有流程”，而是“流程真理源还没有完全收敛”。

## 4. 关键发现

### F1. 高优先级：规则真理源分散，违反 SDD 的单一真理源原则

**现象**

- `hard-gate.ts` 保存一套 stage 绑定
- `dispatcher.ts` 保存一套 layer 绑定
- `skills/spec-first/README.md` 保存一套 Skill 阶段表
- 各 `SKILL.md` 再各自解释一套边界

**风险**

- 文档、入口、阻断、验收可能逐步漂移
- 用户看到的流程与系统真正执行的流程不完全一致

**结论**

这是当前最核心的工程治理问题，应列为 `P0`。

### F2. 高优先级：Gate 仍反向依赖 Skill 文档，Docs-as-Logic 耦合过重

**现象**

- `gate-evaluator.ts` 的 C11 检查不只检查 constitution 与 design，还检查 `03-spec/04-design/08-code-review` 文档是否引用 `constitution-authority.md`

**风险**

- Skill 文档缺一行引用文字，就可能影响正式 Gate 判定
- 文档说明层与业务门禁层边界被打穿

**结论**

这不符合软件工程里的“规则在引擎，文档在接口层”原则，应列为 `P0`。

### F3. 高优先级：`09-test` 的阶段绑定与 TDD 叙述冲突

**现象**

- `09-test/SKILL.md` 的内容明显强调 TDD、测试前置、测试建模
- `hard-gate.ts` 当前却把 `test` 绑定在 `05_verify`

**风险**

- 用户在认知上会把 `test` 当成验证阶段才执行的动作
- 与 `code` 前的 RED/GREEN 实践形成冲突

**结论**

`test` 应明确前移为测试规格 / TC 建模入口，`verify` 只保留放行验收。

### F4. 中高优先级：`code-review` 与 `verify` 的 `completion` 边界未彻底切开

**现象**

- `dispatcher.ts` 允许 `code-review --layer <single|cross|completion>`
- `12-verify/SKILL.md` 又说明 completion 应由 verify 承担

**风险**

- 审查通过和阶段放行可能被混为一件事
- 证据链与责任边界不清晰

**结论**

`code-review` 应只保留实现阶段质量审查；`verify` 独占 completion。

### F5. 中优先级：TDD 与审批证据仍依赖 `findings.md` 文本协议

**现象**

- `hard-gate.ts` 通过文本块和关键字解析 `findings.md`
- `tests/unit/hard-gate.test.ts` 也围绕文本格式写断言

**风险**

- 容易受格式漂移影响
- 不利于做统计、审计、迁移、自动补全

**结论**

需要在 `P1` 引入结构化证据账本，但应保留对 `findings.md` 的兼容读取以平滑迁移。

### F6. 中优先级：CLI 命令族与 Skill 心智仍分裂

**现象**

- CLI 里 `feature list/current/switch` 已统一
- Skill 层仍保留 `17-feature-list/18-feature-switch/19-feature-current` 三个独立入口
- `catchup/status/plan` 也存在恢复、观察、决策职责交叉

**风险**

- 用户学习成本偏高
- 入口过多，且职责边界靠经验记忆

**结论**

这一问题不应在 `P0` 先动大手术，但应在 `P1` 作为产品化收敛重点处理。

## 5. 设计结论

推荐采用“**真理源收敛优先**”路线，而不是先改命令名或只修文档。

### 目标四层模型

1. `Spec/State` 真理层
2. `Engine` 执行层
3. `CLI/Runtime` 编排层
4. `Skill/Docs` 交互层

核心原则：

- **SDD**：规则先在 engine 可执行，再由 Skill 解释
- **TDD**：TDD 作为运行时前置条件，而不是文案建议
- **Single Source of Truth**：阶段、层、门禁、状态各自只有一个主定义位置
- **Docs as Interface, Not Logic**：文档不再承担正式业务门禁逻辑

## 6. P0 / P1 / P2 落地计划

### 6.1 P0：规则收口与语义止血

**优先级**

- 最高优先级

**目标**

- 先消除规则真理源分散
- 先修复最影响 SDD/TDD 一致性的冲突

**建议动作**

1. 建立统一治理注册表
   - 建议新增：`src/core/skill-runtime/skill-governance.ts`
   - 统一导出 stage binding、layer binding、command family

2. 收紧 `test` / `code-review` / `verify` 边界
   - `test` 前移为测试规格 / TC 建模入口
   - `code-review` 只保留 `single` / `cross`
   - `verify` 独占 `completion`

3. 清除 Gate 对 Skill 文档内容的正式依赖
   - `gate-evaluator.ts` 仅判定真正的治理数据和产物
   - 文档完整性移入 docs/catalog 测试

4. 清理 Skill 索引与阶段表的漂移项
   - `skills/spec-first/README.md`
   - 相关 `SKILL.md`

**影响面**

- `src/core/skill-runtime/*`
- `src/core/gate-engine/*`
- `skills/spec-first/08-code-review/SKILL.md`
- `skills/spec-first/09-test/SKILL.md`
- `skills/spec-first/12-verify/SKILL.md`
- `skills/spec-first/README.md`
- 单测：`hard-gate`、`skill-runtime`、`gate-evaluator`

**验证方式**

- `npx vitest run tests/unit/hard-gate.test.ts tests/unit/skill-runtime.test.ts tests/unit/gate-evaluator.test.ts`
- 抽样验证：
  - `code-review --layer completion` 被拒绝
  - `verify --layer completion` 正常
  - `test` 的阶段绑定与文档叙述一致

**完成标志**

- 运行时和文档中的阶段/层语义不再冲突
- Gate 不再把 Skill 文案当作正式放行条件

### 6.2 P1：证据结构化与命令面收敛

**优先级**

- 第二优先级

**目标**

- 让系统更可审计、更易维护
- 让用户视角的命令面和内部治理模型对齐

**建议动作**

1. 引入结构化证据账本
   - 建议新增：`specs/<featureId>/runtime/evidence.jsonl`
   - 记录 TDD RED/GREEN、plan approval、review result、verify result

2. 保留 `findings.md` 作为人类可读摘要
   - 机器判定优先读 ledger
   - `findings.md` 保留兼容与说明角色

3. 收敛命令族
   - `feature` 继续作为正式命令族
   - `17/18/19` 退化为薄包装或迁移提示
   - `catchup/status/plan` 按“恢复 / 观察 / 决策”重新定位

4. 收敛 Skill 说明层
   - Skill 只保留流程说明和模板
   - 删除重复阈值、重复阶段表、重复边界叙述

**影响面**

- `src/core/skill-runtime/hard-gate.ts`
- `src/core/process-engine/advance.ts`
- `src/cli/commands/feature.ts`
- `skills/spec-first/02-catchup/SKILL.md`
- `skills/spec-first/11-plan/SKILL.md`
- `skills/spec-first/14-status/SKILL.md`
- `skills/spec-first/17-feature-list/SKILL.md`
- `skills/spec-first/18-feature-switch/SKILL.md`
- `skills/spec-first/19-feature-current/SKILL.md`

**验证方式**

- `npx vitest run tests/unit/hard-gate.test.ts tests/unit/skill-runtime.test.ts tests/e2e/core-flow.test.ts`
- 抽样检查：
  - 有 ledger 时无需依赖 `findings.md` 文本格式
  - `feature list/current/switch` 成为主入口
  - `catchup/status/plan` 文案职责不再互相覆盖

**完成标志**

- 关键治理证据可结构化读取
- 用户学习命令的心智模型明显简化

### 6.3 P2：状态驱动编排与角色化闭环

**优先级**

- 第三优先级

**目标**

- 把 `orchestrate` 从“阶段线性跳转器”升级为“状态驱动调度器”
- 把角色入口与治理模型真正接起来

**建议动作**

1. `orchestrate` 基于 stage + gate + evidence 决定建议
2. 减少过时的硬编码技能映射
3. 强化 `onboarding / first / init` 的角色分工
4. 让控制层真正服务主链，而不是与主链竞争入口

**影响面**

- `src/core/skill-runtime/orchestrate-args.ts`
- `src/core/skill-runtime/dispatcher.ts`
- `skills/spec-first/13-orchestrate/*`
- 入口类 Skill 与相关文档

**验证方式**

- `npx vitest run tests/unit/skill-runtime.test.ts tests/e2e/core-flow.test.ts`
- 真实流程回放：
  - 新 Feature 启动
  - 进行中 Feature 接续
  - 实现阶段进入 review / verify
  - 收口阶段进入 archive

**完成标志**

- orchestrate 的推荐不再与实际治理规则冲突
- 角色入口、主链阶段、控制层之间边界稳定

## 7. 推荐实施顺序

推荐按以下顺序推进，不建议跳步：

1. `P0-1` 统一治理注册表
2. `P0-2` 收紧 `test` / `code-review` / `verify`
3. `P0-3` 把 Gate 从 Skill 文档逻辑中解耦
4. `P1-1` 引入结构化证据账本
5. `P1-2` 收敛 feature/context 命令面
6. `P2-1` 升级 orchestrate 为状态驱动

原因很直接：

- 不先收口真理源，后续所有命令合并都会只是“换入口，不换逻辑”
- 不先明确审查与放行边界，TDD 和 verify 永远会相互牵扯
- 不先把证据结构化，后续状态驱动编排会继续依赖脆弱文本协议

## 8. 审查结论

### 结论一句话

`spec-first` 当前已经具备真正的流程引擎骨架，但还没有完全达到“规范是唯一真理源、引擎负责执行、Skill 只做交互”的成熟状态。

### 最重要的三件事

1. 先收敛规则真理源
2. 再结构化证据与命令面
3. 最后升级编排与角色入口

### 当前建议

如果只做一轮最值钱的优化，应优先做 `P0`。  
如果要形成真正稳定的工程产品，应至少完成 `P0 + P1`。  
`P2` 是体验与规模化收益最大的一层，但必须建立在前两层已经收口的前提上。

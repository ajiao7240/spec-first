## ⚠️ 需要明确的关键问题

### 问题 1（仅 Path Y）：Reference 加载策略与全局约束的关系

**方案第 71-79 行**：
```markdown
### 2.3 Reference 读取按职责收敛

不要再让每个 subagent 都吃到 7-10 个 reference 文件。

更好的方式是：
- 主线程保留少量核心合同
- subagent 只加载自己的主题规范 + 输出 schema
- 公共规则只放一处，减少重复维护
```

**全局约束**（SKILL.md 第 61 行）：
```markdown
| 场景 | 必须加载 | 说明 |
|---|---|---|
| 所有执行 | 7 个 reference 文件 | 主线程与 wave 调度的共同契约 |
```

**问题**：
1. 这类“按职责减少 reference”的思路，只在 Path Y 才会与全局约束冲突
2. 当前 live docs 已经选择 Path X：保留 7 个全局 reference，只优化 Serena 与共享证据包
3. 因此，这里不再是当前实施阻塞，而是未来合同变更的决策点

**建议修正**：
- **方案 A（保守）**：明确说明"遵守全局约束，所有 subagent 仍然加载 7 个 reference 文件"
  - 优点：不破坏全局约束
  - 缺点：token 成本节省有限

- **方案 B（修改全局约束）**：明确说明"需要修改 SKILL.md 第 61 行的全局约束，将'所有执行必须加载'改为分层加载"
  - 优点：token 成本可以显著降低
  - 缺点：需要修改全局约束，影响面大

**推荐**：方案 A（保守），遵守全局约束，当前实施口径即 Path X。

**理由**：
1. 全局约束是 first skill 的核心契约，不应该为了性能优化而破坏
2. token 成本节省虽然有限，但执行时间节省仍然显著（Serena 替代 glob/grep/Read）
3. 如果要修改全局约束，应该作为独立的变更，而不是优化方案的一部分

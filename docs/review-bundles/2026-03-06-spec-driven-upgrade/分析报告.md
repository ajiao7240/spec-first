# Spec-First 对标 OpenSpec / Spec-Kit / Superpowers / Planning-with-Files / Trellis 的深度分析报告

> 日期：2026-03-06
> 范围：文档、源码、skills、hooks、模板、运行时机制
> 目标：判断 `spec-first` 还可以从哪些项目借鉴，才能把“需求→设计→编码→测试→交付”从人工驱动升级为规范驱动

---

## 1. 结论摘要

`spec-first` 目前已经不是“有没有规范驱动框架”的问题，而是“怎样把现有规范体系继续执行化、运行时化、交付化”的问题。

当前项目的强项非常明确：

- 已有完整阶段状态机、Gate、追踪矩阵、RFC/缺陷管理、catchup、Context Pack、HARD-GATE、模板哈希、manifest 迁移等能力。
- 相比五个对标项目，`spec-first` 在流程治理和追踪治理上明显更强，已经具备平台雏形。

当前项目的核心短板也很明确：

- 规范仍然主要驱动“人和 agent 的行为”，还没有完全驱动“运行时决策与产物生成”。
- 一些能力已经有“基础版”，但还没有形成强闭环，例如升级回滚、worktree 命令化、任务级上下文注入、变更依赖图、交付证据链。

一句话判断：

`spec-first` 已经完成了“规范化流程引擎”的大半，但距离“真正的规范驱动研发系统”，还需要把规范对象模型、agent 运行时和交付治理三条链路再向前推进一层。

## 2. 调研方法

本次分析不是只看 README，而是按三层做交叉验证：

1. 文档层：看项目对外宣称的理念和工作流。
2. 执行层：看 CLI、脚本、hook、runtime 是否真正实现这些理念。
3. 约束层：看 skill/template/prompt 是否能稳定约束 agent 行为。

主要核对材料包括：

- OpenSpec 的 `README.md`、`docs/concepts.md`、`src/commands/workflow/instructions.ts`、`src/core/artifact-graph/instruction-loader.ts`、`src/core/command-generation/registry.ts`
- Spec-Kit 的 `spec-driven.md`、`templates/commands/specify.md`、`templates/commands/plan.md`、`templates/commands/tasks.md`
- Superpowers 的 `README.md`、`skills/subagent-driven-development/SKILL.md`、`skills/test-driven-development/SKILL.md`
- Planning-with-Files 的 `README.md`、`skills/planning-with-files/SKILL.md`、`scripts/session-catchup.py`、`scripts/check-complete.sh`
- Trellis 的 `README.md`、`.trellis/workflow.md`、`.claude/hooks/session-start.py`、`.claude/hooks/inject-subagent-context.py`、`.claude/commands/trellis/parallel.md`、`src/utils/template-hash.ts`、`src/migrations/index.ts`、`src/commands/update.ts`

## 3. 五个项目真正值得借鉴的“硬能力”

### 3.1 OpenSpec：把“变更”做成一等对象

OpenSpec 最值得借鉴的不是 proposal/design/tasks 这些文档名，而是它把“change”当成一个明确的执行对象。

从代码上看，它的关键机制是：

- 用 `openspec/changes/<change>/` 承载完整变更生命周期，而不是让需求散落在对话里。
- 用 schema + artifact graph 决定“当前该生成什么”“缺什么依赖”“apply 是否 ready”。
- `instructions` 命令不是返回一段固定 prompt，而是动态返回 context、rules、template、dependencies、outputPath。
- 通过 command adapter registry 把同一套 workflow 映射到不同 AI 宿主。

这说明 OpenSpec 的强项是：

- 变更对象模型清晰
- artifact readiness 可计算
- 多宿主输出层有统一抽象

对 `spec-first` 的价值：

- 现在 `spec-first` 很强的是“阶段”，但相对弱的是“变更对象模型”。
- 如果要真正做到规范驱动，`Feature` 之外还需要一个更强的 `Change/Artifact Graph` 层。
- 当前 `Context Pack` 更像上下文摘要，不像 OpenSpec 那样能精确说明“下一个产物是什么、为何可生成、依赖是否满足”。

### 3.2 Spec-Kit：用模板强行塑造高质量规格

Spec-Kit 最强的地方不是 CLI，而是模板本身就是行为约束器。

从模板可以看出：

- `specify.md` 强制 WHAT / WHY 与 HOW 分离。
- 对 `[NEEDS CLARIFICATION]` 有数量上限、优先级、回填流程。
- `plan.md` 强制先 research，再 data-model / contracts / quickstart。
- `tasks.md` 强制按 User Story 组织任务，而不是按技术层组织任务。

这意味着 Spec-Kit 的核心价值是：

- 模板不是“示例”，而是“对 LLM 的结构化约束”
- 规格质量、澄清质量、任务拆解质量都被模板化
- 从规范到设计到任务的转换路径很清晰

对 `spec-first` 的价值：

- 这套思想 `spec-first` 已经吸收了不少，但还可以更彻底。
- 现在 `spec-first` 的很多能力存在于 skill 描述里，下一步应更多下沉为模板级、结构级、状态机级约束。
- 尤其是 story-oriented task generation 和 clarification budget，仍有继续产品化空间。

### 3.3 Superpowers：把“不要偷懒”写成制度

Superpowers 的价值不是功能多，而是它几乎把所有常见 agent 合理化都堵死了。

从 skill 可以确认：

- TDD 有铁律：没有 failing test 就不许写 production code。
- subagent-driven-development 强制“实现 → spec review → code quality review”的两阶段审查顺序。
- planning / review / finish 都有强流程，而不是建议。

它真正提供的是：

- 行为纪律
- 质量守门
- 流程不可跳过性

对 `spec-first` 的价值：

- `spec-first` 已经集成了大量这类思想，尤其是 hard-gate、反合理化、verify 证据链。
- 但要继续避免“能力写进 skill，执行却仍然靠 agent 自觉”的问题。
- 下一步要把更多行为守卫变成 runtime contract，而不只是 prompt contract。

### 3.4 Planning-with-Files：把上下文丢失问题产品化

Planning-with-Files 的核心创新非常简单，但非常硬：

- 用 `task_plan.md`、`findings.md`、`progress.md` 把上下文放到磁盘上。
- 用 SessionStart hook 自动执行 catchup。
- 用 Stop hook 做完成度守门。
- 用 5-question reboot test 判断恢复质量。

代码上最关键的是两点：

- `session-catchup.py` 真正在解析上一轮会话，找 planning files 最后一次更新之后的“未同步上下文”。
- `check-complete.sh` 真正在会话结束时读取计划文件并输出完成状态。

它真正解决的是：

- AI 会话有失忆性
- 重要信息必须从上下文窗口搬到文件系统

对 `spec-first` 的价值：

- `spec-first` 已经借鉴得比较深入，catchup、task-context、stop-guard 都在代码里。
- 但当前仍偏“提示式同步”，还可以继续强化为“状态式同步”和“证据式同步”。
- 比如 findings / progress / task_plan 的职责边界、自动更新策略、恢复冲突判定，还可更严格。

### 3.5 Trellis：把规范注入、并发执行和升级治理做成工程系统

Trellis 是这次调研里最值得重视的一个，因为它补的是 `spec-first` 目前最关键的一类短板：运行时和升级治理。

从代码和 hook 可以确认，Trellis 的强项不只是理念，而是真落在实现里：

- SessionStart hook 自动注入 workflow、guidelines、workspace context
- PreToolUse hook 在 subagent 调度前注入 task-specific jsonl context
- `.trellis/tasks/`、`.trellis/workspace/`、`.trellis/spec/` 分层明确
- `template-hash.ts` 做模板哈希追踪
- `migrations/index.ts` 做版本到版本的 manifest 迁移装配
- `update.ts` 按模板变更、迁移、平台配置做升级治理
- `parallel.md` 明确主仓 orchestrator 与 worktree worker 分离

它真正解决的是：

- 规范如何稳定注入，而不依赖 agent 记忆
- 多 agent 如何在物理隔离下运行
- 模板和宿主配置如何安全升级

对 `spec-first` 的价值最大，主要有五点：

1. 规范分层
2. 任务级上下文注入
3. worktree 多 agent 编排
4. 模板升级安全链路
5. 宿主配置统一注册表

## 4. 对 `spec-first` 当前状态的判断

### 4.1 已经具备的强项

从源码和 hooks 看，`spec-first` 已经具备以下核心能力：

- 阶段状态机与阶段守门：不是只有文档，而是有 `stage-state`、gate、hard-gate、matrix、traceability
- 会话恢复：`catchup.ts` 已经把 stage/task/findings/todo/auto-loop 整合在一起
- 上下文压缩：`context-pack.ts` 已有 control zone + references + slicing
- 行为守卫：`hard-gate.ts` 已经把 stage、TDD、plan approval、worktree 风险接进运行时判断
- 宿主注入：`session-hook.ts`、`ai-runtime-hook.ts` 已经在注册 SessionStart / PreToolUse / PostToolUse / Stop
- 升级治理基础：`hash-registry.ts`、`manifest-engine.ts`、`update.ts` 已经吸收 Trellis 的基础思路

所以不能低估当前项目。它已经不是一个“prompt 工具包”，而是具备 runtime 特征的流程系统。

### 4.2 真正的缺口

尽管能力很多，但从“规范驱动”目标反推，当前缺口主要有四类。

#### 缺口 A：规范对象模型还不够执行化

`spec-first` 当前的主轴是 `Feature -> Stage -> Matrix`。

这套模型很适合治理，但还不够适合“执行驱动”。相比之下：

- OpenSpec 有更强的 artifact graph
- Spec-Kit 有更强的 spec → plan → task 结构化转换

当前问题是：

- 很多运行时判断还是围绕文件是否存在、阶段是否推进
- 还没有把“哪些 artifact 已 ready、哪些 contract 已满足、哪些 change 影响哪些 downstream”做成统一对象模型

#### 缺口 B：上下文注入还不够 agent-specific

`spec-first` 已有 Context Pack 和 TaskContextPack，但仍偏“摘要型”。

相比 Trellis：

- Trellis 是 implement/check/debug/research 各自拥有独立 JSONL 上下文文件
- 注入发生在 subagent 调用前，而且和任务目录绑定

当前 `spec-first` 还缺：

- agent-specific context contract
- task workspace 级产物目录
- review/debug/research 的上下文分道

#### 缺口 C：升级治理已有基础，但还没形成“安全链路”

`spec-first update` 已经做了：

- 模板哈希比对
- manifest 迁移
- 宿主刷新

但还缺最关键的两个动作：

- 更新前快照备份
- 回滚演练与审计闭环

这意味着当前升级治理更像“能升级”，还不是“可安全升级”。

#### 缺口 D：交付链路还不够规范驱动

现在 `spec-first` 很强的是需求、设计、编码、验证治理。

但从“测试到交付”这一段看，还存在三件事没有完全打通：

- 测试证据如何由规范自动汇总
- 交付门禁如何直接消费规范与验证产物
- release / archive / deploy 的证据链如何沉淀为可回放资产

也就是说，它现在更接近“规范辅助驱动研发”，还没完全做到“规范直接驱动交付”。

## 5. 最值得借鉴、且应优先落地的完善点

### P0：把现有“基础版能力”补成闭环

这些能力 `spec-first` 已经有半套或基础版，收益最高，优先级也最高。

1. 更新前快照备份 + 一键回滚
   - 来源：Trellis
   - 价值：把 update/migration 从“可执行”升级为“可恢复”

2. worktree 命令化
   - 来源：Trellis + Superpowers
   - 价值：把高风险任务的隔离执行从“建议”变成“默认入口”

3. 冲突策略与迁移审计输出
   - 来源：Trellis
   - 价值：让 manifest migration 不再只是 Skip，而是有可解释性与可追踪性

4. Stop/Verify/Release 证据闭环
   - 来源：Planning-with-Files + Superpowers
   - 价值：结束会话、推进阶段、准备交付时都要看到新鲜证据，而不是状态推断

### P1：把规范从“阶段输入”升级为“执行对象”

1. 引入 Artifact Graph / Change Graph
   - 来源：OpenSpec
   - 价值：可计算 readiness、依赖、阻塞、影响范围

2. 引入 dependsOn / provides 语义
   - 来源：OpenSpec
   - 价值：不只是追踪“谁对应谁”，而是追踪“谁生产什么能力，谁消费什么能力”

3. 引入 contract-first task derivation
   - 来源：Spec-Kit
   - 价值：task 不是人工拆分结果，而是由 spec/design/contracts/data-model 推导

### P1：把 agent runtime 做细

1. implement/check/debug/research 独立上下文注入
   - 来源：Trellis
   - 价值：减少上下文污染，提高多 agent 可控性

2. 任务目录化运行时
   - 来源：Trellis
   - 价值：每个任务拥有自己的 PRD、context、log、review output

3. context cost 报告
   - 来源：Trellis
   - 价值：让上下文治理从经验变成可测量对象

### P2：把“编码驱动测试”继续升级为“规范驱动测试与交付”

1. 从 spec / scenario 自动生成验证视图
2. 把 verify 结果沉淀为 release evidence pack
3. 把 archive/release 阶段做成基于证据的 gate，而不是状态推进的附属动作

## 6. 推荐的演进路线

### 阶段一：补齐工程治理闭环

目标：先把现有基础版能力补完整。

建议动作：

- `update` 前自动生成 `.spec-first/.backup-*`
- 增加 `spec-first rollback` 或 `spec-first update --rollback <snapshot>`
- 增加 `spec-first worktree create/status/cleanup`
- 增强 migration 冲突策略、执行日志和失败恢复说明

### 阶段二：补齐任务级运行时

目标：把“当前 TASK”变成真正的一等运行时对象。

建议动作：

- 为 implement/check/debug/research 建立独立 context schema
- 在 `specs/<feature>/` 之外，引入更强的 task runtime 目录
- 将 review 输出、debug 记录、evidence pack 与 task 绑定

### 阶段三：补齐规范对象模型

目标：把规范从静态文档推进为可计算对象图。

建议动作：

- 为 FR/DS/TASK/TC 之外增加 artifact/change dependency graph
- 引入 `provides` / `dependsOn` / `requires` / `blocks` 等语义
- 让 `orchestrate`、`verify`、`release` 直接消费这些语义

### 阶段四：补齐交付驱动

目标：从“规范约束开发”进化到“规范驱动交付”。

建议动作：

- 生成 release evidence pack
- 将 gate / metrics / tests / defects / RFC 汇总为交付判断对象
- 在 release 阶段沉淀部署前检查、回滚说明、风险摘要

## 7. 如何从“规格准确”推进到“自动开发闭环”

上面的分析说明了 `spec-first` 该补哪些能力，但还没有把“当 PRD、技术方案、任务拆解已经足够准确之后，系统如何自动完成后续研发动作”讲透。

要做到这一点，关键不是再增加一层 prompt，而是补一个真正可执行的闭环：

- 上游把 PRD、设计、任务变成可计算约束
- 中游把约束转成 TDD 执行单元
- 下游把测试证据、改动范围、风险摘要转成提测资产

这意味着 `spec-first` 需要从现在的 `Feature -> Stage -> Matrix` 继续演进为：

`Feature -> Change Graph -> Execution Plan -> TDD Runtime -> Evidence Pack -> QA Submission Pack`

其中最重要的不是名称，而是每一层都要有清晰的输入、输出和 gate。

### 7.1 自动化闭环成立的前提

只有在以下前提满足时，系统才应该进入自动研发模式：

- PRD 已经完成核心澄清，关键需求不存在高优先级 `[NEEDS CLARIFICATION]`
- 技术方案已经明确边界、契约、数据模型、非功能约束、风险与回滚思路
- 开发任务已经不是“泛泛待办”，而是具备依赖、影响面、验收标准、测试义务的执行单元
- 追踪矩阵已经能把 `FR / NFR / Scenario / Contract / Task / Test Obligation` 串起来

如果这些前提不成立，系统不应该继续自动编码，而应该在 spec/design/task gate 直接阻断。

### 7.2 需要新增的判断标准

“规格准确”不能只靠人工主观判断，至少要有一组可检查规则：

- 完整性：每个 FR 都有对应场景、边界条件、异常路径、非目标
- 可设计性：每个需求都能落到具体 contract、data model、状态变迁或界面行为
- 可编码性：每个任务都能回答“修改哪些模块、引入什么测试、怎样判定 done”
- 可验证性：每个任务都能生成明确的 test obligations，而不是事后再补测试
- 可交付性：每个 change 都能说明对外行为变化、风险点、兼容性与回归范围

这套判断标准应该从文档审查规则继续下沉为结构化字段和 gate 规则。

## 8. 自动 TDD 开发流水线

真正的自动 TDD 不是“让 agent 自己去写测试和代码”，而是先把任务转换成严格的执行单元，再进入受限循环。

### 8.1 推荐的流水线形态

建议把自动开发拆成 7 个离散步骤：

1. `derive-execution-plan`
   - 从 PRD、设计、任务、追踪矩阵生成 `execution-plan.json`
   - 明确 change scope、影响模块、依赖关系、测试义务、风险标签

2. `derive-test-obligations`
   - 为每个任务生成测试义务，而不是直接生成测试代码
   - 输出单元测试、契约测试、集成测试、回归测试、冒烟测试的 required set

3. `select-runtime-strategy`
   - 根据风险等级、改动范围、是否跨模块、是否改 schema / contract 决定执行模式
   - 低风险可单 agent 自动执行
   - 高风险默认切到 worktree + reviewer/checker 隔离模式

4. `generate-failing-tests`
   - 先写 failing tests
   - gate 要求至少一个新失败测试或更新后的失败断言出现，否则不得进入 production code

5. `implement-to-green`
   - 实现代码直到当前任务的 required tests 转绿
   - 期间只能修改 execution plan 允许的 change scope，超范围改动直接触发阻断或升级评审

6. `run-self-verification`
   - 不只运行新测试，还要按影响范围运行回归、自测、lint、typecheck、静态检查
   - 生成结构化 evidence

7. `assemble-qa-submission-pack`
   - 汇总改动说明、影响范围、测试证据、风险与回滚方案
   - 自动产出提测报告

### 8.2 每个任务需要的最小执行对象

当前 `todo-state.json` 已经有任务状态与依赖，但要支撑自动 TDD，还不够。每个任务至少需要扩展为下面这些字段：

- `taskId`
- `goal`
- `dependsOn`
- `changeScope`
- `acceptanceCriteria`
- `testObligations`
- `allowedPaths`
- `blockedPaths`
- `riskLevel`
- `runtimeMode`
- `reviewMode`
- `evidenceRequired`

换句话说，任务不应只是标题和依赖，而应是一个可执行 contract。

### 8.3 为什么必须先生成 test obligations

很多自动编码失败，本质上不是模型写不出代码，而是任务拆解没有把“验证责任”提前定义。

建议 `spec-first` 不直接从任务跳到“写测试”，而是先落一层：

- 任务需要证明什么行为
- 哪类测试最适合证明这个行为
- 哪些旧测试必须回归
- 哪些不该由当前任务承担

这样做的好处是：

- TDD 不再依赖 agent 的临场判断
- verify 阶段可以对照 obligation 检查“有没有漏测”
- 提测报告可以直接引用 obligation 完成度

### 8.4 自动 TDD 的 gate 设计

建议新增四道硬 gate：

1. `task-ready gate`
   - 没有 `changeScope`、`testObligations`、`acceptanceCriteria` 的任务不得进入自动开发

2. `red gate`
   - 没有 failing test 证据，不得开始实现

3. `green gate`
   - 任务 required tests 不全绿，不得标记任务 done

4. `scope gate`
   - 修改文件超出 `allowedPaths` 或引入未声明 contract 变更，必须升级为人工确认或 reviewer 子代理复核

## 9. 自动自测与证据归档

如果自动编码是第一层闭环，那么自动自测就是第二层闭环。没有结构化自测证据，所谓“自动完成”只会停在代码生成。

### 9.1 自测不应只是一串命令

建议把自测拆成四类证据：

- 静态证据：lint、typecheck、schema 校验、contract diff、禁止项扫描
- 行为证据：单元测试、契约测试、集成测试、关键业务场景测试
- 范围证据：本次 change 影响了哪些模块、接口、页面、脚本、配置
- 风险证据：是否涉及迁移、数据修改、权限、缓存、异步任务、外部依赖

这四类证据都应该写成结构化产物，而不只是控制台输出。

### 9.2 建议新增的 evidence 产物

建议在 `specs/<feature>/runtime/` 或 task runtime 目录下新增：

- `execution-plan.json`
- `test-obligations.json`
- `change-scope.json`
- `verification-evidence.json`
- `risk-summary.json`
- `qa-submission-report.md`
- `qa-submission-report.json`

其中：

- `verification-evidence.json` 记录测试命令、结果、时间戳、失败重试、最终状态
- `change-scope.json` 记录声明影响面与实际 diff 影响面的对比
- `risk-summary.json` 汇总高风险标签、是否需灰度、是否需回滚预案

### 9.3 自测执行策略

建议 verify runtime 不使用“一刀切全量测试”，而采用分层策略：

- 任务级：运行当前任务 obligations 对应的最小测试集合
- 变更级：运行当前 change scope 涉及的模块回归
- 特征级：运行 feature 关键路径验证
- 交付级：在生成提测包前再跑一次最小可信冒烟集

这样才能兼顾速度和可信度。

### 9.4 自动失败处理

一旦自测失败，系统不能只把任务打回 `blocked`，还要给出明确失败类型：

- 测试生成失败
- 红灯未建立
- 实现未转绿
- 回归失败
- 范围越界
- 契约破坏
- 风险超阈值

这些失败类型应该进入审计日志和 evidence pack，后续才能支持复盘和 retry policy。

## 10. 改动范围提测报告生成机制

提测报告不应该在研发结束后由人手写，它应该是 runtime 的直接产物。

### 10.1 提测报告的核心目标

提测报告至少要回答 QA 最关心的五个问题：

1. 本次到底改了什么
2. 哪些地方可能被影响
3. 开发已经自测了什么
4. 哪些地方风险最高
5. 如果出问题，怎么快速回滚或定位

如果不能自动回答这五个问题，就说明“规范驱动交付”还没有完成。

### 10.2 报告生成的数据来源

提测报告不应依赖人工总结，而应自动汇总以下来源：

- PRD 中的业务目标、范围、非目标
- 技术方案中的契约、数据模型、风险与兼容性说明
- 任务图中的 `changeScope`、`allowedPaths`、`testObligations`
- 实际 diff、提交文件、生成/修改的 artifact
- 自测 evidence、失败记录、剩余风险、回滚说明

也就是说，提测报告本质上是一个聚合视图，而不是新文档。

### 10.3 提测报告建议结构

建议默认产出一份机器可读 JSON 和一份给 QA/测试负责人阅读的 Markdown：

- 变更摘要
- 需求映射
- 改动文件与模块范围
- 接口 / 数据 / 配置 / 页面影响面
- 已执行自测项与结果
- 未覆盖项与已知风险
- 回归建议
- 发布注意事项
- 回滚说明

其中“改动范围”建议分成两层：

- 声明范围：设计和任务中预计影响的范围
- 实际范围：基于 git diff、contract diff、schema diff 得到的真实范围

如果两者偏差过大，提测报告应自动标红。

### 10.4 如何自动识别改动范围

推荐采用“声明范围 + 静态推导 + 实际 diff”三路合并：

- 声明范围来自 `changeScope`
- 静态推导来自依赖图、接口引用、模块映射、路由映射、schema 引用
- 实际 diff 来自文件修改、符号引用变化、contract/schema 差异

最终得到：

- `changedFiles`
- `changedModules`
- `changedContracts`
- `changedSchemas`
- `affectedScenarios`
- `requiredRegressionAreas`

这比单纯列文件更接近 QA 真正需要的信息。

## 11. 为了实现这条闭环，`spec-first` 需要新增哪些核心能力

### 11.1 从 matrix 升级到 execution graph

当前 traceability matrix 很适合追踪，但还不够适合自动执行。

建议新增 `execution graph` 层，把这些关系显式化：

- `feature -> change`
- `change -> artifact`
- `artifact -> contract`
- `contract -> task`
- `task -> testObligation`
- `task -> allowedPaths`
- `task -> evidence`

这样 `orchestrate`、`code`、`verify`、`release` 才能围绕同一个对象模型协同。

### 11.2 新增 task runtime 目录

建议每个任务拥有独立 runtime 空间，例如：

- `specs/<feature>/tasks/<task-id>/context/`
- `specs/<feature>/tasks/<task-id>/runtime/`
- `specs/<feature>/tasks/<task-id>/evidence/`
- `specs/<feature>/tasks/<task-id>/review/`

这样做有三个好处：

- 多 agent 并发时，文档和证据不会混在 feature 根目录
- implement/check/debug/research 可以各自写入独立产物
- 提测报告可以从 task 粒度向上聚合

### 11.3 新增 execution-plan 派生器

建议新增一个专门的派生命令，例如：

- `spec-first derive execution-plan`
- `spec-first derive test-obligations`
- `spec-first derive qa-pack`

这些命令的职责是把静态规范转成运行时对象，而不是直接调 agent 写代码。

### 11.4 新增自动 TDD runtime

现有 `auto-loop` 和 `todo-runner` 已经提供了雏形，但要进一步升级为：

- 能识别 red/green/refactor 三段状态
- 能记录每次失败测试和修复迭代
- 能限制修改范围
- 能在任务 done 前自动拉起 checker/reviewer

也就是说，`auto-loop` 需要从“任务推进器”进化为“受控 TDD 执行器”。

### 11.5 新增 QA Submission Pack

建议把 release 之前的提测产物收敛成标准包：

- `qa-submission-report.md`
- `qa-submission-report.json`
- `verification-evidence.json`
- `risk-summary.json`
- `rollback-notes.md`

后续不论是给测试、给发布负责人、还是给 archive 复盘，都消费同一套证据包。

## 12. 分阶段落地路线图

### P0：先补可执行闭环，不追求完全自动

目标：先做到“规格准确后，可以半自动、强约束地完成 TDD、自测、提测输出”。

建议动作：

- 给任务模型补 `changeScope / acceptanceCriteria / testObligations / allowedPaths`
- 增加 `derive execution-plan` 与 `derive test-obligations`
- 扩展 `auto-loop` 记录 red/green/refactor 状态
- 生成基础版 `verification-evidence.json` 和 `qa-submission-report.md`

### P1：再补 agent 隔离与范围治理

目标：把自动开发从“能跑”升级为“可控、可审计、可回放”。

建议动作：

- 默认对高风险任务启用 worktree 隔离
- implement/check/debug/research 使用 task-specific context 注入
- 增加 `scope gate` 与 contract/schema diff 检查
- 引入 `change-scope.json` 和 `risk-summary.json`

### P2：最后补真正的规范驱动交付

目标：让 verify / release / archive 都直接消费执行证据，而不是依赖人工总结。

建议动作：

- release 阶段直接读取 QA Submission Pack
- archive 阶段自动归档任务证据、失败模式、回滚记录、风险复盘
- 建立“需求准确率 -> 任务偏差率 -> 测试补写率 -> 提测通过率”的闭环指标

只有走到这一步，`spec-first` 才算真正完成从规范治理到规范驱动交付的升级。

## 13. 最终判断

如果只看理念，`spec-first` 与这五个项目已经非常接近。

如果看代码落地，`spec-first` 目前最明显的进步是：已经把很多外部项目的优点吸收进来了，而且不是停留在文档口号层面。

但如果看“是否真正完成从人工驱动到规范驱动的升级”，答案是：

**还差最后一层。**

这最后一层不是继续写更多 skill，也不是继续补更多流程文档，而是把规范变成：

- 可计算的对象模型
- 可注入的运行时上下文
- 可审计的升级链路
- 可判定的交付证据

只有这样，`spec-first` 才会从“规范治理引擎”真正升级为“规范驱动研发系统”。

---

## 14. 优先级建议

按价值 / 成本比排序，建议优先级如下：

1. `update` 快照备份 + 回滚
2. worktree 命令化 + 高风险任务默认隔离
3. task runtime 目录 + agent-specific context 注入
4. artifact/change dependency graph
5. release evidence pack + 交付 gate

最不建议优先做的事情：

- 继续扩充 skill 文案但不补 runtime
- 继续增加阶段规则但不补交付证据链
- 继续做多宿主适配但不补统一运行时抽象

因为这些会继续增强“规则感”，但不会显著提升“规范驱动程度”。

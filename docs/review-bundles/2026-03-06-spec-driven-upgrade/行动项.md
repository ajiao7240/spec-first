# Spec-Driven Upgrade Action Items

## P0

- 建立 `execution-plan.json` 与 `test-obligations.json` 派生能力，命令入口放到 `spec-first derive`
- 扩展 `src/core/ai-orchestrator/todo-runner.ts` 的任务结构，使任务具备 `changeScope`、`acceptanceCriteria`、`testObligations`、`allowedPaths`
- 扩展 `src/core/ai-orchestrator/auto-loop.ts`，增加 `red gate`、`green gate`、`scope gate`
- 生成基础版 `verification-evidence.json`

## P1

- 为 `implement/check/debug/research` 增加 task-specific context 注入
- 增加 `specs/<feature>/tasks/<task-id>/context|runtime|evidence` 目录规范
- 生成 `change-scope.json` 与 `risk-summary.json`
- 让 `hard-gate` 对高风险任务默认要求 worktree 或 reviewer 确认

## P2

- 生成 `qa-submission-report.json` 与 `qa-submission-report.md`
- 让 05_verify 直接消费 evidence pack
- 让 07_release 直接消费 QA Submission Pack
- 在 `analyze` / `validate` 中增加 execution-plan、evidence、QA pack 一致性校验

## Dependency Order

1. 先做 execution-plan 派生，否则后续 gate 没有稳定输入
2. 再做任务 contract 扩展，否则 auto-loop 无法受控
3. 再做 task runtime，否则多 agent 上下文会继续混在 feature 级
4. 再做 evidence / QA pack，否则 verify 仍然只能读控制台输出
5. 最后做 CLI 汇总和 e2e 回归，否则缺少闭环验证

## Review Checkpoints

- Checkpoint 1: `derive execution-plan` 命令与单测通过
- Checkpoint 2: `TodoItem` contract 扩展与 `auto-loop` red gate 生效
- Checkpoint 3: task runtime 与 agent-specific context 注入稳定
- Checkpoint 4: verify / release 能消费结构化 evidence
- Checkpoint 5: QA submission report 可被测试团队直接使用

## Done Definition

- 任一 feature 在 03_plan 结束后可以生成 `execution-plan.json`
- 任一实现任务在进入 04_implement 前都能生成 `test-obligations.json`
- 自动编排不能跳过 failing test 直接写实现
- verify 报告包含结构化证据而不是纯人工总结
- 交付前自动产出改动范围提测报告

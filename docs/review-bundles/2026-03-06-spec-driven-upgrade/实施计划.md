# Spec-Driven TDD Automation Implementation Plan

> **For Claude:** REQUIRED SUB-SKILL: Use `superpowers:executing-plans` to implement this plan task-by-task.

**Goal:** 在 `spec-first` 内部补齐“规格准确 -> 自动 TDD 开发 -> 自动自测 -> 自动提测报告”的运行时闭环。

**Architecture:** 以现有 `Feature -> Stage -> Matrix` 为治理骨架，新增 `Execution Plan / Test Obligations / Evidence Pack / QA Submission Pack` 四层运行时对象。CLI 负责派生与汇总，`auto-loop` 负责受控执行，`verify` 与 `release` 负责消费证据而不是重新猜测状态。

**Tech Stack:** TypeScript, Node.js, Vitest, existing `spec-first` CLI/runtime modules

---

## Scope

本计划不追求一次性做完“完全无人值守开发”，只做三件事：

- 让任务从待办项升级为可执行 contract
- 让自动编排从任务推进器升级为受控 TDD 执行器
- 让 verify / release 直接读取结构化证据和提测包

不在本轮内做的内容：

- 新宿主平台适配
- 大规模 UI 改造
- 复杂静态分析平台或语言服务集成

## Target Runtime Artifacts

新增或标准化以下产物：

- `specs/<feature>/runtime/execution-plan.json`
- `specs/<feature>/runtime/test-obligations.json`
- `specs/<feature>/runtime/change-scope.json`
- `specs/<feature>/runtime/verification-evidence.json`
- `specs/<feature>/runtime/risk-summary.json`
- `specs/<feature>/runtime/qa-submission-report.json`
- `specs/<feature>/qa-submission-report.md`
- `specs/<feature>/tasks/<task-id>/context/*.json`
- `specs/<feature>/tasks/<task-id>/evidence/*.json`

## Acceptance Criteria

- `spec-first derive execution-plan <featureId>` 可生成运行时执行计划
- `TodoItem` 不再是纯标题待办，而是具备 `changeScope`、`testObligations`、`allowedPaths`
- `auto-loop` 能阻止“无 red 证据直接写实现”
- `verify` 可读取结构化 evidence，而不是只消费终端输出
- `release` 可读取 QA Submission Pack，而不是依赖人工总结
- QA 能从 `qa-submission-report.md` 直接看到改动范围、已测范围、风险和回滚说明

## Rollout Order

1. 先落 Task 1，确保有 execution plan 与 obligations
2. 再落 Task 2，避免 auto-loop 无约束运行
3. 接着做 Task 3，保证多 agent / task runtime 不串上下文
4. 再做 Task 4，让 verify / release 真正消费证据
5. 最后做 Task 5，把 CLI、验证和 e2e 打通

## Risks

- 如果 `task_plan.md` 结构仍然过于自由，execution plan 派生会不稳定
- 如果 diff 范围识别只依赖文件路径，QA report 会过于粗糙
- 如果 05_verify 仍允许跳过 evidence freshness，自动提测报告会失真
- 如果 worktree 仍然只是建议，高风险任务的自动开发会继续污染主工作区

## Recommended First Slice

最小可交付切片建议是：

- 先完成 `derive execution-plan`
- 再给 `TodoItem` 增加 `testObligations`
- 然后让 `auto-loop` 支持 red gate
- 最后生成基础版 `verification-evidence.json`

这样可以在不重写全部 verify/release 的前提下，先建立最关键的执行闭环。

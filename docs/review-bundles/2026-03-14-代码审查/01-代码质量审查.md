# 代码质量审查（校正版）

## 结论

本次只保留已被代码直接证明的问题。上一版中的部分结论方向正确，但存在严重程度偏高、文件定位不准或描述过度的问题，已移除。

## 已验证问题

### CQ-001 [High] batch executor 仍是占位实现

位置：`src/core/batch-executor/concurrent-executor.ts`

证据：

```ts
// TODO: 调用 Agent tool
await new Promise((resolve) => setTimeout(resolve, 100));
return { taskId: task.id, success: true, message: '执行成功' };
```

影响：并发执行链路会给出“成功”结果，但没有真实实现任务执行。

### CQ-002 [High] `saveTodoState()` 无并发锁

位置：`src/core/ai-orchestrator/todo-runner.ts`

现状：采用 `writeFileSync(tmp) + renameSync()` 做原子替换，但未使用仓库已有的 `src/shared/file-lock.ts`。

影响：多进程同时写 `todo-state.json` 时，仍可能发生最后写入覆盖。

### CQ-003 [Medium] `stop_on_blocked` 处理重复

位置：`src/core/ai-orchestrator/auto-loop.ts`

现状：失败分支和 post-write guard 分支各自内联了一套近似相同的：

- `cascadeBlocked()`
- `writeAuditLog()`
- `haltState()`
- `checkpoint()`

影响：后续修改容易只改一处。

### CQ-004 [Medium] 审计日志写入存在线性退化

位置：`src/core/ai-orchestrator/audit-log.ts`

现状：每次 `writeAuditLog()` 都先 `readFileSync()` 全文件，再解析最后一行取 `prevHash`。

影响：日志越大，单次写入越慢。

### CQ-005 [Medium] 完成检测与 slop 检测都允许配置正则直编译

位置：

- `src/core/ai-orchestrator/completion-detector.ts`
- `src/core/ai-orchestrator/slop-checker.ts`

现状：两处都对配置输入执行 `new RegExp(...)`，无安全过滤或复杂度限制。

影响：灾难性回溯模式会拖慢或阻塞主流程。

### CQ-006 [Medium] `loadTodoState()` 默认宽松读取

位置：`src/core/ai-orchestrator/todo-runner.ts`

现状：`strict` 不是默认值，默认路径下损坏状态文件会返回 `undefined` 而不是抛错。

影响：定位状态损坏时，错误信号偏弱。

## 纠正项

- “batch-executor 无任何测试”不成立。
  - 实际存在 `tests/unit/batch-executor/integration.test.ts` 和 `concurrent-executor.test.ts`
  - 但这些测试验证的是当前占位执行器，不是实际 Agent 调用
- “所有 High 级问题都会阻断当前交付”不成立。
  - 例如审计日志的性能问题更适合标为 Medium

## 建议

1. 先把 batch executor 明确标为 experimental，避免误导
2. 为 `saveTodoState()` 增加文件锁
3. 抽取统一 `haltOnBlocked()` 辅助函数
4. 给正则类配置增加校验或白名单

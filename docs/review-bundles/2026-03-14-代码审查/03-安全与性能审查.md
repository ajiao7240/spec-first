# 安全与性能审查（校正版）

## 已验证安全问题

### SEC-000 [Critical] 迁移执行器命令注入

位置：`src/core/migrations/manifest-engine.ts`

证据：

```ts
const fullCommand = step.command + (step.args?.length ? ' ' + step.args.join(' ') : '');
execSync(fullCommand, { cwd, stdio: 'ignore' });
```

说明：这是典型的拼接 shell 字符串执行。`step.command` / `step.args` 只要来自不可信输入，就具备注入面。

### SEC-001 [High] AI Runtime 扩展 Hook 命令注入

位置：`src/core/tool-integration/ai-runtime-hook.ts`

证据：

```ts
command: `sh -c 'SPEC_FIRST_EXTENSION_NAMESPACE=${ext.namespace}; ${hook.command}'`
```

说明：`ext.namespace` 和 `hook.command` 都被拼接进 shell 命令。

### SEC-002 [Medium] 正则配置可触发 ReDoS

位置：

- `src/core/ai-orchestrator/completion-detector.ts`
- `src/core/ai-orchestrator/slop-checker.ts`

说明：两处都直接编译配置中的正则，没有复杂度限制或超时保护。

### SEC-003 [Medium] `assertSafePath()` 只做绝对路径约束，不做根目录约束

位置：`src/shared/fs-utils.ts`

现状：会拒绝相对路径和空字节，但不会确认目标路径属于指定白名单根目录。

影响：该函数更像“绝对路径卫生检查”，不是“访问范围控制”。

### SEC-004 [Low] 审计链不可抵抗本地重写

位置：`src/core/ai-orchestrator/audit-log.ts`

现状：基于纯 SHA256 hash chain，没有 HMAC 或外部不可变存储。

影响：适合做一致性校验，不适合宣称强不可抵赖。

## 已验证性能问题

### PERF-001 [Medium] 审计日志写入会随文件增长线性变慢

位置：`src/core/ai-orchestrator/audit-log.ts`

原因：每次写入都全量 `readFileSync()`。

### PERF-002 [Low] Git 调用仍以同步方式分布在多处运行时路径

位置示例：

- `src/core/skill-runtime/hard-gate.ts`
- `src/core/skill-runtime/scope-guard.ts`
- `src/shared/host-bootstrap.ts`

说明：这不一定是 bug，但在 CLI 热路径中应谨慎评估阻塞时间。

## 修正说明

- 上一版将部分“安全意识良好但存在边界”的实现描述成了已失效设计，本版已收敛
- 安全问题以“代码直接可见的注入面 / 复杂度风险 / 信任边界不足”为准，不再扩展到未证实场景

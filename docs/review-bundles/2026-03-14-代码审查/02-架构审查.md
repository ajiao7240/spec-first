# 架构审查（校正版）

## 核心判断

当前最明显的架构问题不是“所有子系统都失控”，而是“规范、Gate、阶段推进三套真相没有完全对齐”。

## 已验证问题

### AR-001 [High] Gate 条件与 Skill 文档存在双重事实源

证据：

- `skills/spec-first/12-verify/SKILL.md`
- `src/core/gate-engine/condition-registry.ts`

已确认的不一致：

- `G-IMPL-01`
  - Skill：C4 ≥ 80%
  - 代码：C4 ≥ 60%
- `G-VERIFY-01`
  - Skill：C4 = 100%
  - 代码：C4 ≥ 80%

这已经足以说明 Gate 与 Skill 没有单一事实源。

### AR-002 [High] `07_release` 的阶段语义分裂

证据：

- `condition-registry.ts` 为 `07_release` 注册了 `G-REL-01` / `G-REL-02`
- `golive.ts` 的 `GL-05` 也要求 release 证据
- `advance.ts` 又会在进入 `07_release` 后自动推进到 `08_done`

当前处置：

- 保留 `07_release` 作为正式发布阶段
- 当前实现允许在进入 `07_release` 后自动收口到 `08_done`
- release 证据要求继续保留，作为当前产品语义的一部分

结论：原始“是否冲突”问题已被产品决策收敛，后续重点不再是删改语义，而是确保文档、测试和 CLI 提示持续与这一实现一致。

### AR-003 [Medium] Gate 评估包含副作用

位置：`src/core/gate-engine/condition-registry.ts`

`evaluateAnalyzeCriticalFindings()` 会在报告过期时直接调用 `analyzeArtifacts()` 刷新结果。

影响：Gate 不再是纯评估过程，耗时和副作用都不可预测。

### AR-004 [Medium] 状态持久化策略不统一

现状：

- `trace-engine/id-generator.ts` 已使用 `withFileLock()`
- `todo-runner.ts` 没有使用锁

影响：不同状态文件的并发保护策略不一致。

## 不再保留的结论

- “多状态机缺乏统一协调层”作为长期架构观察可以成立，但当前 bundle 没有足够代码证据支撑其 Critical 评级
- “RFC/Defect 一定导致 Gate 失真”没有直接复现实证，本版不保留

## 建议

1. 让 Gate 条件从 Skill/规则源自动生成，或至少引入一致性测试
2. 保持 `07_release` 自动收口语义在文档、测试、CLI 提示中持续一致
3. 把 Gate 中的刷新动作外移到显式预处理步骤

# <需求名称> 增量需求文档

> 本模板是「证券行业通用增量需求」主模板，适配所有 surface 的 80% 场景。它对应 `spec-prd` 的核心体验，遵循 spec-first 的 **WHAT not HOW** 边界。
>
> **使用规则**：标 **[core]** 的 section 始终必填；标 **[conditional]** 的 section 按需求规模与 surface 决定是否展开。surface 专属关注点请叠加对应 lens 模板（10-App / 20-Admin / 30-Backend）。
>
> **证券行业**：写券商需求前，先对照 `90-证券行业需求关注点与参考附录.md` 的「行业横切关注点必查清单」（合规/适当性/AML-KYC/风控/行情/交易订单/资金清结算/多展业地/审计/精度时区），逐项判断是否适用；适用项必须在下方 core 或对应 conditional section 写清。
>
> **WHAT not HOW**：本文只写产品行为、业务规则、范围、验收。数据库 schema、接口字段设计、文件改动、任务拆解、技术选型一律不写，留给 `spec-plan`。

## Summary [core]

用一句话讲清「给谁、在什么场景、解决什么问题、做成什么样」。

> 例：为 CRM 运营提供统一的入金审核备注能力，在处理入金单据时记录沟通、复核意见和后续追溯依据。

## Change Delta [core]

> 增量需求的核心。说清这次相对现有系统**改了什么**，把新需求和现有能力分清楚。

| 变化类型 | 内容 | 涉及现有能力 | 资金/交易/合规影响 | 证据 tag |
| --- | --- | --- | --- | --- |
| keep（保持不动） | | | | |
| extend（在现有上扩展） | | | | |
| replace（替换现有行为） | | | | |
| remove（下线/移除） | | | | |
| unknown（待确认） | | | | |

证据 tag 取值：`confirmed-source`（已读源码/测试/文档确认）/ `user-stated`（用户明确）/ `gitnexus-pointer`（图谱线索，未经源码确认）/ `external-research`（外部资料，需标日期与来源）/ `assumption`（推断）。

涉及历史逻辑时，不得写「沿用历史逻辑」或「与之前一致」作为唯一说明；必须写清差异、保留点和影响范围。

> **写好示例（增量风格）**：
> ✗ 模糊：`extend｜优化用户列表`
> ✓ 具体：`extend｜用户列表增加"批量禁用"｜复用现有列表筛选与分页(confirmed-source: admin/user_list)｜涉及操作审计(C9)｜confirmed-source`
> 关键差别:替换了哪条现有行为、复用了什么、不碰什么,要让 plan 不必猜。

## Requirements [core]

> 一条需求 = 一个可独立验收的最小行为。用编号 + EARS 句式：「当 <触发条件> 时，<角色/系统> 应 <可观察的行为>」。

| 编号 | 触发条件 | 角色 | 系统行为 | 用户可见结果 | 覆盖行业关注点 |
| --- | --- | --- | --- | --- | --- |
| R-01 | | | | | Cx |
| R-02 | | | | | Cx |

业务规则单独编号（避免散落在截图/口头说明）：

- BR-001：<一条规则，"当 X 时应 Y">
- BR-002：

> **写好示例（增量风格 + 反模糊词）**：
> ✗ 模糊：`支持批量操作，体验更好`（"等/相关/更好"这类词让 plan 无从验收）
> ✓ 具体：`R-01｜当管理员在用户列表勾选 ≥1 行并点"批量禁用"时，系统应弹二次确认、确认后将选中用户置为已禁用并写审计日志，列表实时刷新被禁用状态`
> 反模糊词:不写"等/相关/合适的/更好";数量、状态、边界都要可验收。

**优先级分级（必填——专业 PRD 与功能清单的分水岭）**：每条需求标 P0/P1/P2 或 MoSCoW（Must/Should/Could/Won't）。回答"排期被砍时先砸哪些、哪些可降级、哪些阻塞上线"。证券场景中合规/风控/资金安全相关需求通常不可降级，须显式标注。

| --- | --- | --- | --- |
| R-01 | P0 / Must | | 是 / 否 |
| R-02 | | | |

## Acceptance Examples [core]

> 一条核心需求至少 1 条正向 + 1 条异常验收。用 Given/When/Then 或 EARS 等价。每条标注覆盖的 R 编号和行业关注点。

```text
AC-01（对应 R-01 / Cx）
Given <前置状态>
When <用户动作>
Then <可观察结果>

AC-02（对应 R-02 / Cx，异常）
Given <前置状态>
When <异常输入/操作>
Then <系统拒绝并提示，且不产生资金/订单/审计副作用>
```

涉及资金或交易的验收必须覆盖：重复提交、状态冲突、弱网/超时、部分成功、用户取消、低权限用户、低适当性客户、展业地差异。

## Handoff

- 产品行为已明确、下游 `spec-plan` 无需发明 WHAT → 推荐进入 `spec-plan`。
- 需要审查完整性/一致性 → `spec-doc-review`。
- 仍有未决产品行为 → 继续 refine 或记入 Outstanding Questions。
- **不要**直接进入 `spec-work`。

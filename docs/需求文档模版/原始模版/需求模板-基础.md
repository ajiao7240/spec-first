# 需求模板\-基础

## 设计前提

- **作者**：产品经理；**首要读者**：AI 编码助手 \+ 研发 \+ 测试

- **写作哲学**：把\&\#34;做什么\&\#34;写到 AI 不用猜，把\&\#34;为什么\&\#34;写到研发不用问，把\&\#34;怎么验\&\#34;写到测试不用编

- **粒度**：一条需求 = 一个可独立验收的最小行为

- **句式**：尽量用 EARS —— \&\#34;当 \&lt;触发条件\&gt; 时，\&lt;角色/系统\&gt; 应 \&lt;可观察的行为\&gt;\&\#34;

## 0\. 一句话需求

> 写法：用一句话讲清\&\#34;给谁、解决什么问题、做成什么样\&\#34;。
> 
> 例：给 KAZ 注册用户提供修改昵称的能力，使其在客户端展示和好友搜索中生效。
> 
> 

## 1\. 背景与目标

- **当前是什么样**：现状一两句（含痛点 / 数据 / 反馈来源）

- **要变成什么样**：目标一两句

- **成功指标**：可量化（数值 / 转化率 / 错误率 / 完成时长）；实在量化不了，写\&\#34;上线后客服关于 X 的工单减少\&\#34;这类可观察口径

- **不做这件事的风险**：让研发理解优先级

## 2\. 范围与不做的事

- **本期做**：要点 bullet

- **本期不做**（重要，AI 最容易自己加戏）：明确列出\&\#34;看起来相关但本期不动\&\#34;的功能

- **依赖外部**：依赖哪些团队、哪些已有接口 / 系统、谁负责什么

## 3\. 用户与场景

- **角色**：用户类型（新用户 / 老用户 / 客服 / 管理员…）

- **进入场景**：从哪里进来、带着什么状态进来（已登录 / 未登录、首次 / 复访…）

- **退出去向**：完成后跳到哪、看到什么反馈

## 6\. 数据与字段

- **新增 / 修改字段**：字段名（中文）、说明、来源（用户输入 / 系统生成 / 外部接口）、是否必填、长度限制、合法字符

- **示例数据**（强烈建议放 1\-2 个真实样例）：

    - 合法：小明

    - 合法：María（含拉丁字符）

    - 非法：含 emoji 🌟

- **同一概念在不同地方的展示**：列出昵称会出现在哪几个页面 / 通知 / 消息中

## 7\. 业务规则与异常

> 用\&\#34;当 X 时，应 Y\&\#34;句式，一行一条。
> 
> 

- **正常规则**：当昵称重复时，应提示\&\#34;该昵称已被使用\&\#34;

- **边界规则**：当用户在 30 天内已修改过 1 次，应禁止再次修改并提示剩余天数

- **异常处理**：当后端接口超时 \&gt; 3 秒，应展示\&\#34;网络不佳，请稍后重试\&\#34;，且不应清空用户输入

- **不应清单**（防 AI 自由发挥）：不应在保存前发送任何短信；不应自动同步到第三方平台

## 8\. 兼容性与影响面

> PM 视角能写的：
> 
> 

- **影响哪些已有功能**：好友列表、推送通知、客服后台搜索…

- **老数据怎么办**：现有用户的历史昵称是否需要迁移、是否需要默认值

- **客户端版本**：低版本 App 是否需要兼容、是否强制升级

- **多端表现**：iOS / Android / H5 / 后台是否需要同步

## 9\. 非功能要求

- **性能感知**：用户操作到看到结果应 ≤ N 秒

- **可用性**：上线后该入口的成功率应 ≥ 99%

- **安全 / 合规**：是否涉及个人信息、是否要脱敏展示、是否要审计日志

- **多语言**：涉及哪些语言版本的文案（提示语、错误码）

## 10\. 验收标准（Given\-When\-Then 或示例驱动）

> 一条功能 = 至少 1 条正向 \+ 1 条异常验收用例。研发和 AI 直接照这个写测试。
> 
> 

```
用例 A-01（对应 F-04）
Given 用户已登录，当前昵称是"小明"
When 用户提交昵称"小红"
Then 编辑页关闭，资料页昵称展示为"小红"，后台数据库该用户昵称字段更新为"小红"

用例 A-02（对应 F-03）
Given 用户在编辑页
When 用户提交昵称"aaaaaaaaaaaaaaaaaaaaa"（21 字符）
Then 提交按钮拒绝，页面提示"昵称最多 20 字"，不发起后端请求
```

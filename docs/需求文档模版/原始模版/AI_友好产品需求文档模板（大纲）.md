# AI 友好产品需求文档模板（大纲）

## 0\. AI 读取说明

> 本节用于约束文档本身，不写技术方案。
> 
> 

示例：

```text
本文是产品需求文档，不包含数据库设计、接口设计、代码实现方案。
本文的重点是：业务目标、用户角色、功能范围、页面行为、业务规则、异常场景、验收标准。
需求必须按模块拆分，不要把多个独立模块写在同一篇大文档里。
如果技术实现需要补充，由研发在技术方案中单独承接。
```

## 1\. 一句话需求

用一句话说清楚“给谁，在什么场景下，解决什么问题”。

示例：

```text
为 CRM 运营人员提供统一的任务备注能力，方便在处理开户、入金、出金等单据时记录和查看沟通信息。
```

## 2\. 需求拆分规则

> 这是 AI 友好需求文档的前置要求。先拆分，再写正文。
> 
> 

### 2\.1 为什么必须拆分

一个大 PRD 同时包含多个模块，会导致 AI 读取时上下文过大、规则互相干扰、验收口径混乱。产品需要把需求按可独立理解、独立评审、独立验收的模块拆开。

示例：

```text
不推荐：
《CRM 出入金需求总文档》

推荐拆成：
1. 《入金任务列表需求》
2. 《入金任务详情需求》
3. 《入金审核操作需求》
4. 《出金任务列表需求》
5. 《出金任务详情需求》
6. 《出金审核操作需求》
7. 《通用备注能力需求》
```

### 2\.2 拆分原则

示例：

```text
1. 按业务模块拆：开户、入金、出金、退款、CRA&EDD 分开写。
2. 按页面拆：列表页、详情页、操作弹窗、配置页分开写。
3. 按流程节点拆：提交、初审、复审、驳回、撤回分开写。
4. 按用户角色拆：运营端、主管端、客户侧、前台页面分开写。
5. 通用能力单独写：备注、附件、操作日志、权限、导出等不要散落在多个业务文档里。
```

### 2\.3 单篇文档大小限制

示例：

```text
1. 一篇文档只描述一个独立模块或一个独立能力。
2. 一篇文档建议控制在 800 行以内。
3. 一篇文档不超过 20K tokens。
4. 如果截图、表格、规则太多，需要继续拆子文档。
5. 跨文档引用必须写清楚文档名和章节名，不能只写“见上文”“见大需求”。
```

### 2\.4 拆分后的文档关系

如果多个文档属于同一个大需求，需要建立一个总索引文档。总索引只放导航和边界，不承载具体规则。

示例：

```text
总索引：《CRM 出入金需求索引》

包含文档：
1. 《入金任务列表需求》：只描述入金列表筛选、字段、操作入口。
2. 《入金任务详情需求》：只描述详情展示、备注、附件、操作记录。
3. 《入金审核操作需求》：只描述通过、驳回、补充材料等审核动作。
4. 《出金任务列表需求》：只描述出金列表筛选、字段、操作入口。
5. 《通用备注能力需求》：描述所有任务共用的备注能力。

规则：
1. 具体业务规则写在子文档，不写在总索引。
2. 子文档之间有依赖时，用明确链接引用。
3. 同一个规则只能放在一个子文档里，其它文档只引用。
```

## 3\. 背景与问题

说明为什么要做，重点写业务现状和问题，不写实现。

示例：

```text
当前不同任务页面的备注入口和展示方式不一致，运营需要在多个页面之间切换查看历史记录。
这导致处理效率低，也容易遗漏关键沟通信息。
```

## 4\. 目标与衡量标准

写清楚本需求想达到什么效果，以及如何判断做成了。

示例：

```text
目标：
1. 运营可以在任务详情页直接查看备注历史。
2. 运营可以新增、编辑、删除备注。
3. 同一类任务的备注展示和操作方式保持一致。

衡量标准：
1. 运营处理任务时不需要跳转到其它系统查看备注。
2. 测试验收时，所有支持的任务类型都能完成备注增删改查。
```

## 5\. 用户角色与使用场景

说明谁会用、在什么场景用、目标是什么。

|角色|使用场景|用户目标|
|---|---|---|
|运营人员|处理任务详情时补充沟通记录|记录处理过程，方便后续追溯|
|运营主管|查看任务历史备注|了解处理过程和风险点|
|测试人员|验收备注功能|验证不同任务类型行为一致|

## 6\. 需求范围

### 6\.1 本期要做

只写产品能力和业务范围。

示例：

```text
1. 在任务详情页展示备注列表。
2. 支持新增备注。
3. 支持编辑自己或他人的备注，具体权限见第 9 节。
4. 支持删除备注，删除后列表不再展示。
5. 支持以下任务类型：开户、入金、出金、退款。
```

### 6\.2 本期不做

明确不做什么，避免 AI 或研发自动扩展。

示例：

```text
1. 不做历史备注数据迁移。
2. 不支持备注附件。
3. 不支持 @人提醒。
4. 不支持富文本，只支持纯文本。
5. 不新增消息通知。
```

### 6\.3 本文档与其它模块的关系

如果本需求依赖其它模块，必须写清楚依赖关系，避免 AI 把多个模块混在一起实现。

示例：

```text
1. 本文档只描述“通用备注能力”。
2. 开户详情页如何接入备注，见《开户任务详情需求》。
3. 入金详情页如何接入备注，见《入金任务详情需求》。
4. 权限菜单如何配置，见《CRM 权限配置需求》。
```

## 7\. 页面与入口

描述用户从哪里进入、看到什么、能做什么。

### 7\.1 页面入口

示例：

```text
入口位置：CRM 后台 -> 任务管理 -> 任务详情页 -> 备注区域。
展示位置：任务详情页底部，跟随任务基础信息一起展示。
```

### 7\.2 页面元素

|元素|是否必须|展示规则|示例|
|---|---|---|---|
|备注列表|是|有备注时展示，按时间倒序|最新备注展示在最上方|
|新增按钮|是|用户有备注权限时展示|“新增备注”|
|编辑按钮|是|符合权限时展示|“编辑”|
|删除按钮|是|符合权限时展示|“删除”|
|空状态|是|无备注时展示|“暂无备注”|

## 8\. 用户流程

用编号步骤写清楚用户怎么完成操作。流程图可以有，但不能只放图。

### 8\.1 新增备注流程

示例：

```text
1. 用户进入任务详情页。
2. 用户点击“新增备注”。
3. 系统打开备注输入框。
4. 用户输入备注内容。
5. 用户点击“保存”。
6. 系统保存成功后关闭输入框，并刷新备注列表。
7. 新增备注展示在列表最上方。
```

### 8\.2 编辑备注流程

示例：

```text
1. 用户在备注列表中点击“编辑”。
2. 系统展示原备注内容。
3. 用户修改内容后点击“保存”。
4. 系统保存成功后刷新该条备注。
5. 备注展示最新内容和最后修改人。
```

### 8\.3 删除备注流程

示例：

```text
1. 用户点击“删除”。
2. 系统弹出二次确认。
3. 用户确认删除。
4. 系统删除成功后，该备注从列表中消失。
```

## 9\. 业务规则

> 这是产品需求最重要的部分。每条规则单独编号，避免散落在截图和口头说明里。
> 
> 

### 9\.1 内容规则

示例：

```text
BR-001：备注内容不能为空。
BR-002：备注内容最多 2000 个字符。
BR-003：备注内容只支持纯文本，不支持图片、附件、富文本。
BR-004：备注内容前后空格保存时自动去除。
```

### 9\.2 权限规则

示例：

```text
BR-101：有任务详情查看权限的用户，可以查看备注列表。
BR-102：有任务处理权限的用户，可以新增备注。
BR-103：是否允许编辑他人备注，需要产品明确。
BR-104：是否允许删除他人备注，需要产品明确。
```

### 9\.3 展示规则

示例：

```text
BR-201：备注列表按创建时间倒序展示。
BR-202：每条备注展示备注内容、创建人、创建时间。
BR-203：如果备注被编辑过，需要展示最后修改人和最后修改时间。
BR-204：删除后的备注不在列表中展示。
```

### 9\.4 任务类型规则

示例：

```text
BR-301：开户、入金、出金、退款任务均支持备注。
BR-302：不同任务类型的备注入口和交互保持一致。
BR-303：如果某个任务类型暂不支持，必须在第 5.2 节写入“不在本期范围”。
```

## 10\. 异常与边界场景

把用户可能遇到的失败、边界、空数据写清楚。

|场景|系统表现|用户提示|
|---|---|---|
|备注内容为空|不允许保存|“请输入备注内容”|
|备注内容超长|不允许保存|“备注最多支持 2000 个字符”|
|无备注数据|展示空状态|“暂无备注”|
|删除时用户取消|不删除备注|关闭确认弹窗|
|保存失败|保留输入内容|“保存失败，请稍后重试”|

## 当前结构分析

- **Skills**: 41 个（包含 5 个 spec-* 核心 skills）
- **Agents**: 6 个
- **Scripts**: 6 个
- **Lib**: 5 个 CLI 实现文件
- **Bin**: 1 个 CLI 入口

```

lib/init.js                # 命令生成
lib/templates.js           # 模板处理
templates/claude/commands/spec/*.md  # 命令模板
package.json               # npm 配置
```

**核心 skills**:
```
skills/spec-brainstorm/
skills/spec-plan/
skills/spec-work/
skills/spec-review/
skills/spec-compound/
```

## 问题识别

### ❌ 问题1: 大量非核心 skills

**现状**: 41 个 skills，其中只有 5 个是 spec-* 核心

**问题**:
- 36 个 skills 来自 compound-engineering 迁移
- 与 npm CLI 模式无关
- 增加包体积（如果发布到 npm）
- 用户困惑（不知道哪些是核心功能）

**示例**:
```
skills/andrew-kane-gem-writer/    # Ruby gem 相关
skills/dhh-rails-style/           # Rails 风格
skills/gemini-imagegen/           # 图片生成
skills/frontend-design/           # 前端设计
```

### ❌ 问题2: scripts/ 目录用途不明

**现状**: 6 个脚本文件

**问题**:
- 部分脚本是旧 plugin 模式的（task-manager.sh, stage-gate.sh）
- 与 npm CLI 模式不一致
- 用户手册中已不再引用

### ❌ 问题3: agents/ 目录定位模糊

**现状**: 6 个 agent 目录

**问题**:
- npm CLI 模式不使用 agents
- 如果保留，应该说明用途

### ❌ 问题4: docs/ 结构混乱

**现状**:
```
docs/01-需求分析/
docs/02-架构设计/
docs/03-实施方案/
docs/04-一致性检查/
docs/05-用户手册/
docs/superpowers/
docs/业界分析/
```

**问题**:
- 开发文档和用户文档混在一起
- 中英文混用
- 部分目录可能过时

## 优化建议

### 方案A: 最小化发布（推荐）

**目标**: npm 包只包含核心功能

**结构**:
```
spec-first/
├── bin/                    # CLI 入口
├── lib/                    # CLI 实现
├── templates/              # 命令模板
├── package.json
├── README.md
└── LICENSE
```

**优点**:
- 包体积最小
- 职责清晰
- 易于维护

**缺点**:
- 失去扩展 skills

**实施**:
1. 移除 skills/、agents/、scripts/
2. 更新 package.json files 字段
3. 创建独立仓库存放扩展 skills

---

### 方案B: 核心+扩展分离

**目标**: 保留扩展能力，但分离核心和扩展

**结构**:
```
spec-first/
├── bin/
├── lib/
├── templates/
├── core/                   # 核心 skills
│   ├── spec-brainstorm/
│   ├── spec-plan/
│   ├── spec-work/
│   ├── spec-review/
│   └── spec-compound/
├── extensions/             # 扩展 skills（不发布）
│   ├── frontend-design/
│   ├── gemini-imagegen/
│   └── ...
├── docs/
│   ├── user/              # 用户文档
│   └── dev/               # 开发文档
├── package.json
└── README.md
```

**优点**:
- 核心功能清晰
- 保留扩展能力
- 文档分类明确

**缺点**:
- 需要重构目录

**实施**:
1. 创建 core/ 目录，移入 5 个核心 skills
2. 创建 extensions/ 目录，移入其他 skills
3. 更新 package.json files: ["bin/", "lib/", "templates/", "core/"]
4. 重组 docs/

---

### 方案C: 保持现状+清理

**目标**: 最小改动，只清理明显过时内容

**操作**:
1. 删除 scripts/ 中的旧脚本
2. 删除 agents/（如果不用）
3. 整理 docs/
4. 更新 .npmignore

**优点**:
- 改动最小
- 风险最低

**缺点**:
- 包体积大
- 结构不够清晰

---

## 推荐方案

### 🎯 推荐：方案B（核心+扩展分离）

**理由**:
1. 符合 npm CLI 模式的定位
2. 核心功能清晰
3. 保留扩展能力
4. 便于未来发展

**实施步骤**:

**第1步**: 创建新目录结构
```bash
mkdir -p core extensions docs/user docs/dev
```

**第2步**: 移动核心 skills
```bash
mv skills/spec-* core/
```

**第3步**: 移动扩展 skills
```bash
mv skills/* extensions/
```

**第4步**: 重组文档
```bash
mv docs/05-用户手册/* docs/user/
mv docs/01-需求分析 docs/02-架构设计 docs/03-实施方案 docs/dev/
```

**第5步**: 更新 package.json
```json
{
  "files": [
    "bin/",
    "lib/",
    "templates/",
    "core/",
    "README.md"
  ]
}
```

**第6步**: 更新 .npmignore
```
extensions/
docs/dev/
scripts/
agents/
tests/
.claude/
```

## 1. 设计目标与约束

### 1.1 设计目标

| # | 目标 | 量化指标 | 优先级 |
|---|------|---------|--------|
| G1 | **全链路可追踪** | FR→DS→API→TASK→TC→PR 6层追踪链，覆盖率 = 100% | P0 |
| G2 | **自动化门禁** | 4 道关键 Gate（Design Ready / Code Ready / Release Ready / Go Live）自动校验，人工终审放行，Gate 首次通过率 > 85% | P0 |
| G3 | **规范一致性** | 3 次合并触发 SCA（sca_design / sca_code / sca_test），不一致项当阶段修复率 = 100% | P0 |
| G4 | **AI 协作无缝** | Agent 路由准确率 > 95%，会话恢复成功率 > 95% | P1 |
| G5 | **度量可运营** | 12项指标自动采集，运营闭环可执行 | P2 |
| G6 | **渐进式落地** | 3步路线图，每步有明确验收标准和超时策略 | P0 |

### 1.2 设计约束

| 约束类型 | 约束内容 | 来源 |
|---------|---------|------|
| **运行环境** | CLI 工具链，不依赖 Web Server；本地 Git 仓库 + CI/CD Pipeline | 产品边界 |
| **技术栈** | Node.js 为主（校验脚本）、Shell 脚本（Git Hook）、YAML/Markdown（数据格式） | v5 工具链映射 |
| **兼容性** | 纯人工场景（无 AI）必须可独立运行，Layer B Hook 承载全部 Gate | 双层 Hook 设计 |
| **团队规模** | 10-50 人研发团队，不做企业级 PLM | 目标用户 |
| **零侵入** | 不修改业务代码结构，仅通过 Git Hook + CI Pipeline + 文件规范约束 | 架构原则 |
| **渐进式** | MVP 阶段手动维护，验证价值后再工具化 | 落地路线图 |

### 1.3 设计原则

| 原则 | 说明 |
|------|------|
| **Convention over Configuration** | 目录结构、文件命名、ID 格式均有默认约定，减少配置 |
| **Fail-Fast** | Hook 校验不通过立即阻断，不允许不合规产物进入下一阶段 |
| **Append-Only** | 运行态三文件、AI 统计采用追加模式，避免并发冲突 |
| **Idempotent** | 校验脚本可重复执行，结果一致 |
| **Offline-First** | 核心校验在本地完成（Git Hook），CI 作为兜底防线 |

### 2.2 架构决策记录

> **[Arch]**: 采用分层架构而非微服务，原因：Spec-First 是 CLI 工具链而非在线服务，分层架构足够且部署简单。
> **[DevEff]**: 同意。开发者日常接触的是 Git Hook 和 CLI 命令，不需要启动额外服务进程。
> **[QA]**: 关键是执行层的双层 Hook 必须解耦——Layer A 不存在时 Layer B 独立运行。
> **[决议]**: 采用 4 层架构（交互→编排→核心服务→数据），执行层作为横切层嵌入。Layer A/B 通过统一接口调用核心服务，互不依赖。

## 3. 技术选型决策

### 3.1 核心技术栈

| 领域 | 选型 | 备选 | 决策理由 |
|------|------|------|---------|
| **校验引擎** | Node.js (ESM) | Python, Deno | 前端团队熟悉、npm 生态丰富、remark/unified 原生支持 |
| **Markdown 解析** | remark + unified | markdown-it, gray-matter | AST 级操作能力强，插件生态完善，支持自定义 transformer |
| **YAML 解析** | js-yaml | yaml (npm) | 轻量、零依赖、YAML 1.2 完整支持 |
| **正则引擎** | Node.js 内置 RegExp | RE2 | ID 格式校验场景简单，无 ReDoS 风险，内置即可 |
| **Git Hook** | Shell + Node.js | Husky, lefthook | Shell 做入口分发，Node.js 做复杂校验；避免引入额外依赖 |
| **CI 集成** | GitHub Actions | GitLab CI, Jenkins | 团队主用 GitHub，YAML 配置，Marketplace 丰富 |
| **AI Hook** | Claude Code Hooks | 自定义 MCP | 原生支持 PreToolUse/PostToolUse/Stop 三种时机 |
| **统计存储** | JSONL (本地文件) | SQLite, LevelDB | 追加写入、无需额外进程、grep 友好、轮转简单 |
| **Jira 集成** | REST API v3 | Jira Automation | API 灵活度高，可按需集成，不依赖 Jira 插件 |

### 3.2 专家会审记录

#### 议题 1：校验引擎语言选择

> **[Arch]**: Node.js vs Python。Python 在 AST 解析和脚本编写上更简洁，但团队以前端为主。
> **[DevEff]**: Node.js 的 remark/unified 生态是 Markdown 处理的事实标准，且团队 10+ 人熟悉 JS/TS。Python 需要额外学习成本。
> **[QA]**: 校验脚本需要在 CI 和本地 Git Hook 中运行，Node.js 在两个环境都有原生支持。
> **[决议]**: **Node.js (ESM)**。理由：团队技能匹配 + remark 生态 + CI/本地双环境兼容。

#### 议题 2：数据存储方案

> **[Arch]**: 追踪矩阵和度量数据是否需要数据库？SQLite 可提供查询能力。
> **[DevEff]**: 引入数据库增加部署复杂度。Markdown/YAML 文件可直接 Git 版本控制，开发者无需额外工具。
> **[QA]**: S 规模用 Markdown 表格，M/L 用 YAML——这已经在需求中定义了。度量数据用 JSONL 追加写入，按月轮转，6 个月保留。
> **[决议]**: **文件即数据库**。Markdown（S 规模）/ YAML（M/L 规模）存储追踪矩阵，JSONL 存储 AI 统计。不引入额外数据库。

#### 议题 3：Git Hook 实现方式

> **[Arch]**: Husky 是社区标准，但引入了 npm 依赖。直接用 Shell 脚本更轻量。
> **[DevEff]**: Husky 的 `.husky/` 目录约定已被广泛接受，但 Spec-First 的 Hook 逻辑较复杂（阶段感知），需要 Node.js 处理。
> **[QA]**: 关键是 commit-msg Hook 的阶段感知规则——根据提交路径识别阶段，允许对应 ID 标签。这用纯 Shell 实现会很脆弱。
> **[决议]**: **Shell 入口 + Node.js 校验**。`.git/hooks/commit-msg` 用 Shell 脚本做入口，调用 `scripts/check-commit-msg.mjs` 做实际校验。不依赖 Husky。

### 6.4 专家会审记录

> **[Arch]**: 目录结构遵循 Convention over Configuration 原则，Feature 目录命名 `<飞书需求ID>-<需求名称>`，天然与需求源头对齐，便于审计和追踪。
> **[DevEff]**: `.spec-first/` 目录集中管理工具链配置，开发者只需关注 `specs/<飞书需求ID>-*/` 下的产出物。模板机制降低上手成本。
> **[QA]**: 追踪矩阵按规模选择 `.md` 或 `.yaml` 格式，这在 Gate 校验脚本中需要做格式自动检测。运行态文件的 `.jsonl` 追加模式避免并发冲突。
> **[决议]**: 目录结构确认。补充约定——`runtime/` 目录默认加入 `.gitignore`，但 `gate-history.jsonl` 可选择性提交以保留审计轨迹。

### 7.4 专家会审记录

> **[Arch]**: 部署架构的核心是"零服务进程"——所有校验在 Git Hook 或 CI Pipeline 中按需执行，不需要常驻后台服务。这大幅降低了运维复杂度。
> **[DevEff]**: `spec-first init` 一键初始化是关键体验。开发者 clone 仓库后执行一次即可，后续通过 Git 操作自然触发校验。需要注意 Git Hook 的 symlink 在 Windows 上的兼容性。
> **[QA]**: CI Pipeline 作为兜底防线至关重要——即使开发者绕过本地 Hook（`--no-verify`），CI 仍会拦截不合规提交。建议 CI Gate 校验失败时自动 block PR merge。
> **[决议]**: 采用"本地优先 + CI 兜底"双保险策略。`spec-first init` 负责本地环境初始化，CI Pipeline 配置随仓库提交。Windows 兼容性列入技术风险。

## 8. 技术风险与应对

### 8.1 风险矩阵

| # | 风险 | 影响 | 概率 | 等级 | 应对策略 |
|---|------|------|------|------|---------|
| R1 | **Markdown AST 解析边界** | remark 对非标准 Markdown 扩展（如 Mermaid、自定义容器）解析失败 | 中 | 🟡 | 校验仅处理结构化元素（标题/表格/ID标签），非标准语法透传不校验 |
| R2 | **大规模追踪矩阵性能** | L 规模 Feature（100+ FR）的矩阵校验耗时过长，阻塞 commit | 低 | 🟢 | 增量校验：仅校验本次 commit 涉及的文件变更，全量校验放 CI |
| R3 | **Git Hook 绕过** | 开发者使用 `--no-verify` 跳过本地 Hook | 高 | 🟡 | CI Pipeline 作为兜底防线，PR 合并前强制通过 Gate 校验 |
| R4 | **AI Agent 路由准确率** | Agent 路由错误导致错误阶段的操作 | 中 | 🟡 | Context Pack 携带阶段信息，Agent 执行前二次确认阶段匹配 |
| R5 | **Windows 环境兼容性** | Shell 脚本、symlink 在 Windows 上行为不一致 | 中 | 🟡 | 提供 PowerShell 版 Hook 入口，或推荐 WSL2 环境 |
| R6 | **JSONL 文件膨胀** | AI 统计数据长期积累导致文件过大 | 低 | 🟢 | 按月轮转，保留 6 个月，超期自动归档压缩 |
| R7 | **多人并发编辑追踪矩阵** | 同一 Feature 多人同时修改矩阵导致 Git 冲突 | 中 | 🟡 | YAML 格式按 FR-ID 分块，降低冲突概率；冲突时以 Gate 校验结果为准 |

### 8.2 专家会审记录

> **[Arch]**: R1 和 R2 是技术层面最需要关注的。remark 的 AST 解析对标准 GFM 支持良好，但自定义语法块需要明确"不校验"边界。大规模矩阵的增量校验需要在详细设计中给出具体算法——基于 `git diff` 的文件变更集做范围收窄。
> **[DevEff]**: R3 是现实中最高频的风险。开发者赶工时 `--no-verify` 是常见操作。CI 兜底是必须的，但更好的策略是让 Hook 校验足够快（< 2s），降低开发者绕过的动机。
> **[QA]**: R7 的并发冲突在多人协作的 M/L 规模项目中不可避免。建议追踪矩阵的 YAML 格式按 FR-ID 排序，每个 FR 独立一个 YAML block，最大化 Git 三路合并的成功率。
> **[决议]**: 7 项风险全部纳入跟踪。R3（Hook 绕过）和 R5（Windows 兼容）作为 MVP 阶段重点关注项。其余风险在对应模块的详细设计中给出具体应对方案。

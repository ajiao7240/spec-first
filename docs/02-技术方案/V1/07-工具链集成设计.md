## 1. 模块职责与边界

### 1.1 核心职责

| 职责 | 说明 |
|------|------|
| **Git Hook 安装与管理** | 管理 commit-msg / pre-push Hook 的安装（symlink）、升级、卸载，确保 Layer B 在所有开发环境可用 |
| **Layer A Hook 配置** | 维护 `claude-hooks.json` 配置文件，定义 PreToolUse / PostToolUse / Stop 三种 AI Runtime Hook |
| **Layer B Hook 脚本** | 实现 commit-msg 格式校验（Conventional Commits）、pre-push 增量 SCA 校验的入口脚本和 Node.js 校验逻辑 |
| **CI Pipeline 配置** | 提供 GitHub Actions YAML 模板，实现 PR 级全量 Gate 校验、度量报告生成、PR Comment 输出 |
| **Jira 状态同步** | 封装 Jira REST API v3 调用，实现阶段流转 → Jira Status 自动同步、缺陷数据双向同步 |
| **CLI 命令框架** | 提供 `spec-init` / `spec-id` / `spec-gate` / `spec-metrics` / `spec-matrix` 等 CLI 命令入口 |
| **Agent 注册表加载** | 从 `constitution.md` 解析 Agent 注册表，供 M5 AIOrchestrator 路由使用 |

### 1.4 专家会审记录

> **[Arch]**: M7 是「执行层胶水」——它不定义校验逻辑（M3 的职责），不定义度量指标（M6 的职责），而是负责将这些逻辑「安装」到正确的执行环境中（Git Hook、CI Pipeline、CLI）。M7 与 M3 的分工是：M3 定义「校验什么」，M7 负责「在哪里校验」和「怎么触发」。这种分离确保校验逻辑可以在不同执行环境中复用。
> **[DevEff]**: M7 直接影响开发者日常体验——Hook 安装是否顺畅、commit-msg 校验是否快速（< 2s）、CLI 命令是否直观。建议 M7 的所有用户交互都遵循「零配置即可用」原则：`spec-first init` 一键完成所有 Hook 安装和配置初始化，开发者无需手动编辑任何配置文件。
> **[QA]**: M7 的关键质量属性是「可靠性」——Hook 脚本不能因为自身 Bug 阻塞开发者的正常 Git 操作。建议所有 Hook 脚本都有 try-catch 兜底：校验脚本异常时输出警告但不阻断提交，避免「工具链故障 → 团队停工」的级联失败。
> **[决议]**: M7 定位为「执行层胶水」，不定义校验逻辑。遵循「零配置即可用」原则。Hook 脚本异常时降级为警告不阻断（但 CI Pipeline 不降级，作为最终防线）。

## 2. Git Hook 安装与管理

### 2.1 Hook 文件体系

```
.spec-first/
├── hooks/                          # Hook 源文件（Git 跟踪）
│   ├── commit-msg.sh              # Layer B: commit-msg 入口脚本
│   ├── pre-push.sh                # Layer B: pre-push 入口脚本
│   └── claude-hooks.json          # Layer A: Claude Code Hooks 配置
├── scripts/                        # 校验脚本（Git 跟踪）
│   ├── check-commit-msg.mjs       # commit-msg 校验逻辑
│   ├── check-id-format.mjs        # ID 格式校验
│   ├── check-traceability.mjs     # 追踪矩阵校验
│   ├── check-sca.mjs             # SCA 一致性校验
│   ├── calc-coverage.mjs          # 覆盖率计算
│   └── gate-evaluator.mjs         # Gate 条件评估
```

**安装方式**：通过 symlink 将 `.spec-first/hooks/` 下的脚本链接到 `.git/hooks/`，不直接修改 `.git/hooks/` 目录。

### 2.2 安装流程

```
spec-first init（Hook 安装部分）：

  1. 检测 .git/ 目录是否存在
     ├── 不存在 → 报错：请先 git init
     └── 存在 → 继续

  2. 检测 Node.js 版本
     ├── < 20 → 报错：需要 Node.js >= 20 LTS
     └── >= 20 → 继续

  3. 备份已有 Hook（如有）
     ├── .git/hooks/commit-msg 存在 → 重命名为 commit-msg.backup
     └── .git/hooks/pre-push 存在 → 重命名为 pre-push.backup

  4. 创建 symlink
     ├── ln -sf ../../.spec-first/hooks/commit-msg.sh .git/hooks/commit-msg
     └── ln -sf ../../.spec-first/hooks/pre-push.sh .git/hooks/pre-push

  5. 设置执行权限
     └── chmod +x .spec-first/hooks/*.sh

  6. 验证安装
     ├── 执行 .git/hooks/commit-msg --version → 输出版本号
     └── 执行 .git/hooks/pre-push --version → 输出版本号

  7. 输出安装摘要
     └── "✓ Git Hooks installed: commit-msg, pre-push"
```

### 2.3 跨平台兼容性

| 平台 | symlink 支持 | 解决方案 |
|------|-------------|---------|
| **macOS / Linux** | 原生支持 | 直接 `ln -sf` 创建 symlink |
| **Windows (Git Bash)** | 支持 | Git for Windows 内置 symlink 支持 |
| **Windows (CMD/PowerShell)** | 需管理员权限或开发者模式 | 提供 `spec-first init --copy` 模式，复制文件而非 symlink |
| **WSL2** | 原生支持 | 与 Linux 行为一致 |

**`--copy` 模式**：当 symlink 不可用时，将 Hook 脚本复制到 `.git/hooks/`。缺点是 Hook 源文件更新后需重新执行 `spec-first init` 同步。安装时记录模式到 `.spec-first/runtime/hook-install-mode.json`，后续升级时自动选择对应模式。

### 2.4 Hook 升级与卸载

```
升级流程（spec-first upgrade）：
  1. 检测当前安装模式（symlink / copy）
  2. symlink 模式 → 无需操作（源文件已更新，symlink 自动生效）
  3. copy 模式 → 重新复制 Hook 脚本到 .git/hooks/
  4. 校验版本号一致

卸载流程（spec-first uninstall）：
  1. 删除 .git/hooks/commit-msg 和 .git/hooks/pre-push
  2. 若存在 .backup 文件 → 恢复原有 Hook
  3. 保留 .spec-first/ 目录（配置和产出物不删除）
  4. 输出卸载摘要
```

### 2.5 Hook 脚本完整性校验（安全加固）

为防止 Hook 脚本被恶意篡改（如通过恶意 PR 修改 `.spec-first/hooks/*.sh`），引入 SHA256 完整性校验机制。

**校验和清单文件**：

```yaml
# .spec-first/hooks/integrity.yaml
schema_version: "1.0"
generated_at: "2026-02-08T10:00:00Z"
generated_by: "spec-first init"
checksums:
  commit-msg.sh: "sha256:a1b2c3..."
  pre-push.sh: "sha256:d4e5f6..."
  pre-commit.sh: "sha256:g7h8i9..."
```

**生成时机**：`spec-first init` 安装 Hook 时自动生成 `integrity.yaml`。

**校验逻辑**：

| 场景 | 校验行为 | 不匹配时处理 |
|------|---------|-------------|
| Hook 脚本执行时（本地） | 比对自身 SHA256 与 `integrity.yaml` 记录 | ⚠️ 输出警告，**不阻断**（避免误报） |
| CI Pipeline 中 | 强制校验 Hook 脚本完整性 | 🔴 阻断，报告篡改风险 |
| `spec-first doctor` | 检查所有 Hook 脚本完整性 | 输出诊断结果 |

**设计要点**：
- 本地执行时仅警告不阻断，避免因 `integrity.yaml` 未及时更新导致开发者无法提交
- CI 环境做强制校验，作为安全兜底
- `integrity.yaml` 本身也纳入 Git 版本控制，篡改需同时修改脚本和校验和，增加攻击成本

### 2.6 专家会审记录

> **[Arch]**: symlink 方案的核心优势是「源文件 Git 跟踪 + 执行文件自动同步」——团队成员 `git pull` 后 Hook 自动更新，无需手动重装。`--copy` 模式是 Windows 兼容的降级方案，不是推荐方案。建议在 `spec-first init` 输出中明确提示当前安装模式。
> **[DevEff]**: Hook 安装必须是幂等的——多次执行 `spec-first init` 不应产生副作用。已有 symlink 时跳过创建，已有 backup 时不重复备份。建议增加 `spec-first doctor` 命令，检查 Hook 安装状态、Node.js 版本、脚本权限等环境健康度。
> **[QA]**: Hook 脚本的版本号管理很重要——如果 Hook 脚本更新了但开发者的 `.git/hooks/` 还是旧版本（copy 模式），可能导致校验行为不一致。建议 Hook 脚本执行时检查版本号，版本不匹配时输出升级提示。
> **[决议]**: symlink 为默认模式，`--copy` 为降级方案。安装幂等。增加 `spec-first doctor` 命令。Hook 脚本内置版本检查。

## 3. Layer A Hook 配置

### 3.1 Layer A 定位与依赖

Layer A 是 Claude Code Hooks 机制提供的 AI Runtime Hook，**仅在 AI 辅助开发场景下生效**。Layer A 不存在时，Layer B 独立承载全部 Gate 校验。

| 属性 | 说明 |
|------|------|
| **依赖** | Claude Code 最新稳定版（内置 Hooks 机制） |
| **配置文件** | `.spec-first/hooks/claude-hooks.json` |
| **触发方式** | Claude Code 在 AI 会话中自动加载并执行 |
| **与 Layer B 关系** | 独立运行，无数据传递；Layer A 是增强层，Layer B 是防线层 |
| **校验严格度** | 柔性引导（Level 0-1），不强制阻断 |

**Layer A 的价值**：AI 辅助场景下，产出物在写入文件后**立即校验**，发现问题可让 AI 自动修正——比等到 commit-msg 时才发现问题效率高得多。

### 3.2 `claude-hooks.json` 完整 Schema

> **回应 M3 遗留问题 O2**：定义 Layer A Claude Code Hooks 的完整配置 Schema。

```json
{
  "$schema": "https://spec-first.dev/schemas/claude-hooks.v1.json",
  "version": "1.0.0",
  "hooks": {
    "PreToolUse": [
      {
        "id": "stage-gate-precheck",
        "matcher": {
          "tool_names": ["write_to_file", "edit_file", "create_file"]
        },
        "command": "node .spec-first/scripts/hook-pre-tool-use.mjs",
        "timeout_ms": 1000,
        "on_failure": "warn"
      }
    ],
    "PostToolUse": [
      {
        "id": "artifact-validation",
        "matcher": {
          "tool_names": ["write_to_file", "edit_file", "create_file"]
        },
        "command": "node .spec-first/scripts/hook-post-tool-use.mjs",
        "timeout_ms": 2000,
        "on_failure": "warn"
      }
    ],
    "Stop": [
      {
        "id": "session-completion-check",
        "matcher": {},
        "command": "node .spec-first/scripts/hook-stop.mjs",
        "timeout_ms": 5000,
        "on_failure": "ignore"
      }
    ]
  }
}
```

**Schema 字段说明**：

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `version` | string | ✅ | Schema 版本号，用于兼容性检查 |
| `hooks` | object | ✅ | 三种 Hook 类型的配置容器 |
| `hooks.<Type>[].id` | string | ✅ | Hook 唯一标识，用于日志和度量 |
| `hooks.<Type>[].matcher` | object | ✅ | 触发条件匹配器 |
| `matcher.tool_names` | string[] | ❌ | 匹配的工具名列表，空数组或省略表示匹配所有 |
| `hooks.<Type>[].command` | string | ✅ | 执行的校验脚本命令 |
| `hooks.<Type>[].timeout_ms` | number | ✅ | 超时限制（毫秒） |
| `hooks.<Type>[].on_failure` | enum | ✅ | 失败处理：`"warn"` 输出警告 / `"block"` 阻断操作 / `"ignore"` 静默跳过 |

### 3.3 PreToolUse Hook 校验逻辑

**触发时机**：AI 执行写操作（`write_to_file` / `edit_file` / `create_file`）**之前**。

**校验流程**：

```
AI 请求写操作
    │
    ▼
hook-pre-tool-use.mjs 接收上下文（toolName, filePath）
    │
    ├── 1. 解析 filePath → 推断目标 Feature 和阶段
    │     ├── specs/NNN-feat/spec.md → 01 Specify
    │     ├── specs/NNN-feat/design.md → 02 Design
    │     ├── specs/NNN-feat/tasks.md → 03 Plan
    │     ├── specs/NNN-feat/contracts/*.yaml → 02 Design
    │     ├── src/**/*.ts → 04 Implement
    │     ├── specs/NNN-feat/tests/*.test.md → 05 Verify
    │     └── 其他路径 → 跳过校验（非 Spec-First 管控文件）
    │
    ├── 2. 查询当前阶段（M1.getCurrentStage）
    │     └── 使用内存缓存，避免文件 I/O
    │
    ├── 3. 阶段匹配校验
    │     ├── 目标文件属于当前阶段 → PASS
    │     ├── 目标文件属于已完成阶段 → WARN（提示走 RFC 变更流程）
    │     └── 目标文件属于未来阶段 → WARN（提示当前阶段尚未完成）
    │
    └── 4. 输出校验结果
          ├── PASS → 静默通过
          └── WARN → 输出提示信息到 AI 上下文
```

**文件路径 → 阶段映射表**：

| 文件路径模式 | 归属阶段 | 说明 |
|-------------|---------|------|
| `specs/NNN-*/spec.md` | 01 Specify | 需求规格 |
| `specs/NNN-*/design.md` | 02 Design | 设计文档 |
| `specs/NNN-*/contracts/*.yaml` | 02 Design | API 契约 |
| `specs/NNN-*/data-model.md` | 02 Design | 数据模型 |
| `specs/NNN-*/tasks.md` | 03 Plan | 任务清单 |
| `specs/NNN-*/checklist.md` | 03 Plan | 实施 Checklist |
| `src/**/*`, `lib/**/*` | 04 Implement | 业务代码 |
| `specs/NNN-*/tests/*.test.md` | 05 Verify | 测试用例 |
| `specs/NNN-*/reports/*` | 05 Verify | 测试/安全报告 |
| `specs/NNN-*/retro.md` | 06 Wrap-up | 复盘文档 |
| `specs/NNN-*/release-note.md` | 06 Wrap-up | 发布说明 |

### 3.4 PostToolUse Hook 校验逻辑

**触发时机**：AI 执行写操作（`write_to_file` / `edit_file` / `create_file`）**之后**。

**校验流程**：

```
AI 写操作完成
    │
    ▼
hook-post-tool-use.mjs 接收上下文（toolName, filePath, fileContent）
    │
    ├── 1. 判断文件类型（Markdown / YAML / 代码）
    │
    ├── 2. ID 格式校验（仅 Markdown/YAML 产出物）
    │     ├── 提取文件中所有 ID（正则匹配）
    │     ├── 逐个调用 M2.validateId(id)
    │     └── 不合法 ID → 输出修正建议（正确格式 + 示例）
    │
    ├── 3. traces/verifies 引用完整性（仅 Markdown 产出物）
    │     ├── 提取 traces: / verifies: 声明中的引用 ID
    │     ├── 校验引用的 ID 是否已存在
    │     └── 悬空引用 → 输出可能的正确 ID（模糊匹配）
    │
    ├── 4. [AI-GENERATED] 标记检查（仅 AI 新建文件）
    │     ├── 新建文件 → 检查是否包含 [AI-GENERATED] 标记
    │     └── 缺失 → 提示 AI 在文件头部添加标记
    │
    └── 5. 输出校验结果
          ├── 全部通过 → 静默
          └── 有问题 → 输出修正建议列表，AI 可自动修正
```

**`[AI-GENERATED]` 标记规范**：

> **回应 M5 遗留问题 O6**：定义 AI 生成内容标记的插入位置。

| 文件类型 | 标记位置 | 标记格式 |
|---------|---------|---------|
| Markdown 产出物 | 文件头部元信息区（front-matter 之后，正文之前） | `> [AI-GENERATED] by <Agent> at <ISO8601>` |
| YAML 契约文件 | 文件顶部注释 | `# [AI-GENERATED] by <Agent> at <ISO8601>` |
| 代码文件 | 文件顶部注释 | `// [AI-GENERATED] by <Agent> at <ISO8601>` |
| 测试用例 | 文件头部元信息区 | `> [AI-GENERATED] by <Agent> at <ISO8601>` |

### 3.5 Stop Hook 校验逻辑

**触发时机**：AI 会话结束时（用户退出或 AI 主动结束）。

**校验流程**：

```
AI 会话即将结束
    │
    ▼
hook-stop.mjs 接收上下文（sessionId, featureId）
    │
    ├── 1. 三文件完成度校验
    │     ├── 检查 spec.md / tasks.md / checklist.md 是否存在
    │     ├── 检查各文件的关键区块是否已填充（非空）
    │     └── 输出完成度百分比（已填充区块 / 总区块）
    │
    ├── 2. 追踪矩阵同步检查
    │     ├── 本次会话新增的 ID 是否已录入追踪矩阵
    │     ├── 本次会话修改的产出物是否与矩阵一致
    │     └── 不同步项 → 输出待同步清单
    │
    ├── 3. AI 统计数据采集
    │     ├── 记录会话时长、操作次数、涉及文件数
    │     ├── 记录 Agent 类型、阶段、Feature ID
    │     └── 追加写入 .spec-first/runtime/ai-stats.jsonl
    │
    └── 4. 输出会话摘要
          ├── 本次会话完成的工作项
          ├── 未完成的待办项
          └── 下次会话建议的起始点
```

**AI 统计数据 JSONL 格式**：

```jsonl
{"ts":"2026-02-08T15:30:00Z","sid":"sess-abc123","feat":"001-user-auth","stage":"02_design","agent":"DesignArchitect","duration_s":1800,"ops":42,"files_touched":5,"ids_created":["DS-AUTH-001","DS-AUTH-002"],"completion_delta":{"design.md":0.3}}
```

| 字段 | 说明 |
|------|------|
| `ts` | 会话结束时间戳（ISO 8601） |
| `sid` | 会话唯一 ID |
| `feat` | Feature ID |
| `stage` | 当前阶段 |
| `agent` | Agent 类型（M5 路由结果） |
| `duration_s` | 会话时长（秒） |
| `ops` | 写操作次数 |
| `files_touched` | 涉及文件数 |
| `ids_created` | 本次会话新建的 ID 列表 |
| `completion_delta` | 各产出物完成度增量 |

### 3.6 专家会审记录

> **[Arch]**: `claude-hooks.json` 的 Schema 设计遵循「声明式配置」原则——配置文件只描述「何时触发」和「执行什么」，不包含校验逻辑本身。校验逻辑在 `.spec-first/scripts/` 下的 Node.js 脚本中实现，配置文件仅做路由。`on_failure` 字段的三种模式（warn/block/ignore）覆盖了所有需要的阻断层级，与 M3 定义的 Level 0-1 阻断策略对齐。
> **[DevEff]**: PreToolUse 的文件路径→阶段映射是 Layer A 的核心创新——AI 在写文件前就知道「这个文件属于哪个阶段」，避免了跨阶段误操作。但映射表需要与目录结构约定严格同步，建议映射规则从 `config.yaml` 动态加载而非硬编码，便于项目定制。PostToolUse 的 `[AI-GENERATED]` 标记对审计很重要——Code Review 时可以快速识别 AI 生成的内容。
> **[QA]**: Stop Hook 的 AI 统计数据采集是度量体系的重要数据源——M6 的 AI 协作效率指标（AI 辅助占比、会话效率）依赖这些数据。JSONL 格式的 `completion_delta` 字段支持增量统计，避免全量扫描。建议 Stop Hook 在采集统计数据时做数据校验（如 `duration_s > 0`、`feat` 非空），防止脏数据污染度量。
> **[决议]**: Schema 采用声明式配置。文件路径→阶段映射从 `config.yaml` 动态加载。`[AI-GENERATED]` 标记强制执行。Stop Hook 统计数据增加基础校验。

## 4. Layer B Hook 实现

### 4.1 commit-msg Hook 入口脚本

**文件**：`.spec-first/hooks/commit-msg.sh`

```bash
#!/usr/bin/env bash
# Spec-First commit-msg Hook v1.0.0
# 入口脚本：Shell 分发 → Node.js 校验

set -euo pipefail

HOOK_VERSION="1.0.0"
SCRIPT_DIR="$(cd "$(dirname "$0")/../scripts" && pwd)"
COMMIT_MSG_FILE="$1"

# --version 支持（安装验证用）
if [[ "${1:-}" == "--version" ]]; then
  echo "spec-first commit-msg hook v${HOOK_VERSION}"
  exit 0
fi

# 版本检查：源文件版本 vs 安装版本
if [[ -f "${SCRIPT_DIR}/../hooks/commit-msg.sh" ]]; then
  SOURCE_VERSION=$(grep '^# Spec-First commit-msg Hook v' \
    "${SCRIPT_DIR}/../hooks/commit-msg.sh" | head -1 | sed 's/.*v//')
  if [[ "${SOURCE_VERSION}" != "${HOOK_VERSION}" ]]; then
    echo "⚠️  Hook version mismatch: installed=${HOOK_VERSION}, source=${SOURCE_VERSION}"
    echo "   Run 'spec-first init' to upgrade."
  fi
fi

# 核心校验：调用 Node.js 脚本
if command -v node &>/dev/null; then
  node "${SCRIPT_DIR}/check-commit-msg.mjs" "$COMMIT_MSG_FILE" || {
    EXIT_CODE=$?
    if [[ $EXIT_CODE -eq 1 ]]; then
      # 校验失败 → 阻断提交
      exit 1
    else
      # 脚本异常（exit code > 1）→ 降级为警告
      echo "⚠️  commit-msg hook encountered an error (exit code: ${EXIT_CODE})"
      echo "   Commit allowed. Please check hook script."
      exit 0
    fi
  }
else
  echo "⚠️  Node.js not found. Skipping commit-msg validation."
  echo "   Install Node.js >= 20 for full Spec-First support."
  exit 0
fi
```

**关键设计**：
- Shell 做入口分发，Node.js 做复杂校验——Shell 启动快（< 50ms），Node.js 处理格式校验和 FEAT 缩写验证逻辑
- 异常降级：Node.js 脚本 exit code = 1 表示校验失败（阻断），> 1 表示脚本异常（降级放行）
- Node.js 不存在时降级为警告，不阻断开发者正常 Git 操作

### 4.2 commit-msg 格式校验逻辑（Conventional Commits）

**文件**：`.spec-first/scripts/check-commit-msg.mjs`

**格式规范**（Conventional Commits + FEAT scope）：

> commit-msg Hook 校验提交消息是否符合 Conventional Commits 格式，scope 为 FEAT 缩写。TASK ID 关联从 commit 级别移至 PR 级别（PR 描述中的 `Covers:` 区块），降低每次提交的认知负担。

**格式定义**：

```
<type>(<scope>): <description>

type  = feat | fix | docs | refactor | test | chore
scope = FEAT 缩写（如 ADDR、AUTH），必须已注册
```

**示例对比**：

| 优化前（阶段感知标签） | 优化后（Conventional Commits） |
|---------------------|------------------------------|
| `[SPEC-ADDR] 完成需求规格` | `docs(ADDR): 完成需求规格` |
| `[DESIGN-ADDR] 完成技术设计` | `docs(ADDR): 完成技术设计与 API 契约` |
| `[IMPL-ADDR] TASK-ADDR-001 创建表结构` | `feat(ADDR): 创建 address 表结构` |
| `[VERIFY-ADDR] 完成测试用例` | `test(ADDR): 完成地址管理测试用例` |

**TASK ID 关联方式**：在 PR 描述中使用 `Covers:` 区块关联 TASK ID：

```markdown
## Covers
- TASK-ADDR-001 数据库表结构创建
- TASK-ADDR-002 地址 CRUD 接口实现
- TASK-ADDR-003 默认地址逻辑
```

**校验算法**：

```
function checkCommitMsg(commitMsgFile):
  msg = readFile(commitMsgFile)

  // Step 1: 校验 Conventional Commits 格式
  ccMatch = msg.match(/^(feat|fix|docs|refactor|test|chore)\(([A-Z][A-Z0-9]{1,15})\): .+/)
  if !ccMatch:
    return FAIL("格式错误。期望: <type>(<FEAT>): <description>")

  type = ccMatch[1]        // feat | fix | docs | ...
  featAbbr = ccMatch[2]    // FEAT 缩写

  // Step 2: 校验 FEAT 缩写已注册
  if !M2.validateFeatAbbr(featAbbr).valid:
    return FAIL("FEAT 缩写 '{featAbbr}' 未注册。已注册: {registeredList}")

  // Step 3: 校验 description 非空且有意义
  description = msg.substring(ccMatch[0].indexOf(": ") + 2)
  if description.length < 5:
    return FAIL("描述过短，至少 5 个字符")

  return PASS
```

### 4.3 pre-push Hook 实现

**文件**：`.spec-first/hooks/pre-push.sh`（入口）+ `.spec-first/scripts/check-sca.mjs`（校验逻辑）

**触发时机**：`git push` 执行前。

**校验内容**：增量 Spec-Consistency-Analysis（SCA），基于 `git diff` 仅校验本次推送涉及的产出物。

**校验流程**：

```
git push 触发
    │
    ▼
pre-push.sh 入口脚本
    │
    ├── 1. 获取待推送的 commit 范围
    │     └── git log --oneline origin/<branch>..HEAD
    │
    ├── 2. 获取变更文件列表
    │     └── git diff --name-only origin/<branch>..HEAD
    │
    ├── 3. 过滤 Spec-First 管控文件
    │     ├── specs/**/* → 纳入 SCA 校验
    │     ├── src/**/* → 检查 traces 注释
    │     └── 其他文件 → 跳过
    │
    ├── 4. 调用 check-sca.mjs 执行增量 SCA
    │     ├── 提取变更文件中的 ID 集合
    │     ├── 构建局部 ID 引用图
    │     ├── 检测断裂的追踪链
    │     └── 检测新增的孤儿项
    │
    └── 5. 输出校验结果
          ├── PASS → 允许 push
          ├── FAIL → 阻断 push，输出不一致项清单
          └── 异常 → 降级为警告，允许 push
```

**超时控制**：pre-push Hook 超时限制 10s。超时时降级为警告放行，CI Pipeline 做全量兜底。

**增量 SCA 与全量 SCA 的差异**：

| 维度 | 增量 SCA（pre-push） | 全量 SCA（CI Pipeline） |
|------|---------------------|----------------------|
| 范围 | 仅本次推送涉及的文件 | 整个 Feature 的全部产出物 |
| 精度 | 可能遗漏间接影响 | 完整覆盖 |
| 耗时 | < 10s | < 120s |
| 阻断 | 超时降级放行 | 超时标记 FAIL |
| 定位 | 开发者快速反馈 | PR 合并最终防线 |

### 4.4 多 Feature 混合提交处理

> **回应 M3 遗留问题 O7**：定义多 Feature 混合提交时 featureId 的推断逻辑。

```
function inferFeatureId(stagedFiles):
  featureIds = Set()
  for file in stagedFiles:
    match = file.match(/^specs\/(\d{3}-[a-z0-9-]+)\//)
    if match:
      featureIds.add(match[1])

  if featureIds.size == 0:
    // 非 specs/ 目录的提交（如纯代码提交）
    // 从 commit message 的 scope（FEAT 缩写）推断
    return inferFromCommitScope(commitMsg)

  if featureIds.size == 1:
    return featureIds.first()

  if featureIds.size > 1:
    // 多 Feature 混合提交 → 拒绝，提示拆分
    return FAIL("本次提交涉及多个 Feature: {featureIds}。请拆分为独立提交。")
```

**规则**：一次 commit 只允许涉及一个 Feature 的产出物文件。跨 Feature 的提交被拒绝并提示拆分。纯代码提交（不涉及 `specs/` 目录）通过 commit message 标签推断 Feature。

### 4.5 专家会审记录

> **[Arch]**: Layer B 的 Shell + Node.js 分层设计是正确的——Shell 做入口保证启动速度（< 50ms），Node.js 做复杂逻辑保证可维护性。exit code 约定（1=校验失败，>1=脚本异常）是 Unix 惯例的合理扩展，确保异常不阻断开发。pre-push 的增量 SCA 是性能与精度的权衡——已知可能遗漏间接影响，CI 全量 SCA 兜底。
> **[DevEff]**: commit-msg 采用 Conventional Commits 格式（`feat(ADDR): ...`），降低每次提交的认知负担。TASK ID 关联从 commit 级别移至 PR 级别（`Covers:` 区块），更符合实际开发节奏。多 Feature 混合提交的拒绝策略可能在大型重构时造成不便，建议增加 `--allow-multi-feature` 临时旗标（仅限 Tech Lead 使用），但 CI 仍做严格校验。
> **[QA]**: `--no-verify` 绕过事件的度量采集很重要（M3 遗留问题 O6）。建议在 CI Pipeline 中检测：如果 commit 没有经过 commit-msg Hook（通过检查 commit message 格式的一致性间接判断），记录为「疑似绕过」事件，纳入度量报告。pre-push 增量 SCA 的精度需要在集成测试中验证——对比增量结果与全量结果的偏差。
> **[决议]**: Shell + Node.js 分层确认。exit code 约定 1=FAIL/>1=异常。多 Feature 混合提交默认拒绝，预留 `--allow-multi-feature` 旗标。CI 检测疑似绕过事件。

## 5. CI Pipeline 配置

### 5.1 Pipeline 总览

> **回应 M3 遗留问题 O3**：定义 CI Pipeline 的 GitHub Actions YAML 配置模板。

Spec-First 提供两个 GitHub Actions Workflow：

| Workflow | 文件 | 触发时机 | 职责 |
|----------|------|---------|------|
| **Gate 校验** | `.spec-first/ci/spec-first-gate.yml` | PR 创建/更新 | 全量 Gate 校验 + SCA + 覆盖率，Block PR merge |
| **度量报告** | `.spec-first/ci/spec-first-report.yml` | PR 合并到主分支 | 度量采集 + 报告生成 + PR Comment |

### 5.2 Gate 校验 Workflow

```yaml
# .spec-first/ci/spec-first-gate.yml
name: Spec-First Gate Check

on:
  pull_request:
    types: [opened, synchronize, reopened]
    paths:
      - 'specs/**'
      - 'src/**'
      - '.spec-first/**'

permissions:
  contents: read
  pull-requests: write          # PR Comment 权限

jobs:
  gate-check:
    runs-on: ubuntu-latest
    timeout-minutes: 5
    steps:
      - uses: actions/checkout@v4
        with:
          fetch-depth: 0        # 全量历史，支持 diff 比较

      - uses: actions/setup-node@v4
        with:
          node-version: '20'

      - name: Detect Feature ID
        id: detect
        run: |
          # 从 PR 变更文件推断 Feature ID
          FEAT_ID=$(git diff --name-only origin/${{ github.base_ref }}...HEAD \
            | grep '^specs/' | head -1 | cut -d'/' -f2)
          echo "feat_id=${FEAT_ID}" >> $GITHUB_OUTPUT

      - name: Run Full SCA
        run: node .spec-first/scripts/check-sca.mjs --mode=full --feature=${{ steps.detect.outputs.feat_id }}

      - name: Run Coverage Check
        run: node .spec-first/scripts/calc-coverage.mjs --feature=${{ steps.detect.outputs.feat_id }}

      - name: Run Gate Evaluation
        run: node .spec-first/scripts/gate-evaluator.mjs --feature=${{ steps.detect.outputs.feat_id }} --output=json > gate-result.json

      - name: Post Gate Report to PR
        if: always()
        uses: actions/github-script@v7
        with:
          script: |
            const fs = require('fs');
            const result = JSON.parse(fs.readFileSync('gate-result.json', 'utf8'));
            const body = formatGateReport(result);  // 格式化为 Markdown
            await github.rest.issues.createComment({
              owner: context.repo.owner,
              repo: context.repo.repo,
              issue_number: context.issue.number,
              body: body
            });
```

**关键设计**：
- `fetch-depth: 0` 获取全量 Git 历史，支持 `git diff` 比较
- `paths` 过滤：仅 `specs/`、`src/`、`.spec-first/` 变更时触发，避免无关 PR 浪费 CI 资源
- `if: always()` 确保 Gate 报告即使在校验失败时也能输出到 PR Comment
- Gate 结果以 JSON 格式输出，便于后续步骤解析和格式化

### 5.3 度量报告 Workflow

```yaml
# .spec-first/ci/spec-first-report.yml

  pull_request:

permissions:
  contents: read

jobs:
  metrics-report:

      - uses: actions/setup-node@v4

      - name: Detect Feature ID
```

### 5.4 专家会审记录

> **[Arch]**: CI Pipeline 的核心定位是「最终防线」——即使开发者绕过本地 Hook（`--no-verify`），CI 仍会拦截不合规提交。Gate 校验 Workflow 的 `paths` 过滤是性能优化的关键——非 Spec-First 管控文件的 PR 不触发校验，避免 CI 资源浪费。两个 Workflow 分离（Gate 校验 vs 度量报告）是因为触发时机不同：Gate 在 PR 创建时，度量在 PR 合并后。
> **[DevEff]**: PR Comment 报告是开发者与 CI 交互的主要界面——报告必须包含「当前状态 + 未覆盖项 + 修复建议」三部分，让开发者一眼看到问题和解决方案。`Suggested Actions` 输出可执行的 CLI 命令（如 `spec-id next TC AUTH`），降低修复成本。建议报告支持折叠长列表（使用 GitHub `<details>` 标签），避免 PR Comment 过长。
> **[QA]**: CI Pipeline 超时设为 5 分钟（Gate 校验）和 3 分钟（度量报告），对 L 规模 Feature（100+ FR）可能不够。建议增加 `timeout-minutes` 的可配置性，通过 `config.yaml` 的 `ci.gate_timeout_minutes` 字段控制。CI 校验失败时应自动设置 PR 为 `changes_requested` 状态，防止意外合并。
> **[决议]**: 两个 Workflow 分离确认。PR Comment 报告包含三段式内容。超时可通过 `config.yaml` 配置。CI 校验失败自动 Block PR merge（通过 GitHub Branch Protection Rules）。

## 6. Jira 集成设计

### 6.1 集成架构与成熟度模型

> **回应 M4 遗留问题 O3 和 M6 遗留问题 O3**：定义 Jira REST API 认证方式、调用频率和字段映射。

Jira 集成按团队成熟度分三级，渐进式落地：

| 级别 | 方式 | 适用阶段 | 实现路径 |
|------|------|---------|---------|
| L1 | **手动同步** | MVP 阶段 | Gate 通过后由 PM 手动流转 Jira Status |
| L2 | **半自动（Webhook）** | 增强阶段 | Gate 通过 → CI 调用 Jira REST API 自动流转，失败降级为手动 |
| L3 | **全自动（双向同步）** | 全量阶段 | Jira Automation Rule 监听 Status → 触发 Gate；Gate 通过 → 回写 Status |

**MVP 阶段（L1）不需要任何 Jira API 集成**——仅通过文档约定 Jira Status 与阶段的映射关系，PM 手动操作。

### 6.2 Jira Status 映射表

| Spec-First 阶段 | Jira Status | 触发条件 | Transition API |
|-----------------|------------|---------|---------------|
| 00 Init | `INIT` | Feature 创建 | `POST /transitions` → INIT |
| 01 Specify | `SPECIFYING` | CLI 内联校验通过 | `POST /transitions` → SPECIFYING |
| 02 Design | `DESIGNING` | DoR Sign-off | `POST /transitions` → DESIGNING |
| 03 Plan | `PLANNING` | Design Review 通过 | `POST /transitions` → PLANNING |
| 04 Implement | `IN_PROGRESS` | Task Review 通过 | `POST /transitions` → IN_PROGRESS |
| 05 Verify | `IN_TESTING` | Code CR 通过 | `POST /transitions` → IN_TESTING |
| 06 Wrap-up | `WRAPPING_UP` | UAT Sign-off | `POST /transitions` → WRAPPING_UP |
| 07 Release | `RELEASING` | 归档清单通过 | `POST /transitions` → RELEASING |
| 08 Done | `DONE` | 发布成功 + 观察窗口通过 | `POST /transitions` → DONE |

**Jira Workflow 前置要求**：Jira 项目需预配置上述 9 个 Status 及合法流转路径，与 Spec-First 阶段状态机一一对应。

### 6.3 Jira REST API 认证与调用

**认证方式**：

| 方式 | 适用场景 | 配置位置 |
|------|---------|---------|
| **API Token（推荐）** | Jira Cloud | 环境变量 `JIRA_API_TOKEN` 注入，不存储在配置文件中 |
| **OAuth 2.0** | 企业级 Jira Server | `config.yaml` → `jira.auth.oauth`（仅存 client_id，secret 走环境变量） |
| **Personal Access Token** | Jira Data Center | 环境变量 `JIRA_PAT` 注入，不存储在配置文件中 |

> **安全原则**：凭证（Token、Secret、PAT）永远不落地到版本控制文件中。CI 环境通过 GitHub Secrets / GitLab CI Variables 注入，本地环境通过 `.env` 文件（已在 `.gitignore` 中）或 shell profile 注入。

**`config.yaml` Jira 配置节**：

```yaml
# .spec-first/config.yaml — Jira 集成配置
jira:
  enabled: false                    # 默认关闭，按需开启
  maturity: "L1"                   # L1=手动 / L2=半自动 / L3=全自动
  base_url: "https://your-org.atlassian.net"
  project_key: "PROJ"
  auth:
    method: "api_token"            # api_token / oauth2 / pat
    # Token 通过环境变量注入，不存储在配置文件中
    token_env: "JIRA_API_TOKEN"
    email_env: "JIRA_USER_EMAIL"
  rate_limit:
    max_requests_per_minute: 30    # Jira Cloud 默认限制
    retry_on_429: true             # 429 Too Many Requests 时自动重试
    retry_delay_ms: 2000           # 重试间隔
  field_mapping:                   # Jira 自定义字段映射
    feature_id: "customfield_10001"
    spec_first_stage: "customfield_10002"
    coverage_rate: "customfield_10003"
```

**API 调用封装**：

```
function transitionJiraIssue(issueKey, targetStatus):
  // Step 1: 获取可用 Transitions
  transitions = GET /rest/api/3/issue/{issueKey}/transitions
  targetTransition = transitions.find(t => t.to.name == targetStatus)

  if !targetTransition:
    return FAIL("Jira Status '{targetStatus}' 不可达。当前可用: {availableStatuses}")

  // Step 2: 执行 Transition
  POST /rest/api/3/issue/{issueKey}/transitions
    body: { transition: { id: targetTransition.id } }

  // Step 3: 更新自定义字段
  PUT /rest/api/3/issue/{issueKey}
    body: { fields: { [field_mapping.spec_first_stage]: targetStatus } }

  return SUCCESS
```

**失败处理**：API 调用失败时记录到 `findings.md`，**不阻塞开发流程**。Jira 同步是辅助功能，不是 Gate 条件。

### 6.4 缺陷数据双向同步

**同步方向**：

| 方向 | 触发条件 | 同步内容 |
|------|---------|---------|
| **Spec-First → Jira** | M4 缺陷管理创建缺陷记录 | 缺陷 ID、严重级别、关联 FR/TC、发现阶段 |
| **Jira → Spec-First** | Jira Bug 状态变更（Resolved/Closed） | 修复状态、修复 PR、验证结果 |

**Jira Bug 字段映射**：

| Spec-First 字段 | Jira 字段 | 说明 |
|-----------------|----------|------|
| `defectId` | `summary` 前缀 | 如 `[BUG-AUTH-001] 登录异常` |
| `severity` | `priority` | Critical→Highest, Major→High, Minor→Medium |
| `relatedFR` | `customfield_10004` | 关联的 FR ID |
| `relatedTC` | `customfield_10005` | 发现缺陷的 TC ID |
| `foundInStage` | `customfield_10006` | 发现阶段 |
| `fixPR` | `customfield_10007` | 修复 PR 链接 |

**同步流程（L2 半自动）**：

```
缺陷创建（M4 → Jira）：
  1. M4.createDefect() 创建本地缺陷记录
  2. 调用 Jira API 创建 Bug Issue
  3. 将 Jira Issue Key 回写到本地缺陷记录
  4. 失败时记录到 findings.md，不阻塞

缺陷关闭（Jira → Spec-First）：
  1. CI Pipeline 定期轮询（或 Webhook 触发）
  2. 检测 Jira Bug 状态变更为 Resolved/Closed
  3. 更新本地缺陷记录状态
  4. 触发回归验证提醒
```

### 6.5 专家会审记录

> **[Arch]**: Jira 集成的核心原则是「不阻塞开发流程」——Jira 同步失败不应成为 Gate 条件，不应阻断提交或推送。Jira 是项目管理工具，不是质量防线。三级成熟度模型（L1→L2→L3）确保团队可以渐进式采用，MVP 阶段零 API 依赖。Token 通过环境变量注入而非存储在 `config.yaml` 中，是安全最佳实践。
> **[DevEff]**: L2 半自动模式是大多数团队的最佳平衡点——Gate 通过后自动流转 Jira Status，开发者无需手动操作。但 Jira API 的 Rate Limit（Cloud 默认 30 req/min）在大团队并发场景下可能成为瓶颈，建议增加请求队列和批量处理机制。缺陷同步的 `customfield` 映射需要在 Jira 管理员配合下完成，建议提供 `spec-first jira setup` 命令自动检测和配置自定义字段。
> **[QA]**: 缺陷数据双向同步对质量度量至关重要——M6 的缺陷密度、逃逸率等指标依赖 Jira 中的缺陷数据。Jira → Spec-First 方向的同步（缺陷关闭）需要可靠的触发机制——Webhook 优于轮询（实时性更好），但 Webhook 配置需要 Jira 管理员权限。建议 L2 阶段用轮询（CI Pipeline 定期执行），L3 阶段升级为 Webhook。
> **[决议]**: Jira 同步不作为 Gate 条件。三级成熟度渐进落地。Token 环境变量注入。L2 用轮询，L3 升级 Webhook。增加 `spec-first jira setup` 命令。

**`spec-id next` 内部算法**：

```
function specIdNext(type, featAbbr, options):
  // Step 1: 校验 FEAT 缩写已注册
  featResult = M2.validateFeatAbbr(featAbbr)
  if !featResult.valid:
    exit(FAIL, "FEAT '{featAbbr}' 未注册。已注册: {registeredList}")

  // Step 2: 扫描已有 ID，获取最大序号
  featureDir = findFeatureDir(featAbbr)
  existingIds = M2.extractIds(featureDir, type)
  maxSeq = max(existingIds.map(id => id.sequence)) || 0

  // Step 3: 生成下一个 ID
  nextSeq = maxSeq + 1
  nextId = formatId(type, featAbbr, nextSeq, options)

  // Step 4: 唯一性校验
  if M2.validateId(nextId).errors.includes("DUPLICATE"):
    nextSeq += 1  // 跳过冲突序号
    nextId = formatId(type, featAbbr, nextSeq, options)

  // Step 5: 输出
  if options.quiet:
    print(nextId)
  else:
    printFormatted(nextId, type, featAbbr, maxSeq, nextSeq)
```

### 7.8 专家会审记录

> **[Arch]**: CLI 命令设计遵循「单一职责」原则——每个子命令对应一个功能域，不混合职责。`spec-id next` 的内部算法保证了 ID 序号的单调递增和唯一性，即使在并发场景下（两个开发者同时执行 `spec-id next`），通过文件锁或 Git 提交冲突检测也能避免重复。`spec-matrix trace` 的追踪链查询是 Impact Analysis 的基础能力，M4 ChangeMgr 依赖此接口。
> **[DevEff]**: CLI 的交互体验是开发者采纳度的关键。三个设计要点：(1) 所有命令支持 `--quiet` 和 `--output=json`，满足人机两种场景；(2) 错误输出包含可执行的修复命令（如 `spec-id next TC AUTH --level E2E`），降低修复成本；(3) `spec-first doctor` 是新成员 onboarding 的第一个命令，必须覆盖所有环境依赖。建议增加 `spec-first completion` 命令生成 Shell 自动补全脚本（bash/zsh/fish）。
> **[QA]**: `spec-gate check --fix-hints` 的修复建议质量直接影响 Gate 首次通过率——建议越精确，开发者修复越快。`spec-metrics coverage --uncovered` 输出的未覆盖项清单是 QA 追踪的重要输入。建议所有 CLI 命令的退出码遵循统一约定：0=成功，1=校验失败，2=参数错误，3=环境错误，便于脚本集成。
> **[决议]**: CLI 命令按功能域划分确认。退出码统一约定（0/1/2/3）。增加 Shell 自动补全支持。`--fix-hints` 输出可执行修复命令。

## 10. 错误处理与边界场景

### 10.1 错误分类与处理策略

M7 作为执行层胶水，错误来源涵盖 Hook 安装、脚本执行、CI Pipeline、外部 API 四大领域：

| 错误类别 | 触发场景 | 严重度 | 处理策略 | 用户提示 |
|---------|---------|--------|---------|---------|
| **Hook 安装失败** | `.git/hooks/` 目录不存在或无写权限 | 🔴 阻断 | Fail-Fast，输出修复命令 | `git init` 或 `chmod` 命令 |
| **Hook 脚本不存在** | `.spec-first/hooks/*.sh` 文件缺失 | 🔴 阻断 | 提示重新初始化 | `spec-first init --repair` |
| **Symlink 创建失败** | Windows 无 symlink 权限 | 🟡 降级 | 降级为文件复制模式 | 提示启用开发者模式或使用 WSL2 |
| **Node.js 版本不兼容** | Node.js < 20 或未安装 | 🔴 阻断 | Fail-Fast，输出版本要求 | `nvm install 20` 命令 |
| **校验脚本执行超时** | 脚本执行超过配置的超时时间 | 🟡 降级 | 终止脚本，返回 TIMEOUT 退出码 | 提示检查脚本或增大超时配置 |
| **校验脚本运行时错误** | 脚本内部异常（语法错误、依赖缺失） | 🔴 阻断 | 捕获 stderr，输出错误摘要 | 输出错误行号 + 修复建议 |
| **CI Pipeline 配置错误** | YAML 语法错误或 Action 版本不兼容 | 🔴 阻断 | CI 平台报错，本地 `doctor` 可预检 | `spec-first doctor` 检查 CI 配置 |
| **Jira API 认证失败** | Token 过期或权限不足 | 🟡 降级 | 跳过 Jira 同步，记录警告 | 提示重新配置 `spec-first jira setup` |
| **Jira API 限流** | 请求频率超过 Jira 限制 | 🟡 降级 | 指数退避重试（最多 3 次） | 提示稍后重试或调整 `rateLimit` 配置 |
| **Jira API 网络超时** | 网络不可达或响应超时 | 🟡 降级 | 跳过 Jira 同步，不阻断主流程 | 提示检查网络或 Jira 服务状态 |
| **Agent 注册表格式错误** | constitution.md 中 Agent 定义不合法 | 🟡 警告 | 使用默认 Agent 路由，输出警告 | `spec-first doctor` 检查注册表 |
| **Layer A 配置错误** | `claude-hooks.json` JSON 语法错误 | 🟡 降级 | Layer A 失效，Layer B 独立运行 | 提示修复 JSON 语法 |

### 10.2 错误处理原则

M7 的错误处理遵循三条核心原则：

```
原则 1: 主流程不可阻断（Jira/Layer A 故障不影响 Git 操作）
  ├── Jira API 失败 → 跳过同步，记录警告，主流程继续
  ├── Layer A Hook 失败 → Layer B 独立运行，不受影响
  └── Agent 注册表异常 → 使用默认路由，输出警告

原则 2: 校验类错误必须阻断（不合规产物不可进入下一阶段）
  ├── commit-msg 校验失败 → 阻断 commit，退出码 1
  ├── pre-push 校验失败 → 阻断 push，退出码 1
  └── CI Gate 校验失败 → 阻断 PR merge

原则 3: 错误提示必须可操作（What-Why-How 三段式）
  ├── What: 发生了什么错误
  ├── Why: 为什么会发生
  └── How: 如何修复（给出具体命令或操作步骤）
```

**错误输出格式标准**：

```
✗ [ERROR_CODE] 错误简述
  原因: 具体原因说明
  修复: 具体修复命令或操作步骤
  文档: 相关文档链接（可选）
```

**示例**：

```
✗ [HOOK_NOT_INSTALLED] commit-msg Hook 未安装
  原因: .git/hooks/commit-msg 文件不存在，可能未执行初始化
  修复: 运行 spec-first init --repair 重新安装 Hook
```

### 10.3 边界场景处理

#### 场景 1：Hook 被第三方工具覆盖

```
触发：开发者安装 Husky、lefthook 等工具，覆盖了 .git/hooks/ 下的 Hook 文件
检测：spec-first doctor 检查 Hook 文件内容是否包含 spec-first 标识
影响：commit-msg / pre-push 校验失效，不合规提交可进入仓库
处理：
  1. doctor 检测到 Hook 被覆盖时，输出警告 + 修复命令
  2. 提供两种修复方案：
     a. 独占模式：spec-first init --repair 重新安装（覆盖第三方 Hook）
     b. 共存模式：在第三方 Hook 中追加 spec-first 调用
        示例（Husky .husky/commit-msg）：
        #!/bin/sh
        # Husky 原有逻辑
        npx commitlint --edit $1
        # Spec-First 追加
        .spec-first/hooks/commit-msg.sh "$1"
  3. CI Pipeline 作为兜底防线，即使本地 Hook 失效也能拦截
```

#### 场景 2：Windows 环境 Symlink 限制

```
触发：Windows 默认不允许非管理员创建 symlink
影响：Hook 安装的 symlink 模式失败
处理：
  1. 检测操作系统，Windows 环境自动降级为文件复制模式
  2. 复制模式下，Hook 文件直接复制到 .git/hooks/（非 symlink）
  3. 复制模式的代价：.spec-first/hooks/ 更新后需手动重新安装
     （symlink 模式下自动生效）
  4. spec-first doctor 在 Windows 下额外检查：
     - 复制模式的 Hook 文件是否与源文件一致（MD5 对比）
     - 不一致时提示 spec-first init --repair
  5. 推荐方案：提示开发者使用 WSL2 或启用 Windows 开发者模式
     （开发者模式允许非管理员创建 symlink）
```

#### 场景 3：CI Pipeline 与本地 Hook 校验结果不一致

```
触发：本地 Hook 通过但 CI 失败，或本地 Hook 失败但 CI 通过
原因分析：
  - Node.js 版本差异（本地 v20，CI v22）
  - 依赖版本差异（本地 npm install 后未提交 lock 文件）
  - 文件编码差异（Windows CRLF vs Linux LF）
  - 增量 vs 全量校验差异（本地增量，CI 全量）
处理：
  1. spec-first doctor 输出本地环境指纹（Node 版本、npm 版本、OS）
  2. CI Pipeline 在输出中打印环境指纹，便于对比
  3. 校验脚本内部记录 --verbose 日志，输出每步判定依据
  4. 以 CI 结果为准——CI 是最终防线，本地 Hook 是辅助
  5. 建议团队统一 .nvmrc 文件锁定 Node.js 版本
```

#### 场景 4：Jira 字段映射不匹配

```
触发：不同 Jira 项目的字段 Schema 不同，自动检测的映射不准确
影响：状态同步失败或写入错误字段
处理：
  1. spec-first jira setup 的 detectJiraFields 自动检测可用字段
  2. 检测结果展示给用户确认，不自动写入配置
  3. 用户确认后写入 config.yaml 的 jira.fieldMapping 节
  4. 首次同步时做 dry-run（--dry-run），预览变更不实际执行
  5. 同步失败时记录失败详情到 .spec-first/runtime/jira-sync.log
  6. 提供 spec-first jira test 命令验证映射正确性
```

#### 场景 5：代码注释中 traces 声明的多风格解析

```
触发：不同语言的代码注释风格不同，需统一解析 traces/implements 声明
  （回应 M2 O6：代码注释中 traces 声明的解析规则）
支持的注释风格：
  - 单行注释（//）：JavaScript, TypeScript, Java, Go, C/C++, Rust
    // implements: TASK-AUTH-001, traces: FR-AUTH-001
  - 井号注释（#）：Python, Ruby, Shell, YAML
    # implements: TASK-AUTH-001, traces: FR-AUTH-001
  - 块注释（/* */）：JavaScript, Java, CSS
    /* implements: TASK-AUTH-001, traces: FR-AUTH-001 */
  - 文档注释（/** */）：JSDoc, JavaDoc
    /** @traces FR-AUTH-001 */
解析正则：
  /(implements|traces|verifies)\s*:\s*((?:[A-Z]+-[A-Z0-9]+-\d{3}(?:,\s*)?)+)/g
处理：
  1. 按文件扩展名选择注释风格（.js → //, .py → #）
  2. 先剥离注释标记，再用统一正则提取 ID
  3. 支持单行多 ID（逗号分隔）和多行分别声明
  4. 忽略字符串字面量中的疑似 ID（避免误提取）
  5. 提取结果用于 PR 合规率（C8）和孤儿项检测（C9）
```

#### 场景 6：Layer A 与 Layer B 同时触发的去重

```
触发：Claude Code 执行 git commit 时，Layer A（PostToolUse）和
      Layer B（commit-msg Hook）可能对同一产出物重复校验
影响：校验执行两次，浪费时间；两次结果可能不一致
处理：
  1. Layer A 校验通过后，在环境变量中设置标记：
     export SPEC_FIRST_LAYER_A_CHECKED=1
  2. Layer B commit-msg.sh 入口检测该环境变量：
     if [ "$SPEC_FIRST_LAYER_A_CHECKED" = "1" ]; then
       # Layer A 已校验，Layer B 仅做轻量复核（ID 格式 + 阶段标签）
       node .spec-first/scripts/check-commit-msg.mjs "$1" --lite
     else
       # 纯人工场景，Layer B 做完整校验
       node .spec-first/scripts/check-commit-msg.mjs "$1" --full
     fi
  3. --lite 模式：仅校验 commit message 格式，跳过产出物内容校验
  4. --full 模式：完整校验（格式 + FEAT 缩写验证 + ID 引用合法性）
  5. 去重仅在同一次 commit 操作中生效，环境变量不持久化
```

### 10.4 专家会审记录

> **[Arch]**: 错误处理的分层策略正确——校验类错误（关乎质量）必须阻断，集成类错误（Jira/Layer A）允许降级。场景 6 的 Layer A/B 去重机制通过环境变量实现，轻量且无副作用。场景 5 的代码注释解析回应了 M2 O6 的遗留问题，统一正则 + 按扩展名选择注释风格是务实的方案。需要注意正则的性能——大文件（10000+ 行）中逐行匹配的开销需控制在 500ms 以内。
> **[DevEff]**: 场景 1（Hook 被覆盖）是实际中最常见的问题——团队中有人用 Husky，有人用 Spec-First，两者冲突。共存模式的方案（在 Husky Hook 中追加 Spec-First 调用）是最佳实践，但需要在文档中给出具体的集成示例。场景 3（CI 与本地不一致）的根因是环境差异，`.nvmrc` + `package-lock.json` 提交是基本要求，建议 `spec-first doctor` 检查这两个文件是否存在。
> **[QA]**: 场景 5 的注释解析需要覆盖测试——至少覆盖 JS/TS（`//`）、Python（`#`）、Java（`/* */`）三种风格，以及单行多 ID、多行分别声明、字符串字面量误匹配三种边界。场景 6 的去重机制需要集成测试验证——模拟 Claude Code 触发 commit 时 Layer A 和 Layer B 的执行顺序和环境变量传递。
> **[决议]**: 错误处理遵循「校验阻断 + 集成降级」分层策略。代码注释解析支持三种注释风格，性能控制在 500ms 以内。Layer A/B 去重通过环境变量实现。`spec-first doctor` 增加 `.nvmrc` 和 `package-lock.json` 存在性检查。

## 3. 待决策选型

### 3.1 类型系统：TypeScript 5.x (strict)

**建议**：TypeScript 5.x，启用 `strict: true`。

| 维度 | TypeScript | 纯 JavaScript (JSDoc) |
|------|-----------|----------------------|
| 类型安全 | 编译期强制，接口/枚举原生支持 | 运行时才暴露，JSDoc 注解冗长 |
| 设计文档对齐 | 01~07 设计文档全部使用 TS interface 定义 | 需手动翻译，易漂移 |
| IDE 体验 | 自动补全、重构、跳转完整 | 部分支持 |
| 构建成本 | 需编译步骤 | 零构建 |
| 生态兼容 | remark/unified/js-yaml 均有 @types | — |

**决策理由**：

1. 设计文档（01-流程引擎 §7、02-追踪体系 §6、03-质量门禁 §5）已定义 40+ 个 TypeScript interface，直接复用可避免翻译漂移
2. 7 个子系统间接口契约复杂（M1↔M3 Gate 调用、M2 覆盖率查询），类型系统是接口正确性的编译期保障
3. Node.js 20 LTS 原生支持 `--experimental-strip-types`（Node 22+ 正式支持），但考虑稳定性仍建议走 tsup 编译

**TypeScript 配置要点**：

```jsonc
// tsconfig.json 关键配置
{
  "compilerOptions": {
    "target": "ES2022",
    "module": "NodeNext",
    "moduleResolution": "NodeNext",
    "strict": true,
    "outDir": "dist",
    "rootDir": "src",
    "declaration": true,
    "sourceMap": true,
    "esModuleInterop": true,
    "skipLibCheck": true
  }
}
```

**结论**：✅ 采用 TypeScript 5.x (strict)。

### 3.2 CLI 框架：Commander.js

**候选对比**：

| 维度 | Commander.js | yargs | oclif |
|------|-------------|-------|-------|
| 包体积 | 52 KB | 280 KB | 2+ MB（含框架） |
| 学习曲线 | 低 | 中 | 高 |
| 子命令支持 | 原生 `.command()` | `.command()` | 插件式 |
| TypeScript | 内置类型 | 内置类型 | 原生 TS |
| 周下载量 | 1.5 亿+ | 8000 万+ | 50 万+ |
| 适用场景 | 中小型 CLI | 复杂参数解析 | 大型插件化 CLI |

**决策理由**：

1. Spec-First CLI 命令结构清晰（`spec-first <domain> <action>`），Commander 的子命令模式天然匹配
2. 轻量无依赖，符合"零服务进程"理念
3. 社区最广泛，团队上手成本最低

**命令结构映射**：

```
spec-first
├── init                    # 项目初始化
├── feature
│   └── init                # 启动 Feature
├── stage
│   └── current <id>        # 查看当前阶段
├── gate
│   └── request <id> --to   # 请求阶段转换
├── coverage <id>           # 覆盖率报告
├── trace <id>              # 追踪链查询
├── matrix
│   └── show <id>           # 追踪矩阵
├── doctor                  # 环境检查
└── report <id>             # 度量报告
```

**结论**：✅ 采用 Commander.js。

### 3.3 测试框架：Vitest

**候选对比**：

| 维度 | Vitest | Jest | node:test (内置) |
|------|--------|------|-----------------|
| ESM 支持 | 原生，零配置 | 需 `--experimental-vm-modules` | 原生 |
| TypeScript | 原生（esbuild 转译） | 需 ts-jest 或 @swc/jest | 需手动配置 |
| 执行速度 | 快（esbuild + 并行） | 中等 | 快 |
| Watch 模式 | 内置，HMR 级响应 | 内置 | 无 |
| 覆盖率 | 内置 `--coverage` (v8/istanbul) | 内置 | 需 c8 |
| 断言库 | 内置 (chai 兼容) | 内置 (expect) | 内置 (assert) |
| Mock | 内置 `vi.mock()` | 内置 `jest.mock()` | 需手动 |
| 生态成熟度 | 高（Vite 生态） | 最高 | 低 |

**决策理由**：

1. ESM 原生支持——Spec-First 全量 ESM，Jest 的 ESM 支持仍标记 experimental
2. TypeScript 零配置——无需 ts-jest 等额外转译层
3. API 与 Jest 高度兼容（`describe/it/expect`），团队迁移成本为零
4. 内置覆盖率报告，满足 Gate 校验中"代码覆盖率 ≥ 80%"的自动化需求

**结论**：✅ 采用 Vitest。

### 3.4 构建工具：tsup (esbuild)

**候选对比**：

| 维度 | tsup | tsc (原生) | unbuild |
|------|------|-----------|---------|
| 构建速度 | 极快（esbuild） | 慢（全量类型检查） | 快（rollup） |
| 配置复杂度 | 极低（约 10 行） | 中（tsconfig） | 低 |
| 产物格式 | ESM / CJS / IIFE | ESM / CJS | ESM / CJS |
| 类型声明 | `--dts` 自动生成 | 原生 | 自动生成 |
| Bundle | 支持（可选） | 不支持 | 支持 |
| Tree-shaking | 支持 | 不支持 | 支持 |

**决策理由**：

1. CLI 工具需要快速构建反馈——tsup 基于 esbuild，构建速度比 tsc 快 10-100x
2. 配置极简，`tsup.config.ts` 约 10 行即可覆盖全部需求
3. 支持同时输出 ESM 产物 + 类型声明，满足 `@spec-first/cli` 发布需求
4. 类型检查由 `tsc --noEmit` 单独执行（CI 中并行），不阻塞构建

**配置示例**：

```typescript
// tsup.config.ts
import { defineConfig } from 'tsup';

export default defineConfig({
  entry: ['src/index.ts'],
  format: ['esm'],
  target: 'node20',
  dts: true,
  clean: true,
  sourcemap: true,
  splitting: false,
});
```

**结论**：✅ 采用 tsup。

### 3.5 包管理器：pnpm

**候选对比**：

| 维度 | pnpm | npm | yarn (berry) |
|------|------|-----|-------------|
| 磁盘效率 | 硬链接，节省 50%+ | 每项目完整副本 | PnP 模式零副本 |
| 安装速度 | 快（并行 + 缓存） | 中 | 快 |
| 幽灵依赖 | 严格隔离，不允许 | 允许（扁平化） | PnP 严格 |
| Monorepo | `workspace:` 原生 | `workspaces` | `workspaces` |
| 锁文件 | `pnpm-lock.yaml` | `package-lock.json` | `yarn.lock` |
| 社区趋势 | 快速增长 | 最广泛 | 稳定 |

**决策理由**：

1. 严格的依赖隔离——防止幽灵依赖，确保 `@spec-first/cli` 发布后依赖完整
2. 磁盘效率高——开发机上多项目共享依赖缓存
3. 未来若拆分 monorepo（如 `@spec-first/core` + `@spec-first/cli`），pnpm workspace 原生支持
4. Node.js 16.13+ 内置 corepack，`corepack enable pnpm` 即可使用

**结论**：✅ 采用 pnpm。

### 3.6 代码规范：ESLint 9 + Prettier

**选型**：

| 工具 | 版本 | 职责 |
|------|------|------|
| ESLint | 9.x (Flat Config) | 代码质量 + TypeScript 规则 |
| Prettier | 3.x | 代码格式化（与 ESLint 分离） |
| @typescript-eslint | 8.x | TS 类型感知规则 |

**决策理由**：

1. ESLint 9 Flat Config 是新标准，配置更简洁，避免 `.eslintrc` 的继承链混乱
2. Prettier 专注格式化，ESLint 专注质量——职责分离，不互相干扰
3. `@typescript-eslint` 提供类型感知规则（如 `no-floating-promises`），对异步 Gate 评估逻辑尤为重要

**关键规则**：

```typescript
// eslint.config.ts 关键规则
{
  rules: {
    '@typescript-eslint/no-floating-promises': 'error',  // Gate 异步必须 await
    '@typescript-eslint/strict-boolean-expressions': 'error',
    'no-console': ['error', { allow: ['warn', 'error'] }],  // CLI 输出走专用 logger
    'eqeqeq': 'error',
  }
}
```

**结论**：✅ 采用 ESLint 9 + Prettier。

## 4. 项目工程结构

### 4.1 单包 vs Monorepo

| 维度 | 单包 (Single Package) | Monorepo |
|------|----------------------|----------|
| 复杂度 | 低，一个 package.json | 高，需 workspace 配置 |
| 发布 | `npm publish` 一步完成 | 需 changeset / lerna |
| 内部引用 | 直接 import | workspace 协议 |
| 适用阶段 | MVP / 早期 | 成熟期拆分 |

**决策**：MVP 阶段采用**单包**，所有 M1-M7 模块在 `src/` 下按目录组织。未来规模增长后可拆分为 `@spec-first/core` + `@spec-first/cli`。

### 4.3 结构设计说明

| 设计决策 | 理由 |
|---------|------|
| `commands/` 与 `core/` 分离 | 交互层（CLI 参数解析、输出格式化）与业务逻辑解耦，便于未来接入其他交互方式（如 MCP Server） |
| `core/` 按 M1-M7 子目录 | 与设计文档 1:1 映射，代码可直接追溯到对应设计章节 |
| 每个子模块 `index.ts` 导出 | 统一对外接口，内部实现可自由重构而不影响调用方 |
| `shared/` 共享层 | 类型定义、错误处理、文件 IO 等横切关注点集中管理 |
| `parsers/` 独立 | Markdown/YAML/ID 解析是多模块共享的基础能力，独立封装避免重复 |
| `templates/` 使用 Handlebars | 产出物模板需支持条件渲染（Mode/Size 裁剪），Handlebars 语法简洁且无运行时依赖 |

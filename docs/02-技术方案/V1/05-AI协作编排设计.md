## 1. 模块职责与边界

### 1.1 核心职责

| 职责 | 说明 |
|------|------|
| **Agent 路由分派** | 根据当前阶段和任务类型，从路由矩阵中选择最合适的 Agent，生成委派指令 |
| **Context Pack 构建** | 跨 Agent 委派时构建标准化上下文包，确保任意 Agent 可恢复完整语境 |
| **Session Catchup** | 会话中断后自动恢复上下文（读取运行态三文件 + 追踪矩阵 + 阶段定位） |
| **并行执行编排** | Size M/L 场景下编排独立 TASK 的并行执行，管理 Worktree 隔离和合并 |
| **AI 统计采集** | 会话结束时采集代码/文档变更统计，追加写入 `ai-stats.jsonl` |
| **AI 能力边界管控** | 标记 AI 生成内容（`[AI-GENERATED]`），确保人类终审不被跳过 |

### 1.2 模块边界

### 1.3 与其他模块的接口

| 调用方 | 接口 | 说明 |
|--------|------|------|
| M5 → M1 | `getCurrentStage(featureId)` | 路由分派和 Session Catchup 时查询当前阶段 |
| M5 → M1 | `getStageConfig(featureId, stage)` | 获取裁剪后的阶段配置，确定产出物列表 |
| M5 → M2 | `getMatrixSnapshot(featureId)` | 构建 Context Pack 时获取追踪矩阵快照 |
| M5 → M2 | `getCoverage(featureId, type)` | Session Catchup 时校验覆盖率一致性 |
| M5 → M6 | `recordAIStats(stats)` | 会话结束时写入 AI 统计数据 |
| M5 → M7 | `getAgentRegistry()` | 从 constitution.md 加载 Agent 注册表 |
| M3 → M5 | Layer A Hook 触发 | PreToolUse/PostToolUse/Stop 三种时机调用 M5 |

### 1.4 专家会审记录

> **[Arch]**: M5 是「AI 协作的编排层」——它不执行具体的 AI 任务（那是各 Agent 的事），而是管理 Agent 的选择、上下文传递和会话连续性。M5 与 M1 的关系是「消费者」——M5 依赖 M1 提供阶段信息来驱动路由决策，但不触发阶段转换。M5 与 M3 的关系是「被调用方」——Layer A Hook 在 AI Runtime 中触发 M5 的统计采集和上下文校验。
> **[DevEff]**: AI 协作编排的核心矛盾是「上下文完整性 vs Token 效率」。Context Pack 必须携带足够信息让 Agent 理解任务，但不能把所有文件都塞进去（Token 成本过高）。建议 Context Pack 采用「元数据 + 按需加载」模式——Pack 中只包含文件路径和摘要，Agent 按需读取完整内容。
> **[QA]**: AI 生成内容的质量保障是 M5 的隐性职责——`[AI-GENERATED]` 标记确保人类审核不被跳过，但标记本身可能被 AI 遗漏。建议 Layer A PostToolUse Hook 自动检测新增内容是否包含 AI 标记，缺失时自动补充。
> **[决议]**: M5 定位为「AI 编排层」，不执行具体任务。Context Pack 采用「元数据 + 按需加载」模式。`[AI-GENERATED]` 标记由 PostToolUse Hook 自动补充兜底。

## 2. Agent 路由体系

### 2.1 路由架构

```
AI 任务请求
    │
    ▼
识别任务类型（需求分析/架构设计/编码/测试/文档等）
    │
    ▼
查询 M1.getCurrentStage() 获取当前阶段
    │
    ▼
查询路由矩阵（阶段 × 任务类型 → Agent）
    │
    ├── Size S → 使用默认 Agent（不启用多代理路由）
    │
    └── Size M/L → 查询 Agent Registry
            │
            ├── 匹配到专业 Agent → 生成 Context Pack → 委派
            │
            └── 未匹配 → 使用默认 Agent
    │
    ▼
Agent 执行任务，产出物标记 [AI-GENERATED]
    │
    ▼
人类终审（Sign-off）→ 标记 [HUMAN-REVIEWED]
```

### 2.2 能力路由矩阵

路由矩阵分为两层：**能力层**（规范定义，不可变）和**映射层**（配置定义，可替换）。

**能力层**（规范约束）：

| 阶段 | 任务类型 | 所需能力 | 输入 | 期望产出 | AI 占比 |
|------|---------|---------|------|---------|--------|
| 01 Specify | 需求分析 | 业务理解 + 结构化输出 | 业务意图、领域知识 | `spec.md` 草稿 | 60-70% |
| 02 Design | 架构设计 | 系统设计 + 技术选型 | `spec.md`、技术约束 | `design.md` 草稿 | 50-60% |
| 02 Design | 外部调研 | 信息检索 + 对比分析 | 技术选型问题 | `findings.md` 更新 | 70-80% |
| 03 Plan | 任务编排 | 任务分解 + 依赖分析 | `design.md`、FR 列表 | `tasks.md` | 60-70% |
| 04 Implement | 代码生成 | 编码 + 测试生成 | TASK 定义、API 契约 | 源码 + 单元测试 | 70-80% |
| 04 Implement | 代码搜索 | 代码库索引 + 语义检索 | 搜索意图 | 相关代码片段 | 90%+ |
| 05 Verify | 测试生成 | AC 理解 + TC 设计 | `spec.md` AC + 代码 | TC + 测试报告 | 60-70% |
| 06 Wrap-up | 文档生成 | 技术写作 + 格式化 | 全阶段产出物 | README + API 文档 | 70-80% |

**映射层**（维护在 `constitution.md` → `agent-registry` 节）：

```yaml
# constitution.md → agent-registry
capabilities:
  需求分析: { agent: "oracle", source: "omo-skills" }
  架构设计: { agent: "sisyphus", source: "omo-skills" }
  外部调研: { agent: "librarian", source: "omo-skills" }
  任务编排: { agent: "do", source: "myclaude" }
  代码生成: { agent: "codeagent-wrapper", source: "myclaude" }
  代码搜索: { agent: "explore", source: "omo-skills" }
  测试生成: { agent: "default", source: "built-in" }
  文档生成: { agent: "document-writer", source: "omo-skills" }
```

### 2.3 路由规则与 Size 适配

| 规则 | 说明 | 实现方式 |
|------|------|---------|
| **默认最小集** | Size S 仅使用默认 Agent，不启用多代理路由 | 路由函数检测 Size，S 直接返回 `default` |
| **按需升级** | Size M/L 或跨端场景启用专业 Agent 分工 | 路由函数查询 Agent Registry |
| **人类终审** | 所有 Agent 产出物必须经人类 Sign-off | PostToolUse Hook 检查 `[HUMAN-REVIEWED]` 标记 |
| **能力降级** | 专业 Agent 不可用时自动降级为默认 Agent | 路由函数 try-catch，失败回退 |
| **阶段约束** | Agent 只能操作当前阶段允许的产出物 | Context Pack 中 `current_phase` 限定操作范围 |

**路由算法**：

```
function routeAgent(taskType, featureMeta):
  // 规则 1: Size S 不启用多代理路由
  if featureMeta.size === "S":
    return { agent: "default", reason: "Size S uses default agent" }

  // 规则 2: 查询 Agent Registry
  registry = loadAgentRegistry()
  entry = registry.capabilities[taskType]

  if entry is null:
    return { agent: "default", reason: "No agent registered for " + taskType }

  // 规则 3: 检查 Agent 可用性
  if not isAgentAvailable(entry.agent, entry.source):
    return { agent: "default", reason: entry.agent + " unavailable, fallback" }

  return { agent: entry.agent, source: entry.source }
```

### 2.4 AI 能力边界与标记规范

**AI 生成占比基线**（按任务类型）：

| 任务类型 | AI 占比 | AI 职责边界 | 人类不可委托职责 |
|---------|--------|------------|----------------|
| 需求分析 | 60-70% | 结构化整理、歧义检测、AC 生成建议 | 业务决策、优先级判定、利益相关方确认 |
| 架构设计 | 50-60% | 方案生成、Trade-off 分析、技术选型调研 | 架构决策拍板、非功能需求量化、风险承担 |
| 外部调研 | 70-80% | 信息检索、对比矩阵、趋势分析 | 结论判定、适用性评估、成本决策 |
| 任务编排 | 60-70% | 任务拆解、依赖分析、工时估算建议 | 优先级排序、资源分配、里程碑承诺 |
| 代码生成 | 70-80% | 代码编写、单元测试、追踪注释 | Code Review 终审、安全审计、性能基线确认 |
| 代码搜索 | 90%+ | 代码库索引、语义检索、相关性排序 | 搜索意图确认 |
| 测试生成 | 60-70% | TC 设计、测试数据生成、覆盖率分析 | UAT 验收、安全测试终审、性能测试基线 |
| 文档生成 | 70-80% | 文档撰写、格式化、交叉引用检查 | 技术准确性终审、对外发布审批 |

**标记规范**：

| 标记 | 格式 | 添加时机 | 添加方式 |
|------|------|---------|---------|
| `[AI-GENERATED]` | 文件头部或章节头部 | AI 产出内容写入时 | PostToolUse Hook 自动检测并补充 |
| `[HUMAN-REVIEWED]` | 紧跟 `[AI-GENERATED]` 之后 | 人类完成审核后 | 人工手动添加，Gate 校验检查 |
| `[AI-ASSISTED]` | 文件头部 | AI 辅助但人类主导的内容 | 人工标注 |

**标记检测算法**：

```
function checkAIMarker(filePath, diffContent):
  // PostToolUse Hook 调用
  if isNewFile(diffContent) or hasSubstantialAddition(diffContent, threshold=20):
    if not containsMarker(filePath, "[AI-GENERATED]"):
      appendMarker(filePath, "[AI-GENERATED]")
      return { action: "marker_added", file: filePath }
  return { action: "no_change" }

function hasSubstantialAddition(diff, threshold):
  addedLines = diff.filter(line => line.startsWith("+")).length
  return addedLines >= threshold
```

### 2.5 专家会审记录

> **[Arch]**: 路由体系的两层设计（能力层 + 映射层）是关键架构决策——能力层定义「需要什么能力」是规范约束，映射层定义「谁来提供」是配置项。这意味着更换 Agent 工具链（如从 Claude Code 迁移到其他 AI IDE）只需修改映射层，能力层不变。路由算法的降级策略（专业 Agent 不可用时回退 default）保证了系统可用性。
> **[DevEff]**: AI 占比基线是「参考值」而非「硬约束」——实际占比取决于任务复杂度和 AI 能力演进。建议在 `ai-stats.jsonl` 中记录每次任务的实际 AI 占比，用于校准基线。`[AI-GENERATED]` 标记的自动补充机制（PostToolUse Hook）是开发者体验的关键——开发者不需要手动记住标记，系统自动兜底。
> **[QA]**: 能力边界表中「人类不可委托职责」是质量保障的底线——AI 可以辅助但不能替代人类在决策、审批、验收环节的角色。`[HUMAN-REVIEWED]` 标记必须由人工手动添加，不可由 AI 自动生成，否则失去审核意义。建议 Gate 校验时检查：所有 `[AI-GENERATED]` 内容必须有对应的 `[HUMAN-REVIEWED]`。
> **[决议]**: 两层路由设计确认。AI 占比基线作为参考值，实际占比记录到统计。`[HUMAN-REVIEWED]` 必须人工添加。Gate 校验增加「AI 标记配对检查」。

## 3. Context Pack 标准

### 3.1 设计目标与约束

Context Pack 是跨 Agent 委派时的标准化上下文包，解决「AI 会话无状态」问题。

| 目标 | 说明 |
|------|------|
| **语境完整** | 任意 Agent 仅凭 Context Pack 即可理解任务全貌，无需额外口头说明 |
| **Token 高效** | 采用「元数据 + 按需加载」模式，Pack 本身 < 2KB，Agent 按需读取完整文件 |
| **格式统一** | 所有 Agent 使用相同的 YAML Schema，降低解析成本 |
| **可校验** | Pack 中的文件路径、阶段信息、任务 ID 均可自动校验合法性 |

**强制约束**：

| # | 约束 | 校验方式 |
|---|------|---------|
| C1 | 每次 Agent 委派必须生成 Context Pack，禁止口头传递上下文 | PreToolUse Hook 检测委派指令是否携带 Pack |
| C2 | `artifacts` 路径必须指向实际存在的文件 | Pack 生成时校验 `fs.existsSync()` |
| C3 | `current_phase` 必须与 M1 `getCurrentStage()` 一致 | Pack 生成时查询 M1 校验 |
| C4 | `current_task` 必须与 `task_plan.md` 记录一致 | Pack 生成时解析 task_plan.md 校验 |
| C5 | Pack 版本号必须与当前工具链版本兼容 | 版本号比对，不兼容时拒绝委派 |

### 3.2 Context Pack YAML Schema

```yaml
# context-pack.yaml — 跨 Agent 统一输入格式
context_pack:
  version: "1.0"                    # Pack Schema 版本

  # ── Feature 元数据 ──
  feature_meta:
    id: "001-user-auth"             # Feature 目录名
    title: "用户认证模块"             # Feature 标题
    feat_abbr: "AUTH"               # FEAT 缩写
    mode: "N"                       # N=新建 / I=迭代
    size: "S"                       # S/M/L
    platforms: ["H5", "Backend"]    # 涉及端

  # ── 产出物路径索引（元数据，Agent 按需读取） ──
  artifacts:
    spec: "specs/001-user-auth/spec.md"
    design: "specs/001-user-auth/design.md"
    tasks: "specs/001-user-auth/tasks.md"
    checklist: "specs/001-user-auth/checklist.md"
    matrix: "specs/001-user-auth/traceability-matrix.md"
    contracts:                      # API 契约文件列表
      - "specs/001-user-auth/contracts/API-AUTH-001.yaml"
    reports: []                     # 阶段报告（按需填充）

  # ── 运行态文件索引 ──
  runtime:
    stage_state: ".spec-first/runtime/stage-state.json"
    gate_history: ".spec-first/runtime/gate-history.jsonl"
    ai_stats: ".spec-first/runtime/ai-stats.jsonl"

  # ── 当前执行上下文 ──
  execution:
    current_phase: "04_implement"   # 当前阶段（与 M1 一致）
    current_task: "TASK-AUTH-001"   # 当前执行的 TASK ID
    task_description: "实现用户邮箱注册 API"  # TASK 简述
    related_frs: ["FR-AUTH-001"]    # 关联的 FR 列表
    related_apis: ["API-AUTH-001"]  # 关联的 API 契约

  # ── 规范引用 ──
  constitution: ".spec-first/constitution.md"

  # ── 摘要信息（减少 Agent 读取完整文件的需求） ──
  summaries:
    spec_summary: "用户认证模块：支持邮箱注册、登录、密码重置三个核心功能"
    design_decisions: ["JWT 无状态认证", "bcrypt 密码加密", "Redis Session 缓存"]
    pending_tasks: 3                # 剩余未完成 TASK 数
    coverage_snapshot:              # 覆盖率快照
      task_coverage: 1.0
      api_coverage: 1.0
      test_coverage_fr: 0.67
```

### 3.3 Context Pack 构建流程

```
Agent 委派请求
    │
    ▼
Step 1: 查询 M1.getCurrentStage(featureId) → current_phase
    │
    ▼
Step 2: 查询 M1.getStageConfig(featureId, stage) → 产出物列表
    │
    ▼
Step 3: 扫描 Feature 目录，收集已存在的产出物路径
    │    校验：路径必须指向实际文件（C2）
    ▼
Step 4: 查询 M2.getMatrixSnapshot(featureId) → 覆盖率快照
    │
    ▼
Step 5: 解析 task_plan.md → 当前 TASK + 剩余 TASK 数
    │    校验：current_task 与 task_plan.md 一致（C4）
    ▼
Step 6: 生成摘要信息
    │    - spec_summary: 从 spec.md 提取首段摘要
    │    - design_decisions: 从 design.md 提取 ADR 标题列表
    │    - pending_tasks: 从 tasks.md 统计未完成 TASK 数
    ▼
Step 7: 组装 context-pack.yaml → 写入缓存目录
    │    路径: .spec-first/runtime/session-context/FEAT-NNN-session.json
    ▼
Step 8: 返回 Pack 路径给路由分派器
```

**构建性能约束**：

| 指标 | 目标 | 实现方式 |
|------|------|---------|
| Pack 构建耗时 | < 500ms | M1/M2 查询走内存缓存，文件扫描用 `glob` 批量匹配 |
| Pack 文件大小 | < 2KB | 仅存元数据和摘要，不嵌入完整文件内容 |
| 缓存命中率 | > 80% | 同一 Feature 同一阶段的 Pack 缓存复用，阶段变更时失效 |

### 3.4 按需加载策略

Context Pack 仅携带文件路径和摘要，Agent 根据任务需要按需读取完整文件内容。按需加载矩阵按阶段定义：

| 当前阶段 | 必读文件 | 按需读取 | 不加载 |
|---------|---------|---------|--------|
| 01 Specify | constitution.md | — | design.md, tasks.md, 代码 |
| 02 Design | spec.md, constitution.md | research.md, findings.md | tasks.md, 代码 |
| 03 Plan | spec.md, design.md | contracts/*.yaml | 代码, 测试 |
| 04 Implement | tasks.md（当前 TASK）, contracts/*.yaml | spec.md（相关 FR）, design.md（相关 DS） | 其他 TASK, 测试报告 |
| 05 Verify | spec.md（AC 部分）, 代码（被测模块） | design.md, tasks.md | research.md |
| 06 Wrap-up | 全阶段产出物索引 | 按需读取各文件 | — |

**加载优先级算法**：

```
function resolveLoadPriority(pack, taskType):
  phase = pack.execution.current_phase
  loadMatrix = PHASE_LOAD_MATRIX[phase]

  // 必读文件：立即加载到 Agent 上下文
  mustRead = loadMatrix.must_read.map(path => readFile(path))

  // 按需文件：生成加载指令，Agent 执行时按需触发
  onDemand = loadMatrix.on_demand.map(path => {
    return { path, summary: extractSummary(path), loadCmd: "read " + path }
  })

  return { mustRead, onDemand }
```

### 3.5 专家会审记录

> **[Arch]**: Context Pack 的「元数据 + 按需加载」模式是 Token 效率的关键——Pack 本身 < 2KB，但通过路径索引可以访问整个 Feature 的产出物。`summaries` 节是重要的优化点——Agent 通过摘要判断是否需要读取完整文件，避免无差别加载。Pack 缓存策略需要明确失效条件：阶段变更、产出物文件修改、TASK 切换三种情况都应触发缓存失效。
> **[DevEff]**: 按需加载矩阵按阶段定义「必读/按需/不加载」三级，这对 Agent 的 Token 消耗控制至关重要。04 Implement 阶段只需读取当前 TASK 和相关契约，不需要加载整个 spec.md——这可以将单次 Agent 调用的 Token 消耗降低 40-60%。建议 Pack 构建时自动计算预估 Token 数，超过阈值时警告。
> **[QA]**: Context Pack 的 5 条强制约束（C1-C5）是质量保障的基础——特别是 C3（阶段一致性）和 C4（任务一致性），防止 Agent 在错误的上下文中执行任务。建议 Pack 校验失败时不仅阻断委派，还要输出具体的不一致项和修复建议。
> **[决议]**: Pack 缓存失效条件明确为三种（阶段变更/文件修改/TASK 切换）。Pack 构建时计算预估 Token 数。校验失败输出具体不一致项 + 修复建议。

## 4. Session Catchup 机制

### 4.1 触发场景

| 触发条件 | 触发方式 | 频率 | 说明 |
|---------|---------|------|------|
| `/clear` 命令执行 | 用户主动触发 | 中频 | 用户清空上下文后需要恢复 |
| 上下文窗口截断 | AI Runtime 自动检测 | 低频 | Token 超限导致早期上下文丢失 |
| IDE / CLI 重启 | 启动时自动检测 | 中频 | 进程重启后上下文全部丢失 |
| Agent 切换 | 路由分派器触发 | 高频（M/L） | 从 Agent A 切换到 Agent B 时恢复语境 |
| 长时间闲置恢复 | 用户重新交互时 | 低频 | 超过 30 分钟无交互后恢复 |

### 4.2 恢复流程

```
Session Catchup 触发
    │
    ▼
Step 1: 读取运行态三文件
    ├── task_plan.md   → 当前规划状态
    ├── stage-state.json    → 当前阶段状态
    └── findings.md    → 已有发现和决策
    │
    ▼
Step 2: 读取追踪矩阵
    └── traceability-matrix.md/yaml → 追踪状态快照
    │
    ▼
Step 3: 定位当前阶段和 TASK
    ├── 查询 M1.getCurrentStage(featureId) → current_phase
    └── 解析 task_plan.md → current_task + 完成百分比
    │
    ▼
Step 4: 按 current_phase 动态加载阶段核心文件（见 4.3）
    │
    ▼
Step 5: 一致性校验
    ├── task_plan.md 记录 vs 实际文件状态
    ├── 追踪矩阵 vs 产出物文件存在性
    └── stage-state.json vs task_plan.md 阶段一致
    │
    ├── 一致 → Step 6
    └── 不一致 → 生成不一致报告，标记需修正项
    │
    ▼
Step 6: 生成恢复摘要 → 输出到终端
    │
    ▼
Step 7: 构建 Context Pack → 恢复完成，Agent 可继续工作
```

### 4.3 动态加载文件矩阵

Session Catchup 的 Step 4 按 `current_phase` 动态选择加载文件，避免全量加载导致 Token 浪费：

| current_phase | 必须加载 | 可选加载 | 不加载 |
|---------------|---------|---------|--------|
| 01 Specify | `constitution.md` | — | design.md, tasks.md, 代码 |
| 02 Design | `spec.md` | `research.md`, `findings.md` | tasks.md, 代码 |
| 03 Plan | `spec.md`, `design.md` | `contracts/*.yaml` | 代码, 测试 |
| 04 Implement | `tasks.md`, `checklist.md` | `design.md`（相关 DS 节） | 其他 TASK, 测试报告 |
| 05 Verify | `spec.md`（AC 部分）, `tasks.md` | `tests/*.test.md` | research.md |
| 06 Wrap-up | `spec.md`（摘要） | 全部产出物（按需） | — |

**加载策略说明**：

- **必须加载**：恢复后 Agent 立即需要的文件，在 Catchup 流程中主动读取
- **可选加载**：Agent 执行任务时可能需要的文件，生成 `loadCmd` 指令供按需触发
- **不加载**：当前阶段不相关的文件，不出现在恢复上下文中

### 4.4 一致性校验规则

Session Catchup 的 Step 5 执行一致性校验，检测运行态数据与实际产出物之间的偏差：

| # | 校验项 | 校验逻辑 | 不一致处理 |
|---|--------|---------|-----------|
| V1 | 阶段一致性 | `stage-state.json.currentStage` 与 `task_plan.md` 中阶段记录一致 | 以 `stage-state.json` 为准，修正 `task_plan.md` |
| V2 | TASK 完成状态 | `task_plan.md` 标记完成的 TASK → 对应代码/PR 是否存在 | 标记为 `[INCONSISTENT]`，提示人工确认 |
| V3 | 矩阵填充完整性 | 当前阶段应填充的列是否已填充 | 输出缺失列清单，提示补充 |
| V4 | 文件存在性 | `artifacts` 路径指向的文件是否实际存在 | 输出缺失文件列表，提示创建 |
| V5 | ID 引用合法性 | `task_plan.md` 中引用的 TASK/FR ID 是否在矩阵中存在 | 输出悬空引用列表 |

**校验算法**：

```
function validateConsistency(catchupData):
  issues = []

  // V1: 阶段一致性
  stageFromState = readStageState().currentStage
  stageFromProgress = parseProgress().currentPhase
  if stageFromState !== stageFromProgress:
    issues.push({ type: "STAGE_MISMATCH", fix: "auto", detail: ... })
    fixProgressPhase(stageFromState)  // 自动修正

  // V2: TASK 完成状态
  completedTasks = parseProgress().completedTasks
  for task in completedTasks:
    if not hasCorrespondingOutput(task):
      issues.push({ type: "TASK_NO_OUTPUT", fix: "manual", detail: ... })

  // V3-V5: 矩阵/文件/ID 校验
  // ... 类似逻辑

  return {
    consistent: issues.length === 0,
    autoFixed: issues.filter(i => i.fix === "auto").length,
    manualRequired: issues.filter(i => i.fix === "manual"),
  }
```

### 4.6 专家会审记录

> **[Arch]**: Session Catchup 的核心价值是「无损恢复」——AI 会话中断不应导致上下文丢失或工作重复。恢复流程的 7 步设计覆盖了从数据读取到一致性校验的完整链路。关键设计点：Step 5 的一致性校验区分「自动修正」和「人工确认」两类问题——阶段不一致可以自动修正（以 stage-state.json 为权威源），但 TASK 完成状态不一致需要人工确认（涉及外部系统如 Git/PR）。
> **[DevEff]**: 恢复摘要的终端输出格式是开发者体验的关键——开发者在 `/clear` 后需要快速了解「我在哪、做到哪、还剩什么」。摘要中的覆盖率快照和已完成百分比提供了即时的进度感知。建议恢复耗时控制在 500ms 以内，超过 1s 会影响交互流畅度。
> **[QA]**: 一致性校验的 5 项规则（V1-V5）覆盖了主要的数据偏差场景。V2（TASK 完成状态校验）是最容易出现不一致的场景——AI 标记 TASK 完成但代码未提交、PR 未合并。建议 V2 校验时不仅检查 PR 存在性，还检查 PR 状态（merged vs open vs closed）。
> **[决议]**: 恢复耗时目标 < 500ms。V2 校验增加 PR 状态检查。一致性校验结果记录到 `gate-history.jsonl` 作为审计轨迹。

## 5. 并行执行模型

### 5.1 并行条件判定

Size M/L 场景下，独立 TASK 可并行执行以缩短实施周期。并行执行需满足以下前置条件：

| # | 条件 | 判定方式 | 不满足时处理 |
|---|------|---------|------------|
| P1 | TASK 间无依赖 | `tasks.md` 中 `depends_on` 为空 | 按依赖顺序串行执行 |
| P2 | 文件变更范围无交叉 | 预分析 TASK 涉及的文件路径集合，取交集 | 有交叉则串行执行 |
| P3 | Size ≥ M | 从 `stage-state.json` 读取 Size | Size S 不启用并行 |
| P4 | TASK 已标记 `parallel: true` | 解析 `tasks.md` 中的 TASK 元数据 | 未标记则默认串行 |

**执行模式决策矩阵**：

| TASK 间依赖 | 文件交叉 | 执行模式 | 隔离方式 |
|------------|---------|---------|---------|
| 无依赖 | 无交叉 | **并行执行** | Git Worktree 隔离 |
| 无依赖 | 有交叉 | **串行执行** | 主分支顺序提交 |
| 有依赖 | — | **按依赖顺序串行** | 主分支顺序提交 |

### 5.2 Git Worktree 隔离机制

并行 TASK 通过 Git Worktree 实现文件系统级隔离，避免并发修改冲突：

```
并行执行编排流程：

Step 1: 从 tasks.md 提取可并行 TASK 集合
    │    过滤条件: parallel=true && depends_on=[]
    ▼
Step 2: 文件交叉预分析
    │    对每个 TASK 预判涉及的文件路径集合
    │    取两两交集，有交叉的 TASK 降级为串行
    ▼
Step 3: 为每个并行 TASK 创建 Worktree
    │    git worktree add ../worktree-TASK-FEAT-NNN feature/TASK-FEAT-NNN
    │    每个 Worktree 独立分支，独立文件系统
    ▼
Step 4: 分派 Agent 到各 Worktree
    │    每个 Agent 携带独立 Context Pack
    │    Agent 在 Worktree 目录中执行任务
    ▼
Step 5: 等待所有并行 TASK 完成
    │    超时机制: 单 TASK 超时不阻塞其他 TASK
    ▼
Step 6: 顺序合并回主分支
    │    git merge feature/TASK-FEAT-NNN --no-ff
    │    冲突时暂停，等待人工解决
    ▼
Step 7: 清理 Worktree
    │    git worktree remove ../worktree-TASK-FEAT-NNN
    ▼
Step 8: 运行覆盖率校验，确认合并后数据一致
```

**Worktree 命名规范**：

```
../worktree-{TASK-ID}/
  例: ../worktree-TASK-AUTH-001/
  例: ../worktree-TASK-AUTH-003/
```

### 5.3 并发写入保护

并行 Agent 同时操作运行态三文件和追踪矩阵时，需要专门的并发保护策略。

**运行态三文件（task_plan.md / findings.md / stage-state.json）**：

| 策略 | 说明 |
|------|------|
| **Append-Only 写入** | 并行 Agent 只追加内容，不修改已有行。每条追加记录包含 `[TASK-FEAT-NNN]` 前缀标识来源 |
| **时间戳排序** | 每条追加记录附带 ISO 8601 时间戳，合并时按时间排序 |
| **合并窗口** | 并行 TASK 全部完成后，由主 Agent 执行一次合并整理，消除重复、解决冲突 |
| **冲突检测** | 若两个 Agent 修改了同一行（非追加），合并时标记 `[CONFLICT]` 由人工裁决 |

**追踪矩阵并发更新策略**：

| 场景 | 策略 |
|------|------|
| 各 TASK 对应不同 FR 行 | 直接合并，无冲突（各改各的行） |
| 多个 TASK 对应同一 FR 行 | 最后完成的 TASK 负责合并该行的 Task Ref / PR Ref 列 |
| Status 列更新 | 以最低状态为准（如一个 TASK 已 Verified 但另一个仍 In Progress，则保持 In Progress） |

**合并流程**：

```
并行 TASK 全部完成
    │
    ▼
主分支执行 git merge（按 TASK 编号顺序逐个合并）
    │
    ├── 无冲突 → 继续下一个
    └── 有冲突 → 暂停，通知 Tech Lead 手动合并
    │
    ▼
运行覆盖率校验脚本（全量模式）
    │
    ├── 通过 → 并行执行完成
    └── 未通过 → 输出不一致项，人工修正
```

### 5.4 专家会审记录

> **[Arch]**: 并行执行模型的核心约束是「隔离性」——Git Worktree 提供了文件系统级隔离，每个并行 TASK 在独立目录中执行，互不干扰。合并阶段按 TASK 编号顺序逐个合并（而非一次性合并所有），降低了冲突解决的复杂度。文件交叉预分析（Step 2）是关键的前置校验——如果两个 TASK 都会修改同一个文件（如 `design.md`），必须降级为串行，否则合并时必然冲突。
> **[DevEff]**: 并行执行的收益在 Size L（15+ TASK）时最为显著——串行执行可能需要数天，并行可缩短至 1/3。但并行也增加了编排复杂度和合并风险。建议并行 TASK 数量上限为 5（受限于 AI Agent 并发能力和人类监控能力）。Worktree 的创建和清理应自动化，开发者无需手动操作 Git 命令。
> **[QA]**: 并发写入保护的「以最低状态为准」规则是保守但安全的策略——避免了某个 TASK 提前标记 Verified 但其他 TASK 尚未完成的误导。合并后的全量覆盖率校验是质量兜底——确保并行合并没有引入追踪链断裂。建议合并后除覆盖率校验外，还运行 SCA 一致性校验。
> **[决议]**: 并行 TASK 上限 5 个。合并后运行覆盖率 + SCA 双重校验。Worktree 生命周期自动管理。文件交叉预分析作为并行的必要前置条件。

## 6. AI 统计采集

### 6.1 采集时机与触发方式

| 触发时机 | 触发方式 | 采集内容 | 说明 |
|---------|---------|---------|------|
| AI 会话结束 | Layer A Stop Hook | 代码变更 + 文档变更 + 会话元数据 | 主要采集入口 |
| Git pre-push | Layer B Hook（兜底） | 代码变更 + 文档变更 | Layer A 未触发时的兜底采集 |
| 手动触发 | CLI 命令 `spec-stats collect` | 同 Stop Hook | 调试和补录场景 |

### 6.2 采集流程

```
AI 会话结束（Stop Hook 触发）
    │
    ▼
Step 1: 读取 baseline commit hash
    │    来源: .spec-first/.baseline（Start Hook 写入）
    ▼
Step 2: 采集开发人员信息
    │    git config user.name + git config user.email
    ▼
Step 3: 解析分支名提取需求 ID
    │    分支格式: feature/<FR-ID>-<description>
    │    提取: FR-ID → 关联到追踪矩阵
    ▼
Step 4: 统计代码变更
    │    git diff --stat <baseline>..HEAD -- '*.ts' '*.js' '*.vue' ...
    │    输出: 文件数、+additions、-deletions
    ▼
Step 5: 统计文档变更
    │    git diff --stat <baseline>..HEAD -- '*.md' '*.yaml'
    │    输出: 文件数、+additions、-deletions
    ▼
Step 6: 组装统计记录 → 追加写入 ai-stats.jsonl
    ▼
Step 7: 终端输出统计摘要
    │    格式: AI Stats | Leo | FR-AUTH-001 | 04-implement | Code: 5 files +127 -34
    ▼
Step 8: 清理 .baseline 文件
```

### 6.4 运维策略

| 维度 | 策略 | 说明 |
|------|------|------|
| **Git 管理** | `.gitignore` 排除，定期归档 | JSONL 频繁追加不适合逐次提交；按月归档到 `stats/` 目录后纳入 Git |
| **文件轮转** | 按月自动轮转 | 文件名格式 `ai-stats-YYYY-MM.jsonl`，`ai-stats.jsonl` 为当月软链接 |
| **并发写入** | 文件锁 + 追加模式 | `flock`（Linux）/ `lockfile`（macOS）确保多 Agent 并发写入不丢数据 |
| **容量上限** | 单文件 ≤ 10MB | 超过阈值自动轮转，避免文件过大影响解析性能 |
| **数据清理** | 保留最近 6 个月 | 超期归档文件压缩存储或移至外部存储 |

**轮转机制实现**：

```
function ensureStatsFile():
  currentMonth = formatDate(now(), "YYYY-MM")
  targetFile = ".spec-first/runtime/ai-stats-" + currentMonth + ".jsonl"
  symlinkPath = ".spec-first/runtime/ai-stats.jsonl"

  if not exists(targetFile):
    createFile(targetFile)
    // 更新软链接指向新月份文件
    removeSymlink(symlinkPath)
    createSymlink(targetFile, symlinkPath)

  // 检查容量上限
  if fileSize(targetFile) > 10 * 1024 * 1024:  // 10MB
    archivePath = targetFile + "." + timestamp() + ".gz"
    compress(targetFile, archivePath)
    truncate(targetFile)

  return symlinkPath
```

### 6.5 专家会审记录

> **[Arch]**: AI 统计采集的架构定位是「被动记录」而非「主动干预」——统计数据用于事后分析和度量运营，不影响 Agent 执行流程。Stop Hook 是主要采集入口，pre-push Hook 是兜底——两层采集确保数据不丢失。JSONL 格式的选择是正确的：追加写入无锁竞争、grep 友好、按行解析无需加载全文件。
> **[DevEff]**: 统计采集对开发者应该是「零感知」的——Stop Hook 自动触发，终端输出一行摘要即可，不应阻塞或干扰开发流程。`session.durationMinutes` 字段对度量运营很有价值——可以分析 AI 辅助编码的效率趋势。建议 `code.languages` 按扩展名分组统计，便于分析不同语言的 AI 辅助效果。
> **[QA]**: 统计数据的完整性是度量运营的基础——如果 Stop Hook 未触发（如进程异常退出），pre-push Hook 必须能补录。建议 pre-push Hook 检测：自上次统计记录以来是否有未记录的 commit，有则补录。`baseline` 字段记录基线 commit 是关键——确保统计的是本次会话的增量变更，而非累积变更。
> **[决议]**: 统计采集零感知设计。pre-push Hook 补录未记录的 commit。JSONL 按月轮转 + 6 个月保留。统计数据供 M6 度量引擎消费。

### 8.6 专家会审记录

> **[Arch]**: 5 个接口的职责划分遵循单一职责原则——AgentRouter 管「选谁」，ContextBuilder 管「带什么」，SessionManager 管「恢复什么」，ParallelExecutor 管「怎么并行」，StatsCollector 管「记什么」。接口间通过数据结构（ContextPack、CatchupResult、ParallelPlan）传递信息，无直接调用依赖。`RouterDiagnostics` 是运维友好的设计——Agent 注册表加载失败或 Agent 不可用时，诊断接口可快速定位问题。
> **[DevEff]**: `ContextBuilder.resolveLoadPriority` 返回 `estimatedTokens` 是 Token 成本控制的关键——开发者和 AI Agent 都能感知本次委派的 Token 消耗预估。建议设置 Token 预警阈值（如 > 50K tokens），超过时提示优化加载策略。`StatsCollector.formatStatsSummary` 的一行格式设计确保统计输出不干扰开发流程——终端只显示一行摘要，详细数据写入 JSONL。
> **[QA]**: `PackValidationResult` 的 `violations` 数组精确到约束编号（C1-C5），便于自动化测试断言。`SessionManager.validateConsistency` 返回 `ConsistencyReport` 区分 `autoFixed` 和 `manualRequired`，这对测试策略很重要——自动修正项需要验证修正逻辑正确性，人工确认项需要验证提示信息的准确性。`ParallelExecutor.analyzeFileOverlap` 的 `recommendation` 字段将分析结果转化为可执行建议，降低人工判断成本。
> **[决议]**: 5 接口划分确认。Token 预警阈值设为 50K。统计摘要保持一行格式。Pack 校验精确到约束编号。文件交叉分析输出可执行建议。

## 9. 错误处理与边界场景

### 9.1 错误分类与处理策略

| 错误类别 | 触发场景 | 处理策略 | 用户提示 |
|---------|---------|---------|---------|
| **Agent 不可用** | 专业 Agent 来源（omo-skills/myclaude）不可达 | 自动降级为 default Agent，记录降级事件 | 输出降级通知 + 原因 + 降级后 Agent 名称 |
| **Context Pack 构建失败** | 产出物文件缺失、M1 查询超时、矩阵数据不一致 | Fail-Fast，阻断委派操作 | 输出失败的约束编号（C1-C5）+ 具体缺失项 + 修复建议 |
| **Session Catchup 失败** | 运行态三文件损坏、stage-state.json 不可读 | 尝试从 Git 历史恢复，失败则降级为「冷启动」模式 | 输出损坏文件 + 恢复尝试结果 + 冷启动提示 |
| **并行 Worktree 创建失败** | 磁盘空间不足、Git 版本不支持、分支名冲突 | 降级为串行执行，记录降级原因 | 输出失败原因 + 自动切换串行模式通知 |
| **并行合并冲突** | 两个 TASK 修改了预分析未检测到的同一文件 | 暂停合并，标记冲突文件，等待人工解决 | 输出冲突文件列表 + 冲突 TASK 对 + 手动合并命令 |
| **统计采集失败** | baseline 文件缺失、git diff 执行失败 | 跳过本次采集，记录失败事件到日志 | 输出警告（不阻断）+ 手动补录命令 |
| **AI 标记缺失** | PostToolUse Hook 检测到新增内容无 `[AI-GENERATED]` 标记 | 自动补充标记（兜底） | 输出自动补充通知 |
| **Token 预算超限** | Context Pack 预估 Token 数超过 50K 阈值 | 警告但不阻断，建议优化加载策略 | 输出当前预估 Token 数 + 阈值 + 优化建议 |
| **Agent Registry 解析失败** | constitution.md 中 agent-registry 节格式错误 | 回退使用 default Agent，记录解析错误 | 输出解析错误位置 + 期望格式 |
| **一致性校验发现不一致** | Session Catchup V1-V5 校验发现数据偏差 | 自动修正项立即修正，人工确认项标记等待 | 输出不一致详情 + 自动修正结果 + 待确认项 |

### 9.2 边界场景处理

#### 场景 1：Agent 工具链迁移

```
触发：团队从 Claude Code 迁移到其他 AI IDE（如 Cursor、Windsurf）
影响：Layer A Hook 机制变更，Agent 映射层需要更新
处理：
  1. 能力层（路由矩阵）不变——「需要什么能力」是规范约束
  2. 映射层（constitution.md → agent-registry）更新 Agent 名称和来源
  3. Context Pack YAML Schema 不变——任何 AI IDE 都可解析标准 YAML
  4. Session Catchup 的运行态三文件格式不变——与 AI IDE 无关
  5. 统计采集的 Stop Hook 需适配新 IDE 的 Hook 机制
  6. 迁移期间可同时保留新旧映射，通过 source 字段区分
```

#### 场景 2：Context Pack 缓存过期但未被检测

```
触发：产出物文件被外部工具（非 Spec-First 流程）修改，缓存未感知
影响：Agent 基于过期的 Context Pack 执行任务，可能产出不一致结果
处理：
  1. Pack 构建时计算 artifacts 路径下所有文件的 hash 摘要
  2. 缓存中记录 hash 摘要，获取缓存时重新计算并比对
  3. hash 不匹配 → 缓存失效，重新构建 Pack
  4. 性能优化：仅计算文件 mtime + size 的组合 hash，避免全文件 hash 开销
  5. 极端情况（mtime 被篡改）：Gate 校验时做全量一致性检查兜底
```

#### 场景 3：并行 TASK 执行中某个 Agent 异常退出

```
触发：并行执行的 5 个 TASK 中，1 个 Agent 进程崩溃
影响：该 TASK 的 Worktree 处于不完整状态
处理：
  1. 其他并行 TASK 不受影响，继续执行
  2. 失败 TASK 状态标记为 failed，记录失败时间和原因
  3. 所有并行 TASK 完成后，先合并成功的 TASK
  4. 失败 TASK 的 Worktree 保留（不自动清理），供人工排查
  5. 提示 Tech Lead：可选择重试失败 TASK 或手动完成
  6. 重试时复用已有 Worktree，从断点继续（若 Worktree 状态可恢复）
```

#### 场景 4：纯人工场景（无 AI）下的 M5 行为

```
触发：团队未配置 AI IDE，Layer A Hook 不存在
影响：M5 的 Agent 路由、Context Pack、Session Catchup 均不适用
处理：
  1. Agent 路由：不触发（无 AI Agent 可分派）
  2. Context Pack：不生成（无跨 Agent 委派需求）
  3. Session Catchup：不触发（无 AI 会话需要恢复）
  4. 并行执行：仍可使用——开发者手动在 Worktree 中并行开发
  5. AI 统计采集：不触发（Stop Hook 不存在）
  6. AI 标记：不适用（所有内容均为人工编写）
  7. 结论：纯人工场景下 M5 处于「休眠」状态，不影响其他模块运行
```

#### 场景 5：AI 会话上下文窗口耗尽

```
触发：长时间 AI 会话导致 Token 窗口接近上限，早期上下文被截断
影响：Agent 丢失任务初始上下文，可能产出偏离需求的结果
处理：
  1. AI Runtime 检测到上下文截断时，自动触发 Session Catchup
  2. Catchup 从运行态三文件恢复完整语境（不依赖会话历史）
  3. 恢复后重新构建 Context Pack，确保 Agent 拥有最新上下文
  4. 建议开发者在长会话中定期执行 /clear + Catchup，主动释放上下文
  5. Context Pack 的 summaries 节设计正是为此场景优化——
     Agent 通过摘要快速理解全貌，无需重读完整文件
```

### 9.3 专家会审记录

> **[Arch]**: 错误处理的核心原则是「优雅降级」——Agent 不可用降级为 default、Worktree 创建失败降级为串行、统计采集失败不阻断主流程。M5 作为编排层，其自身故障不应导致开发流程中断。场景 1（工具链迁移）验证了两层路由设计的价值——能力层不变，映射层可替换，这是架构层面的关键解耦点。场景 4（纯人工场景）确认了 M5 的「可选」定位——无 AI 时 M5 休眠，不影响 M1-M4 和 M6-M7 的正常运行。
> **[DevEff]**: 场景 2（缓存过期）的 mtime+size 组合 hash 是性能与准确性的平衡——全文件 hash 在大文件场景下开销过高，mtime+size 在 99% 场景下足够准确。场景 5（上下文耗尽）是 AI 协作中最常见的痛点——Context Pack 的 summaries 节和 Session Catchup 机制正是为此设计。建议在 AI 统计中记录「Catchup 触发次数」，用于分析上下文管理的效率。
> **[QA]**: 场景 3（Agent 异常退出）的「失败 Worktree 保留」策略是正确的——自动清理可能丢失调试信息。但需要设置保留期限（如 7 天），超期自动清理避免磁盘膨胀。并行合并冲突（9.1 表格）的处理需要在集成测试中覆盖——模拟文件交叉预分析遗漏的场景，验证冲突检测和人工介入流程。
> **[决议]**: 错误处理遵循「优雅降级」原则。缓存校验用 mtime+size 组合 hash。失败 Worktree 保留 7 天后自动清理。AI 统计增加 Catchup 触发次数字段。

## 1. 模块职责与边界

### 1.1 核心职责

| 职责 | 说明 |
|------|------|
| **阶段状态管理** | 维护 Feature 的 8+2 阶段生命周期状态机（8 个主阶段 00-07 + 2 个终态），确保阶段流转合法 |
| **三层规范合并** | Feature 启动时合并 Layer 0 + Layer 1 + Layer 2 规则，生成定制化流程实例 |
| **双模式裁剪** | 根据 Mode（N/I）和 Size（S/M/L）裁剪每个阶段的产出物深度和 Gate 条件 |
| **Gate 触发** | 4 道关键 Gate（Design Ready / Code Ready / Release Ready / Go Live）转换时触发 Gate 评估，调用 M3 GateEngine 执行校验；其余阶段通过 CLI 内联校验或自动流转 |
| **阶段感知** | 为其他模块提供"当前阶段"查询能力，支撑 Hook 校验和 AI 路由 |

### 1.2 模块边界

```
                    ┌─────────────────────────┐
                    │   M1 ProcessEngine       │
                    │                         │
  输入              │  ┌───────────────────┐  │  输出
  ─────────────────▶│  │ Stage FSM         │  │──▶ 当前阶段状态
  constitution.md   │  │ (阶段状态机)       │  │
  config.yaml       │  └────────┬──────────┘  │
  Feature 元数据    │           │              │
                    │  ┌────────▼──────────┐  │
                    │  │ Tailoring Engine   │  │──▶ 裁剪后流程实例
                    │  │ (裁剪引擎)         │  │
                    │  └────────┬──────────┘  │
                    │           │              │
                    │  ┌────────▼──────────┐  │
                    │  │ Spec Merger        │  │──▶ 合并后规范
                    │  │ (规范合并器)        │  │
                    │  └───────────────────┘  │
                    └─────────────────────────┘
                              │ 调用
                    ┌─────────▼───────────────┐
                    │   M3 GateEngine          │
                    └─────────────────────────┘
```

### 1.3 与其他模块的接口

| 调用方 | 接口 | 说明 |
|--------|------|------|
| M1 → M3 | `evaluateGate(stage, featureId, conditions?)` | 阶段转换时触发 Gate 评估，conditions 可选（默认从裁剪结果加载） |
| M1 → M2 | `getCoverage(featureId, type)` | 获取覆盖率数据（Gate 条件之一） |
| M3 → M1 | `getCurrentStage(featureId)` | Gate 校验时查询当前阶段 |
| M5 → M1 | `getCurrentStage(featureId)` | AI 编排查询当前阶段，用于 Agent 路由 |
| M7 → M1 | `getCurrentStage(featureId)` | Hook 校验查询当前阶段，用于 commit-msg 阶段感知 |

### 1.4 专家会审记录

> **[Arch]**: M1 是编排中枢，但不应承担具体校验逻辑。Gate 评估委托给 M3，覆盖率计算委托给 M2，M1 只负责"何时触发"和"状态流转"。
> **[DevEff]**: 阶段感知是多个模块的共享需求（Hook、AI 路由、度量采集），M1 必须提供高效的查询接口，避免每次都解析文件。
> **[QA]**: 状态机的合法性是整个体系的基石——非法的阶段跳转必须被拒绝。建议状态机实现为显式 FSM，而非隐式 if-else 链。
> **[决议]**: M1 定位为"编排中枢 + 状态守护"，不做具体校验。状态机采用显式 FSM 实现，状态持久化到 `stage-state.json`。

## 2. 阶段状态机设计

### 2.1 状态定义

```
状态枚举（StageId）:

  INIT        = "00_init"
  SPECIFY     = "01_specify"
  DESIGN      = "02_design"
  PLAN        = "03_plan"
  IMPLEMENT   = "04_implement"
  VERIFY      = "05_verify"
  WRAP_UP     = "06_wrap_up"
  RELEASE     = "07_release"
  DONE        = "08_done"        # 终态：发布成功
  CANCELLED   = "09_cancelled"   # 终态：Feature 取消
```

每个阶段的内部状态（StageStatus）：

| 状态 | 说明 |
|------|------|
| `pending` | 尚未进入该阶段 |
| `active` | 当前正在执行 |
| `gate_checking` | Gate 评估中 |
| `gate_passed` | Gate 已通过，等待进入下一阶段 |
| `gate_failed` | Gate 未通过，需修正后重新提交 |
| `completed` | 阶段已完成（Gate 通过且已进入下一阶段） |
| `skipped` | 被裁剪跳过（仅限特定场景，如 Hotfix） |

### 2.2 状态转换图

```
                         ┌──────────┐
                         │  INIT    │
                         │ (00)     │
                         └────┬─────┘
                              │ Gate: 目录就绪 + Mode/Size/端确认
                              ▼
                         ┌──────────┐
                         │ SPECIFY  │
                         │ (01)     │
                         └────┬─────┘
                              │ Gate: DoR Sign-off + 无歧义 + ID已分配
                              ▼
                         ┌──────────┐
                         │ DESIGN   │
                         │ (02)     │
                         └────┬─────┘
                              │ Gate: Design Review + API覆盖率=100%
                              ▼
                         ┌──────────┐
                         │  PLAN    │
                         │ (03)     │
                         └────┬─────┘
                              │ Gate: Task Review + 覆盖率=100%
                              ▼
                         ┌──────────┐
                         │IMPLEMENT │
                         │ (04)     │
                         └────┬─────┘
                              │ Gate: Code CR + PR合规率=100%
                              ▼
                         ┌──────────┐
                         │ VERIFY   │
                         │ (05)     │
                         └────┬─────┘
                              │ Gate: UAT Sign-off + Test覆盖率=100%
                              ▼
                         ┌──────────┐
                         │ WRAP_UP  │
                         │ (06)     │
                         └────┬─────┘
                              │ Gate: 归档清单 + 实现覆盖率=100%
                              ▼
                         ┌──────────┐
                         │ RELEASE  │
                         │ (07)     │
                         └────┬─────┘
                              │ Gate: Smoke Test + 指标无异常

                         ┌──────────┐
                         │  DONE    │
                         │ (终态)   │
                         └──────────┘

```

### 2.3 转换规则表

| 当前阶段 | 目标阶段 | 触发条件 | Gate 条件 | Gate Owner |
|---------|---------|---------|----------|------------|
| — | INIT | `spec-init` 命令或手动创建 | — | — |
| INIT | SPECIFY | CLI 内联校验通过 | 目录就绪 + Mode/Size/端确认 + FEAT 缩写已注册（内联到 CLI，非独立 Gate） | Tech Lead |
| SPECIFY | DESIGN | CLI 内联校验通过 | 无歧义标记 + 所有 FR/NFR 已分配 ID（内联到 CLI，非独立 Gate） | Tech Lead |
| DESIGN | PLAN | ★ Gate 1: Design Ready | spec_signoff + design_review + api_coverage + sca_design | Tech Lead / Architect |
| PLAN | IMPLEMENT | 自动流转 | Plan 校验合并到 Gate 2（Task 覆盖率在 Code Ready 时统一校验） | Tech Lead |
| IMPLEMENT | VERIFY | ★ Gate 2: Code Ready | task_coverage + task_compliance + code_cr + pr_task_link + code_coverage + sca_code | Tech Lead / Peer |
| VERIFY | WRAP_UP | 自动流转 | Wrap-up 校验合并到 Gate 3（矩阵终审在 Release Ready 时统一校验） | — |
| WRAP_UP | RELEASE | ★ Gate 3: Release Ready | uat_signoff + test_coverage_fr + security_clear + matrix_final + sca_test | QA Lead + PM |
| RELEASE | DONE | ★ Gate 4: Go Live | smoke_test + release_signoff | Tech Lead + Ops |
| 任意 | CANCELLED | Feature 取消决策 | Tech Lead + PM 审批 | Tech Lead |

### 2.4 转换约束（不变量）

```
规则 1: 严格顺序——不允许跳过阶段（INIT→SPECIFY→...→DONE）
规则 2: 不可回退——已完成的阶段不可重新进入（变更走 RFC 流程）
规则 3: Gate 必过——阶段转换必须经过 Gate 评估，无 Gate 旁路
规则 4: 单活跃——同一 Feature 同一时刻只有一个阶段处于 active 状态
规则 5: 终态不可逆——DONE 和 CANCELLED 为终态，不可再转换
```

### 2.5 内部状态转换（单阶段内）

```
pending ──▶ active ──▶ gate_checking ──▶ gate_passed ──▶ completed
                              │                              ▲
                              ▼                              │
                        gate_failed ──(修正后重新提交)────────┘
```

### 2.6 专家会审记录

> **[Arch]**: 状态机设计清晰。补充一点：`gate_checking` 状态需要考虑超时——如果 Gate 评估脚本异常退出，不能让 Feature 永远卡在 `gate_checking`。建议增加超时自动回退到 `active` 的机制。
> **[DevEff]**: `skipped` 状态需要严格限制使用场景。需求文档明确"不跳过阶段，而是调节深度"，所以 `skipped` 仅用于 Hotfix 等不适用场景，正常流程中不应出现。
> **[QA]**: 转换约束的"不可回退"规则很关键。实际中如果 Design 阶段发现 Specify 有遗漏，应该走 RFC 变更流程在当前阶段修正，而不是回退到 Specify。这需要在文档中明确说明。
> **[决议]**: 增加 `gate_checking` 超时机制（默认 60s）。`skipped` 仅限 Hotfix 场景。回退需求通过 RFC 变更流程处理，不破坏状态机单向性。

## 3. 三层规范合并引擎

### 3.1 三层规范体系回顾

```
Layer 0（通用框架）    Layer 1（模式×规模）    Layer 2（端特有）
┌─────────────────┐  ┌─────────────────┐  ┌─────────────────┐
│ 8+2 阶段定义     │  │ Mode N / Mode I  │  │ APP / PC / H5   │
│ 追踪体系规则     │  │ Size S / M / L   │  │ Backend / ...   │
│ 横切机制定义     │  │ 产出物深度调节    │  │ 端特有检查项     │
│ ID 格式规范      │  │ Gate 条件调节     │  │ 端特有产出物     │
└────────┬────────┘  └────────┬────────┘  └────────┬────────┘
         │                    │                    │
         └────────────────────┼────────────────────┘
                              ▼
                    ┌─────────────────────┐
                    │  合并后流程实例       │
                    │  (Feature-specific)  │
                    └─────────────────────┘
```

### 3.2 数据源定义

| Layer | 数据源文件 | 格式 | 加载时机 |
|-------|-----------|------|---------|
| Layer 0 | `.spec-first/config.yaml` → `layer0` 节 | YAML | 项目初始化时加载，全局缓存 |
| Layer 1 | `.spec-first/config.yaml` → `layer1.{mode}.{size}` 节 | YAML | Feature 启动时按 Mode×Size 选择 |
| Layer 2 | `platform-rules/<platform>-rules.md` | YAML front-matter | Feature 启动时按涉及端动态加载 |

### 3.3 合并算法

```
function mergeSpecs(featureMeta):
  // Step 1: 加载 Layer 0（基线）
  base = loadLayer0()

  // Step 2: 应用 Layer 1 规则（Mode × Size）
  layer1Rules = loadLayer1(featureMeta.mode, featureMeta.size)
  merged = applyLayer1(base, layer1Rules)

  // Step 3: 逐端合并 Layer 2
  for platform in featureMeta.platforms:
    layer2Rules = loadLayer2(platform)
    merged = applyLayer2(merged, layer2Rules)

  // Step 4: 冲突检测与解决
  conflicts = detectConflicts(merged)
  if conflicts.length > 0:
    merged = resolveConflicts(merged, conflicts)

  return merged
```

### 3.4 合并规则详述

**Layer 1 应用规则**（覆盖式）：

| 规则类型 | 合并策略 | 示例 |
|---------|---------|------|
| 产出物深度 | Layer 1 覆盖 Layer 0 默认值 | Size S: `research.md` 可省略 |
| Gate 条件 | Layer 1 可放宽或收紧 | Size S: 不强制 AC 级覆盖率 |
| 额外活动 | Layer 1 追加到 Layer 0 | Mode I: 追加"历史产物定位" |

**Layer 2 应用规则**（追加式）：

| 规则类型 | 合并策略 | 示例 |
|---------|---------|------|
| entry_criteria | 追加到阶段准入条件 | APP: "兼容性矩阵已确认" |
| deliverables | 追加到阶段产出物 | APP: "UI 适配方案" |
| exit_gate_items | 追加到 Gate 检查项 | APP: "性能预算评审通过" |

### 3.5 冲突解决机制

**多端规则合并时的冲突处理**：

| 冲突类型 | 解决规则 | 实现方式 |
|---------|---------|---------|
| 同类规则重复 | 取并集（全部保留，去重） | `Set.union()` |
| 阈值类冲突 | 各端各自适用，不互相覆盖 | 按 `platform` 字段隔离 |
| 互斥规则 | 标记为 `conflict`，要求 Tech Lead 在 Init 阶段裁定 | 输出冲突报告，阻断直到人工解决 |

**优先级链**：Layer 0 < Layer 1 < Layer 2。Layer 2 覆盖 Layer 0/1 时，必须在端规则文件中标注 `overrides` 字段。

### 3.6 专家会审记录

> **[Arch]**: 合并算法的关键是幂等性——同样的输入（Mode/Size/Platforms）必须产出完全相同的合并结果。建议合并结果缓存到 `stage-state.json` 中，避免重复计算。
> **[DevEff]**: Layer 2 的端规则文件使用 YAML front-matter 嵌入 Markdown，这样端规则既可被脚本解析，也可被人类阅读。但需要明确 YAML Schema，避免格式不一致。
> **[QA]**: 冲突检测必须在 Init 阶段完成，不能让冲突带入后续阶段。建议 Init Gate 增加"规范合并无冲突"作为必要条件。
> **[决议]**: 合并结果缓存到 `stage-state.json`。端规则 YAML Schema 在 `config.yaml` 中定义。Init Gate 增加合并无冲突检查。

## 4. 双模式裁剪引擎

### 4.1 裁剪维度

裁剪引擎基于两个维度组合裁剪流程实例：

| 维度 | 取值 | 裁剪影响 |
|------|------|---------|
| **Mode** | N（New Feature）/ I（Iteration） | 控制是否追加增量活动（历史定位、Impact Analysis、回归验证） |
| **Size** | S / M / L | 控制每个阶段的产出物深度和 Gate 条件严格度 |

### 4.2 Size 判定算法

```
function determineSize(featureMeta):
  dimensions = [
    { name: "modules",    value: featureMeta.moduleCount,    S: [1,2], M: [3,5], L: [6,Inf] },
    { name: "ac_count",   value: featureMeta.acCount,        S: [1,5], M: [6,15], L: [16,Inf] },
    { name: "api_changes", value: featureMeta.apiChangeCount, S: [0,1], M: [2,5], L: [6,Inf] },
    { name: "data_model", value: featureMeta.dataModelLevel,  S: 0, M: 1, L: 2 },
    { name: "cross_team", value: featureMeta.crossTeamCount,  S: 0, M: 1, L: [2,Inf] }
  ]
  // 5 个维度取最高级别
  return max(dimensions.map(d => classify(d)))
```

### 4.3 Size 裁剪矩阵

| 阶段 | 产出物/活动 | S | M | L |
|------|-----------|---|---|---|
| 01 Specify | `spec.md` | 简版（核心 FR + AC） | 标准版 | 完整版（含 NFR 量化） |
| 01 Specify | AC 级覆盖率校验 | 不强制 | 强制 ≥90% | 强制 ≥90% |
| 02 Design | `research.md` | 可省略 | 标准版 | 完整版（含 Trade-off） |
| 02 Design | `contracts/*.yaml` | 可内嵌 design.md | 独立文件 | 独立文件 + Mock Server |
| 02 Design | `data-model.md` | 无变更可省略 | 字段级变更必须 | 完整 ERD + 状态机 |
| 02 Design | Architect 参与 | 可选 | 推荐 | 必须 |
| 03 Plan | 任务粒度 | 1-3 个 TASK | 5-15 个 TASK | 15+ 个 TASK |
| 04 Implement | 并行策略 | 串行 | 分支并行 | Git Worktree 隔离 |
| 05 Verify | 安全扫描 | OWASP + SCA | + SAST | + DAST + 渗透测试 |
| 05 Verify | NFR-SEC 叠加 | 上浮至 SAST | — | — |
| 07 Release | 部署策略 | Rolling Update | Blue-Green | Canary |
| 07 Release | 观察窗口 | 30 分钟 | 2 小时 | 24 小时 |

### 4.4 Mode 裁剪规则

**核心原则**：Mode N 和 Mode I 共享相同的 8 阶段骨架（00 Init ~ 07 Release），Mode I 在特定阶段追加增量活动。

| 阶段 | Mode N（全新） | Mode I（迭代）追加活动 |
|------|--------------|---------------------|
| 00 Init | 标准流程 | + 历史产物定位（找到已有 spec/plan/contracts/code） |
| 01 Specify | 标准流程 | + Impact Analysis → `impact-analysis.md` |
| 02 Design | 标准流程 | + 增量设计（diff 模式，仅描述变更部分） |
| 03 Plan | 标准流程 | + 回归任务规划（标记需回归验证的模块） |
| 04 Implement | 标准流程 | + 增量实现（关注变更点，保护已有功能） |
| 05 Verify | 标准流程 | + 回归验证 → `regression-report.md` |
| 06 Wrap-up | 标准流程 | + 增量归档（仅归档变更部分，引用历史产物） |

**Mode I 子类型**（影响 Impact Analysis 深度）：

| 子类型 | 定义 | Impact Analysis 深度 |
|--------|------|---------------------|
| Enhancement | 功能增强 | 标准：影响范围 + 风险评估 |
| Optimization | 性能/体验优化 | 轻量：仅性能基线对比 |
| Bug Fix | 缺陷修复 | 标准：根因分析 + 回归范围 |
| Refactoring | 代码重构 | 深度：全量影响分析 + 行为等价验证 |

### 4.5 专家会审记录

> **[Arch]**: 裁剪引擎的核心是"不跳过阶段，只调节深度"。这意味着裁剪结果是一个"产出物深度矩阵"，而不是"阶段跳过列表"。数据结构上，每个阶段都有 `required_deliverables` 和 `optional_deliverables` 两个列表。
> **[DevEff]**: Size 判定的 5 维度取最高级别规则简单明确，但实际中可能出现"4 个维度是 S，1 个维度是 L"的情况。建议增加"维度权重"或"人工覆盖"机制，允许 Tech Lead 在 Init 阶段手动调整 Size。
> **[QA]**: Mode I 的回归验证是质量保障的关键。Refactoring 子类型的"行为等价验证"需要在详细设计中明确——是通过 Snapshot Testing、Contract Testing 还是其他方式实现。
> **[决议]**: 裁剪结果为"深度矩阵"而非"跳过列表"。Size 判定支持 Tech Lead 手动覆盖（记录覆盖理由到 `stage-state.json`）。Refactoring 的行为等价验证方式在 M3 详细设计中定义。

## 5. Gate 触发与评估机制

### 5.1 M1 与 M3 的职责分工

| 职责 | M1 ProcessEngine | M3 GateEngine |
|------|-----------------|---------------|
| Gate 触发时机 | ✅ 决定何时触发 | — |
| Gate 条件定义 | ✅ 从裁剪结果中提取 | — |
| Gate 条件评估 | — | ✅ 执行具体校验 |
| Gate 结果处理 | ✅ 根据结果决定状态转换 | — |
| Gate 历史记录 | — | ✅ 写入 gate-history.jsonl |

### 5.2 Gate 触发流程

```
用户请求阶段转换（如 SPECIFY → DESIGN）
    │
    ▼
M1: 校验转换合法性（FSM 规则）
    │ 不合法 → 拒绝，返回错误
    ▼ 合法
M1: 将当前阶段内部状态设为 gate_checking
    │
    ▼
M1: 从裁剪结果中提取当前阶段的 Gate 条件列表
    │
    ▼
M1: 调用 M3.evaluateGate(stage, featureId, conditions)
    │
    ├── M3 返回 PASS ──▶ M1: 状态设为 gate_passed → completed
    │                         下一阶段设为 active
    │
    ├── M3 返回 FAIL ──▶ M1: 状态设为 gate_failed
    │                         返回失败原因列表
    │
    └── M3 超时(60s) ──▶ M1: 状态回退为 active
                              记录超时事件到日志
```

### 5.3 Gate 条件分类

| 类别 | 说明 | 评估方式 | 示例 |
|------|------|---------|------|
| **自动校验** | 脚本可自动判定 | M3 调用校验脚本 | ID 格式、覆盖率计算、SCA |
| **人工签核** | 需要人类确认 | M3 检查签核记录是否存在 | DoR Sign-off、Design Review |
| **混合型** | 自动校验 + 人工终审 | 自动校验通过后等待人工确认 | Code CR（自动 lint + 人工 review） |

### 5.4 专家会审记录

> **[Arch]**: M1 和 M3 的职责分工清晰——M1 管"何时"，M3 管"如何"。Gate 超时回退机制是必要的安全阀，但需要记录超时事件以便排查。
> **[DevEff]**: Gate 条件的"人工签核"类需要明确签核记录的存储位置和格式。建议统一存储在对应阶段的产出物中（如 `spec.md` 末尾的 Sign-off 区块），M3 通过解析文件检测签核是否存在。
> **[QA]**: Gate 失败后的修正-重提交循环需要有次数限制吗？从质量角度看不应限制（必须修到通过为止），但从度量角度看，Gate 失败次数是"返工率"的重要数据源。
> **[决议]**: Gate 失败不限次数，但每次失败都记录到 `gate-history.jsonl`，作为返工率的数据源。签核记录统一存储在产出物文件的 Sign-off 区块中。

### 7.4 核心类型定义

```typescript
/** 阶段枚举 */
type StageId =
  | "00_init" | "01_specify" | "02_design" | "03_plan"
  | "04_implement" | "05_verify" | "06_wrap_up" | "07_release"
  | "08_done" | "09_cancelled";

/** 阶段内部状态 */
type StageStatus =
  | "pending" | "active" | "gate_checking"
  | "gate_passed" | "gate_failed" | "completed" | "skipped";

/** 模式 */
type Mode = "N" | "I";

/** 规模 */
type Size = "S" | "M" | "L";

/** Mode I 子类型 */
type ModeISubType = "Enhancement" | "Optimization" | "BugFix" | "Refactoring";

/** 平台 */
type Platform = "APP" | "PC" | "H5" | "Backend" | "MiniProgram";

/** Feature 元数据（initFeature 输入） */
interface FeatureMeta {
  featureId: string;           // 如 "001-user-auth"
  featAbbr: string;            // 如 "AUTH"
  mode: Mode;
  size?: Size;                 // 可选，未提供时自动判定
  sizeOverride?: Size;         // Tech Lead 手动覆盖
  sizeOverrideReason?: string; // 覆盖理由
  platforms: Platform[];
  modeISubType?: ModeISubType; // 仅 Mode I 时必填
  moduleCount: number;         // Size 判定维度
  acCount: number;
  apiChangeCount: number;
  dataModelLevel: 0 | 1 | 2;  // 0=无变更, 1=字段级, 2=表级
  crossTeamCount: number;
  createdBy: string;
}

/** 阶段转换结果 */
interface TransitionResult {
  success: boolean;
  fromStage: StageId;
  toStage: StageId;
  gateResult?: GateResult;
  failReasons?: GateFailItem[];  // 失败时的具体原因列表
  timestamp: string;             // ISO 8601
}

/** Gate 评估结果 */
interface GateResult {
  verdict: "PASS" | "FAIL" | "TIMEOUT";
  checkedItems: GateCheckItem[];
  duration_ms: number;
}

/** Gate 单项检查结果 */
interface GateCheckItem {
  conditionId: string;           // 如 "design_review", "sca_design"
  type: "auto" | "manual" | "hybrid";
  passed: boolean;
  message?: string;              // 失败时的说明
}

type GateFailItem = GateCheckItem & { passed: false };

/** Feature 完整状态快照 */
interface FeatureState {
  version: string;               // schema 版本，如 "1.0"
  featureId: string;
  featureMeta: FeatureMeta;
  currentStage: StageId;
  stages: Record<StageId, StageState>;
  mergedSpec: MergedSpecMeta;
}

/** 单阶段状态 */
interface StageState {
  status: StageStatus;
  enteredAt?: string;            // ISO 8601
  completedAt?: string;
  gateAttempts: number;          // Gate 尝试次数（用于返工率统计）
  lastGateResult?: GateResult;
}

/** 裁剪后的阶段配置 */
interface TailoredStageConfig {
  stageId: StageId;
  requiredDeliverables: Deliverable[];   // 必须产出物
  optionalDeliverables: Deliverable[];   // 可选产出物
  gateConditions: GateCondition[];       // Gate 检查项
  modeIAdditions?: Activity[];           // Mode I 追加活动
}

/** 产出物定义 */
interface Deliverable {
  id: string;                  // 如 "spec.md", "research.md"
  depth: "minimal" | "standard" | "full";  // 深度级别
  description: string;
  platform?: Platform;         // Layer 2 端特有产出物
}

/** Gate 条件定义 */
interface GateCondition {
  conditionId: string;
  type: "auto" | "manual" | "hybrid";
  description: string;
  threshold?: number;          // 阈值类条件（如覆盖率 ≥ 80%）
  platform?: Platform;         // Layer 2 端特有条件
}

/** 活动定义（Mode I 追加） */
interface Activity {
  id: string;
  name: string;
  stage: StageId;
  output?: string;             // 产出物文件名
}
```

### 7.5 专家会审记录

> **[Arch]**: 接口设计遵循单一职责——ProcessEngine 对外暴露 6 个方法，TailoringEngine 和 SpecMerger 作为内部组件不对外暴露。`getCurrentStage` 是高频调用（Hook、AI 路由、度量采集都依赖），必须有内存缓存，避免每次读取 `stage-state.json`。
> **[DevEff]**: `requestTransition` 返回 `TransitionResult` 包含完整的 Gate 评估详情，开发者可以清楚知道哪些条件未通过。建议 CLI 输出时按 `failReasons` 逐项列出，附带修复建议。
> **[QA]**: `GateCheckItem` 的 `type` 字段区分了自动/人工/混合三类校验，这对测试策略很重要——自动类可以单元测试覆盖，人工类需要集成测试模拟签核记录。`gateAttempts` 字段是返工率的数据源，必须在每次 Gate 评估时递增。
> **[决议]**: `getCurrentStage` 实现内存缓存 + 文件变更监听（`fs.watch`）。CLI 输出 Gate 失败详情时附带修复建议。`gateAttempts` 在 `gate_checking` 状态进入时递增。

## 8. 错误处理与边界场景

### 8.1 错误分类与处理策略

| 错误类别 | 触发场景 | 处理策略 | 用户提示 |
|---------|---------|---------|---------|
| **配置错误** | `config.yaml` 格式错误、缺少必要字段 | Fail-Fast，阻断操作 | 输出具体字段路径 + 期望格式 |
| **状态文件损坏** | `stage-state.json` 格式错误或数据不一致 | 尝试自动修复，失败则阻断 | 输出损坏位置 + 建议手动修复命令 |
| **非法状态转换** | 请求跳过阶段、回退阶段、重复进入 | 拒绝操作，返回合法转换列表 | 输出当前阶段 + 允许的目标阶段 |
| **Gate 评估超时** | M3 GateEngine 响应超过 60s | 回退到 `active` 状态，记录超时事件 | 提示重试或检查校验脚本 |
| **规范合并冲突** | 多端 Layer 2 规则互斥 | 标记冲突，阻断直到人工裁定 | 输出冲突详情 + 需要裁定的规则对 |
| **文件缺失** | 产出物文件不存在（如 `spec.md` 未创建） | Gate 校验时报告缺失项 | 输出缺失文件列表 + 创建模板命令 |

### 8.2 边界场景处理

#### 场景 1：Feature 取消后的资源清理

```
触发：cancelFeature(featureId, reason)
处理流程：
  1. 校验当前阶段不是终态（DONE/CANCELLED）
  2. 将当前活跃阶段状态设为 completed（标记 cancelledAt）
  3. 将所有 pending 阶段状态设为 skipped
  4. 设置 currentStage = "09_cancelled"
  5. 记录取消原因到 stage-state.json
  6. 不删除任何产出物文件（保留审计轨迹）
```

#### 场景 2：Gate 评估期间进程崩溃

```
触发：gate_checking 状态下进程异常退出
检测：下次启动时检查 stage-state.json 中是否有阶段处于 gate_checking
恢复：
  1. 检查 gate_checking 状态的持续时间
  2. 若超过 gate_timeout_ms（默认 60s），自动回退到 active
  3. 若未超时，保持 gate_checking 状态，等待用户重新触发
  4. 记录异常恢复事件到日志
```

#### 场景 3：config.yaml 变更后的流程实例一致性

```
触发：项目 config.yaml 被修改（如 Gate 条件调整）
影响：已创建的 Feature 流程实例可能与新配置不一致
处理：
  1. 已完成阶段：不受影响（历史数据不可变）
  2. 当前活跃阶段：下次 Gate 评估时使用新配置
  3. 未来阶段：自动使用新配置
  4. mergedSpec 中记录 layer0Hash，检测到 hash 变化时提示用户
```

### 8.3 状态恢复机制

```
启动时自检流程（ProcessEngine.selfCheck）：

  1. 读取 stage-state.json
     ├── 文件不存在 → 报错：Feature 未初始化
     └── 文件存在 → 继续

  2. 校验 JSON Schema 合法性
     ├── 不合法 → 尝试从 Git 历史恢复最近合法版本
     └── 合法 → 继续

  3. 校验状态一致性
     ├── 多个阶段处于 active → 保留最后一个，其余设为 completed
     ├── 有阶段处于 gate_checking 且超时 → 回退到 active
     └── currentStage 与 stages 不一致 → 以 stages 为准修正 currentStage

  4. 校验 mergedSpec hash
     ├── layer0Hash 不匹配 → 警告：配置已变更，建议重新合并
     └── 匹配 → 正常
```

### 8.4 专家会审记录

> **[Arch]**: 状态恢复机制是系统健壮性的关键。`selfCheck` 应在每次 ProcessEngine 实例化时自动执行，开销可控（仅读取一个 JSON 文件 + 几项校验）。Git 历史恢复是最后手段，正常情况下不应触发。
> **[DevEff]**: 错误提示的质量直接影响开发者体验。每个错误必须包含：(1) 发生了什么，(2) 为什么发生，(3) 如何修复。避免输出原始堆栈信息，转化为可操作的修复建议。
> **[QA]**: 场景 2（进程崩溃恢复）和场景 3（配置变更一致性）需要专门的集成测试用例。建议在测试计划中增加「异常恢复」测试套件，覆盖所有 `selfCheck` 分支。
> **[决议]**: `selfCheck` 在 ProcessEngine 构造时自动执行。错误提示遵循「What-Why-How」三段式。异常恢复测试套件纳入 M1 测试计划。

## 1. 模块职责与边界

### 1.1 核心职责

| 职责 | 说明 |
|------|------|
| **ID 格式校验** | 8 种 ID 类型的正则校验，确保格式合规 |
| **ID 注册与分配** | FEAT 缩写注册、ID 唯一性校验、序号自增管理 |
| **追踪矩阵 CRUD** | 矩阵的创建、渐进填充、查询、终审校验 |
| **矩阵格式自适应** | 根据 Size 自动选择 Markdown（S）或 YAML（M/L）格式 |
| **覆盖率计算** | 9 种覆盖率/合规率指标的计算，支撑 Gate 校验 |
| **ID 追踪链查询** | 基于 ID 的正向/反向追踪链查询，支撑 Impact Analysis |

| M6 → M2 | `getCoverage(featureId, type)` | 度量采集时获取覆盖率数据 |

### 1.4 专家会审记录

> **[Arch]**: M2 是数据中枢——ID 和追踪矩阵是整个体系的数据基础。M2 不做流程编排（M1 的职责），不做 Gate 评估（M3 的职责），只负责数据的 CRUD 和计算。
> **[DevEff]**: 覆盖率计算是高频调用（每次 Gate 都会触发），必须高效。建议矩阵数据在内存中维护索引，避免每次都解析文件。增量计算优于全量计算。
> **[QA]**: ID 格式校验是第一道防线——格式不合法的 ID 不应进入追踪矩阵。建议 ID 校验在产出物写入时就执行（Layer A PostToolUse），而不是等到 Gate 阶段才发现。
> **[决议]**: M2 定位为「数据中枢」，提供 ID 校验 + 矩阵 CRUD + 覆盖率计算三大能力。覆盖率计算支持增量模式。ID 校验在写入时和 Gate 时双重执行。

## 2. ID 体系设计

### 2.1 ID 类型总览

| 前缀 | 全称 | 格式 | 示例 | 定义阶段 | 粒度 |
|------|------|------|------|---------|------|
| `FR` | Functional Requirement | `FR-<FEAT>-NNN` | `FR-AUTH-001` | 01 Specify | 条目级 |
| `NFR` | Non-Functional Req | `NFR-<DIM>-NNN` | `NFR-SEC-001` | 01 Specify | 条目级 |
| `DS` | Design Section | `DS-<FEAT>-NNN` | `DS-AUTH-001` | 02 Design | 条目级 |
| `API` | API Endpoint | `API-<SVC>-NNN` | `API-AUTH-001` | 02 Design | 条目级 |
| `TASK` | Implementation Task | `TASK-<FEAT>-NNN` | `TASK-AUTH-001` | 03 Plan | 条目级 |
| `TC` | Test Case | `TC-<LVL>-<FEAT>-NNN` | `TC-E2E-AUTH-001` | 05 Verify | 条目级 |
| `ADR` | Architecture Decision | `ADR-NNN` | `ADR-001` | 02 Design | 文件级 |
| `RFC` | Request for Change | `RFC-NNN` | `RFC-001` | 横切机制 | 文件级 |

**粒度说明**：
- **条目级**：一个文件包含多个 ID（如 `spec.md` 包含多个 FR）
- **文件级**：一个文件对应一个 ID（如 `rfc/001-*.rfc.md` 对应 `RFC-001`）

### 2.2 正则校验规则

```javascript
const ID_PATTERNS = {
  FR:   /^FR-[A-Z][A-Z0-9]{1,15}-\d{3}$/,
  NFR:  /^NFR-[A-Z][A-Z0-9]{1,7}-\d{3}$/,
  DS:   /^DS-[A-Z][A-Z0-9]{1,15}-\d{3}$/,
  API:  /^API-[A-Z][A-Z0-9]{1,15}-\d{3}$/,
  TASK: /^TASK-[A-Z][A-Z0-9]{1,15}-\d{3}$/,
  TC:   /^TC-(UT|IT|E2E|ST)-[A-Z][A-Z0-9]{1,15}-\d{3}$/,
  ADR:  /^ADR-\d{3}$/,
  RFC:  /^RFC-\d{3}$/,
};

// 通用 ID 匹配（任意类型）
const ANY_ID = /^(FR|NFR|DS|API|TASK|TC|ADR|RFC)-/;
```

**变量说明**：

| 变量 | 约束 | 说明 |
|------|------|------|
| `<FEAT>` | `[A-Z][A-Z0-9]{1,15}` | Feature 缩写，2-16 位，首位必须为字母 |
| `<DIM>` | `[A-Z][A-Z0-9]{1,7}` | NFR 维度缩写，2-8 位 |
| `<SVC>` | `[A-Z][A-Z0-9]{1,15}` | 服务/模块缩写，与 FEAT 规则相同 |
| `<LVL>` | `UT\|IT\|E2E\|ST` | 测试级别，固定枚举 |
| `NNN` | `\d{3}` | 三位数字序号，从 001 递增 |

### 2.3 ID 声明位置与解析规则

| ID 类型 | 声明文件 | 声明格式 | 解析方式 |
|---------|---------|---------|---------|
| FR | `spec.md` | `#### FR-FEAT-NNN: 标题` | Markdown H4 标题解析 |
| NFR | `spec.md` | `#### NFR-DIM-NNN: 标题` | Markdown H4 标题解析 |
| DS | `design.md` | `### DS-FEAT-NNN: 标题` | Markdown H3 标题解析 |
| API | `contracts/*.yaml` | `operationId: API-SVC-NNN` | YAML 字段解析 |
| TASK | `tasks.md` | `#### TASK-FEAT-NNN: 标题` | Markdown H4 标题解析 |
| TC | `tests/*.test.md` | `#### TC-LVL-FEAT-NNN: 标题` | Markdown H4 标题解析 |
| ADR | `design.md` 或独立文件 | 文件名 `ADR-NNN-*.md` | 文件名解析 |
| RFC | `rfc/NNN-*.rfc.md` | 文件名 `RFC-NNN` | 文件名解析 |

**引用（traces）声明格式**：

| 产出物 | traces 声明方式 | 示例 |
|--------|---------------|------|
| `tasks.md` | `- **traces**: FR-AUTH-001, FR-AUTH-002` | TASK 引用 FR/NFR |
| `tests/*.test.md` | `- **verifies**: FR-AUTH-001/AC-1` | TC 引用 FR/AC 或 NFR |
| `contracts/*.yaml` | `x-traces: [FR-AUTH-001]` | API 引用 FR |
| `design.md` | `- **traces**: [FR-AUTH-001]` | DS 引用 FR |
| 代码注释 | `// implements: TASK-AUTH-001, traces: FR-AUTH-001` | 代码引用 TASK/FR |
| PR 描述 | `Implements: TASK-AUTH-001` | PR 引用 TASK |
| Commit message | `[TASK-AUTH-001] 提交描述` | Commit 引用 TASK |

### 2.4 ID 生命周期与不变量

```
ID 生命周期：

  Created ──▶ Active ──▶ Accepted (🎯)
                │
                └──▶ Cancelled (🚫)  // 需求废弃，ID 不可复用
```

**不变量（Invariants）**：

```
规则 1: 唯一性——同一 Feature 内同类型 ID 不可重复
规则 2: 稳定性——ID 一次分配，终身不改；废弃后不可复用
规则 3: 序号连续——同类型 ID 序号从 001 递增，不跳号
规则 4: 先注册后使用——FEAT 缩写必须先在注册表中登记
规则 5: 阶段约束——ID 只能在对应阶段创建（FR 在 01，TASK 在 03）
规则 6: 引用合法——traces/verifies 引用的 ID 必须已存在
```

### 2.5 专家会审记录

> **[Arch]**: ID 体系的核心价值是「可解析」——所有 ID 都能通过正则提取，不依赖上下文。这使得校验脚本可以纯文本处理，无需理解 Markdown 语义。DS ID 的引入替代了非结构化的章节号引用（如 `§2.1`），确保 Design 层追踪链不随编辑漂移。
> **[DevEff]**: 8 种 ID 类型看似复杂，但开发者日常只接触 TASK 和 FR——commit message 写 `[TASK-AUTH-001]`，代码注释写 `traces: FR-AUTH-001`。其余 ID 由 Tech Lead 和 QA 管理。建议 CLI 提供 `spec-id next FR AUTH` 命令自动生成下一个可用 ID。
> **[QA]**: ID 不变量的「序号连续」规则在实际中可能被打破——如果 FR-AUTH-002 被废弃，FR-AUTH-003 仍然存在，序号就不连续了。建议改为「序号单调递增」，允许跳号但不允许回退。
> **[决议]**: 序号规则调整为「单调递增，不可回退」，允许因废弃导致的跳号。CLI 提供 `spec-id next` 命令。DS ID 正式纳入追踪体系。

## 3. FEAT 注册表管理

### 3.1 注册表文件格式

注册表文件位于 `specs/.feat-registry.md`，采用 Markdown 表格格式：

```markdown
# FEAT 注册表

| FEAT 缩写 | 全称 | 注册人 | 注册日期 | 状态 |
|-----------|------|--------|---------|------|
| AUTH | Authentication | Leo | 2026-01-15 | Active |
| AUTHZ | Authorization | Leo | 2026-01-15 | Active |
| PAY | Payment | Leo | 2026-01-20 | Active |
| ORDER | Order Management | Leo | 2026-02-01 | Deprecated |
```

### 3.2 治理规则

| # | 规则 | 校验时机 | 违规处理 |
|---|------|---------|---------|
| R1 | FEAT 缩写全局唯一 | Init 内联校验 | 冲突则阻塞，要求更换缩写 |
| R2 | 先注册后使用 | ID 校验时 | 未注册的 FEAT 缩写拒绝通过 |
| R3 | 禁止歧义缩写 | 注册时 | 同一业务域不得注册多个缩写 |
| R4 | 废弃不复用 | 注册时 | Deprecated 状态的缩写不可被新 Feature 使用 |
| R5 | 格式合规 | 注册时 | 缩写必须匹配 `[A-Z][A-Z0-9]{1,15}` |

### 3.3 注册表操作

```
注册新 FEAT 缩写：
  1. 校验格式合规（R5）
  2. 校验全局唯一（R1）——扫描注册表，包括 Deprecated 条目
  3. 校验无歧义（R3）——检查是否与已有缩写语义重叠
  4. 追加新行到注册表，状态设为 Active
  5. 写入 Git（注册表变更需提交）

废弃 FEAT 缩写：
  1. 校验该 FEAT 下无 Active 状态的 FR/NFR/TASK 等 ID
  2. 将注册表中对应行状态改为 Deprecated
  3. 该缩写永久保留在注册表中，不可删除
```

### 3.4 专家会审记录

> **[Arch]**: 注册表采用 Markdown 表格而非 YAML/JSON，是因为它需要人类可读且 Git diff 友好。注册表是低频写入（仅 Init 阶段），高频读取（每次 ID 校验），性能不是瓶颈。
> **[DevEff]**: `spec-first init` 命令应自动引导 FEAT 缩写注册，避免开发者手动编辑注册表。注册冲突时给出已有缩写列表，帮助选择。
> **[QA]**: R3「禁止歧义缩写」是最难自动化的规则——「AUTH」和「AUTHN」是否歧义需要人工判断。建议自动化检查前缀包含关系（如 AUTH 是 AUTHZ 的前缀），人工确认语义是否重叠。
> **[决议]**: 注册表保持 Markdown 格式。CLI 自动引导注册。歧义检查采用「前缀包含 + 人工确认」双重机制。

## 4. 追踪矩阵设计

### 4.1 矩阵结构（8 列标准）

| 列 | 字段名 | 填充阶段 | 说明 |
|----|--------|---------|------|
| 1 | FR/NFR ID | 01 Specify | 需求标识，矩阵的行主键 |
| 2 | Description | 01 Specify | 需求简述 |
| 3 | Design Ref | 02 Design | DS-FEAT-NNN 引用 |
| 4 | API/Data Ref | 02 Design | API-SVC-NNN 或 Data Model 引用 |
| 5 | Task Ref | 03 Plan | TASK-FEAT-NNN 引用 |
| 6 | Test Case Ref | 05 Verify | TC-LVL-FEAT-NNN 引用 |
| 7 | PR Ref | 04 Implement | PR 链接或编号 |
| 8 | Status | 渐进更新 | 状态标记（见下方） |

**Status 状态枚举**：

| 状态 | 符号 | 含义 | 出现时机 | 覆盖率计算 |
|------|------|------|---------|-----------|
| Planned | `📋` | 需求已录入，尚未开始设计/实现 | 01 Specify | 计入 Active |
| Implemented | `🔨` | 已有 TASK 和 PR，尚未测试验证 | 04 Implement | 计入 Active |
| Verified | `✅` | 已有 TC 且测试通过 | 05 Verify | 计入 Active |
| Accepted | `🎯` | UAT 签核通过，正式验收 | 06 Wrap-up | 计入 Active |
| Deferred | `⏸️` | 需求推迟到后续版本 | 任意阶段 | **排除出 Active** |
| Cancelled | `🚫` | 需求已取消（需记录取消原因） | 任意阶段 | **排除出 Active** |
| Exception | `⚠️` | 需求因客观原因豁免（已登记 Known Exception List） | 任意阶段 | **排除出 Active** |
| Not Implemented | `❌` | 需求未被实现（遗漏） | 06 Wrap-up | 计入 Active（负面） |

> **覆盖率分母定义**：`Active = Total - (Deferred + Cancelled + Exception)`。所有覆盖率指标（C1-C9）均基于 Active 集合计算，确保延期、取消和豁免的需求不影响覆盖率判定。

### 4.2 矩阵格式自适应

根据 Feature Size 自动选择矩阵存储格式：

| Size | 格式 | 文件名 | 理由 |
|------|------|--------|------|
| S | Markdown 表格 | `traceability-matrix.md` | FR 数量少（≤5），表格直观可读 |
| M/L | YAML | `traceability-matrix.yaml` | FR 数量多，YAML 按 ID 分块，Git 合并友好 |

**格式自动检测算法**：

```
function detectMatrixFormat(featureDir):
  if exists(featureDir + "/traceability-matrix.yaml"):
    return "yaml"
  if exists(featureDir + "/traceability-matrix.md"):
    return "markdown"
  // 新建时根据 Size 决定
  size = getFeatureSize(featureDir)
  return size === "S" ? "markdown" : "yaml"
```

### 4.3 矩阵格式示例

**Markdown 格式（Size S）**：

```markdown
| FR/NFR ID | Description | Design Ref | API/Data Ref | Task Ref | Test Case Ref | PR Ref | Status |
|-----------|-------------|------------|--------------|----------|---------------|--------|--------|
| FR-AUTH-001 | 用户邮箱注册 | DS-AUTH-001 | API-AUTH-001 | TASK-AUTH-001 | TC-E2E-AUTH-001 | #42 | 🎯 |
| FR-AUTH-002 | 用户登录 | DS-AUTH-002 | API-AUTH-002 | TASK-AUTH-002 | TC-E2E-AUTH-002 | #43 | 🎯 |
| NFR-SEC-001 | 密码加密存储 | DS-AUTH-001 | — | TASK-AUTH-003 | TC-ST-AUTH-001 | #42 | 🎯 |
```

**YAML 格式（Size M/L）**：

```yaml
# traceability-matrix.yaml
version: "1.0"
featureId: "001-user-auth"
entries:
  FR-AUTH-001:
    description: "用户邮箱注册"
    designRef: ["DS-AUTH-001"]
    apiDataRef: ["API-AUTH-001"]
    taskRef: ["TASK-AUTH-001"]
    testCaseRef: ["TC-E2E-AUTH-001"]
    prRef: ["#42"]
    status: "accepted"       # accepted | cancelled | not_implemented | ...
  FR-AUTH-002:
    description: "用户登录"
    designRef: ["DS-AUTH-002"]
    apiDataRef: ["API-AUTH-002"]
    taskRef: ["TASK-AUTH-002"]
    testCaseRef: ["TC-E2E-AUTH-002"]
    prRef: ["#43"]
    status: "accepted"
  NFR-SEC-001:
    description: "密码加密存储"
    designRef: ["DS-AUTH-001"]
    apiDataRef: []
    taskRef: ["TASK-AUTH-003"]
    testCaseRef: ["TC-ST-AUTH-001"]
    prRef: ["#42"]
    status: "accepted"
```

### 4.4 渐进填充流程

```
01 Specify ──▶ 创建矩阵，填入所有 FR/NFR ID + Description
                Status = 📝 Draft
                其余列为空

02 Design  ──▶ 填充 Design Ref 列（DS-FEAT-NNN）
                填充 API/Data Ref 列（API-SVC-NNN）

03 Plan    ──▶ 填充 Task Ref 列（TASK-FEAT-NNN）

04 Implement ─▶ 填充 PR Ref 列（PR 链接或编号）
                Status = 🔄 In Progress

05 Verify  ──▶ 填充 Test Case Ref 列（TC-LVL-FEAT-NNN）
                Status = ✅ Verified

06 Wrap-up ──▶ 最终校验：所有列无空（或标记为不适用）
                Status = 🎯 Accepted 或 🚫 Cancelled
```

**填充约束**：
- 每个阶段只能填充对应列，不可提前填充后续阶段的列
- 已填充的列不可清空（废弃需求改 Status 为 🚫，不删除行）
- 一个 FR/NFR 可关联多个 TASK/TC/PR（一对多关系）

### 4.5 专家会审记录

> **[Arch]**: YAML 格式按 FR-ID 分块是关键设计——每个 FR 独立一个 YAML block，最大化 Git 三路合并的成功率。多人同时编辑不同 FR 的矩阵行时，Git 可自动合并。
> **[DevEff]**: 渐进填充的约束需要在 Gate 校验中强制执行——★ Gate 2: Code Ready 时检查 Task Ref 列是否已填充，而不是检查 Test Case Ref 列（那是 05 的事）。这避免了「一次性填完所有列」的反模式。
> **[QA]**: 矩阵的「不可清空」规则很重要——它保证了审计轨迹。如果需求变更导致 FR 废弃，应该保留该行并标记 🚫，而不是删除。这样在 Wrap-up 复盘时可以看到完整的需求演变历史。
> **[决议]**: YAML 按 FR-ID 分块。渐进填充在 Gate 中按阶段校验。废弃需求保留行标记 🚫。矩阵变更历史通过 Git 版本控制追溯。

## 5. 覆盖率计算引擎

### 5.1 覆盖率指标体系

#### 正向覆盖率（需求是否被充分实现）

| # | 指标 | 公式 | 校验阶段 | 阈值 |
|---|------|------|---------|------|
| C1 | Task 覆盖率 | Active FR∪NFR with ≥1 TASK / Active FR+NFR | 03 Plan | = 100% |
| C2 | API 覆盖率 | FR(需API) with ≥1 API / Total FR(需API) | 02 Design | = 100% |
| C3 | Test 覆盖率(FR级) | Active FR∪NFR with ≥1 TC / Active FR+NFR | 05 Verify | = 100% |
| C4 | Test 覆盖率(AC级) | Active AC with ≥1 TC / Active AC 总数 | 05 Verify | ≥ 90%(M/L) |
| C5 | 实现覆盖率 | Active FR∪NFR with ≥1 PR / Active FR+NFR | 06 Wrap-up | = 100% |

#### 反向合规率（实现是否有需求依据）

| # | 指标 | 公式 | 校验阶段 | 阈值 |
|---|------|------|---------|------|
| C6 | Task 合规率 | TASK with ≥1 FR/NFR ref / Total TASK | 03 Plan | = 100% |
| C7 | TC 合规率 | TC with ≥1 FR/NFR ref / Total TC | 05 Verify | = 100% |
| C8 | PR 合规率 | PR with ≥1 TASK ref / Total PR | 04 Implement | = 100% |

#### 孤儿项检测

| # | 指标 | 公式 | 校验阶段 | 阈值 |
|---|------|------|---------|------|
| C9 | 孤儿项率 | 未关联 FR/NFR 的(TASK+TC+PR) / 全部(TASK+TC+PR) | 06 Wrap-up | = 0% |

### 5.2 计算算法

```javascript
/**
 * 通用覆盖率计算框架
 * @param matrix - 追踪矩阵数据
 * @param sourceField - 分子来源列（如 "taskRef"）
 * @param filterFn - 分母过滤函数（筛选 Active 的 FR/NFR）
 */
function calcCoverage(matrix, sourceField, filterFn) {
  const activeEntries = matrix.entries.filter(filterFn);
  const coveredEntries = activeEntries.filter(
    entry => entry[sourceField] && entry[sourceField].length > 0
  );
  return {
    covered: coveredEntries.length,
    total: activeEntries.length,
    rate: activeEntries.length === 0
      ? 1.0
      : coveredEntries.length / activeEntries.length,
    uncovered: activeEntries
      .filter(e => !e[sourceField] || e[sourceField].length === 0)
      .map(e => e.id),  // 输出遗漏需求清单
  };
}
```

### 5.3 Known Exception List 豁免机制

部分 FR/NFR 可能因合理原因无法被某类产物覆盖（如纯文档类 NFR 无需 API），通过豁免机制从分母排除：

```yaml
# traceability-matrix.yaml 中的豁免声明
exceptions:
  - id: "NFR-OBS-001"
    exemptFrom: "apiDataRef"
    reason: "可观测性需求不涉及 API 变更，通过基础设施配置实现"
    approvedBy: "Leo"
    approvedAt: "2026-02-08"
```

**豁免约束**：

| 约束 | 说明 |
|------|------|
| 豁免比例上限 | 单个覆盖率指标的豁免条目 ≤ 总条目的 10% |
| 审批要求 | 每条豁免必须有 Tech Lead 审批记录 |
| 理由必填 | `reason` 字段不可为空 |
| 不可滥用 | Gate 校验时检查豁免比例，超限则阻断 |

### 5.4 增量计算模式

```
全量计算：读取完整矩阵，计算所有覆盖率指标
  适用：Gate 校验、CI Pipeline、Wrap-up 终审
  开销：O(N)，N = FR/NFR 总数

增量计算：基于 git diff 确定变更范围，仅重算受影响指标
  适用：commit-msg Hook、pre-push Hook
  开销：O(ΔN)，ΔN = 本次变更涉及的 FR/NFR 数量

增量计算流程：
  1. git diff --name-only 获取变更文件列表
  2. 从变更文件中提取涉及的 ID 集合
  3. 在矩阵中定位这些 ID 对应的行
  4. 仅对这些行重算覆盖率
  5. 若增量结果异常，提示开发者（不阻断，阻断留给 CI 全量校验）
```

### 5.5 专家会审记录

> **[Arch]**: 覆盖率计算的核心是「分母准确」——Active 的 FR/NFR 才计入分母，Cancelled 的排除。Known Exception List 进一步从分母排除合理豁免项。分母不准会导致覆盖率虚高或虚低。
> **[DevEff]**: 增量计算对开发者体验至关重要——commit-msg Hook 必须在 2s 内完成，全量计算在 L 规模（100+ FR）时可能超时。增量模式只在 Hook 中使用，CI 仍做全量校验作为兜底。
> **[QA]**: AC 级覆盖率（C4）是 M/L 规模的重要指标——FR 级覆盖率可能掩盖「一个 FR 有 10 个 AC 但只测了 1 个」的情况。AC 级覆盖率阈值设为 ≥90% 而非 100%，是因为部分 AC 可能通过集成测试隐式覆盖。
> **[决议]**: 增量计算用于 Hook，全量计算用于 CI/Gate。AC 级覆盖率 M/L 规模 ≥90%，S 规模不强制。豁免比例上限 10%。

### 7.5 专家会审记录

> **[Arch]**: 三个接口职责清晰——IdRegistry 管 ID 的「生」与「验」，MatrixManager 管矩阵的「存」与「查」，CoverageCalculator 管覆盖率的「算」与「报」。`getTraceChain` 放在 MatrixManager 而非独立接口，是因为追踪链本质上是矩阵数据的图遍历，数据源在矩阵中。`TraceLink` 记录 `sourceFile` 和 `lineNumber`，支撑 IDE 跳转和 CI 报告中的精确定位。
> **[DevEff]**: `getUncoveredReport` 的 `suggestedActions` 字段是开发者体验的关键——Gate 失败时不仅告诉「哪些没覆盖」，还告诉「怎么修」。建议 CLI 输出时将 `suggestedActions` 格式化为可执行的命令提示（如 `spec-id next TC AUTH`）。增量计算接口 `calcIncrementalCoverage` 接受 `changedFiles` 而非自行调用 `git diff`，这是正确的——保持接口纯函数特性，`git diff` 由调用方（Hook 脚本）负责。
> **[QA]**: `validateExceptions` 独立为接口方法很重要——豁免机制容易被滥用，必须有专门的校验入口。建议在每次 Gate 评估时自动调用此方法，而不是仅在手动触发时校验。`ExceptionViolation` 的 `reason` 字段应包含具体的违规类型（如 "RATE_EXCEEDED"、"REASON_EMPTY"、"APPROVER_MISSING"），便于自动化处理。
> **[决议]**: 三接口划分确认。`getUncoveredReport` 输出可执行修复建议。`validateExceptions` 在 Gate 评估时自动调用。`ExceptionViolation.reason` 采用结构化错误码 + 人类可读描述双字段模式。

## 8. 错误处理与边界场景

### 8.1 错误分类与处理策略

| 错误类别 | 触发场景 | 处理策略 | 用户提示 |
|---------|---------|---------|---------|
| **ID 格式错误** | ID 不匹配正则规则（如 `FR-auth-001` 小写） | Fail-Fast，阻断写入 | 输出期望格式 + 正确示例 |
| **FEAT 未注册** | 使用了未在 `.feat-registry.md` 中注册的缩写 | 阻断，提示注册 | 输出已注册缩写列表 + 注册命令 |
| **ID 重复** | 同一 Feature 内出现重复 ID | 阻断，拒绝写入 | 输出冲突 ID + 所在文件位置 |
| **序号跳号** | 新 ID 序号不是当前最大值 +1（非废弃导致） | 警告，不阻断 | 提示期望序号 + 允许手动确认 |
| **引用悬空** | traces/verifies 引用了不存在的 ID | Gate 阶段阻断 | 输出悬空引用列表 + 可能的正确 ID |
| **矩阵格式错误** | Markdown 表格列数不匹配或 YAML 解析失败 | Fail-Fast，阻断操作 | 输出错误位置 + 期望格式 |
| **矩阵列越权填充** | 在非对应阶段填充了后续列（如 01 阶段填 Task Ref） | 阻断，拒绝更新 | 输出当前阶段 + 允许填充的列 |
| **覆盖率计算异常** | 分母为 0（无 Active FR/NFR）或数据不一致 | 返回默认值 1.0 + 警告 | 提示矩阵可能未初始化 |
| **豁免比例超限** | 单指标豁免条目 > 总条目 10% | Gate 阶段阻断 | 输出超限指标 + 当前比例 + 上限 |
| **注册表损坏** | `.feat-registry.md` 格式解析失败 | 尝试从 Git 历史恢复 | 输出损坏位置 + 恢复命令 |

### 8.2 边界场景处理

#### 场景 1：ID 废弃后的序号处理

```
触发：FR-AUTH-002 被标记为 Cancelled
影响：FR-AUTH-003 已存在，序号出现跳号
处理：
  1. 废弃的 ID 保留在矩阵中，Status 设为 🚫
  2. 序号不回收——下一个可用序号仍为 max(现有序号) + 1
  3. getNextSequence() 扫描所有已分配序号（含 Cancelled），返回 max + 1
  4. 覆盖率计算时，Cancelled 的 FR/NFR 从分母排除
```

#### 场景 2：多人并发编辑追踪矩阵

```
触发：开发者 A 和 B 同时修改 traceability-matrix.yaml
冲突概率：
  - YAML 按 FR-ID 分块 → 编辑不同 FR 时 Git 可自动合并
  - 编辑同一 FR 的不同列 → Git 可能冲突
处理：
  1. YAML 格式按 FR-ID 排序，每个 FR 独立 block
  2. 建议团队约定：同一 FR 的矩阵更新由该 FR 的 Owner 负责
  3. Git 冲突时，以最新 Gate 校验结果为准重建冲突行
  4. CI Pipeline 在 PR 合并前做全量矩阵校验，确保合并后数据一致
```

#### 场景 3：矩阵格式迁移（Markdown → YAML）

```
触发：Feature 初始判定为 Size S（Markdown），后因需求膨胀升级为 Size M
影响：需要将 traceability-matrix.md 迁移为 traceability-matrix.yaml
处理：
  1. 提供 CLI 命令 `spec-matrix migrate <featureId>`
  2. 解析 Markdown 表格，转换为 YAML 结构
  3. 校验转换后数据完整性（行数、列数、ID 一致）
  4. 保留原 .md 文件重命名为 .md.bak，新建 .yaml 文件
  5. 后续操作自动检测到 .yaml 文件，切换为 YAML 模式
```

#### 场景 4：跨 Feature 的 ID 引用

```
触发：Feature B 的 TASK 需要引用 Feature A 的 FR（跨 Feature 依赖）
约束：当前 ID 体系以 Feature 为边界，矩阵按 Feature 独立管理
处理：
  1. 跨 Feature 引用使用完整路径格式：`FR-AUTH-001@001-user-auth`
  2. 校验时需加载被引用 Feature 的矩阵，确认 ID 存在且 Active
  3. 跨 Feature 引用不计入本 Feature 的覆盖率分母
  4. Impact Analysis 时，跨 Feature 引用作为「外部依赖」标记
  5. 被引用 Feature 的 FR 废弃时，通知所有引用方
```

#### 场景 5：ID 提取遇到非标准格式

```
触发：产出物中 ID 格式接近但不完全匹配（如 "FR-Auth-001" 大小写错误）
处理：
  1. 严格模式（Gate 校验）：不匹配即报错，不做模糊匹配
  2. 提示模式（编辑时）：检测疑似 ID，输出修正建议
  3. 疑似 ID 检测规则：
     - 前缀匹配但大小写不对 → 提示正确大小写
     - 序号位数不对（如 FR-AUTH-01）→ 提示补零
     - FEAT 缩写未注册但格式合法 → 提示注册
```

### 8.3 专家会审记录

> **[Arch]**: 错误处理的核心原则是「写入时严格，查询时宽容」——ID 写入产出物时必须格式合法（Fail-Fast），但覆盖率查询时分母为 0 返回 1.0 而非报错（Graceful Degradation）。场景 4 的跨 Feature 引用是架构层面的重要扩展点，`@featureId` 后缀格式保持了 ID 的自描述性，无需额外上下文即可定位。
> **[DevEff]**: 场景 5 的「疑似 ID 检测」对开发者体验至关重要——大小写错误是最常见的 ID 格式问题。建议 Layer A Hook（AI Runtime）在 PostToolUse 时做疑似检测并自动修正，Layer B Hook（Git）在 commit-msg 时做严格校验。两层 Hook 的校验严格度不同，形成「柔性引导 + 刚性兜底」的双层防线。
> **[QA]**: 场景 2 的并发冲突处理需要在集成测试中覆盖——模拟两个开发者同时修改同一 YAML 文件的不同 FR block，验证 Git 三路合并的成功率。场景 3 的格式迁移必须有数据完整性校验——迁移前后的 FR/NFR 数量、ID 列表、Status 分布必须完全一致。
> **[决议]**: 错误处理遵循「写入严格 + 查询宽容」原则。疑似 ID 检测在 Layer A 柔性引导，Layer B 刚性校验。跨 Feature 引用采用 `@featureId` 后缀。格式迁移增加数据完整性校验断言。

## 9. 专家会审总结

### 9.1 关键设计决策汇总

| # | 决策项 | 决策结论 | 决策理由 | 提出方 |
|---|--------|---------|---------|--------|
| D1 | M2 定位 | 数据中枢：ID 校验 + 矩阵 CRUD + 覆盖率计算 | 不做流程编排（M1）、不做 Gate 评估（M3），只负责数据 | Arch |
| D2 | ID 序号规则 | 单调递增，不可回退，允许因废弃跳号 | 原「连续」规则在废弃场景下不可维持 | QA |
| D3 | DS ID 纳入追踪 | DS-FEAT-NNN 正式纳入 8 种 ID 类型 | 替代非结构化章节号引用，防止编辑漂移 | Arch |
| D4 | 注册表格式 | Markdown 表格（非 YAML/JSON） | 人类可读、Git diff 友好、低频写入高频读取 | Arch |
| D5 | 歧义缩写检查 | 前缀包含自动检测 + 人工确认 | 语义重叠无法完全自动化判断 | QA |
| D6 | 矩阵格式自适应 | Size S 用 Markdown，Size M/L 用 YAML | S 规模表格直观，M/L 规模 YAML 按 ID 分块利于 Git 合并 | Arch |
| D7 | YAML 按 FR-ID 分块 | 每个 FR 独立 YAML block | 最大化 Git 三路合并成功率 | QA |
| D8 | 废弃需求处理 | 保留矩阵行标记 🚫，不删除 | 保证审计轨迹完整性 | QA |
| D9 | 覆盖率增量计算 | Hook 用增量，CI/Gate 用全量 | Hook 必须 < 2s，全量计算在 L 规模可能超时 | DevEff |
| D10 | AC 级覆盖率阈值 | M/L ≥ 90%，S 不强制 | 部分 AC 通过集成测试隐式覆盖，100% 过于严苛 | QA |
| D11 | 豁免比例上限 | 单指标 ≤ 10% | 防止豁免机制被滥用 | Arch |
| D12 | 跨 Feature 引用 | `ID@featureId` 后缀格式 | 保持 ID 自描述性，无需额外上下文 | Arch |

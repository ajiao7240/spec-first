## 2. ID 体系

### 2.1 ID 格式定义（6 种）

| 类型 | 格式 | 示例 | 说明 |
|------|------|------|------|
| Feature | `FSREQ-YYYYMMDD-<FEAT>-NNN` | FSREQ-20260210-AUTH-001 | 特性唯一标识 |
| FR | `FR-<FEAT>-NNN` | FR-AUTH-001 | 功能需求（含 NFR，加 `[NFR:<DIM>]` 标签区分维度） |
| DS | `DS-<FEAT>-NNN` | DS-AUTH-001 | 设计方案（含 API 契约引用） |
| TASK | `TASK-<FEAT>-NNN` | TASK-AUTH-001 | 开发任务 |
| TC | `TC-<LVL>-<FEAT>-NNN` | TC-UT-AUTH-001 | 测试用例 |
| RFC | `RFC-NNN` | RFC-001 | 变更请求（含 ADR 决策记录） |

**精简决策**：
- **NFR → FR**：NFR 本质是需求，用 `[NFR:SEC]`、`[NFR:PERF]` 等标签区分维度，不单独编号
- **API → DS 属性**：API 契约是设计的一部分，在 DS 中通过 `api_ref` 字段引用，不独立编号
- **ADR → RFC**：架构决策本质是变更请求的子类型，用 RFC 的 `级别: ADR` 标记
- **DEF/EXC → 无独立 ID**：缺陷用 Feature 内自增序号（`#1, #2...`）；Exception 条目登记在 `known-exceptions.md`，并强制关联 RFC

### 2.2 NFR 标签维度

FR 条目通过 `[NFR:<DIM>]` 标签标记非功能需求维度：

`SEC`（安全）、`PERF`（性能）、`AVAIL`（可用性）、`MAINT`（可维护性）、`COMPAT`（兼容性）、`A11Y`（无障碍）。

示例：`FR-AUTH-005 [NFR:SEC]` — 表示该需求属于安全类非功能需求。

### 2.3 注册规则

1. **唯一来源**：仅 `spec-first id next` 可分配新 ID
2. **不可复用**：已分配 ID 不可回收重用
3. **缩写唯一**：FEAT 缩写全局唯一，注册在 `specs/.feat-registry.md`
4. **序号递增**：同类型同缩写下序号单调递增（001 → 002 → ...）

### 2.4 ID 生成算法

```text
nextId(type, abbr, featureId)
  │
  ├── 1. 读取 traceability-matrix.md
  ├── 2. 扫描所有已存在的同类型同缩写 ID
  ├── 3. 取最大序号 + 1（无已有则从 001 开始）
  ├── 4. 组装 ID 字符串
  ├── 5. 校验格式合法性（正则匹配）
  └── 6. 写入矩阵新行（Status = Planned）
```

### 2.5 模糊搜索

`spec-first id search <query>` 支持：

1. **前缀匹配**：`FR-AUTH` → 匹配所有 FR-AUTH-* ID
2. **缩写匹配**：`AUTH` → 匹配所有含 AUTH 的 ID
3. **类型过滤**：`--type FR` → 仅搜索 FR 类型

## 3. 追踪矩阵模型

### 3.1 文件格式

文件路径：`specs/<featureId>/traceability-matrix.md`

```markdown
| 需求 ID | Design Ref | Task Ref | TC Ref | PR Ref | Status |
|---------|-----------|---------|--------|--------|--------|
| FR-AUTH-001 | DS-AUTH-001 | TASK-AUTH-001 | TC-UT-AUTH-001 | #42 | Implemented |
| FR-AUTH-005 [NFR:SEC] | DS-AUTH-002 | TASK-AUTH-003 | TC-IT-AUTH-001 | #43 | Verified |
```

### 3.2 Status 枚举

| 状态 | 含义 |
|------|------|
| `Planned` | 已规划，尚未实现 |
| `Implemented` | 已实现，待验证 |
| `Verified` | 已通过测试验证 |
| `Accepted` | 已通过 UAT 验收 |
| `Deferred` | 延期到后续版本 |
| `Cancelled` | 已取消 |
| `Exception` | 已知例外（关联 RFC） |

### 3.3 追踪链方向

```text
FR → DS → TASK → TC → PR
│                       │
└────── 正向覆盖 ───────┘
│                       │
└────── 反向合规 ───────┘
```

## 4. 覆盖率算法

### 4.1 C1-C9 指标（公式权威定义见 v2-11 §2.1）

M2 TraceEngine 负责计算以下 9 项指标，完整公式与权重见 v2-11-度量与健康分。此处仅列出 M2 实现视角的语义说明：

| ID | 指标 | 类型 | M2 实现语义 |
|----|------|------|------------|
| C1 | Design Coverage | 正向覆盖 | 需求是否都有设计 |
| C2 | API Coverage | 正向覆盖 | 设计是否都有契约 |
| C3 | Task Coverage | 正向覆盖 | 需求是否都拆解了任务 |
| C4 | Test Coverage (FR) | 正向覆盖 | 需求是否都有测试 |
| C5 | Test Coverage (AC) | 正向覆盖 | 验收条件是否都有测试 |
| C6 | Impl Coverage | 正向覆盖 | 需求是否都实现了 |
| C7 | PR Compliance | 反向合规 | 是否有无任务依据的提交 |
| C8 | Task Compliance | 反向合规 | 是否有无需求依据的任务 |
| C9 | TC Compliance | 反向合规 | 是否有无需求依据的测试 |

### 4.3 辅助指标（信息性，不纳入健康分）

| 指标 | 公式 | 说明 |
|------|------|------|
| 孤儿项率 | `无任何上游关联的产物数 / 产物总数` | `matrix check` 时输出警告，不参与 C1-C9 计算 |

### 4.4 有效分母计算

> **与 v2-11 §3.1 完全对齐。**

"Active" 指排除以下状态的条目：`Deferred`、`Cancelled`、`Exception`。

- Exception 条目从分母排除，但必须关联有效 RFC
- Deferred/Cancelled 条目从分母排除，无额外约束
- **分母为 0 时，该指标值定义为 100%**（无需覆盖）

## 5. Known Exception 处理

文件路径：`specs/<featureId>/known-exceptions.md`

> **数据源定位**：`known-exceptions.md` 是**豁免登记主文件**，条目必须关联已审批的 RFC（`rfc_id` 必填）。CLI 命令 `spec-first rfc create` 审批通过后可辅助写入条目，也允许手动编辑（需通过 commit-msg Hook 校验）。Gate 评估时直接读取此文件中的有效条目。

### 5.1 Exception 记录格式

> Exception 不设独立 ID，通过关联的 RFC ID 追踪。
> **权威 schema** 以 `v2-06-质量门禁与豁免机制.md` §6.2 为准；下表仅用于矩阵视图展示。

```markdown
| 关联 RFC | 影响范围 IDs | 原因 | 过期时间 | 回滚点 |
|---------|--------------|------|---------|--------|
| RFC-002 | FR-AUTH-003, TC-IT-AUTH-001 | 第三方 SDK 未就绪 | 2026-03-15 | commit abc123 |
```

### 5.2 规则

1. Exception 从覆盖率分母排除
2. 必须关联有效 RFC（RFC 状态为 approved，且豁免未过期）
3. 必须有 `expires_at`（过期时间）和 `rollback_point`
4. 过期后自动升级为 P0 风险，Gate check 时报 FAIL

## 6. 最小实现清单

1. `nextId(type, abbr, featureId)` — ID 生成与注册
2. `validateId(id)` — 格式校验（正则匹配）
3. `searchId(query, featureId)` — 模糊搜索
4. `checkMatrix(featureId)` — 矩阵完整性校验（孤儿项、断链）
5. `getCoverage(featureId)` — C1-C9 九项覆盖率计算（与 v2-11 口径一致）
6. `validateExceptions(featureId)` — Exception 有效性校验

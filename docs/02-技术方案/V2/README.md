---

## 1. 方案范围

本目录提供 V2 需求文档的可实施技术方案，覆盖以下能力：

1. 双层架构（Skill 驱动 + CLI 原子执行）
2. 流程状态机与阶段推进
3. ID/追踪矩阵/覆盖率计算
4. Gate/SCA/变更管理
5. Context Pack 与 Catchup
6. 多端规范合并（仅技术端）
7. 产出物与模板
8. 度量体系与上线门槛

## 3. 实施顺序（推荐）

1. 先实现基础能力：`v2-03` + `v2-04` + `v2-05`
2. 再实现质量闭环：`v2-06` + `code-review-integration.md` + `v2-07`
3. 再实现 AI 协作：`v2-02` + `v2-08`
4. 再实现扩展与运营：`v2-09` + `v2-10` + `v2-11`
5. 最后执行上线门槛：`v2-12`

---

## 4. 设计约束

1. 用户统一入口为 `/spec-first:*`，禁止用户裸调 CLI。
2. CLI 保持确定性与幂等，Skill 不承担状态持久化职责。
3. 所有关键状态变更必须可审计（JSONL 留痕）。
4. Mode I 的回归验证与影响分析必须可追踪。
5. 不引入超出需求文档范围的复杂框架。

---

## 5. 验收标准

1. 文档与 `docs/01需求文档/v2` 术语、命令、状态一致。
2. 结构要求按文档类型执行：
   - 核心实施文档（`v2-03` ~ `v2-11`）至少包含：目标、关键数据结构/接口、流程、异常或降级策略、最小实现清单。
   - 总览/路线图文档（`v2-01`、`v2-12`）至少包含：目标、边界、关键对象、实施路径、风险与回退、最小实现清单。
3. 关键决策可落地为可执行任务，不依赖模糊描述。

## 1. 目标

用统一口径计算覆盖率、合规率、缺陷逃逸率与健康分，支撑治理决策，识别瓶颈阶段。

## 2. 12 项核心指标

### 2.1 A 类：覆盖率与合规率（9 项）

| ID | 指标名 | 公式 | 目标 |
|----|--------|------|------|
| C1 | Design Coverage | 被 DS 覆盖的 Active FR∪NFR / Active FR+NFR | 100% |
| C2 | API Coverage | 需接口的 FR 被 API 覆盖 / 需接口的 FR | 100% |
| C3 | Task Coverage | 被 TASK 覆盖的 Active FR∪NFR / Active FR+NFR | 100% |
| C4 | Test Coverage (FR) | 被 TC 覆盖的 Active FR∪NFR / Active FR+NFR | 100% |
| C5 | Test Coverage (AC) | 被 TC 覆盖的 AC / 总 AC | ≥90% (M/L) |
| C6 | Impl Coverage | Active FR∪NFR with ≥1 PR / Active FR+NFR | 100% |
| C7 | PR Compliance | 关联 TASK ID 的 PR / 总 PR | 100% |
| C8 | Task Compliance | 关联 FR/NFR/DS 的 TASK / 总 TASK | 100% |
| C9 | TC Compliance | 关联 AC/FR 的 TC / 总 TC | 100% |

### 2.2 B 类：效率指标（1 项）

| ID | 指标名 | 公式 | 用途 |
|----|--------|------|------|
| E1 | Stage Cycle Time | 各阶段 `entered_at` 到 `exited_at` 的时间差 | 识别瓶颈阶段 |

### 2.3 C 类：质量指标（1 项）

| ID | 指标名 | 公式 | 目标 |
|----|--------|------|------|
| Q1 | Defect Escape Rate | 上线后缺陷 / 总缺陷 | < 2%（S1/S2 = 0%） |

### 2.4 D 类：综合指标（1 项）

| ID | 指标名 | 公式 | 用途 |
|----|--------|------|------|
| H1 | Health Score | 加权综合分（见第 3 节） | 项目健康度一览 |

## 3. 计算规则

### 3.1 有效分母定义

"Active" 指排除以下状态的条目：`Deferred`、`Cancelled`、`Exception`。

- Exception 条目从分母排除，但必须关联有效 RFC
- Deferred/Cancelled 条目从分母排除，无额外约束
- 分母为 0 时，该指标值定义为 100%（无需覆盖）

### 3.2 健康分公式

> **量纲说明**：C1-C9 取值范围 0.0-1.0，公式结果 ×100 输出为 0-100 整数刻度。

```text
H1 = (w1×C1 + w2×C2 + w3×C3 + w4×C4 + w5×C5
    + w6×C6 + w7×C7 + w8×C8 + w9×C9) × 100 - penalty(Q1)
```

**默认权重**（可通过 config.yaml 覆盖）：

| 权重 | 值 | 对应指标 | 权重理由 |
|------|-----|---------|---------|
| w1 | 0.10 | C1 Design Coverage | 设计是下游实现的前提，但缺失可通过后续阶段补救，取基础权重 |
| w2 | 0.10 | C2 API Coverage | 接口契约是跨端协作基础，但仅影响有接口需求的 FR 子集 |
| w3 | 0.10 | C3 Task Coverage | 任务拆解影响实现可追踪性，但不直接影响交付质量 |
| w4 | 0.15 | C4 Test Coverage (FR) | 测试覆盖直接关联缺陷逃逸率，是质量保障核心环节，赋予最高档权重 |
| w5 | 0.10 | C5 Test Coverage (AC) | AC 级覆盖细化验收粒度，但 C4 已覆盖 FR 级，此处为补充 |
| w6 | 0.15 | C6 Impl Coverage | 实现覆盖率直接反映交付完成度，与 C4 并列为最高档权重 |
| w7 | 0.10 | C7 PR Compliance | PR 关联 TASK 是追踪链闭合的最后一环，取合规基线权重 |
| w8 | 0.10 | C8 Task Compliance | TASK 关联 FR/DS 保证任务有据可依，取合规基线权重 |
| w9 | 0.10 | C9 TC Compliance | TC 关联 AC/FR 保证测试有据可依，取合规基线权重 |

> **权重分档逻辑**：三档权重（0.15 / 0.10 / —），总和 = 1.0。C4+C6 为交付质量双核心取 0.15；其余 7 项各 0.10。未采用更细粒度分档以降低调参复杂度。

**惩罚项**：`penalty(Q1) = max(0, (Q1 - 0.02)) × 100`

- Q1 ≤ 2% 时无惩罚
- Q1 = 5% 时惩罚 3 分
- Q1 = 10% 时惩罚 8 分

### 3.3 健康等级

| 分数区间 | 等级 | 含义 |
|----------|------|------|
| ≥ 90 | 🟢 Healthy | 项目健康，可正常推进 |
| ≥ 70 | 🟡 Warning | 存在风险，需关注瓶颈 |
| < 70 | 🔴 Critical | 严重问题，需立即干预 |

## 4. 存储与导出

### 4.1 存储格式

文件路径：`specs/<featureId>/metrics.jsonl`

```json
{
  "timestamp": "2026-02-10T10:30:00Z",
  "featureId": "FSREQ-20260210-AUTH-001",
  "stage": "03_plan",
  "metrics": {
    "C1": 1.0, "C2": 0.85, "C3": 1.0, "C4": 0.0,
    "C5": 0.0, "C6": 0.0, "C7": 0.0, "C8": 1.0, "C9": 0.0
  },
  "E1": { "00_init": 120, "01_specify": 3600, "02_design": 7200 },
  "Q1": 0.0,
  "healthScore": 87.5,
  "healthLevel": "warning"
}
```

### 4.2 写入时机

| 触发事件 | 写入内容 |
|----------|---------|
| `gate check` 完成 | 全量 12 项指标快照 |
| `metrics coverage` 手动调用 | 全量 12 项指标快照 |
| `stage advance` 成功 | 全量 12 项指标快照 |

### 4.3 轮转策略

- 月度轮转：当月文件超过 1000 行时归档为 `metrics-YYYY-MM.jsonl`
- 归档文件只读，不再追加
- `metrics report` 默认读取当前文件，`--history` 参数可读取归档文件

## 5. 瓶颈分析规则

M6 MetricsEngine 内置 5 条瓶颈检测规则，`metrics report` 时自动执行：

| 规则 | 触发条件 | 建议动作 |
|------|----------|---------|
| R1 设计瓶颈 | C1 < 80% 且阶段 ≥ 03_plan | 补充 DS 覆盖，检查是否有 FR 遗漏设计 |
| R2 测试瓶颈 | C4 < 100% 且阶段 ≥ 05_verify | 补充 TC，优先覆盖高风险 FR |
| R3 实现滞后 | C6 < 50% 且阶段停留超过 E1 中位数 ×2 | 检查任务拆解粒度，考虑拆分大任务 |
| R4 合规缺口 | C7 或 C8 < 90% | 检查 PR/TASK 关联，补充 traces 注释 |
| R5 缺陷逃逸 | Q1 > 10% | 加强测试设计，增加边界用例覆盖 |

**输出格式**：

```text
⚠️ 瓶颈检测结果:
  R2 测试瓶颈: C4=75% (目标 100%), 当前阶段 05_verify
  → 建议: 补充 TC，优先覆盖 FR-AUTH-003, FR-AUTH-005
```

触发的规则写入 `findings.md`，同时在终端输出。

## 6. 最小实现清单

1. `computeCoverage(featureId)` — 计算 C1-C9 九项覆盖率（基于追踪矩阵）
2. `computeCycleTime(featureId)` — 计算 E1 各阶段耗时（基于 stage-state.json history）
3. `computeDefectEscapeRate(featureId)` — 计算 Q1 缺陷逃逸率（基于缺陷记录）
4. `computeHealthScore(featureId)` — 加权计算 H1 + 健康等级判定
5. `detectBottlenecks(featureId)` — 执行 R1-R5 瓶颈规则 + 输出建议
6. `appendMetrics(featureId, snapshot)` — JSONL 追加写入
7. `generateReport(featureId)` — 渲染 health-report.md（基于 Handlebars 模板）

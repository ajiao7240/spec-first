# V2-06 质量门禁与豁免机制

> **对齐需求**: core-05-cross-cutting · core-03-traceability · aux-02-cli-system
> **版本**: v2.1 | **日期**: 2026-02-10 | **原则**: KISS

---

## 1. 目标

在每阶段设置最小、明确、可执行的 Gate，避免"口头通过"。Gate 结果可审计、可追溯。

---

## 2. Gate 模型

### 2.1 结果语义

| 结果 | 含义 | 允许推进 |
|------|------|---------|
| `PASS` | 所有条件满足 | ✅ |
| `PASS_WITH_WAIVER` | 部分条件未满足，但有有效豁免 | ✅ |
| `FAIL` | 条件未满足，无有效豁免 | ❌ |

> **异常状态 `GATE_UNAVAILABLE`**：当 GateEngine 自身不可用时（如阶段 A 尚未交付 M3），`gate check` 抛出 `GateUnavailableError` 异常而非返回 GateResult。该异常不属于 Gate 三态语义，由调用方（`stage advance`）根据 `config.yaml` 中 `gate.pilot_mode` 决定是否降级放行。详见 v2-12 §阶段 A Gate 降级策略。

```text
gate check <featureId> [--stage <stageId>]
  │
  ├── 1. 读取 stage-state.json（确定当前阶段 + 合并后的 Gate 条件）
  ├── 2. 逐项评估每个 Gate 条件
  │     ├── 条件通过 → mark PASS
  │     ├── 条件未通过 → 检查 known-exceptions.md
  │     │     ├── 有有效豁免 → mark WAIVER
  │     │     └── 无豁免 → mark FAIL
  ├── 3. 汇总结果
  │     ├── 全部 PASS → 返回 PASS
  │     ├── 有 WAIVER 无 FAIL → 返回 PASS_WITH_WAIVER
  │     └── 有 FAIL → 返回 FAIL + 修复建议
  └── 4. 写入 gate-history.jsonl
```

## 3. 阶段 Gate 条件

| 阶段 | Gate 条件 |
|------|----------|
| 00_init | 目录就绪；Mode/Size/Platforms 已确认；Mode I 完成历史产物定位 |
| 01_specify | DoR 签核；无歧义标记；FR/NFR 完整分配 ID；SCA 通过 |
| 02_design | 设计评审通过；API 覆盖率 = 100%；SCA 通过 |
| 03_plan | Task 覆盖率 = 100%；Task 合规率 = 100%；SCA 通过 |
| 04_implement | Code Review 通过；单元测试代码覆盖率 ≥ 80%；PR 合规率 = 100%；SCA 通过 |
| 05_verify | UAT Sign-off（Gate Owner: QA Lead + PM）；Test 覆盖率(FR) = 100%；Test 覆盖率(AC) ≥ 90%（M/L）；TC 合规率 = 100%；安全无高危；（仅 Mode I）回归通过 |
| 06_wrap_up | 文档完整性；实现覆盖率 = 100%；矩阵所有条目 Status ∈ {Accepted, Cancelled, Exception}（Exception 须有有效豁免，对齐 core-04-process §06 Exit Gate） |
| 07_release | Smoke Test 通过率 100%；错误率 ≤ 基线 110%；P95 延迟 ≤ 基线 120%；回滚率 = 0（发布后 1 小时观测窗口） |

### 3.1 07_release 外部指标源 fallback

07_release Gate 依赖外部监控系统（APM/日志平台）提供错误率、P95 延迟、回滚率数据。当外部指标源不可用时：

| 场景 | fallback 策略 |
|------|--------------|
| APM 不可用（超时/连接失败） | 降级为手动填报：Gate Owner 提供截图证据 + 人工签核，写入 findings.md 审计记录 |
| 基线数据缺失（首次发布无历史基线） | 跳过基线对比条件，仅校验 Smoke Test 通过率 + 回滚率 = 0 |
| 部分指标缺失 | 可用指标正常评估，缺失指标标记为 `WAIVER`（需 RFC 审批） |

> **对齐 v2-12 全局降级策略**：外部指标源 fallback 是 v2-12 §降级与回退策略的专项细化，不替代全局降级机制。

## 4. SCA（规范一致性校验）

### 4.1 触发时机

| 时机 | 校验范围 |
|------|---------|
| Specify 完成后 | spec 内部一致性（FR/NFR ID 唯一、AC 完整） |
| Design 完成后 | spec ↔ design（每个 FR 有 DS 映射） |
| Plan 完成后 | spec ↔ tasks（每个 FR 有 TASK 映射） |
| Implement 完成后 | spec ↔ code（TASK traces 完整） |
| Verify 完成后 | spec ↔ test results（TC verifies FR/AC） |

### 4.2 SCA 与 Gate 的关系

SCA FAIL 等同 Gate FAIL — 阻断阶段推进。SCA 结果写入 Gate 条件评估结果中。

---

## 5. 安全严重级别

| 级别 | 含义 | Gate 行为 | 可豁免 |
|------|------|----------|--------|
| S1 | 数据泄露/RCE/权限绕过 | 强制 FAIL | ❌ 禁止 |
| S2 | XSS/CSRF/注入 | 默认 FAIL | 仅紧急场景 |
| S3 | 信息泄露/配置不当 | 可豁免放行 | ✅ |
| S4 | 低风险/建议项 | 不阻断 | 不需要 |

**"安全无高危"定义**：无 S1，且无未获批豁免的 S2。

## 6. 豁免快通道

### 6.1 流程

> **数据源定位**：`known-exceptions.md` 是**豁免登记主文件**（对齐 core-03-traceability §Known Exception List），条目必须关联已审批的 RFC（`rfc_id` 必填）。CLI 命令 `spec-first rfc create` 审批通过后可辅助写入条目，也允许手动编辑。Gate 评估时直接读取此文件中的有效条目。

```text
1. 创建 RFC（spec-first rfc create）
2. RFC 审批通过后，在 known-exceptions.md 登记/更新豁免条目（CLI 可辅助写入，也允许人工维护）
3. Gate 评估时检测到豁免 → 返回 PASS_WITH_WAIVER
4. 到期前关闭 RFC 并补充闭环证据
```

### 6.2 豁免记录必须字段

| 字段 | 说明 |
|------|------|
| waiver_id | 豁免唯一标识（Feature 内唯一，格式 `WVR-NNN`） |
| rfc_id | 关联的 RFC ID |
| scope_ids | 豁免覆盖的受影响 ID 列表（严格限定 6-ID 模型合法类型：`FR-*`、`DS-*`、`TASK-*`、`TC-*`）。NFR 引用格式：`FR-AUTH-003[NFR:PERF]`；API 引用格式：`DS-AUTH-001.api_ref`。校验正则：`^(FR\|DS\|TASK)-[A-Z][A-Z0-9]{1,15}-\d{3}(\[NFR:[A-Z]+\])?(\.api_ref)?$\|^TC-(UT\|IT\|E2E\|ST)-[A-Z][A-Z0-9]{1,15}-\d{3}$` |
| reason | 豁免原因（必填） |
| risk_level | 风险等级（Low/Medium/High） |
| mitigation | 缓解措施（必填） |
| rollback_point | 回滚点（commit SHA 或 PR 编号） |
| owner | 豁免负责人（必填） |
| approved_by | 审批人（角色+姓名，必填） |
| created_at | 创建时间（ISO8601，必填） |
| expires_at | 过期时间（ISO8601） |
| closure_evidence | 闭环证据（关闭时填写） |

### 6.3 过期处理

豁免过期后，Gate check 自动将对应条件标记为 FAIL，升级为 P0 风险。

### 6.4 豁免约束

1. 单次 Gate 的豁免条目数量不得超过 Active FR+NFR 总数的 10%
2. 禁止无到期时间的豁免（`expires_at` 必填）
3. 超限或字段不完整时，`gate check` 返回 FAIL

### 6.5 waiver_id 生成规则

- 正则：`^WVR-\d{3}$`
- 作用域：Feature 内唯一，不跨 Feature 共享序号
- 分配方式：按 `known-exceptions.md` 已有最大序号 + 1（从 `WVR-001` 开始）

## 7. Hook 双层执行

| 层 | 名称 | 触发时机 | 行为 |
|----|------|---------|------|
| Layer A | AI Runtime Hook | Skill Phase 5 | 实时提示（软阻断） |
| Layer B | Git/CI Hook | prepare-commit-msg / commit-msg / pre-push / CI | 硬阻断（ExitCode ≠ 0 则拒绝） |

**KISS 实现顺序**：先实现 Layer B（Git Hook + CI Pipeline），再补 Layer A。

### 7.1 prepare-commit-msg Hook

从当前分支名 / `.spec-first/current` + `task_plan.md` 提取 TASK ID，自动预填：

- **Subject line**：`[TASK-<FEAT>-NNN] ` 前缀（人类可读，`git log --oneline` 友好）
- **Git trailer**：`traces: TASK-<FEAT>-NNN`（机器解析，工具链消费）

> 与需求文档 core-05-cross-cutting §Layer B 对齐，降低手动输入 ID 的心流中断。

### 7.2 commit-msg Hook

校验 commit message 格式，需满足**至少一条合法 ID 标识规则**（推荐同时满足）：

1. Subject line 以 `[TASK-<FEAT>-NNN]` 或 `[RFC-NNN]` 合法前缀开头
2. Message body 包含 `traces: TASK-<FEAT>-NNN` 或 `refs: RFC-NNN` trailer

两条都不满足时拒绝提交，并输出修复提示（期望格式 + 当前可用 TASK ID 列表）。prepare-commit-msg 默认预填前缀和 trailer，以尽量让大多数提交一次通过。

### 7.3 pre-push Hook

执行增量 SCA（调用 `spec-first matrix check <featureId>` 的增量评估路径，由引擎按本次变更范围计算）；不一致则阻断 push。

### 7.4 CI Pipeline Hook

PR 创建/更新时执行全量一致性校验 + 追踪覆盖率校验（全量 SCA + 关键覆盖率阈值）；不通过时阻止合并。

### 7.5 Layer A — AI Runtime Hook 接口定义

> **交付阶段**：阶段 C（P2），对齐 core-05-cross-cutting §Layer A。Layer A 不存在时，所有 Gate 校验由 Layer B 承载。

```typescript
// src/core/tool-integration/runtime-hooks.ts

type HookType = 'PreToolUse' | 'PostToolUse' | 'Stop';

interface RuntimeHookResult {
  allowed: boolean;          // true=放行, false=软阻断
  message?: string;          // 阻断时的提示信息
  suggestions?: string[];    // 修复建议
}

interface RuntimeHookContext {
  featureId: string;
  stage: string;
  skillName?: string;        // 当前执行的 Skill
  phase?: string;            // 当前 Phase（如 'P4_WRITE'）
  targetFile?: string;       // 写操作目标文件路径
}

interface RuntimeHookEngine {
  /** PreToolUse: AI 执行写操作前，校验当前阶段 Gate 前置条件 */
  preToolUse(ctx: RuntimeHookContext): Promise<RuntimeHookResult>;
  /** PostToolUse: AI 执行写操作后，校验产出物是否符合当前阶段规范 */
  postToolUse(ctx: RuntimeHookContext): Promise<RuntimeHookResult>;
  /** Stop: AI 会话结束时，输出完成度报告 */
  stop(ctx: RuntimeHookContext): Promise<RuntimeHookResult>;
}
```

**注入机制**：Skill Phase 5（副作用执行）中调用 `RuntimeHookEngine`，由 Skill Dispatcher 在进入 Phase 4/5 前后自动触发对应 Hook。

**软阻断语义**：`allowed=false` 时输出 warning 并提示修正，不强制终止 Skill 执行（区别于 Layer B 的硬阻断）。

## 8. 审计日志

文件路径：`specs/<featureId>/gate-history.jsonl`

```json
{"timestamp":"2026-02-10T10:30:00Z","stage":"01_specify","result":"PASS","conditions":[{"name":"DoR签核","result":"PASS"},{"name":"FR/NFR ID完整","result":"PASS"}]}
{"timestamp":"2026-02-10T11:00:00Z","stage":"02_design","result":"PASS_WITH_WAIVER","conditions":[{"name":"API覆盖率","result":"WAIVER","waiver":"RFC-002"}]}
```

每次 `gate check` 追加一行，不修改历史记录。

## 9. 最小实现清单

1. `evaluateGate(featureId, stage)` — 逐项评估 Gate 条件
2. `evaluateSCA(featureId, stage)` — SCA 校验（基于矩阵追踪链）
3. `evaluateSecuritySeverity(featureId)` — 安全级别汇总
4. `validateWaiver(exceptionId)` — 豁免有效性校验（未过期 + RFC=approved）
5. `appendGateHistory(featureId, result)` — JSONL 审计写入
6. `installHooks()` — Git/CI Hook 安装（prepare-commit-msg + commit-msg + pre-push + CI Pipeline）

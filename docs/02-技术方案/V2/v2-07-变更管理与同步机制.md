> **对齐需求**: core-05-cross-cutting · core-06-case · aux-02-cli-system

## 1. 目标

把需求/设计变更纳入可审计状态机，防止"先改后补"。

---

## 2. RFC 状态机

### 2.1 状态定义（4 态）

```text
draft → approved → closed
          ↘ rejected
```

| 状态 | 含义 |
|------|------|
| `draft` | 草案，可编辑 |
| `approved` | 已批准，执行中（执行过程由 Skill/任务驱动，不在状态机内建模） |
| `closed` | 已完成并关闭（终态） |
| `rejected` | 评审不通过（终态） |

### 2.2 合法转换表

| 当前状态 | 允许目标 |
|---------|---------|
| draft | approved, rejected |
| approved | closed, rejected |
| rejected | — (终态) |
| closed | — (终态) |

**设计决策**：执行过程（planning → executing → verifying）由 Skill 和任务计划驱动，不在 RFC 状态机内建模。状态机仅管理"决策节点"（是否批准、是否完成），避免状态爆炸。

### 2.3 RFC 记录格式

文件路径：`specs/<featureId>/rfc/<RFC-NNN>.rfc.md`

```markdown
# RFC-001: 新增邮箱验证码登录

- **级别**: Major
- **状态**: approved
- **发起人**: Leo
- **创建时间**: 2026-02-10T14:00:00Z
- **影响产物**: spec.md, design.md, contracts/, task_plan.md, tests/
- **影响 ID**: FR-AUTH-001, DS-AUTH-001, TASK-AUTH-001

## 变更描述
...

## 影响分析
...

## 审批记录
| 时间 | 操作 | 操作人 |
|------|------|--------|
| 2026-02-10T14:00:00Z | draft → approved | Leo |
```

## 3. 变更分级

| 级别 | 判定条件 | 处理方式 |
|------|---------|---------|
| Minor | 影响 ≤2 类产物，不需重新触发 Gate | `/spec-first:sync` 自动回填 |
| Major | 影响 3-5 类产物，或需重新触发 Gate | 标准 RFC 流程 |
| Critical | 涉及架构变更 / Constitution 修改 / 跨团队影响 | 完整评审 + Tech Lead 签核 |

**判定规则**：两个维度（影响产物数、是否需重新触发 Gate）取较高级别。

## 4. 影响分析

### 4.1 自动影响分析

基于追踪矩阵的 ID 链，自动计算变更影响范围：

```text
变更 FR-AUTH-001
  │
  ├── 正向追踪：DS-AUTH-001 → TASK-AUTH-001 → TC-UT-AUTH-001
  └── 影响产物：spec.md, design.md, contracts/, task_plan.md, tests/
```

### 4.2 影响分析产出物

文件路径：`specs/<featureId>/impact-analysis.md`（Mode I 必须产出）

## 5. Sync 机制

入口：`/spec-first:sync <file_path>`

### 5.1 执行流程

```text
sync <file_path>
  │
  ├── 1. 扫描改动文件，识别变更类型
  ├── 2. 反查受影响 ID（基于追踪矩阵）
  ├── 3. 判定变更级别（Minor/Major/Critical）
  │     ├── Minor → 自动回填文档与矩阵
  │     └── Major/Critical → 生成 RFC 草案，提示用户审批
  ├── 4. 触发增量 matrix check + gate check
  └── 5. 更新 findings.md（记录同步操作）
```

### 5.2 适用场景

- Hotfix 后需回填规范
- 手工改动代码后需同步文档
- 外部依赖变更后需更新契约

## 6. 缺陷管理

### 6.1 严重级别

与安全级别统一口径：S1（致命）/ S2（严重）/ S3（一般）/ S4（建议）。

### 6.2 缺陷记录

> 缺陷不设独立 ID 体系，使用 Feature 内自增序号（`#1, #2...`）。

```typescript
interface DefectRecord {
  seq: number;            // Feature 内自增序号
  featureId: string;
  severity: 'S1' | 'S2' | 'S3' | 'S4';
  title: string;
  reporter: string;       // 报告人（必填）
  discoveredIn?: string;  // 发现阶段（如 '04_implement'、'05_verify'），用于缺陷逃逸率计算
  linkedFr?: string;      // 关联 FR ID
  linkedTc?: string;      // 发现该缺陷的 TC ID
  status: 'open' | 'fixing' | 'fixed' | 'verified' | 'wontfix';
  createdAt: string;
}
```

### 6.3 缺陷状态转换表

| 当前状态 | 允许目标 |
|---------|---------|
| open | fixing, wontfix |
| fixing | fixed, open（放弃修复时回退） |
| fixed | verified, open（验证不通过时回退） |
| verified | — (终态) |
| wontfix | — (终态) |

## 7. 最小实现清单

1. `createRfc(featureId, opts)` — 创建 RFC + 自动分配 ID
2. `submitRfc(rfcId, featureId)` — 提交 RFC 审批（便捷入口，等价于 `transitionRfc(rfcId, 'approved', featureId)` + 豁免条目写入）
3. `transitionRfc(rfcId, status, featureId)` — RFC 状态流转 + 合法性校验
4. `listRfc(featureId)` — 列出 Feature 下所有 RFC
5. `registerDefect(featureId, opts)` — 注册缺陷
6. `analyzeImpact(featureId, changedIds)` — 基于追踪链的影响分析
7. `syncBackfill(featureId, filePath)` — 反向同步回填

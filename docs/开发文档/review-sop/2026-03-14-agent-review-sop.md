# Agent 审查 SOP

> 版本：v1
> 日期：2026-03-14
> 适用对象：Codex / Claude Code / 多 Agent 协同审查
> 目标：产出可复核、可追责、可回写的代码审查结论，避免猜测、错定位、旧统计和过度定级

---

## 1. 适用场景

在以下任一场景必须使用本 SOP：

- 用户要求核对已有审查报告或 review bundle 是否准确
- 用户要求生成新的代码审查报告
- 多个 Agent 分头审查后，需要统一收敛结论
- 需要核对“代码实现 vs 文档/Skill/Gate/规范”是否一致
- 需要输出可直接用于修正文档或追踪整改的问题清单

---

## 2. 审查目标

本 SOP 解决以下问题：

- 审查报告混入推测而非代码事实
- 文件数、测试数、Skill 数等统计失真
- 问题定位到错误文件或不存在的代码
- 将“证据不足的问题”写成“已经证实的问题”
- 多 Agent 输出口径不一致，无法收敛
- 严重等级随意升级，影响团队判断

---

## 3. 证据源优先级

审查必须按以下优先级取证：

1. 本地源码
2. 本地测试、类型检查、覆盖率配置、构建结果
3. 本地文档、`SKILL.md`、配置文件、模板
4. `context7`
   - 仅用于核对官方库、框架、SDK、语言特性最佳实践
5. `fetch`
   - 仅用于核对外部网页、远程规范、在线文档

硬规则：

- 本地代码事实优先于外部文档描述
- 未核实的历史报告数字不能直接复用
- 外部资料只能补充解释，不能覆盖本地证据

---

## 4. 输出分级规则

所有问题必须归入以下三类之一：

- `已验证问题`
  - 有明确代码位置、行为证据或命令输出支撑
- `证据不足`
  - 方向可能成立，但当前没有足够证据
- `不成立/已修正`
  - 原报告结论与当前仓库事实不符

禁止：

- 把“证据不足”包装成阻断项
- 把“历史上可能存在”写成“当前仍存在”
- 用模糊表述代替文件位置和证据

---

## 5. 严重程度判定

### Critical

仅在以下情况可使用：

- 明确的远程/本地命令注入、任意代码执行、提权面
- 直接破坏系统核心承诺，且已由代码确认
- 当前交付会被实质性绕过或错误放行

### High

用于：

- 功能核心路径未实现或错误实现
- 规范与实现存在重大漂移
- 并发/状态一致性风险明确存在
- 自动化门禁缺失，导致质量无法兜底

### Medium

用于：

- 性能退化风险
- 默认行为不稳健
- 校验时机偏后
- 测试覆盖不完整但已有部分保障

### Low

用于：

- 设计边界不严谨
- 安全意识良好但保护范围有限
- 长期治理项或文档化问题

---

## 6. 标准执行步骤

必须按顺序执行，不得跳步。

### Step 1: 明确审查对象

识别本次要核对的是：

- 单份报告
- 一组 review bundle
- 整个仓库
- 某个子系统

产出：

- 审查范围说明
- 待核对文档路径列表

### Step 2: 盘点真实规模

使用命令统计当前仓库事实，例如：

```bash
rg --files src | wc -l
rg --files tests | wc -l
rg --files skills -g 'SKILL.md' | wc -l
rg -n "\b(it|test)\(" tests | wc -l
```

产出：

- 当前 `src/` 文件数
- 当前 `tests/` 文件数
- 当前仓库内 `SKILL.md` 数量
- 当前测试声明数量口径

### Step 3: 抽取报告中的关键断言

优先识别：

- 安全漏洞
- Gate / Skill / 文档一致性问题
- 核心功能未实现
- 测试/CI/覆盖率类结论
- 架构级 Critical/High 结论

产出：

- “待核对断言清单”

### Step 4: 逐条找代码证据

对每个关键断言执行：

- 全局检索相关符号
- 打开对应文件
- 定位函数/配置/条件实现
- 核对报告描述与代码是否一致

推荐命令：

```bash
rg -n "pattern" src tests skills
sed -n 'start,endp' path/to/file.ts
```

产出：

- 每条断言的证据路径
- 成立 / 不成立 / 证据不足

### Step 5: 运行必要验证命令

最少执行：

```bash
pnpm test -- --run
pnpm typecheck
```

视情况补充：

- 覆盖率配置读取
- CI 目录检查
- Gate / release / audit / skill 相关测试抽样

产出：

- 可复核的运行结果摘要

### Step 6: 纠正原报告口径

重点识别以下误判：

- “无测试”其实是“有测试但只覆盖占位实现”
- “无门禁”其实是“门禁存在但被自动跳过”
- “后置校验”被写成“输出写入后才发现”
- “仓库内 Skill 数”混入用户级或外部技能目录

产出：

- 修正说明列表

### Step 7: 回写审查文档

文档必须改写成证据驱动格式，至少包含：

- 已验证问题
- 证据不足项
- 不成立/已修正项
- 本次核验使用的命令
- 当前仓库真实统计口径

### Step 8: 最终一致性检查

检查输出是否满足：

- 所有高优问题都有具体文件位置
- 所有数字都可由当前仓库复核
- 没有把推测写成结论
- 没有遗漏“旧结论已失效”的修正说明

---

## 7. 常见误判纠正规则

### 规则 1：统计必须现场重算

以下数字不得沿用历史报告：

- 源码文件数
- 测试文件数
- 测试数量
- Skill 数量

### 规则 2：找不到代码位置就不能下结论

如果问题描述无法对应到当前仓库的具体文件、函数、配置或测试：

- 标记为 `证据不足`
- 不得继续保留为 `High` 或 `Critical`

### 规则 3：实现存在 ≠ 行为正确

如果代码有实现，但语义与文档冲突：

- 归为“规范与实现漂移”
- 不得写成“完全未实现”

### 规则 4：测试存在 ≠ 风险已覆盖

如果测试只覆盖 mock、stub、占位路径：

- 应写成“测试存在，但未覆盖真实链路”
- 不得写成“完全无测试”

### 规则 5：多处证据必须分别说明

如果同一问题横跨：

- `SKILL.md`
- `condition-registry.ts`
- `advance.ts`
- `golive.ts`

则要写清每处证据的角色，不要混成一条模糊结论。

---

## 8. 文档输出契约

若输出 review bundle，推荐至少包含：

- `00-综合审查报告.md`
- `01-代码质量审查.md`
- `02-架构审查.md`
- `03-安全与性能审查.md`
- `04-测试审查.md`
- `05-Skill与Gate审查.md`

每份文档的最小结构：

1. 审查范围
2. 核验方法
3. 已验证问题
4. 修正项
5. 建议优先级

综合报告还必须额外包含：

- 当前仓库真实统计口径
- 本次执行过的验证命令
- 对上一版结论的集中修正说明

---

## 9. 推荐命令清单

### 盘点规模

```bash
rg --files src | wc -l
rg --files tests | wc -l
rg --files skills -g 'SKILL.md' | wc -l
rg -n "\b(it|test)\(" tests | wc -l
```

### 代码检索

```bash
rg -n "execSync|execFileSync|shell:\s*true" src tests skills
rg -n "required_mcps|completion_markers|G-IMPL-01|G-VERIFY-01" src tests skills
rg -n "AUTO_ADVANCE|smoke-test-report|release-note" src tests skills
```

### 核验命令

```bash
pnpm test -- --run
pnpm typecheck
```

### 文档检查

```bash
git diff --stat -- docs/review-bundles/<bundle>
git status --short docs/review-bundles/<bundle>
```

---

## 10. Agent 执行硬规则

- 不要把猜测写成事实
- 不要引用未复核的旧数字
- 不要把外部资料当作本地事实
- 不要为了“显得全面”硬凑问题数
- 不要把长期治理项标成当前阻断项
- 不要忽略“原报告哪些结论已失效”

如果无法确认，统一写：

> 当前证据不足，暂不保留为已验证问题。

---

## 11. 成功标准

完成一次审查后，必须满足：

- 文档中的核心结论都能追溯到当前仓库证据
- 高优先级问题都有准确文件定位
- 统计口径与当前仓库一致
- 原报告中的错误结论被明确纠正
- 多 Agent 结论最终能收敛到统一版本

---

## 12. 推荐使用方式

推荐按以下顺序使用：

1. 盘点范围与统计
2. 提取关键断言
3. 逐条取证
4. 运行测试与类型检查
5. 按“已验证 / 证据不足 / 不成立”分类
6. 回写 bundle
7. 再做一次最终一致性检查

这份 SOP 的目的不是让审查显得更重，而是让审查结论更可信。

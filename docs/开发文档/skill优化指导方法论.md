# Skill 优化指导方法论

更新时间：2026-03-15  
适用范围：本仓库所有 `skills/**/SKILL.md` 及其 `references/` 配套文档

## 目标

这份文档总结的是一套“如何把 skill 从设想稿优化成可执行手册”的方法论。核心目标不是把 skill 写得更长，而是让它：

- 和当前实现一致
- 对 agent 可执行
- 对多 agent 可收敛
- 对审查和回归可验证

一句话原则：

**skill 的价值不在于描述理想流程，而在于稳定指导当前仓库里的真实执行。**

## 什么时候要优化 skill

出现以下任一信号，就应该进入 skill 优化：

- skill 描述的能力超过当前代码真实能力
- agent 按 skill 执行时频繁出现误判、误用或过度发挥
- skill 的配置项、字段名、产物名与实现侧真理源不一致
- reference 文档越来越像通用教程，而不是当前仓库执行规范
- 审查时发现“文档说支持，代码其实没接线”

## 核心原则

### 1. 真理源优先于文档习惯

skill 不能自说自话。所有关键约束都必须回到实现真理源确认，例如：

- 配置项来自哪个 schema
- 状态文件由哪个模块读写
- 守卫逻辑在哪个文件执行
- 报告结构由哪个生成器输出

如果实现和文档冲突：

- 先承认冲突
- 再决定是改文档还是改代码
- 不允许继续保留“双真理源”

### 2. 当前模式和目标模式必须拆开

很多 skill 的最大问题不是错，而是把“目标态”写成了“当前态”。

标准做法：

- `当前模式`：今天就能稳定执行的流程
- `目标模式`：未来希望接线或自动化的能力

写法要求：

- 当前模式要保守、可证明
- 目标模式可以保留，但必须明确标注为未来能力

### 3. 主文档负责流程，reference 负责约束

最稳的结构不是把所有内容都塞进 `SKILL.md`，而是分层：

- `SKILL.md`
  - 触发条件
  - 执行流程
  - 硬规则
  - 角色边界
  - 当前模式 / 目标模式
- `references/*`
  - 上下文包格式
  - 报告模板
  - TDD 守卫
  - 测试模板
  - traces 规范
  - 验证清单

判断标准：

- agent 一次执行时先读主文档就能开始
- 遇到具体格式和约束再跳 reference

### 4. skill 要压缩，不要膨胀

优化 skill 通常不是“继续加内容”，而是：

- 去掉重复表达
- 删掉教程型通用内容
- 保留当前仓库的硬约束
- 收敛到最小可执行信息

对 agent 来说，密度比长度更重要。

### 5. 多 agent 场景必须先写边界

凡是涉及并发、subagent、共享状态文件的 skill，最先要写清楚的不是“怎么并发”，而是：

- 谁负责读计划
- 谁负责更新共享状态
- 谁能写哪些文件
- 哪些场景必须串行

如果边界不清，skill 会先在状态管理上失败，而不是在功能实现上失败。

## 标准优化流程

```text
发现漂移
   |
   v
盘点 skill 全目录
   |
   v
建立文档 <-> 实现 对照表
   |
   v
区分 当前模式 / 目标模式
   |
   v
逐文件判定:
可保留 / 需收敛 / 已过时
   |
   v
先改主文档，再改 reference
   |
   v
补测试或回归现有文档测试
   |
   v
输出优化审查文档
```

## 具体操作步骤

### Step 1. 盘点 skill 全目录

先列出完整范围：

- `SKILL.md`
- `references/` 下所有文件
- 如有 `scripts/`、`assets/`、模板文件，也一起纳入

目标：

- 不漏文件
- 不只审主文档

### Step 2. 建立实现真理源映射

给 skill 里的每类约束找到对应代码文件：

- 配置约束 -> `config-schema.ts`
- 字段格式 -> parser / type 定义
- 守卫规则 -> runtime guard / validator
- 状态文件 -> checkpoint / state manager
- 报告结构 -> report generator
- 测试约束 -> 当前测试框架与已有测试

建议输出一个简单映射表：

```text
文档项                -> 实现真理源
max_parallel          -> config-schema.ts
TDD 预检              -> guards.ts
context pack          -> context-packer.ts
batch report          -> report-generator.ts
checkpoint            -> checkpoint.ts
```

### Step 3. 逐文件判断“是否仍有价值”

每个文件只做三类判断：

- `已对齐`
- `部分对齐，需要收敛`
- `已过时，需要重写`

这一步不要空泛评价，要说明为什么：

- 字段名错了
- 命令过时了
- 内容太泛化
- 实现未接线
- 重复表达太多

### Step 4. 先收敛主文档

`SKILL.md` 优先级最高，因为 agent 最先读它。

主文档优化时重点看这 7 项：

1. 是否清楚区分当前模式和目标模式
2. 是否绑定真实配置入口
3. 是否说明主进程 / subagent 边界
4. 是否明确共享文件写入规则
5. 是否写清串行降级条件
6. 是否包含最小实现、Simplicity First、Surgical Changes 一类硬约束
7. 是否仍有 FAQ、重复章节、重复守卫描述

### Step 5. 再收敛 reference

reference 不求全，求准。

重点检查：

- 示例命令是不是当前仓库真实命令
- 字段名是不是当前类型定义里的字段名
- 模板是不是和当前输出结构一致
- 内容是不是通用教程而不是当前执行规范

经验规则：

- 如果一个 reference 80% 都是通用教程，应该砍
- 如果一个 reference 只剩 20% 关键约束，也值得保留

### Step 6. 为 skill 建立最小一致性测试

skill 不是只能人工审。

至少要有一类文档一致性测试，验证这些关键锚点：

- 主文档是否包含关键硬规则
- 主文档和 reference 是否引用同一真理源
- 不允许已删除的关键约束被“优化掉”

这种测试不求覆盖所有自然语言内容，但能防止回归。

### Step 7. 输出优化审查文档

优化结束后，不只改文件，还要留一份审查结论文档。标准内容包括：

- 审查范围
- 实现真理源
- 文档关系图
- 逐文件结论
- 已对齐项
- 待优化项
- 优先级排序
- 验证结果

这样下一次优化不会从头开始。

## 最常见的反模式

### 1. 把未来能力写成当前能力

表现：

- 文档写“自动并发”“自动 checkpoint”“自动调 subagent”
- 实现里其实还是占位逻辑

修正：

- 拆成当前模式 / 目标模式

### 2. 用不存在的配置项当真理源

表现：

- skill 写了 `runtime.code.*`
- 实际 schema 里根本没有

修正：

- 只引用当前代码里真实存在的配置入口

### 3. 模板命令漂移

表现：

- 文档里还是 `npm test`
- 仓库已经统一成 `pnpm`

修正：

- 所有 reference 模板命令统一跟当前仓库命令体系

### 4. reference 变成教程

表现：

- 大量通用 TS / React / Node 教学内容
- 和当前 skill 的执行关系很弱

修正：

- 删掉通用知识，只保留当前仓库必须遵守的约束

### 5. 多 agent 边界写不清

表现：

- 写了“可并发”
- 但没写共享文件谁来更新

修正：

- 先写边界，再写并发流程

## 推荐的输出结构

一个高质量 skill 最终建议稳定在下面这套结构：

```text
SKILL.md
references/
  context-pack-schema.yaml
  tdd-guard.md
  report-template.md
  test-template.md
  traces-trailer.md
  target-env-verification.md
  code-standards.md
  diff-template.md
tests/
  <skill-docs-consistency>.test.ts
docs/review-bundles/
  <skill-优化方案>.md
```

不是每个 skill 都必须有全部 reference，但凡存在就要说明其必要性。

## 判定一个 skill 是否已经“优化到位”

满足以下条件，基本就算到位：

- agent 按主文档就能开始执行
- reference 都有明确职责，不是泛化资料库
- 文档中所有关键字段和命令都能在实现里找到真理源
- 当前模式和目标模式没有混写
- 有最小一致性测试兜底
- 审查文档能说清楚哪些文件还需继续优化

## 07-code 案例沉淀

这次 `07-code` 的优化经验可以概括成 4 句话：

1. 先承认 skill 比实现超前，再收敛成当前模式。
2. 先改 `SKILL.md`，再改 reference，不要反过来。
3. 先让字段名、命令、状态文件名和实现对齐，再谈最佳实践。
4. 最后用文档测试守住关键锚点，避免后续“优化过头”。

## 后续建议

如果以后要优化别的 skill，建议直接复用这套顺序：

1. 盘点目录
2. 建立真理源映射
3. 区分当前模式 / 目标模式
4. 主文档收敛
5. reference 收敛
6. 补最小文档测试
7. 输出优化审查文档

这会比“想到哪改到哪”稳定得多，也更适合多 agent 协作。

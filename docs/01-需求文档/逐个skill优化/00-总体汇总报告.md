## 核心问题深度分析

### 1. Skill Runtime 核心问题

#### 问题根因
- **RUNTIME_COMMANDS 覆盖不完整**: 设计时未考虑所有 CLI 命令，导致路由逻辑不完整
- **REMOVED_SKILLS 数据污染**: 可能是数组合并错误或导出逻辑问题

#### 影响评估
- **严重性**: 🔴 高
- **影响范围**: 所有 skills
- **用户影响**: 部分命令无法通过 `/spec-first:xxx` 语法调用

#### 修复优先级
1. **立即修复**: REMOVED_SKILLS 数据污染（影响类型安全）
2. **本周内**: 补全 RUNTIME_COMMANDS 或明确筛选标准

### 2. orchestrate Skill 问题

#### 问题根因
- **Stage 映射不完整**: SKILL.md 和 skill-mapping.md 只覆盖到 06_wrap_up，未考虑 07_release 和 08_done
- **参数语义不清晰**: --auto-advance 与 --auto 的区别未在文档中体现

#### 影响评估
- **严重性**: 🔴 高
- **影响范围**: 主工作流编排
- **用户影响**: 无法自动推进到 release 和 done 阶段

#### 修复优先级
1. **立即修复**: 补全 Stage 映射（阻塞 release 流程）
2. **本周内**: 明确参数语义

### 3. code Skill 问题

#### 问题根因
- **失败率控制未实现**: changelog 声称支持但配置缺失，可能是版本迭代遗留问题
- **缺少 SHARED.md 约束**: 未遵守 3-Strike Error Protocol 和 Worktree First 原则

#### 影响评估
- **严重性**: 🔴 高
- **影响范围**: 代码实现质量
- **用户影响**: 批量执行时无法根据失败率自动停止，高风险任务可能污染主工作区

#### 修复优先级
1. **立即修复**: 删除失败率控制声明或实现该功能
2. **本周内**: 补充 3-Strike Error Protocol 和 Worktree First 原则

### 4. catchup Skill 问题

#### 问题根因
- **缺少 Skill 类型声明**: 未明确声明属于"路由控制型"
- **P0-P5 术语冲突**: 信息源优先级与 SHARED.md 的执行阶段使用相同标记

#### 影响评估
- **严重性**: 🟡 中
- **影响范围**: 会话恢复流程
- **用户影响**: AI 可能误解 catchup 的执行流程

#### 修复优先级
1. **立即修复**: 增加 Skill 类型声明和 confirm_policy 字段
2. **本周内**: 将信息源优先级改用不同标记（Priority-0/1/2）

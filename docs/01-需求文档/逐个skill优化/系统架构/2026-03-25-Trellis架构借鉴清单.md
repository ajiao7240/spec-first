# Trellis 架构借鉴清单

> 文档日期：2026-03-25
>
> 分析对象：`/Users/kuang/xiaobu/Trellis`
>
> 目标：从 Trellis 的工作流治理、记忆分层、任务建模、适配器注册、会话追踪等设计中，提炼出对 `spec-first` 系统架构真正有价值的借鉴点。

## 1. 结论先行

Trellis 最值得借鉴的不是某个单独脚本，而是一整套“工作流治理架构”：

- 把长期规范、任务状态、会话历史拆成不同状态域。
- 把任务当成一级实体，而不是普通文档附件。
- 把平台差异收敛到 registry / adapter，而不是散落在 if / else 中。
- 把工作流的关键检查做成前置治理，而不是事后补救。
- 把思考指南、恢复机制、会话日志做成可执行资产，而不是纯说明文档。

对 `spec-first` 来说，这些能力的价值不在于“照搬实现”，而在于把系统从“能跑”提升到“能长期协作、能恢复、能治理、能扩展”。

## 2. Trellis 的可借鉴架构能力

### 2.1 三层记忆模型

Trellis 把长期规范、当前任务、会话历史拆成不同目录和状态域，形成稳定的分层记忆。

可参考：

- [workflow.md](/Users/kuang/xiaobu/Trellis/.trellis/workflow.md)
- [trellis-memory-system.md](/Users/kuang/xiaobu/Trellis/docs/trellis-memory-system.md)
- [workspace/index.md](/Users/kuang/xiaobu/Trellis/.trellis/workspace/index.md)

对 `spec-first` 的价值：

- 减少把所有上下文塞进单一文档或单一运行态的风险。
- 让规范、任务、会话不互相污染。
- 让 AI 继续工作时可以明确恢复边界。

建议落点：

- `docs/` 只承载规范说明和分析结论。
- `specs/` 只承载 Feature 运行态和阶段信息。
- `.spec-first/` 只承载项目级真源和当前工作态。
- `workspace/` 类目录承载会话日志和恢复上下文。

### 2.2 任务作为一级实体

Trellis 的 `task.json` 不是附属文件，而是任务本体，包含状态、负责人、分支、下一步动作、子任务和关联文件。

可参考：

- [.trellis/scripts/task.py](/Users/kuang/xiaobu/Trellis/.trellis/scripts/task.py)
- [.trellis/scripts/common/task_store.py](/Users/kuang/xiaobu/Trellis/.trellis/scripts/common/task_store.py)

对 `spec-first` 的价值：

- 任务可以变成可追踪、可恢复、可分派的结构化对象。
- 任务推进不再只依赖自然语言和会话记忆。
- 任务完成标准、依赖关系和执行记录都能结构化沉淀。

建议落点：

- 把“任务文档”升级为“任务对象 + 任务状态 + 任务产物”的组合。
- 让 `spec-first` 的计划、执行、验证、归档形成稳定状态机。

### 2.3 registry / adapter 驱动的适配方式

Trellis 把宿主平台差异集中到 registry 和 adapter，不把不同平台的分支逻辑散布在核心流程里。

可参考：

- [.trellis/scripts/common/registry.py](/Users/kuang/xiaobu/Trellis/.trellis/scripts/common/registry.py)
- [.trellis/scripts/common/cli_adapter.py](/Users/kuang/xiaobu/Trellis/.trellis/scripts/common/cli_adapter.py)
- [.trellis/spec/cli/backend/platform-integration.md](/Users/kuang/xiaobu/Trellis/.trellis/spec/cli/backend/platform-integration.md)

对 `spec-first` 的价值：

- 宿主适配可以从业务逻辑中解耦出来。
- 新增 Claude / Cursor / Codex / 其他宿主时，不需要到处改核心模块。
- CLI 路由、hook 注入、上下文拼装都可以用 registry 统一管理。

建议落点：

- `src/core/host-adapters/` 与 `src/core/tool-integration/` 应该继续收敛为独立适配层。
- 所有宿主差异优先通过注册表声明，不要靠硬编码条件分支。

### 2.4 会话日志与恢复机制

Trellis 把会话历史视为可恢复资产，而不是临时聊天记录。

可参考：

- [.trellis/scripts/add_session.py](/Users/kuang/xiaobu/Trellis/.trellis/scripts/add_session.py)
- [.trellis/workspace/index.md](/Users/kuang/xiaobu/Trellis/.trellis/workspace/index.md)

对 `spec-first` 的价值：

- 可以更可靠地续接跨 session 的工作。
- 可以把“上次做到哪一步”变成可机读状态。
- 可以把开发过程里的关键决策记录下来，降低上下文遗失成本。

建议落点：

- 给每个 Feature 增加固定的会话摘要和恢复入口。
- 把“会话记录”与“任务状态”分离，避免混写。

### 2.5 思考指南前置

Trellis 把跨层、跨平台的思考指南显式化，要求在执行前先建立边界感。

可参考：

- [.trellis/spec/guides/index.md](/Users/kuang/xiaobu/Trellis/.trellis/spec/guides/index.md)
- [.trellis/spec/guides/cross-layer-thinking-guide.md](/Users/kuang/xiaobu/Trellis/.trellis/spec/guides/cross-layer-thinking-guide.md)
- [.trellis/spec/guides/cross-platform-thinking-guide.md](/Users/kuang/xiaobu/Trellis/.trellis/spec/guides/cross-platform-thinking-guide.md)

对 `spec-first` 的价值：

- 能减少“实现前没想清楚边界，导致后续重构成本高”的问题。
- 尤其适合 `skill-runtime`、`ai-orchestrator`、`tool-integration` 这类容易耦合的模块。

建议落点：

- 为关键模块补“边界检查清单”。
- 让架构原则先于实现动作生效。

### 2.6 工作流治理前置

Trellis 的流程不是“写代码 -> 再补文档”，而是把规范、任务、会话、验证、归档串成一条治理链。

可参考：

- [.trellis/workflow.md](/Users/kuang/xiaobu/Trellis/.trellis/workflow.md)
- [trellis-workflow-analysis.md](/Users/kuang/xiaobu/Trellis/docs/trellis-workflow-analysis.md)
- [trellis-existing-project-iteration.md](/Users/kuang/xiaobu/Trellis/docs/trellis-existing-project-iteration.md)

对 `spec-first` 的价值：

- 新功能和存量改造可以收敛到同一条流程链。
- 任务推进不只是执行，而是执行、检查、记录、归档一体化。

建议落点：

- `spec-first` 的 stage gate 可以进一步吸收 Trellis 的治理思路。
- 让“计划、实施、验证、总结”形成连续链路，而不是松散文档堆叠。

## 5. 对 spec-first 的直接影响

如果把 Trellis 的借鉴点真正吸收到 `spec-first`，系统架构会出现四个明确改善：

1. 任务将从“文档推进”升级成“结构化对象推进”。
2. 会话恢复将从“人工回忆”升级成“可恢复状态”。
3. 宿主适配将从“散落逻辑”升级成“注册表驱动”。
4. 工作流将从“阶段串联”升级成“治理链路”。

这四点和当前 `spec-first` 的系统架构是兼容的，不需要推倒重来，只需要继续收敛边界。

## 6. 建议的落地顺序

1. 先补任务对象化和会话摘要机制。
2. 再把宿主适配和上下文分发统一进 registry / adapter。
3. 再把关键模块的边界检查清单固化到文档和执行前检查里。
4. 最后把工作流治理链与现有 stage gate 合并成统一路径。

## 7. 结论

Trellis 对 `spec-first` 的最大价值，不是提供一个可直接复制的技术栈，而是提供一个更成熟的“工作流治理模型”。

如果只选三件事优先吸收：

1. 任务一级实体化。
2. 规范 / 任务 / 会话三层分离。
3. registry 驱动的宿主适配。

这三件事能直接增强 `spec-first` 的扩展性、可恢复性和可治理性。

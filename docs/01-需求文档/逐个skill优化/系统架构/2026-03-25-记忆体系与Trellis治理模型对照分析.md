# 记忆体系与 Trellis 治理模型对照分析

> 文档日期：2026-03-25
>
> 关联对象：
>
> - [Spec-First 记忆体系深度分析](./2026-03-25-记忆体系深度分析.md)
> - [Trellis 架构借鉴清单](./2026-03-25-Trellis架构借鉴清单.md)
>
> 目的：把 `spec-first` 的分层记忆体系与 Trellis 的工作流治理模型放到同一张分析框架里，明确两者的对应关系、可合并点和后续架构启发。

## 1. 结论先行

这两份分析看似在讲不同的东西，实际上在回答同一个问题：

- `spec-first` 关注的是“记忆如何分层、如何写入、如何投影、如何守卫”。
- `Trellis` 关注的是“任务如何推进、会话如何恢复、规范如何约束、适配如何收口”。

把它们合并来看，可以得到一个更完整的判断：

> `spec-first` 已经有了“真源/投影/守卫/策略”的技术骨架，Trellis 补上的则是“任务化/会话化/治理化”的运行机制。

因此，后续演进的重点不应该是增加更多文档，而是把下面四件事做实：

1. 任务对象化。
2. 会话恢复标准化。
3. 真源 / 投影 / 守卫边界继续收紧。
4. registry / adapter 驱动的上下文与宿主分发。

## 2. 两套体系的本质对应关系

### 2.1 `spec-first` 记忆体系在解决什么

`spec-first` 的记忆体系深度分析已经明确，它不是单一存储，而是分层知识与运行态系统：

- 真源层保存项目级认知事实。
- Feature 运行态层保存任务/阶段/上下文。
- docs 投影层给人阅读。
- 会话 / Hook 层负责触发和守卫。
- skill 输入契约层负责决定上下文喂入方式。

它的核心目标是：

- 让记忆可恢复。
- 让记忆可同步。
- 让记忆可投影。
- 让记忆可审计。

### 2.2 Trellis 治理模型在解决什么

Trellis 的核心不是代码分层，而是工作流治理：

- 规范、任务、会话、平台适配拆成不同状态域。
- 任务本身就是结构化对象。
- 会话历史是可恢复资产。
- registry / adapter 收口平台差异。
- hooks / validation 做前置治理。

它的核心目标是：

- 让工作流可持续。
- 让任务可推进。
- 让状态可恢复。
- 让宿主差异可管理。

### 2.3 二者的重叠区

二者真正重叠的地方，不是“存了什么”，而是“如何让状态持续有效”：

- `spec-first` 的真源层，和 Trellis 的规范层，都是长期认知资产。
- `spec-first` 的 Feature 运行态层，和 Trellis 的 task/workspace 层，都是工作推进资产。
- `spec-first` 的会话 / Hook 层，和 Trellis 的 session/hook 层，都是持续协作资产。
- `spec-first` 的 context-resolver，和 Trellis 的 registry / adapter，都是分发与路由资产。

## 3. 对照矩阵

| 维度 | spec-first | Trellis | 对架构的共同启发 |
| --- | --- | --- | --- |
| 长期知识 | `.spec-first/runtime/first/*` 真源 | `.trellis/spec/*` 长期规范 | 不能把规范当作临时文档 |
| 工作态 | `specs/<featureId>/stage-state.json` 等 | `.trellis/tasks/*`、`workspace/*` | 任务应当是结构化状态，而不是纯文本 |
| 输出层 | `docs/first/*` 投影 | `docs/*`、workspace 记录 | 输出层必须和真源分离 |
| 会话恢复 | `session-hook.ts`、`ai-runtime-hook.ts`、`catchup.ts` | `add_session.py`、workspace 日志 | 恢复能力应成为系统标准能力 |
| 分发策略 | `context-resolver.ts`、`skill-input-contracts.yaml` | registry / adapter | 上下文与宿主差异应由策略层统一管理 |
| 质量守卫 | `gate-engine`、`first-governance.ts` | hooks / validation | 前置校验比事后补救更重要 |
| 工作流治理 | stage gate、transition 流程 | workflow + task 流程 | 流程本身应是产品能力的一部分 |

## 4. 结合后的关键判断

### 4.1 记忆不是“档案”，而是“运行态的一部分”

这是两份分析合并后最重要的结论。

`spec-first` 的记忆体系已经说明，记忆是分层状态流转，不是单点存档。Trellis 进一步说明，任务、会话、规范和平台适配必须一起运行，才能形成真正可持续的工作流。

这意味着：

- 真源不是供回忆的资料，而是供系统继续运行的事实底座。
- 任务不是说明书，而是可推进的状态对象。
- 会话不是聊天记录，而是可恢复工作现场。

### 4.2 任务对象化是下一阶段最关键的升级

Trellis 把 `task.json` 做成一级实体，这对 `spec-first` 的启发非常直接。

`spec-first` 目前已经有 `specs/<featureId>/stage-state.json`、`task_plan.md`、`findings.md` 这类运行态，但如果要提升长期协作能力，下一步应该更明确地把“任务状态、上下文、产物、依赖关系”收敛成一个结构化任务对象。

这样做的收益是：

- 任务可恢复。
- 任务可分派。
- 任务可追踪。
- 任务完成条件可校验。

### 4.3 会话恢复应该被提升到系统入口级

`spec-first` 已经有 `catchup` 和 hook 机制，但从 Trellis 的做法看，会话恢复不应只是附加命令，而应该是标准入口。

这意味着：

- 每次 session 启动都应该知道当前 Feature 的恢复位置。
- 恢复逻辑不应依赖人工回忆。
- 会话日志与任务状态应分离，但能互相索引。

### 4.4 真源 / 投影 / 守卫边界已经正确，但还要继续收紧

`spec-first` 的记忆体系已经把 `docs/first/*` 明确成投影层，这是正确的。

Trellis 的启发在于：不仅要分层，还要保证每层只做一件事。

因此后续要继续避免：

- docs 反向变成运行真源。
- hook 里承载业务规则。
- task/state 混写到一处。
- 适配逻辑散落在各个命令里。

### 4.5 registry / adapter 是上下文分发的关键

如果把 `spec-first` 的 `context-resolver.ts` 和 Trellis 的 registry / adapter 思路结合起来，可以得出一个更明确的架构方向：

- 宿主差异应该通过 registry 声明。
- 上下文选择应该通过策略层决定。
- 资产加载应该通过契约驱动，而不是硬编码。

这会直接提升系统可扩展性。

## 5. 对 `spec-first` 的直接影响

### 5.1 架构层面的变化

合并这两份分析后，`spec-first` 的架构应更清晰地向下面四个方向收敛：

1. **任务对象化**
   - 把工作流推进单位从文档升级成结构化对象。

2. **会话标准化**
   - 把 session 恢复、上下文续接、日志记录统一成标准机制。

3. **边界继续收紧**
   - 真源、投影、守卫、策略各司其职。

4. **分发能力注册化**
   - 宿主、上下文、适配、资产加载都尽量 registry 驱动。

### 5.2 组织层面的变化

从“文档管理”转向“运行态治理”之后，`spec-first` 的文档体系也应该更克制：

- 文档应该说明状态如何流转。
- 文档应该说明谁负责写入、谁负责更新、谁负责实现。
- 文档不应该重复承载运行态。

### 5.3 扩展性的变化

这套结合后的模型对扩展性有直接帮助：

- 新 Feature 更容易接入。
- 新宿主更容易适配。
- 新策略更容易替换。
- 新会话更容易恢复。

## 6. 风险提醒

### 6.1 不要把 Trellis 的机制误解成“脚本更强”

Trellis 的价值不是 Python 脚本多，而是它把脚本变成了治理入口。

对 `spec-first` 来说，应该吸收的是治理思想，而不是脚本中心化本身。

### 6.2 不要把记忆体系误解成“多存一点”

记忆体系的重点不是存储量，而是状态边界。

如果边界不清，记忆越多越乱；如果边界清楚，少量状态也能恢复整个工作流。

### 6.3 不要把任务对象化做成新的文档膨胀

任务对象化的目标是把状态结构化，不是把文档数量翻倍。

真正要增加的是结构，不是重复说明。

## 7. 建议的下一步

如果继续推进，建议按这个顺序做：

1. 明确 `spec-first` 的任务对象字段和状态边界。
2. 定义会话恢复的标准输入和标准输出。
3. 收紧 `context-resolver`、`hook`、`gate` 三层边界。
4. 把 registry / adapter 的思想落实到宿主和上下文分发。
5. 再回头修正文档索引，让文档顺着运行态走。

## 8. 最终结论

把 `spec-first` 的记忆体系和 Trellis 的治理模型合起来看，可以得到一个更稳定的方向：

> `spec-first` 不是单纯的“知识记录系统”，而是一个“以真源为底座、以任务为单位、以会话为通道、以治理为约束”的研发运行系统。

而 Trellis 的价值，恰好是补齐了这套系统里最容易被忽略的那部分：

- 任务如何持续推进。
- 会话如何可靠恢复。
- 平台如何统一适配。
- 工作流如何持续治理。

这也是后续架构优化最值得优先吸收的部分。

## 2. 扩展产物（业界最佳实践）

### 2.1 环境与运维产物

## 4. 智能端类型检测规则

### 4.1 检测规则（7 种主类型 + 子类型）

```yaml
端类型检测:
  后台服务:
    - pom.xml, build.gradle, build.gradle.kts  # Java/Kotlin
    - go.mod, go.sum                            # Go
    - requirements.txt, pyproject.toml          # Python
    - Cargo.toml                                # Rust
    - 且无前端目录（无 src/pages, src/components 等）

  前端 Web:                                     # 合并 Admin + H5
    Admin 子类型:
      - package.json + ant-design/antd/element-plus/arco-design/@alifd/next
    H5 子类型:
      - package.json + vant/@nutui/nutui/mint-ui
      - 或检测到移动端适配代码
    通用前端:
      - package.json + react/vue/angular/svelte（无 Admin/H5 特征）

  移动端 App:                                    # 合并 iOS + Android
    iOS 子类型:
      - *.xcodeproj, *.xcworkspace, Podfile
      - Package.swift + iOS target
    Android 子类型:
      - AndroidManifest.xml
      - build.gradle + com.android.application

  跨平台:
    Flutter: pubspec.yaml
    React Native: package.json + react-native
    UniApp: manifest.json + uni
    KMP: build.gradle.kts + kotlin("multiplatform")

  PC 桌面:                                       # 合并 Win/Mac/Linux
    Windows 子类型: *.csproj/*.sln / CMakeLists.txt + WinAPI/MSVC
    macOS 子类型: *.xcodeproj + macOS target / Package.swift + macOS
    Linux 子类型: CMakeLists.txt/Makefile + GTK/Qt/SDL
    跨平台桌面: Electron(package.json + electron) / Tauri(Cargo.toml + tauri)

  Monorepo:
    - turbo.json                               # Turborepo
    - nx.json                                  # Nx
    - lerna.json                               # Lerna
    - pnpm-workspace.yaml                      # pnpm workspaces

  后端+前端混合:
    - 同时检测到后台服务特征 + 前端 Web 特征
    - 目录结构包含: backend/ + admin/ 或 server/ + web/ 或 api/ + frontend/
```

> **说明**：子类型影响模板选择（如 architecture.md 的布局差异），但不影响产物集的选择。

### 4.2 检测优先级

当同时检测到多种端类型特征时，按以下优先级判断：

1. **Monorepo** - 优先判断（可能包含多种端类型）
2. **后端 + 前端混合** - 同时检测到后端和前端特征
3. **跨平台** - Flutter/RN/UniApp/KMP
4. **移动端 App** - iOS/Android
5. **PC 桌面** - Windows/macOS/Linux/Electron/Tauri
6. **前端 Web** - Admin/H5/通用前端
7. **后台服务** - 默认兜底

## 6. 核心产物 vs 扩展产物

### 6.1 分层策略

| 层级 | 产物数 | 说明 | 执行时间 |
|------|--------|------|----------|
| **Layer 0 核心** | 3 个 | tech-stack + codebase-overview + quick-commands | 秒级 |
| **Layer 1 标配** | 4-6 个 | 按端类型自动选择 | 分钟级 |
| **Layer 2 扩展** | 2-4 个 | 按需生成或 `depth=deep` | 按需 |

### 6.2 Layer 0 核心产物（所有端必须，quick 模式产物）

> Layer 0 即 quick 模式产物，所有端类型必须生成。deep 模式额外生成 README.md 作为索引。

| 产物 | 生成方式 | 说明 |
|------|----------|------|
| tech-stack.md | 主线程 | 技术栈摘要 |
| codebase-overview.md | 主线程 | 目录结构概览 |
| quick-commands.md | 主线程 | 构建/测试/运行命令 |

### 6.3 Layer 1 标配产物（按端类型）

| 端类型 | 标配产物 |
|--------|----------|
| 后台服务 | architecture, api-docs, database-er, domain-model, external-deps |
| 前端 Web | architecture, api-docs, component-library, state-management |
| 移动端 App | architecture(简化), api-docs, app-release, compatibility |
| 跨平台 | architecture, api-docs, platform-bridge, compatibility |
| PC 桌面 | architecture, api-docs, desktop-native, release |
| Monorepo | architecture, module-graph, env-config |
| 后端+前端混合 | 后台服务 + 前端 Web 全部产物 |

### 6.4 Layer 2 扩展产物（按需）

| 产物 | 触发条件 |
|------|----------|
| call-graph.md | `depth=deep` 或显式要求 |
| monitoring.md | 检测到监控配置 |
| performance.md | `depth=deep` |
| i18n.md | 检测到多语言配置 |

### 7.4 与 00-first 对比分析

| 维度 | 00-first | 市场 Skill |
|------|----------|------------|
| **产物数量** | 9-16 个完整文档 | 通常 1-3 个 |
| **端类型适配** | 7 种主类型 + 子类型（后端/前端Web/移动端/跨平台/PC桌面/Monorepo/混合） | 通用，无端类型区分 |
| **语言/框架支持** | 12 语言 + 20 框架 + 多端检测 | 通常针对特定技术栈 |
| **证据标注** | 强制（file:line + 代码片段 + 证据类型） | outfitter 有置信度追踪，其他无 |
| **并发执行** | 8 个 Agent 三波派发 | 通常单线程 |
| **质量保证** | 交叉验证（4 项）+ 抽样验证 | 无 |
| **持久化记忆** | 生成到 docs/first/ | oimiragieo 写入 .claude/context/memory/ |
| **置信度追踪** | 无 | outfitter 有 0-5 级置信度条 |

**00-first 差异化优势**：
1. **全栈覆盖**：同时支持前后端、多端类型
2. **产物完整**：一个命令生成完整的项目认知文档集
3. **质量保证**：证据标注 + 交叉验证，防止 AI 幻觉
4. **企业级**：安全脱敏、降级策略、超时控制

**市场 Skill 可借鉴点**：
1. **outfitter/codebase-recon**：置信度追踪机制（0-5 级进度条）
2. **oimiragieo/project-onboarding**：Greenfield vs Brownfield 检测
3. **wshobson/architecture-patterns**：Clean/Hexagonal/DDD 架构模式知识库

# 03-spec / 04-design / 06-task / 07-code / 12-verify 深挖：问题点 / 修改点 / 验证点

> 目标：继续把主链路 5 个节点拆到可执行优化级别。
>
> 事实边界：
> - 只使用 `skills/spec-first/*/SKILL.md` 与 `src/core/**`、`src/cli/**` 里能核实的事实
> - 不把 `docs/first`、`docs/requirements`、`reports` 里的文字倒写成真源
> - 技能文档中出现、但代码注册表里没有的条件，单独标为“文档声明，代码未见对应注册”

## 0. 主链路骨架

```text
03-spec
  ├─ PRD 生成
  ├─ FR/AC 收敛
  ├─ 宪法一致性检查
  └─ document-links / findings / spec.md

04-design
  ├─ spec -> DS
  ├─ research 按需 companion
  ├─ 宪法权威检查
  └─ design.md / contracts / document-links / findings

06-task
  ├─ spec + design -> task_plan
  ├─ 任务依赖解析
  ├─ 并行语义 / 粒度控制
  └─ document-links / findings

07-code
  ├─ TASK 执行
  ├─ batch / checkpoint / report
  ├─ dependency-check + hard-gate
  └─ findings / batch-report / batch-checkpoint / task_plan

12-verify
  ├─ gate check
  ├─ docs links validate
  ├─ metrics report
  └─ findings / stage advance 建议
```

## 1. 03-spec

### 问题点

| 问题点 | 代码 / 文档事实 | 风险 |
|---|---|---|
| 03-spec 同时承担 PRD、FR/AC、宪法检查、文档关联、ID 注册，职责密度很高 | `skills/spec-first/03-spec/SKILL.md` 的 Phase 0 + Step 0-8 流程；`src/core/document-links.ts` 的 stage 校验；`src/core/gate-engine/condition-registry.ts` 的 G-SPEC 条件 | 入口太多，容易把“需求定义”做成“多轮综合处理” |
| C-PRD、C10、宪法一致性是三套不同的质量面 | `03-spec` 里明确 `C-PRD >= 85%`、C10、P1.5 宪法检查；gate 里有 `G-SPEC-00/01/02/03` | 容易出现“分数过了，但内容没收敛” |
| `document-links.yaml` 既是声明面，也是 gate 依赖面 | `validateStageDocumentLinks()` 在 `SPECIFY` 阶段要求 `spec.md`；`loadDocumentLinks()` 验证结构和引用 | 一旦链接缺失，后续 stage 会被连锁影响 |
| `findings.md` 既装发现又装过程记录 | `03-spec` 成功标准要求 `findings.md` 包含 Phase 0.0-0.6 + Step 0-8 记录 | 如果记录过长，后续 review / status 读起来很重 |

### 修改点

| 修改点 | 目标 |
|---|---|
| 把 `Phase 0` 与 `Step 0-8` 拆成两个固定区块展示 | 让读者先知道“先产 PRD，再收敛 FR/AC” |
| 将 `C-PRD / C10 / constitution` 做成并列健康卡 | 减少把不同维度混为一个通过/不通过 |
| 给 `document-links.yaml` 增加更显式的“声明 / 引用 / 缺失”提示 | 降低后续 gate 失败的排查成本 |
| 把 `findings.md` 记录粒度压缩成“结论 + 证据路径 + 下一步” | 降低噪音，保留可回放信息 |

### 验证点

| 验证点 | 通过标准 |
|---|---|
| 是否能区分 PRD、FR/AC、宪法、链接四种检查 | 每种都有独立检查结果 |
| 是否能看到 `spec.md` 与 `document-links.yaml` 的双向关系 | `spec.md` 不是孤立文件 |
| 是否能把 `findings.md` 当作过程证据而不是正文 | 记录可回放但不替代 spec |

## 2. 04-design

### 问题点

| 问题点 | 代码 / 文档事实 | 风险 |
|---|---|---|
| design 阶段对 `research` 的依赖是按需触发，但文档与 runtime 都很强调 | `04-design` 明确 companion contract；`05-research` 输出 `research.md` + `findings.md`；`design` 回流字段固定 | 如果没有清晰的触发条件，research 可能被滥用或漏用 |
| design 需要先过硬门槛，再做宪法检查 | `hard-gate.ts` 要求 `spec.md` 存在；`condition-registry.ts` 有 `G-DESIGN-01/02/03` | 很容易把“设计方案写出来”误判为“设计已可推进” |
| design 的输出还牵扯 contracts 和 document-links | `skills/spec-first/04-design/SKILL.md` 输出 `contracts/*.yaml`（按需）和 `document-links.yaml` | 设计文本与契约文件可能不同步 |
| 简洁性守卫明确禁止投机性层次 | `Simplicity First` 明确禁止为未来可能性提前加层 | 若不自觉，很容易把设计写得过宽 |

### 修改点

| 修改点 | 目标 |
|---|---|
| 把“何时必须 research”写成最小判定列表 | 减少触发歧义 |
| 把 `design.md` 的必写项和禁写项做成正反表 | 让设计输出更容易自检 |
| 将 `contracts/*.yaml` 标注为按需，而不是默认必产 | 降低无谓输出 |
| 让宪法检查结果和设计回流字段固定成同一个顺序 | 便于后续 review / verify 消费 |

### 验证点

| 验证点 | 通过标准 |
|---|---|
| 是否能说明这次 design 有没有触发 research | 有清晰触发依据 |
| 是否能看出设计是否引用了 spec | `design.md` 中可追溯 |
| 是否能区分“必要设计”与“投机性层次” | 投机层次不进入正式输出 |

## 3. 06-task

### 问题点

| 问题点 | 代码 / 文档事实 | 风险 |
|---|---|---|
| 任务拆解必须读取 spec + design + document-links，但任务计划常被写成纯文本列表 | `06-task` 明确 `task_plan.md` 需要 ID / owner / 工期 / 依赖 / 验收标准 / 验证命令；`parser.ts` 只解析表格型任务 | 一旦不按表格写，parser 会读不全 |
| canonical 状态与 legacy 状态混用 | `normalizeTaskPlanStatus()` 把 complete/completed/done/verified 归一；`toTaskNodes()` 再映射到 done | 容易出现 UI / 执行器 / 文档三边状态不一致 |
| `[P]` 并行语义依赖任务计划里明确标记 | `06-task` 文档定义 `[P]`，`orchestrate` 也依赖它判断并行风险 | 如果标记不统一，并行调度会失真 |
| 任务粒度控制是硬要求 | `2-4h` 任务、`5-30min` step；`When to Stop and Ask` 要求粒度模糊就问 | 粒度过粗会直接把后续实现拖爆 |

### 修改点

| 修改点 | 目标 |
|---|---|
| 把 `task_plan.md` 模板严格固定成 parser 友好的表格 | 避免解析失败 |
| 把 canonical / legacy 状态写成一张对照表 | 减少状态混用 |
| 将 `[P]` 并行标记单独写在任务标题或字段中 | 让 orchestrate 更容易消费 |
| 给每个任务强制补“验证命令”一栏 | 让 code 阶段有落点 |

### 验证点

| 验证点 | 通过标准 |
|---|---|
| parser 是否能读出所有任务 | `readTaskPlan()` 返回完整任务列表 |
| `task_plan.md` 是否能映射到 TaskNode | `toTaskNodes()` 无丢失字段 |
| 是否能从任务计划直接看出依赖拓扑 | depends_on 可机器读 |

## 4. 07-code

### 问题点

| 问题点 | 代码 / 文档事实 | 风险 |
|---|---|---|
| code 的现实能力和文档目标态存在差异 | `07-code/SKILL.md` 明确 batch executor 仍未完全接入真实 subagent 链路，失败率控制 / 超时裁剪 / 多层并发控制仍是目标态 | 容易把提示词里的“自动批量模式”误当真实 runtime 已完全落地 |
| 代码执行前置门槛很多 | `hard-gate.ts` 要求 `design.md`、`task_plan.md`、阶段匹配、worktree 高风险守卫 | 如果门槛没过，后续实现动作都不应发生 |
| `findings.md`、`batch-checkpoint.json`、`batch-report.md` 是共享执行痕迹 | `07-code` 明确主进程写共享文件，subagent 禁止直接写汇总 | 共享文件写错会污染后续恢复 / review |
| `findings.md` 只作执行记录，不是 code 门禁 | `07-code` 里写明 findings 不作为 code 进入门禁 | 很容易把执行记录当成验收证据 |

### 修改点

| 修改点 | 目标 |
|---|---|
| 把“提示词层的批量模式”与“runtime 已接线能力”写成两列 | 避免虚假能力感 |
| 给共享文件的写入边界增加显式表格 | 防止主进程 / subagent 越权写文件 |
| 把高风险守卫的触发条件与 remediation 统一成固定格式 | 便于人工接管 |
| 将 findings 与门禁结论分区写 | 不让执行记录遮蔽验收证据 |

### 验证点

| 验证点 | 通过标准 |
|---|---|
| 是否能区分“提示词层批量”与“真实 runtime 批量” | 两者不混写 |
| 是否能从 task_plan 直接推导 code 执行顺序 | 依赖解析可用 |
| 高风险变更是否会被 worktree 守卫拦住 | 保护分支 + 高风险条件时返回 BLOCKED |

## 5. 12-verify

### 问题点

| 问题点 | 代码 / 文档事实 | 风险 |
|---|---|---|
| verify 必须认新鲜证据，但人容易复用旧输出 | `12-verify/SKILL.md` 明确“五步 Gate Function：IDENTIFY -> RUN -> READ -> VERIFY -> ONLY THEN” | 如果不看新鲜命令输出，就会产生假通过 |
| verify 需要同时看 gate、docs links、metrics、TDD | `12-verify` 里明确 Gate 与 TDD 证据分离；`findings.md` 缺口要单列 | 很容易把门禁通过误当 TDD 闭环完整 |
| `G-VERIFY-02` 在 skill 文档存在，但当前内置 registry 未见对应项 | `condition-registry.ts` 只注册 `G-VERIFY-01`、`G-VERIFY-03`；`12-verify` 文档仍列出 `G-VERIFY-02` | 文档与代码口径不一致，容易误读 |
| verify 的输出既要给出 PASS/FAIL，也要给出下一步命令 | `12-verify` 文档要求 stage advance 建议 | 如果只报结果不报动作，闭环不完整 |

### 修改点

| 修改点 | 目标 |
|---|---|
| 把 gate 结果、docs 结果、TDD 结果拆成三个独立区块 | 防止混读 |
| 对 `G-VERIFY-02` 这类文档-代码不一致项做显式注记 | 避免把文档当代码 |
| 在输出中强制附带“建议下一步命令” | 让 verify 结果可执行 |
| 把 PASS_WITH_WAIVER 的豁免说明单独写清 | 避免通过状态被误解 |

### 验证点

| 验证点 | 通过标准 |
|---|---|
| 是否能确认当前验证证据是新鲜的 | 不是复用旧结果 |
| 是否能把门禁通过与 TDD 闭环区分开 | 两者独立展示 |
| 是否能识别 registry 与 skill 文档的不一致 | `G-VERIFY-02` 不被误当代码事实 |

## 6. 五个节点的联动问题

```text
03-spec 产出的 spec.md / document-links.yaml 是 04-design 的输入
04-design 产出的 design.md / contracts 是 06-task 的输入
06-task 产出的 task_plan.md 是 07-code 的输入
07-code 产出的 findings / checkpoint / report 是 12-verify 的输入
12-verify 的 PASS / WAIVER / FAIL 决定能否 stage advance
```

### 联动问题点

| 问题点 | 事实依据 | 风险 |
|---|---|---|
| 任何一层记录不规范，都会顺着链路放大 | spec-links、design-links、task_plan、findings、gate history 都是下游输入 | 小问题在下游变成门禁问题 |
| findings.md 被多节点共用 | `03-spec`、`04-design`、`06-task`、`07-code`、`12-verify` 都会写 findings | 容易互相覆盖语义 |
| 阶段推进依赖 gate 与 dependency-check 同时成立 | `advance.ts` 先 dependency-check 再 gate 校验 | 只修一个不够 |

### 联动修改点

| 修改点 | 目标 |
|---|---|
| 把各节点 findings 的写入语义做成前缀规范 | 区分 plan / design / execute / verify |
| 让 task_plan / design / spec 三者形成稳定引用链 | 降低断链 |
| 让 verify 输出同时包含 gate 结论和下一步命令 | 保持闭环 |

### 联动验证点

| 验证点 | 通过标准 |
|---|---|
| 是否能从 verify 追到 code，再追到 task，再追到 design/spec | 追溯链完整 |
| 是否能从 findings 判断记录属于哪个节点 | 前缀 / 区块清晰 |
| 是否能区分 dependency-check 失败和 gate 失败 | 两者分别可见 |

## 7. 这五个节点的最小优化顺序

1. 先收紧 `03-spec` 的输出边界，让 spec / links / findings 不再混乱
2. 再收紧 `04-design` 的 research 触发与 contracts 输出
3. 再把 `06-task` 任务表格式和状态归一做稳
4. 再把 `07-code` 的真实 runtime 能力与目标态分开写清
5. 最后让 `12-verify` 把新鲜证据、TDD、gate、下一步命令同时落盘

# init/spec 执行日志问题整改方案

**整理日期**: 2026-03-18  
**问题来源**: [2026-03-18-211127-command-messagespec-firstinitcommand-message.txt](/Users/kuang/xiaobu/spec-first/2026-03-18-211127-command-messagespec-firstinitcommand-message.txt)  
**目标**: 针对 `init -> spec -> stage advance -> skill render` 首轮体验链路中的卡点，形成一份可执行、可验收、可分阶段落地的整改方案。

---

## 1. 总体结论

这份日志暴露的不是单点 bug，而是首轮体验链路没有收口：

1. `init` 完成后的下一步建议与真实阶段要求冲突
2. 控制面命令契约不统一，AI 和用户都容易调用错误
3. `skill render / Context Pack` 输出形态不适合真实交互宿主
4. `docs/first` 当前更像结构索引，却被期待成高质量项目认知文档
5. `init` 阶段生成过多低价值模板，后续又要被 LLM 重写，造成来回折返

当前系统能跑通，但首轮体验不稳定，且会制造“看起来已完成、实际需要人工绕路”的假象。

## 2. 日志确认的问题

### 2.1 你已发现的问题

#### 问题 A：`docs/first` 的 md 内容过于精简，缺少有效认知信息

**结论**: 成立  

当前 `docs/first` 大量内容来自：
- [first-bootstrap.ts](/Users/kuang/xiaobu/spec-first/src/core/skill-runtime/first-bootstrap.ts)
- [first-doc-projection.ts](/Users/kuang/xiaobu/spec-first/src/core/skill-runtime/first-doc-projection.ts)

它不擅长：

因此当前 `docs/first` 适合作为“事实层”，不适合作为“最终解释层”。

- `spec-first skill render <skill>` 输出过长
- Context Pack 警告在宿主里呈现成错误样式
- Bash / Read / AskUserQuestion / 长 markdown 混杂在同一轮输出中
- 宿主界面折叠长输出时，用户容易误判为命令未结束

#### 问题 C：多个阶段执行命令因为缺少 `--yes` 报错

**结论**: 成立  

日志中明确出现：
- `stage advance` 因缺少 `--yes` 报错
- 后续重试又因为参数形式不对报错

这说明：
- 确认策略确实生效了
- 但帮助文案、skill 示例、AI 调用习惯还没完全对齐

## 3. 额外发现的问题

### 3.1 `init skill / AI 引导层` 的下一步建议是错的

日志里 `init` 完成后，AI 给出的下一步是直接执行 `/spec-first:spec`，但当前阶段仍是 `00_init`，而 `spec` skill 需要 `01_specify`。结果就是：

- 用户照着建议执行
- 立刻被 hard gate 拦截
- 然后再倒回去做 `gate check + stage advance`

这属于明显的流程断裂。

需要说明的是：
- [init.ts](/Users/kuang/xiaobu/spec-first/src/cli/commands/init.ts) 本身并没有直接输出 `/spec-first:spec`
- 问题出在 `01-init` skill 的引导口径和 AI 执行链路没有和真实阶段状态收口
- 因此这里应视为“skill/AI 引导问题”，不是单纯的 CLI 文案问题

涉及实现：
- [init.ts](/Users/kuang/xiaobu/spec-first/src/cli/commands/init.ts)
- [hard-gate.ts](/Users/kuang/xiaobu/spec-first/src/core/skill-runtime/hard-gate.ts)
- [stage.ts](/Users/kuang/xiaobu/spec-first/src/cli/commands/stage.ts)

### 3.2 控制面命令语法不统一

日志里连续出现三种错用：

- `--platform`，实际应为 `--platforms`
- `stage advance --feature <id>`，实际应为 `stage advance <featureId>`
- `strict` 策略命令未补 `--yes`

这说明当前控制面不够 agent-friendly，也不够 human-friendly。

### 3.3 `skill render` 默认输出过重

当前：
- [skill.ts](/Users/kuang/xiaobu/spec-first/src/cli/commands/skill.ts) 直接把完整渲染结果打印到终端

这会导致：
- 长文本滚屏
- 终端噪音很大
- 宿主上下文膨胀
- “卡顿感”被放大

### 3.4 `init` 的模板生成范围偏大

从日志看，`init` 阶段会立即生成：
- `prd.md`
- `task_plan.md`
- `impact-analysis.md`
- `findings.md`
- `traceability-matrix.md`
- `stage-state.json`

问题是：
- `prd.md` 很快会被 spec 重写
- `task_plan.md` 在任务阶段之前价值有限
- 模板太早生成，会制造“产物已完成”的错觉

### 3.5 spec 阶段默认假设过多

日志中生成的 PRD 自动补入了：
- SEO
- Lighthouse 指标
- 暗色模式
- CSP
- 部署平台假设
- 多个业务指标

这说明当前 spec 提示链在简单场景下补全过度，容易把合理默认项误写成需求。

## 4. 根因分析

### 4.1 阶段流转设计断裂

当前逻辑是：
- `init` 完成后 Feature 仍停留在 `00_init`
- 但界面提示直接去 `/spec-first:spec`
- `spec` 又要求 `01_specify`

这导致首轮链路天然需要一次额外绕路。

### 4.2 控制面契约没有形成单一事实来源

命令真实语法分散在：
- CLI help
- skill 文档
- references
- AI 输出示例
- router 的确认策略

它们目前没有完全一致，因此用户和 agent 都容易用错。

### 4.3 输出层没有为“交互宿主”做优化

当前输出更多是“便于调试”和“便于开发者看完整内容”，而不是“便于 Claude/Cursor/Gemini 这类宿主流畅消费”。

### 4.4 facts 层和 insights 层没有明确分离

当前 `docs/first` 同时承担：
- runtime truth 的事实投影
- 项目认知解释文档

但前者适合 CLI 生成，后者更适合 LLM 增强。这两个层没有分开，导致用户期待和实现能力不匹配。

## 5. 文档生成分层方案

### 5.1 哪些适合 CLI 脚本生成

适合脚本生成的内容应满足：
- 可枚举
- 可验证
- 可从仓库结构直接推断
- 歧义小

建议保留为 CLI truth/facts 层的内容：
- 模块拓扑
- 入口点
- 命令面 / API 面
- 外部依赖
- 测试/构建工具
- runtime truth index
- 可探测到的 domain/schema 骨架

对应文档：
- [docs/first/README.md](/Users/kuang/xiaobu/spec-first/docs/first/README.md)
- [docs/first/api-docs.md](/Users/kuang/xiaobu/spec-first/docs/first/api-docs.md)
- [docs/first/codebase-overview.md](/Users/kuang/xiaobu/spec-first/docs/first/codebase-overview.md)
- [docs/first/external-deps.md](/Users/kuang/xiaobu/spec-first/docs/first/external-deps.md)

### 5.2 哪些适合 LLM 生成或增强

适合 LLM 生成的内容应满足：
- 需要抽象和解释
- 需要权衡
- 需要把散乱事实组织成认知结构

建议改成 LLM-enhanced insight 层的内容：
- 项目摘要
- 架构总览
- steering
- critical flows 的解释
- entry guide
- conventions 的解释性说明

对应文档：
- [docs/first/summary.md](/Users/kuang/xiaobu/spec-first/docs/first/summary.md)
- [docs/first/architecture.md](/Users/kuang/xiaobu/spec-first/docs/first/architecture.md)
- [docs/first/steering.md](/Users/kuang/xiaobu/spec-first/docs/first/steering.md)
- [docs/first/critical-flows.md](/Users/kuang/xiaobu/spec-first/docs/first/critical-flows.md)
- [docs/first/entry-guide.md](/Users/kuang/xiaobu/spec-first/docs/first/entry-guide.md)
- [docs/first/conventions.md](/Users/kuang/xiaobu/spec-first/docs/first/conventions.md)

### 5.3 推荐模型

推荐采用两层模型：

1. `facts`  
CLI 生成，要求结构稳定、字段稳定、适合 diff

2. `insights`  
LLM 基于 facts 生成，要求可解释、可读、可辅助 agent 决策

原则：
- CLI 不要假装输出高质量认知结论
- LLM 不要直接改写 runtime truth

## 6. 解决方案

## 6.1 P0 必修项

### P0-1. 修复 `init -> spec` 阶段链路

目标：
- init 完成后，用户看到的下一步必须可直接执行

推荐方案：
- 方案 A：`init` 成功后自动推进到 `01_specify`
- 方案 B：`/spec-first:spec` 在 `00_init` 且 gate PASS 时自动先执行一次 `stage advance`

推荐：
- 优先方案 B  
原因：
  - 更保守
  - 不让 `init` 自动产生额外副作用
  - 把“进入 spec 阶段”的动作留在 spec skill 自己内部收口

可行性说明：
- 当前 `00_init` 阶段已经支持 `gate check PASS`
- 日志中已实际出现 `Gate 检查 — ... (00_init) -> PASS`
- 代码和测试也已覆盖 `00_init` 的 gate 评估路径

验收标准：
- `init` 后直接执行 `/spec-first:spec` 不再被 hard gate 拦截
- 不需要用户手工先跑 `gate check` 再 `stage advance`

### P0-2. 统一控制面命令契约

目标：
- 用户和 agent 按任一官方文档操作，都不会撞到参数陷阱

整改内容：
- 统一 `--platforms`
- 明确 `stage advance <featureId>`
- 明确 strict 命令都需要 `--yes`，其中当前日志已确认 `stage advance` 会经由 router 触发确认检查
- 所有帮助文案、skill 文档、references、错误提示统一

增强建议：
- 为高频命令增加 agent-friendly alias
  - 支持 `stage advance --feature <id>`
  - 兼容 `stage advance <id>`

验收标准：
- AI 再次按推荐路径执行时，不会在参数层面连续报错

### P0-3. 降噪 `Context Pack` 超限告警

目标：
- 超限要可见，但不能看起来像失败

整改内容：
- 不再输出高干扰中文错误风格
- 改为轻量 warning
- 默认只提示“已裁剪/已降级”
- 详细尺寸信息放 debug 模式

推荐输出：
- `WARN context pack too large, using compact view`

验收标准：
- 用户不会把它误解为命令失败
- agent 不会因该提示中断主流程判断

### P0-4. 缩短 `skill render` 默认输出

目标：
- 减少滚屏、减少假卡死、减少上下文污染

整改内容：
- 默认输出 summary 版：
  - 当前阶段
  - 关键前置条件
  - 当前 skill 的目标
  - 下一步动作
- 完整正文仅在 `--full` 或内部模式输出

验收标准：
- `spec-first skill render spec` 默认输出显著缩短
- 宿主界面不再频繁出现长折叠块

## 6.2 P1 应修项

### P1-1. 重构 `docs/first` 生成策略

目标：
- 让用户明确知道“哪些是事实层，哪些是解释层”

整改内容：
- 对 `docs/first` 增加分层说明
- 或直接拆为：
  - `docs/first/facts/*`
  - `docs/first/insights/*`
- 至少在文档头部写明：
  - 机械投影
  - LLM 增强摘要

验收标准：
- 用户不再把机械投影误当成高质量认知报告

### P1-2. 缩减 `init` 阶段模板生成

目标：
- init 只生成当前阶段真正需要的骨架

建议保留：
- `prd.md`
- `findings.md`
- `traceability-matrix.md`
- `stage-state.json`

建议延迟生成：
- `task_plan.md`
- 其他后阶段专属产物

验收标准：
- init 之后的工作区更干净
- 不再制造“任务计划已经准备好了”的假象

### P1-3. 收紧 spec 阶段的默认补全

目标：
- 简单需求不要自动灌入过多默认假设

整改内容：
- 收紧默认加项：
  - 暗色模式
  - SEO 详细指标
  - 分析平台
  - 部署平台
  - 泛化 KPI
- 仅在用户明确要求或命中特定规则时补入

验收标准：
- 生成的 PRD 更贴近用户输入
- `[ASSUMED]` 项减少

## 6.3 P2 优化项

### P2-1. 为控制面增加恢复型建议

例如在失败时显式给出：
- 缺 `--yes` 怎么补
- 当前阶段不对时怎么推进
- Context Pack 过大时已采取什么降级

### P2-2. 为首轮体验增加 compact mode

例如：
- `init` 默认简洁输出
- `--verbose` 时才打印完整过程

### P2-3. 为 `docs/first` 增加 LLM-enhance 刷新命令

比如：
- `spec-first first --facts`
- `spec-first first --insights`

让事实层和解释层可以分开刷新。

## 7. 实施顺序

建议按下面顺序做：

1. `P0-1` 修复 `init -> spec` 阶段链路
2. `P0-2` 统一控制面命令契约
3. `P0-3` 降噪 Context Pack 超限告警
4. `P0-4` 缩短 `skill render` 默认输出
5. `P1-2` 缩减 init 模板生成
6. `P1-3` 收紧 spec 默认补全
7. `P1-1` 重构 docs/first 生成分层
8. `P2-*` 做体验增强

原因：
- 前 4 项直接影响“能不能顺畅走完整条链路”
- 后 3 项才是“生成质量和认知质量”

## 8. 验收标准

### 8.1 首轮体验验收

目标链路：
- `/spec-first:init`
- 成功创建 Feature
- 直接进入 `/spec-first:spec`
- 无需人工绕路补 `gate check` / `stage advance`

验收标准：
- 用户按系统推荐路径可以一次走通
- 不再出现推荐下一步和真实阶段要求冲突

### 8.2 控制面验收

验收标准：
- `--platforms` / `stage advance` / `--yes` 的帮助、skill、错误提示一致
- agent 按官方示例执行不再连续撞参数错误

### 8.3 宿主输出验收

验收标准：
- `Context Pack` 超限提示不再表现为“像报错”
- `skill render` 默认输出长度明显下降
- 不再出现“像卡死一样”的长滚屏体验

### 8.4 文档质量验收

验收标准：
- `docs/first` 中 facts 与 insights 有明确边界
- 机械投影文档不再冒充高价值认知总结
- 简单需求生成的 PRD 默认假设显著减少

---

## 9. 推荐落地策略

本轮不建议一开始就追求“最聪明”。

推荐策略：

1. 先修流程断裂
2. 再统一控制面契约
3. 再压低输出噪音
4. 最后提升 first/docs 和 spec 的生成质量

也就是说，优先级应该是：

`先可顺畅执行，再可正确理解，最后再可高质量生成`

---

## 10. 一句话结论

这份日志反映的核心问题不是“某个命令坏了”，而是 `首轮上手链路没有被产品化收口`。  

必须优先解决的，是：
- `init -> spec` 阶段断裂
- 控制面参数/确认语法不一致
- `skill render / Context Pack` 输出不适合真实交互宿主

在此之后，再处理：
- `docs/first` 的 CLI/LLM 分层
- spec 生成内容过度补全

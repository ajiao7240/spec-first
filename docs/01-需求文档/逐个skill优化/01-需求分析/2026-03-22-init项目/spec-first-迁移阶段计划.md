# spec-first 迁移阶段计划

文档日期：2026-03-22
目标：把 `gstack` 完整迁入 `spec-first`，不改变任何功能

## 1. 迁移主线

迁移主线应该收敛成 4 个阶段，而不是把所有任务平铺处理。

```text
阶段 A：完整迁入
    ->
阶段 B：四个中枢改造
    ->
阶段 C：skill 逐个迁移与全仓收口
    ->
阶段 D：验证与定版
```

这 4 个阶段里，真正决定成败的是阶段 B。

更具体的执行原则应当是：

```text
先迁移整个框架
    ->
再逐个 skill 迁移
    ->
逐个测试验证
```

也就是说，不能一开始就把每个 skill 当作独立迁移单元。

正确顺序是：

1. 先把公共框架层迁过来
   - 安装
   - 生成
   - runtime
   - sidecar
   - 状态目录
2. 等框架层稳定后，再逐个 skill 收口
3. 每迁一个 skill，就做对应验证

这是后续整个执行过程的总原则。

## 2. 四阶段定义

### 阶段 A：完整迁入

目标：

- 先把 `gstack` 作为完整系统搬进来。

范围：

- 顶层文件
- skill 目录
- `browse/`
- `scripts/`
- `bin/`
- `test/`
- `supabase/`

不做：

- 不改名
- 不改路径
- 不改状态目录

阶段产物：

- 一个“已经在 `spec-first-pro` 仓库里，但内部仍叫 gstack”的中间态

阶段通过标准：

- 文件树完整
- 能识别出后续所有改造点

### 阶段 B：四个中枢改造

目标：

- 把系统真正从 `gstack` 切到 `spec-first`

必须改的四个中枢：

1. `setup`
2. `scripts/gen-skill-docs.ts`
3. `bin/gstack-*` helper
4. `browse` 路径与状态目录解析

这阶段要解决：

- 安装目录改名
- 状态目录改名
- helper 命令前缀改名
- agents 输出目录改名
- skill 生成路径改名

阶段产物：

- 一个“已能按 spec-first 路径安装、生成、运行”的核心系统

阶段通过标准：

- `setup` 跑通
- skill 可重新生成
- browse runtime 可运行

### 阶段 C：skill 逐个迁移与全仓收口

目标：

- 把中枢改造扩散到全仓，让所有文档、测试、生成物、模板一致。

范围：

- `SKILL.md.tmpl`
- 各 skill `SKILL.md`
- `.agents/skills/spec-first-*`
- `bin/*`
- `test/*`
- 顶层文档

阶段产物：

- 一个“名称、路径、生成物、测试”全部一致的 `spec-first`

阶段通过标准：

- 仓库内部不再存在未受控的 `gstack` 残留

这一阶段应明确采用：

```text
逐个 skill 迁移
    +
逐个 skill 验证
```

而不是一次性把所有 skill 文档全部改完再统一测。

原因：

- skill 数量多
- 生成链路复杂
- 某些 skill 依赖 review/qa/bin/sidecar 资产
- 一次性改完后很难定位是哪一个 skill 引入了断裂

所以阶段 C 的执行粒度应该是：

1. 选一个 skill
2. 改模板/路径/文案/运行时引用
3. 重新生成产物
4. 跑对应验证
5. 确认通过后再进入下一个 skill

### 阶段 D：验证与定版

目标：

- 证明迁移没有破坏功能，并决定兼容层保留范围。

范围：

- build
- setup
- 生成脚本
- 单测
- 关键 skill 路径
- 兼容层验证

阶段产物：

- 可发布的迁移版本

阶段通过标准：

- 新路径可用
- 老状态可兼容
- 文档可指导安装和升级

## 3. 阶段依赖关系

### 3.1 为什么不能先改文档

因为：

- 文档先改，只会让系统“看起来像 spec-first”
- 但安装、生成、运行时仍可能全部指向 `gstack`

所以文档必须排在中后段。

### 3.2 为什么必须先打通中枢

因为：

- `setup` 决定能否安装
- `gen-skill-docs.ts` 决定生成物是否可用
- `browse` 路径逻辑决定运行时是否真的活着

这四个点没通，其他都是表层工作。

## 4. 当前最合理的执行顺序

把现有文档收敛后，真正推荐的推进顺序如下：

1. `A1` 基线冻结
2. `A2` 原仓完整迁入
3. `B1` 改 `setup`
4. `B2` 改 `scripts/gen-skill-docs.ts`
5. `B4` 改 `bin/*` 命名前缀
6. `B3` 改 browse 状态目录与路径解析
7. `C1` 改 skill 模板
8. `C2` 重新生成产物
9. `C3` 改 agents sidecar
10. `C4` 改测试
11. `C5` 改顶层文档
12. `D1` 跑 build / test / setup 验证
13. `D2` 跑兼容层验证
14. `D3` 决定是否保留旧别名

如果进一步按你的思路收束，执行节奏应该理解成：

### 第一段：框架迁移

- `A1`
- `A2`
- `B1`
- `B2`
- `B3`

### 第二段：skill 逐个迁移

- 从阶段 C 开始，按 skill 为单位推进

### 第三段：skill 逐个验证

- 每完成一个 skill，就立即验证，而不是最后一次性总测

## 5. 每阶段的关键输出

### 阶段 A 输出

- 完整迁入代码树
- 基线记录
- 引用清单

### 阶段 B 输出

- `spec-first` 命名下可运行的安装/生成/运行时主干

### 阶段 C 输出

- 全仓命名与路径一致
- 生成物与测试同步

### 阶段 D 输出

- 可交付版本
- 兼容策略
- 升级说明

## 6. 风险优先级

迁移风险按优先级排序如下：

### P0

- `setup` 改完但安装目录与 sidecar 不一致
- `gen-skill-docs.ts` 改完但生成产物仍引用旧路径
- browse 运行时仍写入 `.gstack`

### P1

- `bin/*` 已改名但模板未更新
- `.agents/skills/spec-first-*` 与 sidecar 目录不一致
- 测试大量硬编码旧路径

### P2

- 文档残留旧名
- CHANGELOG 与 README 叙述不一致

## 7. 现在应该做什么

如果从当前状态继续推进，最正确的下一步不是继续写泛方案，而是开始阶段 A 与阶段 B 的执行准备：

1. 列出要迁入的目录和要忽略的目录
2. 列出四个中枢中的具体改造点
3. 把阶段 B 拆成可执行 patch 级别任务

## 8. 结论

迁移方案到这里已经足够清晰：

- 顶层上，项目应被视为一个完整系统，而不是一组 skill
- 执行上，必须按阶段推进，不能平铺改
- 成败上，关键只看四个中枢是否被打通

如果再进一步压缩成一句话，就是：

```text
先迁移整个框架，再逐个 skill 迁移，再逐个测试验证
```

接下来如果继续，最有价值的是直接进入：

`阶段 A 执行清单`

以及

`阶段 B 中枢改造清单`

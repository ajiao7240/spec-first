# Quality Agents 与多人协作上层系统边界说明

文档状态：Draft v1
文档日期：2026-03-22

## 1. 文档目标

本文档用于明确：

- `Quality Agents` 当前产品边界
- “多人协作下 AI 辅助开发质量提升”需求文档所代表的问题域
- 两者的关系与切分方式

## 2. 当前产品：Quality Agents

Quality Agents 当前被定义为：

- 独立产品
- 前置质量判断系统
- 核心问题：需求/方案跑偏
- 核心资产：Quality Memory

其核心能力聚焦在：

- `/clarify`
- `/challenge`
- `/scope-lock`
- `/design-review`

它的目标是：

- 更早把问题说清楚
- 更早识别错误 framing
- 更早锁定范围边界
- 更早暴露设计缺口

## 3. 上层问题域：多人协作 AI 研发控制系统

文档《多人协作下 AI 辅助开发质量提升需求文档》描述的是一个更高一层的问题域：

- 多人协作
- 多端协同
- 多 owner
- 多 git 工程
- 协议一致性
- 联调协作
- 变更重算
- 发布反馈

这类问题的本质不是“前置质量判断”，而是“团队协作与交付控制”。

## 4. 两者关系

两者不是同一个产品的当前边界。

更准确地说：

- `Quality Agents` 是一个更小、更硬的前置质量判断产品
- 多人协作控制系统是未来可能服务的更大场景，或未来另一层产品

关系示意：

```text
多人协作 AI 研发控制系统
    └── 可消费 / 接入 Quality Agents
            ├── clarify
            ├── challenge
            ├── scope-lock
            └── design-review
```

## 5. 为什么必须切分

如果现在把两者直接合并，会立即出现以下问题：

- 产品目标混杂
- MVP 范围失控
- 质量判断系统重新长成流程控制平面
- 价值验证变慢

切分的目的不是否认上层系统价值，而是保证当前产品先证明自己成立。

## 6. 当前结论

当前正式结论是：

- `Quality Agents` 不并入多人协作全链路控制系统边界
- 多人协作需求文档被视为更高层场景需求
- 当前产品先把“前置质量判断”做成

## 7. 后续演进建议

后续若 Quality Agents 被证明成立，再考虑：

1. 将其作为多人协作系统中的前置质量引擎
2. 为多人协作场景增加协议、一致性、联调、变更重算等上层能力
3. 重新评估是否需要新产品层或新产品线

# 第二期：飞书 -> 需求工程 -> Git 仓库 -> DevOps 发布集成设计

文档日期：2026-03-22
文档类型：第二期集成设计文档
适用场景：
- 公司内多人协作研发
- 飞书管理需求
- GitLab / GitHub 承载代码
- 自研 DevOps 系统承载提测、发布、环境和上线

## 1. 目标

第二期不再继续扩展需求工程内核，而是在现有企业系统之上补齐系统接缝：

```text
飞书
  -> 需求工程
  -> Git 仓库
  -> DevOps 发布
```

目标是让需求、方案、代码、发布形成一条可追踪链路，而不是散落在多个系统里各自独立存在。

## 2. 一句话定位

```text
第一期做需求工程内核
第二期做企业系统接缝
```

第一期回答：

- 需求怎么分流
- 怎么拆到各端
- 怎么落到工程
- 怎么形成方案、协议、任务包

第二期回答：

- 飞书里的需求如何进入需求工程主线
- Git 仓库如何和需求、任务、方案关联
- DevOps 发布如何接住提测和发布材料
- 发布结果如何回写并形成完整回溯

## 3. 总体设计原则

### 3.1 不替换现有系统

第二期不是重建一套需求系统、代码系统或发布系统，而是：

```text
在 飞书 + GitLab/GitHub + 自研 DevOps 之上
增加统一的需求工程主线
```

### 3.2 Source of Truth 分层

每类信息应有明确主来源：

- 需求主来源：飞书
- 代码主来源：GitLab / GitHub
- 发布主来源：DevOps
- 需求工程主来源：本方案产出的需求/方案/协议/任务文档链

### 3.3 链路可回溯

必须能做到：

- 从飞书需求找到相关 repo、任务、MR/PR、发布单
- 从 MR/PR 找到对应需求、方案、协议
- 从 DevOps 发布单回溯到需求和代码变更

### 3.4 先关联，后自动化

第二期落地顺序应是：

```text
先把字段和链接打通
再逐步做自动同步和自动回写
```

不要一开始就追求全自动集成。

## 4. 目标链路

```text
飞书需求
   |
   v
需求工程分流
   |
   +--> 新项目
   +--> 增量需求
   +--> 混合型需求
   |
   v
需求/方案/协议/任务文档链
   |
   v
GitLab / GitHub 仓库与 MR/PR
   |
   v
DevOps 提测 / 发布 / 上线
   |
   v
发布结果回写
   |
   v
飞书与需求工程链路闭环
```

## 5. 分模块设计

### 5.1 飞书接入

目标：

- 让飞书成为需求入口
- 让需求工程有稳定的源需求链接

建议承接的信息：

- 飞书需求 ID
- 需求标题
- 需求链接
- 需求负责人
- 优先级
- 评审结论
- 需求状态

需求工程侧需要生成或维护：

- 需求类型：`new_project / incremental / hybrid`
- 交付模式：`multi-team-by-end / fullstack-owner`
- 各端需求文档链接
- 方案文档链接
- 协议文档链接
- 任务文档链接

#### 飞书接入最小目标

```text
从飞书需求进入后
能生成一份需求工程工作项
并保留双向链接
```

### 5.2 Git 仓库接入

目标：

- 让需求工程结果落到具体代码仓库
- 让 MR/PR 与需求、任务、方案关联

建议承接的信息：

- repo 列表
- base branch
- owner
- MR/PR 链接
- diff 范围
- 评审状态

需求工程侧需要输出：

- `/focus-requirements` 结果
- handoff 文档结果
- 当前 owner 的 workspace 上下文

关键要求：

- 一个需求可关联多个 repo
- 一个任务包可关联一个或多个 MR/PR
- MR/PR 标题或描述中应带需求 ID / 任务 ID

### 5.3 DevOps 接入

目标：

- 让提测和发布承接需求工程产物
- 让发布系统不只是“拿代码发”，而是“拿有上下文的需求包发”

建议承接的信息：

- 提测单 / 发布单 ID
- 环境信息
- 发布批次
- 发布范围
- 发布状态
- 回滚状态

需求工程侧需要输出：

- `generate-self-test-checklist`
- `generate-integration-test-checklist`
- `generate-acceptance-pack`
- `summarize-release-impact`

关键要求：

- 提测单可挂验收包
- 发布单可挂需求 ID、任务包、MR/PR
- 发布完成后回写发布结果摘要

### 5.4 回写与闭环

目标：

- 让需求、代码、发布彼此可追踪

最小闭环要求：

- 飞书需求页能看到：
  - 关联方案
  - 关联任务
  - 关联 repo / MR/PR
  - 关联发布单
- 发布单能回溯：
  - 对应哪个飞书需求
  - 哪些 repo 改动
  - 哪些任务包进入本次发布

## 6. 第二期建议输出产物

- 飞书字段映射表
- 需求工程文档 ID 规则

## 8. 第二期范围边界

第二期建议只做“接缝打通”，不建议一开始就做：

- 全量自动化审批流
- 全量自动生成所有飞书文档
- 强绑定某一家 Git 平台能力
- 重写现有 DevOps 逻辑

更合理的方式是：

```text
先打通标识和链接
再打通字段和状态
最后再做自动同步和自动回写
```

## 9. 推荐分阶段落地

### 9.1 第二期-A：关联打通

- 飞书需求 ID 进入需求工程文档
- 需求工程文档关联 repo
- MR/PR 关联需求 ID
- 发布单关联需求 ID 和 MR/PR

### 9.2 第二期-B：字段打通

- 飞书字段和需求工程状态映射
- 任务包和 MR/PR 状态映射
- 验收包和提测单字段映射

### 9.3 第二期-C：自动回写

- 发布结果回写飞书
- 需求工程文档自动更新发布影响总结
- 支持按需求查看完整交付轨迹

## 10. 一句话总结

```text
第二期不是做新平台
而是把飞书、GitLab/GitHub、DevOps 接进同一条需求工程主线
```

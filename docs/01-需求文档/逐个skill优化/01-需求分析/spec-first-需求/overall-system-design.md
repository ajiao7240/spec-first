## 1. 一句话定义

这个系统的核心设计思路是：

```text
把“产品需求到工程上下文”从口头协作
重构成
可分流、可产物化、可追踪、可交接的需求工程系统
```

它不是单纯的“AI 写代码流程”，而是一个：

```text
需求工程系统
+ 多端协作前半链路系统
+ 文档驱动系统
+ handoff 系统
+ LLM 辅助判断系统
```

## 1.1 当前分期边界

当前文档体系按两期理解：

```text
一期:
  聚焦需求工程内核

二期:
  聚焦企业系统接缝
  飞书 -> 需求工程 -> Git 仓库 -> DevOps 发布
```

一期范围内重点解决：

- 需求分流
- 交付模式判断
- 多端拆分
- 混合型需求处理
- 工程落位
- 工程影响面识别
- workspace 上下文组装
- 当前 owner 需求聚焦
- 基础质量增强
- handoff 文档生成
- 固定 gstack 接手点约定

二期范围内重点解决：

- 飞书接入
- GitLab / GitHub 关联
- DevOps 提测与发布接入
- 外部系统状态回写

因此当前主线应理解为：

```text
先把需求工程内核跑通
把产品需求稳定转换成工程上下文
再交给 gstack 承接后续研发流程
最后再把企业现有系统接进来
```

其中，一期的具体范围、流程和 skill 落位，应结合：

- `new-project-vs-incremental-demand-flow.md`
- `phase-1-complete-solution.md`
- `phase-1-p0-skills-spec.md`

共同理解。

## 2. 系统要解决什么问题

这个系统主要解决公司研发里最常见的几类断层：

- 产品需求很大，但研发不知道会改哪些工程
- 多端协作时，前后端理解不一致
- 后端被前端页面牵着拆，能力域被打碎
- 需求拆分后，文档、方案、任务互相漂移
- 多人并行开发时，不知道谁依赖谁、谁阻塞谁
- 需求变化后，前面的方案和任务不会自动重算
- 提测、验收、发版之后，缺少完整沉淀

所以系统目标不是：

```text
让某个人更快写代码
```

而是：

```text
让需求从产品语言
先稳定变成本端可执行的工程上下文
并在交接前留下结构化产物
```

## 3. 总体设计原则

## 3.1 文档驱动，不靠聊天驱动

核心原则：

```text
需求不是聊天记录
方案不是会议结论
任务不是口头分配
```

系统要求将关键过程沉淀成结构化文档：

- 总需求文档
- 本端需求文档
- 工程影响分析文档
- workspace 上下文文档
- handoff summary

也就是说：

```text
系统以“文档链”驱动
不是以“对话上下文”驱动
```

## 3.2 分阶段，而不是从需求直接跳到开发

一期系统把主流程切成明确阶段：

```text
需求分流阶段
-> 工程落位阶段
-> workspace 组装阶段
-> 本端需求拆分阶段
-> 基础质量增强阶段
-> handoff 阶段
```

每个阶段只回答一类问题：

- 需求分流阶段：这是什么类型的需求
- 工程落位阶段：本端会影响哪些工程
- workspace 组装阶段：执行上下文是什么
- 本端需求拆分阶段：我到底负责什么
- 基础质量增强阶段：问题是否说清、是否跑偏、范围是否稳定
- handoff 阶段：从哪里接入 gstack

这本质上是在做：

```text
认知降维
```

让一个节点不需要同时解决所有问题。

## 3.3 多端协作靠“接口契约”，不是靠共享所有代码上下文

多人分端模式下：

- App 有自己的 workspace
- H5 有自己的 workspace
- Backend 有自己的 workspace

大家不是共享一个超大目录，而是共享这些协作产物：

- 总需求文档
- 本端 / 本模块需求文档
- 工程影响分析
- workspace 上下文
- handoff bundle

所以系统里的关键抽象是：

```text
workspace = 本地工程上下文
handoff bundle = gstack 接手输入
```

## 3.4 后端按能力域拆，不按前端页面拆

产品层可以按载体拆：

- App
- H5
- 后台

但技术层不能让后端跟着前端页面拆。

正确原则是：

```text
产品按载体拆
技术按职责拆
后端按能力域拆
联调按契约拆
```

这决定了：

- 后端是共享能力提供者
- 不是某个前端页面的附属实现层

## 3.5 新项目和增量需求必须分流

系统不使用一条流程硬套全部场景，而是先做分流判断。

分界点不是：

```text
项目新不新
```

而是：

```text
是否已经存在稳定的工程承载体
```

工程承载体包括：

- repo 是否已存在
- 端边界是否已存在
- owner 是否明确
- 接口风格是否稳定
- 服务边界是否稳定

因此系统入口先判断：

```text
新项目 / 新业务
还是
存量项目 / 增量需求
```

## 3.6 同时适配两种交付模式

系统显式支持两种模式：

```text
1. 多人分端
2. 全栈整包
```

区别：

- 多人分端：按端或模块 owner 协作
- 全栈整包：一个 fullstack owner 对自己负责范围内的 App/H5/Backend 整体负责

但不管哪种模式，一期都至少保留：

- 本端 / 本包需求文档
- 工程影响分析
- workspace 上下文
- handoff summary

也就是说：

```text
交付模式可以不同
工程上下文不能缺失
```

## 3.7 实施策略：先一期主链，后扩展能力

当前阶段更合适的实现策略不是先做完整角色编排，也不是先把方案/协议/任务主链全部实现，而是：

```text
先把一期主链跑通
再扩展二期与深度能力
```

原因是：

- 当前最关键的问题不是“方案写得不够早”，而是“研发拿到需求后缺少稳定工作起点”
- 如果一期同时承担方案、协议、任务主链，系统会重新变重
- 先跑通 owner 级 handoff，最容易验证真实价值
- gstack 已经能承接下游 CEO / Eng review 与实现链路

因此建议分两阶段推进。

第一阶段：

```text
一期主链先行
```

目标：

- 先固化需求到工程上下文的转换链
- 先固化“手动准备 workspace 后”的需求收口链
- 先验证 gstack 是否能稳定接手

优先实现的能力包括：

- `/focus-requirements`
- 固定 handoff 文档
- `backend-domain-split`
- `clarify`
- `challenge`
- `scope-lock`

其中：

- `明确工程列表`
- `手动准备 workspace`

属于研发已完成的人工前置条件，不再作为当前一期核心 P0 skill。

第二阶段：

```text
在一期主链稳定后
再扩展方案、协议、任务与系统接缝能力
```

这一阶段再去解决：

- 更重的方案/协议/任务主链
- feature.json 执行追踪层
- Judge eval 共享仓库
- 多人共享产物方案
- 企业系统接缝

这样做的原则是：

```text
先固化“会什么”
再固化“怎么串起来”
```

也就是说：

- 现在保留流程抽象
- 但实现上优先做 skill
- 暂不把上层角色编排当成第一阶段交付目标

## 9. 系统为什么是工程化的

它具备典型工程系统特征：

### 9.1 可追踪

每一步都有输入和输出文档。

### 9.2 可重跑

接口协议变了，只需重跑协议和任务相关节点，不必全部重来。

### 9.3 可分工

不同团队、不同 owner 可以在不同阶段接入。

### 9.4 可治理

支持状态同步、一致性检查、变更影响分析。

### 9.5 可适配组织结构

既支持多人分端，也支持全栈整包。

### 9.6 可与 LLM 结合

LLM 不再直接“从需求跳代码”，而是在每个阶段处理更明确的问题。

## 10. 最终总结

把这套系统压缩成一句话：

```text
这是一个“文档驱动 + 阶段驱动 + skill 驱动”的需求工程系统
```

再展开一点：

```text
先把需求交付过程拆成阶段
再把每个阶段产物化
再把每个节点 skill 化
最后让多人分端、全栈整包、多端联动都能在同一套工程约束下运行
```

# 多人协作下 AI 辅助开发质量提升需求文档

文档日期：2026-03-22
文档类型：需求文档
适用场景：
- 公司内多人协作研发
- 多端项目：App / H5 / Backend / 后台
- 一个需求涉及多个模块、多个 owner、多个 git 工程
- 同时覆盖新项目 / 新业务建设，与存量项目增量迭代
- 同时适配多人分端与全栈整包两种交付模式

## 1. 文档目标

本文档定义一套“多人协作下 AI 辅助开发质量提升”的正式需求，目标不是单纯提升个人编码速度，而是让 AI 进入团队研发主流程，提升需求正确性、方案正确性、跨端协作一致性、需求变更可控性，以及测试和发布稳定性。

## 2. 背景与问题

在公司多人协作研发场景中，交付质量问题通常不只发生在编码阶段，而是沿着整条协作链路扩散：

- 需求评审完成后，研发仍不清楚具体会改哪些工程
- 同一需求下，各端对范围、边界、依赖理解不一致
- 后端容易被前端页面驱动拆碎，能力域失稳
- 技术方案、接口协议和任务文档形成得太晚
- 联调阻塞经常在开发后期才暴露
- 需求变化后，前面产出的方案、任务、测试清单难以同步重算
- 验收、提测、发布反馈缺少结构化沉淀

当前很多团队引入 AI 后，主要还是把 AI 当成“个人编码助手”。这种方式可以提升局部产出速度，但并不能天然提升团队级质量，甚至可能放大多人协作中的理解偏差和协议漂移。

因此，需要建设一套以文档、阶段、状态和产物为核心的 AI 辅助需求工程体系，让 AI 作为“团队流程助手”进入主链路。

## 3. 建设目标

本系统需要达成以下目标：

1. 让 AI 从“个人编码助手”升级为“团队研发流程助手”。
2. 让需求从产品语言逐层转化为工程语言，并在每一层形成结构化产物。
3. 在需求、方案、协议、任务、测试、发布反馈各阶段引入 AI 辅助，降低协作失真。
4. 让多人分端与全栈整包两种交付模式都能在同一套工程约束下运行。
5. 让需求变化后，影响面、任务包、验收清单可快速重算，而不是依赖人工重新梳理。

## 4. 需求范围

### 4.1 范围内

本需求覆盖以下能力：

- 需求识别与分流
- 总需求到本端需求的拆分
- 后端按能力域拆分
- 受影响工程识别
- workspace 组织
- 技术方案生成与评审辅助
- 接口协议草案与一致性检查
- 任务包生成与状态跟踪
- 自测 / 联调 / 验收清单生成
- 需求变更影响分析
- 发布反馈沉淀

### 4.2 范围外

本需求不直接覆盖以下内容：

- 具体业务代码实现细节
- IDE 级代码补全产品设计
- 单一模型或单一厂商绑定方案
- 代码仓库权限治理本身
- 测试平台、CI 平台、发布平台的实现细节

## 5. 核心设计原则

### 5.1 AI 不只辅助编码

AI 必须覆盖整条链路，而不只停留在“写代码”阶段。

### 5.2 文档驱动

需求、方案、协议、任务、验收必须文档化，不能只依赖聊天记录、口头同步或会议纪要。

### 5.3 阶段驱动

流程必须拆成清晰阶段，每个阶段只解决一类问题。

### 5.4 多人协作优先

系统首先服务多人协作和多端协同场景，再兼容全栈整包模式。

### 5.5 后端按能力域拆

后端不跟着前端页面拆，而按共享能力域设计和演进。

### 5.6 接口契约作为协作边界

多人分端场景下，各端共享的是需求、方案、协议和联调产物，而不是一个超大代码上下文。

## 6. 目标流程

### 6.1 总流程

```text
需求评审完成
   |
   v
需求识别与分流
   |
   +--> 新项目 / 新业务
   +--> 存量项目 / 增量需求
   |
   v
交付模式判断
   |
   +--> 多人分端
   +--> 全栈整包
   |
   v
需求拆分
   |
   v
技术方案
   |
   v
接口协议
   |
   v
技术评审
   |
   v
任务拆分
   |
   v
开发 / 自测 / 联调 / 提测
   |
   v
发布反馈
```

### 6.2 新项目与增量需求的差异

新项目不能直接从“手动准备 workspace 后进入需求聚焦”开始，因为工程承载体本身可能尚未稳定。新项目在进入各端 workspace 之前，需要先完成：

- 跨端职责划分
- 工程拓扑设计
- 仓库 / 服务边界设计
- 接口边界设计

增量需求则可以在已有 repo、owner、接口风格和服务边界基础上，进入“手动准备 workspace -> 本端需求范围与需求聚焦 PRD -> handoff 给 gstack”的链路。

### 6.3 多人分端与全栈整包的差异

- 多人分端：App、H5、Backend 各自维护本端 workspace，通过接口协议和方案文档协作。
- 全栈整包：一个 fullstack owner 对自己负责范围内的 App/H5/Backend 整体负责，但仍保留需求文档、方案文档、协议文档和任务拆分。

## 7. AI 在流程中的职责

说明：

- 本节描述的是系统完整形态下的长期职责范围
- 不等于当前一期立即实现范围
- 当前一期以“手动准备 workspace 后，输出需求聚焦 PRD 和 handoff”为准

AI 在长期完整形态下应辅助以下节点：

- 判断需求类型：新项目还是增量需求
- 判断交付模式：多人分端还是全栈整包
- 辅助研发聚焦自己负责范围内的需求文档
- 将后端需求按能力域拆分
- 在 handoff 后辅助 gstack 完成方案收口
- 生成接口协议草案和兼容性说明
- 检查前后端协议一致性
- 辅助跨端技术方案评审
- 生成任务包和开发顺序
- 生成自测、联调、验收清单
- 跟踪状态、识别阻塞与依赖
- 分析需求变更影响并重算任务
- 生成发布影响总结

## 8. 功能需求

### 8.1 需求阶段

系统应支持：

1. 判断当前需求属于新项目 / 新业务，还是存量项目 / 增量需求。
2. 判断当前交付模式属于多人分端，还是全栈整包。
3. 将总产品需求拆分为各端需求文档。
4. 将后端需求按能力域拆分，而不是按前端载体拆分。
5. 识别需求涉及的用户场景、验收标准和约束条件。

### 8.2 方案阶段

系统应支持：

1. 识别本次需求涉及哪些 git 工程。
2. 区分 direct change、dependency review、integration only 三类工程。
3. 组装各端或全栈 workspace。
4. 基于 workspace 和本端需求文档生成本端技术方案。
5. 生成跨端依赖分析、风险清单和改动点清单。

### 8.3 协议阶段

系统应支持：

1. 生成接口协议草案。
2. 输出字段、状态码、兼容性和错误处理约束。
3. 检查前后端协议一致性。
4. 输出协议冲突、待确认项和缺失项。

### 8.4 评审阶段

系统应支持：

1. 组织跨端技术方案评审辅助。
2. 复核工程影响面、依赖关系和边界定义。
3. 输出评审结论、待补充项和阻塞项。
4. 在评审通过前，不允许进入正式开发拆分。

### 8.5 开发阶段

系统应支持：

1. 生成各端任务包或全栈整包任务包。
2. 规划开发先后顺序和并行关系。
3. 跟踪任务执行状态。
4. 检测阻塞与依赖问题。
5. 同步需求、方案、协议、任务文档状态。

### 8.6 变更控制阶段

系统应支持：

1. 当需求变化时，自动分析影响哪些端、哪些工程、哪些协议和哪些任务。
2. 基于变更结果重算相关任务包、测试清单和验收材料。
3. 标识受影响的文档并提示同步修正。

### 8.7 测试与验收阶段

系统应支持：

1. 生成自测清单。
2. 生成联调清单。
3. 生成验收包。
4. 输出验收路径、环境说明、风险说明和已知限制。

### 8.8 发布反馈阶段

系统应支持：

1. 生成发布影响总结。
2. 总结本次需求最终影响的端、能力域、接口和风险关闭情况。
3. 为后续复盘和增量迭代提供结构化输入。

## 9. 非功能需求

### 9.1 可追踪

每个阶段必须有明确输入、输出和状态，不允许只有口头结论。

### 9.2 可重跑

协议、任务、影响分析、清单生成等节点应可单独重跑，不要求全链路重来。

### 9.3 可扩展

系统必须同时支持：

- 多人分端模式
- 全栈整包模式
- 新项目 / 新业务
- 存量项目 / 增量需求

### 9.4 可治理

系统必须支持：

- 文档状态同步
- 文档一致性检查
- 任务状态跟踪
- 阻塞识别
- 需求变更影响分析

### 9.5 可集成

系统应能与现有 git 工程、文档系统、测试流程、发布流程协作，而不是要求团队替换整套工程环境。

## 10. 关键产物要求

系统至少应输出以下产物：

- 总需求文档
- App 需求文档
- H5 需求文档
- Backend 能力域文档
- 技术方案文档
- 接口协议文档
- 任务文档
- 状态跟踪表
- 自测清单
- 联调清单
- 验收包
- 发布影响总结

## 11. 成功标准

如果该系统落地成功，应在团队层面看到以下变化：

1. 需求拆分更稳定，跨端范围偏差减少。
2. 技术方案成稿更快，方案补漏次数下降。
3. 前后端协议冲突在开发前暴露，而不是联调后暴露。
4. 联调阻塞更早被发现，阻塞定位时间缩短。
5. 需求变化后的返工成本下降。
6. 提测、验收、发布材料更完整，复盘输入更清晰。

## 12. 不做什么

本系统不追求以下目标：

- 让 AI 代替研发做全部技术决策
- 让多人协作退化成“每个人各自和 AI 聊天”
- 通过增加文档数量替代工程判断
- 让后端围绕前端页面做被动拆分

## 13. 一句话总结

```text
多人协作下，AI 的价值不应只停留在“帮个人写代码”
而应升级为“辅助整个团队流程提升质量”
```

# spec-first 最小保留流程图

> 目标：在不丢失高质量交付边界的前提下，把 `spec-first` 的运行模型压缩到最小可保留集。
>
> 结论先行：可以减少 `ID` 关联层级，但不能删除 `stage gate`、`缺陷回流` 和 `RFC 豁免` 这三个质量闭环；它们只能被收敛，不能被抹掉。

## 1. 最小保留原则

当前代码和文档已经把系统拆成三层：

1. `skill 流程层`
2. `交付产物层`
3. `质量控制层`

最小保留方案只要求：

- `skill 流程层` 记录“发生了什么”
- `交付产物层` 记录“改了什么”
- `质量控制层` 决定“能不能推进”

也就是说，`skill` 节点可以成为主视图，但它不能替代产物 ID。

## 2. 最小保留流程图

┌──────────────────────────────┐

│  TASK / TC / RFC / Defect    │

└──────────────┬───────────────┘

               │ 只保留必要关联
               │

## 3. 三条最小闭环

### 3.1 主交付闭环

```text
Feature -> PRD / requirements -> REQ
REQ -> spec.md / requirements convergence -> FR
FR -> DS -> TASK
```

这条链是最小的交付闭环。

- `Feature` 是容器
- `PRD / requirements` 阶段只生成 `REQ`
- `REQ` 是 PRD 需求陈述
- `FR` 是 `spec.md` 收敛后的功能需求
- `DS` 是设计约束
- `TASK` 是执行切片

这条链必须保留，但不要求每一层都在执行层被完整展开。

### 3.2 缺陷回流闭环

```text
Defect -> linkedFr / linkedTc -> 回流到 FR / TC
```

缺陷记录不是“额外负担”，而是质量闭环的一部分。

- `linkedFr` 负责把缺陷归因到需求
- `linkedTc` 负责把缺陷归因到测试
- `status` 负责缺陷状态流转

如果去掉缺陷回流，只剩下“有问题”但没有“问题属于谁、修到哪一步”的信息。

### 3.3 RFC 豁免闭环

```text
RFC -> approved -> known-exceptions.md -> EX
```

RFC 豁免只用于“明确允许偏离”的情况。

- `RFC` 负责提出例外
- `approved` 负责确认例外可接受
- `known-exceptions.md` 负责留下审计痕迹
- `EX` 负责在质量控制层被识别

如果去掉 RFC 豁免，系统会更严格，但任何临时例外都只能靠人肉沟通，后续无法审计。

## 4. 可以删掉什么，不能删掉什么

### 4.1 可以删掉的复杂度

- 不需要把所有 `ID` 当成同等级主链
- 不需要把 `TC`、`RFC`、`Defect`、`EX` 全部拉进执行层
- 不需要把 `relationship-graph` 变成唯一消费接口
- 不需要让 `skill` 节点和产物 ID 一一强绑定

### 4.2 不能删掉的能力

- 不能删 `stage-gate`
- 不能删 `hard-gate`
- 不能删 `defect` 回流
- 不能删 `RFC` 豁免
- 不能删 `document-links` 的阶段校验

## 5. 最小职责划分

| 层级 | 最小职责 | 是否必须带 ID |
|---|---|---|
| Skill 流程节点 | 记录执行轨迹、阶段推进、审计日志 | 否 |
| 交付产物层 | 记录需求、设计、任务、测试、变更、缺陷 | 是 |
| 质量控制层 | 判断能否推进、能否豁免、能否回流 | 是 |

## 6. 典型场景

### 6.1 新增需求

最小路径：

```text
skill 节点 -> Feature -> PRD / requirements -> REQ -> spec.md 收敛 -> FR -> DS -> TASK -> stage-gate
```

如果缺少阶段产物，门禁失败，流程会被卡住，但不会“崩溃”。

### 6.2 新增缺陷

最小路径：

```text
skill 节点 -> Defect -> linkedFr / linkedTc -> 状态流转 -> 回归验证
```

缺陷可以独立存在，不要求先把它塞回主需求链。

### 6.3 非需求内代码变更

最小路径：

```text
skill 节点 -> RFC / Defect / TASK 中其一 -> gate 校验 -> 推进或阻断
```

如果是明确偏离，走 RFC 豁免；如果是质量问题，走 Defect；如果只是实现切片，留在 TASK。

#### 判定表

| 变更类型 | 最小落点 | 说明 |
|---|---|---|
| 当前 `TASK` 内的实现细化 | `TASK` | 不改变 `REQ / FR / DS` 语义，只推进既有任务 |
| 需求外的新能力、新行为、新接口 | `RFC` -> 评审 -> 可能回写 `REQ / FR` | 这是范围变化，不应伪装成普通实现 |
| 发现缺陷、回归失败、线上异常 | `Defect` | 缺陷必须可回流到 `FR / TC`，不能只写在讨论里 |
| 已批准的临时偏离 | `RFC` -> `known-exceptions.md` -> `EX` | 例外要可审计、可过期、可回滚 |
| 纯流程审计、执行轨迹记录 | `skill` 节点 | 不需要挂到具体需求 ID，但不能替代产物层 |

#### 选择原则

- **优先 TASK**：只要不改变 `REQ / FR / DS` 语义，就尽量留在当前任务内收敛
- **优先 Defect**：只要是“出问题了”，先把质量问题独立出来，不要直接改成需求讨论
- **优先 RFC**：只要涉及范围、接口、责任边界或临时偏离，就不要直接塞回 TASK
- **优先 EX**：只有 RFC 已批准，且例外需要长期留痕时，才进入 `known-exceptions.md`

## 7. 对当前代码的现实含义

结合最新代码，这个最小模型已经能对上当前实现：

- `task-plan` 执行层仍然只消费 `FR / DS`
- `relationship-graph` 负责更完整的追踪视图
- `gate-evaluator` 负责放行或阻断
- `defect` 负责缺陷登记与回流
- `rfc` 负责例外审批与豁免落盘

这说明我们已经具备“流程节点可独立记录、产物 ID 仍可保留”的条件。

## 8. 结论

最小保留，不等于只剩流程节点。

真正可行的最小模型是：

```text
skill 节点 + 主交付链 + 单一 stage gate + 缺陷回流 + RFC 豁免
```

再往下删，就会把“高质量交付”削成“只记录发生过什么”。

# ID / Gate 收敛详细改造方案 - 实施计划

> **For Claude:** REQUIRED SUB-SKILL: Use superpowers:executing-plans to implement this plan task-by-task.

**Goal:** 将当前分散的 ID 体系和 Gate 体系收敛为一套更少层次、更少名词、单一主链路可解释的治理模型，降低认知成本、排障成本和文档/代码漂移风险。

**Architecture:** 保留现有 stage machine、dependency check、hard gate、stage gate、release gate 和 RFC/Defect 机制，但收敛命名与职责边界。ID 侧保留必要类型，不再让多个模块各自解释关系；关系解释以一张 canonical graph 为准，其他视图只做补充。第一波只做收敛，不引入新的状态枚举或新的豁免模型，避免“为简化而再造复杂度”。

**Tech Stack:** TypeScript, Vitest, Markdown, YAML, existing `spec-first` CLI/core modules.

**当前进度:** Phase 0-4 已完成核心实现和回归验证；Phase 5 的文档术语统一、兼容性收口与残余校验仍在继续，文档中的少量待补项保留为后续小修任务。

## 1. 目标态定义

### 1.1 Gate 目标态

Gate 相关概念统一成四个职责层：

- `precondition`
  - 负责执行前置条件，典型如依赖存在、环境准备、文件可读。
- `stage-gate`
  - 负责 stage 的阶段条件检查，主要落在 `condition-registry.ts` + `gate-evaluator.ts`。
- `hard-gate`
  - 负责 skill 入口硬约束，避免带着明显不满足的上下文进入错误阶段。
- `release-gate`
  - 负责发布前最终校验，落在 `golive.ts`。

`confirm-policy` 只负责人工确认策略，不再被称为 gate。

### 1.2 ID 目标态

ID 体系按两层看：

- 主链路 ID
  - `Feature -> REQ -> FR -> DS -> TASK`
- 验证/补充 ID
  - `TC`
  - `RFC`
  - `EX`
  - `Defect`
  - `MatrixRow`

主链路负责推进和交付解释，补充 ID 负责治理、豁免、回流和验证留痕。
其中 `relationship-graph.ts` 当前只负责 row-tier 可表达的 `TC / RFC` 分类；`EX`、`Defect`、`MatrixRow` 保留在各自治理/矩阵模块里，不进入 row-tier 节点集合。

### 1.3 不做项

- 不新增 `PASS_WITH_ENV_ISSUES` 之类的新状态枚举。
- 不把 `TC` 强行塞进 `TaskNode` 的执行上下文。
- 不引入新的豁免文件模型。
- 不重写 stage machine。
- 不要求旧的持久化状态文件做 schema migration，除非后续真的改了持久化格式。

## 2. 实施顺序

### Phase 0: 冻结当前语义并建立回归基线

目标是先把“当前到底有哪些 gate、哪些 ID、哪些关系”固定下来，再动代码。

### Phase 1: Gate 收敛

目标是把所有 gate 语义收成一套命名和一组职责，不再有多套同义词。

### Phase 2: ID 收敛

目标是把 ID 类型和允许集合收成一个共享来源，消灭各处重复定义。

### Phase 3: 关系图收敛

目标是把 `upstream/downstream`、`linkedFr/linkedTc`、`rfcId/frId`、`document-links` 收成一张可解释的 canonical graph。

### Phase 4: 消费端对齐

目标是让 `id` CLI、task parser、trace context、feature resolver 的解释一致。

### Phase 5: 文档和回归收口

目标是把文档术语和代码实现对齐，并用测试锁住收敛结果。

### 2.1 兼容策略

在不改持久化格式的前提下，默认保持以下兼容性：

- `stage-state.json` 继续按现有格式读取，不做迁移。
- `gate-history.jsonl` 继续按现有格式追加，不改存储协议。
- CLI 命令的退出码保持不变。
- CLI 输出文案可以收敛术语，但不要在第一轮就改成机器协议。
- 如果后续要引入机器可读输出，再单独设计版本标记或 `--format json`，不要和当前收敛混在一起。

## 9. 风险控制

### 风险 1: 只改名不改职责

如果只是改标签，不收敛职责，复杂度会继续保留。

**控制方式：**
- 每个 gate 层只保留一个主模块负责。
- 任何重复判断点都必须删除或降级为辅助输出。

### 风险 2: canonical graph 变成“新一套全量大图”

如果把所有关系一次性塞进主图，复杂度会反弹。

**控制方式：**
- 主图只保留真正驱动推进的链路。
- 其余关系只能是补充链，不准提升权重。

### 风险 3: 为了统一而扩张 `TaskNode`

把 `TC` 加进执行节点会让执行层变胖。

**控制方式：**
- 维持 `TaskNode` 窄结构。
- 验证侧关系通过报告或 trace view 展示。

---

## 10. 验收标准

1. 同一件事只会出现在一层 gate 语义里，不会同时被叫做 gate、policy、dependency check。
2. `id next / validate / search / list` 的允许类型集合只来自一份共享定义。
3. 文档中的 gate id 列表与代码 registry 一致，不再有悬空项。
4. 关系解释只剩两层：
   - 主链路
   - 补充链路
5. `TaskNode` 结构不被验证侧关系膨胀。
6. 测试可以稳定证明这套收敛不会回退。

---

## 11. 最终建议执行顺序

1. 先改 gate taxonomy 和 registry 对齐。
2. 再改 ID taxonomy 的共享来源。
3. 再改 canonical graph 和 relation consumers。
4. 最后统一 CLI、文档和测试。

这个顺序的原因是：
- gate 是最容易产生误解和漂移的地方，先收口能减少后续改动的语义噪音。
- ID taxonomy 是各处重复定义的源头，收掉后可以减少很多下游修补。
- 关系图是最容易“越修越大”的部分，必须放在 gate 和 ID 收敛之后做。

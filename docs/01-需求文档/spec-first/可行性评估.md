## 核心结论

### 定位判断

**Spec-First v7.1 在“流程语义完整性 + 追踪量化 + AI协作标准化”三个维度已形成业界第一梯队能力**：8+2 阶段主流程、FR/NFR/DS/API/TASK/TC/ADR/RFC 八类 ID、双层 Hook Gate、12 项度量与健康分，均已在规范层闭环。

### 关键发现

**可行性判断从“能力是否存在”升级为“规范能力与交付成熟度是否一致”**。v7.1 目前呈现“规范完备、交付部分就绪”的状态：

| 能力维度 | 规范定义（Normative） | 当前交付（Delivery） | 可行性影响 |
|---------|------------------------|--------------------|-----------|
| 流程引擎 | 8+2 阶段 + 终态定义 | M1 已交付 | 主流程可执行 |
| 追踪与度量 | 八类 ID + 覆盖率 + 12 指标 | M2/M6 已交付 | 可量化治理可落地 |
| 变更管理 | RFC/Defect FSM + 豁免机制 | M4 已交付 | 变更闭环可落地 |
| Gate 自动化 | GateEngine 自动条件解析 | M3 部分交付 | 需人工兜底 |
| AI 编排 | Context/Catchup/Stats | M5 部分交付 | 需限制 auto 模式 |
| 工具集成 | Hook 安装/CI 模板/feature/commit | M7 规划中 | DX 与规模化受限 |

**核心问题已从“怎么拼四工程”转为“怎么消除 Normative/Delivery 差距”**，重点是 M3/M5 修复与 M7 落地。

### 量化预期

| 指标 | 当前（手动） | 集成后（AI辅助） | 提升幅度 |
|------|------------|-----------------|---------|
| Spec→PR 周期（Size S） | 5-10天 | 3-5天（试点） | 40-60% |
| 追踪矩阵维护 | 手动填充 | CLI计算+人工抽检 | 60-70% |
| Gate校验 | 人工检查 | 自动校验+Owner终审 | 50-70%（M3修复前） |
| 会话中断恢复 | 重新理解上下文 | Catchup + 运行态三文件 | 70-85%（M5修复后可提升） |

### 各方案详细对比

#### OpenSpec

- **官网**: https://openspec.pro
- **定位**: 轻量级、棕地优化的SDD框架，兼容20+AI编码工具
- **核心理念**: Token高效、diff驱动、最小规范层
- **三步工作流**: Proposal（创建规范）→ Apply（AI按规范编码）→ Archive（归档变更）
- **产出物**: 每个变更独立文件夹（`proposal.md` / `specs.md` / `design.md` / `tasks.md`）
- **优势**: 极低门槛、广泛AI工具兼容、适合迭代开发
- **劣势**: 无追踪体系、无质量门禁、缺少架构指导
- **v7.1可借鉴**: 轻量路径可直接映射到 `S + strict` 试点策略，降低首月推行阻力

#### GitHub Spec Kit

- **维护者**: GitHub工程师 Den Delimarsky & John Lam
- **当前版本**: v0.0.77（实验阶段）
- **四阶段工作流**: Specify → Plan → Tasks → Implement & Validate
- **产出物目录**: `.specify/`（`constitution.md` / `spec.md` / `plan.md` / `tasks.md`）
- **支持AI Agent**: GitHub Copilot、Claude Code、Gemini CLI
- **特色**: `/analyze` 命令检测规范与代码的偏差
- **优势**: GitHub生态深度集成、Constitution概念、自动创建Git分支
- **劣势**: 无ID追踪、无变更管理、无多端扩展、实验阶段不稳定
- **v7.1可借鉴**: `/analyze` 思路可强化 `verify quick/full` 的增量与全量一致性校验分层

#### Autospec

- **定位**: YAML-first的规范驱动CLI工具，脱胎于Spec Kit但独立演进
- **四阶段工作流**: Spec(`spec.yaml`) → Plan(`plan.yaml`) → Tasks(`tasks.yaml`) → Implement
- **核心特色**: 所有产出物为YAML格式，支持程序化访问和自动化编排
- **依赖管理**: `tasks.yaml` 内置 `dependencies` 字段，按依赖顺序执行
- **优势**: 机器可解析性极强、自动重试机制、适合CI/CD集成
- **劣势**: YAML可读性不如Markdown、对简单项目过重、缺少质量门禁
- **v7.1可借鉴**: YAML结构化字段可继续沉淀到 `task_plan.md` 依赖/并行调度模型

#### SpecifyPlus

- **定位**: Spec Kit的增强分支，面向多Agent企业级治理
- **核心特色**:
  - **Constitution Gates**: `.specify/memory/constitution.md` 作为不可违背的项目宪法
  - **[P]并行标记**: 任务定义中标记可并行执行的任务
  - **PHR决策追踪**: Prompt History Records，完整记录AI辅助决策链路
- **优势**: 多Agent系统支持、强治理能力、决策可追溯
- **劣势**: 复杂度高、学习曲线陡、小项目过重
- **v7.1可借鉴**: PHR思路可补充到 `gate-history.jsonl + ADR/RFC` 的统一审计链

#### TypeSpec（微软）

- **官网**: https://typespec.io
- **版本**: GA 1.0（2025年5月）
- **定位**: API-first设计语言，单一真理源生成多语言SDK
- **工作流**: Define → Validate → Generate（OpenAPI/Protobuf/JSON Schema/多语言客户端）
- **AI集成**: TypeSpec for Copilot（Build 2025发布）
- **优势**: 编译器级类型安全、微软生态深度集成、多协议支持
- **劣势**: 仅覆盖API设计阶段、微软生态偏向、非全流程方案
- **v7.1可借鉴**: 在 02 Design 引入 TypeSpec 可降低 API-SVC-NNN 分配与契约漂移风险

#### Specmatic

- **官网**: https://specmatic.io
- **定位**: 契约驱动测试平台，API规范即可执行契约
- **工作流**: Design → Mock → Test → Validate → Deploy
- **2026路线图**: MCP支持、Genie AI（自然语言→API规范）、API治理质量评分
- **优势**: 多协议支持（OpenAPI/GraphQL/gRPC/AsyncAPI）、向后兼容性测试、AI-Ready API
- **劣势**: 仅覆盖API测试、微服务专用、单体应用不适用
- **v7.1可借鉴**: 契约测试可作为 05 Verify 的强制子项，支撑 GL-04 E2E 质量门槛

### v7.1的竞争优势

1. **流程完整性**：8+2 阶段 + 终态不可逆定义，覆盖从 Init 到 Release 的全生命周期
2. **治理可量化**：C1-C9 + E1/Q1/H1 形成“覆盖率-效率-质量”一体化指标体系
3. **双层可扩展架构**：Skill 编排 + CLI 确定性执行，天然适配多宿主 AI 平台
4. **审计与放行机制**：Gate/SCA/豁免/GL 门槛形成“试点到公司级推广”的治理闭环
5. **交付透明化**：Normative 与 Delivery 分离呈现，避免“规范能力被误判为已交付能力”

## 业界SDD最佳实践综述

### 核心工作流演进

2025-2026 年业界共识已稳定为：**先规范、再生成、后验证**。  
v7.1 的差异不在“是否遵循五阶段”，而在将五阶段工程化为 **8+2 可执行状态机**，并补齐发布与终态审计。

| 业界五阶段共识 | v7.1 对应阶段 | v7.1 增量价值 |
|---------------|--------------|--------------|
| Specify | 00 Init + 01 Specify | 增加 mode/size/platforms 前置裁剪，避免后续返工 |
| Design/Plan | 02 Design + 03 Plan | 把“设计可行”拆成“设计完成 + 任务可执行”两道 Gate |
| Implementation | 04 Implement | 引入 `/spec-first:code-review` 与 A/B 九维度审查 |
| Review & Acceptance | 05 Verify + 06 Wrap-up | 测试与归档分离，形成可审计证据链 |
| （业界常省略） | 07 Release + 08_done/09_cancelled | 发布指标门禁 + 终态不可逆，防止“伪完成” |

### 人机协作模式演进（对齐 v7.1）

“AI 辅助开发”在业界已从单轮补全转向流程协作，但落地成效取决于**控制策略**而非模型能力本身。

| 协作层级 | 行业常见形态 | 主要风险 | v7.1 对应策略 |
|---------|-------------|---------|---------------|
| L1 助手式 | IDE补全/问答 | 局部最优、上下文断裂 | 通过 `/spec-first:*` 统一入口减少随机工作流 |
| L2 流程式 | Spec Kit/Kiro 的阶段模板 | 阶段切换松散、证据弱 | Gate + SCA + `verify quick/full` 强化阶段准出 |
| L3 治理式 | 企业多 Agent 编排 | 规范与实现漂移 | Normative/Delivery 双轨 + GL-01~GL-04 准入 |

**v7.1 执行原则**：默认 `confirm_policy=strict`，把 AI 定位为“高吞吐草拟者”，把人定位为“Gate Owner + 风险签核者”。

### 人类角色转变

**2026年最有价值的技能**：**需求工程**（Requirement Engineering）

- 将模糊业务需求转化为精确AI规范的能力
- 开发者从"编码者"转向"架构师+验证者"

**时间分配变化**：

| 模式 | 编码/生成 | 设计/规范 | 审核/优化 |
|------|----------|----------|----------|
| 传统开发 | 80% | 10% | 10% |
| SDD模式 | 20%（AI生成） | 30% | 50% |

### 主流AI开发生态分层（v7.1 采纳策略）

| 生态层 | 代表形态 | 能力边界 | v7.1 采纳策略 |
|-------|---------|---------|--------------|
| IDE 助手层 | Copilot/Cursor/Windsurf | 局部生成强、全流程弱 | 作为 04 Implement 的局部生产力插件 |
| 流程模板层 | Spec Kit/Kiro/OpenSpec | 阶段模板强、治理深度有限 | 作为 Skill 指令与模板设计参考 |
| 运行治理层 | Hooks/CI/策略引擎 | 可阻断、可审计 | 作为 v7.1 Gate/SCA/GL 的硬约束承载层 |
| 企业编排层 | 多 Agent Orchestrator | 规模化协作强、复杂度高 | 在 GL 通过后逐步扩展到 L 场景 |

### 关键最佳实践（映射到 v7.1）

1. 阶段切换必须“有证据地通过”：每次 `stage advance` 前执行 `verify full`。  
2. 先守住合规率，再追求效率：C7/C8/C9 未达 100% 时不讨论提速。  
3. 把豁免当债务而非特权：`PASS_WITH_WAIVER` 必须绑定到期与回滚点。  
4. 把上下文当资产管理：`task_plan.md/findings.md/stage-state.json` 是会话连续性的最小集合。  
5. 以 GL 门槛管理推广节奏：未通过 GL-01~GL-04 时只允许试点，不做公司级默认化。

### 测试驱动生成（TDG）升级为 Gate 驱动生成（GDG）

业界 TDG 已被证明有效，但在工程落地中仍需要 Gate 约束；v7.1 采用“TDG + Gate”的 GDG（Gate-Driven Generation）模式：

1. 提供测试用例（输入 + 预期输出）
2. AI生成代码
3. 自动验证代码是否通过测试
4. 迭代优化直到通过所有测试

在 v7.1 中，上述 4 步分别映射为：`01/03 产出 AC+Checklist → 04 实现与CR → 05 验证 → Gate 放行`。  
**结论**：v7.1 的最短落地路径不是“只做 TDG”，而是“TDG 在 Gate 内运行”。

## 端到端可行性评估（基于 v7.1 最新需求）

### AI辅助端到端开发路径（Size S + Mode N，8+2）

```text
人类输入业务意图
    ↓
[AI] 00 Init       → 创建 Feature 工作区 + mode/size/platforms
    ↓
[AI] 01 Specify    → 生成 spec.md + FR/NFR ID + 矩阵初始化      ← DoR Sign-off
    ↓
[AI] 02 Design     → 生成 design.md/contracts/data-model/ADR    ← Design Review + API Coverage=100%
    ↓
[AI] 03 Plan       → 生成 task_plan.md/checklist                ← Task Review + Task Coverage/Compliance=100%
    ↓
[AI] 04 Implement  → /spec-first:code + /spec-first:code-review ← Code CR + Coverage≥80% + PR Compliance=100%
    ↓
[AI] 05 Verify     → 生成 TC + 执行测试 + 安全扫描 + UAT         ← Test Coverage=100% + TC Compliance=100%
    ↓
[AI] 06 Wrap-up    → 归档 + 复盘 + 矩阵闭环                      ← Impl Coverage=100% + Status闭合
    ↓
[AI/CI] 07 Release → Build + Smoke Test + 核心指标窗口观测
    ↓
08_done / 09_cancelled（不可逆终态）
```

### 可行性成立前提（Go/No-Go）

| 前提条件 | Go阈值 | No-Go处置 |
|---------|--------|----------|
| Feature 级 Gate 闭环 | 00-06 各阶段 Exit Gate 全通过 | 禁止 `stage advance` |
| 覆盖率与合规率 | C1-C4/C6-C9 达到阶段阈值（核心项 100%） | 阻断并返回当前阶段修复 |
| 安全门禁 | 无 S1，且无未审批豁免的 S2 | 强制 `FAIL` |
| 豁免机制合规 | `PASS_WITH_WAIVER` 必须有有效 `known-exceptions.md` 条目 | 视同 `FAIL` |
| 上线准入门槛 | GL-01~GL-04 全部通过 | 维持试点，不允许公司级默认推广 |

### v7.1 交付成熟度约束（影响可行性的真实边界）

| 模块 | 规范状态 | 交付状态 | 对可行性的影响 | 当前策略 |
|------|---------|---------|---------------|---------|
| M1 ProcessEngine | Required | ✅ | 阶段推进可执行 | 正常使用 |
| M2 TraceEngine | Required | ✅ | ID 与追踪矩阵可自动化 | 正常使用 |
| M3 GateEngine | Required | 🔧 Partial | Gate 自动评估存在缺口 | Gate Owner 人工兜底 + 禁用 `--force` |
| M4 ChangeMgr | Required | ✅ | RFC/Defect 可闭环 | 正常使用 |
| M5 AIOrchestrator | Required | 🔧 Partial | context/catchup/stats 稳定性受限 | `confirm_policy=strict` 为默认 |
| M6 MetricsEngine | Required | ✅ | 12 指标可计算 | 用于试点决策 |
| M7 ToolIntegration | Required | 📋 Planned | Hook 安装/CI模板/commit封装未完整 | 先手工模板 + 渐进接入 |

### 各阶段 AI 能力边界（v7.1）

| 阶段 | AI可承担 | 人类必须介入 | AI生成占比（建议） |

## 关键差距与补齐方案（v7.1 对齐版）

### Gap 1: M3 GateEngine 自动条件解析链路未闭合

**现状**：Gate 判定在规范层定义完整，但自动条件解析尚未完全注入，存在 `--force` 被滥用风险。  
**影响**：自动 Gate 可信度不足，阶段推进仍依赖人工兜底。  
**补齐方案**：

1. 固化 Gate DSL 与阶段条件映射，收敛“同义条件”表达。
2. 在 `gate check` 输出中强制展示“判定证据链”。
3. 禁止试点仓库使用 `--force`，仅保留管理员破窗流程。

### Gap 2: M5 AIOrchestrator 存在命令签名漂移（typecheck 未归零）

**现状**：`ai context/catchup/stats` 在命令层与核心模块签名不一致。  
**影响**：会话恢复与上下文注入稳定性不足，限制 L 规模特性可行性。  
**补齐方案**：

1. 统一命令签名与类型约束，`npm run typecheck` 归零作为合并门槛。
2. `confirm_policy` 默认 `strict`，仅低风险白名单允许 `assisted/auto`。
3. 增加 Context Pack v2.0 合规测试（control <2KB + references 按需读取）。

### Gap 3: M7 ToolIntegration 未交付（Hook安装/CI模板/commit封装）

**现状**：流程规范要求双层 Hook 与 CI 模板，但工具化仍处于规划。  
**影响**：团队迁移成本高、执行一致性依赖人工。  
**补齐方案**：

1. 优先交付 Hook 安装器（prepare-commit-msg / commit-msg / pre-push）。
2. 提供 CI 最小模板（Gate + SCA + Coverage）。
3. 落地 `feature`/`commit` 命令组，降低日常操作摩擦。

### Gap 4: Skill 体系“定义完成”与“联调完成”仍有差距

**现状**：16 个 Skill 已定义，但 `/spec-first:xxxx` 到 Runtime/CLI 的端到端联调尚未全面验收。  
**影响**：编排链路在边界场景（多 Feature 切换、异常恢复）下仍有不确定性。  
**补齐方案**：

1. 以 Daily Path（`plan → code → code-review → verify`）为主线做联调回归。
2. 每个 Skill 输出统一“输入/输出契约”与失败回退策略。
3. 将 GL-04 作为联调完成的硬验收口。

## AI可消费的规范结构最佳实践

### 核心原则：人机双可读

业界共识：**纯Markdown对人友好但AI解析不稳定，纯YAML对机器友好但人类审核困难**。最佳实践是**YAML Front Matter + Markdown正文**的混合结构。

### 结构化层次模型

```text
Level 0: 纯自然语言Markdown     → AI解析准确率 ~70%
Level 1: 模板化Markdown（固定标题层级） → AI解析准确率 ~80%
Level 2: YAML Front Matter + Markdown  → AI解析准确率 ~95%
Level 3: 纯YAML/JSON（Autospec模式）  → AI解析准确率 ~99%
```

**v7.1推荐目标**：Level 2（兼顾人机可读性）。

### spec.md Front Matter 最佳实践

```yaml
---
# spec.md YAML Front Matter
feature_id: "001-user-auth"
title: "用户认证模块"
mode: N          # N=新建 / I=迭代
size: S          # S/M/L
platforms: [H5, Backend]
priority: P0
status: draft    # draft → reviewed → approved → implemented → verified

requirements:
  functional:
    - id: FR-AUTH-001
      title: "邮箱注册"
      priority: P0
      acceptance_criteria:
        - ref: AC-1
          given: "用户提供有效邮箱和密码"
          when: "提交注册表单"
          then: "返回201 + JWT Token"
          test_data:
            input: {email: "user@example.com", password: "Pass123!"}
            expected: {status: 201, has_token: true}
    - id: FR-AUTH-002
      title: "邮箱登录"
      priority: P0
  non_functional:
    - id: NFR-PERF-001
      dimension: performance
      metric: "登录接口P99 < 200ms"

traces:
  design: "design.md"
  tasks: "task_plan.md"
  matrix: "traceability-matrix.md"
---
```

### task_plan.md Front Matter 最佳实践

```yaml
---
# task_plan.md YAML Front Matter
feature_id: "001-user-auth"
phase: "03-plan"
total_tasks: 4
parallel_groups:
  - group: 1
    tasks: [TASK-AUTH-001, TASK-AUTH-002]  # 可并行
  - group: 2
    tasks: [TASK-AUTH-003]                  # 依赖group 1
  - group: 3
    tasks: [TASK-AUTH-004]                  # 依赖group 2

tasks:
  - id: TASK-AUTH-001
    title: "实现注册API"
    traces: [FR-AUTH-001]
    size: S
    parallel: true
    worktree: false
    acceptance_criteria:
      - ref: FR-AUTH-001/AC-1
        test_input: {email: "user@example.com", password: "Pass123!"}
        expected_output: {status: 201, has_token: true}
    test_cases: [TC-UT-AUTH-001, TC-IT-AUTH-001]
---
```

### 反模式与设计原则

**反模式（避免）**：

| 反模式 | 问题 | 正确做法 |
|--------|------|---------|
| 纯散文式需求 | AI无法稳定提取结构化信息 | 用YAML Front Matter固定关键字段 |
| ID仅出现在正文中 | AI可能遗漏或误解引用关系 | ID在Front Matter中显式声明追踪链 |
| AC用自然语言描述 | 无法自动转化为测试用例 | 用Given-When-Then + test_data结构 |
| 一个文件包含多个Feature | 上下文窗口浪费、追踪混乱 | 一Feature一目录，文件职责单一 |
| 手动维护追踪矩阵 | 易过时、易遗漏 | Front Matter声明traces，工具自动生成矩阵 |

**设计原则**：

1. **单一真理源**：每个ID只在一处定义，其他位置通过引用关联
2. **渐进式结构化**：Size S只需基础Front Matter，Size L才需完整YAML
3. **向后兼容**：Front Matter是增量增强，不破坏现有Markdown可读性
4. **工具可生成**：所有Front Matter字段均可由AI在对应阶段自动填充

## 风险提醒

### Risk 1: Normative 与 Delivery 漂移

**风险等级**：🔴 高

**表现**：规范定义完整，但 M3/M5/M7 未完全交付，若按“全能力已可用”推进，会导致流程承诺与实际体验不一致。

**缓解措施**：
- 以 Delivery 状态为准，不以 Normative 目标替代现状
- 试点默认 `strict` 模式，禁止 `--force` 常态化
- 每周发布“规范-交付差距清单”

### Risk 2: Gate 自动化缺口导致误放行

**风险等级**：🔴 高

**表现**：M3 自动条件解析未完全闭合时，可能出现 Gate 误判或人工误读。

**缓解措施**：
- Gate Owner 人工终审不降级
- Gate 输出强制证据化（失败条目映射到具体 ID）
- 对 `PASS_WITH_WAIVER` 做到期自动升级阻断

### Risk 3: 上下文恢复不稳定导致重复劳动

**风险等级**：🟡 中

**表现**：M5 漂移导致 `catchup/context` 不稳定，AI 重复分析与遗漏追踪链。

**缓解措施**：
- context 协议测试纳入 CI
- 运行态三文件（`task_plan.md/findings.md/stage-state.json`）强制更新
- `full verify` 作为阶段切换前置

### Risk 4: 迁移成本过高导致团队绕过流程

**风险等级**：🟡 中

**表现**：命令多、路径长，开发者绕过 `/spec-first:*` 直接手工操作，造成追踪断链。

**缓解措施**：
- 优先推广 Daily Path 四步（plan/code/code-review/verify）
- 先交付 `feature`/`commit` 命令降低心流中断
- 将失败提示从“报错”改为“可执行修复建议”

### Risk 5: 澄清债务堆积导致后期返工

**风险等级**：🟡 中

**表现**：`NEEDS CLARIFICATION` 项目未在01/02阶段清零，进入03/04后以“需求变更”形式爆发，导致任务重排和代码返工。

**缓解措施**：

- 为每个Feature设置澄清预算（默认≤3），超限自动触发澄清会
- 将“澄清项清零”作为进入03阶段硬门槛
- 在复盘中统计“澄清延迟关闭率”，连续超阈值则回退流程复杂度

### 风险矩阵总览

| 风险 | 等级 | 影响 | 缓解核心 |
|------|------|------|---------|
| 规范-交付漂移 | 🔴高 | 承诺失真、推广失败 | 以 Delivery 状态驱动节奏 |
| Gate缺口 | 🔴高 | 误放行、质量事故 | 人工终审 + 证据化判定 |
| 上下文不稳定 | 🟡中 | 重复劳动、追踪断链 | M5修复 + 三文件强制更新 |
| 迁移成本高 | 🟡中 | 绕流程、工具闲置 | Daily Path + DX 命令补齐 |
| 澄清债务 | 🟡中 | 后期返工、任务重排 | 澄清预算 + 前置清零门槛 |
| 完美主义 | 🟡中 | 交付延迟、范围蔓延 | 分阶段验收 + 时间盒 |

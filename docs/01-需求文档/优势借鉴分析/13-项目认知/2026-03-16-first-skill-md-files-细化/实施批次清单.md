# First Skill MD 文件重构实施批次清单

> 目标：将逐文件改造说明转化为可执行批次，确保重构按依赖顺序推进，避免先改文档后改契约、或先改投影后改真源导致反复返工。

---

## 一、批次划分原则

实施顺序遵循 4 条原则：

1. 先改 **总契约**，再改 **分项说明**。
2. 先改 **skill 层文档**，再改 **docs/first 投影视图**。
3. 先改 **基础 canonical docs**，再改 **专题型 docs**。
4. 先改 **默认生成内容**，再改 **条件生成内容**。

---

## 二、批次总览

| 批次 | 目标 | 范围 | 产出 |
|------|------|------|------|
| Batch 1 | 收口总契约 | `SKILL.md` + 核心 references | 新的 first 正式口径 |
| Batch 2 | 收口基础流程与质量规则 | 执行流、检测、QA、测试策略 | 可执行的 runtime-first 流程规范 |
| Batch 3 | 收口专题能力规范 | code/api/setup/domain/database 五类专题 references | 专题型资产与文档的生成规范 |
| Batch 4 | 重写第一批基础 canonical docs | `README` + 12 个基础/派生 docs | 第一批稳定投影视图 |
| Batch 5 | 收口剩余基础 canonical docs | `tech-stack/api-docs/codebase-overview/domain-model` | 基础文档全集收口完成 |
| Batch 6 | 接入专题型 docs | `architecture/call-graph/external-deps/local-setup/development-guidelines` | runtime-first 专题阅读层 |
| Batch 7 | 接入条件型 docs | `database-er` | 条件型数据库认知能力 |

---

## 三、详细批次

### Batch 1：总契约收口

目标：

- 统一 first 的正式产品定义
- 删除 quick/deep 双模式口径
- 删除旧 Agent 直接生成 markdown 的表述

文件：

- `skills/spec-first/00-first/SKILL.md`
- `skills/spec-first/00-first/references/execution-flow.md`
- `skills/spec-first/00-first/references/subagent-architecture.md`

完成标志：

1. `first` 被定义为单一标准模式。
2. 正式资产、条件型资产、专题型文档都有清晰定义。
3. 生成分层与消费契约已进入主文档。

### Batch 2：基础流程与规则收口

目标：

- 确立项目识别、生成规则、测试策略和质量门禁

文件：

- `skills/spec-first/00-first/references/detection-rules.md`
- `skills/spec-first/00-first/references/端类型产物映射.md`
- `skills/spec-first/00-first/references/quality-assurance-rules.md`
- `skills/spec-first/00-first/references/testing-strategy.md`

完成标志：

1. 项目类型到资产生成策略映射明确。
2. QA 与测试矩阵基于 runtime assets 而不是旧文档模式。
3. orphan / projection / ghost / roundtrip / consumption gates 全部落到文档口径。

### Batch 3：专题能力规范收口

目标：

- 为结构、接口、环境、领域、数据库五类专题建立正式规范

文件：

- `skills/spec-first/00-first/references/agents-code-analysis.md`
- `skills/spec-first/00-first/references/agents-api-deps.md`
- `skills/spec-first/00-first/references/agent-guidelines-setup.md`
- `skills/spec-first/00-first/references/agent-domain-model.md`
- `skills/spec-first/00-first/references/agent-database.md`
- `skills/spec-first/00-first/references/database-config.md`

完成标志：

1. 所有专题文档都不再被视为真源。
2. 每类专题都明确 runtime 来源、投影规则和条件生成规则。
3. `database-er.md` 已明确为条件型文档。

### Batch 4：基础 canonical docs 重写

目标：

- 建立第一批稳定可投影的 docs/first 入口层

文件：

- `docs/first/README.md`
- `docs/first/summary.md`
- `docs/first/role-views.md`
- `docs/first/stage-views.md`
- `docs/first/steering.md`
- `docs/first/conventions.md`
- `docs/first/critical-flows.md`
- `docs/first/change-map.md`
- `docs/first/entry-guide.md`
- `docs/first/reboot-guide.md`
- `docs/first/common-playbooks.md`
- `docs/first/known-risks-and-traps.md`

完成标志：

1. 所有基础 docs 都具备明确 runtime 来源。
2. `README.md` 已成为全量入口，而非历史兼容页。
3. 基础认知、导航、风险和 playbook 能形成第一层项目级认知。

### Batch 5：收口剩余基础 canonical docs

目标：

- 将剩余 4 个基础 canonical docs 收口为正式投影视图

文件：

- `docs/first/tech-stack.md`
- `docs/first/api-docs.md`
- `docs/first/codebase-overview.md`
- `docs/first/domain-model.md`

完成标志：

1. 这 4 个文件不再带有 legacy 语义。
2. 它们都只从 runtime assets 投影生成。
3. 基础 canonical docs 的 16 个正式文档定义全部收口完成。

### Batch 6：专题型 docs 接入

目标：

- 把高认知价值的旧专题文档重新纳入新流程

文件：

- `docs/first/architecture.md`
- `docs/first/call-graph.md`
- `docs/first/external-deps.md`
- `docs/first/local-setup.md`
- `docs/first/development-guidelines.md`

完成标志：

1. 专题文档全部有 runtime 来源。
2. `architecture`、`external-deps`、`local-setup`、`development-guidelines` 可默认生成。
3. `call-graph` 已明确为条件生成。

### Batch 7：条件型数据库认知接入

目标：

- 让数据库项目具备正式的数据认知能力

文件：

- `docs/first/database-er.md`

依赖：

- `database-schema.json` 的 schema 与生成规则已明确

完成标志：

1. `database-er.md` 只在条件满足时生成。
2. 不适用项目不会被强制生成低质量数据库文档。
3. 数据驱动项目可获得显著增强的项目认知能力。

---

## 四、优先级建议

### P0：必须先做

- Batch 1
- Batch 2
- Batch 3

原因：

- 这些批次决定 first 的正式 contract、规则体系和专题能力规范，是后续 docs 层改造的前置条件。

### P1：建议紧接着做

- Batch 4
- Batch 5
- Batch 6

原因：

- 这些批次决定基础 docs、转正文档和正式专题文档是否能高质量落入新体系。

### P2：条件能力最后做

- Batch 7

原因：

- 数据库认知很有价值，但不适用于所有项目，适合作为条件能力单独推进。

---

## 五、每批验收问题

每批完成后，至少回答以下问题：

### Batch 1 验收问题

1. `first` 是否还存在 quick/deep 双模式表述？
2. 是否还存在“Agent 直接生成 markdown 真源”的说法？

### Batch 2 验收问题

1. QA 和测试是否已经以 runtime assets 为中心？
2. 端类型映射是否已经转成生成策略，而非旧文档组合？

### Batch 3 验收问题

1. 每类专题是否都有明确 runtime 来源？
2. 条件生成规则是否已经清楚？

### Batch 4 验收问题

1. 基础 docs 是否全部可从 runtime 重建？
2. `README.md` 是否已经成为正式入口？

### Batch 5 验收问题

1. 这 4 个剩余基础 canonical docs 是否全部收口完成？
2. 是否彻底移除了它们的旁路生成语义？

### Batch 6 验收问题

1. 专题型 docs 是否都不再依赖自由生成？
2. 是否区分了默认生成与条件生成？

### Batch 7 验收问题

1. `database-er.md` 是否只在适用项目生成？
2. 数据认知是否具备稳定 runtime 真源支持？

---

## 六、建议执行顺序

建议实际执行顺序如下：

1. 先统一口径：Batch 1
2. 再统一规则：Batch 2
3. 再推进专题能力规范：Batch 3
4. 再统一基础 docs：Batch 4
5. 再收口剩余基础 canonical docs：Batch 5
6. 再接入专题 docs：Batch 6
7. 最后接入数据库条件能力：Batch 7

这样做的原因是：

- 先保证“正式系统成立”
- 再补“高价值专题认知”
- 最后再补“条件型专业能力”

---

## 七、结论

逐文件细化已经完成，但若没有批次策略，执行时仍容易混乱。

因此本批次清单的作用是把重构从“文件很多”收口成“7 个可执行阶段”：

> 先立契约，再立规则，再立基础认知层，再吸收旧文档价值，最后补条件型能力。

# 第一批专家集成改造任务文档

> 日期：2026-03-15  
> 范围：第一批核心 skill  
> 路线：先沉淀 `profile` 真源，再按 `03-spec`、`04-design`、`06-task`、`08-review`、`12-verify`、`13-orchestrate` 吸收  
> 本版目标：基于当前实现，把任务文档收敛成可实施、可验证、与 runtime 一致的单一路线版本

## 一、改造目标

| 项目 | 内容 |
|------|------|
| 保持不变 | 不改阶段状态机、不改 Gate 真源、不引入第二套 orchestrator |
| 正式路线 | 先建立 `profile` 真源与映射，再改 skill 文本，再补最小 runtime 接入，最后补测试 |
| 第一批 skill | `03-spec`、`04-design`、`06-task`、`08-review`、`12-verify`、`13-orchestrate` |
| 第一批 profile | `planning-consult`、`architecture-review`、`code-quality-review`、`security-review`、`execution-orchestration`、`api-verification`、`frontend-evidence-check` |
| Canonical 边界 | 第一批 7 个 profile 构成唯一顶层 canonical profile 注册表；第二、第三批默认复用或扩展，不再新增顶层 profile |
| 本版修正重点 | 补齐 `13-orchestrate` 的 CLI/runtime 任务、`06-task` 的 schema/parser 任务、所有改造的测试任务 |

## 二、当前实现约束

| 约束 | 当前实现 | 对任务拆分的影响 |
|------|------|------|
| `orchestrate` 主行为在 CLI | `src/cli/commands/orchestrate.ts` | 不能只改 `13-orchestrate/SKILL.md`，必须补 CLI/runtime 子任务 |
| TASK 结构有既有解析器 | `src/core/task-plan/parser.ts` | 新增 handoff / DoD 若要被消费，必须补 schema/parser/消费端任务 |
| Skill 渲染走统一 runtime | `src/core/skill-runtime/dispatcher.ts`、`src/core/skill-runtime/prompt-assembler.ts` | profile 注入应挂在 skill runtime，而不是散落在 skill 文本里 |
| 仓库已有 skill 文档与渲染测试 | `tests/unit/*skill-docs*.test.ts`、`tests/integration/skill-render.test.ts`、`tests/unit/skill-runtime.test.ts` | 每个改造项都要明确落到现有测试面 |

## 三、问题修复方案

| 问题 | 现状 | 修复方案 |
|------|------|------|
| `13-orchestrate` 任务只写 skill 文本改造 | 无法覆盖真正的 CLI 编排路径 | 增加 `orchestrate.ts` 的 profile 选择、输出注入、回退逻辑任务 |
| `06-task` 引入 handoff/DoD 但无消费链路 | 目前 parser 不识别相关结构 | 增加 task plan schema、parser、消费端对齐任务 |
| 公共前置任务偏文档化 | 缺测试与回归要求 | 把公共任务拆成 profile 真源、runtime 接入、schema、测试四条并行线 |
| 验证方式过于抽象 | 无法直接转成开发/验收动作 | 每个任务补“修改文件 + 新增测试 + 运行命令” |

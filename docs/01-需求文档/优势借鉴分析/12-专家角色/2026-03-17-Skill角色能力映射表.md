## 使用说明

本文档是**纯文档集成方案**实施完成后的参考手册，用于：
- 快速查询某个 Skill 融合了哪些专家能力
- 理解改造后的 Skill 有什么新能力
- 指导团队成员使用增强后的 Skill

**关键概念**：
- **专家角色**：从 agency-agents、oh-my-opencode、Trellis 等项目借鉴的专家视角
- **核心能力**：专家角色带来的决策标准、输出模板、检查清单、停止条件
- **使用场景**：什么情况下这些能力会发挥作用

## 详细说明

### 第一批（P0）- 核心治理入口

#### 08-review - 代码审查

**融合角色**：Code Reviewer + Security Engineer + Oracle

**核心能力**：
- 问题分级机制：🔴 阻断 / 🟡 建议 / 💭 讨论
- 三要素评论法：位置 + 原因 + 建议
- STRIDE 威胁模型：识别安全风险
- 只读审查模式：不直接写代码，只给建议

**使用场景**：
- 核心逻辑变更 → 架构审查模式（关注状态机、并发、一致性）
- 安全敏感变更 → 安全审查模式（输入验证、鉴权、敏感数据）
- 常规功能 → 质量审查模式（可维护性、测试覆盖、回归风险）

**新增能力**：审查模式选择矩阵、安全敏感检查清单、Findings 输出模板

---

#### 12-verify - 验证验收

**融合角色**：Reality Checker + API Tester + Check Agent

**核心能力**：
- 默认 NEEDS WORK 态度：证据不足时默认不通过
- 证据驱动："看起来可以"不被接受
- 三轴测试：功能、安全、性能
- 覆盖率要求：≥95% 才算通过
- 迭代限制：最多 5 次

**使用场景**：
- API 项目 → 接口契约 + 集成测试 + 性能基线
- Frontend 项目 → 视觉 + 交互 + 无障碍 + 性能
- 安全敏感 → 额外执行安全验证

**新增能力**：验证模式矩阵、证据要求模板、默认怀疑规则

---

#### 04-design - 架构设计

**融合角色**：Software Architect + Oracle

**核心能力**：
- 强制多方案对比：至少 2 个候选方案 + trade-off 说明
- ADR 模板：架构决策记录
- 拒绝推进规则：无法解释权衡则拒绝
- 只读审查模式：不直接写代码

**使用场景**：
- 架构决策 → 必须提供权衡表（方案 A vs 方案 B）
- 设计前检查 → 需求边界、技术约束、关键假设、依赖服务
- 风险评估 → 失败模式分析（5 大场景）

**新增能力**：架构权衡表、设计边界检查、失败模式分析

---

#### 13-orchestrate - 任务编排

**融合角色**：Atlas + Agents Orchestrator + Prometheus

**核心能力**：
- 只编排不实现：明确编排者边界
- Notepad 学习：积累经验
- QA 门控：未通过前不推进
- 重试限制：最多 3 次
- 5 项清单门控：边界确认

**使用场景**：
- 需求澄清 → 选择 planning-consult 专家
- 架构设计 → 选择 architecture-review 专家
- 代码审查 → 选择 code-quality-review / security-review 专家
- 验证验收 → 选择 api-verification / frontend-evidence-check 专家

**新增能力**：专家选择规则、交接模板、阻塞回流处理


#### 03-spec - 需求规格

**融合角色**：Prometheus + Metis + Momus + Plan Agent

**核心能力**：
- 5 项清单门控：目标明确、范围收敛、成功标准、约束条件、风险识别
- 5 类缺口检测：隐含意图、歧义表达、AI 风险、缺失 AC、边界情况
- 4 维度审查：清晰度、完整性、可测试性、一致性
- 4 类拒绝条件：目标模糊、任务不独立、无验收标准、依赖未明

**使用场景**：
- 需求澄清 → 必须通过 5 项清单才能输出 FR
- 歧义扫描 → 检测 5 类常见缺口
- AC 检查 → 可观察、可测试、可复核、无歧义

**新增能力**：需求边界澄清协议、隐含假设扫描、AC 可测试性检查

---

#### 06-task - 任务规划

**融合角色**：Prometheus + Momus + Atlas

**核心能力**：
- 范围收敛：明确做什么、不做什么
- 任务独立性：输入明确、输出明确、可独立验证、时间可估
- 可分派性验证：任务可以交给他人执行

**使用场景**：
- 任务规划 → 检查任务独立性（4 项标准）
- 任务定义 → 明确 Task DoD（功能完成、测试通过、文档更新、验收标准）
- 依赖管理 → 明确并行、强依赖、弱依赖

**新增能力**：任务独立性检查、Task DoD 模板、并行与依赖约束

### 第二批（P1）- 流程扩展

#### 05-research - 技术调研

**融合角色**：Librarian + Oracle

**核心能力**：
- 证据强度分级：强证据（官方文档、源码、实测）/ 中证据（博客、案例）/ 弱证据（观点）
- 外部资料查证：不猜测原则
- 只读模式：不直接实现，只给建议

**使用场景**：
- 技术选型 → 证据分级评估
- 资料不足 → 停止调研，标记"需要 POC"

**新增能力**：证据强度分级、资料不足停止条件

---

#### 21-analyze - 问题分析

**融合角色**：Oracle + Reality Checker

**核心能力**：
- 只读诊断模式：不直接修复，只分析
- 证据导向：基于事实分析
- 默认怀疑态度：不轻易下结论

**使用场景**：
- 问题诊断 → 评估症状清晰度、复现稳定性、影响范围、紧急程度
- 优先级建议 → P0（立即）/ P1（本周）/ P2（本月）/ P3（排期）

**新增能力**：背景质量诊断矩阵、修复优先级建议

---

#### 07-code - 代码实现

**融合角色**：Hephaestus + Security Engineer + Implement Agent

**核心能力**：
- 研究先于行动：实现前必读规范
- 实现前风险自检：边界、测试策略、安全约束
- 遇规范冲突停下澄清：不猜测
- 最小变更原则：只改必要范围
- 默认不信任外部输入：安全约束

**使用场景**：
- 实现前 → 检查规范已读、边界已明确、测试策略已明确、安全约束已识别
- 实现中 → 范围外问题记录到 findings.md，不立即修复
- 实现后 → 功能符合 spec、测试通过、类型检查通过、代码格式化

**新增能力**：实现前风险自检、范围外问题 findings.md 协议、完成后最小交付要求

---

#### 11-plan - 计划制定

**融合角色**：Prometheus + Metis + Momus + Plan Agent

**核心能力**：
- 5 项清单 + 边界明确
- 计划补漏：识别遗漏
- 4 维度审查 + 量化阈值：清晰度、完整性、可测试性、一致性
- 4 类拒绝条件：目标模糊、任务不独立、无验收标准、依赖未明

**使用场景**：
- 计划审查 → 通过 4 维度检查
- 执行前批准 → 提供风险评估、资源确认、依赖明确
- Plan Summary → 目标、范围、里程碑、风险

**新增能力**：计划质量检查表、执行前批准证据要求、Plan Summary 最小字段

---

#### 20-spec-review - 规格评审

**融合角色**：Momus + Software Architect

**核心能力**：
- 4 维度审查 + 量化阈值
- 可实现性检查：技术路径、依赖服务、性能要求、团队能力
- 识别无理由的抽象

**使用场景**：
- 完整性评审 → 检查 FR/AC 是否完整
- 可实现性评审 → 检查技术可行性
- 一致性评审 → 检查 FR-AC 是否一致

**新增能力**：规格评审模式矩阵、FR-AC 一致性检查、可实现性与依赖风险检查、评审结论模板

---

#### 10-archive - 归档收尾

**融合角色**：Reality Checker + Check Agent

**核心能力**：
- 证据闭环：必须收集完整证据
- 默认 NEEDS WORK：证据不足不允许归档
- 完成标记检查：不能跳过
- 不能跳过：强制执行

**使用场景**：
- 归档前 → 收集验证报告、审查记录、变更日志、追踪矩阵
- 遗留风险分层 → 已解决、已缓解、已记录
- 知识沉淀 → 更新 README、CLAUDE.md、docs/

**新增能力**：复盘证据回收清单、遗留风险与行动项分层、知识沉淀要求、归档失败停止条件


### 第三批（P3）- 轻量引导

> **注**：第三批 skill 不引入具名专家角色，只增加轻量级的引导建议。

#### 00-first - 首次使用

**核心能力**：
- 项目认知优先级：README → CLAUDE.md → config.yaml
- 关键风险阅读提示：brownfield 先 doctor，greenfield 先 init
- 边界说明：只负责引导，不执行具体任务

**使用场景**：首次使用 spec-first 时的引导

**新增能力**：项目认知优先级、关键风险阅读提示、边界说明

---

#### 00-onboarding - 项目入职

**核心能力**：
- 角色视角推荐：开发者、架构师、PM 不同路径
- 场景路径映射：新功能、修 bug、查状态
- 退化场景兜底：文档缺失时的保守建议

**使用场景**：新成员加入项目时的引导

**新增能力**：角色视角推荐、场景路径映射、退化场景兜底

---

#### 01-init - 项目初始化

**核心能力**：
- 初始化前检查清单：项目类型、技术栈、团队规模
- 缺失前提的保守建议：不猜测，询问用户

**使用场景**：初始化新项目或接入现有项目

**新增能力**：初始化前检查清单、缺失前提的保守建议

---

#### 02-catchup - 上下文恢复

**核心能力**：
- 恢复摘要模板：当前阶段、最后操作、待办任务
- 阻塞识别规则：stage 停滞、任务无进展、Gate 失败
- 保守回流建议：上下文不清晰时不主动推进

**使用场景**：中断后恢复工作时的上下文重建

**新增能力**：恢复摘要模板、阻塞识别规则、保守回流建议

---

#### 14-status - 状态查询

**核心能力**：
- 风险解释规则：stage 停滞、Gate 失败、覆盖率低
- 下一步建议生成规则：根据当前 stage 给出建议
- 展示边界：只展示状态，不执行操作

**使用场景**：查询项目当前状态和健康度

**新增能力**：风险解释规则、下一步建议生成规则、展示边界

---

#### 15-doctor - 健康诊断

**核心能力**：
- 诊断分层规则：🔴 阻断 / 🟡 警告 / 💭 建议
- 修复优先级建议：P0（具体命令）/ P1（修复建议）/ P2（参考文档）
- 协作边界：只诊断，不自动修复

**使用场景**：诊断项目健康度和问题

**新增能力**：诊断分层规则、修复优先级建议、协作边界

---

#### 16-sync - 同步操作

**核心能力**：
- 同步策略选择规则：追踪矩阵、文档、状态
- 冲突解释模板：描述冲突、分析原因、给出选项
- 边界说明：只同步，不修改业务逻辑

**使用场景**：同步追踪矩阵、文档、状态

**新增能力**：同步策略选择规则、冲突解释模板、边界说明

---

#### 17-feature - Feature 管理

**核心能力**：
- Feature 导航建议：查看、切换、创建
- 多 Feature 场景提示：提示当前 Feature、其他 Feature 状态
- 命令边界：只管理 Feature 元数据，不执行具体开发任务

**使用场景**：管理多个 Feature 的切换和状态

**新增能力**：Feature 导航建议、多 Feature 场景提示、命令边界

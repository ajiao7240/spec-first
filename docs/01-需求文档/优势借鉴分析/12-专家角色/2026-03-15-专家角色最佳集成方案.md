# 专家角色最佳集成方案

> 日期：2026-03-15  
> 范围：仅保留当前确定方案，不再保留备选路线对比  
> 核心结论：采用三层集成模型，先建立第一批 `canonical profiles`，再通过第二、第三批 `fragments` 与 `command-local guidance` 分层吸收专家能力

## 零、全局结论

| 问题 | 结论 |
|------|------|
| 当前方案是否准确 | 准确。它与 `spec-first` 现有 `stage + gate + skill runtime + orchestrate` 骨架一致，不引入第二套流程真源 |
| 当前方案是否合理 | 合理。它把专家能力分层收敛为 `canonical profiles`、`fragments`、`command-local guidance`，避免顶层角色膨胀 |
| 是否符合最佳实践 | 基本符合。更接近 workflow-based organization、明确依赖与回退、渐进式接入，而不是重型多 agent runtime |
| 是否适合单期集中交付 | 不适合。应坚持“单一路线、分阶段实施”，而不是一次性把三层和三批全部落完 |
| 推荐实施口径 | 先第一批核心治理入口，再第二批扩展治理入口，最后第三批命令侧 guidance |

## 一、方案概述

| 项目 | 内容 |
|------|------|
| 集成方式 | 不引入独立专家运行时，不复制外部项目完整 orchestrator；在 `spec-first` 内建立三层集成模型：`canonical profiles`、`expert fragments`、`command-local guidance` |
| 目标 | 提升需求澄清、架构设计、任务拆解、代码审查、阶段验收、编排交接的专业深度 |
| 第一批适用范围 | `03-spec`、`04-design`、`06-task`、`08-review`、`12-verify`、`13-orchestrate` |
| 后续扩展范围 | 第二批通过 `fragment` 覆盖 `05/07/10/11/20/21`，第三批通过 `fragment + command-local guidance` 覆盖 `00/01/02/14/15/16/17` |
| 保持不变 | `stage-state.json`、`gate`、`traceability-matrix.md`、`todo-state.json`、现有 `orchestrate`/`auto-loop` 主干 |
| 一句话原则 | 专家角色只提供“视角、模板、交付要求、审查维度”，不替代当前流程真源 |
| 实施原则 | 单路线，不单次集中交付；所有落地都应有回退策略与现有主路径兼容保证 |

## 二、当前实现下的集成边界

| 当前实现约束 | 含义 | 集成要求 |
|------|------|------|
| `StageState` / `GateResult` 是流程真源 | 流程推进已由阶段机和 gate 决定 | profile 不能改变流程真源，只能增强 skill 输出 |
| `orchestrate` 已是主编排 Skill | 主编排入口已经存在 | 不新建第二套 orchestrator，只增强其协作模板 |
| `auto-loop` 已有任务级循环 | 任务执行与恢复机制已存在 | profile 不承接运行时状态，只提供任务类型视角 |
| `dispatcher` + `prompt-assembler` 已是 Skill 注入入口 | 已有统一渲染链路 | 专家能力应优先接到 skill 渲染链路 |
| `context-pack` 已负责上下文裁剪 | 已有上下文控制机制 | profile 不额外发明上下文存储，复用现有裁剪 |
| 一部分 CLI 命令无稳定 `feature/stage/task` 上下文 | `first/init/doctor/feature/onboarding` 更多直接依赖 cwd、CLI 参数和已有资产 | 这类命令不能硬接 `taskType -> profile`，只能走静态 fragment 或 command-local guidance |

## 三、三层集成模型

| 层级 | 适用范围 | 运行时地位 | 典型落点 |
|------|------|------|------|
| Layer 1: `canonical profiles` | 第一批核心 skill | 顶层稳定语义，可进入 `taskType -> profile` 解析链路 | `src/core/agents/orchestrator-profiles.ts`、`skills/spec-first/experts/*.md` |
| Layer 2: `expert fragments` | 第二批扩展 skill，第三批静态 skill | 只作为 `skillName -> fragments` 的附着层，不进入顶层 registry | `src/core/agents/expert-fragments.ts`、`skills/spec-first/experts/fragments/*.md` |
| Layer 3: `command-local guidance` | 无稳定上下文的 CLI 命令 | 只影响命令侧提示与建议，不参与动态 profile 解析 | `src/cli/commands/first.ts`、`init.ts`、`doctor.ts`、`feature.ts`、`onboarding.ts` |

## 五、第二批与第三批的 Fragment / Guidance 约束

| 类型 | 约束 | 说明 |
|------|------|------|
| 第二批 fragment | 只能补强已有 canonical profile，不新增顶层 runtime 角色 | 适用于 `05/07/10/11/20/21` |
| 第三批 fragment | 只能做轻量认知、恢复、诊断、同步建议 | 适用于 `00/01/02/14/15/16/17` |
| command-local guidance | 只给命令输出补“下一步建议 / 风险提示 / 回退建议” | 适用于 `first/onboarding/init/doctor/feature` |
| 禁止事项 | 第二、第三批不得扩张 `orchestrator-profiles.ts` 的 canonical 列表 | 防止顶层 profile 数量失控 |

## 八、最佳实践约束

| 约束 | 原则 | 含义 |
|------|------|------|
| 组织方式 | 按 workflow / stage 组织，而不是按角色宇宙组织 | 专家能力应附着到现有 skill 与阶段，而不是反客为主形成新系统 |
| 真源控制 | 保持单一流程真源 | `stage-state.json`、`gate`、`traceability-matrix.md`、`todo-state.json` 继续是唯一流程真源 |
| 复杂度控制 | 顶层稳定语义最少化 | 只有第一批 7 个 `canonical profile` 进入顶层 registry |
| 运行时边界 | 轻接入，重回退 | 任何 profile / fragment / guidance 未命中时，都必须安全退回现有行为 |
| 上下文边界 | 不为专家能力新建并行上下文系统 | 统一复用 `prompt-assembler`、`context-pack`、已有 CLI 上下文 |
| 交付方式 | 单路线、分阶段 | 先做核心治理入口，再做扩展入口，最后做命令侧体验增强 |

## 九、实施策略

| 层面 | 推荐做法 | 不推荐做法 |
|------|------|------|
| 路线选择 | 采用单一路线架构 | 同时维护多套候选架构 |
| 交付节奏 | 分阶段落地，每阶段有独立完成标准 | 把三批任务一次性合并成交付包 |
| 第一批重点 | 优先 `12-verify`、`08-review`、`04-design`，再进 `06-task`、`13-orchestrate` 的 runtime 接入 | 一开始就同时动 parser、orchestrate、全部 skill 文本 |
| 第二批重点 | 作为 `fragment` 附着层扩展治理深度 | 把第二批做成新的顶层 profile 集合 |
| 第三批重点 | 作为 `command-local guidance` 改善 CLI 可用性 | 试图让低上下文命令参与动态 profile 解析 |

| Phase 2 | 先改 `12-verify` 与 `08-review` | 尽快获得质量收益 |
| Phase 3 | 再建立 `fragment` 真源与 `skillName -> fragments` 附着 | 为第二、第三批铺底座 |
| Phase 4 | 再改 `04-design` 与 `13-orchestrate` | 提升设计与编排质量 |
| Phase 5 | 最后改 `06-task` 与 `03-spec`，并补第三批 `command-local guidance` | 收紧前置澄清与命令侧建议 |

## 十四、单期实施边界

| 问题 | 建议 |
|------|------|
| 是否单期做完三批 | 不建议 |
| 单期最合理覆盖范围 | 第一批 + 第二批基础设施，不含第三批大面积 CLI guidance |
| 单期必须完成的基础设施 | `orchestrator-profiles.ts`、`expert-fragments.ts`、`prompt-assembler.ts`、shared contract 覆盖 |
| 单期应避免的高风险组合 | 同时大改 `task parser`、`orchestrate CLI`、第三批多个命令输出 |
| 单期完成定义 | 第一批核心入口已能稳定命中 canonical profile，第二批 fragment 附着链路已建立，未命中时全部安全回退 |

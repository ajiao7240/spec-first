# 研发专家角色参考文档

> 日期：2026-03-15  
> 范围：`agency-agents` + `oh-my-opencode` 中所有偏研发相关角色  
> 目的：为 `spec-first` 后续的 `expert profiles`、`orchestrate` 协作模板、`review/verify` 专项增强提供统一参考真源

## 一、筛选边界与说明

| 维度 | 说明 |
|------|------|
| 纳入范围 | 软件研发直接相关角色，包括：架构、实现、安全、测试、运维、编排、技术文档、研发流程治理 |
| 排除范围 | 纯营销、销售、财务、品牌、公关、泛业务运营角色 |
| 来源差异 | `agency-agents` 偏“专家角色库 + 工作流模板”；`oh-my-opencode` 偏“多智能体编排系统 + 内建研发代理” |
| 使用原则 | 该文档用于“提取可借鉴专家语义”，不是要求把源项目完整移植到 `spec-first` |
| 接入目标 | 优先服务 `design` / `orchestrate` / `review` / `verify` / `wrap_up` 五类入口 |

## 二、推荐级别定义

| 级别 | 含义 | 适用判断 |
|------|------|----------|
| `P0` | 首批应接入 | 与 `spec-first` 当前主流程强相关，能直接增强设计评审、代码审查、验证或编排 |
| `P1` | 第二批增强 | 有明确价值，但依赖更多上下文、工具或专项场景 |
| `P2` | 备用/专项 | 只适用于特定项目类型、技术栈或成熟度更高的团队 |

## 七、对 `spec-first` 的接入映射建议

| `spec-first` 入口 | 推荐专家组合 | 目标 | 建议接法 |
|------|------|------|------|
| `design` | Prometheus + Software Architect + Oracle | 在设计前明确边界，在设计后补足架构审查 | 作为“规划/评审型 profile”挂在设计前后 |
| `task` / `plan` | Prometheus + Metis + Momus + Senior Project Manager | 让任务拆解更清晰、更可执行、更可验证 | 作为“任务质量增强模板” |
| `orchestrate` | Atlas + Agents Orchestrator + Project Shepherd | 提升复杂任务协作提示、handoff 模板和推进策略 | 只增强提示与交付要求，不新建第二套状态机 |
| `review` | Code Reviewer + Security Engineer + Oracle | 提升代码审查的风险识别和安全深度 | 作为按任务类型切换的 review profile |
| `verify` | Reality Checker + API Tester + Evidence Collector + Performance Benchmarker + Accessibility Auditor | 强化最终验证、证据化、专项验收 | 作为 verify 的多模板输出与检查清单来源 |
| `wrap_up` | Technical Writer + Test Results Analyzer + Jira Workflow Steward | 强化交付文档、验证摘要、流程记录 | 作为收口文档增强器 |

## 九、结论

| 结论项 | 判断 |
|------|------|
| `agency-agents` 最适合借什么 | 大量可复用的“研发专家人格 + 交付物导向模板” |
| `oh-my-opencode` 最适合借什么 | “规划-补漏-审查-执行编排”这一套研发代理分工语义 |
| 不建议借什么 | 两边完整的运行时实现、整套多代理基础设施、完整大而全角色宇宙 |
| 最佳落地方式 | 先把专家角色收敛成 `spec-first` 的少量 `expert profiles`，按任务类型切换提示词、交付物和审查维度 |
| 推荐节奏 | 先接 `P0`，再补 `P1`，`P2` 保持专项扩展 |

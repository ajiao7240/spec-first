## 1. 文档定位

### 1.1 目标

本手册用于指导 `spec-first` 在不重做 runtime 的前提下，把参考项目中的专家能力以“文档增强”的方式接入当前 skill 体系。

本文档关注的是：

- 专家能力接到哪些真实节点
- 各节点该接什么级别的专家角色
- 实施顺序如何安排
- 如何判断集成是否真正生效

### 1.2 非目标

本方案不做以下事情：

- 不复制 `agency-agents` 或 `oh-my-opencode` 的完整运行时
- 不新增“用户手动选专家”的独立机制
- 不引入第二套状态机
- 不把专家能力硬编码成新的 TypeScript 编排逻辑
- 不把固定命令、固定工具、固定阈值写成跨项目默认前提
- 不打破当前 `spec-first` 的阶段状态机、Gate、追溯体系、覆盖率体系与既有主流程能力

## 2. 核心原则

### 2.1 真实路径优先

只以当前真实 skill 路径为准。

这里必须区分两层概念：

- 物理目录：真实实施路径，当前就是 `03-spec`、`04-design`、`08-review`、`12-verify`、`13-orchestrate`
- 逻辑节点名：便于表达和讨论时使用的 `spec`、`design`、`review`、`verify`、`orchestrate`

实施时：

- 文档可以使用逻辑节点名表达
- 文件改造、reference 新增、render 验证必须落到真实编号目录

### 2.2 render 闭环优先

新增文档只有在以下条件之一满足时，才算真正生效：

1. `SKILL.md` 正文已直接内联新的专家规则或模板要求
2. render 组装逻辑已显式把该文档正文拼入最终 prompt

以下情况当前不能单独视为“已生效”：

- 仅在 `SKILL.md` 中提到文件名
- 仅新增 `references/*.md` 或 `experts/*.md`，但未把正文并入 prompt
- 仅把文档加入“参考清单”，但 render 输出里不可见

如果做不到这一点，只能称为“文档真源已建立”，不能称为“运行时已生效”。

### 2.3 分层专家化

不是所有节点都需要同等重量的专家人格。

本方案采用三层模型：

- 强专家节点：高判断密度，接主专家 + 辅专家
- 中专家节点：保留主判断视角，但不堆叠过多人格
- 轻引导节点：定义工作姿态，不引入重型 borrowed experts

### 2.4 文档增强优先于代码增强

第一阶段以：

- 专家文档
- references 模板
- render 可见性

为主，而不是先改 CLI/runtime 代码。

### 2.4.1 流程不变，节点增强

本方案的底线约束是：

> 不重构 `spec-first` 主流程，只增强 skill 节点能力。

这意味着：

- 不改阶段推进模型
- 不改 Gate 与覆盖率主机制
- 不改追溯 ID 与变更管理主机制
- 不改现有 `orchestrate` 的流程地位

允许改的范围仅包括：

- `SKILL.md` / render 模板的表达增强
- `references/*.md` 的检查清单、模板、停止条件增强
- `experts/*.md` 的共享能力文档
- 节点级的治理表达、证据表达、上下文表达增强

判断一项改造是否越界，可以用一句话测试：

> 这项改动是在增强某个 skill 节点的判断与输出，还是在重做 `spec-first` 的流程机制？

如果答案偏向后者，本轮不做。

### 2.5 禁止写死跨项目默认命令

例如以下内容不应被写成通用前提：

- `npm run test:api`
- `npm run test:integration`
- `npm run test:perf`
- 固定 Lighthouse 分数阈值
- 固定安全扫描工具名

正确做法是：

- 先从 `task_plan.md`、仓库脚本、CI 配置中发现现有命令
- 若不存在，再写入可复现的手动验证步骤

### 2.6 先盘点现有能力，再决定借鉴

跨工程借鉴的前提不是“别人的做法先进，所以直接搬”，而是先确认当前 `spec-first` 是否已经具备等价能力。

当前已在现有 skill / references 中落地、无需重复发明的能力包括：

- `research` 已具备 `2-Action Rule`、证据落盘、`findings.md` 同步约束
- `catchup` 已具备 `5-Question Reboot` 风格的上下文恢复机制
- `code` 已具备 TDD 守卫、RED / WAIVER / GREEN 证据约束
- `verify` 已具备 Gate、覆盖率、证据化记录与失败修复建议约束
- `design` / `verify` 已具备 Constitution / C11 合规相关约束
- `archive` 已具备 `break-loop` 风格复盘模板
- `first` 已具备 runtime-first 认知与 `steering` 类运行时产物

因此，后续借鉴必须遵守：

1. 先识别当前已有等价能力
2. 再判断是“补强”还是“新增”
3. 避免把已有规则换个名字再实现一遍

原因：

- 当前仓库已有等价或近似能力
- 本轮目标是增强 skill 能力，不是重构系统内核

## 4. 规划、分析、执行路线

这一节定义真正可执行的路线，确保“先分析、再规划、再执行、最后验证”，而不是直接写文档碎片。

### 4.1 Step A：先做能力盘点

对每个节点先回答 4 个问题：

1. 当前 skill 已经有哪些硬约束或 hooks？
2. 当前 references 已经覆盖了哪些规则？
3. 参考项目提供的是新能力，还是更好的表达方式？
4. 本次增强是“新增文档”、还是“增强现有 reference”？

输出：

- 节点能力盘点表
- 借鉴类型判断：`新增 / 补强 / 不采纳`

### 4.2 Step B：再做借鉴映射

对每个可借鉴点，统一映射为下列 4 类之一：

- 专家角色
- 检查清单
- 治理约束
- 上下文机制

只有映射完成后，才允许进入文档实施。

输出：

- 借鉴映射矩阵
- 节点到 reference 的落点清单

### 4.3 Step C：按节点分批实施

实施顺序必须按风险和收益排序：

1. 强专家节点
2. 中专家节点
3. 轻引导节点
4. 全局治理增强
5. 上下文与记忆增强

这样可以避免：

- 轻节点先做重
- 治理层缺失导致后续返工
- render 验证跟不上文档增长

### 4.4 Step D：每批实施都做 render 验证

每完成一批，不是检查“文件写没写”，而是检查：

- `spec-first skill render <node>` 是否看得到增强
- 是否破坏了现有 skill 的职责边界
- 是否引入了和已有规则冲突的新约束

### 4.5 Step E：最后再做真实 Feature 演练

真实演练至少应覆盖：

`spec -> research -> design -> task -> code -> review -> verify -> archive`

原因：

- 这条链路能验证专家角色、治理约束、上下文链是否协同
- 只测单一节点，无法证明方案真的可用

## 5. 专家分层模型

### 3.1 强专家节点

这些节点直接承担澄清、设计、审查、验收、编排等高判断密度工作：

- `spec`
- `research`
- `design`
- `task`
- `review`
- `verify`
- `orchestrate`

要求：

- 必须定义主专家
- 必须定义辅专家
- 必须定义停止条件
- 必须定义输出质量标准

### 3.2 中专家节点

这些节点承担质量门、只读分析、计划解释、实现约束、归档收口等次高判断工作：

- `plan`
- `spec-review`
- `analyze`
- `code`
- `archive`

要求：

- 定义主专家
- 可选接入 1-2 个辅专家
- 保持节点原有职责边界

### 3.3 轻引导节点

这些节点更适合定义为工作姿态，而不是完整专家人格：

- `init`
- `catchup`
- `status`
- `sync`
- `feature`
- `doctor`
- `first`
- `onboarding`

要求：

- 低噪声
- 强稳定
- 易恢复
- 不越界承担重判断任务

## 10. 全局治理增强

这一节描述的不是某个节点的专家人格，而是整个体系都应遵守的治理约束。

### 10.1 权威层级

建议采用以下优先级：

`Constitution > Spec > Design > Task > Code > Verify Evidence`

含义：

- Constitution 定义不可违背的全局原则
- Spec 定义需求边界与验收目标
- Design 定义技术方案与权衡
- Task 定义执行粒度与依赖关系
- Code 是实现结果，不得反向篡改上层契约
- Verify Evidence 是结果证明，不得伪造、推断或跳过

### 10.2 审查分层

建议在 `review` 和 `verify` 中统一采用三层检查：

1. 单点检查：当前文件/当前改动是否正确
2. 跨产物检查：是否与 spec / design / task / findings 一致
3. 完成检查：是否满足收口条件，是否仍有未闭环风险

### 10.3 证据纪律

建议将以下规则提升为全局约束：

- 无证据，不声称通过
- 无复现，不声称失败已定位
- 无明确边界，不推进到下游节点
- 无上游契约，不允许下游自行发挥

### 10.4 抗幻觉规则

建议明确：

- 不凭单一模糊线索生成高置信结论
- 不把“看起来合理”当成“已被验证”
- 不把角色人格包装成事实来源

## 11. 上下文与记忆增强

这一节用于补齐“轻节点为什么仍然重要”的设计理由。

### 11.1 最小上下文链

当前最小可行上下文链建议是：

`first -> catchup -> status -> findings.md -> 当前 skill`

含义：

- `first` 解决“项目整体认知”
- `catchup` 解决“当前 Feature 在哪”
- `status` 解决“当前阶段和下一步是什么”
- `findings.md` 解决“最近结论、阻塞、证据是什么”

### 11.2 关键结论落盘原则

建议在以下节点统一强化：

- `research`
- `design`
- `task`
- `review`
- `verify`

原则：

- 关键结论必须能落到 `findings.md` 或对应产出物
- 不允许只存在于会话上下文里
- 被引用的结论必须可追溯到文件位置或命令输出

### 11.3 轻引导节点的真实价值

轻引导节点不是“弱化版节点”，而是承担三件事：

1. 恢复上下文
2. 稳定入口
3. 压缩运行时噪声

因此它们不需要重型专家人格，但需要稳定的职责定义。

## 12. render 接入清单

这一节用于回答“谁来接、接到哪里、怎么验收”。

### 12.1 接入方式

每个节点的专家能力接入，当前实现下只应认定以下两种方式为“运行时已生效”：

1. 在对应节点的 `SKILL.md` 正文中直接加入规则、模板或约束
2. 修改 render 组装逻辑，把专家文档正文显式拼入最终 prompt

以下方式当前只能算“文档已存在”，不能单独算“运行时已生效”：

- 仅在 `references/` 中新增文件
- 仅在 `SKILL.md` 中增加文件链接或文件名
- 仅在 README、矩阵、清单里登记该文档

### 12.1.1 专家文档复用规则

同一专家在多节点使用时，采用“单一真源 + 节点补充”的模式：

- 专家定义：`experts/{expert-name}.md`
- 节点差异：在各节点自身的 `SKILL.md` 或 `references/*.md` 中补充场景约束
- 节点生效：必须把节点最终需要执行的规则写进 `SKILL.md` 正文，或通过 render 装配逻辑显式拼入 prompt

这意味着：

- 不在多个节点里复制粘贴整份专家文档
- 不为节点差异去改写共享专家真源
- 不把“节点里提到专家文件名”误当成 render 已生效

示例：

- `experts/architecture-review.md` 定义通用架构权衡与失败模式检查
- `04-design/SKILL.md` 写入 design 场景下必须执行的权衡与失败模式要求
- `08-review/SKILL.md` 只吸收与代码审查有关的架构风险判断，不复制完整 design 语境

### 12.2 节点级接入清单

| 节点 | 目标注入位置 | 最低要求 | 验证命令 | 通过标准 |
|---|---|---|---|---|
| `spec` | `03-spec/SKILL.md` 正文或 render 装配逻辑 | 可见澄清协议、缺口识别、AC 可测试性检查 | `spec-first skill render spec --feature <有效 feature>` | 输出含澄清/假设/停止条件 |
| `research` | `05-research/SKILL.md` 正文或 render 装配逻辑 | 可见证据分级、不猜测原则、待验证项 | `spec-first skill render research --feature <有效 feature>` | 输出含证据强弱或待验证标记 |
| `design` | `04-design/SKILL.md` 正文或 render 装配逻辑 | 可见权衡表、失败模式、待验证项 | `spec-first skill render design --feature <有效 feature>` | 输出含权衡/失败模式要求 |
| `task` | `06-task/SKILL.md` 正文或 render 装配逻辑 | 可见任务独立性、DoD、依赖关系 | `spec-first skill render task --feature <有效 feature>` | 输出含任务粒度/依赖约束 |
| `review` | `08-review/SKILL.md` 正文或 render 装配逻辑 | 可见模式矩阵、findings 结构 | `spec-first skill render review --feature <有效 feature>` | 输出含审查模式和 findings 要求 |
| `verify` | `12-verify/SKILL.md` 正文或 render 装配逻辑 | 可见证据要求、未证明规则 | `spec-first skill render verify --feature <有效 feature>` | 输出含证据链与未证明判断 |
| `orchestrate` | `13-orchestrate/SKILL.md` 正文或 render 装配逻辑 | 可见 handoff、回流、推进规则 | `spec-first skill render orchestrate --feature <有效 feature>` | 输出含 handoff / 回流条件 |

### 12.3 执行责任

实施时必须明确三类责任：

- 文档责任：谁维护 `experts/*.md` 和 `references/*.md`
- 模板责任：谁修改 render 模板或 skill 引用
- 验证责任：谁执行 `spec-first skill render <name>` 并确认输出可见

如果三类责任没有落到具体人或具体任务，方案在实践中会停留在“文档存在，运行时不生效”的状态。

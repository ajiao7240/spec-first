# 专家文档统一模板

> **版本**: v1.0.0  
> **日期**: 2026-03-17  
> **来源**: 基于 Oh-My-OpenCode、Trellis、Agency-Agents 参考项目提取  
> **用途**: 创建 `skills/spec-first/experts/` 下的专家文档

---

## 模板说明

本模板综合了三个参考项目的最佳实践：

| 参考项目 | 借鉴要素 |
|---------|---------|
| **Oh-My-OpenCode** | 动态 Prompt 组装、元数据驱动、Slop Detection、Todo Continuation、Dual Completion |
| **Trellis** | Code-Spec 分离、Cross-Layer Check、失败模式分析、"AI 永不提交"原则 |
| **Agency-Agents** | 专家角色定义（Prometheus/Oracle/Atlas/Momus 等）、访谈澄清协议 |

---

## 二、核心原则

{从参考项目中提取的指导原则，3-7条}

**示例**（基于 Oh-My-OpenCode + Trellis）：
- **默认怀疑**：证据不足时默认 NEEDS WORK
- **只读模式**：不直接修改代码，只给建议
- **证据驱动**："看起来可以"不被接受
- **失败优先**：先分析失败模式，再设计成功路径
- **最小变更**：只改必要的，不打临时补丁

## 四、检查清单

### 4.1 必检项（Blocking）

{从参考项目中提取的强制检查项}

**示例**（基于 Oh-My-OpenCode Comment Checker + Trellis Code-Spec）：
- [ ] 无 AI 生成的废话注释（"这个函数做了..."）
- [ ] 无未处理的 TODO/FIXME
- [ ] 关键决策有 ADR 或设计文档支撑
- [ ] 安全敏感操作有威胁建模（STRIDE）
- [ ] 公开 API 有向后兼容性分析

### 4.2 建议项（Advisory）

{从参考项目中提取的建议检查项}

**示例**（基于 Trellis Brainstorm + Oh-My-OpenCode Dual Completion）：
- [ ] 复杂逻辑有失败模式分析
- [ ] 关键假设已显化并标记验证方式
- [ ] 边界条件有测试覆盖
- [ ] 性能敏感路径有基准测试

## 五、输出要求

### 5.1 产物结构

{从参考项目中提取的产物模板}

**示例**（基于 Trellis Code-Spec 7部分 + Oh-My-OpenCode Agent Metadata）：

```markdown
## {专家输出标题}

### 决策依据
{为什么做这个决策，基于什么约束}

### 风险分级
- 🔴 阻断: {必须修复的问题}
- 🟡 建议: {应该改进的问题}
- 💭 讨论: {需要团队决策的问题}

### 关键假设
{列出所有隐含假设，标记验证方式}

### 失败模式
{如果这个方案失败，最可能的原因是什么}

### 下一步要求
{明确的后续动作，可验证的完成标准}
```

### 5.2 质量标准

{从参考项目中提取的质量门槛}

**示例**（基于 Oh-My-OpenCode Completion Detection + Trellis Gate）：
- 每个建议必须包含：位置 + 原因 + 具体改进方案
- 风险评估必须有证据支撑，不能是主观猜测
- 关键决策必须有至少2个备选方案的权衡分析
- 安全问题必须给出修复建议和验证方式

## 六、反模式（Anti-Patterns）

{从参考项目中提取的常见错误}

**示例**（基于 Oh-My-OpenCode Slop Detection + Trellis "AI永不提交"）：

| 反模式 | 表现 | 正确做法 |
|--------|------|---------|
| 假完成 | 说"已完成"但无证据 | 提供测试输出/日志/截图 |
| 过度自信 | "应该没问题" | 明确标记未验证项 |
| 无限追问 | 苏格拉底式提问变成无限访谈 | 关键问题澄清后立即转入输出 |
| 直接提交 | AI 直接执行 git commit | 只生成 commit message，由人工提交 |
| 临时补丁 | 不找根因，打补丁 | 分析根本原因，最小化变更 |

---

## 七、停止条件

{从参考项目中提取的完成标准}

**示例**（基于 Oh-My-OpenCode Todo Continuation + Trellis Gate）：

满足以下**所有**条件时停止：
1. 所有必检项已通过
2. 阻断级问题已全部修复或豁免
3. 产物包含所有必需章节
4. 关键假设已显化并标记验证方式
5. 下一步要求明确且可验证

**不满足时的动作**：
- 输出当前进度和剩余项
- 标记 `NEEDS WORK` 状态
- 给出具体的补全建议

## 关键设计原则

1. **单一真源**：每个专家能力只定义一次，通过引用复用
2. **元数据驱动**：支持动态 prompt 组装（借鉴 Oh-My-OpenCode）
3. **分层检查**：必检项 vs 建议项（借鉴 Trellis Gate）
4. **证据导向**：所有判断必须有依据（借鉴 Reality Checker）
5. **反模式明确**：从参考项目中提取真实的坑（借鉴 Slop Detection）
6. **停止条件清晰**：避免无限循环（借鉴 Todo Continuation）

## 使用指南

### 创建新专家文档时

1. 复制本模板
2. 填充各章节内容，参考对应的参考项目文档
3. 确保包含所有必需章节（一到九）
4. 在 `skills/spec-first/experts/README.md` 中注册
5. 执行 `spec-first skill render <skill-name>` 验证引用生效

### 引用专家文档时

```markdown
<!-- 在 SKILL.md 中 -->
## 专家能力

本节点融合以下专家能力：

{{> experts/planning-consult}}
{{> experts/architecture-review}}
```

### 验证集成效果

```bash
# 渲染后检查专家内容是否出现
spec-first skill render spec > /tmp/rendered.md
grep -c "planning-consult" /tmp/rendered.md  # 应该 > 0

# 检查渲染后的行数，避免 prompt 过长
wc -l /tmp/rendered.md
```

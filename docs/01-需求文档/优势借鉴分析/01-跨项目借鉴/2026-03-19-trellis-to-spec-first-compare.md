# Trellis → Spec-First 可借鉴能力对照表

> 目的：把 Trellis 里值得借鉴的设计，按 **直接可借 / 需改造 / 不建议借** 三类映射到 Spec-First，避免把“安装器型产品”的方案原样搬到“研发流程 skill 集合”里。

## 1. 结论摘要

Trellis 最值得借鉴的不是某个 CLI 命令，而是它把以下几件事做成了产品能力：

- 单一真源 + 派生注册表
- 模板资产版本化
- 更新时尊重用户改动
- 并行 worktree / 多 agent 执行
- 会话回放与上下文恢复
- 契约测试锁定系统边界

这些能力对 Spec-First 都有价值，但需要按产品定位重映射：

- Spec-First 不是安装器，不需要照搬 Trellis 的“分发到多平台目录”的模型
- Spec-First 更像一套研发流程 skill 集合，应该优先借鉴其 **版本化、派生化、回放化、并行化** 的思想

---

## 2. 直接可借

### 2.1 单一真源，全部派生

**Trellis 做法**
- 用一个 registry 作为单一真源，其他平台目录、命令、模板集合都从 registry 派生。

**Spec-First 可借点**
- `truth-source.ts` 应继续作为阶段 / skill / route 的单一真源。
- 所有门禁、路由、辅助索引、建议命令都应该派生自真源，而不是散落在多个文档里各写各的。

**收益**
- 减少重复定义
- 降低文档与实现分叉
- 让新增 skill / gate 有明确入口

### 2.2 模板资产版本化

**Trellis 做法**
- workflow、spec、commands、hooks 作为模板资产打包交付。

**Spec-First 可借点**
- `skills/spec-first/*` 可以被视作“可版本化的 skill 资产包”。
- `runtime`、`docs`、`records` 的边界应保持稳定，但 skill 文本、流程说明、门禁说明可以版本演进。

**收益**
- 有利于发布、回滚、迁移
- 有利于对比不同版本的 skill 行为

### 2.3 契约测试锁边界

**Trellis 做法**
- 测试直接锁定 registry 唯一性、路径冲突、迁移逻辑、模板完整性。

**Spec-First 可借点**
- 用契约测试锁定：
  - `truth-source`
  - gate 条件是否与文档一致
  - runtime / docs / records 的职责边界
  - `docs-index.json` 是否只做辅助索引

**收益**
- 防止门禁语义漂移
- 防止文档口径与代码口径再次分叉

### 2.4 会话 / 运行态回放

**Trellis 做法**
- 通过 journal / index 的形式把会话摘要持久化，供后续恢复上下文。

**Spec-First 可借点**
- `findings.md`、`gate-history.jsonl`、`stage-state.json` 本质上就是类似的“外置记忆”。
- 这套记录机制应该继续保留，并进一步强调“可回放、可追责、可恢复”。

**收益**
- 中断后容易恢复
- 审查时有明确证据链
- AI 上下文更稳定

---

## 3. 需改造后借

### 3.1 并行 worktree / 多 agent 执行

**Trellis 做法**
- 把并行任务和 worktree 作为产品内建能力。
- 通过工作流和配置，直接调度多个 agent 在独立 worktree 里执行。

**Spec-First 对应思路**
- 可以借“并行 + worktree”的思想，但不能照搬 Trellis 的安装器语义。
- 对 Spec-First 来说，更合理的是：
  - 让高风险改动默认建议 worktree
  - 把并行任务、调度和恢复做成明确的 skill / gate 策略
  - 不把多 agent 的组织方式写成一个安装到用户目录的配置复制系统

**改造重点**
- 并行是执行策略，不是模板分发策略
- worktree 是隔离手段，不是产品主目标

### 3.2 更新时尊重用户修改

**Trellis 做法**
- 通过 hash / migration / conflict action 管理模板更新，避免粗暴覆盖。

**Spec-First 对应思路**
- 适合借鉴到：
  - `docs/first`
  - runtime 认知资产
  - skill 文本更新
  - 将来可能的门禁 / 记录模板更新

**改造重点**
- 不能把用户手写内容和系统生成内容混为一谈
- 更新时要识别“机器可重建”与“用户手改”

**收益**
- 降低误覆盖风险
- 更符合研发团队对知识资产的保护需求

### 3.3 路由 / 模板 / 配置的三层分离

**Trellis 做法**
- 平台注册表、模板层、配置层分开。

**Spec-First 对应思路**
- 可以拆成：
  - `truth-source`：单一真源
  - `runtime`：机器上下文
  - `docs`：人类阅读
  - `records`：审计记录

**改造重点**
- 继续保持三层或四层分离
- 不要把“阅读索引”抬升成真源

---

## 4. 不建议借

### 4.1 安装器 / 分发器心智

**Trellis 特征**
- 主要面向“把模板/命令/工作流安装到用户项目里”。

**为什么不建议照搬**
- Spec-First 不是安装器，不需要复制 Trellis 的平台目录分发模型。
- Spec-First 的价值在于研发流程治理，不在于多平台配置发放。

### 4.2 过重的模板复制机制

**Trellis 特征**
- 大量模板复制、迁移、冲突处理逻辑。

**为什么不建议照搬**
- Spec-First 主要是 skill / runtime / docs / records 的流程系统。
- 过重的复制机制会增加维护成本。

### 4.3 过多面向最终用户的工具目录

**Trellis 特征**
- 目录结构和命令体系非常面向终端用户安装和使用。

**为什么不建议照搬**
- Spec-First 更适合让用户接触少量高层 skill。
- 内部实现细节应尽量收敛在 runtime、records、docs 这三层。

---

## 5. 对 Spec-First 的落地建议

### 5.1 可直接采用的原则

- 单一真源，派生所有门禁与路由
- 版本化 skill / gate 文本
- 契约测试锁边界
- 外置记忆支持回放与恢复
- worktree 作为高风险改动隔离策略

### 5.2 需要改造后采用的原则

- 并行 agent 调度
- 更新时保留用户改动
- 目录与配置分层

### 5.3 不建议采用的原则

- 把 Spec-First 做成安装器型产品
- 把模板分发机制做成主架构
- 把终端用户导向大量内部目录和复制逻辑

---

## 6. 优先级建议

### P0

1. **单一真源 + 派生注册表**
   - 这是最值得直接借的能力。
2. **契约测试**
   - 防止流程语义再次漂移。

### P1

1. **外置记忆 / 回放**
   - 继续强化 findings / gate-history / stage-state 的闭环。
2. **用户改动保护**
   - 更新运行态产物时不要粗暴覆盖。

### P2

1. **并行 worktree / 多 agent**
   - 借思想，不照搬实现。
2. **模板版本化**
   - 仅限 skill / runtime / docs 的可版本化层。

---

## 7. 结论

Trellis 最值得借鉴的，不是“多一套 CLI 命令”，而是它把 **流程、模板、迁移、并行、回放、契约测试** 做成了一个产品闭环。

对 Spec-First 来说，最合适的借鉴方式是：

- 借“单一真源 + 派生”这一类架构原则
- 借“版本化 + 迁移 + 契约测试”这一类工程原则
- 借“worktree + 多 agent”这一类执行策略
- 不借“安装器型分发系统”的产品定位

一句话总结：

**Trellis 可以提供方法论，但 Spec-First 的产品边界应该更窄、更聚焦：它是一套研发流程 skill 集合，而不是一套安装到全平台的工作流分发器。**

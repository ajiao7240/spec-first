## A1 — PreToolUse Hook 注意力刷新

**来源**: Planning-with-Files §1（Manus Principle 4: Manipulate Attention Through Recitation）

**解决问题**: Risk 3（上下文恢复不稳定）+ Gap 4（Skill 联调一致性）

**原理**: AI 模型存在"lost in the middle"效应——上下文中间的信息容易被忽略，而末尾的信息获得最多注意力。通过在每次工具调用前自动读取计划文件，目标和当前阶段被推到注意力窗口的末尾，保持聚焦。

**落地位置**: `.claude/settings.json` — 追加到现有 `PreToolUse` 数组中（与 gate check 并存）

**适用范围**:
- Claude Code：按本节直接落地
- Codex：本轮先不做宿主 Hook 注入，使用 A2/A3 的 Skill 约束补偿（待 P1 再评估统一 Hook 适配）

**具体内容**:

在 `.claude/settings.json` 的 `hooks.PreToolUse` 数组中追加一条（不替换现有 gate check）：

```json
{
  "matcher": "write|edit|create",
  "hooks": [
    {
      "type": "command",
      "command": "sh -c 'FEAT=$(head -1 .spec-first/current 2>/dev/null); if [ -n \"$FEAT\" ] && [ -f \"specs/$FEAT/task_plan.md\" ]; then echo \"=== TASK 上下文刷新 ===\"; head -30 \"specs/$FEAT/task_plan.md\"; fi'"
    }
  ]
}
```

> 合并策略：追加到 `PreToolUse` 数组末尾，现有 gate check 条目保持不变。matcher 使用小写并与当前实现一致（`write|edit|create`）。
> 兼容性说明：`task_plan.md` 可以是 YAML Front Matter + 正文结构；该 Hook 仅用于“注意力刷新”，不依赖固定行号语义。

**刷新内容应包含**:
- 当前 TASK ID + 标题
- 关联的 FR/DS/TASK traces
- AC（验收标准）
- 当前阶段状态

**验收标准**: 每次 Write/Edit/Create 操作前，AI 上下文末尾出现当前 TASK 摘要

**工作量**: 0.5 天

## A2 — `/spec-first:code` 反合理化表 + Read/Write 决策矩阵

**来源**: Superpowers §1（Anti-Rationalization）+ §7（Systematic Debugging）+ Planning-with-Files §2（Filesystem as External Memory）

**解决问题**: Risk 4（绕流程）+ Risk 3（上下文不稳定导致信息丢失）

**落地位置**: `skills/spec-first/07-code/SKILL.md`

### 追加内容 ① — 反合理化守卫

在 SKILL.md 中新增 `## 反合理化守卫` 段落：

```markdown
## 反合理化守卫

当你产生以下念头时，立即停止并回到流程：

| AI 的借口 | 封堵 |
|-----------|------|
| "这个改动太小，不需要走 code-review" | 小改动也有回归风险，review 耗时 < 2 分钟 |
| "我已经手动检查过了" | 手动检查 ≠ 自动校验证据 |
| "先写完再补测试" | 事后测试证明不了什么，TDD 先行 |
| "这只是重构，不影响功能" | 重构不改行为 ≠ 重构不引入 bug |
| "快速修一下，之后再调查" | 快速修复掩盖根因，系统化调试更快 [SP §7] |
| "我看到问题了，让我直接修" | 看到症状 ≠ 理解根因 [SP §7] |
| "同时改多处，一起测试" | 无法隔离哪个改动有效，会引入新 bug [SP §7] |
| "再试一次修复"（已失败 2+ 次） | 3 次失败 = 架构问题，停止修复，质疑设计 [SP §7] |
| "我记得刚才看到的内容" | 上下文会被压缩，记忆不可靠，写入文件 |
```

### 追加内容 ② — Read/Write 决策矩阵

在 SKILL.md 中新增 `## 上下文持久化规则` 段落：

```markdown
## 上下文持久化规则

Context Window = RAM（易失、有限），Filesystem = Disk（持久、无限）。
任何重要的东西都写入磁盘。

| 场景 | 动作 | 原因 |
|------|------|------|
| 刚写完一个文件 | 不要读 | 内容还在上下文中 |
| 查看了图片/PDF/网页 | 立即写 findings.md | 多模态内容压缩时丢失 |
| 浏览器/MCP 返回数据 | 写入文件 | 外部数据不持久 |
| 开始新 TASK | 读 task_plan.md + findings.md | 重新定向上下文 |
| 发生错误 | 读相关文件 | 需要当前状态来修复 |
| 间隔后恢复 | 读所有运行态文件 | 恢复状态 |

压缩必须可恢复：即使丢弃了内容，也要保留 URL / 文件路径 / ID 指针。
```

**验收标准**:
1. code skill 执行时，AI 遵循完整流程（code-review、TDD、根因分析后再修复），可通过抽查至少 5 次 code skill 执行记录验证
2. 研究发现、技术决策、API 响应样本等及时持久化到 findings.md（每次 code skill 执行后 findings.md 有对应更新）

**工作量**: 0.5 天

## A3 — `/spec-first:verify` 证据铁律 + Common Failures 表

**来源**: Superpowers §2（Iron Law: NO COMPLETION CLAIMS WITHOUT FRESH VERIFICATION EVIDENCE）

**解决问题**: Gap 1（M3 GateEngine 自动条件解析未闭合）+ Risk 2（Gate 缺口导致误放行）

**落地位置**: `skills/spec-first/12-verify/SKILL.md`（主）+ `skills/spec-first/13-orchestrate/SKILL.md`（阶段推进时引用）

### 追加内容 ① — 证据铁律（五步 Gate Function）

在 SKILL.md 中新增 `## 证据铁律` 段落：

```markdown
## 证据铁律

铁律：声称任何阶段通过前，必须提供新鲜的验证证据。

五步 Gate Function：
1. **IDENTIFY** — 什么命令能证明这个声明？
2. **RUN** — 执行完整命令（新鲜的、完整的）
3. **READ** — 完整输出，检查退出码，计数失败项
4. **VERIFY** — 输出是否确认了声明？
5. **ONLY THEN** — 发出声明

禁止使用的表述：
- "should pass" / "looks good" / "已完成" / "我检查过了"
- "应该没问题" / "上一轮通过了" / "差不多了"
```

### 追加内容 ② — Spec-First Common Failures 表

```markdown
## Common Failures 表

| 声明 | 需要的证据 | 不充分的证据 |
|------|-----------|-------------|
| Gate 通过 | `spec-first gate check <featureId>` 输出: PASS | "我检查过了"、"应该没问题" |
| 覆盖率达标 | `spec-first metrics coverage <featureId>` 输出: C1-C9 ≥ 阈值 | "所有 FR 都有对应 TASK" |
| 阶段可推进 | `spec-first gate check <featureId>` 输出 PASS + `spec-first matrix check <featureId>` 退出码 0 | "上一轮通过了" |
| TASK 完成 | 测试命令输出 + code-review 通过 | "代码写完了" |
| Feature 可归档 | `spec-first gate check <featureId>` + `spec-first matrix check <featureId>` + 归档产物证据 | "所有 TASK 都标记完成了" |
```

**补充要求**: 在 Gate 输出中展示"判定证据链"——失败条目映射到具体 ID + 修复建议（由 Skill 约束实现，不是 CLI 强制）。

**验收标准**: AI 在声称任何 Gate/阶段通过时，必须贴出对应命令的完整输出

**工作量**: 0.5 天

## A5 — Description Trap 巡检（19 Skill）

**来源**: Superpowers §8（v4.0.0 经验：description 写了流程摘要 → 模型按描述执行，不读正文）

**解决问题**: Gap 4（Skill 定义完成但执行一致性不足）

**原理**: 若 skill 的 front matter `description` 写了流程摘要，模型会直接按描述执行，而不读完整 SKILL 正文。这导致阶段守卫、确认策略、证据链要求被绕过。

**落地位置**: 全部 19 个 SKILL.md 的 Front Matter `description` 字段

### 规范定义

**description 字段只写触发条件（"Use when X"），禁止写执行流程、策略摘要、捷径提示。**

| 类型 | 示例 | 判定 |
|------|------|------|
| 合规 | `"定位 Feature 并校验阶段为技术设计（02_design）"` | ✅ 触发条件 |
| 合规 | `"定位进行中的 TASK 并执行代码实现"` | ✅ 触发条件 |
| 违规 | `"先读 spec，再生成 design，最后校验一致性"` | ❌ 流程摘要 |
| 违规 | `"加载上下文后按 P0-P5 六阶段执行"` | ❌ 策略摘要 |
| 违规 | `"快速模式：跳过 P3 确认直接生成"` | ❌ 捷径提示 |

### 执行方式

逐一检查 19 个 Skill 的 description 字段：

## A6 — "Spirit vs Letter" 原则 🔴 v1.4 新增

**来源**: Superpowers §11（v4.3.1 源码验证确认）

**原理**: Superpowers 多个核心 skill 中明确声明："Violating the letter of these rules is violating the spirit of these rules." 这直接封堵了一整类合理化——"我理解这个规则的核心思想，所以可以灵活调整执行方式"。字面规则的违反就是精神的违反，不存在"精神上遵守但字面上变通"的说法。

**落地位置**: `skills/spec-first/03-spec/SKILL.md` + `07-code/SKILL.md` + `12-verify/SKILL.md` + `04-design/SKILL.md`（所有 P0 级别 skill）

### 追加内容

在 4 个核心 skill 开头的原则声明部分加入：

```markdown
## 字面即精神原则

**Violating the letter of these rules is violating the spirit of these rules.**
```

配套反合理化表条目：

| AI 的借口 | 封堵 |
|-----------|------|
| "我理解核心思想，可以灵活执行" | 字面规则的违反就是精神的违反，不存在灵活变通 |
| "这是精神而非仪式" | 仪式（字面规则）是精神的体现，跳过仪式就是违背精神 |

| "具体情况具体分析" | 规则已考虑常见情况，例外需明确讨论而非自行变通 |

**验收标准**: AI 在 skill 执行时遵循字面规则，不使用"精神上遵守但字面上变通"的借口（可通过抽查至少 5 次 skill 执行记录验证）

**工作量**: 0.5 天（4 个 skill 文件）

## A7 — Stop Hook 完成度守门 🔴 v1.4 新增

**来源**: Planning-with-Files §4（v2.16.0 源码验证确认）

**解决问题**: Risk 2（Gate 缺口导致误放行）+ Gap 1（GateEngine 自动条件解析未闭合）

**原理**: PwF 在 Stop Hook 中运行 `check-complete.sh` 脚本，当 AI 尝试结束会话时自动检查 `task_plan.md` 中所有 TASK 的完成状态。如果有未完成的 TASK，通过 `followup_message` 机制触发自动继续。这是一种"软守门"——让 AI 自己看到证据后做出正确判断。

**落地位置**: `.claude/settings.json` — Stop Hook 配置

**具体内容** (Prompt-based 方案，推荐):
```json
{
  "Stop": [{
    "matcher": "*",
    "hooks": [{
      "type": "prompt",
      "prompt": "在结束会话前，检查 task_plan.md 中的 TASK 完成状态。如果有未完成的 TASK，输出未完成的 TASK ID 列表并继续工作。"
    }]
  }]
}
```

**Command-based 备选方案**:
```yaml
Stop:
  - matcher: "*"
    hooks:
      - type: command
        command: |
          FEAT=$(head -1 .spec-first/current 2>/dev/null)
          if [ -n "$FEAT" ] && [ -f "specs/$FEAT/task_plan.md" ]; then
            COMPLETE=$(grep -c "Status.*complete" "specs/$FEAT/task_plan.md" || echo 0)
            TOTAL=$(grep -c "TASK-" "specs/$FEAT/task_plan.md" || echo 0)
            if [ "$COMPLETE" -lt "$TOTAL" ]; then
              echo "⚠️ TASK 未完成: $COMPLETE/$TOTAL，请继续完成剩余 TASK"
            fi
          fi
```

**验收标准**: AI 结束会话前自动检查 TASK 完成度，防止提前收工

**工作量**: 0.5 天

## T1 — 模板哈希 + 变更分级更新（Trellis 借鉴，P0）

**来源**: Trellis `src/utils/template-hash.ts` + `src/commands/update.ts`（`analyzeChanges` 分类逻辑）

**解决问题**: 当前 `spec-first update` 更像“宿主刷新器”，无法区分“模板升级”与“用户手改”，存在误覆盖风险。

**落地位置**:
- `src/core/update-engine/hash.ts`
- `src/core/update-engine/diff.ts`
- `src/cli/commands/update.ts`
- `.spec-first/.template-hashes.json`

**核心机制**:
1. 初始化托管文件哈希注册表（首批覆盖 `.spec-first` 与 `skills` 相关受管产物）。
2. 每次 update 对文件做三态分类：
   - `unchanged`（当前内容等于目标模板）
   - `auto-update`（当前哈希命中历史模板，可安全升级）
   - `user-modified`（当前哈希不在模板历史，进入冲突决策）
3. `--dry-run` 输出分类结果，不直接改写文件。

**验收标准**:
- `spec-first update --dry-run` 能按文件输出分类和处理策略。
- 用户手改文件默认不自动覆盖。
- 单元测试覆盖 hash 读写、分类判定和异常场景。

**工作量**: 1.5 天

## T2 — Manifest 迁移引擎 + 冲突策略（Trellis 借鉴，P0）

**来源**: Trellis `src/migrations/index.ts` + `src/migrations/manifests/*.json`

**解决问题**: 版本升级缺乏可审计迁移编排，无法稳定处理 rename/delete/目录迁移等变更。

**落地位置**:
- `src/migrations/index.ts`
- `src/migrations/manifests/*.json`
- `src/core/update-engine/migrations.ts`
- `src/cli/commands/update.ts`

**核心机制**:
1. 引入 `.spec-first/.version` 记录项目资产版本。
2. 迁移 manifest 描述动作（`rename` / `delete` / `rename-dir` / `copy-if-missing`）。
3. 执行前先 classify，再走冲突策略：
   - `auto`: 自动执行
   - `confirm`: 需要确认（CLI 可设 `--skip-all` 或 `--create-new`）
   - `conflict`: 生成冲突报告并保留现场
4. 补齐 CLI 参数：`--migrate --force --skip-all --create-new --allow-downgrade`。

**验收标准**:
- 至少 2 个历史版本升级样例可复现并通过。
- rename/delete 场景中，用户改动不被静默覆盖。
- 迁移报告包含“执行项/跳过项/冲突项”统计。

**工作量**: 2 天

### B1 — 新鲜上下文隔离（Fresh Context Per Task）

**来源**: Superpowers §6

**解决问题**: Risk 3（上下文恢复不稳定）+ 多 TASK 执行时的上下文污染问题

**原理**: 每个 TASK 启动时只接收必要的上下文，不传递前一个 TASK 的完整执行历史。优势：
- 无上下文污染（前一个 TASK 的错误假设不会传播）
- 控制器精确策展上下文（而非让 subagent 自己摸索）
- 并行安全（subagent 之间不干扰）

**落地位置**: `skills/spec-first/13-orchestrate/SKILL.md`

**具体做法**:

在 orchestrate skill 的 P1_CONTEXT 阶段增加上下文裁剪规则：

```markdown
## 上下文裁剪规则（Fresh Context Per Task）

为每个 TASK 启动全新的 subagent 时，只提供：

1. **当前 TASK 全文**：从 task_plan.md 提取当前 TASK 的完整描述
2. **相关 spec 片段**：仅与当前 TASK traces 关联的 FR/NFR
3. **必要的设计上下文**：仅相关的 DS/API 定义
4. **不传递**：
   - 前一个 TASK 的完整执行日志
   - 其他 TASK 的调试信息
   - 无关的 spec/design 章节
```

**验收标准**:
- 多 TASK 并行执行时，每个 TASK 的 subagent 不会看到其他 TASK 的上下文
- Context Pack 的 control 段控制在 2KB 以内

**工作量**: 1 天（orchestrate skill 修改 + Context Pack 优化）

### B2 — 用户故事组织的任务分解

**来源**: Spec-Kit §6

**解决问题**: Gap 10（TASK 缺少并行性标注）+ Risk 11（任务按技术层组织导致无法独立交付）

**原理**: Spec-Kit 的任务分解以**用户故事**为组织单元，而非技术层。每个 US 是一个可独立交付的 MVP 增量，而非"先做完所有后端再做前端"。

**落地位置**: `skills/spec-first/06-task/SKILL.md` + task_plan.md 模板

**具体做法**:

1. **TASK 条目增加并行标记**：
```markdown
- [ ] TASK-001 [P] [US1] 创建 Album 模型
- [ ] TASK-002 [P] [US1] 实现 Album CRUD API
- [ ] TASK-003 [US1] 添加拖放排序功能
```

2. **task_plan.md 按 Phase 分组**：
```markdown
## Phase 1: Setup（基础设置）
## Phase 2: User Stories - P1（核心价值）

### US1 — Photo Album Management (P1)
- [ ] TASK-001 [P] [US1] Create Album model
- [ ] TASK-002 [P] [US1] Implement Album CRUD API

### US2 — Photo Tile Preview (P2)
- [ ] TASK-003 [US2] Create Photo model
```

**标记说明**:
- `[P]` = 可并行执行（不依赖其他任务）
- `[US#]` = 所属用户故事（可独立交付和测试）

**验收标准**:
- TASK 条目包含 `[P]` 并行标记
- task_plan.md 按 Phase 分组（Setup / Foundational / User Stories / Polish）
- `orchestrate --auto` 能识别 `[P]` 标记并调度并行 TASK

**工作量**: 1 天（task skill Prompt 修改 + task_plan 模板更新）

## 实施原则

1. **双轨实施**: A 轨先稳行为纪律，T 轨再补工程韧性，避免“有流程没恢复”或“有工具没纪律”。

3. **可恢复优先于可升级**: T3 先于大规模迁移，先保证失败可回滚。
4. **渐进验证**: 先在 Daily Path（plan → code → code-review → verify）试点，再推广全阶段。
5. **量化反馈**: 同时跟踪流程合规率（C7/C8/C9）与升级可靠性指标（冲突率、回滚成功率、升级失败率）。

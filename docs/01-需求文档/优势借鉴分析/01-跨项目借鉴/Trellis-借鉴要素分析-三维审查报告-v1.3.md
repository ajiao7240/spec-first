## 三、性能影响审查

### 3.1 性能风险矩阵

| 要素 | 风险点 | 影响程度 | 缓解措施 |
|------|--------|---------|---------|
| **JSONL 分层注入** | Token 消耗 + I/O 延迟 | **高** | 内存缓存、异步预加载、摘要注入 |
| **Hook 自动化** | 命令执行延迟 | **中** | Hook 异步化、性能预算 |
| **Brainstorm 九步流程** | 交互轮次增加 | **中** | 复杂度分流、快速通道 |
| **Cross-Layer Check** | 验证时间增加 | **低** | 按需触发（3+ 模块变更） |
| **Skill 数量增长** | 路由表膨胀 | **极低** | 现有路由已支持 22 个，增量可控 |

### 3.2 Token 消耗分析

| 机制 | Token 增量 | 触发频率 | 风险等级 |
|------|-----------|---------|---------|
| **JSONL 分层注入** | 每次子代理 +500-2000 tokens | 每个子代理启动 | **高** |
| **Brainstorm 九步流程** | 每轮 +200-500 tokens | 复杂任务需求阶段 | **中** |
| **Code-Spec 深度规则** | +1500-3000 tokens | Infra/跨层变更 | **中** |

**Token 预算建议**:

| 任务复杂度 | 预算增量 | 分配 |
|-----------|---------|------|
| Trivial | +0 | 跳过所有增强 |
| Simple | +500 | 轻量 JSONL 注入 |
| Moderate | +2000 | JSONL + 轻量 Brainstorm |
| Complex | +5000 | 完整九步 + Code-Spec 深度 |

### 3.3 用户体验影响

| 操作 | 当前延迟 | 增加延迟 | 用户感知 |
|------|---------|---------|---------|
| 子代理启动 | 1-2s | +0.3-0.5s | 轻微，可接受 |
| Hook 执行 | <0.1s | +0.05-0.1s | 不可感知 |
| 需求澄清（Complex） | 5-10 轮 | +2-3 轮 | 明显，但价值高 |

### 3.4 性能优化建议

#### 短期（Week 1-2）
1. **JSONL 注入优化**: 实现内存缓存（Map + TTL）、异步预加载
2. **Hook 性能监控**: 添加性能埋点，设置性能预算（<200ms）
3. **复杂度分流**: 明确 Trivial/Simple 判断规则，实现快速通道

#### 中期（Week 3-4）
4. **Token 预算管理**: 实现上下文预算分配器，动态裁剪
5. **懒加载机制**: debug.jsonl 只在首次错误时加载
6. **并行化优化**: 独立 Hook 并行执行

### 3.5 实施优先级矩阵（性能调整版）

| 要素 | 性能影响 | 业务价值 | 实施建议 | 前置条件 |
|------|---------|---------|---------|---------|
| Code-Spec vs Guide 分离 | 极低 | 极高 | **立即实施** | 无 |
| 任务复杂度分类 | 极低 | 高 | **立即实施** | 无 |
| Onboard 引导 | 极低 | 中 | **立即实施** | 无 |
| Break-Loop 调试闭环 | 低 | 高 | **Week 2** | 无 |
| Code-Spec 深度规则 | 低 | 高 | **Week 2** | 无 |
| Cross-Layer Check | 中 | 中 | **Week 3** | 性能监控就绪 |
| Brainstorm 九步流程 | 中 | 高 | **Week 3** | 复杂度分流就绪 |
| **JSONL 分层注入** | **高** | 极高 | **Week 3** | **缓存机制就绪** |
| Meta/Local 分离 | 低 | 低 | **暂缓** | P0/P1 验证完成 |

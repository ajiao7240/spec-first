### 1.1 未集成要素详细功能说明（U1-U5）

#### U1 Hash 锚定编辑（Hashline Edit）
- 功能目标：在 AI 多轮改写过程中，稳定定位并修改“指定内容块”，降低错改/漏改风险。
- 关键能力：为目标段生成稳定锚点（hash/指纹）；编辑前先校验锚点是否仍匹配；不匹配时触发重定位而非盲写。
- 触发时机：跨文件批量改动、多人并发改动后继续编辑、长会话恢复后继续编辑。
- 预期收益：减少“改错位置”“覆盖他人改动”问题，提升自动化编辑可审计性。
- 验收标准：同一段落多次编辑命中率稳定；锚点失效时给出明确冲突提示与重定位结果。

#### U2 类别化模型委派（Category-Based Delegation）
- 功能目标：按任务类别自动选择最合适的执行策略/模型角色，而非统一单通道处理。
- 关键能力：任务分类（如规划、实现、审查、文档）；分类到执行路由映射；路由结果可回溯（记录为何分配到该类别）。
- 触发时机：`orchestrate` 批处理、复杂任务拆分、多子任务并行执行。
- 预期收益：提高复杂任务吞吐与质量一致性，减少“一个策略处理所有问题”的性能和准确率损失。
- 验收标准：分类命中率可统计；不同类别任务质量指标优于统一策略基线。

#### U3 Gemini 意图守卫（Intent Gate）
- 功能目标：在进入 Gemini 相关调用前做“意图级”准入判断，避免误用高成本或高风险能力。
- 关键能力：识别用户意图是否满足调用条件；不满足时拒绝或降级；记录拒绝原因和替代路径。
- 触发时机：准备调用 Gemini 文本/图像/多模态能力前。
- 预期收益：降低误调用成本与安全风险，提升外部模型调用可控性。
- 验收标准：守卫命中日志完整；误调用率下降；被拒绝请求有可执行替代建议。

#### U4 Brainstorm 九步流程
- 功能目标：把开放式需求澄清转为结构化九步收敛流程，避免“想法很多但不可执行”。
- 关键能力：固定九步阶段（问题定义、约束、备选、权衡、反例、决策等）；每步有明确产出；支持中断恢复。
- 触发时机：需求不清晰、方案分歧大、进入 `spec/design` 前的高不确定性阶段。
- 预期收益：决策过程更可追溯，减少后期返工。
- 验收标准：每轮输出均包含九步结构；最终结论可直接映射到 FR/DS/TASK。

#### U5 trellis-meta vs trellis-local 分离
- 功能目标：把“平台通用能力”和“项目本地定制”分层隔离，降低升级时冲突。
- 关键能力：meta 层承载官方基线；local 层承载项目覆盖；升级仅替换 meta，local 通过声明式补丁合并。
- 触发时机：版本升级、模板升级、跨项目复用同一能力栈。
- 预期收益：升级可控、定制可持续，避免每次升级手工冲突清理。
- 验收标准：升级后 local 定制保留率高；冲突项可自动识别并给出合并策略。

## 五、T1 + U5 联合实施方案

> 代码调研日期：2026-03-02
> 核心发现：U5 的 meta/local 分离天然解决了 T1 "不覆盖用户修改"的核心问题，两者应联合设计。

### 5.1 U5 方案：meta/local 目录分离

**现状**：`.spec-first/` 下包级基线与项目定制混放，升级时无法区分"该覆盖"和"该保留"。

**已有可复用代码**：
- `layer-merger.ts` — 三层合并架构（L0 基线 → L1 模式×规模 → L2 平台 YAML），扩展为四层即可
- `loadConfig()` in `config-schema.ts` — 已有缓存机制和默认值合并，增加一级读取链即可
- `loadEnabledExtensions()` in `extensions.ts` — 已有扫描+校验逻辑，增加 meta 目录扫描即可

**目标目录结构**：

```
.spec-first/
├── meta/                    # 包级基线（spec-first update 管理，用户不直接编辑）
│   ├── config.yaml          # 基线配置
│   ├── templates/           # 基线模板
│   ├── rules/               # 基线规则
│   └── template-hashes.json # 模板哈希注册表（T1 用）
├── local/                   # 项目定制（用户编辑，升级时保留）
│   ├── config.yaml          # 项目覆盖配置
│   ├── templates/           # 自定义模板（覆盖同名 meta 模板）
│   └── rules/               # 自定义规则
├── config.yaml              # 现有配置（迁移期兼容，最终废弃）
├── current                  # 当前 feature（不变）
└── ...
```

**配置合并链**（优先级递增）：

```
DEFAULT_SPEC_FIRST_CONFIG → meta/config.yaml → local/config.yaml → config.yaml(兼容)
```

**实现要点**：
1. `loadConfig()` 增加 meta → local 两级读取，合并策略为 deep-merge（local 覆盖 meta 同名键）
2. `renderTemplate()` 增加模板查找链：`local/templates/` → `meta/templates/` → 包内 `templates/`
3. `spec-first update` 只写 `meta/` 目录，不触碰 `local/`
4. 迁移脚本：首次运行时将现有 `.spec-first/` 内容按归属拆分到 meta/local

**工期**：0.5-1d（`loadConfig` 改动约 30 行，模板查找链约 20 行，迁移脚本约 50 行）

### 5.2 T1 方案：模板哈希 + 变更分级更新

**现状**：`spec-first update` 模板刷新为全量覆盖，无法识别"哪些变了""用户改过没有"。

**已有可复用代码**：
- `sha256()` in `audit-log.ts` — 直接复用
- `RfcLevel` 类型（Minor/Major/Critical）— 复用为变更分级

**基于 U5 的简化设计**：

meta/local 分离后，T1 的核心逻辑大幅简化：

| 场景 | meta hash 变化？ | local 有同名覆盖？ | 动作 |
|------|:-:|:-:|------|
| 模板未变 | 否 | — | 跳过 |
| 模板变了，用户无定制 | 是 | 否 | 自动更新 meta |
| 模板变了，用户有定制 | 是 | 是 | 提示用户：查看 diff / 保留 local / 接受新 meta |
| 关键配置变更 | 是 | — | 阻断，必须显式确认 |

**哈希注册表**（`.spec-first/meta/template-hashes.json`）：

```json
{
  "version": "1.0.0",
  "generated": "2026-03-02T00:00:00Z",
  "templates": {
    "feature-spec.hbs": { "hash": "a1b2c3...", "level": "Major" },
    "task-card.hbs": { "hash": "d4e5f6...", "level": "Minor" },
    "config.yaml": { "hash": "g7h8i9...", "level": "Critical" }
  }
}
```

**变更分级规则**：
- `Minor`：文档模板、注释模板 → 自动更新
- `Major`：流程模板、Skill 模板 → 有 local 覆盖时提示确认
- `Critical`：配置文件、规则文件 → 始终阻断确认

**实现要点**：
1. `spec-first update` 流程增加 hash 比对步骤（在模板写入前）
2. 新增 `computeTemplateHashes(dir): Record<string, string>` 工具函数（复用 `sha256()`）
3. 比对逻辑：读旧注册表 → 计算新包 hash → diff → 按分级执行
4. 有 local 覆盖的模板，输出 diff 摘要供用户决策

**工期**：1d（hash 计算约 20 行，比对+分级约 40 行，update 流程集成约 30 行）

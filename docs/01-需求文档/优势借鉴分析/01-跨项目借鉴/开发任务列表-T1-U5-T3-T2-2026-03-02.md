## 模块二：T1 模板哈希 + 变更分级更新（1d，依赖 U5）

### T1-1 创建模板哈希注册表管理模块
- **新建文件**：`src/core/template/hash-registry.ts`
- **功能**：
  - `computeTemplateHashes(dir): Record<string, string>` — 计算目录下所有模板哈希
  - `loadHashRegistry(projectRoot): HashRegistry` — 加载 `.spec-first/meta/template-hashes.json`
  - `saveHashRegistry(registry, projectRoot)` — 保存哈希注册表
  - `compareHashes(old, new): ChangeDiff[]` — 比对哈希差异
- **复用**：`sha256()` from `audit-log.ts`
- **验收**：能正确计算和比对模板哈希

### T1-2 创建变更分级分类器
- **新建文件**：`src/core/template/change-classifier.ts`
- **功能**：
  - `classifyChange(templateName, changeType): RfcLevel` — 分类为 Minor/Major/Critical
  - `isCriticalTemplate(name): boolean` — 判断是否为关键模板
  - 分类规则：
    - Minor：文档模板、注释模板
    - Major：流程模板、Skill 模板
    - Critical：config.yaml、规则文件
- **验收**：分类规则符合预期

### T1-3 实现变更决策逻辑
- **新建文件**：`src/core/template/update-decision.ts`
- **功能**：
  - 基于 meta/local 分离的四种场景决策
  - `decideUpdate(metaHashChanged, hasLocalOverride, changeLevel): UpdateAction`
  - 决策类型：SKIP / AUTO_UPDATE / PROMPT / BLOCK
- **验收**：决策矩阵与设计一致

### T1-4 集成到 update 命令流程
- **文件**：`src/cli/commands/update.ts`
- **改动**：
  - 在模板写入前执行哈希比对
  - 按决策类型执行动作
  - PROMPT/BLOCK 时输出 diff 摘要并等待用户确认
- **验收**：update 流程集成完整

- **新建文件**：`src/core/backup/snapshot-manager.ts`

  - `listSnapshots(projectRoot): SnapshotInfo[]` — 列出快照

- **快照内容**：`.spec-first/` 目录、关键配置文件
- **验收**：快照创建、恢复、删除功能正常

- **新建文件**：`src/core/backup/rollback-engine.ts`
- **功能**：
  - `buildRollbackPlan(snapshotId): RollbackPlan` — 构建回滚计划
  - `executeRollback(plan): RollbackResult` — 执行回滚
  - `dryRun(plan): RollbackPlan` — 演练模式（验证回滚链路）
- **复用**：现有 `buildRollbackPlan()` 逻辑
- **验收**：回滚引擎可执行和演练

### T3-3 创建 rollback 命令
- **新建文件**：`src/cli/commands/rollback.ts`
- **功能**：
  - `spec-first rollback list` — 列出快照
  - `spec-first rollback create [--label]` — 创建快照
  - `spec-first rollback restore <snapshotId>` — 恢复快照
  - `spec-first rollback delete <snapshotId>` — 删除快照
  - `--dry-run` 选项用于演练
- **验收**：命令可用，交互友好

### T3-4 update 命令集成自动快照
- **文件**：`src/cli/commands/update.ts`
- **改动**：
  - update 执行前自动创建快照
  - 快照失败时阻断更新
  - 更新失败时提示可用快照 ID
- **验收**：update 有自动快照保护

### T3-5 快照存储目录设计
- **目录**：`.spec-first/.snapshots/`
- **结构**：
  ```
  .spec-first/.snapshots/
  ```

- **验收**：目录结构清晰，易于管理

### T2-1 定义 Manifest 结构与类型
- **新建文件**：`src/core/migrations/manifest-schema.ts`
- **类型定义**：
  - `MigrationManifest` 接口
  - `MigrationStep` 类型（rename/delete/copy/patch/execute/mkdir）
  - `VersionRange` 类型
  - `ConflictStrategy` 枚举（skip/overwrite/abort/prompt）
- **验收**：类型定义完整

### T2-2 实现 Manifest 加载与校验
- **新建文件**：`src/core/migrations/manifest-loader.ts`
- **功能**：
  - `loadManifest(path): MigrationManifest` — 加载 YAML
  - `validateManifest(manifest): ValidationResult` — 校验结构
  - `listManifests(projectRoot): MigrationManifest[]` — 列出所有 manifest
- **验收**：能正确加载和校验 manifest

### T2-3 实现版本区间匹配
- **新建文件**：`src/core/migrations/version-matcher.ts`
- **功能**：
  - `matches(current, range): boolean` — 判断版本是否匹配
  - `parseRange(rangeStr): VersionRange` — 解析版本区间语法
  - 支持语义化版本（semver）
- **验收**：版本匹配逻辑正确

### T2-4 实现迁移步骤执行引擎
- **新建文件**：`src/core/migrations/manifest-engine.ts`
- **功能**：
  - `executeStep(step, projectRoot, strategy): StepResult` — 执行单个步骤
  - `executeManifest(manifest, projectRoot, conflictStrategy): ExecutionResult` — 执行完整 manifest
  - 支持的步骤类型：mkdir/rename/delete/copy/patch/execute
- **验收**：各步骤类型执行正确

### T2-5 集成到 update 命令
- **文件**：`src/cli/commands/update.ts`
- **改动**：
  - update 流程中增加 manifest 执行阶段

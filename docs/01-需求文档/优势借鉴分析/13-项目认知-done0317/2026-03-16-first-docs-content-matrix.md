# First 文档承载内容矩阵

> 目标：明确 `first` 文档体系中每个正式文档分别承载什么内容，避免文档之间职责重叠、边界模糊或信息缺口。

---

## 一、文档分层

当前文档体系分为三层：

1. **基础 canonical docs**
   - 承载项目级认知底座
   - 适合所有角色和阶段优先读取

2. **正式专题文档**
   - 围绕重要主题做深入展开
   - 适合在进入开发、设计或审查前定向阅读

3. **条件型正式文档**
   - 仅在项目满足条件时生成
   - 用于补足特定项目的重要认知维度

---

## 二、基础 canonical docs 内容矩阵

| 文档 | 文档定位 | 主要承载内容 | 主要读者 / 场景 |
|------|----------|--------------|-----------------|
| `README.md` | 总入口 | 文档导航、阅读顺序、条件型文档状态、整体使用说明 | 所有角色首次进入 `docs/first` 时 |
| `summary.md` | 项目总览 | 项目是做什么的、主要能力、主要入口、核心风险、基础证据摘要 | 所有角色首次理解项目 |
| `role-views.md` | 角色导航 | 按开发、产品、测试、架构等角色组织的优先阅读路径与关注点 | 新手接手项目、跨角色协作 |
| `stage-views.md` | 阶段导航 | 按 spec/design/task/code/review/verify 阶段组织的最小背景输入 | 下游 skill、阶段切换时 |
| `steering.md` | 项目导向与边界 | 产品目标、技术约束、结构边界、权威顺序、核心原则 | 设计前、实现前、评审前 |
| `conventions.md` | 项目规范总览 | 编码规范、测试规范、配置规范、协作与交付约定的总览视图 | 实现前、review 前、verify 前 |
| `critical-flows.md` | 关键链路总览 | 核心业务链路、关键流程、高风险路径与不可轻易破坏的主干流程 | 设计前、实现前、回归分析时 |
| `change-map.md` | 变更导航图 | 常见改动入口、影响面、热点区域、验证提示与回归关注点 | 老需求迭代、任务拆解、实现前 |
| `entry-guide.md` | 最小进入路径 | 新手进入项目时从哪里开始读、先看什么、常见改动从哪开始 | 新手开发、需求接手 |
| `reboot-guide.md` | 恢复路径 | 会话中断后如何快速恢复上下文、重新定位工作 | 长间隔恢复、交接后接手 |
| `common-playbooks.md` | 常见改动剧本 | 改接口、改页面、改配置、改数据结构等典型任务的推荐步骤 | 老需求迭代、实现前准备 |
| `known-risks-and-traps.md` | 风险与陷阱汇总 | 已知高风险链路、历史坑点、常见误改点、回归盲区 | 修改前、review 前、verify 前 |
| `tech-stack.md` | 技术栈专题视图 | 语言、框架、构建方式、测试基础设施、关键技术组件与技术约束 | 新手接手项目、设计前、环境判断时 |
| `codebase-overview.md` | 代码结构总览 | 模块划分、关键目录、入口关系、阅读顺序、代码导航地图 | 新手理解代码、设计前、实现前 |
| `domain-model.md` | 业务领域模型 | 核心业务对象、概念边界、对象关系、关键语义、易混淆概念 | 需求设计、实现前、评审前 |

## 三、正式专题文档内容矩阵

| 文档 | 文档定位 | 主要承载内容 | 主要读者 / 场景 |
|------|----------|--------------|-----------------|
| `architecture.md` | 架构专题视图 | 系统整体结构、技术边界、模块协作关系、关键架构路径、架构风险 | 设计、重构、复杂需求实现前 |
| `call-graph.md` | 调用链专题视图 | 入口到核心逻辑的关键调用路径、扩散点、高风险传播关系 | 改复杂需求、定位影响面、debug |
| `external-deps.md` | 外部依赖专题视图 | 第三方依赖、外部服务、集成边界、依赖耦合风险、升级注意事项 | 接手老项目、环境排查、风险分析 |
| `local-setup.md` | 本地环境专题视图 | 环境要求、安装步骤、启动步骤、本地调试方法、常见环境问题 | 新手开发、环境搭建、问题排查 |
| `development-guidelines.md` | 开发规范专题视图 | 代码规范、测试规范、配置规范、提交流程、必须遵守的项目开发约定 | 实现前、review 前、团队协作 |

## 五、职责边界

为避免文档之间重复，职责边界应保持如下：

| 文档 | 不应承载的内容 |
|------|----------------|
| `summary.md` | 不展开具体架构、调用链、环境步骤 |
| `steering.md` | 不展开完整开发规范细节，不替代 `development-guidelines.md` |
| `entry-guide.md` | 不承载完整环境安装手册，不替代 `local-setup.md` |
| `codebase-overview.md` | 不承载架构判断与深层调用分析，不替代 `architecture.md` / `call-graph.md` |
| `domain-model.md` | 不替代数据库结构视图，不替代 `database-er.md` |
| `architecture.md` | 不重复基础目录清单，不替代 `codebase-overview.md` |
| `external-deps.md` | 不替代完整技术栈说明，不替代 `tech-stack.md` |
| `development-guidelines.md` | 不替代规范总览，只做展开说明 |
| `database-er.md` | 不替代领域模型，只负责数据结构认知 |

## 六、结论

这套文档体系的设计原则是：

1. **基础 canonical docs 负责建立项目级认知底座**
2. **正式专题文档负责深挖关键主题，帮助快速进入开发**
3. **条件型正式文档负责补足特定项目的重要认知维度**

一句话总结：

> `first` 文档体系不是“很多文档的堆叠”，而是一套按层次承载项目认知、开发导航、风险提示和专题深入内容的结构化文档系统。

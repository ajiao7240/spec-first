# 2026-03-18 first 三阶段开发任务文档

## 1. 文档目标

本任务文档用于把 `first` 的后续落地工作收口成三阶段实施路线，服务以下目标：

1. `first` 只有一个正式入口：`spec-first first`
2. `first` 负责生成项目级认知真源与投影视图
3. 下游 skill 节点按需消费 `first` 文档，提高节点质量
4. `archive / sync` 在 Feature 完成时回写项目级认知，保证 `first` 持续最新
5. `first` 永远是增强层，不是硬前置；缺失时任何节点都不得阻断流程

本方案不以“当前已经完成多少”为前提，而以目标态落地顺序为前提。

## 2. 总体范围

本轮实施只覆盖三块：

1. `first` 主链生成项目级文档
2. 各 skill 节点引入项目级文档作为辅助上下文
3. 归档阶段将需求变更回写到项目级文档

不在本轮范围：

1. 恢复 `quick / deep` 双模式
2. 把 `first` 变成所有节点的硬前置依赖
3. 让需求级文档与项目级文档混放
4. 让节点默认全文注入全部项目级文档

## 3. 三阶段总览

| 阶段 | 名称 | 核心目标 | 主要产出 |
| --- | --- | --- | --- |
| 第一阶段 | `first` 生成项目级文档 | 建立统一入口、统一真源、统一投影视图 | `.spec-first/runtime/first/*`、`docs/first/*` |
| 第二阶段 | 各节点接入项目级文档 | 让下游节点按需消费 `first`，提升质量但不阻断流程 | 节点输入矩阵、`context-resolver` / `dispatcher` 接入 |
| 第三阶段 | 归档回写项目级文档 | 在 Feature 完成时将结构性变更同步回 `first` | `archive / sync` 回写链路、治理日志、增量刷新 |

## 4. 第一阶段：`first` 执行并创建相关文档

### 4.1 阶段目标

让 `spec-first first` 成为唯一正式入口，执行后能够稳定生成项目级 runtime 真源与项目级投影视图。

### 4.2 阶段任务

#### 任务 1：收口入口协议

目标：

1. 固定正式入口为 `spec-first first`
2. 移除 quick / deep、旧别名、双默认路径心智
3. 明确 CLI 主链优先于 skill 编排入口

实施点：

1. CLI 参数协议
2. `skills/spec-first/00-first/SKILL.md`
3. 宿主命令路由
4. 测试与帮助文案

验收：

1. 用户只需执行 `spec-first first`
2. 对外文案不再宣传双模式
3. 相关测试与帮助输出统一到单入口口径

#### 任务 2：固定项目级 runtime 真源模型

目标：

1. runtime 真源只承载项目级稳定认知
2. 移除需求级或阶段级内容混入

项目级正式 runtime 资产固定为 9 个：

1. `summary.json` - 项目摘要
2. `steering.json` - 项目导向
3. `conventions.json` - 项目规范
4. `critical-flows.json` - 关键流程
5. `entry-guide.json` - 入口指南
6. `api-contracts.json` - 接口契约
7. `structure-overview.json` - 结构总览
8. `domain-model.json` - 领域模型
9. `database-schema.json` - 数据库结构

索引文件：

- `index.json` - 记录健康度、新鲜度、投影一致性

实施点：

1. runtime types / store / bootstrap / refresh
2. artifact mapping
3. health / governance / index
4. 文档与测试同步

验收：

1. runtime 模型不再包含 `role / stage / change / reboot`
2. 所有 runtime 资产都有清晰职责
3. `index.json` 能表达资产存在性、健康度、投影状态

#### 任务 3：固定项目级投影视图

目标：

1. `docs/first` 只承载项目级正式文档
2. 文档数量收口到正式主集合

项目级正式文档主集合固定为 14 个（含 1 个条件型）：

1. `README.md` - 总入口
2. `summary.md` - 项目总览
3. `steering.md` - 项目导向
4. `conventions.md` - 项目规范总览
5. `critical-flows.md` - 关键流程总览
6. `entry-guide.md` - 入口指南
7. `api-docs.md` - 接口规范文档
8. `codebase-overview.md` - 代码库总览
9. `domain-model.md` - 领域模型文档
10. `architecture.md` - 架构视图
11. `call-graph.md` - 调用链视图
12. `external-deps.md` - 外部依赖视图（组合投影，真源来自 summary/steering/conventions）
13. `development-guidelines.md` - 开发规范指南
14. `database-er.md` - 数据库 ER 视图（条件型，仅在 database-schema.status === healthy 时生成）

文档分层：
- 启动层（5份）：README, summary, entry-guide, codebase-overview, steering
- 开发层（4份）：development-guidelines, api-docs, critical-flows, external-deps
- 专题层（6份）：architecture, call-graph, domain-model, database-er, steering, conventions

建议删除、不再作为项目级正式文档：

1. `role-views.md`
2. `stage-views.md`
3. `change-map.md`
4. `reboot-guide.md`
5. `common-playbooks.md`
6. `known-risks-and-traps.md`
7. `tech-stack.md`
8. `local-setup.md`

实施点：

1. `first-doc-projection.ts`
2. docs registry
3. renderer 章节结构
4. 文档测试
5. 投影与真源映射校正

验收：

1. `docs/first` 只生成正式项目级文档
2. docs 与 runtime 一一对应，不混需求级内容
3. docs 作为阅读视图，不回灌为真源
4. `external-deps.md` 明确为组合投影，真源来自 `summary.json + steering.json + conventions.json`，不新增 `external-deps.json`
5. `tech-stack.md` 删除后，技术栈信息统一并入 `summary.json.techStack`，并在 `summary.md` 中展示
6. `steering.md` 在启动层只承担“快速浏览目标与边界”，在专题层承担“完整约束与禁止模式”

#### 任务 4：保证 `first` 缺失时不阻断其他节点

目标：

1. `first` 是增强层，不是硬前置
2. 没有 runtime / docs 时，任何节点都不报错、不停流程

实施点：

1. `context-resolver`
2. 背景状态与 fallback
3. `dispatcher`
4. skill docs 与 contract

验收：

1. `missing / partial / stale` 三种状态都有清晰降级
2. 没有 `first` 时，节点继续执行
3. 只降低背景质量与置信度，不阻断流程

### 4.3 第一阶段交付物

1. 单入口 `spec-first first`
2. 项目级 runtime 真源
3. 项目级正式 docs
4. runtime / docs / health / projection 测试

## 5. 第二阶段：各个 skill 节点引入文档作为上下文改造

### 5.1 阶段目标

让各节点在不改变自身主职责的前提下，按需使用 `first` 项目级文档提升质量。

原则：

1. 节点主职责优先
2. `first` 只提供项目级背景增强
3. 节点仍能在 `first` 缺失时继续执行
4. 默认最小注入，不默认全文注入

### 5.2 节点接入分类

#### A. 不应依赖 `first` 的需求级节点

这类节点主恢复源或主状态源来自 `specs/{featureId}`，不应依赖 `first` 才能工作：

1. `catchup`
2. 需求级阶段状态恢复相关节点

要求：

1. 不把 `first` 当正式输入来源
2. 需求级上下文独立闭环
3. 不把项目级认知和需求级恢复混放

#### B. 以需求级主流程为主，但可读取 `first` 增强质量的节点

这类节点仍是需求级主流程，但项目级认知能显著提升质量：

1. `spec`
2. `design`
3. `task`
4. `plan`
5. `research`
6. `code`
7. `review`
8. `verify`
9. `status`
10. `orchestrate`
11. `doctor`
12. `archive`
13. `sync`
14. `spec-review`
15. `analyze`

要求：

1. `required` / `optional` 输入矩阵代码化
2. 只注入必要摘要
3. 缺失时自动降级，不得阻断

### 5.3 阶段任务

#### 任务 1：建立统一输入矩阵

目标：

1. 把每个节点可消费的 `first` 资产统一登记
2. 明确 `required / optional / degradation`
3. 固定默认注入层级为 `L0 / L1 / L2`

建议新增：

1. `src/core/skill-runtime/first-skill-input-matrix.ts`

至少包含：

1. `requiredRuntimeAssets`
2. `optionalRuntimeAssets`
3. `allowedProjectionDocs`
4. `degradationLevel`
5. `recommendedActionWhenMissing`
6. `defaultInjectionLevel`
7. `l1SummarySchema`

验收：

1. 不再把输入矩阵散落在各 skill 文档里
2. `context-resolver` 和 `dispatcher` 可以共享同一份 contract
3. 节点输入矩阵已基于 SDD 最佳实践优化：
   - `spec` 必需 5 个资产（+conventions.json）
   - `design` 必需 6 个资产（+critical-flows.json, +domain-model.json）
   - `task` 必需 6 个资产（+4 个关键资产）
   - `code` 必需 7 个资产（+4 个关键资产）
   - `archive` 必需 9 个资产（覆盖所有 runtime 资产）
4. 每个 runtime 资产都有可程序校验的 `L1` 摘要字段定义（参见最佳实践方案 6.2.1 节）

#### 任务 2：改造 `context-resolver`

目标：

1. 每个节点只装载自己的最小项目级背景
2. docs 只作为辅助阅读层

实施点：

1. required / optional 分层读取
2. runtime 优先
3. fallback / missing / partial / stale
4. 节点级别差异化读取
5. `L1` 摘要优先注入

验收：

1. 无节点默认全量读 `docs/first`
2. 节点只拿自己需要的项目级资产
3. 缺失时返回背景告警，不返回阻断错误
4. 默认注入以 `L1` 摘要为主，不直接拼接全文

#### 任务 3：改造 `dispatcher`

目标：

1. 将背景质量、依赖强度、推荐动作统一投影给节点
2. 不在展示层发明与事实无关的新语义

实施点：

1. runtime notice
2. 节点级 guidance
3. 风险提示与推荐动作
4. `background_input_status / background_status` 一致性

验收：

1. 各节点收到的项目背景有统一格式
2. 缺失 `first` 时展示“降级执行”，而不是“不可执行”

#### 任务 5：实现 token 控制策略

目标：

1. 避免多个节点重复注入相同项目级背景导致 token 成本线性上升
2. 默认只注入最小摘要，不注入整份 runtime 或整篇 markdown
3. 同一主题默认优先注入 runtime 摘要，不同时注入对应 projection docs 全文

token 控制策略（L0/L1/L2 三层注入）：

**L0 元信息**：
- 资产是否存在
- 健康状态（healthy/partial/stale/missing）
- 最后更新时间

**L1 关键字段摘要**（已有完整 schema 定义）：

| 运行时资产 | L1 关键字段 |
| --- | --- |
| `summary.json` | `projectWhat`, `mainCapabilities`, `techStack`, `platformType` |
| `steering.json` | `goals`, `nonGoals`, `constraints`, `boundaries` |
| `conventions.json` | `apiRules`, `moduleRules`, `testingRules`, `projectRules` |
| `critical-flows.json` | `names`, `count`, `invariants` |
| `entry-guide.json` | `readFirst`, `thenRead`, `avoidEntry`, `categories` |
| `api-contracts.json` | `interfaceNames`, `count`, `interfaceTypes`, `authModes` |
| `structure-overview.json` | `moduleNames`, `entryPoints`, `topDirs`, `count` |
| `domain-model.json` | `entityNames`, `count`, `glossaryTerms` |
| `database-schema.json` | `status`, `provider`, `tableNames`, `count` |

**L2 全文或长片段**：
- 仅当节点显式需要时才展开

详细 schema 定义参见《2026-03-17-first-最佳实践目标态方案.md》第 6.2.1 节。

验收：

1. 节点表中的"可访问资产"不等于"默认全文注入"
2. 连续节点执行时 token 不因重复背景线性膨胀
3. `L1` 摘要字段成为节点默认消费层，能直接被测试验证

### 5.4 第二阶段交付物

1. 输入矩阵 contract
2. 节点级 `context-resolver` 改造
3. 节点级 `dispatcher` 改造
4. 缺失 `first` 的降级测试

## 6. 第三阶段：归档完成时同步更新项目级文档

### 6.1 阶段目标

让 `archive / sync` 成为项目级认知持续更新的正式入口，确保 `first` 始终反映当前项目真实状态。

### 6.2 阶段原则

1. Feature 级变更完成后，结构性变化必须回流到项目级真源
2. `code / review / verify` 不直接改 `docs/first`
3. 正式 writeback 由 `archive / sync` 负责
4. 先更新 runtime，再刷新 docs
5. 保留“校验”和“刷新”能力，但不扩成新的正式用户命令

### 6.3 阶段任务

#### 任务 1：定义回写触发条件

目标：

1. 明确什么样的 Feature 完成后需要回写 `first`

建议触发类型：

1. 接口契约变化
2. 关键流程变化
3. 项目规范变化
4. 模块边界变化
5. 领域对象变化
6. 数据结构变化
7. 项目入口与导航变化

验收：

1. `archive / sync` 能识别“是否存在项目级认知影响”
2. 无影响的 Feature 不做无意义刷新

#### 任务 2：实现 `archive / sync` 回写协议

目标：

1. 在 Feature 归档时判断是否改变了项目级认知
2. 把结构性变更同步回 `first`
3. archive 节点必须检查所有 9 个 runtime 资产

回写范围（覆盖所有 9 个 runtime 资产）：

1. 项目总览变化 → `summary.json`
2. 项目目标/约束变化 → `steering.json`
3. 规范变化 → `conventions.json`
4. 关键流变化 → `critical-flows.json`
5. 开发入口变化 → `entry-guide.json`
6. API 变化 → `api-contracts.json`
7. 结构变化 → `structure-overview.json`
8. 领域模型变化 → `domain-model.json`
9. 数据结构变化 → `database-schema.json`

结构性变更判断标准：
- 新增/删除模块（src/ 下新增/删除目录）
- API 变更（新增/删除/修改接口签名）
- 数据库 schema 变更（新增/删除/修改表或字段）
- 核心流程变更（关键入口或主调用路径变化）
- 依赖变更（package.json 中新增/删除/升级依赖）

验收：

1. `archive` 节点能识别所有 9 个 runtime 资产是否需要更新
2. 结构性变更能被准确识别并回写
3. 回写后 `index.json` 更新健康度和新鲜度

#### 任务 3：实现 runtime writeback 与 docs refresh

目标：

1. `archive / sync` 结束时刷新受影响 runtime
2. 再刷新对应 `docs/first`

实施点：

1. `first-governance`
2. `first-context`
3. 变更映射
4. 增量投影
5. 回写后自动校验 runtime 真源完整性与 docs 投影一致性

验收：

1. Feature 结束后，项目级文档仍保持最新
2. 回写路径稳定、可重复、可审计
3. 校验与刷新作为系统能力存在，不新增新的正式入口命令

#### 任务 4：补治理日志与审计能力

目标：

1. 让项目级文档更新具备追溯性

治理日志至少记录：

1. 哪个 Feature 触发
2. 影响了哪些 runtime 资产
3. 刷新了哪些 docs
4. 主要证据来自哪些 Feature 文件
5. 结果是成功、降级还是跳过

验收：

1. 项目级认知更新可追溯
2. 后续问题可审计，不靠人工回忆

### 6.4 第三阶段交付物

1. `archive / sync` 回写协议
2. runtime 增量刷新
3. docs 增量投影
4. 治理日志与审计链

## 7. 推荐实施顺序

### 第一批

1. 单入口
2. 项目级 runtime 真源
3. 项目级 docs 主集合
4. 缺失 `first` 不阻断

### 第二批

1. 输入矩阵代码化
2. `context-resolver` 接入
3. `dispatcher` 接入
4. token 控制

### 第三批

1. `archive / sync` 回写协议
2. 受影响资产识别
3. runtime / docs 增量刷新
4. 治理日志

## 8. 阶段验收总表

| 阶段 | 验收重点 |
| --- | --- |
| 第一阶段 | `spec-first first` 可稳定生成项目级 runtime 与 docs，且不混需求级文档 |
| 第二阶段 | 下游节点能按需消费 `first`，缺失时只降级不阻断，且 token 注入受控 |
| 第三阶段 | Feature 归档后能把结构性变化同步回 `first`，项目级认知持续最新 |

## 9. 一句话结论

这三阶段的正确落地顺序是：

1. 先让 `first` 稳定生成项目级文档
2. 再让下游节点按需消费这些文档提升质量
3. 最后让 `archive / sync` 把需求变更回写到项目级认知底座

只有这样，`first` 才会从“一次性文档生成器”真正变成“项目级持续认知系统”。

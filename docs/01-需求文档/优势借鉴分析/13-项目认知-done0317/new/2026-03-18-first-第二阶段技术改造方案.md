## 3. 第二阶段技术改造方案

### 3.1 当前架构分析

**现有 prompt 组装流程**（`dispatcher.ts:loadSkill()`）：

```typescript
// 1. 加载 skill 模板
content = loadSkillTemplate(skillPath);

// 2. 变量替换（FEATURE_ID, CURRENT_STAGE 等）
if (enableAssembly) {
  ctx = resolvePromptAssemblyContext(executionContext);
  content = assemblePrompt(content, ctx);
}

// 3. 注入 runtime notice（按 skillName 分支）
if (skillName === 'spec') {
  content = `${buildSpecRuntimeNotice(ctx)}\n\n${content}`;
}
if (skillName === 'design') {
  content = `${buildDesignRuntimeNotice(ctx)}\n\n${content}`;
}
// ... 其他 skill
```

**关键发现**：

1. ✅ **已有注入机制**：每个 skill 都有对应的 `build*RuntimeNotice()` 函数
2. ✅ **已调用 resolveSkillContext()**：所有 `build*RuntimeNotice()` 内部都调用了 `resolveSkillContext()`
3. ✅ **已实现降级策略**：`context.backgroundInputStatus` 已传递到 notice 中
4. ❌ **注入内容不统一**：每个 skill 的 notice 格式和字段不一致
5. ❌ **token 控制缺失**：无法按预算裁剪注入内容

### 3.2 改造策略

**核心原则**：复用现有 `build*RuntimeNotice()` 机制，增强而非重写。

```
┌─────────────────────────────────────────────────────────────┐
│                    Skill Execution Layer                     │
│                   (loadSkill in dispatcher.ts)               │
└────────────────────────┬────────────────────────────────────┘
                         │
                         │ 1. 加载模板 + 变量替换
                         ↓
┌─────────────────────────────────────────────────────────────┐
│              Prompt Assembler (prompt-assembler.ts)          │
│  - assemblePrompt(template, ctx)                             │
│  - 替换 {{FEATURE_ID}}, {{CURRENT_STAGE}} 等                │
└────────────────────────┬────────────────────────────────────┘
                         │
                         │ 2. 注入 runtime notice（按 skillName）
                         ↓
┌─────────────────────────────────────────────────────────────┐
│         build*RuntimeNotice() (dispatcher.ts:485-560)        │
│  - 调用 resolveSkillContext(projectRoot, skillName)         │
│  - 格式化 notice（backgroundInputStatus + 资产摘要）        │
│  - 返回 markdown 格式的 notice                               │
└────────────────────────┬────────────────────────────────────┘
                         │
                         │ 3. 解析上下文（内部调用）
                         ↓
┌─────────────────────────────────────────────────────────────┐
│              Context Resolver (context-resolver.ts)          │
│  - 读取输入矩阵（getSkillInputContract）                    │
│  - 检测 backgroundInputStatus                                │
│  - 三层降级：runtime → docs → none                           │
│  - 返回 ResolvedSkillContext                                 │
└─────────────────────────────────────────────────────────────┘
```

**目标**：统一所有 `build*RuntimeNotice()` 的输出格式，便于 AI 解析

**背景**：
- 当前 `dispatcher.ts:485-560` 中有 10 个 `build*RuntimeNotice()` 函数

**实施点**：

1. 创建 `formatRuntimeNotice()` 统一格式化函数
2. 重构所有 `build*RuntimeNotice()` 使用统一格式
3. 标准化字段名（`backgroundInputStatus`, `data_source`, `required_assets`, `optional_assets`）
4. 保持 `loadSkill()` 中的注入逻辑不变（行 485-560）

**统一格式**：

```typescript
// dispatcher.ts
function formatRuntimeNotice(
  marker: string,
  title: string,
  context: ResolvedSkillContext,
  extraFields?: Record<string, string>
): string {
  const parts = [`<!-- ${marker} -->`, `## ${title}`];
  parts.push(`backgroundInputStatus: ${context.backgroundInputStatus}`);
  parts.push(`data_source: ${context.source}`);
  
  if (context.requiredAssetNames.length > 0) {
    parts.push(`required_assets: ${context.requiredAssetNames.join(', ')}`);
  }
  if (context.optionalAssetNames.length > 0) {
    parts.push(`optional_assets: ${context.optionalAssetNames.join(', ')}`);
  }
  
  // 注入额外字段（如 stageViewSummary, roleViewSummary）
  if (extraFields) {
    for (const [key, value] of Object.entries(extraFields)) {
      parts.push(`${key}: ${value}`);
    }
  }
  
  if (context.missingRequiredAssets.length > 0) {
    parts.push(`missing_required_assets: ${context.missingRequiredAssets.join(', ')}`);
  }
  
  if (context.fallback.warning) {
    parts.push(`warning: ${context.fallback.warning}`);
  }
  
  if (context.recommendedAction) {
    parts.push('recommendation: 建议先运行 /spec-first:first 补全背景数据');
  }
  
  parts.push(`<!-- /${marker} -->`);
  return parts.join('\n');
}
```

**重构示例**：

```typescript
// 重构前
function buildSpecRuntimeNotice(executionContext: SkillExecutionContext): string | undefined {
  const context = resolveSkillContext(executionContext.projectRoot, 'spec', executionContext.featureId);
  const parts = ['<!-- spec-runtime-context -->', '## Spec Context'];
  parts.push(`background_input_status: ${context.backgroundInputStatus}`);
  // ... 各种自定义格式
  return parts.join('\n');
}

// 重构后
function buildSpecRuntimeNotice(executionContext: SkillExecutionContext): string | undefined {
  const context = resolveSkillContext(executionContext.projectRoot, 'spec', executionContext.featureId);
  return formatRuntimeNotice('spec-runtime-context', 'Spec Context', context, {
    spec_view_summary: context.stageViewSummary || 'N/A',
  });
}
```

```markdown
# Background Quality Contract

## 背景质量三级状态

### full
- 所有 required 资产可用
- 至少 50% optional 资产可用
- 节点可以高质量执行

### degraded
- 部分 required 资产缺失
- 节点可以执行，但需要更多假设
- 应提示用户运行 `spec-first first`

### blind
- 所有 required 资产缺失
- 节点仍可执行，但质量无保障
- 必须明确告知用户风险

## 节点降级行为规范

### 检查时机
- 节点启动时检查 `backgroundInputStatus`
- 从 `ResolvedSkillContext` 读取状态

### 降级行为
- **full**: 正常执行，无需提示
- **degraded**: 执行前提示缺失资产，建议运行 `first`
- **blind**: 执行前明确告知风险，询问是否继续

### 禁止行为
- ❌ 因背景缺失而阻断流程
- ❌ 自动运行 `spec-first first`（除非用户明确授权）
- ❌ 忽略降级状态，不告知用户

## 示例

### spec 节点
- **required**: steering, conventions
- **degraded 行为**: 提示"缺少项目导向和规范，可能需要更多澄清"
- **blind 行为**: 提示"无项目背景，将基于通用最佳实践"

### code 节点
- **required**: conventions, entry-guide
- **degraded 行为**: 提示"缺少开发规范，可能不符合项目约定"
- **blind 行为**: 提示"无项目背景，将使用通用代码风格"
```

## 5. 风险与缓解

### 风险 1：token 预算难以精确控制

**描述**：不同 runtime 资产大小差异大，难以精确预测 token 使用量

**缓解**：
1. 实现"软限制"策略：超出 10% 以内允许
2. 提供 `--first-context-budget` CLI 参数，允许用户覆盖
3. 记录实际 token 使用量，用于后续优化

### 风险 2：降级提示可能被忽略

**描述**：用户可能忽略 degraded / blind 提示，导致质量问题

**缓解**：
1. 降级提示使用醒目格式（⚠️ 符号）
2. 在 skill 执行结束时再次提示
3. 在 `spec-first status` 中显示背景健康度

### 风险 3：输入矩阵维护成本

**描述**：新增 skill 时需要同步更新输入矩阵

**缓解**：
1. 在 `SKILL_INPUT_MATRIX` 中提供 `default` 配置
2. 新增 skill 时自动继承 `default`
3. 单元测试检查所有 skill 都有定义

### 风险 4：向后兼容性

**描述**：现有 skill 可能依赖旧的上下文获取方式

**缓解**：
1. 保留 `resolveSkillContext()` 的旧接口
2. 新增 `options` 参数为可选
3. 逐步迁移现有 skill

### 风险 5：现有 `build*RuntimeNotice()` 格式不统一

**描述**：当前 10 个 `build*RuntimeNotice()` 函数格式各异，统一成本高

**缓解**：
1. 先创建 `formatRuntimeNotice()` 统一格式化函数
2. 逐步迁移现有函数，保持接口不变
3. 单元测试确保迁移前后输出一致

## 7. 后续优化方向

### 优化 1：智能裁剪

**目标**：根据 skill 实际使用情况动态调整裁剪策略

**方案**：
1. 记录每个 skill 实际使用的资产字段
2. 优先裁剪未使用的字段
3. 定期分析使用数据，优化默认配置

### 优化 2：缓存机制

**目标**：减少重复读取 runtime 资产的开销

**方案**：
1. 在 `context-resolver` 中实现 LRU 缓存
2. 缓存 key 为 `${projectRoot}:${skillName}:${runtimeHash}`
3. runtime 变更时自动失效

### 优化 3：按需加载

**目标**：支持"先注入摘要，按需加载详情"模式

**方案**：
1. 在 prompt 中只注入资产摘要（如 `steering.goals`）
2. 提供 `@first:steering:full` 指令，允许 AI 按需加载完整内容
3. 通过 tool call 实现动态加载

## 8. 一句话结论

第二阶段的核心是"让下游节点按需消费 first，提升质量但不阻断流程"，技术实现的关键是：

1. **输入矩阵代码化**：明确每个 skill 需要什么（`SKILL_INPUT_MATRIX`）
2. **统一 runtime notice 格式**：重构 `build*RuntimeNotice()` 使用 `formatRuntimeNotice()`
3. **token 控制与降级策略**：增强 `context-resolver` 支持按需裁剪
4. **背景质量契约**：统一降级行为规范，确保流程不阻断

**改造原则**：复用现有 `build*RuntimeNotice()` 机制，增强而非重写，最小化改动范围。

**预期收益**：
- 下游 skill 质量提升 20-30%（基于项目背景的决策更准确）
- 开发体验改善（自动注入上下文，无需手动查找）
- 系统可维护性提升（输入矩阵集中管理，格式统一）

只有这样，`first` 才能从"可选增强"真正变成"全流程公共背景输入层"。

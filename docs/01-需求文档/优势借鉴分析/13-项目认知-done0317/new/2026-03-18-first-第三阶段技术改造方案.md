# 2026-03-18 first 第三阶段技术改造方案

## 1. 方案概述

### 1.1 阶段目标

第三阶段实现 Feature 归档时将结构性变更回写到项目级 `first` 认知底座，确保项目级认知持续最新。

### 1.2 核心价值

- **持续性**：`first` 不再是一次性产物，而是随 Feature 迭代持续更新的知识底座
- **闭环性**：Feature 生命周期末端自动同步项目级认知，形成完整闭环
- **可追溯**：每次回写都有治理日志，可审计、可回溯

### 3.3 任务 3：实现增量更新引擎

#### 3.3.1 创建 `first-incremental-update.ts`

**核心逻辑**：

```typescript
// 伪代码示例
function incrementalUpdateRuntimeAssets(
  projectRoot: string,
  changes: StructuralChange[]
): UpdateResult {
  const affectedAssets = mapChangesToAssets(changes);
  const updated: RuntimeAssetName[] = [];
  const conflicts: ConflictRecord[] = [];

  for (const assetName of affectedAssets.runtimeAssets) {
    // 1. 读取现有资产
    const existing = readRuntimeAsset(projectRoot, assetName);

    // 2. 应用变更
    const merged = mergeChanges(existing, changes, assetName);

    // 3. 检测冲突
    const conflict = detectConflicts(existing, merged);
    if (conflict) {
      conflicts.push(conflict);
      continue;
    }

    // 4. 写回资产
    writeRuntimeAsset(projectRoot, assetName, merged);
    updated.push(assetName);
  }

  // 5. 更新 index.json
  updateIndexTimestamps(projectRoot, updated);

  return { updated, conflicts, skipped: [] };
}
```

**合并策略**：

基础去重后，所有内容冲突交给 LLM 智能处理。

| 资产类型              | 基础去重策略           | LLM 智能合并场景                           |
| --------------------- | ---------------------- | ------------------------------------------ |
| `summary.json`        | 完全相同的模块名去重   | 同名模块不同实现 → LLM 判断是否合并/重命名 |
| `api-contracts.json`  | 完全相同的接口名去重   | 同名接口不同签名 → LLM 判断版本演进/冲突   |
| `critical-flows.json` | 完全相同的 flowId 去重 | 同 flowId 不同步骤 → LLM 判断是否互补      |
| `conventions.json`    | 完全相同的规则文本去重 | 相似规则不同表述 → LLM 提取最佳表述        |

**LLM 冲突解决接口**：

```typescript
interface ConflictResolution {
  strategy: 'merge' | 'rename' | 'keep-both' | 'manual';
  llmReasoning: string;
  suggestedAction: string;
  mergedContent?: any;
}

async function resolveMergeConflict(
  existing: any,
  incoming: any,
  context: {
    featureId: string;
    assetType: RuntimeAssetName;
    projectContext: FirstRuntimeSummary;
  }
): Promise<ConflictResolution>;
```

**LLM 处理示例**：

场景：Feature A 新增模块 `auth-module`，Feature B 也新增模块 `auth-module`

```typescript
// LLM 分析上下文
const prompt = `

- 职责：用户认证与授权

- 依赖：passport, oauth2
- 来源：specs/FEAT-B/design.md

判断：这是同一个模块的不同描述，还是两个不同的模块？
如果是不同模块，建议重命名方案。
`;

// LLM 返回
{
  strategy: 'rename',
  llmReasoning: '这是两个不同的认证模块，Feature A 是基础认证，Feature B 是第三方登录',
```

#### 3.7.1 回滚机制设计

**目的**：处理回写失败场景，确保系统状态一致性

**备份策略**：

```typescript
interface BackupSnapshot {

  featureId: string;

function createBackup(projectRoot: string, affectedAssets: RuntimeAssetName[]): BackupSnapshot;
```

**回滚触发条件**：

- 增量合并失败（LLM 冲突解决失败）
- 写入 runtime 资产失败（磁盘空间不足）
- 投影刷新失败（模板渲染错误）
- 用户手动取消回写操作

**实现要点**：

- 回写前创建备份快照
- 备份存储在 `.spec-first/runtime/first/.backup/` 目录
- 回滚后清理备份文件
- 记录回滚操作到治理日志

#### 3.7.2 幂等性保证设计

**目的**：支持重试，避免重复数据

**幂等性检查**：

```typescript
interface IdempotencyCheck {
  featureId: string;
  changeFingerprint: string; // 变更内容的哈希值
  lastApplied?: string; // 上次应用时间
}

function checkIdempotency(
  projectRoot: string,
  featureId: string,
  changes: StructuralChange[]
): 'new' | 'duplicate' | 'partial';
```

**实现策略**：

- 计算变更内容的哈希指纹
- 在治理日志中记录已应用的变更指纹
- 重复执行时跳过已应用的变更
- 支持部分重试（只应用未成功的变更）

**幂等性保证场景**：

- 用户修正 Feature 产物后重新归档
- CI 流程中归档失败后自动重试
- 手动触发增量刷新

#### 3.7.3 错误处理策略

**分层错误处理**：

| 层级       | 错误类型         | 处理策略                                | 是否回滚 |
| ---------- | ---------------- | --------------------------------------- | -------- |
| 变更识别层 | 产物格式不规范   | 记录警告，跳过该 Feature                | 否       |
| 资产映射层 | 映射规则缺失     | 降级到保守模式（只更新核心资产）        | 否       |
| 增量更新层 | LLM 冲突解决失败 | 标记冲突，等待人工介入                  | 是       |
| 增量更新层 | 磁盘空间不足     | 立即回滚，报错终止                      | 是       |
| 投影刷新层 | 模板渲染失败     | 标记 docs 为 stale，不阻断 runtime 更新 | 否       |
| 治理日志层 | 日志写入失败     | 记录到临时文件，不阻断流程              | 否       |

**错误恢复路径**：

- 可恢复错误：记录警告，继续执行
- 不可恢复错误：回滚并报错终止
- 部分失败：标记失败项，成功项保留

## 7. 关键技术点

### 7.1 变更识别准确性

**挑战**：Feature 产物格式不规范时如何识别变更

**解决方案**：

- 使用 LLM 语义分析而非正则匹配
- 提供多种证据来源（spec.md, design.md, task_plan.md）
- 容错处理：识别失败时记录警告，不阻断流程

### 7.2 LLM 冲突解决可靠性

**挑战**：LLM 判断可能不准确

**解决方案**：

- 提供充分的上下文（项目现有资产、Feature 定义）
- 要求 LLM 输出推理过程（llmReasoning）
- 用户可审查并修改 LLM 建议
- 关键冲突强制人工确认

### 7.3 增量合并性能优化

**挑战**：大型 runtime 资产加载慢

**解决方案**：

- 只加载受影响的资产（按需加载）
- 使用流式解析（避免全量加载到内存）
- 缓存已加载的资产（同一次回写中复用）

### 7.4 幂等性保证

**挑战**：如何判断变更是否已应用

**解决方案**：

- 计算变更内容的哈希指纹
- 在治理日志中记录已应用的指纹
- 重复执行时跳过已应用的变更

## 8. 风险与缓解

### 8.1 风险：LLM 冲突解决失败

**影响**：回写流程中断

**缓解措施**：

- 提供降级策略（保守合并）
- 标记冲突项，不阻断其他更新
- 记录失败原因，便于人工介入

### 8.2 风险：回写破坏现有数据

**影响**：项目级认知损坏

**缓解措施**：

- 回写前创建备份
- 失败时自动回滚
- 治理日志记录所有操作

### 8.3 风险：性能问题

**影响**：大型项目回写慢

**缓解措施**：

- 按需加载资产
- 增量更新而非全量重建
- 异步处理非关键操作（如 docs 投影）

## 3. 第二阶段技术改造方案

### 3.1 当前架构分析

**现有 prompt 组装流程**（`dispatcher.ts:loadSkill()`）：

```typescript
// 1. 加载 skill 模板
content = loadSkillTemplate(skillPath);

// 2. 变量替换（FEATURE_ID, CURRENT_STAGE 等）
if (enableAssembly) {
  ctx = resolvePromptAssemblyContext(executionContext);
  content = assemblePrompt(content, ctx);
}

// 3. 注入 runtime notice（按 skillName 分支）
if (skillName === 'spec') {
  content = `${buildSpecRuntimeNotice(ctx)}\n\n${content}`;
}
if (skillName === 'design') {
  content = `${buildDesignRuntimeNotice(ctx)}\n\n${content}`;
}
// ... 其他 skill
```

**关键发现**：

1. ✅ **已有注入机制**：每个 skill 都有对应的 `build*RuntimeNotice()` 函数
2. ✅ **已调用 resolveSkillContext()**：所有 `build*RuntimeNotice()` 内部都调用了 `resolveSkillContext()`
3. ✅ **已实现降级策略**：`context.backgroundInputStatus` 已传递到 notice 中
4. ❌ **注入内容不统一**：每个 skill 的 notice 格式和字段不一致
5. ❌ **token 控制缺失**：无法按预算裁剪注入内容

### 3.2 改造策略

**核心原则**：复用现有 `build*RuntimeNotice()` 机制，增强而非重写。

```
┌─────────────────────────────────────────────────────────────┐
│                    Skill Execution Layer                     │
│                   (loadSkill in dispatcher.ts)               │
└────────────────────────┬────────────────────────────────────┘
                         │
                         │ 1. 加载模板 + 变量替换
                         ↓
┌─────────────────────────────────────────────────────────────┐
│              Prompt Assembler (prompt-assembler.ts)          │
│  - assemblePrompt(template, ctx)                             │
│  - 替换 {{FEATURE_ID}}, {{CURRENT_STAGE}} 等                │
└────────────────────────┬────────────────────────────────────┘
                         │
                         │ 2. 注入 runtime notice（按 skillName）
                         ↓
┌─────────────────────────────────────────────────────────────┐
│         build*RuntimeNotice() (dispatcher.ts:485-560)        │
│  - 调用 resolveSkillContext(projectRoot, skillName)         │
│  - 格式化 notice（backgroundInputStatus + 资产摘要）        │
│  - 返回 markdown 格式的 notice                               │
└────────────────────────┬────────────────────────────────────┘
                         │
                         │ 3. 解析上下文（内部调用）
                         ↓
┌─────────────────────────────────────────────────────────────┐
│              Context Resolver (context-resolver.ts)          │
│  - 读取输入矩阵（getSkillInputContract）                    │
│  - 检测 backgroundInputStatus                                │
│  - 三层降级：runtime → docs → none                           │
│  - 返回 ResolvedSkillContext                                 │
└─────────────────────────────────────────────────────────────┘
```

#### 任务 2：context-resolver 增强

**目标**：增强 `resolveSkillContext()` 以支持 token 控制和字段裁剪

**实施点**：

1. 新增 `tokenBudget` 参数
2. 实现字段裁剪逻辑（按优先级裁剪 optional 资产）
3. 新增 `projectionDocs` 支持（按需返回投影文档路径）
4. 优化降级提示生成

**接口变更**：

```typescript
// context-resolver.ts
export interface ResolvedSkillContext {
  // ... 现有字段
  tokenBudget?: number;
  actualTokenUsage?: number; // 实际注入的 token 数
  trimmedAssets?: string[]; // 因 token 限制被裁剪的资产
  projectionDocs?: string[]; // 需要的投影文档路径
}

export function resolveSkillContext(
  projectRoot: string,
  skillName: string,
  options?: {
    tokenBudget?: number; // 覆盖默认 token 预算
    forceRuntime?: boolean; // 强制使用 runtime（测试用）
  }
): ResolvedSkillContext;
```

**裁剪策略**：

1. 优先保留 `required` 资产
2. 按以下顺序裁剪 `optional` 资产：
   - `reboot-guide`（最低优先级）
   - `entry-guide`
   - `change-map`
   - `critical-flows`
   - `conventions`（最高优先级）
3. 如果仍超限，裁剪 `required` 资产的非关键字段

**验收标准**：
- ✅ token 预算控制生效
- ✅ 裁剪策略符合预期
- ✅ `trimmedAssets` 正确记录被裁剪的资产
- ✅ 单元测试覆盖裁剪场景

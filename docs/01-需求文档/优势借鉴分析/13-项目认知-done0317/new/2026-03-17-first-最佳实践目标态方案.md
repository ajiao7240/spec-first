# 2026-03-17 first 最佳实践目标态方案

## 1. 方案目标

本方案不以“当前代码做到哪”为起点，而以最佳实践为起点，重新定义 `first` 在 `spec-first` 体系中的目标态。

核心目标只有一个：

> 让新人接手存量项目做需求迭代开发时，能以最低认知成本快速进入需求分析、技术方案、任务拆分、开发、测试与归档闭环。

因此，`first` 不应只是“生成一堆文档”，而应成为：

1. 项目级认知真源层
2. 全流程 skill 的公共背景输入层
3. Feature 完成后可持续更新的项目级知识底座

## 2. 最佳实践下的核心结论

### 2.1 单入口

`first` 只能有一个正式入口：

```bash
spec-first first
```

不保留：

- quick / deep 双模式
- CLI 与 skill 双默认入口
- 多 Agent 编排作为默认执行路径

增强分析可以存在，但只能是可选增强，不是默认主链。

### 2.2 单真源

正式真源固定为：

- `.spec-first/runtime/first/*.json`

正式投影视图固定为：

- `docs/first/*.md`

必须坚持：

1. 先生成 runtime 真源
2. 再统一投影 docs
3. 下游节点优先读取 runtime
4. docs 只作为阅读视图，不得回灌为真源

### 2.3 强治理、弱前置

`first` 对系统质量是强约束，但对流程执行不是硬前置。

换句话说：

- 有 `first` 时，下游 skill 应获得更高质量上下文
- 没有 `first` 时，任何 skill 仍必须继续执行
- 差别只能体现在背景质量和结论置信度上，不能体现在“能不能运行”上

### 2.4 全链路持续更新

`first` 不是初始化产物，而是持续治理对象。

目标态要求：

1. 项目初次接手时生成
2. 日常开发中被下游节点消费
3. 在 `wrap_up / done` 时自动同步更新
4. 始终保持为当前项目的最新认知底座

## 3. 目标态架构

### 3.1 三层架构

#### 第一层：Runtime Truth Layer

职责：

- 存放结构化、机器可消费的项目级认知资产
- 为下游 skill 提供稳定上下文切片

项目级正式 runtime 资产建议收口为 9 个：

| 中文名 | 英文文件 | 作用说明 |
| --- | --- | --- |
| 项目摘要 | `summary.json` | 项目级总入口真源，汇总项目名称、平台类型、技术栈、模块、能力、入口、数据模型、API 面、风险与证据，供其他 runtime 资产和 `README/summary` 投影汇总使用。 |
| 项目导向 | `steering.json` | 固化项目的产品目标、核心场景、非目标、术语表，以及技术约束、禁止模式、结构边界和入口规则，是设计与方案阶段的重要约束真源。 |
| 项目规范 | `conventions.json` | 以 `api / module / testing / projectRules` 四组约定记录观察到的模式、偏差、推荐规范和证据，用于代码实现、审查、验证和本地协作。 |
| 关键流程 | `critical-flows.json` | 记录项目的关键流程清单，每条流程包含入口、核心模块、不变量和验证钩子，用于设计、实现、回归和风险分析。 |
| 入口指南 | `entry-guide.json` | 按任务类别给出 `readFirst / thenRead / avoidEntry / relatedFlows`，用于新人接手项目和不同类型任务的最小进入路径。 |
| 接口契约 | `api-contracts.json` | 记录接口契约字段及其 `interfaceType / name / method / path / handler / request / response / auth / errors / evidence`，只服务 API 契约，不承载外部依赖综述，也不以“统计系统有多少接口”为目标。 |
| 结构总览 | `structure-overview.json` | 记录代码拓扑、模块清单、各模块职责、关键路径、入口关系和阅读顺序，是 `codebase-overview / architecture / tech-stack` 等投影的核心结构真源。 |
| 领域模型 | `domain-model.json` | 记录核心实体或概念的类型、说明、不变量、关系和证据，并补充 glossary，用于需求理解、方案设计和审查。 |
| 数据库结构 | `database-schema.json` | 记录数据库状态、提供方、表结构、字段、关系、风险和证据；同时决定 `database-er.md` 是否以 `healthy / not_applicable / degraded` 状态生成。 |

索引文件：

- `index.json`

不再纳入 `first` 核心模型的内容：

- `role-views.json`
- `stage-views.json`
- `change-map.json`
- `reboot-guide.json`

#### 第二层：Projection Docs Layer

职责：

- 提供适合人阅读的项目认知文档
- 让不同角色在不读 JSON 的情况下快速进入项目

项目级正式文档建议收口为 14 个主集合，其中包含 1 个条件型文档：

| 中文名 | 英文文件 | 作用说明 |
| --- | --- | --- |
| 总入口 | `README.md` | `docs/first` 的统一入口，负责文档导航、阅读顺序、条件型文档状态和使用建议。 |
| 项目总览 | `summary.md` | 面向所有角色的项目摘要，说明项目是什么、主要能力、主要入口、核心风险和证据摘要。 |
| 项目导向 | `steering.md` | 展示项目目标、能力边界、技术约束、结构边界和核心原则。 |
| 项目规范总览 | `conventions.md` | 汇总 API、模块、测试、项目规则四类约定，是实现、审查和验证前的规则入口。 |
| 关键流程总览 | `critical-flows.md` | 展示核心业务/技术链路、关键入口、不变量和验证钩子。 |
| 入口指南 | `entry-guide.md` | 面向新人或新任务，说明应先读什么、再读什么、避免从哪里进入。 |
| 接口规范文档 | `api-docs.md` | 只服务正式接口规范，展示已识别接口的请求规范、响应规范、鉴权要求、错误语义和证据，不以“全量接口统计”作为目标。 |
| 代码库总览 | `codebase-overview.md` | 展示目录结构、模块划分、入口关系、阅读顺序和代码导航地图。 |
| 领域模型文档 | `domain-model.md` | 展示核心业务对象、概念边界、关系、关键语义和 glossary。 |
| 架构视图 | `architecture.md` | 深入展示系统整体结构、模块协作关系、技术边界和架构风险。 |
| 调用链视图 | `call-graph.md` | 展示关键入口到核心逻辑的主调用路径、扩散点与高风险传播关系。 |
| 外部依赖视图 | `external-deps.md` | 展示第三方依赖、外部服务、集成边界、依赖风险和升级注意事项。该文档为组合投影，真源来自 `summary.json`、`steering.json`、`conventions.json`，不单独新增 `external-deps.json`。 |
| 开发规范指南 | `development-guidelines.md` | 展开说明开发规范、测试规范、配置约定、协作方式和项目级开发守则。 |
| 数据库 ER 视图 | `database-er.md` | 条件型文档，仅在 `databaseSchema.status === healthy` 时生成，用于展示表结构、关系、关键字段和数据风险。 |

建议删除、不再作为项目级正式文档的内容：

- `role-views.md`
- `stage-views.md`
- `change-map.md`
- `reboot-guide.md`
- `common-playbooks.md`
- `known-risks-and-traps.md`
- `tech-stack.md`
- `local-setup.md`

#### 第三层：Skill Consumption Layer

职责：

- 为每个 downstream skill 定义其最小必要输入
- 控制不同节点读取哪些 runtime / docs
- 当 `first` 缺失时提供分层降级

### 3.2 分析能力分层

方案层不应绑定具体工具，而应绑定分析能力。

`first` 的目标态分析能力应固定为六类：

1. 结构分析
   - 产出 `structure-overview.json`
   - 识别目录结构、模块边界、阅读顺序、入口分布

2. 接口契约提取
   - 产出 `api-contracts.json`
   - 提取接口契约、请求/响应约束、鉴权与错误语义

3. 领域模型提取
   - 产出 `domain-model.json`
   - 识别核心概念、实体边界、关系与关键语义

4. 数据结构提取
   - 产出 `database-schema.json`
   - 识别数据库提供方、表结构、关系与数据风险

5. 配置与依赖解析
   - 补强 `summary.json`、`steering.json`、`conventions.json`
   - 识别技术栈、依赖约束、环境前提与工程规则

6. 项目导航生成
   - 产出 `entry-guide.json`
   - 组织项目进入路径、阅读路径和最小导航提示

### 3.3 推荐文档分层

#### 启动层

默认优先暴露：

- 总入口 - `README.md`
- 项目总览 - `summary.md`
- 入口指南 - `entry-guide.md`
- 代码库总览 - `codebase-overview.md`
- 项目导向（快速浏览目标与边界） - `steering.md`

作用：

- 帮新人快速知道项目是什么
- 从哪里开始读
- 从哪里开始改
- 改动会影响哪里
- 技术栈信息统一并入 `summary.json.techStack`，并在 `summary.md` 中展示，不再单列 `tech-stack.md`

#### 开发层

按开发任务读取：

- 开发规范指南 - `development-guidelines.md`
- 接口规范文档 - `api-docs.md`
- 关键流程总览 - `critical-flows.md`
- 外部依赖视图 - `external-deps.md`

作用：

- 帮研发进入具体实现、验证和回归

#### 专题层

复杂场景按需读取：

- 架构视图 - `architecture.md`
- 调用链视图 - `call-graph.md`
- 领域模型文档 - `domain-model.md`
- 数据库 ER 视图 - `database-er.md`
- 项目导向（完整版：含技术约束与禁止模式） - `steering.md`
- 项目规范总览 - `conventions.md`

原则：

- 只保留对项目级认知有长期价值的专题文档
- 但默认阅读路径必须尽量短

## 4. 可观测性与质量门禁

### 4.1 可观测性目标

`first` 不只要能生成，还必须可观测、可判断、可解释。

目标态至少应具备以下观察能力：

1. 资产健康度
   - 每个 runtime 资产都有 `healthy / degraded / not_applicable`

2. 投影一致性
   - 能判断 runtime 与 docs 是否一致
   - 能判断是否存在缺失投影、孤儿投影、过期投影

3. 新鲜度
   - 能标记 `fresh / stale`
   - stale 不阻断流程，但必须降低置信度

4. 背景质量
   - downstream skill 能看到当前是 `full / degraded / blind`
   - 并据此调整结论强度

### 4.2 Index 的最佳实践职责

`index.json` 在目标态中不只是“路径清单”，还应承担四项职责：

1. 资产注册表
2. 健康状态索引
3. 文档投影索引
4. 新鲜度与更新时间索引

但 `index.json` 不应膨胀成复杂历史数据库。

### 4.3 最小门禁

目标态至少应有以下门禁：

1. runtime schema 校验
2. projection completeness 校验
3. cross-asset consistency 校验
4. 条件型文档适用性校验

示例：

- `databaseSchema.status !== healthy` 时不得生成 `database-er.md`
- `api-contracts.json` 中涉及的核心概念，应能在 `domain-model.json` 中找到对应语义对象或证据

## 5. first 的正式职责

### 5.1 对用户的职责

`first` 应向用户提供以下价值：

1. 快速理解项目定位、核心能力和风险
2. 快速定位主要代码结构和入口
3. 快速理解项目接口规范、结构边界和核心领域对象
4. 为下游各阶段提供统一项目背景

### 5.2 对系统的职责

`first` 应向系统提供以下能力：

1. 稳定的项目级背景输入
2. 可增量更新的认知资产
3. 可治理、可审计的项目认知写回
4. 跨 skill 的统一输入 contract
5. 可观测、可校验、可解释的运行状态

### 5.3 项目级别文档表

| 文档/资产 | 是否应由 `first` 维护 | 是否应由 `init/feature` 维护 | 原因 |
| --- | --- | --- | --- |
| `summary.json` / `summary.md` | 是 | 否 | 描述整个项目是什么、能力边界、风险、入口，跨多个需求复用。 |
| `steering.json` / `steering.md` | 是 | 否 | 记录项目目标、技术约束、结构边界，属于全局规则，不绑定单一需求。 |
| `conventions.json` / `conventions.md` | 是 | 否 | 记录 API、模块、测试、项目规则等共用规范。 |
| `critical-flows.json` / `critical-flows.md` | 是 | 否 | 记录项目关键链路，供多个需求共用。 |
| `entry-guide.json` / `entry-guide.md` | 是 | 否 | 帮新人理解“从哪里进入项目”，不是某个需求独有。 |
| `api-contracts.json` / `api-docs.md` | 是 | 否 | 描述项目接口契约与接口规范，属于全局认知；不等于统计系统一共有多少个接口。 |
| `structure-overview.json` / `codebase-overview.md` | 是 | 否 | 描述整体结构和模块边界，显然是项目级。 |
| `domain-model.json` / `domain-model.md` | 是 | 否 | 描述核心领域对象和关系，通常跨需求复用。 |
| `database-schema.json` / `database-er.md` | 是 | 否 | 描述数据库结构，是系统底层事实。 |

### 5.4 明确不负责的事情

`first` 不应负责：

1. 直接推动 Feature 生命周期
2. 替代 spec / design / task / code / verify 本身的专业能力
3. 作为 hard gate 阻断流程
4. 让多个 Agent 直接生成 markdown 真源

## 6. 下游 Skill 的最佳实践接入方式

### 6.1 总原则

下游 skill 消费 `first` 时，必须遵守：

1. 优先读 runtime 真源
2. docs 只作为辅助阅读视图
3. 每个 skill 只读与自己职责有关的资产
4. 不允许泛化写成“读取 docs/first/”
5. 装载逻辑必须支持按需加载，而不是默认读全量资产
6. 任意节点在 `first` 文档或 runtime 缺失时，都只能降级执行，不得阻断主流程

### 6.2 按需加载

按需加载是 `first` 成为公共上下文层的关键前提。

目标态要求：

1. `context-resolver` 只装载当前 skill 的 `requiredRuntimeAssets`
2. 若有需要，再装载 `optionalRuntimeAssets`
3. projection docs 只作为二级补充，不作为默认全量加载内容
4. 下游节点允许在执行过程中显式请求更多资产，但默认不预装

这样做的收益：

1. 降低上下文体积
2. 减少 token 浪费
3. 降低无关信息污染
4. 让每个节点的“背景质量下降”更可解释
5. 保证 `first` 永远是增强层，而不是硬前置依赖

### 6.2.1 上下文注入与 Token 控制

为避免多个节点重复注入相同项目级背景导致 token 成本线性上升，目标态应增加以下约束：

1. 表中列出的资产表示“推荐访问”或“允许访问”，不表示默认全文注入。
2. 默认只注入最小摘要，不注入整份 runtime 或整篇 markdown。
3. 同一主题默认优先注入 runtime 摘要，不同时注入对应 projection docs 全文。
4. docs 只在需要自然语言章节说明时按需展开。
5. 连续节点执行时，应复用同一份项目级摘要缓存，不重复拼装相同上下文。

建议把注入粒度固定为三层：

1. `L0` 元信息
   - 资产是否存在
   - 健康状态
   - 最后更新时间
2. `L1` 关键字段摘要
   - 只暴露每个 runtime 资产的最小可判定字段
3. `L2` 全文或长片段
   - 仅当节点显式需要时才展开

默认规则：

1. 先用 `L0/L1`
2. 不默认进入 `L2`
3. 不默认同时注入 `runtime + 对应 docs`
4. 同一轮链路中优先复用已有摘要，而不是重复重新注入

`L1` 建议固定为可校验的摘要 schema，而不是自由文本。建议如下：

| 运行时资产 | `L1` 关键字段摘要 |
| --- | --- |
| `summary.json` | `projectWhat: string`、`mainCapabilities: string[]`、`techStack: string[]`、`platformType: string` |
| `steering.json` | `goals: string[]`、`nonGoals: string[]`、`constraints: string[]`、`boundaries: string[]` |
| `conventions.json` | `apiRules: string[]`、`moduleRules: string[]`、`testingRules: string[]`、`projectRules: string[]` |
| `critical-flows.json` | `names: string[]`、`count: number`、`invariants: string[]` |
| `entry-guide.json` | `readFirst: string[]`、`thenRead: string[]`、`avoidEntry: string[]`、`categories: string[]` |
| `api-contracts.json` | `interfaceNames: string[]`、`count: number`、`interfaceTypes: string[]`、`authModes: string[]` |
| `structure-overview.json` | `moduleNames: string[]`、`entryPoints: string[]`、`topDirs: string[]`、`count: number` |
| `domain-model.json` | `entityNames: string[]`、`count: number`、`glossaryTerms: string[]` |
| `database-schema.json` | `status: string`、`provider: string`、`tableNames: string[]`、`count: number` |
| `index.json` | `healthyAssets: string[]`、`degradedAssets: string[]`、`staleAssets: string[]`、`lastUpdated: string` |

约束：

1. `L1` 只允许摘要字段，不允许直接内嵌长文本章节。
2. 每个资产的 `L1` 字段必须可被程序校验，不能依赖人工解释。
3. 节点矩阵中的 `required/optional` 默认消费的就是这些 `L1` 字段，而不是全文。

### 6.3 节点级输入矩阵

下表给出各 downstream skill 最值得读取的项目级文档。原则是：

1. 先读 runtime 真源
2. 再按需补读 projection docs
3. 每个节点只读能直接提升其结论质量的那一小组文档

统一约束：

1. 表中的“必需”表示“在 `first` 存在时优先装载的最小项目级背景”，不是流程阻断条件。
2. 表中的“可选”表示“存在时进一步提升质量的补充背景”。
3. 无论 `first` 文档是否存在，节点都必须继续执行自己的主流程，不得因为 `first` 缺失而报错或停止。
4. 表中资产默认按摘要级别注入，不表示整份文档全文进入 prompt。

### 6.4 输入矩阵的最佳实践

目标态不应把输入矩阵散落在各 skill 文档里，而应有一份统一 declarative contract。

建议新增：

- `src/core/skill-runtime/first-skill-input-matrix.ts`

至少定义：

1. `requiredRuntimeAssets`
2. `optionalRuntimeAssets`
3. `allowedProjectionDocs`
4. `degradationLevel`
5. `recommendedActionWhenMissing`

这样：

- `context-resolver.ts` 可直接消费
- `dispatcher.ts` 可直接消费
- skill 文档只是对外说明，不再是唯一真理源

### 6.5 校验与刷新能力

目标态必须具备“真源校验”和“增量刷新”两类能力，但不建议把它们扩展成新的正式用户入口命令。

原因：

1. `first` 的正式入口仍应保持唯一：`spec-first first`
2. `validate / refresh` 更适合作为 `first` 内部能力、`doctor` 诊断能力或实现细节，不再额外扩出新的命令面

建议保留的能力：

1. 真源校验能力
   - 校验 `index.json` 是否完整登记正式 runtime 资产
   - 校验 runtime schema、projection 完整性、一致性与条件型文档适用性
   - 可由 `first` 执行结束后自动触发，也可被 `doctor` 复用

2. 增量刷新能力
   - 基于变更检测刷新受影响 runtime 资产与对应 docs
   - 作为 `first` 的内部流程存在，不要求用户切换到单独子命令

结论：

- 保留“校验”和“刷新”能力
- 不在目标态方案中把它们定义成新的正式用户命令

## 7. 07 / 08 节点与项目级认知回写

### 7.1 最佳实践判断

Feature 做完以后，如果项目级认知没有同步更新，这套 `first` 体系就会快速失效。

因此，最佳实践下必须要求：

- 每个 Feature 的结构性变更都能回流到 `first`

### 7.2 正确分工

#### code / review 的职责

不直接修改 `docs/first`。

只负责：

1. 识别本次变更影响了哪些项目级认知资产
2. 输出认知影响信号
3. 在审查阶段标记是否存在未同步的认知变更

#### archive / sync 的职责

负责真正执行项目级认知治理。

具体包括：

1. 判断是否需要刷新 `first`
2. 刷新哪些 runtime 资产
3. 再统一投影哪些 docs
4. 输出可审计 writeback 结果

### 7.3 回写时点

必须在两个时点触发：

1. `06_wrap_up -> 07_release`
2. `07_release -> 08_done`

### 7.4 回写范围

项目级认知回写至少覆盖：

- 项目总览变化 -> `summary`
- 核心能力边界变化 -> `steering`
- 关键流变化 -> `critical-flows`
- 开发入口变化 -> `entry-guide`
- API 变化 -> `api-contracts`
- 结构变化 -> `structure-overview`
- 领域模型变化 -> `domain-model`
- 数据结构变化 -> `database-schema`
- 规范变化 -> `conventions`

### 7.5 最佳实践约束

项目认知 writeback 必须：

1. 由治理层统一执行
2. 基于 runtime-to-docs projection registry
3. 有日志、有证据、有状态
4. 不允许由业务节点手工改 `docs/first/*.md`

### 7.6 追溯性

目标态需要的不是优先新增很多查询命令，而是先把追溯能力落到治理日志中。

最小追溯字段应包括：

1. `triggerStage`
2. `featureId`
3. `changedFiles`
4. `updatedRuntimeAssets`
5. `updatedDocs`
6. `decision`
7. `writebackPerformed`
8. `evidence`

这样可以回答三个关键问题：

1. 哪个 Feature 影响了项目级认知
2. 它更新了哪些 runtime 资产
3. 它刷新了哪些 docs/first 文档

## 8. 缺失 first 时的最佳实践

### 8.1 核心原则

没有 `first` 时，系统必须是：

- 可继续执行的
- 可解释的
- 低置信度但不报错的

### 8.2 三层降级

#### 第一层：前置层

节点：

- `init`
- `spec`
- `design`
- `task`

策略：

- 继续执行
- 提示“存量项目建议先执行 first”

#### 第二层：开发验证层

节点：

- `research`
- `code`
- `review`
- `verify`

策略：

- 继续执行
- 明确提示“背景不足会降低影响面判断和回归质量”

#### 第三层：治理分析层

节点：

- `catchup`
- `orchestrate`
- `status`
- `doctor`
- `archive`
- `sync`
- `spec-review`
- `analyze`

策略：

- 继续执行
- 明确声明当前是弱背景模式
- 降低结论置信度

### 8.3 fallback 顺序

最佳实践下，fallback 顺序必须固定：

1. runtime
2. docs
3. none

且必须满足：

1. docs fallback 不依赖 runtime index
2. docs fallback 只是阅读辅助，不伪造 runtime slices
3. `none` 也是合法状态，不是异常状态

### 8.4 partial / stale / missing

缺失 `first` 不能只分成“有 / 没有”两类，目标态应至少区分三种背景退化：

1. `missing`
   - 没有 runtime，也没有 docs
   - 属于最弱背景状态

2. `partial`
   - 有部分 runtime 资产，或只有 docs fallback
   - 可继续执行，但必须降低结论完整性

3. `stale`
   - 资产存在，但超过合理更新时间窗口，或被变更检测判定过期
   - 可继续执行，但必须降低置信度，并给出刷新建议

### 8.5 错误处理原则

缺失 `first` 时不允许：

1. 抛异常
2. 中断节点执行
3. 输出与真实行为不一致的“必须先运行 first”文案

允许：

1. 告警
2. 标记 `background_input_status=degraded/blind`
3. 给出建议动作 `spec-first first`

## 9. 文档与实现的最佳实践

### 9.1 文档必须服务执行，不服务展示

`docs/first` 的目标不是“完整”，而是“帮助人尽快开始做事”。

因此每份文档都应回答一个明确问题：

- 这是什么项目
- 从哪里开始看
- 从哪里开始改
- 风险在哪
- 验证什么

### 9.2 避免重叠

目标态应控制：

1. `summary` 不展开架构
2. `entry-guide` 不替代环境文档
3. `codebase-overview` 不替代架构与调用链
4. `domain-model` 不替代数据库结构
5. `api-docs` 只服务接口规范，不承载外部依赖总览

### 9.3 中文输出 contract

正式 projection docs 默认使用中文，保留：

- 路径
- 命令
- 标识符
- 技术术语

如果实现与文档规范冲突，优先修 renderer 与测试，不接受规范和产物长期漂移。

## 10. 推荐目标态执行流程

### 10.1 默认路径

```text
用户执行 spec-first first
  -> 判断是否已有健康 runtime
  -> 若无：bootstrap 全量 runtime
  -> 若有：按变更增量刷新 runtime
  -> 更新 index.json
  -> 最后统一投影 docs/first
```

### 10.2 增强路径

只有以下场景才按需增强：

1. `call-graph`
2. `architecture`
3. `domain-model`
4. `api-docs`

增强路径可以引入：

- 更深符号分析
- Serena / LSP
- Agent 执行提示

但不能改变：

- 单入口
- runtime-first
- docs 最终统一投影

### 10.3 持续治理路径

```text
code / review 识别项目认知影响
  -> archive / sync 触发治理判断
  -> 选择需刷新 runtime assets
  -> 统一更新 docs/first
  -> 写回治理日志
```

## 11. 目标态实施优先级

### P0

把 `first` 正式定义为：

- 单入口
- 单真源
- 强治理
- 弱前置

### P1

把下游 skill 输入矩阵代码化，避免继续散落在 skill 文档。

### P2

把 `archive / sync` 的项目认知 writeback 升级为正式治理 contract。

### P3

把缺失 `first` 的降级规则彻底统一到：

- `context-resolver`
- `dispatcher`
- skill 文档
- 测试

### P4

把项目级正式文档主集合按“启动层 / 开发层 / 专题层”组织，不再平铺暴露。

## 12. 验收标准

满足以下条件，才算达到最佳实践目标态：

1. `first` 只有一个正式入口
2. `.spec-first/runtime/first` 是唯一正式真源
3. `docs/first` 全部由 runtime 投影生成
4. downstream skill 有统一输入矩阵
5. 没有 `first` 时，任何 skill 都不会报错
6. `archive / sync` 能自动把结构性变更回写到 `first`
7. 新人第一次接手项目时，默认阅读路径不超过 6 份核心文档
8. 开发、评审、验证节点都能基于 `first` 获得更高质量上下文
9. 系统能区分 `missing / partial / stale` 三种背景状态
10. `index.json` 能反映健康度、新鲜度和投影一致性

## 13. 最终结论

最佳实践下，`first` 的终态不是“文档生成器”，而是：

- 项目级认知真源系统
- 全流程 skill 的公共背景输入层
- Feature 生命周期末端自动更新的项目知识底座

从用户价值倒推，真正应该做到的是：

1. 新人接手老项目时，第一时间知道看什么、改什么、风险在哪。
2. 后续需求、设计、任务、开发、测试节点都能自动带上更好的项目背景。
3. 每次需求迭代完成后，这套项目认知仍然是最新的。

这就是 `first` 最佳实践方案的终态定义。

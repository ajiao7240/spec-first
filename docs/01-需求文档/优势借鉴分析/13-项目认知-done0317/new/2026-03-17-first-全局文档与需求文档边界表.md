# 2026-03-17 first 全局文档与需求文档边界表

## 目标

明确 `first` 应维护哪些**项目全局文档**，以及哪些内容应由 `init + Feature 生命周期` 维护为**需求维度文档**，避免项目级认知和需求级推进状态混放。

## 边界原则

1. 跨多个需求长期复用的，归项目全局文档。
2. 只服务某个 `featureId` 的，归需求维度文档。
3. 描述项目稳定事实的，归 `first`。
4. 描述需求推进状态、阶段动作、交付物的，归 `specs/{featureId}`。

## 项目级别文档表

| 文档/资产 | 是否应由 `first` 维护 | 是否应由 `init/feature` 维护 | 原因 |
| --- | --- | --- | --- |
| `summary.json` / `summary.md` | 是 | 否 | 描述整个项目是什么、能力边界、风险、入口，跨多个需求复用。 |
| `steering.json` / `steering.md` | 是 | 否 | 记录项目目标、技术约束、结构边界，属于全局规则，不绑定单一需求。 |
| `conventions.json` / `conventions.md` | 是 | 否 | 记录 API、模块、测试、项目规则等共用规范。 |
| `critical-flows.json` / `critical-flows.md` | 是 | 否 | 记录项目关键链路，供多个需求共用。 |
| `entry-guide.json` / `entry-guide.md` | 是 | 否 | 帮新人理解“从哪里进入项目”，不是某个需求独有。 |
| `api-contracts.json` / `api-docs.md` | 是 | 否 | 描述项目接口契约与接口规范，属于全局认知；不等于统计系统一共有多少个接口。 |
| `structure-overview.json` / `codebase-overview.md` | 是 | 否 | 描述整体结构和模块边界，显然是项目级。 |
| `domain-model.json` / `domain-model.md` | 是 | 否 | 描述核心领域对象和关系，通常跨需求复用。 |
| `database-schema.json` / `database-er.md` | 是 | 否 | 描述数据库结构，是系统底层事实。 |
| `role-views.json` / `role-views.md` | 可并入，不建议单独强保留 | 否 | 更像阅读导航包装层，可并入 `README.md` 或 `entry-guide.md`。 |

## 需求级别文档表

| 文档/资产 | 是否应由 `first` 维护 | 是否应由 `init/feature` 维护 | 原因 |
| --- | --- | --- | --- |
| `stage-views.json` / `stage-views.md` | 否，建议迁出 | 是 | 回答的是“某阶段该看什么”，更像 Feature 生命周期节点上下文，不是全局项目事实。 |
| `specs/{featureId}/prd.md` / `spec.md` | 否 | 是 | 只描述某个需求的范围、目标、验收。 |
| `specs/{featureId}/design.md` | 否 | 是 | 只描述某个需求的技术方案。 |
| `specs/{featureId}/task_plan.md` | 否 | 是 | 只服务当前需求拆解与推进。 |
| `specs/{featureId}/traceability-matrix.md` | 否 | 是 | 只追踪当前需求链路。 |
| `specs/{featureId}/findings.md` | 否 | 是 | 只记录当前需求执行中的发现与审计结果。 |
| `specs/{featureId}/retro.md` | 否 | 是 | 是本次需求复盘，不是全局项目认知。 |
| `specs/{featureId}/impact-analysis.md` | 否 | 是 | 只针对当前需求的影响分析。 |
| `stage-state.json` / `.spec-first/current` | 否 | 是 | 属于 Feature 生命周期和当前上下文状态，不是 `first` 的职责。 |

## 收口建议

### 项目全局文档

应由 `first` 负责维护，并在 `wrap_up / done` 时通过治理机制同步更新。

### 需求维度文档

应由 `init / spec / design / task / code / verify / archive / sync` 这条 Feature 生命周期负责维护。

### 需要重点迁出的内容

1. `stage-views.json`
2. `stage-views.md`

### 需要重点降级的内容

1. `role-views.json`
2. `role-views.md`

### 当前不纳入 first 项目级正式文档的内容

1. `change-map.json`
2. `change-map.md`
3. `reboot-guide.json`
4. `reboot-guide.md`

## 结论

`first` 应维护的是**项目全局稳定认知**，不是**需求推进状态**。  
凡是绑定 `featureId`、阶段推进、任务执行、审计与复盘的内容，都应归需求维度文档，而不是继续堆进 `first`。

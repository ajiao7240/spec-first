# 2026-03-17 first 缺失场景分层降级优化方案

> 当前状态：`核心实现已完成`。已落地内容包括：`context-resolver` 分层降级、`dispatcher` 分层提示、`docs fallback` 与 `index.json` 解耦、相关节点 `SKILL.md` 文档口径同步与回归测试。

## 一、问题背景

当前 `spec-first` 已将 `first` 定义为项目级认知输入层，但从代码实现看，`first` 并不是严格前置条件：

1. `context-resolver.ts` 在 `first` 缺失时会返回 `source: 'none'`
2. `dispatcher.ts` 会注入告警与建议，但不会直接阻断节点执行
3. 真正的 hard-gate 仍然是阶段、Feature、Gate、TDD、Worktree 等约束，而不是 “必须先执行 first”

这说明当前系统的主方向是正确的：

- `first` 是辅助输入
- 不是所有流程的必要条件

但当前降级策略仍然过于粗糙，主要问题有三类：

1. 所有节点统一使用“建议先运行 `/spec-first:first`”的弱提示，缺少场景分层
2. `docs fallback` 仍依赖 `index.json` 健康状态，导致“只有 docs、没有 runtime”的弱回退链路不成立
3. 节点文档中的“需要 first 资产”表述较强，而代码实际行为只是“缺失时带告警继续”

因此，需要为“未执行 first / first 产物缺失 / first 只存在部分产物”的场景建立分层降级策略。

## 二、目标

本方案的目标不是把 `first` 升级为强依赖，而是把“缺失 first 时如何继续执行”设计成稳定、清晰、可解释的系统行为。

目标如下：

1. 新项目在没有代码、没有 `docs/first`、没有 runtime 的情况下，流程仍可继续
2. 存量项目在缺少 `first` 时，系统能明确告知风险，而不是静默退化
3. 不同节点根据其对项目级认知的依赖程度，使用不同级别的降级提示
4. `docs/first` 若真实存在，即使没有 runtime index，也应允许作为弱 fallback
5. 节点文档、上下文装载、提示语和测试口径保持一致

## 三、核心设计原则

### 3.1 first 是辅助输入，不是硬前置

`first` 的定位保持不变：

- 用于增强项目级背景输入质量
- 用于帮助存量项目、老需求迭代、会话恢复
- 不是新 Feature 生命周期的强前置条件

这意味着：

- `init / spec / design / task` 不能因为没有 first 而被阻断
- `code / review / verify / doctor / analyze / status / archive` 也不应直接阻断
- 但后者需要更强的风险提示

### 3.2 降级行为必须按节点分层

不能继续用单一策略覆盖所有节点。

更合理的做法是按“对项目级认知的依赖强度”分层：

1. 需求与设计前置层
2. 开发与验证核心层
3. 治理、诊断与分析层

### 3.3 docs fallback 必须独立于 runtime index

当前 docs fallback 的判断依赖：

- `index.json` 存在
- docs projection 状态为 healthy

这会导致一种不合理情况：

- `docs/first/*.md` 明明真实存在
- 但因为没有 `.spec-first/runtime/first/index.json`
- 系统仍然不能把 docs 当作弱 fallback

终态应该改为：

- runtime fallback 与 docs fallback 分离判断
- docs 是否可用，优先看 docs 文件本身是否存在且可解析

## 四、分层降级策略

### 4.1 第一层：需求与设计前置层

适用节点：

- `init`
- `spec`
- `design`
- `task`

策略：

1. 缺少 `first` 时默认继续执行
2. 不阻断，不升级为 hard-gate
3. 提示语应明确区分两种场景：
   - 新项目：`可跳过 first，继续当前流程`
   - 存量项目：`建议补跑 first，以获取项目级背景输入`

推荐提示语：

> 当前未检测到 `first` 项目认知资产。若这是新项目或尚未形成代码结构，可继续当前流程；若这是存量项目或需要在旧代码上迭代，建议先运行 `/spec-first:first` 补齐项目级背景输入。

### 4.2 第二层：开发与验证核心层

适用节点：

- `code`
- `review`
- `verify`
- `research`

策略：

1. 缺少 `first` 时仍继续执行
2. 但告警级别明显高于第一层
3. 需明确说明：
   - 可继续执行
   - 但项目级背景不足
   - 当前输出可信度、完整性或回归质量会下降

推荐提示语：

> 当前未检测到完整的 `first` 项目认知资产。流程可以继续，但项目级背景输入不足，当前实现/审查/验证结果的完整性与可信度会下降。若项目已存在代码或需要识别影响面、关键流、规范约束，建议优先运行 `/spec-first:first`。

### 4.3 第三层：治理、诊断与分析层

适用节点：

- `doctor`
- `analyze`
- `status`
- `archive`
- `sync`
- `catchup`
- `orchestrate`
- `spec-review`

策略：

1. 缺少 `first` 时继续执行
2. 但应显式降级结果类型
3. 输出中要标注：
   - 当前结论是“弱背景模式”下的结果
   - 哪些判断因为缺少 `first` 而不能成立
   - 哪些建议仅是保守建议，而非高置信结论

推荐输出元信息：

- `background_mode: weak`
- `project_cognition_status: missing`
- `confidence_impact: degraded`
- `recommended_action: /spec-first:first`

## 五、docs fallback 优化方案

### 5.1 当前问题

当前 `resolveSkillContext()` 中，docs fallback 走的是：

1. 判断对应 docs 是否存在
2. 再判断 `hasHealthyCanonicalDocsFallback(projectRoot)`

而 `hasHealthyCanonicalDocsFallback()` 又要求：

- runtime index 存在
- docs projection 在 index 中标记 healthy

这使 docs fallback 不是独立 fallback，而是 runtime 健康检查的附属能力。

### 5.2 目标行为

应将 docs fallback 重构为独立链路：

1. 若 runtime 可用，优先使用 runtime
2. 若 runtime 不可用，但 docs 文件真实存在且可解析，则使用 docs fallback
3. 若 docs 也不存在，再返回 `source: 'none'`

### 5.3 推荐拆分

建议将当前逻辑拆成三组判断：

1. `hasHealthyRuntimeFallback(projectRoot)`
2. `hasAvailableDocsFallback(projectRoot, requiredDocs?)`
3. `canParseDocsFallback(projectRoot, skillName, stageKey?)`

其中第二组不再依赖 `index.json`，而改为：

- 文件存在
- 至少关键 docs 可读取
- 标题/章节结构满足最小解析要求

### 5.4 docs fallback 的定位

必须明确：

- docs fallback 是弱 fallback
- 只提供阅读层辅助，不提供结构化 runtime slices
- 若走 docs fallback，`required` / `optional` 中的 runtime 真源对象仍为空
- 但允许提供：
  - `stageViewSummary`
  - `roleViewSummary`
  - 基于 docs 的弱提示

## 六、代码改造建议

### 6.1 context-resolver.ts

主要改造点：

1. 将“节点分层”显式化
   - 新增 `FIRST_CONTEXT_DEGRADATION_LEVEL`
   - 至少区分 `soft / elevated / diagnostic`

2. 调整 `recommendedAction` 生成逻辑
   - 不再统一只给 `run-first`
   - 改为根据节点层级和项目形态生成不同提示

3. 重构 docs fallback
   - 不再依赖 `hasHealthyCanonicalDocsFallback()`
   - 改为独立文件存在性与最小可解析判断

4. 新增输出字段
   - `backgroundMode?: 'full' | 'weak' | 'none'`
   - `confidenceImpact?: 'none' | 'low' | 'medium' | 'high'`
   - `degradationLevel?: 'soft' | 'elevated' | 'diagnostic'`

### 6.2 dispatcher.ts

主要改造点：

1. 节点 notice 提示分层
   - 第一层节点：提示可继续
   - 第二层节点：提示质量下降
   - 第三层节点：提示结果降级

2. 注入统一元信息
   - `background_mode`
   - `project_cognition_status`
   - `confidence_impact`
   - `degradation_level`

3. recommendation 文案按层级变化

### 6.3 SKILL.md 与节点文档

主要改造点：

1. 把“需要 first 资产”收紧成：
   - `优先使用`
   - `存在时读取`
   - `缺失时按弱背景模式继续`

2. 对第二层、第三层节点补充一条统一规则：

> 当 `first` 项目认知资产缺失时，流程可以继续，但必须在输出中显式说明背景不足带来的可信度下降，不能把弱背景结果表述成高置信结论。

### 6.4 测试

必须新增三类测试：

1. **无 first 场景**
   - `.spec-first/runtime/first` 不存在
   - `docs/first` 不存在
   - 节点仍能继续

2. **仅 docs fallback 场景**
   - 无 runtime index
   - 有 `docs/first/*.md`
   - 节点可走 docs fallback

3. **分层告警场景**
   - `spec/design/task` -> soft warning
   - `code/review/verify/research` -> elevated warning
   - `doctor/analyze/status/archive/...` -> diagnostic degradation

## 七、推荐实施顺序

### Phase A：规则收口

1. 明确三层节点分组
2. 明确统一提示语与元信息字段
3. 明确 docs fallback 最小可用标准

### Phase B：resolver 改造

1. 给 `resolveSkillContext()` 增加分层降级规则
2. 将 docs fallback 从 index 依赖中解耦
3. 补 `backgroundMode / confidenceImpact / degradationLevel`

### Phase C：dispatcher 改造

1. 按层级注入不同 runtime notice
2. 调整 recommendation 文案
3. 对治理、诊断类节点补“弱背景模式”提示

### Phase D：节点文档与测试收口

1. 修改节点 `SKILL.md`
2. 补 fallback 场景测试
3. 校验“first 是辅助输入，不是前置条件”的主口径一致

## 八、最终验收标准

满足以下条件时，视为本优化完成：

1. 在完全未执行 `first` 的新项目中：
   - `init / spec / design / task` 可以正常继续
   - 且提示文案明确说明“新项目可跳过”

2. 在未执行 `first` 的存量项目中：
   - `code / review / verify / research` 可以继续
   - 但输出中明确说明“背景不足，结果可信度下降”

3. 在无 runtime、仅有 `docs/first` 的场景中：
   - 节点可以走 docs fallback
   - 不要求 `index.json` 先存在

4. 在治理、诊断、分析节点中：
   - 缺失 `first` 时输出必须带弱背景标识
   - 不能把弱背景判断包装成高置信结论

5. 节点文档、代码、测试三者口径一致：
   - `first` 是辅助输入
   - 不是必要前置条件
   - 缺失时按层级降级，而不是统一一句“建议补跑”

## 九、结论

最佳优化方向不是把 `first` 变成强依赖，而是把“缺失 `first` 时如何继续执行”设计成正式能力。

终态应满足三条：

1. `first` 缺失时流程仍可继续
2. 不同节点按依赖强度分层降级
3. docs fallback 独立成立，不再依赖 runtime index

这样才能同时支持：

- 新项目、无代码场景
- 存量项目、老需求迭代场景
- runtime-first 主线
- `first` 仅作为辅助输入而非阻断条件的产品定位

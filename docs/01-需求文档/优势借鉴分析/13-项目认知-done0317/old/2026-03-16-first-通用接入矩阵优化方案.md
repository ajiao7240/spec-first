# 2026-03-16 first 通用接入矩阵优化方案

## 一、结论

当前的 [first 节点接入改造计划](/Users/kuang/xiaobu/spec-first/docs/01-需求文档/优势借鉴分析/13-项目认知/2026-03-16-first-节点接入改造计划.md) 对当前 `spec-first` 仓库是合理的，适合作为第一阶段落地方案。

但如果从长期演进和最佳实践角度看，它还不是最优形态。

原因是：

1. 它主要按当前节点写死接入关系
2. 它还没有把不同端类型、不同开发语言、不同项目拓扑下的差异抽象出来
3. 它还没有形成“通用阶段输入层 + 项目类型适配层”的两层结构

因此，最佳实践不应是只有一份“当前节点接入矩阵”，而应升级为：

1. **通用阶段输入矩阵**
2. **项目类型适配矩阵**
3. **节点映射矩阵**

三层组合。

需要特别说明的是，这三层模型当前主要是“最佳实践目标模型”，不是“代码已经完整实现的统一框架”。

当前代码已具备的基础包括：

1. `platformType`
2. 节点级资产 contract
3. 完整 runtime / projection registry

因此，本文件的正确定位是：

**在已有 `platformType + 节点级资产映射 + 完整 runtime contract` 基础上，定义下一步的更优抽象模型。**

## 二、为什么当前矩阵还不是最佳实践

### 2.1 按节点硬编码，不利于跨项目复用

当前矩阵的问题不是方向错，而是抽象层级偏低。

例如它直接写：

- `03-spec` 接什么
- `04-design` 接什么
- `07-code` 接什么

这种写法适合当前仓库，但不适合作为跨项目、跨语言、跨端的长期规范。

更通用的表达应该先回答：

- 需求阶段应接入什么项目认知资产
- 设计阶段应接入什么项目认知资产
- 实现阶段应接入什么项目认知资产
- 验证阶段应接入什么项目认知资产

然后再映射到具体节点。

### 2.2 没有区分通用真源与适配型真源

并不是所有 `first` 产物都具有同等通用性。

#### 通用真源

这类资产几乎对所有项目都成立：

- `summary.json`
- `role-views.json`
- `stage-views.json`
- `steering.json`
- `change-map.json`
- `entry-guide.json`
- `reboot-guide.json`
- `critical-flows.json`

#### 适配型真源

这类资产强依赖项目类型、语言、端形态：

- `api-contracts.json`
- `structure-overview.json`
- `domain-model.json`
- `database-schema.json`
- `external-deps.md`
- `local-setup.md`
- `development-guidelines.md`

如果不分层，容易把面向后台服务的输入模式硬套到前端 H5、App、CLI 或 Monorepo 项目上。

### 2.3 没有把项目类型适配规则纳入模型

不同项目的认知重点不同：

- Backend 项目更重 API、数据库、外部依赖
- Frontend 项目更重结构、页面流、状态与交互边界
- App 项目更重页面流、模块边界、Native Bridge、集成关系
- Monorepo 项目更重 package topology、workspace 边界、共享规范

所以最佳实践不应是“一张固定矩阵通吃”，而应是：

**固定核心输入层 + 可变项目类型覆盖层**

### 2.4 当前代码已有部分基础，但抽象层级还不够

不能把现状理解成“完全没有代码支撑”。

当前代码里已经有：

1. `platformType` 贯穿 `summary/context/cli`
2. `context-resolver.ts` 中按 skill 的 required/optional 资产映射
3. `first-artifact-mapping.ts` 中完整 runtime / projection contract

真正的缺口在于：

- 分类粒度仍停留在 `platformType`
- 还没有 `projectType / primaryLanguage / 适配权重` 这类更通用字段
- 节点输入规则仍是散落实现，不是统一 declarative matrix

因此正确的优化路径不是“先推翻现有实现”，而是：

**先基于现有代码推进节点接入，再逐步把散落规则抽象成三层模型。**

## 三、最佳实践模型

### 3.1 三层模型

最佳实践建议使用以下三层：

#### 第一层：通用阶段输入矩阵

定义每个阶段“原则上应该接入哪些认知资产”。

#### 第二层：项目类型适配矩阵

定义不同类型项目在不同阶段下，哪些资产是必选、加权、可选、禁用。

#### 第三层：节点映射矩阵

把当前仓库中的 `03-spec / 04-design / 07-code / 08-review / 12-verify` 等节点映射到前两层。

这样做的好处是：

1. 当前仓库可以落地
2. 其他项目可以复用
3. 新增语言/端类型时无需推翻旧矩阵

### 3.2 与当前代码的对应关系

| 层级 | 当前代码状态 | 说明 |
| --- | --- | --- |
| 通用阶段输入矩阵 | 部分已存在 | 已有 skill 级资产 contract，但尚未集中建模 |
| 项目类型适配矩阵 | 基础不足 | 目前只有 `platformType`，还没有完整项目类型字段 |
| 节点映射矩阵 | 部分已存在 | 节点到资产的关系已散落在 `context-resolver.ts` 和各 skill 文档中 |

因此，这个三层模型不应被理解为“节点接入前必须先整体代码化”的前置条件，而应被理解为：

1. 先按现有代码基础推进接入
2. 再把接入经验收敛为统一三层配置

## 六、语言适配原则

### 6.1 不按语言直接决定文档集合

语言不应该直接决定“生成哪些正式文档”，而应主要影响：

1. `structure-overview.json` 的抽取策略
2. `api-contracts.json` 的抽取策略
3. `domain-model.json` 的归纳策略
4. `database-schema.json` 的识别方式

因此最佳实践是：

- 正式文档全集保持稳定
- 语言主要影响真源提取方式，而不是文档 contract

### 6.3 当前仓库尚未具备完整语言适配字段

当前代码中并没有：

- `projectType`
- `primaryLanguage`

这意味着文档中的“语言适配原则”当前仍属于设计层，而不是现成可用的代码字段。

短期实施上，应采用：

- 文档层先定义项目类型与语言适配逻辑
- 代码层先继续使用 `platformType + 项目结构特征` 做近似判断

而不应把“必须先新增 `projectType/primaryLanguage` 字段”当成节点接入的阻断前置条件。

### 6.2 语言影响的是真源适配器，不是节点接入框架

例如：

- Java/Spring
  - 更容易从 controller/service/entity 抽 API 与领域模型
- Node/TypeScript
  - 更容易从 route/handler/schema/type 抽接口边界
- Go
  - 更依赖 package/handler/service 的结构约定
- 前端 TS/React/Vue
  - 更依赖 pages/components/store/api 的结构识别

这些都属于“真源生成层”的问题，不应反向改变下游接入矩阵的总体结构。

## 七、节点矩阵如何升级

### 7.1 当前矩阵的正确定位

当前 [first 节点接入改造计划](/Users/kuang/xiaobu/spec-first/docs/01-需求文档/优势借鉴分析/13-项目认知/2026-03-16-first-节点接入改造计划.md) 应被视为：

- **当前仓库的第一阶段节点落地计划**

而不是：

- 长期通用规范的最终形态

### 7.2 推荐升级方式

下一版节点接入计划应按以下逻辑组织：

1. 先引用通用阶段输入矩阵
2. 再声明当前项目类型
3. 再映射到当前 skill 节点

这样每个节点的接入动作都能解释为：

- 它属于哪个阶段
- 当前项目类型要求它额外接什么
- 它为何接入这些资产而不是其他资产

### 7.3 推荐实施顺序

为了避免理论模型阻塞实施，建议顺序明确为：

#### 第一步：基于现有代码基础推进节点接入

直接利用当前已存在的：

- `platformType`
- `context-resolver` 节点资产 contract
- `first` runtime / projection registry

先补主开发链节点。

#### 第二步：补标准化项目类型字段

在节点接入已经验证有效后，再按需要补：

- `projectType`
- `primaryLanguage`
- 更规范的项目分类信息

#### 第三步：统一抽象为 declarative matrix

最后再把：

- 阶段输入层
- 项目类型适配层
- 节点映射层

沉淀成统一配置框架。

## 八、最终建议

最佳实践不是推翻当前矩阵，而是把它升级成：

1. **通用阶段输入矩阵**
2. **项目类型适配矩阵**
3. **当前仓库节点映射矩阵**

对应落地建议：

### 第一阶段

保留当前节点改造计划，继续服务本仓库落地。

### 第二阶段

基于本文件补一版升级版节点接入计划，把每个节点接入动作都映射到：

- 阶段输入层
- 项目类型适配层

### 第三阶段

在必要时补齐标准化项目类型字段，并将这套模型抽象回 `00-first` 与后续各节点，使其成为 `spec-first` 的长期标准。

## 九、总结

结论可以压缩成一句话：

**当前接入矩阵对当前仓库是合理的，但还不是最佳实践；最佳实践应升级为“通用阶段输入矩阵 + 项目类型适配矩阵 + 节点映射矩阵”的三层模型。**

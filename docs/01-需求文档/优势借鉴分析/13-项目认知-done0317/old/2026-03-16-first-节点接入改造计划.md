## 二、目标

将 `first` 从“生成完整项目认知资产”进一步升级为“被后续节点系统性消费的正式输入层”。

本轮目标不是继续扩展 `first` 自身产物，而是：

1. 按阶段把项目认知资产接入后续节点
2. 让不同端、不同语言项目都能复用接入框架
3. 让当前仓库的节点接入逻辑能解释为“阶段需求 + 项目类型适配”

当前已可复用的代码基础包括：

1. `platformType`
2. `context-resolver.ts` 中已有的节点级 required/optional 资产 contract
3. `first-artifact-mapping.ts` 中完整 runtime / projection registry

因此，本轮节点接入不以“先新增 `projectType/primaryLanguage` 字段”为前置条件。

## 三、三层接入模型

### 3.1 第一层：通用阶段输入层

#### 需求阶段

- `stage-views.json`
- `change-map.json`

#### 审查阶段

默认输入：

#### 验证阶段

默认输入：

- `critical-flows.json`
- `common-playbooks.md`
- `known-risks-and-traps.md`

### 3.2 第二层：项目类型适配层

### 当前仓库的判定

当前 `spec-first` 仓库更接近：

- `CLI / Tooling`
- `Node / TypeScript`
- 带有 `Monorepo / 多模块工具链` 特征

但需要强调：

- 当前代码层真正稳定存在的字段只有 `platformType`
- `Node / TypeScript`、`Monorepo / 多模块工具链` 在本计划中属于**基于项目结构和依赖特征的实施判断**
- 它们当前还不是 `first` runtime 中的正式标准字段

因此本仓库在通用阶段输入层之上，应叠加以下适配权重：

#### CLI / Tooling 加权

- `structure-overview.json`：必选
- `conventions.json`：推荐
- `local-setup.md`：推荐
- `api-contracts.json`：可选

#### Monorepo / 多模块工具链加权

- `structure-overview.json`：必选
- `steering.json`：必选
- `conventions.json`：必选
- `external-deps.md`：推荐

### 3.3 第三层：节点映射层

当前仓库中的每个节点，应被解释为：

- 属于哪个阶段
- 继承该阶段的默认输入
- 再叠加本项目类型的适配输入

这样接入动作不再是“拍脑袋补几个文件”，而是有统一解释框架。

## 三点五、实施边界

### 3.5.1 本轮先做“节点接入”，不先做“三层框架代码化”

本轮实施的重点是：

- 修改各节点 `SKILL.md`
- 修改各节点 `references/*.md`
- 补充模板、上下文装载和测试

而不是先做：

- `STAGE_INPUT_MATRIX`
- `PROJECT_TYPE_WEIGHTS`
- `projectType`
- `primaryLanguage`

这些统一配置或字段。

### 3.5.2 三层模型代码化属于后续增强

当主开发链节点已经验证接入有效后，再考虑：

1. 把散落在 `context-resolver.ts` 中的规则集中配置化
2. 补标准化项目类型字段
3. 把项目类型适配逻辑从文档升级为代码 contract

## 七、实施方式

### 7.1 先改节点文档 contract

每个节点优先修改：

- `SKILL.md`
- `references/*.md`

先把 first 输入层写进节点正式说明，再做后续上下文装载与测试。

### 7.1.1 复用当前已有节点级 contract

在节点接入时，应优先复用当前已有的：

- `context-resolver.ts` 中的 required/optional 资产逻辑

避免在本轮一开始就平行发明第二套未落地的装载机制。

### 7.2 再改上下文装载与模板

如果节点存在：

- 模板
- 输出格式文档
- 场景映射
- 恢复逻辑

应同步把 first 资产接入进去。

### 7.3 最后补测试

每个关键节点至少补三类断言：

1. 是否声明接入 first 资产
2. 是否引用正确的 runtime/docs 路径
3. 是否有缺失资产时的降级策略

### 7.4 后续再抽统一配置

当 Batch 1-3 完成后，再评估是否需要把：

- 阶段输入层
- 项目类型适配层
- 节点映射层

上移为统一 declarative matrix。

## 八、验收标准

完成本轮节点接入改造后，应满足：

1. 主开发链节点都能解释为“阶段输入 + 项目类型加权”的组合结果
2. `03-spec` 到 `12-verify` 均至少显式接入 2 个以上 first 资产
3. `04-design`、`07-code` 必须显式接入符合 CLI / Tooling + 多模块工具链项目特征的资产
4. `external-deps.md`、`local-setup.md`、`development-guidelines.md` 至少各有 1 个下游节点显式消费
5. `modules.json` 不再作为任何节点引用对象
6. 主开发链节点接入已落地时，不要求 `projectType/primaryLanguage` 同步完成

## 九、后续演进

如果后续要把这套模型推广到其他项目，建议：

1. 保持“通用阶段输入层”稳定
2. 为 Backend / Frontend / App / Monorepo / CLI 分别补标准适配模板
3. 让各节点只声明“属于哪个阶段、当前项目类型是什么、因此接入哪些资产”

这样才能真正做到：

- 跨语言可复用
- 跨端可复用
- 不随单个仓库结构变化而频繁重写节点矩阵

## 十二、总结

**从“当前仓库节点写死接入关系”升级为“阶段输入层 + 项目类型适配层 + 当前节点映射层”的第二版接入计划。**

这使当前接入动作既能服务本仓库实施，也能为未来跨端、跨语言推广保留统一框架。

# 01-init Skill 场景化优化完整方案

> 日期：2026-03-16
> 范围：`skills/spec-first/01-init`、`src/cli/commands/init.ts`、`src/core/process-engine/init.ts` 及相关场景文档
> 目标：将当前 `init` 从“仅能稳定创建 Feature 工作区”的半成品入口，升级为面向新项目、存量项目首次引入、已接入项目日常需求三类场景的标准化初始化入口

---

## 1. 执行摘要

当前 `01-init` 的核心问题不是“Feature 初始化内核不稳”，而是“入口语义失真”：

- 文档与场景手册把 `spec-first init` 描述为“项目首次初始化入口”
- 代码实现已经把它收紧为“`first` 完成后的严格 Feature 初始化入口”
- Skill 文档、交互规范、CLI 实现三者之间存在明显漂移

因此，本次优化不建议推翻现有 `process-engine/init` 内核，而应采用：

**外层场景编排 + 内层严格 Feature Init Core**

即：

1. 保留现有 `Feature Init Core` 的原子性、幂等性、锁机制和骨架生成能力
2. 将 `/spec-first:init` 升级为“初始化路由器”
3. 让路由器在不同场景下自动进入不同初始化轨道
4. 对存量项目首次引入补齐“legacy baseline”最佳实践
5. 对新项目首次接入补齐 project onboarding 与 `first` 的前置编排

---

## 2. 当前现状与根因

## 2.1 当前真实流程

当前代码层真实流程为：

```text
first -> init(feature scaffold) -> spec -> design -> task -> code -> verify -> wrap_up -> release
```

其中：

- `first` 提供 runtime 真源
- `init` 负责创建 `00_init` 阶段的 Feature 工作区
- `init` 本身并不是项目接入总入口

## 2.2 当前主要断层

### D1. 入口语义断层

`spec-first init` 在文档中被描述为：

- 新项目首次引入入口
- 老项目首次接入入口
- 新 Feature 初始化入口

但代码只稳定支持第 3 类。

### D2. Skill 与 CLI 行为断层

Skill 约定中存在：

- `degraded` 背景下继续初始化
- 中文 FEAT 候选
- 键盘导航式交互
- bootstrap 默认关闭

但 CLI 真实实现中：

- `first` 缺失直接阻断，不可降级
- 无中文 FEAT 候选
- 无箭头导航，仅 `readline` 问答
- bootstrap 默认开启

### D3. 场景最佳实践未产品化

文档中已提出存量项目首次引入时，应创建：

`FSREQ-000000-legacy-baseline`

但当前 `init` 不会识别：

- 是否为首次引入
- 是否为存量工程
- 是否应推荐建立 baseline
- 是否为 `Mode I` 场景补齐 `impact-analysis.md`

### D4. 命令模型历史包袱未清理

仓库里仍并存多种口径：

- `spec-first init`
- `spec-first init feature ...`
- `spec-first feature init`

这会让用户和 Agent 都误判入口能力。

## 3. 优化目标

本次优化目标分为四层。

### G1. 语义一致

让以下四类对象完全一致：

- Skill 文档
- CLI 帮助
- 场景手册
- 代码真实行为

### G2. 场景可用

覆盖三类核心场景：

1. 全新项目全新工程
2. 存量项目首次引入 Spec-First
3. 已接入项目的日常新需求 / 迭代需求

### G3. 核心不退化

不得破坏当前 `Feature Init Core` 已有能力：

- FEAT 注册锁
- 原子写入
- 幂等恢复
- 骨架文档生成
- `background_input_status` 写入
- `.spec-first/current` 更新

### G4. 最小心智负担

用户只记一个高层入口：

`/spec-first:init`

内部由系统判断当前处于哪个初始化阶段。

## 4. 推荐总体方案

## 4.1 总体原则

采用双层架构：

```text
Layer A: Init Router / Onboarding Orchestrator
  ├── 场景识别
  ├── 前置检查
  ├── 路由到 project onboarding / baseline init / feature init
  └── 统一交互与摘要输出

Layer B: Feature Init Core
  ├── FEAT 校验
  ├── Feature ID 生成
  ├── 原子目录创建
  ├── 文档骨架生成
  └── current / registry 更新
```

## 4.2 为什么不建议弱化 `first` 硬前置

不建议简单把“缺少 `first` 仍继续初始化”改回来，原因是：

1. 下游 Skill 已显著依赖 `first` runtime 真源
2. 缺少 runtime 真源会让后续上下文注入质量失真
3. `degraded` 可作为临时诊断状态，但不应成为默认 Feature 启动路径

因此正确做法不是弱化硬门禁，而是：

**把硬门禁上移到初始化路由器里，以“自动补前置”替代“用户手动碰壁”。**

## 5. 新的初始化模型

## 5.1 三条初始化轨道

### Track A: Greenfield Project Onboarding

适用：

- 新仓库
- 尚未建立 `.spec-first/`
- 尚未有 `first` runtime

目标：

- 完成项目级接入
- 生成基础配置
- 引导完成 `first`
- 可选创建首个 Feature

### Track B: Brownfield First Adoption

适用：

- 已有代码工程
- 首次引入 Spec-First
- 历史规范文档为空或不完整

目标：

- 完成项目接入
- 生成 `first` runtime
- 引导创建 `legacy-baseline`
- 再进入真实需求

### Track C: Existing Project Feature Init

适用：

- 项目已接入 Spec-First
- 已存在 `first` runtime
- 用户要创建新 Feature

目标：

- 直接进入严格 Feature 初始化

## 5.2 路由判定规则

建议在 `handleInit` 入口新增项目状态检测：

```text
if !git repo:
  进入 greenfield-precheck，提示先 git init

if !.spec-first exists OR !project meta exists:
  route = greenfield_onboarding

else if !first runtime healthy:
  route = onboarding_then_first

else if no legacy baseline and repo is brownfield:
  route = brownfield_first_adoption

else:
  route = feature_init
```

其中 brownfield 判定可采用启发式：

- 已存在源码目录、构建文件、依赖清单
- 但 `specs/` 为空或不存在
- 无 `FSREQ-000000-LEGACY-BASELINE`

---

## 6. 新的 `/spec-first:init` 目标行为

## 6.1 对用户暴露的统一语义

`/spec-first:init` 定义为：

> Spec-First 初始化总入口。根据当前项目状态，自动执行项目接入、基线建档或 Feature 初始化。

CLI 帮助不再只写“初始化 Feature 工作区”，而应改成：

> 初始化 Spec-First 工作流：包含项目接入、存量基线建档、Feature 工作区创建三种路径。

## 6.2 建议新增子模式

为了兼顾显式控制和兼容老用法，建议：

```bash
spec-first init
spec-first init --track project
spec-first init --track baseline
spec-first init --track feature
spec-first init --feat AUTH --mode N --size M --platforms h5,backend
```

其中：

- 默认无 `--track` 时自动路由
- 显式 `--track` 时走指定分支
- `--feat/...` 仅在 `feature` 轨道必填

## 7. 各场景详细优化方案

## 7.1 场景 A：全新项目全新工程

### 当前问题

当前场景手册要求：

```bash
spec-first init
spec-first doctor
```

但现有代码实际上会因为没有 `first` runtime 直接失败。

### 优化后流程

```text
Step 0: 检查 git / node / package manager / host baseline
Step 1: 生成 .spec-first 基础目录与 meta/config
Step 2: 引导创建 layer2 平台 YAML
Step 3: 运行 first（quick 或 deep）
Step 4: 输出项目接入摘要
Step 5: 询问是否立刻创建首个 Feature
```

### 交互建议

首屏不是问 `feat`，而是问：

```text
检测到当前项目尚未完成 Spec-First 接入。

请选择初始化目标：
1. 新项目首次接入
2. 存量项目首次引入
3. 已接入项目，创建新 Feature
```

对新项目，后续问题应关注：

- 项目类型
- 平台集合
- 是否立即跑 `first`
- `first` 选择 quick 还是 deep
- 是否创建首个 Feature

### 产出物

- `.spec-first/meta/config.yaml`
- `.spec-first/layer2/*.yaml`
- `.spec-first/constitution.md`
- `.spec-first/runtime/first/*`
- 可选首个 `specs/<featureId>/`

## 7.2 场景 B：存量项目首次引入

### 当前问题

存量项目最佳实践只存在于文档中，没有被 `init` 产品化。

### 优化后流程

```text
Step 0: 检测到 brownfield
Step 1: 引导完成 project onboarding
Step 2: 运行 first
Step 3: 明确提示“建议先建立 legacy baseline”
Step 4: 自动生成 baseline Feature 默认参数
Step 5: 进入 baseline Feature 初始化
Step 6: 输出下一步：补齐 spec/design/matrix
```

### baseline 默认策略

建议预置：

- `featureId = FSREQ-000000-LEGACY-BASELINE`
- `mode = I`
- `size = M`
- title = `存量系统可分析基线`

允许用户修改，但推荐默认值。

### baseline 特有骨架

对 baseline Feature，额外生成：

- `impact-scope.md`
- `baseline-checklist.md`
- `spec.md` 使用“当前能力摘要模板”
- `design.md` 使用“现状架构模板”

## 7.3 场景 C：已接入项目的新 Feature / 迭代 Feature

### 当前问题

这部分是当前实现最接近可用的路径，但交互和文档仍有漂移。

### 优化后流程

若项目已经满足：

- `.spec-first` 完整
- `first` runtime healthy

则直接进入 `Feature Init Core`。

### 必须补齐的差异

1. `Mode I` 时增加迭代上下文提示
2. 若仓库中不存在 `legacy baseline`，且为 brownfield，给出警告而不是静默继续
3. `Mode I` 默认附加 `impact-analysis.md`
4. 输出“关联 baseline Feature”建议

## 9. CLI 层优化方案

## 9.1 `handleInit` 重构建议

建议拆成：

```text
handleInit()
  ├── detectInitTrack()
  ├── runProjectOnboardingTrack()
  ├── runBrownfieldBaselineTrack()
  └── runFeatureInitTrack()
```

## 9.2 新增接口建议

```ts
interface InitProjectState {
  gitReady: boolean;
  specFirstReady: boolean;
  firstRuntimeHealthy: boolean;
  hasLegacyBaseline: boolean;
  likelyBrownfield: boolean;
  discoveredPlatforms: string[];
}

type InitTrack =
  | 'project-onboarding'
  | 'brownfield-baseline'
  | 'feature-init';
```

## 9.3 路由函数建议

```ts
function detectInitTrack(state: InitProjectState, args: string[]): InitTrack
```

职责：

- 用户显式指定 `--track` 时优先采用
- 否则按项目状态自动判定

## 9.4 保留不动的内核

以下内核函数不建议大改：

- `init(opts)`
- `commitFeatureInit`
- `registerFeat`
- `writeFeatureSkeleton`

这些已经足够稳定，应尽量复用。

## 9.5 建议新增包装层

### `runProjectOnboardingTrack()`

负责：

- 创建 `.spec-first/meta/config.yaml`
- 引导创建平台 YAML
- 调用 `first`
- 可选调用 `runFeatureInitTrack()`

### `runBrownfieldBaselineTrack()`

负责：

- 检查并创建 baseline 默认参数
- 调用 `runFeatureInitTrack({ baselinePreset })`
- 输出 baseline 补齐任务

### `runFeatureInitTrack()`

负责：

- 调用现有 `normalizeInitInput`
- 调用现有 `init(opts)`
- 执行 post-init setup

## 10. 交互设计优化方案

## 10.1 交互原则

1. 先识别场景，再收参数
2. 先解释为什么，再让用户选
3. 无 TTY 时必须支持纯参数模式
4. TTY 模式只承诺当前真实可实现的文本交互

## 10.2 推荐交互顺序

### 全局入口

```text
Step 1/3: 识别当前项目状态
Step 2/3: 选择初始化轨道
Step 3/3: 进入该轨道专属参数收集
```

### project-onboarding

```text
project type -> platforms -> first mode -> bootstrap -> create first feature?
```

### brownfield-baseline

```text
platforms -> baseline preset confirm -> first mode -> bootstrap
```

### feature-init

```text
feat -> mode -> size -> platforms -> title -> feature-id -> bootstrap
```

## 10.3 关于 TTY 能力的现实约束

当前实现使用 `node:readline/promises`，按 Node 官方能力更适合：

- 普通提问
- 基于文本的多选切换
- TTY 检测与非交互模式回退

不适合在文档里承诺：

- 复杂箭头导航
- 富交互菜单
- 稳定的即时刷新式 TUI

如果要兑现这些能力，应后续引入专门 TUI 方案，而不是继续用文档虚构。

## 11.1 统一命令口径

全仓统一只保留：

- `spec-first init`

若要表达 Feature 初始化，也应写成：

- `spec-first init --track feature --feat AUTH --mode N ...`

避免继续混用“项目初始化”“需求初始化”“Feature 初始化”而不区分层级。

## 12. 测试与验收方案

## 12.1 单元测试

新增覆盖：

- `detectInitTrack` 三轨识别
- brownfield 判定逻辑
- baseline 默认参数生成
- `Mode I` 时 `impact-analysis.md` 骨架生成

## 12.2 集成测试

新增场景：

1. 新仓库首次执行 `spec-first init`
2. brownfield 首次执行 `spec-first init`
3. 已接入项目执行 `spec-first init --feat ...`
4. 无 TTY 且参数不足时提示帮助
5. 显式 `--track` 与自动路由一致性

反向/边界场景（必须覆盖）：

6. healthy project 不被误路由到 brownfield 轨道（防止已接入项目被拉入 baseline 流程）
7. 已存在 `FSREQ-000000-LEGACY-BASELINE` 时，`brownfield-baseline` 轨道不重复创建
8. `Mode N` Feature 初始化完成后，目录中不含 `impact-analysis.md`
9. `first` 缺失时 `--track feature` 给出明确错误，不静默退出

## 12.3 E2E 测试

至少补三条完整链路：

1. Greenfield:
   `init(project) -> first -> init(feature)`
2. Brownfield:
   `init(project) -> first -> init(legacy baseline) -> init(real feature)`
3. Existing:
   `init(feature) -> stage current -> advance`

## 12.4 文档一致性测试

建议新增或扩展文档测试，确保：

- Skill 声明的默认值与 CLI 一致
- 文档中的命令格式不再出现旧写法（`spec-first init feature`、`spec-first feature init`）
- bootstrap 默认值、交互顺序、前置条件都与实现一致
- 文档不再宣称未实现的交互能力（箭头导航、中文 FEAT 候选）
- 非 TTY 下纯参数模式的行为与 Skill 文档描述一致

## 13. 分阶段实施路线

## Phase 1：语义对齐

目标：

- 先消除“文档说能做，代码做不到”的问题

动作：

- 重写 `01-init` Skill 文档
- 修正文档中的命令写法
- 明确 `init` 当前真实职责

风险：

- 低

收益：

- 立即降低误用率

## Phase 2：增加初始化路由器

目标：

- 让 `/spec-first:init` 真正成为总入口

动作：

- 新增 `detectInitTrack`
- 增加 `project-onboarding` / `brownfield-baseline` / `feature-init` 三轨
- 保持 `Feature Init Core` 不动

风险：

- 中

收益：

- 三大场景可正式落地

## Phase 3：补 baseline 与 Mode I 增强

目标：

- 把存量项目最佳实践产品化

动作：

- baseline 预设
- `impact-analysis.md` 自动骨架
- baseline 缺失时的警告与引导

风险：

- 中

收益：

- 真正覆盖企业真实高频场景

## Phase 4：交互增强

目标：

- 提升体验，不再让文档先跑到实现前面

动作：

- 决定是否引入正式 TUI
- 若不引入，则收敛文档承诺到真实能力

风险：

- 中

收益：

- 降低 Skill/CLI 漂移复发

## 14. 明确不建议的方案

## 14.1 不建议仅修文档不改入口

这只能“解释失败”，不能“消除失败”。

## 14.2 不建议去掉 `first` 硬前置

这会把当前下游 Skill 的上下文质量保证打穿。

## 14.3 不建议重写 `process-engine/init.ts`

当前 Feature 初始化内核已经是项目里较稳定的一层，问题不在这里。

## 15. 最终推荐结论

### 结论一句话

`01-init` 应从“Feature 脚手架技能”升级为“初始化总入口技能”，但其底层仍复用现有严格可靠的 Feature Init Core。

### 推荐落地方向

优先级按顺序为：

1. 统一语义与文档
2. 引入三轨初始化路由器
3. 产品化 brownfield `legacy-baseline`
4. 补齐 `Mode I` 增量场景产物
5. 视需要再升级 TUI 体验

### 改造后的理想用户心智

用户只需要知道：

```text
第一次进入项目，运行 /spec-first:init
以后创建需求，还是运行 /spec-first:init
系统会根据项目状态自动把我带到正确轨道
```

这才是符合 SDD 最佳实践的初始化入口设计。

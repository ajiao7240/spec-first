# 01-init Skill 重构设计文档

> 日期：2026-03-16
> 目标：将 `01-init` 从单一 Feature 初始化入口，重构为覆盖项目接入、存量基线建档、Feature 初始化的统一初始化路由器
> 适用范围：`skills/spec-first/01-init`、`src/cli/commands/init.ts`、`src/core/process-engine/init.ts`、相关文档与测试

---

## 1. 设计目标

本次重构的核心目标不是重写初始化能力，而是把初始化拆成两层：

1. 外层负责“识别当前是什么初始化场景”
2. 内层负责“稳定创建 Feature 工作区”

重构后应满足：

- 新项目首次接入时，`/spec-first:init` 不再因缺少 `first` 直接失败
- 存量项目首次引入时，`/spec-first:init` 能主动引导建立 `legacy-baseline`
- 已接入项目创建新 Feature 时，继续复用当前稳定的原子初始化逻辑
- Skill 文档、CLI 帮助、场景文档与实际行为一致

## 3. 总体架构

## 3.1 分层模型

```text
+-----------------------------------------------------------+
| Init Router                                               |
|  - detect project state                                   |
|  - choose track                                           |
|  - collect track-specific input                           |
|  - invoke onboarding / first / baseline / feature init    |
+-----------------------------------------------------------+
                |                     |
                |                     |
                v                     v
+-------------------------------+   +----------------------+
| Project/Baseline Flow         |   | Feature Init Core    |
| - project bootstrap           |   | - FEAT validate      |
| - first runtime generation    |   | - Feature ID         |
| - legacy baseline guidance    |   | - atomic scaffold    |
| - handoff summary             |   | - registry/current   |
+-------------------------------+   +----------------------+
```

## 3.2 设计原则

### P1. 路由层可变，内核层稳定

问题集中在入口编排，因此不应破坏当前 Feature Init Core。

### P2. 统一入口，场景分支

用户入口仍保持为 `/spec-first:init`，内部自动进入不同轨道。

### P3. 硬前置不取消，只做前移处理

`first` runtime 真源仍是 Feature 初始化的强前提，但不再由用户手动撞墙后自己恢复，而由初始化路由器处理。

### P4. brownfield 优先产品化

企业真实高频场景不是 greenfield，而是已有系统首次纳入 Spec-First，因此 `legacy-baseline` 必须从文档最佳实践升级为产品能力。

---

## 4. 场景模型

## 4.1 三类核心轨道

### Track A：project-onboarding

适用：

- 新项目
- `.spec-first/` 不完整
- 尚未建立 `first` runtime

职责：

- 建立项目级目录和配置
- 收集平台信息
- 执行 `first`
- 可选继续创建首个 Feature

### Track B：brownfield-baseline

适用：

- 已有工程首次引入
- 可识别为 brownfield
- 尚不存在 `legacy-baseline`

职责：

- 生成 `first` runtime
- 创建 `FSREQ-000000-LEGACY-BASELINE`
- 给出 baseline 产出与下一步要求

### Track C：feature-init

适用：

- 项目已接入
- `first` runtime 健康
- 用户正在创建新 Feature

职责：

- 进入现有严格 Feature 初始化流程

## 4.2 状态判断输入

设计一个统一的项目状态对象：

```ts
interface InitProjectState {
  gitReady: boolean;
  specFirstDirExists: boolean;
  metaConfigExists: boolean;
  firstRuntimeHealthy: boolean;
  hasAnyFeature: boolean;
  hasLegacyBaseline: boolean;
  projectMaturity: 'greenfield' | 'brownfield';
  discoveredPlatforms: string[];
}
```

其中 `projectMaturity` 可优先复用已有：

- `src/core/skill-runtime/first-platform-detector.ts`

避免重复造 brownfield 判定逻辑。

---

## 5. 路由设计

## 5.1 自动路由规则

推荐规则：

```text
if !gitReady:
  fail with guidance: run git init first

else if !specFirstDirExists || !metaConfigExists:
  route -> project-onboarding

else if !firstRuntimeHealthy:
  route -> project-onboarding-with-first

else if projectMaturity == brownfield && !hasLegacyBaseline && !hasAnyFeature:
  route -> brownfield-baseline

else:
  route -> feature-init
```

## 5.2 显式覆盖规则

新增参数：

```bash
spec-first init --track project
spec-first init --track baseline
spec-first init --track feature
```

优先级：

1. `--track` 显式指定
2. 自动路由

> ⚠️ 推荐首版同时支持 `--track` 与自动路由。若首版只实现自动路由，需在实施前书面说明并获得 Tech Lead 确认，否则不得开工（详见审查清单 §4.1）。

## 5.3 兼容策略

当前 CLI 用户已习惯直接传：

```bash
spec-first init --feat AUTH --mode N --size M --platforms h5,backend
```

因此兼容规则为：

- 若用户传入 `--feat` 等 Feature 参数，默认优先尝试 `feature-init`
- 若此时 `first` 不健康，不直接报错，而提示“先补齐 project-onboarding/first”，并提供自动继续选项

## 6. 交互设计

## 6.1 入口交互

当前交互的问题是上来就问 `feat`，没有识别当前项目处于哪个初始化阶段。

重构后建议先展示：

```text
检测到当前项目状态：
- Git: ready
- Spec-First: missing
- first runtime: missing
- project maturity: brownfield

建议轨道：brownfield-baseline
```

然后给用户确认：

```text
请选择：
1. 按建议继续
2. 选择其他轨道
3. 退出
```

## 6.2 轨道专属交互

### project-onboarding

建议收集：

- 项目类型
- 平台 YAML 是否自动生成
- `first` 模式：quick / deep
- 是否执行 bootstrap
- 是否创建首个 Feature

### brownfield-baseline

建议收集：

- 主要平台集合
- baseline 默认 Feature ID 是否接受
- baseline 标题是否修改
- 是否立即创建真实需求 Feature

### feature-init

保留现有字段：

- feat
- mode
- size
- platforms
- title
- feature-id
- bootstrap

## 6.3 关于交互能力的现实收敛

当前基于 `node:readline/promises`，建议文档仅承诺：

- 逐步问答
- 文本多选
- TTY 检测
- 非 TTY 下纯参数模式

不承诺：

- 真正的上下键菜单
- 高保真 TUI

## 7. CLI 结构重构

## 7.1 建议的 `src/cli/commands/init.ts` 新结构

```text
parseInitCliInput
detectInitProjectState
detectInitTrack
resolveInitCliInput
runProjectOnboardingTrack
runBrownfieldBaselineTrack
runFeatureInitTrack
handleInit
```

## 7.2 关键职责拆分

### `detectInitProjectState(projectRoot)`

负责：

- 检查 git
- 检查 `.spec-first/`
- 检查 `meta/config.yaml`
- 检查 `first` runtime
- 检查 `legacy-baseline`
- 扫描平台
- 复用 maturity detector

### `detectInitTrack(state, args)`

负责：

- 识别 `--track`
- 根据状态自动判定轨道

### `runProjectOnboardingTrack()`

负责：

- 补 `.spec-first/meta/config.yaml`
- 若无平台配置，引导创建 `.spec-first/layer2/*.yaml`
- 调用 `handleFirst` 或内聚的 first 执行逻辑
- 输出 onboarding 摘要

### `runBrownfieldBaselineTrack()`

负责：

- 生成 baseline 预设输入
- 调用 `runFeatureInitTrack`
- 输出 baseline handoff

### `runFeatureInitTrack()`

负责：

- 复用当前 `normalizeInitInput`
- 复用当前 `init(opts)`
- 复用 post-init setup

## 11. 错误处理设计

## 11.1 Git 未初始化

行为：

- 直接失败
- 明确提示 `git init`

## 11.2 平台配置缺失

行为：

- 在 onboarding 轨道中允许交互创建模板
- 在 feature-init 轨道中提示切换到 onboarding 或手动补齐

## 11.3 first 缺失

行为：

- 在自动路由下切到 onboarding
- 在显式 `--track feature` 下提示先执行 onboarding/first

## 11.4 baseline 缺失但 brownfield

行为：

- 不阻断 feature-init
- 但在输出中给出高优先级警告与建议命令

## 11.5 非 TTY 环境

行为：

- 检测 `process.stdout.isTTY === false` 时切换到纯参数模式
- 参数充足（含 `--feat`、`--track` 等必要字段）：直接执行，不发起 `readline` 交互
- 参数不足：输出使用帮助文本并以非零状态码退出，**不挂起等待**
- 禁止在非 TTY 下调用 `readline/promises`，防止 CI/CD 场景命令挂死

## 14. 测试设计

## 14.1 单元测试

新增：

- `detectInitProjectState`
- `detectInitTrack`
- baseline preset builder
- Mode I impact-analysis scaffold

## 14.2 集成测试

新增：

- 无 `.spec-first` 的 project-onboarding 路径
- brownfield + no baseline 路径
- healthy project 的 feature-init 路径

反向/边界场景（必须覆盖）：

- healthy project 不被误路由到 brownfield 轨道
- 已存在 `FSREQ-000000-LEGACY-BASELINE` 时不重复创建
- 非 TTY + 参数不足时输出帮助并退出，不挂起
- `Mode N` 初始化不生成 `impact-analysis.md`
- `first` 缺失时给出明确错误提示，不静默失败

## 14.3 文档一致性测试

新增：

- CLI 帮助与 Skill 默认值一致
- 不再出现 `spec-first init feature`

## 15. 兼容与迁移策略

## 15.1 用户兼容

保持：

```bash
spec-first init --feat AUTH --mode N --size M --platforms h5,backend
```

继续有效。

## 15.2 文档迁移

先做文档纠偏，再上代码路由改造，避免发布窗口内继续扩散错误心智。

## 15.3 代码迁移

优先新增函数，不大规模重排当前 init core。

---

## 16. 最终推荐

最终推荐设计为：

- `01-init` 作为总入口 Skill
- `spec-first init` 作为总入口 CLI
- 通过三轨路由将 onboarding、baseline、feature-init 串起来
- 保留当前 `process-engine/init.ts` 作为底层可靠内核

这是在不牺牲当前稳定性的前提下，把 `init` 补齐为真正可推广入口的最小正确路径。

2. **改完以后，怎样才算真正通过验收？**

因此它是实施前 gate，不是方案正文，也不是开发任务列表。

## 2. 改造目标基线

本次改造的最低正确目标是：

- `spec-first init` 成为统一初始化入口
- 支持三条轨道：
  - `project-onboarding`
  - `brownfield-baseline`
  - `feature-init`
- 保留现有 `Feature Init Core` 的稳定性
- 文档、Skill、CLI、测试四层行为一致

以下不属于最低目标，不应在本轮范围蔓延：

- 引入完整 TUI 框架
- 改造后续 `spec/design/task/code` 流程
- 重写 Gate 体系
- 扩大到全 Skill 系统重构

## 3. 实施前必须通过的审查清单

## 3.1 范围审查

开始改代码前，必须明确以下结论：

- 本轮只重构 `01-init` 入口与相关文档，不扩散到非初始化链路
- `src/core/process-engine/init.ts` 仅允许做最小增强，不允许大规模重写
- `first` 仍然是 Feature 初始化的强前提，不被取消
- brownfield `legacy-baseline` 必须纳入产品化范围，不再只是文档建议

若以上任一结论未达成一致，不得开工。

## 3.2 入口语义审查

必须先确认以下术语已经统一：

- `project onboarding` = 项目接入
- `brownfield baseline` = 存量基线建档
- `feature init` = Feature 工作区初始化
- `/spec-first:init` = 初始化总入口

如果文档和设计里仍存在以下混用，必须先修正：

- “项目初始化”与“Feature 初始化”混用
- `spec-first init feature ...`
- `spec-first feature init ...`

## 3.3 架构边界审查

以下边界必须在改造前锁定：

- 路由逻辑放在 `src/cli/commands/init.ts`
- Feature 原子初始化逻辑继续留在 `src/core/process-engine/init.ts`
- project maturity / brownfield 判定尽量复用已有检测能力
- `first` 的执行尽量复用现有命令或底层上下文能力，不复制逻辑

若边界不清晰，后续实现极易演变为“先改能跑，再补设计”，必须阻断。

## 3.4 交互现实性审查

开始改造前，必须确认文档承诺与当前技术能力一致：

- 当前只支持基于 `readline/promises` 的文本交互
- 当前不支持真正的箭头导航式 TUI
- 当前不支持中文 FEAT 自动候选，除非本轮明确实现

也就是说，实施前必须先决定：

### 选项 A：收敛文档承诺

把文档降到真实能力。

### 选项 B：补齐交互能力

真正实现文档已承诺能力。

若这一步不先做选择，代码与文档必然再次漂移。

本轮推荐：**选项 A**。

## 3.5 风险审查

必须识别并接受以下风险：

- 新增轨道会影响 `handleInit` 主流程，属于入口级高风险改造
- 文档口径清理可能影响现有用户心智
- brownfield 判定如果过于激进，可能把已接入项目误导向 baseline 轨道
- baseline 默认 Feature ID 如果与现网已有约定冲突，可能引入迁移成本

如未写清这些风险及对应缓解措施，不得进入编码。

## 4. 实施前必须明确的设计决策

下列问题在开始编码前必须有明确答案。

## 4.1 `--track` 是否纳入首版

必须二选一：

- 方案 1：首版就支持 `--track`
- 方案 2：首版只做自动路由，`--track` 后续补

本轮推荐：**方案 1**

原因：

- 更利于测试
- 更利于排错
- 更利于向后兼容与文档表达

> ⚠️ 若选择方案 2，须在开工前书面说明理由并获得 Tech Lead 确认，否则视为偏离推荐方案，不得进入编码。

## 4.2 brownfield 判定是否复用现有 maturity detector

必须二选一：

- 方案 1：复用现有 `first-platform-detector`
- 方案 2：在 `init` 中再写一套启发式逻辑

本轮推荐：**方案 1**

原因：

- 避免双份 brownfield 判定逻辑
- 减少长期维护分叉

> ⚠️ 若选择方案 2，须在开工前书面说明理由并获得 Tech Lead 确认，否则视为偏离推荐方案，不得进入编码。

## 4.3 baseline 是否自动创建

必须二选一：

- 方案 1：brownfield 且无 baseline 时自动创建
- 方案 2：brownfield 且无 baseline 时默认推荐，用户确认后创建

本轮推荐：**方案 2**

原因：

- baseline 是高价值动作，但仍属于高影响写入
- 需要给用户保留控制权

> ⚠️ 若选择方案 1，须在开工前书面说明理由并获得 Tech Lead 确认，否则视为偏离推荐方案，不得进入编码。

## 4.4 `Mode I` 是否补 `impact-analysis.md`

必须一锤定音：

- `Mode I` 初始化是否自动写入 `impact-analysis.md`

本轮推荐：**必须写入**

原因：

- 当前流程与 Gate 已把它视为关键产出
- 不在初始化时补骨架，会把问题推迟到后续阶段暴露

> ⚠️ 若决定不在初始化时写入，须在开工前书面说明替代补齐方案并获得 Tech Lead 确认，否则不得进入编码。

## 5. 实施前阻断条件

出现以下任一情况，应阻断编码：

- 设计文档仍允许“去掉 `first` 强前置”
- 任务拆解仍包含大规模重写 `process-engine/init.ts`
- 文档仍未统一命令口径
- 没有为三轨分别定义测试样例
- 没有明确 baseline 的默认策略
- 没有明确无 TTY 场景如何回退
- 没有决定交互能力是“收敛文档”还是“补功能”

这些阻断项不解决，后续改造极大概率演变为“行为临时拼接”。

---

## 6. 实施中必须保持的技术约束

## 6.1 核心稳定性约束

以下能力不得回退：

- FEAT 注册锁
- 原子目录创建
- 幂等恢复
- `.spec-first/current` 更新
- `background_input_status` 保持写入

## 6.2 最小改动约束

以下事项必须优先“新增包装层”而不是“重写核心”：

- 项目状态检测
- 轨道识别
- onboarding / baseline 路由

## 6.3 文档一致性约束

每改完一个阶段，必须同步确认：

- Skill 文档是否已匹配代码
- CLI 帮助是否已匹配代码
- 场景文档是否已匹配代码

不允许最后再统一补文档。

## 6.4 非 TTY 约束

无交互终端时，`init` 入口必须降级处理：

- 存在足够参数（`--feat`、`--track` 等）：按参数执行，不等待交互输入
- 参数不足时：输出帮助文本或明确的校验错误，不得挂起等待
- 禁止在非 TTY 环境下发起 `readline/promises` 交互，防止 CI/CD 场景下命令挂死

## 7. 实施完成后的验收标准

## 7.1 功能验收

必须全部满足：

### A. 新项目场景

在一个新的 Git 仓库中执行：

```bash
spec-first init
```

应能够：

- 进入项目接入轨道
- 补齐 `.spec-first/meta/config.yaml`
- 处理平台配置
- 串联 `first`
- 输出明确下一步

### B. 存量项目首次引入

在 brownfield 工程执行：

```bash
spec-first init
```

应能够：

- 识别为 brownfield
- 推荐或引导建立 `legacy-baseline`
- 生成 baseline Feature
- 输出 baseline 补齐要求

### C. 已接入项目新 Feature

在 healthy 项目执行：

```bash
spec-first init --feat AUTH --mode N --size M --platforms h5,backend
```

应能够：

- 直接进入 feature-init
- 生成正常 Feature 目录
- 保持现有核心行为不变

### D. Mode I 迭代 Feature

执行：

```bash
spec-first init --feat ORDER --mode I --size M --platforms backend
```

应能够：

- 生成 Feature 工作区
- 自动包含 `impact-analysis.md`

## 7.2 兼容性验收

必须满足：

- 旧的 `spec-first init --feat ...` 用法仍然可用
- 不因引入新轨道而破坏 bootstrap 能力
- 不因文档更新而引入新旧命令长期并存

## 7.3 文档验收

必须满足：

- Skill 文档不再宣称未实现的能力
- CLI 帮助与实际参数一致
- 场景文档只描述已实现路径
- 仓库中不再保留 `spec-first init feature` 作为推荐写法

## 7.4 测试验收

### TDD 流程合规性（前置要求）

在运行测试前，必须确认：

- 新增功能测试遵循 TDD 流程：先写失败测试（RED），再写实现（GREEN）
- commit 记录中须包含 RED 阶段证据，或提供明确的 WAIVER 说明（格式：`Scope / Reason / Alternative Verification / Approver`）
- 不得出现"只有 GREEN、没有 RED/WAIVER"的伪闭环

### 正向场景（必须通过）

必须至少通过：

```bash
npx vitest run tests/unit/init*.test.ts tests/unit/init-router.test.ts tests/integration/init-bootstrap.test.ts tests/e2e/core-flow.test.ts
npm run typecheck
```

建议通过：

```bash
npm test
```

### 反向/边界场景（必须覆盖）

以下场景必须有对应测试用例：

- healthy 项目不被误路由到 brownfield 轨道
- 已有 baseline Feature 不被重复创建
- 无参数 + 非 TTY 下不挂起，应输出帮助或校验错误
- `Mode N` 不生成 `impact-analysis.md`
- `first` 缺失时给出明确错误，不静默失败

## 8. 回归重点清单

以下是实现完成后必须重点回归的风险点：

- `first` 缺失时是否仍会意外直接失败
- healthy 项目是否被误判为 brownfield baseline
- baseline preset 是否会和已有 FEAT 注册冲突
- `Mode N` 是否误生成 `impact-analysis.md`
- `.spec-first/current` 是否在各轨道下都正确更新
- 非 TTY 下是否能正确回退到帮助或参数校验
- bootstrap 默认值是否与文档一致

---

## 9. 建议的代码审查重点

代码审查时，应优先盯以下问题：

### R1. 路由逻辑是否可读

不能把所有分支继续堆在 `handleInit` 一处，必须拆成清晰函数。

### R2. 是否复用了已有能力

尤其是：

- maturity detector
- first 执行逻辑
- Feature Init Core

### R3. 是否出现无必要的新状态

不要为了解决入口问题引入额外复杂全局状态。

### R4. 文档是否同步提交

若代码提交不带文档同步，默认视为不通过。

### R5. 测试是否覆盖三轨

如果没有三轨测试，默认视为改造不完整。

---

## 10. 最终放行条件

只有同时满足以下条件，才允许认为 `01-init` 改造完成：

1. 三轨初始化路径都能跑通
2. `Feature Init Core` 稳定性未退化
3. `Mode I` 骨架补齐到位
4. Skill / CLI / 场景文档已统一
5. 关键测试通过
6. 回归重点项已人工 smoke check

---

## 11. 建议执行方式

本清单建议作为实施前会议或实现前自审 checklist 使用。

推荐流程：

1. 先逐条打勾本清单
2. 再进入开发任务拆解文档执行
3. 每完成一阶段，回到本清单复核相应验收项

如果你要进入代码实现，建议下一步直接按开发任务拆解文档执行，并把本文件作为 review gate。

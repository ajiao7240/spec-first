## 3. 分级发现（Findings）

## 3.1 HIGH

### F-01 声明式上下文“单路径化”后，阶段覆盖不完整
- 现状：文档将上下文文件限定为 `01~05`，并要求移除 `STAGE_LAYERS` 静态映射。
- 证据：
  - 指南定义：`contexts/01_specify.jsonl` ~ `05_verify.jsonl`
  - 任务卡要求：移除 `STAGE_LAYERS`
  - 代码现实：阶段枚举包含 `00_init/06_wrap_up/07_release/08_done/09_cancelled`
- 风险：当 Feature 处于 06+ 阶段执行 `ai context` 或相关流程时，单路径上下文可能无定义，导致构建失败或行为未定义。
- 建议：
  1. 明确阶段覆盖策略（二选一，不保留模糊地带）：
     - A. 补齐 `00/06/07/08/09` 的上下文声明文件模板；
     - B. 明确规定 06+ 阶段上下文构建禁用并给出结构化提示。
  2. 将该策略写入指南和任务卡验收标准，避免实施分歧。

### F-02 Plan Reject Guard 的落点设计偏“解析层”，执行层闭环不足
- 现状：任务卡将 Plan Reject Guard 主要接入 `dispatcher.ts`。
- 证据：`dispatcher.ts` 当前职责是“命令解析与路由”，不承担 feature 状态判定和执行控制。
- 风险：把守卫放在解析层，容易出现“能报错但无法形成统一执行闸门”的碎片化治理。
- 建议：
  1. 将 Guard 核心落在执行入口（如 orchestrate 运行时入口 + hard-gate 协同）；
  2. `dispatcher.ts` 仅做参数合法性检查，不承载业务拒绝决策。

### F-03 更新引擎方案缺“版本语义与迁移元数据”闭环
- 现状：文档强调 hash/migration/backup，但缺 version gate、breaking 提示、迁移指南聚合。
- Trellis 借鉴点：
  - `CLI version vs project version` 门禁
  - `allow-downgrade` 明确开关
  - `migration metadata`（changelog/breaking/guide/aiInstructions）
- 风险：迁移执行可能在错误版本上下文触发，造成不可预期变更。
- 建议：
  1. 在 spec-first 增加 `.spec-first/.version` 与版本比较逻辑；
  2. 增加迁移元数据结构（至少含 breaking + guide）；
  3. `update --dry-run` 输出版本差异和迁移摘要。

### F-04 迁移执行缺少“冲突分级 + 交互策略”规范
- 现状：任务卡有 `executeMigration()` 和回滚，但未规范冲突分类输出契约。
- Trellis 借鉴点：`auto/confirm/conflict/skip` 四分法 + 执行顺序排序 + 冲突交互策略。
- 风险：实施团队容易做成“遇冲突直接失败/直接覆盖”的二元逻辑，难以稳定推广。
- 建议：
  1. 将迁移分类模型写入任务卡验收标准；
  2. 规定 `--dry-run` 必须输出分类统计与逐项计划；
  3. 保留非交互模式（CI）策略：有冲突即失败并输出修复建议。

## 3.2 MEDIUM

### F-05 HostRegistry 目标正确，但接口尚缺“模板采集能力”
- 现状：接口重点在 path/capability/installers。
- Trellis 借鉴点：平台注册同时提供模板采集（用于 update 差异分析）。
- 建议：在 HostRegistry 增加 `collectTemplates()` 或等价能力，以支撑更新引擎统一管理。

### F-06 Workspace Journal 索引目标合理，但缺旋转与归档规则
- 现状：任务卡定义索引与查询性能。
- Trellis 借鉴点：journal 轮转与 index 自动维护。
- 建议：补充单文件上限、轮转触发、重建策略，防止长期运行后索引退化。

## 3.3 LOW

### F-07 文件名版本与文档头版本不一致

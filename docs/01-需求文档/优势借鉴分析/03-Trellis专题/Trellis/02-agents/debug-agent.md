# Debug Agent 详解

> 本文档详细分析 Trellis 的 Debug Agent（调试代理）

---

## 1. 核心定位

### 1.1 角色定义

**问题修复专家** — 分析实现或检查阶段的问题，执行精确修复。

```
┌─────────────────────────────────────────────────────────────┐
│                    Debug Agent 定位                          │
├─────────────────────────────────────────────────────────────┤
│  核心能力：                                                  │
│  ├─ 分析错误日志和堆栈跟踪                                    │
│  ├─ 定位问题根因                                             │
│  ├─ 执行精确修复（最小变更原则）                              │
│  └─ 验证修复有效性                                           │
│                                                              │
│  触发条件：                                                  │
│  ├─ Implement Agent 失败                                     │
│  ├─ Check Agent 发现问题                                     │
│  ├─ 测试失败                                                 │
│  └─ 用户报告 Bug                                             │
│                                                              │
│  输出：                                                     │
│  ├─ 问题根因分析                                             │
│  ├─ 修复代码                                                 │
│  └─ 验证结果                                                 │
└─────────────────────────────────────────────────────────────┘
```

### 1.2 设计原则

| 原则 | 说明 |
|------|------|
| **最小变更** | 只修改修复问题必要的代码 |
| **根因分析** | 找到根本原因，不做临时补丁 |
| **验证修复** | 修复后必须验证问题已解决 |
| **规范遵循** | 修复代码必须符合项目规范 |

## 2. 执行流程

### 2.1 调试流程

```
┌─────────────────────────────────────────────────────────────┐
│                    Debug Agent Flow                          │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│  Step 1: 接收问题                                            │
│          从 Dispatch Agent 接收错误上下文                     │
│          │                                                   │
│          ▼                                                   │
│  Step 2: Hook 注入上下文                                     │
│          PreToolUse Hook 读取 debug.jsonl                    │
│          注入调试相关规范                                      │
│          │                                                   │
│          ▼                                                   │
│  Step 3: 分析错误信息                                        │
│          ├─ 解析错误日志                                      │
│          ├─ 分析堆栈跟踪                                      │
│          └─ 识别错误类型                                      │
│          │                                                   │
│          ▼                                                   │
│  Step 4: 定位问题位置                                        │
│          ├─ 确定问题文件                                      │
│          ├─ 确定问题行号                                      │
│          └─ 确定问题范围                                      │
│          │                                                   │
│          ▼                                                   │
│  Step 5: 根因分析                                            │
│          ├─ 分析代码逻辑                                      │
│          ├─ 检查数据流                                        │
│          └─ 识别根本原因                                      │
│          │                                                   │
│          ▼                                                   │
│  Step 6: 设计修复方案                                        │
│          ├─ 评估修复影响                                      │
│          ├─ 选择最小变更方案                                  │
│          └─ 验证方案可行性                                    │
│          │                                                   │
│          ▼                                                   │
│  Step 7: 执行修复                                            │
│          ├─ 应用代码修改                                      │
│          └─ 更新相关测试                                      │
│          │                                                   │
│          ▼                                                   │
│  Step 8: 验证修复                                            │
│          ├─ 运行相关测试                                      │
│          ├─ 检查是否引入新问题                                │
│          └─ 确认问题已解决                                    │
│          │                                                   │
│          ▼                                                   │
│  Step 9: 输出修复报告                                        │
│          根因分析 + 修复内容 + 验证结果                        │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

## 3. 问题分析框架

### 3.1 五维度分析

```
┌─────────────────────────────────────────────────────────────┐
│                   五维度问题分析                              │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│  维度 1: 错误类型                                            │
│  ├─ 编译错误                                                 │
│  ├─ 运行时错误                                               │
│  ├─ 逻辑错误                                                 │
│  ├─ 性能问题                                                 │
│  └─ 安全问题                                                 │
│                                                              │
│  维度 2: 影响范围                                            │
│  ├─ 单个文件                                                 │
│  ├─ 单个模块                                                 │
│  ├─ 跨模块                                                   │
│  └─ 全局影响                                                 │
│                                                              │
│  维度 3: 优先级                                              │
│  ├─ P0: 阻塞问题（必须立即修复）                              │
│  ├─ P1: 严重问题（本迭代修复）                                │
│  ├─ P2: 一般问题（下迭代修复）                                │
│  └─ P3: 优化建议（可延后）                                    │
│                                                              │
│  维度 4: 修复复杂度                                          │
│  ├─ 简单（单行修改）                                          │
│  ├─ 中等（多行/多文件）                                       │
│  ├─ 复杂（逻辑重构）                                          │
│  └─ 困难（架构调整）                                          │
│                                                              │
│  维度 5: 风险评估                                            │
│  ├─ 低风险（无副作用）                                        │
│  ├─ 中风险（局部影响）                                        │
│  └─ 高风险（全局影响）                                        │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

## 4. 修复策略

### 4.1 最小变更原则

```
┌─────────────────────────────────────────────────────────────┐
│                    最小变更原则                               │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│  ✅ 应该做：                                                 │
│  ├─ 只修改问题相关的代码                                      │
│  ├─ 保持现有代码风格                                          │
│  ├─ 不改变公共接口                                            │
│  ├─ 不添加不必要的功能                                        │
│  └─ 验证修复后运行测试                                        │
│                                                              │
│  ❌ 不应该做：                                               │
│  ├─ 重构与问题无关的代码                                      │
│  ├─ 添加"顺便优化"的改动                                      │
│  ├─ 修改不相关的文件                                          │
│  ├─ 改变现有行为                                              │
│  └─ 引入新的依赖                                              │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

### 4.2 修复策略选择

| 问题类型 | 修复策略 | 说明 |
|----------|----------|------|
| **编译错误** | 直接修复 | 修改类型/语法 |
| **运行时错误** | 添加检查 | 空值检查/边界检查 |
| **逻辑错误** | 修正逻辑 | 修改条件/算法 |
| **性能问题** | 优化实现 | 算法优化/缓存 |
| **安全问题** | 加固代码 | 输入验证/转义 |

### 5.1 协作流程

```
┌──────────────┐    失败     ┌──────────────┐
│  Implement   │────────────▶│    Debug     │
│    Agent     │             │    Agent     │
└──────────────┘             └──────┬───────┘

┌──────────────┐    问题     │      │
│    Check     │─────────────┤      │
│    Agent     │             │      │
└──────────────┘             │      ▼
                             │ ┌──────────────┐
                             │ │    修复后    │
                             │ │    重新检查  │
                             │ └──────────────┘
                             │
                             ▼
                      ┌──────────────┐
                      │   返回给     │
                      │   Dispatch   │
                      └──────────────┘
```

### 5.2 触发条件

```python
# Dispatch Agent 调用 Debug Agent 的条件
def should_dispatch_debug(previous_agent, result):
    if previous_agent == "implement" and result.failed:
        return True

    if previous_agent == "check" and result.has_issues:
        return True

    return False
```

## 7. 核心规则

### 7.1 六大铁律

| 规则 | 说明 |
|------|------|
| **1. 根因分析** | 必须找到根本原因，不做临时补丁 |
| **2. 最小变更** | 只修改必要的代码 |
| **3. 验证修复** | 修复后必须验证问题已解决 |
| **4. 规范遵循** | 修复代码必须符合项目规范 |
| **5. 无副作用** | 不引入新的问题 |
| **6. 文档更新** | 必要时更新相关文档 |

### 7.2 调试检查清单

```markdown
## Debug Agent 检查清单

- [ ] 已分析错误日志和堆栈
- [ ] 已定位问题位置
- [ ] 已确定根本原因
- [ ] 已评估修复影响
- [ ] 已选择最小变更方案
- [ ] 已执行修复
- [ ] 已验证修复有效
- [ ] 已检查无新问题引入
```

## 8. 对 spec-first 的借鉴价值

### 8.1 核心借鉴点

| 借鉴点 | 优先级 | 说明 |
|--------|--------|------|
| **五维度分析框架** | P1 | 结构化问题分析方法 |
| **最小变更原则** | P0 | 修复只触及必要范围 |
| **根因分析模板** | P2 | 标准化问题分析文档 |
| **验证机制** | P1 | 修复后必须验证 |

### 8.2 实现建议

```python
class DebugAgent:
    """Debug Agent - 问题修复"""

    def execute(self, error_context, context: InjectedContext):
        """执行调试任务"""
        # 1. 分析错误
        error_analysis = self.analyze_error(error_context)

        # 2. 定位问题
        problem_location = self.locate_problem(error_analysis)

        # 3. 根因分析
        root_cause = self.find_root_cause(problem_location)

        # 4. 设计修复方案
        fix_plan = self.design_fix(root_cause)

        # 5. 执行修复
        fix_result = self.apply_fix(fix_plan)

        # 6. 验证修复
        verification = self.verify_fix(fix_result)

        return DebugOutput(
            root_cause=root_cause,
            fix_result=fix_result,
            verification=verification
        )

    def apply_fix(self, fix_plan):
        """应用修复 - 遵循最小变更原则"""
        # 只修改必要的代码
        changes = []
        for change in fix_plan.changes:
            if change.is_necessary:
                changes.append(apply_change(change))
        return changes
```

#### MF-1：`generic` 宿主 `supportsSkills` 与参考文档口径不一致

**位置**：`src/core/tool-integration/capability-matrix.ts` 第 39 行

**问题代码**：
```typescript
{
  host: 'generic',
  supportsSkills: true,   // ← 写死 true
  supportsMcp: false,
  supportsHooks: false,
  supportsSessionStart: false,
  supportsViewer: false,
  supportsBrowser: false,
  supportsProjectScopedConfig: false,
},
```

**文档声明**（`docs/reference/host-capability-matrix.md`）：
```
| Generic | planned | ✅ | ❌ | ❌ | ❌ | ❌ | ❌ | ❌ | 弱宿主，仅支持 skills 导出与策略降级 |
```

**核验结果**：
``` 
代码：getHostCapability('generic')?.supportsSkills → true
当前文档：host-capability-matrix.md 已将 Generic 标为 Skills ✅
补充核验：shared/skill-commands.ts 已实现 generic skills 安装链路，tests/unit/skill-commands.test.ts 也有 generic host target 覆盖
契约测试：tests/unit/host-adapters.test.ts 已补充 generic supportsSkills/supportsMcp 断言
结论：本项已按“generic = 支持 skills 导出的弱宿主”口径完成对齐
```

**影响**：原始问题会导致文档、代码、`doctor/update` 输出分叉，并误导下游对 generic 能力边界的判断。

**处理决定**：
```typescript
采用当前实现口径：
generic = 支持 skills 导出，但不支持 MCP / browser / hooks / session / viewer 的弱宿主。
本次通过修正文档并补契约测试完成对齐，不再要求修改代码。
```

**最佳实践优化方案（按 SDD 分层）**：
- 规范决策：
  - 先定义 `supportsSkills` 的字段语义，它必须只表达一种含义：要么是“宿主静态支持该能力”，要么是“当前环境已具备该能力”，不能混用。
  - 将宿主能力定义收敛为单一事实源。推荐做法是：代码中的能力表为主，文档由代码生成或至少由代码驱动校对。
  - 为 `generic` 明确定义产品定位。推荐口径：`generic` 是“弱宿主/导出宿主”，支持 skills 导出，不支持 MCP、browser、hooks、session/viewer。
- 当前必做：
  - 修订 `host-capability-matrix.md`，将 Generic 更新为“弱宿主，仅支持 skills 导出与策略降级”。
  - 补一组能力契约测试，断言 `generic` 的能力声明与 `doctor/update` 输出口径一致。
- 后续演进：
  - 再考虑引入“能力声明层 + 环境探测层”的双层模型，避免能力字段继续混入运行态状态。

#### MF-2：`code-analysis` 场景硬编码返回 `serena`，未过滤不支持的宿主

**位置**：`src/core/tool-integration/tool-selection.ts` 第 21-22 行

**问题代码**：
```typescript
case 'code-analysis':
  return { primary: ['serena'], fallback: ['shell-rg'] };  // 硬编码，不检查宿主
```

**对比（同文件 external-research 已正确实现）**：
```typescript
case 'external-research':
  return {
    primary: ['fetch', 'context7'].filter((tool) => isToolSupported(host, tool)),  // ✅ 正确
    fallback: ['browser', 'manual-official-docs'],
  };
```

**serena 的宿主定义**（`tool-registry.ts`）：
```typescript
{
  id: 'serena',
  hosts: ['claude', 'codex', 'gemini', 'cursor'],  // 不含 generic
  requiredMcps: ['serena'],
  ...
}
```

**验证结果**：
```
调用：selectToolsForScenario('generic', 'code-analysis')
当前输出：{ primary: ['serena'], fallback: ['shell-rg'] }
期望输出：{ primary: [],         fallback: ['shell-rg'] }
是否有 bug：true
```

**影响**：`generic` 宿主不支持 MCP（`supportsMcp: false`），无法运行需要 MCP 的 `serena`，但当前实现仍会将其推荐为 primary 工具，导致工具选择策略失效。

**修复方案**：
```typescript
case 'code-analysis':
  return {
    primary: isToolSupported(host, 'serena') ? ['serena'] : [],
    fallback: ['shell-rg'],
  };
```

**最佳实践优化方案（按 SDD 分层）**：
- 规范决策：
  - 将“工具支持矩阵”和“场景策略”拆成两个独立概念：注册表负责“工具在哪些 host 可用”，选择器负责“当前场景如何排序和回退”。
- 当前必做：
  - 修复当前硬编码逻辑，`code-analysis` 必须基于宿主支持情况过滤 `serena`。
  - 补参数化测试，至少覆盖 `generic + code-analysis`、`codex + code-analysis`、`generic + external-research`。
- 后续演进：
  - 工具选择逐步收敛为注册表驱动，避免在 `tool-selection.ts` 中继续硬编码 primary 工具。
  - 如场景继续增加，再抽出统一的场景解析助手，例如“按 scenario 找候选工具，再按 host 过滤，再按策略排序”。

- 明确定义 capability 字段语义，特别是 `supportsSkills` 属于静态能力还是运行态可用性。
- 明确 `generic` 的产品定位，并将其作为唯一口径同步到代码、文档、`doctor/update` 输出。
- 明确能力矩阵与环境探测的边界，避免同一字段混合表达“声明能力”和“当前环境状态”。

### 2. 当前迭代必做

- 修复 MF-2：`code-analysis` 按宿主过滤 `serena`。
- 补齐 `host × scenario` 的关键测试矩阵，至少覆盖 `generic + code-analysis`。
- 对齐 MF-1 的代码与文档口径，在一个迭代内完成，不允许继续分叉。
- 修正 SF-2：避免暴露可变注册表引用。

### 3. 后续演进

- 将工具选择进一步收敛为注册表驱动，而不是分散硬编码。
- 引入能力声明层与环境探测层分离的模型。
- 若 host 数量和状态复杂度继续增加，再为 adapter 引入单次探测上下文/状态快照。
- 对 `doctor/update` 建立契约测试，确保文档、能力矩阵、命令输出三者长期一致。

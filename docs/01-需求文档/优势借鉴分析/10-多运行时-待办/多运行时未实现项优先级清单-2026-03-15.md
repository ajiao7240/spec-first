# 多运行时未实现项优先级清单

> 范围：对照 `Spec-First多运行时参考方案-2026-03-15.md` 与 `Spec-First-Host-Adapter开发任务文档.md`，整理当前尚未完整实现、但后续是否值得继续推进的事项。
>
> 日期：2026-03-15
> 结论基线：当前多运行时能力已具备 `Host Adapter + capability + update/doctor 多宿主接入`，但尚未形成文档定义的完整平台闭环。

---

## 一、结论

当前最值得做的，不是继续新增宿主，而是先把多运行时底层抽象收口。

优先级建议如下：

- `P0 / 现在值得做`：补齐 `mutation plan -> executor -> validator` 三段式闭环，并继续把 `init / postinstall / manifest` 收敛到 adapter 架构。
- `P1 / 可后置`：新增 `OpenCodeAdapter` 作为第三宿主验证样板；评估是否提升 `generic` 宿主级别。
- `P2 / 暂不建议现在做`：仅为追齐文档文件名而补空壳模块，或在底层抽象未稳定前继续扩更多宿主。

---

## 二、P0：现在值得做

### 1. Host Mutation Planner

- 当前状态：未实现独立 planner。
- 为什么值得现在做：
  - 这是把多运行时从“能用”升级为“可扩展”的关键分层。
  - 没有 planner，`update` 仍然会混合宿主识别、变更决策和执行逻辑。
- 当前问题：
  - 难以做真正的 `dry-run / 审计 / 回滚 / 测试隔离`。
  - 每接入新宿主都更容易把命令层越改越大。
- 建议动作：
  - 抽出 `HostMutation`、`HostMutationPlan`、`planSkillSync`、`planMcpConfig`、`planHooks`。
- 目标结果：
  - Adapter 只产出计划，不直接写文件。

### 2. Host Mutation Executor

- 当前状态：未实现独立 executor。
- 为什么值得现在做：
  - 没有 executor，planner 即使做出来也无法真正统一落盘逻辑。
  - 当前执行路径分散在 `update`、bootstrap、hook 安装等多处。
- 建议动作：
  - 抽出统一执行器，负责 `apply / dry-run / backup / rollback`。
- 目标结果：
  - 多宿主文件写入行为可复用、可测试、可审计。

### 3. Host Validator / ValidationResult

- 当前状态：`doctor` 能诊断，但没有独立 validator 层。
- 为什么值得现在做：
  - `doctor` 目前既做诊断聚合，也承接部分宿主规则。
  - 没有 validator，宿主健康判断难以在 `doctor / init / postinstall` 之间复用。
- 建议动作：
  - 抽出 `HostValidationIssue / HostValidationResult`。
  - 让 `doctor` 只负责渲染报告，不直接承载全部宿主规则。
- 目标结果：
  - 诊断逻辑复用化，报告输出与校验逻辑解耦。

### 4. init / postinstall 继续走 adapter

- 当前状态：`update`、`doctor` 已较多接入 adapter；`postinstall` 仍保留明显的宿主直写和 `claude/codex` 分支。
- 为什么值得现在做：
  - 如果只有一半命令走新架构，系统会长期处于双轨状态。
  - 新增宿主时会出现“update 支持，postinstall 不支持”的行为裂缝。
- 建议动作：
  - 让 `init / postinstall` 至少复用 registry、capability、validator 结果。
- 目标结果：
  - 多运行时行为在安装、初始化、诊断、升级四个入口保持一致。

### 5. 宿主配置真源继续 adapter 化

- 当前状态：`bootstrap-manifest.ts` 仍保留较强的 `claude/codex` 双字段结构。
- 为什么值得现在做：
  - 这是阻碍后续新增宿主的主要历史结构之一。
  - 若不继续收敛，adapter 层会变成外面新架构、里面旧模型的包裹层。
- 建议动作：
  - 逐步把宿主专属字段收敛为 capability + adapter plan。
- 目标结果：
  - 新增宿主时尽量只加 adapter，而不是继续扩大 manifest 分支。

## 三、P1：可后置

### 6. OpenCodeAdapter

- 当前状态：未实现。
- 为什么不是现在最急：
  - 它主要用于验证扩展性，不是当前用户价值最高的缺口。
  - 如果底层三段式未完成，先接第三宿主只会放大结构债。
- 什么时候值得做：
  - planner / executor / validator 基本稳定后。
- 建议定位：
  - 作为“新宿主接入样板”，验证 adapter 模型是否足够通用。

### 7. generic 提升为正式 full host

- 当前状态：`generic` 仍是弱宿主。
- 为什么后置：
  - generic 的价值更偏导出和兼容，而不是高能力宿主。
  - 把它提升为 full host 的投入，未必优于先把真实宿主链路做稳。
- 什么时候值得做：
  - 当产品明确需要一个“非 Claude/Codex/Gemini/Cursor 的一等宿主”时。

---

## 四、P2：暂不建议现在做

### 8. 为追齐文档而补空壳模块

- 当前状态：文档中提到的 `base-adapter.ts`、`mutation-plan.ts`、`mutation-executor.ts`、`validation.ts` 并未全部存在。
- 为什么不建议现在直接补：
  - 如果只是为了让文件名齐全，而没有真实职责边界，会制造新的伪抽象。
  - 这类“形式实现”会让后续重构更困难。
- 建议：
  - 只在职责已经明确时落文件，不为追文档目录结构而补壳。

### 9. 在底层未稳定前继续扩更多宿主

- 当前状态：已经有 `gemini/cursor`，但平台闭环还不完整。
- 为什么不建议现在做：
  - 每新增一个宿主，都会放大当前架构未收口的问题。
  - 容易把多运行时做成大量宿主特判的堆积系统。
- 建议：
  - 先收敛底层，再扩宿主数量。

---

## 五、建议执行顺序

建议按以下顺序推进：

1. 补 `Host Mutation Planner`
2. 补 `Host Mutation Executor`
3. 补 `Host Validator / ValidationResult`
4. 让 `init / postinstall` 继续复用 adapter
5. 继续收敛宿主配置真源
6. 再引入 `OpenCodeAdapter` 做第三宿主样板
7. 最后再决定是否提升 `generic`

---

## 六、最小判断标准

可以认为“多运行时底层值得继续往下做”的完成标准是：

- 新增一个宿主时，主要工作是新增 adapter，而不是修改多处 CLI 分支。
- `update / doctor / init / postinstall` 对宿主的理解来自同一套 registry + capability + validation。
- `dry-run` 不再只是命令级打印，而是基于统一 mutation plan 的可预览执行。
- 宿主健康检查、修复建议、落盘执行三者职责分离。

在达到这些标准之前，优先补底层收敛，优先级高于继续扩宿主数量。

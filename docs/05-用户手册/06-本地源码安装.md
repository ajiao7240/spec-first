# 本地源码安装指南

这份指南描述的是“下载源码后，本地打包并安装到目标项目里生成 Claude Code 的 `/spec:*` 命令，以及 Codex 的 `$spec-*` skills”的流程。

## 前置要求

- Node.js >= 20
- npm >= 8
- Claude Code 或 Codex CLI 已安装

## 模式说明

当前仓库已与 `spec-first` CLI 模型对齐：

- 发布入口是 npm 包 `spec-first`
- 本地安装模型是 `npm install -g spec-first` 后执行 `spec-first init --claude` 或 `spec-first init --codex`
- Claude Code 的 `/spec:*` 命令来自项目内 `.claude/commands/spec/`
- Codex 的 `$spec-*` 入口来自项目内 `.agents/skills/spec-*/SKILL.md`，不会生成 `.codex/commands/spec/`
- `graph-bootstrap` 是 external graph-provider readiness facts 编译入口
- Claude `commands/spec/*.md` 会委托给对应平台运行时目录中的 workflow skill；Codex 直接交付 workflow skill
- `spec-first init` 会把随包发布的 standalone skills 同步到 Claude Code 的 `.claude/skills/` 或 Codex 的 `.agents/skills/`
- `spec-first init` 会把 Claude command-backing workflow skills 同步到 `.claude/spec-first/workflows/`
- `spec-first init` 会把随包发布的 agents 同步到 Claude Code 的 `.claude/agents/` 或 Codex 的 `.codex/agents/`
- `spec-first init` 会把随包发布的 agent support files 同步到 Claude Code 的 `.claude/agents/` 或 Codex 的 `.codex/agents/`
- `spec-first init` 还会把命令模板同步到 Claude Code 的 `.claude/commands/spec/`
- `spec-first init` 还会在项目内写入 `.claude/spec-first/.developer` 或 `.codex/spec-first/.developer`
- `spec-first init` 还会在项目内写入严格 schema 的 `.claude/spec-first/state.json` 或 `.codex/spec-first/state.json`
- `spec-first init` 支持显式传入 `-u/--user` 和 `--lang`；如果不传用户名，会优先回退到全局 `~/.spec-first/.developer`，再回退到 `git config user.name`
- 发布包内运行时资产清单由治理 JSON、command template frontmatter、`skills/` 和 `agents/` 生成，不再发布 `.claude-plugin/plugin.json`
- 不再推荐把当前仓库直接注册成 Claude plugin

## 推荐安装

### 1. 获取源码

```bash
git clone https://github.com/sunrain520/spec-first.git
cd spec-first
```

### 2. 打包源码

```bash
npm pack
```

这一步会生成一个本地 tarball，例如 `spec-first-<version>.tgz`。

### 3. 安装本地 tarball

```bash
npm install -g ./spec-first-<version>.tgz
```

> 当前发布包不再携带旧内置图运行时的 native parser 依赖；如果安装期仍出现 peer dependency 警告，优先按实际缺失包名排查本机 npm 缓存或全局环境，而不是按旧内置图路径处理。

这一步就是”全局安装”到系统 PATH。安装成功后，直接运行：

```bash
which spec-first
spec-first --version
spec-first -v
```

如果 `which spec-first` 仍然指向旧路径，先执行：

```bash
hash -r
```

再重复上面的检查。

`spec-first -v` 会显示欢迎页和版本信息；安装阶段的 `postinstall` 提示不作为稳定承诺。

### 4. 诊断宿主环境并初始化目标平台

安装完成后的**推荐顺序**：先 `doctor` 检查环境，再 `init` 初始化项目。

```bash
# 安装后第一个命令：检查环境
spec-first doctor
# 或显式指定平台
spec-first doctor --claude
spec-first doctor --codex

# 初始化目标项目
spec-first init --claude
# 或
spec-first init --codex
```

如果你希望显式指定初始化身份，可以改成：

```bash
spec-first init --claude -u <name> --lang <zh|en>
spec-first init --codex -u <name> --lang <zh|en>
```

如果 `doctor` 提示 `legacy managed state detected`，或者 `clean` 明确拒绝当前 legacy 安装，不要先手工删除 runtime 目录，直接重新运行对应平台的 `spec-first init`。当前版本采用 hard-cut 升级策略：

- `init` 是唯一支持的 legacy 升级入口
- `init` 会先执行 managed hard reset，再全量重建运行时
- `clean` 只清理当前受管资产，不负责 legacy 迁移

如果你之前装过 pnpm 版本或旧的全局 shim，先在新终端里执行 `hash -r`，或者直接重开一个 shell，再运行上面的命令。这样可以避免命令缓存继续指向旧路径。

### 5. 重启目标平台

**重要**：`spec-first init` 写入项目内运行时目录后，必须完全退出并重启 Claude Code 或 Codex 才能稳定识别新入口。

目录存在只能证明资产已生成，不能单独证明宿主已经发现 `spec-graph-bootstrap`；必须完全重启宿主进程后再验证。

**macOS 退出方式：**
- 使用 `Cmd+Q` 完全退出应用
- 或运行 `pkill -f "claude"` 杀掉进程
- 不是关闭窗口，必须退出应用

然后重新启动：
```bash
# Claude Code
claude

# Codex
codex
```

### 6. 验证安装

```bash
# 检查项目内命令目录
# Claude Code
ls .claude/commands/spec

# 检查项目内技能目录
# Claude Code
ls .claude/skills
ls .claude/spec-first/workflows
# Codex
ls .agents/skills | rg '^spec-'

# 检查项目内代理目录
# Claude Code
find .claude/agents -type f | head
# Codex
find .codex/agents -type f | head

# 在目标平台中测试
# Claude Code
/spec:ideate --help
/spec:brainstorm --help
/spec:graph-bootstrap
/spec:skill-audit

# Codex
$spec-ideate
$spec-brainstorm
$spec-graph-bootstrap
$spec-skill-audit
```

如果 Claude Code 能识别 `graph-bootstrap` / `skill-audit` 入口，或 Codex 能在 skill discovery / 最小探测调用中看到 `spec-graph-bootstrap` / `spec-skill-audit`，说明当前 supporting workflows 安装成功。

注意：

- `mcp-setup` 负责安装和验证 required runtime、MCP servers、graph providers 与 helper tools
- `graph-bootstrap` 在 setup ready 后编译 external graph-provider readiness facts，并写入 `.spec-first/graph/`、`.spec-first/impact/` 和 `.spec-first/providers/`
- `skill-audit` 审查 source skills、治理投递、runtime drift 与安全信号，并写入 `.spec-first/audits/skill-audit/`
- `compound` 适合更偏知识捕获与复合上下文整理的场景
- 如果宿主内看不到 `spec-graph-bootstrap` 或 `spec-skill-audit` 相关资产，说明当前运行时没有完成安装或未重启，需要重新 `init` 后再完全重启宿主
- 对 Codex 而言，正式可用性以 `.agents/skills/spec-graph-bootstrap/` 和 `.agents/skills/spec-skill-audit/` 的宿主发现结果为准

> 提示：`npm install -g ./spec-first-<version>.tgz` 里的 `<version>` 应替换成你刚刚 `npm pack` 生成的实际版本号。

如果需要确认全局命令版本，可执行：
```bash
spec-first --version
```

## 源码模式

如果只是想从当前源码仓库确认安装路径，可运行：

```bash
./install-local.sh
```

这个脚本现在只输出与源码打包安装一致的本地验证步骤，不再写入 `~/.claude/plugins`。
它也会明确提示：`graph-bootstrap` 是 external graph-provider readiness facts 编译入口。

## 测试安装

运行完整测试套件：

```bash
# Smoke + Integration E2E 测试
npm test

# 单元 / smoke / integration 分层验证
npm run test:unit
npm run test:smoke
npm run test:integration
```

`npm test` 应该看到：
```
=== E2E 测试全部通过 ✓ ===
```

`npm run test:unit`、`npm run test:smoke` 和 `npm run test:integration` 分别覆盖单元契约、CLI/init/doctor smoke，以及 workflow 级集成检查。需要一次性跑主链路时使用 `npm test`。

## 开发模式

如果需要修改代码：

### 快速重载（推荐）

```bash
# 修改代码后运行
./dev-reload.sh

# 然后按提示执行 doctor/init
```

此脚本现在会提醒使用与 npm 发布版一致的本地验证流程。

### 手动重载

```bash
# 1. 重新诊断并生成项目命令
spec-first doctor
spec-first doctor --claude
spec-first doctor --codex
spec-first init --claude
spec-first init --codex

# 2. 完全退出目标平台
# Claude Code
pkill -f "claude"
# Codex
pkill -f "codex"

# 3. 重新启动
# Claude Code
claude
# Codex
codex
```

### 测试修改

```bash
# 运行测试
npm test

# 在目标平台中测试
# Claude Code
/spec:brainstorm "测试功能"

# Codex
$spec-brainstorm "测试功能"
```

**注意**：当前平台运行时不支持热加载，每次修改后必须重启

## 卸载

```bash
# 卸载全局 CLI
npm uninstall -g spec-first

# 删除项目内受管资产
spec-first clean --claude
spec-first clean --codex

# 删除源码
cd ..
rm -rf spec-first
```

重启 Claude Code 或 Codex 完成卸载。

## 故障排查

### 问题1：源码安装失败

```bash
# 检查 Node.js 是否安装
node --version

# 如果没有，安装 Node.js
brew install node  # macOS
```

### 问题2：找不到 skills

```bash
# 检查项目命令目录
# Claude Code
ls -la .claude/commands/spec

# 检查项目技能目录
# Claude Code
ls -la .claude/skills
ls -la .claude/spec-first/workflows
# Codex
ls -la .agents/skills

# 重新生成
spec-first init --claude
spec-first init --codex

# 运行环境诊断
spec-first doctor
spec-first doctor --claude
spec-first doctor --codex

# 重新打包并重新安装
npm pack
npm install -g ./spec-first-<version>.tgz
```

### 问题3：修改不生效

```bash
# 重新生成项目命令
spec-first init --claude
spec-first init --codex

# 完全退出并重启目标平台
```

### 问题4：安装时出现 `npm warn ERESOLVE overriding peer dependency`

当前版本已移除旧内置图运行时的 native parser 依赖，正常安装不应再依赖这些旧解析器组合。如果你仍看到 peer dependency 警告，通常来自本机 npm 缓存、全局依赖或周边工具链。先确认 `spec-first` 自身是否可用：

验证安装是否真正成功：

```bash
spec-first -v
spec-first doctor
```

如果以上命令正常返回，说明 package CLI 安装成功。graph provider readiness 由宿主内的 `spec-mcp-setup` / `spec-graph-bootstrap` workflow 继续验证。如果以上命令无法正常返回，按以下步骤排查：

1. 清除 npm 缓存后重装：`npm cache clean --force && npm install -g ./spec-first-<version>.tgz`
2. 使用 `--legacy-peer-deps` 安装：`npm install -g ./spec-first-<version>.tgz --legacy-peer-deps`
3. 确认 Node.js 版本 >= 20：`node --version`

### 问题5：`spec-first doctor` 仍然指向旧路径

如果报错路径里出现 `packages/cli/bin/spec-first.js`，说明你的 shell 还缓存着旧的 pnpm shim。处理方式：

```bash
hash -r
which spec-first
spec-first doctor
```

如果还不对，直接关闭当前终端并重新打开，再执行 `which spec-first` 和 `spec-first doctor`。

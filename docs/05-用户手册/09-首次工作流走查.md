# 首次工作流走查

本文用一个很小的需求句子展示 `spec-first` 的真实使用方式：

```text
Improve onboarding for first-time CLI users
```

目标不是把 workflow 变成固定状态机，而是让每一步都留下下一步能读取的高质量上下文。脚本负责确定性的安装、校验、生成和路径管理；LLM 负责需求判断、范围取舍、计划拆分和实现决策。

## 前置条件

这一步只在首次使用、换宿主、升级后重建 runtime assets，或 MCP/helper 环境变化时需要执行。后续日常需求不必每次重新安装。

### 0.1 安装 CLI

先在终端安装 `spec-first` CLI：

```bash
npm install -g spec-first
```

如果你在本仓库源码中验证发布物，也可以改用本地 pack 安装：

```bash
npm pack
npm install -g ./spec-first-<version>.tgz
```

安装完成后，在目标项目根目录检查 CLI 与当前宿主状态：

```bash
spec-first doctor
```

### 0.2 初始化当前宿主

只初始化你实际使用的宿主。Codex 项目运行：

```bash
spec-first init
```

Claude Code 项目运行：

```bash
spec-first init
```

初始化后重启宿主或新开会话，让宿主加载刚生成的 runtime assets。后续 `/spec:*` 或 `$spec-*` 入口都在宿主会话里运行，不是 shell 命令。

### 0.3 准备 MCP / helper runtime

进入新的 Claude Code 或 Codex 会话后，先运行当前宿主的环境准备入口：

```text
# Claude Code
/spec:mcp-setup

# Codex
$spec-mcp-setup
```

`mcp-setup` 负责安装并验证 required harness runtime、MCP servers、graph providers 和 helper tools。它输出的是确定性环境事实，不替代后续的需求判断或实现判断。

如果当前目录是父 workspace 且包含多个独立 child Git repos，`mcp-setup` 会默认逐个处理全部 child repos，并只在父目录写 `.spec-first/workspace/*summary.json` advisory summary。后续真正修改、测试或 review 某个项目时，仍需要明确 child repo scope。

### 0.4 编译代码图谱 readiness

当 setup 报告 baseline ready，或提示需要 graph bootstrap 时，继续运行：

```text
# Claude Code
/spec:graph-bootstrap

# Codex
$spec-graph-bootstrap
```

`graph-bootstrap` 会基于 setup 产出的 provider 配置，编译 GitNexus external graph-provider readiness facts，写入 `.spec-first/graph/*`、`.spec-first/providers/*` 和 `.spec-first/impact/*`。这些 facts 的消费边界不同：plan/work/debug 主要使用 GitNexus 做全局图谱理解，code-review 使用 GitNexus review-impact evidence 做 diff impact / review context；如果它 unavailable、stale、definitions-only 或 degraded，LLM 应明确说明限制并退回 bounded direct repo reads，而不是假装已有完整图谱认知。

后续切换分支、pull、rebase、merge 或产生新的 dirty worktree 后，不会自动 rebuild GitNexus index；下一个 graph consumer 会通过 `source_revision` / `worktree_status_hash` 检测 stale。轻量任务可以继续 bounded direct reads，graph-heavy 任务再显式运行 `graph-bootstrap`。

父 workspace 下无参数运行 `graph-bootstrap` 也会默认走 all-child maintenance：每个 child repo 写自己的 canonical graph artifacts，父目录只写 `.spec-first/workspace/graph-bootstrap-summary.json`。

## 1. 从 brainstorm 开始

Codex 会话中：

```text
$spec-brainstorm "Improve onboarding for first-time CLI users"
```

Claude Code 会话中：

```text
/spec:brainstorm "Improve onboarding for first-time CLI users"
```

第一次 brainstorm 通常只生成一个 requirements brief，例如：

```text
docs/brainstorms/2026-05-01-001-cli-onboarding-requirements.md
```

这个文档应该回答：

- 用户是谁，当前卡在哪里
- 哪些需求是当前必须解决的
- 哪些想法明确不在本轮范围内
- 成功标准是什么
- 后续 planning 还需要注意哪些边界

如果这些内容还不清楚，应继续对话澄清，而不是急着进入实现。

## 2. 用 plan 承接需求

当需求 brief 已经足够稳定后，在宿主会话中进入 plan：

```text
$spec-plan
```

或：

```text
/spec:plan
```

计划通常写入：

```text
docs/plans/2026-05-01-001-feat-cli-onboarding-plan.md
```

plan 的职责是把 requirements 转成可评审、可执行的工程决策上下文。它不需要替代 executor 写微步骤，也不应该把所有实现细节提前锁死。

一个好的 plan 至少要说明：

- 实施目标和非目标
- 需要修改或新增的大致文件区域
- 依赖关系和风险点
- 验证方式
- 哪些问题留到 implementation-time 决策

## 3. 大计划再编译 task pack

如果 plan 很小，可以直接进入 work。如果 plan 涉及多个模块、多个阶段或需要多人/多 agent 交接，可以使用已安装的 standalone `write-tasks` skill，把 plan 编译成 task pack：

```text
docs/tasks/2026-05-01-001-feat-cli-onboarding-tasks.md
```

task pack 的价值是确定性 handoff：它记录 source plan、hash、task graph、wave 和验证信号。它不是新的需求真相源；如果 plan 变了，应重新编译或验证 task pack freshness。

## 4. 进入 work 执行

当 plan 或 task pack 可执行后：

```text
$spec-work
```

或：

```text
/spec:work
```

work 阶段会读取当前请求、plan/task pack、AGENTS/CLAUDE 指令、相关源码和测试，然后做最小可验证改动。执行结果通常体现为：

- 代码或文档 diff
- 对应测试或检查命令
- 验证记录和残余风险说明
- 必要时更新 `CHANGELOG.md`

work 不应该修改 generated runtime copies 来绕过 source truth。涉及 skill、agent、template 或 CLI 的变更，应改源码资产，再按需要通过 `spec-first init` 并选择目标宿主 重建 runtime。

## 5. 合并前 review，完成后 compound

如果本次是移动 App 改动，并且需要在模拟器、真机、打包或 QA 前检查 PRD / Figma / source 一致性，可以在 code review 前后插入：

```text
$spec-app-consistency-audit prd:<path> figma-context:<path> source:<path>
```

或：

```text
/spec:app-consistency-audit prd:<path> figma-context:<path> source:<path>
```

`figma-context:<path>` 是本地 materialized JSON evidence；`figma-ref:<id-or-url>` 只是 reference。Figma MCP 是可选能力，不属于 required setup baseline。

合并前可运行：

```text
$spec-code-review
```

或：

```text
/spec:code-review
```

review 的重点是 bug、行为回归、测试缺口和残余风险，不是为改动写表扬稿。

当一个问题被稳定解决，并且经验值得复用时，再运行：

```text
$spec-compound
```

或：

```text
/spec:compound
```

compound 会把可复用经验写入 `docs/solutions/`，供后续会话读取。

## 常见判断

| 情况 | 推荐入口 | 原因 |
| --- | --- | --- |
| 只有一句想法，还没定义成功标准 | `$spec-brainstorm` / `/spec:brainstorm` | 先提升需求输入质量 |
| 需求已清楚，但不知道怎么落地 | `$spec-plan` / `/spec:plan` | 先把工程路径变成可评审计划 |
| 已有 plan 或 task pack | `$spec-work` / `/spec:work` | 直接执行最小可验证改动 |
| 现有实现失败或测试报错 | `$spec-debug` / `/spec:debug` | 先找根因，不要把 bug 当普通 feature 做 |
| 准备合并，想确认风险 | `$spec-code-review` / `/spec:code-review` | 用评审发现阻断问题和残余风险 |
| 移动 App 改动需要 QA 前静态一致性检查 | `$spec-app-consistency-audit` / `/spec:app-consistency-audit` | 先审 PRD、Figma、本地 source、路由、架构、埋点和 i18n 的一致性 |
| 父 workspace 下准备多仓环境 | `$spec-mcp-setup` 后 `$spec-graph-bootstrap` | 默认逐个 child repo 做维护，父目录只保留 advisory summary |

## 关键边界

- `docs/brainstorms/`、`docs/plans/`、`docs/tasks/` 和 `docs/solutions/` 是长期协作文档层。
- `.claude/`、`.codex/` 和 `.agents/skills/` 是 generated runtime assets，不是 source truth。
- `skills/`、`agents/`、`templates/` 和 `src/cli/` 才是 spec-first 自身能力的源码真相源。
- `.spec-first/workspace/` 是父 workspace advisory summary，不是 child repo 的 canonical truth。
- 脚本做确定性检查，LLM 做语义判断；不要把 workflow 设计成多状态强编排系统。

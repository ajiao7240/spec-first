# 最佳实践

本节默认你已经完成 `spec-first doctor` 和 `spec-first init --claude` 或 `spec-first init --codex`。

## Ideate

### 推荐做法

- 先发散候选，再决定是否进入 brainstorm
- 把想改善什么、约束是什么、不能碰什么写清楚
- 保留被否决的想法和原因，方便后续复盘

### 避免做法

- 还没比较选项就直接收敛到单一路径
- 只给出模糊愿望，不写约束
- 把 ideate 直接当成 implementation planning

## Brainstorm

### 推荐做法

- 先收敛边界，再展开方案
- 把关键决策写进需求文档
- 验收标准必须具体、可测试

### 避免做法

- 跳过需求探索直接开工
- 需求描述只停留在一句话
- 没有明确的完成条件

## Plan

### 推荐做法

- 按可独立验证的粒度拆任务
- 先识别风险，再决定技术选型
- 让计划能直接支撑 `work` 阶段执行

### 避免做法

- 计划过于粗糙
- 风险项没有显式记录
- 选型只凭偏好，不看约束

## Work

### 推荐做法

- 小步提交，保持变更可回滚
- 边实现边验证
- 把关键问题和解决方式留在实施记录里

### 避免做法

- 一次性做完所有内容再测试
- 代码完成后才开始补测试
- 过程信息完全不记录

## Review

### 推荐做法

- 基于标准输出 findings
- 明确区分非阻断和阻断问题
- 给出可执行的修复建议

### 避免做法

- 只给主观评价，不给结论
- 评分口径前后不一致
- 忽略关键阻断项

## Compound

### 推荐做法

- 只沉淀可复用、重复出现的模式
- 让 solution 文档结构固定、便于检索
- 把可迁移资产写清楚

### 避免做法

- 什么都沉淀
- 分类混乱
- 没有复用说明

## Stage-0 协作与 Git 合并

### 推荐做法

- 把 `docs/contexts/<slug>/` 当 durable docs 管理，把 `.spec-first/` 当 control plane 管理，两者不要混提
- 只有在架构边界、公开入口、集成关系、测试面或高风险模块变化时，才刷新对应的 Stage-0 文档
- 刷新后先看 diff，再决定提交哪些 durable docs 文件
- 多人协作时优先按文件族分工，例如 `architecture/*`、`pitfalls/*`、`layers/*`
- 同一个 `slug` 的 `README.md`、`00-summary.md` 这类聚合文件尽量只保留一个主修改人
- 合并冲突时先合并代码，再运行 `spec-first crg build --repo=.` 和 bootstrap / graph-bootstrap 重建，最后审查 durable docs 的最终 diff

### 避免做法

- 把 `.spec-first/workflows/bootstrap/<slug>/` 一起提交到 Git
- 每次小改都全量刷新 `docs/contexts/<slug>/`
- 多个人同时改同一个聚合文档
- 合并时手工拼接大段机器生成内容后直接提交
- 不审查 diff 就把 AI 生成文档整批提交

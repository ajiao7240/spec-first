# 最佳实践

本节默认你已经完成 `spec-first doctor` 和 `spec-first init` 并选择目标宿主。

## Ideate

### 推荐做法

- 先发散候选，再决定是否进入 brainstorm
- 把想改善什么、约束是什么、不能碰什么写清楚
- 保留被否决的想法和原因，方便后续复盘

### 避免做法

- 还没比较选项就直接收敛到单一路径
- 只给出模糊愿望，不写约束
- 把 ideate 直接当成 implementation planning

## Brainstorm

### 推荐做法

- 先收敛边界，再展开方案
- 把关键决策写进需求文档
- 验收标准必须具体、可测试

### 避免做法

- 跳过需求探索直接开工
- 需求描述只停留在一句话
- 没有明确的完成条件

## Plan

### 推荐做法

- 按可独立验证的粒度拆任务
- 先识别风险，再决定技术选型
- 让计划能直接支撑 `work` 阶段执行

### 避免做法

- 计划过于粗糙
- 风险项没有显式记录
- 选型只凭偏好，不看约束

## Work

### 推荐做法

- 小步提交，保持变更可回滚
- 边实现边验证
- 把关键问题和解决方式留在实施记录里
- 在父 workspace 中执行写入、测试、review autofix 或 commit 前，确认 plan/task pack 有 `target_repo` 或 per-unit/per-task `target_repo`

### 避免做法

- 一次性做完所有内容再测试
- 代码完成后才开始补测试
- 过程信息完全不记录
- 从父 workspace 直接让 work/review 推断要改哪个 child repo

## Review

### 推荐做法

- 基于标准输出 findings
- 明确区分非阻断和阻断问题
- 给出可执行的修复建议

### 避免做法

- 只给主观评价，不给结论
- 评分口径前后不一致
- 忽略关键阻断项

## App Consistency Audit

### 推荐做法

- 移动 App 改动在模拟器、真机、打包或 QA 前运行 `spec-app-consistency-audit`
- 提供本地 `prd:<path>`、`figma-context:<path>` 和 `source:<path>`，把 `figma-ref:<id-or-url>` 当 reference 而不是 evidence
- 缺 PRD 或 Figma 时披露 degraded coverage，限制结论范围

### 避免做法

- 把缺失 Figma context 直接升级成 confirmed issue
- 把 Figma MCP 当 required setup baseline；它只是可选 materialization 能力
- 把 `.spec-first/app-audit/` 当长期手工维护文档提交

## Compound

### 推荐做法

- 只沉淀可复用、重复出现的模式
- 让 solution 文档结构固定、便于检索
- 把可迁移资产写清楚

### 避免做法

- 什么都沉淀
- 分类混乱
- 没有复用说明

## Runtime 产物与 Git 合并

### 推荐做法

- 把 `.spec-first/audits/`、`.spec-first/workspace/`、`.spec-first/workflows/` 这类 runtime/control-plane 或执行产物当可重建本地证据管理，不提交到 Git
- 架构边界、公开入口、集成关系或测试面变化后，运行 `spec-mcp-setup` 校验 required runtime 和 helper 配置
- 切换分支、pull、rebase、merge 或 dirty worktree 变化后，计划、实现、调试和 review 中的当前事实结论仍要用 direct source reads、测试、ast-grep、git diff 或日志交叉确认
- 在父 workspace 下，`spec-mcp-setup` 无参数默认处理全部 child repos；`--repo <child>` 用于单 child 收窄，`--all-repos` 是显式等价入口
- 只读代码问题可以先用 bounded direct reads / `rg` 定位候选 repos，但最终 repo 相关性由 LLM 根据问题判断
- 普通 plan/work/debug/review/compound context 默认排除 `.spec-first/audits/**` 和 generated mirrors（`.claude/**`、`.codex/**`、`.agents/skills/**`）；只有 setup/update/runtime-drift/audit 任务或用户明确点名路径时才按需读取
- 修改 `skills/`、workflow 入口或 dual-host governance 后，运行 `spec-skill-audit` 查看 source skill、治理投递、runtime drift 与安全信号；把 scorecard 当 review signal，不当 gate
- 需要面向团队沉淀的经验写入 `docs/solutions/`，不要把 runtime/control-plane 产物转成长期手工维护文档
- 合并冲突时先合并代码，再基于 direct source reads、git diff、tests/logs 或聚焦验证判断验证范围

### 避免做法

- 把 `.spec-first/workspace/`、`.spec-first/audits/`、`.spec-first/workflows/` 这类 runtime/control-plane 或执行产物一起提交到 Git
- 把 `.spec-first/audits/**`、`.claude/**`、`.codex/**` 或 `.agents/skills/**` 加入普通上下文扫描、reviewer prompt 或 worker prompt，除非当前任务明确需要这些 runtime artifacts
- 手改 `.claude/`、`.codex/`、`.agents/skills/` 来修 runtime drift；应修改 source truth 后重新 `spec-first init`
- 为了“留上下文”恢复旧文档包或手工维护机器生成索引
- 合并时用过期摘要替代当前代码事实和验证结果
- 在普通 plan/work/debug/review 中隐藏运行 provider repair、默认 hooks、watchers 或 daemons
- 在普通 plan/work/debug/review 中隐藏运行 `group_sync`
- 把 advisory summary 当成新的 canonical source truth，或用 fallback posture 替代可信度判断
- 把 stale/degraded direct source evidence evidence 当成 downstream workflow 失败；正确做法是披露限制、降级 direct source reads，并只把 graph findings 当 advisory pointer
- 仅凭 `graph_evidence` 声明 debug root cause，或仅凭 graph output 提 review finding；root cause/finding 必须有 reproduction/source/log/test/diff/contract 证据确认
- 把 setup projection 的 `source_tags[]` 写成 host config、dependency-ready 或 workspace-advisory 这类 setup-internal diagnostics；这些只能作为 `source_provenance`、fingerprint facts 或 limitations
- 把父 workspace 的 advisory summary 当成所有 child repo 的统一 canonical truth

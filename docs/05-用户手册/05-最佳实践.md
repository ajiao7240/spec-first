# 最佳实践

本节默认你已经完成 `spec-first doctor` 和 `spec-first init --claude` 或 `spec-first init --codex`。

## Brainstorm

### 推荐做法

- 先收敛边界，再展开方案
- 把关键决策写进需求文档
- 验收标准必须具体、可测试

### 避免做法

- 跳过需求探索直接开工
- 需求描述只停留在一句话
- 没有明确的完成条件

## Plan

### 推荐做法

- 按可独立验证的粒度拆任务
- 先识别风险，再决定技术选型
- 让计划能直接支撑 `work` 阶段执行

### 避免做法

- 计划过于粗糙
- 风险项没有显式记录
- 选型只凭偏好，不看约束

## Work

### 推荐做法

- 小步提交，保持变更可回滚
- 边实现边验证
- 把关键问题和解决方式留在实施记录里

### 避免做法

- 一次性做完所有内容再测试
- 代码完成后才开始补测试
- 过程信息完全不记录

## Review

### 推荐做法

- 基于标准输出 findings
- 明确区分非阻断和阻断问题
- 给出可执行的修复建议

### 避免做法

- 只给主观评价，不给结论
- 评分口径前后不一致
- 忽略关键阻断项

## Compound

### 推荐做法

- 只沉淀可复用、重复出现的模式
- 让 solution 文档结构固定、便于检索
- 把可迁移资产写清楚

### 避免做法

- 什么都沉淀
- 分类混乱
- 没有复用说明

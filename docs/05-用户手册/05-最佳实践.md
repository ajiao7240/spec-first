# 最佳实践

本节默认你已经完成 `spec-first doctor` 和 `spec-first init --claude` 或 `spec-first init --codex`。

## Ideate

### 推荐做法

- 先发散候选，再决定是否进入 brainstorm
- 把想改善什么、约束是什么、不能碰什么写清楚
- 保留被否决的想法和原因，方便后续复盘

### 避免做法

- 还没比较选项就直接收敛到单一路径
- 只给出模糊愿望，不写约束
- 把 ideate 直接当成 implementation planning

## Brainstorm

### 推荐做法

- 先收敛边界，再展开方案
- 把关键决策写进需求文档
- 验收标准必须具体、可测试

### 避免做法

- 跳过需求探索直接开工
- 需求描述只停留在一句话
- 没有明确的完成条件

## Plan

### 推荐做法

- 按可独立验证的粒度拆任务
- 先识别风险，再决定技术选型
- 让计划能直接支撑 `work` 阶段执行

### 避免做法

- 计划过于粗糙
- 风险项没有显式记录
- 选型只凭偏好，不看约束

## Work

### 推荐做法

- 小步提交，保持变更可回滚
- 边实现边验证
- 把关键问题和解决方式留在实施记录里

### 避免做法

- 一次性做完所有内容再测试
- 代码完成后才开始补测试
- 过程信息完全不记录

## Review

### 推荐做法

- 基于标准输出 findings
- 明确区分非阻断和阻断问题
- 给出可执行的修复建议

### 避免做法

- 只给主观评价，不给结论
- 评分口径前后不一致
- 忽略关键阻断项

## Compound

### 推荐做法

- 只沉淀可复用、重复出现的模式
- 让 solution 文档结构固定、便于检索
- 把可迁移资产写清楚

### 避免做法

- 什么都沉淀
- 分类混乱
- 没有复用说明

## Graph Readiness 协作与 Git 合并

### 推荐做法

- 把 `.spec-first/graph/` 和 `.spec-first/audits/` 当 runtime/control-plane / execution artifacts 管理，不提交到 Git
- 架构边界、公开入口、集成关系或测试面变化后，先运行 `spec-mcp-setup` 校验 provider 配置，再按需运行 `spec-graph-bootstrap` 刷新 readiness facts
- 修改 `skills/`、workflow 入口或 dual-host governance 后，运行 `spec-skill-audit` 查看 source skill、治理投递、runtime drift 与安全信号；把 scorecard 当 review signal，不当 gate
- 需要面向团队沉淀的经验写入 `docs/solutions/`，不要把 runtime/control-plane 产物转成长期手工维护文档
- 合并冲突时先合并代码，再刷新 graph readiness facts，最后基于 provider facts 或 direct repo reads 判断验证范围

### 避免做法

- 把 `.spec-first/graph/`、`.spec-first/audits/`、`.spec-first/workflows/` 这类 runtime/control-plane 或执行产物一起提交到 Git
- 手改 `.claude/`、`.codex/`、`.agents/skills/` 来修 runtime drift；应修改 source truth 后重新 `spec-first init`
- 为了“留上下文”恢复旧文档包或手工维护机器生成索引
- 合并时用过期摘要替代当前代码事实和 provider readiness 判断

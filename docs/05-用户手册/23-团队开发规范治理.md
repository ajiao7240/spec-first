# 团队开发规范治理

团队开发规范用于告诉 plan/work/review/debug 这类 workflow：当前项目有哪些已确认的工程约束。它解决的是“团队以后改这类代码时必须遵守什么”，不是“某个能力现在对用户表现是什么”。

## Source 边界

当前推荐 source surface：

- `docs/contracts/team-standards.md`：定义 trust、lifecycle、promotion、owner、selection、fallback 和 consumer boundary。
- `docs/standards/index.md`：声明 Surface / Layer / Capability Registry、Owner Registry 和 Rule Index。
- `docs/standards/*.md`：承载 confirmed rule cards。
- `docs/standards/candidates/**`：承载 proposal-only 候选、冲突和 promotion proposals，永远不是 hard context。

不要恢复 `$spec-standards`、`/spec:standards`、`skills/spec-standards/` 或 `.spec-first/standards/`。团队规范治理现在是 source 文档和 standalone skill `spec-team-standards-governance`，不是新的 public workflow。

## 消费规则

workflow 消费团队规范时必须 summary-first：

1. 先读 `docs/contracts/team-standards.md`。
2. 再读 `docs/standards/index.md`。
3. 根据 workflow、changed paths、surface、layer、capability 和 category 选择 rule IDs。
4. 只打开命中的 rule files。
5. 只有 `trust=confirmed,lifecycle_state=active` 且 scope 命中的规则可作为 hard context。

`observed`、`suggested`、`imported`、`conflict`、`confirmed-draft`、deprecated/archived rules 和 `docs/standards/candidates/**` 只能作为 advisory、blocked 或 promotion evidence。

## 与其他文档的区别

| 文档 | 写什么 | 不写什么 |
| --- | --- | --- |
| `docs/specs/<capability>/spec.md` | 当前能力行为真相、业务状态、API/事件/错误/权限语义 | 团队协作流程、代码风格、review 规则 |
| `docs/standards/**` | 工程约束、架构边界、测试策略、review 规则、安全/隐私规则 | 单个能力完整需求、一次需求验收标准 |
| `docs/contracts/**` | workflow、artifact、schema、consumer contract | 业务能力行为和团队偏好正文 |
| `docs/solutions/**` | 已解决问题的复用经验 | confirmed team policy |

## Brownfield 初始化

从显式来源开始：`AGENTS.md`、`CLAUDE.md`、README、contributing docs、architecture docs、lint/test/schema/CI 配置。代码结构、graphify/codegraph、review findings 和 `docs/solutions/**` 只能生成 `observed` 或 `suggested` candidates；它们不是 confirmed policy。

大型项目按单一 slice 初始化：一个 repo、一个 surface、一个 capability 或 sub-domain、明确 include/exclude 范围。不要做全仓总结再把它叫做团队规范。

写入候选前必须经过 secret、PII、本机绝对路径和 prompt-injection hygiene 检查。不通过的内容只能作为 report-only limitation，不能落进 git-tracked candidates。

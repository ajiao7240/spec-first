# 常见问题

## 安装问题

### Q: 如何验证安装成功？

A: "安装成功"和"宿主内 workflow 可见"是两个阶段，需要分别确认。

**阶段一：确认 CLI 安装成功**

安装完成后，第一个命令应该是 `spec-first doctor`：

```bash
spec-first doctor
# 或显式指定平台
spec-first doctor --claude
spec-first doctor --codex
```

如果 `doctor` 提示当前平台的 `commands/spec` 缺失，先执行对应平台的 `spec-first init`。
如果 `doctor` 提示 `.claude/skills`、`.claude/spec-first/workflows`、`.agents/skills`、`.codex/commands/spec` 或 `.codex/agents` 缺失，重新执行对应平台的 `spec-first init`。

```bash
spec-first init --claude
# 或
spec-first init --codex
```

**阶段二：确认宿主内 workflow 可见**

CLI 安装成功不代表宿主（Claude Code / Codex）已经识别 `/spec:*` 入口。必须完全退出并重启宿主后，在宿主内测试：

```bash
# Claude Code
/spec:ideate --help
/spec:brainstorm --help

# Codex
/spec:ideate --help
/spec:brainstorm --help
```

如果宿主能显示帮助信息，说明 workflow 已可见。如果目录文件存在但宿主不识别，说明没有完全重启宿主（macOS 下需要 `Cmd+Q` 退出，不是关闭窗口）。

对于 Stage-0，应确认两个入口都已安装：

- `bootstrap`：默认稳定入口
- `graph-bootstrap`：CRG CLI Tier 1 驱动的 Phase 0-4 全功能入口，两者均正式可用

### Q: 安装后找不到 Claude 或 Codex 的 `/spec:*` 入口？

A:
1. 检查项目内命令目录：
```bash
# Claude Code
ls .claude/commands/spec

# Codex
ls .codex/commands/spec
```

如果你先想确认 Stage-0 双入口是否都已安装，Claude Code 和 Codex 里都应能看到 `bootstrap.md` 和 `graph-bootstrap.md`。

2. 检查项目内技能目录：
```bash
# Claude Code
ls .claude/skills
ls .claude/spec-first/workflows

# Codex
ls .agents/skills

# Codex 代理
find .codex/agents -type f | head
```

其中 Stage-0 相关目录应同时包含：

- Claude Code：`.claude/spec-first/workflows/spec-graph-bootstrap/`
- Codex：`.agents/skills/spec-graph-bootstrap/`

3. 重新执行项目初始化：
```bash
spec-first init --claude
# 或
spec-first init --codex
```

4. 运行环境诊断：
```bash
spec-first doctor
spec-first doctor --claude
spec-first doctor --codex
```

5. 完全退出并重启目标平台（Claude Code 或 Codex）

补充说明：

- `graph-bootstrap` 是 CRG CLI Tier 1 驱动的 Phase 0-4 全功能入口，正式可用
- `compound` 适合更偏知识捕获与复合上下文整理的场景
- 对 Codex 来说，正式可用性以 `.agents/skills/spec-graph-bootstrap/` 的宿主发现结果为准

### Q: 本地安装后 Claude 或 Codex 的 `/spec:*` 不生效？

A:
1. 确认使用的是 npm CLI 模型，而不是旧的 plugin 安装模型

2. 重新执行：
```bash
spec-first doctor
spec-first doctor --claude
spec-first doctor --codex
spec-first init --claude
# 或
spec-first init --codex
```

3. 检查当前平台的 `.claude/commands/spec` / `.claude/skills` / `.claude/spec-first/workflows` 或 `.codex/commands/spec` / `.agents/skills` 是否存在且包含对应目录

4. 完全退出目标平台（不是关闭窗口）

5. 重新启动 Claude Code 或 Codex

### Q: `spec-first init` 为什么没有提示我输入用户名？

A: 因为这个命令默认会做身份回退推断，不会强制交互询问。优先级是：

1. 显式传入的 `-u/--user`
2. 全局 `~/.spec-first/.developer`
3. `git config user.name`

如果这三层都拿不到用户名，命令才会报错。  
如果你想明确指定项目身份，可以直接执行：

```bash
spec-first init --claude -u <name> --lang <zh|en>
# 或
spec-first init --codex -u <name> --lang <zh|en>
```

### Q: `npm install -g spec-first` 后为什么没看到欢迎页？

A: `postinstall` 脚本会输出欢迎信息，但 npm 默认不保证展示所有 lifecycle 输出，因此 `postinstall` 不是稳定的欢迎页入口。

**稳定可见的版本确认方式**是运行：

```bash
spec-first -v
```

如果你想观察安装期脚本输出，可以使用：

```bash
npm install -g spec-first --foreground-scripts
```

### Q: 安装时出现大量 `npm warn ERESOLVE overriding peer dependency` 怎么办？

A: 这是已知的 tree-sitter 兼容性噪音，本版本的治理目标是逐步消除此类警告。

**原因**：CRG 模块依赖 `tree-sitter@~0.21.0` 作为解析器运行时，而各语言 grammar 包（tree-sitter-javascript、tree-sitter-cpp 等）的 `peerDependency` 元数据要求更高版本（`^0.22.1` 以上）。npm 会检测到版本不匹配并输出警告，但最终以 overriding 方式完成安装。

**验证安装是否真正成功**：

```bash
spec-first -v
spec-first doctor
spec-first crg stats --repo=.    # 若已有图谱，应返回 JSON
```

如果以上命令正常返回，说明安装成功，peer dep 警告不影响运行时正确性。全套 Jest 单元测试和 E2E 测试在此版本组合下均通过。

如果以上命令无法正常返回，说明安装存在问题，请按以下步骤排查：

1. 清除 npm 缓存后重装：`npm cache clean --force && npm install -g spec-first`
2. 使用 `--legacy-peer-deps` 安装：`npm install -g spec-first --legacy-peer-deps`
3. 确认 Node.js 版本 >= 20：`node --version`

## 使用问题

### Q: 如何查看生成的入口？

A: 按平台查看：
```bash
# Claude Code
ls .claude/commands/spec

# Codex
ls .agents/skills | rg '^spec-'
```

如果你关注 Stage-0，并行期至少应看到：

- command: `graph-bootstrap.md`
- skill: `spec-graph-bootstrap`

### Q: 如何重新生成命令？

A: 重新运行初始化，命令会默认覆盖现有文件：
```bash
spec-first init --claude
# 或
spec-first init --codex
```

如果你只想检查某个平台是否完整，可以使用：

```bash
spec-first doctor --claude
spec-first doctor --codex
```

### Q: `doctor` 提示 `legacy managed state detected` 怎么办？

A: 这表示当前项目里存在旧版 `state.json` 形状，当前版本不会继续做字段兼容迁移。正确处理方式是直接重新运行对应平台的 `init`：

```bash
spec-first init --claude
# 或
spec-first init --codex
```

当前版本的升级策略是：

- `init` 是唯一支持的 legacy 升级入口
- `init` 会先执行 managed hard reset，再按当前版本全量重建运行时
- 不需要先手工删除 `.claude/`、`.codex/` 或 `.agents/skills/` 下的 spec-first 目录

### Q: 如何清理 spec-first 生成的项目内资产？

A: 使用：
```bash
spec-first clean --claude
# 或
spec-first clean --codex
```
这个命令会删除 spec-first 受管的 Claude Code 或 Codex 资产，同时保留未受管的自定义技能或代理。

补充说明：

- `clean` 只负责删除当前受管集合
- 如果当前项目属于 legacy managed state，`clean` 会拒绝迁移并提示你先执行 `spec-first init`
- 也就是说，legacy 升级入口是 `init`，不是 `clean`

### Q: Review 不通过怎么办？

A:
1. 查看 `/spec:review` 输出中的阻断问题和文件引用
2. 修复代码
3. 重新运行 /spec:work
4. 再次 /spec:review

### Q: 多个任务如何管理？

A: 当前发布版主链路以 `docs/brainstorms/`、`docs/plans/`、`docs/solutions/` 这些稳定产物为主；如果仓库自己额外维护 `.claude/tasks/`，那属于仓库内部流程，不是 npm 用户的标准路径。

### Q: 执行过程中，如何查看某个 skill 的调用链路？

A: 当前没有一个专门的 runtime trace viewer 去把每次 workflow 执行自动展开成可视化调用树。最可靠的办法是按下面顺序查看静态链路：

1. 先看 command wrapper
2. 再看对应 workflow skill 的 `SKILL.md`
3. 再追到其中引用的 agents、references、assets

以 `spec:plan` 为例，入口 wrapper 会先要求宿主读取 `.claude/spec-first/workflows/spec-plan/SKILL.md`，再按 skill contract 继续执行。因此 command wrapper 负责“入口跳转”，真正的链路定义主要写在 `SKILL.md` 里。

建议优先查看这些位置：

- `templates/claude/commands/spec/*.md`
- `skills/spec-*/SKILL.md`
- `agents/**` 或 `docs/10-prompt/agents/**`
- `docs/02-架构设计/全局分析/2026-04-16-spec-first-体系完整性与skill执行链路深度审查.md`

如果你想在执行时快速看某个 skill 实际会 dispatch 哪些子 agent，可以直接搜索：

```bash
rg -n "Task spec-first:|Read |always-on|parallel|references/" skills/spec-plan/SKILL.md
rg -n "Task spec-first:|Read |always-on|parallel|references/" skills/spec-brainstorm/SKILL.md
rg -n "Task spec-first:|Read |always-on|parallel|references/" skills/spec-review/SKILL.md
rg -n "Task spec-first:|Read |always-on|parallel|references/" skills/spec-ideate/SKILL.md
```

这会直接显示：

- 会调用哪些 research / review agents
- 哪些是 always-on
- 哪些是 conditional
- 还会继续读哪些 `references/` 或 support files

如果还要继续往下追，就对被调用的 agent 文件再做一次同类搜索。例如 `spec-plan`、`spec-brainstorm`、`spec-review`、`spec-ideate` 中和 `docs/solutions/` 相关的链路，最终都会落到 `learnings-researcher`。

补充说明：

- `spec-first doctor` 只做运行时健康检查，不展示 workflow 调用链
- `omm scan` / `omm view` 更偏代码结构与架构浏览，不是 skill orchestration trace viewer
- 当前最稳定的查看方式仍然是 `command wrapper -> SKILL.md -> agent/references`

### Q: `docs/contexts/<slug>/` 是 AI 生成的，为什么还能进 Git？如何保证可合并？

A: 可以进 Git，但前提是你把它当成 **durable docs**，而不是当成运行时缓存。

先分清两类产物：

- `docs/contexts/<slug>/`：长期上下文资产，可审查、可协作、可提交
- `.spec-first/workflows/bootstrap/<slug>/`：control plane，只供本地或 CI 运行时消费，不提交

是否容易 Git merge，关键不在于“是不是 AI 生成”，而在于下面这些约束是否成立：

1. durable docs 和 control plane 已分层
2. 文档按文件族拆分，而不是所有内容塞进一个大文件
3. 刷新时尽量最小改写，不反复重写无变化文件
4. 提交前人工审查 diff，确认不是噪音和错误事实
5. 冲突时优先重跑生成链，而不是手工硬拼机器生成大段内容

推荐处理流程：

```bash
# 1. 先合并代码
git merge <branch>

# 2. 如果你走 graph-bootstrap，先重建图谱事实
spec-first crg build --repo=.
```

第 3 步：重跑 Stage-0

- 走图谱入口：`/spec:graph-bootstrap` 或 `$spec-graph-bootstrap`
- 走知识捕获入口：`/spec:compound` 或 `$spec-compound`

然后只做两件事：

1. 审查 `docs/contexts/<slug>/` 的最终 diff
2. 只提交 durable docs，不提交 `.spec-first/`

如果团队经常多人并发刷新同一个 `slug`，应额外遵守两条纪律：

1. 同一个 durable doc 文件只设一个主作者
2. 优先按文件分工，不要多人同时改 `README.md` 或 `00-summary.md`

## 仓库维护问题

以下问题只适用于你正在维护这个仓库源码，而不是普通 npm 用户。

### Q: 如何跳过某个阶段？

A: 不建议跳过。如果你是在仓库源码里调试任务流程，可以手动操作：
```bash
# 手动创建产物文件
touch .claude/tasks/<task-id>/0X-stage.md

# 更新状态
./scripts/task-manager.sh update --task-id <id> --stage <next-stage>
```

### Q: 如何自定义 Review 标准？

A: 如果你在仓库源码里调试本地流程，可以修改 `scripts/review-judge.sh` 中的判定规则；日常 npm 用户通常不需要改这个脚本。

### Q: 如何扩展新的 skill？

A: 如果你在仓库源码里扩展能力，可以参考现有 `skills/` 创建新的 `skill.md`；日常 npm 用户通常只需要使用现有工作流入口。

### Q: 如何集成到 CI/CD？

A: 在 CI 中运行测试：
```bash
npm test
```

# 常见问题

## 安装问题

### Q: 如何验证安装成功？

A: "安装成功"和"宿主内 workflow 可见"是两个阶段，需要分别确认。

**阶段一：确认 CLI 安装成功**

安装完成后，第一个命令应该是 `spec-first doctor`：

```bash
spec-first doctor
# 或显式指定平台
spec-first doctor --claude
spec-first doctor --codex
```

如果 `doctor` 提示当前平台的 `commands/spec` 缺失，先执行 `spec-first init` 并选择目标宿主。
如果 `doctor` 提示 `.claude/skills`、`.claude/spec-first/workflows`、`.agents/skills` 或 `.codex/agents` 缺失，重新执行 `spec-first init` 并选择目标宿主。

```bash
spec-first init
# 按引导选择目标宿主
```

**阶段二：确认宿主内 workflow 可见**

CLI 安装成功不代表宿主（Claude Code / Codex）已经识别对应入口。必须完全退出并重启宿主后，在宿主内测试：

```text
# Claude Code
/spec:ideate --help
/spec:brainstorm --help

# Codex
$spec-ideate --help
$spec-brainstorm --help
```

如果宿主能显示帮助信息，说明 workflow 已可见。如果目录文件存在但宿主不识别，说明没有完全重启宿主（macOS 下需要 `Cmd+Q` 退出，不是关闭窗口）。

如果你要确认 setup、审计和知识沉淀入口，应检查：

- `mcp-setup`：required runtime、MCP servers 和 helper tools 的安装/验证入口
- `skill-audit`：source skill 质量、治理投递、runtime drift 与安全信号审计入口
- `compound`：工作完成后的稳定知识捕获入口
- `tasks`：task pack hash / validate 的确定性工具入口

### Q: `doctor --json` 里的状态字段应该怎么理解？

A: `doctor --json` 是给脚本、CI 和维护者读取的 machine-readable 运行面报告。它检查的是 CLI 安装、managed runtime assets、host readiness 和 workflow verification evidence，不是 MCP/provider/graph 的完整健康检查。

常见字段含义：

- `install_health`：Node、Git、package 级 CLI 运行检查，回答“`spec-first` 这个命令能不能正常工作”。
- `runtime_asset_health`：`spec-first init` 生成的受管 runtime assets 是否存在、匹配 source truth、没有 drift。
- `host_readiness`：Claude Code / Codex CLI、宿主配置、项目级 developer profile 等宿主侧 wiring 是否可用。
- `decision_input_health`：当前仍是 `not_checked`；不要把它理解成当前源码证据已经可用或不可用。下游 workflow 应直接读取当前 source、git diff、tests/logs 或用户提供证据。
- `workflow_runnability`：对“宿主 workflow surface 是否可运行”的汇总，取值是 `verified`、`simulated` 或 `not_verified`。
- `workflow_runnability_basis`：解释 runnability 的事实依据，包括 runtime surface、managed state、verification evidence 路径、新鲜度和 `fallback_reason`。

`workflow_runnability` 的三个状态：

| 状态 | 含义 | 下一步 |
|---|---|---|
| `verified` | runtime surface 已就绪，并且存在 schema 有效且足够新的 verification evidence | 可以进入下游 workflow；仍需按具体任务运行测试和 review |
| `simulated` | runtime surface 已就绪，但 verification evidence 缺失、过期或不完整 | 先按当前任务运行必要 workflow/验证；如果是 setup 场景，继续 `spec-mcp-setup` |
| `not_verified` | runtime assets、managed state 或 workflow surface 仍不完整 | 先运行对应的 `spec-first init`，重启宿主后再检查 |

常见 `fallback_reason`：

- `verification_evidence_missing`：尚未记录 workflow verification evidence。
- `verification_evidence_schema_invalid`：verification evidence 文件存在但不符合 schema。
- `verification_gates_unresolved`：evidence 没有可识别的 effective gate。
- `verification_evidence_not_relevant`：evidence 不满足当前有效 gate。
- `verification_evidence_stale`：evidence 存在但超过当前可信时间窗口，需要重新验证。
- `verification_evidence_freshness_unknown`：evidence 缺少可解析的 `captured_at` 时间，无法判断新鲜度。

如果目标是准备 MCP/helper runtime，请运行当前宿主内的 `spec-mcp-setup`。`doctor` 不会替代 setup workflow，也不会把临时探测结果写成当前源码证据真相源。

### Q: 父级 workspace 显示 broken worktree 或只覆盖了部分子仓怎么办？

A: 先区分两个问题：Git metadata 是否健康，以及当前 workspace 里哪些目录会被自动覆盖。

`spec-mcp-setup` 的 target resolver 会输出 `project-target.v2` 诊断。关键字段是：

- `git_health.status="broken-worktree"`：当前目录的 `.git` 是 worktree pointer 文件，但指向的真实 gitdir 已不存在。
- `git_health.status="corrupted-gitdir"`：当前 `.git` 存在但损坏或无法被 Git 识别。
- `coverage_gap.uncovered_top_level_dirs`：父级 workspace 里没有被 child Git repo discovery 覆盖的顶级目录数量。
- `candidates_diagnostics[]`：有 broken/corrupted `.git` 的 child 目录，不会进入 child repo 列表。

如果是 broken worktree，先运行 dry-run 诊断：

```bash
spec-first repair-worktree --dry-run
```

这个命令只打印 `.git` pointer 摘要、unlink preview 和手动修复建议，不会删除 `.git`。`--apply` / `--unlink` 当前会拒绝执行。

如果是 corrupted gitdir，按提示运行 Git 自身诊断，例如：

```bash
git fsck
```

如果 `coverage_gap` 显示父级还有很多非 Git 目录未覆盖，这是预期的 advisory：默认 `--all-repos` 只处理独立 child Git repos，不会自动把父级普通文件夹纳入索引。需要索引这类非 Git 普通文件夹时，向 `spec-mcp-setup` 显式传 `--folder <path>`；自动 mixed topology 覆盖仍是后续 contract/provider gate，不是当前默认行为。

### Q: 安装后找不到 Claude 的 `/spec:*` 或 Codex 的 `$spec-*` 入口？

A:
1. 检查项目内命令目录：
```bash
# Claude Code
ls .claude/commands/spec
```

如果你先想确认 setup 入口是否已安装，Claude Code 里应能看到 `mcp-setup.md`，并且 Codex skill discovery 能看到 `spec-mcp-setup`。

2. 检查项目内技能目录：
```bash
# Claude Code
ls .claude/skills
ls .claude/spec-first/workflows

# Codex
ls .agents/skills

# Codex 代理
find .codex/agents -type f | head
```

其中 setup workflow 相关目录应包含：

- Claude Code：`.claude/spec-first/workflows/spec-mcp-setup/`
- Codex：`.agents/skills/spec-mcp-setup/`

skill audit workflow 相关目录应包含：

- Claude Code：`.claude/commands/spec/skill-audit.md`
- Claude Code：`.claude/spec-first/workflows/spec-skill-audit/`
- Codex：`.agents/skills/spec-skill-audit/`

3. 重新执行项目初始化：
```bash
spec-first init
# 按引导选择目标宿主
```

4. 运行环境诊断：
```bash
spec-first doctor
spec-first doctor --claude
spec-first doctor --codex
```

5. 完全退出并重启目标平台（Claude Code 或 Codex）

补充说明：

- `mcp-setup` 是 required runtime 与 helper setup 入口，正式可用
- `skill-audit` 是 source skill 审计入口，审计产物写入 `.spec-first/audits/skill-audit/`
- `compound` 适合更偏知识捕获与复合上下文整理的场景
- 对 Codex 来说，正式可用性以 `.agents/skills/spec-mcp-setup/` 的宿主发现结果为准

### Q: 如何运行 skill audit？产物要不要提交？

A: 如果你在维护 spec-first 自身、修改 `skills/`、新增 workflow，或者想检查 skill trigger / boundary / governance / runtime drift，可以运行 `spec-skill-audit`。

宿主入口：

```text
# Claude Code
/spec:skill-audit
/spec:skill-audit skills/spec-debug
/spec:skill-audit --runtime --patch-preview

# Codex
$spec-skill-audit
$spec-skill-audit skills/spec-debug
$spec-skill-audit --runtime --patch-preview
```

直接脚本入口：

```bash
node skills/spec-skill-audit/scripts/write-audit-artifacts.js --repo .
node skills/spec-skill-audit/scripts/write-audit-artifacts.js --repo . --target skills/spec-debug
node skills/spec-skill-audit/scripts/write-audit-artifacts.js --repo . --runtime --patch-preview
```

结果目录：

```text
.spec-first/audits/skill-audit/latest/
```

优先读取：

- `skill-audit-summary.md`
- `skill-improvement-plan.md`
- `skill-audit-report.json`
- `expert-scorecard.json`
- `security-risk-report.json`
- `promise-implementation-report.json`
- `governance-drift-report.json`
- `runtime-drift-report.json`

`.spec-first/audits/` 是 gitignored 执行产物，不是 source truth，不提交。`expert-scorecard.json` 是信号，不是 gate；P0/P1 finding 需要结合 signal、evidence、counter-evidence、decision、reason、recommendation、confidence 由 LLM 判断是否修复。`promise-implementation-report.json` 只指出文档承诺和脚本事实是否一致，不替 LLM 下最终结论。runtime drift 只能通过 `spec-first init` 并选择目标宿主修复，不手改 `.claude/`、`.codex/`、`.agents/skills/`。

### Q: 本地安装后 Claude 的 `/spec:*` 或 Codex 的 `$spec-*` 不生效？

A:
1. 确认使用的是 npm CLI 模型，而不是旧的 plugin 安装模型

2. 重新执行：
```bash
spec-first doctor
spec-first doctor --claude
spec-first doctor --codex
spec-first init
# 按引导选择目标宿主
```

3. 检查当前平台的 `.claude/commands/spec` / `.claude/skills` / `.claude/spec-first/workflows` 或 `.agents/skills` / `.codex/agents` 是否存在且包含对应目录

4. 完全退出目标平台（不是关闭窗口）

5. 重新启动 Claude Code 或 Codex

### Q: `spec-first init` 如何确定用户名？

A: `spec-first init` 会在交互式引导中展示用户名输入，默认值按以下优先级回退：

1. 当前 host 的项目级 `.developer`
2. 全局 `~/.spec-first/.developer`
3. `git config user.name`

直接输入新值即可覆盖默认值。如果这些来源都拿不到用户名且你也未输入，命令会要求补全。

```bash
spec-first init
```

### Q: `npm install -g spec-first` 后为什么没看到欢迎页？

A: 新版本不在 `postinstall` 阶段展示欢迎页，以避免 npm install-script 审计噪音；安装完成后的稳定欢迎页入口是 CLI 本身。

**稳定可见的版本确认方式**是运行：

```bash
spec-first -v
```

如果你想观察安装期脚本输出，可以使用：

```bash
npm install -g spec-first --foreground-scripts
```

### Q: 安装时出现 `npm warn ERESOLVE overriding peer dependency` 怎么办？

A: 当前版本已移除旧内置图运行时的 native parser 依赖，正常安装不应再依赖这些旧解析器组合。如果你仍看到 peer dependency 警告，先按实际包名判断来源，通常是本机 npm 缓存、全局依赖或周边工具链引起。

**验证 spec-first package CLI 是否真正成功**：

```bash
spec-first -v
spec-first doctor
```

如果以上命令正常返回，说明 package CLI 安装成功。provider / helper readiness 由宿主内的 `spec-mcp-setup` workflow 继续验证。

如果以上命令无法正常返回，说明安装存在问题，请按以下步骤排查：

1. 清除 npm 缓存后重装：`npm cache clean --force && npm install -g spec-first`
2. 使用 `--legacy-peer-deps` 安装：`npm install -g spec-first --legacy-peer-deps`
3. 确认 Node.js 版本 >= 20：`node --version`

## 跨平台与 Windows 排障

### Q: Win64 原生环境下安装或运行失败，应该先检查什么？

A: 先在原生 Windows shell 中确认 CLI、Node、npm 和 Git 是否都来自预期路径。Git Bash、MSYS2 和 WSL 可以用于 POSIX 工具，但不能替代 PowerShell 或 `cmd.exe` 的原生验证，因为 npm 会生成 `.cmd` shim，`PATH`、quoting、code page 和路径规则也不同。

PowerShell 7+ / Windows PowerShell 5.1：

```powershell
node --version
npm --version
git --version
Get-Command spec-first
spec-first doctor
```

`cmd.exe`：

```bat
node --version
npm --version
git --version
where spec-first
spec-first doctor
```

如果找不到 `spec-first`，先看全局 prefix：

```powershell
npm config get prefix
npm prefix -g
```

常见修复是把 npm global prefix 对应的 shim 目录加入用户 `PATH`。Windows 上通常是 `%APPDATA%\npm`；如果你自定义了 prefix，以 `npm prefix -g` 输出为准。修改 `PATH` 后请新开终端，不要只在旧窗口里重试。

### Q: PowerShell 提示 execution policy、编码或中文路径问题怎么办？

A: `spec-first` 的 package CLI 入口优先走 npm shim，不需要直接执行 `.ps1`。如果你在维护仓库源码并需要运行 PowerShell helper 脚本，请使用显式参数，避免 profile 和 execution policy 干扰：

```powershell
pwsh -NoProfile -ExecutionPolicy Bypass -File .\skills\spec-mcp-setup\scripts\check-health.ps1
```

Windows PowerShell 5.1 可用，但默认编码和 native argument passing 与 PowerShell 7+ 不完全一致。遇到中文路径、emoji、box drawing 字符显示异常时，优先用 Windows Terminal + PowerShell 7+ 复验；如果必须在 `cmd.exe` 中验证，可以先执行 `chcp 65001`，但最终判断以 exit code、`doctor --json` 和 artifacts 为准，不以终端视觉字符为准。

### Q: `npx` 或 provider 安装失败，应该清什么缓存？

A: 先区分 package CLI 和 provider/helper 工具。`spec-first -v` 与 `spec-first doctor` 验证的是 package CLI；`spec-mcp-setup` 和 `spec-mcp-setup` 才负责 provider readiness。

可以按顺序排查：

```bash
npm cache verify
npm cache clean --force
npx --yes --version
spec-first doctor --json
```

如果是 provider 安装或探测失败，重新运行当前宿主里的 setup workflow，再查看本地 artifacts：

```text
/spec:mcp-setup

$spec-mcp-setup
```

重点检查 `.spec-first/config/` 和 `.spec-first/workspace/` 下的 status / summary。Local evidence 可能包含绝对路径；不要未经筛选上传到公开 issue。

### Q: 安装时看到 native dependency build、`better-sqlite3` 或 parser 相关错误怎么办？

A: 当前 `spec-first` package 已经退役旧内置图运行时的 `better-sqlite3`、tree-sitter parser 等 native dependency。正常 `npm install -g spec-first` 不应再编译这些 native modules。

如果错误来自 `spec-first` 自身，请先确认安装的是当前包：

```bash
npm view spec-first version
spec-first -v
npm ls -g spec-first --depth=0
```

如果错误来自 agent-browser 或其它 provider/helper 工具，它属于 provider readiness 问题，不代表 package CLI 安装失败。按 `spec-mcp-setup` 的报告处理，并保留 provider raw log 作为证据。

### Q: `agent-browser install` 下载 Chrome 超时，但本机已有 Chrome 怎么办？

A: 不要手动伪造 `~/.agent-browser/spec-first-install.json`。这个 marker 只表示 `agent-browser install` 成功，不是系统 Chrome 可用性的通用证明。`agent-browser` 现在是 non-blocking browser automation helper capability；默认 setup 会跳过它，不阻断 `baseline_ready`。只有需要浏览器截图、浏览器测试或 UI polish 自动化时，设置 `SPEC_FIRST_BROWSER_HELPER_REQUIRED=1` 后重跑 `spec-mcp-setup`，或设置 `AGENT_BROWSER_EXECUTABLE_PATH` 指向现有 Chrome/Chromium 可执行文件再修复。

### Q: Windows long path、特殊字符路径或文件锁导致失败怎么办？

A: 推荐先在路径含空格、中文、方括号的普通目录中验证：

```text
C:\tmp\spec first smoke
C:\tmp\spec-first-中文路径
C:\tmp\spec-first-[brackets]
```

如果项目路径非常深，Windows 可能触发 legacy `MAX_PATH` 限制。可以启用系统 long paths，并让 Git 支持长路径：

```powershell
git config --global core.longpaths true
```

同时避免把 repo 放在同步盘、杀毒软件实时锁定目录或非常深的嵌套目录中。Windows 文件锁会让删除、重命名和 clean 操作失败；关闭占用文件的编辑器、终端、索引器或 provider 进程后重试。

### Q: tmux、CI、无 TTY 或重定向环境里会卡住吗？

A: package CLI 不要求 tmux，也不把 tmux 当作 readiness 条件。CI、管道、重定向或 tmux detached session 中建议设置：

```bash
CI=true NO_COLOR=1 spec-first doctor
```

如果是在 PowerShell 中：

```powershell
$env:CI = "true"
$env:NO_COLOR = "1"
spec-first doctor
```

如果输出被重定向，优先读取 exit code、`doctor --json` 和 `.spec-first/*` artifacts。不要把颜色、emoji 或终端布局当作唯一成功标准。

## 使用问题

### Q: 如何查看生成的入口？

A: 按平台查看：
```bash
# Claude Code
ls .claude/commands/spec

# Codex
ls .agents/skills | rg '^spec-'
```

如果你关注 setup 事实入口，至少应看到：

- command: `mcp-setup.md`
- skill: `spec-mcp-setup`

### Q: 如何重新生成命令？

A: 重新运行初始化，命令会默认覆盖现有文件：
```bash
spec-first init
# 按引导选择目标宿主
```

如果你只想检查某个平台是否完整，可以使用：

```bash
spec-first doctor --claude
spec-first doctor --codex
```

### Q: `doctor` 提示 `legacy managed state detected` 怎么办？

A: 这表示当前项目里存在旧版 `state.json` 形状，当前版本不会继续做字段兼容迁移。正确处理方式是直接重新运行 `spec-first init` 并选择目标宿主：

```bash
spec-first init
# 按引导选择目标宿主
```

当前版本的升级策略是：

- `init` 是唯一支持的 legacy 升级入口
- `init` 会先执行 managed hard reset，再按当前版本全量重建运行时
- 不需要先手工删除 `.claude/`、`.codex/` 或 `.agents/skills/` 下的 spec-first 目录

### Q: 如何清理 spec-first 生成的项目内资产？

A: 使用：
```bash
spec-first clean --claude
# 或
spec-first clean --codex
```
这个命令会删除 spec-first 受管的 Claude Code 或 Codex 资产，同时保留未受管的自定义技能或代理。

补充说明：

- `clean` 只负责删除当前受管集合
- 如果当前项目属于 legacy managed state，`clean` 会拒绝迁移并提示你先执行 `spec-first init`
- 也就是说，legacy 升级入口是 `init`，不是 `clean`

### Q: Codex 启动时出现两段完全相同的 SessionStart 注入怎么办？

A: 这是**全局 hook 污染**导致的双注入。

根因：你曾在某个目录运行过 `spec-first init`，而那个目录的 `.codex` 恰好是 Codex 实际读取 hook 的全局位置（`CODEX_HOME`，默认 `~/.codex`）。结果是全局层 hook 对所有项目生效，与每个项目自己的 SessionStart hook 同时触发，于是注入两遍。

排查与清理：

```bash
# 1. 确认是否存在全局污染
spec-first doctor --codex

# 2A. 在 CODEX_HOME 目录（默认 ~/.codex 的父目录，即 ~）运行 clean
spec-first clean --codex

# 2B. 若 clean 提示 “No spec-first managed project assets found”（旧版本写入、无受管 state），
#     手动删除 <CODEX_HOME>/hooks.json 里的 spec-first SessionStart entry；
#     若该文件没有其它 hook，可整文件删除。默认路径为 ~/.codex/hooks.json
```

说明：

- 新版本的 `spec-first init` 已会**自动跳过**在 `CODEX_HOME` 目录写入 SessionStart hook（仍正常安装 skills/agents/AGENTS.md），并在正常项目 init 收尾时检测并提示已存在的全局污染，所以这是历史遗留问题的一次性清理。
- 清理只针对 `CODEX_HOME/hooks.json` 里的 SessionStart entry。`~/.codex/config.toml` 里可能残留 `[hooks.state...]` / `[projects."~"]` 痕迹，它们不影响双注入，可忽略。
- 该问题为 Codex 特有：Codex 会叠加执行全局层与项目层 hook；Claude Code 对 user/project 设置中相同的 hook 命令做去重，不会因此双注入。

### Q: Review 不通过怎么办？

A:
1. 查看 `/spec:code-review` 输出中的阻断问题和文件引用
2. 修复代码
3. 重新运行 /spec:work
4. 再次 /spec:code-review

### Q: 多个任务如何管理？

A: 当前发布版主链路以 `docs/brainstorms/`、`docs/plans/`、`docs/tasks/`、`docs/solutions/` 这些稳定产物为主。计划较大、涉及多个模块或需要并行/交接时，使用 standalone `write-tasks` skill 从 plan 派生 task pack；它依赖 source plan 和 hash freshness，不是新的需求真相源。

### Q: 执行过程中，如何查看某个 skill 的调用链路？

A: 当前没有一个专门的 runtime trace viewer 去把每次 workflow 执行自动展开成可视化调用树。最可靠的办法是按下面顺序查看静态链路：

1. 先看 command wrapper
2. 再看对应 workflow skill 的 `SKILL.md`
3. 再追到其中引用的 agents、references、assets

以 `spec:plan` 为例，入口 wrapper 会先要求宿主读取 `.claude/spec-first/workflows/spec-plan/SKILL.md`，再按 skill contract 继续执行。因此 command wrapper 负责“入口跳转”，真正的链路定义主要写在 `SKILL.md` 里。

建议优先查看这些位置：

- `templates/claude/commands/spec/*.md`
- `skills/spec-*/SKILL.md`
- `agents/**` 或 `docs/10-prompt/agents/**`
- `docs/02-架构设计/全局分析/2026-04-16-spec-first-体系完整性与skill执行链路深度审查.md`

如果你想在执行时快速看某个 skill 实际会 dispatch 哪些子 agent，可以直接搜索：

```bash
rg -n "Task spec-first:|Read |always-on|parallel|references/" skills/spec-plan/SKILL.md
rg -n "Task spec-first:|Read |always-on|parallel|references/" skills/spec-brainstorm/SKILL.md
rg -n "Task spec-first:|Read |always-on|parallel|references/" skills/spec-code-review/SKILL.md
rg -n "Task spec-first:|Read |always-on|parallel|references/" skills/spec-ideate/SKILL.md
```

这会直接显示：

- 会调用哪些 research / review agents
- 哪些是 always-on
- 哪些是 conditional
- 还会继续读哪些 `references/` 或 support files

如果还要继续往下追，就对被调用的 agent 文件再做一次同类搜索。例如 `spec-plan`、`spec-brainstorm`、`spec-code-review`、`spec-ideate` 中和 `docs/solutions/` 相关的链路，最终都会落到 `learnings-researcher`。

补充说明：

- `spec-first doctor` 只做运行时健康检查，不展示 workflow 调用链
- `omm scan` / `omm view` 更偏代码结构与架构浏览，不是 skill orchestration trace viewer
- 当前最稳定的查看方式仍然是 `command wrapper -> SKILL.md -> agent/references`

### Q: runtime 产物要不要进 Git？如何保证可合并？

A: 不要把本地 runtime/control-plane 产物或 skill audit 执行产物提交到 Git。当前 `spec-mcp-setup` 默认写 setup-owned facts；`spec-skill-audit` 默认产物是 `.spec-first/audits/skill-audit/` 下的审计报告。它们可以本地或 CI 重建。

先分清两类内容：

- `.spec-first/config/`、`.spec-first/workspace/`：runtime setup facts / advisory summaries，不提交
- `.spec-first/audits/`：skill audit 执行产物，不提交
- `docs/solutions/`、`docs/plans/`：长期协作文档，可审查、可提交

是否容易 Git merge，关键在于下面这些约束是否成立：

1. runtime/control-plane 不进入 Git
2. 需要长期沉淀的知识由 `spec-compound` 写入 `docs/solutions/`
3. required runtime 或 helper 配置变化后重新运行 `spec-mcp-setup`
4. 评审或合并前直接读取当前 source、git diff、tests/logs；证据不足时披露限制并回退 bounded direct repo reads

推荐处理流程：

```text
# 1. 先合并代码
git merge <branch>

# 2. 在宿主内刷新 required runtime
/spec:mcp-setup        # Claude Code

# 或 Codex
$spec-mcp-setup
```

如果本次合并产生了可复用经验，再运行 `/spec:compound` 或 `$spec-compound` 写入长期知识；提交时只提交源码和 durable docs，不提交 `.spec-first/`。

### Q: 父 workspace 下有多个 child Git repos，setup 和 mcp-setup 怎么运行？

A: 当前父 workspace 是 advisory 层，不拥有 child repo 的 repo-local truth。

从父 workspace 无参数运行：

```text
/spec:mcp-setup

$spec-mcp-setup
```

默认会走 all-child-repos maintenance 路径：逐个 child repo 写各自的 `.spec-first/config/*`，父目录只写 `.spec-first/workspace/*summary.json`。

如果只想处理一个 child repo，使用脚本层 `--repo <child>`：

```bash
bash skills/spec-mcp-setup/scripts/install-mcp.sh --repo project-a
bash skills/spec-mcp-setup/scripts/verify-tools.sh --repo project-a
bash skills/spec-mcp-setup/scripts/bootstrap-providers.sh --repo project-a
```

`--all-repos` 是显式等价入口，不改变边界。后续 plan、work、review 只要会写文件、跑测试、改 changelog、autofix 或 commit，都必须有明确 `target_repo` 或 per-unit/per-task `target_repo`。

### Q: Figma MCP 要不要放进 mcp-setup required baseline？

A: 不要。Figma MCP 是 App consistency audit 的可选 materialization 能力，不属于 required harness runtime baseline。

正确边界是：

- `figma-context:<path>`：本地 materialized JSON，能作为 evidence。
- `figma-ref:<id-or-url>`：只是 reference，不能自动当作 evidence。
- 默认交互模式下，如果宿主已有 Figma MCP 能力，可以 materialize 本地 context。
- `mode:headless` 和 `mode:report-only` 不应远程 fetch Figma；缺 context 时应记录 degraded coverage。

## 仓库维护问题

以下问题只适用于你正在维护这个仓库源码，而不是普通 npm 用户。

### Q: 如何跳过某个阶段？

A: 当前 npm CLI 模型没有旧式 `.claude/tasks/<task-id>/0X-stage.md` 阶段状态机，也不建议手动伪造阶段产物。按当前状态选择入口即可：需求不清楚走 brainstorm，路径不清楚走 plan，已有 plan/task pack 走 work，失败走 debug，准备合并走 code-review。

### Q: 如何自定义 Review 标准？

A: 日常 npm 用户通常不需要自定义内部 reviewer。维护本仓库源码时，优先修改 source truth：`skills/spec-code-review/SKILL.md`、相关 `agents/`、`templates/` 或 contract tests。不要手改 `.claude/`、`.codex/`、`.agents/skills/` 的 runtime copies。

### Q: 如何扩展新的 skill？

A: 如果你在仓库源码里扩展能力，参考现有 `skills/<skill-name>/SKILL.md` 创建 source skill，并同步 dual-host governance、入口模板、README/用户手册、CHANGELOG 和 contract tests。日常 npm 用户通常只需要使用现有工作流入口。

### Q: 如何集成到 CI/CD？

A: 在 CI 中运行测试：
```bash
npm run test:unit
npm run test:smoke
npm run test:integration
npm test
```

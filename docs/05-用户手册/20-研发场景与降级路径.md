# 研发场景与降级路径

本文说明 `spec-first` 如何把不同研发拓扑整理成 advisory facts，让 workflow 在直接读源码、清理本地残留或重新 setup 之间做出可解释判断。

核心边界：

- Scripts prepare, LLM decides。脚本只写确定性事实；workflow 根据用户意图、源码、测试和这些事实做语义判断。
- Scenario fingerprint 不是 gate、approval、状态机或风险评分。
- `.spec-first/workspace/*` 是 parent/workspace advisory facts，不替代 child repo 内的 `.spec-first/config/*` 或当前源码。
- Generated runtime mirrors 不手改；需要刷新宿主指引时运行 `spec-first init`。

## 关键产物

| 产物 | 生成者 | 用途 |
| --- | --- | --- |
| `.spec-first/workspace/scenario-fingerprint-setup.json` | `spec-mcp-setup` / `verify-tools.*` | setup-time 场景事实，schema 为 `developer-scenario-fingerprint-setup.v1` |
| `.spec-first/workspace/scenario-fingerprint.json` | `spec-mcp-setup` / `bootstrap-providers.*` | bootstrap-time 合并事实，schema 为 `developer-scenario-fingerprint.v1` |
| `.spec-first/workspace/graph-targets.json` | `resolve-workspace-graph-targets.*` | child repo、Gradle/npm build targets、coverage summary 和 graph coverage class |
| `.spec-first/workspace/parent-artifact-quarantine.json` | `spec-mcp-setup` | parent workspace 中不应被当成 child truth 的 stale/orphan graph artifacts |
| `.spec-first/workspace/mcp-setup-summary.json.quality_signals` | `spec-mcp-setup --all-repos` | mcp-setup 质量基线，例如 process result、command failure、dirty advisory、build-target coverage 和 host drift |

这些产物都是可重建的 runtime/control-plane evidence。它们可以帮助 workflow 选择证据路径，但不能替代源码读取、测试、日志或 reviewer judgment。

## 场景类别

`state_class` 是 matrix interpretation label，不是 durable artifact classifier。当前 provisional 场景包括：

| 场景 | 推荐姿态 |
| --- | --- |
| `clean-single-repo` | 正常使用当前 workflow；需要 direct evidence 时先确认 freshness |
| `dirty-single-repo` | 保持 diff scope 明确；提交、PR 或 review 前披露 dirty graph-affecting paths |
| `first-time-git-repo` | setup-heavy 或 graph-heavy 任务先跑 `$spec-mcp-setup` / `/spec:mcp-setup`；轻量任务可直接读源码 |
| `multi-repo-workspace` | 写入、测试、commit、review autofix 前必须明确 `target_repo` 或 per-child scope |
| `multi-repo-dirty-workspace` | 只对明确 scoped 且 dirty 状态可解释的 child repo 行动；其他 child repo 仅作风险或 follow-up |
| `foreign-residual-workspace` | 先运行 `spec-first clean --workspace-orphans` 预览，并用 `spec-first init` 或 setup/bootstrap 刷新本地 harness |
| `non-git-folder` | 可做文件定位和直接源码审查；不要声称 Git diff、commit freshness 或 review impact evidence |
| `non-git-build-workspace` | direct source evidence 覆盖只对 covered git roots 成立；uncovered Gradle/npm modules 需要 direct source reads |
| `provider-degraded` | graph-backed claim 降级；graph-heavy 任务先跑 `$spec-mcp-setup` / `/spec:mcp-setup` 或改用 bounded direct evidence |

## Capability Matrix

完整解释见 `docs/contracts/workflows/scenario-capability-matrix.md`。默认 capability class 有 5 类：

- `full`：可按正常 workflow 使用 graph/source/test evidence。
- `bounded`：可以继续，但必须显式限定 repo/file scope 并披露限制。
- `partial`：只对覆盖面内的 repo、module 或 provider surface 成立；未覆盖面需要直接源码证据。
- `fallback-only`：不能用 graph/provider/fingerprint 支撑相关 claim，只能用 direct source/test/log evidence 或重新 setup/bootstrap。
- `blocked-action-required`：在清理或刷新前，不做写入、autofix、commit、root-cause claim 或 graph-backed review claim，除非用户明确接受 degraded evidence。

`spec-work`、`spec-code-review` 和 `spec-debug` 有 high-risk override：遇到 foreign residual、provider unavailable 或 non-git build workspace coverage gap 时会更保守。

## 常见操作

老项目升级到场景指纹：

```bash
# 先刷新 source-owned host guidance / runtime mirrors
spec-first init

# 再在当前宿主运行 setup（含可选 provider readiness）
$spec-mcp-setup
```

父 workspace 残留清理：

```bash
spec-first clean --workspace-orphans
spec-first clean --workspace-orphans --confirm
```

第一条命令只预览，不删除。第二条命令只删除 `parent-artifact-quarantine.json` 中记录且通过白名单、repo-relative 和 symlink escape 校验的 parent orphan 路径。

build-target 覆盖缺口处理：

1. 读取 `.spec-first/workspace/graph-targets.json.coverage_summary`。
2. 如果 `graph_coverage_class=partial-build-targets`，只把 direct source evidence evidence 用于 covered git roots。
3. 对 `non_git_build_modules[]` 中 uncovered 的 Gradle/npm module 直接读源码、运行相关测试或记录 limitation。
4. 不要自动索引非 Git package/module 目录。

## 质量信号

`mcp-setup-summary.json.quality_signals` 是 Evaluation Harness baseline，不是优化目标本身。当前字段：

- `child_count`
- `process_results_rate`
- `command_failed_rate`
- `dirty_advisory_child_rate`
- `build_target_coverage_ratio`
- `impact_probe_with_test_provenance_rate`
- `host_instruction_drift_rate`

`spec-optimize` 可以消费这些信号决定先衡量 direct evidence 质量、刷新 host instruction、收窄 graph-dependent experiment scope，或对 uncovered build modules 采用 direct-source fallback。最终优化目标、实验范围和 winner selection 仍由批准的 optimization spec 和测量结果决定。

## 反模式

- 把 `state_class` 当成审批状态或风险分。
- 在普通 plan/work/debug/review 中隐式运行 direct source evidence analyze、group sync、provider repair、hooks、watchers 或 daemons。
- 用 parent workspace 的 `.spec-first/workspace/*` 覆盖 child repo canonical facts。
- 在 generated runtime mirrors 中手改 skill、agent 或 host guidance。
- 因为 graph-targets 命名了相邻 repo 或 module 就扩大 plan/work/review scope。
- 把 advisory facts 写成 confirmed truth，或把缺失 fingerprint 当作老用户工作流失败。

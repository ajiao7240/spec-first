当前 spec-first 要支持的三种开发模式可以这样列：

| 模式 | 形态                    | 典型例子                                                                             | spec-first 处理方式                                                                        |
| -- | --------------------- | -------------------------------------------------------------------------------- | -------------------------------------------------------------------------------------- |
| 1  | **单个 Git 工程单项目**      | 一个 repo 里就是一个应用 / SDK / CLI                                                      | `.spec-first/*`、graph facts、plan/work/review 都以当前 repo 为边界                             |
| 2  | **单个 Git 工程多 module** | monorepo、multi-package、backend/frontend 同 repo、Android 多 module                  | 一个 `.git` 根目录，`.spec-first/*` 放 repo root；GitNexus / CRG 覆盖 repo 内多个 module            |
| 3  | **父目录下多个独立 Git 工程**   | `workspace/project-a`、`workspace/project-b`、`workspace/project-c` 每个都有自己的 `.git` | 父 workspace 只做候选发现和提示；每个 child repo 独立 setup / graph-bootstrap / plan / work |

代码侧也能支撑这个判断：`spec-mcp-setup` 只在 selected Git repo 内写 `.spec-first/config/graph-providers.json`、`.spec-first/config/runtime-capabilities.json`、`.spec-first/config/provider-artifacts.json`，这些是 repo-local setup facts；`spec-graph-bootstrap` 再消费这些事实并把 canonical graph 产物写到 selected repo 的 `.spec-first/graph/*` 和 `.spec-first/impact/*`。

---

## 1. 单个 Git 工程单项目

这是最基础模式：

```text
my-app/
  .git/
  .spec-first/
  src/
  docs/
  README.md
```

适合：

```text
单服务项目
单 CLI 项目
单 SDK 项目
单前端项目
单后端项目
```

职责划分：

```text
GitNexus      → 当前 repo 全局代码认知
CRG           → 当前 repo diff / change impact
spec-first    → brainstorm / plan / tasks / work / review / docs 闭环
```

这是当前实现最自然、最稳定的模式。

---

## 2. 单个 Git 工程多 module

也就是一个 repo 下多个 module：

```text
platform/
  .git/
  .spec-first/
  packages/
    cli/
    core/
    web/
  services/
    api/
    worker/
  docs/
```

或者：

```text
android-app/
  .git/
  app/
  feature-login/
  feature-payment/
  core-network/
  core-ui/
```

这种不是多个 repo，而是**一个 Git 工程里的多个项目模块**。

职责划分：

```text
GitNexus      → repo 级全局代码图谱，识别 module 间关系
CRG           → repo 内变更影响分析，识别跨 module 影响半径
spec-first    → 负责把需求、计划、任务按 module 边界拆分和路由
```

这个模式下 `.spec-first` 应该仍然只放在 repo root：

```text
platform/.spec-first/
```

不要每个 module 都放一套 `.spec-first`，否则 plan/work/review 的事实源会分裂。

---

## 3. 父目录下多个独立 Git 工程

这是 workspace / multi-repo 模式：

```text
workspace/
  spec-first/
    .git/
    .spec-first/
  product-website/
    .git/
    .spec-first/
  backend-api/
    .git/
    .spec-first/
  mobile-app/
    .git/
    .spec-first/
```

这里的关键是：**父目录不是统一 Git repo，每个子目录都是独立 Git 工程。**

当前实现的规则是：

```text
父 workspace 可以运行 spec-mcp-setup 做 host runtime 检查
父 workspace 未选择 child repo 时，project rows 显示 workspace-target-required
从父 workspace 操作 child repo 时，显式传 --repo <child>
每个 repo 有自己的 .spec-first/graph/*
每个 repo 有自己的 plan/work/review 证据闭环
```

示例：

```bash
# 在父 workspace 下发现候选，但不写 parent repo-local state
$spec-mcp-setup

# 显式选择 child repo 后写 child/.spec-first/config/* 和 child/.serena/*
bash skills/spec-mcp-setup/scripts/install-mcp.sh --repo project-a
bash skills/spec-mcp-setup/scripts/verify-tools.sh --repo project-a

# 显式选择 child repo 后写 child/.spec-first/graph/* 和 child/.spec-first/impact/*
bash skills/spec-graph-bootstrap/scripts/bootstrap-providers.sh --repo project-a
```

因为 `spec-graph-bootstrap` 的契约是从 selected repo 的 `.spec-first/config/*` 读取 setup facts，然后输出 selected repo 的 canonical graph readiness artifacts；`spec-plan` 是第一个下游消费者，读取 selected repo 的 graph facts，并在 stale / unavailable 时降级到直接读该 repo。

所以这个模式下不要理解成：

```text
父目录只有一套 .spec-first 管所有 repo
父目录自动选择唯一 child repo
父目录默认对所有 child repos 批处理
```

更合理的是：

```text
每个 Git repo 一套 .spec-first
父 workspace 只输出 candidates、recommended next command 和可选 advisory facts
single-repo plan 写 target_repo: <child>
cross-repo plan 在每个 implementation unit/task 上写 target_repo
review 按 Git repo 分组 diff、graph readiness 和 impact evidence
```

父 workspace 不拥有 repo-local truth。不要在父 workspace 下生成 `.spec-first/config/*`、`.spec-first/graph/*`、`.spec-first/impact/*`、`.serena/*` 或 `.spec-first/config.local*.yaml`。如果未来支持 `--all-repos`，它也必须是显式 maintenance action，逐个 child repo 运行并写各自的 canonical artifacts；不能变成默认 workflow 路径。

# 最终三种模式

可以固定成这三个名称：

```text
1. Single Repo / Single Project
   单个 Git 工程单项目

2. Single Repo / Multi Module
   单个 Git 工程多 module

3. Multi Repo Workspace
   父目录下多个独立 Git 工程
```

中文可以写成：

```text
1. 单仓单项目模式
2. 单仓多模块模式
3. 多仓工作区模式
```

我的建议是，在 spec-first 文档里统一用这组命名：

```text
单仓单项目
单仓多模块
多仓工作区
```

这三个模式就能覆盖你前面说的：

```text
单个 git 工程单项目
单个 git 工程下多个 module
父目录下多个独立 git 工程
```

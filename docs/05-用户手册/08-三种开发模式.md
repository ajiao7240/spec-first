当前 spec-first 要支持的三种开发模式可以这样列：

| 模式 | 形态                    | 典型例子                                                                             | spec-first 处理方式                                                                        |
| -- | --------------------- | -------------------------------------------------------------------------------- | -------------------------------------------------------------------------------------- |
| 1  | **单个 Git 工程单项目**      | 一个 repo 里就是一个应用 / SDK / CLI                                                      | `.spec-first/*`、local evidence facts、plan/work/review 都以当前 repo 为边界                             |
| 2  | **单个 Git 工程多 module** | monorepo、multi-package、backend/frontend 同 repo、Android 多 module                  | 一个 `.git` 根目录，`.spec-first/*` 放 repo root；direct source reads 可覆盖 repo 内多个 module；不要把 module 当独立 repo scope |
| 3  | **父目录下多个独立 Git 工程**   | `workspace/project-a`、`workspace/project-b`、`workspace/project-c` 每个都有自己的 `.git` | 父 workspace 只保存 advisory summaries；每个 child repo 独立 setup / plan / work |

代码侧也能支撑这个判断：`spec-mcp-setup` 只在 selected Git repo 或父 workspace 选中的 child repo 内写 `.spec-first/config/*` 这类 repo-local setup facts。父 workspace 只允许写 `.spec-first/workspace/*summary.json` 这类 advisory summary。

---

## 1. 单个 Git 工程单项目

这是最基础模式：

```text
my-app/
  .git/
  .spec-first/
  src/
  docs/
  README.md
```

适合：

```text
单服务项目
单 CLI 项目
单 SDK 项目
单前端项目
单后端项目
```

职责划分：

```text
direct source evidence      → 当前 repo 全局代码认知
direct source evidence impact → 当前 repo diff / change impact
spec-first    → brainstorm / plan / tasks / work / review / docs 闭环
```

这是当前实现最自然、最稳定的模式。

---

## 2. 单个 Git 工程多 module

也就是一个 repo 下多个 module：

```text
platform/
  .git/
  .spec-first/
  packages/
    cli/
    core/
    web/
  services/
    api/
    worker/
  docs/
```

或者：

```text
android-app/
  .git/
  app/
  feature-login/
  feature-payment/
  core-network/
  core-ui/
```

这种不是多个 repo，而是**一个 Git 工程里的多个项目模块**。

职责划分：

```text
direct source evidence      → repo 级全局代码图谱，识别 module 间关系
direct source evidence impact → repo 内变更影响分析，识别跨 module 影响半径
spec-first    → 负责把需求、计划、任务按 module 边界拆分和路由
```

这个模式下 `.spec-first` 应该仍然只放在 repo root：

```text
platform/.spec-first/
```

不要每个 module 都放一套 `.spec-first`，否则 plan/work/review 的事实源会分裂。

单仓多模块也不要因为存在 `packages/`、`apps/` 或 Android modules 就启用 multi-repo workspace 的 direct source evidence group readiness。它仍然是一个 Git root，查询和 refresh 都按 repo-local local evidence facts 处理。

---

## 3. 父目录下多个独立 Git 工程

这是 workspace / multi-repo 模式：

```text
workspace/
  spec-first/
    .git/
    .spec-first/
  product-website/
    .git/
    .spec-first/
  backend-api/
    .git/
    .spec-first/
  mobile-app/
    .git/
    .spec-first/
```

这里的关键是：**父目录不是统一 Git repo，每个子目录都是独立 Git 工程。**

当前实现的规则是：

```text
父 workspace 可以运行 spec-mcp-setup 做 host runtime 检查、只读候选发现和 all-child maintenance
父 workspace 未传 --repo <child> 时，mcp-setup 默认逐个处理全部 child repos
从父 workspace 操作单个 child repo 时，显式传 --repo <child>
--all-repos 是父 workspace 默认 all-child maintenance 的显式等价入口
每个 repo 有自己的 .spec-first/config/*
每个 repo 有自己的 plan/work/review 证据闭环
```

示例：

```bash
# 在父 workspace 下运行 setup：默认逐个 child 处理，只在父目录写 advisory summary
$spec-mcp-setup

# 显式选择 child repo 后写 child/.spec-first/config/* setup facts
bash skills/spec-mcp-setup/scripts/install-mcp.sh --repo project-a
bash skills/spec-mcp-setup/scripts/verify-tools.sh --repo project-a

# 批量维护：无 --repo 的父 workspace 默认行为；--all-repos 是显式等价入口
bash skills/spec-mcp-setup/scripts/install-mcp.sh
bash skills/spec-mcp-setup/scripts/verify-tools.sh

# 显式等价写法
bash skills/spec-mcp-setup/scripts/install-mcp.sh --all-repos
bash skills/spec-mcp-setup/scripts/verify-tools.sh --all-repos
```

只读代码问题可以先用 `rg`、git status 和 bounded direct reads 找到 candidate repos。LLM 根据用户问题做语义选择；脚本只提供确定性 workspace facts。

因为 `spec-mcp-setup` 的契约是写入 selected repo 的 `.spec-first/config/*` setup facts；`spec-plan`、`spec-work` 和 `spec-code-review` 的当前源码结论仍要直接读 selected repo。

所以这个模式下不要理解成：

```text
父目录只有一套 .spec-first 管所有 repo
父目录自动选择唯一 child repo
普通 plan/work/review 默认跨所有 child repos 写入或测试
父目录的 advisory summary 等同于 child repo 的 source truth
```

更合理的是：

```text
每个 Git repo 一套 .spec-first
父 workspace 只输出 candidates、recommended next command 和可选 advisory facts
父 workspace 下 mcp-setup 默认 all-child maintenance，但不成为 plan/work/review 的默认写入范围
single-repo plan 写 target_repo: <child>
cross-repo plan 在每个 implementation unit/task 上写 target_repo
review 按 Git repo 分组 diff 和验证证据
```

父 workspace 不拥有 repo-local truth。不要在父 workspace 下生成 child repo 的 `.spec-first/config/*` 或 `.spec-first/config.local*.yaml`。父 workspace 默认 all-child maintenance 和显式 `--all-repos` 都只能写 `.spec-first/workspace/*summary.json` 这类 advisory summary；repo-local setup facts 必须仍写到各个 child repo。

“dirty-advisory / stale evidence” 与 “query unavailable” 是两件事：dirty worktree 下的 direct source evidence 证据可能降级，但仍可能作为 stale/advisory read-only evidence 使用。不要为了问一个只读多仓问题而强制用户先 commit/stash/clean；需要对当前源码或测试结论负责时，再直接读取当前源码或运行相应验证。

# 最终三种模式

可以固定成这三个名称：

```text
1. Single Repo / Single Project
   单个 Git 工程单项目

2. Single Repo / Multi Module
   单个 Git 工程多 module

3. Multi Repo Workspace
   父目录下多个独立 Git 工程
```

中文可以写成：

```text
1. 单仓单项目模式
2. 单仓多模块模式
3. 多仓工作区模式
```

我的建议是，在 spec-first 文档里统一用这组命名：

```text
单仓单项目
单仓多模块
多仓工作区
```

这三个模式就能覆盖你前面说的：

```text
单个 git 工程单项目
单个 git 工程下多个 module
父目录下多个独立 git 工程
```

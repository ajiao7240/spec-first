当前 spec-first 要支持的三种开发模式可以这样列：

| 模式 | 形态                    | 典型例子                                                                             | spec-first 处理方式                                                                        |
| -- | --------------------- | -------------------------------------------------------------------------------- | -------------------------------------------------------------------------------------- |
| 1  | **单个 Git 工程单项目**      | 一个 repo 里就是一个应用 / SDK / CLI                                                      | `.spec-first/*`、graph facts、plan/work/review 都以当前 repo 为边界                             |
| 2  | **单个 Git 工程多 module** | monorepo、multi-package、backend/frontend 同 repo、Android 多 module                  | 一个 `.git` 根目录，`.spec-first/*` 放 repo root；GitNexus / CRG 覆盖 repo 内多个 module            |
| 3  | **父目录下多个独立 Git 工程**   | `workspace/project-a`、`workspace/project-b`、`workspace/project-c` 每个都有自己的 `.git` | 每个 repo 独立 setup / graph-bootstrap / plan / work；spec-first 负责跨 repo workflow 编排和上下文路由 |

代码侧也能支撑这个判断：`spec-mcp-setup` 只在 Git repo 内写 `.spec-first/config/graph-providers.json`、`.spec-first/config/runtime-capabilities.json`、`.spec-first/config/provider-artifacts.json`，这些是 repo-local setup facts；`spec-graph-bootstrap` 再消费这些事实并把 canonical graph 产物写到当前 repo 的 `.spec-first/graph/*` 和 `.spec-first/impact/*`。([GitHub][1])

---

## 1. 单个 Git 工程单项目

这是最基础模式：

```text
my-app/
  .git/
  .spec-first/
  src/
  docs/
  README.md
```

适合：

```text
单服务项目
单 CLI 项目
单 SDK 项目
单前端项目
单后端项目
```

职责划分：

```text
GitNexus      → 当前 repo 全局代码认知
CRG           → 当前 repo diff / change impact
spec-first    → brainstorm / plan / tasks / work / review / docs 闭环
```

这是当前实现最自然、最稳定的模式。

---

## 2. 单个 Git 工程多 module

也就是一个 repo 下多个 module：

```text
platform/
  .git/
  .spec-first/
  packages/
    cli/
    core/
    web/
  services/
    api/
    worker/
  docs/
```

或者：

```text
android-app/
  .git/
  app/
  feature-login/
  feature-payment/
  core-network/
  core-ui/
```

这种不是多个 repo，而是**一个 Git 工程里的多个项目模块**。

职责划分：

```text
GitNexus      → repo 级全局代码图谱，识别 module 间关系
CRG           → repo 内变更影响分析，识别跨 module 影响半径
spec-first    → 负责把需求、计划、任务按 module 边界拆分和路由
```

这个模式下 `.spec-first` 应该仍然只放在 repo root：

```text
platform/.spec-first/
```

不要每个 module 都放一套 `.spec-first`，否则 plan/work/review 的事实源会分裂。

---

## 3. 父目录下多个独立 Git 工程

这是 workspace / multi-repo 模式：

```text
workspace/
  spec-first/
    .git/
    .spec-first/
  product-website/
    .git/
    .spec-first/
  backend-api/
    .git/
    .spec-first/
  mobile-app/
    .git/
    .spec-first/
```

这里的关键是：**父目录不是统一 Git repo，每个子目录都是独立 Git 工程。**

当前实现更适合：

```text
每个 repo 独立运行 /spec:mcp-setup
每个 repo 独立运行 /spec:graph-bootstrap
每个 repo 有自己的 .spec-first/graph/*
每个 repo 有自己的 plan/work/review 证据闭环
```

因为当前 `spec-graph-bootstrap` 的契约是从当前 repo 的 `.spec-first/config/*` 读取 setup facts，然后输出当前 repo 的 canonical graph readiness artifacts；`spec-plan` 也是第一个下游消费者，读取当前 repo 的 graph facts，并在 stale / unavailable 时降级到直接读仓库。([GitHub][2])

所以这个模式下不要理解成：

```text
父目录只有一套 .spec-first 管所有 repo
```

更合理的是：

```text
每个 Git repo 一套 .spec-first
spec-first workflow 可以在父目录层面协调多个 repo
GitNexus 可以作为 global knowledge provider 提供跨 repo 认知
CRG 默认仍按具体 repo 的变更集做 impact context
```

# 最终三种模式

可以固定成这三个名称：

```text
1. Single Repo / Single Project
   单个 Git 工程单项目

2. Single Repo / Multi Module
   单个 Git 工程多 module

3. Multi Repo Workspace
   父目录下多个独立 Git 工程
```

中文可以写成：

```text
1. 单仓单项目模式
2. 单仓多模块模式
3. 多仓工作区模式
```

我的建议是，在 spec-first 文档里统一用这组命名：

```text
单仓单项目
单仓多模块
多仓工作区
```

这三个模式就能覆盖你前面说的：

```text
单个 git 工程单项目
单个 git 工程下多个 module
父目录下多个独立 git 工程
```

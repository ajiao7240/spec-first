# 项目规范与胶水基线

`spec-standards` 用来把项目里隐含的工程约定、shared standards、graph evidence 和可复用能力整理成 `.spec-first/standards/` 下的本地 reviewable baseline artifacts。该目录由 `init` 默认加入 `.gitignore`；需要团队共享的 confirmed standards 应 promote 到明确 source 路径。

它适合在 brainstorm、plan、work、code-review 之前使用，让下游 workflow 先知道当前项目是什么形态、哪些规范已经确认、哪些只是观察到的实践、哪些共享标准还需要项目确认。

## 入口

Claude Code：

```bash
/spec:standards
/spec:standards --quick
/spec:standards --refresh --domain cli
/spec:standards --deep
/spec:standards --baseline --import-source ../shared-standards
```

Codex：

```bash
$spec-standards
$spec-standards --quick
$spec-standards --refresh --domain cli
$spec-standards --deep
$spec-standards --baseline --import-source ../shared-standards
```

如果当前宿主里还看不到入口，先重新生成 runtime assets：

```bash
spec-first init --claude
spec-first init --codex
```

不要手改 `.claude/`、`.codex/` 或 `.agents/skills/` runtime mirror 来修入口。

## 什么时候使用

| 场景 | 推荐命令 |
| --- | --- |
| 新项目或首次为项目建立规范上下文 | `$spec-standards --baseline` |
| 想快速判断当前 baseline 是否还可用 | `$spec-standards --quick` |
| 某个 domain、module 或 child repo 发生变化 | `$spec-standards --refresh --domain <name>`、`--module <path>` 或 `--repo <child>` |
| 大型项目需要更深 graph evidence 和多 lens 审查 | `$spec-standards --deep` |
| 团队已有共享规范，想导入后再做项目对齐 | `$spec-standards --baseline --import-source <git-or-path>` |

`--import-source` 可以和 `--baseline`、`--quick`、`--refresh` 或 `--deep` 组合。导入项默认是 `imported`，不是 `confirmed`。

## 模式说明

### `--baseline`

默认模式。用于首次建立项目规范和 glue/reuse capability baseline。

脚本阶段会准备：

- `project-shape.json`
- `standards-plan.json`，其中包含候选规范 shape、状态词表、证据预算、写回策略和 downstream consumers
- `glue-map.json`，其中包含可复用能力和后续 workflow 消费边界

LLM 阶段再基于这些事实合成：

- `standards-candidates.json`
- `standards-preview.md`

baseline 不会自动写 `repo-profile.yaml`。

### `--quick`

快速 freshness check。它只生成或更新：

- `standards-update-decision.json`

这个模式用于回答“现有 standards baseline 还能不能复用”。它不重新生成候选规范，也不改 repo profile。

### `--refresh`

有边界地刷新某个范围。常见写法：

```bash
$spec-standards --refresh --domain cli
$spec-standards --refresh --module src/cli
$spec-standards --refresh --repo packages/app
```

如果传入 `--module`，`project-shape.json` 会记录 `module_detection` 和对应 module facts；如果 inventory 扫描被截断，这些 facts 会标记为 partial。没有传 `--domain`、`--module` 或 `--repo` 时，就按 repo-level refresh 处理，并在输出中说明。

在父级多仓 workspace 中，`--repo <child>` 会把 child repo 作为目标 repo root。默认产物写入 `<child>/.spec-first/standards/`，父级 workspace 不写 child-local standards artifacts。

### `--deep`

用于显式需要更深证据的场景。它会生成：

- `graph-query-index.json`

这个文件只是 deterministic graph query plan。Live GitNexus / MCP 查询结果属于 session-local evidence，不能把 raw query dump 当作 confirmed project policy 提交。

### `--import-source <git-or-path>`

导入共享规范来源。常见写法：

```bash
$spec-standards --baseline --import-source ../team-standards
$spec-standards --quick --import-source https://example.com/team/shared-standards.git
```

本地路径会扫描 `.md`、`.json`、`.yaml`、`.yml` 文件并生成 imported items。远程 git URL 只记录 source descriptor 和 `remote-fetch-not-performed`，不会在 deterministic script 里联网拉取。

导入相关产物：

- `standards-sources.json`
- `import-lock.json`
- `imported-standards.json`

## 直接运行 deterministic script

需要只验证确定性产物时，可以直接运行脚本：

```bash
node skills/spec-standards/scripts/prepare-baseline.js --quick
node skills/spec-standards/scripts/prepare-baseline.js --refresh --domain cli
node skills/spec-standards/scripts/prepare-baseline.js --deep
node skills/spec-standards/scripts/prepare-baseline.js --baseline --import-source ../shared-standards
```

脚本只准备 facts、freshness decision、graph query plan 和 import lock。候选规范、冲突解释、preview 内容和是否写入 repo profile 仍由 LLM 与用户确认决定。

## Artifact 质量门

当 LLM 已经生成 `standards-candidates.json` 和 `standards-preview.md` 后，应先运行 validator，再把它们交给下游 workflow 当作可信 baseline：

```bash
node skills/spec-standards/scripts/validate-artifacts.js --standards-dir .spec-first/standards --json
```

也可以显式指定产物路径：

```bash
node skills/spec-standards/scripts/validate-artifacts.js \
  --candidates .spec-first/standards/standards-candidates.json \
  --preview .spec-first/standards/standards-preview.md \
  --plan .spec-first/standards/standards-plan.json \
  --json
```

validator 只检查 artifact handoff contract：JSON 结构、状态词表、source_type、按状态分级的 support、conflict/unknown 是否可见、preview 是否声明 `repo-profile.yaml` 未修改、patch 是否只引用 confirmed candidate。它不是语义裁判，不判断某条候选规范是否“业务上正确”。

validator pass 是 trusted baseline 的完成标准。markdownlint、cSpell 或 IDE diagnostics 是辅助信号：可以修复不改变契约的格式问题，但不要为了消除诊断而改写 contract heading、candidate id、命令名、路径、工具名或作者名。若诊断与 validator contract 或真实证据冲突，应保留 contract/evidence，并把诊断作为 allowlist/noise 后续处理。

退出码含义：

- `0`：`status=pass` 且 `trust_level=trusted`，可以作为 trusted standards baseline 交给下游消费。
- `1`：validation fail，下游不能当作 trusted baseline。
- `2`：CLI usage error。
- `3`：internal error。
- `4`：`status=pass` 且 `trust_level=degraded`。例如显式传入 `--allow-fallback-vocabulary` 且缺少 `standards-plan.json` 时，validator 只能证明结构可读，不能证明这是可信 baseline。

默认情况下，缺少 `standards-plan.json` 会失败；只有显式使用 `--allow-fallback-vocabulary` 才允许 degraded pass。下游 workflow 看到 `trust_level=degraded`、validator fail 或缺少 validator result 时，只能 advisory consumption，不能把这些 standards 当硬约束。

candidate `id` 必须唯一，因为 `conflicts[]`、`unknowns[]`、外部确认文件和 `repo-profile.patch.yaml` 都用 id 做稳定引用。确认文件或 patch 文件如果不在 `.spec-first/standards/` 默认目录里，可以通过 `--confirmations <path>` 或 `--patch <path>` 显式传入。

## 产物边界

默认产物目录：

```text
.spec-first/standards/
```

本地 review artifacts：

- `project-shape.json`
- `standards-plan.json`
- `glue-map.json`
- `standards-update-decision.json`
- `graph-query-index.json`
- `standards-sources.json`
- `import-lock.json`
- `imported-standards.json`
- `standards-candidates.json`
- `standards-preview.md`

`.spec-first/standards/` 默认由 `init` 加入 `.gitignore`，上述 artifacts 和 scratch/runtime evidence 都不应直接提交。需要团队共享时，应把已确认内容 promote 到明确 source 路径。

scratch/runtime evidence 包括：

- `.spec-first/standards/work/`
- `.spec-first/standards/tmp/`
- `.spec-first/standards/cache/`
- `.spec-first/standards/raw/`
- `.spec-first/standards/graph-query-raw/`
- `*.log`

下游 workflow 只能把 `confirmed` standards 当作硬约束。`observed`、`imported` 和 `suggested` 是 advisory context；`conflict` 是 risk context；`unknown` 是 question context。`advisory` 不是 candidate status，只是消费模式。

下游消费边界由 `standards-plan.json` 的 `synthesis_contract.downstream_consumers` 和 `glue-map.json` 的 `downstream_consumers` 描述。`spec-plan`、`spec-write-tasks`、`spec-work` 和 `spec-code-review` 可以读取这些产物，但不能把未确认候选升级为硬规则。

`glue-map.json` 只用于 reuse-first 判断，帮助发现已有能力和避免重复实现；它不是 workflow 状态机，也不能覆盖 plan、task pack、work scope 或 review 判断。

## 常见路径

日常推荐路径：

1. 新项目先跑 `$spec-standards --baseline`
2. 下游 brainstorm / plan / work / review 读取 `.spec-first/standards/` 作为项目上下文
3. 日常变更后用 `$spec-standards --quick` 检查 freshness
4. 局部变化用 `$spec-standards --refresh --domain <name>` 或 `--module <path>`
5. 团队规范变化时用 `--import-source` 导入，再让 LLM 做 project alignment preview

如果不确定是否需要 standards baseline，先跑 `--quick`；它的成本最低，并会给出下一步建议。

# 快速开始指南

本指南默认你使用的是当前 `spec-first` npm CLI 版本。当前实现里，Claude Code 使用 `/spec:*`，Codex 使用 `$spec-*`，两者不是同一组命令面。

![Spec-First 总览图](../assets/svg/spec-first-overview.svg)

## 1. 安装 CLI

推荐直接从 npm 安装：

```bash
npm install -g spec-first
```

如果你要验证源码发布物，也可以使用：

```bash
git clone https://github.com/sunrain520/spec-first.git
cd spec-first
npm pack
npm install -g ./spec-first-<version>.tgz
```

如果 shell 之前缓存过旧路径，执行：

```bash
hash -r
```

详细说明见 [本地源码安装指南](./06-本地源码安装.md)。

## 2. 检查环境并初始化

安装完成后，**第一个命令**应该是 `spec-first doctor`，确认当前环境状态，再执行 `spec-first init`：

```bash
# 安装后第一个命令：检查环境
spec-first doctor
# 或显式指定平台
spec-first doctor --claude
spec-first doctor --codex

# 确认版本
spec-first -v

# 初始化目标项目
spec-first init
```

按引导多选 Claude Code 和/或 Codex、确认开发者姓名与语言，预览写入内容后再确认。多选框会自动预勾选你上一次选择的宿主（记录在全局 `~/.spec-first/.developer`，跨项目共享一份），首次安装时不预勾选；可按需增减后回车。也可以用 `spec-first init --codex` 或 `spec-first init --claude --codex` 跳过宿主选择；脚本场景用 `spec-first init -y -u <name> --lang zh` 跳过全部提示。

完成后，项目内会生成：

- Claude Code：`.claude/commands/spec`、`.claude/skills`、`.claude/spec-first/workflows`、`.claude/agents`
- Codex：`.agents/skills`、`.codex/agents`
- 平台状态：`.claude/spec-first/*` 或 `.codex/spec-first/*`

其中 `.claude/spec-first/.developer` 或 `.codex/spec-first/.developer` 记录本次项目初始化使用的开发者名称、语言偏好、初始化时间和 CLI 版本。

`spec-first -v` 会稳定显示欢迎页和版本信息；`npm install -g spec-first` 的安装期提示不作为对外保证。

### 最小可用 fast path

完成 `spec-first doctor`、`spec-first init` 并重启宿主后，可以先进入轻量 workflow。这适合 docs-only、小 bugfix、轻量 plan/work/review 和首次项目试用：

```text
/spec:ideate / /spec:brainstorm / /spec:plan / /spec:work / /spec:code-review
$spec-ideate / $spec-brainstorm / $spec-plan / $spec-work / $spec-code-review
```

如果 workflow 需要项目级证据、MCP helper 或跨模块/跨仓影响分析，优先使用 bounded direct source reads、`rg`、ast-grep、git diff、tests/logs 和用户提供证据。缺失或过期的辅助 evidence 只能作为 degraded evidence 披露，不能包装成成功的 direct evidence，也不是所有 workflow 的硬前置。

证据准备触发节点可以按这张表理解：

| 事件或需求 | 默认动作 |
| --- | --- |
| 首次 setup、MCP 缺失或 helper package projection 过期 | 运行 `spec-mcp-setup`；它刷新 required runtime 和 setup-owned facts |
| 需要当前源码证据 | 直接读取 bounded source、git diff、tests/logs 或运行聚焦验证命令 |
| 切换分支、pull、rebase、merge 或 dirty worktree 变化 | 下一个 workflow 重新读取当前 source / diff / tests；不依赖自动 rebuild index |
| docs、typo、小型本地 bug 或首次试用 | local evidence facts stale / unavailable 时披露限制，并继续 bounded direct repo reads |
| shared API/route/provider contract、core workflow、跨模块变更或高风险 review | 扩大 direct-source 读取、运行聚焦测试，并在无法证明影响面时把限制写入 Coverage |

当前推荐的事实准备、专项审查与知识沉淀入口：

- `spec-mcp-setup`：安装并验证 required harness runtime、MCP servers 和 helper tools
- `spec-app-consistency-audit`：移动 App 在运行时验证前的 PRD / Figma / source / route / architecture / analytics / i18n 静态一致性审查入口
- `spec-skill-audit`：审查本仓库 `skills/` source assets 的结构、trigger、治理投递、runtime drift、安全信号和改进计划
- `spec-compound`：在完成工作后捕获稳定知识，沉淀到 `docs/solutions/`
- `spec-first tasks hash|validate`：在 `spec-work` 消费派生 task pack 前做确定性校验

历史遗留的内置图入口、静态上下文文档包和手工注入索引不再是当前入口或 fallback。

如果当前目录是包含多个独立 child Git repos 的父 workspace：

- `spec-mcp-setup` 从父 workspace 无参数运行时，默认按 all-child-repos maintenance 路径处理每个 child repo。
- `--repo <child>` 用于收窄到单个 child repo；`--all-repos` 是显式等价入口。
- 父 workspace 可以写 `.spec-first/workspace/*summary.json` 这类 advisory summary。child repo 的 `.spec-first/config/*` 必须写在 child repo 内。
- 后续 plan、work、review 如果会写文件、跑测试、改 changelog、autofix 或 commit，必须有明确 `target_repo` 或 per-unit/per-task `target_repo`。
- 如果父级 `.git` 是失效 worktree pointer，setup 会把它报告为 `git_health.status="broken-worktree"`，并在 readiness ledger 的 `parent_workspace_advisory` 中提示 `spec-first repair-worktree --dry-run`。该命令只做预览，不会删除 `.git`。
- 如果父级包含 child Git repos 之外的普通业务目录，`coverage_gap` 会显示未覆盖目录数量。默认 all-repos 不会自动把这些普通目录加入 graph bootstrap；需要时使用显式 `--folder <path>` 路径。

如果你想显式指定身份，直接在引导中的用户名和语言步骤输入目标值：

```bash
spec-first init
```

姓名默认值会优先回退到全局 `~/.spec-first/.developer`，再回退到 `git config user.name`。

如果 `doctor` 报告 `legacy managed state`，或者 `clean` 提示当前安装属于 legacy，不要先手动删目录，直接重新运行 `spec-first init` 并选择目标宿主。当前版本会由 `init` 自动执行 managed hard reset 并重建运行时。

## 3. 启动第一个任务

![Spec-First 五阶段工作流](../assets/svg/spec-first-workflow.svg)

从候选发散开始：

```text
# Claude Code
/spec:ideate "添加用户登录功能"
/spec:brainstorm "添加用户登录功能"

# Codex
$spec-ideate "添加用户登录功能"
$spec-brainstorm "添加用户登录功能"
```

然后按标准链路推进：

```text
# Claude Code
/spec:ideate
/spec:plan
/spec:work
/spec:code-review
/spec:compound

# 增强 readiness：首次 setup、MCP 缺失或 helper 配置变化时再运行
/spec:mcp-setup        # 首次或环境变化后准备 required runtime

# Codex
$spec-ideate
$spec-plan
$spec-work
$spec-code-review
$spec-compound

# 增强 readiness：首次 setup、MCP 缺失或 helper 配置变化时再运行
$spec-mcp-setup        # 首次或环境变化后准备 required runtime
```

如果当前任务是移动 App，并且需要在 QA、模拟器、真机或打包前审查 PRD / Figma / source 一致性，可以插入 App consistency audit：

```text
# Claude Code
/spec:app-consistency-audit prd:<path> figma-context:<path> source:<path>

# Codex
$spec-app-consistency-audit prd:<path> figma-context:<path> source:<path>
```

审查产物写入 `.spec-first/app-audit/runs/<run-id>/`。`figma-context:<path>` 是本地 materialized JSON evidence；`figma-ref:<id-or-url>` 只是 reference，缺少本地 context 时不会自动成为 Figma 证据。

这条链路会分别产出：

- `docs/brainstorms/*.md`
- `docs/plans/*.md`
- 代码与测试变更
- 结构化评审结论
- `docs/solutions/**/*.md`

如果你正在维护 spec-first 自身或新增/修改 skill，可以额外运行 skill audit：

```text
# Claude Code
/spec:skill-audit
/spec:skill-audit skills/spec-debug
/spec:skill-audit --runtime --patch-preview

# Codex
$spec-skill-audit
$spec-skill-audit skills/spec-debug
$spec-skill-audit --runtime --patch-preview
```

也可以直接运行脚本，不依赖宿主重新加载：

```bash
node skills/spec-skill-audit/scripts/write-audit-artifacts.js --repo .
node skills/spec-skill-audit/scripts/write-audit-artifacts.js --repo . --target skills/spec-debug
node skills/spec-skill-audit/scripts/write-audit-artifacts.js --repo . --runtime --patch-preview
```

审计结果写入 `.spec-first/audits/skill-audit/latest/`。这是 gitignored 执行产物，不是 source truth；先读 `skill-audit-summary.md`、`skill-improvement-plan.md`、`promise-implementation-report.json` 和相关 JSON 报告，再由 LLM 判断哪些信号需要修复。

## 4. 验证是否可用

在目标项目里再次执行：

```bash
spec-first doctor
# 或显式指定当前平台
spec-first doctor --claude
spec-first doctor --codex
```

如果 Claude Code 的 `commands/spec`、`skills`、`agents` 或 Codex 的 `.agents/skills`、`.codex/agents` 都通过检查，说明项目级工作流已经安装完成。

如需确认当前 setup workflow 资产已落地，可额外检查：

- Claude Code：`.claude/commands/spec/mcp-setup.md`
- Claude Code：`.claude/spec-first/workflows/spec-mcp-setup/SKILL.md`
- Codex：`.agents/skills/spec-mcp-setup/SKILL.md`

如果宿主内仍看不到 `spec-mcp-setup`，说明本地安装需要重新 `init` 并完全重启宿主。

如需确认 skill audit 入口已落地，可额外检查：

- Claude Code：`.claude/commands/spec/skill-audit.md`
- Claude Code：`.claude/spec-first/workflows/spec-skill-audit/SKILL.md`
- Codex：`.agents/skills/spec-skill-audit/SKILL.md`

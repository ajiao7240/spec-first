# 完整示例

下面的示例假设你已经在目标项目里完成了 `spec-first doctor` 和 `spec-first init --claude` 或 `spec-first init --codex`。Claude Code 使用 `/spec:*` 命令入口；Codex 使用 `$spec-*` skill 入口。

## 场景：添加用户登录功能

### 步骤 0: 准备运行环境和代码事实

首次进入项目、换宿主、升级后重建 runtime assets，或 MCP/helper 环境变化后，先在终端完成 CLI 与宿主初始化：

```bash
spec-first doctor
spec-first init --claude -u <name> --lang zh
# 或
spec-first init --codex -u <name> --lang zh
```

重启 Claude Code 或 Codex 后，在宿主会话内准备 required runtime 和 graph readiness：

```text
/spec:mcp-setup
/spec:graph-bootstrap

$spec-mcp-setup
$spec-graph-bootstrap
```

`mcp-setup` 和 `graph-bootstrap` 只准备确定性事实。后续需求、计划、实现和评审判断仍由 LLM 基于这些 facts、当前代码和用户目标完成。

### 步骤 1: Ideate + Brainstorm

```text
/spec:ideate "添加用户登录功能"
/spec:brainstorm "添加用户登录功能"

$spec-ideate "添加用户登录功能"
$spec-brainstorm "添加用户登录功能"
```

**对话过程**

```
Claude: 让我们探索一下登录功能的需求。请问：
1. 需要支持哪些登录方式？
   A. 用户名密码
   B. 邮箱密码
   C. 第三方登录（Google/GitHub）

用户: A 和 B

Claude: 好的。关于安全性：
1. 是否需要验证码？
2. 是否需要记住登录状态？

用户: 都需要

Claude: 明白了。我会生成需求文档...
```

**生成文档**：`docs/brainstorms/2026-03-29-user-login-requirements.md`

```markdown
---
date: 2026-03-29
topic: user-login
---

# 用户登录功能需求

## 核心需求
- 支持用户名/邮箱 + 密码登录
- 登录验证码
- 记住登录状态

## 验收标准
- 用户可以成功登录
- 错误提示清晰
- 登录状态持久化
```

### 步骤 2: Plan

```text
/spec:plan
$spec-plan
```

**生成计划**：`docs/plans/2026-03-29-001-feat-user-login-plan.md`

```markdown
---
date: 2026-03-29
type: feat
origin: docs/brainstorms/2026-03-29-user-login-requirements.md
---

# 实施计划

## 技术选型
- 后端：JWT token
- 验证码：图形验证码
- 存储：Redis session

## 任务拆解
1. 创建登录 API
2. 实现验证码生成
3. 实现 JWT 认证
4. 前端登录表单
5. 测试用例
```

### 步骤 3: 可选 task pack

如果计划涉及多个模块、多个阶段、并行执行或明确交接，可以先用已安装的 standalone `write-tasks` skill 从 plan 派生 task pack：

```text
docs/tasks/2026-03-29-001-feat-user-login-tasks.md
```

task pack 不是新的需求真相源。它依赖 source plan 和 hash freshness；plan 变化后应重新编译或验证。

### 步骤 4: Work

```text
/spec:work
$spec-work
```

**执行开发**，更新代码与测试，并按 plan 推进实现。

```markdown
# 实施结果

## 已完成
- ✅ 登录 API：POST /api/login
- ✅ 验证码生成：GET /api/captcha
- ✅ JWT 认证中间件
- ✅ 前端登录表单
- ✅ 单元测试

## 测试结果
- 所有测试通过
```

### 步骤 5: Review

```text
/spec:code-review
$spec-code-review
```

**评审结果**：输出结构化 findings，并在需要时引用 `docs/plans/2026-03-29-001-feat-user-login-plan.md`

```markdown
## 优点

- 测试覆盖完整

## 建议
- 可以提取验证码生成为独立模块
```


## 总结

完整流程通常会产出：
- 运行环境事实、graph readiness facts、需求文档、实施计划、可选 task pack、评审结论、解决方案文档
- 完整的代码实现
- 可复用的知识资产

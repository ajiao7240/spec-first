# Spec-First 内部培训使用讲稿

> 适用场景：面向第一次接触 `spec-first` 的研发同学，进行一场 `90` 分钟左右的内部使用培训。
>
> 本讲稿只回答一个问题：
>
> **第一次怎么正确把 `spec-first` 用起来。**

## 培训目标

这场培训结束后，学员应该能做到：

- 理解 `spec-first` 是 workflow system，而不是 prompt 技巧集合
- 知道第一次进入项目后应该先做什么
- 知道 Claude 和 Codex 的入口差异
- 知道 setup / graph-bootstrap 是准备工具和代码事实，不是业务判断
- 知道单仓、多模块和多仓 workspace 的 `.spec-first` 边界
- 看懂一次最小闭环会产生什么 artifact
- 避开第一次最容易踩到的错误路径

## 讲师提醒

- 这不是架构设计分享，不要展开讲完整实现细节
- 这不是产品发布会，不要堆能力列表
- 主线只围绕：`doctor -> init -> restart -> mcp-setup / graph-bootstrap -> workflow -> artifact`
- 讲解时以当前仓库 canonical 文档为准：
- 首次使用路径：`docs/05-用户手册/01-快速开始.md`
- 核心概念：`docs/05-用户手册/02-核心概念.md`
- 完整示例：`docs/05-用户手册/03-完整示例.md`
- 宿主入口真相：`docs/contracts/dual-host-governance/README.md`
- source-of-truth 边界：`docs/solutions/workflow-issues/modify-source-not-artifacts-2026-04-13.md`

## 第一部分：它解决什么问题（15 分钟）

### 这一段要回答的问题

**为什么我们需要 `spec-first`，而不是继续把 AI coding 当成 prompt 对话？**

### 建议开场讲法

很多 AI coding 的失败，不是模型不会写代码，而是模型拿到的上下文、目标、验证标准和历史知识都不稳定。

`spec-first` 解决的不是“怎么写一条更厉害的 prompt”，而是把原来零散的对话过程，收口成一个可重复的 workflow：

`Brainstorm -> Plan -> Work -> Review -> Compound`

当前完整工程闭环还包括前置的 `mcp-setup / graph-bootstrap`，以及按场景插入的 `debug`、`app-consistency-audit`、`skill-audit`、`compound-refresh` 等 supporting workflows。培训时不要把这些讲成必须按顺序跑完的状态机；它们是不同问题状态的入口。

如果要把这句话讲得再短一点，可以直接说：

> `spec-first` 不是帮你多生成一点代码，而是帮你把 AI coding 变成一个有输入、有过程、有产物、有复用的工程闭环。

### 这一段只讲 4 个概念

#### 1. Workflow

核心不是“有没有命令”，而是“每一步在解决什么问题”。

- `Brainstorm`：明确要做什么
- `Plan`：明确怎么做
- `Work`：按计划完成实施
- `Review`：结构化检查问题
- `Compound`：把这次经验沉淀成以后还能复用的知识

#### 2. Artifact

`spec-first` 不只是聊天，它会留下可以继续消费的 artifact。

例如：

- `docs/brainstorms/*.md`
- `docs/plans/*.md`
- `docs/solutions/**/*.md`

这些文件不是“聊天记录备份”，而是 workflow 继续往下走的输入。

#### 3. Decision Input

这个项目最重要的判断是：

> AI coding 的质量，受限于 decision input 的质量，而不是 orchestration 的重量。

课堂上不用展开术语，只要让大家记住：

- 不要急着让模型“开始写”
- 先让模型拿到更清楚的上下文、目标、边界和验证方式

#### 4. 正确进入比高级技巧更重要

第一次培训，最重要的不是覆盖所有能力，而是先防止误入。

如果第一次就：

- 跳过 `doctor`
- 没有 `init`
- 没重启宿主
- 用错 Claude / Codex 入口

后面的体验通常都会偏掉。

### 这一段收口话术

可以这样收口：

> 所以今天这场培训，不是讲 `spec-first` 的全部，而是只讲一件事：第一次怎么正确用起来。

## 第二部分：第一次怎么开始（20 分钟）

### 这一段要回答的问题

**一个第一次使用的人，应该按什么顺序进入？**

### canonical 顺序

必须按这个顺序讲：

1. 安装 CLI
2. `spec-first doctor`
3. `spec-first init` 并选择目标宿主
4. 重启宿主
5. 运行 `mcp-setup` 准备 required runtime
6. 按提示运行 `graph-bootstrap` 编译代码图谱 readiness
7. 进入业务 workflow

这 5 步里，最容易被跳过的是：

- `doctor`
- `restart`
- `mcp-setup`

要强调：这些步骤不是可选礼节，而是第一次使用路径的一部分。

### Claude 与 Codex 的入口差异

这里必须讲得非常明确，不要模糊带过。

| Host | 用户入口 |
|---|---|
| Claude Code | `/spec:*` |
| Codex | `$spec-*` |

讲师口播建议：

> Claude 用户看到的是 `/spec:*`。  
> Codex 用户看到的是 `$spec-*`。  
> 不要把这两者混用。

### 建议现场展示的最小命令

```bash
spec-first doctor
spec-first init
```

或：

```bash
spec-first doctor
spec-first init
```

然后立刻补一句：

> 初始化完成后，必须重启当前宿主；进入新会话后先跑当前宿主的 `mcp-setup`，再按提示跑 `graph-bootstrap`。

### 这一段要明确但不要展开的内容

- `doctor` 是做环境与状态检查
- `init` 是安装项目级 runtime 资产
- runtime 资产生成后，宿主需要重新加载

不要在这一段展开讲：

- runtime asset 同步细节
- template 分发细节
- governance schema 细节

### 推荐阅读顺序

如果学员会后只读 3 份材料，推荐顺序固定为：

1. `docs/05-用户手册/01-快速开始.md`
2. `docs/05-用户手册/02-核心概念.md`
3. `docs/05-用户手册/03-完整示例.md`

## 第三部分：最小 live demo（25 分钟）

### 这一段要回答的问题

**一次最小闭环到底长什么样？**

### demo 目标

这次 demo 不追求展示全能力，只展示 4 件事：

1. 第一次进入项目后先做什么
2. workflow 是怎么被触发的
3. artifact 是怎么出现的
4. 这些 artifact 后续怎么继续被消费

### 建议 demo 主线

建议用一条最短路径，不要现场探索：

1. `spec-first doctor`
2. `spec-first init` 并选择目标宿主
3. 重启宿主
4. 运行 `mcp-setup` 和 `graph-bootstrap`
5. 运行一个 workflow 入口
6. 打开对应 artifact

### workflow 入口示例

Claude 示例：

```bash
/spec:ideate "添加用户登录功能"
/spec:plan
```

Codex 示例：

```bash
$spec-ideate "添加用户登录功能"
$spec-plan
```

### demo 时要边做边说的解释

每跑完一步，都要立刻解释“这一步意味着什么”。

例如：

- 跑完 `doctor`：这是第一次进入项目前的环境检查，不是可选步骤
- 跑完 `init`：项目级 runtime 资产已经安装，但宿主还没重新加载
- 跑完 `mcp-setup`：required runtime、MCP、graph providers 和 helper tools 已被安装或验证
- 跑完 `graph-bootstrap`：代码图谱 readiness facts 已编译，后续 workflow 可读取；不可用时要披露限制并回退 direct reads
- 重启后进入 workflow：现在用户入口才真正可用
- 看到 artifact：这些文件不是临时记录，而是 workflow 下一步的输入

### demo 中至少展示 2 类 artifact

建议至少展示：

- `docs/brainstorms/*.md`
- `docs/plans/*.md`

如果时间允许，再补充：

- `docs/solutions/**/*.md`

### demo 收口话术

可以这样说：

> 大家现在看到的重点，不是“命令运行成功了”，而是 `spec-first` 把一次 AI coding 过程变成了一个可继续消费的 workflow。

## 第四部分：artifact 怎么读（15 分钟）

### 这一段要回答的问题

**这些生成出来的文件到底有什么用？**

### 最少讲清 4 类 artifact

#### 1. `docs/brainstorms/*.md`

作用：记录这次任务“要做什么”。

你可以把它理解成：

- 需求澄清结果
- scope 边界说明
- success criteria 的初步落地

#### 2. `docs/plans/*.md`

作用：记录这次任务“怎么做”。

它不是代码实现本身，而是实施前的设计与执行边界。

#### 3. `docs/solutions/**/*.md`

作用：沉淀“这次问题以后怎么复用”。

这是 `spec-first` 和普通 prompt 驱动使用方式差别很大的地方。  
普通对话往往结束就结束了，而这里希望把可复用经验留下来。

#### 4. runtime artifacts

例如：

- `.agents/skills/`
- `.claude/`
- `.codex/`
- `.spec-first/`

课堂上要讲清楚：

这些更多是运行时副本、控制面产物或生成结果，**不是默认手改位置**。

父 workspace 下的 `.spec-first/workspace/` 也只是 advisory summary。每个 child repo 仍然拥有自己的 `.spec-first/config/`、`.spec-first/graph/`、`.spec-first/impact/` 和 `.spec-first/providers/`。

### source-of-truth 和 runtime artifact 的课堂表达

建议讲得非常朴素，不必上来用太重术语：

> `skills/`、`agents/`、`templates/` 这类更接近真源。  
> `.agents/skills/`、`.claude/`、`.codex/`、`.spec-first/` 更多是运行时副本或生成产物。  
> 发现内容不对时，先判断你看到的是源头还是副本。

### 这一段收口话术

> 所以 `spec-first` 不是“帮你多存几个文件”，而是把原来会丢失的中间决策，变成以后还能继续利用的 artifact。

## 第五部分：常见误区与 Q&A（15 分钟）

### 这一段要回答的问题

**第一次最容易错在哪？**

这一段不要只列 warning，要用“错误路径 -> 后果 -> 正确路径”的方式讲。

### 误区 1：把 `spec-first` 当成 prompt 技巧集合

**错误路径**
- 只记命令名
- 不关心 workflow 边界和 artifact

**后果**
- 使用过程退化成“换了个壳继续聊天”
- 看不到这套体系真正的价值

**正确路径**
- 把它当成 workflow system 使用
- 关注每一步的输入、输出和下一步消费关系

### 误区 2：跳过 `doctor / init / restart`

**错误路径**
- 安装完就直接进入 workflow

**后果**
- 宿主内入口不可见
- 运行时状态不完整
- 容易误判为“工具不好用”

**正确路径**
- 严格按首次使用顺序进入：
- `spec-first doctor`
- `spec-first init`
- 重启宿主
- `/spec:mcp-setup` 或 `$spec-mcp-setup`
- `/spec:graph-bootstrap` 或 `$spec-graph-bootstrap`
- 再进入 workflow

### 误区 3：在 Codex 中误用 `/spec:*`

**错误路径**
- 把 Claude 的入口口径带到 Codex

**后果**
- 直接用错入口
- 对双宿主产品面产生错误心智

**正确路径**
- 记住：
- Claude 用 `/spec:*`
- Codex 用 `$spec-*`

### 误区 4：修改 runtime artifact 而不是 source-of-truth

**错误路径**
- 看到 `.agents/skills/`、`.claude/`、`.codex/` 里的内容不对，就直接进去改

**后果**
- 改动不可持续
- 下次同步或重新安装后被覆盖
- 容易误把副本问题当成源码问题

**正确路径**
- 先判断当前文件是源头还是运行时副本
- 优先修改 source-of-truth，再通过同步链路刷新副本

### 误区 5：把父 workspace 当成一个统一 repo

**错误路径**
- 在父目录下直接要求 plan/work/review 修改所有 child repos
- 把 `.spec-first/workspace/*summary.json` 当成跨 repo canonical truth

**后果**
- scope 不可追踪
- review 和 graph facts 混在一起
- 容易把一个 child repo 的事实误用到另一个 child repo

**正确路径**
- setup 和 graph-bootstrap 可从父 workspace 做 all-child maintenance
- 真正写入、测试、review autofix 或 commit 前，plan/task pack 必须写明 `target_repo` 或 per-unit/per-task `target_repo`

### Q&A 建议优先回答的问题

课堂最后优先回答这些问题：

1. 我第一次应该先看哪几份文档？
2. Claude 和 Codex 到底有什么不同？
3. 为什么要先 `doctor`？
4. 为什么我已经 `init` 了，宿主里还看不到入口？
5. 哪些文件可以长期保留，哪些只是运行时副本？
6. 多仓 workspace 里父目录和 child repo 的边界是什么？
7. 一次最小 workflow 到底至少要走到哪一步？

## 结尾收口

最后建议用一句话收尾：

> 第一次使用 `spec-first`，不要追求把所有能力都学完。  
> 先把正确进入、正确触发、正确看懂 artifact 这三件事做对，后面的学习成本会低很多。

## 课后行动建议

培训结束后，建议给学员一个最小行动路径：

1. 重新阅读 `docs/05-用户手册/01-快速开始.md`
2. 对照 `docs/05-用户手册/03-完整示例.md` 复现一次最小闭环
3. 记录自己第一次使用时最卡的一步
4. 再回头看 `docs/05-用户手册/02-核心概念.md`，理解 workflow 与 artifact 的关系

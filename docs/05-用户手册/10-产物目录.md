# 产物目录

本文说明 `spec-first` 常见产物由谁生成、谁读取、是否应提交，以及哪些目录是 generated runtime assets。它的重点是边界，不是把 workflow 固化成状态机。

## 核心原则

- 长期协作知识写入 `docs/` 下的 requirements、plans、tasks 和 solutions。
- CLI、skill、agent、template 的 source truth 位于 `src/cli/`、`skills/`、`agents/` 和 `templates/`。
- `.claude/`、`.codex/` 和 `.agents/skills/` 是 generated runtime assets，可由 `spec-first init` 重建，不应作为 source truth 手改。
- `.spec-first/` 下多为 runtime/control-plane facts，通常不提交到 Git。
- 脚本负责确定性事实和格式校验，LLM 负责需求、取舍、实现和评审判断。

## Workflow 文档产物

| 路径 | 主要生成者 | 主要读取方 | Git 边界 | 说明 |
| --- | --- | --- | --- | --- |
| `docs/brainstorms/*-requirements.md` | `$spec-brainstorm` / `/spec:brainstorm` | `$spec-plan` / `/spec:plan`、doc review、后续维护者 | 通常提交 | 需求 brief，记录问题、范围、非目标和成功标准 |
| `docs/plans/*-plan.md` | `$spec-plan` / `/spec:plan` | `$spec-work` / `/spec:work`、standalone `write-tasks` skill、code/doc review | 通常提交 | 实施计划，是执行前的主要决策上下文 |
| `docs/tasks/*-tasks.md` | standalone `write-tasks` skill | `$spec-work` / `/spec:work` | 视团队协作需要提交 | 从 plan 派生的 executable handoff，依赖 source plan 和 hash freshness |
| `docs/solutions/**/*` | `$spec-compound` / `/spec:compound` | 后续 brainstorm、plan、work、debug、review 会话和维护者 | 通常提交 | 解决问题后的可复用经验，不是当前需求或计划的替代品 |
| `CHANGELOG.md` | 执行变更的 agent / 维护者 | reviewer、release、用户 | 提交 | 本仓库要求任何源码或文档变更都同步记录 |

## Review、debug 与验证产物

| 产物形态 | 主要生成者 | 主要用途 | 边界 |
| --- | --- | --- | --- |
| 结构化 findings / residual risks | `$spec-code-review` / `/spec:code-review` | 合并前发现 bug、回归和测试缺口 | 以当前 diff、计划和测试为范围，不替代实际验证 |
| App consistency audit report | `$spec-app-consistency-audit` / `/spec:app-consistency-audit` | 在移动 App 运行时验证前审查 PRD、Figma、source、route、architecture、analytics 和 i18n 一致性 | `.spec-first/app-audit/` 是执行产物；Figma ref 不等于 materialized evidence |
| root cause / fix / verification notes | `$spec-debug` / `/spec:debug` | 解释失败原因并证明修复 | 优先复现，再收敛最小修复 |
| test output / command output | shell、npm scripts、CI | 提供确定性验证事实 | 不应由 LLM 伪造；失败要保留真实限制 |

这些产物不一定每次都落成单独文件。是否写入 docs 取决于对应 workflow 的职责和当前任务价值。

## Generated runtime assets

| 路径 | 生成方式 | 是否 source truth | 是否手改 | 说明 |
| --- | --- | --- | --- | --- |
| `.claude/commands/spec/` | `spec-first init --claude` | 否 | 否 | Claude Code `/spec:*` command runtime |
| `.claude/skills/` | `spec-first init --claude` | 否 | 否 | Claude standalone / internal skill runtime |
| `.claude/spec-first/workflows/` | `spec-first init --claude` | 否 | 否 | Claude command-backed workflow skill copies |
| `.claude/agents/` | `spec-first init --claude` | 否 | 否 | Claude agent runtime |
| `.codex/agents/` | `spec-first init --codex` | 否 | 否 | Codex agent runtime |
| `.agents/skills/` | `spec-first init --codex` | 否 | 否 | Codex workflow、standalone 与 internal skill runtime |

如果这些目录漂移，修复方式是重新运行对应宿主的 init，而不是直接编辑 runtime copy。

## Source truth 资产

| 路径 | 角色 | 修改后通常需要 |
| --- | --- | --- |
| `src/cli/` | CLI 行为、命令实现、contract 校验 | 单元/集成/smoke 测试，必要时 build |
| `skills/` | source skill 定义和脚本 | source 文件检查、contract 测试，必要时 fresh-source eval |
| `agents/` | source agent 定义 | source 文件检查、contract 测试，必要时 fresh-source eval |
| `templates/` | runtime 生成模板 | init / smoke / governance contract 测试 |
| `docs/` | 协作文档、计划、手册和长期知识 | Markdown link、内容 contract 或相关文档测试 |
| `tests/` | 回归和 contract 保障 | 对应测试命令 |

## `.spec-first/` control-plane facts

`.spec-first/` 下的目录用于 setup、graph bootstrap、App consistency audit、verification 和 quality gate 的机器事实。更完整的 control-plane 映射见 [Workflows 与产物地图](./04-workflows-artifacts-map.md)。

常见目录：

- `.spec-first/config/`
- `.spec-first/providers/`
- `.spec-first/graph/`
- `.spec-first/impact/`
- `.spec-first/audits/`
- `.spec-first/app-audit/`
- `.spec-first/workflows/`

这些目录回答“当前机器事实是什么”，不是长期手工维护知识库。若事实 stale、blocked 或 degraded，下游 workflow 应说明限制，并回退到 bounded direct repo reads 或已配置 provider。

## 选择与维护建议

- 只想记录需求：写 `docs/brainstorms/`。
- 需要执行前共识：写 `docs/plans/`。
- 计划很大、需要交接或并行执行：从 plan 派生 `docs/tasks/`。
- 问题已经解决且经验可复用：写 `docs/solutions/`。
- runtime 看起来坏了：先判断 source truth 是否正确，再用 `spec-first init --claude` 或 `spec-first init --codex` 重建。
- 不确定某个产物是否该提交：优先提交 durable docs 和源码资产，不提交可重建 runtime/control-plane facts。

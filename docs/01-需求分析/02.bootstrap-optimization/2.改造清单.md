 **基于当前 `spec-graph-bootstrap` 的改造版 task 清单 + PRD 模板骨架**，目标只围绕一件事：

> **让 bootstrap 的产物更适合后续节点注入，从而提升 AI 编码质量。**

我会尽量遵守当前 skill 的已有骨架，不搞推倒重来。

---

# 一、先基于当前实现定边界

当前 `spec-graph-bootstrap` 已经明确了这些事实：

* 它是 **Stage-0 supporting workflow**，目标是在 `docs/contexts/<slug>/` 下生成长期上下文资产。([GitHub][1])
* 当前版本范围是 **“generate context assets only”**，自动注入后续五阶段还是 future capability。([GitHub][1])
* 现有 Phase 2 已有固定任务：`summary-context`、`architecture-context`、`pitfalls-context`，并按 layer、database 条件增补任务。([GitHub][1])
* 运行时控制面已经存在：会在 `.context/spec-first/bootstrap/<slug>/tasks/<task-id>/prd.md` 里为每个任务生成 PRD。([GitHub][1])

所以最合理的优化方式不是换框架，而是：

**保留现有宿主检查、模式探测、Phase 1/2/3 结构，补充更面向“执行注入”的任务和产物。**

---

# 二、改造目标

把当前产物从：

* 项目理解文档

推进到：

* **项目理解文档 + 代码事实索引 + 任务型上下文包 + 注入优先级规则**

也就是让 bootstrap 产物能直接服务后续：

* brainstorm
* plan
* work
* review

而不是只做“项目知识归档”。

---

# 三、建议的改造版 task 清单

## A. 保留现有任务，但增强其输出目标

### 1. `summary-context`

保留输出：

* `docs/contexts/<slug>/00-summary.md`

但内容要求增强为：

* 项目一句话定位
* 主语言 / 主框架 / 启动入口
* 核心目录与模块入口
* 推荐优先阅读文档
* 高风险区域概览
* 当前 bootstrap mode（Full / Enhanced / Basic）

当前 skill 已要求至少识别主语言、主框架、顶层结构；这是很好的基础。([GitHub][1])

---

### 2. `architecture-context`

保留输出：

* `architecture/system-overview.md`
* `architecture/module-map.md`
* `architecture/integration-boundaries.md`（条件生成）

增强方向：

* `system-overview.md` 增加运行路径视角
* `module-map.md` 增加“做某类需求该先去哪”
* `integration-boundaries.md` 增加接口约束与风险提示

当前 skill 已经把这三个文件定义为整体结构、目录职责、集成边界三块，这个设计是对的。([GitHub][1])

---

### 3. `pitfalls-context`

保留输出：

* `pitfalls/index.md`

增强方向：

* 由“按文件列风险”改成“按任务类型 + 风险类型列风险”
* 每个坑点补“改动前检查项”
* 明确关联哪些目录/模块/测试

当前 skill 已要求要有文件位置、行号范围、风险类型、成因和缓解建议，这已经很好，下一步只是更贴近执行。([GitHub][1])

## B. 新增任务 1：`code-facts-context`

这是最重要的新增任务。

### 目标

生成**代码事实层**，给后续节点提供稳定、低 token、尽量少主观解释的上下文。

### 建议输出

```text
docs/contexts/<slug>/code-facts/
  public-entrypoints.md
  core-dependencies.md
  test-map.md
  high-risk-modules.md
```

### 这四个文件的作用

#### `public-entrypoints.md`

告诉 agent：

* 项目从哪里进入
* HTTP / CLI / worker / cron / MQ 各自入口在哪
* 哪些入口最常被改动涉及

#### `core-dependencies.md`

告诉 agent：

* 哪些模块依赖最重
* 哪些 shared/core/util 是高扇出节点
* 哪些目录是公共基础层

#### `test-map.md`

告诉 agent：

* 核心模块对应哪些测试
* 哪些区域测试薄弱
* 哪些任务类型应优先联看哪些测试

#### `high-risk-modules.md`

告诉 agent：

* 哪些模块一改容易波及全局
* 哪些模块复杂度高、依赖多、测试弱
* 哪些是 review 时必须重点关注的区域

---

## C. 新增任务 2：`injection-context`

### 目标

把“项目知识”转成“任务型上下文包”。

### 建议输出

```text
docs/contexts/<slug>/context-packs/
  add-endpoint.md
  bugfix.md
  refactor-shared.md
  review-change.md

docs/contexts/<slug>/injection-index.yaml
```

### 这些文件的作用

#### `add-endpoint.md`

新增接口时先看什么：

* 路由入口
* DTO/schema
* service/repository
* 相关测试
* 错误处理约定

#### `bugfix.md`

修 bug 时先看什么：

* 入口链路
* 异常处理点
* 常见脆弱区
* 相关日志和测试

#### `refactor-shared.md`

重构公共模块时先看什么：

* 下游依赖
* 影响半径
* 兼容性点
* 相关测试

#### `review-change.md`

review 改动时先看什么：

* 常见漏改区域
* 相关测试
* 风险信号
* 影响面提示

#### `injection-index.yaml`

定义不同阶段/任务的文档注入优先级。

这个文件很关键，因为当前 skill 明确说自动注入还是未来能力；那现在就应该先把“怎么注入”显式写成规则文件。([GitHub][1])

# 五、每个 task 的建议输入 / 输出 / 验收标准

## 1. `summary-context`

**输入**

* 顶层目录
* 技术栈配置
* 启动入口
* mode 结果
* layer 检测结果

**输出**

* `00-summary.md`

**验收**

* 能回答“这个项目是什么、怎么启动、主要模块在哪里”
* 能给后续 agent 一个明确入口
* 不写空泛结论

---

## 2. `architecture-context`

**输入**

* 目录结构
* 路由/启动链
* 模块依赖
* 外部集成点

**输出**

* `system-overview.md`
* `module-map.md`
* `integration-boundaries.md`

**验收**

* 能回答“系统怎么组织”
* 能回答“这个需求大概率先去哪”
* 能回答“系统边界和外部依赖在哪”

---

## 3. `pitfalls-context`

**输入**

* 高复杂度代码
* 全局状态
* 隐式约定
* 低覆盖区域
* 异常吞掉区域

**输出**

* `pitfalls/index.md`

**验收**

* 每个坑点都有位置、原因、风险、缓解建议
* 能直接指导 review 或改代码
* 不写“注意性能/注意安全”这种空话

---

## 4. `code-facts-context`

**输入**

* 入口文件
* import/call graph
* 测试目录和模块映射
* 高扇出/高复杂度模块

**输出**

* `public-entrypoints.md`
* `core-dependencies.md`
* `test-map.md`
* `high-risk-modules.md`

**验收**

* 内容尽量客观、少解释
* 可直接服务 plan/work/review
* 能帮助 agent 做最小上下文选择

---

## 5. `injection-context`

**输入**

* summary
* architecture
* pitfalls
* code-facts
* layer outputs
* database outputs（如有）

**输出**

* `context-packs/*.md`
* `injection-index.yaml`

**验收**

* 不同任务类型有不同注入包
* 注入顺序清晰
* 能直接供后续 workflow 读取

# 六、改造版 PRD 模板骨架

当前 skill 已经有每 task 一份 `prd.md` 的控制面机制。([GitHub][1])
我建议把 PRD 模板统一成下面这种，适合所有 context task。

## 通用 PRD 模板

```md
# Task PRD: <task-id>

## Goal
本任务的目标是什么。
说明它要解决的“后续节点上下文缺口”。

## Target Output
- docs/contexts/<slug>/<path1>
- docs/contexts/<slug>/<path2>

## Why It Matters
说明这些产物将如何提升 brainstorm / plan / work / review 的质量。

## Data Sources
优先数据来源：
- source code
- config files
- routing/bootstrap files
- schema / migrations
- tests

次级数据来源：
- package metadata
- env examples
- CI config

## Extraction Rules
- 先抽事实，再做摘要
- 禁止脱离代码现实编造架构
- 发现不确定项时明确标注
- 不要把推断写成确定事实

## Output Requirements
- 内容结构
- 必须包含的栏目
- 必须包含的位置/文件引用
- 必须避免的空话

## Acceptance Criteria
- 读者能用它回答什么问题
- 能否直接帮助后续节点减少搜索范围
- 是否足够具体、可执行

## Risks / Caveats
- 哪些地方容易误判
- 哪些内容需要保守表达

## Suggested Workflow
1. 扫描主要事实来源
2. 提取结构信息
3. 归纳重点
4. 生成文档
5. 自检并修正文风
```

# 七、两个新增 task 的专用 PRD 骨架

## `code-facts-context` PRD

```md
# Task PRD: code-facts-context

## Goal
生成项目代码事实索引，为后续 plan/work/review 提供稳定、低 token、客观的上下文层。

## Target Output
- docs/contexts/<slug>/code-facts/public-entrypoints.md
- docs/contexts/<slug>/code-facts/core-dependencies.md
- docs/contexts/<slug>/code-facts/test-map.md
- docs/contexts/<slug>/code-facts/high-risk-modules.md

## Data Sources
优先：
- main/bootstrap/router/worker/cron 入口文件
- import / call relationships
- tests and fixtures
- shared/core/util modules

## Extraction Rules
- 以事实为主，解释为辅
- 避免泛泛架构描述
- 明确写出入口、依赖、测试、高风险区域
- 不确定项标注为“推断”

## Acceptance Criteria
- 后续 agent 可以据此决定先看哪些代码
- 可以快速识别高风险模块和联看测试
- 文档不依赖读者事先熟悉仓库
```

## `injection-context` PRD

```md
# Task PRD: injection-context

## Goal
把项目知识和代码事实转换为任务型上下文包，供后续 workflow 节点按需注入。

## Target Output
- docs/contexts/<slug>/context-packs/add-endpoint.md
- docs/contexts/<slug>/context-packs/bugfix.md
- docs/contexts/<slug>/context-packs/refactor-shared.md
- docs/contexts/<slug>/context-packs/review-change.md
- docs/contexts/<slug>/injection-index.yaml

## Data Sources
- 00-summary.md
- architecture/*
- pitfalls/index.md
- code-facts/*
- layers/*
- database/* (if available)

## Extraction Rules
- 按任务组织，不按知识域组织
- 每个任务包必须包含：先读什么、关注什么、容易漏什么、联看什么测试
- injection-index.yaml 必须给出优先级

## Acceptance Criteria
- agent 能按任务类型快速装配上下文
- 注入包足够短、足够实用
- 能减少盲目扫描仓库
```

# 九、最值得先落地的改造顺序

别一次改太大，我建议按这个顺序：

## 第一批

* `code-facts-context`
* `injection-context`
* `injection-index.yaml`

## 第二批

* 增强 `README.md`
* 增强 `pitfalls/index.md`
* 增强 `module-map.md`

## 第三批

* 细化 layer 输出
* 优化 database 输出到执行视角

这样风险最低，收益最大。

# 会话启动注入 `using-superpowers` 流程说明

## 1. 结论先行

`/plugin marketplace add obra/superpowers-marketplace` 这一步本身**不是**“每次对话都进入 brainstorming”的直接原因。

真正决定行为的链路是：

1. Claude Code 安装并加载 `superpowers` 插件；
2. 插件在 `SessionStart` 时注入 `using-superpowers` 的完整内容；
3. `using-superpowers` 明确要求“先检查是否有 skill 可用”；
4. `using-superpowers` 又把 `brainstorming` 放在 process skills 的优先级最前面；
5. `brainstorming` skill 自己再把“创意/功能/行为修改”强制导向设计与澄清流程。

所以，用户看到的现象不是 marketplace 直接触发 brainstorming，而是**会话启动注入 + skill 优先级规则 + brainstorming 的硬门禁**共同造成的。

---

## 2. 整体链路

可以把这条链理解成四层：

```text
Marketplace 注册
  -> 插件安装
  -> SessionStart 注入 using-superpowers
  -> using-superpowers 要求优先检查 skill
  -> brainstorming 成为首个 process skill
```

流程图如下：

```text
+--------------------------------------------------------------------------------------+
| 触发条件                                                                            |
| 用户在 Claude Code 中完成一次会话启动、清空会话或压缩会话                            |

+-----------------------------------------+--------------------------------------------+
                                          |
                                          v
+--------------------------------------------------------------------------------------+
| Hook 注册                                                                            |
| 文件：hooks/hooks.json                                                               |

+-----------------------------------------+--------------------------------------------+
                                          |
                                          v
+--------------------------------------------------------------------------------------+
| Hook 分发                                                                            |
| 文件：hooks/run-hook.cmd                                                             |

+-----------------------------------------+--------------------------------------------+
                                          |
                                          v
+--------------------------------------------------------------------------------------+
| 注入脚本执行                                                                         |
| 文件：hooks/session-start                                                            |
```

对应到仓库里的实际实现：

- Marketplace 只是提供安装入口
- `hooks/hooks.json` 决定会不会在会话开始时运行 hook
- `hooks/session-start` 决定注入什么上下文
- `skills/using-superpowers/SKILL.md` 决定 Claude 如何理解“先用 skill”
- `skills/brainstorming/SKILL.md` 决定 brainstorming 的执行规则

## 3. 第一步：Marketplace 只负责安装，不负责流程

用户执行：

```bash
/plugin marketplace add obra/superpowers-marketplace
```

这一步的作用只是把 marketplace 注册到 Claude Code，让后续可以安装 `superpowers` 插件。

它本身不决定：

- 是否进入 brainstorming
- 是否在每次对话开头注入上下文
- 是否强制走设计流程

这些行为都发生在插件被加载之后。

## 4. 第二步：插件在 `SessionStart` 上挂了 hook

`superpowers` 插件的 hook 配置在 [hooks/hooks.json](/Users/kuang/xiaobu/superpowers/hooks/hooks.json#L1)。

这里最关键的是：

- hook 类型是 `SessionStart`
- matcher 是 `startup|clear|compact`
- 执行命令是 `hooks/run-hook.cmd session-start`

也就是说，只要是新会话启动，或者清空、压缩这类会话状态变化，Claude Code 就会触发这个 hook。

这一步决定的是：

- 什么时候注入上下文
- 注入逻辑是否只在启动时生效
- 是否覆盖到“每次新对话”

## 5. 第三步：`session-start` 把 `using-superpowers` 整段塞进上下文

真正的注入逻辑在 [hooks/session-start](/Users/kuang/xiaobu/superpowers/hooks/session-start#L1)。

这个脚本做了几件事：

1. 找到插件根目录
2. 读取 `skills/using-superpowers/SKILL.md`
3. 把整个 skill 内容做 JSON 转义
4. 拼成一段 `session_context`
5. 通过 `additionalContext` 或 `hookSpecificOutput.additionalContext` 注入到会话里

关键点在于：它不是只注入一句“请使用技能”，而是把 `using-superpowers` 的**完整内容**写进了 session context。

这意味着 Claude 一进入会话，就会看到一段很强的系统级指令：

> 你有 superpowers。下面是 `using-superpowers` 的完整内容。

这就是为什么它不是普通提示词，而是一个“会话级行为约束”。

## 6. 第四步：`using-superpowers` 规定了“先查 skill”

`using-superpowers` 的核心是“技能优先”。

在 [skills/using-superpowers/SKILL.md](/Users/kuang/xiaobu/superpowers/skills/using-superpowers/SKILL.md#L1) 里，最关键的几句是：

- 如果有哪怕 `1%` 的可能性某个 skill 适用，就必须调用它
- 在任何响应或动作前，先检查是否有 skill 可用
- `brainstorming` 属于 process skills，优先级高于 implementation skills

这会导致 Claude 在面对一个新请求时，不会先直接回答“怎么写代码”，而是先判断：

- 这个请求是不是一个要进入流程的任务
- 有没有合适的 process skill
- 如果有，先切到 brainstorming

所以 `using-superpowers` 的作用不是“直接让你 brainstorming”，而是**把“先检索、先匹配、先走流程”设成默认行为**。

## 7. 第五步：为什么最先命中的是 `brainstorming`

`using-superpowers` 里已经把 process skills 的顺序说清楚了：

1. `brainstorming`
2. `debugging`

也就是说，只要 Claude 判断当前任务属于“要开始做某件事”，它最先会想到的是 brainstorming。

而 [skills/brainstorming/SKILL.md](/Users/kuang/xiaobu/superpowers/skills/brainstorming/SKILL.md#L1) 又进一步把范围放得很宽：

- 创建功能
- 构建组件
- 添加能力
- 修改行为

这类请求都要先经过 brainstorming。

因此，用户会感觉“每次一聊就进 brainstorming”，本质上是：

- 入口层把 `using-superpowers` 注入了会话
- 总规则层把 skill 优先级提到了最高
- `brainstorming` 自己又覆盖了绝大多数“要做事”的场景

## 8. 第六步：`brainstorming` 自己又是一个硬门禁

`brainstorming` skill 不是“可选建议”，而是一个强约束流程。

它要求：

- 先了解项目上下文
- 一次只问一个澄清问题
- 提出 2-3 个方案
- 先给设计，再要用户批准
- 没有设计批准前，不能进入实现

这意味着只要 Claude 进入了 `brainstorming`，它就会天然表现为：

- 先问问题
- 先讨论方案
- 先组织设计
- 暂时不直接写实现

所以用户感知到的不是“它读到了一个 skill”，而是“它开始进入审题和设计模式”。

## 9. 这个流程里，哪些地方“决定了行为”

如果按决定权从高到低排序，可以这样看：

### 9.1 会话级注入

`hooks/session-start` 决定每个新会话拿到什么初始上下文。

这是最直接的入口层决策。

### 9.2 技能优先级规则

`using-superpowers` 决定 Claude 如何看待 skill：

- 先查 skill
- 先用 process skills
- `brainstorming` 排在最前面

这是行为策略层决策。

### 9.3 brainstorming 的流程约束

`brainstorming` 决定一旦进入该 skill，必须先设计、后实现。

这是执行层决策。

### 9.4 Marketplace

Marketplace 只决定：

- 插件从哪里装
- 版本来源是什么

它不决定会话时如何思考。

## 10. 为什么看起来像“每次都会进”

因为这个设计的目标本来就是“默认先走流程”。

只要用户的对话内容被 Claude 认为是：

- 想做一个功能
- 想改一个行为
- 想解决一个问题
- 想开始一个工程任务

那么 `using-superpowers` 就会把它归到 skill 检索路径里，而 `brainstorming` 又是最前面的 process skill，于是就形成了“几乎每次都先 brainstorming”的体感。

换句话说，这不是误触发，而是**设计本来就倾向于把新任务前置到 brainstorming**。

---

## 11. 如果要改这个行为，应该改哪里

如果你的目标是“不要每次都先 brainstorming”，优先改这三个地方：

1. `hooks/session-start`
   - 不再全量注入 `using-superpowers`
   - 改成更轻的启动提示

2. `skills/using-superpowers/SKILL.md`
   - 降低“1% 也必须用 skill”的强度
   - 调整 process skills 的优先级

3. `skills/brainstorming/SKILL.md`
   - 收窄适用范围
   - 避免把所有“想做点什么”的对话都划进 brainstorming

如果不改这三处，单改 marketplace 没有效果。

## 12. 对比表

| 维度 | Superpowers 的 `brainstorming` | spec-first 的 `brainstorm` | 核心差异 |
| --- | --- | --- | --- |
| 定位 | 通用过程技能 | 五阶段工作流的第一阶段 | 一个是行为约束，一个是项目阶段 |
| 触发方式 | 会话启动自动注入后，按 skill 规则隐式触发 | 用户显式执行 `/spec:brainstorm` | 一个默认前置，一个显式命令 |
| 入口文件 | `hooks/session-start` -> `skills/using-superpowers/SKILL.md` -> `skills/brainstorming/SKILL.md` | `templates/claude/commands/spec/brainstorm.md` -> `skills/spec-brainstorm/SKILL.md` | 一个是会话级 bootstrap，一个是命令级工作流 |
| 主要目标 | 先澄清、先设计、后实现 | 产出 requirements 文档，为 `/spec:plan` 提供输入 | 一个管思考方式，一个管需求定稿 |
| 作用范围 | 几乎覆盖所有“要做事”的请求 | 只覆盖 spec-first 的需求澄清阶段 | 一个更宽，一个更窄 |
| 输出物 | 主要是更早进入设计式对话 | `docs/brainstorms/YYYY-MM-DD-<topic>-requirements.md` | 一个无固定产物，一个有明确持久化产物 |
| 强制程度 | 高，且通过会话上下文持续施压 | 高，但仅限 `/spec:brainstorm` 工作流内 | 一个持续生效，一个局部生效 |
| 终点 | 跳转到 `writing-plans` | 跳转到 `/spec:plan` | 一个面向实现规划，一个面向项目工作流 |
| 是否要求用户显式发起 | 否，可能被自动带入 | 是，通常由用户或命令显式触发 | 一个自动路由，一个手动入口 |

## 13. 一句话总结

`obra/superpowers-marketplace` 只负责把插件装进 Claude Code；真正让 Claude 在新会话里先进入 brainstorming 的，是 `SessionStart` 注入 `using-superpowers`，再由 `using-superpowers` 的“skill 优先”规则把任务导向 `brainstorming`。

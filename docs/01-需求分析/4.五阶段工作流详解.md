# Spec-First 五阶段工作流详解

本文基于当前仓库代码与运行态资产，梳理 `brainstorm → plan → work → review → compound` 五个阶段在 `spec-first` 中的真实执行机制，包括：

- 每个阶段的入口
- 每个阶段的输入、输出、持久化产物
- 每个阶段的执行策略
- 每个阶段会调用哪些 skill / agent
- 阶段之间如何衔接

本文描述的不是理想状态，而是当前代码实际实现的行为。

## 1. 当前工作流的真实入口层

当前 `spec-first` 不是“用户直接运行 skill”，而是三层结构：

1. 用户入口层：`/spec:*` 命令
2. 工作流编排层：5 个 workflow skill
3. 执行资产层：通用 `skills/` + `agents/`

当前统一资产清单定义在 [plugin.json](/Users/kuang/xiaobu/spec-first/.claude-plugin/plugin.json)：

- commands 目录：`templates/claude/commands/spec`
- skills 目录：`skills`
- agents 目录：`agents`

其中 5 个对外命令分别映射到 5 个核心 workflow skill：

| 用户命令 | 命令模板 | 核心 workflow skill | 主要职责 |
| --- | --- | --- | --- |
| `/spec:brainstorm` | `templates/claude/commands/spec/brainstorm.md` | `skills/spec-brainstorm/SKILL.md` | 定义“要做什么” |
| `/spec:plan` | `templates/claude/commands/spec/plan.md` | `skills/spec-plan/SKILL.md` | 定义“怎么做” |
| `/spec:work` | `templates/claude/commands/spec/work.md` | `skills/spec-work/SKILL.md` | 执行计划并交付代码 |
| `/spec:review` | `templates/claude/commands/spec/review.md` | `skills/spec-review/SKILL.md` | 审查变更、路由问题、给出 verdict |
| `/spec:compound` | `templates/claude/commands/spec/compound.md` | `skills/spec-compound/SKILL.md` | 沉淀解决方案到知识库 |

这意味着：

- 用户看到的是 5 个 `/spec:*` 命令
- 真正驱动每个阶段的是对应的 workflow skill
- agent 不是顶层入口，而是被 skill 按条件调度的执行资产

## 2. 当前运行态资产规模

当前仓库初始化后会把发布资产同步到项目内 `.claude/` 目录。

以当前仓库运行态为准：

- 对外命令：5 个
- 发布 skill：41 个
- 发布 agent：47 个

按 agent 分类统计：

- `agents/research/`：6 个
- `agents/review/`：26 个
- `agents/document-review/`：7 个
- `agents/design/`：3 个
- `agents/workflow/`：4 个
- `agents/docs/`：1 个

当前五阶段工作流并不会用到全部 41 个 skill 和 47 个 agent，但这些资产构成了阶段内可调度的能力池。

### 2.1 运行态受管机制

项目运行态不是“复制完就结束”，而是受 `spec-first` 自己管理：

- `spec-first init --claude`
  - 读取 `.claude-plugin/plugin.json`
  - 同步 `commands / skills / agents`
  - 写入 `.claude/spec-first/state.json`
  - 清理上一版受管但当前已废弃的资产

- `spec-first doctor`
  - 校验 Node / Git / Claude Code
  - 校验 `.claude-plugin/plugin.json`
  - 校验 `.claude/spec-first/state.json`
  - 校验 `.claude/commands/spec`、`.claude/skills`、`.claude/agents`

- `spec-first clean --claude`
  - 只删除 `spec-first` 受管资产
  - 保留用户自定义的未受管 `.claude/skills` / `.claude/agents`

这意味着当前工作流并不是散装 markdown 集合，而是带有：

- 统一 manifest
- 受管运行态
- 版本化同步
- 项目内清理能力

的一套插件式资产模型。

## 3. 五阶段总体衔接关系

五阶段的核心产物流转如下：

```text
/spec:brainstorm
  -> docs/brainstorms/YYYY-MM-DD-<topic>-requirements.md

/spec:plan
  -> docs/plans/YYYY-MM-DD-NNN-<type>-<descriptive-name>-plan.md

/spec:work
  -> 代码 / 测试 / PR / 截图 / 可选 review 输入

/spec:review
  -> 审查报告 / 可选 autofix / 可选 .context 产物 / 可选 todo

/spec:compound
  -> docs/solutions/<category>/<filename>.md
```

这条链的设计意图是：

- `brainstorm` 负责产品定义
- `plan` 负责技术规划
- `work` 负责实现
- `review` 负责门禁与修正
- `compound` 负责知识沉淀

## 4. `/spec:brainstorm`

### 4.1 入口与驱动关系

入口模板：[brainstorm.md](/Users/kuang/xiaobu/spec-first/templates/claude/commands/spec/brainstorm.md)

该命令模板明确要求：

- 先读取 `.claude/skills/spec-brainstorm/SKILL.md`
- 将该 skill 视为 source of truth
- 将产物写入 `docs/brainstorms/YYYY-MM-DD-<topic>-requirements.md`

所以当前执行链是：

```text
/spec:brainstorm
-> commands/spec/brainstorm.md
-> skills/spec-brainstorm/SKILL.md
-> requirements 文档 / handoff 选项
```

### 4.2 输入

`brainstorm` 的输入可以来自三类来源：

1. 用户直接输入的 feature idea / problem
2. 已存在的 `docs/brainstorms/*-requirements.md`
3. 轻量 repo scan 发现的上下文约束

skill 会在 Phase 0 做三件事：

- 判断是否有现有 brainstorm 文档可续写
- 判断这次是否真的需要 brainstorm
- 判断 scope 是 Lightweight / Standard / Deep

### 4.3 执行策略

`brainstorm` 的执行策略是“交互优先、实现后置、文档为核心”。

核心特点：

- 先做轻量 repo scan
- 再做产品压力测试
- 再做一问一答式协作澄清
- 如有多个合理方向，再给 2-3 个 approach
- 最后沉淀 requirements 文档

它的默认策略不是并行 agent orchestration，而是：

- 单线程对话驱动
- 一次只问一个问题
- 尽量把产品决策在本阶段解决

### 4.4 输入产物与输出产物

**主要输入：**

- 用户问题描述
- 现有 brainstorm 文档
- 轻量 repo 上下文

**主要持久化输出：**

- `docs/brainstorms/YYYY-MM-DD-<topic>-requirements.md`

requirements 文档的目标不是技术实现，而是提供：

- Problem Frame
- Requirements
- Success Criteria
- Scope Boundaries
- Key Decisions
- Dependencies / Assumptions
- Outstanding Questions
- Next Steps

### 4.5 agents / skills 使用情况

`brainstorm` 本身没有定义“总是要启动的 agent”。

当前阶段特征是：

- 核心驱动 skill：1 个，即 `spec-brainstorm`
- always-run agents：0 个
- 并行 subagents：0 个

但在 handoff 阶段，有 1 个可选 skill：

- `document-review`

如果用户选择 “Review and refine”，则会进入 `document-review` skill，而不是在 brainstorm 主流程里自动启动 reviewer agents。

### 4.6 向下一阶段的 handoff

`brainstorm` 的 handoff 是显式的。

当 `Resolve Before Planning` 已清空时，主要下一步是：

- 推荐进入 `/spec:plan`
- 在轻量场景下允许直接进入 `/spec:work`
- 或进入 `document-review`

因此 `brainstorm` 的阶段完成标志不是“聊完了”，而是：

- requirements 文档已形成
- 关键产品决策足够稳定
- planning 不需要再发明产品行为

## 5. `/spec:plan`

### 5.1 入口与驱动关系

入口模板：[plan.md](/Users/kuang/xiaobu/spec-first/templates/claude/commands/spec/plan.md)

该命令模板明确要求：

- 先读取 `.claude/skills/spec-plan/SKILL.md`
- 用该 skill 作为 planning 的 source of truth
- 输出到 `docs/plans/YYYY-MM-DD-NNN-<type>-<descriptive-name>-plan.md`
- 本阶段不写代码

所以当前执行链是：

```text
/spec:plan
-> commands/spec/plan.md
-> skills/spec-plan/SKILL.md
-> plan 文档 / post-generation 选项
```

### 5.2 输入

`plan` 的输入主要来自四类：

1. `docs/brainstorms/*-requirements.md`
2. 用户直接给出的 feature description
3. repo 研究结果
4. 可选外部研究结果

其中 requirements 文档是优先输入源。

如果存在相关 requirements 文档，skill 要求：

- 读取完整文档
- 继承问题定义、范围边界、成功标准、关键决策
- 不允许 silently omit 上游内容

### 5.3 执行策略

`plan` 的执行策略是“研究优先、再结构化规划、最后做 confidence check”。

核心阶段是：

1. Phase 0：找上游 requirements，分类 blocker 和 plan depth
2. Phase 1：收集上下文
3. Phase 2：解决 planning questions
4. Phase 3：构造 plan 结构和 implementation units
5. Phase 4：写 plan 文档
6. Phase 5：自动做 confidence check / deepen

和 `brainstorm` 的最大区别是：

- `brainstorm` 以人机澄清为主
- `plan` 明确使用多 agent research

### 5.4 agents / skills 使用情况

`plan` 是五个阶段里最明显依赖 research agents 的阶段之一。

#### 固定 always-run agents

在 `1.1 Local Research (Always Runs)` 中，固定并行运行 2 个 agent：

- `spec-first:research:repo-research-analyst`
- `spec-first:research:learnings-researcher`

这是当前 `spec:plan` 会出现 “2 agents finished” 的根本原因。

#### 条件性 research agents

当 `1.2 Decide on External Research` 判断需要外部研究时，再并行运行 2 个 agent：

- `spec-first:research:best-practices-researcher`
- `spec-first:research:framework-docs-researcher`

#### 条件性 workflow agent

对于 Standard / Deep plan，或 user flow 还不清晰时，会再运行：

- `spec-first:workflow:spec-flow-analyzer`

#### confidence check 阶段的 targeted agents

写完 plan 后，`5.3 Confidence Check and Deepening` 还可能按 section gap 定向调度 agent。

这里的策略是：

- 每个 section 只选择最小必要 agent 集
- 每个 section 通常 1-3 个 agent
- 总数通常不超过 8 个

可被调度的 agent 来自：

- research
- review
- workflow

### 5.5 输出产物

`plan` 的主要持久化产物是：

- `docs/plans/YYYY-MM-DD-NNN-<type>-<descriptive-name>-plan.md`

这个 plan 不是执行脚本，而是决策与实施边界文档。它通常包含：

- Overview
- Problem Frame
- Requirements Trace
- Scope Boundaries
- Context & Research
- Key Technical Decisions
- Open Questions
- High-Level Technical Design
- Implementation Units
- System-Wide Impact
- Risks & Dependencies

### 5.6 向下一阶段的 handoff

`plan` 写完后，不会直接编码，而是给出 post-generation 选项：

- Open plan
- Run `document-review`
- Share to Proof
- Start `/spec:work`
- Start `/spec:work` in another session
- Create Issue

因此 `plan` 的完成标志是：

- plan 文件已写入磁盘
- confidence check 已完成或明确跳过
- 下一步执行入口已明确

## 6. `/spec:work`

### 6.1 入口与驱动关系

入口模板：[work.md](/Users/kuang/xiaobu/spec-first/templates/claude/commands/spec/work.md)

该模板要求先读取：

- `.claude/skills/spec-work/SKILL.md`

并以该 skill 作为：

- execution flow
- tasking
- verification
- handoff

的 source of truth。

### 6.2 输入

`work` 的核心输入是 plan 文档，也兼容 specification / todo file。

输入重点不是“照抄步骤”，而是读取 plan 中的：

- Implementation Units
- Requirements Trace
- Files
- Test Scenarios
- Verification
- Deferred to Implementation
- Scope Boundaries
- Execution note

### 6.3 执行策略

`work` 的执行策略是“快速澄清 → 建 task list → 选择执行模式 → 持续测试 → 交付”。

核心阶段：

1. Quick Start
2. Execute
3. Quality Check
4. Ship It

当前实现里，`work` 不是默认重 agent 化执行，而是先按 plan 拆任务，再决定三种执行策略：

- Inline
- Serial subagents
- Parallel subagents

也就是说，`work` 的默认思想是：

- 小任务内联做
- 多任务按依赖串行 subagents
- 无依赖任务再并行 subagents

### 6.4 agents / skills 使用情况

`work` 没有定义固定 always-run agent。

当前阶段的 agent 策略是“按任务规模动态调度”：

- always-run agents：0 个
- 默认 subagents：按 Implementation Units 数量和依赖关系决定
- reviewer agents：仅复杂或高风险变更时建议使用
- agent teams：只有用户明确要求 swarm mode 且平台支持时才使用

可选能力包括：

- `ce:review` / `spec:review` 作为复杂变更的质量检查
- `git-worktree` 作为工作隔离能力
- `agent-browser` / `imgup` 用于 UI 截图与 PR 材料

### 6.5 输出产物

`work` 的输出不是单一文档，而是交付物集合：

- 代码变更
- 测试变更
- 可选截图
- commit
- PR
- PR 描述中的 Post-Deploy Monitoring & Validation
- 可能更新 plan frontmatter 中的 `status: completed`

所以 `work` 是五阶段里最偏“实际交付”的节点。

### 6.6 向下一阶段的 handoff

`work` 的下游通常是 `review`。

进入 `review` 前，`work` 应至少满足：

- 任务已完成
- tests pass
- linting pass
- requirement trace 已对齐
- deferred implementation questions 已处理

## 7. `/spec:review`

### 7.1 入口与驱动关系

入口模板：[review.md](/Users/kuang/xiaobu/spec-first/templates/claude/commands/spec/review.md)

它明确委托：

- `.claude/skills/spec-review/SKILL.md`

并把 reviewer selection、severity rules、output structure 交给该 skill。

### 7.2 输入

`review` 的输入不是单一文件，而是“代码变更范围 + 意图 + 可选 plan”。

典型输入包括：

- 当前 branch 的 diff
- 指定 branch
- PR number / GitHub URL
- `base:<sha-or-ref>`
- `plan:<path>`

skill 会先建立：

- diff base
- file list
- unified diff
- untracked file scope
- intent summary
- 可选 requirements completeness 依据

### 7.3 执行策略

`review` 的执行策略是五阶段里最强的“并行审查 + 合并去重 + 路由决策”模型。

核心阶段：

1. Determine scope
2. Intent discovery
3. Select reviewers
4. Spawn sub-agents
5. Merge findings
6. Synthesize and present
7. 按 mode 决定是否 autofix / handoff

它的本质不是“读一下 diff”，而是一个有 reviewer 选择和 merge pipeline 的 orchestrator。

### 7.4 agents / skills 使用情况

`review` 是当前 agent 使用最密集的阶段。

#### Always-on reviewers：6 个

每次 review 固定运行：

- correctness-reviewer
- testing-reviewer
- maintainability-reviewer
- project-standards-reviewer
- agent-native-reviewer
- learnings-researcher

其中前 4 个偏 persona reviewer，后 2 个是 CE always-on agent。

#### Conditional reviewers：最多 11 个

根据 diff 选择：

- cross-cutting conditional：6 个
  - security
  - performance
  - api-contract
  - data-migrations
  - reliability
  - adversarial
- stack-specific conditional：5 个
  - dhh-rails
  - kieran-rails
  - kieran-python
  - kieran-typescript
  - julik-frontend-races

#### CE migration-specific conditional：2 个

- schema-drift-detector
- deployment-verification-agent

因此单次 review 的 reviewer 规模是：

- 最小：6 个
- 最大：19 个

### 7.5 输出产物

`review` 的输出分为即时报告与可选持久化产物两类。

**即时输出：**

- Findings
- Requirements Completeness
- Applied Fixes
- Residual Actionable Work
- Learnings & Past Solutions
- Agent-Native Gaps
- Schema Drift Check
- Deployment Notes
- Coverage
- Verdict

**持久化输出：**

- interactive / autofix 模式下的 `.context/spec-first/ce-review/<run-id>/`
- autofix 模式下按需创建的 todo 文件

report-only 模式不会写 `.context`，也不会写 todo。

### 7.6 向下一阶段的 handoff

`review` 的下游通常是：

- fix 后继续 review
- clean verdict 后进入 PR / merge 决策
- 或进入 `compound`

所以 `review` 的阶段职责是门禁，而不是简单建议。

## 8. `/spec:compound`

### 8.1 入口与驱动关系

入口模板：[compound.md](/Users/kuang/xiaobu/spec-first/templates/claude/commands/spec/compound.md)

它显式委托：

- `.claude/skills/spec-compound/SKILL.md`

并要求将最终知识沉淀到 `docs/solutions/`。

### 8.2 输入

`compound` 的输入通常是：

1. 刚解决的问题上下文
2. 会话中的调查与修复记录
3. 可选 brief context 参数
4. auto memory 中与问题相关的补充笔记
5. `docs/solutions/` 中已有相关文档

### 8.3 执行策略

`compound` 默认不是单线程整理，而是“先并行研究，再单线程 assembly”。

默认 Full Mode 的策略：

1. Phase 0.5：读 auto memory
2. Phase 1：3 个 subagents 并行研究
3. Phase 2：主控组装并写入唯一文档
4. Phase 2.5：判断是否建议 / 调用 `compound-refresh`
5. Phase 3：可选 specialized reviews

注意该 skill 有一个硬约束：

- 只有 orchestrator 写文件
- Phase 1 subagents 只返回 text data

### 8.4 agents / skills 使用情况

#### Full Mode always-run subagents：3 个

并行固定运行：

- Context Analyzer
- Solution Extractor
- Related Docs Finder

#### Optional specialized agents：按问题类型触发

当前 Phase 3 可选 specialized agents 包括：

- performance-oracle
- security-sentinel
- data-integrity-guardian
- cora-test-reviewer
- kieran-rails-reviewer
- code-simplicity-reviewer

其中 code-heavy issue 还可能再加两位 reviewer。

**实现现状说明：**

当前 `skills/spec-compound/SKILL.md` 确实声明了 `cora-test-reviewer`，但当前打包的 `agents/` 与运行态 `.claude/agents/` 中并没有这个 agent 文件。也就是说：

- `performance-oracle`
- `security-sentinel`
- `data-integrity-guardian`
- `kieran-rails-reviewer`
- `code-simplicity-reviewer`

当前都有对应 agent 资产；

但 `cora-test-reviewer` 目前属于 **skill 文档声明存在、实际打包资产中缺失** 的一项能力。分析当前实现时，应把它视为“声明中的可选能力”，而不是已经验证可用的运行态 agent。

需要注意一处当前实现偏差：

- `skills/spec-compound/SKILL.md` 提到了 `cora-test-reviewer`
- 但当前仓库的 `agents/` 目录中并不存在该 agent 文件

因此这条规则当前更接近“设计意图”，不是已经被发布资产完整实现的可执行能力。

#### Compact-Safe Mode

compact-safe mode 会跳过并行 subagents：

- always-run agents：0 个
- specialized review：默认跳过
- 由 orchestrator 单线程完成全部工作

### 8.5 输出产物

`compound` 的持久化产物非常明确：

- `docs/solutions/[category]/[filename].md`

如果与已有解决方案高度重叠，则不是新建，而是：

- 更新已有 doc
- 添加 `last_updated: YYYY-MM-DD`

因此 `compound` 是五阶段里最明确的知识沉淀节点。

### 8.6 向后续维护的 handoff

`compound` 的后续不是继续开发，而是知识库维护。

它可能：

- 直接结束
- 推荐 `/spec:compound-refresh`
- 标记已有 pattern / learning 文档需要刷新

## 9. 五阶段的执行策略对比

| 阶段 | 主策略 | 是否默认并行 | always-run agents | 条件 agents / subagents | 主要持久化产物 |
| --- | --- | --- | --- | --- | --- |
| brainstorm | 对话澄清 + requirements 收敛 | 否 | 0 | `document-review` 仅后续可选 | `docs/brainstorms/*-requirements.md` |
| plan | research-first + plan writing + confidence check | 是 | 2 | 外部 research 2 个、flow 1 个、deepening 最多约 8 个 | `docs/plans/*.md` |
| work | task execution + continuous testing + shipping | 视任务而定 | 0 | serial/parallel subagents、可选 reviewer agents、可选 agent teams | 代码、测试、PR、截图 |
| review | 多 persona 并行审查 + merge/dedup + verdict | 是 | 6 | 条件 reviewer 最多 13 个 | 报告、可选 `.context`、可选 todo |
| compound | 并行研究 + 单文件 assembly | 是 | 3 | specialized reviewers 按需触发；compact 模式为 0 | `docs/solutions/**/*.md` |

## 10. 各阶段输入/输出的真实边界

从当前代码看，五阶段边界很明确：

- `brainstorm` 不负责技术实现
- `plan` 不负责写代码
- `work` 不负责定义产品目标
- `review` 不负责重新设计需求
- `compound` 不负责继续编码

这也是当前工作流最核心的设计原则：

**每个阶段都围绕一个主要问题展开，并把可复用的信息写成持久化产物，供下一阶段继续消费。**

## 11. 当前项目里 skill 与 agent 的真实层级关系

五阶段工作流已经体现出明确的层级关系：

```text
用户 /spec:* 命令
  -> command template
  -> workflow skill
  -> 通用 skill / agents / subagents
  -> 产物写盘
```

职责分工是：

- command template：稳定用户入口
- workflow skill：阶段编排器
- agents：专业化子任务执行器
- 持久化文档：跨阶段上下文载体

因此，当前 `spec-first` 的真实模型不是“用户直接调 agent”，而是：

**用户通过 `/spec:*` 进入，workflow skill 负责编排，agent 负责被按需调度执行。**

## 12. 当前实现中的命名与资产残留

当前代码已经完成了 `/spec:*` 命令入口迁移，但 skill 文本里仍然保留了一部分历史命名。

典型表现：

- `spec-plan` 中仍多次写 `ce:brainstorm`
- `spec-work` 中仍写 “Load the `ce:review` skill”
- `spec-compound` 中仍写 `ce:compound-refresh`

这说明当前项目的真实状态是：

- **入口层** 已经迁移到 `spec:*`
- **部分 skill 文案** 仍保留 compound-engineering 时代的 `ce:*` 历史残留

这类残留暂时不会阻断核心 workflow 理解，因为命令模板已经明确把 `/spec:*` 委托到对应 workflow skill。
但如果后续继续做 plugin-first、多平台适配或自动化编排，这些命名残留应继续清理，否则会让：

- 用户入口名称
- skill 内部描述
- 文档中的工作流名称

三者出现不必要的歧义。

## 13. 结论

当前 `spec-first` 五阶段工作流已经形成稳定的阶段闭环：

- `brainstorm` 产出 requirements
- `plan` 产出 implementation plan
- `work` 产出代码与交付物
- `review` 产出审查 verdict 与修复路由
- `compound` 产出知识文档

其中最关键的实现特征是：

1. 5 个 `/spec:*` 命令只是入口层
2. 真正驱动流程的是 5 个 workflow skill
3. 不同阶段的 agent 使用强度不同：
   - brainstorm 最轻
   - plan 中等偏强
   - work 动态调度
   - review 最强
   - compound 中等且结构固定
4. 每个阶段都要求把关键结果写成持久化产物，而不是只留在对话上下文里

如果后续继续做 plugin-first / 多平台扩展，这份分层关系应保持不变：

- 对用户暴露稳定命令入口
- 对内部保持 skill 编排
- 对复杂子任务使用 agent 资产池
- 对跨阶段状态使用文档产物承载

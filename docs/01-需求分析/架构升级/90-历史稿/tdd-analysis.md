# TDD 分析：为何 Implement 不使用测试驱动开发

> 本文档分析 spec-first 项目中 implement agent 为何不采用 TDD 模式，以及改进建议

---

## 1. 当前设计分析

### 1.1 现有流程

```
┌─────────────────────────────────────────────────────────────────────────────────┐
│                        当前 implement 流程                                       │
└─────────────────────────────────────────────────────────────────────────────────┘

  implement → check (循环) → finish
      │            │            │
      │            │            │
      ▼            ▼            ▼
  实现 + lint    spec验证    最终检查
  typecheck     (最多5次)
```

### 1.2 隐含假设

| 假设 | 说明 | 问题 |
|------|------|------|
| lint + typecheck 足够 | 认为静态检查能发现大部分问题 | ❌ 无法发现逻辑错误 |
| check agent 会验证 | spec 验证覆盖功能正确性 | ⚠️ 依赖 spec 质量 |
| 人工 review 兜底 | 最终由人审核 | ✅ 有效但效率低 |

### 1.3 缺失的验证层

```
┌─────────────────────────────────────────────────────────────────────────────────┐
│                        验证层级对比                                              │
└─────────────────────────────────────────────────────────────────────────────────┘

  当前有:
  ✅ Lint 检查 (eslint)
  ✅ TypeCheck (typescript)
  ✅ Spec 验证 (check agent)
  ✅ 人工 Review

  当前缺失:
  ❌ 单元测试 (unit test)
  ❌ 集成测试 (integration test)
  ❌ 功能测试 (e2e test)
```

## 2. 为何不用 TDD？

### 2.1 AI 编码的特殊性

```
┌─────────────────────────────────────────────────────────────────────────────────┐
│                        传统 TDD vs AI 编码                                       │
└─────────────────────────────────────────────────────────────────────────────────┘

  传统开发 (TDD 有效):
  ┌──────────┐    ┌──────────┐    ┌──────────┐
  │ 1. 写测试 │ ──►│ 2. 写实现 │ ──►│ 3. 重构  │
  │   (Red)  │    │  (Green) │    │(Refactor)│
  └──────────┘    └──────────┘    └──────────┘
       │
       └── 测试由人编写，独立于实现思维


  AI 编码 (TDD 价值降低):
  ┌──────────┐    ┌──────────┐
  │ AI 生成  │ ──►│ 同一模型  │
  │ 测试代码 │    │ 生成实现  │
  └──────────┘    └──────────┘
       │               │
       └───────┬───────┘
               │
               ▼
        测试和实现同源
        可能有相同的理解错误
```

### 2.2 同源偏见问题

| 问题 | 说明 | 示例 |
|------|------|------|
| **理解偏差** | AI 对需求的误解会同时影响测试和实现 | 需求说"升序"，测试和实现都写成"降序" |
| **边界遗漏** | AI 没考虑的边界情况，测试也不会覆盖 | 空数组、null、超大数 |
| **假设一致** | 测试和实现可能基于相同的错误假设 | 都假设输入是有效 JSON |

### 2.3 时间成本

```
当前流程时间:
  implement: 30 min (实现 + lint/typecheck)
  check:     15 min (spec 验证)
  ─────────────────
  总计:       45 min

TDD 流程时间:
  test-design: 15 min (设计测试用例)
  implement:   30 min (实现功能)
  test-run:    10 min (运行测试)
  check:       15 min (spec 验证)
  ─────────────────
  总计:         70 min

时间增加: 56%
```

### 2.4 项目特点

| 特点 | 影响 |
|------|------|
| **CLI 工具** | 更依赖命令行测试而非单元测试 |
| **模板生成** | 生成代码的正确性难以用单元测试验证 |
| **多平台支持** | 8 个平台的差异使测试复杂化 |

## 3. 是否应该加 TDD？

### 3.1 适合加 TDD 的场景

```
┌─────────────────────────────────────────────────────────────────────────────────┐
│                        适合 TDD 的场景                                           │
└─────────────────────────────────────────────────────────────────────────────────┘

  ✅ 纯逻辑功能 (无副作用)
     └── 工具函数、算法、数据处理

  ✅ API 契约验证
     └── 请求/响应格式、错误码

  ✅ 边界条件多的功能
     └── 输入验证、状态转换

  ✅ 回归风险高的模块
     └── 核心业务逻辑
```

### 3.2 不适合 TDD 的场景

```
┌─────────────────────────────────────────────────────────────────────────────────┐
│                        不适合 TDD 的场景                                         │
└─────────────────────────────────────────────────────────────────────────────────┘

  ❌ UI 组件
     └── 视觉效果难以用单元测试验证

  ❌ 文件系统操作
     └── Mock 成本高，集成测试更有效

  ❌ 快速原型
     └── 测试会拖慢迭代速度

  ❌ 探索性开发
     └── 需求不确定，测试要反复修改
```

## 4. 改进建议

### 4.1 方案 A: 让 Implement 生成测试

**修改 `implement.md`:**

```markdown
### 3. Implement Features

- Write code following specs and technical design
- **Write corresponding unit tests**  ← 新增
- Follow existing code patterns
- Only do what's required, no over-engineering

### 4. Verify

Run project's lint, typecheck, **and test** commands to verify changes.
                                    ↑ 新增
```

**优点:**
- 简单，不需要修改架构
- 测试和实现一起生成，保持一致

**缺点:**
- 同源偏见问题依然存在
- 增加 implement 阶段时间

---

### 4.2 方案 B: 添加独立的 Test Phase

**修改 dispatch 流程:**

```
当前: implement → check → finish

改进: implement → test → check → finish
              ↑
           新增阶段
```

**新增 `test.md` agent:**

```yaml
---
name: test
description: |
  Test runner and validator. Runs tests and reports failures.
  Does NOT write code, only runs and validates.
tools: Read, Bash, Glob, Grep
model: opus
---
```

**优点:**
- 职责分离，更清晰
- 可以独立运行测试验证

**缺点:**
- 增加系统复杂度
- 需要新增 agent 定义

---

### 4.3 方案 C: 在 Check 中包含测试 (推荐)

**修改 `ralph-loop.py`:**

```python
# 当前
def get_verify_commands(repo_root: str) -> list[str]:
    # 从 worktree.yaml 读取
    ...

# 改进: 默认包含测试
DEFAULT_VERIFY_COMMANDS = ["pnpm lint", "pnpm typecheck", "pnpm test"]

def get_verify_commands(repo_root: str) -> list[str]:
    commands = read_from_worktree_yaml(repo_root)
    if not commands:
        return DEFAULT_VERIFY_COMMANDS
    return commands
```

**修改 `worktree.yaml` 模板:**

```yaml
verify:
  - pnpm lint
  - pnpm typecheck
  - pnpm test  # 新增
```

**优点:**
- 最小改动
- 利用现有循环机制
- 可配置，项目可自定义

**缺点:**
- 测试失败时没有专门的修复指导

---

### 4.4 方案 D: 分层测试策略

```
┌─────────────────────────────────────────────────────────────────────────────────┐
│                        分层测试策略                                              │
└─────────────────────────────────────────────────────────────────────────────────┘

  Layer 1: 静态检查 (当前有)
  ├── Lint (eslint)
  └── TypeCheck (typescript)

  Layer 2: 单元测试 (建议加)
  ├── 纯函数测试
  └── 工具函数测试

  Layer 3: 集成测试 (可选)
  ├── CLI 命令测试
  └── 文件生成测试

  Layer 4: E2E 测试 (可选)
  ├── 完整工作流测试
  └── 多平台兼容性测试


  执行策略:
  ┌─────────────────────────────────────────────────────────────┐
  │  implement: Layer 1 (快速反馈)                              │
  │  check:     Layer 1 + Layer 2 (质量保证)                    │
  │  finish:    Layer 1 + Layer 2 + Layer 3 (完整验证)          │
  └─────────────────────────────────────────────────────────────┘
```

## 5. 推荐方案

### 5.1 短期 (P0)

采用 **方案 C**: 在 Check 中包含测试

```diff
# ralph-loop.py
- # 默认不运行测试
+ DEFAULT_VERIFY_COMMANDS = ["pnpm lint", "pnpm typecheck", "pnpm test"]
```

### 5.2 中期 (P1)

采用 **方案 A**: 让 implement 生成测试

```diff
# implement.md
  ### 3. Implement Features
  - Write code following specs and technical design
+ - Write corresponding unit tests
```

### 5.3 长期 (P2)

采用 **方案 D**: 分层测试策略

- 定义清晰的测试层级
- 不同 phase 运行不同层级的测试
- 建立 CI/CD 自动化测试流程

## 6. 总结

### 6.1 为何不用 TDD (当前)

| 原因 | 权重 |
|------|------|
| AI 编码的同源偏见 | 高 |
| 时间成本增加 | 中 |
| 项目特点 (CLI 工具) | 中 |
| 现有验证机制足够 | 低 |

### 6.2 建议改进

| 优先级 | 方案 | 改动量 | 效果 |
|--------|------|--------|------|
| P0 | Check 包含测试 | 小 | 中 |
| P1 | Implement 生成测试 | 中 | 高 |
| P2 | 分层测试策略 | 大 | 高 |

### 6.3 核心权衡

```
         速度 ◄──────────────────────────────► 质量
           │                                    │
  当前设计  ●────────────────────────────────────┤
           │                                    │
  加 TDD    ├────────────────────────────────────●
           │                                    │
           └────────────────────────────────────┘
                    增加 ~25% 时间
                    提升 ~40% 质量
```

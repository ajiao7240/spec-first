## 2. 总体目标

这轮方案的核心目标有七条：

1. 建立统一的 `decision_hints` 结构化契约
2. 将决策生产前移到 `task creation / plan`
3. 将决策执行下沉到 runtime hooks
4. 保持 `dispatch` 纯调度，不承担复杂策略推导
5. 让 check 阶段测试要求成为硬门控，而不是软提示
6. 保证多平台语义一致
7. 保证旧任务可回退，不要求破坏式迁移

一句话总结：

> 用最小结构化 contract 固定关键边界，用 runtime 执行关键 gate，把 LLM 自主性保留在 phase 执行层。

---

## 3. 问题定义

当前系统已经具备多 Agent 流水线，但仍有三个核心缺口：

1. 关键策略缺少结构化表达
   - TDD 强度
   - check 跑哪些命令
   - 是否需要跨层检查
   - bugfix 的修复策略

2. 关键约束缺少 runtime enforcement
   - prompt 可以提醒
   - 文档可以建议
   - 但没有进入 gate，就不构成工程约束

3. workflow、policy、runtime、agent 自主性边界不够清晰
   - 容易把复杂度重新堆回 `dispatch`
   - 容易把 schema 设计成第二份 PRD
   - 容易把主 workflow 扩张成 phase engine

因此，这一轮方案不是要“再造一个中央控制系统”，而是要：

> 建立一个最小但稳定的 Harness Engineering 骨架。

## 4. 总体设计

## 4.1 设计原则

### 1. `dispatch` 保持纯调度

`dispatch` 只负责：

- 读取 `next_action`
- 启动对应 phase

`dispatch` 不负责：

- 推导 TDD 策略
- 推导测试命令
- 推导 debug 策略
- 成为 gate engine

### 2. `next_action` 只负责 workflow topology

`next_action` 只表达：

- 有哪些 phase
- phase 顺序如何

它不表达：

- phase 的策略
- phase 的验证规则
- phase 的内部执行方式

### 3. `decision_hints` 只负责最小 phase policy

`decision_hints` 负责表达最关键、最稳定、最需要跨平台一致的策略：

- implement 的模式强度
- check 的验证命令

它不应该承载：

- 全量业务语义
- 详细实现步骤
- 完整 prompt 文本
- 所有局部判断

### 4. runtime hooks 负责 enforcement

关键 gate 必须下沉到 runtime：

- `inject-subagent-context.py` 负责注入 phase policy
- `ralph-loop.py` 负责执行 check gate

### 5. LLM 自主性只保留在执行层

LLM 的自主决策主要存在于：

- `plan`
- `implement`
- `check`
- `debug`
- `finish`

不应主要存在于：

- `task.json`
- `dispatch`
- runtime hooks

### 6. 不过度设计

本方案明确拒绝以下方向：

- 完整 workflow DSL
- 完整 gate engine
- 过大的 `decision_hints` schema
- 在没有 runtime consumer 前先膨胀 phase/action 语义

## 5. 终局架构

## 5.1 四层边界

终局架构应收敛为四层。

### Layer 1: Workflow Topology

载体：

- `task.json.next_action`
- `workflow_type`（仅作为 task creation / plan 阶段的输入标签）

责任：

- 定义 phase 存在性
- 定义 phase 顺序

约束：

- 运行时唯一执行输入是 `task.json.next_action`
- `workflow_type` 不应成为第二份运行时 workflow truth
- 如果保留 `workflow_type`，它必须在 task creation / plan 阶段被编译为 `next_action`

### Layer 2: Phase Policy

载体：

- `task.json.decision_hints`

责任：

- 定义各 phase 的最小关键约束

生产者：

- `task creation`
- `plan`

约束：

- `decision_hints` 必须由明确的 producer 写入
- 不允许主要依赖 runtime 或 dispatch 现场猜测默认值

### Layer 3: Runtime Enforcement

载体：

- `inject-subagent-context.py`
- `ralph-loop.py`
- 各平台对应 runtime

责任：

- 注入 policy
- 执行 gate
- 阻止绕过

### Layer 4: LLM Autonomy

载体：

- `plan`
- `implement`
- `check`
- `debug`
- `finish`

责任：

- 在边界内做语义理解
- 决定具体实现路径
- 做局部 tradeoff
- 完成具体执行

## 5.3 每层只回答一个问题

| 层 | 回答的问题 | 不回答的问题 |
|----|------------|--------------|
| Workflow Topology | 任务经过哪些 phase | 每个 phase 怎么做 |
| Phase Policy | 每个 phase 的关键约束是什么 | 代码如何实现 |
| Runtime Enforcement | 哪些约束必须被系统执行 | 业务设计是否合理 |
| LLM Autonomy | 在边界内如何最优完成任务 | 系统边界如何定义 |

## 5.4 优化前后的流程对比

### 优化前

```text
用户需求
   │
   ▼
task_store.py
   │
   └── 写死 next_action
         implement -> check -> finish -> create-pr
   │
   ▼
dispatch
   │
   └── 按固定 phase 调度
   │
   ▼
implement / check
   │
   ├── 关键策略主要靠 prompt / 文档暗示
   ├── TDD 是否采用由 agent 临场判断
   └── check gate 主要依赖 worktree.yaml / markers
```

特点：

- `next_action` 固定写死
- 没有 task-level policy contract
- runtime 不知道 implement/check 的任务级策略
- hard gate 与 phase policy 没有统一

### 优化后

```text
用户需求
   │
   ▼
task creation / plan
   │
   ├── 规范化用户输入（workflow / preset / alias）
   ├── 生成 workflow_type（创建期标签）
   ├── 编译出 next_action（运行时唯一 topology truth）
   └── 写入 decision_hints（最小 phase policy contract）
   │
   ▼
dispatch
   │
   └── 只按 next_action 调度
   │
   ▼
runtime hooks
   │
   ├── inject-subagent-context.py 注入 policy
   └── ralph-loop.py 执行 verify gate
   │
   ▼
implement / check / debug / finish
   │
   └── 在既定边界内自主执行
```

特点：

- topology、policy、runtime、autonomy 四层分离
- policy 有 producer，runtime 有 consumer
- verify gate 与 task-level contract 对齐
- `dispatch` 保持纯调度

## 6. 关键数据模型

第一阶段的 `decision_hints` 只建议落最小 schema：

```json
{
  "decision_hints": {
    "implement": {
      "mode": "tdd_recommended"
    },
    "check": {
      "verify_commands": ["pnpm lint", "pnpm typecheck", "pnpm test"]
    }
  }
}
```

推荐原因：

1. 已覆盖 implement 的策略强度
2. 已覆盖 check 的硬门控命令
3. 字段更少，便于跨平台一致
4. runtime 有明确消费方

关键前提：

1. `task_store.py` 必须为新任务写入默认 `decision_hints`
2. `plan` 可以按任务类型覆写默认值
3. runtime 只消费，不负责成为默认值生产者
4. 文中 `verify_commands` 仅为示例，真实命令必须由 producer 按任务/项目写入 task

扩展字段如：

- `implement.test_layers`
- `implement.must_show_red_green`
- `check.cross_layer_required`
- `debug.fix_strategy`
- `finish.update_spec_expected`

应作为第二阶段或第三阶段增强，而不是第一阶段的硬要求。

## 7. 主 workflow 与可选能力的边界

主 workflow 只保留最稳定的骨架。

### 7.1 主 workflow

推荐第一阶段只保留三种 topology：

1. `default`
   - `implement -> check -> finish -> create-pr`

2. `quick-fix`
   - `implement -> check -> create-pr`

3. `docs-only`
   - `implement -> check`

补充约束：

- `workflow_type` 只用于创建阶段表达意图
- 创建完成后必须生成对应的 `next_action`
- `dispatch` 运行时只读取 `next_action`
- 不允许 `workflow_type` 与 `next_action` 长期并行充当两份事实来源

### 7.2 不进入主 workflow 的能力

以下能力不建议第一阶段进入主 workflow：

1. `with-review`
2. `research`
3. 独立 `tdd` phase
4. 独立 `debug-systematic` phase

原因不是单一的“为了复用”，而是同时为了：

1. 保持主 workflow 稳定
2. 避免把能力误建模成固定 phase
3. 提高跨场景复用性
4. 避免过早承诺 runtime 语义

更准确的建模方式是：

- 主 workflow = 拓扑骨架
- `review / research / tdd / debug` = 可选能力或策略预设

## 8. 三阶段实施拆分

## 8.1 Phase 1：数据模型落地与最小运行时接入

### 目标

先把最关键的闭环建立起来：

1. 新任务能稳定写出结构化 `workflow_type + next_action + decision_hints`
2. runtime 开始消费最小 phase policy
3. check gate 开始基于 task-level contract 工作

### 范围

#### Step 1: 新建 `workflow.py`

职责：

- 定义最小 workflow topology
- 提供从 `workflow_type` 到 `next_action` 的编译能力

建议提供：

- `DEFAULT_WORKFLOWS`
- `get_next_action(workflow_type)`
- `list_workflows()`

Phase 1 只支持 3 个 workflow：

- `default`
- `quick-fix`
- `docs-only`

约束：

- `workflow_type` 只是创建期标签
- `next_action` 是 dispatch 的唯一运行时输入
- `workflow_type` 不参与运行时决策
- 不保证手工编辑 `next_action` 后自动回写或同步 `workflow_type`

#### Step 2: 新建 `decision_hints.py`

职责：

- 定义最小 policy defaults
- 提供默认 `decision_hints` 生成函数

建议提供：

- `DECISION_HINTS_DEFAULTS`
- `get_default_decision_hints(workflow_type, repo_root, task_context=None)`
- `validate_decision_hints()`

Phase 1 只支持最小字段：

- `implement.mode`
- `check.verify_commands`

约束：

- 不在 Phase 1 依赖 runtime 推断 `verify_commands`
- 默认值由代码显式生成，不由 hook 或 dispatch 现场猜测
- `verify_commands` 是 task/repo-specific contract，不是固定的 `pnpm` 三连
- `verify_commands` 的默认生成应显式继承 repo 约定

推荐优先级：

1. task-level override
2. repo-level explicit verify config
3. producer-time deterministic repo-aware default
4. completion markers fallback

#### Step 3: 修改 `task_store.py`

这是 Phase 1 的核心改造点。

需要完成：

- `cmd_create` 增加 `--workflow` 参数
- 默认 `workflow_type = default`
- 根据 `workflow_type` 生成 `next_action`
- 写入 `workflow_type`
- 写入默认 `decision_hints`

producer ownership 明确如下：

- `task_store.py` 为新任务写入默认 topology 和 policy
- `plan` 只负责后续最小覆写，不负责补齐缺失默认值

#### Step 4: 修改 `task.py`

需要完成：

- `create` 子命令支持 `--workflow`
- 参数校验只允许 3 个值：
  - `default`
  - `quick-fix`
  - `docs-only`

#### Step 5: 修改 `ralph-loop.py`

需要完成：

- 优先读取 `task.json.decision_hints.check.verify_commands`
- 若 task-level contract 不存在，再 fallback：
  - `.spec-first/worktree.yaml verify`
  - completion markers

这一步是 check gate runtime 化的关键。

#### Step 6: 修改 `inject-subagent-context.py`

需要完成：

- 读取最小 `decision_hints`
- 注入 implement/check prompt

最小注入目标：

- implement 至少看到 `implement.mode`
- check 至少看到 `check.verify_commands`

#### Step 7: 验证

至少验证三件事：

1. 创建任务时能写出新结构

```bash
python3 ./.spec-first/scripts/task.py create "Test" --workflow quick-fix
```

2. 新建 `task.json` 包含：

- `workflow_type`
- `next_action`
- `decision_hints`

3. check gate 优先消费 task-level `verify_commands`

### Phase 1 成功标准

1. `dispatch` 仍然纯调度
2. 新任务创建时即可产出最小 topology + policy
3. check gate 开始真正执行 task-level verify commands
4. 旧任务无需破坏式迁移

Phase 1 完成后，系统必须满足以下闭环：

1. task creation 能稳定产出默认 `decision_hints`
2. `workflow_type` 不会成为悬空元数据
3. runtime 能消费 task-level verify contract
4. 旧任务仍可通过 fallback 路径工作

### Phase 1 不做

1. 不支持 7 个 workflow_type
2. 不引入完整 12 字段 schema
3. 不引入 runtime 推断 `verify_commands`
4. 不在 Phase 1 引入 `check.cross_layer_required`
5. 不新增 `review` / `research` 主 workflow
6. 不引入完整 workflow DSL

## 8.2 Phase 2：高价值能力快速集成

### 目标

在不破坏四层边界的前提下，快速增强最有价值的常见场景。

### 原则

第二阶段做“预设策略集成”，不做“新增 phase 引擎”。

Phase 2 的核心不是扩 topology，而是：

- 在既有 `workflow_type` 之上增加更强的 policy preset
- 继续复用 `next_action + decision_hints` 模型
- 不新增新的运行时 workflow truth

输入分层原则：

- `--workflow` 只负责 topology
- `--preset` 只负责 policy enhancement

推荐第一批 `preset`：

- `with-tdd`
- `debug`

### 优先支持

#### 1. `with-tdd`

推荐实现：

- topology 仍使用 `default`
- 通过 `decision_hints` 增强 implement/check policy

示例：

```json
{
  "workflow_type": "default",
  "decision_hints": {
    "implement": {
      "mode": "tdd_required",
      "must_show_red_green": true
    },
    "check": {
      "verify_commands": ["pnpm lint", "pnpm typecheck", "pnpm test"]
    }
  }
}
```

可执行落地建议：

##### Step 1: 在 `decision_hints.py` 增加 TDD preset

建议提供：

- `get_tdd_preset()`
- 或在 `get_default_decision_hints()` 中支持 `preset=with-tdd`

##### Step 2: 在 `task_store.py` 或 `plan` 中支持 `--preset`

推荐方式：

- 用户输入 `--preset with-tdd`
- 内部编译为：
  - `workflow_type = default`
  - 更强的 `decision_hints.implement/check`

##### Step 3: 在 `inject-subagent-context.py` 中补强 implement 注入

至少注入：

- `implement.mode = tdd_required`
- `must_show_red_green = true`

##### Step 4: 验证

至少验证：

1. 新任务能生成 TDD 风格 policy
2. implement prompt 能看到 TDD 要求
3. check gate 仍基于 `verify_commands` 工作

#### 2. `debug`

推荐实现：

- topology 使用 `quick-fix` 或 `default`
- 通过 `decision_hints` 增强 bugfix policy

示例：

```json
{
  "workflow_type": "quick-fix",
  "decision_hints": {
    "implement": {
      "mode": "tdd_required"
    },
    "debug": {
      "fix_strategy": "reproduce_then_fix",
      "analysis_depth": "deep",
      "prefer_test_reproduction": true
    },
    "check": {
      "verify_commands": ["pnpm lint", "pnpm typecheck", "pnpm test"]
    }
  }
}
```

接入前提：

1. `inject-subagent-context.py` 已具备对 `decision_hints.debug.*` 的消费能力
2. debug policy 能被稳定注入到 debug 执行上下文
3. 至少已有一条清晰的 bugfix policy 路径，而不是只新增字段不消费

可执行落地建议：

##### Step 1: 在 `decision_hints.py` 增加 debug preset

建议提供：

- `get_debug_preset()`
- 或在 `get_default_decision_hints()` 中支持 `preset=debug`

##### Step 2: 在 `inject-subagent-context.py` 增加 debug policy 注入

至少注入：

- `fix_strategy`
- `analysis_depth`
- `prefer_test_reproduction`

##### Step 3: 明确 debug 的入口不是新 workflow phase

内部编译方式应保持：

- `workflow_type = quick-fix` 或 `default`
- 通过 `decision_hints.debug.*` 表达修复策略

##### Step 4: 验证

至少验证：

1. debug preset 能生成稳定字段
2. debug agent 实际能收到 policy
3. 不新增 `debug-systematic` phase

### 不建议在 Phase 2 做的事

1. 不新增 `tdd` 独立 phase
2. 不新增 `review` 独立 phase
3. 不新增 `research` 主 workflow phase
4. 不引入完整 workflow DSL

### Phase 2 成功标准

1. `with-tdd` 可稳定使用
2. `debug` 在补齐 policy consumer 后可使用
3. 不膨胀 `next_action` 的 action 语义
4. 第二阶段能力通过 preset/policy 映射，而不是 phase engine 扩张

### Phase 2 不做

1. 不把 `with-tdd` 升级成独立 workflow topology
2. 不把 `debug` 升级成独立 workflow topology
3. 不把 `with-review` 纳入主 workflow
4. 不把 `research` 纳入主 workflow
5. 不让用户输入别名直接变成第二份运行时 truth

### 推荐输入与内部编译关系

为兼顾用户体验和内部收口，推荐采用如下映射：

| 用户可见名称 | 内部编译结果 |
|--------------|--------------|
| `--preset with-tdd` | `workflow_type=default` + TDD preset |
| `--preset debug` | `workflow_type=quick-fix` 或 `default` + debug preset |
| `--preset with-review` | 暂不支持主 workflow 集成 |
| `--preset research` | 不作为主 workflow preset，保留为 plan / brainstorm 前置能力 |

补充说明：

- `with-tdd` 的成熟度高于 `debug`
- `debug` 只有在补齐 policy consumer 后才进入“可快速使用”范围

---

## 8.3 Phase 3：谨慎扩展与平台收口

### 目标

在 Phase 1 和 Phase 2 已稳定运行后，再考虑更高阶能力和多平台收口。

### 原则

Phase 3 不是“继续加字段”，而是：

- 只在已有 consumer 的前提下扩 schema
- 只在已有稳定 runtime 的前提下扩能力
- 只在语义已经稳定时做多平台收口

### 可考虑方向

#### 1. 扩展 schema，但只在有 runtime consumer 时扩

例如：

- `implement.test_layers`
- `debug.analysis_depth`
- `finish.update_spec_expected`

扩展前必须满足：

1. 字段语义稳定
2. 有明确生产者
3. 有明确消费者
4. 不会把 schema 变成巨型控制面

可执行落地建议：

##### Step 1: 建立字段准入清单

任何新字段进入主 schema 前，必须回答四个问题：

1. 谁生产它
2. 谁消费它
3. 不写它时 fallback 是什么
4. 它是否真的属于最小 contract，而不是开放上下文

##### Step 2: 先实现 consumer，再写入默认值

顺序必须是：

1. consumer ready
2. 字段语义冻结
3. 写入默认值
4. 才允许进入主 schema

##### Step 3: 为新增字段补验证样例

至少包括：

- 有该字段的 task 示例
- 无该字段的 fallback 示例
- runtime 实际消费示例

#### 2. 评估 `with-review`

只有在以下条件满足后才进入主方案：

1. review 输出格式稳定
2. review 的阻断/通过状态明确
3. runtime 能消费 review 结果

可执行落地建议：

##### Step 1: 明确 review contract

至少定义：

- review 输入来源
- review 输出结构
- review 的 pass / block 状态

##### Step 2: 明确 runtime consumer

至少回答：

- 是由 `dispatch` 调用，还是由 hook 注入
- review 结果如何影响后续 phase
- review 失败是否阻断 create-pr

##### Step 3: 小范围试点

先在单一平台或单一任务类型中试点，不直接进入主 workflow。

#### 3. 评估 `research`

`research` 更可能作为：

- `plan` 前置动作
- `brainstorm` 的能力增强

而不是主 workflow phase。

可执行落地建议：

##### Step 1: 明确 research 的位置

先只允许两种形态：

- plan 前置探索
- brainstorm 的辅助能力

##### Step 2: 禁止直接提升为主 workflow phase

除非同时满足：

1. 输入输出稳定
2. 有明确完成条件
3. 有 runtime consumer

否则 research 只能保持旁路能力。

#### 4. 多平台模板收口

将核心语义统一到：

- `workflow_type`
- `decision_hints`
- fallback 顺序
- `verify_commands` 的 enforcement 逻辑

同时保持一条原则：

- 多平台共享的是语义模型，不是重复维护两套运行时 truth

可执行落地建议：

##### Step 1: 先统一语义，再统一实现

优先统一：

- `workflow_type`
- `decision_hints`
- fallback 顺序
- `verify_commands` 语义

不要求第一时间统一：

- 所有平台的 prompt 文本
- 所有平台的 hook 代码细节

##### Step 2: 建立平台对照表

至少列出：

| 平台 | workflow producer | policy consumer | gate consumer |
|------|-------------------|-----------------|---------------|
| Claude | task_store / plan | inject hook | ralph-loop |
| iFlow | 待定义 | 待定义 | 待定义 |
| OpenCode | 待定义 | plugin/runtime | 待定义 |

##### Step 3: 逐平台接入，不做一次性全平台重写

先接入已有 hook/runtime 能力的平台，再处理其余平台。

### Phase 3 成功标准

1. 扩展能力不破坏四层边界
2. 新字段先有 consumer，再进 schema
3. 多平台共享同一语义模型

### Phase 3 不做

1. 不为了“完整”而提前引入无 consumer 字段
2. 不为了“统一”而强推所有平台同步重写
3. 不把 research/review 在语义未稳定前纳入主 workflow
4. 不把开放上下文错误塞进 `decision_hints`

---

## 9. 为什么是三阶段，而不是一次做完

如果一次做完整方案，最容易出现四类问题：

1. `dispatch` 重新变重
2. schema 膨胀成第二份 PRD
3. workflow action/phase 语义提前膨胀
4. 没有 runtime consumer 的概念先进入主系统

三阶段拆分的价值在于：

1. 先建立最小闭环
2. 再接高价值场景
3. 最后才做谨慎扩展

这比“一次性设计完整 workflow engine”更符合当前项目目标。

---

## 10. 最终结论

整体方案可以收敛成四句话：

1. `next_action` 只管 workflow topology
2. `decision_hints` 只管最小 phase policy
3. runtime hooks 执行最关键 gate
4. LLM 自主性保留在 phase execution layer

三阶段路线则可以收敛成三句话：

1. **Phase 1**：先做最小可行闭环
2. **Phase 2**：再做高价值 preset 集成
3. **Phase 3**：最后谨慎扩展 schema、能力和多平台收口

一句话总结全文：

> 这不是要把 spec-first 做成一个更重的中央控制系统，而是要把它收敛成一个“拓扑清晰、契约最小、gate 可执行、执行层仍保留智能”的工程化 Harness。

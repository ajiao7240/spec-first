## 2. 核心结论

当前方案不应继续沿着“更复杂的 workflow template + 更复杂的 hook”演进，而应收敛成：

> **Spec / Policy -> Task Compiler -> Task Contract -> Runtime Enforcement -> Evidence**

也就是说：

1. 需求、工程规则、工作流能力是上游输入
2. `task.json` 是编译后的运行时 contract，不是上游真相本身
3. `dispatch` 只调度，不做策略推导
4. hook 只消费 contract，不生成核心决策
5. 最终判定任务完成，应该越来越依赖 evidence，而不是 agent 声称

## 2.1 设计原则

为了避免系统失控，必须遵守以下 6 条原则：

1. **dispatch 保持纯调度** - 只读取 next_action 并启动 phase，不做策略推导
2. **next_action 只负责 topology** - 只表达 phase 顺序，不表达 phase 策略
3. **decision_hints 只负责最小 policy** - 只表达最关键、最稳定的策略
4. **runtime hooks 负责 enforcement** - 关键 gate 必须下沉到 runtime
5. **LLM 自主性保留在执行层** - plan/implement/check/debug 可以自主决策
6. **不过度设计** - 拒绝完整 workflow DSL、完整 gate engine、过大的 schema

## 3. 当前系统的真实问题

当前仓库已经有一套可运行的多 Agent harness，但存在四类结构性问题。

### 3.1 没有统一的编译入口

当前新任务主要由 [task_store.py](/Users/kuang/xiaobu/spec-first/.spec-first/scripts/common/task_store.py#L147) 直接写出 `task.json`，而 [plan.py](/Users/kuang/xiaobu/spec-first/.spec-first/scripts/multi_agent/plan.py#L100) 也是直接复用 `cmd_create()`。

这说明当前天然的收口点其实已经存在：

- `task_store.py` / `cmd_create()`

但它目前只是在“写文件”，还没有真正承担：

- 编译 workflow
- 编译 policy defaults
- 写入统一 contract

### 3.2 workflow 模板层语义混杂

当前 [workflow_templates.py](/Users/kuang/xiaobu/spec-first/.spec-first/scripts/common/workflow_templates.py#L38) 同时包含：

- 拓扑模板：`default`、`quick-fix`、`docs-only`
- 预设能力：`with-tdd`、`debug`
- 尚未成熟的主流程候选：`with-review`、`research`

这会导致三个问题：

1. topology 和 policy preset 混在一起
2. runtime action 语义被过早承诺
3. 文档和代码很容易出现“双重真相”

### 3.3 runtime 还没有消费 task-level policy

当前 runtime hooks 的职责已经很清楚：

- [inject-subagent-context.py](/Users/kuang/xiaobu/spec-first/.claude/hooks/inject-subagent-context.py#L107) 负责 phase 推进和上下文注入
- [ralph-loop.py](/Users/kuang/xiaobu/spec-first/.claude/hooks/ralph-loop.py#L80) 负责 check stop gate

但它们还没有真正消费 `task.json.decision_hints`：

- `inject-subagent-context.py` 还没有注入 task-level phase policy
- `ralph-loop.py` 还在优先读取 repo-level `worktree.yaml verify`

### 3.4 任务完成仍然偏“状态驱动”，不够“证据驱动”

当前系统主要有：

- `current_phase`
- `next_action`
- marker / verify

但缺少更清晰的 evidence 模型。例如：

- 哪些 verify command 实际执行过
- 哪些要求的产物已经生成
- 哪些 gate 是 pass，哪些是 fallback pass

这不是 Phase 1 要全部做完的事，但它决定了 Phase 3 的正确方向。

## 4. 目标架构

正确的目标架构不是四层，而是六层。

## 4.1 六层边界

### Layer 0: Intent / Spec

载体：

- `prd.md`
- 任务说明
- 项目级规范

责任：

- 定义任务要解决什么问题
- 定义验收目标和约束

### Layer 1: Policy / Capability

载体：

- 项目 workflow 能力定义
- phase policy 规则文档

责任：

- 定义系统允许哪些 topology
- 定义哪些策略是硬约束，哪些只是建议

### Layer 2: Compilation

载体：

- `task_store.py`
- 未来收口后的 compiler 模块

责任：

- 把 intent + policy 编译为 task-level runtime contract

这是当前项目最关键、也最缺的一层。

### Layer 3: Runtime Contract

载体：

- `task.json`

责任：

- 保存编译后的 `workflow_type`
- 保存 `next_action`
- 保存 `decision_hints`

约束：

- runtime 只能消费这里的 contract
- 运行时不能再把上游文档当作主要 truth

### Layer 4: Runtime Enforcement

载体：

- `dispatch`
- `inject-subagent-context.py`
- `ralph-loop.py`
- `create_pr.py`

责任：

- phase 调度
- policy 注入
- verify gate 执行
- terminal action 执行

### Layer 5: Evidence / Completion

载体：

- verify 结果
- phase 完成状态
- PR / session / spec 更新结果

责任：

- 证明任务真的完成，而不是“agent 说完成”

## 5. 当前仓库中的正确收口点

结合现有代码，当前最现实、最正确的改造方式如下。

### 5.1 让 `task_store.py` 成为唯一编译入口

原因：

1. 新任务本来就在这里创建
2. `plan.py` 已经复用 `cmd_create()`，见 [plan.py](/Users/kuang/xiaobu/spec-first/.spec-first/scripts/multi_agent/plan.py#L100)
3. 从这里收口，改造面最小、收益最大

结论：

- `task_store.py` 负责写入默认 `workflow_type + next_action + decision_hints`
- `plan` 只负责建议 override，不负责另起一套写入逻辑

### 5.2 把 `workflow_templates.py` 拆成两轴

当前文件不应继续作为“大一统 workflow registry”。

正确拆法：

1. `topology registry`
   - 只保留 `default`
   - 只保留 `quick-fix`
   - 只保留 `docs-only`

2. `policy preset registry`
   - Phase 2 再接 `with-tdd`
   - Phase 2 再接 `debug`

这条边界非常关键：

- `workflow_type` 只表达 topology
- `preset` 只表达 policy enhancement

### 5.3 `create-pr` 是 terminal pipeline action，不是 policy-bearing phase

当前系统确实把 `create-pr` 放进 `next_action`，并在 [create_pr.py](/Users/kuang/xiaobu/spec-first/.spec-first/scripts/multi_agent/create_pr.py#L588) 中设置最终 `current_phase`。

这可以保留，但要明确它的定位：

- 它属于 terminal pipeline action
- 它在 `next_action` 中占一个顺序编号，用于统一 pipeline 进度与最终状态推进
- 它不属于 policy-bearing execution phase
- 它不承载独立 `decision_hints` phase schema
- 它不应反过来污染 topology / preset 设计

这里要明确区分两个概念：

1. `ordered pipeline step`
   - 只表示任务流水线中的顺序位置
   - `create-pr` 属于这一类，因此保留 `phase` number 是合理的

2. `policy-bearing execution phase`
   - 由 agent / hook / policy contract 驱动的执行阶段
   - 例如 `implement`、`check`、`finish`
   - `create-pr` 不属于这一类

因此，`create-pr` 当前的正确建模不是“普通 phase”，也不是“完全脱离状态机的后置脚本”，而是：

- 一个参与 `current_phase` 进度推进的 terminal pipeline step
- 一个不进入 `decision_hints` phase policy 模型的终端动作

### 5.4 hook 只消费 contract

`inject-subagent-context.py` 和 `ralph-loop.py` 不应该再承担默认值生产。

正确职责：

- `inject-subagent-context.py` 读取 `decision_hints`，注入 implement/check/debug policy
- `ralph-loop.py` 执行 `check.verify_commands`
- 缺字段时可以 fallback，但 fallback 不是主路径

### 5.5 `decision_hints` 的 prompt 注入格式必须统一

为了避免不同平台各自拼接 prompt，`decision_hints` 的注入格式需要统一。

推荐规则：

1. 注入位置固定在：
   - `## Your Context` 之后
   - `## Your Task` 之前
2. 注入块使用独立小节：
   - `## Phase Policy`
3. 注入内容只做“结构化 policy 翻译”，不重复整份 `task.json`
4. 注入内容必须短、硬、可执行，不夹带解释性长文

推荐格式：

```md
## Phase Policy

- `implement.mode`: `direct`
- `check.verify_commands`: `["<repo-specific commands>"]`

Follow this policy as a hard task contract when executing this phase.
```

对于 `implement`，推荐注入为：

```md
## Phase Policy

- `implement.mode`: `direct`

Execution requirements:
- Implement directly unless a stronger preset/policy is injected
- Do not perform git commit
```

对于 `check`，推荐注入为：

```md
## Phase Policy

- `check.verify_commands`: `["<repo-specific commands>"]`

Execution requirements:
- You must run the verification commands above
- Treat these commands as the task-specific gate for this task
- If they fail, continue checking/fixing instead of claiming completion
```

实现要求：

1. `build_implement_prompt()` 负责渲染 `implement` policy block
2. `build_check_prompt()` 负责渲染 `check` policy block
3. 当字段缺失时：
   - 不渲染空 block
   - 继续走 fallback 文案
4. Phase 1 不要求 `finish/debug` 注入独立 policy block

## 6. Phase 1 的正确目标

Phase 1 不是“支持很多 workflow”，而是建立最小闭环。

## 6.1 Phase 1 最小 contract

Phase 1 只落这三个核心点：

1. `workflow_type`
2. `next_action`
3. `decision_hints`

其中 `decision_hints` 在 Phase 1 只支持两个字段：

- `implement.mode`
- `check.verify_commands`

### Phase 1 完整 Schema 定义

```typescript
// task.json 新增字段
interface TaskData {
  // ... existing fields ...
  workflow_type: "default" | "quick-fix" | "docs-only";
  decision_hints: DecisionHints;
  evidence?: Evidence | null;  // Phase 1 预留，Phase 3 实现
}

// Phase 1 最小 schema
interface DecisionHints {
  implement?: {
    mode: "standard";  // Phase 1 固定值
  };
  check?: {
    verify_commands: string[];  // 例如: ["pnpm lint", "pnpm typecheck"]
  };
}

// Phase 3 预览：evidence 结构
interface Evidence {
  verify_results: VerifyResult[];
  phase_completions: PhaseCompletion[];
  artifacts: string[];
}

interface VerifyResult {
  command: string;
  exit_code: number;
  timestamp: string;
  passed: boolean;
}
```

### 默认值生成逻辑

```python
# task_store.py
DEFAULT_DECISION_HINTS = {
    "default": {
        "implement": {"mode": "standard"},
        "check": {"verify_commands": ["pnpm lint", "pnpm typecheck"]}
    },
    "quick-fix": {
        "implement": {"mode": "standard"},
        "check": {"verify_commands": ["pnpm lint"]}
    },
    "docs-only": {
        "implement": {"mode": "standard"},
        "check": {"verify_commands": ["pnpm lint"]}
    }
}

def generate_default_hints(workflow_type: str) -> dict:
    return DEFAULT_DECISION_HINTS.get(
        workflow_type,
        DEFAULT_DECISION_HINTS["default"]
    )
```

- `implement.mode`
- `check.verify_commands`

Phase 1 的 canonical schema 应直接写死为：

```ts
interface DecisionHintsPhase1 {
  implement?: {
    mode: "direct" | "tdd_recommended" | "tdd_required";
  };
  check?: {
    verify_commands: string[];
  };
}
```

当前实施基线再收紧一层：

- Phase 1 producer 默认写入 `implement.mode = "direct"`
- `tdd_recommended / tdd_required` 作为合法枚举保留，供 Phase 2 `--preset with-tdd` 产出
- `check.verify_commands` 必须由 producer 按 task/repo-specific contract 写入
- 文档中的 `pnpm lint / pnpm typecheck / pnpm test` 仅为示例，不是固定默认值

`check.cross_layer_required` 不进入 Phase 1 主 contract。原因很简单：

1. 当前没有稳定 runtime consumer
2. 进入 schema 会制造“字段已存在但系统不真正执行”的假象
3. 当前最关键的 enforcement 字段只有 `verify_commands`

## 6.2 Phase 1 拓扑范围

Phase 1 只允许三种 topology：

1. `default`
   - `implement -> check -> finish -> create-pr`

2. `quick-fix`
   - `implement -> check -> create-pr`

3. `docs-only`

不允许在 Phase 1 继续保留主流程级：

- `with-tdd`
- `with-review`
- `debug`

## 7. Phase 2 的正确目标

Phase 2 只做一件事：

> 在不扩张 topology 的前提下，增加高价值 policy preset。

## 7.1 Phase 2 只接两个 preset

1. `with-tdd`
2. `debug`

## 7.2 Canonical compile rule

Phase 2 允许灵活组合 topology 和 preset：

```bash
# 用户可以这样调用
python3 task.py create "Fix bug" --workflow quick-fix --preset debug
python3 task.py create "Add feature" --workflow default --preset with-tdd
```

**编译规则**:

```python
def compile_task_contract(workflow_type: str, preset: str | None) -> dict:
    # 1. 获取 base topology
    workflow = get_workflow(workflow_type)

    # 2. 应用 preset (如果有)
    if preset:
        preset_config = get_preset(preset)
        # 增强 decision_hints
        workflow["decision_hints"] = merge_hints(
            workflow.get("decision_hints", {}),
            preset_config["enhancements"]
        )
        # 可选: 插入额外的 phase (如 tdd)
        if preset_config.get("insert_phases"):
            workflow["phases"] = insert_phases(
                workflow["phases"],
                preset_config["insert_phases"]
            )

    return workflow
```

**推荐组合**:
- `--workflow default --preset with-tdd` - 标准流程 + TDD
- `--workflow quick-fix --preset debug` - 快速修复 + 系统化调试
- `--workflow default --preset debug` - 标准流程 + 系统化调试

为了避免双真相，Phase 2 必须固定编译规则。

推荐规则：

- `--preset with-tdd`
  - 编译为 `workflow_type=default`
  - 增强 `decision_hints.implement/check`

- `--preset debug`
  - 编译为 `workflow_type=quick-fix`
  - 增强 `decision_hints.debug/check`

这里故意不再保留“`debug` 可以编译成 `quick-fix` 或 `default`”这种模糊说法。  
如果真的需要例外，必须由显式 CLI 参数或 plan override 指定，而不是文档层默认模糊化。

2. 不新增 `review` 独立 phase
3. 不新增 `debug-systematic` 独立 phase
4. 不把 `research` 纳入主 workflow

## 8. Phase 3 的正确目标

Phase 3 才开始处理以下问题：

1. evidence 模型
2. 更丰富但已有 consumer 的 schema
3. 多平台语义收口

## 8.1 `evidence` 是 Phase 3 顶层扩展，不在 Phase 1 预留空字段

`evidence` 的方向是对的，但不应在 Phase 1 提前预留为“空字段”。

原因：

1. 当前没有稳定 producer
2. 当前没有明确 consumer
3. 当前没有收敛后的数据模型
4. 提前写入空字段会制造“schema 已支持但系统无行为”的假象

这里需要明确一个实现事实：

- `task.json` 当前采用可追加字段模型
- [TaskData](/Users/kuang/xiaobu/spec-first/packages/cli/src/templates/spec-first/scripts/common/types.py#L21) 是 `TypedDict(total=False)`，且仅用于 read-path type hints
- 写回路径保留原始 dict，不应因为类型提示丢弃未知字段

因此，未来在 Phase 3 新增：

```json
{
  "evidence": {
    "...": {}
  }
}
```

属于安全的顶层追加，不应视为破坏性变更。

正确策略是：

- Phase 1：不预留 `evidence`
- Phase 2：继续保持“evidence by concept, not by schema”
- Phase 3：在 producer、consumer、结构都收敛后再正式引入

## 8.2 只有有 consumer 的字段才能进入主 schema

Phase 3 才考虑以下字段：

- `check.cross_layer_required`
- `implement.test_layers`
- `implement.must_show_red_green`
- `debug.fix_strategy`
- `finish.update_spec_expected`

前提是同时满足：

1. 字段语义稳定
2. 有唯一 producer
3. 有明确 consumer
4. 有 fallback

## 8.3 多平台统一的是语义，不是实现细节

平台统一目标应是：

- 同一个 `workflow_type` 语义
- 同一个 `decision_hints` 语义
- 同一个 fallback 顺序
- 同一个 verify enforcement 语义

不要求第一时间统一：

- hook 实现细节
- prompt 文本
- 每个平台内部目录布局

---

## 9. 为什么必须按三阶段做

如果当前直接“做完整”，最容易出现四个错误：

1. 把 `dispatch` 再次做重
2. 把 `decision_hints` 扩成第二份 PRD
3. 把还没有 consumer 的能力提前写进 schema
4. 把 preset 错误地建模成 workflow phase

三阶段拆分的价值是：

1. 先建立 compiler -> contract -> runtime 的最小闭环
2. 再用 preset 承载高价值场景
3. 最后才引入 evidence 和扩展 schema

---

## 10. 最终结论

当前项目最需要的不是更多模板、更多 hook 分支、更多文档枚举，而是三个收口动作：

1. 让 `task_store.py` 成为唯一 task compiler 入口
2. 把 `workflow_templates.py` 拆成 topology 和 preset 两轴
3. 让 runtime hooks 真正消费 `task.json` contract

一句话总结：

> 正确方向不是把 spec-first 继续做成更复杂的 workflow engine，而是把它收敛成一个“由 compiler 产出最小 contract、由 runtime 执行 contract、最终以 evidence 判定完成”的工程化 harness。

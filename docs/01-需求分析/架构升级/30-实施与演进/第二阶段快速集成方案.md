# 第二阶段快速集成方案

> **版本**: v0.2  
> **定位**: 第一阶段完成后的快速扩展方案  
> **目标**: 在不破坏终局架构边界的前提下，快速集成最有价值的第二阶段能力。

---

## 1. 前提

本方案默认第一阶段已经完成以下最小能力：

1. `next_action` 支持最小 workflow topology 配置化
2. `workflow_type` 已存在，至少支持：
   - `default`
   - `quick-fix`
   - `docs-only`
3. `decision_hints` 已最小落地，至少支持：
   - `implement.mode`
   - `check.verify_commands`
4. `ralph-loop.py` 已开始消费 `check.verify_commands`

也就是说，第二阶段不是从零开始，而是建立在“拓扑已轻量配置化、phase policy 已最小结构化、check gate 已初步 runtime 化”的基础上。

---

## 2. 第二阶段目标

第二阶段不追求完整 workflow engine，而只做：

1. 快速增强常见高价值场景
2. 保持 `dispatch` 纯调度
3. 不引入新的复杂 phase action
4. 尽量复用已有 `workflow_type + decision_hints` 模型
5. 只承诺当前已有稳定 producer / consumer 的最小 contract

一句话：

> 第二阶段应以“预设策略集成”为主，而不是“新增 phase 引擎”为主。

补充约束：

> Phase 2 的 canonical contract 只固定 `implement.mode` 和 `check.verify_commands`。更丰富字段可以继续研究，但不应在第二阶段文档中被当作“默认已落地能力”。

---

## 3. 核心判断

对于用户提到的四类能力：

- `with-tdd`
- `with-review`
- `debug`
- `research`

不建议一视同仁地直接做成 workflow template。

推荐分类如下：

| 能力 | 第二阶段建议 | 原因 |
|------|--------------|------|
| `with-tdd` | 立即支持 | 高价值，且可映射到 `implement.mode` |
| `debug` | 立即支持 | 高价值，且可先映射到 `quick-fix` topology + 更强 implement/check contract |
| `with-review` | 延后 | 缺少稳定 runtime consumer |
| `research` | 不纳入 `next_action` 主工作流 | 更适合 plan / brainstorm 前置流程 |

---

## 4. 不建议的做法

第二阶段不建议这样做：

1. 新增 `tdd` 为独立 phase
2. 新增 `review` 为独立 phase
3. 新增 `debug-systematic` 为独立 phase
4. 新增 `research` 为主执行流水线 phase

原因：

1. 这些动作会迅速膨胀 `next_action` 的 action 语义
2. 当前 runtime 没有为这些 phase 提供稳定消费方
3. 会打破已经确认的四层边界：
   - topology
   - policy
   - runtime enforcement
   - LLM autonomy

---

## 5. 推荐实现方式

第二阶段推荐继续保持两轴结构：

### 5.1 轴一：Workflow Topology

仍然只负责 phase 顺序：

- `default`
- `quick-fix`
- `docs-only`

### 5.2 轴二：Policy Preset

通过 `decision_hints` 组合实现常见场景能力。

如果需要用户可见名称，可以支持轻量别名，但本质上仍然映射到：

- `workflow_type`
- `decision_hints`

而不是新增 action 类型，也不是新增一个持久化的 `policy_profile` 主字段。

推荐原则：

1. `preset` 是 task creation / compile 输入，不是新的 runtime truth
2. 编译结果仍然只收敛到 `workflow_type + next_action + decision_hints`
3. 如果 `plan` 参与建议，也只能产出 preset / override 建议，不能绕过 compiler 直接生成另一套 contract

---

## 6. `with-tdd` 的快速集成

## 6.1 定位

`with-tdd` 不是新的 workflow topology，而是：

> `default` workflow + 更强的 implement/check policy

## 6.2 推荐映射

```json
{
  "workflow_type": "default",
  "decision_hints": {
    "implement": {
      "mode": "tdd_required"
    },
    "check": {
      "verify_commands": ["<repo-specific commands>"]
    }
  }
}
```

说明：

1. `with-tdd` 在 Phase 2 中首先只是“更强 implement/check contract”
2. `verify_commands` 必须是 repo / task specific contract，不应在文档中固化为某一组通用命令
3. `must_show_red_green` 这类 richer field 可以继续保留在规则文档中讨论，但在当前第二阶段文档里不作为默认已落地 contract 承诺

## 6.3 为什么这比独立 `tdd` phase 更好

1. TDD 本质上是 implement 的执行模式，不是独立 topology
2. 不需要扩展 `next_action` action 语义
3. 更符合“policy 与 topology 分离”
4. 改动面小，可快速落地

## 6.4 第二阶段建议

`with-tdd` 建议作为第二阶段优先级最高的能力接入。

---

## 7. `debug` 的快速集成

## 7.1 定位

`debug` 不是新的 workflow topology，而是：

> `quick-fix` workflow + 更强的 bugfix policy

## 7.2 推荐映射

```json
{
  "workflow_type": "quick-fix",
  "decision_hints": {
    "implement": {
      "mode": "tdd_required"
    },
    "check": {
      "verify_commands": ["<repo-specific commands>"]
    }
  }
}
```

说明：

1. `debug` preset 在 Phase 2 的稳定价值，首先来自 `quick-fix` topology + 更强 implement/check contract
2. `debug.fix_strategy`、`analysis_depth`、`prefer_test_reproduction` 这类 richer field 只有在 runtime 注入和消费路径稳定后，才适合升级为 Phase 2 默认 contract
3. 在当前仓库状态下，第二阶段不应把这些字段写成“默认已经可依赖”的能力边界

## 7.3 为什么不引入 `debug-systematic` phase

1. 目前没有稳定的独立 runtime consumer
2. debug 更像修复策略，而不是主流程拓扑
3. 独立 phase 会让 action 语义快速膨胀

## 7.4 第二阶段建议

`debug` 可以作为第二阶段第二优先级能力接入。

---

## 8. `with-review` 的判断

## 8.1 当前不建议快速接入

原因：

1. 当前缺少稳定的 review runtime consumer
2. 缺少 review phase 的状态推进模型
3. 缺少 review gate 的可靠执行面
4. 很容易变成“多一个 phase 名字”，但没有真实系统能力

## 8.2 什么时候适合做

只有在以下条件同时具备时才建议进入 workflow：

1. review 输出格式稳定
2. review 的阻断/通过状态模型明确
3. runtime 能消费 review 结果

## 8.3 当前建议

第二阶段不做 `with-review` workflow 集成。

如有需要，应先作为：

- 独立 skill
- 或 finish 前的人工/半自动审查动作

而不是 workflow topology。

---

## 9. `research` 的判断

## 9.1 不建议纳入 `next_action`

`research` 更适合：

- `start`
- `brainstorm`
- `plan`

而不是 task 主执行流水线 phase。

## 9.2 原因

1. research 是前置探索，不是标准执行 phase
2. 放进 `next_action` 会让主流程边界变模糊
3. 会出现“任务已创建，但主流程前两步还在探索”的混乱状态

## 9.3 当前建议

如果要增强 research 能力，应做成：

1. plan 阶段自动触发的 research 子步骤
2. 或某种 `exploratory` preset / planning mode

但不进入 `next_action` 主模板集合。

---

## 10. 第二阶段推荐产物

第二阶段最合适的产物不是 4 个新 workflow template，而是：

### 10.1 保持 topology 不变

继续只保留：

- `default`
- `quick-fix`
- `docs-only`

### 10.2 增加 policy preset

例如：

- `with-tdd`
- `debug`

它们不是新的 runtime 顶层字段，而是 compiler 输入。

推荐映射：

| 用户名称 | 实际映射 |
|----------|----------|
| `with-tdd` | `workflow_type=default` + `decision_hints.implement.mode=tdd_required` |
| `debug` | `workflow_type=quick-fix` + `decision_hints.implement.mode=tdd_required` |
| `with-review` | 暂不支持 |
| `research` | 不属于 next_action workflow |

如果需要额外增强 `check.verify_commands`，也应在 compile 时一并生成，但仍然落在既有 `decision_hints.check` 下。

---

## 11. 第二阶段最小实施顺序

### Step 1

支持 `with-tdd` 作为 preset 别名：

- 不改 topology
- 只生成更强的 `decision_hints`

### Step 2

支持 `debug` 作为 preset 别名：

- 不改 topology
- 只生成 Phase 2 可稳定消费的更强 implement/check contract

### Step 3

支持显式 preset 输入，并固定优先级：

1. 显式 CLI 参数
2. plan 给出的 override 建议
3. 系统默认值

第二阶段先不引入“按任务类型自动选择 preset”的隐式推断逻辑，避免在编译入口之外形成第二套默认值系统。

### Step 4

暂不引入：

- `with-review`
- `research` workflow

## 13. 最终结论

第一阶段做完后，第二阶段确实可以快速推进，但正确姿势是：

> 快速集成“预设策略”，而不是快速扩张“phase 引擎”。

因此：

### 可以快速做

1. `with-tdd`
2. `debug`

### 不建议快速做

1. `with-review`
2. `research`

### 原因

因为对当前项目而言，最稳的演进路径是：

- workflow topology 保持少而稳
- phase policy 逐步增强
- runtime 只加强最关键 gate
- LLM 自主性继续保留在执行层

并且第二阶段要额外遵守两个收敛原则：

- 只有已有稳定 consumer 的字段，才进入第二阶段默认 contract
- `task_store.py` 继续作为唯一 compiler，`plan` 只提建议，不另起一套写入逻辑

这条路线既符合终局架构，也符合“不过度设计、不脱离目标”的约束。

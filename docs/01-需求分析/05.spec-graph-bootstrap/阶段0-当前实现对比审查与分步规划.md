## 一、阶段0范围收敛

结合 [阶段0-CRG-NodeJS集成技术方案.md](/Users/kuang/xiaobu/spec-first/docs/01-需求分析/spec-graph-bootstrap需求/阶段0-CRG-NodeJS集成技术方案.md) 与 [阶段0-方案文档审查意见.md](/Users/kuang/xiaobu/spec-first/docs/01-需求分析/spec-graph-bootstrap需求/阶段0-方案文档审查意见.md)，阶段0的真实目标不是“完整替代 `code-review-graph`”，而是先把下面这条链路做闭环：

- 输入收敛
- 增量检测
- AST 解析
- SQLite 入库
- raw edge 解析为 canonical edge
- 基础 postprocess：flows / communities / FTS / architecture 基础查询
- 通过 `spec-first crg <subcommand>` 暴露稳定 JSON 契约

不属于阶段0闭环的内容：

- facts 文档生成
- refresh 策略
- review 文档生产链路
- MCP server 注册与 prompt 层能力
- embedding / wiki / refactor / multi-repo

这里还需要额外加一条边界澄清：

- 阶段0做的是 **CRG 事实底座的首个可运行采集层**，不是 CRG 的最终产品形态

也就是说，阶段0聚焦 AST/代码图构建是合理的，但不能把“阶段0使用 AST 图”误解为“CRG 的定义就是 AST 图工具”。

## 二、当前实现总体判断

当前 `spec-first` 的状态，不再是“只有方案文档”，而是已经进入“阶段0中段”的实现态：

- 已有 `src/crg/` 模块化实现，覆盖 parser、graph、incremental、communities、flows、search、changes、CLI 路由。
- 已有 SQLite schema 与 `better-sqlite3` 路径。
- 已有 `resolveEdges()` 两阶段以上的 canonical 解析逻辑，用于解决阶段0文档里最关键的 P0 闭环问题之一。
- 已有社区重建的外键安全顺序修正，明显是按此前审查意见在补齐。
- 已有部分单元测试，至少覆盖了重名符号串线这一类核心回归风险。

但它距离“阶段0完成”仍有明显差距：

- 解析能力仍显著低于 `code-review-graph`。
- schema 与查询契约虽然补了一部分字段，但与文档、命令覆盖、真实查询语义仍未完全闭环。
- 一部分命令名已经出现在 router 里，但是否都具备与文档一致的行为，还不能仅凭当前结构认定为已完成。
- iOS / Kotlin / Swift / ObjC 适配仍基本停留在方案层，不是可交付实现。

结论：

`spec-first` 当前不是“尚未开始”，也不是“阶段0已完成”，而是“核心骨架已落地，离可验收还差解析能力、命令闭环、平台适配和一致性收口”。

## 四、关键差异解读

### 4.1 已经补上的关键问题

这些点说明当前实现不是停留在纸面：

1. `resolveEdges()` 已把“raw 引用如何落到 canonical target_id”做成实际代码，而不是只留在文档里。
2. `migrations.js` 已把 `confidence/source_tier/evidence/inference_reason` 加进 `nodes`，至少解决了文档中“schema 没有字段但查询假设有字段”的一类内在矛盾。
3. `communities.js` 已采用外键安全重建顺序，说明此前 P0 风险已被吸收进实现。
4. CLI 已不是单纯占位，已经具备明确的 command surface。

### 4.2 当前最大的真实缺口

这几项才是阶段0是否能收口的决定因素：

1. 语言覆盖与平台适配不足。
当前 `parser.js` 明确不支持 `swift`、`kt`、`m`、`mm`，这与阶段0文档里大量 iOS/移动端场景设计明显不一致。

2. 测试面明显偏窄。
现在能看到的强测试主要压在 edge resolve 上，但阶段0还需要覆盖：
   - input convergence 规则优先级
   - fingerprints 增量更新
   - postprocess 连续执行稳定性
   - FTS 对账
   - CLI JSON envelope 契约
   - communities 结果传播与健康状态

3. 文档目标和当前代码目标并不完全同步。
文档中强调了“17 个工具 + iOS 特例 + graphify 借鉴算法 + 阶段0验收矩阵”，但当前实现的成熟度更像是“先把 Node.js CRG 核心骨架搭出来”，还没到全面兑现所有承诺。

4. 与 `code-review-graph` 的真正差距，不在 router 数量，而在 parser 深度、成熟度和回归验证。

## 五、阶段0结论

补充一个终局视角判断：

- 从终局看，CRG 应该成为 spec-first 的工程事实内核、工作流上下文引擎和质量控制底座。
- 从阶段0看，当前实现只是这个终局目标的第一层基础设施，即“结构事实采集与构建流水线”。
- 因此，当前阶段0实践是否准确，不取决于它是不是已经变成完整 CRG，而取决于它是否为后续更高层的事实归一、上下文注入、质量控制打下了高质量底座。

如果按“阶段0方案文档”验收，当前实现状态应判定为：

- `spec-first` 已完成阶段0的基础骨架搭建。
- 关键 P0 之一的 edge canonical 化已经进入可运行实现。
- 数据库存储、社区重建安全性、CLI 集成形态基本成立。
- 但尚不能宣称“阶段0完成”，因为语言覆盖、iOS 适配、命令级对账测试、输入收敛与 postprocess 完整验证仍有实质缺口。

更准确的判断是：

`spec-first` 当前已达到“阶段0可继续实施，不需要回退重写方案”的状态，但还没有达到“可冻结 contract 并转入阶段1”的状态。

## 六、建议的分步规划

下面的规划按“先闭环、再扩面、最后对账”的顺序安排。

### Step 1：冻结阶段0最小验收面

目标：

- 明确阶段0必须通过的最小子命令集，只保留：
  - `build`
  - `stats`
  - `query`
  - `impact`
  - `flows`
  - `communities`
  - `architecture`
  - `search`
  - `postprocess`

动作：

- 将其余命令标记为“阶段0非阻塞项”或“P1”。
- 把 router 已存在但未充分验证的命令，从验收口径中剥离。

原因：

- 当前最大的风险不是“命令少”，而是“命令面过宽导致阶段0迟迟无法收口”。

### Step 2：补齐 parser 与 edge 解析的阶段0必测集

目标：

- 把最核心的数据正确性先锁死。

动作：

- 新增 fixture 与单测，至少覆盖：
  - import alias
  - relative import
  - index module
  - 同名函数跨文件
  - 同文件优先消歧
  - unresolved 边计数
  - tree-sitter 缺失时的降级行为

原因：

- 当前 `resolveEdges` 已经是阶段0最重要的“硬资产”，应优先把回归测试补齐。

### Step 3：补输入收敛与 fingerprint 对账

目标：

- 确保 build 的输入集、增量集、最终落库状态三者一致。

动作：

- 为 `input-convergence.js` 增加规则优先级测试。
- 为 `incremental.js` 增加 changed/unchanged/deleted 测试。
- 增加“路径从输入集中移除后，旧节点会被清理”的集成测试。

原因：

- 没有这一层，图数据库很容易“看起来能跑，实际上越跑越脏”。

### Step 4：补 postprocess 稳定性与 SQLite 审计

目标：

- 证明社区、flow、FTS 的重建是可重复、可审计的。

动作：

- 扩展现有 SQLite audit 脚本覆盖：
  - 连续两次 `postprocess` 结果不崩
  - `community_id` 无悬挂
  - `fts_nodes` 与 `nodes` 行数一致
  - CLI 输出与 DB 真值一致

原因：

- 这是阶段0能否宣称“图谱基础设施稳定”的关键证据。

### Step 5：明确 iOS 范围，二选一收口

目标：

- 解决“文档承诺很强，但代码尚未落地”的最大偏差。

建议二选一：

1. 如果近期不做 iOS parser 落地：
   - 立即把阶段0文档改成“iOS 适配设计已完成，真实支持延期到阶段0.5 或阶段1前置”。

2. 如果阶段0必须保留 iOS 承诺：
   - 先补 `swift`/`kt`/`m`/`mm` 的最小 parser 支持和 fixture，不要继续只停留在方案文档。

原因：

- 这是当前方案和实现差异最大的地方，必须尽快消歧，否则后续所有阶段计划都会建立在假契约上。

### Step 6：再决定是否继续追平 `code-review-graph` 的广度

目标：

- 在阶段0闭环后，再讨论是否补齐更多语言和更多分析命令。

动作：

- 对照 `code-review-graph` 逐项评估：
  - 哪些是真正被 `spec-graph-bootstrap` 需要的
  - 哪些只是“看起来很完整，但不在当前主链路”

原因：

- `code-review-graph` 的价值主要是成熟度参照，不应变成 `spec-first` 阶段0的范围膨胀来源。

## 七、实施顺序建议

推荐按下面顺序执行：

1. 先收紧阶段0验收范围。
2. 再补 parser/edge 解析测试。
3. 再补输入收敛与 fingerprint 测试。
4. 再补 postprocess/SQLite 审计。
5. 最后处理 iOS 承诺与实现的一致性。

这样做的好处：

- 能先把最核心的数据闭环锁住。
- 不会被 `code-review-graph` 的大而全能力带偏节奏。
- 能尽快得出“阶段0是否可宣告完成”的硬证据。

---

## 八、最终建议

当前最合理的项目动作不是“继续扩命令数量”，而是：

- 把阶段0当作“图基础设施闭环工程”来收尾。
- 优先做 correctness 和 contract 对账。
- 对 iOS 支持作出真实决策，而不是继续同时保留强承诺和弱实现。

一句话总结：

`spec-first` 当前已经走到“可以靠补测试和收合同完成阶段0”的位置，但前提是先承认它与 `code-review-graph` 的主要差距仍在解析广度和成熟度，而不是命令表面数量。

## 九、是否存在过度设计

结论：

存在，但不是整体过度设计，而是“阶段0目标尚未收口时，命令面和承诺面先铺宽了”。

需要把“过度设计”拆成三类看：

### 9.1 不属于过度设计的部分

这些是阶段0必须存在的基础设施，不应因为实现较完整就被误判为过度设计：

- `graph.js` 的 canonical edge 解析、去歧义、unresolved 统计
- `migrations.js` 的 SQLite schema、外键、WAL、FTS
- `incremental.js` 的 fingerprint 与 changed/unchanged/deleted 检测
- `communities.js` 的安全重建顺序
- `input-convergence.js` 的输入收敛链路

原因：

- 阶段0不是 demo，它的核心目标就是把“可持续建图、可增量更新、可查询”的基础设施做成闭环。
- 这些模块虽然看起来偏工程化，但本质上是在消除脏库、串线、悬挂引用、输入污染这类硬问题。

### 9.2 存在轻度过度设计的部分

这部分不是“完全不该做”，而是“在阶段0还没闭环时铺得太早、太宽”：

- `router.js` 一次性暴露 17+1 个子命令
- `review-context`、`detect-changes`、`affected-flows`
- `god-nodes`、`surprising-connections`
- 过早把后续分析能力包装成已对外承诺的 surface

问题不在于这些能力没价值，而在于：

- 它们抬高了阶段0验收成本
- 容易让外部误判“命令已存在 = 行为已成熟”
- 会分散本应优先投入到 parser 正确性、输入收敛、postprocess 稳定性的注意力

更准确地说，这属于“阶段顺序上的过度设计”，不是算法本身有问题。

### 9.3 明显过度设计的部分

最明显的过度设计不在代码量，而在契约承诺：

- 文档对 iOS / Swift / ObjC / Kotlin 的阶段0承诺较强

## 十、结合 `graphify` 再看“数据分析阶段”是否高质量

本节只讨论 `graphify` 的数据分析阶段，即：

`build → cluster → analyze → report`

重点不看它的输入发现或导出能力，而看“分析结果本身是否可信、可解释、可复用”。

### 10.1 结论

结论分两层：

- 如果标准是“作为研究型图分析工具，是否有高质量启发式和良好解释性”，答案是：**是，整体质量较高**。
- 如果标准是“是否已经达到可直接作为 `spec-first` 阶段0强契约依赖的工程级分析内核”，答案是：**还不完全够，仍偏研究型/启发式型高质量实现**。

换句话说：

`graphify` 的数据分析阶段不是“拍脑袋”，也不是“玩具级 demo”；它已经具备清晰的方法论、可解释输出和较好的测试意识。但它强调的是“对人有启发的分析”，不是“强 schema、强稳定性、强可回归 contract”。

### 10.2 为什么说它质量较高

从代码和测试看，`graphify` 的分析质量主要体现在四点。

#### 1. 分析逻辑是成体系的，不是零散规则堆砌

它的主链路非常清晰：

- `cluster.py` 负责社区划分
- `analyze.py` 负责高价值信号提炼
- `report.py` 负责把分析结果转成可读审计报告

这种拆分是干净的，且模块职责稳定：

- 社区发现与社区评分分离
- 图上“重要节点”和“惊喜连接”分离
- 结构分析与输出呈现分离

这意味着它的数据分析阶段不是把所有逻辑揉在一个“万能 report 函数”里，而是有明确的分析层。

#### 2. 惊喜连接算法有可解释的多因子评分，不是单指标排序

`graphify/analyze.py` 的 `_surprise_score()` 不是简单按 `confidence` 排序，而是组合了多个信号：

- 置信度层级
- 跨文件类型
- 跨顶层目录
- 跨社区
- peripheral → hub 的结构异常

而且输出里保留了 `why` 解释文本。

这点非常重要：

- 它不是只给结果
- 它还给“为什么这个结果值得你注意”

这使它更像“辅助分析器”，而不是一个只会吐排名的黑盒。

#### 3. 它对“结构噪音”有主动抑制

`analyze.py` 里对以下对象有明确过滤：

- file-level hub nodes
- method stub
- concept node

这说明作者意识到图分析里最大的风险之一不是“算不出来”，而是“算出来的都是垃圾高频节点”。  
这类噪音过滤本质上是在提升 precision，而不是盲目追求 recall。

对 `spec-first` 来说，这一点尤其值得借鉴，因为阶段0如果不先处理噪音节点，后续所有 architecture / communities / surprising 结果都会被污染。

#### 4. 测试覆盖了“分析逻辑是否真的成立”

`graphify` 的测试不是只测“返回值存在”，而是覆盖了一些分析语义：

- `test_surprising_connections_ambiguous_scores_higher_than_extracted`
- `test_surprising_connections_cross_type_scores_higher`
- `test_surprising_connections_excludes_concept_nodes`
- `test_cluster_covers_all_nodes`
- `test_cluster_does_not_write_to_stdout/stderr`
- `test_report_shows_raw_cohesion_scores`

这类测试说明它在验证：

- 算法排序方向是否符合设计预期
- 噪音是否被正确排除
- 报告是否保留原始分数而非虚假打标

这已经明显高于“只测命令能运行”的层次。

### 10.3 为什么又说它还不是工程强契约级

虽然质量较高，但它的数据分析阶段仍然有几个明显的“研究型工具”特征。

#### 1. 社区检测质量依赖外部算法实现，稳定性不是完全自控

`graphify/cluster.py` 的策略是：

- 优先 Leiden（`graspologic`）
- fallback 到 Louvain（`networkx`）

这对研究型工具很合理，因为质量优先；但对工程型基础设施来说会带来两个问题：

- 不同环境下结果可能不一致
- 算法行为的一部分不掌握在项目自己手里

因此它的 community 质量“通常不错”，但稳定契约不如 `spec-first` 现在走的确定性 3-Pass 路线。

#### 2. 分析结论仍大量依赖启发式，而非强约束数据模型

例如：

- `_is_file_node()` 依赖 label 和 source_file 约定
- `_is_concept_node()` 依赖 source_file 是否像真实文件路径
- `_surprise_score()` 的分数权重是经验权重，不是从稳定 schema 推导

这些规则并不差，反而很实用；但它们的本质是启发式。

这意味着：

- 适合“辅助理解”
- 不太适合直接作为“强验收真值”

#### 3. report 产物更偏解释型审计，而不是严格机器契约

`report.py` 的 `GRAPH_REPORT.md` 很适合人看：

- 有 Summary
- 有 God Nodes
- 有 Surprising Connections
- 有 Communities
- 有 Ambiguous Edges
- 有 Knowledge Gaps

这类报告质量是高的，因为可读、可解释、有叙事结构。

但它的问题是：

- 输出更偏 markdown 审计文档
- 不是强 schema 的机器消费接口
- 某些 section 的阈值和表述较经验化

所以它适合启发人，不适合直接充当 `spec-first` 阶段0的稳定分析 contract。

#### 4. 分析结果的“正确性上界”受前置抽取质量强烈影响

`graphify` 的分析做得再好，也还是建立在前面的 extraction quality 之上。

而它的 build 层是 NetworkX + JSON 合并逻辑，存在这类特点：

- 节点合并时“后写覆盖前写”
- 图分析较依赖 label/source_file 等属性约定
- 结构边与语义边混在同一个图里做排序

这并不一定错误，但说明它更像“高质量分析器”，而不是“高保真图数据库上的确定性查询层”。

### 10.4 对 `spec-first` 的借鉴价值

因此，`graphify` 对 `spec-first` 最有价值的不是“整套搬过来”，而是下面三类能力：

**应该借鉴**

- 多因子惊喜连接的思路
- 噪音节点过滤意识
- 分析输出必须附带解释字段
- 对排序方向和过滤逻辑写测试

**谨慎借鉴**

- Leiden/Louvain 这类外部社区算法依赖
- 过多经验阈值直接进入阶段0 contract
- 报告式输出替代结构化 JSON 契约

**不应直接借鉴为阶段0基础**

- 把启发式分析结果当作“真值层”
- 把 markdown 报告当作主数据接口
- 把环境依赖型算法当作阶段0稳定核心

### 10.5 对当前阶段0判断的影响

结合 `graphify` 之后，可以更准确地修正前面的结论：

1. `spec-first` 阶段0的重点仍应是“确定性图基础设施闭环”，不是先追求 `graphify` 这种分析层丰富度。
2. `graphify` 的分析阶段本身是高质量的，但它更适合做：
   - P1/P2 分析增强
   - report 层启发式信号来源
   - architecture / risk signal 的补充
3. 如果在阶段0就把 `graphify` 式分析大量前置，会放大过度设计风险，因为它本身就是高质量启发式层，而不是最小闭环层。

### 10.6 最终判断

一句话判断：

`graphify` 的数据分析阶段是“高质量的启发式分析层”，但不是“可直接替代阶段0确定性基础设施层的工程内核”。

因此对 `spec-first` 最合理的策略是：

- 阶段0只吸收它最成熟、最可解释的分析思想
- 不把它整套分析框架前置成当前必须兑现的 contract
- 等 `spec-first` 的 build/query/postprocess 闭环稳定后，再逐步引入 `graphify` 式高阶分析

# `spec-graph-bootstrap` 阶段 2：事实抽取与文档生成需求

> 本文是 `spec-graph-bootstrap` 的阶段 2 需求文档。
>
> 本文定义 v1 最小闭环如何从目标仓库抽取事实并生成可消费产物。
>
> 如与 [修订终版.md](./修订终版.md) 冲突，以总版为准。
>
> 本文中的“阶段 2”指实施推进阶段，不等同于 skill 执行过程中的 `Phase 0` 到 `Phase 4`。

## 1. 目标

阶段 2 的目标是把 `spec-graph-bootstrap` 从“可安装的 skill”推进到“可生成最小闭环产物的 Stage-0 系统”。

这个阶段不是简单创建 markdown，而是建立以下链路：

* 先抽取事实层
* 再基于事实层生成摘要、代码事实与任务包
* 最后生成可判定的注入路由

阶段 2 完成后，系统应能在不依赖外部 graph 项目的前提下，稳定生成 v1 核心产物。

## 2. 范围

本阶段包含：

* Host Readiness Gate
* mode / slug / rerun backup
* `Fact Inventory` 及相关控制面产物
* 最小摘要层与代码事实层产物
* `review-change` task pack
* `injection-index.yaml`

本阶段不包含：

* 后续 workflow 的自动消费接入
* 复杂的 refresh / partial invalidation
* 完整 layers/database 全展开
* 外部 `graphify` 主链路集成

## 3. v1 最小闭环产物

阶段 2 要求能够稳定生成以下核心产物：

### 3.1 控制面产物

* `.context/spec-first/bootstrap/<slug>/fact-inventory.json`
* `.context/spec-first/bootstrap/<slug>/risk-signals.json`
* `.context/spec-first/bootstrap/<slug>/test-surface.json`
* `.context/spec-first/bootstrap/<slug>/fingerprints.json`

其中 `fingerprints.json` 在阶段 2 的最低要求是：

* 文件可稳定生成并可被读取
* 顶层结构至少包含 `inputs`、`outputs`、`updated_at`
* 能记录最基本的输入文件、analyzer 版本和已生成产物
* 此处的“最低要求”仅指 v1 初版 contract，不等同于阶段 4 的完整 refresh 依赖模型

更完整的依赖模型、失效传播和局部刷新语义由阶段 4 完善，不要求在阶段 2 一次完成。

### 3.2 长期上下文产物

* `docs/contexts/<slug>/README.md`
* `docs/contexts/<slug>/00-summary.md`
* `docs/contexts/<slug>/architecture/module-map.md`
* `docs/contexts/<slug>/pitfalls/index.md`
* `docs/contexts/<slug>/code-facts/public-entrypoints.md`
* `docs/contexts/<slug>/code-facts/test-map.md`
* `docs/contexts/<slug>/code-facts/high-risk-modules.md`
* `docs/contexts/<slug>/context-packs/review-change.md`
* `docs/contexts/<slug>/injection-index.yaml`

## 4. 输入

阶段 2 的输入包括：

* 目标仓库源码
* 项目配置
* 测试目录与测试配置
* schema / migration / model / DTO 等数据形态资产
* 宿主 marker 与 runtime probe 结果
* `spec-first crg` CLI 输出（CRG Node.js 内嵌实现，阶段 0 已完成）

> **工具来源说明**：CRG 能力以 `spec-first crg <command>` CLI 形式内嵌，无需独立安装 Python 依赖或注册 MCP server。Serena 仍以 MCP 形式使用（Enhanced/Basic 降级路径）。详见[阶段2-集成方案.md](./阶段2-集成方案.md) §1。

## 5. 输出职责边界

### 5.1 事实层

事实层负责提供可追溯事实，不负责直接写 narrative 结论。

每条事实至少应支持以下字段：

* `path`
* `symbol`
* `kind`
* `summary`
* `confidence`
* `inference_reason`
* `evidence`
* `updated_at`

### 5.2 摘要层

摘要层负责压缩事实，而不是重新扫源码。

* `README.md` 是上下文控制台
* `00-summary.md` 是最小事实摘要
* `module-map.md` 和 `pitfalls/index.md` 是稳定的项目理解入口

### 5.3 路由层

路由层负责把产物树变成可消费规则：

* `context-packs/review-change.md` 提供 review 场景的最小上下文包
* `injection-index.yaml` 提供按 `stage` / `task_type` / `fact.*` / `output_exists.*` 判定的可读规则

## 6. 关键设计约束

### 6.1 Facts First

在本阶段内，任何正式产物都必须优先消费事实层。

默认规则：

* renderer 不得自由回扫源码补洞
* 如确需回读源码，必须显式记录 fallback 原因和范围
* `Inferred` 结论必须带 `inference_reason`

### 6.2 路径兼容

本阶段不得改变以下路径约定：

* `docs/contexts/<slug>/`
* `.context/spec-first/bootstrap/<slug>/`

### 6.3 降级可见

所有 `block / degrade / skip` 行为都必须对用户可见，对后续节点也可见。

不得出现“能力缺失但静默跳过”的行为。

### 6.4 最小闭环优先

本阶段优先保证 v1 核心产物稳定生成，不追求一次性覆盖完整产物树。

如果某个增强产物会明显拖慢闭环，应延后到阶段 4 或后续版本。

### 6.5 路由求值契约必须与总版一致

阶段 2 生成的 `injection-index.yaml` 必须与总版的最小求值模型保持一致。

最低要求包括：

* `selection_rules` 至少支持 `stage`、`task_type`、`fact.*`、`output_exists.*`
* `output_exists.*` 只表达“消费方基于实际文件存在性做实时求值”的契约，不新增独立状态文件
* v1 至少稳定覆盖 `always / plan / work / review / unknown`
* `unknown` 的最小回退集合必须可读、可人工消费、且不阻断 workflow

## 7. 执行流程

### 7.1 Phase 0：Readiness

1. 判定当前宿主和 marker
2. 验证 runtime baseline 是否就绪（Serena MCP probe）
3. 检测 CRG 图状态：检查 `<target>/.spec-first-graph/graph.db` 是否存在并有效（`spec-first crg stats`）
4. 若图未就绪，询问用户是否执行 `spec-first crg build`，或降级到 Enhanced/Basic
5. 决定 `Basic / Enhanced / Full` 或等价能力边界
6. 输出 `graph_support_state`、`db_support_state`、`code_facts_confidence`

> 工具实现细节见[阶段2-集成方案.md §3](./阶段2-集成方案.md)。

### 7.2 Phase 1：事实抽取

1. 项目身份识别
2. 入口点识别
3. 模块结构识别
4. 集成边界识别
5. 测试表面识别
6. 风险信号识别
7. 数据形态识别
8. 写入控制面事实产物

### 7.3 Phase 2：任务规划

1. 为固定产物生成 PRD task contracts
2. 为条件产物判定是否创建 task
3. 将事实证据写入 PRD，而不是只写抽象说明

### 7.4 Phase 3：文档生成

**前置检查**：进入 Phase 3 前，`fact-inventory.json` 必须存在且非空。如不满足，停止并提示用户：事实层不完整，不得在无事实层的情况下继续渲染。

1. 生成 `00-summary.md`
2. 生成 `module-map.md`
3. 生成 `pitfalls/index.md`
4. 生成 `code-facts/*`
5. 生成 `context-packs/review-change.md`（由 `spec-first crg review-context` 驱动，见集成方案 §8）
6. 串行生成 `README.md`

### 7.5 Phase 4：路由生成

1. 生成 `injection-index.yaml`
2. 写入 `always / stages / task_types / selection_rules / advice`
3. 保证 `unknown` task type 有最小回退集合
4. 明确 `output_exists.*` 作为消费期实时求值契约写入路由说明

## 8. 验收标准

### 8.1 事实层验收

* `fact-inventory.json`、`risk-signals.json`、`test-surface.json`、`fingerprints.json` 全部存在
* 每条 `Inferred` 事实都有 `inference_reason`
* 事实层字段和置信度值域稳定

### 8.2 文档层验收

* v1 最小闭环产物全部生成且非空
* `README.md` 能作为阅读控制台使用
* 各产物职责边界清楚，避免大面积重复堆叠

### 8.3 路由层验收

* `injection-index.yaml` 为合法 YAML
* 至少覆盖 `always / plan / work / review / unknown`
* `selection_rules` 已包含 `output_exists.*` 相关求值契约
* `unknown` 路由不阻断 workflow

### 8.4 稳定性验收

* 可在至少 2 个真实项目上运行
* 不依赖外部 graph 工具也能完成最小闭环
* token 成本与运行时长可接受

## 9. 与其他阶段的关系

* 阶段 2 依赖阶段 1 的安装与入口集成
* 阶段 3 的消费逻辑必须建立在阶段 2 的产物合同稳定之后
* 阶段 4 的 refresh 与 invalidation 必须建立在阶段 2 的产物依赖关系明确之后

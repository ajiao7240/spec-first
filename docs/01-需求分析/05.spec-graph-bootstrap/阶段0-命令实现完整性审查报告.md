## Blocker 缺陷详细说明

### B-1：`flows` — criticality 评分无意义

**当前实现（`src/crg/flows.js:189`）**：
```javascript
// flows.js:189
const criticality = pageRankScores.get(entry.id) || 0;
```
criticality 直接取该节点的 PageRank 分数。

**实际效果（算法必然结论，非样本依赖）**：
```sql
-- 任意构建后执行（数字因构建状态不同而异，但比较结果恒定）：
SELECT COUNT(DISTINCT criticality) FROM flows;
-- → 结果为 1：所有 flow 的 criticality 值完全相同
```
PageRank 在图稀疏时（calls 边大量 unresolved）结果均匀化，所有节点分数趋近于相同值——**这是算法的数学性质，与具体构建状态无关**，不可能通过某次特殊构建规避。

**Python 原版算法（`code-review-graph/flows.py:compute_criticality`）**：
```python
criticality = (
    file_spread      * 0.30   # 涉及文件数：1 file→0.0, 5+→1.0
  + external_score  * 0.20   # 外部调用数：0→0.0, 5+→1.0
  + security_score  * 0.25   # 安全关键词节点占比（auth/token/encrypt...）
  + test_gap        * 0.15   # 1 - 测试覆盖率
  + depth_score     * 0.10   # BFS 深度：0→0.0, 10+→1.0
)
```

**补充发现（源码对比新增）**：除 criticality 算法外，还存在以下差距：
1. **Entry point 检测缺失装饰器和命名规则**：Python `detect_entry_points()`（`flows.py:76-112`）支持"无入度检测 + 框架装饰器识别（`@app.route`等）+ 命名模式（`main/test_*/handle_*`）"三层逻辑；JS 仅用无入度 SQL 查询（`flows.js:151-157`），漏掉所有框架入口点。
2. **BFS 深度限制从 15 降到 5，每 flow 节点数上限从无限改为 20**：`flows.js:151` 中 maxdepth=5, maxNodesPerFlow=20；Python `flows.py` maxdepth=15，无节点数上限。
3. **flows 表缺失字段**：JS 只写入 `id, entry_node_id, criticality, node_count`（`flows.js:162-165`）；Python 另写入 `name, depth, file_count, path_json`（`flows.py:328-340`）。

**修复方向**：在 `flows.js` 中实现等价的 5 因子加权评分。关键参数需从已解析的 `flow_nodes`、`edges`、`nodes` 表计算：
- `file_spread`：`SELECT COUNT(DISTINCT file_path) FROM nodes WHERE id IN (flow_nodes)`
- `security_score`：节点名称匹配安全关键词（见下方常量，**须提取为共享模块**）
- `test_gap`：`SELECT COUNT(*) FROM nodes WHERE is_test=1 AND id IN (flow_nodes)`
- `depth_score`：已有 `node_count`，可用 BFS 最大 depth 近似
- `external_score`：流节点调用 `target_id IS NULL` 的边占比（间接通过 unresolved 边估算）

**⚠️ 安全关键词须使用完整列表（共 25 个，来自 `constants.py:7-12`）**：
```javascript
// src/crg/constants.js（新建共享模块，flows.js 和 changes.js 均须引用）
const SECURITY_KEYWORDS = new Set([
  'auth', 'login', 'password', 'token', 'session', 'crypt', 'secret',
  'credential', 'permission', 'sql', 'query', 'execute', 'connect',
  'socket', 'request', 'http', 'sanitize', 'validate', 'encrypt',
  'decrypt', 'hash', 'sign', 'verify', 'admin', 'privilege',
]);
```
> 注意：`sql`、`query`、`execute` 是 SQL 注入相关关键词，对数据访问层评分至关重要。若只用 9 个关键词的子集，数据库操作代码的 `security_score` 将系统性为 0。

### B-2：`detect-changes` — 算法维度严重缩水

**当前实现（`src/crg/changes.js:41-55`）**：
```javascript
function assessFileRisk(filePath, db) {
  const fanIn = ...; // 该文件 module 节点的入边总数
  if (fanIn >= 10) return 'High';
  if (fanIn >= 3)  return 'Medium';
  return 'Low';
}
// 输出：每个文件 + risk_level（High/Medium/Low）
```

**Python 原版算法（`code-review-graph/changes.py:compute_risk_score`）**：
```python
def compute_risk_score(store, node):
    score = 0.0
    score += min(flow_count * 0.05, 0.25)            # F1 flow 参与度 （上限 0.25）
    score += min(cross_community * 0.05, 0.15)       # F2 社区跨越调用者数（上限 0.15）
    score += 0.05 if has_test else 0.30              # F3 测试覆盖（有测试+0.05，无测试+0.30）
    score += 0.20 if security_keyword_match else 0   # F4 安全关键词
    score += min(caller_count / 20.0, 0.10)         # F5 调用者数量（上限 0.10）
    return round(min(score, 1.0), 4)                 # 0-1.0 浮点
```

**差距对比**：

| 维度 | spec-first | Python 原版 |
|------|-----------|------------|
| 粒度 | **文件级** | **节点（函数）级** |
| 输出 | 三档枚举 | 0-1.0 浮点精度 |
| 因子 | 1 个（fan-in） | 5 个因子 |
| flow 参与度 | ❌ 无 | ✅ 有 |
| 测试覆盖缺口 | ❌ 无 | ✅ 有 |
| 安全关键词 | ❌ 无 | ✅ 有 |
| 社区跨越度 | ❌ 无 | ✅ 有 |
| test_gaps 输出 | ❌ 无 | ✅ 有（无测试覆盖的高风险函数） |
| review_priorities | ❌ 无 | ✅ 有（top 10 按风险排序） |

**⚠️ 前置条件（实现前须验证）**：
```sql
-- F2 因子依赖 community_id 正确传播。在修复 changes.js 前，执行：
SELECT COUNT(*) FROM nodes WHERE community_id IS NULL;
-- 结果必须为 0，否则 cross_community 因子恒为 0，风险评分系统性偏低
```

### M-1：`context` — top_hubs 排序错误

**当前实现（`src/crg/cli/context.js:74-87`）**：

-- 改为按入度排序：
SELECT n.*, COUNT(e.id) as in_degree
FROM nodes n
LEFT JOIN edges e ON e.target_id = n.id
WHERE n.kind != 'module'

- **confidence_weight**（+10）：边的源或目标节点为推断型（`confidence='Inferred'`）
- **cross_language**（+30）：跨编程语言的依赖（如 Python 调用 JS，或 JS 调用 Ruby）
- **cross_community**（+40）：跨社区边（最强信号）
- **peripheral_to_hub**（+20）：入度为 0 的外围节点调用高入度中枢节点

过滤阈值 `score < 30` 排除噪音。此命令在 Python 原版中无对应，是 spec-first 的原创贡献。

### E-2：`god-nodes`（明确的过度中心化检测）
按 `in_degree DESC` 取前 5%（上限 10 个，下限 1 个），排除 module 节点。Python 原版通过 `get_impact_radius` 间接发现高影响节点，但无专用命令。

# `spec-graph-bootstrap` 阶段 3：相关文档消费需求

> 本文是 `spec-graph-bootstrap` 的阶段 3 需求文档。
>
> 本文定义后续 workflow 如何消费阶段 2 生成的 Stage-0 产物。
>
> 如与 [修订终版.md](./修订终版.md) 冲突，以总版为准。
>
> 本文中的“阶段 3”指实施推进阶段，不等同于 skill 执行过程中的 `Phase 0` 到 `Phase 4`。

## 1. 目标

阶段 3 的目标是证明 `spec-graph-bootstrap` 生成的上下文资产不是静态文档集合，而是后续 `plan / work / review` 可以可靠消费的输入层。

阶段 3 采用两步推进：

* `3A` 非侵入式消费验证：先通过需求文档中的治理说明草案与人工消费规范验证路由是否有效
* `3B` skill 消费固化：把验证通过的规则最小化写入 `spec-plan / spec-work / spec-code-review` 的 `SKILL.md`

阶段 3 完成后，系统应具备以下能力：

* 后续 workflow 能按 `stage` 与 `task_type` 读取最小必要上下文
* 当 task type 无法稳定识别时，能回退到合理的最小集合
* 当某些产物缺失、为空、过期或解析失败时，后续 workflow 仍可继续运行
* Stage-0 产物消费始终是增强项，不是主任务前置阻断项

补充说明：

* `3A` 证明的是“人工模拟消费规则成立”
* `3B` 才证明“真实 workflow 已内建该消费能力”

## 2. 范围

本阶段包含：

* `3A`：
  * 在需求文档中定义项目级治理说明草案
  * 定义 `plan / work / review` 的人工消费顺序与验证方法
  * 验证 `stage + task_type + selection_rules` 的路由是否合理
* `3B`：
  * 把验证通过的消费规则最小化固化到 `spec-plan / spec-work / spec-code-review` 的 `SKILL.md`
  * 固化统一降级逻辑和 fallback 路径
* `stage + task_type + selection_rules` 的消费契约
* 手动消费方式和后续自动消费的接口预留

本阶段不包含：

* 重写 `spec-plan / spec-work / spec-code-review` 的全部主逻辑
* 修改 command 层入口以承载消费逻辑
* 新增复杂的上下文装配框架
* 语义 task type 分类器的复杂实现
* `brainstorm / compound` 的正式消费接入验证；它们仍属于总版定义的终局覆盖范围，但不纳入阶段 3 的 v1 验收

## 3. 输入

阶段 3 消费的上游输入包括：

* `docs/contexts/<slug>/README.md`
* `00-summary.md`
* `pitfalls/index.md`
* `code-facts/*`
* `context-packs/*`
* `injection-index.yaml`
* 阶段 2 输出的 freshness / confidence 信息
* 基于 `docs/contexts/<slug>/` 实际文件存在性实时求值得到的 `output_exists.*`

## 4. 消费模型

### 4.1 最小消费单位

本阶段不以“整棵 context 树”作为默认输入，而以以下最小消费单位为准：

* `always` 集合
* `stage` 集合
* `task_type` 集合
* `selection_rules` 命中的条件扩展集合

### 4.2 默认回退集合

当 `task_type` 无法稳定识别，或 `injection-index.yaml` 不可用时，应回退到固定最小集合：

* `00-summary.md`
* `pitfalls/index.md`
* `code-facts/public-entrypoints.md`
* `code-facts/test-map.md`

回退的目标是降级，不是阻断。

### 4.3 阅读优先级

在消费时，应遵循以下优先级：

* `review` 优先风险文档和 review task pack
* `work` 优先 code-facts 和 task pack
* `plan` 优先 module map 和入口清单
* narrative 说明文档不应压过 task pack 和 code-facts

### 4.4 Task Type 最低识别契约

第一阶段的 task type 不要求复杂自动分类，但必须满足：

* 由消费 workflow 负责判定当前 task type
* 判定优先级：
  1. 用户显式指定
  2. workflow 基于任务描述做稳定映射
  3. 无法稳定识别时回退到 `unknown`
* `unknown` 是正式兜底类型，不是异常状态

### 4.5 非阻断降级契约

`spec-graph-bootstrap` 产物消费是增强项，不是主任务阻断条件。统一降级契约如下：

1. 优先读取 `injection-index.yaml`
2. 读取失败时退回固定最小集合
3. 固定最小集合仍缺失时，退回 workflow 自身的 repo research / diff analysis / 计划文件直接阅读
4. 记录降级级别与原因，但不得因为 Stage-0 产物缺失而中止主任务

只有主任务自身输入不足，例如需求不清、计划缺失或 review scope 无法建立，才允许阻断。

### 4.6 文件级健壮性要求

以下情况都只能触发降级，不能触发 workflow 失败：

* 文件不存在
* 文件为空
* YAML / JSON 解析失败
* freshness 不明或明显过期
* 内容只达到 `Inferred` 或 `Unknown`

## 5. 各 workflow 的消费职责

### 5.1 `3A` 非侵入式消费验证

`3A` 通过需求文档中的“项目级治理说明草案”承载消费约定：

* 在本需求文档中给出优先消费顺序、fallback 规则与“缺失即降级、不阻断”原则的文案草案
* 草案用于指导后续开发与验证，不直接修改当前项目的 `AGENTS.md / CLAUDE.md`
* 用真实任务手工验证 `plan / work / review` 是否按约定读取产物

`3A` 的目标是验证产物设计和路由规则，不在本子阶段固化到 workflow 主逻辑。

v1 边界：

* 阶段 2 最小闭环只强制提供 `context-packs/review-change.md`
* 因此 `3A` 与 `3B` 对 `spec-plan`、`spec-work` 的验证，应以 `code-facts + fallback` 为主，不以专用 task pack 命中作为前提
* `spec-code-review` 是 v1 中唯一保证存在专用 task pack 的消费方

### 5.2 `3B` skill 消费固化

`3B` 只做最小侵入式固化：

* 修改 `skills/spec-plan/SKILL.md`
* 修改 `skills/spec-work/SKILL.md`
* 修改 `skills/spec-code-review/SKILL.md`
* 不改 command 层
* 不扩大到其他 workflow

只有 `3A` 验证通过的规则，才允许进入 `3B` 固化。

### 5.2.1 `3A` 项目级治理说明草案

以下草案用于指导后续开发阶段在宿主项目中如何添加非侵入式消费说明，不代表当前仓库已经采用该规则。

#### 总原则

* `docs/contexts/<slug>/` 下的 Stage-0 产物是增强上下文，不是主任务前置阻断项
* 当 `injection-index.yaml`、`context-packs/*`、`code-facts/*` 缺失、为空、过期或解析失败时，只允许降级，不允许阻断主任务
* 只有主任务自身输入不足，例如需求不清、计划缺失、review scope 无法建立时，才允许阻断

#### 建议消费顺序

* `spec-plan`
  * 优先读取 `docs/contexts/<slug>/injection-index.yaml`
  * 然后读取 `README.md`、`00-summary.md`、`architecture/module-map.md`
  * 若命中 task type，再读取对应 `context-packs/*`
* `spec-work`
  * 优先读取 `docs/contexts/<slug>/injection-index.yaml`
  * 然后读取 `code-facts/public-entrypoints.md`、`code-facts/test-map.md`
  * 若命中 task type，再读取对应 `context-packs/*`
* `spec-code-review`
  * 优先读取 `docs/contexts/<slug>/injection-index.yaml`
  * 然后读取 `context-packs/review-change.md`、`code-facts/high-risk-modules.md`、`code-facts/test-map.md`

#### 建议降级规则

* `injection-index.yaml` 缺失或非法：退回 `README.md` 与固定最小集合
* 固定最小集合为：
  * `00-summary.md`
  * `pitfalls/index.md`
  * `code-facts/public-entrypoints.md`
  * `code-facts/test-map.md`
* 固定最小集合仍缺失：退回 workflow 自带的 repo research、diff analysis 或 plan reading
* `task_type` 无法稳定识别：回退到 `unknown`
* `unknown` 不是错误状态，必须继续执行

#### 建议执行要求

* 消费 Stage-0 产物时，优先使用 task pack 和 code-facts，不要让 narrative 文档压过执行型上下文
* 若文档只提供 `Inferred` 或 `Unknown` 结论，必须视为弱信号，不得当作强事实依赖
* 当发生降级时，应在响应中简要说明降级原因，但不要求用户先补 bootstrap 产物
* 草案验证通过后，再进入 `3B` 写入相关 `SKILL.md`

### 5.2.2 `3A` 验证记录模板

为避免 `3A -> 3B` 的准入判断过于主观，阶段 3 必须为每个验证任务产出一份统一记录。模板字段至少包括：

* `task_name`：本次验证任务名称
* `task_goal`：要完成的主任务
* `stage`：`plan / work / review`
* `task_type`：命中的 task type，若未识别则写 `unknown`
* `context_slug`：验证使用的 context slug
* `expected_inputs`：按治理草案预期应命中的上下文文件
* `actual_inputs`：实际读取到的上下文文件
* `fallback_triggered`：是否触发 fallback
* `fallback_reason`：触发 fallback 的原因
* `degrade_level`：`none / partial / full-fallback`
* `missing_outputs`：缺失但不阻断的产物清单
* `misleading_points`：造成误导、噪音或弱信号过强的问题
* `useful_outputs`：真正帮助任务推进的产物
* `verdict`：`pass / pass-with-gaps / fail`
* `allow_enter_3b`：`yes / no`
* `gate_reason`：是否允许进入 `3B` 的依据

记录要求：

* `plan / work / review` 各至少保留一份真实任务记录
* `allow_enter_3b = yes` 只允许在“主路径可工作、降级可解释、无关键误导”的前提下给出
* 若 `verdict = fail` 或 `allow_enter_3b = no`，必须明确是路由问题、产物问题还是治理草案问题
* 该模板是阶段 3 的准入证据，不要求当前仓库实现自动收集

### 5.3 `spec-plan`

`spec-plan` 应至少消费：

* `always`
* `stages.plan`
* 命中的 `selection_rules`

最小目标：

* 规划时知道从哪里进入系统
* 知道核心模块边界
* 知道相关测试和高风险区域
* v1 下允许主要通过 `code-facts + fallback` 完成消费，不要求专用 task pack 必定存在
* 当上下文资产缺失时，直接退回本地 repo research，而不是中止 planning

### 5.4 `spec-work`

`spec-work` 应至少消费：

* `always`
* `stages.work`
* 与测试和高风险相关的条件扩展

最小目标：

* 实施前知道先看什么文件
* 知道影响哪些测试
* 知道什么地方容易过度修改
* v1 下允许主要通过 `code-facts + fallback` 完成消费，不要求专用 task pack 必定存在
* 当上下文资产缺失时，直接退回计划文件与本地代码扫描，而不是中止 work

### 5.5 `spec-code-review`

`spec-code-review` 应至少消费：

* `always`
* `stages.review`
* `context-packs/review-change.md`
* 高风险模块和测试映射

最小目标：

* review 不只盯 diff
* review 能补看风险传播面和相关测试
* 当上下文资产缺失时，直接退回基于 diff 的常规 review，而不是中止 review

## 6. 关键设计约束

### 6.1 消费不能依赖完美产物

本阶段必须接受以下现实：

* task type 可能识别失败
* 某些条件产物可能不存在
* 某些事实可能只达到 `Inferred`

因此消费规则必须内建 graceful degradation。

### 6.2 不能反向破坏总版路径

消费方只能读取已定义的 Stage-0 产物路径，不得引入新的私有路径约定。

### 6.3 先验证再固化

阶段 3 必须先经过 `3A` 的非侵入式验证，再进入 `3B` 的 skill 固化。不得跳过验证，直接把未经验证的路由规则写入 workflow 主契约。

### 6.4 手动可消费优先于自动化完美

在尚未统一引入自动装配器之前：

* YAML 必须先支持人工 / agent 手动消费
* 自动消费是后续增强，不是本阶段前提

### 6.5 缺失即降级，不得阻断

只要 `plan / work / review` 的主输入仍成立，Stage-0 产物缺失都只能触发降级，不得让 workflow 直接失败或要求用户先补 bootstrap 产物。

## 7. 执行流程

### 7.1 `3A` 非侵入式验证流程

1. 先在本需求文档中确认治理说明草案
2. 选择真实任务，分别验证 `plan / work / review`
3. 按统一验证记录模板记录命中路径、降级路径、误导点与缺失点
4. 收敛出允许固化到 `3B` 的最小规则集

其中：

* `plan / work` 优先验证 `code-facts + fallback` 是否足够支撑主任务
* `review` 验证 `review-change` task pack 是否能带来额外价值

### 7.2 `3B` skill 固化流程

1. 将 `3A` 验证通过的规则写入三个 workflow 的 `SKILL.md`
2. 为三个 workflow 分别加入最小 fallback 说明
3. 验证缺失产物时仍能继续运行
4. 验证消费顺序与人工约定一致

### 7.3 消费前准备

1. 识别当前 `stage`
2. 推断或指定 `task_type`
3. 读取 `README.md` 了解 freshness 与可用产物
4. 读取 `injection-index.yaml`
5. 根据 `docs/contexts/<slug>/` 的实际文件存在性求值 `output_exists.*`

### 7.4 路由求值

1. 加载 `always`
2. 加载对应 `stage`
3. 加载对应 `task_type`
4. 对 `selection_rules` 做条件求值
5. 若 task type 未命中，则回退 `unknown`

### 7.5 降级处理

* YAML 缺失：退回固定最小集合
* YAML 非法：记录 warning，退回固定最小集合
* 条件未命中：跳过条件扩展，不阻断
* 产物缺失：跳过缺失文件并记录原因
* 固定最小集合也缺失：退回 workflow 自带的 repo research / diff analysis / plan reading
* 任何单文件失败都不得让 workflow exit non-zero

## 8. 验收标准

### 8.1 `3A` 非侵入式验证验收

* 本文中的治理说明草案足以指导至少两类真实任务
* 已按统一模板记录命中、降级、误导点和缺失点
* 已形成允许进入 `3B` 的最小固化规则集
* 已明确区分人工模拟消费成立，与真实 workflow 已接入不是同一结论
* `3B` 准入判断可追溯到具体验证记录，而不是口头结论

### 8.2 `3B` skill 固化验收

* `spec-plan / spec-work / spec-code-review` 的 `SKILL.md` 已写入最小消费与 fallback 规则
* 不依赖人工提醒也能优先消费 Stage-0 产物
* 缺失产物时仍能稳定降级

### 8.3 Plan 消费验证

至少完成一组实验，证明 `spec-plan` 能消费：

* `architecture/module-map.md`
* `code-facts/public-entrypoints.md`
* 正确的 fallback 集合；若存在专用 task pack，可作为加分项验证

### 8.4 Work 消费验证

至少完成一组实验，证明 `spec-work` 能消费：

* `code-facts/test-map.md`
* 正确的 fallback 集合；若存在专用 task pack，可作为加分项验证
* 命中的条件扩展

### 8.5 Review 消费验证

至少完成一组实验，证明 `spec-code-review` 能消费：

* `code-facts/high-risk-modules.md`
* `context-packs/review-change.md`
* `code-facts/test-map.md`

### 8.6 降级验收

* task type 未识别时，workflow 不阻断
* 部分产物缺失时，workflow 仍能继续
* `docs/contexts/<slug>/` 整体缺失时，workflow 仍能继续
* `injection-index.yaml` 缺失或非法时，workflow 仍能继续
* 消费逻辑能区分 `Observed / Inferred / Unknown`

## 9. 与其他阶段的关系

* 阶段 3 依赖阶段 2 产物合同稳定
* 阶段 3 的验证结果会直接反馈到总版中的替换策略与 MVP 验收
* 阶段 4 的刷新策略不能破坏本阶段定义的消费路径和回退集合

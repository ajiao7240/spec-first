范围**只聚焦第一层：项目级规范层（repo-profile）**，并严格遵循你确认的指导思想：

> **轻 contract + 明确边界 + 让 LLM 决策**
> 通过提供更高质量的决策输入提升质量，
> **不要**把质量控制做成“多状态流转 + 强编排”的状态机。

# spec-first 第一层：项目级规范层（repo-profile）

## 最终完整技术方案文档 v1.0

---

## 1. 背景

在 AI coding / workflow-first 开发模式中，质量问题往往不来自“模型不会写”，而来自“模型拿到的决策输入质量不够高”。

常见问题包括：

* 项目长期原则没有显式表达
* 项目边界只能靠模型从代码和文档里猜
* review 关注点不稳定，随人而变
* 团队成员与 AI 在同一个项目上的约束理解不一致
* 每次任务都要重新理解“这个项目默认怎么做”

因此，需要在代码事实层之上，引入一层**轻量、稳定、团队共享**的项目级规范输入层。

这层不是为了接管流程，也不是为了增加控制状态；
而是为了：

**让 LLM 在进入 plan / work / review 前，拿到更高质量、更低歧义、更贴近项目现实的项目级规范输入。**

---

## 2. 目标

第一层 repo-profile 只解决一个问题：

> **为当前项目建立一份团队共享、机器可读、轻量稳定的项目级规范真相源，用来提升 LLM 的决策输入质量。**

它只回答这些问题：

* 这个项目是什么类型
* 这个项目长期稳定的原则是什么
* 这个项目有哪些不可违反项
* 这个项目 review 默认关注什么

---

## 3. 非目标

为了避免过度设计，第一层**明确不做**以下事情：

* 不做代码事实提取
* 不做团队级规范沉淀
* 不做候选规则抽取
* 不做复杂评分体系
* 不做状态流转控制
* 不做质量 gate 状态机
* 不做强编排式流程控制
* 不做多文件规则引擎
* 不做自动 promote / sync / verify 闭环
* 不做前端 / 后端 / 通用多目录模板体系

---

## 4. 最高指导思想

### 4.1 轻 contract

repo-profile 只表达**最小必要约束**，不表达复杂执行状态。

它应该告诉 LLM：

* 优先遵守什么原则
* 哪些边界不要打破
* 哪些约束必须满足
* review 默认该关注什么

它**不应该**告诉 LLM：

* 当前在哪个状态
* 下一步必须进哪个状态
* 哪个 gate 是否通过
* 哪个规则触发哪个流程分支

---

### 4.2 明确边界

repo-profile 是**项目级规范输入层**，不是：

* 代码事实层
* 当前任务规范层
* 团队规范层
* 控制引擎
* 状态机 runtime

它负责提供项目级规范输入，
不负责控制 workflow。

---

### 4.3 让 LLM 决策

repo-profile 的目标不是替代 LLM，而是帮助 LLM。

正确关系是：

```txt id="31k8gi"
项目级规范输入
+ 代码事实输入
+ 当前任务输入
= 更高质量的 LLM 决策
```

而不是：

```txt id="uyc7qc"
更多状态
+ 更多 gate
+ 更强编排
= 更高质量
```

---

## 5. 在整个体系中的位置

第一层 repo-profile 位于代码事实层与任务层之间，但不依赖复杂编排。

推荐的整体分层关系：

```txt id="0prm1k"
代码事实层（graph-bootstrap / code facts）
-> 项目级规范层（repo-profile）
-> 当前任务规范层（task spec）
-> LLM 决策（plan / work / review）
```

其中：

* **代码事实层**：告诉模型“代码库现在是什么”
* **repo-profile**：告诉模型“这个项目长期怎么做”
* **task spec**：告诉模型“这次任务要做什么”

---

## 6. 核心定位

一句话定版：

> **repo-profile is an input layer, not a control engine.**

中文版：

> **repo-profile 是输入层，不是控制引擎。**

这意味着：

* 它提供约束
* 它不接管流程
* 它帮助决策
* 它不替代决策

---

## 7. 目录设计

## 7.1 主目录

主文件放在：

```txt id="cc8o6q"
.spec-first/
  specs/
    repo-profile.yaml
```

原因：

* `.spec-first/` 已是当前仓库共享工作区目录
* 它和 `graph/`、`workflows/` 一样，属于 repo-shared control-plane / input 资产
* `specs/` 可明确表达这里承载的是规范输入，而不是代码事实或运行时状态
* 它比 `docs/` 更适合作为 machine-readable 的规范输入源

---

## 7.2 为什么不放 `docs/`

因为 `docs/` 更适合：

* 面向人阅读的架构说明
* 使用手册
* ADR / 设计文档
* 产品文档

而 repo-profile 的本质是：

* workflow input
* machine-readable asset
* 系统真相源

所以：

* **真相源**放 `.spec-first/specs/`
* **说明书**可放 `docs/`

例如可选补充：

```txt id="jsrj8z"
docs/repo-profile.md
```

---

## 7.3 为什么不拆前端 / 后端 / 通用目录

第一层不建议按：

* frontend
* backend
* common

拆目录。

原因：

1. 第一层是**当前项目规范**，不是模板库
2. 对前后端未分离或混合项目，这种分类会制造歧义
3. 项目级规范更适合表达“整体原则”，而不是预设技术分区
4. 分类目录更适合未来 Team Pack，而不是第一层 repo-profile

因此第一层采用：

* **单目录**
* **单文件**
* **单真相源**

## 8. 团队协作模型

## 8.1 共享一份主规范

团队共享一份：

```txt id="knhkc0"
.spec-first/specs/repo-profile.yaml
```

并提交到仓库。

这份文件是项目级规范真相源。

---

## 8.2 不采用“每人一份主规范”

不建议每个人维护一份本地主规范。
否则会导致：

* 规范分叉
* AI 行为不一致
* 团队经验无法沉淀
* review 无法对齐

---

## 8.3 允许个人本地偏好，但不是主规范

如果未来需要，可支持本地私有偏好：

```txt id="dvjcx4"
.spec-first/local/user-overrides.yaml
```

并加入 `.gitignore`。

但这不是第一层主方案的必要部分。

## 9. 核心对象定义

第一层只保留一个核心对象：

## `RepoProfile`

它描述当前项目的长期稳定规范，包含 5 类内容：

1. 项目身份
2. 项目意图
3. 核心原则
4. 不可违反项
5. review 默认关注点

---

## 10. Schema 设计

第一版采用最小字段集。

### `repo-profile.yaml`

```yaml id="2v3c6g"
schema_version: 1
repo_id: my-repo
project_type: ai-workflow-cli
languages:
  - typescript

project_intent:
  summary: Build a workflow-first AI coding CLI with explicit stage boundaries.

principles:
  - Prefer explicit stage boundaries
  - Prefer fact-backed context over assumption-heavy summaries
  - Avoid hidden coupling between major workflow stages

non_negotiables:
  - Generated files must not be edited directly
  - Contract changes require compatibility review
  - Fallback or degraded paths must be explicit

review_defaults:
  - contract compatibility
  - boundary integrity
  - fallback clarity
```

---

## 11. 字段定义

## 11.1 `schema_version`

Schema 版本号，用于未来兼容升级。

---

## 11.2 `repo_id`

仓库标识。
建议使用仓库名或根目录名。

---

## 11.3 `project_type`

项目类型。
第一版建议用粗粒度分类：

* `backend-service`
* `frontend-app`
* `fullstack-app`
* `cli-tooling`
* `ai-workflow-cli`
* `library`

不做更细分层。

---

## 11.4 `languages`

项目主语言列表，例如：

* `typescript`
* `python`
* `go`

---

## 11.5 `project_intent.summary`

一句话表达项目意图。

要求：

* 短
* 稳
* 帮助 LLM 建立项目语境
* 不追求替代 README

---

## 11.6 `principles`

项目长期稳定原则。

要求：

* 3 到 5 条为宜
* 最多不超过 7 条
* 必须短句
* 必须稳定
* 不写临时 workaround

好的例子：

* Prefer explicit stage boundaries
* Avoid hidden coupling between major modules

不好的例子：

* We should always keep quality high
* Try to write elegant and maintainable code

---

## 11.7 `non_negotiables`

不可违反项。

要求：

* 可执行
* 可判断
* 能直接影响决策边界

好的例子：

* Generated files must not be edited directly
* Contract changes require compatibility review

不好的例子：

* Code should be clean
* Please be careful when changing files

---

## 11.8 `review_defaults`

review 默认关注点。

要求：

* 只表达默认审查方向
* 不做完整 checklist
* 不做状态驱动规则

例如：

* contract compatibility
* boundary integrity
* fallback clarity

---

## 12. 写法约束

为了保持 repo-profile 轻量可维护，强制约束如下：

### 12.1 原则短句化

每条原则尽量一行。

### 12.2 只写长期约束

不写任务临时规则。

### 12.3 不可违反项必须可操作

模糊规范不应写入。

### 12.4 review_defaults 只做方向提示

不做执行引擎。

### 12.5 文件尽量控制在 30~60 行

避免快速膨胀成“大配置”。

---

## 13. 明确禁止的设计

第一层为了严格贯彻指导思想，明确禁止：

## 13.1 状态机字段

不允许加入：

* `workflow_state`
* `stage_status`
* `quality_gate_level`
* `transition_rules`
* `route_on_failure`
* `next_action_if_conflict`

---

## 13.2 强编排语义

不允许加入：

* “命中某规则必须自动进入某阶段”
* “修改某类文件必须自动触发某 reviewer”
* “低于某分数则阻止下阶段执行”

---

## 13.3 复杂 rule runtime

不允许把 repo-profile 做成：

* policy engine
* routing engine
* orchestrator
* stateful gate system

---

## 14. 与其他层的边界关系

## 14.1 与代码事实层

* 代码事实层负责“代码是什么”
* repo-profile 负责“项目原则是什么”

它们并列，而不是互相覆盖。

---

## 14.2 与任务规范层

* task spec 负责“这次任务要做什么”
* repo-profile 负责“这个项目长期怎么做”

repo-profile 不承载当前任务计划。

---

## 14.3 与团队规范层

未来如果有 Team Pack：

* Team Pack 负责跨项目复用
* repo-profile 负责当前项目

但第一层设计**不依赖 Team Pack** 存在。

## 15. Ownership 设计

为了保持简单，第一层只分为两类字段：

## 15.1 机器辅助生成

可自动推断：

默认人工维护：

* `project_intent.summary` 的最终表述
* `principles`
* `non_negotiables`
* `review_defaults`

### 核心规则

> 机器可以给初稿，人负责确认和维护核心内容。

## 16.1 `spec-first init --claude` / `spec-first init --codex`

### 作用

在初始化 host runtime 资产的同时，补种 repo-level shared spec seeds：


* 探测仓库根目录

* 写入最小模板

```bash id="n3530z"

spec-first init --codex
```

## 17. init 输入来源

`spec-first init --claude` / `spec-first init --codex` 可读取以下来源辅助生成初稿：

* 仓库目录名
* `package.json`
* `pyproject.toml`
* `go.mod`
* `Cargo.toml`
* `README.md`

但第一层 init 不追求“深度理解仓库”，只做**轻量初始化**。其中 `project_intent.summary` 必须采用固定优先级与固定回退句式，不能让脚本做自由式语义总结。

## 18. init 推断内容

init 只自动推断：

* `repo_id`
* `project_type`
* `languages`
* `project_intent.summary` 的初稿

对于原则、不可违反项和 review 默认关注点，第一版使用**单一基础模板**，由人确认修改；不按 `project_type` 分支，避免把脚本膨胀成规则映射器。

---

## 19. 基础模板策略

为了稳定可维护，init 第一版不依赖自由式 LLM 生成，也不引入按 `project_type` 分支的多套默认模板，而是使用**单一 repo-level 基础模板**。

### 19.1 基础模板

默认模板如下：

```yaml id="spzk8e"
principles:
  - Prefer explicit stage boundaries
  - Prefer fact-backed context over assumption-heavy summaries
  - Avoid hidden coupling between major workflow stages

non_negotiables:
  - Generated files must not be edited directly
  - Contract changes require compatibility review
  - Fallback or degraded paths must be explicit

review_defaults:
  - contract compatibility
  - boundary integrity
  - fallback clarity
```

---

### 19.2 `project_intent.summary` 初稿策略

`project_intent.summary` 的初稿必须可测试、可回退、可解释，建议采用以下固定优先级：

1. 优先读取 manifest 中现成的描述字段，例如 `package.json.description`
2. 若缺失，则读取 `README.md` 首个标题后的首段摘要
3. 若仍缺失，则回退为固定句式，例如 `Repository <repo_id> is a <project_type> project.`

这一步的目标是生成**最小可读初稿**，不是让脚本“理解项目”。

```ts id="0dqr8j"
async function planSharedSpecSeeds(repoRoot: string): Promise<void> {
  const profilePath = join(repoRoot, ".spec-first/specs/repo-profile.yaml");
  const readmePath = join(repoRoot, ".spec-first/specs/README.md");

  await ensureDir(dirname(profilePath));

  if (!(await exists(profilePath))) {
    const repo_id = detectRepoName(repoRoot);
    const project_type = await detectProjectType(repoRoot);
    const languages = await detectLanguages(repoRoot);
    const summary = await buildProjectIntentSummaryDraft(repoRoot, {
      repo_id,

    const profile = {
      schema_version: 1,
      repo_id,
      project_type,
      languages,
      project_intent: {
        summary,
      },
      principles: BASELINE_PRINCIPLES,
      non_negotiables: BASELINE_NON_NEGOTIABLES,
      review_defaults: BASELINE_REVIEW_DEFAULTS,
    };

    RepoProfileSchema.parse(profile);
    await writeYaml(profilePath, profile);
  }

  if (!(await exists(readmePath))) {
    await writeFile(readmePath, renderSpecsReadme());
  }
}
```

## 25. 质量提升机制

repo-profile 之所以有价值，不是因为它控制流程，而是因为它提升输入质量。

它的价值主要来自 4 个方面：

### 25.1 降低歧义

让 LLM 快速知道项目长期原则。

### 25.2 补足项目约束上下文

减少每次从代码与 README 中重新猜测项目规范。

### 25.3 提前暴露边界

让 LLM 在 plan / work / review 前就知道哪些不能做。

### 25.4 统一团队输入

让不同成员运行 workflow 时，AI 至少拿到一致的项目级规范输入。

---

## 26. 混合项目适配策略

对于前后端未分离、全栈或混合型项目，不采用目录切分，而用：

* 单文件主规范
* `project_type`
* 必要时项目整体原则表达

例如：

```yaml id="jghjv0"
schema_version: 1
repo_id: my-fullstack-app
project_type: fullstack-app
languages:
  - typescript

project_intent:
  summary: Build a full-stack product with shared contracts and clear module boundaries.

principles:
  - Keep UI concerns separate from domain logic
  - Prefer explicit boundaries between transport, domain, and infra
  - Avoid hidden coupling across major modules

non_negotiables:
  - Contract changes require compatibility review
  - Generated files must not be edited directly
  - Side-effectful flows must remain explicit

review_defaults:
  - contract compatibility
  - boundary integrity
  - shared module impact
```

这样不会因为前后端未分离而混乱。

---

## 26.1 与 `docs/contexts/` 的边界

`docs/contexts/<slug>/` 更适合承载：

* 代码事实摘要
* module map
* public entrypoints
* test map
* pitfalls / context pack

而 `repo-profile.yaml` 只承载：

* 项目长期原则
* 不可违反项
* review 默认关注点
* 轻量项目身份与意图

因此：

* `docs/contexts/` 负责“项目现在是什么”
* `repo-profile.yaml` 负责“项目长期怎么做”

二者互补，不互相覆盖。

---

## 27. 防过度设计约束

为了确保第一层长期可维护，强制遵守以下限制：

### 27.1 只允许一个主文件

不拆子文件。

### 27.2 第一版不新增 repo-profile 专用命令

第一版不新增 repo-profile 专用命令，只通过现有 `spec-first init --claude` / `--codex` 种下 seed。

### 27.3 不做状态流转

没有状态机，没有 transition。

### 27.4 不做评分

没有 score、confidence、gate level。

### 27.5 不做自动闭环

第一版不做 refresh、verify、doctor、sync，也不做自动回写或自动维护闭环。

### 27.6 不引入多分类目录

不做 frontend/backend/common 分目录。

---

## 28. 成功标准

第一层是否成功，不看功能多不多，而看以下 5 条：

1. 团队能接受仓库里存在一份共享项目规范文件
2. 新成员能快速理解项目长期约束
3. AI workflow 能稳定消费这份文件
4. 规范修改能进入正常 PR/review 流程
5. 文件长期保持轻量，没有膨胀成复杂配置系统

---

## 29. 失败信号

如果出现以下现象，说明设计开始偏离：

* 想给 repo-profile 增加状态流转字段
* 想让 repo-profile 承担流程路由职责
* 想把 quality gate 写进 repo-profile
* 想把 task spec / candidates / team-pack 混进去
* 一个主文件开始拆成很多子规则文件
* init 开始演化成复杂 merge engine

## 31. 最终结论

第一层 repo-profile 的正确设计不是“规范操作系统”，而是：

> **一份团队共享、轻量、机器可读、项目级的约束输入层。**

它严格遵循：

* 轻 contract
* 明确边界
* 让 LLM 决策

它通过提升输入质量提升输出质量，
而不是通过多状态流转和强编排去替代 LLM 的判断。

---

## 32. 一句话定版

> **repo-profile 提供项目级原则、边界和不可违反项，帮助 LLM 在更高质量输入下做决策，但它本身不承担状态机式质量控制。**

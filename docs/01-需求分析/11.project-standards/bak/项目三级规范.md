# spec-first Convention Mining & Team Pack

## 技术方案文档 v0.1

## 1. 背景与目标

### 1.1 背景

当前 `spec-first` 已经具备较强的 **代码事实层** 能力：通过 `graph-bootstrap` 将代码库转化为结构化事实、最小上下文、阶段注入索引，并作为后续 workflow 的 Stage-0 输入。此前分析中，项目已明确将 `graph-bootstrap` 定位为代码事实基础设施，而不是重型编排器。其 README 也强调清晰控制面边界，以及 `repo profile` 这类对象应被独立建模。

与此同时，现有体系更强于：

* 识别代码现实
* 组织任务级 spec
* 在 plan / work / review 中消费事实化上下文

但还缺少一条能力：

**从真实项目实践中抽取团队级公共规范，并供未来项目复用。**

这意味着需要在现有体系之上新增一条“规范蒸馏链”：

```text
代码事实
-> 项目规范
-> 候选公共规范
-> 团队审核确认
-> Team Pack
-> 新项目初始化 / 阶段注入 / Review 默认规则
```

### 1.2 问题定义

当前系统缺少的不是单一 repo 的说明文档，而是以下能力：

1. 从正在开发的项目中识别稳定工程模式
2. 区分项目局部约束与团队可复用规则
3. 将可复用规则版本化沉淀为 Team Pack
4. 在未来项目中复用 Team Pack，形成组织级工程默认行为
5. 让规范的来源、证据、置信度、适用范围可追踪

### 1.3 建设目标

本方案的目标是新增一套 **Convention Mining & Team Pack** 能力，满足：

* 从单项目中抽取 candidate conventions
* 对 convention 进行证据化、评分、分级、审查
* 产出可版本化的 Team Pack
* Team Pack 可在新项目 init、plan、work、review 时复用
* 与现有 `graph-bootstrap`、`repo-profile`、`task spec` 分层协同
* 保持 `spec-first` 一贯的“轻契约、显式边界、事实优先”设计哲学

## 3. 总体设计

## 3.1 分层模型

新增后的体系建议分为 4 层：

### Layer 0：Code Facts

由 `graph-bootstrap` 提供：

* AST facts
* symbol / dependency / flow / impact
* minimal-context
* injection-index
* provenance / confidence / fallback

回答：**代码库现在是什么**

### Layer 1：Repo Profile

当前仓库的项目级规范：

* 本项目的架构原则
* 本项目的边界规则
* 本项目的测试要求
* 本项目的 review 默认关注点
* 本项目的风险区域

回答：**这个项目应该如何被修改**

### Layer 2：Convention Extract

从当前项目抽取候选公共规范：

* 哪些模式是稳定的
* 哪些是团队可复用的
* 哪些仅是项目局部约束
* 哪些规则证据不足

回答：**这个项目里哪些实践值得上升为团队规范**

### Layer 3：Team Pack

经过审核确认、可版本化、可复用的团队规范包：

* 可用于新项目初始化
* 可作为 plan/work/review 默认注入
* 可跨项目共享
* 可按语言/项目类型继承

回答：**团队未来默认如何做**

---

## 3.2 核心能力链

```text
graph-bootstrap
  -> repo-profile
  -> convention-extract
  -> candidates + evidence + scores
  -> review queue
  -> team-pack promote
  -> future repo init / stage injection
```

## 4. 架构原则

### 4.1 事实与规范分离

* `graph-bootstrap` 负责事实提取
* `repo-profile` 负责项目级规范
* `convention-extract` 负责公共规范抽取
* `team-pack` 负责团队级规范沉淀

不得将“事实是什么”和“应该怎样做”混为一个对象。

### 4.2 规范必须 machine-readable 优先

* Markdown 用于人类阅读
* YAML / JSON 用于注入、合并、验证、版本控制

### 4.3 每条规则必须可追踪

所有 candidate / rule 必须记录：

* source_type
* confidence
* evidence_refs
* scope
* exceptions
* promote_status

### 4.4 人工确认是升格门槛

自动抽取只生成 candidate。
**进入 Team Pack 必须经过人工确认或显式 promote。**

### 4.5 规范必须支持 drift 检测

随着代码演进：

* 规范可能过期
* 规则可能局部失效
* Team Pack 在单 repo 中可能被绕过

因此需要 verify / doctor / drift-report。

## 5. 功能模块设计

## 5.1 Repo Profile

Repo Profile 是当前仓库的项目级规范画像，作为 Convention Extract 的直接输入之一。

职责：

* 汇总项目级 architecture / boundaries / testing / review / risk
* 区分 human-authored 与 machine-derived sections
* 为 plan / work / review 提供 stage-aware 注入上下文

核心产物：

* `repo-profile.yaml`
* `conventions/*.yaml`
* `stage-context/*.json`
* `diagnostics/*.json`

---

## 5.2 Convention Extract

Convention Extract 是本方案的核心新增模块。

职责：

* 从 graph facts + repo-profile + docs/tests/config 中识别稳定工程模式
* 生成 candidate conventions
* 为每条 candidate 提供证据、置信度、适用范围、例外项
* 将 candidate 放入 review queue
* 支持 promote 到 Team Pack

---

## 5.3 Team Pack

Team Pack 是组织级、团队级的规范包。

职责：

* 存储已确认的团队默认规范
* 提供 pack metadata、rulesets、catalog、examples、templates
* 供新项目 init、repo-profile sync、workflow stage injection 复用

## 8. 核心流程设计

## 8.1 Convention Extract 流程

### Step 1：收集输入

输入来源：

* graph-bootstrap facts
* repo-profile
* README / docs
* tests layout
* config files
* 变更热点与 review traces（如未来可用）

### Step 2：Pattern Mining

从输入中识别稳定模式：

* 分层结构
* 依赖方向
* 目录布局
* 命名规范
* 测试组织方式
* review 关注点
* generated/safety 规则

### Step 3：Rule Synthesis

将模式转成候选规范：

* statement
* rationale
* exceptions
* proposed scope
* evidence refs

### Step 4：Scoring

为 candidate 计算：

* stability_score
* coverage_score
* consistency_score
* doc_alignment_score
* review_signal_score
* locality_penalty
* exception_penalty

### Step 5：Classification

分类为：

* org-default
* team-reusable
* domain-specific
* project-local

### Step 6：Queueing

将 candidate 推入：

* proposed
* needs-review
* accepted
* rejected
* accepted-with-scope-limit

---

## 8.2 Promote 流程

### 输入

* candidate ids
* target pack
* scope override（可选）
* reviewer decision

### 处理

* 校验 evidence 完整性
* 写入 team-pack 对应 conventions 文件
* 更新 catalog.yaml
* 记录 changelog
* 可选同步到当前 repo-profile

### 输出

* 更新后的 Team Pack
* promote report
* version bump

---

## 8.3 Sync 流程

### 作用

将 Team Pack 同步到当前 repo 或新项目模板。

### 输入

* team-pack
* target repo-profile
* sync policy

### 输出

* merged repo-profile defaults
* updated stage injection
* sync diff report

## 9. 抽取与评分模型

## 9.1 评分公式

```text
Convention Score =
  stability_score
+ coverage_score
+ consistency_score
+ review_signal_score
+ doc_alignment_score
- exception_penalty
- locality_penalty
```

### 维度说明

#### stability_score

模式是否跨时间稳定存在。

#### coverage_score

模式是否跨多个目录/模块存在。

#### consistency_score

模式内部是否一致，是否经常被绕过。

#### review_signal_score

该模式是否经常体现为 review 重点。

#### doc_alignment_score

是否与已有文档/约定一致。

#### exception_penalty

例外越多，扣分越高。

#### locality_penalty

越局部、越特定于单项目，扣分越高。

---

## 9.2 阈值建议

* `score >= 0.85`：promote-ready
* `0.70 ~ 0.85`：needs-review
* `0.50 ~ 0.70`：hold
* `< 0.50`：reject or keep project-local

## 10. 规则分级模型

## P0：org-default

适合全团队默认使用。

例：

* generated 文件不允许直接手改
* schema 变更必须说明兼容性

## P1：team-reusable

大多数项目适用。

例：

* review 先检查 contract，再看实现
* 高风险目录变更要附影响说明

## P2：domain-specific

只适用于特定项目类型。

例：

* AI workflow 项目必须暴露 fallback quality signaling

## P3：project-local

仅当前仓库适用。

例：

* 特定模块命名
* 某个历史兼容目录布局

## 13. Stage Injection 设计

## 13.1 注入来源

最终某个 stage 的输入建议由三部分组成：

```text
stage input =
  team-pack stage-context
  + repo-profile stage-context
  + graph-bootstrap minimal-context
  + current task spec
```

### 含义

* Team Pack：团队默认规范
* Repo Profile：项目特有约束
* Graph Bootstrap：代码事实
* Task Spec：当前任务目标

---

## 13.2 stage-aware 注入示例

### Work

* naming
* edit-safety
* path ownership
* generated file policy

### Review

注入：

* review defaults

## 14. 验证与诊断设计

## 14.1 Profile Verify

检查：

* profile 是否缺关键 section
* repo-profile 与 graph facts 是否冲突
* protected_paths 是否失效
* stage injection 是否缺映射

输出：

* `diagnostics/profile-health.json`
* `diagnostics/drift-report.json`

---

## 14.2 Convention Verify

检查：

* candidate evidence 是否缺失
* score 是否过低
* promoted rule 是否与 repo 现实冲突
* Team Pack 应用后是否出现严重不适配

---

## 14.3 Drift 检测

drift 示例：

* Team Pack 要求显式 stage boundaries，但 repo 中出现隐式共享状态
* Team Pack 要求 generated 文件不手改，但 repo 中存在直接修改
* repo-profile 规定 transport 不能直连 persistence，但 graph facts 检测到反例

## 15. 与现有系统的集成关系

## 15.1 与 graph-bootstrap 的关系

`graph-bootstrap` 仍然保持纯事实层定位。
它不负责 Team Pack，不负责 rule promote。

它只提供：

* AST / symbol / impact / flow
* minimal-context
* quality signals

Convention Extract 消费其产物，但不反向污染 graph-bootstrap。

---

## 15.2 与 repo-profile 的关系

repo-profile 是 Convention Extract 的直接上游。

* repo-profile 描述当前项目规范
* convention-extract 从 repo-profile 和事实层抽取可复用部分
* Team Pack 可反向 sync 到 repo-profile

---

## 15.3 与 plan/work/review 的关系

Team Pack 不替代 task spec，而是作为默认规范注入层：

* plan：限制设计空间
* work：限制实现边界
* review：提供默认检查清单

---

## 16. 数据所有权策略

## 16.1 human-authored

由维护者手工编写，自动流程不得直接覆盖。

## 16.2 machine-derived

由 refresh / extract 自动生成，可被重建。

## 16.3 mixed sections

采用 merge 策略，human 优先，machine 提供候选 patch。

---

## 17. Merge 策略

### 原则

* human-owned 不覆盖
* machine-derived 可刷新
* promoted rules 进入 Team Pack 后只能通过显式修改变更
* sync 到 repo-profile 时应生成 diff report

### 策略类型

* overwrite-never
* append-only
* merge-by-id
* replace-machine-section-only

## 19. 风险与应对

## 风险 1：抽出来的是“现状”，不是“规范”

应对：

* 引入 locality_penalty
* 强制人工 promote
* 保留 exceptions 与 scope

## 风险 2：单仓库抽取样本偏差

应对：

* 初期仅作为 candidate
* v2 再做多仓库共识

## 风险 3：Team Pack 过于僵化

应对：

* 支持 scope、exceptions、domain-specific
* repo-profile 始终可覆盖 pack 默认值

## 风险 4：规范与代码现实漂移

应对：

* verify / doctor / drift-report
* refresh 与 sync 流程

## 风险 5：与现有 graph-bootstrap 职责混淆

应对：

* 坚持事实层与规范层解耦
* 只消费，不反向耦合

* verify 能识别主要 drift

## 20.3 业务指标

* 新项目初始化减少重复规范建设
* review 一致性提升
* plan/work/review 的默认质量提高
* 团队规范沉淀成本下降

## 21. 对外定义建议

### 英文

**Convention Extract mines evidence-backed reusable conventions from a live project, separates project-local constraints from team-level defaults, and promotes validated rules into versioned Team Packs.**

### 中文

**Convention Extract 会从真实项目中挖掘有证据支撑的可复用规范，区分项目局部约束与团队默认规则，并将验证通过的规则沉淀为可版本化的 Team Pack。**

---

## 22. 最终结论

本方案不是为 `spec-first` 再添加一个静态“规范文件夹”，而是新增一条完整的能力链：

**事实层 → 项目规范层 → 候选公共规范层 → 团队规范包层**

它与现有体系关系如下：

* `graph-bootstrap`：告诉模型代码库现在是什么
* `repo-profile`：告诉模型当前项目应该如何被修改
* `convention-extract`：从项目中抽取值得团队复用的工程规范
* `team-pack`：将这些规范沉淀为未来项目可继承的默认规则

这条链将使 `spec-first` 从“帮助 AI 理解当前代码库”，进一步升级为：

**从真实项目中学习团队工程习惯，并将其产品化为可复用规范资产。**

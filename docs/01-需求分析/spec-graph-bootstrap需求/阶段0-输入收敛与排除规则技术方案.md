# 阶段0：输入收敛与排除规则技术方案

> 目标：将 CRG 的输入集合从“全盘扫描”收敛为“可维护、可预测、可解释”的最小集合，降低大仓构建耗时与噪声污染风险。

---

## 一、问题定义

在大型仓库中，若不做输入收敛，`crg build` 会受到以下问题影响：

1. 性能开销过大：依赖目录与构建产物会放大 SHA/解析/入库成本。
2. 分析结果失真：`hub_nodes`、`communities`、`impact` 容易被三方代码污染。
3. 增量不稳定：生成物频繁变化导致 `changed_files` 假阳性增加。
4. 行为不可解释：同一仓库在不同机器上输入集合不一致。

---

## 二、设计目标

1. 输入集合最小化：默认仅扫描有价值源码。
2. 行为可预测：同仓库多次构建输入稳定。
3. 可配置：仓库级可增删排除规则，无需改代码。
4. 多端友好：支持前端/App/PC/后端混合仓库。
5. 安全优先：敏感文件和运行时产物默认不入图。

---

## 三、配置文件定义

### 3.1 文件名与位置

- 配置文件：`<repo_root>/.spec-first-graphignore`
- 语法：兼容 `.gitignore`（支持 `*`、`**`、`?`、`!`、注释行）

### 3.2 配置职责

- 定义“额外排除规则”与“项目定制排除规则”
- 不承担语言识别职责（语言识别由 parser 决定）

### 3.3 默认建议内容

- 已在仓库根目录提供默认模板：`/.spec-first-graphignore`
- 硬排除（所有项目）：`.git/**`、`.spec-first-graph/**`、`.code-review-graph/**`、`graphify-out/**`
- 构建产物排除规则见 §6.1（`build/` 细化方案）
- `Pods/` 的排除**由 iOS 适配器自动处理**，用户无需在配置文件中维护 Pod 命名约定（见 §6.2）

### 3.4 与 `.gitignore` 的关系

1. `.gitignore` 解决“版本控制是否跟踪”的问题。
2. `.spec-first-graphignore` 解决“CRG 是否索引”的问题。
3. 两者不互相替代；建议将二者都维护在仓库根目录。
4. 若同一路径在 `.spec-first-graphignore` 被排除，默认不进入 CRG 输入集合。
5. 白名单 `!` 规则只对索引范围生效，不应绕过安全硬规则（敏感文件过滤）。

---

## 四、输入收敛流水线

`crg build` 前置输入收敛应采用如下顺序：

1. 获取候选文件集合：优先 `git ls-files`（tracked-only）
2. 非 git 仓库回退：全目录递归枚举（Node.js 侧用 `fs.readdirSync` 递归或 `fast-glob`）
3. 应用内置默认排除规则
4. 应用 `.spec-first-graphignore` 用户规则
5. 执行二进制过滤与敏感文件过滤
6. 输出最终输入集合（`final_inputs`）
7. 从 `final_inputs` 推导 `present_languages`（不再做第二次全盘扫描）
8. `final_inputs` 供 `detectChangedFiles` 与 parser 使用，`present_languages` 供 grammar 按需加载使用

> 说明：默认模式为 `tracked-only`。本地调试可选 `tracked+untracked`，但仍需经过同一排除链路。
>
> **iOS 项目自动升级规则**：`extraIncludes` 是候选集过滤器，不能将未被 `git ls-files` 输出的文件注入候选集。因此，当 `isIos=true` 且 Podfile.lock 中存在本地 Pod（EXTERNAL SOURCES 有 `:path:` 字段）时，`buildIosAdapter` **自动将模式升级为 `tracked+untracked`**，并在 stderr 输出 warning 说明原因。用户可通过 `--mode=tracked-only` 手动覆盖（此时本地 Pod 无法入图，Pod 边界社区检测退化）。
>
> 无本地 Pod 的 iOS 项目（纯三方依赖 + 自有代码全部在 git 跟踪范围内）维持 `tracked-only`。

### 4.2 语言集合推导约定

- `detectPresentLanguages` 的输入应为 `final_inputs`（文件路径数组），而不是 `repoRoot` 目录递归扫描。
- 语言集合推导必须复用输入收敛结果，避免”索引范围”和”语言检测范围”两套口径不一致。
- 推导规则建议：
  1. 常规扩展名：`ext = path.extname(file).slice(1).toLowerCase()`
  2. 扩展名别名：如 `.mm -> m`
  3. `.h` 特殊判定：基于文件内容与同名实现文件启发式路由（ObjC/C++/C）

**两套机制分工**：

- `.spec-first-graphignore` 流水线负责”哪些文件进入 `final_inputs`”（输入集合过滤）。
- `detectPresentLanguages(final_inputs)` 的内部 `ignore` 参数用于目录递归遍历时跳过已知无关目录（仅在传入 `repoRoot` 时生效）；当传入已收敛的 `final_inputs` 时，`ignore` 参数不再起作用。
- 两套机制不重复：流水线管输入范围，`detectPresentLanguages` 管语言推导逻辑。

### 4.1 输入模式矩阵（建议）

| 模式 | 候选文件来源 | 适用场景 | 风险 |
| --- | --- | --- | --- |
| `tracked-only`（默认） | `git ls-files` | CI、日常稳定构建 | 新建但未 `git add` 文件不会入图；iOS `Pods/` 通常不被跟踪（见上方 iOS 说明） |
| `tracked+untracked` | `git ls-files` + `git status --porcelain` 未跟踪文件 | 本地开发调试、iOS Pod 社区检测 | 输入集合波动较大，增量命中率下降 |
| `all-files`（不推荐默认） | 全目录递归枚举 | 非 git 仓库、一次性迁移 | 噪声最高，性能最不稳定 |

## 五、规则优先级

优先级从高到低：

1. 安全硬规则（敏感文件、路径安全规则）
2. 显式命令行参数规则（如后续支持 `--exclude` / `--include`）
3. `.spec-first-graphignore` 仓库级规则
4. 内置默认排除规则
5. 语言支持过滤（`detectPresentLanguages(final_inputs)`）

建议保留 `!` 白名单能力，但不允许绕过安全硬规则。

### 5.1 规则冲突处理

1. 同一路径同时命中“排除规则”和“白名单规则”时，先按规则顺序计算。
2. 若白名单目标命中安全硬规则（如敏感文件模式），仍应拒绝入图。
3. 建议在构建日志中输出 `ignored_by_rule` 统计，便于解释“为什么没被索引”。

## 六、多端项目推荐排除集

### 6.1 必选（所有项目）

**图谱元数据（硬排除，不可白名单）**

- `.spec-first-graph/**`（spec-graph-bootstrap 产物目录）
- `.code-review-graph/**`（code-review-graph Python 版与 Node.js 版共用的产物目录）
- `.git/**`
- `graphify-out/**`

**构建产物**

`build/` 的排除粒度按项目类型不同：

| 项目类型 | 推荐规则 | 说明 |
| --- | --- | --- |
| Web / Node / iOS / Go | `build/**` | 整个 build 目录均为产物 |
| Android / KMP | `build/intermediates/**`<br>`build/outputs/**`<br>`build/tmp/**` | 保留 `build/generated/**`（AIDL、kapt、BuildConfig 等注解生成源，属 API 契约，有入图价值） |

```
# .spec-first-graphignore 示例（Android 项目）
build/intermediates/**
build/outputs/**
build/tmp/**
# build/generated/** 不排除
```

`dist/**` 与 `target/**`：

- `dist/**`：Web/Node 构建输出，通常应排除。**例外**：monorepo 中已发布的库包（`packages/*/dist/` 被提交到仓库），可用白名单 `!packages/*/dist/**` 保留。
- `target/**`：Rust、Java Maven 构建输出，通常应排除。

### 6.2 移动端（iOS/Android）

**iOS — Pods 目录（适配器驱动，无需用户维护白名单）**

`Pods/` 同时包含外部三方 Pod 和内部业务 Pod，二者处理方式相反：

| 类型 | 判定依据 | 处理 |
| --- | --- | --- |
| 外部三方 Pod | `Podfile.lock` PODS 段有版本号，EXTERNAL SOURCES 无 `:path:` | 排除 |
| 内部业务 Pod | `Podfile.lock` EXTERNAL SOURCES 段有 `:path: ../PodName` | **保留**（§14.8.3 Pod 边界社区检测依赖） |

**判定由 `computePodExcludePaths(repoRoot, podlockPath)` 自动完成**（见《阶段0-核心算法实现参考》§14.8.1）。适配器在 `collectInputFiles` 前注入 `extraExcludes / extraIncludes`，精确排除三方 Pod 目录，保留本地 Pod 目录。

`.spec-first-graphignore` 中**不需要**写 `Pods/**` 或任何 Pod 相关规则——适配器已处理。

> **降级策略**：若 `Podfile.lock` 不存在或无 EXTERNAL SOURCES，`computePodExcludePaths` 安全降级为排除整个 `Pods/**`。

其余 iOS/Android 排除项：

- `DerivedData/**`（Xcode 编译缓存）
- `Carthage/**`（Carthage 依赖目录）
- `.gradle/**`（Gradle 构建缓存）
- `.m2/**`（Maven 本地仓库）

Android 特有（见 §6.1 中的 build/ 细化规则）：

```
build/intermediates/**
build/outputs/**
build/tmp/**
```

### 6.3 Web / Node

- `node_modules/**`
- `dist/**`（应用层项目；库包按需白名单）
- `.next/**`
- `.nuxt/**`
- `.turbo/**`
- `.cache/**`（webpack / babel / jest 缓存）
- `coverage/**`（Jest / Istanbul 覆盖率报告）
- `.nyc_output/**`

### 6.4 Python

- `.venv/**`
- `venv/**`
- `__pycache__/**`
- `*.egg-info/**`
- `dist/**`（setuptools 打包产物）
- `.pytest_cache/**`

### 6.5 Go / 后端通用

- `vendor/**`（Go modules vendor 目录，等同 node_modules，不应入图）
- `bin/**`（编译输出）
- `target/**`（Rust / Java Maven 构建产物）

### 6.6 本项目专项

- `graphify-out/**`（已在 §6.1 列出，此处作显式提醒）

## 七、验收与观测

### 7.1 验收标准

1. 在包含 `Pods/`（含三方和本地 Pod）、`graphify-out/`、`build/`、`dist/`、`.git/` 的仓库中：三方 Pod 和产物目录输入文件数为 0；本地 Pod 正常入图（iOS 项目）。
2. 同仓库连续两次 `crg build`，无源码变更时 `changed_files=0`。
3. 输入收敛后 `crg build` p50/p95 时间下降（记录基线对比）。
4. `communities` 与 `hub_nodes` 不再出现依赖目录主导结果。

### 7.2 建议指标

1. `input_files_total`
2. `input_files_after_ignore`
3. `input_files_by_language`
4. `ignored_files_by_rule`
5. `build_duration_ms`

### 7.3 最小验收用例（建议纳入 fixture）

| 用例 | 输入条件 | 预期 |
| --- | --- | --- |
| Pod 精确隔离（iOS） | 仓库含 `Pods/AFNetworking/`（三方）与 `Pods/HSTradeModule/`（本地，Podfile.lock 有 `:path:`） | 三方 Pod 计数为 0；本地 Pod 正常入图（`computePodExcludePaths` 自动计算，无需配置文件白名单） |
| Pod 全排除降级 | iOS 项目 Podfile.lock 中 EXTERNAL SOURCES 为空 | 整个 `Pods/**` 安全排除，不报错 |
| 非 iOS 项目 Pods 目录 | 非 iOS 项目偶然存在 `Pods/` 目录 | 无 Podfile.lock → `isIos=false` → 走默认流水线；`.spec-first-graphignore` 中的 `Pods/**` 规则生效 |
| 产物目录隔离 | 仓库含 `graphify-out/`、`dist/` | 产物目录不入图，不影响 `changed_files` |
| tracked-only 稳定性 | 无源码变更连续两次 build | 第二次 `changed_files=0` |
| 白名单语义 | 排除 `generated/**` 且白名单 `!generated/keep.ts` | 仅 `keep.ts` 入图（若未命中安全硬规则） |
| 安全硬规则优先 | 白名单尝试包含敏感文件 | 敏感文件仍被拒绝入图 |

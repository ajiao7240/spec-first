## 1. 架构决策总览

### 1.1 核心原则

`spec-graph-bootstrap` 的 Fact Inventory 提取遵循三层工具栈：

```
Tier 1: spec-first crg CLI（内嵌，无需注册）← primary，AST-grade Observed 事实
Tier 2: Serena MCP                          ← secondary，semantic-level 补充
Tier 3: Built-in（Grep/Glob）               ← fallback，heuristic-level Inferred 事实
```

**Tier 1 的调用方式**：通过 `Bash(spec-first crg <command> --repo=<target>)` 执行，输出 JSON 到 stdout，封装在统一 envelope 内（见 §1.4）。无需 MCP server 注册，无需重启宿主，安装 `spec-first` 即可用。

选型依据：

| 判断维度 | spec-first crg CLI | Serena | Built-in |
|---|---|---|---|
| 事实来源 | tree-sitter AST（确定性） | 语义索引（推断性） | 文本匹配（启发式） |
| 入口点识别 | `crg flows` 直接返回 | `find_symbol` 手动 | Grep 启发式 |
| 调用关系 | `CALLS` / `IMPORTS_FROM` 边 | `find_referencing_symbols` | 不支持 |
| 测试映射 | `is_test=1` 标记 + `imports_from` 启发式（`tests_for` 查询，`inference_reason: naming_convention`）| 路径启发式 | 路径启发式 |
| 风险信号 | fan-out 可查、impact 可算 | 需手动 | 不支持 |
| 增量更新 | `crg build` 内建 SHA 变更检测 | 无 | 无 |
| 持久化 | SQLite（跨会话复用） | 无 | 无 |

### 1.2 为什么选 spec-first crg CLI 为主工具

**决策：spec-first crg CLI（CRG Node.js 内嵌实现）作为 Tier 1，Serena 作为 Tier 2，Built-in 作为 Tier 3。**

理由：

1. **零安装开销**：CRG 能力已内嵌在 `spec-first` npm 包中（`src/crg/`），`npm install -g spec-first` 即完成，无需额外 Python 环境、无需手动注册 MCP server、无需重启宿主。
2. **Observed 事实最可信**：tree-sitter 解析出的是确定性结构——`CALLS` 边存在就是存在。这比语义推断和文本匹配更符合 "Facts First" 原则。
3. **数据模型直接对应**：CRG 的 `NodeInfo + EdgeInfo` 与 Fact Inventory schema 几乎 1:1 映射。
4. **增量刷新天然对齐**：CRG 内建文件 SHA 检测和图增量更新，`fingerprints.json` 可直接复用其结果。
5. **统一 envelope 输出**：所有子命令输出 `{ schema_version, generated_at, repo_root, degraded, warnings, data }` 格式，Claude 读取结果只需关注 `.data` 字段。

### 1.3 与现有 spec-bootstrap 的兼容性

本方案不改 spec-bootstrap 的任何逻辑或产物。分析模式扩展为：

```
spec-bootstrap:       Enhanced(Serena) / Basic
spec-graph-bootstrap: Full(CRG+Serena) / Enhanced(Serena) / Basic
```

新增 `Full` 模式仅在 CRG 图已构建时启用，不改变 `Enhanced` 和 `Basic` 的已有语义。

### 1.4 CLI 输出 envelope 规范

所有 `spec-first crg <command>` 输出格式：

```json
{
  "schema_version": "crg-cli/v1",
  "generated_at": "<ISO datetime>",
  "repo_root": "<absolute path>",
  "degraded": false,
  "warnings": [],
  "data": { ... }
}
```

**使用规则**：
- Claude 读取 Bash 输出后，始终从 `.data` 字段提取具体内容
- `degraded: true` 表示 CLI 内部降级（如 better-sqlite3 加载失败），需触发 Enhanced 模式降级
- `warnings[]` 中的告警须传播到 `generation_errors[]` 或降级报告

## 2. 工具能力清单

### 2.1 spec-first crg CLI 子命令（Tier 1）

数据库路径：`<repo_root>/.spec-first-graph/graph.db`（WAL 模式）

阶段 2 使用以下子命令：

| 子命令 | 用途 | 阶段 2 使用场景 |
|---|---|---|
| `spec-first crg build --repo=<path>` | 全量/增量构建图（含 postprocess） | Phase 0：用户确认后构建 |
| `spec-first crg stats --repo=<path>` | 图统计（节点数、边数、构建时间） | Phase 0：探测图是否存在与有效 |
| `spec-first crg context --repo=<path>` | 超紧凑概览（~100 token） | Phase 1 入口：获取项目全局概览 |
| `spec-first crg flows --repo=<path>` | 执行流列表（按 criticality 排序） | Phase 1 Stage B：入口点识别 |
| `spec-first crg flow --id=<flow_id> --repo=<path>` | 单条执行流详情（`flow_id` 来自 `crg flows` 返回的 `data.items[].flow_id`）| Phase 1 Stage B：入口点深度分析 |
| `spec-first crg communities --repo=<path>` | 代码社区列表；每项含 `community_id`（字符串，如 "tests"/"crg/1"） | Phase 1 Stage B：模块结构识别 |
| `spec-first crg community --id=<community_id> --repo=<path>` | 单个社区详情（`community_id` 来自 `crg communities` 返回的 `data.items[].community_id`）| Phase 1 Stage B：模块深度分析 |
| `spec-first crg architecture --repo=<path>` | 架构总览（跨社区耦合） | Phase 1 Stage B：项目级结构摘要 |
| `spec-first crg large-functions --min-lines=N --repo=<path>` | 超阈值函数/类 | Phase 1 Stage C：风险信号 |
| `spec-first crg impact --symbol=<node_id> --depth=N --repo=<path>` | 符号影响半径（反向 BFS）；`node_id` 须为节点 ID 字符串（symbol_key 格式） | Phase 1 Stage C：fan-out 风险信号 |
| `spec-first crg search <keyword> --repo=<path>` | FTS5 关键词搜索节点，返回含 `node_id`（字符串 symbol_key）的结果列表 | Phase 1 Stage A：data_shapes / layer 识别，及 crg query 前置 ID 查找 |
| `spec-first crg query --pattern=<type> --symbol/--module/--subject=<node_id> --repo=<path>` | 8 种图查询；参数须为**节点 ID 字符串（symbol_key 格式）** | Phase 1 Stage B/C：各类关系查询 |
| `spec-first crg god-nodes --repo=<path>` | 过度中心化节点（in_degree 前 5%，排除 module 节点），以 FactItem[] 格式输出；`confidence: 'Inferred'` | Phase 1 Stage C：high_fanout 风险信号 / `risk-signals.json.crg_metrics.top_hubs` |
| `spec-first crg detect-changes --since=<sha> --repo=<path>` | 变更风险评分 | Phase 4 刷新 |
| `spec-first crg review-context --since=<sha> --repo=<path>` | diff + 影响分析 + 测试 | review workflow 按需分析工具（`spec-review` 消费，需指定 git ref；非阶段 2 批量执行，不用于生成静态 review-change.md）|
| `spec-first crg affected-flows --since=<sha> --repo=<path>` | 受变更影响的执行流列表 | Phase 4 刷新 / `spec-review` 按需调用（非阶段 2 批量执行）|
| `spec-first crg surprising-connections --repo=<path>` | 跨社区意外耦合边列表 | 可选：Phase 1 Stage B 补充架构异常信号（非必须，视项目规模决定是否调用）|
| `spec-first crg postprocess --repo=<path>` | 构建后处理（社区检测、流分析等） | 由 `crg build` 自动调用，**不在 spec-graph-bootstrap 中单独执行** |

**`crg query` 支持的 pattern 及其参数名**：

| pattern | 参数名 | 含义 | 对应 Fact Inventory 字段 |
|---|---|---|---|
| `callers_of` | `--symbol` | 谁调用了 X | risk_signals.fan_in |
| `callees_of` | `--symbol` | X 调用了谁 | modules 核心依赖方向 |
| `importers_of` | `--module` | 谁导入了 X | integrations / risk_signals |
| `importees_of` | `--module` | X 导入了什么 | modules 依赖方向（补充）|
| `dependents_of` | `--module` | X 的上游依赖方 | integrations 依赖链 |
| `dependencies_of` | `--module` | X 依赖什么 | modules 核心依赖方向 |
| `tests_for` | `--subject` | X 的测试有哪些 | testing_surface |
| `similar_to` | `--symbol` | 与 X 同名/同签名的符号 | 重复实现检测 |

> ⚠️ **不支持的 pattern**（原 MCP 版有，CLI 无）：
> - `file_summary`（列出文件内所有节点）→ 替代方案：直接 Read 源文件，或 `crg search <basename>`
> - `children_of`（列出节点的子成员）→ 替代方案：直接 Read 源文件获取字段列表

## 3. Phase 0：探测扩展

在继承 spec-bootstrap 的 Host Readiness Gate 基础上，新增 CRG 图状态检测和分析模式判定。

### 3.0 Slug 生成规则

`<slug>` 用于构造所有产物路径（`.context/spec-first/bootstrap/<slug>/` 和 `docs/contexts/<slug>/`）。

**生成规则**（在 Phase 0 最先确定，后续所有步骤使用同一值）：

```
slug = path.basename(path.resolve(<target>))
     # 即目标仓库根目录名，如 /Users/me/projects/my-app → slug = "my-app"
```

**特殊情况**：
- 若 `slug` 包含空格或特殊字符，替换为连字符（`-`）
- 若宿主提供 `--slug=<custom>` 参数，使用用户指定值覆盖
- Monorepo 子包分析（v1 不支持）：v1 始终以仓库根目录名作为 slug

**slug 确定后立即输出给用户**：

```
📁 Slug: my-app

             docs/contexts/my-app/
```

### 3.2 CRG 图状态检测（CLI 方式）

```
Step 3: CRG 图状态检测

3a. 检查 DB 文件是否存在：
    DB 路径: <target>/.spec-first-graph/graph.db

    [1] DB 文件不存在
        → crg.indexed = false
        → 跳转到 3b（提示用户构建）

    [2] DB 文件存在 → 执行: spec-first crg stats --repo=<target>

        [2a] 命令返回非零退出码，或 envelope.degraded = true
             → crg.indexed = false
             → 输出 warnings 中的错误信息到用户
             → 跳转到 3b

        [2b] 命令成功，但 data.node_count = 0 且 data.edge_count = 0
             → crg.indexed = false（图空，构建未完成）
             → 跳转到 3b

        [2c] 命令成功，data.node_count > 0（图非空）
             → crg.indexed = true
             → 记录 data.last_built, data.node_count, data.edge_count 供 fingerprints.json 使用
             → 跳转到 3c（stale 检测）

3b. 图未就绪时的用户提示：
    "检测到 spec-first crg CLI 可用，但当前项目尚未构建图索引（或图为空）。
     是否现在构建？（耗时视项目大小：普通项目 2-5 分钟，大型 monorepo 可达 10 分钟）
     选择「是」将执行 spec-first crg build 后继续。
     选择「否」将降级到 Enhanced/Basic 模式。"

    → 用户选「是」:
      Bash: spec-first crg build --repo=<target>
      等待完成（如超时 >10 分钟 → 自动降级，写入 generation_errors）
      构建成功 → crg.indexed = true，继续 Step 3c
      构建失败 → crg.indexed = false，自动降级到 Enhanced 模式

    → 用户选「否」:
      crg.indexed = false → 降级到 Enhanced 或 Basic

3c. Stale 检测（仅在 crg.indexed = true 且 fingerprints.json 已存在时执行）：
    读取 fingerprints.json.inputs.crg.graph_last_built
    对比当前 crg stats 返回的 data.last_built
    → 不一致：打印 ⚠️ "CRG 图已更新（上次: X，当前: Y）
                受影响产物: code-facts/public-entrypoints.md, code-facts/test-map.md, code-facts/high-risk-modules.md"

    读取 fingerprints.json.inputs.files 中的关键文件 SHA
    对比当前实际文件 SHA（Read 后计算）
    → 不一致：打印 ⚠️ "文件 <path> 已变更，受影响产物: <依赖该文件的产物列表>"

    如有任何 stale 信号：
    追加 "⚠️ 检测到 N 个 stale 信号，建议重新运行 spec-graph-bootstrap 以刷新产物。"
    stale 不阻断流程，用户可选择继续使用现有产物。
```

### 3.3 分析模式判定

```
if crg.indexed = true:
    mode = Full
    graph_support_state = local-available
    code_facts_confidence = high

elif serena.ready:
    mode = Enhanced
    graph_support_state = crg-not-indexed  # CRG CLI 可用但图未构建/未索引，不是"仅本地"
    code_facts_confidence = medium

else:
    mode = Basic
    graph_support_state = unavailable
    code_facts_confidence = low
```

> **注**：
> - 原方案的 `graph_support_state = "external-available"` 改为 `"local-available"`，因为 CRG 现在是内嵌 CLI，不是外部 MCP server。
> - Enhanced 模式的状态由 `"local-only"` 改为 `"crg-not-indexed"`，消除歧义：Enhanced 模式表示"CRG CLI 存在但图未构建/为空/不可用"，不表示"只能本地访问"。`unavailable` 保留给 Basic 模式（Serena 也不可用的最低限度情况）。

## 4. Phase 1：事实提取流程

### 4.1 提取器总览

> **"并行"的执行语义**：本文件中"Stage 内并行"指在同一 Claude response 中批量发出该 Stage 的全部 **Bash tool call**（Claude 支持在单个 response 中并行执行多个独立 Bash 命令）。Stage 之间有数据依赖，必须顺序执行（等待上一 Stage 所有 Bash 调用返回并完成 JSON 写入后，再发出下一 Stage 的 Bash 调用）。Stage B 内部同样有 Round 间依赖：B-Round1 → B-Round2 → B-Round3 各自有前置数据要求，不得合并为同一 response 发出（B-Round2 需要 B-Round1 的 flow ID / community_id，B-Round3 需要 B-Round2 的 `crg search` 节点 ID）。

**Phase 1 触发（单独一步，先于 Stage A）**：

```bash
# 单独执行，获取全局概览用于驱动后续 Stage 优先级
spec-first crg context --repo=<target>
```

读取 `data.top_flows`、`data.top_communities`、`data.top_hubs`，决定 Stage B/C 中哪些查询优先、哪些社区需要深度分析。

> ⚠️ **`crg context` 字段说明**：
> - `crg context` 与 `crg flows` 均返回 `flow_id` 字段（字符串 symbol_key 格式，如 `"flow:src/foo.js#function#bar#L10:20"`），可直接传给 `crg flow --id=<flow_id>`
> - `top_communities[i].community_id`（`crg context` 用 `community_id`）与 `crg communities` 的 `items[i].community_id` 名称一致
> - `top_hubs` 目前基于节点 rowid 降序排列（PageRank 尚未实现），不代表真实连通度；v1 中优先依赖 `top_flows` 和 `top_communities` 驱动优先级，`top_hubs` 仅供参考

### 4.2 Project Identity Extractor

目标：识别项目名、主语言、主框架、仓库形态。

| 工具层 | Bash 调用 | 置信度 |
|---|---|---|
| **Full** | `spec-first crg stats --repo=<target>`（语言分布）+ `Read(package.json 等)` | Observed |
| **Enhanced** | `Read(<主配置文件>)` + Glob 框架标记文件 | Observed |
| **Basic** | `Glob("package.json")` + `Glob("go.mod")` + `Read` | Observed |

说明：项目身份无论哪个模式都能达到 Observed，因为元文件是确定性的。

输出字段：
```yaml
project_identity:
  name: string          # 从 package.json.name 或目录名
  primary_language: string
  primary_frameworks: [string]
  repo_shape: monorepo | service | library | cli
```

### 4.3 Entrypoint Extractor

目标：识别 HTTP 入口、CLI 入口、worker/cron 入口、SDK/public export 入口。

| 工具层 | Bash 调用 | 置信度 |
|---|---|---|
| **Full** | `spec-first crg flows --repo=<target>` → 按 criticality 排序的执行流（每项含 `flow_id`、`entry_node`、`criticality`、`node_count`；`flow_id` 为字符串，无 `name` 字段）| Observed |
| **Full 补充** | `spec-first crg flow --id=<flow_id> --repo=<target>` → 单流详情（关联节点列表、entry_node）；**仅对 criticality 最高的 top-5 流调用**，避免大量 Bash 调用；`flow_id` 来自 `crg flows` 的 `data.items[].flow_id`（`crg context` 的 `top_flows[i].flow_id` 同名一致）| Observed |
| **Full 细化** | `Read(<flow_node.file_path>)` → 读入口节点源文件（`crg flow` 返回的 `data.nodes[0].file_path`，`nodes` 按 `position` 排序，position=0 为入口节点），提取导出符号；替代原 `file_summary` 模式 | Observed |
| **Enhanced** | `mcp__serena__search_for_pattern(substring_pattern="@app.get\|@router.post\|main(")` | Inferred |
| **Basic** | `Grep("app\\.get\\|app\\.post\\|router\\.\\|main\\s*\\(")` + `Glob("src/main.*")` | Inferred |

> **Full 模式优势**：`crg flows` 内建框架装饰器模式识别（`@app.get`、`@router.post`、`@RequestMapping` 等），直接给出入口函数的限定名、位置、调用深度。这是 AST 级别的 Observed 事实。

输出字段（每条）：
```yaml
- path: string
  symbol: string          # 限定名，如 src/server.ts::app.listen
  kind: http | cli | worker | sdk
  summary: string
  confidence: observed | inferred
  inference_reason: string | null   # observed 时填 null；inferred 时必填
  evidence:
    - file: string
      symbol: string
  updated_at: ISO datetime
```

### 4.4 Module Structure Extractor

目标：顶层目录职责、模块边界、模块间关系。

| 工具层 | Bash 调用 | 置信度 |
|---|---|---|
| **Full** | `spec-first crg communities --repo=<target>` → 代码社区（CRG 内建 3-Pass 算法） | Observed |
| **Full 补充** | `spec-first crg community --id=<community_id> --repo=<target>` → 社区内节点列表（`data.members[]`，含各节点的 `id`、`name`、`file_path`）；`community_id` 来自 `crg communities` 结果的 `data.items[].community_id`（注意：字段名是 `community_id`，不是 `id`）| Observed |
| **Full 补充** | `spec-first crg query --pattern=importees_of --module=<member_node_id> --repo=<target>` → 模块间依赖方向；`module` 取 `crg community` 返回的 `data.members[0].id`（成员节点 ID 字符串 symbol_key，非文件路径）；不硬编码路径 | Inferred |
| **Full 概览** | `spec-first crg architecture --repo=<target>` → 基于社区结构的架构总览（含 `hub_nodes`）；`hub_nodes` 服务于 Phase 3 `module-map.md`，`risk-signals.json.crg_metrics.top_hubs` 由 Stage C `crg god-nodes` 独立填充（消费方不同，非重复计算）| Observed |
| **Enhanced** | `mcp__serena__get_symbols_overview` → 文件级结构 | Inferred |
| **Basic** | `Glob("src/*")` → 顶层目录 + `Read` 关键 index 文件 | Inferred |

输出字段（每条）：
```yaml

  name: string
  responsibility: string
  community_id: string | null  # Full 模式独有（如 "tests"/"crg/1"）；Enhanced/Basic 填 null
```

### 4.5 Integration Extractor

| 工具层 | Bash 调用 | 置信度 |

### 4.6 Test Surface Extractor

目标：unit/integration/e2e 测试布局、模块与测试的对应关系、覆盖空白。

| 工具层 | Bash 调用 | 置信度 |
|---|---|---|
| **Full** | `spec-first crg query --pattern=tests_for --subject=<node_id> --repo=<target>` → 找带 `is_test=1` 标记且通过 `imports_from` 边指向 `subject` 的测试节点；`node_id` 为节点 ID 字符串（symbol_key 格式） | Inferred |
| **Full 批量** | 从 Stage B-Round2 `crg community --id=<X>` 返回的 `data.members[]` 中选取被 `entrypoints` 登记的节点，补充选 non-test 成员节点，总上限 **10 个**；对每个 `member.id`（字符串 symbol_key）调用 `crg query --pattern=tests_for --subject=<member.id>` | Inferred |
| **Enhanced** | `Glob("test*/**")` + `Glob("spec*/**")` + `mcp__serena__find_symbol` | Inferred |
| **Basic** | `Glob("test*/**")` + `Glob("spec*/**")` + `Glob("__tests__/**")` | Inferred |

> ⚠️ **API 说明**：`crg query --pattern=tests_for` 的参数名为 `--subject`（不是 `--symbol` 或 `--target`）。`--subject` 须为**节点 ID 字符串（symbol_key 格式，如 `src/foo.js#function#bar#L10`）**，不接受裸节点名称或文件路径。节点 ID 从 Stage B-Round2 `crg community --id=<X>` 返回的 `data.members[].id` 获取，无需额外 search 查询。

> ⚠️ **置信度说明**：`tests_for` 返回值由 CRG 内部标注为 `Inferred`（`inference_reason: "naming_convention"`）。实现原理：找带 `is_test=1` 标记且有 `imports_from` 边指向 subject 的节点——没有独立的 `TESTED_BY` 边类型。此方式只覆盖**直接 import** 了 subject 的测试文件，通过中间层或端到端方式覆盖 subject 的测试不会被检出（已知局限，非真实覆盖空白的充分证明）。

> **`test_kind` 分类规则**（路径启发式，`inference_reason: "path-naming-pattern"`）：CRG 不返回 test 类型，由 skill 基于 `file_path` 推断：
> - `file_path` 含 `e2e` / `end-to-end` / `playwright` / `cypress` → `e2e`
> - `file_path` 含 `integration` / `api-test` / `contract` → `integration`
> - 其余 `is_test=1` 节点 → `unit`（兜底）
>
> `test_kind` 的 `confidence` 始终为 `inferred`，路径命名约定是启发式，非构建元数据。

输出字段（每条）：
```yaml
- test_path: string
  test_symbol: string
  target_path: string      # 被测文件
  target_symbol: string    # 被测函数/类
  test_kind: unit | integration | e2e
  confidence: observed | inferred
  evidence: [...]
```

### 4.7 Risk Signal Extractor

目标：高扇出模块、高复杂度热点、shared/core/common/util 风险点、弱测试区域。

| 工具层 | Bash 调用 | 置信度 |
|---|---|---|
| **Full** | `spec-first crg impact --symbol=<flow.entry_node> --depth=2 --repo=<target>` → 反向 BFS 影响半径；`entry_node` 为节点 ID 字符串（symbol_key 格式），直接来自 `crg flows` 结果 | Observed |
| **Full** | `spec-first crg large-functions --min-lines=100 --repo=<target>` → 大函数/大文件 | Observed |
| **Full** | `spec-first crg god-nodes --repo=<target>` → in_degree 前 5% 的过度中心化节点（至多 10 个，排除 module 节点）；结果写入 `risk-signals.json.crg_metrics.top_hubs`；无需任何前置 ID，与 impact/large-functions 同一批次并行发出 | Inferred（`inference_reason: "call_graph_traversal"`）|
| **Full** | 从 Stage B-Round2 `crg community --id=<X>` 成员中筛选 `file_path` 含 `core/` / `shared/` / `common/` / `util/` 的节点，对每个节点的 `id`（字符串 symbol_key）调用 `crg query --pattern=dependents_of --module=<node_id> --repo=<target>` → 谁依赖了共享模块（调用上限 5 次）| Inferred（`inference_reason: "crg-dependents-evidence"`）|
| **Enhanced** | `mcp__serena__search_for_pattern(substring_pattern="TODO\|FIXME\|HACK")` → 技术债 | Inferred |
| **Basic** | `Grep("TODO\|FIXME\|HACK")` + `Glob("src/core/**")` + `Glob("src/shared/**")` | Inferred |

> ⚠️ **API 说明**：`crg impact` 的 `--symbol` 须为**节点 ID 字符串（symbol_key 格式）**，不接受裸符号名称。
> **取参规则**：`--symbol` 值从 `crg flows` 结果的 `data.items[]` 取 criticality 最高的 top-5 条目的 `entry_node` 字段（已是 symbol_key 字符串，无需额外 search）。`crg flows` 返回字段为 `{flow_id, entry_node, criticality, node_count}`，**无 `name` 字段**。
>
> **`crg god-nodes` vs `crg architecture` 的分工**：`crg architecture`（§4.4）内部也调用 `godNodes()` 返回 `hub_nodes`，但该结果服务于架构总览（Phase 3 `module-map.md`）。`crg god-nodes` 独立调用的目的是将 hub 节点数据写入 `risk-signals.json.crg_metrics.top_hubs`，供 risk 维度消费——两条调用链路消费方不同，不是重复计算。

输出字段（每条）：
```yaml
- path: string
  symbol: string
  kind: high_fanout | high_complexity | shared_core | weak_test | tech_debt
  summary: string
  severity: high | medium | low
  confidence: observed | inferred
  evidence: [...]
```

### 4.8 Data Shape Extractor

目标：DTO、schema、entity/model、migration、validation。

| 工具层 | Bash 调用 | 置信度 |
|---|---|---|
| **Full** | 6 个关键词**并行独立搜索**（同一 response 内批量发出）：<br>`spec-first crg search "schema" --repo=<target>`<br>`spec-first crg search "entity" --repo=<target>`<br>`spec-first crg search "model" --repo=<target>`<br>`spec-first crg search "dto" --repo=<target>`<br>`spec-first crg search "migration" --repo=<target>`<br>`spec-first crg search "validation" --repo=<target>`<br>→ 合并去重后得到候选节点列表 | Observed（节点存在由 AST 确认）|
| **Full 字段详情** | 对候选节点中 `kind=class\|interface\|struct` 的条目，直接 `Read(<node.file_path>)` 提取字段列表；替代原 `children_of` 模式 | Observed |
| **Enhanced** | `mcp__serena__find_symbol(name_path_pattern="Schema\|Model\|DTO\|Entity")` + `Glob("migration*/**")` | Inferred |
| **Basic** | `Glob("src/models/**")` + `Glob("src/schema/**")` + `Glob("migration*/**")` + `Grep("interface\\|type\\|class.*Schema")` | Inferred |

> ⚠️ **FTS5 搜索语义**：`crg search` 底层使用 SQLite FTS5，多词之间的空格被视为短语匹配（phrase），`"schema entity model"` 不会返回分别含这三个词的节点，而是要求它们连续出现。因此 Full 模式**必须**对每个关键词单独调用，不得合并为一次多词搜索。

> **Full 模式优势**：`crg search` 使用 AST 节点名搜索而非路径匹配，对任意目录结构均有效。Stage A 可安全并行执行，无需等待 `project_identity`。

> **字段详情替代方案**：原方案中使用 `children_of` 模式查询节点子成员，CLI 不支持此模式。改为直接 `Read` 对应源文件并让 Claude 从源码文本中提取字段清单，所得结果同样为 Observed 置信度。

**`data_shapes.kind` 分类规则**（从搜索词 + 节点属性推断，`inference_reason: "crg-semantic-search-evidence"`）：

| 命中搜索词 | 节点 `kind` | 补充条件 | 分类为 `kind` |
|---|---|---|---|
| `migration` | 任意 | 文件名含数字前缀（如 `20231201_`）或 `migrate`/`migration` 字样 | `migration` |
| `schema` | `class`\|`interface`\|`struct` | — | `schema` |
| `dto` | `class`\|`interface`\|`struct` | 名称含 `Dto`/`DTO`/`Request`/`Response` | `dto` |
| `entity` \| `model` | `class` | 名称含 `Entity`/`Model` | `entity` |
| `validation` | `class`\|`function` | 名称含 `Validator`/`Validation`/`Schema`+Zod/Joi | `validation` |
| 其余 | — | 无法明确分类 | `schema`（兜底，最通用） |

> 分类为 `inferred`：FTS5 命中节点的存在性是 Observed，但名称→类型映射是启发式。

### 4.9 Layer Detection

目标：识别项目中的 frontend/backend/mobile/desktop/cli/shared/data 层。

沿用 spec-bootstrap Phase 1.4 的检测规则，但增强置信度：

| 检测结果 | Full 模式置信度 | Enhanced/Basic 置信度 |
|---|---|---|
| `crg search` 返回前端框架节点（见下方定义）| Observed | — |
| package.json 中有 React/Vue 依赖 | Observed | Observed |
| 目录结构模式匹配（android/, ios/）| Inferred | Inferred |

**Full 模式 Layer 识别逻辑**：

```
对每个层，并行执行对应关键词搜索（同一 response 内批量发出）：
  Bash: spec-first crg search "<关键词>" --repo=<target>

⚠️ 关键词选择原则：搜索的是代码中实际出现的**符号名/函数名/类名**，而非框架描述性短语。
"React component"、"express router" 这类描述性短语在代码节点名中几乎不出现，应改用框架特征 API 词。

关键词 → 层（按框架特征 API，任一命中即判定该层存在）：
  frontend(React):   "useState" / "useEffect" / "JSX"
  frontend(Vue):     "defineComponent" / "createApp"
  frontend(Angular): "Injectable" / "Component" / "NgModule"
  backend(Express):  "router" / "middleware" / "app.listen"
  backend(FastAPI):  "APIRouter" / "Depends" / "HTTPException"
  backend(Django):   "urlpatterns" / "views" / "models.Model"
  mobile(Android):   "Activity" / "Fragment" / "ViewModel"
  mobile(iOS):       "UIViewController" / "SwiftUI" / "UIView"
  desktop(Electron): "BrowserWindow" / "ipcMain"
  desktop(Tauri):    "invoke" / "tauri"
  cli:               "commander" / "yargs" / "argparse" / "cobra"

判定规则（按优先级）：
  1. crg search 返回 ≥1 个节点
     → present=true, confidence=observed
     → evidence: [{file: <node.file_path>, symbol: <node.name>}]（取前 3 个）
     → 停止，不继续后续检测

  2. crg search 返回 0 个节点，但 package.json / go.mod 中有框架依赖
     → present=true, confidence=observed（元文件是确定性来源）
     → inference_reason: null

  3. 两者均无，但目录模式匹配（android/, ios/, web/, frontend/）
     → present=true, confidence=inferred
     → inference_reason: "directory-naming-pattern"

  4. 三者均无
     → present=false, confidence=unknown
     → inference_reason: "no-evidence"
```

输出字段：
```yaml
layers:
  frontend:
    present: bool
    confidence: observed | inferred | unknown
    inference_reason: string | null
    evidence: [{file: string, symbol: string}]
  backend: { 同上结构 }
  # mobile / desktop / cli / shared / data 同上结构
```

## 5. 置信度推导规则

### 5.1 总则

`confidence` 只允许三个值：`observed`、`inferred`、`unknown`。推导必须基于可判定规则，不允许自由推断。

### 5.2 Observed 的来源

事实被标记为 `observed` 当且仅当：

| 来源 | 条件 | 示例 |
|---|---|---|
| CRG AST（CLI）列表命令 | `crg flows`、`crg communities`、`crg community` 等直接从 AST 结构返回的节点/社区 | `crg flows` 返回的入口函数，`crg community` 返回的成员节点 |
| 配置文件 | 直接 Read package.json、go.mod 等元文件 | `project_identity.primary_language` |
| CRG 统计（CLI） | `crg large-functions`、`crg impact` 返回的结构化数据 | 超阈值函数的行数和位置 |

约束：
- `crg query` 的所有结果由 CLI 内部标注为 `Inferred`（见 §5.3），不属于 Observed 来源
- CRG 返回的节点一定有 `file_path` + `line_start` + `line_end`，这些作为 `evidence` 字段
- CRG 返回的边一定有 `source` + `target` + `kind`，这些作为 `evidence` 字段

### 5.3 Inferred 的来源

事实被标记为 `inferred` 当且仅当：

| 来源 | 条件 | 必须附加 |
|---|---|---|
| 目录命名模式 | 路径匹配预定义模式 | `inference_reason: "directory-naming-pattern"` |
| Glob 路径模式 | Glob 匹配特定目录/文件名模式 | `inference_reason: "glob-path-pattern"` |
| Serena 模式搜索 | `search_for_pattern` 命中 | `inference_reason: "serena-pattern-match"` |
| 路径近邻 | 测试文件路径与源码路径的近邻关系 | `inference_reason: "path-proximity"` |
| 框架标记推断 | 从 import 语句推断使用的框架 | `inference_reason: "framework-import-inference"` |
| CRG importers 证据 | `crg query --pattern=importers_of` 返回 AST 级 IMPORTS_FROM 边，但库名→语义分类仍需预置清单映射 | `inference_reason: "crg-importers-evidence"` |
| CRG 语义搜索证据 | `crg search` 返回 AST 节点，但 kind 分类由节点名称关键词推断 | `inference_reason: "crg-semantic-search-evidence"` |
| Grep import 推断 | 文本匹配 import 语句 | `inference_reason: "grep-import-pattern"` |

约束：
- 每条 `inferred` 事实**必须**带 `inference_reason`
- `inferred` 不得单独触发高严重性结论（`high` severity），除非有 ≥2 个独立信号共同支持
- `confidence` 字段只写 `observed` / `inferred` / `unknown`，不写扩展标注

### 5.4 Unknown 的来源

事实被标记为 `unknown` 当且仅当缺少足够直接证据，也缺少足够启发式支持。`unknown` 只允许进入降级路由，不得进入强结论分支。

### 5.5 模式与置信度的关系

```
Full 模式（CRG CLI）:
  entrypoints      → observed（crg flows/flow AST 级）
  modules          → observed（crg communities/community 社区检测级）
  testing_surface  → inferred（crg query tests_for，inference_reason: "naming_convention"；CLI 内部固定输出 Inferred）
  risk_signals     → observed（crg large-functions / crg impact BFS 统计级）
  integrations     → inferred（inference_reason: "crg-importers-evidence"，有 AST 边支持）
  data_shapes      → inferred（inference_reason: "crg-semantic-search-evidence"）

Enhanced 模式（Serena）:
  entrypoints      → inferred（模式搜索级）
  modules          → inferred（符号结构推断）
  testing_surface  → inferred（路径近邻推断）
  risk_signals     → inferred（模式匹配推断）[severity 上限: medium]
  integrations     → inferred（模式匹配推断）
  data_shapes      → inferred（模式匹配推断）

Basic 模式（Built-in）:
  全部字段         → inferred（文本/路径匹配推断）[severity 上限: medium]
```

**Enhanced / Basic 模式 risk_signals severity 约束：**

| 模式 | 条件 | severity 上限 |
|---|---|---|
| Enhanced | 单一 Serena 模式搜索命中 | `medium` |
| Enhanced | ≥2 个独立 Serena 信号同时命中 | `high` 可用 |
| Basic | 单一 Grep 模式命中 | `medium` |
| Basic | ≥2 个独立 Grep 信号同时命中 | `high` 可用 |

### 6.3 test-surface.json

路径：`.context/spec-first/bootstrap/<slug>/test-surface.json`

**`coverage_gaps.severity` 判定规则（Full 模式）**：

| 条件 | severity |
|---|---|
| `crg impact` 返回 `data.blast_radius ≥ 5` 且该符号是 `crg flows` 中 criticality=high 流的入口 | `high` |

## 7. 降级与容错

### 7.1 三级降级

```
Full (CRG CLI)
  ↓ crg.indexed = false（DB 不存在 / 为空 / 构建失败 / 超时）
Enhanced (Serena)
  ↓ Serena 不可用
Basic (Built-in)
```

降级不阻断主流程。降级后的事实置信度按 §5.5 规则降级。

### 7.2 单工具失败处理

| 场景 | 动作 |
|---|---|
| `crg flows` 返回非零退出码 | 降级到 Grep 获取入口，该部分事实标 `inferred` |
| `crg query --pattern=tests_for` 返回空 | 该查询维度标为 `unknown`，不影响其他维度 |
| `mcp__serena__find_symbol` 失败 | 降级到 Grep，该部分事实标 `inferred` |
| 任何工具调用异常 | 写入 `generation_errors[]`，继续其他维度 |
| `crg build` 超时（>10 分钟）| 自动降级到 Enhanced；向用户报告"构建超时，已降级"；写入 `generation_errors[{extractor:"crg_build", error:"timeout", fallback_applied:true}]` |
| `crg build` 返回错误 | 显示具体错误；降级到 Enhanced；写入 `generation_errors[]` |
| 构建过程中用户中断 | 等效"用户选择否"，直接降级到 Enhanced |
| `degraded: true` in CLI envelope | 触发降级链，不继续尝试 Full 模式 |

### 7.3 部分产物缺失策略

```yaml
missing_file_strategy:
  code-facts/public-entrypoints.md: "跳过，使用 00-summary.md 替代"
  context-packs/review-change.md: "跳过，使用 unknown 回退集合"
  code-facts/test-map.md: "跳过，不影响 review 路由"
```

任何缺失都必须在 `README.md` 的 Freshness 章节和 `injection-index.yaml` 的 `caveats` 中可见。

**`context-packs/review-change.md` 生成方式（静态事实组装）**：

`review-change.md` 是面向 review 场景的静态上下文包，必须基于已生成的事实层组装，不得依赖特定 git diff：

```
组装来源（按优先级）：
  1. risk-signals.json → "高风险区域" 章节（high severity 优先）
  2. test-surface.json → "测试覆盖空白" 章节（coverage_gaps）
  3. fact-inventory.json.entrypoints → "关键入口点" 章节（kind=http/worker 优先）
  4. fact-inventory.json.integrations → "外部依赖边界" 章节（high-risk 类型）

格式：
  ## 高风险区域
  ## 关键入口点
  ## 外部依赖边界
  ## 测试覆盖空白
  ## 使用说明（本文为静态快照，生成于 <timestamp>）
```

> **`crg review-context` 的定位**：`spec-first crg review-context --since=<git-ref>` 是面向"当次变更"的按需分析工具（需要指定 git ref，不适合生成不依赖特定 diff 的静态上下文包）。该命令在后续 review workflow（`spec-review`）中按需调用，不在本阶段批量执行。

数据流约束（Facts First 执行规则）：
- Worker **必须优先**读取控制面事实文件（`fact-inventory.json` 等）
- Worker **默认不得**回扫源码补洞
- 如确需回读，PRD 中必须设置 `source_read_allowed = true` 并记录 `fallback_reason`

## 10. 验收标准

### 10.1 工具链验收

| 检查项 | 标准 |
|---|---|
| CRG 探测 | 能正确区分 indexed / not-indexed 两种状态（通过 DB 文件检查 + `crg stats`）|
| 降级链路 | `crg.indexed=false` 时能自动降级到 Serena/Basic，不阻断 |
| CLI envelope | `degraded: true` 时能正确触发降级，不继续尝试 Full 模式 |

| 置信度合规 | 所有 `inferred` 事实都有 `inference_reason` |

### 10.2 模式对比验收

至少在 1 个真实项目上分别用 Full 和 Basic 模式运行，对比：
- Full 模式事实层的 `observed` 比例应显著高于 Basic
- Basic 模式不得出现无 `inference_reason` 的 `inferred` 事实
- 两种模式的最终产物目录结构一致

### 10.3 增量基础验收

- `fingerprints.json` 正确记录 `crg stats` 返回的 `last_built`（`graph_last_built` 字段）、`node_count`、`edge_count` 和关键输入文件 SHA
- `fingerprints.json` 的 `outputs` 部分正确记录产物依赖关系
- v1 stale 检测：当 `fingerprints.json` 存在且 CRG 图有更新时，Phase 0 必须在探测输出中打印 ⚠️ stale 警告
- v1 stale 检测：当关键文件 SHA 变更时，Phase 0 必须指出具体变更文件和受影响产物
- v1 不要求自动重建，stale 警告不阻断主流程
- `fingerprints.json` 的两段写入（`in_progress` → `complete`）能正确反映运行中断状态

### 10.4 降级可见性验收

- 降级事件在执行摘要中向用户报告
- 降级在 `fact-inventory.json` 的 `analyzer_mode` 中记录
- 降级在 `README.md` Freshness 章节中体现

## 11. 风险与约束

### 11.1 已知风险

| 风险 | 影响 | 缓解 |
|---|---|---|
| CRG 图构建耗时（普通项目 2-5 分钟，大型 monorepo 可达 10 分钟）| 用户等待时间过长 | Phase 0 给出含时间范围的明确提示，允许跳过；超过 10 分钟自动降级 |
| CRG 对某些语言支持不完整 | 事实层部分字段为空 | 降级到 Serena/Built-in 补充 |
| `better-sqlite3` native 模块安装失败 | CLI `degraded: true`，无法进入 Full 模式 | Phase 0 检测 envelope.degraded 字段，自动降级；向用户输出安装指引 |

### 11.2 实施约束

- SKILL.md 中的 crg 调用始终显式传入 `--repo=<target>`，不依赖工作目录默认值
- `<target>` 在 Phase 0 确定后作为 `$TARGET_REPO` 变量传递，后续所有 Bash 调用复用此变量
- CRG CLI 是内嵌实现，不需要也不应通过 MCP server 调用；SKILL.md 中不得出现 `mcp__code-review-graph__*` 前缀
- Serena MCP 调用保持不变（仍为 `mcp__serena__*`）

### 11.3 Monorepo 边界约束（v1）

**v1 的 monorepo 支持范围：**

| 情况 | v1 行为 |
|---|---|
| `repo_shape = monorepo` | `spec-first crg build --repo=<整个 monorepo 根>`，CRG 扫描全局 |
| `crg communities` | 返回跨 package 的全局社区，`modules[]` 中条目可能跨越多个子包 |
| `crg impact` | 可跨 package 计算 fan-out，v1 不限制 |
| 子包级单独分析 | v1 不支持——`--repo` 只能是整个 monorepo，不能是单个子包 |
| 子包级产物隔离 | v1 不支持——`fact-inventory.json` 描述整个 repo，不区分子包 |

**约束说明**：monorepo 场景下，CRG 的社区检测可能将不同子包的代码识别为同一社区。这是预期行为——v1 以"整体项目理解"为目标，子包级精度留给后续阶段。实现时应在 `project_identity.repo_shape = "monorepo"` 标注中提示用户这一限制。

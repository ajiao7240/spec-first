# `spec-graph-bootstrap` 阶段 2：集成方案

> 本文是阶段 2 的具体集成方案，定义 Fact Inventory 提取的工具选型、调用链路、数据映射和降级策略。
>
> 本文中"阶段 2"指实施推进阶段，不等同于 skill 执行过程中的 `Phase 0–4`。
>
> 如与 [修订终版.md](./修订终版.md) 或 [阶段2-事实抽取与文档生成需求.md](./阶段2-事实抽取与文档生成需求.md) 冲突，以总版为准。

---

## 1. 架构决策总览

### 1.1 核心原则

`spec-graph-bootstrap` 的 Fact Inventory 提取遵循三层工具栈：

```
Tier 1: code-review-graph MCP  ← primary，AST-grade Observed 事实
Tier 2: Serena MCP             ← secondary，semantic-level 补充
Tier 3: Built-in (Grep/Glob)   ← fallback，heuristic-level Inferred 事实
```

选型依据：

| 判断维度 | code-review-graph | Serena | Built-in |
|---|---|---|---|
| 事实来源 | tree-sitter AST（确定性） | 语义索引（推断性） | 文本匹配（启发式） |
| 入口点识别 | `list_flows` 直接返回 | `find_symbol` 手动 | Grep 启发式 |
| 调用关系 | `CALLS` / `IMPORTS_FROM` 边 | `find_referencing_symbols` | 不支持 |
| 测试映射 | `TESTED_BY` 边，`tests_for` 查询 | 路径启发式 | 路径启发式 |
| 风险信号 | fan-out 可查、impact radius 可算 | 需手动 | 不支持 |
| 增量更新 | `incremental.py` 内建 | 无 | 无 |
| 持久化 | SQLite（跨会话复用） | 无 | 无 |
| 增量对齐 | 文件 SHA 变化检测 | N/A | N/A |

### 1.2 为什么选 code-review-graph 为主工具

**决策：code-review-graph（CRG）作为 Tier 1，Serena 作为 Tier 2，Built-in 作为 Tier 3。**

理由：

1. **Observed 事实最可信**：tree-sitter 解析出的是确定性结构——`CALLS` 边存在就是存在。这比语义推断和文本匹配更符合 "Facts First" 原则。
2. **数据模型直接对应**：CRG 的 `NodeInfo + EdgeInfo` 与 Fact Inventory schema 几乎 1:1 映射，不需要额外转换层。
3. **增量刷新天然对齐**：CRG 内建文件 SHA 检测和图增量更新，`fingerprints.json` 可直接复用其结果。
4. **业界最佳实践对齐**：GitHub Copilot Workspace、Sourcegraph Cody 均采用 AST graph → 事实层 → 上下文注入架构。
5. **已在用户工具链中**：CRG 是已部署的 MCP server，无需额外引入新依赖。

### 1.3 与现有 spec-bootstrap 的兼容性

本方案不改 spec-bootstrap 的任何逻辑或产物。分析模式扩展为：

```
spec-bootstrap:     Enhanced(Serena) / Basic
spec-graph-bootstrap: Full(CRG+Serena) / Enhanced(Serena) / Basic
```

新增 `Full` 模式仅在 CRG 可用时启用，不改变 `Enhanced` 和 `Basic` 的已有语义。

## 2. 工具能力清单

### 2.1 code-review-graph MCP 工具

CRG 以 FastMCP 实现，注册名 `code-review-graph`。MCP 工具调用前缀为 `mcp__code-review-graph__`。

阶段 2 会使用以下工具：

| 工具函数 | 用途 | 阶段 2 使用场景 |
|---|---|---|
| `build_or_update_graph_tool` | 构建/增量更新图 | Phase 0：确保目标项目已索引 |
| `list_graph_stats_tool` | 图统计（节点数、边数） | Phase 0：探测图是否存在 |
| `query_graph_tool` | 图查询（callers_of, tests_for 等） | Phase 1：提取各类事实 |
| `get_minimal_context_tool` | 超紧凑上下文（~100 token） | Phase 1：快速获取项目概览 |
| `list_flows_tool` | 列出执行流（按 criticality 排序） | Phase 1：入口点识别 |
| `get_flow_tool` | 单条执行流详情 | Phase 1：入口点深度分析 |
| `list_communities_tool` | 列出代码社区 | Phase 1：模块结构识别 |
| `get_community_tool` | 单个社区详情 | Phase 1：模块深度分析 |
| `get_architecture_overview_tool` | 架构总览 | Phase 1：项目级结构摘要 |
| `find_large_functions_tool` | 大函数/大类/大文件 | Phase 1：风险信号 |
| `get_impact_radius_tool` | 变更影响半径 | Phase 1：风险信号（fan-out） |
| `detect_changes_tool` | 变更检测 + 风险评分 | Phase 4：增量刷新 |
| `semantic_search_nodes_tool` | 语义搜索节点 | Phase 1：特定模式查找 |

`query_graph_tool` 支持以下查询模式（`pattern` 参数）：

| pattern | 含义 | 对应 Fact Inventory 字段 |
|---|---|---|
| `callers_of` | 谁调用了 X | risk_signals.fan_in |
| `callees_of` | X 调用了谁 | modules 核心依赖方向 |
| `imports_of` | X 导入了什么 | integrations |
| `importers_of` | 谁导入了 X | risk_signals.fan_out |
| `children_of` | X 包含哪些节点 | modules 模块内部结构 |
| `tests_for` | X 的测试有哪些 | testing_surface |
| `inheritors_of` | 谁继承了 X | modules 继承关系 |
| `file_summary` | 文件内所有节点 | 各 extractor 的文件级证据 |

## 3. Phase 0：探测扩展

在继承 spec-bootstrap 的 Host Readiness Gate 基础上，新增三个探测字段和一个 CRG 索引检查。

### 3.1 新增探测步骤

```
原 spec-bootstrap 探测顺序:
  Step 0: 宿主选择
  Step 1: mcp-setup marker
  Step 2: MCP runtime probe (context7)

spec-graph-bootstrap 新增:
  Step 3: CRG MCP probe
  Step 4: 分析模式判定
```

### 3.2 CRG 探测流程

```
Step 3: CRG MCP probe

3a. 调用 list_graph_stats_tool(repo_root=<target-project>)，按以下优先级匹配第一个成立的情况：

    [1] 工具未注册或调用抛出异常（MCP 层错误）
        → crg.available = false
        → 后续跳过所有 CRG 调用

    [2] 工具调用成功，但响应缺少 graph_mtime 字段（图数据库未初始化）
        → crg.available = true, crg.indexed = false
        → 不依赖具体错误字符串，只检查字段是否存在

    [3] 工具调用成功，graph_mtime 存在，但 node_count=0 且 edge_count=0
        → crg.available = true, crg.indexed = false
        → 图文件存在但内容为空，视为构建未完成
        （注：此情况极少，真正空项目可安全降级）

    [4] 工具调用成功，graph_mtime 存在，node_count≥0（含 0）且响应完整
        → crg.available = true, crg.indexed = true
        → 图已就绪，node_count=0 的空项目属于已索引状态，不触发构建提示

3b. 如果 crg.available = true 且 crg.indexed = false:
    → 提示用户:
      "检测到 code-review-graph MCP 可用，但当前项目尚未构建图索引。
       是否现在构建？（耗时视项目大小：普通项目 2-5 分钟，大型 monorepo 或 100k+ 行代码项目可达 10 分钟）
       选择「是」将执行 build_or_update_graph_tool 后继续。
       选择「否」将降级到 Enhanced/Basic 模式。"

    → 用户选择「是」:
      调用 build_or_update_graph_tool(repo_root=<target>, postprocess="full")
      等待完成 → crg.indexed = true

    → 用户选择「否」:
      → crg.indexed = false，降级到 Enhanced 或 Basic
```

### 3.3 分析模式判定

```
if crg.available AND crg.indexed:
    mode = Full
    graph_support_state = external-available
    code_facts_confidence = high

elif serena.ready:
    mode = Enhanced
    graph_support_state = local-only
    code_facts_confidence = medium

else:
    mode = Basic
    graph_support_state = unavailable
    code_facts_confidence = low
```

### 3.4 探测输出格式

```
🔍 检测项目工具就绪状态...

CRG:      available=yes, indexed=yes, nodes=234, edges=891
Serena:   ready=yes, project=/path/to/target

📊 分析模式: Full | Code Facts: high | DB: inferred | Graph Support: external-available

（如 fingerprints.json 存在且检测到 stale，追加以下输出）
⚠️ 检测到 2 个 stale 信号：
  - CRG 图已更新（上次: 2026-04-01T10:00:00Z，当前: 2026-04-09T08:30:00Z）
    受影响产物: code-facts/public-entrypoints.md, code-facts/test-map.md, code-facts/high-risk-modules.md
  - 文件 package.json SHA 已变更
    受影响产物: code-facts/public-entrypoints.md
建议重新运行 spec-graph-bootstrap 以刷新产物。继续使用现有产物请忽略此警告。
```

### 3.5 State-to-Action 决策表（补充修订终版 §11.4）

| Signal 组合 | CRG 可用？ | Action | Effect on Output |
|---|---|---|---|
| `crg.available=true AND crg.indexed=true` | 是 | `continue` | Full 模式；CRG 作为事实层主数据源；所有 code-facts 产物为 AST-grade Observed |
| `crg.available=true AND crg.indexed=false AND user_declined_build=true` | 部分（可探测，不可查询）| `degrade` | 降级到 Enhanced；CRG 不可用于事实提取；用户已被告知 |
| `crg.available=false AND serena.ready=true` | 否 | `continue` | Enhanced 模式；Serena 作为主数据源；code-facts 为语义级 Inferred |
| `crg.available=false AND serena.ready=false` | 否 | `continue`（最低限度）| Basic 模式；仅 Built-in 工具；所有 code-facts 必须标记为 Inferred |

## 4. Phase 1：事实提取流程

### 4.1 提取器总览

> **"并行"的执行语义**：本文件中"Stage 内并行"指在同一 Claude response 中批量发出该 Stage 的全部 MCP tool call（Claude 的 parallel tool use 机制）。Stage 之间有数据依赖，必须顺序执行（等待上一 Stage 所有 tool call 返回并完成 JSON 写入后，再发出下一 Stage 的 tool call）。

```
Phase 1 执行顺序（三阶段，Stage 间顺序执行，Stage 内批量 tool call）：

  ── Stage A（无依赖，同一 response 内批量发出）─────────────────────
  4.2 Project Identity Extractor  ──→ fact-inventory.project_identity
  4.9 Layer Detection             ──→ fact-inventory.layers
  4.10 Database Detection         ──→ fact-inventory.database
  4.8 Data Shape Extractor        ──→ fact-inventory.data_shapes
       (Full: semantic_search_nodes + children_of；Enhanced/Basic: Glob+Grep)
  ──────────────────────────────────────────────────────────────────

  ── Stage B（依赖 Stage A 结果，同一 response 内批量发出）────────────
  4.3 Entrypoint Extractor        ──→ fact-inventory.entrypoints
       (依赖: project_identity.primary_language 确定框架入口模式)
  4.4 Module Structure Extractor  ──→ fact-inventory.modules
       (依赖: project_identity.repo_shape 确定社区检测范围)
  4.5 Integration Extractor       ──→ fact-inventory.integrations
       (依赖: project_identity.primary_language 确定已知库清单范围)
  ──────────────────────────────────────────────────────────────────

  ── Stage C（依赖 Stage B 结果，同一 response 内批量发出）────────────
  4.6 Test Surface Extractor      ──→ fact-inventory.testing_surface / test-surface.json
       (依赖: modules 确定核心模块列表，驱动 tests_for 批量查询)
  4.7 Risk Signal Extractor       ──→ risk-signals.json
       (依赖: entrypoints.criticality 判定高风险入口；modules 确定 shared_core 候选)
  ──────────────────────────────────────────────────────────────────

  4.11 写入控制面（所有 Stage 完成后，见 §4.11）
```

### 4.2 Project Identity Extractor

目标：识别项目名、主语言、主框架、仓库形态。

| 工具层 | 工具调用 | 置信度 |
|---|---|---|
| **Full** | `get_minimal_context_tool(repo_root)` + `list_graph_stats_tool()` 获取语言分布 | Observed |
| **Enhanced** | `serena get_current_config` 获取语言 + Glob 框架标记文件 | Observed |
| **Basic** | `Glob("package.json")` + `Glob("go.mod")` + `Glob("requirements.txt")` + `Read` | Observed |

说明：项目身份无论哪个模式都能达到 Observed，因为元文件（package.json 等）是确定性的。

输出字段：
```yaml
project_identity:
  name: string          # 从 package.json.name 或目录名
  primary_language: string
  primary_frameworks: [string]
  repo_shape: monorepo | service | library | cli
```

### 4.3 Entrypoint Extractor

目标：识别 HTTP 入口、CLI 入口、worker/cron 入口、SDK/public export 入口。

| 工具层 | 工具调用 | 置信度 |
|---|---|---|
| **Full** | `list_flows_tool(repo_root)` → 按 criticality 排序的执行流 | Observed |
| **Full 补充** | `get_flow_tool(flow_id=<id>)` → 单流详情（入口文件、调用链深度、覆盖文件数） | Observed |
| **Full 细化** | `query_graph_tool(pattern="file_summary", target=<flow.entry_file>)` → 文件内所有符号（`target` 取 `list_flows_tool` 返回的流的入口文件路径，非硬编码）| Observed |
| **Enhanced** | `serena search_for_pattern(substring_pattern="@app.get|@router.post|main(")` | Inferred |
| **Basic** | `Grep("app\\.get\\|app\\.post\\|router\\.\\|main\\s*\\(")` + `Glob("src/main.*")` | Inferred |

Full 模式优势：CRG 的 `detect_entry_points()` 内建了框架装饰器模式识别（`@app.get`、`@router.post`、`@RequestMapping` 等），直接给出入口函数的限定名、位置、调用深度。这是 AST 级别的 Observed 事实，不需要额外启发式。

输出字段（每条）：
```yaml
- path: string
  symbol: string          # 限定名，如 src/server.ts::app.listen
  kind: http | cli | worker | sdk
  summary: string
  confidence: observed | inferred
  inference_reason: string | null   # observed 时填 null；inferred 时必填
  evidence:
    - file: string
      symbol: string
  updated_at: ISO datetime
```

### 4.4 Module Structure Extractor

目标：顶层目录职责、模块边界、模块间关系。

| 工具层 | 工具调用 | 置信度 |
|---|---|---|
| **Full** | `list_communities_tool(repo_root)` → 代码社区（CRG 内建 Leiden/Louvain） | Observed |
| **Full 补充** | `get_community_tool(community_id=<id>)` → 社区内节点列表 | Observed |
| **Full 补充** | `query_graph_tool(pattern="imports_of", target=<社区代表路径>)` → 模块间依赖方向；`target` 取 `list_communities_tool` 返回的最大社区的代表文件路径（`representative_file` 字段），若无此字段则取社区内第一个节点的 `file_path`；不硬编码 `src/core` | Observed |
| **Full 概览** | `get_architecture_overview_tool()` → 基于社区结构的架构总览 | Observed |
| **Enhanced** | `serena get_symbols_overview` → 文件级结构 | Inferred |
| **Basic** | `Glob("src/*")` → 顶层目录 + `Read` 关键 index 文件 | Inferred |

Full 模式优势：CRG 的社区检测自动识别代码内聚模块，不需要依赖目录命名约定。比手动目录分析更准确——即使代码物理目录与逻辑模块不一致，社区检测仍能正确分组。

输出字段（每条）：
```yaml
- path: string           # 目录或文件路径
  name: string           # 模块名
  responsibility: string # 一句话职责
  community_id: int | null  # CRG 社区 ID（Full 模式独有；Enhanced/Basic 模式填 null）
  confidence: observed | inferred
  inference_reason: string | null   # observed 时填 null；inferred 时必填（如 "directory-naming-pattern"）
  evidence: [...]
```

### 4.5 Integration Extractor

目标：DB client、cache、MQ、外部 HTTP/gRPC/SDK、auth/session 边界。

| 工具层 | 工具调用 | 置信度 |
|---|---|---|
| **Full** | `query_graph_tool(pattern="imports_of", target=<主配置文件路径>)` → 获取配置文件的直接依赖边；主配置文件按语言确定：Node→`package.json`，Python→`requirements.txt`/`pyproject.toml`，Go→`go.mod`，Java→`pom.xml`/`build.gradle`。此调用给出"谁导入了配置文件"，为 Observed 事实，但最终 integration 条目因 kind 分类仍标 inferred（见说明）| Inferred（`inference_reason: "crg-importers-evidence"`）|
| **Full 补充** | 对语言清单中每个库名逐一调用：`query_graph_tool(pattern="importers_of", target="knex")`、`target="prisma"` …（见下方库名清单）→ 谁使用了这些库 | Inferred（`inference_reason: "crg-importers-evidence"`）|
| **Enhanced** | `serena search_for_pattern(substring_pattern="createConnection|new Redis|new Kafka")` | Inferred |
| **Basic** | `Grep("import.*from.*['\"](?:knex|prisma|redis|ioredis|amqplib)")` | Inferred |

**已知库名清单（按语言选择，Stage B 已知 `primary_language`）**

实现规则：Stage B 执行时已从 Stage A 获得 `project_identity.primary_language`，按语言选取对应清单，对清单中每个库名逐一调用 `importers_of`，过滤非空结果。

| 语义分类 | Node.js / JavaScript / TypeScript | Python | Go | Java / Kotlin |
|---|---|---|---|---|
| database | `knex`, `prisma`, `typeorm`, `sequelize`, `mongoose`, `pg`, `mysql2`, `sqlite3` | `sqlalchemy`, `psycopg2`, `django.db`, `tortoise-orm`, `pymongo`, `aiomysql` | `gorm`, `pgx`, `database/sql`, `go-sqlite3`, `mongo-driver` | `hibernate`, `spring-data`, `mybatis`, `jpa` |
| cache | `redis`, `ioredis`, `memcached`, `node-cache` | `redis`, `aioredis`, `django-redis`, `pylibmc` | `go-redis`, `rueidis`, `bigcache` | `jedis`, `lettuce`, `redisson` |
| message_queue | `amqplib`, `kafkajs`, `bullmq`, `bull` | `celery`, `pika`, `kafka-python`, `aio-pika` | `sarama`, `nats.go`, `rabbitmq` | `kafka-clients`, `spring-amqp`, `activemq` |
| http_client | `axios`, `node-fetch`, `got`, `superagent` | `requests`, `httpx`, `aiohttp`, `urllib3` | `net/http`, `resty`, `fasthttp` | `okhttp`, `httpclient`, `webclient` |
| auth | `passport`, `jsonwebtoken`, `bcrypt`, `oauth2` | `authlib`, `python-jose`, `passlib`, `django-allauth` | `golang-jwt`, `casbin`, `go-oauth2` | `spring-security`, `nimbus-jose-jwt` |

> 对于未在清单内的语言（Rust、Ruby、Swift 等），跳过 `importers_of` 批量调用，仅执行 Enhanced/Basic 降级路径。清单可随版本迭代扩展。

**API 约束说明**：`query_graph_tool` 的 `target` 参数为单一字符串，不支持正则或多值。多目标检测必须拆分为多次独立调用。实现时：
1. 从 `project_identity.primary_language` 确定使用哪一列清单
2. 优先读取项目依赖清单（`package.json` / `requirements.txt` / `go.mod` / `pom.xml`）缩小候选范围
3. 对候选库名逐一调用 `importers_of`，收集非空结果

说明：Integration 的识别跨度较大。CRG 能准确给出 `IMPORTS_FROM` 边（谁导入了某库），但将库名映射到语义分类（"knex 是数据库客户端"）仍依赖预置库名清单。因此 Integration 的事实标记为 `inferred`，`inference_reason` 填 `"crg-importers-evidence"`（区别于 Grep 的 `"grep-import-pattern"`），表示推断有 AST 级证据支撑。

输出字段（每条）：
```yaml
- path: string
  symbol: string
  kind: database | cache | message_queue | http_client | auth | other
  summary: string
  confidence: inferred                        # Integration 固定为 inferred，见§5.3
  inference_reason: "crg-importers-evidence"  # Full 模式；Basic 模式为 "grep-import-pattern"
  evidence: [...]                             # Full 模式：CRG IMPORTS_FROM 边；Basic 模式：Grep 匹配行
```

### 4.6 Test Surface Extractor

目标：unit/integration/e2e 测试布局、fixture/mock/setup、模块与测试的对应关系。

| 工具层 | 工具调用 | 置信度 |
|---|---|---|
| **Full** | `query_graph_tool(pattern="tests_for", target="<module-or-function>")` → TESTED_BY 边 | Observed |
| **Full 批量** | 从 Stage B `modules[]` 中取 `confidence=observed` 的条目，优先选与 `entrypoints` 有 `CALLS` 关系的模块，上限取 **10 个**；对每个模块的 `path` 调用 `query_graph_tool(pattern="tests_for", target=<path>)`；若返回空则用 `find_large_functions_tool` 定位该模块内的关键函数后再查测试 | Observed |
| **Full 覆盖** | 先 `Glob("src/core/**/*")` 获取文件列表，再 `get_impact_radius_tool(changed_files=[<file_list>], max_depth=2)` → 查看影响到的测试节点（`changed_files` 接受文件路径列表，不接受目录；Glob 结果上限取前 20 个文件，避免超出 token 预算）| Observed |
| **Enhanced** | `Glob("test*/**")` + `Glob("spec*/**")` + `serena find_symbol` 按测试文件模式查找 | Inferred |
| **Basic** | `Glob("test*/**")` + `Glob("spec*/**")` + `Glob("__tests__/**")` | Inferred |

Full 模式优势：CRG 内建 `TESTED_BY` 反向边和 `is_test` 标记。`tests_for` 查询直接给出"函数 X 被测试 A、测试 B 覆盖"的精确映射。这是 Fact Inventory 中最关键的高价值数据——没有 CRG 时只能做路径近邻启发式。

输出字段（每条）：
```yaml
- test_path: string
  test_symbol: string
  target_path: string      # 被测文件
  target_symbol: string    # 被测函数/类
  test_kind: unit | integration | e2e
  confidence: observed | inferred
  evidence: [...]
```

### 4.7 Risk Signal Extractor

目标：高扇出模块、高复杂度热点、shared/core/common/util 风险点、弱测试区域。

| 工具层 | 工具调用 | 置信度 |
|---|---|---|
| **Full** | `get_impact_radius_tool(changed_files=[...核心文件], max_depth=2)` → fan-out 分析（"核心文件" = `list_flows_tool` 返回的 criticality=high 流的入口文件，或 `list_communities_tool` 返回的最大社区的代表文件，取两者并集，上限 10 个）| Observed |
| **Full** | `find_large_functions_tool(threshold=100)` → 大函数/大文件 | Observed |
| **Full** | 从 Stage B `modules[]` 中筛选路径含 `core/` / `shared/` / `common/` / `util/` 的条目，对每个条目的 `path` 调用 `query_graph_tool(pattern="importers_of", target=<path>)` → 谁依赖了共享模块（调用上限 5 次，避免 N 过大；若 `modules[]` 无符合条目则跳过此调用）| Observed |
| **Full 覆盖** | `list_flows_tool()` → 按 criticality 排序，高 criticality 流经的节点为高风险 | Observed |
| **Enhanced** | `serena search_for_pattern(substring_pattern="TODO|FIXME|HACK")` → 技术债 | Inferred |
| **Basic** | `Grep("TODO|FIXME|HACK")` + `Glob("src/core/**")` + `Glob("src/shared/**")` | Inferred |

Full 模式优势：CRG 的 impact radius 直接给出扇出度（total_impacted 数量），`find_large_functions_tool` 直接给出超阈值函数列表。这些都是结构化数据，不需要人工判断。

输出字段（每条）：
```yaml
- path: string
  symbol: string
  kind: high_fanout | high_complexity | shared_core | weak_test | tech_debt
  summary: string
  severity: high | medium | low
  confidence: observed | inferred
  evidence: [...]
```

### 4.8 Data Shape Extractor

目标：DTO、schema、entity/model、migration、validation。

| 工具层 | 工具调用 | 置信度 |
|---|---|---|
| **Full** | `semantic_search_nodes_tool(query="data model entity schema DTO migration validation", repo_root=<target>)` → 返回类型/接口/类节点，不依赖路径命名 | Observed |
| **Full 补充** | 对返回节点中 `node_type=class\|interface\|struct` 的条目，调用 `query_graph_tool(pattern="children_of", target=<node.qualified_name>)` 获取字段列表 | Observed |
| **Enhanced** | `serena find_symbol(name_path_pattern="Schema\|Model\|DTO\|Entity\|Struct")` + `Glob("migration*/**")` | Inferred |
| **Basic** | `Glob("src/models/**")` + `Glob("src/schema/**")` + `Glob("migration*/**")` + `Grep("interface\\|type\\|class.*Schema")` | Inferred |

Full 模式优势：使用语义搜索而非路径匹配，对任意目录结构（`src/models/`、`app/entities/`、`domain/`、`pkg/`）均有效，真正做到语言无关。Stage A 可安全并行执行，无需等待 `project_identity`。

### 4.9 Layer Detection

目标：识别项目中的 frontend/backend/mobile/desktop/cli/shared/data 层。

沿用 spec-bootstrap Phase 1.4 的检测规则，但增强置信度：

| 检测结果 | Full 模式置信度 | Enhanced/Basic 置信度 |
|---|---|---|
| CRG 社区中识别到前端框架节点（见下方定义）| Observed | — |
| package.json 中有 React/Vue 依赖 | Observed | Observed |
| 目录结构模式匹配（android/, ios/）| Inferred | Inferred |

**Full 模式 CRG 识别前端框架节点的具体逻辑：**

```
对每个已知前端框架关键词，调用：
  semantic_search_nodes_tool(query="<关键词>", repo_root=<target>)

关键词 → 层（language-agnostic）：
  "React component" / "Vue component" / "Angular component"  → frontend
  "express router" / "fastapi route" / "django view"         → backend
  "Android Activity" / "iOS UIViewController"                → mobile
  "Electron main process" / "Tauri window"                   → desktop
  "commander" / "yargs" / "argparse"                        → cli

判定规则（按优先级顺序）：
  1. semantic_search_nodes 返回 ≥1 个节点
     → present=true, confidence=observed
     → evidence: [{file: <node.file_path>, symbol: <node.name>}]（取前 3 个结果）
     → 停止，不继续后续检测

  2. semantic_search_nodes 返回 0 个节点，但 package.json / go.mod / requirements.txt 中有框架依赖
     → present=true, confidence=observed（元文件是确定性来源）
     → inference_reason: null（observed 无需 inference_reason）
     → evidence: [{file: "package.json", symbol: "dependencies.<framework>"}]

  3. 两者均无，但目录模式匹配（`android/`, `ios/`, `web/`, `frontend/`）
     → present=true, confidence=inferred
     → inference_reason: "directory-naming-pattern"

  4. 三者均无结果
     → present=false, confidence=unknown
     → inference_reason: "no-evidence"
     → 不得推断为 present=true，不进入强结论分支
```

输出字段：
```yaml
layers:
  frontend:
    present: bool
    confidence: observed | inferred | unknown
    inference_reason: string | null   # observed 时填 null；inferred/unknown 时必填
    evidence: [{file: string, symbol: string}]
  backend:
    present: bool
    confidence: observed | inferred | unknown
    inference_reason: string | null
    evidence: [{file: string, symbol: string}]
  # mobile / desktop / cli / shared / data 同上结构
```

沿用 spec-bootstrap Phase 1.5 的数据库检测流程，不变。CRG 不直接参与数据库 schema 检测（CRG 专注代码结构，不查数据库）。

### 4.11 写入控制面

所有 Stage 完成后，将各 Extractor 的输出合并写入控制面产物文件。

**写入原子性规则**：

- `fact-inventory.json`：全量写入（所有 Stage 结果合并后一次性写入，不分条目增量写入）；任何 Stage 失败时，已完成的部分**仍然写入**，但对应字段填充已有结果，缺失字段填 `null` 并在顶层增加 `generation_errors` 数组记录失败详情。
- `risk-signals.json`、`test-surface.json`：同上规则，独立文件分别原子写入。
- `fingerprints.json`：最后写入，在三个主文件写入完成后执行。`fingerprints.json` 的 `outputs` 中只记录**实际已成功写入**的文件；如某文件因错误未能生成，在 `generation_errors` 中记录，不写入 `outputs`。

**写入后校验**：

```
1. 读取写入后的 fact-inventory.json，校验 JSON 合法性
2. 校验 schema_version 字段存在
3. 校验 analyzer_mode 值为 full | enhanced | basic 之一
4. 如校验失败 → 向用户报告"控制面写入异常：<具体错误>"，不继续后续 Phase
```

**`generation_errors` 格式**（追加到各 JSON 顶层，仅在有错误时存在）：

```yaml
generation_errors:
  - extractor: string        # 如 "entrypoint_extractor"
    stage: A | B | C
    error: string            # 简短错误描述
    affected_fields: [string] # 受影响的字段名
    fallback_applied: bool   # 是否已应用降级填充
```

## 5. 置信度推导规则

### 5.1 总则

`confidence` 只允许三个值：`observed`、`inferred`、`unknown`。

推导必须基于可判定规则，不允许自由推断。

### 5.2 Observed 的来源

事实被标记为 `observed` 当且仅当：

| 来源 | 条件 | 示例 |
|---|---|---|
| CRG AST | CRG 解析出的节点/边，有确定的 AST 节点类型和源码位置 | `entrypoints` 中 CRG `list_flows` 返回的入口函数 |
| 配置文件 | 直接读取 package.json、go.mod 等元文件获得的确定性信息 | `project_identity.primary_language` |
| CRG 查询 | `query_graph` 返回的非空结果集 | `tests_for` 返回的 TESTED_BY 边 |
| CRG 统计 | `find_large_functions`、`get_impact_radius` 返回的结构化数据 | 超阈值函数的行数和位置 |

约束：
- CRG 返回的节点一定有 `file_path` + `line_start` + `line_end`，这些作为 `evidence` 字段
- CRG 返回的边一定有 `source` + `target` + `kind`，这些作为 `evidence` 字段

### 5.3 Inferred 的来源

事实被标记为 `inferred` 当且仅当：

| 来源 | 条件 | 必须附加 |
|---|---|---|
| 目录命名模式 | 路径匹配预定义模式（`src/controllers/` → controller 层）| `inference_reason: "directory-naming-pattern"` |
| Glob 路径模式 | Glob 匹配特定目录/文件名模式（如 `src/models/**` → data_shapes）| `inference_reason: "glob-path-pattern"` |
| Serena 模式搜索 | `search_for_pattern` 命中 | `inference_reason: "serena-pattern-match"` |
| 路径近邻 | 测试文件路径与源码路径的近邻关系 | `inference_reason: "path-proximity"` |
| 框架标记推断 | 从 import 语句推断使用的框架 | `inference_reason: "framework-import-inference"` |
| CRG importers 证据 | CRG `importers_of` 返回 AST 级 IMPORTS_FROM 边，但库名→语义分类仍需预置清单映射 | `inference_reason: "crg-importers-evidence"` |
| CRG 语义搜索证据 | `semantic_search_nodes_tool` 返回 AST 节点，但 kind 分类（dto/entity/schema）由节点名称关键词推断 | `inference_reason: "crg-semantic-search-evidence"` |
| Grep import 推断 | 文本匹配 import 语句 | `inference_reason: "grep-import-pattern"` |

约束：
- 每条 `inferred` 事实**必须**带 `inference_reason`
- `inferred` 不得单独触发高严重性结论（如 `high` severity risk signal），除非有 ≥2 个独立信号共同支持
- Basic 模式下，大多数事实为 `inferred`
- `confidence` 字段在 JSON 中只写 `observed` / `inferred` / `unknown` 三种值，不得写扩展标注（如 `Inferred(Observed-evidence)`）；证据类型通过 `inference_reason` 字段区分

### 5.4 Unknown 的来源

事实被标记为 `unknown` 当且仅当：

- 缺少足够直接证据，也缺少足够启发式支持
- `unknown` 只允许进入降级路由，不得进入强结论分支

### 5.5 模式与置信度的关系

```
Full 模式 (CRG):
  entrypoints       → observed（AST 级）
  modules            → observed（社区检测级）
  testing_surface    → observed（TESTED_BY 边级）
  risk_signals       → observed（fan-out/complexity 统计级）
  integrations       → inferred（inference_reason: "crg-importers-evidence"，有 AST 边支持）
  data_shapes        → inferred（inference_reason: "crg-semantic-search-evidence"；节点存在由 AST 确认，kind 分类由节点名称关键词推断）

Enhanced 模式 (Serena):
  entrypoints       → inferred（模式搜索级）
  modules            → inferred（符号结构推断）
  testing_surface    → inferred（路径近邻推断）
  risk_signals       → inferred（模式匹配推断）
  integrations       → inferred（模式匹配推断）
  data_shapes        → inferred（模式匹配推断）

Basic 模式 (Built-in):
  entrypoints       → inferred（文本匹配推断）
  modules            → inferred（目录结构推断）
  testing_surface    → inferred（路径匹配推断）
  risk_signals       → inferred（Grep 模式推断）[severity 上限: medium，见下方约束]
  integrations       → inferred（Grep import 推断）
  data_shapes        → inferred（Glob + Grep 推断）
```

**Enhanced 模式 risk_signals severity 约束：**

Enhanced 模式下所有 risk_signals 均为 `inferred`，复用 §5.3 的 "≥2 个独立信号" 规则：

| 条件 | severity 上限 |
|---|---|
| 单一 Serena 模式搜索命中（如 `TODO\|FIXME` 数量 > 阈值）| `medium` |
| ≥2 个独立 Serena 信号同时命中（如大文件 + `find_referencing_symbols` 显示高引用数）| `high` 可用 |
| 仅来自符号名称推断（如函数名含 "core" / "shared"）| `medium`（命名推断不升级为 high）|

**Basic 模式 risk_signals severity 约束：**

Basic 模式下所有 risk_signals 均为 `inferred`，依据 §5.3 "单一 inferred 信号不得触发 `high` severity" 规则，Basic 模式的 severity 适用以下规则：

| 条件 | severity 上限 |
|---|---|
| 单一 Grep 模式命中（如 `TODO\|FIXME` 数量 > 阈值）| `medium` |
| ≥2 个独立 Grep 信号同时命中（如大文件 + 无测试目录）| `high` 可用 |
| 来自 `src/core/`, `src/shared/` 等路径的推断 | `medium`（路径推断不升级为 high）|

这确保 Enhanced / Basic 模式不因启发式匹配误判产生虚假的高风险告警，与 Full 模式的 AST 级 `high` severity 形成可信度区分。

失效原则（v1 执行 stale 检测提示，不做自动重建；阶段 4 实现自动重建）：
- CRG graph_mtime 变化 → 所有 `crg:*` 依赖的产物应标记为 stale
- 文件 SHA 变化 → 对应 `files.*` 依赖的产物应标记为 stale
- analyzer 版本变化 → 对应依赖的产物应标记为 stale
- 上游产物 stale → 下游产物连带 stale

**v1 实现：写入依赖关系 + 运行时 stale 检测警告（不自动重建）。阶段 4 实现自动失效与重建。**

v1 stale 检测逻辑（在 Phase 0 探测阶段执行）：

```
如果 fingerprints.json 存在：
  1. 读取 inputs.crg.graph_mtime，对比当前 list_graph_stats_tool 返回的 mtime
     → 不一致：打印 ⚠️ "CRG 图已更新（上次: X，当前: Y），以下产物可能过期: <依赖 crg:* 的产物列表>"
  2. 读取 inputs.files 中的关键文件 SHA，对比当前实际 SHA
     → 不一致：打印 ⚠️ "文件 <path> 已变更，以下产物可能过期: <依赖该文件的产物列表>"
  3. 如有任何 stale 信号，在探测输出末尾追加：
     "⚠️ 检测到 N 个 stale 信号，建议重新运行 spec-graph-bootstrap 以刷新产物。"
  4. stale 不阻断流程，用户可选择继续使用现有产物

如果 fingerprints.json 不存在（首次运行）：
  → 跳过 stale 检测，正常继续
```

这保证 v1 的 fingerprints.json 不只是静态文档——Phase 0 的 stale 检测给用户提供可见的新鲜度反馈，同时为阶段 4 的自动重建奠定基础。

## 7. 降级与容错

### 7.1 三级降级

```
Full (CRG)
  ↓ CRG 不可用或未索引
Enhanced (Serena)
  ↓ Serena 不可用
Basic (Built-in)
```

降级不阻断主流程。降级后的事实置信度按 §5.5 规则降级。

### 7.2 单工具失败处理

| 场景 | 动作 |
|---|---|
| CRG `list_flows` 超时 | 降级到 `Grep` 获取入口，该部分事实标 `inferred` |
| CRG `query_graph` 返回空 | 该查询维度标为 `unknown`，不影响其他维度 |
| Serena `find_symbol` 失败 | 降级到 `Grep`，该部分事实标 `inferred` |
| 任何工具调用异常 | 写入 `fact-inventory.json` 顶层的 `generation_errors[]`（格式见 §4.11），继续其他维度 |
| `build_or_update_graph_tool` 超时（>10 分钟）| 中止构建，自动降级到 Enhanced 模式；向用户报告"构建超时，已降级"；`crg.indexed` 保持 `false`；写入 `generation_errors[{extractor:"crg_build", error:"timeout", fallback_applied:true}]` |
| `build_or_update_graph_tool` 返回错误 | 向用户显示具体错误信息；自动降级到 Enhanced 模式；将错误写入 `generation_errors[]` |
| 构建过程中用户中断 | 等效于"用户选择否"，直接降级到 Enhanced 模式；已中断的图数据库不参与后续查询 |

### 7.3 部分产物缺失策略

```yaml
# injection-index.yaml 中缺失产物处理
missing_file_strategy:
  code-facts/public-entrypoints.md: "跳过，使用 00-summary.md 替代"
  context-packs/review-change.md: "跳过，使用 unknown 回退集合"
  code-facts/test-map.md: "跳过，不影响 review 路由"
```

任何缺失都必须在 `README.md` 的 Freshness 章节和 `injection-index.yaml` 的 `caveats` 中可见。

| Fact Inventory 字段 | 消费者 | 最终产物 |

| 所有字段 | `injection-context` worker | `context-packs/review-change.md` |

> **v1 范围说明**：`integrations` 和 `data_shapes` 在 v1 中以"补充节"形式注入现有产物（`pitfalls/index.md` 和 `architecture/module-map.md`），不单独生成新文件。若未来需要独立产物（如 `code-facts/integrations.md`），由阶段 3 规划。

数据流约束（Facts First 执行规则）：
- Worker **必须优先**读取控制面事实文件（`fact-inventory.json` 等）
- Worker **默认不得**回扫源码补洞
- 如确需回读，PRD 中必须设置 `source_read_allowed = true` 并记录 `fallback_reason`
- Worker 输出中超出事实层的推断，必须标为 `inferred`

## 10. 验收标准

### 10.1 工具链验收

| 检查项 | 标准 |
|---|---|
| CRG 探测 | 能正确区分 indexed / not-indexed / unavailable 三种状态 |
| 降级链路 | CRG 不可用时能自动降级到 Serena/Basic，不阻断 |
| 事实层完整性 | `fact-inventory.json` 包含所有 8 个数据字段（`project_identity`、`entrypoints`、`modules`、`integrations`、`testing_surface`、`data_shapes`、`layers`、`database`）及 4 个元字段（`schema_version`、`analyzer_mode`、`generated_at`、`source_snapshot`）|
| 置信度合规 | 所有 `inferred` 事实都有 `inference_reason` |

### 10.2 模式对比验收

至少在 1 个真实项目上分别用 Full 和 Basic 模式运行，对比：
- Full 模式事实层的 `observed` 比例应显著高于 Basic
- Basic 模式的产物不得出现无 `inference_reason` 的 `inferred` 事实
- 两种模式的最终产物目录结构一致

### 10.3 增量基础验收

- `fingerprints.json` 正确记录 CRG `graph_mtime`、`node_count`、`edge_count` 和关键输入文件 SHA
- `fingerprints.json` 的 `outputs` 部分正确记录产物依赖关系（`depends_on` 字段）
- v1 stale 检测：当 `fingerprints.json` 存在且 CRG 图有更新时，Phase 0 必须在探测输出中打印 ⚠️ stale 警告
- v1 stale 检测：当关键文件 SHA 变更时，Phase 0 必须指出具体变更文件和受影响产物
- v1 不要求自动重建，stale 警告不阻断主流程
- `outputs` 的依赖结构必须可被阶段 4 直接扩展为自动重建逻辑，不需要 schema 变更

### 10.4 降级可见性验收

- 降级事件在执行摘要中向用户报告
- 降级在 `fact-inventory.json` 的 `analyzer_mode` 中记录
- 降级在 `README.md` Freshness 章节中体现

## 11. 风险与约束

### 11.1 已知风险

| 风险 | 影响 | 缓解 |
|---|---|---|
| CRG 图构建耗时（普通项目 2-5 分钟，大型 monorepo 可达 10 分钟）| 用户等待时间过长 | Phase 0 给出含时间范围的明确提示，允许用户选择跳过；超过 10 分钟自动降级 |
| CRG 对某些语言支持不完整 | 事实层部分字段为空 | 降级到 Serena/Built-in 补充 |
| MCP 工具名前缀因配置而异 | SKILL.md 中硬编码工具名可能不匹配 | Phase 0 探测时动态确认工具可用性 |

### 11.2 实施约束

- SKILL.md 中的 CRG 工具名使用逻辑名（如 `list_flows`），实际 MCP 调用名由 Phase 0 探测后确定
- 不在本阶段引入新的 MCP server 安装逻辑——CRG 的安装由 mcp-setup skill 负责
- 不修改 CRG 源码——仅消费其 MCP 接口

### 11.3 Monorepo 边界约束（v1）

**v1 的 monorepo 支持范围：**

| 情况 | v1 行为 |
|---|---|
| `repo_shape = monorepo` | `build_or_update_graph_tool(repo_root=<整个 monorepo 根>)`，CRG 扫描全局 |
| `list_communities_tool` | 返回跨 package 的全局社区，`modules[]` 中条目可能跨越多个子包 |
| `get_impact_radius_tool` | 可跨 package 计算 fan-out，v1 不限制 |
| 子包级单独分析 | v1 不支持——`repo_root` 只能是整个 monorepo，不能是单个子包 |
| 子包级产物隔离 | v1 不支持——`fact-inventory.json` 描述整个 repo，不区分子包 |

**约束说明**：monorepo 场景下，CRG 的社区检测可能将不同子包的代码识别为同一社区（因为它们共享依赖）。这是预期行为——v1 以"整体项目理解"为目标，子包级精度留给后续阶段。实现时应在产物的 `project_identity.repo_shape = "monorepo"` 标注中提示用户这一限制。

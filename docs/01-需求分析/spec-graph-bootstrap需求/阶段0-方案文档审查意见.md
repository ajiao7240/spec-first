## P0 问题

### 1. `CALLS / IMPORTS_FROM` 缺少“从原始引用归一化到 `target_id`”的权威闭环

严重性：P0

影响：

- `crg query`
- `crg impact`
- `crg flows`
- `crg review-context`
- 同名函数去歧义能力

问题说明：

参考实现中，`extractCalls()` 产出的是 `target_name`，`extractImports()` 产出的是 `target_path_raw`。但主方案中的 `edges` 表只接受 `source_id / target_id` 这一类 canonical 边。文档没有补上“原始引用如何解析成唯一目标节点”的权威步骤。

这会导致两个直接后果：

1. 关系边无法可靠落库。
2. 即便能落库，也无法保证同名符号不串线。

这不是实现细节，而是数据流缺了一段关键闭环。

证据：

- [阶段0-核心算法实现参考.md:213](/Users/kuang/xiaobu/spec-first/docs/01-需求分析/spec-graph-bootstrap需求/阶段0-核心算法实现参考.md:213)
- [阶段0-核心算法实现参考.md:232](/Users/kuang/xiaobu/spec-first/docs/01-需求分析/spec-graph-bootstrap需求/阶段0-核心算法实现参考.md:232)
- [阶段0-CRG-NodeJS集成技术方案.md:278](/Users/kuang/xiaobu/spec-first/docs/01-需求分析/spec-graph-bootstrap需求/阶段0-CRG-NodeJS集成技术方案.md:278)
- [阶段0-CRG-NodeJS集成技术方案.md:581](/Users/kuang/xiaobu/spec-first/docs/01-需求分析/spec-graph-bootstrap需求/阶段0-CRG-NodeJS集成技术方案.md:581)

建议修订：

1. 在主方案中新增“边解析阶段”并写成权威流程：
   - `parse`
   - `write raw refs`
   - `resolve raw refs -> canonical edges`
   - `postprocess`
2. 显式定义中间态存储，建议增加：
   - `unresolved_calls`
   - `unresolved_imports`
3. 写明解析优先级：
   - `CALLS`：同文件 > 同类/同作用域 > 同名唯一 > 失败则 unresolved
   - `IMPORTS_FROM`：路径规范化 > module 解析 > 失败则 unresolved
4. 写明 `query / impact / flows / review-context` 只消费已解析 canonical 边，不消费 unresolved 边。

建议增加验收：

- 同名函数 fixture 下不会串线。
- import alias / relative import / index module 三类路径解析 fixture。
- unresolved 边有明确统计输出。

### 2. `surprising-connections` 依赖 `nodes.confidence`，但 schema 无该字段

严重性：P0

影响：

- `crg surprising-connections`
- `confidence/source_tier/inference_reason` 契约一致性

问题说明：

主方案把 `confidence/source_tier/inference_reason` 定义为输出契约，但 schema 中并没有 `nodes.confidence` 字段。参考实现的 `surprisingConnections()` 却直接在 SQL 中读取 `sn.confidence`。

这意味着按文档实现会直接失败，属于明确的内部矛盾。

证据：

- [阶段0-核心算法实现参考.md:758](/Users/kuang/xiaobu/spec-first/docs/01-需求分析/spec-graph-bootstrap需求/阶段0-核心算法实现参考.md:758)
- [阶段0-CRG-NodeJS集成技术方案.md:255](/Users/kuang/xiaobu/spec-first/docs/01-需求分析/spec-graph-bootstrap需求/阶段0-CRG-NodeJS集成技术方案.md:255)
- [阶段0-CRG-NodeJS集成技术方案.md:474](/Users/kuang/xiaobu/spec-first/docs/01-需求分析/spec-graph-bootstrap需求/阶段0-CRG-NodeJS集成技术方案.md:474)

建议修订：

必须统一存储模型，建议二选一：

1. 若 `confidence` 是持久化字段，则加进 schema，且明确是 node 级还是 edge 级。
2. 若 `confidence` 只是 CLI 输出字段，则参考实现不得直接查询 `nodes.confidence`。

更合理的建议是：

- `nodes` 不存 `confidence`
- `edges` 或 `unresolved_*` 存 `confidence/source_tier/inference_reason`
- CLI 输出 facts 时统一映射

建议增加验收：

- `Inferred` 边必须带 `inference_reason`
- `Observed` 边不要求 `inference_reason`
- `surprising-connections` 在无 `confidence` 持久化字段时仍可运行

### 3. 通用社区算法只给 `module` 节点分社区，无法支撑函数级跨社区分析

严重性：P0

影响：

- `crg communities`
- `crg architecture`
- `crg surprising-connections`
- 后续模块耦合分析

问题说明：

当前 3-Pass 社区检测以 `kind='module'` 的节点为输入分组，并只将这些节点写回 `community_id`。但后续高价值分析消费的却是函数/类/方法级语义边。

这会导致函数和类节点很可能没有社区归属，跨社区分析在语义节点层面失效。

尤其 `surprisingConnections()` 的 SQL 过滤条件要求 `sn.community_id IS NOT NULL` 且跨社区；如果函数节点没有社区归属，就会被大面积排除。

证据：

- [阶段0-核心算法实现参考.md:559](/Users/kuang/xiaobu/spec-first/docs/01-需求分析/spec-graph-bootstrap需求/阶段0-核心算法实现参考.md:559)
- [阶段0-核心算法实现参考.md:678](/Users/kuang/xiaobu/spec-first/docs/01-需求分析/spec-graph-bootstrap需求/阶段0-核心算法实现参考.md:678)
- [阶段0-核心算法实现参考.md:767](/Users/kuang/xiaobu/spec-first/docs/01-需求分析/spec-graph-bootstrap需求/阶段0-核心算法实现参考.md:767)
- [阶段0-CRG-NodeJS集成技术方案.md:269](/Users/kuang/xiaobu/spec-first/docs/01-需求分析/spec-graph-bootstrap需求/阶段0-CRG-NodeJS集成技术方案.md:269)

建议修订：

1. 明确社区边界是文件级边界，不是仅 module 节点边界。
2. 文件社区确定后，文件内所有语义节点应继承同一个 `community_id`。
3. iOS Pod 社区检测也应遵循同样传播逻辑，而不是只给 module 打标。

推荐表述：

- `community_id` 表示“节点所属社区，默认从其 `file_path` 映射得到”

建议增加验收：

- 函数节点可参与跨社区惊喜边分析。
- `architecture.hub_nodes` 中的实体节点具有有效 `community_id`。

### 4. 社区重建事务顺序与外键关系冲突，存在脏引用风险

严重性：P0

影响：

- `crg postprocess`
- 连续重建稳定性
- 外键完整性

问题说明：

主 schema 明确 `nodes.community_id REFERENCES communities(id)`。但通用社区和 iOS Pod 社区两个写入实现都先执行 `DELETE FROM communities`，再插入新社区并更新节点。

如果启用 SQLite 外键检查，这一顺序可能直接失败；即使未开启，也会在事务过程中产生悬空引用，给调试和后续查询带来不确定性。

证据：

- [阶段0-CRG-NodeJS集成技术方案.md:269](/Users/kuang/xiaobu/spec-first/docs/01-需求分析/spec-graph-bootstrap需求/阶段0-CRG-NodeJS集成技术方案.md:269)
- [阶段0-CRG-NodeJS集成技术方案.md:294](/Users/kuang/xiaobu/spec-first/docs/01-需求分析/spec-graph-bootstrap需求/阶段0-CRG-NodeJS集成技术方案.md:294)
- [阶段0-核心算法实现参考.md:694](/Users/kuang/xiaobu/spec-first/docs/01-需求分析/spec-graph-bootstrap需求/阶段0-核心算法实现参考.md:694)
- [阶段0-核心算法实现参考.md:1546](/Users/kuang/xiaobu/spec-first/docs/01-需求分析/spec-graph-bootstrap需求/阶段0-核心算法实现参考.md:1546)

建议修订：

统一为安全事务模板，至少二选一：

1. 先 `UPDATE nodes SET community_id = NULL`，再删旧社区，再插新社区，再回填。
2. 用临时表重建并 swap，避免直接删父表。

同时必须在主方案中明确：

- 是否启用 `PRAGMA foreign_keys=ON`
- 社区重建必须满足外键一致性

建议增加验收：

- 外键开启时，连续两次 `crg postprocess` 不报错。
- 社区重建中途失败不会留下悬空 `community_id`。

严重性：P1

影响：

- iOS 项目输入收敛
- Pod 边界社区检测
- 验收标准可达性

### 6. native fallback 策略范围过大，能力边界未写实

严重性：P1

影响：

- 安装可用性承诺
- 运行时一致性
- 测试范围和支持矩阵

问题说明：

主方案一方面以 `better-sqlite3`、WAL、FTS5、同步 API 为基础设计实现；另一方面又提出 `sql.js` 和 tree-sitter WASM fallback。

这两套实现模型并不等价。当前文档没有补足以下关键约束：

- fallback 是否支持 WAL
- fallback 是否支持 FTS5
- 数据是否仍落盘到同一路径
- API 是否仍维持同步 prepared statements 语义
- 哪些命令在 fallback 模式下降级或禁用

在没有明确抽象层和能力矩阵前，这个 fallback 更像风险扩散点，而不是缓解措施。

## 推荐修订顺序

# `spec-graph-bootstrap` 方案本质问题总结

## 一、根本问题：AI 没有持久化的、可信的代码结构知识

当前 AI 编码工作流每次都从零开始理解项目。它要么**全量扫仓库**（Token 浪费、时间长），要么**靠推断猜结构**（幻觉风险高）。无论哪种，它都没有一个稳定的、可复用的、可信度标注的项目知识库。

这导致五类连锁问题：

| 症状 | 根因 |
|---|---|
| AI 改了错误的地方，漏改了真正的入口 | 不知道真实入口点在哪里 |
| AI 修复了函数，但触发了隐藏调用者的回归 | 不知道影响半径 |
| AI 写的代码覆盖了空白区，测试却在另一个目录 | 不知道测试映射 |
| 每次启动新任务都重新扫一遍代码 | 没有增量缓存，知识不持久 |
| 上下文加载了大量无关文档，或加载不足 | 没有任务级别的最小上下文路由 |

---

## 二、现有 `spec-bootstrap` 的缺口

现有方案产出的是**叙事型文档**，不是可被机器消费的结构化事实：

1. **缺代码事实层**：没有真实入口点、依赖方向、测试映射、高风险模块的结构化记录
2. **缺任务型上下文包**：产物是知识文档，而不是"改这类代码时先读什么"的行动指南
3. **缺注入路由**：没有机器可读的规则来决定哪个阶段/任务类型加载哪些上下文
4. **缺增量思维**：每次全量重扫，没有 SHA 变更检测和局部刷新
5. **缺消费闭环验证**：生成的文档是否真的被后续 workflow 利用，没有验证机制

---

## 三、解法框架：三层知识系统

```
目标仓库
  ↓ Phase 1: 事实抽取（AST + 语义 + 启发式，三级工具栈）
  ┌─────────────────────────────────────────────┐
  │ 事实层（控制面，机器读）                      │
  │  fact-inventory.json   risk-signals.json    │
  │  test-surface.json     fingerprints.json    │
  └─────────────────────────────────────────────┘
  ↓ Phase 3: 文档生成（只消费事实层，不回扫源码）
  ┌─────────────────────────────────────────────┐
  │ 摘要层（人可读）                             │
  │  00-summary.md         module-map.md        │
  │  pitfalls/index.md     code-facts/*.md      │
  └─────────────────────────────────────────────┘
  ↓ Phase 4: 路由生成
  ┌─────────────────────────────────────────────┐
  │ 注入层（机器可消费）                         │
  │  context-packs/review-change.md             │
  │  injection-index.yaml                       │
  └─────────────────────────────────────────────┘
          ↓ 按 stage × task_type 路由注入
  plan / work / review / compound 工作流
```

---

## 四、五条核心设计原则

| 原则 | 含义 | 关键约束 |
|---|---|---|
| **Facts First** | 先抽事实，再写文档 | 摘要层 worker 不得默认回扫源码 |
| **Honest Confidence** | 区分 Observed / Inferred / Unknown | `Inferred` 必须带 `inference_reason`，不允许无证据的 `Observed` |
| **Graph-Informed, Not Graph-Dependent** | 借 AST 结构思维，不强依赖图系统才能运行 | Full/Enhanced/Basic 三级降级，不阻断 |
| **Injection-Oriented** | 输出服务后续节点，不追求文档完备 | `injection-index.yaml` 是可判定路由，不是推荐列表 |
| **Incremental-Friendly** | 首次全量，后续按变化刷新 | `fingerprints.json` 记录输入 SHA + 产物依赖，支持局部失效 |

## 五、工具栈与置信度模型

```
Tier 1: code-review-graph（tree-sitter AST）→ Observed  （确定性）
Tier 2: Serena MCP（语义索引）            → Inferred  （推断性）
Tier 3: Built-in Grep/Glob               → Inferred  （启发式）
```

任何层的失败都触发降级，降级必须对用户和后续节点可见，不得静默跳过。

### 实现侧红线（v1）

为避免“原则正确、实现走偏”，v1 需额外满足以下工程红线：

1. **标识无歧义**：事实与关系优先使用稳定标识（如 `symbol_key`），避免仅用裸 `name` 造成同名串线。
2. **契约可判定**：`confidence / source_tier / evidence / inference_reason` 字段和枚举固定，能被 schema 测试直接判定。
3. **降级不阻断**：图能力故障（如未构建/损坏）只触发降级链，不能阻断 `plan/work/review` 主流程。
4. **收益可量化**：除“可生成”外，必须追踪 token 降幅、fallback 命中率、review 覆盖率等效果指标。

## 六、四个实施阶段各自解决的本质子问题

整套方案分四个阶段推进，每个阶段都对应一个本质子问题：

### 阶段 1：如何安全替换生产工具而不破坏现有用户

**本质问题**：工具系统的迁移往往是最容易被低估的风险。旧 skill 有真实用户，新旧并存阶段一旦安装逻辑有冲突，就会破坏现有工作流。

**解法**：
- 双入口并行，旧入口 `/spec:bootstrap` 不动，新入口 `/spec:graph-bootstrap` 独立存在
- `spec-first init` 保持幂等，重复执行不产生脏状态
- 产物路径复用原有约定（`docs/contexts/<slug>/`），不引入新的平行目录
- 平台差异（Claude vs Codex）由 adapter 层吸收，产品契约统一表述

**阶段 1 的退出条件**：新入口可被宿主发现并调用，旧入口不受影响。

---

### 阶段 2：如何从代码库中抽取可信的结构化事实

**本质问题**：AI 能读代码，但"读"得到不等于"理解"得对。叙事型总结容易把推断写成确定，把局部推广为全局，把缺失的部分用想象填充。需要一个机制让每条事实都带着"它从哪来、有多可信"。

**解法（阶段 2A：事实层）**：
- 三级工具栈，不同工具产出不同置信度：AST 给 `Observed`，语义搜索给 `Inferred`，Grep 给 `Inferred`
- 每条 `Inferred` 事实必须带 `inference_reason`，枚举值固定，不允许自由填写
- 控制面事实文件（`fact-inventory.json` 等）是唯一事实源，后续产物只能消费它，不得自行回扫
- `fingerprints.json` 记录输入文件 SHA、分析器版本、产物依赖，为增量刷新提供基础

**解法（阶段 2B：摘要与路由生成）**：
- 摘要层只压缩事实，不重新产生结论
- `injection-index.yaml` 必须是可判定的路由规则（`stage × task_type × output_exists.*`），不是推荐列表
- `unknown` task type 必须有最小回退集合，不允许阻断 workflow

**阶段 2 的退出条件**：v1 最小闭环产物稳定生成，路由规则可被人工消费。

---

### 阶段 3：如何证明这些产物对后续 AI 工作流真的有用

**本质问题**：生成了文档不等于被使用，被使用不等于有帮助。上下文准备和编码质量之间有一条"最后一公里"——如果 `plan / work / review` 不按路由规则读正确的文档，整套系统价值归零。

**解法（阶段 3A：非侵入式验证）**：
- 先在需求文档里定义"项目级治理说明草案"，不直接改 skill 代码
- 用真实任务手工验证 `plan / work / review` 是否按约定读取正确产物
- 填写统一验证记录（`expected_inputs` vs `actual_inputs`、`fallback_reason`、`verdict`、`allow_enter_3b`）
- **只有 3A 验证通过的规则，才允许进入 3B 固化**——这是防止把未经验证的规则写死进系统的关键 gate

**解法（阶段 3B：skill 消费固化）**：
- 把验证通过的消费顺序和 fallback 规则写入 `spec-plan / spec-work / spec-review` 的 `SKILL.md`
- 降级链：`injection-index.yaml` → 固定最小集合 → workflow 自带的 repo research / diff analysis
- Stage-0 产物是**增强项**，不是主任务的前置阻断条件；任何单文件缺失只能触发降级，不能让 workflow 失败

**阶段 3 揭示的更深层问题**：
> 消费质量的上限，不是事实层有多准，而是 task pack 里的内容结构和 workflow 的 prompt 设计有多好。`context-packs/review-change.md` 里究竟写什么，直接决定了 `spec-review` 在 review 时能补看多少风险传播面。这是整套系统效果的决定层。

**阶段 3 的退出条件**：至少 2 类真实任务验证通过，降级链不阻断主任务，3A 记录可追溯。

---

### 阶段 4：如何让知识系统不随代码演进而腐烂

**本质问题**：代码在变，但生成一次的文档不会自动变。过期的上下文比没有上下文更危险——它会以"确定性"的口吻提供错误的事实，下游 AI 会高置信度地做出错误决策。

**解法**：
- `fingerprints.json` 不是简单的文件哈希清单，而是"输入文件 SHA + 分析器版本 + schema 版本 + 产物依赖关系"的统一索引
- 局部失效映射（入口变 → 只重算 `public-entrypoints.md`；测试变 → 只重算 `test-map.md`），不因一个文件变化而全量重跑
- 全量失效触发条件明确（顶层目录结构变化、多入口同时迁移、层级识别变化），不靠人工判断
- 刷新结果对外可见：本次是全量还是局部、重算了哪些、跳过了哪些及原因
- 非 v1 必需产物（如 `context-packs/add-endpoint.md`）未生成时，刷新引擎必须 `skip/no-op`，不得升级为失败

**阶段 4 是删除旧 `/spec:bootstrap` 之前的最后一道门**：增量刷新稳定、至少 2 个真实项目验证通过、plan/work/review 质量指标没有明显恶化，才可以做入口切换。

## 七、本质问题的完整链路

```
[问题] AI 工作流每次冷启动
    → 全量扫 or 靠推断猜 → 幻觉 / Token 浪费 / 低质量上下文
         ↓
[方案] 建立持久化结构化知识层
    ├── 阶段1：安全接入现有工具体系（不破坏旧链路）
    ├── 阶段2：抽取可信事实（置信度标注 + Facts First）
    ├── 阶段3：证明产物被正确消费（验证优先于固化）
    └── 阶段4：让知识随代码演进保持新鲜（增量刷新）
         ↓
[终局] AI 工作流冷启动时，以最少 Token 加载最准确的项目知识
       不靠猜，不靠全量扫，靠持久化 + 可信度标注 + 增量刷新的事实系统
```

## 八、价值边界（诚实评估）

**这个系统能解决的：**
- 结构性幻觉（AI 误判入口、模块边界、测试路径）
- 大型仓库的 Token 预算问题（结构化导航替代全量扫）
- 重复分析的冗余成本（增量刷新复用）
- 上下文选择错误（injection-index 路由）
- 知识腐烂问题（fingerprints 驱动的局部失效）

**这个系统解决不了的：**
- 业务逻辑正确性（AST 不知道折扣规则对不对）
- 模型本身的推理能力上限
- task type 的可靠自动识别（v1 默认 `unknown`，靠 workflow 手动判定）

**最关键的一条认知**：

> 阶段 2 把"上下文准备"做得很扎实，但整套系统的最终效果取决于**阶段 3 的消费质量**——task pack 里的内容结构 + downstream workflow 的 prompt 设计，才是编码质量能否真正提升的决定因素。没有 3A 的验证记录，系统只是"理论上有用"。

---

## 九、一句话总结

> **`spec-graph-bootstrap` 本质上是在解决一个问题：让 AI 工作流在每个任务开始时，用最少的 Token 加载最准确的项目知识——不靠猜，不靠全量扫，靠一个持久化、可信度标注、可增量刷新的结构化事实系统。而这套系统能否真正提升编码质量，最终取决于消费端的 workflow 是否真的按路由规则使用了正确的上下文。**

# `spec-graph-bootstrap` 阶段 4：相关文档更新与刷新需求

> 本文是 `spec-graph-bootstrap` 的阶段 4 需求文档。
>
> 本文定义 `spec-graph-bootstrap` 在 rerun、增量刷新、freshness 和 confidence 维度上的长期维护行为。
>
> 如与 [修订终版.md](./修订终版.md) 冲突，以总版为准。
>
> 本文中的“阶段 4”指实施推进阶段，不等同于 skill 执行过程中的 `Phase 0` 到 `Phase 4`。

## 1. 目标

阶段 4 的目标是把 `spec-graph-bootstrap` 从“一次性上下文生成器”升级为“可长期维护的 Stage-0 知识层”。

阶段 4 完成后，系统应具备以下能力：

* rerun 时能识别已有上下文并复用 slug
* 能根据输入变化做局部失效和局部重算
* 产物能展示 freshness / confidence / analyzer version
* 用户能看见本次是全量刷新还是局部刷新

## 2. 范围

本阶段包含：

* `fingerprints.json` 的依赖模型完善
* rerun refresh 规则
* 局部失效与全量失效判定
* freshness / confidence 的对外展示
* 部分产物重算时的装配与可见性

本阶段不包含：

* 完整知识仓库导出体系
* 多仓库集中式刷新
* embedding / semantic search 主链路

## 3. 输入

阶段 4 依赖以下输入：

* 上一次运行留下的控制面状态
* 当前目标仓库文件快照
* analyzer version
* schema version
* 产物依赖关系
* 当前运行 mode 与支持状态

## 4. 核心产物

### 4.1 `fingerprints.json`

`fingerprints.json` 不是简单的文件哈希清单，而是统一索引：

* 输入文件指纹
* analyzer 版本
* schema 版本
* 产物依赖关系
* 上次刷新时间

约束：

* 阶段 2 必须先产出一个可读取的初版结构
* 阶段 2 的初版 contract 只要求稳定提供 `inputs`、`outputs`、`updated_at` 及最基本元信息
* 阶段 4 在不破坏已有顶层结构的前提下扩展依赖模型
* 阶段 4 不得通过重定义 schema 破坏阶段 2 已产出的基础文件

### 4.2 freshness 信息

正式产物应尽量展示：

* `updated_at`
* `analyzer_version`
* `schema_version`
* `source_snapshot`
* `confidence`

### 4.3 rerun 状态

用户应能看到：

* 本次是否复用已有 slug
* 本次是否触发 backup
* 本次是全量刷新还是局部刷新
* 哪些产物被重算
* 哪些产物被保留

## 5. 关键设计约束

### 5.1 增量刷新不能靠“看情况”

局部刷新必须基于可判定规则，而不是人工猜测。

### 5.2 刷新不能破坏消费合同

无论是局部刷新还是全量刷新，都不能改变阶段 3 已定义的消费路径和 fallback 集合。

### 5.3 freshness 必须对用户可见

如果产物可能过期，就必须显式暴露，而不是默认假设“最近一次生成仍然可信”。

### 5.4 部分失效必须透明

当某些产物因为依赖未命中、分析器能力不足或异常失败而未刷新时：

* 用户必须可见
* 后续节点必须可见
* 不得伪装为“最新”

### 5.5 非 v1 必需产物未生成时必须 no-op

阶段 4 的失效映射可以引用终局产物名，但 v1 并不要求这些产物已实际生成。

因此必须满足：

* 像 `context-packs/add-endpoint.md` 这类非 v1 必需产物，如果当前 slug 下从未生成，refresh engine 必须将其视为 `skip / no-op`
* `skip / no-op` 不是异常，不得升级为 rerun 失败，不得单独触发全量刷新
* 对外报告中可以说明“该产物当前未生成，因此本次跳过刷新”，但不得把它表述为错误
* 只有 v1 必需产物缺失，才需要进入异常诊断或回退路径讨论

## 6. 刷新模型

### 6.1 首次运行

首次运行默认生成“当前版本要求的全部目标产物”：

* facts
* 摘要
* code-facts
* v1 必需 task pack，以及当前命中的条件/增强 task pack
* injection index

### 6.2 rerun 复用

如果目标项目已有 `docs/contexts/<slug>/`：

1. 复用既有 slug
2. 先做 backup
3. 读取上次的 `fingerprints.json`
4. 计算输入变化和版本变化
5. 判定局部失效还是全量失效

### 6.3 局部失效映射

至少支持以下映射：

* 入口文件变化 -> `public-entrypoints.md`
* 测试变化 -> `test-map.md`
* `shared/core/common/util` 变化 -> `high-risk-modules.md`
* 路由/API 边界变化 -> `review-change.md`、相关 task pack；若某个 task pack 在当前 slug 下未生成，则该目标直接 `skip / no-op`
* analyzer / schema 版本变化 -> 相关依赖产物重算

### 6.4 重大结构变化

以下情况应升级为全量评估：

* 顶层目录结构明显变化
* 多个关键入口同时迁移
* 层级识别结果变化
* 大规模 rename / move 导致局部映射不再可信

## 7. 执行流程

### 7.1 rerun 前检查

1. 探测已有 contexts 和 slug
2. 读取旧 `fingerprints.json`
3. 比较输入与版本变化

### 7.2 失效判定

1. 标记直接失效产物
2. 标记连带失效产物
3. 对当前不存在的条件/增强产物标记 `skip / no-op`
4. 判断是否升级为全量刷新

### 7.3 刷新执行

1. 重跑受影响 extractors
2. 重跑受影响 renderers
3. 重写 freshness / confidence / source snapshot
4. 更新 `fingerprints.json`

### 7.4 对外报告

至少报告：

* 本次刷新类型：全量 / 局部
* 触发原因
* 重算的产物
* 跳过的产物及原因
* 当前 freshness 与 confidence 状态

## 8. 验收标准

### 8.1 rerun 验收

* 同一项目重复运行默认复用同一 slug
* rerun 前会创建有效 backup
* rerun 失败时能按既定策略恢复

### 8.2 增量验收

* 入口文件变化时，只重算相关入口产物
* 测试变化时，只重算测试映射相关产物
* 重大结构变化时，能升级为全量刷新
* 非 v1 必需且当前未生成的产物，在 rerun 中会被稳定视为 `skip / no-op`

### 8.3 freshness 验收

* 正式产物中能看见 freshness / confidence
* `Inferred` 结论在刷新后仍保留明确标记
* 分析器版本变化会触发相关产物重算

## 9. 与其他阶段的关系

* 阶段 4 依赖阶段 2 的产物模型和阶段 3 的消费路径已经稳定
* 阶段 4 的刷新行为不得改变阶段 1 定义的安装路径和阶段 2 定义的产物根目录
* 阶段 4 是删除旧 `/spec:graph-bootstrap` 前的重要稳定性门槛

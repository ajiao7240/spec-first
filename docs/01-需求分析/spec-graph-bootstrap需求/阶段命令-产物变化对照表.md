# 阶段命令 -> 产物变化对照表（spec-graph-bootstrap）

> 版本基线：对齐 `阶段0-CRG-NodeJS集成技术方案.md` + `修订终版.md`（2026-04-10）
> 目标：明确“执行哪个命令，会改哪些文件/目录，哪些只是只读查询”。

---

## 1. 路径变量与约定

- `<repo_root>`：被分析的目标仓库根目录
- `<slug>`：项目稳定命名空间（同时用于文档树与控制面）
- 图数据库目录：`<repo_root>/.spec-first-graph/`
- 输入收敛配置：`<repo_root>/.spec-first-graphignore`（仓库级配置文件，非运行时产物）
- 文档产物目录：`<repo_root>/docs/contexts/<slug>/`
- 控制面目录：`<repo_root>/.context/spec-first/bootstrap/<slug>/`

## 2. Phase 0（CRG）命令写入矩阵

说明：`spec-first crg` 大多数命令是只读查询。真正会改磁盘的主要是 `build` / `postprocess`。

| 命令 | 写入/更新文件 | 是否改写磁盘 | 备注 |
|---|---|---|---|
| `spec-first crg stats --repo=<repo_root>` | 无 | 否 | 读取 `graph.db` 统计信息；图不存在时 exit 2 |
| `spec-first crg context --repo=<repo_root>` | 无 | 否 | 只输出 JSON |
| `spec-first crg query ... --repo=<repo_root>` | 无 | 否 | 只输出 JSON |
| `spec-first crg impact ... --repo=<repo_root>` | 无 | 否 | 只输出 JSON |
| `spec-first crg large-functions ... --repo=<repo_root>` | 无 | 否 | 只输出 JSON |
| `spec-first crg search <keyword> --repo=<repo_root>` | 无 | 否 | 只输出 JSON |
| `spec-first crg flows --repo=<repo_root>` | 无 | 否 | 只输出 JSON |
| `spec-first crg flow <name> --repo=<repo_root>` | 无 | 否 | 只输出 JSON |
| `spec-first crg communities --repo=<repo_root>` | 无 | 否 | 只输出 JSON |
| `spec-first crg community <name> --repo=<repo_root>` | 无 | 否 | 只输出 JSON |
| `spec-first crg architecture --repo=<repo_root>` | 无 | 否 | 只输出 JSON |
| `spec-first crg detect-changes --since=<sha> --repo=<repo_root>` | 无 | 否 | 读取 git diff + 图，输出风险 JSON |
| `spec-first crg review-context --since=<sha> --repo=<repo_root>` | 无 | 否 | 只输出 JSON |
| `spec-first crg affected-flows --since=<sha> --repo=<repo_root>` | 无 | 否 | 只输出 JSON |
| `spec-first crg surprising-connections --repo=<repo_root>` | 无 | 否 | 只输出 JSON |
| `spec-first crg build --repo=<repo_root>` | `<repo_root>/.spec-first-graph/graph.db`（及 `-wal/-shm`） | 是 | 首次建库或增量更新；更新 nodes/edges/meta/fts 等 |
| `spec-first crg build --force --repo=<repo_root>` | 同上（全量重建） | 是 | 全量重算，覆盖旧索引结果 |
| `spec-first crg postprocess --repo=<repo_root>` | 同上（库内后处理结果） | 是 | 重算 flows/communities/FTS 等后处理结果 |

## 3. `spec-first init` 相关产物变化

> 该命令主要安装 skill/command 运行时资产；不写 MCP 配置。
> 说明：`.spec-first-graphignore` 属于仓库级 CRG 输入治理配置，建议由仓库维护（模板可由文档提供），`init` 默认不强制覆写。

| 命令 | 典型写入路径 | 是否改写磁盘 | 备注 |
|---|---|---|---|
| `spec-first init --claude` | `<repo_root>/.claude/skills/spec-graph-bootstrap/SKILL.md`；`<repo_root>/.claude/commands/spec/graph-bootstrap.md` | 是 | 幂等安装 |
| `spec-first init --codex` | `<repo_root>/.agents/skills/spec-graph-bootstrap/SKILL.md`；`<repo_root>/.codex/commands/spec/graph-bootstrap.md` | 是 | 幂等安装 |
| `spec-first init`（通用） | 宿主运行时资产（skills/commands） | 是 | 可选提示图未构建（检测 `.spec-first-graph/graph.db`） |

## 4. `/spec:graph-bootstrap`（阶段化执行）产物变化

> 下表按阶段说明“同一次 workflow 执行”会落哪些文件。入口命令在宿主侧表现为 `/spec:graph-bootstrap`（或等价 command 入口）。

## 5. rerun / refresh（同 slug 重跑）产物变化矩阵

> 规则来源：`修订终版.md` 的增量刷新契约（17.2/17.5）。

| 触发变化 | 局部刷新目标（写入/更新） | 不存在时处理 |
|---|---|---|
| 入口文件变化 | `docs/contexts/<slug>/code-facts/public-entrypoints.md` | — |
| `shared/core/common/util` 变化 | `docs/contexts/<slug>/code-facts/high-risk-modules.md`；`core-dependencies.md`（若已存在） | 未生成则 `skip/no-op` |
| 测试变化 | `docs/contexts/<slug>/code-facts/test-map.md` | — |
| 路由/API 边界变化 | `docs/contexts/<slug>/context-packs/add-endpoint.md`；`context-packs/review-change.md` | 未生成则 `skip/no-op` |
| 配置/依赖变化 | `high-risk-modules.md`；`injection-index.yaml`；`core-dependencies.md`（若已存在） | 未生成则 `skip/no-op` |
| rename/delete（关键边界） | 受影响模块相关文件；必要时升级全量 | 按全量评估策略处理 |
| 重大结构变化 | 全量重算 `docs/contexts/<slug>/` + 控制面关键文件 | 不走局部刷新 |

补充：

- `fingerprints.json` 在每次成功 refresh 后都会更新（输入 SHA、版本、依赖关系、刷新时间）。
- `skip/no-op` 属于正常分支，不应视为失败。

## 6. 快速判断：一个命令是否“会改文件”

- **会改文件**：`spec-first crg build`、`spec-first crg postprocess`、`spec-first init`、`/spec:graph-bootstrap`（首次或 rerun/refresh）
- **只读查询**：除 `build/postprocess` 外的大多数 `spec-first crg <subcommand>`

## 8. 文件级说明（作用 / 功能 / 主要内容）

### 8.1 图数据库运行时文件（`.spec-first-graph/`）

| 文件 | 作用 | 功能 | 主要内容 | 主要消费方 |
|---|---|---|---|---|
| `.spec-first-graph/graph.db` | 图数据主文件 | 提供 `crg` 全部查询与分析的数据基础 | SQLite 数据库，核心表含 `nodes / edges / communities / flows / flow_nodes / graph_meta / fts_nodes` | `crg` 子命令；间接服务 `plan/work/review` |
| `.spec-first-graph/graph.db-wal` | WAL 事务日志 | 提升写入吞吐并保证崩溃恢复 | SQLite WAL 增量日志（运行时自动维护） | `crg build/postprocess` 运行时 |
| `.spec-first-graph/graph.db-shm` | WAL 共享内存 | 支撑 WAL 并发读写协调 | SQLite 共享状态文件（运行时自动维护） | `crg build/postprocess` 运行时 |

### 8.2 控制面文件（`.context/spec-first/bootstrap/<slug>/`）

| 文件 | 作用 | 功能 | 主要内容 | 主要消费方 |
|---|---|---|---|---|
| `fact-inventory.json` | 事实总清单 | 作为后续文档渲染的唯一事实源 | 项目身份、入口、模块、集成、数据形态、风险等结构化事实；含置信度字段 | 渲染器；间接服务 `plan/work/review` |
| `risk-signals.json` | 风险信号集 | 支撑 review 与高风险模块文档 | 高扇出、高复杂度、低测试覆盖、跨层传播等风险条目与证据 | `review` 主消费；`work` 次消费 |
| `test-surface.json` | 测试覆盖面清单 | 支撑 `test-map.md` 与 review 关联测试推荐 | 测试目录结构、测试类型、模块到测试映射、空白区提示 | `work/review` |
| `fingerprints.json` | 增量刷新索引 | 驱动局部失效判断与 refresh 计划 | 输入文件 SHA、分析器/Schema 版本、产物依赖关系、上次刷新信息 | refresh 引擎；维护侧 |
| `tasks/<task-id>/prd.md` | 任务控制面文档 | 固化一次执行的任务约束与验收口径 | 任务目标、输入、输出、风险、验收标准、执行备注 | 当前任务执行 agent |

### 8.3 v1 必需文档产物（`docs/contexts/<slug>/`）

| 文件 | 作用 | 功能 | 主要内容 | 主要消费方 |
|---|---|---|---|---|
| `README.md` | 上下文控制台 | 给后续节点提供“先读什么”的导航入口 | Snapshot、阅读顺序、关键风险、可用 context packs、freshness 状态 | `plan/work/review`（兜底） |
| `00-summary.md` | 最小事实摘要 | 快速建立项目全局心智模型 | 项目定位、主技术栈、真实入口、核心模块、高风险摘要 | `always`（全阶段） |
| `architecture/module-map.md` | 模块地图 | 解释模块边界与连接关系 | 模块职责、依赖方向、常见变更入口、边界说明 | `plan` 主消费 |
| `pitfalls/index.md` | 坑位索引 | 预防常见误改与过度修改 | 按任务类型组织的风险点：位置、原因、风险、缓解建议、相关测试 | `always`（全阶段） |
| `code-facts/public-entrypoints.md` | 入口事实清单 | 指导“改哪里”与“从哪进” | HTTP/CLI/worker/SDK 等入口列表、路径与关键符号 | `plan/work` |
| `code-facts/test-map.md` | 测试映射清单 | 指导“改完测哪里” | 模块/函数与测试文件映射、覆盖提示、缺口提醒 | `work/review` |
| `code-facts/high-risk-modules.md` | 高风险模块清单 | 指导 review/work 的优先关注点 | 高耦合/高扇出/低覆盖模块、影响半径与风险说明 | `review` 主消费；`work` 次消费 |
| `context-packs/review-change.md` | review 任务包 | 为 review 场景提供最小必要上下文 | Purpose、When to use、Read first、Key modules、Related tests、Pitfalls、Avoid overreach、Escalation hints | `review` |
| `injection-index.yaml` | 注入路由表 | 决定不同 stage/task_type 的上下文加载集合 | `always`、`plan/work/review` 规则、条件分支、fallback 行为 | 路由装配器；`plan/work/review` 共同依赖 |

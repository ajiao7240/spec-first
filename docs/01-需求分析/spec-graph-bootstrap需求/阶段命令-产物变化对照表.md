# `spec-graph-bootstrap` 阶段命令 → 产物变化对照表

> 目标：把阶段 0 到阶段 4 内部每一步新增/修改的产物，以及这些产物对命令行为的影响做成可核对对照表。
>
> 说明：
>
> - 这里的“阶段命令”指 `spec-graph-bootstrap` 内部执行阶段，不是外部 CLI 子命令。
> - `spec-bootstrap` 继续保留；此表只描述 `spec-graph-bootstrap`。
> - 第一版控制面目录采用 `.spec-first/workflows/bootstrap/<slug>/`，文档目录采用 `docs/contexts/<slug>/`。
> - 旧 `.context/spec-first/bootstrap/<slug>/` 与 `fingerprints.json` 仅视为历史迁移阶段口径，不再作为当前实现验收标准。
> - 如与总版需求冲突，以 `修订终版.md` 为准。

## 1. 总览

| 阶段 | 主要动作 | 新增/更新产物 | 对外可见变化 |
| --- | --- | --- | --- |
| Phase 0 | readiness / slug / 模式判定 / rerun 准备 | `.spec-first/workflows/bootstrap/<slug>/artifact-manifest.json`（初版） | 尚无正式文档产物，但运行控制面建立 |
| Phase 1 | 事实抽取与风险整理 | `fact-inventory.json`、`risk-signals.json`、`test-surface.json`、`artifact-manifest.json`（补全 inputs） | 仍以控制面产物为主 |
| Phase 2 | 文档与路由生成 | `README.md`、`00-summary.md`、`architecture/module-map.md`、`pitfalls/index.md`、`code-facts/*`、`context-packs/review-change.md`、`injection-index.yaml`、`artifact-manifest.json`（补全 outputs） | `docs/contexts/<slug>/` 形成最小闭环 |
| Phase 3 | graph 增强 / 证据升级 / 消费固化 | 更新 `fact-inventory.json`、更新 `risk-signals.json`、按 v1 收敛 `injection-index.yaml` 与消费顺序 | 文档内容质量上升，命中更精确 |
| Phase 4 | rerun refresh / freshness 更新 | 增量更新 `artifact-manifest.json` 与受影响文档 | 相关文档进入可刷新维护状态 |

## 2. Phase 0：readiness 与控制面初始化

### 2.1 目标

在不生成正式上下文文档前，先建立本次 bootstrap 的运行控制面，确保后续阶段有稳定根目录、稳定模式判定和 rerun 基线。

### 2.2 产物变化

| 产物 | 路径 | 变化类型 | 说明 |
| --- | --- | --- | --- |
| 控制面目录 | `.spec-first/workflows/bootstrap/<slug>/` | 新建 | 当前 bootstrap run 的控制面根目录 |
| `artifact-manifest.json` | `.spec-first/workflows/bootstrap/<slug>/artifact-manifest.json` | 新建 | 首次写入基础元信息、输入来源、状态 |

### 2.3 建议最小字段

`artifact-manifest.json` 首次写入至少包含：

- `schema_version`
- `generated_at`
- `updated_at`
- `status`
- `inputs`

### 2.4 对命令行为的影响

- rerun 时可以复用 `<slug>`
- 后续阶段统一从控制面目录读取当前 run 状态
- readiness 失败或降级时，可在控制面中记录 mode / graph_support_state

## 3. Phase 1：事实层建立

### 3.1 目标

建立 facts-first 的结构化事实层，为后续文档渲染提供稳定输入。

### 3.2 产物变化

| 产物 | 路径 | 变化类型 | 说明 |
| --- | --- | --- | --- |
| `fact-inventory.json` | `.spec-first/workflows/bootstrap/<slug>/fact-inventory.json` | 新建 | 记录项目身份、模块、入口、测试、集成、数据形状等事实 |
| `risk-signals.json` | `.spec-first/workflows/bootstrap/<slug>/risk-signals.json` | 新建 | 记录风险信号、高风险模块、关键热点 |
| `test-surface.json` | `.spec-first/workflows/bootstrap/<slug>/test-surface.json` | 新建 | 记录测试结构、缺口、风险与样本 |
| `artifact-manifest.json` | `.spec-first/workflows/bootstrap/<slug>/artifact-manifest.json` | 更新 | 补充事实层输入、依赖与生成状态 |

### 3.3 对命令行为的影响

- 后续渲染器不需要默认回扫源码才能生成 v1 文档
- `Observed / Inferred / Unknown` 成为稳定值域
- `Inferred` 事实必须带 `inference_reason`

## 4. Phase 2：文档与路由生成

### 4.1 目标

基于事实层生成 v1 最小闭环文档与路由索引。

### 4.2 产物变化

| 产物 | 路径 | 变化类型 | 说明 |
| --- | --- | --- | --- |
| `README.md` | `docs/contexts/<slug>/README.md` | 新建 | 上下文控制台首页 |
| `00-summary.md` | `docs/contexts/<slug>/00-summary.md` | 新建 | 高层摘要 |
| `architecture/module-map.md` | `docs/contexts/<slug>/architecture/module-map.md` | 新建 | 模块与边界图谱 |
| `pitfalls/index.md` | `docs/contexts/<slug>/pitfalls/index.md` | 新建 | 风险与坑点索引 |
| `code-facts/public-entrypoints.md` | `docs/contexts/<slug>/code-facts/public-entrypoints.md` | 新建 | 对外入口事实 |
| `code-facts/test-map.md` | `docs/contexts/<slug>/code-facts/test-map.md` | 新建 | 测试映射与覆盖面 |
| `code-facts/high-risk-modules.md` | `docs/contexts/<slug>/code-facts/high-risk-modules.md` | 新建 | 高风险模块 |
| `context-packs/review-change.md` | `docs/contexts/<slug>/context-packs/review-change.md` | 新建 | review 专用上下文包 |
| `injection-index.yaml` | `docs/contexts/<slug>/injection-index.yaml` | 新建 | 定义 `always + stages + selection_rules + advice` 的 v1 注入规则 |
| `artifact-manifest.json` | `.spec-first/workflows/bootstrap/<slug>/artifact-manifest.json` | 更新 | 记录 outputs 与文档依赖 |

### 4.3 对命令行为的影响

- `plan / work / review` 开始具备可消费的 Stage-0 产物
- `README.md` 可作为人工导航台
- `injection-index.yaml` 成为后续消费方的主入口索引

## 5. Phase 3：增强分析与消费精化

### 5.1 目标

在 v1 最小闭环之上，利用 graph 或更强分析能力提升事实质量，并把已验证通过的最小消费规则固化到下游 workflow。

### 5.2 产物变化

| 产物 | 路径 | 变化类型 | 说明 |
| --- | --- | --- | --- |
| `fact-inventory.json` | `.spec-first/workflows/bootstrap/<slug>/fact-inventory.json` | 更新 | 补强入口、模块关系、集成、数据流事实 |
| `risk-signals.json` | `.spec-first/workflows/bootstrap/<slug>/risk-signals.json` | 更新 | 提升风险判断准确度 |
| `test-surface.json` | `.spec-first/workflows/bootstrap/<slug>/test-surface.json` | 更新 | 补强测试关联与薄弱区 |
| `injection-index.yaml` | `docs/contexts/<slug>/injection-index.yaml` | 更新 | 保持 v1 路由结构，`selection_rules` 当前只固化 `output_exists.*` |
| workflow skills | `skills/spec-plan/SKILL.md`、`skills/spec-work/SKILL.md`、`skills/spec-review/SKILL.md` | 更新 | 固化 Stage-0 消费顺序与 fallback 规则 |

### 5.3 对命令行为的影响

- 命中产物更精准，减少无关文档注入
- graph 可用时，`architecture/module-map.md` 与高风险模块判断质量提升
- 当前 v1 消费顺序固定为 `always[] -> stages.<stage>[] -> selection_rules(output_exists.*) -> advice.<stage>`
- `fact.*` 规则作为后续增强方向保留，当前消费链显式跳过

## 6. Phase 4：刷新与长期维护

### 6.1 目标

让 Stage-0 从一次性生成物变成可刷新、可声明 freshness 的长期资产。

### 6.2 产物变化

| 产物 | 路径 | 变化类型 | 说明 |
| --- | --- | --- | --- |
| `artifact-manifest.json` | `.spec-first/workflows/bootstrap/<slug>/artifact-manifest.json` | 更新 | 写入依赖模型、刷新结果、跳过原因 |
| 各 markdown 产物 | `docs/contexts/<slug>/...` | 局部更新或全量重写 | 根据变更影响范围刷新 |
| freshness 元信息 | 控制面/文档内嵌字段 | 新增/更新 | 暴露 freshness、confidence、analyzer version |

### 6.3 对命令行为的影响

- rerun 不再总是全量重算
- 可对消费方声明哪些产物是新鲜、哪些是降级或过时
- 刷新失败时可回退到上一个可用快照

## 7. 一张图看命令与产物关系

```text
Phase 0
  -> 建立 .spec-first/workflows/bootstrap/<slug>/
  -> 写入 artifact-manifest.json (初版)

Phase 1
  -> 写入 fact-inventory.json / risk-signals.json / test-surface.json
  -> 更新 artifact-manifest.json (inputs / status)

Phase 2
  -> 生成 docs/contexts/<slug>/ 全套 v1 文档
  -> 生成 injection-index.yaml
  -> 更新 artifact-manifest.json (outputs)

Phase 3
  -> 补强事实与风险判断
  -> 固化 Stage-0 消费顺序与 fallback 规则
  -> 保持 injection-index.yaml 为 always + stages + selection_rules + advice

Phase 4
  -> 按依赖模型局部或全量刷新
  -> 更新 artifact-manifest.json 与 freshness 信息
```

## 8. 当前建议如何使用这张表

这张表建议用于以下场景：

1. 审查某一阶段是否“真的有产物变化”，而不是只有提示词描述；
2. 核对控制面与文档面是否职责混淆；
3. 判断某个命令行为变化应该归因于哪类产物；
4. 未来设计 refresh 时，快速识别哪些文件属于控制面、哪些属于消费面。

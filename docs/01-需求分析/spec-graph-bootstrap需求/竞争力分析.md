# spec-graph-bootstrap 竞争力分析

> 基于 [业界方案调研.md](./业界方案调研.md) 提炼，做诚实的优劣势评估。
>
> 分析对象：spec-graph-bootstrap 整体方案（阶段 1-4）。

---

## 一、先明确竞争对象

业界方案分两类，竞争性质完全不同：

| 类型 | 代表 | 关系 |
|---|---|---|
| **IDE 内联工具**（Cursor、Copilot、Cody）| 嵌入编辑器，用户无感知索引 | **非直接竞争**，它们解决的是"AI 聊天时的上下文补全" |
| **工作流编排层**（Aider、OpenHands、SWE-Agent）| 驱动 AI 完成多步任务 | **直接竞争**，同样需要"任务前加载正确上下文" |
| **知识图谱工具**（Graphify、CodePrism、GraphRAG）| 建图 + 查询 | **能力重叠**，spec-graph-bootstrap 的 Tier 1 工具（CRG）本身就是这一类 |

spec-graph-bootstrap 的真实定位：**为 `plan / work / review` 等 AI workflow 提供可路由、可验证的上下文注入层**。这在业界几乎没有直接对标。

---

## 二、核心优势

### 优势 1：消费路由层（injection-index.yaml）— 业界空白

**业界现状**：所有主流工具在"上下文准备好了"之后就停止。消费端靠语义相似度模糊匹配，或靠用户手动 `@指定`。没有任何工具有"按 stage × task_type 路由最小必要上下文"的机制。

**spec-graph-bootstrap 的设计**：
```yaml
# 可判定规则，不是推荐列表
stages:
  review:
    always_include: [context-packs/review-change.md, code-facts/high-risk-modules.md]
    if: output_exists.test-map → include code-facts/test-map.md
```

**价值**：AI workflow 从"什么都加载 or 随机加载"变成"每种任务类型加载最小必要集合"。这是 token 效率提升的最后一公里。

---

### 优势 2：消费验证闭环（3A → 3B 门控）— 独有机制

**业界现状**：没有任何工具有"先验证消费规则有效，再固化到 skill"的机制。业界普遍是"写规则 → 直接上线 → 靠用户反馈发现问题"。

**spec-graph-bootstrap 的设计**：
- 3A：用真实任务手工验证路由是否命中正确文档，填写 `expected_inputs vs actual_inputs` 记录
- 只有 `allow_enter_3b = yes` 的规则才允许写入 workflow SKILL.md
- 验证记录可追溯，不靠口头判断

**价值**：防止把未经验证的路由规则变成 AI workflow 的硬依赖，避免"上下文系统反而降低编码质量"的失效模式。这是**质量保证机制**，不是功能。

---

### 优势 3：置信度感知消费（Observed / Inferred / Unknown）— 稀有设计

**业界现状**：所有工具的 context 对消费方来说是"一个大袋子"，没有区分哪些是确定性事实、哪些是推断。下游 AI 会以同等置信度对待所有 context。

**spec-graph-bootstrap 的设计**：
- 每条事实带 `confidence` 和 `inference_reason`
- 消费契约明确：`Observed` 可直接依赖，`Inferred` 是弱信号，`Unknown` 不得作为强事实

**价值**：AI workflow 知道"哪些结论可以直接信，哪些只是参考"。这减少了因为弱推断被当成强事实而产生的错误。

---

### 优势 4：知识腐烂显性化（fingerprints.json staleness 契约）

**业界现状**：Cursor、Cody 等工具的索引是否过期，用户和下游 workflow 通常不感知。"最近一次生成"默认被当作"仍然可信"。

**spec-graph-bootstrap 的设计**：
- `fingerprints.json` 记录输入文件 SHA + 分析器版本 + 产物依赖关系
- Phase 0 强制检测 staleness 并输出 ⚠️ 警告
- `README.md` Freshness 章节对用户可见
- 降级事件必须对后续节点可见，不得静默

**价值**：下游 AI workflow 知道当前上下文是新鲜的还是过期的，从而决定是否需要先触发刷新。

---

### 优势 5：任务型上下文包（context-packs）— 超越知识文档

**业界现状**：Codified Context 论文提到了类似"冷知识文档库"的概念，但是静态的。大多数工具的上下文是"项目说明文档"，不是"执行前检查清单"。

**spec-graph-bootstrap 的设计**（`context-packs/review-change.md` 结构）：
```
1. Purpose（任务场景）
2. When to use（使用条件）
3. Read first（先读什么）
4. Key modules / directories
5. Related tests
6. Common pitfalls
7. Avoid overreach（不要过度修改的边界）
8. Escalation hints（什么时候停下来）
```

**价值**：不是"这个项目是什么"，而是"执行这类任务时应该怎么做"。行动导向 > 知识导向。

---

## 三、核心劣势（必须正视）

### 劣势 1：安装摩擦高——不是"开箱即用"

**业界标准**：Cursor 打开项目就自动索引，用户零配置。Copilot 插件安装完就可用。

**spec-graph-bootstrap 的现实**：
- 需要手动运行 `spec-first init`
- 需要 CRG 已安装且已构建图（2-10 分钟）
- 需要 Serena MCP 已配置
- v1 只有 Full/Enhanced/Basic 三种模式，不是"打开就工作"

**影响**：新项目的冷启动成本高，工具普及阻力大。

---

### 劣势 2：没有向量语义检索能力

**业界标准**：Cursor、Cody、Augment Code 都有语义搜索（"找所有处理用户认证的地方"）。

**spec-graph-bootstrap 的现实**：
- 事实抽取依赖预定义的 extractor（入口点、模块、测试等）
- 没有"给我找相关代码片段"的语义检索能力
- Tier 2 的 Serena MCP 提供部分语义能力，但不是 embedding-based

**影响**：对于"发现型"任务（找相关代码、找相似实现）能力弱；主要强项在"结构型"事实（入口在哪、影响半径多大）。

---

### 劣势 3：仍是 spec，尚未实现——没有经过真实项目验证

**业界标准**：Aider、Cursor 已经过数百万用户验证，有公开的 benchmark 数据。

**spec-graph-bootstrap 的现实**：
- 目前是需求文档阶段，核心产物（SKILL.md 实现、context-packs 生成逻辑）尚未存在
- 消费路由（injection-index.yaml）是否真的对 plan/work/review 有帮助，还需要 3A 验证才能回答
- Token 效率提升是假设，不是实测数据

**影响**：所有优势目前是设计优势，不是实测优势。

---

### 劣势 4：没有 IDE 原生集成

**业界标准**：Cursor/Copilot/Continue 在编辑器里实时感知用户行为（光标位置、当前文件、最近修改）。

**spec-graph-bootstrap 的现实**：
- 是 CLI skill，不是 IDE 插件
- 上下文注入靠 SKILL.md 规则驱动，不是实时感知
- task type 识别依赖 workflow 手动判定，v1 默认 `unknown`

**影响**：对于需要实时上下文感知的场景（实时补全、边写边查）不适用。这不是 spec-graph-bootstrap 的目标场景，但用户可能期待。

---

### 劣势 5：阶段 3（消费质量）仍是最大风险

上一节分析过，整套系统的最终效果取决于 context-packs 里的内容质量和 workflow 的 prompt 设计。这是目前 spec 定义最模糊的地方——说"先读什么"，但没说"读完之后 AI 怎么利用这些信息做出更好的决策"。

---

## 四、竞争力地图

```
                          上下文路由与消费设计
                                   ↑
                                   │
              spec-graph-bootstrap ●  ←── 这里是真正的差异化空间
                                   │
                                   │         Codified Context（论文）
                                   │
                ─────────────────────────────────────────────→
           纯索引                                    索引 + 路由 + 消费
                                   │
                   Graphify ●       │
                 CodePrism ●       │
               code-review-graph ● │
                        Aider ●    │
                               Cursor ●  Cody ●
                                   │
                                   ↓
                             工具建设完善度
```

**核心洞察**：
- Cursor/Cody 在"索引"这个轴上远超所有人
- spec-graph-bootstrap 的差异化在"路由与消费"这个轴上
- **这两个轴是不同的战场，不是同一场竞争**

---

## 五、真正的竞争力在哪里

不是与 Cursor 竞争"谁的索引更快"，而是解决了一个 Cursor 根本不打算解决的问题：

> **Cursor 告诉 AI 去哪里找代码，spec-graph-bootstrap 告诉 AI 在执行不同任务时该加载哪些知识、以什么置信度信任它们、消费后如何验证效果有没有改善。**

这是两个层次：
- **工具层**（Cursor/Cody）：上下文检索基础设施
- **工作流层**（spec-graph-bootstrap）：上下文编排与消费治理

类比：Cursor 是"图书馆"，spec-graph-bootstrap 是"图书管理员 + 阅读指南 + 效果评估体系"。

---

## 六、建议的竞争策略

### 短期：聚焦已有差异化，不要补短

不需要做语义 embedding（那是 Cursor 的战场）。聚焦：
1. 把 `injection-index.yaml` 做成业界第一个"任务路由即代码"的可执行规范
2. 把 3A 验证记录模板变成可分享的"消费质量评估标准"
3. 把 context-packs 的 8 段式结构确立为任务上下文包的行业标准格式

### 中期：拥抱 MCP 生态，而不是独立建楼

- 把 CRG 的 22 个工具 + spec-graph-bootstrap 的路由层封装成标准 MCP server
- 让任何 MCP 兼容的 workflow（Continue.dev、OpenHands、Codex）都能直接消费
- 这样 spec-graph-bootstrap 变成**通用 AI workflow 的上下文编排中间层**，而不是只服务 spec-first 内部

### 长期：消费质量指标（必须补上的缺口）

目前最大的战略盲点是：没有量化指标证明"注入了 context-packs 之后，AI 编码质量提升了多少"。

业界同样缺这个——这是先发者的机会：
- 设计 review 质量 A/B 对比实验（有 context-pack vs. 没有）
- 用 3A 验证记录积累真实消费效果数据
- 把数据变成方案可信度的核心证据

---

## 七、一句话竞争力定位

> 业界已经解决了"如何建索引"，没有人解决"如何把正确的上下文，按任务类型，以正确的置信度，路由给正确的 AI workflow，并验证效果"。这是 spec-graph-bootstrap 的真实竞争力所在，也是它必须兑现的承诺。

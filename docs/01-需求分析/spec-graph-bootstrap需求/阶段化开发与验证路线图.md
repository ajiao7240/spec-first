## 1. 路线图目标

本路线图的目标是把 `spec-graph-bootstrap` 从需求定义推进到可落地实施，并保证每个阶段都有：

* 明确的开发边界
* 明确的验证出口
* 明确的进入下一阶段 gate
* 明确的主要风险与止损点

本路线图遵循以下总原则：

* 第一版保留 `/spec:graph-bootstrap`，新建 `spec-graph-bootstrap` 并行验证
* 第一版复用现有 `docs/contexts/<slug>/` 与 `.spec-first/workflows/bootstrap/<slug>/` 路径
* 不把安装、生成、消费、刷新混在同一个阶段里做
* 旧 `.context/spec-first/bootstrap/<slug>/` 仅视为历史迁移阶段口径，不再作为当前实现验收标准
* 阶段 2 内部分成 `2A 事实层` 与 `2B 文档/路由生成` 两个执行批次
* 阶段 3 内部分成 `3A 非侵入式消费验证` 与 `3B skill 消费固化` 两个执行批次

## 2. 总体推进顺序

1. 阶段 1：skill 安装集成
2. 阶段 2A：事实层
3. 阶段 2B：文档与路由生成
4. 阶段 3A：非侵入式消费验证
5. 阶段 3B：skill 消费固化
6. 阶段 4：相关文档更新与刷新

说明：

* 这里的“阶段”是实施推进阶段，不等同于 skill 内部执行时的 `Phase 0` 到 `Phase 4`
* 阶段 2A 与 2B 属于同一个需求阶段的两个实施批次，目的是控制风险，而不是改变需求结构
* 阶段 3A 与 3B 属于同一个需求阶段的两个实施批次，目的是先验证消费规则，再最小化固化到 workflow skill

## 3. 阶段 1：skill 安装集成

### 3.1 目标

把 `spec-graph-bootstrap` 安全接入当前 `spec-first` 系统，不影响现有 `spec-graph-bootstrap`。

### 3.2 开发任务

* 新增 `skills/spec-graph-bootstrap/`
* 新增 `templates/claude/commands/spec/graph-bootstrap.md`
* 更新 `.claude-plugin/plugin.json`，新增 `graph-bootstrap` command
* 确保 `spec-first init --claude` 和 `spec-first init --codex` 都能同步新 skill 与新 command
* 更新 README / 用户手册中的双入口并行说明

### 3.3 验证任务

* 执行 `spec-first init --claude`
* 执行 `spec-first init --codex`
* 检查新旧 runtime 资产同时存在
* Claude 宿主重启后完成新入口最小调用
* Codex 宿主验证新 skill 已被发现；若保留命令兼容层，再额外验证兼容入口可调用
* 验证 `doctor` 或等价检查能看到新增 managed assets

### 3.4 退出条件

* 新旧入口并行存在且不互相覆盖
* `init` 重复执行保持幂等
* Claude 侧新入口可调用
* Codex 侧新 skill 已被宿主发现；兼容命令入口仅作为附加兼容性验证

### 3.5 风险

* plugin manifest 与 command 模板不同步，导致宿主无法发现新入口
* `init` 的 managed asset 清理逻辑误删新入口
* README 过早写成“已切换到新入口”，与当前双轨策略冲突

## 4. 阶段 2A：事实层

### 4.1 目标

先把 `Fact Inventory` 及控制面产物做实，确保后续渲染层有稳定输入。

### 4.2 开发任务

* 实现 readiness / mode / slug / rerun backup 基础逻辑
* 定义事实模型与置信度规则
* 生成：
  * `fact-inventory.json`
  * `risk-signals.json`
  * `test-surface.json`
  * `artifact-manifest.json` 初版结构
* 固定 `Observed / Inferred / Unknown` 值域与最小字段集合
* 当前实现的控制面根目录统一为 `.spec-first/workflows/bootstrap/<slug>/`

### 4.3 验证任务

* 在至少 2 个真实项目上跑出控制面产物
* 检查 `Inferred` 事实是否包含 `inference_reason`
* 检查 `artifact-manifest.json` 顶层结构是否至少包含 `schema_version`、`generated_at`、`updated_at`、`status`、`inputs`
* rerun 时确认 slug 可复用
* 若已生成文档产物，检查 `artifact-manifest.json` 第二次写入后包含 `outputs`

### 4.4 退出条件

* 控制面产物可稳定生成
* facts-first 合同成立
* 渲染器后续无需默认回扫源码

### 4.5 风险

* 把事实层做成 narrative 摘要，失去结构化价值
* `artifact-manifest.json` 首版结构过弱，后续无法平滑扩展
* `Inferred` 过多但没有证据说明，后续消费会放大误判
* 控制面路径与当前 `.spec-first/workflows/bootstrap/<slug>/` 契约不一致，导致后续文档与验证脚本继续漂移

## 5. 阶段 2B：文档与路由生成

### 5.1 目标

基于阶段 2A 的事实层打通 v1 最小闭环产物。

### 5.2 开发任务

* 生成 `README.md`
* 生成 `00-summary.md`
* 生成 `architecture/module-map.md`
* 生成 `pitfalls/index.md`
* 生成：
  * `code-facts/public-entrypoints.md`
  * `code-facts/test-map.md`
  * `code-facts/high-risk-modules.md`
* 生成 `context-packs/review-change.md`
* 生成 `injection-index.yaml`

### 5.3 验证任务

* 最小闭环产物全部生成且非空
* `README.md` 能作为上下文控制台使用
* `injection-index.yaml` 是合法 YAML
* `unknown` task type 存在正式回退集合

### 5.4 退出条件

* v1 最小闭环成立
* 路由规则可人工消费
* 不依赖外部 graph 也能稳定生成

### 5.5 风险

* 文档职责边界不清，造成 `README.md`、`00-summary.md`、`pitfalls/index.md` 重复堆叠
* `selection_rules` 写成建议而不是可判定规则
* 过早追求完整产物树，拖慢最小闭环

## 6. 阶段 3A：非侵入式消费验证

### 6.1 目标

先以非侵入方式验证 Stage-0 产物能被 `plan / work / review` 实际利用，而不是停留在“理论可读”。

### 6.2 开发任务

* 在阶段 3 文档中定义项目级治理说明草案与 fallback 规则
* 定义 `spec-plan` 的人工消费契约
* 定义 `spec-work` 的人工消费契约
* 定义 `spec-code-review` 的人工消费契约
* 固定 v1 消费顺序：`always[] -> stages.<stage>[] -> selection_rules(output_exists.*) -> advice.<stage>`
* 固定 `output_exists.*` 的实时求值方式
* 明确 `fact.*` 规则在 v1 显式跳过
* 固定“缺失即降级，不得阻断”规则

### 6.3 验证任务

* Plan 实验：
  * 命中 `architecture/module-map.md`
  * 命中 `code-facts/public-entrypoints.md`
* Work 实验：
  * 命中 `code-facts/test-map.md`
  * 命中专用 task pack，或回退到固定最小集合 / fallback 集合
* Review 实验：
  * 命中 `code-facts/high-risk-modules.md`
  * 命中 `context-packs/review-change.md`
* 降级实验：
  * `injection-index.yaml` 缺失时继续运行
  * `docs/contexts/<slug>/` 缺失时继续运行
* 记录实验：
  * 按统一验证记录模板填写 `expected_inputs / actual_inputs / fallback_reason / verdict / allow_enter_3b`

### 6.4 退出条件

* 已证明人工消费链成立
* `unknown` 与缺失产物场景下仍能继续运行
* 已形成允许进入 `3B` 的最小固化规则集
* 准入结论可追溯到统一验证记录，而不是口头判断
* 至少 2 类真实任务验证通过

### 6.5 风险

* 项目级治理说明草案过长、过散，导致指引不稳定
* task type 识别契约不清，消费逻辑无法稳定落地
* 产物路径或文件名在消费前继续变化，导致接入反复返工

## 7. 阶段 3B：skill 消费固化

### 7.1 目标

把 `3A` 验证通过的最小消费规则固化到 `spec-plan / spec-work / spec-code-review` 的 `SKILL.md`，形成稳定消费契约。

### 7.2 开发任务

* 修改 `skills/spec-plan/SKILL.md`
* 修改 `skills/spec-work/SKILL.md`
* 修改 `skills/spec-code-review/SKILL.md`
* 为三个 workflow 写入最小消费顺序与 fallback 规则
* 保持 command 层不变

### 7.3 验证任务

* 三个 workflow 都能优先消费 Stage-0 产物
* Stage-0 产物缺失时，三个 workflow 都能自动降级
* 三个 workflow 的消费顺序与 `3A` 人工约定一致

### 7.4 退出条件

* 不依赖人工提醒也能稳定消费
* 缺失产物场景下仍不阻断主任务
* skill 内契约与阶段 3 文档一致

### 7.5 风险

* 过早把未验证规则写死进 skill，导致后续返工
* skill 内 fallback 规则不一致，造成三条链路行为漂移
* 把 Stage-0 产物消费写成硬依赖，反向阻断主任务

## 8. 阶段 4：相关文档更新与刷新

### 8.1 目标

把 `spec-graph-bootstrap` 从一次性生成器推进到可长期维护的 Stage-0。

### 8.2 开发任务

* 完善 `artifact-manifest.json` 依赖模型
* 明确局部失效与全量失效判定
* 实现 rerun refresh 行为
* 输出 freshness / confidence / analyzer version
* 对外报告本次刷新类型、重算产物和跳过原因

### 8.3 验证任务

* 入口文件变化，只重算入口相关产物
* 测试变化，只重算测试相关产物
* 重大结构变化，升级为全量刷新
* rerun 失败时按策略恢复

### 8.4 退出条件

* 增量刷新契约成立
* freshness 与 degradation 对用户可见
* 不破坏阶段 3 已定义的消费路径

### 8.5 风险

* 刷新规则靠人工判断，无法长期维护
* `fingerprints.json` schema 扩展破坏阶段 2 初版兼容性
* 局部刷新后产物 freshness 状态不透明

## 9. 阶段 gate

### 9.1 阶段 1 -> 阶段 2A

必须满足：

* 新入口已被宿主识别
* `init` 幂等成立
* 新旧入口并行不冲突

### 9.2 阶段 2A -> 阶段 2B

必须满足：

* 事实层字段和值域稳定
* `fingerprints.json` 初版结构固定
* facts-first 合同已经明确

### 9.3 阶段 2B -> 阶段 3A

必须满足：

* v1 最小闭环产物路径固定
* `injection-index.yaml` 合法且可人工消费
* `README.md` 已能作为控制台使用

### 9.4 阶段 3A -> 阶段 3B

必须满足：

* 已完成至少 2 类真实任务的人工消费验证
* 已形成最小固化规则集
* 降级规则已验证不会阻断主任务
* 已按统一验证记录模板沉淀证据，且 `allow_enter_3b` 结论明确

### 9.5 阶段 3B -> 阶段 4

必须满足：

* 消费路径固定
* task type 最低识别契约固定
* fallback 规则固定

### 9.6 阶段 4 -> 删除旧入口评估

必须满足：

* rerun / refresh 稳定
* 至少 2 个真实项目验证通过
* `plan / work / review` 质量指标没有明显恶化
* 文档与测试覆盖足以支撑切换

## 10. 推荐排期方式

建议按以下方式排期，而不是把阶段 2 作为一个大包一次性交付：

1. 迭代 1：阶段 1
2. 迭代 2：阶段 2A
3. 迭代 3：阶段 2B
4. 迭代 4：阶段 3A
5. 迭代 5：阶段 3B
6. 迭代 6：阶段 4

原因：

* 阶段 2 是最大不确定性来源
* 事实层未稳定前，不应该并行推进消费和刷新
* 文档与路由一旦进入消费阶段，文件名和路径应视为冻结

## 11. 实施建议

* 先写阶段 1 的实现计划，不要一开始直接进入阶段 2
* 阶段 2A 完成后，先做一次字段和路径冻结评审
* 阶段 3A 先做手动消费验证，再进入阶段 3B 的 skill 固化
* 阶段 4 只在阶段 3B 路径与消费契约稳定后启动

## 12. 成功定义

这条路线图成功，不是因为写出了更多文档，而是因为做到了：

* 新 skill 能并行接入当前系统
* facts-first 闭环成立
* 后续 workflow 能消费这些产物
* rerun 与 refresh 变成系统能力
* 删除旧 `/spec:graph-bootstrap` 之前，已经有足够证据证明新链路更稳

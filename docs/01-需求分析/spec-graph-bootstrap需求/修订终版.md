# `spec-graph-bootstrap` Skill 最终技术方案

## 1. 文档目标

本文档是 `spec-graph-bootstrap` skill 的最终技术方案。

该 skill 的定位是：**替代现有 `spec-bootstrap` 的下一代 Stage-0 实现**，为后续 `brainstorm / plan / work / review / compound` 五阶段提供更高质量、更低 token、可路由、可维护的上下文资产。

现有 `spec-bootstrap` 已具备以下基础能力：

* 作为 Stage-0 supporting workflow
* 生成 `docs/contexts/<slug>/` 下的上下文资产
* 具备 Host Readiness Gate
* 具备 `Full / Enhanced / Basic` 三种分析模式
* 通过 `.context/spec-first/bootstrap/<slug>/tasks/<task-id>/prd.md` 管理任务控制面

但当前范围仍然以生成 context assets 为主，自动注入后续五阶段仍属于未来能力。

`spec-graph-bootstrap` 的目标不是简单复制现有能力，而是在保持 Stage-0 外部职责连续的前提下，借鉴两类能力：

* 从 `code-review-graph` 借鉴：**AST、依赖/调用/测试关系、影响面思维、增量更新思维**
* 从 `graphify` 借鉴：**一页式摘要入口、持久化知识图文件、缓存与 update/watch 思路、对事实与推断的谨慎表达**

## 2. 方案结论

### 2.1 目录策略：软替换

保留两个目录，采用“软替换”方式演进：

```text
skills/spec-bootstrap/         # 冻结，保持现状，不再增强
skills/spec-graph-bootstrap/   # 所有新能力都在这里演进
```

### 2.2 开发约定

* `spec-bootstrap` 不做任何处理，维持现状
* 所有 bootstrap 新功能都只补充到 `spec-graph-bootstrap`
* `spec-graph-bootstrap` 稳定并验证后，删除 `skills/spec-bootstrap/`
* 最终仅保留 `skills/spec-graph-bootstrap/`

### 2.3 职责定位

`spec-graph-bootstrap` 是一个 **graph-informed 的增强版 Stage-0 skill**，其职责是：

1. 从项目代码、配置、测试、schema 中抽取结构化事实
2. 生成项目级上下文资产
3. 生成代码事实层资产
4. 生成任务型上下文包
5. 生成上下文注入索引
6. 支持后续节点按任务类型和阶段进行最小必要上下文注入

### 2.4 命令入口策略

先区分两层：

* 产品层统一语义：保留 `bootstrap` 与 `graph-bootstrap` 这两个 workflow 名称
* 平台层具体落地：由 Claude / Codex adapter 按各自宿主能力做不同实现

开发与验证阶段，在产品层保持双入口并行语义：

* 旧入口：`/spec:bootstrap`
* 新入口：`/spec:graph-bootstrap`

替换完成后：

* 删除对外 `/spec:bootstrap`
* 对外统一保留 `/spec:graph-bootstrap` 作为长期正式入口
* `spec-graph-bootstrap` 成为唯一实现

这样可以避免长期维护双命名兼容层，并让命令、技能、测试、文档统一收敛到同一入口。

### 2.5 平台差异与适配边界

`spec-graph-bootstrap` 是统一产品能力，不等于 Claude 与 Codex 在宿主侧有完全相同的交互模型。需求文档必须明确区分：

* 哪些是统一产品契约
* 哪些是平台实现契约
* 哪些差异由 adapter 层吸收

统一产品契约：

* 产物路径统一为 `docs/contexts/<slug>/` 与 `.context/spec-first/bootstrap/<slug>/`
* workflow 名称统一为 `spec-bootstrap` 与 `spec-graph-bootstrap`
* facts-first、降级不阻断、先验证再固化等原则在两个平台保持一致

Claude 平台实现特征：

* 以 `CLAUDE.md` 作为项目级指令文件
* 以 `.claude/commands/spec/*.md` 承载显式 slash command
* 以 `.claude/skills/`、`.claude/agents/` 承载 runtime 资产

Codex 平台实现特征：

* 以 `AGENTS.md` 作为项目级指令文件
* 以 `.agents/skills/` 作为 skill 主发现目录
* `.codex/commands/spec/` 是 `spec-first` 当前提供的兼容命令层，不应在需求文档中误表述为 Codex 平台的天然前提
* v1 对 Codex 的正式可用性承诺，以 skill 已安装到 `.agents/skills/` 且可被宿主发现为准；兼容命令入口仅用于 `spec-first` 兼容性验证

适配边界：

* 需求文档描述产品能力时，应优先使用 workflow 名称与产物契约，不应把某个平台的命令语法当作统一事实
* 需求文档描述安装与运行时布局时，应显式按 Claude / Codex 分开写
* adapter 层负责把统一产品契约映射到不同宿主的目录、命令与指令文件模型
* 任何仅由 `spec-first` 兼容层提供的能力，都必须在文档中标注“兼容层”，不能写成平台天然能力

## 3. 背景问题

现有 `spec-bootstrap` 更偏“项目上下文文档生成器”。如果终局目标是**提升 AI 编码质量**，还存在以下缺口：

### 3.1 缺少代码事实层

当前更偏 narrative 文档，缺少：

* 真实入口点
* 核心依赖方向
* 测试映射
* 高风险模块

### 3.2 缺少任务型上下文包

当前产物更像“知识文档”，而不是：

* 新增接口要先看什么
* 修 bug 要先看什么
* 改 shared 模块要先看什么
* review 变化要先看什么

### 3.3 缺少注入路由

当前 bootstrap 以生成 context assets 为主，自动注入后续五阶段仍不是已完成能力。需要一个机器可读的注入路由文件，明确不同阶段和任务类型应该先加载哪些产物。

### 3.4 缺少增量与缓存思维

每次 bootstrap 都全量扫仓库、全量重写所有文档，会带来：

* token 浪费
* 时间成本高
* 同一窗口多节点上下文重复加载
* 文档维护成本高

## 4. 设计目标

### 4.1 业务目标

* 降低后续节点冷启动成本
* 降低盲目全仓扫描
* 提高实现、修改、review 的正确性
* 帮助后续节点选择更小、更准的上下文
* 支持长期维护与增量刷新
* 支持未来知识仓库集中维护

### 4.2 技术目标

* 保持与现有 bootstrap 的目录与控制面兼容
* 新增代码事实层
* 新增任务型上下文层
* 新增注入索引层
* 支持置信标记
* 支持增量刷新
* 允许可选增强地消费外部 graph 结果

---

## 5. 非目标

本 skill 不负责：

* 替代运行时 diff review 引擎
* 完整模拟 `code-review-graph` 的所有 MCP 行为
* 完整模拟 `graphify` 的全部多模态功能
* 强依赖外部图服务才能运行
* 自动改写后续五阶段实现逻辑

---

## 6. 设计原则

### 6.1 Facts First

先抽事实，再做摘要和任务包。不允许先凭空写架构，再去找代码佐证。

### 6.2 Graph-Informed, Not Graph-Dependent

借 graph 的结构化思维，不强依赖外部图系统才能运行。默认模式下自给自足完成基础事实抽取；检测到外部图产物时再增强。

### 6.3 Injection-Oriented

输出必须服务后续节点注入，而不是只追求“文档完备”。

### 6.4 Honest Confidence

统一区分：

* `Observed` — 有代码证据直接支持
* `Inferred` — 基于模式推断，无直接证据
* `Unknown` — 无法判断

### 6.5 Incremental-Friendly

首次按当前版本要求全量生成目标产物，后续按变化刷新。

### 6.6 Compatible-by-Path

产物路径与控制面路径尽量延续现有 bootstrap 约定，以降低迁移成本。

## v1 正式回填附录

本附录只做增量补充，不删除前文内容，不重写前文结构。它的作用是把 v1 方案正式落到当前文档里，作为后续评审和实现的规范补强。

### A.1 回填原则

- 保留前文全部内容，不做删减
- 新增内容以 v1 正式规范口径补充
- 旧内容与 v1 不冲突的部分保留原样
- 与 v1 更强约束冲突的部分，以本附录为准

### A.2 阅读导航

正式评审时，建议按以下顺序阅读：

1. 文档目标
2. 方案结论
3. 设计原则
4. 输出产物结构
5. Phase 1 到 Phase 4
6. 增量刷新与外部适配

这个顺序对应的是“知识层控制台”的阅读方式，而不是传统说明书的阅读方式。

### A.3 事实层收紧

`Fact Inventory` 必须被视为唯一事实源。后续产物必须优先消费事实层，不允许默认回扫源码补洞。

事实层建议统一输出以下字段：

- `path`
- `symbol`
- `kind`
- `summary`
- `confidence`
- `inference_reason`
- `evidence`
- `updated_at`

其中 `confidence` 必须显式区分：

- `Observed`
- `Inferred`
- `Unknown`

### A.4 摘要层边界

`README.md` 的角色是上下文控制台，负责告诉后续节点先读什么、当前 freshness 如何、可用的 task pack 有哪些。

`00-summary.md` 的角色是最小事实摘要，只回答：

- 这是什么项目
- 主技术栈是什么
- 从哪里进入
- 核心模块有哪些
- 高风险区域有哪些
- 建议先读什么

摘要层之间不应重复堆叠同一批事实。

### A.5 产物结构边界

v1 只要求先打通最小闭环，不要求一次性展开全部终局产物。

建议 v1 的最小闭环产物包括：

- `README.md`
- `00-summary.md`
- `architecture/module-map.md`
- `pitfalls/index.md`
- `code-facts/public-entrypoints.md`
- `code-facts/test-map.md`
- `code-facts/high-risk-modules.md`
- `context-packs/review-change.md`
- `injection-index.yaml`

其余内容保留为增强项或条件项，按事实命中和后续版本逐步补齐。

### A.6 Task Pack 的正式要求

每个 `context-packs/*.md` 应统一使用以下结构：

1. Purpose
2. When to use
3. Read first
4. Key modules / directories
5. Related tests
6. Common pitfalls
7. Avoid overreach
8. Escalation hints

Task Pack 的目标不是展开百科式说明，而是让执行者快速知道先读什么、重点看什么、什么情况要停下来升级。

### A.7 Injection Routing 的正式要求

`injection-index.yaml` 必须是可判定的路由规则，而不是纯推荐列表。

它至少应表达：

- `stage`
- `task_type`
- `fact.*.present`
- `fact.*.confidence`
- `output_exists.*`

默认最小集合建议为：

- `00-summary.md`
- `pitfalls/index.md`
- `code-facts/public-entrypoints.md`
- `code-facts/test-map.md`

当 task type 识别失败时，应回退到最小集合，而不是阻断 workflow。

### A.8 Freshness 与 confidence

所有正式产物都应尽量展示：

- `updated_at`
- `analyzer_version`
- `schema_version`
- `source_snapshot`
- `confidence`

目的不是增加噪音，而是让后续节点知道哪些结论可直接信、哪些只是推断、哪些已经过期。

### A.9 增量刷新正式要求

`fingerprints.json` 不只是文件 SHA，而是输入、分析器版本、schema、产物依赖关系的统一索引。

建议至少支持以下失效映射：

- 入口文件变化 -> `public-entrypoints.md`
- 测试变化 -> `test-map.md`
- shared/core 变化 -> `high-risk-modules.md`
  - 若 `core-dependencies.md` 已生成，则一并刷新；未生成则对该目标执行 `skip / no-op`
- 路由/API 变化 -> `add-endpoint.md`、`review-change.md`
  - 若 `add-endpoint.md` 等可选产物当前未生成，则对应刷新目标直接 `skip / no-op`

### A.10 降级与容错

正式方案必须显式区分：

- `block`
- `degrade`
- `skip`

要求是：

- 宿主前置条件失败时才 `block`
- 能继续但能力受限时 `degrade`
- 某类产物不命中时 `skip`

任何降级都不得静默失败，必须对用户与后续节点可见。

### A.11 v1 验收标准

v1 通过的标准应至少包括：

- 后续节点能快速知道先读什么
- 事实层和摘要层边界清楚
- `injection-index.yaml` 可用于手动消费
- 所有推断都有证据或明确标记
- 增量刷新可以局部失效
- 降级场景不会阻断主流程
- 文档内容不再重复堆叠
- 文档能体现 freshness 与 confidence
- 任务包能直接指导执行，而不是只作说明

### A.12 结语

这份正式版不是为了写得更长，而是为了让 `spec-graph-bootstrap` 真正变成一个可被后续节点消费、可增量维护、可复核审计的知识层系统。

## 7. 总体架构

### 7.1 架构概览

```text
Target Repository
  -> Phase 0: Host Readiness Gate
  -> Phase 1: Repository Analysis + Fact Inventory Extraction
  -> Phase 2: Context Task Planning + PRD Generation
  -> Phase 3: Context Asset Rendering
  -> Phase 4: Injection Routing Generation
  -> docs/contexts/<slug>/ 产物树
```

### 7.2 三层模型

#### 第一层：事实层

输入：

* 代码
* 配置
* 测试
* schema / migration
* 启动入口
* 路由 / worker / CLI 入口

输出（控制面）：

* `fact-inventory.json`
* `risk-signals.json`
* `test-surface.json`
* `fingerprints.json`

#### 第二层：摘要层

基于事实层生成：

* `00-summary.md`
* `architecture/*`
* `pitfalls/index.md`
* `code-facts/*`
* `README.md`

#### 第三层：注入层

基于摘要层和事实层生成：

* `context-packs/*`
* `injection-index.yaml`

## 9. 输入 / 输出定义

### 9.1 输入

* 项目代码仓库
* 项目配置
* 测试目录
* schema / migration
* 可选 DB 访问能力
* 可选外部图分析产物

### 9.2 输出根目录

继续使用：

```text
docs/contexts/<slug>/
```

### 9.3 控制面根目录

继续使用：

```text
.context/spec-first/bootstrap/<slug>/
```

不单独新起 `graph-bootstrap` 控制面目录，以降低迁移成本。

### 9.4 Slug Resolution Rules

`<slug>` 是项目级稳定 namespace，同时用于产物树与控制面路径。

生成优先级：

1. 用户显式传入 slug
2. 仓库根目录名
3. 项目元信息中的稳定名称（如 `package.json.name` / repo name）
4. 若仍冲突，则附加短 hash

规范要求：

* slug 仅允许小写字母、数字与 `-`
* 空格、下划线统一转换为 `-`
* `docs/contexts/<slug>/` 与 `.context/spec-first/bootstrap/<slug>/` 必须使用相同 slug

复用与冲突策略：

* 同一项目默认复用同一 slug
* 默认行为是复用与增量刷新，而不是每次创建新 slug
* 若同名 slug 已存在但指向不同项目，必须显式报冲突，不允许 silent overwrite
* 若用户明确需要新的上下文版本，可使用 `<slug>-v2` 或 `<slug>-<date>`

## 11. Phase 0：Host Readiness Gate

沿用现有 bootstrap 的 readiness 思路与阻断逻辑。

### 11.1 目标

* 验证当前运行位置正确
* 验证宿主环境
* 验证必要工具可用性
* 验证可选工具探测状态
* 如失败则停止

### 11.2 扩展探测字段

在 readiness 结果中增加：

```yaml
graph_support_state: local-only | external-available | unavailable
code_facts_confidence: high | medium | low
db_support_state: connected | inferred | unavailable
```

### 11.3 输出示例

```text
[Graph Bootstrap] Analysis mode: Enhanced | Code Facts: high | DB: inferred | Graph Support: local-only
```

### 11.4 State-to-Action 决策表

| Signal | Value | Action | Effect on Output | User-facing Note |
| --- | --- | --- | --- | --- |
| `graph_support_state` | `external-available` | `continue` | 尝试消费外部 graph 结果并增强事实层/README | 记录已启用外部增强 |
| `graph_support_state` | `local-only` | `continue` | 仅使用本地 Fact Inventory 生成全部核心产物 | 提示外部增强未启用 |
| `graph_support_state` | `unavailable` | `degrade` | 跳过外部增强探测，仅保留本地抽取链路 | 提示 graph 支持缺失但不阻断 |
| `code_facts_confidence` | `high` | `continue` | 正常生成 `code-facts/*` 与 `context-packs/*` | 无额外提示 |
| `code_facts_confidence` | `medium` | `continue` | 正常生成产物，但高风险结论需优先引用证据 | 提示部分结论含推断 |
| `code_facts_confidence` | `low` | `degrade` | 仍生成最小闭环产物，但 `code-facts/*` 与 task pack 必须显式标记“仅供参考” | 提示建议人工复核关键模块 |
| `db_support_state` | `connected` | `continue` | 允许生成 `database/*` 的已验证产物 | 标注 DB 为 verified |
| `db_support_state` | `inferred` | `degrade` | 可生成 `database/*`，但必须标记为 `Inferred` | 提示未直连 DB |
| `db_support_state` | `unavailable` | `skip` | 跳过 `database/*`，不影响 summary / architecture / code-facts / injection | 提示 DB 上下文未生成 |

阻断规则：

* 宿主未完成 `mcp-setup`、宿主未重启导致 MCP runtime 不可调用、或目标仓库路径无效时，动作统一为 `block`
* `block` 仅用于宿主级前置条件失败，不用于 feature-level 能力缺失

### 11.5 Mode-to-Capability Mapping

为保持与现有 `spec-bootstrap` 的连续性，保留 `Full / Enhanced / Basic` 三种分析模式，并映射到新架构：

| Mode | 能力边界 |
| --- | --- |
| `Basic` | 仅启用 `project_identity`、`entrypoints`、`module_structure`、基础 `test_surface`；生成固定任务与最小 `code-facts` / `injection-index`；不生成数据库产物，不启用外部增强 |
| `Enhanced` | 启用全部核心 analyzer；条件任务按触发规则执行；数据库文档允许 `inferred` 版；外部增强可选；作为默认推荐模式 |
| `Full` | 启用 `Enhanced` 全部能力，并增加 DB connected 分析、外部 graph 结果增强、完整依赖指纹与全部条件任务 |

补充规则：

* `Basic` 与 `Enhanced` 必须在没有外部 graph 结果时独立运行
* `Full` 是增强能力全部可用的模式，不应成为系统能否运行的前提

## 12. Phase 1：Repository Analysis + Fact Inventory

这是新 skill 的核心改造点。

### 12.1 目标

在生成任何文档前，先形成一个统一的中间事实集。

### 12.2 为什么要这样

`code-review-graph` 的关键价值，在于它先构建结构化代码图，再在图上做 blast radius、dependency chains、test coverage gaps 等查询，而不是每个问题重新扫仓库。

`spec-graph-bootstrap` 不一定完整复制这套实现，但必须借鉴“**先有事实层，再有文档层**”的架构。

### 12.3 Fact Inventory 模型

```yaml
project_identity:
  name:
  primary_language:
  primary_frameworks: []
  repo_shape:         # monorepo | service | library | cli

entrypoints: []
modules: []
integrations: []
data_shapes: []
testing_surface: []
risk_signals: []
layers: []
database: {}
```

为支持后续路由判定，`Fact Inventory` 中会被 `selection_rules` 读取的分支项，必须采用统一对象结构，而不是裸数组或松散对象。推荐最小结构如下：

```yaml
layers:
  frontend:
    present: true
    confidence: observed
    inference_reason: ""
    evidence: []
  backend:
    present: false
    confidence: unknown
    inference_reason: ""
    evidence: []

database:
  present: true
  confidence: inferred
  inference_reason: "ORM model and migration files co-exist under data layer"
  evidence: []
```

每条事实统一字段：

```yaml
path:
symbol:
kind:
summary:
confidence: observed | inferred | unknown
inference_reason: ""
evidence:
  - file:
  - symbol:
```

### 12.4 子提取器

#### 12.4.1 Project Identity Extractor

识别：项目名、主语言、主框架、仓库形态

数据源：顶层目录、package/go/cargo/java 等元文件、框架标记文件

#### 12.4.2 Entrypoint Extractor

识别：HTTP 入口、CLI 入口、worker/cron/queue 入口、SDK/public export 入口

数据源：`main.*`、`server.*`、`app.*`、router/bootstrap 文件、export/index 文件

#### 12.4.3 Module Structure Extractor

识别：顶层目录职责、service/controller/repository/model 模式、shared/core/common/util 层、模块自然边界

#### 12.4.4 Integration Extractor

识别：DB client、cache、MQ、外部 HTTP/gRPC/SDK、auth/session 边界

#### 12.4.5 Test Surface Extractor

识别：unit/integration/e2e 测试布局、fixture/mock/setup、模块与测试的近邻关系

#### 12.4.6 Risk Signal Extractor

识别：高扇出模块、高复杂度热点、shared/core/common/util 风险点、弱测试区域、全局状态、异常吞掉点、配置中心/基础设施中心

#### 12.4.7 Data Shape Extractor

识别：DTO、schema、entity/model、migration、validation

#### 12.4.8 Confidence Derivation Rules

`Observed / Inferred / Unknown` 必须来自明确规则，而不是自由推断。

* `Observed`：可由 AST、配置、schema、测试、显式注册、显式导出或显式引用直接证明
* `Inferred`：只能来自预定义启发式，例如目录命名模式、文件邻接关系、router/handler 近邻、migration/ORM/schema 共现、测试与模块的路径近邻
* `Unknown`：缺少足够直接证据，也缺少足够启发式支持

对 `Inferred` 的约束：

* 每条 `inferred` 事实必须带 `inference_reason`
* `inferred` 不得单独触发强结论，尤其不能单独触发高风险判断，除非有多个独立信号共同支持
* 当推断器版本变化时，依赖该启发式的结论必须失效重算

对 `selection_rules` 会读取的分支项，还必须满足：

* `present` 表示该分支是否可用于路由决策
* `confidence` 只能使用 `observed / inferred / unknown`
* 当 `present = true` 但 `confidence = unknown` 时，只允许进入降级路由，不得进入高风险强结论分支

### 12.5 控制面输出

```text
.context/spec-first/bootstrap/<slug>/
  fact-inventory.json
  risk-signals.json
  test-surface.json
  fingerprints.json
```

### 12.6 Fact-to-Document Enforcement

为避免重复扫描和推断失控，定义如下执行约束：

* `fact-inventory.json`、`risk-signals.json`、`test-surface.json` 只保存**可追溯事实**与置信度
* `summary / architecture / pitfalls / code-facts / context-packs` 必须优先基于事实层生成
* renderer 默认只能读取事实层与已生成的上游产物，不能直接重新扫源码
* 文档中若出现超出事实层的推断，必须显式标记为 `Inferred`

允许例外：

* 只有在控制面显式声明 `source_read_allowed = true` 时，renderer 才能回退读源码
* 回退必须记录 `fallback_reason` 与 `fallback_scope`
* 若 renderer 输出了事实层中不存在、且没有 fallback 记录支撑的事实，视为违规输出

### 13.4 Conditional Task Trigger Rules

#### `layer-context`

触发依据：`fact_inventory.layers.<layer>.present`

* 当 `confidence = observed` 时，必须生成
* 当 `confidence = inferred` 时，允许生成，但文档中必须显式标记为 `Inferred`
* 当未命中或 `confidence = unknown` 时，直接 `skip`
* 不允许为未命中的 layer 生成空 `index.md`

#### `database-context`

触发依据：`db_support_state` 与 `fact_inventory.database.present`

* 当 `db_support_state = connected` 且数据库证据存在时，必须生成 verified 数据库文档
* 当 `db_support_state = inferred` 且存在明确 ORM/schema/migration 证据时，允许生成 inferred 数据库文档
* 当 `db_support_state = unavailable` 或 `fact_inventory.database.present = false` 时，直接 `skip`
* 不允许为未命中的数据库上下文生成空 `database-index.md`

## 16. Phase 4：Injection Routing

### 16.1 `injection-index.yaml`

这是新 skill 的关键产物，用于把 context tree 变成可路由的注入规则。

### 16.2 设计目标

* 最小必要注入
* 按阶段选择（`brainstorm / plan / work / review / compound`）
* 按任务类型选择（终局目标：`add-endpoint / bugfix / refactor-shared / review-change / data-change / general-implementation / unknown`）
* 缺失文件可 graceful skip
* task pack 优先于长 narrative
* review 优先风险文档
* work 优先 code-facts

v1 边界说明：

* 第一阶段只强制生成 `context-packs/review-change.md`
* 因此 `plan / work` 在 v1 主要依赖 `code-facts + fallback`，不以专用 task pack 为前提
* `review` 是 v1 中唯一保证存在专用 task pack 的消费方

### 16.3 Task Type Resolution Rules

终局目标支持以下核心 task types：

* `add-endpoint`
* `bugfix`
* `refactor-shared`
* `review-change`
* `data-change`
* `general-implementation`
* `unknown`

v1 路由要求：

* 必须稳定支持 `review-change`
* 必须稳定支持 `unknown`
* 允许声明其他 task types，但在 v1 中它们默认更多承担“路由占位”和“后续扩展位”角色
* 对 `plan / work` 而言，若对应专用 task pack 尚未生成，应回退到 `code-facts + fallback`

降级规则：

* 当无法稳定识别 task type 时，进入 `unknown`
* `unknown` 默认加载最小通用上下文：`00-summary.md`、`pitfalls/index.md`、`code-facts/public-entrypoints.md`、`code-facts/test-map.md`
* `task_type` 是路由优化器，不是运行前提；识别失败不能阻断 workflow，只能降级到通用上下文

### 16.4 典型规则

| 阶段 / 场景 | 注入内容 |
| --- | --- |
| `always` | `00-summary.md`, `pitfalls/index.md` |
| `plan` | `architecture/module-map.md`, `code-facts/public-entrypoints.md` |
| `work` | `code-facts/test-map.md`, `context-packs/<task-type>.md`（若不存在则回退 fallback） |
| `review` | `code-facts/high-risk-modules.md`, `context-packs/review-change.md` |

### 16.5 建议结构

以下 YAML 是终局目标结构示意，不代表 v1 必须一次性填满全部 `task_types` 分支。v1 至少应保证：

* `always`
* `stages.plan / work / review`
* `task_types.review-change`
* `task_types.unknown`

补充说明：

* `stages.brainstorm / compound` 属于 `injection-index.yaml` 的终局设计覆盖范围
* 但它们不属于阶段 3 的 v1 消费验证范围；阶段 3 只要求验证 `plan / work / review`
* 因此示意 YAML 中出现 `brainstorm / compound`，不代表当前版本必须完成对应消费接入

```yaml
always:
  - 00-summary.md

stages:

  work:

task_types:

selection_rules:

  - when:

  work: "优先 context-packs 和 test-map，而非 architecture"
```

### 16.6 Injection Rule Evaluation Model

`selection_rules` 必须是可判定规则，而不是语义标签。

第一阶段只支持以下谓词：

* `stage == <value>`
* `task_type == <value>`
* `fact.<path>.present == true`
* `fact.<path>.confidence in {observed, inferred}`
* `output_exists.<relative_path> == true`

设计约束：

* `selection_rules` 只负责“是否注入”
* `advice` 只负责“优先读取顺序”，不参与规则求值
* 第一阶段不引入自由布尔表达式或复杂 DSL

`output_exists.*` 的取值来源必须明确：

* 不额外新增独立状态文件
* 由消费方在读取 `injection-index.yaml` 时，根据 `docs/contexts/<slug>/` 下实际文件是否存在做实时求值
* `README.md` 中列出的已生成产物清单可作为人工消费时的辅助依据
* 若某个路径不存在，则对应 `output_exists.<relative_path> = false`

### 16.6.1 Task Type 最低识别契约

第一阶段不要求实现复杂分类器，但必须定义最低可执行契约：

* task type 由当前消费 workflow 负责判定
* 判定优先级：
  1. 用户显式给出的任务类型
  2. workflow 根据当前任务描述做的稳定映射
  3. 无法稳定识别时回退到 `unknown`
* `unknown` 是正式兜底类型，不得视为错误状态
* 第一阶段允许人工指定 task type，不以自动识别准确率作为上线前提

### 16.7 Injection Consumption Contract

在后续 workflow 尚未统一接入自动注入前，`injection-index.yaml` 至少支持两种消费方式：

#### 方式一：人工 / agent 手动消费

执行节点根据当前：

* `stage`
* `task type`
* `condition`

读取对应 required/optional 文档。

#### 方式二：未来自动消费

如果后续 spec-first 增加统一上下文装配器，可直接读取该 YAML 自动装配上下文。

换句话说，`injection-index.yaml` 必须先做到“**人工可依赖、未来可自动化**”。

消费失败策略：

* YAML 缺失：降级为手动读取 `README.md` + `00-summary.md` + `pitfalls/index.md`
* YAML 非法：记录 warning，降级为只消费 `always` 最小集合
* task type 未命中：回退到 `stage` 规则，不阻断 workflow
* condition 未命中：跳过条件扩展文档，不阻断 workflow

### 16.8 Initial Consumers

第一阶段明确这几个消费方：

* `spec-plan`：消费 `stage=plan`
* `spec-work`：消费 `stage=work`
* `spec-review`：消费 `stage=review`

在未接统一自动装配器前，由执行这些 skill 的 agent 按 `stage + task_type + selection_rules` 手动消费 `injection-index.yaml`。

## 17. 增量刷新设计

### 17.1 首次 bootstrap

首次 bootstrap 生成“当前版本要求的全部目标产物”，而不是无条件展开终局产物树中的全部文件。

### 17.2 后续 refresh 规则

| 变化区域                | 刷新产物                                           |
| ------------------- | ---------------------------------------------- |
| 入口文件变化              | `public-entrypoints.md`                        |
| shared/core 变化      | `high-risk-modules.md`；若 `core-dependencies.md` 已生成则一并刷新，未生成则对该目标执行 `skip / no-op` |
| 测试变化                | `test-map.md`                                  |
| 路由/handler/API 边界变化 | `add-endpoint.md`、`review-change.md`；若某个可选 task pack 尚未生成，则对该目标执行 `skip / no-op` |
| 重大结构变化              | 建议全量 refresh                                   |

### 17.3 Fingerprint Model

`fingerprints.json` 不能只记录输入文件 SHA，必须同时覆盖输入指纹、分析器版本和产物依赖关系。

建议最小结构采用“输入 + 产物依赖”的统一索引，不再把它描述成单纯的文件 SHA 列表：

```yaml
inputs:
  files:
    "src/server.ts": "<sha>"
  analyzer_versions:
    entrypoints: "v1"
    module_structure: "v1"
  schema_versions:
    fact_inventory: "v1"
    injection_index: "v1"

outputs:
  "code-facts/public-entrypoints.md":
    depends_on:
      - "src/server.ts"
      - "analyzer:entrypoints@v1"
      - "schema:fact_inventory@v1"
```

失效原则：

* 输入文件变化 -> 直接依赖产物失效
* analyzer 或 schema 版本变化 -> 对应产物失效
* 上游产物失效 -> 下游产物连带失效

### 17.4 指纹文件

`fingerprints.json` 存放在：

```text
.context/spec-first/bootstrap/<slug>/fingerprints.json
```

它的职责是记录：

* 输入文件 SHA
* analyzer / schema 版本
* 产物依赖关系
* 上次成功刷新时间

它不是单纯的文件哈希清单。

### 17.5 Incremental Invalidation Contract

增量刷新必须基于明确的失效规则，而不是“看情况全量”。

建议规则如下：

| 变化类型 | 判定方式 | 动作 |
| --- | --- | --- |
| 入口文件新增/修改/删除 | 命中 `main.*`、`server.*`、`app.*`、router/bootstrap/export 入口集合 | 局部刷新 `public-entrypoints.md`、相关 task pack；未生成的可选 task pack 直接 `skip / no-op` |
| `shared/core/common/util` 变化 | 路径命中共享核心目录集合 | 局部刷新 `high-risk-modules.md`；若 `core-dependencies.md` 已生成则一并刷新，未生成则对该目标执行 `skip / no-op` |
| 测试变化 | 命中 unit/integration/e2e、fixture/mock/setup 集合 | 局部刷新 `test-map.md` |
| 路由/API 边界变化 | 命中 handler/controller/route/api schema 集合 | 局部刷新 `add-endpoint.md`、`review-change.md`；当前未生成的可选产物直接 `skip / no-op` |
| 配置或依赖变化 | lockfile、核心配置、代码生成输入变化 | 强制重算 `high-risk-modules.md`、`injection-index.yaml`；若 `core-dependencies.md` 已生成则一并重算，未生成则对该目标执行 `skip / no-op` |
| rename/delete | 文件重命名、目录搬迁、关键入口删除 | 至少重算受影响模块对应产物；若涉及入口或层级边界则升级为全量评估 |

以下情况判定为“重大结构变化”，应执行全量 refresh：

* 顶层目录结构明显变化
* 多个关键入口同时迁移
* 层级识别结果变化
* 大规模重命名导致局部映射失真

## 18. 可选外部增强适配

### 18.1 适配原则

不强依赖，做可选增强。`spec-graph-bootstrap` 默认自给自足。

### 18.2 `code-review-graph` 适配

如果检测到本地已有 `code-review-graph` 结果，可增强：

* `core-dependencies.md` — 依赖关系
* `high-risk-modules.md` — 高风险模块
* `review-change.md` — 变化影响提示
* `test-map.md` — 测试关联建议

借鉴点：Tree-sitter AST、imports/calls/inheritance/tests 图、blast radius 思维、增量更新思维。

### 18.3 `graphify` 适配

`graphify` 只作为后续增强适配对象，不进入第一阶段主链路。如果未来启用，可增强：

* `README.md` — 总控页
* 知识仓库导出
* 一页式摘要
* confidence 表达
* 文档/图/代码混合场景下的知识组织

借鉴点：`GRAPH_REPORT.md`、`graph.json`、`cache/`、多模态与持久化知识层。

### 18.4 External Adapter Contract

外部增强适配必须定义清晰的输入协议，避免实现阶段“看情况解析”。

建议最小契约：

* 输入来源优先级：显式参数传入 > 固定扫描路径 > 不启用外部增强
* 最小 schema：必须能定位项目标识、模块关系或摘要主体，否则视为无效输入
* 版本/格式校验：若版本未知或 schema 不满足最小字段，记为 invalid
* 解析成功后只能作为增强信息，不得覆盖本地 Fact Inventory 中的 `Observed` 事实

### 18.5 External Adapter Fallback Rules

外部增强结果永远是“可选附加信息”，不能成为生成核心产物的单点依赖。

规则如下：

* 若检测不到 `code-review-graph`，则使用本地 Fact Inventory 继续生成
* 若检测不到 `graphify`，则使用本地 README 总控页模板继续生成
* 若外部结果格式异常，则记录 warning，不阻断 bootstrap
* 外部增强失败时，不影响最小闭环产物生成

## 21. MVP Success Criteria

第一阶段上线时，只要求 `spec-graph-bootstrap` 在**不依赖外部 graph 项目**的前提下，稳定生成以下核心产物：

* `README.md`
* `00-summary.md`
* `architecture/module-map.md`
* `pitfalls/index.md`
* `code-facts/public-entrypoints.md`
* `code-facts/test-map.md`
* `code-facts/high-risk-modules.md`
* `context-packs/review-change.md`
* `injection-index.yaml`

并满足：

* 可在至少 2 个真实项目上运行
* 后续 `plan / work / review` 节点具备可验证的消费路径
* 生成时间与 token 成本可接受
* 不依赖外部 graph 工具也能完成最小闭环

补充说明：

* 在 v1 中，`review` 应验证专用 task pack 消费
* 在 v1 中，`plan / work` 主要验证 `code-facts + fallback` 是否足够支撑主任务
* “后续节点能实际消费”在阶段化推进上应拆成：
  * `3A`：人工模拟消费验证成立
  * `3B`：真实 workflow 接入后成立

### 21.1 Outcome Metrics

除“能生成产物”外，还应验证它是否真实改善后续 workflow 质量。

建议至少跟踪以下指标：

* 平均上下文加载 token
* `plan / work / review` 冷启动耗时
* 一次通过率
* review 返工率

度量要求：

* 先建立旧链路 baseline，再比较 `spec-graph-bootstrap` treatment
* 至少在 2 个真实项目上观测
* 观察窗口建议为 2 周或 4 周
* 若指标无改善或明显恶化，应暂停替换并复盘

### 21.2 Token Budget Guidelines

为避免第一阶段在中大型项目上失控，按仓库规模定义轻量预算规则：

| 规模 | 判定信号 | 默认策略 |
| --- | --- | --- |
| `small` | 文件数较少、单应用、单语言 | 允许一次全量 bootstrap，优先生成全部固定任务 |
| `medium` | 文件数中等、存在多层结构 | 优先最小闭环；`layer-context` / `database-context` 严格按条件触发；限制 narrative 扩写范围 |
| `large` | 文件数很多、monorepo 或多应用结构 | 优先事实层与最小路由层；默认不全量展开 layers/database；必要时建议分模块 bootstrap |

每次运行至少记录：

* 扫描文件数
* 生成产物数
* 是否触发降级
* token 粗估或成本级别
* 项目规模档位

## 22. 验收标准

一个成功的 `spec-graph-bootstrap` 运行后，应能支持后续节点回答：

1. 这是什么项目？
2. 从哪里进入？
3. 做常见改动时先看哪里？
4. 哪些模块高风险？
5. 哪些测试应联看？
6. review 除了 diff 外还应看什么？
7. 不同任务类型该注入哪组上下文？
8. `task_type = unknown` 时是否仍回退到合理的最小上下文？

如果这些问题回答不了，说明它还只是“多了一套文档”，没有真正实现“提升 AI 编码质量”。

### 22.1 Validation Procedure

为让“后续节点能消费这些产物”可证伪，第一阶段至少执行两个最小实验：

#### 实验 1：Plan 消费验证

任务：新增一个简单接口。

检查：

* `spec-plan` 是否实际使用了 `architecture/module-map.md`
* `spec-plan` 是否实际使用了 `code-facts/public-entrypoints.md`
* 是否命中了正确的 fallback 集合；若存在专用 task pack，可作为增强项验证

#### 实验 2：Review 消费验证

任务：修改 shared 模块或高风险模块。

检查：

* `spec-review` 是否实际使用了 `code-facts/high-risk-modules.md`
* `spec-review` 是否实际使用了 `context-packs/review-change.md`
* `spec-review` 是否实际使用了 `code-facts/test-map.md`

每次实验至少记录：

* 命中的上下文文件
* 命中的 `stage`
* 命中的 `task_type`
* 是否触发 fallback 及其原因
* 缺失但被降级跳过的产物
* 是否存在漏注入、错注入或明显减少无关文件扫描

阶段 3 进入 `3B` 前，以上实验记录应统一采用固定模板，至少包含：`task_name`、`stage`、`task_type`、`expected_inputs`、`actual_inputs`、`fallback_reason`、`misleading_points`、`verdict`、`allow_enter_3b`。

## 23. 替换策略

### 23.1 当前阶段

* 保留 `skills/spec-bootstrap/`
* 新能力只进入 `skills/spec-graph-bootstrap/`

### 23.2 稳定判断

满足以下条件后，可删除旧 skill：

* 覆盖原 `spec-bootstrap` 的核心产物
* 新增产物被后续节点实际消费
* 生成成本可接受
* 至少在若干真实项目上验证通过

### 23.3 Migration Runbook

#### 阶段一：双入口开发期

* `/spec:bootstrap` 保持旧实现
* `/spec:graph-bootstrap` 指向新实现
* 所有新增能力仅进入 `spec-graph-bootstrap`

#### 阶段二：收敛切换期

* README、用户手册、命令模板、初始化产物、smoke tests 全部迁移到 `/spec:graph-bootstrap`
* 若真实项目验证出现阻断问题，立即停止删除动作并继续保留旧入口

#### 阶段三：删除旧入口

* 删除 `/spec:bootstrap` 对外入口
* 删除 `skills/spec-bootstrap/`
* 仅保留 `/spec:graph-bootstrap`

删除前检查清单：

* 命令模板已不再引用 `spec-bootstrap`
* `init` 产物与 smoke tests 已切换到新入口
* README / 用户手册 / 版本更新文档已同步
* 至少 2 个真实项目验证通过

### 23.4 最终状态

仅保留 `skills/spec-graph-bootstrap/` 与 `/spec:graph-bootstrap`，删除 `skills/spec-bootstrap/` 与 `/spec:bootstrap`。

## 24. 总结

`spec-graph-bootstrap` 的本质不是“多写几篇 bootstrap 文档”，而是把 Stage-0 升级成一个：

* **facts-first** — 先从代码中抽事实，再做摘要
* **graph-informed** — 借鉴 graph 结构化思维，但不强依赖外部图系统
* **injection-oriented** — 产物面向后续节点注入，不是文档齐全
* **incremental-friendly** — 首次全量，后续增量刷新

的上下文工程 skill。

它从代码、配置、测试与结构中抽取可信事实，再组织成：

* 项目摘要
* 架构与风险
* 代码事实索引
* 任务型上下文包
* 注入路由规则

最终服务于后续所有节点的上下文加载与编码质量提升。

# `spec-graph-bootstrap` Skill 最终技术方案

## 1. 文档目标

本文档是 `spec-graph-bootstrap` skill 的最终技术方案。

该 skill 的定位是：**替代现有 `spec-bootstrap` 的下一代 Stage-0 实现**，为后续 `brainstorm / plan / work / review / compound` 五阶段提供更高质量、更低 token、可路由、可维护的上下文资产。

现有 `spec-bootstrap` 已具备以下基础能力：

* 作为 Stage-0 supporting workflow
* 生成 `docs/contexts/<slug>/` 下的上下文资产
* 具备 Host Readiness Gate
* 具备 `Full / Enhanced / Basic` 三种分析模式
* 通过 `.context/spec-first/bootstrap/<slug>/tasks/<task-id>/prd.md` 管理任务控制面

但当前范围仍然以生成 context assets 为主，自动注入后续五阶段仍属于未来能力。

`spec-graph-bootstrap` 的目标不是简单复制现有能力，而是在保持 Stage-0 外部职责连续的前提下，借鉴两类能力：

* 从 `code-review-graph` 借鉴：**AST、依赖/调用/测试关系、影响面思维、增量更新思维**
* 从 `graphify` 借鉴：**一页式摘要入口、持久化知识图文件、缓存与 update/watch 思路、对事实与推断的谨慎表达**

---

## 2. 方案结论

### 2.1 目录策略：软替换

保留两个目录，采用“软替换”方式演进：

```text
skills/spec-bootstrap/         # 冻结，保持现状，不再增强
skills/spec-graph-bootstrap/   # 所有新能力都在这里演进
```

### 2.2 开发约定

* `spec-bootstrap` 不做任何处理，维持现状
* 所有 bootstrap 新功能都只补充到 `spec-graph-bootstrap`
* `spec-graph-bootstrap` 稳定并验证后，删除 `skills/spec-bootstrap/`
* 最终仅保留 `skills/spec-graph-bootstrap/`

### 2.3 职责定位

`spec-graph-bootstrap` 是一个 **graph-informed 的增强版 Stage-0 skill**，其职责是：

1. 从项目代码、配置、测试、schema 中抽取结构化事实
2. 生成项目级上下文资产
3. 生成代码事实层资产
4. 生成任务型上下文包
5. 生成上下文注入索引
6. 支持后续节点按任务类型和阶段进行最小必要上下文注入

### 2.4 命令入口策略

开发与验证阶段：

* 旧入口：`/spec:bootstrap`
* 新入口：`/spec:graph-bootstrap`

替换完成后：

* 对外统一保留 `/spec:bootstrap` 作为用户入口
* 内部实现切换到 `spec-graph-bootstrap`
* `/spec:graph-bootstrap` 可移除，或仅作为兼容别名保留一个过渡周期

这样可以保持用户心智稳定，降低迁移成本。

## 3. 背景问题

现有 `spec-bootstrap` 更偏“项目上下文文档生成器”。如果终局目标是**提升 AI 编码质量**，还存在以下缺口：

### 3.1 缺少代码事实层

当前更偏 narrative 文档，缺少：

* 真实入口点
* 核心依赖方向
* 测试映射
* 高风险模块

### 3.2 缺少任务型上下文包

当前产物更像“知识文档”，而不是：

* 新增接口要先看什么
* 修 bug 要先看什么
* 改 shared 模块要先看什么
* review 变化要先看什么

### 3.3 缺少注入路由

当前 bootstrap 以生成 context assets 为主，自动注入后续五阶段仍不是已完成能力。需要一个机器可读的注入路由文件，明确不同阶段和任务类型应该先加载哪些产物。

### 3.4 缺少增量与缓存思维

每次 bootstrap 都全量扫仓库、全量重写所有文档，会带来：

* token 浪费
* 时间成本高
* 同一窗口多节点上下文重复加载
* 文档维护成本高

---

## 4. 设计目标

### 4.1 业务目标

* 降低后续节点冷启动成本
* 降低盲目全仓扫描
* 提高实现、修改、review 的正确性
* 帮助后续节点选择更小、更准的上下文
* 支持长期维护与增量刷新
* 支持未来知识仓库集中维护

### 4.2 技术目标

* 保持与现有 bootstrap 的目录与控制面兼容
* 新增代码事实层
* 新增任务型上下文层
* 新增注入索引层
* 支持置信标记
* 支持增量刷新
* 允许可选增强地消费外部 graph 结果

---

## 5. 非目标

本 skill 不负责：

* 替代运行时 diff review 引擎
* 完整模拟 `code-review-graph` 的所有 MCP 行为
* 完整模拟 `graphify` 的全部多模态功能
* 强依赖外部图服务才能运行
* 自动改写后续五阶段实现逻辑

---

## 6. 设计原则

### 6.1 Facts First

先抽事实，再做摘要和任务包。不允许先凭空写架构，再去找代码佐证。

### 6.2 Graph-Informed, Not Graph-Dependent

借 graph 的结构化思维，不强依赖外部图系统才能运行。默认模式下自给自足完成基础事实抽取；检测到外部图产物时再增强。

### 6.3 Injection-Oriented

输出必须服务后续节点注入，而不是只追求“文档完备”。

### 6.4 Honest Confidence

统一区分：

* `Observed` — 有代码证据直接支持
* `Inferred` — 基于模式推断，无直接证据
* `Unknown` — 无法判断

### 6.5 Incremental-Friendly

首次全量生成，后续按变化刷新。

### 6.6 Compatible-by-Path

产物路径与控制面路径尽量延续现有 bootstrap 约定，以降低迁移成本。

## 7. 总体架构

### 7.1 架构概览

```text
Target Repository
  -> Phase 0: Host Readiness Gate
  -> Phase 1: Repository Analysis + Fact Inventory Extraction
  -> Phase 2: Context Task Planning + PRD Generation
  -> Phase 3: Context Asset Rendering
  -> Phase 4: Injection Routing Generation
  -> docs/contexts/<slug>/ 产物树
```

### 7.2 三层模型

#### 第一层：事实层

输入：

* 代码
* 配置
* 测试
* schema / migration
* 启动入口
* 路由 / worker / CLI 入口

输出（控制面）：

* `fact-inventory.json`
* `risk-signals.json`
* `test-surface.json`
* `fingerprints.json`

#### 第二层：摘要层

基于事实层生成：

* `00-summary.md`
* `architecture/*`
* `pitfalls/index.md`
* `code-facts/*`
* `README.md`

#### 第三层：注入层

基于摘要层和事实层生成：

* `context-packs/*`
* `injection-index.yaml`

## 12. Phase 1：Repository Analysis + Fact Inventory

这是新 skill 的核心改造点。

### 12.1 目标

在生成任何文档前，先形成一个统一的中间事实集。

### 12.2 为什么要这样

`code-review-graph` 的关键价值，在于它先构建结构化代码图，再在图上做 blast radius、dependency chains、test coverage gaps 等查询，而不是每个问题重新扫仓库。

`spec-graph-bootstrap` 不一定完整复制这套实现，但必须借鉴“**先有事实层，再有文档层**”的架构。

### 12.3 Fact Inventory 模型

```yaml
project_identity:
  name:
  primary_language:
  primary_frameworks: []
  repo_shape:         # monorepo | service | library | cli

entrypoints: []
modules: []
integrations: []
data_shapes: []
testing_surface: []
risk_signals: []
layers: []
database: {}
```

每条事实统一字段：

```yaml
path:
symbol:
kind:
summary:
confidence: observed | inferred | unknown
evidence:
  - file:
  - symbol:
```

### 12.4 子提取器

#### 12.4.1 Project Identity Extractor

识别：项目名、主语言、主框架、仓库形态

数据源：顶层目录、package/go/cargo/java 等元文件、框架标记文件

#### 12.4.2 Entrypoint Extractor

识别：HTTP 入口、CLI 入口、worker/cron/queue 入口、SDK/public export 入口

数据源：`main.*`、`server.*`、`app.*`、router/bootstrap 文件、export/index 文件

#### 12.4.3 Module Structure Extractor

识别：顶层目录职责、service/controller/repository/model 模式、shared/core/common/util 层、模块自然边界

#### 12.4.4 Integration Extractor

识别：DB client、cache、MQ、外部 HTTP/gRPC/SDK、auth/session 边界

#### 12.4.5 Test Surface Extractor

识别：unit/integration/e2e 测试布局、fixture/mock/setup、模块与测试的近邻关系

#### 12.4.6 Risk Signal Extractor

识别：高扇出模块、高复杂度热点、shared/core/common/util 风险点、弱测试区域、全局状态、异常吞掉点、配置中心/基础设施中心

#### 12.4.7 Data Shape Extractor

识别：DTO、schema、entity/model、migration、validation

### 12.5 控制面输出

```text
.context/spec-first/bootstrap/<slug>/
  fact-inventory.json
  risk-signals.json
  test-surface.json
  fingerprints.json
```

### 12.6 Fact-to-Document Boundary

为避免重复扫描和推断失控，定义如下边界：

* `fact-inventory.json`、`risk-signals.json`、`test-surface.json` 只保存**可追溯事实**与置信度
* `summary / architecture / pitfalls / code-facts / context-packs` 必须优先基于事实层生成
* renderer 不应绕过事实层直接重新扫源码写各自结论
* 文档中若出现超出事实层的推断，必须显式标记为 `Inferred`

## 13. Phase 2：Task Planning

继续沿用现有 bootstrap 的 PRD task 机制。

### 13.1 固定任务

#### 13.1.1 `summary-context`

输出：`00-summary.md`

要求：

* 一句话项目定位
* 主语言 / 主框架
* 真实入口
* 核心目录
* 高风险区域摘要
* 推荐优先阅读顺序

#### 13.1.2 `architecture-context`

输出：

* `architecture/system-overview.md`
* `architecture/module-map.md`
* `architecture/integration-boundaries.md`（条件）

要求：

* 运行路径视角
* 模块职责
* 常见变更入口
* 集成边界
* 跨层风险边界

#### 13.1.3 `pitfalls-context`

输出：`pitfalls/index.md`

要求：

* 按任务类型组织
* 每个 pitfall 有位置、原因、风险、缓解建议、相关测试

### 13.2 新增固定任务

#### 13.2.1 `code-facts-context`

输出：

* `code-facts/public-entrypoints.md`
* `code-facts/core-dependencies.md`
* `code-facts/test-map.md`
* `code-facts/high-risk-modules.md`

作用：建立代码事实层，降低实现和 review 的冷启动成本。

#### 13.2.2 `injection-context`

输出：

* `context-packs/add-endpoint.md`
* `context-packs/bugfix.md`
* `context-packs/refactor-shared.md`
* `context-packs/review-change.md`
* `injection-index.yaml`

作用：把 context 文档树转成任务型上下文包 + 注入路由规则。

### 13.3 条件任务

#### 13.3.1 `layer-context`

输出：`layers/*/index.md`

增强要求：本层入口、常见改动起点、高风险点、相关测试、邻接层关系

#### 13.3.2 `database-context`

输出：`database/*`

增强要求：哪些代码目录依赖该数据层、哪类改动风险高、哪些约束最关键

## 14. PRD 模板规范

每个任务 PRD 应统一结构：

1. **Goal** — 本任务要交付什么
2. **Target Output** — 输出文件列表
3. **Why It Matters** — 对后续节点的价值
4. **Data Sources** — 从哪里取事实
5. **Extraction Rules** — 抽取与组织规则
6. **Output Requirements** — 格式与内容要求
7. **Acceptance Criteria** — 验收标准
8. **Risks / Caveats** — 风险与注意事项
9. **Suggested Workflow** — 建议执行步骤

## 15. Phase 3：Context Asset Rendering

### 15.1 通用输出要求

所有文件必须：

* 结构化、可扫描
* 少空话，引用具体路径/模块/测试
* 明确标注 `Observed / Inferred / Unknown`
* 以“供后续节点消费”为优先目标，而不是文案美观

### 15.2 置信表达

适用文件：

* `system-overview.md`
* `integration-boundaries.md`
* `high-risk-modules.md`
* `context-packs/*.md`

### 15.3 README 总控页

借鉴 `graphify` 的一页式摘要模式（`GRAPH_REPORT.md`），将 `README.md` 升级为上下文总控页。

固定章节：

1. Project Snapshot
2. Read This First
3. High-Risk Areas
4. Main Entrypoints
5. Architecture
6. Code Facts
7. Context Packs
8. Layers
9. Database
10. Freshness / Caveats

### 15.4 README 与 `00-summary.md` 的角色边界

#### `00-summary.md`

定位：**最小事实摘要**

负责回答：

* 这是什么项目
* 主技术栈是什么
* 从哪里进入
* 核心模块有哪些
* 高风险区域有哪些

#### `README.md`

定位：**上下文控制台**

负责回答：

* 先读什么
* 有哪些上下文资产
* 哪些 task pack 可用
* 当前分析模式与 freshness 状态
* 如何扩展读取更多上下文

### 15.5 `high-risk-modules.md` 排序规则

高风险模块建议按以下维度综合排序：

1. 扇出高
2. shared/core/common/util 层
3. 测试薄弱
4. 复杂度高
5. 跨层传播风险高

即使第一阶段不做复杂打分，也应遵循上述统一排序口径。

### 15.6 `context-packs/*.md` 固定模板

每个 task pack 文档必须使用统一结构：

1. Purpose
2. When to use
3. Read first
4. Key modules / directories
5. Related tests
6. Common pitfalls
7. Avoid overreach
8. Escalation hints

其目标是保证后续节点可快速消费，而不是让每份 pack 风格分散。

## 16. Phase 4：Injection Routing

### 16.1 `injection-index.yaml`

这是新 skill 的关键产物，用于把 context tree 变成可路由的注入规则。

### 16.2 设计目标

* 最小必要注入
* 按阶段选择（`brainstorm / plan / work / review / compound`）
* 按任务类型选择（`add-endpoint / bugfix / refactor-shared / review-change`）
* 缺失文件可 graceful skip
* task pack 优先于长 narrative
* review 优先风险文档
* work 优先 code-facts

### 16.3 典型规则

| 阶段 / 场景  | 注入内容                                       |
| -------- | ------------------------------------------ |
| `always` | `00-summary.md`, `pitfalls/index.md`       |
| `plan`   | `module-map.md`, `public-entrypoints.md`   |
| `work`   | `test-map.md`, task-specific pack          |
| `review` | `high-risk-modules.md`, `review-change.md` |

### 16.4 建议结构

```yaml
always:
  - 00-summary.md
  - pitfalls/index.md

stages:
  brainstorm:
    - architecture/system-overview.md
  plan:
    - architecture/module-map.md
    - code-facts/public-entrypoints.md
  work:
    - code-facts/test-map.md
  review:
    - code-facts/high-risk-modules.md
  compound:
    - README.md

task_types:
  add-endpoint:
    - context-packs/add-endpoint.md
  bugfix:
    - context-packs/bugfix.md
  refactor-shared:
    - context-packs/refactor-shared.md
  review-change:
    - context-packs/review-change.md

selection_rules:
  - condition: "has_database"
    inject: ["database/database-er.md"]
  - condition: "has_layer:frontend"
    inject: ["layers/frontend/index.md"]

expansion_hints:
  review: "优先 code-facts 和 risk signals，而非 narrative"
  work: "优先 context-packs 和 test-map，而非 architecture"
```

### 16.5 Injection Consumption Contract

在后续 workflow 尚未统一接入自动注入前，`injection-index.yaml` 至少支持两种消费方式：

#### 方式一：人工 / agent 手动消费

执行节点根据当前：

* stage
* task type
* condition

读取对应 required/optional 文档。

#### 方式二：未来自动消费

如果后续 spec-first 增加统一上下文装配器，可直接读取该 YAML 自动装配上下文。

换句话说，`injection-index.yaml` 必须先做到“**人工可依赖、未来可自动化**”。

### 17.2 后续 refresh 规则

## 18. 可选外部增强适配

### 18.1 适配原则

不强依赖，做可选增强。`spec-graph-bootstrap` 默认自给自足。

### 18.2 `code-review-graph` 适配

如果检测到本地已有 `code-review-graph` 结果，可增强：

* `core-dependencies.md` — 依赖关系
* `high-risk-modules.md` — 高风险模块
* `review-change.md` — 变化影响提示
* `test-map.md` — 测试关联建议

借鉴点：Tree-sitter AST、imports/calls/inheritance/tests 图、blast radius 思维、增量更新思维。

### 18.3 `graphify` 适配

如果检测到本地已有 `graphify` 结果，可增强：

* `README.md` — 总控页
* 知识仓库导出
* 一页式摘要
* confidence 表达
* 文档/图/代码混合场景下的知识组织

借鉴点：`GRAPH_REPORT.md`、`graph.json`、`cache/`、多模态与持久化知识层。

### 18.4 External Adapter Fallback Rules

外部增强结果永远是“可选附加信息”，不能成为生成核心产物的单点依赖。

规则如下：

* 若检测不到 `code-review-graph`，则使用本地 Fact Inventory 继续生成
* 若检测不到 `graphify`，则使用本地 README 总控页模板继续生成
* 若外部结果格式异常，则记录 warning，不阻断 bootstrap
* 外部增强失败时，不影响最小闭环产物生成

## 22. 验收标准

一个成功的 `spec-graph-bootstrap` 运行后，应能支持后续节点回答：

1. 这是什么项目？
2. 从哪里进入？
3. 做常见改动时先看哪里？
4. 哪些模块高风险？
5. 哪些测试应联看？
6. review 除了 diff 外还应看什么？
7. 不同任务类型该注入哪组上下文？

如果这些问题回答不了，说明它还只是“多了一套文档”，没有真正实现“提升 AI 编码质量”。

## 24. 总结

`spec-graph-bootstrap` 的本质不是“多写几篇 bootstrap 文档”，而是把 Stage-0 升级成一个：

* **facts-first** — 先从代码中抽事实，再做摘要
* **graph-informed** — 借鉴 graph 结构化思维，但不强依赖外部图系统
* **injection-oriented** — 产物面向后续节点注入，不是文档齐全
* **incremental-friendly** — 首次全量，后续增量刷新

的上下文工程 skill。

它从代码、配置、测试与结构中抽取可信事实，再组织成：

* 项目摘要
* 架构与风险
* 代码事实索引
* 任务型上下文包
* 注入路由规则

最终服务于后续所有节点的上下文加载与编码质量提升。

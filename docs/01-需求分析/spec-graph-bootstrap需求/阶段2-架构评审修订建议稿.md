## 1. 评审结论

当前方案整体方向正确，核心架构判断成立：

* 以事实层为中心组织控制面与长期上下文产物，方向正确
* `Full / Enhanced / Basic` 三级降级模型合理
* 将控制面产物与 markdown 消费产物分离，是可扩展的演进路径
* 将 `fingerprints.json` 作为阶段 4 refresh / invalidation 的前置基础，思路正确

但当前文档仍存在一组会直接影响落地的 contract 缺口与语义冲突，主要集中在以下四类：

* 状态机定义不互斥，导致 Phase 0 无法稳定判定
* 事实层 schema 未真正统一，导致 Worker 难以“统一消费”
* 部分高成本 extractor 没有闭环到最终产物
* 路由层与任务规划层在需求文档中是核心交付，但集成方案未定义到可执行级别

建议将修订项按 `P0 / P1 / P2` 三档处理。`P0` 需先修再实现，`P1` 应在实现前补齐，`P2` 可与实现并行收敛。

## 2. P0：必须先修的阻断项

### P0-1. Phase 0 中 `crg.indexed` 判定逻辑互相冲突

**涉及文档**

* [`阶段2-集成方案.md`](/Users/kuang/xiaobu/spec-first/docs/01-需求分析/spec-graph-bootstrap需求/阶段2-集成方案.md)

**问题说明**

`§3.2 CRG 探测流程` 中同时存在以下两条规则：

* `graph_mtime` 存在且 `node_count=0 && edge_count=0` → `indexed=false`
* `graph_mtime` 存在且 `node_count≥0（含 0）` → `indexed=true`

这两个条件对“空图 / 空项目”场景的解释互相覆盖，导致：

* 是否提示“现在构建图索引”不可判定
* 是否进入 `Full` 模式不可判定
* Phase 0 状态机不是互斥完备状态机

**修订建议**

将 `CRG probe` 状态收敛为互斥四态：

1. `unavailable`
2. `available_but_uninitialized`
3. `available_and_indexed_empty_repo`
4. `available_and_indexed_nonempty`

建议改写为：

```text
若工具调用失败 -> unavailable
若工具调用成功但缺少 graph_mtime -> available_but_uninitialized
若 graph_mtime 存在，且明确返回 repo_is_empty=true -> available_and_indexed_empty_repo
若 graph_mtime 存在，且节点/边统计有效 -> available_and_indexed_nonempty
```

如果 CRG API 没有 `repo_is_empty` 这类显式字段，则不要试图仅凭 `node_count=0` 同时表达“未建索引”和“空项目已索引”。

**建议新增一句**

> 在缺少显式空仓库标志位时，`node_count=0 && edge_count=0` 一律视为 `available_but_uninitialized`，避免误判进入 `Full` 模式。

### P0-2. 事实层最小 contract 未统一，违背“统一消费”目标

**涉及文档**

* [`阶段2-事实抽取与文档生成需求.md`](/Users/kuang/xiaobu/spec-first/docs/01-需求分析/spec-graph-bootstrap需求/阶段2-事实抽取与文档生成需求.md)
* [`阶段2-集成方案.md`](/Users/kuang/xiaobu/spec-first/docs/01-需求分析/spec-graph-bootstrap需求/阶段2-集成方案.md)

**问题说明**

需求文档 `§5.1` 明确要求每条事实至少支持：

* `path`
* `symbol`
* `kind`
* `summary`
* `confidence`
* `inference_reason`
* `evidence`
* `updated_at`

但集成方案中的若干事实结构未满足这一统一 contract：

* `modules` 缺少 `kind / summary / updated_at`
* `testing_surface` 缺少 `inference_reason / updated_at`
* `data_shapes` 缺少 `inference_reason / updated_at`
* `risk-signals` 未和统一事实 contract 对齐

这会直接导致：

* Worker 无法按统一字段读取事实
* 不同事实类型需要单独分支解析
* “Facts First” 退化为“事实文件存在，但 schema 各自为政”

**修订建议**

在集成方案中新增一个“统一事实基类”约束，并明确：

```yaml
fact_base:
  path: string
  symbol: string
  kind: string
  summary: string
  confidence: observed | inferred | unknown
  inference_reason: string | null
  evidence: [object]
  updated_at: ISO datetime
```

然后说明：

* `entrypoints / modules / integrations / testing_surface / data_shapes / risk_signals` 均以该基类为基础扩展
* 允许某些事实类型增加专有字段，如 `severity`、`community_id`、`test_kind`
* 不允许删除基类字段

**落地建议**

如果担心 `testing_surface` 或 `layers` 不自然，可分两类：

* `fact_record[]` 型：强制继承统一字段
* `state_record` 型：如 `layers.frontend.present`，允许独立 schema

但必须在文档里明确哪类属于“事实条目”，哪类属于“状态聚合”。

### P0-3. `data_shapes` 的置信度与 `inference_reason` 前后冲突

**涉及文档**

* [`阶段2-集成方案.md`](/Users/kuang/xiaobu/spec-first/docs/01-需求分析/spec-graph-bootstrap需求/阶段2-集成方案.md)

**问题说明**

当前文档中同时存在两套彼此冲突的定义：

* `§4.8` 把 Full 模式 `semantic_search_nodes + children_of` 定义为 `Observed`
* `§5.5` 又把 Full 模式 `data_shapes` 写成 `inferred（inference_reason: "crg-importers-evidence"）`

此外，`"crg-importers-evidence"` 明显属于 `Integration Extractor` 的语义，不适用于 `data_shapes`。

**修订建议**

先统一语义，再修正命名：

**方案 A：保持 `Observed`**

适用条件：

* 只要 CRG 返回的节点/字段直接作为 `data_shapes` 事实输出
* 不对 `DTO / entity / schema / validation` 做额外语义映射

则应写为：

```text
data_shapes -> observed
```

**方案 B：维持 `Inferred`**

适用条件：

* 虽有 AST 节点证据，但“该节点属于 DTO 还是 entity”仍依赖命名规则或预置分类

则应写为：

```text
data_shapes -> inferred
inference_reason: "crg-children-ast-evidence"
```

建议优先采用 **方案 B**，因为当前文档想表达的是：

* 结构存在是 AST 事实
* 语义类别仍需规则映射

这更符合 `inferred with observed evidence` 的实际含义。

### P0-4. `integrations` 与 `data_shapes` 提取链已定义，但产物消费链未闭合

**涉及文档**

* [`阶段2-集成方案.md`](/Users/kuang/xiaobu/spec-first/docs/01-需求分析/spec-graph-bootstrap需求/阶段2-集成方案.md)

**问题说明**

`§8 Fact Inventory → 最终产物映射` 中未为 `integrations`、`data_shapes` 指定：

* 下游 consumer
* 对应 worker
* 最终 markdown 产物

这两个 extractor 都有明确的提取成本，尤其 `integrations` 可能触发大量 `importers_of` 查询。如果没有消费闭环，会出现：

* 高成本提取但不产出用户价值
* 阶段范围虚胖
* Worker 不知道哪些事实需要在文档中体现

**修订建议**

二选一明确写入文档：

**方案 A：补下游消费**

例如：

* `integrations` → `architecture-context` worker / `00-summary.md` / `module-map.md`
* `data_shapes` → `code-facts-context` worker / `pitfalls/index.md` / 新增 `code-facts/data-shapes.md`

**方案 B：明确阶段边界**

在 `§8` 或 `§6.1` 增补：

> `integrations` 与 `data_shapes` 在 v1 中只写入控制面产物，不直接驱动 markdown 产物；其消费接入放到阶段 3。

如果目标是“最小闭环优先”，建议优先使用 **方案 B**。

### P0-5. `injection-index.yaml` 是核心交付，但没有被定义为可实现的 contract

**涉及文档**

* [`阶段2-事实抽取与文档生成需求.md`](/Users/kuang/xiaobu/spec-first/docs/01-需求分析/spec-graph-bootstrap需求/阶段2-事实抽取与文档生成需求.md)
* [`阶段2-集成方案.md`](/Users/kuang/xiaobu/spec-first/docs/01-需求分析/spec-graph-bootstrap需求/阶段2-集成方案.md)

**问题说明**

需求文档明确要求：

* `selection_rules` 至少支持 `stage`、`task_type`、`fact.*`、`output_exists.*`
* 覆盖 `always / plan / work / review / unknown`
* `unknown` 需提供最小回退集合

但集成方案中没有给出：

* `injection-index.yaml` 的最小 schema
* 规则求值顺序
* 多条规则命中时的优先级
* `output_exists.*` 的运行时求值边界
* `unknown` 回退集合的内容定义

这会直接阻断实现，因为开发者既不知道 YAML 应该长什么样，也不知道消费方应如何解释它。

**修订建议**

新增独立小节：`§6.x injection-index.yaml 最小 schema`

建议最小结构：

```yaml
schema_version: "v1"
generated_at: ISO datetime

always:
  - "00-summary.md"

stages:
  plan:
    include: ["00-summary.md", "architecture/module-map.md"]
  work:
    include: ["00-summary.md", "code-facts/public-entrypoints.md"]
  review:
    include: ["00-summary.md", "code-facts/high-risk-modules.md", "context-packs/review-change.md"]

task_types:
  unknown:
    include: ["00-summary.md", "README.md"]

selection_rules:
  - when:
      stage: "review"
      output_exists.context_packs_review_change: true
    include: ["context-packs/review-change.md"]
  - when:
      task_type: "unknown"
    include: ["README.md", "00-summary.md"]

caveats:
  - "context-packs/review-change.md missing, fallback to unknown set"
```

同时明确：

* `output_exists.*` 是消费时基于文件存在性的实时求值，不从状态文件读取
* 规则优先级按文档顺序求值，后匹配可追加，不覆盖 `always`

## 3. P1：实现前应补齐的重要项

### P1-1. `fingerprints.json` 与需求文档 contract 不一致

**问题说明**

需求文档要求 `fingerprints.json` 顶层至少包含：

* `inputs`
* `outputs`
* `updated_at`

但集成方案目前给的是：

* `schema_version`
* `generated_at`
* `inputs`
* `outputs`

缺少 `updated_at`。

**修订建议**

两种修法任选其一：

* 将 `generated_at` 重命名为 `updated_at`
* 同时保留两个字段，并说明：
  * `generated_at` = 当前本次生成时间
  * `updated_at` = 最后一次成功写入该文件的时间

若目标是最小化 schema，建议直接统一为 `updated_at`。

---

### P1-2. `§4.11 写入控制面` 小节缺失

**问题说明**

`§4.1` 总览中出现了 `4.11 写入控制面`，但正文没有该章节。

**修订建议**

补充 `§4.11`，至少写清：

* 写入是否采用临时文件 + rename 的原子写入
* 当部分 extractor 失败时，是否允许部分写入
* 写入前是否做 schema 校验
* 写入后是否立即更新 `fingerprints.json`

推荐策略：

* 控制面 JSON 逐文件原子写入
* 某 extractor 失败时，允许其字段写为 `unknown` 或空列表，不阻断其他文件落盘
* 所有控制面文件写完后，再统一写 `fingerprints.json`

---

### P1-3. `inference_reason` 对 `observed` 事实的 nullable 规则未写清

**问题说明**

当前 schema 中多个事实条目的 `inference_reason` 表示为 `string`，而规则层面只要求 `inferred` 必填。

**修订建议**

全局统一为：

```yaml
inference_reason: string | null
```

并新增全局规则：

* `observed` → `inference_reason = null`
* `inferred` → `inference_reason` 必填
* `unknown` → `inference_reason` 可填 `"no-evidence"` 或 `null`，二者择一并全局统一

建议 `unknown` 使用 `"no-evidence"`，可读性更强。

### P1-4. `last_build_commit` 字段来源未定义

**问题说明**

`fingerprints.json.inputs.crg.last_build_commit` 已在 schema 中定义，但没有定义来源。

**修订建议**

补一条显式规则：

> `last_build_commit` 优先取 CRG 工具返回的 commit 字段；如 CRG 不提供，则以目标仓库 `git rev-parse HEAD` 结果填充；若仓库非 git 项目或获取失败，填 `null`。

同时说明：

* detached HEAD 也允许，直接记录 commit hash
* 不应把 branch name 写入该字段

### P1-5. `Enhanced` 模式的 `risk_signals.severity` 提升边界未量化

**问题说明**

当前文档只给了 Basic 模式的 severity 约束表，未定义 Enhanced 模式在“多个独立 inferred 信号共同支持”时是否允许升到 `high`。

**修订建议**

补一张 Enhanced 模式约束表，建议写为：

| 条件 | severity 上限 |
|---|---|
| 单一 Serena 模式命中 | `medium` |
| 2 个独立 Serena 信号交叉命中 | `high` |
| Serena + 路径近邻共同支持 | `high` |
| 仅依赖目录命名 | `medium` |

同时定义“独立信号”的最小判定规则：

* 不同工具来源，或
* 同一工具但不同模式命中类型

### P1-6. 构建索引的用户提示时长与超时阈值需统一

**问题说明**

当前文档中：

* 提示用户时说“通常 1-5 分钟”
* 失败处理时超时阈值写“>10 分钟”

**修订建议**

统一为：

> 通常 1-5 分钟，大型项目最长可达 10 分钟。超过 10 分钟视为超时并自动降级。

这样用户预期和系统行为一致。

---

### P1-7. `并行执行` 的运行时语义应明确

**问题说明**

当前文档多次使用“阶段内并行执行”，但未明确：

* 是逻辑无依赖，顺序不限
* 还是同一轮响应内批量发起 tool call

**修订建议**

将措辞统一改为：

> 同一 Stage 内任务逻辑上无依赖，可并行发出 tool call；若宿主不支持并行调用，则允许按任意顺序串行执行，结果 contract 不变。

这样不会误导实现方去引入线程或协程语义。

### P1-8. `coverage_gaps.severity` 判定阈值未定义

**问题说明**

`test-surface.json.coverage_gaps[].severity` 只写了“根据 fan-out 和 criticality 判定”，没有阈值。

**修订建议**

定义最小量化规则，例如：

| 条件 | severity |
|---|---|
| `criticality=high` 且 `fan_out >= 10` | `high` |
| `criticality=medium` 或 `fan_out 3-9` | `medium` |
| 其他情况 | 不进入 `coverage_gaps` |

如果 CRG 不直接返回 `criticality` 数值，则改成离散规则并写死。

## 4. P2：建议同步修整的清洁度与可维护性问题

### P2-1. `get_impact_radius_tool(changed_files=["src/core/"])` 是否接受目录路径需补 API contract

**问题说明**

当前文档示例把目录路径直接传给 `changed_files`。如果 CRG 只接受文件列表，这里会出现静默空结果或运行时报错。

**修订建议**

不要直接在方案里写目录输入。改成：

> 若风险分析目标为目录，先展开为文件列表，再传入 `get_impact_radius_tool`。

这样规避对未验证 API 行为的依赖。

---

### P2-2. stale 检测表述应从“自动失效检测”改为“过期提示检测”

**问题说明**

当前文档一处写“v1 只记录依赖关系，不做自动失效检测”，后文又详细描述了 stale 检测逻辑。

**修订建议**

统一措辞为：

> v1 会做 stale 检测提示，但不做自动重建。

这是准确表述，也更符合当前方案真实能力边界。

---

### P2-3. `README.md` Freshness 章节需最小 schema

**问题说明**

文档多次提到降级、缺失产物、stale 信号应在 `README.md` 的 `Freshness` 章节中体现，但没有定义该章节的最小结构。

**修订建议**

为 `README.md` 新增最小 contract：

```markdown
## Freshness

- Analyzer mode: Full | Enhanced | Basic
- Generated at: ISO datetime
- Stale signals:
  - ...
- Missing outputs:
  - ...
- Degrade events:
  - ...
```

这样后续实现、测试与 review 都有稳定锚点。

## 5. 建议补充的明确文本

### 5.1 统一事实字段规则

```text
除 layers/database 这类状态聚合对象外，所有事实条目均必须继承统一事实字段：
path / symbol / kind / summary / confidence / inference_reason / evidence / updated_at。
observed 事实的 inference_reason 填 null；inferred 事实必须填写原因；unknown 事实不得进入强结论分支。
```

### 5.2 `integrations` / `data_shapes` 的 v1 边界

```text
v1 中 integrations 与 data_shapes 默认写入控制面，不强制驱动 markdown 产物。
如后续产物需要消费这两类事实，应在阶段 3 中新增明确的 consumer 与输出文件。
```

### 5.3 `injection-index.yaml` 的最小执行语义

```text
always 为无条件包含集合。
stages / task_types 为静态建议集合。
selection_rules 按文档顺序求值，命中时附加 include 集合。
output_exists.* 在消费时基于文件系统实时判定，不依赖缓存状态文件。
unknown 必须提供不阻断 workflow 的最小回退集合。
```

### 5.4 控制面写入语义

```text
每个控制面文件采用临时文件写入后原子替换。
单个 extractor 失败不会阻断其他控制面文件写入，但必须将失败原因记录为 fallback_reason，并在 README.md 的 Freshness 章节对用户可见。
```

## 6. 推荐修订顺序

建议按以下顺序修文档，再进入实现：

1. 先修 `Phase 0` 状态机互斥性
2. 再统一事实层最小 schema
3. 再确定 `data_shapes` 的置信度语义
4. 明确 `integrations / data_shapes` 的阶段边界
5. 补齐 `Phase 2 / 3 / 4` 的最小可执行 contract
6. 最后整理 stale、Freshness、缺失产物和降级可见性

这样可以先保证“能实现”，再保证“实现后可验证”。

## 7. 最终建议结论

当前方案不需要推倒重来，但不建议直接进入编码。

更准确的判断是：

* 架构方向正确
* 阶段边界基本合理
* 但 contract 还没有收敛到“不同实现者会得到同一结果”的程度

只要先完成本修订建议稿中的 `P0` 与 `P1`，这套方案就能从“好思路”收敛为“可执行的系统设计”。

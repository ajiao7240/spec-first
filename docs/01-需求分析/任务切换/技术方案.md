# 任务切换 Skill 技术方案

## 1. 背景

当前 spec-first 已经有完整的任务体系：

- `task create` 创建任务目录
- `task init-context` 生成实现 / 检查 / 调试上下文
- `current_task.py switch` 切换当前任务指针 `.spec-first/.current-task`
- `task.py start` 保持为低层兼容原语
- `get_context.py` 和各平台 hook 根据当前任务指针恢复上下文

问题在于，切换任务这件事对使用者来说仍然是“记命令”，而不是“语义化入口”。

因此需要新增一个 skill，把“切换任务”封装成更自然的用户操作。

---

## 2. 目标

新增一个 `current-task` skill，用于：

- 让用户用 skill 语义切换当前任务
- 只修改当前任务锚点，不改写任务内容
- 复用现有 `current_task.py` 作为统一入口，`task.py start` 作为底层原语
- 保持行为可预测、低风险、易回滚

---

## 3. 非目标

这个 skill 不负责：

- 新建任务
- 自动写 PRD
- 自动生成 `implement.jsonl` / `check.jsonl` / `debug.jsonl`
- 自动归档旧任务
- 自动修改 `spec/`

如果需要完整任务初始化，仍然应该显式执行：

```bash
python3 ./.spec-first/scripts/task.py init-context <task> <dev_type>
python3 ./.spec-first/scripts/current_task.py switch <selection>
```

---

## 4. 现状判断

### 4.1 切换任务的真实锚点

当前任务切换的唯一状态源是：

```text
.spec-first/.current-task
```

`current_task.py switch` 会把任务目录写进去，`task.py start` 保留为低层原语。
后续的上下文恢复、会话注入、任务显示，都会读取这个锚点。

### 4.2 上下文来源

后续 skill 或 session hook 读取任务时，依赖的是：

- `.spec-first/.current-task`
- 当前任务目录下的 `task.json`
- `prd.md`
- `implement.jsonl`
- `check.jsonl`
- `debug.jsonl`

也就是说，**skill 不需要发明新的状态存储**，只要切换当前任务指针即可。

---

## 5. 方案对比

### 方案 A：模糊匹配后直接切换

skill 先尝试解析任务名，再决定具体切哪个。

优点：

- 输入短
- 少一次用户确认

缺点：

- 规则复杂
- 容易产生歧义
- 和现有任务切换链路不一致

### 方案 B：先列列表，再让用户选择

skill 先展示任务列表，再由用户明确指定要切换的任务。

优点：

- 最简单
- 没有歧义
- 不需要维护模糊匹配逻辑
- 与当前实现能力一致

缺点：

- 多一次选择步骤

### 方案 C：切换 + 自动补上下文

skill 切换任务时顺带初始化上下文。

优点：

- 一次性把体验做完整

缺点：

- 会修改更多文件
- 容易让“切换任务”和“初始化任务”概念混淆
- 风险明显高于前两种

---

## 6. 推荐方案

推荐采用 **方案 B + 轻量提示**。

核心原则：

- `current-task` 默认只负责“列出任务并让用户选”
- 用户明确选择后，再调用 `current_task.py switch <selection>`
- 如果目标任务上下文不完整，skill 只提示，不自动改写

这样可以保证：

- 切换行为稳定
- 不需要模糊解析
- 不会误覆盖任务数据
- 与现有 `get_context` / session hook 机制完全兼容

---

## 7. Skill 设计

### 7.1 统一命名

统一名称建议使用：

```text
current-task
```

Codex 的触发形式是：

```text
$current-task
```

其他平台则使用同名命令入口：

```text
current-task
```

### 7.2 Skill 职责

skill 的职责只包含：

1. 获取当前任务
2. 列出可切换任务
3. 获取用户选择
4. 校验所选任务是否存在
5. 调用 `current_task.py switch`
6. 必要时提示是否需要 `init-context`
7. 检查所选任务是否已是当前任务，若是则只提示不切换

### 7.3 输入解析规则

输入解析建议采用“列表优先、明确选择、路径直达”的规则。

| 输入形式 | 解析结果 | 说明 |
|---|---|---|
| `current-task` | 列出任务并引导选择 | 无参数时不报错，先展示列表 |
| `current-task list` | 列出任务 | 显式只读模式 |
| `current-task switch` | 列出任务并引导选择 | 兼容写法，等同于无参数调用 |
| `current-task <task>` | 切换到目标任务 | 仅在用户已明确选定任务后使用 |
| `<task>` = `03-25-my-task` | exact match | 精确目录名 |
| `<task>` = `.spec-first/tasks/03-25-my-task` | 相对路径 | 直接解析为仓库内路径 |
| `<task>` = `/abs/path/to/task` | 绝对路径 | 仅在路径确实指向任务目录时接受 |
| 选择项 = 当前任务 | 提示“已是当前任务” | 不重复写入 `.current-task` |
| `my-task` 这样的短名 | 仅作为列表中的可选展示项 | 先由用户在列表中明确选中，再切换 |

### 7.4 无参数行为

无参数调用时，默认行为不是报错，而是：

1. 列出活跃任务
2. 标记当前任务
3. 提示用户从列表中选择要切换的任务，并在选定后执行 `current_task.py switch`
4. 如果选中的任务就是当前任务，提示“已是当前任务”并结束

这样更符合“先看列表，再切换”的使用习惯。

## 8. 执行流程

```text
用户输入 $current-task
        │
        ▼
无参数？
        ├─ 是 → 列出任务并引导选择
        └─ 否 → 用户明确选择任务
        │
        ▼
读取当前任务 (.current-task)
        │
        ▼
用户选择任务
        │
        ├─ 不存在 → 提示错误并退出
        ├─ task.json 缺失 / 不可读 → 提示错误并退出
        │
        ├─ 已存在但无上下文 → 提示先 init-context
        │
        ├─ 当前任务有未提交变更 / 未收尾工作 → 轻量提醒先 commit 或 add_session，但不阻断切换
        │
        ▼
        执行 current_task.py switch <selection>
        │
        ▼
        写入 .spec-first/.current-task
        │
        ▼
后续 get_context / session hook 读取新任务上下文
```

## 9. 关键边界

### 9.1 只切指针，不切数据

这个 skill 不应该修改：

- `task.json`
- `prd.md`
- `implement.jsonl`
- `check.jsonl`
- `debug.jsonl`
- `spec/`

### 9.2 不替代 `finish`

如果当前用户希望显式结束当前任务，仍然应该执行：

```bash
python3 ./.spec-first/scripts/task.py finish
```

`current-task` 只负责切换，不负责生命周期收尾。

### 9.3 不替代 `init-context`

如果目标任务是新任务，skill 可以提示：

- 先初始化上下文
- 再切换当前任务

但不建议在 skill 内默认自动执行。

### 9.4 不忽略任务元数据

切换前至少要检查：

- `task.json` 是否存在
- `task.json` 是否可读
- 当前任务是否仍有未完成工作

如果 `task.json` 缺失或损坏，直接提示用户修复任务目录，不要静默切换。

---

## 10. 风险与对策

### 风险 1：切到一个还没初始化上下文的任务

对策：

- skill 提示用户先跑 `task init-context`
- 不自动补文件

### 风险 2：用户误以为 skill 会创建任务

对策：

- 文案明确写“只切换当前任务”
- 帮助信息里列出前置命令

### 风险 3：多个任务目录名接近，用户选错

对策：

- skill 在切换前展示任务全路径
- 必要时先让用户确认一次

---

## 11. 测试建议

建议至少覆盖这些场景：

1. `current-task` 无参数时列出任务并引导选择
2. `current-task list` 正常列出任务
3. `current-task switch` 无参数时等同于 `current-task`
4. 在列表中选择一个 exact 任务名后，能成功切换
5. 在列表中选择一个相对路径后，能成功切换
6. 在列表中选择一个绝对路径后，能成功切换
7. 当前已是目标任务时，给出“已是当前任务”提示或静默成功
8. 目标任务不存在时失败退出
9. 目标任务存在但缺少 `init-context` 时输出提醒
10. 当前任务有未提交变更时输出轻量提醒，不阻断切换
11. `task.json` 缺失或损坏时拒绝切换
12. 切换后 `.current-task` 更新为最后一次选择
13. 切换后 `get_context.py` 能读到新的当前任务

## 12. 落地建议

推荐的实现位置：

- `packages/cli/src/templates/codex/skills/current-task/SKILL.md`
- `packages/cli/src/templates/spec-first/scripts/current_task.py`
- 如需平台同步，再补各平台对应入口文案

推荐的实现策略：

1. 先做统一的 `current_task.py`
2. 平台入口统一调用这个脚本
3. `task.py start` 保持为底层原语，不承担交互选择

### 12.1 Breaking Change 与迁移说明

这是一次有意的接口收口，不是回滚到旧的模糊匹配行为。

变化点：

- 用户级任务切换入口从 `task.py start` 收敛到 `current_task.py switch`
- `task.py start` 不再接受短名后缀匹配，例如 `my-task -> 03-25-my-task`
- `current-task switch` 无参数时按列表展示处理，不再报错

影响：

- 现有自动化脚本如果依赖 `task.py start my-task` 这类短名调用，需要迁移
- 精确目录名或仓库内精确路径仍然可以作为兼容输入

迁移建议：

```bash
# 旧写法（已不推荐）
python3 ./.spec-first/scripts/task.py start my-task

# 新写法：先列出任务，再明确选择
python3 ./.spec-first/scripts/current_task.py list
python3 ./.spec-first/scripts/current_task.py switch <selection>

# 或者直接使用精确目录
python3 ./.spec-first/scripts/task.py start .spec-first/tasks/03-25-my-task
```

## 13. 多平台安装策略

如果这个能力要真正覆盖多个端，安装层应该支持平台分发，但**核心语义不能分裂**。

### 13.1 统一语义

所有平台都应该围绕同一个概念：

```text
current-task
```

也就是：

- `current-task list`
- `current-task` 先列出任务并引导选择

Codex 的触发前缀 `$` 只是技能调用语法，不是新的命名体系。

核心行为仍然只依赖：

- `python3 ./.spec-first/scripts/current_task.py list`
- `python3 ./.spec-first/scripts/current_task.py switch <selection>`

### 13.2 平台适配

不同平台只负责“入口形式”不同，不负责改变任务切换规则。

推荐映射：

- **Codex**: `packages/cli/src/templates/codex/skills/current-task/SKILL.md`
- **Claude Code**: `.claude/commands/spec/current-task.md`
- **Cursor**: `.cursor/commands/spec-current-task.md` (Cursor 不支持子目录，使用 `spec-` 前缀)
- **iFlow**: `.iflow/commands/spec/current-task.md`
- **Shared script**: `packages/cli/src/templates/spec-first/scripts/current_task.py`

如果某个平台更适合命令式入口，就做命令；如果更适合 skill，就做 skill。  
但所有入口都应调用同一套任务切换逻辑。

### 13.3 安装原则

安装脚本需要做到三件事：

1. 安装平台入口文件
2. 保留共享语义文案
3. 不在平台层重复实现任务状态机

换句话说：

- **平台层负责“怎么触发”**
- **spec-first scripts 负责“切换什么”**

### 13.4 分阶段落地

建议分两步：

1. 先支持 Codex
2. 再同步 Claude / Cursor / 其他平台

原因很简单：

- Codex 已经是 skill 机制，最适合先验证
- Claude / Cursor / iFlow 入口形式不同，需要分别适配
- 先把核心语义跑通，再扩展平台，风险最低

---

## 14. 结论

如果目标是“让用户更自然地切换任务”，最稳妥的方案不是新增复杂状态机，而是：

- 新增一个 `current-task` skill
- skill 只调用统一的 `current_task.py`
- 任务上下文仍然由 `.current-task` 驱动

这样可以把“任务切换”收敛成一个非常薄的语义层，而不破坏现有工作流。

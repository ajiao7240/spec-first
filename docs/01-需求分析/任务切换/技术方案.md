# 任务切换 Skill 技术方案

## 1. 背景

当前 spec-first 已经有完整的任务体系：

- `task create` 创建任务目录
- `task init-context` 生成实现 / 检查 / 调试上下文
- `task start` 切换当前任务指针 `.spec-first/.current-task`
- `get_context.py` 和各平台 hook 根据当前任务指针恢复上下文

问题在于，切换任务这件事对使用者来说仍然是“记命令”，而不是“语义化入口”。

因此需要新增一个 skill，把“切换任务”封装成更自然的用户操作。

---

## 2. 目标

新增一个 `switch-task` skill，用于：

- 让用户用 skill 语义切换当前任务
- 只修改当前任务锚点，不改写任务内容
- 复用现有 `task.py start` 作为唯一事实来源
- 保持行为可预测、低风险、易回滚

---

## 3. 非目标

这个 skill 不负责：

- 新建任务
- 自动写 PRD
- 自动生成 `implement.jsonl` / `check.jsonl` / `debug.jsonl`
- 自动归档旧任务
- 自动修改 `spec/`

如果需要完整任务初始化，仍然应该显式执行：

```bash
python3 ./.spec-first/scripts/task.py init-context <task> <dev_type>
python3 ./.spec-first/scripts/task.py start <task>
```

---

## 4. 现状判断

### 4.1 切换任务的真实锚点

当前任务切换的唯一状态源是：

```text
.spec-first/.current-task
```

`task.py start` 会把任务目录写进去。
后续的上下文恢复、会话注入、任务显示，都会读取这个锚点。

### 4.2 上下文来源

后续 skill 或 session hook 读取任务时，依赖的是：

- `.spec-first/.current-task`
- 当前任务目录下的 `task.json`
- `prd.md`
- `implement.jsonl`
- `check.jsonl`
- `debug.jsonl`

也就是说，**skill 不需要发明新的状态存储**，只要切换当前任务指针即可。

---

## 5. 方案对比

### 方案 A：只包装 `task.py start`

skill 只做一件事：

```bash
python3 ./.spec-first/scripts/task.py start <task>
```

优点：

- 最简单
- 风险最低
- 与现有机制完全一致
- 不会引入重复状态

缺点：

- 如果目标任务还没有上下文，用户仍要手动补 `init-context`

### 方案 B：切换前先检查，再切指针

skill 先读取当前任务和目标任务，再决定是否提示用户：

- 当前任务是否需要先 `finish`
- 目标任务是否存在
- 目标任务是否已经有上下文

优点：

- 交互更友好
- 减少误切

缺点：

- 逻辑比方案 A 多一层
- skill 变成“半流程编排”，维护成本上升

### 方案 C：切换 + 自动补上下文

skill 切换任务前自动执行 `init-context`。

优点：

- 一次性把体验做完整

缺点：

- 会修改更多文件
- 容易让“切换任务”和“初始化任务”概念混淆
- 风险明显高于前两种

---

## 6. 推荐方案

推荐采用 **方案 A + 轻量提示**。

核心原则：

- skill 只负责“切当前任务”
- `task.py start` 是唯一执行入口
- 如果目标任务上下文不完整，skill 只提示，不自动改写

这样可以保证：

- 切换行为稳定
- 不会误覆盖任务数据
- 与现有 `get_context` / session hook 机制完全兼容

---

## 7. Skill 设计

### 7.1 Skill 名称

建议命名为：

```text
switch-task
```

调用方式建议统一为：

```text
$switch-task
```

### 7.2 Skill 职责

skill 的职责只包含：

1. 获取当前任务
2. 解析目标任务
3. 校验目标任务是否存在
4. 调用 `task.py start`
5. 必要时提示是否需要 `init-context`

### 7.3 输入形式

推荐两种输入方式：

- 任务目录名
- 任务相对路径

例如：

```text
$switch-task my-task
$switch-task .spec-first/tasks/03-25-my-task
```

---

## 8. 执行流程

```text
用户输入 $switch-task
        │
        ▼
读取当前任务 (.current-task)
        │
        ▼
识别目标任务
        │
        ├─ 不存在 → 提示错误并退出
        │
        ├─ 已存在但无上下文 → 提示先 init-context
        │
        ▼
执行 task.py start <target>
        │
        ▼
写入 .spec-first/.current-task
        │
        ▼
后续 get_context / session hook 读取新任务上下文
```

ASCII 简图：

```text
┌──────────────────────┐
│   $switch-task       │
└─────────┬────────────┘
          │
          ▼
┌──────────────────────┐
│ 解析目标任务         │
└─────────┬────────────┘
          │
          ├── 不存在 → 失败退出
          │
          ▼
┌──────────────────────┐
│ 可选：提示 init-context│
└─────────┬────────────┘
          │
          ▼
┌──────────────────────┐
│ task.py start <task>  │
└─────────┬────────────┘
          │
          ▼
┌──────────────────────┐
│ 更新 .current-task   │
└─────────┬────────────┘
          │
          ▼
┌──────────────────────┐
│ 后续上下文自动切换   │
└──────────────────────┘
```

## 9. 关键边界

### 9.1 只切指针，不切数据

这个 skill 不应该修改：

- `task.json`
- `prd.md`
- `implement.jsonl`
- `check.jsonl`
- `debug.jsonl`
- `spec/`

### 9.2 不替代 `finish`

如果当前用户希望显式结束当前任务，仍然应该执行：

```bash
python3 ./.spec-first/scripts/task.py finish
```

`switch-task` 只负责切换，不负责生命周期收尾。

### 9.3 不替代 `init-context`

如果目标任务是新任务，skill 可以提示：

- 先初始化上下文
- 再切换当前任务

但不建议在 skill 内默认自动执行。

---

## 10. 风险与对策

### 风险 1：切到一个还没初始化上下文的任务

对策：

- skill 提示用户先跑 `task init-context`
- 不自动补文件

### 风险 2：用户误以为 skill 会创建任务

对策：

- 文案明确写“只切换当前任务”
- 帮助信息里列出前置命令

### 风险 3：多个任务目录名接近，用户选错

对策：

- skill 在切换前展示任务全路径
- 必要时先让用户确认一次

---

## 11. 测试建议

建议至少覆盖这些场景：

1. 当前任务切到另一个已存在任务
2. 目标任务不存在
3. 目标任务存在，但没有 `init-context`
4. 连续切换两个任务，确认 `.current-task` 始终是最后一次选择
5. 切换后 `get_context.py` 能读到新的当前任务

---

## 12. 落地建议

推荐的实现位置：

- `packages/cli/src/templates/codex/skills/switch-task/SKILL.md`
- 如需平台同步，再补各平台对应入口文案

推荐的实现策略：

1. 先只做 Codex skill
2. skill 内只包装 `task.py start`
3. 成熟后再考虑同步到 Claude / OpenCode / iFlow 的对应命令体系

---

## 13. 多平台安装策略

如果这个能力要真正覆盖多个端，安装层应该支持平台分发，但**核心语义不能分裂**。

### 13.1 统一语义

所有平台都应该围绕同一个概念：

```text
current-task
```

也就是：

- `current-task list`
- `current-task switch <task>`

核心行为仍然只依赖：

- `python3 ./.spec-first/scripts/task.py list`
- `python3 ./.spec-first/scripts/task.py start <task>`

### 13.2 平台适配

不同平台只负责“入口形式”不同，不负责改变任务切换规则。

推荐映射：

- **Codex**: `packages/cli/src/templates/codex/skills/current-task/SKILL.md`
- **Claude Code**: `.claude/commands/spec/current-task.md`
- **Cursor**: `.cursor/commands/spec-current-task.md`
- **iFlow**: `.iflow/commands/spec/current-task.md`

如果某个平台更适合命令式入口，就做命令；如果更适合 skill，就做 skill。  
但所有入口都应调用同一套任务切换逻辑。

### 13.3 安装原则

安装脚本需要做到三件事：

1. 安装平台入口文件
2. 保留共享语义文案
3. 不在平台层重复实现任务状态机

换句话说：

- **平台层负责“怎么触发”**
- **spec-first scripts 负责“切换什么”**

### 13.4 分阶段落地

建议分两步：

1. 先支持 Codex
2. 再同步 Claude / Cursor / 其他平台

原因很简单：

- Codex 已经是 skill 机制，最适合先验证
- Claude / Cursor / iFlow 入口形式不同，需要分别适配
- 先把核心语义跑通，再扩展平台，风险最低

---

## 14. 结论

如果目标是“让用户更自然地切换任务”，最稳妥的方案不是新增复杂状态机，而是：

- 新增一个 `switch-task` skill
- skill 只调用 `task.py start`
- 任务上下文仍然由 `.current-task` 驱动

这样可以把“任务切换”收敛成一个非常薄的语义层，而不破坏现有工作流。

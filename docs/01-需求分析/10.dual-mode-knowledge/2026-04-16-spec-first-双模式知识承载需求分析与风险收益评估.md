# spec-first 双模式知识承载需求分析与风险收益评估

> 文档性质：需求分析稿
> 撰写日期：2026-04-16
> 适用范围：`spec-first init`、`spec-graph-bootstrap`、`spec-graph-bootstrap`、`context-routing`、`plan/work/review` 消费链路
> 当前阶段：需求收敛与架构判断，不直接作为开发实施手册

---

## 1. 需求背景

`spec-first` 当前已经具备较强的 repo 内长期文档产出能力：

- repo 内长期文档默认落在 `docs/contexts/<slug>/`
- 控制面产物默认落在 `.spec-first/workflows/bootstrap/<slug>/`
- workspace 场景已有最小聚合能力，但仍以 repo 为主承载单位

这一模式在单仓或单项目协作下工作良好，但在以下场景中开始暴露上限：

1. `monorepo`
2. `workspace`
3. “一个 workspace 目录下挂多个独立 git repo”的 polyrepo 微服务工作区

尤其在 polyrepo 微服务模式下，一个需求往往需要联动多个工程代码，长期知识分散在多个代码仓库内，会带来：

- 跨工程上下文割裂
- AI 读取入口不稳定
- 团队协作时文档真源分散
- repo 级知识与 workspace 级知识边界不清
- 多工程联动场景下的 review / verification / ownership 难以形成稳定知识层

因此，需要对 `spec-first` 的知识承载模型进行升级。

---

## 2. 需求目标

本次需求不是用外挂知识库替代现有模式，而是让 `spec-first` 同时支持两种知识承载模式：

### 2.1 模式 A：项目内文档模式

不传入外部知识库 git 地址时，系统工作在 `项目内文档` 模式：

- repo 级长期文档写入当前代码仓库
- 当前代码仓库既承载代码，也承载 durable knowledge

这里强调的是**产品模式保留**，不是要求实现层必须兼容当前旧路径、旧 state 或旧测试 contract。

### 2.2 模式 B：外挂知识库模式

在 `spec-first init` 阶段传入一个外部知识库 git 地址时：

- `spec-first` 自动将当前代码 repo 注册进该知识库项目
- repo 级长期文档也迁移到外挂知识库作为主真源
- 当前代码仓库不再作为这类长期文档的正式 durable source of truth
- 哪些微服务工程放入同一个知识库 git 项目，由用户自行决策

这意味着外挂模式不是“补充一个 docs repo”，而是增加一个正式的外部 knowledge backend。

---

## 3. 需求边界

### 3.1 本次明确要解决的问题

1. 支持 `项目内文档` 与 `外挂知识库` 两种模式并存
2. 由 `init` 参数决定当前项目采用哪种模式
3. 外挂模式下自动注册当前 repo 到知识库项目
4. 外挂模式下 repo 级长期文档也落到外挂知识库
5. 允许用户自行决定多个微服务 repo 的归组关系
6. 让后续 `bootstrap / plan / work / review` 可以稳定读取正确知识源

### 3.2 本次暂不直接承诺解决的问题

1. 自动发现组织内所有 repo 并自动归组
2. 自动迁移历史全部文档并清洗冲突
3. 跨知识库的全局搜索与统一索引
4. 复杂权限平台、在线文档平台或中心化 web UI
5. 生产级知识同步守护进程

---

## 4. 当前理解下的正确问题定义

这次需求的本质，不是“多加一个 git 地址参数”，而是：

> 让 `spec-first` 的 durable knowledge 存储层，从单一的 repo-local 模式，升级为可切换的双模式知识承载架构。

更准确地说，是引入一个统一的知识承载抽象层：

- 上层 workflow 只关心知识真源在哪里、如何读、如何写、如何校验
- 底层可以落在：
  - 当前代码 repo 内
  - 外挂 knowledge repo

如果没有这个抽象层，而只是临时给 `init` 增加 `--knowledge-git` 一类参数，后续会快速出现：

- 读写路径分叉
- 真源不一致
- 双写与漂移
- 迁移和回滚缺乏 contract

---

## 5. 三类场景下的价值判断

### 5.1 monorepo

收益存在，但不是最高。

原因：

- monorepo 本身已经天然集中
- 外挂知识库的主要价值在于知识与代码解耦，而不是跨 repo 聚合
- 若团队协作文档很多、非研发角色需要直接参与，外挂模式仍有价值

### 5.2 workspace

收益中高。

原因：

- workspace 是天然的协作单元
- 但当前 workspace 通常只是目录聚合，不天然具备稳定知识控制面
- 外挂知识库适合作为 workspace 级 durable knowledge 层

### 5.3 polyrepo 微服务工作区

收益最高。

原因：

- 一个需求经常联动多个独立 git repo
- 跨服务知识、边界、owner、验证路径并不天然存在于单一代码仓库中
- AI 在该场景下最缺的不是“看代码”，而是“看稳定的跨工程知识层”

结论：

> 该需求的最高价值场景是 polyrepo 微服务工作区，其次是 workspace，最后才是 monorepo。

---

## 6. 预期收益分析

### 6.1 对 AI workflow 的收益

1. 为 `bootstrap / plan / work / review` 提供更稳定的知识真源
2. 降低 AI 在多 repo 场景下的盲搜与误读概率
3. 提高 cross-repo context assembly 的稳定性
4. 为后续 ownership、routing、verification、workspace-level context 提供更合理的 durable knowledge 落点

### 6.2 对团队协作的收益

1. 把知识资产从业务代码发布节奏中解耦
2. 允许多个工程围绕同一个 knowledge repo 协作
3. 让 TL、架构师、QA、产品等角色更容易参与文档维护
4. 支持按业务域、服务群、团队边界自行归组

### 6.3 对知识治理的收益

1. repo 级知识与 workspace 级知识可以放在同一控制平面治理
2. 更容易沉淀 service boundary、cross-repo flow、共享术语、验证矩阵
3. 为后续增量更新、刷新、质量校验提供统一落点

### 6.4 对产品形态的收益

1. `spec-first` 不再只是“在代码仓库内生成文档”的工具
2. 而是具备“可切换知识后端”的长期架构潜力
3. 这使它更适合作为团队级 AI 研发效能基础设施

---

## 7. 核心风险分析

### 7.1 真源复杂化风险

这是最大的系统性风险。

一旦同时支持两种模式：

- `项目内文档`
- `外挂知识库`

就必须严肃回答：

1. 当前项目处于哪种模式
2. 当前 durable knowledge 的唯一真源是哪一处
3. 哪些路径是主写入路径
4. 哪些路径只是兼容读取或导出层

若没有明确 contract，后果会是：

- 双写
- 文档漂移
- agent 读错源
- 路径分叉不可维护

### 7.2 生命周期治理风险

外挂模式下，不能只考虑 `init` 成功，还必须考虑长期演进：

1. repo 重命名怎么办
2. repo 从一个知识库组迁到另一个组怎么办
3. knowledge repo 地址变更怎么办
4. 绑定关系损坏如何修复
5. 当前 repo 退出外挂模式后如何回退

若这些动作无正式 contract，后续会出现脏绑定和孤儿目录。

### 7.3 多人协作合并冲突风险

共享 knowledge repo 之后，会出现更高频的文档协作与 merge 冲突：

1. 多个 repo 同时改 workspace 级索引
2. 自动生成文档覆盖人工整理文档
3. 高频更新导致 PR 噪音和 review 负担上升
4. agent 在不清楚边界时修改顶层目录与索引

### 7.4 默认体验变差风险

如果外挂模式设计过重，会影响 `spec-first init` 的首次成功路径：

1. 初始化时 clone / pull / register / bootstrap / migrate 串得过长
2. 失败路径复杂，用户难理解
3. 未使用外挂模式的用户也被迫感知复杂度

### 7.5 代码仓自解释性下降风险

外挂模式下 repo 级长期文档迁出后，代码仓本身的“自解释性”会下降。

如果不保留明确的指针和绑定元数据，维护者只看代码仓时会不知道：

- 长期文档在哪
- 当前 repo 归属哪个 knowledge repo
- 后续应该去哪里读和更新 durable knowledge

### 7.6 权限与边界风险

多个微服务可以进入同一个 knowledge repo，但不代表它们天然适合共享全部知识。

如果知识归组不当，会带来：

- 跨团队可见范围扩大
- 不同业务域知识混淆
- 评审和 ownership 边界不清

因此归组权必须交给用户，系统只做承载，不做隐式组织推断。

---

## 8. 总体架构判断

结合当前 `spec-first` 能力与团队场景，这种模式的总体判断是：

> 值得做，但必须作为正式的知识承载架构能力来设计，不能当成一个临时附加参数。

原因有三：

1. 对 polyrepo 微服务团队是高收益方向
2. 与 `spec-first` 正在推进的 workspace / cross-repo / spec-graph-bootstrap 能力高度协同
3. 若设计草率，会直接引入长期不可维护的真源复杂度

因此，正确定位应当是：

> `spec-first` 的双模式 durable knowledge backend 能力。

而不是：

- “文档仓库小增强”
- “init 多一个 git 参数”
- “简单独立 docs repo 支持”

---

## 9. 正确设计必须守住的原则

### 9.1 保留 `in_repo` 作为正式模式，但不要求旧实现兼容

不传知识库 git 地址时，系统必须继续支持 `项目内文档` 这一正式模式。

但当前仍处于开发阶段，因此不需要把“兼容当前旧路径实现、旧 state 结构、旧测试断言”作为约束。可以直接以新架构替换旧实现。

### 9.2 外挂模式单一真源

外挂模式下必须有唯一 durable knowledge 真源，禁止“外挂 repo + 代码 repo”同时作为正式主写入源。

### 9.3 绑定关系 machine-readable

当前代码 repo 与 knowledge repo 的绑定关系必须有结构化记录，不能靠目录扫描、人工约定或隐式推断。

### 9.4 自动注册必须可审计

`init` 自动注册当前 repo 到 knowledge repo 时，必须能追踪：

- 当前 repo 身份
- 绑定时间
- 归属 knowledge repo
- 在 knowledge repo 中的目录落点

### 9.5 写入边界必须分层

不同级别文档必须有明确边界：

- repo 级知识
- workspace / cross-repo 级知识
- 全局共享知识

不能让任意 agent 自由修改顶层索引和全局 registry。

### 9.6 必须支持回退与迁移

任何正式模式都必须考虑：

- 从项目内模式切到外挂模式
- 从外挂模式回退
- 从知识库 A 迁到知识库 B

否则后续无法稳定推广。

---

## 10. 与当前实现的关系判断

从当前代码与方案看，已有能力主要是：

1. repo 内 durable docs 路径已经稳定
2. workspace 最小聚合已经存在
3. cross-repo 主链正在按 `docs/plans/` 系列方案推进

但当前还没有真正具备：

1. 统一的 knowledge backend 抽象
2. 外挂知识库绑定 contract
3. repo 自动注册与目录骨架初始化
4. 双模式下统一的读写 resolver

因此，这个需求不是一个局部补丁，而是一个明确的产品/架构扩展项。

---

## 11. 不应采取的错误路线

以下路线看似省事，实际上会造成长期问题：

### 11.1 只加一个 `--knowledge-git` 参数

问题：

- 没有知识模式抽象
- 没有真源边界
- 没有后续消费协议

### 11.2 靠扫描父目录自动发现所有 repo 并自动归组

问题：

- 组织边界不可推断
- 权限边界不可推断
- 误归组成本很高

### 11.3 外挂模式下继续双写 repo 内文档

问题：

- 真源漂移
- agent 读写混乱
- 协作不可控

### 11.4 把所有知识、缓存、运行时产物都塞进外挂 knowledge repo

问题：

- durable / generated / transient 混在一起
- 文档仓库失控
- 维护成本快速上升

---

## 12. 当前阶段的合理结论

在当前阶段，最合理的结论不是“立即实现所有能力”，而是：

1. 先确认双模式知识承载作为正式需求成立
2. 再输出一份完整的架构设计方案
3. 设计中重点回答：
   - 模式定义
   - 真源边界
   - `init` 行为
   - knowledge repo 目录结构
   - 绑定 contract
   - 读写 resolver
   - 团队协作与 merge 策略
   - 迁移 / 回退机制

一句话总结：

> 这不是“是否支持独立文档仓库”的问题，而是 `spec-first` 是否要升级为支持可切换知识后端的团队级 AI 研发效能基础设施。

对当前团队场景，尤其是 polyrepo 微服务协作现实，这个方向是合理且高价值的；但必须以强约束架构来落地，不能以松散约定方式推进。

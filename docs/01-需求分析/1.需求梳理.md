# AI辅助工作框架需求文档

## 1. 文档概述

### 1.1 文档名称

AI辅助工作框架需求文档

### 1.2 编写背景

本次建设工作的核心目标，不是挑选单一AI工具进行零散使用，而是围绕统一方法论搭建一套可落地、可扩展、可沉淀的AI辅助工作框架。该框架旨在将AI能力纳入标准化工作流和工程化体系中，使AI从“聊天工具”升级为“可管理、可复用、可审查、可持续优化”的工作支撑系统。

本框架以 **Harness Engineering** 为指导思想。Harness 的核心精神不是单纯提升 AI 使用效率，而是推动软件交付方式升级：

**让软件交付从脆弱、手工、割裂，走向智能、自动、可控。**

因此，本框架服务的对象不是单次零散任务，而是围绕需求、研发、测试协同链路的软件交付过程，通过统一方法论、标准流程、角色协同、质量校验与资产沉淀，重塑软件交付方式。

### 1.3 建设目标

本框架的建设目标如下：

1. 一期围绕 Claude 建设完整的工具支撑体系，形成可复制的落地样板；
2. 将 AI 能力嵌入需求、研发、测试协同链路，提升软件交付连续性；
3. 降低需求理解、任务拆解、执行落地、质量校验过程中的脆弱性和手工性；
4. 通过流程标准、状态控制、人工兜底和资产沉淀提升整体交付可控性；
5. 在框架母体不变的前提下，后续逐步扩展到 Codex 等其他模型的工具支撑。

### 1.4 建设原则

本框架遵循以下原则：

* **方法论先行**：优先统一软件交付协作方式，而非优先堆叠工具；
* **一期收敛**：首期仅聚焦 Claude 工具支撑，不做多模型协同编排；
* **交付导向**：框架服务于需求—研发—测试的交付主链路；
* **流程闭环**：任务尽量纳入统一闭环流程管理；
* **标准优先**：明确输入输出、状态流转、质量标准、沉淀规则和治理边界；
* **人机协同**：AI增强交付效率与质量，关键决策和高风险动作保留人工把关；
* **资产沉淀**：每轮协作产物都应尽可能反哺为可复用资产。

## 2. 框架定位

### 2.1 核心定位

本项目要建设的不是提示词集合，也不是单一AI工具使用规范，而是一套**面向软件交付全流程的 AI 工具支撑框架**。该框架以工作流为主线，以规范为约束，以质量可控为前提，以资产沉淀为长期价值来源，以 Claude 工具支撑为一期核心建设内容。

### 2.2 一期定位

一期定位为：

**围绕 Claude 建设一套完整、可复用、可管理的工具支撑框架。**

重点不是支持多个模型同时协作，而是先将 Claude 在不同交付场景下的使用方式、阶段职责、输入输出规范、质量校验方式和资产沉淀路径建立清楚，形成稳定的方法样板。

### 2.3 后续扩展定位

后续扩展方向为：

* 在框架母体不变的前提下，逐步扩展到 Codex 等其他模型；
* 每新增一个模型，补充该模型的工具支撑规范；
* 扩展方式为“单模型支撑能力复制”，而非“多模型混合编排协作”。

## 3. 指导思想：Harness Engineering

### 3.1 核心理念

Harness Engineering 的核心，不是研究如何写更复杂的提示词，而是通过工程化方式驾驭AI，让AI在边界清晰、流程清晰、标准清晰、状态可控的体系中稳定发挥作用。

### 3.2 在本框架中的落地要求

在本框架中，Harness Engineering 体现为：

1. 明确 AI 在不同阶段承担的角色与职责；
2. 明确不同任务进入 AI 协作流程的条件；
3. 明确每一阶段的输入输出规范与状态切换规则；
4. 明确质量审查、失败处理与人工兜底规则；
5. 明确哪些成果需要沉淀、如何沉淀、谁负责维护；
6. 通过模板化、规则化、可追踪化降低 AI 输出随机性；
7. 通过反馈回路与资产积累持续提升整体交付效率和质量。

### 3.3 Harness 核心目标映射

结合软件交付场景，Harness 的目标拆解如下：

* **脆弱 → 稳定**：通过标准输入输出、阶段门禁、review 机制和人工兜底，降低交付波动；
* **手工 → 自动**：通过模板、自动生成、自动检查、自动推荐资产降低重复劳动；
* **割裂 → 连续**：通过产品、研发、测试三角色协同打通需求到交付链路；
* **智能、自动、可控**：让 AI 在受控系统中持续增强软件交付能力。

## 4. 一期建设范围

### 4.1 一期核心范围

一期仅围绕 Claude 的工具支撑展开建设，重点包括：

* Claude 在各工作流阶段的职责边界定义；
* Claude 在不同任务场景下的输入输出模板；
* Claude 的任务使用规范、状态流转规则与结果校验机制；
* Claude 协作成果的知识沉淀与回灌机制；
* 一期重点角色的使用适配；
* 基础治理、日志留痕与人工兜底机制。

### 4.2 一期不包含内容

一期暂不纳入以下内容：

* 多模型协同编排机制；
* Claude 与 Codex 等模型的联动调度；
* 运维、运营等角色的深度适配；
* 高复杂度自动代理链路；
* 无人工干预的高风险智能决策替代机制。

### 4.3 一期重点角色

考虑落地效率、协同链路完整性和质量闭环，一期优先覆盖以下角色：

* 产品人员
* 研发人员（含后端开发、前端开发、全栈开发人员）
* 测试人员

其中：

* **产品人员**负责需求发起、方案定义、业务目标确认与结果验收；
* **研发人员**负责技术方案分析、执行落地与技术问题处理，其中全栈开发人员还承担需求讨论、需求输出、前后端衔接、集成适配、技术联动等职责；
* **测试人员**负责质量校验、缺陷识别、场景补充与风险把关。

### 4.4 一期重点任务场景

一期优先覆盖以下任务场景：

1. 产品方案发散、需求整理、PRD/MRD辅助撰写；
2. 技术方案分析、研发任务拆解、代码相关说明与问题分析；
3. 前后端协同设计、接口衔接分析、全栈开发相关任务支持；
4. 测试方案设计、测试计划制定、测试执行支持、缺陷分析、回归测试支持；
5. 围绕需求—研发—测试链路的跨角色评审与质量闭环任务；
6. 通用的会议总结、结构化分析、知识整理等标准文本任务。

## 5. 核心工作流设计

框架采用五步闭环：

**brainstorm → plan → work → review → compound**

该流程不是僵化的顺序链条，而是一套标准阶段模型与状态流转体系。不同任务可以按复杂度裁剪，但必须明确当前所处阶段、阶段输出、完成标准、可回退路径和是否需要人工接管。

### 5.1 Brainstorm（头脑风暴）

#### 5.1.1 目标

用于思路发散、需求理解、方案探索、问题拆视角、备选路径生成。

#### 5.1.2 输入

* 问题定义
* 背景信息
* 业务目标
* 约束条件
* 已知风险或限制

#### 5.1.3 输出

* 候选方案列表
* 可行性分析初稿
* 风险点和取舍维度
* 推荐方向建议

#### 5.1.4 完成标准

* 至少形成多个可比较的备选方向；
* 输出包含取舍维度，而非仅有观点堆砌；
* 可为 plan 阶段提供明确输入。

### 5.2 Plan（规划制定）

#### 5.2.1 目标

将发散思路收敛为可执行方案，形成任务计划与实施路径。

#### 5.2.2 输入

* brainstorm 阶段产出
* 已确认的优选方案
* 资源与依赖信息
* 交付目标

#### 5.2.3 输出

* 目标定义
* 任务拆解
* 执行顺序
* 负责人建议
* 依赖与风险项
* 验收标准

#### 5.2.4 完成标准

* 任务可执行；
* 风险与依赖可识别；
* 后续 work 阶段无需重新理解任务目标。

### 5.3 Work（落地执行）

#### 5.3.1 目标

围绕既定计划完成实际产出。

#### 5.3.2 输入

* 计划单
* 模板或规范
* 相关资料与上下文

#### 5.3.3 输出

* 文档草稿
* 分析结论
* 技术说明
* 代码相关辅助内容
* 测试执行记录、缺陷分析说明、验证结论等

#### 5.3.4 完成标准

* 输出符合既定格式和目标要求；
* 可进入 review 阶段校验；
* 可追溯到对应计划项。

### 5.4 Review（复盘校验）

#### 5.4.1 目标

对 work 阶段产出进行质量校验与问题识别。

#### 5.4.2 输入

* 执行产出物
* 原始目标与验收标准
* 相关规范或评审维度

#### 5.4.3 输出

* 问题清单
* 修订建议
* 通过/返工结论
* 风险说明

#### 5.4.4 完成标准

* 明确指出关键问题与优先级；
* 给出可执行修订建议；
* 能支持人工判断是否通过。

### 5.5 Compound（资产沉淀）

#### 5.5.1 目标

将本轮协作成果沉淀为可复用资产，形成长期价值。

#### 5.5.2 输入

* 已通过 review 的成果
* 过程中的高质量模板、方法、案例
* 可抽象复用的知识内容

#### 5.5.3 输出

* 模板
* SOP
* 知识卡片
* 评审清单
* 场景案例
* 提示词规范

#### 5.5.4 完成标准

* 沉淀内容可复用；
* 可归类、可检索、可维护；
* 能在后续同类任务中直接调用。

## 6. 状态流转与回退规则

为使五步闭环从“流程列表”升级为“可控 Harness 状态机”，本框架定义如下状态规则。

### 6.1 基本流转

默认状态流转为：

**Brainstorm → Plan → Work → Review → Compound**

### 6.2 进入条件

* Brainstorm 可在目标、背景、问题定义基本清晰时进入；
* Plan 仅在 Brainstorm 已形成可比较候选方案时进入；
* Work 仅在 Plan 已形成可执行任务和验收标准时进入；
* Review 仅在 Work 已形成明确交付物时进入；
* Compound 仅接收已通过 Review 的成果。

### 6.3 回退规则

* Brainstorm 输出候选方案不足或目标不清时，不得进入 Plan；
* Plan 缺失验收标准、依赖关系或关键任务拆解时，不得进入 Work；
* Work 输出不完整或不符合格式要求时，不得进入 Review；
* Review 存在阻断问题时，必须回退到 Work；
* 任一阶段若上下文不足、目标冲突或信息缺失，可回退到上游阶段；
* 任一高风险任务在关键判断节点可升级为人工接管。

### 6.4 终止与升级

任务在以下场景可被终止或升级：

* 输入长期不完整；
* 输出连续不满足要求；
* 工具异常无法恢复；
* 涉及高风险业务判断；
* 需要人工定夺的争议性结论。

此时任务需保留状态记录和失败原因，用于后续复盘与模板优化。

## 7. Claude 一期工具支撑定位

### 7.1 Claude 的核心角色

一期中，Claude 的定位不是全能执行代理，而是以下三类核心支撑角色：

1. **Brainstorm 主引擎**：负责思路发散、视角补充、方案探索；
2. **Plan 结构化生成器**：负责任务拆解、路径设计、执行计划生成；
3. **Review 质量审查助手**：负责逻辑检查、完整性检查、问题识别与修订建议生成。

在 Work 阶段，Claude 可承担文档起草、分析整理、技术说明、测试辅助说明等工作，但在高风险环节不替代人工最终决策。

### 7.2 Claude 在五步流程中的能力映射

#### Brainstorm

* 多角度发散
* 提出假设
* 生成候选方案
* 给出取舍维度

#### Plan

* 结构化拆解任务
* 生成执行路径
* 标记风险与依赖
* 给出阶段验收标准

#### Work

* 辅助撰写文档
* 输出技术说明
* 辅助补充测试思路
* 协助整理分析材料

#### Review

* 审查逻辑完整性
* 检查缺漏
* 识别表述问题
* 辅助发现测试盲点和质量风险

#### Compound

* 提炼 SOP
* 提炼模板
* 总结复盘结论
* 输出可复用知识卡片

### 7.3 Claude 的使用边界

Claude 一期使用边界如下：

* 可用于辅助分析、规划、生成、检查、总结；
* 不直接替代正式审批、上线决策、对外发布决策；
* 涉及高风险业务判断时，必须由人工最终确认；
* review 结论可作为辅助判断依据，但不直接等同于最终验收。

## 8. 控制面与执行面设计

Harness Engineering 要求执行与控制分离，避免 AI 既做内容生成又做最终裁决。

### 8.1 控制面

控制面负责：

* 任务分级；
* 状态切换；
* 阶段规则校验；
* 是否允许进入下阶段；
* 是否需要人工确认；
* 是否允许进入 compound；
* 日志留痕与失败记录。

### 8.2 执行面

执行面负责：

* Claude 的分析、生成、检查与提炼；
* 文档草稿输出；
* 结构化计划输出；
* 测试思路和技术说明输出；
* review 建议和沉淀建议输出。

### 8.3 原则

* Claude 可以是执行器，但不应天然等于控制器；
* 是否通过、是否返工、是否沉淀、是否升级人工，必须由控制规则和人工共同约束；
* AI 输出不应直接跳过 review 或人工兜底环节。

## 9. 任务分级机制

为兼顾效率与质量，框架对任务进行分级管理。

### 9.1 按复杂度分级

#### L1 轻任务

特点：

* 单轮协作即可完成；
* 风险低；
* 输出较短。

示例：

* 会议纪要整理
* 简单文案优化
* 测试点补充
* 技术说明润色

流程要求：

* 可走 brainstorm/work/review 的简化链路。

#### L2 标准任务

特点：

* 需要完整闭环；
* 有明确目标与交付物；
* 需要 review。

示例：

* PRD 辅助撰写
* 技术方案分析
* 测试方案设计
* 缺陷分析报告

流程要求：

* 应走完整的 brainstorm → plan → work → review；
* 有价值成果进入 compound。

#### L3 复杂任务

特点：

* 多角色参与；
* 高风险或高复杂度；
* 需要多轮 review 和人工确认。

示例：

* 跨团队需求方案
* 核心模块设计说明
* 复杂测试策略设计
* 关键问题根因分析

流程要求：

* 必须走完整闭环；
* 必须人工审查；
* 可沉淀为重点案例资产。

### 9.2 按风险分级

* **低风险**：AI 可直接辅助产出，人工快速检查即可；
* **中风险**：必须经过 review 和人工确认；
* **高风险**：必须保留人工审批，不得直接采用 AI 结果。

## 10. 重点角色适配需求

### 10.1 产品人员

#### 核心定位

框架的需求主导者与业务定义者，侧重方案发散、需求梳理、计划制定、结果验收和资产沉淀。

#### 核心使用场景

* brainstorm：产品方案发散、需求挖掘、场景补全；
* plan：产品规划、需求拆解、迭代计划制定；
* work：PRD、MRD、需求说明等文档辅助撰写；
* review：需求合理性、可行性、完整性评审；
* compound：需求模板、评审清单、PRD模板沉淀。

#### 适配要求

* 界面简洁，突出方案生成、计划制定、文档辅助、评审支持等核心能力；
* 支持需求类模板快速调用；
* 支持自定义评审维度，如合理性、完整性、实现可行性；
* 支持与办公系统的后续集成扩展。

### 10.2 研发人员（含全栈开发人员）

#### 核心定位

框架的执行者与技术落地者，侧重需求理解、技术分析、任务拆解、执行辅助、问题排查和技术资产沉淀。研发人员范围包括后端开发、前端开发以及**全栈开发人员**。

其中，全栈开发人员是一类重点适配对象，尤其适用于一人公司、小团队或资源有限场景下，承担从需求讨论、方案设计、开发实现到联调验证的全流程职责，不仅参与技术实现，也深度参与需求理解与需求输出。

#### 核心使用场景

* **brainstorm**：需求讨论、业务场景理解、技术方案选型、实现路径讨论、系统衔接思路发散；
* **plan**：需求拆解、任务规划、前后端协同安排、接口设计拆分、风险点识别；
* **work**：需求说明整理、技术说明输出、代码相关说明、接口联调分析、前后端协作内容整理；
* **review**：需求与实现一致性检查、逻辑检查、接口衔接检查、实现方案检查、说明文档校验；
* **compound**：技术模板、需求分析模板、接口联调 SOP、问题案例、调试经验沉淀。

#### 对全栈开发人员的重点适配说明

全栈开发人员不仅是研发执行角色，也可能是需求承接和需求输出角色。在一人公司或小团队场景下，其职责往往覆盖：

* 参与需求讨论，梳理业务目标与场景；
* 将业务需求转化为可实现的功能需求或技术任务；
* 负责前后端整体方案设计与拆解；
* 承担开发、联调、问题排查和交付支持；
* 在缺少专职产品或专职测试时，补足需求澄清和质量校验工作。

因此，框架在适配全栈开发人员时，不能仅把其视为“技术实现者”，而要视为：

**兼顾需求、方案、开发、联调、质量校验的复合型核心角色。**

#### 通用适配要求

* 支持从需求讨论到技术实现的连续性工作流；
* 支持需求整理、需求输出与技术拆解一体化处理；
* 支持前后端联动场景下的任务拆解与协同规划；
* 支持接口设计、接口联调、页面与服务逻辑衔接等场景分析；
* 支持跨模块问题排查与结构化总结；
* 支持沉淀适用于需求—研发—测试一体化协作的模板、案例和 SOP。

### 10.3 测试人员

#### 核心定位

框架的质量把关者与风险识别者，侧重测试方案设计、测试执行、缺陷分析、回归验证与质量资产沉淀。

#### 核心使用场景

* **brainstorm**：测试场景发散、边界情况补充、异常路径补充；
* **plan**：测试计划、测试范围、测试用例设计、验证策略制定；
* **work**：执行测试用例、记录测试结果、整理缺陷描述、输出验证结论；
* **review**：AI 输出质量检查、测试覆盖性检查、兼容性与风险检查；
* **compound**：测试用例模板、测试 SOP、缺陷案例库沉淀。

#### 适配要求

* 支持测试维度自定义，如功能、兼容性、边界、异常、性能；
* 支持测试计划、测试用例和缺陷分析结构化生成；
* 支持在 work 阶段围绕测试执行输出结果、缺陷和验证结论；
* 支持在 review 阶段对其他角色产出进行质量校验；
* 支持测试资产可复用、可检索、可标签化管理。

### 10.4 跨角色协同适配

#### 协同目标

确保产品、研发、测试三类角色在同一闭环中高效协作，形成“需求定义—研发落地—测试校验”的质量闭环。

#### 协同要求

* brainstorm 阶段支持多角色共同参与问题发散；
* plan 阶段明确产品、研发、测试的职责分工；
* review 阶段支持跨角色评审，测试人员重点负责质量与缺陷视角，研发人员配合修订，产品人员确认需求匹配；
* compound 阶段形成可跨角色复用的资产；
* 治理层需支持按角色配置不同权限。

## 11. 上下文装配规范

为降低 Claude 输出波动，需要对各阶段上下文进行标准装配。

### 11.1 Brainstorm 上下文

应包括：

* 问题定义
* 背景
* 业务目标
* 已知约束
* 当前假设

### 11.2 Plan 上下文

应包括：

* Brainstorm 结论
* 已选方案
* 时间、资源与依赖约束
* 验收目标

### 11.3 Work 上下文

应包括：

* Plan 结果
* 参考资料
* 输出模板
* 输出格式要求
* 特殊边界或限制条件

### 11.4 Review 上下文

应包括：

* 原始目标
* 当前交付物
* 质量标准
* 风险边界
* 历史同类问题（如有）

### 11.5 Compound 上下文

应包括：

* 已通过的结果
* 过程中的高质量片段
* 可复用场景说明
* 标签与分类信息

### 11.6 上下文约束原则

* 上下文必须结构化提供，不允许仅凭临时口述；
* 不同阶段不得无控制地复用全部历史上下文；
* 上下文应按阶段裁剪，避免无关信息污染输出；
* 涉及敏感或高风险信息时，需经过权限与边界控制。

## 13. 质量评价体系

为了让 review 从“主观看看”升级为“可判断、可复用的标准动作”，本框架建立统一质量评价体系。

### 13.1 评价维度

所有关键产出至少从以下四个维度进行评价：

1. **正确性**：事实、逻辑、结论是否正确；
2. **完整性**：是否覆盖任务要求与关键场景；
3. **可执行性**：建议或方案是否可落地；
4. **可复用性**：是否值得沉淀为后续资产。

### 13.2 评价方式

每个维度采用 1~5 分评分，并输出：

* 总体评价
* 是否通过
* 是否需要返工
* 是否建议沉淀

### 13.3 问题等级

review 阶段问题分为：

* **阻断问题**：不修复不能通过；
* **重要问题**：建议修复后再通过；
* **优化建议**：不影响通过，但建议改进。

## 14. 异常与失败处理机制

Harness 不仅设计成功路径，也必须设计失败路径。

### 14.1 输入失败

典型场景：

* 需求不完整；
* 背景不足；
* 目标不明确；
* 上下文冲突。

处理方式：

* 返回补充输入模板；
* 不进入下一阶段；
* 记录失败类型，用于优化输入规范。

### 14.2 生成失败

典型场景：

* 输出空泛；
* 逻辑跳跃；
* 与约束冲突；
* 结构不符合要求。

处理方式：

* 允许一次受控重试；
* 重试失败后进入 review 失败态；
* 根据问题类型回退上游阶段。

### 14.3 评审失败

典型场景：

* 阻断问题过多；
* 关键缺漏；
* 与业务目标不一致；
* 无法满足交付标准。

处理方式：

* 强制返工；
* 标注失败原因；
* 不得进入 compound。

### 14.4 系统失败

典型场景：

* Claude 调用异常；
* 工具不可用；
* 上下文丢失；
* 记录失败。

处理方式：

* 人工接管；
* 留痕记录；
* 不中断任务可追踪性。

### 14.5 失败回灌

所有失败类型应进入复盘体系，用于：

* 优化输入模板；
* 优化 review checklist；
* 形成反例案例库；
* 优化阶段切换规则。

---

## 15. Compound 资产沉淀与回灌机制

### 15.1 沉淀目标

Compound 不是简单归档，而是把协作结果转化为可复用、可积累、可提升效率的交付资产。

### 15.2 资产类型

一期沉淀资产分为四类：

#### 1）Prompt 资产

* 高质量提示模板
* 角色指令模板
* Review 检查模板

#### 2）Workflow 资产

* 任务 SOP
* 标准流程模板
* 协同流程模板

#### 3）Domain 资产

* 产品知识卡
* 技术说明模板
* 测试策略清单
* 术语表

#### 4）Case 资产

* 优秀案例
* 失败案例
* 修订前后对比案例

### 15.3 沉淀门槛

并非所有任务都进入资产库，满足以下条件之一时建议沉淀：

* 同类任务重复出现 3 次以上；
* 可被跨角色复用；
* 明显提升效率或质量；
* 具有培训与示范价值。

### 15.4 资产管理要求

* 资产需有标签、场景说明、适用范围；
* 资产需支持检索与版本更新；
* 资产需明确维护责任人。

### 15.5 回灌机制

沉淀资产需反向作用于后续任务：

* 新任务开始时优先匹配相似模板与案例；
* Review 失败案例进入负面案例库；
* 高频阻断问题沉淀为 review checklist；
* 高复用模板进入默认推荐集；
* 低复用、低质量模板定期淘汰。

---

## 16. 模板治理机制

为了避免模板、SOP、案例库失控增长，需要建立模板治理机制。

### 16.1 治理目标

确保模板可识别、可维护、可迭代、可淘汰，防止资产层变成文档堆积区。

### 16.2 治理要求

* 模板应有统一命名规范；
* 模板应有版本号；
* 模板应标记适用场景和使用边界；
* 模板应明确维护责任人；
* 模板需支持效果评估与淘汰。

### 16.3 模板生命周期

模板生命周期包括：

* 创建
* 试用
* 评估
* 正式发布
* 优化迭代
* 淘汰归档

---

## 17. 治理与人工兜底机制

### 17.1 治理目标

保障框架使用过程中的可控性、可追踪性和风险可管。

### 17.2 一期治理最小集

一期治理至少应覆盖以下内容：

1. **权限分级**
   按角色区分任务查看、模板维护、资产发布等权限。

2. **日志留痕**
   留存任务输入、AI 输出、review 结论、人工处理记录。

3. **高风险动作人工确认**
   正式发布、关键需求确认、核心技术结论采纳等必须人工确认。

4. **问题回溯能力**
   重要任务需支持从结果回溯到输入、过程与评审意见。

### 17.3 人工介入节点

以下场景必须保留人工把关：

* 需求最终定稿；
* 研发关键方案采纳；
* 测试最终放行判断；
* 对外输出内容；
* 正式纳入知识库的核心资产。

### 17.4 角色-阶段-权限矩阵原则

* 产品人员可发起 brainstorm、plan，可参与 review，可申请沉淀需求资产；
* **研发人员（含全栈开发人员）可参与 brainstorm、plan、work、review、compound，可沉淀技术资产；**
* **测试人员可参与 brainstorm、plan、work、review、compound，可发起质量返工；其中 work 阶段重点包括执行测试用例、记录测试结果、提交缺陷、回归验证与输出测试结论；**
* 模板维护者或管理员可配置规则、发布模板、归档资产。

## 18. 非功能性需求

### 18.1 扩展性需求

#### 目标

保障框架可在母体不变的前提下扩展到其他模型支撑、更多角色与更多场景。

#### 要求

* 框架母体与模型支撑规范解耦；
* 后续可新增 Codex 工具支撑规范；
* 资产结构支持模板、案例、SOP、知识卡片等多种类型；
* 场景扩展时主要新增规范，不重构核心流程。

### 18.2 性能需求

#### 目标

保障框架在日常使用中响应及时、流程顺畅、检索高效。

#### 建议指标

* 轻量任务响应目标：3~5 秒；
* 中等复杂任务响应目标：10~20 秒；
* 资产检索响应目标：1~2 秒；
* 支持基础并发使用能力，并在上线后用实际数据校准指标。

说明：一期性能目标以“可用、稳定、可观测”为先，避免过早采用过于理想化的 SLA。

### 18.3 稳定性需求

#### 目标

保障框架稳定可用，避免流程中断、记录丢失、结果不可追踪。

#### 要求

* 关键流程异常时可人工接管；
* 重要记录可追溯；
* 资产支持备份与版本回溯；
* 版本迭代不破坏历史资产与配置。

### 18.4 用户使用简单

#### 目标

降低产品、研发、测试三类用户的学习和操作成本。

#### 要求

* 核心功能入口清晰；
* 操作步骤尽量控制在 3 步内完成主要动作；
* 默认配置足够可用；
* 提供流程提示、示例模板和常见问题说明；
* 让不同角色能快速找到各自核心能力入口。

### 18.5 集成方便

#### 目标

降低一期工具支撑接入和后续系统扩展成本。

#### 要求

* Claude 相关调用方式标准化；
* 后续可扩展对接办公系统与专业工具；
* 接入扩展不破坏现有流程与资产体系；
* 为后续 Codex 支撑保留扩展接口。

## 19. 可观测性指标与反馈回路

Harness 系统必须可观测，否则无法持续优化。

### 19.1 运行指标

* 任务总量
* 各阶段停留时长
* 阶段回退率
* 人工接管率
* Claude 调用失败率

### 19.2 质量指标

* Review 通过率
* 首次通过率
* 阻断问题占比
* Compound 入库率
* 入库后复用率

### 19.3 业务指标

* PRD 输出效率提升
* 研发拆解效率提升
* 测试执行与问题发现效率提升
* 跨角色协同返工率下降

### 19.4 反馈回路原则

* 指标不是展示用，而是用于驱动模板优化、状态规则优化、失败规则优化和资产淘汰；
* 高失败率场景优先治理；
* 高复用率资产优先推广；
* 高频阻断问题优先沉淀为标准检查规则。

## 20. 一期成功标准

一期是否成功，建议从以下角度判断：

1. 是否形成 Claude 的标准化使用规范；
2. 是否让产品、研发、测试三类角色能够在典型任务中稳定使用；
3. 是否跑通 brainstorm → plan → work → review → compound 的闭环；
4. 是否建立状态流转、失败处理、人工兜底和日志留痕机制；
5. 是否形成首批可复用资产与基础回灌体系。

可进一步量化为：

* 典型任务交付效率提升；
* Review 返工率下降；
* 模板与案例复用率提升；
* 一期角色的活跃使用率提升；
* 沉淀资产数量与复用次数增长。

---

## 21. 路线图建议

### 21.1 一期

聚焦 Claude 工具支撑，覆盖产品、研发、测试三类角色，打通核心闭环，建立状态控制、质量评价、失败处理和资产沉淀机制。

### 21.2 二期

在框架母体不变的前提下，扩展 Codex 工具支撑；补强资产检索、模板治理和更多角色适配。

### 21.3 三期

继续扩展更多模型支撑、系统集成、治理看板与更完善的跨角色协同机制。

---

## 22. 结论

本次要建设的，是一套以 **Harness Engineering** 为指导思想、以 **Claude** 为一期核心、以 **brainstorm → plan → work → review → compound** 为闭环、优先服务产品人员、研发人员、测试人员的 AI 辅助工作框架。研发人员中需重点支持全栈开发人员，尤其适配其在一人公司或小团队中从需求讨论到开发落地的全流程工作场景。

该框架的重点不在于多模型协作，而在于先把单模型的工具支撑体系做深、做实、做成标准样板，再在框架母体不变的基础上逐步扩展到其他模型支撑。通过状态控制、上下文装配、失败处理、质量评价、资产沉淀与反馈回路，推动软件交付从脆弱、手工、割裂，走向智能、自动、可控。

## 23. 用户最终使用模式

### 23.1 设计目标

为降低用户接入门槛、统一使用入口、强化流程约束，本框架在用户侧的最终使用模式定义为：

**通过 npm 安装 spec-first 工具包，并通过 Claude 命令行以 `/spec:xxx` 指令模式使用框架能力。**

该模式的目标是让用户不需要理解底层复杂机制，只需要通过统一安装方式和统一命令入口，即可进入标准化的 AI 辅助交付流程。

### 23.2 最终交互形态

用户最终交互形态定义如下：

#### 安装方式

用户通过命令安装框架工具：

```bash
npm i spec-first
```

或在全局安装场景下：

```bash
npm i -g spec-first
```

安装完成后，用户即可在本地环境中接入框架能力。

#### 使用入口

用户通过 **Claude 命令行** 作为主要使用入口，调用框架能力。

即：

* Claude 命令行是用户主要操作界面；
* `spec-first` 提供标准化能力封装；
* 用户通过 slash command 风格命令触发不同阶段、不同任务、不同 spec 模块。

### 23.3 核心交互模式

用户通过如下形式触发能力：

```bash
/spec:xxx
```

其中：

* `/spec:` 是统一命令前缀；
* `xxx` 表示具体 spec 类型、任务类型或流程能力；
* 命令由 Claude 命令行解析并进入对应流程节点或能力模块。

这意味着框架的用户体验不是“自由聊天式”，而是：

**受控命令入口 + 标准 spec 驱动 + Claude 执行支撑。**

### 23.4 为什么采用 spec-first 模式

采用 `spec-first` 作为最终使用模式，目的在于：

#### 1）统一入口

避免不同角色、不同任务各自使用不同提示词、不同启动方式、不同习惯，导致交付方式继续分裂。

#### 2）先有规格，再有执行

用户先通过 spec 明确任务意图、上下文和目标，再进入 Claude 的分析、生成、review、compound 过程。
这符合 Harness Engineering 中“先控制、再执行”的核心思想。

#### 3）降低随机性

通过 `/spec:xxx` 明确任务类型，可以减少用户直接自由描述带来的上下文漂移和输出不稳定。

#### 4）便于扩展

后续扩展 Codex 时，也可以延续同样的 spec-first 入口模式，而不需要重构用户习惯。

### 23.5 用户侧标准使用链路

用户最终使用链路建议定义为：

#### 第一步：安装工具

```bash
npm i spec-first
```

#### 第二步：进入 Claude 命令行环境

用户在 Claude CLI 中工作。

#### 第三步：通过 `/spec:xxx` 发起任务

例如：

```bash
/spec:brainstorm
/spec:plan
/spec:review
/spec:prd
/spec:test
/spec:fullstack
```

#### 第四步：补充上下文

用户按命令要求输入必要上下文，如：

* 需求背景
* 目标
* 限制条件
* 当前资料
* 交付预期

#### 第五步：Claude 按 spec 模式执行

Claude 在受控 spec 下完成：

* 分析
* 规划
* 输出
* 评审
* 沉淀建议

#### 第六步：结果进入后续阶段或沉淀

输出结果进入：

* 下一个流程阶段
* 人工确认
* review
* compound 资产沉淀

### 23.6 `/spec:xxx` 的分类建议

为了让框架既清晰又可扩展，建议将 `/spec:xxx` 分为三类。

#### 一）流程类 spec

用于映射五步闭环流程：

```bash
/spec:brainstorm
/spec:plan
/spec:work
/spec:review
/spec:compound
```

作用：

* 直接进入某一流程阶段；
* 强制采用该阶段的输入输出模板和规则。

#### 二）任务类 spec

用于映射实际交付任务：

```bash
/spec:prd
/spec:tech-plan
/spec:test-plan
/spec:defect
/spec:meeting
```

作用：

* 面向实际使用场景；
* 内部自动绑定对应流程模板和角色上下文。

#### 三）角色类 spec

用于映射不同角色工作模式：

```bash
/spec:pm
/spec:dev
/spec:fullstack
/spec:qa
```

作用：

* 根据角色自动加载默认上下文结构、输出格式和检查规则。

### 23.7 推荐的首批 spec

结合一期范围，建议优先支持以下 spec：

#### 流程类

* `/spec:brainstorm`
* `/spec:plan`
* `/spec:work`
* `/spec:review`
* `/spec:compound`

#### 产品类

* `/spec:prd`
* `/spec:requirement`
* `/spec:feature`

#### 研发类

* `/spec:dev`
* `/spec:fullstack`
* `/spec:tech-plan`
* `/spec:api`

#### 测试类

* `/spec:qa`
* `/spec:test-plan`
* `/spec:test-exec`
* `/spec:defect`

### 23.8 全栈开发人员使用模式

考虑到你前面强调全栈开发人员适配到一人公司 / 小团队场景，全栈角色建议支持更完整的命令模式，例如：

```bash
/spec:fullstack
/spec:fullstack-brainstorm
/spec:fullstack-plan
/spec:fullstack-review
```

其中 `/spec:fullstack` 可以作为复合入口，默认覆盖：

* 需求讨论
* 需求输出
* 技术拆解
* 前后端协同
* 接口联调
* 问题排查
* 交付复盘

这比把全栈只当普通 dev 子集更符合你的实际使用场景。

### 23.9 测试人员使用模式

测试角色建议明确区分“设计”和“执行”：

```bash
/spec:test-plan
/spec:test-exec
/spec:defect
/spec:qa-review
```

其中：

* `/spec:test-plan` 用于测试计划与测试用例设计
* `/spec:test-exec` 用于执行测试用例、记录结果、输出验证结论
* `/spec:defect` 用于缺陷描述、根因整理、回归建议
* `/spec:qa-review` 用于质量评审与覆盖性检查

这样就不会混淆“测试人员 work 是执行测试用例”这一定义。

### 23.10 用户体验要求

该最终使用模式需要满足以下体验要求：

#### 1）命令统一

所有能力统一通过 `/spec:xxx` 调用，不引入多套入口。

#### 2）低学习成本

用户只需记住少量高频 spec，即可覆盖主要工作场景。

#### 3）上下文引导

每个 spec 自动提示所需输入，避免用户自行猜测格式。

#### 4）结果结构化

每个 spec 输出应默认结构化，便于 review、回退和 compound。

#### 5）可演进

后续新增 spec 时，不破坏已有命令习惯。

### 23.11 与 Harness Engineering 的关系

该使用模式与 Harness Engineering 高度一致，原因在于：

* 它不是让用户直接自由聊天；
* 而是让用户先进入受控 spec；
* 再由 Claude 在 spec 约束下执行任务；
* 最终通过标准输出进入 review、compound 和反馈回路。

所以从 Harness 的角度看，这种模式的本质是：

**把 AI 使用方式从“即兴提问”升级为“基于规格的可控交付入口”。**

### 23.12 结论

本框架的用户最终使用模式定义为：

> 用户通过 `npm i spec-first` 安装框架工具，并通过 Claude 命令行使用 `/spec:xxx` 作为统一交互入口，基于 spec-first 模式进入受控、标准化、可复用的 AI 辅助交付流程。

这一模式将作为一期产品化落地的核心使用方式，并作为后续扩展 Codex 工具支撑时的统一用户入口。

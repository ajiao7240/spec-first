# AI辅助工作框架需求文档

## 1. 文档概述

### 1.1 文档名称

AI辅助工作框架需求文档

### 1.2 编写背景

本次建设工作的核心目标，不是挑选单一AI工具进行零散使用，而是围绕统一方法论搭建一套可落地、可扩展、可沉淀的 AI 辅助工作框架。该框架旨在将 AI 能力纳入标准化工作流和工程化体系中，使 AI 从“聊天工具”升级为“可管理、可复用、可审查、可持续优化”的工作支撑系统。

本框架以 **Harness Engineering** 为指导思想。Harness 的核心精神不是单纯提升 AI 使用效率，而是推动软件交付方式升级：

**让软件交付从脆弱、手工、割裂，走向智能、自动、可控。**

因此，本框架服务的对象不是单次零散任务，而是围绕需求、研发、测试协同链路的软件交付过程，通过统一方法论、标准流程、角色协同、质量校验与资产沉淀，重塑软件交付方式。

### 1.3 建设目标

本框架的建设目标如下：

1. 一期围绕 Claude 建设完整的工具支撑体系，形成可复制的落地样板；
2. 将 AI 能力嵌入需求、研发、测试协同链路，提升软件交付连续性；
3. 降低需求理解、任务拆解、执行落地、质量校验过程中的脆弱性和手工性；
4. 通过流程标准、状态控制、人工兜底和资产沉淀提升整体交付可控性；
5. 在框架母体不变的前提下，后续逐步扩展到 Codex 等其他模型的工具支撑。

### 1.4 建设原则

本框架遵循以下原则：

* **方法论先行**：优先统一软件交付协作方式，而非优先堆叠工具；
* **一期收敛**：首期仅聚焦 Claude 工具支撑，不做多模型协同编排；
* **交付导向**：框架服务于需求—研发—测试的交付主链路；
* **流程闭环**：任务尽量纳入统一闭环流程管理；
* **标准优先**：明确输入输出、状态流转、质量标准、沉淀规则和治理边界；
* **人机协同**：AI 增强交付效率与质量，关键决策和高风险动作保留人工把关；
* **资产沉淀**：每轮协作产物都应尽可能反哺为可复用资产。

## 2. 框架定位

### 2.1 核心定位

本项目要建设的不是提示词集合，也不是单一 AI 工具使用规范，而是一套**面向软件交付全流程的 AI 工具支撑框架**。该框架以工作流为主线，以规范为约束，以质量可控为前提，以资产沉淀为长期价值来源，以 Claude 工具支撑为一期核心建设内容。

### 2.2 一期定位

一期定位为：

**围绕 Claude 建设一套完整、可复用、可管理的工具支撑框架。**

重点不是支持多个模型同时协作，而是先将 Claude 在不同交付场景下的使用方式、阶段职责、输入输出规范、质量校验方式和资产沉淀路径建立清楚，形成稳定的方法样板。

### 2.3 后续扩展定位

后续扩展方向为：

* 在框架母体不变的前提下，逐步扩展到 Codex 等其他模型；
* 每新增一个模型，补充该模型的工具支撑规范；
* 扩展方式为“单模型支撑能力复制”，而非“多模型混合编排协作”。

## 3. 指导思想：Harness Engineering

### 3.1 核心理念

Harness Engineering 的核心，不是研究如何写更复杂的提示词，而是通过工程化方式驾驭 AI，让 AI 在边界清晰、流程清晰、标准清晰、状态可控的体系中稳定发挥作用。

### 3.2 在本框架中的落地要求

在本框架中，Harness Engineering 体现为：

1. 明确 AI 在不同阶段承担的角色与职责；
2. 明确不同任务进入 AI 协作流程的条件；
3. 明确每一阶段的输入输出规范与状态切换规则；
4. 明确质量审查、失败处理与人工兜底规则；
5. 明确哪些成果需要沉淀、如何沉淀、谁负责维护；
6. 通过模板化、规则化、可追踪化降低 AI 输出随机性；
7. 通过反馈回路与资产积累持续提升整体交付效率和质量。

### 3.3 Harness 核心目标映射

结合软件交付场景，Harness 的目标拆解如下：

* **脆弱 → 稳定**：通过标准输入输出、阶段门禁、review 机制和人工兜底，降低交付波动；
* **手工 → 自动**：通过模板、自动生成、自动检查、自动推荐资产降低重复劳动；
* **割裂 → 连续**：通过产品、研发、测试三角色协同打通需求到交付链路；
* **智能、自动、可控**：让 AI 在受控系统中持续增强软件交付能力。

## 4. 一期建设范围

### 4.1 一期核心范围

一期仅围绕 Claude 的工具支撑展开建设，重点包括：

* Claude 在各工作流阶段的职责边界定义；
* Claude 在不同任务场景下的输入输出模板；
* Claude 的任务使用规范、状态流转规则与结果校验机制；
* Claude 协作成果的知识沉淀与回灌机制；
* 一期重点角色的使用适配；
* 基础治理、日志留痕与人工兜底机制。

### 4.2 一期不包含内容

一期暂不纳入以下内容：

* 多模型协同编排机制；
* Claude 与 Codex 等模型的联动调度；
* 运维、运营等角色的深度适配；
* 高复杂度自动代理链路；
* 无人工干预的高风险智能决策替代机制。

### 4.3 一期重点角色

考虑落地效率、协同链路完整性和质量闭环，一期优先覆盖以下角色：

* 产品人员
* 研发人员（含后端开发、前端开发、全栈开发人员）
* 测试人员

其中：

* **产品人员**负责需求发起、方案定义、业务目标确认与结果验收；
* **研发人员**负责技术方案分析、执行落地与技术问题处理，其中全栈开发人员还承担需求讨论、需求输出、前后端衔接、集成适配、技术联动等职责；
* **测试人员**负责质量校验、缺陷识别、场景补充与风险把关。

### 4.4 一期重点任务场景

一期优先覆盖以下任务场景：

1. 产品方案发散、需求整理、PRD/MRD 辅助撰写；
2. 技术方案分析、研发任务拆解、代码相关说明与问题分析；
3. 前后端协同设计、接口衔接分析、全栈开发相关任务支持；
4. 测试方案设计、测试计划制定、测试执行支持、缺陷分析、回归测试支持；
5. 围绕需求—研发—测试链路的跨角色评审与质量闭环任务；
6. 通用的会议总结、结构化分析、知识整理等标准文本任务。

## 5. 工作流索引

框架采用五步闭环：

**brainstorm → plan → work → review → compound**

这套工作流的详细机制、阶段输入输出、执行策略、skill/agent 调度和运行态资产，统一以 [4.五阶段工作流详解.md](/Users/kuang/xiaobu/spec-first/docs/01-需求分析/4.五阶段工作流详解.md) 为准。

因此，本文件只保留上位需求与框架定位，不再重复展开五阶段细节。

## 6. 角色与治理要求

### 6.1 角色适配要求

框架必须同时支持：

* 产品人员的需求发起、方案定义、结果验收；
* 研发人员的技术分析、任务拆解、执行落地；
* 测试人员的质量校验、风险识别、回归验证。

### 6.2 治理要求

框架必须具备以下治理能力：

* 明确输入输出边界；
* 明确阶段流转与回退规则；
* 明确人工接管条件；
* 明确资产沉淀与复用规则；
* 明确 Claude 一期能力边界。

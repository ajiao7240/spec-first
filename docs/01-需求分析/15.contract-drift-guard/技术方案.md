## 1. 结论先行

本方案新增一个轻量、常驻、纯确定性的契约漂移 guard：

- `tests/unit/contract-drift-guard.test.js`
- `tests/unit/fixtures/contract-capability-probes.js`

它要解决的不是“能力实现质量好不好”，而是更基础的 honesty 问题：**角色契约/治理文档声称某能力或入口存在时，代码侧必须有可验证承载；如果尚未实现，契约侧必须逐词显式标注 `(aspirational)`。**

这是 script-owned deterministic fact check，落在角色契约 §4 的 Scripts / Tools 职责：文件发现、schema/contract 校验、machine-readable reason 输出。它不让脚本判断架构取舍、能力成熟度、业务优先级或 review finding 是否成立。

本版较初稿有三处关键收敛：

1. `/spec:*` 入口完整清单不从角色契约 §5 示例句解析，而从 `src/cli/contracts/dual-host-governance/skills-governance.json` 的 `workflow_command` registry 派生。
2. `(aspirational)` 必须绑定到具体能力词，不允许“同一 markdown 表格行任意位置有标记就整行通过”。
3. `grep` probe 是概念名，Jest 实现用 Node `fs` 递归扫描，不依赖外部 `grep` / `rg` 命令，也不扫描 generated runtime mirrors。

## 2. 背景与问题

本轮契约对照审计发现一类高杠杆 honesty bug：文档声称某治理能力、harness 能力或公开入口已存在，但当前 source 没有对应 producer、runtime delivery 或测试承载。HC-001/003/004、WEG-01、GEH-01/02 都属于这个族群。

这类问题不应长期靠大型多-agent 审计反复发现。它的核心事实大多可机械核对：

- 契约 §2 Harness 表中的覆盖能力词是否有对应 source 承载；
- 未实现词是否被显式标为 aspirational；
- 公开 workflow command registry 是否与 Claude command 模板一致；
- generated runtime mirror 是否被误当作 source 证据；
- 已有 high-value runtime integrity anchor 测试是否登记为同一 drift-guard family。

因此，本方案把“声明 ↔ source 承载”的最低限度核对固化进 unit tests，作为 release/test 链路里的早期失败信号。

## 3. Goals / Non-goals

### Goals

- G1：角色契约 §2 Harness 表里每个覆盖能力词，要么有显式注册的 probe 且 probe 命中当前 source，要么该词自身带 `(aspirational)` 标记且注册表声明为 `aspirational`。
- G2：§2 覆盖能力词集合与 `contract-capability-probes.js` 注册表双向一致。新增能力词不登记 probe 必红；删除能力词不清注册表也必红。
- G3：公开 workflow command 的完整 expected set 来自 `skills-governance.json`，并校验每个 `workflow_command.command_name` 有对应 Claude command template。
- G4：guard 失败时输出足够复现的 reason：layer、term、source line、probe type、paths、pattern、hit count、missing command/template。
- G5：把已存在的 runtime integrity anchor 测试归档为同一 drift-guard family 的先例，不重写它。

### Non-goals

- NG1：不判断能力实现质量、成熟度、架构合理性或是否“足够好”。存在性不是质量证明。
- NG2：不引入 NLP 或语义解析。能力词由 §2 markdown 表格确定，probe 由人工显式登记。
- NG3：不把角色契约 §5 示例句升级为完整 workflow registry。§5 是治理说明，不是 command manifest。
- NG4：不新增 JSON schema 或 runtime artifact writer。Jest fixture 足够表达 v1 guard。
- NG5：不手改 `.claude/`、`.codex/`、`.agents/skills/` generated runtime mirrors。
- NG6：不实现 Evaluation Harness 指标采集。若某指标未 instrument，本 guard 只保证它在契约里被诚实标注。

## 5. Source-of-truth 与边界

### Source

- 能力声明 source：`docs/10-prompt/结构化项目角色契约.md` §2。
- workflow command registry source：`src/cli/contracts/dual-host-governance/skills-governance.json`。
- Claude command template source：`templates/claude/commands/spec/*.md`。
- probe registry：`tests/unit/fixtures/contract-capability-probes.js`，是 test-owned mirror，不是 runtime source。

### Generated runtime

以下路径不作为 probe hit 的默认证据：

- `.claude/**`
- `.codex/**`
- `.agents/skills/**`
- `.spec-first/audits/**`

原因：它们是 generated runtime / audit output。普通 drift guard 必须回到 source-of-truth 或 source tests，而不是让 runtime mirror 反向证明 source 声明。

### Script vs LLM

- Script 负责：抽词、读 registry、查路径、扫描 source、计数、输出 reason。
- LLM/维护者负责：决定一个未命中的能力词应该实现、降级标 `(aspirational)`，还是从契约删除。

## 6. 设计

### 6.1 文件落点

```text
tests/unit/contract-drift-guard.test.js
tests/unit/fixtures/contract-capability-probes.js
```

不新增 `src/cli/contracts/**` 文件，因为 v1 guard 只被 Jest 消费，不是 runtime 或 downstream workflow artifact。

### 6.2 Probe registry

```js
'use strict';

module.exports = {
  'Context Harness': {
    'bounded direct source reads': {
      type: 'content',
      pattern: 'bounded direct source reads',
      paths: ['skills', 'docs/contracts', 'README.md'],
    },
    '`rg`': {
      type: 'content',
      pattern: '`rg`',
      paths: ['skills', 'docs/contracts', 'README.md'],
    },
    'ast-grep': {
      type: 'content',
      pattern: 'ast-grep',
      paths: ['skills/spec-mcp-setup', 'skills/spec-plan', 'README.md'],
    },
    'context bundle': {
      type: 'path',
      anyOf: ['src/cli/helpers/context-bundle.js', 'docs/contracts/context-bundle.md'],
    },
    'docs/solutions': {
      type: 'path',
      anyOf: ['docs/solutions'],
    },
  },
  'Evaluation Harness': {
    'debug 命中率': {
      type: 'aspirational',
      reason: 'HC-001: 当前未 instrument debug hit-rate',
    },
  },
};
```

Probe 类型：

| type | 含义 | 判定 |
|------|------|------|
| `path` | source 文件/目录存在即可证明承载存在 | `anyOf` 至少一个 repo-relative path 存在 |
| `content` | source 文本中存在稳定承载锚点 | Node `fs` 扫描 `paths` 下文本文件，`pattern` 命中数 > 0 |
| `aspirational` | 明确未实现或未 instrument | §2 中该 term 自身必须带 `(aspirational)`；注册表必须有 `reason` |

说明：初稿称 `grep` probe，本版改名为 `content`。如果保留 `grep` 字段名，也必须在实现里用 Node 内置扫描，不调用外部命令。

### 6.3 §2 能力词抽取

抽取规则：

1. 只读 `docs/10-prompt/结构化项目角色契约.md` 中 `## 2. 核心链路与 Harness 层` 下的 markdown 表格。
2. 表格列 1 为 `layer`，列 3 为 coverage text。
3. 用中文顿号 `、` 切分 coverage text。
4. 保留原始 term 文本用于注册表 key 对比。
5. 去掉 markdown emphasis，不去掉 backtick。也就是说当前 `rg` term 是 `` `rg` ``，除非契约同步改成无 backtick。
6. 每个 term 记录 `layer`、`lineNumber`、`rawCell`、`termText`、`hasAspirationalMarker`。

`hasAspirationalMarker` 必须 term-local，而不是 row-local。推荐实现：

- 先按顿号切片；
- 对每个切片独立判断是否含 `(aspirational)`；
- 用去掉 marker 后的文本作为 normalized key；
- 若整行末尾需要批量标记多个词，必须逐词写 marker，例如 `debug 命中率 (aspirational)、review 漏判率 (aspirational)`。

### 6.4 T1：§2 term ↔ probe registry 双向一致

```text
actualTerms = parseHarnessCoverageTerms(roleContract)
registeredTerms = flattenProbeRegistry(contractCapabilityProbes)

assert no missingProbe: actualTerms - registeredTerms
assert no staleProbe: registeredTerms - actualTerms
assert each registered term is under the same layer as contract term
```

失败输出示例：

```text
missing_probe: layer="Governance Harness" term="hook budget" line=35
stale_probe: layer="Evidence Harness" term="old audit metric"
layer_mismatch: term="context bundle" contract="Context Harness" registry="Evidence Harness"
```

### 6.5 T2：非 aspirational probe 必须命中；aspirational 必须逐词标注

`path`：

- `anyOf` 为空或不是数组：失败 `invalid_probe_shape`。
- 所有 path 都不存在：失败 `path_probe_missed`。

`content`：

- `paths` 为空或不是数组：失败 `invalid_probe_shape`。
- 只扫描 repo 内 source path。
- 默认排除 generated runtime/audit 路径。
- 对 `.md`、`.js`、`.json`、`.sh`、`.ps1`、`.cjs`、`.mjs`、`.yaml`、`.yml`、`.txt` 等文本文件计数。
- `pattern` 可以是 string 或 RegExp。string 用 `includes`，RegExp 用 `test`。
- 命中数为 0：失败 `content_probe_missed`。

`aspirational`：

- 注册表必须有非空 `reason`。
- §2 中对应 term 必须 term-local 带 `(aspirational)`。
- 如果 term 没有 marker：失败 `aspirational_marker_missing`。
- 如果 term 有 marker 但 registry 不是 `aspirational`：失败 `unexpected_aspirational_marker`，防止“已实现能力被文档误标为规划中”。

### 6.6 T3：workflow command registry ↔ Claude command template

T3 不解析角色契约 §5 的示例入口列表。正确 expected set：

```js
const governance = JSON.parse(fs.readFileSync(GOVERNANCE_PATH, 'utf8'));
const workflowCommands = governance.skills
  .filter((record) => record.entry_surface === 'workflow_command')
  .map((record) => record.command_name)
  .sort();
```

断言：

1. 每个 `workflow_command` 必须有非空 `command_name`。
2. 每个 `command_name` 必须存在 `templates/claude/commands/spec/<command_name>.md`。
3. `templates/claude/commands/spec/*.md` 中每个 basename 必须出现在 `workflow_command.command_name`。
4. 每个 workflow record 的 `host_delivery.claude` 必须是 `command`。
5. 每个 dual-host workflow record 的 `host_delivery.codex` 必须是 `skill` 或当前 registry 允许的合法 delivery；v1 不检查 `.codex/commands`，因为 `CodexAdapter.hasCommands = false`。

角色契约 §5 只做轻量 honesty guard：

- 不应包含 `/spec:update`。
- 若列出 runtime 维护类动作，必须表述为 terminal CLI：`spec-first update`、`spec-first init`、`spec-first clean`、`spec-first doctor`。
- 不要求 §5 枚举所有 workflow commands。

### 6.7 SRB-001 family 登记

不改写 `tests/unit/spec-plan-contracts.test.js`。只在 guard 文档或测试注释中登记：

- `spec-plan` runtime integrity anchors 属于 `contract-drift-guard` family 的既有特化测试；
- 它防的是 source/reference/runtime projection anchor 漂移；
- 新 `contract-drift-guard.test.js` 防的是 role-contract capability/entry declaration 漂移。

## 7. 实现细节

### 7.1 Helper 函数

建议在 `contract-drift-guard.test.js` 内保持局部 helper，不新增共享模块：

```js
function readRepoFile(relativePath) {}
function parseHarnessCoverageTerms(markdown) {}
function flattenProbeRegistry(registry) {}
function listTextFiles(relativePaths, options = {}) {}
function countContentHits({ pattern, paths }) {}
function compareSets(actual, expected) {}
function workflowCommandNames() {}
function claudeCommandTemplateNames() {}
```

只有当后续第二个测试文件真实复用这些 helper 时，再考虑抽到 `tests/unit/helpers/`。v1 不提前抽象。

### 7.2 失败 reason 输出

Jest 默认 diff 对复杂对象不够直观。建议手工收集 `errors` 数组，最后：

```js
expect(errors).toEqual([]);
```

每个 error 是结构化对象：

```js
{
  reason_code: 'content_probe_missed',
  layer: 'Evidence Harness',
  term: 'debug ledger',
  line: 33,
  probe: {
    type: 'content',
    pattern: 'debug ledger',
    paths: ['skills/spec-debug', 'docs/contracts'],
  },
  hits: 0,
}
```

这满足“机器可读、可复现”的目标，同时不新增 JSON schema。

### 7.3 文本扫描排除

默认排除：

```js
const EXCLUDED_PREFIXES = [
  '.claude/',
  '.codex/',
  '.agents/skills/',
  '.spec-first/audits/',
  'node_modules/',
  'graphify-out/',
];
```

`graphify-out/` 是 project graph output，不作为本 guard 的 source evidence。若未来有专门 project-graph drift guard，应另行定义 source/consumer 边界。

### 7.4 Probe 稳定性准则

优先选择稳定 source 承载：

- 文件路径优先于宽泛词命中；
- helper/contract 文件优先于历史 validation docs；
- workflow skill/source contract 优先于 README marketing prose；
- tests 可以作为承载补充，但不能单独证明用户可见能力已存在；
- 历史 `docs/plans/**`、`docs/validation/**` 默认不作为实现承载，除非 probe 明确是在登记历史先例。

## 8. 落地顺序

1. **契约诚实化**
   给 §2 中当前未实现或未 instrument 的 term 逐词加 `(aspirational)`，或删除不应保留的路线图词。每个改动同步 `CHANGELOG.md`。

2. **建立 probe registry**
   新增 `tests/unit/fixtures/contract-capability-probes.js`，覆盖 §2 当前所有 term。初次登记时可以先把确认为未实现的词设为 `aspirational`，避免为了过 test 编造弱 probe。

3. **实现 T1/T2**
   先只跑 §2 term/probe 双向一致与 probe hit。验证命令：

   ```bash
   npx jest tests/unit/contract-drift-guard.test.js --runInBand
   ```

4. **实现 T3 registry/template guard**
   使用 `skills-governance.json` 派生 expected command set，校验 Claude command template 双向一致；添加 §5 lightweight honesty check。

5. **负例反向验证**
   临时制造并回滚三个失败：
   - §2 加假能力词但不登记 probe，T1 应红；
   - 移除某 aspirational term 自身 marker，T2 应红；
   - 新增临时 Claude command template 但不登记 governance，T3 应红。

6. **回归验证**
   运行最窄到最宽：

   ```bash
   npx jest tests/unit/contract-drift-guard.test.js --runInBand
   npm run test:unit
   ```

7. **文档/family 登记**
   更新审计报告 §4-B 状态，并在本方案或相关测试注释中登记 SRB-001 runtime integrity anchors 属于同一 guard family。

## 11. 风险与缓解

| 风险 | 影响 | 缓解 |
|------|------|------|
| Probe 注册表变成第二 truth source | 维护者同时改契约和 fixture，产生镜像负担 | 明确 fixture 是 test-owned mirror；T1 双向一致强制清理 stale probe |
| `content` probe 过宽导致假阳性 | 文档提到词但无实现承载也通过 | 优先 path/helper/contract source；probe path 限定到承载目录，默认排除历史 docs |
| `content` probe 过窄导致频繁红 | 重命名或移动实现时需要同步 fixture | 这是可接受 forcing function；失败 reason 指向具体 probe |
| Aspirational 标记污染已实现能力 | 能力被误降级 | T2 增加 `unexpected_aspirational_marker` |
| §5 入口校验过度约束 prose | 角色契约示例句被迫枚举全量入口 | T3 改用 `skills-governance.json`，§5 只查 runtime CLI 不被伪装成 `/spec:*` |
| 扫描 generated runtime 造成自证循环 | runtime mirror 替 source 背书 | 默认排除 generated runtime/audit path |

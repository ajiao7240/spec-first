## 文档 1：`code-facts-context/prd.md`

```md
# Task PRD: code-facts-context

## Goal

生成项目的“代码事实层”上下文资产，为后续的 brainstorm / plan / work / review 提供稳定、低 token、少主观解释的代码级上下文。

本任务要解决的问题：

- 后续节点不知道从哪里开始看代码
- 改动时容易漏看真实入口和关键依赖
- review 时不知道哪些模块风险高
- 实现时不知道应该联看哪些测试
- 每次都重复冷启动扫描仓库

本任务应优先从源码、配置、测试、入口文件中抽取“事实”，而不是产出泛泛的项目介绍。

---

## Target Output

本任务必须生成以下文件：

- `docs/contexts/<slug>/code-facts/public-entrypoints.md`
- `docs/contexts/<slug>/code-facts/core-dependencies.md`
- `docs/contexts/<slug>/code-facts/test-map.md`
- `docs/contexts/<slug>/code-facts/high-risk-modules.md`

如果仓库规模很小，仍然应生成上述文件，但允许内容更简洁。

---

## Why It Matters

这些文件的目标不是“解释项目”，而是帮助后续节点快速回答：

- 程序从哪里进入？
- 新增功能时应先看哪些入口？
- 哪些 shared/core 模块改动风险高？
- 哪些测试最可能需要联看？
- 哪些区域虽然 diff 小，但影响面大？

成功的产物应该减少：

- 无边界地扫描仓库
- 盲目从目录名猜职责
- 漏掉相关测试
- 漏判高风险模块
- 仅凭 diff 做浅层 review

---

## Data Sources

### Priority 1: Source Code
优先从以下代码事实中提炼：

- 主启动文件（如 `main.*`, `app.*`, `server.*`, `bootstrap.*`, `index.*`）
- 路由注册文件
- CLI 入口
- worker / cron / queue consumer 入口
- service / controller / handler / repository / model 等典型职责模块
- shared / core / common / utils 等公共模块
- 公共导出文件（如 `index.ts`, `exports`, SDK public API）

### Priority 2: Tests
优先识别：

- unit tests
- integration tests
- e2e tests
- fixtures / mocks / stubs
- snapshot tests
- test setup / bootstrap files

### Priority 3: Config and Metadata
如有必要，可使用：

- `package.json`
- `pnpm-workspace.yaml`
- `tsconfig.json`
- `vite.config.*`
- `next.config.*`
- `go.mod`
- `Cargo.toml`
- `pom.xml`
- `build.gradle`
- CI 配置
- lint / formatter 配置

### Priority 4: Existing Bootstrap Outputs
如果以下文件已存在，可作为辅助来源，但不得覆盖代码事实：

- `docs/contexts/<slug>/00-summary.md`
- `docs/contexts/<slug>/architecture/system-overview.md`
- `docs/contexts/<slug>/architecture/module-map.md`
- `docs/contexts/<slug>/pitfalls/index.md`

---

## Extraction Rules

### Rule 1: Facts First
先抽取代码事实，再做最小必要归纳。  
不要先写结论，再倒推代码依据。

### Rule 2: Low Interpretation
尽量减少主观推断。  
例如：

- 可以写：`src/server/router.ts` 注册了主要 HTTP 路由
- 不要写：系统采用高度解耦的现代模块化架构  
  除非有明确代码依据支撑

### Rule 3: Mark Uncertainty
如果某条关系是根据目录命名、调用模式或惯例推断出来的，应明确标记为：

- `Observed`
- `Likely`
- `Inferred`
- `Unknown`

不要把推断写成确定事实。

### Rule 4: Prefer Scannable Structure
输出优先使用：

- 短段落
- 条目列表
- 表格
- 路径清单
- 模块清单

不要写长篇散文式描述。

### Rule 5: Optimize for Later Injection
写作目标不是“文档好看”，而是让后续节点能快速读取并决定：

- 先看哪里
- 哪些模块不能轻易动
- 哪些测试要一起看
- 哪些区域 review 时必须提高警惕

## Output Requirements

### 1. `public-entrypoints.md`

必须包含：

- 项目主要运行入口
- HTTP 入口（如有）
- CLI 入口（如有）
- worker / cron / queue 入口（如有）
- 公共导出 API 或 SDK 入口（如有）
- 每类入口对应的关键后续链路起点
- “做常见功能改动时通常先看哪里”

建议结构：

- Overview
- HTTP Entrypoints
- CLI Entrypoints
- Background/Async Entrypoints
- Public Export Entrypoints
- First Files to Inspect by Task Type

---

### 2. `core-dependencies.md`

必须包含：

- 主要模块间依赖方向
- shared/core/common/util 模块
- 高扇出模块
- 关键跨层依赖
- 哪些模块很多功能都会间接依赖
- 哪些模块不建议轻易重构

建议结构：

- Dependency Overview
- Shared/Core Modules
- High Fan-out Modules
- Cross-layer Dependencies
- Modules with Broad Change Impact

---

### 3. `test-map.md`

必须包含：

- 单测、集成测试、端到端测试的位置
- 关键模块对应的测试位置
- 测试薄弱区域
- 哪类改动应优先联看哪些测试
- 哪些 test setup / fixture 对理解行为很关键

建议结构：

- Test Layout Overview
- Module-to-Test Mapping
- Critical Test Entry Points
- Weakly Tested Areas
- Test Review Hints by Change Type

---

### 4. `high-risk-modules.md`

必须包含：

- 高扇出模块
- 高复杂度模块
- 低测试覆盖且重要的模块
- 共享工具类/基础模块
- 可能隐藏全局副作用的模块
- 说明“为什么风险高”
- 说明“改动时应联看什么”

建议结构：

- Risk Ranking Overview
- Broad Impact Modules
- Fragile Shared Utilities
- Complexity Hotspots
- Low-confidence Areas
- Review Tips

---

## File Style Requirements

所有文件都应满足：

- 使用明确标题
- 路径尽量具体
- 模块名尽量具体
- 适当引用符号名、类名、函数名
- 内容短而实用
- 不写“请注意性能”“注意安全”这类空话
- 能被后续 agent 快速消费

## Acceptance Criteria

任务完成后，后续节点应能够仅依赖这些产物回答：

1. 这个项目从哪里启动？
2. 新功能通常从哪个入口开始改？
3. 哪些 shared/core 模块一改容易波及全局？
4. 哪些测试最值得联看？
5. 哪些区域高风险，不应轻率修改？
6. review 某类改动时，应重点看哪些模块和测试？

若上述问题无法回答，则任务视为未完成。

---

## Risks / Caveats

### Risk 1: Over-Inference
不要把目录命名或框架惯例直接当作事实。

### Risk 2: Missing Dynamic Behavior
某些依赖关系可能是动态注册、反射、配置驱动，静态读取未必完全覆盖。  
如果有这种情况，应明确说明局限性。

### Risk 3: Test Mapping Ambiguity
模块和测试的关系可能不是一一对应。  
允许给出“likely related tests”，但要标明是 likely。

### Risk 4: Monorepo / Workspace Complexity
如果仓库是 monorepo，应明确当前文档覆盖的范围和子项目边界，避免混淆。

---

## Suggested Workflow

1. 扫描顶层目录和主要配置，确认项目形态
2. 定位启动入口、路由入口、异步入口、公共导出入口
3. 粗略识别核心模块和 shared/core/common/util 层
4. 识别测试目录、fixtures、test bootstrap 文件
5. 识别高扇出、高复杂度、高风险模块
6. 先写事实清单，再整理为四个目标文件
7. 自检：是否足够具体、是否可供后续节点快速注入

---

## Minimum Quality Bar

至少做到：

- `public-entrypoints.md` 不空泛，能指出真实入口
- `test-map.md` 不只是列目录，能指出测试与模块的关系
- `high-risk-modules.md` 不只是列“复杂文件”，而是说明风险原因
- `core-dependencies.md` 能体现哪些模块有广泛影响面

如果只能产出“目录级泛泛总结”，则质量不达标。

---

## Output Constraints

- 不泄露任何敏感信息
- 不写入密钥、token、连接串
- 不伪造覆盖率数据
- 不捏造不存在的模块关系
- 不把“猜测”包装成“结论”

---

## Final Reminder

本任务生成的是“代码事实索引”，不是传统项目说明文。  
优先保证：

- 客观
- 具体
- 可执行
- 可注入
```

## 文档 2：`injection-context/prd.md`

````md
# Task PRD: injection-context

## Goal

将已有的项目理解资产和代码事实资产，进一步加工为“任务型上下文包”，用于后续 brainstorm / plan / work / review 节点按需注入上下文。

本任务要解决的问题：

- 即使已有 summary / architecture / pitfalls，后续节点仍然不知道“当前任务该先读哪几份文档”
- 不同任务类型需要的上下文不同，不能一股脑全部注入
- 缺少统一的注入优先级规则
- 上下文资产虽然存在，但还没有转成可直接服务执行的格式

本任务应产出短小、任务导向、可注入的 context packs，而不是再写一批泛泛说明文档。

---

## Target Output

本任务必须生成以下文件：

- `docs/contexts/<slug>/context-packs/add-endpoint.md`
- `docs/contexts/<slug>/context-packs/bugfix.md`
- `docs/contexts/<slug>/context-packs/refactor-shared.md`
- `docs/contexts/<slug>/context-packs/review-change.md`
- `docs/contexts/<slug>/injection-index.yaml`

如果某项目不具备某类任务的典型场景，也应保留该文件，但可明确写出适用范围有限。

---

## Why It Matters

这些产物的目标是帮助后续节点快速回答：

- 这个任务现在该先读什么？
- 哪些上下文是必需的，哪些可以稍后再看？
- 哪些模块最可能相关？
- 哪些坑点最相关？
- 哪些测试要一起看？
- 哪些常见过度修改应该避免？

成功的输出应减少：

- 每个节点重新理解项目结构
- 无边界地读取大量文档和代码
- 上下文注入过量
- 任务和上下文不匹配
- 实现和 review 的关注点漂移

---

## Data Sources

### Primary Inputs
优先使用以下已生成上下文资产：

- `docs/contexts/<slug>/00-summary.md`
- `docs/contexts/<slug>/architecture/system-overview.md`
- `docs/contexts/<slug>/architecture/module-map.md`
- `docs/contexts/<slug>/architecture/integration-boundaries.md`（如有）
- `docs/contexts/<slug>/pitfalls/index.md`
- `docs/contexts/<slug>/code-facts/public-entrypoints.md`
- `docs/contexts/<slug>/code-facts/core-dependencies.md`
- `docs/contexts/<slug>/code-facts/test-map.md`
- `docs/contexts/<slug>/code-facts/high-risk-modules.md`
- `docs/contexts/<slug>/layers/*/index.md`（如有）
- `docs/contexts/<slug>/database/*`（如有）

### Secondary Inputs
必要时可回到源代码或配置补充：

- 路由入口
- 测试位置
- shared/core 模块
- DB schema
- 错误处理和 validation 位置

---

## Extraction Rules

### Rule 1: Organize by Task, Not by Knowledge Domain
本任务不是再按“架构 / 模块 / 数据库”组织内容，  
而是按“任务类型”组织内容。

每个 context pack 都必须直接服务一种典型动作：

- add endpoint
- bugfix
- refactor shared code
- review change

### Rule 2: Tell the Next Node What to Read First
每个 context pack 都必须明确写出：

- first files/docs to inspect
- likely relevant modules
- likely relevant tests
- common pitfalls
- overreach to avoid

### Rule 3: Keep Packs Short and Actionable
这些文件不是大而全总结。  
目标是 1 次注入就能用，而不是让 agent 再读一篇长文。

### Rule 4: Reuse Existing Assets
优先重组已有产物，而不是重新发明一套解释。

### Rule 5: Separate Must-Read from Nice-to-Read
每个任务包要区分：

- 必读
- 建议读
- 需要时再读

### Rule 6: Explicit Priority Rules
`injection-index.yaml` 必须明确不同阶段和任务类型的优先注入顺序。

## Output Requirements

## 1. `context-packs/add-endpoint.md`

本文件应服务“新增接口 / 新增路由 / 新增 handler / 新增 API 能力”。

必须包含：

- When to use
- Read first
- Typical path of implementation
- Related contracts / boundaries
- Common pitfalls
- Related tests
- Avoid these mistakes

重点应回答：

- 先看哪个入口
- DTO/schema/validation 在哪
- service/repository 往哪里走
- 哪些 shared 模块或中间件会受影响
- 需要联看哪些测试
- 哪些不该随便改

---

## 2. `context-packs/bugfix.md`

本文件应服务“定位并修复已有缺陷”。

必须包含：

- When to use
- Read first
- How to trace the behavior
- Common fragile spots
- Related error handling / logging points
- Related tests
- Avoid these mistakes

重点应回答：

- 出问题时先看入口还是异常处理
- 哪些地方容易误判问题根因
- 哪些模块虽然不在报错点，但常是根因来源
- 哪些测试最适合帮助复现/验证

---

## 3. `context-packs/refactor-shared.md`

本文件应服务“修改 shared/core/common/util 等公共模块”。

必须包含：

- When to use
- Read first
- Dependency impact considerations
- Compatibility concerns
- Related tests
- High-risk modules involved
- Avoid these mistakes

重点应回答：

- 下游依赖多不多
- 哪些 API/调用方最容易被破坏
- 哪些模块改动需要特别保守
- review 时应如何放大检查范围

---

## 4. `context-packs/review-change.md`

本文件应服务“review diff / review implementation / review refactor”。

必须包含：

- When to use
- Read first
- Diff-adjacent context to inspect
- Common missed areas
- Test review hints
- High-risk signals
- Review checklist

重点应回答：

- 除了 diff 外还该看什么
- 哪些 related files/tests 经常被漏掉
- 哪类变更一看就要提高警惕
- 高风险区域有哪些典型信号

## 5. `injection-index.yaml`

这是最关键的调度文件。

它必须定义：

- always inject
- stage-based injection
- task-type-based injection
- optional fallback reads

建议 schema 如下：

```yaml
always:
  - 00-summary.md
  - pitfalls/index.md

brainstorm:
  - architecture/system-overview.md

planning:
  - architecture/module-map.md
  - code-facts/public-entrypoints.md

implementing:
  default:
    - code-facts/public-entrypoints.md
    - code-facts/test-map.md
  add-endpoint:
    - context-packs/add-endpoint.md
  bugfix:
    - context-packs/bugfix.md
  refactor-shared:
    - context-packs/refactor-shared.md

reviewing:
  default:
    - pitfalls/index.md
    - code-facts/high-risk-modules.md
  review-change:
    - context-packs/review-change.md
````

实际 schema 可调整，但必须表达：

* 哪些文档始终优先
* 哪些文档适合 planning
* 哪些文档适合 implementing
* 哪些文档适合 reviewing
* 针对不同任务类型补哪些 pack

## Structure Requirements for Every Context Pack

每个 `context-packs/*.md` 文件都应尽量使用同一结构：

* Purpose
* When to use
* Read first
* Key modules / directories
* Related tests
* Common pitfalls
* Avoid overreach
* Escalation hints

其中：

### Read first

必须给出优先阅读顺序，而不是只列材料名。

### Avoid overreach

必须写出常见误区，例如：

* 不要一上来重构 shared 层
* 不要只改 handler 不看 validation
* 不要只看 diff 不联看相关测试
* 不要因为 bug 出在表层就忽略底层依赖

### Escalation hints

必须说明什么时候需要扩大上下文范围。

## Acceptance Criteria

任务完成后，后续节点应能够：

1. 根据任务类型快速选出该读的上下文
2. 明确区分必读和补充阅读
3. 在实现前知道哪些模块最相关
4. 在 review 时知道哪些相关区域容易漏看
5. 在不同任务类型下读取不同的 context pack
6. 避免将整套 `docs/contexts/<slug>/` 全量注入

## Risks / Caveats

### Risk 1: Packs Too Generic

如果 context pack 写成“通用工程建议”，则没有价值。
必须和当前项目的具体结构绑定。

### Risk 2: Packs Too Long

如果每个 pack 都变成长篇文档，后续节点仍然会读不动。
应保持短、准、可执行。

### Risk 3: Wrong Priority Rules

`injection-index.yaml` 若未区分任务类型，会导致上下文注入失焦。

### Risk 4: Stale Context

当项目结构显著变化时，这些 pack 可能过期。
必要时应在文档中提示 freshness caveat。

---

## Suggested Workflow

1. 读取 summary / architecture / pitfalls / code-facts
2. 按任务类型归纳真正需要的上下文
3. 提炼每种任务的“先看什么、联看什么、避免什么”
4. 写出 4 个 context packs
5. 基于这些 pack 设计 injection-index.yaml
6. 自检：是否足够短、足够具体、适合后续节点直接注入

基于 **当前 `spec-graph-bootstrap` 的现有实现** 来看，它已经有一条比较清晰的流水线：

* 先做 **Host Readiness Gate**，不满足就中止。([GitHub][1])
* 再做 **项目分析 + 工具探测 + mode 选择**，在 Full / Enhanced / Basic 三种模式下退化执行。([GitHub][1])
* 然后在 **Phase 2** 生成 PRD task contracts，固定产出 summary / architecture / pitfalls，并按 layer、database 条件增补。([GitHub][1])
* 最终目标是把产物沉淀到 `docs/contexts/<slug>/`，作为后续 workflow 可复用的长期上下文。只是当前版本也明确写了：**自动注入五阶段 workflow 还不是现有能力，而是 future capability**。([GitHub][1])

所以如果以你的目标——**提升 AI 编码质量、为后续节点注入上下文**——来审视，结论是：

## 总判断

**当前实现方向正确，但“项目理解产物”强于“执行注入产物”。**
也就是说，它现在更擅长生成：

* 项目摘要
* 架构总览
* 分层说明
* 坑点文档
* 数据库文档

但还不够擅长生成：

* 任务型上下文包
* 注入优先级
* 代码事实索引
* review/change-oriented context

这不是推翻重做，而是适合做 **增量优化**：保留现有 Phase 1/2 骨架，在 task 设计和产物结构上补一层“面向 AI 执行”的资产。

# 一、基于当前实现的优化方案

## 1. 保留现有三类固定任务，但重写“产物定义”

当前固定任务是：

* `summary-context` → `00-summary.md`
* `architecture-context` → `system-overview.md` / `module-map.md` / `integration-boundaries.md`
* `pitfalls-context` → `pitfalls/index.md` ([GitHub][1])

这三类都该保留，因为它们是最基础的项目理解资产。
但建议把它们从“描述性文档”改成“可注入入口”。

### 建议改法

`00-summary.md` 增加：

* 项目一句话定位
* 主语言 / 主框架 / 启动入口
* 关键目录入口
* 本项目后续最该先读的 3 份文档
* 高风险区域摘要

`module-map.md` 增加：

* 每类需求优先进入的目录
* 不建议直接修改的目录
* 公共依赖层 / 高扇出模块

`pitfalls/index.md` 增加：

* 按任务类型分组，而不只是按文件罗列
* 每个坑点对应“改动前检查项”

这样仍然符合现有 task 边界，但更适合后续节点注入。

## 2. 在 Phase 2 新增两类任务

当前 Phase 2 只有 fixed tasks、layer tasks 和 conditional database task。([GitHub][1])
建议在不破坏现有流程的前提下，再加两类任务。

### A. `code-facts-context`

建议新增产物：

* `docs/contexts/<slug>/code-facts/public-entrypoints.md`
* `docs/contexts/<slug>/code-facts/core-dependencies.md`
* `docs/contexts/<slug>/code-facts/test-map.md`
* `docs/contexts/<slug>/code-facts/high-risk-modules.md`

作用：

* 让后续节点先拿到“代码事实层”
* 降低 summary / architecture 过度主观化的问题

### B. `injection-context`

建议新增产物：

* `docs/contexts/<slug>/context-packs/add-endpoint.md`
* `docs/contexts/<slug>/context-packs/bugfix.md`
* `docs/contexts/<slug>/context-packs/refactor-shared.md`
* `docs/contexts/<slug>/context-packs/review-change.md`
* `docs/contexts/<slug>/injection-index.yaml`

作用：

* 把“项目知识”转成“任务注入策略”
* 为后续 plan/work/review 节点提供可直接消费的 context pack

这两类任务，是当前实现最值得补的地方。

---

## 3. 强化 Phase 1 的“数据提炼规则”，不要只做项目识别

当前 Phase 1 已经会做：

* mode 检测
* layer detection
* DB config detection
* 最低质量保证：`00-summary.md` 要识别主语言、主框架、顶层模块；`module-map.md` 要有顶层目录职责。([GitHub][1])

这很好，但还缺“数据来源 → 产物字段”的明确映射。
建议在 Phase 1 增加一层内部 schema，比如：

* entrypoints
* routes
* configs
* cross-module dependencies
* tests
* external integrations
* error handlers
* data entities

先抽成中间事实集，再写文档。
这样产物一致性会高很多，也更利于未来自动注入。

---

## 4. 把 `README.md` 从目录页升级为“上下文控制台”

当前 skill 里说 README 作为 contexts 目录入口很合理，但更适合进一步变成总控页。([GitHub][1])

建议它固定包含：

* 项目一句话简介
* 最近 bootstrap 时间
* 当前分析模式（Full / Enhanced / Basic）
* 本项目推荐优先阅读顺序
* 高风险区域
* context packs 导航
* layers 导航
* database 状态

这样后续 agent 只读 README，就能决定下一步读什么。

---

## 5. 明确“注入优先级”，解决当前 future capability 缺口

当前 skill 直接写明：**自动注入到五阶段 workflow 是未来能力**。([GitHub][1])
所以现在最合理的补法不是等框架改完，而是先把优先级显式写成文件。

建议新增：

`docs/contexts/<slug>/injection-index.yaml`

示例结构：

```yaml
always:
  - 00-summary.md
  - pitfalls/index.md

planning:
  - architecture/system-overview.md
  - architecture/module-map.md

implementing:
  - code-facts/public-entrypoints.md
  - code-facts/test-map.md
  - context-packs/bugfix.md

reviewing:
  - pitfalls/index.md
  - code-facts/high-risk-modules.md
  - context-packs/review-change.md
```

这能把“文档资产”转成“注入规则资产”。

这五个最直接影响 AI 改代码和 review 的质量。

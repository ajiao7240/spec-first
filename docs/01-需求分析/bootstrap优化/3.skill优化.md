# Spec-First Bootstrap

Bootstrap a durable **project context + execution context** library for a target repository.

This is a **Stage-0 supporting workflow** — it runs once (or on demand) to generate reusable context assets that later spec-first workflows can consume.

**Claude entry point:** `/spec:bootstrap [target-repo-path-or-slug]`  
**Codex entry point:** `$spec-bootstrap [target-repo-path-or-slug]`

## Why This Exists

spec-first's five-stage workflow (brainstorm → plan → work → review → compound) works best when it can draw on stable, project-level context:

- project summary
- architecture and module boundaries
- integration boundaries
- known pitfalls
- code facts
- task-oriented context packs
- injection priority rules

Without this foundation, each workflow session starts cold, leading to repeated inference, noisy file search, weak task routing, and lower coding/review quality.

`spec-bootstrap` solves this by producing a reusable context library at:

`docs/contexts/<slug>/`

It is not a sixth stage of the main workflow — it is a prerequisite asset generator you run when onboarding a new project or refreshing context after significant changes.

**Current scope of this workflow:**
1. generate durable project context assets
2. generate task-oriented execution context assets
3. generate injection priority metadata

Automatic injection into the five-stage workflow may still be implemented separately by the host framework. This workflow must therefore emit explicit injection metadata that downstream workflows can consume.

## Design Principles

### 1. Facts first, summaries second
Always extract project facts from source code, config, schema, tests, and runtime entrypoints before writing summaries or guidance.

### 2. Execution-oriented output
Do not stop at descriptive project documentation. Outputs must help later workflows answer:
- where should I look first?
- what is likely impacted?
- which tests should I inspect?
- what are the common pitfalls for this type of change?

### 3. Conservative inference
Do not present guesses as facts. If something is inferred rather than directly grounded in code/config/schema, mark it clearly.

### 4. Small, composable context
Prefer concise, structured outputs that are easy for later workflows to inject selectively.

### 5. No framework drift
Describe the repository as it exists. Do not rewrite the project into an imagined architecture.

## Phase 1: Analyze the Target Repository

**Goal:** Build enough grounded understanding to define high-quality context tasks and populate execution-oriented outputs.

Keep the existing responsibilities:
- establish slug
- backup existing context dir if needed
- exclude previous context output from analysis
- probe tools
- detect DB access
- determine mode

### Extend Phase 1 with a normalized fact inventory

Before Phase 2 task generation, the workflow should extract or estimate the following fact categories:

- `project_identity`
  - project name
  - primary language
  - primary framework(s)
  - repo shape (single app / monorepo / service / library / CLI / mixed)

- `entrypoints`
  - app bootstrap files
  - router registration files
  - CLI entrypoints
  - workers / cron / queue consumers
  - public exported SDK entrypoints

- `module_structure`
  - top-level directories
  - likely responsibility of each module
  - shared/core/common utility layers

- `integration_points`
  - database clients
  - cache clients
  - message queues
  - external HTTP/gRPC/SDK integrations
  - auth/session boundaries

- `data_shapes`
  - DTO/schema/model/entity definitions
  - migrations
  - ORM mappings
  - validation rules

- `testing_surface`
  - unit/integration/e2e test locations
  - fixtures/mocks
  - rough module-to-test adjacency

- `risk_signals`
  - high fan-out modules
  - high complexity hotspots
  - global state / singleton usage
  - implicit conventions
  - low-test-confidence areas
  - fragile shared utilities

- `layer_detection`
  - frontend/backend/mobile/desktop/cli/shared/data/guides

- `database_detection`
  - current DB detection logic
  - verification state
  - DB count / schema scope

This fact inventory is an internal extraction step. It does not need to be emitted directly as a file, but all Phase 2 tasks must use it.

## Phase 2: Generate Context Tasks

**Goal:** Generate task contracts that produce durable, execution-oriented context assets.

Keep the existing control plane location:

`.context/spec-first/bootstrap/<slug>/tasks/<task-id>/prd.md`

### 2.1 Fixed tasks (required)

#### Task: `summary-context`
**Outputs**
- `docs/contexts/<slug>/00-summary.md`

**Purpose**
Produce the shortest durable entrypoint into the project.

**Must include**
- project one-line description
- primary language / framework
- primary entrypoints
- top-level module map (brief)
- what later workflows should read next
- current analysis mode caveats
- top 3 risk areas

---

#### Task: `architecture-context`
**Outputs**
- `docs/contexts/<slug>/architecture/system-overview.md`
- `docs/contexts/<slug>/architecture/module-map.md`
- `docs/contexts/<slug>/architecture/integration-boundaries.md` (conditional)

**Purpose**
Describe how the system is organized and where major responsibilities live.

**Must include**
- startup / request / execution flow overview
- top-level module responsibilities
- external system boundaries
- important internal boundaries
- where new features usually enter
- where cross-cutting changes are risky

---

#### Task: `pitfalls-context`
**Outputs**
- `docs/contexts/<slug>/pitfalls/index.md`

**Purpose**
Capture concrete, file-grounded pitfalls that later workflows should avoid.

**Must include for each pitfall**
- path
- line range or symbol
- pitfall type
- why it is risky
- what kind of change commonly breaks here
- suggested mitigation
- related tests if known

Prefer grouping pitfalls by task relevance:
- adding endpoints
- bug fixing
- refactoring shared code
- reviewing changes
- touching data layer

---

### 2.2 New fixed task: `code-facts-context`
**Outputs**
- `docs/contexts/<slug>/code-facts/public-entrypoints.md`
- `docs/contexts/<slug>/code-facts/core-dependencies.md`
- `docs/contexts/<slug>/code-facts/test-map.md`
- `docs/contexts/<slug>/code-facts/high-risk-modules.md`

**Purpose**
Emit a low-interpretation “code facts” layer that later workflows can trust.

**File requirements**

##### `public-entrypoints.md`
Must list:
- HTTP entrypoints
- CLI entrypoints
- worker/cron/queue entrypoints
- public package exports / SDK entrypoints if present
- where to begin for common feature work

##### `core-dependencies.md`
Must summarize:
- shared/core modules
- high fan-out modules
- major dependency directions
- modules that many features transitively depend on

##### `test-map.md`
Must summarize:
- where unit/integration/e2e tests live
- likely test companions for important modules
- weakly tested critical areas
- what kinds of changes should trigger test review

##### `high-risk-modules.md`
Must summarize:
- fragile shared modules
- broad-impact modules
- low-test-confidence hotspots
- modules with unusually high complexity or dependency centrality

**Rules**
- facts first
- concise structure
- avoid long prose
- mark uncertain relationships as inferred

---

### 2.3 New fixed task: `injection-context`
**Outputs**
- `docs/contexts/<slug>/context-packs/add-endpoint.md`
- `docs/contexts/<slug>/context-packs/bugfix.md`
- `docs/contexts/<slug>/context-packs/refactor-shared.md`
- `docs/contexts/<slug>/context-packs/review-change.md`
- `docs/contexts/<slug>/injection-index.yaml`

**Purpose**
Translate project understanding into task-oriented injection packs.

**Context pack requirements**

Each context pack must answer:
- when this pack should be used
- what files or modules to inspect first
- what dependencies or boundaries to remember
- what pitfalls commonly apply
- what tests should be reviewed
- what overreach to avoid

##### `add-endpoint.md`
For new API / route / handler work.

##### `bugfix.md`
For diagnosing and patching existing behavior.

##### `refactor-shared.md`
For changing shared/core/common utilities and cross-cutting code.

##### `review-change.md`
For reviewing diffs and validating likely impact.

##### `injection-index.yaml`
Defines injection priority by workflow stage and task type.

Suggested schema:

```yaml
always:
  - 00-summary.md
  - pitfalls/index.md

brainstorm:
  - architecture/system-overview.md

planning:
  - architecture/module-map.md
  - code-facts/public-entrypoints.md

implementing:
  - code-facts/public-entrypoints.md
  - code-facts/test-map.md
  - context-packs/bugfix.md

reviewing:
  - pitfalls/index.md
  - code-facts/high-risk-modules.md
  - context-packs/review-change.md
````

## Phase 3: Execute Tasks and Write Outputs

Keep the current backup / restore / partial-failure behavior unchanged.

### Additional output quality rules

All generated outputs must follow these rules:

1. prefer bulletized, scannable structure over long narrative
2. include paths, symbols, modules, or tests wherever useful
3. avoid vague advice like “be careful with performance”
4. distinguish:

   * observed fact
   * likely inference
   * unknown / unverified
5. optimize for later workflow injection, not for standalone documentation beauty

## README Assembly Rules

`docs/contexts/<slug>/README.md` should be assembled last and act as the context control panel.

It must include:

* project title / slug
* bootstrap timestamp
* analysis mode
* recommended reading order
* high-risk areas
* available context packs
* available layer docs
* database doc status
* injection-index presence
* quick links to all major outputs

Suggested section order:

1. Project Snapshot
2. Read This First
3. High-Risk Areas
4. Architecture
5. Code Facts
6. Context Packs
7. Layers
8. Database
9. Caveats and Freshness Notes

## PRD Contract Template

Every generated task PRD should follow this structure:

* Goal
* Target Output
* Why It Matters
* Data Sources
* Extraction Rules
* Output Requirements
* Acceptance Criteria
* Risks / Caveats
* Suggested Workflow

### PRD requirements

#### Goal

Describe what context gap this task closes for later workflows.

#### Data Sources

List prioritized sources:

* source code
* config
* schema
* tests
* entrypoints
* integrations
* optional MCP analysis

#### Extraction Rules

Always say:

* extract facts first
* do not invent architecture
* mark uncertainty
* prefer concise actionable output

#### Acceptance Criteria

Must be framed in terms of usefulness to later workflows:

* can later workflows decide where to look first?
* can they identify likely risks?
* can they find likely related tests?
* can they avoid scanning the whole repo?

---

## Minimum Quality Bar

A bootstrap run is only considered successful if, at minimum, later workflows can use the generated outputs to answer:

1. what is this project?
2. where does execution start?
3. where do major responsibilities live?
4. what are the riskiest areas?
5. which files/tests are likely relevant for common changes?
6. what should be injected first for implementation and review?

If those questions cannot be answered, the run should be treated as incomplete even if files were generated.

---

## Non-Goals

This workflow does not:

* rewrite application architecture
* guarantee perfect semantic understanding
* replace live code reading
* replace repository-specific review tools
* inject context automatically by itself unless the host framework explicitly wires that behavior

Its job is to create durable context assets that make later workflows faster, more grounded, and more consistent.

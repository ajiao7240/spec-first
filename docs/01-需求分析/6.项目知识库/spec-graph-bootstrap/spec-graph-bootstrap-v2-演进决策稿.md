# spec-graph-bootstrap v2 演进决策稿

> 本文用于收束 `spec-graph-bootstrap` 下一版的演进方向。
> 它不是再补一份背景分析，而是明确：为什么要升级、升级到哪里、优先做什么、哪些事情暂时不做。

---

## 1. 决策结论

`spec-graph-bootstrap` 下一版应从“项目上下文生成”升级为“研发质量资产生成”。

这意味着：

1. Stage-0 的目标不再只是帮助读懂仓库
2. Stage-0 要显式生产后续研发阶段可直接消费的高价值资产
3. 产物模型要从单一“架构认知”扩展为：
   - Understanding
   - Rules
   - Patterns
   - Decisions
   - Risks
4. 设计必须具备跨场景通用性，不能过拟合 Web 中后台前端

一句话概括：

> v2 的 `spec-graph-bootstrap` 不是“文档生成器”，而是“研发质量资产生产器”。

---

## 2. 为什么现在必须升级

## 2.1 v1 解决了什么

v1 已经解决了一个重要问题：

- 把项目最基本的上下文资产显式化
- 让仓库不再完全依赖口口相传
- 为后续阶段提供一个初步的项目认知入口

这些价值是成立的，尤其是：

- `00-summary.md`
- `architecture/*`
- `pitfalls/index.md`
- `layers/*`
- `database/*`

已经构成了一个可用的 Stage-0 基础。

## 2.2 v1 没解决什么

但如果目标是提升研发质量，v1 还有明显短板：

1. 文档偏“介绍型”，不够“执行型”
2. 没有独立的规则层，硬约束不够显式
3. 没有独立的样板层，最该抄的代码模式没有沉淀出来
4. 很少解释“为什么这样做”，缺关键决策资产
5. 高价值结论的证据链不够强
6. 对前端、后端、App、PC、CLI、数据等不同项目类型的统一抽象还不够

这会导致一个典型问题：

> 后续节点能“理解项目”，但不一定能“沿着团队最佳路径做事”。

这正是研发质量提升的关键瓶颈。

---

## 3. v2 要解决的核心问题

v2 不是简单扩充文档数量，而是解决 4 个上游问题：

### 3.1 如何让后续阶段少原创

必须让 agent 更容易命中：

- 最近似参考实现
- 标准骨架
- 稳定组合方式

否则 agent 只能从零写，导致风格漂移和 review 返工。

### 3.2 如何让后续阶段少犯规

必须把“不能写错什么”前置成一级资产，而不是散落在经验里。

否则会频繁出现：

- 能跑但不合规
- 逻辑对但绕过封装
- 看起来合理但破坏了平台约束

### 3.3 如何让后续阶段少误判

必须增强：

- 证据
- 可信度标记
- 过时区域提示
- 风险点前置

否则后续节点会把弱推断当强事实。

### 3.4 如何让后续阶段理解历史取舍

必须让关键设计决策显性化。

否则后续节点只看到代码表面，不理解：

- 为什么不能直接重构
- 为什么这里看起来“怪”
- 为什么这个方案不是技术上最优但仍必须保留

---

## 4. v2 的目标定义

## 4.1 目标

v2 的目标定义为：

1. 为目标仓库生成长期可复用的研发质量资产
2. 降低后续阶段的原创代码比例
3. 降低后续阶段违反项目约束的概率
4. 提高后续阶段命中既有代码模式的概率
5. 提高后续阶段对关键风险和历史决策的感知能力
6. 为后续 `plan / work / review / compound` 提供上游资产基础

## 4.2 非目标

v2 当前不做：

1. 不自动把所有资产注入后续阶段
2. 不做完整自动双向同步
3. 不替代人工架构决策
4. 不要求所有高价值资产都 100% 自动生成
5. 不追求用 Stage-0 一步解决整个知识治理体系

这个边界很重要。

v2 的重点是：

> 先把“生产什么资产”定义对，再逐步解决“怎么消费这些资产”。

---

## 5. v2 的核心设计决策

## D1. 从单一上下文资产升级为五类资产

v2 采用以下五类资产：

1. Understanding
2. Rules
3. Patterns
4. Decisions
5. Risks

原因：

- `Understanding` 解决看懂项目
- `Rules` 解决不能犯规
- `Patterns` 解决少原创、少走偏
- `Decisions` 解决理解历史取舍
- `Risks` 解决防错和防误判

这是 v2 最根本的结构调整。

## D2. Rules 和 Patterns 必须独立成层

不能把规则和样板埋在架构文档里。

原因：

1. 规则是执行底线，不是背景知识
2. 样板是实现骨架，不是架构说明
3. 后续阶段检索“约束”和“参考实现”时，需要直接命中一级资产

## D3. 文档必须任务导向

关键文档必须回答：

- 遇到什么任务时先看我
- 看完后去哪些文件
- 最像的参考实现在哪里
- 哪些地方不要动

这要求文档从“介绍型”变成“执行型”。

## D4. 所有高价值结论必须带证据和可信度

统一引入：

- `Verified`
- `Inferred`
- `Unknown`

高价值结论应尽量附：

- 真实路径
- 符号
- 配置键
- schema
- 参考实现

## D5. v2 必须场景中立

v2 不能默认项目是前端页面型工程。

要覆盖：

- 前端 Web
- 后端服务
- App
- 桌面端
- CLI
- 数据/ETL
- 异步任务

因此：

- `patterns/` 不能只围绕 UI
- worker 激活要由项目信号驱动
- 产物模型要先通用、后收敛

## D6. 自动生成与人工策展分层

建议分成两类：

### 自动生成层

- `summary`
- `architecture`
- `layers`
- `database`
- `evidence`
- `freshness`

### 人工策展层

- `rules`
- `patterns`
- `decisions`
- 高价值 `pitfalls`

原因：

- 机械事实适合自动化
- 高价值规则和样板不适合完全依赖自动推断

---

## 6. 产物演进决策

## 6.1 保留项

以下 v1 产物保留，但需增强：

- `README.md`
- `00-summary.md`
- `architecture/system-overview.md`
- `architecture/module-map.md`
- `architecture/integration-boundaries.md`
- `layers/*`
- `guides/index.md`
- `database/*`

## 6.2 新增项

新增 3 类核心资产目录：

- `rules/`
- `patterns/`
- `decisions/`

同时重构 `pitfalls/`，从单文件或弱结构风险说明升级成风险资产族。

## 6.3 弱化项

v1 中过于“介绍型”的结构应弱化，避免大量低密度文档堆积。

具体表现：

- 不鼓励只有概念介绍、没有路径和证据的章节
- 不鼓励写泛化框架说明替代项目内模式说明
- 不鼓励把规则、样板、风险混在一起

---

## 7. worker 演进决策

## 7.1 固定 worker 升级

在 v1 基础上，固定 worker 应扩展为：

- `summary-context`
- `architecture-context`
- `rules-context`
- `patterns-context`
- `decisions-context`
- `pitfalls-context`

这是 v2 的基本盘。

## 7.2 条件 worker 策略

条件 worker 不再围绕“前端专题”扩张，而是围绕“项目能力面”扩张。

例如：

- screen
- UI
- API
- state
- client-platform
- jobs
- CLI
- testing
- database
- stale risk
- historical constraints

这能保证同一套 worker 框架可服务多种仓库类型。

## 7.3 PRD 合同升级

PRD 必须从“文件写作说明”升级为“质量资产合同”。

新增关键字段：

- `Primary Questions`
- `Candidate Signals`
- `Required Evidence`
- `Task Entry Expectations`
- `Confidence Policy`
- `Freshness Risks`

这几项是 v2 的关键控制面升级。

---

## 8. 通用性决策

## 8.1 为什么必须强调通用性

如果 v2 过拟合中后台前端 CRUD，短期会显得很强，但长期会带来三个问题：

1. 一旦切到后端或 CLI 项目，价值快速衰减
2. 产物模型被任务类型绑死，难以复用
3. worker 和 PRD 设计会越来越像一套前端专项工具，而不是 Stage-0 通用能力

因此，v2 必须采用：

> 资产类型优先，具体项目类型后收敛

## 8.2 通用性落实方式

### 在产物层

- `rules/` 对所有项目类型成立
- `patterns/` 允许按项目能力面生成不同专题
- `layers/` 覆盖前端、后端、App、PC、CLI、data

### 在 worker 层

- 固定 worker 不依赖具体技术栈
- 条件 worker 由项目类型信号激活

### 在 PRD 层

- 不预设“页面开发”是默认任务
- 强调路径、约束、模式、证据这些跨场景通用问题

---

## 9. 质量衡量决策

v2 成功与否，不能再只看“文件生成齐不齐”。

建议采用三层指标：

## 9.1 产物指标

- 高价值结论带证据的比例
- 标记 `Verified` 的结论比例
- 高价值任务的样板覆盖率
- 高价值约束的规则覆盖率

## 9.2 下游指标

- 后续 `plan` 中内联示例代码长度下降
- 后续 `work` 的原创比例下降
- `review` 中风格偏差类问题下降
- `review` 中违反规则类问题下降
- 参考实现命中率上升

## 9.3 治理指标

- stale 文档比例
- rerun 后失效参考实现数量
- 新规则/新样板回灌速度

也就是说：

> v2 质量要通过下游误差是否减少来反推，而不是只看上游产物数量。

---

## 10. 分阶段落地决策

## Phase A：结构升级

先完成：

- `rules/`
- `patterns/`
- `decisions/`
- `pitfalls/` 重构
- 核心文档统一章节模板

目标：

- 把“产什么”定义清楚

## Phase B：控制面升级

再完成：

- worker 扩展
- PRD 模板升级
- assembly 质量检查增强

目标：

- 把“怎么稳定地产”定义清楚

## Phase C：证据与 freshness 升级

再补：

- `Verified/Inferred/Unknown`
- `Freshness`
- rerun 差异提示
- 参考实现失效检测

目标：

- 把“产物能不能信”做起来

## Phase D：消费对接

最后再接：

- `brainstorm`
- `plan`
- `work`
- `review`
- `compound`

目标：

- 把“好资产真正被用起来”接上

这个顺序的核心原因是：

> 如果上游资产模型不先定对，后面的消费编排只会放大错误。

---

## 11. 明确不做的事情

为了防止 v2 范围失控，以下事项暂时不纳入本轮设计：

1. 不做完整知识运营系统
2. 不要求自动生成所有高价值规则
3. 不要求自动识别所有历史决策
4. 不在 v2 里解决所有文档过时问题
5. 不做“代码与 spec 完全实时同步”
6. 不要求每类项目都在第一版支持到极致

v2 的目标是建立正确骨架，而不是一次性做完全部治理。

---

## 12. 需要立即推进的事项

基于当前知识库状态，建议立刻推进以下事项：

1. 以 `spec-graph-bootstrap-v2-产物清单明细表.md` 作为新的产物定义底稿
2. 以 `spec-graph-bootstrap-v2-worker-任务拆分与PRD模板升级稿.md` 作为控制面升级底稿
3. 把现有 v1 文档明确标记为历史版，避免误用
4. 下一步进入“v2 规范化文稿”，形成更接近最终设计文档的版本

---

## 13. 最终判断

`spec-graph-bootstrap` 的下一版不能只做“一轮增强”，因为问题不是“现有文档还不够详细”，而是**产品定义本身还停留在认知层**。

真正的升级方向是：

> 从“帮人理解仓库”  
> 升级为  
> “为后续研发阶段预先准备好规则、样板、决策、风险和证据”

这才是能长期提升研发质量的 Stage-0。

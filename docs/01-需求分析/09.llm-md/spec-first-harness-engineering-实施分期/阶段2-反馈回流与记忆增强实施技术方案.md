# 阶段2：反馈回流与记忆增强实施技术方案

本文是主方案第二阶段的展开稿，对应主方案第 13.2 节。

适用范围：

- `spec-code-review`
- `spec-compound`
- `history`
- `history-spec-index`
- shared knowledge 双路召回
- bootstrap / work / review trace

本文假定阶段 1 已完成最小闭环，否则阶段 2 会失去稳定输入。

## 1. 阶段定位

阶段 2 的目标，不是再加一层新流程，而是把阶段 1 已经落地的执行资产变成复利资产。

换句话说，阶段 1 解决“这次怎么做”，阶段 2 解决：

- 下次怎么少犯
- 下次怎么少原创
- 经验到底该回流到哪里
- 相似历史需求如何稳定召回

## 2. 阶段目标

### 2.1 业务目标

- 让 `spec-code-review` 从一次性判题器升级为候选提炼器。
- 让 `spec-compound` 从“手工写总结”升级为“信号驱动知识路由器”。
- 让 `history / decision notes / history-spec-index` 能被 plan/work/review 真正复用。
- 让 greenfield 早期的 starter/template 经验开始转化为 repo-local 经验。

### 2.2 技术目标

- 补齐 review 双候选输出与快速更新提案通道。
- 建立 decision notes 与 history index 的召回链。
- 建立 shared knowledge 的双路召回：语义召回 + 索引导航召回。
- 建立 trace taxonomy，避免 probe、work、review failure 混写。

## 3. 范围与边界

### 3.1 In Scope

1. `spec-code-review/<run-id>/meta.json`
2. `rule-candidates.json`
3. `pattern-candidates.json`
4. `fast-track-proposals.json`
5. `suggested_destination`
6. `pitfalls-specialist` 等专项 reviewer 路由
7. `.context/spec-first/history/<spec-id>/decision-notes.md`
8. `.context/spec-first/knowledge/history-spec-index.json`
9. `spec-compound` 的 procedure / pattern / failure / automation candidate
10. work/review/bootstrap trace 统一建模
11. shared knowledge 双路召回

### 3.2 Out of Scope

- `spec-improve`
- readiness 评分
- 自动写回 verify hints / instruction-context / reference 真源
- 中心化知识平台或远程控制台

## 4. 阶段架构原则

1. `candidate first, writer later`
   阶段 2 的核心是提出有证据的候选，而不是自动治理。
2. `review is not just judge`
   review 必须同时产生当前 diff 的问题和未来可复用的改进候选。
3. `knowledge routing over knowledge pile-up`
   目标是把经验送回正确源头，不是不断扩大一个大而全经验库。
4. `history is executable context`
   `decision-notes` 和 `history-spec-index` 不是档案馆，而是下次 planning 和 review 的输入。

## 5. 关键实现对象

### 5.1 `spec-code-review`

#### 新职责

阶段 2 是 `spec-code-review` 候选输出能力的**首次引入**。以下四类产物在阶段 1 中被列为 Out of Scope，由本阶段首次实现：

- 输出 `findings.json`
- 输出 `rule-candidates.json`（首次产出，非继承自阶段 1）
- 输出 `pattern-candidates.json`（首次产出，非继承自阶段 1）
- 输出 `fast-track-proposals.json`（首次产出，非继承自阶段 1）
- 根据 `high_risk_areas`、pitfalls 和 preflight / verification warning 触发专项 reviewer

**运行产物路径：** `spec-code-review` 每次执行的产物写入 `.context/spec-first/spec-code-review/<run-id>/`。此路径与阶段 1 的 `.context/spec-first/work/<run-id>/` 平行，统一受 `run_id` 规范约束（格式 `YYYYMMDDTHHmmss-XXXX`）。若现有 skill 实现中使用了其他路径，在阶段 2 实施前须迁移到 `.context/spec-first/spec-code-review/<run-id>/`，并更新 doctor 诊断路径。

#### 最小数据契约

##### `findings.json`

`spec-code-review` 每次执行后写入 `.context/spec-first/spec-code-review/<run-id>/findings.json`，是后续 candidates / proposals 的原始输入来源，也是阶段 3 `spec-improve` 的直接输入。

最小字段：

- `schema_version`
- `generated_at`
- `run_id`
- `spec_id`（关联 plan 文档标识，与 work meta.json 保持一致）
- `reviewer_personas`（参与本次 review 的 persona 列表）
- `findings[]`
  - `finding_id`
  - `severity`（`blocking` | `major` | `minor` | `info`）
  - `category`（`correctness` | `security` | `architecture` | `pitfall` | `style` | `other`）
  - `location`（文件路径或范围，允许为 `null`）
  - `description`
  - `evidence`（具体代码片段或信号来源，至少一条）
  - `is_recurring`（是否与历史 finding 同类，布尔值）
  - `suggested_action`（`fix` | `review-manual` | `escalate-to-candidate` | `ignore`）

消费约束：

- `findings[]` 是 `rule-candidates.json`、`pattern-candidates.json`、`fast-track-proposals.json` 的数据来源，不允许直接跳过 findings 生成 candidate
- `spec-improve` 读取时只能读 `findings[]` 的聚合统计，不得把单次 finding 当成长期规则

##### `rule-candidates.json`

最小字段：

- `schema_version`
- `generated_at`
- `candidates[]`
  - `candidate_type`
  - `source_finding`
  - `pattern`
  - `suggested_target`
  - `suggested_destination`
  - `confidence`

##### `pattern-candidates.json`

最小字段：

- `schema_version`
- `generated_at`
- `candidates[]`
  - `candidate_type`
  - `source_finding`
  - `path`
  - `reason`
  - `suggested_destination`
  - `confidence`

##### `fast-track-proposals.json`

最小字段：

- `schema_version`
- `generated_at`
- `proposals[]`
  - `proposal_type`
  - `source_finding`
  - `target_asset`
  - `reason`
  - `evidence[]`
  - `human_action_required`

#### 回流门槛

只有同时满足以下条件才能进入 candidate 或快速提案：

- 可重复
- 可抽象
- 有足够证据
- 不仅是一次性业务噪音

#### 快速更新通道的边界

- 它只产生提案，不直接修改真源
- 它只适用于高危、明确、复发概率高的问题
- 首版只覆盖少数 target asset：`verify-hints`、`instruction-context`、reference 质量、pitfall entry

### 5.2 专项 reviewer 路由

本阶段至少落地一类高价值专项 reviewer：

- `pitfalls-specialist`

**触发条件（满足任一即路由）：**

- `findings.json` 中存在 `category == "pitfall"` 的 finding
- `instruction-context.json` 的 `high_risk_areas` 中有字段与本次变更的目标路径重叠
- `preflight.json` 或 `verification.json` 中存在 `WARN` 级条目

**路由机制：** `spec-code-review` 主流程在生成 `findings.json` 后，检查上述条件。若触发，将以下上下文传给 `pitfalls-specialist` 子 reviewer，独立运行后合并结果到 `findings.json`（新增或更新对应 finding 的 `suggested_action`）。

输入：

- `pitfalls/index.md`（路径：`docs/contexts/<slug>/pitfalls/index.md`）
- `verify-hints.json`
- `instruction-context.json` 中的 `high_risk_areas`
- `preflight.json` 的 `matched_rules`
- `verification.json` 的警告项

输出：

- 是否命中已知风险（对应 `is_recurring: true`）
- 是否复现历史反模式（可升级 `suggested_action` 为 `escalate-to-candidate`）
- 是否应升级为 rule candidate 或 fast-track proposal（写入对应 candidate 文件）

**降级约束：** 若 `pitfalls/index.md` 不存在，`pitfalls-specialist` 不阻断主流程，只在 `findings.json` 元数据中标记 `pitfalls_specialist_skipped: true`。

### 5.3 `spec-work` 的 history 沉淀

本阶段 `spec-work` 新增 `decision-notes.md` 的稳定写入语义。

写入路径：`.context/spec-first/history/<spec_id>/decision-notes.md`

写入触发条件：

- 实现与 design 明显偏离
- 发现新的强约束或隐性依赖
- reference 的复用理由与边界值得保留

**写入责任链：**

1. `spec-work` 在执行结束前，若触发任一写入条件，写入 `decision-notes.md`
2. `spec-work` 写入后，**主动通知**（通过在 `meta.json` 的 `decision_note_written: true` 字段标记）下游有更新
3. `history-spec-index.json` 的更新由 `spec-compound` 在 compound 执行时消费 `meta.json` 并更新，而非由 `spec-work` 直接写入 index
4. 若 `spec-compound` 未运行，index 可能存在滞后；消费方（`spec-plan`、`spec-code-review`）应能容忍 index 与 notes 存在不一致，以 `decision-notes.md` 文件内容为准

要求：

- `decision-notes.md` 是 `history-spec-index.json` 的摘要真源
- index 只能存摘要和索引信息，不得扩展成比 notes 更完整的新真源

### 5.4 `history-spec-index.json`

目标：

- 为 `spec-plan` 提供 Historical Analogs
- 为 `spec-code-review` 提供历史风险与决策上下文
- 为 `spec-compound` 提供去重和更新判断

最小字段建议：

- `schema_version`
- `entries[]`
  - `spec_id`
  - `slug`
  - `source_plan`
  - `source_section`
  - `decision_note_path`
  - `summary`
  - `keywords`
  - `related_paths`

### 5.5 `spec-compound`

#### 新职责

- 从 work/review/bootstrap 信号判断“是否值得沉淀”
- 判断沉淀应该去哪，而不是默认写 `docs/solutions/`

#### 四类 memory

1. `solution memory`
2. `procedure memory`
3. `pattern memory`
4. `failure memory`

#### automation candidates

当同类流程满足以下门槛时，才可输出 automation candidate：

- `evidence_count >= 3`（同类流程至少成功 3 次，跨不同 spec_id）
- `stability = high`（最近 3 次执行均无 `partial` 或 `failed` outcome，步骤顺序无明显变化）
- 没有未解决的 `blocking` 级 finding 关联到该流程

输出字段：

- `task_family`（流程类型标识，如 `add-api-endpoint`、`migrate-schema`）
- `evidence_count`（满足门槛的成功执行次数）
- `stability`（`high` | `medium` | `low`）
- `evidence_spec_ids[]`（来源 spec_id 列表，供追溯）
- `suggested_next_step`（建议人工采纳的下一步，如 `promote-to-template` | `review-steps-for-script`）

**人工确认要求（与全局 never-auto-write 约束对齐）：**

automation candidate 不是 never-auto-write 约束的例外。输出 automation candidate 仅表示"有证据支持对该流程进一步自动化"，但：

- 不自动生成脚本
- 不自动替换现有 workflow 步骤
- `suggested_next_step` 必须是人工评估后才能采纳的建议（如 `promote-to-template` 或 `review-steps-for-script`），不得为任何自动执行类动作
- 采纳路径：人工审阅 candidate → 确认纳入 → 由人工或显式触发的 CLI 命令执行写入

若未来需要引入自动采纳路径，须在方案中额外说明权限边界、证据门槛和回滚策略，不能在此约束范围内隐式升级。

## 6. 双路召回设计

阶段 2 必须把 shared knowledge 从“存在入口”推进到“可以稳定检索”。

### 6.1 两条召回路径

1. **语义召回** — 解决”应该看什么”

   最小实现：LLM in-context 排序。将 `knowledge/points.json` 中相关分类的 knowledge point 列表传入 prompt，由 LLM 根据当前任务描述排序并选出最相关的 3-5 条作为参考上下文。**不要求**引入向量数据库或外部嵌入服务；阶段 2 只需 LLM 能在单次对话中完成排序判断即可。

2. **索引导航召回** — 解决”精确去哪里”

   最小实现：关键词匹配。`history-spec-index.json` 的每条 entry 有 `keywords[]` 字段，召回时对当前任务关键词做精确匹配（子串即可），命中后按 `decision_note_path` 定位 `decision-notes.md`。**不要求**模糊匹配或向量相似度；阶段 2 只需精确词匹配能命中即可。

### 6.2 最小落地点

- `knowledge/sources.json`
- `knowledge/points.json`
- `knowledge/retrieval-policy.json`
- `history-spec-index.json`
- `decision-notes.md`
- `docs/contexts/<slug>/patterns/index.md`

### 6.3 消费侧接入说明

各下游 skill 在不同阶段触发召回的最小接入方式：

**`spec-plan`**

1. 读取 `knowledge/retrieval-policy.json`，确认当前任务类型匹配哪类知识源
2. 用任务关键词在 `history-spec-index.json` 的 `keywords` 字段做精确索引导航，定位 Historical Analogs
3. 若索引命中，按 `decision_note_path` 读取对应 `decision-notes.md`，提取决策约束与差异点
4. 若索引未命中，退化为语义召回：读取 `knowledge/points.json` 中相关分类的 knowledge point，作为参考上下文

**`spec-work`**

1. 读取 `reference-index.json` 中 `validated` 引用作为首选参考（阶段1 已有）
2. 额外读取 `docs/contexts/<slug>/patterns/index.md`，确认当前任务领域是否有已验证的代码模式
3. 实现过程中若偏离 design，在 `decision-notes.md` 写入决策记录，并通知 history-spec-index 更新摘要

**`spec-code-review`**

1. 读取 `knowledge/points.json` 中 `failure` 类 knowledge point，与 `findings.json` 做交叉比对（识别 `is_recurring`）
2. 读取 `history-spec-index.json`，检查当前 spec 的 Related Paths 是否有历史决策未被本次 plan 引用
3. 以上两类命中，可升级对应 finding 的 `suggested_action` 为 `escalate-to-candidate`

> **降级约束：** 任何召回路径在资产缺失时均不得报硬错误，只能在输出元数据中标记 `retrieval_available: false`。

### 6.5 阶段 2 的保守约束

- 不要求做统一向量平台
- 不要求做远程 source registry
- 不要求引入复杂召回编排器
- 先让 repo-local 与 runtime-managed pack 走通双路召回即可

## 7. Trace 统一建模

阶段 2 要解决的不是”多存日志”，而是 failure taxonomy 混乱。

### 7.1 Trace 分类与存放路径

| 文件 | 归属阶段 | 路径 | 语义 |
| --- | --- | --- | --- |
| `probe-failures.json` | bootstrap | `.context/spec-first/bootstrap/<slug>/analysis/` | 探针探测失败，缺 evidence |
| `worker-failures.json` | bootstrap | `.context/spec-first/bootstrap/<slug>/analysis/` | worker 执行中断、输出不完整 |
| `verify-warnings.json` | bootstrap | `.context/spec-first/bootstrap/<slug>/analysis/` | 验证阶段的警告，非阻断 |
| `signals.json` | work | `.context/spec-first/work/<run-id>/` | 执行过程中的关注信号（已由阶段1定义） |
| `findings.json` | review | `.context/spec-first/spec-code-review/<run-id>/` | review 发现（已由本阶段 §5.1 定义） |

### 7.2 各 Trace 文件最小字段

#### `probe-failures.json`

- `schema_version`
- `slug`
- `generated_at`
- `failures[]`
  - `probe_id`（探针标识）
  - `target`（探测目标路径或命令）
  - `reason`（失败原因，`not-found` | `command-error` | `parse-error` | `timeout` | `other`）
  - `impact`（对下游 analysis 资产的影响说明）

#### `worker-failures.json`

- `schema_version`
- `slug`
- `generated_at`
- `failures[]`
  - `worker_id`（worker 标识）
  - `asset`（尝试产出的目标资产名）
  - `reason`（失败原因）
  - `partial_output`（是否有部分产出，布尔值）
  - `fallback_applied`（是否已触发降级，布尔值）

#### `verify-warnings.json`

- `schema_version`
- `slug`
- `generated_at`
- `warnings[]`
  - `warning_id`
  - `asset`（触发警告的资产名）
  - `field`（具体字段，允许为 `null`）
  - `message`
  - `severity`（`warn` | `info`，不得为 `error`）

### 7.3 类型隔离要求

- Phase 1 探针失败（`probe-failures`）不能和 worker 失败（`worker-failures`）混类写入同一文件
- `verify-warnings` 只允许 `warn` / `info`，阻断级错误必须通过 worker-failures 体现
- review candidate 不能和真实 finding 混类（finding 是事实，candidate 是推断）
- signal 只能表示”值得关注”，不能冒充知识或直接作为 rule 依据

## 9. 阶段验收标准

1. `spec-code-review` 能稳定输出 `meta.json`、`rule-candidates.json`、`pattern-candidates.json`。
2. candidate 都带 `suggested_destination`，且 destination 语义不含糊。
3. `spec-code-review` 能输出 `fast-track-proposals.json`，且不会静默写回主资产。
4. 至少一种专项 reviewer 能基于 `pitfalls + high_risk_areas + preflight/verification` 真正触发。
5. `spec-work` 能写 `decision-notes.md`，且 `history-spec-index.json` 能索引到它。
6. `spec-plan` 能消费 `history-spec-index.json` 与 `decision-notes.md` 做 Historical Analogs。
7. `spec-compound` 不再只写 `docs/solutions/`，而是能输出 procedure / pattern / failure / automation candidate。
8. shared knowledge 至少在 repo-local 范围支持“语义召回 + 索引导航”两条路径。

## 10. 阶段风险与缓解

### 10.1 风险：review 变成另一个 writer

缓解：

- candidate 与 fast-track proposal 默认都是提案
- 任何写回动作都留给人工采纳或后续显式 capability

### 10.2 风险：knowledge routing 变成知识堆积

缓解：

- `suggested_destination` 必须分类清楚
- compound 先做路由判断，再决定是否落文档

### 10.3 风险：history index 演化成第二真源

缓解：

- `decision-notes.md` 才是决策细节真源
- `history-spec-index.json` 只存摘要、关键词与索引

### 10.4 风险：双路召回过早平台化

缓解：

- 阶段 2 只做 repo-local 与 runtime-managed pack 的最小召回
- 暂不引入重型 source registry 或统一后台

## 11. 本阶段完成标志

完成阶段 2 的标志不是“生成了更多候选文件”，而是下面三件事同时成立：

1. review 的结果开始影响下次 planning 和 work，而不是留在一次性报告里。
2. 历史决策和已验证 pattern 能被后续需求稳定召回。
3. compound 开始把经验送回正确源头，而不是把所有内容都堆进 solutions 或经验库。

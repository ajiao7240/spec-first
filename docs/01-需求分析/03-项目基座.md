# Spec-First 一期改造清单

> 说明：本文件保留 Every 的方法论背景作为历史参考，但当前实现与安装方式已经切换为 `spec-first` 的 npm CLI 模型。

## 一、改造目标

以 `spec-first` 的 npm CLI 入口为实现载体，吸收五步闭环方法论作为工作流骨架，落地一套面向 Claude Code 的 **Spec-First Framework** 一期版本，实现：

* 五步闭环可用
* 任务按目录聚合
* 最小状态管理可用
* review 形成质量闸门
* compound 可沉淀资产
* 后续需求可复用经验

不引入多模型协同、复杂工作流引擎和重运行时能力。

---

## 二、一期保留内容

## 1. 保留五步主链骨架

从历史参考中保留并映射以下主链能力：

* `brainstorm`
* `plan`
* `work`
* `review`
* `compound`

改造后的统一命名：

* `/spec:brainstorm`
* `/spec:plan`
* `/spec:work`
* `/spec:review`
* `/spec:compound`

这是一期最核心的保留部分。

## 2. 保留“闭环 + 复利”方法论

保留历史参考中的核心精神，但用当前实现语言重写为：

* 需求先发散再收敛
* 执行前有计划
* 执行后有 review
* 合格成果进入 compound
* compound 产物进入后续任务复用

这一层是方法论骨架，不需要重写发明。

## 3. 保留 review 作为独立阶段

一期必须保留 review 的独立地位，不把它并入 work。
因为你后面的质量提升和经验沉淀，都依赖 review 先做过滤。

## 4. 保留 compound 作为独立阶段

compound 不只是总结，而是进入资产沉淀入口。
这一步必须保留，否则“自我进化”目标落不下来。

## 三、一期必须改造的内容

## 1. 命令命名改造

把历史参考中的 `ce:*` 体系改成当前的 `spec:*`：

* `/ce:brainstorm` → `/spec:brainstorm`
* `/ce:plan` → `/spec:plan`
* `/ce:work` → `/spec:work`
* `/ce:review` → `/spec:review`
* `/ce:compound` → `/spec:compound`

目的：

* 与你的设计文档统一
* 建立自有品牌和清晰语义
* 避免直接继承上游命名包袱

## 2. 产物目录改造

不要沿用 Every 原有的知识沉淀目录作为运行态主目录。
一期统一改成你的任务目录模型：

```text
.claude/tasks/<task-id>/
  task.yaml
  01-brainstorm.md
  02-plan.md
  03-work.md
  04-review.md
  05-compound.md
```

这项是最重要的工程改造之一。
因为你要的是**按任务聚合的 spec-first 运行态**，不是直接复用 Every 原有的目录语义。

## 3. 增加 `task.yaml` 最小控制面

历史参考中的上游实现不是围绕你定义的 `task.yaml` 工作的，所以一期要补上这一层：

最小字段集保留你文档定义：

* `task_id`
* `title`
* `role`
* `level`
* `current_stage`
* `status`
* `created_at`
* `updated_at`

用途：

* 保存当前任务最小状态
* 支撑阶段流转
* 支撑恢复最基本上下文
* 不演化成复杂工作流引擎

## 4. 输出格式统一改造

五个阶段的主要产物统一使用：

* `frontmatter + markdown`

特别是 review 和 compound，要形成可机器消费、可人工阅读的双层结构。

## 5. 参数体系统一

五个 `spec:*` skills 统一支持：

```bash
--role <product|dev|fullstack|test>
--level <L1|L2|L3>
--task-id <task-id>
```

说明：

* `--role` 是视角模式，不是独立岗位系统
* `--level` 默认自动建议并采用
* `--task-id` 未传时自动生成

这套参数层是你自有框架的统一入口，要在一期直接定住。

## 6. review 机制改造成正式守门节点

一期要把 review 从”可选检查”改造成”正式守门阶段”。

必须实现的内容：

* 四维评分（正确性、完整性、可执行性、可复用性）
* 问题分级（blocking / major / minor）
* 判定字段（`pass` / `rework_required` / `compound_recommended`）
* review 正文说明
* 双重判定规则（关键维度底线 + 阻断问题优先 + 总分参考）

输出格式：frontmatter + markdown 混合格式

## 7. compound 改造成“任务沉淀 + 资产迁移”双层结构

一期不要只生成 `05-compound.md` 就结束。
需要明确分成两层：

### 任务内沉淀

* `05-compound.md`

### 资产沉淀

* `.claude/assets/prompts/`
* `.claude/assets/workflows/`
* `.claude/assets/domain/`
* `.claude/assets/cases/`

也就是：

* `05-compound.md` 是本任务记录
* `.claude/assets/` 是长期复用资产区

## 四、一期必须实现的核心机制

以下机制是设计文档中定义的核心能力，必须在一期实现。

## 1. 阶段契约与守门机制

每个 skill 执行时的标准流程（六步）：

1. **上游产物检查**：验证上一阶段产物文件是否存在，并满足基本可消费条件
   - 文件存在且非空
   - 如果有 frontmatter，则 `status` 不处于失败态
   - 对 review 之后的流转，必须 `pass=true`

2. **元数据读取**：读取 `task.yaml`，确认当前状态和任务信息

3. **level 决策**：如未指定 `--level`，根据任务描述分析并建议任务级别，默认采用并提示可覆盖

4. **输入完整性检查**：根据确定的 level 验证必需输入是否完整

5. **执行与输出**：生成阶段产物（markdown 文件）

6. **状态更新**：更新 `task.yaml` 的 `current_stage`、`status` 和 `updated_at`，遵循以下规则：
   - skill 成功完成并产出可消费结果时，方可推进 `current_stage`
   - skill 执行失败时更新 `status=failed` 或 `blocked`
   - review 不通过时不得推进到 compound

## 2. Rationale capture

在 `brainstorm`、`plan`、`compound` 中增加固定章节：

* 为什么选这个方案
* 为什么不选其他方案
* 当前选择的适用边界

这是支持"自我进化"的关键机制。

## 3. Review 双重判定机制

review 阶段必须实现完整的判定规则：

**规则1：关键维度底线**
- 正确性 < 3 → 不通过
- 完整性 < 3 → 不通过

**规则2：阻断问题优先**
- 存在 `blocking_issues > 0` → 默认不通过

**规则3：总分参考**
- 总分 < 12：通常不通过
- 总分 12-15：作为人工参考区间
- 总分 ≥ 16：可进入通过候选

`summary_score` 仅作为参考，不作为唯一通过依据；是否通过以阻断问题、关键维度和返工要求综合判断。

## 五、一期明确删除或不引入的 Every 内容

## 1. 不纳入主链的命令/skill

一期不集成或不优先集成：

* `ideate`
* `compound-refresh`
* `changelog`
* `feature-video`
* `sync`
* `onboarding`
* `todo-resolve`
* `todo-triage`
* `test-browser`
* `test-xcode`
* `reproduce-bug`
* `report-bug-ce`
* `resolve-pr-feedback`

原因：
这些属于外围工具型、周边协作型、平台生态型能力，不是一期主链刚需。

## 2. 不引入大而全 reviewer persona 体系

一期不保留 Every 那套很厚的 review persona 矩阵。
只保留你的轻量 review 机制即可。

原因：

* 降低理解成本
* 降低维护成本
* 避免把 review 做成复杂平台

## 3. 不引入平台分发和多宿主转换逻辑

历史上的平台型能力一期全部不接。

不做：

* 多宿主插件转换
* marketplace 相关能力
* 跨 CLI 分发

因为当前一期已明确聚焦 Claude Code CLI 的 npm CLI 使用模式。

## 六、一期明确不引入 Adam 的内容

为了防止后续跑偏，也一起写死：

* 多模型 red-team
* session recovery

* beads/task tracking runtime
* memory tiers
* classify-stats / process stats
* worktree/session 级复杂隔离
* 高复杂度自动代理链路

这些全部延期到二期以后。

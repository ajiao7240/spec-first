# 阶段3：系统自进化实施技术方案

本文是主方案第三阶段的展开稿，对应主方案第 13.3 节。

适用范围：

- `spec-improve`
- readiness 评分
- 过程代理指标
- outcome metrics
- Highest ROI gaps

本文假定阶段 1 和阶段 2 已经稳定提供：

- bootstrap 控制面
- work 持久化
- review candidates / proposals
- history / compound / routing 输入

否则阶段 3 只能输出失真的评分。

## 1. 阶段定位

阶段 3 的目标，不是让系统“自动修自己”，而是让系统开始稳定识别：

- 哪些地方最缺收益
- 哪些资产在漂移
- 哪些过程指标正在失真
- 哪些改造最值得优先投入

因此，`spec-improve` 的默认定位必须是：

- auditor
- aggregator
- proposer

而不是：

- automatic writer
- universal fixer
- hidden governance engine

## 2. 阶段目标

### 2.1 业务目标

- 让 `spec-first` 从 workflow 集合演进为能自我诊断的交付系统。
- 给团队一个稳定的“下一步最值钱改哪里”的判断入口。
- 用结果指标反校验过程指标，避免系统只优化代理变量。

### 2.2 技术目标

- 定义 `spec-improve` 的输入面、输出面和权限边界。
- 定义 readiness 评分维度。
- 定义运营指标与 outcome metrics。
- 定义 Highest ROI gaps 的生成规则。

## 3. 范围与边界

### 3.1 In Scope

1. `spec-improve`
2. agent-readiness audit
3. process metrics 汇总
4. outcome metrics 汇总
5. Highest ROI gaps
6. stale source / stale asset 的识别提案
7. routing / template / rule / pattern 的更新提案

### 3.2 Out of Scope

- 自动写回 `verify-hints.json`
- 自动写回 `instruction-context.json`
- 自动改 `reference-index.json`
- 自动改 repo-root instruction file
- 自动生成脚本并替换现有 workflow

## 4. 核心原则

1. `improve is proposer, not writer`
   首版只输出报告、提案、汇总索引。
2. `outcome validates process`
   过程指标只有在带来结果改善时才算有效。
3. `highest ROI over maximal completeness`
   阶段 3 不是把所有缺口罗列一遍，而是优先识别最值得做的少数项。
4. `human judgment remains in the loop`
   评分和提案可以自动生成，采纳与写回必须显式确认。

## 5. `spec-improve` 输入面设计

至少读取：

- bootstrap analysis asset
- work `meta.json`
- work `preflight.json`
- work `verification.json`
- work `signals.json`
- review `findings.json`
- review `rule-candidates.json`
- review `pattern-candidates.json`
- review `fast-track-proposals.json`
- history `decision-notes.md`
- `history-spec-index.json`
- compound memory / automation candidates

**Harness-enabled 与 Reduced-harness 输入矩阵：**

| 输入资产 | Harness-enabled 要求 | Reduced-harness 处理 |
| --- | --- | --- |
| `analysis.json` | 必须存在，作为 readiness 基线 | 缺失时 `instruction_readiness` / `context_readiness` 评分降级，标注 `evidence: none` |
| `reference-index.json` | 必须存在，作为 `reference_readiness` 依据 | 缺失时 `reference_readiness` 评分降级 |
| `verify-hints.json` | 必须存在，作为 `verification_readiness` 依据 | 缺失时 `verification_readiness` 评分降级 |
| `instruction-context.json` | 必须存在 | 缺失时 `instruction_readiness` 最低评级 |
| `preflight.json` | 至少 1 条历史记录 | 缺失时过程指标 `bootstrap_context_hit_rate` 无法计算，标注 `unavailable` |
| `verification.json` | 至少 1 条历史记录 | 缺失时 `first_pass_success_rate` 等 outcome metrics 无法计算 |
| `signals.json` | 至少 1 条历史记录 | 缺失时 `warn_count` 等代理指标不可用 |
| `findings.json` | 至少 1 条历史记录 | 缺失时 `feedback_loop_readiness` 最低评级 |
| `rule-candidates.json` | 可选，有则更佳 | 缺失不影响评分，但 Highest ROI gaps 中 "feedback loop" 方向的 gap 更难定量 |
| `pattern-candidates.json` | 可选 | 同上 |
| `fast-track-proposals.json` | 可选 | 同上 |
| `decision-notes.md` | 可选，有则为 `knowledge_ops_readiness` 提供证据 | 缺失时 `decision_note_reuse_rate` 不可计算 |
| `history-spec-index.json` | 可选 | 缺失时 `knowledge_ops_readiness` 只能依赖 sources/points.json |

**总体降级策略：** Reduced-harness 模式下，`spec-improve` 不得因输入缺失而硬失败。每个缺失资产对应的维度评分降级为 `insufficient-data`（不评分但标注），Highest ROI gaps 中自动纳入"补齐该资产"作为一条高优先级 gap。

要求：

- 读取时只能消费已有真源或稳定投影
- 不得自行发明一套新的长期状态文件作为第四真源

## 6. 输出面设计

### 6.1 Agent Readiness Audit

最小输出：

- readiness 各维度分数
- Highest ROI gaps
- 推荐改造顺序

### 6.2 提案类输出

允许输出：

- 模板建议
- rule / pattern 汇总提案
- knowledge routing 建议
- stale source 修复提案
- content source update proposal

要求：

- 输出的是 `proposal`，不是直接修改
- 每个 proposal 都应带证据来源和影响面说明

## 7. 评分维度设计

首版至少包含：

1. `instruction_readiness`
2. `context_readiness`
3. `reference_readiness`
4. `knowledge_ops_readiness`
5. `verification_readiness`
6. `feedback_loop_readiness`

### 7.1 评分解释要求

- 分数必须能解释，不允许纯黑箱
- 每个分数至少对应：
  - 当前证据
  - 主要缺口
  - 一条可执行提升建议

### 7.2 Highest ROI gaps 生成规则

至少综合：

- 风险严重度
- 复发频率
- 对交付主链的影响
- 修复成本
- 是否已经有候选提案可直接落地

## 8. 指标体系设计

### 8.1 过程代理指标

阶段 3 沿用并汇总：

1. `bootstrap_context_hit_rate`
2. `reference_anchor_rate`
3. `inline_template_ratio`
4. `doubt_resolution_rate`
5. `pitfalls_hit_rate`
6. `pitfalls_action_rate`
7. `decision_note_reuse_rate`
8. `rewrite_ratio`

### 8.2 Outcome Metrics

必须显式纳入：

1. `first_pass_success_rate`
2. `spec_to_review_lead_time`
3. `blocking_finding_density`
4. `post_plan_rework_count`

### 8.3 指标解释原则

- 过程指标变好但 outcome 不变，优先怀疑代理指标失真
- outcome 变坏时，不允许只靠再加 guardrail 解决
- 指标只用于定位问题，不直接变成自动惩罚或自动写回依据

**指标失真检测机制（可操作）：**

`spec-improve` 在每次 readiness 报告中必须执行以下背离检测，若检测到背离则在报告中标注 `metric_divergence_warning`：

| 检测项 | 触发条件 | 建议动作 |
| --- | --- | --- |
| 过程指标与结果指标背离 | 过程指标（如 `bootstrap_context_hit_rate`、`reference_anchor_rate`）环比提升 ≥ 10%，但 `first_pass_success_rate` 或 `blocking_finding_density` 环比无改善或下降 | 标注"代理指标可能失真"，建议人工核查最近 3 条 spec 的执行路径 |
| 指标覆盖率过低 | 计算某一过程指标所依赖的输入资产数 < 3（即样本不足） | 标注"统计样本不足，指标仅供参考"，不使用该指标参与 Highest ROI gap 排序 |
| 同一维度连续最高分 | 某 readiness 维度连续 3 次评分为最高等级，但对应 outcome 指标无改善 | 提示"该维度可能已是非瓶颈，考虑将优化资源转向其他维度" |

## 9. 与前两阶段的依赖关系

### 9.1 对阶段 1 的依赖

- 没有 analysis / reference / verify-hints / instruction-context，就没有稳定的 readiness 基线
- 没有 preflight / verification / signals，就没有可分析的执行信号

**阶段 1 退出验证代理指标（`spec-improve` 可读取作为历史基线）：**

| 代理指标 | 来源资产 | 最小可计算条件 |
| --- | --- | --- |
| `bootstrap_context_hit_rate` | `meta.json` 的 `bootstrap_slug` 非 null 比例 | 至少 2 条 `meta.json` 记录 |
| `reference_anchor_rate` | plan 文档中 `References` section 引用 `validated` 条目的比例 | 至少 2 条 plan 文档 |
| `preflight_coverage` | `preflight.json` 的 `matched_rules` 非空执行次数 / 总 Harness-enabled 执行次数 | 至少 3 条 Harness-enabled 记录 |

### 9.2 对阶段 2 的依赖

- 没有 review candidates / fast-track proposals，就没有稳定的改造建议来源
- 没有 decision notes / history index / compound routing，就没有跨需求复利输入

**阶段 2 退出验证代理指标（`spec-improve` 可读取作为历史基线）：**

| 代理指标 | 来源资产 | 最小可计算条件 |
| --- | --- | --- |
| `decision_note_reuse_rate` | `history-spec-index.json` 中被 `spec-plan` 引用的 entry 比例 | 至少 3 条 spec 完成了 review + compound |
| `pitfalls_hit_rate` | `findings.json` 中 `category == pitfall` 的 finding 占比 | 至少 2 次 review 执行 |
| `recurring_finding_rate` | `findings.json` 中 `is_recurring: true` 的 finding 比例（越低越好） | 至少 3 次 review 执行 |

## 10. 需要改动的资产

### 10.1 Skill / prompt

- 新增 `spec-improve`
- 可能需要对 `spec-review`、`spec-compound`、`spec-work` 做少量 metadata 补强，以保证 improve 有稳定输入

### 10.2 CLI / runtime

- improve 的读取与汇总能力
- readiness 评分与报告输出
- stale source / routing 健康度的 doctor 扩展

### 10.3 文档

- 核心概念中增加 improve 的定位
- 用户手册中增加 improve 的输入/输出说明

## 11. 阶段验收标准

1. `spec-improve` 能读取前两阶段的关键控制面资产，不依赖人工拼接输入。
2. `spec-improve` 能稳定输出 6 个 readiness 维度。
3. `spec-improve` 能稳定输出 Highest ROI gaps，并且每个 gap 有证据来源与建议动作。
4. `spec-improve` 能汇总过程代理指标与 outcome metrics。
5. `spec-improve` 默认不自动写回任何主资产。
6. 若未来要引入自动写回，文档必须额外声明权限边界、证据门槛和回滚策略，不能隐式升级。

## 12. 阶段风险与缓解

### 12.1 风险：improve 变成万能修复器

缓解：

- 文档明确 proposer 定位
- 默认只输出报告、提案、汇总索引

### 12.2 风险：评分变成黑箱 KPI

缓解：

- 每个分数都必须能解释证据和缺口
- Highest ROI gaps 的推导逻辑必须可追溯

### 12.3 风险：过程指标绑架系统

缓解：

- 强制纳入 outcome metrics
- 当过程指标与结果指标冲突时，优先相信结果指标

### 12.4 风险：把 stale source 修复误做成自动写回

缓解：

- stale source 只能先形成 update proposal
- 采纳与执行必须显式确认

## 13. 本阶段完成标志

完成阶段 3 的标志不是“多了一个评分器”，而是下面三件事同时成立：

1. 团队能看出当前系统最值钱的改造点，而不是靠经验拍脑袋。
2. 过程指标与结果指标能一起解释“为什么最近质量变好了或变差了”。
3. `spec-improve` 能推动下一轮改进，但不会偷偷演化成自动治理系统。

# review 机制

Spec-First 当前的 review 机制不是单一命令，而是三层配合：

- 用户侧的 `/spec:code-review` 命令，用于审查代码变更
- 任务流水线里的 review 阶段，用于控制 `work -> review -> compound` 的状态流转
- `spec-code-review` skill 里的多视角审查规范，用于把 review 做成可重复、可合并、可门禁化的流程

和 [Review 机制详细设计](/Users/kuang/xiaobu/spec-first-doc/业界学习/03-机制分析/1.Review机制详细设计.md) 相比，当前项目保留了“多 persona、置信度门禁、阶段回退”的核心思想，但实现方式更轻：以 Claude Code 命令模板、shell 门禁脚本和任务产物文件为主，不引入独立的工作流引擎。

## 1. 当前 review 解决什么问题

review 的目标不是简单给建议，而是把“是否可以继续往下走”这件事结构化。

- 识别代码变更里的正确性、回归、覆盖率、可维护性问题
- 判断当前 work 产物是否可以进入 compound
- 把阻断问题和非阻断问题分流
- 保留 review 结论，作为后续返工和沉淀的依据

## 2. 用户侧入口

当前用户主要通过 `/spec:code-review` 触发 review。

- 入口命令由 `spec-first init --claude` 生成到项目的 `.claude/commands/spec/code-review.md`
- 当前模板要求 review 关注 correctness、regressions、coverage gaps、maintainability
- 如果没有发现问题，要明确写出“没有问题”以及残余风险，而不是只给空结论

这意味着 review 是一个 Claude Code 命令层能力，不是 `spec-first` Node CLI 里的独立子命令。

## 3. review 的执行层

### 3.1 先确定 review 范围

review 先确认自己到底在看什么。

- 可以 review 当前分支、指定分支、PR，或者一段变更摘要
- 以 merge-base 为基准计算 diff，避免只看到工作区的局部变化
- 处理 untracked 文件时要谨慎；未加入版本控制的文件通常不进入正式 review 范围
- 如果有 plan 文档，还会额外做需求追踪，检查计划里的要求是否真的落到实现里

### 3.2 再选择 reviewer 视角

`spec-code-review` 的核心不是单个 reviewer，而是多个 reviewer 并行。

- always-on：correctness、testing、maintainability、project-standards
- CE always-on：agent-native-reviewer、learnings-researcher
- conditional：security、performance、api-contract、data-migrations、reliability、adversarial
- stack-specific：dhh-rails、kieran-rails、kieran-python、kieran-typescript、julik-frontend-races

选择 reviewer 时看的是 diff 的实际内容和风险，不是机械地按文件后缀套模板。

### 3.3 并行审查

每个 reviewer 独立产出结构化 findings。

- findings 必须有文件、行号、严重级别、原因、证据和建议路由
- reviewer 还要补充 residual_risks 和 testing_gaps
- 这一步的目标是“多视角发现问题”，不是让不同 reviewer 重复写同一句话

### 3.4 合并、去重和门禁

合并层会把多个 reviewer 的输出收敛成一个结论。

- 先做 schema 校验，过滤掉不符合结构的输出
- 再做 confidence gate，低于 0.60 的发现默认不报
- 再按 `file + line bucket + title` 去重
- 再把结果路由到 `safe_auto`、`gated_auto`、`manual`、`advisory`
- `pre_existing: true` 的问题会单独列出，不算这次 diff 造成的新问题

## 4. review 产物长什么样

review 的输出通常包含这些部分：

- Findings
- Requirements Completeness
- Applied Fixes
- Residual Actionable Work
- Pre-existing
- Learnings & Past Solutions
- Agent-Native Gaps
- Schema Drift Check
- Deployment Notes
- Coverage
- Verdict

如果有 plan 文档，review 还会顺手做需求完成度核对。

- explicit plan：未完成的需求会直接变成更强的 review 问题
- inferred plan：只作为提示，不作为强契约

## 5. 任务流水线里的 review

Spec-First 的 task 流程里，review 还是一个明确的阶段闸门。

`task.yaml` 记录当前阶段和状态：

- `current_stage`
- `status`
- `role`
- `level`

阶段文件按顺序推进：

```text
01-brainstorm.md -> 02-plan.md -> 03-work.md -> 04-review.md -> 05-compound.md
```

`stage-gate.sh` 负责检查上游产物是否齐全。

| 目标阶段 | 必要条件 |
| --- | --- |
| plan | 必须存在 `01-brainstorm.md` |
| work | 必须存在 `02-plan.md` |
| review | 必须存在 `03-work.md` |
| compound | 必须存在 `04-review.md`，且 review 通过 |

这保证了 review 不是“看完就算”，而是会实际阻断下游阶段。

## 6. review 的判定规则

`review-judge.sh` 给 review 提供了一个可执行的通过标准。

- correctness < 3 或 completeness < 3，直接不通过
- 存在 blocking issues，直接不通过
- 四项总分低于 12，直接不通过
- 总分达到 16 及以上时，标记 `compound_recommended: true`

四个维度分别是：

- correctness
- completeness
- executability
- reusability

因此，当前项目的 review 不是“主观感觉不错就过”，而是有最低分线和阻断问题优先级的。

## 7. review 通过后的流转

review 通过后，才能继续 compound。

- `review` 不通过时，必须回到 `work` 修复
- `compound` 只能接收 `pass: true` 的 review 结果
- 如果 review 发现问题，优先修复阻断项，再处理残余建议

## 8. 当前实现边界

当前项目的 review 机制已经形成闭环，但它不是一个独立的流程引擎。

- 没有额外的数据库或队列来驱动 review 状态
- 主要依赖文件系统产物、task.yaml 和命令模板
- 主要依赖 Claude Code 执行 `/spec:code-review`
- 复杂的多 persona 逻辑是 review 规范的一部分，不是 npm CLI 的 Node 子命令

## 9. 与业界分析文档的关系

[Review 机制详细设计](/Users/kuang/xiaobu/spec-first-doc/业界学习/03-机制分析/1.Review机制详细设计.md) 里的两类 review 思路，当前项目都吸收了：

- 文档/计划审查的“多 persona + 合并 + 置信度门禁”思想，被收敛成 review skill 的结构化输出
- 代码变更审查的“scope -> reviewer -> merge -> route -> verdict”流程，被收敛成 `/spec:code-review`、`review-judge.sh` 和 `stage-gate.sh`

所以，Spec-First 当前的 review 机制可以概括为一句话：

**以 `/spec:code-review` 为入口，以多 persona 结构化审查为核心，以 `review-judge.sh` 和 `stage-gate.sh` 作为门禁，最终决定 work 是否可以进入 compound。**

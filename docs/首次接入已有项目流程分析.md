# 首次接入已有项目流程分析

本文档只讨论一个场景：

- 项目已经存在
- 这是第一次接入 spec-first
- `.spec-first/spec/` 里还没有可直接复用的项目规范

这个场景的关键点不是“先做第一个需求”，而是“先让项目接入 spec-first 的工作系统”。

---

## 0. 先判断要不要 bootstrap

第一件事是看 `.spec-first/spec/` 是否已经有真实内容，而不是空模板。

判断结果很直接：

- 为空或只有模板时，先 bootstrap
- 已经有真实规范时，直接进入常规迭代

也就是说，首次接入的真实顺序通常是：

1. 初始化开发者
2. 读取当前上下文
3. 检查 spec 状态
4. 必要时先 bootstrap
5. 再开始第一轮需求迭代

从代码分层看，这里其实有两层动作：

- `spec-first init`：先生成工作区结构和 `spec/` 模板骨架
- `create_bootstrap`：再创建一个“补齐规范”的任务，推动你把骨架写完整

---

## 1. 首次接入的完整流程

```text
已有项目首次接入 spec-first
├─ init_developer
├─ get_context
├─ 检查 .spec-first/spec/
│  ├─ 为空 / 模板
│  │  └─ create_bootstrap
│  └─ 已可用
│     └─ 直接进入需求迭代
└─ 创建首个可交付任务
   ├─ task create
   ├─ task init-context
   └─ task start
```

这条路径的重点是先把“规范记忆”补起来，再让 AI 进入后续任务。

可以把首次接入的执行链看成这样：

```text
已有项目首次接入 spec-first
        │
        ▼
spec-first init
        │
        ├─ init_developer
        ├─ createWorkflowStructure(...)
        ├─ createSpecTemplates(...)
        └─ 生成 .spec-first/spec/ 初始模板
        │
        ▼
get_context
        │
        ▼
检查 .spec-first/spec/ 是否仍只是空模板
        │
        ├─ 是 -> create_bootstrap
        │          ├─ 生成 00-bootstrap-guidelines
        │          ├─ 写 task.json
        │          ├─ 写 prd.md
        │          └─ 写 .current-task
        │
        └─ 否 -> 直接进入常规任务流
```

---

## 2. 为什么 bootstrap 是必须的

已有项目第一次接入时，AI 最缺的不是语法能力，而是项目上下文：

- 目录怎么分层
- 组件、服务、路由放哪里
- 错误怎么捕获和返回
- 日志怎么写
- 哪些模式是团队约定

bootstrap 的目标不是产出业务功能，而是把仓库里的真实模式提炼成 `.spec-first/spec/`。

### 2.1 相关代码

- `.spec-first/scripts/create_bootstrap.py`
- `packages/cli/src/commands/init.ts`

bootstrap 会写入三样东西：

- `task.json`
- `prd.md`
- `.current-task`

其中 `task.json` 的关键字段是：

- `status: in_progress`
- `dev_type: docs`
- `priority: P1`
- `relatedFiles` 指向规范目录

但 bootstrap 本身不是“写业务知识”的动作。它只是把“补规范”这件事变成一个可追踪任务。真正把项目知识写进 `.spec-first/spec/`，仍然依赖后续人工分析、文档补写，或者 `update-spec` 体系把新知识沉淀回去。

### 2.2 bootstrap PRD 写什么

bootstrap PRD 强调的是“仓库真实存在的模式”，不是理想化最佳实践。

内容通常包括：

- 后端目录如何分层
- 前端组件 / 页面 / hook 如何组织
- 错误处理和日志约定
- 质量要求
- 需要保留的 anti-pattern

如果是 monorepo，还要按 package 维度分别整理规范。

## 3. 首次接入的推荐顺序

### 3.1 初始化开发者

系统先需要知道“谁在工作”：

- 创建 `.developer`
- 创建 `.spec-first/workspace/<developer>/`
- 创建 `journal-1.md`
- 创建 `index.md`

### 3.2 获取当前上下文

`get_context` 会展示：

- 当前分支
- 工作区是否干净
- 当前是否有活跃任务
- journal 写到哪了

这样可以先看现场，再决定要不要进入 bootstrap。

### 3.3 检查 spec 是否可用

这是首次接入最重要的分支点：

- 如果只是空目录，就先补规范
- 如果已经有真实规范，就直接进入正常任务流

### 3.4 补齐规范

优先补和仓库形态强相关的内容：

- 后端项目优先补 `backend/`
- 前端项目优先补 `frontend/`
- fullstack 项目两边一起补

文档写法要遵守一个原则：**记录现实，不写愿景**。

---

## 4. 补规范时最先写什么

建议先写这些内容：

- 目录结构
- 模块边界
- 组件 / 服务 / 路由的组织方式
- 错误处理
- 日志规范
- 质量标准

如果仓库本来就有 `CLAUDE.md`、`AGENTS.md`、`.cursorrules` 或类似文档，也应该先把它们的约束提炼进 `.spec-first/spec/`。

---

## 5. 首次接入最容易犯的错

- 把 bootstrap 当成第一个业务需求
- 还没补规范，就直接开始写功能
- 只写理想模板，不写仓库真实模式
- 忽略 monorepo 的 package 维度
- 在 `update` 流程里试图改写 `spec/`
- 误以为 `spec-first update` 会补规范

正确做法是：

1. 先把规范从代码和既有文档里提炼出来
2. 再让后续任务读取这些规范
3. 让所有后续需求都在同一套上下文下推进

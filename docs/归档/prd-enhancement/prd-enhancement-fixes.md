# PRD 增强方案 - 问题修复清单

> **版本**: v2.1.0
> **日期**: 2026-03-05

---

## 🔍 审查发现的问题

### 问题 1: Phase 流程定义不完整 ⚠️ 高

**问题描述**:
- 总流程包含 Phase 0.3/0.4/0.6
- 但详细设计只有 0.0/0.1/0.2/0.5/0.6
- 阶段划分未给 0.3/0.4 工时与交付

**修复方案**:
```
Phase 0.3: PRD 初稿生成
- 工作量: 0.5 天
- 交付物: prd.md（初稿）
- 执行: 基于 raw-requirement.md + findings.md 生成

Phase 0.4: PRD 自检
- 工作量: 0.5 天
- 交付物: C-PRD 评分结果
- 执行: 运行 C-PRD 评分算法
```

---

### 问题 2: 门禁逻辑冲突 ⚠️ 高

**问题描述**:
- 质量评分 ≥85 直接"跳过 0.5"
- 但流程又规定 C-PRD <85 必须进 0.5
- 且 C-PRD 在 0.4 才计算

**修复方案**:
```
统一使用 C-PRD 评分（Phase 0.4）作为唯一决策依据：

if C-PRD >= 70%:
    跳过 Phase 0.5，直接进入 Phase 0.6
else:
    执行 Phase 0.5 补全对话

删除 Phase 0.2 的"质量评分 ≥85 跳过"逻辑
```

---

### 问题 3: 初始化脚本变量未展开 ⚠️ 高

**问题描述**:
- 使用了单引号 heredoc `<< 'FINDINGS'`
- 导致 `$(date)` 和 `$FEATURE_ID` 不会展开

**修复方案**:
```bash
# 错误写法
cat > file << 'EOF'
$(date)
EOF

# 正确写法
cat > file << EOF
$(date)
EOF
```

---

### 问题 4: 术语定义评分方向错误 ⚠️ 中

**问题描述**:
- 原文：术语定义 | 未定义术语 | 每个术语 +2
- 逻辑错误：应该奖励"已定义术语"，而非"未定义术语"

**修复方案**:
```
术语定义维度（10%）:
- 检测文档中的专业术语（大写缩写、领域词汇）
- 检查是否有术语表或定义说明
- 评分：已定义术语数 × 2，最多 10 分
```

### 问题 5: 实现细节可执行性歧义 ⚠️ 中

**问题描述**:
- 解析库写 `xlsx.js`，依赖写 `xlsx`
- JSON 代码块内含注释，不可直接使用

**修复方案**:
```json
{
  "dependencies": {
    "mammoth": "^1.6.0",
    "xlsx": "^0.18.5",
    "pdf-parse": "^1.1.1"
  }
}
```

说明文档中统一使用 `xlsx` 包名。

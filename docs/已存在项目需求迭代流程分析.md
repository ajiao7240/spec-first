# 已存在项目需求迭代流程分析

本文档只分析“存量项目需求迭代”这条流程，不包含新项目 bootstrap。

目标是回答四件事：

1. 流程怎么走
2. 每个节点产出什么
3. 上下文怎么分层管理
4. 过程记录写到哪里、写什么

---

## 0. 适用范围

适用场景：

- 修 bug
- 加功能
- 重构
- 性能优化
- 文档或规范迭代

不展开内容：

- 新项目 0-1 初始化
- 具体业务实现细节
- 平台差异化命令适配

---

## 1. 统一流程图

```text
已存在项目需求进入
├─ 需求澄清 / 复杂度判断
│  ├─ 复杂 / 不清晰 → brainstorm
│  └─ 简单 / 明确 → 直接建任务
├─ task create
├─ 写 PRD
├─ task init-context
├─ task start
├─ 实现 / 检查 / 收尾
├─ add_session
└─ archive / create PR
```

这条链路的核心不是“先写代码再说”，而是先把任务、上下文和记录都组织好。

---

## 2. 流程分段说明

### 2.1 需求进入与分流

这一段的目标不是立刻写代码，而是判断这次需求属于哪种复杂度：

- 简单且边界清晰，直接进入任务创建
- 复杂、跨模块、需求存在不确定性，先走 brainstorm

判断标准通常是：

- 是否需要研究
- 是否需要拆分阶段
- 是否需要跨 package / 跨层协调
- 是否已有足够明确的验收标准

### 2.2 创建任务

相关代码：

- `.spec-first/scripts/common/task_store.py`

`task create` 会创建任务目录并写入 `task.json`。最重要的字段是：

- `status = planning`
- `base_branch = 当前分支`
- `next_action` 按实现、检查、收尾、提 PR 的顺序排列
- `children` / `parent` 用于任务树
- `package` 用于 monorepo 场景

这一步完成后，任务已经可追踪，但还没有进入上下文注入。

### 2.3 编写 PRD

PRD 的作用不是“写一份漂亮说明”，而是把需求契约写清楚：

- 需求范围
- 验收标准
- 技术约束
- 边界和风险

也就是说，PRD 是任务的契约，而不是设计随笔。

### 2.4 初始化上下文

相关代码：

- `.spec-first/scripts/common/task_context.py`

`task init-context` 负责生成：

- `implement.jsonl`
- `check.jsonl`
- `debug.jsonl`

默认注入的内容不一样：

- `implement.jsonl` 会带上 `workflow.md`
- `backend` / `frontend` 会带上对应的 `spec/.../index.md`
- `check.jsonl` 会带上 `finish-work` 和 `check`
- `debug.jsonl` 主要聚焦 `check`

这说明实现阶段和检查阶段看的不是同一套上下文。

### 2.5 进入执行

`task start` 之后，任务会被写入当前执行锚点。后续上下文恢复会围绕 `.current-task` 和 workspace 展开。

这个阶段真正关心的是：

- 当前任务是谁
- 当前分支是哪条
- 当前工作目录是否干净
- 当前 journal 记录到哪一条

### 2.6 记录 session

相关代码：

- `.spec-first/scripts/add_session.py`

`add_session` 会把本次工作结果写进 journal，并同步 `index.md`。

记录内容至少包括：

- 标题
- 摘要
- commit
- 分支
- 测试结果
- 下一步

如果当前 journal 太长，会自动拆分成新的 journal 文件。

---

## 3. 上下文分层应该怎么理解

### 3.1 实现上下文

实现上下文只需要最相关的最小集合：

- `workflow.md`
- `spec`
- 当前任务说明

### 3.2 检查上下文

检查上下文偏向质量与收尾：

- `finish-work`
- `check`

### 3.3 调试上下文

调试上下文更偏向代码质量检查和问题定位：

- `check`

这种分层的好处是避免“所有阶段都读同样的东西”。

---

## 4. 任务状态与分支约束

### 4.1 分支来源

记录 session 时，分支会按顺序解析：

1. CLI `--branch`
2. 当前任务 `task.json.branch`
3. Git 当前分支

### 4.2 单包 / 多包

monorepo 下如果没有明确 package，系统会尝试解析默认 package；如果解析不到，`init-context` 会要求显式指定 `--package` 或配置默认 package。

### 4.3 子任务关系

`task.json` 支持 `parent` / `children` 关系，`cmd_add_subtask` 和 `cmd_remove_subtask` 会同步维护双向链接。

这意味着任务不是平面的，而是可以构建树状结构的。

---

## 5. 结论

存量项目迭代的主目标是：**把需求变成可执行、可记录、可恢复的任务**。

因此最重要的不是“写了多少代码”，而是：

- 任务元数据是否完整
- 上下文是否分层正确
- journal 是否能回放
- 后续是否能顺利归档

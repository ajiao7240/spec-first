# GitNexus 安装、生成、使用流程 10 轮审查报告

## 第 1 轮

- 问题描述：GitNexus 版本虽然集中在 `mcp-tools.json`，但当前仍通过 `installation.unix.args[1]` 反推 package spec，缺少显式 `version_policy` / `pin_reason` / `blocked_versions` 机器字段，后续容易被“全部最新版本”规则误改回不稳定 rc。
- 等级：P2
- 建议优化方案：在 GitNexus 条目增加显式 `version_policy.package_spec`、`pin_reason`、`blocked_versions`，安装、host config、provider projection 全部读取该字段，不再从 argv 下标推断版本。

## 第 2 轮

- 问题描述：`install-mcp` 的 warmup 阶段会执行 `npx` 下载/准备 GitNexus，但 setup 输出没有明确提示“首次运行可能下载中”；用户容易把静默等待误判为卡死。
- 等级：P3
- 建议优化方案：在每个 warmup 命令前输出统一提示：`spec-mcp-setup: preparing <tool>; dependencies may download on first use...`，不实现复杂进度条。

## 第 3 轮

- 问题描述：build/cache/runtime/binary artifacts 的排除规则目前只用于 GitNexus query probe token 选择，不会实际约束 `gitnexus analyze --force` 的索引范围；如果这些文件被 Git 跟踪或 GitNexus 未按预期忽略，仍可能进入索引。
- 等级：P1
- 建议优化方案：新增 GitNexus index policy 投影，优先生成或校验 `.gitnexusignore`，覆盖 `build/**`、`cache/**`、`runtime/**`、`.gradle/**`、`*.apk`、`*.so` 等；若未能确认排除策略实际生效，在 provider status 中标记 `index_policy_applied=false`。

## 第 4 轮

- 问题描述：`gitnexus analyze --force` 默认可能更新 `AGENTS.md` / `CLAUDE.md` 的 GitNexus managed block，导致 graph bootstrap 除 `.spec-first/` 外还修改根 instruction 文件，制造非预期工作区脏变更。
- 等级：P1
- 建议优化方案：bootstrap 生成命令改为追加 `--skip-agents-md`，必要时再加 `--no-stats`；同步更新命令 shape allowlist 和测试，保证 graph bootstrap 只写 graph readiness artifacts。

## 第 5 轮

- 问题描述：GitNexus analyze/query 当前使用 repo basename 作为 repo identity，`query --repo <repo-name>` 在全局 registry 中遇到同名仓库时可能查错索引或产生歧义。
- 等级：P1
- 建议优化方案：为每个目标 repo 生成稳定 alias，例如 `<basename>-<root-hash>`；analyze 使用 `--name <alias>`，query 使用同一 alias，并把 alias 写入 `graph-providers.json`。

## 第 6 轮

- 问题描述：bootstrap 命令安全校验仍允许 GitNexus bootstrap 为 `analyze` 且不带 `--force`；旧配置或手写配置可能绕过强制重建，继续复用缺失 BM25/FTS 的旧索引。
- 等级：P2
- 建议优化方案：当前阶段只允许 `analyze --force` 作为 GitNexus bootstrap 命令；如需兼容旧配置，先标记 `unsupported-provider-command` 或 `stale-command-shape`，要求重新运行 `spec-mcp-setup`。

## 第 7 轮

- 问题描述：当源码 basename 无法生成 expected-hit probe 时，fallback token 仍可能通过任意 BM25 命中让 GitNexus 被判为 `query_ready=true`，证明强度不足。
- 等级：P2
- 建议优化方案：GitNexus `query_ready=true` 必须同时满足 `query_probe_policy.expected_hit=true` 和 `processes/process_symbols` 非空；`expected_hit=false` 时最多标记 `query-unverified`。

## 第 8 轮

- 问题描述：PowerShell 版 `bootstrap-providers.ps1` 生成的 `graph-facts.json` 缺少 Unix 版已有的 `capabilities.query_global_graph`、`capabilities.impact_context` 和 `staleness_hints`，跨平台 artifact contract 不一致。
- 等级：P1
- 建议优化方案：补齐 PowerShell `graph-facts` 字段，与 Unix 版保持同构；在 `mcp-setup-powershell-contracts.test.js` 中增加字段级断言。

## 第 9 轮

- 问题描述：provider 命令没有脚本级 timeout；GitNexus analyze、npm 首次下载或 native 卡死只能依赖宿主外层 timeout，失败后缺少结构化 `timeout` reason code。
- 等级：P2
- 建议优化方案：在 provider command metadata 增加 `timeout_sec`，bootstrap 执行器按命令级 timeout 截断，并输出 `failure_class=provider-timeout`、`reason_code=gitnexus-analyze-timeout` 或通用 timeout reason。

## 第 10 轮

- 问题描述：后续 workflow “优先尝试 live MCP” 目前主要是 prose 约束，没有记录 live MCP 实际加载版本、host config 版本、响应结果和限制；当前会话可能仍加载旧 rc server，但计划/执行输出无法机器化表达这个风险。
- 等级：P2
- 建议优化方案：把 `runtime_mcp_evidence` 收敛为轻量固定字段：`tool_loaded`、`response_ok`、`host_config_package`、`session_version_known`、`evidence_scope`、`limitations`；成功只作为 session-local evidence，不回写 compiled readiness。

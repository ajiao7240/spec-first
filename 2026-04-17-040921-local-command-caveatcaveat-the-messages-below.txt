  🔴 A1. resolveEdges 阶段 5 的"全局符号消歧"会制造错误 calls 边

  位置： src/crg/graph.js:250-280

  const rows = getAllSymbolRows.all(name);        // 所有同名节点
  if (rows.length === 1) symbolCache[name] = rows[0].id;   // 全局唯一就用它

  问题： 只要全局碰巧只有一个 run / init / parse，即便 source 文件根本没 import
  过 target 的 module，也会被连上 calls 边。对一个用 CommonJS + 大量 run() /

  - 阶段 5 必须加守卫：source_module 到 target_module 之间存在 imports_from
  边（或同文件），才允许消歧为 calls 边。
  - 否则降级为 unresolved。代价：unresolved
  数字会上升，但这是"诚实的上升"，比"静默假阳"好得多。
  - 可用一次性预计算 moduleReachability: Map<srcModId, Set<tgtModId>>，O(|E|)
  一次，查询 O(1)。

  一、正确性与语义类缺陷（高优先，会污染下游 AI agent 判断）

  C1. 社区检测不是社区检测——本质是"按顶层目录分组"

  位置：communities.js:75-102

  Pass 1 按 file_path 第一个非容器目录分组，直接决定 node.community_id。Pass 2
  只评估"健康度标签"，Pass 3
  只对超大社区做连通分量拆分。整个流程没有任何图拓扑聚类（Louvain / Leiden /
  Label Propagation 都没有），本质上是 groupBy(filePath.split('/')[k])。

  后果传染 surprising_connections（analyze.js:112）——cross_community=40 是 4
  因子里最大权重，但社区边界本身是"目录边界"，因此 surprising_connections
  实际上在报告"跨目录调用"，这对任何合理模块化的代码库都是海量噪音。项目介绍文档
  宣称的"社区检测 + 4 因子惊喜连接"在算法层面是名不副实。

  建议：引入 graphology-communities-louvain（纯 JS，零原生依赖），在 module
  级别跑 Louvain，目录框架降级为 fallback；或至少把
  surprising_connections.cross_community 权重下调到 15
  并要求与另一维度共同触发。

  C2. resolveEdges 的 AMBIGUOUS 缓存没有起到缓存作用

  位置：graph.js:255-280

  const cached = symbolCache[name];
  if (cached !== AMBIGUOUS) return cached;
  // 全局重名：尝试同文件精确匹配
  const rows = getAllSymbolRows.all(name);  // ← 每次 AMBIGUOUS 都重复扫

  当某个 name 在全局有多个同名节点（如 run、build、resolve），每次遇到同名 raw
  edge 都重新跑 SELECT id, file_path FROM nodes WHERE name = ?。对 JS 仓库这种
  name 冲突频繁的场景，这是主 bottleneck 之一。

  修复：缓存应存 rows[] 本身：

  symbolCache[name] = { status: 'ambiguous', rows };
  // 后续 srcFile 过滤从缓存读，零 SQL

  C3. resolveEdges 的相对路径扩展名探测只覆盖 JS/TS

  位置：graph.js:313-316

  for (const suffix of ['', '.js', '.mjs', '.ts', '/index.js']) {

  缺 .tsx、.cjs、.jsx、.py、.rb、.go。Python from .utils import X、Ruby
  require_relative './helper'、Go（不相关）、TSX 组件相对 import
  全部直接进入阶段 2 全局符号兜底——多语言仓库解析率大幅虚降，unresolved 被夸大。

  C4. basename fallback 只对 C/ObjC 头文件生效

  位置：graph.js:322-330

  const isHeaderExt = /\.(h|hpp|hh|hxx|m|mm|c|cc|cpp|cxx)$/i.test(rawPath);

  Java import foo.Bar（parser 会不会产出纯 basename 另说）、Python 模块名、Ruby
  常量引用若以 basename 形式出现都跳过 fallback。建议扩展为"任何无路径分隔符 +
  属于 INDEXABLE_EXTS 的 target_name/target_path_raw 都走 basename proximity"。

  C5. chunking.js 切分对 line_end=0 的节点产生废块

  位置：chunking.js:18-27

  const span = Math.max((node.line_end || 0) - (node.line_start || 0) + 1, 1);
  const chunkCount = Math.max(1, Math.ceil(span / effectiveMaxLines));
  // ...
  const end = Math.min((node.line_end || 0), start + effectiveMaxLines - 1);

  当 parser 对某类节点没填 line_end（或 line_end < line_start，TS
  语法异常时可能发生）时，end = Math.min(0, ...)=0，同时 start > 0，产出
  {line_start: X, line_end: 0} 的颠倒块。chunks 表里这些 row
  会成为"非法数据点"污染 retrieval。

  修复：line_end <= line_start → 单一 chunk 且 line_end = line_start，并在
  chunking 入口校验。

  C6. semantic-rerank 与项目宣称严重不符

  位置：retrieval/semantic-rerank.js:13-18

  const haystack = `${item.name} ${item.retrieval_text}`.toLowerCase();
  const overlap = terms.reduce((sum, term) => sum + (haystack.includes(term) ? 1
   : 0), 0);

  这是朴素子串计数，不是语义。没有 embedding、没有
  IDF、没有词干化、没有近义词映射。项目介绍说"semantic-rerank
  词项重排"——诚实的命名应该是 lexical-overlap-rerank。此外 retrieval_text 仅是
  "${file_path} ${kind} ${name}"（parser.js:139），完全不含代码文本或 
  docstring，所以哪怕上了真 embedding，信号密度也是零。这是 CRG 作为"AI
  代码理解底座"的最大系统性弱点。

  C7. changes.js 不做行级 diff 过滤——一处改动影响整个文件所有节点

  位置：changes.js:27-39

  getChangedFilesFromGit 只用 git diff --name-only，assessNodePriorities
  把文件里所有 function/method/class
  都打分并返回。结果：修一个注释、改一个字符串常量，review-context 把同文件 50
  个函数都列为"变更"，并对它们做 2-hop BFS（每个触发一次批量 risk
  scoring），产生大量假阳性 blast radius。

  修复：改用 git diff -U0 --numstat -- <file> 或 git diff --unified=0 提取
  hunks，过滤 nodes WHERE [line_start, line_end] 与任一 hunk 区间相交。这是 AI
  review 体验的决定性优化。

  ---
  二、算法复杂度类缺陷（中高优先）

  P1. BFS 普遍用 Array.shift()，均摊 O(n²)

  位置：flows.js:59、communities.js:47、impact.js:30、review-context.js:167

  JS shift() 是 O(n)。对 flows 限 20 节点可忽略；但 communities.bfsComponents
  扫社区内所有模块节点，impact.reverseBfs 深度 5
  可能遍历数千调用者，review-context.reverseBfs 对每个 affected node 都跑一次。

  修复：标准替换为头索引队列：

  const queue = [startId];
  let head = 0;
  while (head < queue.length) {
    const cur = queue[head++];
    // ...
    queue.push(...);
  }

  零字符改动，实测对大仓库能省一半以上的 BFS 时间。

  P2. writeCommunities 子社区内外边统计是 O(E × Components)

  位置：communities.js:253-270

  for (const { src, tgt } of moduleEdges) { if (component.has(src) &&
  component.has(tgt)) subIntraEdges++; }
  for (const { src, tgt } of moduleEdges) { /* subInterEdges */ }

  对每个连通分量都完整遍历 moduleEdges 两次。仓库 10K 边 + 20 个子社区就是 40
  万次迭代。

  修复：单次遍历 moduleEdges，预先构建 moduleToSubComponent: Map<moduleId, 
  componentIdx>，对每条边 O(1) 分类：

  for (const {src, tgt} of moduleEdges) {
    const si = moduleToSubComp.get(src), ti = moduleToSubComp.get(tgt);
    if (si === undefined || ti === undefined) continue;
    if (si === ti) subStats[si].intra++;
    else { subStats[si].inter++; subStats[ti].inter++; }
  }

  P3. 社区 community_id 传播使用关联子查询——N+1 反模式

  位置：communities.js:346-354

  UPDATE nodes SET community_id = (
    SELECT m.community_id FROM nodes m
    WHERE m.file_path = nodes.file_path AND m.kind = 'module' LIMIT 1
  ) WHERE kind != 'module'

  SQLite 会对每行非 module 节点执行子查询一次。用 idx_nodes_file_path_kind
  索引能优化但仍是每行一次查询。

  修复：构造 filePath → community_id 临时表，用 JOIN 替换子查询，或在 JS
  侧批量收集后 UPDATE ... WHERE id IN (...) 分批运行。

  P4. detectFlows 入口检测 NOT IN (SELECT target_id ...)

  位置：flows.js:163-169

  SQLite 对 NOT IN (subquery) 执行效率不佳，对大仓库是 O(N × E)。已经在
  loadAdjacency 里加载了全部 calls 边到 Map，应该改用 JS 侧 Set 过滤：

  const hasIncoming = new Set();
  for (const [, targets] of adjacency) targets.forEach(t => hasIncoming.add(t));
  const entryNodes = db.prepare("SELECT id, name FROM nodes WHERE kind != 
  'module'").all()
    .filter(n => !hasIncoming.has(n.id));

  P5. rebuildFTS 每次 postprocess 全量重建

  位置：search.js:85-126

  DROP + CREATE + 全表 INSERT。对 10 万节点仓库是秒级操作，且每次 build 都跑。

  修复：增量模式——只对 changedFiles 对应 node_id DELETE FROM fts_nodes WHERE 
  node_id IN (...) + 重新 INSERT。FTS5 external content 有坑，但独立 FTS
  虚表支持直接 DELETE。

  P6. searchNodesByLike 是全表扫，每个 term 一次

  位置：retrieval/seed.js:27-34

  WHERE lower(name) LIKE lower(?) OR lower(retrieval_text) LIKE lower(?)

  零索引支持（LIKE '%x%' 无法走索引）。当 FTS match 返回空（极常见，因为 FTS
  对复合标识符切词不友好），每个 term 都会全表扫 nodes。

  修复：
  1. 优先让 FTS5 配 tokenize='trigram'，这样 LIKE '%foo%' 的语义可由 FTS 覆盖；
  2. 保留 LIKE 时加 name 上的函数索引或直接用 name GLOB '*x*'（SQLite GLOB
  不支持索引但至少不需 lower）。

  P7. assessFileRisk 每个文件都查 COUNT(*) FROM edges WHERE target_id = ?

  位置：changes.js:56

  对 N 个变更文件做 N 次查询。改成一次 GROUP BY target_id：

  SELECT target_id, COUNT(*) c FROM edges WHERE target_id IN (...) GROUP BY
  target_id

  P8. review-context 加载全图 calls 边到 Map

  位置：review-context.js:152-159

  无 .filter 边界——对 100K 边仓库会占用约 10MB+ 内存 +
  同步加载延迟。可改为只加载 "affected 2-hop 反向闭包内" 的边：先查直接
  callers，再查 callers 的 callers，分两批 SQL。

  ---
  三、SQL/Schema/数据结构层面

  S1. edges(target_id, kind) 缺复合索引

  位置：migrations.js:147-150

  高频查询 WHERE target_id = ? AND kind = 'calls'（impact/callers/changes
  全用到）。现有 idx_edges_target + idx_edges_kind 独立索引，SQLite
  只能选一个；应加 CREATE INDEX idx_edges_target_kind ON edges(target_id, 
  kind)，同理 (source_id, kind)。

  S2. unresolved_edges 无唯一约束，可能重复写入

  位置：migrations.js:104-112、graph.js:416-438

  replace 逻辑是 truncate + insert，本身幂等，但如果同一 raw edge 在 rawEdges
  数组里重复出现，当前会插入两次。加约束 UNIQUE(source_id, edge_kind, 
  target_name, target_path_raw) + INSERT OR IGNORE。

  S3. expand.js 无分块，接近 999 参数上限

  位置：retrieval/expand.js:14-24

  WHERE (e.source_id IN (${placeholders}) OR e.target_id IN (${placeholders}))
    AND n.id NOT IN (${placeholders})

  三倍占位符。当 seed_limit=12 是 36 个，离 999 尚远，但未来调大 seed_limit
  时会爆。建议加 CHUNK_SIZE=300，或把 NOT IN 移到 JS 侧。

  S4. FTS5 关键字转义只去除双引号

  位置：search.js:31

  const safeKeyword = '"' + keyword.replace(/"/g, '') + '"';

  未处理 * 尾通配（实际算特性）、反斜杠、control chars。在 spec-first
  这种用户可传查询的场景有 SQL injection-like 的语义泄漏风险（FTS operator
  injection）。建议同时 strip FTS5 操作符字符 [*":()]。

  S5. nodes.id 依赖 filePath#kind#name#lineStart——函数改名/挪位置都破坏稳定性

  位置：parser.js:108

  单次重命名会使旧 id 变 stale，所有指向它的 edges 在 resolveEdges 里都找不到旧
  target（因为旧 node 已被 deleteStaleNodes 清掉），但同时新 name
  的同名全局查找可能命中别处——语义漂移。

  这是设计取舍，不是容易修的 bug，但应该在文档里明确声明"id 非稳定，仅供一次
  build 周期内引用"；对外给 AI agent 的 FactItem 应该额外携带 canonical 标识（如
   symbol_path: "src.crg.graph::resolveEdges"）以便跨 build 复用。

  四、设计层面（需产品侧讨论）

  D1. retrieval_text 信号量接近零

  parser.js:139 把 retrieval_text 设为 "${file_path} ${kind} 
  ${name}"，FTS/LIKE/semantic-rerank 全部在这个字符串上跑。真正可 retrieve
  的信号（代码体、docstring、首行注释、参数名）完全缺失。

  建议：buildNode 时从 tsNode.text 截取前 N 行，去除纯空白/分号行，作为
  retrieval_text 的主体。chunks 表已经有 line 范围，完全可以直接存 chunk
  源码前若干行。这是把 CRG 从"骨架"升级为"血肉"的决定性一步。

  D2. "混合检索"名下其实只有词法层

  seed = FTS + LIKE + 文件名匹配；expand = 1-hop 邻居；rerank =
  权重加分；semantic-rerank = 子串计数。全链路无任何 embedding、无任何 learned
  ranker。

  真要做 semantic，建议：
  - 集成 @xenova/transformers（纯 JS，本地 small MiniLM，~22MB）对 chunks 做离线
   embedding，存 chunks.embedding BLOB；
  - retrieval 时 query 也 embed，cosine 相似度打分 + 现有权重融合。

  这可以做成 Opt-in（profile=review --semantic=dense），不破坏默认快路径。

  D3. 社区/flows/risk 的权重都是魔法数字

  flows.criticality 0.30/0.20/0.25/0.15/0.10，changes.assessNodeRisk
  同样，analyze.surprising 10/30/40/20。没有 weights.json、没有校准数据、没有
  A/B。benchmarks/ 目录存在但未驱动这些权重。

  建议：抽一个 weights/ 配置包，tests/unit 里加权重漂移保护测试，benchmark
  里加"已知 high-risk PR → top-k recall"指标。

  D4. Chunking 完全不 AST-aware

  chunking.js:22-24 按 line_start + index * maxLines 硬切。对于 500 行的 god
  class，切成 7 块，每块前后缺失上下文。tree-sitter 已有完整 AST，完全可以按
  statement/block 边界对齐切分。

  D5. calls 边不含 call-site 行号

  parser 阶段提取调用关系时丢失了 "source 节点内部哪一行调用了 target"
  的信息。AI agent 做精准 patch 时需要这个。建议 edges 表加 call_line 
  INTEGER，或至少加 site_offset INTEGER 供后续 chunk 定位。

  D6. parser.js 1841 行单文件，16 语言混在一起

  无明显算法问题但维护性严重受限。建议拆为 parser/index.js +
  parser/languages/{javascript,objc,...}.js，每个子文件只暴露
  extractNodesAndEdges(tree, filePath)。

# 2026-05-01 App Audit 升级方案 30 轮小步审查记录

## 背景

审查对象：`docs/02-架构设计/spec_app_consistency_audit_升级技术方案.md`

判断基线：`docs/10-prompt/结构化项目角色契约.md`

核心原则：Light contract；Explicit boundaries；Scripts prepare, LLM decides；不把 app-audit 做成隐藏规则引擎或强状态机。

本轮修订目标：围绕 app-audit 升级方案做 30 个小步诊断与修订，使方案更贴近 `spec-first` 全局 workflow 链路，并降低文档与真实实现之间的漂移风险。

---

## Round 01：权威流水线缺少实现对齐说明

- 问题描述：方案已经定义了权威总流水线，但没有明确当前代码已经具备哪些能力，容易让读者误以为所有能力已经实现。
- 具体修复：新增“当前实现对齐与迁移边界”，列出 `SKILL.md`、scripts、schemas、prompts、template、dual-host governance、tests 和方案文档的 source-of-truth。
- 修复结论：方案从“理想架构描述”变为“升级目标 + 当前事实 + 迁移策略”。
- 修复效果：降低文档领先实现造成的误用风险，符合角色契约中“文档描述不能领先代码实现太多”的治理要求。

## Round 02：source-of-truth 与 generated runtime 边界不够具体

- 问题描述：方案强调不手改 runtime，但缺少与本能力相关的具体 source 文件清单。
- 具体修复：在实现对齐章节列出 `skills/`、`templates/`、`src/cli/contracts/`、`tests/`、`docs/` 等具体维护面。
- 修复结论：后续实现者能明确修改 source，而不是去改 `.claude/`、`.codex/`、`.agents/skills/`。
- 修复效果：减少 runtime drift 与多真相源风险。

## Round 03：当前 app-audit 真实实现能力被高估

- 问题描述：升级方案包含 Audit Planner、Validation Pass、headless handoff 等设计，但当前脚本主要是静态抽取与宽松校验。
- 具体修复：补充当前实现事实：已有 preflight、contract extraction、context merge、rule-pack selection、evidence gate、report assembly；但 Agent IO、strict issue schema、Validation Pass 尚未完整落地。
- 修复结论：方案明确哪些是已有基础、哪些是升级目标。
- 修复效果：避免任务拆分时把未来能力当成现有能力。

## Round 04：schema 演进策略缺失

- 问题描述：方案要求 `claim_type`、`affected_surface`、`validation_status`、`code_review_handoff`，但当前 `audit-report.schema.json` 是宽松 v1。
- 具体修复：新增“Schema 演进原则”，采用 additive hardening，新增独立 issue schema，优先 strict mode，不急于 v2。
- 修复结论：schema 迁移变成可落地步骤，而不是一次性破坏 consumer。
- 修复效果：保持 light contract，同时为后续严格校验留出路径。

## Round 05：validate-artifacts 职责边界不清

- 问题描述：方案只说 schema 校验，但没有约束 validator 不能做语义判断。
- 具体修复：新增 `validate-artifacts.js` 职责边界：只校验字段、枚举、hash、path、evidence 结构和 reason_code，不判断语义支持关系。
- 修复结论：确定性校验与 LLM Evidence Auditor 的职责重新分清。
- 修复效果：符合 Scripts prepare, LLM decides。

## Round 06：confirmed report 与 script candidate 默认策略冲突

- 问题描述：当前 `validateArtifact` 默认要求 script-produced artifact 为 candidate，但 audit report 可能是 confirmed。
- 具体修复：方案中明确 audit-report 若 `contract_status: confirmed`，必须走明确的 `--allow-confirmed` 或 report writer 专用调用路径。
- 修复结论：confirmed 产物不再被普通脚本路径静默生成。
- 修复效果：避免 advisory facts 被误当 confirmed truth。

## Round 07：Figma materialization 仍可能落旧全局路径

- 问题描述：当前 skill 里仍有 `.spec-first/app-audit/input/figma-context.json` 的旧路径表达，升级方案需要 run-scoped。
- 具体修复：新增 “Input Materialization 与路径边界”，规定 default/headless 写入 `runs/<run-id>/input/`，report-only 禁止写 input。
- 修复结论：Figma 输入 materialization 进入 run-scoped 证据链。
- 修复效果：避免不同 run 之间读取旧 Figma context。

## Round 08：Figma reference 与 evidence 的边界仍需强化

- 问题描述：Figma URL/node id 容易被误当设计证据。
- 具体修复：明确 `figma-ref:<id-or-url>` 只是外部引用，不等于 evidence；extractor 只能消费本地 materialized JSON。
- 修复结论：设计证据必须先 materialize。
- 修复效果：减少模型基于引用臆测设计状态的风险。

## Round 09：report-only no-write 需要覆盖 input materialization

- 问题描述：上一版已修正 report-only 不写 artifact，但 materialized input 也是写入行为。
- 具体修复：明确 report-only 禁止写 `input/`、preview 和 run artifact，只能读取已存在本地文件。
- 修复结论：report-only 成为真正可并行的只读审查模式。
- 修复效果：与 spec-code-review 的 report-only 语义对齐。

## Round 10：artifact 索引缺失

- 问题描述：run-scoped 目录里 artifact 多，但没有统一索引，downstream consumer 仍可能靠路径猜测。
- 具体修复：新增 `artifact-manifest.json`，记录 path、schema_version、artifact_id、producer、consumer、sha256、freshness、sensitivity、deprecated_aliases、reason_code。
- 修复结论：manifest 是 run 内部索引，不承载审查结论。
- 修复效果：提升 downstream 读取可靠性，避免路径硬编码扩散。

## Round 11：metadata 缺少 contract versions

- 问题描述：metadata 只有 run 基本信息，无法判断 schema/contract 版本。
- 具体修复：在 `metadata.json` 增加 `contract_versions` 和 `depth`。
- 修复结论：run 可被后续 workflow 按 contract 版本解释。
- 修复效果：为 schema 演进和兼容读取提供依据。

## Round 12：失败 run 的 artifact 语义不完整

- 问题描述：方案只描述成功产物，缺少 failed/degraded 时 metadata 和 latest 指针规则。
- 具体修复：新增失败语义：run 创建后失败仍写 metadata/manifest/status_reason_codes；failed run 不更新 latest-summary。
- 修复结论：失败也成为可诊断 evidence，而不是自然语言尾注。
- 修复效果：增强 workflow 证据链。

## Round 13：latest-summary 仍可能成为第二真相源

- 问题描述：latest-summary 方便消费，但容易被 downstream 当事实源。
- 具体修复：继续强化 latest-summary 只作为指针，并要求消费者校验 metadata branch/head/source_hash。
- 修复结论：run-scoped artifact 仍是唯一事实来源。
- 修复效果：降低 stale artifact 误读风险。

## Round 14：reason_code 没有标准集合

- 问题描述：方案多处提 degraded/failed，但没有稳定 reason_code，脚本和报告难以聚合。
- 具体修复：新增 reason_code 最小集合，包括 scope、input、graph、contract、expert、gate、validator、report writer 等类别。
- 修复结论：失败/降级从自然语言变成 machine-readable。
- 修复效果：便于 Coverage、headless envelope 和测试断言复用。

## Round 15：Graph/MCP 边界不明确

- 问题描述：app-audit 可能消费 graph/provider evidence，但 provider 不应成为 scope authority。
- 具体修复：新增 Graph/MCP 边界：provider 只提供 candidate pointers；stale/degraded 回退 source scan；最终 issue 必须引用项目特定 source evidence。
- 修复结论：graph 增强输入质量，但不替代审查判断。
- 修复效果：对齐 Codebase -> Graph -> Spec 链路，同时避免 provider 细节泄漏。

## Round 16：live MCP 证据可能被误写为 canonical artifact

- 问题描述：成功的 live MCP 查询是 session-local 事实，不等于 compiled readiness 或 run artifact。
- 具体修复：方案中明确 live MCP 只能作为 session-local evidence pointer，不能替代 run-scoped canonical artifact。
- 修复结论：session evidence 与 durable artifact 分层。
- 修复效果：减少跨会话不可追踪判断。

## Round 17：CLI e2e 仍以旧扁平目录为目标

- 问题描述：当前 `spec-app-consistency-audit-cli-e2e.test.js` 使用 `.spec-first/app-audit/` 扁平输出。
- 具体修复：在 v0.1 与实现优化方向中加入“CLI e2e 覆盖 run-scoped path，不再只测扁平目录”。
- 修复结论：测试成为迁移抓手。
- 修复效果：防止实现继续向旧路径写入。

## Round 18：legacy path 兼容策略不清

- 问题描述：直接删除旧扁平路径可能破坏现有测试；继续写旧路径又会制造双真相源。
- 具体修复：规定旧扁平路径只允许作为 migration compatibility 读取来源，不能作为新写入目标；legacy alias 必须在 manifest 标记 deprecated。
- 修复结论：兼容不等于第二真相源。
- 修复效果：迁移更稳，同时保持 explicit boundaries。

## Round 19：SKILL.md 与升级方案存在漂移风险

- 问题描述：source skill 仍是旧静态优先流程，没有表达新 mode、run-scoped、audit-plan、validation、handoff。
- 具体修复：在 P0 优化方向中加入先更新 `skills/spec-app-consistency-audit/SKILL.md`。
- 修复结论：实现入口必须先对齐 source skill。
- 修复效果：避免用户通过 skill 入口执行到旧流程。

## Round 20：Claude command argument hint 可能继续暗示旧参数

- 问题描述：template 的 argument-hint 仍是 `[prd path] [source path] [figma context or options]`，与 token contract 不一致。
- 具体修复：在 P0 优化方向中加入更新 template argument-hint。
- 修复结论：入口文案要反映 canonical token contract。
- 修复效果：减少用户输入歧义。

## Round 21：README 可能过早承诺未落地能力

- 问题描述：README 入口可见，若写入完整升级能力会造成用户误解。
- 具体修复：在 P0 优化方向中要求 README 只写稳定能力，不提前承诺 Audit Planner / Validation Pass 全自动链路。
- 修复结论：公开文案跟随实现成熟度。
- 修复效果：降低 user-visible promise drift。

## Round 22：strict issue schema 没有独立测试路径

- 问题描述：当前 audit report schema 宽松，新增字段若只写文档无法保证落地。
- 具体修复：v0.1 增加 strict issue schema + compatibility validation mode，并要求同步 validate test。
- 修复结论：issue 协议升级有测试闭环。
- 修复效果：增强 schema/downstream consumer 可维护性。

## Round 23：Agent IO 与 report-only 的关系需要可执行

- 问题描述：report-only 不写 full artifact，所以 compact return 必须足够完整。
- 具体修复：前序方案已把 compact finding 设为 gate-ready；本轮记录其必要性并纳入 30 轮审查结论。
- 修复结论：compact return 不再只是摘要。
- 修复效果：report-only/headless/default 都能用同一 issue 合成协议。

## Round 24：Validation Pass 容易变成第二个规则门

- 问题描述：validator 如果失败就统一降级，会让验证基础设施可用性影响真实问题。
- 具体修复：前序方案已区分 `validator_rejected` 与 `validator_unavailable`；本轮确认该原则保留。
- 修复结论：validator unavailable 不是反证。
- 修复效果：避免强证据 confirmed finding 被基础设施波动误杀。

## Round 25：code-review handoff 需要机器字段而非文本

- 问题描述：如果只在 headless envelope 写 mapping，code-review 难以稳定消费。
- 具体修复：前序方案已新增 `code_review_handoff`；本轮确认它必须写入 issues.json。
- 修复结论：handoff 成为 artifact contract。
- 修复效果：提升 workflow-to-workflow 协作可靠性。

## Round 26：depth 与 mode 容易混淆

- 问题描述：`deep` 是审查深度，不是交互/写入 mode。
- 具体修复：前序方案已改为 `depth:deep`，并在 metadata 增加 `depth`。
- 修复结论：mode 管交互和 artifact，depth 管审查强度。
- 修复效果：减少参数冲突和实现分支复杂度。

## Round 27：Figma redaction 需要进入 materialized source_inputs

- 问题描述：只说 redaction mode 不够，必须在 raw input 和 contract 里可追踪。
- 具体修复：Input Materialization 章节要求 materialized raw input 写 source_inputs，并标记 data_sensitivity 与 redaction mode。
- 修复结论：隐私策略成为证据元数据的一部分。
- 修复效果：减少 Figma 文案泄漏风险。

## Round 28：MVP 分期需要从功能优先转为 contract hardening 优先

- 问题描述：如果优先做更多专家/规则包，会放大不稳定 contract。
- 具体修复：v0.1 增加 artifact-manifest、strict issue schema、run-scoped CLI e2e；v0.2 才深化 Figma/PRD。
- 修复结论：先稳核心闭环，再扩专家域。
- 修复效果：符合 80/20 和最小可维护落地顺序。

## Round 29：当前测试策略缺少 report-only no-write

- 问题描述：report-only 的价值在只读，但缺测试很容易回归。
- 具体修复：P0 优化方向新增 report-only no-write 回归测试，确认不会写 preflight/input/preview/run artifacts。
- 修复结论：只读模式需要测试保障。
- 修复效果：支持与其他只读验证并行运行。

## Round 30：方案缺少本轮审查证据沉淀

- 问题描述：用户要求持续围绕方案做小步迭代，如果没有 loop 记录，后续难以判断哪些问题已审过。
- 具体修复：新增本文件，记录 30 轮问题描述、修复细节、结论和效果。
- 修复结论：方案演化过程本身成为可追踪 artifact。
- 修复效果：增强 Knowledge 节点沉淀，为后续实现任务拆分提供上下文。

## 本轮总体结论

升级方案的方向保持正确：`spec-app-consistency-audit` 应作为 App 专项一致性审查 workflow，不并入 `spec-code-review`，通过 run-scoped artifact、headless envelope 和 machine-readable handoff 与 code-review 协作。

本轮主要优化不是扩大功能，而是补齐落地边界：

```text
1. 当前实现事实与迁移边界
2. run-scoped input/materialization
3. artifact-manifest.json
4. schema additive hardening
5. reason_code 最小集合
6. Graph/MCP provider 边界
7. report-only no-write 测试要求
8. source skill / command template / CLI e2e 的同步落地路径
```

最终效果：方案更贴近 `Codebase -> Graph -> Spec -> Plan -> Tasks -> Code -> Review -> Knowledge` 链路，减少隐式上下文和不可追踪判断，同时保持轻契约与可演进架构。

# 2026-05-01 再次 50 轮全流程 Skill / Agent 细致审查记录

本轮基于全项目 workflow、41 个 source skill 审计信号、50 个 source agent 清单、`spec-code-review` / `spec-doc-review` / `spec-skill-audit` 的设计思路，以及 `docs/10-prompt/结构化项目角色契约.md` 的要求，对升级方案进行 Round 31-80 的小步复审与修订。

## Round 31：app-audit 在全链路中的位置仍可能被误解

- 问题描述：方案强调 App 专项审查，但没有把它明确放回 `Codebase -> Graph -> Spec -> Plan -> Tasks -> Code -> Review -> Knowledge` 链路，容易被实现成新的总控入口。
- 具体修复：新增“全流程 Skill / Agent 协同边界”，声明 app-audit 是专项 Review lens，只消费上游事实、补充 review 证据、输出 handoff 建议。
- 修复结论：app-audit 不再承担跨链路编排职责。
- 修复效果：避免架构漂移为强流程引擎，符合 light contract。

## Round 32：Graph readiness 不应由 app-audit 修复

- 问题描述：Graph / MCP 不可用时，app-audit 如果自行 setup，会越界到 `spec-mcp-setup` / `spec-graph-bootstrap`。
- 具体修复：方案新增 Graph 不可用、stale、definitions-only 时只记录 degraded 并回退 bounded source scan，同时输出 handoff。
- 修复结论：Graph 是可选输入，不是 app-audit 的职责本体。
- 修复效果：提升 provider 边界清晰度，避免下游 workflow 依赖 provider 内部细节。

## Round 33：Graph provider 输出容易被误当 scope authority

- 问题描述：GitNexus、Serena 或其他 provider 的 pointers 若被当成审查范围权威，会让 provider 噪声影响 App 审查结论。
- 具体修复：保留“provider 只能给 candidate pointers”的原则，并在全流程章节补充 Graph 只服务 Codebase / Graph 节点。
- 修复结论：最终 issue 仍必须引用项目特定 source evidence。
- 修复效果：降低 provider stale 或 definitions-only 带来的误报。

## Round 34：PRD 缺失时 Product Expert 不应补写需求

- 问题描述：缺 PRD 时，Product Expert 可能根据代码或术语猜测产品目标。
- 具体修复：新增 handoff 建议映射：PRD、验收标准、用户流程缺失时建议 `spec-brainstorm` / `spec-plan`，app-audit 只输出 advisory。
- 修复结论：产品语义缺口回到 Spec / Plan 节点处理。
- 修复效果：减少隐式上下文和不可追踪产品判断。

## Round 35：文档质量问题不能混入 App issue

- 问题描述：PRD 或 plan 本身不一致时，app-audit 可能把文档质量问题误报成实现问题。
- 具体修复：新增 `spec-doc-review` handoff：文档范围、验收、非目标、逻辑冲突交给 doc-review。
- 修复结论：app-audit 只审实现与已知 contract 的一致性。
- 修复效果：减少跨 workflow 误分类。

## Round 36：任务拆分与执行不属于 app-audit

- 问题描述：审查发现问题后，方案可能诱导 app-audit 直接拆 task 或写实现计划。
- 具体修复：全流程边界中明确 app-audit 不拆任务、不改代码、不替代 `spec-work` / `spec-work-beta`。
- 修复结论：Tasks / Code 节点保留给对应 workflow。
- 修复效果：避免 App 审查扩成实施编排器。

## Round 37：code-review 与 app-audit 的主次关系需更硬

- 问题描述：App 专项 finding 和通用 correctness/security/test coverage finding 容易互相吞并。
- 具体修复：新增 handoff mapping：通用代码正确性、安全、性能、测试覆盖、API contract 仍回到 `spec-code-review`。
- 修复结论：app-audit 只提供 App 专项 lens 和 `code_review_handoff`。
- 修复效果：保留 code-review 的通用审查心智模型。

## Round 38：skill-audit deterministic scorecard 不能成为 app-audit gate

- 问题描述：全量 skill-audit 产生 P0/P1/P2 信号，但其中很多是结构化信号，需要 LLM 语义复核。
- 具体修复：方案新增“Skill-Audit 信号处理”：scorecard 只能作为设计输入，不作为 App 审查失败门。
- 修复结论：治理信号不污染 App issue 协议。
- 修复效果：避免把 advisory facts 当 confirmed truth。

## Round 39：runtime generated assets 边界需要在 app-audit 中复述

- 问题描述：spec-update 审计信号显示 runtime 资产可能被误改，app-audit 方案也必须防止类似误用。
- 具体修复：全局安全边界和 agent catalog 边界重复声明不得手改 `.claude/`、`.codex/`、`.agents/skills/`。
- 修复结论：source-first 成为 app-audit 落地硬约束。
- 修复效果：减少 source/runtime drift。

## Round 40：report-only no-write 需要覆盖 handoff

- 问题描述：新增 `workflow-handoff-suggestions.json` 后，如果 report-only 仍写该文件，会破坏 no-write contract。
- 具体修复：handoff contract 明确 report-only 不写文件，只在当前响应 Coverage / Follow-ups 输出。
- 修复结论：report-only 的只读边界覆盖所有新增 artifact。
- 修复效果：支持并行只读验证，不污染工作树。

## Round 41：缺少跨 workflow handoff 的机器协议

- 问题描述：只在报告里写“建议运行某 workflow”不利于 parent workflow 稳定消费。
- 具体修复：新增 `workflow-handoff-suggestions.json` schema，包含 `target_workflow`、`reason_code`、`recommended_action`、`blocking`、`confidence`。
- 修复结论：handoff 从自然语言建议升级为轻量 artifact contract。
- 修复效果：提高 workflow-to-workflow 协作可靠性。

## Round 42：code_review_handoff 与 workflow_handoff_suggestions 容易混淆

- 问题描述：前者是单条 finding 移交，后者是下一步 workflow 建议，混用会让 code-review 消费逻辑复杂化。
- 具体修复：P1 issue 协议新增区分规则：`code_review_handoff` 属于 issue，`workflow_handoff_suggestions` 属于 run-level follow-up。
- 修复结论：finding-level 和 run-level 协作协议分离。
- 修复效果：减少 downstream consumer 的分支判断。

## Round 43：Agent catalog 边界不够显式

- 问题描述：app-audit 专家 lens 如果被直接放进全局 `agents/`，会污染跨 workflow agent 目录。
- 具体修复：新增 Agent Catalog 章节：skill-local prompts 是 App 专项专家 source-of-truth，`agents/` 只放跨 workflow 稳定通用 agent。
- 修复结论：专家资产拥有清晰归属。
- 修复效果：降低 agent collection 化风险。

## Round 44：App 专家升级为全局 agent 缺少准入条件

- 问题描述：某个 App lens 未来可能复用，但没有治理门槛会导致随意晋升。
- 具体修复：新增 graduation policy：必须更新 agents source、dual-host governance、触发边界、fresh-source eval 和回归测试。
- 修复结论：跨 workflow agent 晋升有明确证据要求。
- 修复效果：保持全局 agent catalog 稳定。

## Round 45：不应复制全局 reviewer 的 source 内容

- 问题描述：app-audit 可能为了完整审查复制 security、testing、architecture reviewer 内容，造成双真相源。
- 具体修复：方案声明可引用全局 agent 方法论，但不复制 source；App 专项内容留在 skill-local prompts。
- 修复结论：避免 reviewer 逻辑分叉。
- 修复效果：提高长期维护性。

## Round 46：Validation Pass 不应拥有最终语义裁决权

- 问题描述：独立 validator 容易被误实现成第二个规则门或最终裁判。
- 具体修复：Agent Catalog 章节补充：Validation Pass 只读验证 finding，不拥有修改权或最终语义裁决权。
- 修复结论：最终分区仍由 Evidence Gate + LLM Evidence Auditor + validation_status 共同决定。
- 修复效果：防止强编排和硬 gate 扩张。

## Round 47：远程脚本执行风险需要成为 app-audit 默认禁止项

- 问题描述：skill-audit P0 暴露 remote script pipe 风险，app-audit 虽不安装依赖，也应明确禁止。
- 具体修复：全局安全边界新增“不执行 remote script pipe，不下载未知脚本，不在审查过程中安装依赖”。
- 修复结论：审查流程保持静态、只读、可预测。
- 修复效果：降低 supply-chain 和权限风险。

## Round 48：secrets / .env 边界需要写入方案

- 问题描述：全量 skill 审计中存在 `.env*` 复制或读取信号，app-audit 处理 App 配置时可能触碰 secrets。
- 具体修复：新增禁止读取、复制、输出或 materialize `.env` / `.env.*` / token / credentials；只允许白名单非敏感字段。
- 修复结论：secret 默认阻断或脱敏。
- 修复效果：减少审查报告泄露敏感信息的风险。

## Round 49：默认网络和运行验证边界需收紧

- 问题描述：App 审查容易自然延伸到 Figma 拉取、浏览器、模拟器、真机、云真机。
- 具体修复：全局安全边界新增默认不启动网络请求、build、simulator、real device、cloud device、browser。
- 修复结论：运行验证仅作为后续建议，不是默认审查动作。
- 修复效果：保持静态审查入口轻量可复现。

## Round 50：Figma / PRD 长文本泄漏风险需要统一治理

- 问题描述：Figma 和 PRD 可能包含内部文案、个人信息或敏感业务规则。
- 具体修复：全局安全边界要求长文本、PII、凭证样式字符串必须经过 redaction mode，不能直接进入报告。
- 修复结论：redaction 不只属于 Figma extractor，也属于全局报告策略。
- 修复效果：提升数据治理一致性。

## Round 51：data_sensitivity 需要贯穿新增 artifact

- 问题描述：新增 handoff、manifest 和 eval 后，如果没有 sensitivity 字段，会破坏 artifact 消费安全。
- 具体修复：handoff schema 和安全边界要求 source_inputs / artifacts 标记 data_sensitivity 或 redacted 状态。
- 修复结论：数据敏感度成为跨 artifact 元数据。
- 修复效果：后续 consumer 可以按敏感等级处理输出。

## Round 52：manifest 需要纳入 handoff artifact

- 问题描述：`workflow-handoff-suggestions.json` 如果不进入 manifest，parent workflow 无法验证它属于当前 run。
- 具体修复：输出 artifact 树新增该文件，并声明 manifest 是 run 内部索引。
- 修复结论：handoff artifact 与其他 run-scoped artifact 一致。
- 修复效果：减少 latest 指针或旧产物误读。

## Round 53：failed run 与 latest 指针关系仍需保持

- 问题描述：新增安全阻断和 handoff 后，failed/degraded 场景更多，latest 指针不能随意更新。
- 具体修复：保留 failed run 不更新 latest 的语义，并让 handoff 作为 run 内 artifact，不改变 latest 规则。
- 修复结论：失败语义未被新增 artifact 破坏。
- 修复效果：避免 downstream workflow 读取失败产物当最新结论。

## Round 54：eval readiness 不能只写“需要测试”

- 问题描述：skill-audit 显示 eval readiness 是系统性弱点，方案需要具体 case 类型。
- 具体修复：新增 v0.1 最小 eval / 回归集，覆盖 trigger、boundary、mode、artifact、evidence、safety、handoff。
- 修复结论：eval 从泛泛要求变成可落地清单。
- 修复效果：实现阶段可直接转为 fixtures 和 unit/e2e 测试。

## Round 55：trigger case 需要证明 app-audit 不滥触发

- 问题描述：App/KMP/UI 改动应推荐 app-audit，但纯后端或机械改动不应触发。
- 具体修复：eval readiness 增加 trigger cases：App diff 推荐，非 App 后端 diff 不触发。
- 修复结论：触发精度成为测试目标。
- 修复效果：降低用户被专项审查打扰的概率。

## Round 56：boundary case 需要覆盖非 App issue

- 问题描述：缺 PRD、Figma ref-only、skill 质量问题都容易被误报成 App issue。
- 具体修复：eval readiness 增加 boundary cases，要求这些场景输出 degraded 或 handoff，而不是 confirmed issue。
- 修复结论：边界错误可以被回归测试捕获。
- 修复效果：减少跨域误报。

## Round 57：mode case 需要覆盖失败 envelope

- 问题描述：headless、report-only、from:code-review 的行为最容易在实现中回归。
- 具体修复：eval readiness 增加 mode cases：report-only no-write、headless 缺 scope failed envelope、from:code-review 不询问用户。
- 修复结论：mode contract 可测试。
- 修复效果：提高 parent workflow 调用可靠性。

## Round 58：evidence case 需要覆盖误确认路径

- 问题描述：rule-pack-only、未确认 industry、validator_unavailable 是最容易导致误判的路径。
- 具体修复：eval readiness 增加 evidence cases，覆盖 rule-pack-only rejection、industry advisory、validator_unavailable 保留/降级规则。
- 修复结论：证据门禁和语义审计有最小验证集。
- 修复效果：降低 hallucinated finding 外部化风险。

## Round 59：safety case 需要成为 P0 测试

- 问题描述：安全边界如果只写在文档中，容易在 extractor 或未来 runtime 验证中被绕过。
- 具体修复：P0 稳定执行清单新增 secrets、remote execution、network、runtime validation 默认阻断或 deferred 测试。
- 修复结论：安全不是 P2 优化，而是默认执行前提。
- 修复效果：减少高影响回归。

## Round 60：handoff case 需要验证 reason_code 与目标 workflow

- 问题描述：handoff 如果没有测试，可能退化为自由文本或错误路由。
- 具体修复：eval readiness 增加 handoff cases，验证 `target_workflow`、`reason_code`、`confidence` 和 headless envelope 同步。
- 修复结论：handoff 成为可验证 contract。
- 修复效果：提升 workflow coordination layer 的稳定性。

## Round 61：v0.1 artifact 范围需要加入 handoff 但保持克制

- 问题描述：新增 handoff artifact 后，v0.1 可能继续膨胀到全量 contract。
- 具体修复：v0.1 必做列表只新增 `workflow-handoff-suggestions.json` 和安全/eval 边界，仍暂缓全量行业/运行验证。
- 修复结论：核心闭环增强，但默认产物仍保持小。
- 修复效果：符合 80/20 和 MVP hardening 优先。

## Round 62：专家数量仍需由 planner 控制

- 问题描述：全流程和全 agent 视角可能诱导每次启用全部专家。
- 具体修复：保留 LLM Audit Planner 按 facts 选择最小专家组，并明确跨 workflow 建议写 handoff，不调度外部 workflow。
- 修复结论：专家不是越多越好。
- 修复效果：降低成本、噪声和上下文污染。

## Round 63：industry confirmed 仍必须依赖用户或 profile 确认

- 问题描述：行业术语候选在 App 项目中很常见，容易被过度解释。
- 具体修复：继续保留未确认 industry 只能 advisory/candidate，并在 eval evidence case 中覆盖。
- 修复结论：行业专家不能单独制造 confirmed issue。
- 修复效果：减少行业 rule-pack 误伤。

## Round 64：Rule Pack 只做上下文增强

- 问题描述：规则包越丰富，越容易变成隐藏规则引擎。
- 具体修复：方案保留 rule-pack-only confirmed rejection，并把它列入 eval。
- 修复结论：Rule Pack 不替代 LLM 判断和项目证据。
- 修复效果：保持 “Scripts prepare, LLM decides”。

## Round 65：spec-work / spec-work-beta 与 app-audit 边界需要明确

- 问题描述：app-audit finding 可能看起来像可自动修复任务，诱导直接实施。
- 具体修复：全流程边界声明 app-audit 不改代码、不生成 task pack，只可移交给 work 类 workflow。
- 修复结论：审查与执行分离。
- 修复效果：降低自动修错造成产品语义偏差的风险。

## Round 66：spec-polish-beta / runtime validation 不应进入默认路径

- 问题描述：移动端视觉、键盘、安全区、真机权限等问题需要运行验证，但默认启动会变重。
- 具体修复：handoff mapping 将 runtime/browser/simulator/real-device 验证交给后续 workflow，app-audit 只输出 runtime_verification_suggestions。
- 修复结论：运行验证是 follow-up，不是 app-audit 默认步骤。
- 修复效果：保持静态审查的可重复性。

## Round 67：spec-compound 知识沉淀不能自动写

- 问题描述：审查中发现模式或规范后，直接写 docs/solutions 会跳过用户确认和知识质量评估。
- 具体修复：handoff mapping 中把 durable learnings 交给 `spec-compound` / `spec-compound-refresh`。
- 修复结论：Knowledge 节点通过专门 workflow 承接。
- 修复效果：提高知识沉淀质量，避免文档噪声。

## Round 68：session / slack / web research 不是默认 App 证据

- 问题描述：组织讨论、历史会话或外部资料可提供背景，但不能替代当前项目 evidence。
- 具体修复：全流程边界要求最终 issue 引用项目特定 evidence；外部或历史上下文只能作为 handoff 或 advisory background。
- 修复结论：App issue 不依赖不可复现口头上下文。
- 修复效果：提升审查可追溯性。

## Round 69：spec-update 只能作为 runtime stale 建议

- 问题描述：发现 host runtime 可能过期时，app-audit 如果直接更新会破坏审查边界。
- 具体修复：handoff mapping 增加 `spec-update`，并新增 `host_runtime_stale_suspected` reason_code。
- 修复结论：更新行为由用户和 update workflow 控制。
- 修复效果：避免审查过程产生环境突变。

## Round 70：using-spec-first 只做入口治理

- 问题描述：app-audit 方案可能把入口路由判断写成内部逻辑。
- 具体修复：handoff mapping 增加 using-spec-first：当请求本质是 workflow 入口判断时，app-audit 不接管。
- 修复结论：入口治理仍由 meta skill 承担。
- 修复效果：减少重复路由规则。

## Round 71：全部 agents 不代表全部调用

- 问题描述：项目有 50 个 source agent，但 app-audit 不应把它们作为可全量调度池。
- 具体修复：Agent Catalog 章节区分全局通用 agent 与 skill-local app experts，并强调按需引用方法论。
- 修复结论：agent catalog 是能力边界，不是专家大会清单。
- 修复效果：降低上下文膨胀。

## Round 72：全局 reviewer 方法论可借鉴但不复制

- 问题描述：spec-design、product、security、testing 等 reviewer 的方法论对 App 审查有价值，但复制会带来漂移。
- 具体修复：方案声明 app-audit 可引用方法论，不复制 source；如需跨 workflow 复用必须走 agent graduation。
- 修复结论：复用通过边界和治理实现。
- 修复效果：避免 prompt fork。

## Round 73：安全 reviewer 与 App 安全 lens 需要分层

- 问题描述：App 安全问题可能和通用 security review 重叠。
- 具体修复：全流程边界把通用安全 review 交给 code-review，app-audit 聚焦 token、隐私日志、WebView、Deep Link、移动端权限等 App 一致性相关风险。
- 修复结论：安全审查不重复、不缺位。
- 修复效果：提升问题归属清晰度。

## Round 74：测试 reviewer 与 App eval readiness 需要分工

- 问题描述：app-audit 的 eval readiness 是 workflow 自测，不等于产品代码测试覆盖。
- 具体修复：方案将 eval readiness 限定为 workflow contract 回归；产品测试覆盖仍由 code-review/testing reviewer 判断。
- 修复结论：workflow 测试和产品测试分离。
- 修复效果：避免把“方案可执行性”误当“产品质量已验证”。

## Round 75：API / contract review 不应被 App contract extraction 替代

- 问题描述：App codebase contract、analytics contract 等可能被误认为通用 API contract review。
- 具体修复：全流程边界声明 API contract、安全、性能等通用问题仍回到 code-review；app-audit contract 是一致性证据输入。
- 修复结论：contract extraction 不越权。
- 修复效果：保持 downstream consumer 对 contract 类型的理解一致。

## Round 76：issue 字段需要支持跨 workflow 区分

- 问题描述：新增 handoff 后，issue schema 如果不明确边界，report writer 可能把 run-level 建议混入 issue。
- 具体修复：P1 issue 协议明确 `code_review_handoff` 属于 issue，而 `workflow_handoff_suggestions` 属于 run-level。
- 修复结论：issue 字段语义更清晰。
- 修复效果：降低 report synthesis 错误。

## Round 77：最终报告需要展示 handoff 但不制造硬 gate

- 问题描述：用户需要看到下一步建议，但报告不应表现成必须执行的状态机。
- 具体修复：最终报告结构新增 “Workflow Handoff Suggestions”，只列建议和 reason，不作为 verdict gate。
- 修复结论：建议可见但不强制。
- 修复效果：增强行动性，同时保持轻契约。

## Round 78：schema 演进需要包含 handoff schema

- 问题描述：此前 schema hardening 只覆盖 issue，新增 handoff artifact 后也需要版本化。
- 具体修复：handoff contract 定义 `spec-app-consistency-audit-handoff-suggestions.v1`。
- 修复结论：新增 artifact 有独立 schema_version。
- 修复效果：便于未来 consumer 兼容演进。

## Round 79：迁移仍必须 source-first

- 问题描述：方案新增 SKILL.md、command template、eval、handoff 等落地项，容易被实现者先改 runtime。
- 具体修复：新增章节和 P0 清单反复声明先改 source-of-truth，不手改 generated runtime。
- 修复结论：迁移路径与项目角色契约一致。
- 修复效果：减少双宿主 drift。

## Round 80：本轮总收敛需要回到项目目标

- 问题描述：50 轮审查若只扩大能力，会偏离 spec-first 的长期演化方向。
- 具体修复：本轮修订聚焦边界、handoff、安全、eval、source/runtime、agent catalog，而不是新增更多专家或规则包。
- 修复结论：方案更像 workflow harness 的专项节点，而不是规则引擎或 agent collection。
- 修复效果：提升输入质量、证据链、可维护性和用户真实研发效率，同时保持轻契约、显式边界和可演进架构。

## 再次 50 轮总体结论

本轮没有把 `spec-app-consistency-audit` 扩大成全流程总控，而是把它更准确地嵌入 spec-first 全链路：

```text
1. 对上游：缺 Graph / PRD / Figma / plan 时输出 degraded 与 handoff，不伪造事实。
2. 对中游：只做 App 专项一致性审查，不拆任务、不改代码、不替代 code-review。
3. 对下游：通过 code_review_handoff 移交 finding，通过 workflow_handoff_suggestions 建议下一步 workflow。
4. 对治理：吸收 skill-audit 信号，但不把 scorecard 变成 App 审查 gate。
5. 对安全：默认阻断 secrets、remote execution、network 和 runtime validation 的隐式扩张。
6. 对验证：把 trigger、boundary、mode、artifact、evidence、safety、handoff 纳入 v0.1 最小 eval。
```

最终效果：升级方案从“App 审查能力设计”进一步收敛为“spec-first workflow harness 中的 App 专项 Review 节点设计”。它更能服务核心链路，减少隐式上下文和临时决策，同时避免重状态、强编排、硬规则和过度工程。

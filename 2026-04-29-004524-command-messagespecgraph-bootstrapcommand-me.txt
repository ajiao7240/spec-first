⏺ 根因已完全明确。以下是完整分析：

  ---
  根因分析

  直接原因：KuzuDB 原生崩溃（SIGSEGV）

  崩溃发生在 gitnexus 打印 banner
  之后、任何文件解析之前，这个时序强烈指向数据库初始化/open 
  阶段，而不是代码索引阶段。

  .gitnexus/ 里有：
  lbug     105MB  ← KuzuDB 数据文件（魔数 LBUG 确认）
  lbug.wal  61B   ← Write-Ahead Log，只有 61 字节，异常小
  meta.json       ← 上次索引：commit 417c86a0，11:29 UTC

  与当前 HEAD 8b4672ce（22:46 UTC）差了 4
  个提交，说明这是上次会话留下的残留数据库。

  崩溃链

  gitnexus analyze --force
    └─ 尝试打开 .gitnexus/lbug（KuzuDB native C++ 层）
         └─ WAL 恢复阶段：lbug.wal 只有 61B，状态不一致
              └─ KuzuDB C++ 读取非法内存指针
                   └─ SIGSEGV（exit 139）

  lbug.wal 的 61B 表明上一次 analyze 在提交事务前就崩溃了，留下了半完成的 
  WAL。KuzuDB 的 C++ native 层在 WAL 恢复时触发段错误——这是 KuzuDB 的 known
  issue（文档归类为"concurrent analyze corruption"触发的数据库损坏场景）。

  为何 --force 没有解决问题

  --force 是 gitnexus 的标志，跳过增量检测、强制重新索引，但它不清理 KuzuDB 
  本身。KuzuDB 在 open 时如果发现损坏的 WAL，在 C++ 层就已崩溃，Node.js
  侧根本没机会执行任何清理逻辑。

  ---
  修复步骤

  第一步：清理损坏的 KuzuDB 文件
  rm /Users/kuang/xiaobu/spec-first/.gitnexus/lbug \
     /Users/kuang/xiaobu/spec-first/.gitnexus/lbug.wal

  （meta.json 可保留或一并删除，gitnexus 会重新生成）

  第二步：可选——加堆内存保险（981 个 git 文件问题不大，但预防性加上）
  NODE_OPTIONS="--max-old-space-size=8192" npx -y gitnexus@1.6.4-rc.21 analyze
  --force

  第三步：重新运行 bootstrap
  bash .claude/spec-first/workflows/spec-graph-bootstrap/scripts/bootstrap-provi
  ders.sh

  ---
  预防建议

  在仓库根加 .gitnexusignore，排除生成目录，减少索引负担：
  .spec-first/
  node_modules/
  .git/

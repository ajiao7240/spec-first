## 1. Skill 概述

### 1.1 核心定义

**focus-requirements** 是一个**需求边界切割器**。

核心任务：把"已审过的全局需求"收敛成"某个 owner 范围的 PRD"。

```
┌─────────────────┐         ┌─────────────────┐
│  全局需求文档    │  ──────► │  Owner PRD      │
│  (多团队相关)    │         │  (单一责任边界)  │
└─────────────────┘         └─────────────────┘
                                   │
                    ┌──────────────┼──────────────┐
                    ▼              ▼              ▼
              focus-req.md   side-req.md    handoff-summary.md
```

### 1.2 解决的问题

| 痛点 | 这个 Skill 的解法 |
|------|-------------------|
| 大需求跨多团队，边界不清 | 明确 In/Out Scope |
| 依赖关系模糊 | 显式的 Dependencies 章节 |
| 验收标准各说各话 | 可验证的 Acceptance Criteria |
| 不敢说"不知道" | 诚实的 Open Questions |

### 1.3 三个输出文件

| 文件 | 给谁看 | 作用 |
|------|--------|------|
| `docs/requirements/focus-requirements.md` | 自己 | 我知道我要做什么 |
| `handoff/side-requirements.md` | 依赖方 | 我知道我依赖谁 |
| `handoff/handoff-summary.md` | 审查方 | 让管理层知道我准备好了 |

## 2. 深度分析

### 2.1 五个成功标准的本质

这五个标准本质上是回答五个问题：

| # | 成功标准 | 回答的问题 |
|---|----------|-----------|
| 1 | Owner Scope 清晰 | **谁做？** |
| 2 | In/Out Scope 清晰 | **做什么？不做什么？** |
| 3 | Dependencies 隔离 | **需要谁配合？** |
| 4 | Acceptance 可验证 | **怎么算做完？** |
| 5 | Open Questions 诚实 | **还有什么不确定？** |

这五个问题回答清楚了，后续的技术设计、任务拆解、代码实现才能顺利开始。

### 2.2 交互策略：最小干预原则

**默认模式**：直接收敛（边界清楚就不问）

**触发确认的条件**（三种歧义触发器）：
1. `Owner Boundary Ambiguity` - 谁负责不清楚
2. `Dependency Ownership Ambiguity` - 依赖归谁不清楚
3. `Acceptance Boundary Ambiguity` - 完成标准不清楚

这是一个**最小干预原则**的设计：
- 不做固定访谈脚本
- 只问必要的澄清问题
- 问题解决后立即继续

### 2.3 视觉证据规则

这个 skill 有一个非常特别的设计：**图片是一手源材料，不是装饰**。

关键规则：
- 必须先读截图，再压缩文本
- 提取精确的标签、数字、UI 状态
- 如果是流程图，要重建成 ASCII 图
- 如果图片暗示了文本没说的需求，要标注"从视觉证据推断"
- 如果图片看不清，要停下来问（NEEDS_CONTEXT）

这解决了什么问题？**产品经理习惯用截图说话，但开发需要精确的文字描述**。

### 2.4 非目标边界

明确不做：
- 架构设计
- API 设计
- 任务拆解
- 工作量估算
- 跨 owner 协调
- 悄悄补全缺失的业务规则

只做一件事：**需求边界切割**。

### 3.2 输入依赖问题

**当前状态**（已验证）：

| 产物 | 当前要求 | 验证方法 |
|------|----------|----------|
| `summary` | **推荐**（非必需） | `grep -A5 "输入上下文"` |

✅ **此问题已解决**，无需额外修复。

### 3.3 输出路径问题

当前输出三个固定文件：
- `docs/requirements/focus-requirements.md`
- `handoff/side-requirements.md`
- `handoff/handoff-summary.md`

| 问题 | 影响 | 验证方法 |
|------|------|----------|
| 路径硬编码 | 无法适配不同项目的目录结构 | `grep "docs/requirements" SKILL.md` |
| 文件名固定 | 同一项目多个 owner 需求会覆盖 | 同上 |
| `handoff/` 目录 | 很多项目中不存在 | 实际项目测试 |

### 3.4 规模化能力不足

当前设计：**一次只处理一个 owner**。

实际场景：大型需求可能需要同时切割给 3-5 个 owner。

| 问题 | 后果 | 验证方法 |
|------|------|----------|
| 需要运行多次 | 用户体验差 | 用户反馈 |
| 每次可能不一致 | 边界划分冲突 | 多次执行对比 |
| 无全局视图 | 无法检查覆盖完整性 | 检查输出文件 |

### 3.5 模板结构问题

当前 `focus-requirements.md` 模板有 10 个章节：

1. Background
2. Owner Scope
3. In Scope
4. Out of Scope
5. Relevant Flows
6. Dependencies
7. Acceptance Criteria
8. Non-Functional Constraints
9. Assumptions
10. Open Questions

| 问题 | 影响 | 验证方法 |
|------|------|----------|
| 简单需求章节冗余 | 很多章节会是空的或只有占位符 | 实际使用测试 |
| 模板静态 | 不会根据需求复杂度调整 | 检查模板文件 |

### 3.6 示例覆盖问题

当前有 3 个示例：

| 示例 | 场景 | 验证方法 |
|------|------|----------|

| simple-profile-copy-update | 简单直接收敛 | 同上 |
| ambiguity-confirmation | 歧义触发场景 | 同上 |

**缺少的场景**：

- 新项目需求（从零开始，不是增量）
- 纯后端需求
- 需求变更场景

## 8. 核心洞察

### 8.1 设计亮点

1. **最小干预原则**：能直接判断的就不问，只在必要时确认
2. **边界清晰**：明确只做需求切割，不做技术设计
3. **视觉证据优先**：把截图当作一手源材料处理
4. **诚实标注**：不确定的就是不确定，不瞎猜
5. **降级模式完备**：first 产物缺失时仍可工作

### 8.2 落地摩擦点

这个 skill 的设计理念很好，但在**实际落地**时有摩擦点：

| 摩擦点 | 表现 | 影响 | 状态 |
|--------|------|------|------|
| ~~前置条件太重~~ | ~~需要 first 产物~~ | ~~用户被阻塞~~ | ✅ 已有降级模式 |
| **输出不够灵活** | 路径/文件名固定 | 适配困难 | ⏳ 待修复 |
| **规模化能力不足** | 一次只能处理一个 owner | 大需求效率低 | ⏳ 待修复 |
| **下游链路断裂** | 引用不存在的 skill | 工作流无法继续 | ⏳ 待修复 |

### 8.3 一句话总结

> **把"大家一起做"变成"你做这个，他做那个，你们这样配合"。**

这是一个**降低团队协作熵**的工具。核心价值不是"生成文档"，而是**消除协作中的模糊性**。

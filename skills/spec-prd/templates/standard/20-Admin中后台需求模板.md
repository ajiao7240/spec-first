## Requirements [core]

（同通用模板：EARS + BR 编号）

Admin 需求尤其要把以下规则写成 R 或 BR：

- 哪些角色可见、可查、可导出、可审批、可复核、可撤销。
- 敏感字段何时明文、何时脱敏、谁能解密查看。
- 危险操作是否需要二次确认、原因填写、四眼复核、操作留痕。
- 任务状态、SLA、批量操作、导出上限、跨页选择、异常恢复。

## 权限矩阵 [conditional，Admin 专属]

> 权限控制粒度：菜单级 + 按钮级 + 字段级 + 数据范围。本文写"哪个角色能见/能做什么"，不写权限码命名设计。

| 角色 | 可见菜单 | 可用按钮 | 数据范围 | 敏感字段（是否脱敏/可见） | 是否可导出 | 是否可复核 |
| --- | --- | --- | --- | --- | --- | --- |

角色隔离建议逐项确认：

- 经办人与复核人是否必须不同人。
- 不同运营角色是否可以互相查看或修改处理结果。
- 超级管理员是否可绕过业务复核；若可以，必须写审计和使用限制。

## 作业流与复核机制 [conditional，Admin 专属]

| 节点 | 操作者 | 前置条件 | 可执行动作 | 下一状态 | 是否需复核 | 失败/驳回结果 |
| --- | --- | --- | --- | --- | --- | --- |
| 待处理 | | | 通过/驳回/补材料/转人工 | | 是 / 否 | |
| 复核中 | | | 复核通过/复核驳回 | | | |
| 已完成 | | | 查看/导出/归档 | 终态 | | |

必须写清：是否允许同一人提交并复核、是否允许撤回、是否允许二次修改、驳回后用户或运营看到什么。

### 字段联动与业务规则

- 批量操作规则：勾选数、跨页保留、批量上限、部分成功。
- 危险操作：删除、禁用、驳回、强制通过、手工调整等需二次确认与原因。
- 导出规则：数量上限、超限提示、异步导出、文件有效期、下载权限。
- 搜索规则：敏感条件是否允许模糊搜索。

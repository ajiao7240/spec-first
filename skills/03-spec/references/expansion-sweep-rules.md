# Expansion Sweep 规则

Step 5 Expansion Sweep（发散扫描）的执行策略。

---

## 核心原则

**主动识别边界/失败场景/NFR，避免遗漏。**

---

## 四大扫描维度

### 1. 边界扫描

**目标**：识别输入边界值

**检查项**：
- 最小值（0, null, empty）
- 最大值（MAX_INT, 超长字符串）
- 非法值（负数、特殊字符、SQL 注入）
- 边界组合（最小+最大、null+empty）

**示例**：
```markdown
## FR-AUTH-001: 短信验证码登录

边界条件：
- 手机号：11 位数字，不支持国际号码
- 验证码：6 位数字，有效期 120s
- 重试次数：最多 3 次/手机号/小时
- 空值处理：手机号/验证码为空时返回 400
```

---

### 2. 失败场景扫描

**目标**：识别异常路径

**检查项**：
- 网络失败（超时、断连）
- 服务失败（第三方 API 不可用）
- 权限不足（未登录、无权限）
- 资源耗尽（并发限制、配额用尽）
- 数据异常（重复、冲突、不一致）

**示例**：
```markdown
## FR-AUTH-001: 短信验证码登录

失败场景：
- 短信发送失败 → 返回 503，提示"服务暂时不可用"
- 验证码过期 → 返回 400，提示"验证码已过期"
- 验证码错误 → 返回 401，剩余重试次数 -1
- 重试次数耗尽 → 返回 429，锁定 1 小时
```

---

### 3. NFR 扫描

**目标**：识别非功能需求

**检查项**：
- 性能（响应时间、吞吐量）
- 安全（加密、认证、授权）
- 可用性（SLA、容错、降级）
- 可维护性（日志、监控、告警）
- 可扩展性（水平扩展、垂直扩展）

**示例**：
```markdown
## NFR-AUTH-001: 短信验证码登录性能要求

- 响应时间：P95 < 500ms
- 吞吐量：1000 QPS
- 可用性：99.9%
- 安全：验证码加密存储，传输 HTTPS
- 监控：发送成功率、验证成功率、平均响应时间
```

---

### 4. 依赖风险扫描

**目标**：识别外部依赖的失败影响

**检查项**：
- 依赖不可用时的降级策略
- 依赖超时时的重试策略
- 依赖限流时的排队策略
- 依赖数据不一致时的修复策略

**示例**：
```markdown
## 依赖风险：短信服务商

- 依赖：阿里云短信服务
- 失败影响：用户无法登录
- 降级策略：切换备用短信服务商（腾讯云）
- 超时策略：3s 超时，重试 1 次
- 监控：短信发送成功率 < 95% 时告警
```

---

## 执行流程

### Step 1: 读取 FR 列表

从 PRD 或 spec.md 读取所有 FR。

### Step 2: 逐个 FR 扫描

对每个 FR 执行四大维度扫描。

### Step 3: 补充到 spec.md

将扫描结果补充到 spec.md 对应章节。

### Step 4: 更新 findings.md

记录扫描发现的边界/失败/NFR/风险。

---

## 跳过条件

**Trivial 复杂度**：跳过 Expansion Sweep

**理由**：单文件微调，边界和失败场景明确，无需发散扫描。

**记录格式**：
```markdown
## Step 5: Expansion Sweep
SKIPPED — Trivial 路径无需发散扫描
```

---

## 最佳实践

### DO ✅

1. **主动识别** — 不要等用户提出，主动扫描
2. **四维完整** — 边界/失败/NFR/依赖都要扫描
3. **具体可验证** — 边界值要具体，失败场景要可测试
4. **补充到 spec** — 扫描结果写入 spec.md
5. **记录到 findings** — 重要发现记录到 findings.md

### DON'T ❌

1. **不要只扫描边界** — 四维都要扫描
2. **不要模糊描述** — "可能失败"→"超时 3s 后失败"
3. **不要忽略 NFR** — NFR 是质量保证的基础
4. **不要忘记依赖** — 外部依赖是高风险点
5. **不要只读不写** — 扫描结果必须落盘

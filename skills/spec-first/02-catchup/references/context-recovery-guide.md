# Context Recovery Guide

上下文恢复策略与最佳实践。

## 信息源优先级

### P0 - 必读（关键信息）

| 文件 | 包含内容 | 读取目的 | 失败处理 |
|------|----------|----------|----------|
| `.spec-first/current` | 当前 Feature ID | 定位 Feature | 执行 feature list 选择 |
| `stage-state.json` | 阶段、状态、时间戳 | 确定当前阶段 | 执行 init 初始化 |

### P1 - 重要（工作状态）

| 文件 | 包含内容 | 读取目的 | 失败处理 |
|------|----------|----------|----------|
| `task_plan.md` | 任务列表、状态、Owner | 了解任务进度 | 执行 task 拆解任务 |
| `findings.md` | 发现、决策、阻塞 | 恢复工作上下文 | 执行 status 生成状态 |

### P2 - 辅助（追溯信息）

| 文件 | 包含内容 | 读取目的 | 失败处理 |
|------|----------|----------|----------|
| `traceability-matrix.md` | FR/DS/TASK/TC 映射 | 理解追溯关系 | 可选，不阻塞恢复 |
| `spec.md` | 需求规格 | 理解功能需求 | 可选，不阻塞恢复 |
| `design.md` | 技术设计 | 理解设计决策 | 可选，不阻塞恢复 |

### P3 - 可选（历史记录）

| 文件 | 包含内容 | 读取目的 | 失败处理 |
|------|----------|----------|----------|
| Git 历史 | 最近提交、变更 | 了解代码变更 | 可选，不阻塞恢复 |
| `gate-history.jsonl` | Gate 检查历史 | 了解质量状态 | 可选，不阻塞恢复 |

## 恢复步骤详解

### Step 1: 定位 Feature

**目标**: 确定当前工作的 Feature

**操作**:
```bash
# 读取当前 Feature ID
cat .spec-first/current
```

**成功标准**:
- 获得 Feature ID（如 `FSREQ-20260305-AUTH-001`）

**失败处理**:
```bash
# 列出所有 Feature
spec-first feature list

# 选择 Feature
spec-first feature switch <featureId> --yes
```

---

### Step 2: 加载阶段状态

**目标**: 确定当前阶段和状态

**操作**:
```bash
# 读取阶段状态
cat specs/<featureId>/stage-state.json
```

**成功标准**:
- 获得当前阶段（如 `04_implement`）
- 获得阶段状态（如 `in_progress`）
- 获得时间戳

**失败处理**:
```bash
# 重新执行初始化入口
/spec-first:init
```

---

### Step 3: 加载任务进度

**目标**: 了解任务完成情况

**操作**:
```bash
# 读取任务计划
cat specs/<featureId>/task_plan.md
```

**成功标准**:
- 获得任务列表
- 识别 in_progress 任务
- 统计任务状态分布

**失败处理**:
```bash
# 拆解任务
/spec-first:task
```

---

### Step 4: 加载最近发现

**目标**: 恢复工作上下文

**操作**:
```bash
# 读取发现记录
tail -n 50 specs/<featureId>/findings.md
```

**成功标准**:
- 获得最近 3-5 条发现
- 识别最后一个有效结论
- 识别阻塞项

**失败处理**:
```bash
# 生成当前状态
/spec-first:status
```

---

### Step 5: 检查文件完整性

**目标**: 识别缺失文件

**操作**:
```bash
# 检查核心文件
ls -la specs/<featureId>/{spec.md,design.md,task_plan.md,findings.md}
```

**成功标准**:
- 核心文件存在
- 文件更新时间合理（< 7 天）

**失败处理**:
- 缺少 spec.md → 执行 `/spec-first:spec`
- 缺少 design.md → 执行 `/spec-first:design`
- 缺少 task_plan.md → 执行 `/spec-first:task`

---

### Step 6: 识别风险

**目标**: 标记阻塞项和风险

**检查项**:
- [ ] 是否有 blocked 状态任务？
- [ ] 是否有覆盖率不足？
- [ ] 是否有 Gate 失败？
- [ ] 是否有文件过期？

**风险分级**:
- 🔴 高风险: 阻塞阶段推进
- 🟡 中风险: 影响进度或质量
- 🟢 低风险: 轻微问题

---

### Step 7: 生成恢复报告

**目标**: 输出 6 步报告

**报告结构**:
1. Feature 基本信息
2. 任务进度
3. 最近发现
4. 文件完整性检查
5. 风险识别
6. 建议下一步

---

### Step 8: 回答 5-Question

**目标**: 验证恢复质量

**问题清单**:
1. 当前 Feature 与阶段是什么？
2. 当前 in_progress TASK 是什么？
3. 上次中断前最后一个有效结论是什么？
4. 当前最大阻塞是什么？
5. 下一步最小可执行命令是什么？

**验证标准**:
- 所有问题都能回答 → 恢复成功
- 有问题无法回答 → 标记信息缺口

---

## 信息缺口处理

### 缺口类型 1: Feature 未定位

**检测**: `.spec-first/current` 不存在或为空

**影响**: 无法继续恢复

**补齐方案**:
```bash
# 列出所有 Feature
spec-first feature list

# 选择 Feature
spec-first feature switch <featureId> --yes
```

---

### 缺口类型 2: 阶段状态缺失

**检测**: `stage-state.json` 不存在

**影响**: 无法确定当前阶段

**补齐方案**:
```bash
# 重新执行初始化入口
/spec-first:init
```

---

### 缺口类型 3: 任务计划缺失

**检测**: `task_plan.md` 不存在

**影响**: 无法了解任务进度

**补齐方案**:
```bash
# 拆解任务
/spec-first:task
```

---

### 缺口类型 4: 发现记录为空

**检测**: `findings.md` 不存在或为空

**影响**: 无法恢复工作上下文

**补齐方案**:
```bash
# 生成当前状态
/spec-first:status
```

---

### 缺口类型 5: 无 in_progress 任务

**检测**: 所有任务为 planned/complete

**影响**: 不知道下一步做什么

**补齐方案**:
```markdown
从 task_plan.md 选择下一个 planned 任务：

1. 检查依赖关系（depends_on）
2. 选择无依赖或依赖已完成的任务
3. 更新任务状态为 in_progress
4. 开始执行任务
```

---

## 恢复质量评估

### 优秀恢复（90-100 分）

**特征**:
- 所有 5-Question 都能回答
- 无信息缺口
- 下一步明确
- 风险已识别

**行动**:
- ✅ 可以立即继续工作

---

### 良好恢复（70-89 分）

**特征**:
- 4/5 Question 能回答
- 有少量信息缺口
- 下一步基本明确
- 主要风险已识别

**行动**:
- ⚠️ 补齐信息缺口后继续

---

### 中等恢复（50-69 分）

**特征**:
- 3/5 Question 能回答
- 有明显信息缺口
- 下一步不够明确
- 部分风险未识别

**行动**:
- ❌ 必须补齐信息缺口

---

### 较差恢复（0-49 分）

**特征**:
- < 3 Question 能回答
- 严重信息缺口
- 下一步不明确
- 风险未识别

**行动**:
- 🚫 停止工作，重新初始化

---

## 最佳实践

### DO ✅

1. **按优先级读取** — 先读 P0，再读 P1，最后读 P2
2. **标记缺口** — 无法回答的问题必须标记
3. **提供方案** — 每个缺口都给出补齐方案
4. **明确下一步** — 提供可执行命令
5. **写入 findings** — 恢复报告追加到 findings.md

### DON'T ❌

1. **不要跳过 P0** — 必须先定位 Feature 和阶段
2. **不要假设记忆** — 不要假设用户记得上下文
3. **不要忽略缺口** — 信息缺失必须标记
4. **不要模糊建议** — 下一步必须具体可执行
5. **不要只读不写** — 恢复报告必须落盘

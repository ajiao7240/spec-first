# 设计约束规则

> Design Skill 的设计边界与模板驱动约束

---

## 核心原则

**系统级 HOW vs 实现级 HOW**

design 阶段输出系统级设计，不输出实现细节。

---

## 必须写的内容

### 1. 模块边界

**定义**：系统的逻辑分层与模块划分

**示例**：
```
- auth-service: 认证服务
- user-service: 用户管理
- notification-service: 通知服务
```

---

### 2. 接口契约

**定义**：模块间的 API 定义

**示例**：
```
POST /api/auth/login
Request: { phone, code }
Response: { token, userId, expiresAt }
```

---

### 3. 数据模型

**定义**：核心实体与关系

**示例**：
```
users (id, phone, created_at)
sessions (id, user_id, token, expires_at)
```

---

### 4. 一致性策略

**定义**：分布式场景下的数据一致性保证

**示例**：
```
- 最终一致性：用户注册后异步发送欢迎邮件
- 强一致性：支付扣款与订单创建在同一事务
```

---

### 5. 回滚策略

**定义**：失败场景的恢复机制

**示例**：
```
- 支付失败：自动退款 + 订单状态回滚
- 库存扣减失败：释放预占库存
```

---

## 禁止写的内容

### 1. 类/函数级实现

**错误示例**：
```typescript
class AuthService {
  async login(phone: string, code: string) {
    // 实现细节
  }
}
```

**正确做法**：只描述模块职责，不写具体代码

---

### 2. 具体代码片段

**错误示例**：
```python
def send_sms(phone, code):
    client = TwilioClient(api_key)
    client.send(phone, code)
```

**正确做法**：描述接口契约，不写实现

---

### 3. 与需求无关的库细节

**错误示例**：
```
使用 Redis 作为缓存，配置如下：
- maxmemory: 2GB
- eviction-policy: allkeys-lru
```

**正确做法**：只说明"使用缓存提升性能"，具体配置留给实现阶段

## 自我修正上限

**规则**：`{{MAX_SELF_CORRECTION}}` 轮（默认 3）

**触发条件**：
- 设计方案被用户拒绝
- gate check 失败
- 覆盖率检查不通过

**超限处理**：
- 记录到 `findings.md`
- 标记为 `[NEEDS CLARIFICATION]`
- 等待用户明确指导

---

## 澄清标记规则

**格式**：`[NEEDS CLARIFICATION][TYPE]`

**类型**：
- `[ARCH]` - 架构决策不明确
- `[API]` - 接口定义有歧义
- `[DATA]` - 数据模型不清晰
- `[PERF]` - 性能要求未明确
- `[SEC]` - 安全策略待确认

**限制**：每轮最多 3 项

**示例**：
```markdown
## 待澄清问题

1. [NEEDS CLARIFICATION][ARCH] 用户服务与订单服务是否需要拆分为独立微服务？
2. [NEEDS CLARIFICATION][PERF] 列表查询的分页大小上限是多少？
3. [NEEDS CLARIFICATION][SEC] 是否需要支持多因素认证（MFA）？
```

## 设计深度指南

| 层级 | 设计阶段 | 实现阶段 |
|------|----------|----------|
| 系统架构 | ✅ 必须 | - |
| 模块划分 | ✅ 必须 | - |
| 接口契约 | ✅ 必须 | - |
| 数据模型 | ✅ 必须 | - |
| 类设计 | ❌ 禁止 | ✅ 实现时 |
| 函数签名 | ❌ 禁止 | ✅ 实现时 |
| 代码实现 | ❌ 禁止 | ✅ 实现时 |

## 字面即精神原则

**核心理念**：违反字面规则就是违反精神规则

### 反合理化表

| AI 的借口 | 封堵 |
|-----------|------|
| "我理解核心思想，可以灵活执行" | 字面规则的违反就是精神的违反，不存在灵活变通 |
| "这是精神而非仪式" | 仪式（字面规则）是精神的体现，跳过仪式就是违背精神 |
| "实质重于形式" | 在流程守卫上，形式（字面规则）= 实质（精神） |
| "具体情况具体分析" | 规则已考虑常见情况，例外需明确讨论而非自行变通 |

## Plan Mode 协同

**适用场景**：
- 架构分层决策
- 接口边界划分
- 数据一致性策略

**协同流程**：
1. 在 Plan Mode 中先收敛设计方案
2. 关键结论同步到 `findings.md`
3. 在 `design.md` 中保留可追溯引用

**示例**：
```markdown
## 架构决策

采用微服务架构，拆分为 3 个服务：
- auth-service
- user-service
- order-service

详见 findings.md § 2026-03-05 架构讨论
```

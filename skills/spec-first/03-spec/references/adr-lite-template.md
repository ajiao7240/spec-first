# ADR-lite 模板

Architecture Decision Record（轻量级）模板，用于 Step 7 记录技术决策。

---

## 使用场景

**触发条件**：
- 复杂度为 Complex
- 存在多个技术方案需要权衡

**跳过条件**：
- 复杂度为 Trivial/Simple/Moderate
- 技术方案唯一且明确

---

## 示例 1: 缓存方案选择

```markdown
## ADR-AUTH-001: 验证码缓存方案

**状态**: Accepted

**日期**: 2026-03-05

**上下文**:
短信验证码需要临时存储（有效期 120s），需要选择合适的缓存方案。

**候选方案**:

### 方案 A: Redis
- **描述**: 使用 Redis 作为缓存存储
- **优势**:
  - 支持 TTL 自动过期
  - 性能高（P99 < 5ms）
  - 支持集群部署
- **劣势**:
  - 需要额外部署 Redis
  - 增加运维成本
- **成本**: 部署成本 2 天，运维成本 0.5 天/月
- **风险**: Redis 故障会导致验证码功能不可用

### 方案 B: 内存缓存（Map）
- **描述**: 使用应用内存 Map 存储
- **优势**:
  - 无需额外部署
  - 实现简单
- **劣势**:
  - 不支持分布式（多实例数据不共享）
  - 应用重启数据丢失
  - 需要手动实现 TTL 清理
- **成本**: 实现成本 0.5 天
- **风险**: 多实例场景下验证码不一致

**决策**: 选择方案 A（Redis）

**理由**:
1. 系统已部署 Redis，无额外部署成本
2. 支持分布式，满足多实例部署需求
3. TTL 自动过期，无需手动清理

**后果**:
- **正面影响**: 支持分布式部署，性能稳定
- **负面影响**: 依赖 Redis 可用性
- **缓解措施**: Redis 集群部署 + 降级策略（Redis 不可用时使用内存缓存）

**相关人员**:
- 提议人: 张三
- 决策人: 李四（Tech Lead）
- 评审人: 王五、赵六

**参考资料**:
- Redis 官方文档: https://redis.io/docs/
- 内部 Redis 部署规范: docs/redis-deployment.md
```

## 示例 2: 权限模型选择

```markdown
## ADR-USER-001: 权限模型选择

**状态**: Proposed

**日期**: 2026-03-05

**上下文**:
系统需要实现细粒度权限控制，支持角色、资源、操作三维度权限管理。

**候选方案**:

### 方案 A: RBAC（Role-Based Access Control）
- **描述**: 基于角色的访问控制
- **优势**:
  - 实现简单，易于理解
  - 适合层级化组织结构
  - 业界成熟方案
- **劣势**:
  - 角色爆炸问题（角色数量过多）
  - 不支持动态权限
  - 难以处理特殊场景
- **成本**: 实现成本 5 天
- **风险**: 角色数量增长后维护困难

### 方案 B: ABAC（Attribute-Based Access Control）
- **描述**: 基于属性的访问控制
- **优势**:
  - 支持动态权限
  - 灵活性高
  - 可处理复杂场景
- **劣势**:
  - 实现复杂
  - 性能开销大
  - 学习成本高
- **成本**: 实现成本 15 天
- **风险**: 规则复杂度高，难以维护

### 方案 C: RBAC + 资源级权限
- **描述**: RBAC 基础上增加资源级权限控制
- **优势**:
  - 兼顾简单性和灵活性
  - 支持大部分场景
  - 渐进式扩展
- **劣势**:
  - 需要设计资源权限模型
  - 比纯 RBAC 复杂
- **成本**: 实现成本 8 天
- **风险**: 资源权限模型设计不当会影响性能

**决策**: 选择方案 C（RBAC + 资源级权限）

**理由**:
1. 当前需求大部分场景 RBAC 可满足
2. 资源级权限可覆盖特殊场景（如"只能编辑自己创建的文档"）
3. 实现成本适中，可渐进式扩展

**后果**:
- **正面影响**: 满足当前和未来需求，扩展性好
- **负面影响**: 比纯 RBAC 复杂，需要设计资源权限模型
- **缓解措施**: 先实现 RBAC，再逐步增加资源级权限

**相关人员**:
- 提议人: 张三
- 决策人: 待定
- 评审人: 待定

**参考资料**:
- NIST RBAC 标准: https://csrc.nist.gov/projects/role-based-access-control
- ABAC 指南: https://www.nist.gov/publications/guide-attribute-based-access-control-abac-definition-and-considerations
```

## 最佳实践

### DO ✅

1. **至少 2 个方案** — 必须对比至少 2 个方案
2. **量化对比** — 优劣势要具体，避免模糊描述
3. **明确理由** — 决策理由要充分，有说服力
4. **评估后果** — 正面和负面影响都要评估
5. **记录参与人** — 明确提议人、决策人、评审人

### DON'T ❌

1. **不要只有 1 个方案** — 至少对比 2 个方案
2. **不要模糊描述** — "性能好"→"P99 < 5ms"
3. **不要忽略后果** — 必须评估正面和负面影响
4. **不要跳过评审** — Complex 决策需要评审
5. **不要忘记更新** — 决策变更时更新状态为 Deprecated

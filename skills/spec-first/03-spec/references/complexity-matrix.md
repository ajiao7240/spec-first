# 复杂度判定矩阵

本文档详细说明复杂度判定标准和执行路径。

---

## 四档判定标准

**多维取最高档原则**：任一维度达到某档位，则整体判定为该档位。

| 档位 | 受影响文件数 | 歧义点数量 | 方案分支数 | 外部依赖 | 典型特征 |
|------|------------|----------|----------|---------|---------|
| Trivial | ≤1 | 0 | 1 | 无 | 单文件微调、文案修改、配置变更 |
| Simple | 2-3 | 1-2 | 1 | 无或已明确 | 单模块功能、清晰边界、无架构影响 |
| Moderate | 4-8 | 3-5 | 2 | 1-2 个 | 跨模块协作、存在技术选型、需轻量调研 |
| Complex | ≥9 | ≥6 | ≥3 | ≥3 个 | 架构级变更、多方案权衡、需深度调研 |

---

## 判定流程

1. **统计受影响文件数**（Phase 0.2 已收集）
2. **识别歧义点数量**（扫描 PRD 中的未定义术语/多解语义/缺失边界）
3. **检测方案分支数**（关键词：或者/可选/待定/考虑）
4. **统计外部依赖数量**（Phase 0.2 已收集）
5. **按表格规则判定档位**，边界情况向上取整

---

## 四档对应执行路径

| 复杂度 | 执行路径 | 说明 |
|--------|---------|------|
| Trivial | Phase 0 + Step 0 + Step 2 + Step 8 | 跳过 Step 3-7 |
| Simple | Phase 0 + Step 0 + Step 2-3 + Step 6 + Step 8 | 跳过 Step 4-5, 7 |
| Moderate | Phase 0 + Step 0 + Step 2-6 + Step 8 | 跳过 Step 7 |
| Complex | Phase 0 + Step 0 + Step 2-8 | 全量执行 |

**注**: Step 1 已合并到 Phase 0.2，所有路径都不再单独执行 Step 1。

---

## 判定示例

### 示例 1: Trivial

**需求**: 修改登录页面的按钮文案

**判定依据**:
- 受影响文件数: 1（login.tsx）
- 歧义点数量: 0
- 方案分支数: 1
- 外部依赖: 无

**结论**: Trivial

**执行路径**: Phase 0 + Step 0 + Step 2 + Step 8

---

### 示例 2: Simple

**需求**: 添加用户头像上传功能

**判定依据**:
- 受影响文件数: 3（UserProfile.tsx, api.ts, types.ts）
- 歧义点数量: 1（图片大小限制未明确）
- 方案分支数: 1
- 外部依赖: 无（使用现有 OSS）

**结论**: Simple

**执行路径**: Phase 0 + Step 0 + Step 2-3 + Step 6 + Step 8

---

### 示例 3: Moderate

**需求**: 实现订单支付流程

**判定依据**:
- 受影响文件数: 6（Order, Payment, Notification, API, Types, DB）
- 歧义点数量: 4（支付方式/超时处理/失败重试/通知机制）
- 方案分支数: 2（同步支付 vs 异步支付）
- 外部依赖: 2（支付网关、短信服务）

**结论**: Moderate（多维取最高档）

**执行路径**: Phase 0 + Step 0 + Step 2-6 + Step 8

---

### 示例 4: Complex

**需求**: 重构整个权限系统为 RBAC

**判定依据**:
- 受影响文件数: 15+（所有模块）
- 歧义点数量: 8（角色定义/权限粒度/继承规则/动态权限/审计日志/迁移策略/兼容性/性能影响）
- 方案分支数: 3（RBAC vs ABAC vs 混合模式）
- 外部依赖: 4（认证服务、审计服务、缓存、数据库）

**结论**: Complex

**执行路径**: Phase 0 + Step 0 + Step 2-8（全量）

---

## 边界情况处理

**原则**: 边界情况向上取整

**示例**:
- 受影响文件数 = 3，其他维度都是 Trivial → 判定为 Simple
- 歧义点数量 = 5，其他维度都是 Simple → 判定为 Moderate
- 任一维度达到 Complex → 整体判定为 Complex

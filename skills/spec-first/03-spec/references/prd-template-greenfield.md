## 元信息

```yaml
---
scenario: greenfield
scenario_reason: "无现有代码/文档，全新功能"
evidence_paths: []
complexity: "待判定"
created_at: "YYYY-MM-DD HH:mm:ss"
last_updated: "YYYY-MM-DD HH:mm:ss"
---
```

**字段说明**：
- `scenario`: 场景类型（greenfield/iteration）
- `scenario_reason`: 场景判定理由
- `evidence_paths`: 证据文件路径列表（greenfield 为空）
- `complexity`: 复杂度档位（Trivial/Simple/Moderate/Complex，初始为"待判定"）
- `created_at`: 创建时间
- `last_updated`: 最后更新时间

## 示例：短信验证码登录（Greenfield）

```yaml
---
scenario: greenfield
scenario_reason: "无现有登录功能，全新实现"
evidence_paths: []
complexity: "待判定"
created_at: "2026-03-05 10:00:00"
last_updated: "2026-03-05 10:00:00"
---
```

### 1. 业务目标

#### 1.1 问题陈述

**当前痛点**：
用户需要记忆密码才能登录，密码遗忘后找回流程复杂，导致用户流失。

**目标用户**：
所有注册用户，特别是移动端用户。

**使用场景**：
用户在移动端打开 App，希望快速登录查看内容。

#### 1.2 业务价值

**预期收益**：
- 降低登录门槛，提升用户留存率
- 减少密码找回工单，降低客服成本

**成功指标**：
- 短信验证码登录占比 >= 30%
- 登录成功率 >= 95%
- 平均登录时长 < 30s

---

### 2. 用户故事

#### 2.1 核心用户故事

**作为** 普通用户
**我想要** 通过手机号和验证码快速登录
**以便于** 无需记忆密码

#### 2.2 用户旅程

```
[打开 App] → [输入手机号] → [点击获取验证码] → [收到短信] → [输入验证码] → [登录成功]
```

**详细步骤**：
1. 用户打开 App，进入登录页面
2. 用户输入手机号，点击"获取验证码"
3. 系统发送 6 位数字验证码到用户手机
4. 用户收到短信，输入验证码
5. 系统验证通过，返回 JWT token，登录成功

---

### 3. 功能边界

#### 3.1 范围内（In Scope）

**核心功能**：
- 发送短信验证码
- 验证码登录

**必需特性**：
- 手机号格式校验
- 验证码有效期（120s）
- 重试次数限制（3 次/小时）
- 防暴力破解

#### 3.2 范围外（Out of Scope）

**本期不做**：
- 邮箱验证码登录
- 第三方登录（微信、支付宝）
- 国际手机号支持

**理由**：
- 优先验证短信验证码登录的可行性
- 第三方登录需要额外对接，工期较长

---

### 4. 约束条件

#### 4.1 技术约束

- 使用阿里云短信服务
- 验证码存储在 Redis（有效期 120s）
- 登录成功后返回 JWT token

#### 4.2 业务约束

- 短信成本控制在 0.05 元/条
- 单用户每日最多发送 10 次验证码

#### 4.3 时间约束

- 预计上线时间: 2026-03-10
- 关键里程碑: 2026-03-08 完成开发

---

### 5. 成功标准

#### 5.1 功能完整性

- [ ] 发送验证码功能已实现
- [ ] 验证码登录功能已实现
- [ ] 用户旅程可完整走通

#### 5.2 质量标准

- [ ] 单元测试覆盖率 >= 80%
- [ ] 集成测试通过
- [ ] 性能测试达标（P95 < 500ms）
- [ ] 安全扫描通过

#### 5.3 业务指标

- [ ] 短信验证码登录占比 >= 30%
- [ ] 登录成功率 >= 95%

---

#### 6.1 待澄清问题

|------|--------|--------|------|
| 验证码有效期是 60s 还是 120s？ | High | 产品 | Resolved（120s） |

|------|------|------|----------|

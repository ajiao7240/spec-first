# 收敛问答规则

Step 6 Q&A Loop（收敛确认）的一问一答策略。

---

## 核心原则

**逐项确认 FR/AC，一问一答，避免遗漏。**

---

## 一问一答规则

### 规则 1: 每次只确认 1 个 FR

**错误示例** ❌：
```markdown
请确认以下 3 个 FR：
- FR-AUTH-001: 短信验证码登录
- FR-AUTH-002: 邮箱密码登录
- FR-AUTH-003: 第三方登录
```

**正确示例** ✅：
```markdown
请确认 FR-AUTH-001: 短信验证码登录

描述：用户输入手机号和验证码，系统验证后返回 JWT token。

是否确认？
```

---

### 规则 2: FR 确认后再确认 AC

**流程**：
1. 先确认 FR 描述
2. FR 确认后，逐条确认 AC
3. 所有 AC 确认后，进入下一个 FR

**示例**：
```markdown
[第 1 轮] 请确认 FR-AUTH-001: 短信验证码登录
→ 用户确认

[第 2 轮] 请确认 AC-AUTH-001-01: 手机号格式校验
→ 用户确认

[第 3 轮] 请确认 AC-AUTH-001-02: 验证码有效期校验
→ 用户确认

[第 4 轮] 请确认 AC-AUTH-001-03: 重试次数限制
→ 用户确认

[第 5 轮] FR-AUTH-001 所有 AC 已确认，进入 FR-AUTH-002
```

---

### 规则 3: 允许用户修订

用户可以修订 FR/AC，修订后重新确认。

**示例**：
```markdown
[AI] 请确认 AC-AUTH-001-01: 手机号必须是 11 位数字

[用户] 不对，应该支持国际号码

[AI] 已修订为：AC-AUTH-001-01: 手机号支持国内 11 位和国际号码（+86/+1 等）

请重新确认？

[用户] 确认
```

---

## 确认格式

### FR 确认格式

```markdown
## FR-<ABBR>-<SEQ>: <标题>

**描述**：<功能描述>

**优先级**：P0/P1/P2

**验收标准**：
- AC-<ABBR>-<SEQ>-01: <验收标准 1>
- AC-<ABBR>-<SEQ>-02: <验收标准 2>

**是否确认？**
```

---

### AC 确认格式

```markdown
## AC-<ABBR>-<SEQ>-<NN>: <标题>

**Given**：<前置条件>
**When**：<操作>
**Then**：<预期结果>

**是否确认？**
```

## 生成 FR/AC 的依据

### 依据来源（优先级）

1. **PRD**（Phase 0 产出）
2. **Step 1 Auto-Context**（上下文收集）
3. **Step 3 Question Gate**（用户回答）
4. **Step 5 Expansion Sweep**（发散扫描）

### 生成流程

1. 从 PRD 提取核心功能 → 生成 FR
2. 从 Step 3 回答提取边界 → 生成 AC
3. 从 Step 5 扫描提取失败场景 → 生成 AC
4. 从 Step 5 扫描提取 NFR → 生成 NFR-*

---

## 确认后写入

### 写入 spec.md

用户确认后，立即写入 `spec.md`。

**格式**：
```markdown
# 功能需求

## FR-AUTH-001: 短信验证码登录

**描述**：用户输入手机号和验证码，系统验证后返回 JWT token。

**优先级**：P0

**验收标准**：
- AC-AUTH-001-01: 手机号格式校验（11 位数字）
- AC-AUTH-001-02: 验证码有效期校验（120s）
- AC-AUTH-001-03: 重试次数限制（3 次/小时）

**确认时间**：2026-03-05T10:00:00Z
```

### 注册到追踪矩阵

确认后，立即注册到 `traceability-matrix.md`。

**命令**：
```bash
spec-first id next FR AUTH --feature <featureId>
spec-first matrix update <featureId> FR-AUTH-001 --title "短信验证码登录" --status Planned
```

## 最佳实践

### DO ✅

1. **一问一答** — 每次只确认 1 个 FR
2. **FR 先 AC 后** — FR 确认后再确认 AC
3. **允许修订** — 用户可以修订，修订后重新确认
4. **立即写入** — 确认后立即写入 spec.md
5. **立即注册** — 确认后立即注册到追踪矩阵

### DON'T ❌

1. **不要批量确认** — 不要一次确认多个 FR
2. **不要跳过 AC** — FR 确认后必须确认 AC
3. **不要拒绝修订** — 用户修订是正常流程
4. **不要延迟写入** — 确认后立即写入
5. **不要忘记注册** — 确认后必须注册到矩阵

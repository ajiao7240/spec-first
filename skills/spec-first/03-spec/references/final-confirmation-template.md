## 示例：短信验证码登录

```markdown
# 最终确认包

## 1. Goal（业务目标）

**Feature ID**: FSREQ-20260305-AUTH-001

**Feature 标题**: 短信验证码登录

**业务目标**:
为用户提供便捷的短信验证码登录方式，降低登录门槛，提升用户体验。

**成功指标**:
- 短信验证码登录占比 >= 30%
- 登录成功率 >= 95%
- 平均登录时长 < 30s

---

## 2. Requirements（功能需求）

### FR-AUTH-001: 发送短信验证码

**描述**: 用户输入手机号，系统发送 6 位数字验证码到用户手机。

**优先级**: P0

**验收标准**:
- AC-01: 手机号格式校验（11 位数字）
- AC-02: 验证码 6 位随机数字
- AC-03: 验证码有效期 120s
- AC-04: 同一手机号 60s 内只能发送 1 次

### FR-AUTH-002: 验证码登录

**描述**: 用户输入手机号和验证码，系统验证后返回 JWT token。

**优先级**: P0

**验收标准**:
- AC-01: 验证码正确且未过期时登录成功
- AC-02: 验证码错误时返回错误提示，剩余重试次数 -1
- AC-03: 验证码过期时返回错误提示
- AC-04: 重试次数耗尽（3 次）时锁定 1 小时

---

## 3. Non-Functional Requirements（非功能需求）

### NFR-AUTH-001: 性能要求

**类型**: Performance

**要求**:
- 发送验证码响应时间 P95 < 500ms
- 验证码登录响应时间 P95 < 300ms
- 支持 1000 QPS

**验证方式**: 压力测试

### NFR-AUTH-002: 安全要求

**类型**: Security

**要求**:
- 验证码加密存储（AES-256）
- 传输使用 HTTPS
- 防止暴力破解（重试次数限制 + 锁定机制）

**验证方式**: 安全扫描 + 渗透测试

---

## 4. Acceptance Criteria（验收标准汇总）

| AC ID | 描述 | 关联 FR | 优先级 |
|-------|------|---------|--------|
| FR-AUTH-001 / AC-01 | 手机号格式校验（11 位数字） | FR-AUTH-001 | P0 |
| FR-AUTH-001 / AC-02 | 验证码 6 位随机数字 | FR-AUTH-001 | P0 |
| FR-AUTH-001 / AC-03 | 验证码有效期 120s | FR-AUTH-001 | P0 |
| FR-AUTH-001 / AC-04 | 同一手机号 60s 内只能发送 1 次 | FR-AUTH-001 | P0 |
| FR-AUTH-002 / AC-01 | 验证码正确且未过期时登录成功 | FR-AUTH-002 | P0 |
| FR-AUTH-002 / AC-02 | 验证码错误时返回错误提示 | FR-AUTH-002 | P0 |
| FR-AUTH-002 / AC-03 | 验证码过期时返回错误提示 | FR-AUTH-002 | P0 |
| FR-AUTH-002 / AC-04 | 重试次数耗尽时锁定 1 小时 | FR-AUTH-002 | P0 |

---

## 5. Definition of Done（完成标准）

- [ ] 所有 FR 已实现（FR-AUTH-001, FR-AUTH-002）
- [ ] 所有 AC 已验证（8 条 AC）
- [ ] Gate check 通过（C1-C9 覆盖率达标）
- [ ] 代码审查通过
- [ ] 单元测试覆盖率 >= 80%
- [ ] 集成测试通过
- [ ] 性能测试通过（P95 < 500ms/300ms）
- [ ] 安全扫描通过
- [ ] 文档已更新（API 文档、用户手册）

---

## 6. Out of Scope（明确不做）

**本期不做**:
- 邮箱验证码登录
- 第三方登录（微信、支付宝）
- 国际手机号支持

**理由**:
- 优先验证短信验证码登录的可行性
- 第三方登录需要额外对接，工期较长
- 国际手机号需求不明确

**后续计划**:
- V2.0 增加邮箱验证码登录
- V3.0 增加第三方登录

---

## 7. Dependencies（依赖项）

**外部依赖**:
- 阿里云短信服务: 发送短信验证码
- Redis: 缓存验证码

**内部依赖**:
- 用户服务: 查询/创建用户
- JWT 服务: 生成 token

---

## 8. Risks（风险识别）

| 风险 | 等级 | 影响 | 缓解措施 |
|------|------|------|----------|
| 短信服务商故障 | High | 用户无法登录 | 接入备用短信服务商（腾讯云） |
| Redis 故障 | Medium | 验证码功能不可用 | Redis 集群部署 + 降级策略 |
| 验证码被暴力破解 | Medium | 账号安全风险 | 重试次数限制 + 锁定机制 |

---

## 9. Implementation Plan（实施计划）

**下一步**: 执行 `/spec-first:design`

**预计工期**: 5 天

**里程碑**:
1. 技术设计完成: 2026-03-06
2. 代码实现完成: 2026-03-08
3. 测试验收完成: 2026-03-10

---

## 10. Approval（确认签字）

**确认人**: 张三（产品经理）

**确认时间**: 2026-03-05 10:00:00

**确认意见**: 同意

**修改意见**: 无
```

## 最佳实践

### DO ✅

1. **完整性** — 10 个章节都要填写
2. **具体化** — 避免模糊描述，要具体可验证
3. **优先级** — 明确 P0/P1/P2 优先级
4. **风险识别** — 主动识别风险并提供缓解措施
5. **用户确认** — 必须获得用户确认签字

### DON'T ❌

1. **不要跳过章节** — 10 个章节都要填写
2. **不要模糊描述** — "性能好"→"P95 < 500ms"
3. **不要忽略 OOS** — 明确不做的内容要列出
4. **不要忽略风险** — 风险识别是必须的
5. **不要跳过确认** — 必须获得用户确认

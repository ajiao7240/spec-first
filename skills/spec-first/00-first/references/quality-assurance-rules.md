# 统一 QA 规则（00-first）

> 本文档定义 00-first 全部分析主题与 runtime 资产共用的质量约束。
> 当前正式 contract 为单一标准模式：所有正式产物均执行同一证据规则。

---

## 0. 输出语言（强制）

- **默认输出语言：中文**
- 所有生成的 markdown 文档必须使用中文撰写
- **技术术语保留英文原文**（如 TypeScript、Vitest、Feature、Gate、Stage）
- **代码标识符保留英文原文**（如函数名、变量名、文件路径）
- **专有名词保留英文**（如 npm、pnpm、ESM、CommonJS）
- **混合写法示例**：
  - ✅ "Vitest 测试框架配置"
  - ✅ "Feature 生命周期管理"
  - ✅ "Gate 门禁校验"
  - ❌ "测试框架配置"（缺少具体名称）
  - ❌ "功能生命周期管理"（Feature 不应翻译）
- **强门禁**：
  - 如果 projection 输出与中文 contract 不一致，优先修 renderer 与测试
  - 不接受“规范要求中文，但实现临时英文”的漂移状态

---

## 1. 证据标注（强制）

所有技术结论必须附带证据源，唯一合法格式：

```markdown
- <结论> (`<file_path>:<line>` — `<关键代码片段>` — `[证据类型]`)
```

**证据类型分类**：

| 类型 | 说明 | 示例 |
|------|------|------|
| `[显式]` | 代码/注释/文档明确声明 | 配置文件中的值、注释说明、类型定义 |
| `[推断]` | 从代码行为分析得出 | 从 if 条件推断业务规则、从方法调用推断依赖 |
| `[待确认]` | 无法确定或需要用户确认 | 缺少文档的复杂逻辑、隐式约定 |

默认规则：未标注证据类型时，默认为 `[显式]`。  
违规判定：无证据标注且未标记 `[待确认]` 的裸结论，视为不合格产出。

---

## 2. 抽样验证（完稿前执行）

统一流程：

1. 从已写结论中随机抽取样本（样本量见下表）
2. 打开对应源文件，核对结论与证据是否一致
3. 按结果处理
   - ✅ 准确：保持不变
   - ⚠️ 有偏差：修正文档内容和证据标注
   - ❌ 无法验证：标记 `[待确认]`

验证过程不写入最终文档，但发现错误必须先修正再输出。

## 3. 分析主题 / runtime 资产最低要求矩阵

| 分析主题 / 资产 | 必须标注证据的内容 | 抽样规模 |
|----------------|--------------------|----------|
| 代码结构 / `structure-overview.json` | 模块职责说明、入口文件识别、构建命令来源 | 2-3 条 |
| 架构关系 / `structure-overview.json` | 模块依赖边、层次边界、关键协作关系来源 | 2-3 条 |
| 调用链 / `critical-flows.json` | 关键入口、主调用关系、扩散风险来源 | 2-3 条 |
| API 契约 / `api-contracts.json` | 每个正式接口的 route / command 注册位置 | 2-3 条 |
| 外部依赖 / `summary.json` | 第三方服务、外部边界、依赖声明或配置引用位置 | 2-3 条 |
| 规范与环境 / `conventions.json` / `entry-guide.json` / `summary.json` | 规则来源、运行时/命令/环境变量来源 | 2-3 条 |
| 数据库 / `database-schema.json` | DB 类型识别、表结构、外键关系、字段注释来源 | 2-3 张表 |
| 领域模型 / `domain-model.json` | 概念、关系、状态转换、业务规则、验证规则来源 | 3-5 条 |

## 4. 数据库特例（纯 SQL 项目）

当无可引用模型文件时，允许使用以下替代证据格式：

```markdown
- <结论> (`<db_type>:<query_type>` — `<查询结果摘要>`)
```

示例：

```markdown
- users 表有 email 字段 (`mysql:DESCRIBE users` — `email varchar(255) NOT NULL`)
```

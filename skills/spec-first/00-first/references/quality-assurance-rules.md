# 统一 QA 规则（00-first）

> 本文档定义 00-first 全部 Agent 共用的质量约束。  
> A1/A2/A3/B/C1/C2/D/A4 必须统一遵循，不得各自定义冲突版本。
> **注意**：quick 模式不强制要求证据标注，仅 deep 模式严格执行。

---

## 1. 证据标注（强制）

所有技术结论必须附带证据源，唯一合法格式：

```markdown
- <结论> (`<file_path>:<line>` — `<关键代码片段>` — `[证据类型]`)
```

**证据类型分类**：

| 类型 | 说明 | 示例 |
|------|------|------|
| `[显式]` | 代码/注释/文档明确声明 | 配置文件中的值、注释说明、类型定义 |
| `[推断]` | 从代码行为分析得出 | 从 if 条件推断业务规则、从方法调用推断依赖 |
| `[待确认]` | 无法确定或需要用户确认 | 缺少文档的复杂逻辑、隐式约定 |

默认规则：未标注证据类型时，默认为 `[显式]`。  
违规判定：无证据标注且未标记 `[待确认]` 的裸结论，视为不合格产出。

---

## 2. 抽样验证（完稿前执行）

统一流程：

1. 从已写结论中随机抽取样本（样本量见下表）
2. 打开对应源文件，核对结论与证据是否一致
3. 按结果处理
   - ✅ 准确：保持不变
   - ⚠️ 有偏差：修正文档内容和证据标注
   - ❌ 无法验证：标记 `[待确认]`

验证过程不写入最终文档，但发现错误必须先修正再输出。

---

## 3. Agent 最低要求矩阵

| Agent | 必须标注证据的内容 | 抽样规模 |
|-------|--------------------|----------|
| A1 | 模块职责说明、入口文件识别、构建命令来源 | 2-3 条 |
| A2 | Mermaid 图中每条模块依赖边（import 来源） | 2-3 条 |
| A3 | 依赖矩阵中每个调用关系（import/require 位置） | 2-3 条 |
| B | 每个 API 端点的路由定义位置 | 2-3 条 |
| C1 | 每个外部服务的依赖声明或配置引用位置 | 2-3 条 |
| C2 | 规范条目的配置来源、运行时/命令/环境变量来源 | 2-3 条 |
| D | DB 类型识别、表结构、外键关系、字段注释来源 | 2-3 张表 |
| A4 | 概念、关系、状态转换、业务规则、验证规则来源 | 3-5 条 |

---

## 4. Agent D 特例（纯 SQL 项目）

当无可引用模型文件时，允许使用以下替代证据格式：

```markdown
- <结论> (`<db_type>:<query_type>` — `<查询结果摘要>`)
```

示例：

```markdown
- users 表有 email 字段 (`mysql:DESCRIBE users` — `email varchar(255) NOT NULL`)
```

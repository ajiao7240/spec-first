# Task Review Checklist

任务拆解质量检查清单，用于在输出 `task_plan.md` 前自检。

## A. 任务完整性（必查）

- [ ] 所有 FR 都有对应的 TASK 映射
- [ ] 所有 DS 都有对应的 TASK 映射
- [ ] 任务覆盖了设计中的所有接口和数据模型
- [ ] 边界情况和异常处理有对应任务

## B. 任务粒度（必查）

- [ ] 单个任务可在 2-4 小时内完成
- [ ] 单个任务包含 3-7 个具体步骤
- [ ] 每个步骤可在 5-30 分钟内完成
- [ ] 无"调研"、"分析"等模糊动作（需拆为具体步骤）

## C. 依赖关系（必查）

- [ ] 所有依赖使用 TASK ID 引用（禁止自然语言）
- [ ] 依赖关系无循环（A→B→A）
- [ ] 关键路径任务已识别
- [ ] 可并行任务已标记 `[P]`

## D. 验收标准（必查）

- [ ] 每个任务有明确的验收标准
- [ ] 验收标准可测试/可验证
- [ ] 验收标准对应 FR/DS 要求
- [ ] 边界情况有验收条件

## E. 责任分工（必查）

- [ ] 每个 TASK 有唯一的 Owner
- [ ] Owner 角色明确（FE/BE/DEV/QA/DEVOPS）
- [ ] 跨角色协作任务有明确交接点

## F. 用户故事映射（必查）

- [ ] 每个 TASK 标记所属用户故事 `[US#]`
- [ ] 用户故事可独立交付
- [ ] 用户故事有端到端验收条件

## G. 输出格式（必查）

- [ ] 使用统一的 task_plan.md 模板
- [ ] 任务明细表格式一致
- [ ] TASK ID 命名规范一致
- [ ] traceability-matrix.md 已同步更新

## H. 常见陷阱（自检）

- [ ] 无"TODO"、"待补充"等占位符
- [ ] 无"实现XXX功能"这种任务（需拆为具体步骤）
- [ ] 无"优化"、"改进"等模糊动作（需定义具体目标）
- [ ] 无超过 1 天工期的任务（需进一步拆解）
- [ ] 无"协调"、"沟通"等非执行性任务（需转化为具体输出）

## I. 可执行性验证（按需）

### I1. 陌生开发者测试

假设一个完全不熟悉项目的开发者拿到任务计划：
- [ ] 能理解每个任务要做什么
- [ ] 知道从哪里开始
- [ ] 知道需要哪些文件/资源
- [ ] 知道什么时候算完成

### I2. 中断恢复测试

假设任务进行到一半中断：
- [ ] 能从任务计划恢复上下文
- [ ] 知道当前进度
- [ ] 知道下一步做什么
- [ ] 不会丢失已完成工作

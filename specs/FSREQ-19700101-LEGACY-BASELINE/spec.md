# Spec — FSREQ-19700101-LEGACY-BASELINE

> 存量系统可分析基线

**Feature ID**: `FSREQ-19700101-LEGACY-BASELINE`

## 1. 业务目标

### 1.1 背景

当前 spec-first 仓库已经具备完整的 CLI、Core、Skill、Template 与宿主集成能力，但这些能力分布在代码、运行态资产和技能目录中。若没有一份统一基线，后续设计与计划阶段会继续依赖隐含假设，难以保持一致性。

### 1.2 目标

本 Feature 的目标是把当前系统的事实整理成一份可审查、可追溯、可继续传递到下一阶段的基线规格，作为后续 `spec-review`、`design` 和 `task` 的统一输入。

### 1.3 业务价值

- 统一当前系统认知，减少团队与 AI 对现状的不同理解
- 让后续设计建立在事实证据上，而不是经验猜测上
- 提前暴露能力缺口、风险和技术债务，降低返工成本
- 宿主安装态、宿主命令注册、同步结果和外部集成边界被纳入同一份基线后，后续阶段能在单一事实底座上继续推进

### 1.4 成功指标

- 当前系统的核心模块、关键流程、宿主边界和运行态资产均被覆盖
- 关键缺口和风险均被分级并附带后续动作
- 基线文档可以直接进入 `spec-review`
- 宿主安装态、宿主命令注册和外部集成边界均有明确证据路径

## 2. 功能需求

| REQ ID | 需求声明 | 说明 |

| REQ-BASE-001 | 必须形成当前系统能力总览 | 覆盖 CLI、Core、Skill、Template 与宿主集成 |
| REQ-BASE-002 | 必须识别架构边界和关键流程 | 覆盖阶段流转、技能分发、初始化、验证、发布等主链路 |
| REQ-BASE-003 | 必须识别缺口、技术债务与风险 | 供后续规划和评审使用 |
| REQ-BASE-004 | 必须产出可评审、可追溯的基线包 | 可进入 spec-review，并作为 design 输入 |
| REQ-BASE-005 | 必须纳入宿主安装态与外部边界证据 | 覆盖宿主技能目录、命令注册、同步结果和外部集成/部署边界 |

| AC-04 | 任何不确定事实都必须标记为 `[ASSUMED]` 或 `[NEEDS CLARIFICATION]`，不得静默猜测 | ST |

**Upstream REQ**: `REQ-BASE-002`

**说明**：需要把阶段流转、技能分发、初始化、门禁、验证与追溯相关的主流程写清楚，明确当前系统的边界在哪里。

| AC-04 | 任何边界或外部集成点都必须附带证据路径，缺失则记录为开放问题 | ST |

### 2.3 FR-BASE-003：缺口、技术债务与风险可用于后续规划

**Upstream REQ**: `REQ-BASE-003`

| AC-04 | 规格中不允许把历史分析材料当成当前机制真源；历史材料只能作为背景证据 | ST |

### 2.4 FR-BASE-004：基线包可评审、可追溯、可进入下一阶段

**Upstream REQ**: `REQ-BASE-004`

### 2.5 FR-BASE-005：宿主安装态与外部边界可被纳入基线

| AC ID | 验收标准 | 建议测试层级 |

## 3. 非功能需求

### 3.1 追溯性

- 所有关键结论必须能追溯到至少一个证据路径
- 所有 FR 都必须有对应的 REQ upstream 引用
- 所有开放问题都必须在 `findings.md` 中有记录

### 3.2 完整性

- 必须覆盖 runtime summary 中列出的核心模块
- 必须覆盖 critical flows 中列出的关键流程
- 必须覆盖当前 Feature 的主链路与宿主边界

### 3.3 可靠性与确定性

- 同一组证据反复生成，结论应保持稳定
- 不同时间点的文本措辞可以调整，但事实结论不得漂移
- 不得把推测写成既定事实

### 3.4 安全与合规

- 不得输出密钥、凭据、个人数据或宿主敏感配置
- 外部信息不足时应标记为待确认，而不是擅自补全
- 宿主安装态只作为必要证据，不扩展成不受控的环境描述

### 3.5 可观测性

- 规格中的每个重要结论都应有证据或状态标记
- 未确认项必须能在 findings 中快速定位
- 文档之间的关联关系应可由 document-links 解析
- 宿主安装态与外部边界证据应能从 findings.md 或补充记录中定位

## 4. 约束条件

### 4.1 范围约束

- 只做当前系统基线分析，不设计新功能实现
- 不承诺向下兼容
- 不把历史文档里的旧 gate / trace / doc-links 机制当作当前主链路

### 4.2 证据约束

- 以仓库内代码、运行态资产和技能文档为主证据
- 宿主侧信息仅在明确相关时作为辅助证据
- 证据不足时必须显式标记 `[NEEDS CLARIFICATION]`

### 4.3 输出约束

- FR/AC 必须可测试、可判定
- 每个 AC 必须只描述一个断言
- 不允许把实现细节、类名、函数名写成需求本体

## 5. 已确认范围

| 项目 | 优先级 | 负责人 | 状态 |
|------|--------|--------|------|
| 宿主安装态与外部集成/部署边界 | Medium | 待确认 | Resolved |
| 生产/运维信息若仓库内无证据，则仅作为后续补充信息，不作为本次基线阻断项 | Medium | 待确认 | Resolved |

## 6. 术语表

| 术语 | 定义 |
|------|------|
| Baseline | 当前系统的事实底座，用于后续设计和规划 |
| Evidence Path | 支撑某条结论的文件、代码或运行态资产路径 |
| Gap | 当前能力与目标能力之间的差距 |
| Technical Debt | 会增加后续变更成本的遗留实现或约束 |
| Host Artifact | 安装在宿主环境中的技能、配置或同步产物 |
| Traceability | 需求、证据、任务与结果之间可追溯的关系 |

## 7. 风险与依赖

### 7.1 依赖

- `.spec-first/runtime/first/` 下的 runtime 资产必须可读取，否则基线会降级为不完整状态
- `specs/FSREQ-19700101-LEGACY-BASELINE/constitution.md` 必须存在，否则无法完成一致性判断
- `document-links.yaml`、`findings.md`、`task_plan.md` 需要保持同步，否则基线包无法稳定传递到后续阶段

### 7.2 风险

- **范围漂移**：如果把宿主安装态、历史分析材料和仓库事实混为一谈，基线会失真
- **证据不足**：外部集成点或部署信息缺失时，容易把推测写成结论
- **旧模型污染**：gate / trace / doc-links 的历史材料若未明确降级，可能干扰当前认知

### 7.3 回退约束

- 本 Feature 处于规格定义阶段，若 `spec-review` 未通过，应回到 `spec.md` / `prd.md` 修订，而不是推进设计
- 当前阶段不涉及实现回滚，也不需要产品级兼容方案
- 任何回退只影响文档层，不应影响宿主安装态或运行时代码

## 8. 下一步

- 运行 `spec-review` 校验 FR / AC 的完整性、歧义与一致性
- 若审查通过，再进入 `design` 阶段补充技术方案

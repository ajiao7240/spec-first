# 需求文档（Spec）

## 1. 背景与目标

当前登录仅支持账号密码，存在两类问题：

1. 密码登录失败率偏高（忘记密码、输入错误）。
2. 在弱网场景下重试体验差，用户流失明显。

本次迭代目标：

1. 在不破坏原账号密码登录的前提下，新增短信验证码登录。
2. 登录成功率提升，且安全风险可控。
3. 全链路可追踪（FR/NFR → DS/API → TASK → TC → 报告）。

## 2. 范围

### 2.1 In Scope

1. H5 登录页增加"短信登录"入口与交互流程。
2. 后端新增验证码发送与验证码登录接口。
3. 风控策略：验证码有效期、重试次数、频控与防重放。
4. 关键链路埋点与监控。
5. 邮箱验证码登录：发送邮件验证码、邮箱登录接口、邮箱频控策略。

### 2.2 Out of Scope

1. 第三方社交登录（微信/Apple/Google）。
2. 密码找回流程重构。
3. 多租户认证体系改造。

## 3. 角色与用户故事

### 3.1 角色

1. 未登录用户（终端用户）。
2. 安全风控系统（内部系统角色）。
3. 运维与值班人员（监控告警消费方）。

### 3.2 用户故事

1. 作为用户，我希望可以通过短信验证码快速登录，以减少记密码负担。
2. 作为用户，我希望在验证码过期或错误时得到明确提示，并可重新获取。
3. 作为风控系统，我需要限制异常请求，防止短信轰炸与撞库攻击。
4. 作为用户，我希望可以通过邮箱验证码登录，以在没有手机的场景下也能快速登录。

## 4. 功能需求（FR）

### FR-AUTH-001 账号密码登录体验优化

描述：在保留原有账号密码登录能力下，优化错误提示与重试体验。

验收标准：

1. FR-AUTH-001/AC-1
Given 用户输入正确账号密码
When 点击登录
Then 登录成功并进入首页。

2. FR-AUTH-001/AC-2
Given 用户输入错误密码
When 点击登录
Then 返回统一错误提示，不暴露账号存在性。

3. FR-AUTH-001/AC-3
Given 连续多次失败
When 超过重试阈值
Then 触发节流策略并提示稍后再试。

### FR-AUTH-002 新增短信验证码登录

描述：新增手机号 + 验证码登录方式，支持验证码发送、校验、登录。

验收标准：

1. FR-AUTH-002/AC-1
Given 用户输入合法手机号
When 点击“发送验证码”
Then 系统成功发送验证码并返回倒计时状态。

2. FR-AUTH-002/AC-2
Given 用户输入正确验证码且未过期
When 点击登录
Then 登录成功并签发会话。

3. FR-AUTH-002/AC-3
Given 用户输入错误验证码
When 点击登录
Then 返回错误提示并累计失败次数。

4. FR-AUTH-002/AC-4
Given 验证码已过期
When 点击登录
Then 返回“验证码已过期，请重新获取”。

5. FR-AUTH-002/AC-5
Given 同一手机号在倒计时期间重复请求
When 再次点击发送
Then 前端禁止触发，后端返回频控响应。

### FR-AUTH-003 新增邮箱验证码登录

描述：新增邮箱 + 验证码登录方式，支持验证码发送、校验、登录。

验收标准：

1. FR-AUTH-003/AC-1
Given 用户输入合法邮箱地址
When 点击"发送验证码"
Then 系统成功发送验证码邮件并返回倒计时状态。

2. FR-AUTH-003/AC-2
Given 用户输入正确验证码且未过期
When 点击登录
Then 登录成功并签发会话。

3. FR-AUTH-003/AC-3
Given 用户输入错误验证码
When 点击登录
Then 返回错误提示并累计失败次数。

4. FR-AUTH-003/AC-4
Given 验证码已过期
When 点击登录
Then 返回"验证码已过期，请重新获取"。

5. FR-AUTH-003/AC-5
Given 同一邮箱在倒计时期间重复请求
When 再次点击发送
Then 前端禁止触发，后端返回频控响应。

## 5. 非功能需求（NFR）

### NFR-SEC-001 安全

1. 验证码 TTL = 5 分钟（短信）/ 10 分钟（邮箱）。
2. 单验证码最大尝试次数 = 5 次。
3. 防重放：验证码一次性消费，成功后立即失效。
4. 频控：手机号维度、IP 维度联合限制。
5. 邮箱频控：邮箱维度 60 秒内最多 1 次发送。

### NFR-PERF-001 性能

1. 发送验证码接口 P95 < 300ms（不含短信通道外部延迟）。
2. 验证码登录接口 P95 < 200ms。

### NFR-OBS-001 可观测性

1. 关键事件需埋点：发送验证码、验证码登录成功/失败、频控触发。
2. 必须可按 `featureId` 与 `taskId` 聚合日志与指标。

### NFR-REL-001 稳定性

1. 第三方短信通道异常时，返回可理解错误并支持快速降级。
2. 失败不应影响原密码登录可用性。

## 6. 约束与假设

1. 短信通道由现有供应商提供，签名模板已备案。
2. 登录态签发逻辑沿用现有会话体系。
3. 本需求不改动用户主表结构，仅新增 OTP 会话存储。
4. 邮件通道由现有邮件服务提供，模板已配置。

## 7. 风险与缓解

1. 短信通道抖动导致发送失败率上升。
缓解：增加重试与降级提示，监控告警阈值前置。

2. 风控参数过严影响正常用户。
缓解：灰度阶段采用可配置阈值并实时观测。

3. 前端倒计时与后端频控不一致。
缓解：以后端判定为准，前端仅做体验层限制。

## 8. 验收与度量

### 8.1 业务验收

1. 短信登录主路径可用。
2. 原密码登录无回归。
3. 安全策略（TTL、重试、频控）生效。

### 8.2 指标目标

1. 登录成功率提升（相对基线 +3% 以上）。
2. 短信登录占比达到可观测阈值（用于后续迭代决策）。
3. 安全高危告警为 0。

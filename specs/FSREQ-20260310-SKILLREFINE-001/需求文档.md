# Spec-First 需求细化与优化方案

## 一、需求优化框架
将原始需求拆解为 **「代码库建模 → Skill 阶段验证 → 全流程健壮性审查 → 多Agent视角Skill审计 → 问题清单与优化方案输出」** 五个闭环阶段，每个阶段明确目标、任务、交付物与验证标准。


## 二、分阶段细化执行方案

### 阶段 1：代码库深度理解与架构建模
**目标**：建立代码库的全局认知模型，为后续验证提供基础。

**核心任务**：
1. **代码库结构扫描**：
   - 分析目录结构，识别 Skill 层、CLI 层、运行时层的核心模块与文件组织。
   - 梳理模块依赖关系（如 Skill 如何调用 CLI、运行时层如何支撑状态流转）。
2. **核心概念提取**：
   - 定义 `Context Pack`、`Gate`、`追踪矩阵`、`阶段状态机` 等核心概念的代码实现位置与逻辑。
   - 梳理 Skill 的分类与职责边界（如需求澄清Skill、设计Skill、任务分解Skill等）。
3. **技术栈与依赖分析**：
   - 记录核心依赖库、框架（如 Node.js/Docker 相关集成）。
   - 识别配置文件、状态存储方式（如是否用本地文件/数据库追踪流程）。

**输入**：完整 Spec-First 代码库。  
**输出**：《代码库架构模型文档》（含模块依赖图、核心概念映射表、技术栈清单）。  
**验证标准**：能通过模型快速定位任意 Skill/CLI 命令的实现位置与调用链路。


### 阶段 2：Skill 阶段深度验证
**目标**：逐个验证 Skill 层各阶段的逻辑完整性、与 Spec-First 目标的契合度。

**核心任务**：
1. **Skill 清单梳理**：
   - 枚举所有 Skill，明确每个 Skill 的「输入条件 → 处理逻辑 → 输出交付物 → 触发下一阶段的 Gate 规则」。
2. **单 Skill 逻辑验证**：
   - 模拟输入（如需求文档片段、代码上下文），验证 Skill 输出是否符合预期（如需求澄清是否覆盖关键歧义点、设计是否关联可追踪的任务）。
   - 检查 Skill 对「异常输入」的处理（如需求模糊时是否主动追问、代码冲突时是否提示风险）。
3. **Skill 间衔接验证**：
   - 验证前一阶段 Skill 的输出是否能作为后一阶段的输入（如需求确认后的文档是否能直接驱动设计 Skill）。
   - 检查状态机流转：Skill 执行完成后，是否正确更新阶段状态、触发 Gate 校验。

**输入**：《代码库架构模型文档》、核心 Skill 代码。  
**输出**：《单 Skill 验证报告》（含每个 Skill 的测试用例、执行结果、逻辑缺陷记录）。  
**验证标准**：每个 Skill 至少覆盖「正常流程、异常流程、边界场景」三类测试用例，且通过率≥80%。


### 阶段 3：全流程健壮性审查
**目标**：模拟从需求澄清到发布的完整闭环，验证流程的可追踪性、可恢复性与自主推进能力。

**核心任务**：
1. **端到端流程模拟**：
   - 以「一个虚拟功能需求」为起点，触发需求澄清 Skill → 设计 Skill → 任务分解 Skill → 代码生成 Skill → 校验 Skill → 发布 Skill 的全流程。
   - 记录每个阶段的状态变更、Context Pack 内容变化、追踪矩阵的更新。
2. **可恢复性验证**：
   - 模拟流程中断（如手动终止、异常报错），验证是否能通过 Context Pack 恢复会话，无需重复输入项目背景。
   - 检查「人工介入修改」后的流程衔接（如人工调整设计后，Skill 是否能基于修改后的内容继续推进）。
3. **Gate 规则有效性验证**：
   - 故意制造「不满足 Gate 条件」的场景（如需求未确认就进入设计、代码未通过校验就触发发布），验证流程是否被正确阻断。

**输入**：《单 Skill 验证报告》、虚拟需求场景。  
**输出**：《全流程健壮性审查报告》（含端到端执行日志、中断恢复测试结果、Gate 规则缺陷记录）。  
**验证标准**：流程能自主推进≥3个连续阶段，且中断后能100%恢复上下文。


### 阶段 4：多Agent视角 Skill 审计
**目标**：从不同角色（AI协同开发者、流程治理负责人、团队协作场景）出发，审查 Skill 的适配性与问题。

**核心任务**：
1. **角色1：AI协同开发者视角**
   - 审查 Skill 是否能「降低重复沟通成本」（如需求澄清是否覆盖高频歧义点、代码生成是否结合项目已有规范）。
   - 验证 Skill 对「人工控制权」的保留（如是否在关键节点提供「确认/修改/拒绝」选项，而非强制推进）。
2. **角色2：流程治理负责人视角**
   - 检查 Skill 是否能「沉淀流程证据」（如追踪矩阵是否记录需求→代码的关联、Gate 校验是否留存日志）。
   - 审查 Skill 对「变更管理」的支持（如需求变更时，是否能追溯影响的任务与代码）。
3. **角色3：团队协作场景视角**
   - 模拟多成员协作，验证 Skill 输出是否「易读、易交接」（如文档生成是否符合团队规范、任务分解是否明确责任人）。

**输入**：《全流程健壮性审查报告》、角色场景模拟脚本。  
**输出**：《多Agent视角审计报告》（含各角色场景下的问题记录、适配性建议）。  
**验证标准**：每个角色视角至少发现3个与目标场景相关的问题或改进点。


### 阶段 5：问题整理与优化清单输出
**目标**：汇总所有问题，形成结构化的优化清单，明确优先级与落地路径。

**核心任务**：
1. **问题分类与优先级定义**：
   - 按「逻辑缺陷、流程断点、体验优化、架构改进」分类问题。
   - 定义优先级：P0（阻塞核心流程）、P1（影响核心目标）、P2（体验/扩展性优化）。
2. **优化方案制定**：
   - 针对每个问题，给出「代码修改建议、Skill 逻辑调整建议、流程规则补充建议」。
   - 明确优化后的验证标准（如修复后需通过哪类测试用例）。
3. ** roadmap 规划**：
   - 按优先级输出分阶段优化计划（如P0问题1周内修复，P1问题2周内迭代）。

**输入**：前四个阶段的所有报告。  
**输出**：《Spec-First Skill 优化清单与落地计划》（含问题列表、优化方案、优先级、时间规划）。  
**验证标准**：清单中100%的问题有明确的优化方向与验证标准，P0问题有可直接执行的修复步骤。


## 三、整体执行路径总结
1. **先建模**：通过阶段1建立代码库认知，避免后续验证的盲目性。  
2. **再验证**：从单 Skill 到全流程，层层递进验证逻辑与健壮性。  
3. **多视角审计**：结合目标用户角色，确保 Skill 贴合实际场景需求。  
4. **结构化输出**：最终输出可落地的优化清单，而非泛泛的问题描述。

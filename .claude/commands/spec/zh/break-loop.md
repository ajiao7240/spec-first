# 打破循环 - 深度 Bug 分析

当调试完成时，使用此命令进行深度分析，以打破"修复 bug -> 忘记 -> 重复"的循环。

---

## 分析框架

从这 5 个维度分析你刚修复的 bug：

### 1. 根本原因类别

这个 bug 属于哪个类别？

| Category | Characteristics | Example |
|----------|-----------------|---------|
| **A. 缺少规范** | 没有关于如何做的文档 | 新功能没有检查清单 |
| **B. 跨层约定** | 层之间的接口不清楚 | API 返回的格式与预期不同 |
| **C. 变更传播失败** | 改了一处，漏了其他 | 更改了函数签名，漏了调用点 |
| **D. 测试覆盖缺口** | 单元测试通过，集成测试失败 | 单独工作，组合时失败 |
| **E. 隐式假设** | 代码依赖未记录的假设 | 时间戳秒 vs 毫秒 |

### 2. 为什么修复失败（如果适用）

如果你在成功之前尝试了多次修复，分析每次失败：

- **表面修复**：修复了症状，不是根本原因
- **不完整范围**：找到了根本原因，没有覆盖所有情况
- **工具限制**：grep 漏掉了，类型检查不够严格
- **心智模型**：一直在同一层找，没有考虑跨层

### 3. 预防机制

什么机制可以防止这种情况再次发生？

| Type | Description | Example |
|------|-------------|---------|
| **文档** | 写下来让人们知道 | 更新思维指南 |
| **架构** | 在结构上使错误不可能 | 类型安全包装器 |
| **编译时** | TypeScript strict，没有 any | 签名更改导致编译错误 |
| **运行时** | 监控、警报、扫描 | 检测孤立实体 |
| **测试覆盖** | E2E 测试、集成测试 | 验证完整流程 |
| **代码审查** | 检查清单、PR 模板 | "你检查了 X 吗？" |

### 4. 系统性扩展

这个 bug 揭示了哪些更广泛的问题？

- **类似问题**：其他地方可能存在这个问题吗？
- **设计缺陷**：是否存在根本性的架构问题？
- **流程缺陷**：是否有开发流程改进？
- **知识缺口**：团队是否缺少某些理解？

### 5. 知识捕获

将见解固化到系统中：

- [ ] 更新 `.spec-first/spec/guides/` 思维指南
- [ ] 更新 `.spec-first/spec/backend/` 或 `frontend/` 文档
- [ ] 创建问题记录（如果适用）
- [ ] 为根本修复创建功能工单
- [ ] 如果需要，更新检查命令

## 输出格式

请以以下格式输出分析：

```markdown
## Bug Analysis: [简短描述]

### 1. Root Cause Category
- **Category**: [A/B/C/D/E] - [类别名称]
- **Specific Cause**: [详细描述]

### 2. Why Fixes Failed (if applicable)
1. [第一次尝试]: [为什么失败]
2. [第二次尝试]: [为什么失败]
...

### 3. Prevention Mechanisms
| Priority | Mechanism | Specific Action | Status |
|----------|-----------|-----------------|--------|
| P0 | ... | ... | TODO/DONE |

### 4. Systematic Expansion
- **Similar Issues**: [列出有类似问题的地方]
- **Design Improvement**: [架构级建议]
- **Process Improvement**: [开发流程建议]

### 5. Knowledge Capture
- [ ] [要更新的文档 / 要创建的工单]
```

## 核心哲学

> **调试的价值不在于修复 bug，而在于使这类 bug 永远不再发生。**

三个层次的见解：
1. **战术**：如何修复这个 bug
2. **战略**：如何预防这类 bug
3. **哲学**：如何扩展思维模式

30 分钟的分析节省 30 小时的未来调试。

---

## 分析后：立即行动

**重要**：完成上述分析后，你必须立即：

1. **更新 spec/guides** - 不要只是列出 TODO，实际更新相关文件：
   - 如果是跨平台问题 → 更新 `cross-platform-thinking-guide.md`
   - 如果是跨层问题 → 更新 `cross-layer-thinking-guide.md`
   - 如果是代码复用问题 → 更新 `code-reuse-thinking-guide.md`
   - 如果是领域特定的 → 更新 `backend/*.md` 或 `frontend/*.md`

2. **同步模板** - 更新 `.spec-first/spec/` 后，同步到 `src/templates/markdown/spec/`

3. **提交 spec-first 更新** - 这是主要输出，不只是分析文本

> **如果分析只留在聊天中，它是无价值的。价值在于更新的规范。**
